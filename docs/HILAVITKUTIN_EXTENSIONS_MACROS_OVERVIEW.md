# hilavitkutin-extensions-macros

Proc-macro crate paired with `hilavitkutin-extensions`. Emits the `#[repr(C)]` `ExtensionDescriptor` constant, the `__hilavitkutin_extension_descriptor` exported fn, the `CapabilityEntry` table, and per-capability C-ABI trampolines on behalf of an extension author. Shipping this macro as a sibling of the host crate means every real consumer gets the same descriptor plumbing for free and the plumbing stays consistent across downstream ecosystems.

The crate is inherently `std`-backed at macro-execution time (proc-macros run in the compiler's host context). Its emitted output is constrained to the same `no_std` posture as `hilavitkutin-extensions`: no `alloc`, no `dyn`, no runtime hashing, no linker-magic registration. The macro emission path uses `::hilavitkutin_extensions::*` qualified paths so the emitted code is correct regardless of the consumer's import layout.

## Public items

- `export_extension` is the primary attribute macro. Placed on a struct (or on an `impl` block in some variants) that implements the capability traits the extension exports, it emits the full descriptor surface.

The crate has no public types. Its contract is the macro attribute and the shape of the code it emits. The emitted code refers to types defined in `hilavitkutin-extensions` (`ExtensionDescriptor`, `CapabilityEntry`, `CapabilityId`, `ExtensionMeta`, `InitHandler`, `ShutdownHandler`, `CapabilityExport`, `HOST_ABI_VERSION`, `DESCRIPTOR_SYMBOL`, `ExtensionAbiStatus`, `ExtensionVersion`).

## Macro behaviour

`#[export_extension]` reads two sources of information about the extension:

1. Attribute parameters on the invocation site. Supported keys: `name = "..."` (byte-string literal; defaults to the struct ident in kebab-case), `version = "MAJOR.MINOR.PATCH"` (string literal; defaults to reading `CARGO_PKG_VERSION` via `env!`), `required_host_caps = [CAP_A, CAP_B, ...]` (path expressions resolving to `CapabilityId` const items; defaults to empty).
2. Trait implementations on the target type in the surrounding module. `ExtensionMeta`, `InitHandler`, and `ShutdownHandler` are discovered via path-qualification in emitted code; the macro does not parse arbitrary `impl` blocks to find them. `CapabilityExport` impls on the target type (one per capability) drive the capability table generation.

The emission has four components:

1. A `const DESCRIPTOR: ExtensionDescriptor = ExtensionDescriptor { ... }` item whose fields are filled from the attribute parameters and trait-derived values. The `capabilities_ptr` field points at a second static constant (`const CAPABILITIES: &[CapabilityEntry] = &[ ... ];`), populated from each `CapabilityExport` impl on the target type.
2. `#[no_mangle] pub extern "C" fn __hilavitkutin_extension_descriptor() -> *const ExtensionDescriptor { &DESCRIPTOR }`. The fn name uses the `DESCRIPTOR_SYMBOL` byte-string content (without the trailing null). Rust compile-time checks that the fn name matches the constant is not enforced by the macro (both come from the same crate-version pairing).
3. Per-capability trampolines: for each `CapabilityExport` impl, an `extern "C" fn trampoline_<cap>(args...) -> Ret` that calls into the typed Rust method on the target type with appropriate argument marshalling. The specific signatures depend on consumer-defined domain types that implement `CapabilityExport`; this crate only provides the dispatching shell.
4. Optional `init_fn` / `shutdown_fn` trampolines emitted only when the target type implements `InitHandler` / `ShutdownHandler`. When those traits are absent, the descriptor's `init_fn` / `shutdown_fn` fields are `None`.

## Customisation path

Attribute parameters cover the common case. Trait implementations cover the author-controlled behaviour. For consumer contract crates that want a domain-native attribute, the recommended pattern is: ship a wrapping proc-macro (e.g. `#[viola::plugin]`) whose expansion is equivalent to `#[export_extension(<params set by consumer>)] <input item>`. The macro-wraps-macro idiom is standard Rust and does not require any cooperation from this crate beyond exposing `export_extension` as reachable from generated code.

## Arbitrary-time linking invariant

The emitted output preserves the invariant because it is fundamentally per-extension: one `DESCRIPTOR` constant per extension, one exported fn per extension, one capability table per extension.
