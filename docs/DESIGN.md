# hilavitkutin — Architecture Design

> Morsel-driven pipeline execution engine. `#![no_std]`, no alloc.
> Consumers declare WorkUnits with typed read/write access sets; the
> engine analyses them into phases/trunks/fibers, compiles per-core
> dispatch programs, runs them on a pre-allocated thread pool.

hilavitkutin-ctx, -persistence, -str, -kit, -providers are standalone ecosystem crates that consumers opt into. ctx, persistence, str are pure ecosystem (engine does not depend on them). kit is a contract crate the engine depends on for its `.add_kit()` method, but it has no reverse dependency on the engine; consumers can author Kits without pulling the engine in. providers ships default Kit impls consumers wire in (interner, future ColumnStorage / Clock / Memory defaults); it consumes api + str + kit + substrate, no reverse dep.

hilavitkutin-linking, hilavitkutin-extensions, hilavitkutin-extensions-macros make up the plugin-host layer. Downstream consumers (viola, future loimu, future clause extension surfaces) build domain-specific contract crates on top.

## Principles (see PRINCIPLES.md.tmpl)

1. `#![no_std]`, no alloc, every crate.
2. Monomorphisation is the dispatch (no dyn, no TypeId).
3. Pre-allocated thread pool; no runtime spawn beyond plan-time.
4. Schedule once, reuse across frames. Plan recompute is rare.
5. Self-hosting meta pipeline. The scheduler schedules itself.
