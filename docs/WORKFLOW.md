# Workflow

How a consumer uses hilavitkutin from a Rust project, what shape the
dependency takes, and how the engine evolves.

## Adding hilavitkutin to a project

```toml
[dependencies]
hilavitkutin = { version = "0.0", default-features = false }
hilavitkutin-api = { version = "0.0" }
```

Pull `hilavitkutin-api` for the consumer-facing types (`WorkUnit`,
`AccessSet`, `Column`, `Resource`, `Virtual`, `Context`, the platform
contracts) and `hilavitkutin` for the scheduler implementation. The
ecosystem crates (`hilavitkutin-ctx`, `hilavitkutin-persistence`,
`hilavitkutin-str`, `hilavitkutin-kit`, `hilavitkutin-providers`) are
opt-in; pull only the ones your app needs.

Build-time optimisations (LLVM passes, cfg emission, PGO/BOLT
workflows) live in `hilavitkutin-build`. Add it as a build dependency
in your `build.rs` only; it does not link into the runtime binary.

## Authoring a WorkUnit

Implement `WorkUnit` on a marker type. Declare `Read` and `Write` as
type-level access sets. Implement `execute` against a `Context` typed
to the access bounds. The scheduler analyses the access sets at
compile time, plans phases / trunks / fibers, compiles per-core
dispatch programs, and runs them on a pre-allocated thread pool.

Static composition only: the scheduler builder accepts WorkUnits at
compile time; there is no runtime registration. No `dyn`, no heap
allocations, no `thread::spawn` at runtime.

## Pre-1.0 stability

hilavitkutin is in pre-1.0 design phase. Public surfaces stabilise at
`0.0.x`; expect breaking changes between minor versions. The
authoritative API contract is `hilavitkutin-api`; depending on engine
internals (`hilavitkutin::*`) outside the documented public surface is
not supported.

## Reporting issues

Issues and feature discussion happen on the GitHub repo. Patches
