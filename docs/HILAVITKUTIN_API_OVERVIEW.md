# hilavitkutin-api: Design

Consumer-facing types, traits, and platform contracts. Downstream
consumer applications depend on this crate to define WorkUnits. The
engine crate (hilavitkutin) also depends on it. `#![no_std]`, no
alloc. All index, count, and size types use arvo newtypes.

Depends on `hilavitkutin-ctx` for its `provider_generic!` /
`provider_generic2!` macros and the `Context<P>` provider-tuple
wrapper. WU execute parameter types and the WU-context accessor
traits are expressed via those macros; see "WU Context providers"
below.

The crate uses `provider` as a single workspace-wide vocabulary.
Two trait families share the noun:

1. The `Provider` trait declared in this section gates registration
   on the SchedulerBuilder. Every value passed to `.with(value)`
   impls it. See `## Provider, the unified registration contract`.
2. The ctx-framework accessor traits (`HasResourceProvider`,
   `HasColumnReader`, etc.) generated by `provider_generic!`
   expose runtime providers on the WU Context tuple. See
   `## WU Context providers` below.

The two families connect: a `Resource<T>` registered via
`.with(Resource::new(value))` becomes reachable in WU Context
tuples through `HasResourceProvider<R>`. Registration is the
build-time half; accessors are the run-time half. Both refer to
the same concept (a thing the scheduler provides), so both wear
the `provider` label.

## Provider, the unified registration contract

`SchedulerBuilder` takes one verb: `.with(value)`. Every type
accepted by `.with` impls the sealed `Provider` trait. The trait
gates registration and routes per-kind typestate updates through
its `Dispatch` associated type.

```rust
pub trait Provider {
    type Init;
    const KIND: ProviderKind;
    type Dispatch: Dispatch;
}
```

The associated `Init` names the construction-time value type. For
stateful wrappers (`Resource<T>`) it is the inner `T`. For the
zero-data wrappers (`Column<T>`, `Virtual<T>`, `LinkedBin<dyn Trait>`)
it is `()`. For unit-struct WorkUnits, Kits, and platform-impl types
it is `Self` (the registration value IS the construction value).

`Dispatch` is the per-kind typestate router (next subsection). The
constant `KIND: ProviderKind` is a documentation and debugging aid;
load-bearing dispatch is via `Provider::Dispatch`, not the constant.

The trait carries `#[diagnostic::on_unimplemented]`. The message
points at the constructors a consumer reaches for: `Resource::new(value)`,
`Column::<T>::new()`, `Virtual::<T>::new()`, `LinkedBin::<dyn TraitFamily>::new()`,
plus the unit-struct WorkUnit, Kit, and platform-impl shapes. A type
that does not impl `Provider` cannot reach `.with`; the diagnostic
names the gap directly.

`Provider` is sealed via a private `Token` pattern. Consumer crates
cannot impl `Provider` directly; the substrate mints the trait for
the four router shapes (Unit, Store, Kit, Platform) plus the
consumer-reachable unit-struct shapes (WorkUnits, Kits, platform
impls).

## Dispatch: per-kind typestate routing

```rust
pub trait Dispatch {
    type NextWus<Wus>;
    type NextStores<Stores>;
    type NextPlatform<Platform>;
}
```

Each generic-associated type computes the next typestate accumulator
when a provider of the corresponding kind registers. The four router
structs sit beside `Provider`:

| Router | Routes |
|---|---|
| `UnitDispatch<W>` | Prepends `W` to the `Wus` accumulator. |
| `StoreDispatch<S>` | Prepends `S` to the `Stores` accumulator. |
| `KitDispatch<K>` | Concats `K::Units` into `Wus` and `K::Owned` into `Stores`. |
| `PlatformDispatch<P>` | Prepends `P` to the platform-tuple accumulator. |

For routers that do not affect a given accumulator, the
corresponding GAT is identity. `UnitDispatch<W>::NextWus<Wus> =
Cons<W, Wus>`; its `NextStores` and `NextPlatform` pass `Stores` and
`Platform` through unchanged. `StoreDispatch<S>::NextStores<Stores>
= Cons<S, Stores>`; its other two GATs are identity.
`KitDispatch<K>::NextWus<Wus> = <K::Units as Concat<Wus>>::Out`;
analogous for stores; platform GAT is identity.
`PlatformDispatch<P>::NextPlatform<Platform> = Cons<P, Platform>`;
its other two GATs are identity.

A single blanket impl on `SchedulerBuilder` over `P: Provider` reads
`P::Dispatch` and walks the per-kind GAT. The trait solver fails on
`Provider` directly when a non-`Provider` value is passed to
`.with`, surfacing the right diagnostic at the call site. Per-kind
blankets would route the failure through the wrong trait first (an
error pointing at `WorkUnit` when the user passed a `u32`).
Validated empirically; see
`mock/research/sketches/202605091700-builder-provider-shape/`.

## Store types (domain 06)

Three store types, one scheduling mechanism:

```rust
pub struct Resource<T>(PhantomData<T>); // singleton, one value
pub struct Column<T>(PhantomData<T>); // collection, morsel-chunked
pub struct Virtual<T>(PhantomData<T>); // zero-data, DAG edge only
```

The scheduler sees StoreId pairs for edge construction. Does not care
about cardinality. `Write<Resource<X>> → Read<Resource<X>>` builds
the same edge as `Write<Column<Y>> → Read<Column<Y>>`.

`StoreId` is a dense index assigned at plan time. Wraps an arvo
fixed-point newtype (not raw u16). `AccessMask` is the arvo-bitmask
`Mask64` (<=64 stores) or `Mask256` (<=256 stores). All scheduler ops
= single-instruction bitwise.

`AccessSet` is a sealed marker on the cons-list typestate. `Empty`
and `Cons<H, T>` are the substrate primitives consumer code
encounters either directly or through the `read!` / `write!`
macros (which expand to nested `Cons` chains). Compile-time store
descriptors. Plan phase converts to runtime AccessMask via
monomorphised generic functions (the monomorphised function IS the
type identity, no TypeId, no std::any).

ZST erasure: column marker types constrain the DAG, enforce type
safety, and determine morsel sizes at compile time. Erased to plain
indices by execution time.

### Store wrappers as Providers

`Resource<T>`, `Column<T>`, and `Virtual<T>` each impl `Provider` so
they reach `.with` on the SchedulerBuilder. Their `Dispatch` is
`StoreDispatch<Self>`. `Resource<T>::Init = T` (the wrapper carries
the inner value at construction). `Column<T>::Init = ()` and
`Virtual<T>::Init = ()` (zero-data markers).

Construction at the call site is uniform:

```rust
.with(Resource::new(value))     // stateful wrapper, takes inner T
.with(Column::<T>::new())       // zero-data wrapper, generic turbofish
.with(Virtual::<T>::new())      // zero-data wrapper, generic turbofish
```

`Column` and `Virtual` impl `HasTrivialCtor` (from notko); the
`new()` method on each calls through to the trait's const-callable
zero-arg constructor. `Resource` keeps its inherent
`pub const fn new(value: T) -> Self` constructor because it carries
state; it does not impl `HasTrivialCtor`.

## LinkedBin

```rust
pub struct LinkedBin<T: ?Sized>(PhantomData<*const T>);
```

Type-system marker for an extension-loaded provider bin keyed by a
trait family. `LinkedBin<dyn LinterApi>` reserves a slot in the
SchedulerBuilder typestate that runtime instances populate via
`hilavitkutin-extensions`. `LinkedBin` impls `Provider<Init = ()>`
with `Dispatch = StoreDispatch<Self>`. It impls `HasTrivialCtor`
for the `LinkedBin::<dyn TraitFamily>::new()` form.

The wrapper is the type-system anchor only. Runtime instance
population happens in `hilavitkutin-extensions` plus `hilavitkutin`;
this declaration ships the marker so WUs can declare reads and
writes against `LinkedBin<dyn TraitFamily>` and the typestate
proves coverage at `.build()` time.

`?Sized` permits `dyn TraitFamily` parameters. The `PhantomData<*const T>`
field carries no `Send` / `Sync` requirements onto the wrapper
itself; bounds propagate from the consumer-side trait definition.

## ColumnValue trait (domain 05, R4)

```rust
pub trait ColumnValue: Copy + 'static {
 const BIT_WIDTH: usize; // no default on trait itself
}

// Blanket impl via min_specialization
impl<T: Copy + 'static> ColumnValue for T {
 default const BIT_WIDTH: usize = core::mem::size_of::<Self>() * 8;
}
```

Any `Copy + 'static` type ≤16 bytes is column-storable. Zero friction
for consumers. BIT_WIDTH informs the storage engine for bitpacking.
Sub-byte arvo types specialise (e.g. `UFixed<1,0>` → BIT_WIDTH=1).
No pack/unpack: type IS the storage (R3, R4).

## WorkUnit trait (domain 09)

```rust
pub trait WorkUnit<Schedule = Always>:
    Provider<Init = Self> + Send + Sync + 'static
{
 type Read: AccessSet;
 type Write: AccessSet;
 type Hint: SchedulingHint = (Immediate, Atomic, Normal);
 type Ctx<'frame>: HasColumnReader<Self::Read>
 + HasColumnWriter<Self::Write>
 + HasResourceProvider<Self::Read>
 + HasVirtualFirer<Self::Write>
 + HasEach<Self::Read, Self::Write>
 + HasBatch<Self::Read, Self::Write>
 + HasReduce<Self::Read, Self::Write>;
 const COMMUTATIVE: bool = false;
 fn execute<'frame>(&self, ctx: &Self::Ctx<'frame>);
}
```

The `Ctx` associated type is a Generic Associated Type (GAT) over a
`'frame` lifetime. The engine instantiates it at dispatch time with the
live frame lifetime so the WU can read borrows into `PoolFrame`-derived
state (progress slots, predicted-wait atomic readers, etc.) without
forcing the consumer's `impl WorkUnit` declaration to carry an
explicit `<'frame>` parameter. Validated by sketch
`mock/research/sketches/202605101036-poolframe-lifetime/`. Dispatch
sites use HRTB: `Wu: for<'a> WorkUnit<Ctx<'a> = SomeCtxFor<'a>>`.

The `Provider<Init = Self>` supertrait carries the registration
contract. WorkUnits register on the SchedulerBuilder via
`.with(MyWu)` (unit struct) or `.with(MyWu::new())` (when the WU
has init data). `Provider::Dispatch` for WorkUnits is
`UnitDispatch<Self>`, prepending the WU type onto the `Wus`
typestate accumulator.

No NAME const. WU identity is the type itself. Naming is a consumer
concern; they can implement their own trait or use wrappers.

Schedule conditions:
- `Always`: unconditional (default)
- `On<V>`: when virtual V fires

Write-write ambiguity resolved by scheduling hint priority across
three dimensions. Tie-break: most deps first, then deterministic
fallback.

WU authoring guideline: ≤8 unique columns per WU (register pressure
guideline, not engine constraint).

## WU Context providers (domains 09, 19)

The WU execution parameter is a ctx-framework provider tuple
(`Self::Ctx` on the `WorkUnit` trait) bound by seven accessor
traits declared via `hilavitkutin-ctx`'s `provider_generic!` /
`provider_generic2!` macros. Each accessor carries the relevant
access-set type parameter so compile-time `Contains<S>` checks
can enforce WU-declared access at the call site.

Provider API traits (access-set-aware via ctx parameterised macros):

```rust
trait ColumnReaderApi<R: AccessSet> {
 unsafe fn read<T: ColumnValue>(&self, i: USize) -> T
 where R: Contains<Column<T>>;
}
trait ColumnWriterApi<W: AccessSet> {
 unsafe fn write<T: ColumnValue>(&self, i: USize, v: T)
 where W: Contains<Column<T>>;
}
trait ResourceProviderApi<R: AccessSet> {
 fn resource<T: 'static>(&self) -> &T
 where R: Contains<Resource<T>>;
}
trait VirtualFirerApi<W: AccessSet> {
 fn fire<V: 'static>(&self)
 where W: Contains<Virtual<V>>;
}
trait EachApi<R: AccessSet, W: AccessSet> {
 fn run<F>(&self, f: F); // per-element, fusible
}
trait BatchApi<R: AccessSet, W: AccessSet> {
 fn run<F>(&self, f: F); // full morsel slice, non-fusible
}
trait ReduceApi<R: AccessSet, W: AccessSet> {
 fn run<A, F>(&self, init: A, f: F) -> A;
}
```

Accessor traits (generated by `provider_generic!` /
`provider_generic2!`):

```rust
provider_generic!(<R: AccessSet> ColumnReaderApi as HasColumnReader => reader);
provider_generic!(<W: AccessSet> ColumnWriterApi as HasColumnWriter => writer);
provider_generic!(<R: AccessSet> ResourceProviderApi as HasResourceProvider => resources);
provider_generic!(<W: AccessSet> VirtualFirerApi as HasVirtualFirer => virtuals);
provider_generic2!(<R: AccessSet, W: AccessSet> EachApi as HasEach => each);
provider_generic2!(<R: AccessSet, W: AccessSet> BatchApi as HasBatch => batch);
provider_generic2!(<R: AccessSet, W: AccessSet> ReduceApi as HasReduce => reduce);
```

`Contains<S>` is a sealed compile-time membership witness on the
cons-list typestate. Accessing a store not declared in `Read` or
`Write` fails to compile at the relevant API-trait method's
where-clause. Implemented as `#[marker]`, so overlapping head and
tail impls coexist:

```rust
// Head: Cons<X, T> contains X.
impl<X: 'static, T: 'static + AccessSet>
  Contains<X> for Cons<X, T> {}

// Tail: Cons<H, T> contains M if T does.
impl<H: 'static, T: 'static + AccessSet, M: 'static>
  Contains<M> for Cons<H, T> where T: Contains<M> {}
```

Membership lookup walks the cons-list at trait-solver time. Depth
scales linearly with list length; round 202605042200 measured 0.53
seconds at depth 5 with 25 WorkUnits across 13 stores (S1b
finding), well within the rustc default recursion limit.

`AccessSet::LEN` reports the cons-list cardinality at compile time.
`Empty::LEN = USize::ZERO`. `Cons<H, T>::LEN = USize(T::LEN.0 + 1)`.
The recursive impl computes correctly at any depth and is callable
in const contexts via the const-arith surface on `arvo::USize`.

`&self` receiver on every provider method is load-bearing: prevents
LLVM noalias metadata that would reorder writes across fused WUs.
Raw-pointer access `unsafe read / write` pinky-swears ownership
discipline that the engine proves at plan time.

## Foundational primitives for Kit (domain 23, )

The api ships the foundational primitives that `Kit` (in
`hilavitkutin-kit`), `SchedulerBuilder` (in the engine), and the
app-level `replace_resource` API rest on. The api crate documents
the contracts because `WorkUnit` authors, `.build()` callers, and
`Kit` authors all reference them at consumer-API positions.

The contract surface is intentionally thin. Round-4's design (topic
`202605042200_topic_kit_trait_split.md`) reduced the Kit-side
contract surface to four trait shapes plus one opt-in marker. The
old `Buildable<Stores>`, `WuSatisfied<A>`, `BuilderExtending<B>`,
and `BuilderResource<T>` traits are gone (deleted entirely per
`no-legacy-shims-pre-1.0.md`); their roles are taken by
`ContainsAll<L>` plus `WorkUnitBundle::AccumRead`/`AccumWrite`
projections plus the new `Kit { type Units; type Owned }` shape.
Layered enforcement framing per
`202605042200_topic_round_4_layered_enforcement.md`: the
typestate proves correctness; visibility lives in Rust language
rules; `Replaceable` is the only enforced annotation.

### `WorkUnitBundle` trait

```rust
pub trait WorkUnitBundle {
 type AccumRead: AccessSet;
 type AccumWrite: AccessSet;
}

impl WorkUnitBundle for Empty {
 type AccumRead = Empty;
 type AccumWrite = Empty;
}

impl<W: WorkUnit, T: WorkUnitBundle> WorkUnitBundle for Cons<W, T>
where
 W::Read: Concat<T::AccumRead>,
 W::Write: Concat<T::AccumWrite>,
 <W::Read as Concat<T::AccumRead>>::Out: AccessSet,
 <W::Write as Concat<T::AccumWrite>>::Out: AccessSet,
{
 type AccumRead = <W::Read as Concat<T::AccumRead>>::Out;
 type AccumWrite = <W::Write as Concat<T::AccumWrite>>::Out;
}
```

A `WorkUnitBundle` is a recursive cons-list of `WorkUnit`s with
projected per-bundle `AccumRead` and `AccumWrite` totals. The two
recursive impls accumulate each WU's `Read` and `Write` sets via
`Concat`. The accumulator carries duplicates (the same store
appearing in multiple WUs' Read sets shows up once per
occurrence); logical dedup via `#[marker] Contains` makes
duplicates correctness-irrelevant. See the audit M2 investigation
at `mock/research/sketches/202605050800_dedup_concat/FINDINGS.md`
for the cost analysis; physical dedup is deferred to round-5+.

### `StoreBundle` trait

```rust
pub trait StoreBundle {}
impl StoreBundle for Empty {}
impl<H, T: StoreBundle> StoreBundle for Cons<H, T> {}
```

A marker trait for the Kit's `Owned` declarations. Two-impl
recursive shape. No projections; the bundle's content is read by
the engine's `KitDispatch<K>` routing during `.with(MyKit)`, not by
the trait itself. `StoreBundle` carries `Resource<T>`, `Column<T>`,
`Virtual<T>`, and `LinkedBin<T: ?Sized>` markers in the cons-list
shape produced by `read!` / `write!` macros from a flat-list syntax.

### `Replaceable` opt-in marker

```rust
pub trait Replaceable {}
```

A marker trait with no methods and no associated types. Per-type
opt-in via `impl Replaceable for Foo {}` declarations from the
kit author. The static `T: Replaceable` bound on
`Scheduler::replace_resource::<T>(value)` (see the engine's
DESIGN) rejects override attempts on non-opt-in types at compile
time, with a high-quality E0277 diagnostic that lists the types
that DO impl `Replaceable`. See sketch S5 at
`mock/research/sketches/202605050615_kit_taxonomy/` for the
empirical evidence.

Replaceability is the only enforced annotation on Owned types.
Visibility is per-kit-crate (whole-kit pub vs whole-kit pub(crate)
in its own crate); not a per-Owned-type axis. See
`202605042200_topic_round_4_layered_enforcement.md` for the full
layered-enforcement framing the doc CL paraphrases.

### `Kit` trait at the api boundary

```rust
pub trait Kit: Provider<Init = Self> {
 type Units: WorkUnitBundle;
 type Owned: StoreBundle;
}
```

The trait is declared in `hilavitkutin-kit`, not in api; the api
documents the contract because consumers (kit authors) and the
engine (`.with(MyKit)` caller) both reference it at their boundary.
`Provider<Init = Self>` is the registration supertrait: Kits reach
the builder via the same `.with` verb as WorkUnits and stores.
`Provider::Dispatch` for Kits is `KitDispatch<Self>`, which concats
`K::Units` into the `Wus` accumulator and `K::Owned` into the
`Stores` accumulator at the type level.
The kit crate's `DESIGN.md` carries the authoring guide and
examples; this section names the shape so neighbouring core
primitives can be read in context.

The trait carries no `Required` member and no `requirements_doc()`
helper. "What a kit requires" lives in prose (the kit author lists
required external Resources in the Kit's rustdoc). The typestate
proves correctness via `ContainsAll<AccumRead>` plus
`ContainsAll<AccumWrite>` at `.build()`; the resulting compile
error names the missing marker on the first inline line of the
diagnostic per the S1b depth-5 sketch outcome.

### `ContainsAll<L>`

```rust
#[marker]
pub trait ContainsAll<L>: AccessSet {}
impl<S: AccessSet> ContainsAll<Empty> for S {}
impl<S: AccessSet, H, T> ContainsAll<Cons<H, T>> for S
where
 S: Contains<H> + ContainsAll<T>,
{
}
```

Replaces the deleted `Buildable<Stores>` and `WuSatisfied<A>`
traits. Two-impl recursive shape: empty list trivially contained;
non-empty list satisfies when the head is contained and the tail
is recursively contained. The `#[marker]` attribute permits the
overlapping resolution shape Contains relies on. Sealed by being
declared in api alongside `Contains`; consumers cannot impl
directly because the only legitimate shape is the recursive walk.

The trait solver's recursion grows with the length of `L`, which
equals the count of distinct Read/Write records across all WUs in
the bundle (with duplicates per the M2 finding). Empirical: a
25-WU bundle at depth 5 with 13 distinct stores produces a
36-element accumulated chain; `.build()` resolves in 0.53s on
nightly rustc per S1b. See
`mock/research/sketches/202605050700_deep_stacking_d5/FINDINGS.md`.

### `read!` / `write!` macros

Consumers declare WU access sets through declarative macros that
convert flat-tuple syntax to the cons-list shape used by
`WorkUnitBundle::AccumRead/Write` accumulation:

```rust
#[macro_export]
macro_rules! read {
 () => { Empty };
 ($T:ty $(,)?) => { Cons<$T, Empty> };
 ($T:ty, $($rest:ty),+ $(,)?) => { Cons<$T, $crate::read!($($rest),+)> };
}
// write! has identical shape.
```

A `WorkUnit` impl writes:

```rust
type Read = read![Resource<Interner>, Column<FileInfo>, Resource<Workspace>];
type Write = write![Column<Diagnostic>];
```

Empty `read![]` yields `Empty`; single-store `read![Resource<X>]`
expands to `Cons<Resource<X>, Empty>`. Both macros live in
`hilavitkutin-api` and are re-exported from the crate root.

### `Depth` trait

```rust
#[allow(private_bounds)]
pub trait Depth: Sealed {
 const D: USize;
}

impl Depth for Empty { const D: USize = USize::ZERO; }
impl<H, R: Depth> Depth for Cons<H, R> {
 const D: USize = R::D + USize::ONE;
}
```

Two impls. No specialization. No marker overlap. Coherence
trivially holds because `Empty` and `Cons<H, R>` are disjoint
shapes.

`Depth` is impl'd ONLY on cons-list shapes. Plan-stage code
consumes engine accumulators (cons-list by construction) and
consumer-declared `Read` / `Write` (cons-list via the `read!` /
`write!` macros), so any value the plan stage counts already
comes from a cons-list source.

`AccessSet::LEN` and `Depth::D` coexist with explicit, distinct
contracts. `LEN` is shape-based (immediate-cell count, always 2
for cons-list cells); `D` is total-element-based (the count after
recursive descent). Both stay documented at their respective trait
declarations.

### `Contains` diagnostic

`Contains<S>` carries `#[diagnostic::on_unimplemented]` so the
unsatisfied-trait-bound error reads as a stable, consumer-friendly
message instead of a bare `Contains` mention:

```text
note: store `{Self}` does not contain `{T}`. Register it with
 `.with(Resource::new(value))`, `.with(Column::<T>::new())`,
 `.with(Virtual::<T>::new())`, or install a Kit that owns it via
 `.with(MyKit)`. If `.build()` reports `overflow evaluating the
 requirement`, declare `#![recursion_limit = "1024"]` at your crate
 root.
```

The attribute is purely a DX win; trait semantics are unchanged.
The annotation lives on the existing `Contains<S>` declaration in
`access.rs`.

### Diagnostic coverage across the public surface

  `ColumnValue`, `Replaceable`.
- Sealed substrate axes: `SchedulingHint`, `UrgencyValue`,

`hilavitkutin_api::prelude` re-exports the consumer-facing

```rust
// Now in scope:
```

`Dispatch` and the four router structs (`UnitDispatch`,

### `.build()` where-clause (engine side)

The engine's `SchedulerBuilder::build` carries:

```rust
where Wus: WorkUnitBundle, Stores: AccessSet + StoreBundle
 + ContainsAll<Wus::AccumRead> + ContainsAll<Wus::AccumWrite>
```

This unfolds via the recursive `ContainsAll` impl into the
conjunction of per-store `Contains<...>` proofs against `Stores`.
The cons-list recursion impl on `Contains` handles arbitrary-depth
`Stores` accumulators. With `#[marker]` on both `Contains` and
`ContainsAll`, the trait solver picks any matching impl and stops;
duplicates in `Wus::AccumRead/Write` extend traversal length but
do not duplicate proof work.

When a consumer registers a WU via `.with(MyWu)` whose `Read`
mentions a store not added via `.with(Resource::new(value))` /
`.with(Column::<T>::new())` / `.with(Virtual::<T>::new())` /
`.with(MyKit)`, rustc reports the missing membership using the
diagnostic message above, naming the missing store by type. The
inline error leads with the `Empty: Contains<...>` failure on the
first line; the verbose `Cons<...>` accumulator chain follows but
does not bury the marker name.

### Recursion limit

The substrate's typestate machinery walks cons-list recursions at
trait-solver time. Default rustc `recursion_limit = 128`
accommodates apps up to roughly 30 WUs by 30 stores. The
`hilavitkutin-api`, `hilavitkutin`, and `hilavitkutin-kit` crates
declare `#![recursion_limit = "512"]` internally for their own
machinery. Consumer crates approaching the threshold (4+ nested
Kit layers, 30+ flat WUs across the build) declare:

```rust
#![recursion_limit = "1024"]
```

at their own crate root.

Earlier substrate designs proposed shipping a
`recursion_limit_for_kits!()` macro that expanded to the inner
attribute. Current rustc rejects macro-expansion of crate-level
inner attributes ("an inner attribute is not permitted in this
context"); the design retreats to documenting the directive
plainly. The trait diagnostics on `Contains`, `ContainsAll`,
`Kit`, and `WorkUnit` mention the recursion-limit directive in
their notes when the bound failure could plausibly stem from
depth.

When `.build()` reports `overflow evaluating the requirement`, the
fix is the inner attribute above; no other configuration is
needed.

### Marker-trait keystone risk

`Contains` and `ContainsAll` rely on `feature(marker_trait_attr)`
(rust-lang/rust#29864), an unstable feature without an active
stabilisation track. The crate accepts this risk explicitly.
A future rustc that removes or restricts marker-trait coherence
would break the recursive cons-list AccessSet pattern; the planned
response is a bitset-backed AccessSet over a per-app StoreIndex
ConstParamTy, sketched in
`mock/research/marker_trait_attr_bitset_fallback.md`. The fallback
is not built in round-4. Other nightly features the crate
depends on (`const_trait_impl`, `adt_const_params`,
`generic_const_exprs`) have differing stabilisation profiles;
analogous response notes can be written if and when their
role becomes load-bearing.

### Why these live in api, not in kit

`WorkUnitBundle`, `StoreBundle`, `ContainsAll`, `Replaceable`, and
`Depth` are contract-shape artefacts. `WorkUnitBundle` and
`StoreBundle` are referenced in `Kit` associated-type bounds.
`ContainsAll` is referenced at the engine's `.build()` where-clause
and in any consumer-side trait bound that proves a particular
access-set state. `Replaceable` is referenced at the
`replace_resource` call site. `Depth` is referenced by plan-stage
code that walks accumulators. They belong with `AccessSet` and
`Contains` in api. The `Kit` trait in `hilavitkutin-kit` is purely
the consumer-facing declarative shape; the proofs above sit a
layer below it. The `Kit: Provider<Init = Self>` supertrait is also
declared in this api crate alongside `Provider` itself; the kit
crate inherits the registration shape rather than defining its own.

## Scheduling hints (domain 09)

Three axes. Each axis is a sealed trait with a fixed set of ZST
carrier types. Each trait's `VALUE` const is typed as the narrowest
arvo `UFixed<N, 0, Hot>` alias that fits the axis.

| Axis | Alias | Width | Markers | Discriminants |
|------|-------|------:|---------|---------------|
| Urgency | `Urgency = UFixed<2, 0, Hot>` | 2 bits | `Immediate`, `Steady`, `Relaxed`, `Deferred` | 3, 2, 1, 0 |
| Divisibility | `Divisibility = UFixed<2, 0, Hot>` | 2 bits | `Atomic`, `Adaptive`, `Interruptible` | 2, 1, 0 |
| Significance | `Significance = UFixed<3, 0, Hot>` | 3 bits | `Critical`, `Important`, `Normal`, `Opportunistic`, `Optional` | 4, 3, 2, 1, 0 |

`SchedulingHint` trait on tuples: `(U: UrgencyValue, D:
DivisibilityValue, S: SignificanceValue)`. Higher discriminant =
higher priority (ordering derived from `UFixed`'s `PartialOrd`).

Three axes pack into 7 bits; u8-discriminant thinking would take
24 bits. Exact-width primitives are how the stack pays this memory
back at every WU.

## Platform traits (domain 02)

```rust

 unsafe fn allocate(&self, len: usize, align: usize) -> *mut u8;
 unsafe fn deallocate(&self, ptr: *mut u8, len: usize);
 unsafe fn protect(&self, ptr: *mut u8, len: usize,
 read: bool, write: bool);
}

pub trait ThreadPool: Send + Sync {
 fn spawn(&self, f: impl FnOnce() + Send + 'static);
 fn worker_count(&self) -> usize;
}

pub trait Clock: Send + Sync {

}
```

All consumed via monomorphisation. The Scheduler is generic over the

ride the hilavitkutin sweep (#72); this round intentionally does not
retune them.

## Runtime contracts (round 202605101036)

Locked surface from the runtime megaround. These types ship in
`hilavitkutin-api` so consumers can reference them in their impls; the
engine crate provides the default implementations.

### `Executor: Sealed`

Open trait, sealed via private supertrait. Default impl is
`HybridExecutor` in the engine crate. `Scheduler<'frame, Cfg, E: Executor = HybridExecutor>` propagates the type so consumers can swap to a custom Executor (typically only for advanced custom parking or work-stealing impls).

The sealed bound enforces the Topic 3 S7 atomic-ordering protocol:
non-substrate Executor impls cannot violate the cross-thread happens-
before contract.

```rust
pub trait Executor: private::Sealed {
    fn run<const MAX_CORES: usize, const MAX_PHASES: usize>(
        &self,
        frame: Pin<&PoolFrame<MAX_CORES, MAX_PHASES>>,
        core_id: USize,
    );
    // Plus per-phase-barrier hooks and parking primitive accessors;
    // concrete shape drafted in the src CL.
}
```

`Executor::try_steal` is BACKLOG behind a `work-stealing` feature
flag for v2. Work-stealing is not in v1.

### `PlanAffecting: BuilderInput + Sealed`

Sealed marker trait. Resources implementing `PlanAffecting` dirty the

without it are cheap value-swaps. Default is NOT plan-affecting.


keyed by `PlanAffectingId`. Default cap `MAX_PLAN_AFFECTING_RESOURCES = 16`;
consumers override via the Scheduler const generic.

Engine-internal state with cross-thread lifetime requirements (PoolFrame,

through const-known `'frame` and `'phase` lifetimes. The Topic 6 axis C
sketch validates that these lifetimes stay contained inside the engine
boundary: consumer `impl WorkUnit` blocks remain `<>`-clean via the
`Ctx<'frame>` GAT pattern; consumer `impl PlanAffecting` declarations
similarly don't carry explicit lifetime parameters.

## Resource model (domain 07, R5)

```rust
pub struct Field<T: ColumnValue>(PhantomData<T>); // scalar ≤16 bytes
pub struct Seq<T, const N: usize>(PhantomData<T>); // const-sized array
pub struct Map<K, V, const N: usize>(PhantomData<(K, V)>); // const-sized map
```

No dynamic collections. Allocation size is const generic. Field<T>
is stack-local cached (LLVM promotes to registers). Seq/Map are
arena-allocated. Write resource collections count toward L1 morsel
budget; read-only ride L2 prefetcher.

Resource data is behind pointer indirection to external slab storage.
NOT inline in Context. Writing inline data through `&self` is UB.

`Seq<T, N>` and `Map<K, V, N>` are type-level descriptors, not
containers with a `push` method. Consumers receive items into them
via `Push<T>` (or `BoundedPush<T>` for overflow-aware pushes); see
the contract surface below. The scheduler owns the arena; these
types name the slot, not the storage.

## Virtual flag system (domain 10)

Bit-packed fire flags using arvo-bitmask types. One bit per registered
Virtual type. Packed into arvo words, grouped into Mask64/Mask256.
Hierarchical check: word == 0 → skip. Affinity-ordered bit assignment
at plan time. Epoch-based reset (increment, not memset).

Per-(virtual, consumer) bit tracking. Clear-on-dispatch. Re-fire
during execution → consumer runs again next pass. No lost fires.

## Consumer contract surface (capability / sink / codec)

The modules above cover pipeline execution. A parallel surface covers
the OTHER boundary: API positions that receive or emit items outside
WU execution (a parser's diagnostic stream, a codegen target's byte
output, a serialiser's encoded bytes). That surface lives in three
modules: `capability`, `sink`, `codec`.

Framing: every position that accepts items is a trait bound, not a
concrete container. The trait is the contract; the caller brings the
type. `Vec<T>` never crosses this boundary; neither does `Seq<T, N>`.

### `capability`: the atoms

```rust
pub trait Push<T> { fn push(&mut self, item: T); }
pub trait BulkPush<T>: Push<T> {
 fn push_bulk(&mut self, items: &[T]) where T: Copy {
 for i in items { self.push(*i); }
 }
}
pub trait Len {
 fn len(&self) -> USize;
 fn is_empty(&self) -> Bool { /* default via len */ }
}
pub trait Capacity {
 fn capacity(&self) -> USize;
 fn remaining(&self) -> USize;
}
pub trait BoundedPush<T>: Push<T> + Capacity {
 fn try_push(&mut self, item: T) -> Outcome<(), Full>;
}
pub struct Full;
```

`Push<T>` is the atom: receive one item. `BulkPush<T>` adds slice
push (with a loop default implementors may override for memcpy-style
fastpaths). `Len` / `Capacity` are introspection atoms. `BoundedPush<T>`
is overflow-aware; `try_push` returns `Outcome::Err(Full)` on refusal
and MUST NOT silently drop items.

### `sink`: named composites + combinators

Named traits are bounds over capabilities; blanket impls make them
shorthand:

```rust
pub trait Collector<T>: Push<T> {}
impl<T, S: Push<T> + ?Sized> Collector<T> for S {}

pub trait DiagnosticSink<E>: Push<E> + Len {}
impl<E, S: Push<E> + Len + ?Sized> DiagnosticSink<E> for S {}

pub trait ByteEmitter: Push<u8> + BulkPush<u8> {}
impl<S: Push<u8> + BulkPush<u8> + ?Sized> ByteEmitter for S {}
```

Combinators:

- `NullSink`: unit struct; implements `Push<T>` + `BulkPush<T>` for
 every `T` as no-ops. For tests and dry-run counting.
- `CountingSink<T>`: counts pushes, stores nothing; implements
 `Push<T>` + `Len`. For "how many diagnostics were emitted".
- `TeeSink<'a, A, B>`: fans each push to two downstream sinks.
 Requires `T: Clone` (pushes distribute across both).

Tuple blanket impls ship for arity 2 and 3 so consumers can pass
`(&mut diagnostics, &mut bytes)` satisfying disjoint `Push<T>` and
`Push<u8>` slots via destructured field access. Higher arities land
when a concrete consumer surfaces a need.

### `codec`: encode / decode

Primary form is stateful incremental:

```rust
pub trait Encoder<T> {
 fn feed<B: ByteEmitter>(&mut self, v: &T, out: &mut B);
 fn finish<B: ByteEmitter>(self, out: &mut B);
}

pub trait Decoder<T> {
 fn feed<'a, S: Push<T>>(&mut self, chunk: &'a [u8], out: &mut S)
 -> Outcome<&'a [u8], DecodeError>;
 fn finish(self) -> Outcome<(), DecodeError>;
}

#[non_exhaustive]
pub enum DecodeError { Truncated, Invalid, OverLength }
```

Oneshot adapters as extension traits with blanket impls:

```rust
pub trait EncoderExt<T>: Encoder<T> + Sized {
 fn encode_one<B: ByteEmitter>(self, v: &T, out: &mut B);
}
pub trait DecoderExt<T>: Decoder<T> + Sized {
 fn decode_all<S: Push<T>>(self, bytes: &[u8], out: &mut S)
 -> Outcome<(), DecodeError>;
}
```

Streaming consumers (chunked bytecode loading, paradox save parsing)
call `feed`/`finish` directly. All-at-once consumers use the `Ext`
methods. One trait pair serves both shapes.

### Example: reshape a Vec-return

```rust
// Before the sweep:
fn parse(tokens: &[Token]) -> Result<Ast, Vec<SyntaxError>> {... }

// After the sweep (contracts from this round):
fn parse<E>(tokens: &[Token], errors: &mut E) -> Outcome<Ast, ParseAborted>
where
 E: DiagnosticSink<SyntaxError>;
```

Call sites supply whatever concrete sink they want (a `Vec` under a
std feature in tests, a `CountingSink` in "just count" mode, a
bounded on-stack arena in production). The parser stays trait-generic.

### What is NOT here

- Blanket impls for `Vec<T>` / `Seq<T, N>`. These live in test-helper
 crates (#81) under an optional `std` or `seq` feature.
- Specific codec formats (JSON, clause bytecode, paradox save). Each
 format ships in its owner crate; this crate defines the contract.
- `Flush` / `Fuse` / back-pressure semantics for buffered byte
 emitters. Deferred; see BACKLOG.

- Engine internals (scheduling, dispatch codegen, phase analysis)
- Platform implementations (mmap, pthread; those are in hilavitkutin)

- Concrete codec format implementations (consumer-owned; this crate
