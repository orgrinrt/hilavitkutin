# hilavitkutin-api — Design

Consumer-facing types, traits, and platform contracts. Consumers
(polka-dots, saalis, loimu) depend on this crate to define WorkUnits.
The engine crate (hilavitkutin) also depends on it. `#![no_std]`,
no alloc. All index, count, and size types use arvo newtypes.

Depends on `hilavitkutin-ctx` for its `provider_generic!` /
`provider_generic2!` macros and the `Context<P>` provider-tuple
wrapper. WU execute parameter types and the WU-context accessor
traits are expressed via those macros; see "WU Context providers"
below.

## Store types (domain 06)

Three store types, one scheduling mechanism:

```rust
pub struct Resource<T>(PhantomData<T>);  // singleton, one value
pub struct Column<T>(PhantomData<T>);    // collection, morsel-chunked
pub struct Virtual<T>(PhantomData<T>);   // zero-data, DAG edge only
```

The scheduler sees StoreId pairs for edge construction. Does not care
about cardinality. `Write<Resource<X>> → Read<Resource<X>>` builds
the same edge as `Write<Column<Y>> → Read<Column<Y>>`.

`StoreId` — dense index assigned at plan time. Wraps an arvo
fixed-point newtype (not raw u16). `AccessMask` — arvo-bitmask
`Mask64` (≤64 stores) or `Mask256` (≤256 stores). All scheduler ops
= single-instruction bitwise.

`AccessSet` — trait on tuples. `(Column<PosX>, Column<PosY>,
Resource<Gravity>)` implements AccessSet. Compile-time store
descriptors. Plan phase converts to runtime AccessMask via
monomorphised generic functions (the monomorphised function IS the
type identity — no TypeId, no std::any).

ZST erasure: column marker types constrain the DAG, enforce type
safety, and determine morsel sizes at compile time. Erased to plain
indices by execution time.

## ColumnValue trait (domain 05, R4)

```rust
pub trait ColumnValue: Copy + 'static {
    const BIT_WIDTH: usize;  // no default on trait itself
}

// Blanket impl via min_specialization
impl<T: Copy + 'static> ColumnValue for T {
    default const BIT_WIDTH: usize = core::mem::size_of::<Self>() * 8;
}
```

Any `Copy + 'static` type ≤16 bytes is column-storable. Zero friction
for consumers. BIT_WIDTH informs the storage engine for bitpacking.
Sub-byte arvo types specialise (e.g. `UFixed<1,0>` → BIT_WIDTH=1).
No pack/unpack — type IS the storage (R3, R4).

## WorkUnit trait (domain 09)

```rust
pub trait WorkUnit<Schedule = Always>: Send + Sync + 'static {
    type Read: AccessSet;
    type Write: AccessSet;
    type Hint: SchedulingHint = (Immediate, Atomic, Normal);
    type Ctx: HasColumnReader<Self::Read>
            + HasColumnWriter<Self::Write>
            + HasResourceProvider<Self::Read>
            + HasVirtualFirer<Self::Write>
            + HasEach<Self::Read, Self::Write>
            + HasBatch<Self::Read, Self::Write>
            + HasReduce<Self::Read, Self::Write>;
    const COMMUTATIVE: bool = false;
    fn execute(&self, ctx: &Self::Ctx);
}
```

No NAME const. WU identity is the type itself. Naming is a consumer
concern — they can implement their own trait or use wrappers.

Schedule conditions:
- `Always` — unconditional (default)
- `On<V>` — when virtual V fires

Write-write ambiguity resolved by scheduling hint priority across
three dimensions. Tie-break: most deps first, then deterministic
fallback.

WU authoring guideline: ≤8 unique columns per WU (register pressure
guideline, not engine constraint).

## WU Context providers (domains 09, 19)

The WU execution parameter is a ctx-framework provider tuple —
`Self::Ctx` on the `WorkUnit` trait — bound by seven accessor
traits declared via `hilavitkutin-ctx`'s `provider_generic!` /
`provider_generic2!` macros. Each accessor carries the relevant
access-set type parameter so compile-time `Contains<S>` checks
can enforce WU-declared access at the call site.

Provider API traits (access-set-aware via ctx parameterised macros):

```rust
trait ColumnReaderApi<R: AccessSet> {
    unsafe fn read<T: ColumnValue>(&self, i: USize) -> T
    where R: Contains<Column<T>>;
}
trait ColumnWriterApi<W: AccessSet> {
    unsafe fn write<T: ColumnValue>(&self, i: USize, v: T)
    where W: Contains<Column<T>>;
}
trait ResourceProviderApi<R: AccessSet> {
    fn resource<T: 'static>(&self) -> &T
    where R: Contains<Resource<T>>;
}
trait VirtualFirerApi<W: AccessSet> {
    fn fire<V: 'static>(&self)
    where W: Contains<Virtual<V>>;
}
trait EachApi<R: AccessSet, W: AccessSet> {
    fn run<F>(&self, f: F);   // per-element, fusible
}
trait BatchApi<R: AccessSet, W: AccessSet> {
    fn run<F>(&self, f: F);   // full morsel slice, non-fusible
}
trait ReduceApi<R: AccessSet, W: AccessSet> {
    fn run<A, F>(&self, init: A, f: F) -> A;
}
```

Accessor traits (generated by `provider_generic!` /
`provider_generic2!`):

```rust
provider_generic!(<R: AccessSet> ColumnReaderApi as HasColumnReader => reader);
provider_generic!(<W: AccessSet> ColumnWriterApi as HasColumnWriter => writer);
provider_generic!(<R: AccessSet> ResourceProviderApi as HasResourceProvider => resources);
provider_generic!(<W: AccessSet> VirtualFirerApi as HasVirtualFirer => virtuals);
provider_generic2!(<R: AccessSet, W: AccessSet> EachApi as HasEach => each);
provider_generic2!(<R: AccessSet, W: AccessSet> BatchApi as HasBatch => batch);
provider_generic2!(<R: AccessSet, W: AccessSet> ReduceApi as HasReduce => reduce);
```

`Contains<S>` is a sealed compile-time membership trait on tuples.
Accessing a store not declared in Read / Write fails to compile at
the relevant API-trait method's where-clause. Implemented as
`#[marker]`, so overlapping impls are permitted.

Two impl families are shipped:

1. **Flat-tuple impls.** One `Contains<Tᵢ>` impl per position per
   arity, for arities 0..=12. These cover `WorkUnit::Read` and
   `WorkUnit::Write` declarations, which are flat tuples at the
   consumer level.
2. **Cons-list recursion impl.** One additional impl that
   propagates membership down arbitrarily-deep cons-list-shaped
   tuples:

   ```rust
   impl<H: 'static, R: 'static, T: 'static> Contains<T> for (H, R)
   where R: Contains<T> {}
   ```

   This impl exists for the scheduler builder's `Stores`
   accumulator, which grows as `(H, (H, (H, ())))` per
   registration. Combined with the arity-2 head match
   (`Contains<T0> for (T0, T1)`), membership chains correctly down
   any depth. Permitted because `Contains` is `#[marker]`.

`&self` receiver on every provider method is load-bearing: prevents
LLVM noalias metadata that would reorder writes across fused WUs.
Raw-pointer access `unsafe read / write` pinky-swears ownership
discipline that the engine proves at plan time.

## Builder support (domain 23, rounds 202605010900 + 202605011500)

Several sealed contracts ship in api so the engine's
`SchedulerBuilder<MAX_*, Wus, Stores>` can carry where-clauses on
`.build()` and on `add_kit`. These contracts live here, not in the
engine, because `WorkUnit` authors, `.build()` callers, and `Kit`
authors all reference them at consumer-API positions.

### `Buildable<Stores>` sealed trait

```rust
pub trait Buildable<Stores: AccessSet>: Sealed {}

impl<Stores: AccessSet> Buildable<Stores> for () {}

impl<H, R, Stores> Buildable<Stores> for (H, R)
where
    H: WorkUnit,
    R: Buildable<Stores>,
    Stores: AccessSet
         + WuSatisfied<H::Read>
         + WuSatisfied<H::Write>,
{}
```

Two impls only: a base case for `()` (empty Wus accumulator) and a
recursive case for the cons-list cell `(H, R)`. The trait solver
takes N steps for N registered WUs; each step adds two
`WuSatisfied` obligations. Linear, unbounded depth. No per-arity
cap. Coherence trivially holds because `()` and `(H, R)` are
disjoint shapes. Sealed via private supertrait; consumers cannot
impl directly.

The default `recursion_limit = 128` accommodates apps up to
roughly 30 WUs by 30 stores; larger workloads need
`#![recursion_limit = "512"]` at the consumer crate root. See the
"Recursion limit" subsection below.

### `WuSatisfied<A>` sealed trait

```rust
pub trait WuSatisfied<A: AccessSet>: Sealed<A> {}

impl<S: AccessSet> WuSatisfied<()> for S {}

impl<S, H: 'static, R> WuSatisfied<(H, R)> for S
where
    S: Contains<H> + AccessSet + WuSatisfied<R>,
    R: AccessSet,
{}
```

Two impls only. `A` is a cons-list shape produced by the `read!`
and `write!` macros from a consumer's flat-tuple syntax (see the
next subsection). Recursion proves `Contains<Tᵢ>` for every `Tᵢ`
in `A` against `Self`. No per-arity cap. Sealed.

### `read!` / `write!` macros

Consumers declare WU access sets through declarative macros that
convert flat-tuple syntax to the cons-list shape `WuSatisfied`
expects:

```rust
#[macro_export]
macro_rules! read {
    () => { () };
    ($T:ty $(,)?) => { ($T, ()) };
    ($T:ty, $($rest:ty),+ $(,)?) => { ($T, $crate::read!($($rest),+)) };
}
// write! has identical shape.
```

A `WorkUnit` impl writes:

```rust
type Read = read![Resource<Interner>, Column<FileInfo>, Resource<Workspace>];
type Write = write![Column<Diagnostic>];
```

Empty `read![]` yields `()`; single-store `read![Resource<X>]`
works identically to multi-store. Both macros live in
`hilavitkutin-api` and are re-exported from the crate root.

### `BuilderExtending<B>` sealed trait

```rust
#[allow(private_bounds)]
pub trait BuilderExtending<B>: Sealed<B> {}

impl<Wus, Stores, NewStores>
    BuilderExtending<SchedulerBuilder<MAX_*, Wus, Stores>>
    for SchedulerBuilder<MAX_*, Wus, NewStores>
where
    Wus: AccessSet,
    Stores: AccessSet,
    NewStores: AccessSet + WuSatisfied<Stores>,
{}
```

Constrains `Kit::Output` so that a Kit cannot wipe prior
registrations. `Wus` is identical between input and output (Kits
never add WUs, only stores; consumers register WUs with `.add::<W>()`
themselves). `NewStores: WuSatisfied<Stores>` proves every store
from the input `Stores` is still present in `NewStores`. The
engine's `add_kit` carries `K::Output: BuilderExtending<Self>` as a
where-clause; a buggy Kit returning `SchedulerBuilder<..., (), ()>`
fails the bound at the call site with a clean compile error pointed
at the Kit's `Output` declaration, not at a downstream `.build()`.

Sealed via private supertrait `Sealed<B>`; only the engine's own
`SchedulerBuilder` impl is permitted.

### `BuilderResource<T>` sealed trait

```rust
#[allow(private_bounds)]
pub trait BuilderResource<T: 'static>: Sealed<T> {
    type WithResource;
    fn with_resource(self, init: T) -> Self::WithResource;
}

impl<MAX_UNITS, MAX_STORES, MAX_LANES, Wus, Stores, T>
    BuilderResource<T>
    for SchedulerBuilder<MAX_UNITS, MAX_STORES, MAX_LANES, Wus, Stores>
where
    T: 'static,
    Wus: AccessSet,
    Stores: AccessSet,
    (Resource<T>, Stores): AccessSet,
{
    type WithResource =
        SchedulerBuilder<MAX_UNITS, MAX_STORES, MAX_LANES, Wus, (Resource<T>, Stores)>;

    fn with_resource(self, init: T) -> Self::WithResource {
        self.resource(init)
    }
}
```

Lets a Kit declared in a crate that cannot import the engine call
`builder.with_resource(value)` against any builder type implementing
the trait. The single legal impl is the engine's, forwarding to the
existing inherent `SchedulerBuilder::resource(init)`. The associated
type `WithResource` carries the same type-state extension the
inherent method produces.

`hilavitkutin-providers`'s `InternerKit` is the first caller. Its
Kit impl is generic over `B: BuilderResource<T>` for the relevant
`T`, so the impl never names `SchedulerBuilder` directly. The
forbidden-imports rule on `hilavitkutin-providers` (which blocks
`hilavitkutin::*`) stays intact.

Sealed via private supertrait `Sealed<T>`; only the engine's
`SchedulerBuilder` impl is permitted. A future round adds symmetric
`BuilderColumn<T>` and `BuilderVirtual<T>` traits when consumer Kits
need to register columns or virtuals; v0.1 ships
`BuilderResource<T>` only because the InternerKit registers a
single resource.

### `Depth` trait

```rust
#[allow(private_bounds)]
pub trait Depth: Sealed {
    const D: USize;
}

impl Depth for () { const D: USize = USize::ZERO; }
impl<H, R: Depth> Depth for (H, R) {
    const D: USize = R::D + USize::ONE;
}
```

Two impls. No specialization. No marker overlap. Coherence
trivially holds because `()` and `(H, R)` are disjoint shapes.

`Depth` is impl'd ONLY on cons-list shapes. Flat tuples of arity
3 or more deliberately do not impl `Depth`. Plan-stage code
consumes engine accumulators (cons-list by construction) and
consumer-declared `Read` / `Write` (cons-list via the `read!` /
`write!` macros), so any value the plan stage counts already comes
from a cons-list source. If a future code path wants `Depth` on a
flat tuple, that is the wrong abstraction and the call site should
refactor to cons-list.

`AccessSet::LEN` and `Depth::D` coexist with explicit, distinct
contracts. `LEN` is shape-based (immediate-tuple-arity, always 2
for cons-list cells); `D` is total-element-based (the count after
recursive descent). Both stay documented at their respective trait
declarations.

### `Contains` diagnostic

`Contains<S>` carries `#[diagnostic::on_unimplemented]` so the
unsatisfied-trait-bound error reads as a stable, consumer-friendly
message instead of a bare `Contains` mention:

```text
note: store `{Self}` does not contain `{T}`. Register it with
      `.resource::<T>(initial)`, `.column::<T>()`, `.add_virtual::<T>()`,
      or install a Kit that registers it.
```

The attribute is purely a DX win; trait semantics are unchanged.
The annotation lives on the existing `Contains<S>` declaration in
`access.rs`.

### `.build()` where-clause (engine side)

The engine's `SchedulerBuilder::build` carries:

```rust
where Wus: Buildable<Stores>, Stores: AccessSet
```

This unfolds via the recursive `Buildable` impl into the
conjunction of per-WU `WuSatisfied<...>` proofs, which unfold via
the recursive `WuSatisfied` impl into per-store `Contains<...>`
proofs against `Stores`. The cons-list recursion impl on `Contains`
handles arbitrary-depth `Stores` accumulators.

When a consumer registers a WU whose `Read` mentions a store not
in `Stores`, rustc reports the missing membership using the
diagnostic message above, naming the missing store by type. The
consumer reads "I forgot to register `Resource<Interner>`" and
adds the appropriate `.resource(Interner)`, `.column::<Interner>()`,
`.add_virtual::<Interner>()`, or `.add_kit(InternerKit)` call.

### Recursion limit

If a consumer hits `overflow evaluating the requirement` during
`.build()`, declare `#![recursion_limit = "512"]` (or higher) at
the consumer crate root. The default 128 accommodates apps up to
roughly 30 WUs by 30 stores; larger workloads need explicit
lifting. The `hilavitkutin-api`, `hilavitkutin`, and
`hilavitkutin-kit` crates already declare 512 internally for their
own machinery.

### Why these live in api, not in kit

`Buildable`, `WuSatisfied`, `BuilderExtending`, and `Depth` are
contract-shape artefacts. The first three are referenced in
`WorkUnit::Ctx` bound conjunctions, at `.build()` call sites, and
in the engine's `add_kit` where-clause. `Depth` is referenced by
plan-stage code that walks the same accumulators. They belong with
`AccessSet` and `Contains` in api. The `Kit` trait in
`hilavitkutin-kit` is purely about composing onto whatever the
builder is; the proofs above sit a layer below it.

## Scheduling hints (domain 09)

Three axes. Each axis is a sealed trait with a fixed set of ZST
carrier types. Each trait's `VALUE` const is typed as the narrowest
arvo `UFixed<N, 0, Hot>` alias that fits the axis.

| Axis | Alias | Width | Markers | Discriminants |
|------|-------|------:|---------|---------------|
| Urgency | `Urgency = UFixed<2, 0, Hot>` | 2 bits | `Immediate`, `Steady`, `Relaxed`, `Deferred` | 3, 2, 1, 0 |
| Divisibility | `Divisibility = UFixed<2, 0, Hot>` | 2 bits | `Atomic`, `Adaptive`, `Interruptible` | 2, 1, 0 |
| Significance | `Significance = UFixed<3, 0, Hot>` | 3 bits | `Critical`, `Important`, `Normal`, `Opportunistic`, `Optional` | 4, 3, 2, 1, 0 |

`SchedulingHint` trait on tuples: `(U: UrgencyValue, D:
DivisibilityValue, S: SignificanceValue)`. Higher discriminant =
higher priority (ordering derived from `UFixed`'s `PartialOrd`).

Three axes pack into 7 bits; u8-discriminant thinking would take
24 bits. Exact-width substrate is how the stack pays this memory
back at every WU.

## Platform traits (domain 02)

```rust
pub trait MemoryProvider: Send + Sync {
    unsafe fn allocate(&self, len: usize, align: usize) -> *mut u8;
    unsafe fn deallocate(&self, ptr: *mut u8, len: usize);
    unsafe fn protect(&self, ptr: *mut u8, len: usize,
                      read: bool, write: bool);
}

pub trait ThreadPool: Send + Sync {
    fn spawn(&self, f: impl FnOnce() + Send + 'static);
    fn worker_count(&self) -> usize;
}

pub trait Clock: Send + Sync {
    fn now_ns(&self) -> u64;
}
```

All consumed via monomorphisation. The Scheduler is generic over the
platform types. Storing `dyn MemoryProvider` would violate the dyn
ban. Each tier (no_os, os, std) ships defaults. Consumer overrides
via scheduler builder.

Note: `len`, `align`, `worker_count`, `now_ns` parameters and return
types should use arvo newtypes in the final API (e.g. `ByteCount`,
`Alignment`, `CoreCount`, `Nanoseconds`). Exact arvo type mappings
ride the hilavitkutin sweep (#72); this round intentionally does not
retune them.

## Resource model (domain 07, R5)

```rust
pub struct Field<T: ColumnValue>(PhantomData<T>);  // scalar ≤16 bytes
pub struct Seq<T, const N: usize>(PhantomData<T>);  // const-sized array
pub struct Map<K, V, const N: usize>(PhantomData<(K, V)>);  // const-sized map
```

No dynamic collections. Allocation size is const generic. Field<T>
is stack-local cached (LLVM promotes to registers). Seq/Map are
arena-allocated. Write resource collections count toward L1 morsel
budget; read-only ride L2 prefetcher.

Resource data is behind pointer indirection to external slab storage.
NOT inline in Context. Writing inline data through `&self` is UB.

`Seq<T, N>` and `Map<K, V, N>` are type-level descriptors, not
containers with a `push` method. Consumers receive items into them
via `Push<T>` (or `BoundedPush<T>` for overflow-aware pushes) —
see the contract surface below. The scheduler owns the arena; these
types name the slot, not the storage.

## Virtual flag system (domain 10)

Bit-packed fire flags using arvo-bitmask types. One bit per registered
Virtual type. Packed into arvo words, grouped into Mask64/Mask256.
Hierarchical check: word == 0 → skip. Affinity-ordered bit assignment
at plan time. Epoch-based reset (increment, not memset).

Per-(virtual, consumer) bit tracking. Clear-on-dispatch. Re-fire
during execution → consumer runs again next pass. No lost fires.

## Consumer contract surface (capability / sink / codec)

The modules above cover pipeline execution. A parallel surface covers
the OTHER boundary — API positions that receive or emit items outside
WU execution (a parser's diagnostic stream, a codegen target's byte
output, a serialiser's encoded bytes). That surface lives in three
modules: `capability`, `sink`, `codec`.

Framing: every position that accepts items is a trait bound, not a
concrete container. The trait is the contract; the caller brings the
type. `Vec<T>` never crosses this boundary; neither does `Seq<T, N>`.

### `capability` — the atoms

```rust
pub trait Push<T>      { fn push(&mut self, item: T); }
pub trait BulkPush<T>: Push<T> {
    fn push_bulk(&mut self, items: &[T]) where T: Copy {
        for i in items { self.push(*i); }
    }
}
pub trait Len {
    fn len(&self) -> USize;
    fn is_empty(&self) -> Bool { /* default via len */ }
}
pub trait Capacity {
    fn capacity(&self) -> USize;
    fn remaining(&self) -> USize;
}
pub trait BoundedPush<T>: Push<T> + Capacity {
    fn try_push(&mut self, item: T) -> Outcome<(), Full>;
}
pub struct Full;
```

`Push<T>` is the atom: receive one item. `BulkPush<T>` adds slice
push (with a loop default implementors may override for memcpy-style
fastpaths). `Len` / `Capacity` are introspection atoms. `BoundedPush<T>`
is overflow-aware; `try_push` returns `Outcome::Err(Full)` on refusal
and MUST NOT silently drop items.

### `sink` — named composites + combinators

Named traits are bounds over capabilities; blanket impls make them
shorthand:

```rust
pub trait Collector<T>: Push<T> {}
impl<T, S: Push<T> + ?Sized> Collector<T> for S {}

pub trait DiagnosticSink<E>: Push<E> + Len {}
impl<E, S: Push<E> + Len + ?Sized> DiagnosticSink<E> for S {}

pub trait ByteEmitter: Push<u8> + BulkPush<u8> {}
impl<S: Push<u8> + BulkPush<u8> + ?Sized> ByteEmitter for S {}
```

Combinators:

- `NullSink` — unit struct; implements `Push<T>` + `BulkPush<T>` for
  every `T` as no-ops. For tests and dry-run counting.
- `CountingSink<T>` — counts pushes, stores nothing; implements
  `Push<T>` + `Len`. For "how many diagnostics were emitted".
- `TeeSink<'a, A, B>` — fans each push to two downstream sinks.
  Requires `T: Clone` (pushes distribute across both).

Tuple blanket impls ship for arity 2 and 3 so consumers can pass
`(&mut diagnostics, &mut bytes)` satisfying disjoint `Push<T>` and
`Push<u8>` slots via destructured field access. Higher arities land
when a concrete consumer surfaces a need.

### `codec` — encode / decode

Primary form is stateful incremental:

```rust
pub trait Encoder<T> {
    fn feed<B: ByteEmitter>(&mut self, v: &T, out: &mut B);
    fn finish<B: ByteEmitter>(self, out: &mut B);
}

pub trait Decoder<T> {
    fn feed<'a, S: Push<T>>(&mut self, chunk: &'a [u8], out: &mut S)
        -> Outcome<&'a [u8], DecodeError>;
    fn finish(self) -> Outcome<(), DecodeError>;
}

#[non_exhaustive]
pub enum DecodeError { Truncated, Invalid, OverLength }
```

Oneshot adapters as extension traits with blanket impls:

```rust
pub trait EncoderExt<T>: Encoder<T> + Sized {
    fn encode_one<B: ByteEmitter>(self, v: &T, out: &mut B);
}
pub trait DecoderExt<T>: Decoder<T> + Sized {
    fn decode_all<S: Push<T>>(self, bytes: &[u8], out: &mut S)
        -> Outcome<(), DecodeError>;
}
```

Streaming consumers (chunked bytecode loading, paradox save parsing)
call `feed`/`finish` directly. All-at-once consumers use the `Ext`
methods — one trait pair serves both shapes.

### Example — reshape a Vec-return

```rust
// Before the sweep:
fn parse(tokens: &[Token]) -> Result<Ast, Vec<SyntaxError>> { ... }

// After the sweep (contracts from this round):
fn parse<E>(tokens: &[Token], errors: &mut E) -> Outcome<Ast, ParseAborted>
where
    E: DiagnosticSink<SyntaxError>;
```

Call sites supply whatever concrete sink they want (a `Vec` under a
std feature in tests, a `CountingSink` in "just count" mode, a
bounded on-stack arena in production). The parser stays trait-generic.

### What is NOT here

- Blanket impls for `Vec<T>` / `Seq<T, N>`. These live in test-helper
  crates (#81) under an optional `std` or `seq` feature.
- Specific codec formats (JSON, clause bytecode, paradox save). Each
  format ships in its owner crate; this crate defines the contract.
- `Flush` / `Fuse` / back-pressure semantics for buffered byte
  emitters. Deferred; see BACKLOG.

## What this crate does NOT contain

- Engine internals (scheduling, dispatch codegen, phase analysis)
- Platform implementations (mmap, pthread — those are in hilavitkutin)
- Build-time optimisations (LLVM passes — those are in hilavitkutin-build)
- Naming/identity for WUs (consumer concern)
- Concrete codec format implementations (consumer-owned; this crate
  ships `Encoder<T>` / `Decoder<T>` contracts only)
