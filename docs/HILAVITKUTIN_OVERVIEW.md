# hilavitkutin — Design

Morsel-driven pipeline execution engine. Takes WorkUnit declarations
(from hilavitkutin-api), analyses them into phases/trunks/fibers,
compiles dispatch programs, and runs them on a pre-allocated thread
pool. `#![no_std]`, no alloc. All index/count/weight types use arvo
newtypes.

The consolidation topic (202603181200) is the authoritative design
source. This document maps domains to engine modules.

## Plan stage (domains 11-16)

Pure analysis, no runtime state. Takes WU declarations (Read/Write
AccessSets, scheduling hints, COMMUTATIVE flag) and produces a
complete execution plan:

1. Build DAG from AccessMask overlap (arvo-bitmask)
2. Topological sort + node renumbering (arvo-graph)
3. Upward rank → critical path (arvo-graph)
4. Waist detection → phase boundaries (arvo-graph)
5. RCM reordering → fiber grouping order + arena layout (arvo-sparse)
6. Block diagonal + D-M → phase validation + dead column elimination
   (arvo-sparse)
7. Spectral partitioning → trunk formation when >5 fibers
   (arvo-spectral)
8. Fiber grouping: greedy (arvo-comb) or matrix chain DP for >10 ops
   (arvo-comb)
9. Per-fiber morsel sizing (write-only formula, domain 12)
10. Per-phase adaptive configs (MAX_FUSE / BALANCED / MAX_SPLIT)
11. Column classification per fiber (internal / input / output)
12. Dirty propagation masks for incremental skip (arvo-bitmask)

Plan recomputes on any pipeline structure change (new WUs, record
count change, DAG modification). Microsecond cost on const-generic
bitmask structures.

## Dispatch codegen (domain 17)

Turns the plan into executable code. Each fiber gets a monomorphised
dispatch function constructing a LOCAL `&[WuFn]` slice (devirtualised
by LLVM). The flattener emits the rust-pipe pattern:

1. Cache resources to stack-local array
2. Read fiber-input columns at morsel start
3. Pure function pipeline with locals (no intermediate stores)
4. Fiber-internal columns → register-to-register (DSE)
5. All output stores grouped at end (store-buffer friendly)

Per-core compiled dispatch: each physical core gets a monomorphised
function encoding its entire pipeline (phases, record ranges, morsel
boundaries, sync points). No dynamic dispatch at runtime.

Progress counters: AtomicUsize per fiber, monotonic record index.
Plain store (Release) — not RMW, cheapest atomic. Lock-free by
construction (library guarantee, T3 decisions #103-104).

Dispatch approach selected by record count:
- <10K records: IndirectPerFiber or TrunkMega
- \>10K records: ScheduleMega

ExpandedLto pragma (domain 03, hilavitkutin-build) REQUIRED for
devirtualisation.

## Resource handling (domain 19)

Stack-local resource caching: snapshot accessed resources to stack
before morsel loop, redirect res_ptr. LLVM proves stack doesn't alias
heap column data → promotes to registers. Zero resource memory ops in
hot loop. Write back mutable resources after loop.

Pointer separation: resource storage and column storage at separate
provenance. Both through `&self` receiver (load-bearing: prevents
noalias UB when WUs are fused within a fiber).

Convergence accumulators: each head+tail thread gets its own stack-
local accumulator. Merged after convergence (addition for additive,
sequential fallback for non-commutative).

## Threading (domain 20)

Pre-allocated thread pool: `physical_core_count` threads, hybrid wake
(spin 128 → park). Pool persists for pipeline lifetime.

Heterogeneous core awareness: P-cores get critical path trunks +
larger morsels. E-cores get branches/leaves + proportionally smaller
ranges. Thread count = min(physical_cores, parallelisable_width + 1).

Core-pinned trunks: each trunk assigned to specific pool thread at
plan time (warm L1). Leftover threads do convergence → branches →
bridges in priority order.

Head+tail convergence: two threads process same commutative fiber
from opposite ends. Every commutative fiber gets ~2x parallelism.

Pipeline parallelism between phases: phases overlap via AtomicUsize
progress counters. Phase N+1 starts when N produces enough records
for one morsel. Total time ≈ max(phases) + fill latency.

Work-stealing is NOT default. Deterministic morsel assignment (pre-
partitioned at plan time). Consumer can provide stealing Executor
via trait override for variable per-record workloads.

## Strategy selection (domain 21)

Plan-time selection based on record count + pipeline shape:
- <10K records → Sequential
- Deep (depth > fibers/2, roots ≤ 2) → Sequential
- Wide (roots > depth/2) → Adaptive / PipeChase
- Mixed (phases have parallel trunks) → Phased

Per-phase config selected independently based on phase topology.
Schedule reuse across frames (construct once, reuse). Morsel-to-core
affinity pinned across frames. Cadence via SchedulerMetrics resource
(consumer reads, consumer decides policy).

## Runtime adaptation (domain 22)

Static analysis at plan time: cache pressure profile, data flow
volume map, column lifetime map, memory watermark.

Runtime analysis across frames: per-morsel timing, change frequency
(hot/warm/cold), morsel temperature → core assignment, frame time
prediction (EMA, 1/8 decay), throughput trending.

Executor reorganisation triggers between frames (not during
execution): EMA threshold comparison → recompute morsel sizes,
re-select per-phase configs, or full plan recompute.

Predictive parking at sync points: <100ns spin, 100ns-10µs
spin_loop, >10µs park.

## Scheduler builder + meta (domain 23)

`Scheduler<MAX_UNITS, MAX_STORES, MAX_LANES>` keeps its three
const-generic capacity parameters governing engine-internal sizes.

`SchedulerBuilder<MAX_UNITS, MAX_STORES, MAX_LANES, Wus, Stores>`
gains two phantom-tuple type parameters for type-state
accumulation. `Wus` carries the registered WU types. `Stores`
unifies registered `Resource<T>`, `Column<T>`, `Virtual<T>`
markers (matching how `WorkUnit::Read` is shaped in
`hilavitkutin-api`). Initial state on `Scheduler::builder()` is
`SchedulerBuilder<..., (), ()>`.

Builder API:
```
Scheduler::<MAX_UNITS, MAX_STORES, MAX_LANES>::builder()
    .add::<MyWU>()
    .resource::<T>(initial)
    .resource_default::<T>()
    .column::<T>()
    .add_virtual::<T>()
    .add_kit(MyKit)
    .memory(my_provider)
    .threads(my_pool)
    .clock(my_clock)
    .build()
```

Method shapes (registration methods evolve the type-state):

```rust
impl<..., Wus, Stores> SchedulerBuilder<..., Wus, Stores>
where Wus: AccessSet, Stores: AccessSet,
{
    pub fn add<W: WorkUnit>(self) -> SchedulerBuilder<..., (W, Wus), Stores>
    where (W, Wus): AccessSet;

    pub fn resource<T: 'static>(self, _init: T)
        -> SchedulerBuilder<..., Wus, (Resource<T>, Stores)>
    where (Resource<T>, Stores): AccessSet;

    pub fn column<T: 'static>(self) -> SchedulerBuilder<..., Wus, (Column<T>, Stores)>
    where (Column<T>, Stores): AccessSet;

    pub fn add_virtual<T: 'static>(self) -> SchedulerBuilder<..., Wus, (Virtual<T>, Stores)>
    where (Virtual<T>, Stores): AccessSet;

    pub fn add_kit<K>(self, k: K) -> K::Output
    where
        K: Kit<Self>,
        K::Output: BuilderExtending<Self>;

    pub fn memory<M: 'static>(self, _provider: M) -> Self;
    pub fn threads<P: 'static>(self, _pool: P) -> Self;
    pub fn clock<C: 'static>(self, _clock: C) -> Self;
}

impl<..., Wus, Stores> SchedulerBuilder<..., Wus, Stores>
where
    Wus: Buildable<Stores>,
    Stores: AccessSet,
{
    pub fn build(self) -> Scheduler<MAX_UNITS, MAX_STORES, MAX_LANES>;
}
```

`.add::<W>()`, `.resource(_)`, `.column()`, `.add_virtual()` each
prepend a marker to one of the type-state tuples. `.add_kit(k)`
returns whatever `k.install(self)` produces (Kit-defined). The
platform-provider methods (`.memory`, `.threads`, `.clock`) thread
through `Self` unchanged for this round; type-state tracking for
platform providers is a future round if a call site demands it.

`.add_kit` carries `K::Output: BuilderExtending<Self>` from
`hilavitkutin-api`. The bound proves the Kit kept the same `Wus`
(Kits never add WUs, only stores) and only extended `Stores`
(`WuSatisfied<Stores>` proves no prior store was wiped). A buggy
Kit returning `SchedulerBuilder<..., (), ()>` fails this bound at
the call site with a clean compile error pointed at the Kit's
`Output` declaration, not at a downstream `.build()`.

`.build()` carries `Wus: Buildable<Stores>` from `hilavitkutin-api`.
This unfolds (recursive cons-list impls) into the conjunction of
`Stores: WuSatisfied<Wᵢ::Read> + WuSatisfied<Wᵢ::Write>` for every
`Wᵢ`, which itself unfolds into per-store `Stores: Contains<Tⱼ>`
proofs. The cons-list recursion impls handle arbitrary-depth
`Wus` and `Stores` accumulators with no per-arity cap.

When a registered WU's `Read` mentions a store that was not
registered, `.build()` fails to compile with a pointed error. For
example, a WU with `type Read = read![Resource<Interner>]` and no
`.resource(Interner)` or `.add_kit(InternerKit)` triggers the
`Contains` `#[diagnostic::on_unimplemented]` message:

```text
note: store `()` does not contain `Resource<Interner>`. Register it with
      `.resource::<Interner>(initial)`, `.column::<Interner>()`,
      `.add_virtual::<Interner>()`, or install a Kit that registers it.
```

The error names the missing store directly, by type, and lists
the four registration paths.

The engine declares `#![recursion_limit = "512"]` at its crate
root. This accommodates `.build()` against 50-WU by 50-store apps
without trait-solver overflow. Consumers whose apps exceed this
scale should bump the limit at their own crate root; see the
`hilavitkutin-api` "Recursion limit" subsection.

Static composition (R6): all WUs registered at compile time. No
runtime registration. Plan parameters adapt; composition does not.

Kit composition: see `hilavitkutin-kit/DESIGN.md`. A Kit is a
Bevy-style preset that bundles a set of `.resource` / `.column` /
`.add_virtual` calls under one name. Kits chain freely with each
other and with raw builder calls; each `.add_kit(k)` returns the
type-state produced by the Kit's install method, constrained by
`BuilderExtending<Self>`.

Self-hosting meta pipeline: PlanStage → ScheduleReady → PassStart →
ScheduleEnd. Meta resources: Dag, ExecutionPlan, LaneAssignment,
SchedulerMetrics. Irreducible kernel ~50 lines.

Version stamps: parallel arrays (SoA), capture-before + compare-after.
Changed stores = bitmask. Increment only on completion.

Error handling (R7): execute() → (). Abort on failure. Errors via
data flow (error columns/virtuals). Fiber failure → freeze progress
→ poison dependents → independent fibers continue. PipelineResult:
per-fiber status (Completed / Failed / Poisoned).

R2 APIs: evict/dump + inject/import for column persistence.
hilavitkutin owns generation counters; consumer owns storage.

## Platform implementations (domain 02)

os tier (default): raw mmap/munmap for MemoryProvider, raw
pthread_create/clone for ThreadPool, clock_gettime for Clock. Direct
syscalls, no libc where possible.

std tier (fallback): std::alloc, std::thread, std::time::Instant.
catch_unwind available but abort preferred (R7).

no_os tier: trait contracts only. Consumer provides all impls.
