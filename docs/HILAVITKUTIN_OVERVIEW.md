# hilavitkutin, Design

Morsel-driven pipeline execution engine. Takes WorkUnit declarations
(from hilavitkutin-api), analyses them into phases/trunks/fibers,
compiles dispatch programs, and runs them on a pre-allocated thread
pool. `#![no_std]`, no alloc. All index/count/weight types use arvo
newtypes.

The consolidation topic (202603181200) is the authoritative design
source. This document maps domains to engine modules.

## Plan stage (domains 11-16)

Pure analysis, no runtime state. Takes WU declarations (Read/Write
AccessSets, scheduling hints, COMMUTATIVE flag) and produces a
complete execution plan:

1. Build DAG from AccessMask overlap (arvo-bitmask). DependencyGraph
   uses CSR backing (`row_offsets` + `col_indices` + `edge_kinds`)
   per Topic 9 axis B, bench-locked canonical at every N (no threshold).
2. Topological sort + node renumbering (arvo-graph)
3. Upward rank → critical path (arvo-graph)
4. Waist detection → phase boundaries (arvo-graph)
5. RCM reordering → fiber grouping order + arena layout (arvo-sparse)
6. Block diagonal + D-M → phase validation + dead column elimination
 (arvo-sparse). Consumes `core_class: [CoreClass; MAX_CORES]` from
 heterogeneous-core detection per Topic 6 axis E.
7. Spectral partitioning → trunk formation when >5 fibers
 (arvo-spectral)
8. Fiber grouping: greedy (arvo-comb) or matrix chain DP for >10 ops
 (arvo-comb)
9. Per-fiber morsel sizing (write-only formula, domain 12). Output
   includes `micro_morsel_interval: USize` alongside `morsel_size: USize`
   per Topic 7 amendment. Default `MICRO_MORSEL_INTERVAL = 64` records.
   Adapt sidecar cache footprint subtracted from `l1_write_budget`
   per Topic 5 M10 amendment.
10. Per-phase adaptive configs (MAX_FUSE / BALANCED / MAX_SPLIT)
11. Column classification per fiber (internal / input / output)
12. Dirty propagation masks for incremental skip (arvo-bitmask)
13. FiberShape classification per Topic 3 amendment (Topic 4 axis D):
    typed projection over the unique fiber shape set (K, not 2^N).
    Codegen-emitted; informs per-core dispatch monomorphisation.

Plan recomputes on `Scheduler::run()` entry when the dirty bit is set
(Topic 1 axis 4). Dirty bit is keyed per `PlanAffecting`-bound Resource
(Topic 8 axis B); resources without that marker are cheap value-swaps.

Plan recompute is single-threaded in v1 (sequential through steps 1-13).
Parallelisation deferred until consumer pipelines push beyond ~256 WUs.

## Dispatch codegen (domain 17)

Turns the plan into executable code. Codegen is shaped by Topic 4 axes:

- Sealed `trait DispatchCodegen<Cfg: RunCfg>` with TAIT
  `type CoreDispatch = impl Fn(...)`. Single shipped impl
  `StandardCodegen` lowers to closed-form arithmetic, zero `blr`.
- Per-fiber-shape generic monomorphisation: `fn run_fiber<S: FiberShape>(...)`
  over the sealed `FiberShape: Sealed` marker carrying `type WuTuple` +
  `const SHAPE_ID`. Combinatorics K (unique shapes), not 2^N.
  Sealed-trait dispatch measured free vs hand-written direct calls
  per `mock/benches/dispatch_static_n*_findings.md`.
- Codegen-wrapped via `#[inline(always)] fn invoke_<wu>_in_fiber_<n>`
  shim fns; LLVM elides the shim layer.

The fused-pipeline pattern within a fiber:

1. Cache resources to stack-local array
2. Read fiber-input columns at morsel start
3. Pure function pipeline with locals (no intermediate stores)
4. Fiber-internal columns → register-to-register (DSE)
5. All output stores grouped at end (store-buffer friendly)

Per-core compiled dispatch: each physical core gets a monomorphised
`CoreProgram<...>` encoding its phases, record ranges, morsel
boundaries, and sync points. Worker entry receives
`Pin<&'frame CoreProgram<...>>` per the Topic 6 axis C lifetime
contract (validated by the PoolFrame lifetime sketch).

Progress counters: `AtomicUsize` per fiber, monotonic record index.
Plain store (Release), not RMW; cheapest atomic per the
`atomic_ordering_cost` bench (Relaxed beats Acquire/SeqCst by 3-4x
on aarch64). Lock-free by construction.

`DispatchApproach` shipped in v1 (`#[non_exhaustive]`):
- `ScheduleMega`: approach A.
- `TrunkMega`: approach B (within margin of A per bench).
- `IndirectPerFiber`: approach D + E. Single-target indirect dispatch
  per fiber; BTB-warm above N≈256 per the `dispatch_dynamic` bench
  (volatile fn pointer converges to baseline once warmed).

Selection driven by record count + per-fiber shape entropy. Codegen
emits `dmb st` (aarch64) / `sfence` (x86_64) between final FiberOutput
non-temporal `stnp` writes and progress-counter Release store
(Topic 3 invariant S3).

ExpandedLto pragma (domain 03, hilavitkutin-build) REQUIRED for
devirtualisation.

## Resource handling (domain 19)

Stack-local resource caching: snapshot accessed resources to stack
before morsel loop, redirect res_ptr. LLVM proves stack doesn't alias
heap column data → promotes to registers. Zero resource memory ops in
hot loop. Write back mutable resources after loop.

Pointer separation: resource storage and column storage at separate
provenance. Both through `&self` receiver (load-bearing: prevents
noalias UB when WUs are fused within a fiber).

Convergence accumulators: each head+tail thread gets its own stack-
local accumulator. Merged after convergence (addition for additive,
sequential fallback for non-commutative).

## Threading (domain 20)

Locked shape from Topic 6 (round 202605101036). Pre-allocated thread
pool: `physical_core_count` threads, hybrid wake. Pool persists for
pipeline lifetime.

### Executor trait

Open `trait Executor: Sealed`, monomorphised. Default impl
`HybridExecutor` ships in this crate. `Scheduler<'frame, Cfg, E: Executor = HybridExecutor>`
propagates the type so consumers can swap in custom executors.
Sealed bound enforces Topic 3 S7 cross-thread atomic-ordering protocol.

### Three-tier parking

Per-OS atomic-wait primitives (Topic 6 axis B):

- Linux: `futex(2)` `FUTEX_WAIT_PRIVATE` / `FUTEX_WAKE_PRIVATE`.
- macOS: `__ulock_wait2` / `__ulock_wake`.
- Windows: `WaitOnAddress` / `WakeByAddressSingle`.

Tier selected by per-phase `predicted_wait_ns` slot in `PoolFrame`
(Topic 6 axis J):

- `< 2µs`: spin-only (skip middle + park tiers).
- `2µs to 50µs`: spin budget exhausted → futex/ulock/WaitOnAddress.
- `> 50µs`: skip spin, park immediately.

Thresholds in `WakeStrategy`. Per-CoreClass spin defaults: P=128, E=32
(Topic 6 axis K; subject to bench validation in follow-up).

### PoolFrame

`PoolFrame<MAX_CORES, MAX_PHASES>` is engine-internal data plane on
per-pool cache lines (Topic 3 amendment M11):

```rust
pub struct PoolFrame<const MAX_CORES: usize, const MAX_PHASES: usize> {
    pub shutdown: AtomicBool,
    pub phase_arrived: AtomicU32,
    pub predicted_wait_ns: [AtomicU32; MAX_PHASES],
    pub progress_slots: [AtomicUsize; MAX_CORES],
}
```

Workers receive `Pin<&'frame PoolFrame>` per Topic 6 axis C, validated
by the lifetime sketch under `mock/research/sketches/202605101036-poolframe-lifetime/`.

### Phase barrier

Centralised atomic counter + futex-wake-all (Topic 6 axis I). Worker
on phase exit: `phase_arrived.fetch_add(1, Release)`. If value < N:
futex-wait on the address. Last arriver (value == N): reset counter,
futex-wake-all. Tree-barrier deferred to follow-up if 32+ core bench
shows ping-pong on the centralised counter.

### Heterogeneous-core awareness

Per-OS P/E classification ships in v1 (Topic 6 axis E):

- x86_64: CPUID leaf 0x1A.
- Linux: `/sys/devices/system/cpu/*/topology/cluster_id` + cpufreq.
- macOS: `sysctlbyname("hw.perflevel0.physicalcpu")`.
- Windows: `GetSystemCpuSetInformation`.

`CoreAssignment` (existing) extends with `core_class: [CoreClass; MAX_CORES]`.
P-cores get critical-path trunks + larger morsels. E-cores get
branches/leaves + proportionally smaller ranges.

Head+tail convergence: two threads process same commutative fiber
from opposite ends. Every commutative fiber gets ~2x parallelism
(Topic 3 axis E).

Pipeline parallelism between phases: phases overlap via `progress_slots[]`
atomic counters. Phase N+1 starts when N produces enough records
for one morsel.

### Shutdown protocol

`PoolFrame.shutdown: AtomicBool`. Workers Relaxed-load after each morsel
boundary; on shutdown=true, exit mainloop cleanly. `Scheduler::Drop`:

1. Stores `true` into `shutdown`.
2. Calls futex/ulock/WakeByAddressAll on every parked worker.
3. Joins each ThreadHandle.

Per Topic 6 axis G. No thread cancellation primitives used.

### Work-stealing

NOT in v1 (Topic 6 axis D). Deterministic morsel assignment (pre-
partitioned at plan time) covers v1's workloads. `Executor::try_steal`
deferred to v2 behind a `work-stealing` feature flag.

## Strategy selection (domain 21)

Plan-time selection based on record count + pipeline shape:
- <10K records → Sequential
- Deep (depth > fibers/2, roots ≤ 2) → Sequential
- Wide (roots > depth/2) → Adaptive / PipeChase
- Mixed (phases have parallel trunks) → Phased

Per-phase config selected independently based on phase topology.
Schedule reuse across frames (construct once, reuse). Morsel-to-core
affinity pinned across frames. Cadence via SchedulerMetrics resource
(consumer reads, consumer decides policy).

## Runtime adaptation (domain 22)

Static analysis at plan time: cache pressure profile, data flow
volume map, column lifetime map, memory watermark.

Runtime analysis across frames: per-morsel timing, change frequency
(hot/warm/cold), morsel temperature → core assignment, frame time
prediction (EMA, 1/8 decay), throughput trending.

Executor reorganisation triggers between frames (not during
execution): EMA threshold comparison → recompute morsel sizes,
re-select per-phase configs, or full plan recompute.

Predictive parking at sync points: <100ns spin, 100ns-10µs
spin_loop, >10µs park.

## Scheduler builder + meta (domain 23)

`Scheduler<Wus, Stores>` is the engine's runtime handle. The two
phantom-tuple type parameters carry the typestate accumulated at
build time: `Wus` is the registered WUs (cons-list of `WorkUnit`
declarations), `Stores` is the registered stores (cons-list unifying
`Resource<T>`, `Column<T>`, `Virtual<T>` markers). The v0.1
`MAX_UNITS` / `MAX_STORES` / `MAX_LANES` capacity const generics
are dropped in round-4; the typestate alone carries the shape, and
runtime sizes are derived from the typestate's `Depth::D`
projections rather than from explicit caps.

`SchedulerBuilder<Wus, Stores, Platform>` accumulates typestate
across each registration call. Initial state on
`Scheduler::builder()` is `SchedulerBuilder<Empty, Empty, Empty>`.

The builder takes one verb: `.with(value)`. Every type accepted by
`.with` impls the sealed `Provider` trait declared in
`hilavitkutin-api`. The trait routes per-kind typestate updates
through its `Dispatch` associated type; see the api crate's
`## Provider, the unified registration contract` section.

Example call site:

```rust
Scheduler::builder()
    .with(MyMemory::new(4096))           // platform: memory
    .with(MyThreadPool::new(8))          // platform: threads
    .with(MyClock)                       // platform: clock
    .with(Resource::new(GameState::default())) // store: stateful
    .with(Column::<Player>::new())       // store: zero-data
    .with(Virtual::<Tick>::new())        // store: zero-data
    .with(LinkedBin::<dyn LinterApi>::new()) // store: extension bin
    .with(InternerKit)                   // kit: declarative bundle
    .with(SpawnerWu)                     // workunit
    .with(PhysicsWu)                     // workunit
    .build()
```

Method signature:

```rust
impl<Wus, Stores, Platform> SchedulerBuilder<Wus, Stores, Platform>
where Wus: AccessSet, Stores: AccessSet,
{
 pub fn with<P: Provider>(self, _provider: P) -> SchedulerBuilder<
     <P::Dispatch as Dispatch>::NextWus<Wus>,
     <P::Dispatch as Dispatch>::NextStores<Stores>,
     <P::Dispatch as Dispatch>::NextPlatform<Platform>,
 >;
}

impl<Wus, Stores, Platform> SchedulerBuilder<Wus, Stores, Platform>
where
 Wus: WorkUnitBundle,
 Stores: AccessSet + StoreBundle
 + ContainsAll<Wus::AccumRead>
 + ContainsAll<Wus::AccumWrite>,
{
 pub fn build(self) -> Scheduler<Wus, Stores, Platform>;
}
```

The single `.with` method routes each registration through
`P::Dispatch`. A `WorkUnit` provider (kind `WorkUnit`) carries
`Dispatch = UnitDispatch<Self>`, which prepends `Self` onto the
`Wus` accumulator. A store provider (`Resource<T>`, `Column<T>`,
`Virtual<T>`, `LinkedBin<T>`) carries `Dispatch = StoreDispatch<Self>`,
which prepends onto `Stores`. A `Kit` provider carries `Dispatch
= KitDispatch<Self>`, which concats `K::Units` into `Wus` and
`K::Owned` into `Stores`. A platform provider (memory, thread pool,
clock) carries `Dispatch = PlatformDispatch<Self>`, which prepends
onto the platform tuple. Each kind's GAT is identity for the
accumulators it does not touch. The blanket impl on
`SchedulerBuilder` reads `P::Dispatch` once per call and the trait
solver computes the next typestate via the GAT walks.

Kit registration is type-driven: the Kit instance flows through
`.with(MyKit)` purely so `Self` participates in
`Provider::Dispatch` resolution; the engine reads `K::Units` and
`K::Owned` at compile time. Unit-struct Kits register zero-cost.
Stateful Kits (rare) construct via `.with(MyKit::new(args))` and
the constructor body computes whatever runtime data the Kit needs;
the typestate accumulation is the same.

The v0.1 Kit shape (`pub trait Kit<B> { type Output; fn install(self, B) -> Self::Output }`)
is gone. The new declarative `pub trait Kit { type Units; type Owned }`
shape (in `hilavitkutin-kit`) lets the engine accumulate
without invoking an install method. The v0.1 `BuilderExtending<B>`
sealed trait that bounded `K::Output` is therefore also gone; the
new shape has no `Output` to bound.

`.build()` carries `Wus: WorkUnitBundle, Stores: ContainsAll<Wus::AccumRead> + ContainsAll<Wus::AccumWrite>`
from `hilavitkutin-api`. The accumulator is computed once at the
type level via the recursive `WorkUnitBundle for Cons<W, T>` impl;
the proof unfolds via the recursive `ContainsAll` impl into
per-store `Stores: Contains<Tᵢ>` proofs. With `#[marker]` on both
`Contains` and `ContainsAll`, overlapping resolutions are
permitted; duplicates in `Wus::AccumRead/Write` extend traversal
but do not duplicate proof work.

When a registered WU's `Read` mentions a store that was not
registered, `.build()` fails to compile with a pointed error. For
example, a WU with `type Read = read![Resource<Interner>]` and no
`.with(Resource::new(interner))` or `.with(InternerKit)` triggers
the `Contains` `#[diagnostic::on_unimplemented]` message:

```text
note: store `Empty` does not contain `Resource<Interner>`. Register it with
 `.with(Resource::new(value))`, `.with(Column::<Interner>::new())`,
 `.with(Virtual::<Interner>::new())`, or install a Kit that owns it
 via `.with(MyKit)`.
```

The error names the missing store directly, by type, on the first
inline line of the diagnostic. The verbose accumulator chain
follows but does not bury the marker name. Empirically validated
to depth 5 with 25 WUs and 13 stores; see
`mock/research/sketches/202605050700_deep_stacking_d5/FINDINGS.md`.

The engine declares `#![recursion_limit = "512"]` at its crate
root. This accommodates `.build()` against 50-WU by 50-store apps
without trait-solver overflow. Consumers whose apps exceed this
scale should bump the limit at their own crate root; see the
`hilavitkutin-api` "Recursion limit" subsection.

Static composition (R6): all WUs registered at compile time. No
runtime registration. Plan parameters adapt; composition does not.

Kit composition: see `hilavitkutin-kit/DESIGN.md`. A Kit is a
declarative bundle: `type Units` (cons-list of WorkUnits) plus
`type Owned` (cons-list of stores). Kits chain freely with each
other and with raw builder calls; each `.with(MyKit)` (or
`.with(MyKit::new(args))` for stateful kits) accumulates `K::Units`
into the bundle and `K::Owned` into the stores via `KitDispatch<K>`
typestate routing. Order matters for the `ContainsAll` proof at
`.build()`: a Kit whose WUs read from another Kit's Owned state
must come after the owning Kit in the builder chain (or after the
relevant `.with(Resource::new(value))` / `.with(Column::<T>::new())`
calls).

### `Scheduler::replace_resource`

```rust
impl<Wus, Stores> Scheduler<Wus, Stores> {
 pub fn replace_resource<T: Replaceable>(&mut self, _new: T)
 where
 Stores: Contains<Resource<T>>;
}
```

App-level method to swap the existing `Resource<T>` instance in
the scheduler's data plane. The static `T: Replaceable` bound
rejects override attempts on types that did not opt in via
`impl Replaceable for T {}`. The `Stores: Contains<Resource<T>>`
bound proves the resource exists in the registered stores.

Replaceable opt-in is the only enforced annotation on
Owned types (per audit C2/M4 plus the S5 visibility-axis
finding); see `hilavitkutin-api::Replaceable` and
the design notes.

When a consumer attempts `scheduler.replace_resource(NotReplaceable)`,
rustc emits an E0277 with a high-quality diagnostic that names
the offending type, lists the types that DO impl `Replaceable`,
and points at the bound declaration. See
`mock/research/sketches/202605050615_kit_taxonomy/FINDINGS.md`
for the empirical evidence.

`replace_resource` is app-only. Called between `.build()` and
`scheduler.run()`. Not a hot-path operation; the typestate change
is logical only (the `Resource<T>` slot's value is replaced; its
type-level identity in `Stores` is unchanged).

Self-hosting meta pipeline: PlanStage → ScheduleReady → PassStart →
ScheduleEnd. Meta resources: Dag, ExecutionPlan, LaneAssignment,
SchedulerMetrics. Irreducible kernel ~50 lines.

Version stamps: parallel arrays (SoA), capture-before + compare-after.
Changed stores = bitmask. Increment only on completion.

Error handling (R7): execute() → (). Abort on failure. Errors via
data flow (error columns/virtuals). Fiber failure → freeze progress
→ poison dependents → independent fibers continue. PipelineResult:
per-fiber status (Completed / Failed / Poisoned).

R2 APIs: evict/dump + inject/import for column persistence.
hilavitkutin owns generation counters; consumer owns storage.

## Platform implementations (domain 02)

os tier (default): raw mmap/munmap for MemoryProvider, raw
pthread_create/clone for ThreadPool, clock_gettime for Clock. Direct
syscalls, no libc where possible.

std tier (fallback): std::alloc, std::thread, std::time::Instant.
catch_unwind available but abort preferred (R7).

no_os tier: trait contracts only. Consumer provides all impls.
