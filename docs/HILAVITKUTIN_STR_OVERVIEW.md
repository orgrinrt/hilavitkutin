# hilavitkutin-str тАФ Design Overview

`#![no_std]`, zero external dependencies. Interned string system
for the hilavitkutin ecosystem. Shared across saalis, polka-dots,
loimu. Lives in saalis workspace for now, will be extracted to
hilavitkutin repo when stabilised.

## `Str` тАФ interned string handle

```rust
#[repr(C)]
#[derive(Copy, Clone, Eq, PartialEq, Hash)]
pub struct Str(u32);
```

u32 handle into the string table. Comparison is integer equality.
`Copy`, `repr(C)`, ABI-stable, 4 bytes everywhere.

Bit layout (nibble-aligned):
- bit 31: `0` = const (compile-time), `1` = runtime (arena)
- bits 30-28: reserved flags
- bits 27-0: 28-bit ID (268M unique entries)

Const IDs are content hashes (deterministic, stable across
sessions and builds). Runtime IDs are sequential (fast, ephemeral).

## `str_const!()` тАФ compile-time string creation

```rust
macro_rules! str_const {
    ($s:literal) => {{
        #[used]
        #[link_section = ".hilavitkutin_strings"]
        static __ENTRY: StaticStrEntry = StaticStrEntry {
            hash: Str::__make(const_fnv1a($s) & 0x0FFF_FFFF),
            value: $s,
        };
        Str::__make(const_fnv1a($s) & 0x0FFF_FFFF)
    }};
}
```

One macro call produces the handle AND registers the string in a
linker section. No external deps тАФ no `inventory`. Platform-
specific section walking at startup:
- Linux: `__start_hilavitkutin_strings` / `__stop_hilavitkutin_strings`
- macOS: `getsectiondata` on `__DATA,hilavitkutin_strings`

`Str::__make` is not public тАФ `str_const!` is the only way to
create const handles.

## `StringInterner<A>` тАФ wrapper

Wraps `ArenaInterner` with const table handling. The const table

Methods:
- `intern(&self, s: &str) -> Str` тАФ check const table, fall back

- `resolve(&self, s: Str) -> &str` тАФ if const: linker section;

## Content hashing

`const_fnv1a` тАФ FNV-1a hash computable in const context. Used
for `str_const!` handle derivation. Also used by
`hilavitkutin-persistence` for stable on-disk string identity.

The content hash is the file-level identity of a string. The
sequential runtime ID is the session-level identity. They never
need to agree тАФ the hash is stable, the runtime ID is ephemeral.
