# hilavitkutin-ctx — Design Overview

`#![no_std]`, zero dependencies. Provider-gated context framework.
Reusable across saalis, polka-dots, loimu.

## Consumer DX

One impl per capability. `Self::Ctx` just works:

```rust
impl Cataloguer for Steam {
    type Ctx = (SteamConn, MetaWriter);
    fn catalogue(&self, ctx: &Self::Ctx) -> Result<(), ()> {
        ctx.connection().check()?;
        ctx.connection().get(str!("/apps"))?;
        ctx.writer().insert(42)?;
        Ok(())
    }
}
```

## How it works

### Provider API traits

Domain-defined contracts for what a provider can do:

```rust
trait ConnectorApi {
    fn check(&self) -> Result<(), ()>;
    fn get(&self, endpoint: Str) -> Result<(), ()>;
}

trait WriterApi {
    fn insert(&self, id: EntityId) -> Result<(), ()>;
    fn upsert(&self, id: EntityId) -> Result<(), ()>;
}

trait QueryApi {
    fn query(&self) -> Result<(), ()>;
}
```

### Accessor traits

Generated by `provider!` macro. One per provider kind.
"The context has this provider available":

```rust
trait HasConnector {
    type Connector: ConnectorApi;
    fn connection(&self) -> &Self::Connector;
}

trait HasWriter {
    type Writer: WriterApi;
    fn writer(&self) -> &Self::Writer;
}

trait HasQuery {
    type Query: QueryApi;
    fn querier(&self) -> &Self::Query;
}
```

### Context<P>

Delegates to accessor traits on `P`:

```rust
pub struct Context<P> { providers: P }

impl<P: HasConnector> Context<P> {
    fn connection(&self) -> &P::Connector {
        self.providers.connection()
    }
}
// ... same for each accessor
```

### Tuple accessor impls

Fixed position per arity. Only ONE generic bound per impl
(no coherence conflicts). Generated by `tuple!` macros:

```rust
// 2-tuple: pos 0 = connector, pos 1 = writer
impl<A: ConnectorApi, B> HasConnector for (A, B) {
    type Connector = A;
    fn connection(&self) -> &A { &self.0 }
}
impl<A, B: WriterApi> HasWriter for (A, B) {
    type Writer = B;
    fn writer(&self) -> &B { &self.1 }
}

// 3-tuple: pos 0 = connector, pos 1 = query, pos 2 = writer
impl<A: ConnectorApi, B, C> HasConnector for (A, B, C) {
    type Connector = A;
    fn connection(&self) -> &A { &self.0 }
}
impl<A, B: QueryApi, C> HasQuery for (A, B, C) {
    type Query = B;
    fn querier(&self) -> &B { &self.1 }
}
impl<A, B, C: WriterApi> HasWriter for (A, B, C) {
    type Writer = C;
    fn writer(&self) -> &C { &self.2 }
}
```

Position convention per arity:
- 2-tuple: (connector, writer)
- 3-tuple: (connector, query, writer)
- Wrong order → compile error (bound not satisfied)

### Domain traits

Define `type Ctx` with accessor trait bounds:

```rust
trait Cataloguer {
    type Ctx: HasConnector + HasWriter;
    fn catalogue(&self, ctx: &Self::Ctx) -> Result<(), ()>;
}

trait Enricher {
    type Ctx: HasConnector + HasQuery + HasWriter;
    fn enrich(&self, ctx: &Self::Ctx) -> Result<(), ()>;
}
```

Arbitrary methods, arbitrary return types. The framework doesn't
prescribe method shapes.

## Framework DX — macros

### `provider!` — one per provider kind

```rust
provider!(ConnectorApi as HasConnector => connection);
provider!(QueryApi as HasQuery => querier);
provider!(WriterApi as HasWriter => writer);
```

Generates:
1. Accessor trait (`HasConnector`)
2. Context delegation (`impl<P: HasConnector> Context<P> { fn connection()... }`)

### `tuple!` — one per layout

```rust
tuple!(ConnectorApi: HasConnector => connection,
       WriterApi: HasWriter => writer);

tuple!(ConnectorApi: HasConnector => connection,
       QueryApi: HasQuery => querier,
       WriterApi: HasWriter => writer);
```

Generates accessor trait impls on the tuple for each position.

### `define_providers!` — facade (optional)

Wraps `provider!` + `tuple!` calls:

```rust
define_providers! {
    providers {
        ConnectorApi as HasConnector => connection,
        QueryApi as HasQuery => querier,
        WriterApi as HasWriter => writer,
    }
    layouts {
        (ConnectorApi, WriterApi),
        (ConnectorApi, QueryApi, WriterApi),
    }
}
```

One macro call generates everything. The `layouts` section
declares which tuple shapes exist.

## Self::Ctx resolution

Each domain trait has its own `type Ctx`. `Self::Ctx` resolves
per-trait, not per-type. A single struct can impl `Cataloguer`
AND `Enricher` with different `Ctx` types — no ambiguity.

The trait bounds on `type Ctx` (e.g. `HasConnector + HasWriter`)
enforce at compile time which providers must be in the tuple.
If the tuple at that position doesn't impl the right provider
API trait → compile error.

## Multiple capabilities on one type

```rust
struct Steam;

impl Cataloguer for Steam {
    type Ctx = (SteamConn, MetaWriter);
    fn catalogue(&self, ctx: &Self::Ctx) -> Result<(), ()> {
        ctx.connection().get(str!("/apps"))?;
        ctx.writer().insert(42)?;
        Ok(())
    }
}

impl Enricher for Steam {
    type Ctx = (SteamConn, HotQuery, MetaWriter);
    fn enrich(&self, ctx: &Self::Ctx) -> Result<(), ()> {
        ctx.connection().get(str!("/enrich"))?;
        ctx.querier().query()?;
        ctx.writer().insert(99)?;
        Ok(())
    }
}
```

Different `Ctx` types → different provider sets → different
available methods. `Self::Ctx` unambiguous in each trait's methods.

## No HasCtx needed

Unlike the earlier hilavitkutin WorkUnit pattern, this design
does NOT use `HasCtx<K>`. The provider tuple IS `Self::Ctx`
directly, with accessor trait bounds enforcing the contract.
Simpler, fewer moving parts, same consumer DX.

## Parameterised accessor / API traits

Some consumers need accessor traits that carry a type parameter —
typically to express a compile-time bound (like "this reader only
touches stores in set R") or to host a generic method whose bound
changes per call site. hilavitkutin's WorkUnit `Context` is the
driving example: a `ColumnReader` accessor parameterised over
`R: AccessSet`, with a generic `read<T>` method bounded by
`R: Contains<Column<T>>`.

Two macros ship alongside `provider!` / `tuple!` for this case:

```rust
// API trait has a type parameter; its methods may be generic.
trait ColumnReaderApi<R: AccessSet> {
    fn read<T: ColumnValue>(&self, i: USize) -> T
    where R: Contains<Column<T>>;
}

// Accessor trait carries the same parameter.
provider_generic!(<R: AccessSet> ColumnReaderApi<R> as HasColumnReader<R> => reader);

// Tuple impls thread the parameter through.
tuple_generic!(
    <R: AccessSet>
    ColumnReaderApi<R>: HasColumnReader<R> => reader,
    WriterApi: HasWriter => writer,
);
```

Generated shape mirrors `provider!` / `tuple!` with the type
parameter threaded through the accessor trait, API-trait bound,
and Context delegation impl. Consumers can mix generic and
concrete providers in the same tuple layout.

Multi-parameter variants (`HasX<R, W>` for accessors needing two
type parameters, as hilavitkutin's op registrars do) ship the same
macro shape extended with additional `<R, W>` syntax.

## Key types

| Type | Purpose |
|------|---------|
| `Context<P>` | Provider-gated context wrapper |
| `provider!` | Generates accessor trait + Context delegation (concrete methods) |
| `tuple!` | Generates accessor impls per tuple layout (concrete-method providers) |
| `provider_generic!` | Variant accepting type parameters on the accessor / API traits |
| `tuple_generic!` | Tuple impls threading one or more type parameters |
| `define_providers!` | Facade combining `provider!` + `tuple!` |

## Design rule

No std. No alloc. No deps. Consumer-agnostic. Provider API
traits, accessor traits, and domain traits are defined by the
consumer. This crate provides `Context<P>` + the macros.
