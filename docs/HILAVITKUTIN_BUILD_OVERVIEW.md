# hilavitkutin-build — Design

Shared build-dependency crate. Every hilavitkutin crate and every
consumer's build.rs calls `hilavitkutin_build::configure().run()`.
Independent of hilavitkutin runtime — optimises HOW code is compiled,
not what it does. Standalone, no runtime deps.

## Entry point

```rust
// in any crate's build.rs:
fn main() {
    hilavitkutin_build::configure()
        .profile("release", |p| p.enable::<(
            ExpandedLto, FastMath, LoopOptimization, Pgo, Bolt,
        )>())
        .run();
}
```

`configure().run()` reads cargo features and target env vars
(CARGO_CFG_TARGET_ARCH, CARGO_CFG_TARGET_VENDOR,
CARGO_CFG_TARGET_FEATURE), emits `cargo:rustc-cfg` lines and
`cargo:rerun-if-changed`, and generates the rustc wrapper and config
files. Does NOT do live CPUID/sysctl detection — hardware topology
(L1 sizes, core counts, P/E ratio) is a runtime concern handled by
LazyLock in hilavitkutin's scheduler.

## Four-part build configuration model (T4 §Q1)

**1. hilavitkutin-build (this crate):** reads cargo env vars, emits
cfg flags, generates the rustc wrapper script and config file. Runs
in every crate's build.rs. Does NOT handle LLVM codegen flags
directly — that's the wrapper's job.

**2. No custom cargo features (T4 §Q2):** the compiler already
provides everything needed. ISA capabilities via `target_feature`.
Vendor identity via `CARGO_CFG_TARGET_VENDOR`. Architecture via
`CARGO_CFG_TARGET_ARCH`. Cache/topology is runtime detection.
`compile_error!` guards for any future mutually exclusive feature
groups.

**3. RUSTC_WORKSPACE_WRAPPER:** the primary mechanism for injecting
LLVM flags. A thin shell script generated by hilavitkutin-build,
wrapping every rustc invocation for workspace crates. Inspects
CARGO_CFG_TARGET_ARCH, CARGO_CFG_TARGET_VENDOR,
CARGO_CFG_TARGET_FEATURE and computes flags from three axes:
- **Target:** which target-specific flags to inject
- **Tier:** inferred from `-C opt-level=N` in the rustc args
- **Passes:** which LLVM passes to activate beyond default pipeline

Each (target, tier) pair gets EXACTLY the flags optimal for it.
Can grow to hundreds of lines — more precision is more value.

Configured in `.cargo/config.toml`:
```toml
[build]
rustc-workspace-wrapper = "target/hilavitkutin-build/rustc-wrapper"
```

Only affects workspace crates, not dependencies. Full access to
CARGO_CFG_* env vars for conditional logic.

**4. Generated config file:** pragmas that need Cargo profile
settings (ExpandedLto, Pgo, BuildStd) write to
`target/hilavitkutin-build/hilavitkutin-config.toml`. This file
is generated, gitignored, and consumed by the Cargo profile.

## cfg flag emission

hilavitkutin-build emits NO custom cfg flags for hardware detection.
The compiler provides `target_feature` and `CARGO_CFG_TARGET_FEATURE`
natively. arvo detects hardware capabilities in its own build.rs
via `CARGO_CFG_TARGET_FEATURE`, not custom flags from this crate.

The only cfg this crate emits:
- `arvo_fast_math` — when the FastMath pragma is active. arvo uses
  this to gate fast-math float semantics.

## Pragma system (T4 §Q4f)

Typed pragmas with dependency enforcement:

```rust
pub trait Pragma: 'static {
    const NAME: &'static str;
    type Requires: PragmaRequirement = ();
    fn rustc_flags() -> Vec<String> { vec![] }
    fn cfg_flags() -> Vec<String> { vec![] }
    fn post_build() -> Option<PostBuildAction> { None }
}
```

Requirement combinators: `All<(A, B)>` (all must be active),
`Any<(A, B)>` (at least one). Checked at `configure().run()` time.

**All built-in pragmas:**

| Pragma | What it does | Mechanism |
|---|---|---|
| `LoopOptimization` | IRCE, LoopPredication, SimplifyCFG, LoopInterchange, LoopDistribute, LoopDataPrefetch, SeparateConstOffsetFromGEP via polka-passes.so | Rustc wrapper |
| `Polly` | Polyhedral optimizer (tiling, fusion, GEMM detection). Requires Polly-enabled LLVM or container workflow | Rustc wrapper |
| `MathPeephole` | Float math peephole rewrites via math-peephole.so. **Requires: FastMath** | Rustc wrapper |
| `FastMath` | LLVM unsafe-fp-math flag + `arvo_fast_math` cfg | Rustc wrapper |
| `ExpandedLto` | fat LTO + codegen-units=1. REQUIRED for devirtualisation (domain 17, 12.6x penalty without) | Generated config |
| `Pgo` | Consume PGO profiles if they exist on disk | Generated config |
| `Bolt` | Post-link binary rewriting. Linux ELF only, no-op on macOS | Post-build hook |
| `Profiling` | Run profiling benchmarks post-build. **Requires: Any<(Pgo, Bolt)>** | Post-build hook |
| `BuildStd` | Rebuild std from source with optimisation flags. NOT in defaults (build time) | Generated config |
| `ParallelCodegen<N>` | `-Zthreads=N`. 0 = auto-detect via available_parallelism() | Rustc wrapper |
| `SharedGenerics` | `-Zshare-generics=y`. Reduces link time in debug builds | Rustc wrapper |
| `LoopFusion` | Fuse adjacent loops. EXPERIMENTAL. Not in defaults. Benchmark before enabling | Rustc wrapper |
| `MimallocAllocator` | Recommends mimalloc. Consumer sets #[global_allocator] | Advisory |

**Default pragma sets per profile:**

| Profile | Default pragmas |
|---|---|
| dev | ParallelCodegen, SharedGenerics |
| dev-opt | ParallelCodegen, SharedGenerics, LoopOptimization |
| release | ParallelCodegen, LoopOptimization, MathPeephole, FastMath, ExpandedLto, Bolt, Pgo |
| profiling | ParallelCodegen, LoopOptimization, MathPeephole, FastMath, Profiling, Pgo, Bolt |

Not in any default: Polly (requires special LLVM), BuildStd (build

## LLVM pass plugin (T4 §Q4c)

A ~40-line C++ file compiled against LLVM headers. Registers passes
at two extension points in the LLVM pipeline:

**VectorizerStartEP (before vectorizer):**
1. IRCE — bounds check elimination
2. LoopPredication — guard hoisting
3. SimplifyCFG — dead branch cleanup after IRCE/Predication
4. LoopInterchange — fix nested loop order for sequential access
5. LoopDistribute — break dependent loops for partial vectorization

**OptimizerLastEP (after vectorizer):**
6. LoopDataPrefetch — prefetch the vectorized access pattern
7. SeparateConstOffsetFromGEP — CSE of column base pointers
8. LoopFusion — fuse adjacent loops (conditional: only when
   LoopFusion pragma is active)

The IR cleanup pipeline (passes 1-3) runs BEFORE Polly to make Rust
IR Polly-friendly (remove bounds checks, panic branches, dead code).
Callback registration order at VectorizerStartEP determines execution
order — cleanup callbacks register before Polly's callbacks.

**Platform constraint:** `-Z llvm-plugins` only works on Linux
(dynamic LLVM). macOS/Windows ship static LLVM — no plugin loading.
Dev builds on macOS use `-C passes=` and `-C llvm-args=` (built-in
passes, no plugin). Release builds use the container workflow.

## Flag ownership by pragma

Every flag has exactly one owner pragma:

**LoopOptimization:**
- `-Z llvm-plugins=<path>/polka-passes.so`
- `-C llvm-args=-enable-loop-distribute`
- `-C llvm-args=-enable-loop-versioning-licm`
- `-C llvm-args=-enable-interleaved-mem-accesses`
- `-C llvm-args=-loop-prefetch-writes`
- `-C llvm-args=-enable-histogram-loop-vectorization`

**MathPeephole:**
- `-Z llvm-plugins=<path>/math-peephole.so`

**FastMath:**
- `-C llvm-args=-enable-unsafe-fp-math`
- `--cfg arvo_fast_math`

**Polly:**
- `-Z llvm-plugins=<path>/libPolly.so` (or `-C llvm-args=--polly`)

**ExpandedLto:**
- `lto = "fat"` (in generated config)
- `codegen-units = 1` (in generated config)

**Pgo (when profiles exist):**
- `-C profile-use=<path>/merged.profdata`
- `-C llvm-args=-vectorize-with-block-frequency`
- `-C llvm-args=-split-machine-functions`
- `-C llvm-args=-profile-guided-section-prefix`

**ParallelCodegen<N>:**
- `-Zthreads=N`

**SharedGenerics:**
- `-Zshare-generics=y`

**Wrapper implicit (always-on):**
- `-C target-cpu=native` when TARGET == HOST

Cargo profile settings (`panic = "abort"`, `strip`, `opt-level`)
remain in the consumer's Cargo.toml. NOT pragma-controlled.

## PGO + BOLT workflows (T4 §Q5)

**Release build 1 (no profiles):**
1. Cargo builds → binary produced
2. Post-build hook: static BOLT runs immediately (seconds, Linux ELF,
   reorders blocks/functions with static heuristics, 3-5% gain)
3. Background: runs benchmarks with PGO instrumentation + perf record.
   Generates `.profraw` + `perf.data`. Merges profiles.

**Release build 2+ (profiles exist):**
1. `configure()` finds `merged.profdata` → adds `-C profile-use=...`
   and PGO-complementary flags to generated config
2. Cargo recompiles with PGO (10-20% branch/inline improvement)
3. Post-build hook: profile-guided BOLT (additional 5-10%)
4. Background: re-profiles with optimised binary

**Profile staleness:**
- Profiles in `target/hilavitkutin-build/pgo/` (gitignored)
- `cargo clean` deletes them
- `configure()` records git commit hash when profiles generated
- If HEAD diverges >50 commits, emits `cargo:warning=stale profiles`
- Never refuses to build; stale profiles degrade gracefully

**Optimisation tiers (additive, automatic):**

| Tier | What | Benefit |
|---|---|---|
| Stock | nothing extra | baseline |
| Plugins | LLVM pass plugins via wrapper | loop optimization |
| Static BOLT | post-build reorder (Linux ELF) | 3-5% layout |
| PGO | profile-use from benchmarks | 10-20% branch/inline |
| Profile-guided BOLT | BOLT with profiling data | additional 5-10% |
| Machine function split | compile-time flag (all targets) | partial BOLT for macOS |

BOLT is Linux ELF only (no Mach-O). macOS gets
`-C llvm-args=-split-machine-functions` as partial alternative.

## Container workflow (T4 L596-663)

`-Z llvm-plugins` requires dynamic LLVM (Linux nightly only).
Dev builds on macOS: no plugins, use `-C passes=` and `-C llvm-args=`
for built-in passes. Release builds: Apple Container (Apache 2.0,
macOS Virtualization.framework, boots <1 second) running minimal
Debian with rustup nightly. Runs aarch64-linux natively on Apple
Silicon. No VM overhead, no Rosetta, no cross-compilation.

## Existing implementation

The following already exist in the repo:

- `scripts/llvm-passes/polka-passes.cpp` — the LLVM pass plugin

  LoopInterchange, LoopDistribute at VectorizerStartEP.
  LoopDataPrefetch, SeparateConstOffsetFromGEP at OptimizerLastEP.
  Compiles against LLVM headers inside the release container.

- `scripts/llvm-passes/math-peephole.cpp` — float math peephole
  rewrite plugin. Fired by the MathPeephole pragma.

- `scripts/release-container.sh` — Apple Container workflow (407
  lines). Persistent named container, first-time toolchain setup
  (rustup nightly + matching LLVM binary release downloaded
  automatically), hash-based plugin compilation skip, cargo build
  with auto-detected plugin `.so` loading, interactive shell mode.

These scripts are the runtime infrastructure that hilavitkutin-build's
pragma system drives. The `configure().run()` API generates the rustc
wrapper that references the plugin `.so` paths and injects the correct
flags per pragma.
