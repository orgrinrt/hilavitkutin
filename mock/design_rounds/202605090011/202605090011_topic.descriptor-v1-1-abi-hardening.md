# ExtensionDescriptor v1.1 ABI hardening

`ExtensionDescriptor` is the sole `#[repr(C)]` contract between an
extension and the host. Today's shape (`HOST_ABI_VERSION = 1`) was
flagged by the substrate-completion audit on three independent axes.
The round closes those axes by reshaping the descriptor into a v1.1
form gated by a literal magic tag, sized by a `descriptor_size` field
the host consults before reading any further fields, with all length
positions on platform-stable `u32` instead of pointer-width `USize`,
and with all length values bounded against an explicit ceiling before
the host reads any pointer-len pair as a slice. The `abi_version`
field is bumped from 1 to 2 (no shims; pre-1.0 deletion only). The
`#[export_extension]` proc-macro emits the new shape automatically;
extension authors recompile and pick up the change without source
edits. The cross-repo blast radius is exactly that: viola's plugin
fixtures (`viola-test-plugin-fixture`,
`viola-test-runner-fixture`) regenerate clean.

## Audit findings closed

The synthesis memo (2026-05-04) names three findings on the
plugin-host layer that this round addresses, all on the
`ExtensionDescriptor` shape:

- **F1 — pointer-width-dependent wire layout.** The current shape
  uses `USize` for `name_len`, `capabilities_len`, and
  `required_host_caps_len`. `USize` is repr-transparent over `usize`
  and therefore changes byte width depending on the build target
  (4 bytes on 32-bit, 8 bytes on 64-bit). The descriptor's literal
  byte layout differs across platforms. An extension built on a
  64-bit host loaded by a 32-bit host (or vice-versa) reads at the
  wrong offsets. Even on monomorphic deployments the ambiguity is a
  defect waiting to fire under unusual cross-compilation cases. The
  fix is to commit the lengths to a fixed wire width.
- **F2 — no `descriptor_size` field for forward evolution.** The
  current shape gives the host no way to detect that a loaded
  extension's descriptor was compiled against an older or newer
  hilavitkutin-extensions version. v2 cannot add fields without
  silently changing offsets that v1 hosts read into garbage. The fix
  is to put a `descriptor_size` field early in the layout so a host
  reading a future v2 descriptor knows to stop after the v1.1 prefix
  it understands, and so a host reading a past v0 descriptor (none
  exist; this is forward-looking) detects the mismatch.
- **F3 — unbounded-trust slice access.** Today's host code
  dereferences the descriptor's pointer-and-length pairs directly to
  build slices. Nothing constrains the length values. An extension
  (malicious or accidentally-broken) can advertise a pointer plus
  `name_len = u64::MAX` and the host attempts to materialise that
  slice. The fix is to validate every length field against a
  reasonable ceiling at descriptor-read time, before any pointer is
  followed.

## Decisions for this round

### D1. Magic tag at the very beginning of the descriptor

**Decision:** the descriptor's first field is `pub tag: u32`. The host
reads the tag before any other field. A tag mismatch surfaces the
new `ExtensionError::DescriptorTagMismatch { found: u32 }` and
aborts the load. The tag value is the literal `u32` little-endian
encoding of the four ASCII bytes `H`, `I`, `L`, `E`, which is
`0x454C4948` (the ASCII bytes laid out little-endian: `H = 0x48` at
the lowest address, then `I = 0x49`, then `L = 0x4C`, then `E =
0x45`). The constant ships as
`pub const EXTENSION_DESCRIPTOR_TAG: u32 = 0x454C4948;`.

Why ASCII "HILE": the proc-macro's emitted symbol prefix is
`__hilavitkutin_extension_descriptor`; the four-byte ASCII tag is
recognisable in a hex-dump of an extension's static rodata and
unambiguously identifies the descriptor as a hilavitkutin extension
descriptor (rather than a stray same-shape struct from a different
plugin framework). Using ASCII characters that fit in one `u32` is
the conventional pattern (`PE\0\0`, `\x7FELF`, `JFIF`, etc.).

The tag is the first field rather than coexisting with
`abi_version` because the tag is the gate: a wrong tag means the
host has no business attempting to interpret any other field,
including `abi_version`, since the layout itself is suspect.

### D2. `descriptor_size` second, before any other interpretation

**Decision:** the descriptor's second field is
`pub descriptor_size: u32`. It records the byte size of the
descriptor as the extension's compile-time `core::mem::size_of` of
the descriptor type. The host validates
`descriptor_size >= core::mem::size_of::<ExtensionDescriptor>()`
where the `size_of` on the right is the host's own. A smaller value
surfaces `ExtensionError::DescriptorSizeTooSmall { advertised, minimum }`.

The check is `>=` (not `==`) because forward-evolution is the point:
a v2 extension built against a future hilavitkutin-extensions adds
trailing fields, the v2 descriptor's `descriptor_size` is larger,
the v1.1 host validates the prefix it knows is intact and ignores
the trailing bytes. A v1.1 host reading a v1.1 descriptor sees
exact equality.

The descriptor_size field is named `descriptor_size` (not
`payload_size` or `header_size`) because it is the size of the
descriptor record itself, the entire `#[repr(C)]` struct, not of
any out-of-line storage the descriptor's pointers reach.

### D3. Bump `abi_version` from 1 to 2; delete v1 outright

**Decision:** `HOST_ABI_VERSION` becomes `AbiVersion(2)`. The pre-1.0
no-legacy-shims rule applies: there is no `AbiVersion(1)` deprecation
alias, no compat-read path that handles the old layout, no version-1
test fixture preserved as historical artefact. The v1 shape ceases
to exist in source. Extensions built against v1 fail load with
`ExtensionError::AbiVersionMismatch { expected: AbiVersion(2), got:
AbiVersion(1) }`.

The bump is to 2 (not 1.1) because semver pre-release suffixes do
not exist on `AbiVersion`'s `u32` representation. The integer is the
identifier; whatever round names this v1.1, the wire identifier is
the integer 2.

The field type changes from bare `pub abi_version: u32` to
`pub abi_version: AbiVersion`. `AbiVersion` is already
`#[repr(transparent)]` over `u32`, so the wire layout is identical,
but the host's read-path types stay typed (no bare-primitive
construction at the comparison site).

### D4. Length fields explicit `u32`

**Decision:** `name_len`, `capabilities_len`, and
`required_host_caps_len` all change from `USize` to bare `u32`. The
choice of `u32` (not `u64`) is deliberate: it is wire-stable, and
the implied ceiling of just under 4 billion entries is well above
any plausible extension's surface area. A `u64` would be wider on
the wire than necessary; a `u16` would cap real workloads at 65
535. `u32` is the stable middle.

The bare `u32` choice is consistent with how the existing
`abi_version: u32` field carries its `lint:allow(no-bare-numeric, ...)
tracked: #206` annotation. The same FFI-wire-primitive policy
applies. When arvo ships a `UWire<32>` newtype (per the
hilavitkutin-extensions BACKLOG entry), all three length fields plus
`abi_version`, `descriptor_size`, and `tag` flip in one pass.

The fields bear `lint:allow(arvo-types-only, no-bare-numeric,
no-public-raw-field) tracked: #206` annotations identical to the
existing pattern.

### D5. Bounds ceiling: hard cap at 1 048 576 entries

**Decision:** every length field is validated at descriptor-read
time against a `pub const MAX_DESCRIPTOR_LIST_LEN: u32 = 1 << 20;`
(1 048 576 entries). A value above the ceiling surfaces
`ExtensionError::DescriptorBoundsExceeded { field: &'static str,
len: u32 }`. The `field` is one of the literal strings `"name"`,
`"capabilities"`, or `"required_host_caps"`.

The ceiling is one million entries. The reasoning:

- The current host uses these lengths to materialise slices via
  `core::slice::from_raw_parts`. The 1 << 20 ceiling caps the slice
  byte size at `1_048_576 * size_of::<largest entry type>()` which
  is bounded by `size_of::<CapabilityEntry>` (an id `u64` plus a
  `*const c_void`, 16 bytes on 64-bit), giving 16 MiB worst case.
  That is well within any reasonable host's address-space slack.
- One million capability entries per extension is two-and-a-half
  orders of magnitude larger than viola's full plugin-API surface
  (the viola-plugin-abi crate ships three constants today). The
  ceiling is generous, not tight.
- 1 << 20 is a power of two that fits in `u32` with room. It is not
  load-bearing; if a future ecosystem needs more, the constant
  bumps in a follow-up round and the field width is already `u32`.

### D6. Bounds-check ordering; tag first, size second, version third, lengths last

**Decision:** the host's validation order is fixed:

1. Tag check. Wrong tag means the layout is not what is claimed;
   read no further.
2. `descriptor_size` check. With the layout established, ensure the
   host reads only what it understands.
3. `abi_version` check. The v1.1 layout is known sound up to here;
   a wrong version still aborts but the failure-mode is "incompatible
   version" rather than "garbage layout".
4. Length-field bounds checks against `MAX_DESCRIPTOR_LIST_LEN`,
   one for `name_len`, one for `capabilities_len`, one for
   `required_host_caps_len`. The first failure surfaces.
5. Required-host-capability check (the existing v1 step, unchanged).

The order is deliberate: each step builds confidence that the next
step's read is sound. Reordering, for example by checking
`abi_version` before `tag`, lets a wrong-tag descriptor pass the
version check by coincidence (the second u32 in the new layout
happens to fall in the version slot of the old layout, depending on
field offsets). Defensive ordering forecloses that.

### D7. No backwards compat; viola plugins recompile, no source edits

**Decision:** there is no v1 read path in the host. There is no
descriptor adapter that reads a v1 layout into a v1.1 in-memory
shape. Pre-1.0 deletion. Viola plugins (`viola-test-plugin-fixture`,
`viola-test-runner-fixture`) use `#[export_extension]`; the proc-
macro emits the new shape automatically; the cross-repo update is
viola's `cargo build` after viola's hilavitkutin-extensions git
ref bumps to the post-merge commit.

Viola does not import `ExtensionDescriptor` directly anywhere: the
14-file Capability count from the parked #382 round was on
`CapabilityId` and `CapabilityEntry`, both unchanged in this round.
Direct `ExtensionDescriptor` references in viola are limited to one
doc-comment in `viola-plugin-abi/src/vtable.rs` which references
`hilavitkutin_extensions::CapabilityEntry::vtable_ptr` (a
`CapabilityEntry` field, also unchanged).

The hilavitkutin-extensions BACKLOG already records the
"flip bare-primitive FFI-wire sites to `UWire<N>`" follow-up. This
round expands the bare-primitive set (adding `tag`,
`descriptor_size`, three explicit `u32` length fields) but the
flip-when-arvo-ships story is the same.

## What does NOT change

- `CapabilityId(u64)`: unchanged. (Vocabulary rename is the parked
  #382 round.)
- `CapabilityEntry { id, vtable_ptr }`: unchanged.
- `ExtensionVersion { major, minor, patch, _reserved }`: unchanged.
- `ExtensionAbiStatus`: unchanged.
- `DESCRIPTOR_SYMBOL`: unchanged. The symbol name stays
  `__hilavitkutin_extension_descriptor`.
- `ExtensionHost::load` signature: unchanged. The new validation
  steps are internal.
- `ExtensionRequirement` / `policy_translate`: unchanged.
