**Date:** 2026-04-20

**Blocks:** #71 (arvo sweep), #72 (hilavitkutin sweep), #73 (clause sweep).

## Purpose

Ship the missing contract layer that lets public APIs emit items
without returning heap collections. Today, signatures like
`parse() -> Result<Ast, Vec<SyntaxError>>` and
`CodegenTarget::emit() -> Outcome<CodegenArtifact { bytes: Vec<u8>, diagnostics: Vec<Diagnostic> }, _>`
exist in shipped clause code. The sweeps need to reshape them to take
trait-bound sinks and return scalar outcomes. Those traits do not
exist yet in this workspace. This round creates them.

## Current state

| `work_unit` | `WorkUnit<Schedule>`, `Always`, `On<V>` |

These are all **pipeline-execution contracts**. A WU body talks to
`Context<R, W>` through them.

### What is missing

No trait covers "emit items to a consumer-provided destination" for
uses **outside** pipeline execution. Clause's parser, validator, and
codegen have no sink trait to take, so the polish rounds (now in the
#74 revisit list) returned `Vec<T>`. That pattern must not cross the
API boundary.

### Heritage — saalis

`saalis-contracts` (in `~/Dev/saalis/mock/crates/`) split trait
contracts into `shared/` + `host/` + `client/` modules. It defined
domain-specific provider APIs (`ConnectionApi`, `WriterApi`,
`QueryApi`, `RowWriter<M>`) that all follow the same pattern: a
consumer-provided type is the storage; the trait is the contract.

Saalis did not ship a generic `Sink<T>` or `ByteEmitter`. Its
collection-facing traits were always domain-typed. The decision for
us is whether to follow the generic-core path or the domain-typed
path (see Q2).

### Constraints (hard)

- `#![no_std]`, no `alloc`, no `dyn`, no `TypeId`.
- Public signatures must obey the forbidden-vocabulary list: no
  `Option`/`Result`/`Vec`/`String`/`usize`/`f32`/`bool` in pub
  positions. Use `notko::Maybe` / `Outcome`, `arvo::USize`,
  `arvo::FastFloat`, `arvo_bits::Bool`.
- No blanket `Vec<T>` or `Seq<T, N>` impls in the API crate — those
  belong in consumer test-helpers (#81).
- `hilavitkutin-api` must not depend on `hilavitkutin`,
  `hilavitkutin-ctx`, `hilavitkutin-persistence`, `hilavitkutin-str`.

## The designed contract family (proposed vocabulary)

For consistent naming across the crate, every trait in this family
represents a **capability the caller provides** that the callee
drives. Each is trait-generic over the item type.

1. **`Collector<T>`** — generic "receive items". `push(&mut self, item: T)`.
   The atom. Any producer emits items to a collector.

2. **`DiagnosticSink<E>`** — alias-style specialisation:
   `trait DiagnosticSink<E>: Collector<E> { ... }` or an independent
   trait (Q2). Semantics identical to `Collector<E>` but the name
   signals diagnostic/error domain at call sites.

3. **`ByteEmitter`** — `push_byte`, `push_bytes(&[u8])`. Separate
   from `Collector<u8>` because it carries byte-specific optimisation
   (bulk `push_bytes` is not naturally expressible via single-item push).

4. **`Codec`** family — Encoder + Decoder pair for serialisation.
   Consumes / produces `ByteEmitter` + `&[u8]`. Shape TBD (Q4).

The task #83 title calls out four terms (sink / buffer / collector / codec).
Mapping:
- "sink" → `Collector<T>` + `DiagnosticSink<E>` (generic + named alias)
- "buffer" → no separate trait; buffers are implementations of the
  above (a Vec-backed buffer, a ring-backed buffer, a null sink)
- "collector" → `Collector<T>` (the base)
- "codec" → `Encoder<T>` + `Decoder<T>`

Pre-decision: buffer is NOT a separate trait in this round. A
bounded buffer is whatever implements `Collector<T>` with a `capacity()`
query. See Q5.

## Open questions (to be answered by the user, one at a time)

### Q1 — Scope envelope for this round

How broad should this round be?

- **Minimal**: ship only `Collector<T>` + `DiagnosticSink<E>` +
  `ByteEmitter`. Defer the codec pair to a later round when clause
  bytecode serialisation needs it.
- **Canonical**: ship all four families — `Collector<T>`,
  `DiagnosticSink<E>`, `ByteEmitter`, `Encoder<T>` + `Decoder<T>`.
  Covers every near-term consumer need the sweeps may surface.
- **Maximal**: canonical plus sink combinators (`NullSink`, `CountingSink`,
  `TeeSink`) and length/capacity query supertraits. Test ergonomics
  covered in the same round.

### Q2 — Root abstraction relationship

How do the traits relate?

- **Independent**: each trait stands alone. `DiagnosticSink<E>` has
  its own `push`, `ByteEmitter` has `push_byte`, etc. No supertrait.
- **`Collector<T>` as root**: `DiagnosticSink<E>: Collector<E>`;
  `ByteEmitter: Collector<u8>` + bulk-write extension. One
  base abstraction.
- **Capability composition**: smaller traits like `Push<T>`, `BulkPush<T>`,
  `Capacity`, `Len`. Larger traits are bounds over those.

### Q3 — Push-fallibility

Is `push` infallible or fallible?

- **Infallible**: `fn push(&mut self, item: T)`. Overflow / full-sink
  handling is the implementor's problem (discard, panic in debug,
  etc.). Simple, matches `core::iter::Extend`-style ergonomics.
- **Fallible with dedicated error**: `fn push(&mut self, item: T) -> Outcome<(), SinkFull>`.
  Explicit overflow signalling at every call site.
- **Fallible returning a capacity hint**: `fn push(&mut self, item: T) -> Maybe<Cap>`
  — `Maybe::Isnt` means "refused"; `Maybe::Is(n)` means "n slots free".

### Q4 — Codec shape

Assuming Q1 includes codecs, what's the shape?

- **Trait pair**: `Encoder<T> { fn encode<B: ByteEmitter>(&self, v: &T, out: &mut B); }`
  + `Decoder<T> { fn decode<B: &[u8]>(bytes: B) -> Outcome<T, DecodeError>; }`.
- **Single bidirectional**: `trait Codec<T> { fn encode ...; fn decode ...; }`.
  One trait per type.
- **Defer**: skip codec this round (pushes scope to minimal).

### Q5 — Length / capacity queries

Do all sinks expose counts?

- **No**: sinks are push-only. Implementors can add their own methods.
- **Yes, required**: every sink must provide `len() -> USize`.
  Consumers can branch on "did we emit anything".
- **Yes, optional supertrait**: `Collector<T>` is push-only;
  `BoundedCollector<T>: Collector<T> { fn len() -> USize; fn capacity() -> Cap; }`
  for implementations that know.

### Q6 — Module organisation

Where in `hilavitkutin-api` do these traits live?

- **Single `contract` module**: all four families live in
  `hilavitkutin-api::contract::{sink, codec, ...}`.
- **One file per family**: `src/{collector,sink,emitter,codec}.rs`
  alongside existing `src/{access,context,...}.rs`.
- **Under `api` subdir + `pipeline` subdir split**: existing
  pipeline-execution traits move to `pipeline/`, new consumer
  traits land in `contract/`. More reshuffling; clarifies intent.

### Q7 — Default method on `Collector<T>` for slices

Should `Collector<T>` provide `push_slice(&[T]) where T: Copy`
as a default method?

- **Yes**: default iterates + pushes one at a time. Implementors
  override for bulk optimisation (`ByteEmitter` style).
- **No**: push one at a time; bulk is a separate capability that
  byte-specific traits carry.

## Decisions recorded

### Q1 — scope: **Maximal**

Ship:
- `Collector<T>`, `DiagnosticSink<E>`, `ByteEmitter`.
- `Encoder<T>` + `Decoder<T>` codec pair.
- Sink combinators: `NullSink`, `CountingSink`, `TeeSink`.
- Length / capacity query supertraits.
- Tuple blanket impls for pairs/triples (so a consumer can pass
  `(&mut diagnostics, &mut bytes)` where each implements the right
  trait).

Larger round; justified because it ships a complete, testable
surface in one pass and avoids a "test-ergonomics follow-up" round
that would mirror the #50 deferral pattern.

### Q2 — root abstraction: **Capability composition**

Small capability atoms:

```rust
pub trait Push<T>      { fn push(&mut self, item: T); }
pub trait BulkPush<T>  { fn push_bulk(&mut self, items: &[T]) where T: Copy; }
pub trait Len          { fn len(&self) -> USize; fn is_empty(&self) -> Bool { ... } }
pub trait Capacity     { fn capacity(&self) -> Cap; fn remaining(&self) -> Cap { ... } }
```

Named traits are bounds over atoms. Blanket impl turns each name into
a shorthand:

```rust
pub trait Collector<T>: Push<T> {}
impl<T, S: Push<T>> Collector<T> for S {}

pub trait DiagnosticSink<E>: Push<E> + Len {}
impl<E, S: Push<E> + Len> DiagnosticSink<E> for S {}

pub trait ByteEmitter: Push<u8> + BulkPush<u8> {}
impl<S: Push<u8> + BulkPush<u8>> ByteEmitter for S {}
```

Combinators fall out clean:
- `NullSink` — unit struct, implements `Push<T>` for all `T` (body empty).
- `CountingSink` — wraps a `USize` counter, implements `Push<T>` + `Len`.
- `TeeSink<A, B>` — `impl<T, A: Push<T>, B: Push<T>> Push<T> for TeeSink<A, B>`.

Trade-off: slightly more trait names to read, much more composition.
The capability-composition choice pushes Q3's "push fallibility" into
a capability-specific decision (infallible `Push<T>` vs. bounded
`BoundedPush<T>` may coexist).

### Q3 — push fallibility: **Infallible `Push<T>` + separate `BoundedPush<T>`**

```rust
pub trait Push<T> {
    fn push(&mut self, item: T);
}

pub trait BoundedPush<T>: Push<T> {
    /// Attempt a push. `Outcome::Err(Full)` when the sink refuses.
    fn try_push(&mut self, item: T) -> Outcome<(), Full>;

    /// Remaining capacity in elements. `0` means the next `push`
    /// call's behaviour is impl-defined (discard / panic / wrap);
    /// use `try_push` to get a refusal instead.
    fn remaining(&self) -> Cap;
}

/// Marker type returned by `BoundedPush::try_push` on refusal.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Full;
```

Callers that don't care about overflow bind to `Push<T>`. Callers
that need overflow-safety bind to `BoundedPush<T>`. Unbounded sinks
only implement `Push<T>`; bounded sinks implement both. `try_push`
failure never silently drops data.

### Q4 — codec shape: **Stateful incremental + oneshot extension**

Primary form is chunked:

```rust
pub trait Encoder<T> {
    /// Consume one value and emit bytes. May buffer internally.
    fn feed<B: ByteEmitter>(&mut self, v: &T, out: &mut B);

    /// Flush any remaining framing / trailer bytes. Consumes self.
    fn finish<B: ByteEmitter>(self, out: &mut B);
}

pub trait Decoder<T> {
    /// Feed a chunk. Emits decoded values to `out`. Returns unconsumed
    /// bytes so a buffered decoder can either retain them internally
    /// OR return them to the caller depending on the impl contract.
    fn feed<'a, S>(&mut self, chunk: &'a [u8], out: &mut S)
        -> Outcome<&'a [u8], DecodeError>
    where
        S: Push<T>;

    /// Finalise. Error if the decoder carries an unfinished frame.
    fn finish(self) -> Outcome<(), DecodeError>;
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
#[non_exhaustive]
pub enum DecodeError {
    /// Input ended mid-value.
    Truncated,
    /// Input bytes did not match the expected encoding.
    Invalid,
    /// Framed container indicates more bytes than available.
    OverLength,
}
```

Oneshot adapters as `Ext` traits with blanket impls:

```rust
pub trait EncoderExt<T>: Encoder<T> + Sized {
    /// All-at-once convenience: feed one value then finish.
    fn encode_one<B: ByteEmitter>(mut self, v: &T, out: &mut B) {
        self.feed(v, out);
        self.finish(out);
    }
}
impl<T, E: Encoder<T>> EncoderExt<T> for E {}

pub trait DecoderExt<T>: Decoder<T> + Sized {
    /// All-at-once convenience: feed a complete input then finish.
    /// Errors if any bytes remain unconsumed after `feed`.
    fn decode_all<S: Push<T>>(
        mut self,
        bytes: &[u8],
        out: &mut S,
    ) -> Outcome<(), DecodeError> {
        let rest = self.feed(bytes, out)?;
        if !rest.is_empty() {
            return Outcome::Err(DecodeError::OverLength);
        }
        self.finish()
    }
}
impl<T, D: Decoder<T>> DecoderExt<T> for D {}
```

Rationale: stateful incremental is strictly more capable than
all-at-once. Streaming consumers (clause bytecode loader reading
chunks, future persistence layer) use `feed`/`finish` directly.
All-at-once consumers (simple in-memory tests, one-shot serialise)
use the `Ext` methods which compose trivially. One trait pair, two
ergonomic paths.

## Finalised design sketch

### `capability.rs`

```rust
use arvo::newtype::USize;
use arvo_bits::Bool;
use notko::Outcome;

/// Receive one item by value. Infallible; overflow is the
/// implementor's concern. See `BoundedPush<T>` for refusal.
pub trait Push<T> {
    fn push(&mut self, item: T);
}

/// Receive a slice of `Copy` items in one call.
///
/// Default implementation pushes per-item; implementors with a
/// bulk-optimised path (e.g., byte emitters with memcpy) override.
pub trait BulkPush<T>: Push<T> {
    fn push_bulk(&mut self, items: &[T])
    where
        T: Copy,
    {
        for item in items {
            self.push(*item);
        }
    }
}

/// Report the current item count.
pub trait Len {
    fn len(&self) -> USize;

    fn is_empty(&self) -> Bool {
        Bool::new(self.len() == USize::ZERO)
    }
}

/// Report total and remaining item capacity.
pub trait Capacity {
    fn capacity(&self) -> USize;
    fn remaining(&self) -> USize;
}

/// Overflow-aware push. `try_push` returns `Outcome::Err(Full)` when
/// the sink refuses; implementors MUST NOT silently drop items.
pub trait BoundedPush<T>: Push<T> + Capacity {
    fn try_push(&mut self, item: T) -> Outcome<(), Full>;
}

/// Marker returned by `BoundedPush::try_push` on refusal.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Full;
```

### `sink.rs`

```rust
use arvo::newtype::USize;
use core::marker::PhantomData;

use crate::capability::{BulkPush, Len, Push};

/// Intent-signalling alias: "this receives items of type T".
pub trait Collector<T>: Push<T> {}
impl<T, S: Push<T> + ?Sized> Collector<T> for S {}

/// Push-with-count: sinks that know how many they've received.
pub trait DiagnosticSink<E>: Push<E> + Len {}
impl<E, S: Push<E> + Len + ?Sized> DiagnosticSink<E> for S {}

/// Byte-stream: per-byte push plus bulk-write extension.
pub trait ByteEmitter: Push<u8> + BulkPush<u8> {}
impl<S: Push<u8> + BulkPush<u8> + ?Sized> ByteEmitter for S {}

/// Discards everything pushed to it.
#[derive(Debug, Clone, Copy, Default)]
pub struct NullSink;

impl<T> Push<T> for NullSink {
    #[inline(always)]
    fn push(&mut self, _: T) {}
}
impl<T> BulkPush<T> for NullSink {
    #[inline(always)]
    fn push_bulk(&mut self, _: &[T]) where T: Copy {}
}

/// Counts pushes but stores nothing.
#[derive(Debug, Clone, Copy, Default)]
pub struct CountingSink<T> {
    count: USize,
    _m: PhantomData<fn(T)>,
}

impl<T> Push<T> for CountingSink<T> {
    #[inline(always)]
    fn push(&mut self, _: T) {
        self.count = self.count + USize::ONE;
    }
}
impl<T> Len for CountingSink<T> {
    #[inline(always)]
    fn len(&self) -> USize { self.count }
}

/// Fans pushes out to two downstream sinks.
pub struct TeeSink<'a, A: ?Sized, B: ?Sized> {
    pub a: &'a mut A,
    pub b: &'a mut B,
}

impl<'a, T, A, B> Push<T> for TeeSink<'a, A, B>
where
    T: Clone,
    A: Push<T> + ?Sized,
    B: Push<T> + ?Sized,
{
    #[inline(always)]
    fn push(&mut self, item: T) {
        self.a.push(item.clone());
        self.b.push(item);
    }
}

// Tuple blanket impls (pairs and triples) so a consumer can pass
// `(&mut diags, &mut bytes)` as "one sink" satisfying two Push<T>
// slots via destructuring. Concretely — these enable the pattern
// `emit<S: Push<Diag> + Push<u8>>(s: &mut S, ...)` to be satisfied
// by disjoint sinks. Declared here; detailed impls in the doc CL.
```

### `codec.rs`

```rust
use notko::Outcome;

use crate::capability::Push;
use crate::sink::ByteEmitter;

/// Chunked encoder. Feed values; finish to flush framing.
pub trait Encoder<T> {
    fn feed<B: ByteEmitter>(&mut self, v: &T, out: &mut B);
    fn finish<B: ByteEmitter>(self, out: &mut B);
}

/// Chunked decoder. Feed bytes; produces values via `out`.
/// `feed` returns unconsumed bytes so the caller can either buffer
/// them (streaming) or treat leftovers as an error (oneshot).
pub trait Decoder<T> {
    fn feed<'a, S>(&mut self, chunk: &'a [u8], out: &mut S)
        -> Outcome<&'a [u8], DecodeError>
    where
        S: Push<T>;

    fn finish(self) -> Outcome<(), DecodeError>;
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
#[non_exhaustive]
pub enum DecodeError {
    /// Input ended mid-value.
    Truncated,
    /// Input bytes did not match expected encoding.
    Invalid,
    /// Framed container indicates more bytes than available.
    OverLength,
}

/// Oneshot encode helper. Feeds one value then finishes.
pub trait EncoderExt<T>: Encoder<T> + Sized {
    fn encode_one<B: ByteEmitter>(mut self, v: &T, out: &mut B) {
        self.feed(v, out);
        self.finish(out);
    }
}
impl<T, E: Encoder<T>> EncoderExt<T> for E {}

/// Oneshot decode helper. Feeds a complete input; errors if any
/// bytes remain unconsumed.
pub trait DecoderExt<T>: Decoder<T> + Sized {
    fn decode_all<S>(mut self, bytes: &[u8], out: &mut S)
        -> Outcome<(), DecodeError>
    where
        S: Push<T>,
    {
        let rest = self.feed(bytes, out)?;
        if !rest.is_empty() {
            return Outcome::Err(DecodeError::OverLength);
        }
        self.finish()
    }
}
impl<T, D: Decoder<T>> DecoderExt<T> for D {}
```

## Remaining mechanical decisions (auto-decided; flag during DOC CL if wrong)

1. **Tuple blanket impls** — arity 2 and 3 only. Pattern: `(&mut A, &mut B)`
   where `A: Push<T>` and `B: Push<U>` gives you a pair that satisfies
   disjoint Push requirements. Larger arities not ship now; extend in
   a later round if consumers demand.
2. **BulkPush<T> default** — pushes per-item. Implementors SHOULD override.
3. **Cap vs USize** — runtime counts use `USize` (arvo newtype). `Cap` is
   reserved for const-generic capacity (e.g., `Seq<T, N: Cap>`).
4. **DecodeError shape** — `#[non_exhaustive]` enum, Copy. Consumers may
   match and extend will not break them; future variants (e.g., `Checksum`)
   land in a compatible bump.
5. **Sized bounds** — explicit `?Sized` on blanket impls (`for S where S: Push<T> + ?Sized`)
   so `&mut dyn Push<T>`-style uses compose even though we don't use dyn.
6. **Full** — unit struct, not enum (one refusal reason). If future
   variants land, upgrade to `#[non_exhaustive]` enum.
7. **Clone bound on `TeeSink::push`** — `T: Clone` required. For `Copy`
   types the clone is free; for non-Copy consumers, tee requires explicit
   clone because it distributes the item across two downstream sinks.
