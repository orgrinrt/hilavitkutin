A new subsection `### Diagnostic coverage across the public surface` lists the traits that gained `#[diagnostic::on_unimplemented]` in this round. Body text:

```
The following public traits carry `#[diagnostic::on_unimplemented]`
as of round 202605082321 in addition to the previously-shipped
`Contains<S>` and `ContainsAll<L>`:

- Type-level: `Concat<L>`, `WorkUnitBundle`, `StoreBundle`,
  `ColumnValue`, `Replaceable`.
- Sealed substrate axes: `SchedulingHint`, `UrgencyValue`,
  `DivisibilityValue`, `SignificanceValue`, `Depth`. The diagnostics
  on the sealed axes name the available substrate-provided ZST
  markers so consumers see the menu in the error trail.
- Capability layer: `Push<T>`, `BulkPush<T>`, `Len`, `Capacity`,
  `BoundedPush<T>`, `Collector<T>`, `DiagnosticSink<E>`,
  `ByteEmitter`.
- Codec: `Encoder<T>`, `Decoder<T>`. The extension traits
  `EncoderExt<T>` / `DecoderExt<T>` are not annotated; their
  default-method default impls rarely surface raw bound failures.
- Platform contracts: `MemoryProviderApi`, `ThreadPoolApi`,
  `ClockApi`, `HasMemoryProvider`, `HasThreadPool`, `HasClock`.
- Provider API: `ColumnReaderApi<R>`, `ColumnWriterApi<W>`,
  `ResourceProviderApi<R>`, `VirtualFirerApi<W>`, `EachApi<R, W>`,
  `BatchApi<R, W>`, `ReduceApi<R, W>`.
- WorkUnit: `WorkUnit<Schedule>`.

Each diagnostic carries a one-line `message` plus, where the fix
points at a specific call, a `note` naming the remediation. The
attribute changes nothing about trait semantics; consumers see
better error trails on identical substrate behaviour.
```

### Replace "### Recursion limit" subsection

Current text (lines 422-430 of the file as of branch HEAD `82c681f`) covers the `#![recursion_limit = "512"]` recommendation. Replace with the expanded text below to reflect the diagnostic-guided manual-attribute approach (the macro design from round 202605042200 cannot work in current rustc; sketch S1 finding 2026-05-08).

Replacement text:

```
### Recursion limit

The substrate's typestate machinery walks cons-list recursions at
trait-solver time. Default rustc `recursion_limit = 128`
accommodates apps up to roughly 30 WUs by 30 stores. The
`hilavitkutin-api`, `hilavitkutin`, and `hilavitkutin-kit` crates
declare `#![recursion_limit = "512"]` internally for their own
machinery. Consumer crates approaching the threshold (4+ nested Kit
layers, 30+ flat WUs across the build) declare:

```rust
#![recursion_limit = "1024"]
```

at their own crate root.

Earlier substrate designs proposed shipping a
`recursion_limit_for_kits!()` macro that expanded to the inner
attribute. Current rustc rejects macro-expansion of crate-level
inner attributes ("an inner attribute is not permitted in this
context"); the design retreats to documenting the directive
plainly. The trait diagnostics on `Contains`, `ContainsAll`, `Kit`,
and `WorkUnit` mention the recursion-limit directive in their
notes when the bound failure could plausibly stem from depth.

When `.build()` reports `overflow evaluating the requirement`, the
fix is the inner attribute above; no other configuration is
needed.
```

### Add new section "### `prelude` module"

Add immediately after the new "Diagnostic coverage" subsection. Body:

```
### `prelude` module

`hilavitkutin_api::prelude` re-exports the consumer-facing
substrate primitives so a single glob import covers the typical
WorkUnit + Kit author's needs:

```rust
use hilavitkutin_api::prelude::*;

// Now in scope:
//   Empty, Cons, AccessSet, Contains, ContainsAll, Concat
//   Resource, Column, Virtual, StoreBundle, Replaceable
//   WorkUnit, WorkUnitBundle, Always, On
//   read!, write!
```

Provider-side and platform-contract traits (`HasColumnReader`,
`MemoryProviderApi`, etc) stay out of the prelude. Consumers that
need them name them directly so import sets stay
self-documenting at provider-tuple boundaries.
```
