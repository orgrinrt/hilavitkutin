## Purpose

Land the type-surface skeleton for hilavitkutin-build per its
301-line DESIGN. Full implementation of the LLVM pass plugin, the
rustc workspace wrapper script generation, and the PGO/BOLT
integration tiers defers to follow-up rounds — each of those is
its own engineering chunk with external tooling (LLVM, BOLT)
that warrants dedicated focus.

What ships this round:

1. **Pragma enum** — every pragma from DESIGN Q4f table:
   `LoopOptimization`, `Polly`, `MathPeephole`, `FastMath`,
   `ExpandedLto`, `Pgo`, `Bolt`, `Profiling`, `BuildStd`,
   `ParallelCodegen(n)`, `SharedGenerics`, `LoopFusion`,
   `MimallocAllocator`. Composable as a `PragmaSet`.
2. **Profile enum** — `Dev`, `DevOpt`, `Release`, `Profiling`,
   `Ci`. Each with a default-pragma bundle.
3. **Axis enums** — `Target`, `Tier`, `Passes` per DESIGN's three
   axes from which rustc flags compute.
4. **`BuildConfig`** — the composition point: profile + pragma
   overrides + detected target / tier info.
5. **`bootstrap_from_buildscript()`** function — the one
   consumers' `build.rs` calls. Stub: reads `CARGO_CFG_*`, emits
   `arvo_fast_math` cfg when FastMath is active, writes a stub
   `hilavitkutin-config.toml` to `$OUT_DIR`. Full rustc-wrapper
   generation deferred.
6. **`PragmaRequirement` matrix** — which pragmas require which
   external tooling (Polly needs Polly-enabled LLVM; Pgo needs
   profraw+profdata; Bolt needs llvm-bolt).
7. **`compile_error!` guard helpers** — DESIGN Q2 says
   hilavitkutin-build doesn't invent custom cargo features; it
   uses `compile_error!` with helpful messages when ISA / vendor
   / arch mismatches break compilation. Ship the helper macros
   even though the full mapping table lands in a follow-up round.

## Deferred to follow-up rounds

- Actual rustc workspace wrapper shell script generation (DESIGN
  Q4 part 3). The wrapper inspects `CARGO_CFG_*` at rustc
  invocation time and computes flags; authoring that bash / PS1
  code lives in its own round after the pragma types are stable.
- LLVM pass plugin C++ source (~40 lines per DESIGN). Its own
  round; delivery is a pre-built dylib that ships alongside the
  toolchain.
- PGO / BOLT tier integration (DESIGN Q5). Each tier is an
  independent round with its own measurement + verification story.
- `hilavitkutin-config.toml` full schema — skeleton writes a
  placeholder; follow-up round ships the real pragma-serialisation
  format.
- Wiring the `hilavitkutin-build` dep into consumer build scripts.
  Every hilavitkutin crate's `build.rs` currently calls
  `mockspace::bootstrap::bootstrap_from_buildscript()`; the
  migration to ALSO calling `hilavitkutin_build::bootstrap` is a
  separate infra round (or lands piecewise as the build layer
  proves valuable).

All deferred items captured in the new `BACKLOG.md.tmpl`.
