## Decisions locked (mostly derived from prior topics)

### Axis A. Morsel boundary computation

Already locked by Topic 3 step 9 (`compute_morsel_size`). Inputs: SIMD-width-aware base size from arvo-strategy + per-fiber working-set size + L1 write budget (Topic 5 M10 amendment: minus adapt sidecar). Output: per-fiber `morsel_size: USize` baked into ExecutionPlan at plan time. Worker reads from CoreProgram, not recomputed each loop.

### Axis B. Record iteration shape

Forward-only by default. Topic 3 axis E adds head+tail convergence (`Convergence` struct already in `mock/crates/hilavitkutin/src/thread/convergence.rs`) for commutative + single-trunk-phase + threshold-eligible fibers. Codegen materialises dual-thread program; the morsel loop body is symmetric (one thread iterates from head index forward, the other from tail backward).

### Axis C. Micro-morsel boundary criterion

A morsel is the cache-friendly window over a record range; a **micro-morsel** is a sub-morsel sync point inside the morsel loop where workers cross-validate progress before continuing. The criterion: micro-morsel boundary fires every `MICRO_MORSEL_INTERVAL` records OR at morsel end, whichever comes first.

`MICRO_MORSEL_INTERVAL` default = 64 records. Rationale: matches typical L1 cache line stride (8 records of 8-byte fields = 64 bytes = one cache line), so the sync point happens at a natural cache-eviction boundary anyway.

### Axis D. Per-record vs per-morsel WU dispatch granularity

Per-morsel for column-shaped WUs (the common case): the WU's `execute` receives a `Morsel<T>` window, iterates records itself. This matches Topic 4's dispatch codegen — WU body sees the slice, vectorises freely, branches predictably.

Per-record dispatch is the explicit opt-in shape for WUs that can't vectorise (e.g., resource-only WUs that read from Resource<T> and don't iterate over a Column<T>). Detection inferred from AccessSet shape per Topic 3 axis D ("morsel-vs-scalar dispatch inferred per-WU from AccessSet shape").

### Axis E. FiberOutput commit shape

Already locked by Topic 3 S3: codegen emits `dmb st` (aarch64) / `sfence` (x86_64) between the morsel loop's final `stnp` writes to FiberOutput columns and the progress-counter Release store. The morsel loop body itself uses regular stores; the fence + Release-counter combination publishes the morsel's writes atomically from the consumer's perspective.

### Axis F. Sync point implementation

Two flavours:

- **Inline sync** (micro-morsel boundary inside the morsel loop): `core::hint::spin_loop` budget then proceed. No futex. Workers race-ahead is acceptable inside a morsel; the inline sync only catches gross drift. Implementation: a per-fiber `AtomicUsize` "max progress" counter; each micro-morsel boundary, owning worker writes its progress with Release, peer worker reads with Acquire; if peer is behind by more than `MAX_DRIFT_RECORDS = 32`, spin briefly. No park inside morsel loop.

- **Phase barrier sync** (between morsels-of-different-phases): already locked by Topic 6 axis I (centralised atomic counter + futex-wait via `PoolFrame.phase_arrived`). Workers park between phases, not between morsels-within-phase.

### Axis G. Morsel cache-line alignment

Already covered by Topic 3 step 12 + the arena 64-byte alignment invariant from Round 2 audit. FiberOutput buffers are 64-byte aligned at arena alloc time; morsel writes land on aligned addresses; stnp is sound.

## User-attention call (autonomous, can be revisited)

### `MICRO_MORSEL_INTERVAL = 64` and `MAX_DRIFT_RECORDS = 32`

These are concrete defaults I picked autonomously. Both have natural rationales (cache-line boundary + half-morsel drift tolerance) but neither is bench-validated yet. Either can shift by ~2x without breaking the design. If a future bench shows different optimums, swap the constants without redesign.

Specifically, `MICRO_MORSEL_INTERVAL` could be tuned per-CoreClass (P-cores might tolerate longer intervals since they retire records faster); for v1 it's a single pool-wide constant.
