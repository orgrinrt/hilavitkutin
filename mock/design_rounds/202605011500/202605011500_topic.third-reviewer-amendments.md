## Findings the third reviewer caught (and all of us missed)

### A. AccessSet works only by structural coincidence

Original topic line 53: "`WuSatisfied<A>` stays per-arity 0..=12 because `A` is the consumer's declared `Read` / `Write` flat tuple. 12 stores per WU is comfortable headroom; the cap there doesn't bite real workloads."

Reviewer pushback: the dismissal was wrong-shape. A vehje typecheck WU legitimately reads from 16+ stores (Interner, TypeEnv, ConstEnv, ImportTable, ScopeStack, SealedRegistry, OrphanRules, Definitions, Bodies, Mir, Spans, ResolvedTypes, NameTables, Macros, Diagnostic, Errors). 13+ stores per WU is normal, not a smell.

### B. Kit::Output is not structurally constrained

A buggy Kit could declare `type Output = SchedulerBuilder<(), ()>` and silently wipe prior registrations. The reviewer flagged this as a soundness foothole, not just an ergonomics gap.

### C. Plan-stage code needs accurate depth

`AccessSet::LEN` reports immediate-tuple-arity (always 2 for cons-list cells). The reviewer noted this is "silently wrong" for cons-list shapes and asked for explicit declaration of intent. After user discussion: plan-stage code IS load-bearing for follow-up work; deferring a `Depth` trait pushes future complexity that we can address now while in the right thematic context.

### D. rustc trait-solver depth limits

Default `recursion_limit = 128` won't accommodate 50-WU × 50-store apps. Need `#![recursion_limit = "512"]` declared somewhere consumer-visible.

### 1. WuSatisfied uncap via `read!` / `write!` macros

Ship two declarative macros in `hilavitkutin-api` that convert flat-tuple syntax to cons-list shape:

```rust
#[macro_export]
macro_rules! read {
    () => { () };
    ($T:ty $(,)?) => { ($T, ()) };
    ($T:ty, $($rest:ty),+ $(,)?) => { ($T, $crate::read!($($rest),+)) };
}

#[macro_export]
macro_rules! write { /* identical shape */ }
```

Replace per-arity `WuSatisfied<A>` impls (0..=12) with two recursive impls:

```rust
impl<S: AccessSet> WuSatisfied<()> for S {}
impl<S, H: 'static, R> WuSatisfied<(H, R)> for S
where S: Contains<H> + AccessSet + WuSatisfied<R>, R: AccessSet,
{}
```

Consumer migration: `type Read = (Resource<X>, Column<Y>);` becomes `type Read = read![Resource<X>, Column<Y>];`. Single-store case `read![Resource<X>]` works the same.

The migration touches `mock/crates/hilavitkutin-api/tests/work_unit.rs` (1 site) and `mock/crates/hilavitkutin/tests/scheduler_builder.rs` (3 sites in Stub-shim WU impls, all introduced this PR).

### 2. Kit::Output constrained via `BuilderExtending<B>`

New sealed trait in `hilavitkutin-api`:

```rust
mod extending_sealed { pub trait Sealed<B> {} }

#[allow(private_bounds)]
pub trait BuilderExtending<B>: extending_sealed::Sealed<B> {}

impl<Wus, Stores, NewStores> BuilderExtending<SchedulerBuilder<MAX_*, Wus, Stores>>
    for SchedulerBuilder<MAX_*, Wus, NewStores>
where
    Wus: AccessSet,
    Stores: AccessSet,
    NewStores: AccessSet + WuSatisfied<Stores>,
{}
```

The `Wus` parameter is identical between input and output (Kits cannot add WUs, only stores). `NewStores: WuSatisfied<Stores>` proves every store from `Stores` is still in `NewStores`. Engine's `add_kit` adds a where-clause:

```rust
pub fn add_kit<K>(self, k: K) -> K::Output
where
    K: Kit<Self>,
    K::Output: BuilderExtending<Self>,
{ k.install(self) }
```

A buggy Kit returning `SchedulerBuilder<(), ()>` fails the `BuilderExtending` bound at the call site with a clean compile error.

### 3. Depth trait (sealed, cons-list-shape-only)

```rust
mod depth_sealed { pub trait Sealed {} }

#[allow(private_bounds)]
pub trait Depth: depth_sealed::Sealed {
    const D: USize;
}

impl depth_sealed::Sealed for () {}
impl<H, R: Depth> depth_sealed::Sealed for (H, R) {}

impl Depth for () { const D: USize = 0; }
impl<H, R: Depth> Depth for (H, R) { const D: USize = R::D + 1; }
```

Two impls. No specialization. No marker overlap. Coherence trivially holds because `()` and `(H, R)` are disjoint shapes.

`Depth` is impl'd ONLY on cons-list shapes. Flat tuples of arity 3+ deliberately don't impl `Depth`. This is correct: plan-stage code consumes engine accumulators (cons-list by construction) and consumer-declared `Read` / `Write` (cons-list via macros). If a future code path needs `Depth` on a flat tuple, that's the wrong abstraction; refactor to cons-list.

Compile-time assertion `<FiftyWusType as Depth>::D == 50` validates the recursion.

`AccessSet::LEN` is documented explicitly as shape-based (immediate-tuple-arity). Plan-stage code uses `<X as Depth>::D` for total count. The two consts coexist with clear, distinct contracts.

### 4. Recursion limit

`#![recursion_limit = "512"]` declared in:

- `mock/crates/hilavitkutin-api/src/lib.rs` (where the trait machinery lives)
- `mock/crates/hilavitkutin/src/lib.rs` (where `.build()` is called inside the engine's own tests)
- `mock/crates/hilavitkutin-kit/src/lib.rs` (defensive; if a consumer adds a Kit chain that pushes the trait solver here)

Consumer crates (viola, vehje, future apps) inheriting heavy WU/store counts may need to declare it themselves. Document this in `hilavitkutin-api/DESIGN.md.tmpl`:

> If your consumer hits "overflow evaluating the requirement" during `.build()`, declare `#![recursion_limit = "512"]` (or higher) at your crate root. The default 128 accommodates apps up to roughly 30 WUs × 30 stores; larger workloads need explicit lifting.
