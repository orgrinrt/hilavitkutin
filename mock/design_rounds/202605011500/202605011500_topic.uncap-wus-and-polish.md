## Decisions locked in TOPIC

### 1. Replace per-arity `Buildable` macro with recursive impl

Current shape in `mock/crates/hilavitkutin-api/src/builder.rs`:

```rust
impl_buildable!(W0);                // (W0, ())
impl_buildable!(W0, W1);            // (W0, (W1, ()))
...
impl_buildable!(W0, ..., W11);      // depth 12. HARD CAP
```

New shape (one base case + one recursive impl):

```rust
// Base.
impl buildable_sealed::Sealed for () {}
impl<Stores: AccessSet> Buildable<Stores> for () {}

// Recursion. Sealed impl carries no Stores param; Buildable impl does.
impl<H, R> buildable_sealed::Sealed for (H, R) {}
impl<H, R, Stores> Buildable<Stores> for (H, R)
where
    H: WorkUnit,
    R: Buildable<Stores>,
    Stores: AccessSet + WuSatisfied<H::Read> + WuSatisfied<H::Write>,
{}
```

Coherence: `()` and `(H, R)` are disjoint shapes; no `#[marker]` needed. Trait solver does N steps for N WUs, each step adds 2 `WuSatisfied` obligations. Linear, unbounded depth.

`WuSatisfied<A>` stays per-arity 0..=12 because `A` is the consumer's declared `Read` / `Write` flat tuple. 12 stores per WU is comfortable headroom; the cap there doesn't bite real workloads.

The `impl_buildable!` macro is deleted entirely.

### 2. Fix two `Stub` shim soundness foot-guns in tests

`mock/crates/hilavitkutin/tests/scheduler_builder.rs`:

```rust
// ColumnReaderApi::read body:
   unsafe { core::mem::zeroed::<T>() }   // ⚠ UB for niche-bearing T
=> unimplemented!()                      // ✓ where-clauses gate reachability

// ResourceProviderApi::resource body:
   unsafe { &*(self as *const _ as *const T) }   // ⚠ UB: ZST as &T
=> unimplemented!()                              // ✓ same gating
```

Both unsafe blocks are unreachable by the test where-clauses today. But the shim is now a copy-paste template for future test authors. The reviewer flagged that someone will copy it into a test that DOES exercise those paths and hit immediate UB. `unimplemented!()` removes the unsafety entirely; the where-clause guarantees we never call it.

### 4. `#[diagnostic::on_unimplemented]` on `Contains<S>`

Currently when a consumer registers a WU referencing an unregistered store, the error reads:

```
trait bound `(): Contains<Resource<Interner>>` was not satisfied
which is required by `(ReadInterner, ()): Buildable<()>`
```

That's correct but cryptic. the consumer doesn't recognize `Contains` from their own code. `#[diagnostic::on_unimplemented]` lets us attach a message:

```
note: store `Resource<Interner>` was not registered on the SchedulerBuilder.
      Call `.resource::<Interner>(initial)`, `.column::<Interner>()`, or
      install a Kit that registers it.
```

5-line addition in `mock/crates/hilavitkutin-api/src/access.rs` on the `Contains<S>` trait declaration.

### 5. Tighten `hilavitkutin-kit` forbidden-imports

Current rule in `mock/mockspace.toml`:

```toml
{ scope = "hilavitkutin-kit", forbidden = "hilavitkutin::*, hilavitkutin_api::*, hilavitkutin_ctx::*, hilavitkutin_persistence::*, hilavitkutin_str::*", reason = "..." }
```

Add `arvo*, notko*, arvo_bits*` so the zero-deps promise is lint-enforced (currently only Cargo.toml enforces it):

```toml
{ scope = "hilavitkutin-kit", forbidden = "hilavitkutin::*, hilavitkutin_api::*, hilavitkutin_ctx::*, hilavitkutin_persistence::*, hilavitkutin_str::*, arvo::*, arvo_bits::*, arvo_bitmask::*, arvo_graph::*, arvo_sparse::*, arvo_spectral::*, arvo_comb::*, arvo_hash::*, notko::*", reason = "hilavitkutin-kit defines the Kit preset trait only; standalone, no other deps including arvo and notko substrate. Engine depends on it, not the reverse" }
```

Additive lint at designed severity. Not a gate downgrade.

## Out of scope

- `add_kit::<K: Kit<Self> + Default>()` overload (filed as follow-up; reviewer flagged this as ergonomics improvement once a real downstream Kit lands).
- `#[diagnostic::do_not_recommend]` on the cons-list `Contains` recursion impl, OR named-newtype wrappers for `Wus`/`Stores`. Both reduce the verbosity of error messages dumping deep cons-lists. File as follow-up; not blocking.
- Trybuild fixtures (#296). Already tracked.
- Promoting #295 (app! macro) priority. Already noted; user has been explicit that macros are not a priority.
- **WuSatisfied recursion (deliberately deferred).** `WuSatisfied<A>` operates on `A = WU::Read | WU::Write`, which the consumer declares as a flat tuple. Rust cannot recurse over flat tuples (no built-in Head/Tail decomposition); to recurse, the consumer would have to write a cons-list shape, which is a DX regression. Three options to remove this cap exist:
  1. Migrate consumer-side declarations to cons-list shape (ugly).
  2. Provide a `read![]` / `write![]` macro that converts flat to cons-list (small new macro; consumer adopts non-tuple syntax).
  3. Support both via `#[marker]` overlap (coherence ceremony; sealing supertrait needs adjustment).
  
  Per-WU 12-cap differs in nature from the per-app `Wus` cap that this round fixes. A single WU touching 13+ stores is itself an architectural smell (the WU is doing too much). A 50-WU app with each WU touching 3-7 stores hits no cap. Asked of the third reviewer: does any planned consumer (vehje typecheck, validator framework, viola lint pipeline) realistically have a single WU with >12 store accesses? If yes, file a follow-up round to apply option (2). If no, document as soft ceiling.

## Open question for the third reviewer

This round was opened specifically because two prior reviewers (and the author) all missed the Wus = 12 cap. The TOPIC asks the third reviewer:

**"What ELSE did all three of us miss?"**

The Wus cap was a clear scaling failure that escaped reviewer attention because the original framing said "matches AccessSet's arity-12 cap" and that sounded reasonable. The reviewer should probe for similar "looks reasonable on inspection but fails at consumer scale" issues:

- Is the `Stores` cons-list recursion actually free of trait-solver depth limits at 50+ stores? (rustc default recursion limit is 128.) What about 200 stores?
- Are there other arities bottoming out per-arity (anywhere in the new `builder.rs` + `access.rs`) that I missed?
- Does the Kit trait shape itself have a hidden cap when consumers compose 5+ Kits in series? (Each Kit's `Output` is a fresh builder type; deep Kit chains stack type-level transformations.)
- Is the `WuSatisfied<A>` 0..=12 cap on per-WU Read/Write actually fine at 50-WU scale, or do consumers in real workloads (vehje typecheck, validator framework) hit per-WU Read/Write with more stores than that?
- Anything in the test shim or the round artefacts that has a "looks fine, breaks at scale" smell?

The third reviewer's job is the not-yet-found problems, not re-confirming the Buildable fix.
