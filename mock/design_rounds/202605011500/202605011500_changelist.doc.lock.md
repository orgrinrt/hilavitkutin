### `mock/crates/hilavitkutin-api/DESIGN.md.tmpl`

Rewrite the existing "Builder support (domain 23, round 202605010900)" subtree (lines 177 to 257). The section was authored in round 202605010900 against the per-arity blanket-impl shape; round 202605011500 replaces that shape with cons-list recursion plus three new contracts. Section-level edits:

1. **`Buildable<Stores>` sealed trait subsection.** Replace the per-arity-impl prose ("Implemented per `Wus`-tuple arity (0..=12, matching the `AccessSet` cap)") with the recursive cons-list framing. Two impls only:

   ```rust
   impl Buildable<Stores> for ()                      // base: no WUs
       where Stores: AccessSet;
   impl<H, R, Stores> Buildable<Stores> for (H, R)    // recursion
       where H: WorkUnit,
             R: Buildable<Stores>,
             Stores: AccessSet
                  + WuSatisfied<H::Read>
                  + WuSatisfied<H::Write>;
   ```

   Prose: trait solver does N steps for N WUs, each step adds two `WuSatisfied` obligations. Linear, unbounded depth (no per-arity cap). Coherence trivially holds because `()` and `(H, R)` are disjoint shapes. Sealed via private supertrait. Document the recursion-limit consequence: rustc's default `recursion_limit = 128` accommodates apps up to roughly 30 WUs × 30 stores; larger workloads need `#![recursion_limit = "512"]` at the consumer crate root (covered in the new "Recursion limit" subsection below).

2. **`WuSatisfied<A>` sealed trait subsection.** Replace the per-arity-impl prose ("Implemented per `A`-tuple arity (0..=12, the same `AccessSet` cap)") with the cons-list recursion framing. Two impls only:

   ```rust
   impl<S: AccessSet> WuSatisfied<()> for S {}        // base
   impl<S, H: 'static, R> WuSatisfied<(H, R)> for S    // recursion
       where S: Contains<H> + AccessSet + WuSatisfied<R>,
             R: AccessSet;
   ```

   Prose: `A` is a cons-list shape produced by the `read!` / `write!` macros from a consumer's flat-tuple syntax. Recursion proves `Contains<Tᵢ>` for every `Tᵢ` in `A` against `Self`. No per-arity cap.

3. **New subsection: `read!` / `write!` macros.** New subsection explaining the consumer-facing migration. Consumers write `type Read = read![Resource<X>, Column<Y>];` instead of `type Read = (Resource<X>, Column<Y>);`. The macro converts flat-tuple syntax to the cons-list shape `WuSatisfied` expects. Single-store `read![Resource<X>]` works identically to multi-store. Empty `read![]` yields `()`. Document that the same shape applies to `Write`. Cite the specification once in code-block form:

   ```rust
   #[macro_export]
   macro_rules! read {
       () => { () };
       ($T:ty $(,)?) => { ($T, ()) };
       ($T:ty, $($rest:ty),+ $(,)?) => { ($T, $crate::read!($($rest),+)) };
   }
   // write! has identical shape.
   ```

   Prose: macros live in `hilavitkutin-api` and are re-exported from the crate root.

4. **New subsection: `BuilderExtending<B>` sealed trait.** Constrains `Kit::Output` so a buggy Kit cannot wipe prior registrations. Trait declaration:

   ```rust
   #[allow(private_bounds)]
   pub trait BuilderExtending<B>: Sealed<B> {}

   impl<Wus, Stores, NewStores>
       BuilderExtending<SchedulerBuilder<MAX_*, Wus, Stores>>
       for SchedulerBuilder<MAX_*, Wus, NewStores>
   where
       Wus: AccessSet,
       Stores: AccessSet,
       NewStores: AccessSet + WuSatisfied<Stores>,
   {}
   ```

   Prose: `Wus` is identical between input and output (Kits never add WUs, only stores; consumers register WUs themselves with `.add::<W>()` after kits compose stores). `NewStores: WuSatisfied<Stores>` proves every store from the input `Stores` is still present in `NewStores`. Engine's `add_kit` adds `K::Output: BuilderExtending<Self>` as a where-clause; a buggy Kit returning `SchedulerBuilder<(), ()>` fails the bound at the call site with a clean compile error. Sealed via private supertrait `Sealed<B>`; only the engine's own `SchedulerBuilder` impl is permitted.

5. **New subsection: `Depth` trait.** Plan-stage code consumes engine accumulators (cons-list by construction) and consumer-declared `Read` / `Write` (cons-list via `read!` / `write!`). `AccessSet::LEN` reports immediate-tuple-arity (always 2 for cons-list cells), which is correct for that contract but not for total-element counts. `Depth` ships the cons-list-only count:

   ```rust
   #[allow(private_bounds)]
   pub trait Depth: Sealed {
       const D: USize;
   }

   impl Depth for () { const D: USize = 0; }
   impl<H, R: Depth> Depth for (H, R) {
       const D: USize = R::D + 1;
   }
   ```

   Prose: two impls, no specialization, no marker overlap. Coherence trivially holds because `()` and `(H, R)` are disjoint shapes. `Depth` is impl'd ONLY on cons-list shapes; flat tuples of arity 3+ deliberately do not impl `Depth`. This is correct-by-contract: any value the plan stage counts comes from a cons-list source. If a future code path needs `Depth` on a flat tuple, that is the wrong abstraction and the call site should refactor to cons-list. `AccessSet::LEN` and `Depth::D` coexist with explicit, distinct contracts (the former shape-based, the latter total-element-based). Both stay documented at their respective trait declarations.

6. **`Contains` diagnostic.** New small subsection (or fold into existing Contains documentation if it lives nearby). The trait gains `#[diagnostic::on_unimplemented]` so the unsatisfied-trait-bound error reads as a stable, consumer-friendly message instead of the bare `Contains` mention. Specify the message text:

   ```text
   note: store `{Self}` does not contain `{T}`. Register it with
         `.resource::<T>(initial)`, `.column::<T>()`, `.add_virtual::<T>()`,
         or install a Kit that registers it.
   ```

   Prose: the attribute is purely a DX win; trait semantics unchanged. Only an additive 5-to-7-line annotation on the existing trait declaration in `access.rs`.

7. **New subsection: Recursion limit.** Document the consumer guidance:

   > If your consumer hits `overflow evaluating the requirement` during `.build()`, declare `#![recursion_limit = "512"]` (or higher) at your crate root. The default 128 accommodates apps up to roughly 30 WUs × 30 stores; larger workloads need explicit lifting. The `hilavitkutin-api`, `hilavitkutin`, and `hilavitkutin-kit` crates already declare 512 internally for their own machinery.

   Position: directly after the `Depth` subsection. One paragraph, one example.

8. **`.build()` where-clause subsection (lines 224 to 248).** Mostly stays. Edit the inline error-text example to match the upgraded `Contains` `#[diagnostic::on_unimplemented]` message (so the example shows the new message, not the bare unsatisfied-bound text). Edit the call-site fix-up sentence so it lists `.add_virtual` (not `.virtual_`).

9. **"Why these live in api, not in kit" subsection.** Extend slightly: also names `BuilderExtending` and `Depth` as living here, with the same rationale (contract-shape artefacts referenced by engine where-clauses and plan-stage code; `Kit` is purely composition).

### `mock/crates/hilavitkutin-kit/DESIGN.md.tmpl`

Two small edits:

1. **Add a "Crate-internal recursion limit" note.** One paragraph in a new short subsection at the end (or folded into "Why a separate crate"). Wording: this crate declares `#![recursion_limit = "512"]` defensively. If a consumer composes deep Kit chains where each Kit's `install` body builds against the previous output's accumulators, the trait solver runs through the cons-list at each step. Authoring a Kit that tried to drop registrations would fail at the engine's call site, not at the Kit declaration. Single sentence pointing at the api crate's `BuilderExtending` subsection for the contract details.
