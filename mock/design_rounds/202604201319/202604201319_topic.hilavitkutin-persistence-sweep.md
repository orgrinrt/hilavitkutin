## Purpose

Migrate `hilavitkutin-persistence` from bare std primitives to the stack vocabulary:

- `Option<T>` → `notko::Maybe<T>`
- `Result<T, E>` → `notko::Outcome<T, E>`
- bare `u32`/`u64`/`usize`/`bool` in public positions → domain-typed newtypes or arvo primitives
- bare string-in-docs framing → adjusted to describe the new shape

The crate is `#![no_std]`, no alloc. The sweep is docs-first (`DESIGN.md.tmpl` prescribes bare `u32` today; that is the drift). Source migrates to match.

## Open questions (provisional decisions recorded; override any via AskUserQuestion response below)

### Q1 / Q2 — ContentHash home and shape

**Decision (user, 2026-04-20, resolved via #116):** `ContentHash` lives in **arvo-hash** (a new L2 crate, sibling of arvo-bitmask). Shape: `pub type ContentHash = arvo_bits::Bits<28>`. Persistence imports `arvo_hash::ContentHash` directly; `NameHash` is a semantic alias used contextually, not a distinct type.

Why the arvo home:

- Hashes are numeric bit-pattern algorithms. They belong in the numeric substrate, not in a downstream persistence or str crate.
- `arvo_bits::Bits<const N: u8>` is the generic opaque bit-pattern container (shipped in #116). `ContentHash` aliases `Bits<28>`.
- Cross-cutting: clause and future consumers also want this type family. Arvo-level home keeps it reusable.

Alternatives considered during #116: a persistence-local newtype (too narrow; loses cross-crate reuse), a hilavitkutin-str newtype (hashes aren't a string-domain concern), a hilavitkutin-api entry (conflicts with the contract-first framing from #82).

### Q3 — Byte-buffer offsets / lengths shape?

`bytes_offset: u32` and `bytes_len: u32` in `StringTableEntry` are buffer-position indices. Choices:

- **A (recommended provisional)** — newtype `pub struct BufferOffset(pub USize)` + `pub struct BufferLen(pub USize)` local to `hilavitkutin-persistence`. Uses `arvo::USize` under the hood.
- **B** — `arvo::USize` directly. Risks semantic-alias-nudge lint fire at this boundary.
- **C** — use existing `arvo::Cap` (capacity) for length, `USize` for offset.
- **D** — other.

**Provisional decision:** A — persistence-local newtypes over `USize`. BufferOffset and BufferLen are specific enough to deserve names; they're not generic "a count" (cf. `RowCount`).

### Q4 — Count types (rows, columns, cardinality)?

- **A (recommended provisional)** — persistence-local newtypes `RowCount(USize)`, `ColumnCount(USize)`, `Cardinality(USize)`. Each one different domain.
- **B** — use `arvo::USize` directly for all three.
- **C** — use `arvo::Cap` for column bounds (const max-capacity), `USize` for runtime counts.
- **D** — other.

**Provisional decision:** A — explicit domain aliases. These appear in pub API surface; naming them makes call sites self-documenting. `MAX_COLUMNS_PER_TABLE: usize = 64` becomes `MAX_COLUMNS_PER_TABLE: Cap` since it IS a const-capacity value.

### Q5 — Schema version type?

- **A (recommended provisional)** — newtype `pub struct SchemaVersion(u32)` with `#[derive(...)]` + `next() -> Self` increment helper. Opaque ordinal.
- **B** — `arvo::UFixed<32, 0>` or similar numeric.
- **C** — just use `ContentHash`-style opaque bits (version is hash-like in being compared for equality, not arithmetic).
- **D** — other.

**Provisional decision:** A — newtype over `u32`. Schema versions are ordinal (v1 < v2) but arithmetic is never meaningful (v7 - v3 has no meaning). Newtype with comparison + `next()` matches the domain.

### Q6 — Booleans (`visited`, `is_empty()`)?

- **A (recommended provisional)** — use `arvo_bits::Bool` (`arvo::Bool` re-export) everywhere at pub API boundary. Private fields that back them can remain bare `bool` as impl detail (Bool wraps bool).
- **B** — use bare `bool` everywhere (pragmatic exemption).
- **C** — other.

**Provisional decision:** A — `arvo::Bool` at boundary. `fn is_empty(&self) -> Bool`. Private struct fields can be bare `bool` internals.

### Q7 — Sieve eviction weight?

`weight: u64` is an eviction-priority score. Choices:

- **A (recommended provisional)** — persistence-local newtype `pub struct EvictionWeight(u64)`. Comparable, no arithmetic.
- **B** — `arvo::UFixed<64, 0, Cold>` — forces Strategy marker, wrong semantically.
- **C** — bare `u64` kept, local exemption.
- **D** — other.

**Provisional decision:** A — `EvictionWeight` newtype. Domain-clear, doesn't force arithmetic semantics.

### Q8 — rkyv `Archived` type exemption?

`type Archived = u32` on `impl Archive for Str`. The `Archived` associated type names a disk-layout ABI value, not a runtime API value. Choices:

- **A (recommended provisional)** — explicit carve-out in the doc CL: rkyv `Archived = <bare int>` is permitted (like core trait impls). Document the exception in DESIGN.md under a "Disk-layout types" subsection.
- **B** — newtype the archived shape too (`type Archived = ArchivedContentHash;`).
- **C** — other.

**Provisional decision:** A — exemption for rkyv `Archived` bit-layout types. Document it in DESIGN.md.tmpl. Same pattern as `fn fmt() -> fmt::Result` being permitted.

### Q9 — Persistence-local error → Outcome migration detail?

`PersistenceError` is an existing crate-local enum. After migration:

- **A (recommended provisional)** — `fn open(...) -> Outcome<Self, PersistenceError>` — keep error type, swap wrapper. Zero shape change.
- **B** — introduce `PersistenceOutcome<T>` type alias (`= Outcome<T, PersistenceError>`) for readability.
- **C** — other.

**Provisional decision:** A — plain `Outcome<T, PersistenceError>`. Alias add zero value over the generic form; the `Outcome` spelling carries intent.

### Q10 — Scope boundary: does this round also touch the rkyv Archive impls in source?

- **A (recommended provisional)** — no. rkyv impls stay; adjust only docs + pub surfaces. rkyv integration has its own follow-up round (tracked in `BACKLOG.md.tmpl`).
- **B** — yes, rework rkyv impls to operate on the new newtypes.
- **C** — other.

**Provisional decision:** A — out of scope for this round. rkyv Archived types keep their bare-int layout; the exception in Q8 covers them.

## Finalised design sketch (under provisional decisions above)

### Imports from arvo-hash (shipped in #116)

```rust
use arvo_hash::ContentHash; // = arvo_bits::Bits<28>
```

Used directly at API boundaries in place of bare `u32` content-hash values. `NameHash` stays as a contextual variable name rather than a distinct type; the 28-bit FNV content-hash domain is one type, two spellings at different call sites.

### New types introduced this round
