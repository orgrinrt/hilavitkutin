# Doc Changelist: hilavitkutin-persistence primitive-vocabulary sweep

**Date:** 2026-04-20
**Phase:** DOC
**Scope:** `hilavitkutin-persistence/DESIGN.md.tmpl` primary. Incidental touches to other templates only if grep surfaces stale references.
**Topic:** `202604201319_topic.hilavitkutin-persistence-sweep.md`
**Type:** additive on domain-alias vocabulary; adjusts the existing bare-`u32` / `u64` struct field types to their semantic newtypes. No change to algorithmic behaviour.

## Motivation

`hilavitkutin-persistence` ships with bare `u32` / `u64` / `usize` / `bool` / `String` / `Option` / `Result` in pub positions. The primitive-vocabulary discipline (CLAUDE.md, per-repo PRINCIPLES, no-bare-* lints) rejects these. `arvo-hash::ContentHash` now exists (shipped via #116 round `202604201338`), so the sweep can replace the `u32` FNV-hash values with the cross-crate domain type.

Per-site drift was audited in the topic file. Mapping summary:

| Current | Target |
|---|---|
| `content_hash: u32`, `name_hash: u32`, return `u32` from FNV paths | `arvo_hash::ContentHash` |
| `bytes_offset: u32`, `bytes_len: u32` | `BufferOffset(USize)`, `BufferLen(USize)` (new, persistence-local) |
| `bit_width: u32` | `BitWidth(u32)` (new, persistence-local; semantic alias despite same backing) |
| `cardinality: u64`, `row_count: u64`, `column_count: usize`, `count: usize` | `Cardinality(USize)`, `RowCount(USize)`, `ColumnCount(USize)` |
| `version: u32` | `SchemaVersion(u32)` |
| `weight: u64` | `EvictionWeight(u64)` |
| `Option<T>` in pub methods | `notko::Maybe<T>` |
| `Result<T, PersistenceError>` in pub methods | `notko::Outcome<T, PersistenceError>` |
| `bool` at pub API boundary | `arvo::Bool` |

`type Archived = u32` on rkyv impls stays bare (disk-layout ABI exemption, called out explicitly in DESIGN.md).

## Changes proposed

### 1. `mock/crates/hilavitkutin-persistence/DESIGN.md.tmpl`

**Update the Manifest code block** (currently lines ~127–138) to reflect the new types. The exact before/after code changes are mechanical; doc CL execution applies them verbatim.

Before:

```rust
struct Manifest {

}

    name_hash: u32,
    version: u32,
    row_count: u64,
```

After:

```rust

    tables: /* map of table_id → TableMeta */,
    count: ColumnCount,
}

struct TableMeta {
    name_hash: ContentHash,
    version: SchemaVersion,
    row_count: RowCount,
    column_count: ColumnCount,
    columns: /* column metadata */,
}
```

**Update the Str Archive block** (currently ~lines 40–44) to show the disk-layout discipline explicitly:

Before:

```rust
impl Archive for Str {
    type Archived = u32;  // content hash
}
```

After:

```rust
impl Archive for Str {
    // Disk-layout ABI. Runtime API uses ContentHash; the rkyv
    // Archived type stays bare u32 to pin on-disk byte layout.
    type Archived = u32;
}
```

**Add a new subsection** `## Disk-layout types (rkyv ABI)` under `## Serialisation boundary` (inserted before the `## Cold store` heading). Roughly:

```
## Disk-layout types (rkyv ABI)

Every value that crosses the rkyv boundary on disk is specified by
its byte layout, not by its runtime type surface. The `Archived`
associated type on an `impl Archive` spells the disk form; the
runtime type spells the in-memory form.

Current disk-layout types (pinned):

- `Str::Archived = u32` — low 28 bits carry the FNV content hash;
  bit 31 is the const/runtime flag (const-interned strings on disk
  are just their content hash).
- Integer primitive columns archive to themselves (identity).

The runtime API uses `arvo_hash::ContentHash` (= `arvo_bits::Bits<28>`)
for the hash value. Conversion at the rkyv boundary is a single
masked cast; no format change.

Disk-layout types are exempt from the primitive-vocabulary
discipline. They exist to pin a byte layout across sessions; giving
them a semantic newtype would add a conversion call at the hottest
serialisation path without changing the on-disk format.
```

**Update the Key types table** at the bottom (currently lines ~151–158) to mention the new domain aliases. Add rows for `ContentHash`, `StringTable`'s use of `BufferOffset` / `BufferLen`, and the count aliases.

**Preserve** backtick discipline during the doc CL: names that land in source during the src CL stay **unbackticked** until then (same rule as used for arvo-hash doc CL).
