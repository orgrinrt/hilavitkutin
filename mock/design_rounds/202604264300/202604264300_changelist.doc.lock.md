## Deliverables

### Fix 1 — `Str::CONST_MASK` is a misleading zero

`hilavitkutin-str/src/handle.rs` defines `CONST_MASK = 0 << 31`,
which evaluates to `0`. The bit layout is:

- bit 31 = `0` for const-origin handles, `1` for runtime-origin
- bits 30-28 reserved
- bits 27-0 ID

`is_const()` checks `(self.0 & Self::RUNTIME_MASK) == 0`, i.e. it
detects a const-origin handle by the *absence* of the runtime bit.
So the `CONST_MASK = 0` value is numerically correct — the
const-origin flag pattern at bit 31 is literally zero — but the
name `CONST_MASK` implies a non-zero bit mask that a caller could
AND with a handle. No external caller uses `CONST_MASK` today
(internal grep confirms only `RUNTIME_MASK` is referenced by
`__runtime` and `is_const`), so the rename is free.

Chosen fix: rename `CONST_MASK` to `CONST_ORIGIN_FLAG` with a doc
comment clarifying that the const-origin polarity at bit 31 is a
*cleared* bit, and that `is_const()` / `is_runtime()` are the
intended query APIs for callers — masking against the constant is
not a correct origin test.

### Fix 2 — Cargo.toml package metadata

`mock/Cargo.toml` has `[workspace.package]` with `version`,
`edition`, `license` but no `description` or `repository`. Each
per-crate `Cargo.toml` inherits those three via `.workspace = true`.
crates.io rejects publishes without a description and repository.
Even pre-publish, the metadata aids discoverability via
`cargo metadata` + rustdoc.

Chosen fix:
- Add `description = "Pipeline execution engine — provider-driven, const-generic, no_std"` to `[workspace.package]`.
- Add `repository = "https://github.com/orgrinrt/hilavitkutin"` to `[workspace.package]`.
- Each crate's `Cargo.toml` gets `description` and `repository`
  declared as per-crate values (not workspace-inherited), with
  a concise crate-scoped description each. The workspace
  description stays as a fallback for any future crate that
  hasn't yet customised.

Per-crate descriptions:
- `hilavitkutin-api` — "WorkUnit + provider trait surface for the hilavitkutin pipeline engine"
- `hilavitkutin` — "Pipeline execution engine — provider-driven, const-generic, no_std"
- `hilavitkutin-build` — "Build-time helpers for the hilavitkutin pipeline engine"
- `hilavitkutin-ctx` — "Per-WorkUnit context framework for hilavitkutin"
- `hilavitkutin-persistence` — "Zero-copy persistence for hilavitkutin pipeline state"
- `hilavitkutin-str` — "Interned string handle + arena + linker-section const table"

### Fix 3 — `inject_str` sentinel vs `StringInterner::resolve`

`hilavitkutin-str`'s `StringInterner::resolve(&self, Str) -> &str`
returns the empty string on a const-table miss (via
`lookup_const_by_handle(s).unwrap_or("")`). `inject_str` in
`hilavitkutin-persistence/src/archive_str.rs` then detects the
miss via `!resolved.is_empty()`. Two problems:

1. An interned empty string is a legitimate value; it would be
   indistinguishable from a miss.
2. The sentinel is an invisible API contract — a caller reading
   `resolve`'s signature would never know about the empty-string
   fallback.

Chosen fix: change `StringInterner::resolve` to return
`Option<&str>`. `Some(s)` on hit (either const or runtime),
`None` on const-miss. The runtime branch currently delegates to
`ArenaInterner::arena_resolve` which returns `&str` unconditionally
(no miss signal); we keep `arena_resolve` unchanged and always
return `Some(...)` for runtime handles. The const-miss path is
the only `None`.

Callers:
- `inject_str` — uses the new `Option` return cleanly: `Some(s)`
  triggers the const-confirm path, `None` falls through to the
  runtime string-table lookup. Removes the empty-string sentinel.
- `evict_str` (same file) — wraps `interner.resolve(handle)` on
  the runtime branch; updated to `.expect("runtime resolve")` or
  equivalent since runtime handles always resolve (invariant per
  interner contract).
- Tests — both `hilavitkutin-str/tests/interner.rs` and
  `hilavitkutin-persistence/tests/archive_str.rs` update their
  `interner.resolve(h)` equality asserts to match against
  `Some("…")`.

## BACKLOG — non-engine post-review notes

### `hilavitkutin-api/BACKLOG.md.tmpl`

Append a new `## Non-engine post-review notes` section with three
items:

- **`WorkUnit::Ctx` carries all 7 provider bounds unconditionally.**
  The `Ctx` associated type on every WorkUnit requires
  `HasEach + HasBatch + HasReduce + HasMemoryProvider +
  HasThreadPool + HasClock + HasWorker` (or current equivalent)
  regardless of which iteration APIs the WU actually uses. Two
  possible follow-ups: (a) per-method bound splitting, e.g.
  `fn execute_each` bounded only by `HasEach`; or (b) document
  the blanket no-op impl pattern for unused APIs on provider
  tuples. Design decision deferred.
- **`EachApi` / `BatchApi` / `ReduceApi` all name their method
  `run()`.** A provider tuple implementing all three forces
  fully-qualified syntax (`<P as EachApi>::run(...)`) at any call
  site that wants to pick a specific iteration kind. Consider
  renaming to `run_each` / `run_batch` / `run_reduce` for
  ergonomics. Breaking change; BACKLOG only.
- **`HasMemoryProvider` / `HasThreadPool` / `HasClock` accessor
  methods don't match the `provider!` macro conventions.** The
  accessors are `memory()` / `threads()` / `clock()`, which
  diverge from whatever the `provider!` macro would generate from
  the trait name. Either document the asymmetry or rename for
  consistency.

### `hilavitkutin-str/BACKLOG.md.tmpl`

Append a single item:

- **`str_const!` macro hardcodes `0x0FFF_FFFF`** in two places
  (the static-entry `hash` field and the returned handle value),
  instead of referencing `$crate::Str::ID_MASK`. If the bit
  layout ever shifts the ID width, `str_const!` silently desyncs.
  Replace the hex literal with `$crate::Str::ID_MASK` references.
  Tiny follow-up; BACKLOG only.

## Impl-time decisions

1. **`CONST_ORIGIN_FLAG` over `CONST_ORIGIN_VALUE`.** The field
   is conceptually a flag bit pattern (one bit, at position 31),
   not a value in a numeric sense. `FLAG` reads cleaner.
2. **Per-crate `description` and `repository`.** The workspace
   value covers the default, but per-crate descriptions convey
   more at a glance on crates.io. Added to every crate's own
   `[package]`.
3. **`resolve` returns `Option<&str>` outright; no parallel
   `resolve_opt`.** Migration cost is small (5 call sites across
   source + tests) and the signature change is the right answer.
   Parallel methods stay out.
4. **`arena_resolve` signature unchanged.** The `ArenaInterner`
   trait is a host contract; runtime handles are always resolvable
   by construction (the interner assigned the id). Leaving the
   trait alone avoids forcing every future host impl to opt into
   `Option`.
