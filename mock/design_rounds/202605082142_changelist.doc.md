### Replace lines 33 to 37 ("AccessSet is a trait on tuples" framing)

Current text (verbatim):

```
`AccessSet` is a trait on tuples. `(Column<PosX>, Column<PosY>,
Resource<Gravity>)` implements AccessSet. Compile-time store
descriptors. Plan phase converts to runtime AccessMask via
monomorphised generic functions (the monomorphised function IS the
type identity, no TypeId, no std::any).
```

Replacement text:

```
`AccessSet` is a sealed marker on the cons-list typestate. `Empty`
and `Cons<H, T>` are the substrate primitives consumer code
encounters either directly or through the `read!` / `write!`
macros (which expand to nested `Cons` chains). Compile-time store
descriptors. Plan phase converts to runtime AccessMask via
monomorphised generic functions (the monomorphised function IS the
type identity, no TypeId, no std::any).
```

Rationale: post-round-4 the canonical encoding is the cons-list. The "trait on tuples" framing is misleading after this round and was already misleading at the time of round 202605042200 (the round locked HList; this CL aligns the description).

### Replace lines 146 to 170 ("Contains is a sealed compile-time membership trait on tuples" + "Two impl families are shipped" section)

Current text (the whole block from "`Contains<S>` is a sealed..." through the cons-list-recursion code block ending "Permitted because `Contains` is `#[marker]`."):

Replacement text:

```
`Contains<S>` is a sealed compile-time membership witness on the
cons-list typestate. Accessing a store not declared in `Read` or
`Write` fails to compile at the relevant API-trait method's
where-clause. Implemented as `#[marker]`, so overlapping head and
tail impls coexist:

```rust
// Head: Cons<X, T> contains X.
impl<X: 'static, T: 'static + AccessSet>
  Contains<X> for Cons<X, T> {}

// Tail: Cons<H, T> contains M if T does.
impl<H: 'static, T: 'static + AccessSet, M: 'static>
  Contains<M> for Cons<H, T> where T: Contains<M> {}
```

Membership lookup walks the cons-list at trait-solver time. Depth
scales linearly with list length; round 202605042200 measured 0.53
seconds at depth 5 with 25 WorkUnits across 13 stores (S1b
finding), well within the rustc default recursion limit.

`AccessSet::LEN` reports the cons-list cardinality at compile time.
`Empty::LEN = USize::ZERO`. `Cons<H, T>::LEN = USize(T::LEN.0 + 1)`.
The recursive impl computes correctly at any depth and is callable
in const contexts via the const-arith surface on `arvo::USize`.
```

Rationale: drops the legacy "two impl families" framing in favour of the single canonical HList shape. Documents the recursive `LEN` computation that the source CL fixes. Preserves the `#[marker]` rationale (overlapping impls), since that is still a load-bearing aspect of the canonical shape.
