# Changelist (src): arvo Round D alignment — ABANDONED

## Status

Abandoned mid-execution. The topic file's scope claim ("adapter only, 7 src files in hilavitkutin-api + 1 in hilavitkutin-str, 4 test files") understated the actual migration footprint by a factor of three or more. The work was descoped before any code commits landed; all source edits and the Cargo.lock bump have been reverted from the working tree.

## What was discovered

Investigating the bumped lock revealed Round D's reach across hilavitkutin is broader than the topic anticipated:

1. **Pattern A and B (the originally scoped patterns) work.** `arvo::newtype::*` re-exports moved to `arvo::*`; `IBits(N)` literal forms converted to `ibits(N)` helper-fn calls. These migrations apply cleanly to the hilavitkutin-api files listed in the topic and were tested green.

2. **`arvo_bits::bitfield!` macro relocated to `arvo::bitfield!`.** The bitfield primitive moved from L1 contracts to L0 in Round B/C/D. hilavitkutin-str/src/handle.rs uses the macro to declare `StrLayout`. Migration is import path only at the macro site, but emission produces code using `Bits::from_raw` instead of the (removed) `Bits::new`, which forces matching changes at every call site of the generated wrapper's accessors.

3. **`Bits::new` removed; `Bits::from_raw` is the construction surface.** Affects every consumer that constructs `Bits<N, S>` values. Found at: hilavitkutin-str/src/{handle,interner}.rs, hilavitkutin-persistence/src/{archive_str,manifest,sieve,cold_store}.rs, plus tests under each crate.

4. **`Bits::bits()` accessor renamed to `Bits::to_raw()`.** Same call-site footprint as #3.

5. **`Bits<N, S>: From<u64>` not implemented.** Old code used `(value as u64).into()` to construct `Bits<28, Hot>`. New code requires explicit `Bits::<28, Hot>::from_raw((value as u64) as u32)` because the container width is keyed on N (u32 for N=28) rather than the source value's width.

6. **`ContentHash` (alias for `Bits<64, Hot>`) inherits all of #3 to #5.** Used throughout hilavitkutin-persistence.

Item-count estimate: 12 errors in hilavitkutin-persistence on first build attempt; further crates (hilavitkutin-ctx, hilavitkutin engine, plus their tests) likely surface similar patterns. Total source surface is more than 30 files across 5 of the 8 hilavitkutin crates.

## Why abandoned in this shape

The topic file claimed "no public API surface change" and scoped the round as a mechanical adapter. Items 2 to 6 stay below the public API surface, so the topic's framing remains correct. The scope claim does not.

Per workspace rule (mock-workspace.md, Design rounds): "Each round has 'Current state' and 'Changes proposed' sections." A topic that materially understates the changes proposed is not a sound base for execution. The honest move is to abandon and reopen with a topic that names the full scope.

The other path, widening this round in-flight, is rejected because:

- Topic files are frozen once committed; the topic can't be updated to reflect the discovered scope.
- The src CL can be expanded (it is living during SRC phase) but the divergence between the frozen topic and the executing CL becomes a permanent paper trail of "the topic was wrong, see the CL for what actually shipped." That is the failure mode the design-round flow exists to prevent.

## What lands now

Nothing. The branch carries the topic + doc CL + this src CL as a discovery artefact. Working tree is reverted to the pre-execution state (matches `dev` HEAD modulo the round artefacts).
