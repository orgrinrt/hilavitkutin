# Topic: arvo Round D alignment

**Date:** 2026-04-29
**Phase:** TOPIC
**Scope:** `hilavitkutin-api` (5 src files + 4 test files), `hilavitkutin-str` (1 src file). Adapter round to track arvo's Round D substrate change. No public API surface change for hilavitkutin's own consumers; only internal `use` paths and const-generic-position literal forms.

## Current state

arvo Round D consolidated changelist (clause-dev workspace, arvo dev `bcc38304`) reshaped two surface details that hilavitkutin-api currently consumes:

1. The `Bool`, `Cap`, `USize`, `FBits`, `IBits`, `Width` wrappers moved from `arvo::newtype::*` to the crate root re-export at `arvo::*`. The `arvo::newtype` path no longer exists. Round D collapsed `newtype` into `arvo_storage` and re-exports the public surface from `arvo`.

2. The `IBits`, `FBits`, `Width` ConstParamTy newtypes changed their inner field from `u8` to `MetaCarrier(u16)`. The change resolves a rustc trait-solver cycle in the `Hot: UContainerFor<{ ufixed_bits(I, F) }>` projection (research note in arvo: layout-soundness gates for `repr(transparent)` chains and `MetaCarrier` transmute, round D-aux). The literal-construction form changed from `IBits(N)` (raw integer) to either `IBits(MetaCarrier::from_raw(N))` (verbose) or the helper `ibits(N)` / `fbits(N)` / `width(N)` (idiomatic). The helpers are const fns and live alongside the wrappers in the same `arvo::*` re-export.

Both changes are visible from any consumer that names these types or constructs literal const-generic-position values. hilavitkutin-api uses both patterns. hilavitkutin's own `mock/Cargo.lock` pins arvo at `a98a3f49` (pre-Round-D), so its in-workspace builds stay green; viola consumes hilavitkutin via path-deps and against local arvo at the post-Round-D state, which surfaces 38 trait-bound errors when building any viola crate that transitively touches `hilavitkutin-api` (every code-touching #231 prerequisite).

## Changes proposed

Two mechanical sweeps inside the crates listed above.

1. Replace `arvo::newtype::{...}` with `arvo::{...}` everywhere. The imported names stay the same. No trait or method change. Pure path migration.
2. Replace `IBits(N)` and `FBits(N)` (and any `Width(N)`) literal forms in const-generic positions with `ibits(N)` / `fbits(N)` / `width(N)` helper-fn calls. The helper produces an equivalent value with the new `MetaCarrier`-backed payload. Both helpers are const fns, so const-generic positions accept them. Sites where the type appears as a runtime value (rare; hilavitkutin doesn't do this) would use `IBits(MetaCarrier::from_raw(N))`.

No public API change in hilavitkutin. The exposed types (`Urgency`, the various `pub type` aliases over `UFixed<I, F, S>`) keep identical bit widths, identical strategy markers, identical raw container types. The change is in how the hilavitkutin source spells the const-generic argument.

No doc-side change. None of `mock/crates/*/DESIGN.md.tmpl` reference `arvo::newtype` as an import path or `IBits(N)` as a const-construction form. Verified by grep: no `newtype::` mentions outside source; no `IBits(` outside source.

## Why a round at all

The diff is 7 files of mechanical token replacement plus a Cargo.lock bump. Two reasons it goes through the design-round flow rather than a direct chore:

1. The `changelist-required` lint blocks any `crates/*.rs` edit at the HARD_ERROR phase gate when no src CL is open. The repo's invariant is "no source change without a CL", and this is a real source change crossing the public lint surface (the type-position changes are visible to the lint pack).
2. The cross-repo coordination is non-trivial state. Capturing the Round D dependency, the affected files, and the verification steps in a topic + doc CL + src CL leaves a paper trail for the next adapter round (when arvo lands its next consolidating change).

The round stays small: one topic file, one near-empty doc CL (no doc-template edits), one src CL, two `cargo mock lock` calls, one `cargo mock close`.
