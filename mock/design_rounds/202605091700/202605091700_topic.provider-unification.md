# Provider unification: capability becomes provider; extensions provide bins, not slots

## Background

Two things converge in this topic. First, `hilavitkutin-extensions` currently
uses the vocabulary `Capability` / `CapabilityId` for the wire shape carried
across the C ABI link boundary. The wider workspace already uses `Provider` for
two adjacent things: `hilavitkutin-ctx` Context providers (memory, thread pool,
clock, persistence) accessed via typestate-tracked `Has*Provider` bounds, and
loimu's framing where capability and API are collapsed into one concept (a
provider both declares the API surface and gates access to it).

Second, the scheduler-builder shape that landed in round 202605042200 (`with::<M>`
unified accessor) was partially walked back in PR #69 (#381), which introduced
per-kind methods (`add_unit`, `with_resource`, `with_column`, etc.). The round-2
unification intent reads cleanly when described in prose. The PR-#69 surface
reads as messy and disjoint when typed at a call site.

The two questions are linked. Once "provider" is the uniform vocabulary for
anything a context surfaces (statically linked or extension-loaded), the
builder API shape follows: one verb, one type parameter, uniform call shape.

## Decisions

### Decision 1: vocabulary settles on provider

`Capability` / `CapabilityId` retire. The wire-shape vocabulary in
`hilavitkutin-extensions` becomes `Provider` / `ProviderId`. The descriptor's
`capabilities: *const CapabilityDescriptor` becomes `providers: *const
ProviderDescriptor`. The plugin-host API (`get_capability`, `dispatch_capability`)
becomes `get_provider`, `dispatch_provider`.

The vocabulary covers a single uniform concept across the workspace:

- A statically linked `Resource<T>` is a provider.
- A statically linked `Column<T>` is a provider.
- A `Has*Provider` accessor in `hilavitkutin-ctx` is a provider gate.
- An extension-shipped wire shape carrying a vtable across the link boundary is
  a provider.

Loimu's framing (`define_provider!` produces interface plus permission gate as
one concept) is the precedent. The reason this is the right collapse:
authorisation, access surface, and lifetime ownership are all the same axis at
the consumer's call site. Naming them three different things made writing
sound like three things. They are not.

### Decision 2: typestate over trait families, registry over instances

The C ABI link boundary cannot carry Rust types or trait identities. What
survives is function pointers, repr-C POD, static symbols, and trivial
constants. A host that wants to integrate dynamically loaded extensions into
the typestate-tracked context system therefore cannot track per-extension
typestate (the host does not know at compile time which extensions exist).

The split: typestate tracks **trait families** the host has imported (via
plugin-abi crates the host depends on); the registry tracks **instances** of
those trait families that are loaded at runtime. Concretely:

- The host imports `viola-plugin-abi` (or whichever plugin-abi crate) and
  declares `with_linked::<dyn LinterApi>()` on the scheduler builder. This
  reserves a typestate-tracked bin for `LinterApi` extensions.
- Extensions ship cdylibs that export descriptors satisfying the
  `viola-plugin-abi`-declared wire shape. They load through
  `hilavitkutin-extensions` at any time (arbitrary-time-linking invariant
  preserved).
- Inside a WU body that declared `LinkedBin<dyn LinterApi>` in its `Ctx`, the
  WU iterates the bin: `for linter in ctx.linked::<dyn LinterApi>().iter()
  { ... }`.

The host does not know any specific extension at compile time. It knows trait
families. Extensions populate bins; bins are iterated.

### Decision 3: blanket impl on host-side `LinkedProvider<{ID}>`

The mechanism that makes the typestate clean: host-side blanket impls of trait
family contracts on `LinkedProvider<const ID: u64>`, where `ID` is the
content-derived hash of the trait family identifier (e.g.,
`const_fnv1a("viola.linter.v1")`). The plugin-abi crate exports both the
constant and the impl:

```rust
// In viola-plugin-abi
pub trait LinterApi { fn lint(&self, ast: &Ast) -> Diagnostics; }

#[repr(C)] pub struct LinterVtable {
    pub lint: unsafe extern "C" fn(*const Ast, *mut Diagnostics)
}

pub const LINTER_API_ID: u64 = const_fnv1a("viola.linter.v1");

impl<const ID: u64> LinterApi for LinkedProvider<{ID}> {
    fn lint(&self, ast: &Ast) -> Diagnostics {
        let v = self.vtable as *const LinterVtable;
        let mut out = Diagnostics::default();
        unsafe { ((*v).lint)(ast as *const _, &mut out as *mut _); }
        out
    }
}
```

The blanket form covers all hashes for one trait family. Semantics are "trust
the declaration, validate at link-time": the loader verifies the descriptor's
wire-shape ID matches the ID the trait family advertises before placing the
provider into the matching bin. Mismatched extensions never land in the wrong
bin.

The orphan rule does not bite because `LinkedProvider` is host-defined and the
trait is plugin-abi-defined; both crates are imported by the host and the
extension. The blanket impl lives in the plugin-abi crate.

### Decision 4: plugin-abi crate is the granule

The unit of "host knows about this trait family" is a Rust crate. The host
imports a plugin-abi crate (`viola-plugin-abi`, `vehje-plugin-abi`, etc.).
Extensions import the same crate. Trait identity hash plus repr-C vtable
layout plus blanket impl all ship together.

This is the natural granularity. A plugin-abi crate is small (a few traits, a
few vtable layouts, a constant per trait family). Updating the contract is a
crate version bump on both sides. Versioning works through standard Cargo
mechanisms; extensions specify which plugin-abi version they target, the host
checks the version at link time.

### Decision 5: three-tier extensibility framing

Most "real" extension surfaces in the workspace's consumers (viola, vehje,
future apps) are coarse-granular. A linter processes a whole file per call.
A formatter renders a whole document. A new emission backend handles a whole
IR module. At that granularity, the vtable indirection cost (~1 to 2 ns per
call) is invisible against the workload (millseconds per call).

Extension dispatch carries two costs that statically linked providers do not
pay: dynamic dispatch through one function pointer, and iteration over the
bin (one WU body iterates all loaded extensions of a family; each cannot be
broken into a per-instance monomorphised WU because the typestate does not
know the instance count at compile time). Both costs are real for inner-loop
hot-path work; both are negligible for frame-level work.

The framing for hilavitkutin's consumers:

- **Tier A (primary): static linking with cargo features.** Most extensibility
  is workspace-internal. New linters, new emission backends, new validators
  ship as workspace crates with cargo features. Full typestate, full DAG
  optimisation, full monomorphisation. Dispatched as their own WUs.
- **Tier B (secondary): dlopen extensions through plugin-abi crates.** For
  end-users who genuinely cannot rebuild the host. Coarse-granular protocol
  design (one indirect call per file, document, or frame). Dispatched as one
  iterating WU per trait family.
- **Tier C (when warranted): declarative-data extensions.** Pattern data the
  host's static matcher consumes. Extensions ship bytes (patterns plus handler
  IDs); the matcher is fully optimised, statically linked. Trades expressiveness
  for performance.

Inner-loop hot-path work (per-byte tokenisation, per-record column-store
kernels, SIMD microkernels) does not go through tier B. It statically links.
The constraint is that tier B is intentionally chosen, not the default. The
host author writes a contract (the plugin-abi crate) and accepts the cost.
Well-designed contracts make tier B's cost invisible at the consumer's
workload granularity.

### Decision 6: plan re-bake on extension load or drop

When an extension loads (or drops), the registry changes which means the
relevant bins gain or lose entries. The dispatch shape inside the iterating
WU body cares: it can shard the iteration across cores, prefetch instance
data, allocate per-instance morsel space.

The plan is recomputed when bin contents change for any trait family
referenced by a live WU. The recomputation is cheap relative to the load
operation itself (dlopen costs milliseconds; plan recompute is microseconds).
The arbitrary-time-linking invariant is preserved: load and drop happen at
arbitrary points; the engine reacts by recomputing the plan, not by
rejecting the change.

### Decision 7: optional providers

Optional providers are supported via standard `Maybe<...>` typing on the
context. A WU's Ctx declares `Maybe<HasFooProvider>`; the WU body
pattern-matches on the Maybe to handle the not-present case. This works
identically for static and linked providers. For linked providers, an empty
bin (no extensions loaded) reads as the Maybe being `Isnt`; a non-empty bin
reads as `Is(bin)`. WU bodies that handle Maybe handle both.

## What this enables

- Single uniform vocabulary across context, registry, plugin-host, and consumer
  call sites. "Provider" names the thing wherever the thing lives.
- Statically linked configurations preserve full typestate, full
  monomorphisation, and the optimisable WU DAG.
- Dynamically loaded extensions integrate cleanly through trait-family-keyed
  bins. The compile-time typestate knows the families; the runtime registry
  knows the instances.
- Plugin-abi crates are the granule of "what trait families this host
  recognises." Versioning piggybacks on Cargo.
- The cost of going dynamic is intentional and contained at the consumer's
  Tier-B contract design.
- Arbitrary-time-linking invariant preserved. No "all loaded before any
  invoked" assumption anywhere.

## What this constrains

- Extensions cannot ship arbitrary trait shapes. They can only ship impls of
  trait families the host pre-imported via a plugin-abi crate.
- Extensions cannot be broken into per-instance monomorphised WUs. One
  iterating WU per trait family is the dispatch shape.
- Inner-loop hot-path work is not extensible via dlopen. Static linking
  (Tier A) is the path; cargo features select subsets at compile time.
- Trait family contracts must be designed at coarse granularity to keep
  vtable indirection invisible. Per-byte or per-record extensibility is
  not the right shape; per-file or per-frame is.
