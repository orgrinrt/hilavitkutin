# Builder shape: one verb, value-driven, dispatch-via-associated-type

## Background

Round 202605091700 topic 1 settled that "provider" is the single uniform
vocabulary covering statically linked Resources and Columns, ctx
`Has*Provider` accessors, and extension-shipped wire shapes. Topic 2
applies the same uniformity lens to the SchedulerBuilder.

The current builder (`hilavitkutin/mock/crates/hilavitkutin/src/scheduler/mod.rs:79-141`)
exposes nine methods: `add_unit`, `add_resource`, `add_resource_default`,
`add_column`, `add_virtual`, `add_kit`, `with_memory`, `with_threads`,
`with_clock`. The split tracks domain-vs-platform, which the unified
provider framing dissolves. The result is messy at the call site:
`add_*` and `with_*` interleave; the reader's eye has to track which
verb prefix matches which kind. Under the unified framing the split
should not exist.

Sketch `mock/research/sketches/202605091700-builder-provider-shape/`
validated the shape under nightly 1.96.0 (positive: ten-line call
site compiles + runs; negative: `u32` produces clean Provider
diagnostic via on_unimplemented). See `OUTCOME.md` in that
directory for the full validation.

## Decisions

### Decision 1: one verb, value-driven, `.with(value)`

The builder exposes one registration method:

```rust
impl<Wus, Stores, Plat> SchedulerBuilder<Wus, Stores, Plat> {
    pub fn with<P: Provider>(self, _provider: P) -> SchedulerBuilder<
        <P::Dispatch as Dispatch>::NextWus<Wus>,
        <P::Dispatch as Dispatch>::NextStores<Stores>,
        <P::Dispatch as Dispatch>::NextPlatform<Plat>,
    > {
        SchedulerBuilder { _phantom: PhantomData }
    }
}
```

Every registration is a value passed to `.with`. The trait dispatch
on the value's type discriminates kind. Reads as ten lines of
`.with(value)` with consistent shape:

```rust
let scheduler = Scheduler::builder()
    .with(MyMemory::new(arena))
    .with(MyThreadPool::new(8))
    .with(MyClock)
    .with(Resource::new(GameState::default()))
    .with(Column::<Player>::new())
    .with(Virtual::<Tick>::new())
    .with(LinkedBin::<dyn LinterApi>::new())
    .with(InputKit)
    .with(SpawnerWu)
    .with(PhysicsWu)
    .build();
```

All current `add_*` and `with_*` methods retire. Per
`no-legacy-shims-pre-1.0.md`, no deprecated aliases.

### Decision 2: Provider trait carries Dispatch as associated type

```rust
#[diagnostic::on_unimplemented(
    message = "`{Self}` is not a Provider; pass a registered provider value to `.with(...)`",
    label = "not a Provider",
    note = "use `Resource::new(value)` for singleton state, `Column::<T>::new()` / `Virtual::<T>::new()` / `LinkedBin::<dyn Trait>::new()` for type-keyed declarations, or impl `Provider` on your unit-struct WUs/Kits/platform impls."
)]
pub trait Provider: Sized {
    type Init;
    const KIND: ProviderKind;
    type Dispatch: Dispatch;
}
```

`Init` carries the construction-time value type (`()` for stateless
wrappers, `T` for `Resource<T>`, `Self` for unit structs and platform
providers).

`KIND` is a documentation-and-debugging aid; runtime code does not
branch on it. The compile-time dispatch lives entirely in `Dispatch`.

`Dispatch` is the per-kind typestate router. The associated-type
dispatch (rather than per-kind blanket impls of a helper trait) is
load-bearing: the trait solver fails on `Provider` directly when a
non-Provider type is passed, surfacing the right diagnostic. Sketch
validated this: with per-kind blankets the error misleadingly
pointed at the wrong supertrait.

### Decision 3: Dispatch trait + four routers (full words)

```rust
pub trait Dispatch {
    type NextWus<Wus>;
    type NextStores<Stores>;
    type NextPlatform<Platform>;
}

pub struct UnitDispatch<W>(PhantomData<W>);
pub struct StoreDispatch<S>(PhantomData<S>);
pub struct KitDispatch<K>(PhantomData<K>);
pub struct PlatformDispatch<P>(PhantomData<P>);
```

Note: full words on `PlatformDispatch`, not the abbreviated `Plat`
that appeared in the sketch. The sketch's truncation was scratch-
prose ergonomics; production names spell out.

Per-router behavior:

- `UnitDispatch<W>`: WUs land on the `Wus` cons-list; Stores and
  Platform untouched.
  ```rust
  type NextWus<Wus> = Cons<W, Wus>;
  type NextStores<Stores> = Stores;
  type NextPlatform<Platform> = Platform;
  ```
- `StoreDispatch<S>`: Resources, Columns, Virtuals, LinkedBins land
  on `Stores`.
  ```rust
  type NextWus<Wus> = Wus;
  type NextStores<Stores> = Cons<S, Stores>;
  type NextPlatform<Platform> = Platform;
  ```
- `KitDispatch<K>`: kits append `K::Units` (a `WorkUnitBundle`)
  onto `Wus` and `K::Owned` (a `StoreBundle`) onto `Stores`. Uses
  the existing `Concat` machinery from round 4.
  ```rust
  type NextWus<Wus> = <K::Units as Concat<Wus>>::Out;
  type NextStores<Stores> = <K::Owned as Concat<Stores>>::Out;
  type NextPlatform<Platform> = Platform;
  ```
- `PlatformDispatch<P>`: memory provider, thread pool, clock, and
  any future platform-bound provider lands on `Platform`.
  ```rust
  type NextWus<Wus> = Wus;
  type NextStores<Stores> = Stores;
  type NextPlatform<Platform> = Cons<P, Platform>;
  ```

`KitDispatch` ships first-class out the door, not deferred. Round
4 already shipped the `Concat` machinery; this round wires it into
the dispatch routing.

### Decision 4: HasTrivialCtor — granular trait, lives in notko

The trait that formalizes "this type has a no-arg const constructor"
is named `HasTrivialCtor` (not `Marker`, not `ProviderWithTrivialCtor`).
It is independent of `Provider`. Granular, tightly scoped, reusable.

```rust
// In notko::ctor (or notko::traits::HasTrivialCtor)
pub const trait HasTrivialCtor: Sized {
    fn new() -> Self;
}
```

Reasoning:

- The trait describes a property of a type ("ships a `const fn new()`
  with no arguments"), not a position in any particular surface. It
  reads cleanly anywhere a const no-arg constructor is meaningful.
- It is reusable beyond hilavitkutin: any workspace crate that wants
  the same convention (arvo PhantomData wrappers, vehje markers,
  viola plugin types) can impl it without dragging in
  `hilavitkutin-api`.
- "Marker" was rejected because it overloads with `core::marker`
  conventions and reads as too generic for what is in fact a
  ctor-shape contract.
- "ProviderWithTrivialCtor" was rejected because tying the trait to
  Provider leaks two concerns into one name. The two are orthogonal:
  Provider is "registerable on the SchedulerBuilder"; HasTrivialCtor
  is "constructible with `T::new()` in const context". Conjoining
  them at the trait level prevents reuse.

The trait lives in **notko** (the foundation primitives crate). Notko
is the right home: zero deps, workspace-wide reach, exactly the
class of small reusable primitive notko exists for.

Library-side hilavitkutin wrappers impl it:

```rust
// In hilavitkutin-api
impl<T: 'static> const HasTrivialCtor for Column<T> {
    fn new() -> Self { Column(PhantomData) }
}
impl<T: 'static> const HasTrivialCtor for Virtual<T> {
    fn new() -> Self { Virtual(PhantomData) }
}
impl<T: ?Sized + 'static> const HasTrivialCtor for LinkedBin<T> {
    fn new() -> Self { LinkedBin(PhantomData) }
}
```

`Resource<T>` does NOT impl HasTrivialCtor: `Resource::new(value)`
takes the actual value, no trivial form. The wrapper's inherent
constructor is what users call.

User-side WUs / Kits / platform impls also impl HasTrivialCtor when
they are unit structs (which most are). This enables both the bare-
name form (`MyKit`) and the trait-method form (`MyKit::new()`) at
call sites; the bare-name form is preferred for unit structs but
the trait method form is available for generic positions.

### Decision 5: WorkUnit and Kit supertrait Provider

Existing traits become supertraits-of-Provider:

```rust
pub trait WorkUnit: Provider<Init = Self> {}
pub trait Kit: Provider<Init = Self> + WorkUnitBundle {
    type Units: WorkUnitBundle;
    type Owned: StoreBundle;
}
```

`WorkUnit::Init = Self` because the WU value IS the registration
data (typically a unit struct). `Kit::Init = Self` for the same
reason. Production blanket impls: `impl<W: WorkUnit> Provider for W`
with `Dispatch = UnitDispatch<W>`; `impl<K: Kit> Provider for K`
with `Dispatch = KitDispatch<K>`.

Platform providers (memory, threads, clock) impl Provider directly
in the consumer crate (or via a blanket impl on
`MemoryProviderApi` / `ThreadPoolApi` / `ClockApi` in
`hilavitkutin-providers` / `hilavitkutin-api`, depending on where
the trait lives). Their `Dispatch = PlatformDispatch<Self>`.

### Decision 6: ProviderKind enum

```rust
pub enum ProviderKind {
    WorkUnit,
    Resource,
    Column,
    Virtual,
    LinkedBin,
    Kit,
    Memory,
    Threads,
    Clock,
}
```

Documentation-and-debug aid only; the load-bearing dispatch goes
through `Dispatch`. `KIND` is `pub const` on each impl. Future kinds
extend the enum (and add a Dispatch router) without breaking the
trait surface.

## What this enables

- Single uniform call-site shape. `.with(value)` everywhere.
- Compile-time provider validation via `Provider`'s
  on_unimplemented diagnostic.
- Per-kind typestate routing through `Dispatch` associated type;
  one blanket impl handles all kinds.
- Reusable `HasTrivialCtor` trait available to any workspace crate.
- KitDispatch first-class: kits register at full fidelity from
  this round.
- Future provider kinds (whatever the next sibling provider type
  is) extend cleanly: new Dispatch struct, new ProviderKind
  variant, new Provider impl. No builder method changes.

## What this constrains

- All `add_*` / `with_*` methods retire. Pre-1.0 churn applies; no
  deprecated aliases.
- Every Provider impl must declare its Dispatch type. Mechanical
  but mandatory.
- HasTrivialCtor must land in notko before hilavitkutin-api can
  impl it on the wrappers. Round dependency: notko changes first
  (within the same round, but ordered in the src CL).
- Platform-provider impls move out of `with_memory` / `with_threads`
  / `with_clock` ergonomic specials and into the unified `.with`
  surface. Consumer code that called `with_memory(impl)` becomes
  `with(impl)` (mechanical migration; same value, one method).

## Open items the doc CL must resolve

These are not unresolved design questions; they are mechanical
specifications the doc CL fills in:

1. Exact module placement: `notko::ctor` vs `notko::traits::ctor` vs
   inline at notko crate root. Notko's existing layout dictates.
2. Whether `Provider`, `Dispatch`, and the four router structs live
   in `hilavitkutin-api::store` (existing module), `::provider` (new),
   or at crate root. Existing layout dictates.
3. The `ProviderKind` enum's exact variant ordering and whether it
   carries `repr` for debug-friendly discriminants.
4. Whether `WorkUnit` + `Kit` keep their existing supertrait sets
   or get reshaped (likely no; just add Provider as supertrait).
