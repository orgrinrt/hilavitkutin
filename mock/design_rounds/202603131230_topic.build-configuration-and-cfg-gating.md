## Purpose

The dispatch and optimisation topic established that morsel sizing, core
assignment, and cache-tier decisions depend on hardware detection (L1/L2
per core type, core counts, cache line sizes). Three research documents
catalogue the full space of compiler flags, nightly features, linker
options, and LLVM passes available to us. This topic decides:

- What goes where: static config (Cargo.toml/config.toml) vs computed
  (build script / xtask) vs runtime detection
- What `hilavitkutin-build` owns and its public API
- Which cfg flags the project defines and how they're emitted
- How profiles scale (dev -> test -> bench -> release -> release-optimised)
- When/how Polly, PGO, BOLT, and AutoFDO enter the pipeline
- What nightly feature gates every crate enables

## Context from prior discussion

The following positions were reached in conversation but never recorded
as design decisions:

**Three-layer split proposed:**
1. **xtask/build orchestrator:** controls LLVM/rustc flags per invocation
   (build-time, affects codegen quality). Calls cargo with computed
   RUSTFLAGS. Handles CPUID/sysctl detection for flag selection.
2. **Detection crate build.rs:** emits `cargo:rustc-cfg=...` flags for
   code-level gating (which code paths are compiled in). Workspace-wide
   via dependency propagation.
3. **Runtime detection in the library:** CPU topology for morsel sizing,
   core assignment, heterogeneous dispatch. Lives in the execution engine.

**Target matrix dimensions identified:**
1. ISA vector width: NEON 128-bit / AVX2 256-bit / AVX-512 512-bit / SVE variable
2. Core uniformity: homogeneous vs heterogeneous (P/E, big.LITTLE)
3. Vendor uarch quirks: Intel errata, AMD split-AVX2 (Zen1/2), Apple 128B cache lines
4. Cache topology: L1 per core type, L2 sharing, cache line size (64 vs 128)

**cfg flags proposed (SUPERSEDED -- all handled by existing
mechanisms: target_feature, target_vendor, target_arch, or the
RUSTC_WORKSPACE_WRAPPER. No custom cfg flags in the final model):**
```
cfg(cpu_heterogeneous_cores)   -- runtime detection (LazyLock)
cfg(cpu_vendor_intel)          -- cfg(target_vendor) + wrapper
cfg(cpu_vendor_amd)            -- cfg(target_vendor) + wrapper
cfg(cpu_cache_line_128)        -- WRONG: 64B everywhere for alignment
cfg(cpu_sve)                   -- cfg(target_feature = "sve")
cfg(cpu_avx512)                -- cfg(target_feature = "avx512f")
cfg(cpu_split_avx2)            -- handled by -C target-cpu=znver1/2
```

**Profile hierarchy discussed (STALE -- superseded by Q3 decisions):**
- dev: LLVM default (not Cranelift)
- dev-opt: LLVM with optimization for testing MIR/LLVM behavior
- bench: LLVM, release-level optimisation, no strip
- release: opt=3, lto, codegen-units=1, panic=abort, strip
- profiling: release + line-tables-only debug info

## Already decided (from prior conversation, not yet committed)

**D1. hilavitkutin-build is a shared crate, not an xtask.**

"Build script is exactly the correct thing, we just write a separate
shared crate for the shared stuff and then use it as build time dep."

hilavitkutin-build is a shared crate used as `[build-dependencies]` by
crates that need cfg emission. Each consumer crate's build.rs calls
into hilavitkutin-build. No xtask wrapper around cargo.

**D2. No unilateral decisions on what to pursue.**

Polly, PGO, BOLT, AutoFDO, and any other optimisation technique are
not dismissed without explicit user decision. The research documents
catalogue the options; the design round decides adoption.

**D3. Nightly Rust with feature gates.**

The project uses nightly. Unstable features are adopted after auditing
for likelihood of stabilisation and known ICE avoidance. Already
identified for adoption: min_specialization, min_generic_const_args,
adt_const_params, const_trait_impl.

**D4. No YAGNI, no "push to v2."**

"We have no release/ship pressure, we always design for the ideal."
Build infrastructure is designed for the full target, not a minimal
viable version.

**D5. Vendor-specific flags are not applied globally.**

Intel-specific mitigations (SKX102 branch alignment) should not apply
to AMD targets. Detection and gating are required.

## Resolved questions

**Q1. What does hilavitkutin-build own?** -- RESOLVED

hilavitkutin-build is a shared build-dependency crate (D1). What scope
does it cover?

Candidates:
- CPUID detection (x86) and sysctl/cpuinfo (ARM) for hardware topology
- cfg flag emission (cpu_vendor_intel, cpu_heterogeneous_cores, etc.)
- LLVM flag computation (inline-threshold, vendor-specific flags)
- Profile-aware flag selection
- Polly .so detection and -Z llvm-plugins flag emission
- MIR inlining threshold configuration

Or is it purely detection + cfg, with LLVM flags staying in static
config (.cargo/config.toml)?

**Decision: four-part build configuration model.**

(Originally five parts. Part 2 (custom cargo features) and part 3
(profile-rustflags) were superseded by the wrapper discovery. See
strikethrough text below for history.)

**1. hilavitkutin-build** (build-dep crate) handles cfg flag emission.

Named with meta prefix = fully internal dependency. Every crate's
build.rs calls:

```rust
fn main() {
    hilavitkutin_build::configure().run();
}
```
(SUPERSEDED — see Q4f for the full builder API. The `Meta::prepare()`
pattern was replaced by `configure()` which handles both cfg emission
and all build infrastructure setup.)

Scope:
- Reads cargo features and target env vars (CARGO_CFG_TARGET_ARCH,
  CARGO_CFG_TARGET_VENDOR, CARGO_CFG_TARGET_FEATURE)
- Emits cargo:rustc-cfg flags based on the active feature set
- Runs in every crate's build.rs (cfg flags are per-crate)
- resolve() prints all cargo:rustc-cfg lines and cargo:rerun-if-changed
- Does NOT do live CPUID/sysctl detection -- features declare intent,
  no need to detect what you already know you're targeting
- Does NOT handle LLVM codegen flags

build.rs is pre-build only. resolve() IS the complete build.rs action.

**2. No custom cargo features for hardware targeting.**

~~Originally "Cargo features as single source of truth for hardware
target." Superseded: the wrapper + existing CARGO_CFG_* + target_feature
handle everything. Custom features would create a parallel system
with divergence risk.~~

ISA capabilities: `#[cfg(target_feature = "avx2")]` from `-C target-cpu`.
Vendor identity: `CARGO_CFG_TARGET_VENDOR` in build.rs and wrapper.
Architecture: `CARGO_CFG_TARGET_ARCH` in build.rs and wrapper.
Cache/topology: runtime detection via LazyLock, with hilavitkutin-build
emitting const approximations from CARGO_CFG_TARGET_FEATURE heuristics.

compile_error! guards remain for any future mutually exclusive feature
groups if needed, but the current model needs none.

**3. RUSTC_WORKSPACE_WRAPPER** handles ALL LLVM codegen flags.

~~profile-rustflags was originally here. Dropped: nightly cargo feature
with known bugs (not propagated to CARGO_ENCODED_RUSTFLAGS, doesn't
trigger recompile, incompatible with build-std, no stabilisation
target). The wrapper replaces it entirely.~~

The wrapper computes flags from three heuristic axes:
- **Target** (CARGO_CFG_TARGET_ARCH, CARGO_CFG_TARGET_VENDOR,
  CARGO_CFG_TARGET_FEATURE): which target-specific flags to inject
- **Tier** (inferred from `-C opt-level=N` in the rustc args the
  wrapper receives): which optimisation-tier flags to inject
- **Passes** (`-C passes=` and `-C llvm-args=--enable-*`): which
  LLVM passes to activate beyond the default pipeline

Each combination of (target, tier) gets EXACTLY the flags optimal for
it. Not "universal safe flags." Not "flags that happen to be no-ops at
low opt-levels." Explicit, per-combination tuning.

The wrapper script (`target/hilavitkutin-build/rustc-wrapper`) can grow to
hundreds or thousands of lines. That is fine. More precision is more
value. The concrete flag assignments per (target, tier) pair are
decided as part of Q3, informed by the LLVM flags research.

Cargo profiles handle only the standard profile keys (opt-level, lto,
codegen-units, panic, strip, debug). No rustflags in profiles.

**4. compile_error!** guards for mutually exclusive features.

Features are additive in cargo and unify across the workspace. Mutual
exclusivity is enforced by compile_error! on conflicting combinations.
The error message includes a hint pointing to the correct usage.

**5. target_feature** for ISA capabilities.

ISA extensions (avx2, avx512, neon, sve) use the existing stable
`#[cfg(target_feature = "...")]` mechanism. These come from
`-C target-cpu` and `-C target-feature` -- no hilavitkutin-build
involvement needed. hilavitkutin-build handles only the things cargo
can't express natively (vendor, topology, cache characteristics).

**Q2. What does the project need custom features for?** -- RESOLVED

The initial Q2 resolution defined a three-tier feature model with
custom SIMD, L1, cache-line, and core-count features. This was
superseded by the discovery that the Rust compiler and Cargo already
handle most of what we were trying to build.

**What the compiler already provides (no custom features needed):**

1. `-C target-cpu=native` (or specific: skylake, znver3, apple-m1)
   handles vendor-specific LLVM tuning automatically. LLVM knows the
   uarch and applies the right scheduling model, branch alignment
   mitigations, and vectorisation strategy.

2. `#[target_feature(enable = "avx2")]` provides function-level SIMD
   dispatch. No cargo features needed for SIMD code gating.

3. `#[cfg(target_feature = "avx2")]` for compile-time gating of SIMD
   code paths. Automatically set by `-C target-cpu`.

4. `std::arch::x86_64::*` for explicit SIMD intrinsics when
   auto-vectorisation falls short. portable_simd is unsuitable (broke
   Polars Jan 2026, no stabilisation after 4+ years).

5. `CARGO_CFG_TARGET_FEATURE` env var in build.rs contains the
   comma-separated list of enabled target features. hilavitkutin-build
   can read this to derive constants.

**What the compiler does NOT know (but we handle without custom
features):**

- L1 cache sizes per core type
- Core counts and P/E ratio
- Cache line sizes (64B data alignment vs 128B prefetch on Apple)
- Whether cores are heterogeneous

No custom cargo features for these. Instead, feed values into existing
built-in mechanisms (CARGO_CFG_*, target_feature, the wrapper's
heuristics, runtime LazyLock detection). More data and more accurate
data through built-in channels; no parallel custom flag system that
creates divergence points.

Derivability and approximation:

- **Cache line**: default 64B. Apple Silicon has 128B coherency units
  relevant for prefetch configuration (`--cache-line-size=128`) though
  the actual CPU cache line is 64B. hilavitkutin-build generates a const
  for cache line size; default 64, adjusted per target if evidence
  warrants it. If CPUs with larger actual cache lines surface, we
  re-adapt and approximate those rather than special-casing.
- **L1 sizes**: may be approximable by synthesising target features
  (e.g. AVX-512 + specific feature set implies a known CPU generation
  with known L1 sizes). Needs research.
- **Heterogeneity**: modern Apple Silicon is always P/E. Intel 12th
  gen+ has hybrid. Heuristic: apple + aarch64 = heterogeneous.
  Total core count available at startup; P/E split can use static
  context rather than const (research needed on whether this is
  sufficient for LLVM vectorisation).
- **Core counts**: total available at startup. P/E split may need
  runtime detection but could use `once_cell` / `LazyLock` for
  static initialisation.

**Decision: RESOLVED — no custom cargo features needed.**

hilavitkutin-build emits no custom cfg flags. The compiler provides
`target_feature` and `CARGO_CFG_TARGET_FEATURE`. The rustc wrapper
injects per-crate flags (pragma-driven) and `arvo_fast_math` when
the FastMath pragma is active. arvo detects hardware capabilities
in its own build.rs via `CARGO_CFG_TARGET_FEATURE`.

L1 cache sizes, core counts, P/E heterogeneity detection: these are
runtime concerns handled by LazyLock in hilavitkutin's scheduler,
not build-time cfg flags. They don't need custom cargo features.
Deferred to the hilavitkutin design round.

## Cargo configuration tools (discovered mid-discussion)

The following Cargo capabilities were discovered during Q2/Q3
discussion and significantly change the design space:

**1. `cargo --config` (CLI config override)**

Dynamic config injection on the command line. Can set any config.toml
value without editing files:
```
cargo build --config "build.rustflags=['-C','llvm-args=--foo']"
cargo --config "target.'cfg(target_arch = \"x86_64\")'.rustflags=[...]"
```

**2. `.cargo/config.toml` `include`**

Config files can include other config files:
```toml
include = [
    { path = "vendor-flags.toml", optional = true },
    "shared-config.toml",
]
```
Paths relative to the including config file. Only `.toml` files.

**3. `[env]` table in config.toml**

Set environment variables for all Cargo-invoked processes:
```toml
[env]
POLKA_TARGET = "apple-m1"
SOME_VAR = { value = "path", relative = true }
FORCED = { value = "val", force = true }
```

**4. `CARGO_CFG_TARGET_FEATURE`**

Available in build.rs. Contains comma-separated list of enabled
target features (mmx,sse,sse2,avx2,...). hilavitkutin-build can read
this to derive hardware properties and compute constants.

**5. `CARGO_CFG_<cfg>` env vars in build.rs**

Full set of target cfg values available:
- CARGO_CFG_TARGET_ARCH (x86_64, aarch64)
- CARGO_CFG_TARGET_VENDOR (apple, unknown)
- CARGO_CFG_TARGET_OS (macos, linux)
- CARGO_CFG_TARGET_FEATURE (mmx,sse,avx2,...)
- CARGO_CFG_FEATURE (activated cargo features)

**6. `[target.<cfg>]` rustflags**

Architecture-level rustflags in config.toml:
```toml
[target.'cfg(target_arch = "x86_64")']
rustflags = ["-C", "target-cpu=x86-64-v3"]
```
Cannot distinguish Intel from AMD (same triple), but handles
architecture-level defaults.

**7. `rustc-workspace-wrapper`**

Wraps every rustc invocation for workspace members:
```toml
[build]
rustc-workspace-wrapper = "path/to/wrapper"
```
Could inject vendor-specific flags based on env vars or config.

**8. `[profile.<name>.build-override]`**

Override build-script compilation settings within a profile.

**9. `[profile.<name>.package.<name>]`**

Per-package profile overrides. Same keys as normal profile (minus
panic, lto, rpath).

**10. Custom target specifications**

JSON target spec files defining architecture, data layout, linker,
pre-link-args. Accessible via:
```
rustc +nightly -Z unstable-options --print target-spec-json
```

**11. `-C target-feature=+avx2,-sse4a`**

Explicit control over individual target features via rustflags.
Can be set in config.toml, profiles, or RUSTFLAGS. Fine-grained
cross-compilation without custom cargo features.

**Implications for the build model:**

- Profiles can be tier-only (dev, bench, release, profiling). No
  vendor axis. `-C target-cpu` handles vendor tuning.
- SIMD: use `#[target_feature]`, `std::arch`,
  and `#[cfg(target_feature)]`. Trust LLVM auto-vectorisation via
  the wrapper's per-target tuning. No custom features. No
  portable_simd (unsuitable).
- Vendor-specific LLVM flags (the few that remain): `cargo --config`
  or `include` or `.cargo/config.toml` per-target sections.
- hilavitkutin-build reads CARGO_CFG_TARGET_FEATURE in build.rs to
  derive constants (morsel sizes, alignment, etc.) from the known
  target feature set.

## Research findings (completed)

Full details in mock/research/build-configuration-research.md.

**1. L1 sizes cannot be derived from target features.** Feature sets
only encode ISA capabilities, never cache geometry. LLVM has internal
cache models but they are not exposed to Rust. No CARGO_CFG var for
cache info.

**2. Runtime detection is the path for hardware topology.** LazyLock
overhead is one `mov` (x86) or `ldar` (aarch64) per function entry.
LLVM vectorises loops with runtime trip counts just fine; morsel size
and vector width are orthogonal. macOS sysctl provides exact P/E
counts + L1/L2 per core type. Linux sysfs provides hybrid topology.

**3. Cache line size is nuanced.** The actual CPU cache line is 64B
on all current architectures including Apple Silicon. But Apple
Silicon has 128B SoC-level DMA coherency units, which matter for
prefetch configuration: LLVM's `--cache-line-size=128` controls
prefetch distance and is appropriate for Apple targets. Crossbeam
uses 128 for false-sharing prevention (different concern). For data
alignment, 64B is correct everywhere. For LLVM prefetch tuning, 128B
is correct on Apple.

**4. Auto-vectorisation is the primary SIMD strategy.** DataFusion
found auto-vec 40% faster than hand-written SIMD. portable_simd is
unsuitable (broke Polars Jan 2026, no stabilisation timeline).
`#[target_feature(enable = "avx2")]` enables auto-vec with wider
registers for entire function bodies. `multiversion` crate handles
multi-ISA dispatch for distributed binaries.

**5. RUSTC_WORKSPACE_WRAPPER is the strongest stable mechanism for
conditional codegen flags.** Wraps every rustc call for workspace
members. Has access to all CARGO_CFG_* env vars. Can inject -C
target-cpu and any other codegen flags based on target arch/vendor.
Build scripts cannot emit -C flags (deliberate Cargo design decision).

**6. No major database engine does core-type-aware scheduling.**
SiliconDB (VLDB 2019) demonstrated 2x gains with explicit P/E
awareness. This is an area where polka-dots can lead. That "nobody
does it" is context for how, not an argument against pursuing it.

**7. `-C passes=` injects individual LLVM passes.** 551 passes
available in rustc's bundled LLVM 22.1.0 without any plugins. Key
data-relevant passes already built in: loop-distribute, loop-fusion,
loop-interchange, loop-flatten, loop-data-prefetch, load-store-
vectorizer, loop-unroll-and-jam. Activated via `-C passes=<name>` or
`-C llvm-args=--enable-<name>`. The wrapper injects these per
(target, tier) pair.

**8. `-C passes=` and `-C llvm-args=--enable-*` are distinct.**
`-C passes=` appends an extra run of a pass. `--enable-*` enables
a pass within the existing pipeline schedule. Per-pass, one or the
other is preferred depending on where in the pipeline the pass needs
to run.

**9. Diagnostic flags for tuning.** `-Z print-llvm-passes` shows
which passes run. `-Z time-llvm-passes` times each. `-C remark=<pass>`
prints optimisation remarks. `-Z llvm-time-trace` generates Chrome-
tracing JSON. These feed the benchmark/profiling infrastructure and
guide per-(target, tier) flag tuning.

**10. MIR-level optimisation is a separate layer.** `-Z mir-opt-level`,
`-Z inline-mir`, `-Z inline-mir-threshold` run before LLVM and are
orthogonal. MIR inlining can cross crate boundaries that LLVM misses
without fat LTO. Nightly-only; goes through the wrapper.

**11. Cross-function fusion requires inlining.** LLVM will not fuse
loops across function boundaries unless the callee is inlined. For
operator chains in the morsel pipeline, this means chain execution
functions must be inlineable or monomorphised (generics, not dyn).
Confirms the dyn ban. --inline-threshold tuning directly impacts
whether chained operators get fused.

**Q2b. How do vendor-specific LLVM flags reach the compiler?**
-- RESOLVED

**Decision: RUSTC_WORKSPACE_WRAPPER.**

A thin shell script wraps every rustc invocation for workspace
crates. It inspects CARGO_CFG_TARGET_ARCH, CARGO_CFG_TARGET_VENDOR,
and optionally CARGO_CFG_TARGET_FEATURE to determine which -C flags
to inject.

```bash
#!/bin/bash
RUSTC="$1"; shift
if [ "$CARGO_CFG_TARGET_VENDOR" = "apple" ]; then
    exec "$RUSTC" "$@" -C target-cpu=apple-m1
elif [ "$CARGO_CFG_TARGET_ARCH" = "x86_64" ]; then
    exec "$RUSTC" "$@" -C target-cpu=x86-64-v3
fi
exec "$RUSTC" "$@"
```

Configured in `.cargo/config.toml`:
```toml
[build]
rustc-workspace-wrapper = "target/hilavitkutin-build/rustc-wrapper"
```

Properties:
- Stable, works today
- Only affects workspace crates, not dependencies
- Full access to CARGO_CFG_* env vars for conditional logic
- Artifacts cache separately per wrapper config
- Can be overridden per-developer via env var
- For host builds: `-C target-cpu=native` gives best results

The wrapper's primary value is per-target flag dispatch: it sees the
exact target triple decomposed into CARGO_CFG_TARGET_ARCH,
CARGO_CFG_TARGET_VENDOR, CARGO_CFG_TARGET_FEATURE and injects flags
specific to that exact target. This is the vendor axis. The profile
dimension (tier) is orthogonal and handled by Cargo profiles.

The wrapper handles BOTH axes simultaneously: target-specific flags
from CARGO_CFG_* env vars, tier-specific flags from inspecting the
opt-level in the rustc arguments it receives. Cargo profiles set
opt-level/lto/codegen-units/panic/strip; the wrapper reads opt-level
and computes the full LLVM flag set for that (target, tier) pair.

This eliminates profile-rustflags, vendor profiles, and universal
flag configs. One mechanism: the wrapper.

**Q3. How do profiles scale?** -- RESOLVED (see Q4f pragma system + Q7 config)

With Q2b deciding on RUSTC_WORKSPACE_WRAPPER for vendor flags,
profiles are tier-only (no vendor axis).

**Decided — REVISED to match Q7 config and Q4f pragma system:**

Five profiles. Cargo-native settings in Cargo.toml; hilavitkutin
pragmas per-profile via the builder API.

- dev: Cranelift for workspace crates, LLVM for deps (cached). No
  wrapper pragmas. Fast iteration. `cargo build`.
- dev-opt: LLVM for all crates, opt-level=2. LoopOptimization
  pragmas active. For testing MIR/LLVM behavior. `cargo build --profile dev-opt`.
- release: LLVM, opt-level=3. Full pragma set: LoopOptimization,
  FastMath, ExpandedLto, Bolt, Pgo. `cargo build --release`.
- profiling: inherits release, keeps debug info for flamegraphs.
  LoopOptimization, MathPeephole, FastMath, Profiling, Pgo, Bolt pragmas. Runs benchmarks
  post-build to generate PGO/BOLT profiles.
  `cargo build --profile profiling`.
- ci: inherits release, thin LTO. Same pragmas as release. Tuned for
  CI caching. `cargo build --profile ci`.

The wrapper reads the active profile (see Q4f — profile detection via
generated context file) and applies the pragmas the consumer configured
for that profile. Per (target, profile) pair: exactly the flags that
profile's pragma set implies. No universal flags; explicit tuning.

**Research complete.** Two documents catalogue the full flag space:
- `202603131400_research.rustc-llvm-optimization-flags.md` — 1000+
  lines covering 11 flag categories, all delivery mechanisms, per-tier
  recommendations, PGO/BOLT/AutoFDO workflows
- `202603131400_research.llvm-plugin-passes.md` — `-Z llvm-plugins`,
  Polly, built-in LLVM passes (551 in LLVM 22.1.0), custom pass

**Key mechanisms the wrapper uses:**
- `-C target-cpu=<name>`: vendor-specific LLVM tuning per target
- `-C target-feature=+feat,-feat`: fine-grained ISA control per target
- `-C llvm-args=--flag=value`: LLVM backend tuning per (target, tier)
- `-C passes=<pass-name>`: inject specific LLVM passes beyond default
  pipeline per (target, tier)
- `-Z` flags: nightly rustc flags (MIR opt level, inline-mir, etc.)

**Flag categories to assign per (target, tier):**
1. Inlining: --inline-threshold, -Z inline-mir-threshold, function
   specialization (--funcspec-*)
2. Vectorisation: --force-vector-interleave, --enable-interleaved-mem-
   accesses, --enable-histogram-loop-vectorization
3. Loop passes: --enable-loop-distribute, --enable-loopinterchange,
   --enable-unroll-and-jam, --enable-loop-flatten
4. Memory: --enable-gvn-hoist, --enable-gvn-sink, DSE tuning
5. Code layout: --hot-cold-split, --enable-ext-tsp-block-placement,
   --enable-machine-outliner
6. Prefetch: --aarch64-enable-loop-data-prefetch, --cache-line-size,
   --prefetch-distance
7. Branch: --x86-branches-within-32B-boundaries (Intel only)
8. MIR: -Z mir-opt-level, -Z inline-mir, -Z cross-crate-inline-threshold
9. Diagnostics: -Z print-llvm-passes, -Z time-llvm-passes, -C remark

Concrete per-(target, tier) flag tables are the next step for Q3
resolution.

**Q4. Polly and LLVM plugin passes**

Research findings from three documents:
- 202603131600_research.polly-llvm-rust-ir.md (Polly vs Rust IR)
- 202603131500_research.per-crate-backends-and-domain-optimizers.md
- 202603131700_research.llvm-passes-beyond-polly.md

Plus live container test (aarch64-linux nightly, Apple Container).

1. **Polly works but has friction with Rust.** SCoP detection struggles
   with Rust-generated LLVM IR (bounds checks, panics, enum
   discriminants). Version coupling is strict (must match rustc's LLVM
   exactly). Compile time overhead ~1.4%. Polly-friendly Rust patterns:
   `get_unchecked` loops, `chunks_exact(N)`, `#[repr(C)]` structs,
   `#[inline(always)]`. Error block heuristic (`-polly-allow-error-blocks`)
   helps in theory but is defeated by LLVM's earlier optimization passes
   in practice.

2. **No platform limitation for plugin loading.** `-Z llvm-plugins`
   works on aarch64-unknown-linux-gnu with stock nightly. The restriction
   is dynamic LLVM linking, not architecture. Official nightly Linux
   builds already use dynamic LLVM. macOS and Windows ship static LLVM
   in their nightly builds, so plugin loading does not work there natively.

3. **25+ non-default built-in LLVM passes.** Beyond O3 defaults, LLVM
   ships passes that are disabled by default but available via `-C passes=`
   or `-C llvm-args=`: loop-distribute, loop-fusion, loop-interchange,
   loop-data-prefetch, IRCE, loop-predication, histogram vectorization,
   unroll-and-jam, loop-flatten, hot-cold-split, function specialization.
   These work on all platforms, no plugin needed.

4. **Pass ordering matters.** `-C passes=` appends after the default
   pipeline, which is suboptimal for pre-vectorizer passes. Loop-interchange,
   IRCE, and loop-predication should run BEFORE the vectorizer;
   loop-data-prefetch should run AFTER. The PassBuilder extension point
   API (`registerVectorizerStartEPCallback`, `registerOptimizerLastEPCallback`)
   places passes at the correct pipeline position. This requires an LLVM
   pass plugin (~40 lines of C++), loadable via `-Z llvm-plugins`.

5. **Polly adds value beyond built-ins.** Polyhedral loop tiling,
   GEMM detection, combined fusion+tiling that individual built-in passes
   cannot achieve. Worth pursuing for hot loop nests in the morsel
   processing pipeline. Available in Homebrew LLVM (statically linked
   into LLVM tools, `LLVM_POLLY_LINK_INTO_TOOLS=ON`) and NixOS LLVM
   (`enablePolly = true` by default).

6. **No custom rustc build needed.** The build orchestrator concept
   (building custom rustc with modified bootstrap.toml) is unnecessary.
   Stock nightly on Linux already ships dynamic LLVM. The workflow:
   - Dev builds: native macOS, stock nightly, no plugins
   - Release builds: aarch64-linux container (Apple Container on macOS 26),
     stock nightly, load plugin `.so` via `-Z llvm-plugins` or RUSTFLAGS
   - The LLVM pass plugin is a small C++ file compiled against LLVM
     headers in seconds, not a full LLVM/rustc rebuild

7. **Container workflow for release builds.** Apple Container (Apache 2.0,
   uses macOS Virtualization.framework, boots in <1 second) runs
   aarch64-linux natively on Apple Silicon. A minimal Debian container
   with rustup nightly is the release build environment. No VM overhead,
   no Rosetta, no cross-compilation complexity.

8. **Industry confirmation.** Intel, AMD, ARM, Apple, Google all upstream
   their optimizations. No secret compiler sauce exists outside LLVM trunk.
   Academic research (Minotaur, domain-specific vectorization) is prototype
   stage. The passes we need are already in LLVM; the gap is pipeline
   ordering and activation, solved by the plugin approach.

Resolved questions:
- Polly requires `-Z llvm-plugins` for the full polyhedral optimizer; the
  individual named passes (`polly-canonicalize`, `polly-opt-isl`,
  `polly-codegen`) can load via `-C passes=` but miss Polly's scheduling.
- No need to automate rustc building. Automate plugin compilation only
  (small C++ file against LLVM headers from nightly sysroot).
- Container-based release builds replace the "Linux VM" concept; Apple
  Container is lighter, faster, and native.
- Yes, writing targeted LLVM pass plugins is the recommended path: a
  single plugin `.so` that registers passes at correct extension points.

**Q4a. IR cleanup passes for Polly viability**

The research catalogued Polly's SCoP detection failures (bounds checks,
panics, landing pads, non-affine trip counts) and separately catalogued
all available passes. The missing analysis: which passes fix which
Polly-hostile patterns, and in what order.

Three Rust IR patterns kill Polly. Each has a corresponding fix chain:

**Pattern 1: bounds checks (the primary killer).**
Every `slice[i]` generates `icmp uge → branch → panic_bounds_check →
unreachable`. Three passes address this directly:
- IRCE: splits loop into pre/main/post; main loop body has no range
  checks. The main loop becomes a clean affine region for Polly.
- LoopPredication: converts per-iteration `i < len` guard into a single
  loop-invariant check hoisted before the loop. Same effect as
  `get_unchecked()` without `unsafe`.
- ConstraintElimination: already in O3 default, but running again after
  IRCE/Predication catches redundant checks they exposed.

If IRCE eliminates the bounds check, the panic path becomes unreachable.
Then SimplifyCFG removes the dead branch. What remains is a clean loop
body with affine accesses — exactly what Polly's SCoP detector wants.

**Pattern 2: panic/unwind paths.**
- `panic = "abort"` in Cargo profile eliminates ALL landing pads and
  unwind paths. No LLVM pass needed; it is a codegen setting. Critical
  for Polly viability.
- HotColdSplit extracts cold code (panic handlers, error paths) into
  separate functions. The cold call remains in its own basic block,
  which Polly can treat as an error block via `-polly-allow-error-blocks`.

**Pattern 3: non-affine trip counts from iterator machinery.**
`indvars` and `loop-rotate` (both in O3 default) handle simple cases.
Complex iterator chains remain opaque to SCEV. No pass solves this
generically; the fix is coding patterns (simple `for i in 0..n` with
direct indexing, `chunks_exact(N)`, `get_unchecked`).

**The preparation pipeline.** These passes should run BEFORE Polly, in
this order, at the VectorizerStartEP extension point:

1. IRCE — eliminate bounds checks
2. LoopPredication — hoist remaining per-iteration guards
3. SimplifyCFG — remove dead panic branches (now unreachable)
4. ConstraintElimination — mop up leftover redundant comparisons

Since Polly also registers at VectorizerStartEP, callback registration
order determines execution order. Our cleanup callbacks must register
BEFORE Polly's callbacks so the IR is clean by the time Polly sees it.

**Q4b. Passes that directly optimize columnar access patterns**

Beyond Polly cleanup, these passes address the specific patterns in a
morsel-driven columnar engine. All are built into LLVM (Apache 2.0 with
LLVM Exception), already linked into `libLLVM.so`. The plugin calls
their constructors and registers them at the correct pipeline position.

| Pass | Pattern it addresses | Our use case |
|------|----------------------|-------------|
| LoopInterchange | Wrong nested loop order (row vs column) | Multi-column processing where outer loop iterates columns |
| LoopDistribute | Carried dependency preventing vectorization | Loop reading column A, computing, writing column B with dependency on B |
| LoopFusion | Adjacent loops over same range | Back-to-back operators on same morsel (filter then project) |
| LoopDataPrefetch | Sequential streaming exceeding L1 | Every column scan; prefetch next cache line |
| SeparateConstOffsetFromGEP | Multi-index GEPs with invariant base | Column base pointer invariant, row offset varies; split enables LICM on base |
| LoopVersioningLICM | Aliased pointers preventing LICM | Multi-column ops through slice pointers where LLVM cannot prove non-aliasing |
| Histogram vectorization | `buckets[indices[i]] += step` | GROUP BY aggregation; exactly this access pattern |
| LoopIdiomVectorize | Find-first-mismatch, find-first-of | Filter predicates scanning a column for matches |

**Q4c. Full plugin pass registration plan**

The plugin registers passes at two extension points. No pass source code
is copied; these are existing LLVM pass classes instantiated and placed
at the correct pipeline position.

VectorizerStartEP (before vectorizer):
1. IRCE — bounds check elimination
2. LoopPredication — guard hoisting
3. SimplifyCFG — dead branch cleanup after IRCE/Predication
4. LoopInterchange — fix nested loop order for sequential access
5. LoopDistribute — break dependent loops for partial vectorization

OptimizerLastEP (after vectorizer):
6. LoopDataPrefetch — prefetch the vectorized access pattern
7. SeparateConstOffsetFromGEP — CSE of column base pointers
8. LoopFusion — fuse adjacent loops (EXPERIMENTAL, only when LoopFusion
   pragma is active). Placed at OptimizerLastEP because it needs to see
   adjacent loops from inlined functions. NOTE: if results are
   undesirable, experiment with VectorizerStartEP placement instead —
   fusing before vectorization may produce better vector code but
   misses loops that become adjacent only after inlining.

**Flag ownership by pragma (every flag has exactly one owner):**

`LoopOptimization` injects:
- `-Z llvm-plugins=<path>/polka-passes.so` (plugin: IRCE, LoopPredication,
  SimplifyCFG, LoopInterchange, LoopDistribute, LoopDataPrefetch,
  SeparateConstOffsetFromGEP)
- `-C llvm-args=-enable-loop-distribute` (also enables built-in pipeline slot)
- `-C llvm-args=-enable-loop-versioning-licm`
- `-C llvm-args=-enable-interleaved-mem-accesses`
- `-C llvm-args=-loop-prefetch-writes`
- `-C llvm-args=-enable-histogram-loop-vectorization`

`MathPeephole` injects:
- `-Z llvm-plugins=<path>/math-peephole.so`

`FastMath` injects:

`Polly` injects:

- `lto = "fat"`
- `codegen-units = 1`

**Q4d. Third-party LLVM pass plugins**

Surveyed the ecosystem. Two categories: directly useful and learn-from.

Directly useful:
- **Polly** (Apache 2.0, in LLVM tree) — polyhedral loop optimizer. Tiling,
  fusion, parallelization, GEMM detection. Ships with Homebrew/NixOS LLVM.
  Our IRCE/LoopPredication cleanup (Q4a) is specifically designed to make
  Polly's SCoP detection viable on Rust IR.
- **xpass / MathPeephole** (MIT, github.com/shibatch/xpass) — floating-point
  peephole optimizer by the SLEEF author. 7 rewrite rules that eliminate
  division and sqrt operations: fraction combining (`a/b + c/d → (ad+bc)/(bd)`),
  division-free comparisons (`a/b > d` → sign-bit logic), sqrt elimination
  (`w*sqrt(x) + y > z` → square both sides). Requires `-ffast-math`.
  Benchmarks show 35-77% latency reduction on matching patterns (Zen 2,
  Goldmont+, Coffeelake). Single C++ file, MIT license.

  Relevance: morsel aggregation (AVG = sum/count, STDDEV = sqrt(variance)),
  filter predicates with division, cost model computations. Plan: vendor
  the source into `scripts/llvm-passes/`, port from LLVM 14 to LLVM 22 API,
  integrate into `polka-passes.so` as a function pass at OptimizerLastEP.
  The rewrite logic is pure IR pattern matching; the port is mostly pass
  registration boilerplate.

- **Enzyme** (Apache 2.0, github.com/EnzymeAD/Enzyme) — automatic
  differentiation at LLVM IR level. Being integrated into rustc behind
  `#[autodiff]`. Not needed now, but relevant if we add gradient
  computation for ML-guided scheduling or cost models.

Learn-from:
- **llvm-tutor** (MIT, github.com/banach-space/llvm-tutor) — 11 example
  passes. Best reference for pass API patterns.
- **llvm-plugin-rs** (233 stars, github.com/jamesmth/llvm-plugin-rs) —
  LLVM passes in Rust via Inkwell. Supports LLVM 11-20 (not 22 yet).
  No loop pass support. Worth watching.
- **CompilerGym/RLCompOpt** (Meta, github.com/facebookresearch/CompilerGym) —
  RL for pass ordering. 123 LLVM passes as action space. Future direction
  if we want to learn optimal pass orderings for our IR patterns.

Not useful but noted:
- AFL++ instrumentation passes (fuzzing, not optimization)
- obfuscator-llvm (CFG flattening, anti-analysis)

Notable gaps: no AoS-to-SoA pass, no cache-oblivious tiling pass, no
profile-guided prefetch pass, no domain-specific data processing pass
exist anywhere. Our plugin is the first "data engine optimization preset"
targeting columnar workloads at the LLVM level.

**Pragma system**

Pragmas are build directives — instructions to hilavitkutin-build that
control HOW the build system processes crates. They don't change what
is being built, only how. The name comes from Greek "πρᾶγμα" (deed/
thing done), same etymology as C's `#pragma`.

Each pragma is a type implementing the `Pragma` trait:
```rust
pub trait Pragma: 'static {
    const NAME: &'static str;

    /// Pragmas that must also be active. Checked at configure() time.
    /// Use bare type for single requirement, All<(A, B)> for all-of,
    /// Any<(A, B)> for at-least-one-of. Default: no requirements.
    type Requires: PragmaRequirement = ();

    /// Flags to inject into rustc args when active.
    fn rustc_flags() -> Vec<String> { vec![] }
    /// Cfg flags to inject when active.
    fn cfg_flags() -> Vec<String> { vec![] }
    /// Post-build action to run after last crate compiles.
    fn post_build() -> Option<PostBuildAction> { None }
}

/// Requirement combinators.
pub struct All<T: IntoPragmaSet>(PhantomData<T>);  // every listed pragma
pub struct Any<T: IntoPragmaSet>(PhantomData<T>);  // at least one

/// Implemented for (), single pragmas, All<...>, Any<...>.
/// check() returns Ok or Err with the missing pragma name.
pub trait PragmaRequirement {
    fn check(active: &PragmaSet) -> Result<(), String>;
}

/// Enables multiple pragmas at once. Implemented for individual
/// pragmas and tuples up to arity 12.
pub trait IntoPragmaSet {
    fn pragmas() -> Vec<PragmaId>;
}
```

Example — `MathPeephole` requires `FastMath`:
```rust
impl Pragma for MathPeephole {
    const NAME: &'static str = "math-peephole";
    type Requires = FastMath;
    // ...
}
```

Example — `Profiling` requires at least one of `Pgo` or `Bolt`:
```rust
impl Pragma for Profiling {
    const NAME: &'static str = "profiling";
    type Requires = Any<(Pgo, Bolt)>;
    // ...
}
```

`configure().run()` walks every active pragma, calls
`T::Requires::check(&active_set)`. On failure, emits
`cargo::error=MathPeephole requires FastMath — add FastMath to the
pragma set for profile "release"`. The error message is generated
automatically from the pragma names. No manual error strings.

Built-in pragmas:
```rust
/// Loop restructuring: IRCE, LoopPredication, SimplifyCFG,
/// LoopInterchange, LoopDistribute (via polka-passes.so)
pub struct LoopOptimization;

/// Polyhedral optimizer: tiling, fusion, GEMM detection
/// (Polly .so or -C llvm-args=--polly). Not in defaults —
/// requires Polly-enabled LLVM or container workflow.
pub struct Polly;

/// Float math peephole rewrites: fraction reduction, sqrt
/// elimination, division-free comparisons (via math-peephole.so).
/// Requires: FastMath.
pub struct MathPeephole;

/// Fast-math semantics (arvo_fast_math cfg, unsafe-fp-math LLVM flag)
pub struct FastMath;

/// Fat LTO + codegen-units=1 in generated config
pub struct ExpandedLto;

/// BOLT post-build binary rewriting (Linux ELF only, no-op otherwise)
pub struct Bolt;

/// Consume PGO profiles if they exist on disk
pub struct Pgo;

/// Run profiling benchmarks post-build (generates PGO + BOLT profiles).
/// Requires: at least one of Pgo or Bolt.
pub struct Profiling;

/// Rebuild std from source with optimization flags.
/// Not in defaults (build time cost).
pub struct BuildStd;

/// Parallel LLVM codegen (-Zthreads=N). Const generic controls
/// thread count. 0 = auto-detect via available_parallelism() at
/// configure() time.
pub struct ParallelCodegen<const N: usize = 0>;

/// Share monomorphized generics across crates (-Zshare-generics=y).
/// Reduces link time by deduplicating generic instantiations.
pub struct SharedGenerics;

/// Fuse adjacent loops over the same iteration range. EXPERIMENTAL.
/// LLVM's profitability model is a stub (always fuses). Beneficial
/// for back-to-back morsel operators. May hurt other patterns.
/// Not in any default set. Benchmark before enabling.
pub struct LoopFusion;
```

Parameterized pragmas use const generics. The default value (0 for
ParallelCodegen) is resolved at `configure()` time. The consumer
writes:
```rust
// auto-detect thread count
.enable::<(ParallelCodegen, SharedGenerics, LoopOptimization)>()

// explicit 4 threads
.enable::<(ParallelCodegen<4>, SharedGenerics, LoopOptimization)>()
```

This pattern extends to any pragma that needs a parameter, e.g. a
hypothetical `Prefetch<const DISTANCE: usize = 256>` for prefetch
distance tuning. The const generic composes naturally in the enable
tuple — no separate builder method, no runtime configuration.

**Pragma dependencies (via `type Requires`):**

| Pragma | `type Requires` | rationale |
|--------|----------------|-----------|
| `MathPeephole` | `FastMath` | xpass rewrites reassociate float ops; only correct under fast-math semantics |
| `Profiling` | `Any<(Pgo, Bolt)>` | benchmark profiles need a consumer; without Pgo or Bolt they go nowhere |
| all others | `()` | independent |

Dependencies are enforced automatically at `configure().run()` via
`PragmaRequirement::check()`. The error message is derived from the
pragma names — no manual error strings. Emitted via `cargo::error=`.

No implicit pull-in. Every active pragma is explicitly listed by the
consumer. No surprises.

**Pragma effects by mechanism:**

| Mechanism | Pragmas that use it |
|-----------|-------------------|
| Generated config (hilavitkutin-config.toml) | ExpandedLto, Pgo, BuildStd |
| Rustc wrapper (per-crate flag injection) | LoopOptimization, Polly, MathPeephole, FastMath |
| Post-build hook | Bolt, Profiling |

All resolved at `configure()` time, written into the generated artifacts.

**Default pragma sets per profile:**

| Profile detection | Default pragmas |
|-------------------|----------------|
| dev (OPT_LEVEL=0) | ParallelCodegen, SharedGenerics |
| dev-opt (OPT_LEVEL>0, debug) | ParallelCodegen, SharedGenerics, LoopOptimization |
| release (OPT_LEVEL=3, no debug) | ParallelCodegen, LoopOptimization, MathPeephole, FastMath, ExpandedLto, Bolt, Pgo |
| profiling (OPT_LEVEL=3, debug) | ParallelCodegen, LoopOptimization, MathPeephole, FastMath, Profiling, Pgo, Bolt |

Not in any default: Polly (requires special LLVM), BuildStd (build time),
LoopFusion (experimental profitability model).
ParallelCodegen defaults to 0 (auto-detect) in all profiles.
SharedGenerics only in dev/dev-opt (its benefit is link-time reduction
in debug builds; release builds use LTO which subsumes it).

**Consumer-defined pragmas:**

Consumers implement the `Pragma` trait for custom build directives:
```rust
struct MyCustomPass;
impl hilavitkutin_build::Pragma for MyCustomPass {
    const NAME: &'static str = "my-custom-pass";
    type Requires = ();
    fn rustc_flags() -> Vec<String> {
        vec!["-C llvm-args=-my-custom-flag".into()]
    }
}
```

The wrapper reads flags from the pragma's type definition. No
hardcoded knowledge needed for custom pragmas. Dependencies are
enforced the same way as built-in pragmas.

**Profile builder**

Per-profile configuration via closures. Each profile gets its own
builder scoping pragmas, benchmarks, and overrides to that profile:

```rust
impl ProfileBuilder {
    /// Enable one or more pragmas. Accepts single type or tuple.
    fn enable<T: IntoPragmaSet>(&mut self) -> &mut Self;

    /// Disable one or more pragmas (overrides defaults).
    fn disable<T: IntoPragmaSet>(&mut self) -> &mut Self;

    /// Clear all defaults for this profile.
    fn no_defaults(&mut self) -> &mut Self;

    /// Add a bench target from the current crate.
    fn bench(&mut self, name: &str) -> &mut BenchBuilder;

    /// Add a bench target from another workspace crate.
    fn bench_in(&mut self, crate_name: &str, bench_name: &str) -> &mut BenchBuilder;

    /// Disable a specific built-in benchmark.
    fn disable_builtin_bench(&mut self, name: &str) -> &mut Self;

    /// Disable all built-in benchmarks for this profile.
    fn no_builtin_benches(&mut self) -> &mut Self;
}

impl BenchBuilder {
    /// Additional arguments passed to the bench binary.
    fn with_args(self, args: &[&str]) -> Self;
}
```

Bench targets are referenced by name, not path. `configure()` resolves
names to binary paths using `CARGO_TARGET_DIR`, `CARGO_MANIFEST_DIR`,
and `cargo metadata`. The consumer never hard-codes paths.

**Detection-based defaults**

When the consumer does not call `.profile()` for a given profile,
`configure()` detects the active profile from build.rs env vars
(`PROFILE`, `OPT_LEVEL`, `DEBUG`) and applies sane defaults:

| Detection | Default pragmas |
|-----------|----------------|
| PROFILE=debug, OPT_LEVEL=0 | (none) |
| PROFILE=debug, OPT_LEVEL>0 | LoopOptimization |
| PROFILE=release, DEBUG=true | LoopOptimization, MathPeephole, FastMath, Profiling, Pgo, Bolt |
| PROFILE=release, DEBUG=false | LoopOptimization, MathPeephole, FastMath, ExpandedLto, Bolt, Pgo |

Custom profiles that inherit from release/dev are detected via
`PROFILE` (which cargo sets to the base profile name: `debug` or
`release`). The consumer calls `.profile()` to override when the
heuristic is wrong.

`.no_defaults()` on the outer builder disables all detection. Profiles
not explicitly configured via `.profile()` get nothing.

New pragmas added in future versions of hilavitkutin-build are NOT
added to the default set without a semver-major bump. No surprise
behavior changes on upgrade.

**Profile detection in the wrapper**

The wrapper needs to know which profile is active to apply the right
pragmas. `configure()` writes `target/hilavitkutin-build/build-context`
on every build from build.rs, capturing `PROFILE`, `OPT_LEVEL`, `DEBUG`,
`TARGET`, and `CARGO_CFG_TARGET_FEATURE`. The wrapper reads this file
to determine the active profile and its pragma set.

The generated wrapper has a baked-in profile → pragma mapping from the
consumer's `.profile()` calls. The build-context file provides the
runtime values to select which mapping entry applies.

**Package scopes (per-crate overrides)**

`.profile()` sets the baseline for a build profile. `.package()` overrides
pragmas and opt-level for specific crates within that baseline. Both use
the same `BuildContext` trait:

```rust
/// Shared interface for profile and package configuration.
pub trait BuildContext {
    fn enable<T: IntoPragmaSet>(&mut self) -> &mut Self;
    fn disable<T: IntoPragmaSet>(&mut self) -> &mut Self;
    fn opt_level(&mut self, level: impl Into<OptLevel>) -> &mut Self;
    fn no_defaults(&mut self) -> &mut Self;
}
```

`ProfileBuilder` implements `BuildContext` plus bench methods.
`PackageBuilder` implements `BuildContext` (no bench methods — benches
are profile-level, not per-crate).

`.package()` takes a string: exact crate name or glob (detected by
presence of `*`). One method, not separate `package` and `package_glob`.

Resolution: profile pragmas are the baseline. Package `enable` adds
pragmas for that crate. Package `disable` removes pragmas for that
crate. Package `opt_level` overrides the profile's opt-level for that
crate (generates `[profile.release.package.foo] opt-level = ...` in
the generated config).

Common package configurations can be extracted as plain functions:

     .disable::<(LoopOptimization, MathPeephole, FastMath)>();
}

fn traits_package(p: &mut PackageBuilder) {

**`configure()` mechanics**

Idempotent. On first call (no setup exists):
1. Detects platform: macOS Tahoe+ → Apple Container, Linux → native
2. Container lifecycle: create, LLVM download + trim + fstrim, persist
3. Plugin compilation (hash-based skip, all `*.cpp` in plugin dir)
4. Resolves profile pragmas + package overrides → per-crate flag sets
5. Validates pragma dependencies (`type Requires`) via `cargo::error=`
6. Generates `target/hilavitkutin-build/hilavitkutin-config.toml` (artifact, gitignored)
7. Generates `target/hilavitkutin-build/rustc-wrapper` with resolved flag table
8. Writes `target/hilavitkutin-build/build-context` (PROFILE, OPT_LEVEL, etc.)
9. Adds optional include to `.cargo/config.toml` (one line)
10. Emits `cargo:warning=hilavitkutin optimization active on next build`
11. Emits `cargo:rerun-if-changed` on generated artifacts and plugin sources

On subsequent calls (setup exists, hashes match): no-op.

**Two-build bootstrap (automatic, no manual intervention):**

Cargo parses `.cargo/config.toml` BEFORE any `build.rs` runs. So the
generated config does not take effect for the build that creates it.

Build 1 (first ever `cargo build`, `cargo check`, `cargo test`, any
command): `configure()` runs setup, build completes with stock
optimization. The generated config now exists on disk. `configure()`
emits `cargo:rerun-if-changed` on the generated config and plugin
sources so build.rs reruns when they change.

Build 2 (any subsequent cargo command): Cargo reads the include at
startup, finds the generated config, activates the wrapper. Cargo's
fingerprint check sees polka-* crates have different flags (new wrapper,
per-package profiles), recompiles them with plugins loaded. Non-polka
crates hit cache. This is the only "extra" recompile.

Build 3+: everything cached. Wrapper is transparent.

No env vars, no manual rebuild triggers. Cargo's own fingerprint system
detects the flag changes and forces recompilation of exactly the crates
that need it. `configure()` uses `cargo:rerun-if-changed` on generated
artifacts and plugin sources to re-check hashes when they change on
disk.

**No cargo subcommand.** `configure()` replaces `cargo polka setup`.
`cargo build --release` replaces `cargo polka build`. No `cargo-polka`
binary. The builder API replaces the CLI entirely.

**Optional include in `.cargo/config.toml`**

`configure()` adds to the consumer's `.cargo/config.toml`:
```toml
include = [
    { path = "../target/hilavitkutin-build/hilavitkutin-config.toml", optional = true },
]
```

`optional = true`: if the file does not exist (first build, or after
`cargo clean`), cargo silently skips the include. Zero cost. No error.

If the config already has an `include` array, append our entry.
If it has `include = "single.toml"`, convert to array form.
If no include exists, create one. The include line is the ONLY thing
written to the consumer's committed config.

**Generated `target/hilavitkutin-build/hilavitkutin-config.toml`**

Artifact (gitignored, blown away by `cargo clean`). Contents resolved
from profile pragmas + package overrides. No runtime resolution:
```toml

# from .package("polka-pipeline", |p| p.opt_level(3))
[profile.release.package.polka-pipeline]
opt-level = 3

# from .package("polka-cli", |p| p.opt_level("s"))
[profile.release.package.polka-cli]
opt-level = "s"
```

When the `ExpandedLto` pragma is active for a profile (it is by default
for release), the generated config includes:
```toml
[profile.release]
lto = "fat"
codegen-units = 1
```

Why these matter:
- `lto = "fat"` — merges all crates into one LLVM module at link time.
  Our plugins see all code (polka + consumer) in a single optimization
  unit. Cross-crate loop interchange, fusion, and inlining become
  possible. Critical for generic-heavy code where monomorphizations
  live in the calling crate, not the defining crate.
- `codegen-units = 1` — all functions within each crate in one LLVM
  module. Maximum inlining opportunity within a crate.

Other profile-level settings (`panic = "abort"`, `strip = "symbols"`)
belong in the consumer's Cargo.toml, not in the generated config.
hilavitkutin pragmas control only the things Cargo can't express or
that need to be generated dynamically. The consumer sets their Cargo
profiles how they want; the `ExpandedLto` pragma adds the LTO settings
that the LLVM plugins need to be effective.

The consumer can override in their Cargo.toml. Cargo.toml profile
values take precedence over config.toml profile values (manifest >
config).

The rustc wrapper is generated with a flat crate → flags map derived
from the resolved profile pragmas + package overrides. Every crate in
the optimization set is listed explicitly by name or glob.

**Q5. PGO and BOLT workflows**

PGO and BOLT are multi-step pipelines (build-instrument-run-rebuild).
They integrate into `hilavitkutin-build` via the builder API and the
rustc wrapper's post-build hook. AutoFDO is noted for completeness but
dropped from the design — PGO is strictly better when we control the
benchmarks (exact counts vs. sampling).

**The workflows:**

PGO (Profile-Guided Optimization):
1. Build with instrumentation: `-C profile-generate=<dir>`
2. Run benchmarks → `.profraw` files
3. Merge: `llvm-profdata merge *.profraw -o merged.profdata`
4. Rebuild: `-C profile-use=merged.profdata`

BOLT (Binary Optimization and Layout Tool):
1. Build release binary
2. Static BOLT: `llvm-bolt <bin> -o <bin>.bolt -reorder-blocks=ext-tsp
   -reorder-functions=hfsort+ -split-functions` (no profiles needed)
3. For profile-guided BOLT: run with `perf record -b`, then optimize
   with `-data=<profile>`

AutoFDO: noted, not designed. Uses `perf record` sampling instead of
compiler instrumentation. Lower quality than PGO. Only relevant when
you cannot deploy an instrumented binary (production traffic profiling).
Not our use case.

**Platform constraints:**

BOLT operates on ELF binaries only (no Mach-O). The binary format is
determined by the TARGET, not the build host:
- Linux target → ELF → BOLT works
- macOS target → Mach-O → BOLT does not work
- Cross-compiling Linux→macOS still produces Mach-O → no BOLT

For macOS targets: `-C llvm-args=-split-machine-functions` at compile
time gives partial BOLT-like benefits (hot/cold function splitting).
This works on all targets because it operates on LLVM IR, not the
final binary. Enabled automatically by `configure()` when PGO profiles
are present.

**Builder API:**

Minimal (defaults enable Pgo + Bolt on release, Profiling on profiling):
```rust
fn main() {
    hilavitkutin_build::configure().run();
}
```

Extended (consumer benchmarks on the profiling profile):
```rust
fn main() {
    hilavitkutin_build::configure()
        .profile("profiling", |p| p
            .bench("query_throughput")
            .bench_in("saalis-bench", "aggregation")
                .with_args(&["--exact", "morsel_hot_path"])
        )
        .run();
}
```

Selective (disable specific built-in benchmarks):
```rust
.profile("profiling", |p| p
    .disable_builtin_bench("morsel-throughput")
    .bench("custom_bench")
)
```

Bench targets are referenced by name, not path. `configure()` resolves
names to binary paths using `CARGO_TARGET_DIR`, `CARGO_MANIFEST_DIR`,
the active profile, and `cargo metadata`. The consumer never hard-codes
a target/ path or assumes a "release" profile exists.

`.bench("name")` — bench target in the current crate.
`.bench_in("crate", "name")` — bench target in another workspace crate.
`.with_args(...)` — additional arguments passed to the bench binary.

**Built-in benchmarks:**

`hilavitkutin-build` ships built-in benchmarks that exercise polka
framework hot paths: morsel iteration, column access, scheduler
dispatch, pipeline stage execution. These use the consumer's polka
config (reads `polka.toml`) with fixed synthetic input data.

Built-in benches run by default on release builds. They always merge
additively with consumer benchmarks. More benchmark coverage = more
complete profiles = better optimization. Built-in + consumer profiles
is strictly better than either alone; PGO profile merging sums execution
counters, and BOLT profile merging concatenates branch samples.

**PGO and BOLT share the same benchmark runs.**

A single benchmark execution produces both PGO and BOLT profiles
simultaneously:
- PGO: compiler instrumentation is compiled into the binary, writes
  `.profraw` on process exit
- BOLT: `perf record -b -e cycles:u` runs externally, writes `perf.data`

Both instruments are independent. One benchmark run → two profile
artifacts. No redundant work.

**Profile merging:**

After all bench runs complete:
1. Each run produces `.profraw` (PGO) and contributes to `perf.data` (BOLT)
2. `llvm-profdata merge *.profraw -o merged.profdata`
3. BOLT profiles merge via `perf` tooling
4. Results: `target/hilavitkutin-build/pgo/merged.profdata` and
   `target/hilavitkutin-build/bolt/merged.fdata`
5. Next release build: `configure()` detects profiles, adds flags

Merging is automatic and internal. The consumer does not call merge
functions or manage profile files.

**Automatic pipeline via the rustc wrapper post-build hook:**

The rustc wrapper detects "last crate compiled" (it knows the full
crate set from the generated config). After the last crate:

Release build 1 (no profiles exist yet):
1. Cargo finishes → binary produced
2. Wrapper post-build hook:
   a. Static BOLT runs immediately (seconds, unconditional on Linux ELF,
      reorders blocks/functions with static heuristics)
   b. Background: runs configured benchmarks with PGO instrumentation
      + perf record. Generates `.profraw` + `perf.data`. Merges.
3. Consumer gets their binary immediately. Background profiling is
   invisible. Static BOLT already applied (3-5% code layout improvement).

Release build 2 (profiles exist from previous run):
1. `configure()` finds `merged.profdata` → adds `-C profile-use=...`
   and PGO-complementary flags to generated config
2. Cargo recompiles polka-* crates with PGO
3. Wrapper post-build hook:
   a. Profile-guided BOLT replaces static BOLT (additional 5-10%)
   b. Background: re-profiles with PGO-optimized binary (profiles
      improve because the optimized binary has different hot paths)

Release build 3+: cached unless code changed. If code changed,
profiles from previous build still apply (LLVM discards mismatched
basic blocks, retains matching ones).

**Profile staleness:**

Profiles are NOT incrementally improving. Each instrumented run produces
fresh counters. `llvm-profdata merge` sums counters across runs but
merging old profiles (from changed code) with new profiles is harmful —
basic block IDs shift, counters reference wrong blocks. LLVM silently
ignores mismatched blocks.

Therefore: profiles live in `target/hilavitkutin-build/pgo/` (gitignored).
`cargo clean` deletes them. This is correct — stale profiles from old
code are worthless. Regeneration happens automatically on the next
release build via the post-build hook.

`configure()` records the git commit hash when profiles were generated.
If current HEAD diverges significantly (configurable threshold, default
50 commits), emits `cargo:warning=PGO profiles may be stale`. Never
refuses to build; stale profiles degrade gracefully (mismatched blocks
ignored, matching blocks still help).

**Complementary LLVM flags enabled with PGO data:**

When `configure()` detects PGO profiles, it also enables:
- `-C llvm-args=-vectorize-with-block-frequency` — PGO-enhanced
  vectorization (aggressive in hot, conservative in cold)
- `-C llvm-args=-split-machine-functions` — hot/cold function splitting
  (works on all targets including macOS; partial BOLT alternative)
- `-C llvm-args=-profile-guided-section-prefix` — groups hot functions
  for icache locality

Without profiles these flags do nothing. With profiles they compound.

**Optimization tiers (no user intervention needed per tier):**

| Tier | When | What runs | Benefit |
|------|------|-----------|---------|
| Stock | dev builds | nothing extra | baseline |
| Plugins | release, any platform | LLVM pass plugins via wrapper | loop optimization, prefetch |
| Static BOLT | release, Linux ELF | post-build reorder (seconds) | 3-5% code layout |
| PGO | release, second build onward | profile-use from prev benchmarks | 10-20% branch/inline |
| Profile-guided BOLT | release, Linux ELF, second build | BOLT with profiling data | additional 5-10% layout |
| Machine function split | release, all targets, with PGO | compile-time flag | partial BOLT for macOS |

Each tier is additive and automatic. The consumer configures pragmas
per-profile via `.profile("release", |p| p.enable::<(Pgo, Bolt)>())`.
Defaults already enable these for release. Everything else — profiling,
merging, detection, flag injection, post-build optimization — is
handled internally by `configure()` and the wrapper.

**Summary: what lives where**

| Concern | Where | When |
|---------|-------|------|
| Plugin compilation | `configure()` in build.rs | every build (hash-skip) |
| Package overrides → flags | `configure()` in build.rs | every build (hash-skip) |
| PGO profile detection + flags | `configure()` in build.rs | release builds (auto) |
| PGO + BOLT profiling | wrapper post-build hook (background) | after release builds |
| Static BOLT | wrapper post-build hook (immediate) | after release builds, Linux ELF |
| Profile-guided BOLT | wrapper post-build hook (immediate) | when BOLT profiles exist |
| Profile merging | wrapper post-build hook (internal) | after benchmarks complete |
| Built-in benchmarks | shipped with hilavitkutin-build | run by default on release |
| Staleness check | `configure()` in build.rs | every build (auto) |

**Q5a. Numeric primitives: `arvo` crate**

The math mode discussion (float vs fixed-point vs mixed, hardware-aware
backend selection, fast-math compatibility) produced a design for a
standalone numeric primitives crate. This is cross-project: polka,
loimu, and saalis all depend on it.

**Name:** `arvo` (Finnish: "value" / "worth" / "estimate"). Free on
crates.io. Lives in the polka-dots repo for now; extracted to its own
repo later. Other projects depend via git:
```toml
arvo = { git = "https://github.com/orgrinrt/polka-dots", branch = "main" }
```

**Scope:** `#![no_std]` numeric library. Fixed-point types (ported from
loimu-primitives), fast-math float wrappers (LLVM intrinsics), composable
numeric traits, hardware-aware backend selection.

**No monolithic `Numeric` trait.** `core::ops` (`Add`, `Sub`, `Mul`, `Div`)
is already available in `no_std`. Arvo adds only what `core::ops` doesn't
cover, as small composable traits:

- `TotalOrd` — min, max, clamp (float: assumes no NaN under fast-math)
- `Sqrt` — square root (hardware intrinsic or integer approximation)
- `Recip` — multiplicative inverse
- `Abs` — absolute value
- `FromConstant` — zero, one construction

Generic code composes bounds as needed: `T: Add + Mul + Sqrt` for
standard deviation, `T: Add + FromConstant` for count. No function
asks for more than it needs.

**Types:**

Fixed-point (ported from loimu):
- `UFixed<const I: u8, const F: u8>` — unsigned, `#[repr(transparent)]` over `u64`
- `IFixed<const I: u8, const F: u8>` — signed, `#[repr(transparent)]` over `i64`
- Strategy markers: `Hot` (lane-capped), `Precise` (full-width), `Warm` (runtime)
- `Fixed<I, F, S: Strategy>` — strategy-parameterized wrapper

Float wrappers (new for arvo):
- `FastFloat` — newtype over f32/f64, emits LLVM intrinsics, fast-math compatible, no_std
- `StrictFloat` — IEEE 754 strict, for correctness-sensitive paths

Semantic aliases (combined from loimu + polka domains):
- From loimu: `Ratio`, `Factor`, `Norm`, `Coord`, `Angle`
- For data processing: `Count`, `Average`, `Deviation`

**Mixing is the default.** A single morsel can have columns using different
numeric types. COUNT → `UFixed<32,0>`, AVG → `FastFloat`, VERSION →
`UFixed<8,8>`. The column's numeric type is part of its schema. The
pipeline dispatches per column via the trait bounds.

**Hardware-aware.** `hilavitkutin-build::configure()` emits cfg flags
(`polka_hw_sqrt`, `polka_hw_fma`, `polka_simd_width`) based on target
features. Arvo's `Sqrt` impl on `FastFloat` uses `#[cfg(polka_hw_sqrt)]`
to select hardware intrinsic vs integer approximation. Invisible to
callers.

**LLVM plugin integration.** The math-peephole pass (xpass) fires on
`FastFloat` operations (fraction reduction, sqrt elimination). Fixed-point
columns use integer arithmetic — no float passes needed, but loop
interchange and prefetch still apply. The per-crate flag set in
hilavitkutin-build depends on whether the crate uses fast-float operations.

**This topic records the design direction. A proper design round for
arvo will happen separately when implementation begins.** The crate
layout, trait details, loimu port scope, and conversion trait design
need their own topic.

**Q5b. Pipeline execution engine: `hilavitkutin` crate**

The pipeline and scheduler design is shared across three projects:
polka-dots (data compiler), saalis (query engine), and loimu (UI
framework, scheduler abstractions only). Rather than each project
reimplementing the same DAG scheduling, work-unit dispatch, and
pipeline stage abstractions, these extract into a standalone crate.

**Name:** `hilavitkutin` (Finnish: a contraption, a whirligig). Free on
crates.io, npm, PyPI, GitHub repo. Lives in the polka-dots repo for
now; extracted later. Other projects depend via git:
```toml
hilavitkutin = { git = "https://github.com/orgrinrt/polka-dots", branch = "main" }
```

**Scope:** `#![no_std]` (with `alloc`) pipeline execution engine.

What it owns:
- DAG scheduling (topological ordering, dependency tracking)
- Work-unit dispatch (the `WorkUnit` abstraction: has `execute()`,
  context is ECS-style `Read<T>` / `Write<T>`)
- Morsel chunking (splitting work into cache-friendly units)
- Pipeline stage abstraction (N→IR, IR→IR, IR→M transforms)
- Work-stealing scheduler (per-core queues, steal from neighbors)
- Progress tracking, cancellation, backpressure

What it does NOT own:
- Column storage, data model, schema (arvo, polka-data)
- Query planning, optimization (saalis-specific)
- UI layout, rendering, compositing (loimu-specific)
- IO, filesystem, network (platform-specific, std-gated consumers)

**Usage per project:**

polka-dots: full pipeline. `polka-pipeline` and `polka-scheduler` become
thin wrappers around `hilavitkutin` with polka-specific stage types
(template compilation, config generation, audit). The morsel execution
model from the dispatch-and-optimisation topic maps directly.

saalis: full pipeline. Query operators are pipeline stages. Morsels are
column chunks. The morsel-driven execution model (Leis et al.) is
`hilavitkutin`'s core abstraction.

loimu: scheduler abstractions only. Uses the work-unit dispatch and
DAG scheduling for frame-budget-aware task scheduling (render, layout,
animation). Does NOT use morsel chunking or the full pipeline — loimu's
"pipeline" is the compositor, which has different semantics.

**Relationship to hilavitkutin-build:**

`hilavitkutin` is a runtime crate, not a build tool. But it benefits
from the LLVM plugins: the morsel processing hot loops are exactly
where loop interchange, IRCE, prefetch, and PGO pay off. The
The `LoopOptimization` pragma applies to `hilavitkutin`
and any crate that uses its pipeline execution.

**This topic records the design direction. A proper design round for
hilavitkutin will happen separately.** The work-unit model, scheduler
design, morsel abstraction, and stage trait contracts need their own
topic. The dispatch-and-optimisation topic (202603121800) already
covers much of the execution model — hilavitkutin is the extraction
of that design into a reusable crate.

**Q6. Nightly feature gates -- confirmed assignments**

Research: mock/research/rust-nightly-features-for-type-constraints.md

**Always-on set (every crate in the workspace):**

`min_specialization` — layered trait defaults in the SDK. A
`ColumnValue` trait has a default impl; specific types specialize for
performance. Depends on the next-gen trait solver for soundness; known
unsoundness around lifetimes (#79457). Our specialization usage is
simple (no lifetime-dependent specialization). If the fix breaks our
code, we fix our code. Research score: 5.0.

`min_generic_const_args` — the `Check<{ size_of::<T>() <= 15 }>: IsTrue`
pattern for compile-time layout enforcement. THE solve for the column
value constraint problem (128-bit stride, 15-byte payload). 7/18 tasks
remaining as of research date. Research score: 5.0.

`adt_const_params` — newtypes and enums as const generic parameters.
Our `repr(C)` atoms are const-generic in several contexts. Known ICEs
exist in specific patterns (recursive ADTs in const position, unsized
const params). Resolution: document avoidance rules, not cfg-gate. The
ICE-triggering patterns are outside our usage. If an ICE surfaces, we
adjust the code, not disable the feature.

`const_trait_impl` — `const fn` in trait impls. Needed for compile-time
schema introspection: `ColumnValue::LAYOUT` as a const method. The
stdlib dropped and re-adopted this feature; it is actively worked on.
We use it for our own traits, not stdlib traits, so internal API churn
is manageable. Research score: 4.7.

**Where-useful set (enabled per-crate):**

`explicit_tail_calls` — `become` keyword for guaranteed TCO. Enables:
polka-lang-polka (recursive AST traversal: parser, expander). Without
it, deep nesting can stack overflow. Only relevant in crates with
recursive tree walking.

`loop_match` — jump-table dispatch for state machine inner loops.
Enables: polka-pipeline (pipeline stage dispatch), polka-lang-polka
(token state machine). Compiles match arms into a jump table instead
of a branch ladder.

`never_type` (`!`) — infallible return types. A pipeline stage that
cannot fail returns `Result<T, !>`. The compiler optimizes away the
error path. Useful across core crates. Not critical — `Infallible`
exists on stable as a workaround, but `!` is more ergonomic and the
compiler can reason about it better.

`build_std` — rebuild stdlib from source with our optimization
settings. Enables LTO across the std boundary. With our LLVM plugins,
rebuilding std means the stdlib's hot paths (iterator machinery, slice
operations) get our loop optimization passes too. This is a
a `hilavitkutin-build` pragma (`BuildStd`). Not in the default
pragma set due to build time cost. Enable per-profile:
`.profile("release", |p| p.enable::<BuildStd>())`.

**Framework-level feature flag:**

`no_std` default / `std` opt-in — all polka crates are `#![no_std]` by
default. A `std` feature flag gates std-dependent paths. This is
independent of `build_std` in hilavitkutin-build:
- `no_std` = the framework has no std dependency (portability, minimal
  deps, embeddable, WASM-compatible)
- `build_std` = the consumer's std is rebuilt from source with LLVM
  plugins (optimization of std's hot paths in the final binary)

A consumer can use both: polka crates compiled without std, consumer's
own binary links a plugin-optimized std. No conflict.

Every polka crate writes `no_std` first, then `#[cfg(feature = "std")]`
for the std-dependent path. The `std` feature propagates through the
dependency graph: enabling `std` on `polka-sdk` enables it on
`polka-primitives`. `alloc` is available in `no_std` mode via
`extern crate alloc`.

**Dropped:**

`macro_metavar_expr` — `${count()}`, `${index()}` in macro_rules.
Dropped. If a macro gets complex enough to need these, it should be a
proc macro instead.

**Tracking (not yet usable, design-aware):**

`scalable_vectors` — sized hierarchy for SVE/RISC-V V types. Morsel
processing directly benefits: variable-length SIMD without scalar
fallback loops. 2026 goal, not usable yet. Track for adoption when
available.

`reflection_and_comptime` — compile-time type reflection. Potential
replacement for proc-macro-based schema derivation. Groundwork-only in
2026. Research score: 4.7.

`dictionary_passing_style` — potential impact on our no-dyn
monomorphization strategy. 2026 experimental goal. Monitor for
conflicts with our architecture.

**Q7. Static config structure — confirmed split**

Research: mock/research/nightly-and-stable-optimisation-config.md (section 12)

Four config files, three committed, one generated. The principle: committed
configs are the reasonable defaults; generated config layers on top for the
optimized build. Cargo merges them (generated takes precedence on conflict).

**`rust-toolchain.toml`** (committed, workspace root)

Pins nightly channel and required components:
```toml
[toolchain]
channel = "nightly"
components = ["rust-src", "rustc-codegen-cranelift-preview"]
```

`rust-src` is needed for `build-std` (when enabled via hilavitkutin-build).
`rustc-codegen-cranelift-preview` is needed for fast dev builds (Cranelift
backend for workspace crates, LLVM for dependencies).

**`Cargo.toml`** (committed, workspace root)

Static build configuration. Profiles, feature gates, workspace lints,
unstable table:
```toml
[workspace]
members = [
    "mock/crates/*",
    "arvo",
    "hilavitkutin",
    "hilavitkutin-build",
    "hilavitkutin-build",
]

[workspace.package]
edition = "2024"
rust-version = "nightly"

# --- unstable features ---
[unstable]
codegen-backend = true

# --- profiles ---
[profile.dev]
opt-level = 0
debug = 1                    # line-tables-only (significant macOS speedup)
incremental = true
codegen-backend = "cranelift" # fast compile for workspace crates

[profile.dev.package."*"]
codegen-backend = "llvm"     # LLVM for deps — cached, perf matters for tests
opt-level = 2

[profile.release]
opt-level = 3
lto = "thin"                 # default; hilavitkutin-build upgrades to "fat"
codegen-units = 16           # default; hilavitkutin-build overrides to 1
strip = "symbols"

[profile.profiling]
inherits = "release"
debug = "line-tables-only"
strip = "none"

[profile.profiling.build-override]
opt-level = 0
codegen-units = 256

[profile.ci]
inherits = "release"
lto = "thin"
codegen-units = 1
```

Note: `[profile.release]` uses thin LTO and 16 codegen units as defaults.
These are the "reasonable without hilavitkutin-build" settings. When
the `ExpandedLto` pragma is active, the generated config overrides
these with fat LTO, codegen-units=1, panic=abort. Cargo merges the
generated config on top, so the upgrade is seamless.

Feature gates (nightly) are in each crate's `Cargo.toml`, not the
workspace root. The always-on set (`min_specialization`,
`min_generic_const_args`, `adt_const_params`, `const_trait_impl`) is
declared per-crate via `#![feature(...)]` in `lib.rs`.

**`.cargo/config.toml`** (committed, minimal)

Only three things: the hilavitkutin-build optional include, per-target
dev-build flags, and linker selection:
```toml
# generated config (gitignored artifacts, optional)
include = [
    { path = "../target/hilavitkutin-build/hilavitkutin-config.toml", optional = true },
]

[target.aarch64-apple-darwin]
rustflags = [
    "-Zthreads=8",
    "-Zshare-generics=y",
    "-C", "target-cpu=native",
]

[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = [
    "-Zthreads=8",
    "-Zshare-generics=y",
    "-C", "target-cpu=native",
    "-C", "link-arg=-fuse-ld=mold",
]

[target.aarch64-unknown-linux-gnu]
linker = "clang"
rustflags = [
    "-Zthreads=8",
    "-Zshare-generics=y",
    "-C", "target-cpu=native",
    "-C", "link-arg=-fuse-ld=mold",
]
```

`-Zthreads=8` parallelizes LLVM codegen within rustc (separate from
codegen-units, which is parallelism across compilation units).
`-Zshare-generics=y` shares monomorphized generic code across crates
in debug builds (significant link-time reduction).
`-C target-cpu=native` ensures LLVM uses the host CPU's features for
vectorization and scheduling. In CI, this changes to a specific target
like `x86-64-v3` or `apple-m4`.
`mold` linker on Linux for fast linking. macOS uses the default Apple
linker (fastest on macOS per Bevy benchmarks).

**`target/hilavitkutin-build/hilavitkutin-config.toml`** (generated, gitignored)

This is what `hilavitkutin_build::configure()` produces. It layers on
top of the committed config. Contents depend on what the consumer
enabled:
```toml
# always generated
[build]
rustc-wrapper = "target/hilavitkutin-build/rustc-wrapper"

# per-package profiles from .package() overrides
[profile.release.package.polka-pipeline]
opt-level = 3

# when ExpandedLto pragma is active
[profile.release]
lto = "fat"
codegen-units = 1
panic = "abort"

# when BuildStd pragma is active
[unstable]
build-std = ["core", "alloc", "std"]
```

**What hilavitkutin-build CANNOT emit from build.rs:**

`cargo:rustc-flags` does not exist. build.rs can emit `cargo:rustc-cfg`,
`cargo:rustc-env`, `cargo:rustc-link-lib`, but NOT arbitrary `-C` codegen
flags. This is deliberate (cargo#1293). The workaround is the generated
config.toml include + rustc wrapper, which is exactly what Q4f designs.

**Interaction between committed and generated config:**

Cargo's config merging: values from included files load first, then
the including file's own values merge on top (taking precedence). So
the committed `.cargo/config.toml` values (per-target rustflags) take
precedence over the generated config. This is correct — the generated
config controls the wrapper and profiles, the committed config controls
the per-target flags and linker.

For `[profile.release]`, the committed Cargo.toml sets the defaults;
the generated config overrides specific fields when the `ExpandedLto`
pragma is active. Cargo.toml profile values take precedence over config.toml
profile values (manifest > config), so if the consumer explicitly sets
`lto = "thin"` in their Cargo.toml, the generated `lto = "fat"` does
NOT override it. This is the right behavior — consumer's explicit
choice wins over generated defaults.

**Summary of ownership:**

| Setting | Committed file | Generated? |
|---------|---------------|------------|
| Nightly channel + components | rust-toolchain.toml | no |
| Profile defaults (dev, release, profiling, ci) | Cargo.toml | no |
| Cranelift for dev builds | Cargo.toml | no |
| Feature gates (#![feature(...)]) | per-crate lib.rs | no |
| Per-target rustflags (threads, share-generics, target-cpu) | .cargo/config.toml | no |
| Linker selection (mold, Apple ld) | .cargo/config.toml | no |
| Optional include line | .cargo/config.toml | no (written once) |
| Rustc wrapper path | generated config | yes |
| Per-package overrides (opt-level, pragma enable/disable) | generated config + wrapper | yes |
| Full release overrides (fat LTO, codegen-units=1) | generated config | yes |
| build-std | generated config | yes |
| PGO profile-use flag | generated config | yes (when profiles exist) |

**Q8. Workspace lint configuration — confirmed, three layers**

Research: mock/research/nightly-and-stable-optimisation-config.md (section 8)

Three lint layers, each enforcing a different level of the architecture.

**Layer 1: Workspace lints (Cargo.toml, standard rustc/clippy)**

```toml
# Cargo.toml (workspace root)
[workspace.lints.rust]
# safety
unsafe_op_in_unsafe_fn = "deny"

# code quality
unreachable_pub = "warn"
missing_debug_implementations = "warn"
unused_crate_dependencies = "warn"
redundant_imports = "warn"
let_underscore_drop = "warn"

# bug prevention
trivial_casts = "warn"
trivial_numeric_casts = "warn"
explicit_outlives_requirements = "warn"

[workspace.lints.clippy]
all = { level = "warn", priority = -1 }
pedantic = { level = "warn", priority = -1 }
# override noisy pedantic lints
module_name_repetitions = "allow"
must_use_candidate = "allow"
missing_errors_doc = "allow"
```

Per-crate opt-in:
```toml
# each crate's Cargo.toml
[lints]
workspace = true
```

Individual crates can override specific lints when justified (e.g.,
polka-derive allows `clippy::too_many_lines` because proc macros are
inherently long).

`missing_docs` is NOT enabled. We write docs where they add value,
not everywhere. Enforcing docs on every `pub` item creates filler.

`clippy::nursery` stays at allow (default). Nursery lints are
unstable and change between clippy releases; too volatile for CI.

CI enforcement:
```sh
RUSTFLAGS="-D warnings" cargo clippy --all-targets
```
All warnings become errors in CI. Developers see warnings locally;
CI blocks on them.

**Layer 2: Mockspace architecture lints (`cargo mock lint`)**

These are the existing mock workspace lints that check structural
rules. They run as part of `cargo mock` and block on violations:

- Every crate has `src/lib.rs` and `README.md.tmpl`
- Dependencies in `Cargo.toml` match the target architecture layers
- No upward dependencies (extensions → SDK only, core → SDK, CLI → core)
- No layer violations (data types don't leak upward, UI doesn't leak downward)
- Pub items in mock stubs match the design docs

These operate on the `Cargo.toml` dependency graph and the `lib.rs`
pub item signatures. They do NOT inspect code patterns within function
bodies.

**Layer 3: Custom domain lints (`mock/lints/`, source code patterns)**

These enforce design decisions that rustc and clippy cannot know about.
They inspect source code within crate files — struct fields, function
signatures, type annotations. They run as part of `cargo mock lint`
alongside the architecture lints.

| Lint file | What it rejects | Scope |
|-----------|----------------|-------|
| `no-raw-primitives.rs` | `f32`, `f64`, `i8`..`u128` in struct fields and pub fn signatures | All framework crates. Exempt: loop counters, local variables, indices, const values, FFI/`repr(C)` boundary types |
| `no-dyn.rs` | `dyn Trait` anywhere | All crates. No exceptions. |
| `no-std-any.rs` | `std::any::TypeId`, `std::any::Any`, `core::any::*` | All crates. No type erasure. |
| `arvo-types-only.rs` | Non-arvo numeric types in data-facing struct fields | polka-data, polka-pipeline, polka-catalogue, arvo. Exempt: internal computation locals |
| `no-unwrap.rs` | `.unwrap()` outside `#[cfg(test)]` modules | All crates. Already a CLAUDE.md rule; now enforced by lint. |
| `no-toml-value.rs` | `toml::Value` usage outside polka-data | All crates except polka-data |
| `no-ui-below-cli.rs` | `&Ui` parameter in function signatures | All crates except polka-cli |
| `sdk-traits-only.rs` | Concrete `impl` blocks (non-trait) in SDK crates | polka-primitives, polka-derive, polka-sdk. SDK exports only traits, `repr(C)` types, and proc macros. |

Exemption mechanism: `// lint:allow(no-raw-primitives)` comment on the
line above the violation. The lint parser recognizes this and skips.
This is intentional friction — the comment documents WHY the exemption
exists, and it is greppable for audit.

Difference from CLAUDE.md rules: CLAUDE.md states the rules in prose
for humans and agents. `mock/lints/` enforces them mechanically. If a
rule is in CLAUDE.md, it should have a corresponding lint. If a lint
exists, the CLAUDE.md rule can reference it instead of restating the
logic.

**`no-raw-primitives` in detail:**

This is the most nuanced lint. It must distinguish:

Rejected (data-facing positions):
```rust
pub struct ColumnStats {
    pub mean: f64,           // REJECT: use arvo::FastFloat or arvo::Average
    pub count: u64,          // REJECT: use arvo::Count (UFixed<64,0>)
}
```

Allowed (non-data positions):
```rust
pub fn process(morsel_idx: usize) {  // OK: index, not data
    for i in 0..n {                   // OK: loop counter
        let offset: usize = i * 16;  // OK: local computation
    }
}

#[repr(C)]
pub struct AbiHeader {
    pub version: u32,                 // OK: FFI boundary
}
```

The lint checks: struct fields, enum variant fields, pub fn parameter
types, pub fn return types. It does NOT check: local `let` bindings,
loop variables, const/static declarations, `#[repr(C)]` struct fields.

**`arvo-types-only` vs `no-raw-primitives`:**

These overlap but have different scopes. `no-raw-primitives` is broad
(all framework crates, all raw numeric types). `arvo-types-only` is
narrow (data-facing crates only, requires specifically arvo types not
just "not raw"). A struct field using a custom newtype that isn't from
arvo passes `no-raw-primitives` but fails `arvo-types-only`.

Both are needed: `no-raw-primitives` catches the obvious mistakes
everywhere; `arvo-types-only` enforces the numeric discipline in the
data layer specifically.
