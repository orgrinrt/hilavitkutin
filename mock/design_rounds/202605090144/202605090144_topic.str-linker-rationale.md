# Linker-section rationale for hilavitkutin-str

The workspace rule in `hilavitkutin/.claude/CLAUDE.md` forbids
"linker-magic registration (`inventory::submit!`, `#[ctor]`,
`#[distributed_slice]`, `.init_array`-based registries) in any
hilavitkutin crate". The `hilavitkutin-str` crate ships a custom
`.hilavitkutin_strings` linker section walked at startup-once to
populate the const interner. This is functionally similar to a
distributed-slice pattern, so the rule's surface seems to forbid
it. But it is permitted by the rule's spirit. The DESIGN.md.tmpl
needs to record the reasoning so a future agent or contributor
reading the rule alongside the code does not flag the section as a
violation.

This round adds a "Linker-section exemption rationale" subsection
to `mock/crates/hilavitkutin-str/DESIGN.md.tmpl`. No template
deletion, no source edits.

## Why hilavitkutin-str's linker section is exempt

The no-linker-magic rule targets a specific failure mode: runtime
discovery of dispatch targets. The engine's correctness contract
depends on the WorkUnit graph being known at compile time. Linker-
magic patterns let consumer code register additional dispatch
targets at link time, defeating the static composition guarantee.
Concretely:

- `inventory::submit!` registers values that a runtime walker reads.
  When the engine reads such a registry, the WU graph is a function
  of which crates linked, not which crates the consumer explicitly
  added to the SchedulerBuilder.
- `#[ctor]` runs code at startup. If it modifies global runtime
  state the engine reads, the engine's behaviour depends on link
  order.
- `#[distributed_slice]` is the same shape as `inventory` with
  different ergonomics.
- `.init_array` is the cross-platform mechanism each of the above
  ultimately uses.

The shared anti-pattern: **a linker-time artefact that mutates the
engine's runtime dispatch behaviour**.

The hilavitkutin-str linker section does not match this pattern.
Specifically:

1. **The data is strings, not dispatch targets.** Each
   `StaticStrEntry` is a `(hash, content_bytes)` pair. The
   interner reads the section once at startup, populates a lookup
   table, and the rest of the program never touches the section.
   The string interner's behaviour is a pure function of the
   strings the consumer invoked `str_const!` on; no consumer can
   smuggle in an extra dispatch target by linking a crate that
   declares strings.

2. **The trust boundary is different.** Linker-magic registration
   patterns are dangerous because the registered values participate
   in load-bearing dispatch. The engine's WU graph drives plan
   compilation; getting an extra WU into it would change the plan.
   String interning does not drive plan compilation. The interner
   is a content-addressed data structure; an extra string in the
   table costs O(1) lookup and no behaviour difference for code
   that does not reference that string.

3. **The macro emits no code that runs at consumer dispatch time.**
   `str_const!` produces a static `StaticStrEntry` placed in the
   linker section plus a compile-time `Str` handle. The handle is
   the dispatch identity; the linker section is a side-table
   consulted only by the interner during a one-shot startup walk.

4. **There is no consumer-side instability risk.** The string
   interner exposes `intern(&str) -> Str` and `resolve(Str) ->
   &str`. The contract is content-stable: same string in, same
   `Str` out. Adding more strings to the table does not change
   that contract for any pre-existing consumer.

5. **The technique replaces what would otherwise need
   `OnceLock<HashMap>` or similar.** Without the linker section,
   const string interning would require either runtime
   construction of a hash table (alloc-dependent, breaks the
   `no_alloc` rule) or a `phf`-style perfect-hash crate
   (substantial dependency, build-time cost). The linker section
   provides O(1) const lookup with zero runtime allocation. This
   matches the design intent of `hilavitkutin-str`: const strings
   are first-class, runtime strings are arena-allocated, both flow
   through the same `Str` handle.

The rule is preserved in spirit. The technique is exempt because
the data it registers is non-dispatch and read-only.

## What gets added to DESIGN.md.tmpl

existing `## StaticStrEntry, linker-section registration` section,

explicitly cross-references the workspace rule
(`hilavitkutin/.claude/CLAUDE.md` "linker-magic registration"
clause) and names the specific anti-pattern that the
`.hilavitkutin_strings` section does not match.
