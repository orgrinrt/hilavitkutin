## CHANGE: ADD `InternerKit<BYTES, ENTRIES>` struct + Kit impl to hilavitkutin-providers

File: `mock/crates/hilavitkutin-providers/src/interner.rs` (extends
the existing module).

Adds at the bottom of the existing module:

```rust
use hilavitkutin_api::BuilderResource;
use hilavitkutin_kit::Kit;

/// Kit preset that registers a default-configured
/// `Resource<StringInterner<MemoryArena<BYTES, ENTRIES>>>`.
///
/// Composes via the api-level `BuilderResource<T>` sealed trait so
/// the impl never names the engine's `SchedulerBuilder` directly.
/// Use as `Scheduler::builder().add_kit(InternerKit::<BYTES, ENTRIES>)`.
pub struct InternerKit<const BYTES: usize, const ENTRIES: usize>;

impl<B, const BYTES: usize, const ENTRIES: usize> Kit<B>
    for InternerKit<BYTES, ENTRIES>
where
    B: BuilderResource<StringInterner<MemoryArena<BYTES, ENTRIES>>>,
{
    type Output = B::WithResource;

    fn install(self, builder: B) -> Self::Output {
        builder.with_resource(default_interner::<BYTES, ENTRIES>())
    }
}
```

The two const-generic parameters use bare `usize` (the lint:allow
pattern matches the existing `default_interner` and `MemoryArena`
const-generic shape, tracked: #121).

## CHANGE: ADD smoke test for InternerKit Kit installation against a stub builder

File: `mock/crates/hilavitkutin-providers/tests/smoke.rs`
(extends the existing test file).

Adds a new `#[test] fn internerkit_installs_via_stub_builder()` that:

1. Defines a private `StubBuilder<T>` type implementing
   `BuilderResource<T>` with `WithResource = StubBuilder<T>` (a
   self-loop, sufficient to type-check the trait bound).
2. Implements the sealing supertrait
   `hilavitkutin_api::builder::builder_resource_sealed::Sealed<T>`
   for the stub. The sealing module is `pub use`-d via
   `#[doc(hidden)]` re-export so test crates can name it (added in
   the CHANGE block to api/src/lib.rs above).

   Note: if the sealing module's path is not reachable from a tests
   crate (it is `#[doc(hidden)]` but visible by full path), the
   smoke test falls back to constructing the Kit and calling
   `install` against a real engine `SchedulerBuilder` instead. This
   pulls in the engine as a dev-dep on providers, matching the
   pattern used by hilavitkutin-kit's own `tests/kit.rs`.

3. Calls `InternerKit::<128, 8>.install(stub)` and asserts the
   returned type compiles (no runtime assertion needed; type-check
   is the test).

The sealing-module visibility is the load-bearing decision in this
CL. The simpler path (engine as dev-dep on providers tests) avoids
that entirely; if the sealing module needs `pub` visibility for
test-side stubs, a separate round handles it. SRC application picks
the simpler path: dev-dep on the engine for the test only.

Cargo.toml edit: add `[dev-dependencies] hilavitkutin.workspace =
true` to the providers Cargo.toml. Dev-deps are not gated by the
runtime forbidden-imports rule (they only affect the test binary).
