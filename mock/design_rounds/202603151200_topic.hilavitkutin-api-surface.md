## Q1. Store types and AccessSet

From core design Q1: three store types, one scheduling mechanism.

**The Store trait and three types:**

```rust
/// Marker trait for all store types. Each concrete store maps to a
/// StoreId used for DAG edge construction.
pub trait Store: 'static {
    type Item: Send + Sync + 'static;
}

/// Singleton store. One value of type T.
pub struct Resource<T: Send + Sync + 'static>(PhantomData<T>);
impl<T: Send + Sync + 'static> Store for Resource<T> {
    type Item = T;
}

/// Collection store. N values of type T, morsel-chunked.
pub struct Column<T: Send + Sync + 'static>(PhantomData<T>);
impl<T: Send + Sync + 'static> Store for Column<T> {
    type Item = T;
}

/// Zero-data store. Creates DAG edges with no storage.
pub struct Virtual<T: 'static>(PhantomData<T>);
impl<T: 'static> Store for Virtual<T> {
    type Item = (); // no data
}
```

**AccessSet as tuples:**

```rust
/// Trait for declaring what stores a work unit accesses.
/// Implemented for single stores and tuples up to arity 12.
pub trait AccessSet {
    fn store_ids() -> &'static [StoreId];
}

// Single store
impl<S: Store> AccessSet for S {
    fn store_ids() -> &'static [StoreId] { /* ... */ }
}

// Tuples
impl<A: Store, B: Store> AccessSet for (A, B) {
    fn store_ids() -> &'static [StoreId] { /* ... */ }
}

impl<A: Store, B: Store, C: Store> AccessSet for (A, B, C) {
    fn store_ids() -> &'static [StoreId] { /* ... */ }
}

// ... up to arity 12 via macro
```

**StoreId and bitmask representation:**

```rust
/// Dense index assigned at plan time. Maps to a bit position in
/// the access bitmask.
#[derive(Clone, Copy, PartialEq, Eq, Hash)]
pub struct StoreId(u16);

/// Fixed-width bitmask for access set operations.
/// u64 for ≤64 stores, [u64; 4] for ≤256 stores.
#[derive(Clone, Copy)]
pub struct AccessMask(u64);

impl AccessMask {
    pub const fn empty() -> Self { Self(0) }
    pub const fn bit(id: StoreId) -> Self { Self(1 << id.0) }
    pub const fn union(self, other: Self) -> Self { Self(self.0 | other.0) }
    pub const fn intersects(self, other: Self) -> bool { (self.0 & other.0) != 0 }
    pub const fn is_empty(self) -> bool { self.0 == 0 }
}
```

**Decision: hybrid const/runtime bitmask.**

AccessSet provides compile-time store descriptors. The plan phase
assigns bit positions and converts to runtime bitmasks via monomorphized
generic functions (no TypeId, no const hash — the monomorphized function
IS the type identity). All dispatch-time operations use pre-computed
masks (single AND/OR per operation).

```rust
pub trait AccessSet: 'static {
    /// Compile-time: store descriptors known from associated types.
    const STORES: &'static [StoreDescriptor];
}

pub struct StoreDescriptor {
    pub kind: StoreKind,
    // type identity via monomorphization, not TypeId
}

pub enum StoreKind { Resource, Column, Virtual }
```

Plan phase converts per-impl const STORES into runtime AccessMask:
```rust
// called once per impl at plan time (monomorphized per AccessSet)
fn to_mask<A: AccessSet>(assignment: &BitAssignment) -> AccessMask {
    let mut mask = AccessMask::empty();
    for desc in A::STORES {
        mask = mask.union(AccessMask::bit(assignment.position_for::<A>(desc)));
    }
    mask
}
```

Bitmask width: u64 for ≤64 stores (typical). For >64 stores, use
`[u64; 4]` (256 stores). The width is a const generic on the scheduler,
matching MAX_STORES.

## Q2. WorkUnit trait

From core design Q2: generic-conditioned trait with scheduling hints.

```rust
/// The core work unit trait. The generic parameter controls when
/// this impl is dispatched.
pub trait WorkUnit<Schedule = Always>: Send + Sync + 'static {
    /// Human-readable name for diagnostics and metrics.
    const NAME: &'static str;

    /// What this impl reads. Write implies Read.
    type Read: AccessSet;

    /// What this impl writes.
    type Write: AccessSet;

    /// Scheduling hints as a tuple of ZST axis values.
    type Hint: SchedulingHint = (Immediate, Atomic, Normal);

    /// Convenience alias for the typed context.
    type Ctx = Context<Self::Read, Self::Write>;

    /// Execute the work. Called once per scalar dispatch or once
    /// per morsel for Column-containing access sets.
    fn execute(&self, ctx: Self::Ctx);
}
```

**Schedule condition types:**

```rust
/// Unconditional. Dispatched every pass.
pub struct Always;

/// Dispatched when virtual V fires.
pub struct On<V: 'static>(PhantomData<V>);
```

**Decision: After<U> dropped.** Every real sequencing need is covered by:
- Data dependencies (Write → Read overlap creates DAG edges automatically)
- Event dependencies (fire a virtual, consumer gates with On<V>)

`After<U>` was unnecessary sugar that introduced ambiguity (which impl
of a multi-impl type does it reference?). With After gone, the only
schedule conditions are Always and On<V>. Simpler, unambiguous.

If a work unit truly needs to run "after U but doesn't read anything U
writes," U should fire a completion virtual and the consumer gates on it.
This makes the ordering explicit in the API rather than implicit in
a type reference.

**Decision: `Self::Ctx` via HasCtx<S> blanket + supertrait.**

The `Self::Ctx` signature works via a blanket-implemented supertrait:

```rust
trait HasCtx<S> {
    type Ctx;
}

trait WorkUnit<S = Always>: HasCtx<S> + Sized {
    type Read: AccessSet;
    type Write: AccessSet;
    fn execute(&self, ctx: Self::Ctx);
}

// blanket: every WorkUnit<S> automatically gets HasCtx<S>
impl<T: WorkUnit<S>, S> HasCtx<S> for T {
    type Ctx = Context<T::Read, T::Write>;
}
```

`Self::Ctx` resolves correctly per-impl because HasCtx<S> is a
supertrait parameterized by S, and S is fixed per impl block. The
blanket is NOT circular despite HasCtx being a supertrait — Rust
resolves the blanket from the WorkUnit impl's Read/Write, which are
concrete per impl.

Multi-impl types work: `Self::Ctx` in `impl WorkUnit<On<ResponseReady>>`
resolves to `<Self as HasCtx<On<ResponseReady>>>::Ctx`, which the blanket
provides as `Context<this_impl::Read, this_impl::Write>`.

Compile-time access enforcement: Context methods use Contains<S, Idx>
bounds on the Read/Write tuples. Accessing a store not declared in this
impl's Read or Write is a compile error.

No unstable features. No macros. No type aliases. No unsafe.

Verified by compilation: the pattern compiles on stable Rust 2021
edition with multi-impl types, per-impl enforcement, and shared
struct data across impls.

From core design Q1 + Q2: compile-time safe access via Contains bounds.

**Questions for this section:**
- The Contains trait with tuple impls causes overlapping impl issues
  when the same store appears in both Read and Write. How to resolve?
  Type-level index (Contains<S, Idx>) avoids overlap but adds complexity.
- Should column() return a slice or a custom iterator type that enables
  zip patterns? `ctx.zip::<(A, B, C)>()` appeared in examples but isn't
  shown in the trait.
- How does the morsel scope work? The Context holds a range (start, end)
  into the full column, and column() returns only that slice?

## Q5. Virtual flag system

From core design Q1 + Q3: per-(virtual, consumer) bits, clear-on-dispatch,
epoch-based reset.

```rust
/// The virtual flag array. Per-(virtual, consumer) bytes storing
/// the epoch when last fired.
pub struct VirtualFlags {
    /// One byte per (virtual, consumer) pair. Value = epoch when fired.
    flags: [u8; MAX_VIRTUAL_CONSUMER_PAIRS],
    /// Current epoch. Incremented each pass. "Fired" = flag == epoch.
    epoch: u8,
    /// Mapping: virtual index → (base_offset, consumer_count)
    layout: [VirtualLayout; MAX_VIRTUALS],
}

struct VirtualLayout {
    base: u16,   // offset into flags array
    count: u8,   // number of consumers
}

impl VirtualFlags {
    /// Fire a virtual for all its consumers.
    pub fn fire(&mut self, virtual_idx: u16) {
        let layout = &self.layout[virtual_idx as usize];
        let base = layout.base as usize;
        let count = layout.count as usize;
        // Write current epoch to all consumer slots
        for i in base..base + count {
            self.flags[i] = self.epoch;
        }
    }

    /// Atomic fire from external thread.
    pub fn fire_external(&self, virtual_idx: u16) {
        let layout = &self.layout[virtual_idx as usize];
        let base = layout.base as usize;
        let count = layout.count as usize;
        let epoch = self.epoch; // read is atomic for u8
        for i in base..base + count {
            // atomic store
            unsafe {
                let ptr = &self.flags[i] as *const u8 as *mut u8;
                core::intrinsics::atomic_store_relaxed(ptr, epoch);
            }
        }
    }

    /// Check if a specific consumer's virtual has fired.
    pub fn is_fired(&self, consumer_slot: u16) -> bool {
        self.flags[consumer_slot as usize] == self.epoch
    }

    /// Clear a consumer's flag (on first dispatch).
    pub fn clear_on_dispatch(&mut self, consumer_slot: u16) {
        // Write a non-epoch value
        self.flags[consumer_slot as usize] = self.epoch.wrapping_sub(1);
    }

    /// Advance epoch (pass boundary). No memset needed.
    pub fn advance_epoch(&mut self) {
        self.epoch = self.epoch.wrapping_add(1);
        // Every 256 passes: full clear to avoid false positives
        // from epoch wraparound
        if self.epoch == 0 {
            for f in self.flags.iter_mut() {
                *f = 255; // anything != 0 (the new epoch)
            }
        }
    }
}
```

**Questions:**
- The epoch wraparound every 256 passes requires a full clear. Is this
  acceptable? For 60fps: wraparound every ~4 seconds. For single-shot
  pipelines: never wraps.
- Should the fire_external use Relaxed ordering or Release? Relaxed is
  cheaper but the scheduler scan might not see the write immediately.
  Release + Acquire on the scan side guarantees visibility.

## Q9. SlabColumn

From core design Q7 + Q10: the universal storage atom.

```rust
/// Contiguous, fixed-stride, zero-copy-growth column.
/// Backed by mmap (os tier) or consumer-provided memory (no_os).
pub struct SlabColumn<T: Send + Sync + 'static> {
    base: *mut T,
    len: usize,
    capacity: usize,  // in elements, not bytes
    _marker: PhantomData<T>,
}

impl<T: Send + Sync + 'static> SlabColumn<T> {
    /// Create with initial capacity. Allocates via MemoryProvider.
    pub fn with_capacity(cap: usize, mem: &dyn MemoryProvider) -> Self {
        let size = cap * core::mem::size_of::<T>();
        let align = core::mem::align_of::<T>();
        let base = unsafe { mem.allocate(size, align) as *mut T };
        Self { base, len: 0, capacity: cap, _marker: PhantomData }
    }

    pub fn len(&self) -> usize { self.len }
    pub fn is_empty(&self) -> bool { self.len == 0 }

    pub fn push(&mut self, value: T) {
        debug_assert!(self.len < self.capacity);
        unsafe {
            self.base.add(self.len).write(value);
        }
        self.len += 1;
    }

    pub fn as_slice(&self) -> &[T] {
        unsafe { core::slice::from_raw_parts(self.base, self.len) }
    }

    pub fn as_mut_slice(&mut self) -> &mut [T] {
        unsafe { core::slice::from_raw_parts_mut(self.base, self.len) }
    }

    /// Morsel-scoped view.
    pub fn morsel(&self, start: usize, end: usize) -> &[T] {
        &self.as_slice()[start..end]
    }

    pub fn morsel_mut(&mut self, start: usize, end: usize) -> &mut [T] {
        &mut self.as_mut_slice()[start..end]
    }
}
```

## Q10. Version stamps

From core design Q2 + Q10 + Q12: parallel arrays, SoA layout.

```rust
/// Version stamp manager. Parallel arrays for before/current comparison.
pub struct VersionStamps<const MAX_STORES: usize = 64> {
    current: [u64; MAX_STORES],
    before: [u64; MAX_STORES],
}

impl<const N: usize> VersionStamps<N> {
    /// Capture before-versions for a work unit's write set.
    pub fn capture_before(&mut self, write_mask: AccessMask) {
        let mut bit = write_mask.0;
        while bit != 0 {
            let idx = bit.trailing_zeros() as usize;
            self.before[idx] = self.current[idx];
            bit &= bit - 1; // clear lowest set bit
        }
    }

    /// Check if any stamp changed. Returns bitmask of changed stores.
    pub fn changed(&self, write_mask: AccessMask) -> AccessMask {
        let mut changed = 0u64;
        let mut bit = write_mask.0;
        while bit != 0 {
            let idx = bit.trailing_zeros() as usize;
            if self.before[idx] != self.current[idx] {
                changed |= 1 << idx;
            }
            bit &= bit - 1;
        }
        AccessMask(changed)
    }

    /// Increment a store's version.
    pub fn increment(&mut self, store_id: StoreId) {
        self.current[store_id.0 as usize] += 1;
    }
}
```

## Q11. Fusion execution model

From core design Q12 (loimu fusion research): work units push per-element
ops via `ctx.each()`. The storage engine collects all ops from all work
units in the same step, groups them by shared columns, and iterates the
data ONCE per group, calling all fused ops per element.

**The consumer API:**

```rust
impl WorkUnit for TransformAndScale {
    type Read = (Column<RawValue>, Resource<ScaleFactor>);
    type Write = (Column<Scaled>,);

    fn execute(&self, ctx: Self::Ctx) {
        let scale = ctx.resource::<ScaleFactor>();

        ctx.each(|raw: &RawValue, out: &mut Scaled| {
            *out = raw.apply(*scale);
        });
    }
}
```

`execute()` runs once at pass start. It does NOT run per element.
Each `ctx.each()` call registers an op descriptor. The closures run
later inside the fused iteration loop.

Resources are loaded before the loop via `ctx.resource()`. The `each`
closure captures them. Only column elements appear as closure params.

Three op kinds:
- `ctx.each(|...| { ... })` — per-element, fusible
- `ctx.batch(|...| { ... })` — full morsel slice, non-fusible
- `ctx.reduce(init, |acc, ...| acc + ...)` — accumulation, partially fusible

**Fusion mechanics:**

Within a step, multiple work units' `each` ops that share columns are
grouped by the fusion engine. Multiple writes to the same column from
different work units CAN fuse — they run sequentially per element in
DAG order. The fused loop iterates once, calling all ops per element:

```
for i in morsel_start..morsel_end {
    // work_unit_1's each (DAG order)
    scaled[i] = raw[i].apply(scale);
    // work_unit_2's each (reads what 1 just wrote)
    normalized[i] = scaled[i].normalize();
}
```

Data stays in L1 across all fused ops.

**Critical finding: LLVM devirtualizes and inlines function pointer
arrays with fat LTO.**

Tested: an array of `fn(&mut [u64], usize)` function pointers called
in a tight loop, compiled with `-C opt-level=3 -C lto=fat -C
codegen-units=1`. LLVM:

1. Devirtualized the function pointers (proved their concrete targets)
2. Inlined both ops into the loop body
3. Fused them into one loop iteration
4. Unrolled 4x with paired loads/stores
5. Replaced `mul 3` with `add x, x, lsl #1`

Zero indirect calls. The generated assembly is identical to
hand-written fused code.

This means: **runtime-determined fusion groups produce compile-time-
quality codegen.** No compile-time fusion grouping needed. No type-level
tuple composition. No explicit fusion groups in the builder API.

The path:
1. Consumer writes `ctx.each()` closures with `#[inline(always)]`
2. execute() runs at pass start, pushes op function pointers into arrays
3. The fusion engine groups ops by shared columns (runtime)
4. The fused loop iterates with the function pointer array
5. Fat LTO devirtualizes, inlines, and fuses everything
6. LLVM sees one loop body with all ops, vectorizes and optimizes

Requirements for this to work:
- `#[inline(always)]` on the each closures (or small enough for LLVM
  to inline at its discretion)
- Fat LTO enabled (hilavitkutin-build's ExpandedLto pragma)
- `codegen-units = 1`
- The function pointer array is constructed once and not mutated
  (LLVM must prove the contents are constant)

Fallback without LTO: function pointer dispatch per op per element.
Still benefits from fusion's cache locality. Loses the inlining.
Estimated overhead: ~1-2ns per indirect call per element, amortized
by the per-element work being heavier.

**Critical constraint: local/static slices, NOT heap Vec.**

Tested: LLVM devirtualizes `&[fn_ptr_a, fn_ptr_b, fn_ptr_c]` (local
slice with known values) into fully inlined fused loop. Tested: LLVM
does NOT devirtualize `Vec<EachOp>` (heap-allocated, contents not
provable).

This is not a problem for hilavitkutin: we have NO alloc, NO heap.
All storage is SlabColumn or stack arrays. The function pointer array
for a fused chain is always a local slice or a SlabColumn-backed
fixed array — both of which LLVM can devirtualize. The no-alloc
constraint is load-bearing for the fusion optimization.

Verified assembly (aarch64, fat LTO, codegen-units=1): three ops
(scale, add, normalize) fused into one loop body. Zero indirect
calls. Register reuse across ops (output of op N becomes input of
op N+1 without reload). LLVM even vectorized the non-division ops.

**Validation needed:** unit tests covering different function array
payloads, data types, and op combinations. Verify LLVM devirtualizes
in each case. Tests are isolated (don't need hilavitkutin) — just
function pointer arrays + tight loops + fat LTO. Later expanded to
test with hilavitkutin's actual storage layer.

## Q12. Internal execution pipeline trace

From core design Q7 (partitions, chains, chain groups, morsels) and
Q11 (fusion): tracing through what the scheduler writes internally to
verify the full pipeline works under our constraints (no alloc, no heap,
const-generic sizing, LLVM devirtualization requirement).

### Data structures (all const-generic, all stack/SlabColumn)

**Plan state (computed once at build or on DAG change):**

```rust
struct PlanState<const MU: usize, const MS: usize> {
    // per-unit bitmasks, indexed by dense u16 unit ID
    read_masks:  [AccessMask; MU],
    write_masks: [AccessMask; MU],

    // access matrix: unit × store adjacency (read | write per row)
    access_matrix: [AccessMask; MU],

    // DAG adjacency: bit-matrix
    // row i has bit j set if unit i must run before unit j
    dag: [u64; MU],  // MU ≤ 64 for u64, else [u64; K]

    unit_count: u16,
    store_count: u16,
}
```

DAG edges computed from access matrix: for every pair (i, j) where
`(write_masks[i] & read_masks[j]) != 0`, set `dag[i] |= 1 << j`.
O(N²) ANDs on u64 values. For 50 units: 2500 AND operations, ~1µs.

**Partition map (connected components of column-sharing graph):**

```rust
struct PartitionMap<const MU: usize> {
    partition_id: [u8; MU],  // unit → partition
    count: u8,
}
```

Found via stack-allocated DFS: `visited` bitmask (u64 for ≤64 units),
fixed `[u16; MU]` stack. Two units share a partition if their access
masks intersect (directly or transitively). Zero heap.

**Chain plan (contiguous ranges of topo-ordered units):**

```rust
struct ChainPlan<const MU: usize> {
    chain_start: [u16; MU],   // chain i starts at unit chain_start[i]
    chain_len:   [u16; MU],   // chain i has chain_len[i] units
    chain_count: u16,

    // fused op function pointers, packed contiguously
    ops: [EachFn; MU],        // all ops across all chains
    op_offsets: [u16; MU],    // chain i's ops start at op_offsets[i]
    op_counts: [u8; MU],      // chain i has op_counts[i] ops
}
```

Chain formation: Kahn's topo sort (fixed `[u16; MU]` queue + `[u16; MU]`
in-degree array), then greedy fusion walk. Break when write set exceeds
L1 budget, or fan-in, or pipeline breaker.

**Chain group plan (spectral partitioning for L2 co-location):**

```rust
struct ChainGroupPlan<const MU: usize> {
    group_id: [u8; MU],           // chain → group
    group_count: u8,
    cluster_assignment: [u8; 8],  // group → L2 cluster
}
```

For ≤5 chains: single group. For >5: Fiedler vector via power iteration
on a `[[u8; 64]; 64]` weight matrix (4 KB stack) with `[f32; 64]`
eigenvector (256 bytes stack). 30 iterations on a 20×20 matrix: ~12,000
multiply-adds. Microseconds. No heap.

**Morsel plan (pre-partitioned ranges per worker):**

```rust
struct MorselPlan<const MU: usize, const ML: usize> {
    morsel_size: [u32; MU],                   // elements per morsel
    worker_ranges: [[(u32, u32); ML]; MU],    // per-chain, per-worker
}
```

Morsel size: `L1_CACHE_SIZE / (N_columns * size_of::<T>())`. Worker
ranges: divide total morsels evenly, remainder distributed round-robin.

### Dispatch loop (per worker, per pass)

```rust
fn worker_dispatch(
    lane: u8,
    scheduler: &Scheduler<MU, MS, ML>,
) {
    let chain_plan = &scheduler.chain_plan;
    let morsel_plan = &scheduler.morsel_plan;
    let stores = &scheduler.stores;

    for g in scheduler.groups_for_lane(lane) {
        for c in scheduler.chains_in_group(g) {
            let (m_start, m_end) = morsel_plan.worker_ranges[c][lane as usize];
            let m_size = morsel_plan.morsel_size[c] as usize;

            // slice into the ops array — local &[EachFn]
            let ops = &chain_plan.ops[
                chain_plan.op_offsets[c] as usize..
                chain_plan.op_offsets[c] as usize + chain_plan.op_counts[c] as usize
            ];

            for m in m_start..m_end {
                let row_start = m as usize * m_size;
                let row_end = core::cmp::min(row_start + m_size, stores.total_rows());

                // THE FUSED LOOP
                for i in row_start..row_end {
                    for op in ops {
                        op(stores, i);
                    }
                }
            }
        }
    }
}
```

### Size budget (typical: 30 units, 40 stores, 4 workers)

| Structure | Size |
|-----------|------|
| `read_masks: [AccessMask; 256]` | 2 KB |
| `write_masks: [AccessMask; 256]` | 2 KB |
| `dag: [u64; 256]` | 2 KB |
| `chain_plan.ops: [EachFn; 256]` | 2 KB |
| `morsel_plan.worker_ranges: [[(u32,u32); 8]; 256]` | 16 KB |
| `fiedler weights (temporary, plan only)` | 4 KB |
| Hot metadata: `[HotMeta; 256]` | 2.8 KB |
| **Total scheduler state** | **~30 KB** |

Fits in L1. All const-generic arrays, zero heap.

### TESTED: LLVM does NOT devirtualize struct-field fn pointer arrays

**Test file:** `mock/research/struct_field_devirt_test.rs`

The previous devirtualization test used a local `&[fn(...)]` slice with
known values. The actual scheduler would store function pointers in
`ChainPlan.ops: [EachFn; MU]` — a field of a struct. The dispatch loop
takes a slice of this field: `&chain_plan.ops[offset..offset+count]`.

**Result: LLVM does NOT devirtualize this pattern.**

Tested six dispatch patterns, compiled with `-C opt-level=3 -C lto=fat
-C codegen-units=1`, benchmarked on aarch64 (Apple Silicon M1):

| Pattern | ns/elem | vs hand-fused | Devirt? | Assembly |
|---------|---------|---------------|---------|----------|
| hand-fused (baseline) | 0.41 | 1.0x | n/a | NEON vectorized, `add.2d`/`ushr.2d` |
| local `&[fn]` slice | 0.39 | 0.9x | YES | identical to hand-fused |
| monomorphized trait | 0.41 | 1.0x | YES | inlined into caller |
| unrolled (3 fn params) | 0.39 | 0.9x | YES | identical to hand-fused |
| const-generic `&[fn; N]` param | 2.40 | 5.8x | NO | 3x `blr` (indirect) |
| struct-field slice | 5.19 | 12.6x | NO | `blr x8` in loop |

**Root cause:** LLVM devirtualizes function pointers only when their
values are visible in the function that uses them. When function
pointers cross a function boundary via `&struct` or `&[fn; N]`
parameter, LLVM cannot prove their contents. The struct was populated
in `main()`, but the dispatch function only sees an opaque reference.
Even fat LTO does not save this — the indirection through a struct
reference is opaque to LLVM's value tracking.

**What works:**

1. **Local slice** — `let ops: &[EachFn] = &[op_a, op_b, op_c]`
   constructed inside the dispatch function. LLVM sees the values
   directly. Full devirtualization, inlining, NEON vectorization.

2. **Monomorphized trait** — `trait ChainDispatch { fn dispatch(...) }`
   with a concrete impl that constructs the local slice internally.
   The trait method is monomorphized per concrete type. LLVM inlines
   the concrete dispatch into the caller. 1.0x hand-fused.

3. **Unrolled parameters** — pass each `fn` as a separate parameter:
   `dispatch(stores, op_a, op_b, op_c, start, end)`. LLVM proves
   each parameter's value at the call site. Full devirtualization.

**What does NOT work:**

1. **Struct field** — function pointers stored in `plan.ops[..]` and
   sliced. 12.6x slower. `blr x8` (indirect branch) in hot loop.

2. **Const-generic `&[fn; N]` parameter** — function pointers in a
   fixed array passed by reference. 5.8x slower. Three separate
   `blr` instructions (LLVM unrolled the inner loop but didn't
   devirtualize).

**Design implication:** the dispatch loop CANNOT iterate a pre-built
function pointer array stored in a struct. The function pointer values
must be visible at the point of the loop.

### Resolution: monomorphized chain dispatch

Each chain gets a monomorphized dispatch function generated at compile
time. The dispatch function constructs the local `&[EachFn]` slice
internally, where LLVM can see the values.

The chain formation still happens at plan time (runtime). But the set
of possible chains is bounded: each chain is a subsequence of the
registered work units. At registration time (compile time via generics),
we know every work unit's `each` closure. The plan phase selects which
work units group into which chain, then dispatches to a monomorphized
function for that combination.

Two viable approaches:

**Approach A: per-chain-type trait (monomorphized).**

```rust
trait ChainDispatch: Send + Sync {
    fn dispatch(&self, stores: &mut StoreMap, start: usize, end: usize);
}

// generated per concrete chain combination
impl ChainDispatch for (ScaleUnit, OffsetUnit, NormUnit) {
    fn dispatch(&self, stores: &mut StoreMap, start: usize, end: usize) {
        let ops: &[EachFn] = &[
            ScaleUnit::each_fn(),
            OffsetUnit::each_fn(),
            NormUnit::each_fn(),
        ];
        for i in start..end {
            for op in ops { op(stores, i); }
        }
    }
}
```

The tuple type encodes the chain composition. Each combination
generates a different monomorphized `dispatch`. LLVM sees the local
slice with known function pointer values. Full devirtualization.

**Approach B: unrolled dispatch with fn parameters.**

```rust
fn dispatch_chain(
    stores: &mut StoreMap,
    ops: (EachFn, EachFn, EachFn),  // passed as values, not slice
    start: usize,
    end: usize,
) {
    for i in start..end {
        (ops.0)(stores, i);
        (ops.1)(stores, i);
        (ops.2)(stores, i);
    }
}
```

Function pointers as tuple values (in registers) rather than array
elements (behind a pointer). LLVM sees each value. Works for chains
up to ~8 ops (register pressure is the limit on aarch64 with 30
GPRs, but the fn ptrs are loaded before the loop).

**Approach C: hybrid — monomorphized inner loop, runtime chain selection.**

At plan time, the scheduler selects which chain to dispatch. The
dispatch table is an array of `fn(&mut StoreMap, usize, usize)` —
each entry is a monomorphized chain dispatch function. The outer
selection is one indirect call per chain per morsel (cheap). The
inner loop is fully devirtualized.

```rust
// plan time: build dispatch table
let chain_dispatch: [fn(&mut StoreMap, usize, usize); MAX_CHAINS] = [
    // each entry is a monomorphized function with local slice
    chain_0_dispatch,
    chain_1_dispatch,
    ...
];

// dispatch time: one indirect call per chain per morsel
chain_dispatch[chain_idx](stores, morsel_start, morsel_end);
```

The single indirect call per morsel costs ~1-2ns. The inner loop
(1000+ elements) runs fully devirtualized. Net overhead: <0.002 ns
per element. Negligible.

~~**Preferred: Approach C.**~~ Approach C was the initial candidate:
one indirect call per chain per morsel, inner loop devirtualized.
Measured at 1.17x hand-fused — measurable 17% overhead.

### TESTED: partition mega-dispatch (Approach D) — 1.02x hand-fused

**Test file:** `mock/research/dispatch_optimization_test.rs`

Approach D eliminates all indirect calls by composing `#[inline(always)]`
chain dispatch functions into a single per-partition function:

```rust
#[inline(always)]
fn chain0_dispatch(stores: &mut StoreMap, start: usize, end: usize) {
    let ops: &[EachFn] = &[op_scale, op_add_offset, op_normalize];
    for i in start..end {
        for op in ops { op(stores, i); }
    }
}

#[inline(always)]
fn chain1_dispatch(stores: &mut StoreMap, start: usize, end: usize) {
    let ops: &[EachFn] = &[op_combine, op_finalize];
    for i in start..end {
        for op in ops { op(stores, i); }
    }
}

/// ONE function for the entire partition. No indirect calls.
#[inline(never)]
fn partition_mega_dispatch(stores: &mut StoreMap, start: usize, end: usize) {
    chain0_dispatch(stores, start, end);
    chain1_dispatch(stores, start, end);
}
```

LTO inlines both chain dispatches. The partition function contains both
chains' inner loops with all ops devirtualized, inlined, and vectorized.

**Benchmark results (aarch64, Apple Silicon M1, fat LTO):**

| Pattern | ns/elem | vs hand-fused |
|---------|---------|---------------|
| hand-fused baseline | 0.98 | 1.0x |
| **partition mega-dispatch** | **1.00** | **1.02x** |
| fully inlined (no fn ptr) | 1.09 | 1.11x |
| indirect per-chain (Approach C) | 1.15 | 1.17x |

The mega-dispatch (1.02x) beats even hand-written inline code (1.11x).
This is because LTO has more optimization context when composing
`#[inline(always)]` functions — it can make better register allocation
and instruction scheduling decisions than hand-written sequential code.

**Assembly verification (chain 1 in mega-dispatch):**

```asm

ldp  q2, q3, [x13], #64
ldp  q4, q5, [x14, #-32]      ; paired 128-bit loads from col_d
ldp  q6, q7, [x14], #64
add.2d  v0, v4, v0             ; vector add (combine op)
add.2d  v1, v5, v1
add.2d  v2, v6, v2
add.2d  v3, v7, v3
```

### TESTED: explicit prefetch HURTS on Apple Silicon

| Pattern | ns/elem | vs hand-fused |
|---------|---------|---------------|
| hand-fused + prefetch | 1.91 | 1.96x SLOWER |
| mega-dispatch + prefetch | 1.87 | 1.92x SLOWER |
| mega + morsel pipelining | 1.44 | 1.47x SLOWER |

Explicit `core::intrinsics::prefetch_read_data` with locality hint 3
(L1) makes everything nearly 2x slower. Even morsel-level software
pipelining (prefetch next morsel into L2 while processing current)
is 1.47x slower.

**Root cause:** Apple Silicon M1's hardware prefetcher is excellent for
sequential access patterns. Column data is accessed sequentially per
morsel. The hardware prefetcher already has the next cache lines ready.
Explicit prefetch instructions pollute the pipeline, compete with the
hardware prefetcher, and waste decode bandwidth.

**Caveat: tested on Apple Silicon M1 only.** The target deployment
for polka-dots and saalis is likely older Linux x86 hardware, which
may have weaker prefetchers. The prefetch finding may not generalize
to all targets. Explicit prefetch should be profiling-gated and
`#[cfg(target_arch)]`-conditional, not blanket-applied or blanket-banned.

**Design implication:** do NOT add explicit prefetch by default for
sequential column traversal. Reserve for:
- Random/pointer-chasing access patterns (e.g., hash table lookups)
- Architectures where profiling shows the hardware prefetcher is
  insufficient (some older x86, embedded cores)
- Cross-morsel prefetch on non-Apple architectures (needs testing)

Explicit prefetch is `#[cfg(target_arch)]` + profiling-gated, not
day-one default. The microkernel table entry for `prfm`/`prefetcht0`
should specify this gating.

### Preferred: Approach D (partition mega-dispatch)

Each partition gets a single monomorphized dispatch function containing
all its chains, fully inlined. The scheduler stores one fn ptr per
partition (not per chain). Dispatch:

```rust
fn worker_dispatch(lane: u8, scheduler: &Scheduler<MU, MS, ML>) {
    let morsel_plan = &scheduler.morsel_plan;
    let stores = &mut scheduler.stores;

    for p in 0..scheduler.partition_count {
        let dispatch = scheduler.partition_dispatch[p]; // fn ptr
        let (m_start, m_end) = morsel_plan.partition_ranges[p][lane as usize];
        let m_size = morsel_plan.morsel_size[p] as usize;

        for m in m_start..m_end {
            let row_start = m as usize * m_size;
            let row_end = core::cmp::min(row_start + m_size, stores.total_rows());

            // ONE indirect call per morsel — entire partition inlined
            dispatch(stores, row_start, row_end);
        }
    }
}
```

The `partition_dispatch` array is `[fn(&mut StoreMap, usize, usize);
MAX_PARTITIONS]`. Each entry is a monomorphized function containing
all chains for that partition. One indirect call per partition per
morsel (not per chain). The inner loops are fully devirtualized,
inlined, and NEON-vectorized.

For a typical pipeline with 2-3 partitions and 1024-element morsels:
1 indirect call per morsel, 1024+ fully-inlined iterations. Overhead:
~0.001 ns per element. Effectively zero.

**How mega-dispatch functions are generated:**

The partition's chain composition is known at plan time. But the
individual chain dispatch functions are monomorphized at compile time
(they use local `&[EachFn]` slices with known function pointers).
The mega-dispatch composes these via `#[inline(always)]` calls.

Generation options:
1. **Proc macro** — `#[register]` generates per-WorkUnit dispatch
   helpers. A plan-time macro composes them into partition dispatches.
2. **Const-generic recursion** — type-level chain list
   `Partition<(Chain<A, B, C>, Chain<D, E>)>` generates the dispatch
   via recursive trait impl. Each level is `#[inline(always)]`.
3. **Build-time code generation** — if the schedule is static (polka:
   known at build time from polka.toml), generate the mega-dispatch
   in build.rs.

All three produce the same assembly. Option 2 (const-generic recursion)
is the most general — works for both static and dynamic schedules,
requires no proc macros, and lets LLVM inline at its discretion.

**Open question:** for dynamic schedules (work units registered at
runtime, e.g., plugin loading), the mega-dispatch can't be generated
at compile time. Fallback: Approach C (indirect per-chain, 1.17x).
Or: generate a bounded set of partition dispatches for common
combinations, with runtime fallback for uncommon ones.

**Results (aarch64, Apple Silicon M1, fat LTO, 4M elements):**

| Pattern | ns/elem | vs hand-fused |
|---------|---------|---------------|
| hand-fused (baseline) | 1.71 | 1.0x |
| **schedule mega-dispatch** | **1.66** | **0.97x** |

The full-schedule mega-dispatch matches hand-fused code. Zero overhead

**Intrinsics findings:**

- **`likely` + `get_unchecked` combo: 1.40x SLOWER.** The `if
  likely(i < n)` guard prevents LLVM from vectorizing the loop body.
  The branch exists even though it's always true. Worse than bounds
  checks, which LLVM can prove are unnecessary and eliminate.

- **Pure `get_unchecked`: 1.02x (neutral).** LLVM already eliminates
  bounds checks for `start..end` iteration over known-length Vecs.
  Unchecked indexing adds no value here. In real hilavitkutin, the
  morsel range is guaranteed within bounds by the scheduler — so
  unchecked is safe and may help if LLVM can't prove the bounds
  (e.g., when iterating a SlabColumn where length is a runtime field).

- **Conclusion: for the inner dispatch loop, LLVM is already optimal.**
  `likely`/`unlikely` hurt vectorization. `get_unchecked` is neutral.
  The useful intrinsics are elsewhere: `ctlz`/`cttz` for bitmask
  scanning (scheduler ready-queue), `copy_nonoverlapping` for bulk
  column ops (SlabColumn growth), `write_bytes` for column init.
  Not in the dispatch loop.

### TESTED: multi-threaded morsel dispatch — memory bandwidth bound

**Test file:** `mock/research/full_schedule_dispatch_test.rs`

Pre-spawned thread pool with barrier synchronization (no per-iteration
thread creation overhead). 4M elements, 8 columns, tested at 1/2/4/8
threads on M1 Air (4 P-cores + 4 E-cores).

| Threads | mega-dispatch | unchecked | hand-fused | Best speedup |
|---------|---------------|-----------|------------|-------------|
| 1 | 1.62 ns | 1.68 ns | 1.71 ns | 1.0x |
| 2 | 1.44 ns | 1.49 ns | 1.59 ns | 1.1x |
| 4 | 1.43 ns | 1.33 ns | 1.34 ns | 1.3x |
| 8 | 1.74 ns | 1.53 ns | 1.57 ns | 1.1x |

**Peak: 1.3x speedup at 4 threads (4 P-cores). 8 threads is WORSE.**

The E-cores (~40% slower than P-cores) drag down performance when work
is evenly distributed. Even 4 P-cores only achieve 33% parallel
efficiency.

**Root cause: memory bandwidth saturation.** Working set: 8 columns ×
4M elements × 8 bytes = 256 MB per pass. The ops (multiply, add, shift,
xor) are so cheap that the bottleneck is loading data from DRAM, not
processing it. Adding cores doesn't help when all cores share the same
memory bus (~68 GB/s on M1).

**Design implications:**

1. **Morsel parallelism's value is NOT raw throughput on cheap ops.**
   For trivially cheap per-element work (arithmetic), memory bandwidth
   is the bottleneck. Parallelism helps when ops are COMPUTE-heavy
   (template expansion, string processing, complex math, serialization).

2. **Heterogeneous core awareness matters.** Assigning equal morsel
   ranges to E-cores wastes P-core time waiting for slow E-cores.
   The scheduler should assign proportionally smaller ranges to slower
   cores (the "morsel temperature → core assignment" mechanism from
   the dispatch topic).

3. **polka-dots / saalis workloads will scale better.** polka-dots
   has hundreds of entries with per-entry template expansion, file I/O,
   and DSL parsing. saalis has complex query operations per record.
   These are compute-heavy ops where parallelism helps. This micro-

4. **Tested on M1 Air only.** Older Linux x86 hardware (the likely
   deployment target) has different memory bandwidth characteristics,

   Multi-threaded scaling may be better or worse depending on the

### CRITICAL: slice aliasing UB — Context must use raw pointer access

**Test file:** `mock/research/unchecked_pattern_test.rs`

The `c()`/`cm()` pattern (returning `&[u64]`/`&mut [u64]` from
`from_raw_parts`) causes LLVM to produce **wrong results** when
work units are inlined into fused loops. The `&mut self` on `cm()`
gets `noalias` metadata from LLVM, causing it to reorder column
accesses across work units. Verified: hand-fused checked (c/cm)
and hand-fused unchecked (cu/cw) produce **different output** on
every row for a 3-op ECS movement chain.

**The fix: `cu()`/`cw()` raw pointer access methods on `&self`.**

```rust
impl StoreMap {
    // WRONG: returns slice with bounds check, &mut self gets noalias
    fn cm(&mut self, id: ColId) -> &mut [u64] {
        unsafe { slice::from_raw_parts_mut(self.col_ptrs[id.0 as usize], ...) }
    }

    // CORRECT: raw pointer write through &self, no noalias
    unsafe fn cw(&self, id: ColId, i: usize, val: u64) {
        *self.col_ptrs[id.0 as usize].add(i) = val;
    }
}
```

Key properties:
1. `cw` takes `&self`, not `&mut self` — no noalias metadata
2. Write goes through `*mut u64` stored in `col_ptrs` — raw pointer
   writes have no aliasing assumptions
3. `cu` reads through the same raw pointer — no bounds check
4. Verified: `cu`/`cw` output matches raw pointer dispatch exactly

**Design implication for Context<R, W>:**

The Context type from Q3 must NOT return slices. Instead:

```rust
impl<'a, R: AccessSet, W: AccessSet> Context<'a, R, W> {
    /// Read one element from a column. No bounds check.
    pub unsafe fn read<T>(&self, i: usize) -> T
    where R: Contains<Column<T>> { ... }

    /// Write one element to a column. No bounds check.
    /// Takes &self, not &mut self — writes through raw pointer.
    pub unsafe fn write<T>(&self, i: usize, val: T)
    where W: Contains<Column<T>> { ... }
}
```

The `&self` receiver is load-bearing. If Context methods take
`&mut self`, LLVM adds noalias and may reorder writes across
work units when they're fused. With `&self`, the raw pointer
writes have no aliasing assumptions and execute in program order.

This also means `ctx.each()` closures capture `&Context` (shared),
not `&mut Context`. The closure writes through raw pointers
internally. This is safe because the morsel dispatch ensures
non-overlapping access and the scheduler prevents concurrent
writes to the same column.

## UNDOCUMENTED: Morsel context memory model — what a WU actually sees

The docs (§8 memory-layout-and-platform) describe `ColumnStorage::slice()`
and `process_batch()` at the trait level, but never describe the concrete
machine-level memory layout a WU sees during morsel execution. This gap
caused the hilavitkutin benchmark (`mock/research/hilavitkutin_bench.rs`)
to model the wrong access pattern, producing results that may not transfer
to the real engine.

### What the WU context must contain

**Columns: pre-offset cursor + known stride.**

The chain grouper statically knows which columns a chain touches. At morsel
dispatch, the executor resolves each column to a cursor (pointer to the
first element in the morsel window): `slab_base + morsel_start * STRIDE`.
The WU walks forward by stride. Only the columns in this chain are present
in the context — no global column array lookup.

The stride is `size_of::<T>()` (type-native, per Q8 decision — supersedes
§1's universal 128-bit stride). The WU indexes from 0 to morsel_len,
advancing the cursor. This means:
- One pointer load per column per morsel (amortized over all rows)
- Sequential access pattern per column (prefetcher-optimal)
- No array-index-by-column-ID at the element level

**Resources: pointer to external singleton storage.**

Resource scalar fields are sharded by type into shared columns. A resource
with a u64 field lives in a `resource_field<u64>` column, at a row
determined by the resource ID. Collections inside resources use SlabColumn
handles.

During morsel execution, resources are accessed via pointer to their
canonical storage location. Read-only resources are loaded once into a
register. Accumulating resources read/write through the pointer per
element (or accumulate locally and write back once, when the WU owns
exclusive access to that resource for the morsel).

The resource data is NOT inline in the context struct. It is behind
pointer indirection to slab-backed storage. This is load-bearing:
writing inline data through `&self` (shared Context reference) is UB.
Pointer-to-external writes are sound because the pointed-to memory has
no aliasing relationship with the `&self` reference.

### What the benchmark got wrong

The hilavitkutin benchmark uses:

| Aspect | Benchmark model | Real model |
|--------|----------------|------------|
| Column access | `col_ptrs[col_id].add(global_i)` | pre-offset cursor + stride walk |
| Column ID lookup | array index per element | resolved at morsel dispatch, one ptr per column |
| Resource storage | `res: [u64; MAX_RES]` inline | pointer to external slab storage |
| Resource write | UB: `core::ptr::write` through `&self` to inline data | sound: raw ptr write to external storage |
| Stride | 8 bytes (u64) | size_of::<T>() (type-native, Q8) |
| Morsel indexing | global `st..en` | local `0..morsel_len` |

The dispatch strategy insights (function pointer overhead, inlining,
devirtualization, mega-dispatch) are about call overhead and likely still
hold. But absolute throughput numbers, cache behavior observations, and
the resource UB bug all stem from the wrong memory model.

### TESTED: chain-level resource caching eliminates aliasing overhead

**Test file:** `mock/research/hilavitkutin_bench.rs` (Stats-Pipeline scenario)

When WU functions access resources via `ru()`/`rw()` through `res_ptr`
(a raw pointer in the Store struct), LLVM cannot prove that resource
writes don't alias with column writes (both are raw pointers from the
same `&self`). Result: LLVM reloads resources from memory EVERY
iteration instead of keeping them in registers. Measured overhead:
1.28-1.40x vs hand-written code that uses local variables.

**Assembly evidence (aarch64):**

Without caching: 21 memory ops per iteration, including 6 resource
accesses through `[x0, #N]` (res_ptr). Read-only resources reloaded
every iteration.

With caching: the chain dispatch copies resources to a stack-local
array before the loop, points `res_ptr` at it. LLVM proves stack
memory doesn't alias heap column data. All 4 resources promoted to
registers (x3, x9, x10, x13). ZERO resource memory ops in the loop.
Written back once after the loop via `stp`.

**The fix:** before the morsel loop, the chain dispatcher snapshots
the chain's resources into a stack-local array and redirects `res_ptr`
to it. WU functions use `ru()`/`rw()` unchanged — they go through
`res_ptr` which now points to the stack. After the loop, the dispatcher
writes back mutable resources to canonical storage.

**Design implication:** the chain context must separate resource
storage from column storage at the pointer level. Resources should
live in a stack-local or `restrict`-qualified region that LLVM can
prove non-aliasing with column pointers. The chain dispatcher handles
this automatically — it knows the resource set from the WorkUnit
declarations' `type Read`/`type Write`.

This is NOT a `&self` vs `&mut self` issue (the cu/cw noalias finding).
This is a pointer provenance issue: two raw pointers from the same
struct may alias. The fix is pointer separation, not reference type.

### TESTED: inter-WU store-reload is the remaining overhead — flattened code eliminates it

**Test file:** `mock/research/hilavitkutin_bench.rs` (Stats-Pipeline: 8 WUs, deep chain)

After resource caching, the pipeline still shows ~14% overhead vs hand-written
baseline on the Stats-Pipeline (deep single chain). Three variants tested:

| Approach | vs baseline | Inner loop instructions | Column loads in loop |
|---|---|---|---|
| Hand baseline (raw ptrs, one loop) | 1.00x | 41 | 2 |
| Pipeline (fn ptr devirt + inline) | 1.13-1.16x | 50 | 5 |
| Mega-dispatch (#[inline(always)] chain) | 1.14-1.16x | 99 (includes morsel mgmt) | 5 |
| **Flattened (inline WU bodies, cu/cw)** | **1.00-1.03x** | matches baseline | 2 |

**Root cause:** the fn-pointer-based WU composition forces inter-WU data
flow through column memory. `stats_fft_pass1` stores `c4` to column 4
via `cw()`. `stats_fft_pass2` reloads it from column 4 via `cu()`. Even
fully inlined, LLVM cannot prove that the `cw()` store to column 4 doesn't
alias with a `cu()` read from column 6 (both are raw pointers from the
same struct). So it reloads instead of reusing the local.

The flattened version writes the same `cw(c(4), i, c4)` store, but uses
the LOCAL variable `c4` for subsequent computation instead of reloading
via `cu(c(4), i)`. LLVM sees that `c4` is already available and skips
the reload. The column store still happens (other chains may read it),
but the value is reused from the register.

**Key finding:** `cu()`/`cw()` access methods have ZERO overhead vs raw
pointers when used within a single scope (flattened = 1.00x baseline).
The overhead comes exclusively from the fn-pointer WU composition pattern
forcing store-then-reload between WU boundaries.

**Design implication:** the schedule flattener must emit code that reuses
local variables across WU boundaries. If WU A writes `c4` and WU B reads
`c4`, the flattened code should:

```rust
let c4 = compute_a(...);
cw(c(4), i, c4);       // store for downstream chains
let c6 = compute_b(c4); // reuse local, NOT cu(c(4), i)
cw(c(6), i, c6);
```

This is a CODE GENERATION problem, not a runtime dispatch problem. The
schedule is known at compile time (static analysis of WorkUnit declarations).
A proc macro or const-generic flattener can emit this pattern directly.

**Results by scenario (with resource caching):**

- ECS-Game (4 chains, 10 WUs): 1.00-1.01x baseline. Multi-chain scenarios
  have no overhead because chains are `#[inline(never)]` boundaries anyway —
  inter-chain data flow goes through columns by design.
- Partition (4 chains, 6 WUs): 0.97-1.04x baseline. Same — chain boundaries
  are natural materialization points.
- Stats-Pipeline (1 deep chain, 8 WUs): 1.13-1.16x baseline with fn ptrs,
  1.00x with flattened. The overhead exists only in deep single-chain
  scenarios where many WUs pass data through columns within one chain.

**Conclusion:** the execution model is sound. The morsel + chain structure
adds zero overhead for multi-chain workloads. For deep single-chain
workloads, the schedule flattener must generate code with local variable
reuse across WU boundaries to eliminate the 14% store-reload penalty.
This is achievable at compile time since the chain composition is static.

### Chain-internal columns are dead stores — only chain outputs need materialization

The chain grouper places WUs in the same chain precisely because their
column dependencies are internal: WU A writes column X, WU B reads
column X, and no other chain reads X between A and B. That's the
definition of "fusible" — the data flow is local to the chain.

This means intermediate column stores within a chain are dead stores
from the perspective of correctness. Only columns that are READ by a
downstream chain (or are final consumer outputs) need to be written to
memory. Everything else can stay in registers as locals.

**The chain as a pure function pipeline:**

Within a chain, processing row n is: `d(c(b(a(input_n))))`. Each WU is
a pure function that takes column values and produces column values. The
chain is a composition of these functions. The intermediate "columns"
are just named wires between stages — they don't need memory backing
within the chain.

The memory layout makes this natural. All columns in the chain are
co-located in one arena. Column A at row n is `arena_base + A_offset + n`.
Column B is `arena_base + B_offset + n`. The cursor for row n accesses
all columns at known offsets from one base pointer. The flattener emits:

```rust
for n in morsel_start..morsel_end {
    // read chain inputs (columns written by upstream chains)
    let input_a = *col_a_ptr.add(n);
    let input_b = *col_b_ptr.add(n);

    // WU pipeline — locals flow between stages, no intermediate stores
    let stage1 = wu_1(input_a, input_b);  // reads a, b → produces intermediate
    let stage2 = wu_2(stage1);            // reads intermediate → produces next
    let stage3 = wu_3(stage2, input_a);   // can read chain inputs too
    let output  = wu_4(stage3);

    // write ONLY chain outputs (columns read by downstream chains)
    *col_output_ptr.add(n) = output;
}
```

No intermediate column stores. No intermediate column reloads. Data
flows through registers for the entire chain depth. The only memory
operations are: read chain inputs (from upstream chains' outputs) and
write chain outputs (for downstream chains to read).

**Why this is correct:**

The chain grouper guarantees that no external consumer reads a chain-
internal column before the chain completes. The chain processes all
its WUs per row before moving to the next row. So a "column" that's
only written by WU A and read by WU B within the same chain never
needs to exist in memory — it's a register-to-register data path.

If a column IS read by a downstream chain, it's a chain OUTPUT. The
chain grouper placed the consumer in a different chain precisely
because there's a materialization boundary there (fan-in, max_cols
exceeded, or pipeline breaker). The chain output must be written to
memory at the chain boundary.

**What the flattener needs to know (all statically available):**

1. Which columns are chain-internal (written and read only within this
   chain) — these become locals, no memory ops.
2. Which columns are chain inputs (written by an upstream chain, read
   by this chain) — these are read from memory at the start.
3. Which columns are chain outputs (written by this chain, read by a
   downstream chain or the consumer) — these are written to memory
   at the end.
4. The WU execution order within the chain (topological sort, already
   computed by the chain grouper).

All of this information is available at schedule-build time from the
WorkUnit declarations' `type Read` and `type Write` associated types
and the chain grouper's dependency analysis.

**Design approach: the chain is a generated function.**

Each chain becomes a monomorphized function generated at compile time
(proc macro on `#[register]`, or const-generic composition). The
function:
1. Caches resources to a stack-local array (resource caching finding)
2. Reads chain input columns once per row
3. Calls WU logic as inline pure functions, passing locals
4. Writes chain output columns once per row

The WU's `each` closure signature changes from
`|col_a: &T, col_b: &mut U| { ... }` to a pure function
`fn(input_a: T, input_b: T) -> (U, V)` that takes values and returns
values. The flattener wires the returns to the next stage's inputs.

Columns that are both chain-internal AND chain-output (written by an
early WU, used by a later WU, and also read by a downstream chain)
get both: the local is reused within the chain, AND the value is
stored to memory for the downstream chain.

**Expected result:** zero overhead vs hand-written code for ALL chain
depths, not just multi-chain scenarios. The 14% deep-chain overhead
disappears because there are no intermediate stores or reloads.

4. Register allocation that avoids the recompute penalty of register-

### TESTED: scheduling strategies — pipeline-chase beats all alternatives

**Test file:** `mock/research/hilavitkutin_bench.rs` (Partition scenario, 4 chains)

8 scheduling variants tested with frame-based measurement (200-2000 frames).
The Partition scenario has independent chains A and B (cols 0-7, 8-15),
then dependent bridge chain C (reads outputs of both) and final chain.

**Results at 250K-2M rows (p50 ns/elem, ratio vs sequential):**

| variant | 250K | 2M | strategy |
|---|---|---|---|
| sequential | 9.49ns | 9.89ns | A→B→C, single core |
| AB-thread | 0.93x | 0.92x | A‖B on 2 threads, C sequential |
| interleave | 1.00x | 1.02x | A+B share L1 (half morsel each) |
| split-C | 0.77x | 0.78x | A‖B, C row-split with local copy |
| rowsplit-AB | 0.93x | 0.94x | half(A)+half(B) per thread, C seq |
| 4way-C | 0.79x | 0.78x | A‖B, C 4-way row-split |
| **pipe-chase** | **0.62x** | **0.58x** | **A+B interleaved core 0, C chases core 1** |
| steal-C | 0.88x | 0.87x | A‖B, CAS work-stealing for C |

**pipe-chase at 2M: 0.58x sequential — 42% faster. Scales BETTER
at larger sizes (0.67x at 50K → 0.58x at 2M).**

**Why pipe-chase wins (and the others don't):**

1. **Full overlap.** A+B runs on core 0 (interleaved morsels, both fit
   in L1). C runs on core 1, starting as soon as ONE morsel of A+B
   completes. C never waits for A+B to fully finish — it processes
   rows progressively. The total time ≈ max(A+B time, C time) instead
   of A+B time + C time.

2. **Zero copy overhead.** C reads directly from A+B's output columns.
   The co-located arena means A's output (col 7) and B's output (col 15)
   are in the same allocation. Core 1 reads them via cache coherence —
   on M1's unified memory, this is essentially free once the data is
   written by core 0.

3. **Minimal synchronization.** One `AtomicUsize` per chain (A progress,
   B progress). Core 0 does `store(end, Release)` after each morsel.
   Core 1 does `load(Acquire)` to check how far it can process. On
   aarch64, Release = `stlr` (one instruction), Acquire = `ldar` (one
   instruction). No locks, no barriers, no CAS.

4. **No contention.** Core 0 only writes to A+B columns. Core 1 only
   writes to C columns. Different cache lines (columns are separate
   arrays in the arena). No false sharing.

**Why the others are worse:**

- **interleave (1.00x):** sharing L1 between A and B doesn't help because
  each already fits independently. Halving the morsel just adds overhead.

- **split-C (0.77x):** the copy of input columns to thread-local buffers
  costs O(N) per frame. At 2M rows, copying ~16MB of column data per
  frame eats the parallelism gain. pipe-chase avoids copying entirely.

- **steal-C (0.87x):** CAS contention. Both threads compete for C morsels
  via `compare_exchange`. Failed CAS = wasted cycles. pipe-chase assigns
  C to one thread, no contention.

- **4way-C (0.78x):** more parallelism within C but still pays the copy
  cost. No better than 2-way split-C.

- **rowsplit-AB (0.93x):** splitting A and B by rows doesn't help because
  they're already independent — running them on separate threads (AB-thread)
  gives the same 0.92x. The bottleneck is C, not A+B.

**Design validation:** the pipe-chase pattern IS the §7.3 design:
`AtomicUsize` progress per chain, downstream starts as soon as
`progress >= needed_row`. The benchmark empirically validates that
this design produces near-optimal scheduling for dependent chain
graphs on multi-core hardware.

**Design: plan-time strategy selection.**

The scheduler knows at plan time:
1. Each chain's column count (determines morsel size)
2. Each chain's WU count and complexity (from `#[register]` metadata)
3. The dependency graph (which chains are independent vs dependent)
4. Total row count (from the data source or previous chain)

From this, it can estimate the compute weight ratio between producers
(independent chains) and consumers (dependent chains). The selection
rule:

```rust
fn select_strategy(
    producer_weight: f64,  // estimated total cycles for A+B
    consumer_weight: f64,  // estimated total cycles for C
    row_count: usize,
) -> Strategy {
    if row_count < 10_000 {
        return Sequential;  // thread overhead dominates
    }
    let ratio = consumer_weight / producer_weight;
    if ratio > 0.5 {
        // C is heavy relative to A+B — both threads should help with C
        ChaseSteal
    } else if producer_weight < LIGHT_THRESHOLD {
        // A+B is light — producer finishes fast, steal C tail
        Adaptive
    } else {
        // default: overlap A+B with C chasing
        PipeChase
    }
}
```

The weight estimate doesn't need to be precise. It's a rough
classification (light/medium/heavy) derived from:
- WU count × estimated ops per WU (from column read/write counts)
- Column count (affects morsel size, therefore iterations per chain)

The selection happens once at plan time (microsecond cost). Wrong
selection results in ~10-20% suboptimal performance, not catastrophic
failure — all strategies are correct, just differently efficient.

**Budget selection:**

The frame budget (`morsels_per_frame`) also affects performance:
- budget=4: best for pipe-chase at large N (more overlap opportunities)
- budget=64: best for chase+steal (less CAS contention, longer steal windows)
- adaptive ignores budget entirely (runs unbounded)

For budgeted execution (e.g. game loops with time budgets), the
scheduler should use larger budgets for heavier chains to reduce
per-frame dispatch overhead. For unbounded execution (batch pipelines),
adaptive is always best or within 5% of best.

### TESTED: deep-chase (both-ends convergence) — doesn't help serial paths

**Strategy:** for every chain, maintain front + tail atomics. Producer
advances from front, chaser advances from tail. Both converge on
the middle of each chain. Applied to ALL chains including serial deps.

**Results at 2M rows:**

| pipeline | deep-chase total | vs sequential |
|---|---|---|
| Partition (wide) | 17.0ms | 0.45x (helps, but worse than adaptive's 0.32x) |
| PolkaPipeline (deep) | 22.0ms | 0.94-1.15x (no help, sometimes worse) |

**Why it fails on deep pipelines:** the chaser can only process chain 1's

With 5 chains in a serial path, the chaser must wait for each preceding
chain to finish before starting the next. The per-chain convergence
overhead (CAS per morsel, atomic dep checks) exceeds any savings from

**Why it's worse than adaptive on wide pipelines:** adaptive runs A+B

The serial dependency depth is a fundamental limit. No 2-thread
strategy can parallelize a chain that depends on the previous chain's
complete output. The only way to speed up deep serial paths is compute
optimization (DSE, resource caching, LLVM codegen) — not threading.

### SYNTHESIS: pipeline-adaptive optimisations from benchmark findings

Six concrete optimisations the engine should implement, all informed
by the benchmark results above.

**1. Dual-mode chain formation: single-core vs multi-core.**

The chain grouper currently fuses aggressively to minimise chain count
(= minimise data walks). This is optimal for single-core but creates
dominant chains that can't be parallelised. Example: the game world's
6-WU physics+collision+terrain chain (15 cols) dominates compute.

Multi-core mode should split chains at natural boundaries when the
split creates parallelisable width. Physics (4 WUs, 9 cols) could
run on one core while collision+terrain (2 WUs, 10 cols) chases.
The extra data walk costs ~10% but the parallelism gains ~40%.

Selection: based on core count and row count. <10K rows or 1 core:
single-core mode. >10K rows and 2+ cores: multi-core mode.

**2. Thread count adapts to parallelisable width.**

Currently fixed at 2 (producer + chaser). The game world has 3
independent roots — with 4 cores, three producers + one chaser.
Thread count = min(physical_cores, max_parallelisable_width + 1).
The "+1" is the chaser thread for dependent chains.

**3. Compute-weighted morsel interleaving.**

When the producer interleaves independent chains, it should weight
by compute cost. If chain A is 10x heavier per morsel than chain B,
process 10 A morsels per 1 B morsel so both progress counters advance
at roughly the same rate. The chaser then has consistently available
work instead of waiting for the slow chain.

Weight estimate: morsel_time ≈ morsel_size × wu_count × avg_col_accesses.
Doesn't need precision — 2x accuracy is sufficient for interleaving.

**4. Codegen flattener with DSE + stores-at-end.**

For each chain, emit a monomorphised function:
1. Cache resources to stack-local array
2. Loop over morsel range
3. All WU computation using locals (no intermediate column reads)
4. Only chain-output columns stored (chain-internal skipped)
5. All stores grouped at end of loop body

The chain grouper marks each column as chain-internal or chain-output.
The flattener uses this to eliminate dead stores and group remaining
stores.

**5. Automatic per-chain resource caching.**

The chain dispatcher snapshots all accessed resources to a stack-local
array before the morsel loop, redirects `res_ptr`. LLVM proves stack
doesn't alias heap column data, promotes all resource accesses to
registers. After the loop, writes back mutable resources.

Mechanical — derived directly from the WU declarations' resource
read/write sets. The codegen emits it automatically.

**6. Compute-balanced chain splitting.**

The grouper should consider compute balance, not just column count.
If splitting a chain creates two chains of roughly equal compute weight,
the split enables parallelism. If it creates one heavy and one trivial
chain, the split wastes a data walk without useful parallelism.

Weight proxy: WU count × column access count per WU. The grouper
prefers splits that create balanced partitions when in multi-core mode.

### THEORY: spanning tree decomposition + adaptive chain configurations

The WU dependency graph is a DAG. To reason about parallelism and
chain formation, we decompose it into a spanning tree with a trunk
(critical path) and branches (side paths that join at fan-in points).
Chain boundaries become adaptive — the executor picks between
multiple pre-computed chain configurations at plan time based on
pipeline shape and hardware.

#### DAG → spanning tree

A DAG has fan-in (node with multiple parents) and fan-out (node
whose output feeds multiple children). A tree has no fan-in. To
convert, we identify the **critical path** (longest weighted path
from any root to any leaf) and make it the trunk. All other paths
become branches that join the trunk at fan-in points.

**Algorithm:**

1. Compute the weight of each WU: `wu_weight = col_reads + col_writes`
   (proxy for per-element compute cost).
2. For each root (WU with no predecessors), compute the longest
   weighted path to any leaf. The globally longest path is the trunk.
3. Walk the trunk from root to leaf. At each fan-out point, the trunk
   continues along the heaviest child. Other children become branch
   roots.
4. At each fan-in point (where a branch rejoins the trunk), record
   the join. The fan-in WU's "primary parent" is the trunk predecessor.
   All other parents are branch joins.
5. Recurse: each branch is itself a sub-DAG that can be decomposed
   into a sub-trunk + sub-branches.

The result is a tree where:
- The trunk represents the serial bottleneck (critical path)
- Branches represent parallelisable work
- Fan-in points are natural barrier boundaries
- Branch leaves are natural chain endpoints

**Example (game world):**

```
TRUNK (critical path):
  gravity(0) → integ_vel(1) → integ_pos(2) → spatial_hash(3)
  → coll_resp(4) → ai_decide(5) → move_target(6)

BRANCHES (join the trunk at fan-in points):
  wind(0) ─────────────────→ joins integ_vel
  hunger(0) ────────────────→ joins ai_decide
  terrain_samp(3) → weather → joins ai_decide
  gather(6) → deposit(7) ──→ (leaf: consume)
  view_matrix(6) → frustum_cull(7) → (leaf: render_prep)
```

The trunk has depth 7 (7 WUs in sequence). Branches have depth
1-3. The maximum parallelism at any point is 4 (gravity + wind +
hunger all independent at order 0; terrain + spatial_hash at order 3).

#### Finding natural barriers

A **natural barrier** is a fan-in point where multiple branches
must synchronise before a downstream WU can start. These are the
mandatory chain break points — you CANNOT fuse across a fan-in
because the fused chain would need data from multiple independent
branches.

Natural barriers in the game world:
- `integ_vel`: needs gravity + wind (barrier between roots and physics)
- `ai_decide`: needs collision + terrain + hunger (barrier before AI)
- `render_prep`: needs frustum_cull + AI state (barrier before render)

Everything between two consecutive barriers (along any path) is a
candidate chain segment. The chain grouper's job is to decide which
segments to fuse and which to keep separate.

#### Walking back from leaves

Starting from each leaf and walking back to the nearest barrier
gives you the minimum chain that must exist:

```
render_prep ← frustum_cull ← view_matrix  (3 WUs, render chain)
consume ← deposit ← gather                (3 WUs, resource chain)
move_target                                (1 WU, movement chain)
```

These are "forced chains" — they can't be split further because
they're between two barriers (or between a barrier and a leaf).

Walking further back from the barriers gives the segments between
barriers:

```
ai_decide                                  (1 WU, AI chain)
coll_resp ← spatial_hash ← integ_pos ← integ_vel  (4 WUs, physics chain)
weather ← terrain_samp                    (2 WUs, terrain chain)
hunger_tick                                (1 WU, hunger chain)
gravity                                    (1 WU, gravity chain)
wind                                       (1 WU, wind chain)
```

This gives the MAXIMUM chain count (10 chains). Fusing adjacent
segments (between the same barriers) gives fewer chains.

#### Adaptive chain configurations

Instead of a single chain configuration, the grouper produces
multiple configs ranked by parallelism vs cache locality trade-off:

**Config MAX-FUSE (fewest chains, best cache):**
Fuse everything that fits in L1. Result: the 12-chain config the
current grouper produces. Best for single-core, 1 data walk per
fused segment.

**Config BALANCED (split at compute balance points):**
Split the dominant chain (physics+collision+terrain, 6 WUs) into
physics (4 WUs, 9 cols) and collision+terrain (2 WUs, 10 cols).
Result: 13 chains, but the two halves of the dominant chain can
run on separate cores. The extra data walk costs ~10% but enables
~30% parallelism gain.

**Config MAX-SPLIT (most chains, maximum parallelism):**
Every barrier-to-barrier segment is its own chain. Result: 10
chains. Maximum parallelism but more data walks. Useful when cores
are plentiful and row count is large (data walk cost is amortised).

The executor selects the config at plan time:

```
if cores == 1 || rows < 10_000 {
    Config::MAX_FUSE
} else if cores == 2 {
    Config::BALANCED
} else {
    Config::MAX_SPLIT
}
```

#### Adaptive chains ≈ adaptive morsels

Morsel size adapts to L1 and column count: more columns → smaller
morsels → more morsel boundaries but all data fits in cache.

Chain granularity adapts to core count and row count: more cores →
more chains → more data walks but more parallelism.

Both are the same trade-off: **granularity vs locality.** Finer
granularity enables more parallelism (morsels across rows, chains
across WUs) but costs locality (morsel boundary overhead, chain
boundary data walk).

The optimal point depends on the specific pipeline + hardware:
- polka-dots (200 rows, 1 core): MAX_FUSE + large morsels
- saalis (100K rows, 2 cores): BALANCED + medium morsels
- loimu (1M rows, 8 cores): MAX_SPLIT + small morsels

The spanning tree decomposition gives the vocabulary for this:
trunk depth determines how much of the pipeline is inherently
serial. Branch width determines how much can be parallelised.
Natural barriers are the fixed points that all configs must respect.
The configs differ only in how much they fuse between barriers.

#### What WUs need to provide

For all of this to work automatically, WUs must declare:
1. **Read column set** — which columns this WU reads (but doesn't write)
2. **Write column set** — which columns this WU produces
3. **Order** — relative execution order among WUs that write the same
   column (resolves write-write ambiguity in the DAG)

That's it. Everything else — trunk identification, barrier detection,
chain formation, config generation, morsel sizing, strategy selection —
is derived by the scheduler from these three declarations.

The user writes:
```rust
#[work_unit(order = 3)]
fn spatial_hash(
    #[read] pos_x: &Column<PosX>,
    #[read] pos_y: &Column<PosY>,
    #[read] pos_z: &Column<PosZ>,
    #[write] coll_flag: &mut Column<CollFlag>,
    #[write] coll_nx: &mut Column<CollNX>,
) {
    // ...
}
```

The `#[work_unit]` proc macro extracts read/write sets and order,
registers the WU, and the scheduler does the rest.

### THEORY: phased execution with trunks, branches, and spatial morsels

A more complete decomposition of the WU DAG that exploits ALL
available parallelism. Replaces the simple "chain grouper + strategy
selector" model with a layered structure.

#### Vocabulary

**Phase**: a wide section of the DAG between narrow bottlenecks.
Detected by finding "waist" points where the number of concurrently
live paths drops to a minimum. Each phase has maximum internal
parallelism. Phases execute in sequence (each phase starts when the
previous phase produces enough data — pipeline parallel via progress
counters).

**Trunk**: a sequential critical path within a phase that shares NO
columns with other trunks in the same phase. Trunks are fully
independent — they can run on separate cores with zero synchronisation,
zero shared cache lines, zero atomics. A trunk is the old "partition"
concept, but derived from the DAG structure, not hand-declared.

**Branch**: a side path attached to a trunk. It shares some columns
with its trunk (reads trunk output, or writes trunk input). Branches
are chains within the trunk's execution scope — they execute as
chaser chains during the trunk's morsel walk, or fused into the trunk
if column count permits.

**Bridge**: a branch that reads from multiple trunks. It creates a
cross-trunk dependency but doesn't break trunk independence. The bridge
runs AFTER all its parent trunks reach the required row range (the
chaser pattern with multiple progress counters). Bridges are the
fan-in resolution mechanism.

**Morsel**: a spatial window into a chain: rows [st..en] × columns
[the columns this chain touches]. The morsel size is the largest
window that fits in L1. Not just a row range — the column dimension
matters too. The chain's column set determines the "width" of the
window, the row range determines the "height."

**Time**: the third dimension. Frame budgets, pipeline parallelism
between phases, latency hiding. Phases overlap: the first trunk of
phase N+1 can start as soon as phase N's last trunk produces enough
rows (atomic progress). Time budgets control how many morsels each
trunk processes per frame.

#### DAG decomposition algorithm

**Step 1: Phase detection.**

Find waist points — positions in the topological order where the
concurrent path count is minimal. A waist is a set of WUs through
which ALL paths from any root to any leaf must pass.

```
Concurrent paths at each order level:
  order 0: gravity, wind, hunger = 3 paths (wide)
  order 1: integ_vel = 1 path (waist!)
  order 2: integ_pos = 1 path (waist)
  order 3: spatial_hash, terrain_samp = 2 paths (widening)
  order 4: coll_resp, weather = 2 paths
  order 5: ai_decide = 1 path (waist!)
  order 6: move_target, gather, view_matrix = 3 paths (wide)
  order 7: deposit, frustum_cull = 2 paths
  order 8: consume, render_prep = 2 paths (leaves)
```

Waists at orders 1-2 and 5. These define phase boundaries:
- Phase A: orders 0-2 (roots → integ_pos). Waist: integ_vel/pos.
- Phase B: orders 3-5 (collision, terrain → ai_decide). Waist: ai_decide.
- Phase C: orders 6-8 (movement, gathering, rendering → leaves).

Phase A is narrow (3 roots converging to 1 WU). Phase B widens
(2 parallel paths) then narrows (1 WU). Phase C is wide (3 parallel
paths to 2 leaves).

**Step 2: Trunk identification per phase.**

Within each phase, find maximal sets of WUs that share no columns.
Each set is a trunk.

Phase A: gravity and wind both write accel columns → they share
column space → they're in the same trunk. hunger writes hunger/stamina
(disjoint from accel) → separate trunk.
- Trunk A1: gravity → wind → integ_vel → integ_pos (physics)
- Trunk A2: hunger_tick (independent)

Phase B: spatial_hash and terrain_samp read pos (from phase A output,
not from each other) → they share pos columns as READ inputs but
different WRITE columns → they CAN be separate trunks if we treat
read-only shared columns as non-conflicting (reads don't conflict
with reads).
- Trunk B1: spatial_hash → coll_resp (collision)
- Trunk B2: terrain_samp → weather (terrain/weather)
- Bridge: ai_decide (reads from B1 + B2 + A2, fan-in)

Phase C: move_target, gather, and view_matrix all read from AI output
but write to disjoint columns → separate trunks.
- Trunk C1: move_target
- Trunk C2: gather → deposit → consume (resources)
- Trunk C3: view_matrix → frustum_cull → render_prep (rendering)

**Step 3: Column sharing analysis.**

Read-read sharing between trunks is fine — reads don't conflict.
Write-write or read-write sharing forces trunks into the same trunk
or creates a branch relationship.

This means: trunks within a phase share NO write columns. They MAY
share read columns (both trunks read from the same upstream output).
The upstream output is produced by a previous phase and is immutable
during the current phase — so reads are safe.

**Step 4: Branch assignment.**

Branches are WUs that share write columns with a trunk, or that
read from a trunk's write output within the same phase. They
execute as chaser chains within the trunk's morsel scope.

In the game world, most WUs are either trunk nodes (sequential
critical path within a phase) or bridges (fan-in between trunks).
True branches (side paths that rejoin the same trunk) appear when
a WU writes to a column that the trunk later reads — this forces
the WU to run before the trunk's consumer within the same chain.

#### Execution model

```
Phase A:
  Core 0: Trunk A1 (gravity → wind → integ_vel → integ_pos)
  Core 1: Trunk A2 (hunger_tick)
  → A2 finishes almost instantly, core 1 idles (or steals A1 tail)

Phase B (starts as Phase A produces rows):
  Core 0: Trunk B1 (spatial_hash → coll_resp)
  Core 1: Trunk B2 (terrain_samp → weather)
  Bridge: ai_decide chases on whichever core finishes first

Phase C (starts as Phase B produces rows):
  Core 0: Trunk C1 (move_target) then helps with C2 or C3
  Core 1: Trunk C3 (render pipeline)
  Core 0→: Trunk C2 (resource pipeline) after C1 finishes
```

Total parallelism: 2 trunks per phase × 3 phases. With 2 cores,
each phase has one trunk per core. Phases pipeline-overlap via
progress counters.

Compared to the current approach (12 chains, sequential + chaser):
- Current: 38ms for 1M agents (sequential), 36ms (adaptive, 0.93x)
- Predicted with phased execution: each phase's parallel trunks cut
  its time roughly in half. 3 phases × ~half = ~0.5x sequential.
  Estimated: ~19ms for 1M agents. This would match the Partition
  scenario's 0.58x ratio.

#### Why this works better than the current approach

The current grouper fuses WUs into chains greedily along dependency
edges. It doesn't distinguish between trunks (independent parallel
paths) and branches (dependent side paths). As a result, it fuses
physics+collision+terrain into one 6-WU chain that becomes the
serial bottleneck.

The phased decomposition:
1. Identifies that collision and terrain are INDEPENDENT (trunk B1 ‖ B2)
   even though they both depend on physics (trunk A1). The current grouper
   fuses them because they share columns transitively.
2. Enables pipeline parallelism between phases. Phase B starts processing
   rows as soon as Phase A produces them — no need to wait for Phase A
   to finish entirely.
3. Naturally assigns work to cores: one trunk per core per phase.
   No CAS contention, no progress counter polling between trunks.

#### What the scheduler needs

From WU declarations (read set, write set, order), the scheduler:
1. Builds the DAG
2. Detects phases (waist analysis)
3. Identifies trunks per phase (column conflict graph)
4. Assigns branches and bridges
5. Computes chain formation within each trunk (fuse or split based
   on L1 budget + compute balance)
6. Sizes morsels per chain (L1 / (chain_cols × sizeof(T)))
7. Selects execution strategy per phase (sequential if narrow,
   parallel if wide, adaptive if mixed)

All computed at plan time from static WU metadata. Zero runtime cost.

#### The morsel as spatial window

A morsel is rows × columns. The row range is determined by the
morsel sizing formula. The column set is determined by the chain's
column union. Together they define a rectangular region of the
columnar store that must fit in L1.

This is already what the current morsel sizing does implicitly —
`L1 / (col_count × sizeof(T))` computes the row range that makes
the full column set fit. Making it explicit as a spatial window opens
the door to future optimisations:
- Splitting a chain's columns across morsels (partial column coverage)
- Adapting morsel shape to the access pattern (column-major vs row-major)
- NUMA-aware morsel placement (different morsels on different nodes)

These are future work. The current row-only morsel is correct and
sufficient for all benchmarked workloads.

### THEORY: within-chain head+tail convergence

Within a chain, rows are independent. The chain processes each row
through the same WU sequence. Row 0 and row N have no dependency on
each other — the execution order of rows does not matter for
correctness.

Therefore: EVERY chain can be split into head+tail processing with
2 threads. Thread A processes rows 0→middle, thread B processes rows
N→middle. Both run the identical fused WU pipeline. They converge
in the middle. Every chain gets ~2x parallelism.

This is orthogonal to trunk parallelism:
- Trunk parallelism: independent chains on separate cores
- Head+tail convergence: same chain on 2 cores from opposite ends

**Combined with phased execution:**

Phase 1 has one big serial trunk (6 WUs, 15 cols). Currently runs
on one core. With head+tail convergence, it runs on 2 cores — each
processing half the rows. The phase completes in ~half the time.

With 4 cores: 2 trunks × 2 directions each = 4 cores utilised.
With 2 cores: single-trunk phases still get 2x from convergence.

**The serial bottleneck disappears.** Phase 1's 6-WU trunk was the
bottleneck because it ran on one core. With head+tail convergence,
it runs on 2 cores. The phased executor's 0.82x could improve to
~0.55-0.6x if Phase 1 (the dominant phase) gets halved.

**Resource accumulation caveat:** WUs that accumulate resources
(ru/rw per element) have a cross-row dependency through the resource.
Row N's resource write affects row N+1's resource read. Head+tail
convergence breaks this — the head thread and tail thread would
accumulate into the same resource concurrently.

Fix: each thread gets its own resource accumulator (same as the
stack-local resource caching pattern). After convergence, merge the
two accumulators. For additive accumulation: just add them. For
non-commutative accumulation: split the range at a morsel boundary
and process sequentially within each half, merge at the end.

**When to use head+tail convergence:**
- Always, when cores > 1 and rows > morsel_size * 2
- Even for single-trunk phases (the serial bottleneck case)
- Skip for chains with non-commutative resource accumulation
  (rare — most accumulation is additive: counters, sums, max)

**Default behaviour:** the executor should ALWAYS use head+tail
convergence unless:
1. Only 1 core available
2. Fewer than 2 morsels worth of rows
3. Chain has non-commutative resource accumulation (detected at
   plan time from WU metadata)

### THEORY: pipelined phases + core-pinned pool + adaptive wake

Three improvements to the phased execution model, building on
the benchmark findings.

#### 1. Thread pool with hybrid wake strategy

Spinning wastes power and competes with worker threads. Parking
(`thread::park`/`unpark`) has microsecond wake latency. The hybrid:

```
fn pool_thread_wait(signal: &AtomicUsize) {
    // spin briefly (hot path — wakes in nanoseconds)
    for _ in 0..128 {
        if signal.load(Ordering::Acquire) != 0 { return; }
        core::hint::spin_loop();
    }
    // park (cold path — saves power, wakes in microseconds)
    while signal.load(Ordering::Acquire) == 0 {
        std::thread::park();
    }
}
```

The dispatcher calls `thread.unpark()` when signalling work.
`park`/`unpark` is available on all std targets (futex on Linux,
os_unfair_lock on macOS, etc.). No platform-specific features needed.

Pool size = physical_core_count (detected at startup). Threads are
created once and persist for the pipeline's lifetime.

#### 2. Core-pinned trunks + leftover threads for convergence

Each trunk is assigned to a specific pool thread at plan time.
The trunk stays on that thread (and therefore that core) across all
its morsels — warm L1, no migration.

When a phase has fewer trunks than pool threads, the leftover threads
pick up:
- **Convergence work:** process the tail end of a trunk's row range
  (head+tail convergence)
- **Long branches:** branches with significant compute that benefit
  from parallel execution
- **Bridge processing:** fan-in bridges that can start as soon as
  their parent trunks produce enough rows

Assignment priority:
1. One thread per trunk (guaranteed)
2. Remaining threads do convergence on the heaviest trunk
3. If convergence is done, pick up branches/bridges

#### 3. Pipelined phases — the key insight

**The current model runs phases sequentially: phase 1 must complete
all rows before phase 2 starts. This wastes the pipeline's depth.**

Phase 2 only needs phase 1's output for the rows it's about to
process. It doesn't need ALL of phase 1's rows — just enough for
one morsel window. Once phase 1 has produced `morsel_size_phase2`
rows, phase 2 can start processing from row 0.

This is the §7.3 atomic progress pattern applied between phases:

```
// Phase 1 trunk (producer):
for morsel in morsels {
    process_morsel(morsel);
    phase1_progress.store(morsel.end, Release);
}

// Phase 2 trunk (consumer — runs on a different thread):
for morsel in morsels {
    // wait until phase 1 has produced enough rows
    while phase1_progress.load(Acquire) < morsel.end {
        // yield to pool or spin briefly
    }
    process_morsel(morsel);
    phase2_progress.store(morsel.end, Release);
}
```

**The effect:** all phases overlap. Phase 1 is on morsel 5 when
phase 2 is on morsel 3 when phase 3 is on morsel 1. The pipeline
is full. With 4 phases and 4 cores: all phases active simultaneously,
each processing a different row window.

**Total time changes from sequential (sum of phases) to pipelined
(max of phases + fill latency):**

- Current sequential: T = T_phase0 + T_phase1 + T_phase2 + T_phase3
- Pipelined: T ≈ max(T_phase0, T_phase1, T_phase2, T_phase3) + fill

For the game world at 1M rows:
- Phase 1 dominates (~60% of compute)
- Current phased-pool: 27.3ms (phases sequential with convergence)
- Predicted pipelined: ~T_phase1 + fill ≈ 0.6 × 40ms + 1ms ≈ 25ms
  (similar to phased-conv's 25ms, but with consistent small-N behaviour)

But the real win is on deeper pipelines with more phases. A 10-phase
pipeline currently takes 10 × T_avg. Pipelined: max(T_phase) + 9 × fill.
For balanced phases: ~T_avg + 9 × fill ≈ T_avg + 0.1ms = nearly 10x
faster than sequential.

**The convergence + pipeline combination:**

Each phase's trunk runs with head+tail convergence (2 threads).
The pipeline overlaps phases on additional threads. With a 4-core
pool and 4 phases:
- Core 0: phase 1 trunk (head)
- Core 1: phase 1 trunk (tail convergence)
- Core 2: phase 2 trunk (head, chasing phase 1's progress)
- Core 3: phase 2 trunk (tail convergence, chasing phase 1's progress)

As phase 1 completes morsels, phase 2 starts. As phase 2 completes,
phase 3 starts on cores 0-1 (which finished phase 1). The pipeline
fills and stays full.

**What the scheduler computes at plan time:**

From the phase decomposition + core count:
1. Assign trunks to cores (one trunk per core per phase)
2. Assign convergence to leftover cores
3. Compute pipeline fill order (which phase starts on which core)
4. Compute morsel sync points (phase N+1 starts when phase N has
   produced enough rows for one morsel)
5. Set up atomic progress counters per phase

All computed once. The runtime cost is one `load(Acquire)` per morsel
to check the upstream phase's progress — the same cost as the
chain-level progress counters we already benchmark at ~0 overhead.

### THEORY: per-core compiled dispatch with predictive parking

The typed dispatch result reveals the final architecture. At plan
time, the scheduler knows everything: the DAG, the phases, the
trunks, the chain composition, the morsel sizes, the core count.
It can compile a complete per-core execution plan — a monomorphised
function for each core that contains the entire pipeline as a
sequence of devirtualised WU calls with baked-in morsel boundaries
and phase transitions.

**Per-core compiled dispatch:**

For each physical core, the scheduler generates (via proc macro or
const-generic composition) a single `#[inline(never)]` function
that encodes:
1. Which phases this core participates in
2. Which trunks/row ranges this core processes per phase
3. The fused WU sequence per chain (local `&[WuFn]` slices)
4. Morsel boundaries (compile-time constants from L1 / col_count)
5. Phase sync points (where to check upstream progress)

The per-core function is the entire "program" for that core. It
runs start to finish. No runtime dispatch decisions. No function
pointer lookups. No dynamic scheduling. Pure sequential execution
with static sync points.

```rust
// generated for core 0: head half of all phases
fn core0_program(store: &mut Store, rows: usize, progress: &[AtomicUsize]) {
    let mid = rows / 2;

    // phase 0: roots (head half)
    let ops_gravity: &[WuFn] = &[wu_gravity]; // LOCAL — devirtualised
    for_morsel(store, ops_gravity, 0, mid, 3072);
    let ops_wind: &[WuFn] = &[wu_wind];
    for_morsel(store, ops_wind, 0, mid, 4096);
    let ops_integ_vel: &[WuFn] = &[wu_integrate_vel];
    for_morsel(store, ops_integ_vel, 0, mid, 2048);
    progress[0].store(mid, Release);

    // phase 1: fused trunk (head half)
    let ops_phys: &[WuFn] = &[wu_integ_pos, wu_spatial_hash, ...];
    for_morsel(store, ops_phys, 0, mid, 768);
    progress[1].store(mid, Release);

    // phase 2 (wait for phase 1 tail to catch up)
    while progress[1].load(Acquire) < mid { park_predictive(); }
    ...
}
```

**Predictive parking at sync points:**

At each sync point (phase boundary where this core waits for another),
the core knows approximately how long the wait will be. The scheduler
estimates it from the compute weight of the other core's remaining
work. If the wait is:
- < 100ns: spin (the other core is almost done)
- 100ns - 10µs: `spin_loop()` with backoff
- > 10µs: `thread::park()` with `unpark()` from the other core

The prediction is computed at plan time from morsel counts and
estimated per-morsel cost. It doesn't need to be precise — the
categories (spin/backoff/park) are coarse. Wrong prediction costs
at most one park/unpark cycle (~1µs).

**Why this is the endgame:**

Every prior optimisation is subsumed:
- Local `&[WuFn]` slices: baked into the per-core function
- Resource caching: emitted as stack-local snapshots per chain
- DSE + stores-at-end: the codegen flattener emits optimal loop body
- Phase pipelining: the sync points are compile-time constants
- Head+tail convergence: each core's row range is baked in
- Morsel sizing: compile-time constants from L1 / col_count
- No dynamic dispatch: the entire program is a monomorphised function

The runtime is just: spawn N threads, each runs its compiled program,
join when all complete. No scheduler loop. No work queues. No atomics
except at sync points. Pure execution.

### THEORY: write-only morsel sizing + read-as-resource

Within a phase, columns fall into two categories:
1. **Write columns**: written by some WU in this phase. These must
   be in the morsel's L1 working set — the WU writes to them per
   element, and the written data must be available for subsequent
   WUs in the same chain.
2. **Read-only columns**: written by a PREVIOUS phase, only read in
   this phase. These are immutable inputs. They don't need to be
   budgeted in the morsel's L1 working set.

**Current morsel sizing:** `morsel = L1 / (all_cols * sizeof(T))`
where all_cols = read + write columns. This overestimates the L1
pressure because read-only columns don't compete for write bandwidth.

**Proposed:** `morsel = L1 / (write_cols * sizeof(T))`. Only write
columns count for the morsel budget. Read-only columns are accessed
sequentially per element (the WU still does `cu(c(POS_X), i)`) but
the hardware prefetcher brings them into L1 as additional sequential
streams. M1 tracks 8-16 independent prefetch streams. The read-only
columns ride on the prefetcher — they're in L1 when needed but don't
evict write columns.

**Impact on the game world:**

Phase 1 fused trunk: 15 unique columns total.
- Write columns: ACC_X/Y/Z, VEL_X/Y/Z, POS_X/Y/Z, COLL_*, TERRAIN,
  MOISTURE, AI_STATE, ACTION, TARGET_* — roughly 17 writes
- Read-only (from phase 0): VEL_Y (read by gravity but then written
  by integ_vel — so it's actually write)

Hmm, in this particular phase most columns are read-modify-write.
The write-only optimization helps most when a phase reads from many
upstream columns but writes to few. Example: the render chain reads
POS, HEALTH, AI_STATE (6 read-only cols) but writes RENDER_*, VISIBLE
(6 write cols). With write-only sizing: morsel doubles from
`96K / (12 * 8) = 1000` to `96K / (6 * 8) = 2000`.

**The per-phase read-only detection:**

At plan time, for each phase:
1. Collect all columns written by any WU in this phase → write_set
2. Collect all columns read by any WU in this phase → read_set
3. read_only = read_set - write_set (columns read but never written)
4. Morsel size uses write_set.len() instead of (read_set ∪ write_set).len()

**The treatment of read-only columns:**

Read-only columns within a phase are analogous to resources:
- They're immutable for the phase's duration
- They can be loaded once and kept in cache/registers
- Multiple trunks can read them concurrently (no write conflicts)
- The prefetcher handles sequential access even if not L1-budgeted

For scalar-like read-only values (where the same value is reused
across many rows), the resource caching pattern applies: copy to
a stack-local before the morsel loop. For per-row read-only columns,
the prefetcher handles it — no special treatment needed beyond
not counting them in the morsel budget.

**What this enables:**

Larger morsels → fewer morsel transitions → less dispatch overhead.
For phases where read columns dominate (e.g., render phase reads 6,
writes 6), the morsel size doubles. For phases where most columns
are read-modify-write (e.g., physics), the improvement is smaller.

The detection is free — it's a set subtraction on the column sets
that the chain grouper already computes. No runtime cost.

### ASM INSPECTION: remaining codegen issues in the write-sized dispatch

**File:** `target/release/deps/hilavitkutin_bench-*.s`, function
`dispatch_write_sized` for the 6-WU fused trunk.

The function is fully devirtualised (0 `blr`). 221 instructions in
the inner loop, 19 stores, 38 loads. Four issues identified:

**1. Constant rematerialisation inside the loop.**

Hash constants like `0xFF51AFD7ED558CCD` are constructed from 4
`mov`+`movk` instructions EVERY iteration:
```asm
mov  x22, #36045
movk x22, #60757, lsl #16
movk x22, #45015, lsl #32
movk x22, #65361, lsl #48
```
This happens for 2 hash constants = 8 wasted instructions per element.
LLVM should hoist these outside the loop but can't because register
pressure is too high — all 30 GPRs are consumed by column pointers +
intermediates. The constants get spilled and rematerialised.

**Root cause:** the 6-WU fused trunk touches 17+ columns. Each column
needs a base pointer register. 17 column pointers + 4 hash constants +
loop counter + intermediates = 30+ registers needed. aarch64 has 31
GPRs (x0-x30). Register pressure forces spills.

**Potential fix:** the fusion is too aggressive. 6 WUs with 15+ unique
columns in one fused chain exceeds the register file. Splitting into
two chains of 3 WUs each (~8 columns) would fit in registers without
spills. The morsel sizing wouldn't change (same L1 budget), but the
codegen quality would improve dramatically — no constant remateriali-
sation, no stack spills for column pointers.

This connects to the column-count-driven phase strategy: the threshold
for chain splitting shouldn't just be L1 capacity but also register
file capacity. On aarch64: ~12-14 columns per chain before register
spills start. On x86-64: ~8-10 (fewer GPRs).

**2. Store-then-reload of intermediate column data.**

Position integration stores vel_x then immediately reloads it for the
collision hash:
```asm
str  x22, [x6, x16, lsl #3]   ; store vel_x
...
ldr  x22, [x6, x16, lsl #3]   ; reload vel_x (same value!)
```
LLVM doesn't eliminate this because it can't prove the store to one
column pointer doesn't alias with a read from another. This is the
same cu/cw aliasing issue from the Stats-Pipeline findings — the DSE
pattern (keep intermediates as locals, store only at the end) would
eliminate these reloads.

8 unnecessary loads per element in this function. At 1M elements:
8M wasted memory operations.

**3. AI state machine as cascaded compares (not jump table).**

The 6-state AI decision (`ai_state = 0..5`) compiles to:
```asm
cmp  x26, #1
b.le LBB_17
cmp  x26, #2
b.eq LBB_23
cmp  x26, #3
b.eq LBB_24
cmp  x26, #5
b.eq LBB_21
```
5 cascaded compares instead of a jump table. The branch predictor
handles this OK (skewed distribution), but a jump table would be
1 instruction regardless of state count.

**4. Column pointers spilled to stack at morsel boundaries.**

The function uses 304 bytes of stack. Column pointers are loaded
from stack (`ldr x, [sp, #N]`) at the start of each morsel. This
is the morsel boundary overhead — 17+ column pointer loads per
morsel. With smaller morsels, this overhead is proportionally larger.

**The register pressure problem is the key issue.** It causes both
#1 (constant rematerialisation) and #4 (stack spills). The fix:
don't fuse 6 WUs with 15+ columns into one chain. Split at the
register file boundary (~12 columns on aarch64). This is a chain
grouper decision, not a codegen issue — the grouper currently fuses
based on L1 capacity (93 columns) which is far above the register
file limit (12-14 columns).

**New chain splitting threshold:**

```
max_chain_cols = min(
    L1_usable / (MIN_MORSEL * sizeof(T)),  // L1 capacity limit
    REGISTER_COUNT - RESERVED_REGS,         // register file limit
)
```

On M1: min(93, ~14) = 14. The 15-column fused trunk exceeds this
by 1 column — enough to trigger spills.

### THEORY: signal-ordered execution + branch-free compiled dispatch

The virtual column/signal system (`ctx.on::<Signal>`) already captures
the branching structure of work units. Instead of `if alive { ... }`,
the user writes `ctx.on::<Alive>(|...| { ... })`. The scheduler knows
EXACTLY which entities have each signal fired — it's a bitset per
signal type, computed before execution.

**Row reordering by signal state:**

If entities are reordered so that all `Alive=true` entities come first
and all `Alive=false` entities come last, the `ctx.on::<Alive>` handler
runs on a contiguous range [0..n_alive) and the else handler runs on
[n_alive..total). No per-element branch. The compiled program emits
two specialised loop bodies and switches at the boundary.

The row reordering is a full permutation of all column data. In an
ECS, row ordering has no semantic meaning — entity 0 doesn't need to
be at row 0. The scheduler can reorder freely as long as all columns
are permuted consistently.

**What the compiled program looks like:**

```rust
// without signal ordering: branch per element
for i in 0..rows {
    let alive = cu(c(ALIVE), i);
    if alive != 0 {
        // alive path: AI, movement, gathering
        wu_ai_decide(s, i);
        wu_move_to_target(s, i);
    } else {
        // dead path: skip AI, just render as corpse
        wu_render_corpse(s, i);
    }
}

// with signal ordering: branch-free
let alive_end = signal_boundary[Alive]; // pre-computed at plan time
// alive path (contiguous range, no branch)
for i in 0..alive_end {
    wu_ai_decide(s, i);
    wu_move_to_target(s, i);
}
// dead path (contiguous range, no branch)
for i in alive_end..rows {
    wu_render_corpse(s, i);
}
```

The inner loops have no branches. LLVM can vectorise them freely
(no branch misprediction, no predicated execution needed). Each
loop body is a pure sequential walk through contiguous data.

**Why this works without changing the user API:**

The user already writes `ctx.on::<Signal>` — this IS the declared
branch structure. The scheduler just:
1. Scans signal bitsets before the phase
2. Computes the row permutation that groups same-signal entities
3. Permutes all column data (one-time cost per phase)
4. Emits specialised loop bodies per signal combination
5. Records the signal boundaries for the compiled program

The user never sees the permutation — they write per-entity logic
via `ctx.on`, and the scheduler handles the rest.

**Signal ordering vs virtual column writes:**

The ordering is by signal READ — which entities have the signal
active. The signal WRITE (firing a signal) changes the bitset for
the next phase. Within a phase, the signal state is immutable
(same as read-only columns). So the ordering is stable for the
entire phase's execution.

Between phases, signals may change (a WU in phase N fires a signal
that phase N+1 reads). The row permutation is recomputed at each
phase transition where signal state changes. This is part of the
phase transition cost — alongside the read-only column detection.

**Multiple signals → multi-key sort:**

If a phase has two signal-gated WUs (`ctx.on::<Alive>` and
`ctx.on::<Colliding>`), the sort key is the combination:
- Alive + Colliding: rows 0..a
- Alive + !Colliding: rows a..b
- !Alive + Colliding: rows b..c
- !Alive + !Colliding: rows c..n

Four contiguous ranges, four specialised loop bodies, zero branches.
The number of ranges is 2^(number of signals in the phase). For
2-3 signals this is 4-8 ranges — manageable. For more signals,
the scheduler picks the most discriminating signals for ordering
and falls back to per-element branching for the rest.

**Bool/flag columns without explicit signals:**

Some columns are de facto signals: `collision_flag` is 0 or 1,
`action` is an enum 0-5. These can be auto-detected at plan time
(the column's value domain is small and discrete). The scheduler
treats them as implicit signals and applies the same ordering.

Detection: at data load time, scan columns for low cardinality.
If a column has ≤8 unique values across all rows, treat it as an
orderable signal. The scan is O(N) per column, amortised over the
pipeline's lifetime.

**Cost of row permutation:**

Permuting N rows across C columns: O(N × C) memory ops. For 1M rows
× 36 columns: 36M u64 writes = ~288MB. At 100GB/s M1 bandwidth:
~3ms. This is a one-time cost per phase transition where signal
state changes. For the game world with 4 phases, worst case: 12ms
of permutation overhead. This must be recouped by the branch-free
execution savings.

For the game world's AI WU (heavy branching: 6 states), removing
per-element branching on 1M elements saves approximately:
- 6 branch mispredictions × ~10ns penalty × 1M / 6 = ~10ms
The 3ms permutation cost is recouped.

For simple bool branches (alive/dead), the misprediction rate is
lower (branch predictor learns the dominant path). The savings are
smaller but the permutation cost is also smaller (only the signal
column matters, not all columns — the permutation can be lazy,
only reordering the columns actually accessed in the signal-gated
WU).

#### Shared read columns between trunks — two approaches

Trunks within a phase may share read-only columns (e.g., trunks B1
and B2 both read `pos_x/y/z` from phase A's output). Two approaches
to handle this:

**Approach 1: snapshot-to-local (simpler, higher cost at scale).**

At phase transition, each trunk copies the shared read columns into
its own local buffer. The trunk reads from its local copy. Zero
cross-core cache contention — each core works on private data.

Same principle as resource caching: if data is read-only for a scope,
copy it locally. LLVM can reason about aliasing on local data, and
the CPU never contends on shared cache lines.

Cost: one memcpy per shared column per trunk at phase start.
For pos_x/y/z at 1M rows: 3 cols × 8 bytes × 1M = 24MB copy per
trunk. At ~100GB/s M1 bandwidth: ~0.24ms. Phase B takes ~15ms at
1M — so copy overhead is ~1.6%.

The co-located arena extends: each trunk's arena contains ALL its
columns — both its private write columns and its snapshotted read
columns. The morsel window covers rows × all columns, physically
contiguous, all in one core's L1. Perfect spatial locality.

This approach is simpler because:
- No synchronisation between trunks during execution
- Each trunk owns its data entirely — no shared pointers
- The copy cost is predictable and bounded
- The memory layout is optimal per trunk

But it may be worse than approach 2 at large scale because:
- The copy duplicates data that's already in the arena
- At very large N, the copy itself becomes significant
- If many trunks share many columns, the total copy volume grows

**Approach 2: aligned morsel synchronisation (zero copy, more complex).**

Instead of copying shared columns, both trunks read from the SAME
column data. The morsels are aligned: trunk B1's morsel [0..1024)
and trunk B2's morsel [0..1024) access the same rows at the same
time. Because both are sequential reads, the hardware prefetcher
on each core brings in the same cache lines simultaneously.

On M1 (unified L2, shared memory): this is nearly free. One core's
prefetch warms the cache for the other core. No copying needed.

On x86 NUMA: cache lines bounce between L1 caches of different cores.
The cost depends on the NUMA distance. Same-socket cores share L2/L3
and have cheap sharing. Cross-socket cores have expensive coherence.

This approach is more complex because:
- Morsel boundaries must be synchronised across trunks (not just
  progress counters but actual morsel start/end alignment)
- If trunks have different morsel sizes (different column counts),
  aligning them requires choosing the smallest morsel size for all
  trunks that share columns, sacrificing per-trunk optimal morsel sizing
- The execution becomes coupled: one trunk can't advance past the
  other's morsel boundary without risking cache eviction of shared data

#### Column-count-driven phase strategy

The column count of a fused phase determines whether to keep it
as one trunk (fuse aggressively) or split into parallel trunks:

```
fused_cols = union of all WU columns in the phase (deduplicated)
max_cols = L1_usable / (MIN_MORSEL * sizeof(T))

if fused_cols < max_cols * 0.5:
    → FUSE: single trunk, one data walk, large morsels.
    Plenty of L1 headroom. Morsel is large → low dispatch overhead.
    Threading adds no value (one trunk = one serial path).

if fused_cols >= max_cols * 0.5 AND fused_cols < max_cols:
    → CONSIDER SPLIT: fusing works but morsels are getting small.
    Check if a balanced split into 2+ trunks exists where each trunk
    has ~fused_cols/2 unique columns. If yes AND row_count > 10K
    AND cores > 1: split. Otherwise: fuse.

if fused_cols >= max_cols:
    → MUST SPLIT: morsel would be too small fused (< MIN_MORSEL rows).
    Find the minimum number of trunks such that each trunk's column
    count < max_cols. Each trunk gets viable morsel sizing.
```

**Game world Phase 1:** 25 unique columns, max_cols ~93 on M1.
25 < 46 (= 93 * 0.5). Fuse. One trunk, one walk. This matches the
benchmark: the fused approach (0.93x seq) is already nearly optimal
for this phase because the column set fits easily in L1.

**When splitting helps:** a phase with 60 unique columns across
10 WUs. Fused morsel: 96KB / (60 × 8) = 200 rows (tiny, lots of
dispatch overhead). Split into two 30-column trunks: 400 rows per
morsel + two cores = better morsel sizing AND parallelism. The
morsel sizing improvement alone justifies the split, even without
threading.

**The column count threshold is the same trade-off as morsel sizing:**
- Below 50% max_cols: locality dominates (fuse, one walk)
- Above 50% max_cols: morsel sizing degrades (split for better morsels)
- Above max_cols: forced split (can't fit in L1 at all)

**Recommendation:** benchmark both. Try approach 1 first (simpler,
predictable) to establish whether the phased decomposition helps at
all. Then try approach 2 to see if avoiding the copy is worthwhile.
If the copy overhead is <5% of phase time, approach 1 wins on
simplicity. If the copy is >10%, approach 2 may be needed.

The choice may also be platform-dependent: approach 1 for NUMA x86
(copies avoid cross-socket coherence), approach 2 for unified-memory
ARM (sharing is cheap).

#### What WUs need to provide (unchanged)

Same as before: read set, write set, order. No additional metadata
needed. The phased decomposition is a scheduler-side analysis that
works entirely from these three declarations.
