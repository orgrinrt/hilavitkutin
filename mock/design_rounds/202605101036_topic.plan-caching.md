**Date:** 2026-05-11
**Phase:** TOPIC, closed (autonomous trivial calls + one bench-gated value).
**Scope:** when does plan recompute fire, `DependencyGraph` backing (dense vs CSR), dirty-detection runtime.
**Source topics:** Topic 1 (axis 4 auto-recompute on dirty), Topic 8 (PlanAffecting marker).

## Decisions locked

### Axis A. Plan recompute trigger

`Scheduler::run()` (Topic 1 axis 4) checks the dirty bit on entry. If set: recompute plan via the Topic 3 step chain, clear dirty bit, dispatch. If clear: dispatch directly from cached plan.

Dirty bit setters:
1. `replace_resource<R: PlanAffecting>` (Topic 8 axis B).
2. Consumer-explicit `Scheduler::dirty_plan()` API (for advanced use, e.g., debug rebuild).
3. WorkUnit registration changes via builder (only valid pre-build, but mockspace lint catches builder-after-build attempts).

### Axis B. `DependencyGraph` backing — CSR vs dense — BENCH-LOCKED CSR

**Locked: CSR canonical at all N, no threshold.** Bench result from `mock/benches/dep_graph_csr_9` (2026-05-11):

| N nodes | CSR ns/op | Dense ns/op | Dense slowdown vs CSR |
|---|---|---|---|
| 8 | 146 | 724 | +397% (5x slower) |
| 16 | 206 | 1239 | +502% (6x slower) |
| 32 | 302 | 2911 | +862% (10x slower) |
| 64 | 645 | ~12500 | +1840% (20x slower) |
| 128 | 1225 | ~41300 | +3270% (34x slower) |
| 256 | 2546 | ~151000 | +5830% (60x slower) |

**Predicted crossover N=16 is wrong.** CSR wins even at N=8 because dense O(N²) cell-scan beats CSR's edge-direct iteration's loop overhead only when N is so small the matrix fits trivially in registers, which is essentially never. At realistic DAG densities (3-6 edges/node), CSR's E iterations always beat dense's N² scans.

Hybrid variant (dense for N≤16, CSR else) measured within noise of pure CSR at N≥32, slightly faster at N=128 (-12%) and N=256 (-22%) due to cache effects on the unused col_indices allocation, but pure CSR is the canonical shape for simplicity. The hybrid variant is dropped.

CSR shape: three columnar arrays at the engine level — `row_offsets: [USize; MAX_UNITS + 1]`, `col_indices: [UnitId; MAX_EDGES]`, `edge_kinds: [EdgeKind; MAX_EDGES]` (read/write/control). Aligned to arvo-sparse conventions (which already provides CSR primitives the engine consumes via the arvo-sparse → engine boundary).

Findings audit trail: `mock/benches/dep_graph_csr_9_n*_findings.md` (6 files, N=8/16/32/64/128/256).

### Axis C. Dirty-detection granularity

Per-`PlanAffecting`-resource. The dirty bit array (`[AtomicBool; MAX_PLAN_AFFECTING_RESOURCES]` from Topic 8) IS the dirty-detection runtime. No finer granularity (per-column, per-field, etc.); too much bookkeeping for too little benefit. If a resource changes, we treat the whole plan as potentially affected.

Consumers who want finer granularity can split a fat PlanAffecting resource into multiple smaller ones; each gets its own dirty bit.

### Axis D. Plan-cache eviction

The plan is cached forever as long as no dirty bit fires. There is no eviction; the cache is just `ExecutionPlan<'frame, ...>` held by the Scheduler. Scheduler lives for the pipeline's lifetime; cache lives with it.

If consumer wants to drop the cache (e.g., low-memory mode), they call `Scheduler::dirty_plan()` to force recompute. No explicit eviction API; the dirty bit IS the eviction signal.

### Axis E. Multi-version cache

Not in v1. A single cached plan. Consumers who need multiple plans (e.g., different shapes per input class) build multiple Schedulers.

### Axis F. Plan recompute parallelism

Plan recompute is single-threaded (runs on the scheduler-owning thread before dispatch). The Topic 3 chain is sequential; some steps (RCM, spectral) could parallelise but the costs are <100µs at realistic N, not worth the coordination overhead. Sequential v1.

If a consumer's pipeline is structurally massive (10K+ WUs), revisit. Realistic ceiling per `MAX_UNITS` const generic is ~256.

## User-attention call (bench-gated)

### CSR threshold N

Pending bench. The bench tests CSR vs dense `DependencyGraph::add_edge` + `iter_successors(u)` at N = 8, 16, 32, 64, 128, 256 with edge density ~2-6 per node. Crossover N is what the bench finds; lock the value into engine code as `CSR_THRESHOLD_N`. Until bench lands, CSR is the working default for all N.
