# Hilavitkutin test rehab post-arvo-Round-D

Several hilavitkutin tests fail to compile against current arvo dev:

- `USize::ZERO` resolves to the `Identity` trait associated const.
  Tests that don't import `Identity` see "no associated item named
  ZERO". Fix is one new import per affected test file.
- `assert!(b)` where `b: arvo::Bool` fails because `Bool` doesn't
  impl `Not`. The canonical pattern (per arvo's own test code, see
  `arvo-storage/tests/nusize_const_arith.rs`) is `assert!(b.0)` and
  `assert!(!x.0)`. Tests using the `Bool` directly need the `.0`
  field access.
- `assert_eq!(maybe_returning_fn(), Some(...))` where the function
  returns `notko::Maybe<T>` fails the type check. Replace with
  `Maybe::Is(...)` / `Maybe::Isnt`.
- Integer literals at sites expecting `USize` need wrapping in
  `USize(N)` (the explicit constructor).

These are mechanical migrations against the current substrate
contract. No design questions; the patterns are already in active
use in arvo's own test suites.

## Affected files

`cargo test --workspace --no-run` fails on these test crates with
the symptoms above:

- `mock/crates/hilavitkutin-persistence/tests/sieve.rs` — 1 site,
  `USize::ZERO`.
- `mock/crates/hilavitkutin-persistence/tests/cold_store.rs` — 1
  site, `USize::ZERO`.
- `mock/crates/hilavitkutin-persistence/tests/manifest.rs` — 5
  sites, `USize::ZERO`.
- `mock/crates/hilavitkutin-persistence/tests/archive_str.rs` — 8
  sites: `assert!(<Bool>)` (3 sites), `USize::ZERO` (3 sites),
  `Some(...)` for `Maybe<&str>` return (2 sites).
- `mock/crates/hilavitkutin/tests/plan_types.rs` — 32 sites:
  `assert!(<Bool>)` (12 sites), integer-literal-where-USize-expected
  (12 sites), method-argument-type mismatches downstream (8 sites,
  same root cause once the Bool / USize sites land).

## Decisions

### D1. Use `Identity::ZERO` (trait-disambiguated path) for USize::ZERO sites

The cleanest test-side fix is `use arvo::Identity;` at the top of
each affected file. With `Identity` in scope, `USize::ZERO` resolves
to the trait's associated const. The alternative (`USize(0)`) works
but loses the typed-const intent that the tests are explicitly
exercising. Match arvo's own test convention: prefer `Identity`.

### D2. `.0` field access for `assert!(<Bool>)` sites

Per arvo's own test suites (`nusize_const_arith.rs`, `bits_const_arith.rs`).
Inline form: `assert!(b.0)`, `assert!(!x.0)`. No alternative pattern
ships in arvo today.

### D3. `Maybe::Is(...)` for `Some(...)` at Maybe-typed sites

`notko::Maybe::Is` / `notko::Maybe::Isnt` are the canonical
constructors. The `Maybe` import is already present in every
affected test; the substitution is purely textual.

### D4. `USize(N)` wrapping for integer literals at USize-typed sites

The explicit constructor preserves intent and reads cleanly. No
change to the bare-numeric lints because test code paths already
permit literals; the wrap is a type-coercion fix, not a lint
question.

### D5. No production-source changes in this round

The fixes are test-only. No `mock/crates/*/src/**.rs` edits needed.
DESIGN.md.tmpl and BACKLOG.md.tmpl stay untouched. The doc CL is
empty-of-template-edits but still ships to step the phase machine.
