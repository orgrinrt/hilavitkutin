# Purpose

With the ten "pending-sweep" lints now at `error` across commit / build / push (severity flip applied in `mock/mockspace.toml` by the human overseer per workspace rule), lint scan reports 107 error occurrences (~60 unique source lines, several with multi-lint overlap). All are in `hilavitkutin-extensions`, the crate whose sole role is the C-ABI descriptor layer between the host and dlopen'd extensions.

Classification of the 60 unique sites:

- **~55 FFI-boundary**: `#[repr(C)]` descriptor struct fields, `#[repr(u32)]` enum discriminants, FNV-1a `u64` hash constants, `&[u8]` byte-string parameters for the dlopen path and the extension name slice, `Option<unsafe extern "C" fn()>` nullable function pointers whose None representation is specifically the null-pointer niche (wrapping in `notko::Maybe` would break ABI), and the `&str` parameter on `CapabilityId::from_name` (const-fn input type). These are load-bearing. Each gets a trailing `// lint:allow(<name>) reason: FFI descriptor boundary; tracked: #206` annotation referring to the FFI descriptor boundary allow policy task.
- **~5 test sites in `src/lib.rs` `mod tests`**: layout assertions that validate `size_of::<ExtensionVersion>() == 8`, `ExtensionAbiStatus` discriminant values, and related invariants. They necessarily reference raw primitives because they verify the ABI layout itself. Same `#206` annotation.
- **1 genuine fix** in `src/host.rs`: `ExtensionHost::has_capability(&self, id: CapabilityId) -> bool` returns a bare `bool`. The correct return type is `arvo_bits::Bool`. No FFI justification; this is internal logic.

# Deliverables

## S1. Genuine fix: `ExtensionHost::has_capability`

Change the return type from `bool` to `arvo_bits::Bool`. Add `arvo-bits` to `hilavitkutin-extensions`' `Cargo.toml` `[dependencies]` (workspace inherit). Update the one caller site inside `load` that compares the return value to `true`. Update the test `host_advertises_declared_capabilities` to assert against `Bool::TRUE` / `Bool::FALSE` or to take `.is()` on the wrapper.

## S2. FFI-boundary `lint:allow` annotations

Add trailing `// lint:allow(<lint>) reason: ...; tracked: #206` comments on the following lines. The `<lint>` list contains every lint that fires on that line so a single comment covers overlapping reports.

### `src/descriptor.rs`

- Line 15 `pub const HOST_ABI_VERSION: u32 = 1;` — `no-bare-numeric`, `arvo-types-only`. Reason: C-ABI version counter; `u32` width is contract-fixed.
- Line 21 `pub const DESCRIPTOR_SYMBOL: &[u8] = b"..."` — `no-bare-numeric`, `arvo-types-only`. Reason: null-terminated byte string matching the linker-level symbol; bytes are the FFI unit.
- Line 29 `pub struct CapabilityId(pub u64);` — `no-bare-numeric`, `arvo-types-only`, `no-public-raw-field`. Reason: `#[repr(transparent)]` over `u64` is the FNV-1a hash width + ABI contract; no-public-raw-field yields because the tuple field IS the ABI shape.
- Line 36 `pub const fn from_name(name: &str) -> Self` — `no-bare-string`. Reason: const-fn byte-string input; `Str` has no const constructor usable at the macro-expansion time that `CapabilityId::from_name` is invoked.
- Lines 37-43 FNV-1a constants + loop — `no-bare-numeric`, `arvo-types-only`. Reason: FNV-1a algorithm constants are fixed-width; the step casts are the algorithm definition.
- Lines 58-61 `ExtensionVersion { major: u16, ... }` — `no-bare-numeric`, `arvo-types-only`, `no-public-raw-field`. Reason: `#[repr(C)]` ABI layout.
- Line 69 `pub enum ExtensionAbiStatus { Ok = 0, ... }` — `no-bare-numeric`, `arvo-types-only`. Reason: `#[repr(u32)]` discriminants are the C-ABI return value.
- Lines 102-109 `ExtensionDescriptor` fields (`abi_version: u32`, `name_len: usize`, `capabilities_len: usize`, `required_host_caps_len: usize`, function-pointer slots) — `no-bare-numeric`, `arvo-types-only`, `no-public-raw-field`. Reason: `#[repr(C)]` ABI layout.
- Lines 110, 113 `Option<unsafe extern "C" fn(...) -> ExtensionAbiStatus>` — `no-bare-option`. Reason: `Option<fn>` is the Rust idiom for nullable function pointers. The `None` variant has the null-pointer-niche layout guaranteed by the language; `notko::Maybe<fn>` has no such guarantee and would break the C-ABI.

### `src/error.rs`
