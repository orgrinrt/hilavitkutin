# Scheduler::run() signature

This is the foundational design decision for hilavitkutin runtime task B1 (#334) within the larger runtime megaround (B1 + C1–C7). Every other axis (plan algorithm chain, dispatch codegen, adapt subsystem, thread pool, morsel loop, resource resolution, plan caching) hangs on what shape `Scheduler::run()` takes.

This topic enumerates the locked-decisions that constrain the design space, the open axes the user must decide, and records each decision as it lands.

## Locked decisions (from prior rounds; cannot relitigate)

These constrain the design space and must be honoured by any signature this topic produces.

### A1. run() returns PipelineResult only; no result extraction

Locked at consolidation topic round 202604200055 / domain 23, lines 2165-2167. Rule: "Pipeline::run() returns PipelineResult status only. No result extraction from Pipeline::run() — consumer reads ColumnStorage directly." `PipelineResult` is shipped at `mock/crates/hilavitkutin/src/scheduler/result.rs` with three variants: `Completed`, `Failed`, `Poisoned`.

### A2. Error model: WU::execute returns (); errors flow through data

Locked, same source, lines 2147-2167 + DESIGN.md.tmpl lines 330-336. Rule: "execute() on WorkUnit returns (). Abort on failure, no unwind tables. Errors surface through data flow: error columns or error virtuals. The scheduler detects failure via version stamps or error virtuals, then poisons dependent fibers. Independent fibers continue."

### A3. Frame model: consumer-owned loop; scheduler provides one pass

Locked in consolidation topic / domain 21, lines 1984-1991, and CLAUDE.md. Rule: "Consumer reads metrics via `On<meta::ScheduleEnd>` WU and implements their own cadence logic. hilavitkutin provides measurement data, consumer decides policy." `scheduler.run()` is a single-pass call. Engine never loops itself.

### A4. Adapt fires between frames only, not during execution

Locked at `adapt/mod.rs` lines 6-10 + DESIGN.md.tmpl lines 143-148. Rule: "The executor-reorganisation triggers live here; per DESIGN they fire between frames (not during execution, which would invalidate monomorphised dispatch)."

### A5. Plan reused across frames; recompute only on structure change

Locked at DESIGN.md.tmpl lines 54-55 and CLAUDE.md. Rule: "Schedule once, reuse across frames. Plan recompute is rare (triggered by structure change). Microsecond cost on const-generic bitmask structures."

### A6. Self-hosting meta pipeline

Locked at consolidation topic lines 2136-2137 and DESIGN.md.tmpl line 325-327. Rule: four meta virtuals (`PlanStage`, `ScheduleReady`, `PassStart`, `ScheduleEnd`) and four meta resources (`Dag`, `ExecutionPlan`, `LaneAssignment`, `SchedulerMetrics`). Irreducible kernel ~50 lines: "fire PlanStage, wait for plan WUs, fire ScheduleReady, dispatch consumer WUs, fire ScheduleEnd."

### A7. replace_resource happens between build and run

Locked at DESIGN.md.tmpl lines 320-323. Rule: "replace_resource is app-only. Called between `.build()` and `scheduler.run()`."

### A8. Resource&lt;T&gt; is a PhantomData stub until C6

Per `mock/research/notes/202605091700_src_apply_decisions.md` D1. Rule: `Resource<T>` is a type-marker stub; `Resource::new(value)` drops the value at the constructor. The runtime data plane is explicitly built in C6.

### A9. Scheduler post-build is a ZST (no type params)

Per source state at `scheduler/mod.rs`. After round 202605091700, `Scheduler` carries no type-level WU/store identity at runtime (intentional erasure). run() must work against opaque Scheduler.

## Open axes for this topic

The signature has multiple independent axes. Each is decided in turn.

### Axis 1: input shape (what does run() take?)

**DECIDED.** `run()` is parameterless. `RunCfg` is a substrate-provided Provider: the substrate ships `impl Provider for RunCfg` so consumers pass it bare via the `.with()` machinery. No `Resource::new(...)` wrap; the substrate handles the storage shape under the hood. No special configure() method, no special-cased builder API.

The wrap-or-bare rule (per the validated builder-provider sketch at `mock/research/sketches/202605091700-builder-provider-shape/sketch.rs` lines 43-47):

1. Generic provider types (`Resource<T>`, `Column<T>`, `Virtual<T>`, `LinkedBin<T>`) keep their wrapper because T parameterises the slot identity. `.with(Resource::new(GameState::default()))`, `.with(Column::<FileInfo>::new())`.
2. Non-generic provider types (WUs, Kits, platform impls, RunCfg, and any other substrate-special framework type) impl `Provider` directly and pass bare. `.with(MyRunner)`, `.with(InputKit)`, `.with(MyClock)`, `.with(RunCfg::default())`, `.with(RunCfg { record_count: USize(2048) })`.

Concrete shape:

```rust
// builder path. RunCfg auto-inserts via Default impl + HasTrivialCtor;
// consumer never has to mention it.
let mut sched = Scheduler::builder()
    .with(Resource::new(Interner))     // generic provider, wrapped
    .with(Column::<Record>::new())     // generic provider, wrapped
    .with(MyRunner)                    // WU, bare
    .with(MyKit)                       // Kit, bare
    .build();   // RunCfg::default() implicitly registered

// frame loop. consumer-owned. parameterless run() per pass.
loop {
    match sched.run() {
        PipelineResult::Completed => { /* consumer reads columns */ }
        PipelineResult::Failed | PipelineResult::Poisoned => break,
    }
}

// to shape the run differently, the consumer passes RunCfg bare via
// .with(), OR calls scheduler.replace_resource(RunCfg { ... }) between
// frames per A7.
let cfg = RunCfg { record_count: USize(1024) /* future fields */ };
let mut sched = Scheduler::builder()
    .with(cfg)                         // RunCfg, bare. No wrapper.
    .with(MyRunner)
    .build();

// later, between frames:
sched.replace_resource(RunCfg { record_count: USize(2048) });
```

Properties:

1. No special-casing in the API surface. RunCfg uses the same `.with()` route as every other provider; the special-ness lives entirely in the substrate's Provider impl for RunCfg, hidden from consumers.
2. WUs that need `record_count` read it via whatever accessor pattern the substrate exposes for RunCfg. Specific accessor shape (e.g. `&Resource<RunCfg>` in Ctx vs a substrate-specific helper) is decided in axis 4 (resource resolution) and the doc CL.
3. Dirty-detection for free. RunCfg participates in the plan-cache invariants like any other provider. Replacing or otherwise mutating it dirties the plan if the change implies a structure shift (record_count change, etc). Exact dirty trigger semantics in axis 4.
4. Default ergonomics. `RunCfg: Default + HasTrivialCtor` (or equivalent) lets the builder auto-insert when the consumer didn't explicitly configure. The consumer who needs to override hands a constructed RunCfg directly to `.with()`, no new API surface.
5. Consumer-extensible. Consumers can ship their own richer config types as siblings (via Resource<MyCustomCfg> wrapper or their own Provider impl, depending on shape) if they need knobs the substrate doesn't provide.
6. Replaceable. RunCfg implements `Replaceable` so `scheduler.replace_resource(RunCfg { ... })` is the between-frames update path per A7.

Open follow-up notes (deferred to other axes or to the doc CL execution):

1. Whether RunCfg fields are pub or accessor-only (RunCfg API surface lands in the doc CL).
2. Whether the auto-insert-default behaviour requires a new builder method or falls out of `Default + HasTrivialCtor` blanket logic.
3. Concrete RunCfg field set v1: at minimum `record_count: USize`. Future-extensible (frame_id, budget hints, debug flags) but minimal v1.

### Axis 2: return shape

**DECIDED.** `Scheduler::run()` returns `notko::Outcome<T, E>` whose concrete `T` and `E` are determined by the consumer's `RunCfg` type, propagated through the builder typestate via the existing dispatch-routing mechanism. The substrate's old `PipelineResult { Completed | Failed | Poisoned }` enum is retired; the result type dogfoods notko's foundation primitive.

#### Mechanism: new dispatch variant

A fifth dispatch variant joins the existing four:

```rust
pub struct RunCfgDispatch<C>(PhantomData<C>);  // bare PhantomData, like StoreDispatch / UnitDispatch
```

Consumers pick which type is "the run cfg" by setting `BuilderInput::Dispatch = RunCfgDispatch<Self>`. This is the same `.with()`-routed disambiguator the builder already uses for Resource / Column / Virtual / WorkUnit / Kit / platform impls. No `.set_runcfg_type::<C>()` separate API; no turbofish on `run()`.

#### The RunCfg trait

```rust
pub trait RunCfg:
    BuilderInput<Dispatch = RunCfgDispatch<Self>>
    + HasRecordCount   // required-field accessor traits as supertrait bounds
    // + HasFrameId    // future fields added here
    + Sized + 'static
{
    type Out;
}
```

Field requirements live as supertrait bounds on `RunCfg` (Shape B per the topic conversation), matching the existing Resource / WorkUnit / Kit pattern: each sub-trait declares its requirements as supertrait bounds; the `Dispatch` struct stays a bare `PhantomData<C>`. Compile-error on missing field-trait points at the `impl RunCfg for ...` line, which reads cleanly to a consumer ("the trait I'm implementing requires `HasRecordCount`").

#### Substrate ships a default

```rust
pub struct DefaultRunCfg { pub record_count: USize }

impl HasRecordCount for DefaultRunCfg { fn record_count(&self) -> USize { self.record_count } }
impl BuilderInput for DefaultRunCfg { type Dispatch = RunCfgDispatch<Self>; }
impl RunCfg for DefaultRunCfg { type Out = Outcome<(), ()>; }
```

If the consumer does not register a RunCfg type via `.with()`, the builder auto-inserts `DefaultRunCfg::default()` via `HasTrivialCtor` (mechanism inherited from axis 1). Simple consumers (viola CLI, vehje builds) get `Outcome<(), ()>` for free. The success variant carries no payload because consumers read run results via sidecar Resources (Topic 1 axis 1 lock); the error variant is `()` for "I failed; no structured reason", with heavy consumers overriding to a structured Err type. **Audit m4 fix:** earlier draft used `Outcome<Bool, ()>` which smuggled a Bool as success-vs-success-with-flag, design rot from the retired `PipelineResult` enum.

#### Heavy-consumer override

```rust
struct LoimuCfg { record_count: USize, /* ... */ }

impl HasRecordCount for LoimuCfg { /* ... */ }
impl BuilderInput for LoimuCfg { type Dispatch = RunCfgDispatch<Self>; }
impl RunCfg for LoimuCfg { type Out = Outcome<LoimuRunSummary, LoimuFailure>; }
```

Used at call site exactly like any other registered type:

```rust
let mut sched = Scheduler::builder()
    .with(LoimuCfg::default())  // <-- normal .with(), Dispatch picks the slot
    .with(MyRunner)
    .with(MyKit)
    .build();  // -> Scheduler<LoimuCfg>

match sched.run() {  // -> LoimuCfg::Out = Outcome<LoimuRunSummary, LoimuFailure>
    Outcome::Ok(summary) => bus.emit(ScheduleEnd::from(summary)),
    Outcome::Err(failure) => bus.emit(BehaviorFaulted::from(failure)),
}
```

#### Single-slot enforcement

The builder typestate has one named slot for the run-cfg type. First `.with(SomeCfg)` where `SomeCfg: RunCfg` transitions `RunCfgSlot::Empty` → `RunCfgSlot::Filled<SomeCfg>`. A second `.with(OtherCfg)` where `OtherCfg: RunCfg` attempts `Filled<SomeCfg>` → `Filled<OtherCfg>`, which the typestate rejects: compile error referencing the slot. "What if many" is impossible by construction; the consumer literally cannot register two RunCfg types.

#### Storage / WU read-side

`RunCfgDispatch<C>` internally registers `C` into the resource store too, so any WorkUnit can read the run cfg via the standard `ctx.resource::<TheRunCfg>()` accessor. The dispatch variant is "store + runcfg-slot" in one. Consumers do NOT separately impl `Resource` for their RunCfg; the `RunCfg` trait subsumes that storage role through the dispatch.

**Multi-responsibility coupling acknowledged.** `RunCfgDispatch<C>` deliberately carries three responsibilities on one type-level handle:
1. Builder slot population (typestate slot for "the run cfg type").
2. Resource-store registration (so WUs can read it via the standard accessor).
3. Type-parameter propagation through to `Scheduler<C>::run() -> C::Out`.

These responsibilities are coupled by design, not accidentally. They all fire from the same consumer call (`.with(MyRunCfg)`); separating them would force three separate registrations or a bespoke `.set_runcfg_type::<C>()` API that the round explicitly rejected. The dispatch-routing-as-disambiguator pattern's whole purpose is to fold all three onto one `BuilderInput::Dispatch` declaration. Pay for the coupling at design-tax review time; refuse to pay it at consumer call sites.

#### Scheduler type after build

A9 ("post-build Scheduler is ZST, no type params") relaxes slightly: one phantom type parameter carries the run-cfg handle. Storage stays effectively zero (PhantomData); only type identity widens.

```rust
pub struct Scheduler<Cfg: RunCfg> {
    _cfg: PhantomData<Cfg>,
    /* ... ZST internals ... */
}

impl<Cfg: RunCfg> Scheduler<Cfg> {
    pub fn run(&mut self) -> Cfg::Out;
}
```

#### Why this shape

1. **Dogfooding.** The substrate mandates `notko::Outcome` everywhere else; the runtime return is no exception. Removing the parallel `PipelineResult` enum removes a primitive that competed with the workspace primitive.
2. **Single-API uniformity.** Disambiguation flows through the existing dispatch-routing mechanism. Same `.with()` machinery for everything; `BuilderInput::Dispatch` is the one switch the consumer touches.
3. **Heavy-vs-simple.** Simple consumers get `Outcome<(), ()>` automatically (DefaultRunCfg auto-inserts). Heavy consumers parameterise to rich `Outcome<RunSummary, RichFailure>`. Hilavitkutin imposes no fault model.
4. **Forward-compatible.** Adding fields to a consumer's RunCfg or extending its Err enum doesn't break hilavitkutin. New required-field traits (`HasFrameId`, `HasBudgetHints`) can be added as supertrait bounds on `RunCfg` later without touching the dispatch machinery.

#### Open follow-ups (deferred to sketch + doc CL)

1. **Sketch needed.** Validate the typestate-slot machinery: builder must (a) detect that a `.with(T)` registration's `T::Dispatch = RunCfgDispatch<T>` qualifies for the run-cfg slot, (b) transition the slot from `Empty` to `Filled<T>` exactly once, (c) reject a second runcfg `.with()` with a coherent compile error, (d) carry `T` through to `Scheduler<T>` post-build. This is the kind of trait-solver-and-typestate question that needs an empirical compile before doc CL locks. Sketch lives at `mock/research/sketches/202605101036-runcfg-dispatch-slot/`. Per cl-claim-sketch-discipline.md.
2. **Field set v1.** At minimum `HasRecordCount` (record_count drives morsel sizing). `HasFrameId`, `HasBudgetHints` and similar added when downstream WUs prove they need them. Start small.
3. **Notko derive macro.** Separate notko-side task: ergonomic `#[derive(Outcome)]` (or attribute macro) for writing custom Outcome-shaped types — analogous to `thiserror` for stdlib Result. Tracked in this round's BACKLOG; doesn't block this axis's lock.
4. **Compile-error wording for double-runcfg-registration.** The slot's typestate transition error should ideally produce a `do_not_recommend` / on_unimplemented annotation that explains "you tried to register a second RunCfg type; the first was X." Polish, not blocking.

### Axis 3: receiver

**DECIDED.** `&mut self`. The receiver is the obvious choice given the locked decisions:

1. A5 (plan reused across frames; recompute only on structure change) implies the plan cache mutates inside run() when dirty. Recompute writes to internal plan-storage state.
2. The frame counter, dirty flag, persistence cursor, adapt metrics, and any other engine-internal scratch state mutate per pass. Reading without mutating is impossible for a single-pass run.
3. A9 (Scheduler post-build is a ZST, modulo the axis-2 phantom type param) does not preclude `&mut self`. Storage is zero; the borrow carries permission to mutate engine-internal state, not data.

`&self` with interior mutability (cell-based plan cache, atomic counters) would force a more elaborate internal layout to spare the consumer from a trivial borrow constraint. The Scheduler is its own owned value; consumers do not share it across borrows. The trade is unjustified.

Concrete signature shape:

```rust
impl<RunOut: RunOutcomeShape> Scheduler<RunOut> {
    pub fn run(&mut self) -> RunOut::Out;
}
```

No additional parameters per axis 1 (parameterless). No lifetime parameter on Out (the Out type is owned, not borrowed from the scheduler).

### Axis 4: plan-cache interaction

**DECIDED.** Auto-recompute on dirty, inside `run()`, before dispatch. The consumer never reasons about the plan cache; recompute happens transparently when the engine's dirty flag is set. Recompute failures surface through `Cfg::Out`'s `Err` variant.

#### Trigger

`run()` checks an internal `plan_dirty: Bool` at entry. If set: recompute the plan, clear the flag, dispatch. If unset: dispatch the cached plan directly.

The dirty flag is set by the substrate at structural-change boundaries:
- `replace_resource(...)` of a Replaceable that participates in the plan (most Replaceables do not; the substrate marks specific Replaceables as plan-affecting via a marker trait).
- Extension load/unload through the ExtensionSurface mechanism (per round 202605091700 reshape).
- `replace_resource(RunCfg)` if the new RunCfg has plan-affecting fields (record_count change).

**Interim behaviour pre-`PlanAffecting` (Topic 8) lands.** `replace_resource(RunCfg)` unconditionally dirties the plan in v1. Pessimistic but correct: a RunCfg replacement that touches `record_count` requires morsel-size recompute, and v1 cannot distinguish that from a no-op replacement until the `PlanAffecting` marker trait ships. Once the marker lands, only RunCfg types impl-ing `PlanAffecting` for the changed field set trigger the dirty flag. Locking "dirty-always" as the conservative interim rather than deferring the decision; consumer pays one full plan recompute per `replace_resource(RunCfg)` call until Topic 8.

The dirty flag is NOT set on:
- Per-frame resource mutation by WUs (the common case — billions of writes per frame, structural shape unchanged).
- Column writes (records mutate; plan stays valid).

#### Recompute failure path

If plan recomputation itself fails (cycle introduced by a newly-loaded extension, missing required Resource, dependency unsatisfiable), `run()` returns `Cfg::Out`'s Err variant. For DefaultRunCfg this is `Outcome::Err(())`; for a heavy consumer's RunCfg it carries whatever structured failure reason their `Out` enum encodes.

#### Why not caller-driven recompute

A `scheduler.recompute_plan()` separate method was considered (Alt B from the topic conversation). Rejected because:
1. Consumer must remember to call it. Forgetting silently runs against a stale plan, which is a wrong-behavior bug rather than a fail-fast compile error.
2. Adds API surface for no consumer-side benefit. Auto-recompute is cheap (the dirty check is a single bool read).
3. Loimu's prior art (RecalculateSchedulePlan signal) confirms auto-recompute-between-passes is the validated shape.

#### Why not exposed as scheduler.is_plan_dirty()

Defer. No known consumer needs to time their `run()` calls based on dirty state. If one emerges, the accessor is additive; landing it in v1 is premature.

#### Open follow-ups

1. **Marker trait for plan-affecting Replaceables.** Substrate ships `PlanAffecting: Replaceable` (or similar). Replaceables that affect plan structure opt in. Topic 8 (Resource resolution + persistence wiring, C6+B2) is the right place to land the trait.
2. **Recompute concurrency.** If a future config wants recompute to overlap with the previous frame's tail, the auto-on-entry shape needs revisiting. Defer until a consumer asks.

### Axis 5: adapt invocation path

**DECIDED.** Adapt is a meta-WU that observes the `ScheduleEnd` virtual (per A6). The substrate ships a default `AdaptWu` impl in `hilavitkutin-providers`; consumers register it via `.with(AdaptWu::default())` if they want substrate's default behaviour, ship their own `AdaptWu` impl if they want custom adapt logic, or omit it entirely if they don't want adapt.

#### Why this shape

1. **Dogfoods A6.** A6 already commits to a self-hosting meta pipeline with `ScheduleEnd` as one of four meta-virtuals. Adapt observing `ScheduleEnd` and mutating `AdaptMetrics` + `LaneAssignment` is the same pattern any consumer WU would use.
2. **Honours A4.** A4 says adapt fires between frames, not during execution. Meta-WUs run at meta-virtual boundaries which ARE between frames by construction. No special-casing.
3. **Heavy-vs-simple.** Simple consumers (viola, vehje) probably don't need adapt; they `.build()` without registering AdaptWu and pay zero cost. Heavy consumers register the default substrate impl or ship their own. Loimu, which already wraps adapt at policy-timer level, may register a minimal adapt-WU that forwards measured EMA to its policy threads.
4. **No special API.** `.with(AdaptWu)` is the same `.with()` everything else uses. No `.enable_adapt()`, no `.disable_adapt()`, no flag in builder config. The adapt-WU's *presence* in the registered WU set is the on/off switch.

#### Substrate-default AdaptWu shape (impl detail, locked at Topic 5)

The substrate's `AdaptWu` reads execution-duration EMAs from per-fiber metrics, computes a new `LaneAssignment` if drift exceeds threshold, writes the new assignment to a meta-resource. The next pass's plan recompute (axis 4) picks up the new lane assignment via its dirty-flag.

The actual algorithm (mode select, EMA constants, threshold) belongs in Topic 5 (adapt subsystem, C3 + B5). This axis only locks the *invocation shape*: adapt is a meta-WU, not a scheduler-internal phase, not a separate `.adapt()` method.

#### Why not inside run() (Alt A)

Tying adapt cadence to "exactly one per `run()` call" fixes a tradeoff the consumer should own. A consumer might want adapt every 16 frames; another every frame; another never. As a meta-WU, the consumer controls cadence via standard WU scheduling (signal observation, frame counters in their own WU).

#### Why not separate scheduler.adapt() method (Alt B)

Adds API surface and a "remember to call this" footgun. Same rejection as the corresponding alt for axis 4.

#### Open follow-ups

1. **AdaptWu shape.** Topic 5 locks the algorithm + the meta-resources it reads/writes.
2. **Substrate-default adapt-WU registration ergonomics.** If the substrate-shipped default is what most consumers want, a "convenience kit" (`.with(DefaultRuntimeKit)`) that bundles adapt + persistence wiring + thread-pool impl into one `.with()` call may be useful. Defer until use sites prove the bundling is worth a kit.

## Decisions (consolidated)

All five axes locked. Concrete shape post-build:

```rust
pub trait RunCfg:
    BuilderInput<Dispatch = RunCfgDispatch<Self>>
    + HasRecordCount
    + Sized + 'static
{
    type Out;
}

pub struct Scheduler<Cfg: RunCfg> {
    _cfg: PhantomData<Cfg>,
    /* ZST internals: counters, plan-cache state, persistence cursor */
}

impl<Cfg: RunCfg> Scheduler<Cfg> {
    pub fn run(&mut self) -> Cfg::Out;
}
```

Lock summary per axis:

1. **Input** — parameterless. RunCfg propagates as a Resource via `.with()`; auto-fills DefaultRunCfg via HasTrivialCtor if consumer didn't register one.
2. **Return** — `notko::Outcome<Cfg::Ok, Cfg::Err>` parameterised by the consumer's `Cfg::Out` associated type. Disambiguator is the dispatch routing slot (`RunCfgDispatch<Self>`); single-occupant typestate slot rejects multi-registration. Field requirements (`HasRecordCount`, future ones) are supertrait bounds on the `RunCfg` trait. The substrate's `PipelineResult` enum is retired.
3. **Receiver** — `&mut self`. Carries permission to mutate counters + plan-cache; A9 ZST stays effectively zero (PhantomData<Cfg>).
4. **Plan-cache** — auto-recompute on dirty inside `run()` before dispatch. Dirty set by structural-affecting Replaceables and extension load/unload. Recompute failure surfaces through `Cfg::Out::Err`.
5. **Adapt** — meta-WU observing `ScheduleEnd` (per A6). Substrate ships default `AdaptWu` in hilavitkutin-providers; consumer registers via `.with(AdaptWu::default())` for default behaviour, ships their own AdaptWu impl for custom, or omits entirely. No special API.

#### Topic-wide open follow-ups

1. **Sketch needed before doc CL lock.** `mock/research/sketches/202605101036-runcfg-dispatch-slot/` validates the typestate slot + `RunCfg` trait + dispatch routing + Scheduler<Cfg> propagation as a self-contained compilable artefact. Per cl-claim-sketch-discipline.md.
2. **Notko derive macro for custom Outcomes.** Out-of-round task; tracked separately. Not blocking this lock; consumer can hand-write Outcome-shaped enums until the macro lands.
3. **PlanAffecting marker trait.** Lands in Topic 8 (resource resolution + persistence wiring).
4. **Default convenience kit.** `DefaultRuntimeKit` bundling AdaptWu + DefaultRunCfg + a persistence-wiring stub. Defer until use sites prove the bundling is worth a kit.
