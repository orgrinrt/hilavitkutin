## 1. SlabColumn: the universal storage atom

SlabColumn is the single building block under every data structure in loimu. Every
collection (SparseArray, DenseColumn, SparseSet, NodeMap, Dictionary) is built on
it. Every view column, every temporal buffer, every signal arena is backed by it.

### Memory model

SlabColumn reserves a large contiguous virtual address range at creation via
`mmap` with `PROT_NONE`. Physical pages are committed on demand as slots are used.
Growth commits the next page; no realloc, no copy, existing pointers remain
valid. Virtual reservation grows to accommodate the maximum key ever inserted
(not the current count), which is fine on 64-bit with lazy page commit.

```
Virtual range: [reserved ==========================================]
Physical:      [committed===]              <- only this uses RAM
               [slot0][slot1]...[slotN]    <- contiguous, fixed stride
                                   ^ growth just commits next page
```

### Stride and access

Stride is `size_of::<T>()` rounded to alignment, known at compile time from
`HasShape`. Access is direct pointer arithmetic:

```
slot_ptr = base_ptr + index * STRIDE
```

No software bounds check on the hot path. Virtual address bounds are enforced by
the memory mapping system (uncommitted pages fault). The compiler const-folds
`STRIDE` and can reduce the multiply to a shift when stride is power-of-two.

For type-erased access (plugin types), stride comes from `Shape` at load time.
Same arithmetic, not const-folded but still branchless.

### OOM policy

Page commit failure (physical memory exhaustion after successful virtual
reservation) is fatal. The process aborts. The framework does not attempt
recovery from OOM.

### Occupancy

Sparse variants use a separate `BitSet` for occupancy (one bit per slot) instead
of `Option<T>` wrappers. This keeps the column data contiguous and
alignment-friendly.

### Platform fallback

If `mmap` is unavailable (WASM, embedded), fall back to a linked-page allocator
with the same API. Page size is configurable. Iteration crosses page boundaries
with one pointer chase per page (amortized over hundreds of slots).

| Platform         | SlabColumn backing                     | Dirty index              |
|------------------|----------------------------------------|--------------------------|
| macOS, Linux     | mmap (reserve + commit)                | Embedded in page 0       |
| Windows          | VirtualAlloc (MEM_RESERVE + MEM_COMMIT)| Embedded in page 0       |
| WASM, embedded   | Linked pages (no virtual mem)          | Separate proportional alloc |

### Data structure primitives

Three building blocks built on SlabColumn, all gap-free and cache-friendly:

| Primitive          | Structure                       | Operations                    |
|--------------------|---------------------------------|-------------------------------|
| `SparseArray<V>`   | `SlabColumn<V>` + `BitSet`      | O(1) get/insert by integer key|
| `DenseColumn<T>`   | `SlabColumn<T>`                 | O(1) push, O(1) swap-remove  |
| `SparseSet<T>`     | sparse + dense + keys           | O(1) all ops, linear iteration|

`SparseArray` replaces HashMap for key-to-index mappings. Direct indexing; no
hashing, no collision chains. `DenseColumn` is the contiguous data store;
swap-remove on delete keeps data packed. `SparseSet` combines both: sparse array
maps keys to dense indices, dense column holds values.

`NodeMap<T>` wraps `SparseSet<T>`. `DenseNodeMap<T>` wraps `DenseColumn<T>` with
a `SparseArray` index. Both are internal to the view system.

## 2. The 16-byte stride model and bit-packing

### Fixed-point type system

loimu uses no `f32`/`f64` anywhere in framework code. All numeric values use
fixed-point types from `loimu-primitives`. Two base types:

| Type                              | Layout                         | Description       |
|-----------------------------------|--------------------------------|-------------------|
| `UFixed<const I: u8, const F: u8>`| I integer + F fractional bits  | Unsigned          |
| `IFixed<const I: u8, const F: u8>`| 1 sign + I integer + F frac    | Signed            |

Both are `#[repr(transparent)]` newtypes wrapping the minimum standard-width
container.

### Type metadata drives column layout

Every fixed-point type carries compile-time metadata: `BIT_WIDTH`, `FRAC_BITS`,
`SIGNED`. The storage engine reads this metadata from `StorageRecord` field
descriptors to determine:

1. Column bit width (how many bits per element)
2. Packing strategy (how elements pack into machine words)
3. SIMD unpack kernel (how to widen for computation)

### Concrete packing examples

```
prec_f16 column: [16b][16b][16b][16b]...  <- 2 elements per 32-bit word
wide_f32 column: [32b][32b][32b][32b]...  <- standard stride
```

A `prec_f16` field occupies 16 bits per element, not 32. This halves memory and
doubles cache line utilization for that column.

### Shape aliases

Three shape patterns control the integer/fractional split at a given total width:

| Pattern      | Meaning                   | Distribution        |
|------------- |---------------------------|---------------------|
| `wide_f{N}`  | Wide integer range        | 75% int / 25% frac  |
| `prec_f{N}`  | High fractional precision | 25% int / 75% frac  |
| `std_f{N}`   | Balanced                  | 50% int / 50% frac  |

Signed by default; `u` prefix for unsigned (e.g., `u_std_f16`).

### Semantic aliases

| Alias    | Type            | Purpose                       |
|----------|-----------------|-------------------------------|
| `Ratio`  | `UFixed<0,8>`   | 0..1 range                    |
| `Factor` | `UFixed<2,8>`   | Multiplier (0..4 typical)     |
| `Norm`   | `IFixed<1,14>`  | Normalized -1..1              |
| `Coord`  | `IFixed<16,16>` | Spatial coordinate            |
| `Angle`  | `UFixed<9,7>`   | Degrees (0..360 with frac)    |
| `Bool`   | `uint1`         | Boolean as numeric            |
| `Nibble` | `uint4`         | 4-bit value                   |

### Storage vs computation width

- Storage: bit-packed at declared width in SlabColumn
- Computation: unpacked to next power-of-2 SIMD lane width
- A pre-computed compile-time table maps bit widths to packing strategy, unpack
  kernel, values per cache line, and chunk size

### Arithmetic strategy annotations

| Annotation            | Strategy          | Description                                 |
|-----------------------|-------------------|---------------------------------------------|
| `#[hot]`              | Lane-capped       | May truncate to stay in narrow SIMD lanes    |
| `#[cold]`             | Full-width        | No precision loss until final write          |
| `#[warm]` (default)   | Configurable      | Uses `FixedPointConfig` resource             |
| `#[precise]`          | Full-width        | For correctness-critical paths (prediction)  |

Overflow: saturating by default. `#[hot]` can switch to wrapping.

### Scoped lint enforcement

Inside `define_record!`, `define_resource!`, `define_behavior!` field types,
standard integers (`u32`, `i64`, etc.) and floats (`f32`, `f64`) are rejected at
compile time. Local variables, loop counters, indices, and FFI boundaries are
exempt.

## 3. Three storage backends

### Overview

| Backend | Purpose                     | Buffering                    | Allocation              |
|---------|-----------------------------|------------------------------|-------------------------|
| Record  | Persistent structured data  | SwapBuffer + SnapshotBuffer  | Views, SoA              |
| Snap    | Ephemeral one-pass data     | Double-buffered arenas       | Bump, pointer swap      |
| Reel    | Pre-computed future values  | FIFO feed, NOT ring buffer   | Contiguous + read cursor|

### Record backend

All persistent data shapes defined via `define_record!` or `#[derive(Record)]`
are stored in the record backend. Views group columns; each column is a
SlabColumn. Non-temporal fields get 2 SlabColumns (frozen + live). Temporal
fields get 3 (old + current + projected). Spring temporal fields get 4 (old +
current + projected + velocity).

Record data is double-buffered per-column within views. Each column's
SnapshotBuffer performs a dirty-region copy from live to frozen at pass boundary.
Idle passes with no writes pay zero memcpy.

### Snap backend

Signals are NOT records. They use the snap backend: double-buffered arenas
(`ExchangeBuffer<ErasedArena<SignalId>>`). The signal bus uses two `ErasedArena`
instances; the read arena (previous pass, frozen) and write arena (current pass,
mutable). At pass boundary the scheduler swaps them and resets the old read
arena. Bump allocation within each arena; pointer swap between passes.

`ErasedArena<K>` maps key `K` to `ErasedColumn`, backed by `SparseArray`.
Supports typed push/iter (via `HasShape` + pointer cast) and raw push/field
access (for plugin types). Cleared per schedule.

At startup, the signal crate's `SignalSetRegistry` inspects every behavior's
observed signals. Signals observed together form a `SignalSet`. The bus
co-locates their columns for cache locality during iteration.

### Reel backend

For tolerance > 2, the reel backend stores pre-computed future values. Reel is a
FIFO feed (NOT a ring buffer):

- Contiguous array with a read cursor
- Bulk-write m values at production time
- Sequential-read one per pass at consumption time
- Size = tolerance
- Advancement is storage engine infrastructure at each pass boundary

Memory layout: reel storage is separate from views (uniform view stride
preserved). Each temporal field with tolerance > 2 gets its own reel. Reels are
SlabColumn-backed with a separate cursor column. The storage engine manages reel
advancement; behaviors just read the current value.

When the reel is exhausted, the last value is held (no wrap).

Beyond temporal prediction, reels serve any pre-computed sequence: animation
curves (keyframe values pre-sampled), procedural generation (noise values
pre-computed), AI decisions (planned moves pre-evaluated), physics substeps
(intermediate states pre-integrated). The reel is agnostic to content.

## 4. SwapBuffer and SnapshotBuffer

### Unified double-buffering

```rust
struct Exchange;  // pointer swap + reset (ephemeral data)
struct Snapshot;  // memcpy live -> frozen copy (persistent data)

struct SwapBuffer<T, Mode = Exchange> { a: T, b: T }

type ExchangeBuffer<T> = SwapBuffer<T, Exchange>;
type SnapshotBuffer<T> = SwapBuffer<T, Snapshot>;
```

| Method       | Exchange                        | Snapshot                              |
|--------------|---------------------------------|---------------------------------------|
| `front()`    | Read side (frozen)              | Snapshot (frozen copy)                |
| `back_mut()` | Write side (mutable)            | Live side (mutable)                   |
| `swap()`     | Pointer swap; old front resets  | Copy dirty regions live to snapshot   |
| Swap cost    | O(1)                            | O(dirty), zero when idle              |

In Snapshot mode, `a` is always the live buffer and `b` is always the frozen
snapshot. `back_mut()` returns `&mut a` (live), `front()` returns `&b`
(snapshot). The snapshot is the destination of the copy, frozen after the copy
completes.

### Instantiation table

| Consumer       | Type                                    | Mode     | Purpose                |
|----------------|-----------------------------------------|----------|------------------------|
| Record backend | `SwapBuffer` + `SnapshotBuffer`         | Snapshot | Frozen + live per view |
| Signal bus     | `ExchangeBuffer<ErasedArena<SignalId>>`  | Exchange | Read/write arena pair  |
| GPU bindings   | `ExchangeBuffer<SlabColumn<T>>`         | Exchange | CPU write / GPU read   |

### Embedded dirty index

Every SlabColumn reserves one page (4KB) at the start of its virtual allocation
for a dirty tracking header. Slot data starts at a fixed offset (`PAGE_SIZE`,
compile-time constant). When tracking is disabled, the header page is never
committed; zero physical memory cost.

```
[page 0: dirty index            ][page 1+: slot data           ]
 flags(4B) count(4B) spill(8B)    [slot0][slot1]...[slotN]
 range entries (up to 509)         ^ base_ptr = alloc_ptr + PAGE_SIZE
```

Page 0 layout:
- `flags: u32` -- tracking enabled, all-dirty, spilled
- `count: u32` -- number of range entries in page 0
- `spill_ptr: *mut SpillBuffer` -- null unless overflowed
- Range entries `[(start: u32, len: u32)]` -- up to 509 in one page

When dirty ranges exceed page 0 capacity, a heap-allocated SpillBuffer extends
the index. Spill allocation is lazy (created on first overflow) and freed at pass
boundary.

Both `a` and `b` have page 0 reserved. `a`'s (live) page 0 is used for write
tracking. `b`'s (snapshot) page 0 is used for the undo journal.

### Merge strategies

Selected at compile time by the consumer macro:

| Strategy          | Write cost             | Boundary cost         | Best for             |
|-------------------|------------------------|-----------------------|----------------------|
| `MergeOnWrite`    | O(1) amortized (cursor)| O(1) (already sorted) | CPU SnapshotBuffer   |
| `AppendAndMerge`  | O(1) (append)          | O(n log n) (sort)     | GPU bindings         |

MergeOnWrite: cursor-optimized sorted insertion. A cursor tracks the last
insertion position. Spatially local writes (common case in UI and game workloads)
hit the cursor and merge in O(1). Scattered writes fall back to binary search
O(log n). Used by `define_resource!` with `#[double_buffered]`.

AppendAndMerge: O(1) appends during the pass. Tracks bounding min/max
incrementally. Sort and merge into contiguous ranges at pass boundary. Used by
`define_binding!` with `buffering: Double`.

### Dirty index API

```
is_any_dirty() -> bool                  // coarsest check; skip sync if clean
dirty_bounds() -> Option<(u32, u32)>    // bounding min/max (GPU bulk DMA)
dirty_ranges() -> impl Iterator<(u32, u32)>  // exact ranges (CPU memcpy)
clear_dirty()                           // reset index + free spill
```

### Dirty-tracked sync flow

1. Pass start: no copy. The snapshot from the previous pass is still valid for
   readers.
2. During pass: writers modify the live buffer. Each write records the slot
   index in `a`'s page 0 dirty index (via the selected merge strategy).
3. Pass boundary (`swap()`): `is_any_dirty()` -- if clean, skip entirely.
   Otherwise, iterate `dirty_ranges()` and copy only those slot ranges from
   `a`'s data pages to `b`'s data pages. Clear the index.

Idle passes pay zero memcpy. The pass boundary is a synchronization point (DAG
guarantees no concurrent writes), so no atomics are needed on the dirty index.

### Undo journal for behavior rollback

If a behavior panics or traps (WASM fuel exhaustion), its writes must be rolled
back. The rollback strategy is determined at plan time by the scheduler based on
DAG analysis. Two strategies exist, both using existing buffer memory:

**Sole-writer rollback:** When the DAG shows a behavior is the only writer to a
resource in the current pass, buffer `b` (the snapshot) already contains the
pre-behavior state. On panic: copy dirty ranges from `b` back to `a` (reverse of
normal sync direction), then clear the dirty index in `a.page0`. Cost during
normal execution: zero. No journal, no extra writes, no memory.

**Shared-writer rollback:** When multiple behaviors write to the same resource
(serialized in the DAG), buffer `b` predates all writers. Instead, the undo
journal in `b.page0` records each field write:

```
b.page0 layout (undo journal):
  count: u32            -- number of entries
  entries: [(offset: u32, len: u16, old_value: [u8; len])]
```

4KB holds enough entries for typical field-level writes (~100-170 entries
depending on field sizes). Overflow spills to heap.

Per-behavior flow:
1. Reset `b.page0` journal count to 0
2. Run behavior. Each resource field write appends `(offset, len, old_value)` to
   `b.page0`
3. Success: journal abandoned in place (count reset at next behavior)
4. Panic: replay journal in reverse, memcpy old values back to `a`

Strategy selection: The scheduler computes `sole_writer: bool` per (behavior,
resource) pair during plan computation. This is a static property of the DAG, not
a runtime check. Mode is a zero-size marker type, monomorphized at compile time.
No runtime branching.

## 5. Views: engine-managed archetypes

### Two-layer system

1. **Static schema (build-time):** Behavior access declarations produce a column
   co-access graph. Affinity analysis groups frequently co-accessed columns into
   view templates.
2. **Dynamic population (runtime):** View instances are created and destroyed by
   population thresholds with hysteresis (promote at 1.5x, demote at 0.5x). All
   threshold-based transitions use the `Hysteresis` primitive; no ad hoc threshold
   logic.

### Affinity analysis algorithm

At startup, the engine collects every handler's access declarations:

```
HandlerA: reads [position, velocity], writes [position]
HandlerB: reads [position, color]
HandlerC: reads [velocity, mass]
```

Co-access frequency matrix:

```
           position  velocity  color  mass
position      -        2        1      0
velocity      2        -        0      1
color         1        0        -      0
mass          0        1        0      -
```

Affinity groups are computed by clustering columns with high co-access frequency.
In this example: {position, velocity} and {color} and {mass}. Each group becomes
a view template.

### View templates vs view instances

A view template defines which columns are together. A view instance is a concrete
allocation holding elements matching a particular marker combination. Multiple
marker combinations that access the same column set share the same view template;
each gets its own view instance (separate dense storage).

### Population thresholds with hysteresis

- Promote at 1.5x threshold: sparse elements migrated to a new view
- Demote at 0.5x threshold: view destroyed, elements return to sparse

Views are defined by access pattern, not by marker combination. Narrow views (few
columns, many elements) are more cache-friendly than wide views (many columns,
few elements). The affinity analysis naturally produces narrow views.

### Physical layout

SoA-only. Each column within a view is a SlabColumn, dense-packed, with
append/swap-remove.

```
View "spatial" (columns: position, velocity):
  position: SlabColumn<Vec3<Coord>>  [p0][p1][p2]...
  velocity: SlabColumn<Vec3<Coord>>  [v0][v1][v2]...

  index 0: element for NodeId 42
  index 1: element for NodeId 7
```

Node-to-view mapping: `SparseArray<ElementId, (ViewId, DenseIndex)>` per record
type, O(1) lookup.

### View transitions

When an element needs to move between views (marker set change):
1. Mutation is deferred to pass boundary
2. Storage engine determines new view membership
3. Batch mode (scheduler saturation < 0.7): all pending moves execute in bulk
4. Scatter mode (scheduler saturation >= 0.7): PendingMoves queue; relocate only
   elements about to be accessed; rest stay pending

Scatter mechanism:
1. Pre-batch analysis against op queue
2. Relocate only elements scheduled for access this pass
3. Batch planning proceeds on correct state
4. Remaining moves stay in PendingMoves for next pass boundary

All threshold-based decisions (batch vs scatter) use `Hysteresis`.

### View restructuring

Restructuring triggered when fitness metric < 0.6 persists for ~60 passes:
1. Flush PendingMoves first
2. Snapshot current membership
3. Run affinity analysis on actual population (not just static schema)
4. Diff new templates against current templates
5. Bulk-copy elements to new view layout
6. Update all node-to-view mappings
7. Free old memory

View restructuring uses a separate budget threshold from scatter/batch. Also
driven by `Hysteresis`.

### Fallback sparse storage

Elements with rare marker combinations fall below population threshold and live
in sparse storage: one `SparseSet` per record type. When population crosses the
promote threshold (with hysteresis), a view is created and elements migrate.

### Memory budget estimation

Per-view memory = (element_count x column_stride) x column_count.

Breakdown per field type:
- Non-temporal field: 2 x column_stride (frozen + live)
- Temporal field: 2 x 3 x column_stride (frozen + live, each with old/current/projected)
- Bit-packed field: column_stride = ceil(element_count x bit_width / 8)

Sparse fallback overhead per record: one SparseSet (sparse array + dense column +
key column). More memory per element than a view, but acceptable for
rare-combination populations below the promote threshold. The storage engine
tracks per-view memory usage and accounts for memory cost of new layouts before
committing to restructuring.

## 6. Operation fusion

### Why deferred writes enable fusion

Behaviors do not write directly to columns during execution. They push
per-element operations via `each` blocks. The storage engine collects all ops,
groups them by data affinity, and executes fused iteration passes. Because frozen
data is immutable during a pass, there are no dependencies between readers of the
same column.

### Per-element op model

Each `each` block in a handler pushes an op descriptor:

```
Op {
    handler_id: HandlerId,
    view_id: ViewId,
    read_columns: &[ColumnId],
    write_columns: &[ColumnId],
    access_mode: AccessMode,
    fn_ptr: fn(&mut ElementCtx),
}
```

The storage engine receives ops in scheduler DAG order (correctness ordering is
already resolved by the scheduler).

### Two-level scheduling

| Level          | Concern      | Mechanism                                  |
|----------------|--------------|--------------------------------------------|
| Scheduler      | Correctness  | DAG ordering, barrier enforcement, lanes   |
| Storage engine | Efficiency   | Data-affinity fusion, column co-iteration  |

The scheduler emits an ordered sequence of scheduling units. The storage engine
receives this sequence and fuses compatible units within each barrier-free span.

### Fusion algorithm

1. Receive ordered units from scheduler
2. Partition into fusible groups at barrier boundaries
3. Within each group, cluster ops by shared read columns and view membership
4. One iteration pass per cluster: loop over shared columns, dispatch all
   compatible ops per element in scheduler order
5. Barriers run standalone between groups

### Three fusion tiers

| Tier | Name          | Access modes          | Fusibility                        |
|------|---------------|-----------------------|-----------------------------------|
| 1    | Unconditional | Pinned, Approx, Defer | Always fusible; relaxed ordering  |
| 2    | Ordered       | &T, &mut T            | Per-element order preserved       |
| 3    | Chaos         | Sample, Post          | Fusible, relaxed ordering         |

Access mode semantics:
- **Pinned:** Element-local state only, no cross-element dependencies. Always
  safe to fuse.
- **Approx:** Result may vary with execution order but remains within acceptable
  bounds (e.g., force accumulation). Fusible; order relaxed.
- **Defer:** Write is deferred to pass boundary. Always safe to fuse since the
  write does not affect other ops in this pass.
- **&T:** Read-only. Fusible with other reads and with &mut T on different
  columns.
- **&mut T:** Per-element write. Fusible with reads on different columns. Two
  &mut T on the SAME column: NOT fusible (serialize).
- **Sample:** Reads a statistical sample of the column. Order does not matter.
- **Post:** Runs after all other ops for this element. Fusible with other Post
  ops.

### Compatibility matrix

```
         Pinned  Approx  Defer  &T    &mut T  Sample  Post
Pinned     Y       Y       Y     Y      Y       Y      Y
Approx     Y       Y       Y     Y      Y       Y      Y
Defer      Y       Y       Y     Y      Y       Y      Y
&T         Y       Y       Y     Y      Y*      Y      Y
&mut T     Y       Y       Y     Y*     X**     Y      Y
Sample     Y       Y       Y     Y      Y       Y      Y
Post       Y       Y       Y     Y      Y       Y      Y

* &T and &mut T fusible only on DIFFERENT columns
** Two &mut T on same column: NOT fusible (serialize)
```

### Fused iteration structure

```
for element in view.dense_iter() {
    // All Tier 1 ops (any order)
    op_a.execute(element);
    op_c.execute(element);

    // All Tier 2 ops (scheduler DAG order)
    op_b.execute(element);
    op_d.execute(element);

    // All Tier 3 ops (any order)
    op_e.execute(element);
}
```

Within a fused pass, the element pointer advances once; all compatible ops
execute per-element before moving to the next element.

### Fusion invariant

Correctness guarantee: for any set of ops, the result of fused execution is
identical to executing each op in a separate iteration pass in scheduler order.
Verifiable by disabling fusion and comparing outputs.

### Non-fusible patterns

- Tree traversal: requires parent-child ordering, not element-local
- Reductions: aggregate over all elements (sum, max, count)
- Scalar ops: operate on singleton resources, no per-element iteration
- Conditional logic: ops with control flow that depends on other elements' state

Non-fusible units run standalone. The storage engine assigns them to
cache-affinity lanes when possible (prefer the lane that last touched the same
columns).

### Vectorized execution

Fused iteration processes elements in ~1024-value chunks sized to fit in L1
cache. The chunk size is pre-computed per view based on column widths.

Within each chunk:
1. Load columns into registers/cache
2. Execute all fused ops per element in scheduler order
3. Write results to output columns
4. Advance to next chunk

## 7. Temporal storage

### Temporal trait

```rust
pub trait Temporal {
    fn predict(old: &Self, current: &Self, factor: Factor) -> Self;
}
```

Factor semantics:
- < 1.0: interpolation between old and current
- = 1.0: identity (returns current)
- > 1.0: extrapolation beyond current

Factor type is `Factor` (`UFixed<2,8>` from `loimu-primitives`), not `f32`.

### Default blanket impl

```rust
impl<T: Add<Output=T> + Sub<Output=T> + Mul<std_f32, Output=T> + Copy> Temporal for T {
    fn predict(old: &Self, current: &Self, factor: Factor) -> Self {
        let delta = *current - *old;
        *current + delta * factor.to_std_f32()
    }
}
```

Linear extrapolation. Specialized prediction via newtypes (e.g.,
`PhysicsPosition(Vec3)` overrides `predict` with quadratic).

### SpringTemporal trait

```rust
pub trait SpringTemporal: Temporal {
    fn spring_predict(
        old: &Self, current: &Self, velocity: &Self,
        params: &SpringParams, dt: Factor,
    ) -> (Self, Self);
}
```

Returns `(new_value, new_velocity)`. Semi-implicit Euler integration: force from
spring equation, then velocity update, then position update.

### Spring parameters

```rust
pub struct SpringParams {
    pub style: SpringParamStyle,
}

pub enum SpringParamStyle {
    ResponseDamping { response: Factor, damping: Factor },
    StiffnessDampingMass { stiffness: Factor, damping_coeff: Factor, mass: Factor },
}
```

ResponseDamping (SwiftUI style): response controls settling speed, damping
controls overshoot (1.0 = critical). Converted to physics parameters:
`stiffness = (2*PI/response)^2`, `damping_coeff = 4*PI*damping/response`.

### 3-column layout

| Field type              | Columns                           | Purpose                        |
|-------------------------|-----------------------------------|--------------------------------|
| Non-temporal            | 2 (frozen + live)                 | Double-buffering               |
| Temporal (tolerance=2)  | 3 (old + current + projected)     | Inline triple buffer           |
| Temporal (spring)       | 4 (old + current + proj + vel)    | Extended for spring dynamics   |
| Temporal (tolerance>2)  | 3 inline + reel                   | Spillover to reel backend      |

Buffer rotation at pass boundary: pointer swap (zero-copy), old becomes
previous-old, current becomes old, projected becomes current. For spring fields,
the velocity slot rotates alongside the other three.

### CPU buffer multiplier

| Field type      | Slots | Swap | Total |
|-----------------|-------|------|-------|
| Non-temporal    | 1     | 2x   | 2x    |
| Temporal        | 3     | 2x   | 6x    |
| Spring temporal | 4     | 2x   | 8x    |

### Cadence-driven prediction

`TemporalPredictionBehavior` runs prediction for all temporal fields. It is a
behavior (not a storage engine operation), scheduled by the normal DAG.
Auto-registered when any record has `#[temporal]` fields. Triggers `CadenceMap`
auto-registration via auto-defaults.

Prediction runs at the cadence of each field's signal type. The behavior iterates
all elements with temporal fields of a given cadence, calls
`Temporal::predict()` (or `SpringTemporal::spring_predict()` for spring fields),
and writes the projected column. Per-type batching: all elements of the same type
in the same view are processed contiguously, maximizing cache utilization.

### CadenceMap resource

```rust
define_resource!(#[singleton] CadenceMap {
    entries: Dictionary<SignalId, CadenceEntry>,
});
```

`CadenceEntry`: `current_interval` (hot), `last_commit_time` (hot), `policy`
(cold). Read-only for behaviors, write-only for scheduler. Immutable during a
schedule.

### Cadence policies

```rust
pub enum CadencePolicy {
    Fixed { interval: CadenceBound },
    Curved { heuristic: CadenceHeuristic, curve: EasingCurve },
    Tracking { tracks: SignalId, ratio: Factor, floor: CadenceBound, ceiling: CadenceBound },
}

pub enum CadenceBound {
    None,
    Literal(Duration),
    Signal(SignalId),
}
```

- Fixed: constant interval. `CadenceBound::Signal` ties to another signal's cadence.
- Curved: adapts based on heuristic along an easing curve. `CadenceHeuristic` is
  extensible; `SchedulerLoad` is the initial variant.
- Tracking: follows another signal at a ratio, clamped to floor/ceiling.

Circular tracking (A tracks B tracks A) is a registration-time error. The
scheduler alone computes cadence adaptation. No user behavior writes
`CadenceMap`. EMA smoothing is scheduler-global, not per-policy.

### Reel model

For tolerance > 2: FIFO feed (NOT a ring buffer), contiguous array with a read
cursor, bulk-write m values at production time, sequential-read one per pass at
consumption time. Size = tolerance. Advancement is storage engine infrastructure.

## 8. Marker storage

### Declaration forms

Three forms via `define_marker!`:

```rust
// Boolean tag (no data)
define_marker!(Visible);

// Combined: generates record + marker + association
define_marker!(Spatial2d {
    local: Transform2D = Transform2D::IDENTITY,
    world: Transform2D = Transform2D::IDENTITY,
});

// Explicit association (record defined separately)
define_record!(PhysicsData {
    mass: Coord = Coord::ONE,
    #[temporal(cadence = PhysicsStep)]
    velocity: Vec3<Coord> = Vec3::ZERO,
});
define_marker!(Physics: PhysicsData);
```

### Marker-record separation

Marker = boolean tag (FilterCache bitset membership only). Record = data shape
(`StorageRecord` impl). Association = `define_marker!(M: R)` links the two.

The storage engine places record data in views based on marker combinations. The
marker itself carries no data; it determines which records a node has.

### Storage engine placement

When a node gains marker `M` (associated with record `R`):
1. Marker is added to the node's marker bitset
2. FilterCache is invalidated for `M`
3. Storage engine determines the node's new view membership based on its full
   marker set
4. Default record data for `R` is inserted into the appropriate view
5. Node-to-view index is updated

Removal is the reverse.

### Double-buffered access

Record data in views is double-buffered (frozen + live) by default. Behaviors
reading `nodes M` get the frozen snapshot (parallel); behaviors writing `nodes M`
get the live buffer (exclusive).

For guaranteed immediate consistency, annotate the record:
```rust
define_record!(#[single_buffer] FocusState {
    focused: Bool = Bool::FALSE,
});
define_marker!(Focus: FocusState);
```

Single-buffered records use normal DAG edges; readers and writers are serialized.

### DAG edge rules

| Access                      | Double-buffered               | Single-buffered    |
|-----------------------------|-------------------------------|--------------------|
| `read nodes M`              | No DAG edge to writers        | Normal DAG edge    |
| `write nodes M`             | Exclusive, serialized         | Exclusive          |
| read vs read                | Parallel                      | Parallel           |
| write vs write              | Conflict (sequential)         | Conflict           |

For double-buffered records, read-vs-write is always disjoint in
`AccessSet::is_disjoint()`, enabling parallel execution.

### FilterCache

The scheduler maintains a `FilterCacheMap<MarkerId>` that caches which nodes
carry each marker type. Caches are invalidated at pass start and lazily populated
on first access.

`FilterCacheMap<K>` uses a `SparseArray` with `TypeId -> slot` indirection for
O(1) per-marker-type cache lookup after first access. Each `FilterCache<K>`
stores matching IDs in a `DenseColumn<K>` (sorted) for cache-local contiguous
iteration and O(log n) membership tests via binary search.

```
pass start:   filter_caches.invalidate_all()
first access: compute nodes_with_marker::<M>(), populate FilterCache
subsequent:   return cached &[NodeId] slice (sorted, contiguous)
```

Marker mutations during the pass take effect next pass.

### Context API

```rust
// Read (frozen snapshot for double-buffered, data for single-buffered)
ctx.nodes::<M>()           // -> impl Iterator<Item = (NodeId, &R)>
ctx.node_data::<M>(id)     // -> Option<&R>

// Write (live buffer for double-buffered, data for single-buffered)
ctx.nodes_mut::<M>()       // -> impl Iterator<Item = (NodeId, &mut R)>
ctx.node_data_mut::<M>(id) // -> Option<&mut R>
```

These methods are only available if the handler declared the corresponding
access. Calling `nodes_mut::<M>()` without `write nodes M` is a compile error.

## 9. Type erasure

### ErasedColumn

`CollectionStorage` holds heterogeneous types in a single container via
`ErasedColumn`:

```
ErasedColumn {
    data: NonNull<u8>,
    len: usize,
    item_layout: Layout,
    shape: Shape,
    type_id: Option<TypeId>,
    drop_fn: Option<fn(*mut u8)>,
}
```

For static types, `push_typed<T>()` and `get_typed<T>()` use TypeId guards and
compile-time field access (zero overhead). For plugin types,
`get_field_raw(index, offset)` does `base + index * stride + offset` pointer
arithmetic with offsets validated at plugin load time.

### Shape

A `Shape` is the runtime equivalent of a struct's memory layout:

```
Shape { name, size, align, fields: [ShapeField] }
ShapeField { name, offset, size, field_type }
```

`ShapeField` carries an `is_temporal: bool` flag set by `define_record!` when a
field has `#[temporal]`. This enables view-level buffer operations: the storage
engine allocates 3-buffer rotation for temporal columns instead of the standard
double-buffer pair.

### HasShape trait

For static Rust types, `define_record!` generates `HasShape` impls using
`offset_of!`. For WASM plugin types, `Shape` is computed from the TOML manifest
via `schema_to_shape()` (in loimu-reflect).

### Plugin projection

When a plugin declares `projects_to = "core::ThemeState"`:

1. Host `Shape` for `ThemeState` is looked up (generated by `define_record!`).
2. Plugin `Shape` is computed from its manifest.
3. `ProjectionMap::validate()` checks field-by-field: names, types, sizes,
   offsets.
4. If valid, plugin data goes into the same `ErasedColumn` as the host type.

After validation, host behaviors iterating `ThemeState` see both static and
plugin-provided instances in one contiguous scan. No virtual dispatch, no
per-access coercion.

### Core buffer types summary

| Type                    | Purpose                                               |
|-------------------------|-------------------------------------------------------|
| `SlabColumn<T>`         | Universal storage atom: mmap-backed, fixed-stride     |
| `BitSet`                | Compact occupancy tracking for sparse columns         |
| `ErasedColumn`          | Type-erased column for static + plugin types          |
| `Shape`                 | Runtime layout descriptor (field offsets, sizes)       |
| `ShapeField`            | Individual field within a Shape                       |
| `SwapBuffer<T, Mode>`   | Double-buffered pair with front/back semantics        |
| `Density`               | Dense, Sparse, or Auto                                |
| `FieldLayout`           | AoS, SoA, or Auto                                    |
| `StorageLayout`         | Layout hint: Density x FieldLayout, Arena, Auto       |
| `StorageStats`          | Runtime statistics (column count, bytes, pages)       |
| `DirtyIndex`            | Private internal dirty tracking for SlabColumn        |
| `ErasedArena<K>`        | Type-erased arena for ephemeral signal storage        |
| `Reel<T>`               | Pre-computed FIFO feed (bulk write, sequential read)  |
| `MergeStrategy`         | MergeOnWrite or AppendAndMerge                        |
| `UndoEntry`             | Undo journal entry (offset, length, old value bytes)  |
| `UndoJournal`           | Behavior rollback journal (ordered undo entries)      |
| `ProjectionMap`         | Plugin type projection validation                     |
| `Hysteresis<T>`         | Threshold-based transition primitive                  |

## 10. Selection vectors

Selection vectors (index arrays) filter elements by marker membership without
copying data. When a fused iteration only needs elements with a specific marker,
the selection vector contains the dense indices of matching elements within the
view.

```
View "spatial" (1000 elements):
  Selection for "Visible" marker: [0, 2, 5, 7, 12, ...]
  -> iterate only those indices, skip the rest
```

Selection vectors are built from FilterCache bitsets intersected with view
membership. They are zero-copy: the selection vector points into the view's
existing dense columns, no data movement.

For views where all elements match the filter (common case), the selection vector
is elided and iteration is a straight sequential scan. This is the typical path
for views that exist precisely because a marker combination is common; the
selection vector is a fallback for cross-cutting queries against views that
contain a superset of the needed elements.

## 11. Design rationale and critique

### What works well

**SlabColumn as universal atom.** Having a single backing type for everything
eliminates allocation diversity. One growth policy, one OOM policy, one access
pattern. The mmap-based zero-copy growth is genuinely zero-cost for the common
case; no realloc, no copy, existing pointers stable. The embedded dirty index in
page 0 is clever: zero physical cost when disabled, O(dirty) sync cost, and it
avoids a separate allocation for tracking metadata.

**Deferred writes enabling fusion.** By forcing all writes through `each` blocks
and keeping frozen data immutable during a pass, the storage engine gains
complete freedom to reorder and fuse operations. The fusion invariant (disabling
fusion produces identical results) is a strong correctness guarantee that makes
the optimization transparent.

**Two-level scheduling.** Separating correctness (scheduler DAG) from efficiency
(storage engine fusion) is a clean division. The scheduler never needs to know
about column layout, and the storage engine never needs to know about dependency
ordering.

**View affinity analysis.** Defining views by access pattern rather than by
marker combination avoids the combinatorial explosion that plagues traditional
ECS archetype systems. Two marker combinations needing the same columns share a
template, reducing the total number of distinct layouts.

**Type erasure with validated projection.** Validating plugin Shapes once at load
time and then using the same ErasedColumn for both host and plugin types is an
effective approach. Zero per-access overhead after validation. Host behaviors see
all instances in one contiguous scan.

### What is complex

**Seven access modes.** Pinned, Immediate, MutImmediate, Sample, Post, Approx,
Defer. The compatibility matrix is 7x7 with footnotes. The distinction between
Approx and Sample is subtle (both relax ordering, but Approx tolerates
approximation while Sample reads a subset). The three fusion tiers add another
classification layer on top. This is powerful for a UI framework doing
signal-driven scheduling, but the cognitive load is high.

**Temporal column multiplier.** A spring temporal field requires 4 slots x 2
(swap) = 8x memory. For records with multiple spring temporal fields, memory
grows fast. The reel backend for tolerance > 2 adds yet another storage location
separate from views. The interaction between temporal columns, views, and reels
creates three-way coupling in the storage engine.

**View transitions under load.** The batch/scatter adaptive transition, the
fitness metric, the ~60-pass persistence threshold, the separate restructuring
budget: these are many parameters to tune. The PendingMoves queue means views may
be temporarily inconsistent (elements in the wrong view), which the access layer
must handle.

**Undo journal dual-purpose page 0.** Buffer `a`'s page 0 is the dirty index;
buffer `b`'s page 0 is the undo journal. During a swap, `b` becomes `a` and its
page 0 transitions from journal to dirty index use. This works, but the same
physical memory serves two different purposes at different lifecycle stages,
which makes reasoning about invariants harder.

**CadenceMap and cadence policies.** Three policy types (Fixed, Curved, Tracking)
with `CadenceBound` that can be None, Literal, or Signal, plus circular tracking
detection: this is a complex configuration space. Useful for loimu's
signal-driven model, but most consumers would need only Fixed.

### What hilavitkutin should adopt

1. **SlabColumn as the universal atom.** The mmap-backed, fixed-stride,
   zero-copy growth model is ideal for any columnar store. The embedded dirty
   index is worth keeping.
2. **Deferred writes with fusion.** The two-level scheduling model (correctness
   from DAG, efficiency from storage engine) is the right separation of concerns.
3. **SoA views with affinity analysis.** Even a simplified version (static-only,
   no dynamic restructuring) would give good cache behavior.
4. **SnapshotBuffer for read/write separation.** The dirty-tracked sync with
   zero cost for idle passes is directly applicable.
5. **Selection vectors.** Zero-copy filtered iteration with FilterCache bitsets
   is a clean primitive.

### What hilavitkutin should simplify

1. **Access modes.** Read and Write (possibly ReadExclusive) are sufficient. The
   seven-mode system is loimu-specific. Fusion tiers collapse to: read-only ops
   are always fusible, write ops on different columns are fusible, two writes to
   the same column serialize.
2. **Temporal storage.** Not needed in hilavitkutin. Polka-dots runs once and
   exits; saalis does not do prediction. loimu should layer temporal columns on
   top of hilavitkutin's base storage.
3. **Cadence policies.** hilavitkutin does not need signal-driven cadence. The
   DAG executes when work is submitted.
4. **View restructuring.** Dynamic restructuring, fitness metrics, ~60-pass
   persistence thresholds: these add complexity for long-running UI processes.
   For a data compiler (polka) or query engine (saalis), static views from
   build-time analysis are sufficient.
5. **Spring dynamics.** Specific to loimu's animation system.
6. **Seven-mode undo journal.** Sole-writer rollback (zero-cost via snapshot) is
   worth keeping. Shared-writer undo journal is loimu-specific complexity.

## 12. Relevance to hilavitkutin

### Q1: unified store

The hilavitkutin topic asks whether `Read<T>` / `Write<T>` should unify columns
and resources. Loimu's answer is yes: both are backed by SlabColumn, the
scheduler does not distinguish row count, and morsel chunking is internal to
execute(). `Read<HttpClient>` (singleton) and `Read<Entities>` (collection)
create DAG edges the same way. hilavitkutin should adopt this.

### Q7: storage abstraction

The hilavitkutin topic asks about a storage trait for all three projects.
Loimu's `StorageFacade` trait is implemented by all backing storage structs.
hilavitkutin needs a simpler version:

- Contiguous slice access for morsel processing (polka, loimu)
- Materialized view from DB query results (saalis)
- Both singleton and collection access patterns

The SlabColumn model works for polka and loimu. saalis needs a bridging
implementation that materializes SQLite query results into a SlabColumn-like
contiguous buffer for processing, then writes back.

### What the unified store should look like

Based on loimu's storage architecture, hilavitkutin's store should provide:

1. **Column<T>** backed by a simplified SlabColumn (mmap growth, fixed stride,
   embedded dirty tracking). This is the universal atom.
2. **Read<T> / Write<T>** that work identically for singletons and collections.
   The scheduler sees them as DAG edges with no cardinality distinction.
3. **Static views** from build-time affinity analysis. No dynamic population
   thresholds, no restructuring, no PendingMoves. Views are computed once at
   startup from access declarations and do not change.
4. **SnapshotBuffer** for read/write separation with dirty-tracked sync. Idle
   resources pay zero sync cost.
5. **Selection vectors** for filtered iteration over view columns.
6. **ErasedColumn + Shape** for plugin type erasure. Validated projection so
   plugin data and host data share the same column.

loimu layers temporal columns, spring dynamics, reel backends, cadence policies,
signal arenas, and seven access modes on top of these primitives. Those layers
stay in loimu. hilavitkutin provides the foundation they build on.

### Record model

hilavitkutin should adopt the record model concept: a trait describing field
layout, default values, and type metadata. polka-dots already has schema-as-traits
from the paradigm topic. The `StorageRecord` trait with `FIELDS`, `COMPONENTS`,
and `PARAMS` maps directly to polka's `Column<In<_>, As<_>>` IR definition
pattern. hilavitkutin unifies the two by making `StorageRecord` (or its
equivalent) the base trait that both polka's schema-as-traits and loimu's
`define_record!` generate.

### Dependency graph

loimu-storage depends on loimu-buffers depends on loimu-primitives (plus id,
error-primitives, visibility-macros, codepath-macros, diagnostics, registry).
hilavitkutin should own the buffer primitives (SlabColumn, SwapBuffer, BitSet,
ErasedColumn, Shape) and the record trait. The primitives crate (arvo, in
hilavitkutin's case) provides the numeric types that record fields use. Storage
is a thin layer on top of buffers, providing views, fusion, and the storage
engine itself.
