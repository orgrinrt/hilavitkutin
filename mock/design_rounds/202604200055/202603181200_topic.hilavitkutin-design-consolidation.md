Runtime dependency graph:
```
consumer
  └─ hilavitkutin
       ├─ hilavitkutin-api
       ├─ arvo, arvo-bitmask, arvo-graph, arvo-sparse
       ├─ arvo-spectral (deps: arvo-sparse)

```

hilavitkutin-build is a standalone build-dep with no runtime deps.
It provides cfg emission, rustc wrapper scripts, LLVM pass config,
PGO/BOLT workflows, and pragma-based build configuration. All three
hilavitkutin crates and all consumers use it in their build.rs.

No dependency on polka-*, std, alloc, or external crates in any of
the three crates. Designed for extraction to a separate repository.

## Vocabulary

Canonical terms for this design. Dead terms listed to prevent
regression to stale mental models.

**Hierarchy (coarsest → finest):**
```
pipeline → core → phase ↔ waist → trunk → fiber ↔ branch ↔ bridge → morsel → micro-morsel → entry
```

| Term | Definition |
|---|---|
| **record** | One data point in a column. The global identity. A column has N records. A morsel windows into a range of records. Not "entity" (no ECS connotation), not "row" (no tabular connotation). |
| **entry** | One record × one op. Register level. The innermost unit. |
| **micro-morsel** | Inner tiling when peak_live > L1. Sub-morsel record range. Single-level by default; activates at ECS scale. |
| **morsel** | 3D window into a fiber's co-located arena: records × columns × temporal. L1-sized. Multiple morsels per fiber. |
| **fiber** | Unit of column co-location and morsel windowing. WUs within a fiber share columns. Morsels window into the fiber's arena. Execution strategy (fused vs sequential passes) is decided by the codegen, not the fiber definition. Replaces "chain." |
| **branch** | Side path attached to a trunk. Peer of fiber within a trunk. |
| **bridge** | Fan-in node reading from multiple trunks. Peer of fiber. Runs after parent trunks reach required record range. |
| **trunk** | Column-disjoint maximal path within a phase. Contains fibers, branches, bridges. Trunks share NO write columns — zero sync. |
| **waist** | Narrow bottleneck in the DAG where concurrent path count drops to minimum. Defines phase boundaries. |
| **phase** | Wide DAG section between waists. Phases execute with pipeline parallelism. Each phase has its own execution strategy. |
| **core** | Physical CPU core used as an execution unit. One pool thread per physical core. Core-pinned trunks. Not the Rust `core` crate. |
| **pipeline** | The entire DAG. Coarsest level. |

**Dead terms (do not use):**
| Dead term | Replacement | Why |
|---|---|---|
| chain | fiber | "Chain" causes regression to stale T2/T3 mental model |
| chain group | trunk | Trunks replaced chain groups |
| partition | phase | Phases replaced partitions |
| archetype | fiber | Archetype was just column co-access, which IS fiber formation |
| entity | record | "Entity" implies ECS; hilavitkutin is not an ECS |
| row | record | "Row" implies tabular joins; columns are independent |
| order (on WU) | scheduling hints | WUs declare hints, scheduler derives ordering |

### 01. arvo primitives

**CONSOLIDATED.** Expanded from numerics-only to include mathematical
primitives used across all arvo consumers. Detailed design of each
crate deferred to its own design round.

**What arvo is:**
- Standalone `#![no_std]` primitive library family, cross-project
  (polka, saalis, loimu, hilavitkutin)
- Lives in hilavitkutin workspace, extracted to own repo later (same
  lifecycle)
- Name: arvo (Finnish: "value/worth/estimate")
- All crates: `#![no_std]`, no alloc, const-generic sizing, same
  LLVM optimisation discipline as numerics. Define once, reuse everywhere.

**Crate structure:**

```
arvo                 numeric primitives (UFixed/IFixed, floats, aliases)
arvo-bitmask         fixed-width bitmask types, bit-matrix, set ops
arvo-sparse          sparse matrix (CSR, RCM, block diagonal, D-M)
arvo-graph           DAG algorithms (topo sort, components, rank, waist)
arvo-spectral        Laplacian, power iteration, Fiedler, bisection
arvo-comb            combinatorial optimisation (DP, greedy, bin-pack)
```

Dependency flow:
```
arvo                 (no deps within arvo — numerics are independent)
arvo-bitmask         (no deps within arvo)
arvo-sparse          (deps: arvo-bitmask, arvo)
arvo-graph           (deps: arvo-bitmask, arvo)
arvo-spectral        (deps: arvo-sparse)
arvo-comb            (deps: arvo-bitmask, arvo)
```

arvo-graph, arvo-sparse, and arvo-comb depend on arvo for weight and
cost types. The arvo-types-only lint applies to all arvo sub-crates:
weights use arvo fixed-point newtypes (not raw u32/usize), giving
validation and semantic clarity. After compilation these translate to
the same machine instructions as raw integers.

**arvo (numerics):**
- Fixed-point: `UFixed<I, F>` / `IFixed<I, F>` (repr(transparent) over
  u64/i64)
- Strategy markers: `Hot` (lane-capped), `Precise` (full-width),
  `Warm` (runtime)
- Float wrappers: `FastFloat` (LLVM intrinsics, fast-math),
  `StrictFloat` (IEEE 754)
- Small composable traits: `TotalOrd`, `Sqrt`, `Recip`, `Abs`,
  `FromConstant`
- Semantic aliases: `Ratio`, `Factor`, `Norm`, `Coord`, `Angle`,
  `Count`, `Average`, `Deviation`
- No monolithic Numeric trait — compose bounds: `T: Add + Mul + Sqrt`
- Source: T4 L1856-1924 §Q5a

**arvo-bitmask:**
- Fixed-width bitmask types: `Mask64(u64)`, `Mask256([u64; 4])`
- Bit-matrix adjacency: `[u64; N]` for ≤64 nodes
- Set operations: union, intersection, difference, complement
- Popcount, trailing/leading zeros, iterate set bits
- Dirty propagation (OR sequences): `predecessors[N] & dirty_mask`
- Cross-cutting foundation for all other arvo crates
- Source: T5 L1022-1041 (bitmask access sets), T5 L1079-1087
  (bit-matrix DAG adjacency), T5 L1173-1179 (dirty propagation)

**arvo-sparse:**
- CSR (compressed sparse row) format for >256 nodes
- Sparse binary matrix representation (access matrix)
- RCM reordering: bandwidth minimization, O(n+nnz)
- Block diagonal detection: permutation to block diagonal form
- Dulmage-Mendelsohn decomposition: structural rank, fine/coarse,
  dead column elimination
- SpMV (sparse matrix-vector multiply) — deferred but same family
- Source: T3 L1581-1820 (matrix methods)

**arvo-graph:**
- Topological sort (Kahn's): fixed-size queue + in-degree array
- Connected components: DFS/BFS on bitmask adjacency
- Longest/shortest weighted path
- Upward/downward rank computation: `rank(u) = cost(u) +
  max(rank(successors(u)))`
- Critical path analysis
- Waist detection: concurrent path counting in DAG topo order
- Node renumbering: sequential IDs matching reverse topo order
  (prefetcher-optimal access)
- Source: T5 L1044-1094 (renumbering, step ranges), T6 L2490-2501
  (spanning tree), T6 L2710-2712 (waist detection)

**arvo-spectral:**
- Laplacian construction from weighted adjacency: L = D - W
- Power iteration for eigenvector computation
- Fiedler vector (second-smallest eigenvalue of Laplacian)
- Spectral bisection and k-way partitioning (recursive bisection)
- Const-generic stack-allocated: `[[u8; 64]; 64]` weight matrix
  (4 KB), `[f32; 64]` eigenvector (256 bytes), 30 iterations
- Source: T3 L1752-1835, T6 L944-957

**arvo-comb:**
- Matrix chain ordering DP: optimal interval grouping under
  constraints. Cost function + feasibility check + DP recurrence.
- Greedy interval grouping (fusion walk): constrained sequential
  grouping with feasibility predicate.
- Bin-packing: greedy two-level (e.g. affinity-ordered bit
  assignment from domain 10).
- Source: T3 L1837-1937 (matrix chain DP)

**Relationship to hilavitkutin:**
- hilavitkutin uses arvo (numerics) for all internal numerics
- hilavitkutin uses arvo-bitmask for AccessMask, DAG adjacency,
  virtual flag storage
- hilavitkutin uses arvo-graph for topo sort, waist detection, rank
  computation, connected components
- hilavitkutin uses arvo-sparse for access matrix, RCM, block
  diagonal, Dulmage-Mendelsohn
- hilavitkutin uses arvo-spectral for trunk formation (Fiedler)
- hilavitkutin uses arvo-comb for fiber grouping DP, morsel bin-pack
- "Column data types come from arvo; hilavitkutin doesn't care what's
  in the columns, only the read/write dependency structure"
  (T5 L1228-1229)
- Boolean columns = UFixed<1,0> with 8x density in BitPacked layout
  (T5 L868)

**Build integration:**
- Arvo detects hardware capabilities in its own build.rs via
  `CARGO_CFG_TARGET_FEATURE` (not custom cfg flags from
  hilavitkutin-build). Only cfg from hilavitkutin-build: `arvo_fast_math`

- Arvo's impls select backend via target_feature (hardware intrinsic
  vs software approximation)
- MathPeephole LLVM pass fires on FastFloat operations
- arvo-types-only lint rejects non-arvo numerics in data-facing
  structs (T4 §Q8)

### 02. Platform tiers

**CONSOLIDATED.** No tensions. Primary source T5 §Q11, confirmed by T4 §Q6 and T6 §Q8.

**Three tiers:**
| Tier | Target | Memory | Threads | Clock | Panics |
|---|---|---|---|---|---|
| `no_os` | bare metal, WASM | Consumer provides | Consumer provides | Consumer provides | abort |
| `os` (default) | Linux, macOS | raw mmap / extern "C" | raw clone / pthread | raw clock_gettime | abort |
| `std` | Windows, other | std::alloc | std::thread | std::time::Instant | catch_unwind |

- `os` and `std` mutually exclusive. `os` gets optimisation investment.
- `no_os` = trait contracts only; consumer provides all platform impls.
- `alloc` crate is technically available in no_std mode but hilavitkutin
  does NOT use it. No Vec, Box, String. All dynamic storage via SlabColumn
  or stack arrays. Consumers may use alloc in their own code.

**Platform traits (in hilavitkutin-api):**
```rust
pub trait MemoryProvider: Send + Sync {
    unsafe fn allocate(&self, len: usize, align: usize) -> *mut u8;
    unsafe fn deallocate(&self, ptr: *mut u8, len: usize);
    unsafe fn protect(&self, ptr: *mut u8, len: usize, read: bool, write: bool);
}
pub trait ThreadPool: Send + Sync {
    fn spawn(&self, f: impl FnOnce() + Send + 'static);
    fn worker_count(&self) -> usize;
}
pub trait Clock: Send + Sync {
    fn now_ns(&self) -> u64;
}
```
Each tier ships defaults. Consumer overrides via scheduler builder.
Platform traits use `impl Trait` in argument position — each callsite
monomorphises. The Scheduler is generic over the platform types (not
`dyn`). Storing `dyn MemoryProvider` would violate the dyn ban.

**No alloc crate.** All dynamic storage through SlabColumn (mmap on os,
std::alloc on std, consumer-provided on no_os). Fixed-size structures
use stack arrays, arrayvec, or SlabColumn.

**Hardware detection:** CPUID (x86), sysctl (macOS ARM), sysfs (Linux ARM).
Fallback: 32 KB L1, 1 core. Detection runs once at startup.

**Sources:** T5 L877-958 §Q11 (primary), T4 L2048-2062 §Q6 (no_std/std),
T6 L584-654 §Q8 (trait definitions + os impls), T2 L119-124 §4.1 (detection)

### 03. Build-time optimisation

**CONSOLIDATED.** No tensions. T4 (build-configuration-and-cfg-gating)
is the single authoritative source (2453 lines, Q1-Q8). T5 and T6
reference it but add no conflicting decisions.

**What hilavitkutin-build is:**
- Shared build-dependency crate (`[build-dependencies]`), not an xtask
- Every crate's build.rs calls `hilavitkutin_build::configure().run()`
- Independent of hilavitkutin runtime — optimises HOW code is compiled

**Four-part build config model (T4 §Q1):**
1. hilavitkutin-build (build-dep): cfg flag emission
2. No custom cargo features (compiler provides target_feature, CARGO_CFG_*)
3. RUSTC_WORKSPACE_WRAPPER: all LLVM codegen flags per (target, tier)
4. compile_error! for mutually exclusive features

**Pragma system (T4 §Q4f):**
- Builder API: `configure().profile("release", |p| p.enable::<(LoopOptimization, FastMath, ExpandedLto, Bolt, Pgo)>()).run()`
- Built-in pragmas: LoopOptimization, Polly, MathPeephole, FastMath,
  ExpandedLto, Bolt, Pgo, Profiling, BuildStd, ParallelCodegen<N>,
  SharedGenerics, LoopFusion
- Pragma trait with type Requires for dependency enforcement
- Effects by mechanism: generated config (ExpandedLto, Pgo, BuildStd),
  rustc wrapper (LoopOptimization, Polly, MathPeephole, FastMath),
  post-build hook (Bolt, Profiling)

**LLVM passes (T4 §Q4):**
- 551 passes available in LLVM 22.1.0
- IR cleanup for Polly SCoP detection (IRCE, LoopPredication, SimplifyCFG)
- Pass registration at VectorizerStartEP + OptimizerLastEP
- Polly works on aarch64-linux (confirmed via Apple Container)

**PGO + BOLT workflows (T4 §Q5):**
- Shared benchmark runs produce both PGO (.profraw) and BOLT (perf.data)
- Automatic pipeline via rustc wrapper post-build hook
- BOLT Linux ELF only (no Mach-O); machine-function-split as macOS partial alternative
- Optimization tiers: Stock → Plugins → Static BOLT → PGO → Profile-guided BOLT

**Five profiles (T4 §Q3):**
- dev (Cranelift for workspace, LLVM for deps)
- dev-opt (LLVM, opt-level=2, LoopOptimization)
- release (opt-level=3, full pragmas)
- profiling (inherits release, keeps debug info)
- ci (inherits release, thin LTO for caching)

**Critical for hilavitkutin execution model (T6):**
- ExpandedLto pragma (fat LTO + codegen-units=1) is REQUIRED for
  LLVM to devirtualise monomorphised tuple dispatch. Without it,
  struct-field fn arrays don't devirtualise (12.6x slower).

**Sources:** T4 entire file (authoritative), T5 L1181-1216 (instruction
cache grouping + MimallocAllocator pragma), T6 L815-844 (ExpandedLto
required for devirt)


#### — Level 1: Data representation (depends on L0) —

### 04. Column memory layout

**CONSOLIDATED.** Stride tension resolved. Co-location error from
initial pass corrected (was: "per-column arrays, not co-located").
Vocabulary pass applied (chain→fiber, row→record). Content verified
against full topic read.

**Resolution R3: type-native stride, no flags byte.**
- Stride = `size_of::<T>()` per column. Each column has its own stride.
- No per-entry flags byte. Null/tombstone handled via separate bitmask
  per column (same pattern as fences moving to external progress counters).
- Value constraint becomes `size_of::<T>() <= 16` (was 15 with flags byte).
  Full 16 bytes usable. Clean power-of-two, fits u128, aligns with SIMD.

**Supersedes:** T1 decisions #80-81 (128-bit tagged stride), T2 decision #1
(universal 128-bit), T3 decision #90 (stride IS the SIMD lane). The
T6 bench finding (type-native outperforms 128-bit) is now authoritative.

**What remains from the original design:**
- Contiguous columnar storage (all topics agree)
- Co-located arena within fibers (T6 L1770-1777): all columns in a
  fiber share one arena allocation with known static offsets. One base
  pointer per morsel, columns at `base + col_offset + i * stride`.
  Columns across different fibers are separate allocations.
- 64-byte alignment on column array base addresses (T3 §Q3 L2825-2860)
- Sequential access pattern, prefetcher-optimal (all topics agree)
- Column layout from scheduling hints: ByteAligned/Natural/BitPacked
  still applies per column (T5 §Q10 L840-870), just without the
  uniform stride assumption

**What changes downstream:**
- Morsel formula: `L1_usable / (Σ size_of::<T_i>() for fiber columns)`
  instead of `L1_usable / (col_count * 16)`
- Payload/SlotSize (domain 05): Slot15 → Slot16, sealed set adjusts
- 15-byte compile-time constraint → 16-byte
- T3's variable stride option is now the default (each column has its
  own stride = size_of::<T>())

**Sources:** T6 L1597-1606 (type-native supersedes), T1 L848-876
(original design, superseded), T2 L15-28 + L561-572 (universal stride,
superseded), T3 L2825-2860 (64-byte alignment, retained), T3 L3305-3377
(variable stride analysis, now default), T5 L840-870 (column layout
from hints, retained)

### 05. Column value types

**CONSOLIDATED.** Simplified from T3's Payload/SlotSize system following
R3 (type-native stride, no flags byte, 16-byte max).

**Resolution R4: ColumnValue trait with blanket impl + specialisation.**

The sealed SlotSize system (T3 §Q3 L3100-3377) is dropped. With type-
native stride, pack/unpack is identity — the domain type IS the storage
type directly. The Payload trait simplifies to ColumnValue:

```rust
pub trait ColumnValue: Copy + 'static {
    const BIT_WIDTH: usize;  // no default on trait itself
}

// Blanket impl via min_specialization: everything Copy + 'static + ≤16 bytes
impl<T: Copy + 'static> ColumnValue for T {
    default const BIT_WIDTH: usize = core::mem::size_of::<Self>() * 8;
}

// Specialisation for sub-byte types (must explicitly state BIT_WIDTH)
impl ColumnValue for UFixed<1, 0> {
    const BIT_WIDTH: usize = 1;
}
```

**Key design decisions:**
- **Blanket impl:** any `Copy + 'static` type ≤16 bytes is column-storable.
  Zero friction for consumers. `Column<u32>` just works.
- **min_specialization** (nightly, T4 §Q6): blanket provides `default const
  BIT_WIDTH`. Specific types specialise to override. Trait itself has NO
  default — specialising impls must state BIT_WIDTH explicitly, making the
  override visible and intentional.
- **BIT_WIDTH** informs the storage engine for bitpacking. Default =
  `size_of * 8` (byte-aligned). Sub-byte arvo types override (UFixed<1,0>
  → 1 bit, UFixed<4,0> → 4 bits). Storage uses BIT_WIDTH in BitPacked
  column layout mode (T5 §Q10).
- **No raw primitives** is our internal lint (arvo-types-only, T4 §Q8),
  NOT enforced on consumers. Consumers store whatever they want.
- **16-byte max** (was 15 with flags byte). Enforced at compile time on the
  blanket impl's where clause. Forces interned IDs for String/PathBuf/Vec.
- **repr(C)** on domain newtypes enforced by `#[derive(ColumnValue)]` macro
  or lint, not by the trait itself.
- **No pack/unpack.** Type IS the storage. No SlotSize intermediary.

**What this supersedes:**
- T3 decisions #83-87 (Payload trait, sealed SlotSize, Slot1-15) → replaced
  by ColumnValue with blanket impl
- T3 decision #90 (stride IS the SIMD lane) → superseded by R3
- T1 decisions #82-83 (15-byte constraint) → updated to 16-byte
- T4 §Q6 `Check<size_of ≤ 15>` pattern → updated to ≤ 16

**What is retained:**
- Domain newtypes for our crates (arvo-types-only lint, T4 §Q8)
- Copy + 'static requirement (T3 L2793-2823)
- Compile-time size enforcement
- repr(C) on stored types (T3 §Q3)
- Column<In<IR>, As<T>> is consumer-side (R1); hilavitkutin sees Column<T>

**Sources:** T3 L2793-2823 + L3100-3377 §Q3 (original Payload design,
superseded), T4 L1993-2000 §Q6 (min_specialization), T5 L868-870 §Q10
(sub-byte density), R3 (type-native stride, 16-byte max)


#### — Level 2: Type system and storage (depends on L0-L1) —

### 06. Store types

**CONSOLIDATED.** No tensions. T5 §Q1 defines, T6 §Q1 elaborates.

**Three store types, one scheduling mechanism (T5 L198-208):**
```rust
pub struct Resource<T>(PhantomData<T>);  // singleton, one value
pub struct Column<T>(PhantomData<T>);    // collection, N values, morsel-chunked
pub struct Virtual<T>(PhantomData<T>);   // zero-data, DAG edge only
```

- Scheduler sees StoreId pairs for edge construction. Does not care
  about cardinality. Write<Resource<X>> → Read<Resource<X>> builds
  the same edge as Write<Column<Y>> → Read<Column<Y>>.
- Column<T> where T: ColumnValue (R4). No IR parameter (R1).
- Resource<T>: singleton, accessed via ctx.resource::<T>() → &T
- Column<T>: collection, morsel-scoped access via raw pointers (cu/cw)
- Virtual<T>: no storage, fire-only via ctx.fire::<T>()

**StoreId and AccessMask (T5 L1022-1041, T6 L79-137):**
- `StoreId(u16)` — dense index assigned at plan time
- `AccessMask` — arvo-bitmask type (`Mask64` for ≤64 stores,
  `Mask256` for ≤256 stores)
- All scheduler ops = single-instruction bitwise: AND for conflict,
  OR for union, `== 0` for empty check
- Bit position assignment computed once at plan time

**AccessSet (T5 L270-271, T6 L53-77):**
- Trait on tuples: `(Column<PosX>, Column<PosY>, Resource<Gravity>)`
  implements AccessSet
- Compile-time store descriptors. Plan phase converts to runtime
  AccessMask via monomorphised generic functions (the monomorphised
  function IS the type identity — no TypeId, no std::any)

**ZST erasure (T2 L810-814):**
Column marker types (ZSTs) constrain the DAG, enforce type safety,
and determine morsel sizes at compile time. Erased to plain usize
indices by the time execution begins. The Schedule lowers types to
indices.

**Sources:** T5 L198-208 §Q1 (primary), T6 L21-137 §Q1 (elaboration),
T5 L1022-1041 §plan-phase (bitmask access sets), T2 L810-814 §8.9
(ZST erasure)

### 07. Column storage

**CONSOLIDATED.** ColumnStorage trait and resource storage model resolved.

**Resolution R5: Resource model — scalars + const-sized collections.**

Resources consist of three field types only:
- `Field<T>` where T: ColumnValue (≤16 bytes) — scalar, stack-local
  cached, LLVM promotes to registers. Only accessed fields loaded.
- `Seq<T, const N: usize>` — const-sized array in separate arena.
  Stack-local copied if small, L2-hinted if read-only.
- `Map<K, V, const N: usize>` — const-sized map in separate arena.
  Same caching pattern as Seq.

No dynamic collections. Allocation size is const generic (compile-time).
Element values in Seq/Map are NOT constrained to 16 bytes — only
Field<T> has the ColumnValue limit.

**Morsel sizing for resource collections:**
- Write resource collections count toward L1 morsel budget
- Read-only resource collections do NOT (ride L2 prefetcher)
- Consistent with write-only morsel sizing principle from R3

### 10. Virtual flag system

**CONSOLIDATED.** No tensions. T5 §Q1 + §Q3 defines, T6 §Q5 elaborates.

**Bit-packed fire flags (T5 L219-251):**
- One bit per registered Virtual type
- Packed into u8 bytes, grouped into u64 words (arvo-bitmask types)
- Hierarchical check: u64 == 0 → skip 64, u8 == 0 → skip 8
- Fire: `ctx.fire::<T>()` — one byte load, one OR, one byte store
- T::INDEX const-folded from compile-time type identity

**Affinity-ordered bit assignment (T5 L233-247):**
- Co-firing virtuals packed into same byte/word (plan time)
- Two-level greedy bin-packing: first u64 words, then bytes
- Cold words stay 0 in connector-heavy passes
- Computed once at plan time, zero runtime cost

**Per-(virtual, consumer) bit tracking (T5 L511-548):**
- Each consumer has its own bit for each virtual it gates on
- Fire sets ALL consumer bits for that virtual
- Clear-on-dispatch: bit cleared on first dispatch, not resume/completion
- Re-fire during execution → consumer runs again next pass
- No lost fires

**Epoch-based reset (T5 L1124-1128, T6 L444-454):**
- Global epoch u8, incremented per pass
- "Flag is set" = flag_value == current_epoch
- Reset = one u8 increment (not memset)
- Full clear every 256 passes (epoch wraparound)

**Storage:** ~96 bits = 12 bytes for 64 virtuals × 1.5 consumers avg.
One cache line. fire_external: atomic store relaxed for cross-thread
signalling (T5 L541). Internal fire: non-atomic (T5 L542).

**Virtuals are pure flags, not data carriers (T5 L490).** Data goes
in Column or Resource alongside the Virtual.

**Signal-ordered execution theory (T6 L3507-3585):** record reordering
by signal state for branch-free loops. Theoretical — not bench-validated.

### 11. Cache hierarchy, tiling, and phase decomposition

**CONSOLIDATED. Merged with former domain 18 (phase decomposition).**
T3 cache principles + T6 phase decomposition = one model.

**Phase decomposition (T6 L2661-2864, bench-validated):**

Vocabulary:
- **Phase:** wide DAG section between waist points (concurrent path
  count minimums). Phases execute with pipeline parallelism.
- **Trunk:** sequential critical path within a phase. Shares NO write
  columns with other trunks. Zero sync between trunks.
- **Branch:** side path attached to trunk, shares some columns.
  Executes as chaser within trunk's morsel scope.
- **Bridge:** reads from multiple trunks (fan-in). Runs after parent
  trunks reach required record range.

Algorithm (all at plan time, zero runtime cost):
1. Build DAG from WU read/write sets (arvo-bitmask for adjacency)
2. Detect phases via waist analysis (arvo-graph waist detection)
3. Identify trunks per phase (arvo-spectral for >5 fibers, else
   arvo-graph connected components on column-disjoint paths)
4. Assign branches and bridges
5. Form fibers within each trunk (arvo-comb greedy grouping +
   arvo-sparse RCM for ordering, per L1 + register budget)
6. Size morsels per fiber
7. Select execution strategy per phase

**Cache principles (T3 L330-701, retained):**
- Write columns MUST be L1 (T3 decision #43)
- Read-only columns overflow to L2 (prefetcher ~80-90% L1 throughput)
- Column temperature: write-only, read-write, read-only-dense,
  read-only-sparse, read-only-shared (T3 decision #50)
- Column layout from hints: ByteAligned/Natural/BitPacked (T5 §Q10)
- Register file (~14 aarch64, ~10 x86) is binding constraint for
  fiber width, not L1 capacity (T6 bench)

**Execution strategies (T6, bench-validated):**
- Multi-trunk phases: trunks run in parallel on separate cores
- Single-trunk phases: head+tail convergence (2 threads, opposite
  ends, ~2x parallelism). Skip for non-commutative resource accumulation.
- Pipelined phases: phases OVERLAP via AtomicUsize progress counters.
  Phase N+1 starts when N produces one morsel. Total time ≈ max(phases)
  + fill latency (not sum of phases).
- Core-pinned trunks: each trunk assigned to specific pool thread at
  plan time (warm L1). Leftover threads do convergence → branches →
  bridges in priority order.
- Pre-allocated thread pool with hybrid wake (spin 128 → park).
  Pool eliminates small-N catastrophe (crossover ~2K records vs ~50K).

**Shared read columns between trunks (T6 L3635-3670):**
- Approach 1 (snapshot-to-local): copy shared columns per trunk at
  phase transition. 1.6% overhead. Simpler, no cross-core contention.
- Approach 2 (aligned morsel sync): zero copy, both trunks read same
  data. Works on unified-memory ARM (M1). Complex on NUMA x86.
- Recommendation: approach 1 for ARM, approach 2 for x86 NUMA.

**Column-count-driven phase strategy (T6 L3696-3737):**
- fused_cols < 50% max_cols → FUSE (one trunk, large morsels)
- fused_cols 50-100% max_cols → CONSIDER SPLIT (if balanced + cores)
- fused_cols > max_cols → MUST SPLIT

**Bench results (T6, Game World 1M agents):**
| strategy | vs seq | notes |
|---|---|---|
| sequential | 1.00x | baseline |
| adaptive | 0.93x | flat chase+steal, barely helps deep pipeline |
| phased | 0.82x-0.89x | trunk parallelism |
| phased+convergence | 0.68x | halves serial bottleneck phase |
| phased-pool | 0.68x-0.69x | consistent across all sizes (2K-1M) |
| phased-conv (spawn) | 0.62x at 1M | but 15.54x SLOWER at 200 records |

  fiber → trunk → phase. Levels 0-1 (entry, morsel) and 5 (pipeline)

- T2 decisions #9, #12, #13, #14 — superseded by T3 decision #51

### 12. Morsel model

**CONSOLIDATED.** No tensions. Additive evolution T1→T2→T3→T5→T6.

**Morsel sizing formula (with R3 type-native stride, R5 resource model):**
```
morsel_size = (L1_usable / Σ write_sizes).clamp(MIN_MORSEL, MAX_MORSEL) & !3
```
- L1_usable = detected_L1 × 0.75
- write_sizes = Σ size_of::<T_i>() for write columns + write resource

  collections ride L2 prefetcher, don't consume L1 budget.
- MIN_MORSEL configurable (not a design decision — R3 note)
- MAX_MORSEL = 8192
- Align to multiple of 4 (cache line alignment at any stride)
- Per-fiber morsel size (each fiber has its own)
- Morsel vs scalar dispatch inferred from AccessSet: Column<T> in
  Read/Write → morsel-chunked, Resource/Virtual only → scalar (T5 §Q4)

**Two-level morsel (T3 decision #98, single-level by default):**
- Morsel = L1 scheduling unit (outer loop)
- Micro-morsel = inner tiling when peak_live > L1 (T3 decision #60)
- peak_live = max(op.read ∪ op.write ∪ prev.write) at any Z-slice
- Single-level sufficient for polka/saalis. Micro-morsel activates
  at ECS scale (hundreds of columns per fiber).

**3D morsel cube (T3 decisions #52-58):**
- X (records) × Y (columns) × T (temporal across frames)
- Inner cube: x (micro-morsel) × y (active cols per op) × z (op index)
- Schedule reusable across frames (construct once, execute repeatedly)
- Morsel-to-core affinity pinned across frames (warm L2)
- Stable morsel boundaries (same record ranges unless record count changes)

**Change detection (T3 decisions #57-58, R2 evict/inject):**
- Per-morsel generation counter — bump on write, skip if unchanged
- Propagates through DAG — unchanged root skips all transitive dependents
- Transforms pipeline from batch processor to incremental processor
- evict/inject APIs (R2) handle persistence to consumer storage

**Dispatch integration (T5):**
- Divisibility hint: Atomic=all morsels at once, Interruptible=one
  at a time with steal opportunities, Adaptive=EMA-based (T5 §Q2)
- Pre-partitioned morsel ranges at plan time (no atomic fetch_add)
- Morsel boundaries baked into compiled per-core programs (→ domain 17)

### 14. Fiber formation

**CONSOLIDATED.** Replaces "chain formation." Vocabulary updated: "chain"
→ "fiber," "chain group" and "partition" are dead terms (replaced by
trunk and phase respectively).

**Vocabulary:**
```
entry → micro-morsel → morsel → fiber ↔ branch ↔ bridge → trunk → phase ↔ waist → core → pipeline
```
- **Fiber:** the unit of column co-location and morsel windowing. WUs
  within a fiber share columns, and morsels window into the fiber's
  co-located arena. Whether those ops are fused (one loop body) or run
  as separate passes over the same morsel is an execution strategy
  decision (domain 17), not a definitional property of the fiber.
- **Branch:** side path attached to a trunk (peer of fiber).
- **Bridge:** fan-in node reading from multiple trunks (peer of fiber).
- **Trunk:** column-disjoint maximal path within a phase. Contains
  fibers. Trunks share NO write columns with other trunks — zero sync.
- Fibers, branches, and bridges are peers within a trunk.

**What T1's "archetype" became:** dead concept. Ops sharing columns
don't "run consecutively for cache reuse" — they form a fiber. The
archetype concept is fully subsumed.

**Co-located arena (T6 L1770-1777):**

All columns within a fiber share one arena allocation. The morsel
context has one base pointer. Each column at a known static offset:
```
arena_base + col_A_offset + i * stride_A
arena_base + col_B_offset + i * stride_B
```
Record index `i` advances one cursor covering all columns. Morsel boundary
= one base pointer update. Co-location means:
- All fiber data in one cache-line neighborhood
- Fiber-internal columns are register-to-register data paths (DSE)
- Fiber outputs are the only columns written to memory
- One base register + offset immediates, not N separate pointers

Column layout uses type-native stride (R3, domain 04): `size_of::<T>()`
per column. 64-byte alignment on column array bases.

**Arena addressing — RESOLVED (bench-validated, no constraint needed).**

The concern was that co-located arenas exceeding the aarch64 12-bit
scaled immediate range (~32KB for 8-byte accesses) would force LLVM
to use extra offset registers, degrading codegen. Benchmarked three
access patterns at 2-30 columns, 1M records, morsel=1024, 30 reps,
fat LTO, codegen-units=1 (Apple Silicon M1):

**Three patterns tested:**
- **arena**: `base + col * rows + i` (runtime multiply per access)
- **separate**: `col_ptrs[N].add(i)` (one pointer per column)
- **precomputed**: resolve `col_ptr = base + col * rows` once at morsel
  boundary, inner loop does `col_ptr.add(i)`. This IS the real fiber
  model — the dispatcher resolves column pointers from the arena base
  at each morsel boundary, then the inner loop is pointer arithmetic.

**Results (pre-computed vs separate pointers, const-generic N):**

| read_cols | precomp ns/el | separate ns/el | ratio | morsel arena |
|---|---|---|---|---|
| 2 | 0.42 | 0.63 | **0.67x** | 24 KB |
| 4 | 1.36 | 1.17 | 1.17x | 40 KB |
| 8 | 2.95 | 2.89 | 1.02x | 72 KB |
| 14 | 5.94 | 6.57 | **0.90x** | 120 KB |
| 20 | 10.53 | 11.00 | **0.96x** | 168 KB |
| 30 | 24.58 | 23.50 | 1.05x | 248 KB |

**Key findings:**

1. **No addressing mode cliff.** Performance varies ±17% across all
   column counts with no sudden degradation. At 120 KB morsel arena
   (14 cols, 4x the 32KB immediate range), the precomputed model is
   10% FASTER than separate pointers. At 248 KB (30 cols, 8x the
   range), it's only 5% slower.

2. **LLVM handles it correctly.** When column offsets exceed the
   immediate range, LLVM pre-computes column pointers from the arena
   base and uses `ptr.add(i)` in the inner loop — exactly the
   precomputed pattern. The addressing mode transition is invisible
   at the performance level.

3. **Co-location wins on cache.** At 14+ columns, the precomputed
   arena model beats separate pointers (0.90-0.96x) because all
   columns share one allocation = fewer TLB entries, better hardware
   prefetch across columns in the same cache neighbourhood.

4. **Dynamic column count: arena always wins.** When LLVM cannot see
   the exact column count (runtime loop), the arena model is 6-15%
   faster at every column count tested (0.85-0.97x). This is because
   `base + col * rows + i` compiles to one multiply-add vs loading a
   pointer from a pointer array.

5. **At 2 columns: arena is 33% faster** in both const-generic and
   precomputed models. Small fibers benefit most from co-location
   (fewer cache lines touched total).

**Resolution:** the arena addressing concern is a non-issue. The
holistic feasibility check does NOT need an arena size threshold. The
existing constraints (register file budget, L1 write budget, L1+L2
total, fan-in, pipeline breakers) are sufficient. The co-located arena
with pre-computed column pointers at morsel boundaries is the correct
model. No additional fiber formation constraint needed.

**Bench source:** `mock/research/bench/src/arena_addressing.rs`

**Algorithm:**

The access matrix (T3 L1590-1627) is the unifying representation. All
pipeline operations map to arvo algorithms (see domain 15):
- Fiber formation = bandwidth minimization (arvo-sparse RCM, always
  run, O(n+nnz))
- Phase identification = block diagonal detection (arvo-sparse
  Dulmage-Mendelsohn, always run)
- Trunk formation = spectral graph analysis (arvo-spectral Fiedler,
  when >5 fibers)
- Optimal grouping = matrix chain DP (arvo-comb, when >10 ops)

Core algorithm: arvo-graph Kahn's topological sort → arvo-comb greedy
grouping walk (T2 §8.4, T6 §Q12, T7 §1.3):
1. Build DAG: write→read edges (arvo-bitmask adjacency). Write-write
   ambiguity resolved by scheduling hints (Urgency, Divisibility,
   Significance — domain 09). Higher-priority hint wins. Tie-break:
   most deps first, then deterministic fallback (type name or similar).
2. Topological sort via arvo-graph Kahn's (fixed `[u16; MU]` queue +
   in-degree array, zero heap).
3. Walk topo order. For each WU: if all predecessors are in one fiber
   AND adding this WU passes the holistic feasibility check → group
   into that fiber. Otherwise → new fiber with fence dependencies on
   upstream fiber(s).
4. Compute inter-fiber deps from WU-level predecessors.

Validated: 17-WU game world → 12 fibers (T6 bench).

**Holistic feasibility check (one calculation):**

All compete for the same resources:

Register file: co-located arena base pointer + column offset immediates
+ write resource pointers (R5 Field<T>) + loop counter + intermediates
+ constants ≤ available GPRs. aarch64: ~14 usable. x86-64: ~10.
Co-location reduces column pointer pressure (one base + offsets vs N
separate pointers). Register file is the binding constraint, not L1
capacity (T6 L3428-3505, T7 §1.1).

L1 write budget: Σ write column sizes (`size_of::<T_i>()`, R3) +
Σ write resource collection sizes (R5: Seq/Map const generic N) must
fit L1 write budget. Read-only columns and read-only resource
collections ride L2 prefetcher (domain 11, domain 07). Write-only
morsel sizing from domain 12.

L1+L2 total: all accessed data must fit L1+L2 combined.

No fan-in: all predecessors in exactly one fiber. Multi-fiber
predecessors → new fiber (mandatory barrier = bridge).

No pipeline breaker: ops requiring all input records before output →
materialization boundary.

**Column classification:**

The fiber grouper classifies each column statically from WorkUnit
Read/Write declarations:
- **Fiber-internal:** written by one WU, read only by the next WU in
  the same fiber, not read externally. Register-to-register data path.
  DSE skips memory ops. The fiber as pure function pipeline:
  `d(c(b(a(input_n))))`.
- **Fiber-input:** written by upstream fiber/trunk. Read from memory at
  morsel start.
- **Fiber-output:** read by downstream fiber or final consumer. Written
  to memory at morsel end.
- **Fiber-internal AND fiber-output:** local reused within fiber
  (register) AND stored for downstream.

Feeds the codegen flattener (domain 17): read inputs → pure function
pipeline with locals → output stores grouped at end. DSE: 0.95-0.96x
baseline.

**Column temperature (from domain 11):**

| Access pattern | Cache | Budget |
|---|---|---|
| Write-only | L1 during morsel | L1 write |
| Read-write | L1 during morsel | L1 write |
| Read-only dense | L2 prefetched | L2 read |
| Read-only sparse | L3 tolerable | Minimal |
| Read-only shared | Shared L2/L3 | L2 amortised |

**Unique column counting:**

Fiber's column budget uses the unique set across all grouped WUs, not
the sum. Dedup via Read+Write tuple types (each column type is unique
by definition). The dedup matters; mechanism is implementation detail.

**Spanning tree decomposition:**

Operates within each phase (waist boundaries from domain 11).
1. Weight each WU: `col_reads + col_writes`
2. Longest weighted path = trunk (critical path)
3. Fan-out: trunk follows heaviest child, others become branches
4. Fan-in: record bridge
5. Recurse per branch

Natural barriers (fan-in) = mandatory fiber break points. Between
consecutive barriers: candidate fiber segments. Walking back from
leaves → forced fibers. Walking back from barriers → maximum fiber
count. The grouper decides which segments to group and which to keep
separate.

**Per-phase adaptive configs:**

The grouper produces multiple configs per phase:
- **MAX_FUSE:** group everything that fits. Fewest fibers, best cache.
  Phases with a clear mega-trunk benefit most: sequential pass +
  head+tail convergence beat splitting even with many cores.
- **BALANCED:** split at compute balance points. Weight proxy: WU count
  × column accesses. ~10% extra data walk, ~30% parallelism gain.
- **MAX_SPLIT:** every barrier-to-barrier segment is its own fiber.
  Maximum parallelism.

Selection per phase based on topology. Domain 21 for selection logic.

Column-count heuristic (T6 L3696-3737):
- fused_cols < 50% max_cols → FUSE
- 50-100% → consider split if balanced + cores + record count
- \> max_cols → must split

**Dispatch constraints:**

- Local `&[fn]` slices devirtualise; struct-field arrays don't (12.6x).
  No-alloc load-bearing. ExpandedLto REQUIRED (domain 03).
- Each fiber → monomorphised dispatch with LOCAL `&[WuFn]` slice.
- <10K records: IndirectPerFiber or TrunkMega. >10K: ScheduleMega.
- Deep single-fiber: flattener + DSE (rust-pipe, domain 17).

**Commutativity (from domain 08):**

`COMMUTATIVE: bool` on WU. Fiber is commutative only if ALL WUs are.
Commutative fibers → head+tail convergence + multi-core morsel
dispatch. Resource accumulation: per-thread stack-local accumulator,
merged after. Non-commutative resource accumulation → skip head+tail.

**WU authoring guideline:** ≤8 unique columns per WU. Guideline, not
constraint.

**What WUs must declare:**
1. Read column set (`type Read: AccessSet`)
2. Write column set (`type Write: AccessSet`)
3. Scheduling hints (`type Hint: SchedulingHint` — Urgency,

### 15. Pipeline analysis algorithms

**CONSOLIDATED.** No tensions. T3 defines the algorithms, T5 provides
data structure designs, T6 validates implementations. Vocabulary
updated (partition→phase, chain→fiber, chain group→trunk).

These algorithms are not hilavitkutin-specific. They are reusable
mathematical primitives that live in arvo crates (domain 01). Define
once with the same no_std/no_alloc/const-generic/LLVM-optimised
discipline. hilavitkutin consumes them; so can saalis, loimu, anything.

**The access matrix is the unifying representation** (T3 decision #66).
Matrix A where rows = WUs, columns = stores. All pipeline operations
map to matrix/graph operations provided by arvo crates:

| Pipeline operation | arvo crate | Algorithm |
|---|---|---|
| DAG construction | arvo-bitmask | Bitmask adjacency from Read/Write overlap |
| Topological sort | arvo-graph | Kahn's with fixed-size queue |
| Node renumbering | arvo-graph | Sequential IDs for prefetcher-optimal access |
| Upward rank | arvo-graph | Bottom-up critical path computation |
| Waist detection | arvo-graph | Concurrent path counting in topo order |
| Fiber formation | arvo-sparse | RCM bandwidth minimization |
| Phase identification | arvo-sparse | Block diagonal + Dulmage-Mendelsohn |
| Dead column elimination | arvo-sparse | D-M horizontal/vertical components |
| Trunk formation | arvo-spectral | Fiedler vector spectral bisection |
| Optimal fiber grouping | arvo-comb | Matrix chain ordering DP |
| Greedy fiber grouping | arvo-comb | Constrained interval grouping |
| Bit assignment | arvo-comb | Two-level greedy bin-packing |

**How hilavitkutin applies them (plan-time pipeline):**

The entire plan-time pipeline runs once at schedule construction (or
when the WU set changes). Zero runtime cost. Every step operates on
the same access matrix built from WU declarations.

**Step 1: Build the access matrix** (arvo-bitmask)

Each WU's `type Read: AccessSet` and `type Write: AccessSet` tuple
types are lowered to `AccessMask` bitmasks at registration. The
access matrix is the collection of these masks:
- `read_masks[i]`: bit j set if WU i reads store j
- `write_masks[i]`: bit j set if WU i writes store j

DAG edges: for every pair (i, j), `(write_masks[i] & read_masks[j])
!= 0` → WU i must precede WU j. This is a single AND + compare per
pair. O(N²) on u64 values. For 50 WUs: 2500 ANDs, ~1µs. The result
is a bit-matrix DAG: `dag[i] |= 1 << j` where i precedes j.

Write-write to the same store: both WUs have the bit set in their
write masks. Ordering resolved by scheduling hint priority (domain
09): compare Urgency, then Divisibility, then Significance. Tie-break:
most downstream dependents first, then deterministic fallback.

**Step 2: Topological sort + renumbering** (arvo-graph)

Kahn's on the bit-matrix DAG. After sort, renumber all WU IDs so
they match reverse topological order. This means upward rank
traversal (step 3) accesses the rank array sequentially — the
hardware prefetcher sees a sequential scan, not random access. This
is the same principle as column co-location (domain 04): make access
patterns sequential for the prefetcher.

**Step 3: Upward rank → critical path → trunk skeleton** (arvo-graph)

`rank(u) = cost(u) + max(rank(successors(u)))`. Bottom-up in reverse
topo order. The WU with the highest rank is the trunk root. The trunk
is the path following the highest-ranked successor at each step — this
is the critical path, the serial bottleneck of the phase. Everything
else is branches or bridges.

Weight per WU: `Σ size_of::<T_i>()` for all accessed columns. This
approximates per-element compute cost because more column data = more
memory ops = more time. The critical path computed this way tells us
which sequence of WUs dominates execution time and should receive
priority treatment (core pinning, head+tail convergence).

**Step 4: Waist detection → phase boundaries** (arvo-graph)

Walk topo order. At each level (set of WUs at the same topo depth),
count how many independent paths are alive. A "path" is a WU that
has at least one unsatisfied downstream dependency. Where the count
drops to a local minimum = waist = phase boundary.

This is different from block diagonal (step 6): block diagonal finds
fully independent sub-problems (no shared columns at all). Waist
detection finds narrow bottleneck points WITHIN a connected DAG. A
DAG can be one connected component but still have waists where all
paths converge through a few WUs before fanning out again.

The game world example (T6 L2716-2727): waists at orders 1-2
(integ_vel/integ_pos) and 5 (ai_decide). Three phases:
- Phase A: roots → integ_pos (3 paths → 1 waist)
- Phase B: collision + terrain → ai_decide (2 paths → 1 waist)
- Phase C: movement + resources + rendering (3 paths → leaves)

Each phase gets its own execution strategy (domain 11). Phases
overlap via pipeline parallelism (AtomicUsize progress counters).

**Step 5: RCM reordering → fiber grouping order + arena layout**
(arvo-sparse, always run)

RCM operates on the access matrix and produces two orderings:

Row reordering (WU execution order): RCM reorders WUs so that ops
accessing similar column sets are adjacent. When the greedy fiber
grouping walk (step 8) processes WUs in this order, it encounters
column-sharing WUs in sequence, producing better fiber groupings
than a naive topo order. For wide DAGs with fan-out (many WUs at
the same topo depth), RCM determines which of those equal-depth
WUs get grouped first.

Column reordering (arena memory layout): RCM reorders columns so
that co-accessed columns have adjacent offsets within the co-located
arena (domain 04). When a fiber's morsel walks `arena_base +
col_offset + i * stride`, columns that are read together end up in
adjacent cache lines. The prefetcher brings in both at once. This
is the physical manifestation of "co-location" — RCM determines the
actual byte offsets of columns within the arena.

**Step 6: Block diagonal + Dulmage-Mendelsohn → phase validation +
dead column elimination** (arvo-sparse, always run)

Build bipartite graph (WUs on one side, columns on other side, edges
from access matrix). Connected components = independently schedulable
groups. These should align with the phases from step 4 — if they
don't, something is wrong with the waist detection.

Dulmage-Mendelsohn refines the block structure:
- Horizontal components: WUs whose output columns are never read by
  any other WU. These are dead-end WUs — their output can be
  eliminated entirely unless it's a final consumer output.
- Vertical components: columns that are only inputs (loaded by root
  WUs, never written by transform WUs). These are read-only for all
  phases after loading.
- Square components: matched WU-column pairs where every WU's output
  is consumed. The core of the pipeline.

Dead column elimination directly reduces the column count in fibers,
which means larger morsels (domain 12 formula) and less register
pressure (domain 14 holistic check). A column that no downstream WU
reads is a wasted store — the fiber grouper marks it as eliminable,
and the codegen flattener (domain 17) skips the store entirely.

**Step 7: Spectral partitioning → trunk formation** (arvo-spectral,
when >5 fibers)

After fiber grouping (step 8, which runs first for the initial
greedy pass), build a weighted graph where:
- Nodes = fibers
- Edge weight = Σ size_of::<T_i>() for columns shared between two
  fibers (the data volume that would need L2/L3 coherence if the
  fibers ran on different cores)

The Fiedler vector bisects this graph to minimize cut weight — the
data volume crossing the trunk boundary. The result: trunks where
fibers within a trunk share heavy column sets (kept on one core's
L1), and the cut between trunks is minimal (only read-only columns
from previous phases, or lightweight bridge columns).

Trunks share NO write columns with other trunks (zero sync). They
MAY share read-only columns from previous phase output. The spectral
cut ensures the shared read volume is minimal — trunks that need
the same read columns end up in the same trunk when possible.

For ≤5 fibers: single trunk (spectral is overkill). The waist
detection already identified phase boundaries; within a small phase,
all fibers form one trunk.

**Step 8: Fiber grouping** (arvo-comb)

Two modes depending on WU count:

Greedy grouping (≤10 ops, or as fallback): the walk from domain 14.
Process WUs in RCM-reordered topo order (step 5). For each WU: if
all predecessors are in one fiber AND the holistic feasibility check
passes → group into that fiber. Otherwise → new fiber. The
feasibility predicate is the full holistic check from domain 14
(register file + L1 write budget + L1+L2 total + no fan-in + no
pipeline breaker).

Matrix chain DP (>10 ops): explores the full solution space. Cost
function: `cost(i,j) = record_count × Σ size_of::<T_k>() for union columns
of ops i..j`. This uses the actual type-native strides (R3) and the
co-located arena model — the cost IS the memory bandwidth consumed by
one data walk through the fiber's arena. Feasibility uses the same
holistic check. DP finds the grouping that minimizes total data walks
across all fibers.

**Step 9: Dirty propagation → incremental skip** (arvo-bitmask)

At runtime (not plan-time), before each pass: check generation
counters on root WU inputs. If unchanged since last pass, set
dirty bit to 0. Propagate through the DAG:
`dirty[j] |= dirty[i]` for every edge i→j. A WU whose entire
predecessor set is clean (all dirty bits 0) skips execution entirely.

The propagation is a bitmask OR sequence: `predecessors[N] & dirty_mask`.
One AND per WU. For 50 WUs: 50 AND operations on u64 values. The
entire dirty check is ~50 nanoseconds. This transforms the pipeline
from a batch processor (re-run everything) to an incremental
processor (re-run only what changed). Feeds domain 12 (change
detection per morsel).

**Consumer's responsibility:**

- Manage storage lifecycle (allocation, growth, eviction via R2 APIs)

- Accept whatever columns the consumer provides
- Build the DAG from WU Read/Write declarations (loaders are root nodes

- Recompute morsel splits at plan stage on any significant pipeline
  change (more records, new WUs, structural DAG change)
- Progress counters between phases provide implicit flow control —
  Phase N+1 can't outrun Phase N

**Loaders are just WorkUnits.** They have no special status in the
engine. They happen to have `type Read = ()` which makes them root
nodes in the DAG, but the scheduler treats them identically to any
other WU. The consumer writes them; hilavitkutin schedules them.

**Plan recomputation:** morsel ranges, fiber grouping, trunk formation
— all plan-stage outputs. When the consumer adds records, changes WU
registration, or modifies the pipeline structure, the plan stage
reruns. This is a microsecond cost (domain 15 algorithms on const-
generic bitmask structures). No incremental plan maintenance — full
recompute is cheap enough.

**Cache impact of loader placement** (T3 decisions #75-79): the first
WU in the pipeline (loader at z=0 in the morsel cube) determines the
initial cache state for all subsequent WUs. RCM column reordering
(domain 15) addresses this by placing co-accessed columns adjacent
in the arena, so the loader's write pattern warms the cache optimally
for downstream fibers.

### 17. Dispatch codegen

**CONSOLIDATED.** No tensions. T6 bench findings are authoritative.
T7 §3 distills the rules. Vocabulary updated (chain→fiber).

This domain turns the plan-time fiber/trunk/phase structure (domains
11, 14, 15) into actual executable code. The no-alloc constraint is
load-bearing. ExpandedLto (domain 03) is required.

**Devirtualisation rules (T6 L1024-1082, bench-validated):**

What devirtualises:
- Local `&[fn]` slices with known values
- Monomorphised trait dispatch
- Unrolled function parameters

What does NOT:
- Struct-field fn pointer arrays (12.6x penalty — LLVM can't prove
  contents through struct reference)
- `&[fn; N]` const-generic parameters (5.8x)
- Global `static mut` SLOTS (indirect `blr` per call)
- `run_fiber(&[WuFn], ...)` with `#[inline(never)]` (one fn for all
  callers, LLVM can't prove slice contents)

**Dispatch approaches (T6 L1084-1384, bench-validated):**

| Approach | Pattern | vs hand-fused |
|---|---|---|
| A: mono tuple | per-fiber-type trait, LOCAL `&[WuFn]` | 1.0x |
| B: unrolled params | each fn as separate parameter | 0.9x |
| C: indirect per-fiber | one fn ptr per fiber per morsel | 1.17x |
| D: trunk mega | `#[inline(always)]` fibers into trunk fn | 1.02x |
| E: schedule mega | all trunks in one fn | 0.97x |
| struct-field | fn ptrs in struct field | 12.6x FAIL |

Row size determines approach (T6 L1507-1517):
- <10K records: C or D (smaller function bodies, better icache — LLVM
  makes better register allocation for separated functions)
- \>10K records: E (full-schedule mega-dispatch, call setup amortised
  over morsel iterations)

**The codegen flattener — rust-pipe pattern (T6 L1689-1976):**

For each fiber, the flattener emits a monomorphised function. This
is where the execution strategy for ops within a fiber is decided
(domain 14 notes that fibers are the column co-location unit, not
definitionally fused). The flattener chooses how to execute the
fiber's WUs based on the fiber's properties:

For the common case (ops that benefit from fusion — most fibers):
1. Cache resources to stack-local array (domain 19)
2. Read fiber-input columns at morsel start
3. Pure function pipeline with locals — no intermediate column stores
4. Fiber-internal columns → register-to-register, DSE skips memory ops
5. All output stores grouped at end of loop body (store-buffer friendly)
6. Resource accumulation via stack-cached pointers

This is the `rust-pipe` pattern. 0.95-0.96x baseline (beats hand ASM
— LLVM makes better instruction scheduling decisions).

Deep single-fiber: the flattener eliminates store-reload between WU
boundaries by reusing locals (1.00x vs 1.14x with fn-ptr composition).
Multi-fiber scenarios already achieve 0.95-1.02x because fiber
boundaries are natural materialisation points — no flattening needed.

**Inlining discipline (T7 §3.1):**
- `#[inline(never)]` on fiber dispatch: lets LLVM optimise each fiber
  as a separate compilation unit
- `#[inline(always)]` on WU functions within their fiber: LLVM sees
  the full body, fuses ops, vectorises
- `#[inline(never)]` on per-core program: the outer function that
  sequences fibers and phase sync

**Compiled per-core dispatch (T6 L3203-3297, T7 §3.3):**

Each physical core gets a monomorphised function encoding its entire
pipeline:
1. Which phases this core processes
2. Which record ranges (head half or tail half for convergence)
3. The WU sequence per fiber (devirtualised LOCAL slices)
4. Morsel boundaries (compile-time constants from domain 12 formula)
5. Phase sync points (stack AtomicUsize + spin loop)

Every prior optimisation is subsumed:
- Local `&[WuFn]` slices: baked into the per-core function
- Resource caching: emitted as stack-local snapshots per fiber
- DSE + stores-at-end: the flattener emits the optimal loop body
- Phase pipelining: sync points are compile-time constants
- Head+tail convergence: each core's record range is baked in
- Morsel sizing: compile-time constants
- No dynamic dispatch: the entire program is monomorphised

Runtime: spawn N threads, each runs its compiled program, join when
complete. No scheduler loop. No work queues. No atomics except at
phase sync points.

**Progress counters and sync (T2 §7, T6 L2020-2024, T7 §3.3):**
- AtomicUsize per fiber, monotonic record index
- Producer: `store(end, Release)` — single `stlr` on aarch64. Not
  RMW, cheapest atomic. (T2 L474-498: plain store, not fetch_add or
  fetch_or)
- Consumer: `load(Acquire)` — single `ldar`
- Spin loop at phase boundaries: `isb` + `ldapr` + `cmp` + branch
- Stack AtomicUsize, not Arc — eliminates 6 heap allocations per
  pipeline execution. Raw pointer (`as usize`) passed to thread.

**Lock-free guarantee (T3 decisions #103-104):**
- Lock-free dispatch by construction: static dispatch table,
  one-writer-N-readers, deterministic morsel assignment
- This is a library guarantee, not an Executor implementation detail
- No locks, no CAS in the dispatch path. The only atomics are
  progress counters (plain store/load, not RMW).

**ASM verification checklist (T7 §3.2):**
For any fiber dispatch function, verify:
1. Zero `blr` (indirect branch) in function body
2. All column accesses use indexed addressing `[reg, counter, lsl #3]`
3. No `[sp, ...]` loads in per-record inner loop (prologue/morsel only)
4. Morsel size is an immediate constant
5. No `bl morsel_size` or `bl fiber_dispatch` calls

All 26 dispatch functions in the bench pass all 5 checks (T6
L3888-3903). Zero hidden pathologies.

### 19. Resource handling

**CONSOLIDATED.** No tensions. T6 bench findings are authoritative.
T7 §6 distills the rules. Three concerns: storage, access during
morsel execution, and codegen treatment.

**Storage model (R5, domain 07):**

Resources consist of three field types:
- `Field<T>` where T: ColumnValue (≤16 bytes) — scalar
- `Seq<T, const N: usize>` — const-sized array, separate arena
- `Map<K, V, const N: usize>` — const-sized map, separate arena

All behind pointer indirection to external slab storage. NOT inline
in the context struct. Writing inline data through `&self` is UB
(LLVM adds noalias to `&self`, inline writes violate it).

**Pointer separation (T6 L1612-1687):**

Resource storage and column storage must have separate pointer
provenance. Two raw pointers from the same struct may alias in LLVM's
view — LLVM cannot prove that a `cw()` store to one column doesn't
affect a `ru()` read from a resource if both pointers originate from
the same struct. Result without separation: LLVM reloads resources
from memory every iteration instead of keeping them in registers.
Measured: 1.28-1.40x overhead (T6 L1654-1658).

The fix: resource data lives in a stack-local region that LLVM can
prove non-aliasing with column pointers. The fiber dispatcher handles
this automatically — it knows the resource set from WU Read/Write
declarations.

This is NOT the same issue as `&self` vs `&mut self` (the cu/cw
noalias finding from domain 09). That prevents LLVM from reordering
writes across fused WUs. This is a pointer provenance issue: two raw
pointers from the same struct may alias. The fix is pointer
separation, not reference type.

**Stack-local resource caching (T6 L1649-1687, T7 §6.1):**

Before the morsel loop, the fiber dispatcher:
1. Snapshots all accessed resources to a stack-local array
2. Redirects res_ptr to the stack-local copy

LLVM proves stack memory doesn't alias heap column data → promotes
all resource accesses to registers. Zero resource memory ops in the
hot loop. After the loop, mutable resources are written back to
canonical storage via `stp`.

Bench evidence (T6 L1660-1669, aarch64):
- Without caching: 21 memory ops per iteration, 6 resource reloads
  through `[x0, #N]` (res_ptr). Read-only resources reloaded every
  iteration.
- With caching: all 4 resources promoted to registers (x3, x9, x10,
  x13). Zero resource memory ops in loop. Written back once via `stp`.

The caching is mechanical — derived from the WU declarations' resource
Read/Write sets. The codegen (domain 17 flattener) emits it
automatically as part of the rust-pipe pattern.

**`&self` receiver is load-bearing (T6 L1573-1576, T7 §6.3):**

Context methods take `&self`, not `&mut self`. The shared reference
prevents LLVM from adding noalias metadata that would cause it to
reorder writes across fused WUs. Column writes go through raw pointers
stored in col_ptrs — raw pointer writes have no aliasing assumptions
and execute in program order.

If Context methods took `&mut self`, LLVM would add noalias and could
reorder column writes from one WU past reads from the next WU in
the same fiber. This produces wrong results (verified: hand-fused
checked c/cm and hand-fused unchecked cu/cw produce different output
on every record for a 3-op movement fiber — T6 L1521-1530).

**Resource accumulation + convergence (T6 L2938-2961, T7 §4.3):**

WUs that accumulate resources (ru/rw per element) have cross-record
dependencies through the resource. Head+tail convergence breaks this
— two threads accumulating into the same resource concurrently.

Fix: each convergence thread gets its own stack-local accumulator
(same caching pattern). After convergence:
- Additive accumulation: add the two accumulators
- Non-commutative accumulation: fall back to sequential processing
  for that fiber (skip head+tail)

The stack-local caching pattern handles this naturally — each thread's
snapshot is independent. Detection at plan time from WU metadata:
if any WU in the fiber writes a resource that it also reads, the
fiber has resource accumulation. If the accumulation is non-commutative
(domain 08), skip convergence.

**What this supersedes:**
- T5 `ctx.resource::<T>() → &T` / `ctx.resource_mut::<T>() → &mut T`
  still the consumer API, but the internal implementation uses raw
  pointers through stack-local cache, not direct references to
  canonical storage
- T3 L746 LoopVersioningLICM concern → solved by pointer separation
  (no LLVM pass needed, the data layout itself fixes aliasing)

L259-271 §Q3 (access methods), T6 L1521-1583 (aliasing UB), T6

#### — Level 7: Scheduling and threading (depends on L0-L6) —

### 20. Threading model

**CONSOLIDATED.** No tensions. T6 bench findings authoritative.

**Pre-allocated thread pool (T6 L2997-3034, bench-validated):**

Pool size = `physical_core_count` threads, created once, persist for
the pipeline's lifetime. Hybrid wake strategy:
- Spin 128 iterations (nanosecond wake, hot path)
- `thread::park` (microsecond wake, saves power)
- `park`/`unpark` available on all std targets (futex on Linux,
  os_unfair_lock on macOS)

Pool eliminates the small-N catastrophe: crossover ~2K records (vs ~50K
for spawn/join strategies). At 2K records, spawn/join was 2.33x worse;
the pool is 0.93x (7% faster than sequential).

**Heterogeneous core awareness (T3 decision #37, T6 L1407-1458):**

Detect at startup (once):
- L1/L2 size per core type
- Core count per type (P-cores, E-cores)
- Core type identity

Core assignment:
- Critical path trunks → P-cores (T3 decision #39)
- Leaf/branch fibers → E-cores
- P-cores get larger morsels (larger L1/L2)
- E-cores get proportionally smaller morsel ranges (not equal
  distribution — E-cores at ~40% P-core speed drag down performance
  with even distribution)

Bench: 8 threads (4P+4E) is WORSE than 4 threads (4P only) on M1
(T6 L1407-1458). Thread count = min(physical_cores, parallelisable
width + 1) — the "+1" is the chaser/convergence thread.

**Core-pinned trunks (T6 L3067-3085):**

Each trunk assigned to a specific pool thread at plan time. The trunk
stays on that thread (= that core) across all its morsels — warm L1,
no migration. Leftover threads pick up work in priority order:
1. Convergence work (tail of heaviest trunk's record range)
2. Long branches with significant compute
3. Bridge processing (fan-in nodes after parent trunks complete)

**Head+tail convergence (T6 L2908-2961):**

Within any commutative fiber, records are independent. Two threads
process from opposite ends, converging in the middle. Every
commutative fiber gets ~2x parallelism. Resource accumulation handled
by domain 19 (per-thread stack-local accumulators, merged after).
Non-commutative resource accumulation → skip convergence for that
fiber (domain 08).

**Pipeline parallelism between phases (T6 L3087-3165):**

Phases overlap via AtomicUsize progress counters. Phase N+1 starts as
soon as Phase N produces enough records for one morsel. Total time ≈
max(phases) + fill latency, not sum of phases. With 4 cores and 4
phases: all phases active simultaneously, each processing a different
record window.

**Per-core compiled programs (domain 17):**

Each core's thread runs a monomorphised function. No scheduler loop,
no work queues, no atomics except at phase sync points. Threading =
spawn N threads, each runs its compiled program, join when complete.

**Lock-free by construction (T3 decisions #103-104):**

Static dispatch table, one-writer-N-readers per fiber, deterministic
morsel assignment. No locks, no CAS in the dispatch path. Library
guarantee. The only atomics are progress counters (plain store/load,
not RMW).

**Work-stealing is NOT the default (T3 decisions #96-97):**

The default is deterministic morsel assignment (pre-partitioned at
plan time, no atomic fetch_add — T5 L1110-1113). If variable per-record
time causes imbalance (T3 DF9), the consumer can provide a stealing
Executor via the Executor trait. The library provides the deterministic
one.

**Thread spawn overhead (T7 §3.4):**

`std::thread::spawn` adds 3 `malloc` calls (~100ns at ~17ms per
pipeline — 0.0006%). Negligible. Pre-allocated pool eliminates it.

### 21. Scheduling strategy

**CONSOLIDATED.** No tensions. T6 bench findings authoritative.
T7 §5 distills the rules.

**No single strategy dominates all workloads** (T6 L2203-2210).
Best total wall time depends on compute balance, phase topology,
and record count.

**Record count thresholds (T6 L2075-2077, T7 §5.1):**
- <10K records: sequential always — thread overhead dominates
- 10K-100K: depends on pipeline shape
- \>100K: always consider threading

**Pipeline shape (T7 §5.2):**

| Shape | Indicator | Strategy |
|---|---|---|
| Wide (many roots, low depth) | roots > depth/2 | adaptive / pipe-chase |
| Deep (serial deps, few roots) | depth > fibers/2, roots ≤ 2 | sequential |
| Mixed (some width, some depth) | phases have parallel trunks | phased |

Deep pipelines never significantly benefit from threading (T6
L2283-2355): 7-fiber deep pipeline → threading adds 1.03-1.05x
overhead. Serial dependency depth is a fundamental limit.

**Per-phase config selection:**

Domain 14 produces adaptive configs (MAX_FUSE / BALANCED / MAX_SPLIT)
per phase. This domain selects which config to use for each phase
based on that phase's topology:
- Phase with clear mega-trunk → MAX_FUSE + head+tail convergence
- Phase with wide independent branches → MAX_SPLIT
- Phase with moderate depth and width → BALANCED

This is not a global heuristic. Each phase is evaluated independently.

**Plan-time strategy selection (T6 L2227-2260):**

The scheduler knows at plan time: each fiber's column count (→ morsel
size), each fiber's WU count and complexity (from scheduling hints),
the dependency graph, total record count. From this it estimates
compute weight ratios:

- `record_count < 10K` → Sequential
- `consumer_weight / producer_weight > 0.5` → ChaseSteal (both
  threads contribute to consumer fibers)
- `producer_weight < LIGHT_THRESHOLD` → Adaptive (producer finishes
  fast, steals consumer tail)
- else → PipeChase (overlap producer with consumer chasing)

Weight estimate from WU count × column access counts. Rough
classification (light/medium/heavy). Wrong selection → ~10-20%
suboptimal, not catastrophic.

**Tested strategies (T6 L1982-2355, at 2M records):**

| Strategy | vs seq | Description |
|---|---|---|
| sequential | 1.00x | single core, no overhead |
| pipe-chase | 0.56-0.58x | A+B interleaved core 0, C chases core 1 |
| chase+steal | 0.55-0.56x | pipe-chase + producer steals C tail |
| adaptive | 0.32-0.54x | no frame budget, continuous chaser |

Asymmetric workloads amplify chase+steal: L-H gives 0.34x (66%
faster with 2 threads).

**Per-project defaults (T7 §5.3):**

| Project | Records | Shape | Strategy |
|---|---|---|---|
| polka-dots | ~200 | deep | sequential — 1µs total |
| saalis | 10K-100K | mixed | sequential for deep, adaptive for wide |
| loimu | 100K-10M | wide | phased with convergence |

**Frame budget (T7 §5.4):**

Controls incrementality (frame count), not throughput. Adaptive
strategies ignore budget. Sequential is stable across budgets.
Budget mainly matters for game loops with time budgets. For batch
pipelines (polka, saalis): irrelevant.

**Schedule reuse (T3 decisions #54-56):**
- Schedule constructed once, reused across frames
- Morsel-to-core affinity pinned across frames (warm L2)
- Stable morsel boundaries unless record count changes
- Full recompute at plan stage on pipeline structure change (domain 16)

**Cadence as consumer-layer (T5 §Q5):**

SchedulerMetrics resource: pass_count, ema_pass_duration_ns,
active_units, stolen_count, idle_ns. EMA 1/8 decay. Consumer reads
metrics via `On<meta::ScheduleEnd>` WU and implements their own
cadence logic. hilavitkutin provides measurement data, consumer
decides policy.

**Sources:** T2 L585-596 + L765-781 (two-step, priority), T3
decisions #54-56 (schedule reuse), T5 L391-476 + L567-637 (hints,
cadence), T6 L1507-1517 + L2130-2355 + L2596-2606 + L2900-2906 +
L3770-3811 (bench findings, strategy selection), T7 §5 (distilled)

**Cross-references:**
- Domain 11: phase topology feeds selection
- Domain 14: per-phase adaptive configs (MAX_FUSE/BALANCED/MAX_SPLIT)
- Domain 17: dispatch approach selection by record count
- Domain 20: threading model executes the selected strategy

### 22. Runtime adaptation

**CONSOLIDATED.** No tensions. T3 defines the framework, T5 provides
metrics, T6 validates.

**Two analysis tiers (T3 L1117-1206, decisions #61-65):**

Static analysis (at schedule construction, plan-time):
- Cache pressure profile: per-fiber L1/L2 pressure from column sizes
- Data flow volume map: bytes flowing between fibers
- Column lifetime map: when each column is first written, last read
- Memory watermark: peak concurrent column allocation

Runtime analysis (across frames, between passes):
- Per-morsel timing: actual execution time per morsel
- Change frequency: classify morsels as hot/warm/cold based on
  generation counter activity (domain 12)
- Cache residency prediction: which morsels are likely L1-warm from
  the previous frame
- Frame time prediction: EMA of pass duration for budgeting
- Throughput trending: detect thermal throttling, memory pressure

**Morsel temperature → core assignment (T3 decision #64):**

Hot morsels (frequently changing, high compute) → P-cores. Cold
morsels (rarely changing, often skipped) → E-cores or deferred.
Temperature from change detection counters (domain 12) + per-morsel
timing.

**EMA metrics (T5 §Q5, L601-637):**

SchedulerMetrics resource: `pass_count`, `ema_pass_duration_ns`,
per-unit EMA, `active_units`, `stolen_count`, `idle_ns`. Decay:
`(ema * 7 + measured) / 8`. Vectorized batch update: NEON 4×u32 or
AVX2 8×u32 for all unit EMAs in one pass (T5 L1054-1066).

**Executor reorganisation triggers (T3 decision #62):**

Between frames, not during execution. Compare EMAs against thresholds
(arvo-bitmask dirty propagation pattern). If fiber morsel timing
changed significantly → recompute morsel sizes. If phase balance
shifted → re-select per-phase configs. If record count changed →
full plan recompute. Triggers are cheap bitmask comparisons.

**Predictive parking at sync points (T6 L3254-3267):**

At phase boundaries, estimate wait time from compute weight of other
core's remaining work:
- <100ns → spin
- 100ns-10µs → `spin_loop()` with backoff
- \>10µs → `thread::park()`

Wrong prediction costs at most one park/unpark cycle (~1µs).

**Frame time prediction (T3 decision #65):**

EMA of pass duration enables frame budgeting (game loops) and progress
bars (batch pipelines). Consumer reads via SchedulerMetrics (domain 23).

**Sources:** T3 L1117-1206 + decisions #61-65 (volumetric analysis),
T5 L474 + L601-637 + L1054-1066 (EMA, vectorized update), T6 L1444 +
L1507-1517 + L2183-2281 + L3254-3267 (bench findings, predictive
parking)

**Cross-references:**
- Domain 12: change detection, generation counters
- Domain 17: predictive parking in compiled programs
- Domain 20: core assignment from temperature
- Domain 21: strategy re-selection between frames
- Domain 23: SchedulerMetrics resource

### 23. Scheduler builder and meta

**CONSOLIDATED.** No tensions. T5 §Q8-Q9 defines, T6 §Q6-Q7
elaborates. Three open questions resolved during consolidation.

**Builder API (T6 §Q6, const generics):**

```rust
Scheduler::<MAX_UNITS, MAX_STORES, MAX_LANES>::builder()
    .add::<MyWU>()               // registers any WorkUnit<S>
    .resource::<T>(initial)      // singleton with initial value
    .resource_default::<T>()     // singleton where T: Default
    .column::<T>()               // column registration
    .memory(my_provider)         // override MemoryProvider
    .threads(my_pool)            // override ThreadPool
    .clock(my_clock)             // override Clock
    .build()                     // → Scheduler
```

One `.add::<MyWU>()` handles all schedule conditions — `Always`,
`On<V>`, etc. are on the trait impl, not the builder call. No
separate `add_on`.

Consumer registers WUs via explicit builder API (T5 §Q8). No
`#[register]` in hilavitkutin itself — consumers handle registration
their way (polka: inventory, saalis: Named, loimu: macros). ECS-as-DI:
Read/Write declarations ARE dependency injection.

**Static schedule (resolved):**

Everything via the builder is statically defined. The schedule is
statically analysable and computable. No runtime WU registration, no
dynamic schedules. The plan stage adjusts only what compile time
cannot: morsel sizes from runtime hardware detection, record counts,
EMA-based recomputation. This eliminates the need for fallback dispatch
approaches (Approach C) entirely.

**Self-hosting scheduler (T5 §Q9, L716-802):**

The scheduler is itself a pipeline. Meta work units run at lifecycle
points:

Meta virtuals:
- `meta::PlanStage` — fired when DAG changes (recompute plan)
- `meta::ScheduleReady` — fired when plan-stage units complete
- `meta::PassStart` — fired at start of each pass
- `meta::ScheduleEnd` — fired after all consumer work

Meta resources (restricted via MetaAccess marker trait):
- `meta::Dag` — the dependency graph
- `meta::ExecutionPlan` — fiber/trunk/phase assignments
- `meta::LaneAssignment` — core assignments
- `meta::SchedulerMetrics` — EMA timing data (domain 22)

The irreducible kernel is ~50 lines: fire PlanStage, wait for plan
WUs, fire ScheduleReady, dispatch consumer WUs, fire ScheduleEnd.

**Version stamps (T5 §Q10, T6 §Q10):**

Parallel arrays (SoA): separate contiguous `[u64; MAX_STORES]` for
"before" and "current" stamps. Capture before-versions before WU runs,
compare after. Changed stores = bitmask of actually-modified columns.
Increment only on completion — no partial writes (domain 08). 2-5 u64
per unit stamp view, packed in one cache line.

**Error handling (resolved):**

`execute()` returns `()`. Abort on failure. This is the design:
- `no_os` and `os` tiers: panic = abort already
- `std` tier: `catch_unwind` available but we prefer abort for LLVM
  codegen quality (abort enables more aggressive optimisation —
  no unwind tables, no landing pads)
- If a WU needs to signal failure: write to an error column or fire
  an error virtual. Data flow, not control flow. The scheduler detects
  failure via version stamps or error virtual, then poisons dependents.
- No Result return type on the hot path

**Error propagation (T3 decisions #106-110):**

- Fiber failure → freeze progress counter + poison all dependent fibers
- Independent fibers continue
- PipelineResult: per-fiber status (Completed / Failed / Poisoned)
- Consumer decides if partial failure is acceptable
- No result extraction from Pipeline::run() — consumer reads
  ColumnStorage directly
- Pipeline::run() returns PipelineResult status only

**R2 APIs (domain 07):**

- `evict(column)` / `dump(column)` — persist column to consumer
  storage, release memory
- `inject(column)` / `import(column)` — load column from consumer
  storage
- hilavitkutin owns generation counters + skip logic. Consumer owns
  persistence.

**Resource initialization (resolved):**

`.resource::<T>(value)` is the primary API — consumer provides initial
value. `.resource_default::<T>()` for convenience where `T: Default`.

**Sources:** T1 L90 + L598-607 + L956-965 (registration, DAG, errors),
T2 L591-598 + L866-884 (schedule, consumer API), T3 decisions #106-110
(error propagation), T5 L376-383 + L688-802 + L828 + L1096-1165
(registration, self-hosting, stamps), T6 L468-581 + L713-753
(builder, meta, stamps)

- Domain 02: MemoryProvider, ThreadPool, Clock overrides

- Domain 09: WorkUnit registration, builder accepts WUs

**Architectural constraints (T1 §4, §10):**
- **No-alloc constraint:** no Vec, Box, String, alloc crate. All
  storage via SlabColumn or stack arrays. (T5 L899-906 §Q11)
- **Type taxonomy:** trait / newtype / repr(C) enum / primitive.
  Governs what form every type takes. (T1 L140-162 §3)

## Resolutions

**R1. Column<T> only — IR is consumer-side.** (items 1, 2)
hilavitkutin-api defines `Column<T>` as a typed storage slot. The
`Column<In<IR>, As<T>>` schema-as-traits pattern is consumer-side
(polka-sdk, saalis-sdk). hilavitkutin doesn't know or care about
domain schemas — it schedules based on Read/Write column sets only.
The `In<CatalogueIR, Installed>` syntax in polka-dots is a consumer
wrapper around hilavitkutin's `Column<T>`. Translation happens in
the consumer's registration layer.

**R2. Incremental execution: counters in hilavitkutin, persistence via evict/inject API.** (item 5)
hilavitkutin owns per-morsel generation counters and skip-unchanged
DAG propagation logic (runtime scheduling concern). For persistence,
hilavitkutin provides evict/dump (extract column data out without
creating copies or retaining references) and inject/import (move
column data in) APIs. Column type registration is static (compile-time)
— no runtime column type creation. Consumers build hot-cold storage
layering on top of these primitives. On-disk state is consumer's
responsibility.

**R3. Type-native stride, no flags byte, 16-byte max.** (item 15, domain 04)
Stride = size_of::<T>() per column. No per-entry flags byte — null/
tombstone via separate bitmask. Value constraint size_of::<T>() <= 16
(was 15 with flags byte). Supersedes T1 #80-81, T2 #1, T3 #90.

**R4. ColumnValue trait with blanket impl + specialisation.** (domain 05)
Replaces T3's Payload/SlotSize system. Blanket impl for all Copy +
'static ≤ 16 bytes via min_specialization. BIT_WIDTH const (no default
on trait, default on blanket impl) for sub-byte bitpacking. No
pack/unpack, no SlotSize. Type IS the storage. Consumers store whatever
they want; arvo-types-only lint is our internal discipline only.

**R5. Resource model — scalars + const-sized collections.** (domain 07)
Resources have three field types: Field<T> (≤16 bytes, register-cached),
Seq<T, N> (const-sized array, arena-allocated), Map<K, V, N> (const-sized
map, arena-allocated). No dynamic collections. Write collections count
toward L1 morsel budget; read-only collections ride L2 prefetcher.
ColumnStorage returns raw pointers (not slices — aliasing UB).

**R6. Static composition, adaptive parameters.** (domain 23)
Pipeline composition is static: all WUs registered via the builder at
compile time. The WU set, DAG structure, fiber/trunk/phase topology,
and monomorphised dispatch functions are all fixed at build time. This
is what enables LLVM devirtualisation — the schedule is statically
analysable. No runtime WU registration, no dynamic schedules.

Plan-stage parameters are adaptive: morsel sizes (from runtime hardware
detection), record counts (from consumer data), per-phase strategy
selection (from EMA metrics), and morsel-to-core affinity (from
temperature). These adjust at plan time without changing the pipeline
composition. Eliminates need for fallback dispatch approaches.

**R7. Abort on failure.** (domain 23)
execute() → (). No Result on the hot path. Errors via data flow
(error columns/virtuals). Abort enables better LLVM codegen. On
no_os/os tiers panic = abort already. On std tier we prefer abort.

**R8. "record" is the global data identity.** (vocabulary)
Not "entity" (ECS connotation), not "row" (tabular connotation).
A column has N records. A morsel windows into a range of records.

**R9. "fiber" replaces "chain".** (vocabulary, domain 14)
Fiber = unit of column co-location and morsel windowing. Not
definitionally fused. "chain group" → trunk. "partition" → phase.

All 36 missing items resolved. See resolution status above.
