## 1. Per-crate optimization in Cargo

### What Cargo supports natively

`[profile.<name>.package.<package-name>]` in the workspace root `Cargo.toml`
allows per-package overrides for a limited set of settings.

**Overridable per package:**

| Setting | Type | Effect |
|---------|------|--------|
| `opt-level` | `0`, `1`, `2`, `3`, `"s"`, `"z"` | Optimization level |
| `debug` | `bool`, `0`, `1`, `2`, `"line-tables-only"` | Debug info |
| `split-debuginfo` | string | Debug info splitting |
| `strip` | `"none"`, `"debuginfo"`, `"symbols"` | Binary stripping |
| `debug-assertions` | bool | Enable debug assertions |
| `overflow-checks` | bool | Integer overflow checks |
| `incremental` | bool | Incremental compilation |
| `codegen-units` | integer | Parallelism vs quality tradeoff |

**NOT overridable per package:**

| Setting | Why |
|---------|-----|
| `panic` | ABI-level; must be consistent across all crates |
| `lto` | Link-time; applies to the whole link unit |
| `rpath` | Linker-level; applies to the final binary |
| `rustflags` | Not a profile setting at all |

### Syntax

```toml
# Optimize a specific dependency in dev builds
[profile.dev.package.lalrpop]
opt-level = 3

# Optimize ALL non-workspace dependencies in dev builds
[profile.dev.package."*"]
opt-level = 2

# Override build scripts and proc macros
[profile.dev.build-override]
opt-level = 3

# Specific version override
[profile.dev.package."foo:2.1.0"]
opt-level = 3
```

### Override precedence

1. `[profile.dev.package.name]` -- named package (highest priority)
2. `[profile.dev.package."*"]` -- any non-workspace member
3. `[profile.dev.build-override]` -- build scripts, proc macros, their deps
4. `[profile.dev]` -- profile-level default
5. Cargo built-in defaults

### The generics caveat

Per-package `opt-level` overrides have a critical limitation: generic functions
are monomorphized in the CALLING crate, not the defining crate. Setting
`opt-level = 3` on a library does not affect its generic functions when
instantiated by a caller compiled at `opt-level = 1`.

For data-processing crates where the hot path is generic (e.g., `Column<T>`
operations), this means the library's opt-level is partially irrelevant. The
calling crate's opt-level determines the optimization of instantiated generics.

Recommendation from Cargo docs: `opt-level = 1` is a better compromise than
`opt-level = 0` for libraries, because it applies some optimizations while still
allowing monomorphized items to be shared between crates.

### profile-rustflags (unstable, nightly only)

Tracking issue: [cargo#10271](https://github.com/rust-lang/cargo/issues/10271)

Allows setting `rustflags` per profile:

```toml
cargo-features = ["profile-rustflags"]

[profile.release]
rustflags = ["-C", "target-cpu=native"]
```

Status: NOT accepted. Open design questions remain about interaction with
rustdoc, build scripts, and proc-macros. Known bugs with dependency resolution
when using `--cfg` flags. Not on a stabilization timeline.

This feature applies flags to ALL crates in the profile, not per-package. There
is no `[profile.release.package.foo].rustflags` syntax.

## 2. RUSTC_WORKSPACE_WRAPPER for per-crate flag injection

### The mechanism

`RUSTC_WORKSPACE_WRAPPER` specifies a wrapper program that Cargo invokes
instead of `rustc` for workspace members only. Non-workspace dependencies
(from crates.io) are NOT affected.

```
RUSTC_WORKSPACE_WRAPPER=/path/to/wrapper cargo build
```

The wrapper receives the full `rustc` invocation as arguments. The first
argument is the path to the actual `rustc` binary. All subsequent arguments
are the flags Cargo normally passes.

### Environment variables available inside the wrapper

When the wrapper executes, Cargo sets these env vars that identify the crate:

| Variable | Content | Example |
|----------|---------|---------|
| `CARGO_PKG_NAME` | Package name from Cargo.toml | `polka-pipeline` |
| `CARGO_CRATE_NAME` | Crate name (hyphens become underscores) | `polka_pipeline` |
| `CARGO_MANIFEST_DIR` | Directory containing the crate's Cargo.toml | `/path/to/crates/polka-pipeline` |
| `CARGO_PRIMARY_PACKAGE` | Set if this is a user-selected package | `1` |
| `CARGO_BIN_NAME` | Binary name (only for bin targets) | `polka` |

### Skeleton wrapper script

```bash
#!/usr/bin/env bash
# rustc-per-crate-wrapper.sh
# Usage: RUSTC_WORKSPACE_WRAPPER=./rustc-per-crate-wrapper.sh cargo build

RUSTC="$1"
shift

case "${CARGO_PKG_NAME}" in
  polka-pipeline)
    # Hot data processing: aggressive vectorization, loop passes
    exec "$RUSTC" "$@" \
      -C opt-level=3 \
      -C codegen-units=1 \
      -C target-cpu=native \
      -C llvm-args=-force-vector-width=8 \
      -C llvm-args=-force-vector-interleave=4 \
      -C passes=loop-distribute
    ;;
  polka-cli)
    # CLI: optimize for size, fast compile
    exec "$RUSTC" "$@" \
      -C opt-level=s \
      -C codegen-units=16
    ;;
  polka-sdk | polka-primitives | polka-derive)
    # SDK/traits: fastest compile, minimal optimization
    exec "$RUSTC" "$@" \
      -C opt-level=1 \
      -C codegen-units=16
    ;;
  polka-data)
    # Data store: branch prediction, function specialization
    exec "$RUSTC" "$@" \
      -C opt-level=3 \
      -C codegen-units=1 \
      -C target-cpu=native
    ;;
  *)
    # Default: standard release settings
    exec "$RUSTC" "$@"
    ;;
esac
```

### Caching behavior

The wrapper affects the filename hash, so artifacts produced by the wrapper
are cached separately from non-wrapper builds. Rebuilding without the wrapper
will NOT reuse wrapper-compiled artifacts, and vice versa. This means toggling
the wrapper causes a full rebuild.

### Nesting with RUSTC_WRAPPER

If both `RUSTC_WRAPPER` and `RUSTC_WORKSPACE_WRAPPER` are set:

```
$RUSTC_WRAPPER $RUSTC_WORKSPACE_WRAPPER $RUSTC [flags...]
```

This allows combining sccache (as `RUSTC_WRAPPER`) with per-crate flags
(as `RUSTC_WORKSPACE_WRAPPER`).

### Limitations

1. Only affects workspace members. External dependencies use the profile's
   settings directly (or `[profile.*.package."*"]`).
2. The wrapper is a process-level overhead: one `fork+exec` per crate
   compilation. For 17 workspace crates this is negligible.
3. Flags injected by the wrapper are NOT visible to `cargo build -vv` in
   the standard way. Debugging requires adding logging to the wrapper.
4. Cannot change `panic` strategy, LTO, or anything that needs to be
   consistent across the link unit.

### cargo-fixup: prior art

[cargo-fixup](https://github.com/cecton/cargo-fixup) uses
`RUSTC_WORKSPACE_WRAPPER` to dynamically patch crate source code at build time.
It writes a wrapper script and `config.toml` into `.cargo/`. This demonstrates
the pattern is viable in production.

## 3. Per-crate codegen backend selection

### Current state (as of Cargo 1.89.0 / May 2025)

**PR #15562** (merged May 19, 2025) extended `codegen-backend` to accept
arbitrary dylib paths and added per-package backend configuration capability.

The syntax in `Cargo.toml`:

```toml
[profile.dev]
codegen-backend = "cranelift"  # default for workspace

[profile.dev.package.polka-pipeline]
codegen-backend = "llvm"  # override for hot crate
```

This requires nightly (`-Z codegen-backend`). The feature uses two input types:
1. **Built-in names** (`"llvm"`, `"cranelift"`, `"gcc"`) -- may stabilize
2. **Dylib paths** (`"/path/to/backend.so"`) -- never intended for stabilization

### Mixed-backend compilation

**Technically feasible**: Different codegen units are compiled independently and
linked. The Cranelift maintainer (bjorn3) confirmed: "Mixing cg_clif and
cg_llvm should be completely fine."

**In practice, problematic**: Issue #13208 reports segfaults when mixing backends
with frameworks like Bevy. Possible causes:
- Calling convention mismatches at crate boundaries
- ABI differences in how each backend handles generics
- Different handling of stack frames and unwinding

**LTO is incompatible**: Cross-crate LTO requires all crates to use the same
backend (same IR format). Fat LTO with mixed backends causes linker errors.
Thin LTO may work partially but is untested.

Issue #13208 was **closed as "Not Planned"** (June 2024), labeled as
"S-blocked-external" (blocked on upstream rustc changes).

### The RUSTC_WORKSPACE_WRAPPER approach to backend selection

Since `codegen-backend` can be passed as a `-Z` flag to rustc, the wrapper
script can select backends per crate:

```bash
case "${CARGO_PKG_NAME}" in
  polka-pipeline | polka-data)
    exec "$RUSTC" "$@" -Z codegen-backend=llvm
    ;;
  polka-cli)
    exec "$RUSTC" "$@" -Z codegen-backend=cranelift
    ;;
esac
```

This has the same caveats as the Cargo.toml approach: segfault risk at
crate boundaries, no LTO across backends.

### Verdict

Mixed-backend compilation is not production-ready. Use LLVM for everything
in release builds, and optionally Cranelift for dev builds (all crates) for
faster iteration.

## 4. Per-crate LLVM pass selection

### The mechanism

The `RUSTC_WORKSPACE_WRAPPER` can inject per-crate LLVM passes:

```bash
case "${CARGO_PKG_NAME}" in
  polka-pipeline)
    # Morsel processing: loop distribution for better vectorization
    exec "$RUSTC" "$@" \
      -C passes=loop-distribute \
      -C llvm-args=-enable-loop-distribute
    ;;
  polka-data)
    # Hash maps, B-trees: branch prediction hints
    exec "$RUSTC" "$@" \
      -C llvm-args=-enable-loop-data-prefetch
    ;;
  polka-cli)
    # String formatting, I/O: size optimization, function merging
    exec "$RUSTC" "$@" \
      -C opt-level=s \
      -C passes=mergefunc
    ;;
  polka-sdk | polka-primitives)
    # Pure type definitions: fastest compile
    exec "$RUSTC" "$@" \
      -C opt-level=1
    ;;
esac
```

### Available passes for data-processing crates

The full pass list (551 passes in LLVM 22.1.0) is documented in the companion
research doc. Key passes by crate archetype:

**Pipeline/execution crate (tight numeric loops, morsel processing):**

| Pass | Effect |
|------|--------|
| `loop-vectorize` | SIMD vectorization (on by default at O2+) |
| `slp-vectorizer` | Cross-lane scalar to vector promotion |
| `loop-distribute` | Break complex loops for vectorization |
| `loop-fusion` | Combine adjacent column-processing loops |
| `loop-unroll<O3>` | Full unrolling with O3 heuristics |
| `loop-data-prefetch` | Software prefetch for streaming access |
| `load-store-vectorizer` | Combine adjacent memory ops |

LLVM args for this archetype:
```
-C llvm-args=-force-vector-width=8
-C llvm-args=-force-vector-interleave=4
-C llvm-args=-enable-loop-distribute
```

**CLI crate (string formatting, I/O, argument parsing):**

| Pass | Effect |
|------|--------|
| `mergefunc` | Merge identical functions (reduces binary size) |
| `globaldce` | Remove dead global definitions |
| `constmerge` | Merge duplicate constants |

Use `-C opt-level=s` or `-C opt-level=z` for size.

**SDK/traits crate (pure type definitions, no runtime code):**

Minimal optimization. `-C opt-level=1` for basic inlining, high
`codegen-units` for parallel compilation.

**Data store crate (hash maps, B-trees, typed access):**

| Pass | Effect |
|------|--------|
| `gvn` | Eliminate redundant loads (hash table probing) |
| `licm` | Hoist invariants from search loops |
| `loop-data-prefetch` | Prefetch next B-tree node |
| `simplifycfg` | Clean up match/enum dispatch |

### Custom pass pipeline via -C no-prepopulate-passes

To define a completely custom LLVM pipeline for a specific crate:

```bash
exec "$RUSTC" "$@" \
  -C no-prepopulate-passes \
  -C passes="instcombine simplifycfg sroa gvn licm loop-vectorize slp-vectorizer"
```

This disables ALL default passes and runs only your specified list. Use with
extreme caution: missing passes can cause correctness issues (e.g., skipping
`lower-expect` can break `likely`/`unlikely` hints).

### Printing the default pipeline

```bash
# See what passes rustc runs at O3
RUSTFLAGS="-Z print-llvm-passes" cargo build --release 2>&1 | head -100

# Time individual passes
RUSTFLAGS="-Z time-llvm-passes" cargo build --release

# View IR before/after all passes (extremely verbose)
RUSTFLAGS="-C llvm-args=-print-after-all" cargo build --release
```

### Interaction with LTO

Passes added via `-C passes` run during per-crate compilation. During LTO, LLVM
runs a separate optimization pipeline on the merged IR. Per-crate passes do NOT
re-run during LTO unless the LTO pipeline includes them.

With thin LTO, some passes run during both phases. With fat LTO, the per-crate
passes run first, then a full pipeline runs on merged IR.

## 5. Alternative compiler backends

### Cranelift (`rustc_codegen_cranelift`)

**Purpose:** Fast debug builds. Cranelift prioritizes compilation speed over
code quality.

**Current state:**
- Distributed as `rustc-codegen-cranelift-preview` component via rustup
- Available on Linux (x86_64, aarch64), macOS (x86_64, aarch64), Windows (x86_64)
- Requires nightly Rust

**What it optimizes:**
- Basic instruction selection (good register allocation)
- Simple peephole optimizations
- No loop optimizations, no vectorization, no advanced alias analysis
- No SIMD support for `std::arch` (partial); `std::simd` works

**What it does NOT do:**
- No auto-vectorization
- No loop unrolling, tiling, fusion
- No LTO
- Panic unwinding is experimental; `-Cpanic=abort` is default

**Compilation speed:** Roughly 2-4x faster than LLVM at O0. The advantage
narrows at higher optimization levels because Cranelift has no equivalent.

**Suitable for release builds?** No. Generated code is significantly slower
than LLVM O2/O3 for compute-intensive workloads. Suitable for:
- Debug iteration cycles
- Build servers where compile time matters more than runtime
- Test suites

**Usage:**
```bash
rustup component add rustc-codegen-cranelift-preview --toolchain nightly
CARGO_PROFILE_DEV_CODEGEN_BACKEND=cranelift cargo +nightly build
```

### GCC backend (`rustc_codegen_gcc`)

**Purpose:** Target architectures not supported by LLVM; potentially better
optimization for some workloads via GCC's independent optimizer.

**Current state (December 2025 blog post by Guillaume Gomez):**
- WIP, approximately 100 failing UI tests
- Uses `libgccjit` for AOT compilation (despite the "JIT" name)
- Supports basic cargo builds and rustc compilation
- Not production-ready

**Potential advantages over LLVM:**
- GCC has historically better Fortran-style loop nest optimization (GRAPHITE,
  GCC's polyhedral optimizer, is more mature than Polly for some patterns)
- GCC's `-O3` can produce code ~10% faster on some x86 hardware for C/C++
- Broader architecture support (Dreamcast, older ARM, MIPS variants)
- Independent optimizer means different heuristic tradeoffs; may find
  optimizations LLVM misses and vice versa

**Practical reality:**
- The 100 failing tests indicate many Rust features are not fully supported
- No performance benchmarks exist comparing GCC-backend Rust to LLVM-backend
- The secondary goal is explicitly stated as "checking if using the gcc
  backend will provide any run-time speed improvement"
- Usable for experimentation, not for production

**Usage:**
```bash
# Requires building from source
git clone https://github.com/nickel-lang/nickel.git
# Follow build instructions in rustc_codegen_gcc README
rustc -Zcodegen-backend=/path/to/librustc_codegen_gcc.so input.rs
```

### SPIR-V / GPU backends (rust-gpu)

**Purpose:** Compile Rust to GPU shaders (SPIR-V) for graphics and compute.

**Current state (July 2025):**
- All major GPU backends can run from a single Rust codebase
- Cross-platform GPU compute in Rust is possible but has rough edges
- Uses `rustc -Z codegen-backend` to plug in the SPIR-V backend
- SPIR-T shader IR is mandatory (was optional, now required)
- Requires a specific nightly Rust version

**Limitations:**
- No `SPV_KHR_physical_storage_buffer`
- No sparse residency textures
- No fragment shading rate
- Panic handling is limited
- Many std library features unavailable in shader context

**Relevance to polka-dots:**
- Future GPU offload of morsel processing is theoretically possible
- Would require expressing column operations as compute shaders
- The morsel model (fixed-size data chunks, independent processing) maps
  well to GPU compute dispatch
- Not practical today; would be a multi-year effort

### Summary table

| Backend | Maturity | Use case | Code quality | Compile speed |
|---------|----------|----------|-------------|---------------|
| LLVM | Production | Release builds | Excellent | Baseline |
| Cranelift | Preview | Dev builds | Poor | 2-4x faster |
| GCC | Alpha | Architecture reach | Unknown for Rust | Unknown |
| SPIR-V | Experimental | GPU compute | N/A | N/A |

## 6. MLIR and Rust

### What MLIR is

MLIR (Multi-Level Intermediate Representation) is an LLVM subproject providing
a framework for building domain-specific IRs with reusable optimization
infrastructure. Unlike LLVM IR (which is a single fixed IR), MLIR supports
multiple "dialects" at different abstraction levels that can coexist and be
progressively lowered.

### Key dialects relevant to data processing

| Dialect | Abstraction level | Relevance |
|---------|-------------------|-----------|
| `affine` | Polyhedral loop nests | Loop tiling, interchange, fusion for array code |
| `linalg` | Named linear algebra operations | Column operations, reductions, projections |
| `scf` | Structured control flow (for, if, while) | Loop nest representation |
| `memref` | Multi-dimensional memory references | Column storage representation |
| `vector` | Hardware-independent vector operations | SIMD without target-specific intrinsics |
| `arith` | Basic arithmetic operations | Column arithmetic |
| `tensor` | Immutable multi-dimensional arrays | Functional column representation |

The `affine` dialect is particularly interesting: it represents the same
polyhedral model as Polly but at a higher level, before lowering to LLVM IR.
This means better optimization opportunities because the structure is preserved.

The `linalg` dialect provides named operations like `linalg.generic` that
represent computations over multi-dimensional data with explicit iteration
domains and data access patterns. This is a natural fit for columnar operations.

### Rust integration: Melior

[Melior](https://github.com/raviqqe/melior) provides Rust bindings to the
MLIR C API.

**Status:** Alpha. API is unstable. Available on crates.io.

**What it supports:**
- Context and module creation
- Dialect registration (`register_all_dialects()`)
- Operation construction (arith, func, and other dialects)
- Block and region building
- IR verification
- Code generation

**Example (from Melior docs):**
```rust
use melior::{Context, dialect::{arith, func}, ir::*};

let context = Context::new();
context.register_all_dialects();

let module = Module::new(Location::unknown(&context));
// Build operations using the arith and func dialects
```

**What it does NOT support well:**
- No high-level API for loop nest construction
- Affine dialect bindings may be incomplete
- Linalg dialect bindings may be incomplete
- No automatic lowering from Rust types to MLIR types

### Could Rust target MLIR instead of LLVM IR?

**Not today.** Rustc emits LLVM IR. There is no `rustc_codegen_mlir`. To use
MLIR, you would need to:

1. Emit MLIR from a code generator (not rustc)
2. Use MLIR's optimization passes
3. Lower MLIR to LLVM IR
4. Feed the lowered IR back into the LLVM pipeline

For polka-dots, this means: if you wanted MLIR-optimized morsel processing,
you would generate MLIR for the hot kernel code outside of rustc, compile it
separately, and call it from Rust via FFI. This is the approach used by
database systems like Umbra and HyPer.

### MLIR in database query compilation

Recent academic work (Jungmair et al., VLDB 2022: "Designing an Open Framework
for Query Optimization and Compilation") demonstrates using MLIR for database
query compilation. Key findings:

- MLIR "significantly reduces the efforts needed to add new intermediate
  representations" for query compilation
- Cross-domain optimization (relational + ML + UDF) becomes viable
- Multiple dialects can coexist in a single module, enabling domain-specific
  optimization at each level before lowering

This is directly relevant to polka-dots' pipeline model: each pipeline stage
could emit domain-specific MLIR, optimize at that level, then lower.

### Verdict

MLIR is the most promising long-term path for domain-specific optimization of
morsel processing. However, the Rust integration is immature. The practical path:

1. **Near term:** Use LLVM passes via `-C passes` and `-C llvm-args`
2. **Medium term:** Generate MLIR for hot kernels via Melior, compile
   separately, link via FFI
3. **Long term:** A `rustc_codegen_mlir` backend (does not exist, would be
   a significant project)

## 7. Domain-specific optimizers

### BLIS (BLAS-like Library Instantiation Software)

**What it is:** An award-winning framework for implementing high-performance
dense linear algebra. Won the 2023 J. H. Wilkinson Prize for Numerical Software.

**The micro-kernel architecture:**
BLIS decomposes all level-3 BLAS operations (matrix-matrix multiply, etc.)
into a single "micro-kernel" -- a tiny, hand-optimized routine (typically
written in assembly) that multiplies small register-resident blocks. Everything
else (cache blocking, memory packing, parallelization) is composed above the
micro-kernel in portable C.

Key insight: by isolating the architecture-dependent code into a single ~50-line
assembly routine, porting to a new architecture requires writing only one
micro-kernel. The entire BLAS implementation follows automatically.

**Applicability to morsel processing:**
The BLIS model maps loosely to morsel execution:
- Micro-kernel = the innermost loop body processing a single morsel column
- Cache blocking = morsel sizing (fit working set in L2)
- Memory packing = column layout optimization (contiguous, aligned)

However, BLIS is specifically about matrix multiplication. The "micro-kernel"
concept generalizes, but BLIS itself is not directly usable for filter/project/
aggregate operations. The design PRINCIPLE is transferable: isolate the
architecture-dependent innermost loop, write it once per target, compose
everything else portably above it.

In Rust, this maps to: write the hot inner loop with `#[target_feature]` and
`std::arch` intrinsics for each ISA (NEON, AVX2, AVX-512), dispatch at
runtime based on CPU detection. This is already the established pattern in
Polars and arrow-rs.

### Halide

**What it is:** A language embedded in C++ for writing high-performance image
processing pipelines. Its key innovation is separating ALGORITHM (what to
compute) from SCHEDULE (how to compute it: tiling, vectorization, parallelism,
fusion).

**Scheduling primitives:**
- `tile(x, y, xo, yo, xi, yi, 8, 8)` -- cache-friendly tiling
- `vectorize(xi, 8)` -- SIMD vectorization
- `parallel(yo)` -- multi-threaded parallelism
- `compute_at(consumer, yo)` -- fusion (compute producer inside consumer)
- `store_at(consumer, yo)` -- storage optimization
- `reorder(y, x)` -- loop interchange
- `unroll(xi)` -- loop unrolling

**Rust bindings:**
- `halide-runtime` crate: runtime bindings for executing pre-compiled Halide
  kernels from Rust. Does NOT support defining Halide algorithms in Rust.
- No full Rust frontend exists. You would define the algorithm and schedule
  in C++, compile to a static library, and call from Rust via FFI.

**Applicability to morsel processing:**
Halide's schedule model maps remarkably well to the morsel execution model:
- Algorithm = the pipeline operator chain
- Schedule = morsel sizing, vectorization width, core assignment, fusion
  decisions
- `compute_at` = operator fusion within a chain
- `tile` = morsel blocking for cache

The limitation: Halide is designed for stencil computations (image processing),
where each output depends on a neighborhood of inputs. Columnar data processing
is mostly map/filter/aggregate, which are simpler patterns. Halide is overkill
for these but exactly right for any spatial/temporal operations.

### Apache TVM

**What it is:** An ML compiler framework that auto-tunes operator
implementations across hardware targets.

**Rust integration:**
- `tvm-rt` crate: Rust bindings to TVM's runtime API
- `tvm-sys` crate: Raw FFI bindings
- `tvm` crate: Higher-level compiler + runtime bindings
- `tvm_graph_rt`: Graph executor in pure Rust

These are runtime bindings, not a Rust frontend. Algorithm definition happens
in Python; Rust invokes pre-compiled TVM modules.

**Auto-tuning model:**
TVM generates many candidate implementations of an operation (varying tile
sizes, vectorization widths, loop orderings), benchmarks each on the target
hardware, and selects the fastest. The tuning results are stored in a log file
and reused for subsequent compilations.

**Applicability:**
- Auto-tuning morsel kernel implementations is attractive
- TVM is heavily ML-focused (tensor operations, convolutions)
- Non-ML workloads (filter, project, aggregate) are not first-class
- The overhead of the TVM runtime may exceed the kernel itself for small data

### ISL (Integer Set Library)

**What it is:** The C library underlying Polly's polyhedral model. Manipulates
sets and relations of integer points bounded by linear constraints.

**Operations:**
- Set and relation construction from constraints
- Affine transformation of iteration domains
- Dependence analysis
- Schedule computation (finding optimal loop orderings)
- Code generation from polyhedral models

**Rust bindings:** None found on crates.io. A Nim wrapper exists
([nim-isl](https://github.com/mratsim/nim-isl)). A Rust wrapper would need
to be written or generated from ISL's C API.

**Applicability:**
ISL could be used at build time or init time to compute optimal loop schedules
for morsel processing based on the detected cache hierarchy. For example:
- Input: column count, column width, L1 size, L2 size, vector width
- Output: optimal morsel size, tile dimensions, loop ordering

However, this is compile-time scheduling, which assumes the workload is known
at compile time. For a general-purpose data engine, the workload varies at
runtime. ISL's value would be in pre-computing a TABLE of optimal
configurations indexed by workload parameters, then looking up at runtime.

### MLIR Affine dialect (as optimizer, not backend)

MLIR's affine dialect provides polyhedral optimization at a higher level than
LLVM IR, similar to ISL but integrated into the MLIR infrastructure. Via
Melior (Rust MLIR bindings), you could:

1. Express a morsel kernel as an affine loop nest
2. Apply MLIR's affine optimization passes (tiling, fusion, vectorization)
3. Lower to LLVM dialect
4. JIT-compile or emit native code

This is the same approach used by database query compilers (Umbra, HyPer)
but at a lower level of integration.

### Summary table

| System | Core idea | Rust integration | Fit for polka-dots |
|--------|-----------|-----------------|-------------------|
| BLIS | Micro-kernel isolation | Principle only | Design pattern, not library |
| Halide | Algorithm/schedule separation | FFI only | Good fit conceptually; overkill for simple ops |
| TVM | Auto-tuned operator compilation | Runtime bindings | ML-focused; limited for data ops |
| ISL | Polyhedral scheduling math | No bindings | Build-time schedule computation |
| MLIR Affine | Polyhedral optimization in IR | Melior (alpha) | Best long-term path |

## 8. Rust-specific optimization tooling

### cargo-show-asm

**What:** View assembly, LLVM IR, MIR, or WASM output per function.

**Key features:**
- Partial function name matching
- Source-assembly interleaving
- llvm-mca analysis for performance insights
- Supports release and dev profiles
- Intel and AT&T assembly syntax
- `--context` flag to include related functions recursively

**Usage:**
```bash
cargo install cargo-show-asm
cargo asm --lib polka_pipeline::morsel::process  # assembly
cargo asm --lib --llvm polka_pipeline::morsel::process  # LLVM IR
cargo asm --lib --mir polka_pipeline::morsel::process  # MIR
```

**Can it show LLVM pass effects?** Not directly. But by comparing LLVM IR
output with different `-C passes` flags, you can observe pass effects.

### cargo-llvm-lines

**What:** Counts lines of LLVM IR per generic function instantiation.
Identifies monomorphization bloat.

**Output format:** `Lines | Copies | Function`

**Usage:**
```bash
cargo install cargo-llvm-lines
cargo llvm-lines --lib -p polka-pipeline | head -20
```

**With LTO for full picture:**
```bash
CARGO_PROFILE_RELEASE_LTO=fat cargo llvm-lines --release
```

**Optimization strategies identified by this tool:**
- High copies: extract non-generic internals, use `#[inline(never)]`
- High lines: consider `dyn Trait` for cold paths (NOTE: polka-dots bans dyn)
- High lines from std: check if a simpler API exists

**Relevance:** Critical for polka-dots. With 17 crates and heavy use of
generics (Column<T>, WorkUnit, etc.), monomorphization bloat is a real risk.
This tool identifies which generic functions produce the most LLVM IR,
directly impacting compile time and binary size.

### cargo-bloat

**What:** Analyzes binary size by function and by crate.

**Features:**
- Function-level size ranking
- Crate-level aggregation (`--crates` flag)
- Regex filtering
- JSON output for programmatic analysis
- Supports ELF, Mach-O, PE (not WASM -- use twiggy)

**Usage:**
```bash
cargo install cargo-bloat
cargo bloat --release -n 20  # top 20 functions by size
cargo bloat --release --crates  # size by crate
```

### Miri

**What:** Interpreter for Rust's MIR. Detects undefined behavior.

**NOT for optimization analysis.** Miri detects:
- Out-of-bounds access, use-after-free
- Uninitialized data use
- Alignment violations
- Data races
- Aliasing violations (Stacked/Tree Borrows)

**Indirect relevance:** Miri can validate that `unsafe` code used for
optimization (e.g., `get_unchecked`, aligned pointer casts) is correct.

### Vectorization analysis tools

No dedicated Rust tool exists for analyzing vectorization friendliness.
Available approaches:

1. **cargo-show-asm:** Inspect assembly for vector instructions (vmov, vadd,
   vfma on ARM; vmovaps, vaddps on x86)
2. **LLVM remarks:** `-C llvm-args=-pass-remarks=loop-vectorize` prints
   vectorization decisions
3. **LLVM opt-viewer:** `-C llvm-args=-pass-remarks-output=remarks.yaml`
   generates machine-readable reports
4. **slipstream crate:** Provides vector types that nudge the compiler toward
   autovectorization
5. **Manual inspection:** Compare `-C opt-level=3` assembly with and without
   `-C no-vectorize-loops` to see what was vectorized

## 9. Profile-guided optimization and BOLT

### PGO workflow

PGO uses instrumented builds to collect runtime profiling data, then feeds
that data back to the compiler for informed optimization decisions.

**Steps:**
```bash
# 1. Build instrumented binary
RUSTFLAGS="-Cprofile-generate=/tmp/pgo-data" \
  cargo build --release

# 2. Run representative workload
./target/release/polka build  # typical usage

# 3. Merge profile data
llvm-profdata merge -o /tmp/pgo-data/merged.profdata /tmp/pgo-data/

# 4. Build optimized binary
RUSTFLAGS="-Cprofile-use=/tmp/pgo-data/merged.profdata" \
  cargo build --release
```

**What PGO optimizes:**
- Branch weights (hot/cold path identification)
- Function inlining decisions (inline hot callees, outline cold ones)
- Basic block layout (hot blocks together, cold blocks elsewhere)
- Function ordering (hot functions together for i-cache)

**Per-crate interaction:** PGO profiles are per-function. Functions from all
crates are profiled. The compiler uses profiles regardless of which crate
defines the function.

### BOLT (Binary Optimization and Layout Tool)

BOLT optimizes the binary AFTER linking. It reorders functions and basic blocks
for optimal instruction cache behavior.

**Workflow:**
```bash
# 1. Build with relocations preserved
RUSTFLAGS="-C link-arg=-Wl,--emit-relocs" cargo build --release

# 2. Instrument
llvm-bolt target/release/polka -o polka.inst \
  -instrument -instrumentation-file=/tmp/bolt.fdata

# 3. Run representative workload
./polka.inst build

# 4. Optimize
llvm-bolt target/release/polka -o polka.bolt \
  -data=/tmp/bolt.fdata -reorder-blocks=ext-tsp \
  -reorder-functions=hfsort -split-functions
```

**Tools:** [cargo-pgo](https://github.com/Kobzol/cargo-pgo) automates both
PGO and BOLT workflows.

### Relevance to per-crate optimization

PGO and BOLT are whole-program optimizations. They inherently cross crate
boundaries: a hot function in `polka-pipeline` called from `polka-cli` will be
optimized based on the call pattern. This complements per-crate flag selection:

- Per-crate flags optimize the COMPILATION of each crate
- PGO/BOLT optimize the LAYOUT of the final binary

Both can be used together.

## 10. Real-world per-crate optimization examples

### Bevy (game engine)

Recommended dev configuration:
```toml
[profile.dev]
opt-level = 1

[profile.dev.package."*"]
opt-level = 3
```

**Rationale:** User code (the game) needs fast recompile, so opt-level 1.
Dependencies (Bevy engine, rendering, physics) need good runtime performance,
so opt-level 3. Dependencies are compiled once and cached; rebuild cost is
paid only on version update.

Additional fast-build configuration in `.cargo/config_fast_builds.toml`:
- `mold` linker on Linux
- `-Zshare-generics=y` (nightly) -- share monomorphized generics across crates
- `-Zthreads=0` (nightly) -- parallel rustc frontend

Bevy does NOT use per-crate backend selection or custom LLVM passes.

### Polars (data engine)

```toml
[profile.release]
lto = "thin"
debug = "line-tables-only"

[profile.dist-release]
inherits = "release"
codegen-units = 1
lto = "fat"
```

**Rationale:** Standard release uses thin LTO (fast linking, good optimization).
Distribution release uses fat LTO + codegen-units=1 for maximum optimization
at the cost of long compile times.

Polars does NOT use per-package opt-level overrides or custom LLVM passes.
It relies on standard profiles and manual SIMD (`std::simd`, `std::arch`).

### DataFusion (query engine)

```toml
[profile.release]
codegen-units = 1
lto = true
strip = true

[profile.release-nonlto]
codegen-units = 16
lto = false
inherits = "release"

[profile.ci.package."*"]
debug = false
strip = "debuginfo"
```

**Rationale:** Release uses codegen-units=1 + fat LTO for maximum optimization.
`release-nonlto` provides fast builds for development. CI strips debug info
from all dependencies to minimize artifact size.

DataFusion does NOT use per-package optimization or custom passes.

### Nickel (language implementation)

```toml
[profile.dev.package.lalrpop]
opt-level = 3

[profile.release.package.lalrpop]
opt-level = 3

[profile.release]
lto = "fat"
codegen-units = 1
```

**Rationale:** lalrpop (parser generator) is extremely slow at opt-level 0.
Setting it to opt-level 3 in BOTH dev and release dramatically improves build
times because lalrpop's generated code benefits enormously from optimization.

This is the clearest example of per-package optimization solving a real problem.

### Common pattern

None of these projects use:
- Mixed codegen backends
- Custom LLVM passes
- Per-crate LLVM flag injection via wrapper
- Polly or polyhedral optimization

All of them use:
- Standard Cargo profiles (release, dev, custom)
- Per-package opt-level for slow-to-compile dependencies
- LTO (thin or fat) for release
- codegen-units=1 for distribution builds

## 11. Recommended strategy for polka-dots

### Immediate (works today, stable Rust)

```toml
# Cargo.toml (workspace root)

[profile.dev]
opt-level = 1  # basic optimization for usable dev performance

[profile.dev.package."*"]
opt-level = 2  # dependencies optimized once, cached

[profile.release]
opt-level = 3
lto = "thin"
codegen-units = 1
strip = "symbols"

[profile.release.package.polka-pipeline]
codegen-units = 1  # maximum optimization for hot crate

[profile.dist]
inherits = "release"
lto = "fat"
```

### Near-term (nightly, RUSTC_WORKSPACE_WRAPPER)

Create a wrapper script that applies per-crate optimization flags. Store
in `tools/rustc-wrapper.sh`, configure via `.cargo/config.toml`:

```toml
[build]
rustc-workspace-wrapper = "./tools/rustc-wrapper.sh"
```

The wrapper inspects `CARGO_PKG_NAME` and injects:
- `-C target-cpu=native` for pipeline and data crates
- `-C opt-level=s` for CLI
- `-C opt-level=1` for SDK/traits
- `-C llvm-args=-force-vector-width=8` for pipeline on AVX2 targets
- `-C passes=loop-distribute` for pipeline

### Medium-term (PGO + BOLT)

Use `cargo-pgo` for distribution builds. Profile data collected from
representative workloads (building a real dotfiles project). Expected
improvement: 5-15% for I/O-heavy CLI, 10-20% for pipeline execution
(branch prediction and layout optimization).

### Long-term (domain-specific optimization)

If morsel processing becomes a bottleneck:

1. Write hot kernels with `std::simd` and `std::arch` intrinsics
2. Consider generating MLIR (via Melior) for complex multi-column operations
3. Evaluate ISL for computing optimal tile sizes at init time
4. GPU offload via rust-gpu for very large datasets (saalis scale)

None of the domain-specific optimizers (Halide, TVM, BLIS, ISL) have mature
Rust integration. The micro-kernel PRINCIPLE from BLIS is the most immediately
transferable: isolate architecture-dependent inner loops, compose portably.
