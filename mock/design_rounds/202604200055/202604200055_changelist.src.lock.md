## Deliverables

Add to `mock/crates/hilavitkutin-ctx/src/lib.rs`:

### 1. `provider_generic!` macro — one type parameter

```rust
provider_generic!(<R: SomeBound> ApiTrait<R> as AccTrait<R> => method);
```

Generates:
- `pub trait AccTrait<R: SomeBound> { type Provider: ApiTrait<R>;
  fn method(&self) -> &Self::Provider; }`
- `impl<R: SomeBound, P: AccTrait<R>> Context<P> { pub fn method(&self)
  -> &<P as AccTrait<R>>::Provider { self.providers.method() } }`

### 2. `provider_generic2!` macro — two type parameters

```rust
provider_generic2!(<R: SomeBound, W: OtherBound>
                   ApiTrait<R, W> as AccTrait<R, W> => method);
```

Generates the two-parameter equivalents. Exists as a separate macro
(not a variadic macro) because Rust declarative macros can't match
optional type-parameter lists cleanly. Used by hilavitkutin-api's
`HasEach<R, W>` / `HasBatch<R, W>` / `HasReduce<R, W>`.

### 3. `tuple_generic!` macro — threads type parameter through tuple impls

```rust
tuple_generic!(
    <R: SomeBound>
    ApiTrait<R>: AccTrait<R> => method,
    OtherApi: OtherAcc => other_method,
);
```

Each position in the tuple declares whether it's the parameterised
provider (API-trait carries `<R>`) or a concrete one (no param).
The macro generates one impl per position, with the type parameter
threaded through only where needed.

Arity: 2-tuple and 3-tuple variants ship this round. Higher arities
can be added as consumers surface need. hilavitkutin-api's typical
Context tuple has ~5 slots (Reader<R>, Writer<W>, ResourceProvider<R>,
VirtualFirer<W>, OpRegistrar<R, W>) — this round ships up to arity 5.

### 4. `tuple_generic2!` macro — two type parameters

Same as `tuple_generic!` but for cases where two providers in the
same tuple are parameterised over different types. Ships arity 3-5.

### 5. Unit tests

- `tests/provider_generic.rs` — a toy access-set trait, one
  parameterised provider, confirm generic-method dispatch works.
- `tests/tuple_generic.rs` — tuple layouts mixing concrete +
  parameterised providers, arities 2 / 3 / 5. Call through
  `Context<P>` to verify delegation threads the type parameter.
- Negative compile tests are deferred (need `trybuild` which is
  heavy-weight for this size) — the API shape gets exercised by
  hilavitkutin-api's own tests in #34.

## Impl-time decisions (pre-committed)

1. **Separate `_generic` / `_generic2` macros.** Rust declarative
   macro_rules cannot match a variable-length type-parameter list
   cleanly. The least-friction shape is one macro per parameter
   count. If a 3+ parameter need surfaces later, add `_generic3`;
   so far ctx needs 1 and 2.

2. **Provider bound syntax.** `<R: Bound>` inside the macro
   invocation is the standard Rust generic-param syntax, visually
   consistent with type definitions. Bounds are mandatory
   (typically `R: AccessSet` or similar seal).

3. **Accessor method name kept on the generic form.** The accessor
   method (`fn method(&self) -> &Self::Provider`) does NOT take
   type parameters — only the trait does. Consumers hang their
   per-T logic on the API-trait's generic methods, not on the
   accessor.

4. **Context delegation threads only trait type params.** The
   generated `impl<R, P: AccTrait<R>> Context<P> { fn method... }`
   introduces `R` as an `impl`-level generic parameter. Call site
   `ctx.method()` has `R` inferred from the `P: AccTrait<R>` bound
   (i.e., from the provider tuple's declared impl).

5. **tuple_generic! arity 5 cap.** Covers WU Context's typical 5
   providers. Larger tuples will need additional macro arms; no
   attempt to make it fully variadic this round.

6. **No mixing with legacy `provider!` in same invocation.**
   Consumers with a mixed tuple (some concrete, some generic) use
   `tuple_generic!` for the whole tuple; non-parameterised
   positions pass an API-trait without a type parameter. The macro
   accepts both shapes at each position via declarative-match
   branching on the `<_>` presence.

7. **Existing `provider!` / `tuple!` / `define_providers!`
   unchanged.** Bytes identical to pre-round. Existing saalis /
   consumer callers remain compatible.
