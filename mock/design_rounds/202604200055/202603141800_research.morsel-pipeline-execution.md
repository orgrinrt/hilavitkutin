## 1. The Leis et al. 2014 Paper --- Morsel-Driven Parallelism

The foundation for morsel-driven execution is "Morsel-Driven Parallelism:
A NUMA-Aware Query Evaluation Framework for the Many-Core Age" by Viktor
Leis, Peter Boncz, Alfons Kemper, and Thomas Neumann, published at SIGMOD
2014 from the TU Munich HyPer group. This paper established the morsel as
a scheduling primitive and introduced NUMA-aware cooperative dispatch for
analytical query processing.

### 1.1 Core model

The paper decomposes query execution into two orthogonal axes: **what** to
execute (pipelines of chained operators) and **how** to schedule (morsel
dispatch with work-stealing).

A **pipeline** is a sequence of relational operators (scan, filter, project,
hash probe) that can run without materializing intermediate results between
operators. Data flows through the pipeline one tuple at a time (in HyPer's
compiled model) or one vector at a time (in vectorized models). Pipelines
end at **pipeline breakers**: operators that must consume all input before
producing any output. Hash build, sort, and group-by aggregation are the
canonical breakers.

A **morsel** is a fixed-size chunk of input tuples assigned atomically to
one worker thread. The paper uses approximately 10,000 tuples as the
default morsel size. This number was chosen empirically: large enough to
amortize scheduling overhead (task queue operations, counter updates), small
enough to enable fine-grained load balancing across cores. At 10K tuples
with typical 8-byte column widths, a morsel's working set is 80 KB per
column --- fitting comfortably in L2 cache (256 KB on contemporary Intel
hardware) while allowing multiple columns to coexist.

### 1.2 NUMA-aware dispatch

The paper's central contribution is NUMA-aware morsel assignment. On
multi-socket servers (the target hardware for HyPer), accessing memory
attached to another socket's memory controller incurs 2-4x latency penalty
(~80 ns local vs ~150-200 ns remote on Intel QPI / AMD HyperTransport
interconnects). The dispatcher maintains a topology map of which morsels
reside on which NUMA node and preferentially assigns morsels to worker
threads running on cores attached to the same node.

The dispatch mechanism uses a global dispatcher thread (single point of
scheduling decisions) that maintains per-pipeline morsel queues. When a
worker thread completes its current morsel, it requests the next one from
the dispatcher. The dispatcher selects a morsel from the same NUMA node
when possible, falling back to cross-node assignment only when the local
queue is empty (work-stealing across NUMA boundaries).

### 1.3 Elasticity and load balancing

A key property of morsel-driven scheduling is **elasticity**: the ability
to dynamically redistribute work in response to load imbalance. Because
morsels are small relative to the total dataset, the dispatcher can
reassign pending morsels from overloaded cores to idle ones. If one
pipeline finishes early on some cores, those cores can be redirected to
help with other active pipelines.

The paper demonstrates that morsel granularity creates a natural load
balancing effect. With 10K-tuple morsels on a 100M-row table, there are
10,000 morsels. Even with significant skew (some morsels taking 2x longer
due to filter selectivity variance), the large number of work items ensures
that load balances out statistically. This is a direct analogue of the
central limit theorem applied to task scheduling.

### 1.4 Why morsel size matters

The morsel size is a tension between two opposing forces:

- **Too small** (e.g., 100 tuples): scheduling overhead dominates. Each
  morsel requires an atomic queue operation, a dispatcher decision, and
  potential cache line bouncing on the shared morsel counter. At 100
  tuples with a 50 ns scheduling overhead, the overhead per tuple is
  0.5 ns --- comparable to the actual processing time for simple filters.
- **Too large** (e.g., 1M tuples): load imbalance. With 100M rows and
  1M-tuple morsels, there are only 100 work items for potentially 64+
  cores. Some cores idle while others process their large morsel. Worse,
  if one morsel contains disproportionately expensive rows (complex
  string operations, cache-missing hash probes), the entire core is
  blocked.

The 10K sweet spot gives ~10,000 work items for a 100M-row table, enough
for statistical balancing across even 128 cores, while keeping scheduling
overhead below 1% of processing time.

### 1.5 What survived into production

The morsel concept proved durable. Every major post-2018 analytical engine
adopted some variant:

- **Umbra** (HyPer's successor) kept morsels but moved from a global
  dispatcher thread to task-based scheduling, eliminating the dispatcher
  as a bottleneck.
- **DuckDB** adopted a fixed 2048-value vector size for cache efficiency
  and a separate 122,880-row morsel for parallel scheduling, decoupling
  the cache-fitting unit from the scheduling unit.
- **Velox** uses "splits" (morsel-equivalent chunks) with adaptive sizing
  based on operator complexity.
- **DataFusion** uses Arrow RecordBatch as its morsel analogue, with
  configurable batch sizes.

What did NOT survive unchanged: the global dispatcher thread (replaced by
distributed work-stealing in all successors), the fixed 10K morsel size
(all successors use adaptive or configurable sizing), and the assumption
that NUMA locality is the primary scheduling concern (modern systems
prioritize cache locality and task granularity over NUMA placement).

## 2. Post-2018 Database Execution Engines

### 2.1 Umbra (TU Munich, successor to HyPer)

Umbra, developed by the same TU Munich group (Thomas Neumann, Viktor Leis,
et al.), replaced HyPer as the research platform starting around 2019. The
key paper is "Umbra: A Disk-Based System with In-Memory Performance" (Neumann
and Freitag, CIDR 2020).

**Compilation model.** Like HyPer, Umbra generates native code for each
query, but with a critical improvement: **Flying Start** adaptive
compilation. HyPer compiled every query through LLVM before executing it,
which imposed a noticeable latency for the first result row (50-200 ms for
complex queries). Flying Start uses a tiered approach: queries begin
execution in an interpreter (bytecode-based, sub-millisecond startup),
and hot code paths are compiled to native code via LLVM in the background.
The compiled version replaces the interpreted version mid-execution. This
is architecturally identical to how JIT-compiled language runtimes (HotSpot
JVM, V8) handle tiered compilation.

**Task-based scheduling.** Umbra replaced HyPer's global dispatcher with
a task-based model. Each pipeline stage generates a set of tasks (one per
morsel), and tasks are placed in lock-free work-stealing deques (one per
worker thread). Workers process their local deque and steal from others
when idle. This eliminates the dispatcher as a single point of contention
and scales better beyond 64 cores.

**Variable morsel sizes.** Umbra introduced adaptive morsel sizing based on
operator complexity. Simple scan+filter pipelines use larger morsels (more
tuples per scheduling unit) because the per-tuple processing time is low
and scheduling overhead is relatively higher. Complex pipelines with
hash probes and string operations use smaller morsels because per-tuple
time is high and finer granularity improves load balance. The system
measures processing time per morsel and adjusts the size for subsequent
morsels in the same pipeline.

**Relevance to hilavitkutin.** The task-based scheduling model is directly
applicable. hilavitkutin's WorkUnit model is essentially Umbra's task
model: a self-contained unit of work placed on a queue, processed by a
worker, with dependencies tracked by the scheduler. The variable morsel
sizing validates our design decision (from the memory layout topic, section
3) to make morsel size a function of column width and L1 cache size rather
than a fixed constant.

### 2.2 DuckDB

DuckDB is an in-process analytical database written in C++, targeting the
same use case as SQLite but with columnar storage and vectorized execution.
The key papers are "DuckDB: an Embeddable Analytical Database" (Raasveldt
and Muehleisen, SIGMOD 2019) and subsequent work on parallel execution.

**Push-based vectorized execution.** DuckDB processes data in vectors of
2048 values (the `STANDARD_VECTOR_SIZE` compile-time constant). This size
was chosen to fit in L1 cache: 2048 values at 8 bytes = 16 KB, well within
even the smallest 32 KB L1 data caches. Unlike HyPer's tuple-at-a-time
compilation, DuckDB's vectorized model processes one vector at a time
through each operator, amortizing interpretation overhead across 2048
values while maintaining cache residency.

The execution model is push-based within a pipeline: the source operator
pushes vectors downstream through filter, project, and probe operators.
I/O is pull-based: the source operator pulls from the storage layer (page
cache, Parquet reader) on demand. This hybrid gives the system backpressure
control at the I/O boundary while maximizing throughput within the
computational pipeline.

**Parallel hash join.** DuckDB's approach to parallel hash table operations
is instructive. The build phase partitions the build-side input into thread-
local hash table segments using radix partitioning. Each thread builds its
segment independently (no synchronization). After build completes, segments
are merged into a shared hash table. The probe phase is embarrassingly
parallel: each thread probes its morsel of probe-side input against the
shared (read-only) hash table.

**Pipeline breaker handling.** DuckDB treats pipeline breakers as pipeline
boundaries. The build phase of a hash join is one pipeline; the probe phase
is a separate pipeline. The scheduler creates both pipelines, connects them
with a dependency edge (probe depends on build), and executes build to
completion before starting probe. The hash table is the shared resource
that crosses the pipeline boundary.

This is directly relevant to hilavitkutin's DAG model: pipeline breakers
are DAG edges. A build WorkUnit writes `Write<HashTable<T>>`, a probe
WorkUnit reads `Read<HashTable<T>>`, and the scheduler infers the ordering
from these declarations.

**Relevance to hilavitkutin.** DuckDB validates two design choices: (1)
decoupling the cache-fitting unit (vector of 2048) from the scheduling
unit (morsel of 122,880) --- hilavitkutin's morsel sizing (from the memory
layout topic) makes the same separation; (2) treating pipeline breakers as
DAG edges rather than special-cased synchronization points. DuckDB is also
the closest analogue for saalis's use case (in-process analytical queries
over a catalogue of game library entries).

### 2.3 Velox (Meta)

Velox is not a database but an expression evaluation and operator execution
framework, open-sourced by Meta in 2022. The key paper is "Velox: Meta's
Unified Execution Engine" (Pedreira et al., VLDB 2022). It serves as the
execution backend for Presto (via Prestissimo) and Spark (via Gluten).

**Operator model.** Velox defines operators (TableScan, Filter, HashJoin,
Aggregate) that process data in Arrow-compatible columnar format (RowVector
and its typed variants). Operators form a tree (not a flat pipeline), and
execution follows a pull model: the root operator pulls from children, which
pull from their children recursively.

**Memory arbitration.** Velox's most distinctive feature is its memory
arbitration system. Operators negotiate for memory through a central
arbitrator. When an operator needs more memory (e.g., a hash build growing
its hash table), it requests allocation from the arbitrator. If memory is
tight, the arbitrator can ask other operators to spill to disk or shrink
their buffers. This provides system-wide memory control without requiring
each operator to implement its own memory management.

The arbitration protocol is cooperative: operators implement a `reclaim()`
callback that the arbitrator can invoke to request memory release. This is
conceptually similar to hilavitkutin's Progress::Yield model --- an
external signal asks the work unit to cooperatively release resources.

**Adaptive processing.** Velox adaptively switches between row-wise and
columnar processing based on data characteristics. For highly selective
filters (few rows pass), row-wise processing avoids materializing full
output vectors. For low-selectivity filters, columnar processing amortizes
interpretation overhead. The decision is made per-batch based on observed
selectivity.

**Relevance to hilavitkutin.** The memory arbitration model is interesting
for saalis's use case (large catalogue queries that may need memory
budgeting), but hilavitkutin's target data volumes (KB to low MB for polka,
sub-GB for saalis) make it unnecessary. The adaptive processing concept
validates our morsel sizing decision: different work characteristics
warrant different processing granularities.

### 2.4 DataFusion (Apache Arrow, Rust)

DataFusion is an extensible query execution framework written in Rust,
part of the Apache Arrow ecosystem. It is the closest existing system to
hilavitkutin in terms of language, ecosystem, and design philosophy.

**Volcano-style iterator model.** DataFusion uses a pull-based iterator
model: each operator implements `ExecutionPlan::execute()` which returns a
`SendableRecordBatchStream` (a Rust `Stream` of Arrow `RecordBatch`
values). The root operator pulls batches from children, which pull from
their children. This is the classic Volcano model (Graefe, 1994) adapted
for Rust's async ecosystem.

**Async integration.** DataFusion is deeply integrated with Tokio. Each
operator's `execute()` method is async, and the stream it returns can
yield (via `.await`) when waiting for I/O (Parquet reads, network
fetches). The Tokio runtime handles scheduling of these async tasks across
a thread pool. This provides cooperative multitasking for I/O-bound
operators without any custom scheduling infrastructure.

**RecordBatch as morsel.** Arrow's `RecordBatch` serves as DataFusion's
morsel equivalent. The default batch size is 8192 rows (configurable via
`SessionConfig::with_batch_size()`). This is larger than DuckDB's 2048
because Arrow's RecordBatch has higher per-batch overhead (schema
validation, reference counting) that needs to be amortized over more rows.

**How async I/O interacts with pipeline operators.** This is directly
relevant to hilavitkutin's Q2 (yielding model). In DataFusion, an operator
that needs to do I/O simply `.await`s within its stream implementation. The
Tokio runtime suspends the task and resumes it when I/O completes. From the
operator's perspective, execution is synchronous --- it writes code as if
I/O is blocking. From the scheduler's perspective, the thread is freed to
do other work while the operator waits.

The cost of this approach: every operator must be async (colored function
problem), the entire system depends on Tokio (not no_std compatible), and
async state machines have implicit heap allocation for each suspended
future. For DataFusion's use case (server-side query processing), these
costs are acceptable. For hilavitkutin's constraints (no_std, three
different consumers), they are not.

**Relevance to hilavitkutin.** DataFusion demonstrates that the Volcano/
pull model works well in Rust with async, but also demonstrates the costs:
Tokio dependency, colored functions, no no_std path. hilavitkutin's
explicit Progress::Yield model avoids these costs at the expense of
ergonomics (manual state machines in work units). DataFusion's configurable
batch size validates our approach of making morsel size a runtime parameter
rather than a compile-time constant.

### 2.5 Polars (Rust)

Polars is a DataFrame library written in Rust, designed for single-machine
analytical workloads. It is relevant both as a Rust execution engine and as
a system that handles ownership and lifetime challenges specific to Rust's
type system in a columnar context.

**Lazy evaluation.** Polars separates query specification (lazy API) from
execution (collect). The lazy API builds a logical plan that is optimized
(predicate pushdown, projection pushdown, common subexpression elimination)
before execution. This is analogous to hilavitkutin's DAG: work units
declare their dependencies, the scheduler builds the execution plan, and
execution happens after planning.

**Streaming execution.** For larger-than-memory datasets, Polars supports
streaming execution where data flows through operators in chunks without
full materialization. The chunk size is computed from a formula:

```
thread_factor = max(12 / n_threads, 1)
chunk_size = max(50_000 / n_cols * thread_factor, 1000)
```

This formula accounts for data width indirectly (more columns = smaller
chunks to fit in cache) and thread count (fewer threads = larger chunks
to reduce scheduling overhead). No hardware detection is involved.

**Rayon for parallelism.** Polars uses Rayon's work-stealing thread pool
for parallel execution. `par_iter()` over chunks provides automatic load
balancing. This is a pragmatic choice that leverages Rust's ecosystem
rather than building a custom scheduler. The downside is that Rayon's
scheduling is general-purpose and cannot exploit domain-specific knowledge
(e.g., NUMA locality, operator complexity) the way Umbra's task scheduler
does.

**Rust-specific concerns.** Polars addresses several challenges that are
unique to Rust and relevant to hilavitkutin:

- **Ownership of columnar data.** Arrow2 (Polars' Arrow implementation)
  uses reference-counted buffers (`Arc<dyn Array>`) to allow zero-copy
  sharing of column data across operators. This avoids the clone-heavy
  patterns that naive Rust ownership would require.
- **Lifetimes in iterators.** Polars' chunk iterators borrow the
  underlying column data, requiring careful lifetime annotations.
  The streaming engine uses owned chunks (`DataFrame` values) rather
  than borrowed slices to avoid lifetime entanglement across thread
  boundaries.
- **Send + Sync bounds.** All data structures that cross thread
  boundaries must be `Send + Sync`. Polars achieves this through
  `Arc<dyn Array>` (which is `Send + Sync`) rather than `&[T]` slices
  (which are `Send + Sync` but cannot outlive the borrow).

**Relevance to hilavitkutin.** Polars validates the Rayon-based work-
stealing approach as viable for Rust columnar processing. Its chunk sizing
formula is a useful data point alongside DuckDB's fixed constant and
Umbra's adaptive sizing. The Rust-specific ownership patterns (Arc buffers,
owned chunks across thread boundaries) are directly applicable to
hilavitkutin's ColumnStorage trait design.

## 3. Game Engine Job Systems

Database engines and game engines solve the same fundamental problem:
parallel processing of data through a dependency graph, with the scheduler
maximizing throughput while respecting ordering constraints. The difference
is the constraint profile: databases optimize for throughput (process all
rows as fast as possible), while game engines optimize for latency (finish
all work within a fixed frame budget, typically 16.6 ms for 60 fps or
8.3 ms for 120 fps).

### 3.1 Naughty Dog GDC 2015: Fiber-Based Job System

The reference talk is "Parallelizing the Naughty Dog Engine Using Fibers"
by Christian Gyrling, presented at GDC 2015. This talk established fibers
as the dominant job system architecture in AAA game development and
directly influenced every subsequent engine's approach.

**Fibers.** A fiber is a user-space cooperative thread: it has its own
stack (Naughty Dog uses 64 KB stacks on PS4, with ~160 fibers in the
pool) but no OS thread backing it. Fibers are multiplexed onto a fixed
pool of worker threads (one per hardware core, typically 6 on PS4).
Context switching between fibers is a register save/restore (tens of
nanoseconds), not an OS context switch (microseconds).

**Atomic counter dependencies.** The core scheduling primitive is the
atomic counter. When job A depends on jobs B and C, A's counter is
initialized to 2. When B completes, it atomically decrements A's
counter. When C completes, it decrements again. The decrement that
reaches zero enqueues A for execution. No locks, no condition variables,
no mutexes --- just atomic integers and a lock-free job queue.

The sequence for a dependent job:

1. Create counter, set to N (number of dependencies).
2. Attach counter to the waiting job.
3. Launch N prerequisite jobs, each carrying a pointer to the counter.
4. Each prerequisite job, on completion, does `counter.fetch_sub(1)`.
5. If `fetch_sub` returns 1 (meaning it just hit zero), the completing
   job enqueues the waiting job onto the job queue.

This is wait-free for the common case (counter > 1 after decrement) and
lock-free for the triggering case (counter == 0, enqueue the dependent
job).

**Yield-on-dependency.** The critical insight: when a fiber encounters
a dependency that is not yet satisfied (counter > 0), it yields. The
fiber's stack is frozen in place, and the worker thread picks up another
ready fiber from the queue. When the dependency is later satisfied (counter
reaches zero), the yielded fiber is re-enqueued and will be resumed by
whichever worker thread picks it up.

This transforms blocking dependencies into cooperative scheduling. A fiber
that needs to wait for a GPU readback, a file I/O completion, or another
job's result simply yields and pays no thread cost while waiting. The worker
thread is never idle --- it always has a fiber to run.

**Work-stealing.** Each worker thread has a local job deque (double-ended
queue). Jobs are pushed to the back and popped from the back (LIFO for
cache locality). When a thread's local deque is empty, it steals from the
front of another thread's deque (FIFO, stealing the oldest/coldest jobs).
This is the Chase-Lev work-stealing deque, the same data structure used
by Rayon, Tokio, and most modern work-stealing schedulers.

**Direct parallel to Progress::Yield.** hilavitkutin's `Progress::Yield(
Pending)` is the non-fiber version of Naughty Dog's yield-on-dependency.
The semantic is identical: "I cannot make progress right now; resume me
when my dependency is satisfied." The implementation differs: Naughty Dog
freezes a fiber stack (implicit state preservation), hilavitkutin requires
the work unit to store its continuation state in a struct (explicit state
preservation). The explicit approach is less ergonomic (the work unit
author must manage the state machine) but more portable (no platform-
specific stack switching, no_std compatible).

### 3.2 Molecular Musings Job System

Stefan Reinalter's blog series (2012-2015) on job system design at
Molecular Musings (later The Machinery) is a widely-referenced practical
guide to job system implementation. The key posts cover task granularity,
lock-free deques, and structured concurrency.

**Task granularity.** Reinalter's analysis identifies the same tension as
Leis et al.: too-fine tasks are dominated by scheduling overhead, too-
coarse tasks cause load imbalance. His empirical recommendation for game
engines: tasks should take 100-1000 microseconds. Below 100 us, the
overhead of deque operations and cache pollution from stealing outweighs
the parallelism benefit. Above 1000 us, one long task can block a core
for a significant fraction of the frame budget.

This maps to hilavitkutin's morsel sizing: at our 16-byte stride with
L1-fitting morsels (~1000-2000 entries), per-morsel processing time for
simple column operations is in the 10-50 us range. For complex operations
(hash probe, string comparison), it rises to 100-500 us. Both are within
Reinalter's recommended range, validating our morsel size derivation.

**Parent-child relationships.** Reinalter introduces structured concurrency
for jobs: a parent job spawns child jobs and waits for their completion.
The parent's counter is incremented for each child spawned, decremented
when each child completes. This provides automatic fan-out/fan-in without
explicit dependency graph construction.

This pattern appears in hilavitkutin's morsel dispatch: a column transform
WorkUnit processes one morsel, but the scheduler fans out multiple instances
(one per morsel of input data) and tracks completion via a counter. When
all morsels are processed, downstream work units become eligible. The
counter mechanism is identical to Reinalter's parent-child model.

### 3.3 Our Machinery / The Machinery

The Machinery (Reinalter's later game engine, discontinued in 2023) added
frame-budget awareness to the job system. Tasks declare an expected
duration, and the scheduler uses these estimates to pack tasks into the
frame budget.

**Budget-aware scheduling.** Each task carries a cost estimate (in
microseconds). The scheduler maintains a running total of allocated time
per frame and rejects or defers tasks that would exceed the frame budget.
Deferred tasks are scheduled in the next frame, with priority boosting
to prevent starvation.

This is directly relevant to hilavitkutin's scheduling groups with cadence.
Saalis's Curator subsystem (cadence: 1 hour) and Doctor subsystem (cadence:
5 minutes) are the long-period equivalent of frame-budget allocation. The
scheduler does not run Curator work every tick; it runs it when the cadence
timer expires and there is scheduling budget available. The Machinery's
approach of cost estimation plus budget tracking provides a concrete
implementation model for this.

### 3.4 Unity DOTS (Data-Oriented Technology Stack)

Unity's DOTS/ECS is the most architecturally relevant game engine system
for hilavitkutin because it derives dependency DAGs from component access
declarations --- exactly the model hilavitkutin proposes.

**Component access declarations.** In Unity DOTS, each system (the
equivalent of a WorkUnit) declares which component types it reads and
writes via type annotations:

```csharp
[BurstCompile]
partial struct MovementSystem : ISystem {
    [ReadOnly] public ComponentLookup<Position> positions;
    public ComponentLookup<Velocity> velocities;
}
```

The job scheduler inspects these declarations at system registration time
and automatically constructs a dependency graph. Two systems that both
read Position can run in parallel. A system that writes Position must run
before or after any system that reads or writes Position. The programmer
never explicitly specifies "system A depends on system B" --- the scheduler
infers it from the data access patterns.

This is EXACTLY the model hilavitkutin uses: `type Read = (Position,
Velocity)` and `type Write = (Acceleration,)` declarations on a WorkUnit
generate DAG edges. Two units that both declare `Read<Position>` have no
ordering constraint. A unit that declares `Write<Position>` is serialized
against all other Position accessors.

**Chunk-based iteration.** Unity stores entities in archetypes (groups of
entities with the same set of component types). Each archetype's data is
stored in 16 KB chunks. Systems iterate over chunks, processing all
entities in a chunk before moving to the next. This is the morsel
equivalent: each chunk is a scheduling unit, and the job system fans out
one task per chunk.

The 16 KB chunk size is fixed and corresponds to fitting in L1 cache. This
is a conservative choice (identical reasoning to DuckDB's 2048 vector size)
that works across all target hardware without runtime detection. Our morsel
sizing (from the memory layout topic) is more adaptive, computing morsel
size from column width and L1 cache size, but the principle is the same.

**Burst compiler.** Unity's Burst compiler takes C# job code and compiles
it to LLVM-optimized native code, similar to HyPer/Umbra's query
compilation. Burst enforces restrictions (no managed allocations, no
virtual calls, no reference types) that guarantee the compiled code is
SIMD-friendly and cache-efficient. These restrictions are conceptually
similar to our `repr(C)` requirement on pipeline data types: both
constrain the type system to enable aggressive optimization.

### 3.5 Unreal Engine 5

Unreal Engine 5's task system is less architecturally novel than Unity
DOTS but provides useful patterns for task dependencies and named
scheduling groups.

**TaskGraph.** UE5's `FTaskGraphInterface` manages a DAG of tasks with
explicit prerequisites. The `CreateAndDispatchWhenReady` pattern allows a
task to declare its prerequisites as a list of task handles:

```cpp
FGraphEventArray Prerequisites;
Prerequisites.Add(TaskA);
Prerequisites.Add(TaskB);
TGraphTask<FMyTask>::CreateTask(&Prerequisites)
    .ConstructAndDispatchWhenReady(args...);
```

Unlike Unity DOTS (where the DAG is inferred from data access), UE5
requires explicit dependency specification. This is the traditional
approach and is more flexible (can express arbitrary ordering constraints
that are not data-driven) but less safe (the programmer can forget a
dependency, causing a data race).

hilavitkutin's declarative approach (infer DAG from Read/Write) is safer
and more aligned with Unity DOTS. However, the UE5 pattern shows why
virtual columns / signal edges (hilavitkutin Q3) are useful: some
dependencies are control-flow, not data-flow, and cannot be inferred from
data access declarations alone.

**Named threads.** UE5 has dedicated named threads (GameThread,
RenderThread, RHIThread) alongside the generic task pool. Work that
must run on a specific thread (e.g., rendering commands on the render
thread) is queued to that thread's dedicated queue. This is analogous to
hilavitkutin's scheduling groups: a subsystem that must run on a specific
thread (e.g., a UI subsystem pinned to the main thread) can be assigned
to a named scheduling group with affinity constraints.

## 4. Cooperative Multitasking Approaches Compared

Four approaches to cooperative scheduling within a pipeline, evaluated
against hilavitkutin's constraints: no_std compatibility, three different
consumers with different I/O profiles, and deterministic debuggability.

### 4.1 Coroutine-based (C++20 coroutines, Rust async/await)

**Mechanism.** The compiler transforms an async function into a state
machine. Each `.await` point becomes a state transition. The function's
local variables are stored in a compiler-generated struct (the "future"
in Rust, the "coroutine frame" in C++). An executor polls the state
machine, advancing it through states until completion.

In Rust, this manifests as the `Future` trait with `poll(self: Pin<&mut
Self>, cx: &mut Context) -> Poll<Output>`. The `Pin` requirement prevents
the future from being moved after it has been polled (because internal
self-references may exist). The executor (Tokio, async-std, smol) manages
a thread pool and schedules futures across worker threads.

**Pros.** Composable: async functions can call other async functions,
and the compiler generates the composed state machine automatically. The
programmer writes sequential-looking code with `.await` at suspension
points. Async I/O integration is native: `TcpStream::read().await`
suspends the task until data arrives without blocking the thread.

**Cons for hilavitkutin.**

- *Requires a runtime.* Rust's async is poll-based and needs an executor
  to drive futures to completion. Tokio is the de facto standard but is
  not no_std compatible. no_std executors exist (embassy, async-executor)
  but add significant complexity and may not support all needed I/O
  primitives.
- *Colored function problem.* Once a function is async, all callers must
  also be async (or use `block_on`, which defeats the purpose). This
  virally infects the entire call chain. hilavitkutin's WorkUnit::execute()
  would need to be async, forcing all three consumers to handle async
  ergonomics even when their work is purely synchronous.
- *Heap allocation.* Each suspended future is heap-allocated (unless the
  compiler can prove it is always polled to completion without
  suspension, enabling stack allocation). For morsel processing where
  work units are dispatched thousands of times per pipeline execution,
  the allocation pressure is non-trivial.
- *Debuggability.* Async state machines are opaque: the compiler-generated
  struct has mangled names, and inspecting the current state requires
  understanding the generated code. Stack traces through suspended
  futures are notoriously unhelpful.

**Verdict: NOT suitable** for hilavitkutin's constraints.

### 4.2 Fiber-based (Naughty Dog, Boost.Fiber)

**Mechanism.** User-level stack switching. Each fiber has its own stack
(allocated at fiber creation, typically 2-64 KB). A fiber yields by
saving its register state and switching to another fiber's stack. The
switch is typically implemented in platform-specific assembly (x86-64,
AArch64) or via OS facilities (Windows fibers, POSIX `ucontext`).

**Pros.** No colored functions: any function in the call stack can yield,
and the entire stack is preserved. Existing synchronous code works
unchanged; only the yield points are new. Yield is cheap: a register
save/restore is ~20-50 ns, comparable to a function call.

**Cons for hilavitkutin.**

- *Stack memory per fiber.* Each fiber needs its own stack. At 64 KB per
  fiber with 160 fibers (Naughty Dog's PS4 configuration), that is
  10 MB of stack memory. For a library that targets embedded-adjacent
  use cases (no_std), pre-allocating megabytes of stack space is
  undesirable.
- *Platform-specific assembly.* Stack switching requires per-architecture
  implementation: separate assembly for x86-64, AArch64, RISC-V, WASM.
  Crates like `context` and `may` provide this but are not no_std
  compatible and require OS thread-local storage.
- *no_std hostile.* Stack allocation requires either a heap allocator
  (for dynamic stacks) or static stack arrays (for fixed stacks).
  `ucontext` is a POSIX API not available on bare-metal. Windows
  fibers require kernel32.dll. WASM has no stack switching primitive.
- *Debuggability concerns.* While better than async (the stack is a
  real stack, so stack traces work), fiber-based code is harder to
  reason about than explicit state machines because the yield point
  can be anywhere in the call stack.

**Verdict: NOT suitable** for hilavitkutin's no_std + cross-platform
constraints. Fibers are the right choice for a single-platform game
engine with known hardware (Naughty Dog targets PS4/PS5 exclusively).
They are the wrong choice for a portable library.

### 4.3 Explicit yield (Progress::Yield model)

**Mechanism.** The work unit's `execute()` method returns an enum:

```
enum Progress {
    Complete,
    Yield(YieldReason),
    Failed(Error),
}
```

The scheduler calls `execute()` repeatedly until it returns `Complete` or
`Failed`. Between calls, the scheduler can run other work units. The work
unit stores its continuation state in its own struct fields --- effectively
a hand-written state machine.

**Pros.**

- *no_std compatible.* No runtime, no executor, no stack allocation.
  The work unit is a struct with `execute() -> Progress`. The scheduler
  is a loop that calls `execute()` on ready work units. Both compile
  to no_std without modification.
- *No colored functions.* `execute()` is a regular function. It can call
  other regular functions. There is no async infection.
- *Fully portable.* No platform-specific code. No assembly. Compiles to
  any target Rust supports, including WASM and bare-metal.
- *Debuggable.* The work unit's state is a struct with named fields.
  You can `println!("{:?}", unit)` to see exactly where it is in its
  processing. You can serialize the state, log it, checkpoint it. The
  state is not hidden in a compiler-generated future or a frozen stack
  frame.
- *Deterministic.* Given the same input and the same scheduling order,
  the work unit produces the same sequence of Progress values. This
  makes testing trivial: call execute() in a loop and assert the
  sequence.

**Cons.**

- *Manual state machines.* The work unit author must track continuation
  state explicitly. A work unit that does three sequential I/O calls
  needs an enum with three states and a match in execute() that
  dispatches based on the current state. This is the same code the
  compiler generates automatically for async functions, but the
  programmer writes it by hand.
- *Cannot yield from nested calls.* If `execute()` calls a helper
  function that discovers it needs to yield, the helper cannot yield
  directly. It must return a result that `execute()` interprets as a
  yield signal. This limits composability compared to fibers (where
  any function can yield) and async (where any async function can
  `.await`).

**Verdict: BEST FIT** for hilavitkutin. The cons are real but acceptable:
manual state machines are more code but produce more debuggable, more
testable, more portable work units. The inability to yield from nested
calls is mitigated by keeping work units small (each morsel is a single
execute() call that runs to completion; yielding is only for I/O-bound
work that spans multiple execute() calls).

### 4.4 Green threads (Go goroutines, Erlang processes)

**Mechanism.** Language-level lightweight threads with a runtime scheduler.
Each goroutine has a small initial stack (2-8 KB in Go) that grows
automatically. The Go runtime multiplexes goroutines onto OS threads
(M:N scheduling) with automatic preemption (since Go 1.14, goroutines
are preempted at function call boundaries via compiler-inserted
checkpoints).

**Pros.** Simple programming model: write sequential code, use `go` to
launch concurrent work, use channels to communicate. No colored functions,
no manual state machines.

**Cons for hilavitkutin.** Requires a language runtime. Not applicable to
Rust. Not applicable to no_std. The green thread model is fundamentally
incompatible with Rust's ownership system --- the runtime needs to manage
goroutine stacks, which conflicts with Rust's borrow checker (who owns the
stack? who can reference data on it?). Rust explicitly removed green
threads (libgreen) in 2014 for these reasons.

**Verdict: NOT applicable** to Rust or hilavitkutin.

## 5. Pipeline Breakers and Materialization Points

A pipeline breaker is an operator that must consume all of its input before
producing any output. The three canonical breakers in database query
processing are:

- **Hash build**: reads all build-side tuples, inserts each into a hash
  table, then signals that the hash table is ready for probing.
- **Sort**: reads all input tuples, sorts them (typically via external
  merge sort for large inputs), then produces sorted output.
- **Group-by aggregation**: reads all input tuples, accumulates aggregates
  per group (sum, count, min, max), then emits one result row per group.

Pipeline breakers are significant because they define the boundaries of
parallelism: within a pipeline (between breakers), operators can be fused
and processed morsel-by-morsel without synchronization. At a breaker, all
parallel workers must synchronize before the next pipeline can begin.

### 5.1 DuckDB's approach: breakers as pipeline boundaries

DuckDB models pipeline breakers as the edges between pipelines. The query
optimizer identifies breakers during physical plan generation and creates
separate pipelines for each segment between breakers. The scheduler
executes pipelines in dependency order: the build pipeline runs to
completion (all worker threads finish all morsels), then the probe
pipeline starts.

The shared resource (hash table, sorted run, aggregate state) is the
communication channel between pipelines. It is written by the upstream
pipeline and read by the downstream pipeline. No locking is needed at the
boundary because the upstream pipeline is guaranteed to have completed
before the downstream pipeline starts.

This is clean but limits parallelism: the probe phase cannot start until
the entire build phase is done, even if partial results could be useful.
For analytical queries on large tables, the build phase can take seconds,
and the probe phase sits idle during that time.

### 5.2 Morsel-driven approach: partitioned shared state

The Leis et al. paper handles pipeline breakers differently. The hash build
phase processes morsels of build-side input, with each morsel adding
entries to the hash table. To avoid contention, the hash table is
partitioned: each worker thread has a thread-local partition, and morsels
are assigned to the partition of the thread that processes them.

After all morsels are processed, the thread-local partitions are merged
into a single global hash table (or the probe phase scans all partitions).
The merge step is a synchronization point, but it is typically cheap
compared to the build phase itself.

This approach allows partial overlap: the build and probe phases can
overlap if the probe uses a concurrent hash table that allows reads while
writes are in progress (with appropriate visibility guarantees). HyPer and
Umbra explored this with lock-free hash tables that provide eventual
consistency during the build phase and full consistency after build
completes.

### 5.3 Mapping to hilavitkutin's DAG model

In hilavitkutin, pipeline breakers are expressed as DAG dependencies
between work units. Consider a hash join:

- **BuildHashTable** work unit: `type Read = (BuildSideColumn,)`, `type
  Write = (HashTable,)`. Processes morsels of build-side input, inserting
  into the hash table. Returns `Progress::Complete` when all input
  morsels are processed.
- **ProbeHashTable** work unit: `type Read = (ProbeSideColumn,
  HashTable)`, `type Write = (JoinResult,)`. Reads from the hash table
  and probe-side input. Cannot start until HashTable is available.

The scheduler sees that ProbeHashTable reads HashTable and BuildHashTable
writes HashTable. This creates a DAG edge: BuildHashTable must complete
before ProbeHashTable can start. No special "pipeline breaker" concept is
needed --- it falls out naturally from the Read/Write declarations.

For the morsel-level scheduling: BuildHashTable's `execute()` processes one
morsel and returns `Progress::Complete`. The scheduler fans out N instances
(one per morsel of build-side input). All N instances must complete before
ProbeHashTable becomes eligible. This is identical to Naughty Dog's atomic
counter: the counter starts at N, each morsel completion decrements it, and
when it reaches zero, ProbeHashTable is enqueued.

The yielding model extends this for I/O-bound breakers. A connector work
unit that fetches data from an API returns `Progress::Yield(Pending)` while
waiting for the HTTP response. The scheduler parks the work unit and runs
other eligible work. When the HTTP response arrives (signaled by the host
via a waker mechanism), the scheduler re-enqueues the work unit. This is
the same mechanism as the morsel-driven hash build, but with I/O latency
instead of computation as the reason for multiple execute() calls.

## 6. Relevance to hilavitkutin

### 6.1 Constraint summary

hilavitkutin must satisfy constraints from three consumers:

| Constraint | Source | Reason |
|------------|--------|--------|
| no_std compatible | all three | loimu targets embedded; polka targets minimal runtime |
| DAG from type declarations | all three | Unity DOTS model; safety through declaration |
| Morsel chunking | polka, saalis | column processing; L1-fitting work units |
| Cooperative yielding | polka, saalis | I/O-bound connectors; long-running transforms |
| Work-stealing | all three | load balancing across cores |
| Scheduling groups | saalis | cadence-based subsystem scheduling |
| Deterministic testing | all three | reproducible builds, reproducible test runs |

### 6.2 Model selection

The explicit yield model (`Progress::Yield`) is the best fit across all
constraints. The reasoning, grounded in the systems surveyed above:

**no_std.** Coroutines require an executor (Tokio, embassy). Fibers
require platform-specific stack switching. Green threads require a language
runtime. Explicit yield requires nothing: a struct, a function, an enum
return value. This compiles to any target without dependencies.

**Composable with morsel processing.** A morsel-processing work unit calls
`execute()` once per morsel and returns `Complete` every time. The
scheduler fans out instances across morsels. This is the DuckDB model
(one task per morsel) implemented without DuckDB's custom task scheduler.
A connector work unit calls `execute()` multiple times, returning `Yield(
Pending)` between API calls. The two patterns coexist in the same
scheduler without special-casing.

**DAG-compatible.** Yielding does not break the dependency graph. A work
unit that yields has not yet written its output, so downstream units remain
blocked (their dependency counter has not been decremented). When the
yielded unit eventually returns `Complete`, the counter decrements and
downstream units become eligible. This is identical to Naughty Dog's
counter-based dependency model, implemented without fibers.

**Debuggable.** The work unit's continuation state is a named struct.
You can derive Debug for it. You can serialize it. You can log it between
execute() calls. You can write a test that asserts the exact sequence of
Progress values for a given input. None of this is possible with fiber
stacks (opaque memory) or async futures (compiler-generated state machines
with mangled names).

### 6.3 Comparison to Naughty Dog fibers

The Naughty Dog fiber model is more ergonomic: any function in the call
stack can yield, and the programmer does not need to manage continuation
state. But fibers require stack allocation (64 KB per fiber on PS4),
platform-specific assembly for stack switching, and OS or custom allocator
support for stack memory. These are acceptable for a PS4/PS5-exclusive
game engine. They are not acceptable for a no_std Rust library targeting
desktop, server, WASM, and potentially embedded platforms.

The explicit yield model is the fiber model with the stack factored out.
Instead of freezing a stack frame, the work unit stores its state in a
struct. Instead of resuming by restoring a stack frame, the scheduler
calls execute() again and the work unit dispatches on its stored state.
The semantic is identical; the mechanism is different. The trade-off is
ergonomics (more boilerplate) for portability (no platform dependencies).

### 6.4 Comparison to Rust async

Rust async provides compiler-generated state machines that are
semantically equivalent to hand-written explicit yield state machines.
The compiler does the work that the work unit author does manually in
the explicit yield model. However, Rust async adds three costs:

1. **Executor dependency.** The Future must be polled by an executor.
   The executor manages the thread pool, the waker mechanism, and the
   I/O reactor. This is a large, complex dependency (Tokio is ~50K LOC)
   that is not no_std compatible.
2. **Colored functions.** An async execute() forces the scheduler to be
   async, which forces the host integration to be async. saalis (which
   uses Tokio for HTTP) could handle this. polka-dots (which is a batch
   compiler with no async I/O) would pay the async tax for no benefit.
   loimu (which is signal-driven, not async) would need an impedance
   adapter.
3. **Pin complexity.** Self-referential futures require `Pin<&mut Self>`,
   which interacts poorly with work-stealing (moving a pinned future to
   another thread requires `Unpin` or careful boxing). The explicit yield
   model avoids this entirely: the work unit struct is a plain struct
   with no self-references.

The async model is the right choice for DataFusion (server-side, Tokio
already required, async I/O is the primary concern). It is the wrong
choice for hilavitkutin (no_std required, three consumers with different
async profiles, explicit state is a feature not a bug).

### 6.5 Synthesis: what hilavitkutin takes from each lineage

From **Leis et al. / HyPer**: the morsel concept as a scheduling unit
sized to fit in cache. The principle that scheduling granularity
determines load balance quality.

From **Umbra**: task-based scheduling instead of a global dispatcher.
Variable morsel sizing based on operator complexity. The validation that
morsel-driven execution works with compilation-based and interpretation-
based processing equally well.

From **DuckDB**: the separation of cache-fitting unit (vector) from
scheduling unit (morsel). Pipeline breakers as pipeline boundaries
expressed through data dependencies. The proof that in-process, single-
node columnar execution is viable and performant.

From **DataFusion**: the validation that Rust's type system can express
columnar pipeline execution. The RecordBatch-as-morsel pattern. The lesson
that async is powerful but imposes costs that are avoidable.

From **Polars**: practical Rust patterns for columnar data ownership
(Arc buffers, owned chunks across thread boundaries). Rayon-based work-
stealing as a viable default. Formula-based morsel sizing as an
alternative to hardware detection.

From **Naughty Dog**: atomic counter dependencies as the scheduling
primitive. Yield-on-dependency as the mechanism for cooperative
scheduling. The insight that blocking dependencies become cooperative
scheduling when the yield mechanism is cheap enough.

From **Unity DOTS**: DAG inference from Read/Write declarations. The
proof that declarative data access is sufficient to generate correct
scheduling without explicit dependency specification. Chunk-based
iteration as the morsel equivalent in an ECS context.

From **The Machinery**: budget-aware scheduling for cadence-based
subsystems. Cost estimation as an input to scheduling decisions.

From **Unreal Engine 5**: named scheduling groups with thread affinity.
The distinction between data-flow dependencies (inferred from declarations)
and control-flow dependencies (specified explicitly via prerequisites or
virtual columns).

hilavitkutin combines these into a coherent model: morsel-sized work units
with typed Read/Write declarations, scheduled via work-stealing with atomic
counter dependencies, cooperatively yielding via explicit Progress enum,
grouped into scheduling groups with optional cadence constraints. The model
is no_std, fully portable, deterministically testable, and serves all three
consumers (data compiler, game library manager, UI framework) without
forcing any of them into an execution model that does not fit their
constraints.
