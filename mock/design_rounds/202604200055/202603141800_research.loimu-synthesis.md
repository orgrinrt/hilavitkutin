## 1. What hilavitkutin Should Adopt from loimu

This section enumerates the design elements from loimu that are project-agnostic and should become part of hilavitkutin's shared foundation. For each item: what it is in loimu, why it transfers cleanly, and how it maps to hilavitkutin's API.

### SlabColumn as universal storage atom

SlabColumn is a contiguous virtual-memory-backed array with demand-paged physical commits. It reserves a large virtual range via `mmap(PROT_NONE)`, commits pages on demand, and never reallocates or copies existing data. Access is direct pointer arithmetic: `base_ptr + index * STRIDE`. No software bounds checks on the hot path; uncommitted pages fault at the OS level.

This is project-agnostic because every columnar store needs a growable contiguous array that does not invalidate pointers on growth. polka needs it for IR columns during pipeline execution. saalis needs it for query result buffers and connector output columns. loimu needs it for everything.

In hilavitkutin, `SlabColumn<T>` becomes the primitive backing `DenseColumn<T>`, `SparseArray<V>`, and `SparseSet<T>`. The type parameter carries `HasShape` so the storage engine knows stride and alignment. The WASM/embedded fallback (linked-page allocator with the same API) ships alongside the mmap implementation, gated by target platform. hilavitkutin exposes the data structure directly; consumers never interact with raw mmap calls.

### Read/Write typed access declarations producing DAG edges

In loimu, every scheduling unit declares what it reads and writes. The proc macro derives `Read<T>` and `Write<T>` from handler body patterns (`each (a: A, b: &mut B)` yields Read A, Write B). Two units with overlapping write sets or read-write conflicts get a DAG edge. Two units with disjoint access sets get no edge and can run in parallel.

This is project-agnostic because every DAG scheduler needs typed dependency edges. The mechanism for deriving access sets varies by consumer (loimu uses proc macros on handler bodies; polka uses trait declarations on pipeline stages; saalis uses query schemas), but the edge construction algorithm is identical: `Write<T>` on unit A and `Read<T>` on unit B produces edge A->B.

In hilavitkutin, the `AccessSet` type holds a set of `(TypeId-equivalent, Direction)` pairs. `Dag::add_edge_from_access` takes two `AccessSet` values and inserts conflict edges. The consumer populates the `AccessSet`; hilavitkutin constructs the DAG. The `TypeId-equivalent` is a `define_id!`-generated interned identifier, not `std::any::TypeId` (which is banned).

### Version stamps for phantom write detection and early-release

Every resource in loimu carries a `version: u64`, incremented by the write accessor. The scheduler captures versions before a unit runs and compares after. If the version did not change (the unit wrote the same value back, or conditional logic skipped the write entirely), downstream units depending only on that resource can skip execution. This is the early-cutoff optimization.

Phantom write detection uses bitwise comparison (`memcmp`) for resources under 4KB, covering more than 95% of framework resources. For larger resources, the developer calls `.mark_changed()` explicitly.

This is project-agnostic because every incremental system needs to detect "no actual change." polka needs it for incremental rebuild (if the parser produces the same AST, skip expansion and resolution). saalis needs it for re-query detection (if the data source version has not changed, the cached result is valid). loimu needs it for behavior skipping and frame generation ratio.

In hilavitkutin, the `Version` type is a `u64` newtype. `VersionedResource<T>` wraps any `T` with a version stamp and provides `write_with_detection(new_value) -> bool` that performs the memcmp and increments only on actual change. The `DirtyTracker` integrates version stamps into its forward-sweep propagation.

### List scheduling with upward rank

loimu's scheduler uses list scheduling with upward rank, directly validated by Taskflow benchmarks (IEEE TPDS 2021) showing near-linear scaling to 40+ cores. The algorithm: (1) topological sort via Kahn's to get depth, (2) compute upward rank bottom-up (`rank(u) = cost(u) + max(rank(successors(u)))`), (3) assign units to lanes by picking the highest-rank ready unit and placing it on the lane where it finishes earliest.

This is project-agnostic because it is a pure graph algorithm with no domain assumptions. The cost model is pluggable (loimu uses EMA of actual execution time; polka could use file size or AST node count; saalis could use query complexity).

In hilavitkutin, `compute_plan` accepts a `Dag<N, E>` and a `CostFn: Fn(N) -> u32` and returns an `ExecutionPlan<N>` with steps and per-node metadata including `upstream_rank`, `depth`, and `lane` assignment. The lane count is a parameter, not a fixed constant.

### Work-stealing via overflow lists

When a worker's primary lane is blocked on a dependency, it does not spin-wait. Each unit has a precomputed `lane_compat: u8` bitmask indicating which lanes can safely execute it (no `AccessSet` conflicts). The blocked worker scans the overflow list for the highest-rank compatible unit and executes it. This fills horizontal gaps with work that advances the critical path.

This differs from Cilk/Rayon's randomized stealing in three ways: (1) stealing triggers on dependency block, not deque exhaustion; (2) candidates come from a precomputed overflow list, not a random victim; (3) selection uses upward rank, not FIFO order. This produces better latency on DAG workloads.

In hilavitkutin, the `Executor` trait has a default implementation that performs overflow stealing. The `lane_compat` bitmask is computed during `compute_plan` and stored per-node in the `ExecutionPlan`. Consumers that do not need work-stealing can implement a simpler `Executor` that runs steps sequentially.

### Dirty tracking (hierarchical DirtyTracker)

The `DirtyTracker` is a hierarchical bitset with configurable depth. Each level reduces the number of tracked elements by a factor. Marking a leaf dirty propagates upward. Querying at a coarse level lets callers skip entire subtrees. Clearing is O(levels), not O(elements). No heap allocation after construction.

loimu uses this at three levels: per-element bits (slot index for storage uploads), per-node (tree subtree skip), and per-tile (screen-space rendering skip). hilavitkutin needs the first two; the third is loimu-specific.

In hilavitkutin, `DirtyTracker::new(capacity, levels)` creates the bitset. The API: `mark(index)`, `is_dirty(index)`, `is_any_dirty()`, `dirty_ranges(level) -> Iterator<Range>`, `clear()`. polka uses it for pipeline stage skip-maps; saalis uses it for data store change tracking; loimu uses it for all three levels.

### Subgraph composition (NodeKind::Subgraph with plan caching)

Nodes in the DAG can be either `Leaf` or `Subgraph(ExecutionPlan<N>)`. A subgraph contains a pre-computed execution plan. When the executor reaches a subgraph node, it executes the inner plan's steps before proceeding. The inner plan is computed once and cached; recomputation triggers only on topology change.

This follows Taskflow's `composed_of` model: (1) the subgraph is a single node in the outer graph for ordering, (2) internal parallelism is preserved, (3) the subgraph is a reference not a copy, (4) no special executor support needed.

In hilavitkutin, `NodeKind<N: Id>` is an enum with `Leaf` and `Subgraph(ExecutionPlan<N>)`. The `Dag` accepts both kinds. The executor recurses into subgraphs automatically. Opaque subgraphs (opt-in) disable boundary analysis and run as atomic units.

### Resource aliasing (lifetime-based memory sharing)

Resources with non-overlapping lifetimes can share memory. `AliasingAnalysis` computes per-resource lifetime intervals `[first_step, last_step]` and groups non-overlapping resources via greedy interval coloring. The algorithm is O(n log n), equivalent to linear scan register allocation (Poletto & Sarkar, 1999).

Only transient resources (created and destroyed within a single plan execution) are candidates. Persistent resources are never aliased. Two aliased resources cannot both be dirty simultaneously.

In hilavitkutin, `AliasingAnalysis::compute(plan, dag, access_fn) -> Vec<Vec<ResourceId>>` returns alias groups. The consumer decides how to share memory based on resource sizes. The core just provides the groups.

### Selection vectors (zero-copy filtered iteration)

Rather than materializing filtered subsets, pass an index array alongside column data. The selection vector contains indices of elements that pass a filter predicate. Iteration reads only the selected indices from the column. Zero-copy, composable, and validated by DuckDB's vectorized execution model.

In hilavitkutin, `SelectionVector` is a `DenseColumn<u32>` of indices. Column iterators accept an optional `SelectionVector` and yield only matching elements. This maps directly to loimu's marker-based tree access (iterate only nodes with a given marker) and to polka's catalogue filtering (iterate only entries matching a predicate).

### Operation fusion (deferred writes, per-element op batching)

Fusible scheduling units (`each` blocks in loimu) push per-element operations to the storage engine. The engine groups operations by shared read columns and view membership, then fuses compatible operations into single iteration passes. Within a fused pass, the element pointer advances once; all compatible ops execute per element before moving to the next.

In hilavitkutin, the `fusible` flag on scheduling units is a boolean property. hilavitkutin does not own the fusion logic itself (that is a storage engine concern), but it provides the metadata the storage engine needs: which units are fusible, what their access sets are, and what the scheduler's ordering constraints require.

### Sole-writer detection for simpler rollback

The scheduler detects at plan computation time whether a resource has a single writer across all scheduling units. Sole-writer resources use snapshot restore on fault (copy dirty ranges from the snapshot buffer back to the live buffer). Shared-writer resources use an undo journal. The detection is a static property of the DAG, computed once at plan time.

In hilavitkutin, `sole_writer: bool` is computed per `(unit_id, resource_id)` pair during `compute_plan` and stored in the plan metadata. The consumer uses this flag to select its rollback strategy.

## 2. What hilavitkutin Should NOT Adopt

Each item here is loimu-specific or domain-specific. Including it in hilavitkutin would add complexity without benefit for polka or saalis.

### Seven access modes (Pinned, Immediate, Sample, Post, Approx, Defer, MutImmediate)

loimu needs seven access modes because its real-time GUI scheduler must express freshness guarantees, ordering constraints, and determinism properties per access. `Pinned<T>` reads a frozen snapshot with no ordering edge. `Immediate<T>` reads the current frame with a DAG edge. `Sample<T>` reads without ordering guarantees. `Approx<T>` reads a predicted future value. These distinctions are meaningful only in a frame-based rendering loop where snapshot age, prediction, and ordering relaxation unlock specific scheduling optimizations.

polka and saalis need two modes: Read and Write. polka's pipeline stages read inputs and write outputs. There is no notion of "frozen snapshot" versus "current frame" because there are no frames. saalis connectors read data sources and write results. The ordering is structural (DAG edges), not temporal (frame boundaries). Adding five extra access modes to hilavitkutin would burden consumers with a vocabulary they do not need and create an API surface that invites misuse.

### Signal bus with double-buffered arenas

loimu's signal bus is a full runtime subsystem: two `ErasedArena<SignalId>` instances, pointer swap at pass boundaries, three buffer modes (Queue, Replace, ReplaceBy), signal filtering, SignalSets for compound observation, and the settling guarantee (zero allocations after initialization). It solves the problem of ephemeral inter-behavior communication in a frame-based execution model.

polka does not need signals at all. Its pipeline is a static DAG of transforms with no runtime events. saalis needs event edges (EntityDiscovered, DataChanged) but these are zero-width DAG markers, not arena-stored ephemeral messages. Including the full signal bus in hilavitkutin would force consumers to pay for arena allocation infrastructure, double-buffering mechanics, and buffer mode semantics they will never use.

### Temporal storage (3-column layout, springs, reels, cadence-driven prediction)

Temporal fields in loimu get three columns (old, current, projected) with buffer rotation at pass boundaries. Spring temporal fields get four (adding velocity). The `Temporal::predict()` trait enables linear extrapolation; `SpringTemporal::spring_predict()` enables spring dynamics. The reel backend stores pre-computed future values for tolerance > 2. CadenceMap drives per-field prediction rates.

This is GUI-specific. Temporal prediction exists because loimu needs to display intermediate values between behavior updates for smooth animation. polka builds files; there is nothing to interpolate. saalis queries databases; prediction is meaningless for query results. Including temporal storage in hilavitkutin would triple the per-field memory cost for a feature no non-GUI consumer would use.

### Rotation groups and prediction-based skipping

loimu uses version stamps to determine how many frames a resource was static, informing the frame generation system's ratio decisions. Rotation groups batch temporal field prediction across elements sharing a cadence. Both depend on the concept of continuous frame-rate display.

polka and saalis do not display frames. Rotation groups are a scheduling optimization for a problem they do not have.

### GPU dispatch sub-keywords (compute/draw/copy)

loimu's `define_behavior!` includes `compute { groups, local_size }`, `draw { target, instances }`, and `copy { src, dst, size }` as scheduling units with queue affinity (Compute, Graphics, Copy). These produce `SchedulingUnitDescriptor` entries with `DispatchConfig` and `QueueAffinity`.

polka is a dotfiles build engine. saalis is a data integration platform. Neither has GPU hardware queues. GPU dispatch belongs in loimu's layer, not in the shared library.

### Per-stage lane profiles

loimu associates behaviors with app stages (e.g., `Init`, `Running`, `Teardown`) and computes lane profiles per stage. This optimizes thread utilization across the lifecycle of a GUI application. polka and saalis have no concept of lifecycle stages with different parallelism profiles.

### define_behavior! macro paradigm

loimu's `define_behavior!` runs a 16-stage parsing pipeline: parse handlers, extract access, infer field-level access, detect access modes, detect guards, generate scheduling units, generate subgraph edges, compute portals, parse animation declarations, generate blend graphs, detect animation commands, generate spring setup, parse dispatch blocks. The macro is the only sanctioned way to define a behavior; manual `impl Behavior` is explicitly prohibited.

This level of macro integration is too opinionated for a shared library. hilavitkutin's `WorkUnit` trait should be manually implementable. Macros should be optional convenience, not mandatory gatekeepers. Different consumers have different declaration patterns: loimu uses DSL handler blocks, polka uses trait impls on pipeline stages, saalis uses schema annotations on connectors. Forcing all three through a single macro form creates friction.

### Visibility macros (#[public_api], #[internal_api])

loimu's visibility macros control API surface via the `standalone` feature flag. In framework mode, `#[public_api]` items are `#[doc(hidden)] pub`. In standalone mode, they are `pub`. `#[internal_api]` items become `pub(crate)` in standalone mode. The `no-bare-pub` lint rejects any `pub` item without an annotation.

This is a loimu workspace convention, not a general-purpose API design pattern. polka-dots has its own visibility model (SDK traits are public; core implementations are crate-private). Adopting loimu's visibility macros would impose a framework-specific convention on a library that serves multiple frameworks.

### Frame generation (extrapolation, interpolation, text bypass)

Frame generation synthesizes intermediate display frames from known scene state. Extrapolation warps dirty regions forward using known motion (trivially correct for scrolling, exact for easing curves). Interpolation renders start and end frames and blends. Text always renders at native resolution.

This is a rendering optimization that requires a display pipeline, dirty tiles, transform stacks, and animation curves. It has no analog in build engines or data integration platforms.

## 3. Critique of loimu's Design Choices

### 16-byte stride: optimal for loimu's column values, constrains generic usage

loimu's fixed-point type system means most column values are 1-8 bytes, with `Coord` (`IFixed<16,16>`) at 4 bytes and `Vec3<Coord>` at 12 bytes being the common maximums. The storage engine bit-packs values at their declared width, so the effective stride varies per column.

For hilavitkutin, stride must be parameterized. polka's IR columns may contain `String` references (8 bytes), path buffers (variable), or AST node references (16+ bytes). saalis's query result columns may contain arbitrary-width values from external databases. Fixing stride at any particular value would constrain consumers unnecessarily. hilavitkutin should use `size_of::<T>()` rounded to alignment, exactly as loimu does internally, but without asserting a characteristic width.

### Seven access modes: necessary for loimu's complex scheduling but API surface burden

The seven access modes are not arbitrary; they encode real scheduling distinctions. `Pinned<T>` breaks ordering edges (maximizing parallelism at the cost of snapshot freshness). `Defer<T>` defers writes to pass boundary (enabling fusion without ordering constraints). `Sample<T>` relaxes determinism (enabling chaos-mode fusion).

However, the three fusion tiers (Unconditional, Ordered, Chaos) are the actual scheduling distinctions. The seven modes are syntactic sugar over these three tiers plus a direction bit (read/write). For consumers that do not need frame-level freshness semantics, the mode vocabulary is overhead. The critique is not that the modes are wrong for loimu but that the API surface (seven wrapper types, compile-time error tables for invalid combinations, mode-specific DAG edge rules) is more than a shared library should impose.

### Signal bus: well-designed but heavy for event-edge-only use

The signal bus architecture is internally consistent: arena allocation avoids per-signal heap allocation, double-buffering separates emission from observation, buffer modes handle different signal semantics (discrete events vs continuous values vs keyed deduplication). The settling guarantee (zero allocations after initialization) is a strong correctness property.

The critique is scope, not quality. saalis needs event edges, not a signal bus. An event edge is a zero-width DAG marker: `Write<EntityDiscovered>` creates a dependency edge with no data. This requires neither arena allocation nor double-buffering nor buffer modes. Building a full signal bus into hilavitkutin to serve this use case would be overengineering. The bus should remain in loimu-signal, layered on top of hilavitkutin's DAG edge primitives.

### No yielding: principled for v1 but acknowledged gap

loimu explicitly deferred yielding and the Naughty Dog fiber model to post-1.0. Behaviors are atomic: `execute()` runs to completion. This is simpler to reason about (no partial writes, no suspended DAG state) and simpler to implement (no fiber stacks, no context switching, no signal handling during yield).

The gap is real. I/O-bound work units (network connectors in saalis, file reads in polka) would benefit from yielding: start the I/O, yield the worker, let it steal other work, resume when I/O completes. Without yielding, I/O-bound units block their lane for the entire I/O duration. loimu acknowledges this; the fiber model was explicitly studied and deferred. hilavitkutin should fill this gap.

### Macro-heavy approach: strong for consistency, reduces approachability

loimu's macros solve a real problem: a single type declaration needs 7-10 interconnected trait implementations, and missing any one breaks the system. `define_resource!` generates struct, Resource, HasShape, StorageRecord, Registrable, Default, PersistableResource, inventory::submit!, field accessors, and type validation. Manual implementation risks inconsistency.

The critique is about the mandatory nature. Manual trait implementation is explicitly prohibited ("WRONG: manual impl loses Registrable, HasShape, and breaks registry consistency"). This makes the framework feel closed. A developer who wants to understand what code is generated must expand macros. Debugging macro-generated code is harder than debugging hand-written code. For a shared library used by three different projects, mandating macros imposes loimu's ergonomic preferences on consumers who may prefer explicit, visible implementations.

### Scheduler algorithm: well-validated, leaves some parallelism on the table

List scheduling with upward rank is well-validated by Taskflow's benchmarks and by HEFT in HPC literature. The step-based execution model groups independent nodes into parallel steps separated by barriers.

The barrier cost is the critique. Step boundaries force all workers to synchronize, draining store buffers (memory fence) and potentially thrashing L2/L3 caches. Taskflow benchmarks show step-based execution leaves 10-15% parallelism on the table versus wavefront (counter-based) execution, where tasks become ready the instant their dependency counter reaches zero. loimu acknowledges this and designs for future executor swap: the three-phase separation (Graph/Plan/Execute) means the executor can change without affecting the graph or plan APIs. The step model is a reasonable starting point; the architecture supports upgrading.

### View system: dynamic affinity analysis and population hysteresis are sophisticated

loimu's view system computes column co-access frequency matrices from behavior access declarations, clusters high-frequency co-accesses into view templates, and manages view instances with population thresholds using hysteresis (promote at 1.5x, demote at 0.5x). View restructuring triggers when fitness drops below 0.6 for ~60 passes.

For polka and saalis, static views are sufficient. polka's column sets are determined at compile time from pipeline stage declarations. saalis's column sets come from query schemas. Neither has the dynamic population shifts that justify runtime affinity analysis. hilavitkutin should support views as a concept (grouping columns for cache-friendly co-iteration) but leave dynamic affinity analysis to loimu's layer.

## 4. Improvements hilavitkutin Can Make

Different constraints produce different design choices. hilavitkutin serves three consumers with varying requirements. These improvements address gaps in loimu's design or opportunities created by the broader scope.

### Add yielding (Progress::Yield)

The gap loimu deferred. A work unit's `execute()` returns `Progress`, which is either `Complete` or `Yield(ResumeToken)`. A yielded unit has not written its output; downstream units remain blocked. The DAG handles this naturally: the dependency edge is not satisfied until the unit completes. The yielded unit frees its worker. The worker steals other ready work via the overflow list. When the yield resolves (I/O completes, budget refreshes), the unit re-enters the ready queue with its `ResumeToken`.

This is `no_std` compatible because it does not require fiber stacks or OS context switching. The unit yields from `execute()` only, not from nested calls. Less ergonomic than fibers but simpler and portable.

### Simplify access modes to Read/Write

Consumers add richness on top. hilavitkutin provides `Read<T>` and `Write<T>` as DAG edge constructors. loimu maps its seven modes to these two for DAG construction, then adds ordering constraints as a layer. polka uses `Read<T>` and `Write<T>` directly. saalis uses them directly. No consumer is forced to learn seven modes they do not need.

### Make macros optional

The `WorkUnit` trait is manually implementable. A `#[register]` derive macro generates inventory registration and HasShape, but a consumer can implement these traits by hand with no loss of functionality. The macro is convenience, not mandate. This matches polka-dots' existing philosophy: `#[register]` is already the derive macro for trait impl blocks.

### Parameterize stride instead of fixing at 16 bytes

hilavitkutin's `SlabColumn<T>` uses `size_of::<T>()` rounded to alignment. No characteristic stride is assumed. The storage layer works with any type that implements `HasShape`. Consumers that want fixed-point bit-packing (loimu) implement it in their storage layer. Consumers that want byte-aligned values (polka, saalis) get natural stride.

### Static views instead of dynamic affinity analysis

hilavitkutin supports view templates defined at compile time from `AccessSet` declarations. No runtime population tracking, no hysteresis thresholds, no fitness metrics. loimu adds dynamic affinity analysis as its own layer. polka and saalis define their view templates statically and never change them.

### Virtual columns instead of signal bus

Zero-width marker types create DAG edges without allocating storage. `Write<EntityDiscovered>` and `Read<EntityDiscovered>` produce the same edge as `Write<Vec3<Coord>>` and `Read<Vec3<Coord>>`, but `EntityDiscovered` has no `SlabColumn`, no memory, no version stamp. This covers saalis's event edge use case and the trivial subset of loimu's signal needs. loimu builds its full signal bus on top.

### no_std from day one

loimu is std-only (it uses mmap, thread spawning, file I/O). hilavitkutin should be structured as a `no_std` core with `alloc`-gated features. The numeric primitives, DAG data structure, dirty tracker, and aliasing analysis have no inherent std dependency. The executor (which spawns threads) and the mmap-backed SlabColumn are behind feature gates. This enables polka-dots to run in WASM or embedded contexts without pulling in std.

## 5. The Unified Store Question (Q1 Synthesis)

Can columns (multi-element homogeneous data) and resources (singleton state) share one `Read<T>`/`Write<T>` API?

### loimu's answer: yes

Both columns and singletons are backed by `SlabColumn`. A resource is a record with cardinality 1 (one element in a view with one slot). The scheduler tracks dependencies by typed access declarations, not by storage characteristics. A singleton and a million-row column create the same kind of DAG edge: `Write<InputState>` on unit A and `Read<InputState>` on unit B produces edge A->B regardless of whether `InputState` has one element or a million.

### What makes it work

The scheduler is storage-agnostic. It sees `AccessSet` entries, not storage layout. The edge from `Write<T>` to `Read<T>` is a typed dependency declaration. Whether T is stored as a singleton resource, a dense column, a sparse set, or a zero-width marker is invisible to the scheduler. The DAG edge construction algorithm is identical across all storage modalities.

### The access API differs

Singletons use `.resource::<T>()` for direct access. Collections use morsel iteration (`each` in loimu, column iterators in hilavitkutin). The scheduling is the same; the ergonomics are different. A singleton read is a single pointer dereference. A collection read is a loop over a dense column with optional selection vector. Both declare `Read<T>` and produce the same DAG edge.

### Recommendation for hilavitkutin

Unified at the scheduler level, differentiated at the access level. The `AccessSet` type does not distinguish singletons from collections. The `WorkUnit::execute()` context provides different accessor methods for different storage modalities. This keeps the scheduler simple (one edge type, one conflict algorithm) while giving consumers natural APIs for their data shapes.

## 6. The Yielding Question (Q2 Synthesis)

### What yielding in loimu's architecture would look like

Behaviors are currently atomic: `execute()` runs to completion or faults. Adding yielding means a unit's `execute()` can return early with a `Yield(ResumeToken)` and be rescheduled later.

### DAG impact

A yielded unit has not written its output. Downstream units must wait. The DAG handles this naturally: the dependency edge is not satisfied until the unit completes (returns `Complete`, not `Yield`). The plan does not change; the executor simply does not decrement downstream dependency counters until completion.

### Version stamp interaction

A yielded unit has not incremented version stamps (it has not written anything yet). Early-release does not fire. Downstream stays blocked. When the unit eventually completes and writes its output, the version stamp increments normally, and early-release checks proceed as usual.

### Work-stealing interaction

A yielded work unit frees its worker thread. The worker steals other ready work via the overflow list, using the same `lane_compat` bitmask and `upstream_rank` priority as always. When the yield resolves (I/O completes, timer fires, budget refreshes), the unit re-enters the ready queue. The next idle worker picks it up and resumes execution from the `ResumeToken`.

### loimu's deferred fiber model vs our explicit yield

Fibers yield from arbitrary nested call sites (more ergonomic: a deeply nested function can yield without threading a return value up the call stack). Our model yields from `execute()` only (less ergonomic: the unit must structure its logic to checkpoint and resume at the top level). The tradeoff: fibers require platform-specific stack management (stack allocation, context switching, signal mask preservation, debugger integration) and are inherently `std`-only. Explicit yield is `no_std` compatible, portable, and simpler to reason about.

For saalis's HTTP connectors (start request, yield, resume on response), explicit yield is sufficient. For loimu's GPU fences (submit compute, yield, resume on fence signal), explicit yield is sufficient. Deep nested yields (a behavior calling a library function that needs to yield) are the ergonomic gap; loimu can address this with fibers as a loimu-specific layer on top of hilavitkutin's explicit yield.

## 8. loimu's Own Research

### Source corpus

loimu's SOURCES.md catalogs 38 sources across academic papers, industry presentations, blog posts, and reference implementations. The collection is thorough and well-organized, spanning task graph scheduling, render graph architecture, GPU-driven rendering, virtualized geometry, mobile GPU constraints, memory hierarchy optimization, incremental computation, frame generation, voxel data structures, data-oriented design, GPU pipeline architecture, database engine internals, columnar execution, and MVCC concurrency control.

### Academic papers referenced

**Halide (Ragan-Kelley et al., PLDI 2013):** Algorithm/schedule separation. The foundational insight that what to compute and how to compute it are independent concerns. Three fundamental scheduling tradeoffs: parallelism, locality, and redundant recomputation. loimu's three-phase model (Graph/Plan/Execute) directly follows this separation. hilavitkutin inherits it.

**Taskflow (Huang et al., IEEE TPDS 2021):** Lightweight parallel task graph system with work-stealing executor. Key contribution: `composed_of` subgraphs, conditional tasking, ~2 microsecond per-task overhead, near-linear scaling to 40+ cores. Validates loimu's list scheduling with upward rank. The five task types (static, dynamic, composed_of, condition, cudaFlow) map well to loimu's model.

**Nominal Adapton (Hammer et al., OOPSLA 2015):** Demand-driven incremental computation with first-class names for controlling reuse. Early cutoff optimization. From-scratch consistency proof. Validates loimu's version-stamp dirty tracking. Key difference: Adapton uses runtime dependency tracking (expensive); loimu uses static declarations (free at runtime).

**Poletto & Sarkar (1999):** Linear scan register allocation. Directly applicable to resource aliasing: the greedy interval coloring algorithm is equivalent to linear scan.

### Industry presentations referenced

**Frostbite FrameGraph (O'Donnell, GDC 2017):** The original render graph pattern. Three-phase model (Setup/Compile/Execute), transient resource aliasing, memory savings on Xbox One's 32MB ESRAM. Covered accessibly in Wihlidal's Halcyon talk. Directly informs loimu-graph's aliasing analysis.

**Naughty Dog Fibers (Gyrling, GDC 2015):** Fiber-based job system with atomic dependency counters and work stealing. Maximum parallelism at the cost of harder profiling and cache-unfriendly stealing. loimu studied this and explicitly deferred it. hilavitkutin fills the gap with explicit yield instead of fibers.

**Nanite (Karis et al., SIGGRAPH 2021, GDC 2024):** GPU-driven virtualized geometry. Key patterns: visibility buffer decoupling (geometry -> materials -> lighting), classify-reserve-scatter (counting sort for GPU-side work sorting), hierarchical coarse-to-fine processing. The 2024 GDC talk extended to GPU-driven materials. loimu adopted the classify-reserve-scatter pattern as a reusable compute utility.

**Aaltonen (HypeHype, SIGGRAPH 2023/2024):** Mobile GPU rendering architecture. Key numbers: ~10K draw calls at 60fps, Xbox 360-class bandwidth constraints. Validated that mobile GPU constraints shape graph compile strategies differently from desktop.

### Database engine research

**DuckDB vectorized execution (Kersten et al., VLDB 2018):** Vectorized vs compiled query execution. Vectorized is competitive with compiled, has better SIMD utilization, zero compilation overhead, easier profiling. Validates loimu's batch processing model. The ~1024 vector size (fitting in L1 cache) is a good target for chunk processing.

**DuckDB lightweight compression (Raasveldt, 2022):** Per-segment algorithm selection (constant, RLE, bit packing, FOR, dictionary, FSST, Chimp/Patas). Validates loimu's bit-packing approach and suggests future optimization: analyze actual data distribution and select optimal encoding per column segment.

**DuckDB SQL on Pandas (Raasveldt & Muhleisen, 2021):** 15x performance gap between naive and optimized query execution. Automatic projection pushdown, filter pushdown, and operation fusion. Validates the scheduler's role as the primary source of performance.

**Apache Arrow:** Standardized in-memory columnar format for zero-copy interchange. Arrow does NOT bit-pack below byte boundaries (prioritizes zero-copy reads); loimu does (prioritizes storage density). Both valid for their use cases.

**MVCC (Neumann et al., SIGMOD 2015):** loimu's frozen/live double-buffer is a two-version MVCC. Avoids classic MVCC problems: no version chain accumulation (exactly 2 versions), no garbage collection (swap is atomic), no write skew (DAG ordering prevents conflicting writes).

This synthesis covers all seven loimu research documents (storage, scheduler, DAG, macros, signals, incremental computation, primitives) plus loimu's own 38-source research corpus and design documentation. The boundary between hilavitkutin and loimu is clear: hilavitkutin owns the DAG, plan computation, dirty tracking, aliasing analysis, version stamps, and the compile pass extension point. Everything above that line is a consumer concern. loimu adds seven access modes, field inference, computed portals, signal bus, temporal storage, GPU dispatch, and frame generation as its own layer. polka adds morsel chunking, column dispatch, and incremental rebuild. saalis adds I/O batching, yielding, and event edges. Each builds on the same foundation without contaminating the shared core.

The key design tension throughout is scope versus power. loimu's design is powerful because it is specific: seven access modes encode real GUI scheduling distinctions; the signal bus handles real inter-behavior communication patterns; temporal storage solves real animation prediction problems. hilavitkutin's design must be less powerful but more general: Read/Write suffice for three consumers; virtual columns cover event edges and signal stubs; explicit yield covers I/O and GPU fences. The power gap is filled by each consumer's own layer, built on hilavitkutin's primitives.

The research validates this layered approach. Halide's algorithm/schedule separation says the same thing at a different level: the algorithm (what to compute) belongs in the consumer; the schedule (how to compute it) belongs in the shared library. Taskflow's benchmarks prove that list scheduling with upward rank scales to production workloads. Salsa and Adapton validate version-stamp early cutoff at scale. DuckDB validates columnar vectorized execution as competitive with compilation. Frostbite validates resource aliasing as meaningful savings. The shared library does not need to invent new algorithms; it needs to provide validated ones in a consumer-agnostic form.
