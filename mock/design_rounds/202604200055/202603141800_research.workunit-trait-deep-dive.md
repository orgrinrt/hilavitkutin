## 1. The Minimal WorkUnit Trait

The WorkUnit trait is the only type hilavitkutin mandates. Everything
else (registration, discovery, scheduling hints, cadence) is either
consumer-owned or scheduler-internal. The trait defines what it means to
be a schedulable unit of work in a typed DAG pipeline.

The minimal shape:

```rust
pub trait WorkUnit: Send + Sync + 'static {
    type Read: AccessSet;
    type Write: AccessSet;
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress;
}
```

Three design decisions are embedded in this signature. Each deserves
explicit justification because changing any of them later is an ABI break
for every consumer.

### Why `&mut self`

The work unit holds mutable state across yields. This is the
fundamental requirement from Q2 (yielding model). An HTTP connector
tracks request progress through an internal state machine:
`Idle -> RequestSent -> ResponseReceived -> Processing -> Done`. A hash
build operator accumulates partial hash table state across morsel
batches. A file scanner tracks its position in a directory listing.

`&mut self` gives the work unit exclusive ownership of its continuation
state. Between `execute()` calls, the scheduler does not touch the work
unit's fields. The work unit resumes exactly where it left off, reading
from and writing to its own struct fields without synchronization.

The alternative is `&self`, which is what loimu uses for behaviors. In
loimu, all state lives in resources (typed singletons backed by
SlabColumn). Behaviors are stateless functions that transform resource
state. A behavior does not "continue" from a previous invocation; each
invocation is independent, reading the current state of resources and
writing new state.

This works for loimu because loimu does not yield. Behaviors are atomic:
they run to completion or fault. There is no continuation state because
there is no continuation. The `&self` signature is correct for that model.

hilavitkutin supports yielding (Progress::Yield). A yielding work unit
must hold its continuation state somewhere. Two options: in the work
unit's own fields (`&mut self`), or in a resource (`&self` with
`Write<ContinuationState>`). The second option externalizes internal
implementation detail into the global resource graph. Every work unit
that yields would need a dedicated continuation resource, polluting the
DAG with scheduling artifacts that have nothing to do with data flow.
The scheduler would see edges to these continuation resources and might
mistakenly serialize work units that share no real data dependency.

`&mut self` is cleaner: the continuation state is private to the work
unit. The scheduler does not know about it. The DAG reflects only data
dependencies, not scheduling mechanics. The cost is that the scheduler
must give the work unit back to the same worker thread (or at least
ensure no concurrent access), but since work units are already
dispatched exclusively (one worker runs one work unit at a time), this
is already guaranteed.

### Why `Context<R, W>` and not a world reference

The Context type is parameterized by the work unit's Read and Write
associated types. It provides accessor methods that are valid only for
the stores declared in those types. This is compile-time access control:
attempting to read a store not listed in `type Read` is a type error.

The alternative is a world reference: `fn execute(&mut self, world:
&World) -> Progress`. The work unit would call `world.read::<T>()` and
get a runtime check (panic or Result::Err if T is not in the declared
read set). This is Bevy's model: systems receive a `World` and access
it through typed queries that are validated at system registration, not
at the call site.

The runtime-check model is simpler to implement but pushes errors from
compile time to runtime. A work unit that accidentally reads an
undeclared resource would compile successfully and crash at runtime. With
`Context<R, W>`, the same mistake is a compile error. The type system
enforces that the work unit can only access what it declared.

The cost is more complex generics. `Context<R, W>` carries two type
parameters everywhere. Work unit implementations must thread these
parameters through helper methods. But this cost is paid once per work
unit implementation and caught once per type error, rather than paid
zero times and caught in production.

### Why `Progress` and not `Result`

The return type is `Progress`, not `Result<(), Error>`. `Progress` is
a five-variant enum (Complete, Yield(BudgetExhausted), Yield(Pending),
Yield(Retry { ... }), Failed(Error)) that communicates the work unit's
disposition to the scheduler. `Result` cannot express yielding: `Ok`
means done, `Err` means failed, and there is no room for "I need to be
called again later."

The Progress enum is hilavitkutin's answer to the cooperative scheduling
problem (Q2). It replaces `Poll::Ready / Poll::Pending` from Rust's
async model with a richer vocabulary that includes retry with backoff
and budget exhaustion. The scheduler dispatches differently based on
each variant:

- `Complete`: mark output as written, increment version stamp, wake
  downstream dependents.
- `Yield(BudgetExhausted)`: re-enqueue the work unit for the next
  scheduling quantum. The work unit ran out of time but has more work.
- `Yield(Pending)`: park the work unit. It is waiting for an external
  event (I/O completion, GPU fence, timer). The waker mechanism (atomic
  flag array, scanned by the scheduler each round) will resume it.
- `Yield(Retry { attempt, max, backoff })`: park the work unit with a
  timer. Re-enqueue after the backoff duration. The scheduler adds
  jitter to desynchronize concurrent retries.
- `Failed(Error)`: mark the work unit as permanently failed. Restore
  the resource state via snapshot or undo journal. Skip downstream
  dependents via version stamp (no increment means no change detected).

## 2. Associated Types vs Generic Parameters

The Read and Write declarations use associated types, not generic
parameters. This is a deliberate choice with specific consequences.

### Associated types (current design)

```rust
impl WorkUnit for SteamCataloguer {
    type Read = (Resource<HttpClient>, Resource<ApiConfig>);
    type Write = (Column<Entities>, Virtual<EntityDiscovered>);
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress { ... }
}
```

Each concrete type has exactly one WorkUnit implementation. There is one
SteamCataloguer, and it reads HttpClient + ApiConfig, and it writes
Entities + EntityDiscovered. This is a 1:1 mapping.

### Generic parameters (alternative)

```rust
impl WorkUnit<(Resource<HttpClient>,), (Column<Entities>,)> for SteamCataloguer {
    fn execute(&mut self, ctx: Context<...>) -> Progress { ... }
}
```

With generic parameters, the same SteamCataloguer type could implement
WorkUnit multiple times with different access patterns. This is a 1:N
mapping.

### Why associated types are correct

A work unit's access pattern is intrinsic to its identity. The
SteamCataloguer reads from an HTTP client because that is what it does;
reading from a different set of resources would make it a different work
unit. Generic parameters would allow the same struct to be two different
work units depending on which impl the caller selects. This creates
ambiguity: when the scheduler receives a SteamCataloguer, which impl
does it use?

The precedent is `Iterator`. Each type has one Item type, not a generic
parameter. `Vec<u32>::IntoIter` always yields `u32`. Making Item a
generic parameter (`impl<T> Iterator<T> for Vec<u32>::IntoIter`) would
allow the same iterator to yield different types, which is nonsensical.

The same logic applies to WorkUnit. Each type has one access pattern.
Associated types encode this 1:1 relationship. The compiler enforces it:
you cannot implement `WorkUnit` twice for the same type.

### Edge case: parameterized work units

Some work units are generic over their data type:

```rust
struct ColumnAdder<A, B, C> {
    _phantom: PhantomData<(A, B, C)>,
}

impl<A, B, C> WorkUnit for ColumnAdder<A, B, C>
where
    A: Send + Sync + 'static + Add<B, Output = C>,
    B: Send + Sync + 'static,
    C: Send + Sync + 'static,
{
    type Read = (Column<A>, Column<B>);
    type Write = (Column<C>,);
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        // ...
        Progress::Complete
    }
}
```

Here the work unit type is itself generic, and each monomorphization has
its own associated types. `ColumnAdder<i32, i32, i32>` reads
`Column<i32>` twice and writes `Column<i32>`. `ColumnAdder<f64, f64, f64>`
reads and writes `Column<f64>`. Each monomorphization is a distinct type
with a distinct WorkUnit impl. Associated types handle this correctly.

## 3. Tuple-Based AccessSet

The AccessSet trait describes a set of stores that a work unit reads or
writes. The implementation uses tuples of store types, with trait impls
for tuples up to a fixed arity.

### The AccessSet trait

```rust
/// A set of stores that a work unit accesses.
///
/// Implemented for the unit type (no access), individual store types,
/// and tuples of store types up to arity 12.
pub trait AccessSet: 'static {
    /// Returns the list of (StoreId, Cardinality) pairs in this set.
    ///
    /// The scheduler calls this once during DAG construction to extract
    /// dependency edges. Not called on the hot path.
    fn store_ids() -> &'static [(StoreId, Cardinality)];
}
```

### Implementations

The unit type represents "no access":

```rust
impl AccessSet for () {
    fn store_ids() -> &'static [(StoreId, Cardinality)] {
        &[]
    }
}
```

Individual store types contribute one entry:

```rust
impl<T: Send + Sync + 'static> AccessSet for Resource<T> {
    fn store_ids() -> &'static [(StoreId, Cardinality)] {
        // returns [(StoreId::of::<Resource<T>>(), Cardinality::One)]
        // computed at compile time or cached on first call
        static IDS: OnceLock<Vec<(StoreId, Cardinality)>> = OnceLock::new();
        IDS.get_or_init(|| vec![(Resource::<T>::store_id(), Cardinality::One)])
    }
}

impl<T: Send + Sync + 'static> AccessSet for Column<T> {
    fn store_ids() -> &'static [(StoreId, Cardinality)] {
        static IDS: OnceLock<Vec<(StoreId, Cardinality)>> = OnceLock::new();
        IDS.get_or_init(|| vec![(Column::<T>::store_id(), Cardinality::Many)])
    }
}

impl<T: Send + Sync + 'static> AccessSet for Virtual<T> {
    fn store_ids() -> &'static [(StoreId, Cardinality)] {
        static IDS: OnceLock<Vec<(StoreId, Cardinality)>> = OnceLock::new();
        IDS.get_or_init(|| vec![(Virtual::<T>::store_id(), Cardinality::Zero)])
    }
}
```

Tuples concatenate their elements' store IDs:

```rust
impl<A: AccessSet, B: AccessSet> AccessSet for (A, B) {
    fn store_ids() -> &'static [(StoreId, Cardinality)] {
        static IDS: OnceLock<Vec<(StoreId, Cardinality)>> = OnceLock::new();
        IDS.get_or_init(|| {
            let mut ids = A::store_ids().to_vec();
            ids.extend_from_slice(B::store_ids());
            ids
        })
    }
}
```

A macro generates tuple impls up to arity 12:

```rust
macro_rules! impl_access_set_tuple {
    ($($T:ident),+) => {
        impl<$($T: AccessSet),+> AccessSet for ($($T,)+) {
            fn store_ids() -> &'static [(StoreId, Cardinality)] {
                static IDS: OnceLock<Vec<(StoreId, Cardinality)>> = OnceLock::new();
                IDS.get_or_init(|| {
                    let mut ids = Vec::new();
                    $(ids.extend_from_slice($T::store_ids());)+
                    ids
                })
            }
        }
    };
}

impl_access_set_tuple!(A);
impl_access_set_tuple!(A, B);
impl_access_set_tuple!(A, B, C);
// ... up to arity 12
```

The scheduler flattens all tuple elements into a flat list of
`(StoreId, AccessDirection)` pairs for DAG edge construction. Access
direction (Read or Write) comes from which associated type the
AccessSet appears in: `type Read` entries are Read, `type Write` entries
are Write.

### Why arity 12

Arity 12 is the de facto Rust convention for tuple trait impls. The
standard library implements traits for tuples up to arity 12.
Bevy does the same for system parameters. loimu's access declarations
support up to arity 12. A work unit that reads more than 12 distinct
stores is probably doing too much and should be split.

If a consumer genuinely needs more than 12, they can nest tuples:
`type Read = ((A, B, C, D, E, F, G, H, I, J, K, L), (M, N))`. The
AccessSet impl for the outer tuple concatenates the inner tuples'
store IDs. This is ugly but functional, and the rarity of the case
does not justify increasing the macro-generated arity.

## 4. Context<R, W> Compile-Time Safety

The Context type is the work unit's view of the world. It provides
typed accessor methods that are gated by the Read and Write associated
types. Attempting to access a store not in the access set is a compile
error.

### Accessor methods

```rust
impl<'a, R: AccessSet, W: AccessSet> Context<'a, R, W> {
    /// Read a singleton resource. Available only if Resource<T> is in R or W.
    pub fn resource<T>(&self) -> &T
    where
        R: Contains<Resource<T>>,
    {
        self.world.get_resource::<T>()
    }

    /// Mutably access a singleton resource. Available only if Resource<T> is in W.
    pub fn resource_mut<T>(&mut self) -> &mut T
    where
        W: Contains<Resource<T>>,
    {
        self.world.get_resource_mut::<T>()
    }

    /// Read a column slice (current morsel). Available only if Column<T> is in R or W.
    pub fn column<T>(&self) -> ColumnRef<'_, T>
    where
        R: Contains<Column<T>>,
    {
        self.world.get_column_ref::<T>(self.morsel_range)
    }

    /// Mutably access a column slice (current morsel). Available only if Column<T> is in W.
    pub fn column_mut<T>(&mut self) -> ColumnMut<'_, T>
    where
        W: Contains<Column<T>>,
    {
        self.world.get_column_mut::<T>(self.morsel_range)
    }

    /// Emit a virtual signal. Available only if Virtual<T> is in W.
    pub fn emit<T>(&mut self)
    where
        W: Contains<Virtual<T>>,
        T: 'static,
    {
        self.world.mark_virtual_emitted::<T>();
    }
}
```

### The Contains trait

`Contains<S>` is a marker trait that asserts a specific store type is
present in an AccessSet. It is implemented for individual types
(trivially: a type contains itself) and for tuples (a tuple contains S
if any element is S).

```rust
/// Marker: S is a member of Self.
pub trait Contains<S> {}

// A type contains itself.
impl<S: AccessSet> Contains<S> for S {}

// A 2-tuple contains S if either element is S.
impl<S, A, B> Contains<S> for (A, B) where A: Contains<S> {}
impl<S, A, B> Contains<S> for (A, B) where B: Contains<S> {}
```

The compiler resolves `Contains` during type checking. If a work unit
declares `type Read = (Resource<HttpClient>, Column<Entities>)` and
calls `ctx.resource::<DatabasePool>()`, the compiler looks for
`Contains<Resource<DatabasePool>>` on `(Resource<HttpClient>,
Column<Entities>)`. Neither element matches, so the impl is not found,
and the compiler emits an error.

In practice, the overlapping impls for tuples (both `A: Contains<S>`
and `B: Contains<S>`) require careful handling. Rust's coherence rules
disallow two blanket impls that could overlap. The standard workaround
is to use a helper trait with a type-level index:

```rust
pub trait Contains<S, Idx> {}

// First element of a 2-tuple.
impl<S, B> Contains<S, Here> for (S, B) {}

// Second element of a 2-tuple.
impl<S, A> Contains<S, There<Here>> for (A, S) {}

// First element of a 3-tuple.
impl<S, B, C> Contains<S, Here> for (S, B, C) {}

// Second element of a 3-tuple.
impl<S, A, C> Contains<S, There<Here>> for (A, S, C) {}

// Third element of a 3-tuple.
impl<S, A, B> Contains<S, There<There<Here>>> for (A, B, S) {}
```

`Here` and `There<I>` are zero-sized marker types. The `Idx` parameter
disambiguates overlapping impls. The compiler infers the index
automatically; the user never writes it. This is the same technique
Bevy uses for `SystemParam` resolution and frunk uses for `HList`
indexing.

### Morsel scoping

When the scheduler dispatches a work unit in morsel mode, the Context's
column accessors return only the current morsel's slice, not the entire
column. The Context holds a `morsel_range: Range<usize>` field that
scopes all column access.

```rust
pub struct Context<'a, R: AccessSet, W: AccessSet> {
    world: &'a World,
    morsel_range: Range<usize>,
    _read: PhantomData<R>,
    _write: PhantomData<W>,
}
```

For scalar dispatch (no columns in the access set), `morsel_range` is
`0..0` and column accessors are never called (they would fail the
Contains check at compile time since no Column<T> is in the access set).

The work unit does not know or care about morsel boundaries. It sees a
column slice via `ctx.column::<T>()` and processes all elements in the
slice. The scheduler decides the slice boundaries. This is the morsel
abstraction from Leis et al.: the pipeline stage is written as if it
processes one batch; the scheduler decides how big the batch is.

### Waker access

The Context provides a waker for I/O-bound work units that need to
yield Pending:

```rust
impl<'a, R: AccessSet, W: AccessSet> Context<'a, R, W> {
    /// Returns a waker handle for this work unit.
    ///
    /// The work unit clones this waker and passes it to its I/O code.
    /// When the I/O completes, the I/O code calls waker.wake(), which
    /// sets an atomic flag. The scheduler detects the flag on its next
    /// scan and moves the work unit back to the runnable queue.
    pub fn waker(&self) -> &WakeHandle {
        &self.wake_handle
    }
}
```

The WakeHandle is an index into the scheduler's atomic flag array. It
is `no_std`-compatible (no heap allocation, no vtable, just an array
index and an `AtomicBool` reference). Consumers that want event-driven
waking (Tokio integration) wrap the atomic flag with their own
notification mechanism.

## 5. Morsel vs Scalar Dispatch

Q4 from the topic asks how the work unit indicates whether it processes
morsels or single items. The answer: it does not indicate anything. The
scheduler infers dispatch mode from the access declarations.

### Inference rule

The scheduler examines the AccessSet store IDs and their cardinalities:

- If any store in Read or Write has `Cardinality::Many` (Column<T>),
  the scheduler uses morsel dispatch. The work unit is invoked once per
  morsel. Each invocation receives a Context scoped to the current
  morsel's row range.
- If no store has `Cardinality::Many`, the scheduler uses scalar
  dispatch. The work unit is invoked once. The Context has no morsel
  scoping (morsel_range is `0..0`).

This inference is automatic. No extra trait, no extra associated type,
no extra annotation. The access declarations that the work unit already
provides for DAG construction double as dispatch mode declarations.

### Why this is correct

The dispatch mode is a consequence of the data shape, not an independent
decision. If a work unit reads column data, it must iterate over rows.
Morsel dispatch is the mechanism for parallel iteration over rows. If a
work unit reads only singleton resources and virtual signals, there is
nothing to iterate. Scalar dispatch is the only sensible mode.

A work unit cannot be "scalar" while reading a column. What would scalar
mean? Process one row? That is a morsel of size 1, which morsel
dispatch handles (degenerate case). Process all rows in one call? That
is a morsel of size N (the whole column), which morsel dispatch also
handles (the scheduler can choose morsel_size = total_rows).

Conversely, a work unit cannot be "morsel" while reading only resources.
There are no rows to chunk. Dispatching it multiple times with different
morsel ranges would just call it repeatedly with the same resource state,
which is wasteful and semantically wrong.

The inference rule eliminates a decision point. The work unit author
declares what data they access, and the dispatch mode follows. No
possibility of mismatch (declaring scalar dispatch but reading a column,
or declaring morsel dispatch but reading only resources).

### Implementation in the scheduler

```rust
fn determine_dispatch_mode(access: &[(StoreId, Cardinality)]) -> DispatchMode {
    let has_column = access.iter().any(|(_, c)| *c == Cardinality::Many);
    if has_column {
        DispatchMode::Morsel
    } else {
        DispatchMode::Scalar
    }
}
```

For morsel dispatch, the scheduler computes the morsel size based on the
column lengths and the number of available workers. The Leis et al.
default of 10,000 rows per morsel is a reasonable starting point. The
scheduler can adapt the morsel size based on per-morsel execution time:
if morsels complete too fast (scheduling overhead dominates), increase
the size; if morsels take too long (load imbalance), decrease the size.

### Mixed access: columns and resources

A work unit that reads both columns and resources is dispatched in
morsel mode. Each invocation receives a morsel-scoped view of the
columns and an unscoped view of the resources. The resources are the
same across all morsel invocations; the columns differ.

```rust
struct EnrichEntities;

impl WorkUnit for EnrichEntities {
    type Read = (Column<EntityId>, Resource<EnrichmentCache>);
    type Write = (Column<EnrichedEntity>,);

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        let cache = ctx.resource::<EnrichmentCache>();
        let ids = ctx.column::<EntityId>();
        let mut output = ctx.column_mut::<EnrichedEntity>();

        for (id, out) in ids.iter().zip(output.iter_mut()) {
            *out = cache.lookup(id);
        }

        Progress::Complete
    }
}
```

The scheduler dispatches this in morsel mode because `Column<EntityId>`
has `Cardinality::Many`. Each morsel gets its own slice of
`Column<EntityId>` and `Column<EnrichedEntity>`, but the same
`Resource<EnrichmentCache>`. The resource is read-shared across all
concurrent morsel invocations (multiple workers can call
`ctx.resource::<EnrichmentCache>()` simultaneously with `&` access).

## 6. Stateless vs Stateful Work Units

Most morsel work units are stateless: each execute() call processes one
morsel independently. The `&mut self` receiver is unused. The question:
should we optimize for this?

### Stateless morsel work unit

```rust
pub struct SquareColumn;

impl WorkUnit for SquareColumn {
    type Read = (Column<f64>,);
    type Write = (Column<f64>,);

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        let input = ctx.column::<f64>();
        let mut output = ctx.column_mut::<f64>();
        for (x, out) in input.iter().zip(output.iter_mut()) {
            *out = x * x;
        }
        Progress::Complete
    }
}
```

The struct has no fields. `&mut self` is `&mut ()` (a zero-sized mutable
reference). The compiler optimizes this away entirely: the self parameter
occupies no register, generates no load/store, and has no effect on the
generated machine code. There is no performance cost to the `&mut self`
signature for stateless work units.

We do not need a separate `StatelessWorkUnit` trait or a marker
associated type. The signature accommodates both cases. The compiler
handles the optimization. Adding a stateless specialization would double
the trait surface for zero performance benefit.

### Stateful work unit: hash build

A hash join's build phase accumulates rows into a hash table across
multiple morsel batches. The hash table is the work unit's internal
state.

```rust
pub struct HashBuild<K: Hash + Eq, V> {
    table: HashMap<K, Vec<V>>,
}

impl<K, V> WorkUnit for HashBuild<K, V>
where
    K: Hash + Eq + Send + Sync + 'static,
    V: Send + Sync + 'static + Clone,
{
    type Read = (Column<K>, Column<V>);
    type Write = (Resource<HashTable<K, V>>,);

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        let keys = ctx.column::<K>();
        let vals = ctx.column::<V>();
        for (k, v) in keys.iter().zip(vals.iter()) {
            self.table.entry(k.clone()).or_default().push(v.clone());
        }
        // After all morsels, write the table to the resource.
        // How does the work unit know it is the last morsel?
        // It does not; the scheduler handles finalization (see below).
        Progress::Complete
    }
}
```

The hash table grows across morsel invocations. Each execute() call adds
the current morsel's rows to the table. The `&mut self` receiver gives
the work unit exclusive access to the table between invocations.

This raises a finalization question: when all morsels are processed, the
work unit needs to write its accumulated hash table to the output
resource. But each execute() call returns `Complete` for its morsel; the
work unit does not know whether more morsels remain.

Two approaches:

**A) Scheduler calls a finalize method.** The WorkUnit trait has an
optional `fn finalize(&mut self, ctx: Context<...>)` method (default
no-op) that the scheduler calls after all morsels are dispatched. The
hash build writes its table to the resource in finalize().

**B) The scheduler provides morsel metadata.** The Context includes
`ctx.is_last_morsel() -> bool`. The work unit writes the table on the
last morsel. This is simpler but couples the work unit to morsel
scheduling mechanics.

Option A is cleaner. The finalize method is a lifecycle hook separate
from morsel processing. Not all work units need it (stateless units use
the default no-op). Work units that accumulate state across morsels
override it. The scheduler guarantees that finalize() is called exactly
once, after all morsels have completed.

### Stateful work unit: HTTP connector

An HTTP connector is stateful across yields, not across morsels. It
tracks its progress through a request/response cycle.

```rust
enum FetcherState {
    Ready,
    Fetching { handle: RequestHandle, wake: WakeHandle },
    Processing { body: Vec<u8> },
}

pub struct HttpFetcher {
    url: Url,
    state: FetcherState,
}

impl WorkUnit for HttpFetcher {
    type Read = (Resource<HttpClient>,);
    type Write = (Column<Metadata>, Virtual<EntityDiscovered>);

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        match &mut self.state {
            FetcherState::Ready => {
                let client = ctx.resource::<HttpClient>();
                let handle = client.start_request(&self.url);
                let wake = ctx.waker().clone();
                handle.on_complete(wake.clone());
                self.state = FetcherState::Fetching { handle, wake };
                Progress::Yield(YieldReason::Pending)
            }
            FetcherState::Fetching { handle, .. } => {
                match handle.try_recv() {
                    Some(response) => {
                        self.state = FetcherState::Processing {
                            body: response.body,
                        };
                        // immediately re-execute to process
                        Progress::Yield(YieldReason::BudgetExhausted)
                    }
                    None => {
                        // still waiting; should not happen if waker fired correctly
                        Progress::Yield(YieldReason::Pending)
                    }
                }
            }
            FetcherState::Processing { body } => {
                let metadata = parse_metadata(body);
                let mut col = ctx.column_mut::<Metadata>();
                col.push(metadata);
                ctx.emit::<EntityDiscovered>();
                self.state = FetcherState::Ready;
                Progress::Complete
            }
        }
    }
}
```

The state machine in the work unit's fields is the continuation. Each
execute() call advances the state machine one step. Between calls, the
scheduler runs other work. This is the explicit state machine pattern
from the yielding deep dive: less ergonomic than async/await, but
portable (no_std compatible), debuggable (the state is a plain enum,
inspectable at any point), and transparent (no hidden compiler-generated
state machine).

## 7. Named Work Units

Every work unit needs a name for debugging, logging, scheduler error
messages, and diagnostic tooling. The question: how is the name provided?

### Option A: associated const on WorkUnit

```rust
pub trait WorkUnit: Send + Sync + 'static {
    const NAME: &'static str;
    type Read: AccessSet;
    type Write: AccessSet;
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress;
}
```

The name is part of the trait. Every implementation must provide it. Zero
runtime cost (const, not a method call). The scheduler uses `T::NAME`
directly.

### Option B: separate Named trait

polka-primitives already defines `Named`:

```rust
pub trait Named {
    const NAME: &'static str;
}
```

WorkUnit could require `Named` as a supertrait:

```rust
pub trait WorkUnit: Named + Send + Sync + 'static {
    type Read: AccessSet;
    type Write: AccessSet;
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress;
}
```

This reuses the existing trait and aligns with polka-dots and saalis,
which both use `Named` for identity.

### Option C: std::any::type_name

Derive the name from the Rust type name: `std::any::type_name::<Self>()`.
No explicit name needed. But `type_name` returns the full module path
(`polka_source_brew::cataloguer::SteamCataloguer`), which is noisy for
logging. The output is also not stable across compiler versions (the Rust
documentation explicitly warns against relying on its format).

Additionally, polka-dots bans `std::any` entirely. This option is
disqualified on two grounds: instability and the ban.

### Decision: associated const on WorkUnit

Option A is the simplest and most explicit. The name is right there in
the trait definition. Every implementation must provide it; no separate
trait needed.

We do not use Option B (separate Named supertrait) because hilavitkutin
should not depend on polka-primitives. hilavitkutin is a shared library
across polka-dots, saalis, and loimu. Making it depend on polka-primitives
would create a circular dependency (polka depends on hilavitkutin,
hilavitkutin depends on polka-primitives, which is part of polka). The
NAME const on WorkUnit is self-contained.

Consumers that also implement `Named` (polka and saalis types) can use
the same string constant for both:

```rust
const CATALOGUER_NAME: &str = "steam-cataloguer";

impl Named for SteamCataloguer {
    const NAME: &'static str = CATALOGUER_NAME;
}

impl WorkUnit for SteamCataloguer {
    const NAME: &'static str = CATALOGUER_NAME;
    // ...
}
```

No conflict, no coupling, one source of truth for the string.

## 8. Retry Configuration

Should retry behavior be encoded in the trait (static, per-type) or in
the Progress return value (dynamic, per-invocation)?

### On the trait (static)

```rust
pub trait WorkUnit: Send + Sync + 'static {
    const NAME: &'static str;
    const MAX_RETRIES: u32 = 0;
    const RETRY_BACKOFF_BASE: Duration = Duration::from_millis(100);
    type Read: AccessSet;
    type Write: AccessSet;
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress;
}
```

The scheduler reads the retry limits from the trait consts. Every failure
of this work unit retries with the same parameters. Simple. The work unit
author sets the limits once and does not think about retry in execute().

The problem: different failures warrant different retry strategies. A
rate limit (429) should retry with a long backoff (the Retry-After
header value). A connection timeout should retry with a short backoff.
An authentication failure (401) should not retry at all. Static consts
cannot express this variation.

### On the Progress enum (dynamic)

```rust
pub enum Progress {
    Complete,
    Yield(YieldReason),
    Failed(Error),
}

pub enum YieldReason {
    BudgetExhausted,
    Pending,
    Retry { attempt: u32, max: u32, backoff: Duration },
}
```

The work unit decides per-failure. A 429 returns `Retry { attempt: 1,
max: 5, backoff: Duration::from_secs(30) }`. A timeout returns
`Retry { attempt: 1, max: 3, backoff: Duration::from_millis(500) }`. A
401 returns `Failed(Error::new("authentication failed"))`. Each failure
gets its own retry parameters.

The work unit tracks the attempt count in its own fields. The scheduler
does not need to track retries per work unit; the work unit tells it
everything via the Progress return value.

### Decision: Progress enum only

Dynamic retry in the Progress enum is strictly more expressive than
static consts. Everything static consts can do, the Progress enum can
also do (the work unit can use const values for its retry parameters).
But the Progress enum can also express per-failure variation, which
static consts cannot.

Adding static consts to the trait would be redundant if the Progress
enum already carries retry parameters. It would also create a question
of precedence: if the trait says `MAX_RETRIES = 3` but execute() returns
`Retry { max: 5 }`, which wins? Eliminating the static consts
eliminates the ambiguity.

The scheduler's responsibility is simple: when it receives
`Yield(Retry { attempt, max, backoff })`, it checks `attempt >= max`.
If true, convert to `Failed`. If false, delay by `backoff + jitter` and
re-dispatch. The scheduler adds jitter (the work unit does not know
about global scheduling state and should not try to jitter itself).

## 9. Lifecycle Hooks

Beyond execute(), does the work unit need additional lifecycle methods?

### init

```rust
fn init(&mut self, ctx: Context<Self::Read, Self::Write>) {}
```

Called once when the work unit is first added to the scheduler. The work
unit can perform one-time setup: allocating buffers, establishing
connections, validating configuration. The Context provides access to
the same stores declared in Read and Write.

Without init, the work unit must perform setup on the first execute()
call, guarded by an `initialized: bool` field. This is boilerplate that
every stateful work unit would need to write. The init hook eliminates
it.

### finalize

```rust
fn finalize(&mut self, ctx: Context<Self::Read, Self::Write>) {}
```

Called once after all morsel dispatches complete (morsel mode) or after
the single dispatch completes (scalar mode). The work unit can perform
cleanup, write accumulated results, or release resources.

This is the answer to the hash build finalization problem from section
6. The hash build accumulates across morsels in execute() and writes
the hash table to the output resource in finalize(). The scheduler
guarantees that finalize() is called exactly once, after the last
morsel's execute() returns Complete.

For scalar work units, finalize() is called after execute() returns
Complete. It is redundant for simple scalar units (they can do everything
in execute()), but useful for scalar units that yield: the connector
yields multiple times, and finalize() runs after the final Complete.

### cleanup

```rust
fn cleanup(&mut self) {}
```

Called when the work unit is removed from the scheduler, regardless of
outcome (Complete, Failed, or scheduler shutdown). No Context parameter
because the stores may no longer be available. The work unit releases
any resources it holds: file handles, connection handles, temporary
buffers.

Without cleanup, resource release happens in `Drop`. But `Drop` cannot
return errors and has no access to diagnostic infrastructure. The cleanup
hook provides a structured place for resource release with the ability
to log warnings (though not through the Context, since stores are
unavailable).

### The complete trait with lifecycle hooks

```rust
pub trait WorkUnit: Send + Sync + 'static {
    const NAME: &'static str;

    type Read: AccessSet;
    type Write: AccessSet;

    /// Called once when the work unit is first scheduled.
    fn init(&mut self, _ctx: Context<Self::Read, Self::Write>) {}

    /// Called repeatedly until the work unit returns Complete.
    /// For morsel dispatch: called once per morsel.
    /// For scalar dispatch: called once (or multiple times if yielding).
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress;

    /// Called once after all morsel dispatches complete.
    fn finalize(&mut self, _ctx: Context<Self::Read, Self::Write>) {}

    /// Called when the work unit is removed from the scheduler.
    /// No context available; stores may be gone.
    fn cleanup(&mut self) {}
}
```

All lifecycle methods except execute() have default implementations
(no-op). Most work units implement only execute(). Stateful morsel
processors add finalize(). Complex connectors add init() and cleanup().
The trait remains simple for simple cases and extensible for complex
ones.

## 10. The Complete Trait

Assembling all decisions from sections 1 through 9:

```rust
/// A schedulable unit of work in the hilavitkutin pipeline.
///
/// Each work unit declares its data dependencies via Read and Write
/// associated types. The scheduler builds a DAG from these declarations
/// and dispatches work units in topological order.
///
/// Work units with Column<T> in their access set are dispatched in
/// morsel mode: execute() is called once per morsel (a fixed-size
/// slice of the column). Work units with only Resource<T> and
/// Virtual<T> are dispatched in scalar mode: execute() is called once.
///
/// Work units may yield by returning Progress::Yield. The scheduler
/// parks the work unit and runs other ready work. The work unit holds
/// its continuation state in its own struct fields.
pub trait WorkUnit: Send + Sync + 'static {
    /// Human-readable name for diagnostics, logging, and scheduling.
    const NAME: &'static str;

    /// Stores this work unit reads. Tuple of Resource<T>, Column<T>,
    /// and/or Virtual<T>. Use () for no reads.
    type Read: AccessSet;

    /// Stores this work unit writes. Tuple of Resource<T>, Column<T>,
    /// and/or Virtual<T>. Use () for no writes.
    type Write: AccessSet;

    /// Called once when the work unit is first added to the scheduler.
    /// Use for one-time setup: buffer allocation, connection
    /// establishment, configuration validation.
    fn init(&mut self, _ctx: Context<Self::Read, Self::Write>) {}

    /// Process one morsel (morsel mode) or run the work unit (scalar
    /// mode). Returns Progress indicating the work unit's disposition.
    ///
    /// Complete: done. Output is written.
    /// Yield(BudgetExhausted): ran out of time, re-dispatch next quantum.
    /// Yield(Pending): waiting on I/O, park until waker fires.
    /// Yield(Retry { .. }): failed transiently, retry with backoff.
    /// Failed(Error): permanent failure. Rollback output.
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress;

    /// Called once after all morsel dispatches complete (morsel mode)
    /// or after execute() returns Complete (scalar mode). Use for
    /// writing accumulated results.
    fn finalize(&mut self, _ctx: Context<Self::Read, Self::Write>) {}

    /// Called when the work unit is removed from the scheduler.
    /// No context available. Use for resource cleanup.
    fn cleanup(&mut self) {}
}
```

### Supporting types

```rust
/// The outcome of a single execute() call.
pub enum Progress {
    /// Work unit finished successfully. Output is committed.
    Complete,
    /// Work unit needs to be called again.
    Yield(YieldReason),
    /// Work unit failed permanently. Output is rolled back.
    Failed(Error),
}

/// Why the work unit yielded.
pub enum YieldReason {
    /// Ran out of time. Ready to run again immediately.
    BudgetExhausted,
    /// Waiting on an external event. Park until waker fires.
    Pending,
    /// Transient failure. Retry after backoff.
    Retry {
        /// Current attempt number (0-indexed).
        attempt: u32,
        /// Maximum attempts before converting to Failed.
        max: u32,
        /// Base backoff duration. Scheduler adds jitter.
        backoff: Duration,
    },
}

/// Typed access to the stores declared by a work unit.
pub struct Context<'a, R: AccessSet, W: AccessSet> {
    world: &'a World,
    morsel_range: Range<usize>,
    wake_handle: WakeHandle,
    _read: PhantomData<R>,
    _write: PhantomData<W>,
}

/// Waker handle for I/O-bound work units.
///
/// An index into the scheduler's atomic flag array. When woken, the
/// corresponding flag is set and the scheduler moves the work unit
/// back to the runnable queue.
#[derive(Clone)]
pub struct WakeHandle {
    flag_index: usize,
    flags: Arc<[AtomicBool]>,
}

impl WakeHandle {
    pub fn wake(&self) {
        self.flags[self.flag_index].store(true, Ordering::Release);
    }
}
```

## 11. Comparison with loimu and Bevy

This section compares hilavitkutin's WorkUnit with loimu's Behavior
and Bevy's System across eight dimensions.

### Trait shape

| Dimension | hilavitkutin WorkUnit | loimu Behavior | Bevy System |
|---|---|---|---|
| Receiver | `&mut self` | `&self` (stateless) | `fn` (stateless) |
| Access declaration | Associated types (Read, Write) | Macro-generated access analysis | Function parameter types |
| Return type | `Progress` enum | Unit `()` (atomic, no yield) | Unit `()` |
| Entry point | `fn execute()` | `fn handle()` (generated) | Bare function |
| Lifecycle hooks | init, finalize, cleanup | None (setup via resources) | Startup systems, OnExit |

hilavitkutin and loimu both use typed access declarations for DAG edge
construction. The mechanism differs: hilavitkutin uses associated types
that the programmer writes; loimu uses macro-generated access analysis
that inspects handler function bodies. Bevy uses function parameter
types (Query, Res, ResMut) with trait-based extraction.

loimu's `&self` receiver reflects its atomic execution model. Behaviors
run to completion; there is no continuation state. All mutable state
lives in resources accessed through the handler's parameters.
hilavitkutin's `&mut self` reflects its yielding model: the work unit
holds continuation state across yield points.

Bevy's "system as function" model is the most ergonomic: no struct
definition, no trait impl, just a function with typed parameters. But it
sacrifices explicitness: the access declarations are implicit in the
function signature, discovered via trait magic at registration time. And
it cannot hold continuation state (functions are stateless; closures
can capture state but the capture is opaque to the scheduler).

### State model

| Dimension | hilavitkutin | loimu | Bevy |
|---|---|---|---|
| Work unit state | In struct fields (private) | In resources (shared) | In Local<T> (per-system) |
| Continuation state | In struct fields via yield | N/A (no yielding) | N/A (no yielding) |
| Cross-invocation state | Persisted by scheduler | Persisted by resource store | Persisted by world |
| Visibility | Private to work unit | Visible to other behaviors | Visible to other systems |

hilavitkutin keeps work unit state private. The scheduler does not
inspect it, other work units cannot access it, and it does not appear in
the DAG. This is the correct model for continuation state (the scheduler
should not care about the HTTP connector's internal state machine) and
for accumulation state (the hash build's partial table is an
implementation detail, not a dependency).

loimu externalizes all state into resources. This is consistent with its
design philosophy (resources are the ground truth, behaviors are
transformations) but means that every piece of state is visible in the
DAG and subject to access control. If a behavior needs a scratch buffer,
it must declare a resource for the buffer and include it in its access
set. This creates DAG edges that do not represent real data dependencies.

Bevy's `Local<T>` type provides per-system private state without DAG
visibility, similar to hilavitkutin's approach. It was added
specifically to address the problem of systems needing private scratch
data without polluting the ECS world.

### Access pattern

| Dimension | hilavitkutin | loimu | Bevy |
|---|---|---|---|
| Access granularity | Store type | Field | Component type |
| Access modes | Read, Write | 7 modes (Pinned, Immediate, etc.) | Read, Write (Res, ResMut) |
| Compile-time safety | Via Contains trait | Via macro analysis | Via SystemParam trait |
| Runtime validation | None needed | At registration (discovery) | At system registration |

hilavitkutin's two access modes (Read, Write) are the common
denominator across all three consumers. loimu's seven modes add
scheduling nuance (snapshotting, approximation, deferred writes) that
polka and saalis do not need. loimu can map its seven modes to Read and
Write for DAG construction and apply the finer distinctions in its own
scheduling layer on top of hilavitkutin.

The granularity difference (store type vs field vs component type) is
deliberate. hilavitkutin tracks dependencies at the store level:
`Column<Position>`, not `Column<Position>.x`. loimu refines to field
granularity in its own scheduling pass. Bevy tracks at component type
granularity, which is structurally identical to hilavitkutin's store
type granularity.

### Dispatch

| Dimension | hilavitkutin | loimu | Bevy |
|---|---|---|---|
| Dispatch mode | Morsel or scalar (inferred) | Per-scheduling-unit (atomic) | Per-system (one call) |
| Morsel support | Native (Column in access set) | Via `each` blocks (macro) | Via ParallelIterator |
| Parallelism | Work-stealing across morsels | Lane assignment + work stealing | Multi-threaded executor |
| Load balancing | Morsel granularity | Scheduling unit granularity | System granularity |

hilavitkutin and loimu both support morsel-level parallelism. The
mechanism differs: hilavitkutin dispatches the same work unit multiple
times with different morsel ranges; loimu decomposes a behavior's
`each` block into scheduling units, each processing a slice of the
collection.

Bevy's `ParallelIterator` provides morsel-like parallelism for query
iteration but at a different level: the system itself decides whether
to iterate in parallel (via `par_iter()`), not the scheduler. The
scheduler parallelizes across systems, not within a system.

hilavitkutin's approach (scheduler decides morsel boundaries) is more
in line with database morsel dispatch (Leis et al.) where the
dispatcher owns morsel sizing and assignment. This gives the scheduler
control over load balancing and backpressure.

### Yielding

| Dimension | hilavitkutin | loimu | Bevy |
|---|---|---|---|
| Yielding | Explicit (Progress::Yield) | None (atomic execution) | None (run to completion) |
| Continuation | Self fields (state machine) | N/A | N/A |
| Budget enforcement | BudgetExhausted variant | N/A | N/A |
| I/O parking | Pending variant + waker | N/A | N/A |
| Retry | Retry variant with backoff | N/A | N/A |

This is hilavitkutin's primary contribution beyond loimu and Bevy. Neither
loimu nor Bevy supports yielding in their current designs. loimu
explicitly deferred it to post-1.0 (the Naughty Dog fiber model was
studied and rejected). Bevy's systems run to completion within a single
frame; long-running work is handled by spawning async tasks through
`AsyncComputeTaskPool`.

hilavitkutin's yield model fills the gap identified in the loimu
synthesis: cooperative scheduling for I/O-bound work (saalis connectors),
long-running computation (hash builds that exceed a time quantum), and
retry with backoff (transient failures).

### Naming

| Dimension | hilavitkutin | loimu | Bevy |
|---|---|---|---|
| Name source | `const NAME` on trait | `define_behavior!` macro arg | `std::any::type_name` |
| Name type | `&'static str` | `&'static str` | `&'static str` |
| Stability | Author-controlled | Author-controlled | Compiler-dependent |
| Uniqueness | Not enforced by trait | Enforced at registration | Not enforced |

hilavitkutin uses an explicit name const, which is stable and
author-controlled. loimu uses macro-provided names, also stable and
author-controlled. Bevy uses `std::any::type_name`, which is
compiler-dependent and includes module paths.

hilavitkutin does not enforce name uniqueness in the trait; that is a
scheduler-level concern. The scheduler validates uniqueness at
registration time and rejects duplicates with a diagnostic error. This
matches loimu's approach (BehaviorRegistry validates uniqueness at
discovery time).

### Lifecycle

| Dimension | hilavitkutin | loimu | Bevy |
|---|---|---|---|
| Setup | init() hook | Resources initialized at startup | Startup systems |
| Teardown | cleanup() hook | Drop on resources | N/A |
| Finalization | finalize() hook | N/A (atomic execution) | N/A |
| Fault handling | Failed return + rollback | Panic catch + snapshot restore | Panic = crash |

hilavitkutin provides three lifecycle hooks (init, finalize, cleanup)
with default no-op implementations. loimu does not need lifecycle hooks
because behaviors are stateless (setup and teardown happen on resources,
not on behaviors). Bevy uses startup systems for setup and does not have
explicit teardown.

The finalize hook is unique to hilavitkutin and exists because of morsel
dispatch. When a work unit processes N morsels and accumulates state
(hash build, aggregation), it needs a final call to commit the
accumulated result. loimu does not need this because its scheduling
units are atomic (no accumulation across invocations). Bevy does not
need this because its systems process all entities in a single call.

### Error handling

| Dimension | hilavitkutin | loimu | Bevy |
|---|---|---|---|
| Error signal | Progress::Failed(Error) | Panic (caught at boundary) | Panic (uncaught) |
| Rollback | Sole-writer snapshot / undo journal | Same | None |
| Downstream effect | Skip via version stamp | Skip via version stamp | Crash |
| Retry | Progress::Yield(Retry { .. }) | None | None |
| Circuit breaking | Consumer layer (scheduler policy) | None | None |

hilavitkutin and loimu share the same rollback model: sole-writer
resources are snapshotted before execution, shared-writer resources use
undo journals, and failed writes are reverted. The difference is in
error signaling: hilavitkutin uses a typed return value
(`Progress::Failed`), loimu catches panics at the dispatch boundary.

The typed return value is preferable for two reasons. First, panics are
not meant for control flow in Rust; they indicate programming errors,
not expected failures. A network timeout is an expected failure, not a
bug. Second, `Progress::Failed(Error)` carries structured error
information (the `Error` type includes the failure message, category,
and context). A panic's payload is an opaque `Box<dyn Any>` that
requires downcasting.

Bevy does not handle system panics. A panicking system crashes the
application. This is acceptable for games (a crashing system indicates a
bug that should be fixed, not a transient failure that should be
retried) but not for data pipelines (a network timeout should not crash
the build).

## 12. Three Consumers, One Trait

The final test of the WorkUnit design is whether it serves all three
consumers without forcing any of them into unnatural contortions.

### polka-dots: template pipeline stage

```rust
pub struct ParsePolkaTemplates;

impl WorkUnit for ParsePolkaTemplates {
    const NAME: &'static str = "parse-polka-templates";
    type Read = (Column<RawSource>,);
    type Write = (Column<ParsedAst>,);

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        let sources = ctx.column::<RawSource>();
        let mut asts = ctx.column_mut::<ParsedAst>();
        for (src, ast_slot) in sources.iter().zip(asts.iter_mut()) {
            match parse(src) {
                Ok(ast) => *ast_slot = ast,
                Err(e) => return Progress::Failed(e.into()),
            }
        }
        Progress::Complete
    }
}
```

Morsel dispatch (Column in access set). Stateless (no fields). No
yielding (CPU-bound, completes in one call). No lifecycle hooks needed.
The simplest possible WorkUnit implementation.

### saalis: HTTP connector with retry

```rust
pub struct SteamCataloguer {
    state: CataloguerState,
    pending_ids: Vec<SteamAppId>,
}

enum CataloguerState {
    Ready,
    Fetching { handle: RequestHandle, wake: WakeHandle },
    Parsing { body: Vec<u8> },
}

impl WorkUnit for SteamCataloguer {
    const NAME: &'static str = "steam-cataloguer";
    type Read = (Resource<HttpClient>, Resource<SteamApiConfig>);
    type Write = (Column<AppMetadata>, Virtual<EntityDiscovered>);

    fn init(&mut self, ctx: Context<Self::Read, Self::Write>) {
        let config = ctx.resource::<SteamApiConfig>();
        self.pending_ids = config.app_ids_to_fetch().to_vec();
    }

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        match &mut self.state {
            CataloguerState::Ready => {
                if self.pending_ids.is_empty() {
                    return Progress::Complete;
                }
                let client = ctx.resource::<HttpClient>();
                let id = self.pending_ids.pop().unwrap();
                let handle = client.get(&format!(
                    "https://store.steampowered.com/api/appdetails?appids={}",
                    id
                ));
                let wake = ctx.waker().clone();
                handle.on_complete(wake.clone());
                self.state = CataloguerState::Fetching { handle, wake };
                Progress::Yield(YieldReason::Pending)
            }
            CataloguerState::Fetching { handle, .. } => {
                match handle.try_recv() {
                    Some(Ok(response)) if response.status == 429 => {
                        let backoff = response
                            .header("Retry-After")
                            .and_then(|v| v.parse().ok())
                            .map(Duration::from_secs)
                            .unwrap_or(Duration::from_secs(30));
                        self.state = CataloguerState::Ready;
                        Progress::Yield(YieldReason::Retry {
                            attempt: 0,
                            max: 5,
                            backoff,
                        })
                    }
                    Some(Ok(response)) => {
                        self.state = CataloguerState::Parsing {
                            body: response.body,
                        };
                        Progress::Yield(YieldReason::BudgetExhausted)
                    }
                    Some(Err(e)) => {
                        self.state = CataloguerState::Ready;
                        Progress::Yield(YieldReason::Retry {
                            attempt: 0,
                            max: 3,
                            backoff: Duration::from_millis(500),
                        })
                    }
                    None => Progress::Yield(YieldReason::Pending),
                }
            }
            CataloguerState::Parsing { body } => {
                let metadata = parse_steam_response(body);
                let mut col = ctx.column_mut::<AppMetadata>();
                col.push(metadata);
                ctx.emit::<EntityDiscovered>();
                self.state = CataloguerState::Ready;
                // loop back to Ready to process next pending ID
                Progress::Yield(YieldReason::BudgetExhausted)
            }
        }
    }

    fn cleanup(&mut self) {
        self.pending_ids.clear();
    }
}
```

Scalar dispatch (no Column in Read; writes to Column but the work unit
manages its own iteration over pending_ids). Stateful (state machine
across yields). Uses init for setup, cleanup for teardown. Uses all
three yield variants: Pending for I/O wait, BudgetExhausted for
immediate re-dispatch (state machine transition), Retry for rate limits
and transient errors. Uses Virtual for event emission.

This is the most complex WorkUnit pattern: I/O-bound, stateful,
yielding, with retry. The trait handles it without extensions.

### loimu: behavior adapter

loimu can adapt its existing Behavior model to WorkUnit. Because loimu
behaviors are atomic (no yielding) and stateless (&self), the adapter
is thin:

```rust
pub struct LoimuBehaviorAdapter<B: LoimuBehavior> {
    behavior: B,
}

impl<B: LoimuBehavior> WorkUnit for LoimuBehaviorAdapter<B> {
    const NAME: &'static str = B::NAME;
    type Read = B::Reads;
    type Write = B::Writes;

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        self.behavior.handle(ctx.into());
        Progress::Complete
    }
}
```

The adapter wraps a loimu Behavior, delegates execute() to handle(),
and always returns Complete (loimu behaviors are atomic). loimu's own
scheduling layer sits above hilavitkutin and adds signal-driven wakeup,
rotation groups, temporal prediction, and the seven access modes. None
of those concerns leak into WorkUnit; they are loimu's extensions on
top of hilavitkutin's base.

The adapter demonstrates that WorkUnit is general enough to serve as a
foundation layer without constraining loimu's more elaborate scheduling
model. loimu does not lose any functionality by adopting WorkUnit; it
gains shared infrastructure (DAG construction, morsel dispatch, work
stealing) and contributes its more advanced features as a layer on top.

## 13. Open Questions

Three design questions remain unresolved by this document. Each requires
input from the other research tracks.

**Column write semantics for morsel dispatch.** When a morsel work unit
writes to a Column, does it write to a pre-allocated slot in the output
column (the morsel range determines the write offset), or does it append
to a thread-local buffer that the scheduler merges after all morsels
complete? Pre-allocated slots require the output column size to be known
in advance. Thread-local buffers require a merge phase. The unified
store deep dive should resolve this.

**Version stamp placement.** Where do version stamps live? On each store
(one stamp per Column<T>, one stamp per Resource<T>)? On each work
unit (one stamp per write)? On each morsel (one stamp per morsel write)?
The stamp granularity affects incremental computation and rollback. The
yielding deep dive should resolve this.

**no_std compatibility of Context and WakeHandle.** The current
WakeHandle uses `Arc<[AtomicBool]>`, which requires alloc. For no_std
without alloc (embedded targets), the flag array must be statically
allocated. The scheduler could take a `&'static [AtomicBool]` reference
from a static array, but the array size must be fixed at compile time.
This is a platform portability question for the build configuration
topic.
