# Doc Changelist: hilavitkutin-ctx parameterised providers

**Date:** 2026-04-20
**Phase:** DOC
**Scope:** `hilavitkutin-ctx` crate only.
**Source topics:**
- 202603181200_topic.hilavitkutin-design-consolidation.md (engine model; Context<R, W> superseded per 2026-04-26 user call)

## Purpose

Prerequisite round for #34 (hilavitkutin-api contracts). hilavitkutin's
WorkUnit execute parameter will be a ctx-framework `Context<P>` rather
than a bespoke `Context<R, W>`. For that to carry hilavitkutin's
compile-time access-set enforcement (`Contains<Column<T>>`), ctx's
accessor-trait / API-trait shapes need to support one capability
growth: *type parameters* on the accessor and API traits.

Current ctx assumes concrete-method APIs:
```rust
trait ConnectorApi {
    fn check(&self) -> Result<(), ()>;
}
provider!(ConnectorApi as HasConnector => connection);
```

hilavitkutin needs access-set-aware variants:
```rust
trait ColumnReaderApi<R: AccessSet> {
    fn read<T: ColumnValue>(&self, i: USize) -> T
    where R: Contains<Column<T>>;
}
// accessor trait also parameterised over R:
trait HasColumnReader<R: AccessSet> {
    type Reader: ColumnReaderApi<R>;
    fn reader(&self) -> &Self::Reader;
}
```

This round grows ctx with a parallel macro family — `provider_generic!`,
`tuple_generic!` — that accepts one type parameter on the accessor /
API traits. Existing `provider!` / `tuple!` / `define_providers!`
remain unchanged.

Key shape decisions:

1. **Additive, not replacing.** The existing concrete-method macros
   stay. Existing saalis consumers (scraper connectors) keep working
   unchanged. The new generic-method macros sit alongside.
2. **One type parameter per macro call.** Enough for
   hilavitkutin's `R: AccessSet` / `W: AccessSet` cases. A version
   with *multiple* parameters (e.g. `Has<R, W>` for op registrars)
   can come as a follow-up when the need concretely surfaces —
   hilavitkutin's own `HasEach<R, W>` is the trigger. Ships in this
   round because #34 needs it immediately.
3. **`Context<P>` unchanged.** The wrapper struct already carries
   `providers: P` and doesn't need any shape change — only the
   delegation-impl shape inside the macros changes to thread the
   generic param.

## Per-crate doc changes

### hilavitkutin-ctx — `mock/crates/hilavitkutin-ctx/DESIGN.md.tmpl`

Add a new section after "Framework DX — macros":

- **Parameterised accessor / API traits.** Explain when to reach for
  them (consumer wants a compile-time-checked access bound, or a
  generic method that's monomorphised per call site). Document
  `provider_generic!` and `tuple_generic!` with a worked
  hilavitkutin-style example (ColumnReaderApi<R> and HasColumnReader<R>).

Update the "Design rule" section to note: parameterised variants
stay zero-deps, no std, no alloc — same constraints as the concrete
form. Consumers can mix generic and concrete providers in the same
tuple.

### hilavitkutin-ctx — `mock/crates/hilavitkutin-ctx/BACKLOG.md.tmpl` (new file)

- Multi-parameter accessor variant (`HasX<R, W>` for op registrars).
  Defer until a second concrete consumer surfaces a multi-parameter
  need (current hilavitkutin-api HasEach<R, W> / HasBatch<R, W> /
  HasReduce<R, W> ARE multi-parameter — so this may ship as part of
  this round or a tight follow-up).
- `define_providers_generic!` facade that mixes concrete +
  parameterised providers in one call. Nice-to-have, not needed
  for #34.
