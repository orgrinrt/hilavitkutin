## 1. loisto Architecture Overview

loisto is a custom immutable media appliance OS that owns every layer from PID 1
to pixel. It is not a general-purpose Linux distribution. It is a purpose-built
system where every component is designed, compiled, and deployed as a single
coherent unit. The project targets a specific hardware platform (AMD AM4,
specifically the Lenovo ThinkCentre M75s Gen 2) and runs a known, bounded set of
processes at any given time: typically four to eight active processes at idle.

The system is organized as a 12-node graph architecture where each node owns an
irreducible concern. There are no tiers, no hierarchy, and no catch-all
"infrastructure" layer. The 12 nodes are: Init, IPC Bus, Scheduler, Data Broker,
Pacer, Compositor, Runner, Shell, Input, Audio, Network, and Update. Extensions
(emulators, media players, streaming clients) attach as leaf nodes that declare
capabilities and resource needs via a C ABI contract. The Runner node mediates
between extensions and the rest of the system.

The choice of Zig for system-level components is deliberate. loisto uses Zig for
all daemons and system services that operate below the UI layer: PID 1, init,
the IPC bus, the scheduler, the compositor controller, input handling, audio
routing, the data broker, network daemons, and the extension runner host. The
reasons are: (a) `@cImport` gives Zig native, zero-overhead consumption of C
headers without bindgen or FFI boilerplate, which matters when the entire ABI
layer is defined as C headers; (b) Zig's comptime system enables type-safe
configuration that would require proc macros in Rust; (c) Zig's explicit
allocator model gives fine-grained control over memory allocation patterns, which
matters for a system that governs power budgets per compute unit and bins memory
via cgroups; (d) `zig cc` serves as the cross-language linker for the entire
workspace, enabling cross-language LTO between Zig and Rust components sharing
LLVM IR at the link boundary.

Rust is used for the UI shell (via loimu, a custom reactive UI framework), the
build system orchestrator, and metadata providers. The boundary between Zig and
Rust is the IPC bus: UI crates publish/subscribe, system crates
publish/subscribe, they never import each other directly.

The codebase is organized into six workspace groups: abi/ (14 C ABI contract
crates), boot/ (6 boot chain crates), root/ (20 build infrastructure and driver
crates), sys/ (29 runtime service crates), shell/ (12 user-facing UI crates),
and ext/ (3 extension crates). All 84 crates are members of a single cargo
workspace regardless of source language. Each crate declares metadata including
domain (core, frame, stream, query, cold, transient, kernel, hardware), primary
language, delivery mechanism (rootfs, sysext, esp, initrd, build, none), and
optional language bindings.

The delivery model uses an immutable SquashFS rootfs with dm-verity, updated
atomically via RAUC A/B slot swaps. Hardware drivers and extension dependencies
are loaded dynamically as sysext SquashFS overlays via systemd-sysext. The system
boots without sysexts; init detects hardware and merges the appropriate driver
packs.

## 2. sched_ext BPF Scheduler

### 2.1 What sched_ext Is

sched_ext is a Linux kernel framework (merged in Linux 6.12) that allows loading
custom BPF programs as CPU schedulers. Instead of modifying the kernel's CFS
(Completely Fair Scheduler) directly, sched_ext provides hooks that a BPF program
can implement to make scheduling decisions: which task runs on which CPU, when to
preempt, how to handle migrations between cores. The BPF program runs in kernel
context with safety guarantees enforced by the BPF verifier, making it possible
to ship custom scheduling policies without maintaining a kernel fork of the
scheduler itself.

The key property of sched_ext for loisto is selectivity: the BPF scheduler only
handles processes that loisto manages. Everything else falls through to CFS. This
means the custom scheduling logic applies precisely to the workloads loisto
controls (extensions, frame pipeline components, system daemons), while any
background kernel activity or unmanaged processes use the standard scheduler.

### 2.2 How loisto Uses sched_ext

loisto's sched_ext usage is split across two crates:

**loisto-sched-bpf** (boot/kernel domain) is the BPF program itself. It is
compiled with `bpftool` and the LLVM BPF backend, not by cargo or zig. It lives
in the boot/ workspace group because it is loaded into the kernel during the boot
sequence by init and runs autonomously from that point. The BPF program reads
BPF maps on each scheduling decision and applies the workload assignments
written by userspace.

**loisto-schedctl** (sys/core domain, Zig) is the userspace daemon that
communicates with the BPF program. It loads the BPF scheduler via sched_ext,
writes workload assignments to BPF maps, and sets per-core CPPC hints. It
consumes power budget hints from the Pacer via IPC and workload descriptors from
the Data Broker.

The communication between schedctl and the BPF program happens via BPF maps,
which are shared data structures between userspace and kernel space. There is no
IPC bus involvement in this path; it is direct shared memory.

### 2.3 BPF Map Structure

The BPF maps define two primary data structures:

```
per_cpu_workload[cpu_id] -> { class, cppc_min, cppc_desired, cppc_max }
pid_assignment[pid]      -> { target_cpu, class, flags }
```

The `per_cpu_workload` map is indexed by CPU ID and contains the workload class
currently assigned to that core along with CPPC performance hints (minimum,
desired, and maximum frequency targets). The `pid_assignment` map is indexed by
PID and contains the target CPU, workload class, and behavioral flags for each
managed process.

schedctl writes to these maps. The BPF program reads them on each scheduling
event. This separation means scheduling policy lives in userspace (where it can
be updated, debugged, and reasoned about) while scheduling enforcement lives in
the kernel (where it has the authority and timing precision to actually move
tasks between cores).

### 2.4 CachyOS Kernel Fork

loisto runs a fork of CachyOS's kernel PKGBUILD configuration. CachyOS is a
performance-focused Arch Linux derivative that maintains a kernel with BORE
(Burst-Oriented Response Enhancer, a CFS modification for better interactive
responsiveness), sched_ext support, and ThinLTO (Link-Time Optimization applied
to the kernel itself). loisto's fork takes this as a base and adds custom
configuration for the specific AM4 hardware target: CPU mitigation selection
(AMD-relevant mitigations on, Intel-only off), huge page configuration, and
sched_ext as the default scheduling framework.

The CachyOS base is chosen because it already maintains the kernel patches loisto
needs (sched_ext, BORE, ThinLTO) and has an active maintenance cadence. loisto
does not maintain its own kernel patches; it configures an existing
well-maintained patchset.

## 3. schedctl Userspace Component

### 3.1 CPPC Hints

AMD Collaborative Processor Performance Control (CPPC) is the hardware interface
for setting per-core performance targets. Unlike traditional CPU frequency
scaling via governors (schedutil, performance, powersave), CPPC provides three
continuous parameters per core: minimum performance, desired performance, and
maximum performance. The hardware (CPU firmware) translates these hints into
actual clock frequencies, voltage, and power states.

loisto strips the standard Linux CPU frequency governors entirely. The Pacer
tells the Scheduler what the power budget is, and schedctl translates that into
CPPC hints per core. The path is: Pacer determines power allocation per compute
unit -> schedctl receives this via IPC topic `TOPIC_POWER_HINT` -> schedctl
writes CPPC values to BPF maps -> BPF program applies them via
`scx_bpf_cpuperf_set()` per core.

This direct control, bypassing governors, is possible because loisto owns the
entire power stack. There is no other software on the system competing for
frequency control. The Pacer governs on total power draw (watts), not on clock
speed (MHz). It sets a constant system power envelope and redistributes across
compute units dynamically based on current workload needs.

### 3.2 Synthetic E-Cores

On AMD Zen 3 processors (the primary target, Ryzen 7 5700G with 8 cores / 16
threads), all cores are architecturally identical. There is no hardware
distinction between performance and efficiency cores like on Intel Alder Lake or
Apple Silicon. loisto creates this distinction in software via CPPC.

The mechanism is straightforward: schedctl designates some cores as "economy" by
capping their CPPC max to a low value. Background tasks (system daemons,
monitors, non-latency-sensitive work) are pinned to these economy cores.
Latency-sensitive work (the extension's main thread, the compositor, PipeWire
audio) runs on the remaining "performance" cores with high CPPC hints.

On a 4C/8T system (like the base Ryzen 3 4350G), the split is typically: core 3
as economy (2 SMT threads), cores 0-2 as performance (6 SMT threads). Voltage is
per physical core on AMD, so setting one core to economy speed means both its SMT
threads run at economy speed.

This is a soft hybrid, not a hard partition. There are no cpuset or isolcpus
directives. The BPF scheduler owns all placement decisions and can flex: borrow
the economy core during audio silence gaps, redistribute if a core hits thermal
throttling, promote economy threads to performance cores if the workload demands
it.

### 3.3 Workload Descriptors

Each extension gets a WorkloadDescriptor that is cached in DuckDB by the Data
Broker. The descriptor contains the workload class (what kind of work this
extension does), preferred core assignment, and pinning preferences. schedctl
queries these descriptors from the broker when an extension launches and uses
them to populate the BPF maps.

The workload descriptor is part of a two-layer scheduling design: a static layer
(the descriptor, computed once per content item from metadata and manifest,
cached, reused on every launch) and a dynamic layer (the Pacer, which reads
actual metrics every frame and makes micro-adjustments). The static layer tells
the scheduler what the workload IS; the dynamic layer tells it what the workload
NEEDS right now.

## 4. pacerctl Frame Pacer

### 4.1 Role as Power Budget Arbiter

The Pacer (loisto-pacerctl, sys/frame domain, Zig, controller kind) sits between
the Compositor (which knows frame deadlines from display scanout timing) and the
rest of the frame pipeline (which needs to know how much work it can afford). Its
role is mediation: it receives thermal headroom data from hardware monitors
(hwmon), translates that into per-component power budgets (CPU, GPU, NPU), and
publishes those budgets to the IPC bus.

The Pacer governs on power draw, not clock speed. The total system power budget
is treated as constant. When one compute unit is idle (e.g., the NPU during a
retro game that does not benefit from neural upscaling), its power allocation is
donated to other units. When thermal pressure builds, the Pacer proactively
scales quality before hard thermal limits force frame drops. This is inspired
by PS5's SmartShift (constant power envelope) and Android ADPF (frame-deadline
hints and thermal headroom forecasting).

### 4.2 VRR-Aware Frame Targeting

The Pacer adjusts the target framerate within the display's VRR (Variable
Refresh Rate) window, typically 40-60 Hz on the target hardware. When the system
can sustain 60fps within power and thermal constraints, it targets 60fps. When
thermal pressure or workload intensity makes 60fps expensive, it drops the target
to a lower rate within the VRR window. The display matches the game's actual
frame rate with no tearing or interpolation needed.

Frame interpolation only kicks in when: (a) the game drops below the VRR minimum
(low framerate compensation), or (b) the display does not support VRR at all.
When loisto's own interpolation is active, FreeSync Premium LFC (Low Framerate
Compensation) should be disabled to avoid double-compensating.

VRR is treated as a power tool: targeting a frame time achievable within the
thermal budget and letting VRR absorb natural variance means fewer interpolated
frames, less compute, less power, and less heat.

### 4.3 Quality Decisions

Based on scene classification from the Analyser (an NPU-powered scene
classification system), the Pacer decides whether to enable interpolation, DRS
(Dynamic Resolution Scaling), and render scale adjustments. These are not binary
on/off decisions; they are continuous quality knobs that the Pacer adjusts based
on current power, thermal, and workload state.

User-facing priority knobs (not profiles, not discrete modes) influence the
Pacer's decisions: a "silence" knob caps total power below the fan curve
inflection point, a "visual quality" knob biases toward FSR2 and interpolation
even at higher thermal cost, a "responsiveness" knob biases toward lower latency
over visual quality.

### 4.4 Scene State Machine

The Pacer implements a five-state scene state machine that drives pipeline
behavior:

| State | Interpolation | DRS | Notes |
|-------|--------------|-----|-------|
| Gameplay | yes | yes | Full pipeline active |
| Menu/GUI | no | no | Sharpen + hold last frame |
| Cutscene | yes | no | Quality mode, no resolution scaling |
| Loading | n/a | n/a | Show loisto's own content (tips, art) |
| Scene cut | no | no | Flush history, hold + fade, no interpolation on stale data |

Scene classification is driven by the NPU-based Analyser component, which
detects game phases via heuristics (drop in GPU draw calls indicates menu, disk
I/O spike indicates loading) combined with neural scene inference. The scene
state machine is critical for avoiding visual artifacts: interpolating across a
scene cut produces garbage frames, so the Pacer must detect cuts and flush the
interpolation history.

### 4.5 Frame Timing Budget

At the 40fps target (the lower bound of the VRR window, representing the
worst-case sustainable framerate), the frame budget is 25ms:

- NPU inference (scene classification + depth maps + motion vectors): ~10-12ms
- iGPU compute (FSR2 upscaling + frame interpolation): ~6-8ms
- Compositor (composite + scanout): ~2-3ms
- Headroom: ~2-4ms

The three compute units (dGPU for rendering, NPU for inference, iGPU for compute
shaders) operate in a pipeline: the dGPU renders a frame, the NPU extracts depth
and motion information from it, and the iGPU runs FSR2 and interpolation using
that information. The iGPU has direct access to system RAM, so NPU outputs (depth
maps, motion vectors) are available zero-copy. The only PCIe transfer in the hot
path is dGPU-rendered frame to system RAM.

## 5. IPC Patterns

### 5.1 Topic-Based IPC Bus

loisto's IPC bus (loisto-ipclib, sys/core domain, Zig with Rust bindings)
provides typed messaging between any two nodes in the 12-node graph. The bus uses
a topic-based publish/subscribe model over shared memory ring buffers. There is
no broker in the IPC path; the ring buffers are memory-mapped directly between
publisher and subscriber.

Each topic has a defined message type from the ABI layer (loisto-abi-bus defines
the envelope format, topic IDs, and ring buffer layout). Components subscribe to
topics they consume and publish to topics they produce.

Key scheduling-relevant topics include:

- `TOPIC_FRAME_TIMING` (published by Pacer): deadline and refresh rate for the
  current frame. Consumed by the compositor and frame pipeline stages.
- `TOPIC_POWER_HINT` (published by Pacer): CPU/GPU power budgets. Consumed by
  schedctl for CPPC translation.
- `TOPIC_SCENE_STATE` (published by Analyser): scene classification state.
  Consumed by the Pacer for quality decisions.
- `TOPIC_THERMAL` (published by pwmon): thermal headroom data. Consumed by the
  Pacer for power budget calculation.

### 5.2 brokerd

The Data Broker (loisto-brokerd, sys/query domain, Zig) is the only component
with direct database access. It owns both DuckDB (for structured persistent data:
content catalog, metadata, labels, trophies) and LMDB (for asset index: resolved
paths, texture cache keys). Everything else queries through IPC messages. The
broker manages content in `/data/res/` zones, indexes extension state from
`/data/ext/` directories, and handles content change notifications.

For the scheduler, the broker's role is providing cached WorkloadDescriptors that
schedctl queries when an extension launches. The descriptor is computed from
metadata and manifest data and tells the scheduler what resource envelope the
workload needs.

### 5.3 hostd

The Runner Host (loisto-hostd, sys/transient domain, Zig) loads and monitors
extension execution within sandboxed environments. It implements a pluggable
runner engine architecture with three engine types: libretro (dlopen core, game
loop, callbacks), process (exec binary, PID monitor), and media (dlopen libmpv,
frame render). Extensions are sysext SquashFS images with manifests.

hostd reports frame timing to the Pacer for power/scheduling decisions via IPC
topic `runner.frame`. This is how the Pacer knows the actual frame times of
running content, which feeds into the dynamic scheduling layer and VRR target
adjustment.

## 6. Zig-Specific Insights

### 6.1 Comptime

Zig's comptime (compile-time evaluation) is used throughout loisto's system
crates for configuration that would otherwise require runtime dispatch or proc
macros. Where Rust uses proc macros to generate code from annotations, Zig
evaluates arbitrary code at compile time, including type construction and
validation.

For the ABI layer, Zig consumes C headers via `@cImport(@cInclude(...))`, which
is itself a comptime operation. The result is zero-overhead type definitions that
match the C ABI exactly, with no runtime FFI cost and no generated binding code.

For the build system, loisto's three-dimensional build matrix (targets x profiles
x domains) produces per-crate compilation flags at build time. Zig crates receive
their optimization mode (ReleaseFast, ReleaseSafe, ReleaseSmall, Debug) and
target-specific flags as build parameters, and comptime evaluation propagates
these into type-safe configuration without runtime checks.

### 6.2 Allocator Patterns

Zig's explicit allocator model means every allocation site declares which
allocator it uses. This is directly relevant to loisto's memory governance:
different domains have different memory budgets enforced by cgroups, and memory
binning (categorized service budgets) is a first-class system concern.

For the scheduler specifically, the BPF map updates from schedctl are structured
as fixed-size writes to memory-mapped regions. There is no dynamic allocation in
the hot path. The workload descriptor lookups from the broker are amortized (done
once per launch, cached locally). The Pacer's per-frame decisions operate on
fixed-size state with no allocation.

For frame pipeline crates, huge pages are used aggressively: 2MB `.text`
alignment for code and `MADV_HUGEPAGE` on heap for frame domain crates. This
is more aggressive than general distributions, but the stripped-down system with
known process count means minimal memory fragmentation and abundant contiguous
2MB blocks for THP.

## 7. Relevance to hilavitkutin

hilavitkutin is a userspace library, not an operating system. It does not have
access to kernel scheduling, BPF maps, CPPC registers, or hardware power
management. The relevance of loisto's scheduler design to hilavitkutin is
therefore in POLICY, not in mechanism.

### 7.1 Transferable Policy Design

The two-layer scheduling model (static workload descriptors + dynamic per-frame
adaptation) is directly transferable. hilavitkutin's WorkUnit concept maps to
loisto's workload assignment: both describe a unit of work with known resource
characteristics that a scheduler assigns to execution resources.

In loisto, the static layer is a WorkloadDescriptor cached in DuckDB; in
hilavitkutin, the static layer is the IR that domain transforms produce. In
loisto, the dynamic layer is the Pacer reading frame deadlines and thermal state;
in hilavitkutin, the dynamic layer is the scheduler reading backpressure signals
and throughput metrics from the pipeline.

### 7.2 Synthetic Core Differentiation

loisto's synthetic E-cores concept (treating identical hardware resources as
differentiated pools via software policy) transfers to hilavitkutin's thread pool
management. A userspace scheduler can designate some threads as "economy"
(background, non-latency-sensitive work) and others as "performance"
(pipeline-critical, latency-sensitive work) without any kernel cooperation. The
mechanism changes (thread priorities and affinity hints instead of CPPC caps),
but the policy pattern is identical.

### 7.3 Scene State Machine as Workload Phase Detection

The Pacer's five-state scene state machine is a specialized form of workload
phase detection. hilavitkutin's pipeline scheduler faces the same problem: the
character of work changes over time (initial load, steady-state processing,
wind-down), and the scheduler should adapt its behavior accordingly. The specific
states differ (hilavitkutin does not have "gameplay" or "cutscene"), but the
pattern of detecting phase transitions and adjusting scheduling parameters is
directly applicable.

### 7.4 Power-Budget-as-Scheduling-Constraint

loisto's constant-power governance model, where the Pacer distributes a fixed
power budget across compute units, is analogous to hilavitkutin's resource
budget model. In a userspace library, the "power budget" is thread pool capacity
and memory allocation budget. The Pacer's approach of governing on outcome
(frame deadline met) rather than utilization (CPU percentage) maps to
hilavitkutin governing on pipeline throughput (work units completed per time
window) rather than thread utilization.

### 7.5 BPF Map Structure as Shared State Model

The BPF map model (a structured shared data region written by a policy agent and
read by an enforcement agent) is the same architecture hilavitkutin needs between
its scheduler and its worker threads. In loisto, schedctl writes to BPF maps and
the kernel BPF program reads them. In hilavitkutin, the scheduler writes to a
work assignment structure and worker threads read from it. The mechanism is
different (BPF maps vs. atomic shared memory), but the data flow pattern and the
separation between policy (what should happen) and enforcement (making it happen)
are the same.

### 7.6 VRR-Aware Deadline Targeting

The Pacer's approach to VRR, where it targets the best sustainable framerate
within a window rather than a fixed target, maps to adaptive deadline targeting
in hilavitkutin. A pipeline scheduler can accept a range of acceptable
throughputs and target the best sustainable rate within that range, adjusting
down when the pipeline is overloaded and up when headroom is available. This
avoids the oscillation that comes from targeting a single fixed throughput value.

### 7.7 What Does NOT Transfer

Several loisto scheduler concepts are kernel-specific and do not transfer:

- BPF program authoring and verification constraints
- CPPC hardware register manipulation
- sched_ext hooks and their timing semantics
- CachyOS kernel configuration and BORE interaction
- Governor replacement (userspace libraries have no governors)
- Thermal sensor reading and fan curve interaction
- cgroup memory binning enforcement

The lesson is: study loisto for scheduling POLICY patterns (how to classify work,
how to assign it, how to adapt), not for scheduling MECHANISM (BPF, CPPC, cgroups,
sysfs). hilavitkutin implements the same policy patterns using userspace
mechanisms (thread pools, atomic shared state, priority queues, adaptive rate
control).
