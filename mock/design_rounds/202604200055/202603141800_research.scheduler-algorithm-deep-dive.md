## 1. The Scheduling Problem Statement

hilavitkutin must schedule work units from a DAG across N worker threads. The
challenge is not one workload; it is three workloads overlaid on the same
infrastructure.

**Data-parallel morsel processing.** polka's pipeline transforms columns of
IR data through fused operator chains. Each morsel is 4K-64K elements
depending on cache geometry. Processing one morsel takes microseconds to low
milliseconds. The scheduler must dispatch morsels to workers fast enough
that no worker idles between morsels. The dispatch overhead budget is under
1 microsecond per morsel assignment; anything higher shows up in profiling as
scheduler-bound rather than compute-bound.

**I/O-bound work that yields and resumes.** saalis connectors issue HTTP
requests to external APIs and return `Progress::Yield(Pending)`. The
request takes 50-500 ms of wall-clock time. During that window, the worker
thread must not block. It must save the work unit's state, steal other
ready work, and resume the connector when the response arrives. A pass with
10 connectors all waiting on HTTP should keep all CPU workers saturated
with morsel work, not parked.

**Mixed workloads.** A single pass may contain morsel stages (CPU-bound
column transforms), scalar stages (singleton resource reads, config
validation), and yielding stages (HTTP connectors, file watches). The
scheduler cannot treat all work units identically. Morsel units split
into many small execute() calls. Scalar units run once. Yielding units
run, park, wake, run again. The scheduler's dispatch loop must handle all
three without special-casing any of them.

**Heterogeneous cores.** Apple Silicon has performance cores (P-cores) and
efficiency cores (E-cores) with different cache sizes, clock speeds, and
memory bandwidth. An M4 Pro has 10P + 4E cores. The P-cores have 192KB
L1 and 12MB shared L2. The E-cores have 128KB L1 and 4MB shared L2.
Morsel sizes should differ: P-cores can profitably process larger morsels
(more data fits in cache), while E-cores should receive smaller morsels
to avoid L2 thrashing.

**Dynamic cost estimation.** Before any pass has run, the scheduler has
no measured data. It uses a heuristic: cost = 1 + writes + emits. After
passes execute, measured durations replace the heuristic via exponential
moving average. When cost estimates change significantly, the plan must
be recomputed: upward ranks shift, lane assignments change, the critical
path may move to a different branch of the DAG.

The algorithm described in the following sections must handle all five
concerns simultaneously, in a single unified dispatch loop, without
requiring the work unit author to know about any of them.

## 2. Plan Computation (Static Phase)

Plan computation runs at startup and whenever a recomputation trigger fires
(topology change, cost estimate divergence, explicit `RecomputePlan`
signal). The output is an `ExecutionPlan`: an ordered sequence of steps,
each containing a set of work unit node IDs that can execute in parallel.

### Step 1: Build DAG from Read/Write declarations

Every work unit declares its access set via associated types:

```
type Read = (Read<Config>, Read<Catalogue>);
type Write = (Write<IrColumn>);
```

The scheduler extracts these declarations at registration time and builds
edges: for every pair (A, B) where A writes a resource R and B reads R,
create an edge A -> B. For every pair (A, B) where both write R, create
an edge between them (ordering is determined by registration order or
explicit priority). Virtual columns (zero-sized types like
`Write<SchemaChanged>`) create DAG edges with no storage backing.

The DAG is stored as `Dag<NodeId, EdgeData>` with dual adjacency maps
(forward for successors, reverse for predecessors). The edge data carries
the resource ID that created the edge, used for version stamp early-release
analysis.

### Step 2: Kahn's topological sort + cycle detection

Kahn's algorithm processes nodes by repeatedly removing zero-in-degree
nodes. This produces level-based ordering: all nodes at depth 0 form the
first batch, all nodes at depth 1 form the second batch, and so on.

```
procedure kahn_toposort(dag) -> (order: Vec<NodeId>, depth: Map<NodeId, u16>):
    in_degree = count incoming edges for each node
    queue = all nodes where in_degree == 0
    depth = {node: 0 for node in queue}
    order = []

    while queue is not empty:
        u = queue.pop()
        order.push(u)
        for each successor v of u:
            in_degree[v] -= 1
            depth[v] = max(depth[v], depth[u] + 1)
            if in_degree[v] == 0:
                queue.push(v)

    if order.len() != dag.node_count():
        error("cycle detected: nodes not visited = cycle members")

    return (order, depth)
```

Complexity: O(V + E). For typical polka pipelines (20-50 nodes, 30-100
edges), this completes in under 10 microseconds.

DFS-based toposort also produces valid ordering but does not group nodes
by level. It interleaves deep and shallow paths, making step grouping a
separate pass. Kahn's gives us level grouping for free.

### Step 3: Compute upward rank

Upward rank identifies the critical path. The formula:

```
rank(u) = cost(u) + max(rank(successors(u)))
```

Leaf nodes (no successors) have rank equal to their cost. Each node's rank
is its own cost plus the maximum rank among its successors. The node with
the highest rank in the entire graph is the root of the critical path.

Computation is bottom-up: process nodes in reverse topological order
(leaves first). Each node's rank is computed once.

```
procedure compute_upward_rank(order, dag, costs) -> Map<NodeId, u32>:
    rank = {}
    for u in order.reversed():
        max_successor_rank = 0
        for v in dag.successors(u):
            max_successor_rank = max(max_successor_rank, rank[v])
        rank[u] = costs[u] + max_successor_rank
    return rank
```

Initial cost estimation before runtime data:

| Component | Cost |
|-----------|------|
| Base | 1 |
| Per declared write | +1 |
| Per emitted virtual column | +1 |
| Morsel dispatch hint | +2 (morsel units tend to be heavier) |

Worked example with 6 nodes:

```
DAG:  A -> C -> E -> F
      A -> B -> D -> F

Costs: A=3, B=2, C=4, D=1, E=2, F=3

rank(F) = 3 + 0 = 3                    (leaf)
rank(E) = 2 + rank(F) = 5
rank(D) = 1 + rank(F) = 4
rank(C) = 4 + rank(E) = 9
rank(B) = 2 + rank(D) = 6
rank(A) = 3 + max(rank(C), rank(B)) = 3 + 9 = 12

Critical path: A -> C -> E -> F (total rank 12)
```

### Step 4: Group nodes into parallel steps

Nodes at the same topological depth with no mutual edges form a step.
Within a step, all nodes can execute in parallel.

```
procedure group_into_steps(order, depth) -> Vec<Step>:
    max_depth = depth.values().max()
    steps = vec![Step::empty(); max_depth + 1]
    for u in order:
        steps[depth[u]].nodes.push(u)
    return steps
```

### Step 5: Lane assignment

List scheduling assigns nodes to lanes (worker threads). The highest-rank
ready node goes to the lane that finishes earliest:

```
procedure assign_lanes(steps, ranks, costs, num_lanes) -> Map<NodeId, u3>:
    // Sort all nodes by rank descending across all steps
    all_nodes = steps.flat_map(|s| s.nodes).sort_by(|a, b| ranks[b].cmp(ranks[a]))
    lane_finish = [0u64; num_lanes]
    assignment = {}

    for u in all_nodes:
        best_lane = 0
        best_finish = u64::MAX

        for lane in 0..num_lanes:
            // Check AccessSet conflict with already-assigned nodes on this lane
            if no_conflict(u, lane, assignment):
                finish = lane_finish[lane] + costs[u]
                if finish < best_finish:
                    best_finish = finish
                    best_lane = lane

        assignment[u] = best_lane
        lane_finish[best_lane] = best_finish

    return assignment
```

### Step 6: Precompute overflow lists with lane_compat bitmask

For each node, compute which lanes can safely execute it (no AccessSet
conflicts). This bitmask is checked at runtime in O(1) during steal
operations.

```
procedure compute_lane_compat(nodes, assignments) -> Map<NodeId, u8>:
    compat = {}
    for u in nodes:
        mask = 0u8
        for lane in 0..num_lanes:
            if no_conflict(u, lane, assignments):
                mask |= 1 << lane
        compat[u] = mask
    return compat
```

### When does lane assignment pay off?

Lane assignment adds O(V * L) cost to plan computation, where V is node
count and L is lane count. For typical DAGs (V < 100, L <= 8), this is
sub-millisecond. The benefit is deterministic load balancing: the critical
path is always assigned to the lane that can complete it earliest.

Simpler alternatives (just steps 1-4, round-robin dispatch within steps)
work for regular DAGs where all nodes in a step have similar cost. They
break down when costs are heterogeneous: a step with one 10ms node and
nine 1ms nodes wastes 9ms of worker time at the step barrier if the heavy
node is not placed on the right lane.

For polka's pipeline, costs are heterogeneous. A parse stage (heavy) and a
config validation stage (light) may share a step. Lane assignment ensures
the parse stage starts on the lane that finishes its previous work earliest,
minimizing step barrier wait time.

For saalis, the benefit is even larger. I/O-bound connector stages have
wildly variable costs (50ms to 500ms depending on external API latency).
Lane assignment with EMA-updated costs adapts to actual latencies and
avoids assigning all slow connectors to the same lane.

loimu's workload (dozens to hundreds of behaviors with similar costs) gets
moderate benefit. The critical path is typically short and well-balanced.

The complexity is justified whenever cost heterogeneity exceeds 2x between
nodes in the same step. Below 2x, round-robin is nearly as good.

## 3. Runtime Execution (Dynamic Phase)

The execution loop runs one pass through the plan's steps. Each step
dispatches its ready nodes to workers, waits for completion (or yield),
and advances.

### The core execution loop

```
procedure execute_pass(plan, store, workers):
    for step in plan.steps:
        // Determine which nodes are ready (deps satisfied, not yielded-pending)
        ready = []
        for node in step.nodes:
            if node.state == Done:
                continue  // completed in a previous pass (incremental)
            if node.state == YieldedPending:
                continue  // parked, will be handled by wake
            if all_deps_satisfied(node):
                ready.push(node)

        dispatch_to_lanes(ready, workers)

        // Wait for all nodes in this step to reach a terminal state
        while any_running_or_ready(step):
            for each completed event from workers:
                match event.result:
                    Progress::Complete =>
                        node.state = Done
                        decrement_downstream_dep_counters(node)
                        check_version_stamps(node, store)
                        // Version stamp check: if written resources
                        // have unchanged versions (phantom write),
                        // mark downstream as early-releasable
                        if all_writes_unchanged(node, store):
                            early_release_downstream(node)

                    Progress::Yield(BudgetExhausted) =>
                        // Re-queue for next iteration within this step
                        node.state = Ready
                        ready.push(node)

                    Progress::Yield(Pending) =>
                        // Park the node; worker steals other work
                        node.state = YieldedPending
                        park(node)
                        // Worker becomes available for stealing

                    Progress::Yield(Retry { attempt, max, backoff }) =>
                        if attempt >= max:
                            node.state = Failed
                            skip_downstream(node)
                        else:
                            node.retry_timer = now() + backoff + jitter()
                            node.state = RetryWaiting

                    Progress::Failed(error) =>
                        node.state = Failed
                        rollback_resources(node, store)
                        skip_downstream(node)
                        emit_diagnostic(error)

            // Check parked nodes for wakeup
            for each parked node in park_list:
                if node.wake_flag.load(Relaxed):
                    node.state = Ready
                    ready.push(node)
                    unpark(node)

            // Check retry timers
            for each retry node in retry_list:
                if now() >= node.retry_timer:
                    node.state = Ready
                    ready.push(node)

            // Idle workers steal from overflow
            for each idle worker:
                stolen = steal_from_overflow(worker.lane, plan)
                if stolen.is_some():
                    dispatch(stolen, worker)

        // Step barrier: all nodes done, yielded-pending, or failed
        barrier()

    // Pass complete
```

### The step barrier problem

A step cannot complete if a Pending work unit is parked. The parked unit
has not written its output. Downstream units in future steps may depend on
that output.

Three options, each with distinct trade-offs:

**Option A: Advance if downstream is in future steps.** If the Pending
unit's downstream dependencies all appear in steps strictly after the
next step, advance to the next step. The Pending unit stays parked. When
it wakes, it writes its output and triggers downstream re-evaluation. This
maximizes parallelism but complicates the invariant: a step's nodes are no
longer guaranteed to have all inputs available.

**Option B: Never advance past a Pending unit.** The step barrier waits
until all nodes are either Done, Failed, or YieldedPending. If a node is
YieldedPending, its downstream in future steps simply won't run this pass.
The next pass picks up where this one left off. Simple, deterministic, but
may underutilize the pipeline when Pending waits are long.

**Option C: Separate Pending units into their own virtual step.** At plan
time, if a node might yield Pending (indicated by a type-level marker or
registration hint), place it in its own step with only other potentially-
Pending nodes. Non-Pending downstream gets its own step. The barrier after
the Pending step advances when all non-Pending work at that depth is done.
Pending nodes resolve across passes.

hilavitkutin uses Option B for the initial design. Reasons:

1. Deterministic: the output of a pass is fully determined by the input
   state. No partial results from in-flight Pending units.
2. Simple: no virtual step splitting, no cross-step dependency patching.
3. The yield gap is filled by work-stealing, not by step advancement.
   Workers whose units are Pending steal overflow work from the current
   step and execute it. The step is not idle; workers are doing useful
   work even though one unit is parked.
4. Option A can be added later as an optimization if profiling shows step
   barriers are the bottleneck. The plan/execute separation makes this
   change local to the executor.

### dispatch_to_lanes detail

```
procedure dispatch_to_lanes(ready, workers):
    // Sort ready nodes by rank descending
    ready.sort_by(|a, b| rank[b].cmp(rank[a]))

    for node in ready:
        target_lane = node.assigned_lane
        target_worker = workers[target_lane]

        if target_worker.is_idle():
            target_worker.dispatch(node)
        else:
            // Primary lane busy; try compatible lanes
            for lane in 0..num_lanes:
                if node.lane_compat & (1 << lane) != 0:
                    if workers[lane].is_idle():
                        workers[lane].dispatch(node)
                        break
            else:
                // All compatible lanes busy; push to overflow
                overflow_list.push(node)
```

## 4. Work-Stealing with Yield Integration

When a worker's current unit yields, the worker does not park. It saves
the unit's state and immediately looks for other work.

### Yield handling sequence

```
procedure on_yield(worker, yielded_unit, yield_reason):
    // Step 1: Save state (already in &mut self on the work unit)
    // The work unit's struct fields hold all intermediate state.
    // No explicit save needed; the scheduler just stops calling execute().

    match yield_reason:
        BudgetExhausted =>
            // Re-queue in the current step's ready list
            yielded_unit.state = Ready
            step.ready.push(yielded_unit)

        Pending =>
            // Park with wake slot
            yielded_unit.state = YieldedPending
            park_list.push(yielded_unit)

        Retry { attempt, max, backoff } =>
            yielded_unit.retry_timer = now() + backoff + jitter()
            yielded_unit.state = RetryWaiting
            retry_list.push(yielded_unit)

    // Step 2: Worker looks for ready overflow work
    stolen = steal_work(worker.lane)

    if stolen.is_some():
        // Step 3: Execute stolen work (filling the yield gap)
        worker.dispatch(stolen)
        return

    // Step 4: Check park list for woken units
    for parked in park_list:
        if parked.wake_flag.load(Relaxed):
            if parked.lane_compat & (1 << worker.lane) != 0:
                parked.state = Ready
                unpark(parked)
                worker.dispatch(parked)
                return

    // Step 5: No work available; spin briefly then sleep
    spin_wait(SPIN_ITERS)  // ~100 iterations, ~200ns
    if still_no_work():
        worker.sleep()  // OS-level park, woken by dispatch or wake_flag
```

### Stealing order

The stealing order within the overflow list uses two criteria:

1. **lane_compat filter:** only consider nodes whose `lane_compat` bitmask
   includes the stealing worker's lane. This is an O(1) bitwise AND check.

2. **upstream_rank priority:** among compatible candidates, steal the one
   with the highest upward rank. This greedy heuristic ensures that stolen
   work unblocks the longest remaining critical path.

```
procedure steal_work(lane) -> Option<NodeId>:
    best = None
    best_rank = 0

    for node in overflow_list:
        if node.lane_compat & (1 << lane) == 0:
            continue
        if not all_deps_satisfied(node):
            continue
        if rank[node] > best_rank:
            best = Some(node)
            best_rank = rank[node]

    if best.is_some():
        overflow_list.remove(best)

    return best
```

For small overflow lists (typical: 0-20 entries), linear scan is faster
than maintaining a sorted structure. The scan touches one cache line per
~8 entries (NodeId + lane_compat + rank fit in 16 bytes). For lists
exceeding 64 entries, a priority queue keyed by rank with lane_compat
pre-filtering would be more efficient, but DAGs large enough to produce
64+ overflow entries at once are rare in polka and saalis workloads.

### Cilk comparison

Cilk's stealing model is simpler: when a worker's deque is empty, it picks
a random victim and steals from the top of the victim's deque. This is
optimal for fork-join parallelism (recursive decomposition) where all
tasks are independent and approximately equal cost.

hilavitkutin's DAG workload differs in three ways:

| Property | Cilk | hilavitkutin |
|----------|------|--------------|
| Trigger | Deque empty | Unit yielded or blocked on dep |
| Selection | Random victim, FIFO steal | Overflow list, highest-rank steal |
| Constraint | None (all tasks independent) | lane_compat (AccessSet conflicts) |
| Optimizes for | Throughput on recursive decomposition | Critical path latency on DAGs |

hilavitkutin uses crossbeam-deque for the per-lane local queue
(push/pop is LIFO for locality, steal is FIFO for granularity), but the
cross-lane stealing policy is rank-ordered, not random. This is the key
synthesis: Cilk's data structure (lock-free deque) with loimu's policy
(rank-weighted priority stealing).

## 5. Morsel Dispatch Within Steps

Morsel dispatch handles data-parallel work units that process column data.
A morsel is a contiguous slice of a column, sized to fit in the worker's
cache.

### Morsel sizing

The morsel size formula:

```
morsel_elements = usable_cache / (num_columns * element_size)
```

Where `usable_cache` is the L2 cache available to the worker minus
overhead for hash tables, state, and stack. A conservative estimate uses
75% of L2.

Concrete numbers for Apple M4 Pro:

| Core type | L2 cache | Usable (75%) | Columns=4, elem=8B | Morsel elements |
|-----------|----------|--------------|---------------------|-----------------|
| P-core | 12 MB | 9 MB | 32 bytes/row | 294,912 |
| E-core | 4 MB | 3 MB | 32 bytes/row | 98,304 |

These are upper bounds. In practice, the working set includes hash tables
for joins, accumulator state for aggregations, and operator state. A more
realistic allocation:

| Purpose | P-core budget | E-core budget |
|---------|---------------|---------------|
| Input columns | 4 MB | 1.5 MB |
| Hash table / accum | 3 MB | 1 MB |
| Output columns | 1.5 MB | 0.3 MB |
| Stack + overhead | 0.5 MB | 0.2 MB |
| **Total** | **9 MB** | **3 MB** |

With 4 MB for input columns and 4 columns at 8 bytes each (32 bytes/row),
a P-core morsel is ~131,072 elements. With 1.5 MB for input columns, an
E-core morsel is ~49,152 elements.

For polka's typical pipeline (config data, not analytics), column sizes
are small (hundreds to low thousands of entries). The entire column often
fits in a single morsel. Morsel dispatch still applies; the morsel is just
the entire column. No splitting needed.

For saalis (query results from connectors), columns can reach tens of
thousands of entries (game library catalogs, metadata tables). Here morsel
splitting provides real parallelism.

### Morsel dispatch mechanism

```
procedure dispatch_morsel_unit(unit, column_len, workers, core_types):
    // Determine morsel size per core type
    morsel_sizes = {}
    for core_type in [P, E]:
        morsel_sizes[core_type] = compute_morsel_size(core_type, unit.num_columns, unit.elem_size)

    // Split the column into morsel ranges
    morsels = []
    offset = 0
    while offset < column_len:
        // Round-robin across workers; each worker's morsel size
        // depends on its core type
        for worker in workers:
            if offset >= column_len:
                break
            size = morsel_sizes[core_types[worker.lane]]
            end = min(offset + size, column_len)
            morsels.push(MorselRange { start: offset, end, worker: worker.id })
            offset = end

    // Push morsels to worker-local deques
    for morsel in morsels:
        workers[morsel.worker].local_deque.push(morsel)

    // Workers process morsels via work-stealing
    // Each morsel is a separate execute() call with a range parameter
    // All morsels share the same step
    // The step barrier waits for all morsels to complete
```

### Morsel dispatch is intra-step

All morsels of a single work unit belong to the same step. The step
barrier waits for every morsel to complete before advancing. This means
morsel parallelism is within a step, not across steps. A morsel work
unit's step might have 100+ individual execute() calls (one per morsel),
all running in parallel across workers.

Workers that finish their assigned morsels early steal morsels from other
workers' deques (Cilk-style FIFO steal from the opposite end). This
balances load when some morsels take longer than others (data-dependent
processing time, cache effects from NUMA topology).

### Morsel vs scalar dispatch

Not all work units are morsel-dispatched. The distinction:

| Dispatch type | When | execute() calls per step |
|---------------|------|--------------------------|
| Morsel | Unit reads column data | column_len / morsel_size |
| Scalar | Unit reads only singletons, or is non-chunkable | 1 |

The scheduler determines dispatch type from the work unit's access
declarations. If any `Read<T>` where T is a collection type (column), the
unit is morsel-dispatched. If all reads are singleton resources, the unit
is scalar-dispatched.

The work unit author does not choose dispatch type. It falls out from the
access declarations. A unit that reads `Read<Config>` (singleton) and
writes `Write<IrColumn>` (collection) is scalar-dispatched: it reads
config once and produces the entire output column in one call. A unit
that reads `Read<IrColumn>` (collection) and writes `Write<OutputColumn>`
(collection) is morsel-dispatched: it processes input morsels and produces
output morsels.

## 6. Heterogeneous Core Handling

P-cores and E-cores have different performance profiles. The scheduler
adapts to this asymmetry.

### Apple Silicon cache geometry

| Core | L1d | L2 | Clock (M4 Pro) |
|------|-----|-----|----------------|
| P-core | 192 KB | 12 MB shared/cluster | 4.5 GHz |
| E-core | 128 KB | 4 MB shared/cluster | 3.3 GHz |

The L2 difference is the most impactful for morsel sizing. P-cores can
hold 3x more working data in L2. Clock speed compounds the difference:
a P-core processes each element ~36% faster. Combined, a P-core processes
~4x the data per unit time compared to an E-core.

### Lane-to-core mapping

The scheduler assigns lanes to core types at startup:

```
procedure map_lanes_to_cores(num_p_cores, num_e_cores) -> Vec<CoreType>:
    lanes = []
    // P-core lanes first (they get the critical path)
    for _ in 0..num_p_cores:
        lanes.push(CoreType::P)
    // E-core lanes after (they get overflow and non-critical work)
    for _ in 0..num_e_cores:
        lanes.push(CoreType::E)
    return lanes
```

Lane assignment (Step 5 of plan computation) naturally places critical-path
nodes on P-core lanes because those lanes have the earliest finish times
(faster clock, larger cache). Overflow work stolen by E-core workers is
typically non-critical-path work, which is the correct assignment.

### Per-lane morsel size

Each lane has a morsel size configuration derived from its core type's
cache geometry:

```
struct LaneConfig {
    core_type: CoreType,
    morsel_elements: usize,  // computed from cache geometry
    l2_budget: usize,        // usable L2 bytes
}
```

When dispatching morsels, the scheduler uses the target worker's lane
config to determine morsel size. A P-core worker gets a 131K-element
morsel; an E-core worker gets a 49K-element morsel. Work-stealing across
core types re-sizes the morsel: if an E-core steals a morsel originally
sized for a P-core, it processes a smaller slice and pushes the remainder
back onto the source deque.

### loimu, loisto, hilavitkutin: three levels of heterogeneity

loimu does not handle heterogeneous cores. Its target workload (GUI
behaviors) does not morsel-chunk. All lanes are equal.

loisto handles it at the kernel level via sched_ext BPF and CPPC hints.
The BPF scheduler pins latency-sensitive work to P-cores and background
work to synthetic E-cores (software-capped Zen 3 cores). This is below
userspace; applications are unaware.

hilavitkutin handles it at the userspace level. The scheduler detects core
types at startup (via `sysctl` on macOS, `/sys/devices/system/cpu/` on
Linux), configures per-lane morsel sizes, and assigns lanes to core types.
No kernel cooperation needed. The cost is slight suboptimality compared to
kernel-level scheduling (the OS may migrate a thread to a different core
type between morsel dispatches), but thread affinity hints
(`pthread_setaffinity_np` on Linux, `thread_policy_set` on macOS) mitigate
this.

## 7. Adaptive Cost Estimation

Cost estimates drive the plan: upward rank, lane assignment, and steal
priority all depend on per-node cost values. Bad estimates produce bad
plans.

### Initial heuristic

Before any pass has executed:

```
cost(u) = 1 + count(writes(u)) + count(emits(u))
```

This heuristic is deliberately simple. It correlates with actual cost
because write operations involve memory mutations (cache line
invalidation, store buffer pressure) and emit operations involve signal
dispatch. But it has no knowledge of the actual computation inside
execute().

### EMA convergence

After each pass, the scheduler measures the wall-clock duration of each
work unit's execute() call (or the sum of its morsel durations for morsel
units). The measured duration replaces the heuristic via exponential
moving average:

```
new_estimate = (7 * old_estimate + measured) / 8
```

This is alpha = 1/8, chosen for stability: it takes ~10 samples for the
EMA to converge to within 10% of the true mean for steady-state
workloads. The integer arithmetic (multiply by 7, add measured, shift
right by 3) avoids floating-point, which matters for no_std targets.

After 10 passes with stable workloads, the EMA has effectively replaced
the heuristic. The initial heuristic value washes out exponentially: after
N passes, the heuristic's contribution is (7/8)^N of the estimate. After
10 passes, that is 0.26 (26%); after 20 passes, 0.07 (7%).

### Recomputation threshold

The plan must be recomputed when cost estimates change enough to alter
the critical path or lane assignment. The threshold:

```
if abs(ema[u] - rank_assumed_cost[u]) > 2 * rank_assumed_cost[u]:
    trigger_recomputation()
```

A 2x divergence means the node takes at least twice as long (or half as
long) as the plan assumed. This threshold balances stability (no
thrashing on small fluctuations) with responsiveness (catching genuine
workload shifts).

Recomputation itself is fast (O(V + E) toposort, O(V * L) lane
assignment). For a 50-node, 8-lane graph, the entire plan computation
takes under 100 microseconds. Recomputing every 10-20 passes is
negligible overhead.

### Cold start strategy

The first pass uses heuristic costs and may produce a suboptimal plan.
This is acceptable because:

1. The first pass is typically loading and initialization, not
   steady-state processing.
2. Suboptimality from bad estimates is bounded: the plan is still
   correct (dependencies are respected), just not optimally balanced.
3. By pass 3-5, the EMA has enough signal to produce reasonable
   estimates. By pass 10, estimates are stable.

For saalis, where passes may run minutes apart (Curator: 1h cadence),
EMA convergence is slow in wall-clock time but fast in pass count. The
cost estimates from the first 10 passes persist across scheduler sleeps
and are not reset.

## 8. The Step vs Wavefront Decision

Steps (barriers between levels) and wavefront (atomic counters, no
barriers) represent two execution strategies with different trade-offs.

### Step-based execution

Each step is a synchronization point. All workers complete all nodes in
step N before any worker begins step N+1. The barrier is a simple atomic
counter: each worker decrements when it finishes its last node in the
step; the last worker to decrement signals the barrier.

Properties:
- **Deterministic order.** Given the same input, the same nodes execute
  in the same relative order across passes. This aids debugging: a bug
  that manifests in pass 47 reproduces in pass 47.
- **Bounded concurrency.** Within a step, the set of concurrently
  executing nodes is known at plan time. This simplifies reasoning about
  resource conflicts.
- **Under-utilization.** If one node in a step takes 10ms and all others
  take 1ms, 9ms of worker time is wasted at the barrier. Taskflow
  benchmarks measure this at 10-15% for typical DAGs.

### Wavefront execution

Each node has an atomic dependency counter initialized to its in-degree.
When a predecessor completes, it decrements all successors' counters. A
node whose counter reaches zero is immediately dispatchable, regardless
of what step it is in.

```
procedure on_complete(node):
    for successor in dag.successors(node):
        if successor.dep_counter.fetch_sub(1, AcqRel) == 1:
            // Last dependency satisfied; successor is ready
            dispatch(successor)
```

Properties:
- **Maximum parallelism.** No artificial barriers. A node runs as soon
  as its inputs are ready.
- **Non-deterministic order.** The execution order depends on thread
  scheduling, cache timing, and contention. Two runs with the same
  input may produce different execution traces.
- **Complex profiling.** Without step boundaries, attributing latency
  to specific pipeline stages is harder.

### hilavitkutin's choice: steps

For the initial design, steps. The reasons:

1. **Yielding fills the gap.** The primary source of step barrier waste
   is workers idling while one slow node completes. Yielding and
   work-stealing already address this: a worker whose node is slow
   (or has yielded) steals overflow work from the current step. Workers
   do not idle at barriers; they steal.

2. **Version stamp early-release.** Version stamps allow the scheduler
   to release downstream dependencies before the step barrier if a
   writer produced no change (phantom write). This recovers some of
   the parallelism that wavefront provides, without giving up
   determinism.

3. **Debugging.** polka is a build tool. Users will encounter errors
   in their templates and configurations. Deterministic execution order
   makes error messages reproducible and stack traces meaningful.
   Wavefront execution with non-deterministic interleaving produces
   heisenbugs.

4. **Upgrade path.** The three-phase separation (Graph/Plan/Execute)
   means the executor is a replaceable component. A wavefront executor
   reads the same `ExecutionPlan` as the step executor. Switching is
   a single type parameter change, not an architectural redesign.

Under-utilization from step barriers for polka's typical DAGs (5-10
steps, 2-8 nodes per step) is estimated at 5-10%, lower than the
Taskflow benchmark average because polka's DAGs are shallow and wide
rather than deep and narrow.

## 9. Scheduler State Machine

The scheduler itself transitions through four states:

```
         trigger
Idle ──────────> Computing ──────> Executing ──────> Draining ──> Idle
  ^                                    |                 |
  |                                    |    (all yields  |
  +────────────────────────────────────+     resolved)   |
         (no yields pending)                             |
         fast path: skip Draining                        |
                                                         |
  Idle <─────────────────────────────────────────────────+
```

### Idle

No pass is running. The scheduler is waiting for a trigger: a cadence
timer, an explicit `RunPass` call, or a topology change that requires
re-evaluation.

In this state, the scheduler's worker threads are sleeping (OS-level
park). No CPU is consumed. The scheduler wakes on trigger via a
condition variable or eventfd.

### Computing

A trigger has fired. The scheduler is recomputing the plan (if needed)
before starting execution. Plan computation is single-threaded (the
scheduler thread) and fast (sub-millisecond for typical DAGs).

If no recomputation is needed (topology unchanged, cost estimates within
threshold), this state is skipped: the scheduler transitions directly
from Idle to Executing.

### Executing

A pass is running. Workers are dispatching nodes from the plan's steps.
This is the steady state. The execution loop from section 3 runs in this
state.

### Draining

The pass has completed all steps, but some work units are still
YieldedPending (parked, waiting on I/O). The scheduler cannot declare the
pass complete until these units resolve.

In the Draining state:
- Workers check wake flags of parked units on each spin cycle.
- When a parked unit wakes, the scheduler dispatches it to a worker.
- The unit runs to completion (or yields again, in which case it
  re-enters Pending state).
- When all Pending units are resolved (Complete or Failed), the
  scheduler transitions to Idle.

The Draining timeout is configurable. If a Pending unit does not wake
within the timeout (default: 30 seconds), it is forcibly failed with
`Progress::Failed(Error::DrainTimeout)`. This prevents a crashed I/O
source from hanging the scheduler indefinitely.

For polka (no yielding), Draining is never entered. The scheduler
transitions directly from Executing to Idle.

## 10. Concrete Data Structures

### ExecutionPlan

```rust
struct ExecutionPlan {
    steps: Vec<Step>,
    node_metadata: Vec<NodeMeta>,   // indexed by NodeId
    lane_count: u8,                 // 1..=8
}

struct Step {
    nodes: Vec<NodeId>,
    morsel_nodes: Vec<NodeId>,      // subset of nodes that are morsel-dispatched
    scalar_nodes: Vec<NodeId>,      // subset that are scalar-dispatched
}

struct NodeMeta {
    lane: u8,                       // assigned lane (0..lane_count)
    lane_compat: u8,                // bitmask of compatible lanes
    depth: u16,                     // topological depth
    upstream_rank: u32,             // upward rank
    dispatch: DispatchType,         // Morsel or Scalar
    estimated_cost: u32,            // current cost estimate (heuristic or EMA)
}

enum DispatchType {
    Scalar,
    Morsel { num_columns: u8, elem_size: u16 },
}
```

### WorkUnitStore

```rust
struct WorkUnitStore {
    units: Vec<Box<dyn WorkUnitErased>>,  // indexed by NodeId
    states: Vec<UnitState>,               // indexed by NodeId
}

enum UnitState {
    Ready,
    Running,
    Done,
    YieldedPending { wake_flag_idx: usize },
    YieldedBudget,
    RetryWaiting { timer: Instant, attempt: u32, max: u32 },
    Failed,
    Skipped,
}
```

Note: `Box<dyn WorkUnitErased>` is used here for illustration. The
actual implementation uses typed storage per work unit registration,
avoiding dynamic dispatch on the hot path. Each registered work unit type
gets its own contiguous storage arena.

### ReadyQueue

```rust
struct ReadyQueue {
    per_lane: [Deque<NodeId>; 8],   // crossbeam-deque per lane
}
```

Each lane has its own deque. The owning worker pushes and pops from the
bottom (LIFO for locality). Stealers take from the top (FIFO for
granularity). This is the standard Cilk deque layout implemented by
crossbeam-deque.

### ParkList

```rust
struct ParkList {
    entries: Vec<ParkEntry>,
    wake_flags: Vec<AtomicBool>,    // indexed by wake_flag_idx
}

struct ParkEntry {
    node_id: NodeId,
    wake_flag_idx: usize,
    parked_at: Instant,
}
```

The wake flags are a contiguous array of atomic bools. Scanning for woken
units is a single cache-line sweep per 64 entries (each AtomicBool is 1
byte, 64 fit in a cache line). For typical workloads (0-20 parked units),
the scan touches 1 cache line.

### VersionCache

```rust
struct VersionCache {
    before: Vec<u64>,       // version stamps captured before each unit executes
    current: Vec<u64>,      // current version stamps (post-execution)
}
```

Indexed by resource ID. Captured before each unit executes. After
execution, if `before[r] == current[r]` for all resources r that the
unit writes, the write was a phantom (no actual change). Downstream
units reading only those resources can skip.

### CostEstimates

```rust
struct CostEstimates {
    heuristic: Vec<u32>,    // initial heuristic cost, indexed by NodeId
    ema: Vec<u32>,          // EMA of measured cost, indexed by NodeId
    sample_count: Vec<u16>, // number of measurements, indexed by NodeId
}
```

While `sample_count < 10`, the estimate blends heuristic and EMA.
After 10 samples, the heuristic is effectively washed out and the EMA
is authoritative.

### OverflowList

```rust
struct OverflowList {
    entries: Vec<OverflowEntry>,
}

struct OverflowEntry {
    node_id: NodeId,
    lane_compat: u8,
    upstream_rank: u32,
}
```

Entries are sorted by upstream_rank descending for fast steal (highest-
rank first). The list is rebuilt at each step boundary. Within a step,
entries are added when a dispatch target lane is busy and removed when
stolen.

## 11. Comparison: hilavitkutin vs loimu vs Rayon vs Tokio

### Scheduling model

**loimu.** Signal-driven wakeup. List scheduling with upward rank and
lane assignment. Step-based execution with step barriers. Version stamps
for phantom write detection and early-release. No yielding (behaviors are
atomic, run to completion). Overflow stealing with lane_compat and
upstream_rank priority. Designed for dozens to hundreds of behaviors
running at UI frame rates (60-120 Hz).

**Rayon.** Fork-join parallelism. No DAG, no explicit dependencies, no
priorities. Recursive task decomposition with work-stealing via Chase-Lev
deques. The `join(a, b)` primitive forks two tasks and steals from either
if the current thread finishes first. No yielding (tasks run to
completion). No step barriers (tasks are independent). Designed for
data-parallel computation (parallel iterators, matrix multiplication,
image processing).

**Tokio.** Async task runtime. No DAG (tasks declare dependencies via
.await, which creates implicit sequential ordering). Work-stealing across
a fixed thread pool. Tasks yield at .await points (compiler-generated
state machines). I/O driver integrated (epoll/kqueue/io_uring). Designed
for I/O-bound server workloads (HTTP servers, database clients, message
brokers) with thousands to millions of concurrent tasks.

**hilavitkutin.** DAG-scheduled pipeline engine. List scheduling with
upward rank and lane assignment (from loimu). Step-based execution with
step barriers (from loimu). Morsel dispatch for data-parallel column
processing (from Leis et al. / HyPer). Yielding with Progress enum
(Pending/BudgetExhausted/Retry, not in loimu). Work-stealing with
lane_compat and upstream_rank priority (from loimu, using crossbeam-deque
from Cilk/Rayon). Version stamps for phantom write detection and
early-release (from loimu). Designed for mixed CPU + I/O pipeline
workloads across three consumers.

### What hilavitkutin takes from each

**From loimu:**
- Plan computation algorithm: Kahn's toposort, upward rank, lane
  assignment, overflow lists with lane_compat bitmask.
- Version stamps: phantom write detection, early-release of downstream
  dependencies.
- Step-based execution model: deterministic ordering, step barriers.
- Per-unit metadata: lane, lane_compat, depth, upstream_rank.
- Fault handling: sole-writer snapshot, shared-writer undo journal.

**From Cilk (via crossbeam-deque):**
- Lock-free deque data structure: LIFO push/pop for locality, FIFO
  steal for granularity.
- Per-worker deque ownership: each worker has its own deque, stealing
  is cross-worker.
- The deque is the physical data structure; the stealing policy
  (rank-ordered, not random) is hilavitkutin's own.

**From Tokio:**
- Parking/waking model: the concept of parking a task that cannot
  make progress and waking it via an external signal.
- Waker pattern: a handle that, when invoked, moves the parked unit
  back to the runnable queue.
- hilavitkutin does NOT take Tokio's I/O driver, async runtime,
  or task scheduling policy. It takes only the park/wake concept
  and implements it with atomic flags instead of Waker trait objects.

**From Leis et al. (HyPer/DuckDB morsel model):**
- Morsel dispatch: fixed-size chunks of column data assigned to
  workers for parallel processing.
- Cache-aware morsel sizing: morsel size derived from L2 cache
  geometry.
- Morsel-level work-stealing: workers that finish early steal
  morsels from other workers' deques.

**From Rayon:**
- Nothing directly. Rayon's fork-join model does not map to DAG
  scheduling. hilavitkutin uses crossbeam-deque (which Rayon also
  uses), but the deque is a data structure, not an algorithm.

### Feature matrix

| Feature | loimu | Rayon | Tokio | hilavitkutin |
|---------|-------|-------|-------|--------------|
| DAG scheduling | Yes | No | No | Yes |
| Upward rank | Yes | No | No | Yes |
| Lane assignment | Yes | No | No | Yes |
| Step barriers | Yes | No | No | Yes |
| Morsel dispatch | No | No | No | Yes |
| Yield (Pending) | No | No | Yes (.await) | Yes (Progress) |
| Yield (Budget) | No | No | No | Yes (Progress) |
| Yield (Retry) | No | No | No | Yes (Progress) |
| Work-stealing | Yes | Yes | Yes | Yes |
| Steal priority | Rank | Random | Random | Rank |
| Version stamps | Yes | No | No | Yes |
| Early-release | Yes | No | No | Yes |
| I/O driver | No | No | Yes | No |
| no_std compat | No | No | No | Yes |
| Heterogeneous cores | No | No | No | Yes |

The table makes the synthesis explicit. hilavitkutin combines loimu's
scheduling intelligence (rank, lanes, version stamps), Cilk's data
structures (lock-free deques), Tokio's conceptual model for I/O
integration (park/wake), and HyPer's data-parallel dispatch (morsels),
while adding its own contributions (heterogeneous core support, the
Progress enum, the unified yield model that bridges CPU and I/O workloads).

### Performance characteristics

For polka (CPU-bound, 20-50 nodes, no yielding):
- Plan computation: < 100 us
- Per-morsel dispatch: < 500 ns
- Step barrier overhead: < 10 us per step
- Total scheduler overhead per pass: < 1 ms
- Comparable to: raw Rayon parallel iterator (within 5-15%)

For saalis (mixed CPU + I/O, 10-30 nodes, 5-10 yielding):
- Plan computation: < 100 us
- Per-yield gap fill: < 1 us (steal from overflow)
- Pending wake detection: < 100 ns (atomic flag scan)
- I/O latency is dominant; scheduler overhead is noise
- Comparable to: hand-rolled Tokio task graph (within overhead margins)

For loimu (hundreds of behaviors, no yielding, UI frame rates):
- Plan computation: < 1 ms (recomputed infrequently)
- Per-behavior dispatch: < 500 ns
- Version stamp check: < 100 ns per resource
- Early-release saves: 0-30% of step barrier time (workload-dependent)
- Comparable to: loimu's own scheduler (same algorithm, shared code)

## 12. Summary of Design Decisions

1. **Plan computation uses loimu's 6-step algorithm.** Kahn's toposort,
   upward rank, lane assignment, lane_compat bitmask, overflow lists.
   The complexity is justified by cost heterogeneity in polka and saalis
   workloads.

2. **Steps, not wavefront.** Deterministic execution order, bounded
   concurrency, simpler debugging. Under-utilization is mitigated by
   yield-based work-stealing within steps.

3. **Per-lane morsel sizing for heterogeneous cores.** P-cores get
   larger morsels, E-cores get smaller morsels. Lane-to-core mapping
   at startup. Thread affinity hints for stability.

4. **EMA cost estimation with 2x recomputation threshold.** Converges
   in ~10 passes. Avoids plan thrashing on small fluctuations. Integer
   arithmetic for no_std.

5. **Yield fills the step barrier gap.** Workers whose units yield
   (Pending, Budget, or Retry) steal overflow work instead of idling.
   This recovers most of the parallelism that wavefront would provide,
   without sacrificing determinism.

6. **Atomic flag array for wake detection.** No_std compatible, cache-
   friendly scan, O(1) per parked unit. Consumers layer event-driven
   notification on top (Tokio waker sets the flag then notifies the
   runtime).

7. **The scheduler does not own I/O.** It provides yield/resume
   semantics. Consumers provide I/O integration. The boundary is the
   atomic wake flag.

8. **Crossbeam-deque for per-lane queues.** Lock-free, cache-friendly,
   proven implementation. Stealing policy is rank-ordered (not random),
   layered on top of Cilk's data structure.
