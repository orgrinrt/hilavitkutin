## Decisions locked

### Axis A. Test pyramid

Three layers, all enforced by `cargo test --workspace` clean before any src CL lock:

1. **Unit tests** — co-located with module under test in each crate. `#[cfg(test)]` modules at file bottom. Cover individual types, traits, fns. Existing engine crates have partial coverage already (task #399 landed); src CL extends to cover the runtime megaround's new types (PoolFrame, ResourceSnapshot, PlanAffecting, Executor, etc.).

2. **Integration tests** — `tests/*.rs` per crate. Cross-module within one crate. Test trait impls against each other, builder paths, scheduler construction end-to-end (no actual thread spawn at unit/integration layer; spawn validated at e2e).

3. **End-to-end tests** — `examples/*.rs` doubling as integration tests at the cross-crate level. Three example apps (axis B below). Each example is `cargo run --example <name>`-able; the example does real work (spawns threads, runs morsels, produces output). CI runs each example with assertions on output shape + timing budgets.

### Axis B. Three example apps

The Topic 11 user-attention call. Three candidate apps span the substrate's value props:

#### `examples/word_count.rs`

Canonical map-reduce. Demonstrates:
- `Column<Word>` (column-store over fixed-width tokens via `hilavitkutin-str` interning).
- One `MapWu` per shard, one `ReduceWu` aggregating; morsel-parallel within shards.
- `Resource<WordIndex>` for the interner; `Resource<HistogramOut>` for the accumulator.
- Tests: morsel parallelism (validates Topic 7), phase barrier (validates Topic 6 axis I), Resource access (validates Topic 8).

#### `examples/particle_step.rs`

N-body integrator step. Demonstrates:
- `Column<Particle>` with SoA-style splits (`Column<Position>`, `Column<Velocity>`, `Column<Mass>` as parallel columns over the same record id space).
- `IntegrateWu` doing per-particle position+velocity update using arvo `UFixed`/`IFixed` fixed-point math.
- Head+tail convergence variant (Topic 3 axis E) — pairwise force computation is symmetric, head+tail eligible.
- Tests: SIMD codegen path (validates Topic 5 axis H), head+tail (validates Topic 3 axis E), AdaptMetrics (validates Topic 5 axis B).

#### `examples/audio_mix.rs`

Real-time audio mixer (offline-render mode for test). Demonstrates:
- `Column<Sample>` over fixed-point audio frames (arvo `IFixed`).
- Multiple `MixerWu` instances composing in parallel, one `MasterBusWu` collapsing.
- Predictive parking (Topic 5 axis G) — audio block size known per phase, `predicted_wait_ns` is informative.
- Tests: deterministic latency budgets (validates Topic 6 axis J), bit-exact reproducibility (validates Topic 3 S3 stnp+fence ordering).

### Axis C. Test infrastructure

- `mock/crates/hilavitkutin-test-utils` (NEW crate, not yet in workspace) — shipping helpers for unit/integration tests across engine crates. Includes mock implementations of `ThreadPoolApi` (single-threaded synchronous executor for unit tests), `ClockApi` (deterministic monotonic clock), `MemoryProviderApi` (heap-backed allocator for test builds — yes, std-allocator, but ONLY in test-utils crate behind `#[cfg(test)]` walls).
- The mock thread pool runs everything on the calling thread. Validates control flow, not concurrency. e2e tests use the real OS pool.

### Axis D. Bench-as-test integration

The bench harness (`mock/benches/`) is independent from the test suite at execution time but the variants ARE compile-checked by `cargo test --workspace`. Each bench's `variants/<name>/Cargo.toml` is its own workspace and gets cargo-checked via a top-level `cargo mock check` that walks variant dirs.

The findings.md files in `mock/benches/` become part of the audit trail for design decisions. Don't gitignore them; they're load-bearing.
