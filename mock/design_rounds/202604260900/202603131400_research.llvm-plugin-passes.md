**Date:** 2026-03-13
**Type:** Research (not a design round topic)
**Context:** Morsel-driven columnar execution engine, data-processing workloads

## 1. The `-Z llvm-plugins` Mechanism

### Current Status

Nightly-only. Tracking issue: [rust-lang/rust#127577](https://github.com/rust-lang/rust/issues/127577).
Implemented in PR #86267. Not on any stabilization timeline.

The flag is documented in `rustc -Z help`:

```
-Z llvm-plugins=val -- a list LLVM plugins to enable (space separated)
```

Multiple plugins are space-separated:

```bash
RUSTFLAGS="-Z llvm-plugins=/path/to/plugin1.so /path/to/plugin2.so" cargo build
```

Custom passes can also be injected via:

```bash
RUSTFLAGS="-C passes=my-pass-name" cargo build
```

And LLVM flags forwarded via:

```bash
RUSTFLAGS="-C llvm-args=--some-flag=value" cargo build
```

### Critical Platform Limitation

**`-Z llvm-plugins` only works on x86_64-unknown-linux-gnu.**

On all other targets (including aarch64-apple-darwin, aarch64-unknown-linux-gnu),
LLVM is linked statically into `librustc_driver`. Dynamic plugin loading requires
a shared `libLLVM-XX.so`, which is only distributed for x86_64 Linux.

Confirmed on this machine:

```
$ rustc -vV
rustc 1.95.0-nightly (b3869b94c 2026-02-23)
LLVM version: 22.1.0

$ ls ~/.rustup/toolchains/nightly-aarch64-apple-darwin/lib/libLLVM*
# (no results -- no LLVM dylib on aarch64-apple-darwin)
```

The error you get on non-x86_64-linux:

```
symbol not found in flat namespace '__ZN4llvm17PreservedAnalyses14AllAnalysesKeyE'
```

See [rust-lang/rust#127573](https://github.com/rust-lang/rust/issues/127573) for details.

### Workaround: Build rustc from Source with Shared LLVM

The `bootstrap.example.toml` in rust-lang/rust has:

```toml
# Whether to build LLVM as a dynamically linked library (as opposed to statically linked).
# Under the hood, this passes `--shared` to llvm-config.
#llvm.link-shared = llvm.thin-lto

# Indicates whether the LLVM plugin is enabled or not
#llvm.plugins = false

# Whether to include the Polly optimizer.
#llvm.polly = false
```

To build a plugin-capable rustc for any platform:

```toml
# bootstrap.toml
[llvm]
link-shared = true
plugins = true
# polly = true  # if you want Polly linked into the toolchain
```

Then build rustc from source:

```bash
git clone https://github.com/rust-lang/rust.git
cd rust
cp bootstrap.example.toml bootstrap.toml
# Edit bootstrap.toml with the above settings
./x build
```

This is heavy (hours of compile time) but it is the only way to get
`-Z llvm-plugins` on macOS/ARM.

### Interaction with LTO

Not well documented. Plugins load into the LLVM pass pipeline that runs
on each codegen unit. With thin LTO, passes run both during initial
compilation and during the LTO linking phase. With fat LTO, all IR is
merged first, then the pass pipeline runs once. Plugin passes would
execute during these phases, but the exact interaction (do plugins run
in both pre-LTO and LTO phases?) is not formally specified.

### Interaction with Incremental Compilation

Incremental compilation caches LLVM IR. When a codegen unit is reused
from cache, the plugin pass pipeline does not re-run on it. Only changed
codegen units go through the full pipeline including plugins. This means
plugins that need whole-program analysis (cross-function optimization)
are incompatible with incremental compilation -- use `-C incremental=no`.

### Projects Using `-Z llvm-plugins` Today

- **afl.rs** (American Fuzzy Lop for Rust): Loads AFL's LLVM
  instrumentation pass. Primary user; most bug reports come from afl.rs.
- **Enzyme/Rust**: Automatic differentiation via LLVM plugin. Has its own
  integration path (`-Z llvm-plugins="./LLVMEnzyme-XX.so" -C passes="enzyme"`).
  Now being integrated more natively via `llvm.enzyme = true` in bootstrap.

## 4. Polly (Polyhedral Loop Optimizer)

### What It Optimizes

Polly detects Static Control Parts (SCoPs) -- loop nests with
affine bounds and array accesses -- and applies polyhedral
transformations:

- **Loop tiling**: Reorders iteration to improve cache locality.
  Transforms `for i in 0..N { for j in 0..M { A[i][j] } }` into
  tile-sized blocks that fit in L1/L2 cache.
- **Loop interchange**: Swaps nested loop order for better memory
  access patterns (row-major vs column-major).
- **Loop fusion**: Combines adjacent loops over the same domain to
  reduce memory traffic.
- **Vectorization**: Generates vectorized loop bodies using the
  polyhedral model for legality analysis.
- **Parallelization**: Generates OpenMP parallel loops.
- **GEMM detection**: Recognizes generalized matrix multiplication
  and generates optimized code similar to GotoBLAS/BLIS.

### Performance

Compile time overhead: ~1.4% (measured on Intel Core i7-7700).
Runtime improvement is workload-dependent; GEMM kernels see
significant speedups.

### Polly Passes

Frontend:
- `polly-canonicalize` -- prepare code for Polly
- `polly-detect` -- detect SCoPs in functions
- `polly-scops` -- create polyhedral description

Middle:
- `polly-dependences` -- calculate dependences
- `polly-opt-isl` -- optimize using isl scheduler

Backend:
- `polly-ast` -- execute isl code generation
- `polly-codegen` -- create LLVM-IR from polyhedral info

### Building Polly as a Loadable Plugin

By default, building Polly with `-DLLVM_ENABLE_PROJECTS=polly` links
it into LLVM tools. To build as a separate loadable plugin:

```bash
cmake -G Ninja ../llvm \
  -DCMAKE_BUILD_TYPE=Release \
  -DLLVM_ENABLE_PROJECTS="polly" \
  -DLLVM_TARGETS_TO_BUILD="AArch64;X86" \
  -DLLVM_POLLY_LINK_INTO_TOOLS=OFF \
  -DLLVM_BUILD_LLVM_DYLIB=ON \
  -DCMAKE_INSTALL_PREFIX=$HOME/llvm-22

ninja -j$(nproc)
# Produces: lib/LLVMPolly.so (Linux) or lib/LLVMPolly.dylib (macOS)
```

With `LLVM_POLLY_LINK_INTO_TOOLS=OFF`, Polly becomes a dynamic
plugin that must be explicitly loaded.

### Using Polly with rustc (Theoretical)

On x86_64-linux with shared LLVM rustc:

```bash
RUSTFLAGS='-Z llvm-plugins=/path/to/LLVMPolly.so \
  -C llvm-args=-polly \
  -C llvm-args=-polly-vectorizer=stripmine \
  -C llvm-args=-polly-position=before-vectorizer' \
  cargo build --release
```

### Key Polly Flags

| Flag | Effect |
|------|--------|
| `-polly` | Enable Polly |
| `-polly-vectorizer=stripmine` | Enable Polly vectorization |
| `-polly-parallel` | Generate OpenMP parallel code |
| `-polly-tiling` | Enable loop tiling (default: on) |
| `-polly-tiling=false` | Disable tiling |
| `-polly-position=early` | Run before standard optimizations |
| `-polly-position=before-vectorizer` | Run before LLVM vectorizer |
| `-polly-only-func=name` | Optimize only one function |
| `-polly-num-threads=N` | Thread count (0 = auto) |
| `-polly-scheduling=static` | OpenMP scheduling strategy |

### Known Issues with Polly + Rust

1. **SCoP detection on Rust IR**: Rust generates more complex LLVM IR
   than C (bounds checks, panics, enum discriminants). Polly's SCoP
   detection may fail to recognize loops that C would expose cleanly.
2. **Version coupling**: Polly MUST be built from the exact same LLVM
   revision as rustc's bundled LLVM. Any mismatch causes ABI
   incompatibility and crashes.
3. **Lack of testing**: No one is systematically testing Polly with
   Rust-generated IR. Edge cases are likely.
4. **Platform limitation**: See section 1 -- only x86_64-linux without
   building rustc from source.

## 5. LLVM Passes Already Available in rustc's Bundled LLVM

rustc's bundled `opt` (LLVM 22.1.0) includes 551 passes. The ones
relevant to data processing are already there without any plugin:

### Loop Optimization Passes

| Pass | Purpose |
|------|---------|
| `loop-distribute` | Break complex loops for better vectorization |
| `loop-fusion` | Combine adjacent loops over same data |
| `loop-interchange` | Swap nested loop order (row/column reorder) |
| `loop-flatten` | Flatten nested loops into single loop |
| `loop-unroll` | Unroll loops (O0/O1/O2/O3 presets, partial, runtime) |
| `loop-unroll-and-jam` | Unroll outer + jam inner (improves register reuse) |
| `loop-data-prefetch` | Insert prefetch instructions for loop data |
| `loop-vectorize` | Auto-vectorize loops (SIMD) |
| `loop-load-elim` | Eliminate redundant loads in loops |
| `loop-sink` | Sink instructions out of loops |
| `loop-reduce` | Loop strength reduction |
| `loop-simplify` | Canonicalize loop structure |
| `loop-rotate` | Convert to do-while form |
| `loop-deletion` | Remove dead loops |
| `loop-idiom` | Recognize loop idioms (memset, memcpy) |
| `loop-idiom-vectorize` | Vectorized loop idiom recognition |
| `loop-versioning` | Create fast/slow loop versions |
| `loop-versioning-licm` | Loop versioning for LICM |
| `loop-bound-split` | Split loops at bounds |

### Vectorization Passes

| Pass | Purpose |
|------|---------|
| `loop-vectorize` | SIMD vectorization of loops |
| `slp-vectorizer` | Superword-level parallelism (combine scalar ops) |
| `load-store-vectorizer` | Combine adjacent loads/stores into vector ops |
| `vector-combine` | Combine vector operations |
| `sandbox-vectorizer` | Experimental modular vectorizer |

### Memory Optimization

| Pass | Purpose |
|------|---------|
| `licm` | Loop-invariant code motion |
| `gvn` | Global value numbering (eliminates redundant loads) |
| `dse` | Dead store elimination |
| `memcpyopt` | Optimize memcpy/memmove/memset |
| `sroa` | Scalar replacement of aggregates |

### Activating These Without Plugins

Many of these run at `-C opt-level=3` already. To add specific passes:

```bash
# Add loop distribution (not in default pipeline)
RUSTFLAGS="-C passes=loop-distribute" cargo build --release

# Disable default vectorization and rely on something else
RUSTFLAGS="-C no-vectorize-loops -C no-vectorize-slp" cargo build --release

# Forward LLVM flags
RUSTFLAGS='-C llvm-args=-force-vector-width=8 \
  -C llvm-args=-force-vector-interleave=2' cargo build --release

# See what passes are running
RUSTFLAGS="-Z print-llvm-passes" cargo build --release

# Time individual passes
RUSTFLAGS="-Z time-llvm-passes" cargo build --release
```

## 6. Writing Custom LLVM Passes

### In C++ (Standard Approach)

The new pass manager API requires:

```cpp
// MyPass.h
#include "llvm/IR/PassManager.h"

namespace llvm {
class MyPass : public PassInfoMixin<MyPass> {
public:
  PreservedAnalyses run(Function &F, FunctionAnalysisManager &AM);
  static bool isRequired() { return true; }
};
} // namespace llvm
```

```cpp
// MyPass.cpp
#include "MyPass.h"
using namespace llvm;

PreservedAnalyses MyPass::run(Function &F, FunctionAnalysisManager &AM) {
  // Transform IR here
  return PreservedAnalyses::none(); // or ::all() if no modifications
}
```

Plugin entry point (required for dynamic loading):

```cpp
// Plugin.cpp
#include "llvm/Passes/PassPlugin.h"
#include "llvm/Passes/PassBuilder.h"
#include "MyPass.h"

extern "C" LLVM_ATTRIBUTE_WEAK ::llvm::PassPluginLibraryInfo
llvmGetPassPluginInfo() {
  return {
    LLVM_PLUGIN_API_VERSION, "MyPlugin", "0.1",
    [](PassBuilder &PB) {
      PB.registerPipelineParsingCallback(
        [](StringRef Name, FunctionPassManager &FPM,
           ArrayRef<PassBuilder::PipelineElement>) {
          if (Name == "my-pass") {
            FPM.addPass(MyPass());
            return true;
          }
          return false;
        });
    }
  };
}
```

### CMake for Out-of-Tree C++ Pass

Minimal standalone `CMakeLists.txt`:

```cmake
cmake_minimum_required(VERSION 3.20)
project(my-llvm-pass)

# Point to LLVM installation
set(LT_LLVM_INSTALL_DIR "" CACHE PATH "LLVM installation directory")
list(APPEND CMAKE_PREFIX_PATH "${LT_LLVM_INSTALL_DIR}/lib/cmake/llvm/")
find_package(LLVM REQUIRED CONFIG)

include_directories(SYSTEM ${LLVM_INCLUDE_DIRS})
add_definitions(${LLVM_DEFINITIONS})
set(CMAKE_CXX_STANDARD 17)

if(NOT LLVM_ENABLE_RTTI)
  set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -fno-rtti")
endif()

add_library(MyPass SHARED MyPass.cpp Plugin.cpp)

# macOS needs this to allow undefined LLVM symbols resolved at load time
target_link_libraries(MyPass
  "$<$<PLATFORM_ID:Darwin>:-undefined dynamic_lookup>")
```

Build and use:

```bash
mkdir build && cd build
cmake -DLT_LLVM_INSTALL_DIR=$HOME/llvm-22 ..
make

# Test with opt
opt --load-pass-plugin=./libMyPass.so --passes=my-pass input.ll -S

# Use with rustc (x86_64-linux only with shared LLVM)
RUSTFLAGS="-Z llvm-plugins=./libMyPass.so -C passes=my-pass" cargo build
```

### In Rust (via `llvm-plugin` crate)

The `llvm-plugin` crate (v0.6.0, by jamesmth) wraps the new pass manager
API using Inkwell's type-safe LLVM bindings.

Repository: https://github.com/jamesmth/llvm-plugin-rs
Docs: https://docs.rs/llvm-plugin

Supported LLVM versions: 11-20 via feature flags (`llvm11-0` through `llvm20-1`).
LLVM 22 is not yet supported as of v0.6.0 (would need `llvm22-1` feature).

```toml
# Cargo.toml
[package]
name = "my-pass"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib"]

[dependencies]
llvm-plugin = { git = "https://github.com/jamesmth/llvm-plugin-rs", features = ["llvm20-1"] }
```

```rust
use llvm_plugin::inkwell::values::FunctionValue;
use llvm_plugin::{
    FunctionAnalysisManager, LlvmFunctionPass, PassBuilder,
    PipelineParsing, PreservedAnalyses,
};

#[llvm_plugin::plugin(name = "MyPass", version = "0.1")]
fn plugin_registrar(builder: &mut PassBuilder) {
    builder.add_function_pipeline_parsing_callback(|name, manager| {
        if name == "my-pass" {
            manager.add_pass(MyPass);
            PipelineParsing::Parsed
        } else {
            PipelineParsing::NotParsed
        }
    });
}

struct MyPass;
impl LlvmFunctionPass for MyPass {
    fn run_pass(
        &self,
        function: &mut FunctionValue,
        _manager: &FunctionAnalysisManager,
    ) -> PreservedAnalyses {
        // Inspect/transform LLVM IR via Inkwell API
        for bb in function.get_basic_blocks() {
            for instr in bb.get_instructions() {
                // ...
            }
        }
        PreservedAnalyses::All
    }
}
```

#### Analysis Pass Example

```rust
use std::collections::HashMap;
use llvm_plugin::inkwell::values::{FunctionValue, InstructionOpcode};
use llvm_plugin::{
    AnalysisKey, FunctionAnalysisManager, LlvmFunctionAnalysis,
    LlvmFunctionPass, PassBuilder, PipelineParsing, PreservedAnalyses,
};

struct MyAnalysis;
impl LlvmFunctionAnalysis for MyAnalysis {
    type Result = HashMap<InstructionOpcode, usize>;

    fn run_analysis(
        &self,
        function: &FunctionValue,
        _manager: &FunctionAnalysisManager,
    ) -> Self::Result {
        let mut counts = HashMap::new();
        for bb in function.get_basic_blocks() {
            for instr in bb.get_instructions() {
                *counts.entry(instr.get_opcode()).or_insert(0) += 1;
            }
        }
        counts
    }

    fn id() -> AnalysisKey {
        1 as AnalysisKey
    }
}
```

#### Missing Features in `llvm-plugin`

- No loop pass support (Inkwell lacks loop wrappers)
- No CGSCC pass support
- No FFI over builtin LLVM analyses (dominator tree, alias analysis)
- Incomplete manager proxy API
- Incomplete analysis invalidation API

These gaps matter for data-processing passes: loop tiling or fusion
passes would need the loop pass infrastructure that is currently absent.

## 7. Example: Pass Targeting `chunks_exact()` Patterns

Rust's `chunks_exact()` compiles to a loop with:
- A bounds check on the slice length
- A main loop body processing `chunk_size` elements
- An induction variable incrementing by `chunk_size`
- A remainder handled separately

In LLVM IR, this appears as a loop with:
- An `icmp ult` for the remaining length
- A `getelementptr` advancing by the chunk stride
- The main body operating on a slice pointer + known length

A custom pass could:

1. **Detect the pattern**: Look for loops where the induction variable
   advances by a constant stride and the loop body accesses a contiguous
   memory range of that stride size.
2. **Insert alignment hints**: If the base pointer might not be aligned,
   peel iterations until alignment is reached, then mark the remaining
   loop body with alignment metadata (`!align`).
3. **Insert prefetch instructions**: For the next chunk's memory, insert
   `llvm.prefetch` intrinsic calls at the start of each iteration.
4. **Promote to vector operations**: If the chunk body performs scalar
   operations on each element, attempt to combine them into vector
   operations (complementing LLVM's own SLP vectorizer).

In practice, LLVM's existing passes (loop-vectorize, slp-vectorizer,
loop-data-prefetch) already handle most of these cases at `-O3`. A
custom pass would only add value for patterns the existing passes miss,
such as:

- Multi-column access patterns (two arrays accessed in lockstep with
  different strides)
- Gather/scatter patterns that LLVM's cost model rejects but are
  profitable on modern hardware
- Domain-specific fusion (combining a filter + projection that LLVM
  cannot prove safe to fuse)

## 8. Other LLVM Passes Available as Plugins

### Research / Third-Party Passes

Most research LLVM passes are built as out-of-tree plugins. Examples:

- **llvm-tutor** (banach-space/llvm-tutor): 11 reference passes for
  learning. Includes opcode counting, function call injection, basic
  block duplication, instruction obfuscation.
- **Enzyme**: Automatic differentiation. Transforms LLVM IR to compute
  derivatives. Available as plugin or linked into LLVM.
- **AFL/AFL++ instrumentation**: Coverage-guided fuzzing instrumentation.
  The primary real-world user of `-Z llvm-plugins` in Rust.
- **LLVM string obfuscator**: Encodes string constants in binaries
  (security/anti-reverse-engineering). The `llvm-plugin` Rust crate
  includes a full reimplementation of this as an example.

### Sanitizer Passes

ASAN, TSAN, MSAN, and UBSAN are built into LLVM, not loaded as plugins.
They are integrated into rustc via `-Z sanitizer=address` etc. These
cannot be meaningfully replaced by plugins.

### Custom Instrumentation

Custom profiling passes can be written as plugins to:
- Count specific instruction patterns
- Measure branch prediction accuracy
- Track memory access patterns
- Inject timing instrumentation around specific functions

## 9. Passes Relevant to Columnar Data Processing

### What Matters for Morsel-Driven Execution

In a morsel-driven columnar engine, the hot path is:

```
for each morsel (64KB-256KB of rows):
  for each operator in chain:
    for each column needed:
      process column[row_start..row_start + morsel_size]
```

The critical optimization targets:

**Loop tiling (cache blocking)**:
Already in LLVM as `loop-interchange` and Polly's `-polly-tiling`.
For columnar access, the key is ensuring that when processing multiple
columns, the data for all columns in a tile fits in L1/L2. LLVM's loop
interchange can reorder `for column { for row }` into `for tile { for column { for rows_in_tile } }`.

**Loop distribution**:
Available as `loop-distribute`. Splits a loop with independent
operations into separate loops, enabling better vectorization of each.
For a loop that reads column A and column B and writes column C, loop
distribution can split this into: vectorize the read of A, vectorize the
read of B, vectorize the computation and write of C.

**Loop fusion**:
Available as `loop-fusion`. Combines adjacent loops over the same range.
For chained operators (filter then project), if both are simple loops
over the same morsel, fusion eliminates intermediate materialization.

**SLP vectorization**:
Available as `slp-vectorizer`. Combines independent scalar operations on
different columns into SIMD operations. If you process 4 columns
simultaneously with the same operation, SLP can vectorize across columns.

**Data prefetch**:
Available as `loop-data-prefetch`. Inserts software prefetch instructions
for loop iteration N+K while processing iteration N. Critical for
streaming through column data that exceeds L1 cache.

**Alignment optimization**:
LLVM's loop vectorizer already handles alignment. You can help it via
`#[repr(align(64))]` on your column storage types and `unsafe` pointer
alignment assertions.

### What LLVM Cannot Do (Requiring Custom Passes or Code Design)

1. **Cross-function fusion**: LLVM will not fuse loops across function
   call boundaries unless the callee is inlined. For operator chains,
   this means your chain execution function must be inlineable or
   monomorphized (generics, not `dyn`).

2. **Predicate pushdown**: LLVM does not understand that a bitmap
   filter applied to column A also applies to column B. This is a
   query optimization, not a compiler optimization.

3. **Adaptive morsel sizing**: Choosing morsel size to fit L2 cache
   requires runtime knowledge of the number of columns and their widths.
   No LLVM pass can do this; it is a scheduler decision.

4. **SIMD dispatch**: Choosing between SSE4.2, AVX2, AVX-512, and NEON
   at runtime based on CPU detection. LLVM can target a specific ISA
   (`-C target-cpu=native`) but dynamic dispatch requires explicit
   `#[target_feature]` functions.

### Practical Recommendation

For a Rust data-processing engine, the highest-impact approach is not
custom LLVM passes but rather:

1. **Write vectorization-friendly Rust**: Use `chunks_exact()`,
   avoid bounds checks in hot loops (`get_unchecked`), ensure column
   data is aligned, use `#[repr(C)]` for predictable layout.

2. **Use `-C opt-level=3 -C target-cpu=native`**: This enables the
   loop vectorizer, SLP vectorizer, and loop-data-prefetch with
   hardware-specific tuning.

3. **Consider `-C llvm-args` for targeted tuning**:
   ```bash
   RUSTFLAGS='\
     -C llvm-args=-force-vector-width=8 \
     -C llvm-args=-force-vector-interleave=4 \
     -C llvm-args=-enable-loop-distribute' \
     cargo build --release
   ```

4. **Polly is interesting but impractical for Rust today**: The SCoP
   detection struggles with Rust's IR patterns, and the platform
   limitation (x86_64-linux only, or build rustc from source) is a
   significant barrier.

5. **Custom LLVM passes are a last resort**: The development and
   maintenance cost is high, version coupling is fragile, and the
   platform limitation means you can only use them on x86_64-linux
   in production. Better to write the optimizations in Rust itself
   (manual vectorization with `std::simd` or `std::arch` intrinsics).

---

## 10. Summary Table

| Approach | Effort | Platform | Impact |
|----------|--------|----------|--------|
| `-C opt-level=3 -C target-cpu=native` | Zero | All | High |
| `-C llvm-args=...` tuning flags | Low | All | Medium |
| `-C passes=loop-distribute` etc. | Low | All | Medium |
| Polly via `-Z llvm-plugins` | High | x86_64-linux only* | Medium-High for affine loops |
| Custom C++ LLVM pass | Very High | x86_64-linux only* | Varies |
| Custom Rust LLVM pass (llvm-plugin crate) | High | x86_64-linux only* | Varies |
| Manual SIMD in Rust (`std::simd`/`std::arch`) | Medium | All | High |

*Unless you build rustc from source with `llvm.link-shared = true`.
