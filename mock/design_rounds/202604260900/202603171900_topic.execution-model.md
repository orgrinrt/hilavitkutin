## Purpose

The API surface topic (Q1-Q12) documents the experimental journey:
what was tested, what worked, what didn't. This topic distills the
results into the concrete engineering rules for hilavitkutin's
execution engine. These rules are the specification the doc
changelist and source changelist implement against.

Every rule here is backed by benchmark evidence and ASM inspection.
References point to the API surface topic sections where the evidence
lives.

## 1. Chain formation

### 1.1. Column-count limit: register file, not L1

The chain grouper must cap fused chain column count at the REGISTER
FILE limit, not L1 capacity.

```
max_chain_cols = min(
    L1_usable / (MIN_MORSEL * sizeof(u64)),   // L1 limit (~93 on M1)
    register_budget,                           // aarch64: ~14, x86-64: ~10
)
```

`register_budget` = total GPRs - reserved registers:
- aarch64: 31 GPRs - 17 reserved (loop counter, intermediates, constants,
  stack frame) = ~14
- x86-64: 16 GPRs - 6 reserved = ~10

**Evidence:** a 15-column fused trunk (6 WUs) showed 8 wasted
instructions per element from constant rematerialisation + stack
spills. After splitting to ≤14 cols per chain, the ASM shows zero
per-row stack traffic and all column pointers in registers.

**Ref:** API surface §ASM INSPECTION, §register pressure finding.

### 1.2. Unique column count, not sum

The chain's column count for morsel sizing must be the UNIQUE set of
columns across all fused WUs, not the sum of per-WU col_counts.

`(WuSpatialHash[7], WuCollResp[7])` has 7 unique columns, not 14.
Summing produces morsels ~2x smaller than necessary.

The WorkUnit trait should declare actual column indices (not just a
count) so the chain grouper can compute the union. The dispatch
macro's `col_count()` should return the unique count.

**Evidence:** ASM inspection showed morsel=876 for SpatialHash+
CollResp (from sum=14), expected morsel=1752 (from unique=7).

### 1.3. Chain formation algorithm

Kahn's topological sort → greedy fusion walk:

1. Build DAG: write→read edges, order field resolves write-write cycles
2. Topological sort via Kahn's algorithm
3. Walk topo order. For each WU:
   - If all predecessors are in one chain AND adding this WU keeps
     unique_col_count ≤ max_chain_cols → fuse
   - If predecessors span multiple chains (fan-in) → new chain
   - If column budget exceeded → new chain
4. Compute inter-chain deps from WU-level preds

**Ref:** bench `chain.rs` form_chains(), validated on 17-WU game world
producing 12 chains.

### 1.4. WU authoring rule: ≤8 columns per WU

WUs should be written with ≤8 unique columns each. A 13-column
monolithic WU (wu_ai_decide) caused register pressure even in
isolation. Splitting to 3 focused WUs (3-7 cols each) gave 14%
improvement.

This is a guideline for WU authors, not an engine constraint. The
chain grouper handles whatever WUs are given.

**Ref:** API surface §AI split finding, bench game_world_typed.rs.

## 2. Morsel sizing

### 2.1. Base formula

```
morsel_size = (L1_usable / (chain_col_count * sizeof(T)))
              .clamp(MIN_MORSEL, MAX_MORSEL)
              & !3   // align to 4
```

Constants:
- L1_usable: L1_size * 0.75 (leave 25% for stack, OS, etc.)
  - macOS aarch64: 128 KB * 0.75 = 96 KB
  - Linux x86-64: 32 KB * 0.75 = 24 KB
- MIN_MORSEL: 64 elements
- MAX_MORSEL: 8192 elements
- sizeof(T): 8 bytes for u64 columns (type-native stride per Q8)

### 2.2. Write-only morsel sizing

Use `write_col_count` instead of `total_col_count` in the formula.
Read-only columns (from previous phases) ride the hardware prefetcher
and don't need L1 budget.

Consistent ~5% improvement across all measured workloads.

The WorkUnit trait provides both `col_count()` and `write_col_count()`.
The dispatch uses `write_col_count().max(1)` for sizing.

**Evidence:** write-sized median 16.78ms vs compiled 17.73ms at 1M
agents, confirmed across 30-sample protocol.

### 2.3. Per-chain morsel size

Each chain has its own morsel size, computed from its unique column
count. Different chains in the same pipeline have different morsel
sizes. This is correct — the morsel size adapts to the chain's
working set.

## 3. Dispatch codegen

### 3.1. Monomorphised tuple dispatch (Approach A)

Each chain is a tuple type of WU ZSTs. The `ChainDispatch` impl
constructs a LOCAL `&[WuFn]` slice. LLVM devirtualises all function
pointers.

```rust
impl ChainDispatch for (WuGravity,) {
    #[inline(never)]
    fn dispatch(&self, store: &mut Store, start: usize, end: usize) {
        let ops: &[WuFn] = &[WuGravity::func()];
        let ms = morsel_size(self.col_count());
        // morsel loop with ops
    }
}
```

The `#[inline(never)]` on the chain dispatch lets LLVM optimise each
chain as a separate compilation unit. WU functions are `#[inline(always)]`
within their chain.

**What devirtualises:**
- Local `&[fn]` slices with known values ✓
- Monomorphised trait dispatch ✓

**What does NOT devirtualise:**
- Struct-field fn pointer arrays (12.6x slower)
- `&[fn; N]` const-generic parameters (5.8x slower)
- Global `static mut` SLOTS (indirect `blr` per call)
- `run_chain(&[WuFn], ...)` with `#[inline(never)]` (one fn for all callers)

**Ref:** API surface §Q12 dispatch findings, §struct-field devirt test,
bench dispatch.rs macro.

### 3.2. ASM verification checklist

For any chain dispatch function, verify:
1. Zero `blr` (indirect branch via register) in the function body
2. All column accesses use `[reg, counter, lsl #3]` — indexed addressing
3. No `[sp, ...]` loads in the per-row inner loop (only in prologue/morsel setup)
4. Morsel size is an immediate constant (`mov wN, #value` or `add xN, xM, #value`)
5. No `bl morsel_size` or `bl chain_dispatch` calls (everything inlined/constant-folded)

All 26 chain dispatch functions in the current bench pass all 5 checks.

### 3.3. Compiled dispatch (per-core programs)

For the final execution, each physical core gets a monomorphised
function encoding its entire pipeline:

1. Which phases this core processes
2. Which row ranges (head half or tail half for convergence)
3. The fused WU sequence per chain (devirtualised)
4. Morsel boundaries (compile-time constants)
5. Phase sync points (stack AtomicUsize + spin loop)

The per-core function is `#[inline(never)]`. It calls chain dispatch
functions which are also `#[inline(never)]` (each chain optimised
separately by LLVM). WU functions within chains are `#[inline(always)]`.

**Sync mechanism:** stack `AtomicUsize`, not `Arc<AtomicUsize>`.
Raw pointer (`as usize`) passed to spawned thread. Eliminates 6 heap
allocations per pipeline execution.

**Spin loop pattern:**
```asm
isb              ; spin_loop hint (yield pipeline)
ldapr x9, [x8]  ; load-acquire from AtomicUsize
cmp x9, x20     ; compare against target row count
b.lo loop_back   ; if not done, loop
```

Single `isb` + `ldapr` + `cmp` + conditional branch. Minimal.

**Ref:** bench game_world_compiled.rs, ASM audit confirming 0 blr
in execute_compiled, 1 spin loop for phase boundary.

### 3.4. Thread spawn overhead

`std::thread::spawn` adds 3 `malloc` calls per `execute_compiled`
invocation (JoinHandle packet, closure capture, thread stack setup).
At ~17ms per call, this is ~100ns total — 0.0006%. Negligible.

For production: pre-allocated thread pool eliminates this entirely.
The bench uses `std::thread::spawn` for simplicity.

## 4. Phase decomposition

### 4.1. Algorithm

1. Build DAG from WU read/write sets + order
2. Topological sort
3. Detect waist points: positions where concurrent path count drops
   to minimum. Waists are mandatory phase boundaries.
4. Per phase: identify trunks (column-disjoint maximal paths)
5. Assign bridges (fan-in nodes that read from multiple trunks)
6. Form chains within each trunk (§1 rules)

### 4.2. Trunk parallelism

Trunks within a phase share NO write columns. They may share read
columns (both read from previous phase output). Trunks run on
separate cores with zero synchronisation, zero shared cache lines.

### 4.3. Head+tail convergence

Within any chain, rows are independent. Two threads process the
same chain from opposite ends, converging in the middle. Every chain
gets ~2x parallelism.

Resource accumulation caveat: each thread gets its own stack-local
accumulator. Merged after convergence (addition for additive resources).

### 4.4. Pipeline parallelism between phases

Phase N+1 starts as soon as phase N produces enough rows for one
morsel. Progress tracked via AtomicUsize per phase. Same spin loop
pattern as intra-phase sync.

### 4.5. Results progression (game world, 1M agents)

```
40.4ms  sequential (SLOTS indirect calls)           1.00x
37.8ms  adaptive (flat chase+steal)                  0.94x
28.5ms  phased-pool (trunk parallelism)              0.71x
25.4ms  phased-conv (convergence)                    0.63x
27.4ms  typed-seq (devirt, 1 thread)                 0.68x
19.8ms  typed-pipe (devirt + Arc + overlap)           0.49x
19.1ms  compiled (devirt + stack atomics)             0.47x
18.3ms  write-sized (devirt + stack + write-only)     0.45x
16.8ms  write-sized (latest run, median)              0.42x
```

Each optimisation layer is additive. None are optional — removing
any one regresses measurably.

## 5. Scheduling strategy selection

### 5.1. Row count threshold

| rows | strategy |
|------|----------|
| < 10K | sequential always — thread overhead dominates |
| 10K-100K | depends on pipeline shape (§5.2) |
| > 100K | always consider threading |

### 5.2. Pipeline shape

| shape | indicator | strategy |
|-------|-----------|----------|
| wide (many roots, low depth) | roots > depth/2 | adaptive / pipe-chase |
| deep (serial deps, few roots) | depth > chains/2, roots ≤ 2 | sequential |
| mixed (some width, some depth) | phases have parallel trunks | phased |

### 5.3. Per-project defaults

| project | rows | shape | strategy |
|---------|------|-------|----------|
| polka-dots | ~200 | deep | sequential — 1µs total |
| saalis | 10K-100K | mixed | sequential for deep, adaptive for wide |
| loimu | 100K-10M | wide | phased with convergence |

### 5.4. Frame budget

Budget mainly controls incrementality (frame count), not throughput.
Adaptive strategies ignore budget. Sequential strategies are stable
across budgets (1.0-1.1ms variation at 100K rows regardless of
budget=1 through budget=unbounded).

## 6. Resource handling

### 6.1. Stack-local resource caching

Before the morsel loop, the chain dispatcher snapshots all accessed
resources to a stack-local array, redirects `res_ptr` to it. LLVM
proves stack doesn't alias heap column data → promotes all resource
accesses to registers.

After the loop, mutable resources are written back to canonical
storage via `stp`.

**Evidence:** without caching, 21 memory ops per iteration including
6 resource reloads. With caching: 0 resource memory ops in the hot
loop, all in registers.

**Ref:** API surface §resource caching finding.

### 6.2. Pointer separation

Resource storage must be behind a pointer to EXTERNAL storage, not
inline in the context/store struct. Writing inline data through
`&self` is UB (noalias). Raw pointer writes to external storage are
sound because the pointed-to memory has no aliasing relationship
with the `&self` reference.

### 6.3. &self receiver is load-bearing

Context methods take `&self`, not `&mut self`. The shared reference
prevents LLVM from adding noalias metadata that would cause it to
reorder writes across fused WUs.

Column writes go through raw pointers stored in `col_ptrs`. Raw
pointer writes have no aliasing assumptions and execute in program
order.

## 7. Deep-chain codegen (single-chain optimisations)

### 7.1. DSE: dead store elimination for chain-internal columns

Columns written by one WU and read only by the next WU in the same
chain don't need memory backing. They become register-to-register
data paths.

The chain grouper marks each column as:
- chain-internal: locals, no memory ops
- chain-input: read from memory at start
- chain-output: written to memory at end

### 7.2. Stores-at-end pattern

All output stores grouped after computation in the loop body. This
is store-buffer friendly and gives LLVM maximum scheduling freedom.

```rust
for i in morsel_start..morsel_end {
    let v0 = cu(c(0), i);            // read inputs
    let l1 = v0 ^ (v0 >> 33);        // compute (all locals)
    let l2 = l1.wrapping_mul(r0);
    // ...
    cw(c(1), i, l1);                 // stores grouped at end
    cw(c(2), i, l2);
}
```

### 7.3. Flattened vs pipeline dispatch

| pattern | overhead vs hand-written |
|---------|--------------------------|
| flattened (locals + DSE) | 0.95-1.00x |
| pipeline (fn-ptr devirt+inline) | 1.09-1.18x |

The 14% pipeline overhead exists ONLY in deep single-chain scenarios.
Multi-chain scenarios (game world, partition) already achieve
0.95-1.02x because chain boundaries are natural materialisation points.

**Ref:** API surface §flattened finding, §DSE finding.

## 8. What does NOT help

These were tested and found neutral or negative. Do not implement.

### 8.1. Row sorting (Morton, data-flow, state-based)

Tested: Morton spatial ordering, data-flow producer→consumer ordering,
AI state grouping. All neutral or 5-10% negative on synthetic data.

**Root cause:** sequential scan is already prefetcher-optimal.
Reordering destroys the sequential pattern and causes TLB misses.

**When it would help:** real data with correlated spatial/state
clusters where branch prediction benefits compound. With random
synthetic data, nothing to predict.

**Definitive result:** sorted-reuse 18.58ms vs compiled 17.73ms —
10.7% slower even with fair measurement (same arena, no re-allocation).

### 8.2. Explicit prefetch

`core::intrinsics::prefetch_read_data` with locality hint 3 (L1)
makes everything ~2x slower on Apple Silicon. Even morsel-level
software pipelining (prefetch next morsel) is 1.47x slower.

Hardware prefetcher handles sequential column access optimally.
Explicit prefetch competes with it.

Reserve for: random/pointer-chasing access, architectures with weak
prefetchers. `#[cfg(target_arch)]` + profiling-gated, not default.

### 8.3. `likely`/`unlikely` intrinsics in dispatch loop

`if likely(i < n)` prevents LLVM from vectorising. 1.40x slower.
LLVM already eliminates bounds checks for `start..end` iteration.

### 8.4. Deep-chase (both-ends convergence on serial deps)

Processing every chain from both ends via CAS atomics. 0.94-1.15x
on deep pipelines — no help, sometimes worse. CAS contention
exceeds savings. Only convergence WITHIN a chain (§4.3) helps.

## 10. What WUs must declare

For the entire execution model to work automatically:

1. **Read column set** — columns this WU reads (not writes)
2. **Write column set** — columns this WU produces (write implies read)
3. **Order** — relative execution order among WUs writing the same column

Everything else is derived: DAG, phases, trunks, chains, morsel sizes,
strategy selection, codegen, sync points.

The user writes:
```rust
#[work_unit(order = 3)]
fn spatial_hash(
    #[read] pos_x: &Column<PosX>,
    #[read] pos_y: &Column<PosY>,
    #[read] pos_z: &Column<PosZ>,
    #[write] coll_flag: &mut Column<CollFlag>,
    #[write] coll_nx: &mut Column<CollNX>,
) { ... }
```

The scheduler derives everything from these three declarations.

## Open questions for this topic

1. **col_count in tuple dispatch**: should WorkUnit expose actual column
   indices (not just a count) so the chain grouper can compute unique
   sets? This changes the trait surface.

2. **Thread pool vs std::thread::spawn**: the bench uses spawn for
   simplicity. Production needs a pre-allocated pool (§4 phased-pool
   findings). Pool API surface?

3. **Dynamic schedules**: plugins register WUs at runtime. The mono-
   morphised tuple dispatch requires compile-time chain composition.
   Fallback for dynamic registration? Approach C (indirect per-chain,
   1.17x) is the candidate.

4. **Platform-specific register budgets**: the `register_budget` in
   §1.1 should be a const generic or cfg-conditional. How does this
   integrate with the platform abstraction from the API surface §Q8?
