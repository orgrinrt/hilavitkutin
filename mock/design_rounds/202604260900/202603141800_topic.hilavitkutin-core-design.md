## Purpose

Three projects need pipeline execution: polka-dots (data compiler),
saalis (game library manager / query engine), loimu (UI framework).
Each has independently designed scheduling, work dispatch, and data
flow. This topic designs hilavitkutin as the shared building block
that eliminates duplicated effort while serving all three.

The goal is NOT to copy loimu's full architecture into a shared crate.
Loimu does far more than polka or saalis need (signal bus, seven access
modes, rotation groups, temporal prediction, GPU queue affinity). The
goal is to find the common denominators and build ONLY those, such that:

- polka-dots replaces polka-pipeline and polka-scheduler with thin
  wrappers around hilavitkutin
- saalis uses hilavitkutin for data pipeline work (query execution,
  batch transforms) alongside its connector orchestration
- loimu can adopt hilavitkutin's core scheduling primitives, reducing
  its own scheduler to a thin layer that adds loimu-specific features
  (signal bus, rotation groups, temporal prediction) on top

## Context: what each project has designed independently

### polka-dots (data compiler)

From the paradigm and dispatch topics:
- WorkUnit with `execute()`, context is ECS-style `Read<T>` / `Write<T>`
- Flat DAG pipeline: N->IR, IR->IR, IR->M transforms
- Morsel-driven execution (Leis et al.) for column processing
- Work-stealing scheduler with per-core queues
- Schema-as-traits for IR definition

### saalis (game library manager)

From the saalis team's integration analysis:
- Two kinds of work: I/O-bound connectors (HTTP, DB) and CPU-bound
  data pipelines (query projection, deduplication, aggregation)
- Event-driven DAG: connector tasks emit events (EntityDiscovered,
  HealthCheckFailed) that trigger downstream work
- Subsystem grouping with cadence (Curator: 1h, Doctor: 5min)
- Needs: context parameter for host services, retry policies, scalar
  dispatch for non-chunkable work, DB-backed storage bridging

### loimu (UI framework)

From loimu's scheduler and resource designs:
- Signal-driven scheduling (not frame-based): policies emit signals,
  scheduler wakes, runs DAG, sleeps
- Behaviors decomposed into scheduling units with field-granular
  read/write access analysis
- Resources as typed singletons backed by SlabColumn
- Seven access modes (Pinned, Immediate, MutImmediate, Sample, Post,
  Approx, Defer) controlling ordering and snapshotting
- Version stamps for phantom write detection and early-release
- List scheduling with upward rank, lane assignment, work stealing
- Rotation groups for prediction-based skipping
- NO yielding — behaviors are atomic, run to completion per pass
- Fault handling with rollback (sole-writer snapshot, shared undo journal)

### saalis team insights (from Phase 1 research)

The saalis team's integration analysis was refined during Phase 1:

- **Hot/cold store architecture.** Active query results and pipeline working
  data live in hilavitkutin's in-memory columnar store (hot). Persistent
  catalog and metadata live in SQLite via SeaORM (cold). hilavitkutin does
  not need to abstract over databases; saalis materializes external data
  into hilavitkutin columns before processing and persists results back to
  SQLite after.

- **Revised request list.** The original "context parameter for host
  services" request is unnecessary under the unified store model (Q1):
  HTTP clients, DB pools, and config are singleton resources accessed via
  `Read<T>`. No special context type needed. The original "virtual columns
  for events" request maps to zero-sized DAG edge types (Q3):
  `Write<EntityDiscovered>` creates a dependency edge with no storage. Both
  requests are satisfied by hilavitkutin's core model without saalis-specific
  extensions.

- **Yielding model for connectors.** saalis's primary motivation for
  yielding in hilavitkutin: HTTP connectors start a request, yield the
  worker (Progress::Yield(Pending)), let the worker steal other ready work,
  and resume when the response arrives. Database connectors with slow
  queries follow the same pattern. This is the primary use case that
  justifies the Progress enum over atomic-only execution.

## Common denominators (what hilavitkutin should own)

Candidate shared primitives, to be confirmed or rejected per question:

1. **DAG scheduling** — all three build a dependency graph from
   read/write declarations and execute in topological order
2. **Read/Write access typing** — all three use typed declarations
   to express what each work unit touches
3. **Work-stealing dispatch** — all three benefit from multi-core
   execution with load balancing
4. **Morsel chunking** — polka and saalis need it for column data;
   loimu's `each` blocks over collections are the same concept
5. **Yielding/cooperative scheduling** — polka and saalis need it
   for I/O-bound and long-running work; loimu deferred it to post-1.0
   but wants it (Naughty Dog fiber model)

NOT shared (project-specific layers on top):
- Signal bus (loimu only — polka and saalis use DAG edges, not signals)
- Seven access modes (loimu only — polka and saalis need Read/Write,
  not Pinned/Immediate/Sample/Post/Approx/Defer)
- Temporal prediction / rotation groups (loimu only)
- GPU queue affinity (loimu only)
- Connector orchestration / retry (saalis only, but retry hooks in
  hilavitkutin make it possible)
- Template compilation stages (polka only)

## Key design questions

**Q1. Unified store: columns and resources under one Read/Write?**

The core question. Can `Read<T>` / `Write<T>` mean both "read from a
multi-row column" and "read a singleton resource"? Loimu unifies them
(both backed by SlabColumn, scheduler doesn't distinguish). If we do
the same, the DAG sees all dependencies uniformly. If we split them,
we need two parallel dependency mechanisms.

Implications:
- Unified: `Read<HttpClient>` (singleton) and `Read<Entities>`
  (collection) create DAG edges the same way. The scheduler doesn't
  know or care about row count. Morsel chunking is internal to the
  work unit's execute(), not the scheduler's concern.
- Split: the scheduler tracks column deps and resource deps separately.
  Two systems, more complex, but potentially more precise (different
  ordering rules for singletons vs collections).

Saalis's `type Ctx` request becomes unnecessary under the unified
model: HTTP clients, DB pools, config are all resources accessed via
`Read<T>`. No special context parameter needed.

**Research findings (Q1):** The synthesis confirms unified at the scheduler
level, differentiated at the access level. The scheduler sees `AccessSet`
entries, not storage layout; the edge from `Write<T>` to `Read<T>` is a
typed dependency declaration regardless of whether T is a singleton or a
million-row column. Access ergonomics differ: singletons use
`.resource::<T>()` for direct access, collections use morsel iteration
(column iterators with optional selection vector). loimu validates this
with SlabColumn backing both resources (cardinality 1) and collections
(cardinality N) identically. Bevy separates `Res<T>` from `Query<&T>`
but both create the same scheduling edges, confirming the pattern. Arrow
and DuckDB keep global state (catalog, config) separate from table data
but use the same read/write tracking for scheduling. The common
denominator across all references: typed, contiguous, fixed-stride
storage with read/write tracking. One edge type, one conflict algorithm.

**Q1 Decision: three store types, one scheduling mechanism.**

Three types, one `Store` trait:
```rust
pub struct Resource<T>(PhantomData<T>);  // singleton, one value
pub struct Column<T>(PhantomData<T>);    // collection, N values, morsel-chunked
pub struct Virtual<T>(PhantomData<T>);   // zero-data, DAG edge only
```

Scheduler sees `StoreId` pairs for edge construction. Does not care
about cardinality. `Write<Resource<HttpClient>>` → `Read<Resource<HttpClient>>`
builds the same edge as `Write<Column<Entities>>` → `Read<Column<Entities>>`.

Access API differs by type inside `execute()`:
- `ctx.resource::<T>()` → `&T` (singleton direct access)
- `ctx.column::<T>()` → morsel-scoped iterator (collection)
- Virtual → no access methods (fire only via `ctx.fire::<T>()`)

Compile-time safety: `Context<R, W>` has `Contains<S>` bounds.
Accessing a store not in your Read/Write declaration is a compile error.

**Virtual<T> implementation: bit-packed fire flags with affinity ordering.**

One bit per registered Virtual type. Packed into u8 bytes, grouped
into u64 words. Hierarchical check:
```
u64 word == 0 → skip all 64 types (one comparison)
u8 byte == 0  → skip 8 types (one comparison)
bit set        → type fired, process downstream edges
```

Fire operation: `ctx.fire::<T>()` sets one bit. One byte load, one OR,
one byte store. Index is const-folded from `T::INDEX`. Reset per pass:
one `memset` of ceil(N/8) bytes.

**Affinity-ordered bit assignment.** Virtual types that tend to fire in
the same pass are packed into the same byte and word. The scheduler
computes co-fire affinity at plan time from static DAG analysis:

- If `Write<Virtual<A>>` and `Write<Virtual<B>>` are on work units
  connected by a DAG edge chain, they have high co-fire affinity.
- Two-level greedy bin-packing: first group into u64 words (types that
  fire in the same pass category), then within each word group into
  bytes (types that fire in the same chain).

Effect: cold words (maintenance, health checks) stay 0 in connector-
heavy passes. Hot words (connector pipeline virtuals) have multiple
bits set — one word check, one byte check, process all set bits.

Computed once at plan time. Zero runtime cost. Recomputed on DAG
change (plugin load, reconfiguration).

For typical usage (20-60 Virtual types): entire bitset fits in one
cache line. Coarse check is one u64 comparison. Common case (nothing
fired) costs one integer comparison per pass.

**Q2. Work unit model, phasing, scheduling hints — RESOLVED**

**Research findings (Q2):** loimu explicitly deferred yielding to post-1.0.
The Naughty Dog fiber model was studied and rejected: fibers require
platform-specific stack management, are no_std hostile, and complicate
debugging. DataFusion's `Poll::Pending` is the closest parallel.

**Q2 Decision: generic-conditioned WorkUnit trait with phase splitting.**

The WorkUnit trait takes a single generic parameter: the scheduling
condition. Multi-phase work is expressed as multiple trait impls on the
same type, each with its own condition, Read/Write, and execute body.

```rust
pub trait WorkUnit<Schedule = Always>: Send + Sync + 'static {
    const NAME: &'static str;
    type Read: AccessSet;
    type Write: AccessSet;
    type Hint: SchedulingHint = (Immediate, Atomic, Normal);
    type Ctx = Context<Self::Read, Self::Write>;

    fn execute(&self, ctx: Self::Ctx);
}
```

The generic parameter controls WHEN this impl runs:
```rust
pub struct Always;                    // unconditional (default)
pub struct On<V>(PhantomData<V>);     // when virtual V fires
pub struct After<U>(PhantomData<U>);  // after work unit U completes
```

**Simple morsel processor — one impl, no phases:**
```rust
impl WorkUnit for AddColumns {
    const NAME: &'static str = "add-columns";
    type Read = (Column<A>, Column<B>);
    type Write = (Column<C>,);

    fn execute(&self, ctx: Self::Ctx) {
        for (a, b, c) in ctx.zip::<(A, B, C)>() {
            *c = *a + *b;
        }
    }
}
```

**HTTP connector — two impls, virtual-gated phase:**
```rust
// phase 1: unconditional, fires request
impl WorkUnit for HttpFetcher {
    const NAME: &'static str = "http-fetcher";
    type Read = (Resource<HttpClient>,);
    type Write = (Column<RequestHandle>,);

    fn execute(&self, ctx: Self::Ctx) {
        let client = ctx.resource::<HttpClient>();
        ctx.column_mut::<RequestHandle>().push(client.send(request));
    }
}

// phase 2: runs when ResponseReady fires (externally)
impl WorkUnit<On<ResponseReady>> for HttpFetcher {
    const NAME: &'static str = "http-fetcher-process";
    type Read = (Column<RequestHandle>,);
    type Write = (Column<Entity>,);

    fn execute(&self, ctx: Self::Ctx) {
        for handle in ctx.column::<RequestHandle>() {
            ctx.column_mut::<Entity>().push(parse(handle.response()));
        }
    }
}
```

**Hash join — two impls, sequential via After:**
```rust
impl WorkUnit for HashBuild {
    const NAME: &'static str = "hash-build";
    type Read = (Column<BuildSide>,);
    type Write = (Resource<HashTable>,);
    fn execute(&self, ctx: Self::Ctx) { /* build */ }
}

impl WorkUnit<After<HashBuild>> for HashProbe {
    const NAME: &'static str = "hash-probe";
    type Read = (Resource<HashTable>, Column<ProbeSide>);
    type Write = (Column<JoinResult>,);
    fn execute(&self, ctx: Self::Ctx) { /* probe */ }
}
```

**Switch/branching — multiple On<V> impls:**
```rust
impl WorkUnit for RequestSender {
    type Write = (Column<RequestHandle>,);
    fn execute(&self, ctx: Self::Ctx) { /* send */ }
}

impl WorkUnit<On<ResponseOk>> for RequestSender {
    type Read = (Column<RequestHandle>,);
    type Write = (Column<Entity>,);
    fn execute(&self, ctx: Self::Ctx) { /* happy path */ }
}

impl WorkUnit<On<ResponseError>> for RequestSender {
    type Read = (Column<RequestHandle>,);
    type Write = (Column<FailedRequest>,);
    fn execute(&self, ctx: Self::Ctx) { /* error path */ }
}
```

No switch builder, no match syntax, no phase closures. Each branch
is a separate impl. The scheduler dispatches whichever impl's
condition is satisfied. Each execute body is branchless straight-line
code. LLVM optimizes each independently.

**Per-impl Read/Write:** each impl declares exactly what IT needs.
No union. The scheduler builds narrow, precise DAG edges per impl.
Write implies Read (no need to declare both).

**No partial writes.** Work units buffer internally, write on return.
Downstream doesn't run until the impl completes. Version stamps
increment only on completion.

**Everything is resumable.** The scheduler treats every work unit as
potentially interruptible between morsels. The Hint controls how
aggressively the scheduler tries to avoid interruption, but makes
no hard guarantee. Even an Atomic-hinted unit may be preempted under
pressure. Correctness never depends on atomicity — the DAG and
version stamps handle consistency regardless.

**External virtual firing:** external threads (I/O completion, timer,
platform events) fire virtuals via `scheduler.fire_external::<V>()`.
Atomic bit set on the virtual flag array. The scheduler scans at pass
start. No signal bus, no arena, no double buffering. Just bits.

**Scheduling hints (type Hint):**

Three axes, each a `#[repr(u8)]` enum where named variants are
well-known points on a 0-255 spectrum. The scheduler uses raw u8
values for arithmetic (rank multiplier, dispatch rate, split
threshold). ZST types carry the u8 value as a const. Hints are
expressed as tuples:

```rust
type Hint: SchedulingHint = (Immediate, Atomic, Normal);
```

**Axis 1 — Urgency:** how urgently should this run?
```rust
#[repr(u8)]
pub enum Urgency {
    Deferred = 0,     // only when nothing else pending
    Relaxed = 85,     // no rush
    Steady = 170,     // consistent pacing across passes
    Immediate = 255,  // run now
}
```

**Axis 2 — Divisibility:** can the scheduler interrupt between morsels?
```rust
#[repr(u8)]
pub enum Divisibility {
    Interruptible = 0,  // interrupt anytime, resume whenever
    Adaptive = 128,     // scheduler decides based on load
    Atomic = 255,       // prefer uninterrupted (hint, not guarantee)
}
```

**Axis 3 — Significance:** how important relative to other work?
```rust
#[repr(u8)]
pub enum Significance {
    Optional = 0,        // only when scheduler is idle
    Opportunistic = 64,  // when there's an opportunity
    Normal = 128,        // no special treatment (default)
    Important = 192,     // prefer over Normal work
    Critical = 255,      // must complete before other work
}
```

Named variants are convenience — consumers can define custom points:
```rust
pub struct SlightlyUrgent;
impl UrgencyValue for SlightlyUrgent { const VALUE: u8 = 200; }
```

**Usage examples:**
```rust
// default: run now, don't split, normal importance
impl WorkUnit for CriticalPipeline {
    type Hint = (Immediate, Atomic, Normal);  // or just omit (default)
    ...
}

// background index rebuild
impl WorkUnit for RebuildIndex {
    type Hint = (Relaxed, Interruptible, Opportunistic);
    ...
}

// health check on schedule
impl WorkUnit<On<HealthCheckDue>> for HealthCheck {
    type Hint = (Immediate, Atomic, Critical);
    ...
}

// large data migration, spread across passes
impl WorkUnit for DataMigration {
    type Hint = (Steady, Interruptible, Normal);
    ...
}
```

How the scheduler uses each axis:
- Urgency feeds into pass scheduling: Immediate units dispatch first,
  Deferred units fill leftover time
- Divisibility controls morsel batch size: Atomic gets all morsels at
  once, Interruptible gets one at a time with steal opportunities
  between, Adaptive adjusts based on EMA of execution time
- Significance multiplies upward rank: Critical gets rank × 2,
  Normal × 1, Optional × 0.25. Also controls work-stealing priority.

**Q3. Virtual columns / signal edges — RESOLVED**

**Research findings (Q3):** loimu's signal bus (arena-allocated ephemeral
messages, three buffer modes, double-buffered arenas) is over-engineered
for polka/saalis. The common denominator is zero-data DAG edges.

**Q3 Decision: pure flag virtuals with per-(virtual, consumer) bits.**

`Virtual<T>` is a zero-sized store type (resolved in Q1). It creates
DAG edges with no storage. `On<V>` in WorkUnit impls gates execution
on the virtual firing (resolved in Q2).

**Virtuals are pure flags, not data carriers.** If the event has
associated data, the data goes in a Column or Resource alongside the
Virtual. The virtual is the trigger; the column is the payload:
```rust
// producer fires virtual, writes data to column
impl WorkUnit for Cataloguer {
    type Write = (Column<Entity>, Virtual<EntityDiscovered>);
    fn execute(&self, ctx: Self::Ctx) {
        ctx.column_mut::<Entity>().push(entity);
        ctx.fire::<EntityDiscovered>();
    }
}

// consumer gates on virtual, reads data from column
impl WorkUnit<On<EntityDiscovered>> for Enricher {
    type Read = (Column<Entity>,);
    type Write = (Column<EnrichedEntity>,);
    fn execute(&self, ctx: Self::Ctx) { ... }
}
```

**Per-(virtual, consumer) bit tracking.** Each consumer of a virtual
has its own bit. This solves the persistence and interruption problem:

- Fire sets the bit for ALL consumers of that virtual
- Each consumer's bit is cleared by the scheduler on FIRST dispatch
  (not on resume of interrupted work, not on completion)
- If the virtual fires again during execution, the bit is set again
  (since it was cleared at dispatch). The consumer runs again next pass.
- No lost fires. No premature clearing.

Clear-on-dispatch rule:
```
fire()       → bit = 1 (for all consumers of this virtual)
dispatch()   → bit = 0 (first dispatch only, not resume)
               consumer starts executing
interrupt()  → consumer paused (bit stays 0)
resume()     → consumer continues (bit stays 0, no re-clear)
complete()   → consumer done
               if virtual fired again during execution: bit = 1
               → consumer dispatched again next pass
```

If a consumer is interrupted and the virtual fires again mid-execution,
the bit is set to 1. When the consumer completes and the scheduler
checks next pass, bit = 1 → dispatch again. No lost fires.

**Storage:** one bit per (virtual, consumer) pair. For 64 virtual types
with average 1.5 consumers each = ~96 bits = 12 bytes. One cache line.
Affinity ordering from Q1 applies at the virtual level.

**External firing:** `scheduler.fire_external::<V>()` atomically sets
the bits for all consumers of V. Internal firing via `ctx.fire::<V>()`
is non-atomic (DAG guarantees exclusive access).

**Per-pass lifetime with clear-on-dispatch.** A virtual fired in pass N
is visible until dispatched. If no consumer runs in pass N (scheduler
was busy), the bit persists to pass N+1, N+2, etc. No time-based
expiry. The bit clears only when a consumer is dispatched for it.

**loimu compatibility:** loimu builds its signal bus on top: virtuals
for scheduling edges + loimu-owned arena columns for signal data +
buffer modes (Replace/Queue/ReplaceBy) as loimu-layer logic.
hilavitkutin provides the bit; loimu provides the arena.

**Q4. Morsel dispatch vs scalar dispatch — RESOLVED**

Resolved by Q2: dispatch mode is inferred from the AccessSet. If
the WorkUnit impl's Read or Write contains a `Column<T>`, the
scheduler morsel-chunks (calls execute() once per morsel). If only
`Resource<T>` and `Virtual<T>`, the scheduler dispatches once (scalar).
No explicit dispatch mode annotation needed. The Divisibility axis
in the scheduling hint (Atomic/Adaptive/Interruptible) controls how
aggressively the scheduler splits morsel dispatch across passes.

**Q5. Scheduling groups and cadence — RESOLVED**

**Research findings (Q5):** loisto has a scene state machine with power
budgets. loimu has three policies (Continuous/Adaptive/OnDemand). saalis
has fixed-interval subsystem grouping. polka runs once, exits.

**Q5 Decision: cadence is consumer-layer, built on virtuals + metrics.**

hilavitkutin does not own a cadence mechanism. Cadence = a virtual that
fires on a timer. The timer is consumer-managed:

```rust
// saalis: background thread fires CuratorTick every hour
// every 3600s: scheduler.fire_external::<CuratorTick>()

impl WorkUnit<On<CuratorTick>> for CatalogueRefresh {
    type Hint = (Relaxed, Interruptible, Opportunistic);
    type Read = (Resource<CatalogueState>,);
    type Write = (Column<Entity>,);
    fn execute(&self, ctx: Self::Ctx) { ... }
}
```

loimu builds its Continuous/Adaptive/OnDemand policies as consumer
logic that fires virtuals at computed intervals. loisto drives
cadence from its scene state machine and power budget. polka fires
no cadence virtuals (runs once).

**Scheduler metrics as a Resource for adaptive cadence.**

The scheduler writes `Resource<SchedulerMetrics>` at the START of
each pass, containing the PREVIOUS pass's metrics. This is always
one-pass-behind data, which is consistent and non-paradoxical — the
current pass is still in progress and can't report its own metrics.

```rust
pub struct SchedulerMetrics {
    pub pass_count: u64,
    pub ema_pass_duration_ns: u32,   // EMA with 1/8 decay
    pub active_units: u16,
    pub stolen_count: u16,
    pub idle_ns: u32,
}
```

Per-unit metrics (EMA of execution time, execution count, max
duration) are also available via the Resource. The scheduler already
computes these internally for upward rank cost estimation — the
Resource just exposes them.

Metrics are always computed (no opt-in/opt-out). Cost: ~60ns per
unit per pass (two `Instant::now()` calls for timestamps). For 50
units: ~3µs per pass. 0.02% of a 16ms frame budget. The data the
scheduler already computes for its own cost estimation is simply
published as a Resource.

Consumers use metrics for adaptive cadence:
```rust
impl WorkUnit<On<CadenceCheck>> for AdaptivePacer {
    type Read = (Resource<SchedulerMetrics>,);
    fn execute(&self, ctx: Self::Ctx) {
        let metrics = ctx.resource::<SchedulerMetrics>();
        let saturation = metrics.saturation();
        if saturation > 0.8 {
            // slow down non-critical timers
        }
    }
}
```

EMA contracts (documented and stable):
- `ema = (ema * 7 + measured) / 8` (1/8 decay)
- `saturation = (pass_duration - idle_time) / pass_duration`

The write happens at pass start → it's the first thing in the DAG.
All readers naturally come after. The version stamp increments every
pass (pass_count always changes), so readers always dispatch.

**Q6. Access mode granularity — RESOLVED**

Resolved by Q2: two modes only. Read and Write. Write implies Read.

loimu's seven access modes (Pinned, Immediate, MutImmediate, Sample,
Post, Approx, Defer) control ordering, freshness, and determinism
within the double-buffered view system. These are loimu-specific
concerns tied to its temporal prediction, rotation groups, and
frame generation. polka and saalis need none of this.

hilavitkutin exposes Read and Write. The DAG uses these for edge
construction. The scheduler uses these for conflict detection. loimu
maps its seven modes to Read/Write at the hilavitkutin boundary and
implements the richer semantics in its own layer.

If hilavitkutin later needs a third mode (e.g. ReadExclusive for
consumers that need guaranteed exclusive read access), it can be
added without changing the WorkUnit trait — it would be a new store
wrapper type like `Exclusive<Resource<T>>` in the AccessSet.

**Q7. Storage abstraction — RESOLVED**

Resolved by Q1 + saalis team insights: hilavitkutin owns in-memory
columnar storage. Consumers handle persistence.

Hot/cold store architecture confirmed by saalis (SQLite cold store)
and loisto (DuckDB+LMDB cold store). loimu is all in-memory.
polka loads config once into memory, processes, writes output.

hilavitkutin provides: SlabColumn-backed storage for Resource<T>
(singleton) and Column<T> (collection). Virtual<T> has no storage.
The HasShape/ErasedColumn model from loimu supports type-erased
access for plugin types.

Consumers materialize external data into hilavitkutin columns before
processing. saalis loads query results from SQLite into Column<T>,
runs pipeline work units, writes results back to SQLite. The
bridge is saalis's concern, not hilavitkutin's.

No storage trait abstraction over databases. No pluggable storage
backend. One storage implementation: in-memory, contiguous,
fixed-stride, mmap-backed (with linked-page fallback for no-mmap
platforms).

**Q8. Registration and discovery — RESOLVED**

Consumers handle registration. hilavitkutin accepts work units via
explicit API calls on the scheduler builder. No `#[register]` macro.
No `inventory` dependency. No proc macros.

The WorkUnit trait is manually implementable. Each consumer uses its
own discovery mechanism:
- polka: `#[register]` via polka-derive + inventory
- saalis: Named + inventory
- loimu: `define_behavior!` macros + inventory

Consumers pass discovered work units to the scheduler:
```rust
let mut scheduler = Scheduler::new();
scheduler.add::<AddColumns>();
scheduler.add::<HttpFetcher>();
scheduler.add_on::<ResponseReady, ProcessResponse>();
scheduler.build();
```

The scheduler extracts Read/Write/Hint from the trait impls at build
time. DAG construction uses the associated types. No runtime
reflection, no TypeId, no string matching.

ECS-as-DI: Read/Write declarations ARE the dependency injection.
`Read<Resource<HttpClient>>` is equivalent to "inject HttpClient."
No service locator, no DI container, no factory.

**Q9. Scheduler extensibility — RESOLVED (emerged from research review)**

The scheduler self-hosts its own infrastructure as meta work units.
Plan computation, lane assignment, metrics, rollback — all expressed
as `WorkUnit<On<meta::*>>` impls operating on meta resources.

**Meta virtuals (scheduler lifecycle):**
- `meta::PlanStage` — fired when DAG changes (startup, plugin load).
  Gates plan computation work units.
- `meta::ScheduleReady` — fired when all plan-stage units complete.
  All consumer work units implicitly depend on this (scheduler adds
  the edge automatically). Consumers never declare it.
- `meta::PassStart` — fired at start of each pass. Gates metrics
  write and any per-pass infrastructure.
- `meta::ScheduleEnd` — fired after all consumer work completes.
  Gates epilogue work: error inspection, rollback, cleanup.

**Meta resources (scheduler internals, restricted access):**
- `Resource<meta::Dag>` — the DAG structure. Only readable by
  `WorkUnit<On<meta::PlanStage>>` impls.
- `Resource<meta::ExecutionPlan>` — computed plan (steps, ranks).
- `Resource<meta::LaneAssignment>` — lane assignments.
- `Resource<SchedulerMetrics>` — pass metrics (unrestricted read).

Consumer work units cannot access meta::Dag or meta::ExecutionPlan —
the type system prevents it. Only meta-gated impls can read scheduler
internals.

**Self-hosting means almost no hardcoded scheduler logic:**
1. DAG changes → fire `meta::PlanStage`
2. Plan-stage units complete → fire `meta::ScheduleReady`
3. Pass starts → fire `meta::PassStart`
4. Consumer units dispatch (gated by ScheduleReady + their own On<V>)
5. Consumer units complete → fire `meta::ScheduleEnd`
6. End-stage units complete → pass done

The default scheduler ships built-in meta work units (upward rank,
lane assignment, metrics). Consumers can replace or extend:

```rust
// custom plan-time DAG analysis
impl WorkUnit<On<meta::PlanStage>> for MyCustomAnalysis {
    type Read = (Resource<meta::Dag>,);
    type Write = (Resource<MyAnalysisResult>,);
    fn execute(&self, ctx: Self::Ctx) { /* custom DAG analysis */ }
}

// custom rollback at schedule end
impl WorkUnit<On<meta::ScheduleEnd>> for RollbackOnError {
    type Read = (Resource<ErrorLog>, Resource<SnapshotJournal>);
    type Write = (/* columns to restore */);
    fn execute(&self, ctx: Self::Ctx) {
        let errors = ctx.resource::<ErrorLog>();
        if errors.is_empty() { return; }
        // restore from journal
    }
}
```

**No rollback hardcoded in the scheduler.** Consumers who need rollback
(loimu with WASM plugins) build it from work units: error tracking
as a Resource, snapshot journals as a Resource, rollback logic as a
`WorkUnit<On<meta::ScheduleEnd>>`. Consumers who don't need it (polka)
pay nothing.

**Performance:** meta units are in the same DAG, dispatched the same
way. PlanStage units run only on DAG change (not per-pass). PassStart
and ScheduleEnd units run once per pass each (~nanoseconds). The
implicit ScheduleReady edge adds one virtual flag check per consumer
unit per pass. Zero meaningful overhead.

**The irreducible kernel.** The meta-scheduler cannot fully self-host —
a small hardcoded kernel builds and dispatches the meta DAG. This
kernel uses the same DAG abstraction as the consumer side (same code,
same types), but its execution is not configurable. It is the
"compiler" for the consumer schedule:

- Builds the meta DAG from meta work unit Read/Write declarations
- Dispatches meta units in topological order
- Fires ScheduleReady when done
- Panics in meta units are fatal (like compile errors — the schedule
  cannot be built if plan-stage units fail)

The kernel is ~50 lines of dispatch logic using the shared DAG
abstraction. Everything else (rank computation, lane assignment,
metrics, rollback, column layout analysis) is pluggable via meta
work units.

**Meta resource storage:** meta resources (meta::Dag, meta::ExecutionPlan,
ColumnLayoutMap, SchedulerMetrics) use fixed byte-aligned layout, not
the hint-driven layout computed by ColumnLayoutAnalysis. This avoids
circularity: the layout analysis output configures consumer column
layout, not its own storage. Two tiers: fixed-layout for meta
resources, hint-driven for consumer columns.

**Meta namespace restriction:** a marker trait `MetaAccess` on meta
resource types. Context<R, W> only provides access to meta resources
when the WorkUnit impl is gated on a `meta::*` virtual. ZST marker,
compile-time bound, zero runtime cost.

**Q10. Scheduler memory budget and column layout — RESOLVED
(emerged from Cilk space bound + hint-driven layout discussion)**

**Scheduler memory is pre-computable and bounded.**

Cilk proved O(S1 × P) space for work-stealing (S1 = serial depth,
P = workers). hilavitkutin's situation is tighter: DAG is static,
work units don't recurse. Memory bound is known at plan time:

Per worker lane:
- Ready deque: bounded by max step size (5-20 entries)
- Current unit metadata: 1 entry
- Version stamp cache: 2-5 u64 values (stores accessed by current unit)

Per scheduler:
- ExecutionPlan: steps × nodes. 50 units = ~400 bytes.
- Virtual flag array: ~12 bytes for 64 virtuals × 1.5 consumers.
- Cost estimates: 50 × 4 bytes = 200 bytes.
- Lane assignments: 50 bytes.

Typical 50-unit DAG with 8 workers: under 4KB total scheduler
metadata. One page. Pre-allocatable at plan time. No dynamic
allocation during passes. Fits in L1.

**Column layout derived from scheduling hints (meta work unit).**

A meta::PlanStage work unit analyzes the DAG to determine column
storage layout based on the scheduling hints of consuming work units:

```rust
impl WorkUnit<On<meta::PlanStage>> for ColumnLayoutAnalysis {
    type Read = (Resource<meta::Dag>,);
    type Write = (Resource<ColumnLayoutMap>,);
    fn execute(&self, ctx: Self::Ctx) { /* analyze + write layout */ }
}
```

Layout decision per column:

| Hottest consumer hint | Urgency | Layout |
|---|---|---|
| Critical + Immediate | any | ByteAligned (fastest SIMD) |
| Important + Immediate | any | ByteAligned |
| Normal + Steady | any | Natural (size_of::<T>()) |
| Opportunistic + Relaxed | any | BitPacked (densest) |
| Optional + Deferred | any | BitPacked |

The hottest consumer determines layout: if ANY reader is Critical,
byte-align. Cold readers don't suffer (byte-aligned is never slower
to read than bit-packed, just uses more memory).

Where bit-packing matters most:
- Boolean columns (Bool = UFixed<1,0>): 8× density (1 bit vs 1 byte)
- Sub-byte fixed-point types: 2-4× density
- Regular types (u32, Vec3): no difference (already byte-aligned)

Layout computed at plan time only. Recomputed on DAG change. The
storage engine reads ColumnLayoutMap to configure each column's
backing SlabColumn. Consumer columns are reallocated with the new
layout if it changed.

**Q11. Platform tiers, no_std, and intrinsics — RESOLVED**

**Three platform tiers:** `no_os`, `os` (default), `std`.

| Tier | Target | Memory | Threads | Clock | Panics |
|---|---|---|---|---|---|
| `no_os` | bare metal, WASM | Consumer provides | Consumer provides | Consumer provides | abort |
| `os` (default) | Linux, macOS | raw mmap / extern "C" | raw clone / pthread | raw clock_gettime | abort |
| `std` | Windows, other | std::alloc | std::thread | std::time::Instant | catch_unwind |

```toml
[features]
default = ["os"]
os = []   # linux/macos: raw syscalls, extern "C", hand-tuned
std = []  # windows/other: std abstractions, catch_unwind
# no feature = no_os: consumer provides everything
```

`os` and `std` are mutually exclusive. `os` is where we invest
optimization effort. `std` is the zero-effort fallback via Rust's
abstractions. `no_os` is trait contracts only.

**No alloc crate.** hilavitkutin does not use Vec, Box, String, or
any heap allocator. All dynamic storage goes through SlabColumn
(mmap on `os`, std::alloc on `std`, consumer-provided on `no_os`).
Fixed-size structures use stack arrays, arrayvec, or SlabColumn.

The DAG itself is stored in SlabColumn — the system eats its own
cooking. Ready deques are fixed ring buffers sized at plan time.
Work unit storage is typed SlabColumns per concrete type (no Box<dyn>).

**Raw syscalls on `os` tier.** mmap, munmap, mprotect via extern "C"
(macOS, linked against libSystem which is always present) or inline
asm (Linux, stable syscall ABI). No libc dependency. No std.

```rust
// macOS: extern "C" against libSystem (always linked)
extern "C" {
    fn mmap(addr: *mut u8, len: usize, prot: i32, flags: i32,
            fd: i32, offset: i64) -> *mut u8;
}

// Linux aarch64: raw syscall
unsafe fn mmap_anon(len: usize) -> *mut u8 {
    let addr: usize;
    core::arch::asm!("svc #0",
        in("x8") 222, in("x0") 0, in("x1") len,
        in("x2") 3, in("x3") 0x22, in("x4") !0usize,
        in("x5") 0, lout("x0") addr,
    );
    addr as *mut u8
}
```

**Platform abstraction traits (always available, any tier):**
```rust
pub trait MemoryProvider: Send + Sync {
    unsafe fn allocate(&self, len: usize, align: usize) -> *mut u8;
    unsafe fn deallocate(&self, ptr: *mut u8, len: usize);
    unsafe fn protect(&self, ptr: *mut u8, len: usize,
                      read: bool, write: bool);
}

pub trait ThreadPool: Send + Sync {
    fn spawn(&self, f: impl FnOnce() + Send + 'static);
    fn worker_count(&self) -> usize;
}

pub trait Clock: Send + Sync {
    fn now_ns(&self) -> u64;
}
```

Each tier ships default impls. Consumer overrides via scheduler
builder:
```rust

    .memory(MyCustomMemory)
    .threads(MyCustomThreadPool)
    .clock(MyCustomClock)
    .build();
```

**core::intrinsics usage (throughout, not just arvo):**

hilavitkutin uses core::intrinsics aggressively. Requires nightly
(already a project requirement).

Numeric (arvo):
- `sqrtf32`, `sqrtf64` — hardware sqrt
- `fmaf32`, `fmaf64` — fused multiply-add
- `fabsf32`, `fabsf64` — absolute value
- `minnumf32`, `maxnumf32` — min/max
- `ceilf32`, `floorf32` — rounding
- `copysignf32` — sign copy

Scheduler and storage:
- `likely` / `unlikely` — branch hints in dispatch loop
- `prefetch_read_data` — explicit prefetch in morsel iteration
- `ctlz` / `cttz` — leading/trailing zeros for virtual flag scan
- `copy_nonoverlapping` — fast memcpy for SlabColumn ops
- `write_bytes` — fast memset for virtual flag reset

All unsafe. All justified by performance-critical hot paths. All
covered by property-based tests and Miri verification.

**Unsafe surface (documented, tested):**
- core::intrinsics for math and hardware ops
- Raw pointer arithmetic in SlabColumn
- MaybeUninit for fixed-size morsel buffers
- Inline asm for syscalls (os tier)
- extern "C" for libSystem calls (macOS os tier)

**Q12. Platform microkernels — RESOLVED**

Day-one asm microkernels for operations where LLVM cannot emit the
optimal instruction or where our design knowledge exceeds LLVM's
analysis. Written as `#[cfg(target_arch)]` + `core::arch::asm!`,
no trait indirection, inline(always).

| Microkernel | aarch64 | x86_64 | Rationale |
|---|---|---|---|
| Cache line zero | `dc zva` | write_bytes fallback | flag reset — LLVM never emits dc zva |
| Paired load/store | `ldp`/`stp` | not needed (OOO) | version stamp batch compare |
| LSE atomics | `cas`/`swp`/`ldadd` | `lock cmpxchg` (LLVM) | deque ops — no spurious LL/SC failure |
| Explicit prefetch | `prfm` | `prefetcht0` | morsel dispatch — LLVM can't see access pattern |
| Non-temporal store | `stnp` | `movntdq` | morsel output — avoid cache pollution on writer |
| Memory fences | `dmb ishld`/`ishst` | not needed (TSO) | deque — minimum fence strength |
| Worker parking | `wfe`/`sev` | `pause` + futex | idle workers — low-power wait |
| Trailing zeros | `rbit`+`clz` (native) | `tzcnt` (skip zero check) | virtual flag scan — branch elimination |

Design principle: write idiomatic Rust first. Profile. Replace with
asm where measurement confirms the win. The above table lists ops
where we're confident enough to write asm from day one, based on
known LLVM limitations or guaranteed instruction-level wins.

Additional asm microkernels added by profiling later — the
`#[cfg(target_arch)]` infrastructure is in place from day one.

Fallbacks: every microkernel has a Rust fallback for unsupported
platforms. The fallback uses `core::intrinsics` where possible,
plain Rust otherwise. The `std` tier always uses the Rust fallback.

**Plan-phase optimizations (data structure and computation):**

**Bitmask access sets (core design decision).** Every StoreId maps to
a bit position. Every WorkUnit's Read and Write set is a fixed-width
bitmask (`u64` for ≤64 stores, `[u64; 4]` for ≤256 stores). All
scheduler operations on access sets become single-instruction bitwise
ops:

- Conflict detection: `(write_a & read_b) != 0` — one AND
- Dependency edge construction: same AND, per pair
- Access set union: `a | b` — one OR
- Access set difference: `a & !b` — one AND NOT
- Empty check: `a == 0` — one comparison
- Intersection for lane conflict: `(node_write & lane_access) != 0`

This eliminates HashSet, Vec<StoreId>, sorted arrays, and all
associated allocation/iteration overhead from the scheduler's core
algorithms. Plan computation, lane assignment, conflict checking,
and DAG edge construction all operate on raw integers.

Bit position assignment is computed once at plan time. StoreId to
bit index is a const lookup table. For typical usage (30-60 stores),
everything fits in a single u64.

**Node renumbering for sequential rank access.** After topological
sort, renumber nodes so their IDs match reverse topological order.
The upward rank traversal (bottom-up) then accesses the rank array
sequentially. Hardware prefetcher kicks in. Without renumbering, the
traversal accesses `rank[successor_id]` which is random.

Zero runtime cost — the renumbering is a permutation computed once
at plan time. Node IDs are internal (consumers use their WorkUnit
type, not a numeric ID). The scheduler maps types to internal IDs.

**Vectorized EMA batch update.** All unit cost EMAs are stored in a
contiguous `[u32; N]` array. The EMA formula `ema + (measured - ema)
>> 3` is applied to all N units per pass. With NEON (4×u32) or AVX2
(8×u32): N/4 or N/8 vector iterations instead of N scalar iterations.
For 50 units: 7 AVX2 iterations. The array is already in L1 (200
bytes). Saves ~100ns per pass. Written from day one as a clean
demonstration of the vectorized batch pattern.

**Vectorized DAG change detection.** Every pass: compare all unit EMAs
against their plan-assumed thresholds to detect if replan is needed.
`any(ema > threshold * 2)` across the contiguous EMA array. Same
NEON/AVX2 pattern as the EMA update: vectorized compare with early
exit. Written from day one.

****Non-obvious data structure and representation choices:**

These are choices that look minor individually but compound across
the entire scheduler. Each eliminates a class of overhead (hashing,
indirection, branching, cache misses) from a frequently-executed path.

**11. Dense u16 index for work unit dispatch.** No TypeId, no HashMap
in the hot path. Each concrete WorkUnit type gets a dense u16 index
at plan time. The dispatch table is a flat array `[fn(); N]`. Dispatch
is one array index. Same pattern as StoreId → bit position.

**12. Bit-matrix DAG adjacency (≤64 nodes).** The DAG topology is a
flat bit array: `[u64; N]` where row i has bit j set if edge i→j
exists. Edge check: one bit test. Successor scan: iterate set bits
in one u64 word (using trailing_zeros). Predecessor scan: test bit j
in each row (column scan). For ≤64 nodes: 512 bytes, 8 cache lines.
The entire DAG topology fits in L1.

For >64 nodes: `[u128; N]` or `[[u64; K]; N]` with K = ceil(N/64).
For >256 nodes: compressed sparse row (CSR). But our three consumers
won't hit this.

**13. Step ranges from contiguous renumbered nodes.** Since nodes are
renumbered by reverse topological order, nodes in the same step have
contiguous indices. A step is `(start: u16, count: u16)`. The step
list is `[(u16, u16); S]`. Iterating a step's nodes is iterating a
contiguous slice. Zero indirection, no Vec<Vec<NodeId>>.

**14. Parallel version stamp arrays (SoA layout).** "Before" stamps
and "current" stamps stored as two separate contiguous `[u64; N]`
arrays, not interleaved `[(u64, u64); N]`. Enables vectorized
comparison: load 4 befores (NEON) or 8 befores (AVX2), load matching
currents, compare in one instruction. The comparison loop touches
two contiguous arrays — perfect prefetch.

**15. Contiguous consumer bits per virtual.** At plan time, assign
consumer bit positions per virtual contiguously. Virtual V's consumers
are at positions `[base..base+count]` in the flag array. "Fire V" is
one byte/word write covering all V's consumers. No per-consumer
lookup, no iteration. The flag array is organized by virtual, not by
consumer.

**16. Pre-partitioned morsel ranges (no atomics).** At plan time,
partition morsels across workers: worker 0 gets `[0..12]`, worker 1
gets `[12..24]`, etc. Each worker walks its range sequentially. No
atomic `fetch_add` for morsel claiming. No contention. If a worker
finishes early, it steals morsel ranges from other workers (same
work-stealing pattern, morsel granularity). The partitioning is a
plan-time `[(start, end); P]` array.

**17. Const morsel size from associated types.** The work unit's
Read/Write associated types are known at compile time. Morsel size is
`L1_CACHE_SIZE / (N_columns * size_of::<T>())` — a const expression.
LLVM sees a known trip count in the inner loop. Enables full unroll
or vectorization without runtime morsel size calculation.

**18. Epoch-based virtual flag reset.** Instead of memset every pass,
increment a global epoch u8. "Flag is set" means `flag_value ==
current_epoch`. "Fire" writes current_epoch to the flag byte. "Check"
compares against epoch. Reset is one u8 increment, not a memset.
After 255 passes: one real clear (memset). Eliminates per-pass memset
for the common case where most virtuals don't fire.

**19. Hot/cold metadata split.** Per-work-unit metadata split into hot
(accessed every dispatch: EMA cost, lane, rank, step index — 11 bytes)
and cold (accessed at plan time only: name, access bitmasks, hints —
35 bytes). 50 units × 11 bytes = 550 bytes = 9 cache lines for all
hot metadata. During dispatch, only hot metadata is touched. Cold
metadata never pollutes L1 during the pass.

**20. Power-of-two deque with AND wrap.** Per-lane ready deque sized
as power of two. Index wrap: `index & (SIZE - 1)`. One AND instead
of modulo. Typical: 32 slots (2^5), wrap `index & 31`.

**21. Lane assignment as bitmask.** Lane stored as `1 << lane_id`
(a u8 bitmask), not as a u3 index. "Is this unit on my lane?" becomes
`(unit.lane_bit & my_lane_bit) != 0`. One AND. Same for steal check:
`(unit.lane_compat & my_lane_bit) != 0`. Already a bitmask AND.

**22. Const-generic scheduler sizing.** Internal arrays sized by const
generics:
```rust
pub struct Scheduler<
    const MAX_UNITS: usize = 256,
    const MAX_STORES: usize = 64,
    const MAX_LANES: usize = 8,
> { /* all arrays sized by consts, no heap */ }
```
Consumer chooses caps at build time. LLVM optimizes loops with known
bounds. Embedded consumers shrink for memory savings. Default covers
typical usage.

**23. Per-work-unit version stamp packing.** Pre-compute a "stamp
view" per work unit: only the stamps of stores this unit accesses,
packed contiguously. Work unit A accesses stores 0, 3, 7 → its stamp
view is `[stamp_0, stamp_3, stamp_7]` packed into one cache line.
The "capture before" and "compare after" operations touch one cache
line per work unit, not scattered stamps across the full array.

**24. Lane ID as function parameter, not TLS.** Each worker thread
receives its lane ID at creation. Every function in the dispatch path
takes `lane: u8` as an argument. Lane ID lives in a register for the
entire dispatch loop. No TLS lookup, no segment register access.
Works in no_std (no TLS available).

**25. Dirty propagation as bitmask OR chain.** Forward sweep through
the DAG for dirty propagation: for each node N, check if any
predecessor is dirty. With bit-matrix DAG: `(predecessors[N] &
dirty_mask) != 0`. One AND + compare per node. Entire sweep for 50
nodes: 50 AND operations on u64 values. ~25ns. No branches except
"if dirty, mark self." The dirty_mask is updated with OR as nodes
are marked.

**26. Instruction cache grouping for execute().** Hot execute()
functions grouped into `.text.hot.execute` link section via
`#[link_section]` or linker script. With LTO (ExpandedLto pragma),
the linker sees all execute functions. Grouping keeps dispatch
targets in adjacent instruction cache lines. This is a
hilavitkutin-build concern, not core, but connects build optimization
to scheduler dispatch.

**Deferred to profiling (not day one):****

Vectorized in-degree scan (Kahn's algorithm): scanning the in-degree
array for zero entries could use NEON/AVX2 compare-against-zero.
NOT day one because Kahn's runs only at plan time (startup + DAG
change), which is infrequent. The 50-element scalar scan takes
nanoseconds. Vectorizing saves ~20ns on an operation that runs
once per DAG change. Not worth the complexity until profiling shows
plan computation is a bottleneck (unlikely for <1000 work units).

Vectorized affinity matrix construction: the virtual flag co-fire
affinity matrix (N×N, used for bit packing order) could use SIMD
for pairwise score updates. NOT day one because N is small (20-64
virtual types), the matrix fits in L1, and the computation runs
only at plan time. Standard scalar code is clear and fast enough.
Revisit if virtual type counts grow significantly or if plan
recomputation becomes frequent.

**Allocator recommendation (hilavitkutin-build pragma):**

The consumer sets `#[global_allocator]` for their own heap
allocations. hilavitkutin core never calls the global allocator.
hilavitkutin-build provides a pragma for convenience:

MimallocAllocator pragma: adds mimalloc dependency, generates
`#[global_allocator]` setup. Recommended for morsel-heavy workloads
(batch alloc/dealloc pattern, low fragmentation, thread-local
free lists).

## Notes

The build configuration topic (202603131230) designed hilavitkutin-build
as the build optimization layer. This topic designs hilavitkutin itself
— the runtime crate. The two are independent: hilavitkutin-build
optimizes how hilavitkutin (and everything else) is compiled;
hilavitkutin defines what executes at runtime.

The arvo crate (numeric primitives) is orthogonal. Column data types
come from arvo; hilavitkutin doesn't care what's in the columns, only
the read/write dependency structure.
