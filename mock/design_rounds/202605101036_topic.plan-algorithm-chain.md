### P1. Canonical 12-step chain inherited from polka-dots

Per `~/Dev/polka-dots/mock/crates/hilavitkutin/DESIGN.md.tmpl` lines 36-50, the chain is:

1. **Build DAG** from AccessMask overlap (arvo-bitmask).
2. **Topological sort** + node renumbering (arvo-graph).
3. **Upward rank** → critical path (arvo-graph).
4. **Waist detection** → phase boundaries (arvo-graph).
5. **RCM reordering** → fiber grouping order + arena layout (arvo-sparse).
6. **Block diagonal + D-M** → phase validation + dead column elimination (arvo-sparse).
7. **Spectral partitioning** → trunk formation when >5 fibers (arvo-spectral).
8. **Fiber grouping**: greedy (arvo-comb) or matrix chain DP for >10 ops (arvo-comb).
9. **Per-fiber morsel sizing** (write-only formula, domain 12).
10. **Per-phase adaptive configs** (MAX_FUSE / BALANCED / MAX_SPLIT).
11. **Column classification** per fiber (internal / input / output).
12. **Dirty propagation masks** for incremental skip (arvo-bitmask).

Each step's substrate dependency is named. Steps 5-7 are the sparse-matrix optimisations that distinguish hilavitkutin from loimu's simpler list-scheduling-with-upward-rank approach (see `~/Dev/polka-dots/mock/design_rounds/202603141800_research.scheduler-algorithm-deep-dive.md` for the deep dive comparing approaches).

### P1.5 Pass-fusion: steps 3 and 12 share one backward DAG traversal

Both step 3 (upward rank computation) and step 12 (dirty propagation masks) walk the DAG bottom-up from leaves. Step 3 computes `rank(u) = cost(u) + max(rank(successors(u)))`; step 12 computes `dirty[j] |= dirty[i]` for every edge i→j. These traversals are independent in their data dependencies but share their DAG-walk order.

**The chain runs steps 3 and 12 as a single fused backward pass.** One walk over the topologically-reversed UnitId order; the inner loop body reads each unit's successors once, computes both `upward_rank` and `dirty_mask` updates in the same iteration. Cost: one O(U + E) traversal instead of two.

The 12-step chain is conceptually presented as 12 ordered steps for design clarity (each step has a clear input/output contract and a citable arvo-crate dependency). At the implementation layer, fusion of disjoint-data steps with shared traversal order is mandatory wherever it preserves the per-step contracts. The src CL records each free-function step's signature; the runner that composes them is allowed to inline + fuse where the data dependencies permit.

Other potential fusions surfaced during step-by-step audit:
- Step 5 (RCM) and step 6 (block-diag + D-M) operate on the same access matrix and produce orderings + classifications that can be computed during overlapping passes; sketch will validate whether fusion is profitable or whether the algorithms' internal structure makes them better as separate passes.
- Step 11 (column classification) reads from step 5/6/7 outputs and could fuse with step 12 if the classification fold can run during the same backward sweep that computes dirty masks. Sketch validation needed.

The src CL bench harness (per round task) measures each step in isolation AND end-to-end; fused passes win the end-to-end measurement when bench-validated.

### P2. Plan stage is pure (no I/O, no allocation at plan-fire time)

Plan computation reads WU declarations + cost estimates and writes an `ExecutionPlan`. No filesystem, no syscalls, no heap allocation **at plan-fire time**. Const-generic backing structures throughout per `#![no_std]` + `no_alloc` discipline.

**Plan-stage scratch buffer.** The 12 intermediate analytical types (AccessMatrix, AccessDag, TopoOrder, UpwardRanks, WaistMap, RcmReorder, BlockDiagonalisation, FiberGrouping, SpectralClusters, TrunkComponentAssignment, CoreAssignment, plus column-classification and dirty-mask scratch) total ~30-50 KB at typical sizes (`MAX_UNITS = 256`, `MAX_COLUMNS = 256`). Allocating this from the call stack at every plan-fire risks TLB refill cost on the column arenas afterward. Instead: the scheduler builder allocates a `PlanScratch` region from the consumer's `MemoryProvider` at `.build()` time, sized for the const-generic upper bounds; plan-stage runs walk this pre-allocated scratch in place. The allocation happens once at build, not per-plan-fire. `no_alloc` invariant holds at the `run()` hot path; the build-time allocation is on the same path as resource arena allocation and uses the same `MemoryProvider` interface.

### P2.5 Arena alignment invariants (μarch-required)

The plan-stage data shape commits to alignment guarantees that codegen and SIMD autovectorization rely on. These are part of the `MemoryProvider` contract, not just the plan-stage layout.

1. **Fiber arena base pointers are 64-byte aligned.** Every column arena base address returned by `MemoryProvider::alloc_arena(...)` is aligned to 64 bytes (the largest cache-line size across target architectures: Apple Silicon 64 B / 128 B for some L2 shapes; x86-64 64 B; common pages 4 KB → 64-byte alignment is a guarantee, not a request). The `& !3` morsel-count formula does NOT establish base-pointer alignment; only an aligned allocator does. The plan-stage's `Fiber.arena_size` and `Fiber.column_offsets` assume this alignment; codegen relies on it for aligned NEON/AVX loads.

2. **Plan-stage scratch is also 64-byte aligned.** The `PlanScratch` region (P2 above) is aligned identically. Step-5 RCM operates on the access matrix in scratch; alignment lets the bitmask operations vectorize cleanly.

3. **Per-core program data lives in NUMA-local memory.** See the NUMA invariant under platform constraints below.

### P3. Plan recomputes on structural change only

Per the locked axis 4 (Topic 1), `run()` auto-recomputes when the dirty flag is set. Dirty triggers are: WU set change, record-count change, plan-affecting Replaceable mutation, extension load/unload. Per-frame data mutation does NOT trigger recompute.

### P4. Microsecond cost target

Polka-dots' design assumed plan recompute completes in microseconds for typical pipelines (20-50 nodes, 30-100 edges). Const-generic bitmask structures + arvo-graph's CSR-shaped storage make this achievable. Larger pipelines (clause compiler with 100+ WUs) should still complete in low milliseconds.

### Axis B: ExecutionPlan data-structure shape

**DECIDED.** Final `ExecutionPlan` is shaped after the canonical runtime hierarchy `pipeline → core → phase ↔ waist → trunk → fiber ↔ branch ↔ bridge → morsel → micro-morsel → record` (vocabulary table at line 74 of the consolidation topic). #### Final ExecutionPlan shape

**Const generics: ten total** (corrects two prior undercounts; `MAX_UNITS_PER_FIBER`, `MAX_COLUMNS_PER_FIBER`, and `MAX_TRUNKS_PER_PHASE` are real parameters consumed by nested types). Per M9, three of the ten are derivable from the others via the `CeilingDiv` helper trait (see "Const-generic arithmetic via CeilingDiv" below); listing them all explicitly avoids `generic_const_exprs` arithmetic-expression-in-array-size hazards.

```rust
pub struct ExecutionPlan<
    const MAX_UNITS:               usize,  // 1 WorkUnit registration = 1 UnitId
    const MAX_PHASES:              usize,
    const MAX_TRUNKS:              usize,  // global; sparsely distributed across phases
    const MAX_FIBERS:              usize,  // global
    const MAX_LANES:               usize,  // = MAX_CORES (one lane per physical core)
    const MAX_COLUMNS:             usize,
    const MAX_COMPONENTS_PER_TRUNK: usize, // fibers + branches + bridges per trunk
    const MAX_UNITS_PER_FIBER:     usize,  // = CeilingDiv<MAX_UNITS, MAX_FIBERS>::RESULT
    const MAX_COLUMNS_PER_FIBER:   usize,  // = CeilingDiv<MAX_COLUMNS, MAX_FIBERS>::RESULT
    const MAX_TRUNKS_PER_PHASE:    usize,  // = MAX_TRUNKS (worst case: all trunks in one phase)
> {
    /// Phase sequence. Phases overlap via pipeline parallelism: phase N+1 starts when
    /// phase N produces enough records for one morsel.
    phases: [Phase<MAX_TRUNKS_PER_PHASE, MAX_COLUMNS>; MAX_PHASES],
    phase_count: USize,

    /// Per-UnitId structural metadata. ONE entry per .with(MyWU)-registered WorkUnit.
    /// Carries derived structural-position fields only (fiber, trunk, phase, core, lane_compat,
    /// hint, depth, upward_rank). Mutable runtime cost data lives in CostTable, not here.
    units: [UnitMeta<MAX_LANES>; MAX_UNITS],
    unit_count: USize,

    /// Per-fiber detailed payload.
    fibers: [Fiber<MAX_UNITS_PER_FIBER, MAX_COLUMNS_PER_FIBER>; MAX_FIBERS],
    fiber_count: USize,

    /// Per-trunk payload (component list, core pinning, spectral cluster).
    trunks: [Trunk<MAX_COMPONENTS_PER_TRUNK, MAX_LANES>; MAX_TRUNKS],
    trunk_count: USize,

    /// Step-12 output: per-store dirty propagation masks for incremental skip.
    /// Computed in the same backward pass as upward rank (step 3); see step-3+step-12 fusion below.
    dirty_masks: DirtyMasks<MAX_UNITS>,

    /// RCM-derived column ordering (step 5). Columns co-accessed end up adjacent in arena.
    column_order: [ColumnId; MAX_COLUMNS],
    column_count: USize,

    /// Per-core projection. Built post-fiber-and-trunk-formation by the final synthesis step.
    /// Codegen (Topic 4) walks this directly. See axis F for the projection's compatibility contract.
    core_programs: [CoreProgram<MAX_PHASES, MAX_TRUNKS, MAX_FIBERS>; MAX_LANES],

    /// Per-phase strategy. **Derived quantity** (computed by step 10 from phase topology +
    /// record count); not consumer-configurable. See "Derived vs configured" note below.
    phase_strategy: [PhaseStrategy; MAX_PHASES],
}

/// Sibling to `ExecutionPlan`. Holds runtime-mutable cost estimates so `ExecutionPlan` itself
/// can stay immutable between structural recomputes. The plan stage initialises the heuristic
/// values (per Domain 15 step 3 cost table); the adapt subsystem (Topic 5 / C3) writes EMA
/// updates here without touching `ExecutionPlan`. Plan-stage purity invariant (P2) preserved.
pub struct CostTable<const MAX_UNITS: usize> {
    /// Initial heuristic per UnitMeta declaration:
    ///   base + per-write-column + per-emitted-virtual + morsel-dispatch-hint
    /// EMA-replaced after ~10 frames (Domain 22 EMA decay 1/8). Mutable.
    estimated_cost: [USize; MAX_UNITS],
    /// Per-unit sample count. Until 10, estimated_cost blends heuristic + EMA.
    sample_count:   [USize; MAX_UNITS],
}

pub struct Phase<...> {
    /// Trunks within this phase. Run in parallel on assigned cores.
    /// **Critical invariant from Domain 11:** trunks share NO write columns
    /// with other trunks within the same phase. Zero sync between trunks.
    trunks: [TrunkId; MAX_TRUNKS_PER_PHASE],
    trunk_count: USize,

    /// Step-10 output: per-phase config selected from phase topology.
    config: PhaseConfig,

    /// Step-4 output: waist boundaries delimiting this phase.
    waist_in:  WaistBoundary,
    waist_out: WaistBoundary,

    /// Step-11 output: per-column classification *within this phase*.
    /// Classes: FiberInternal, FiberInput, FiberOutput, FiberInternalAndOutput.
    column_class: ColumnClassMap<MAX_COLUMNS>,

    /// Pipeline-parallelism progress counter ID. The actual AtomicUsize lives at runtime
    /// (stack-allocated, raw-pointer-passed; per Domain 17 progress counters section).
    /// This slot just records the counter's identity for codegen wiring.
    progress_counter: ProgressCounterId,

    /// Single-trunk phase flag. When true, the phase is eligible for head+tail

pub struct Trunk<const C: usize, const L: usize> {
    /// **Components are peers within the trunk.** Fibers, branches, and bridges
    /// are all instances of `TrunkComponent` (per Domain 14 line 953).
    /// Branches chase the trunk's morsels; bridges fan in from multiple trunks.
    components: [TrunkComponent; C],
    component_count: USize,

    /// Step-7 output (post-fiber-grouping): spectral cluster identifier.
    /// Trunks within the same cluster share heavy column volume (kept on one core's L1).
    cluster: TrunkCluster,

    /// **Core pinning** (Domain 11 line 775): each trunk assigned to specific
    /// pool thread at plan time. Lane = core in this engine; one lane per physical core.

    /// the trunk boundary. Useful for adapt subsystem decisions.
    cut_weight: USize,

pub enum TrunkComponent {
    /// Primary path. Column co-location unit + morsel windowing unit.
    Fiber(FiberId),

    /// Side path attached to the trunk. Shares some columns with parent trunk's fibers.
    /// "Executes as chaser within trunk's morsel scope" (Domain 11 line 743-744).
    Branch(BranchId),

    /// Fan-in node reading from multiple trunks. Mandatory barrier point per Domain 14
    /// line 1090. Runs after parent trunks reach the required record range.
    Bridge(BridgeId),

pub struct Fiber<const U: usize, const C: usize> {
    /// UnitIds in this fiber, in execution order (RCM-reordered per step 5).
    units: [UnitId; U],
    unit_count: USize,

    /// Step-9 output: outer-loop morsel size from the Domain 12 formula, *with platform-aware
    /// SIMD-width alignment* (corrected from the canonical `& !3`):
    ///
    ///   morsel_size = ((L1_usable / Σ write_sizes).clamp(MIN, MAX))
    ///                 & !(SIMD_LANE_ELEMENTS_FOR(write_element_kind) - 1)
    ///
    /// Where L1_usable = detected_L1 × 0.75, MAX_MORSEL = 8192, and SIMD_LANE_ELEMENTS_FOR
    /// resolves at codegen time per platform + write column element kind:
    ///   NEON 128-bit  + u32  → 4 lanes  (SIMD_LANE_ELEMENTS = 4)
    ///   AVX2 256-bit  + u32  → 8 lanes
    ///   AVX-512 + u32  → 16 lanes
    ///   NEON 128-bit  + u64  → 2 lanes
    ///   AVX-512 + u64  → 8 lanes
    ///
    /// The original `& !3` (4-record alignment) is correct for NEON 128-bit u32/u64 but
    /// produces a scalar tail for AVX-512 u32 (16-lane wide stores). Aligning to the largest
    /// relevant SIMD width for the target produces a clean autovectorised body with NO
    /// scalar-tail epilogue per morsel.
    morsel_size: USize,

    /// Step-9 output (conditional): inner-loop micro-morsel size.
    /// Active when peak_live > L1 (Domain 12 line 850); otherwise `Maybe::Isnt`.
    /// peak_live = max(op.read ∪ op.write ∪ prev.write) at any Z-slice.
    micro_morsel_size: Maybe<USize>,

    /// 3D morsel cube data (Domain 12 line 854):
    /// - X = records (windowed by morsel_size)
    /// - Y = active columns per op (this field)
    /// - Z = op index within fiber (= unit_count above)
    /// - T = temporal across frames (implicit; schedule reuses across frames per A5)
    active_columns_per_op: USize,

    /// Co-located arena layout (Domain 14 lines 959-976).
    /// All columns within a fiber share one arena allocation. Type-native stride per column.
    arena_size: USize,
    column_offsets: [ColumnOffset; C],
    column_offset_count: USize,

    /// COMMUTATIVE flag (Domain 14 line 1173). Fiber is commutative iff ALL
    /// constituent WUs are commutative. Enables head+tail convergence + multi-core
    /// morsel splitting; resource accumulation uses per-thread stack-local accumulators
    /// merged after convergence.
    commutative: Bool,

    /// Step-9b: head+tail convergence support. Populated when:
    ///   commutative AND fiber is sole trunk component AND single_trunk_phase
    ///   AND record_count meets threshold per Domain 21.
    head_tail: Maybe<HeadTailConvergence>,

    /// Dispatch approach (Domain 17 line 1558):
    ///   <10K records:  IndirectPerFiber (Approach C) or TrunkMega (Approach D)
    ///   >10K records:  ScheduleMega (Approach E)
    /// Local `&[WuFn]` slice contents are baked at codegen time; this enum
    /// just records which structural shape codegen materialises.
    dispatch_approach: DispatchApproach,
}
```

The shape mirrors the canonical hierarchy directly. Dispatch codegen (Topic 4 / C2) walks ExecutionPlan as the input; the `core_programs` projection gives codegen a per-core view without re-walking the global structure.

#### Derived vs configured: PhaseStrategy, PhaseConfig, DispatchApproach

`PhaseStrategy`, `PhaseConfig`, and `DispatchApproach` are **derived quantities**, not consumer-configurable knobs. The plan-stage chain produces them:

- **`PhaseConfig`** (`MaxFuse | Balanced | MaxSplit`) is computed at step 10 from each phase's fiber-grouping output: `fused_cols < 50% max_cols → MaxFuse`, `50-100% → Balanced`, `> max_cols → MaxSplit`.
- **`PhaseStrategy`** (`Sequential | Phased | PhasedConvergence | PipeChase | ChaseSteal | Adaptive`) is computed at step 10b from each phase's record count + topology: `record_count < 10K → Sequential`, single-trunk + commutative + threshold met → `PhasedConvergence`, multi-trunk wide phase → `Phased`, etc.
- **`DispatchApproach`** is computed at step 10c per fiber from record count + fiber count: `<10K → IndirectPerFiber/TrunkMega`, `>10K → ScheduleMega`.

These three enums are **read-only** post plan stage. Adapt (Topic 5) may dirty the plan and trigger recompute, which produces fresh values. Adapt does NOT directly write into `Phase.config` or `Phase.phase_strategy`. The flow is one-way: plan-stage chain → derived enums → consumed by codegen.

This eliminates the surface for inconsistent combinations (e.g. `Phased + MaxFuse` when only `PhasedConvergence + MaxFuse` makes sense). The combinations a consumer can observe are exactly the combinations the plan-stage chain produces; nothing else is reachable.

#### TrunkComponent dispatch invariant for codegen

`TrunkComponent { Fiber | Branch | Bridge }` is an enum at the data-structure level so plan-stage code reasoning about a trunk's structure can match on the variant uniformly. **At the codegen level, the variant must NOT appear in the inner record loop.** The contract:

- Codegen walks `trunk.components` once at outer scope to enumerate which components exist.
- For each variant present, codegen emits a SEPARATE loop body specialised to that variant: the Fiber loop body, the Branch chase body, the Bridge fan-in barrier body.
- The inner record loop never matches on `TrunkComponent`. The variant has been resolved at codegen time; the inner loop dispatches to a single specialised body via direct call (or inlined under `#[inline(always)]`).

A trunk that contains a mix of Fiber, Branch, and Bridge components produces multiple loop bodies in codegen, dispatched in order. The plan-stage data shape never forces a runtime branch on the variant. This is the invariant that lets the ASM verification checklist (zero `blr` in inner loop, no mispredicted match) hold.

Plan stage thus carries the enum freely as data; codegen is responsible for materialising it as separate specialised bodies. Topic 4's design conversation will re-affirm the invariant.

#### Intermediate types (crate-private)

Steps 1-12 produce intermediate analytical artefacts that feed the next step but do not belong in the final ExecutionPlan. They live as `pub(crate)` types in `plan::analysis::*` submodules. Each cites the canonical Domain (15 = analysis algorithms, 14 = fiber formation, 11 = phase decomposition):

```rust
    /// Write mask per WU: bit j set if WU i writes store j. Built from `type Write`.
```

The plan-stage runner is a single function `compute_execution_plan` that chains them in order:

```rust
pub fn compute_execution_plan<...>(
    units: &[...],
    costs: &[USize; MAX_UNITS],
    cfg: &impl RunCfg,
) -> Outcome<ExecutionPlan<...>, PlanError> {
    let dag       = build_dag(...);
    let topo      = topo_sort(&dag);
    if topo.cycle_detected.as_bool() {
        return Outcome::Err(PlanError::Cycle);
    }
    let waists    = compute_waists(&dag, &topo);
    let rcm       = rcm_reorder(&dag, &waists);
    let block     = block_diagonalise(&dag, &waists, &rcm);
    if !block.dm_validation.is_ok() {
        return Outcome::Err(PlanError::PhaseValidation);
    }
    let spectral  = spectral_partition(&dag, &block);
    let fibers    = group_fibers(&dag, &topo, &spectral);

    // P1.5 fusion: step 3 (upward rank) + step 12 (dirty propagation masks) share
    // ONE backward DAG traversal. Single free function returns both outputs.
    // The fusion happens at the function boundary — the runner does NOT call them
    // separately. Algorithm: walk topo.order in reverse; for each unit, compute
    // rank update from successors AND dirty mask OR from predecessors in the same loop.
    let (ranks, dirty) = compute_upward_rank_and_dirty(&dag, &topo, costs, &fibers);

    let morsels   = size_morsels(&fibers, cfg.record_count());
    let configs   = select_phase_configs(&waists, &fibers);
    let columns   = classify_columns(&fibers, &dag);

    Outcome::Ok(assemble_plan(/* all of the above */))
}
```

Why free functions over typestate-threaded builder:
1. **Each step is independently testable.** Free functions take fixed inputs, return fixed outputs. Unit tests pass synthetic inputs without typestate ceremony.
2. **Step bodies are flat.** A typestate-threaded builder pushes step-N's logic into a method on `Plan<AfterStepN-1>`; the body becomes lost in the typestate noise. Free functions stay readable.
3. **Per-step bench harnesses are trivial.** Each step is a callable function; bench code constructs synthetic input + measures the function. No typestate setup.
4. **Cycle / validation early-exit is natural.** The runner returns `Outcome::Err` on early-exit conditions without needing typestate transitions for "did we abort."
5. **Skip-step capability for adapt's recompute.** When adapt only invalidates parts of the plan (e.g. lane assignment), the runner can re-call the affected step's free function with the previous step's cached output. Typestate would force a full re-typestate-walk.

### Axis D: WorkUnit-to-UnitId mapping

**DECIDED.** One `impl WorkUnit for MyWU` registration produces one `UnitId` in the DAG. The DAG is over WorkUnits (one vertex per `.with(MyWU)`), not over sub-handler ops. Per-call dispatch granularity (morsel chunking, scalar vs morsel) is decided *inside* the WU's `execute()` body based on the AccessSet shape; the plan-stage analysis sees only the WU's declared `type Read: AccessSet` and `type Write: AccessSet`.

This is **distinct from loimu's behavior-decomposition model**, where one `behavior` block decomposes into multiple scheduling units (one per `each` call, one per `tree`/`bind`/`dispatch`/`emit`). Hilavitkutin keeps the boundary at the type-level WorkUnit trait impl. The proc-macro on a WorkUnit does not produce sub-handler units.

#### Consequences for the chain

1. **Step 1 (build DAG)** treats each WorkUnit as one row in the access matrix. `read_masks[i]` and `write_masks[i]` are derived from `<MyWU as WorkUnit>::Read` and `<MyWU as WorkUnit>::Write` tuple types lowered to `AccessMask` bitmasks. Per Domain 15 step 1.
2. **Step 7 (fiber grouping)** groups WUs into fibers; each fiber holds N UnitIds. The "ops fused into one loop body" model from Domain 14 line 946 is a codegen-strategy decision (Topic 4 / Domain 17), not a definitional property of the fiber.
3. **Morsel-vs-scalar dispatch** per Domain 12 line 844: a WU declaring `Column<T>` in Read or Write is morsel-chunked; a WU using only Resource or Virtual access is scalar-dispatched. This decision is per-WU, recorded in UnitMeta as `dispatch_kind: DispatchKind`.
4. **Scheduling hints** (`type Hint: SchedulingHint`) per Domain 09 carry per-WU priority info (Urgency, Divisibility, Significance) used for write-write tie-break ordering and adapt subsystem inputs.

#### Why not loimu's decomposition

Loimu's behavior decomposition serves a different tradeoff: behaviors are user-authored at runtime via signals + handlers, and the proc-macro derives access sets from handler bodies because consumers don't write them by hand. Hilavitkutin's WorkUnits are declared statically at the trait level; the consumer writes `type Read = (...)` explicitly. Decomposition adds nothing because access sets are already explicit at the WU boundary. Adding it would force fiber grouping and dispatch codegen to reason at sub-WU granularity, which the AccessMask + holistic feasibility check were specifically designed to avoid.

#### Implications

- `MAX_UNITS` const generic on `Scheduler` is the upper bound on `.with(WU)` registrations. Consumers size it appropriately at builder time.
- A WorkUnit's `execute(&self, ctx: &Self::Ctx)` is the SAME function called per morsel inside the codegen-emitted dispatch program. The function does not know whether it's running over morsel 0 or morsel 47; the codegen materialises the morsel boundaries around the call.
- Internal-to-`execute()` looping (e.g. iterating over a morsel's records) IS authored by the WU writer. The fiber-flattener (Domain 17 rust-pipe) inlines this body and applies DSE on internal column reads/writes.

### Axis E: head+tail convergence mechanism in plan stage

**DECIDED.** Plan stage marks fibers eligible for head+tail convergence and allocates accumulator slot identifiers. Codegen (Topic 4) materialises the dual-thread program with stack-local accumulators and the merge step. Plan stage carries enough data for codegen; runtime carries no head+tail-specific machinery beyond the standard worker dispatch.

#### Eligibility (Domain 11 line 770, Domain 14 line 1174)

A fiber is eligible for head+tail convergence iff ALL of:

1. **Commutative.** Every WorkUnit in the fiber declares `COMMUTATIVE: true`. The fiber's `commutative` field is true iff the AND across all member WUs is true. Per Domain 14 line 1173.
2. **Single-trunk phase.** The fiber's parent trunk is the only trunk in its phase. Per Domain 11 line 770: "Single-trunk phases: head+tail convergence (2 threads, opposite ends, ~2x parallelism)."
3. **Record count threshold.** The phase's record count exceeds the strategy's threshold. Per Domain 21: <10K records → Sequential (no convergence); >10K with single trunk → PhasedConvergence is the right strategy.
4. **Resource accumulation skipped or compatible.** Non-commutative resource accumulation skips head+tail per Domain 14 line 1176. The fiber's hint must permit accumulator merging.

When eligible, plan stage populates `Fiber.head_tail: Maybe<HeadTailConvergence>`.

#### Codegen contract (preview for Topic 4)

Per Domain 17 line 1611: "Head+tail convergence: each core's record range is baked in." Codegen reads the fiber's `head_tail: Maybe<HeadTailConvergence>` slot and:

1. **If `Maybe::Isnt`:** emit a single per-core program that processes the full record range.
2. **If `Maybe::Just(ht)`:** emit two per-core programs:
   - Head program: records `0..mid`, accumulator initialised at `ht.head_accum.slot_id`.
   - Tail program: records `mid..total`, accumulator initialised at `ht.tail_accum.slot_id`.
   - Both programs end with a merge step into `ht.merge_target.slot_id` using `merge_op`.

The bench validation per Domain 11 line 799: phased+convergence runs 0.68x of sequential at 1M records (32% faster), versus phased without convergence at 0.82-0.89x.

#### Bench-validated thresholds

Per Domain 11 line 800-801:
- `phased+convergence` is consistent across all sizes 2K-1M records.
- `phased-conv (spawn)` reaches 0.62x at 1M but is **15.54x SLOWER at 200 records**. Hence pre-allocated thread pool is mandatory; spawn-based convergence catastrophes at small N.

Plan stage assumes pre-allocated pool (Domain 20 line 778); the eligibility check above does not need to consider spawn cost.

#### Why this lives in plan stage, not codegen

Eligibility depends on graph-shape facts (single-trunk-phase, commutative-fiber-AND) that plan stage already computes. Re-deriving in codegen would duplicate work and tie codegen to plan-internal data structures. Plan stage marks the slot; codegen consumes the marker.

The following invariants are part of the plan-stage contract and the `MemoryProvider` / `ThreadPool` / `Clock` platform contracts. Codegen (Topic 4) materialises code that depends on these holding; the plan stage validates them at fiber-formation feasibility-check time.

### M1. SIMD register-pressure unroll multiplier in feasibility check

The fiber-formation holistic feasibility check (Domain 14) currently states "register file budget + L1 write budget + L1+L2 total + no fan-in + no pipeline breaker." The register-file budget calculation MUST account for SIMD unroll factor:

```text
gpr_budget = base_arena_ptr + loop_counter + intermediates + constants + write_resource_ptrs
           + per_out_of_range_column_ptr  (M3 below)
           ≤ TARGET_GPR  (aarch64 ≈ 14, x86-64 ≈ 10)

simd_register_budget
  = (Σ active_write_columns × (simd_width_bits / element_width_bits) × unroll_factor)
  ≤ TARGET_SIMD  (aarch64 NEON: 32; AVX2: 16; AVX-512: 32)
```

Where `unroll_factor` is derived from `dispatch_approach` (TrunkMega and ScheduleMega favour 4-wide unroll for autovectorised loops; IndirectPerFiber stays scalar) and `simd_width_bits` is the platform-specific SIMD width.

A fiber whose grouped WU set passes the scalar register-pressure check but fails the SIMD-pressure check is split. The substrate bench harness measures the scalar vs SIMD outcome per fiber to validate this calculation.

### M2. Per-core program icache budget

Per-core monomorphised functions (codegen Topic 4) inline `#[inline(always)]` WU bodies for every fiber on that core. For consumers with 100+ WUs, the per-core function for ScheduleMega risks exceeding L1 instruction cache (Apple Silicon P-core 192 KB, E-core 128 KB; Skylake 32 KB; Zen3 32 KB).

Plan stage commits to a target budget:

```text
PER_CORE_ICACHE_BUDGET  = min(L1_icache) * 0.75
                       ≈ 24 KB on Skylake (binding constraint platform)
TYPICAL_WU_BODY_SIZE    ≈ 200-1000 bytes (SIMD column loops trend high)
WU_PER_CORE_LIMIT       = PER_CORE_ICACHE_BUDGET / TYPICAL_WU_BODY_SIZE
                       ≈ 24-120 WUs per core
```

The plan stage tracks per-core estimated icache footprint as a derived field on `CoreProgram`:

```rust
pub struct CoreProgram<...> {
    /* ... */
    /// Estimated icache footprint of the monomorphised per-core function in bytes.
    /// Computed from per-WU estimated body size × WU count + per-fiber dispatch overhead.
    estimated_icache_bytes: USize,
}
```

When `estimated_icache_bytes > PER_CORE_ICACHE_BUDGET`, plan stage falls back from ScheduleMega to TrunkMega (per-trunk dispatch with separate functions, costing one `bl` per trunk transition but fitting more aggregate WU bodies). When even TrunkMega exceeds budget, fall back to IndirectPerFiber.

The fallback path is a per-core decision; cores with light WU loads can run ScheduleMega while heavy cores run TrunkMega.

### M3. Arena 32 KB aarch64 immediate-displacement cliff

aarch64's 12-bit scaled immediate covers `imm12 × stride` where stride is the element size (max 8 B for u64). Above 32 KB offset within an arena, codegen needs an extra GPR loaded with `arena_base + 32 KB` for far-column accesses.

Plan stage commits this cost to the register-pressure feasibility check (M1):

```text
out_of_range_columns = column_offsets.iter().filter(|off| off > 32 KB).count()
gpr_budget += out_of_range_columns * 1  // one extra GPR per far column
```

A fiber whose `arena_size > 32 KB` AND whose register budget fails after counting out-of-range columns is split. This makes the "no addressing mode cliff up to 248 KB" claim precise: the cliff exists, the design accepts paying for it via additional GPR pressure, the feasibility check enforces the budget.

For AArch64-only consumers, codegen MAY also use the `add base, base, #imm` variant for moderate offsets, but the GPR-cost accounting is the conservative case.

### M4. Branch component prefetcher coverage

Branch components (chasers within a trunk) access both fiber-shared columns AND branch-unique columns. RCM-derived ordering optimises for fiber-shared access; branch-unique access patterns can produce non-stride-1 streams that saturate the L2 hardware prefetcher (Skylake tracks ≤4 concurrent streams per core; Zen3 has more aggressive depth; Apple Silicon ≥6).

Plan stage tracks per-branch prefetcher stream count:

```rust
pub struct BranchPrefetcherCheck {
    /// Estimated concurrent prefetcher streams the branch produces:
    ///   shared_fiber_stream + branch_unique_streams
    estimated_streams: USize,
    /// Per-platform target: Skylake = 4, Zen3 = 6, M-series P-core = 6.
    /// Stored at scheduler build time from CPU detection.
    target_max_streams: USize,
}
```

When `estimated_streams > target_max_streams`, plan stage emits a warning at build (in development profiles) and either:
1. Reorders branch column placement in the arena to convert non-stride-1 streams into stride-1.
2. Splits the branch into a fan-in bridge (which stops chasing and waits at the morsel boundary, reducing stream count to one).

The branch's `cut_weight` field on `Trunk` already approximates this; the explicit stream count makes the prefetcher invariant checkable.

### M5. Bridge readiness granularity

Bridges (fan-in nodes reading from multiple trunks) need a defined granularity for "when can the bridge start its morsel pass?" The design commits:

**Bridges fire at trunk-completion granularity, not chunk granularity.** A bridge waits for ALL parent trunks to complete their full record range (or their head/tail half for head+tail trunks) before firing. The bridge's reads are therefore from cache-coherent shared memory after the producer trunks' store buffers have drained to L3 / DRAM.

This is conservative: a finer-grained model where a bridge starts on per-morsel completion of its parents would extract more parallelism but requires per-morsel atomic progress signalling per parent trunk, multiplying the AtomicUsize traffic by the per-parent-trunk count. Bench-validated parallelism numbers in this design's reference data come from the trunk-completion-granularity model; finer granularity is a deferred optimisation.

If a future consumer demonstrates a bench-validated win for chunk-granularity bridges, that's a follow-up round with a new variant of `Bridge` (e.g. `BridgeChunked { trigger_threshold: USize }`). v1: trunk-completion only.

### M6. Phase-sync parking threshold calibration

The predictive parking thresholds (<100ns spin, 100ns to 10µs `spin_loop`, >10µs `park`) are NOT fixed constants in the engine. They are calibrated at `ThreadPool` initialisation via a micro-benchmark:

```rust
pub trait ThreadPool {
    /* ... */
    /// Returns calibrated thresholds for predictive parking. Called at .build() time;
    /// runs a brief inter-core round-trip benchmark to measure stlr+ldar latency.
    fn parking_thresholds(&self) -> ParkingThresholds;
}

pub struct ParkingThresholds {
    /// Below this duration (ns), pure spin is cheapest. Typical: 30-150 ns depending
    /// on platform and inter-cluster vs intra-cluster.
    spin_only_max:       USize,
    /// Below this duration (ns), `spin_loop()` with backoff is cheapest. Typical: 1µs-10µs.
    spin_loop_max:       USize,
    /// Above spin_loop_max, park via `wfe`/`pause` + futex.
}
```

The default impl in `hilavitkutin-providers` uses platform-detected defaults; consumers can override with measured values. The `<100ns` figure in the canonical design is a starting heuristic, not a fixed constant.

### M7. `stnp` (non-temporal store) microkernel applies to FiberOutput columns only

The day-one asm microkernel list includes `stnp` (non-temporal store, bypasses L1/L2 write-allocate). This is correct for `FiberOutput` columns (written once per morsel, not read again until the next phase consumes them).

`FiberInternal` and `FiberInternalAndOutput` columns are read back within the same fiber pass. Using `stnp` on those would force the next read to L3, costing 50-100 ns instead of an L1 hit (4-5 ns).

The column classification (step 11) drives microkernel selection: `stnp` only when `column_class == FiberOutput`. This is part of the codegen contract per Topic 4 / Domain 17 rust-pipe pattern.

### M8. NUMA affinity in `MemoryProvider`

For NUMA hosts (multi-socket x86, future ARM Neoverse N-series), per-trunk core-pinned column arenas must be allocated from the local NUMA node of the trunk's pinned core. Otherwise every arena access pays a remote-NUMA latency penalty (~100 ns vs ~30 ns on a 2-socket Skylake; worse on EPYC across CCD boundaries).

`MemoryProvider` extends with a NUMA-affinity hint:

```rust
pub trait MemoryProvider {
    /// Allocate a region of `size` bytes aligned to `align`. Plan-stage scratch and
    /// fiber arenas use this. The `affinity` hint requests NUMA-node placement; on
    /// UMA platforms (Apple Silicon, single-socket x86) the hint is no-op.
    fn alloc_aligned(&self, size: USize, align: USize, affinity: NumaAffinity)
        -> Outcome<*mut u8, AllocError>;
}

pub enum NumaAffinity {
    /// Allocate from the NUMA node of the calling thread.
    Local,
    /// Allocate from the NUMA node of the specified core's local memory.
    Core(Cap<MAX_LANES>),
    /// Interleave across all NUMA nodes (for cross-trunk shared read-only data).
    Interleave,
    /// Ignore NUMA (UMA fallback or test scenarios).
    Ignore,
}
```

The plan stage emits `Core(trunk.core)` for fiber arena allocations and `Local` for plan-stage scratch (built in the calling thread's NUMA node). The default `MemoryProvider` impl in `hilavitkutin-providers` uses `mbind`/`numa_alloc_onnode` on Linux and is a no-op on Apple Silicon (UMA).

### M9. Const-generic monomorphisation cost mitigation

The 9 outer const generics on `ExecutionPlan` produce one full monomorphisation per distinct `Scheduler` configuration. For consumers with multiple Scheduler shapes (viola checker + vehje compilation server), this multiplies codegen work.

Mitigation: derive `MAX_UNITS_PER_FIBER` and `MAX_COLUMNS_PER_FIBER` from outer constants:

```text
MAX_UNITS_PER_FIBER   = MAX_UNITS / MAX_FIBERS  (rounded up via Cap)
MAX_COLUMNS_PER_FIBER = MAX_COLUMNS / MAX_FIBERS  (rounded up via Cap)
```

Reduces the independent dimension count from 9 to 7 on the outer ExecutionPlan. Nested `Fiber<...>` still takes the derived values as const generics; rustc trait-solver computes them via `generic_const_exprs`.

For runtime-decided sizing (where the consumer would otherwise need a different Scheduler per workload), a future-considered alternative is a hybrid: const-generic outer caps + runtime-checked inner sizes (USize counts validated against const-generic upper bounds). This is a deferred optimisation; v1 uses the const-generic-only model because it produces strictly faster code (no runtime bounds check in the inner loop).

### M10. Adapt sidecar cache footprint accounted for in fiber-formation feasibility check (added 2026-05-10 from Topic 5 Axis D)

Topic 5's adapt subsystem allocates a hot region (per-fiber 64B-aligned cache lines holding progress counter + inline metrics) and a cold region (SoA arrays for end-of-pass derived metrics). Topic 5 Axis D establishes that the hot region writes happen mid-run inside the inner pipeline, so the cache lines those writes touch consume L1 write budget that the WU body would otherwise have. If the plan stage's `compute_morsel_size` runs without subtracting this sidecar footprint, the chosen morsel size overflows the real L1 budget and thrashing follows on every active diagnostic axis.

The fiber-formation feasibility check (Domain 14, plan-stage step 8, which `compute_morsel_size` in step 9 depends on) must subtract:

```text
hot_arena_bytes
  = layout_choice_bytes(per_fiber | per_trunk)         // bench-driven, Topic 5
  + sizeof(CoreParkSlot) * MAX_LANES                    // park breadcrumbs
  + axis_writers_active_this_pass(AdaptConfig)          // cadence-gated writers

l1_write_budget_effective = l1_write_budget - hot_arena_bytes
```

The `axis_writers_active_this_pass` term is the union of axes whose `sample_skip` value resolves to writing this pass (per Topic 5 Axis A interleave semantics). Plan stage reads the `AdaptConfig` from the consumer at planning time and computes the conservative-worst-case footprint (every axis active simultaneously) for the feasibility check; the runtime inner loop only writes the axes actually scheduled for that pass, so the conservative budget always covers actual runtime usage.

**Layout choice (per-fiber vs per-trunk) requires bench before doc-CL lock.** The conservative default until bench runs is per-fiber (clean isolation, no inter-fiber line contention). Bench sketch added: `mock/research/sketches/202605101036-adapt-sidecar-layout/`.

Without M10, the substrate's claim that "adapt does not break inner-loop performance" fails the moment the consumer enables any axis with mid-run writes. M10 makes the cache budget adapt-aware so the plan stage chooses smaller morsels when more diagnostic axes are active.

## Soundness invariants (S1-S7)

Surfaced by domain-expert audit on type-system, atomic-ordering, and FFI-soundness lens. These are correctness commitments, not performance hints.

### S1. Const-generic arithmetic via `CeilingDiv` helper trait

`generic_const_exprs` (rustc tracking #76560) and the multi-operator-in-array-size bug (#96695) make division and ceiling-division expressions in const-generic array bounds unreliable. Direct forms like `[T; (A + B - 1) / B]` or `[T; B + 1]` either fail to elaborate or require nested `where [(); EXPR]:` bounds that themselves don't elaborate cleanly.

Workaround locked: `CeilingDiv` helper trait with associated `const RESULT: usize`, implemented for the (small, finite) set of `(N, D)` pairs the substrate ships:

```rust
pub trait CeilingDiv<const N: usize, const D: usize> {
    const RESULT: usize;
}
```

Implementations are emitted by a proc-macro that enumerates `(N, D)` pairs from a config. Substrate ships impls for the typical Scheduler config caps (N up to 1024, D up to 64). Consumers requiring larger caps add impls via the same macro. The pattern is canonical (typenum-style) and works under stable Rust without `generic_const_exprs`.

For `BlockDiagonalisation`'s `[UnitId; B + 1]`, the substitute is `[UnitId; B_PLUS_1]` where `B_PLUS_1` is an additional explicit const generic that the caller threads through (matches the existing approach). No arithmetic in const-generic positions appears anywhere in plan-stage data; all expressions are precomputed by the caller and passed as named parameters.

### S2. Sealed dispatch-determining traits

`RunCfg`, `Resource`, `Column`, `Virtual`, `WorkUnit`, `Kit`, `MemoryProvider`, `ThreadPool`, `Clock` are ALL sealed via a private `Sealed` supertrait. Consumers cannot impl these traits for arbitrary types; the sealed pattern (canonical Rust idiom: a private `Sealed` super-trait defined in `mod sealed`) is required because:

1. Without sealing, a consumer could write `impl Resource for LoimuCfg` AND `impl RunCfg for LoimuCfg` simultaneously, producing an ambiguous `BuilderInput::Dispatch` resolution. The trait solver hits a coherence error with an opaque message.
2. `min_specialization` (rustc #31844) does NOT resolve overlap between two traits-applicable-to-the-same-type cases (only specialization of "more specific" over "less specific"). The pair-impl pattern relies on the type uniquely satisfying ONE dispatch-determining sub-trait at a time.

Sealing pattern:

```rust
mod sealed {
    pub trait Sealed {}
}

pub trait Resource: sealed::Sealed + BuilderInput<Dispatch = StoreDispatch<Self>> + ... {}
pub trait RunCfg:   sealed::Sealed + BuilderInput<Dispatch = RunCfgDispatch<Self>> + ... {}
// ... etc.
```

The substrate provides a derive macro `#[derive(Resource)]` / `#[derive(RunCfg)]` etc. that emits `impl sealed::Sealed for MyType {}` alongside the sub-trait impl. Consumers cannot impl Sealed by hand (it's `pub(crate)` in `notko`-shaped sealing) so they go through the macro, which forces them to pick exactly one dispatch role per type.

### S3. Non-temporal store + Release-store ordering (corrected per audit-2 C1)

M7 commits to `stnp` for FiberOutput columns. ARMv8-A `stnp` does NOT participate in the release semantics of a subsequent `stlr` (ARM ARMv8-A B2.3.2: non-temporal stores are weakly ordered relative to release barriers). Without an **inner-shareable** store-side data memory barrier, a reader on another core that sees the progress counter `stlr` may NOT see the prior `stnp` writes.

**Corrected codegen contract (Topic 4) — exact mnemonics per platform:**

```rust
// Codegen-emitted fiber-tail. Cfg-gated per architecture.

#[cfg(target_arch = "aarch64")]
unsafe {
    // ... last stnp writes for FiberOutput columns ...
    // dmb st (local PE) is INSUFFICIENT for inner-shareable observation.
    // Emit dmb ishst (inner-shareable store domain barrier) so other
    // cores observe the stnp writes before the subsequent stlr.
    core::arch::asm!("dmb ishst", options(nostack, preserves_flags));
    progress_counter.store(end_record, Ordering::Release);  // stlr
}

#[cfg(target_arch = "x86_64")]
unsafe {
    // ... last movntq/movntdq writes for FiberOutput columns ...
    // `core::sync::atomic::fence(Ordering::Release)` lowers to a
    // compiler barrier ONLY on x86 (TSO is free for ordinary stores).
    // For non-temporal stores, the architectural requirement is sfence
    // per Intel SDM Vol 3 §11.3.1. Emit it explicitly.
    core::arch::x86_64::_mm_sfence();
    progress_counter.store(end_record, Ordering::Release);  // mov (TSO)
}
```

Earlier draft said "the fence lowers to `sfence`" on x86 — that's wrong; `fence(Release)` is a no-op on x86. Earlier draft also said `dmb st` on aarch64 — that's local-PE-only and insufficient for cross-core observation; `dmb ishst` is required.

The ASM verification checklist (Topic 3 axis F + Topic 4) gains a sixth check, scoped per-platform:
- **aarch64:** any fiber whose dispatch_approach uses `stnp` for output columns has a `dmb ishst` immediately preceding the progress counter store.
- **x86_64:** any fiber whose dispatch_approach uses `movnt*` has an `sfence` immediately preceding the progress counter store.

This sixth check is a **codegen unit test** (Topic 4 Axis F split), not a bench-time disasm assertion — correctness invariant, not perf metric.

### S4. `Outcome` host/extension FFI boundary

`notko::Outcome<T, E>` has platform-default repr (per `notko/src/outcome.rs` doc comment). `Scheduler::run() -> Cfg::Out` (an `Outcome`-shape) is host-only; extensions never see the `run()` return type directly. Extensions interact with the host via the `hilavitkutin-extensions` capability ABI, which uses `#[repr(C)]` trampolines.

Soundness rule: any type that crosses the host/extension boundary in either direction (capability arguments, capability return values, callback shapes) MUST be `#[repr(C)]`. `Outcome<T, E>` is allowed only if both `T` and `E` are `#[repr(C)]` AND the consumer explicitly opts in via a `#[repr(C)]` wrapper. The capability-trampoline macro (`hilavitkutin-extensions-macros`) emits the wrapper and the pack/unpack code.

Consumers' `Cfg::Out` for run() does NOT cross the boundary in v1 (run() is host-only). If a future extension model exposes run() to extensions, the design needs explicit FFI shims; deferred.

### S5. P1.5 fusion is a free-function-level commitment, not LLVM-level

The runner pseudocode now shows `compute_upward_rank_and_dirty` as a single free function returning both outputs. The fusion happens AT the function boundary, not inside the LLVM optimiser. This is the form the cl-claim-sketch-discipline.md rule requires: the locked CL claim ("steps 3 and 12 share one backward DAG traversal") matches the runner pseudocode literally. There is no possibility of the LLVM optimiser failing to fuse two separate function calls.

The function signature:

```rust
pub(crate) fn compute_upward_rank_and_dirty<const U: usize>(
    dag:    &AccessDag<U>,
    topo:   &TopoOrder<U>,
    costs:  &[USize; U],
    fibers: &FiberGrouping<U, /* F */>,
) -> (UpwardRanks<U>, DirtyMasks<U>);
```

Other fusion candidates (steps 5+6, step 11+12) are explicitly NOT fused in v1; bench-validation in a per-step round may promote them to fused free functions later. Each fusion is a deliberate, bench-justified decision.

### S6. Edge cases for empty / trivial pipelines

The 12-step chain handles small-N edge cases via explicit early-return paths in each free function, NOT via runtime panics:

- `unit_count == 0`: `build_dag` returns an empty `AccessDag` (all masks zero, edge_count = 0). Subsequent steps produce empty intermediate types. Final `ExecutionPlan` has `phase_count = 0`, `unit_count = 0`. `run()` immediately returns `Outcome::Ok(...)` (the trivial pipeline succeeds with zero work).
- `unit_count == 1`: single-WU pipeline. `topo_sort` returns a single-element order. `compute_upward_rank_and_dirty` returns the WU's cost as its rank; dirty mask is all-ones if the WU is on the path of a dirty-marked input. `compute_waists` returns one phase with one trunk with one fiber. Plan stage produces a valid one-fiber plan.
- All-WUs-form-a-cycle: `topo_sort` sets `cycle_detected = TRUE`; runner returns `Outcome::Err(PlanError::Cycle)` per S5 fusion runner.
- `MAX_FIBERS == 0` (consumer mis-configured Scheduler with zero-fibers cap): rejected at `Scheduler::builder().build()` time as a typestate transition error (the const-generic constraint `MAX_FIBERS >= 1` is part of the `RunCfg` validation).
- `record_count == 0`: morsel sizing returns `morsel_size = MIN_MORSEL` (configured floor); fiber dispatch runs zero iterations of the inner loop. `run()` returns `Outcome::Ok(...)` with no work performed.

Per-step bench harnesses include explicit empty + trivial cases. Step bodies that panic on these inputs are bugs to be caught at sketch validation, not a runtime concern.

### S7. Cross-thread atomic-ordering protocol (added 2026-05-10 per audit-2 structural recommendation)

The substrate has multiple cross-thread atomic interactions; the audit-1 + audit-2 rounds found ordering was specified inconsistently across S3 (progress counter), Topic 5 Axis B (snapshot), Topic 5 Axis E (Virtuals), Topic 5 Axis G (parking predictions), and the per-core idle breadcrumb. S7 consolidates the contract; every cross-thread atomic in the design appears in this table with the exact `Ordering` on each side and the architectural mnemonic per platform.

| Site | Producer | Producer ordering | Consumer | Consumer ordering | aarch64 prod / cons | x86_64 prod / cons |
|---|---|---|---|---|---|---|
| Fiber progress counter | Owning fiber's core | `Release` (preceded by `dmb ishst` if any prior `stnp` writes) | Next-phase observer / bridge / parking primitive | `Acquire` | `stlr` / `ldar` | `mov` (TSO-free) / `mov` |
| Per-core park-time accumulator | Owning core (park entry/exit) | `Release` on increment (paired with phase-boundary release) | AdaptWu at `ScheduleEnd` | `Acquire` fence on entry, then `Relaxed` reads of all slots | `stlr` (or `stxr`/`stlxr` if `fetch_add`) / `ldar` | `lock xadd` / `mov` |
| AdaptWu→`metrics::*` snapshot publish | AdaptWu (on whatever core runs the meta-WU) | Write all snapshot fields with `Relaxed`; then `current_buffer_idx.store(new_idx, Release)` (double-buffer pattern) | Consumer WU reading `metrics::*` | `current_buffer_idx.load(Acquire)`; then `Relaxed` reads of snapshot fields | `stlr` (idx only) / `ldar` (idx only) | `mov` / `mov` |
| Virtual<T> fired-flag set | AdaptWu (or any firing WU) | `Release` (engine's Virtual primitive enforces this) | Observer WU's gate check | `Acquire` | `stlr` / `ldar` | `mov` / `mov` |
| AdaptWu reads AdaptArena hot slots | Worker threads (per-morsel writes to inline metrics) | `Relaxed` increments paired with phase-boundary Release | AdaptWu at `ScheduleEnd` | `Acquire` fence at entry, tied to phase-barrier completion | (relaxed increments → eventual visibility through phase barrier) | (relaxed increments → eventual visibility) |
| `stnp` non-temporal output column writes | Owning core | (no atomic ordering — non-temporal) + explicit `dmb ishst` before progress counter Release | Bridge/observer reading the column on another core | Acquire on the progress counter is the gate; once observed, column data is visible | `dmb ishst` then `stlr` / `ldar` (then plain loads OK) | `_mm_sfence()` then `mov` / `mov` |

**Double-buffer pattern for AdaptWu snapshot publish (resolves audit M1):**

```rust
struct MetricsResource<T> {
    buffer_a: T,
    buffer_b: T,
    current: AtomicUsize,  // 0 = a is live, 1 = b is live
}

// AdaptWu at ScheduleEnd:
fn execute(&self, ctx: &Self::Ctx) {
    let mut r = ctx.writer::<MetricsResource<...>>();
    let next_idx = 1 - r.current.load(Ordering::Relaxed);
    let next_buf = if next_idx == 0 { &mut r.buffer_a } else { &mut r.buffer_b };
    // ... fill next_buf from AdaptArena slots (Relaxed reads paired
    //     with phase-barrier Acquire fence at WU entry) ...
    r.current.store(next_idx, Ordering::Release);  // single atomic publish
}

// Consumer WU:
fn execute(&self, ctx: &Self::Ctx) {
    let r = ctx.reader::<MetricsResource<...>>();
    let idx = r.current.load(Ordering::Acquire);
    let buf = if idx == 0 { &r.buffer_a } else { &r.buffer_b };
    // ... read buf with Relaxed; the Acquire on `idx` carries
    //     the visibility of all fields in the published buffer ...
}
```

**Phase-barrier release-fence convention (resolves audit M2):**

At every phase boundary, the engine emits a release fence on the producer side and an acquire fence on the consumer side. Worker writes between phase boundaries (per-morsel inline metric increments, idle accumulator updates) can use `Relaxed`; the phase barrier carries the cross-thread visibility. AdaptWu's `ScheduleEnd` observation sits after the final phase barrier; an `Acquire` fence at its entry suffices for all metrics reads.

**Testing protocol:** Topic 11 test bar adds:
- `miri` runs with `MIRIFLAGS="-Zmiri-strict-provenance -Zmiri-tree-borrows"` on the engine crate.
- `loom` runs for each cross-thread atomic pair in the table above. Each pair gets at least one loom test.
- `cargo asm` checks that `Release` / `Acquire` on the progress counter actually emit `stlr` / `ldar` on aarch64 (the codegen sixth check from Topic 4 Axis F).

This is the single biggest soundness anchor for the runtime; mismatches against this table at SRC-CL-lock time block the lock.

## Decisions (consolidated)

All six axes locked.

1. **Chain confirmation (A).** Ship the polka-dots-ported 12-step canonical chain as listed in P1, no splits/merges in this round. Each step's substrate dependency named (arvo-bitmask, arvo-graph, arvo-sparse, arvo-spectral, arvo-comb). Per-step internals deferred to per-step follow-up rounds.
2. **ExecutionPlan shape (B).** Final type mirrors the canonical runtime hierarchy `pipeline → core → phase ↔ waist → trunk → fiber ↔ branch ↔ bridge → morsel → micro-morsel → record`. Captures: phases with pipeline parallelism + progress counter slots, trunks with zero-sync column-disjointness + spectral cluster + core pinning, **TrunkComponent enum (Fiber | Branch | Bridge)** as peers within a trunk, fibers with morsel + micro-morsel sizing + 3D morsel cube data + co-located arena + COMMUTATIVE flag + head+tail convergence slots + dispatch approach selection, UnitMeta with rank/depth/cost/fiber/trunk/phase/core/lane-compat/scheduling-hint, dirty propagation masks, RCM-derived column ordering, per-core projection. Seven const generics: MAX_UNITS, MAX_PHASES, MAX_TRUNKS, MAX_FIBERS, MAX_LANES, MAX_COLUMNS, MAX_COMPONENTS_PER_TRUNK. Intermediate analytical artefacts under `plan::analysis::*` are `pub(crate)`; not exposed in final ExecutionPlan. Steps composed as free functions, not typestate-threaded builder.
3. **Ship scope (C).** Full 12 steps in this round per the canonical design. Per-algorithm internals at "what algorithm does the substrate ship" granularity (RCM heuristic choice, spectral eigensolver selection, fiber-grouping DP threshold); bench-driven tuning in follow-up rounds.
4. **WU-to-UnitId mapping (D).** One `impl WorkUnit for MyWU` registration produces one UnitId in the DAG. Plan-stage analysis sees only the WU's declared `type Read`/`type Write` AccessSets. Per-call dispatch granularity (morsel chunking, scalar vs morsel) decided inside `execute()`; not at sub-WU granularity. Distinct from loimu's behavior decomposition model. Morsel-vs-scalar dispatch inferred per-WU from AccessSet shape (Domain 12 line 844).
5. **Head+tail convergence (E).** Plan stage marks fiber eligibility (commutative AND single-trunk-phase AND record threshold AND compatible accumulation) and allocates AccumSlot identifiers (head, tail, merge_target). Codegen materialises the dual-thread program. Bench-validated 0.68x at single-trunk phases versus 1.00x sequential per Domain 11 line 799. Pre-allocated thread pool mandatory (spawn-based convergence is 15.54x slower at 200 records per Domain 11 line 800).
6. **Per-core projection compatibility (F).** ExecutionPlan ships `core_programs: [CoreProgram; MAX_LANES]` with phases-this-core-processes, trunks-this-core-owns, per-fiber record ranges (Full / Head / Tail), phase-sync roles, AND `estimated_icache_bytes` per M2. Codegen (Topic 4) consumes this directly. Plan stage carries the inputs codegen needs to emit code that passes the ASM verification checklist (zero `blr`, indexed addressing, no `[sp,...]` in inner loop, immediate-constant morsel size, no `bl morsel_size` calls).

7. **Soundness invariants (S1-S6).** Compilation + correctness commitments:
   - **S1.** Const-generic arithmetic via `CeilingDiv` helper trait + proc-macro-emitted impl set. No `[T; A / B]`, `[T; B + 1]`, or other arithmetic-in-const-generic-array-size expressions; all such expressions resolved by passing precomputed values as named const-generic parameters.
   - **S2.** All dispatch-determining traits (`Resource`, `Column`, `Virtual`, `WorkUnit`, `Kit`, `RunCfg`, `MemoryProvider`, `ThreadPool`, `Clock`) are sealed via private `Sealed` supertrait + derive-macro-only impl path. Eliminates pair-impl coherence overlaps that `min_specialization` cannot resolve.
   - **S3.** Codegen MUST emit a Release fence (`dmb st` / `sfence`) between final fiber-output `stnp` sequence and progress-counter `stlr` / `mov`. ARMv8-A non-temporal stores do NOT participate in `stlr` Release semantics; without the fence, bridges may read stale data.
   - **S4.** `notko::Outcome<T, E>` is host-internal. Extension ABI requires `#[repr(C)]` for any type crossing the boundary; `Outcome` requires explicit `#[repr(C)]` wrapper if exposed. v1 `Scheduler::run()` is host-only.
   - **S5.** P1.5 step-3+step-12 fusion happens at the free-function-call level (`compute_upward_rank_and_dirty` is a single function returning both outputs), not via LLVM optimiser hopes. Aligns with cl-claim-sketch-discipline.md.
   - **S6.** Edge cases (zero / one WU, all-cycle, zero-fibers config, zero-record runs) handled via explicit early-return paths, not panics. Per-step bench harnesses include trivial cases.

8. **Platform-microarchitecture invariants (M1-M9).** Plan stage and platform contracts commit to nine μarch-aware shapes:
   - **M1.** Fiber feasibility check accounts for SIMD register pressure with unroll multiplier, not just scalar GPRs.
   - **M2.** Per-core program tracks `estimated_icache_bytes`; plan stage falls back ScheduleMega → TrunkMega → IndirectPerFiber when budget exceeds L1 icache (24 KB conservative, sized to Skylake's 32 KB at 75%).
   - **M3.** Arena 32 KB aarch64 immediate cliff handled explicitly; out-of-range columns charge an extra GPR to the register-pressure budget.
   - **M4.** Branch components carry estimated prefetcher-stream count; plan stage warns or splits when streams exceed the platform target (Skylake 4, Apple P-core 6).
   - **M5.** Bridges fire at trunk-completion granularity in v1; finer granularity is a deferred follow-up round.
   - **M6.** Parking thresholds (spin / spin_loop / park) calibrated at `ThreadPool::build()` via micro-benchmark; not fixed constants.
   - **M7.** `stnp` non-temporal store microkernel applies to `FiberOutput` columns only; codegen distinguishes column class.
   - **M8.** `MemoryProvider::alloc_aligned` carries `NumaAffinity` hint (`Local` / `Core(c)` / `Interleave` / `Ignore`); plan stage emits `Core(trunk.core)` for fiber arenas.
   - **M9.** `MAX_UNITS_PER_FIBER` and `MAX_COLUMNS_PER_FIBER` derived from outer `MAX_UNITS / MAX_FIBERS` and `MAX_COLUMNS / MAX_FIBERS`; outer ExecutionPlan const-generic surface reduced from 9 independent to 7 independent + 2 derived.
   - **(Bonus, P2.5 above.)** Fiber arena base pointers and plan-stage scratch are 64-byte aligned; this is a `MemoryProvider` contract, not a plan-stage post-condition.
