## Decisions locked

### Axis A. `ResourceSnapshot<R>` v1 semantics

A `ResourceSnapshot<R: Resource>` is a **read-only typed view** captured at a named phase boundary. Concretely:

```rust
pub struct ResourceSnapshot<R: BuilderInput<Dispatch = ResourceDispatch>> {
    // Pin<&'frame R::Owned> behind the lifetime story from Topic 6.
    inner: Pin<&'frame R::Owned>,
    captured_at: PhaseId,
}
```

Snapshot captures only at phase barrier (after futex-wake, before next phase's morsels begin). Workers that hold a `ResourceSnapshot<R>` are guaranteed:
- The underlying `R::Owned` storage is not being written during the snapshot's lifetime.
- The snapshot's content matches what `R::Owned` held at `captured_at`.
- Snapshot lifetime is bounded by the current phase (cannot escape across phase barriers).

Use case: AdaptWu at ScheduleEnd phase observes `ResourceSnapshot<AdaptMetrics>` to derive `predicted_wait_ns[]` writes for the next frame. No mutation; pure read.

Lifetime: `ResourceSnapshot<'phase, R>`. Once `'phase` ends (next phase barrier), the snapshot is invalid; lifetime invariance prevents lift-out.

### Axis B. `PlanAffecting` marker trait

Sealed marker trait in `hilavitkutin-api`. Implementations are opt-in by consumers OR derived by macro:

```rust
pub trait PlanAffecting: BuilderInput + private::Sealed {}
```

When a `BuilderInput` impl carries `PlanAffecting`, replacing its value via `Scheduler::replace_resource` (or the equivalent post-rename method) triggers a plan recompute dirty bit. When it does NOT, replacement is a cheap value swap.

Default: NOT plan-affecting. Consumers opt in for resources whose value shape changes the plan (e.g., the `RunCfg` resource itself — the dispatch routing target). Most app-state resources (column data, intermediate accumulators, diagnostics) are NOT plan-affecting.

Topic 1 axis 4 said "interim: replace_resource unconditionally dirties until Topic 8's PlanAffecting marker lands". Topic 8 lands the marker; the conditional dirty path is now wired.

### Axis C. Persistence spine — hilavitkutin-persistence integration

The persistence spine is the cold-store sibling to the runtime data plane. `hilavitkutin-persistence` ships `PersistenceContext` (existing crate). The runtime engine accesses persistence via:

- `Resource<P>` where `P: PersistenceProvider`. Pluggable per consumer.
- Hot-side (in-frame) reads: copy from cold into a `Column<T>` at PlanStage; runtime touches the column, not persistence directly.
- Cold-side writes: PassEnd WU drains accumulator columns into persistence via `PersistenceProvider::commit`.

The integration is intentionally narrow: the engine never depends on `hilavitkutin-persistence` types directly. Consumers wire it up via Provider tuples. This preserves the standalone-ecosystem-crate rule for hilavitkutin-persistence.

### Axis D. Resource resolution at builder time

Already locked by Topic 1 axis 1 (`.with(value)` API) + Topic 2 (BuilderInput trait + Dispatch sub-trait routing). Topic 8 adds nothing here; just confirms the path:

1. Consumer calls `.with(MyResource::default())`.
2. Builder's typed router (sealed `Dispatch = ResourceDispatch`) places it in the appropriate Resource slot.
3. At `build()` time, the scheduler walks the registered Resource types, calls each `HasTrivialCtor::ctor` for unregistered defaults, and produces the typed `ResourceArena`.
4. Per-WU `Context<P>` accessors (`HasResourceProvider<R>`) project into the arena.

## User-attention call (autonomous, can be revisited)

### `MAX_PLAN_AFFECTING_RESOURCES = 16`

The dirty-bit array is `[AtomicBool; MAX_PLAN_AFFECTING_RESOURCES]` keyed by `PlanAffectingId`. Default cap is 16. This is the count of distinct `PlanAffecting`-bound BuilderInput types per pipeline. Realistic consumer counts (RunCfg + a few stage configs) likely under 8; 16 leaves headroom.

If a consumer registers more than 16 PlanAffecting types, compile-time fails with `#[diagnostic::on_unimplemented]` pointing at `MAX_PLAN_AFFECTING_RESOURCES`. Cap is a const generic on Scheduler; consumers override via `Scheduler::<_, _, 32>::builder()` syntax if needed.
