## 1. The Three-Phase Pattern: Graph, Plan, Execute

loimu-graph is built around a three-phase execution model that every consumer follows identically:

```
1. GRAPH     Build the DAG: nodes, edges, metadata
2. PLAN      Analyze the DAG: filter, order, group, annotate
3. EXECUTE   Run the plan: dispatch work units, collect results
```

Phase 1 (Graph) is structural: add nodes and edges, reject cycles. Phase 2
(Plan) is analytical: topologically sort, group independent nodes into parallel
steps, attach domain-specific strategy hints. Phase 3 (Execute) is mechanical:
iterate steps, dispatch each step's nodes in parallel, barrier, repeat.

The key architectural decision is that the compile phase (which sits between
raw plan computation and execution) is pluggable. Each consumer defines its own
compile pipeline as an ordered tuple of compile passes. The compiler
monomorphizes the entire pipeline, producing zero dynamic dispatch and full
inlining. Each pass implements `CompilePass<N: Id>`:

```rust
pub trait CompilePass<N: Id> {
    type Input;
    type Output;
    fn apply(self, input: Self::Input) -> Self::Output;
}
```

Consumers compose passes into typed pipelines:

```rust
type SchedulerCompile = CompilePipeline<(
    GroupParallelSteps,
    AssignLanes<8>,
    VersionStampEarlyRelease,
)>;

type VulkanRenderCompile = CompilePipeline<(
    GroupParallelSteps,
    InsertBarriers<Vulkan>,
    AllocateTransients,
    AliasResources,
)>;

type MobileRenderCompile = CompilePipeline<(
    GroupParallelSteps,
    MergePassesForTileMemory,
    InsertBarriers<Vulkan>,
    AllocateTransients,
    AliasResources,
)>;
```

This separation is directly validated by Halide's "schedule as transformations"
model (MIT, PLDI 2013) and confirmed across all reference implementations
(Frostbite FrameGraph GDC 2017, Taskflow IEEE TPDS 2021, Nanite SIGGRAPH
2021). The invariant is: the domain builds the DAG (algorithm), and plan
computation (schedule) can be tuned independently.

Four current consumers share this pattern:

| System | Nodes are | Edges mean | Domain concerns |
|---|---|---|---|
| Scheduler | Scheduling units | Signal flow + resource access | Lane assignment, parallel execution, upward rank |
| Animation | AnimationNodes | Data flow (values + blend weights) | Interpolation modes, driver evaluation, GPU batching |
| Shader compilation | Compile jobs | Include/import dependencies | Caching, incremental recompilation, error recovery |
| View affinity | Record fields | Co-access frequency from behavior access declarations | Grouping fields into views for SoA layout |

Prior art cited in the design: Frostbite FrameGraph (GDC 2017), Halide (MIT,
PLDI 2013), Taskflow (IEEE TPDS 2021), Nanite (SIGGRAPH 2021), Intel TBB Flow
Graph.

## 2. Dag<N, E> Structure

The core data structure is generic over node ID type and edge data type:

```rust
pub struct Dag<N: Id, E> {
    forward: IdMap<N, DenseColumn<(N, E)>>,
    reverse: IdMap<N, DenseColumn<N>>,
    metadata: IdMap<N, NodeMeta>,
}

pub struct NodeMeta {
    payload: [u8; 32],
}
```

Node IDs are typed via `define_id!` from `loimu-id`. This prevents
accidentally mixing node IDs from different domains. Edge data `E` is
domain-specific: the scheduler uses signal flow and resource conflict edges;
the animation system uses data flow edges with blend weights.

Dual adjacency maps (forward and reverse) allow O(1) access to both successors
and predecessors. `NodeMeta` carries 32 bytes of domain-specific payload
without requiring an allocation or a trait object.

### Operations

```rust
impl<N: Id, E> Dag<N, E> {
    pub fn add_node(&mut self, id: N) -> bool;
    pub fn remove_node(&mut self, id: N) -> bool;
    pub fn add_edge(&mut self, from: N, to: N, data: E) -> Result<(), CycleError>;
    pub fn remove_edge(&mut self, from: N, to: N) -> bool;
    pub fn has_node(&self, id: N) -> bool;
    pub fn has_edge(&self, from: N, to: N) -> bool;
    pub fn successors(&self, id: N) -> &[(N, E)];
    pub fn predecessors(&self, id: N) -> &[N];
    pub fn node_count(&self) -> usize;
    pub fn edge_count(&self) -> usize;
}
```

### Online Cycle Detection

`add_edge` performs online cycle detection. On insertion, it temporarily adds
the proposed edge, then runs an incremental reachability check: if the target
can reach the source through existing edges, the edge would create a cycle, so
it is rejected with a `CycleError` containing the full cycle path for
diagnostics. Cycles are rejected at load time, not schedule time.

For batch construction (initial graph build), full topological sort via Kahn's
algorithm is more efficient than per-edge incremental checks.

### Topological Sort

```rust
pub fn topological_sort<N: Id, E>(dag: &Dag<N, E>) -> Vec<N>;
```

Uses Kahn's algorithm specifically because it processes nodes by repeatedly
removing zero-in-degree nodes, producing level-based ordering. All nodes at the
same depth appear in the same batch. DFS-based toposort produces valid ordering
but does not group by level; it interleaves deep and shallow paths. Level
grouping feeds directly into the step-based plan.

Panics if cycles are found (should not happen if all edges went through
`add_edge`).

### Invariants

1. The DAG is always acyclic. `add_edge` rejects cycle-creating edges.
2. `topological_sort` produces a valid order.
3. Within a plan step, no edges exist between nodes. Parallel execution is safe.
4. Dirty propagation is transitive and monotonic. Once dirty, stays dirty until `clear()`.
5. Subgraph execution is recursive but bounded.

## 3. Behavior Decomposition into Scheduling Units

Behaviors in loimu are not monolithic nodes in the DAG. Each behavior is a
subgraph: its handler body is decomposed into individual scheduling units, each
of which becomes its own DAG node. Internal edges between units come from access
analysis, not unconditional sequential chains.

### Handler-Based Dispatch

The scheduler invokes handlers by signal type. Each `on Signal => name { ... }`
block is a handler. The proc macro (`define_behavior!` or `#[derive(Behavior)]`)
derives observed signals, emitted signals, and access sets from the handler
body. No manual declarations are needed.

```rust
define_behavior!(ParticleBehavior {
    on PhysicsStep => physics_handler {
        each (pos: &mut Spatial2d, vel: Velocity) {
            pos.position += vel.value;
        }
    }

    on RenderFrame => render_handler {
        each (pos: Spatial2d, color: ParticleColor) {
            // read-only iteration for rendering
        }

        bind ParticleBinding {
            positions: pos.position,
        };
        dispatch ParticlePipeline;
    }
});
```

### Scheduling Unit Types

| Keyword | Purpose | Is scheduling unit? | Fusible? |
|---------|---------|---------------------|----------|
| `each (fields) { body }` | Collection iteration | Yes | Yes |
| `tree<Marker> (node) { body }` | Tree traversal | Yes | No |
| `bind Type { fields }` | GPU binding write | Yes | No |
| `compute { groups, local_size }` | GPU compute dispatch | Yes | No |
| `draw { target, instances }` | GPU draw dispatch | Yes | No |
| `copy { src, dst, size }` | GPU copy dispatch | Yes | No |
| `emit Type { fields }` | Signal emission | Yes | No |
| `ctx.read::<T>()` | Singleton resource read | No (captured) | - |
| `ctx.write::<T>(\|t\| { ... })` | Singleton resource write | No (captured) | - |
| `let name = expr;` | Intermediate computation | No (scalar) | - |

Each handler decomposes into:

```
on PhysicsStep => physics_handler:
  Unit 0: each (pos, vel) { ... }     <- fusible
  Unit 1: emit Collision { ... }      <- standalone

on RenderFrame => render_handler:
  Unit 0: each (pos, color) { ... }   <- fusible
  Unit 1: bind ParticleBinding        <- standalone
  Unit 2: dispatch ParticlePipeline   <- standalone
```

Units within the same handler are connected by edges from access analysis, not
by unconditional sequential chains. Two units touching different fields can run
in parallel. The scheduler sees inside the behavior and can interleave units
from different behaviors.

### Subgraph Diagram

```
Main DAG:
  [SignalEmitter] -> [BehaviorA.handler1.unit0] -> [BehaviorA.handler1.unit1]
                  -> [BehaviorB.handler1.unit0]
                                    \
                    [BehaviorA.handler1.unit2] -> [consumer]
```

`BehaviorGraph` stores instances in `IndexArray<BehaviorId, BehaviorSubgraph>`.
It is a registry of behavior subgraphs, not a `Dag` itself. The graph compiler
in loimu-graph flattens all subgraphs into one global `Dag` (the execution plan).

## 4. Edge Construction from Field-Granular Access Declarations

Edges in the scheduler DAG are not derived from resource-level declarations.
They are derived from field-level access analysis. This is a critical
distinction. Two behaviors accessing different fields of the same record produce
no DAG edge and can run in parallel.

### Access Derivation

| Pattern | Derived access |
|---------|---------------|
| `each (a: A, b: B, c: &mut C) { ... }` | read A, read B, write C |
| `tree<M> (node) { ... }` | read tree M |
| `ctx.read::<R>()` | Read singleton R |
| `ctx.write::<R>(\|t\| { ... })` | Write singleton R (implies read) |
| `emit Type { ... }` | Emits Type signal |
| `bind Type { ... }` | Write binding descriptor |
| `compute { ... }` | GPU compute dispatch (Compute queue) |
| `draw { ... }` | GPU draw dispatch (Graphics queue) |
| `copy { ... }` | GPU copy dispatch (Copy queue) |

### Field-Level Inference

```rust
each (spatial: Spatial2d) {
    let pos = spatial.position;  // reads Spatial2d.position
    // Spatial2d.velocity not accessed -> no edge
}
```

Two behaviors that both access `Spatial2d` but one reads `.position` and the
other reads `.velocity` produce no conflict edge. This granularity is essential
for parallelism: without it, any behavior touching a large struct would
serialize with every other behavior touching the same struct.

`&mut` in an `each` tuple indicates write access. `&T` (bare reference) maps to
`Pinned<T>`, the default read mode.

### Pessimistic Emit Detection

All code paths in a handler are scanned for `emit` statements. If any code
path can emit a signal, the behavior is marked as emitting that signal. This
is conservative: a guarded emit that never fires at runtime still creates a
DAG edge at plan time.

### Seven Access Modes

loimu defines seven access modes. Polka and saalis need only Read and Write;
the full set is loimu-specific:

| Mode | Direction | Freshness | Ordered | Deterministic | Default? |
|------|-----------|-----------|---------|---------------|----------|
| `Pinned<T>` | Read | Frozen snapshot | No | Yes | Default for reads |
| `Immediate<T>` | Read | Current frame | Yes | Yes | Opt-in |
| `MutImmediate` / `&mut T` | Write | Current frame | Yes | Yes | Default for writes |
| `Sample<T>` | Read | Current | No | No | Opt-in |
| `Post<T>` | Write | Current | No | No | Opt-in |
| `Approx<T>` | Read | Future | No | Yes | Opt-in |
| `Defer<T>` | Write | Deferred | No | Yes | Opt-in |

Compile-time errors prevent illegal combinations: `&Pinned<T>` (double wrap),
`&mut Approx<T>` (write with read-only mode), reading `Post<T>` or `Defer<T>`
(write-only modes as reads).

### Core Types

| Type | Purpose |
|------|---------|
| `Behavior` trait | Handler-based logic unit |
| `AccessSet` | Field-granular resource access for parallel scheduling |
| `FieldAccess` | Per-resource field access descriptor |
| `AccessMode` | Pinned, Immediate, MutImmediate, Sample, Post, Approx, Defer |
| `SchedulingUnitDescriptor` | Per-unit metadata including queue_affinity and dispatch_config |
| `BehaviorDescriptor` | Registry entry (generated by `define_behavior!`) |

## 5. Claim and Release Points (Computed Portals)

Portals are exclusively a subgraph concept and are computed, not declared. All
behaviors declare access sets statically. The subgraph's internal topology is
known at plan time. The framework walks the subgraph's DAG and identifies two
boundary points per resource:

- **Claim point** (head-side): first internal node that accesses a given
  resource. External nodes can freely use that resource until the subgraph
  reaches its claim point.
- **Release point** (tail-side): last internal node that accesses a given
  resource. External nodes waiting on that resource can start after the
  release point, not after the entire subgraph finishes.

### Worked Example

```
Subgraph S: [A, B, C, D, E] in topological order

  A: reads R       <- claim point for R (first access)
  B: writes R
  C: reads R       <- release point for R (last access)
  D: writes Q      <- claim AND release point for Q (only access)
  E: unrelated work

Parent scheduler computes:
  - External writers of R can proceed until S.A starts
  - External readers of R can start after S.C completes (not after S.E)
  - External nodes using Q: can write Q until S.D starts, read Q after S.D
```

Both directions (early release on the tail side, delayed claiming on the head
side) fall out from static access analysis. The subgraph author writes normal
behaviors; the framework computes boundaries automatically.

This enables inter-subgraph parallelism: a resource that is released early by
one subgraph can be claimed by another subgraph or leaf node before the first
subgraph has finished its unrelated tail work.

## 6. Subgraph Composition

### NodeKind and Cached Plans

```rust
pub enum NodeKind<N: Id> {
    Leaf,
    Subgraph(ExecutionPlan<N>),
}
```

A `Subgraph` node contains a pre-computed `ExecutionPlan`. When the executor
reaches a subgraph node, it executes the inner plan's steps (potentially in
parallel) before proceeding. The inner plan is computed once and cached. It
only recomputes when the inner graph's topology changes. The `DirtyTracker` can
mark the subgraph node itself as dirty when its inner graph changes.

### Concrete Use Cases

**Scheduler -> Animation:**

```
Scheduler graph:
  InputBehavior -> AnimationBehavior -> RenderBehavior
                       |
              Animation subgraph:
                Driver1, Driver2 -> Blend -> Constraint -> Output
```

**Scheduler -> Module:**

```
Scheduler graph:
  CoreBehaviors -> ModuleA -> ModuleB -> RenderBehaviors
                    |           |
            ModuleA subgraph  ModuleB subgraph
```

### Nesting Depth

Subgraphs can nest. In practice, two levels suffice (scheduler -> animation,
scheduler -> module -> module's behaviors). Deep nesting indicates a design
problem.

### Inner Node ID Space

Inner plan's node IDs live in the same ID space as the outer graph. No ID
collisions because typed IDs from `define_id!` encode domain and scope.

### Execution Strategies

**Sequential expansion** (current): execute inner plan step by step, with
parallel dispatch within each step. Outer graph's step boundary waits for the
entire subgraph.

**Parallel expansion** (future): if the subgraph's first step has no external
dependencies beyond predecessors that are already complete, the subgraph could
begin executing while other nodes in the same outer step are still running.

**GPU offload** (future): for compute-heavy subgraphs, the inner plan could be
lowered to GPU compute dispatches. CPU submits compute work and moves on; a
fence signals completion.

### Opaque Subgraphs

Subgraphs can be explicitly marked opaque, which disables boundary analysis
(computed portals). The parent treats the opaque subgraph as a single atomic
node with full stall on all declared resources. No internal inspection.

Use cases: GPU-driven pipelines where internal synchronization is managed by GPU
hardware; externally synchronized subgraphs where correctness depends on running
as a monolith. Opt-in; monolithic subgraphs that are not marked opaque are a
code smell.

### Subgraph Invariants

1. Subgraph expansion terminates (acyclic graph, cannot reference ancestors).
2. Non-opaque subgraphs allow boundary analysis. Parent may start/complete
   external work between claim and release points.
3. Opaque subgraphs are atomic. Parent waits for entire subgraph before
   releasing any resources.
4. Subgraph plans are cached. Only recompute on topology change.
5. Inner plan's node IDs live in the same ID space as the outer graph.

### Taskflow Comparison

Taskflow's `composed_of` model is the closest analog. Key properties shared
with loimu: (1) subgraph is a single node in outer graph for ordering, (2)
internal parallelism is preserved, (3) subgraph is a reference not a copy,
(4) no special executor support needed. Taskflow also supports dynamic tasking
("subflow"), where a task spawns new tasks at runtime. loimu uses static
subgraphs with replanning on topology change instead; dynamic subgraphs are
deferred.

## 7. Resource Aliasing

### Core Idea

Resources that never live at the same time can share the same memory. Directly
analogous to register allocation in compilers: variables with non-overlapping
lifetimes share the same register. This comes from Frostbite FrameGraph's
compile phase (GDC 2017), where it produced significant memory savings on Xbox
One's 32MB ESRAM.

### AliasingAnalysis

```rust
pub struct AliasingAnalysis<N: Id> {
    step_of: IdMap<N, usize>,
    lifetimes: DenseColumn<(usize, usize)>,
}

impl<N: Id> AliasingAnalysis<N> {
    pub fn compute<E>(
        plan: &ExecutionPlan<N>,
        dag: &Dag<N, E>,
        access: &impl Fn(N) -> &[ResourceAccess],
    ) -> Self;

    pub fn alias_groups(&self) -> Vec<Vec<ResourceId>>;
}
```

### Lifetime Computation

For each resource in the access declarations: (1) find the earliest step that
accesses it (first_step), (2) find the latest step (last_step), (3) the
lifetime interval is `[first_step, last_step]`. Two resources can alias if
their intervals do not overlap: `a.last_step < b.first_step` or
`b.last_step < a.first_step`.

### Greedy Interval Coloring Algorithm

`alias_groups()` uses greedy interval coloring:

1. Sort resources by `first_step`
2. For each resource, try to assign to an existing group where no conflict exists
3. If no group fits, create a new group

Complexity is O(n * k) where n = number of resources and k = number of groups.
Frostbite uses a more sophisticated free-list allocator tracking actual memory
sizes and alignments. loimu-graph returns groups only; the domain decides how to
share memory based on resource sizes.

The problem is equivalent to graph coloring on an interval graph. Interval
graphs are perfect graphs, so the chromatic number equals the maximum clique
size (maximum number of simultaneously live resources). Optimal interval graph
coloring runs in O(n log n): sort by start time, greedily assign colors. This
is the same algorithm as linear scan register allocation (Poletto and Sarkar,
1999).

### Consumer Applications

**Scheduler:** behavior scratch buffers. Some behaviors allocate temporary
buffers during execution. If two behaviors never run in the same step, their
scratch buffers can alias. Pre-allocate a pool of scratch buffers and assign
based on alias groups.

**Animation:** intermediate blend values. Sequential chains (A -> B -> C)
produce intermediates at A that are consumed by B then dead. A's and C's
intermediates can alias. For GPU-batched animation, aliasing reduces buffer
allocations.

**Render graph (future):** alias render targets, depth buffers, temporary
compute buffers. The original Frostbite use case.

### Aliasing Constraints

Only transient resources (created and destroyed within a single plan execution)
can be aliased. Persistent resources are never aliased. Two resources that alias
cannot both be dirty simultaneously (version stamps would conflict). Plan
computation checks for this and falls back to separate allocations when needed.

### Cache Effects

Aliasing has a positive cache effect when the lifetime gap is short: the second
resource's access finds cache lines already warm from the first resource's
access. If the gap is long, cache lines are cold regardless. Savings are in peak
memory, not throughput.

## 8. Three-Level Dirty Tracking

loimu solves dirty tracking at three granularity levels. Each level feeds the
next.

### Level 1: Resource Version Stamps

Every resource carries a hidden `version: u64` field, incremented by the write
accessor when mutation occurs. Not atomic; the DAG guarantees exclusive write
access.

```
Tick N:
  BehaviorA writes ResourceX -> version 41 -> 42
  BehaviorB reads ResourceX, sees version 42, does work

Tick N+1 (with phantom-write detection):
  BehaviorA writes ResourceX, value unchanged -> version stays 42
  BehaviorB reads ResourceX, sees version 42 (same as last tick) -> skips
```

Phantom-write detection strategies:

| Strategy | Cost | Quality |
|----------|------|---------|
| Always increment | Zero | No skipping |
| Bitwise comparison | `memcmp(old, new, size)` | Perfect for small resources |
| Hash comparison | `hash(new) != hash(old)` | Good for large resources |
| Developer hint | `.mark_unchanged()` | Zero cost, requires discipline |

loimu uses bitwise comparison for resources under 4KB (covers >95% of framework
resources). For larger resources, the developer calls `.mark_changed()`
explicitly; default is "assume changed."

Version stamps are chosen over simple dirty flags because they encode staleness:
if a downstream behavior skipped N ticks, the version stamp difference tells the
frame generation system exactly how many frames were static. This informs
frame generation ratio decisions.

### Level 2: Graph Early Cutoff

When a node in the DAG produces the same output as the last tick, all
downstream nodes that depend only on that output can skip execution.

The scheduler captures the version of every resource before and after each
behavior runs. If all written resources have unchanged versions, all downstream
behaviors that read only those resources can skip. Cost: O(behaviors *
resources_per_behavior) version comparisons.

```
[InputBehavior] -> writes InputState (v41 -> v42)
     |
[LayoutBehavior] -> reads InputState (v42), writes LayoutResult (v15 -> v15*)
     |
[RenderBehavior] -> reads LayoutResult (v15, unchanged) -> SKIP

* LayoutBehavior ran but produced the same output
```

Static declarations mean the DAG is known before execution starts. The scheduler
can precompute which behaviors to skip before running any of them. Trade-off:
static declarations cannot express data-dependent dependencies (must
conservatively assume access always occurs).

### Comparison Table

| Aspect | Salsa | Adapton | loimu |
|--------|-------|---------|-------|
| Dependencies | Runtime-discovered | Runtime-discovered | Static (declared) |
| Granularity | Per-function | Per-named-node | Per-behavior |
| Change detection | Value comparison | Memoization | Version stamps |
| Cost of tracking | Hash lookups | Allocation per node | Zero (compile-time) |

### Level 3: Tile-Based Dirty Regions

Hierarchical tile structure:

```
Level 0: Screen (1 region)
  +- Level 1: Macro tiles (64x64 pixels)
       +- Level 2: Tiles (16x16 pixels)
            +- Level 3: Pixels
```

At 4K: Level 1 has 60 x 34 = 2,040 macro tiles (256 bytes bitmask); Level 2
has 240 x 135 = 32,400 tiles (4KB bitmask). A clean macro tile means all 16
child tiles are clean; skip the entire 64x64 region without checking individual
tiles.

Dirty region propagation:

1. Behavior writes resource -> version stamp increments
2. Resource has associated screen region(s) via node markers
3. Scene tree maps nodes to screen-space bounding boxes
4. Bounding boxes mark tiles as dirty in the bitmask
5. Render pass iterates only dirty tiles
6. After rendering: clear dirty bitmask

Steps 2-4 happen at behavior exit (deferred to pass boundary). Cost is
proportional to the number of nodes that changed position or content.

### Dirty Tracker Data Structure

```
struct DirtyTracker {
    node_dirty: BitVec,
    versions_before: Vec<u64>,
    prev_dirty: BitVec,
}
```

After each execution pass: (1) compare `versions_before` with current versions,
(2) for each resource whose version changed, mark all observing nodes dirty,
(3) for each dirty node, propagate to downstream nodes, (4) swap `node_dirty`
and `prev_dirty` for the next tick.

### Generic Hierarchical Dirty Structure (DIRTY_OVERVIEW)

The DIRTY_OVERVIEW document describes a generic structure that tracks dirty
state at multiple granularity levels. Configurable depth. Each level reduces
the number of tracked elements by a factor:

```
level 0: per-element bits (finest)
level 1: groups of N elements
level 2: groups of N*N elements
...
level K: single root bit (coarsest)
```

API:

```
DirtyTracker::new(capacity, levels)
DirtyTracker::mark(index)
DirtyTracker::is_dirty(index) -> bool
DirtyTracker::is_any_dirty() -> bool
DirtyTracker::dirty_ranges(level) -> impl Iterator<Range>
DirtyTracker::clear()
```

Conventions: no heap allocation after construction; clearing is O(levels), not
O(elements); thread-safe reads, exclusive writes (matches scheduler DAG
guarantees).

Instantiations:

| Consumer | Leaf granularity | Use |
|----------|-----------------|-----|
| loimu-render / loimu-gpu | screen-space tile (4px base) | skip clean raster regions |
| loimu-tree | node ID | skip unchanged subtrees during layout |
| loimu-storage | slot index | track which slots need GPU upload |

## 9. Incremental Re-Execution

### Forward Sweep Propagation

When a node is marked dirty, all transitive successors are also marked.
`dirty_sorted` returns only dirty nodes in valid evaluation order. The
propagation is a forward sweep over topological order; each node is visited at
most once. Cost: O(V + E).

```rust
pub struct DirtyTracker<N: Id> {
    dirty: IdSet<N>,
}

impl<N: Id> DirtyTracker<N> {
    pub fn mark_dirty(&mut self, node: N, dag: &Dag<N, impl Any>);
    pub fn is_dirty(&self, node: N) -> bool;
    pub fn clear(&mut self);
    pub fn dirty_sorted(&self, dag: &Dag<N, impl Any>) -> Vec<N>;
}
```

Consumer uses:

- **Scheduler:** `RecalculateSchedulePlan` triggers full dirty (all nodes)
- **Animation:** driver target changed -> dirty propagation through blend tree
- **Shader compilation:** source file changed -> dirty propagation through
  include graph

### Version Comparison

The scheduler captures versions before and after each behavior runs. If all
written resources have unchanged versions (phantom write detection), all
downstream behaviors that read only those resources can skip. This is the
graph early cutoff mechanism.

The version comparison cost is O(behaviors * resources_per_behavior). For a
typical GUI with 100 behaviors and 5 resources per behavior, that is 500 u64
comparisons per tick; negligible.

### React Comparison

React: O(n) in tree size (walks the entire tree looking for changes). loimu
inverts the model: track what changed, render only that. Version stamp check
is O(1) per resource. Dirty region check is O(dirty tiles). For a 10,000-node
UI with 3 nodes changing: React walks 10,000 nodes; loimu checks 3 version
stamps and processes about 10 tiles.

## 10. Fusion Integration

Fusible scheduling units (`each` blocks) push per-element operations to the
storage engine queue. The storage engine groups operations by shared read
columns and view membership, then fuses compatible operations into single
iteration passes.

### Fusible vs Opaque

These are separate concerns:

- **Fusible:** storage engine question (can merge iteration passes?)
- **Opaque:** scheduler question (can see inside the unit?)

An `each` block is a fusible scheduling unit. A `tree` block, `bind`, `compute`,
`draw`, `copy`, and `emit` are non-fusible (standalone) units.

### Three Fusion Tiers

| Tier | Access modes | Fusibility |
|------|-------------|------------|
| 1 (Unconditional) | Pinned, Approx, Defer | Always fusible |
| 2 (Ordered) | Immediate, MutImmediate | Per-element order preserved |
| 3 (Chaos) | Sample, Post | Fusible, relaxed ordering |

Constraint: two `MutImmediate` writers on the same column are NOT fusible; they
serialize.

### Conditional Guards and Fusion

Guarded `each` blocks (an `each` inside an `if config.some_flag` guard) are
pessimistically allocated as scheduling units with full access declarations. The
guard is evaluated once at dispatch time: true means the unit runs normally and
is fusible; false means it returns immediately with zero iteration. The fusion
engine excludes false-guarded units from fused groups.

### Sole-Writer Detection

The scheduler detects at plan computation time whether a resource has a single
writer across all scheduling units. Sole-writer resources use simpler rollback
on fault: snapshot restore instead of undo journal.

## 11. Over-Synchronization Analysis

### Step-Based vs Wavefront

Step-based execution can leave parallelism on the table. After toposort,
`compute_plan` groups nodes into steps. Two nodes share a step if: (1) no
edge exists between them and (2) they have the same topological level. This is
correct, but in complex graphs, nodes at the same level with very different
execution times cause the step barrier to bottleneck on the slowest node.

Example where step-based is optimal:

```
A -> C
B -> D
C -> E

Step 0: A, B (parallel)
Step 1: C, D (parallel)
Step 2: E
```

But for irregular DAGs, step-based execution leaves 10-15% parallelism on the
table (Taskflow benchmarks).

### Wavefront (Counter-Based) Execution

Naughty Dog's fiber system (Gyrling, GDC 2015) avoids step barriers: each task
has an atomic dependency counter, and tasks become ready the instant their
counter reaches zero.

```
for each worker thread:
    loop:
        task = local_queue.pop() or steal_from(other_worker)
        execute(task)
        for successor in task.successors:
            if successor.dependency_counter.decrement() == 0:
                local_queue.push(successor)
```

Maximum parallelism at the cost of harder profiling (non-deterministic order),
cache-unfriendly stealing, and more complex implementation.

### Barrier Cost

Step boundaries have a cache cost. Each barrier is a synchronization point
where worker threads drain store buffers (memory fence) and the next step's
nodes may access completely different memory regions, thrashing L2/L3 caches.
Drepper measurement: L3 miss penalty is 30-60ns on modern hardware.

Within a step, ordering nodes to maximize cache reuse (nodes accessing the same
resources run on the same core) is a scheduling hint, not a graph feature.

### Fiber-Based Execution

Fibers run tasks, yield when encountering a dependency, resume when the
dependency completes. Attractive for behaviors doing I/O or waiting on GPU
fences. Complex to implement correctly (signal handling, TLS, debugger support).
Deferred to post-1.0 in loimu.

### GPU Work Graphs

D3D12 Work Graphs (SM 6.8) let the GPU generate work for itself.
Hardware-level implementation of the software wavefront pattern. A work graph
node can spawn child nodes on the GPU; this is hardware-accelerated dynamic
subgraphing. Not universal yet; deferred.

### loimu's Position

For loimu's typical workloads (dozens to hundreds of behaviors), the step model
is a reasonable starting point. If profiling reveals barrier bottlenecks, the
executor can switch to counter-based dispatch without changing the graph or plan
APIs. The three-phase separation (Graph/Plan/Execute) makes this swap possible:
the `ExecutionPlan` is produced by the plan phase, and the execute phase
consumes it. A different execute strategy just reads the same plan differently.

## 12. Design Decisions

### No Conditional Edges

Conditional edges (runtime if/else branching in the DAG) were rejected.
Reasons:

- Platform-specific paths are compile-time concerns, not runtime graph branches.
  Different platforms use different compile pipeline types.
- Feature toggles cause graph rebuild. Hash-based caching makes rebuild
  near-free when graph structure is unchanged. No production renderer toggles
  features at frame frequency; hysteresis and adaptation algorithms operate on
  multi-hundred-frame timescales.
- LOD selection happens inside opaque subgraphs, not as conditional edges.
- Static analyzability is core to loimu. The scheduler's optimization model
  (DAG analysis, lane assignment, version stamps, early release) depends on
  knowing the full graph shape at plan time. Conditional edges break this
  contract.

Taskflow needs conditional tasks because it is a general-purpose library.
loimu's graph serves specific consumers with known structure.

### Pluggable Compile Phase

The compile phase is the intentional extension point. The graph structure and
plan computation are fixed; the compile phase is where domains diverge. The
scheduler adds lane assignment and version stamp early release. The render graph
adds barrier insertion, transient allocation, and resource aliasing. Mobile
render adds tile-memory pass merging. Each is a compose-able pass with typed
input and output, monomorphized at compile time.

### Subgraph Boundaries (Computed Portals)

Portals are computed, not declared. The parent scheduler inspects the
subgraph's internal DAG to compute claim and release points. This is a
non-obvious design choice: most systems treat subgraphs as opaque. loimu
defaults to transparent subgraphs and requires explicit opt-in for opacity.

### Inter-Frame Resource Persistence

Resources persisting across frames (e.g., Nanite's HZB) are handled by the
existing resource system, not by inter-frame graph edges. Signal bus
double-buffering handles fire-and-forget cross-frame data. Persistent resources
are ordered by DAG read-after-write edges within a single frame's graph.

### Multi-GPU as Scheduler Extension

Multi-GPU partitioning is structurally analogous to lane dispatch. Each node
gets a `target: GpuId` annotation (metadata). The compile phase detects
cross-GPU resource dependencies and inserts transfer nodes. Not immediate
priority but does not require architectural changes.

### Classify-Reserve-Scatter as Framework Utility

The GPU counting sort pattern (classify items by category, reserve contiguous
buffer space, scatter items into slots) appears 3+ times in Nanite. Provided as
a reusable compute utility in loimu-gpu, not loimu-graph; it is a GPU compute
primitive, not a graph concern.

### Per-Unit Fault Handling

Each scheduling unit has independent fault handling:

1. Unit panics (native) or traps (WASM)
2. Sole-writer resources: restore from snapshot buffer
3. Shared-writer resources: replay undo journal in reverse
4. `BehaviorFaulted { id, unit, reason }` signal emitted
5. Remaining units in the handler are skipped

Per-unit WASM fuel budget: each scheduling unit gets its own fuel allocation
(10M default). Fuel exhaustion in one unit does not affect other units. Uses
snapshot-then-flush to discard uncommitted writes.

## 13. Relevance to hilavitkutin

loimu's DAG infrastructure is the most directly portable piece to
hilavitkutin. The three-phase pattern (Graph/Plan/Execute) is universal: polka
builds a dependency graph from transform declarations, saalis builds one from
connector and query dependencies, loimu builds one from behavior access
declarations. All three need topological sort, parallel step grouping, and
domain-specific compile passes.

### What ports directly

**`Dag<N, E>` with typed IDs, forward/reverse adjacency, online cycle
detection.** This is a pure data structure with no domain assumptions. Node IDs
come from `define_id!` (or hilavitkutin's equivalent), edge data is generic.
The operations (add_node, add_edge with cycle rejection, successors,
predecessors) are exactly what all three consumers need.

**`ExecutionPlan` and `Step` with strategy hints.** The plan structure is
domain-agnostic. Steps group independent nodes. Strategy hints carry per-node
metadata without knowing its type. polka uses hints for morsel dispatch;
saalis uses hints for I/O vs CPU routing; loimu uses hints for lane assignment.

**`compute_plan` with filter predicate.** Filter nodes, toposort survivors,
group into parallel steps. All three consumers need this.

**`DirtyTracker` with forward sweep propagation.** Mark dirty, propagate to
transitive successors, return dirty nodes in topological order. O(V+E). polka
uses it for incremental rebuild (only re-run transforms whose inputs changed);
saalis uses it for event-triggered cascading; loimu uses it for behavior
skipping.

**`AliasingAnalysis` with greedy interval coloring.** Resource lifetime
intervals, non-overlapping aliasing, group computation. polka can alias
intermediate IR buffers between transform stages; saalis can alias scratch
buffers between query phases.

**`CompilePass` trait and tuple-based pipeline.** The extension point. polka
defines passes for morsel chunking and column dispatch. saalis defines passes
for I/O batching and retry insertion. loimu defines passes for lane assignment,
barrier insertion, and resource aliasing. Each is a typed, monomorphized
pipeline.

### What simplifies for polka/saalis

**Access modes:** loimu has seven; hilavitkutin needs two (Read, Write). The
DAG edge construction logic is the same, just with a smaller enum. loimu's
Pinned/Immediate/Sample/Post/Approx/Defer are all projections of timing
semantics that polka and saalis do not have. A future loimu layer can map its
seven modes to hilavitkutin's two for DAG construction, then add ordering
constraints on top.

**Subgraph portals:** loimu's computed portals (claim/release points) are
valuable but complex. polka's pipeline is flat; no subgraphs needed in v1.
saalis has two-level grouping (subsystem -> task) but the inner level is simple
enough that opaque subgraphs suffice. hilavitkutin should support subgraphs
and `NodeKind::Subgraph(ExecutionPlan<N>)` from day one, but portal computation
can be a loimu-specific compile pass rather than a hilavitkutin primitive.

**Field-granular access analysis:** loimu does field-level inference from
handler bodies via proc macros. polka and saalis declare access at resource
granularity (Read<T> / Write<T>). hilavitkutin should accept `AccessSet` as
input to edge construction but not mandate how the access set is derived. loimu
derives it from proc macros; polka derives it from trait declarations; saalis
derives it from query schemas.

**Tile-based dirty regions:** purely a rendering concern. hilavitkutin's dirty
tracking stops at level 2 (graph early cutoff). Level 3 (tile dirty regions)
stays in loimu-render.

**Version stamps:** the concept is universal (track whether a resource actually
changed). hilavitkutin should own the version stamp type and the early cutoff
algorithm. Phantom-write detection (bitwise comparison for small values, explicit
marking for large values) is a storage concern that each consumer can implement
differently.

**Fusion:** the three-tier fusion model is storage-engine specific. hilavitkutin
provides fusible/non-fusible as a scheduling unit property. The storage engine
(loimu-storage, polka's column store, saalis's DB bridge) decides how to merge
fusible units. hilavitkutin does not own fusion logic.

**Frame generation:** not relevant to polka or saalis. Stays entirely in loimu.

### Summary: hilavitkutin scope derived from loimu-graph

| loimu-graph feature | hilavitkutin | loimu layer | polka layer | saalis layer |
|---|---|---|---|---|
| `Dag<N, E>` | Yes, core | - | - | - |
| `ExecutionPlan` / `Step` | Yes, core | - | - | - |
| `compute_plan` | Yes, core | - | - | - |
| `DirtyTracker` + propagation | Yes, core | - | - | - |
| `AliasingAnalysis` | Yes, core | - | - | - |
| `CompilePass` pipeline | Yes, core | - | - | - |
| Online cycle detection | Yes, core | - | - | - |
| `NodeKind::Subgraph` | Yes, core | - | - | - |
| Computed portals | No (compile pass) | Yes | No | No |
| 7 access modes | No (Read/Write only) | Yes (maps to R/W) | No | No |
| Field-granular inference | No (accepts AccessSet) | Yes (proc macro) | No | No |
| Tile dirty regions | No | Yes | No | No |
| Fusion tiers | No (fusible flag) | Yes (storage) | Maybe | No |
| Frame generation | No | Yes | No | No |
| Version stamp type | Yes, core | - | - | - |
| Phantom-write detection | Yes, core | - | - | - |
| Step-based executor | Yes, core | - | - | - |
| Wavefront executor | Future | Future | No | Maybe |

The boundary is clear: hilavitkutin owns the DAG, plan computation, dirty
tracking, aliasing analysis, and the compile pass extension point. Everything
above that line is a consumer concern. loimu adds seven access modes, field
inference, computed portals, tile dirty regions, and fusion as its own layer.
polka adds morsel chunking and column dispatch. saalis adds I/O batching and
retry. Each builds on the same foundation without contaminating the shared
core with domain-specific concerns.
