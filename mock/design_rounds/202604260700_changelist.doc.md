## Purpose

First hilavitkutin round. Lays the consumer-facing type surface that
`hilavitkutin` (engine), `hilavitkutin-build`, `hilavitkutin-ctx`,
`hilavitkutin-str`, and downstream consumers (polka-dots, saalis,
loimu) compose against. No engine internals, no dispatch, no
scheduling — only the contracts.

Contract surface (all in `hilavitkutin-api`):

1. **Store types** — `Resource<T>`, `Column<T>`, `Virtual<T>` as
   PhantomData-backed ZSTs. Scheduler-facing identity is the
   type itself.
2. **`StoreId`** — dense runtime index, wraps an arvo fixed-point
   newtype (not raw u16). Plan-time assigned.
3. **`AccessMask`** — alias to arvo-bitmask `Mask64` (≤64 stores)
   or `Mask256` (≤256 stores). Single-instruction bitwise ops.
4. **`AccessSet`** — sealed trait on tuples of store types.
   Compile-time store descriptor; monomorphises to `AccessMask` at
   plan time.
5. **`Contains<S>`** — compile-time membership trait; accessing an
   undeclared store is a compile error.
6. **`ColumnValue`** — `Copy + 'static` + `BIT_WIDTH: usize`.
   Blanket impl via `min_specialization` defaults to
   `size_of::<Self>() * 8`; arvo sub-byte types specialise.
7. **`WorkUnit<Schedule = Always>`** trait —
   `Read: AccessSet`, `Write: AccessSet`,
   `Hint: SchedulingHint = (Immediate, Atomic, Normal)`,
   `const COMMUTATIVE: bool`, `fn execute(&self, ctx)`.
8. **Schedule markers:** `Always`, `On<V>`.
9. **`Context<'a, R, W>`** — `&self` receiver (load-bearing for
   fused-WU noalias correctness). Ops: `read<T>(i) -> T`,
   `write<T>(i, val)`, `resource<T>() -> &T`, `fire<V>()`,
   plus `each` / `batch` / `reduce` registration.
10. **Scheduling hints** — 3-axis: `Urgency`, `Divisibility`,
    `Significance`, each as `repr(u8)` ZST carriers. Discriminants
    per DESIGN. `SchedulingHint` trait on tuples.
11. **Platform traits:** `MemoryProvider`, `ThreadPool`, `Clock`
    (all `Send + Sync + 'static`, generic-monomorphised, not dyn).
12. **Resource model:** `Field<T>`, `Seq<T, const N: usize>`,
    `Map<K, V, const N: usize>`. No dynamic collections.
13. **Virtual flag system** — bit-packed fire flags on arvo-bitmask
    types, epoch-based reset. Per-(virtual, consumer) tracking.

## Comparison to deprecated changelist

Supersedes `202603241500_changelist.doc.deprecated.md` — the seed
doc CL that covered both hilavitkutin AND the arvo crate family
before they were split into separate repos. That CL's arvo
portion has been delivered across arvo's 10 closed rounds
(202604191900..202604260500); this round delivers the
hilavitkutin-api portion, retaining every type, trait, and
semantic the deprecated CL called for.

### Resource<T> / Column<T> / Virtual<T>

Unchanged from the deprecated CL. Still PhantomData ZSTs; scheduler
still keys on type identity; still supports Always / On<V> schedule
variants via the WorkUnit generic parameter.

### AccessMask / AccessSet / Contains

Unchanged. AccessMask is the arvo-bitmask Mask64 / Mask256 alias
(now provably available — arvo dev branch carries both types post
rounds 202604210000 + 202604260000). AccessSet is the tuple trait
as before; Contains is its membership compile-time check.

### ColumnValue

Unchanged. Still `Copy + 'static`, still has `BIT_WIDTH` const, still
uses `min_specialization` for the blanket default. arvo sub-byte
types (UFixed<1,0>, etc.) specialise as before.

### WorkUnit

Unchanged trait shape. `Read`, `Write`, `Hint` associated types;
`COMMUTATIVE` const; `execute(&self, ctx)`. No NAME const (deferred
CL dropped this; staying dropped).

### Context

Unchanged. `&self` receiver still load-bearing for noalias
correctness across fused WUs. `read` / `write` / `resource` / `fire`
/ `each` / `batch` / `reduce` all present.

### Scheduling hints

Unchanged 3-axis model with DESIGN discriminants (Urgency 255/170/
85/0, Divisibility 255/128/0, Significance 255/192/128/64/0). Lower
discriminant = higher priority.

### Platform traits

Unchanged trait shapes. Note in DESIGN says arvo newtype parameters
(`ByteCount`, `Alignment`, `CoreCount`, `Nanoseconds`) are the
final API target but exact mapping is src-CL decision — the src CL
will either land the newtype parameters directly or note any
deferrals.

### Resource model

Unchanged. `Field<T>`, `Seq<T, N>`, `Map<K, V, N>` all const-sized,
no dynamic collections.

### Virtual flag system

Unchanged type-surface story: bit-packed flags on arvo-bitmask
types, epoch-based reset, per-(virtual, consumer) tracking. The
bit-assignment algorithm itself lives in the engine round (#35).

### Deprecated CL's arvo portion

Already delivered: `arvo`, `arvo-bits`, `arvo-bitmask`, `arvo-graph`,
`arvo-sparse`, `arvo-comb`, `arvo-spectral`. All seven crates ship
with `const N: Cap` surfaces, `cap_size` helper, `core::ops` on
float wrappers, and the full strategy-marker / newtype family. See
arvo's closed rounds 202604191900..202604260500.

### What's different from the deprecated CL

One split in repo layout: arvo and hilavitkutin are separate repos
with separate mockspaces now. Cross-repo deps flow from
hilavitkutin-api → `arvo`, `arvo-bits`, `arvo-bitmask` via git dep
pinned at `branch = "dev"`. No other material difference from the
deprecated CL's hilavitkutin-api section.
