## Purpose

logging two deferrals. Not a feature round — the engine public
surface is otherwise frozen as of 5a4/5a6. These fixes unblock
future multi-threaded call sites and reduce the chance that a
caller mis-reads the semantic guarantees of the compiler-fence
wrappers.

## Deliverables

### Fix 1 — Send/Sync for ResourcePtr + ColumnPtr

`ResourcePtr<T>` and `ColumnPtr<T>` (both in
`resource/provenance.rs`) are `#[repr(transparent)]` newtypes over
`core::ptr::NonNull<T>`. `NonNull` deliberately opts out of the
auto `Send`/`Sync` impls, so the newtypes inherit the opt-out. The
first multi-threaded call site (convergence buffers in thread
5a4, dispatch records from 5a3) will hard-block compilation
without these.

The aliasing discipline is enforced at a different level: the
access-set parameter on the owning cache (`resource::cache`) gates
who can construct these pointers, and the plan-stage analysis
guarantees no two WUs write-overlap. There is no thread-local
aliasing concern beyond what the owning cache already enforces —
so the hand-written impls are a sound restoration of the usual
`*mut T`-is-Send-if-T-is-Send pattern.

Impl shape (for each pointer newtype):

```rust
// SAFETY: The pointer is valid for the thread's lifetime when
// T: Send/Sync. The aliasing discipline is enforced at the type
// level by the access set parameter on the owning cache — there
// is no thread-local aliasing concern.
unsafe impl<T: Send> Send for ResourcePtr<T> {}
unsafe impl<T: Sync> Sync for ResourcePtr<T> {}
```

### Fix 2 — Clarify compiler_fence naming

`intrinsics::compiler_fence_release()` and
`intrinsics::compiler_fence_acquire()` each emit exactly one
`core::sync::atomic::compiler_fence(Ordering::…)` call — a
compile-time reorder barrier, never a hardware memory barrier.
The names mirror atomic orderings, which could mislead a caller
into thinking they pair with Acquire/Release *atomic* ops on
another thread (they don't — no HW barrier, so on weak-memory
architectures like ARM or RISC-V they are insufficient for
thread-to-thread handoff).

Two options considered:

- **A (non-breaking)** — prepend a prominent `# WARNING —
  compiler fence only, not a hardware barrier` block to each
  function's doc comment, spelling out that the compiler-level
  reorder barrier carries no CPU-level ordering, and directing
  callers to atomic Release/Acquire loads/stores for
  thread-to-thread handoff.
- **B (breaking)** — rename to `compiler_reorder_barrier_release`
  / `compiler_reorder_barrier_acquire`.

Pick **A**. The engine's public surface is frozen post-5a6 and
renaming would churn every current + imminent consumer for a
purely cosmetic gain. The warning block is prominent enough to
catch a careful caller at the doc-comment level and is visible
in rustdoc.

### BACKLOG — engine skeleton post-review notes

Append a new `## Engine skeleton post-review notes` section to
`BACKLOG.md.tmpl` with two items:

- **POSIX fn*/void* cast in `platform/os.rs` `spawn_fn`.** The
  current pattern smuggles `fn()` via `usize` as a
  `pthread_create` argument. POSIX doesn't guarantee that `fn*`
  and `void*` have matching size; all Rust tier-1 targets happen
  to comply in practice, but the threading-std implementation
  round should switch to `Box<fn()>` or a static work-queue slot
  to be POSIX-clean.
- **`OnceLock` MSRV in `platform/std_tier.rs`.** Uses
  `std::sync::OnceLock` which requires Rust 1.70+. The workspace
  does not yet pin a toolchain. When a toolchain policy is
  decided, either pin 1.70+ in `rust-toolchain.toml` or replace
  with an `AtomicBool` init-once pattern.

## Impl-time decisions

1. **Warning, not rename.** Public surface is frozen post-5a6.
   Breaking the two `compiler_fence_*` names for a cosmetic gain
   costs more than it saves.
2. **Send/Sync impls mirror the standard pattern.** `NonNull`
   opts out by convention, not by correctness — `*mut T` is
   `Send if T: Send` and `Sync if T: Sync` when the aliasing
   discipline is enforced elsewhere (which it is, via the
   access-set parameter on the owning cache).
3. **Optional tiny compile-time test.** A one-liner
   `fn require_send<T: Send>() {}` + monomorphisation check
   asserts the bound holds — cheap, catches a regression
   cleanly. Keep if it lands without friction.
