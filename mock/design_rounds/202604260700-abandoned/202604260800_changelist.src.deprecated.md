## Deliverables

### 1. Store types

```rust
#[repr(transparent)]
pub struct Resource<T>(core::marker::PhantomData<T>);
#[repr(transparent)]
pub struct Column<T>(core::marker::PhantomData<T>);
#[repr(transparent)]
pub struct Virtual<T>(core::marker::PhantomData<T>);
```

All three are zero-sized marker types. `Copy + Clone + Default`
derived via phantom-data pattern. Consumer constructs them only at
the type level; `PhantomData::default()` not exposed publicly.

### 2. StoreId

`pub struct StoreId(pub arvo::newtype::USize)`. `#[repr(transparent)]`,
`Copy + Clone + Eq + PartialEq + Hash + Debug + Default`. Plan-time
dense index; wraps USize so it composes with arvo-bitmask NodeId
algebra.

### 3. AccessMask

`pub type AccessMask = arvo_bitmask::Mask64`. Alias only this round
(Mask256 tracked in BACKLOG). Documented ceiling: 64 stores per
pipeline at this scope.

### 4. AccessSet

Sealed trait on tuples up to 12 elements:

```rust
pub trait AccessSet: sealed::Sealed + 'static {
    const LEN: usize;
    fn to_mask() -> AccessMask;  // monomorphised at plan time
}
```

Impl blocks for `()`, `(A,)`, `(A, B)`, ..., `(A, B, C, D, E, F, G, H,
I, J, K, L)` via macro. Each impl is one-line via the macro. `to_mask`
stub-only for this round (plan-time engine sets mask bits; surface
returns `AccessMask::empty()` at the api-only level — engine overrides
via a separate trait or associated helper when it lands).

### 5. Contains

Sealed trait parameterised over a tuple and a single store type:

```rust
pub trait Contains<S>: AccessSet {}
```

Impl blocks via macro: for each tuple arity 1..=12, for each position,
`impl<S0, S1, ...> Contains<S0> for (S0, S1, ...)`, etc. The compiler
enforces membership at monomorphisation — a `where R: Contains<Column<T>>`
bound on `Context::read` fails to compile when the WU's Read tuple
lacks the store.

### 6. ColumnValue

```rust
#![feature(min_specialization)]
pub trait ColumnValue: Copy + 'static {
    const BIT_WIDTH: usize;
}

// Default specialisation
impl<T: Copy + 'static> ColumnValue for T {
    default const BIT_WIDTH: usize = core::mem::size_of::<Self>() * 8;
}
```

arvo sub-byte specialisations: UFixed<IBits(1), FBits::ZERO, Hot>,
UFixed<IBits(2), FBits::ZERO, Hot>, UFixed<IBits(4), FBits::ZERO,
Hot>. Each with `const BIT_WIDTH = N`.

### 7. WorkUnit trait + schedule markers

```rust
pub struct Always;
pub struct On<V>(core::marker::PhantomData<V>);

pub trait WorkUnit<Schedule = Always>: Send + Sync + 'static {
    type Read: AccessSet;
    type Write: AccessSet;
    type Hint: SchedulingHint;
    const COMMUTATIVE: bool = false;
    fn execute(&self, ctx: Context<Self::Read, Self::Write>);
}
```

`type Hint` has a default via `#![feature(associated_type_defaults)]`;
the default is `(Immediate, Atomic, Normal)`.

### 8. Context<'a, R, W>

```rust
pub struct Context<'a, R: AccessSet, W: AccessSet> {
    _phantom: core::marker::PhantomData<(&'a (), R, W)>,
    // engine adds pointer-bearing fields in its round; api stub has none
}
```

`&self` receiver on all methods (load-bearing per DESIGN).
api-stub impl of read/write/resource/fire/each/batch/reduce panics
with "engine not linked" — the engine crate overrides via a
concrete type alias + inherent impl, not via this stub. The contract
surface fixes the method shape for consumers.

`unsafe fn read<T: ColumnValue>(&self, i: arvo::newtype::USize) -> T
where R: Contains<Column<T>>`
`unsafe fn write<T: ColumnValue>(&self, i: arvo::newtype::USize, v:
T) where W: Contains<Column<T>>`
`fn resource<T: 'static>(&self) -> &T where R: Contains<Resource<T>>`
`fn fire<V: 'static>(&self) where W: Contains<Virtual<V>>`

Op registration: `fn each<F>(&self, f: F)`, `fn batch<F>(&self, f:
F)`, `fn reduce<A, F>(&self, init: A, f: F) -> A`. Stub bodies
compile but panic.

### 9. Scheduling hints

Three axes, each a sealed trait with `repr(u8)` ZST markers. Direct
transcription of DESIGN:

- Urgency: Immediate(255), Steady(170), Relaxed(85), Deferred(0).
- Divisibility: Atomic(255), Adaptive(128), Interruptible(0).
- Significance: Critical(255), Important(192), Normal(128),
  Opportunistic(64), Optional(0).

Each carrier is `#[repr(u8)] pub struct Immediate;` etc. with a
`const VALUE: u8 = 255;` on the trait impl. `SchedulingHint` trait
on 3-tuples: `(U: UrgencyValue, D: DivisibilityValue, S:
SignificanceValue)`. Lower discriminant = higher priority.

### 10. Platform traits

```rust
pub trait MemoryProvider: Send + Sync + 'static {
    unsafe fn allocate(&self, len: arvo::newtype::USize,
                       align: arvo::newtype::USize) -> *mut u8;
    unsafe fn deallocate(&self, ptr: *mut u8, len: arvo::newtype::USize);
    unsafe fn protect(&self, ptr: *mut u8, len: arvo::newtype::USize,
                      read: arvo::newtype::Bool, write: arvo::newtype::Bool);
}

pub trait ThreadPool: Send + Sync + 'static {
    fn spawn(&self, f: impl FnOnce() + Send + 'static);
    fn worker_count(&self) -> arvo::newtype::USize;
}

pub trait Clock: Send + Sync + 'static {
    fn now_ns(&self) -> u64;
}
```

Parameter newtype mapping (pre-committed this round):
- `len` / `align`: `arvo::newtype::USize`.
- `read` / `write` flags: `arvo::newtype::Bool`.
- `worker_count` return: `arvo::newtype::USize`.
- `now_ns` return: raw `u64` (no `Nanoseconds` newtype exists in arvo
  L0 yet; a consumer-side alias is trivial. Adding `Nanoseconds` to
  arvo belongs to a separate L0 round if a consumer demands it).

### 11. Resource model

```rust
#[repr(transparent)]
pub struct Field<T: ColumnValue>(core::marker::PhantomData<T>);
#[repr(transparent)]
pub struct Seq<T, const N: arvo::newtype::Cap>(core::marker::PhantomData<T>);
#[repr(transparent)]
pub struct Map<K, V, const N: arvo::newtype::Cap>(
    core::marker::PhantomData<(K, V)>,
);
```

All three are ZST markers. Seq/Map take `const N: Cap` with
`where [(); arvo_bitmask::cap_size(N)]:` bound when construction
helpers are added; this round ships marker types only, no runtime
construction (engine concern).

### 12. Virtual flag system (surface only)

Virtual<T> marker is covered above. The flag state / bit-assignment
machinery is engine-side. This crate ships the type-surface
identity only; engine (domain 10) implements the runtime.

- AccessSet::LEN values for arities 0..=6.

## Impl-time decisions (pre-committed)

1. **PhantomData for store types.** All markers are
   `PhantomData<T>` tuples with `#[repr(transparent)]`. Zero size,
   composable with generic-const contexts.

2. **sealed module pattern.** Use the arvo `Ieee`-style sealed
   trait for `AccessSet` and `SchedulingHint` subtraits.

3. **Macro-expanded tuple impls.** AccessSet and Contains use a
   declarative macro to produce impls for arities 0..=12. 12 is
   the ceiling per hilavitkutin domain 09 guideline (WUs ≤8 columns
   with headroom).

4. **Context::read/write panic-stub.** The api crate provides the
   method signatures and doc comments. Runtime panic body with
   message "hilavitkutin-api Context stub: link against the
   hilavitkutin engine crate for execution". Engine crate ships a
   type-identical `Context` in its own module that consumers re-
   export? No — engine patches behaviour via a concrete wrapping
   type. api Context surface + engine Context concrete runtime
   together compose the full execution path.

5. **Virtual<T>'s fire flag.** api ships `Virtual<T>` ZST; the
   `ctx.fire<V>()` method shape; nothing else.

6. **Sealing strategy.** `sealed::Sealed` marker in a private
   `sealed` module. Public traits `AccessSet`, `Contains`,
   `UrgencyValue`, etc. all extend `sealed::Sealed` so external
   consumers cannot implement them.

7. **No `alloc::*`.** Every place DESIGN says `Vec` (even in
   sketches) uses const-generic arrays or PhantomData.

8. **`unsafe fn read/write` keywords.** Per DESIGN. Consumers
   pinky-swear ownership discipline; the engine proves it via plan-
   time analysis.

9. **Hint default.** `type Hint = (Immediate, Atomic, Normal)` via
   `#![feature(associated_type_defaults)]`. WU authors can override.

10. **No `Name: &'static str` or identity strings.** DESIGN
    explicitly drops this — WU identity is the type itself.

11. **No visibility macro.** hilavitkutin-ctx's pre-existing
    `no-bare-pub` lint requires `#[public_api]` / `#[internal_api]`
    on every `pub`. hilavitkutin-api has no such enforcement in its
    lint config this round — stays with plain `pub`. If that lint
    applies repo-wide in a later round, a separate cleanup round
    applies the macro uniformly.
