# Topic: arvo-refit adoption + #290 lint:allow sweep

**Date:** 2026-04-30
**Phase:** TOPIC
**Scope:** Adopt arvo's new `Widen<T>` plus `BitsRefitCtor` API
(landed in arvo PR #40, round 202604300826 + 202604301700) and
drop the four `lint:allow tracked: #290` annotations that were
present because arvo previously lacked widen machinery. Also rename
the `arvo-narrow` dep entry to `arvo-refit` per arvo round
202604301700.
**Source topics:** Task #292.

## Why

Arvo PR #40 closed three deferred items that this repo needed:

1. `Widen<T>` trait declared in `arvo-bits-contracts` next to
   `Narrow<T>`. Cross-primitive Widen impls cover the unsigned and
   signed widening matrix.
2. `BitsRefitCtor::from_narrowed<Src>(src)` and `from_widened<Src>(src)`
   ergonomic constructors on `Bits<N, S, Sign>`. Source type is
   inferred at the call site.
3. The `arvo-narrow` re-export crate renamed to `arvo-refit` to
   name the bit-width refit domain (both narrow and widen) rather
   than one of the two directions.

This repo has four sites annotated `lint:allow(no-bare-numeric)`
with `tracked: #290` plus reason text "arvo lacks a Widen
counterpart to Narrow":

- `mock/crates/hilavitkutin-persistence/src/archive_str.rs:29` :
  `handle.id().to_raw() as u64` widens a `Bits<28, Hot>` u32
  carrier to a `ContentHash` u64 carrier.
- `mock/crates/hilavitkutin-persistence/src/archive_str.rs:34` :
  `Str::ID_MASK.to_raw() as u64` widens a u32 mask to AND with a
  u64 hash.
- `mock/crates/hilavitkutin-persistence/src/archive_str.rs:65` :
  `masked_bits.to_raw() as u64` widens a `Bits<28, Hot>` to
  `ContentHash` u64.
- `mock/crates/hilavitkutin-str/src/macros.rs:23` and `:28` (twin
  sites inside the `str_const!` macro) : `(const_fnv1a($s) &
  0x0FFF_FFFF) as u32` narrows a u64 hash to a u32 carrier for
  `Bits<28, Hot>::from_raw`. The note read "arvo lacks a Widen
  counterpart to Narrow" but the actual operation is narrow
  (mask + cast); the lint-allow was triggered by the bare numeric
  expression around the `as u32` cast, and the sweep can replace
  the whole pattern with `Bits::<28>::from_narrowed(const_fnv1a($s))`
  using the new ergonomic constructor.

The new arvo API gives us a way to write each site without bare
numeric casts. The sweep replaces the lint-allowed casts with
explicit `widen_to` / `from_narrowed` / `from_widened` calls, then
removes the `lint:allow` annotations.

## Decisions

### A. Use `widen_to` directly for u32 plus u64 sites

The three persistence sites widen `Bits<28, Hot>` (u32-backed) or
the `Str::ID_MASK` u32 to u64 to feed `ContentHash::from_raw(u64)`.
Two paths could land:

1. `bits.widen_to::<u64>()` then `ContentHash::from_raw(widened)`.
2. `ContentHash::from_widened(bits)` if the cross-domain Bits to
   primitive direct impl is in scope (it is, via arvo PR #40).

`ContentHash` is a typed alias for `Bits<64, Hot>` (per
`arvo-hash`); the `from_widened` constructor takes any `Src` such
that `Src: Widen<u64>` (where u64 is the carrier). This is the
ergonomic path: source type inferred, no turbofish, no intermediate
named binding.

Decision: prefer `ContentHash::from_widened(bits)` where the
expression is one widen step. For sites that combine a widen with
other operations (the mask AND), use `bits.widen_to::<u64>()` plus
explicit operator chaining so the bit-pattern operations remain
visible at the call site.

### B. Use `from_narrowed` for the macro narrow sites

The `str_const!` macro currently writes
`Bits::<28>::from_raw((const_fnv1a($s) & 0x0FFF_FFFF) as u32)`.
The new `Bits::<28>::from_narrowed(const_fnv1a($s))` does the same
work via the const-trait `Narrow<u32>` impl on u64: mask to N bits,
cast to destination carrier. The `0x0FFF_FFFF` magic number
disappears (`Narrow` derives the mask from N).

Decision: replace both macro sites with `from_narrowed`.

### C. Rename `arvo-narrow` dep to `arvo-refit`

Workspace `Cargo.toml` entry plus per-crate `Cargo.toml` deps plus
the two `use arvo_narrow::Narrow;` imports flip. The arvo facade
re-export is also live (`arvo::Widen`, `arvo::Narrow`, etc), so a
secondary refactor could route through the facade; this round
keeps the direct `arvo_refit::*` imports for parity with the prior
shape (changing the import path is the only real edit in those
two files).

Decision: rename the dep entries and update the imports. Stop
short of routing through the facade in this round.

### D. Workspace dep ref

The workspace `Cargo.toml` currently pins arvo at `branch = "dev"`.
After arvo PR #40 merged at commit `52261d8` on dev, a `cargo
update -p arvo-refit` (and siblings) picks up the new commit. The
sweep's commit will include the resulting `Cargo.lock` bump.

Decision: keep the `branch = "dev"` ref. Lockfile bump is part of
the round's src changes.
