## What the reviewer flagged

1. `mock/crates/hilavitkutin/src/scheduler/mod.rs:88` `pub fn add<W: 'static>` accepts any `'static` type. The original round's doc CL pinned `W: WorkUnit` (see `mock/design_rounds/202605010900/202605010900_changelist.doc.lock.md` "Method shapes" section, plus `mock/research/sketches/app-builder-typestate-202605010900/FINDINGS.md` "What the DOC CL writes"). The shipped source drifted from the spec. Result: `.add::<NotAWu>()` compiled silently and only surfaced at `.build()` as a `Buildable` resolution failure naming cons-list shapes rather than the missing `WorkUnit` impl.

2. Platform-API methods `.memory<M: 'static>`, `.threads<P: 'static>`, `.clock<C: 'static>` accept any `'static`. The platform contracts in `hilavitkutin-api::platform` (`MemoryProviderApi`, `ThreadPoolApi`, `ClockApi`) were intended to bind these, per the engine's role as the platform-API consumer. Same drift pattern as (1).

3. Test suite covers only `Wus = ()` cases. The arity-0 `Buildable` impl reduces trivially without exercising the cons-list `Contains` recursion or the per-WU `WuSatisfied` reduction. The actual proof chain that justifies the round (per-WU `WuSatisfied` reducing to per-store `Contains` against the cons-list `Stores`) was tested only in `mock/research/sketches/...`, which is not part of the workspace test surface.

## Changes proposed

### `mock/crates/hilavitkutin/src/scheduler/mod.rs`

Three bound tightenings:

```rust
// Was: pub fn add<W: 'static>(self) -> ...
   pub fn add<W: WorkUnit>(self) -> ...

// Was: pub fn memory<M: 'static>(self, _provider: M) -> Self
   pub fn memory<M: MemoryProviderApi + 'static>(self, _provider: M) -> Self

// Was: pub fn threads<P: 'static>(self, _pool: P) -> Self
   pub fn threads<P: ThreadPoolApi + 'static>(self, _pool: P) -> Self

// Was: pub fn clock<C: 'static>(self, _clock: C) -> Self
   pub fn clock<C: ClockApi + 'static>(self, _clock: C) -> Self
```

Plus the corresponding `use` additions:

```rust
use hilavitkutin_api::platform::{ClockApi, MemoryProviderApi, ThreadPoolApi};
use hilavitkutin_api::work_unit::WorkUnit;
```

### `mock/crates/hilavitkutin/tests/scheduler_builder.rs`

Add WU-bearing positive tests using a `Stub + TestCtx` shim that implements the seven `HasX` accessor traits (pattern lifted from `mock/crates/hilavitkutin-api/tests/work_unit.rs`). Three new tests:

1. `wu_with_raw_resource_builds`: `.resource(Interner).add::<ReadInterner>().build()`. Read = `(Resource<Interner>,)`; proves Stores contains the Resource.
2. `wu_with_kit_builds`: `.add_kit(InternerKit).add::<ReadInterner>().build()`. Same proof, Kit-supplied resource.
3. `two_wus_with_two_kits_build`: `.add_kit(InternerKit).add_kit(WorkspaceKit).add::<ReadInterner>().add::<DiscoverFiles>().build()`. Two WUs, two Kits; non-trivial cons-list depth in both `Wus` and `Stores`.

Update file doc-comment header to reflect the new two-layer test structure (smoke tests with `Wus = ()` plus WU-bearing tests with non-trivial Wus).

Negative case (`.add::<ReadInterner>().build()` without registration) verified manually 2026-05-01 against the post-amendment source. Comment block in the test file captures the exact expected error text:

```
trait bound `(): Contains<Resource<Interner>>` was not satisfied
which is required by `(ReadInterner, ()): Buildable<()>`
```

A trybuild fixture for this case is tracked in #296.
