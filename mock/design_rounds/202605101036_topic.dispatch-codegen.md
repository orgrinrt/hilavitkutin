## Purpose

Turn Topic 3's `ExecutionPlan + CoreProgram` projection into actual executable Rust code: per-core monomorphised functions that the thread pool runs, per-fiber dispatch shapes per the bench-validated approaches, progress-counter sync points, and the inlining discipline that lets LLVM auto-vectorise the inner loop.

The topic is *highly* constrained by polka-dots Domain 17 (lines 1523-1672 of the canonical consolidation), which is bench-validated and load-bearing. Most "what" is decided. The axes below ask: **what concrete Rust surface in `hilavitkutin/src/dispatch/`** carries each of those decisions.

## Canonical reference (binding)

`~/Dev/polka-dots/mock/design_rounds/202603181200_topic.hilavitkutin-design-consolidation.md` lines 1523-1672 (Domain 17). Bench citations: T6 L812-875, L1024-1384, L1689-1976, L3167-3297, L3888-3903.

Decisions Domain 17 already makes (this topic confirms but does not relitigate):

- **Devirtualisation rules:** local `&[fn]` slices with known values WIN. Struct-field fn-ptr arrays FAIL at 12.6x. Global static slots FAIL.
- **Five dispatch approaches** (A-E) bench-ranked: A 1.0x | B 0.9x | C 1.17x | D 1.02x | E 0.97x. Struct-field FAIL.
- **Row-size threshold:** <10K records → C or D | ≥10K → E. Bench-validated (T6 L1507-1517).
- **rust-pipe flattener** for fiber inner body: stack-cached resources, register-to-register pipeline, all stores grouped at loop end (store-buffer friendly). 0.95-0.96x baseline (beats hand-ASM).
- **Inlining discipline:** `#[inline(never)]` on fiber dispatch + per-core program | `#[inline(always)]` on WU bodies within their fiber.
- **Progress counters:** stack `AtomicUsize` per fiber; producer `store(Release)` (single `stlr`); consumer `load(Acquire)` (single `ldar`); raw pointer (`as usize`) passed across threads, not `Arc`.
- **Lock-free by construction:** library guarantee (T3 #103-104), not an Executor option. Zero CAS, zero RMW in dispatch path.
- **ASM verification checklist** (5 checks per Domain 17 + S3 = 6 total): zero `blr`; indexed addressing; no `[sp,...]` in inner loop; immediate morsel size; no `bl` to morsel-size or fiber-dispatch helpers; **dmb st / sfence between final stnp and progress-counter Release store** (Topic 3 S3).

## The load-bearing tension

The plan is **runtime** data (which WUs go in which fiber, head/tail split, morsel sizes — all depend on registered WUs + `RecordCount` + `AdaptMetrics`). But the bench-validated devirtualisation (Domain 17 L1532-1546) requires the WU sequence per fiber to be a **compile-time-known LOCAL slice** for LLVM to inline through.

The set of registered WUs is type-level (Kit pattern from Topic 2 — cons-list `type Units: WorkUnitBundle`). The *partitioning* of that type-level set into fibers/trunks/cores is runtime. Codegen must reconcile this.

Possible resolutions (Axis D below interrogates):

1. **Type-level partitioning** — lift plan results (which WU in which fiber) to type-level. Combinatorial; type-system stress.
2. **Per-fiber-shape generic monomorphisation** — emit a generic `run_fiber<S: FiberShape>(...)` parameterised by a const-bit-set of which WUs fire; monomorphise per unique shape. Bounded explosion if shapes repeat across consumers.
3. **All-WUs-monolithic with const-folded mask** — single monomorphic fiber-runner iterating ALL registered WUs, each guarded by a const-mask check. Some devirt loss; masks may const-fold.
4. **Plan-time-emitted code via proc-macro** — defer codegen until after plan stage; emit Rust source. Fails the no-build-step framework promise.
5. **Adapt at runtime via approach E (schedule-mega) only** — monolithic per-core fn iterating all WUs at all phases, with phase/fiber as runtime indices. Sacrifices C/D for >10K but avoids the type-level explosion entirely.

User input load-bearing here.

## Axes

### Axis A — Codegen entry-point surface in `hilavitkutin/src/dispatch/`

What functions/types comprise the codegen layer?

**Polka-dots-implied shape** (per the C2 task title "codegen_fiber, codegen_core, build_plan"):

```rust
// hilavitkutin/src/dispatch/codegen.rs

pub fn build_plan<Cfg: RunCfg>(
    plan: &ExecutionPlan<...>,
    units: &Cfg::Units,
) -> CompiledDispatch<Cfg>;

pub(crate) fn codegen_core<Cfg: RunCfg, const C: usize>(
    program: &CoreProgram<...>,
    units: &Cfg::Units,
) -> CoreDispatchFn<Cfg, C>;

pub(crate) fn codegen_fiber<Cfg: RunCfg, F: FiberShape>(
    fiber: &FiberPlan<...>,
    units: &Cfg::Units,
) -> FiberDispatchFn<F>;
```

Where `CompiledDispatch<Cfg>` is a value carrying per-core function pointers (note: this is the kind of struct-field pattern Domain 17 says fails at 12.6x — must NOT be how the inner loop reaches the WU functions; only the OUTER orchestration uses these handles, and per-core fns themselves emit LOCAL slices internally).

**Alternatives:**

- **A1: Three free functions** as above (one per layer of the lattice). Mechanical decomposition; matches polka-dots task naming.
- **A2: Single `Dispatch::build`** returning a typestate-threaded compiled value; internal helpers private. Encapsulates the lattice; one public entrypoint.
- **A3: Trait-method emission** via `trait DispatchCodegen<Cfg> { fn build(plan, units) -> CoreDispatchFn<Cfg>; }` impl-driven by an associated type. Heavier type machinery; integrates better with Topic 1 RunCfg slot.

Recommendation (will confirm): **A1 free functions with one public entrypoint `build_plan` and the rest crate-private**. Free-function composition matches Topic 3 Axis B (plan stage already chose free functions over typestate-threaded builder); preserves consistency. The internal `codegen_core` / `codegen_fiber` helpers are pure functions on plan data, no shared mutable state.

### Axis B — `DispatchApproach` selection: where does the A/B/C/D/E pick land?

Topic 3 Axis B added `dispatch_approach: DispatchApproach` to `ExecutionPlan` (one per fiber, derived in plan stage). Question: does each variant cause emission of *literally different* Rust code?

**Options:**

- **B1: Yes, five distinct emission paths.** `codegen_fiber` matches on `DispatchApproach::{A,B,C,D,E}` and emits five fundamentally different fn shapes. High code volume; matches bench-validated discipline; each variant verifiable independently.
- **B2: Three paths (A, D, E).** Drop B (0.9x but micro) and C (1.17x — penalty unacceptable per Domain 17 L1654: "T6 Approach C (initially preferred) → superseded by D/E based on bench findings"). Keep A as the baseline reference, D for `<10K` records, E for `≥10K`. Fewer emission paths; Domain 17 actually says C is superseded.
- **B3: Two paths (D, E).** Drop A as a development reference only; ship D + E as the production approaches. Most aggressive.

Recommendation (will confirm): **B2 (A, D, E)**. Domain 17 explicitly says C is superseded; B is within margin of A and adds emission complexity; A is kept as the baseline reference path so benches and tests have a known-stable comparison. The selector logic (already in Topic 3 plan stage) emits `DispatchApproach::{A, D, E}` only.

If B2 picks: open `BACKLOG.md.tmpl` entry to add C/B back if a future bench surfaces a workload where they win.

### Axis C — Inlining attribute injection mechanism

Domain 17 prescribes:

- `#[inline(never)]` on fiber dispatch fn
- `#[inline(always)]` on WU bodies within their fiber
- `#[inline(never)]` on per-core program

Question: do consumers (WU authors) write these attributes themselves, or does codegen emit them?

**Options:**

- **C1: Consumer-applied** on WU `execute` method: `WorkUnit::execute` trait method specifies `#[inline(always)]` requirement in docs; consumers comply by discipline. Codegen emits `#[inline(never)]` on its own emitted fns (fiber dispatch, per-core program). Simple; no proc-macro needed; relies on consumer discipline. A `lint:warn` could nudge.
- **C2: Codegen-applied via wrapping.** Codegen wraps each WU `execute` call in a `#[inline(always)] fn invoke_<wu_name>(...)` wrapper that LLVM sees as the inline target, regardless of the original `execute` attribute. Robust against consumer discipline drift; adds one layer of fn nesting (LLVM elides).
- **C3: Proc-macro derive on WU.** `#[derive(WorkUnit)]` (or similar) emits the impl with `#[inline(always)]` baked in. Heavier; tight coupling; hard to override per-fiber-position.

Recommendation (will confirm): **C2 codegen-applied via wrapping**. The inlining discipline is load-bearing for the bench-validated 0.95-0.96x baseline; relying on consumer discipline (C1) introduces a regression mode invisible until ASM-audit time. The wrapper layer is one extra fn that LLVM elides at no cost. C3 is too heavy and couples WU authoring to proc-macro state.

### Axis D — Per-core program assembly: bridging type-level WU set ↔ runtime fiber partition

This is the load-bearing axis (per "The load-bearing tension" above). Pick one of the five resolution shapes:

- **D1: Type-level partitioning.** Full type-level enumeration of valid (WU, fiber-position) tuples; partitioning encoded as a type. Probably infeasible without trait-solver explosion.
- **D2: Per-fiber-shape generic monomorphisation.** `fn run_fiber<S: FiberShape>(...)` parameterised by a const-bit-set; LLVM monomorphises per unique shape. Bounded explosion; dovetails with `min_specialization` and Topic 3's sealed dispatch traits (S2). **Likely the bench-validated path** given Domain 17 L1551 says "per-fiber-type trait, LOCAL `&[WuFn]`" for Approach A.
- **D3: All-WUs-monolithic with const-folded mask.** One monomorphic fiber-runner iterating all registered WUs guarded by const-mask. Some devirt loss.
- **D4: Plan-time-emitted source via proc-macro.** Rejected — fails the no-build-step framework promise.
- **D5: Approach-E-only (schedule-mega).** Sacrifice C/D for ≥10K-only workloads; avoids type-level partition entirely. Conflicts with Axis B keeping D for <10K.

Recommendation (will confirm): **D2 per-fiber-shape generic monomorphisation**. This is the shape Domain 17 L1551 references (Approach A: "per-fiber-type trait, LOCAL `&[WuFn]`"). The per-core program is a generic fn parameterised by the *type-level cons-list of fiber shapes the core hosts*, which is determined by the plan stage and lifted to type-level via a sealed marker trait family. Each unique shape monomorphises once; consumers with N fibers in K shape-classes get K monomorphisations, not 2^N.

This requires the plan stage (Topic 3) to *also* lift its fiber-partitioning decisions to type level for codegen consumption. **Concretely:** Topic 3's `FiberPlan` becomes typed by a sealed `FiberShape: Sealed` marker that encodes the WU set + their order. The CoreProgram projection's per-fiber slot is `&FiberPlan<S: FiberShape>`. This adds type-level work to Topic 3 — needs a Topic 3 amendment lock-pending-sketch entry, OR Topic 4 declares this as Topic 4-scope (the plan stage produces typed shape tags as a *codegen-only* projection).

### Axis E — Progress counter / atomic placement

Domain 17: stack `AtomicUsize` per fiber; producer `store(Release)`; consumer `load(Acquire)`; raw pointer across threads.

**Options:**

- **E1: Per-core-program-local stack array.** `let progress: [AtomicUsize; N_FIBERS] = [...; N_FIBERS]` declared in the per-core fn at entry; raw pointer passed to consumer cores via the thread spawn site. Stack lifetime tied to the run() invocation. Requires `RunCtx` (or equivalent) to carry the raw-pointer base.
- **E2: Plan-stage scratch arena.** Plan stage pre-allocates `AtomicUsize` slots in its scratch buffer (Topic 3 P2.5 pre-allocation); codegen emits the loads/stores against fixed offsets. Decouples lifetime from per-core-fn.
- **E3: Resource-style** — `Resource<ProgressCounters<N>>` registered automatically by `build_plan`; accessed via `Context`. Goes through the providers system — but progress counters are per-run scratch, not consumer-owned state.

Recommendation (will confirm): **E2 plan-stage scratch arena**. Topic 3 already commits to plan-stage scratch buffer pre-allocation (P2 expansion). Adding a `progress: &'static mut [AtomicUsize]` view (raw pointer underneath) keeps lifetimes clean and decouples the spawn point from the per-core-fn entry. E1 works but ties progress-counter lifetime to the per-core-fn stack frame, which complicates handoff to the thread pool. E3 misframes per-run scratch as consumer state.

### Axis G — Lock-free guarantee codification

Domain 17 + T3 #103-104: "Lock-free dispatch by construction. This is a library guarantee, not an Executor implementation detail." How is the guarantee codified?

**Options:**

- **G1: Documentation only.** WORKFLOW.md.tmpl + crate doc comment state the invariant. Discipline upholds it.
- **G2: Sealed Executor trait + diagnostic message.** The Executor trait family (Topic 6) is sealed; the only impl is the lock-free one shipped by hilavitkutin. `#[diagnostic::on_unimplemented]` directs consumers seeking custom executors to a deprecated path.
- **G3: Type-level marker.** `LockFreeDispatch` marker trait blanket-impl on the codegen output type. Consumers cannot construct one without going through `build_plan`.

Recommendation (will confirm): **G1 + G3**. Document the invariant (G1) and add a sealed marker (G3) so codegen output carries the type-level evidence. G2 is overkill — the Executor trait is already sealed by virtue of Topic 6's `min_specialization` + `Sealed` supertrait.

### Axis H — Approach-selection threshold (10K records)

Domain 17 picks <10K → C/D | ≥10K → E. Where does the threshold live?

**Options:**

- **H1: Hardcoded const.** `const APPROACH_E_RECORD_THRESHOLD: USize = USize::new(10_000);` in plan stage. Bench-validated; matches Domain 17.
- **H2: Adapt-tunable.** AdaptMode (Topic 5) can tune the threshold per runtime EMA of fiber timings. Adds a knob; defers decision.
- **H3: Const-generic on RunCfg.** `RunCfg::APPROACH_E_THRESHOLD: USize` with default `USize::new(10_000)`. Consumer-overridable; baseline matches bench.

Recommendation (will confirm): **H3 const-generic with default**. Bench-validated default protects new consumers; allows override for edge cases. H2 adds a knob for the adapt subsystem to tune that current bench data does not justify; if it becomes load-bearing, it gets added in Topic 5 then.
