**Scope:** new crate `hilavitkutin-extensions` (contract-bound host orchestration layer) and its sibling `hilavitkutin-extensions-macros` (proc-macro companion), both built on the now-renamed `hilavitkutin-linking`

# Context

The companion topic file in this round (`202604240615_topic.rename-extensions-to-linking.md`) renames the previously-shipped dlopen primitive and reclaims the name `hilavitkutin-extensions` for the contract host layer. This topic file designs that contract host layer.

Every decision here pivots on the arbitrary-time-linking invariant: extensions load, run, and drop at arbitrary points, independent of siblings. No ecosystem-wide "all extensions loaded before any invoked" barrier exists at this layer. The host tolerates and enforces that posture; downstream consumers (viola, future loimu, future clause extension surfaces) build directory scanning, load-order computation, and manifest-driven selection on top.

The prior art informing this design is captured in `hilavitkutin/EXTENSIONS_RESEARCH.md`. In short:

- `loimu` established the explicit-symbol discovery pattern with `#[repr(C)]` descriptors and a clean modules-vs-plugins split.
- `saalis` proved that generic Rust interfaces can cross a cdylib boundary without `dyn Trait` overhead via macro-driven static monomorphisation.
- `polka-dots` benchmarks confirmed that the cdylib + dlsym boundary is a hard LLVM optimisation barrier; relying on Rust `dylib` or shared LTO defeats the whole purpose.
- All three sibling projects mistakenly built domain concepts into their loader layer, which prevented reuse. The reusable split is extensions (binary mechanics, owned by `hilavitkutin-linking`) vs contracts (host orchestration, owned by `hilavitkutin-extensions`). The domain specialisation sits strictly above both, in consumer contract crates like `viola-plugin-abi`.

The sibling proc-macro crate `hilavitkutin-extensions-macros` exists because every real consumer needs a macro for descriptor emission, and shipping that default in the workspace avoids every consumer reimplementing the same trampoline-generation code. The macro is opinionated on the default path (infer name from struct ident, infer version from `Cargo.toml`, empty init/shutdown unless implemented) and extensible through attribute parameters and trait implementations.

# Problem statement

Without a dedicated host layer:

1. Every downstream consumer defines its own `#[repr(C)]` descriptor layout, its own symbol-name convention, its own init / shutdown function pointer types. Three consumers produce three incompatible formats.

2. The consumer's macro has to emit both the descriptor plumbing (identical across consumers) and the domain-specific trampolines (different per consumer). Without a shared base, the descriptor plumbing is duplicated and drifts.

3. Required-vs-optional classification and failure policy have to be reinvented per consumer, with the obvious risk of subtle divergence (one consumer calls missing plugins fatal, another logs a warning and continues).

4. Cross-consumer composability is impossible: a host that wants to load a viola linter cdylib and a clause diagnostic cdylib simultaneously cannot do so if the two contract crates emit incompatible descriptors.

The host layer designed here owns the shared substrate: descriptor shape, discovery protocol, lifecycle contract, capability identification scheme, failure policy. Consumers add domain interpretations of capabilities on top without reinventing any of the substrate.

# Scope

`hilavitkutin-extensions` owns:

- Plugin-host-layer descriptor `#[repr(C)]` shape (`ExtensionDescriptor`), exported-symbol convention (`__hilavitkutin_extension_descriptor`), ABI version gate (`HOST_ABI_VERSION: u32`).
- Pull-based discovery: load the cdylib through `hilavitkutin-linking::Library::load`, resolve the single well-known descriptor symbol via `Library::resolve_static`, validate the descriptor, return an `Extension` handle to the caller.
- Lifecycle orchestration: init called exactly once per load (with a per-extension host-opaque `*mut c_void`), shutdown called exactly once during drop or `Extension::close` (same host-opaque pointer threaded back).
- Capability declaration: stable `CapabilityId` (FNV-64 of ASCII name, const-evaluable at both the extension and host compile-time), descriptor-owned `&'static [CapabilityEntry]` with `(CapabilityId, *const c_void)` pairs. The host stores and looks up by id; the consumer casts the opaque vtable pointer into its domain-specific type.
- Required-vs-optional classification and failure policy hooks (`ExtensionRequirement::{Required, Optional}`, `FailurePolicyFn` as a plain `fn` pointer, `PolicyVerdict::{Continue, Abort}`).
- Structured error model (`ExtensionError`) layered over `hilavitkutin-linking::LinkError`.
- `Extension` handle with explicit `close` and idempotent drop-time shutdown.

`hilavitkutin-extensions-macros` owns:

- `#[export_extension]` attribute macro placed on the extension-side struct. Emits the `#[repr(C)]` descriptor constant, the `#[no_mangle] pub extern "C" fn __hilavitkutin_extension_descriptor` entry point, the capability table, and per-capability `extern "C" fn` trampolines.
- Attribute parameters: `#[export_extension(name = "...", version = "1.2.3", required_host_caps = [CAP_X, CAP_Y])]`.
- Trait hooks consumers implement to feed the macro: `ExtensionMeta`, `InitHandler`, `ShutdownHandler`, `CapabilityExport`.
- Helper macros re-exportable by downstream consumer contract crates so they can ship their own higher-level attribute (e.g. `#[viola::plugin]`) that expands to `#[export_extension]` internally with consumer-specific defaults.

Neither crate owns:

- Domain contracts. NAM, AST shapes, diagnostic formats, runner/grammar/lint vocabularies all live in consumer contract crates.
- Runtime extension discovery (directory walking, manifest parsing, load-order resolution). That is a higher-layer concern; downstream consumers build it on top.
- Cross-extension communication channels. Extensions are independent.
- Sandboxing, resource limits, fuel metering.
- Hot-reload or ref-counted unload. V1 is load-once, drop-once. BACKLOG.
- Remote extension fetching, signature verification, extension stores. Out of scope entirely.

# Non-goals

- Global static registry. Explicitly rejected. No `inventory`, `ctor`, `distributed_slice`, `.init_array`, or any side-effect-table pattern. Discovery is strictly per-extension and pull-based.
- Ecosystem-wide init barrier. Explicitly rejected. No API reads "all extensions must load before any can be invoked". Every extension's lifecycle is independent.
- Runtime generics via `dyn Trait`. Explicitly rejected. Capabilities cross the FFI boundary as `extern "C" fn` pointers baked by the export macro; the consumer casts them back into their Rust interface on the host side via macro-generated trampolines. Saalis's monomorphisation pattern applied.
- Heap allocation. Explicitly rejected. Descriptors are `&'static`, capability tables are `&'static [CapabilityEntry]`, error types are fixed-size variants.
- `std` dependency in `hilavitkutin-extensions`. `#![no_std]`, no `alloc`, matches the rest of the hilavitkutin stack. (The proc-macro crate is necessarily `std`-backed because proc-macros run in the compiler's host context.)

The engine crate (`hilavitkutin`) does not depend on either: static composition is its rule, and runtime extension loading is out-of-scope for the engine itself.

# Key decisions to lock

## D1. `#![no_std]` + no_alloc + no_dyn posture

`hilavitkutin-extensions` is `#![no_std]` with no `alloc` and no `dyn`. Descriptors are `&'static`, capability tables are `&'static [CapabilityEntry]`, errors are fixed-size enums with fixed-width payloads.

`hilavitkutin-extensions-macros` is a proc-macro crate (inherently `std`-backed at macro-execution time). Its emitted output must compile under no_std.

Rationale: the whole hilavitkutin stack commits to the no_std posture. A host crate that leaked `alloc` or `dyn` into consumer code would break the guarantee and force downstream consumers to accept heap dependencies they rejected at their own layer.

## D2. Pull-based discovery via `__hilavitkutin_extension_descriptor`

Extensions expose exactly one well-known C-ABI symbol:

```rust
#[no_mangle]
pub extern "C" fn __hilavitkutin_extension_descriptor() -> *const ExtensionDescriptor;
```

The host calls this symbol immediately after `Library::load` succeeds, reads the descriptor (which is extension-static, `&'static ExtensionDescriptor`), and validates it before any lifecycle function runs. An extension that does not export this symbol is rejected with `ExtensionError::DescriptorMissing`.

Rationale: the only discovery mechanism that works reliably across the OS / linker / rustc version matrix is explicit symbol resolution. Linker-magic approaches fail silently under runtime dlopen (the dynamic loader does not reliably execute `.init_array` entries merged with the host's global registry). Pull-based discovery also directly preserves the arbitrary-time-linking invariant: each extension is discovered independently when it loads, not batch-registered through a shared side-effect.

## D3. `ExtensionDescriptor` `#[repr(C)]` shape

```rust
#[repr(C)]
pub struct ExtensionDescriptor {
    pub abi_version: u32,
    pub extension_name_ptr: *const u8,
    pub extension_name_len: USize,
    pub extension_version: ExtensionVersion,
    pub required_host_caps_ptr: *const CapabilityId,
    pub required_host_caps_len: USize,
    pub capabilities_ptr: *const CapabilityEntry,
    pub capabilities_len: USize,
    pub init_fn: Option<extern "C" fn(host_ctx: *mut c_void) -> ExtensionAbiStatus>,
    pub shutdown_fn: Option<extern "C" fn(host_ctx: *mut c_void) -> ExtensionAbiStatus>,
}

#[repr(C)]
pub struct ExtensionVersion { pub major: u16, pub minor: u16, pub patch: u16, pub _pad: u16 }

#[repr(C)]
pub struct CapabilityEntry {
    pub id: CapabilityId,
    pub vtable_ptr: *const c_void,
}

#[repr(transparent)]
pub struct CapabilityId(pub u64);

#[repr(u32)]
pub enum ExtensionAbiStatus { Ok = 0, InitFailed = 1, InvalidArg = 2, NotSupported = 3, Internal = 4 }
```

All pointers are `*const` to extension-static data; the extension guarantees the data lives as long as the library is loaded. The host never dereferences `vtable_ptr` directly; it passes it back to the consumer-provided trampoline (generated by `#[export_extension]`) which casts it into the domain-specific vtable type.

Rationale: `#[repr(C)]` is the only ABI guarantee available in Rust. Pointer + length fields instead of slice types avoid Rust's unstable slice layout. `Option<extern "C" fn(...)>` is representation-equivalent to a nullable pointer. `USize` and `u16` / `u32` widths are canonical across unix / windows / 32 / 64-bit targets.

## D4. Lifecycle contract and host-opaque context pointer

The `init_fn` and `shutdown_fn` pointers both receive a single `*mut c_void` representing a "host context" for this extension instance. The host allocates this (or passes null), threads it through init, and passes the same pointer back at shutdown. The extension may stash a pointer inside it; the host sees an opaque `*mut c_void` either way.

Key property: the host context is per-extension, not ecosystem-wide. Extension A cannot reach extension B through this channel. This preserves the arbitrary-time-linking invariant at the API level, not just as documented rule.

Invoke is not part of the fixed descriptor. Invoke is always domain-specific: the host asks "does this extension declare capability X?" by looking at `capabilities[]`, retrieves the `vtable_ptr` for that id, and passes it to the consumer-provided trampoline. `hilavitkutin-extensions` never calls domain code directly; it only resolves the pointer.

Rationale: init / shutdown are universal (every extension needs them). Invoke is never universal: each consumer has its own set of capabilities with different signatures. Baking invoke into the descriptor would either force an opaque bytes-in / bytes-out shape (losing monomorphisation) or force this crate to know about specific capability types (breaking domain-agnosticism). Externalising invoke via capability table + consumer trampoline is the right split.

## D5. CapabilityId: FNV-64 of stable ASCII name, compile-time const

```rust
impl CapabilityId {
    pub const fn from_name(name: &'static [u8]) -> Self { Self(fnv1a_64_const(name)) }
}
```

Consumer contract crates define their capability names as const byte strings and derive the id at compile time. Collisions are vanishingly rare at 64 bits; if they ever surface, consumers rename. No runtime hashing, no TypeId dependency, no UUID generation.

Rationale: `CapabilityId` needs to be stable across independently compiled extension and host binaries. TypeId is banned (and would not be stable across crate versions anyway). UUIDs require ceremony per capability and do not compile-time check. FNV-64 of a stable ASCII name is the minimal mechanism that satisfies both constraints.

## D6. Failure policy hooks (required vs optional, pluggable policy)

Each extension load at the host side carries a classification: `ExtensionRequirement::Required` or `Optional`. On load failure:

- Required + failure: the host returns the failure immediately; the loaded library handle is dropped.
- Optional + failure: the host calls a consumer-supplied `FailurePolicyFn` with the structured `ExtensionError`. The policy returns `PolicyVerdict::Continue` (skip and move on) or `PolicyVerdict::Abort` (propagate upward).

Default policy (if the consumer provides none): `Required => Abort, Optional => Continue`.

Rationale: fixed two-valued classification + pluggable policy hook is the smallest API that covers "hard fail on missing required extension", "warn-and-continue on missing optional extension", and "consumer-specific override". Pluggability is fn-pointer-only (no dyn, no alloc); consumers wanting stateful policies wrap state in their own instance and only expose a plain `fn` here.

## D7. Macro placement and customisation (shape of `hilavitkutin-extensions-macros`)

The proc-macro crate ships one primary attribute: `#[export_extension]`. Placed on a user-defined extension struct that implements the required capability traits, it emits:

1. The `#[repr(C)]` descriptor constant, all fields populated from attribute parameters + trait impls.
2. The `__hilavitkutin_extension_descriptor` extern "C" fn returning `&DESCRIPTOR`.
3. Per-capability trampolines: for each trait-method the user implements, an `extern "C" fn` wrapper that calls into the generic Rust method with proper argument marshalling.
4. The `CapabilityEntry` table populated with `CapabilityId` + trampoline pointers.

Customisation:

- Attribute parameters: `#[export_extension(name = "my-extension", version = "1.2.3", required_host_caps = [CAP_NAM, CAP_DIAG])]`.
- Marker traits the user implements (`ExtensionMeta`, `InitHandler`, `ShutdownHandler`) that feed into the macro expansion. The user may omit any of them and accept defaults (empty init, empty shutdown, name defaults to the struct ident in kebab-case, version reads from the consumer crate's `CARGO_PKG_VERSION`).
- Consumer contract crates can wrap `#[export_extension]` under their own attribute name. A consumer macro `#[viola::plugin]` expands internally to `#[export_extension]` with viola-specific trait bounds or attribute parameters pre-filled. This makes the common case `#[viola::plugin] struct MyLinter;` with no other boilerplate.

Rationale: per user decision in this TOPIC round, this is the hybrid between "layered" and "opinionated sibling proc-macro". Ship a full-featured default with sensible behaviour; allow consumer extension through params + trait impls; allow consumer wrapping via domain macros that expand to ours.

## D8. Arbitrary-time-linking as an API-level invariant

The `Extension` handle is what the host works with. It holds:

- A `Library` (from `hilavitkutin-linking`).
- A `&'static ExtensionDescriptor` resolved during load.
- A `*mut c_void` host context.

`Extension::drop` runs the extension's `shutdown_fn` (if present) with the host context, then drops the `Library` (which dlcloses the library). Exactly once, in that order. No global registry the drop removes the instance from, because no global registry exists.

The API provides no `load_all_extensions(manifest)` function. Every load returns a single `Extension`; the consumer chooses whether to keep it in a `Vec`, a `Seq<Extension, N>`, a `HashMap`, or drop it immediately after one invoke. Different consumers legitimately want different orchestration shapes.

Rationale: the invariant must be enforced by the type system, not documentation. A global registry would grow a `for_each` method, which would be called from a context where one extension's shutdown invalidates another extension's running invocation. Absence of the registry closes that whole class of bugs.

## D9. Error model

```rust
#[repr(C)]
#[non_exhaustive]
pub enum ExtensionError {
    LinkFailed { cause: LinkError },
    DescriptorMissing,
    DescriptorInvalid,
    AbiVersionMismatch { expected: u32, got: u32 },
    ExtensionVersionUnsupported,
    RequiredHostCapabilityMissing { cap: CapabilityId },
    InitFailed { status: ExtensionAbiStatus },
    ShutdownFailed { status: ExtensionAbiStatus },
}
```

All linking-layer failures wrap `LinkError` in the `LinkFailed` variant. Host-layer-specific failures have their own variants with fixed-size structured payloads. No heap strings; all diagnostics go through the enum discriminant and any fixed-width payload.

## D10. Compatibility policy

- **ABI version** (`ExtensionDescriptor::abi_version`): host's `HOST_ABI_VERSION: u32 = 1` for v1. Extensions declare their compiled ABI version. Any mismatch rejects the extension with `AbiVersionMismatch`. This version bumps when the descriptor layout or symbol name changes (breaking change to the host ABI).
- **Extension version** (`ExtensionDescriptor::extension_version`): semver triple. The host does not interpret this; consumers use it for their own compat gates. V1 ships unused by the host; available for future.
- **Required host capabilities**: an extension may declare capability IDs the host must offer. The host passes its own registered capability set into `ExtensionHost::load`; if any required cap is missing, the extension is rejected with `RequiredHostCapabilityMissing`. Allows future host-side capability addition without breaking older extensions that do not request it.

Rationale: one hard version gate (abi_version) + one soft capability gate (required_host_caps) + consumer-owned semver gives space to evolve without painting us into a corner. V1 keeps the policy narrow; later versions can add richer matching if deployment reveals the need.

# Surface sketch. `hilavitkutin-extensions` public items

```rust
// Core types
pub struct ExtensionDescriptor { /* #[repr(C)] fields as in D3 */ }
pub struct ExtensionVersion { major: u16, minor: u16, patch: u16 }
pub struct CapabilityEntry { id: CapabilityId, vtable_ptr: *const c_void }
#[repr(transparent)] pub struct CapabilityId(u64);
#[repr(u32)] pub enum ExtensionAbiStatus { Ok, InitFailed, InvalidArg, NotSupported, Internal }
pub const HOST_ABI_VERSION: u32 = 1;
pub const DESCRIPTOR_SYMBOL: &[u8] = b"__hilavitkutin_extension_descriptor\0";

// Capability identity
impl CapabilityId {
    pub const fn from_name(name: &'static [u8]) -> Self;
}

// Extension handle (what the host works with)
pub struct Extension { /* holds Library + &'static ExtensionDescriptor + host_ctx */ }
impl Extension {
    pub fn descriptor(&self) -> &'static ExtensionDescriptor;
    pub fn capability(&self, id: CapabilityId) -> Maybe<*const c_void>;
    pub fn capabilities(&self) -> &'static [CapabilityEntry];
    pub fn name(&self) -> &'static [u8];
    pub fn version(&self) -> ExtensionVersion;
    pub fn close(self) -> Outcome<(), ExtensionError>;
}
impl Drop for Extension { /* calls shutdown_fn then drops library */ }

// Host-side load entry point
pub struct ExtensionHost {
    host_abi_version: u32,
    host_capabilities: &'static [CapabilityId],
    failure_policy: FailurePolicyFn,
}
impl ExtensionHost {
    pub const fn new(host_capabilities: &'static [CapabilityId]) -> Self;
    pub const fn with_policy(self, policy: FailurePolicyFn) -> Self;
    pub fn load(&self, path: &[u8], req: ExtensionRequirement, host_ctx: *mut c_void) -> Outcome<Maybe<Extension>, ExtensionError>;
}

// Policy
pub enum ExtensionRequirement { Required, Optional }
pub enum PolicyVerdict { Continue, Abort }
pub type FailurePolicyFn = fn(&ExtensionError, ExtensionRequirement) -> PolicyVerdict;
pub fn default_policy(err: &ExtensionError, req: ExtensionRequirement) -> PolicyVerdict;

// Errors
#[repr(C)] #[non_exhaustive] pub enum ExtensionError { /* as in D9 */ }

// Trait hooks consumed by the macro (re-exported for macro output)
pub trait ExtensionMeta {
    const NAME: &'static [u8];
    const VERSION: ExtensionVersion;
    const REQUIRED_HOST_CAPS: &'static [CapabilityId] = &[];
}
pub trait InitHandler {
    extern "C" fn init(host_ctx: *mut c_void) -> ExtensionAbiStatus;
}
pub trait ShutdownHandler {
    extern "C" fn shutdown(host_ctx: *mut c_void) -> ExtensionAbiStatus;
}
pub trait CapabilityExport {
    const ID: CapabilityId;
    const VTABLE_PTR: *const c_void;
}
```

`ExtensionHost::load` returns `Outcome<Maybe<Extension>, ExtensionError>`: outer `Outcome` is "did policy allow continuation"; inner `Maybe` is "did an extension actually instantiate". `Outcome::Ok(Maybe::Nothing)` is "optional extension failed, policy said continue, no instance", a normal control-flow path for consumers that load best-effort.

# Surface sketch. `hilavitkutin-extensions-macros` public items

```rust
#[proc_macro_attribute]
pub fn export_extension(attr: TokenStream, item: TokenStream) -> TokenStream;
```

Helper macros for defining stable `CapabilityId` names and marker traits are re-exported from `hilavitkutin-extensions` for use in macro output.

Example usage shape:

```rust
// In viola-plugin-abi:
pub const CAP_LINTER: CapabilityId = CapabilityId::from_name(b"viola.linter.v1");

pub trait Linter { /* domain interface */ }

// Consumer macro wrapping ours:
#[proc_macro_attribute]
pub fn viola_plugin(attr: TokenStream, item: TokenStream) -> TokenStream {
    // Emits:
    //   impl CapabilityExport for MyLinter { const ID = CAP_LINTER; const VTABLE_PTR = ...; }
    //   #[export_extension(name = "...", required_host_caps = [CAP_NAM])]
    //   struct MyLinter;
}
```

The consumer user writes:

```rust
#[viola::plugin]
struct MyLinter;

impl Linter for MyLinter {
    fn lint(&self, input: &Ast) -> Diagnostics { /* ... */ }
}
```

and gets the full `#[repr(C)]` descriptor, trampolines, and exported symbol emission without direct contact with the lower-layer machinery.

# Test plan

Integration tests under `hilavitkutin-extensions/tests/` load a fixture extension built as a cdylib in `hilavitkutin-extensions/tests/fixtures/`. The fixture is the minimum-viable extension: one capability, empty init, empty shutdown, implementing `ExtensionMeta` + `CapabilityExport` manually (not through the macro, to keep the fixture independent of proc-macro coverage; the macro has its own trybuild tests).

Required coverage for v1:

- Happy path: load fixture, descriptor resolves, descriptor validates, init returns Ok, capability lookup returns expected `*const c_void`, shutdown returns Ok, drop is idempotent with close.
- Descriptor missing: load a cdylib not exporting the symbol; expect `DescriptorMissing`.
- Descriptor ABI version mismatch: fixture with `abi_version != HOST_ABI_VERSION`; expect `AbiVersionMismatch`.
- Init failure: fixture whose init returns `InitFailed`; expect propagation.
- Shutdown failure: fixture whose shutdown returns `ShutdownFailed`; expect surfacing through explicit `close()` but swallowed through drop (drop cannot fail).
- Required host capability missing: fixture declaring a required cap the host does not offer; expect `RequiredHostCapabilityMissing`.
- Optional extension load failure + default policy: expect `Outcome::Ok(Maybe::Nothing)`.
- Required extension load failure + default policy: expect `Outcome::Err(...)`.
- Custom failure policy: supply a policy that aborts on optional; verify verdict honoured.
- Arbitrary-time-linking stress: load A, invoke, load B, invoke both, drop A, invoke B still works, drop B.

Macro tests under `hilavitkutin-extensions-macros/tests/` use `trybuild` fixtures:

- `#[export_extension]` on a bare struct with defaults: compiles, emits descriptor.
- `#[export_extension(name = "...", version = "...")]` with parameters: parameters respected.
- `#[export_extension]` on a struct implementing `InitHandler` + `ShutdownHandler`: lifecycle fn pointers emitted.
- `#[export_extension]` on a struct implementing multiple `CapabilityExport`s: capabilities table populated.
- `#[export_extension]` with invalid parameters: compile error with a usable message.

Not required for v1 (deferred to BACKLOG):

- Multi-platform CI verification (unix + windows). Deferred to task #203 (`cargo mock ci`).
- Concurrent load from multiple threads (ExtensionHost must be Sync; thread-testing requires pool infrastructure inherited later from the engine).
- Reload of the same extension path (covered by v1 through drop + load, but not named test).
- Hot-reload with ref-counted unload.
- Wasm backend.
