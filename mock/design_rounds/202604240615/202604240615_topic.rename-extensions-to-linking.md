**Phase:** TOPIC
**Scope:** rename the shipped `hilavitkutin-extensions` crate (landed 2026-04-24 in PR #34) to `hilavitkutin-linking` and reclaim the `extensions` name for the contract-bound host layer introduced by this same round

# Context

The prior round (202604240745) landed the cross-platform dynamic library loader as `hilavitkutin-extensions`. That crate owns dlopen / LoadLibrary, typed symbol resolution, static-data references, and a structured platform error model. It is strictly mechanical: it knows how to map a path to a process-address-space library handle and how to resolve an exported symbol. It knows nothing about descriptors, contracts, capabilities, or lifecycle.

The current round was originally scoped as a companion crate named `hilavitkutin-plugins`: the contract-bound host that layered descriptor discovery, lifecycle orchestration, and capability dispatch on top of the loader. While drafting the topic file, two naming problems surfaced:

1. **"Plugin" carries the wrong register.** In industry use (WordPress plugins, Figma plugins, VSCode plugins, game mod systems), the word implies a runtime-loaded script unit with some amount of sandboxing, permission model, or interpreter between it and the host. The contract layer being designed here is none of those things: it is strictly compile-time-monomorphised C-ABI cdylibs with full native trust, the same trust level as any other crate in the consumer's dependency tree. The only thing dynamic about them is the linkage point. Calling that a "plugin" primes every reader to expect sandboxing that does not exist and misses the static-typing-across-cdylib-boundary property that is the actual design.

2. **"Extensions" is collision-prone at the current assignment.** Hilavitkutin CLAUDE.md already uses "Extensions" as the category header for standalone opt-in sub-libraries (`hilavitkutin-ctx`, `hilavitkutin-persistence`, `hilavitkutin-str`). Naming the dlopen primitive `hilavitkutin-extensions` forces two different meanings of "extension" into the same codebase: the category header (native Rust sub-libraries bundled with the stack) and the crate name (the dynamic-linker primitive). Worse, the more natural meaning of "extensions" in external user vocabulary (a distributed unit of functionality that extends the host, like IDE extensions) matches neither of the two current uses; it matches precisely what this round was originally calling `hilavitkutin-plugins`.

The clean resolution is to shift both names one step:

- `hilavitkutin-extensions` the shipped crate becomes `hilavitkutin-linking`. Linking is what dlopen and LoadLibrary do: they resolve a binary artifact at runtime and link its symbols into the process. The crate is about the mechanical act of linking; "linking" labels that mechanically and uncontroversially.

- `hilavitkutin-extensions` the name is reclaimed for the contract host layer (the crate that was briefly called `hilavitkutin-plugins` while this topic was being drafted). Users think of the thing they install or distribute as an "extension", and at the contract host layer this matches: a contributor publishes an extension cdylib, the host loads it through the extension host, the extension declares capabilities, the host orchestrates its lifecycle.

- The "Extensions" category header in hilavitkutin CLAUDE.md (covering ctx / persistence / str) is renamed to "Ecosystem". Those crates are siblings that travel with the engine but are not part of it; "ecosystem" says that plainly without inventing a cute term.

This round effects all three renames together, plus introduces the new `hilavitkutin-extensions` crate pair (the host + its proc-macro sibling). The companion topic file `202604240615_topic.hilavitkutin-extensions-host.md` designs the contract host layer; this topic file is dedicated to the rename and the category-word change.

# Problem statement

The names as they stand create three compounding confusions:

1. A reader arriving at `hilavitkutin-extensions`'s DESIGN.md sees a dlopen primitive. A reader arriving at hilavitkutin CLAUDE.md sees `hilavitkutin-ctx / -persistence / -str` called "extensions". A reader arriving at the (soon to exist) contract host layer sees it called `hilavitkutin-plugins`. Three distinct meanings of one word across one codebase.

2. The word `plugin` at the contract layer mis-sets expectations about trust boundaries, lifecycle policy, and runtime dynamism. Any consumer reading the stack top-down spends the first read correcting the mental model.

3. Downstream consumer repos (viola-plugin-abi and others-to-come) carry "plugin" in their names too. If the host layer is named "extensions" and the consumer contract crate is named `*-plugin-abi`, the whole stack reads more naturally as an extensibility story than as a WordPress-style plugin story. The rename aligns the vocabulary downstream consumers inherit.

Without the rename, every subsequent round has to either adopt the mis-set names and propagate the confusion, or apologise for them in running prose. Front-loading the rename now, while the shipped crate has exactly one published version and no downstream dependents outside the workspace, is an order of magnitude cheaper than any later fix.

# Non-goals

- Renaming `IncompatibilityError`. It covers semantic incompatibility detected during load-time compatibility checks and is orthogonal to whether the crate is called extensions or linking.
- Renaming `Symbol` or `StaticRef`. Both describe symbol-resolution concepts (function pointer and static-data-pointer typed handles) that do not depend on the crate name.
- Renaming the sealed trait concept. `LibrarySymbol` keeps the sealed-trait pattern; only the trait name changes.
- Changing public method names on the renamed types beyond their owning-type renames (e.g., `Library::load`, `Library::resolve`, `Library::resolve_static`, `Library::close`; these follow the type rename automatically).
- Reorganising the module layout. The split across `error.rs`, `symbol.rs`, `backend/` is independent of the naming story and stays.
- Any change to the compatibility-check stub. Ships as-is through the rename.
- Moving anything in the backend layer (unix / windows continue unchanged except for the occasional internal type name).

# Key decisions to lock

## D1. The dlopen crate is named `hilavitkutin-linking`

Alternatives considered:

- `hilavitkutin-loader`: viable but suggests higher-level orchestration (asset loader, config loader) that the crate deliberately does not own.
- `hilavitkutin-dlopen`: too platform-flavoured; the crate abstracts over dlopen and LoadLibrary.
- `hilavitkutin-libload`: awkward; `libload` reads like a misspelling of `libloading`.
- `hilavitkutin-dylink`: viable but reads as a portmanteau; less immediately obvious.
- `hilavitkutin-linking`: direct, matches the action the crate performs (runtime linking of a shared library into the process), has no negative association.

Decision: `hilavitkutin-linking`.

## D2. The contract host crate is named `hilavitkutin-extensions`

The name is reclaimed from its prior assignment on the dlopen primitive. This reclaim is safe because the only published version of `hilavitkutin-extensions` is the one from PR #34 (2026-04-24), and it has no downstream dependents outside this workspace. Inside the workspace, every dependent is updated by this same round.

## D3. Primary handle of the linking crate is named `Library`

Alternatives considered:

- `Link`: confusable with hyperlinks and HTTP Link headers.
- `Linkage`: more abstract than useful; reads as a build-time concept.
- `Module`: collides with Rust's mod keyword and with loimu's prior-art usage.
- `Dylib`: platform-specific wording.
- `Library`: matches the dlopen mental model ("open a shared library"), matches the libloading crate's vocabulary, reads naturally in type signatures (`Library::load(path)`).

Decision: `Library`.

## D4. Error is named `LinkError`

Alternatives considered:

- `LibraryError`: viable but reads as "error about a library" rather than "error linking".
- `LinkError`: the failure is in the act of linking (path resolution, symbol lookup, platform mismatch), not in the library artifact itself.

Decision: `LinkError`. The discriminants themselves do not change: `PathNotFound`, `LoadFailed { platform_code: USize }`, `SymbolMissing`, `PlatformMismatch`, `IncompatibleVersion`. All remain `#[non_exhaustive]`.

## D5. Sealed trait is named `LibrarySymbol`

Alternatives considered:

- `Resolvable`: too generic; anything can be "resolvable" in some context.
- `CSymbol`: misleading; the trait bounds Rust fn-pointer types, not C symbol types.
- `LibrarySymbol`: couples to the crate's primary concept while staying accurate ("a symbol resolvable from a Library").

Decision: `LibrarySymbol`. The sealed-trait + macro-generated-impls-at-arity-0-through-8 pattern is unchanged.

## D6. CLAUDE.md category header becomes "Ecosystem"

The existing header "Extensions (standalone opt-in sub-libraries)" covers `hilavitkutin-ctx`, `hilavitkutin-persistence`, and `hilavitkutin-str`. With the `extensions` name being reclaimed for the contract host layer, a new label is required.

Alternatives considered:

- `Companions`: evocative but precious.
- `Modules`: collides with Rust mod.
- `Subsystems`: too clinical.
- `Add-ons`: informal, and itself a near-synonym of extensions.
- `Ecosystem`: descriptive, factual, no collision. The crates are siblings in a shared ecosystem around the engine.

Decision: `Ecosystem (standalone opt-in sub-libraries)`.

## D7. Rename and new-crate addition happen in one round

This round is deliberately larger than the "one concern per round" norm. The rename alone produces a rename-only round, and the new crate pair alone produces an introduction round, but doing them separately requires the intermediate state to pass lints and tests (i.e. rename the crate, merge, then reference the new name in the new crate): two PRs, twice the mockspace-state-machine ceremony, and no architectural benefit to the split.

Running both concerns in one round means:

- One PR cradle-to-grave against `dev`.
- Two topic files under this round id (one for the rename, one for the new crate pair), making the scope clear at the topic-file granularity.
- One doc CL covering both rename edits and new-crate doc templates.
- One src CL covering the directory move, the type renames, the new crate source, and the new crate tests.

Rationale: the rename exists because the new crate needs its old name; the new crate needs the rename completed before it can claim the name. The interleaving is tight; splitting the work introduces a strictly temporary intermediate state for no gain.
