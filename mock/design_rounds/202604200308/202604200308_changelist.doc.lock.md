## Purpose

Last engine sub-round. Ships the public type surface for the
threading primitives (domain 20) and the runtime-adaptation
parameter carriers (domain 22). Skeleton only: every coordination
function stubs to `todo!()`; real OS thread spawning gated behind
the future `threading-std` feature; real PMC / perf_event sampling
deferred. Downstream rounds fill in the thread-pool spawn machinery,
work-stealing fallback, heterogeneous-core detection, adaptive-mode
heuristics, and phase-time measurement.

Threading is how the engine maps `ExecutionPlan`'s core assignments
onto actual threads: pre-allocated pool, hybrid wake strategy,
P/E-core awareness, head+tail convergence for commutative fibers.
Adapt is how per-phase tuning parameters (fuse aggressiveness,
morsel size multiplier, split thresholds) respond to runtime
metrics between frames.

Landing the skeleton now completes the engine crate's public
surface: after 5a1-5a3 + 5a6 + this round, every domain the DESIGN
lists has a module with types and at least stub function
signatures. Downstream rounds can fill in bodies in parallel
because the cross-module type surface is frozen.

## Deliverables

### Threading types (domain 20)

All under new `src/thread/` submodule tree.

- `ThreadPool` — pre-allocated pool stub. Fields: `thread_count: u16`,
  `spin_budget: u32`, `wake_strategy: WakeStrategy`. Construction
  via `ThreadPool::new(core_count, wake)`; the actual thread
  spawning is feature-gated on a future `threading-std` feature
  (not introduced this round; documented in BACKLOG).
- `WakeStrategy` enum — three variants:
  - `HybridSpinPark { spin_iters: u32 }` — spin `spin_iters` rounds
    then park. Default strategy (128 iters per DESIGN).
  - `PureSpin` — busy-wait forever, no parking.
  - `PurePark` — no spinning, park immediately.
- `CoreClass` enum { `P`, `E` } — heterogeneous-core class.
  P-cores get critical-path trunks + larger morsels; E-cores get
  branches/leaves + proportionally smaller ranges.
- `CoreAssignment<const MAX_CORES: usize>` — per-core role record.
  Fields: `trunk_index: [u16; MAX_CORES]`, `fiber_assignments: [FiberId; MAX_CORES]`,
  `morsel_size_multiplier: [u16; MAX_CORES]`, `assigned_count: u16`.
- `ThreadHandle` — newtype over `u16` (thread index in pool).
  `#[repr(transparent)]`, Copy, Eq, Hash.
- `Convergence` — head+tail convergence record: two threads
  processing the same commutative fiber from opposite ends. Fields:
  `head_thread: ThreadHandle`, `tail_thread: ThreadHandle`,
  `meeting_record: ProgressCounter`.

### Runtime adaptation types (domain 22)

New `src/adapt/mod.rs`.

- `AdaptConfig` — per-phase adaptive tuning parameters. Fields:
  `phase_id: PhaseId`, `max_fuse_threshold: u32`,
  `morsel_size_multiplier: u16`, `split_threshold: u32`.
- `AdaptMode` — mode selector for per-phase adaptation. This is
  the same concept as `strategy::PhaseStrategy` (MaxFuse /
  Balanced / MaxSplit) shipped in 5a6. To avoid duplication,
  `adapt::AdaptMode` is a **type alias** for
  `crate::strategy::PhaseStrategy`. Strategy remains the
  canonical home for the enum; adapt re-exports it as
  `AdaptMode` for caller ergonomics (the term "adapt mode"
  reads naturally inside the adapt module's docs, while
  "phase strategy" reads naturally from the strategy
  selector's perspective).
- `AdaptMetrics` — runtime counters feeding adapt decisions.
  Fields: `cache_miss_rate: u32`, `branch_miss_rate: u32`,
  `phase_completion_time_ns: u64`. Default-zero.

### Deferrals (all → BACKLOG)

Threading:
- Actual pthread / Win32 / OS-thread spawning (gated on a future
  `threading-std` feature).
- Hybrid spin-park wake implementation (128 spin iters default →
  park; uses `core::hint::spin_loop` during spin phase).
- Heterogeneous-core detection (CPUID leaf 0x1A on x86, sysfs on
  Linux, IOKit / hwloc on macOS).
- Core pinning per platform (`pthread_setaffinity_np` on Linux,
  `thread_policy_set` on macOS, `SetThreadAffinityMask` on
  Windows).
- Work-stealing `Executor` interface (consumer-provided trait
  override for variable per-record workloads).
- Head+tail convergence merge implementations (addition for
  additive/commutative; sequential fallback for non-commutative).
- Phase pipeline parallelism via AtomicUsize progress counters
  (`ProgressCounter` already exists from 5a3; wiring into the
  dispatch loop is a follow-up).

Adapt:
- PMC / `perf_event` integration for `cache_miss_rate` /
  `branch_miss_rate` sampling.
- Phase completion time measurement (monotonic clock via
  `hilavitkutin-api::Clock` trait).
- Adaptive mode selection heuristic (latency vs throughput
  trade-off; feeds `select_adapt_config` body).
- Per-phase config hot-swap during pipeline execution (between
  frames only, per DESIGN — during execution would invalidate
  monomorphised dispatch).
- Telemetry hooks for exporting adapt decisions to the consumer
  (e.g. for observability / tuning dashboards).

## Impl-time decisions

1. **`AdaptMode` is a type alias for `PhaseStrategy`.** The two
   concepts are the same enum with different names in different
   DESIGN sections. Alias keeps a single source of truth while
   letting callers read whichever name fits context.
2. **`ThreadPool` ships no spawn method this round.** The
   `threading-std` feature flag is called out in BACKLOG but not
   introduced to `Cargo.toml` yet — introducing an unused feature
   would add noise. The future round that implements actual
   spawning will add the feature and the spawn method together.
3. **`CoreAssignment` uses parallel arrays, not
   `[Option<PerCore>; MAX_CORES]`.** Matches the existing
   skeleton style in `plan::FiberGrouping` + `plan::PhaseBoundaries`
   (parallel arrays + count). Keeps the struct Copy-friendly for
   const-construction.
4. **`classify_cores` returns `[CoreClass; 256]`.** 256 is the
   documented upper bound for engine-supported core counts in
   DESIGN. A const generic `<const MAX_CORES: usize>` would be
   nicer but the function is a stub — generic signature churn in
   follow-up rounds is cheap.
5. **`steal_fallback` takes `&T` and `FiberId` with no bound.**
   Real signature will constrain `T: Executor` once the
   `Executor` trait ships (BACKLOG). Skeleton parameter shape is
   enough to plumb callers through.
6. **`Convergence::meeting_record` owns a `ProgressCounter`.**
   Consumers pair two threads with one `Convergence` record; the
   meeting-record counter is published by whichever thread
   crosses the midpoint first (implementation: BACKLOG).
