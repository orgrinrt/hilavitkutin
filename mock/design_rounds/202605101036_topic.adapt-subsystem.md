## Purpose

Wire the runtime adaptation subsystem the canonical design depends on. Static plan-time metrics exist already (Topic 3 Axis B carries cache pressure, data flow, column lifetime). Topic 5 covers the *runtime* tier: per-morsel timing, change-frequency classification, EMA tracking, reorganisation triggers between frames, predictive parking input, and the SchedulerMetrics resource consumers read.

The topic is again concretely shaped by polka-dots (Domain 22, lines 2006 to 2068, plus T5 Q5 EMA discipline at lines 601 to 637 of T5). Most decisions are mechanical placement questions; the load-bearing axes are (1) where AdaptMetrics lives in the data plane, and (2) which AdaptModes ship.

## Inputs from prior topics (binding)

- **Topic 1 Axis 5.** Adapt is a meta-WU (`AdaptWu`) observing `ScheduleEnd`. Substrate ships default `AdaptWu` in `hilavitkutin-providers`; consumer registers via `.with(AdaptWu::default())` for substrate behaviour, ships own `AdaptWu` for custom, omits if not wanted. The integration shape is locked. This topic specifies what the WU body does and what metrics structure it reads/writes.
- **Topic 3 Axis B.** ExecutionPlan carries static plan-time analysis results. Topic 5 only covers runtime additions.
- **Topic 3 Axis B + M2.** Per-core program tracks `estimated_icache_bytes`. AdaptMode tuning of approach selection might consume that, but Axis H locked at H3 with const-generic threshold so adapt does not retune approach in v1.
- **Topic 4 Axis F.** AdaptWu must not break the ASM 6-check invariants: writes to AdaptMetrics happen at fiber-completion or phase boundaries, never inside the inner record loop.
- **No-heap principle.** AdaptMetrics is stack or scheduler-arena allocated; never `Vec` or `Box`.

## Vocabulary (canonical, post-Axis-F lock)

The earlier draft of this topic used three overlapping names; post-Axis-F-lock the canonical set is:

- **AdaptArena**: the raw scratch-arena memory region (hot per-fiber lines + per-core park slots + cold SoA arena). Plan-stage allocated. Codegen + AdaptWu read/write directly via raw pointer. Not consumer-readable.
- **`metrics::*` Resources** (per-axis sub-resources from Axis F): `PassDurationMetrics`, `PhaseEmaMetrics`, `FiberEmaMetrics`, `ChangeClassMetrics`, `CacheResidencyMetrics`, `ThroughputMetrics`, `PredictiveParkingMetrics`, `MemoryWatermarkMetrics`, `CoreIdleTimeMetrics`. Consumer-readable. Refreshed from AdaptArena by AdaptWu at `ScheduleEnd`.
- **`MetricsKit`**: optional Kit recipe bundling the standard combination of `metrics::*` Resources for one-call registration.

Earlier draft referenced a unified `Resource<SchedulerMetrics>`. **That shape is retired** by Axis F (F2 chose per-axis sub-resources). Any prose below referencing `SchedulerMetrics` as a singular Resource is stale; the canonical replacement is the per-axis `metrics::*` set. Earlier `Resource<AdaptMetrics<...>>` references (B1 alternative) are also retired by Axis B's B3 lock — `AdaptMetrics` as a Resource type does not ship; `AdaptArena` is the raw scratch (not a Resource).

**MAX_LANES vs MAX_FIBERS reminder:** these are different (`MAX_LANES = MAX_CORES`, one lane per physical core; `MAX_FIBERS` is the global fiber cap). Per-core arena slots size by `MAX_LANES`; per-fiber arena slots size by `MAX_FIBERS`. Topic 3 line 112 has the canonical comment.

## Canonical reference (binding)

## Axes

### Axis A. AdaptConfig multi-axis shape (REFRAMED 2026-05-10)

Originally proposed as four opaque modes (Off / Fast / Standard / Aggressive). User reframed during topic conversation: each metric is its own axis with independent tuning, in line with the workspace's multi-axis design pattern (arvo Strategy markers, WorkUnit Hint).

`AdaptConfig` is a struct of nine specialised sub-configs, one per metric. Sub-configs do **not** share a uniform shape; each carries only the knobs that make sense for that axis. They share a sealed `AdaptAxis: Sealed` trait for the universal operations (`is_enabled() -> arvo::Bool`, `sample_skip() -> arvo::USize`).

Each sub-config impls `sealed::Sealed` and `AdaptAxis`. Specialised methods stay on the per-axis struct.

#### Cadence and interleave

`sample_skip` lets long-lived schedule loops amortise heavy metrics across passes. `skip = 0` means every pass; `skip = N` means every (N+1)th pass. Default skip values pick up the interleaving pattern: pass-duration always on (cheapest); per-fiber EMA every 4th pass; per-phase EMA every 8th; cache-residency every 16th; throughput-trending every 32nd. Tunings are per-consumer; AdaptWu's check `pass_count % (skip + 1) == 0` runs at `ScheduleEnd` to decide whether each axis updates this pass.

#### Convenience presets

```rust
impl AdaptConfig {
    pub const fn off() -> Self { /* every axis enabled = false */ }
    pub const fn fast() -> Self { /* pass_duration only */ }
    pub const fn standard() -> Self { /* default; pass + phase + fiber + change + parking + idle on */ }
    pub const fn aggressive() -> Self { /* standard + cache + throughput + watermark on */ }
}
```

Presets are documented combinations, not opaque enum variants. Consumer can derive a preset and override one axis: `AdaptConfig { phase_ema: PhaseEmaConfig { sample_skip: arvo::USize::new(7), ..Default::default() }, ..AdaptConfig::standard() }`.

#### Core-idle-time axis (new in reframe)

The ninth axis tracks per-core wait time per pass, the cheapest signal of load-distribution quality. High parking/spinning per core indicates uneven distribution or computations that did not behave as the plan expected. Mechanism:

1. Topic 6's parking primitive records `wait_start_ns = clock.now()` at park entry, `wait_end_ns = clock.now()` at exit, accumulating delta into a `[AtomicUsize; MAX_LANES]` slot in the AdaptArena.
2. AdaptWu at `ScheduleEnd` reads the per-core accumulator, divides by pass duration to get per-core idle ratio (Norm-typed).
3. Compare ratios across cores: `max(idle_ratio) - min(idle_ratio) > imbalance_threshold` emits `Virtual<BalanceAnomaly>`.
4. Topic 9 (plan caching) observes the Virtual and triggers a re-plan with focus on the imbalanced cores.

The breadcrumb increments are atomic but on the cold path (parking is already a sync point), so the overhead is negligible. The signal value is high: if core 0 idles 80% while core 7 idles 5%, the plan's core-pinning is wrong.

Recommendation: **A locked at this multi-axis shape with nine specialised sub-configs sharing the AdaptAxis trait, plus four convenience presets via const fn.**

### Axis B. AdaptMetrics data plane

Where does the metrics structure live?

- **B1: `Resource<AdaptMetrics<...>>` registered automatically by `build_plan` when AdaptMode is not Off.** Consumer-readable via `Has<AdaptMetricsProvider>` Ctx bound. Auto-registration matches the Topic 1 / Topic 2 Provider pattern.
- **B2: Plan-stage scratch arena slot alongside progress counters.** Lifetime tied to `Scheduler::run()` frame; AdaptWu reads/writes via fixed offsets. Cheaper allocation but consumer cannot read it (would need a separate Resource snapshot).
- **B3: Hybrid.** Scratch-arena hot path plus a Resource-snapshot view that AdaptWu refreshes at `ScheduleEnd`. Best of both: fast inner-loop access plus consumer-visible state.

Recommendation: **B3 hybrid**. Topic 4 Axis E already commits to plan-stage scratch arena for progress counters; AdaptMetrics joins that arena. AdaptWu (running at `ScheduleEnd`) snapshots the arena view into a `Resource<SchedulerMetrics>` for consumer reads. The snapshot is tiny (per-phase plus per-fiber counters, on the order of MAX_PHASES + MAX_FIBERS u64s) and only happens once per pass, not per morsel.

### Axis C. EMA decay constant placement

Domain 22 fixes EMA decay at `(ema * 7 + measured) / 8`. Where does that constant live in source?

- **C1: Hardcoded constant.** `const EMA_DECAY_NUMERATOR: arvo::USize = arvo::USize::new(7);` plus matching denominator. Bench-validated default; matches Domain 22.
- **C2: Const-generic on AdaptMode variants.** Each mode carries its own decay weight. Aggressive mode might want faster decay (1/4) for more reactive tuning; Standard keeps 1/8.
- **C3: Const-generic on RunCfg with default.** Consumer-overridable like Axis H from Topic 4.

Recommendation: **C1 hardcoded with BACKLOG entry for C2 if benches surface a mode-dependent need**. Adding the knob now is premature per the bench-driven rule. The 1/8 decay is bench-validated; new evidence is what justifies a knob.

### Axis D. Per-phase / per-fiber metrics shape (HILA-RUNTIME-B5)

What lives in `AdaptMetrics`?

- Per-phase: `ema_duration_ns: arvo::UFixed<...>`, `pass_count: arvo::USize`, `last_pass_duration_ns: arvo::UFixed<...>`.
- Per-fiber: `ema_morsel_duration_ns: arvo::UFixed<...>`, `change_class: ChangeClass` (Hot / Warm / Cold from Domain 12), `stolen_count: arvo::USize` (Topic 6 work-stealing input), `cache_residency_estimate: arvo::Bool`.
- Scheduler-level: `frame_count`, `idle_ns_total`, `thermal_throttle_detected: arvo::Bool`.

Layout question: SoA (separate `[T; MAX_PHASES]` arrays per metric) or AoS (`[PhaseMetrics; MAX_PHASES]`)?

Recommendation: **D1 SoA layout**. Domain 22 mentions vectorised batch updates (NEON 4xu32 / AVX2 8xu32 across all unit EMAs in one pass). SoA enables that vectorisation; AoS forces gather-scatter. Plan stage allocates the arrays in the scratch arena per Axis B3.

### Axis E. Reorganisation trigger mechanism

Domain 22: triggers fire between frames as cheap bitmask comparisons. The arvo-bitmask dirty-propagation pattern from Topic 3 step 12 is the load-bearing primitive.

- **E1. AdaptWu computes a `Mask<N>` of "fibers needing re-plan" by comparing each fiber's EMA against threshold ranges baked into the plan.** If any bit set, AdaptWu emits a `ReplanRequested` Virtual that Topic 9 (plan caching + dirty-detection) observes. If no bits set, no work; the plan stays cached.
- **E2. Continuous re-plan attempt every N frames.** Wasteful; bench data does not justify it.
- **E3. Consumer-driven explicit re-plan call.** Forfeits adapt's purpose; consumers cannot know which fibers drifted.

Recommendation: **E1 mask-driven trigger via Virtual<ReplanRequested>**. Matches the dirty-propagation pattern already shipped in Topic 3 step 12 (which uses arvo-bitmask masks for incremental skip). The threshold ranges are static plan-time computed and stored in the plan; AdaptWu compares EMAs against them in O(MAX_FIBERS) time per pass, all bitmask ops.

### Axis F. SchedulerMetrics resource shape

Domain 22 + Domain 23 say `SchedulerMetrics` is a consumer-readable resource. What fields?

```rust

    // stable while internal layout evolves).

```

Auto-registered when AdaptMode is not Off. Consumer reads via standard `Has<SchedulerMetricsProvider>` Ctx bound.

Recommendation: **F1 ship the shape above**. Per-phase / per-fiber detail accessor methods (`fn fiber_ema(&self, FiberId) -> arvo::UFixed<...>`) preserve the Topic 1 Provider pattern. The struct fields are stable; internal storage can change without breaking consumers.

### Axis G. Predictive parking integration boundary

Topic 5 owns the policy (the wait-time prediction); Topic 6 owns the mechanism (`spin_loop()` / `thread::park()`). Where does the boundary cut?

- **G1. AdaptMetrics carries `predicted_wait_ns: arvo::UFixed<...>` per phase boundary.** Topic 6's parking primitive reads it and picks the strategy from the three-tier rule (under 100ns / 100ns to 10µs / over 10µs). AdaptWu computes the prediction at `ScheduleEnd` for the next pass.
- **G2. AdaptWu directly invokes parking decisions.** Wrong layer; AdaptWu should not interact with the thread pool's parking mechanism.
- **G3. Topic 6 reads raw EMAs and computes the prediction itself.** Couples Topic 6 to AdaptMetrics layout.

Recommendation: **G1 prediction lives in AdaptMetrics, mechanism lives in Topic 6**. Clean boundary. AdaptWu produces `predicted_wait_ns`; Topic 6's parking primitive reads it and dispatches. Wrong prediction costs ≤1 park/unpark cycle per Domain 22 line 2058, well within budget.

### Axis H. Vectorised EMA batch update

Domain 22 names NEON 4xu32 / AVX2 8xu32 vectorisation explicitly. Two implementation paths:

- **H1. Auto-vectorised via standard Rust loop + LLVM passes.** Trust the compiler; bench-verify.
- **H2. Hand-rolled SIMD intrinsics via `core::arch::*` cfg-gated per architecture.** Bench-validated; matches arvo's always-optimal-internals rule.
- **H3. Both paths, AdaptMode-controlled.** Standard mode uses H1; Aggressive uses H2.

Recommendation: **H1 with bench-driven escalation to H2**. Per the workspace rule (`arvo-always-optimal-internals.md`): write the naive trait-composition first; ship the optimisation when benches show a meaningful gap. The naive `for` loop over `[arvo::UFixed<...>; MAX_FIBERS]` with `#[inline(always)]` will autovectorise on most targets given the right cfg-gates. If the bench harness shows a >10% gap on a hot workload, escalate to H2 in a follow-up round.
