# Topic: hilavitkutin arvo Round D adoption (workspace-wide)

**Date:** 2026-04-29
**Phase:** TOPIC
**Scope:** All five hilavitkutin crates that consume arvo's reshaped surfaces (`hilavitkutin-api`, `hilavitkutin-str`, `hilavitkutin-persistence`, `hilavitkutin-ctx`, `hilavitkutin` engine), their tests, and `mock/Cargo.lock`. Mechanical adapter round to track arvo's Round D substrate change. No public API surface change for hilavitkutin's own consumers; only internal `use` paths, const-generic-position literal forms, and a handful of internal Bits constructor / accessor migrations.

## Current state

arvo's consolidated Round D (workspace-local at `bcc38304`) reshaped a small set of surface details that the hilavitkutin crates consume in their internals. None of the changes affect hilavitkutin's own public API; the visible types stay byte-identical because of `repr(transparent)` lowering. The work is mechanical token replacement plus a Cargo.lock bump.

`mock/Cargo.lock` currently pins arvo at `a98a3f49` (pre-Round-D), so workspace-local builds stay green. Consumers building hilavitkutin against post-Round-D arvo (viola via path-deps, future repo-side updates) hit 38 trait-bound errors against a single-crate slice plus a cascading wave through str/persistence. Bumping the lock is what surfaces the full migration footprint.

A prior round (202604291932) attempted this same adoption with a topic scoped to hilavitkutin-api source plus its tests, on the assumption that other consumers were untouched. That assumption was wrong: the lock bump revealed five additional patterns (C through G in this topic) across str/persistence and (transitively) ctx/engine. The prior round was abandoned cleanly; this round is the corrected reopening with full scope spelled out.

A second, narrower split (proposed tasks #284 through #288, since deleted) tried to land each crate's adapter as its own round on the same branch. That ran into the mockspace pre-commit hook's workspace-wide `cargo check` gate: each round's source commit needs the workspace to compile cleanly at commit time, but the workspace can only compile cleanly when all five crates' adapter patterns have landed together (the lock bump is atomic; arvo cannot be at the new SHA for one crate and the old SHA for another). The "one wide round" framing here is the working answer; pre-commit cargo-check loosening is tracked separately as #289.

## Changes proposed

Seven mechanical patterns, restricted to the files above. None changes a public API surface visible outside hilavitkutin.

**Pattern A** (path migration): `arvo::newtype::{...}` becomes `arvo::{...}`. Imported names stay identical. Pure path-only rewrite. Round D collapsed `arvo::newtype` into `arvo_storage` and re-exports the public surface from `arvo::*`.

**Pattern B** (literal helper-fn migration): `IBits(N)` / `FBits(N)` literal forms in const-generic positions become `ibits(N)` / `fbits(N)` helper-fn calls. `FBits::ZERO` becomes `fbits(0)`. The helpers are const fns producing `MetaCarrier`-backed payloads. Round D switched the inner field from `u8` to `MetaCarrier(u16)` to break a rustc trait-solver cycle in the `Hot: UContainerFor<{ ufixed_bits(I, F) }>` projection (research note in arvo: `mock/research/D-aux/...`, layout-soundness gates for `repr(transparent)` chains and `MetaCarrier` transmute).

**Pattern C** (macro path relocation): `arvo_bits::bitfield!` becomes `arvo::bitfield!`. The bitfield primitive moved from L1 contracts to L0 in Round B/C/D. hilavitkutin-str's `StrLayout` declaration is the only consumer in the workspace.

**Pattern D** (Bits constructor): `Bits::new(...)` was removed; `Bits::from_raw(...)` is the construction surface. Affects every consumer that constructs `Bits<N, S>` values: hilavitkutin-str/handle.rs (the `StrLayout` macro emission widened the cascade) and hilavitkutin-persistence/{archive_str, manifest, sieve, cold_store}.rs.

**Pattern E** (Bits accessor): `Bits::bits()` accessor was renamed to `Bits::to_raw()`. Same call-site footprint as D.

**Pattern F** (no auto-coerce From<u64>): `Bits<N, S>: From<u64>` is not implemented in Round D. Old code used `(value as u64).into()` to construct `Bits<28, Hot>`; new code requires explicit `Bits::<28, Hot>::from_raw((value as u64) as u32)` (or whichever container width matches N per arvo's dispatch table). Affects str/interner.rs and any persistence sites that did similar.

**Pattern G** (ContentHash cascade): `ContentHash = Bits<64, Hot>` inherits all of D, E, F throughout hilavitkutin-persistence. The same constructor / accessor edits apply at every ContentHash site.

After source edits land, `cargo update` bumps the eight arvo crates plus the five new arvo facade sub-crates plus mockspace plus notko (the lock-bump cascade). `cargo check --workspace` from `mock/` is the verification gate.

No public API change in hilavitkutin. The exposed types (`Urgency`, `Divisibility`, `Significance`, `Str`, `ContentHash`, the various `pub type` aliases over `UFixed<I, F, S>`) keep identical bit widths, identical strategy markers, identical raw container types. The change is in how hilavitkutin's source spells its arvo imports, const-generic argument forms, and Bits constructor / accessor names.

No doc-side change. None of `mock/crates/*/DESIGN.md.tmpl`, `mock/PRINCIPLES.md.tmpl`, `mock/WORKFLOW.md.tmpl`, or `mock/agent/` reference `arvo::newtype` as an import path, `IBits(N)` as a const-construction form, `arvo_bits::bitfield!` as a macro path, `Bits::new` as a constructor, or `Bits::bits()` as an accessor. Verified by grep against the dev tree at TOPIC time. The doc CL is empty.

## Out of scope

- Any deeper Round D adoption work (using new arvo primitives like `MetaCarrier` directly, restructuring around the new `arvo_storage` module). Adapter round only.
- Re-exporting arvo types through `hilavitkutin-api`. Architecture rule: do not re-export arvo through the api.
- Cross-workspace propagation. clause / vehje / notko need their own follow-up adapter rounds against Round D when their consumers next build against the workspace local arvo. Not this round's scope.
- Loosening the pre-commit `cargo check` gate. Tracked as #289.
- Any source edit that goes beyond the seven patterns above. If any site requires more (for example, a Bits algorithm that uses a removed-by-Round-D method that has no direct replacement), it gets surfaced at SRC time and either gets carved out of this round's scope or recorded in a SHAME entry.

## Why a round at all

The diff is mechanical token replacement across roughly 30 to 40 files plus a Cargo.lock bump. Two reasons it goes through the design-round flow rather than a direct chore.

First, the `changelist-required` lint blocks any `crates/*.rs` edit at the HARD_ERROR phase gate when no src CL is open. The repo's invariant is "no source change without a CL", and this is a real source change crossing the public lint surface (the type-position changes are visible to the lint pack).

Second, the cross-crate coordination is non-trivial state. Capturing the Round D dependency, the affected files, the patterns, and the verification steps in a topic plus doc CL plus src CL leaves a paper trail for the next adapter round (when arvo lands its next consolidating change) and for any agent revisiting the workspace at a future compaction. The seven patterns enumerated here are the load-bearing piece; future rounds can map their own work back onto this template.
