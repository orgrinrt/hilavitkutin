# Purpose

Stand up the documentation surface for the new `hilavitkutin-extensions` crate per the topic. The crate owns cross-platform dynamic library loading plus pull-based symbol resolution, with no plugin-contract semantics. This changelist commits three per-crate templates (`README`, `DESIGN`, `BACKLOG`), adds the crate to the top-level crate table, and scopes two agent rules (`cargo.md`, `MAIN.md`) so they permit the new plugin-host layer without loosening the engine's static-composition constraint.

Crate source (Cargo.toml, build.rs, src/lib.rs, workspace-member registration) is deliberately held for the src CL: the mockspace workflow's `changelist-required` lint blocks source edits during DOC phase, and attempting to land a stub to satisfy `design-doc-source-mismatch` under DOC is both fighting the state machine and unnecessary — the design-doc lint skips source-only checks when the crate directory contains doc templates but no source. DESIGN.md.tmpl describes types in prose without backticks during DOC; the src CL expands it with backticked names once source exists and the lint can verify them.

## D5 — `mock/agent/rules/cargo.md.tmpl`: scope forbidden crate types

The existing rule says "Six crates, flat — no proc-macro crates, no cdylib, no plugin-dlopen targets" and "Forbidden crate types: No `crate-type = ['cdylib']` — there are no dynamic plugins; composition is static. No runtime plugin crates of any shape." That was written before the plugin-host layer was designed; the unqualified prohibition now conflicts with the new `hilavitkutin-extensions` + `hilavitkutin-plugins` crates, which are the reusable machinery downstream consumers use to load their own cdylibs at runtime.

Changes:

1. "Six crates, flat" becomes "Eight crates, flat" once `hilavitkutin-plugins` lands in its own round. For this round (which introduces only `hilavitkutin-extensions`), the rule becomes "Seven crates, flat" and is updated again when plugins lands.
2. Add `hilavitkutin-extensions` to the bullet list of crate descriptions, placed in the extensions grouping.
3. Rewrite the "Forbidden crate types" section so the cdylib prohibition is explicit about which crates it applies to (the engine plus every other hilavitkutin crate), and explicitly notes that `hilavitkutin-extensions` is a plain `rlib` that provides abstractions for LOADING third-party cdylibs but is not itself a cdylib. No hilavitkutin crate is ever a cdylib target; cdylib ecosystems exist only downstream of this layer, in consumer hosts.
4. Add a "Plugin host crates" subsection to clarify: `hilavitkutin-extensions` is the reusable lower-layer loader; `hilavitkutin-plugins` (scheduled as its own round) will be the reusable upper-layer host orchestration. Both are domain-agnostic, used by downstream consumers such as viola.
5. Add dependency-direction rules for `hilavitkutin-extensions`: no other hilavitkutin-crate deps (except notko / arvo substrate for primitive vocabulary). Uses libc / windows-sys for raw FFI. Must never depend on `hilavitkutin-api` or `hilavitkutin`.
6. Keep the "No `[lib] proc-macro = true`" clause unchanged.

## D6 — `mock/agent/MAIN.md.tmpl`: scope static-composition rule

The existing "Static composition only. All WUs are registered at compile time via the scheduler builder. No `inventory::`, no `#[ctor]`, no `#[distributed_slice]`, no runtime plugin discovery" clause reads as a stack-wide prohibition. The intent is engine-scoped: the engine's WorkUnit composition is static, and the stack does not use linker-magic registration anywhere in its own crates. Downstream consumers loading plugins via the plugin-host layer are doing something distinct: pull-based, explicit-symbol, `cdylib`-boundary loading. That is not the same as the banned linker-magic pattern.

Changes:

1. Update the "four logical groupings" prose to "five logical groupings" and add a "Plugin host infrastructure" grouping naming `hilavitkutin-extensions` (with a forward pointer to `hilavitkutin-plugins`).
2. Refine the "Static composition only" architecture rule to scope it to the engine: state explicitly that the engine's WorkUnit composition is static, and that the stack does not use inventory / ctor / distributed_slice anywhere in its own crates.
3. Add a clause: "The plugin-host layer (`hilavitkutin-extensions`, and later `hilavitkutin-plugins`) provides pull-based explicit-symbol dynamic loading for downstream consumer hosts. This is distinct from runtime WU discovery — the engine never loads WUs at runtime, and the plugin-host crates do not use linker-magic discovery themselves."
4. Extend "What not to do" with a clarifying bullet: runtime dlopen for downstream-consumer plugins routed through `hilavitkutin-extensions` (and later `hilavitkutin-plugins`) is the permitted mechanism and is distinct from the banned inventory / ctor / distributed_slice patterns.
