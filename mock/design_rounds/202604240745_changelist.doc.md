# Purpose

Stand up the documentation surface and workspace plumbing for the new `hilavitkutin-extensions` crate per the topic. The crate owns cross-platform dynamic library loading plus pull-based symbol resolution, with no plugin-contract semantics. This changelist commits to the public shape (type and error names), creates the three per-crate template files (`README`, `DESIGN`, `BACKLOG`), updates the top-level crate table, adds the crate to the mock workspace members, and removes the prior lesser-agent starter directory outside `mock/`.

# Deliverables this round

## D1 — `mock/crates/hilavitkutin-extensions/README.md.tmpl`: new file

Terse 3-10 line crate summary per workspace convention. Content:

```
# hilavitkutin-extensions

Cross-platform dynamic library loading primitive for the hilavitkutin
plugin stack. Pull-based explicit-symbol resolution over `dlopen` /
`LoadLibrary`, `cdylib`-only, `#![no_std]`, no allocator.

Domain-agnostic: this crate knows how to open a shared library and
resolve a named symbol. It does not know what a plugin is. Higher-level
descriptor and lifecycle orchestration lives in `hilavitkutin-plugins`.
```

## D2 — `mock/crates/hilavitkutin-extensions/DESIGN.md.tmpl`: new file

Shipping contract for the crate. Names every public type and trait that
the src CL will provide; `design-doc-source-mismatch` enforces parity.

Structure:

1. Opening prose (3-4 paragraphs):
   - What the crate is and what it solves (the two-layer loader / plugin-host split).
   - Why `cdylib` + explicit symbols (optimisation barrier, ABI stability).
   - Why `no_std` + no allocator (matches rest of the stack; no heap strings in errors).
   - What it deliberately does not know (plugin descriptors, lifecycle, domain contracts).
2. Public types section, naming:
   - `Extension` (opaque loader handle; RAII closes the library on drop).
   - `Symbol<'ext, T>` (typed symbol handle with lifetime bound to the `Extension`).
   - `ExtensionSymbol` (sealed marker trait for types resolvable via `Symbol`).
   - `StaticDescriptor<'ext, T>` (helper wrapping pointer-to-descriptor patterns).
3. Public error types:
   - `ExtensionError` with variants `PathNotFound`, `LoadFailed`, `SymbolMissing`, `PlatformMismatch`, `IncompatibleVersion`. No heap payloads.
   - `IncompatibilityError` with `VersionSkew`, `ArchMismatch`, `AbiMismatch`.
4. API surface:
   - `Extension::load(path: &[u8]) -> Outcome<Self, ExtensionError>`
   - `Extension::resolve<T: ExtensionSymbol>(&self, name: &[u8]) -> Outcome<Symbol<'_, T>, ExtensionError>`
   - `Symbol<'_, T>::get(&self) -> &T`
   - `compatibility_check(bytes: &[u8]) -> Outcome<(), IncompatibilityError>` (top-level helper; narrow contract)
5. Platform coverage note (macOS / Linux / Windows triples).
6. Dependency note: consumes `notko` for `Outcome` / `Maybe`, `arvo` for `USize` / `Cap`, `arvo-bits` for `Bool`. No `hilavitkutin-api` or `hilavitkutin` dependency. On Windows, `windows-sys` for raw FFI bindings (decided in src CL).
7. Non-scope explicit negation: no descriptor parsing, no lifecycle, no runtime registration, no inventory, no `.init_array` scanning, no Rust `dylib` support, no async, no explicit unload.
8. Cross-reference to the topic file and EXTENSIONS_RESEARCH.md.

All backticked type names in DESIGN.md.tmpl MUST correspond to items declared in `src/lib.rs`; the src CL fleshes out bodies but the names and module paths match from DOC phase onward.

## D8 — update `mock/agent/rules/cargo.md.tmpl`: scope forbidden crate types

The existing rule forbids `cdylib` and "runtime plugin crates of any shape" across the entire stack. That rule was written before the plugin-host layer was designed; the unqualified prohibition now conflicts with the new `hilavitkutin-extensions` + `hilavitkutin-plugins` crates, which are the reusable machinery downstream consumers use to load their own cdylibs at runtime.

The scope needs to be tightened rather than dropped. Changes:

1. "Six crates, flat" → "Eight crates, flat" (add hilavitkutin-extensions + hilavitkutin-plugins rows).
2. Rewrite "Forbidden crate types" section to scope the cdylib prohibition to the engine+api+build+ctx+persistence+str crates, explicitly carving out the plugin-host infrastructure as different. Clarify that the plugin-host crates are themselves plain `rlib` — they provide abstractions for LOADING cdylibs, they are not cdylibs themselves. No hilavitkutin crate is a cdylib target.
3. Add a new "Plugin host crates" subsection explaining the split: engine+ctx+persistence+str+api+build stay at static composition; extensions+plugins are the reusable loader layer downstream consumers (viola, loimu, etc.) use for their plugin ecosystems.
4. Keep the "no `[lib] proc-macro = true`" clause unchanged.
5. Add dependency-direction rules for the two new crates:
   - `hilavitkutin-extensions`: no other hilavitkutin crate deps (except notko/arvo substrate for primitive vocabulary). Uses `libc` / `windows-sys` for raw FFI. Must never depend on `hilavitkutin-api` or `hilavitkutin`.
   - `hilavitkutin-plugins`: depends on `hilavitkutin-extensions` + notko/arvo substrate. Must never depend on `hilavitkutin-api` or `hilavitkutin`. Domain-agnostic — no viola-specific types.

## D9 — update `mock/agent/MAIN.md.tmpl`: scope static-composition rule

The existing "Static composition only. All WUs are registered at compile time via the scheduler builder. No `inventory::`, no `#[ctor]`, no `#[distributed_slice]`, no runtime plugin discovery" clause reads as a stack-wide prohibition. The intent is engine-scoped: the engine's WorkUnit composition is static; the stack does not use linker-magic registration anywhere in its own crates.

Downstream consumers that use `hilavitkutin-extensions` + `hilavitkutin-plugins` to load plugins at runtime are doing something distinct: pull-based, explicit-symbol, cdylib-boundary loading. That is not the same as the banned pattern (inventory / distributed_slice / ctor constructors) and is not a loophole to the ban.

Changes:

1. Update the "four logical groupings" prose to "five logical groupings" and add a "Plugin host infrastructure" grouping naming the two new crates (`hilavitkutin-extensions`, `hilavitkutin-plugins`).
2. Refine the "Static composition only" rule to say it applies to the engine's WorkUnit composition. Keep the linker-magic ban stack-wide (no hilavitkutin crate uses inventory, distributed_slice, or ctor).
3. Add a clause: "The plugin-host layer (hilavitkutin-extensions, hilavitkutin-plugins) provides pull-based explicit-symbol dynamic loading for downstream consumer hosts. This is distinct from runtime WU discovery: the engine never loads WUs at runtime, and the plugin-host crates do not use linker-magic discovery themselves."
4. Extend "What not to do" with a clarification that runtime dlopen in downstream consumer plugins (routed through hilavitkutin-extensions / -plugins) is the permitted mechanism, distinct from inventory/ctor banned patterns.
