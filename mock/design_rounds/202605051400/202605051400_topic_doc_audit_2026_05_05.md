# Doc audit consolidation round (hilavitkutin)

This round consolidates the workspace-wide documentation audit findings for hilavitkutin into a single doc-only round. Two audits ran on 2026-05-05: a per-crate sub-agent audit covering `mock/crates/*/{README,DESIGN,BACKLOG}.md.tmpl` across all 11 crates (no DEEPDIVE_*.md.tmpl files exist in this repo), and a self-audit of the entrypoint surfaces. The findings cluster into seven categories:

1. **Tier 1 leakage in templates that render to public surfaces.** 28 hits across 8 templates: `mock/research/sketches/`, `mock/design_rounds/`, "design round" phrasing, `mockspace` mentions, round/CL ids, and `.tmpl` extensions in prose.
2. **Engine-vocabulary `entry`/`entries` doubling** in `hilavitkutin-providers`, `hilavitkutin-str`, and `hilavitkutin-persistence` DESIGN tmpls. The audit flags this as a substantive design question rather than pure prose drift: the engine vocabulary is `record`, but the relevant crates use `entry` for hash-table entries / interner-arena entries / changelist entries — three legitimate non-record meanings. The cleanup needs to disambiguate per occurrence.
3. **Two ASCII tree diagrams** in `hilavitkutin-extensions/DESIGN.md.tmpl` (lines 110-130) and `hilavitkutin-linking/DESIGN.md.tmpl` (lines 52-61). Both violate the writing-style ASCII-diagram ban; both should convert to prose or nested lists.
4. **`chain` drift** in cons-list/accumulator prose across `hilavitkutin-api`, `hilavitkutin-kit`, `hilavitkutin` DESIGN tmpls. `chain` is a banned dead term in engine context (mapped to `fiber` in the canonical hierarchy). Prose like "accumulated chain of WUs" needs to use `accumulator` (already canonical for type-level fold results) or rephrase.
5. **Heading-level (h1/h3) and DESIGN-title naming inconsistencies** across all 11 README + DESIGN templates. Part of the per-crate template-shape drift; the audit flags it as cross-cutting structural cleanup.
6. **Stale source-vs-doc names**, multiple places: `MemoryProvider`/`ThreadPool`/`Clock` named in `hilavitkutin-api/DESIGN.md.tmpl` while source has `*Api`-suffixed variants; `WeightedSieveCache` in `hilavitkutin-persistence/DESIGN.md.tmpl` while source has plain `SieveCache`; `tuple_generic!` named in `hilavitkutin-ctx/DESIGN.md.tmpl` while no such macro is exported.
7. **Tabular-vocab clash in `hilavitkutin-persistence`.** `RowCount` and `entities` are used in places where the engine's column-not-table model needs `record` / `count`. Per-crate audit flags as substantive.

There is also one cleanup tail: **Round 4 deleted-trait history-keeping** in `hilavitkutin-api` and `hilavitkutin-kit` DESIGN templates. Per `no-legacy-shims-pre-1.0.md`, history-keeping for deleted-and-replaced shapes should clean up once the round ages out. Round 4 (#330) just landed; the history is fresh enough that whether to retain it is a judgment call. This round leans toward keep-for-one-more-cycle, drop in the next round.

This round is doc-only. The src CL referenced in section "## Following src CL (mentioned, not written)" is left for a separate round after the doc CL applies and is locked.

## Decisions

### Decision 1: Scope shape

**Single doc CL covering all seven categories**, executed against this topic. Same rationale as arvo's parallel round: the findings overlap per file; reviewing them together at one lock checkpoint is cheaper.

### Decision 2: Tier 1 leakage

Apply the same treatment as arvo's Decision 2 + 3: round-id references and `mock/design_rounds/` paths in DESIGN body prose get rewritten to the design rationale itself, without the round id. `mock/research/sketches/` references get a similar treatment: the prose names what the sketch verified ("the empirical evidence" or "the depth-5 monomorphisation profile") without naming the sketch path. Sketch paths themselves stay in Tier 2 history files.

### Decision 3: `entry` vs `record` disambiguation

Per `vocabulary.md`, `entry` is fine for hash-table / interner-arena / changelist entries. The audit flags occurrences where `entry` is used in engine-domain prose that should read `record`. The doc CL walks each occurrence and decides per-context:

- `hilavitkutin-str`: arena entries are interner-arena entries (legitimate). `record`-shaped occurrences in column-store narrative prose (if any) flip to `record`.
- `hilavitkutin-persistence`: tabular-vocab clash specifically. `RowCount` and `entities` are not engine-domain; they are storage-spine narrative. Audit suggests `record_count` and `records` as the canonical replacement; this CL adopts that.
- `hilavitkutin-providers`: per-occurrence verify what the underlying domain is (interner store entries are legitimate; engine column references should be `record`).

### Decision 4: ASCII tree diagrams

Convert both to prose or nested lists per writing-style. The `hilavitkutin-extensions` diagram (lines 110-130) describes the host-orchestration call sequence; rewrite as numbered prose. The `hilavitkutin-linking` diagram (lines 52-61) describes the layer stack; rewrite as a one-paragraph dependency narrative or a nested list with "depends on" indents.

### Decision 5: `chain` drift

Each `chain` occurrence in cons-list / accumulator prose gets reread for canonical replacement:

- "accumulated chain of <X>" → "accumulator over <X>" or "cons-list of <X>".
- "chain of WUs" in dispatch narrative → "sequence of WorkUnits in a fiber" (the engine word).
- Documentation-internal "chain" used metaphorically (not engine-domain) stays if it would be substituted with worse prose; sample-check rejects renaming for renaming's sake.

### Decision 6: Stale source-vs-doc names

The three named drift items are mechanical fixes:

- `MemoryProvider` / `ThreadPool` / `Clock` → `MemoryProviderApi` / `ThreadPoolApi` / `ClockApi` per source. Update all backticked occurrences in `hilavitkutin-api/DESIGN.md.tmpl`.
- `WeightedSieveCache` → `SieveCache` per source. Update `hilavitkutin-persistence/DESIGN.md.tmpl`.
- `tuple_generic!` → drop (the macro does not exist in source). The doc paragraph that names it needs verification: was the macro renamed? deleted? not-yet-implemented? `hilavitkutin-ctx/DESIGN.md.tmpl` paragraph reread against source determines whether the prose stays at all.

### Decision 7: Heading-level inconsistencies

The audit flags structural drift across 11 README + DESIGN templates without enumerating each file's exact deviation. The doc CL spec includes a one-line "audit and align heading levels per the workspace README format" task per crate; the lock review confirms each file matches the canonical shape.

### Decision 8: Round 4 deleted-trait history

Keep the Round 4 history paragraphs in `hilavitkutin-api` and `hilavitkutin-kit` DESIGN tmpls for one more cycle. Drop in the next doc round when Round 4 has aged out. Rationale: the history is genuinely useful for current contributors trying to reason about the new declarative Kit shape; dropping today loses transitional context. The dropping itself is a one-line task in a future doc round; not in this CL.

### Decision 9: Title flow

`# hilavitkutin, Architecture Design` (post-em-dash-replacement) → `# Hilavitkutin: Architecture Design`. Mechanical edit on `mock/DESIGN.md.tmpl` line 1.

### Decision 10: WORKFLOW.md.tmpl rendering fate

Same as arvo's Decision 8: verify what `cargo mock` produces. If it renders to `docs/WORKFLOW.md` (Tier 1), rewrite to consumer focus. If it renders only to `mock/WORKFLOW.md` (Tier 2), no edit.

### Decision 11: Auto-gen header workspace finding

Same as arvo's Decision 7. Recorded as workspace-wide; not addressed in this round.
