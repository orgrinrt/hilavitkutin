# Deliverables

## S1 — `StaticRef<'ext, T: 'static>` in `src/symbol.rs`

Add a second typed symbol wrapper for static-data resolution, distinct from `Symbol` (which wraps fn pointers). Structure:

```rust
pub struct StaticRef<'ext, T: 'static> {
    ptr: *const c_void,
    _marker: PhantomData<&'ext T>,
}

impl<'ext, T: 'static> StaticRef<'ext, T> {
    pub fn get(&self) -> &T { unsafe { &*(self.ptr as *const T) } }
    pub fn as_raw(&self) -> *const c_void { self.ptr }
    pub(crate) fn from_raw(ptr: *const c_void) -> Self { ... }
}
```

`T: 'static` ensures the referenced data has no non-static lifetimes; the typical pattern is a `#[repr(C)] pub struct PluginDescriptor { ... }` static exported by a plugin.

## S2 — `Extension::resolve_static<T: 'static>` in `src/lib.rs`

```rust
pub fn resolve_static<T: 'static>(&self, name: &[u8]) -> Outcome<StaticRef<'_, T>, ExtensionError> {
    match backend::platform_resolve(self.handle, name) {
        Outcome::Ok(ptr) => Outcome::Ok(StaticRef::from_raw(ptr)),
        Outcome::Err(e) => Outcome::Err(e),
    }
}
```

Mirrors `resolve` but returns the static-data wrapper.

## S3 — `Extension::close(self) -> Outcome<(), ExtensionError>` in `src/lib.rs`

Explicit unload that surfaces the platform close result. `Drop` continues to work (RAII default); `close` consumes `self` so the compiler prevents double-close, and the return value lets the caller handle unload errors instead of silently dropping them. Implementation: suppress the `Drop` impl for the consumed path via `core::mem::ManuallyDrop` or `core::mem::forget`, call `backend::platform_close(handle)` directly, propagate the result.

## S4 — extend `ExtensionSymbol` arities to 0-8

Replace the hand-written impls with a declarative macro that expands the sealed trait + `ExtensionSymbol` impl for each arity from 0 through 8. Short block, no code repetition.

## S5 — tests for new surface (`tests/basic.rs`)

Add:
- `resolve_static` for a known system-library symbol (for example `stdout` / `__stdoutp` on macOS / `stdout` on glibc — the C runtime's stdout FILE pointer is the canonical pointer-to-static-data test target).
- `explicit_close` happy path: load libc, close explicitly, verify Outcome::Ok.
- `extended_arity` compile-test: resolve a 5-arg or higher-arity fn pointer (synthetic; may not be callable against libc but the resolve + type should compile).

## S6 — CI workflow `.github/workflows/ci.yml`

Add GitHub Actions workflow:

```yaml
name: CI
on:
  push:
    branches: [dev, main]
  pull_request:
    branches: [dev, main]

jobs:
  check-test:
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@nightly
      - working-directory: mock
        run: cargo check -p hilavitkutin-extensions
      - working-directory: mock
        run: cargo test -p hilavitkutin-extensions

  windows-cross-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@nightly
        with: { targets: x86_64-pc-windows-gnu }
      - working-directory: mock
        run: cargo check --target x86_64-pc-windows-gnu -p hilavitkutin-extensions
```

Minimal but covers the crate's two target surfaces. Native Windows runs can be added later if native test execution is needed; cross-compile-check catches compile errors without needing a Windows runner.
