# Context

Round 202604240745 shipped the `hilavitkutin-extensions` crate as a working cdylib loader. Round 202604240831 applied the doc backtick cleanup to clear the `undocumented-type` lint. Both rounds passed lints and shipped working tests, but the v1 public surface reflected a scope cut that the downstream roadmap does not actually accept: static-data symbol resolution was deferred, function-pointer arities were capped at 0-4, explicit close was left RAII-only, and there was no CI.

The immediate downstream consumer is `hilavitkutin-plugins` (task #191). That crate needs to resolve a plugin descriptor from a loaded library. Plugin descriptors are exported as `#[no_mangle] pub static PLUGIN_DESCRIPTOR: PluginDescriptor = ...;` — a static-data pattern, not a function pointer. With the previous round's cut, every consumer would drop to `Symbol::as_raw` and hand-cast via unsafe, which defeats the typed-wrapper purpose.

The principle is simple: what ships now is what every downstream crate uses from here on. "Complete" is not a polite description; it is the acceptance criterion. Deferring for a follow-up round means every consumer has to work around the gap until the follow-up lands, which is worse than spending the extra commits now.

# What this round ships

1. Static-data symbol access as a typed wrapper — `StaticRef<'ext, T: 'static>` plus `Extension::resolve_static<T>(&self, name)`. Pointer-to-static-data deref pattern; lifetime-tied to the `Extension`; sound by construction.
2. Extended `ExtensionSymbol` arities from zero-to-four to zero-to-eight. Every arity expanded by a short declarative macro; no hand-written repetition.
3. Explicit `Extension::close(self)` method. Drop still works (RAII default); `close` is explicit form for consumers that want to surface loader errors at unload time instead of dropping them into the void.
4. CI workflow at `.github/workflows/ci.yml` — cargo check + cargo test on ubuntu-latest and macos-latest; cross-compile check for windows-latest (native Windows runs can be added in a follow-up if needed). Workflow runs on PR and on push to `dev`.
5. Tests cover: `resolve_static` for a known system-library static (`stdout` on unix as an example of pointer-to-static-data), extended-arity function pointer resolve, explicit `close` happy path.

# Out of scope (deferred, correctly)

- Async wrappers — no current consumer needs them; sync dlopen is fast.
- wasm32 targets — `wasm-component-model` is a different mechanism; genuinely warrants its own transport if and when needed.
- Capability-level compatibility negotiation — belongs in hilavitkutin-plugins (task #191).
- Signing and signature verification — consumer-policy concern; not a loader primitive.
- Loader telemetry — no current consumer needs it.

These remain in BACKLOG.md.tmpl. Everything else comes out.
