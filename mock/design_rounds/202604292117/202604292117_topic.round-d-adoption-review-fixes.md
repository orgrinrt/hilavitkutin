## Changes proposed

Three source-side adjustments. None changes any visible API, none touches doc templates.

`crates/hilavitkutin-str/src/interner.rs`. The `intern` and `intern_static` paths construct `Bits::<28, Hot>::from_raw(id)` directly from the arena's u32 id. The reviewer noted that `Bits::from_raw` is a transparent constructor (verified at `arvo-storage/src/bits.rs:77`) and does not mask. The 28-bit invariant is enforced one layer down, by `StrLayout::with_id` (the bitfield-generated setter masks input to the field width; verified by the existing `make_masks_to_28_bits` test at `tests/handle.rs:21-25`). Add a comment at each site that names the invariant carrier explicitly so future readers do not re-derive the soundness story.

`crates/hilavitkutin-persistence/src/archive_str.rs:34` and `crates/hilavitkutin-persistence/tests/archive_str.rs:50`. The expression `const_fnv1a(s) & Str::ID_MASK.to_raw() as u64` parses correctly under Rust's precedence rules (`as` binds tighter than `&`), but reads ambiguous to a reviewer not holding the full precedence table in head. Parenthesise the cast at both sites to make the intent self-evident.

The deferred should-consider (reviewer point 2): the str_const! macro emission still uses `Bits::<28>::from_raw((... & 0x0FFF_FFFF) as u32)`. Routing through `narrow_to::<28>()` would drop the lint:allow and the bare hex constant. Attempted in this polish round; arvo's current `Narrow` implementations cover only cross-width primitive narrows (`u32 → u16 / u8`), not same-container narrows (`u32 → u32`). The cross-`Bits` impl forwards through the underlying primitive `Narrow<T_N>` impl, so `Bits<32, Hot>::narrow_to::<28>()` requires `u32: Narrow<u32>`, which is not implemented. Tracked as #290 (arvo Widen counterpart + facade re-exports + same-container narrow coverage). The macro emission stays at the explicit-mask form for this round.
