# BuilderInput rename + Resource/Column/Virtual reshape

The 202605091700 round validated `.with(value)` as the unified builder method via wrapper structs (`Resource<T>`, `Column<T>`, `Virtual<T>`, `LinkedBin<T>`). That design works but pushes per-kind-disambiguation to the call site, forcing consumers to write `.with(Resource::new(Interner))` and `.with(Column::<Player>::new())` instead of bare `.with(Interner)` / `.with(Player)`.

This topic reshapes the API to mirror the WU/Kit pattern (sub-trait + supertrait constraint + per-type direct impl) so consumers bare-pass everywhere, with kind disambiguation moved to trait identity at decl site.

## Locked decisions for this topic

### D1. Trait rename: `Provider` → `BuilderInput`

`Provider` is misleading. WUs do not "provide" anything to the builder; they consume input via `Read`. Resource provides storage but is really just an input the builder consumes. Every type passed to `.with()` is a builder input. Rename for accuracy.

### D2. Drop `Provider::KIND` const

The const was load-bearing for nothing. `provider.rs:14` already documents it as "documentation and debugging aid; the load-bearing dispatch is via Provider::Dispatch." With sub-traits per kind, the trait identity itself carries the kind information; storing the same fact as a const is bloat.

### D3. Sub-traits per kind, supertrait-constrained on `BuilderInput`

Mirror the existing WU/Kit pattern uniformly. WUs and Kits today have:

```rust
pub trait WorkUnit<S = Always>: Provider<Init = Self> + Send + Sync + 'static { /* ... */ }
pub trait Kit: Provider<Init = Self> { /* ... */ }
```

After this reshape, all kinds use this pattern:

```rust
pub trait BuilderInput: Sized {
    type Init = Self;        // assoc-type default; Resource carries Self, others override
    type Dispatch;           // routes to UnitDispatch / StoreDispatch / KitDispatch / PlatformDispatch
}

pub trait Resource: BuilderInput<Dispatch = StoreDispatch<Self>> + Sized + 'static {}
pub trait Column: BuilderInput<Dispatch = StoreDispatch<Self>, Init = ()> + Sized + 'static + ColumnValue {}
pub trait Virtual: BuilderInput<Dispatch = StoreDispatch<Self>, Init = ()> + Sized + 'static {}
pub trait WorkUnit<S = Always>: BuilderInput<Dispatch = UnitDispatch<Self>> + Send + Sync + 'static {
    type Read: AccessSet;
    type Write: AccessSet;
    type Hint;
    type Ctx;
    fn execute(&self, ctx: &Self::Ctx);
}
pub trait Kit: BuilderInput<Dispatch = KitDispatch<Self>> + Sized + 'static {
    type Units: WorkUnitBundle;
    type Owned: StoreBundle;
}
pub trait MemoryProvider: BuilderInput<Dispatch = PlatformDispatch<Self>> + MemoryProviderApi + Send + Sync + 'static {}
pub trait ThreadPool: BuilderInput<Dispatch = PlatformDispatch<Self>> + ThreadPoolApi + Send + Sync + 'static {}
pub trait Clock: BuilderInput<Dispatch = PlatformDispatch<Self>> + ClockApi + Send + Sync + 'static {}
```

The supertrait bound `Dispatch = StoreDispatch<Self>` enforces consistency at compile time. A consumer impling `Resource for Interner` while having `BuilderInput for Interner { type Dispatch = UnitDispatch<Self>; }` fails the supertrait constraint. Mismatch is a compile error.

### D4. Wrapper structs `Resource<T>` / `Column<T>` / `Virtual<T>` deleted

No longer needed. Consumer types are themselves the BuilderInput/Resource/Column/Virtual. Bare-pass at call site:

```rust
struct Interner { /* fields */ }
impl BuilderInput for Interner { type Dispatch = StoreDispatch<Self>; }
impl Resource for Interner {}

struct Player { /* fields */ }
impl BuilderInput for Player { type Dispatch = StoreDispatch<Self>; type Init = (); }
impl Column for Player {}
impl ColumnValue for Player { /* BIT_WIDTH */ }

struct Tick;
impl BuilderInput for Tick { type Dispatch = StoreDispatch<Self>; type Init = (); }
impl Virtual for Tick {}

// call site (bare-pass everywhere):
let sched = Scheduler::builder()
    .with(Interner { /* ... */ })
    .with(Player::default())
    .with(Tick)
    .with(MyClock)
    .with(SpawnerWu)
    .with(InternerKit)
    .extend::<dyn LinterApi>()
    .build();
```

### D5. `LinkedBin<T>` → `ExtensionSurface<T>` rename + dedicated `.extend::<dyn Trait>()` builder method

LinkedBin is conceptually different from Resource/Column/Virtual. It is not a slot for app-internal state; it is the app declaring "I open an extension surface here for plugins implementing `dyn Trait`". The existing name buries that intent.

Rename: `LinkedBin<T: ?Sized>` → `ExtensionSurface<T: ?Sized>`.

Dedicated builder method instead of `.with(ExtensionSurface::<dyn Trait>::new())`:

```rust
pub fn extend<T: ?Sized + 'static>(self) -> SchedulerBuilder<...>
```

Wraps internally as `ExtensionSurface<T>` and routes via the same store accumulator. Reads at the call site as the app opening a surface, which matches the actual semantics.

### D6. `feature(specialization)` → `feature(min_specialization)` corrective

Audit: `hilavitkutin-api/src/lib.rs:17` enables `feature(specialization)` (full, unsound, never-stabilising). The only specialization use in source is `column_value.rs:26-44` which is compatible with `min_specialization` (concrete-type specialisations of a `Copy + 'static` blanket — "always applicable" rule). Doc-comment at `column_value.rs:4` already says "min_specialization"; the wider feature flag was speculative. Corrective: change the gate to `min_specialization`. Sketch validates `column_value.rs` still compiles.

### D7. `feature(associated_type_defaults)` enabled

Required for `BuilderInput::Init = Self` default. Stable-track feature with no soundness controversy.

## Why not a single blanket `impl<T: Resource> BuilderInput for T`?

Coherence. Multiple blanket impls of `BuilderInput` bounded on different sub-traits (`impl<T: Resource> BuilderInput for T`, `impl<T: Column> BuilderInput for T`, etc.) overlap in Rust's type system because nothing prevents `T: Resource + Column`. Specialization would solve it but is rejected (soundness). `#[marker]` solves marker-trait overlap but doesn't propagate to non-marker `BuilderInput` impls and is unavailable for `Column` (has `type Item`).

The pair-impl pattern (consumer writes both `impl BuilderInput for X { ... }` and `impl Resource for X {}`) is the floor that Rust's coherence allows without specialization. It is what WUs and Kits already do; this topic just extends the same pattern to Resource/Column/Virtual uniformly.
