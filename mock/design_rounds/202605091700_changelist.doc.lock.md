## File: `mock/crates/hilavitkutin-api/DESIGN.md.tmpl`

### Add new section `## Provider, the unified registration contract`

Add after the existing introduction, before the `WorkUnit` /
`AccessSet` sections.

Content covers:

- `Provider` is the sealed trait that gates registration on the
  SchedulerBuilder. Every type accepted by `.with(value)` impls it.
- Associated types: `type Init` (the construction-time value type;
  `()` for stateless wrappers, `Self` for unit-struct WUs/Kits/platform
  impls, the inner T for `Resource<T>`); `type Dispatch: Dispatch`
  (per-kind typestate router, see Dispatch section below); `const
  KIND: ProviderKind` (documentation aid).
- The `#[diagnostic::on_unimplemented]` message named in the topic
  file (verbatim text in the src CL).
- Sealed via `private::Token`-style sealing (mechanical pattern;
  src CL specifies).

### Add new section `## Dispatch — per-kind typestate routing`

Subsection after Provider. Content:

- `pub trait Dispatch` declares three GATs:
  `type NextWus<Wus>`, `type NextStores<Stores>`,
  `type NextPlatform<Platform>`. Each computes the typestate
  accumulator's next value when a provider of the corresponding kind
  registers.
- The four router structs: `UnitDispatch<W>`, `StoreDispatch<S>`,
  `KitDispatch<K>`, `PlatformDispatch<P>` (full words, no truncation).
  Each has the per-kind GAT body the topic file specified
  (UnitDispatch prepends to Wus; StoreDispatch prepends to Stores;
  KitDispatch uses Concat for both Units and Owned; PlatformDispatch
  prepends to Platform).
- Why associated-type dispatch (single blanket impl over Provider)
  beats per-kind blankets: rust trait solver fails on Provider
  directly when a non-Provider type is passed, surfacing the right
  diagnostic. Reference the sketch at
  `mock/research/sketches/202605091700-builder-provider-shape/`.

### Add new section `## ProviderKind`

Public enum with variants WorkUnit, Resource, Column, Virtual,
LinkedBin, Kit, Memory, Threads, Clock. Doc note: this is a
documentation/debugging aid; load-bearing dispatch is via
`Provider::Dispatch`.

### Modify existing `Kit` section

`pub trait Kit: Provider<Init = Self> { type Units: WorkUnitBundle;
type Owned: StoreBundle }`. Same supertrait note as WorkUnit.

### Modify `Resource<T>`, `Column<T>`, `Virtual<T>` sections

Add to each: this type impls `Provider` and `HasTrivialCtor` (via
`pub const fn new() -> Self`). Construction at the SchedulerBuilder
call site is `Resource::new(value)` for stateful, `Column::<T>::new()` /
`Virtual::<T>::new()` for stateless wrappers.

### Add new section `## LinkedBin<T: ?Sized>`

New wrapper introduced this round. Type-system marker for an
extension-loaded provider bin keyed by trait family. Impls
`Provider<Init = ()>` with `Dispatch = StoreDispatch<Self>`. Impls
`HasTrivialCtor` for the `Type::<dyn TraitFamily>::new()` form.
The runtime instance population happens in `hilavitkutin-extensions`
+ `hilavitkutin`; this wrapper is the type-system anchor only.

### Drop / replace any text mentioning `add_unit`, `add_resource`, `add_column`, `add_virtual`, `add_kit`, `with_memory`, `with_threads`, `with_clock`

The doc surface no longer references the old method names. Examples
become `.with(value)` shape.
