# Context

The hilavitkutin stack is growing a general plugin capability that downstream consumers (first and most prominently viola) will use to load third-party code at runtime. Work on the higher-level plugin contract, descriptor registration, lifecycle orchestration, and failure policy has been sketched under `hilavitkutin/crates/hilavitkutin-plugins/TODO.md`. All of that machinery sits above a lower layer that is strictly concerned with one thing: opening a shared library, resolving an explicit symbol, and returning a typed handle to the caller.

Prior art across the workspace shows what happens when the lower layer is absent. Three sibling projects have independently implemented ad-hoc loaders: `loimu-extern` with its explicit manifest symbol convention, `saalis-sdk` with raw dlopen bound to domain-specific descriptors, and `polka-dots`' plugin subsystem with `cdylib`-only plugin packaging plus the `inventory` crate for registration. Two of those three conflate binary-loading mechanics with plugin contract semantics, which couples the loader to the domain and prevents reuse across downstream consumers. The third avoids the coupling but inherits cross-platform fragility from linker-magic registration.

The crate proposed here is the clean separation. It owns the platform-specific mechanics of dynamic library loading and symbol resolution, and nothing else. It does not know what a plugin is, what a descriptor looks like, what a lifecycle contract says, or what role dispatch means. Those concerns live one layer up in `hilavitkutin-plugins`, and further specialisation (NAM, diagnostics, runner/grammar/lint roles) belongs in consumer contract crates such as viola-plugin-abi. This document locks the boundary and the key design decisions before the doc and src changelists land.

# Problem statement

Every downstream consumer that wants to load runtime extensions currently has to pick between:

1. Pulling in `libloading` directly, which introduces a std dependency and forces every consumer to reimplement structured error mapping, typed symbol wrappers, version compatibility checks, and platform artifact extension handling.
2. Writing raw dlopen against `libc`, which avoids std but duplicates the low-level wrapping work across consumers and produces inconsistent error shapes.
3. Reaching for registration shortcuts like `inventory` or `.init_array` constructors, which are unreliable across the OS / linker / rustc version matrix (particularly when plugins are loaded at runtime via dlopen rather than linked at build time).

None of those are acceptable defaults for this workspace. The stack already commits to a `#![no_std]` posture, explicit static composition, and deterministic behaviour across platforms (hilavitkutin CLAUDE.md). The loader layer must match that posture.

# Scope

This crate owns:

- Cross-platform dynamic library loading (`.dylib` on macOS, `.so` on Linux, `.dll` on Windows)
- Pull-based symbol resolution via explicit, named exports
- Typed symbol handles with lifetime bound to the loader handle
- Structured errors covering path-not-found, load failure, link failure, missing symbol, platform mismatch, and basic compatibility markers
- Optional compatibility-check hooks invoked at load time so downstream hosts can reject incompatible artifacts before proceeding to plugin-level validation
- Platform artifact extension handling (inferring the right suffix per target OS when callers pass a stem rather than a full path)

The crate does not own:

- Plugin descriptor semantics (shape, fields, role declarations)
- Plugin lifecycle contracts (initialize, invoke, shutdown)
- Plugin host orchestration (which plugins run, in what order, with what failure policy)
- Domain-specific contracts (runner/grammar/lint roles, NAM schemas, diagnostics formats)
- Procedural macros for static monomorphisation of plugin code
- Plugin registration, capability matching, or capability declaration
- Unload behaviour beyond RAII release of the loader handle (explicit unload is out of v1 scope)

# Non-goals

- An in-process Rust `dylib` path: explicitly rejected. The optimisation barrier that a `cdylib` plus `dlsym` boundary provides is load-bearing for this ecosystem. Rust `dylib` couples compiler versions and breaks the intended performance-barrier semantics demonstrated by polka-dots' benchmarks.
- Linker-magic registration: explicitly rejected. `inventory`, `ctor`, `#[distributed_slice]`, or `.init_array` constructors are banned on both the producer and consumer sides of the boundary. Discovery is strictly pull-based via explicit exported symbols.
- Asynchronous loading: v1 is synchronous. A future evolution may add async wrappers above this API, but the core interface stays sync.
- Remote extension loading, sandboxing, Wasm containers: all out of scope. Those belong to higher-level consumer-specific plugin profiles if they land at all.
- Version-and-manifest parsing: the loader returns raw typed symbols and a narrow compatibility hook. Interpreting a plugin manifest is a `hilavitkutin-plugins` concern.

# Architectural position

```
  viola-plugin-abi      <-- domain-specific plugin contract (runner/grammar/lint, NAM, diagnostics)
  -------------------
  hilavitkutin-plugins  <-- domain-agnostic plugin host orchestration
  -------------------
  hilavitkutin-extensions  <-- THIS CRATE: cross-platform library loading + symbol resolution
  -------------------
  OS dynamic loader (dlopen / LoadLibrary)
```

`hilavitkutin-extensions` depends only on the OS loader and the notko / arvo substrate crates needed for primitive vocabulary. Nothing in the rest of the stack depends upward from it. The engine itself (`hilavitkutin` crate) does not use this crate — it is static composition only, which is the whole point of the engine's contract.

# Key decisions to lock

## D1 — `#![no_std]` posture with a narrow FFI exception

Every hilavitkutin crate is `no_std`. This one is an edge case: FFI to `dlopen` / `LoadLibrary` inherently requires `libc` or platform API bindings, and the typed symbol API needs care to keep heap-free. The decision: the crate is `#![no_std]` at the crate root; FFI is expressed via `core::ffi` types where available and a minimal `libc`-level raw FFI where not. Error types are fixed-size variants, not heap strings.

Rationale: the `libloading` crate is well-engineered but requires `std` transitively for its error types (which allocate), and introduces an impl surface that would force a `#![feature(...)]`-level exception on the consumer. Raw dlopen with `core::ffi::c_void` + `libc` stays inside the no_std posture.

## D2 — cdylib + explicit symbols, nothing else

Only `cdylib` plugins are supported. This enforces the Rust ABI stability boundary (C ABI, not Rust ABI) and preserves the LLVM optimisation barrier demonstrated by polka-dots' benchmarks.

Symbol resolution is strictly pull-based: the host calls `extension.resolve::<T>(b"__hilavitkutin_plugin_descriptor\0")` and the loaded library must export that exact symbol. No linker-section registration is recognised or scanned. `.init_array` constructors inside loaded plugins are tolerated at the OS level (dlopen triggers them) but are not a contract this crate exposes — plugins that depend on constructor execution for registration will fail validation at the `hilavitkutin-plugins` layer.

## D3 — Typed Symbol handles with lifetime tied to the loader

`resolve<T>` returns a typed handle parameterised over `T` with a lifetime tied to the loader's lifetime. Drop order is loader drops after all symbols, enforced by Rust's borrow checker via the lifetime parameter. This makes use-after-unload a compile error, not a runtime bug.

`T` is restricted to types with defined FFI representation: function pointer types, `#[repr(C)]` struct references, `c_void` pointers, and similar. The crate provides validation helpers for the common shapes.

## D4 — Structured error model with no heap

Errors are a fixed enum with inline data. No owned strings, no boxed types. Where an OS error message is useful, the variant carries a fixed-capacity buffer or a stable error code (platform `errno` or `GetLastError` equivalent) that the consumer can format if it has allocator access. Crates in this workspace route these errors through `notko::Outcome` at their own boundary.

## D5 — Compatibility hook is a narrow contract, not a plugin manifest parser

The crate exposes a `compatibility_check(&[u8]) -> Outcome<(), IncompatibilityError>` hook callable before returning the loader handle to the consumer. The input bytes are the raw value of a single well-known compatibility symbol (for example a version bytestring exported by the plugin). The hook is called by the consumer, not automatically; `hilavitkutin-extensions` only provides the resolution primitive. All plugin-level manifest parsing, role checking, capability matching stays in `hilavitkutin-plugins`.

## D6 — Platform coverage

v1 supports macOS (aarch64, x86_64), Linux (x86_64, aarch64), and Windows (x86_64). Platform-specific path extension handling is provided (accepting a stem and inferring `.dylib` / `.so` / `.dll`, or accepting a full path and validating the extension is one of the recognised set). Additional platforms can be added later without changing the public API.

## D7 — Loading backend choice

Raw dlopen via `libc` / `LoadLibraryW` via `windows-sys`. Not `libloading`.

Rationale: `libloading` is a well-maintained crate, but it depends on `std` and on `libc::Error`-derived types that allocate. It also ships its own `Library` and `Symbol` abstractions that would duplicate the ones the loader design calls for. The raw FFI surface is small (four or five functions per platform) and writing it directly is cleaner than depending on an alloc-requiring crate.

The src CL will lock the precise FFI bindings (functions used, `extern "C"` signatures, `#[link(name = ...)]` attributes per target). Open question for the src CL: whether to depend on `windows-sys` for the Windows side (it is `#![no_std]`-compatible and de facto the standard binding) or hand-roll those FFI declarations inline. Defaulting to `windows-sys` with a narrow feature selection, reviewed in the src CL.

## D8 — Primitive-vocabulary discipline

Public signatures follow the workspace discipline:

| Bare type | Replacement |
|-----------|-------------|
| `Option<T>` | `notko::Maybe<T>` |
| `Result<T, E>` | `notko::Outcome<T, E>` |
| `usize` | `arvo::USize` (for capacities and sizes); raw `usize` only at direct FFI boundaries |
| `bool` | `arvo_bits::Bool` |
| `&'static str` / literal byte patterns | symbol name constants stay as `&[u8]` null-terminated byte literals at the FFI boundary; no `String` / `hilavitkutin-str::Str` on pub API since this crate ships below the str crate in the layering |
| `Vec<T>` / `String` / `Box` / heap containers | forbidden outright |
| `dyn Trait` | forbidden |

Where std trait impls force an escape hatch (for example `fmt::Debug` on an error type returning `fmt::Result`), the impl uses the bare types only inside the impl body; the public API stays on notko / arvo primitives.

## D9 — Macro support is out of scope

Procedural macros for `#[export_plugin]`-style monomorphisation belong to consumer contract crates (`viola-plugin-abi` in particular, via a `viola-plugin-macros` sub-crate). This crate ships no macros. The `hilavitkutin-plugins` crate may provide a generic descriptor-emission macro but that is a separate decision taken in its own design round.

# API shape (preview, to be locked in doc CL)

```rust
// module root
pub struct Extension { /* opaque handle */ }
pub struct Symbol<'ext, T> { /* borrowed handle, lifetime-tied */ }

pub enum ExtensionError {
    PathNotFound { /* fixed path buffer */ },
    LoadFailed { platform_code: ISize },
    SymbolMissing { /* fixed symbol name buffer */ },
    PlatformMismatch { /* expected / actual triple */ },
    IncompatibleVersion,
    // ... a small closed set, no String payloads
}

pub enum IncompatibilityError {
    VersionSkew,
    ArchMismatch,
    AbiMismatch,
}

impl Extension {
    pub fn load(path: &[u8]) -> Outcome<Self, ExtensionError>;

    pub fn resolve<T>(&self, name: &[u8]) -> Outcome<Symbol<'_, T>, ExtensionError>
    where
        T: ExtensionSymbol;
}

impl<'ext, T> Symbol<'ext, T> {
    pub fn get(&self) -> &T;
}

pub trait ExtensionSymbol: private::Sealed {}

impl<R> ExtensionSymbol for extern "C" fn() -> R {}
impl<R, A> ExtensionSymbol for extern "C" fn(A) -> R {}
// ... etc for common arities

// helper for fixed-layout pointer-to-descriptor patterns
pub struct StaticDescriptor<'ext, T: 'static> { /* wraps *const T */ }
impl<'ext, T: 'static> ExtensionSymbol for StaticDescriptor<'ext, T> {}
```

The exact trait bounds and sealed marker mechanics are worked out in the doc CL. Lifetimes are the key constraint: no `Symbol` outlives its `Extension`; no owned state crosses the FFI boundary.
