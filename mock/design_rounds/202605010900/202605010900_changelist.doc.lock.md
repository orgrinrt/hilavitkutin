### `mock/crates/hilavitkutin-api/DESIGN.md.tmpl`

Two new subsections under the AccessSet / Contains section (or a new top-level section "Builder support" if a clean home doesn't exist; verify by inspection during DOC apply).

1. **`Buildable<Stores>` sealed trait.** Documents the marker that `.build()` carries as a where-clause: `Wus: Buildable<Stores>`. Per-arity blanket impls (one per arity 0..=12, matching the existing AccessSet cap) reduce to `Stores: WuSatisfied<Wn::Read> + WuSatisfied<Wn::Write>` for every `Wn` in the tuple. Sealed; consumers cannot impl directly. The trait lives in api because it's a contract referenced by `WorkUnit` users at the call site of `.build()`.

2. **`WuSatisfied<A>` sealed trait.** Documents the per-A-arity reduction: "all members of A are present in Self". Per-arity blanket impls (0..=12) reduce to `Self: Contains<T>` for every `T` in `A`. Sealed.

3. **Recursive `Contains` impl.** Documents the new additive impl on the existing `Contains<T>` trait:

   ```rust
   impl<H: 'static, R: 'static, T: 'static> Contains<T> for (H, R)
   where R: Contains<T> {}
   ```

   Prose: this propagates `Contains<T>` membership down cons-list-shaped tuples. The builder accumulates `Stores` as `(H, (H, (H, ())))` per registration; the recursion makes membership chain through arbitrary depths. Overlaps with the existing flat-tuple `Contains<T0> for (T0, T1)` impl; permitted because `Contains` is `#[marker]`. Existing flat-tuple impls stay unchanged; they govern `WorkUnit::Read` / `Write` declarations, which remain flat tuples at the consumer level.

4. **Why these live in api, not in kit.** One sentence: `Buildable` and `WuSatisfied` are contract-shape artefacts referenced in `WorkUnit::Ctx` bound conjunctions and at `.build()` call sites. They belong with `AccessSet` and `Contains`, not with the `Kit` trait that just composes onto whatever the builder is.

### `mock/crates/hilavitkutin/DESIGN.md.tmpl`

Locate the scheduler module section. Replace the "builder skeleton (deferred to 5a2/5a3/5a4)" stance with a description of the new shape:

1. **`Scheduler<MAX_UNITS, MAX_STORES, MAX_LANES>`** keeps its existing const generics for engine-internal capacity bounds. No change in those parameters.

2. **`SchedulerBuilder<MAX_UNITS, MAX_STORES, MAX_LANES, Wus, Stores>`** evolves: gains two phantom-tuple type parameters tracking accumulated WU types (`Wus`) and accumulated store markers (`Stores`). Initial state on `Scheduler::builder()` is `SchedulerBuilder<..., (), ()>`.

3. **Method shapes.** Document the type-state transitions per method:

   ```rust
   impl<..., Wus, Stores> SchedulerBuilder<..., Wus, Stores>
   where Wus: AccessSet, Stores: AccessSet,
   {
       pub fn add<W: WorkUnit>(self) -> SchedulerBuilder<..., (W, Wus), Stores>
       where (W, Wus): AccessSet;

       pub fn resource<T: 'static>(self, _init: T)
           -> SchedulerBuilder<..., Wus, (Resource<T>, Stores)>
       where (Resource<T>, Stores): AccessSet;

       pub fn column<T: 'static>(self) -> SchedulerBuilder<..., Wus, (Column<T>, Stores)>
       where (Column<T>, Stores): AccessSet;

       pub fn virtual_<T: 'static>(self) -> SchedulerBuilder<..., Wus, (Virtual<T>, Stores)>
       where (Virtual<T>, Stores): AccessSet;

       pub fn add_kit<K: Kit<Self>>(self, k: K) -> K::Output;

       pub fn memory<M: 'static>(self, _provider: M) -> Self;
       pub fn threads<P: 'static>(self, _pool: P) -> Self;
       pub fn clock<C: 'static>(self, _clock: C) -> Self;
   }

   impl<..., Wus, Stores> SchedulerBuilder<..., Wus, Stores>
   where
       Wus: Buildable<Stores>,
       Stores: AccessSet,
   {
       pub fn build(self) -> Scheduler<MAX_UNITS, MAX_STORES, MAX_LANES>;
   }
   ```

4. **Compile-fail behaviour.** Add a worked example: a WU declares `Read = (Resource<Interner>,)`, the consumer forgets to call `.resource(Interner)` or `.add_kit(InternerKit)`, `.build()` reports `trait bound (): Contains<Resource<Interner>> was not satisfied`. The error names the missing store directly, by type.

5. **Platform providers (`memory` / `threads` / `clock`).** Note that these stay non-typestate for this round; they thread through `Self` unchanged. Tracking-via-type-state for platform providers is a future round if an actual call site demands it. Currently the WU `Ctx` bound conjunction includes the platform-provider accessors via `HasResourceProvider` projections, so platform-provided values reach WUs through the Resources slot anyway.
