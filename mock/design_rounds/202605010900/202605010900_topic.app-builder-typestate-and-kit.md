# Topic: SchedulerBuilder type-state + Kit primitive

**Date:** 2026-05-01
**Phase:** TOPIC
**Scope:** New crate `hilavitkutin-kit` plus `hilavitkutin/src/scheduler/`. Evolves the no-op builder skeleton into a phantom-tuple type-state shape that proves WU `Ctx` bounds at `.build()`. Introduces the `Kit` trait as a Bevy-style preset for bundling registrations.

**Source task:** #255 (SUB-H6). Related parked: #295 (`app!` macro, P2-likely-never).

## Current state

`hilavitkutin/mock/crates/hilavitkutin/src/scheduler/mod.rs` is a 92-line skeleton. `Scheduler::builder()` returns `SchedulerBuilder<MAX_UNITS, MAX_STORES, MAX_LANES>` whose methods are all no-ops returning `Self`:

```rust
pub fn add<WU: 'static>(self) -> Self { self }
pub fn resource<T: 'static>(self, _init: T) -> Self { self }
pub fn column<T: 'static>(self) -> Self { self }
pub fn memory<M: 'static>(self, _provider: M) -> Self { self }
pub fn threads<P: 'static>(self, _pool: P) -> Self { self }
pub fn clock<C: 'static>(self, _clock: C) -> Self { self }
pub fn build(self) -> Scheduler<MAX_UNITS, MAX_STORES, MAX_LANES> { Scheduler::default() }
```

The mod.rs comment notes this is intentional: "ships the builder skeleton only; the actual execution loop requires 5a2/5a3/5a4 to be in place first". Type-level accumulation was deferred.

`hilavitkutin-api` ships:

- `WorkUnit<Schedule>` with `type Read: AccessSet`, `type Write: AccessSet`, `type Hint: SchedulingHint`, and a `type Ctx` bound that is the conjunction of seven `HasX<...>` accessor traits (`HasColumnReader`, `HasColumnWriter`, `HasResourceProvider`, `HasVirtualFirer`, `HasEach`, `HasBatch`, `HasReduce`).
- `AccessSet`: sealed marker on tuples at arities 0 through 12.
- `Contains<S>`: `#[marker]` membership witness, one impl per position per arity.
- `Resource<T>`, `Column<T>`, `Virtual<T>`: zero-sized store markers.

`hilavitkutin-ctx` ships `provider!`, `provider_generic!`, `provider_generic2!`, `tuple!`, and `define_providers!` macros. The seven `HasX` accessor traits in `hilavitkutin-api/src/context.rs` are emitted from `provider_generic!`/`provider_generic2!`.

No prior round has touched Kit or the builder type-state. Most recent round (`202604510000`) was the FFI-boundary primitive policy.

The "Kit" name was chosen 2026-04-26 as workspace vocabulary (over Bundle/Preset/Package/Module/Extension) to match Rust ecosystem conventions and avoid conflicts with existing terms in the stack.

## Decisions locked in TOPIC

Five decisions were settled in conversation before this file was written:

1. **Round scope is mechanism only.** `app!` / `define_app!` declarative macro is parked at #295 (P2-likely-never). Bevy-style raw builder chains are proven ergonomically sufficient by widespread framework precedent. The chains in `viola/docs/HILAVITKUTIN-APP-SHAPE.md` are about ten calls long; no readability win justifies macro complexity.

2. **`Kit` lives in a new crate `hilavitkutin-kit`.** Slots between `hilavitkutin-ctx` and the engine. Ninth crate in the stack. `hilavitkutin-kit` depends on `hilavitkutin-api` (contracts) and `hilavitkutin-ctx` (provider macros). The engine (`hilavitkutin`) gains a new dep on `hilavitkutin-kit` to wire `.add_kit()` into the builder.

3. **Kit shape is method-only Bevy-style.**

   ```rust
   pub trait Kit<B> {
       type Output;
       fn install(self, builder: B) -> Self::Output;
   }
   ```

   No associated type list of contributed Resources/Columns/Providers. A Kit's `install` body just calls the builder's existing registration methods (`.add::<WU>()`, `.resource::<T>(v)`, `.column::<T>()`). The type-state evolves through whatever the builder's mechanism is. The Kit is a literal preset; nothing Kit-specific lives at the type level.

4. **Builder type-state is phantom-tuple accumulation.** SchedulerBuilder gains tail type parameters carrying tuples of registered WUs, Resources, Columns, and Kit-supplied accessor projections. Each registration method returns a builder whose tail tuples are extended. `.build()` carries a where-clause proving the union of registered WU `Ctx` bounds is satisfied by the registered providers. Reuses the existing `AccessSet` + `Contains<S>` machinery in `hilavitkutin-api/src/access.rs` (already arities 0..=12).

5. **`.add_kit(k)` is the builder method to install a Kit.** Chosen by parallel naming with `.add::<WU>()`. Method signature reads `fn add_kit<K: Kit<Self>>(self, k: K) -> K::Output`.

## Changes proposed

### New crate `hilavitkutin-kit`

Workspace member at `mock/crates/hilavitkutin-kit/`. Stack position between `hilavitkutin-ctx` and the engine. Standalone-usable: depends on `hilavitkutin-api` (contracts) and `hilavitkutin-ctx` (provider macros), no dep on the engine.

Contents:

- `Kit<B>` trait with `type Output` and `fn install(self, builder: B) -> Self::Output`.
- Possibly a small set of helper traits or marker types for common Kit-composition patterns (concrete shape settled in DOC phase).

`mockspace.toml` `[lints.forbidden-imports]` for `hilavitkutin-kit`: matches the same shape as `hilavitkutin-ctx` and `hilavitkutin-str` (no engine, no api-internals leakage; substrate-only).

`mockspace.toml` adds the new crate to `[workspace.crates]` and the appropriate `[crate.*]` table.

`README.md.tmpl`, `DESIGN.md.tmpl`, `BACKLOG.md.tmpl`, `SHAME.md.tmpl` follow the standard mock layer.

### `hilavitkutin` engine: SchedulerBuilder evolves

`mock/crates/hilavitkutin/src/scheduler/mod.rs` rewrites:

- `SchedulerBuilder` gains four tail type parameters carrying phantom tuples: registered WU types, registered Resource value types, registered Column value types, registered accessor-projection markers from Kits. Initial state on `Scheduler::builder()` is `SchedulerBuilder<..., (), (), (), ()>`.
- `.add::<WU>()` returns `SchedulerBuilder<..., (WU, Wus), Resources, Columns, Providers>`.
- `.resource::<T>(v)` returns `SchedulerBuilder<..., Wus, (Resource<T>, Resources), Columns, Providers>`. (`Resource<T>` is the marker type already in `hilavitkutin-api/src/store.rs`.)
- `.column::<T>()` returns `SchedulerBuilder<..., Wus, Resources, (Column<T>, Columns), Providers>`.
- `.add_kit<K: Kit<Self>>(self, k: K) -> K::Output`: delegates to `k.install(self)`.
- `.build()` carries a where-clause whose concrete shape is the load-bearing engineering detail of this round. The intent: prove that for every WU in `Wus`, its `Ctx` bound (the conjunction of seven `HasX<...>` traits) is satisfied by the accumulated providers.

Concrete `.build()` where-clause shape, AccessSet arity ceiling for accumulator tuples (currently 12, may need extension), and exact composition of "providers" (Resources, Columns, Kit projections, platform providers) are settled in DOC phase with code sketches.

### `hilavitkutin-api`: AccessSet arity ceiling

The accumulator tuples may exceed 12 entries in realistic apps. `viola/docs/HILAVITKUTIN-APP-SHAPE.md` already names six WUs, six Resources, three Columns; viola itself fits in 12 if WUs and stores are tracked separately. A larger consumer (vehje compiler) will exceed it.

DOC phase decides whether to:

- Extend `AccessSet` + `Contains` impls to a higher arity (24, 32). Doable; the existing `impl_access_set!` and `impl_contains!` macros are already templated.
- Switch accumulator tuples to a different mechanism (cons-list-style nested pairs `(W0, (W1, (W2, ...)))`).
- Cap consumer apps at the 12 ceiling and document the limit.

The cleanest call is probably to extend the existing macros to a higher fixed arity. Sketch and validate in DOC.

### Test crate

`mock/crates/hilavitkutin/tests/scheduler_builder.rs` already exists with two trivial assertions. This round rewrites it to exercise the type-state: a small fake WU + fake Resources + fake Columns + a fake Kit; `.build()` succeeds when bounds are satisfied; `.build()` fails to compile (compile-fail test or doc-test) when a WU declares a `Read: Contains<Column<X>>` and no `.column::<X>()` was called.

Compile-fail tests are typically `trybuild`-style. hilavitkutin doesn't ship a `trybuild` crate today; DOC phase decides whether to add it as a dev-dep or settle for runtime assertions on `core::any::type_name::<>()` (insufficient for the negative case) or skip the negative case until a follow-up round.

## Out of scope

- `app!` / `define_app!` macro. Parked as #295. Raw builder chains are the surface; macro can be revisited if a real ergonomics complaint surfaces during viola or vehje implementation.
- Concrete Kits like `InternerKit`, `MemoryKit`, `ColumnStorageKit`, `ClockKit`, `ThreadPoolKit`. Those ship in `hilavitkutin-providers` (#253) against the formalised pattern this round lands.
- Runtime dispatch of registered WUs. The scheduler `.run()` loop is still the 5a2/5a3/5a4 deferred work. This round only formalises the build-time shape.
- Replacing `MAX_UNITS / MAX_STORES / MAX_LANES` const generics. Those track engine-internal capacities, not the registered counts. Phantom tuples and const caps coexist.

## Open engineering questions for DOC phase

These are the details the DOC changelist resolves with concrete code sketches. They were not load-bearing enough to settle in TOPIC, but DOC must close them before SRC starts:

1. **AccessSet arity ceiling.** Extend the existing macros to arity 24 or 32, or switch accumulators to a different shape. Sketch both, pick the cleaner one.
2. **`.build()` where-clause shape.** How does Rust express "for every WU type in `Wus`, its `Ctx` is satisfied by the providers in `Resources + Columns + Providers`"? The likely shape is a sealed marker trait on `Wus` (something like `WusBuildable<Resources, Columns, Providers>`) with blanket impls per tuple arity. Sketch arity 1 and arity 4; verify it composes.
3. **Provider categorisation.** A Kit's "accessor projection" sits where in the tuple landscape? Same slot as a `tuple!`-emitted impl on a Resource tuple? Or a separate fourth slot? Sketch what `hilavitkutin-providers`' `InternerKit` would actually contribute, and let that drive the shape.
4. **Compile-fail test infrastructure.** Add `trybuild` as dev-dep, or defer negative-case testing to a follow-up round.
5. **Crate layering verification.** Confirm `hilavitkutin-kit` doesn't introduce a cycle. The dep graph is `notko` → `arvo*` → `hilavitkutin-api` → `hilavitkutin-ctx` → `hilavitkutin-kit` → `hilavitkutin (engine)`. No backwards edges.
