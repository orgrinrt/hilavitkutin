### `mock/crates/hilavitkutin-kit/src/lib.rs`

New file. Single trait declaration, `#![no_std]`:

```rust
//! Kit preset trait for the hilavitkutin scheduler builder.

#![no_std]

/// A preset that bundles a set of registrations onto a builder.
///
/// `B` is the input builder type; `Self::Output` is the resulting
/// builder type after the Kit's contributions have been applied.
/// The implementation calls the builder's existing registration
/// methods; the type-state evolves through whatever mechanism the
/// builder already uses.
///
/// See `hilavitkutin/DESIGN.md` for the engine-side
/// `SchedulerBuilder<MAX_*, Wus, Stores>` shape that consumes Kits
/// via `.add_kit(k)`.
pub trait Kit<B> {
    /// Builder type produced after `install` runs.
    type Output;
    /// Apply the Kit's registrations to `builder`.
    fn install(self, builder: B) -> Self::Output;
}
```

### `mock/crates/hilavitkutin-api/src/builder.rs`

New module file. Sealed traits supporting the engine's `.build()`:

- `Buildable<Stores: AccessSet>` sealed via private `BuildableSealed`. Per-arity blanket impls for `Wus` tuples 0..=12, each reducing to `Stores: WuSatisfied<Wᵢ::Read> + WuSatisfied<Wᵢ::Write>` for every `Wᵢ`.
- `WuSatisfied<A: AccessSet>` sealed via private `WuSatisfiedSealed<A>`. Per-arity blanket impls for `A` tuples 0..=12, each reducing to `Self: Contains<Tⱼ>` for every `Tⱼ` in `A`.

Both traits exported from the module. Sealing private; consumers cannot impl directly.

Macro-generate the per-arity impls; the existing `impl_access_set!` and `impl_contains!` macros in `access.rs` are the closest precedent. Use a fresh `impl_buildable!` plus `impl_wu_satisfied!` macro pair in the new module.

The `WorkUnit` trait import comes from `crate::work_unit::WorkUnit`. The `AccessSet` and `Contains` imports come from `crate::access`.

### `mock/crates/hilavitkutin-api/src/access.rs`

Append one impl after the arity-12 impls (around line 177):

```rust
// Cons-list recursion. The scheduler builder accumulates Stores
// as (H, R) at every step, where R is itself a cons-list. The
// arity-2 Contains<T0> for (T0, T1) impl above covers head
// matches. This recursive impl propagates membership down the
// tail. #[marker] on Contains permits the overlap.
impl<H: 'static, R: 'static, T: 'static> Contains<T> for (H, R)
where
    R: Contains<T>,
{
}
```
