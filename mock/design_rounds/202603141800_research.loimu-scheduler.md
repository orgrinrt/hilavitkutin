## 1. Signal-driven execution model

loimu's scheduler does not have ticks. It does not have frames. It does not have
stages. The execution model is signal-driven: policy timers emit signals, the
scheduler wakes, runs one complete DAG traversal, then sleeps.

The loop, in its entirety:

```
signals arrive -> scheduler runs the DAG -> scheduler finishes -> scheduler sleeps
```

There are no built-in policy signals. Policies are registered via `AppBuilder`.
Each policy runs its own timer in its own thread. When the timer fires, the
policy emits a user-specified signal. The scheduler wakes, emits `ScheduleStart`
into the bus, forwards the policy signal, traverses the DAG in topological
order, emits `ScheduleEnd`, and goes back to sleep.

Behaviors know nothing about policies or timers. They observe signals. An app
might define `InputPoll` at 120 Hz and `SimStep` at 60 Hz. Those are app-level
names, not framework types. The scheduler runs whichever subset of the DAG
observes the signals present on the bus at that moment.

### Signal bus arena swap

At pass boundaries, the signal bus uses a double-buffered pointer swap:

1. Write arena becomes read arena (pointer swap, not copy).
2. Old read arena resets.
3. Scheduler emits `ScheduleStart` into the now-empty write arena.

All signals from the previous pass are visible as the read arena during the
current pass. Signals emitted during the current pass go into the write arena,
invisible until the next swap.

### Signal filtering

Not every unit runs on every pass. Before executing each unit, the scheduler
checks whether its handler's observed signal is present on the bus:

- If the handler's observed signal set is empty, the unit runs unconditionally.
- If the handler observes specific signals, the scheduler checks each `SignalId`
  against the bus via `has_signal_id()`. If none are present, the unit is
  skipped entirely.

Signal observation is derived from `on Signal => handler { ... }` blocks in the
handler body. `ScheduleStart` fires on every wake regardless of which policy
triggered it, so observing it is the most frequent possible observation.
Handlers should prefer observing tight domain signals emitted by upstream
behaviors. The tighter each observation, the fewer hard sync points, the more
parallelism the scheduler can extract.

All matching uses typed IDs (`SignalId`, `ResourceId`, `UnitId`), never string
comparison. IDs are interned (u64 pointer equality), so each edge check is a
single integer comparison.

### Mid-schedule signal arrival

If a policy timer fires while a schedule is already running:

- If no observer of that signal has executed yet in this schedule, inject the
  signal immediately.
- If observers already ran, queue the signal. The scheduler starts a new
  schedule immediately after the current one completes.

This prevents signal loss without breaking causality within a schedule.

### Wake coalescing

If multiple policy timers fire while a pass is running, they are coalesced: all
pending wakes merge into a single pass after the current one finishes. The
scheduler does not queue separate passes per wake. All signals from coalesced
wakes are present in the write arena at pass start.

### Framework signals

`ScheduleStart`, `ScheduleEnd`, `QuitRequested`, `QuitReady`,
`TeardownStarted`, `BehaviorFaulted`, `RecalculateSchedulePlan`.

Signal ownership:
- `QuitRequested`, `TeardownStarted`, `QuitReady` -- loimu-scheduler
- `RuntimeStopping` -- loimu-runtime

## 2. Three policy types

Policies control when the scheduler wakes. Multiple policies can coexist in
the same app.

### Continuous

```
Continuous { interval }
```

Emits its signal at a fixed interval. Straightforward periodic execution.
An input polling policy at 120 Hz would use `Continuous { interval: 8.33ms }`.

### Adaptive

```
Adaptive { baseline, max }
```

Adjusts interval dynamically based on signal throughput:

1. Count signals emitted to the bus during the schedule.
2. Smooth with exponential moving average: `0.8 * old + 0.2 * new`.
3. Map to activity level [0.0, 1.0]: 0 signals = idle (0.0), 10+ signals =
   busy (1.0).
4. Lerp interval between `baseline` (idle) and `max` (busy).

An idle app decays toward `baseline` (low CPU). Active interaction ramps toward
`max`. After interaction stops, it decays back. `SchedulerMetrics` exposes
`activity_level` and current effective interval for diagnostics.

Recommended for UI apps.

### OnDemand

```
OnDemand { housekeeping }
```

Only emits when signals are pending; periodic housekeeping otherwise.
Recommended for background daemons.

### Overrun handling

The scheduler never queues separate passes per wake. If a pass is running when
a timer fires, the wake is coalesced into the next pass. There is no overrun
accumulation, no timer stacking. The scheduler runs one pass, sleeps, runs the
next. Signal injection rules (see section 1) determine whether mid-schedule
signals are injected or queued.

### Duration tracking

The scheduler measures each schedule's wall-clock duration and maintains a
rolling average in `avg_schedule_duration_us` with 1/8 exponential decay:

```
avg = (avg * 7 + measured) / 8
```

### CadenceMap

`CadenceMap` is a singleton resource owned by the scheduler-internal
`CadenceUpdateBehavior`. No user behavior writes `CadenceMap`. Each
`CadenceEntry` stores the last fire time and the policy. Cadence policies are
evaluated at schedule boundaries, not mid-schedule. If a cadence timer fires
while a schedule is running, the wakeup is queued and re-emitted at the start
of the next schedule.

## 3. List scheduling with upward rank

The scheduler uses list scheduling with upward rank, inspired by instruction
scheduling in compilers and HEFT (Heterogeneous Earliest Finish Time) in HPC.
The algorithm is fixed; the optimizations in later sections operate within this
framework.

### Step-by-step plan computation

Plan computation runs at startup and on recomputation triggers.

**Step 1: Decompose behaviors into scheduling units.** Each `on Signal =>
handler { ... }` block is decomposed into scheduling units. An `each` call
produces one fusible unit. `tree`, `bind`, `dispatch`, and `emit` each produce
a non-fusible unit. Behaviors are subgraphs in the DAG, not monolithic nodes.

**Step 2: Build dependency graph from unit-level access sets.** Edges are
field-granular. The proc macro infers per-field read/write sets from handler
bodies via `FieldRef<Owner, T>`. Two units touching different fields of the same
resource produce no edge.

Edge sources:
- Internal subgraph edges (access analysis within same handler)
- Signal flow: handler A emits S, handler B observes S -> edge A's emit unit to
  B's first unit
- Resource access: unit A writes field F, unit B reads F -> edge A to B
- Marker storage: depends on buffer mode (double-buffered read-vs-write is
  disjoint; single-buffered creates normal edge; write-vs-write always creates
  an edge)
- Tree structure: `write tree` to `read tree` creates an edge; two bare
  `write tree` are sequential; marker-scoped `write tree M1` vs `write tree M2`
  may be parallel if FilterCache shows disjoint marker sets
- Binding access: A writes binding B, C reads binding B -> edge A to C

**Step 3: Kahn's algorithm for cycle detection and depth computation.** loimu
uses Kahn's algorithm (not DFS-based toposort) because it naturally produces
level-based ordering: all nodes at the same depth appear in the same batch. If
the topological sort does not visit all units, a cycle exists. The scheduler
emits a `DagCycle` error with involved behavior/unit names and refuses to start.

```
procedure kahn_toposort(dag):
    in_degree = count incoming edges for each node
    queue = all nodes where in_degree == 0
    depth = {} // node -> depth mapping
    order = []

    while queue is not empty:
        u = queue.pop()
        order.append(u)
        for each successor v of u:
            in_degree[v] -= 1
            depth[v] = max(depth[v], depth[u] + 1)
            if in_degree[v] == 0:
                queue.push(v)

    if len(order) != len(dag.nodes):
        error("DagCycle: contradictory dependencies")

    return order, depth
```

Complexity: O(V + E) where V is the number of units and E is the number of
edges.

**Step 4: Compute upward rank per unit (bottom-up).** The rank formula:

```
rank(u) = estimated_cost(u) + max(rank(successors(u)))
```

This naturally identifies the critical path. The unit with the highest rank in
the entire graph is the start of the critical path.

Initial cost estimation (before runtime data):
- Base cost: 1
- +1 per declared write field (writes are heavier than reads)
- +1 per emitted signal type

After the first few passes, the scheduler replaces the heuristic with an EMA of
actual measured execution duration. The EMA alpha is tuned so the estimate
converges within approximately 10 passes for stable workloads.

Example for a graph with 5 units:

```
Unit A: writes 2 fields, emits 1 signal -> initial cost = 1 + 2 + 1 = 4
Unit B: reads only                      -> initial cost = 1
Unit C: writes 1 field                  -> initial cost = 1 + 1 = 2
Unit D: reads only                      -> initial cost = 1
Unit E: writes 1 field, emits 1 signal  -> initial cost = 1 + 1 + 1 = 3

DAG: A -> B -> D
     A -> C -> E

rank(D) = 1 + 0 = 1        (leaf)
rank(E) = 3 + 0 = 3        (leaf)
rank(B) = 1 + rank(D) = 2
rank(C) = 2 + rank(E) = 5
rank(A) = 4 + max(rank(B), rank(C)) = 4 + 5 = 9

Critical path: A -> C -> E (total weight 9)
```

**Step 5: Lane assignment via list scheduling.** Pick the highest-rank ready
unit, assign to the lane where it finishes earliest (considering AccessSet
conflicts with already-assigned units). Repeat until all assigned.

```
procedure assign_lanes(units, num_lanes):
    sorted = sort units by rank descending
    lane_finish_time = [0; num_lanes]

    for u in sorted:
        best_lane = None
        best_finish = infinity

        for lane in 0..num_lanes:
            if no AccessSet conflict with units already on lane:
                finish = lane_finish_time[lane] + estimated_cost(u)
                if finish < best_finish:
                    best_finish = finish
                    best_lane = lane

        u.lane = best_lane
        lane_finish_time[best_lane] = best_finish
```

**Step 6: Precompute overflow lists per lane gap.** For each unit, compute
`lane_compat: u8` bitmask indicating which lanes can safely run this unit
(no AccessSet conflicts). These are used at runtime for work stealing.

**Step 7: WASM plugin integration.** Plugin behaviors participate in normal
lane assignment after string-to-ID resolution at registration. Plugin
load/unload triggers `RecalculateSchedulePlan`; the plan is recomputed between
passes. There is no dedicated WASM lane.

### EMA convergence

The exponential moving average replaces static heuristics after the first few
passes. With alpha tuned for convergence within ~10 passes, the schedule plan
stabilizes quickly for steady-state workloads. Variance tracking triggers
recomputation when actual durations deviate significantly from estimates.

### Adaptive recomputation triggers

Unit assignments are recomputed when:
- `RecalculateSchedulePlan` signal is emitted (by app code or stage changes)
- `AppStage` resource changes (auto-observed)
- Unit duration variance exceeds a threshold (EMA of actual runtimes)

Recomputation happens between passes, never mid-pass.

## 4. Lane assignment and vertical-first dispatch

### Lanes as worker threads

A `Lane` is a thread-level execution context. The lane count is bounded by
`lane: u3` -- a maximum of 8 lanes (0 through 7). Each worker thread owns one
lane and follows its lane's unit list in order.

### Vertical-first execution

Each worker follows its lane's unit list sequentially. This is "vertical-first"
dispatch: a single lane advances through its full dependency chain before
considering horizontal work from other lanes.

Runtime execution per pass:

1. Pass interval elapses per `SchedulePolicy`.
2. Scheduler swaps signal bus arenas (pointer swap).
3. Scheduler emits `ScheduleStart` into the now-empty write arena.
4. Signal filtering: skip units whose observed signals are not present.
5. Each worker follows its lane's unit list in order (vertical-first).
6. Fusible units (`each` blocks) are dispatched to the storage engine for
   fused iteration. Non-fusible units (`tree`, `bind`, `dispatch`, `emit`)
   execute directly on the lane.
7. If blocked on a dependency: check overflow list (see section 5).
8. When the blocked dep resolves, resume the primary lane list.
9. Emit `ScheduleEnd`. Scheduler sleeps until next policy timer.

### Why vertical-first

Advancing a deep chain unlocks more downstream units for all workers. If a
worker instead picked the horizontally closest ready unit (breadth-first), it
would increase latency on long chains, which are typically on the critical
path.

The upward rank ensures that even when a worker pulls overflow work from another
lane (horizontal filling), it picks units that contribute to unblocking the
heaviest chains. Workers are idle only when ALL remaining units depend on
currently-running work.

### Subgraph model and portals

A behavior is a subgraph in the main DAG. The scheduler sees inside behaviors
and can interleave units from different behaviors when their access sets are
disjoint.

Each subgraph has computed portals per resource:
- **Claim point**: first unit accessing resource R.
- **Release point**: last unit accessing resource R.

External units can use R freely before the claim point and after the release
point. If a behavior's last write to R is unit 2 of 5, other behaviors can
start reading R after unit 2 completes. This is fine-grained resource sharing
that avoids locking an entire behavior's duration.

```
Main DAG:
  [SignalEmitter] -> [BehaviorA.handler1.unit0] -> [BehaviorA.handler1.unit1]
                  -> [BehaviorB.handler1.unit0]
                                    \
                    [BehaviorA.handler1.unit2] -> [consumer]
```

### Cache-affinity lane preference

Non-fusible units touching the same view prefer assignment to the same lane.
This is a soft preference in lane assignment, not a hard constraint. It
improves cache locality when multiple units read/write the same view's columns
sequentially. Upward rank and dependency constraints take priority over cache
affinity.

## 5. Work-stealing via overflow lists

When a worker's primary lane is blocked on a dependency, it does not spin-wait.
Instead it pulls compatible work from other lanes.

### Mechanism

At plan time, each unit has a precomputed `lane_compat: u8` bitmask indicating
which lanes can safely execute it (no AccessSet conflicts with the lane's
primary assignments). At runtime:

```
procedure steal_work(blocked_lane, all_lanes):
    candidates = []
    for each unit U in overflow list:
        if U.lane_compat & (1 << blocked_lane) != 0:
            if all dependencies of U are satisfied:
                candidates.push(U)

    if candidates is not empty:
        pick = candidates.max_by(|u| u.upstream_rank)
        execute(pick)
```

The key properties:

1. **lane_compat bitmask**: precomputed at plan time, O(1) to check. A unit
   with `lane_compat = 0b00001111` can run on lanes 0-3 but not 4-7.

2. **upstream_rank priority**: among stealable candidates, pick the one with
   the highest upward rank. This ensures stolen work contributes to unblocking
   the critical path, not just keeping the worker busy.

3. **Horizontal filling that maximizes vertical surface**: stolen work advances
   other lanes' chains, which may unblock the originally blocked lane's
   dependency sooner.

### Comparison to Cilk/Rayon

Cilk and Rayon use randomized work-stealing: when a worker's deque is empty,
it steals from a random victim. The stolen unit is whatever sits at the bottom
of the victim's deque (oldest spawned task).

loimu's approach differs in three ways:

| Aspect | Cilk/Rayon | loimu |
|--------|-----------|-------|
| When to steal | Deque empty | Blocked on dependency |
| What to steal | Random victim, bottom of deque | Precomputed overflow list, highest-rank candidate |
| Selection criterion | FIFO (oldest task) | Critical path weight (upstream_rank) |

Cilk/Rayon optimize for throughput on fork-join workloads. loimu optimizes for
latency on DAG workloads where the critical path determines the pass duration.
Picking the highest-rank stealable unit is the greedy heuristic that minimizes
critical path stalls.

Taskflow (Huang et al., IEEE TPDS 2021) is closer to loimu's model: each
worker maintains a local task queue and steals from others when idle. The
successor becomes ready when its atomic dependency counter reaches zero. Taskflow
does not use precomputed overflow lists or upward rank for steal priority, so
loimu's approach is more targeted but less general.

### No dedicated WASM lane

Plugin units compete for lane assignment like any other unit, based on upward
rank and access set disjointness. A WASM unit with disjoint access can run
alongside native units on separate lanes. Fuel metering is per-unit, not
per-behavior.

## 6. Per-unit metadata

Every scheduling unit carries precomputed metadata, stored compactly for cache
efficiency during dispatch:

```
lane:          u3      // primary worker assignment (0-7)
lane_compat:   u8      // bitmask: which lanes can safely run this unit
depth:         u16     // DAG depth (topological level from Kahn's)
upstream_rank: u32     // upward rank (critical path weight from leaves)
fusible:       bool    // true for `each` units; false for tree/bind/dispatch/emit
```

### lane (u3)

3 bits, values 0 through 7. Maximum 8 worker lanes. The lane is assigned
during plan computation by the list scheduling algorithm (highest-rank-first
assignment to earliest-finishing lane). A typical desktop app uses 4-8 lanes
matching available cores.

### lane_compat (u8)

Bitmask indicating which lanes can safely execute this unit without AccessSet
conflicts. Precomputed at plan time. Used at runtime only during overflow
stealing (section 5). A unit with `lane_compat = 0xFF` can be stolen by any
lane; a unit with `lane_compat = 0x01` can only run on lane 0.

### depth (u16)

Topological depth from Kahn's algorithm. Units at the same depth have no edges
between them and can run in parallel (within a step). Maximum depth of 65535
levels, which is far beyond any practical DAG.

### upstream_rank (u32)

The upward rank computed bottom-up from the DAG. Higher values indicate units
closer to the critical path's root. Used for lane assignment priority, steal
priority during overflow, and tiebreaking. Maximum value of ~4 billion, more
than sufficient for any realistic schedule.

### fusible (bool)

Marks `each` units as candidates for fused iteration by the storage engine.
The storage engine groups fusible units touching the same view and compatible
column sets, then fuses their per-element operations into a single iteration
pass. Non-fusible units (`tree`, `bind`, `dispatch`, `emit`) execute directly
on their assigned lane.

### Additional per-unit metadata

Beyond the precomputed fields, the scheduler tracks per-unit runtime state:

- `sole_writer: bool` per `(unit_id, resource_id)` pair, for fault rollback
  strategy selection (see section 11).
- Duration history (EMA of actual execution times) for cost estimation updates.
- Downstream fan-out score for tiebreaking (see section 10).

## 7. Version stamps and phantom write detection

### The problem

A unit may declare write access to a resource but not actually modify it on a
given pass (conditional logic, early return, no relevant data). Without
detection, all downstream units that depend on that write edge must wait for the
unit to complete, even though no data changed. In UI workloads, most units skip
their writes on most passes.

### Mechanism

Every resource carries a `version: u64` field. The increment is inside the
generated write accessor, transparent and unconditional:

```
resource.version: u64   // incremented by write accessor, not user code
```

The lane executor captures before-versions and compares after the unit returns:

```
1. before = [R1.version, R2.version]       // lane-local reads, same cache line
2. run unit U                              // exclusive write access guaranteed by DAG
3. after = [R1.version, R2.version]        // still lane-local, no sync
4. for each resource where before == after:
     mark that write edge as "phantom"
5. for each unit T waiting on U:
     if ALL of T's remaining pending deps are phantom:
         early-release T
```

### Why this works without new synchronization

The DAG guarantees exclusive write access during a unit's execution. The version
read before dispatch and the comparison after return both happen on the same
lane. No atomics, no locks, no memory fences for the version check itself.

The "notify dependents" path already exists: when a unit completes, the lane
executor decrements readiness counters for dependent units (atomic, since
dependents may be on other lanes). The phantom flag piggybacks on this existing
notification. No new synchronization primitives are introduced.

### Why no explicit behavior opt-in

An earlier design considered a generic `Outcome` type on the Behavior trait,
where behaviors could return `NoChanges` explicitly. This was rejected because:

- Version stamps catch the same information automatically, with zero cognitive
  load on behavior authors.
- The `Outcome` approach adds API surface (new trait, new associated type, new
  macro syntax) for no additional benefit.
- The only case `Outcome` catches that version stamps miss is "wrote the same
  value back" -- not worth the complexity.

The version counter is a pure scheduler implementation detail. Behaviors declare
write access via `write name: Type;` in handler bodies as before; nothing
changes in the behavior API.

### Overhead analysis

| Operation | Cost | When |
|-----------|------|------|
| Version increment | One `u64 += 1` (non-atomic; DAG guarantees exclusive access) | On every resource write (same cache line as data) |
| Before-capture | One `u64` read per declared-write resource | Before dispatching unit (lane-local) |
| After-comparison | One `u64 ==` per declared-write resource | After unit returns (lane-local) |
| Phantom propagation | One extra bit in existing atomic counter decrement | On completion notification (existing path) |

Typical unit declares 1-3 write resources. Total added cost per unit execution:
2-6 `u64` operations, all lane-local, no contention. The atomic decrement in
the notification path happens regardless; the phantom flag is free.

### What this enables

In UI workloads, version stamps collapse large portions of the DAG at runtime
without recomputing the plan. The precomputed plan assumes worst-case latencies
(all writes happen), but actual execution skips waits when operations complete
with no side effects. This is analogous to early-release in instruction
scheduling: the hardware plan assumes worst-case pipeline latencies, but
forwarding and bypass networks release dependent instructions early when the
data arrives sooner.

## 8. Rotation groups and prediction-based skipping

### Concept

The scheduler partitions prediction-tolerant behaviors into N rotation groups.
On each pass, 1/N groups are active. The remaining (N-1)/N groups use predicted
values from the temporal system instead of executing.

### Eligibility

A behavior is rotation-eligible if ALL fields it accesses are temporal.
Non-temporal field access disqualifies the behavior entirely. This is a strict
binary check, not a percentage threshold.

### Effective tolerance

The effective tolerance is the minimum tolerance of all accessed temporal fields
across all units in the behavior.

```
effective_tolerance = min(tolerance(field) for field in behavior.accessed_fields)
```

A behavior touching fields with tolerance 4 and tolerance 8 has effective
tolerance 4. The weakest link determines the group.

### Adaptive tolerance

Compensates for varying schedule intervals:

```
effective_tolerance = declared_tolerance * (baseline_interval / current_interval)
```

EMA-smoothed to avoid jitter. If the schedule runs faster than baseline,
effective tolerance decreases (tighter prediction window -- predictions don't
have time to diverge). If slower, tolerance increases (predictions must cover
a larger gap).

### Prediction-quality-based skipping

The scheduler tracks prediction error per behavior (EMA of |predicted -
actual|). Behaviors whose predicted values consistently match actual values
(low error EMA) are candidates for skipping even outside their rotation slot.

This is a scheduler heuristic, not a guarantee. The scheduler may decide to
run a behavior that could be skipped, but it never skips a behavior whose
prediction error is above threshold.

### Edge constraint reduction

Only 1/N of rotating behaviors have active data dependencies per pass. Inactive
behaviors' edges are temporarily removed from the DAG. This reduces the critical
path length and increases available parallelism for the active subset.

Consider a DAG with 30 behaviors, 20 of which are rotation-eligible in 4
groups. On any given pass, only 5 of the 20 are active. The DAG effectively
shrinks from 30 nodes to 15 nodes (10 always-active + 5 rotating-active),
with proportionally fewer edges. The critical path may shorten significantly
if rotating behaviors were on it.

### Concrete example

```
N = 4 rotation groups
20 rotation-eligible behaviors
5 active per pass

Pass 0: groups {0}    active -> behaviors 0-4
Pass 1: groups {1}    active -> behaviors 5-9
Pass 2: groups {2}    active -> behaviors 10-14
Pass 3: groups {3}    active -> behaviors 15-19
Pass 4: groups {0}    active -> behaviors 0-4   (cycle repeats)
```

Each behavior executes once every 4 passes. Between executions, downstream
consumers read predicted values from the temporal system. Prediction error
determines whether the predicted values are good enough; if error grows too
large, the scheduler may force the behavior to run out-of-turn.

## 9. Per-stage lane profiles

### Motivation

Different app stages (Loading, Playing, Paused, Editor) activate different
subsets of behaviors. A stage change should not require full DAG recomputation
from scratch.

### LaneProfile structure

Each `AppStage` has a precomputed `LaneProfile`:

```
LaneProfile {
    active_units: IdSet<UnitId>,
    lane_assignments: Dictionary<UnitId, LaneId>,
    upward_ranks: Dictionary<UnitId, u32>,
}
```

At registration time (and on `RecalculateSchedulePlan`), the scheduler computes
a LaneProfile for every declared stage. Behaviors declare which stages they
participate in via `stages()` in their descriptor. Behaviors without explicit
stage declarations are active in all stages. Lane assignments are per-unit, not
per-behavior.

### O(1) stage transitions

At pass boundary, when the scheduler detects a stage change:

1. Look up the new stage's precomputed LaneProfile.
2. Swap the active profile (pointer swap, no reallocation).
3. Units not in the new profile's `active_units` are skipped.
4. Lane assignments from the new profile are used immediately.

Total transition cost: one pointer swap. No DAG rebuild, no topological sort,
no rank recomputation.

### When to recompute

Profiles are recomputed when:
- A plugin is loaded/unloaded (adds/removes units)
- `RecalculateSchedulePlan` is emitted
- Stage definitions change (rare; typically only at startup)

Recomputation happens between passes. The scheduler continues using the old
profile until the new one is ready.

### Pre-sorted unit lists

Each LaneProfile stores pre-sorted unit lists with tiebreakers already
resolved. The runtime dispatch loop does not sort or compare -- it follows the
precomputed list in order. This pushes all comparison work to plan time.

### Per-stage storage layout (speculative)

An additional speculative optimization: view-managed column reordering per
stage for cache locality. For each stage, analyze access sets of active units.
Views and columns accessed together should be placed in adjacent memory.

```
LayoutPlan {
    column_order: Collection<ColumnId>,
    hot_set: IdSet<ViewId>,
}
```

The loimu docs mark this as speculative: measure cache miss rates first before
pursuing. Column reordering must not race with unit access (swap at pass
boundary only). SlabColumn addresses are stable within a pass.

## 10. Tiebreaker heuristics

When two or more units have the same upward rank and their dependencies are
both satisfied, the scheduler needs a deterministic tiebreaker. The choice
affects parallelism.

### Primary: downstream fan-out

Prefer the unit whose completion unblocks the most immediately-ready downstream
units. "Immediately-ready" means all of the downstream unit's dependencies are
satisfied once this unit completes.

```
tiebreak_score(unit) = count of units T where:
    T depends on unit, AND
    all other dependencies of T are already completed
```

This maximizes the parallelism surface: completing a high-fan-out unit creates
the most opportunities for the lane pool to stay busy.

### Secondary: estimated duration

If fan-out scores are equal, prefer the shorter unit. This is the shortest-job-
first heuristic: finishing short units quickly frees lanes for longer units,
reducing overall makespan.

Duration estimates come from `DurationTracker` (EMA of recent execution
durations). Units with no history use a default estimate.

### Tertiary: deterministic ID ordering

If both fan-out and duration are equal, fall back to `UnitId` ordering. This
ensures deterministic execution order across runs, which aids reproducibility
and debugging.

### Combined comparator

```rust
fn compare_tiebreak(a: &Unit, b: &Unit) -> Ordering {
    b.downstream_fan_out.cmp(&a.downstream_fan_out)
        .then(a.estimated_duration.cmp(&b.estimated_duration))
        .then(a.unit_id.cmp(&b.unit_id))
}
```

Fan-out scores and duration estimates are precomputed at plan time, stored in
the LaneProfile, and updated when the plan is recomputed.

## 11. Fault handling and rollback

### Fault triggers

- Native panic: caught via `catch_unwind`.
- WASM fuel exhaustion: per-unit fuel allocation, `PluginFuelExhausted` emitted.
- WASM out-of-bounds trap.

Native and WASM behaviors use the same fault handling path.

### Sole-writer detection

During plan computation, the scheduler determines for each (unit, resource)
pair whether the unit is the sole writer to that resource in the current DAG.
A unit is a sole writer if no other unit in the plan declares write access to
the same resource. This is a static property of the plan:

```
sole_writer: bool   // per (unit_id, resource_id) pair
```

### Rollback strategies

The rollback strategy is selected at plan time based on sole-writer analysis:

| Case | Strategy | Normal-path cost |
|------|----------|-----------------|
| Sole writer | Restore dirty ranges from snapshot buffer `b` | Zero |
| Shared writer | Replay undo journal in `b.page0` (reverse order) | ~24 bytes per field write |

Both strategies use existing buffer memory (no separate allocation).

**Sole-writer rollback**: if the unit is the only writer to a resource, the
original data is still in the snapshot buffer. On fault, copy dirty ranges back
from the snapshot. Normal-path cost is zero because the snapshot exists
regardless (it is the double-buffer mechanism).

**Shared-writer rollback**: if multiple units write the same resource, the
snapshot buffer may contain writes from a previous unit. An undo journal in
`b.page0` records each field write's previous value. On fault, replay the
journal in reverse order to restore pre-fault state. Cost: ~24 bytes per field
write (field ID + old value).

### Fault execution flow

On fault:
1. Roll back writes using the appropriate strategy per resource.
2. Emit `BehaviorFaulted` signal with the unit ID and fault reason.
3. Disable remaining units in the faulted handler for this pass.
4. Continue executing the rest of the DAG. Downstream units that depend on the
   faulted unit are skipped (their dependency counter never reaches zero).

The scheduler does not abort the pass. Non-dependent units continue normally.
The faulted behavior is isolated to its subgraph.

### WASM fuel metering

Fuel metering is per-unit, not per-behavior. Each scheduling unit gets its own
fuel allocation. If fuel exhausts, the snapshot-then-flush model ensures no
partial writes commit. The scheduler emits `PluginFuelExhausted` and continues
normally.

## 12. Deferred execution models

loimu's docs reference three execution models considered but deferred to later
work.

### Naughty Dog fiber-based execution

Source: Gyrling, GDC 2015.

The engine replaces threads with fibers. Each fiber runs a task; when it
encounters a dependency, it yields (saves stack to a wait list). When the
dependency completes, any waiting fibers are resumed.

Key differences from work-stealing:
- Fibers can suspend mid-task (threads cannot without OS support).
- Wait lists avoid spin-waiting or polling.
- Stack save/restore is cheap (~64KB fiber stacks).

This model is attractive for behaviors that do I/O or wait on GPU fences.
Deferred to post-1.0 because fibers are platform-specific and complex to
implement correctly (signal handling, TLS, debugger support).

### Work-stealing wavefront execution

Source: Taskflow (Huang et al., IEEE TPDS 2021).

Instead of step-based execution with barriers between levels, each task has an
atomic dependency counter. Tasks become ready the instant their counter reaches
zero. No step barriers. Maximum parallelism at the cost of more complex
synchronization.

```
for each worker thread:
    loop:
        task = local_queue.pop() or steal_from(other_worker)
        execute(task)
        for successor in task.successors:
            if successor.dependency_counter.decrement() == 0:
                local_queue.push(successor)
```

Tradeoffs vs step-based:
- Pro: maximum parallelism, no barrier waste.
- Con: non-deterministic order (harder to reproduce bugs), cache-unfriendly
  stealing, more complex implementation.

The step model works for loimu's typical workloads (dozens to hundreds of
behaviors). If profiling shows barrier waste (steps with one slow node holding
back many fast nodes), the graph and plan APIs do not change -- only
`execute_plan` gets an alternative implementation.

### GPU work graphs

Source: D3D12 Work Graphs (official release March 2024).

Nodes are GPU entry points that can dynamically spawn successor nodes. The GPU
scheduler handles dispatch, synchronization, and memory. For GPU-heavy workloads
(animation blending, render passes), loimu could eventually lower its plan to
a D3D12 work graph rather than executing steps on the CPU thread pool.

The three-phase pattern (graph, plan, execute) remains; only the executor
changes. This is the hardware evolution of the software pattern loimu-graph
implements.

## 13. Relevance to hilavitkutin

hilavitkutin is the shared pipeline execution engine for polka-dots, saalis,
and loimu. This section maps loimu's scheduler design to hilavitkutin's
requirements, identifying what to adopt directly, what to simplify, and what
to add.

### What to adopt

**DAG scheduling with upward rank.** All three projects build dependency graphs
from read/write declarations and execute in topological order. The list
scheduling algorithm with upward rank is general-purpose and benefits all
three. The plan computation steps (Kahn's toposort, bottom-up rank, lane
assignment) are project-agnostic.

**Lane model with work-stealing.** The lane-based dispatch model with overflow
lists is directly applicable. The `lane_compat` bitmask and `upstream_rank`
priority for steal selection are cheap to compute and effective at reducing
critical path stalls. The maximum of 8 lanes (u3) is reasonable for all three
projects' workloads.

**Per-unit metadata.** The compact metadata structure (`lane`, `lane_compat`,
`depth`, `upstream_rank`, `fusible`) is directly usable. hilavitkutin can adopt
this verbatim.

**Version stamps and phantom write detection.** The mechanism is
project-agnostic. Any DAG scheduler benefits from early-release when units
don't modify their declared-write resources. The zero-overhead design (no new
synchronization, piggybacks on existing notification) makes it a clear win.

**Tiebreaker heuristics.** The three-level tiebreaker (fan-out, duration, ID)
is general-purpose. hilavitkutin can adopt this directly.

**Fault handling with rollback.** The sole-writer/shared-writer distinction
and the two rollback strategies are applicable to any DAG executor. The
`BehaviorFaulted` signal becomes a general fault notification.

### What to simplify

**Signal bus.** loimu's signal bus is loimu-specific. polka-dots runs once and
exits; saalis uses event-driven edges but not a full signal bus. hilavitkutin
should expose virtual columns (zero-sized marker types as DAG edge sources)
instead of a signal bus. Consumers that need a bus (loimu) can build one on top
using virtual columns as the edge mechanism.

**Seven access modes.** loimu has Pinned, Immediate, MutImmediate, Sample,
Post, Approx, Defer. polka and saalis need Read and Write. hilavitkutin
should expose Read and Write as the base set. loimu maps its seven modes to
these two for DAG edge purposes and adds mode semantics in its own layer.

**Rotation groups and temporal prediction.** These are loimu-specific
optimizations tied to the temporal system and UI update patterns. hilavitkutin
does not need them. loimu can implement rotation groups as a layer on top of
hilavitkutin's group/filter mechanism.

**Per-stage lane profiles.** The AppStage concept is loimu-specific. However,
the mechanism (precomputed profiles with O(1) swap) is general. hilavitkutin
should expose a generic "profile" concept: a precomputed subset of the DAG
with lane assignments. Consumers define what triggers a profile switch.

**CadenceMap and policies.** Cadence is needed by saalis (Curator: 1h,
Doctor: 5min) and loimu (Adaptive, OnDemand) but not polka. hilavitkutin
should expose a timer/wake interface that consumers can use to implement
policies. The Continuous/Adaptive/OnDemand types live in the consumer layer.

### What to add: yielding

loimu's scheduler has NO yielding. Behaviors are atomic: they run to completion
per pass. The Naughty Dog fiber model is deferred to post-1.0.

hilavitkutin needs yielding from day one because polka and saalis have
long-running and I/O-bound work units:

- polka: template compilation can be slow; morsel processing may need to yield
  at chunk boundaries for fairness.
- saalis: HTTP requests and DB queries block for arbitrary durations; the
  scheduler must not stall all lanes waiting for one I/O call.

The topic document proposes a `Progress` enum:

```
Complete                           -- done, results written
Yield(BudgetExhausted)             -- ran out of time, resume next quantum
Yield(Pending)                     -- waiting on I/O, park until woken
Yield(Retry { attempt, max, backoff }) -- failed, retry with backoff
Failed(Error)                      -- permanent failure
```

This interacts with the DAG: a yielded unit has not written its output yet.
Downstream units cannot run. The scheduler must track partial completion and
resume the yielded unit before releasing dependents. This is conceptually
similar to the fiber model (suspend mid-task, resume when ready) but
implemented without actual fibers -- the work unit returns `Yield` and the
scheduler re-queues it.

Key design questions for yielding in hilavitkutin:

1. **DAG interaction**: does a yielded unit hold its write locks? If so,
   downstream units are blocked until resume. If not, the unit must re-acquire
   on resume, which may violate ordering guarantees.

2. **Partial writes**: can a yielded unit have written some but not all of its
   outputs? If so, the scheduler needs partial-write tracking (which outputs
   are available). If not, the unit must buffer internally until `Complete`.

3. **Budget enforcement**: should the scheduler enforce a time budget and
   preempt (like WASM fuel metering), or trust work units to yield
   cooperatively? Cooperative is simpler but risks starvation.

4. **Integration with version stamps**: a yielded unit that has not written
   anything yet should not trigger phantom detection. The before-capture must
   happen once (at first dispatch), not on each resume.

loimu's decision to defer yielding was pragmatic for a UI framework where
behaviors are short-lived (~microseconds per unit). hilavitkutin serves data
pipeline workloads where individual units may run for seconds. Yielding is not
optional; it is a core scheduling primitive.
