## 1. Algorithm Overview

SIEVE is a cache eviction algorithm that achieves state-of-the-art miss ratios on web workloads while being structurally simpler than LRU. It was designed as a "turn-key" replacement: drop it into any system currently using LRU or CLOCK and get equal-or-better efficiency with superior throughput. The authors demonstrated that integrating SIEVE into five production cache libraries required fewer than 20 lines of code changes on average.

The algorithm is built on two properties that the authors identify as essential for efficient web cache eviction:

- **Lazy promotion** -- defer the decision about whether to keep an object until eviction time, rather than eagerly promoting it on every access.
- **Quick demotion** -- remove newly inserted objects rapidly if they are not re-accessed, because most objects in web workloads are "one-hit wonders" that are never requested again before eviction.

SIEVE is the simplest known algorithm that effectively achieves both properties simultaneously.

## 2. Data Structure and Mechanism

### 2.1 Components

SIEVE requires exactly three things:

1. **A single FIFO queue** (doubly-linked list) that maintains insertion order.
2. **One bit per entry** ("visited") tracking whether the object has been accessed since last considered for eviction.
3. **One pointer** ("hand") pointing to the current eviction candidate, which sweeps from tail toward head.

### 2.2 Operations

**Cache hit:** Set the visited bit of the accessed object to `1`. If it is already `1`, no operation is needed. Critically, SIEVE does *not* move the object within the queue on a hit. This is the "lazy promotion" -- the object stays in its original position, and whether it survives eviction depends on the visited bit state when the hand reaches it.

**Cache miss (insertion):** Insert the new object at the *head* of the queue with its visited bit set to `0`.

**Eviction (when the cache is full):** Starting from the hand's current position:
1. Examine the object the hand points to.
2. If `visited == 1`: reset the bit to `0`, advance the hand toward the head (retain the object *in its current position*).
3. If `visited == 0`: evict this object, set the hand to the next object.
4. If the hand reaches the head, wrap around to the tail.

### 2.3 Pseudocode

```
fn cache_hit(obj):
    obj.visited = true          // single atomic store; no list mutation

fn evict() -> evicted_obj:
    while true:
        candidate = hand.current()
        if candidate.visited:
            candidate.visited = false
            hand.advance_toward_head()
        else:
            hand.set(candidate.next)
            remove(candidate)
            return candidate

fn cache_miss(key, value):
    if cache.is_full():
        evict()
    obj = Node { key, value, visited: false }
    queue.insert_at_head(obj)
```

### 2.4 The Critical Difference from CLOCK / FIFO-Reinsertion

SIEVE is superficially similar to CLOCK (Second Chance / FIFO-Reinsertion), but there is one crucial structural difference: **where retained objects live after surviving eviction**.

| Algorithm | Retained object placement | Effect |
|---|---|---|
| CLOCK / FIFO-Reinsertion | Reinserted at the *head*, mixed with newly inserted objects | Old survivors and new arrivals compete in the same region |
| SIEVE | Stays at its *original position* in the queue | Old survivors and new objects are spatially separated |

This separation is what gives SIEVE its efficiency advantage. By keeping retained objects in place, SIEVE creates an implicit age-stratified structure: the tail region contains old, tested survivors, while the head region contains new, unproven arrivals. The hand naturally sweeps through the new arrivals first (since it moves from tail to head, then wraps), giving them a quick chance to prove themselves or be evicted. Meanwhile, long-surviving popular objects near the tail accumulate visited bits between hand sweeps, insulating them from eviction.

CLOCK/FIFO-Reinsertion, despite using the same visited-bit mechanism, only reduces FIFO's miss ratio by ~15% on average. SIEVE reduces it by ~21% on average, and by more than 42% on the top 10% of traces.

## 3. Why SIEVE Outperforms ARC on Web Workloads

### 3.1 Web Workload Characteristics

Web cache workloads (CDN, key-value stores, object caches) have distinct properties compared to block cache workloads:

- **Zipfian popularity distribution:** A small fraction of objects account for most requests, with a long tail of rarely-accessed objects.
- **High one-hit-wonder ratio:** The majority of objects are accessed exactly once before eviction. On production traces from Meta, Twitter, Wikipedia, and Tencent, one-hit wonders frequently exceed 50% of unique objects.
- **No sequential scans:** Unlike block caches (where sequential reads of address ranges are common), web caches see independent, popularity-skewed requests.

### 3.2 ARC's Overhead

ARC (Adaptive Replacement Cache) maintains four LRU lists: two for recently-accessed and frequently-accessed data, plus two ghost lists tracking recently evicted entries. It adaptively shifts its balance between recency and frequency. This gives ARC strong scan resistance and adaptability, but:

- **Four LRU lists means four sets of pointer mutations on every access.** Each cache hit requires moving an entry between lists, which involves acquiring locks and mutating linked-list pointers.
- **Ghost lists consume metadata memory** for entries that are not even cached.
- **The adaptivity is tuned for block workloads** where scan patterns are common. On Zipfian web workloads, the frequency-vs-recency balance ARC maintains is less useful.

### 3.3 SIEVE's Advantage

On the 1,559 traces evaluated in the paper (from 7 sources: Meta KV, Meta CDN, Twitter, Wikipedia/Wikimedia, Alibaba, Tencent Photo, and proprietary CDN datasets totaling 247 billion requests to 14.8 billion unique objects):

- SIEVE reduces ARC's miss ratio by **up to 63.2%**, with a **mean reduction of 1.5%** across all traces.
- SIEVE achieves this while being dramatically simpler and faster.

The mean of 1.5% may seem modest, but the distribution is highly skewed in SIEVE's favor: SIEVE matches or beats ARC on the vast majority of traces, and dramatically outperforms it on certain workloads while rarely being much worse. For a "turn-key" algorithm with no tuning parameters, this is remarkable.

The reason is straightforward: on Zipfian workloads with high one-hit-wonder ratios, SIEVE's quick demotion aggressively purges the long tail of unpopular objects, while lazy promotion retains the popular head with minimal overhead. ARC's complex bookkeeping provides marginal benefit on these distributions.

## 4. Scan Resistance

SIEVE's scan resistance is nuanced:

- **Against "traditional" scans** (a sequential sweep larger than the cache): SIEVE is scan-resistant. The scanned objects enter at the head with `visited = 0` and are quickly evicted by the hand before they can displace established popular objects near the tail.
- **Against "interleaved" scans** (scan objects mixed with popular objects in a block cache pattern): SIEVE is *not* scan-resistant in this sense. Popular objects and scan objects can intermingle positionally, and the hand may clear popular objects' visited bits before they can be re-accessed.

However, this limitation is largely irrelevant for web caches, where interleaved sequential scans essentially do not occur. The access patterns are independent key lookups following a Zipfian distribution, not sequential address sweeps. For saalis's use case (metadata lookups, catalog queries), this is the right tradeoff.

## 5. Memory Overhead Per Entry

### 5.1 Algorithmic Overhead

SIEVE requires per entry:

- **1 bit** for the visited flag
- **2 pointers** for the doubly-linked list (prev/next) -- same as LRU
- **0 additional pointers** beyond the single global hand pointer

Compare to:
| Algorithm | Per-entry metadata | Global state |
|---|---|---|
| LRU | 2 pointers (16 bytes on 64-bit) | 1 head pointer |
| CLOCK | 1 bit + position in circular buffer | 1 hand pointer |
| ARC | 2 pointers + ghost-list membership flag | 4 list heads + adaptive parameter |
| S3-FIFO | 2 pointers + frequency counter (2 bits) | 3 queue heads + ghost queue metadata |
| **SIEVE** | **2 pointers + 1 bit** | **1 hand pointer** |

### 5.2 Practical Overhead

In a typical Rust implementation (e.g., the `sieve-cache` crate), each cache entry is a node in a doubly-linked list:

```
struct Node<K, V> {
    key: K,
    value: V,
    prev: Option<NonNull<Node<K, V>>>,  // 8 bytes
    next: Option<NonNull<Node<K, V>>>,  // 8 bytes
    visited: bool,                       // 1 byte (padded)
}
```

The linked-list overhead is **16 bytes per entry** (two pointers), identical to LRU. The visited bit adds 1 byte in practice (due to alignment), though a bitfield packing scheme could reduce this. The total per-entry overhead is **17 bytes** in a naive implementation, effectively the same as LRU's 16 bytes.

The real savings come from *runtime* overhead: SIEVE does not mutate the list on cache hits, only on eviction. This eliminates the pointer-chasing and lock contention that makes LRU expensive under concurrency.

## 6. Throughput and Scalability Benchmarks

From the NSDI '24 paper, measured on a Meta production trace:

| Threads | SIEVE (Mops) | Optimized LRU (Mops) | TwoQ (Mops) | Speedup vs LRU |
|---|---|---|---|---|
| 1 | ~16% faster than LRU | baseline | baseline | 1.16x |
| 16 | >2x LRU throughput | baseline | ~same as LRU | >2.0x |

At 16 threads, SIEVE achieves **more than twice the throughput** of an optimized LRU implementation. This is because:

1. **No list mutation on hit:** A cache hit only sets a boolean, which can be a relaxed atomic store. No linked-list surgery, no lock acquisition.
2. **Eviction is localized:** Only the eviction path (which is infrequent relative to hits) mutates the list, and it does so in a single sweep direction, which is cache-line friendly.
3. **No ghost lists:** Unlike ARC or S3-FIFO, there is no auxiliary data structure to maintain.

## 7. Existing Rust Implementations

### 7.1 `sieve-cache` crate (jedisct1)

- **Repository:** [jedisct1/rust-sieve-cache](https://github.com/jedisct1/rust-sieve-cache)
- **Crates.io:** [sieve-cache](https://crates.io/crates/sieve-cache) -- 96,645+ downloads
- **API:** Drop-in compatible with `clock-pro` and `arc-cache` crates.
- **Variants:**
  - `SieveCache` -- single-threaded, highest performance
  - `SyncSieveCache` -- thread-safe via internal locking
  - `ShardedSieveCache` -- sharded for high-concurrency workloads (recommended: 16-32 shards)
  - `WeightedSieveCache` (and weighted sync/sharded variants) -- memory-bounded eviction via a `Weigh` trait, evicting based on entry weight rather than count
- **Features:** `sync`, `sharded`, `weighted` (all enabled by default)

### 7.2 Other Rust Caches Worth Noting

- **moka** ([moka-rs/moka](https://github.com/moka-rs/moka)): High-performance concurrent cache using TinyLFU (LRU eviction + LFU admission). More complex, but battle-tested. Does not use SIEVE.
- **quick_cache** ([quick_cache](https://crates.io/crates/quick_cache)): Optimized for low-overhead use cases. Minimal feature set.
- **foyer** ([foyer.rs](https://foyer.rs)): Hybrid memory+disk cache. Uses its own eviction policy. Outperforms moka in benchmarks.

For saalis, the `sieve-cache` crate's `WeightedSieveCache` variant is directly relevant: it can evict based on the *weight* (size) of entries rather than just count, which maps naturally to column-segment eviction where segments vary in size.

## 8. Application to Columnar Hot Store

### 8.1 The Mapping

In saalis's architecture, the columnar hot store caches frequently-accessed column segments in memory. The eviction unit is not individual key-value pairs but **column segments** -- contiguous chunks of a particular column's data for a range of entities.

SIEVE maps to this as follows:

| SIEVE concept | Columnar hot store equivalent |
|---|---|
| Cache entry | Column segment (e.g., `title` column, rows 0-1023) |
| Key | `(ColumnId, SegmentRange)` composite key |
| Value | The segment data (`Vec<u8>` or typed array) |
| Visited bit | "Was this segment accessed since last eviction sweep?" |
| Weight (in weighted mode) | Segment size in bytes |
| Cache capacity | Total hot store memory budget |

### 8.2 Why SIEVE Fits

Column segment access patterns in a library manager closely resemble web cache patterns:

1. **Zipfian popularity:** A small set of "hot" columns (title, platform, status, thumbnail_url) dominate queries, while the long tail (obscure metadata fields, rarely-viewed columns) is accessed infrequently. Similarly, recently-added or recently-browsed entity ranges are accessed much more than archival ranges.

2. **High one-hit-wonder ratio:** Many column segments will be loaded once for a query or a UI render pass and never touched again -- e.g., when a user scrolls through a catalog page, segments for the visible rows are loaded but the user moves on. Quick demotion handles this naturally.

3. **No sequential scans:** Saalis queries are entity-driven (fetch metadata for a set of entity IDs) or column-driven (fetch a column for a filter/sort), not sequential sweeps through all segments. SIEVE's weaker scan resistance is irrelevant.

4. **Variable-size entries:** Column segments vary widely in size. A `title` segment (strings) might be 10x larger than a `year` segment (u16 array) for the same row range. The `WeightedSieveCache` variant handles this natively by evicting based on byte weight rather than entry count, ensuring the memory budget is respected.

5. **Read-heavy with infrequent writes:** The hot store is overwhelmingly read-heavy (UI rendering, search, filtering), with writes happening asynchronously through the scheduler's fire-and-forget path. SIEVE's zero-cost-on-hit property is ideal: setting a visited bit is a single atomic store, no lock contention, no pointer mutation.

### 8.3 Integration Sketch

```rust
use sieve_cache::{WeightedSieveCache, Weigh};

#[derive(Clone, Hash, Eq, PartialEq)]
struct SegmentKey {
    column_id: ColumnId,
    range: SegmentRange,
}

struct ColumnSegment {
    data: Vec<u8>,
    // ... typed accessor, compression info, etc.
}

impl Weigh for ColumnSegment {
    fn weight(&self) -> u64 {
        self.data.len() as u64
    }
}

// Memory budget: e.g. 128 MiB for hot store on Batocera appliance
const HOT_STORE_BUDGET: u64 = 128 * 1024 * 1024;

let hot_store: WeightedSieveCache<SegmentKey, ColumnSegment> =
    WeightedSieveCache::with_max_weight(HOT_STORE_BUDGET);
```

### 8.4 Considerations and Tradeoffs

**Segment granularity matters.** If segments are too large (e.g., an entire column for all entities), eviction is coarse-grained and the cache cannot adapt to partial popularity. If segments are too small (per-row), the per-entry overhead (17 bytes of linked-list pointers + visited bit) becomes significant relative to the data. A reasonable segment size for saalis would be 1K-4K rows per segment, yielding segments in the 4 KB to 64 KB range depending on column type.

**No built-in prefetching.** SIEVE is a reactive eviction policy; it does not prefetch. For column segments where spatial locality is strong (querying row range N likely means range N+1 is needed soon), a prefetching layer above the cache would complement SIEVE well.

**Single global hand vs. per-column hands.** A single SIEVE instance treats all columns equally. If certain columns should be prioritized (e.g., always keep `entity_type` and `status` hot), this is better handled by pinning those segments outside the eviction pool rather than trying to bias SIEVE.

**Thread safety.** If the hot store is accessed from both the sync core and the async web layer, `ShardedSieveCache` with 16 shards would provide good concurrency without a global lock. The sharded variant partitions keys across independent SIEVE instances, each with its own hand and lock.

## 9. Comparison Summary

| Property | LRU | ARC | CLOCK | S3-FIFO | SIEVE |
|---|---|---|---|---|---|
| Miss ratio (web) | Baseline | ~6% better | ~same | ~better | ~21% better than FIFO; matches/beats ARC |
| Throughput (16 threads) | Baseline | Worse (4 lists) | Good | Good | >2x LRU |
| Per-entry overhead | 16 B | 16 B + ghost | 1 bit* | 16 B + 2 bits | 16 B + 1 bit |
| Implementation complexity | Simple | Complex | Simple | Moderate (3 queues) | Simpler than LRU |
| Scan resistance | None | Strong | Weak | Strong | Sufficient for web |
| Hit-path cost | List mutation | List mutation | Bit set | Bit set | Bit set |
| Tuning parameters | 0 | 0 (adaptive) | 0 | 1 (small queue %) | 0 |

*CLOCK typically uses a circular buffer rather than a linked list, so the pointer overhead depends on implementation.

## 10. Conclusion and Recommendation

SIEVE is an excellent fit for saalis's columnar hot store eviction policy:

1. **Algorithmically sound** for the access patterns we expect (Zipfian, high one-hit-wonder ratio, no sequential scans).
2. **Zero hit-path overhead** aligns with our sync-core design where the hot store must be fast without async machinery.
3. **Weight-based eviction** available out of the box via the `sieve-cache` crate's `WeightedSieveCache`, handling variable-size column segments naturally.
4. **Production-validated** at Meta and adopted across the industry; winner of the NSDI '24 Community Award.
5. **Trivially simple** to implement or integrate -- the `sieve-cache` crate is a direct dependency option, or a from-scratch implementation is ~50 lines of Rust.

The primary alternative worth considering is S3-FIFO (from the same research group, published at SOSP '23), which offers stronger scan resistance at the cost of more complexity (three queues, a ghost queue, a tuning parameter for small-queue size). For saalis's workload profile, the added complexity of S3-FIFO is not justified.
