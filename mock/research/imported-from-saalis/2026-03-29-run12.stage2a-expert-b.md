## Disagreement Statement

I disagree with the Savant's claim that a two-phase feed/poll model
with a single `poll_output()` drain is sufficient for verkko's
protocol layer. The Savant asserts that collapsing transmit outputs
and application events into one `poll_output()` stream prevents
reentrancy structurally. This is true, but the cost is that the
consumer must branch on every output to determine whether to call
`sendto()` or dispatch to the mesh layer. In a protocol that
produces 10,000+ output events per second (heartbeats, SACKs,
data frames, session events), that branch is in the critical path
and it conflates two fundamentally different output domains: I/O
operations (send bytes to the network) and application callbacks
(notify the mesh layer of state changes). The quinn project
separated these for good reason, and I will defend that separation.

Outside my domain -- no comment on CRDT merge semantics or

However, Expert A's proposed replication orchestration pseudocode

mesh layer receives events from the protocol state machine
(heartbeat data, session state changes) and makes replication
decisions. These decisions produce new protocol inputs (stream
creation for transfer sessions, data frames for transfer
requests). The control flow is:

```
protocol.poll_event() -> HeartbeatReceived
  -> mesh layer detects filter divergence
  -> mesh layer decides to replicate
  -> protocol.handle(StreamCreated { ... })
  -> protocol.handle(FrameData { ... })
```

This is not reentrancy (the mesh layer does not call handle()
from within poll_event()), but it is a producer-consumer loop
within a single event loop iteration. My four-method model
handles this cleanly: the consumer drains all poll_event()
outputs, collects mesh-layer decisions, then feeds them back via
handle() in the next sub-iteration. The Savant's two-phase model
handles it similarly (drain all poll_output(), then feed new
inputs). No conflict here.

This classification is the consumer's responsibility. The protocol
trait does not and should not distinguish connection types. This is
the correct sans-IO boundary: the protocol handles bytes, crypto,
streams, SACK. The mesh layer handles topology, health, replication.

## Part 2: Responses to Expert C (NAT Traversal)

### C.Problem 1: hop_count breaks AEAD

Expert C identifies a genuine correctness bug. The fix (exclude
hop_count from AEAD AD) is the right one. From my domain, the
impact on the protocol state machine is:

The AEAD associated data construction in the protocol state machine
must be changed from "outer_header[0..29]" to
"outer_header[0..8] || outer_header[9..29]" (28 bytes). This is a
one-line change in the AEAD computation path inside handle(). The
state machine's internal AEAD handler needs a `build_aead_ad()`
function that skips byte 8. The frame parser already has the outer
header as a contiguous slice; the AEAD function receives a
non-contiguous AD (two slices concatenated). ChaCha20-Poly1305
supports multi-part AD via incremental authentication. No arena
allocation needed for this; the AD can be fed to the AEAD in two
parts.

Concrete state machine impact:

```rust
fn build_aead_ad(outer_header: &[u8; 29]) -> [&[u8]; 2] {
    [&outer_header[0..8], &outer_header[9..29]]
}
```

This is straightforward. I agree with Expert C's Option A.

### C.Problem 2: Transit relay forwarding body

Expert C specifies a forwarding table and per-frame procedure.
From my domain, the key question is whether the relay peer runs
the same Protocol trait or a separate interface.

Expert C's forwarding procedure is:

1. Parse outer header (29 bytes). No AEAD needed.
2. TTL check.
3. Loop detection.
4. Lookup forwarding table.
5. Decrement hop_count.
6. Forward.

This is NOT a Protocol trait operation. The relay does not decrypt,
does not parse the inner payload, does not maintain stream state,
does not generate SACKs. The relay is a frame forwarder that
operates on the outer header only. The Protocol trait handles
encrypted connection state. The relay operates below the Protocol
trait's abstraction level.

I recommend a separate trait for the relay's forwarding function:

```rust
/// Relay forwarding state machine. Separate from Protocol.
/// No AEAD, no stream state, no arena allocation.
/// All operations are O(1) per frame.
trait RelayForwarder {
    /// Process one incoming frame.
    /// Returns the forwarding decision.
    /// Does NOT borrow from an arena; the forwarding table
    /// is self-contained (caller-owned fixed-size buffer).
    fn forward(
        &mut self,
        frame: &[u8],
        source: PeerAddr,
    ) -> ForwardAction;

    /// Process a SIGNAL message.
    /// Modifies the forwarding table.
    fn handle_signal(
        &mut self,
        signal: &SignalMessage,
        source: PeerAddr,
    ) -> SignalResponse;

    /// Get the next forwarding table entry expiry deadline.
    fn poll_expiry(&self) -> Option<u64>;

    /// Expire stale forwarding table entries.
    fn expire(&mut self, now_ms: u64);
}

enum ForwardAction {
    /// Forward the frame to this address.
    /// The frame is modified in-place (hop_count decremented).
    Send { destination: PeerAddr },
    /// Process locally (destination is this relay's cluster).
    ProcessLocally,
    /// Drop the frame (TTL exhausted or no route).
    Drop,
}
```

The relay peer runs BOTH the Protocol trait (for its own Noise
sessions with mesh peers) and the RelayForwarder trait (for
forwarding other peers' frames). The consumer (relay event loop)
first checks if a received datagram is addressed to the relay
itself (destination matches relay's cluster_destination_id); if
so, it feeds it to the Protocol state machine. If not, it feeds
it to the RelayForwarder.

Buffer ownership for RelayForwarder: the frame is caller-owned.
The forwarder reads the outer header, modifies hop_count in-place,
and returns a ForwardAction with the destination address. No arena
allocation. The consumer then calls sendto() with the modified
frame. Total allocation: zero. This is consistent with Expert C's
~300ns per-frame cost estimate.

### C.Problem 3: NAT traversal

The NAT traversal state machine Expert C specifies (IDLE ->
GATHERING -> CHECKING -> CONNECTED | RELAY) has direct protocol
layer implications. The connectivity checks use PATH_CHALLENGE
(msg_type 0x06) and PATH_RESPONSE (msg_type 0x07), which are
already part of the protocol wire format.

From the sans-IO perspective, the NAT traversal state machine is
a mesh-layer concern that produces protocol-layer inputs:

```
// Mesh layer (NAT traversal):
for (local, remote) in candidate_pairs {
    protocol.handle(ProtocolInput::ConnectionRequested {
        destination: remote.addr,
    })
    // ... wait for PathChallengeReceived event ...
}
```

The protocol state machine does not need to know about NAT
traversal. It sends PATH_CHALLENGE frames when instructed via
handle() and delivers PATH_RESPONSE events via poll_event(). The
NAT state machine is a consumer of the Protocol trait, not part of
it. This is the correct sans-IO separation.

Expert C's relay-first recommendation (Tension 2) affects the
state machine indirectly. See Tension 2 analysis below.

### C.Problem 4: Relay failover

Expert C proposes a warm standby relay with promotion on primary
failure. From my domain, this means the relay event loop must
drive two RelayForwarder instances (primary and secondary). The
secondary holds dormant forwarding entries. On primary failure,
the consumer promotes the secondary by activating its entries and
sending SIGNAL REDIRECT through the secondary.

This is a consumer-level (mesh-layer) orchestration concern. The
Protocol trait and RelayForwarder trait do not change. The state
machine does not know about relay failover. The consumer maintains
a relay selection state machine that interacts with the Protocol
trait (for SIGNAL messages) and the RelayForwarder trait (for
forwarding table management).

The failover timing (2-30 seconds) is driven by heartbeat
timeouts, which the Protocol trait exposes via poll_event()
(HeartbeatReceived with connection_id). The mesh layer tracks
missed heartbeats and triggers failover. No protocol-layer change
needed.

### C.Problem 5: Seat failover

Outside my domain -- no comment beyond the sans-IO interaction
(seats use the same Protocol trait, which is already specified).

### C.Problem 6: Multi-hop relay

Expert C recommends v1 supports 1-hop relay only. I agree from a
protocol complexity perspective. Multi-hop relay requires no

regardless of how many hops they traversed). The RelayForwarder

Multi-hop adds complexity only to the mesh layer's relay selection

### Savant Problem 3: The SansIO Amorphous Surface

The Savant proposes a three-method polling model:

```rust
trait VerkkoProtocol {
    fn feed(&mut self, input: ProtocolInput) -> usize;
    fn poll_output(&mut self) -> Option<ProtocolOutput>;
    fn next_deadline(&self) -> Option<Instant>;
}
```

I disagree with this design on three specific points, and I will
defend each in detail.

**Point 1: Collapsing transmit and event outputs is wrong.**

The Savant's poll_output() returns both SendBytes (which the
consumer must pass to sendto()) and application events (which the
consumer must dispatch to the mesh layer). The consumer must branch
on every output:

```rust
while let Some(output) = protocol.poll_output() {
    match output {
        ProtocolOutput::SendBytes { dest, data } => {
            socket.sendto(dest, data);
        }
        ProtocolOutput::HeartbeatReceived { .. } => {
            mesh.handle_heartbeat(..);
        }
        ProtocolOutput::SessionEstablished { .. } => {
            mesh.handle_session(..);
        }
        // ... 10 more variants ...
    }
}
```

Compare with the four-method model:

```rust
while let Some(tx) = protocol.poll_transmit(&arena) {
    socket.sendto(tx.destination, tx.payload);
}
while let Some(event) = protocol.poll_event(&arena) {
    mesh.handle_event(event);
}
```

The four-method model has two advantages:

1. **Transmit-first ordering.** Draining all transmissions before
   processing application events ensures that outgoing frames
   (heartbeats, SACKs, data) are enqueued for the OS socket
   before the mesh layer processes events that might generate
   more protocol inputs. This reduces tail latency for heartbeats:
   the heartbeat frame is handed to the kernel before the mesh
   layer's potentially expensive event processing begins.

2. **Type-level output separation.** The consumer of poll_transmit()
   knows it always gets a (destination, payload) pair. The consumer
   of poll_event() knows it always gets an application event. No
   match statement needed at the drain point. The compile-time
   guarantee that poll_transmit() never returns an application event
   eliminates an entire class of consumer bugs.

The Savant's counterargument is that the unified poll_output()
prevents reentrancy structurally. I address this in Point 3.

**Point 2: Arena lifetime semantics differ, and mine are better.**

The Savant says: "SendBytes data is a slice into the hot arena.
The slice is valid until the next call to feed()."

My model says: "Transmit::payload borrows from the arena. Valid
until arena.reset()."

The difference: the Savant's model invalidates arena slices on
every feed() call. My model invalidates them on explicit
arena.reset(). The Savant's model is simpler for the consumer
(no explicit reset), but it is less flexible:

Consider the case where the consumer receives 32 datagrams via
recvmmsg and needs to process them all before sending any
responses. Under the Savant's model, each feed() call invalidates
the previous output slices. The consumer must copy each
poll_output() result to a separate buffer before calling feed()
again. Under my model, the consumer feeds all 32 datagrams
(each handle() call borrows from the same arena), then drains all
transmissions (all Transmit structs borrow from the same arena),
then sends them all via sendmmsg, then resets the arena.

The recvmmsg/sendmmsg batch pattern is critical for performance on
Linux. The io_uring submit pattern is even more critical. Both
require the ability to hold multiple output buffers simultaneously.
The Savant's model forces per-output copies. My model enables
zero-copy batching.

Let me be concrete about the performance difference. At 10,000
frames per second with a 1232-byte frame size:

- Savant's model: 10,000 memcpy operations per second, each
  copying up to 1232 bytes = ~12.3 MB/s of memory bandwidth for
  copies alone. On a Raspberry Pi 4's memory bus (~4 GB/s), this
  is 0.3% of memory bandwidth. Not catastrophic, but unnecessary.

- My model: zero copies on the transmit path. The arena slice is
  passed directly to sendmmsg's iovec. Total: 0 bytes of copy
  overhead.

At the design envelope (36 peers, 11 inter-cluster connections),
the frame rate is much lower (~1,000 fps). The performance
difference is negligible at this scale. But the design documents
specify UDP as the transport, and sendmmsg/recvmmsg batching is
the standard pattern. The arena-epoch model supports it; the
per-feed invalidation model does not.

**Point 3: Reentrancy prevention is achievable in both models.**

The Savant claims that the two-phase feed/poll model "structurally
prevents" reentrancy because "poll_output() does not call back into
the consumer." My four-method model also prevents reentrancy: all
methods take `&mut self`, which means the Rust borrow checker
prevents calling handle() while holding a reference returned by
poll_transmit() or poll_event().

The reentrancy scenario the Savant is guarding against:

```rust
// WRONG: calling handle() from within poll_event() processing
while let Some(event) = protocol.poll_event(&arena) {
    if event.is_interesting() {
        // This would require &mut self, but we already
        // borrowed &mut self from poll_event().
        // Rust prevents this at compile time.
        protocol.handle(response_input, &arena); // ERROR
    }
}
```

Rust's borrow checker prevents this in BOTH models. The consumer
cannot call handle() while draining poll_event() because both
require `&mut self`. Similarly, the consumer cannot call feed()
while draining poll_output() in the Savant's model.

The Savant's structural prevention argument applies to languages
without ownership/borrowing (Python, C). In Rust, the borrow
checker provides the same guarantee for both models. The Savant's
advantage in this area does not exist in Rust.

**My revised position on the polling model.**

I maintain the four-method model with one revision. The Savant's
`next_deadline(&self) -> Option<Instant>` is cleaner than my
`poll_timeout() -> Option<TimerDeadline>` because the consumer
does not need timer_id -- the consumer only needs the deadline.
When the deadline elapses, the consumer calls
`handle_timeout(now_ms, &arena)` and the protocol processes all
expired timers internally.

Revised trait:

```rust
trait Protocol {
    fn handle<'a>(
        &mut self,
        input: ProtocolInput<'a>,
        arena: &'a ArenaRef<'a>,
    ) -> Result<(), ProtocolError>;

    fn poll_transmit<'a>(
        &mut self,
        arena: &'a ArenaRef<'a>,
    ) -> Option<Transmit<'a>>;

    fn poll_event<'a>(
        &mut self,
        arena: &'a ArenaRef<'a>,
    ) -> Option<Event<'a>>;

    fn next_deadline(&self) -> Option<Instant>;

    fn handle_timeout<'a>(
        &mut self,
        now_ms: u64,
        arena: &'a ArenaRef<'a>,
    ) -> Result<(), ProtocolError>;
}
```

This preserves the four-method output separation (transmit vs
event vs deadline) while adopting the Savant's cleaner deadline
query signature.

## Part 4: Tension Analysis

### Tension 1: SansIO Polling Convention (4-method vs 2-phase)

**My position: four-method model, with Savant's deadline
signature adopted.**

I have defended this at length in Part 3 above. The core
arguments:

1. Transmit/event separation provides type-level output
   classification.
2. Arena-epoch invalidation (my model) supports sendmmsg/io_uring
   batching; per-feed invalidation (Savant's model) forces copies.
3. Reentrancy prevention is provided by Rust's borrow checker in
   both models, so the Savant's structural advantage does not exist
   in Rust.
4. Transmit-first drain ordering reduces heartbeat tail latency.

The Savant's `next_deadline(&self) -> Option<Instant>` is
cleaner than my `poll_timeout() -> Option<TimerDeadline>`. I
adopt it. The timer_id was unnecessary in the external interface
because the consumer never needs to identify which timer expired;
it just calls handle_timeout(now_ms) and the protocol handles all
expired timers internally.

The Savant's `feed() -> usize` returning the number of pending
outputs is a nice ergonomic touch. I adopt it for handle():

```rust
fn handle<'a>(
    &mut self,
    input: ProtocolInput<'a>,
    arena: &'a ArenaRef<'a>,
) -> Result<usize, ProtocolError>;
```

The usize return tells the consumer how many outputs are pending
(transmits + events). If zero, the consumer can skip the drain
loops. This is a minor optimization but useful for idle
connections that receive heartbeats and produce no outputs.

**Revised calling convention:**

```rust
loop {
    // 1. Receive datagrams (batch)
    let datagrams = socket.recvmmsg(batch_size: 32);
    for dgram in datagrams {
        let pending = protocol.handle(
            ProtocolInput::BytesReceived { source, data: dgram },
            &arena,
        )?;
        // pending > 0 means outputs are waiting
    }

    // 2. Process expired timers
    if let Some(deadline) = protocol.next_deadline() {
        if now >= deadline {
            protocol.handle_timeout(now_ms, &arena)?;
        }
    }

    // 3. Drain transmissions (send to network)
    while let Some(tx) = protocol.poll_transmit(&arena) {
        iovecs.push(iovec { base: tx.payload, dest: tx.destination });
    }
    socket.sendmmsg(&iovecs);  // batch send

    // 4. Drain application events (dispatch to mesh)
    while let Some(event) = protocol.poll_event(&arena) {
        mesh.handle_event(event);
    }

    // 5. Reset arena for next iteration
    arena.reset();
}
```

Key property: all Transmit slices in step 3 borrow from the
same arena. They remain valid until arena.reset() in step 5. The
sendmmsg call in step 3 can scatter-gather from the arena without
copies. This is not possible with the Savant's per-feed
invalidation.

### Tension 2: Relay-First vs Hole-Punch-First NAT Traversal

From the protocol state machine perspective, relay-first and
hole-punch-first produce different state machine topologies:

**Hole-punch-first (current design):**
```
ConnectionRequested
  -> GATHERING (collect candidates)
  -> CHECKING (PATH_CHALLENGE each pair)
  -> CONNECTED (direct path)  OR  RELAY (relay fallback)
```

The state machine starts with candidate checking and falls back
to relay. The connectivity check timer (2 seconds per candidate
pair, up to 8 pairs) adds latency before any data can flow.

**Relay-first (Expert C's recommendation):**
```
ConnectionRequested
  -> RELAY (establish relay path immediately)
  -> data flows through relay
  -> CHECKING (background PATH_CHALLENGE)
  -> CONNECTED (migrate to direct if hole punch succeeds)
```

The state machine starts with relay and upgrades in the background.
Data flows immediately (1 RTT to relay, ~200-500ms).

From a state machine complexity perspective, relay-first is
SIMPLER. The hole-punch-first model has a blocking CHECKING state
where no data flows (the connection is not yet established). The
relay-first model has no blocking state; the connection is
established through the relay and data flows immediately. The
background checking is non-blocking and produces a connection
migration event if it succeeds.

Connection migration (relay -> direct) is already modeled in the
protocol state machine via PATH_CHALLENGE/PATH_RESPONSE. QUIC
uses the same mechanism for connection migration between network
paths. The protocol state machine's connection state includes
`current_path: PeerAddr` which changes on successful migration.
The consumer sees updated Transmit destinations (from relay
address to direct address) and the migration is transparent.

The reverse migration (direct -> relay, when a direct path
degrades) is also handled: the consumer detects path degradation
(via missed heartbeats or SACK loss ratio) and establishes a relay
path. The connection migrates to the relay path via
PATH_CHALLENGE through the relay. The protocol state machine
handles this identically to forward migration.

**My position: I support relay-first.** It produces a simpler
state machine (no blocking CHECKING state), lower initial latency
(data flows in 1 RTT), and the connection migration mechanism
already exists in the protocol for the upgrade path. The cost
(transient relay load) is bounded by the mesh size and mitigated
by background hole punching.

The protocol state machine changes for relay-first are minimal:

1. Remove the blocking CHECKING state from the connection
   establishment path.
2. Add a non-blocking BackgroundChecking state that runs
   concurrent with data flow.
3. Emit a PathUpgraded event when background checking succeeds
   and the connection migrates to direct.

These are internal state machine changes. The Protocol trait
interface does not change. The consumer feeds ConnectionRequested
and receives SessionEstablished (via relay) followed optionally
by PathUpgraded (when direct path is established).

### Tension 3: Gateway Handoff -- Teardown vs Migration

Expert A says gateway handoff tears down intra-cluster sessions
and re-establishes them. The Savant says connection_id is preserved
across relay migration (for relay failover).

From the protocol state machine perspective, these are two
different operations:

**Connection teardown (Expert A's model):**
```
Protocol emits ConnectionClosing { reason: GATEWAY_HANDOFF }
Consumer closes the connection.
New connection established via ConnectionRequested.
Noise KK handshake produces new session keys.
New connection_id, new nonce counter, clean state.
```

**Connection migration (Savant's model):**
```
Protocol receives PATH_CHALLENGE on new path.
Protocol verifies PATH_RESPONSE.
Protocol updates current_path.
Same connection_id, same session keys, same nonce counter.
```

These have different implications:

1. **Teardown** guarantees clean state but loses in-flight data.
   The VitalQueue is flushed. Any reliable-ordered stream with
   unacked frames loses those frames. The retransmission timer
   fires for the old connection, but the connection is closed, so
   the retransmissions are dropped. The peer must re-request any
   data that was in-flight.

2. **Migration** preserves in-flight data but carries state from
   the old path. The nonce counter continues, the SACK state
   continues, the retransmission buffers continue. But the new
   gateway must take ownership of the existing session state, which
   requires the old gateway to transfer the Noise session state
   (transport keys, nonce counters, SACK state) to the new gateway.
   This transfer itself must be reliable (it uses the intra-cluster
   session per Expert A's model).

The protocol state machine supports both operations. Teardown is
simpler (no state transfer). Migration is safer for in-flight data
but more complex (state transfer, potential for state corruption if
the transfer is interrupted).

**My position: teardown for gateway handoff, migration for relay
failover.**

Gateway handoff is a rare event (gateway changes only on gateway
failure or explicit admin action). The cost of re-establishing
sessions is acceptable. The simplicity of clean teardown avoids the
state transfer complexity and the risk of corrupted state.

Relay failover is more frequent (relay peers are household devices
that reboot). Connection migration preserves in-flight data and
avoids the reconnection latency. The relay does not hold session
state (it holds only a forwarding table), so the migration is
simpler: the protocol state machine changes its path from
old_relay_addr to new_relay_addr via PATH_CHALLENGE through the
new relay.

This is consistent with Expert A's model (teardown for gateway
handoff) and the Savant's model (migration for relay failover).
They are not contradictory; they describe different events with
different characteristics.

The protocol state machine needs both:

```rust
enum ConnectionEvent {
    /// Clean teardown. Connection is closed.
    /// New connection must be established separately.
    Close { reason: CloseReason },

    /// Path migration. Connection survives.
    /// Same connection_id, same session keys.
    PathMigrated { new_path: PeerAddr },
}
```

Both are already modeled in my Stage 1 proposal (ConnectionClosing
event and implicit path migration via updated Transmit
destinations).

### Tension 4: Replication Orchestration

Outside my domain -- no comment on probabilistic vs priority-queue
replication.

One protocol-layer observation: the Savant's `replication_budget = 0`
in Active/Stress mode is a mesh-layer policy decision that the
protocol state machine does not enforce. The protocol will happily
accept StreamCreated and FrameData inputs for replication streams
regardless of metabolic state. The metabolic state affects the
protocol's internal parameters (heartbeat interval, LEDBAT target)
but not the consumer's ability to create streams.

This means Expert A's proposal to use the Conductor for replication
aggressiveness (rather than halting replication entirely) is
compatible with the protocol layer. The protocol state machine does
not care whether a stream is "replication" or "user-requested
transfer." The distinction is a mesh-layer concern.

### Tension 5: Relay Trust Model (Full Peer vs Pseudo-Peer)

The Savant proposes a pseudo-peer relay (DK/DEK, no CIK, limited
capabilities). Expert C proposes a full Noise KK peer (the relay
IS a mesh peer with DK/DEK/CIK).

From the protocol state machine perspective, both models use the
same Protocol trait. The relay peer establishes Noise sessions
with other mesh peers. The Protocol trait does not know whether the
remote peer is a relay or a full peer. The consumer (mesh layer)
classifies connections.

**My position: pseudo-peer (Savant's model).**

Reason: the Protocol trait and RelayForwarder trait are separate
interfaces. The relay peer runs both. If the relay is a full peer
with CIK, it participates in inter-cluster sessions as a cluster
gateway, which is not its function. The relay's function is
forwarding, not participating in the mesh. A pseudo-peer with
DK/DEK (for Noise session establishment) but no CIK (no cluster
membership) correctly separates forwarding from mesh participation.

The sans-IO consequence: the relay peer's consumer (its event
loop driver) is simpler than a full peer's consumer. It does not
need to handle convergence events, gossip events, or dominance
computation. It processes Noise sessions (Protocol trait) and
frame forwarding (RelayForwarder trait). The consumer code is
approximately 40% of a full peer's consumer.

Expert C's amplification prevention rule ("MUST NOT forward to an
address from which it has not received a recent authenticated
frame within 30 seconds") creates a liveness hazard that the
Tension Map identifies. In Background mode, heartbeats are sent
every 30 seconds. At exactly the limit, the relay may drop the
first heartbeat after a 30-second gap. The fix is simple: the
relay's "recent authenticated frame" window should be 2x the
maximum heartbeat interval. For Background mode (30 seconds),
the window is 60 seconds. This eliminates the off-by-one hazard.

```rust
const RELAY_AUTH_WINDOW_MS: u64 = 60_000; // 2x max heartbeat interval
```

This is a constant in the RelayForwarder implementation, not a
Protocol trait change.

## Part 5: Updated Ideal Design

Based on the cross-review, I update my ideal sans-IO design with
the following changes from Stage 1:

### 5.1 Protocol Trait (revised)

```rust
trait Protocol {
    /// Process one input event. Returns pending output count.
    fn handle<'a>(
        &mut self,
        input: ProtocolInput<'a>,
        arena: &'a ArenaRef<'a>,
    ) -> Result<usize, ProtocolError>;

    /// Drain the next pending transmission (priority-ordered).
    fn poll_transmit<'a>(
        &mut self,
        arena: &'a ArenaRef<'a>,
    ) -> Option<Transmit<'a>>;

    /// Drain the next pending application event.
    fn poll_event<'a>(
        &mut self,
        arena: &'a ArenaRef<'a>,
    ) -> Option<Event<'a>>;

    /// Next timer deadline across all connections.
    fn next_deadline(&self) -> Option<Instant>;

    /// Process all expired timers.
    fn handle_timeout<'a>(
        &mut self,
        now_ms: u64,
        arena: &'a ArenaRef<'a>,
    ) -> Result<(), ProtocolError>;
}
```

Changes from Stage 1:
- `handle()` returns `Result<usize, ProtocolError>` (adopted from
  Savant's feed() return value).
- `poll_timeout()` renamed to `next_deadline()` (adopted from
  Savant's naming). Returns `Option<Instant>` instead of
  `Option<TimerDeadline>`.
- Timer ID removed from external interface. Internal timer wheel
  manages all IDs. Consumer only sees deadlines.

### 5.2 RelayForwarder Trait (new)

```rust
trait RelayForwarder {
    fn forward(
        &mut self,
        frame: &mut [u8],
        source: PeerAddr,
    ) -> ForwardAction;

    fn handle_signal(
        &mut self,
        signal: &SignalMessage,
        source: PeerAddr,
    ) -> SignalResponse;

    fn poll_expiry(&self) -> Option<u64>;

    fn expire(&mut self, now_ms: u64);
}
```

Buffer ownership: frame is caller-owned, modified in-place
(hop_count decrement). No arena allocation. The relay forwarding
path is zero-allocation.

### 5.3 Arena Budget (revised)

```
Hot arena:    64KB   (per-frame transients, reset per iteration)
Pool arena:  256KB   (retransmission buffers, assembly, verification)
Managed:     256KB   (connection tables, stream state, SACK tracking)
  - inter-cluster: ~176KB (11 connections * ~16KB)
  - intra-cluster:  ~50KB (5 connections * ~10KB)
  - protocol state:  ~30KB
──────────────────────────────
Total:       576KB
```

This is within the 25.6MB hard cap specified in the design
documents.

### 5.4 Connection State Machine (revised for relay-first)

```
ConnectionRequested
  |
  v
RELAY_ESTABLISHING (SIGNAL HELLO to relay, 1 RTT)
  |
  v
RELAY_HANDSHAKING (Noise KK through relay, 1 RTT)
  |
  v
ESTABLISHED_VIA_RELAY (data flows through relay)
  |
  +---> BACKGROUND_CHECKING (PATH_CHALLENGE to direct candidates)
  |       |
  |       +--- success ---> MIGRATING (PATH_CHALLENGE on direct path)
  |       |                   |
  |       |                   v
  |       |               ESTABLISHED_DIRECT (data flows direct)
  |       |
  |       +--- all checks failed ---> remain ESTABLISHED_VIA_RELAY
  |                                   (retry after 5 minutes)
  |
  +---> CLOSING (ConnectionClose received or sent)
          |
          v
        CLOSED (2*RTT drain, then destroyed)
```

The BACKGROUND_CHECKING state runs concurrently with
ESTABLISHED_VIA_RELAY. It does not block data flow. The migration
from relay to direct uses PATH_CHALLENGE/PATH_RESPONSE (already
in the wire format). The consumer sees a PathUpgraded event when
migration succeeds.

### 5.5 Timer Management (confirmed)

Internal timer wheel, 4 levels, 256 slots, 1ms resolution.
Timer IDs computed from (connection_index, timer_type, stream_id).
Maximum 64 active timers (bounded by connection count * timers-per-
connection).

New timer type for relay-first model:

```rust
#[repr(u8)]
enum ConnectionTimer {
    Heartbeat = 0,
    PathChallenge = 1,
    CpskRehandshake = 2,
    CloseTimeout = 3,
    DwellTimeout = 4,
    BackgroundCheck = 5,  // new: periodic hole-punch retry
}
```

### 5.6 Reliability Mode (confirmed)

All streams reliable-ordered. Heartbeats and SACKs are
connection-level unreliable (no stream header). No
UnreliableDatagram stream type. Memory cost: ~88KB for
retransmission state across all connections and streams, within
the 256KB pool arena.

### 5.7 Crypto Thread Interaction (confirmed)

CryptoRequest/CryptoResult variants in ProtocolInput/Output.
Synchronous AEAD (ChaCha20-Poly1305 at ~1us). Async Ed25519
and bao verification. The arena epoch must span at least one
crypto round-trip for pending CryptoRequests.

Additional constraint from cross-review: when a CryptoRequest
is pending, the arena MUST NOT be reset. The consumer must wait
for CryptoResult before calling arena.reset(). This adds a
constraint to the calling convention:

```rust
// After draining transmits and events:
if protocol.has_pending_crypto() {
    // Do NOT reset arena yet.
    // Route CryptoRequests to crypto thread.
    // Wait for CryptoResults.
    // Feed CryptoResults back to protocol.
    // Then reset arena.
}
arena.reset();
```

The `has_pending_crypto(&self) -> bool` method is new. It returns
true if any CryptoRequest outputs have been produced but not yet
answered with CryptoResult inputs.

### 5.8 AEAD Associated Data (revised per Expert C)

```rust
fn build_aead_ad(outer_header: &[u8; 29]) -> [&[u8]; 2] {
    // Exclude hop_count at offset 8
    [&outer_header[0..8], &outer_header[9..29]]
}
// Total AD: 28 bytes
// destination(4) + connection_id(4) + relay_key_id(8) + nonce(12)
```

### 5.9 MetabolicGovernor (confirmed with detail)

Threshold table: [0.20, 0.50, 0.80] in Q16.16.
Hysteresis: 0.05 in Q16.16.
Pure function, deterministic, ~150ns evaluation.
Input: ResourcePressures (5 Q16.16 fields).
Output: MetabolicState enum.

No floating-point anywhere in the governor path. All comparisons
are integer comparisons on u32 values.
