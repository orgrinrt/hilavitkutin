# Deep Dive: IGDB and ScreenScraper APIs

Research date: 2026-03-15

This document covers both APIs in depth -- authentication, query patterns, rate limits,
media handling, and how open-source projects use them -- to inform saalis's connector
architecture for game metadata enrichment.

---

## 1. IGDB (Internet Game Database) API v4

Official docs: https://api-docs.igdb.com/
API home: https://www.igdb.com/api
Free for non-commercial use under the Twitch Developer Service Agreement.

### 1.1 Authentication

IGDB is owned by Twitch (Amazon). All requests require Twitch OAuth **client credentials**
(server-to-server flow, no user login needed).

**Setup:**
1. Register an application at https://dev.twitch.tv/console
2. Set Client Type to **Confidential** (required to generate a client secret)
3. Obtain `client_id` and `client_secret`

**Token request:**
```
POST https://id.twitch.tv/oauth2/token
  ?client_id=YOUR_CLIENT_ID
  &client_secret=YOUR_CLIENT_SECRET
  &grant_type=client_credentials
```

Response:
```json
{
  "access_token": "abcdef123456",
  "expires_in": 5000000,
  "token_type": "bearer"
}
```

Tokens are long-lived (roughly 60 days) but should be refreshed before expiry. Every IGDB
request must include two headers:

```
Client-ID: YOUR_CLIENT_ID
Authorization: Bearer YOUR_ACCESS_TOKEN
```

Ref: https://dev.twitch.tv/docs/authentication/getting-tokens-oauth/

### 1.2 The Apicalypse Query Language

IGDB v4 uses **Apicalypse**, a custom query language sent as the POST body to each endpoint.
The base URL is `https://api.igdb.com/v4/{endpoint}`.

Core clauses:

| Clause     | Purpose                                   | Example                                         |
|------------|-------------------------------------------|-------------------------------------------------|
| `fields`   | Select which fields to return             | `fields name, cover.*, genres.name;`            |
| `where`    | Filter results                            | `where platforms = (48,49) & rating > 70;`      |
| `search`   | Full-text keyword search                  | `search "Sonic the Hedgehog";`                  |
| `sort`     | Order results                             | `sort rating desc;`                             |
| `limit`    | Max results (default 10, max 500)         | `limit 500;`                                    |
| `offset`   | Pagination offset                         | `offset 500;`                                   |
| `exclude`  | Omit specific fields                      | `exclude storyline;`                            |

**Operators in `where`:** `=`, `!=`, `>`, `>=`, `<`, `<=`, `~` (like/contains). Combine
with `&` (AND) and `|` (OR). Null check: `where cover != null;`. Set membership:
`where id = (8,9,11);`.

**Field expansion:** By default, related entities are returned as integer IDs. Using dot
notation or `.*` expands them to full objects:

```
fields name, cover.*, genres.name, platforms.abbreviation;
```

This replaces `cover: 12345` with `cover: { id: 12345, url: "//images.igdb.com/...", width: 264, height: 374 }`.

**Search endpoint:** `search` is available on Games, Characters, Collections, Platforms, and
Themes. It uses keyword matching (not fuzzy/Levenshtein), so partial queries like "starcr"
will not match "Starcraft". Client-side fuzzy matching is recommended for interactive search.

### 1.3 Rate Limits

| Constraint                | Value                            |
|---------------------------|----------------------------------|
| Requests per second       | **4**                            |
| Max open connections      | **8** simultaneous               |
| Max results per request   | **500** (free), 3000 (pro tier)  |
| Multi-query sub-queries   | **10** per request               |

Exceeding the rate limit returns HTTP **429 Too Many Requests**. There is no daily cap on
the free tier -- only the per-second throttle.

**Implication for saalis:** At 4 req/s with 500 results each and multi-query batching (10
sub-queries per request), theoretical throughput is **20,000 game records per second** for
ID-based lookups. In practice, field expansion and network latency reduce this, but enriching
1000+ games is easily achievable in under a minute.

### 1.4 Multi-Query Endpoint

The multi-query endpoint is the single most important feature for batch enrichment.

**URL:** `POST https://api.igdb.com/v4/multiquery`

**Syntax:**
```
query games "Halo Games" {
  fields name, cover.*, genres.name, platforms.name, summary;
  where id = (1,2,3,4,5);
  limit 10;
};

query games "Zelda Games" {
  fields name, cover.*, genres.name, platforms.name, summary;
  where id = (100,101,102,103);
  limit 10;
};

query platforms/count "Platform Count" {
};
```

Rules:
- Each sub-query has the form: `query {endpoint} "{result_name}" { ...body... };`
- Maximum **10** sub-queries per multi-query request
- Each sub-query can target a different endpoint (games, platforms, genres, etc.)
- Count queries use the `/count` suffix on the endpoint name
- The result name is user-defined and returned in the response for correlation
- Each sub-query counts as a **single request** against the rate limit (the whole
  multi-query POST counts as one request)

**Response structure:**
```json
[
  {
    "name": "Halo Games",
    "result": [ { "id": 1, "name": "Halo: Combat Evolved", ... } ],
    "count": 5
  },
  {
    "name": "Zelda Games",
    "result": [ { "id": 100, "name": "The Legend of Zelda", ... } ],
    "count": 4
  },
  {
    "name": "Platform Count",
    "count": 200
  }
]
```

### 1.5 Optimal Query Patterns for Batch Enrichment (1000+ Games)

**Strategy 1: Batch ID lookup via `where id = (...)`**

If you already have IGDB IDs (from a prior search pass or external mapping):

```
fields name, summary, storyline, cover.*, genres.name, platforms.name,
       release_dates.date, release_dates.platform.name,
       involved_companies.company.name, involved_companies.developer,
       involved_companies.publisher, age_ratings.*, total_rating,
       total_rating_count, alternative_names.name, external_games.*;
where id = (1,2,3,...,500);
limit 500;
```

At 500 IDs per request and 4 req/s, 1000 games = 2 requests = under 1 second.

**Strategy 2: Multi-query for heterogeneous data**

Use multi-query to fetch games + their related data in one round-trip:

```
query games "batch_1" {
  fields name, cover, genres, platforms, summary;
  where id = (1,2,3,...,500);
  limit 500;
};

query covers "batch_1_covers" {
  fields game, url, image_id, width, height;
  where game = (1,2,3,...,500);
  limit 500;
};
```

This retrieves both game metadata and cover URLs in a single HTTP request.

**Strategy 3: Incremental sync via `updated_at`**

For ongoing library maintenance, track the last sync timestamp and query only changes:

```
fields name, cover.*, updated_at;
where updated_at > 1700000000;
sort updated_at asc;
limit 500;
offset 0;
```

Page through with increasing offset until results are empty.

**Strategy 4: Name-based discovery**

When you only have ROM filenames, use the search endpoint to find candidates:

```
search "Super Mario World";
fields name, platforms.name, cover.*, first_release_date;
limit 10;
```

Then match by platform and year. Since IGDB search uses keyword matching (not fuzzy),
pre-process filenames: strip region tags `(USA)`, version numbers `(Rev 1)`, and file
extensions before searching.

### 1.6 Protobuf Support

IGDB supports Google Protocol Buffers as an alternative to JSON. Append `.pb` to any
endpoint URL:

```
POST https://api.igdb.com/v4/games.pb
```

The proto definition file is at `https://api.igdb.com/v4/igdbapi.proto`. Protobuf responses
are smaller and faster to deserialize -- meaningful for batch workloads. Note: enabling CORS
breaks protobuf responses (irrelevant for server-side usage in saalis).

### 1.7 Image URL Construction

IGDB images are hosted on Cloudinary. The URL pattern:

```
https://images.igdb.com/igdb/image/upload/t_{size}/{image_id}.{format}
```

**Size descriptors:**

| Name              | Dimensions  | Notes                    |
|-------------------|-------------|--------------------------|
| `micro`           | 35x35       | Tiny thumbnail           |
| `thumb`           | 90x90       | Square thumbnail         |
| `cover_small`     | 90x128      | Small cover art          |
| `cover_big`       | 264x374     | Standard cover           |
| `logo_med`        | 284x160     | Publisher/dev logo       |
| `screenshot_med`  | 569x320     | Medium screenshot        |
| `screenshot_big`  | 889x500     | Large screenshot         |
| `screenshot_huge` | 1280x720    | HD screenshot            |
| `720p`            | 1280x720    | Same as screenshot_huge  |
| `1080p`           | 1920x1080   | Full HD                  |

Append `_2x` to any size for Retina/DPR-2.0 quality (e.g., `cover_big_2x` = 528x748).

**Example:**
```
https://images.igdb.com/igdb/image/upload/t_cover_big/co1wyy.jpg
```

The `image_id` (e.g., `co1wyy`) comes from the `cover` or `screenshot` objects when you
expand them with `fields cover.*` or `fields screenshots.*`.

**Supported image types:** Cover, Screenshot, Artwork, CharacterMugShot, CompanyLogo,
GameEngineLogo, PlatformLogo, EventLogo.

### 1.8 External Game ID Mapping

The `external_games` endpoint maps IGDB IDs to external service IDs:

```
fields game, category, uid;
where game = (1,2,3);
```

Categories include Steam, GOG, Epic, PlayStation Store, Xbox Marketplace, Nintendo eShop,
and more. This is critical for cross-referencing: if saalis knows a Steam AppID, it can
resolve the IGDB game ID and vice versa.

You can also query `alternative_names` for localized or historical titles, which helps with
matching ROM filenames that may use Japanese or alternate naming conventions.

### 1.10 Webhooks

Webhooks push data to your endpoint when IGDB records are created, updated, or deleted.

**Availability:** Pro, Ultra, and Enterprise tiers only (not free tier).

**Registration:**
```
POST https://api.igdb.com/v4/games/webhooks/
Content-Type: application/x-www-form-urlencoded

url=https://your-server.com/webhook/igdb
&secret=your_webhook_secret
&method=update
```

`method` is one of: `create`, `update`, `delete`.

**Behaviour:**
- IGDB sends a POST to your URL with the changed entity in the body
- Your endpoint must return 200 OK within **15 seconds**
- After **5 consecutive failures**, the webhook is deactivated
- The `X-Secret` header contains your webhook secret for verification

**Management:**
- List webhooks: `GET /v4/games/webhooks/`
- Test: `POST /v4/games/webhooks/test/{webhook_id}?entityId={entity_id}`
- Delete: `DELETE /v4/webhooks/{webhook_id}`

**Relevance to saalis:** Webhooks are only on paid tiers. For the free tier, polling via
`where updated_at > {last_sync}` is the alternative. Given saalis targets single-user
appliance deployments, polling on a schedule (e.g., daily) is likely sufficient and avoids
needing a publicly-reachable endpoint.

## 2. ScreenScraper API v2

Website: https://www.screenscraper.fr/
API base: `https://www.screenscraper.fr/api2/`
Community-driven database focused on retro gaming media under Creative Commons.

### 2.1 Authentication

ScreenScraper uses a **two-layer** authentication model: developer credentials + user
credentials.

**Required parameters on every request:**

| Parameter     | Description                              |
|---------------|------------------------------------------|
| `devid`       | Developer ID (assigned to your software) |
| `devpassword` | Developer password                       |
| `softname`    | Your software's name and version         |
| `ssid`        | User's ScreenScraper username            |
| `sspassword`  | User's ScreenScraper password            |

**Obtaining developer credentials:** Register on screenscraper.fr, then request developer
API access through their forum. The process is described as straightforward -- submit your
project name and intended use.

**User registration:** Users register at https://www.screenscraper.fr/membreinscription.php.
A registered account is **required** -- the API is closed to anonymous/non-registered
requests during high-load periods, and non-registered users get lower quotas.

**Example request URL:**
```
https://www.screenscraper.fr/api2/jeuInfos.php
  ?devid=mydev
  &devpassword=mydevpass
  &softname=saalis1.0
  &ssid=username
  &sspassword=userpass
  &systemeid=1
  &romnom=Super%20Mario%20World.sfc
  &md5=abc123...
  &sha1=def456...
  &output=json
```

### 2.2 Game Lookup: `jeuInfos.php`

This is the primary endpoint. It identifies games by **ROM hash** or **filename**.

**Identification parameters (in priority order):**

1. **SHA1 hash** (`sha1`) -- most reliable
2. **MD5 hash** (`md5`) -- fallback
3. **CRC32** (`crc`) -- fallback
4. **ROM filename** (`romnom`) -- least reliable, used when hashes are unavailable
5. **System ID** (`systemeid`) -- narrows platform scope

The API attempts to match using hashes first (exact ROM identification), then falls back to
filename matching. This is ideal for ROM-based library managers where file hashes are readily
available.

**Response structure (JSON, abbreviated):**

```json
{
  "header": {
    "APIversion": "2",
    "dateTime": "2026-03-15 12:00:00",
    "commandRequested": "jeuInfos"
  },
  "response": {
    "jeu": {
      "id": "3",
      "noms": [
        { "region": "us", "text": "Super Mario World" },
        { "region": "jp", "text": "Super Mario World: Super Mario Bros. 4" }
      ],
      "synopsis": [
        { "langue": "en", "text": "Mario and Luigi travel to Dinosaur Land..." },
        { "langue": "fr", "text": "Mario et Luigi voyagent..." }
      ],
      "editeur": { "text": "Nintendo" },
      "developpeur": { "text": "Nintendo EAD" },
      "joueurs": { "text": "1-2" },
      "dates": [
        { "region": "us", "text": "1991-08-13" },
        { "region": "jp", "text": "1990-11-21" }
      ],
      "genres": [
        { "noms": [{ "langue": "en", "text": "Platform" }] }
      ],
      "medias": [
        {
          "type": "box-2D",
          "region": "us",
          "url": "https://www.screenscraper.fr/image.php?..."
        },
        {
          "type": "screenshot",
          "url": "https://www.screenscraper.fr/image.php?..."
        }
      ]
    }
  }
}
```

All textual data is multi-language (keyed by `langue`) and multi-region (keyed by `region`).

### 2.3 Available Media Types

ScreenScraper's media coverage is its primary strength, especially for retro content:

| Media Type       | Description                                    |
|------------------|------------------------------------------------|
| `screenshot`     | In-game screenshot                             |
| `fanart`         | Fan-made artwork                               |
| `box-2D`         | Flat box art scan                              |
| `box-3D`         | 3D rendered box art                            |
| `box-texture`    | Unfolded box texture (for 3D rendering)        |
| `wheel`          | Clear logo / wheel art (for overlay themes)    |
| `marquee`        | Arcade marquee art                             |
| `screenmarquee`  | Screen marquee                                 |
| `backcover`      | Back of box                                    |
| `manual`         | Scanned game manual (PDF or images)            |
| `video`          | Short gameplay video clip                      |
| `mixrbv1`        | Pre-composed mix image (Recalbox style v1)     |
| `mixrbv2`        | Pre-composed mix image (Recalbox style v2)     |
| `maps`           | Game maps                                      |
| `bezel`          | Bezel/overlay for RetroArch                    |

ScreenScraper has the best coverage of wheel, marquee, and box art among retro game
databases. It also provides videos and manuals for many titles.

Media URLs point to `screenscraper.fr/image.php` with authentication tokens embedded. Media
files must be **downloaded through the API** -- URLs are not directly hotlinkable without
credentials.

### 2.4 Other Endpoints

| Endpoint             | Purpose                                    |
|----------------------|--------------------------------------------|
| `systemesListe.php`  | List all supported platforms/systems        |
| `jeuInfos.php`       | Game info + media (primary endpoint)        |
| `userInfos.php`      | User account info, thread limits, quotas    |
| `ssuserInfos.php`    | Extended user info                          |
| `botNote.php`        | Game rating submission                      |

The `systemesListe` response includes system IDs, names, company, release date, ROM file
extensions, and supported ROM types. This is essential for mapping saalis's platform
identifiers to ScreenScraper's system IDs.

### 2.5 Rate Limits and Quotas

ScreenScraper's rate limiting is **tier-based** and tied to user contribution level:

| User Tier              | Threads | Daily Quota  | Notes                              |
|------------------------|---------|--------------|------------------------------------|
| Non-registered         | 0       | 0            | API closed during high load        |
| Registered (no contrib)| 1       | ~10,000      | Basic access                       |
| Active contributor     | 2-3     | ~20,000      | DB contributions earn threads      |
| Donor (Patreon/Tipeee) | 3-6+    | ~50,000      | Financial support = more threads   |

The API response includes `Maxthreadfornonmember` and `Maxthreadformember` fields that
indicate current server-side limits. Software should read these values and throttle
accordingly.

**Throttling requirements:**
- Minimum **1.2 seconds** between requests per thread (enforced by convention; Skyscraper
  uses this interval)
- The API may return errors or empty responses during peak usage periods
- Retry with exponential backoff is expected

**Implication for saalis:** With a registered account and 1 thread at 1.2s intervals, 1000
games takes roughly **20 minutes**. With donor status (3+ threads), that drops to ~7 minutes.
ScreenScraper is inherently slower than IGDB for bulk operations but provides richer media
(box scans, videos, manuals) that IGDB does not have.

### 2.6 Registration System

1. Register at https://www.screenscraper.fr/membreinscription.php
2. Account is free; provides basic API access
3. Thread count increases through:
   - **Database contributions:** Adding/correcting game data, uploading media
   - **Financial support:** Patreon (https://www.patreon.com/screenscraper) or Tipeee
     (https://en.tipeee.com/screenscraper)
4. Developer credentials are separate -- request through the forum after user registration
5. The `userInfos` endpoint returns current quota/thread allocation for the authenticated user

## 3. How Open-Source Projects Use These APIs

### 3.1 Playnite (C#, Windows game launcher)

- **Source:** https://github.com/JosefNemec/PlayniteExtensions
- IGDB is a built-in **metadata plugin** providing cover art, descriptions, genres, and
  platform data
- Implements the Twitch OAuth client credentials flow
- Uses the IGDB search endpoint for name-based matching when importing games
- Caches search results to reduce API calls (GitHub issue #185 discusses this)
- Migrated from IGDB v3 to v4 when unlimited requests became available
- Recently updated to use the new `collections` field (replacing deprecated `collection`)
- The IGDB plugin source was moved out of the main repo into PlayniteExtensions

### 3.2 GameVault (TypeScript/NestJS, self-hosted game server)

- **Source:** https://github.com/Phalcode/gamevault-backend
- Has an **official IGDB partnership** -- uses IGDB as the default metadata provider
- Implements a pluggable metadata provider architecture with priority-based merging
- Configuration via environment variables: `METADATA_IGDB_CLIENT_ID`,
  `METADATA_IGDB_CLIENT_SECRET`
- Fetches: name, cover, genres, platforms, age ratings, average playtime (Time To Beat),
  release dates, and game status
- Migrated from RAWG API to IGDB after RAWG became unreliable
- Designed the metadata system to be source-agnostic: each provider registers with a
  numerical priority, and higher-priority sources overwrite lower-priority data

### 3.3 Lutris (Python, Linux game manager)

- **Source:** https://github.com/lutris/lutris
- Uses IGDB primarily for **cover art** retrieval
- Game metadata (installers, compatibility info) comes from Lutris's own API
- IGDB IDs are used for cross-database linking (e.g., matching Lutris IDs to IGDB IDs)
- Feature requests exist for deeper IGDB integration (importing games from IGDB,
  downloading full metadata), but these are not yet implemented
- Uses SteamGridDB as an additional cover art source

### 3.4 Skyscraper (C++, ROM scraper for retro platforms)

- **Source:** https://github.com/muldjord/skyscraper
- The most mature ScreenScraper API client in the open-source ecosystem
- Implements **hash-based ROM identification** (SHA1, MD5, CRC) as the primary lookup method
- Falls back to filename matching when hashes fail
- Enforces a **1.2-second** minimum interval between API requests
- Implements retry logic with configurable max retries
- Reads thread limits from the API response and adjusts concurrency accordingly
- Supports multiple scraping sources: ScreenScraper, IGDB, MobyGames, OpenRetro, The
  Games DB, and local cache
- Uses a **local cache database** to avoid redundant API calls -- scrapes are cached
  and composed into final output separately from the scraping step
- Credentials stored in `~/.skyscraper/config.ini`

### 3.5 RetroArch / Libretro

- RetroArch does **not** directly integrate IGDB or ScreenScraper
- Thumbnails come from the libretro-thumbnails GitHub repository
  (https://github.com/libretro-thumbnails/libretro-thumbnails), which is periodically
  updated and served from https://thumbnails.libretro.com
- Box art, title screens, and in-game screenshots are matched by No-Intro DAT naming
  conventions
- Third-party tools (RetroBat, ARRM, Skyscraper) scrape from IGDB/ScreenScraper and
  produce gamelist XML files that RetroArch-adjacent frontends (EmulationStation, etc.)
  consume

### 3.6 ES-DE (EmulationStation Desktop Edition)

- Integrates ScreenScraper as a built-in scraping source
- Treats `wheel` and `wheel-hd` media types as equivalent
- Downloads box art, screenshots, wheels, marquees, and videos
- Respects ScreenScraper thread limits from the API response

## 4. Comparative Analysis for Saalis

| Dimension              | IGDB                              | ScreenScraper                      |
|------------------------|-----------------------------------|------------------------------------|
| **Focus**              | Modern + retro games              | Retro / emulation focused          |
| **Auth complexity**    | Low (Twitch OAuth, automated)     | Medium (dev + user credentials)    |
| **Rate limits**        | 4 req/s, no daily cap             | 1 thread default, daily cap        |
| **Bulk speed**         | 1000 games in < 1 min             | 1000 games in 7-20 min             |
| **Batch support**      | Multi-query (10 sub-queries)      | Single game per request            |
| **Text metadata**      | Excellent (summary, storyline)    | Good (multi-lang synopsis)         |
| **Cover art**          | Good (Cloudinary CDN)             | Excellent (box-2D, box-3D scans)   |
| **Screenshots**        | Good                              | Good                               |
| **Video**              | None                              | Gameplay clips                     |
| **Manuals**            | None                              | Scanned manuals                    |
| **Wheel/Marquee**      | None                              | Best in class                      |
| **ROM hash lookup**    | No                                | Yes (SHA1, MD5, CRC)               |
| **External ID mapping**| Yes (Steam, GOG, etc.)            | No                                 |
| **Webhooks**           | Yes (paid tiers only)             | No                                 |
| **Protobuf**           | Yes                               | No (JSON or XML only)              |
| **Rust client**        | igdb-rs crate (basic)             | None (build from scratch)          |

### Recommended Strategy for Saalis

1. **IGDB as primary metadata source:** Use for game identification (name search + external
   ID cross-referencing), text metadata (summary, storyline, genres, platforms), cover art,
   and ongoing sync via `updated_at` polling.

2. **ScreenScraper as media enrichment source:** Use for ROM-based identification (hash
   lookup), high-quality box art (2D/3D scans), wheel logos, marquee art, gameplay videos,
   and scanned manuals. Especially valuable for retro content where IGDB may have limited
   media.

3. **Two-pass enrichment pipeline:**
   - Pass 1 (fast): Batch-query IGDB via multi-query for all known games. Resolve IGDB IDs,
     pull text metadata and basic covers. At 4 req/s with 500 results each, 1000 games
     completes in seconds.
   - Pass 2 (slow, background): For games identified as retro/ROM-based, query ScreenScraper
     by hash for rich media. Queue as low-priority background work units. Respect the 1.2s
     throttle per thread.

4. **Cache aggressively:** Both APIs return stable data. Cache IGDB responses keyed by
   game ID + `updated_at` timestamp. Cache ScreenScraper media files locally after download
   (they won't change often). Skyscraper's two-phase approach (cache raw API responses, then
   compose output) is a proven pattern.

5. **Protobuf for IGDB:** Use `.pb` endpoints for batch enrichment workloads. Smaller
   payloads and faster deserialization. The proto file at
   `https://api.igdb.com/v4/igdbapi.proto` can generate Rust types via `prost`.

6. **Token management:** IGDB tokens last ~60 days. Store the token and expiry in settings.
   Refresh proactively (e.g., when remaining lifetime < 7 days). ScreenScraper credentials
   are static (username/password) -- store as `Credential` settings per the saalis
   architecture.
