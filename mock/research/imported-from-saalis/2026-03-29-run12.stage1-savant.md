## Part 1: Independent Crystallographic Assessment

### 1.1 The Lattice as It Stands

After eleven annealing cycles, the verkko crystal has developed well-defined
unit cell geometry in three of its six layers. The crypto layer (L0) is a
near-perfect crystal: every atom has a defined position, every bond has a
measured energy, every lattice constant is calibrated. The protocol layer
(L1) has crystallized its external faces (wire formats, frame budgets,
congestion control) but retains an amorphous core where the consumer
interface should be. The mesh layer (L2) has the most complex internal
structure in the lattice and the most complete specification of that
structure, but several bond sites remain unsaturated.

The seat layer (L2, parallel crystal) grew rapidly during Run 11. It
nucleated from defined concepts into a partially crystallized body with
authentication, consistency modes, and instruction protocols. It remains
a thin crystal, one unit cell deep in most areas.

The relay layer (L3) is the most amorphous region. Its defined concepts
(membrane, territorial re-encryption, constant-rate padding) form isolated
crystallites that have not merged into a continuous crystal. The body text
covers wire formats and re-encryption mechanics but transit forwarding,
the relay's primary function, has no crystalline structure.

The matter layer (L4) crystallized its content-addressing and transfer
protocols during Run 11. The chunk protocol, transfer lifecycle, and
integrity verification form a coherent lattice. But replication policy,
sync reconciliation, and garbage collection remain as interstitial voids.

The ops layer (L5) is the crystal's documentation and test plan. It does
not need crystallization in the same sense; it needs to accurately map the
crystal that exists.

### 1.2 Bond Energy Analysis

I measure bond energy by counting how many invariants constrain the
interface between two layers. High bond count = strong bond. Low bond
count = the layers can slide against each other during implementation,
producing dislocations.

| Bond | Invariant Count | Assessment |
|------|----------------|------------|
| crypto -> protocol | 12 (AEAD, nonces, ratchet, PCS) | Strong covalent |
| protocol -> mesh | 9 (heartbeat, metabolic, convergence) | Strong covalent |
| mesh -> relay | 3 (ConvergenceSignalProvider, ring coloring, re-enc) | Ionic, long-range |
| mesh -> matter | 4 (filter, source selection, convergence, sealed range) | Ionic |
| protocol -> seat | 5 (SansIO, session, transport, frame, stream) | Moderate covalent |
| crypto -> relay | 3 (StorageAEAD, deterministic nonces, channel salt) | Ionic |
| matter -> protocol | 3 (stream, LEDBAT, connection) | Newly declared, moderate |

The weakest bonds are mesh-to-relay and mesh-to-matter. These are the
bonds most likely to fracture during implementation. They have few
constraining invariants and large interface gaps.

### 1.3 Defect Census

Point defects (single missing specification):
- SansIO polling model (B1 remainder): the consumer API surface exists
  but the driving convention does not. An atom with the right valence
  but no bond geometry.
- Intra-cluster communication (B2): the dest_peer field is a vacant
  lattice site. The atom's position is reserved but no atom occupies it.
- Channel lifecycle (B3): CHANNEL_CREATE entry type is reserved (0x0E)
  but the creation semantics are undefined. A labeled vacancy.
- Matter garbage collection (B4): storage grows without bound. A
  pressure defect that will cause lattice fracture at runtime.
- Reliability mode selection (D-013): streams can be reliable or
  unreliable but the selection mechanism is unspecified.

Line defects (missing specification along a boundary):
- Relay transit forwarding: the entire forwarding path from
  source-cluster-gateway through relay to destination-cluster-gateway
  is unspecified. The hop_count field exists. The SIGNAL message types
  exist. The forwarding table and path selection algorithm do not.
  This is a grain boundary running through the relay layer.
- Matter sync/replication: SyncReconciliation and Replication are
  defined concepts with invariants but no body text. The merge rules,
  replica placement algorithm, and background replication work loop
  are missing. A grain boundary in the matter layer.

Planar defects (entire missing specification surface):
- The seat-to-mesh translation layer: HostPeer capabilities are
  enumerated but the translation from SeatRequest to mesh operation
  is declared as "application layer." This is correct for verkko
  scope but creates a planar defect at the verkko-to-saalis boundary.
  The crystal ends at a cleavage plane.

### 1.4 Polymorph Analysis

The metabolic state machine is the crystal's primary polymorph. Four
crystal structures (Background, Idle, Active, Stress) share the same
atom set but arrange the bonds differently. The phase transition
mechanism (Conductor meet projection over resource product lattice)
is well-specified. The hysteresis (dwell time) prevents rapid
oscillation. The phase jitter prevents synchronization.

One concern: the convergence metabolic override bypasses the dwell time
constraint. This is necessary (convergence must force Active/Stress
regardless of current state). But the exit sequence's Step 3 ("restore
metabolic dwell time enforcement") is a flag flip, not a gated
transition. If convergence completes during a Background-to-Active
transition that was already in progress, the dwell time restoration
could interfere with the in-progress transition. The design handles
this by stating that convergence override can jump to any state, but
the interaction between "restore dwell time" and "currently overridden"
is not explicitly sequenced.

### 1.5 Nucleation History

The crystal nucleated from a single cluster (genesis bootstrap) and
grows by invite-token-mediated nucleation. Each new cluster is a new
unit cell that bonds to the existing crystal via CPSK ionic bonds.
The nucleation mechanism is complete: invite token wire format, Noise
XX handshake, SAS verification, enrollment binding, key delivery.

The crystal growth mechanism (adding peers to existing clusters) is
less complete. Intra-cluster communication (B2) is the missing growth
specification. A new peer joining a cluster needs to establish sessions
with existing cluster peers, receive CIK, and begin participating in
intra-cluster heartbeats. The dest_peer field and CIK-based Noise KK
provide the raw material, but the growth sequence is unspecified.

### 1.6 Annealing Assessment

The three convergence fronts are heating stages. Front 1 (control) is
the highest temperature: flood gossip, GSet merge, EMA reset. Front 2
(consensus) is medium temperature: dominance recomputation, ring
coloring. Front 3 (re-encryption) is the lowest temperature: bulk
work under new keys.

The damping coefficients (HealthMode DAMPED: alpha_fast * 0.3,
alpha_slow * 0.1) are the cooling rate. The design correctly identifies
that health scoring during convergence would produce noise, and
attenuates it. The zero-observation bypass (3+ consecutive periods
without heartbeats) correctly exempts genuine liveness signals from
attenuation.

The annealing sequence is well-ordered: wavefront gating prevents
premature Front 1 start, ConvergenceGates enforce front sequencing,
the exit sequence restores steady-state operation. The convergence
duration bound (120-600 seconds) prevents infinite annealing.

### 1.7 Crystal Symmetry

The crystal has two symmetry types:

Internal symmetry (intra-cluster): poorly defined. The cluster is
a unit cell but its internal structure is unspecified. How do 1-6
peers within a cluster communicate? The design says "reliable broadcast
(iterated unicast) for Vital/Reactive messages. Best-effort broadcast
for gossip IHAVE." This is one sentence for an entire communication
pattern. The internal symmetry of the unit cell is the largest gap.

External symmetry (inter-cluster): well-defined. The gateway handles
all inter-cluster communication. The polygon schedule coordinates
CPSK renegotiation. The observation-path forwarding provides health
visibility. The mesh topology is a complete graph of gateways with
CPSK bonds.

### 1.8 Lattice Energy Budget

The design's resource governance (Conductor, GovernedResource,
ThresholdGuard) is the lattice's energy budget. Every operation costs
energy (CPU, flash, bandwidth, memory). The governor's meet projection
ensures no resource dimension is exhausted.

The energy budget is complete for steady-state operation. During
convergence, the budget is partially suspended (metabolic override
forces Active/Stress). The flash budget moderation (diminuendo curve,
P1 exempt) handles the convergence re-encryption load.

Missing: the energy budget for intra-cluster communication. If a
cluster has 6 peers, each peer maintains up to 5 intra-cluster
sessions. The session state, heartbeat overhead, and SACK processing
for these 5 sessions are not accounted for in the resource budget.
The 200KB managed arena budget and the wire cost tables only count
inter-cluster connections (11 connections for a 12-cluster mesh).
A 6-peer cluster adds 5 intra-cluster connections per peer, 30
intra-cluster connections total. This changes the resource picture.

### 1.9 Defect Migration Paths

When a defect forms (peer failure, network partition), it migrates
through the lattice along specific paths:

1. Heartbeat timeout (point defect) -> health EMA degradation ->
   scar timing -> circuit breaker -> gateway re-election ->
   convergence (if partition). This path is fully specified.

2. Content integrity failure (point defect) -> IntegrityFailure scar
   (Severe tier, weight 2.0) -> dominance cascade penalty -> potential
   admin succession. This path is fully specified.

3. Relay failure (point defect) -> ??? -> ???. The relay has no health
   scoring. The relay is not a mesh peer. When a relay dies, how does
   the mesh discover this? How does traffic reroute? The defect migration
   path for relay failure is completely unspecified.

4. Flash exhaustion (pressure defect) -> GovernedResource response
   (diminuendo) -> WritePermit denial -> re-encryption stall. This
   path is specified but the recovery is not: what happens when the
   diminuendo reaches 20% rate and the re-encryption P1 deadline
   (4 hours) is approaching?

### 1.10 Surface Energy

The crystal's surface energy is the cost of maintaining the API
boundary. The SansIO interface is the primary surface. It has the
right shape (ProtocolInput/ProtocolOutput event enumerations) but
the surface is not polished: the polling model, buffer ownership,
and error semantics are missing. An unpolished surface nucleates
implementation defects.

The ConvergenceSignalProvider and MetabolicGovernor are internal
surfaces (trait boundaries between layers). These are clean
cleavage planes. Each side of the plane has defined responsibilities.
The dependency direction is preserved. These are good surfaces.

The HeartbeatExtension is a boundary object surface. Protocol owns the
frame; mesh owns the semantics. The ownership table is explicit. The
error handling is specified. This is a well-polished surface.

## Part 2: The 7 Most Significant Problems

### Problem 1: The Hollow Unit Cell (Intra-Cluster Communication)

**(a) Crystallographic description:**
The unit cell (cluster) has defined external faces (gateway interfaces,
CPSK bonds to other unit cells) but no internal structure. The atoms
(peers) within the unit cell have no bond geometry. In a real crystal,
the internal structure of the unit cell determines all bulk properties:
thermal conductivity, electrical resistance, mechanical strength.
Without internal structure, the crystal cannot propagate forces from
one face to another. A unit cell with external faces but no internal
bonds is a hollow shell.

**(b) Verkko concrete terms:**
Intra-cluster communication has one sentence of specification: "reliable
broadcast (iterated unicast) for Vital/Reactive messages. Best-effort
broadcast for gossip IHAVE." The following are undefined:

- **Session model:** Do intra-cluster peers use Noise KK with CIK as
  the static key? With individual DEKs? Each peer has a DK and DEK.
  The CIK is shared. The Noise KK pattern requires both parties to
  know each other's static keys. For intra-cluster: initiator uses
  CIK as its static key, responder uses CIK as its static key.
  But CIK is shared, so both sides have the same static key. Noise KK
  with identical static keys on both sides degenerates: the handshake
  pattern produces k=DH(CIK, CIK_ephemeral) which is the same for
  both directions. This is a fundamental problem.

  The correct approach: intra-cluster sessions use Noise KK with each
  peer's DEK as the static key. The CIK is for inter-cluster identity.
  This requires peers within a cluster to know each other's DEK public
  keys. This is a key distribution problem within the cluster.

- **dest_peer assignment:** The dest_peer field is u16. How is a peer
  assigned a dest_peer value within its cluster? At enrollment?
  Dynamically? Sequential? The wire format reserves 0x0000 for
  broadcast. What values do individual peers get?

- **Gateway routing within cluster:** When a gateway receives an
  inter-cluster frame with dest_peer != 0x0000, it must route to the
  specific intra-cluster peer. The gateway must maintain a mapping
  from dest_peer values to intra-cluster sessions. This routing table
  is unspecified.

- **Heartbeat extension forwarding:** The gateway aggregates observation
  summaries from all cluster peers. How do non-gateway peers deliver
  their FlowSummary data to the gateway? Via intra-cluster heartbeats?
  Via a dedicated intra-cluster control channel?

- **CIK replication:** When CIK rotates (signed control log entry),
  the new CIK must be distributed to all cluster peers. This
  distribution uses intra-cluster communication. But CIK rotation
  invalidates the current inter-cluster CPSK sessions. The ordering
  between "distribute new CIK intra-cluster" and "renegotiate CPSK
  inter-cluster" is unspecified.

- **Resource accounting:** 5 intra-cluster sessions per peer in a
  6-peer cluster. Each session has: replay window (128 bytes), SACK
  state, LEDBAT state (or simplified variant), heartbeat timer.
  Total per peer: ~2-5KB additional managed arena. At 6 peers: 12-30KB.
  The 200KB managed arena budget does not account for this.

**(c) Emergent behavior with other proposals:**
The hollow unit cell interacts with every mechanism that crosses the
cluster boundary. Gateway election requires intra-cluster consensus on
who the gateway is (max-register CRDT). The CRDT update must propagate
within the cluster. Observation-path forwarding requires FlowSummary
delivery from non-gateway peers to the gateway. Convergence requires
all cluster peers to participate in Front 1 (control log merge) and
Front 2 (dominance recomputation). CIK rotation requires atomic
distribution within the cluster before any inter-cluster renegotiation.

The design implicitly assumes that intra-cluster communication is
fast, reliable, and ordered. These are reasonable assumptions for a
LAN. But the mechanism that delivers these properties is unspecified.
Without it, gateway election may produce inconsistent results (two
peers think they are gateway because the CRDT update has not
propagated), observation-path forwarding may miss flows (FlowSummary
not delivered), and convergence may stall (cluster peer has not
received control log entries from its own cluster).

The interaction between intra-cluster session establishment and the
SansIO interface is also unspecified. Does the consumer create
intra-cluster connections via ConnectionRequested? Are intra-cluster
sessions automatically established when a peer enrolls? The SansIO
event surface treats all connections uniformly (BytesReceived does
not distinguish intra- from inter-cluster). This is correct for the
protocol layer. But the mesh layer must distinguish them to apply
different policies (no LEDBAT for LAN, no relay fallback, different
heartbeat intervals).

### Problem 2: The Amorphous Relay (Transit Forwarding)

**(a) Crystallographic description:**
The relay layer is a grain boundary between two crystallized regions
(protocol and mesh). A grain boundary in a crystal is a region where
the lattice order breaks down. Atoms on either side of the boundary
are well-ordered within their respective grains, but at the boundary
itself, the bond angles are wrong, the spacing is irregular, and
the defect density is high.

The relay is a grain boundary because it sits between two well-defined
layers (protocol knows how to send frames; mesh knows how to route
them) but the relay's own behavior at the junction is amorphous. The
defined concepts (TransitRelay, RelayTopology, Membrane) describe what
the relay is. The body text describes the storage operations (PUT, GET,
DELETE, TOUCH, SIGNAL) and re-encryption mechanics. But the transit
forwarding function, which is the relay's raison d'etre, has no
specification.

**(b) Verkko concrete terms:**
When peer A cannot directly reach peer B (NAT, firewall), traffic
routes through a relay peer R. The following are undefined:

- **Relay session establishment:** How does A establish a relay
  session? The SIGNAL operation has types HELLO, PUNCH, REDIRECT.
  HELLO presumably initiates contact. PUNCH presumably coordinates
  hole-punching. REDIRECT presumably tells A to use a different
  relay. But the SIGNAL payload format is not specified. What does A
  send in a SIGNAL(HELLO)? Its connection_id? Its CIK fingerprint?
  The destination cluster's identifier?

- **Frame forwarding:** When R receives a frame from A destined for
  B, R reads the outer header (destination, connection_id, hop_count).
  R decrements hop_count. R looks up destination in its forwarding
  table. R forwards the frame to B (or to the next relay if multi-hop).
  The forwarding table structure is not specified. How does R learn
  that destination X maps to address Y? Via the SIGNAL mechanism?
  Via a separate relay control protocol?

- **Relay authentication:** R is not a full mesh peer. R does not
  hold channel keys. R does not participate in gossip. How does R
  authenticate incoming connections? Does R have its own DK/DEK?
  Does R accept connections from any peer that presents a valid CIK
  fingerprint? The trust boundary section says "relay sees opaque
  blobs and cluster IDs." But how does R verify that a peer is
  authorized to use relay services?

- **Relay discovery:** PeerDiscovery lists the fallback chain:
  mDNS -> invite hint -> cached peers -> DHT -> relay. The relay is
  the last resort. But how does a peer discover available relays?
  The invite token contains a relay_dht_key. The DHT (Mainline) is
  mentioned. But the DHT interaction protocol (how to publish relay
  availability, how to query for relays) is unspecified.

- **Path selection:** When multiple relay paths exist (R1, R2), how
  does A choose? Health-based? RTT-based? Random? The RelayTopology
  defined concept says "relay selection based on connectivity graph
  and health." But the selection algorithm is not specified. And the
  relay has no health scoring (it is not a mesh peer, so the
  HealthPipeline does not apply).

**(c) Emergent behavior with other proposals:**
The amorphous relay interacts with convergence, NAT traversal, and
the SansIO interface. During convergence, the polygon schedule
coordinates CPSK renegotiation between gateways. If two gateways
communicate through a relay, the Noise KK handshake messages must
transit the relay. The relay must forward handshake frames (which
have a different format: magic 0x56524B4F, not destination field).
The relay's frame disambiguation (pre-handshake vs post-handshake)
must work for forwarded traffic, not just direct traffic.

The relay also interacts with the matter transfer protocol. When a
peer requests matter from a remote cluster and the path goes through
a relay, the bulk-tier LEDBAT-paced chunk stream transits the relay.
The relay must forward these frames without inspecting them. But the
relay's constant-rate padding mechanism operates on relay operations
(PUT, GET), not on transit frames. Transit frames and storage
operations are two different traffic classes sharing the same relay
node, and their interaction is unspecified.

The relay's interaction with the Conductor is also undefined. A peer
using a relay path has lower effective bandwidth (relay adds latency
and may have limited capacity). The LEDBAT controller will naturally
back off, but the Conductor's bandwidth pressure input
(min(WiFi_capacity, transport_capacity)) does not account for relay
path quality. A peer behind a strict NAT using a relay has different
resource characteristics than a peer with direct connectivity, but
the metabolic state machine treats them identically.

### Problem 3: The SansIO Amorphous Surface (Polling Model)

**(a) Crystallographic description:**
The crystal has a well-defined internal structure but its surface is
amorphous. The surface atoms (ProtocolInput, ProtocolOutput) have the
right chemical composition but they are not arranged in a lattice.
The consumer touches this surface, and the lack of ordered surface
sites means the consumer cannot bond to the crystal in a predictable
way.

In surface crystallography, an amorphous surface causes poor epitaxial
growth: the layer grown on top of it will have many defects, even if
the deposited material is pure. The SansIO surface is the substrate
on which the consumer (the mesh layer, the saalis application) grows.
An amorphous substrate guarantees implementation fragmentation.

**(b) Verkko concrete terms:**
The SansIO interface has ProtocolInput and ProtocolOutput event
enumerations. What is missing:

- **Polling convention:** How does the consumer drive the state machine?
  Options:
  1. `fn poll(&mut self, input: ProtocolInput) -> Vec<ProtocolOutput>`
     (synchronous, returns all outputs for one input).
  2. `fn poll(&mut self) -> Option<ProtocolOutput>` with separate
     `fn feed(&mut self, input: ProtocolInput)` (two-phase, decoupled).
  3. `fn step(&mut self, input: ProtocolInput) -> ProtocolOutput`
     (one-in-one-out, iterator-like).

  Each produces different integration patterns. Without specifying the
  convention, two implementations will choose different patterns, and
  the mesh layer cannot be written against a stable API.

- **Buffer ownership:** The ProtocolOutput::SendBytes contains `data:
  bytes`. Where does this buffer come from? The arena? A caller-provided
  buffer? Does the caller own the buffer after receiving the output, or
  does the protocol machine retain ownership? The managed arena
  (199KB baseline) is the obvious answer, but the allocation and
  deallocation lifecycle is not specified.

  The NeedArena output event tells the consumer that the protocol
  machine needs more arena space. But what happens if the consumer
  cannot provide it? Does the machine drop frames? Buffer internally?
  Block?

- **Error semantics:** What happens when the protocol machine encounters
  an error? Does it produce an error output? Does it silently drop the
  input? Does it transition to an error state? The ConnectionClose
  mechanism handles clean shutdown. The PROTOCOL_ERROR reason handles
  fatal protocol violations. But non-fatal errors (buffer exhaustion,
  malformed input that is not a protocol violation) have no specified
  behavior.

- **Timer management:** The SetTimer output tells the consumer to
  set a timer. The TimerExpired input tells the machine that a timer
  fired. But the timer semantics are not specified. Are timers
  one-shot or repeating? Can a timer be cancelled (no CancelTimer
  input exists)? If a timer fires for a session that has been closed,
  what happens? The heartbeat timer, CPSK renegotiation timer,
  convergence wavefront timer, and retransmission timer are all
  critical, and all operate through this unspecified timer interface.

- **Reentrancy:** Can the consumer call feed() from within a callback
  that processes ProtocolOutput? If ProtocolOutput::FrameReady triggers
  the mesh layer to send a response, can the mesh layer immediately
  call feed(BytesReceived) with the response? This produces recursive
  state machine transitions. The design does not specify whether the
  state machine is reentrant.

**(c) Emergent behavior with other proposals:**
The SansIO surface interacts with every layer above it. The mesh layer's
Conductor must provide ConfigChanged inputs. The convergence mechanism
must observe HeartbeatReceived outputs and produce appropriate inputs.
The matter transfer must use StreamCreated/StreamClosed inputs and
FrameReady outputs.

If the polling convention is synchronous (poll returns Vec), the mesh
layer must process all outputs before feeding the next input. This
creates head-of-line blocking if one output takes a long time to
process. If the convention is two-phase (feed/poll), the mesh layer
must poll repeatedly until poll returns None, which requires a loop.

The buffer ownership question interacts with the relay. When the
protocol machine produces SendBytes for a relayed frame, the frame
data includes the outer header with the relay's address. The mesh
layer must know whether to copy the buffer before passing it to the
OS socket or whether the protocol machine's arena will retain it.
If the arena is bump-allocated (per-frame transient, 64KB hot arena),
the buffer is invalidated on the next poll. The consumer must copy.

The timer management interacts with the heartbeat interval. The
metabolic state machine changes the heartbeat interval. The mesh
layer calls ConfigChanged with the new metabolic state. The protocol
machine must cancel the old heartbeat timer and set a new one. But
CancelTimer does not exist as a ProtocolInput. Either the protocol
machine manages timer cancellation internally (and the consumer's
timer infrastructure must support implicit cancellation by
overwriting), or a CancelTimer input is needed.

### Problem 4: The Missing Replication Lattice (Matter Sync)

**(a) Crystallographic description:**
The crystal has atoms (matter) and bonds (transfer sessions), but the
crystal growth algorithm is undefined. In a real crystal, growth
follows specific rules: atoms attach at kink sites (energetically
favorable), the growth rate depends on supersaturation (how much
excess material is available), and the crystal shape depends on the
relative growth rates of different faces.

Matter replication is crystal growth. The mesh has matter distributed
across peers (atoms at lattice sites). Replication creates copies at
new sites. Without a growth algorithm, the crystal grows
haphazardly: some faces grow fast (peers with bandwidth), others
stagnate (peers behind relays), and the resulting crystal is
full of voids (under-replicated content) and protrusions
(over-replicated content).

**(b) Verkko concrete terms:**
The Replication defined concept has invariants:
- Configurable min_replicas and target_replicas per matter priority.
- Background replication work maintains targets.
- Replication IS the backup.
- Relay peers participate.

The following are undefined:

- **Replica placement:** Which peers get replicas? Random? Capacity-
  weighted? Cluster-diversity-aware? If all replicas are in one cluster,
  a cluster failure loses all copies. If replicas are spread across
  clusters, inter-cluster bandwidth is consumed. The placement
  algorithm determines both durability and bandwidth cost.

- **Replication work loop:** How does a peer decide what to replicate?
  Does the admin configure priorities? Does the system automatically
  replicate based on access frequency? The Stash tracks locally held
  matter. The ContentFilter advertises what a cluster holds. But the
  reconciliation between "what should be replicated" and "what exists"
  is not specified.

- **Sync reconciliation:** The SyncReconciliation defined concept says
  "divergence detected via Merkle root comparison in heartbeats" and
  "delta sync for incremental changes." But the merge rules are not
  specified. What happens when two peers modify the same metadata?
  The design says "field-level, deterministic, reproducible on all
  peers." This is an invariant, not a specification. Which fields?
  Which deterministic rule? Last-writer-wins with HLC? CRDT per field?

- **Background work scheduling:** Replication is background work.
  It uses bulk-tier streams (LEDBAT-paced). But how is replication
  work prioritized against matter transfer requests (which also use
  bulk tier)? The VitalQueue has three tiers, but within the bulk
  tier, there is no prioritization between user-requested transfers
  and background replication.

**(c) Emergent behavior with other proposals:**
The missing replication lattice interacts with the relay membrane and
the territorial re-encryption mechanism. Relay peers participate in
replication (encrypted at rest via Membrane). When a relay holds a
replica and the channel key is revoked, the relay must re-encrypt the
replica under the new key. The re-encryption is territorial
(consistent hash ring). But the hash ring only tracks peers with DK
pubkeys. Relay peers are not full mesh participants. Do relay peers
have DK pubkeys for hash ring purposes? If not, how are relay-held
replicas assigned to re-encryption territories?

The replication work loop interacts with the Conductor. Background
replication consumes bandwidth, CPU, and flash. The Conductor's
metabolic state determines the available budget. In Background mode
(30-50% bandwidth), replication must proceed slowly. In Stress mode
(80-90% bandwidth), replication can proceed faster but the system
is under resource pressure for other reasons. The interaction between
replication priority and metabolic state is not specified.

The sync reconciliation interacts with convergence. During
convergence, the mesh merges GSet state (control log). Matter
metadata is not in the control log (it would bloat the GSet). So
matter metadata sync happens outside convergence, via the
SyncReconciliation mechanism that has no body. This means matter
metadata can remain inconsistent after convergence completes. The
convergence exit sequence does not gate on matter sync completion.

### Problem 5: The Relay Defect Migration Gap

**(a) Crystallographic description:**
In a crystal, every atom site has defined defect migration paths. When
an atom vacates its site (vacancy defect), neighboring atoms can
migrate into the vacancy along specific crystallographic directions.
The migration energy barrier determines how fast vacancies diffuse.

The relay has no defect migration paths. When a relay atom vacates
its site (relay goes offline), no neighboring atom migrates into the
vacancy. The vacancy remains until... what? The design does not say.
The peers relying on the relay for NAT traversal lose connectivity.
The matter stored on the relay becomes inaccessible. The territorial
re-encryption territories assigned to the relay's hash ring positions
become orphaned.

This is not a point defect. It is a structural vacancy in the defect
handling system itself.

**(b) Verkko concrete terms:**
Relay failover (D-008) is deferred to Run 13. But the interaction
between relay failure and existing mechanisms is not even sketched:

- **Health scoring for relay paths:** The HealthPipeline scores
  peer-to-peer links. When two peers communicate through a relay,
  the health score reflects the end-to-end path quality. If the relay
  degrades, the health score degrades. But the scar attribution cannot
  distinguish "relay degraded" from "destination peer degraded."
  ScarAttribution has PathFault { path: PathId } which could represent
  a relay path, but PathId is not defined and there is no mechanism to
  determine that a relay is the cause of path degradation.

- **Matter availability:** When a relay holding replicas goes offline,
  those replicas become inaccessible. The ContentFilter still
  advertises them (the filter is updated per cluster, not per relay).
  A peer requesting matter will receive a positive filter hit, attempt
  a transfer, fail, and potentially scar the source peer, not the
  relay. The source peer is not at fault.

- **Re-encryption territory orphaning:** The consistent hash ring
  assigns territories based on DK pubkeys. If a relay holds matter
  in its territory and goes offline, the re-encryption deadline (P1:
  4 hours) may expire. The retain-until-handoff invariant ("a peer
  retains ownership of its territorial partition until it explicitly
  hands off") cannot be satisfied because the offline relay cannot
  perform the handoff.

**(c) Emergent behavior with other proposals:**
The relay defect migration gap interacts with the seat layer. A seat
connects to a host peer via WebSocket (LAN only). But the host peer
may communicate with remote clusters through a relay. If the relay
fails, the host peer loses inter-cluster connectivity. The seat
observes this as degraded mesh health (HEALTH_SUMMARY events) but has
no actionable recovery path. The host peer cannot switch to a
different relay because relay failover is unspecified.

The gap also interacts with convergence. If a partition heals but one
of the reconnecting pairs requires a relay (neither can reach the other
directly), the wavefront timer starts, the polygon schedule begins,
but the CPSK renegotiation for that pair requires relay transit. If
the relay is degraded during convergence, the pair's renegotiation
may fail. The polygon schedule handles this by treating late-arriving
pairs as still-partitioned. But this means the pair remains
unconverged after the convergence event, creating a persistent
partial-partition state.

### Problem 6: The Uncrystallized Channel Lifecycle

**(a) Crystallographic description:**
Channels are the chemical species in the crystal. Each channel is a
distinct atomic species with its own bond characteristics (key, access
control, epoch). The crystal supports 50+ species. But the mechanism
for introducing a new species into the crystal is undefined. New species
appear spontaneously (CHANNEL_CREATE entry type is reserved but
unspecified). The crystal cannot grow new chemical domains.

**(b) Verkko concrete terms:**
The Channel defined concept states: "Each channel has an independently
generated 256-bit key." The control log entry type 0x0E (CHANNEL_CREATE)
is reserved. The following are undefined:

- **Channel creation authorization:** Who can create channels? Only
  admin? Any peer with write access to the control log? The access
  control model defines per-channel, per-peer access (off, opaque,
  read, write). But the access control for channel creation itself is
  unspecified.

- **Channel semantics:** What distinguishes one channel from another?
  The design says "hierarchical data stream." Hierarchical implies
  parent-child relationships between channels. But the hierarchy
  structure is not specified. Is it a flat list with u16 identifiers?
  A tree? What is the root channel?

- **Channel deletion:** Can channels be deleted? The control log is
  a GSet (append-only). A deleted channel would need a CHANNEL_DELETE
  entry type. But deletion interacts with sealed ranges: a peer
  returning from offline may have sealed ranges referencing a deleted
  channel. The design does not address this.

- **Channel-to-content binding:** How is matter associated with a
  channel? The ContentFilter advertises matter per cluster. The
  channel_index is in the encrypted payload (per frame). But the
  binding between "this matter belongs to this channel" is not
  specified. Is it part of the Stash? Part of the matter metadata?

**(c) Emergent behavior with other proposals:**
Channel lifecycle interacts with KeyBundle operations. KeyBundle
operations are per-channel (Grant, Rotate, Revoke). Creating a channel
requires generating a key and distributing it via KeyBundle(Grant).
But the KeyBundle mechanism assumes channels already exist. The
circular dependency (channels need KeyBundle; KeyBundle needs channels)
is resolved by the genesis bootstrap (which creates initial channels)
but not by the runtime (which needs dynamic channel creation).

Channel lifecycle also interacts with EDT. The EDT has scope_flags
(u64 bitmask) controlling which channels the delegate can grant access
to. This implies channels have a persistent identity that the bitmask
references. But the mapping from channel_index (u16) to scope_flags
bit position is not specified. If channels are created dynamically,
the bitmask must be updated, but EDT tokens are signed and immutable.

### Problem 7: The Fixed-Point Conversion Boundary

**(a) Crystallographic description:**
The crystal's internal computations use a different unit system
(fixed-point Q16.16/Q8.24) than the external measurements (floating-
point from OS APIs). The conversion happens at the PhysicalBoundary
gates. This is a phase boundary in the crystal: two phases (floating-
point and fixed-point) meet at a defined interface.

Phase boundaries can be sharp (well-defined conversion) or diffuse
(gradual transition). The design specifies a sharp boundary (sole
conversion point). But several areas of the design still reference
floating-point operations in contexts that should be fixed-point,
creating diffuse regions where the phase boundary is violated.

**(b) Verkko concrete terms:**
The fixed-point arithmetic decision (post-Run-11, resolving OQ-2) is
embedded in the mesh body (Section 2.0a). The health pipeline, EMA,
dominance cascade, and scar weighting use fixed-point. But:

- **Scar decay formula:** `weight(scar, now) = max(0.0, 1.0 - (now -
  scar.observation_epoch) / 180_days)`. This is written with floating-
  point literals (0.0, 1.0). In fixed-point Q16.16: `max(0, 0x10000 -
  (dt_days * 0x10000) / 180)`. The integer division truncates
  differently than floating-point division. The 180-day decay cliff
  (weight goes from ~0.006 to 0) is affected: fixed-point may produce
  weight=0 slightly before or after 180 days depending on truncation
  direction.

- **CorrelatedSuspicion confidence field:** `ScarAttribution::
  CorrelatedSuspicion { ..., confidence: f32 }`. This is an f32 in the
  scar attribution enum. The scar attribution is in the abstract state
  machine. The lint rule "prevents raw floating-point types (f32, f64)
  from appearing as parameters to functions in the abstract state
  machine." The confidence field violates this lint. It needs to be a
  fixed-point type.

- **CapacityRatio guard:** Section 6.1 says "if `expected_throughput
  <= 0.0`, return `capacity_ratio = 1.0`." The <= 0.0 comparison
  implies floating-point. In fixed-point, this is `if
  expected_throughput_q16_16 == 0`. But the comment says "not
  < f64::EPSILON" which explicitly references f64. The guard needs
  to be rewritten in fixed-point terms.

- **EMA update code:** Section 5.1 shows `fn replay_observation` with
  `f64` parameters and `.powf()` calls. The Arithmetic Safety Invariant
  section correctly specifies the fixed-point EMA update formula
  (`old + ((alpha_q8_24 as u64 * diff as u64) >> 24)`). But the
  replay_observation function and the stale cutoff computation use
  floating-point. These are inconsistent. One is the specification;
  the other is the code sample. They disagree.

**(c) Emergent behavior with other proposals:**
The fixed-point conversion boundary interacts with cross-peer
determinism. Fixed-point arithmetic is deterministic across platforms
(same result on ARM and x86). Floating-point arithmetic is not (IEEE
754 rounding mode differences, extended precision on x87). If any
peer uses floating-point where another uses fixed-point, they will
compute different health scores, different dominance orderings,
different convergence gate predicates. The mesh will disagree on who
is admin, which peers are healthy, and when convergence is complete.

The inconsistency between the fixed-point specification and the
floating-point code samples creates a trap for implementors. An
implementor reading the code samples will use f64. An implementor
reading the Arithmetic Safety Invariant will use Q16.16. They will
produce different results. The mesh will partition.

The confidence: f32 in CorrelatedSuspicion is particularly dangerous.
If this field enters the weighted_scar_count computation (it does, via
tier weight), the f32 -> Q16.16 conversion happens mid-computation,
not at the PhysicalBoundary gate. The conversion boundary is violated.

## Part 3: Open Items and Known Limitations

### Open Items

1. **dest_peer assignment convention.** The u16 field exists. The
   assignment mechanism does not. I propose: sequential assignment at
   enrollment, starting from 0x0001. Gateway is always dest_peer 0x0001
   (lowest-numbered enrolled peer in the cluster). Broadcast is 0x0000.
   Simple, deterministic, no coordination.

2. **Intra-cluster Noise pattern.** Noise KK with DEK (not CIK) as
   static key. CIK is the cluster's identity key for inter-cluster
   Noise KK. DEK is the peer's encryption key for intra-cluster
   Noise KK. Peers learn each other's DEK public keys during enrollment
   (enrollment cert contains new_DEK_pub). This is already in the design
   but not connected to intra-cluster session establishment.

3. **Relay peer identity.** Does a relay have a DK/DEK? If yes, it is a
   pseudo-peer with limited capabilities (no channel keys, no gossip,
   no health scoring). If no, it authenticates via a different mechanism
   (PSK from invite token?). This is a design decision, not a
   specification gap.

4. **Channel hierarchy model.** Flat (u16 identifiers, no hierarchy) or
   tree (parent-child relationships with permission inheritance)? The
   design says "hierarchical data stream" in the Channel defined concept.
   The access control model (off, opaque, read, write) is per-channel,
   per-peer, which works for both flat and hierarchical. But
   "hierarchical" implies structure that is not specified.

5. **SansIO timer cancellation.** The ProtocolInput enum needs a
   CancelTimer { timer_id } variant, or the specification must state
   that SetTimer with the same timer_id replaces the previous timer
   (implicit cancellation by overwrite).

6. **Relay health monitoring.** The relay is not a mesh peer but it
   affects end-to-end health. A mechanism is needed to detect relay
   degradation and distinguish it from destination peer degradation.
   I propose: relay path health is a separate EMA track, keyed by
   (source_cluster, relay_address, destination_cluster), fed by
   end-to-end SACK delivery ratios on relay paths. When relay path
   health degrades but direct paths to the same destination cluster
   (via other relays or after hole-punching) are healthy, the scar
   is attributed to the relay path, not the destination.

### Known Limitations

1. **This review does not address the saalis-to-verkko application
   bridge.** The HostPeer capability enumeration and SeatRequest
   translation are explicitly out of verkko scope.

2. **This review does not propose changes to the crypto layer.** The
   crypto layer is near-complete (~90%) and the remaining 10% is
   ProVerif verification and test vectors, not design gaps.

3. **The fixed-point conversion analysis is limited to the code samples
   in the design documents.** The actual implementation may resolve the
   inconsistencies. But the design documents are the source of truth,
   and they are inconsistent.

4. **Relay transit forwarding is a large enough gap that it warrants a
   dedicated design round.** This review identifies the gap and sketches
   the surface area but does not propose a complete specification.

5. **Matter replication and sync reconciliation are large enough gaps
   that they warrant a dedicated design round.** This review identifies
   the invariants that are needed but does not propose specific
   algorithms.

## Part 4: Ideal Design Synthesis -- The Complete Crystal Structure

### 4.1 The Unit Cell Interior

The cluster's internal structure is a simple fully-connected graph of
1-6 peers. Each peer maintains a Noise KK session with every other
cluster peer, using DEK as the static key. The session produces
per-peer transport keys (independent of CIK and CPSK).

Intra-cluster heartbeats run at a fixed 1-second interval (LAN
latency is negligible; the variable interval is for WAN conservation).
The heartbeat carries the same transport header as inter-cluster
heartbeats but uses a simplified 8-byte extension region: filter_root
(8 bytes) only. The other extension fields (epoch_current,
work_capacity, health_canary, observation_summary) are inter-cluster
concerns.

dest_peer assignment: sequential u16 at enrollment, starting from
0x0001, incrementing for each new peer. The assignment is logged in
the enrollment cert (one additional u16 field). dest_peer 0x0000 is
broadcast (to all cluster peers). dest_peer values are never reused
within a cluster's lifetime (even after peer revocation, the value
is retired).

Gateway routing: the gateway maintains a routing table mapping
dest_peer values to intra-cluster session handles. When a frame
arrives from an inter-cluster session with dest_peer != 0x0000, the
gateway looks up the dest_peer in the routing table and forwards the
frame on the corresponding intra-cluster session. The inner header
is not modified; only the outer header changes (new connection_id,
new nonce, decremented hop_count).

Resource accounting: intra-cluster sessions use a separate
ThresholdGuard instance with a 10KB arena allocation per peer
(60KB maximum for a 6-peer cluster). This is added to the managed
arena baseline (200KB + 60KB = 260KB total, well within the 25.6MB
hard cap).

Intra-cluster health scoring: simplified. No dual-EMA. A single
heartbeat counter. 3 missed intra-cluster heartbeats (3 seconds)
triggers a local-only health degradation. 10 consecutive misses
triggers a scar. Intra-cluster health is not gossiped; it is
local state used for gateway election tiebreaking and
observation-path forwarding quality.

### 4.2 The Relay Crystal

The relay is a pseudo-peer with a DK/DEK pair but no CIK (it does not
belong to a cluster). The relay authenticates to the mesh via a
relay-specific enrollment process:

1. Admin generates an invite token with token_type = 0x04 (RELAY).
2. Relay generates DK, DEK, connects via Noise XX to the mesh.
3. Admin signs a relay enrollment cert (distinct from peer enrollment).
4. Relay receives a relay-specific control channel key (read-only
   for heartbeats, revocations, and convergence signals).

The relay does not participate in gossip, dominance, or convergence.
The relay receives a subset of heartbeat information sufficient to
maintain its forwarding table and TTL timers.

Transit forwarding:

```
ForwardingEntry {
    destination: u32,       // BLAKE3(CIK_pub)[0..4]
    next_hop: SocketAddr,   // where to send frames for this destination
    last_seen: HlcTimestamp,
    health: u8,             // simplified 0-255 health score
}
```

The forwarding table is populated via SIGNAL(HELLO) messages from
connecting peers. When peer A sends SIGNAL(HELLO) to relay R, the
SIGNAL payload contains:

```
SignalHello {
    source_destination: u32,    // A's destination field
    target_destination: u32,    // B's destination field (who A wants to reach)
    connection_id: u32,         // A's connection_id for this relay session
}
```

R creates a forwarding entry: frames with destination == B's
destination field are forwarded to A's address (R remembers A's
source address from the UDP datagram).

When B also sends SIGNAL(HELLO) to R with A's destination as target,
R links the two entries and begins bidirectional forwarding. R does
not inspect the encrypted payload. R reads only the outer header
(destination, connection_id, hop_count). R decrements hop_count and
forwards.

Relay health: the gateway monitors relay path quality via end-to-end
SACK delivery ratios. A separate EMA track (keyed by relay address)
tracks relay health. When relay health degrades below 0.50, the
gateway attempts hole-punching directly or falls back to an alternate
relay. Relay path health is included in the FlowSummary as a separate
field (1 byte, u8 relay_health, 0xFF if no relay used).

Relay failover: when a relay stops responding (3 missed keepalive
probes at 70% of mapping lifetime), the gateway queries the DHT for
alternative relays and initiates new SIGNAL(HELLO) exchanges. The
connection_id is preserved across relay migration (same mechanism
as connection migration: PATH_CHALLENGE/PATH_RESPONSE through the
new relay). The connection survives the relay change.

### 4.3 The SansIO Surface Crystal

The polling model:

```rust
trait VerkkoProtocol {
    /// Feed one input event. Returns the number of pending outputs.
    fn feed(&mut self, input: ProtocolInput) -> usize;

    /// Poll one output event. Returns None when no outputs remain.
    fn poll_output(&mut self) -> Option<ProtocolOutput>;

    /// Query the next timer deadline (for epoll/io_uring integration).
    fn next_deadline(&self) -> Option<Instant>;
}
```

Two-phase (feed/poll), decoupled. The consumer feeds one input, then
polls all outputs until poll_output returns None, then feeds the next
input. This prevents reentrancy (feed cannot be called from within
output processing because poll_output does not call back into the
consumer).

Buffer ownership: SendBytes data is a slice into the hot arena.
The slice is valid until the next call to feed(). The consumer must
copy or send the data before the next feed() call. This is zero-copy
on the fast path (consumer can pass the arena slice directly to
sendmmsg) and the ownership is unambiguous.

The NeedArena output indicates that the protocol machine's hot arena
is exhausted. The consumer must provide a new arena allocation via
a ProtocolInput::ArenaProvided { base: *mut u8, len: usize } event
(new input variant). If the consumer cannot provide an arena, the
protocol machine enters a degraded mode where it processes only
heartbeats and SACKs (vital tier), dropping all other frames. This
is explicit in the specification, not an implementation-time decision.

Timer management: SetTimer implicitly cancels any existing timer with
the same timer_id. CancelTimer is not needed as a separate input.
Timer IDs are small integers (u16). The consumer maintains a timer
wheel or priority queue indexed by timer_id. The protocol machine
guarantees that at most 64 timers are active simultaneously (bounded
by connection count * timers-per-connection).

Error handling: the protocol machine never panics. Non-fatal errors
(malformed input, buffer exhaustion, unexpected state) produce
ProtocolOutput::DiagnosticEvent { severity, message } outputs. Fatal
errors produce ConnectionClosing { reason: INTERNAL_ERROR } outputs.
The consumer logs diagnostics and handles connection loss.

### 4.4 The Replication Lattice

Replica placement: cluster-diversity-first. For target_replicas = 3,
at least 2 replicas must be in different clusters (if 2+ clusters
exist). Within the cluster constraint, placement prefers peers with
the most available flash capacity (from work_capacity heartbeat
field). Relay peers count as a separate "cluster" for diversity
purposes.

Replication work loop:

```
on_heartbeat_tick():
    if metabolic_state == Background:
        replication_budget = 1 chunk per tick
    elif metabolic_state == Idle:
        replication_budget = 5 chunks per tick
    else:
        replication_budget = 0  // Active/Stress: no background replication

    for _ in 0..replication_budget:
        matter = most_under_replicated()
        if matter.replica_count < matter.min_replicas:
            target = select_placement_target(matter)
            initiate_transfer(matter, target)
```

Sync reconciliation: metadata sync uses the same Merkle root mechanism
as ContentFilter. Each peer maintains a Merkle tree of its Stash
entries (matter_id, size, status, last_verified). The Merkle root is
included in the heartbeat extension (alongside filter_root, using the
same 8-byte truncated keyed root, alternating: even heartbeats carry
filter root, odd heartbeats carry stash root). On divergence detection,
incremental delta sync proceeds identically to filter reconciliation.

Merge rules for metadata conflicts: last-writer-wins with HLC
timestamp. The HLC provides causal ordering. Ties (identical HLC)
broken by peer_id. This is consistent with the data plane's existing
HLC + peer_id tiebreaker total order.

### 4.5 The Channel Lifecycle

Channels are flat (not hierarchical). The "hierarchical data stream"
description refers to the hierarchical access control (channel-level
permissions inherit from mesh-level defaults), not hierarchical
channel structure. u16 channel indices, 0x0000-0xFFFE usable,
0xFFFF = control.

Channel creation:

1. Admin generates a 256-bit channel key (CSPRNG).
2. Admin creates a CHANNEL_CREATE control log entry (entry_type 0x0E)
   containing:
   - channel_index: u16 (next unused index, monotonically assigned)
   - channel_key (wrapped to each authorized peer's DEK)
   - initial_access_list: [(PeerId, AccessLevel)]
   - channel_metadata: (name, description, priority_tier)
3. The entry is signed by admin DK and propagated via gossip.
4. Peers receiving the entry add the channel to their local channel
   table and KeyBundle state.

Only admin can create channels. EDT holders cannot create channels
(the EDT scope_flags only control access to existing channels).

Channel deletion is not supported. Channels can be revoked
(KeyBundle(Revoke) for all peers) which makes them inaccessible but
does not remove the channel_index from the namespace. Revoked
channels consume 34 bytes of channel table space indefinitely. At 50
channels, this is 1,700 bytes. Negligible.

### 4.6 The Fixed-Point Phase Boundary

The phase boundary between floating-point (hardware measurements)
and fixed-point (abstract state machine) must be sharp. Every line
of the design documents that references floating-point operations
in the abstract state machine must be rewritten in fixed-point terms.

Specific fixes:

1. `CorrelatedSuspicion { ..., confidence: f32 }` -> `confidence:
   u16` (Q8.8, range [0.0, 255.0] with 1/256 resolution). The
   confidence value is a ratio (0.0 to 1.0 in practice), so Q8.8
   provides 0.004 resolution, which is more than sufficient.

2. Scar decay formula rewritten:
   ```
   weight_q16_16(scar, now) =
       let dt_hours = (now - scar.observation_epoch) as u64;
       let decay_q16_16 = (dt_hours * 0x10000u64) / (180 * 24);
       if decay_q16_16 >= 0x10000 { 0 }
       else { 0x10000u32 - decay_q16_16 as u32 }
   ```

3. CapacityRatio guard: `if expected_throughput_q16_16 == 0 { return
   CapacityRatio(0x10000); }` (Q16.16 representation of 1.0).

4. EMA replay_observation function rewritten in fixed-point:
   ```
   fn replay_observation_q16(
       ema: &mut u32,  // Q16.16
       obs: u32,       // Q16.16 HealthScore
       dt_ms: u32,     // milliseconds
       alpha_base_q8_24: u32,
   ) {
       let stale_cutoff_ms = STALE_CUTOFF_MS[alpha_base_q8_24];
       if dt_ms > stale_cutoff_ms {
           *ema = obs;  // hard reset
       } else {
           // alpha_eff = 1 - (1-alpha)^(dt/dt_ref)
           // Approximated via lookup table for (1-alpha)^n
           let alpha_eff = alpha_lookup(alpha_base_q8_24, dt_ms);
           let diff = obs as i64 - *ema as i64;
           *ema = (*ema as i64 + ((alpha_eff as i64 * diff) >> 24)) as u32;
       }
   }
   ```

5. The `powf()` call in the alpha_effective computation cannot be
   done in fixed-point directly. Pre-compute a lookup table indexed
   by (dt_ms / DT_REFERENCE_MS) with 256 entries covering
   dt from 0 to 128 * DT_REFERENCE. Table size: 256 * 4 = 1KB per
   alpha_base value. Two alpha_base values (fast, slow) = 2KB.
   Interpolation between table entries provides sub-entry resolution.
   The lookup table is computed at compile time from the exact
   floating-point formula and stored as Q8.24 fixed-point values.

The lookup table approach has a tradeoff: it consumes 2KB of flash
for deterministic cross-platform computation. The alternative
(integer approximation of pow()) introduces approximation error that
varies by platform. 2KB is negligible in the 200KB managed arena.

### 4.7 Emergent Properties of the Complete Crystal

When all six problems are addressed, the following emergent properties
appear:

**Property 1: Uniform defect migration.** Every defect type (peer
failure, relay failure, flash exhaustion, integrity failure) has a
defined migration path through the lattice. No defect can form without
a recovery mechanism. The crystal is self-healing at every scale.

**Property 2: Recursive health observation.** The health pipeline
scores inter-cluster links. The simplified intra-cluster health
scoring (heartbeat counter) scores intra-cluster links. The relay
health track scores relay paths. Together, these three health
dimensions cover every bond in the crystal. The gateway aggregates all
three into the observation summary. The Conductor consumes the
aggregate. Every bond in the crystal contributes to the metabolic
state.

**Property 3: Layered replication.** Matter replicates across clusters
(cluster diversity constraint). Channels replicate keys within clusters
(CIK distribution). Control log entries replicate everywhere (GSet
gossip). Each replication layer operates at a different timescale
(matter: background, keys: real-time, control: reliable broadcast).
The three layers are independent but the ContentFilter and Merkle
reconciliation provide a unified divergence detection mechanism.

**Property 4: Relay transparency.** The relay forwards frames without
modification (except hop_count decrement). The relay's forwarding
table is populated by SIGNAL messages. The relay's health is monitored
by end-to-end SACK ratios. The relay is invisible to the application
layer. A peer communicating through a relay produces identical
ProtocolInput/ProtocolOutput events as a peer communicating directly.
The SansIO abstraction makes relay transit a transport-level concern,
not a mesh-level concern.

**Property 5: Channel-as-species.** Channels are chemical species in
the crystal. Each species has a defined creation mechanism (admin
CHANNEL_CREATE), a defined bond type (channel key, epoch ratchet),
and a defined access control model. The channel lifecycle is
monotonic: channels are created but never destroyed. Revocation
removes access but not identity. This monotonicity is consistent
with the GSet monotonicity of the control log and the scar GSet.
The entire crystal is monotonic: state grows, weight decays, but
nothing is deleted.

**Property 6: Deterministic cross-platform computation.** The
fixed-point phase boundary ensures that every computation in the
abstract state machine produces identical results on every platform.
Health scores, dominance orderings, convergence gate predicates, and
scar weights are bit-for-bit identical on ARM and x86. The crystal's
structure is platform-independent.

**Property 7: Surface crystallization.** The SansIO surface
(feed/poll_output/next_deadline) is a minimal, non-reentrant API.
The buffer ownership is unambiguous (arena slice valid until next
feed). The timer management is implicit (SetTimer overwrites). The
error handling is explicit (DiagnosticEvent for non-fatal,
ConnectionClosing for fatal). The consumer can bond to this surface
without ambiguity. Two independent implementations of the consumer
will produce identical behavior given identical inputs.
