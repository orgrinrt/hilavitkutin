## 1. What Their Formalizations Revealed

The three experts gave names to things I felt but could not articulate precisely.

**The SpecAuditor named the state machine hiding inside the damping coefficient.** I described a conditional multiplier. They formalized it as a two-mode state machine (NORMAL / DAMPED) with explicit transitions. This is not just a notational preference. A state machine has enumerable transitions. A conditional multiplier has continuous conditions. The difference is testability. I was proposing something untestable and calling it a mechanism. The SpecAuditor turned it into something verifiable. I adopt the two-mode state machine formulation without reservation.

**The DepAnalyst named the firewall principle.** I described ConvergenceGate as a decoupling mechanism. The DepAnalyst extracted the general principle: "opaque grain boundary projections." Every grain boundary defined concept should be a one-way mirror. Consumers see outcomes, not mechanisms. I called this "the gate acts as a firewall" in one sentence. The DepAnalyst turned it into a graph-wide evolution property. The name matters: "opaque projection principle" is a design rule that applies everywhere. My per-mechanism description was not a rule; it was an observation. The DepAnalyst made it a rule.

**The GapHunter named the bootstrap ordering problem.** I prioritized wire contracts by structural importance (grain boundaries first). The GapHunter pointed out that structural priority and implementation priority are different orderings. The first bytes on the wire are the Noise handshake, connection ID, and SchemaSync. None of these are grain boundary concepts. My ordering was correct for design integrity. The GapHunter's ordering is correct for build sequence. Both orderings must exist. I had one; they supplied the other.

**The SpecAuditor named the consistency model.** I described a snapshot mechanism. They named it: "read-your-writes during normal operation, snapshot isolation during convergence." This is not a description; it is a contract. The consistency model survives implementation changes. The snapshot mechanism does not. The SpecAuditor's formulation is strictly more durable than mine. I adopt it.

**The GapHunter named the coordination gap in damping.** I proposed a locally-applied damping mask. The GapHunter traced what happens when peers detect convergence at different times. Peer C observes Peers A and B degrading (convergence disruption) but does not know to damp those observations. The damping mask must be communicated, not just locally applied. The `convergence_active` heartbeat bit is the fix. I should have seen this. The damping mechanism is a distributed system problem, not a local computation problem. My proposal was correct for a single node and broken for a mesh.

## 2. Final Corrections Accepted

### 2.1 The damping coefficients are wrong (SpecAuditor 3.1)

The SpecAuditor checked my recovery time estimates. I said "~10 observations" for fast EMA recovery. The correct number is 29. I said "~100 observations" for slow EMA recovery. The correct number is 299. I was wrong by 3x on both estimates. The mechanism is still correct (damping prevents oscillation) but the recovery timeline is materially different from what I claimed. The coefficients themselves (0.3x, 0.1x) should be treated as calibration items, not protocol constants, until empirical testing on real household meshes validates them. Hard constraint accepted.

### 2.2 The zero-observation bypass is necessary (GapHunter 1.1)

The GapHunter identified that damping should not apply to absent peers (zero heartbeats for 3+ consecutive periods). Zero heartbeats is not a convergence perturbation; it is a dead peer. The damping mask must bypass for zero-observation events. This is a hard constraint that my original proposal lacked. Without it, genuine peer failures during convergence are detected 3.3x slower on the fast EMA, which is exactly when fast detection matters most (gateway election is in flux). Hard constraint accepted.

### 2.3 The scar tier mapping must be exhaustive and criteria-driven (GapHunter 2.2)

My tier table listed six scar types across three tiers. The GapHunter correctly noted this is an illustrative subset, not a complete mapping. The SpecAuditor added that new scar types without explicit tiers must default to Tier 2. The GapHunter went further: define the classification criteria, not just the assignments. Tier 1 = environmental and self-correcting. Tier 2 = behavioral but possibly accidental. Tier 3 = protocol violation or security breach. The criteria framework is necessary. Without it, every new scar type reopens a classification debate. Hard constraint accepted.

### 2.4 The seat isolation valve needs a staleness indicator (GapHunter, SpecAuditor)

Both reviewers independently identified that seat responses during convergence must carry a `stale` flag. The SpecAuditor formalized the full consistency model (CONSISTENT vs SNAPSHOT). The GapHunter identified the conflict resolution gap for deferred writes. Both are hard constraints. A snapshot without a staleness indicator is a lie to the user. A deferred-write queue without conflict resolution is a silent data corruption vector. Hard constraints accepted.

### 2.5 ScarSeverityTier should be a defined concept (SpecAuditor 2.2)

The SpecAuditor correctly argued that the tier assignment belongs in a defined concept, not buried in the dominance scoring section. Multiple consumers need it (dominance, health reporting, admin UI). Making it a defined concept ensures every scar variant has exactly one tier, PartitionAbsence is explicitly Tier 0, and new variants default to Tier 2. Hard constraint accepted.

### 2.6 Wire contracts need canonical generators, not just test vectors (SpecAuditor 2.3)

Test vectors alone have gaps. Two implementations can pass the same vectors and diverge on uncovered edge cases. The canonical generator (pseudocode algorithm producing bytes from inputs) is what makes a specification complete. IETF RFCs always include the algorithm. The wire contract specification must include byte layout + test vectors + canonical generator. Hard constraint accepted.

## 3. Corrections Rejected

### 3.1 I reject the DepAnalyst's reclassification of the seat isolation valve (DepAnalyst 5)

The DepAnalyst argues the seat isolation valve has "zero structural impact" on the dependency graph and should not share a priority list with ConvergenceGate and HeartbeatExtension. The DepAnalyst is correct about the graph. The DepAnalyst is wrong about the priority list.

The priority list is not a graph repair list. It is a design completion sequence. The lattice includes the user. The seat isolation valve is the bond between the crystal and the person holding it. A crystal that cuts the hand that holds it is a design failure regardless of its internal structural perfection. The DepAnalyst treats the dependency graph as the complete model. It is not. The user experience during convergence is a load-bearing surface of the design, and the valve specifies that surface. Structural fixes and behavioral designs are different kinds of work. They are not different levels of importance.

I keep the seat isolation valve at priority #6. Not because it fixes a graph edge. Because it fixes the part that touches the human.

### 3.2 I reject the smooth penalty function for scar scoring (SpecAuditor 1.2 footnote)

The SpecAuditor notes that the `floor()` in the exponential threshold "softens but does not eliminate the cliff" and suggests a smooth penalty function like `1 + max(0, weighted - 10)^1.5` to eliminate it entirely. I reject this.

Crystallographic: cliffs are features, not bugs. A crystal has sharp phase boundaries. The dominance scoring system should have a clear threshold where a peer transitions from "trusted despite some issues" to "demoted due to accumulated concern." A smooth function creates a gradient where every scar matters equally. The tiered system with a threshold creates two regimes: one where transient scars are tolerable and one where accumulated weight triggers intervention. The threshold is the phase boundary. Smoothing it creates an amorphous transition zone where nobody knows whether a peer is trusted or not.

The cliff at 10.0 weighted units is intentional. The tiering ensures only genuinely concerning patterns reach it.

The crystal has seven cells. After three rounds of cross-review, this is what it looks like through the crystallographic lens.

## 5. Anti-Agreement: The GapHunter Is Wrong About DomainSeparatorRegistry

Three reviewers now classify the DomainSeparatorRegistry gap as a process concern. The SpecAuditor says it is not analogous to TLS 1.0's padding oracle because the error is developer-introduced, not attacker-exploitable. The GapHunter says making unregistered usage impossible is "impractical for BLAKE3 domain separators" because Rust cannot prevent raw BLAKE3 calls at the library level.

Both arguments accept the wrong frame. The question is not "can we prevent raw BLAKE3 calls?" The question is "should the protocol's correctness depend on a developer remembering to do something?"

The GapHunter argues that a wrapper adding a registry lookup is impractical. The GapHunter is wrong about what the wrapper does. The wrapper does not look up a registry at runtime. The wrapper is the only API surface for hashing in the protocol. You do not call `blake3::hash(domain, data)`. You call `protocol::domain_hash(DomainSeparator::EpochDerivation, data)`. The `DomainSeparator` enum is the registry. It is a `repr(C)` enum with one variant per registered domain. Adding a new domain requires adding a variant. The variant's existence IS the registration. Raw BLAKE3 is not in the dependency tree of protocol crates; it is re-exported through the domain-separated wrapper only.

This is not a runtime check. This is a type-level constraint. Unregistered usage is not "caught"; it is "unrepresentable." The Rust type system does this trivially. The GapHunter's objection applies to a runtime registry approach. It does not apply to a type-level approach.

I maintain: design defect, not process concern. The fix is a `DomainSeparator` enum that is the sole API for domain-separated hashing. The lint (S2) catches extension crates that might bypass it. Both are needed. The enum is the design fix. The lint is the process guard.

## 6. Savant Re-Sharpening: Naming Domestications

### 6.1 The convergence damping was domesticated to calibration items

The SpecAuditor reclassified the damping coefficients (0.3x, 0.1x) from protocol constants to calibration items. The GapHunter agreed. They are probably right about the coefficients. They are wrong about the state machine transitions.

The NORMAL-to-DAMPED transition is not a calibration item. It is a structural invariant: "when convergence is active, health observations are attenuated." The coefficients are tunable. The existence of the damping mode is not. If the damping mode is a calibration item, an implementor can set the coefficients to 1.0/1.0 (no damping), and the convergence-health feedback loop is back to oscillating. The state machine's existence must be a protocol constant. The coefficients within the DAMPED state can be calibration items with documented defaults and specified recovery bounds.

Sharp edge restored: the DAMPED mode is mandatory. The coefficients within it are tunable.

### 6.2 The seat isolation was domesticated to "body specification work"

The DepAnalyst argues the seat isolation valve belongs in a separate work stream ("seat document body specification"), not in the dependency graph repair list. The SpecAuditor formalized it beautifully (consistency model, stale flags, queue semantics). But both treated it as one feature among many in the seat body.

The isolation valve is not a feature of the seat body. It is the invariant that makes the seat habitable. Without it, the seat is a room whose floor disappears unpredictably. The consistency model (CONSISTENT vs SNAPSHOT) is not a nice-to-have specification element. It is the fundamental contract between the system and the user. Every other seat specification element (authentication, session lifecycle, request types) operates within this contract. The consistency model comes first because everything else depends on knowing which mode the seat is in.

Sharp edge restored: the consistency model is not a seat body feature. It is the seat's structural foundation.

### 6.3 The purpose-specific projections were domesticated to "same work, differently organized"

The SpecAuditor says exposing projections vs exposing dimensions requires the same specification effort. The GapHunter proposes reducing four projections to two. Both treat the projections as a specification cost problem.

The projections are not about cost. They are about coupling direction. When consumers see dimensions, they own the combination logic. When consumers see projections, the pipeline owns the combination logic. The difference is not organizational. It is a coupling direction decision. With dimensions, changing what "good health for source selection" means requires changing matter (the consumer). With projections, it requires changing the health pipeline (the producer). The producer is one place. The consumers are many. Coupling to one place is cheaper to evolve than coupling to many places. The DepAnalyst understood this perfectly (Section 3.2: "opaque projections are stable under formula changes"). The SpecAuditor and GapHunter missed the coupling direction argument by focusing on specification cost.

The GapHunter's two-projection reduction (availability vs throughput) is reasonable for v1 if the projection interface is designed to accommodate future splits. The framework matters more than the initial count. The sharp edge: projections, not dimensions. The count is negotiable. The coupling direction is not.

## 7. Remaining Fracture Lines

Five places the crystal will break under load if left unaddressed.

1. **Matter -> protocol undeclared dependency.** Every reviewer confirmed it across all rounds. Nobody's proposals fix it. It is a one-line declaration plus specification work but it has not been done. The single most concrete structural defect in the lattice.

2. **27 BLOCKING wire-format gaps.** The GapHunter's count went from 59 gaps to 65 after incorporating my proposals (I closed 1, partially addressed 4, opened 6). My proposals addressed emergent behavior risks. The wire-format gaps that halt implementation are untouched. The lattice cannot crystallize if the atomic positions are not specified.

3. **The matter and seat bodies.** These cells are partially hollow. The isolation valve and consistency model give the seat a foundation, but the body (authentication, session lifecycle, request types, error codes) remains unwritten. The matter body (transfer protocol, replication, sync reconciliation) is equally empty. These are not vacancies anymore; they are voids.

4. **Convergence duration bounds.** The GapHunter identified this (NG-5). The seat isolation valve's acceptability depends on how long convergence takes. A 30-second convergence is tolerable with stale data. A 30-minute convergence is not. The spec does not bound expected convergence duration. Without this bound, the seat isolation valve is a mechanism without a contract on when it activates and for how long.

5. **The health pipeline's internal dimensions.** The SpecAuditor stated it: "The health pipeline's internal dimensions are unspecified. Both proposals [projections vs dimensions] require expanding the internal model." Whether we expose projections or dimensions, the pipeline must first define what it computes internally. The current specification has only `effective_health`, a single scalar. The multi-dimensional internal model does not exist yet in specification text.
