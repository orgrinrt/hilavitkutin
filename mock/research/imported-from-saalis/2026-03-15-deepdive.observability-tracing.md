# Deep Dive: Observability & the `tracing` Ecosystem

> Research document. No decisions made. Explores how the Rust `tracing`
> ecosystem works, how it maps to saalis's notification system, and what
> options exist for metrics, persistence, and real-time streaming.

---

## 1. The `tracing` Crate --- Spans, Events, Levels, Structured Fields, Subscribers

The `tracing` crate (from the tokio-rs project) provides application-level
instrumentation built around two primitives: **spans** and **events**.

### Spans

A span represents a period of time with a beginning and an end. Spans can be
entered and exited as execution flows, and they nest naturally to form a tree
of causal context. Every log line emitted inside a span automatically inherits
the span's fields.

```rust
use tracing::{span, Level};

let span = span!(Level::INFO, "download",
    download_id = 42,
    game = "Super Mario World",
);
let _enter = span.enter();
// Everything logged here carries download_id=42 and game="Super Mario World"
```

The `#[instrument]` attribute macro automates this --- it creates a span named
after the function, records parameters as fields (via `fmt::Debug` by default),
and enters/exits the span around the function body:

```rust
#[tracing::instrument(skip(db), fields(download_id = dl.id, game = %dl.title))]
async fn run_download(dl: Download, db: &Database) -> Result<()> {
    tracing::info!("starting download");
    // ...
}
```

**Async caveat:** Holding a span guard (the `_enter` value from `span.enter()`)
across `.await` points produces incorrect traces because the future may be
polled on different threads. `#[instrument]` handles this correctly by using
`Span::in_scope` or `Instrument` from `tracing-futures`. Manual span entry in
async code should use `span.in_scope(|| { ... })` for sync blocks or
`.instrument(span)` for futures.

### Events

An event is a point-in-time occurrence --- the structured equivalent of a log
line. Events happen at a moment; spans wrap a duration.

```rust
tracing::info!(
    download_id = 42,
    duration_ms = elapsed.as_millis() as u64,
    "download complete"
);
```

### Structured Fields

Fields are key-value pairs attached to spans and events. The syntax supports:

| Syntax | Meaning |
|--------|---------|
| `field = value` | Record a literal or variable |
| `field` | Shorthand for `field = field` (local variable) |
| `?value` or `field = ?expr` | Record via `fmt::Debug` |
| `%value` or `field = %expr` | Record via `fmt::Display` |
| `field = tracing::field::Empty` | Declare now, record later via `span.record()` |
| `"dotted.name" = val` | Quoted field names for non-identifier keys |

The trailing positional format string (e.g., `"download complete: {}", name`)
becomes an implicit `message` field.

### Levels

Five severity levels, from most to least verbose:

| Level | Typical use |
|-------|------------|
| `TRACE` | Very fine-grained, per-iteration diagnostics |
| `DEBUG` | Internal state useful during development |
| `INFO` | Lifecycle events, operational milestones |
| `WARN` | Recoverable problems, degraded operation |
| `ERROR` | Failures that affect correctness |

### The Subscriber Model

A `Subscriber` receives notifications about span lifecycle (creation, entry,
exit, close) and events. It decides what to do with the data: format it, filter
it, export it. The key trait methods:

- `enabled(&Metadata)` --- fast pre-filter; if `false`, the span/event is never
  even constructed (zero allocation)
- `new_span(&Attributes)` --- a span was created
- `enter(&Id)` / `exit(&Id)` --- execution entered/exited a span
- `event(&Event)` --- an event occurred
- `record(&Id, &Record)` --- a span's field was recorded after creation

The design separates instrumentation (the macros in library code) from
collection (the subscriber in the binary). Libraries call `tracing::info!()`;
the application decides what happens with those calls.


## 2. `tracing-subscriber` --- Formatting, Filtering, Composition

The `tracing-subscriber` crate provides a composable `Subscriber`
implementation via the **`Layer`** trait and a `Registry` that holds span data.

### The Layer Model

Instead of implementing `Subscriber` from scratch, you compose **layers** on
top of a `Registry`. Each layer sees all span/event notifications and can
filter, transform, or forward them independently.

```rust
use tracing_subscriber::{fmt, registry, layer::SubscriberExt, EnvFilter};

tracing_subscriber::registry()
    .with(fmt::layer().with_target(true))
    .with(EnvFilter::from_default_env())
    .init();
```

### Formatting Layers

The `fmt::Layer` handles human-readable or JSON output:

```rust
// Human-readable to stderr (default)
fmt::layer()
    .with_target(true)
    .with_thread_ids(true)
    .with_file(true)
    .with_line_number(true)

// JSON lines to a file writer
fmt::layer()
    .json()
    .with_writer(non_blocking_writer)
```

The `FormatEvent` trait allows fully custom output formats.

### Filtering: `EnvFilter` and `Targets`

`EnvFilter` parses directives from the `RUST_LOG` environment variable (or
programmatically). The directive syntax:

```
target[span_name{field=value}]=level
```

Each component is optional. Examples:

```bash
RUST_LOG="warn"                            # global warn+
RUST_LOG="saalis=info,saalis::download=debug"  # per-module
RUST_LOG="saalis[run_download]=trace"      # per-span-name
RUST_LOG="saalis[run_download{download_id=42}]=trace"  # per-field-value
```

**Global vs per-layer filtering:** `EnvFilter` implements both `Layer<S>` (for
global filtering of the entire subscriber) and `Filter<S>` (for per-layer
filtering via `.with_filter()`). Per-layer filtering is the key composition
primitive --- it lets one layer log everything at `debug` to a file while
another only prints `warn+` to stderr:

```rust
let file_layer = fmt::layer()
    .json()
    .with_writer(file_writer)
    .with_filter(EnvFilter::new("saalis=debug"));

let stderr_layer = fmt::layer()
    .with_filter(EnvFilter::new("saalis=info"));

registry()
    .with(file_layer)
    .with(stderr_layer)
    .init();
```

The `Targets` filter is a simpler, non-regex alternative to `EnvFilter` that
only filters by target and level --- it is cheaper at runtime and suitable when
field-based or span-based filtering is not needed.

### Adding Directives Programmatically

```rust
let filter = EnvFilter::from_default_env()
    .add_directive("saalis=info".parse()?)
    .add_directive("saalis::download=debug".parse()?);
```


## 3. `tracing-appender` --- File Rotation and Non-Blocking Writers

`tracing-appender` provides two primitives:

### RollingFileAppender

Creates log files that rotate on a time basis:

```rust
use tracing_appender::rolling;

let appender = rolling::daily("/var/log/saalis", "saalis.log");
// Produces files like: saalis.log.2026-03-15
```

Rotation options: `Rotation::MINUTELY`, `HOURLY`, `DAILY`, `NEVER`.

### NonBlocking Writer

`RollingFileAppender` implements `std::io::Write` and blocks on writes.
Wrapping it in `NonBlocking` moves I/O to a dedicated background thread:

```rust
let (non_blocking, _guard) = tracing_appender::non_blocking(appender);
```

The `_guard` is a `WorkerGuard` --- when dropped, it flushes buffered logs.
**This guard must be held for the lifetime of the program**; dropping it early
loses buffered log lines. In `main()`, bind it to a variable that lives until
return.

### Composition for Saalis

```rust
let file_appender = rolling::daily(&config.log_dir, "saalis.log");
let (non_blocking, _guard) = tracing_appender::non_blocking(file_appender);

registry()
    .with(fmt::layer().with_writer(std::io::stderr))              // human stderr
    .with(fmt::layer().json().with_writer(non_blocking))           // JSON file
    .with(EnvFilter::from_default_env())
    .init();
```

On a Batocera appliance with limited storage, daily rotation combined with a
cron job or startup task that deletes logs older than N days keeps disk usage
bounded. Alternatively, if saalis runs under systemd, journal handles rotation
and `stderr` output is sufficient.


## 4. Custom Tracing Layers --- The `notify()` Pattern

This is the most saalis-relevant section. The existing research docs
(`2026-03-09-logging-observability-research.md`) sketch a `NotificationLayer`
that intercepts tracing events bearing a `notify` field and routes them to the
notification bus. Here is how that works mechanically.

### The `Layer` Trait

A custom `Layer<S>` implements callbacks for span/event lifecycle:

```rust
impl<S: Subscriber + for<'a> LookupSpan<'a>> Layer<S> for NotificationLayer {
    fn on_event(&self, event: &tracing::Event<'_>, ctx: Context<'_, S>) { ... }
    fn on_new_span(&self, attrs: &span::Attributes<'_>, id: &span::Id, ctx: Context<'_, S>) { ... }
    fn on_close(&self, id: span::Id, ctx: Context<'_, S>) { ... }
    fn on_enter(&self, id: &span::Id, ctx: Context<'_, S>) { ... }
    fn on_exit(&self, id: &span::Id, ctx: Context<'_, S>) { ... }
}
```

### The Visitor Pattern for Field Extraction

`tracing` never stores field data permanently; you must extract it yourself via
the `Visit` trait. When `on_event` fires, you create a visitor and pass it the
event's fields:

```rust
struct FieldCollector {
    notify: bool,
    audience: Option<String>,
    urgency: Option<String>,
    message: Option<String>,
    fields: BTreeMap<String, String>,
}

impl tracing::field::Visit for FieldCollector {
    fn record_bool(&mut self, field: &tracing::field::Field, value: bool) {
        if field.name() == "notify" {
            self.notify = value;
        }
    }

    fn record_str(&mut self, field: &tracing::field::Field, value: &str) {
        match field.name() {
            "audience" => self.audience = Some(value.to_owned()),
            "urgency" => self.urgency = Some(value.to_owned()),
            "message" => self.message = Some(value.to_owned()),
            other => { self.fields.insert(other.to_owned(), value.to_owned()); }
        }
    }

    fn record_debug(&mut self, field: &tracing::field::Field, value: &dyn std::fmt::Debug) {
        self.fields.insert(field.name().to_owned(), format!("{:?}", value));
    }
}
```

Usage in the layer:

```rust
fn on_event(&self, event: &tracing::Event<'_>, _ctx: Context<'_, S>) {
    let mut collector = FieldCollector::default();
    event.record(&mut collector);

    if collector.notify {
        let audience = collector.audience
            .map(|a| parse_audience(&a))
            .unwrap_or(Audience::Role(UserRole::Admin));
        let notification = Notification {
            title: collector.message.unwrap_or_default(),
            // ... map remaining fields
        };
        let _ = self.bus.send(notification, audience, Persistent::Yes);
    }
}
```

### Ergonomic Macros at the Call Site

With this layer in the subscriber stack, any code in the codebase can trigger a
notification by adding the `notify` field:

```rust
// Normal log --- no notification
tracing::info!(download_id = dl.id, "download started");

// Log + notification
tracing::warn!(
    notify = true,
    audience = "user:42",
    urgency = "high",
    download_id = dl.id,
    "download failed after 3 retries"
);
```

A proc macro could make this even cleaner:

```rust
// Hypothetical convenience macro expanding to tracing::warn! with notify=true
notify_warn!(
    audience = Audience::User(user_id),
    download_id = dl.id,
    "download failed after 3 retries"
);
```

### Span Extensions for Richer Context

The `Layer` trait also receives span lifecycle events. Span **extensions** let
a layer store arbitrary typed data on a span:

```rust
fn on_new_span(&self, attrs: &span::Attributes<'_>, id: &span::Id, ctx: Context<'_, S>) {
    let mut collector = FieldCollector::default();
    attrs.record(&mut collector);
    if let Some(span) = ctx.span(id) {
        span.extensions_mut().insert(collector.fields);
    }
}
```

Later, in `on_event`, the layer can walk the span stack via `ctx.event_span()`
and `span.parent()` to gather inherited context --- for example, adding the
`download_id` from a parent span to a notification even if the event itself
does not repeat it.

### Trade-Off: Layer vs Explicit Calls

The layer approach is elegant but indirect. The existing research doc notes the
tension: "This is an interesting pattern but may be too clever." The
alternative is explicit `notifications.send()` calls at each site. Both can
coexist --- the layer handles the common case (log-and-notify with one macro
call), while complex notifications with multi-audience routing, grouping, or
conditional logic use the explicit bus API.

For saalis's `NotificationPayload`-as-registered-trait design (per the
2026-03-11 architecture), the layer approach works well: the layer checks for
`notify = true`, extracts the `NotificationPayload` type tag from a field,
looks it up in the registry, and dispatches. The payload IS the notification.


## 5. Metrics from Tracing Spans

### `tracing-opentelemetry`

The `tracing-opentelemetry` crate provides a `Layer` that translates tracing
spans into OpenTelemetry spans. It also has a `MetricsLayer` that extracts
metrics (counters, histograms) from span fields and durations, exporting them
via the OpenTelemetry SDK.

```rust
use tracing_opentelemetry::MetricsLayer;
use opentelemetry_sdk::metrics::SdkMeterProvider;

let meter_provider = SdkMeterProvider::builder()
    .with_reader(periodic_reader)
    .build();

registry()
    .with(MetricsLayer::new(meter_provider.clone()))
    .with(fmt::layer())
    .init();
```

With this in place, span durations become histograms and events with numeric
fields can increment counters --- all derived from the same tracing
instrumentation. No separate metrics library needed at the call site.

### Prometheus Integration

Modern guidance recommends the OTLP exporter over the legacy
`opentelemetry-prometheus` crate. Prometheus natively supports OTLP ingestion:

```
tracing spans/events
  -> tracing-opentelemetry MetricsLayer
    -> OpenTelemetry SDK
      -> OTLP exporter
        -> Prometheus (OTLP receiver)
```

### Is This Right for Saalis?

Probably not for v1. Saalis is a single-binary family appliance, not a
microservice mesh. Full OpenTelemetry adds substantial dependency weight
(`opentelemetry`, `opentelemetry-sdk`, `opentelemetry-otlp`, `tonic` for gRPC
transport). The lightweight alternative from the existing research --- explicit
`Instant::now()` / `elapsed()` timing at key points, recorded as span fields
--- gives 80% of the value at 0% of the dependency cost.

If operational metrics become necessary later (e.g., for the Doctor subsystem's
health checks), a minimal custom `Layer` that counts events by level and target,
tracks span durations into fixed-bucket histograms, and exposes them at a
`/metrics` endpoint is simpler than the full OTel stack.

### The `metrics` Crate Alternative

The `metrics` crate provides a facade (like `log` is to `tracing`) for
counters, gauges, and histograms. `metrics-exporter-prometheus` serves a
Prometheus scrape endpoint. This is lighter than OpenTelemetry but requires
separate instrumentation from tracing. For saalis, co-locating metrics with
tracing (via a custom layer that counts events) avoids dual instrumentation.


## 6. Structured Logging to SQLite

### The Concept

A custom `Layer` that writes events to SQLite rather than (or in addition to)
a file. This creates a queryable log store --- filter by level, target, time
range, or any structured field.

### Schema Design

```sql
CREATE TABLE log_events (
    id          INTEGER PRIMARY KEY,
    timestamp   TEXT NOT NULL,
    level       TEXT NOT NULL,       -- TRACE/DEBUG/INFO/WARN/ERROR
    target      TEXT NOT NULL,       -- module path
    message     TEXT,
    span_id     INTEGER,            -- FK to spans table if in a span
    fields_json TEXT                 -- JSON object of structured fields
);

CREATE TABLE log_spans (
    id          INTEGER PRIMARY KEY,
    name        TEXT NOT NULL,
    target      TEXT NOT NULL,
    parent_id   INTEGER REFERENCES log_spans(id),
    opened_at   TEXT NOT NULL,
    closed_at   TEXT,
    fields_json TEXT
);

CREATE INDEX idx_log_level ON log_events(level);
CREATE INDEX idx_log_target ON log_events(target);
CREATE INDEX idx_log_time ON log_events(timestamp);
```

### Implementation Sketch

```rust
struct SqliteLogLayer {
    tx: std::sync::mpsc::Sender<LogCommand>,
}

enum LogCommand {
    Event { timestamp: String, level: String, target: String, message: String, fields: String },
    SpanOpen { id: u64, name: String, target: String, parent: Option<u64>, fields: String },
    SpanClose { id: u64, closed_at: String },
}
```

The layer sends commands over a channel; a dedicated writer thread batches
inserts for performance. This mirrors `tracing-appender`'s non-blocking
pattern, keeping the hot path (the `on_event` call) to a channel send (~40 ns).

### Querying for Debugging

```sql
-- All errors in the last hour
SELECT * FROM log_events
WHERE level = 'ERROR' AND timestamp > datetime('now', '-1 hour');

-- All events for a specific download
SELECT * FROM log_events
WHERE json_extract(fields_json, '$.download_id') = 42
ORDER BY timestamp;

-- Span durations for enrichment tasks
SELECT name, fields_json,
       (julianday(closed_at) - julianday(opened_at)) * 86400 AS duration_secs
FROM log_spans
WHERE name = 'enrichment_batch'
ORDER BY duration_secs DESC;
```

### Practical Considerations

- **Storage cost:** A busy saalis instance might produce 10,000--50,000 events
  per day. At ~200 bytes per row (including JSON fields), that is 2--10 MB/day
  uncompressed. With a 30-day retention window and periodic `DELETE` + `VACUUM`,
  this fits comfortably in the cold store.

- **Write amplification:** SQLite WAL mode handles concurrent reads (from the
  admin log viewer) and writes (from the logging thread) well. Batching inserts
  in transactions of 100--500 rows keeps WAL checkpoint overhead low.

- **Separate database:** The log store should be a separate SQLite file from
  the main entity database. This avoids log writes contending with entity
  queries and allows independent `VACUUM` / rotation.

- **Level gating:** Only persist events at `INFO` or above to SQLite; `DEBUG`
  and `TRACE` go to the file/stderr layers only. This keeps the database small
  and fast.

### Fits the Saalis Architecture

Saalis already uses SQLite as its primary store and has a hot/cold storage
concept. Structured logs in SQLite are the "cold store for diagnostics" ---
queryable through the admin UI, purgeable on a schedule, and accessible without
SSH.


## 7. Performance --- Cost of Tracing in Hot Paths

### When Disabled (Static Level Filtering)

`tracing` supports compile-time level gates via Cargo features:

```toml
[dependencies]
tracing = { version = "0.1", features = [
    "max_level_debug",          # debug builds: DEBUG and above
    "release_max_level_warn",   # release builds: WARN and above
] }
```

When a level is statically disabled, the macro expands to nothing --- zero
runtime cost, no code in the binary. The `STATIC_MAX_LEVEL` constant is checked
at compile time. This means `trace!()` and `debug!()` calls are completely free
in release builds if `release_max_level_info` or stricter is set.

### When Enabled but Filtered by Subscriber

If a level is compiled in but the subscriber's `enabled()` returns `false`, the
cost is a single atomic load of the cached filter result (~1--2 ns). No field
recording, no allocation, no string formatting.

### When Enabled and Recorded

Measured costs from production systems:

| Operation | Approximate cost |
|-----------|-----------------|
| Disabled span/event (static filter) | 0 ns (compiled out) |
| Disabled span/event (subscriber filter) | 1--2 ns |
| Enabled event, simple fields | 50--200 ns |
| Span creation + enter + exit | 100--400 ns |
| Channel send (non-blocking writer) | 20--40 ns |
| Full span with 5+ fields, formatting | 300--800 ns |

The TiKV project (a high-performance distributed KV store) found that naive use
of `tokio-tracing` (the previous name for `tracing`) reduced throughput by more
than 50% in tight loops. Their solution: custom span collection at ~20 ns per
span using `crossbeam::channel`. The lesson: **do not instrument tight inner
loops; instrument the boundaries** (function entry, batch boundaries, I/O
calls).

### Recommendations for Saalis

1. **Use `release_max_level_info`** --- eliminate TRACE/DEBUG from release
   builds entirely. Development builds keep all levels.

2. **Never instrument per-item loops** --- e.g., do not put `#[instrument]` on
   a function called once per ROM file during a full library scan of 10,000
   files. Instrument the batch boundary instead.

3. **Use `skip_all` liberally** --- `#[instrument(skip_all, fields(batch_size = batch.len()))]`
   avoids Debug-formatting every parameter.

4. **Feature-gate verbose instrumentation** --- for the most performance-
   sensitive paths, guard behind a `tracing-verbose` Cargo feature:

   ```rust
   #[cfg_attr(feature = "tracing-verbose", tracing::instrument(skip_all))]
   fn hot_path() { ... }
   ```

5. **Profile before worrying** --- saalis's hot paths are I/O-bound (HTTP
   fetches, disk writes, SQLite queries), not CPU-bound. A 200 ns span on
   a function that does a 50 ms HTTP request is 0.0004% overhead.


## 8. Integration with HTMX SSE --- Streaming Logs to the Admin Panel

### Architecture

The admin log viewer connects to an SSE endpoint. A custom tracing `Layer`
forwards events to a `tokio::broadcast::channel`, and the SSE handler
subscribes each connected admin to that channel.

```
tracing::info!(...)
  -> NotificationLayer::on_event()     (in subscriber stack)
  -> broadcast::Sender<LogEvent>::send()
  -> SSE handler receives via broadcast::Receiver
  -> Sse::new(stream.map(|e| Event::default().data(render_html(&e))))
  -> HTMX sse-swap="log-event" appends to #log-container
```

### The Broadcast Layer

```rust
struct BroadcastLayer {
    tx: broadcast::Sender<LogEvent>,
}

struct LogEvent {
    timestamp: String,
    level: tracing::Level,
    target: String,
    message: String,
    fields: BTreeMap<String, String>,
}

impl<S: Subscriber + for<'a> LookupSpan<'a>> Layer<S> for BroadcastLayer {
    fn on_event(&self, event: &tracing::Event<'_>, ctx: Context<'_, S>) {
        let mut collector = FieldCollector::default();
        event.record(&mut collector);
        let log_event = LogEvent {
            timestamp: chrono::Utc::now().to_rfc3339(),
            level: *event.metadata().level(),
            target: event.metadata().target().to_owned(),
            message: collector.message.unwrap_or_default(),
            fields: collector.fields,
        };
        // Ignore send errors (no receivers = no admin connected)
        let _ = self.tx.send(log_event);
    }
}
```

### SSE Endpoint

```rust
async fn admin_log_stream(
    State(state): State<AppState>,
    Query(params): Query<LogFilterParams>,
) -> Sse<impl Stream<Item = Result<Event, Infallible>>> {
    let mut rx = state.log_broadcast.subscribe();
    let stream = async_stream::stream! {
        while let Ok(log_event) = rx.recv().await {
            if params.matches(&log_event) {
                let html = render_log_line(&log_event);
                yield Ok(Event::default()
                    .event("log-event")
                    .data(html));
            }
        }
    };
    Sse::new(stream).keep_alive(
        axum::response::sse::KeepAlive::new()
            .interval(Duration::from_secs(15))
    )
}
```

### HTMX Client

```html
<div hx-ext="sse" sse-connect="/admin/logs/stream?level=info">
    <div id="log-container"
         sse-swap="log-event"
         hx-swap="beforeend scroll:bottom">
    </div>
</div>

<!-- Client-side level filter (pure CSS or minimal JS) -->
<select onchange="updateSSEFilter(this.value)">
    <option value="error">Errors only</option>
    <option value="warn">Warnings+</option>
    <option value="info" selected>Info+</option>
    <option value="debug">Debug+</option>
</select>
```

The HTMX SSE extension handles automatic reconnection with exponential backoff.
Changing the filter level can either reconnect with new query parameters or
use CSS class-based hiding for instant client-side filtering without
reconnection.

### Convergence with the Notification System

The log streaming layer and the notification layer share the same architecture:
a custom `Layer` that forwards to a `broadcast::channel`, consumed by SSE
handlers. They could even be the same channel with different SSE event types
(`log-event` vs `notification`), or separate channels with separate layers for
clean separation of concerns. Separate is probably better --- the log stream
has higher volume and different filtering needs than notifications.

Both SSE streams can share a single HTMX SSE connection by using different
`sse-swap` event names on different DOM targets:

```html
<div hx-ext="sse" sse-connect="/events/stream">
    <!-- Notifications -->
    <div sse-swap="notification" hx-swap="beforeend" id="toast-container"></div>
    <span sse-swap="unread-count" id="notif-badge"></span>

    <!-- Log viewer (admin only, conditionally rendered) -->
    <div sse-swap="log-event" hx-swap="beforeend" id="log-container"></div>
</div>
```


## 9. Request Tracing Through the Scheduler/Work-Unit Pipeline

Even though saalis is a single binary, **internal request tracing** through the
scheduler pipeline is valuable. A work unit passes through multiple stages:

```
User action (web request)
  -> WorkUnit created (Curator/Housekeeper/Doctor)
    -> Scheduler queues it
      -> Worker picks it up
        -> Executes (may spawn child work units)
          -> Completes/fails
```

### Correlation via Span Hierarchy

Each stage creates a child span, forming a causal tree:

```rust
// Web handler creates the root span
#[instrument(fields(request_id = %Uuid::new_v4()))]
async fn trigger_enrichment(game_id: EntityId) {
    let work_unit = WorkUnit::new::<EnrichGame>(game_id);

    // Scheduler picks up the span context
    scheduler.submit(work_unit, Span::current());
}

// Scheduler worker creates a child span linked to the request
fn execute_work_unit(unit: WorkUnit, parent: Span) {
    let span = info_span!(parent: &parent, "work_unit",
        unit_id = %unit.id,
        unit_type = %unit.type_name(),
    );
    let _enter = span.enter();
    // All logs here carry request_id from the parent
    unit.execute();
}
```

### Work Unit ID as Correlation Key

Every work unit gets a unique ID. Recording it as a span field means every log
line during that unit's execution is queryable:

```sql
-- In the SQLite log store: all events for work unit #abc123
SELECT * FROM log_events
WHERE json_extract(fields_json, '$.unit_id') = 'abc123'
ORDER BY timestamp;
```

### Span Context Propagation Without Distributed Tracing

In a distributed system, trace context is propagated via HTTP headers
(W3C Trace Context, B3, etc.). Saalis does not need this, but the concept maps
cleanly to internal pipeline stages:

- **Trace ID** = request ID from the web handler (or a synthetic ID for
  scheduler-initiated work)
- **Span ID** = per-stage span within the trace
- **Parent Span ID** = links child work units to the work unit that spawned them

The `tracing` crate handles this natively --- `parent: &parent_span` on span
creation establishes the link. No manual ID plumbing needed.

### DAG Scheduler Visibility

For the DAG scheduler (where work units have dependencies), span context reveals
the execution graph:

```
[request_id=abc] trigger_enrichment
  [unit_id=001] FetchMetadata
    [unit_id=002] DownloadCoverArt  (depends on 001)
    [unit_id=003] MatchCompatList   (depends on 001)
  [unit_id=004] UpdateEntity        (depends on 002, 003)
```

Each unit carries a reference to its parent span. When `on_close` fires for a
span with `unit_id`, the layer can record the duration --- giving a timeline of
the DAG execution without any explicit timing code.


## 10. The `tracing` + `log` Compatibility Layer

### The Problem

Many Rust crates use the `log` crate (the older, simpler facade). If saalis
depends on a library that calls `log::warn!(...)`, those messages are invisible
to the tracing subscriber by default.

### `tracing-log`: Bridging the Gap

The `tracing-log` crate provides two directions:

#### `log` -> `tracing` (primary use case)

`LogTracer` is a `log::Log` implementation that converts `log::Record`s into
`tracing::Event`s:

```rust
use tracing_log::LogTracer;

LogTracer::init()?;
// Now all log::info!() calls from dependencies appear as tracing events
```

`tracing-subscriber`'s `fmt::Subscriber` enables this automatically when the
`tracing-log` feature is active (it is by default).

#### `tracing` -> `log` (for legacy subscribers)

The `tracing` crate itself has a `log` feature (enabled by default) that emits
`log::Record`s for each tracing event when no tracing subscriber is active.
This is a fallback: if a library uses `tracing::info!()` but the binary only
sets up a `log` subscriber (e.g., `env_logger`), the events still appear.

The `log-always` feature makes tracing emit log records even when a tracing
subscriber IS active --- useful if you need both systems receiving events
simultaneously.

### `AsTrace` / `AsLog` Conversion

`tracing-log` provides traits to convert between the two crate's types:

```rust
use tracing_log::{AsTrace, AsLog};

let tracing_level: tracing::Level = log::Level::Warn.as_trace();
let log_level: log::Level = tracing::Level::WARN.as_log();
```

### What This Means for Saalis

Saalis uses `tracing` throughout --- correct choice. Adding `LogTracer::init()`
(or relying on `tracing-subscriber`'s default) captures `log`-based output
from dependencies (e.g., `rusqlite`, `ureq`, `axum` internals) into the same
subscriber pipeline. No code changes in dependencies needed.

One consideration: some dependencies are chatty at `DEBUG` level. The
`EnvFilter` handles this: `RUST_LOG="saalis=debug,rusqlite=warn,ureq=warn"`
keeps dependency noise in check.


## Summary of Recommendations for Saalis

| Area | Recommendation |
|------|---------------|
| Subscriber setup | Layered: human stderr + JSON file + optional SQLite |
| Filtering | `EnvFilter` with per-module defaults; `release_max_level_info` |
| File output | `tracing-appender` with daily rotation, non-blocking writer |
| Notification bridge | Custom `NotificationLayer` checking `notify` field |
| Log viewer | Broadcast layer -> SSE -> HTMX admin panel |
| Metrics | Lightweight custom layer; defer OTel until needed |
| SQLite logging | Separate database, INFO+ only, 30-day retention |
| Span strategy | `#[instrument]` on work-unit boundaries, not inner loops |
| Dependency logs | `LogTracer` captures `log` crate output automatically |
| Performance | Static level gating in release; profile before optimizing |

### Crate Dependencies

| Crate | Purpose | Approximate size |
|-------|---------|-----------------|
| `tracing` | Core instrumentation macros | Already present |
| `tracing-subscriber` | Registry, fmt layer, EnvFilter | Already present |
| `tracing-appender` | File rotation, non-blocking I/O | Small (~3 KB) |
| `tracing-log` | `log` crate compatibility | Small (~5 KB), likely already pulled in |
| `chrono` | Timestamps in custom layers | Likely already present |

No need for `opentelemetry`, `tonic`, `prometheus`, or other heavy crates
unless the monitoring requirements grow beyond a single-box appliance.
