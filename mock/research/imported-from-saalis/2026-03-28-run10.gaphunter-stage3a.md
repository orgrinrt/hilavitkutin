## 1. Savant's Mechanisms from an Implementer's Perspective

### 1.1 Convergence Damping Coefficient (Section 3.1)

**Testability:** Medium. The 0.3x/0.1x damping weights are concrete enough to implement, but testing requires simulating convergence-induced topology changes and measuring EMA recovery. Unit-testable in isolation (apply damping mask, verify EMA weights change). Integration-testable only if the test harness can inject convergence state into the HealthPipeline. The convergence state is a boolean ("any gate open"), which is simple to mock.

**Failure modes:** The mechanism has a silent failure mode the savant did not address. If the convergence damping mask is active and a genuine peer failure occurs (not convergence-induced transient disruption but actual hardware death), the HealthPipeline detects it 3.3x slower (1/0.3) on the fast EMA. During convergence, this is exactly when you need fast failure detection most, because gateway election is in flux and a dead peer might be the elected gateway. The damping coefficient trades detection speed for stability. That tradeoff is probably correct for transient disruptions but dangerous for hard failures during convergence.

**Minimum addition to close the gap:** The damping mask needs a bypass for zero-observation events. If a peer sends zero heartbeats for 3 consecutive periods (15 seconds in Active mode), the damping mask should not apply. Zero heartbeats is not a "convergence-induced perturbation"; it is an absent peer. The mask should attenuate reduced delivery ratios (e.g., 60% instead of 95%), not absent peers.

**Complexity cost:** Low. One boolean check (convergence active?), two multiplier constants, one bypass condition. Roughly 20 lines of implementation logic in the HealthPipeline.

### 1.2 Scar Severity Tiering (Section 3.2)

**Testability:** High. The tier assignments, weights, and decay durations are fully specified. A unit test can: create 15 Tier-1 scars, compute weighted_scar_count, verify it equals 7.5, verify it is below the exponential threshold. Straightforward.

**Failure modes:** The tier assignment itself is a new classification problem. Who decides which scar type maps to which tier? The savant lists six scar types across three tiers. But the mesh currently has more scar types than these six (the scar attribution system in verkko-mesh Section 5.1 lists PartitionAbsence, IntegrityFailure, and several others). An implementer needs the complete mapping, not an illustrative subset.

**Minimum addition:** Provide the exhaustive scar-type-to-tier mapping for all scar attribution types defined in the spec. If the spec adds a new scar type, the tier mapping must be updated simultaneously. This is a maintenance burden but a small one.

**Complexity cost:** Low-medium. The weighted_scar_count formula changes from a count to a weighted sum. The decay becomes per-tier instead of global. Storage needs to track scar tier alongside scar type (or derive tier from type at computation time, which is cleaner).

### 1.3 Lattice Rigidity Layer / Prioritized Wire Contracts (Section 3.3)

**Testability:** This is not a mechanism; it is a prioritization framework. The testability question applies to the wire contracts themselves, not to the ordering. Wire contracts are inherently testable: they produce byte sequences that can be compared against test vectors.

**Failure modes:** The prioritization into three tiers (grain boundary, cross-document, internal) is sound in principle. But the savant's estimate of "~8 concepts, ~200 lines" for Priority 1 is optimistic. HeartbeatExtension alone touches six consumers and requires specifying extension field offsets, sizes, and versioning. ConvergenceGate requires four boolean predicates with specified evaluation order. DomainSeparatorRegistry entries require the complete registry (which is what C-10 already flagged as missing). I estimate Priority 1 at ~400-500 lines, not ~200.

**Complexity cost:** The prioritization framework itself costs nothing. The wire contracts cost significant specification effort, but that cost exists regardless of ordering. The ordering helps by ensuring the highest-risk gaps close first.

### 1.4 Seat Isolation Valve (Section 3.4)

**Testability:** Medium-high. The snapshot/live split is a well-known pattern (MVCC, copy-on-write). Testable by: start convergence, issue seat query, verify it returns snapshot data, complete convergence, issue same query, verify it returns live data. The deferred-write queue is also testable: issue seat mutation during convergence, verify it is queued, complete convergence, verify mutation is applied.

**Failure modes:** Three that the savant did not address:

1. **Snapshot staleness visibility.** The seat sees stale data but does not know it is stale. If a user browses their library during a 5-minute convergence, they see data that is 5 minutes old. They might initiate actions based on stale state (e.g., "download game X" but game X was already removed by a mesh peer during convergence). The deferred-write queue processes after convergence, by which time the user's intent may be invalid. The seat needs a "staleness indicator" or the deferred writes need conflict resolution.

2. **Snapshot cost.** "Takes a snapshot of the current entity metadata and stash status" is easy to say, hard to size. If the entity table has 50,000 entries (large game library across multiple platforms), the snapshot is non-trivial in memory. On a Batocera appliance with limited RAM (the target platform), this could be a problem. The spec needs to bound the snapshot cost or define a lazy-snapshot mechanism.

3. **Queue ordering.** If the user issues 5 mutations during convergence, they are queued. After convergence, they replay. But convergence may have changed the state such that mutation #3 depends on state that mutation #2 would have created in real-time but that convergence has invalidated. The replay needs to be validated against post-convergence state, not blindly applied.

**Minimum addition:** (a) Seat responses during convergence carry a `stale: bool` flag. (b) Deferred mutations are validated against post-convergence state before replay; invalid mutations return an error to the seat on reconnect or next request. (c) Snapshot is implemented as a read-only reference to the pre-convergence hot store state, not a full copy (the hot store already supports this if it uses any form of versioning).

**Complexity cost:** Medium-high. The snapshot mechanism, deferred queue, and post-convergence validation are three separate subsystems. This is the most complex of the savant's four proposals. Worth it for UX, but the implementation budget is larger than the savant's framing suggests.

### 1.5 Purpose-Specific Health Projections (Section 4.3)

**Testability:** High. Four functions, each producing a scalar from a known internal vector. Fully unit-testable. The weights for each projection are the specification gap: the savant says "weighted toward throughput, packet loss" but does not specify the weights. An implementer needs the exact projection formula, not the dimension names.

**Failure modes:** The projections create a new tuning surface. Four projection functions, each with multiple weights, each consumed by a different subsystem. Calibrating these weights requires empirical data from real meshes. The spec can provide initial values, but they will need tuning. This is not a failure mode per se, but it means the projections are a v2 concern, not a v1 concern. For v1, the single scalar is adequate. The projection framework should be designed in but the per-projection weights can be calibrated later.

**Minimum addition:** Specify the projection formulas with placeholder weights and a calibration protocol. For v1, all projections can return the same scalar (the current health score). The framework exists but the differentiation is deferred.

**Complexity cost:** Low if deferred (just four wrapper functions returning the same value). Medium if fully specified (four distinct weighted-sum formulas with documented weight rationale).

## 2. Taking the Ideas Further

### 2.1 Convergence Damping Opens a Coordination Gap

The savant's damping coefficient suppresses health perturbations during convergence. But the damping mask is per-peer: each peer decides independently whether convergence is active and whether to apply the mask. In a partition-heal scenario, different peers enter convergence at different times (they discover the partition heal at different moments, depending on which pairs reconnect first). This means:

- Peer A applies the damping mask at T=0 (it detected the heal first).
- Peer B applies the damping mask at T=5s (it detected the heal when Peer A's heartbeat arrived).
- Peer C has not yet detected the heal and is not damping.

From Peer C's perspective, Peers A and B are exhibiting degraded health (convergence-induced disruption). Peer C does not apply the damping mask because it does not know convergence is happening. Peer C's source selection excludes A and B. When Peer C finally detects the heal (T=10s), it applies its own mask, but its health observations of A and B during [0, 10s] are already recorded at full weight.

**Gap opened:** The damping mask needs to be communicated, not just locally applied. Either: (a) the heartbeat carries a "convergence active" flag that remote observers use to apply damping to that peer's health observations, or (b) the damping mask applies retroactively (recompute EMA without the pre-mask observations). Option (a) is simpler and sufficient. The heartbeat extension already needs a field for gossip_mode; a convergence_active flag is one more bit.

### 2.2 Scar Tiering Opens a Classification Governance Gap

The savant's three-tier system is sound for the current scar types. But when new scar types are added (and they will be; the scar system is designed for extensibility), who assigns the tier? The spec needs a decision framework, not just a mapping table. Otherwise, the same classification dispute that exists for severity ratings (is IntegrityFailure transient or concerning?) will recur for every new scar type.

**Gap opened:** Define the tier assignment criteria, not just the tier assignments. Proposed criteria:
- Tier 1 (transient): the cause is environmental and self-correcting (hardware flap, network congestion).
- Tier 2 (concerning): the cause is behavioral but may be accidental (delivery failure, resource exhaustion).
- Tier 3 (severe): the cause is a protocol violation or security breach (revocation non-compliance, crypto violation).

This framework lets an implementer classify new scar types without a spec revision.

### 2.3 The Seat Isolation Valve Implies a Transaction Model

The savant proposes deferred writes during convergence. This is a transaction: the seat's mutations are buffered, then replayed (committed) or rejected (rolled back) after convergence. But the spec has no transaction concept. Introducing deferred-write semantics without a transaction model creates an implicit, undocumented transactional guarantee that will confuse implementers.

**Gap opened:** Either: (a) make the transaction model explicit (seat mutations during convergence form a tentative batch, committed or rejected atomically after convergence), or (b) reject seat mutations during convergence with a MESH_CONVERGING error code and let the seat retry. Option (b) is simpler and avoids the transaction model entirely. The user sees "please wait" instead of stale-but-responsive data. For a household appliance, "please wait for 30 seconds" is acceptable. For a 5-minute convergence, it is not. The choice depends on expected convergence duration, which the spec should bound.

### 2.4 Wire Contract Prioritization Misses the Bootstrap Problem

The savant prioritizes wire contracts by structural importance (grain boundaries first). From an implementer's perspective, the prioritization should also consider the implementation dependency order. The first thing an implementer writes is the handshake. The handshake requires: Noise prologue format (P-3), SchemaSync (P-4), and connection ID (P-1). None of these are grain boundary concepts. They are internal to verkko-protocol. But they are the first bytes on the wire, and without them, no other protocol interaction is possible.

**Gap opened:** The savant's Priority 1 (grain boundary bonds) and the implementer's Priority 1 (first bytes on the wire) are different sets. The spec needs both orderings: structural priority (for design completeness) and implementation priority (for build order). The ops document's build order addresses this partially, but it does not reference specific wire contracts.

## 3. Mechanisms That Will Not Work in Practice

### 3.1 The Seat Snapshot Mechanism Conflicts with the Sync Core Architecture

The savant proposes that the HostPeer maintains two state views: live_state (mutable, mesh) and snapshot_state (read-only, seat). The saalis architecture rule is "sync core, async shell." The hot store (hilavitkutin columnar) serves all reads. The cold store (SQLite) is write-only during normal operation.

Taking a snapshot of the hot store requires either:
- A copy of the columnar data (expensive on limited RAM).
- A versioned hot store with read-at-snapshot semantics (MVCC).

The saalis architecture does not specify MVCC for the hot store. Adding MVCC to hilavitkutin is a significant architectural addition that crosses the verkko/saalis boundary. The verkko spec cannot mandate hilavitkutin internals.

**Preserved insight:** The seat must not block on convergence. The savant is right about the problem.

**Fixed implementation:** Instead of a snapshot, the HostPeer serves seat reads from the hot store without taking a lock. The hot store is a read-optimized structure; reads should not block on mesh writes. The contention point the savant identified (control log merge holding the state lock for 30 seconds) is a lock granularity problem, not a snapshot problem. The fix is: mesh operations write to a staging area during convergence. The hot store continues serving reads from its current state. When convergence completes, the staging area is merged into the hot store in a single atomic swap. Reads are never blocked. The data the seat sees is not stale; it is the pre-convergence truth, which is the same as the savant's snapshot but without the copy.

This is the write-behind pattern. It requires that mesh operations during convergence do not read their own writes from the hot store (they read from the staging area). This is a constraint on the convergence implementation, not on the seat.

### 3.2 Purpose-Specific Health Projections Create an Undertested Surface

The savant proposes four projection functions. Each is consumed by exactly one subsystem. Each has different weights. The interaction between projection weights and subsystem behavior is combinatorial: changing a weight in `health.for_gateway_election()` changes which peer becomes gateway, which changes the network topology, which changes the health observations, which feeds back into all four projections.

This is not a mechanism that will not work. It is a mechanism that will work but is impractical to tune. Four projection functions with (say) 3 weights each = 12 tuning parameters. The interaction space is 12-dimensional. No household mesh provides enough empirical data to calibrate 12 parameters.

**Preserved insight:** Different consumers need different views of health.

**Fixed implementation:** Two projections, not four. (a) `health.for_availability()` = weighted toward uptime and reachability. Used by gateway election and convergence gating. (b) `health.for_throughput()` = weighted toward delivery ratio and recent performance. Used by source selection and metabolic state. Two projections, 2-3 weights each, manageable tuning surface. If empirical data later shows that gateway election and convergence gating need different views, split then.

## 4. Specific Disagreement: The Savant Is Wrong About DomainSeparatorRegistry Classification

The savant concedes priority (moved from #1 to #4) but insists on classification: "This is a design defect, not merely a process concern. A rule that says 'you must register' can be ignored. A mechanism that makes unregistered usage impossible cannot."

I disagree. The savant is asking the protocol to make a category error impossible at the type level. This is admirable in the abstract but impractical for BLAKE3 domain separators. Domain separators are byte strings passed to a hash function. The only way to make unregistered usage "impossible" is to wrap BLAKE3 in a function that checks a registry at every call site. This means:

1. Every BLAKE3 call goes through a wrapper, adding a registry lookup to every hash computation.
2. The registry must be populated before any hash is computed, which creates an initialization ordering dependency.
3. The wrapper must be the only way to call BLAKE3. If any call site uses raw BLAKE3 (which Rust cannot prevent at the library level without removing BLAKE3 from the dependency tree), the mechanism is bypassed.

The savant's TLS 1.0 analogy is inapt. TLS 1.0's padding oracle was a protocol mechanism that amplified an error. The domain separator gap is not a mechanism; it is an omission. The protocol does not amplify anything. The developer calls BLAKE3 with the wrong domain separator and gets the wrong hash. There is no amplification, only a mistake. The correct fix is defense-in-depth: (a) a lint that catches unregistered domain separators at compile time (which the savant also proposes as S2), (b) test vectors for every registered separator (which I flagged as C-10), (c) documentation that says "do not call BLAKE3 directly." Together, these make the mistake unlikely. Making it "impossible" is not achievable without a language-level capability system that Rust does not have.

Classification: process concern with defense-in-depth mitigation. Not a design defect.

## 6. New Gaps Identified From the Savant's Proposals

### NG-1. Convergence damping mask communication [AMBIGUOUS]

The damping mask is locally applied but convergence detection is not simultaneous across peers. Remote observers need to know a peer is in convergence to dampen their health observations of that peer. Requires a heartbeat flag or equivalent.

**Minimum fix:** Add a `convergence_active` bit to the heartbeat extension.

### NG-2. Scar tier assignment criteria [AMBIGUOUS]

The savant specifies six scar-to-tier mappings but not the decision framework for new scar types. Without criteria, future scar types default to Tier 2 (the safe middle), which defeats the purpose of tiering.

**Minimum fix:** Document the tier assignment criteria (environmental/self-correcting = Tier 1, behavioral/accidental = Tier 2, protocol violation/security = Tier 3).

### NG-3. Seat deferred-write conflict resolution [MISSING-ERROR]

If seat mutations are queued during convergence and replayed after, mutations that conflict with post-convergence state need a resolution policy. No policy is specified.

**Minimum fix:** Deferred mutations are validated against post-convergence state. Invalid mutations return an error to the seat.

### NG-4. Seat snapshot staleness indicator [AMBIGUOUS]

The seat serves stale data during convergence but does not indicate staleness to the user. The seat protocol needs a flag or field to communicate data freshness.

**Minimum fix:** Seat responses carry a `stale: bool` field. True during convergence.

### NG-5. Convergence duration bound [AMBIGUOUS]

The seat isolation valve's acceptability depends on convergence duration. A 30-second convergence is tolerable with stale data. A 30-minute convergence is not. The spec does not bound expected convergence duration.

**Minimum fix:** "Convergence wavefront deadline defaults to 120 seconds. Maximum configurable value: 600 seconds. If convergence exceeds the deadline, the system proceeds with available state."

### NG-6. Health projection weight specification [UNTESTABLE]

The savant proposes purpose-specific health projections but does not specify the projection weights. Without weights, two implementations will produce different projection values, leading to different gateway elections and source selections.

**Minimum fix:** Specify initial weights for each projection function. Mark them as calibration parameters with documented defaults.
