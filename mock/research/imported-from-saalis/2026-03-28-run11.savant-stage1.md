# Run 11: Cartographic Savant -- Stage 1

**Date:** 2026-03-28
**Input:** Seven verkko design documents + Run 10 project lead synthesis (23 actions)
**Scope:** Post-action-plan territory survey. What is still blank after the 23 actions land.
**Role:** Cartographic Savant. The map is the system. Gaps in the map are gaps in the system.

## 1. Territory Survey

### verkko-crypto: Well-Mapped

The best-surveyed territory. Every road has a name and a measured distance. The NonceSafety three-layer defense, the dual-AEAD registry, the EpochRatchet/MicroEpochRatchet chain, the BroadcastEncryption envelope -- these are roads with kilometer markers. The DomainSeparatorRegistry is a customs office with a stamped register. You will not get lost here.

**Unmapped regions remaining after Run 10:**
- Key serialization format (how bytes are laid out in memory/on disk for DK, DEK, CIK). Run 10 P3-1 covers this, but the territory has no landmarks yet.
- The CSPRNG self-test "3 samples" criterion. Three samples of what length, what distance metric for "non-repeating"? A traveler following these directions could implement three different tests.
- FrameKey derivation ordering constraint: the spec says `K_frame = HKDF-Expand(prk=K_micro_M, info=nonce, len=32)`, but the nonce is a 96-bit value that comes from NonceSource, which is on a different thread than the key derivation. The handoff protocol between NonceSource's thread and the AEAD thread is unspecified. The double-write persist fires before re-encryption batches, but not before individual frame encryption. The gap: what is the causal ordering between "issue nonce" and "derive frame key" across threads?

### verkko-protocol: Lightly Mapped

Major roads exist (wire format, LEDBAT, heartbeat, metabolic state machine). But the interior villages are blank. Connection ID generation: unspecified. Stream multiplexing: no wire format. SchemaSync: no message format. Cipher negotiation: absent. SACK format: absent. Flow control: absent.

Run 10 P1-3 and P3-2 cover the named gaps. But there is a deeper problem: the SansIO contract is stated but never specified. The spec says "consumer drives I/O" and "all buffers are arena-allocated from a MemoryProvider," but there is no MemoryProvider interface, no arena allocation API, no specification of what the consumer's polling loop looks like, no event type enumeration, no error model for the sans-IO interface. The protocol is specified as a state machine whose API surface is blank.

**Unmapped regions remaining after Run 10:**
- The SansIO consumer interface. No action in Run 10 covers this.
- MemoryProvider contract. No action in Run 10 covers this.
- The reliability mode selection: "reliable ordered" vs "unreliable datagram" per-stream. How is the mode selected? Per-stream at creation? Switchable? What happens to in-flight data on mode change?
- The event loop's edge-triggered epoll behavior under io_uring upgrade path. The two I/O strategies have different backpressure characteristics. No spec text covers which guarantees hold across both.

### verkko-mesh: Well-Mapped (with one major blank interior)

The most densely specified territory. Enrollment, dominance cascade, health pipeline, convergence, gossip, gateway election, resource governance, scar provenance -- all mapped with precision. The formal foundation section reads like a geodetic survey.

**Unmapped regions remaining after Run 10:**
- Channel lifecycle. Channels are created, accessed, revoked. But who creates a channel? Is there a CreateChannel message type? Can a non-admin create channels? What is the wire format for channel creation? The ControlLog entry format is unspecified (covered by P3-3), but even the semantics of channel creation are absent.
- Intra-cluster communication outside of the gateway. The spec says "reliable broadcast (iterated unicast) for Vital/Reactive messages" for intra-cluster dissemination. But this is one sentence. There is no intra-cluster session model, no CIK-based encryption for intra-cluster frames, no specification of how a non-gateway peer in a cluster communicates with its cluster peers. All inter-cluster detail is exhaustive; intra-cluster is a blank interior.
- The "admin" concept's boundary with "user." The spec talks about admin DK, admin authority, admin enrollment. But who IS the admin at the application layer? saalis has a user/role model (Admin, User, Child). The verkko spec never connects "admin" to saalis's role system. This is a phantom border: the map shows a border between "verkko admin" and "saalis admin" but there is no checkpoint crossing.

### verkko-relay: Lightly Mapped

The membrane model (five operations) and territorial re-encryption are well-specified. But the transit relay -- the core function of forwarding frames -- has almost no body text. The SIGNAL operation for connection setup is mentioned but unspecified. Relay authentication is mentioned but unspecified. How a relay peer discovers that it should relay for a given pair is unspecified. Run 10 P2-4 covers transit forwarding, P3-4 covers relay topology selection and SIGNAL.

**Unmapped regions remaining after Run 10:**
- Relay incentive model. Why does a peer relay for others? Cost model, resource consumption. The spec says "relay peer forwards frames without decrypting content" but does not address the CPU/bandwidth/storage cost to the relay peer or how the relay is compensated (or why it accepts the cost).
- Relay failover. If the relay peer goes down, what happens to the relayed connection? Is there a relay backup? How does the sender discover that the relay is gone? The spec mentions "relay topology adapts" but gives no mechanism.
- Multi-hop relay. The outer header has a hop_count field with max 4. This implies multi-hop relay paths. The spec never specifies when multi-hop occurs, how the path is selected, or what the forwarding table looks like.

### verkko-seat: Terra Incognita

The defined concepts exist (Seat, SeatSession, HostPeer, SeatRequest, SeatEvent, SeatInstruction, SeatProgress). Every defined concept has invariants. But the body section is empty. There is no authentication protocol, no message catalog, no error model, no lifecycle specification. Run 10 P2-1 covers this with ~400 lines of estimated spec text.

**Unmapped regions remaining after Run 10:**
- Seat failover. What happens when the host peer goes down? The invariant says "if that peer goes offline, the seat is disconnected." But then what? Does the seat reconnect to another peer in the cluster? Does it wait? Is there a handoff protocol? This is terra incognita even after P2-1 -- the action plan says "authentication and session establishment" but does not mention failover.
- Multi-seat coordination. Multiple seats connect to the same host peer. What happens when two seats issue conflicting writes? The consistency model (CONSISTENT/SNAPSHOT) addresses read consistency but says nothing about write conflicts between seats.
- Seat notification delivery guarantees. SeatEvent is pushed, not polled. But over what transport? WebSocket? SSE? The spec says verkko-protocol's Stream and Frame, but seat depends on protocol, not the other way around. What reliability mode do seat events use? Reliable ordered? Unreliable?

### verkko-matter: Lightly Mapped

Content addressing (blob_id = BLAKE3), cuckoo filters, Merkle reconciliation, sealed-range catch-up, and streaming verification are well-mapped. MatterTransfer, SyncReconciliation, Replication, and BandwidthPolicy have defined concepts with invariants but no body text. Run 10 P2-2 and P2-3 cover the body voids.

**Unmapped regions remaining after Run 10:**
- Garbage collection. Matter is immutable. But what happens when nobody wants a piece of matter anymore? Is it ever deleted? The spec says cuckoo filters use saturating counters supporting deletion. But the deletion policy -- who decides, when, how -- is absent. Stash has a "Missing" status but no "Deleted" status.
- Matter size limits. No maximum blob size. No minimum. No chunking strategy for large blobs. The spec says "A 2GB ROM" is matter, but the wire format uses 1280-byte UDP segments. The chunking bridge between 2GB blobs and 1280-byte frames is unspecified.
- Priority and scheduling of matter transfers. The spec has tiered deadlines for re-encryption (P1 through P4) but no priority model for initial matter transfer. When a peer wants both a 2GB ROM and a 50KB metadata archive, what goes first?
- The external source fallback. MatterTransfer says "Mesh-first: local peer -> mesh peers -> external source." External sources are connectors (IGDB, ScreenScraper, etc.). But verkko-matter never specifies the connector interface. This is because connectors are saalis-layer, not verkko-layer. But the fallback chain references them without specifying the boundary.

### verkko-ops: Reference Territory (not independently mapped)

This is not a territory in the cartographic sense. It is the legend, the key, the scale bar, the compass rose. It consumes all other documents. It defines nothing that others depend on. It is well-organized and internally consistent. Its gaps are inherited from the territories it references.

## 2. Trade Route Analysis

Data flows across document boundaries through defined concepts. Each boundary crossing is a "trade route." Some routes carry too much traffic. Some have no checkpoint at all.

### Major Trade Routes

**Route 1: crypto -> protocol (FrameKey, TransportAEAD, NonceSafety, EpochRatchet)**
Traffic volume: EVERY frame on the wire traverses this route. This is the busiest trade route in the system. Well-defined: the protocol document specifies the exact wire format, the crypto document specifies the exact key derivation. The checkpoint (defined concepts in crypto's export list) is clean.

**Route 2: protocol -> mesh (Heartbeat, MetabolicState, SACK)**
The heartbeat is the system's circulatory system. Every 2-30 seconds, health observations, epoch state, filter roots, and gossip flags flow from protocol to mesh. The HeartbeatExtension split (transport header vs mesh extension) is a grain boundary that Run 10 P0-3 formalizes. This route is busy but well-managed after P0-3.

**Route 3: mesh -> relay (Convergence, Gateway, Cluster)**
Convergence fronts drive re-encryption. Gateway election determines relay paths. Cluster identity determines territorial partitioning. This route is moderately busy and well-defined by the convergence specification. The ring coloring constraint is the border checkpoint.

**Route 4: mesh -> matter (ContentFilter, HealthPipeline, Channel)**
Source selection requires health scores, filter state, and channel access. This route is implicit. The SourceSelectionPredicate in matter references mesh concepts (health, access, reachability) but the predicate is not a formal defined concept in either document. Run 10 P1-1 creates wire contracts for it. Needs formalization.

**Route 5: matter -> relay (Membrane, TerritorialReencryption, BlobId)**
Matter's content is stored at rest via relay's membrane. Re-encryption uses blob_id as nonce derivation input. This route is well-defined by the deterministic re-encryption nonce construction.

**Route 6: seat -> protocol (Connection, Session, Stream, Frame)**
Seat uses protocol for transport. Clean dependency. No issues.

**Route 7: crypto -> mesh (DK, DEK, ChannelKey, BroadcastEncryption, PCSMechanism, FoundingHash)**
Enrollment, key distribution, dominance cascade, convergence -- all use crypto primitives. This route is the second busiest after Route 1. Well-defined by the defined concept boundaries.

### Chokepoints

**Chokepoint 1: HeartbeatExtension.**
Six consumers share 18 bytes. Filter root, epoch counter, work capacity, health canary, observation summary, gossip mode flag. Every mesh mechanism reads this. Every protocol heartbeat produces it. This is the narrowest pass in the system. Run 10 P0-3 formalizes it, but it remains a chokepoint: any change to these 18 bytes ripples through six consumers.

**Chokepoint 2: ChannelKey.**
Channel keys are the currency of the system. Created by crypto, distributed by mesh (KeyBundle), used by protocol (encryption), stored by relay (membrane), filtered by matter (cuckoo filters). Five documents touch ChannelKey. The key lifecycle (Active/Orphaned) is the sole traffic controller.

**Chokepoint 3: Conductor/MetabolicState.**
Every resource-budget parameter flows through this single bottleneck. By design (the "sole coherence point" invariant). But this means every resource-sensitive mechanism in every document is coupled to the governor's output. The chokepoint IS the design; the risk is that it becomes a bottleneck for specification changes, not for runtime performance.

### Missing Checkpoints (undeclared border crossings)

**Crossing 1: matter -> protocol (undeclared dependency).**
Run 10 P0-1 fixes the declaration. But the crossing itself (how MatterTransfer maps to Stream and LEDBAT) is still unspecified. The declaration creates the checkpoint; the body of the checkpoint is still empty.

**Crossing 2: seat -> saalis application layer.**
The seat serves the web UI. But the spec never says how. SeatRequest and SeatEvent are verkko-level concepts. The saalis web layer (axum, Maud, HTMX) consumes them. This crossing is outside verkko's scope, but the absence of any mention creates a blank border where implementers will improvise.

**Crossing 3: mesh intra-cluster -> mesh inter-cluster.**
The spec exhaustively details inter-cluster communication (gateway, CPSK, broadcast encryption). Intra-cluster communication is one sentence: "reliable broadcast (iterated unicast)." The border between these two regimes is unmarked. A traveler implementing the mesh would not know where intra-cluster ends and inter-cluster begins without reading the entire mesh document.

## 3. The Map After the 23 Actions

If all Run 10 actions are applied faithfully, the map looks like this:

### What becomes well-mapped:

- **HeartbeatExtension** (P0-3): grain boundary formalized. Consumers have a defined interface.
- **ConvergenceGate** (P0-5): opaque projection. Consumers see booleans, not mechanisms.
- **ScarSeverityTier** (P0-6): the scar system gets depth perception.
- **DomainSeparator enum** (P0-7): compile-time domain safety.
- **HealthPipeline two-mode** (P0-8): NORMAL/DAMPED with damping coefficients.
- **Wire contracts for grain boundary concepts** (P1-1): HeartbeatExtension, ConvergenceGate, SourceSelectionPredicate get byte-level specifications.
- **Wire contracts for crypto-protocol bonds** (P1-2): FrameKey, TransportAEAD, EpochRatchet, BroadcastEncryption, Frame, Connection, Session.
- **First-bytes-on-the-wire contracts** (P1-3): Noise handshake, Connection ID, SchemaSync, Stream mux.
- **Seat body** (P2-1): authentication, request/response, events, consistency model.
- **Matter transfer body** (P2-2): MatterTransfer -> Stream mapping, LEDBAT variant, BandwidthPolicy.
- **Matter sync and replication** (P2-3): merge rules, replication protocol.
- **Relay transit** (P2-4): forwarding procedure, relay authentication, forwarding path.
- **Convergence duration bound** (P2-5): wavefront deadline defaults.
- **Health pipeline dimensions** (P2-6): two projections (availability, throughput).

### What is STILL blank after all 23 actions:

**B1. The SansIO consumer interface.**
No action in Run 10 specifies the API that the consumer uses to drive the protocol state machine. This is the most dangerous blank on the map. The protocol is defined as a sans-IO state machine, but the state machine's input/output surface is unspecified. An implementer would need to invent:
- The event type enum (what the state machine emits)
- The action type enum (what the consumer can feed in)
- The polling/driving interface
- The buffer management contract (MemoryProvider)
- Error semantics for buffer exhaustion, invalid state transitions

**B2. Intra-cluster communication.**
One sentence in the entire spec. No session model, no encryption model, no message routing within a cluster. All traffic between peers in the same cluster is a blank map.

**B3. Channel lifecycle.**
Channels are created, granted, rotated, revoked, and accessed. But channel creation is unspecified. Who creates? What format? What happens to a channel when the last key is revoked?

**B4. Matter garbage collection.**
No deletion policy. No storage pressure response for matter (as distinct from relay re-encryption). The stash has four states but none of them is "deleted by policy."

**B5. Matter chunking.**
2GB blobs over 1280-byte UDP segments. No chunking protocol. The bao tree implies a chunking structure but the chunk size, transfer scheduling, and resumption protocol are unspecified.

**B6. Multi-hop relay forwarding.**
hop_count max 4, but no forwarding table, no path selection algorithm, no multi-hop specification.

**B7. Relay failover and incentives.**
Why relay? What happens when a relay goes down?

**B8. Seat failover.**
Host peer dies. What does the seat do?

**B9. Seat write conflict resolution between concurrent seats.**
Multiple seats, conflicting writes. No resolution model.

**B10. The verkko-to-saalis application bridge.**
How does the mesh's state become visible in the web UI? How do saalis's entity/metadata concepts map to verkko's matter/channel concepts? This is the entire reason the mesh exists, and it is outside the scope of all seven documents.

**B11. DK encryption at rest.**
Listed as an open design question (Section 10.2 of mesh). Not covered by any Run 10 action.

**B12. Connection teardown.**
Connection establishment is detailed (Noise KK/XX, PATH_CHALLENGE/PATH_RESPONSE). Connection teardown is absent. Does a connection have a close message? What happens to in-flight frames? What about half-closed connections?

## 4. Phantom Borders

Borders that exist on the map but that the data flows do not respect.

### Phantom Border 1: verkko-seat -> verkko-mesh (non-existent edge)

The map says seat depends only on crypto and protocol. The HostPeer concept mediates. But in practice, the HostPeer IS a mesh peer. The seat's requests ("list my games," "download this ROM") require the host peer to perform mesh operations (query cuckoo filters, initiate matter transfer, check channel access). The data flow is: seat -> host peer -> mesh. The border between seat and mesh is not a real border; it is a membrane through HostPeer. This is intentional (Run 10 consensus finding 13 confirms it). But the phantom border means that seat's specification cannot be completed without specifying HostPeer's mesh-facing behavior, which lives in mesh, not seat. P2-1 writes seat's body but does not specify the HostPeer's internal translation layer.

**Cartographic:** The seat territory and the mesh territory are separated by a river (HostPeer). The bridge across the river is drawn on the map, but neither territory claims the bridge as its own. Travelers arriving at the bridge find no customs office on either side.

**Mechanism:** HostPeer receives a SeatRequest (typed, access-controlled). HostPeer must translate this into mesh operations (filter lookup, channel access check, matter transfer initiation, KeyBundle observation). The translation function is unspecified. It is not in seat (seat does not know mesh). It is not in mesh (mesh does not know seat). It lives in the application layer, outside all seven documents.

**Emergent:** Without the translation specification, the HostPeer will be the place where every implementer makes ad-hoc decisions. It will accumulate coupling to both seat and mesh, becoming a de facto dependency from seat to mesh. The phantom border dissolves in practice.

### Phantom Border 2: verkko-protocol / verkko-mesh MetabolicState boundary

Run 10 P0-2 resolves this by creating an abstract governor contract in protocol. But the phantom remains: the MetabolicState's four states (Background, Idle, Active, Stress) are defined in protocol. The concrete triggers that cause transitions are in mesh's Conductor. The abstract governor interface in protocol will specify transition predicates. But the metabolic state machine's behavior (what it does in each state) is split between protocol (heartbeat interval, LEDBAT target delay, queue sizing) and mesh (convergence override, phase jitter). A traveler reading only the protocol document will see a state machine with abstract triggers and concrete behaviors. A traveler reading only the mesh document will see concrete triggers and abstract behaviors. Neither document is self-contained for the metabolic state machine. P0-2 resolves the dependency cycle but does not resolve the comprehension split.

**Cartographic:** The metabolic state machine is a city that straddles a border. The city hall (triggers) is on the mesh side. The market (parameters) is on the protocol side. You cannot understand the city without crossing the border.

**Mechanism:** `MetabolicState::transition(governor_output) -> MetabolicState` lives in protocol. `Conductor::compute(resource_pressures) -> GovernorOutput` lives in mesh. The abstract governor interface is the border checkpoint. But "convergence override: dwell time enforcement suspended" is a mesh-side behavior that modifies a protocol-side state machine invariant.

**Emergent:** Implementers will put the metabolic state machine in one crate. If it goes in the protocol crate (where it is defined), the convergence override must be an abstract interface callback. If it goes in the mesh crate (where it is driven), the protocol parameters must be passed through. Either way, the phantom border forces one side to hold abstractions that the other side concretizes.

### Phantom Border 3: verkko-matter BlobId / verkko-relay Membrane identity

The map draws a clean border: matter defines BlobId (BLAKE3 of plaintext), relay defines relay_key_id (HMAC of channel_salt and key_id). These are different identifiers for the same content. But the border checkpoint does not specify the translation. matter uses BlobId for source selection and cuckoo filters. relay uses relay_key_id for membrane operations. The translation is: `relay_key_id = HMAC-BLAKE3(key=channel_salt, data=key_id)`, where key_id is `BLAKE3(K_epoch_N || channel_index)[0..8]`. Neither BlobId nor the cuckoo filter fingerprint appears in this derivation. The relay_key_id is derived from the channel key and epoch, not from the content. So the same content under different epochs has different relay_key_ids. The border between matter's content identity and relay's storage identity is not crossed by the content hash at all. The "phantom" is that the two identity systems share no common root. This is by design (relay cannot learn content identity), but it means that the matter -> relay route described in Section 2 above is not actually a data identity route. It is a data placement route that uses a completely different addressing scheme.

**Cartographic:** Two nations with different currencies. Matter uses "content addresses." Relay uses "storage receipts." There is no exchange rate. The translation happens inside the peer when it decides to PUT content on the relay -- the peer knows both addresses. But no map shows this translation.

**Mechanism:** Peer holds plaintext. Computes `blob_id = BLAKE3(plaintext)`. Encrypts plaintext under `K_epoch_N`. Computes `relay_key_id = HMAC-BLAKE3(channel_salt, key_id)`. PUTs ciphertext on relay indexed by relay_key_id. The peer is the only entity that holds both identifiers simultaneously. The mapping is local state, not protocol state.

**Emergent:** If the peer crashes between computing blob_id and computing relay_key_id, the stash and the relay are out of sync. The stash says "I have this blob." The relay says "I have this relay_key_id." There is no reconciliation protocol between the two identity systems. The cuckoo filter advertises the blob; the relay stores the ciphertext; the mapping is in the peer's head. If the peer forgets, the content is on the relay but nobody knows how to retrieve it (because nobody can compute the relay_key_id without the channel key and epoch).

### Phantom Border 4: Convergence fronts / steady-state mechanisms

Run 10 consensus finding 14 notes: "Convergence uses the same mechanisms as steady state." The emergent mechanism in mesh says "Convergence as steady state at higher pressure." This means the border between "convergence mode" and "steady state" is not a border in the mechanism space -- it is a border in the pressure space. The same code runs. The same parameters apply (mostly). But the damping mode (P0-8) introduces a real behavioral difference during convergence. The phantom is that the named fronts (ControlFront, ConsensusFront, ReencryptionFront) suggest a separate convergence protocol, but in reality convergence is a sequencing constraint on top of steady-state mechanisms plus flood gossip plus damping. The fronts are gates, not territories.

## 5. Novel Mechanisms

Three mechanisms that would fill the most dangerous unmapped regions. Each in three representations.

### Mechanism N1: SansIO Event Surface

The most dangerous blank. Without this, the protocol cannot be implemented.

**Cartographic:** The protocol territory has no port. Ships arrive with cargo (inbound bytes) but there is no dock, no harbormaster, no manifest system. The interior is well-mapped (state machines, wire formats), but you cannot enter or leave the territory because the port does not exist on the map.

**Protocol mechanism:**

```
enum ProtocolInput {
    BytesReceived { source: SocketAddr, data: &[u8] },
    TimerExpired { timer_id: TimerId },
    ConnectionRequested { peer_identity: DKFingerprint, address: SocketAddr },
    ArenaRefill { arena: &mut Arena },
}

enum ProtocolOutput {
    SendBytes { destination: SocketAddr, data: &[u8] },
    SetTimer { timer_id: TimerId, deadline_ms: u64 },
    SessionEstablished { connection_id: ConnectionId, peer: DKFingerprint },
    SessionLost { connection_id: ConnectionId, reason: DisconnectReason },
    FrameReady { connection_id: ConnectionId, stream_id: StreamId, payload: &[u8] },
    NeedArena { min_bytes: usize },
}

trait ProtocolMachine {
    fn handle(&mut self, input: ProtocolInput) -> SmallVec<[ProtocolOutput; 4]>;
    fn poll_timeout(&self) -> Option<u64>;
}
```

MemoryProvider becomes an ArenaRefill input. The consumer owns the allocation. The protocol borrows. No heap allocation inside the protocol.

**Emergent:** With an explicit event surface, the protocol becomes testable without a network stack. Property-based tests can generate arbitrary ProtocolInput sequences and verify invariants on the ProtocolOutput sequences. The sans-IO contract becomes enforceable, not aspirational.

### Mechanism N2: IntraClusterTransport

The blank interior of mesh's cluster concept.

**Cartographic:** The mesh map shows 12 cities (clusters) connected by highways (inter-cluster CPSK connections). Inside each city, the streets are blank. There are buildings (peers), and the map says they exist, but there is no street map. A traveler inside a city has no way to navigate.

**Protocol mechanism:**

Intra-cluster communication uses the same Noise KK sessions as inter-cluster, but with CIK as the static key for all peers in the cluster. Each intra-cluster session produces a session key. Intra-cluster frames use the same wire format as inter-cluster frames but with destination = 0 (broadcast) or a peer-local index. Gateway aggregation of FlowSummary happens over these sessions.

Key difference from inter-cluster: no CPSK (all peers share CIK), no broadcast encryption (CIK-encrypted direct), no polygon schedule (small group, iterated unicast suffices). The session model is: Noise KK with CIK static key, producing a pairwise session key per peer pair within the cluster.

```
IntraClusterSession {
    peer_index: u8,           // 0-5 within cluster
    session_key: [u8; 32],    // from Noise KK with CIK
    nonce: NonceSource,       // same three-layer defense
}
```

CIK rotation (48-hour dual acceptance window) applies to intra-cluster sessions identically to inter-cluster sessions.

**Emergent:** With an explicit intra-cluster transport, the gateway's role becomes clear: it is a router between the intra-cluster mesh and the inter-cluster mesh. FlowSummary aggregation is the gateway reading intra-cluster health observations and forwarding them to inter-cluster heartbeats. Without this specification, the gateway is a magic box. With it, the gateway is a router with two interfaces.

### Mechanism N3: MatterChunkProtocol

The bridge between 2GB blobs and 1280-byte frames.

**Cartographic:** The matter territory has a resource (content blobs). The protocol territory has a transport (1280-byte UDP segments). Between them is a canyon with no bridge. The map shows content on one side and frames on the other. The bao tree is drawn as a dotted line across the canyon, but the bridge girders are not specified.

**Protocol mechanism:**

```
MatterChunk {
    transfer_id: u64,           // identifies the transfer session
    blob_id: [u8; 32],          // BLAKE3 of complete plaintext
    chunk_index: u32,           // position in the bao tree
    total_chunks: u32,          // for progress reporting
    chunk_data: [u8; 1200],     // max payload per frame (1280 - 55 overhead - 25 chunk header)
    bao_proof: [u8; PROOF_LEN], // incremental bao tree proof for this chunk
}
```

Transfer lifecycle:
1. Receiver sends `TransferRequest { blob_id, chunk_bitmap }` (resumption-capable).
2. Sender replies with `TransferBegin { blob_id, total_chunks, bao_root }`.
3. Sender streams chunks on bulk tier (Queue 3, LEDBAT-paced).
4. Receiver verifies each chunk against bao_proof independently.
5. On verification failure: abort, scar the source (IntegrityFailure).
6. On completion: verify blob_id = BLAKE3(reassembled plaintext).

Multi-source: the chunk_bitmap in TransferRequest enables resumption from a different source. The bao tree structure guarantees that chunks from different sources can be interleaved as long as each chunk passes its individual proof.

**Emergent:** With an explicit chunk protocol, matter transfer becomes resumable, multi-source, and incrementally verifiable. Without it, implementers will invent their own chunking, and two implementations will produce incompatible transfer sessions.

## 6. The Complete Map

My vision of the fully surveyed verkko territory. What it looks like when every region has been walked.

### Layer 0: The Bedrock (verkko-crypto)

Six cryptographic primitives in five isolation domains. Every derivation has a named domain separator. Every key has a lifecycle. Every nonce has a three-layer defense. The bedrock is solid.

What is added: key serialization formats, CSPRNG self-test precision, cross-thread frame key derivation ordering.

### Layer 1: The Roads (verkko-protocol)

The sans-IO state machine with an explicit event surface. Two I/O strategies (epoll, io_uring) behind one ProtocolMachine trait. MemoryProvider as an arena-refill input. Wire format with byte-level specification and canonical generators. Connection lifecycle from establishment through migration through teardown. Stream multiplexing with explicit stream_id wire format and reliability mode selection. LEDBAT with coupled congestion control at the gateway. Heartbeat with transport header and mesh extension region cleanly separated. Version negotiation in the handshake. The metabolic state machine with abstract governor interface.

What is added: SansIO consumer interface, MemoryProvider contract, connection teardown, reliability mode selection per stream, io_uring behavioral guarantees.

### Layer 2: The Cities (verkko-mesh)

Twelve clusters on the mesh. Each cluster has an explicit intra-cluster transport model. The gateway is a router between intra-cluster and inter-cluster networks. Channel lifecycle from creation through access control through revocation. Enrollment, dominance cascade, health pipeline (two-mode, two projections), scar provenance with severity tiering, convergence (three named fronts with opaque gates and a six-step exit), gossip (dual-mode), resource governance (Conductor with Governed<T>). The admin concept connected to saalis's role system through a defined boundary.

What is added: intra-cluster transport model, channel creation semantics, admin-to-role boundary specification, HostPeer translation layer specification.

### Layer 3: The Trade Routes (verkko-relay)

Transit relay with explicit forwarding procedure, relay authentication, multi-hop path selection. Membrane with five operations and byte-level wire contracts. Territorial re-encryption with ring coloring and deterministic nonces. Constant-rate padding. Relay failover with backup relay selection. Relay resource accounting (why a peer accepts relay burden).

What is added: multi-hop forwarding specification, relay failover, relay resource model.

### Layer 4: The Ports (verkko-seat)
