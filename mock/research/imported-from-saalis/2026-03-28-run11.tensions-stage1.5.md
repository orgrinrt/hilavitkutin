# Run 11: Tension Map (Stage 1.5)

**Date:** 2026-03-28
**Input:** Four Stage 1 documents (specwriter, archsurgeon, wireeng, savant)
**Purpose:** Identify productive disagreements for cross-review focus

---

## Tension 1: Version Negotiation Timing — Handshake Payload vs. Pre-Handshake Probe

**SpecWriter** places version negotiation inside the Noise handshake encrypted payload: "Version negotiation occurs during the Noise handshake as part of the encrypted payload in the first application message after handshake completion." The negotiated version is derived from min/max ranges both peers declare post-handshake.

**WireEng** places version negotiation before the Noise handshake as a dedicated pre-handshake UDP exchange: "Version negotiation is the first exchange on any new UDP association, before the Noise handshake." WireEng specifies a 24-byte VersionProbe datagram sent unencrypted, with a nonce-echoing VersionResponse selecting the version before any Noise state is initialized.

These are incompatible placements. SpecWriter's approach means two peers with no common version will complete a Noise handshake before discovering the incompatibility and aborting. WireEng's approach means no Noise work is done on a version mismatch.

**Question for cross-review:** Does version negotiation belong before or inside the Noise handshake, and what is the cost of the alternative?

## Tension 2: BLAKE2b vs. BLAKE2s in the Noise Protocol Name

**SpecWriter** specifies `Noise_KK_25519_ChaChaPoly_BLAKE2b` and `Noise_XX_25519_ChaChaPoly_BLAKE2b` (BLAKE2b, the 64-bit variant).

**WireEng** specifies `Noise_KK_25519_ChaChaPoly_BLAKE2s` and `Noise_XX_25519_ChaChaPoly_BLAKE2s` (BLAKE2s, the 32-bit variant). WireEng notes: "These are the exact Noise protocol name strings used in the prologue and Noise handshake state initialization."

This is not a naming inconsistency. The Noise protocol name string is hashed into the handshake state as the prologue binding context. Two peers using different name strings will produce non-interoperable sessions silently — the handshake will fail with a decryption error but neither peer will understand why.

**Question for cross-review:** Which hash variant is correct for the Noise handshake, and why? What is the performance/security rationale?

## Tension 3: Connection ID Generation — Random vs. Derived

**SpecWriter** specifies `connection_id = random_u32()`, generated at session initiation by the initiator.

**WireEng** specifies a BLAKE3-derived connection ID: `connection_id = BLAKE3("saalis-connid-v1" || local_CIK_pub || remote_CIK_pub || handshake_nonce || session_counter)[0..4]`. WireEng argues this provides unpredictability for off-path observers and migration-safety by binding the ID to the session key material. WireEng also adds "saalis-connid-v1" to the DomainSeparatorRegistry.

SpecWriter's approach is simpler and produces the same collision probability. WireEng's approach introduces a new domain separator, couples connection ID generation to key material, and prevents pre-handshake ID assignment.

**Question for cross-review:** Does the connection ID need to be derived from key material, or is a random u32 sufficient? Does adding a new domain separator for this purpose fit the DomainSeparator enum design?

## Tension 4: Scope of Deliverables — Spec Text vs. Gap Inventory

**SpecWriter** and **WireEng** produce dense normative text covering Run 10's 23 actions: wire formats, test vectors, state machines, error tables, canonical generators. Their deliverables directly extend the design documents.

**Savant** produces a territory survey that finds 12 gaps NOT covered by Run 10's 23 actions (labeled B1 through B12), and argues three of them are more dangerous than everything the other experts produced: "The most dangerous blank is B1 (SansIO consumer interface). Every other mechanism depends on the protocol being implementable. The protocol is specified as a state machine whose input/output surface does not exist on the map."

**ArchSurgeon** acknowledges the 23 actions but focuses only on structural surgery (dependency declarations, back-reference cleanup, DAG verification). Neither the surgeon nor the savant writes the kind of byte-level wire format the specwriter and wire engineer produce.

The divergence is not just style. Savant claims the Run 10 plan is addressing the wrong level of the problem: producing precise text for already-partially-specified areas while leaving the SansIO consumer interface (B1), intra-cluster communication (B2), and matter chunking (B5) entirely blank. The other experts implicitly accept the Run 10 scope.

**Question for cross-review:** Are the gaps Savant names (especially B1, B2, B5) within scope for Run 11, or should they be deferred? If deferred, when do they get addressed?

## Tension 5: The HostPeer Translation Layer — Phantom or Real?

**Savant** identifies the seat/mesh boundary as a "phantom border": "The seat territory and the mesh territory are separated by a river (HostPeer). The bridge across the river is drawn on the map, but neither territory claims the bridge as its own." Savant argues that SpecWriter's P2-1 Seat body spec cannot be complete without specifying HostPeer's internal translation from SeatRequest to mesh operations.

**SpecWriter** produces a complete seat body spec (P2-1) with authentication, consistency model, request/response, event subscription, and instruction delegation. The spec is self-contained at the seat-protocol boundary. It treats HostPeer as opaque: the seat sends SeatRequests, the host peer responds with SeatResponses.

**ArchSurgeon** does not address seat at all, focusing only on protocol-layer concerns.

The tension: SpecWriter's seat spec is implementable from the seat's perspective but leaves the host peer's mesh-facing behavior unspecified. Savant argues this means the spec is incomplete, because the host peer's translation behavior will be invented independently by every implementer, dissolving the phantom border in practice.

**Question for cross-review:** Does the seat spec need to specify HostPeer's internal translation layer (SeatRequest -> mesh operations), or is specifying the seat-side wire protocol sufficient? Where does this translation layer belong?
