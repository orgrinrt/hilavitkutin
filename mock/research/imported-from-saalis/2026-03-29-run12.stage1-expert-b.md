## Part 1: Independent Assessment

### 1.1 State of the design from a sans-IO perspective

The verkko protocol documents define a rich set of wire formats,
cryptographic mechanisms, and convergence algorithms. The ProtocolInput
and ProtocolOutput enumerations (R11-016) are well-chosen: 8 input
events and 13 output events cover the protocol surface. The problem is
that these enumerations are a vocabulary, not an API. The design
currently has nouns (what events exist) but no verbs (how the consumer
feeds them in, how it drains outputs, who owns the buffers in between,
and what happens when capacity is exhausted).

This is the single most critical gap in the entire verkko specification.
Without the polling model, two independent implementations will produce
wire-compatible but integration-incompatible state machines. The quinn
project learned this lesson: the transport and connection state machines
have a precise polling contract (`handle_event() -> Vec<Event>`, then
`poll_transmit() -> Option<Transmit>`) that makes the sans-IO
separation actually useful. Without it, you have a state machine that
nobody knows how to drive.

### 1.2 What exists and what is missing

**Exists (solid):**
- ProtocolInput enum: 8 variants covering bytes, timers, connection
  lifecycle, stream lifecycle, heartbeat ticks, config changes.
- ProtocolOutput enum: 13 variants covering send, timers, session
  events, frame delivery, heartbeat forwarding, SACK, path challenge,
  schema, transfer, connection close, session resumption, arena request.
- Wire format: 55-byte frame overhead, 1232-byte UDP payload limit,
  29-byte outer header, inner payload structure.
- Stream multiplexing: 3-tier stream ID allocation, 7-byte stream
  header, FIN/RST/SACK_PRESENT flags.
- SACK: 18-byte fixed header + 8 bytes per block, max 16 blocks.
- VitalQueue: 3-tier priority with 70/30 weighted fair queuing.
- MetabolicState: 4 states, abstract governor contract defined.
- Memory arenas: hot (64KB bump), pool (256KB fixed-block), managed
  (~200KB system-budgeted).
- NeedArena output variant signals buffer exhaustion.

**Missing (critical):**
- The polling interface: no `fn handle()`, no `fn poll_transmit()`,
  no `fn poll_timeout()`. The consumer has no calling convention.
- Buffer ownership semantics: when BytesReceived hands data to the
  state machine, who owns the data slice? Is it borrowed for the
  duration of handle()? Is it copied into the arena? The design
  mentions arena allocation but never specifies the ownership transfer.
- Output draining: when the state machine produces SendBytes, does
  the consumer get a borrowed slice from the arena, or an owned
  buffer? What lifetime does it have? When is the arena region freed?
- Backpressure: NeedArena exists but the contract is incomplete.
  What happens when the consumer cannot provide more arena space?
  Does handle() return an error? Does it buffer internally? Does it
  drop frames?
- Error semantics: no error type defined. handle() presumably can
  fail (invalid frame, crypto failure, arena exhaustion). The error
  type determines whether the connection survives.
- Timer ownership: SetTimer { timer_id, deadline_ms } is emitted,
  but timer_id is untyped. Who allocates timer IDs? What happens
  when a timer fires after the connection is closed? What is the
  timer_id space?
- Reentrancy: can handle() be called while a previous handle() is
  being processed? The single-threaded event loop (Section 6.2)
  implies no, but the contract does not state this.
- Reliability mode selection per stream (D-013): the protocol
  mentions reliable-ordered and unreliable-datagram modes but does
  not specify how a stream selects its mode or how the mode affects
  the state machine's internal handling.
- MetabolicGovernor concrete implementation (D-012): the abstract
  contract exists; the parameter mapping from resource pressures to
  metabolic state is blank.

### 1.3 Assessment of the arena model

The three-arena model (hot, pool, managed) is appropriate for the
workload. The hot arena (64KB bump) handles per-frame transients.
The pool arena (256KB fixed-block) handles per-epoch transients and
assembly. The managed arena (~200KB) handles long-lived state.

The concern is the interaction between the arena model and the sans-IO
polling interface. In quinn, the connection state machine borrows
buffers from the caller for input and returns owned Transmit structs
for output. In smoltcp, the device provides a receive buffer and a
transmit buffer, and the stack operates on borrowed slices. In verkko,
the arena model implies a third pattern: the protocol owns the memory,
the consumer provides raw bytes as input, and the protocol returns
arena-allocated slices as output. This is viable but requires explicit
lifetime management.

The NeedArena output event suggests a demand-paged model: the protocol
runs until it needs more memory, then signals the consumer. This is
problematic for latency-sensitive paths. If a heartbeat needs to be
sent and the arena is full, the protocol must either drop the heartbeat
(violating liveness) or block (violating sans-IO). The design needs to
specify the arena exhaustion policy per tier.

### 1.4 Assessment of reliability modes

The protocol document mentions two reliability modes in a single
paragraph (Section 3.3): "Reliable ordered: monotonic sequence, SACK
ACKs, exponential backoff (1s/2s/4s/8s, max 30s), adaptive death
timeout. Unreliable datagram: per-source monotonic sequence for gap
detection."

This is insufficient. The reliability mode affects every aspect of
stream processing: buffer management (reliable requires retransmission
buffers), SACK generation (reliable needs per-stream SACK; unreliable
does not), flow control (reliable needs recv_window; unreliable may
not), ordering (reliable preserves order; unreliable delivers
out-of-order), and memory allocation (reliable needs O(window) memory;
unreliable needs O(1)).

The stream multiplexing design (R11-014) assigns stream IDs to tiers
but does not bind tiers to reliability modes. The natural mapping would
be: vital and reactive streams are reliable-ordered; bulk streams are
reliable-ordered for matter transfers but could be unreliable for
some operations. But this mapping is not stated.

### 1.5 Assessment of MetabolicGovernor (D-012)

The abstract contract (R11-008) is clean: a pure function from
resource pressure readings to MetabolicState. The Conductor section
in verkko-mesh provides the governor formula as a meet projection:
`min(CPU, flash, nonce_io_pressure, min(WiFi_capacity, transport_capacity))`.

What is missing is the concrete parameter mapping. How does a CPU
utilization of 0.7 (Q16.16: 0x0000B333) map to a MetabolicState?
The governor needs threshold bands for each resource dimension, and
the meet of all dimensions produces the final state. The threshold
bands exist for GovernedResource (4 thresholds, 4 responses) but the
mapping from GovernedResource thresholds to MetabolicState is not
specified. The parameter mapping is the body of the Conductor
implementation and it determines the system's behavior under load.

### 1.6 Assessment of intra-cluster communication (D-004)

Outside my domain -- no comment. The dest_peer field exists in the wire
format. The session model for intra-cluster traffic is a mesh-layer
concern, not a sans-IO polling concern.

### 1.7 Cross-cutting concerns from my domain

**Timer management.** The protocol has at least 7 distinct timer
types: heartbeat interval, SACK retransmission, connection migration
(PATH_CHALLENGE), CPSK renegotiation, metabolic dwell time, wavefront
deadline, and connection close (2*RTT drain). Timer IDs are u32
(inferred from SetTimer) but the timer ID space, allocation strategy,
and collision handling are unspecified.

In quinn, timers are an enum of fixed timer types per connection. This
is clean but inflexible. In s2n-quic, timers are managed by the
connection and the consumer only needs to track the earliest deadline.
The verkko design should pick one of these patterns.

**Crypto thread interaction.** Section 6.2 specifies a dedicated crypto
thread with SPSC ring buffers. The sans-IO state machine runs on the
event loop thread. When the state machine receives a frame that needs
Ed25519 verification or bao-tree verification, what happens? The state
machine cannot block. It must enqueue the verification and return, then
process the verification result as a new input event. But
ProtocolInput has no variant for "crypto verification completed." This
is a gap.

**Batch AEAD thread interaction.** Similarly, the batch AEAD thread
has its own NonceSource and handles bulk encryption. The state machine
must be able to hand off encryption work and receive results. The
current ProtocolOutput::SendBytes implies synchronous encryption, but
Section 6.2 implies asynchronous encryption for batch operations.

**Connection state machine vs session state machine.** The design
conflates connection-level and session-level state. A Connection
carries multiple Streams, but the Noise handshake produces a Session
which maps 1:1 to a Connection. Connection migration (PATH_CHALLENGE)
preserves the connection ID but changes the underlying address. The
state machine needs clear separation between connection state
(streams, SACK tracking, congestion control) and session state (Noise
keys, nonce counter, replay window). Quinn does this well.

### 1.8 What I would build

Given the design constraints (no_std-compatible state machine, arena
allocation, single-threaded event loop, SPSC ring to crypto thread),
the polling model should follow the quinn pattern adapted for arena
allocation:

1. Consumer calls `protocol.handle(input: ProtocolInput, arena: &Arena)`
   which processes one input event and mutates internal state.
2. Consumer calls `protocol.poll_transmit(arena: &Arena)` in a loop
   to drain all pending transmissions.
3. Consumer calls `protocol.poll_timeout()` to get the next timer
   deadline.
4. Consumer calls `protocol.poll_event()` in a loop to drain all
   pending application-level events (FrameReady, SessionEstablished,
   etc.).
5. handle() returns `Result<(), ProtocolError>` where ProtocolError
   includes arena exhaustion, crypto failure, and protocol violation.

This four-method interface covers the entire interaction surface. The
arena is passed by reference; the protocol borrows from it for output
buffers. The consumer must consume outputs before the arena is reset.

### 1.9 Interaction with existing design elements

The VitalQueue (3-tier priority) should be internal to the state
machine, not exposed to the consumer. The consumer calls
poll_transmit() and gets the next frame to send, in priority order.
The state machine manages the queue internally.

The SACK mechanism should be entirely internal. The consumer never
sees SACK frames directly; it sees application-level delivery
confirmations (if reliable) or nothing (if unreliable). The
SackReceived output event is useful for the mesh layer's health
pipeline but should be a separate poll point (poll_event) from
transmit draining (poll_transmit).

The MetabolicState should flow into the state machine via
ConfigChanged { governor_output } as already defined. The state
machine uses it to adjust heartbeat intervals, LEDBAT targets, and
queue budgets. This is correct.

### 1.10 Relationship to verkko-seat

The SeatTransport defined concept (R11-025) states that the SansIO
consumer interface handles WebSocket-delivered bytes identically to
UDP-delivered bytes. This is good: the state machine is
transport-agnostic. But it means the polling model must handle the
case where BytesReceived delivers a complete WebSocket message (which
may contain exactly one verkko frame) versus a UDP datagram (which
also contains exactly one frame). The design states "The protocol
machine MUST NOT assume that each BytesReceived event corresponds to
exactly one UDP datagram." This is correct and constrains the parser
to handle partial and multi-frame inputs.

However, the design also states the frame size limit of 1232 bytes is
preserved regardless of WebSocket capacity. This means WebSocket
messages carry one frame each. The parser can assume each
BytesReceived delivers at most one complete frame from WebSocket, but
must handle fragmentation from other transports. The distinction
matters for buffer management: if frames arrive one-at-a-time, the
hot arena can be reset between frames. If they arrive in batches
(recvmmsg), the hot arena must hold all frames in the batch.

## Part 2: The 5 Most Significant Problems

### Problem 1: No Polling Interface -- The SansIO Contract is Vacuous

**(a) Problem statement.**
The ProtocolInput and ProtocolOutput enumerations define the vocabulary
of the sans-IO interface but not the calling convention. There is no
trait, no method signature, no return type, no error type. The consumer
does not know how to drive the state machine. This is D-003 and it is
the highest-priority gap in the design.

**(b) Why it matters.**
Without the polling interface, two implementations will use different
driving patterns. One might call `handle(input) -> Vec<Output>` (heap
allocation, forbidden by the design's own rules). Another might use
`handle(input)` then `poll_output() -> Option<Output>`. A third might
use a callback model. These are not interoperable at the API level.
The sans-IO promise is that the protocol logic is decoupled from I/O;
but without a defined interface, the decoupling is aspirational.

Furthermore, the polling interface determines buffer lifetimes, arena
management, and backpressure behavior. Without it, every other design
decision (VitalQueue, SACK, stream multiplexing) floats without an
anchor. The queue cannot be implemented without knowing how outputs
are drained. SACK cannot be generated without knowing when the
consumer provides the send opportunity.

**(c) References and prior art.**

- **quinn (Rust QUIC):** `Connection::handle(now, remote, ecn, data)
  -> Option<DatagramEvent>`. Then `Connection::poll_transmit(now,
  max_datagrams) -> Option<Transmit>`. Then `Connection::poll_timeout()
  -> Option<Instant>`. Then `Connection::poll() -> Option<Event>`.
  Four-method interface. Outputs are owned structs with internal
  buffers. Arena allocation is not used; quinn allocates per-transmit.

- **s2n-quic (AWS):** Similar to quinn but with a `Provider` trait
  that abstracts timer, clock, and I/O operations. The connection is
  a state machine driven by `on_datagram_received()`. Outputs are
  collected via `on_transmit()` callbacks.

- **smoltcp (no_std TCP/IP):** `Interface::poll(device, timestamp)`.
  The device provides `receive()` and `transmit()` methods. The
  interface borrows from the device's buffers. Arena-like pattern:
  the device owns the memory, the stack borrows it.

- **sans-io.readthedocs.io (Python):** Defines the pattern as
  "receive data, produce events; receive events, produce data."
  Two-phase loop. No timer ownership; consumer manages timers.

- **embassy (embedded async):** State machine driven by `poll()`.
  Outputs are written into caller-provided buffers. Zero allocation.

**(d) Recommended approach.**

Define a `Protocol` trait with four polling methods, an arena
parameter, and a concrete error type. The trait is the sole entry
point for the consumer.

```rust
/// Arena-backed buffer for protocol I/O.
/// Lifetime 'a ties outputs to the arena's current epoch.
/// Consumer must drain all outputs before calling arena.reset().
struct ArenaRef<'a> { /* bump allocator reference */ }

/// Opaque address type. Consumer defines what this means.
/// Could be SocketAddr, could be a WebSocket connection ID.
#[repr(C)]
struct PeerAddr { bytes: [u8; 32] }

/// A frame ready to transmit. Borrows from the arena.
struct Transmit<'a> {
    destination: PeerAddr,
    payload: &'a [u8],
}

/// Application-level event delivered to the consumer.
enum Event<'a> {
    SessionEstablished { connection_id: u32, peer: PeerAddr },
    SessionLost { connection_id: u32, reason: CloseReason },
    FrameReady { connection_id: u32, stream_id: u16, payload: &'a [u8] },
    HeartbeatReceived { connection_id: u32, transport: &'a [u8], extension: &'a [u8] },
    SessionResumed { connection_id: u32, peer: PeerAddr },
    TransferRequested { connection_id: u32, request: &'a [u8] },
    ConnectionClosing { connection_id: u32, reason: CloseReason },
    SchemaRequired { connection_id: u32 },
    PathChallengeReceived { connection_id: u32, challenge: [u8; 8] },
}

/// Timer deadline. Consumer must call handle(TimerExpired)
/// when this deadline elapses.
struct TimerDeadline {
    timer_id: u32,
    deadline_ms: u64,
}

/// Errors that can occur during protocol processing.
#[repr(u8)]
enum ProtocolError {
    /// Arena exhaustion. Consumer must provide more arena space.
    ArenaExhausted = 0,
    /// Cryptographic verification failed. Frame dropped.
    CryptoFailure = 1,
    /// Protocol violation by remote peer. Connection destroyed.
    ProtocolViolation = 2,
    /// Nonce safety violation. AEAD halted. See NonceSafety.
    NonceHalted = 3,
}

/// The sans-IO protocol state machine.
/// 'state lifetime is the protocol's own state.
/// 'arena lifetime is the current arena epoch.
trait Protocol {
    /// Process one input event.
    /// May mutate internal state. May enqueue outputs.
    /// Returns Ok(()) on success, Err on fatal error.
    /// On ArenaExhausted: consumer must provide more arena,
    /// then retry the same input.
    fn handle<'a>(
        &mut self,
        input: ProtocolInput<'a>,
        arena: &'a ArenaRef<'a>,
    ) -> Result<(), ProtocolError>;

    /// Drain the next pending transmission.
    /// Returns None when no more transmissions are pending.
    /// Transmissions are returned in priority order (vital first).
    /// The returned Transmit borrows from the arena.
    fn poll_transmit<'a>(
        &mut self,
        arena: &'a ArenaRef<'a>,
    ) -> Option<Transmit<'a>>;

    /// Get the next timer deadline.
    /// Returns None if no timers are active.
    /// The consumer MUST call handle(TimerExpired { timer_id })
    /// when the deadline elapses.
    fn poll_timeout(&self) -> Option<TimerDeadline>;

    /// Drain the next pending application event.
    /// Returns None when no more events are pending.
    /// Events borrow from the arena.
    fn poll_event<'a>(
        &mut self,
        arena: &'a ArenaRef<'a>,
    ) -> Option<Event<'a>>;
}
```

**Calling convention (consumer event loop pseudocode):**

```
loop {
    // 1. Collect I/O
    let datagrams = socket.recvmmsg();
    for dgram in datagrams {
        let input = ProtocolInput::BytesReceived {
            source: dgram.addr.into(),
            data: dgram.payload,
        };
        match protocol.handle(input, &arena) {
            Ok(()) => {},
            Err(ProtocolError::ArenaExhausted) => {
                // drain outputs, reset arena, retry
                drain_all(&mut protocol, &arena, &socket);
                arena.reset();
                protocol.handle(input, &arena).unwrap();
            },
            Err(e) => { log_error(e); }
        }
    }

    // 2. Process expired timers
    while let Some(timer) = expired_timers.pop() {
        protocol.handle(
            ProtocolInput::TimerExpired { timer_id: timer.id },
            &arena,
        ).ok();
    }

    // 3. Drain transmissions
    while let Some(tx) = protocol.poll_transmit(&arena) {
        socket.sendto(tx.destination, tx.payload);
    }

    // 4. Drain application events
    while let Some(event) = protocol.poll_event(&arena) {
        application.handle_event(event);
    }

    // 5. Set next timer
    if let Some(deadline) = protocol.poll_timeout() {
        timer_wheel.schedule(deadline);
    }

    // 6. Reset arena for next iteration
    arena.reset();
}
```

**Buffer ownership rules:**
- `ProtocolInput::BytesReceived::data` is borrowed for the duration
  of the `handle()` call. The protocol copies what it needs into the
  arena or managed state.
- `Transmit::payload` borrows from the arena. Valid until arena.reset().
- `Event::*::payload` borrows from the arena. Valid until arena.reset().
- The consumer MUST drain all poll_transmit() and poll_event() before
  calling arena.reset().

**Backpressure:**
- If poll_transmit() returns None and the internal queue is full,
  subsequent handle() calls that would enqueue new transmissions
  return Err(ArenaExhausted).
- The consumer responds by draining outputs, resetting the arena,
  and retrying.
- Vital-tier frames are never dropped due to arena exhaustion. The
  protocol reserves a fixed portion of the arena for vital frames.
  If the vital reserve is also exhausted, the protocol enters a
  degraded mode where it drops bulk-tier frames to make room.

**Reentrancy:** handle() MUST NOT be called while a previous handle()
is in progress. The single-threaded event loop (Section 6.2) prevents
this by construction. The trait does not take `&mut self` in
poll_transmit/poll_event for flexibility, but the consumer MUST NOT
call handle() between poll_transmit/poll_event calls within the same
drain cycle. (Correction: all methods take `&mut self` to enforce
exclusive access at the type level.)

### Problem 2: Reliability Mode Selection is Unspecified (D-013)

**(a) Problem statement.**
The protocol mentions reliable-ordered and unreliable-datagram modes
but does not specify: (1) how a stream declares its mode, (2) whether
the mode is per-stream or per-frame, (3) what state each mode requires,
(4) how the mode affects SACK generation, retransmission, and flow
control.

**(b) Why it matters.**
The reliability mode determines the memory budget per stream. A
reliable-ordered stream needs a retransmission buffer (window_size *
frame_size bytes), a SACK tracking structure, and flow control state.
An unreliable-datagram stream needs only a sequence number for gap
detection. At 11 connections with potentially dozens of streams each,
the memory difference is substantial. The arena sizing (Section 6.1)
cannot be validated without knowing how many streams of each type
exist.

The reliability mode also affects the sans-IO interface. For reliable
streams, the state machine must track unacknowledged frames and
retransmit them. The retransmission timer is a new timer type that
must flow through the SetTimer output. For unreliable streams, there
is no retransmission, but the consumer needs to know about gaps
(for health scoring).

The stream tier allocation (vital/reactive/bulk) implies a mapping
to reliability modes, but this mapping is not stated. Without it,
an implementor must guess. Guessing wrong means either wasting memory
(reliable mode for fire-and-forget gossip IHAVE) or losing data
(unreliable mode for KeyBundle distribution).

**(c) References and prior art.**

- **QUIC (RFC 9000):** All streams are reliable-ordered. Unreliable
  datagrams are a separate extension (RFC 9221, DATAGRAM frame).
  QUIC chose to keep reliability out of streams and put it in a
  separate frame type.

- **WebTransport:** Supports both reliable streams and unreliable
  datagrams. Streams are always reliable-ordered. Datagrams are
  always unreliable-unordered.

- **SCTP (RFC 4960):** Streams can be ordered or unordered.
  Reliability is per-message via PR-SCTP (RFC 3758). The stream
  declaration includes an ordering flag.

- **Game networking (ENet, RakNet):** Per-channel reliability modes
  declared at channel creation. Typical modes: reliable-ordered,
  reliable-unordered, unreliable-sequenced, unreliable-unordered.

**(d) Recommended approach.**

Bind the reliability mode to the stream tier, declared at stream
creation time. The mode is fixed for the stream's lifetime. This
avoids per-frame mode switching and simplifies the state machine.

```rust
/// Reliability mode for a stream.
/// Declared at stream creation. Fixed for the stream's lifetime.
#[repr(u8)]
enum ReliabilityMode {
    /// Monotonic sequence, SACK ACKs, retransmission.
    /// Required for: control, key distribution, matter transfer.
    ReliableOrdered = 0,

    /// Per-source monotonic sequence for gap detection.
    /// No retransmission. No flow control (recv_window).
    /// Required for: heartbeat, gossip IHAVE, diagnostics.
    UnreliableDatagram = 1,
}

/// Stream creation input event.
/// The consumer declares the reliability mode.
struct StreamCreated {
    stream_id: u16,
    tier: StreamTier,
    reliability: ReliabilityMode,
}
```

**Tier-to-reliability default mapping:**

| Tier | Stream ID range | Default reliability | Rationale |
|------|----------------|-------------------|-----------|
| Control | 0x0000 | ReliableOrdered | Control must not be lost |
| Vital | 0x0001-0x00FF | ReliableOrdered | KeyBundle, enrollment |
| Reactive | 0x0100-0x0FFF | ReliableOrdered | Gossip, filter updates need delivery confirmation |
| Bulk | 0x1000-0xFFFE | ReliableOrdered | Matter transfer needs chunk-level reliability |

Heartbeats and standalone SACKs are connection-level (no stream
header). They are inherently unreliable-datagram: no retransmission,
no SACK tracking. This is already specified ("HEARTBEAT and SACK-only
frames do not carry stream headers").

The remaining use case for unreliable-datagram streams is gossip IHAVE
messages (lazy repair in Plumtree). These are explicitly designed to
tolerate loss (repair after 1 RTT). If gossip IHAVE uses a reactive-
tier stream, it gets reliable delivery, which is wasteful. Two options:

**Option A (simpler):** All streams are reliable-ordered. Gossip IHAVE
is sent on a reactive-tier stream with reliable delivery. The
retransmission overhead is negligible (IHAVE messages are small and
rare in steady state). This matches QUIC's approach.

**Option B (explicit):** Allow reactive-tier streams to be declared as
UnreliableDatagram. The consumer explicitly creates an unreliable
gossip stream. The state machine allocates no retransmission buffer.

I recommend Option A. The memory savings from unreliable gossip streams
are ~2KB per connection (one retransmission buffer). At 11 connections,
that is 22KB -- less than the pool arena's 256KB budget. The complexity
cost of supporting two reliability modes in the stream state machine
outweighs the memory savings at this scale.

**If Option A is adopted:** Remove the ReliabilityMode enum. All
streams are reliable-ordered. Heartbeats and SACKs remain
connection-level unreliable. The StreamCreated input event drops the
reliability field. The retransmission state machine is uniform across
all streams.

**Retransmission state per reliable stream:**

```rust
struct StreamRetransmitState {
    /// Sent but unacknowledged frames.
    /// Ring buffer in pool arena, max 32 entries.
    unacked: RingBuf<UnackedFrame, 32>,
    /// Current retransmission timeout.
    rto_ms: u32,
    /// Retransmission attempt count for oldest unacked.
    retransmit_count: u8,
    /// Cumulative ACK (highest in-order sequence seen by receiver).
    cumulative_ack: u32,
    /// SACK blocks from receiver.
    sack_blocks: [SackBlock; 4],
    sack_block_count: u8,
    /// Receiver window (bytes remaining).
    recv_window: u32,
}

struct UnackedFrame {
    sequence: u32,
    /// Offset into pool arena where the frame payload is stored.
    /// Valid until the frame is acknowledged or the stream is closed.
    arena_offset: u32,
    arena_len: u16,
    /// Monotonic timestamp when first sent.
    first_sent_ms: u64,
    /// Monotonic timestamp of most recent send.
    last_sent_ms: u64,
}
```

Memory per stream: ~32 * 24 bytes (UnackedFrame) + 48 bytes
(StreamRetransmitState) = 816 bytes. At 11 connections * 10 active
streams each = ~88 KB. Fits within the pool arena (256KB).

### Problem 3: MetabolicGovernor Concrete Parameter Mapping is Blank (D-012)

**(a) Problem statement.**
The Conductor section defines the governor as
`min(CPU, flash, nonce_io_pressure, min(WiFi_capacity, transport_capacity))`
and specifies that `Governed<T>` wraps all budget parameters. But the
mapping from the meet's output (a single pressure value in [0, 1]) to
MetabolicState { Background, Idle, Active, Stress } is not defined.
Without threshold bands, the governor is a formula with no output.

**(b) Why it matters.**
The MetabolicState drives heartbeat intervals (2s-30s), LEDBAT target
delays (10ms-100ms), queue budgets, and convergence behavior. Every
subsystem downstream of the governor depends on this mapping. If the
thresholds are wrong, the system either never enters Stress (missing
degradation signals) or oscillates between states (the dwell time
prevents rapid oscillation, but wrong thresholds cause persistent
misclassification).

The governor is also the input to phase jitter computation, convergence
metabolic override, and the health pipeline's damped mode. These
downstream consumers assume the governor produces meaningful state
transitions. A blank mapping means the downstream consumers are
specifying behavior conditional on an undefined trigger.

**(c) References and prior art.**

- **Linux CFS scheduler:** Maps load average to scheduling decisions
  via a table of thresholds. The table is tuned empirically.

- **TCP congestion control (Cubic):** Maps loss events to window
  adjustments via a cubic function. The function parameters are
  standardized.

- **Kubernetes HPA:** Maps CPU utilization to replica count via a
  linear function with configurable thresholds.

- **Game engine frame budgets:** Map frame time to quality settings
  via a lookup table with hysteresis.

**(d) Recommended approach.**

Define the mapping as a threshold table with hysteresis. The meet
output is a Q16.16 value in [0x00000000, 0x00010000] representing
the minimum resource pressure across all dimensions. Higher pressure
means more constrained.

```rust
/// Governor threshold table.
/// pressure_min is the lower bound (inclusive) for the state.
/// pressure_max is the upper bound (exclusive) for the state.
/// Hysteresis: transition up (to higher state) requires exceeding
/// the threshold by hysteresis_margin. Transition down requires
/// falling below by hysteresis_margin.
///
/// All values are Q16.16 fixed-point.
struct GovernorThresholds {
    /// Pressure thresholds for state transitions.
    /// Index 0: Background -> Idle boundary
    /// Index 1: Idle -> Active boundary
    /// Index 2: Active -> Stress boundary
    thresholds: [u32; 3],

    /// Hysteresis margin (same scale as thresholds).
    /// Prevents oscillation at boundaries.
    hysteresis: u32,
}

/// Default threshold values (calibration items).
const DEFAULT_THRESHOLDS: GovernorThresholds = GovernorThresholds {
    thresholds: [
        0x00003333,  // 0.20 -- Background -> Idle
        0x00008000,  // 0.50 -- Idle -> Active
        0x0000CCCC,  // 0.80 -- Active -> Stress
    ],
    hysteresis: 0x00000CCC,  // 0.05
};

/// The Conductor implementation.
/// Pure function from resource pressures to MetabolicState.
fn evaluate(
    pressures: &ResourcePressures,
    current_state: MetabolicState,
    thresholds: &GovernorThresholds,
    last_transition_ms: u64,
    now_ms: u64,
    dwell_time_ms: u64,
    convergence_override: bool,
) -> MetabolicState {
    // 1. Compute meet pressure
    let meet = min(
        pressures.cpu,
        min(
            pressures.flash,
            min(
                pressures.nonce_io,
                min(pressures.wifi_capacity, pressures.transport_capacity),
            ),
        ),
    );

    // 2. Map pressure to candidate state (with hysteresis)
    let candidate = if meet < threshold_with_hysteresis(
        thresholds.thresholds[0], thresholds.hysteresis, current_state, MetabolicState::Background
    ) {
        MetabolicState::Background
    } else if meet < threshold_with_hysteresis(
        thresholds.thresholds[1], thresholds.hysteresis, current_state, MetabolicState::Idle
    ) {
        MetabolicState::Idle
    } else if meet < threshold_with_hysteresis(
        thresholds.thresholds[2], thresholds.hysteresis, current_state, MetabolicState::Active
    ) {
        MetabolicState::Active
    } else {
        MetabolicState::Stress
    };

    // 3. Check dwell time (unless convergence override)
    if convergence_override {
        // Convergence override: skip dwell time, jump directly
        return max(candidate, MetabolicState::Active);
    }

    if candidate != current_state
        && (now_ms - last_transition_ms) < dwell_time_ms
    {
        return current_state;  // dwell time not satisfied
    }

    candidate
}
```

**ResourcePressures struct:**

```rust
/// All values are Q16.16 fixed-point in [0, 0x00010000].
/// 0x00000000 = no pressure (fully available).
/// 0x00010000 = maximum pressure (fully consumed).
#[repr(C)]
struct ResourcePressures {
    cpu: u32,
    flash: u32,
    nonce_io: u32,
    wifi_capacity: u32,
    transport_capacity: u32,
}
```

**Integration with the polling interface:** The consumer periodically
feeds `ProtocolInput::ConfigChanged { governor_output: MetabolicState }`
to the state machine. The governor evaluation happens outside the
protocol state machine (in the mesh layer), consistent with the
abstract contract. The protocol state machine receives the MetabolicState
as an opaque input and adjusts its parameters accordingly.

**Calibration.** The three threshold values and the hysteresis margin
are calibration items (extending the existing 16-item calibration
table in verkko-ops). They should be empirically tuned on target
hardware (Batocera/Raspberry Pi 4/5). The defaults above (0.20, 0.50,
0.80, 0.05 hysteresis) are starting points.

### Problem 4: Crypto Thread Interaction is Not Modeled in the SansIO Interface

**(a) Problem statement.**
Section 6.2 defines a dedicated crypto thread for Ed25519 and bao-tree
verification, connected to the event loop via SPSC ring buffers. The
sans-IO state machine runs on the event loop thread. But the
ProtocolInput enum has no variant for "crypto verification completed"
and the ProtocolOutput enum has no variant for "please verify this
signature." The async crypto path is invisible to the polling model.

**(b) Why it matters.**
Ed25519 verification of control log entries takes ~50us on ARM. Bao
verification of matter chunks takes variable time depending on chunk
size. If the event loop blocks on these operations, it cannot process
other connections' heartbeats. The SPSC ring buffer is the right
architecture, but the sans-IO interface must model it.

Without the crypto thread interaction modeled in the polling interface,
the implementor must choose between: (1) blocking the event loop on
crypto (violating the latency budget), (2) spawning the crypto work
externally and injecting results back (but how? there is no input
variant), or (3) ignoring verification (violating security). None
of these are acceptable.

**(c) References and prior art.**

- **s2n-quic:** Uses a Provider trait for crypto operations. The
  provider can be synchronous or asynchronous. The connection state
  machine models pending crypto operations as internal state and
  resumes processing when the result arrives.

- **quinn:** Ed25519 verification happens inline (fast enough on
  x86). Bao verification is not part of quinn's scope (it is
  application-layer).

- **Game engines (Unreal):** Expensive operations are queued to
  worker threads. Results arrive as events in the next game tick.
  The state machine models "pending verification" as a state.

**(d) Recommended approach.**

Add two variants to the existing enums:

```rust
// New ProtocolInput variant:
enum ProtocolInput<'a> {
    // ... existing variants ...

    /// Result of an asynchronous crypto operation.
    /// The protocol requested this via CryptoRequest output.
    CryptoResult {
        request_id: u32,
        result: CryptoVerifyResult,
    },
}

#[repr(u8)]
enum CryptoVerifyResult {
    /// Signature or proof is valid.
    Valid = 0,
    /// Signature or proof is invalid.
    Invalid = 1,
    /// Verification failed (internal error).
    Error = 2,
}

// New ProtocolOutput variant:
enum ProtocolOutput<'a> {
    // ... existing variants ...

    /// Request asynchronous crypto verification.
    /// Consumer MUST complete this and return CryptoResult.
    CryptoRequest {
        request_id: u32,
        operation: CryptoOperation<'a>,
    },
}

enum CryptoOperation<'a> {
    /// Verify Ed25519 signature.
    VerifyEd25519 {
        public_key: &'a [u8; 32],
        message: &'a [u8],
        signature: &'a [u8; 64],
    },
    /// Verify bao proof for a chunk.
    VerifyBao {
        root_hash: &'a [u8; 32],
        chunk_index: u32,
        chunk_data: &'a [u8],
        proof: &'a [u8],
    },
}
```

**State machine behavior:** When the protocol needs crypto verification
(control log entry, matter chunk), it emits CryptoRequest and enters a
"pending verification" state for that connection/stream. It continues
processing other connections normally. When CryptoResult arrives via
handle(), the protocol resumes processing the verified frame.

**Interaction with the arena:** The CryptoRequest borrows data from
the arena. The consumer MUST NOT reset the arena until all pending
CryptoRequests have been completed or canceled. This introduces a
constraint on arena lifetime: the arena epoch must span at least one
crypto round-trip. At ~50us for Ed25519 on ARM, this is negligible.
For bao verification of large chunks, the constraint is more
significant.

**Alternative considered:** Model AEAD encryption/decryption as
synchronous (inline in handle()). This is appropriate for ChaCha20-
Poly1305 which completes in ~1us. Only Ed25519 and bao verification
are slow enough to warrant async offloading.

### Problem 5: Timer Management is Underspecified

**(a) Problem statement.**
The protocol emits SetTimer { timer_id, deadline_ms } as output and
consumes TimerExpired { timer_id } as input. But timer_id is an opaque
u32 with no allocation strategy, no collision handling, and no
cancellation mechanism. The consumer does not know how many timers to
expect, whether timer IDs are reused, or how to cancel a timer that
is no longer relevant.

**(b) Why it matters.**
The protocol needs at least 7 timer types per connection: heartbeat,
SACK retransmission (per stream), RTO (per stream), path challenge,
CPSK renegotiation, connection close drain, and metabolic dwell. At
11 connections with 10 streams each, that is potentially ~130 active
timers. If timer IDs are monotonically increasing u32 values, the
consumer needs a timer wheel or priority queue indexed by timer_id.
If timer IDs encode the connection and timer type, the consumer can
use a simpler data structure.

Timer cancellation is the critical missing piece. When a stream is
closed, its retransmission timers must be canceled. When a SACK
acknowledges all unacked frames, the RTO timer must be canceled.
Without a CancelTimer output event, the consumer fires stale timers,
causing the protocol to process spurious TimerExpired events. The
protocol must then determine whether the timer is still relevant,
adding complexity to handle().

**(c) References and prior art.**

- **quinn:** Uses an enum of timer types per connection. The
  connection tracks the earliest deadline and reports it via
  `poll_timeout() -> Option<Instant>`. The consumer only tracks one
  deadline per connection. On expiry, the consumer calls
  `handle_timeout(now)` and the connection processes all expired
  timers internally.

- **smoltcp:** Similar to quinn. One timer per interface. The
  interface manages all internal timers.

- **QUIC general:** Timer management is internal to the connection.
  The consumer only needs the next deadline.

- **Tokio timer wheel:** Hierarchical timer wheel with O(1) insert
  and O(1) expire. Used by tokio-runtime. Not applicable here (we
  are sans-IO) but the data structure is relevant.

**(d) Recommended approach.**

Replace the SetTimer/TimerExpired pair with a single poll_timeout()
method on the Protocol trait (already proposed in Problem 1). The
protocol manages all timers internally. The consumer only needs to
track one deadline: the earliest across all connections.

```rust
trait Protocol {
    // ... other methods ...

    /// Get the next timer deadline across all connections.
    /// Returns None if no timers are active.
    /// The consumer calls handle(TimerExpired { now_ms })
    /// when this deadline elapses.
    fn poll_timeout(&self) -> Option<u64>;

    /// Process all timers that have expired at or before now_ms.
    /// This replaces the per-timer TimerExpired input.
    fn handle_timeout(&mut self, now_ms: u64, arena: &ArenaRef)
        -> Result<(), ProtocolError>;
}
```

**Internal timer management:** The protocol uses a timer wheel (4
levels, 256 slots each, 1ms resolution at level 0). Timer entries
are allocated from the managed arena. Each connection has a fixed
set of timer slots:

```rust
/// Timer types per connection.
/// Each connection has exactly one of each.
#[repr(u8)]
enum ConnectionTimer {
    Heartbeat = 0,
    PathChallenge = 1,
    CpskRehandshake = 2,
    CloseTimeout = 3,
    DwellTimeout = 4,
}

/// Timer types per stream.
/// Each stream has at most one of each.
#[repr(u8)]
enum StreamTimer {
    Retransmit = 0,
    IdleTimeout = 1,
}
```

Timer IDs are computed from (connection_index, timer_type, stream_id):
```
timer_id = (connection_index << 16) | (timer_type << 8) | stream_id
```

This makes timer ID allocation collision-free by construction. Timer
cancellation is internal: closing a stream cancels its timers; closing
a connection cancels all its timers.

The consumer's responsibility reduces to:
1. Call `protocol.poll_timeout()` to get the next deadline.
2. Sleep until that deadline (or until I/O arrives).
3. Call `protocol.handle_timeout(now_ms, &arena)` when the deadline
   elapses.

This is identical to quinn's pattern and has proven practical at scale.

## Part 3: Open Items and Known Limitations

### Open items

1. **Batch AEAD thread interaction.** Section 6.2 describes a
   dedicated batch AEAD thread with its own NonceSource. The sans-IO
   interface does not model this. For re-encryption batches and
   donation batches, the protocol needs to hand off AEAD work to the
   batch thread and receive results. This is analogous to Problem 4
   (crypto thread) but for symmetric crypto. The latency of ChaCha20-
   Poly1305 (~1us per frame) suggests inline AEAD is acceptable for
   single frames, but batch operations (re-encryption of hundreds of
   blobs) must be offloaded. The recommended approach is the same as
   Problem 4: CryptoRequest/CryptoResult for batch AEAD operations.

2. **recvmmsg batch processing.** The event loop uses recvmmsg to
   receive multiple datagrams at once. The hot arena must be large
   enough to hold all datagrams in a batch plus all generated outputs.
   The current 64KB hot arena can hold ~52 datagrams (64KB / 1232
   bytes). If recvmmsg returns more than 52 datagrams, the arena
   overflows. The calling convention must specify the maximum batch
   size or the arena must be dynamically sized. I recommend capping
   recvmmsg batch size at 32 (39KB of input), leaving 25KB for
   output buffers.

3. **WebSocket framing interaction.** The SeatTransport specified
   that WebSocket binary messages carry exactly one verkko frame.
   This simplifies parsing (no reassembly) but the sans-IO interface
   must handle the case where a WebSocket message is delivered in
   fragments by the underlying TCP stack. The design states the
   protocol MUST NOT assume one BytesReceived = one frame. The
   parser needs a reassembly buffer per source address. This is not
   arena-allocated (it persists across arena resets); it belongs in
   managed state.

4. **Connection state machine diagram.** The connection lifecycle
   has at least 6 states: VersionProbe, Handshaking, Established,
   Migrating, Closing, Closed. The transitions between these states
   are implied by the wire formats but not drawn explicitly. A state
   machine diagram would clarify edge cases (e.g., what happens if
   a data frame arrives during VersionProbe? Answer: drop it).

5. **Stream state machine diagram.** A stream has states: Idle (not
   yet created), Open (data flowing), HalfClosed (FIN sent or
   received), Closed (both FIN exchanged or RST received). The
   interaction between stream state and reliability mode (if
   unreliable modes are supported) adds complexity.

6. **ProtocolOutput ordering.** When handle() generates multiple
   outputs (e.g., a SACK and a data frame), the order in which
   poll_transmit() returns them matters. The VitalQueue defines the
   priority order, but the contract should state that poll_transmit()
   returns frames in priority order within a single drain cycle.

7. **Error recovery semantics.** ProtocolError::CryptoFailure means
   a received frame had an invalid auth tag. The connection should
   survive (drop the frame, maybe count toward health scoring). But
   ProtocolError::ProtocolViolation means the remote peer is
   misbehaving and the connection should be torn down. The error
   type needs a severity classification.

### Known limitations

1. **Arena-lifetime output coupling.** The proposed design ties output
   lifetimes to the arena epoch. This means the consumer cannot hold
   a reference to a received frame payload across arena resets. If
   the application needs to process a frame asynchronously, it must
   copy the data out of the arena. This is a deliberate tradeoff:
   zero-copy within a single event loop iteration, copy at the
   application boundary.

2. **Single-threaded assumption.** The Protocol trait takes `&mut self`,
   enforcing single-threaded access. Multi-threaded consumers must
   use a mutex or channel to serialize access. This is consistent with
   Section 6.2's single-threaded event loop design but limits
   scalability for consumers that want to process multiple connections
   in parallel. At the design envelope (11 inter-cluster connections +
   6 intra-cluster connections + seats), single-threaded is sufficient.

3. **No zero-copy for small frames.** Heartbeats (91 bytes) and SACK
   frames (~18-146 bytes) are small enough that the copy into/out of
   the arena is not a bottleneck. The arena pattern optimizes for
   large frames (matter chunks at ~1170 bytes) and batches.

4. **Timer precision.** The timer wheel at 1ms resolution is
   sufficient for heartbeat intervals (2000-30000ms) and RTO
   (1000-30000ms) but may be too coarse for LEDBAT measurements
   (which need sub-millisecond precision). LEDBAT timing should use
   kernel timestamps (SO_TIMESTAMPING) directly, not the timer wheel.

## Part 4: Ideal Design Synthesis

The ideal sans-IO interface for verkko integrates the following
principles:

### 4.1 Four-method polling model

The consumer drives the protocol via four methods:
- `handle(input, arena)` -- process one input event
- `poll_transmit(arena)` -- drain next transmission
- `poll_event(arena)` -- drain next application event
- `poll_timeout()` -- get next timer deadline

Plus a fifth method for timer processing:
- `handle_timeout(now, arena)` -- process all expired timers

This is the quinn pattern, adapted for arena allocation. It has
proven robust across thousands of concurrent connections in
production QUIC implementations.

### 4.2 Arena-epoch buffer management

All short-lived buffers (input parsing, output framing, crypto
intermediates) are allocated from the hot arena. The hot arena is
reset once per event loop iteration, after all outputs are drained.
Long-lived state (connection tables, stream state, SACK tracking)
is allocated from the managed arena, which is not reset.

The pool arena handles medium-lived allocations: retransmission
buffers, pending verification queues, and assembly buffers. The pool
arena uses fixed-block allocation (slab allocator) for predictable
performance.

### 4.3 Async crypto via output/input events

Expensive crypto operations (Ed25519, bao) are modeled as
CryptoRequest outputs and CryptoResult inputs. The protocol state
machine remains non-blocking. The consumer routes CryptoRequests to
the crypto thread (SPSC ring buffer) and feeds CryptoResults back
as ProtocolInput events.

### 4.4 Internal timer management

The protocol manages all timers internally. The consumer only needs
the next deadline. Timer IDs are computed from connection index,
timer type, and stream ID. Timer cancellation is automatic on
connection/stream close.

### 4.5 Uniform reliability

All streams are reliable-ordered. Heartbeats and SACKs are
connection-level unreliable (no stream header). This simplifies
the state machine and matches QUIC's proven approach. The memory
cost is acceptable at the design envelope (88KB for retransmit
state across all connections and streams, within the 256KB pool
arena).

### 4.6 MetabolicGovernor with threshold table

The governor maps the meet of resource pressures to MetabolicState
via a threshold table with hysteresis. Three thresholds (0.20, 0.50,
0.80) and one hysteresis margin (0.05) are calibration items. All
values are Q16.16 fixed-point. The governor evaluation is a pure
function, deterministic, and takes ~150ns (as budgeted in Section 8).

### 4.7 Error stratification

Protocol errors are classified by severity:
- **Transient:** frame dropped, connection continues (invalid auth
  tag, unknown stream, arena exhaustion with retry).
- **Connection-fatal:** connection destroyed, other connections
  continue (protocol violation, nonce exhaustion per-session).
- **Global-fatal:** all connections destroyed (nonce persist failure,
  arena corruption).

The error type encodes severity, enabling the consumer to make
recovery decisions without inspecting error details.

### 4.8 Concrete state machine diagram

The complete sans-IO consumer interaction follows this cycle:

```
                     +---------------------------+
                     |    Consumer Event Loop     |
                     +---------------------------+
                               |
                     +---------v---------+
                     | recvmmsg / select  |
                     +---+--------+------+
                         |        |
               (datagrams)  (timer expired)
                         |        |
                +--------v--+  +--v-----------+
                | handle()   |  | handle_timeout|
                | per dgram  |  | (all expired) |
                +--------+--+  +--+-----------+
                         |        |
                     +---v--------v---+
                     | poll_transmit() |<--+
                     | (loop until    |   |
                     |  None)         |---+
                     +-------+--------+
                             |
                     +-------v--------+
                     | poll_event()   |<--+
                     | (loop until    |   |
                     |  None)         |---+
                     +-------+--------+
                             |
                     +-------v--------+
                     | poll_timeout() |
                     | set next timer |
                     +-------+--------+
                             |
                     +-------v--------+
                     | arena.reset()  |
                     +----------------+
```

**Invariant:** Between arena.reset() and the next recvmmsg, no
references into the arena exist. This is enforced by the lifetime
parameter on ArenaRef.

### 4.9 Wire format interaction

The polling model is transparent to the wire format. The consumer
feeds raw bytes in; the protocol parses them according to Section 3.1.
The consumer receives Transmit structs with raw bytes out; it sends
them via sendmmsg. The internal VitalQueue determines the order.
The internal SACK generator and retransmission engine handle
reliability. The consumer never sees SACK state, retransmission
decisions, or congestion control adjustments.

### 4.10 Seat interaction

The seat (WebSocket client) uses the same Protocol trait. The
consumer wraps the WebSocket connection: BytesReceived delivers one
WebSocket binary message, Transmit is sent as one WebSocket binary
message. The protocol's internal SACK and retransmission are
redundant (WebSocket/TCP provides reliability) but harmless. The
implementation MAY optimize by disabling retransmission timers for
connections known to be on reliable transports, as stated in
SeatTransport.

### 4.11 Integration with existing design elements

| Existing concept | Sans-IO integration |
|-----------------|-------------------|
| VitalQueue | Internal to poll_transmit(). Consumer sees priority-ordered output. |
| SACK | Internal. Consumer sees nothing except health events. |
| LEDBAT | Internal. Affects poll_transmit() pacing. Consumer provides SO_TIMESTAMP via handle(). |
| MetabolicState | Received via ConfigChanged input. Adjusts internal parameters. |
| NonceSafety | Internal. Protocol manages NonceSource. Emits ProtocolError::NonceHalted on persist failure. |
| Heartbeat | Internal generation on HeartbeatTick input. External delivery via poll_event(HeartbeatReceived). |
| Connection migration | Internal PATH_CHALLENGE/PATH_RESPONSE. Consumer sees address change via updated Transmit destinations. |
| Stream multiplexing | Internal. Consumer creates streams via StreamCreated input, receives data via FrameReady event. |

### 4.12 What this synthesis does not cover

- **Intra-cluster communication (D-004).** Outside my domain. The
  dest_peer field is a wire format detail. The session model is a
  mesh-layer concern.
- **Convergence protocol details.** Outside my domain except where
  they interact with the polling model (gossip mode switching,
  metabolic override). The convergence fronts and gates are mesh-
  layer state machines that consume the protocol's output events.
- **Relay operations.** Outside my domain except for the batch AEAD
  interaction (noted in open items).
- **Matter transfer orchestration.** Outside my domain. The
  TransferSession and MatterChunk are application-layer concerns
  that use the stream abstraction.

## Summary of Recommendations

| Problem | Severity | Recommendation | Lines of spec |
|---------|----------|---------------|---------------|
| P1: No polling interface | CRITICAL | Four-method Protocol trait with arena parameter | ~120 |
| P2: Reliability modes unspecified | HIGH | All streams reliable-ordered; heartbeats unreliable connection-level | ~40 |
| P3: Governor mapping blank | HIGH | Threshold table with hysteresis, Q16.16 fixed-point | ~60 |
| P4: Crypto thread not modeled | HIGH | CryptoRequest/CryptoResult variants in ProtocolInput/Output | ~50 |
| P5: Timer management underspecified | MEDIUM | Internal timer wheel, single poll_timeout() method | ~40 |

Total new specification text: ~310 lines of normative content.

All five recommendations are compatible with each other and with the
existing design. None require new DAG edges. None require changes to
wire formats. They fill the interior of the sans-IO state machine
without disturbing the protocol's external surface.

The strongest counterargument to this entire approach is that it
over-constrains the implementation. A sans-IO interface should be
minimal, and the four-method polling model is one of many valid
patterns. My response: the whole point of sans-IO is that the
interface IS the specification. Without a concrete interface, sans-IO
is just a buzzword. The four-method pattern is proven (quinn, smoltcp,
s2n-quic) and its constraints are precisely the constraints that make
independent implementations interoperable at the API level.
