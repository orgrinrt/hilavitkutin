# Deep Dive: Maud -- Compile-Time HTML Templates for Rust

## 1. How Maud Works

Maud is a procedural macro crate (`html!`) that compiles HTML markup into Rust code at build
time. Rather than parsing template files at runtime or interpreting a template DSL, Maud's proc
macro transforms an HTML-like DSL embedded directly in Rust source into a sequence of
`fmt::Write` calls that write string slices into a pre-allocated buffer. The result is a
`Markup` value (a newtype around `String`) containing the rendered HTML.

The key mechanism:

1. The `html!` macro receives tokens representing the template body.
2. A proc-macro parser walks the token stream, recognizing elements, attributes, text,
   splices (Rust expressions in parentheses), and control structures.
3. Static text segments are concatenated into string literals at compile time. Dynamic splices
   generate `write!()` / `write_str()` calls with automatic HTML escaping.
4. The compiler type-checks the entire expansion, catching typos in variable names, type
   mismatches, and borrow errors before the binary is produced.
5. The final binary contains only the pre-baked string constants and the write calls -- no
   parser, no template AST, no interpreter.

### Syntax Overview

```rust
use maud::{html, Markup, DOCTYPE};

fn page(title: &str, items: &[String]) -> Markup {
    html! {
        (DOCTYPE)
        html lang="en" {
            head {
                meta charset="UTF-8";
                title { (title) }
            }
            body {
                h1 { "Items" }
                @if items.is_empty() {
                    p { "No items yet." }
                } @else {
                    ul {
                        @for item in items {
                            li { (item) }
                        }
                    }
                }
            }
        }
    }
}
```

Key syntactic elements:

- **Elements**: `div { ... }` -- braces delimit children; void elements use semicolons (`br;`,
  `meta charset="UTF-8";`).
- **Attributes**: `a href="/" class="nav" { "Home" }` -- key=value pairs before the brace.
- **Classes/IDs shorthand**: `div.container#main { ... }`.
- **Conditional classes**: `div.active[is_active] { ... }` -- the class is only emitted when
  the boolean is true.
- **Splices**: `(expression)` -- evaluates a Rust expression, HTML-escapes the result, and
  writes it. Works in text position and attribute-value position.
- **Control flow**: `@if`, `@else`, `@for .. in ..`, `@match`, `@let` -- all prefixed with `@`
  and following Rust syntax inside.
- **PreEscaped**: `(PreEscaped(raw_html))` -- bypasses escaping. Only use with trusted content.
- **DOCTYPE**: The constant `maud::DOCTYPE` emits `<!DOCTYPE html>`.

Maud v0.27+ works on stable Rust. Earlier versions required nightly, but the proc-macro API
stabilisation resolved this.

## 2. Component and Fragment Patterns

Maud has no built-in "component" or "partial" concept. Instead, it relies on Rust's own
abstraction mechanisms -- functions and trait implementations -- to compose markup. This turns
out to be more powerful than most template-inheritance systems because you get the full Rust
type system.

### Functions as Components

The simplest pattern: a function that returns `Markup`.

```rust
fn nav_link(href: &str, label: &str, active: bool) -> Markup {
    html! {
        a href=(href) .nav-link .active[active] { (label) }
    }
}
```

This can be spliced into any template: `(nav_link("/about", "About", false))`.

### Structs as Components via the Render Trait

For components with richer data, implement `Render` on a struct:

```rust
use maud::{html, Markup, Render};

struct Card<'a> {
    title: &'a str,
    body: Markup,
    footer: Option<Markup>,
}

impl Render for Card<'_> {
    fn render(&self) -> Markup {
        html! {
            div.card {
                div.card-header { (self.title) }
                div.card-body { (self.body) }
                @if let Some(ref f) = self.footer {
                    div.card-footer { (f) }
                }
            }
        }
    }
}
```

Then in a template: `(card)` where `card` is a `Card` value. The compiler enforces that every
field is populated.

### Layout Functions (Base Template Pattern)

Since there is no template inheritance, shared layouts are plain functions that accept content
as `Markup`:

```rust
fn base_layout(title: &str, content: Markup) -> Markup {
    html! {
        (DOCTYPE)
        html lang="en" {
            head {
                meta charset="UTF-8";
                meta name="viewport" content="width=device-width, initial-scale=1.0";
                title { (title) }
                link rel="stylesheet" href="/static/styles.css";
                script src="/static/htmx.min.js" defer {}
            }
            body {
                (header_component())
                main.container { (content) }
                (footer_component())
            }
        }
    }
}
```

Page handlers compose into this:

```rust
fn home_page(data: &HomeData) -> Markup {
    base_layout("Home", html! {
        h1 { "Welcome" }
        (data.featured_section)
    })
}
```

## 3. The Render Trait and Composing Markup

The `Render` trait is the core composition mechanism:

```rust
pub trait Render {
    fn render(&self) -> Markup;
    fn render_to(&self, buffer: &mut String) { ... }
}
```

You override either `render()` (returning `Markup`) or `render_to()` (writing directly into a
buffer to avoid an extra allocation). The default implementations call each other, so you must
implement at least one.

Built-in implementations exist for:

- `&str`, `String` -- HTML-escaped text output.
- All integer and float types -- rendered as decimal text.
- `Option<T: Render>` -- renders the inner value or nothing.
- `Markup` itself -- spliced verbatim (already escaped/trusted).

This means you can splice any `Render`-implementing type directly with `(value)` syntax. The
trait is object-safe enough that you can store `Box<dyn Render>` for heterogeneous component
lists, though in practice generic or enum dispatch is preferred in Rust.

### Composition Example

```rust
struct Page {
    title: String,
    breadcrumbs: Vec<Breadcrumb>,
    body: Markup,
}

struct Breadcrumb { label: String, href: String }

impl Render for Breadcrumb {
    fn render(&self) -> Markup {
        html! { li { a href=(self.href) { (self.label) } } }
    }
}

impl Render for Page {
    fn render(&self) -> Markup {
        base_layout(&self.title, html! {
            nav.breadcrumbs {
                ol {
                    @for crumb in &self.breadcrumbs {
                        (crumb)
                    }
                }
            }
            (self.body)
        })
    }
}
```

### Analysis

Maud is among the fastest engines measured, consistently 4-5x faster than Askama and 10-12x
faster than Tera. Sailfish and Markup are the only engines that beat it, and only in the big
table benchmark -- Maud takes the crown on the teams benchmark.

The performance advantage comes from Maud's approach of emitting direct `write_str()` calls
for static segments. Because the proc macro concatenates adjacent string literals at compile
time, the generated code has very few write calls, minimising overhead. Askama, by contrast,
generates code from external `.html` template files using a Jinja2-like parser, which
produces more granular write calls.

Tera and MiniJinja are runtime engines (Jinja2-style), parsing templates on first use and
interpreting them on each render. This adds 10-50x overhead but enables runtime template
loading, hot reload, and user-editable templates.

**For saalis**: since templates are compiled into the binary and hot-reload is not a
requirement (the Batocera appliance ships a single binary), Maud's compile-time approach
imposes no practical cost and delivers the fastest rendering.

## 5. Compile-Time HTML Validation Guarantees

Maud provides several categories of compile-time safety:

**Structural correctness**: The macro syntax enforces matched braces for element nesting. You
cannot accidentally leave a `<div>` unclosed -- the Rust compiler will reject the token stream
if braces are unbalanced. Void elements (`br`, `meta`, `input`) use semicolons, preventing the
common error of giving them children.

**Type safety on splices**: Every `(expression)` in a template is type-checked by rustc.
If a splice references a nonexistent variable, uses a wrong type, or violates borrow rules,
compilation fails with a standard Rust error pointing at the template line.

**Automatic HTML escaping**: All spliced values pass through HTML entity encoding by default
(`<` becomes `&lt;`, `&` becomes `&amp;`, etc.). This provides systematic XSS protection. The
only bypass is the explicit `PreEscaped` wrapper, which serves as a visible code-review signal.

**What Maud does NOT validate at compile time**:

- It does not check that element or attribute names are valid HTML. You can write
  `frobnicate.xyz { }` and it will happily emit `<frobnicate class="xyz"></frobnicate>`.
- It does not enforce HTML semantics (e.g., `<li>` inside `<ul>`, required attributes on
  elements).
- It does not perform context-aware escaping -- the same entity-encoding is used in attribute
  values and text nodes. There is an open issue (#181) for context-aware escaping (e.g.,
  JavaScript contexts), but it has not been implemented.
- It does not validate CSS class names or ID values.

In practice, Maud catches the most common and dangerous classes of template errors (unclosed
tags, XSS from unsanitized data, type errors) while leaving HTML semantic validation to
linters or browser dev tools.

## 6. Integration with HTMX

HTMX and Maud form a natural pairing. HTMX issues HTTP requests and swaps the returned HTML
fragment into the DOM. Maud excels at producing these fragments because every axum handler can
return `Markup` directly.

### Full Page vs Fragment Pattern

The standard approach uses the `HX-Request` or `HX-Boosted` header (from the `axum-htmx`
crate) to decide whether to return a full page or a bare fragment:

```rust
use axum::extract::Query;
use axum_htmx::HxRequest;
use maud::{html, Markup};

async fn items_list(HxRequest(is_htmx): HxRequest, /* ... */) -> Markup {
    let items_fragment = html! {
        @for item in &items {
            li.item hx-delete={"/items/" (item.id)} hx-target="closest li"
                    hx-swap="outerHTML" {
                (item.name)
                button.delete { "x" }
            }
        }
    };

    if is_htmx {
        items_fragment  // bare fragment for HTMX swap
    } else {
        base_layout("Items", html! {
            h1 { "Items" }
            ul#items-list { (items_fragment) }
        })
    }
}
```

### HTMX Attributes in Maud Syntax

HTMX data attributes are written as regular attributes:

```rust
html! {
    button hx-post="/api/action"
           hx-target="#result"
           hx-swap="innerHTML"
           hx-indicator="#spinner" {
        "Do Thing"
    }
}
```

### Supporting Crates

- **axum-htmx** (crate): Provides extractors (`HxRequest`, `HxBoosted`, `HxTarget`) and
  response headers (`HxPushUrl`, `HxRetarget`, `HxTrigger`) for axum handlers.
- Maud's axum feature (`maud = { features = ["axum"] }`) implements `IntoResponse` for
  `Markup`, so handlers can return it directly.

### The MASH / HARM Stack

The community has named this combination:

- **MASH**: Maud + Axum + SQLx + HTMX
- **HARM**: HTMX + Axum/AlpineJS + Rust + Maud

Both acronyms describe the same core pattern: server-rendered HTML fragments with HTMX-driven
interactivity, no JavaScript framework, no JSON API, no client-side state management.

## 7. Internationalization (i18n) Approaches

Maud has no built-in i18n support. Since templates are Rust code, you integrate whichever i18n
crate you choose by calling it from splice expressions. The main options:

### fluent-rs + fluent-templates

Project Fluent (`.ftl` files) is the most capable option for natural-language-aware
localisation (gender, plurals, conjugations). The `fluent-templates` crate provides:

- `static_loader!` -- loads `.ftl` resources at compile time into the binary.
- `ArcLoader` -- loads at runtime for hot-reloadable translations.
- `fl!()` macro -- resolves a message ID with compile-time key validation.

Integration with Maud:

```rust
use fluent_templates::{static_loader, Loader};
use unic_langid::langid;

static_loader! {
    static LOCALES = {
        locales: "./locales",
        fallback_language: "en",
    };
}

fn greeting(name: &str, lang: &str) -> Markup {
    let lang_id = lang.parse().unwrap_or(langid!("en"));
    let mut args = std::collections::HashMap::new();
    args.insert("name", name.into());
    html! {
        p { (LOCALES.lookup_with_args(&lang_id, "greeting", &args)) }
    }
}
```

### rust-i18n

The `rust-i18n` crate offers a simpler `t!()` macro that works well in Maud splices:

```rust
html! {
    h1 { (t!("page.title")) }
    p  { (t!("page.welcome", name = user.name)) }
}
```

### Pattern for saalis

Since saalis compiles everything into the binary, `fluent-templates::static_loader!` is the
natural fit: `.ftl` files are embedded at compile time (no runtime file I/O), the `fl!()` /
`lookup` calls work inside Maud splices, and Fluent's plural/gender rules handle the linguistic
complexity that simple key-value stores cannot.

## 8. Limitations -- What Maud Cannot Do

**No runtime template loading**: Templates are compiled into the binary. You cannot load them
from a database, filesystem, or CMS at runtime. If user-editable templates are needed (e.g., a
theming system where end users write templates), a runtime engine like Tera or MiniJinja is
required.

**No hot reload**: Changing a template requires recompilation. During development, this means
waiting for `cargo build`. Tools like `cargo-watch` mitigate this, but incremental compile
times for proc-macro-heavy crates can still be 2-5 seconds.

**No template inheritance**: There is no `{% extends "base.html" %}` or `{% block content %}`
mechanism. Layouts must be composed via function calls. This is arguably more explicit and
flexible, but it differs from what developers coming from Jinja2/Django/Rails expect.

**No external template files**: Templates live inline in `.rs` files. Designers who work in
HTML files cannot directly edit Maud templates. There is no tooling to convert `.html` files to
Maud syntax automatically.

**No context-aware escaping**: The same HTML entity encoding applies everywhere. Content
injected into JavaScript blocks, CSS, or URL parameters is not given context-appropriate
escaping. Developers must handle these cases manually with `PreEscaped` or dedicated encoding
functions.

**No semantic HTML validation**: Maud does not know or enforce HTML5 content models. It will
not warn about `<div>` inside `<p>`, missing `alt` on `<img>`, or invalid attribute names.

**Increased compile times**: Heavy use of proc macros adds to build times. For large projects
with hundreds of templates, this can become noticeable compared to runtime engines that have
zero compile-time cost for templates.

**DSL learning curve**: The syntax, while close to HTML, is not HTML. Developers must learn the
`@if` / `@for` / `(splice)` conventions. Copy-pasting HTML from a browser inspector or design
tool requires manual conversion.

## 9. Real-World Usage and Open-Source Projects

Maud has over 2.4 million downloads on crates.io (v0.27.0 as of writing) and a steady
maintenance cadence. Notable usage:

- **Rustodon** -- an early ActivityPub/Mastodon-compatible server in Rust used Maud for its web
  interface.
- **hadamove/todo-maud-htmx** -- a canonical example of the MASH stack: server-side rendered
  todo app with HTMX reactivity, Maud templating, and Tailwind styling.
- **SoTeKie/htmx-axum-maud** -- a starter template demonstrating the HTMX + Axum + Maud
  integration pattern.
- **Scour** (Evan Schwartz) -- built with the MASH stack as a real-world demonstration of the
  architecture.
- **pagetop** -- a web framework that uses Maud internally for its HTML rendering layer.
- Various meta-search engines and personal sites use Maud with Actix-web or Axum.

The community is small but active, with the HARM/MASH stack gaining traction in 2025 as a
reaction against JavaScript-heavy frontends. The pattern appeals to developers who want a
single Rust binary with no Node.js build step.

## 10. Structuring a Maud-Based UI for saalis

Given saalis's architecture (single binary, axum web layer, HTMX-driven UI), here is a
recommended file/module structure:

```
saalis-web/
  src/
    templates/
      mod.rs            // re-exports, shared types
      layout.rs         // base_layout(), fragment_layout()
      components/
        mod.rs
        nav.rs          // navbar, sidebar
        card.rs         // content cards
        table.rs        // sortable data tables
        modal.rs        // dialog/modal wrappers
        form.rs         // form field helpers
      pages/
        mod.rs
        home.rs         // home/dashboard page
        library.rs      // library browse/search
        entity.rs       // entity detail view
        settings.rs     // settings pages
      fragments/
        mod.rs
        entity_list.rs  // HTMX partial: entity list rows
        metadata.rs     // HTMX partial: metadata panel
        search.rs       // HTMX partial: search results
    handlers/
      ...               // axum handlers call into templates/
```

### Layout Layer

```rust
// templates/layout.rs
pub fn base(title: &str, content: Markup) -> Markup {
    html! {
        (DOCTYPE)
        html lang="en" {
            head {
                meta charset="UTF-8";
                meta name="viewport" content="width=device-width, initial-scale=1.0";
                title { (title) " | saalis" }
                link rel="stylesheet" href="/static/css/app.css";
                script src="/static/js/htmx.min.js" defer {}
            }
            body hx-boost="true" {
                (super::components::nav::sidebar())
                main#content { (content) }
            }
        }
    }
}
```

### Handler Pattern

```rust
// handlers/library.rs
async fn library_page(
    HxRequest(is_htmx): HxRequest,
    State(db): State<AppState>,
) -> Markup {
    let entities = db.list_entities().await;
    let content = templates::fragments::entity_list::render(&entities);

    if is_htmx {
        content
    } else {
        templates::layout::base("Library", html! {
            h1 { "Library" }
            div#entity-list { (content) }
        })
    }
}
```

### Component Reuse

Components are just functions or Render impls. They compose freely:

```rust
// templates/components/card.rs
pub fn entity_card(entity: &Entity) -> Markup {
    html! {
        div.card hx-get={"/entities/" (entity.id)}
                 hx-target="#detail-panel"
                 hx-swap="innerHTML" {
            img.card-img src=(entity.thumbnail_url) alt=(entity.name);
            div.card-body {
                h3 { (entity.name) }
                span.badge { (entity.entity_type) }
            }
        }
    }
}
```

### Fragment Pattern for HTMX

Fragments are small templates that return only the HTML that HTMX will swap. They never wrap
in the full layout. The handler decides whether to wrap:

```rust
// templates/fragments/search.rs
pub fn search_results(results: &[SearchResult]) -> Markup {
    html! {
        @for result in results {
            (entity_card(&result.entity))
        }
        @if results.is_empty() {
            p.empty-state { "No results found." }
        }
    }
}
```

This structure keeps templates close to their data, enables full Rust type checking across
every component boundary, and aligns with the HTMX fragment-swapping model where each
endpoint can serve either a full page or a bare fragment based on the request headers.
