# Deep Dive: Concurrency Primitives for a Sync-Core Architecture

**Date:** 2026-03-15
**Scope:** Locking, channels, atomics, scheduling models, and deadlock prevention for saalis's sync-core + async-shell design with in-memory columnar store and background scheduler.

---

## 1. parking_lot vs std::sync

### Performance

On x86_64 Linux, `parking_lot::Mutex` benchmarks at roughly 1.5x faster than `std::sync::Mutex` uncontended, and up to 5x faster under heavy contention from multiple threads. `parking_lot::RwLock` results vary by reader/writer ratio, but are almost always faster than `std::sync::RwLock` -- up to 50x faster in some read-dominated scenarios.

The performance advantage comes from parking_lot's architecture: it manages thread queues in user space via a global hash table (the "parking lot," originally from WebKit) indexed by the address of the lock. This avoids the overhead of pthread wrappers that std::sync historically relies on.

### Storage Efficiency

- `parking_lot::Mutex` and `Once`: 1 byte
- `parking_lot::RwLock` and `Condvar`: 1 word (usize)
- `std::sync::Mutex`: typically much larger due to OS-level pthread structures

This matters for a columnar store where you might have per-column or per-shard locks -- hundreds of lock instances where size adds up.

### Fairness: Task-Fair with Eventual Fairness

parking_lot's `RwLock` uses a **task-fair** locking policy that avoids both reader and writer starvation. This means readers trying to acquire the lock will block even if the lock is currently unlocked when there are writers waiting. This is fundamentally different from the "read-priority" or "write-priority" dichotomy in traditional implementations.

The mechanism is **eventual fairness**: a fair unlock is forced on average every 0.5ms. Any critical section longer than 1ms always gets a fair unlock. This strikes a balance -- short critical sections get maximum throughput (unfair fast-path), while longer-held locks or contended scenarios fall back to fair ordering. You can also force fair unlocking explicitly via `RwLockReadGuard::unlock_fair()` / `RwLockWriteGuard::unlock_fair()`.

### std::sync Improvements Tracking

The Rust libs team has been working on improving std's locks (tracking issue rust-lang/rust#93740). The plan involves replacing pthread-based implementations with custom, movable, allocation-free primitives. An attempt to merge parking_lot directly into std stalled because parking_lot's global hash table requires memory allocation, creating a cyclic dependency (can't allocate without locking, can't lock without the hash table). The std improvements take a different approach. As of early 2026, parking_lot still offers meaningful advantages over std::sync, particularly RwLock fairness and the richer API surface (try_lock_for, upgradable reads, fair unlock).

### Recommendation for saalis

Use `parking_lot` for all internal synchronization. The task-fair RwLock is critical for the columnar store (readers must not starve writers doing metadata updates) and the 1-byte Mutex is ideal for per-entity or per-shard locks.

## 2. DashMap and Concurrent HashMaps

### DashMap Internals

DashMap is a concurrent `HashMap` replacement built on **sharding**. Internally it holds `Box<[CachePadded<RwLock<HashMap<K, V>>>]>` -- an array of cache-line-padded shards, each guarded by its own RwLock. Keys are hashed to determine which shard to lock, so operations on different shards proceed in parallel without contention.

### When to Use DashMap vs Mutex<HashMap>

| Scenario | Recommendation |
|---|---|
| High read concurrency, infrequent writes | DashMap or papaya |
| Mixed read/write from many threads | DashMap |
| Single-threaded or low-contention | `HashMap` behind a single lock |
| Need to iterate while mutating | `Mutex<HashMap>` (DashMap iteration holds shard locks) |
| Write-heavy with few readers | `Mutex<HashMap>` may be simpler with similar perf |

### Pitfalls

DashMap's sharded RwLock model means write operations still block readers on the same shard. A case study demonstrated that a DashMap used with a shared atomic read counter suffered severe performance degradation due to constant inter-core cache line transfers -- the atomic counter was the bottleneck, not DashMap itself, but it illustrates how concurrent data structures interact with surrounding code.

DashMap guards hold shard locks for their lifetime. Holding a `Ref` or `RefMut` across an `.await` point (which saalis won't do in sync core, but could happen at the web boundary) is a deadlock risk.

### papaya: The Lock-Free Alternative

papaya is a newer concurrent hash map designed for read-heavy workloads. Unlike DashMap, its API is **lock-free** -- it hands out direct references to stored objects without wrapper guard types. It uses Hyaline (a variant of epoch-based reclamation) for memory management, which handles slow threads better than traditional EBR. For a columnar store's lookup indices (entity ID -> row), papaya's lock-free reads could outperform DashMap's sharded-RwLock reads.

### ArcSwap for Read-Heavy Configuration

For data that changes rarely but is read on every request (configuration, routing tables, cached schema metadata), `ArcSwap` provides lock-free reads with atomic pointer swaps. Readers never block -- they load the current `Arc` pointer. Writers build a new value and atomically swap it in. This is ideal for saalis's settings/configuration layer and any "snapshot" data renewed periodically.

## 3. crossbeam

### Epoch-Based Memory Reclamation

`crossbeam::epoch` provides the foundation for building lock-free data structures. The mechanism works via an epoch counter: threads "pin" themselves to an epoch when accessing shared data, and objects disconnected from the main data structure are queued for deferred deallocation. Once all threads have advanced past the epoch in which an object was removed, it can be safely freed. This avoids per-object atomic reference counting while still providing safe memory reclamation.

### Queue Implementations

- **`ArrayQueue`**: Bounded MPMC queue. Allocates a fixed-capacity buffer on construction. Wait-free for both producers and consumers when the queue is neither full nor empty. Ideal for fixed-size work pools (e.g., download slot queues with known lane count).

- **`SegQueue`**: Unbounded MPMC queue. Allocates segments on demand. Lock-free push/pop. Suitable for work queues where the number of pending items is unpredictable (e.g., the scheduler's incoming work unit queue).

### Scoped Threads

`crossbeam::scope` allows spawning threads that borrow from the parent stack frame -- the scope guarantees all spawned threads join before the scope exits, so the borrow checker accepts stack references. This is directly useful for the scheduler: spawn a scope, fan out work across threads that borrow the columnar store (behind RwLock), join, and continue. No Arc needed for the duration of the scope.

```rust
crossbeam::scope(|s| {
    for chunk in column_data.chunks(CHUNK_SIZE) {
        s.spawn(|_| {
            process_chunk(chunk); // borrows stack data directly
        });
    }
}).unwrap();
```

## 4. rayon

### Parallel Iterators

Rayon's `.par_iter()` automatically splits data into tasks, distributes them across a work-stealing thread pool, and combines results. The splitting is adaptive -- it recursively subdivides until chunks are small enough to not benefit from further parallelism.

For the columnar store, rayon is natural for:
- Parallel scans over entity metadata columns
- Parallel enrichment passes (CPU-bound parsing, normalization)
- Parallel image thumbnail generation
- Any `map`/`filter`/`fold` over collections of entities

### Thread Pool Configuration

```rust
// Dedicated pool for CPU-bound work, separate from I/O threads
let cpu_pool = rayon::ThreadPoolBuilder::new()
    .num_threads(num_cpus::get())  // or fewer on Batocera appliance
    .thread_name(|i| format!("saalis-cpu-{}", i))
    .build()
    .unwrap();

// Execute work on the dedicated pool
cpu_pool.install(|| {
    entities.par_iter().for_each(|e| enrich(e));
});
```

The `install()` method is key -- it runs a closure on a specific pool rather than the global pool. saalis should configure separate pools: one for CPU-bound work (rayon), dedicated threads for I/O (downloads, network), and tokio's runtime for the web layer.

### Integration with Sync Core

Rayon is purely synchronous and fits the sync-core model perfectly. The `F: Send + Sync + Fn` bounds mean all closures passed to rayon must be thread-safe, which Rust enforces at compile time. For the scheduler's CPU-bound subsystems (Curator scoring, Housekeeper filesystem checks), rayon provides parallelism without async machinery.

### Important: Don't Mix rayon and I/O

Rayon's work-stealing pool assumes tasks are CPU-bound and short-lived. Blocking I/O on a rayon thread starves other tasks. Keep I/O on dedicated threads; use rayon only for compute.

## 5. flume

### Channel Types

- **`flume::unbounded()`**: No capacity limit. `send()` never blocks. Useful for fire-and-forget event emission (scheduler notifications, log events).
- **`flume::bounded(n)`**: Fixed capacity. `send()` blocks when full, providing natural backpressure. Ideal for download queues, rate-limited pipelines.

### The Sync-Async Bridge

flume's defining feature for saalis is that the same channel works across the sync/async boundary:

```rust
let (tx, rx) = flume::bounded(64);

// Sync side (scheduler thread): blocking send
tx.send(work_result).unwrap();

// Async side (axum handler): non-blocking receive
let result = rx.recv_async().await.unwrap();
```

The `Sender` has both `send()` (blocking) and `send_async()` (async). The `Receiver` has both `recv()` (blocking) and `recv_async()` (async). This means the sync-core scheduler can produce results that the async web layer consumes without any `spawn_blocking` or `block_on` gymnastics.

### Comparison: flume vs crossbeam-channel vs tokio::sync::mpsc

| Feature | flume | crossbeam-channel | tokio::sync::mpsc |
|---|---|---|---|
| MPMC | Yes | Yes | No (MPSC only) |
| Async support | Native | No (sync only) | Native (async only) |
| Sync support | Native | Native | Requires `blocking_send` |
| `select!` | `Selector` API | `select!` macro | tokio `select!` |
| Performance | Competitive | Fastest pure-sync | Fastest pure-async |
| Zero unsafe | Yes | No | No |
| Cancel safety | Partial (known issues) | N/A | Yes |

### Recommendation for saalis

Use **flume** as the primary channel between sync core and async web shell. Use **crossbeam-channel** for pure sync-to-sync communication within the scheduler (its `select!` macro is more mature and slightly faster in pure-sync benchmarks). Avoid `tokio::sync::mpsc` in the core -- it requires a tokio runtime context.

**Caveat:** flume has documented cancel-safety issues with async send. If a future is dropped mid-send, notifications can be lost. For critical paths, prefer `try_send` with explicit retry or use crossbeam-channel with a `spawn_blocking` bridge.

## 6. Atomics: AtomicUsize, AtomicBool, and Ordering

### Lock-Free Progress Tracking

Atomics are ideal for the hilavitkutin chain progress pattern -- tracking how far along a multi-step pipeline has progressed without locking:

```rust
static ENRICHED_COUNT: AtomicUsize = AtomicUsize::new(0);
static DOWNLOADED_COUNT: AtomicUsize = AtomicUsize::new(0);
static CANCEL_REQUESTED: AtomicBool = AtomicBool::new(false);

// Worker thread:
ENRICHED_COUNT.fetch_add(1, Ordering::Relaxed);

// Progress reader (web handler):
let enriched = ENRICHED_COUNT.load(Ordering::Relaxed);
let downloaded = DOWNLOADED_COUNT.load(Ordering::Relaxed);
```

### Ordering Semantics

- **`Relaxed`**: No ordering constraints beyond atomicity. The compiler and CPU may reorder freely. Safe for independent counters, progress tracking, and statistics where you don't need to synchronize access to other memory. This is the right choice for most counters.

- **`Acquire` (loads) / `Release` (stores)**: Form a pair. A `Release` store guarantees that all preceding writes are visible to any thread that performs an `Acquire` load of the same atomic and sees the stored value. This is the minimum needed for a spin-lock: the `Release` store publishes the critical section's writes, and the `Acquire` load in `try_lock` ensures the new lock holder sees them.

- **`SeqCst`**: Sequentially consistent. All SeqCst operations across all threads form a single total order that every thread agrees on. This is the strongest (and most expensive) ordering. Rarely needed -- use it only when multiple atomics must be observed in a consistent order across threads, which is uncommon in practice.

### Practical Guidance

For saalis's progress counters and cancellation flags, `Relaxed` is almost always correct. The progress values are advisory -- it doesn't matter if a reader briefly sees a stale count. Use `Acquire`/`Release` only when an atomic guards access to non-atomic data (e.g., a flag that signals "initialization is complete, safe to read the config"). Avoid `SeqCst` unless you can articulate exactly why the weaker orderings are insufficient.

## 7. Arc Patterns

### Arc<T> -- Shared Immutable Data

Use plain `Arc<T>` when the data is immutable after creation. The columnar store's schema definitions, interned strings, and static configuration fit here. Clone is cheap (atomic increment), reads require no synchronization.

### Arc<RwLock<T>> -- Read-Heavy Shared Mutable Data

The go-to pattern for data read frequently and written occasionally. The columnar store itself is likely `Arc<RwLock<ColumnStore>>` or a collection of `Arc<RwLock<Column<T>>>`. Multiple reader threads scan columns concurrently; writer threads (scheduler work units completing) take exclusive access briefly to append or update rows.

### Arc<Mutex<T>> -- Write-Heavy or Simple Shared Mutable Data

When reads and writes are roughly equal, or critical sections are very short, Mutex has less overhead than RwLock (no reader count tracking). Use for queues, accumulators, or any state where "readers" also need to mutate (e.g., a counter that's read-modify-write).

### Arc<ArcSwap<T>> -- Lock-Free Read, Rare Write

For configuration, routing tables, or snapshot data. Readers never block (atomic pointer load). Writers build a new `Arc<T>` and swap it in atomically. The old value lives until all readers drop their references. This is strictly better than `Arc<RwLock<T>>` when writes are very rare and reads are on the hot path.

### Anti-Pattern: Arc<Mutex<Vec<T>>> as a Queue

If you find yourself writing `Arc<Mutex<Vec<T>>>` and calling `push`/`pop`, you actually want a channel (`flume::bounded`) or a concurrent queue (`crossbeam::SegQueue`). These have better performance characteristics and clearer semantics.

## 8. Semaphore Patterns for Download Lane Management

### The Problem

saalis needs to limit concurrent downloads (e.g., 3 simultaneous HTTP transfers on Batocera's constrained hardware). A semaphore is the natural primitive -- acquire a permit before starting a download, release when done.

### Sync Semaphore Options

parking_lot does not provide a dedicated `Semaphore` type. The common sync pattern is to build one from `Mutex` + `Condvar`:

```rust
use parking_lot::{Mutex, Condvar};

struct Semaphore {
    state: Mutex<usize>,
    cv: Condvar,
}

impl Semaphore {
    fn new(permits: usize) -> Self {
        Self { state: Mutex::new(permits), cv: Condvar::new() }
    }

    fn acquire(&self) {
        let mut count = self.state.lock();
        while *count == 0 {
            self.cv.wait(&mut count);
        }
        *count -= 1;
    }

    fn release(&self) {
        *self.state.lock() += 1;
        self.cv.notify_one();
    }
}
```

Alternatively, use a `crossbeam::ArrayQueue` of "permits" (unit values) -- `push` to release, `pop` to acquire. This is lock-free but busy-waits if you spin on `pop`.

### Async Semaphore (Web Boundary)

`tokio::sync::Semaphore` is fair (permits are distributed in request order) and integrates with async/await. If download initiation happens from an axum handler, the tokio semaphore is appropriate at that boundary.

### Fair vs Unfair

A **fair** semaphore grants permits in FIFO order, preventing starvation of any single requester. An **unfair** semaphore allows the fastest thread to acquire, maximizing throughput but risking starvation. For download lanes, fairness matters -- a large game download shouldn't permanently block small metadata fetches. The Condvar-based approach above is inherently fair (waiters wake in `notify_one` order, which in parking_lot is FIFO due to its internal queue).

## 9. Thread-Per-Core vs Work-Stealing

### Work-Stealing (Rayon Model)

Each thread maintains a local deque of tasks. When a thread's deque is empty, it steals from another thread's deque. This automatically load-balances uneven workloads.

**Advantages:** Excellent CPU utilization, good tail latency for uneven workloads, trivial to use via rayon's parallel iterators.

**Disadvantages:** Tasks may migrate between cores, causing cache invalidation. All shared data must be `Send + Sync`. Slightly higher per-task overhead from the stealing protocol.

### Thread-Per-Core (Glommio Model)

One thread per CPU core, pinned. No work migration. Each thread has its own data and event loop. Communication between threads is explicit (via channels).

**Advantages:** Eliminates cache line bouncing, no need for `Send + Sync` on thread-local data, predictable latency. Excellent for I/O-heavy workloads where each thread manages its own connections.

**Disadvantages:** Uneven workloads cause some cores to idle while others are busy. Requires explicit partitioning of work. More complex programming model.

### Recommendation for saalis

**Scheduler (CPU-bound work):** Work-stealing via rayon. The scheduler's workload (scoring entities, scanning metadata, verifying files) is naturally uneven -- some entities need more processing than others. Work-stealing ensures all cores stay busy. The work is CPU-bound and short-lived, which is rayon's sweet spot.

**I/O (downloads, HTTP fetches):** Dedicated threads, not rayon, not thread-per-core. On Batocera's constrained hardware (typically 2-4 cores), thread-per-core is overkill. Use a small fixed pool of I/O threads (matching download lane count) that block on network I/O. These threads communicate results back to the scheduler via flume channels.

**Web layer:** tokio's work-stealing runtime (already decided -- async shell). tokio handles the uneven nature of HTTP request processing well.

This three-pool model (rayon for compute, dedicated threads for I/O, tokio for web) keeps each pool's threads doing what they're optimized for and prevents cross-contamination (blocking I/O on rayon threads, CPU-bound work on tokio threads).

## 10. Common Deadlock Patterns and Prevention

### Pattern 1: Lock Ordering Violation

Two threads acquire locks A and B in opposite order:

```
Thread 1: lock(A) -> lock(B)
Thread 2: lock(B) -> lock(A)  // DEADLOCK
```

**Prevention:** Establish a global lock ordering. Document it. All code must acquire locks in the canonical order. For saalis: store-level lock before shard-level lock before entity-level lock. Consider a compile-time approach using phantom type parameters to encode lock levels.

### Pattern 2: Recursive Read Lock on parking_lot::RwLock

Because parking_lot's RwLock is task-fair, attempting to acquire a read lock while already holding one *on the same thread* can deadlock if a writer is waiting:

```
Thread 1: read_lock(A) -> read_lock(A)  // blocks if writer is queued
Thread 2: write_lock(A)                  // queued between the two reads
```

**Prevention:** Never acquire the same RwLock recursively. Restructure code so that functions receiving data by reference don't re-acquire the lock. If recursion is truly needed, use `parking_lot::ReentrantMutex`.

### Pattern 3: Lock Held Across Channel Send

```rust
let guard = store.read();
tx.send(guard.get(key).clone()); // blocks if channel is full
// guard still held -- readers of store are blocked
```

If the channel consumer also needs the store lock, you have a deadlock.

**Prevention:** Clone data out of the lock, drop the guard, then send:

```rust
let value = {
    let guard = store.read();
    guard.get(key).clone()
};  // guard dropped here
tx.send(value);
```

### Pattern 4: Implicit Guard Lifetimes

Rust's temporary lifetime rules can extend a `MutexGuard` longer than intended:

```rust
// The guard lives until end of the if-let block, not just the match
if let Some(v) = map.lock().get("key") {
    // lock is STILL held here
    expensive_operation(v);
}
```

**Prevention:** Bind the guard explicitly and drop it when done. Use `parking_lot`'s `MutexGuard::map()` to project into a sub-field if you need to hold a reference.

### Pattern 5: try_lock and Timeout

For defensive programming, `parking_lot` supports `try_lock()` (returns immediately) and `try_lock_for(Duration)` (returns after timeout). These are useful for:
- Detecting potential deadlocks in debug builds
- Implementing lock acquisition with fallback strategies
- Preventing indefinite hangs in production

```rust
match store.try_write_for(Duration::from_millis(100)) {
    Some(guard) => { /* proceed */ },
    None => {
        tracing::warn!("store write lock contention > 100ms");
        // retry, skip, or report
    }
}
```

### General Prevention Checklist

1. **Minimize lock scope:** Hold locks for the shortest possible duration. Clone data out, drop the lock, then process.
2. **Document lock ordering:** If acquiring multiple locks, the order must be consistent and documented.
3. **Avoid locks in hot paths:** Use atomics, ArcSwap, or lock-free data structures where possible.
4. **No locks across channel operations:** Clone before sending.
5. **Test with `parking_lot`'s deadlock detection:** Enable the `deadlock_detection` feature in debug builds to get backtraces when deadlocks occur.
6. **Prefer single-lock designs:** If you can restructure to use one lock instead of two, do it.

## Summary: Recommended Stack for saalis

| Concern | Primitive | Crate |
|---|---|---|
| Internal locks (Mutex, RwLock, Condvar) | parking_lot | `parking_lot` |
| Concurrent HashMap (indices, caches) | DashMap or papaya | `dashmap` / `papaya` |
| Lock-free config / snapshots | ArcSwap | `arc-swap` |
| Sync channels (scheduler internal) | crossbeam-channel | `crossbeam-channel` |
| Sync-async bridge channels | flume | `flume` |
| CPU-bound parallelism | rayon thread pool | `rayon` |
| Lock-free queues | ArrayQueue / SegQueue | `crossbeam-queue` |
| Progress counters / flags | AtomicUsize / AtomicBool | `std::sync::atomic` |
| Download lane limiting | Condvar semaphore (sync) / tokio Semaphore (async) | `parking_lot` / `tokio` |
| Deadlock detection (debug) | deadlock_detection feature | `parking_lot` |
