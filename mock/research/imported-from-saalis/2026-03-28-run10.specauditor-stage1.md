# Specification Completeness Audit -- Stage 1

**Date:** 2026-03-28
**Reviewer:** Specification Completeness Auditor (25yr IETF/IEEE/ISO protocol review)
**Scope:** All seven verkko design documents + taxonomy
**Method:** Line-by-line reading of all documents. Every finding states what is missing, where, and the consequence.

---

## Classification key

- **MISSING**: the spec says nothing about this topic
- **INCOMPLETE**: the spec says something but not enough for two independent implementations to agree
- **AMBIGUOUS**: the spec could be read two valid ways
- **UNJUSTIFIED**: the spec states a value or choice without explaining why

**Severity:**
- **CRITICAL**: two implementations will produce incompatible behavior
- **HIGH**: a failure mode is unhandled, or a mechanism cannot be implemented from the spec alone
- **MEDIUM**: an edge case is unspecified, or a constant lacks justification
- **LOW**: a clarification would help but the spec is likely implementable

## verkko-crypto (15 findings)

**C1. MISSING / HIGH -- Enrollment certificate format and verification procedure.**
Section 1.1 says "Bound at enrollment (see verkko-mesh: Enrollment)" but the enrollment certificate structure is only shown in verkko-mesh 1.2. verkko-crypto claims to define the identity model but does not specify how a DK/DEK pair is bound to a cluster or how an observer verifies the binding. An implementor of verkko-crypto alone cannot validate an enrollment certificate.

*Fix:* Add a defined concept `EnrollmentCertificate` to verkko-crypto with the exact field layout and verification algorithm (signature over `new_DK_pub || new_DEK_pub || enrollment_epoch || cluster_CIK`), or explicitly state that enrollment certificate verification is verkko-mesh's responsibility and that verkko-crypto provides only the signing primitive.

**C2. INCOMPLETE / HIGH -- CSPRNG self-test is underspecified.**
Section 1.3 step 2 says "32 bytes, non-zero, non-repeating across 3 samples." It does not specify: (a) what "non-repeating" means when comparing 32-byte blocks (exact equality? Hamming distance?), (b) whether the 3 samples must be drawn consecutively, (c) what the minimum entropy estimate must be, (d) whether the test runs at genesis only or at every startup.

*Fix:* "Self-test: draw three consecutive 32-byte samples from the system CSPRNG. If any two samples are byte-equal, or any sample is all-zero, abort with error E_ENTROPY_FAILURE. No persistent state is written. Self-test runs at genesis only; runtime entropy failures are handled by the OS."

**C3. AMBIGUOUS / MEDIUM -- CIK replication encryption target.**
Section 1.1 says CIK is "encrypted to each DEK" but does not specify the AEAD construction used for this wrapping. Is it StorageAEAD (XChaCha20-Poly1305 with deterministic nonce)? TransportAEAD? A separate X25519+AEAD envelope? Two implementations could choose different constructions and produce incompatible wrapped CIK blobs.

*Fix:* "CIK is wrapped to each cluster peer's DEK using X25519 key agreement (DEK) followed by HKDF-Expand(prk=shared_secret, info='cik:wrap:v1', len=32) to derive a wrapping key, then ChaCha20-Poly1305 with a fresh random 96-bit nonce. The wrapped CIK is stored in the control log."

**C4. INCOMPLETE / MEDIUM -- Key identification collision handling.**
Section 1.5 specifies `key_id = BLAKE3(K_epoch_N || channel_index)[0..8]` and notes "Birthday collision at 2^32 keys." No procedure is specified for what happens if two active keys within the same session produce the same 8-byte key_id. The spec says the mesh "will never reach this" but does not define behavior if it does occur (e.g., due to a bug or adversarial key injection).

*Fix:* "If a received frame's key_id matches multiple active keys, the receiver attempts decryption with each matching key in epoch order (lowest first). If none succeed, the frame is treated as an unknown key_id (buffered per Section 2.4)."

**C5. MISSING / MEDIUM -- BroadcastEnvelope version and algorithm fields.**
Section 2.1 includes `version: u8` and `algorithm: u8` but does not define the valid values for either field. What is version 1? What algorithm ID corresponds to ChaCha20-Poly1305? What happens when a receiver encounters an unknown version or algorithm?

*Fix:* "version = 0x01 for this specification. algorithm = 0x01 for ChaCha20-Poly1305. Unknown version: discard envelope, log warning. Unknown algorithm: discard envelope, log warning."

**C6. INCOMPLETE / MEDIUM -- Micro-epoch checkpoint storage and retrieval.**
Section 2.4 describes micro-epoch checkpoints every 720 advances ("34 bytes" per checkpoint, "6,800 bytes/day") but does not specify: (a) what the 34 bytes contain (key? counter? both?), (b) where they are stored (memory? flash?), (c) how they are retrieved on catch-up, (d) what happens if a checkpoint is lost.

*Fix:* "Each checkpoint stores (u32 micro_epoch_counter, [u8; 32] micro_epoch_key) = 36 bytes. Stored in the managed arena. On catch-up: locate the latest checkpoint with counter <= target, derive forward from there. If no checkpoint is available, derive from the macro-epoch key (K_epoch_N)."

**C7. AMBIGUOUS / MEDIUM -- Dual-key window "3 heartbeat intervals" unit.**
Section 2.4 says "both keys for 3 heartbeat intervals (15s active)." This implies 5s heartbeat intervals, but in Background mode heartbeats are 30s, giving a 90s dual-key window. The spec does not clarify whether the dual-key window is always 15s, or adapts to the current metabolic state.

*Fix:* "Both the old and new epoch key are accepted for max(15_seconds, 3 * current_heartbeat_interval). In Background mode (30s interval): 90s. In Active mode (5s interval): 15s."

**C8. INCOMPLETE / MEDIUM -- EpochRatchet defined concept missing initial state.**
The EpochRatchet defined concept states invariants about derivation but does not state what the initial epoch key (K_epoch_0) is. The body (Section 2.4) says `K_epoch_0 = channel_key`, but the defined concept should carry this since dependents rely on it.

*Fix:* Add to EpochRatchet invariants: "K_epoch_0 is the channel key (independently generated 256-bit random value from ChannelKey)."

**C9. MISSING / LOW -- BLAKE3 test vector table is empty.**
Section 4.1 says implementations "must pass the following test vector table before deployment" but the table says "(defined at implementation time with concrete test vectors)." This defers the single most important interoperability check to a future undefined step.

*Fix:* Provide at least one concrete test vector per registered domain separator now. Example: `BLAKE3("saalis-keyid-v1", 0x00 * 32 || 0x00 * 2) = <hex>`. Implementations that cannot reproduce this output are incompatible.

**C10. INCOMPLETE / MEDIUM -- PCSMechanism does not specify which key scopes are affected.**
Section 2.6 says "the derivation per key scope" but does not enumerate the key scopes. `key_scope_id` is used in the info string but never defined. An implementor does not know whether to re-seed the channel key, the CIK, both, or something else.

*Fix:* "key_scope_id is the channel_index (u16, big-endian). PCS re-seed applies to every channel shared between the two clusters whose CPSK was renegotiated. CIK is not re-seeded via PCS."

**C11. UNJUSTIFIED / LOW -- Gap safety factor of 3.**
Section 3.2 justifies factor 3 with "persist delay (worst 1.5x) and throughput ramp (1.3x) anticorrelate." These two factors multiply to 1.95. Factor 3 provides "2.3x margin over ramp alone, 2x over persist delay alone." But the worst case is both occurring simultaneously (not anticorrelating), which gives factor 1.95. If they do anticorrelate, factor 2 suffices. The justification assumes anticorrelation without proving it.

*Fix:* Either provide evidence of anticorrelation on eMMC (e.g., benchmark data) or state: "Factor 3 assumes worst-case independent factors (1.5 * 1.3 = 1.95) plus 50% engineering margin."

**C12. INCOMPLETE / HIGH -- DoubleWritePersist does not specify the version field semantics.**
Section 3.2 says "use valid copy with highest version number" but the persist file structure shows `[u32 version]` without specifying how version is incremented. Is it monotonic? Does it wrap? What happens at u32::MAX?

*Fix:* "version is a monotonically increasing u32. Incremented by 1 on each write. If version == u32::MAX, the next write sets version = 0 and both copies are written atomically (the one-time wrap is safe because it can only happen after 4 billion persists, roughly 130 years at one persist per second)."

**C13. MISSING / MEDIUM -- FoundingHash does not specify the hash of the founding transcript endianness.**
Section 1.3 shows `LEN(DK_pub) || DK_pub || ...` but does not specify the encoding of LEN (u8? u16? u32? big-endian? little-endian?). Two implementations using different LEN encodings will produce different founding hashes and cannot interoperate.

*Fix:* "All LEN fields are u32 little-endian. DK_pub, DEK_pub, CIK_pub, signing_pub are raw 32-byte encodings. enrollment_cert is the DER-encoded certificate. genesis_timestamp is u64 little-endian milliseconds since Unix epoch."

**C14. MISSING / MEDIUM -- KeyZeroization has no test or verification mechanism.**
The defined concept states "No key material survives in memory after its useful lifetime ends" but provides no way to verify this. No mention of `SecretVec`, `zeroize` crate, or compiler fence to prevent dead-store elimination. The NonceSource section mentions `SecretVec` but the defined concept does not.

*Fix:* Add to KeyZeroization invariants: "Zeroization uses the zeroize crate with compiler fence. Verification: integration test that maps process memory after key deletion and scans for the test key pattern."

**C15. INCOMPLETE / LOW -- DomainSeparatorRegistry is not closed.**
Section 4.1 says "Future additions must register a unique domain separator" but provides no mechanism for registration. Who adds entries? How is uniqueness enforced? Is it a compile-time table? A runtime check?

*Fix:* "The registry is a compile-time constant array. Adding a new domain separator requires adding an entry to the array and a compile-time assertion that no two entries share the same string. Duplicate detection is a compile error."

## verkko-protocol (18 findings)

**P1. MISSING / CRITICAL -- Noise handshake message format and state machine.**
Section 3.3 says "Noise KK for established pairs, Noise XX for first contact" and references [Perrin2018] but does not specify: the prologue, the payload in each handshake message, the static key authentication pattern, or the handshake hash computation. Two implementations cannot interoperate without agreeing on the exact Noise protocol name (e.g., `Noise_KK_25519_ChaChaPoly_BLAKE2b` vs `Noise_KK_25519_ChaChaPoly_SHA256`).

*Fix:* Specify the complete Noise protocol name: "Transport sessions use `Noise_KK_25519_ChaChaPoly_BLAKE2b` for established pairs and `Noise_XX_25519_ChaChaPoly_BLAKE2b` for first contact. Prologue: `'verkko-v1' || initiator_connection_id || responder_connection_id`. Handshake payload: SchemaSync hash (32 bytes) + OBSERVED_ADDR (18 bytes)."

**P2. MISSING / CRITICAL -- Connection ID generation and collision handling.**
The wire format (Section 3.1) uses a `u32 connection_id` but never specifies how it is generated. Random? Sequential? Derived from the Noise handshake? What happens when two connections produce the same connection_id?

*Fix:* "connection_id = first 4 bytes of the Noise handshake hash. If a collision is detected (same connection_id from different sessions), the newer session generates a new random connection_id and sends a CONNECTION_ID_CHANGE frame."

**P3. MISSING / HIGH -- Stream multiplexing is a defined concept with zero body.**
The Stream defined concept says "No head-of-line blocking between streams" and "Each stream has independent ordering guarantees" but the body contains no specification of how streams are created, identified, flow-controlled, or closed. There is no stream ID field in the wire format.

*Fix:* At minimum, add a stream_id field to the encrypted payload, specify stream lifecycle (OPEN, HALF_CLOSE, RESET frames), and define per-stream flow control (window-based or credit-based).

**P4. MISSING / HIGH -- SchemaSync is a defined concept with zero body.**
The defined concept says "Client sends schema hash. Server compares. On mismatch, server sends updated schema." But the body provides no message format, no hash algorithm, no transfer encoding for the schema, and no versioning scheme.

*Fix:* "During Noise handshake, initiator sends schema_hash = BLAKE3(local_protobuf_descriptor_set). Responder compares. On mismatch, responder sends SCHEMA_UPDATE frame (reliable, ordered) containing the serialized FileDescriptorSet. Initiator replaces local schema."

**P5. MISSING / HIGH -- 0-RTT resumption is listed in the taxonomy but has no body anywhere.**
The taxonomy lists "0-RTT resumption" as part of verkko-protocol's scope. Neither the defined concepts nor the body mention it. A seat reconnecting after a brief disconnect would benefit from 0-RTT but the mechanism does not exist.

*Fix:* Add a ResumptionToken defined concept: "On session teardown, both peers store a resumption token (session_key || connection_id || schema_hash). On reconnect within 24 hours, initiator sends the token in the first message. If valid, the session resumes without a full Noise handshake. If invalid, fall back to full handshake."

**P6. INCOMPLETE / HIGH -- SACK format and semantics unspecified.**
The SACK defined concept says "Per-session tracking of delivered vs sent frame counts" but the body never shows the SACK frame format. How many selective ack ranges can be included? What is the encoding? Is it piggybacked on data frames or sent as standalone? The explicit congestion feedback (3 bytes) is shown but the actual selective acknowledgment is not.

*Fix:* Specify a SACK block format: "SACK is piggybacked on every outbound frame. Format: [u32 cumulative_ack, u8 block_count, (u32 gap, u32 ack_length) * block_count]. Maximum 4 blocks per frame (33 bytes)."

**P7. INCOMPLETE / MEDIUM -- LEDBAT coupled controller sharing algorithm.**
Section 3.3 says "Single LEDBAT controller per physical uplink. Fair-share weighted scheduler across 11 inter-cluster connections." But the sharing algorithm is not specified. Is it max-min fair? Proportional? Weighted by what? How does the vital queue guarantee interact with fair sharing?

*Fix:* "Fair-share allocation: each of the 11 connections gets equal share of the LEDBAT window. Vital queue traffic from any connection is exempt from the share limit. Bulk traffic is weighted equally. When one connection has no bulk traffic, its share is redistributed."

**P8. MISSING / MEDIUM -- PMTU probing mechanism.**
Section 3.3 says "PMTU probing on connection establishment" but specifies no probing algorithm. Binary search? Linear? What probe sizes? What timeout before concluding a probe failed?

*Fix:* "PMTU probing: binary search between 576 and 1280 bytes. Send probe frames (PADDING type, no payload) at each size. If no SACK within 2*RTT, probe size is too large. Minimum 3 successful probes at a size before accepting it. Default PMTU: 1280 (IPv6 minimum)."

**P9. INCOMPLETE / MEDIUM -- Reliability modes lack frame-level specification.**
Section 3.3 describes "Reliable ordered: monotonic sequence, SACK ACKs, exponential backoff" but does not specify: (a) how the msg_type field distinguishes reliable from unreliable frames, (b) how the sequence number is encoded, (c) how retransmissions are identified (same sequence number? different nonce?).

*Fix:* "msg_type bit 7 indicates reliability: 0 = unreliable, 1 = reliable. Reliable frames carry a u32 sequence number after msg_type. Retransmissions reuse the same sequence number but get a new nonce and new AEAD encryption. The receiver deduplicates by sequence number within the replay window."

**P10. INCOMPLETE / MEDIUM -- Destination field collision probability.**
Section 3.1 uses `[u32 destination] = BLAKE3(CIK_pubkey)[0..4]` for routing. At 12 clusters, collision probability is negligible. But the spec does not address what a relay does if two clusters have the same 4-byte destination hash (birthday bound at ~65,000 clusters, but the spec should address the failure mode).

*Fix:* "If a relay observes the same destination hash for two different CIKs, it delivers to both. The AEAD decryption will fail for the wrong recipient, and the frame is silently dropped."

**P11. MISSING / MEDIUM -- NAT traversal hole-punching algorithm.**
The NATTraversal defined concept says "Hole punching attempted before relay fallback" but the body says only "Adaptive keepalive: discover mapping lifetime via probes." No hole-punching algorithm is described (STUN? ICE? Custom?). No signaling mechanism for coordinating simultaneous open.

*Fix:* "Hole punching uses STUN-like probes (RFC 5389) to discover the external address. Coordinated via relay: both peers send their OBSERVED_ADDR to the relay, relay forwards to the other peer, both send UDP probes to each other's observed address. If bidirectional traffic flows within 5 seconds, direct connection is established."

**P12. MISSING / MEDIUM -- PeerDiscovery mDNS record format.**
Section defines a fallback chain "mDNS (LAN) -> invite hint -> cached peers -> DHT" but specifies no mDNS service type, TXT record format, or port number.

*Fix:* "mDNS service type: _verkko._udp.local. TXT records: dk=<base64 DK_pub>, port=<u16>. SRV record points to the peer's LAN address."

**P13. INCOMPLETE / MEDIUM -- MetabolicState transition conditions.**
Section 6.3 defines four states and a "minimum dwell time" but does not specify the transition trigger conditions. What measurement causes a transition from Idle to Active? From Active to Stress? The governor (Conductor) is in verkko-mesh, but the protocol layer defines the state machine. The transition predicates are in neither document.

*Fix:* Either define the transition predicates in verkko-protocol ("Active -> Stress when governor output < 0.3 for dwell_time_ms consecutive") or explicitly state "Transition predicates are defined by the governor (see verkko-mesh: Conductor). The protocol layer only defines the states and dwell time enforcement."

**P14. UNJUSTIFIED / LOW -- Replay window size W=1024.**
Section 3.1 says W=1024 "128 bytes per session" without justifying why 1024 and not 512 or 2048. For an embedded system sending at low rates, 1024 may be excessive. For burst traffic during convergence, it may be too small.

*Fix:* "W=1024 chosen to handle: 1-second burst at 1000 frames/s with up to 24ms reordering. At 91 bytes/frame, this is ~91 KB/s. The replay window costs 128 bytes per session (11 sessions = 1,408 bytes). At W=512 the burst tolerance halves; at W=2048 the memory doubles for marginal gain."

**P15. MISSING / LOW -- Heartbeat loss_permille field purpose is unclear.**
Section 3.4 includes `loss_permille` in the heartbeat header with "(diagnostics only, not used for gateway election)." If it is not used for any decision, why is it in the heartbeat? It consumes 2 bytes per heartbeat. Is it used for display? Logging? If it serves no functional purpose, it is dead weight.

*Fix:* Either remove loss_permille and save 2 bytes, or state its consumer: "loss_permille is displayed in the admin UI and included in structured diagnostic logs for operator troubleshooting."

**P16. INCOMPLETE / LOW -- Cristian's algorithm clock sync parameters.**
Section 3.3 says "Cristian's algorithm during Noise handshake (32 bytes)" but does not specify the minimum round-trip threshold for accepting a measurement, or how many measurements are taken.

*Fix:* "Take 3 measurements during Noise handshake. Discard any with RTT > 2 * min(RTTs). Use the measurement with lowest RTT. If all 3 are discarded, skip clock sync and rely on HLC alone."

**P17. MISSING / LOW -- Connection migration rate limit enforcement.**
Section 3.3 says "Rate-limit: 1KB/s before validation" but does not specify how the rate limit is enforced, how long validation takes, or what happens to frames that exceed the limit during validation.

*Fix:* "During path validation (PATH_CHALLENGE sent, awaiting PATH_RESPONSE), outbound frames on the new path are queued and rate-limited to 1 KB/s. Frames exceeding the queue capacity (8 KB) are dropped. Validation completes on PATH_RESPONSE receipt or times out after 5 * RTT."

**P18. INCOMPLETE / MEDIUM -- SansIO defined concept does not specify the consumer API.**
The defined concept says "Consumer drives I/O: provides received bytes, collects outbound bytes" but does not specify the function signatures, event types, or timer management interface. A sans-IO protocol needs a well-defined API surface: `handle_input(bytes) -> Vec<Event>`, `poll_transmit() -> Option<Transmit>`, `handle_timeout()`.

*Fix:* Specify at minimum: "The sans-IO API consists of three methods: `handle_datagram(src: SocketAddr, data: &[u8]) -> Vec<OutputEvent>`, `poll_transmit() -> Option<(SocketAddr, Vec<u8>)>`, `handle_timer(now: Instant) -> Vec<OutputEvent>`. OutputEvent is an enum: {Connected, Disconnected, DataReceived{stream, data}, ...}."

## verkko-mesh (20 findings)

**M1. MISSING / HIGH -- InviteFlow has no body.**
The InviteFlow defined concept lists invariants (mesh identity, bootstrap peer address, PSK, SAS verification, one-time-use) but the body contains no specification. No invite token format, no QR code encoding, no expiry handling, no flow diagram. An implementor cannot build the invite flow from this spec.

*Fix:* Add a body section for InviteFlow specifying: (a) invite token format: `founding_hash || bootstrap_addr || PSK || expiry_epoch || nonce`; (b) QR encoding: base64url of the token; (c) consumption: after successful Noise XX handshake using the PSK, the invite is marked consumed in the control log; (d) expiry: token rejected after expiry_epoch.

**M2. INCOMPLETE / HIGH -- Gateway election conflict resolution during partition.**
Section 7.2 says "Max-register CRDT" for gateway election. Section 3.3 (verkko-protocol) says "Gateway singularity is enforced per connected component. During partition, each partition may independently elect a gateway." But the convergence spec (Section 7.1) does not specify how two gateways from two partitions resolve to one gateway on heal. Do both gateways simultaneously claim? How is the CRDT max-register resolved? What happens to in-flight frames during the transition?

*Fix:* "On partition heal, the gateway with the higher max-register value wins. The losing gateway sends a gateway-release frame within one heartbeat interval. In-flight frames from the losing gateway are buffered by the winning gateway for one heartbeat interval."

**M3. INCOMPLETE / HIGH -- Control log compaction is underspecified.**
Section 1.7 says "Compaction: truncate > 90 days with signed snapshot." But: (a) who triggers compaction? (b) what is in the signed snapshot? (c) how do newly joining peers validate a snapshot? (d) what happens if the snapshot signer is no longer trusted? (e) is there a minimum number of entries that must be kept regardless of age?

*Fix:* "Compaction is triggered by admin when the control log exceeds 10,000 entries or 1 MB. The snapshot contains: BLAKE3(all entries <= cutoff), the cutoff Lamport clock, and the admin's Ed25519 signature. Newly joining peers accept the snapshot if signed by the current admin. If the signer is revoked, the snapshot remains valid (the revocation entry is post-cutoff by definition). Minimum 100 entries are always retained."

**M4. INCOMPLETE / MEDIUM -- Channel access levels lack enforcement specification.**
Section Terminology defines four access levels (off, opaque, read, write) but the body never specifies how these are enforced. Where is the access control check? Before encryption? Before decryption? What frame type carries the access level? How is access level distributed (in the KeyBundle? separately?)?

*Fix:* "Access level is a per-peer, per-channel attribute stored in the control log. KeyBundle(Grant) includes the access level. The host peer enforces access control at the application layer: before encrypting a frame for a channel, verify write access; before delivering decrypted content, verify read access. Opaque peers can forward but not decrypt."

**M5. MISSING / MEDIUM -- CIK rotation procedure is incomplete.**
Section 2.3 says "CIK rotation: signed control log entry, accept both CIKs for 48 hours." But: (a) who initiates rotation? (b) how is the new CIK generated and distributed? (c) what happens to active CPSK sessions during the 48-hour window? (d) how does a peer that joins during the rotation window know which CIK is current?

*Fix:* "CIK rotation is initiated by the cluster admin. New CIK generated from CSPRNG. Distributed via KeyBundle(Rotate) to cluster peers, encrypted to each DEK. During the 48-hour window, peers accept Noise handshakes using either CIK. CPSK sessions renegotiate using the new CIK. After 48 hours, old CIK is deleted. Newly joining peers receive only the new CIK."

**M6. INCOMPLETE / MEDIUM -- Partition announcement observer quorum threshold.**
Section 5.2 says `actual_observer_count < expected_observer_count / 2` for deferral. But `expected_observer_count` is never defined. Is it the number of peers in the observer's cluster? The number of peers in the mesh? The number of peers that were connected before the partition? Integer division truncation also matters: at 3 expected, the threshold is 1.

*Fix:* "expected_observer_count = number of peers in the local cluster that were healthy (slow_ema > 0.92) at the start of the partition epoch. Integer division: floor(expected / 2)."

**M7. AMBIGUOUS / MEDIUM -- Dominance score uses BLAKE3 but the output is not a number.**
Section 2.3 says `dominance_score(peer) = BLAKE3(DK_pub || current_epoch) * (1 + weighted_scar_count) * 2^max(...)`. BLAKE3 produces a 256-bit hash, not a number that can be multiplied by a float. The spec does not say how the hash is converted to a numeric value for multiplication.

*Fix:* "BLAKE3(DK_pub || current_epoch) is interpreted as a 256-bit unsigned integer (big-endian). The multiplication `hash_value * (1 + weighted_scar_count) * 2^max(0, weighted_scar_count - 10)` is performed in 512-bit integer arithmetic (to avoid overflow). argmin compares 512-bit values."

**M8. INCOMPLETE / MEDIUM -- Gossip mode ACK for exit transition.**
Section 7.1 Exit Step 1 says "Admin broadcasts gossip_mode=0. Gate 1: All peers ACK gossip_mode=0." But there is no ACK message type defined anywhere. How does a peer ACK the gossip mode switch? Is it an explicit ACK frame? Is the ACK implicit (setting gossip_mode=0 in the next heartbeat)?

*Fix:* "The ACK is implicit: a peer that receives gossip_mode=0 sets its own gossip_mode=0 in its next heartbeat. Gate 1 is satisfied when all peers' most recent heartbeat shows gossip_mode=0."

**M9. INCOMPLETE / MEDIUM -- Fencing token grace window start trigger.**
Section 2.3 says "10-minute grace after partition heal." But when exactly does the 10-minute timer start? At the first CPSK reconnection? At wavefront deadline expiry? At Front 1 gate open? At Front 2 gate open?

*Fix:* "The 10-minute grace window starts at the wavefront deadline expiry (the moment Front 1 begins processing)."

**M10. MISSING / MEDIUM -- ControlLog hash-linking algorithm.**
Section 1.7 says the control log is "hash-linked" but does not specify the hash function, the fields included in the hash, or the chaining structure. Is it `BLAKE3(prev_hash || entry_bytes)`? Does it include the Lamport clock? The scope?

*Fix:* "Each control log entry contains `prev_hash: [u8; 32]`. prev_hash = BLAKE3(previous_entry_serialized). The chain starts with prev_hash = BLAKE3('verkko-control-genesis-v1'). Serialization is the entry's canonical protobuf encoding."

**M11. INCOMPLETE / MEDIUM -- EDT scope_flags bitmask limitation.**
Section 1.6 defines `scope_flags` as a u64 bitmask for channels. But channel_index is u16 (Section 1.7), meaning up to 65,536 channels exist. A u64 bitmask can only represent 64 channels. The spec does not address this mismatch.

*Fix:* "scope_flags is a u64 bitmask covering channels 0-63. Channels 64+ cannot be delegated via EDT; they require direct admin enrollment. At the design envelope of 50 channels, u64 is sufficient."

**M12. UNJUSTIFIED / LOW -- Scar decay period of 180 days.**
Section 5.3 says "Linear 180-day decay" but does not justify why 180 days and not 90 or 365. For a household mesh where devices may be occasionally flaky due to WiFi, 180 days seems long. A flash storage issue that caused scars 5 months ago may have been physically repaired.

*Fix:* "180-day decay period chosen to exceed the longest expected seasonal pattern (6 months of summer heat causing thermal throttling on attic NAS). At 90 days, scars from quarterly patterns would decay before the next occurrence. At 365 days, scars from one-time events would persist excessively."

**M13. INCOMPLETE / MEDIUM -- GovernedResource PI controller lacks tuning rationale.**
Section 6.1 says "PI controller with anti-windup (Kp=2.0, Ki=0.1)" but does not specify: (a) the sampling interval, (b) the setpoint source, (c) the output range, (d) the anti-windup mechanism beyond "integrator clamped to MAX_INTEGRATOR = 10.0."

*Fix:* "PI sampling interval = heartbeat interval (metabolic-dependent). Setpoint = base_setpoint * capacity_ratio. Output range [0.0, 1.0] (0 = fully throttled, 1 = full capacity). Anti-windup: when output saturates at 0.0 or 1.0, integrator is clamped to prevent further integration in the saturating direction. MAX_INTEGRATOR = 10.0 limits the integrator accumulation to prevent overshoot on sudden setpoint changes."

**M14. MISSING / LOW -- Cluster formation procedure.**
The Cluster defined concept says "1-6 devices on the same LAN" but the body never specifies how a cluster is formed. Is the first device automatically a one-device cluster? How does a second device join the cluster vs creating a new cluster? Is cluster membership explicit (admin assigns) or implicit (same LAN = same cluster)?

*Fix:* "A device is always in exactly one cluster. The genesis device forms a one-device cluster. Additional devices join the cluster via enrollment (Section 1.2). Cluster membership is explicit: the enrollment certificate binds a device to a specific CIK. Devices on the same LAN that are enrolled with different CIKs are in different clusters."

**M15. MISSING / LOW -- LKH tree rebalancing algorithm.**
Section 2.7 says "Lazy rebalancing when imbalance > 2" but does not specify the rebalancing algorithm. AVL rotation? Red-black? How does the rebalancing propagate to other peers?

*Fix:* "Rebalancing uses AVL-style rotations. The admin computes the balanced tree and distributes the new tree structure via a control log entry. Peers accept the new tree structure and re-derive their path keys."

**M16. INCOMPLETE / MEDIUM -- ObservationPathForwarding lacks failure mode.**
Section 5.4 says "Every inter-cluster data transfer contributes a health summary within two consecutive intra-cluster heartbeat intervals." But what happens if a cluster peer fails to report a FlowSummary? Does the gateway use stale data? Time out and exclude the peer? The failure mode for missing observations is unspecified.

*Fix:* "If a cluster peer fails to contribute a FlowSummary for 3 consecutive intra-cluster heartbeat intervals, the gateway excludes that peer's observations from the aggregation and logs a diagnostic warning. The peer's previous observations decay naturally via the time-normalized alpha."

**M17. UNJUSTIFIED / LOW -- Maximum 6 devices per cluster.**
The Cluster defined concept says "1-6 devices" without justification. Why 6? A household might have 8 devices (2 phones, 2 tablets, 1 NAS, 1 TV, 1 laptop, 1 desktop).

*Fix:* "6 devices per cluster chosen to bound intra-cluster reliable broadcast cost: iterated unicast to 5 peers per message. At 8 devices, broadcast cost increases by 40%. If a household exceeds 6 devices, the admin can create a second cluster on the same LAN."

**M18. MISSING / MEDIUM -- Concurrent admin operations during dominance disagreement.**
Section 2.3 says "Two peers may simultaneously believe they are admin" during GSet convergence and that this is "bounded by flood mode's mixing time." But the spec does not address what happens to KeyBundle operations issued by the transient false admin. Are they valid? Rejected? Queued?

*Fix:* "A KeyBundle issued by a peer that was transiently admin (but loses dominance after GSet convergence) is treated as follows: Revoke operations are honored (revoke-wins). Grant operations are subject to the fencing token: if the token is lower than the true admin's token, the Grant is rejected. Rotate operations from the transient admin are discarded."

**M19. INCOMPLETE / MEDIUM -- Metabolic override interaction with dwell time.**
Section 6.3 (verkko-protocol) says "Convergence metabolic override: immediately transitions to Active or Stress. Dwell time enforcement is suspended." But verkko-mesh Section 7.1 Exit Step 3 says "Restore metabolic dwell time enforcement." The spec does not address: what happens if convergence takes longer than the normal dwell time? Can the override transition to a lower state (Stress -> Active) during convergence?

*Fix:* "During convergence, the metabolic state is locked to the override value (Active or Stress, whichever the governor selects). The state cannot decrease during convergence. If resource pressure increases during convergence (governor output drops), the state can increase from Active to Stress. Dwell time is not enforced for either direction during convergence."

**M20. MISSING / LOW -- Heartbeat extension field ordering and versioning.**
The heartbeat carries mesh-layer extension fields (Section "Heartbeat extension fields") but does not specify how the extension region is versioned or extended. If a future version adds a new extension field, how does a peer that only knows the current format parse the heartbeat? Is the extension region fixed-length or variable-length?

*Fix:* "The extension region is fixed at 18 bytes for version 1. Future versions extend the region by appending fields. The version is carried in the outer protocol version (not in the heartbeat). Peers ignore trailing extension bytes they do not understand."

## verkko-relay (12 findings)

**R1. MISSING / HIGH -- TransitRelay frame forwarding procedure.**
The TransitRelay defined concept says "Relay peer forwards frames without decrypting content" but the body says nothing about how transit forwarding works. No forwarding table, no routing algorithm, no TTL decrement, no loop detection beyond the hop_count field in the wire format. The taxonomy acknowledges this: "Transit forwarding spec: Missing."

*Fix:* Specify: "Relay maintains a forwarding table: {destination_hash -> next_hop_connection_id}. On receiving a frame with hop_count > 0 and destination_hash not matching any local CIK: decrement hop_count, look up destination_hash in forwarding table, forward on matching connection. If no match, drop frame. Forwarding table populated from mesh topology gossip."

**R2. MISSING / HIGH -- RelayTopology selection algorithm.**
The RelayTopology defined concept says "Relay selection based on connectivity graph and health" but the body provides no algorithm. How does a peer decide which other peer to use as a relay? What metrics? What if multiple relay paths exist?

*Fix:* "Relay selection: when direct connection to a cluster fails after hole-punch attempt, select the peer with the highest effective_health score that has direct connections to both the local cluster and the target cluster. If multiple candidates, prefer the one with lowest RTT. Cache relay selection for the duration of the direct connection failure."

**R3. INCOMPLETE / MEDIUM -- Membrane 5 operations lack message formats.**
Section 4.3 lists five operations (PUT, GET, DELETE, TOUCH, SIGNAL) but specifies message formats only for BatchGET. PUT, DELETE, TOUCH, and SIGNAL have no wire format specification.

*Fix:* Specify formats:
```
PUT:    { relay_key_id: [u8; 8], ciphertext: Vec<u8>, ttl_days: u16 }
DELETE: { relay_key_id: [u8; 8] }
TOUCH:  { relay_key_ids: Vec<[u8; 8]>, ttl_days: u16 }
SIGNAL: { relay_key_id: [u8; 8], signal_type: u8, payload: Vec<u8> }
```

**R4. INCOMPLETE / MEDIUM -- Relay TTL and TOUCH interaction.**
Section 4.3 says "Default TTL: 90 days, TOUCH every 30 days." But: (a) does TOUCH reset the TTL to 90 days from the TOUCH time, or extend by 30 days? (b) what happens if a blob's TTL expires? Is it immediately deleted or garbage-collected? (c) is the 90-day TTL configurable?

*Fix:* "TOUCH resets the TTL to 90 days from the TOUCH timestamp. Expired blobs are garbage-collected in a background sweep (once per hour). The 90-day TTL is not configurable (it is a protocol constant, not a user setting)."

**R5. INCOMPLETE / MEDIUM -- Constant-rate padding batch_size derivation.**
Section 4.5 says "batch_size constant within an epoch (derived from HKDF of epoch key)" but does not specify the HKDF info string, the output range, or the minimum/maximum batch_size.

*Fix:* "batch_size = HKDF-Expand(prk=epoch_key, info='relay:batch-size:v1', len=2) mod (MAX_BATCH - MIN_BATCH) + MIN_BATCH. MIN_BATCH = 5, MAX_BATCH = 20. New batch_size on each macro-epoch advance."

**R6. MISSING / MEDIUM -- Relay authentication and authorization.**
The spec says "Relay peer forwards frames" but does not specify how a relay authenticates the sender, whether it validates that the sender is a mesh member, or whether any peer can use any other peer as a relay.

*Fix:* "A relay only forwards frames from peers with which it has an active Noise session. Relay service is offered only to peers in the same mesh (verified by founding_hash in the Noise prologue). A peer can relay for any mesh peer but is not required to."

**R7. INCOMPLETE / MEDIUM -- Re-encryption batch size and pacing.**
Section 4.4 describes re-encryption priority tiers and deadline-based pacing but does not specify the batch size for re-encryption operations. How many blobs per batch? How is the batch submitted to the relay (one PUT per blob, or a multi-PUT)?

*Fix:* "Re-encryption batches contain up to 16 blobs. Each blob is a separate PUT to the relay. Batches are paced at the LEDBAT bulk rate. P1 blobs are submitted individually (no batching) for minimum latency."

**R8. UNJUSTIFIED / LOW -- TTL values (90-day, 30-day TOUCH).**
Section 4.3 states these values without justification. Why 90 days and not 60 or 180? Why TOUCH every 30 days and not every 60?

*Fix:* "90-day TTL exceeds the longest expected offline period for a household device (vacation). 30-day TOUCH interval ensures at least 2 TOUCH opportunities before expiry (90/30 = 3), tolerating one missed TOUCH cycle."

**R9. MISSING / LOW -- DELETE operation authorization.**
Section 4.3 lists DELETE as one of five membrane operations but does not specify who can delete. Can any mesh peer delete any blob? Only the owner (per territorial partitioning)? Only the admin?

*Fix:* "DELETE is authorized for the territorial owner of the relay_key_id (per the consistent hash ring) and for the mesh admin. Other peers' DELETE requests are silently dropped."

**R10. INCOMPLETE / MEDIUM -- RelayCorrelationThreatModel lacks quantitative bounds.**
The defined concept says "Correlation attacks are within threat model but bounded by constant-rate padding and salted key rotation" but provides no quantitative leakage analysis. How many observations does the relay need to correlate two clusters? What is the advantage of the padding?

*Fix:* "With batch_size B and R real operations per batch, the relay learns log2(C(B, R)) bits per batch about the real operation count. At B=10, R=5: ~8 bits per batch. Over 100 batches: the relay can estimate the real operation rate to within ~10%. Salted key rotation (90-day) prevents long-term traffic correlation across rotation boundaries."

**R11. MISSING / MEDIUM -- Relay storage capacity management.**
The spec does not address what happens when a relay peer's storage is full. No eviction policy, no notification to the mesh, no refusal mechanism for PUT.

*Fix:* "When relay storage exceeds 90% capacity, the relay includes a STORAGE_FULL flag in its heartbeat work_capacity field (value 0). The mesh stops selecting this peer for new relay storage. Existing blobs continue to be served and TOUCH-ed. At 100% capacity, PUT returns ERR_STORAGE_FULL."

**R12. INCOMPLETE / LOW -- SIGNAL operation is undefined.**
Section 4.3 lists SIGNAL as one of five membrane operations ("connection setup") but provides no specification. What signals? What is the payload? What triggers a SIGNAL?

*Fix:* "SIGNAL is used for relay-assisted connection setup. Signal types: CONNECT_REQUEST (payload: initiator connection_id, CIK fingerprint), CONNECT_RESPONSE (payload: responder connection_id, OBSERVED_ADDR). The relay forwards SIGNAL frames between peers that cannot communicate directly."

## verkko-seat (14 findings)

**S1. MISSING / CRITICAL -- verkko-seat has no body.**
The entire document is seven defined concepts with invariants. There is no body section. No wire format, no message types, no state machine, no authentication flow, no error handling. The taxonomy acknowledges this: "Coverage from v10: ~0%."

This is the most significant gap in the specification. Two independent implementations of the seat protocol will produce incompatible results because there is nothing to implement from.

*Fix:* A minimum viable body must specify:
1. Authentication flow: how the seat proves its identity to the host peer
2. Session establishment: what Noise pattern is used (Noise XX with user credential as PSK?)
3. Message format: request/response framing (correlation ID, type, payload)
4. Event subscription: how the seat subscribes, how events are framed
5. Instruction format: the six instruction types and their wire encoding
6. Error codes: standard error responses
7. Reconnection: how resumption tokens work
8. Concurrency: how multiple seats on the same host peer are isolated

**S2. MISSING / HIGH -- SeatSession authentication mechanism.**
The defined concept says "Seat authenticates with a user credential (not a DK/DEK)" but does not specify what a user credential is. Password? Token? Certificate? Biometric? How is the credential transmitted securely? How is it validated by the host peer?

*Fix:* "User credential is a bearer token derived from a password via Argon2id (tuned to 100ms on target hardware). Token = BLAKE3('seat-auth:v1' || Argon2id(password, salt) || host_peer_DK_pub). Transmitted inside the Noise handshake payload. Validated by the host peer against stored salted hashes."

**S3. MISSING / HIGH -- Seat role mapping to mesh access control.**
The Seat defined concept says "A seat has a user identity and a role. The host peer maps the role to mesh-level access control." But the role types are not defined, the mapping is not specified, and the enforcement mechanism is not described.

*Fix:* "Seat roles: Admin, User, Child (matching the saalis role enum). Mapping: Admin seats can issue KeyBundle operations via the host peer. User seats have read/write access to authorized channels. Child seats have read-only access to a configured channel subset. The host peer enforces the mapping before executing any request."

**S4. MISSING / HIGH -- SeatRequest message catalog.**
The defined concept says requests are "typed via protobuf schema" but no request types are enumerated. An implementor does not know what operations a seat can perform.

*Fix:* At minimum define the core request types: QueryEntities, GetEntityDetail, StartDownload, CancelDownload, GetStashStatus, ListChannels, SubscribeEvents, ExecuteInstruction, ReportProgress.

**S5. MISSING / HIGH -- SeatEvent event categories.**
The defined concept says "Seat subscribes to event categories during session setup" but no categories are defined.

*Fix:* Define categories: DownloadProgress, StashChange, MeshStatus, EntityUpdate, InstructionStatus, ErrorNotification.

**S6. MISSING / HIGH -- SeatInstruction execution model.**
The defined concept says instructions are idempotent and lists types (download, place, link, remove, verify, inventory sync) but provides no format, no success/failure reporting, no timeout, no retry policy.

*Fix:* Specify each instruction type: "download(url, target_path, expected_blake3) -> result. place(matter_id, target_path) -> result. link(source_path, target_path) -> result. remove(target_path) -> result. verify(target_path, expected_blake3) -> result. inventory_sync(stash_delta) -> result. Timeout: 5 minutes per instruction. Retry: host peer retries up to 3 times with exponential backoff."

**S7. MISSING / MEDIUM -- Seat session lifecycle.**
No specification of connection establishment, keepalive, disconnect, or reconnect. How does the seat discover the host peer? Is it always on the same LAN? Can a seat connect over WAN?

*Fix:* "Seats discover host peers via mDNS on LAN (_verkko-seat._tcp.local). WAN seats connect via invite link containing the host peer's address and a one-time PSK. Keepalive: ping every 30 seconds. Disconnect: explicit CLOSE frame or 3 missed pings. Reconnect: automatic with resumption token within 24 hours."

**S8. MISSING / MEDIUM -- Seat maximum concurrency.**
The defined concept says "Multiple seats can connect to the same host peer simultaneously" but does not specify a maximum. Is it 2? 10? 100? What happens when the limit is reached?

*Fix:* "Maximum 8 concurrent seats per host peer. Additional connection attempts receive ERR_TOO_MANY_SEATS. Active seats are evicted LRU when memory pressure exceeds the ThresholdGuard limit."

**S9. MISSING / MEDIUM -- SeatProgress format.**
The defined concept says "Streamed incrementally, not batched" but no format is specified. What fields does a progress update contain? How often are updates sent? Is there backpressure?

*Fix:* "Progress update: { instruction_id: u32, bytes_completed: u64, bytes_total: u64, eta_seconds: u32 }. Sent at most once per second per instruction. If the host peer's receive buffer is full, progress updates are dropped (lossy)."

**S10. MISSING / MEDIUM -- Seat disconnect behavior.**
What happens to in-progress instructions when a seat disconnects? Are they cancelled? Do they continue in the background? Who decides?

*Fix:* "In-progress downloads continue in the background after seat disconnect (the mesh peer manages the download). Other instructions (place, link, remove, verify) are cancelled on disconnect. inventory_sync is idempotent and can be retried on reconnect."

**S11. MISSING / MEDIUM -- Seat-to-seat communication.**
Can one seat see another seat's activity? Can a phone seat see that a tablet seat is downloading something? The spec is silent on inter-seat visibility.

*Fix:* "Seats on the same host peer can observe each other's download progress and stash changes via the MeshStatus and DownloadProgress event categories. Seats cannot send messages to each other directly; all communication goes through the host peer."

**S12. MISSING / LOW -- Seat offline behavior.**
What does a seat do when the host peer is unreachable? Cache? Retry? Show stale data?

*Fix:* "When the host peer is unreachable, the seat displays a disconnected indicator and retries connection every 5 seconds with exponential backoff (max 60 seconds). Cached entity metadata from the last session may be displayed as stale. No writes are queued."

**S13. MISSING / LOW -- Seat key material handling.**
The defined concept says seats "do not hold channel keys" but they do hold a session key (TransportAEAD). How long is this key valid? When is it rotated? Is it stored on the seat device?

*Fix:* "The seat's transport session key is valid for the session duration (max 24 hours). It is stored in memory only, never persisted to disk. On session expiry, the seat must re-authenticate."

**S14. MISSING / LOW -- Seat protobuf schema delivery.**
The SeatSession depends on verkko-protocol: SchemaSync, but SchemaSync itself has no body. How does the seat get the protobuf schema for requests and events?

*Fix:* "SchemaSync runs during seat session establishment. The host peer sends the current FileDescriptorSet for seat request/response/event types. The seat uses this schema for all subsequent communication."

## verkko-matter (13 findings)

**MT1. MISSING / HIGH -- MatterTransfer protocol has no body.**
The defined concept says "LAN path: direct streaming with BLAKE3 verification at end. WAN path: bao-tree verified streaming." But the body contains no transfer protocol specification. No handshake, no chunk size, no flow control, no resumption, no error recovery. The taxonomy says "Blob/matter transfer protocol: Missing."

*Fix:* Specify at minimum: "Transfer initiation: requester sends MATTER_REQUEST(matter_id, offset). Provider responds with MATTER_HEADER(matter_id, total_size, chunk_size). Chunks are sent as reliable ordered frames on a dedicated stream. Chunk size: 64 KB (LAN), 16 KB (WAN). LAN transfers verify BLAKE3 hash after final chunk. WAN transfers use bao-tree verification per chunk. On verification failure: abort transfer, scar source, retry with next provider."

**MT2. MISSING / HIGH -- Replication has no body.**
The defined concept lists invariants (min_replicas, target_replicas, background work) but the body provides no specification. No replication algorithm, no replica placement strategy, no re-replication trigger, no bandwidth allocation.

*Fix:* Specify: "Replication target: min_replicas=2, target_replicas=3 (configurable per matter priority). Background replication work unit: select matter with replicas < target, select target peer with lowest load and highest health, initiate MatterTransfer. Replication bandwidth: LEDBAT bulk tier. Re-replication trigger: peer departure or stash status change to Missing/Corrupt."

**MT3. MISSING / HIGH -- SyncReconciliation has no body.**
The defined concept says "Divergence detected via Merkle root comparison in heartbeats. Delta sync for incremental changes. Merge rules: field-level, deterministic." But the body specifies only cuckoo filter reconciliation (which is for content routing, not state sync). There is no specification for how mesh-wide state (entity metadata, ratings, personal data) is synchronized.

*Fix:* This is a fundamental gap. At minimum specify: what state is synchronized (entity metadata, stash status, ratings), the CRDT type for each (LWW-Register, G-Counter, OR-Set), the delta encoding format, the reconciliation trigger (heartbeat Merkle root divergence), and the merge conflict resolution for each state type.

**MT4. INCOMPLETE / MEDIUM -- ContentFilter does not specify the key for cuckoo insertion.**
Section 4.2 says "10-bit fingerprints, 2-bit saturating counters, 4-entry buckets" but does not specify the hash function used for cuckoo hashing (bucket index and fingerprint derivation from blob_id). Two implementations using different hash functions will produce different filter states.

*Fix:* "Cuckoo bucket index: BLAKE3(blob_id)[0..4] mod bucket_count. Fingerprint: BLAKE3(blob_id)[4..6] >> 6 (10 bits). Alternate bucket: bucket_index XOR BLAKE3(fingerprint)[0..4] mod bucket_count."

**MT5. INCOMPLETE / MEDIUM -- BandwidthPolicy has no body.**
The defined concept says "Policies published via gossip (time windows, rate limits). Per-cluster rules." But the body provides no specification. No policy format, no enforcement mechanism, no default policy.

*Fix:* "BandwidthPolicy format: { source_cluster: CIK_fingerprint, time_windows: [(start_hour, end_hour, max_kbps)], blocked_windows: [(start_hour, end_hour)] }. Distributed via control log. Enforced at the gateway: bulk transfers to the target cluster are rate-limited per the policy. Default policy: no restrictions."

**MT6. INCOMPLETE / MEDIUM -- Stash status transitions.**
The Stash defined concept lists four statuses (Present, Downloading, Missing, Corrupt) but does not specify the valid transitions between them, what triggers each transition, or how Corrupt is detected (beyond the bao-tree verification during transfer).

*Fix:* "Valid transitions: Missing -> Downloading (transfer initiated), Downloading -> Present (transfer complete, hash verified), Downloading -> Missing (transfer failed), Present -> Corrupt (periodic verification fails), Corrupt -> Downloading (re-download initiated), Corrupt -> Missing (admin deletes). Corruption detection: background verification of BLAKE3(file) == matter_id, once per 7 days per matter."

**MT7. INCOMPLETE / MEDIUM -- Matter source selection predicate is informal.**
Section 4.2 says: "possession AND access AND reachability AND health(effective_score) AND NOT assembling(raptor)." But no effective_score threshold is given. What health score is "good enough" to use as a source?

*Fix:* "Source health threshold: effective_score >= 0.70 (Degraded or better). Below 0.70 (Bad), the source is excluded. Threshold is not configurable."

**MT8. MISSING / MEDIUM -- Multi-source transfer coordination.**
The MatterTransfer defined concept says "Multi-source: sequential provider fallback" but does not specify the fallback conditions. How long does the requester wait before trying the next source? Is it timeout-based? Error-based? What if no source has the complete matter?

*Fix:* "Sequential fallback: if the current source fails to deliver a chunk within 3 * RTT, or sends a corrupt chunk, the requester disconnects and tries the next source from the ranked list. If all sources are exhausted, the requester waits for one heartbeat interval and retries the full list. If no source has the matter (not in any content filter), the requester falls back to external sources (connector download)."

**MT9. INCOMPLETE / MEDIUM -- SealedRange catch-up does not specify UnsealGrant format.**
Section 2.8 says "admin can issue UnsealGrant" for Absence-type sealed ranges. But UnsealGrant is never defined as a message type. Is it a KeyBundle operation? A control log entry? What fields does it contain?

*Fix:* "UnsealGrant is a control log entry: { peer_id: PeerId, sealed_range: SealedRange, admin_signature: [u8; 64] }. On receipt, the peer marks the sealed range as unsealed and derives the sealed epoch keys via HKDF chain from K_epoch_start."

**MT10. MISSING / LOW -- Matter deletion and garbage collection.**
The spec says matter is immutable (same matter_id always refers to the same bytes) but never specifies how matter is deleted. Can an admin delete matter? Is there a reference count? What happens to replicas?

*Fix:* "Admin can issue MATTER_DELETE(matter_id) via control log. All peers remove the matter from their stash and content filter. Relay peers delete the corresponding membrane blob on next TOUCH cycle (no explicit DELETE sent; TTL expiry handles it). Reference counting is not needed; the control log entry is authoritative."

**MT11. UNJUSTIFIED / LOW -- Cuckoo filter load factor 90%.**
Section 4.2 says "90% load factor" without justification. Standard cuckoo filter literature [Fan2014] recommends 95% as achievable. At 90%, more space is wasted. At 95%, insertion failure rate increases.

*Fix:* "90% load factor chosen as a conservative target: at 95%, the insertion failure rate increases significantly with 4-entry buckets and 10-bit fingerprints. The ~5% extra space cost (8 KB at 100K entities) is acceptable for lower insertion failure probability."

**MT12. INCOMPLETE / MEDIUM -- MerkleReconciliation lacks specification for the Merkle tree structure.**
Section 3.7 says "Merkle tree walk" with "depth-8 Merkle tree cache per filter" but does not specify: the tree construction algorithm, the hash function for internal nodes, whether nodes are hashed over children's hashes or over the filter data directly.

*Fix:* "Merkle tree: binary tree of depth 8 (256 leaves). Each leaf covers filter_size/256 buckets. Leaf hash = BLAKE3(bucket_data). Internal node hash = BLAKE3(left_child_hash || right_child_hash). Root hash = top internal node. 8-byte truncated root in heartbeat; 32-byte full root every 10 minutes."

**MT13. MISSING / LOW -- External source fallback mechanism.**
The MatterTransfer defined concept says "External sources are last resort" but the body does not specify how external sources are invoked. Is this via saalis connectors? How does the mesh coordinate to avoid duplicate external downloads?

*Fix:* "External source download is delegated to the connector subsystem (saalis-connector-*). The requesting peer's gateway coordinates: only one peer per mesh initiates the external download for a given matter_id. Coordination via a DOWNLOAD_CLAIM gossip message. If the claiming peer fails within 5 minutes, another peer may claim."

## Cross-document findings (5 findings)

**X1. MISSING / CRITICAL -- No protobuf schema or message catalog anywhere.**
Multiple documents reference protobuf (verkko-protocol: SchemaSync, verkko-seat: SeatRequest), but no document provides a protobuf schema, message catalog, or even a list of message types. The entire system's wire protocol depends on protobuf, but the message definitions do not exist.

*Fix:* Create a message catalog document or appendix listing all protobuf message types: Heartbeat, KeyBundle, ScarEntry, PartitionAnnouncement, FlowSummary, MatterRequest, MatterHeader, SeatRequest, SeatResponse, SeatEvent, SeatInstruction, SeatProgress. Each with field definitions.

**X2. INCOMPLETE / HIGH -- Channel concept spans three documents with no single specification.**
Channel is mentioned in verkko-crypto (ChannelKey), verkko-mesh (Channel defined concept, access levels), and verkko-matter (content routing). The Channel defined concept in verkko-mesh says "channel_index is a u16" but does not specify: the channel namespace, hierarchical structure (the invariant says "hierarchical data stream" but the spec shows flat u16 indices), channel creation/deletion, or default channels.

*Fix:* Consolidate the Channel specification in verkko-mesh. Define: "Channels are flat (u16 index, not hierarchical despite the term 'hierarchical data stream' in earlier drafts). Channel 0 is the control channel (always exists). Channels 1-49 are user channels (created by admin via control log entry). Maximum 50 channels per mesh."

**X3. INCOMPLETE / MEDIUM -- Cross-document constant consistency.**
Several constants appear in multiple documents with potential for drift:
- Heartbeat payload size: verkko-protocol says "18 bytes fixed header + 18 bytes extension = 36 bytes." verkko-mesh heartbeat extension fields list fields summing to 18 bytes. Consistent.
- Frame overhead: "55 bytes" in verkko-protocol. Verified.
- "12 clusters" appears in verkko-mesh, verkko-protocol, verkko-relay. Consistent.
- However: verkko-crypto Section 2.1 says "11 recipients" (N-1 for broadcast). verkko-mesh says "12 clusters." These are consistent (broadcast to 11 others). But verkko-protocol says "11 connections" in multiple places, which is also correct. The constant is consistent but the relationship (12 clusters -> 11 connections -> 11 recipients) should be stated once and referenced.

*Fix:* Add a constants table to verkko-ops: "MESH_MAX_CLUSTERS = 12. MESH_MAX_CONNECTIONS = MESH_MAX_CLUSTERS - 1 = 11. BROADCAST_RECIPIENTS = MESH_MAX_CLUSTERS - 1 = 11."

**X4. MISSING / MEDIUM -- Error classification and error codes.**
No document defines error codes, error classification, or error handling behavior beyond specific cases (NoncePersistFailed, E_ENTROPY_FAILURE). There is no unified error model. An implementor of verkko-protocol does not know what error to return when a frame fails AEAD decryption, when a connection times out, or when a session renegotiation fails.

*Fix:* Define an error classification in verkko-protocol: "Fatal errors (halt): nonce persist failure, double corruption. This is a circular reference at the specification level. verkko-relay's dependencies list verkko-mesh, and verkko-mesh's convergence spec references verkko-relay for Front 3 re-encryption. The taxonomy says the dependency graph is "strictly acyclic" but the specification text has a circular reference between convergence and re-encryption.

*Fix:* Clarify that the dependency is unidirectional: verkko-relay depends on verkko-mesh for the convergence signal (Front 3 gate). verkko-mesh references verkko-relay's defined concepts (TerritorialReencryption) as the mechanism that satisfies Front 3, but does not depend on relay internals. ## Assessment

The cryptographic layer (verkko-crypto) is the most complete. The mesh layer (verkko-mesh) is dense and well-thought-out but has gaps in operational procedures (invite flow, CIK rotation). The protocol layer (verkko-protocol) has a solid transport section but is missing several defined concepts' bodies. The relay layer (verkko-relay) has good relay-at-rest design but lacks transit forwarding. verkko-seat and verkko-matter are skeletons: defined concepts with invariants but insufficient body content for implementation.

The overall spec is approximately 70% complete for the crypto and mesh layers, 55% for protocol and relay, and 15-25% for seat and matter. The four CRITICAL findings (seat body, Noise protocol name, connection ID, message catalog) must be resolved before any implementation work begins, as they affect wire-level interoperability.
