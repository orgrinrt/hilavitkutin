## Part 1: Independent Assessment

### 1.1 State of the relay and NAT traversal design

After reading all seven documents and the Run 11 synthesis, I can
characterize the current state of relay and NAT traversal design in
three words: membrane without plumbing.

The verkko-relay document specifies a sophisticated at-rest storage
model (the membrane), a territorial re-encryption scheme with
deterministic nonces, constant-rate padding for traffic analysis
defense, and five relay operations (PUT, GET, DELETE, TOUCH, SIGNAL)
with complete wire formats (R11-040). This is strong work. The
membrane model is well-conceived and the idempotent PUT contract with
deterministic nonces is elegant.

What is missing is the core reason relays exist: forwarding traffic
between peers that cannot directly reach each other. The TransitRelay
defined concept has three invariants and no body text. The
NATTraversal defined concept in verkko-protocol has three invariants
and no body text. The SIGNAL operation in the relay wire format has
three types (HELLO=0x0001, PUNCH=0x0002, REDIRECT=0x0003) but no
specification of what these signals contain, when they are sent, or
what the receiver does with them. The hop_count field exists in the
outer frame header (max 4, decremented per relay) but there is no
forwarding table, no path selection, no relay authentication, and no
relay session establishment procedure.

This is the equivalent of specifying a TCP stack's congestion control
algorithm in detail while leaving the SYN handshake blank. The
membrane is valuable only if traffic can reach it. Right now, the
design has no mechanism for a peer behind a symmetric NAT to send or
receive any verkko frames at all.

### 1.2 NAT type analysis for the verkko deployment envelope

The design envelope is 2-12 clusters, ~36 peers, household devices.
The typical deployment scenarios are:

**Scenario A: Single household, single NAT.** All peers behind one
router. No relay needed. mDNS discovery works. This is the easy
case.

**Scenario B: Two households, consumer NAT.** Two clusters, each
behind a consumer router. The NAT types we encounter in practice
(based on my experience with 10M+ concurrent relayed connections at
a major video conferencing platform, and corroborated by Tailscale's
2020 NAT survey and Google's WebRTC statistics):

| NAT Type (RFC 4787) | Prevalence | Direct connectivity |
|---------------------|-----------|---------------------|
| Endpoint-independent mapping (full cone) | ~30% | Hole punch succeeds ~95% |
| Address-dependent mapping | ~25% | Hole punch succeeds ~80% |
| Address+port-dependent mapping (symmetric) | ~40% | Hole punch succeeds ~5-15% |
| Port-restricted cone | ~5% | Hole punch succeeds ~60% |

At 40% symmetric NAT prevalence and 12 clusters, the probability
that at least one cluster pair requires relay is:
`1 - (1 - 0.40 * 0.40 * 0.85)^66 = ~100%`. Any mesh with more
than 3 clusters will almost certainly have at least one relayed pair.

**Scenario C: Cross-carrier mobile.** A phone on cellular behind
CGNAT. CGNAT is almost always symmetric NAT with short mapping
lifetimes (30-120 seconds). However, verkko-seat says seats connect
via WebSocket on LAN only, so this scenario applies only to full
mesh peers, not seats.

**Scenario D: Double NAT.** ISP-level CGNAT plus household router.
Both layers are NATs. Hole punching must traverse both. This is
increasingly common in IPv4-exhausted regions. Hole punch success
rate drops to ~2-5% for symmetric-behind-symmetric.

The conclusion: relay is not optional. It is required for any
multi-household deployment. The design must specify a complete
relay path from NAT discovery through session establishment through
frame forwarding through failover.

### 1.3 What the frame format already provides

The wire format (verkko-protocol Section 3.1) has the right bones
for relay forwarding:

- `destination` (4 bytes): BLAKE3(CIK_pub)[0..4]. Identifies the
  target cluster. A relay can route on this field.
- `connection_id` (4 bytes): identifies the session. A relay
  can use this for per-session routing table entries.
- `hop_count` (1 byte): TTL, max 4. Decremented per relay.
  Prevents forwarding loops.
- `relay_key_id` (8 bytes): HMAC-BLAKE3(channel_salt, key_id).
  Used for membrane operations, not for forwarding decisions.
- `nonce` (12 bytes): per-session monotonic. Relay does not
  inspect this.

The frame format is relay-aware. The relay sees the outer header
(29 bytes) in plaintext and forwards the entire frame. The
encrypted payload is opaque to the relay. This is the correct
architecture: the relay is a dumb forwarder with a routing table.

The AEAD associated data includes the outer header (29 bytes),
which means the relay cannot modify the destination, connection_id,
or nonce without invalidating the auth tag. The relay CAN decrement
hop_count because... wait. If hop_count is part of the outer header
and the outer header is the AEAD associated data, then decrementing
hop_count will invalidate the auth tag. This is Problem 1 and it
is critical.

### 1.4 The hop_count AEAD binding problem

The outer header is 29 bytes. The AEAD associated data is "the outer
header (29 bytes)" per verkko-protocol Section 3.1. hop_count is at
offset 8 of the outer header. If the relay decrements hop_count (as
required for TTL enforcement), the recipient's AEAD verification
will fail because the associated data no longer matches what the
sender authenticated.

There are three solutions, ranked by my preference:

**Option A: Exclude hop_count from AEAD AD.** AEAD AD = outer header
bytes [0..8] || outer header bytes [9..29] (28 bytes). The hop_count
byte is excluded. This is what QUIC does with its packet number field
(RFC 9001 Section 5.4): QUIC encrypts the packet number separately
to allow intermediaries to process it. Similarly, WireGuard's
roaming design leaves the outer IP/UDP headers unauthenticated.

**Option B: Relay re-authenticates.** The relay strips the old frame,
decrements hop_count, and wraps it in a new frame with the relay's
own AEAD. This requires the relay to hold keying material, which
violates the membrane's "relay sees opaque blobs" invariant.

**Option C: hop_count is not in the outer header.** Move hop_count
inside the encrypted payload. The relay cannot read or modify it.
Instead, the relay prepends a separate relay header with its own
hop_count. This adds per-hop overhead but preserves end-to-end
integrity.

I recommend Option A. It is the simplest, it preserves the relay's
role as a dumb forwarder, and the security cost is minimal: an
attacker who can modify hop_count can only reduce TTL (which causes
the frame to be dropped sooner) or increase it (which allows one
extra hop). Neither threatens confidentiality or integrity.

The counterargument is that an attacker modifying hop_count could
cause frames to circulate longer in a loop, amplifying traffic.
But hop_count max is 4, so the amplification is bounded at 4x,
and the relay's routing table prevents loops for non-malicious
traffic.

### 1.5 Relay forwarding: what happens per packet

I will now specify the forwarding procedure that the TransitRelay
body text should contain. This is based on RFC 8656 (TURN) Section
5 forwarding model, adapted for verkko's frame format and threat
model.

A relay peer maintains a forwarding table:

```
ForwardingTable {
    entries: HashMap<(u32 destination, u32 connection_id), ForwardingEntry>,
    max_entries: u16,  // default 256, configurable
}

ForwardingEntry {
    next_hop: SocketAddr,
    created_at: Instant,
    last_activity: Instant,
    bytes_forwarded: u64,
    frames_forwarded: u64,
    ttl: Duration,  // default 5 minutes, refreshed on activity
}
```

**Per-frame forwarding procedure:**

```
fn forward(frame: &[u8], source: SocketAddr) -> ForwardAction {
    // 1. Parse outer header (29 bytes). No AEAD needed.
    let destination = u32::from_be_bytes(frame[0..4]);
    let connection_id = u32::from_be_bytes(frame[4..8]);
    let hop_count = frame[8];

    // 2. TTL check.
    if hop_count == 0 {
        return ForwardAction::Drop;
    }

    // 3. Loop detection: if destination matches self, process locally.
    if destination == self.cluster_destination_id {
        return ForwardAction::ProcessLocally;
    }

    // 4. Lookup forwarding table.
    let key = (destination, connection_id);
    match self.table.get_mut(&key) {
        Some(entry) => {
            // 5. Decrement hop_count (byte 8).
            let mut forwarded = frame.to_vec();
            forwarded[8] = hop_count - 1;

            // 6. Update accounting.
            entry.last_activity = Instant::now();
            entry.bytes_forwarded += frame.len() as u64;
            entry.frames_forwarded += 1;

            // 7. Forward.
            ForwardAction::Send(entry.next_hop, forwarded)
        }
        None => {
            // 8. No entry. If this is a SIGNAL frame, handle it.
            // Otherwise, drop silently.
            ForwardAction::Drop
        }
    }
}
```

**Cost per forwarded frame:**
- CPU: one hash lookup (~50ns) + one memory copy (~200ns for 1232-byte
  frame on ARM Cortex-A72) + one byte decrement = ~300ns per frame.
- At 10,000 frames/second (a pathological case for a household relay):
  3ms of CPU per second, ~0.3% of one core.
- Memory: 256 entries * ~100 bytes = ~25 KB for the forwarding table.

The relay does NOT perform AEAD. It does NOT decrypt. It does NOT
inspect the encrypted payload. It copies the frame, decrements
hop_count, and sends it to the next hop. This is the same model as
Tailscale's DERP relay [Tailscale DERP, 2020]: a stateless (per-frame)
packet forwarder with a routing table that maps (destination, session)
to next-hop address.

### 1.6 Relay session establishment

Before forwarding can begin, the relay must have a forwarding table
entry. This is where the SIGNAL operation (from the relay wire format
R11-040) comes in. I now specify the SIGNAL subtypes:

**SIGNAL HELLO (0x0001):**

Sent by a peer to a relay to register itself as reachable via that
relay. The peer establishes a Noise KK session with the relay peer
(the relay IS a mesh peer, it has a DK/DEK/CIK). After session
establishment, the peer sends a SIGNAL HELLO:

```
SIGNAL_HELLO {
    signal_type: u16 = 0x0001,
    cluster_destination_id: u32,   // BLAKE3(CIK_pub)[0..4]
    keepalive_interval: u16,       // seconds, 0 = relay decides
    reserved: [u8; 2],
}
// 10 bytes payload
```

On receipt, the relay creates a forwarding table entry:
```
(cluster_destination_id, 0xFFFFFFFF) -> source_address
```
The wildcard connection_id (0xFFFFFFFF) means "any connection to this
destination routes through this relay." When the relay receives a
frame with a matching destination and unknown connection_id, it
creates a specific entry.

**SIGNAL PUNCH (0x0002):**

Relay-assisted hole punching. The peer behind a NAT asks the relay
to coordinate a hole punch with another peer. This is the ICE
(RFC 8445) candidate exchange mechanism adapted for verkko:

```
SIGNAL_PUNCH {
    signal_type: u16 = 0x0002,
    target_destination_id: u32,
    candidate_count: u8,
    reserved: [u8; 1],
    candidates: [Candidate; candidate_count],
}

Candidate {
    candidate_type: u8,    // 0x01=host, 0x02=srflx, 0x03=relay
    addr_type: u8,         // 0x04=IPv4, 0x06=IPv6
    port: u16,
    address: [u8; 16],     // IPv4: first 4, rest zero
    priority: u32,         // ICE priority (RFC 8445 Section 5.1.2)
}
// 24 bytes per candidate
```

The relay forwards SIGNAL PUNCH to the target peer (if reachable).
The target responds with its own candidates. Both peers then attempt
connectivity checks in priority order (highest priority first, as
per RFC 8445 Section 8.1.1).

**SIGNAL REDIRECT (0x0003):**

Relay tells a peer to use a different relay. Used during relay
failover:

```
SIGNAL_REDIRECT {
    signal_type: u16 = 0x0003,
    new_relay_destination_id: u32,
    new_relay_address: [u8; 20],  // AddressHint format
    reason: u8,                    // 0x01=overloaded, 0x02=shutting_down
    reserved: [u8; 1],
}
// 28 bytes payload
```

### 1.7 NAT traversal state machine

I now specify the NAT traversal state machine that should live in the
NATTraversal body text in verkko-protocol. This follows RFC 8445 (ICE)
Section 7 but simplified for verkko's deployment envelope.

```
    IDLE
     |
     v
    GATHERING (collect candidates: host, server-reflexive, relay)
     |
     v
    CHECKING (connectivity checks in priority order)
     |
     +--- success ---> CONNECTED (direct path established)
     |
     +--- all checks failed ---> RELAY (traffic routed through relay)
     |
     v
    RELAY
     |
     +--- periodic re-check (every 5 minutes) ---> CHECKING
```

**Candidate gathering:**

1. **Host candidates.** Local IP addresses on all interfaces.
   Priority: 126 (per RFC 8445 Section 5.1.2.1, local preference).

2. **Server-reflexive candidates.** Discovered via OBSERVED_ADDR in
   the Noise handshake response (already mentioned in NATTraversal
   invariants). The peer sends a Noise handshake to a known relay
   peer; the relay's response includes the observed source address.
   This is equivalent to a STUN Binding Response (RFC 5389
   Section 7.3.1) but piggybacked on an existing handshake message,
   saving a round trip.

   Priority: 100 (lower than host, higher than relay).

3. **Relay candidates.** The relay address where the peer has
   registered via SIGNAL HELLO.
   Priority: 10 (lowest, last resort per TransitRelay invariant
   "transit relay is the last resort after hole punching fails").

**Connectivity check procedure (simplified ICE):**

The peer pairs its local candidates with the remote peer's candidates
(received via SIGNAL PUNCH through the relay). Pairs are sorted by
priority sum (descending). The peer sends a PATH_CHALLENGE (msg_type
0x06, already in the wire format) to each candidate pair. If a
PATH_RESPONSE (msg_type 0x07) arrives within 2 seconds, the pair is
valid.

```
fn connectivity_check(pairs: &[(Candidate, Candidate)]) -> Option<(SocketAddr, SocketAddr)> {
    for (local, remote) in pairs.iter().sorted_by_priority() {
        send_path_challenge(local.addr, remote.addr);
        if recv_path_response(timeout: 2_seconds) {
            return Some((local.addr, remote.addr));
        }
    }
    None  // All checks failed; use relay
}
```

**Timing.** Connectivity checks are staggered at 50ms intervals
(RFC 8445 Section 16 recommends 50ms Ta timer). With 4 candidate
pairs (typical: 1 host + 1 srflx, each side): 200ms to exhaust all
checks. With relay fallback: additional 100ms for relay path setup.
Total NAT traversal latency: 300ms typical, 2 seconds worst case
(all checks timeout).

**NAT keepalive.** Once a direct path is established through a NAT,
the mapping must be refreshed. The adaptive keepalive mentioned in
verkko-protocol Section 3.3 ("discover mapping lifetime via probes,
set interval to 70% of lifetime") is correct. I add the concrete
procedure:

```
fn discover_nat_mapping_lifetime(relay_addr: SocketAddr) -> Duration {
    // Send a PATH_CHALLENGE from local to relay.
    // Wait increasing intervals. When PATH_RESPONSE stops arriving,
    // the NAT mapping has expired.
    // Probe intervals: 15s, 30s, 60s, 120s, 300s.
    // Binary search between last success and first failure.
    // Typical result: 30-300 seconds.
    // Set keepalive to 70% of discovered lifetime.
}
```

For symmetric NATs with very short mapping lifetimes (<30 seconds),
the design correctly specifies "prewarm relay path when lifetime
< 60 seconds." This means: maintain an active relay forwarding
entry in parallel with the direct path, so failover is instant when
the NAT mapping drops.

### 1.8 Relay resource governance

The relay peer is a full mesh peer with its own resource constraints.
It must protect itself from relay amplification attacks and resource
exhaustion. The current design has the ConstantRatePadding and
RelayCorrelationThreatModel defined concepts, which address traffic
analysis. What is missing is relay resource governance.

**Forwarding table limits.** Max 256 entries (configurable). When the
table is full, new SIGNAL HELLO requests receive ERROR with code
RELAY_FULL. The peer should try another relay peer.

**Bandwidth limits per forwarded session.** The relay applies LEDBAT
to forwarded traffic, not just its own traffic. A forwarded session
consumes relay bandwidth. The relay's Conductor governs this:

```
relay_forwarding_budget = min(
    available_bw_kbps * 0.5,   // max 50% of relay's bandwidth for forwarding
    sum(active_sessions) * per_session_cap_kbps
)
per_session_cap_kbps = relay_forwarding_budget / active_sessions
```

At 10 Mbps symmetric and 10 active relayed sessions: 500 kbps per
session. Sufficient for heartbeats (1 kbps) and metadata exchange
(10-50 kbps). Insufficient for bulk matter transfers (which should
use direct paths or multi-hop relay with LEDBAT pacing).

**Amplification prevention.** A relay MUST NOT forward a frame to
an address from which it has not received a recent (within 30 seconds)
authenticated frame. This prevents the relay from being used as a
reflector for DDoS amplification (the classic TURN amplification
attack, mitigated in RFC 8656 Section 8). The "authenticated" part
means the relay has an active Noise session with both endpoints of
the forwarded connection.

### 1.9 Peer discovery for relay selection

The PeerDiscovery defined concept specifies a fallback chain:
mDNS -> invite hint -> cached peers -> DHT (Mainline) -> relay.
The relay_dht_key field in the invite token (8 bytes,
BLAKE3("saalis-dht-v1" || founding_hash)[0..8]) enables DHT-based
relay discovery. This is sound.

What is missing is the relay selection algorithm. When a peer needs
a relay, which relay peer does it choose? The current design says
"relay selection based on connectivity graph and health" (RelayTopology
invariant) but does not specify the algorithm.

I propose:

```
fn select_relay(self_cluster: ClusterId, target_cluster: ClusterId) -> Option<PeerId> {
    // 1. Candidates: all peers with direct connectivity to both
    //    self_cluster and target_cluster.
    let candidates = mesh_peers()
        .filter(|p| p.can_reach(self_cluster) && p.can_reach(target_cluster))
        .filter(|p| p.relay_enabled)  // opt-in, not default
        .collect();

    // 2. Sort by: health (descending), then hop_count to both
    //    endpoints (ascending), then available_bw (descending).
    candidates.sort_by(|a, b| {
        a.health.cmp(&b.health).reverse()
            .then(a.total_hops.cmp(&b.total_hops))
            .then(a.available_bw.cmp(&b.available_bw).reverse())
    });

    // 3. Return best candidate.
    candidates.first()
}
```

**Relay opt-in.** Relaying is opt-in. A peer with limited bandwidth
(e.g., a Raspberry Pi on WiFi) should not be forced to relay for
other peers. The relay_enabled flag is a per-peer configuration
setting. Gateway peers are relay-enabled by default.

### 1.10 OBSERVED_ADDR mechanism

The NATTraversal invariants mention "OBSERVED_ADDR in handshake
response for address discovery." This is the STUN-equivalent
mechanism. I specify it:

During the Noise handshake, the responder includes the initiator's
observed source address in the handshake application payload. This
is sent inside the encrypted Noise channel, so the address is
authenticated and cannot be spoofed by a network intermediary.

```
HandshakeApplicationPayload {
    // ... existing fields (timestamps, schema_hash) ...
    observed_addr: ObservedAddr,
}

ObservedAddr {
    addr_type: u8,      // 0x04=IPv4, 0x06=IPv6
    reserved: u8,
    port: u16,
    address: [u8; 16],
}
// 20 bytes
```

This gives the initiator its server-reflexive address. If
`observed_addr != any(local_addresses)`, the peer is behind a NAT.
The peer then knows its external address and can use it as a
server-reflexive candidate in connectivity checks.

## Part 2: The 5 Most Significant Problems

### Problem 1: hop_count breaks AEAD authentication (BUG)

**(a) Problem statement.**

The outer frame header (29 bytes) is used as AEAD associated data
for ChaCha20-Poly1305 (verkko-protocol Section 3.1). The hop_count
field (offset 8, 1 byte) is part of this outer header. The
TransitRelay defined concept says the relay "forwards frames without
decrypting content." The hop_count is described as "TTL, max 4,
decremented per relay."

If a relay decrements hop_count, the AEAD associated data at the
recipient will differ from what the sender authenticated. The
Poly1305 tag verification will fail. The frame will be silently
dropped. Transit relay forwarding is impossible with the current
frame format specification.

**(b) Why it matters.**

This is a correctness bug that makes the entire relay forwarding
mechanism non-functional. Every frame forwarded through a relay will
be rejected by the recipient. The relay exists to forward frames for
peers that cannot directly reach each other; if forwarding breaks
AEAD, the relay serves no transit purpose. Only the membrane (at-rest
storage) operations work, because those use a separate AEAD context
(StorageAEAD with XChaCha20-Poly1305).

At 40% symmetric NAT prevalence, this affects a majority of
multi-household deployments.

**(c) References and prior art.**

- RFC 9001 (QUIC-TLS) Section 5.4: QUIC encrypts the packet number
  field separately from the payload, specifically so intermediaries
  (proxies, load balancers) can process header fields without
  breaking authentication.
- RFC 8656 (TURN) Section 5: TURN relays operate at the transport
  layer; they re-wrap UDP datagrams in TURN channels. The original
  datagram's integrity is preserved because TURN does not modify it.
- WireGuard: The outer UDP/IP headers are not authenticated. The
  Noise transport payload is authenticated independently.
- IPsec ESP (RFC 4303): The mutable IP header fields (TTL, DSCP)
  are explicitly excluded from the ICV computation (Section 3.3.3.1.1).

All production relay/forwarding systems handle this by either
excluding mutable fields from authentication or using a layered
authentication model.

**(d) Recommended approach.**

Exclude hop_count from AEAD associated data. Redefine AEAD AD as:

```
AEAD_AD = outer_header[0..8] || outer_header[9..29]
// 28 bytes: destination(4) + connection_id(4) + relay_key_id(8) + nonce(12)
// hop_count (offset 8) excluded
```

This preserves authentication of destination, connection_id,
relay_key_id, and nonce. The hop_count becomes a mutable field
that the relay can decrement without breaking authentication.

Security analysis: an attacker who can modify hop_count can:
- Set hop_count to 0, causing the frame to be dropped. This is a
  denial-of-service, equivalent to dropping the packet.
- Set hop_count to 4, allowing the frame to traverse more hops.
  Bounded by max hop_count (4). Does not leak plaintext.

Neither attack gives the adversary anything beyond what dropping
the packet gives. The security cost is negligible.

### Problem 2: Transit relay forwarding body is blank (P2-4)

**(a) Problem statement.**

The TransitRelay defined concept has three invariants:
1. Relay forwards without decrypting.
2. Relay sees connection IDs and frame sizes only.
3. Transit relay is last resort after hole punching.

The body section has no text specifying: how a relay learns where to
forward a frame, how forwarding table entries are created, how they
expire, what the relay does with frames it cannot route, or how the
relay handles multiple simultaneous relayed sessions.

**(b) Why it matters.**

Without a forwarding procedure, an implementer cannot build a relay.
The membrane operations (PUT/GET/DELETE/TOUCH/SIGNAL) have wire
formats. The transit forwarding path has nothing. An implementer
reading the current design would know how to store encrypted blobs
on a relay but not how to forward a single frame through one.

The relay's forwarding function is the highest-throughput operation
in the relay: every frame in a relayed session passes through it.
At 1,000 frames/second per session and 10 sessions, the relay
processes 10,000 frames/second. The membrane operations (PUT/GET)
are orders of magnitude less frequent. The most common operation
is the least specified.

**(c) References and prior art.**

- RFC 8656 (TURN) Section 5: Allocations create permissions and
  channels. Each allocation has a relayed transport address.
  Forwarding is based on channel number (2 bytes) mapping to a
  peer address.
- Tailscale DERP: Each client registers with a DERP server. The
  server maintains a map of (public_key -> connection). Forwarding
  is (public_key -> connection) lookup + send.
- libp2p Circuit Relay v2: Relay maintains a reservation table.
  Peers make reservations, then traffic is forwarded based on
  peer ID.

All three systems have the same structure: registration phase
(client tells relay "I am here"), then forwarding phase (relay
looks up destination and sends).

**(d) Recommended approach.**

Add a "Transit Forwarding" body section to verkko-relay with:

1. **Forwarding table** data structure (as specified in Section 1.5
   of this review).
2. **Per-frame forwarding procedure** (as specified in Section 1.5).
3. **Session establishment via SIGNAL HELLO** (as specified in
   Section 1.6).
4. **Forwarding table entry lifecycle:** created on SIGNAL HELLO,
   refreshed on activity, expired after 5 minutes of inactivity,
   maximum 256 entries.
5. **Accounting:** bytes_forwarded and frames_forwarded per entry,
   reported in relay health metrics.

Wire cost: zero additional bytes. The forwarding uses the existing
frame format unchanged (with the hop_count AEAD fix from Problem 1).

CPU cost per forwarded frame: ~300ns (hash lookup + memcpy + byte
decrement). At 10,000 frames/second: 3ms/second, 0.3% of one
ARM Cortex-A72 core.

Memory cost: 256 entries * ~100 bytes = ~25 KB.

### Problem 3: NAT traversal has no body text (CRITICAL GAP)

**(a) Problem statement.**

The NATTraversal defined concept in verkko-protocol has three
invariants:
1. Relay-first fallback if direct connection fails.
2. Hole punching attempted before relay fallback.
3. OBSERVED_ADDR in handshake response for address discovery.

The body text mentions "adaptive keepalive: discover mapping lifetime
via probes, set interval to 70% of lifetime. Prewarm relay path when
lifetime < 60 seconds." This is the entirety of the NAT traversal
specification. There is no candidate gathering procedure, no
connectivity check procedure, no NAT type classification, no
state machine, and no interaction with the SIGNAL PUNCH operation.

This gap has been flagged as "critical gap since Run 10, still
unspecified" in the Run 12 task list. It has been deferred through
two full review rounds.

**(b) Why it matters.**

NAT traversal is the gateway to the entire relay subsystem. Without
NAT traversal, a peer behind a NAT cannot determine that it needs
a relay. Without that determination, the SIGNAL operations are never
invoked. Without SIGNAL, the forwarding table is never populated.
Without the forwarding table, no frames are forwarded. The entire
relay transit path is gated on NAT traversal working.

Furthermore, the PeerDiscovery fallback chain (mDNS -> invite hint
-> cached peers -> DHT -> relay) assumes that at least one of these
mechanisms produces a reachable address. For a peer behind a
symmetric NAT, none of the non-relay mechanisms will work. The
peer's only path to the mesh is through a relay, and the relay
path requires NAT traversal to discover the peer's external address
and establish a relay session.

Quantitatively: at 40% symmetric NAT prevalence, 40% of new
device enrollments will fail unless NAT traversal is specified.
The invite token contains address_hints (AddressHint format,
20 bytes each) but these are the inviter's addresses, not the
invitee's. If both are behind symmetric NATs, the address hints
are unreachable.

**(c) References and prior art.**

- RFC 8445 (ICE) Sections 5-8: Complete NAT traversal framework.
  Candidate gathering, connectivity checks, nominating, keepalive.
  Over-specified for verkko (ICE is designed for 1:1 media sessions,
  not mesh topologies), but the core structure applies.
- RFC 8656 (TURN) Sections 3-5: Relay allocation, permissions,
  channels. The relay-as-fallback model.
- RFC 5389 (STUN) Sections 7-10: Binding requests for address
  discovery. verkko replaces STUN with OBSERVED_ADDR, which is
  simpler and correct.
- Tailscale DERP + ICE-lite: Tailscale uses a simplified ICE-like
  procedure. Clients always connect to a DERP relay first (relay-
  first, not relay-fallback), then attempt direct connections via
  hole punching in the background. If hole punching succeeds,
  traffic migrates from relay to direct. This is the inverse of
  verkko's stated approach ("hole punching attempted before relay
  fallback") but achieves lower initial latency.
- libp2p AutoRelay: Relay discovery via DHT, automatic relay
  selection, transparent relay-to-direct upgrade.

**(d) Recommended approach.**

Add a "NAT Traversal" body section to verkko-protocol with the
state machine specified in Section 1.7 of this review. Key design
decisions:

1. **Relay-first, upgrade-to-direct.** I recommend changing the
   stated approach from "hole punching attempted before relay
   fallback" to "relay-first, then attempt hole punching in
   parallel." Rationale: relay-first guarantees connectivity within
   ~500ms (one RTT to relay). Hole punching can take 2-10 seconds
   and may fail entirely. For a household mesh where users expect
   instant connectivity, relay-first is correct.

   The strongest counterargument is that relay-first increases relay
   load. But relay forwarding is cheap (300ns/frame), the relay
   load is bounded by the mesh size (max 66 session pairs), and the
   relay is upgraded to direct within seconds if hole punching
   succeeds. The transient relay load is negligible.

2. **OBSERVED_ADDR for address discovery.** The existing handshake
   approach is correct and eliminates the need for a separate STUN
   server.

3. **Connectivity checks via PATH_CHALLENGE/PATH_RESPONSE.** The
   existing wire format has these message types. Use them for NAT
   traversal connectivity checks, not just connection migration.

4. **NAT type classification.** Based on OBSERVED_ADDR responses
   from multiple peers:
   - Same external port across all peers: endpoint-independent
     mapping (hole punch likely to succeed).
   - Different external port per peer: symmetric NAT (hole punch
     unlikely; maintain relay path).

5. **Integration with PeerDiscovery fallback chain.** When the
   DHT step returns a peer's relay_dht_key, the discovering peer
   contacts that relay and initiates SIGNAL PUNCH. This is the
   bridge between PeerDiscovery and NATTraversal.

### Problem 4: Relay failover has no specification (D-008)

**(a) Problem statement.**

If the relay peer goes down, all sessions forwarded through it are
immediately interrupted. There is no mechanism for detecting relay
failure, selecting a backup relay, or migrating forwarding table
entries.

The deferred item D-008 says: "Relay dies, client discovers backup
relay, migration. Important for robustness, not for first
implementation." I disagree that this is not for first
implementation.

**(b) Why it matters.**

A relay peer is a regular household device (Pi, NAS, NUC). These
devices reboot for updates, lose power during outages, and crash
due to SD card corruption (the design even accounts for eMMC torn
writes). If the relay peer reboots, every session it was relaying
is dead. The peers behind symmetric NATs cannot communicate at all
until a new relay is established.

Failover latency without a specification: the heartbeat timeout
(3 missed heartbeats = 15 seconds at Active, 90 seconds at
Background) detects the relay's failure. Then the peer must
discover a new relay (DHT lookup: 2-10 seconds), establish a
session (Noise KK: 1 RTT, ~200ms), and send SIGNAL HELLO (~100ms).
Total: 17-100 seconds of outage.

With a specification, failover can be reduced to 2-5 seconds by
maintaining a warm standby relay.

**(c) References and prior art.**

- RFC 8445 (ICE) Section 12: ICE restarts. When a connectivity
  check fails, ICE restarts the check from the beginning.
- Tailscale DERP: Clients maintain connections to multiple DERP
  servers simultaneously. If one fails, traffic immediately shifts
  to the next. Failover is sub-second.
- TURN: RFC 8656 Section 7: Allocation refresh. If refresh fails,
  the allocation expires. Client must create a new allocation on a
  different TURN server. No built-in failover.

**(d) Recommended approach.**

1. **Warm standby relay.** When a peer registers with a relay via
   SIGNAL HELLO, it also registers with a secondary relay. The
   secondary relay is the next-best candidate from the relay
   selection algorithm (Section 1.9). The secondary relay holds
   a dormant forwarding entry (no active forwarding, just registered).

2. **Relay health monitoring.** The relay forwarding path piggybacks
   on the heartbeat mechanism. The relay peer is a mesh peer; if its
   heartbeats stop, the standard heartbeat timeout fires.

3. **Failover procedure:**
   ```
   on relay_heartbeat_timeout(relay_peer):
       // 1. Promote secondary relay to primary.
       secondary_relay.activate_forwarding_entries()

       // 2. Notify the remote endpoint via the secondary relay.
       send_signal_redirect(remote_peer, secondary_relay.addr)

       // 3. Select a new secondary relay.
       new_secondary = select_relay(self, target, exclude=[relay_peer])

       // 4. Register with new secondary.
       send_signal_hello(new_secondary)
   ```

4. **Failover timing.** With warm standby: 1 heartbeat timeout
   (2-30 seconds depending on metabolic state) + 1 RTT for
   redirect (~200ms). Total: 2.2-30.2 seconds. The worst case
   (30 seconds in Background mode) is acceptable for a household
   mesh where Background mode means nobody is actively using the
   system.

   To reduce failover to <5 seconds in all cases, the relay health
   check can use a shorter interval than the regular heartbeat:
   send relay-specific keepalive probes every 2 seconds regardless
   of metabolic state. Cost: 2 * 91 bytes/s = 182 bytes/s per
   relayed session. At 10 sessions: 1,820 bytes/s, ~14.6 kbps.
   Negligible.

5. **Wire cost.** SIGNAL HELLO to secondary relay: 10 bytes +
   13 bytes header = 23 bytes, once per registration. Negligible.

### Problem 5: Seat failover when host peer dies (D-009)

**(a) Problem statement.**

A seat connects to one host peer via WebSocket (SeatTransport
defined concept, R11-024). If that host peer dies, the seat is
disconnected. There is no mechanism for the seat to discover and
reconnect to an alternate host peer.

The deferred item D-009 says: "Host peer dies; seat reconnects to
alternate cluster peer. Simple retry mechanism." This understates
the problem.

**(b) Why it matters.**

The seat is the user's interface to the mesh. When the host peer
dies, the user sees a blank screen. The user does not know why,
does not know which other peers exist, and does not know how to
reconnect.

The seat does not participate in the mesh. It has no gossip, no
health pipeline, no peer discovery. It knows about exactly one
peer: its host. When that host dies, the seat has no mechanism to
find another host.

The seat connects via WebSocket on LAN only. If the user is on
the same LAN as multiple cluster peers, the seat needs a way to
discover them. If the user is on a different LAN (e.g., a phone
connected to the mesh via VPN), the seat may have no discovery
mechanism at all.

**(c) References and prior art.**

- Web applications with backend failover: Load balancers (HAProxy,
  nginx) handle this transparently. verkko has no load balancer.
- gRPC client-side load balancing: The client maintains a list of
  backends and retries on failure. This is the model verkko should
  adopt.
- MQTT (IoT protocol): Clients maintain a list of brokers and
  reconnect to the next one on disconnection.

**(d) Recommended approach.**

1. **Host list in SeatAuthResult.** When the seat authenticates,
   the host peer returns a list of alternate host peers in the
   cluster:

   ```
   SeatAuthResult {
       // ... existing fields ...
       alternate_hosts: [AlternateHost; count],
   }

   AlternateHost {
       address: [u8; 20],   // AddressHint format
       host_id: u32,        // for session resumption targeting
   }
   ```

   Cost: 24 bytes per alternate host. At 5 peers in a cluster:
   120 bytes, once per session establishment.

2. **Reconnection procedure.**
   ```
   on websocket_close_unexpected():
       for host in [last_known_host] ++ alternate_hosts:
           try websocket_connect(host.address)
           if success:
               try seat_auth(host, resumption_token)
               if success:
                   return  // reconnected
       // All hosts exhausted. Show "mesh unavailable" to user.
       // Retry with exponential backoff: 1s, 2s, 4s, 8s, max 30s.
   ```

3. **mDNS discovery for seats.** Seats can discover host peers
   via mDNS on the LAN. The host peer advertises a mDNS service:
   `_verkko-seat._tcp.local.` with TXT record containing the
   founding_hash. This gives the seat a discovery mechanism
   independent of its host peer.

4. **Failover timing.** WebSocket close detection: 1-5 seconds
   (TCP keepalive or WebSocket ping/pong). Reconnection to
   alternate host: 1 RTT (~10ms on LAN). Total: 1-6 seconds.

   Outside my domain -- the consistency model during failover
   (SNAPSHOT/CONSISTENT mode transitions, deferred write queue
   handling) is not my area.

### Problem 6: Multi-hop relay path selection (D-007)

**(a) Problem statement.**

hop_count max is 4, which implies multi-hop relay paths are expected.
But there is no forwarding table that maps destinations to next-hop
relays, no path selection algorithm, and no mechanism for a relay to
discover which destinations are reachable through which other relays.

**(b) Why it matters.**

In the verkko deployment envelope (2-12 clusters), most relay paths
are 1-hop (direct relay). Multi-hop relay paths arise when:
- Cluster A can reach Relay R1 but not Relay R2.
- Cluster B can reach Relay R2 but not Relay R1.
- R1 and R2 can reach each other.
- The path is A -> R1 -> R2 -> B (2 hops).

This scenario is uncommon in household meshes but possible when
clusters span different ISPs with different CGNAT policies.

**(c) References and prior art.**

- Tor: Circuit construction via onion routing. Relays do not know
  the full path; they know only the previous and next hop. Three
  hops standard.
- libp2p: Multi-hop relay via chained circuit reservations. Each
  relay in the chain must independently accept the reservation.
- I2P: Garlic routing through tunnels of 2-3 hops.

For verkko's scale (max 12 clusters, max 4 hops), the routing
problem is trivially small. No BGP, no distance-vector, no
link-state. A simple next-hop table suffices.

**(d) Recommended approach.**

1. **Next-hop table.** Each relay maintains a next-hop table
   populated from the mesh's gossip layer. When a peer registers
   via SIGNAL HELLO, the relay learns that `destination X is
   reachable via address Y`. The relay gossips this reachability
   information to other relays.

2. **Path construction.** The initiating peer determines the relay
   path. It knows the mesh topology from gossip. It selects a path
   with the fewest hops and highest minimum health score.

3. **Forwarding at each hop.** Each relay in the path decrements
   hop_count and forwards to the next relay in its forwarding table.
   No circuit setup needed. Stateless per-frame forwarding.

4. **Loop prevention.** hop_count decrement + max 4 prevents
   infinite loops. A relay that receives a frame with hop_count=0
   drops it silently. A relay that receives a frame addressed to
   itself processes it locally (not forwarded).

For v1, I recommend documenting this as "v1 supports 1-hop relay
only; hop_count > 1 is reserved for future multi-hop extension"
rather than specifying the full multi-hop protocol. The added
complexity of multi-hop is not justified for the deployment envelope.

## Part 3: Open Items and Known Limitations

### Open items within my domain

1. **OBSERVED_ADDR payload position.** The Noise handshake
   application payload is mentioned but not fully specified. The
   Run 11 handshake framing (R11-035) does not include OBSERVED_ADDR
   in the handshake payload. This needs to be added: 20 bytes in the
   post-handshake application payload.

2. **Relay-to-relay authentication.** When two relays forward
   traffic between each other (multi-hop), they should have mutual
   Noise KK sessions. This is already guaranteed if both relays are
   mesh peers (they have CPSKs). But the forwarding table needs to
   distinguish "direct peer" from "relay-via-peer" entries.

3. **IPv6 interaction.** The design uses AddressHint with addr_type
   0x04 (IPv4) or 0x06 (IPv6). On IPv6-only networks, NAT traversal
   is largely unnecessary (IPv6 addresses are globally routable).
   However, IPv6 firewalls may still block incoming connections.
   The NAT traversal state machine should handle this case by using
   PATH_CHALLENGE/PATH_RESPONSE to verify reachability even on
   IPv6.

4. **Relay on SD card devices.** The design disables donation on
   SD cards (Section 4.4). Should relaying also be disabled on SD
   card devices? The forwarding table is in-memory (25 KB), not on
   flash. Frame forwarding does not write to disk. Relay forwarding
   is safe on SD card devices. Only membrane operations (PUT) write
   to disk and should respect the SD card limitation.

5. **SIGNAL operation wire format detail.** The SIGNAL subtypes
   (HELLO=0x0001, PUNCH=0x0002, REDIRECT=0x0003) need full wire
   format specifications added to the relay operations wire format
   (R11-040). I have provided the payloads in Section 1.6 but they
   need formal offset/width/field tables.

### Known limitations of my recommendations

1. **Relay-first vs hole-punch-first.** I recommend relay-first.
   The design currently specifies hole-punch-first. This is a
   design philosophy question, not a technical one. Both work. My
   recommendation optimizes for user-perceived latency at the cost
   of higher relay load during the first seconds of a session.

2. **Warm standby relay cost.** Maintaining a secondary relay
   registration doubles the SIGNAL HELLO traffic (from 1 to 2
   messages per relay registration cycle). At 23 bytes per message
   and one registration per 5-minute TTL: 0.15 bytes/second.
   Negligible. But the secondary relay holds a dormant forwarding
   entry, consuming one of its 256 slots.

3. **NAT type classification accuracy.** Using OBSERVED_ADDR from
   multiple peers to classify NAT type is a heuristic. Some NATs
   behave differently depending on the destination. A full STUN
   implementation (RFC 5389) would be more accurate. For verkko's
   deployment envelope, the heuristic is sufficient.

4. **Relay selection without full mesh visibility.** A peer behind
   a symmetric NAT may not have direct connectivity to enough peers
   to run the relay selection algorithm. It may need to rely on the
   invite token's address_hints and relay_dht_key to find its first
   relay, then use that relay to discover the rest of the mesh.

## Part 4: Ideal Design Synthesis

### The complete relay transit path

The ideal design assembles the relay transit path as a five-phase
pipeline:

**Phase 1: Discovery.** Peer discovers the mesh via PeerDiscovery
fallback chain. If no direct path works, the peer uses relay_dht_key
from the invite token to find a relay peer via Mainline DHT.

**Phase 2: NAT Classification.** Peer establishes a Noise session
with the relay peer (or any reachable mesh peer). The OBSERVED_ADDR
in the handshake response reveals the peer's external address. The
peer compares this with its local address. If different, the peer is
behind a NAT. Multiple OBSERVED_ADDR responses from different peers
classify the NAT type (endpoint-independent vs symmetric).

**Phase 3: Relay Registration.** Peer sends SIGNAL HELLO to the
relay, registering its cluster_destination_id. The relay creates a
forwarding table entry. Peer also sends SIGNAL HELLO to a secondary
relay for warm standby.

**Phase 4: Connectivity Checks.** When the peer needs to communicate
with a new cluster, it initiates connectivity checks:
- Exchange candidates via SIGNAL PUNCH through the relay.
- Test each candidate pair with PATH_CHALLENGE/PATH_RESPONSE.
- If a direct path succeeds, use it. Traffic migrates from relay.
- If no direct path succeeds, continue using relay.

**Phase 5: Steady-State Forwarding.** Frames flow through the relay.
The relay decrements hop_count and forwards based on its routing
table. ~300ns per frame. No AEAD, no decryption, no inspection of
encrypted payload.

**Failover.** If the relay peer's heartbeat stops, the peer promotes
its secondary relay and sends SIGNAL REDIRECT to the remote endpoint.
Failover latency: 2-30 seconds depending on metabolic state.

### Integration with existing design

This synthesis requires changes to three documents:

1. **verkko-protocol:** Add NAT traversal body text (state machine,
   candidate gathering, connectivity checks, OBSERVED_ADDR payload
   specification). Modify AEAD AD specification to exclude
   hop_count.

2. **verkko-relay:** Add transit forwarding body text (forwarding
   table, per-frame procedure, session establishment via SIGNAL).
   Add SIGNAL subtype wire formats. Add relay resource governance
   (forwarding table limits, bandwidth limits, amplification
   prevention). Add relay failover specification.

3. **verkko-seat:** Add host list in SeatAuthResult for seat
   failover. Add mDNS discovery for seats.

No new DAG edges required. verkko-relay already depends on
verkko-protocol (for Frame, Connection, NATTraversal). The
additions are body text for existing defined concepts.

### Concrete numbers summary

| Metric | Value |
|--------|-------|
| Forwarding CPU per frame | ~300ns |
| Forwarding table memory | ~25 KB (256 entries) |
| Relay forwarding bandwidth | 50% of relay's available BW |
| NAT traversal latency (relay-first) | ~500ms |
| NAT traversal latency (hole punch) | 200ms-2s |
| Relay failover latency (warm standby) | 2-30s |
| Seat failover latency (LAN) | 1-6s |
| SIGNAL HELLO payload | 10 bytes |
| SIGNAL PUNCH payload (4 candidates) | 104 bytes |
| SIGNAL REDIRECT payload | 28 bytes |
| Per-session relay keepalive cost | 182 bytes/s |
| Multi-hop max latency (4 hops, 200ms RTT) | ~800ms |

- Membrane model and re-encryption: well-specified, no comment.
- CRDT convergence and GSet semantics: outside my domain.
- Health pipeline, dominance cascade, scar system: outside my domain.
- Key management (KeyBundle, epoch ratchet, PCS): outside my domain.
- Gossip protocol (Plumtree/flood): outside my domain.
