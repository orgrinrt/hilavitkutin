# Run 10 Stage 2a: Devil's Advocate

**Reviewer:** Devil's Advocate (Gap Hunter perspective -- implementer's lens)
**Date:** 2026-03-28
**Input:** All four Stage 1 reviews + tension map

---

## Mandate

The Stage 1 reviewers attacked what was visible: missing wire formats, coupling cycles, empty cells, grain boundaries. They found real problems. But they all share a deeper agreement -- an agreement so complete that none of them examined it. They all accept the decomposition's foundational premises. I attack those premises.

---

## The Three Unquestioned Assumptions

### Assumption 1: "Document = Crate = Compilation Unit"

Every reviewer treats the seven verkko documents as though they will become seven distinct compilation units (crates, libraries, modules) with well-defined API boundaries. DepAnalyst's entire analysis rests on this: the topological sort, the dependency matrix, the interface sufficiency assessment -- all assume that document boundaries become crate boundaries. Savant's crystallographic model reinforces it: each document is a "unit cell" with "bonds" at the API surface. GapHunter implicitly assumes it when rating wire format gaps as BLOCKING per-document ("an implementor of verkko-crypto alone cannot validate an enrollment certificate"). SpecAuditor's per-document finding structure assumes independent implementability.

Nobody asked: will these seven documents actually become seven crates? The taxonomy says they will. But the taxonomy is a decomposition guide, not a build plan. The verkko-ops build order (Phase 1-10) describes implementation phases, not crate boundaries. No document contains a `Cargo.toml` specification. No document specifies pub API surface vs internal items. The defined concepts describe protocol semantics, not Rust module structure.

**Why nobody questioned it.** The decomposition looks so clean on paper that the document-to-crate mapping feels obvious. Seven documents, seven crates, seven API boundaries. It is the kind of 1:1 mapping that architects love and implementers discover does not work.

### Assumption 2: "Defined Concepts Are Sufficient Interface Contracts"

All four reviewers treat defined concepts (name + invariants) as the interface mechanism between documents. DepAnalyst grades interface sufficiency based on whether a document uses only defined concepts from its dependencies. Savant analyzes "bond strength" by whether a consumer uses "only invariants" vs "some internals." SpecAuditor evaluates completeness by whether each defined concept has enough specification for independent implementation.

Nobody asked: are defined concepts the right granularity for interface contracts? A defined concept is a prose description with a list of invariants. It is not a function signature, a trait definition, a wire format, or a type. When DepAnalyst says "verkko-protocol uses FrameKey, TransportAEAD, NonceSafety from verkko-crypto -- interface sufficiency: SUFFICIENT," that assessment means the prose invariants of these concepts are respected. It does not mean there is an actual API boundary that enforces this sufficiency. Two implementation teams could both respect the prose invariants and still produce incompatible code because the prose does not constrain the API shape.

**Why nobody questioned it.** The defined concept pattern is consistent and well-executed. All six documents that define concepts use the same format. The invariants are specific and testable. The reviewers evaluated the pattern on its own terms and found it good. None stepped outside the pattern to ask whether prose invariants translate to enforceable API contracts.

### Assumption 3: "The Health Pipeline Is a Closed System"

The HealthPipeline (defined in verkko-mesh) is the single mechanism that converts raw observations into health scores, which then feed into source selection (matter), relay selection (relay), gateway election (mesh), metabolic state transitions (protocol via Conductor), and convergence gating. Every reviewer touched HealthPipeline. Savant noted it as a strong bond. DepAnalyst marked it as a clean interface. GapHunter flagged the missing source selection threshold (a parameter gap). SpecAuditor flagged the missing PI controller tuning rationale.

Nobody questioned whether the HealthPipeline's inputs are sufficient, or whether its single EMA-based model will survive contact with real network conditions. The pipeline takes heartbeat delivery ratios and SACK statistics as input and produces a scalar health score per peer. Every consumer assumes this scalar is meaningful.

**Why nobody questioned it.** The pipeline is internally consistent. The dual-EMA (fast + slow), the observation weighting, the scar system, the FlowSummary aggregation -- it all fits together. The math is clean. The invariants are specific. But internal consistency is not the same as external validity. No reviewer asked: does a single scalar health score contain enough information for all five consumers to make correct decisions?

---

## The Most Fragile Assumption: Assumption 2

Assumption 2 ("Defined Concepts Are Sufficient Interface Contracts") is the most fragile because it is the one that fails at the exact moment implementation begins, and its failure cascades through every document boundary simultaneously.

Assumptions 1 and 3 are also fragile, but they fail locally. If the seven documents do not map to seven crates (Assumption 1), the fix is reorganization -- painful but bounded. If the health pipeline proves inadequate (Assumption 3), the fix is adding dimensions to the health model -- complex but localized to mesh and its consumers. But if defined concepts are not sufficient interface contracts (Assumption 2), then every document boundary is simultaneously under-specified, and the 59 gaps that GapHunter found are not individual problems but symptoms of a systemic one.

## Failure Scenario: Two Teams, One Spec, Zero Interop

### Setup

Two implementation teams (Team A and Team B) each receive the seven verkko design documents. They implement verkko-crypto and verkko-protocol independently, respecting all defined concept invariants. They do not share code or discuss implementation. After six months, they attempt to connect their implementations.

### What happens

**Week 1: EpochRatchet divergence.**

Team A implements `K_epoch_N = HKDF-Expand(prk=K_epoch_{N-1}, info="epoch:advance", len=32)` using the `hkdf` crate's `Hkdf::new(None, previous_key)` then `expand("epoch:advance", &mut output)`. This calls HKDF-Extract on the previous key before Expand, because the `hkdf` crate's `new()` always runs Extract.

Team B reads the spec literally ("HKDF-Expand") and uses raw HMAC-SHA256 expansion without Extract, since the spec says "HKDF-Expand" not "HKDF" and RFC 5869 distinguishes the two.

Both implementations satisfy the EpochRatchet invariant: "Forward-only: K_epoch_N cannot be derived from K_epoch_{N+1}." Both are deterministic. Both are forward-secure. But they produce different epoch keys from the same starting material.

Result: silent mesh partition. Every frame encrypted by Team A is undecryptable by Team B. The health pipeline classifies Team B peers as 100% packet loss. After 3 heartbeat intervals, Team B peers are marked Dead. From Team A's perspective, Team B's cluster does not exist.

**Estimate:** The EpochRatchet is advanced once per epoch (configurable, default seems to be on the order of hours based on heartbeat intervals and ratchet scope). Within the first epoch advance after connection -- potentially within the first hour of operation -- the two implementations diverge irreversibly.

**Week 2: FoundingHash mismatch.**

Even before the epoch ratchet diverges, the founding hash will differ. The spec says `LEN(DK_pub) || DK_pub || ...` but does not specify LEN encoding (SpecAuditor C13). Team A uses u32 little-endian. Team B uses u16 big-endian. Both are reasonable choices for a length prefix. The founding hashes differ. The peers cannot join each other's meshes because the founding hash is checked during invite flow.

Result: the two implementations cannot form a mesh together at all. The failure is immediate, not gradual.

**Estimate:** ~20 such ambiguity points exist across crypto and protocol alone (SpecAuditor counts 15 findings in crypto, 18 in protocol; GapHunter counts 10 and 13 respectively; overlap is substantial but at least 20 unique ambiguity points remain). Each one is a potential interop-breaking divergence. The probability that two independent implementations agree on all 20 points by chance is approximately 0 (even if each point has only 2 possible interpretations, 2^20 = 1,048,576 possible combinations, with only 1 being compatible).

**Week 4: Noise handshake incompatibility.**

Both teams implement Noise XX for first contact. Team A chooses `Noise_XX_25519_ChaChaPoly_BLAKE2b`. Team B chooses `Noise_XX_25519_ChaChaPoly_SHA256`. Both are valid Noise protocol names. The spec says "Noise XX" without specifying the hash function. SpecAuditor P1 flagged this. GapHunter P-3 flagged this. Both recommended fixing it. But the deeper problem is not this specific gap -- it is that the defined concept "Connection" does not constrain the Noise protocol name because defined concepts operate at the semantic level ("authenticated, encrypted channel") not the wire level ("these exact bytes in this exact order").

**Month 3: The realization.**

The teams discover that even after fixing the 20+ explicit ambiguity points, they still cannot interoperate because their internal APIs are incompatible. Team A's `EpochRatchet` is a struct with methods. Team B's is a free function. Team A's `Frame` struct contains a `nonce` field as `[u8; 12]`. Team B's uses `u96`. Team A returns `Result<DecryptedFrame, CryptoError>`. Team B returns `Option<Vec<u8>>`. None of these differences violate any defined concept invariant. All of them prevent code reuse across the boundary.

**The cost.** Six months of parallel development. Two complete implementations of crypto and protocol. Zero interoperability. The fix requires writing what amounts to an RFC: a byte-level wire specification that both teams implement. This is approximately 3,000-5,000 lines of specification that does not currently exist (GapHunter's 28 BLOCKING gaps, each requiring 50-200 lines of wire-level spec).

## Why This Failure Is Not Hypothetical

The verkko design is explicitly a protocol specification. Protocol specifications that lack byte-level wire formats fail the two-team test every time. This is not a theoretical concern; it is the defining lesson of 40 years of protocol standardization:

- TLS 1.0 required three years of interop testing after the spec was "complete."
- HTTP/2 required a reference implementation (h2o) before the spec was considered implementable.
- The Noise Protocol Framework works precisely because it specifies exact byte sequences, not just security properties.

The verkko spec is currently at the "security properties and algorithmic intent" level. It is not at the "byte sequence" level. The defined concepts are the security properties. The missing wire formats are the byte sequences. The gap between these two levels is where interop dies.

---

## Minimum Change to Survive This Assumption Failing

The assumption that defined concepts are sufficient interface contracts will fail. The question is whether the design can survive that failure without a ground-up rewrite.

### The change: add a Wire Contract layer beneath each defined concept

For every defined concept that crosses a document boundary (approximately 40 concepts across the seven documents), add a mandatory subsection:

```
Wire Contract:
  - Canonical byte layout: [field: type, offset, size]*
  - Hash stability: BLAKE3(canonical_bytes) is deterministic
  - Test vector: input -> expected canonical_bytes -> expected BLAKE3
```

This is not a new document. It is an addition to the existing defined concept format. Each concept gains 10-30 lines of wire-level specification. Total addition: approximately 800-1,200 lines across all seven documents.

### Why this is the minimum

1. It does not change the document structure. The seven-document decomposition survives.
2. It does not change the defined concept pattern. Invariants remain. Wire contracts are added alongside them.
3. It does not require resolving any of the tensions identified in Stage 1.5. The coupling cycles, MetabolicState ownership, and empty cells are orthogonal to wire contracts.
4. It provides the single thing that makes two-team implementation possible: deterministic byte layouts that can be tested with vectors.

### What it does not fix

Wire contracts do not fix the empty cells (seat, matter). Those documents need body content regardless. Wire contracts do not fix the coupling cycles (protocol-mesh, crypto-mesh). Those need architectural decisions. But wire contracts prevent the scenario where two complete, invariant-respecting implementations cannot talk to each other.

### Cost estimate

Adding wire contracts to 40 defined concepts at 20 lines each = 800 lines. Creating test vectors requires a reference implementation of each concept's canonical encoding = approximately 2,000 lines of reference code (Rust or pseudocode). Total: ~3,000 lines of specification work. At the current document density (~3,200 lines across 7 documents), this is roughly a 25% increase in specification volume. This is substantial but bounded.

## Secondary Fragilities

### Assumption 1 failure mode: the "seven crates" illusion

If the seven documents do not map to seven crates, the most likely failure is that crypto and protocol merge into one crate (they are tightly coupled at the Noise handshake level), mesh absorbs relay (convergence is bidirectionally specified), and matter absorbs seat (the user-facing content workflow requires both). This produces three crates instead of seven: `verkko-crypto-protocol`, `verkko-mesh-relay`, `verkko-matter-seat`. The ops document becomes build documentation, not a crate.

The three-crate outcome would invalidate DepAnalyst's entire topological sort and interface sufficiency analysis. It would also invalidate Savant's grain boundary analysis (G1 and G2 become internal boundaries, not cross-crate interfaces). The remediation cost is moderate: restructure documentation to match the three-crate reality, consolidating defined concepts that straddle the old boundaries.

**Estimate:** The probability of the seven-crate decomposition surviving implementation intact is low. Based on the Stage 1 findings alone: 4 of 6 pairwise document boundaries have implicit coupling edges (protocol-mesh, crypto-mesh, mesh-relay, mesh-matter). Only 2 boundaries are clean (seat-crypto, seat-protocol). A decomposition where 67% of boundaries have coupling defects typically consolidates during implementation.

### Assumption 3 failure mode: the health scalar bottleneck

The HealthPipeline produces `effective_health = min(fast_ema, slow_ema)`, a scalar in [0, 1]. This scalar is consumed by:

1. Source selection (matter): needs to know "can this peer deliver content reliably?"
2. Relay selection (relay): needs to know "can this peer forward frames without dropping?"
3. Gateway election (mesh): needs to know "is this peer stable and well-connected?"
4. Metabolic state (protocol): needs to know "is the system under resource pressure?"
5. Convergence gating (mesh): needs to know "are all peers healthy enough to proceed?"

These five consumers ask five different questions. A single scalar cannot answer all of them correctly. Consider: a peer with 99% uptime but 10% packet loss has `effective_health ~ 0.90` (Degraded). For source selection, this peer is unreliable (high retransmit cost). For gateway election, this peer is acceptable (uptime is high). For metabolic state, this peer's condition does not indicate resource pressure (it is a network issue, not a CPU/flash issue).

The failure scenario: during convergence, the gateway election selects the peer with the highest health score. This peer happens to have high uptime but high packet loss. Convergence requires reliable delivery of control log entries (Front 1). The gateway's high packet loss causes control log merge to take 3x longer than expected. The wavefront deadline expires. The mesh enters a prolonged convergence state that exhausts the background metabolic budget.

**Estimate:** In a 12-cluster mesh (the design envelope), with 6 devices per cluster, there are 72 devices and 66 inter-cluster links. The probability of at least one peer having asymmetric health properties (high uptime, poor throughput; or low uptime, excellent throughput when online) is high. The health scalar will produce suboptimal decisions for at least one consumer on at least one peer in any realistic deployment.

## Summary

| Rank | Assumption | Fragility | Failure Cost | Fix Cost |
|------|-----------|-----------|-------------|----------|
| 1 | Defined concepts are sufficient interface contracts | High -- fails on day 1 of two-team implementation | 6 months of wasted parallel development | ~3,000 lines of wire contracts + test vectors |
| 2 | Seven documents map to seven crates | Medium -- fails when implementation discovers coupling | Restructuring work proportional to coupling surface | Moderate: consolidate 7 docs to ~3-4 |
| 3 | Health pipeline scalar is sufficient for all consumers | Medium -- fails under asymmetric network conditions | Suboptimal decisions during convergence, prolonged recovery | Add health dimensions (throughput, latency, uptime as separate signals) |

The decomposition is not unsound. Its architecture is thoughtful, its cryptographic design is careful, and its convergence model is well-reasoned. But it is standing on the assumption that prose invariants are interchangeable with wire specifications. They are not. Every protocol that has tried this has learned the lesson the hard way. The minimum intervention is to add wire contracts before any implementation begins.
