## Part 1: UDP Payload Size Correction

### 1.1 The Error

My Stage 1 used 1280 bytes as the UDP payload limit. This is wrong. 1280 bytes is the IPv6 minimum MTU at the IP layer. The UDP payload limit is:

```
IPv6 minimum MTU:           1280 bytes
  - IPv6 header:              40 bytes
  - UDP header:                8 bytes
  = Maximum UDP payload:    1232 bytes
```

The SpecWriter caught this. The Savant traced the ripple effects. I confirm: every frame size calculation in my Stage 1 is 48 bytes too generous.

### 1.2 Corrected Frame Budget

The verkko data frame structure, from outermost to innermost:

```
UDP payload (max 1232 bytes):
  Outer header (plaintext):   29 bytes
  Encrypted payload:
    Inner header:             10 bytes
    Stream header:             7 bytes  (accepted SpecWriter's flags byte)
    Payload:                  var
    AEAD auth tag:            16 bytes

Max payload = 1232 - 29 - 10 - 7 - 16 = 1170 bytes
```

Let me verify the outer header (29 bytes) field by field:

```
Outer Header (29 bytes, plaintext):
  Offset  Width  Field
  ------  -----  -----
   0       4     magic: 0x56524B4F ("VRKO")
   4       1     frame_type: u8
   5       4     connection_id: u32
   9       4     frame_number: u32
  13       8     epoch_id: u64
  21       1     hop_count: u8
  22       1     priority: u8
  23       2     payload_len: u16
  25       4     header_crc: u32 (CRC-32C of bytes 0..25)

  Total: 4 + 1 + 4 + 4 + 8 + 1 + 1 + 2 + 4 = 29 bytes  [verified]
```

Inner header (10 bytes, inside encrypted payload):

```
Inner Header (10 bytes, encrypted):
  Offset  Width  Field
  ------  -----  -----
   0       1     msg_type: u8 (bit 7 = SACK piggybacked)
   1       1     flags: u8
   2       2     dest_peer: u16
   4       4     hlc_timestamp: u32 (truncated HLC)
   8       2     payload_offset: u16

```

Stream header (7 bytes, accepted from SpecWriter):

```
Stream Header (7 bytes):
  Offset  Width  Field
  ------  -----  -----
   0       2     stream_id: u16

   6       1     flags: u8 (bit 0 = FIN, bit 1 = RST, bit 2 = SACK_PRESENT)
```

## Part 2: BLAKE2b Concession -- Updated Noise Protocol Names

### 2.1 The Change

I conceded BLAKE2b in Stage 2a. The Advocate's argument was decisive: verkko targets 64-bit hardware (Pi 4/5, x86_64 NUCs), BLAKE2b is faster on 64-bit, and the choice is effectively irreversible after v1. All Noise protocol name strings are updated:

**Before (Stage 1):**
```
Noise_KK_25519_ChaChaPoly_BLAKE2s
Noise_XX_25519_ChaChaPoly_BLAKE2s
```

**After (Stage 3a):**
```
Noise_KK_25519_ChaChaPoly_BLAKE2b
Noise_XX_25519_ChaChaPoly_BLAKE2b
```

### 2.2 Affected Wire Artifacts

The Noise protocol name string is hashed into the handshake state during `Initialize()`. Changing the hash function changes the protocol name, which changes the initial chaining key, which changes every subsequent handshake key. This is a complete handshake state change.

Affected items:

1. **Prologue computation.** The prologue itself does not contain the Noise protocol name (it contains `MAGIC || selected_version || hs_type`). The protocol name is processed separately by `Initialize()`. No change to prologue bytes.

2. **Handshake frame wire format.** No change. The Noise protocol name is not transmitted on the wire; it is an implicit contract between both peers. Both peers must use the same name string or the handshake fails.

3. **Test vectors.** Any test vectors derived from my Stage 1 handshake examples are invalidated. New test vectors must use `BLAKE2b` as the hash.

4. **Session export key derivation.** The `GetHandshakeHash()` output changes because the hash function changed. This affects the SessionResumption token encryption key (see Part 5).

### 2.3 What Does NOT Change

- The outer header format (plaintext, no crypto dependency).
- The inner header format.
- The stream header format.
- The SACK format.
- Domain separator strings (those use BLAKE3, not the Noise hash).
- Connection ID generation (now random_u32, no crypto dependency).

## Part 3: random_u32() Concession -- Connection ID Simplification

### 3.1 The Change

I conceded random_u32() in Stage 2a. The Advocate's argument was sound: the BLAKE3 derivation adds layer coupling (protocol needs CIK from mesh), persistence (session counter), and a DomainSeparator entry, all for a 32-bit plaintext routing shortcut. The security benefit is zero because the connection_id is visible in every frame's outer header.

**Before (Stage 1):**
```
connection_id = truncate_u32(
    BLAKE3_keyed("saalis-connid-v1",
        local_CIK_pub || remote_CIK_pub || handshake_nonce || session_counter
    )
)
```

**After (Stage 3a):**
```
connection_id = random_u32()

Reserved values:
  0x00000000 = handshake (pre-session frames)
  0xFFFFFFFF = reserved for future use

All other values are valid. Generate via CSPRNG.
On collision detection (received frame with matching connection_id
  but invalid AEAD), regenerate.
```

### 3.2 Removed Artifacts

1. **DomainSeparator variant `ConnId`** (value 7 in my Stage 1): removed. The `ResumeToken` variant takes value 7 instead.

2. **Two-phase ID scheme**: eliminated. My Stage 1 had `initiator_cid` and `responder_cid` in the handshake frame that differed from the post-handshake derived ID. With random_u32(), the initiator generates its connection_id before the handshake and uses it throughout. No phase transition.

3. **Session counter persistence**: eliminated. No counter file to corrupt, no wraparound to handle, no multi-instance conflicts.

### 3.3 Updated Handshake Frame

The handshake frame simplifies. The initiator's connection_id is known from the first message:

```
Handshake Frame (pre-AEAD, single magic):
  Offset  Width  Field
  ------  -----  -----
   0       4     magic: 0x56524B4F ("VRKO")
   4       1     frame_type: 0x03 (handshake)
   5       1     hs_type: u8 (0x01 = KK, 0x02 = XX)
   6       1     msg_index: u8 (0, 1, or 2)
   7       4     initiator_cid: u32  -- initiator's connection_id (random_u32)
  11       4     responder_cid: u32  -- responder's connection_id (random_u32)
                                     -- 0x00000000 in msg_index 0 (not yet known)
  15       2     hs_payload_len: u16
  17       var   hs_payload: [u8; hs_payload_len]  -- Noise handshake message

  Fixed header: 17 bytes
```

Verification: 4 + 1 + 1 + 1 + 4 + 4 + 2 = 17 bytes. Correct.

The initiator generates `initiator_cid = random_u32()` before sending msg_index 0. The responder generates `responder_cid = random_u32()` and includes it in msg_index 1. Both CIDs are used for the session lifetime.

## Part 4: Matter Chunk Transfer Wire Format (B5)

### 4.1 Frame Budget for Chunks

The Savant and I agree: matter chunking is critical. The system exists to transfer matter. The corrected frame budget for chunk frames:

```
UDP payload:                         1232 bytes
  - Outer header:                      29 bytes
  - AEAD tag:                          16 bytes
  - Inner header:                      10 bytes
  - Stream header:                      7 bytes
  - Chunk header:                      52 bytes
  = Available for chunk_data + bao_proof: 1118 bytes
```

Verification: 1232 - 29 - 16 - 10 - 7 - 52 = 1118 bytes. Correct.

### 4.2 Bao Proof Size Analysis

The bao tree proof size depends on the blob size and the chunk size. For BLAKE3 bao verified streaming:

```
proof_size = ceil(log2(total_chunks)) * 32 bytes

Examples (1024-byte bao chunks):
  Blob size     Total chunks    Proof size    Remaining for data
  1 MB          1,024           10 * 32 = 320    1118 - 320 = 798
  100 MB        102,400         17 * 32 = 544    1118 - 544 = 574
  500 MB        512,000         19 * 32 = 608    1118 - 608 = 510
  1 GB          1,048,576       20 * 32 = 640    1118 - 640 = 478
  2 GB          2,097,152       21 * 32 = 672    1118 - 672 = 446

Examples (512-byte bao chunks):
  Blob size     Total chunks    Proof size    Remaining for data
  1 MB          2,048           11 * 32 = 352    1118 - 352 = 766
  100 MB        204,800         18 * 32 = 576    1118 - 576 = 542
  2 GB          4,194,304       22 * 32 = 704    1118 - 704 = 414

Examples (2048-byte bao chunks):
  Blob size     Total chunks    Proof size    Remaining for data
  1 MB          512             9 * 32 = 288     1118 - 288 = 830
  100 MB        51,200          16 * 32 = 512    1118 - 512 = 606
  2 GB          1,048,576       20 * 32 = 640    1118 - 640 = 478
```

### 4.3 Chunk Size Decision

At 1024-byte bao chunks with a 2GB blob, the proof is 672 bytes and the remaining data capacity is 446 bytes per frame. This means each 1024-byte chunk requires ceil(1024 / 446) = 3 frames. That is inefficient.

At 512-byte bao chunks, each chunk fits in one frame for blobs up to ~100 MB (542 bytes available, 512 needed). For 2GB blobs the proof grows to 704, leaving 414 bytes, so even 512-byte chunks need 2 frames.

The trade-off: smaller chunks mean more chunks total (doubling chunk count halves throughput due to per-chunk overhead) but better per-frame packing. Larger chunks have fewer chunks but worse packing at extreme blob sizes.

**Decision: use 1024-byte bao chunk groups, split across frames.**

Rationale:
1. The bao standard uses 1024-byte chunks. Using the standard avoids incompatibility with existing bao implementations.
2. The chunk_data in a single frame is partial; the receiver reassembles at the stream layer using sequence numbers.
3. The stream multiplexer already handles reassembly (that is what streams do). A 1024-byte chunk spanning 2-3 frames is no different from a 1024-byte application message spanning 2-3 frames.
4. The chunk header's `chunk_len` field (u16) tells the receiver the total chunk size. The stream layer handles fragmentation transparently.

**Important clarification:** The chunk header appears once per chunk, not once per frame. A chunk that spans multiple frames has the chunk header in the first frame; subsequent frames for the same chunk carry stream continuation data (stream header with matching stream_id and incrementing sequence). The receiver reassembles the chunk from stream frames, then verifies the bao proof.

This changes the frame budget analysis. The 52-byte chunk header is amortized over the entire chunk, not per-frame:

```
Frame 1 of a chunk:
  1232 - 29 - 16 - 10 - 7 - 52 = 1118 bytes of (chunk_data + bao_proof)

Frame 2+ of a chunk (stream continuation):
  1232 - 29 - 16 - 10 - 7 = 1170 bytes of continuation data

For a 1024-byte chunk with 672-byte proof (2GB blob):
  Total chunk payload = 1024 + 672 = 1696 bytes
  Frame 1: 1118 bytes
  Frame 2: 1696 - 1118 = 578 bytes
  = 2 frames per chunk
```

Two frames per chunk for the worst case (2GB blob). For blobs under 100 MB, the proof is smaller and most chunks fit in a single frame. This is acceptable.

### 4.5 Transfer Control Frames

Three new message types for transfer lifecycle management:

#### TransferRequest (msg_type = 0x0D)

Sent by the receiver to request a blob transfer from a source peer.

```
TransferRequest (msg_type = 0x0D):
  Offset  Width  Field
  ------  -----  -----
   0       8     transfer_id: u64       -- CSPRNG-generated by requester
   8      32     blob_id: [u8; 32]      -- which blob to transfer
  40       4     have_bitmap_len: u32   -- number of u64 words in bitmap
  44       var   have_bitmap: [u64; have_bitmap_len]
                                        -- bitfield of already-held chunks
                                        -- bit N = 1 means chunk N is held
                                        -- empty (len=0) means "I have nothing"

  Fixed header: 44 bytes  [8+32+4 = 44, verified]
  Max bitmap: for 2GB blob with 1024-byte chunks = 2,097,152 chunks
              = 32,768 u64 words = 262,144 bytes
              (requires stream-level fragmentation across ~225 frames)
```

For fresh downloads (no prior chunks), `have_bitmap_len = 0` and the bitmap is empty. This is the common case. The frame payload is just the 44-byte header, fitting in a single frame.

For resumption after partial download, the bitmap indicates held chunks. The source skips those chunks. The bitmap may be large for very large blobs; the stream layer fragments it transparently.

#### TransferBegin (msg_type = 0x0E)

Sent by the source peer to confirm the transfer and provide verification metadata.

```
TransferBegin (msg_type = 0x0E):
  Offset  Width  Field
  ------  -----  -----
   0       8     transfer_id: u64       -- echoes the request's transfer_id
   8      32     blob_id: [u8; 32]      -- echoes for verification
  40       4     total_chunks: u32      -- total chunk count
  44       8     blob_size: u64         -- total blob size in bytes
  52      32     bao_root: [u8; 32]     -- bao tree root hash for verification
  84       2     chunk_size: u16        -- chunk size in bytes (typically 1024)
  86       2     reserved: u16          -- zero, future use

  Fixed header: 88 bytes  [8+32+4+8+32+2+2 = 88, verified]
```

The `chunk_size` field allows the source to communicate the bao chunk size. The standard is 1024, but the field exists for forward compatibility.

#### TransferAbort (msg_type = 0x0F)

Sent by either peer to cancel an in-progress transfer.

```
TransferAbort (msg_type = 0x0F):
  Offset  Width  Field
  ------  -----  -----
   0       8     transfer_id: u64       -- which transfer to abort
   8       1     reason: u8             -- abort reason code
   9       1     reserved: u8           -- zero
  10       2     detail_len: u16        -- length of diagnostic text
  12       var   detail: [u8; detail_len]  -- UTF-8 diagnostic (max 256 bytes)

  Fixed header: 12 bytes  [8+1+1+2 = 12, verified]

Reason codes:
  0x00 = COMPLETED          -- transfer finished successfully
  0x01 = INTEGRITY_FAILURE  -- bao proof verification failed
  0x02 = SOURCE_GONE        -- source peer disconnected
  0x03 = CANCELLED          -- receiver no longer wants this blob
  0x04 = STORAGE_FULL       -- receiver has no space
  0x05 = RATE_LIMITED       -- source is throttling transfers
  0x06 = CONN_CLOSING       -- underlying connection is closing
  0xFF = UNSPECIFIED         -- generic abort
```

### 4.6 Transfer Session Lifecycle

```
Requester                          Source
    |                                |
    |--- TransferRequest (0x0D) ---->|
    |    transfer_id, blob_id,       |
    |    have_bitmap                 |
    |                                |
    |<--- TransferBegin (0x0E) ------|
    |    transfer_id, total_chunks,  |
    |    bao_root, chunk_size        |
    |                                |
    |<--- CHUNK (0x04) -------------|  \
    |    chunk_index=0               |   |
    |<--- CHUNK (0x04) -------------|   | Chunks flow on bulk-tier
    |    chunk_index=1               |   | stream (0x1000-0xFFFE)
    |<--- CHUNK (0x04) -------------|   |
    |    chunk_index=N               |  /
    |                                |
    |--- TransferAbort (0x0F) ----->|  (optional: cancel mid-transfer)
    |    reason=COMPLETED or cancel  |
    |                                |
```

Transfer control messages (TransferRequest, TransferBegin, TransferAbort) use the reactive-tier stream range (0x0100-0x0FFF). Chunk data flows on a bulk-tier stream (0x1000-0xFFFE) with LEDBAT pacing. This separation ensures transfer control is prioritized over bulk data.

### 4.7 Multi-Source Transfer

For multi-source transfer (sequential provider fallback), the requester opens separate transfer sessions with different sources. Each session has a unique `transfer_id`. The requester coordinates locally:

1. Select best source (per SourceSelection predicates).
2. Send TransferRequest with `have_bitmap` reflecting already-held chunks.
3. Receive chunks from source A until source A fails or completes.
4. If incomplete, select next source, send new TransferRequest with updated `have_bitmap`.
5. Repeat until all chunks received.

The wire format supports this because `have_bitmap` communicates partial state, and `transfer_id` is per-session (different for each source). No multi-source coordination protocol is needed on the wire; the requester manages it locally.

## Part 5: SessionResumption Wire Format

### 5.1 Context

The ArchSurgeon promoted SessionResumption as a defined concept in verkko-protocol. I noted in Stage 2a that it needs an 80-byte wire format. The Savant's Stage 2b accepted this. The DomainSeparator registry now includes `ResumeToken = 7` with string `"saalis-resume-v1"`.

### 5.2 Resumption Token Wire Format

The resumption token is encrypted and stored locally by the client. It is sent to the server during 0-RTT handshake resumption.

```
SessionResumption Token (plaintext, before encryption):

  Offset  Width  Field
  ------  -----  -----
   0      32     session_id: [u8; 32]
                   BLAKE2b hash of the completed Noise handshake state.
                   Uniquely identifies the session to resume.
  32      32     session_key_hash: [u8; 32]
                   BLAKE3(session_transport_keys).
                   Server compares this to its stored key hash to verify
                   the token refers to a valid, unexpired session.
  64       8     expiry: u64
                   HLC timestamp. Token is invalid after this time.
                   Recommended expiry: 24 hours after session creation.
  72       4     ticket_counter: u32
                   Monotonic counter incremented each time the server
                   issues a new resumption token for this session.
                   Prevents replay of old tokens.
  76       4     reserved: [u8; 4]
                   MUST be zero. Future use.

  Plaintext total: 80 bytes  [32+32+8+4+4 = 80, verified]
```

### 5.3 Token Encryption

The token is encrypted with XChaCha20-Poly1305 (StorageAEAD, the same AEAD used for at-rest encryption throughout verkko):

```
Encrypted Resumption Token:

  Offset  Width  Field
  ------  -----  -----
   0      24     nonce: [u8; 24]
                   XChaCha20-Poly1305 nonce.
                   Derived: BLAKE3("saalis-resume-v1" || session_id)[0..24]
  24      80     ciphertext: [u8; 80]
                   Encrypted plaintext token.
  104     16     auth_tag: [u8; 16]
                   Poly1305 authentication tag.

  Total encrypted token: 120 bytes  [24+80+16 = 120, verified]
```

The encryption key is the session's export key, derived from the Noise handshake via `GetHandshakeHash()`. Since we now use BLAKE2b, the handshake hash is 64 bytes (BLAKE2b output). The encryption key is the first 32 bytes of the handshake hash.

```
resume_key = GetHandshakeHash()[0..32]
```

### 5.4 Token on the Wire: Resumption Handshake

When a client resumes a session, it sends the encrypted token in a modified handshake frame:

```
Resumption Handshake Frame:
  Offset  Width  Field
  ------  -----  -----
   0       4     magic: 0x56524B4F ("VRKO")
   4       1     frame_type: 0x03 (handshake)
   5       1     hs_type: 0x03 (RESUME, new value)
   6       1     msg_index: 0x00
   7       4     initiator_cid: u32     -- client's connection_id (random_u32)
  11       4     responder_cid: u32     -- 0x00000000 (not yet known)
  15       2     hs_payload_len: u16    -- length of payload
  17     120     encrypted_token: [u8; 120]  -- the encrypted resumption token
  137     var    early_data: [u8; hs_payload_len - 120]
                   0-RTT application data (optional, replay-vulnerable)

  Min frame size: 17 + 120 = 137 bytes
  Max frame size: 17 + 120 + (1232 - 29 - 17 - 120) = 17 + 1186 = 1203 bytes
                  (with 1066 bytes of early data)
```

Wait. The resumption handshake frame does not go inside the outer header; it is a pre-session frame like the regular handshake. Let me recalculate the max early data:

```
Max UDP payload for resumption frame:
  1232 bytes total
  - 17 bytes fixed header
  - 120 bytes encrypted token
  = 1095 bytes available for early_data
```

The server processes the resumption:

1. Decrypt the token using its stored `resume_key` for the `session_id`.
2. Verify `ticket_counter` is strictly greater than the last seen counter (prevents replay).
3. Verify `expiry` has not passed.
4. Verify `session_key_hash` matches the stored session key hash.
5. If all checks pass: restore session state, process early_data as 0-RTT data.
6. If any check fails: reject with a regular Noise KK or XX handshake (fallback).

### 5.5 Server Response to Resumption

```
Resumption Accept Frame:
  Offset  Width  Field
  ------  -----  -----
   0       4     magic: 0x56524B4F ("VRKO")
   4       1     frame_type: 0x03 (handshake)
   5       1     hs_type: 0x03 (RESUME)
   6       1     msg_index: 0x01
   7       4     initiator_cid: u32     -- echoed from client
  11       4     responder_cid: u32     -- server's connection_id (random_u32)
  15       2     hs_payload_len: u16
  17     120     new_token: [u8; 120]   -- fresh resumption token for next time
  137     var    response_data: [u8; hs_payload_len - 120]
                   1-RTT confirmed data

  Fixed header + token: 137 bytes
```

The server issues a fresh resumption token in the accept message. The `ticket_counter` is incremented. The old token is invalidated.

### 5.6 0-RTT Safety Constraints

Per the ArchSurgeon's invariant: "0-RTT data is replay-vulnerable: only idempotent operations permitted in the 0-RTT window."

The receiver MUST treat all data in the `early_data` field as potentially replayed. Permitted 0-RTT operations:

| msg_type | Operation | 0-RTT safe? | Reason |
|----------|-----------|-------------|--------|
| 0x01 | HEARTBEAT | Yes | Idempotent liveness signal |
| 0x02 | GOSSIP | Yes | Idempotent state merge (GSet) |
| 0x0A | FILTER_UPDATE | Yes | Idempotent set operation |
| 0x03 | CONTROL | No | Control log entries are non-idempotent |
| 0x04 | CHUNK | No | Could cause duplicate data writes |
| 0x0D | TRANSFER_REQUEST | No | Could initiate duplicate transfers |
| 0x10 | CONN_CLOSE | No | Non-idempotent lifecycle event |

The 0-RTT/1-RTT distinction is communicated by the `msg_index` in the handshake frame: msg_index 0 carries 0-RTT data; msg_index 1 from the server confirms the session, after which all data is 1-RTT safe.

## Part 6: Connection Teardown Wire Format (B12)

### 6.1 CONN_CLOSE Frame

```
CONN_CLOSE (msg_type = 0x10, inside encrypted payload, no stream header):

  Offset  Width  Field
  ------  -----  -----
   0       4     last_good_frame: u32   -- highest frame_number the sender
                                        -- has successfully processed
   4       1     reason: u8             -- close reason code
   5       1     reserved: u8           -- zero
   6       2     detail_len: u16        -- length of diagnostic detail
   8       var   detail: [u8; detail_len]  -- UTF-8 diagnostic (max 256 bytes)

  Fixed header: 8 bytes  [4+1+1+2 = 8, verified]
  Max frame: 55 (overhead) + 8 + 256 = 319 bytes
```

```

+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|                      last_good_frame                          |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|    reason     |   reserved    |          detail_len           |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|                                                               |

|                                                               |
```

### 6.2 Reason Codes

```
#[repr(u8)]
enum CloseReason {
    Normal          = 0x00,  // Clean shutdown, no error
    IdleTimeout     = 0x01,  // No traffic for metabolic timeout period
    KeyExpired      = 0x02,  // Session keys expired, need rehandshake
    MigrationFail   = 0x03,  // PATH_RESPONSE not received within timeout
    InternalError   = 0x04,  // Local error prevents continued operation
    VersionMismatch = 0x05,  // SchemaSync discovered incompatibility
    Evicted         = 0x06,  // Peer evicted by dominance cascade
    Convergence     = 0x07,  // Connection closing as part of convergence
    ProtocolError   = 0xFF,  // Unspecified protocol violation
}
```

I added `Evicted` (0x06) and `Convergence` (0x07) beyond my Stage 2a proposal. Eviction by dominance cascade is a clean close with a specific reason (the closed peer should not immediately reconnect). Convergence close happens during cluster reconfiguration.

### 6.3 Close Handshake Procedure

The close handshake is a two-phase process analogous to QUIC CONNECTION_CLOSE:

```
Phase 1: Initiation
  Peer A decides to close.
  Peer A stops accepting new stream creation.
  Peer A sends CONN_CLOSE with reason.
  Peer A enters CLOSING state.
  Peer A starts close_timer = min(2 * RTT, 5000 ms).

Phase 2: Acknowledgement
  Peer B receives CONN_CLOSE.
  Peer B stops accepting new stream creation.
  Peer B drains pending outbound data (SACKs for in-flight frames).
  Peer B sends its own CONN_CLOSE frame (echoing or providing its own reason).
  Peer B enters CLOSING state.
  Peer B starts close_timer = min(2 * RTT, 5000 ms).

Phase 3: Cleanup
  When close_timer expires:
    - Destroy all session state.
    - Zeroize all session keys (per KeyZeroization contract).
    - Release connection_id.
    - Remove all stream state for this connection.
```

### 6.4 Behavior in CLOSING State

While in CLOSING state, a peer:

- MUST process incoming SACK frames (to complete reliability handoff).
- MUST process incoming CONN_CLOSE frames (the peer's acknowledgment).
- MUST NOT create new streams.
- MUST NOT send new application data.
- MAY send SACK frames for already-received data.
- SHOULD retransmit its CONN_CLOSE if no acknowledgment is received within RTT.

If a peer receives a data frame (not SACK, not CONN_CLOSE) while in CLOSING state, it silently discards the frame. No error response.

### 6.5 Abrupt Close (No Handshake)

If a peer detects a fatal protocol violation (invalid AEAD, corrupted frame, replay detection), it MAY send a CONN_CLOSE with reason `ProtocolError` (0xFF) and immediately destroy session state without waiting for acknowledgment. This is the "abrupt close" path.

The remote peer will detect the connection loss via heartbeat timeout (3 missed heartbeats). The CONN_CLOSE frame is best-effort in this case; it may or may not arrive.

### 6.6 Heartbeat Timeout Close

If heartbeat timeout fires (3 missed heartbeats at the current metabolic rate), the local peer:

1. Enters CLOSING state.
2. Sends CONN_CLOSE with reason `IdleTimeout` (0x01).
3. Starts close_timer.
4. If no CONN_CLOSE received from remote peer before timer expires, destroys session state.

This replaces the current implicit behavior (silent session destruction after heartbeat timeout) with an explicit close that gives the remote peer a chance to acknowledge.

## Part 7: Chunk/Close Interaction

### 7.1 The Problem

The Savant identified this interaction in Stage 2b, Section 6.4: what happens when a CONN_CLOSE arrives while a matter transfer is in progress?

### 7.2 Wire-Level Resolution

Two cases:

**Case A: Receiver initiates close during active transfer.**
The receiver wants to close the connection but has an in-progress transfer from this peer.

```
1. Receiver sends TransferAbort (0x0F) with reason CONN_CLOSING (0x06).
2. Receiver waits for source to acknowledge (source stops sending chunks).
3. Receiver sends CONN_CLOSE (0x10).
4. Normal close handshake proceeds.
```

The TransferAbort MUST be sent before CONN_CLOSE. This gives the source peer a clean signal that the transfer is cancelled due to connection teardown, distinct from an integrity failure or policy cancellation.

**Case B: Source peer initiates close during active transfer.**
The source is sending chunks and wants to close.

```
1. Source stops sending chunks.
2. Source sends TransferAbort (0x0F) with reason CONN_CLOSING (0x06).
3. Source sends CONN_CLOSE (0x10).
4. Normal close handshake proceeds.
5. Receiver notes partial transfer. have_bitmap is updated locally.
6. Receiver may resume from another source using TransferRequest with have_bitmap.
```

**Case C: CONN_CLOSE arrives without prior TransferAbort.**
This happens if the closing peer does not know about the transfer (race condition) or uses abrupt close.

```
1. Receiver receives CONN_CLOSE.
2. Receiver implicitly aborts all in-progress transfers on this connection.
3. For each aborted transfer:
   a. Mark transfer state as INTERRUPTED (not FAILED).
   b. Retain have_bitmap for potential resumption with another source.
   c. Do NOT scar the source (the close was not an integrity failure).
4. Receiver enters CLOSING state per normal procedure.
```

### 7.3 Normative Statement

> When a connection enters CLOSING state, all in-progress matter transfers
> on that connection are implicitly aborted. The closing peer SHOULD send
> TransferAbort with reason CONN_CLOSING (0x06) for each active transfer
> before sending CONN_CLOSE. If CONN_CLOSE is received without prior
> TransferAbort, the remote peer MUST treat all active transfers as
> implicitly aborted with reason CONN_CLOSING. Implicitly aborted
> transfers MUST NOT generate scars against the remote peer.

## Part 8: Seat Transport (W5)

### 8.1 The Problem

The Advocate identified in Stage 2a (W5) and the Savant confirmed in Stage 2b: the seat is a web browser. Web browsers cannot send raw UDP. The seat-to-host-peer transport MUST use WebSocket or similar web-compatible transport.

### 8.2 Does This Need a Wire Format Spec?

Yes, but it is minimal. The WebSocket layer is a transport binding, not a new protocol. The wire format is:

```
Seat Transport Binding: WebSocket Binary Message

  Each WebSocket binary message carries exactly one verkko protocol
  frame (the bytes that would otherwise be a UDP payload).

  The WebSocket message boundaries correspond to frame boundaries.
  No additional framing is needed because WebSocket is message-oriented.

  The Noise handshake, authentication, and all subsequent seat protocol
  frames are carried as WebSocket binary messages with no modification
  to their byte layout.
```

### 8.3 WebSocket Frame Mapping

```
UDP transport (mesh peers):
  UDP datagram -> verkko frame (1 datagram = 1 frame, max 1232 bytes)

WebSocket transport (seat to host peer):
  WebSocket binary message -> verkko frame (1 message = 1 frame)
  No max size constraint from UDP; WebSocket messages can be larger.
  However, the SansIO protocol machine enforces the same frame size
  limit (1232 bytes) regardless of transport, to maintain consistent
  behavior and prevent the seat from triggering different code paths.
```

### 8.4 Why Not a Separate REST/SSE Protocol?

The Savant considered two options: verkko frames over WebSocket vs. a separate REST/SSE protocol. The verkko-seat dependency declaration says the seat depends on verkko-protocol (Connection, Session, Frame, Stream, SchemaSync, SansIO). This dependency only makes sense if the seat uses verkko protocol frames.

A separate REST/SSE protocol would mean the seat does NOT depend on verkko-protocol, contradicting the declared dependency. The seat would have its own framing, its own authentication, its own state management. This is a larger scope change than simply specifying the WebSocket transport binding.

The WebSocket approach preserves:
- Noise-based authentication (the same handshake, the same session keys).
- SchemaSync (protobuf schema exchange works identically).
- Stream multiplexing (seats can use streams for concurrent operations).
- The SansIO contract (the consumer delivers bytes from WebSocket instead of UDP).

### 8.5 Normative Text for verkko-seat

The following should be added as a defined concept in verkko-seat:

> **SeatTransport**
>
> The transport binding for seat-to-host-peer communication.
>
> Invariants:
> - Seats connect to the host peer via WebSocket (RFC 6455) over TLS.
> - Each WebSocket binary message carries exactly one verkko protocol frame.
> - The Noise handshake and all subsequent protocol operations proceed
>   identically to UDP transport. The protocol machine does not distinguish
>   between UDP-delivered and WebSocket-delivered bytes.
> - The WebSocket endpoint is served by the host peer's HTTP server
>   (axum) at a well-known path (e.g., `/verkko/seat`).
> - The maximum frame size is 1232 bytes, matching the UDP transport
>   limit, regardless of WebSocket's larger message capacity.
> - Connection lifecycle (heartbeats, CONN_CLOSE) operates identically
>   over WebSocket. The WebSocket close frame is NOT used for verkko
>   session teardown; CONN_CLOSE is used instead. WebSocket close is
>   treated as an abrupt transport failure.

### 8.6 TLS Consideration

The WebSocket MUST use TLS (wss://) even though the verkko frames are already Noise-encrypted. Reasons:

1. Web browsers require wss:// for WebSocket connections from HTTPS pages.
2. The Noise handshake messages in the first few frames are not yet encrypted (the handshake itself is being established). TLS protects these initial messages.
3. The TLS certificate is the host peer's self-signed certificate or a LAN-local CA certificate. This is a LAN-only connection.

The TLS layer adds latency (TLS handshake before Noise handshake) but this is negligible on LAN (sub-millisecond). The double encryption (TLS + Noise) is redundant for data frames but necessary for handshake frames and required by browser security policy.

### 10.2 Overhead Analysis for Matter Transfer

For a 2GB blob with 1024-byte bao chunks:

```
Total chunks:              2,097,152
Bao proof per chunk:       672 bytes (21 levels * 32 bytes)
Chunk payload:             1024 + 672 = 1696 bytes per chunk

Frame 1 per chunk:         1118 bytes of chunk payload
Frame 2 per chunk:         578 bytes of chunk payload (1696 - 1118)
Frames per chunk:          2

Total frames:              2,097,152 * 2 = 4,194,304 frames
Total wire bytes:          4,194,304 * 1232 = ~5.16 GB
Efficiency:                2 GB / 5.16 GB = 38.8%

Overhead breakdown:
  Outer header:            29 * 4,194,304 = ~116 MB
  Inner header:            10 * 4,194,304 = ~40 MB
  Stream header:           7 * 4,194,304 = ~28 MB
  AEAD tag:                16 * 4,194,304 = ~64 MB
  Chunk header:            52 * 2,097,152 = ~104 MB (once per chunk)
  Bao proofs:              672 * 2,097,152 = ~1.34 GB
  Actual data:             1024 * 2,097,152 = ~2.0 GB
  Frame padding (slack):   ~1.16 GB (unused space in frame 2)
```

The 38.8% efficiency for a 2GB blob looks poor. The dominant cost is bao proofs (1.34 GB, 26% of wire traffic) and frame slack (unused capacity in the second frame of each chunk).

For a 100 MB blob:

```
Total chunks:              102,400
Bao proof per chunk:       544 bytes (17 levels * 32 bytes)
Chunk payload:             1024 + 544 = 1568 bytes per chunk

Frame 1 per chunk:         1118 bytes
Frame 2 per chunk:         450 bytes (1568 - 1118)
Frames per chunk:          2

Total frames:              204,800
Total wire bytes:          204,800 * 1232 = ~240 MB
Efficiency:                100 MB / 240 MB = 41.7%
```

For a 1 MB blob:

```
Total chunks:              1,024
Bao proof per chunk:       320 bytes (10 levels * 32 bytes)
Chunk payload:             1024 + 320 = 1344 bytes per chunk

Frame 1 per chunk:         1118 bytes
Frame 2 per chunk:         226 bytes (1344 - 1118)
Frames per chunk:          2

Total frames:              2,048
Total wire bytes:          2,048 * 1232 = ~2.4 MB
Efficiency:                1 MB / 2.4 MB = 41.7%
```

The efficiency is consistent at ~40% for multi-frame chunks. For LAN transfers at 1 Gbps, a 2GB blob takes ~41 seconds at wire level (5.16 GB / 1 Gbps). For WAN at 10 Mbps, it takes ~69 minutes. These are acceptable for background transfers under LEDBAT pacing.

### 10.3 Optimization Note: Proof Batching

The Savant and SpecWriter noted that bao proofs can be sent separately from chunk data. If proofs are batched (e.g., send proof for chunks 0-7 in one frame, then send 8 chunks of pure data), the per-chunk frame count drops. The chunk header would carry `proof_len = 0` for data-only frames, and a separate msg_type or a flag in the chunk header would indicate a proof batch.

I do not specify this optimization in Stage 3a. The current format is correct and sufficient. Proof batching is a v2 optimization that can be added without breaking the chunk frame format (proof_len = 0 already works; the receiver would need to buffer chunks until the proof batch arrives).

## Part 11: Prologue Binding (Final)

The Advocate (W3) and Savant (Section 8.1, recommendation 2) both flagged that the prologue must be specified exactly once. The final prologue is:

```
Noise prologue for KK and XX handshakes:

  Offset  Width  Field
  ------  -----  -----
   0       4     magic: 0x56524B4F
   4       2     selected_version: u16 (from version negotiation)
   6       1     hs_type: u8 (0x01 = KK, 0x02 = XX)

  Total: 7 bytes  [4+2+1 = 7, verified]

  Both peers compute: prologue = [0x56, 0x52, 0x4B, 0x4F, lo(ver), hi(ver), hs_type]
  Little-endian for selected_version.
```

The prologue is passed to `Noise.Initialize(prologue)` before any handshake messages. Both peers MUST compute identical prologue bytes. If version negotiation selected version 1:

```
prologue = [0x56, 0x52, 0x4B, 0x4F, 0x01, 0x00, 0x01]  (KK)
prologue = [0x56, 0x52, 0x4B, 0x4F, 0x01, 0x00, 0x02]  (XX)
```

For session resumption (hs_type = 0x03), no Noise handshake occurs, so no prologue is computed. The resumption token verification replaces the Noise handshake.
