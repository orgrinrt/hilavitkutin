# Deep Dive: Accessibility (a11y) for an HTMX + SSE TV/Gamepad-First Web UI

## Context

Saalis serves a web UI from a single Rust binary on a Batocera/loisto appliance. The primary
interaction model is gamepad/remote on a TV screen, but the UI must also work on desktop browsers
with keyboard and mouse. The rendering stack is server-side HTML (Maud templates) delivered via
HTMX partial swaps and SSE for live updates. This document surveys how to make that stack
accessible.

---

## 1. ARIA Attributes in Server-Rendered HTML

Because Maud produces complete HTML on the server, every response is an opportunity to embed
correct semantics from the start --- no hydration step, no client-side rendering race.

### Roles

Use native HTML elements wherever possible (`<nav>`, `<main>`, `<aside>`, `<dialog>`,
`<button>`) instead of `<div role="button">`. The native element carries its own implicit ARIA
role, keyboard behavior, and focus management for free. Reserve explicit `role` attributes for
cases where no semantic element exists:

- `role="status"` for a toast/notification container.
- `role="alert"` for error messages that demand immediate attention.
- `role="region"` with `aria-label` for named landmark sections within the content grid.
- `role="tablist"`, `role="tab"`, `role="tabpanel"` for category tabs in the library browser.

### States

Server-rendered HTML can set state attributes directly:

- `aria-current="page"` on the active nav item.
- `aria-selected="true"` on the focused card in the content grid.
- `aria-expanded="true/false"` on collapsible detail panels.
- `aria-busy="true"` on a container whose contents are being fetched via HTMX, removed when the
  swap completes.
- `aria-disabled="true"` on controls that are not currently actionable (e.g. "Download" when no
  source is configured).

### Live Regions for SSE Updates

SSE pushes (download progress, notification toasts, scheduler status) inject HTML fragments into
containers. Those containers need `aria-live` so screen readers announce changes:

- **Polite** (`aria-live="polite"`): Download progress bars, background task status, metadata
  enrichment results. The screen reader announces when idle.
- **Assertive** (`aria-live="assertive"`): Error alerts, critical warnings (disk full, connection
  lost). The screen reader interrupts immediately.
- **Off** (`aria-live="off"`): High-frequency updates (per-second byte counters) that would
  overwhelm a screen reader. Provide a summary endpoint instead.

A critical implementation detail: the live region container must exist in the DOM *before*
content is injected. If the container is added dynamically, screen readers need approximately
two seconds to register it with the accessibility API before announcements will work. The
safest pattern is to render an empty container in the initial page load:

```html
<div id="notifications" role="status" aria-live="polite" aria-atomic="true"></div>
```

Then SSE events swap content into it. The `aria-atomic="true"` attribute tells assistive
technology to announce the entire region contents as a single unit, rather than just the
changed fragment, which prevents confusing partial announcements.

## 2. Screen Reader Compatibility with HTMX Partial Swaps

HTMX's `hx-swap` replaces DOM subtrees. This is the most significant accessibility challenge
because it can silently destroy focus, break live region registration, and leave screen readers
with a stale virtual buffer.

### The Core Problem

When htmx replaces a DOM element, any focus on or within that element is lost. The screen
reader's virtual buffer (a cached representation of the page) may not update immediately. The
Wagtail CMS team's [2025 analysis](https://wagtail.org/blog/htmx-accessibility-gaps-data-and-recommendations/)
found that htmx-powered sites in the Django ecosystem scored worse on automated accessibility
audits than React-powered sites, largely because htmx's documentation oversimplifies
accessibility as "just use HTML." The library provides mechanisms, but the developer must apply
them deliberately.

### Mitigation Strategies

**Use morph swaps for stateful regions.** HTMX supports morphing via extensions (idiomorph).
Instead of replacing the entire subtree, morphing diffs the new HTML against the existing DOM and
patches in-place. This preserves focus, scroll position, video playback state, and form input
values. For the content grid where a card might be focused, morph swaps are strongly preferred:

```html
<div hx-get="/library?page=2" hx-swap="morph:innerHTML" hx-target="#content-grid">
```

**Manage focus explicitly after swap.** HTMX fires the `htmx:afterSwap` event. Use it to
restore focus:

```js
document.addEventListener("htmx:afterSwap", function(evt) {
  const target = evt.detail.target;
  // If the swap target contains a focusable element, focus it
  const focusable = target.querySelector("[autofocus], [data-focus-target]");
  if (focusable) focusable.focus();
});
```

Server-rendered fragments can include `autofocus` or a `data-focus-target` attribute on the
element that should receive focus after the swap completes.

**Use `hx-preserve` for persistent containers.** Elements marked with `hx-preserve` and an `id`
are not replaced during swaps. This is useful for the notification live region, audio/video
players, or any element whose destruction would cause accessibility breakage.

**Set `aria-busy` during requests.** Add `aria-busy="true"` to the swap target when the request
starts, remove it after the swap completes. HTMX's `htmx:beforeRequest` and `htmx:afterSwap`
events can toggle this. Screen readers that support `aria-busy` will hold announcements until the
update finishes, preventing partial or stale information from being read.

**Announce swap completion to screen readers.** For swaps that change significant page content
(e.g. navigating between library sections), inject a visually-hidden status message into a live
region: "Now showing: Retro Games, page 2 of 14."

## 3. Keyboard Navigation

### Tab Order

Server-rendered HTML with semantic elements naturally produces a correct tab order: the DOM
order is the tab order. This is straightforward when the server controls layout. Key rules:

- Never use positive `tabindex` values (1, 2, 3...). Use only `tabindex="0"` (add to tab order)
  or `tabindex="-1"` (programmatically focusable but not in tab order).
- Ensure all interactive elements are natively focusable (`<a>`, `<button>`, `<input>`,
  `<select>`) or given `tabindex="0"` with an appropriate `role`.
- For the content grid, each card should be a single tab stop (the card itself is focusable),
  with arrow keys used for spatial navigation within the grid. This avoids dozens of tab stops.

### Focus Trapping in Modals

When a modal dialog opens (game details, settings, confirmation prompts):

1. Move focus to the first interactive element inside the modal, or to the modal container
   itself if it has `tabindex="-1"`.
2. Trap focus: Tab and Shift+Tab must cycle only through elements within the modal.
3. Escape key closes the modal.
4. On close, return focus to the element that triggered the modal.

The native `<dialog>` element handles most of this automatically, including focus trapping. Use
`dialog.showModal()` rather than manual visibility toggling. Since saalis targets modern
Chromium (Batocera typically ships CEF or a Chromium kiosk), `<dialog>` has full support.

ARIA attributes for modals:

```html
<dialog aria-labelledby="dialog-title" aria-describedby="dialog-desc">
  <h2 id="dialog-title">Game Details</h2>
  <p id="dialog-desc">Metadata for Super Mario Bros.</p>
  <!-- interactive content -->
  <button autofocus>Close</button>
</dialog>
```

### Skip-to-Content Links

A skip link is the first focusable element in the `<body>`. It is visually hidden until it
receives keyboard focus, then it becomes visible. It lets keyboard users bypass repetitive
navigation to jump directly to `<main>`:

```html
<body>
  <a href="#main-content" class="skip-link">Skip to main content</a>
  <nav><!-- sidebar/header nav --></nav>
  <main id="main-content" tabindex="-1">
    <!-- library content -->
  </main>
</body>
```

CSS for the skip link:

```css
.skip-link {
  position: absolute;
  left: -9999px;
  top: auto;
  width: 1px;
  height: 1px;
  overflow: hidden;
}
.skip-link:focus {
  position: fixed;
  top: 0;
  left: 0;
  width: auto;
  height: auto;
  padding: 0.5em 1em;
  background: var(--color-surface);
  color: var(--color-on-surface);
  z-index: 9999;
}
```

On a TV interface, the skip link is less relevant since gamepad navigation uses spatial/arrow
keys rather than Tab, but it remains essential for desktop keyboard users and screen reader users.

## 4. Color Contrast

### WCAG 2.1 AA Requirements

- **Normal text** (below 18pt / 24px, or below 14pt / 18.66px bold): minimum contrast ratio of
  **4.5:1** against background.
- **Large text** (18pt+ / 24px+, or 14pt+ / 18.66px+ bold): minimum contrast ratio of **3:1**.
- **UI components and graphical objects**: minimum contrast ratio of **3:1** (WCAG 2.1 SC 1.4.11).
  This includes focus indicators, icon buttons, form field borders, and chart elements.

### Ensuring Contrast in a Themeable System

Saalis plans a theme system (deferred design topic). To prevent themes from breaking contrast:

**Define color roles, not raw colors.** Use CSS custom properties organized by semantic role:

```css
:root {
  --color-surface: #1a1a2e;
  --color-on-surface: #e0e0e0;       /* text on surface: must be 4.5:1 */
  --color-surface-variant: #252542;
  --color-on-surface-variant: #c0c0d0;
  --color-primary: #4a9eff;
  --color-on-primary: #000000;
  --color-error: #ff6b6b;
  --color-on-error: #000000;
}
```

Each `--color-on-*` is guaranteed to contrast against its corresponding `--color-*` at the AA
ratio. Theme authors must satisfy these pairs.

**Validate at theme registration time.** When a user installs a custom theme, compute the
contrast ratio between each foreground/background pair using the relative luminance formula from
WCAG. Reject or warn about themes that fail AA minimums. This can be done in Rust at build time
or theme-load time --- no runtime cost:

```
contrast_ratio = (L1 + 0.05) / (L2 + 0.05)
```

where L1 is the lighter color's relative luminance and L2 is the darker.

**Never rely on color alone.** Use icons, patterns, underlines, or text labels alongside color
to convey meaning (WCAG SC 1.4.1). A download status should not be indicated solely by
green/yellow/red; add text labels like "Complete", "In Progress", "Failed."

## 5. Reduced Motion

The `prefers-reduced-motion` media query detects when a user has enabled the "Reduce motion"
setting in their OS (macOS, Windows, iOS, Android all support this). Saalis should respect it
because excessive animation can trigger vestibular disorders, migraines, and seizures, and is
distracting for users with ADHD.

### Implementation

Define a CSS custom property that gates all animation durations:

```css
:root {
  --transition-duration: 200ms;
  --animation-play-state: running;
}

@media (prefers-reduced-motion: reduce) {
  :root {
    --transition-duration: 0ms;
    --animation-play-state: paused;
  }
}
```

All CSS transitions and animations reference these properties:

```css
.card {
  transition: transform var(--transition-duration) ease;
  animation-play-state: var(--animation-play-state);
}
```

For HTMX swap transitions (view transitions API, CSS transitions on swap), wrap them in a
`prefers-reduced-motion: no-preference` check so they only run when the user has not requested
reduced motion.

Specific concerns for a TV/gamepad UI:

- **Focus-highlight animations** (glow, scale-up on the selected card) should be replaced with
  instant state changes (border/outline change, no scale animation) under reduced motion.
- **Page transition animations** (slide-in for new content) should be replaced with instant
  swaps.
- **Loading spinners** are generally acceptable even under reduced motion, but provide a text
  alternative ("Loading...") for screen readers via `aria-label`.

## 6. High Contrast Mode

### Windows High Contrast / Forced Colors

Windows provides a "High Contrast" mode (renamed "Contrast Themes" in Windows 11) that
overrides all website colors with a limited system palette. The `forced-colors` CSS media query
detects this:

```css
@media (forced-colors: active) {
  .card {
    border: 2px solid ButtonText;
  }
  .card:focus {
    outline: 3px solid Highlight;
  }
  .icon {
    forced-color-adjust: auto; /* let the browser remap colors */
  }
}
```

Browser support for `forced-colors` is approximately 93%. The older `-ms-high-contrast` media
query is deprecated and should not be used in new code.

### Key Considerations

- **Borders become critical.** In forced-colors mode, `background-color` and `box-shadow` may
  be overridden. Visible borders on cards, buttons, and interactive elements ensure they remain
  distinguishable.
- **`forced-color-adjust: none`** opts an element out of color remapping. Use sparingly --- only
  for elements like cover art images where the original colors are meaningful.
- **System color keywords** (`Canvas`, `CanvasText`, `LinkText`, `Highlight`, `HighlightText`,
  `ButtonFace`, `ButtonText`) map to the user's chosen high-contrast palette. Use these in
  `forced-colors` media queries instead of hardcoded hex values.
- **Focus indicators** must be visible. In forced-colors mode, rely on `outline` rather than
  `box-shadow` or `background-color` changes, since outlines are preserved.

For the TV/appliance context, forced-colors is primarily relevant for desktop browser users.
The Batocera kiosk environment is unlikely to have high-contrast mode enabled, but the CSS is
low-cost insurance.

## 7. Large Text and Zoom (WCAG Reflow)

WCAG 2.1 SC 1.4.4 (Resize Text) requires text to be resizable up to 200% without loss of
content or functionality. SC 1.4.10 (Reflow) goes further, requiring that at 400% zoom
(equivalent to a 320px viewport width), content reflows into a single column with no horizontal
scrolling.

### Implementation Strategy

- **Use `rem` or `em` for all font sizes.** Never use `px` for text. This ensures text scales
  with the user's browser font-size setting and zoom level.
- **Use CSS Grid and Flexbox for layout.** The content card grid should use
  `grid-template-columns: repeat(auto-fill, minmax(200px, 1fr))` so that at higher zoom levels
  (where the viewport effectively shrinks), columns collapse gracefully --- from a 6-column grid
  on a TV to a single column at 400% zoom on desktop.
- **Avoid fixed-width containers.** Use `max-width` with percentage or viewport units, not
  `width: 1200px`.
- **Test at 320px equivalent.** Set the browser to 400% zoom and verify that all content is
  readable, no horizontal scroll bar appears, and all interactive elements remain accessible.
- **Allow text containers to grow vertically.** Never set `height` or `max-height` on text
  containers in a way that would clip overflow. Use `overflow: visible` or `auto` with scroll.

For the TV context, the base font size should already be larger than typical desktop (body text
at ~20-24px equivalent for readability at 3-meter viewing distance). This means desktop zoom
requirements are more demanding; the TV layout is already "zoomed."

## 8. Alt Text for Cover Art

Every `<img>` element needs an `alt` attribute. For a content library manager, cover art images
are the primary visual content.

### Auto-Generating Alt Text from Metadata

When no human-written description is available (which will be the common case for game cover
art), generate alt text from the entity's metadata:

```
alt="Cover art for Super Mario Bros. (Nintendo, 1985, NES)"
```

Template: `Cover art for {title} ({publisher}, {year}, {platform})`

This gives screen reader users the essential identifying information without requiring a visual
description of the artwork itself. The alt text should be concise --- under 100 characters is
the general guideline, as longer alt text creates a poor screen reader experience.

### Edge Cases

- **Missing metadata:** If only the title is known, use `alt="Cover art for {title}"`.
- **No image available (placeholder):** Use `alt=""` (empty alt) with `aria-hidden="true"` on
  the placeholder image, and ensure the game title is visible as text nearby. An empty alt tells
  screen readers to skip the image entirely, which is correct when the image carries no
  information.
- **Decorative images:** Background textures, separator lines, and other purely decorative
  images should have `alt=""` so screen readers skip them.
- **User-uploaded images:** These will lack alt text. Prompt the user to provide a description,
  or flag them for moderation/enrichment.

### Implementation in Maud

In the Maud template, the alt text is constructed server-side from the entity's metadata fields:

```rust
img src=(cover_url) alt=(format!("Cover art for {}", entity.display_name()));
```

This is a strength of server-side rendering: alt text generation is trivial Rust string
formatting, no client-side logic needed.

## 9. Gamepad-First UI and Accessibility Intersection

Saalis's primary input is a gamepad or TV remote. This creates a natural alignment with
accessibility needs, but also some unique challenges.

### Spatial Navigation Benefits

Gamepad navigation uses directional input (D-pad / left stick) to move between focusable
elements based on spatial position. This is functionally identical to the
[W3C WICG Spatial Navigation specification](https://wicg.github.io/spatial-navigation/), which
was designed for exactly these use cases: TV remote controls, game console pads, and
accessibility.

The specification is not yet implemented natively in any browser (Chromium has an experimental
flag `--enable-spatial-navigation`). For production use, a polyfill is needed. Options:

- **[spatial-navigation-polyfill](https://www.npmjs.com/package/spatial-navigation-polyfill)**:
  The WICG's own polyfill. Handles directional focus movement based on element geometry. Not
  yet spec-complete but functional.
- **[js-spatial-navigation](https://github.com/nicedoc/js-spatial-navigation)**: Originally by
  Mozilla. Battle-tested in TV apps. Supports sections (navigation containers), custom
  algorithms, and focus-change events.
- **Custom implementation**: For a server-rendered HTMX app, a minimal approach is feasible.
  Listen for gamepad D-pad events (via the Gamepad API) and arrow key events, find the nearest
  focusable element in the pressed direction, and call `.focus()` on it.

The CSS property `spatial-navigation-contain` (from the spec) creates logical navigation
containers --- focus wraps within a section rather than escaping to unrelated regions. Until
browsers implement it, replicate this with JavaScript by assigning sections and constraining
directional focus movement within them.

### How Gamepad Navigation Helps Screen Readers

A gamepad-first design forces the developer to think about:

- **A clear visual focus indicator on every interactive element.** This directly benefits
  keyboard and screen reader users.
- **Logical grouping of elements into navigable regions.** This maps to ARIA landmarks and
  regions.
- **Reduced reliance on hover states.** There is no hover on a gamepad. All states must be
  conveyed through focus and selection, which are accessible to all input methods.
- **Predictable navigation patterns.** A grid of cards navigated with arrow keys is intuitive
  for both gamepad users and keyboard-only users.

### Where Gamepad and Accessibility Diverge

- **Screen readers add a layer of abstraction.** A screen reader user navigates the accessibility
  tree, not the visual layout. Spatial navigation is meaningless if the accessibility tree does
  not reflect the visual structure. Ensure DOM order matches visual order.
- **Gamepad users do not use Tab.** Desktop keyboard users do. Both input methods must work.
  Arrow keys for spatial navigation within grids, Tab for moving between landmark regions, Enter
  for activation.
- **Text-to-speech on the appliance.** If running on Batocera (Linux), there may be no system
  screen reader. Consider bundling or recommending an accessible browser extension, or
  implementing basic TTS announcements (using the Web Speech API) for gamepad-only users who
  need auditory feedback.

### Gamepad API Integration

The [Gamepad API](https://www.w3.org/TR/gamepad/) provides access to connected controllers.
Map buttons to keyboard-equivalent actions:

| Gamepad Input  | Keyboard Equivalent | Action                 |
|----------------|---------------------|------------------------|
| D-pad / L-stick| Arrow keys          | Spatial navigation     |
| A / Cross      | Enter               | Activate / Select      |
| B / Circle     | Escape              | Back / Cancel / Close  |
| Start          | ---                 | Open menu              |
| Shoulder L/R   | PageUp / PageDown   | Previous/next section  |
| Triggers       | ---                 | Scroll content         |

By translating gamepad input to keyboard events (or shared handler functions), the same
navigation code serves both input methods, and screen readers benefit from the keyboard events.

## 10. Testing Accessibility

### Automated Tools

Automated tools catch approximately 30-57% of accessibility issues (Deque's axe-core
documentation cites up to 57% with zero false positives). They are essential but insufficient
alone.

**axe-core** (by Deque Systems):
- Open-source JavaScript engine. Rules library covers WCAG 2.0, 2.1, and 2.2 at A, AA, and AAA.
- Can be integrated into Rust test suites by running a headless browser (via `chromiumoxide` or
  similar) and injecting axe-core, then asserting on the results.
- As of 2025, axe-core 4.8 shows a 75% reduction in testing time while maintaining 92% accuracy.
- Zero false-positive design philosophy means every reported issue is a genuine problem.

**Pa11y / Pa11y CI**:
- Command-line accessibility testing tool. Wraps axe-core (or HTML_CodeSniffer) with browser
  automation.
- Pa11y CI is designed for CI pipelines: provide a list of URLs, it tests them all and reports.
- Well-suited for saalis because pages are server-rendered at known URLs; no client-side routing
  to handle.
- Can be run against the running saalis binary in a CI pipeline:
  1. Start saalis in test mode with a known SQLite database.
  2. Run `pa11y-ci` against `http://localhost:PORT/library`, `/settings`, etc.
  3. Fail the build if any WCAG AA violations are found.

**Lighthouse**:
- Google's audit suite covers accessibility alongside performance, SEO, and best practices.
- The accessibility score is axe-core under the hood, so findings overlap.
- Useful for a quick holistic view but Pa11y CI is better for dedicated a11y gating in CI.

### Browser Developer Tools

- **Chrome DevTools Accessibility pane**: Inspect the accessibility tree, check computed ARIA
  properties, and verify that the accessible name and role are correct for each element.
- **Firefox Accessibility Inspector**: Similar capabilities; also highlights tab order and shows
  keyboard navigation flow.
- **Chrome DevTools "Rendering" tab**: Emulate `prefers-reduced-motion`, `prefers-contrast`,
  `forced-colors`, and vision deficiencies (protanopia, deuteranopia, tritanopia, achromatopsia)
  without changing OS settings.

### Manual Testing

Automated tools miss context-dependent issues. Manual testing is required for:

- **Screen reader walkthrough**: Test with NVDA (Windows, free), VoiceOver (macOS/iOS, built-in),
  and TalkBack (Android, built-in). Navigate the entire library flow: browse, search, select a
  game, view details, trigger a download, check notifications. Verify that:
  - Every control has an accessible name.
  - Dynamic updates (SSE notifications, HTMX swaps) are announced.
  - Focus is never lost after a swap.
  - Modal dialogs trap focus correctly.
- **Keyboard-only testing**: Unplug the mouse, navigate entirely with Tab, Shift+Tab, Arrow
  keys, Enter, Escape, and Space. Verify that every feature is reachable and operable.
- **Gamepad testing**: Connect a controller, navigate the full UI flow. Verify that spatial
  navigation covers every interactive element, no element is unreachable, and the focus indicator
  is always visible.
- **Zoom testing**: Set browser zoom to 200% and 400%. Verify content reflows without horizontal
  scrolling, no text is clipped, and all controls remain operable.
- **Color contrast spot checks**: Use a contrast checker (WebAIM Contrast Checker, Colour
  Contrast Analyser) on custom themes.

### CI Pipeline Integration

A practical CI accessibility gate for saalis:

```yaml
# In GitHub Actions or equivalent
accessibility-test:
  steps:
    - run: cargo build --release
    - run: ./target/release/saalis --test-mode &
    - run: npx pa11y-ci --config .pa11yci.json
    - run: npx axe-cli http://localhost:8080/library --exit
```

`.pa11yci.json`:
```json
{
  "defaults": {
    "standard": "WCAG2AA",
    "runners": ["axe"],
    "chromeLaunchConfig": {
      "args": ["--no-sandbox"]
    }
  },
  "urls": [
    "http://localhost:8080/",
    "http://localhost:8080/library",
    "http://localhost:8080/settings"
  ]
}
```

## Summary: Priority Checklist for Saalis

| Priority | Area                        | Effort | Impact |
|----------|-----------------------------|--------|--------|
| P0       | Semantic HTML in Maud       | Low    | High   |
| P0       | Skip-to-content link        | Low    | High   |
| P0       | Focus management on swap    | Medium | High   |
| P0       | aria-live for SSE regions   | Low    | High   |
| P0       | Alt text from metadata      | Low    | High   |
| P1       | Color contrast enforcement  | Medium | High   |
| P1       | Keyboard navigation         | Medium | High   |
| P1       | prefers-reduced-motion      | Low    | Medium |
| P1       | Modal focus trapping        | Medium | High   |
| P1       | Morph swaps for grids       | Medium | High   |
| P2       | forced-colors support       | Low    | Medium |
| P2       | Reflow at 400% zoom         | Medium | Medium |
| P2       | Spatial nav polyfill        | Medium | High   |
| P2       | CI accessibility testing    | Medium | High   |
| P3       | Theme contrast validation   | Medium | Medium |
| P3       | TTS for appliance mode      | High   | Medium |

The overarching insight: a gamepad-first, server-rendered architecture is *already closer to
accessible* than most SPAs. The server controls the HTML, semantic elements are natural in Maud,
there is no hydration gap, and the gamepad requirement forces visible focus indicators and
logical navigation regions. The main risks are HTMX swap focus loss (mitigated by morph swaps
and explicit focus management) and the temptation to skip ARIA attributes because "it's just
HTML" (the Wagtail team's data shows this is a real trap).
