### P0-2. MetabolicState Abstract Governor Contract

**Addresses:** Run 10 P0-2, Tension 3
**Target document:** verkko-protocol (MetabolicState defined concept, Section 6.3)
**Depends on:** nothing

#### Draft normative text

Replace the current MetabolicState defined concept invariant "MetabolicState transitions
are driven by a governor that provides metabolic state as input. See verkko-mesh: Conductor
for the governor specification." with the following:

```
### MetabolicGovernor (abstract contract)

An abstract contract that drives MetabolicState transitions. The protocol
layer defines the state machine and this contract; mesh or other layers
provide concrete implementations.

Invariants:
- MetabolicGovernor is a pure function from resource pressure readings
  to MetabolicState.
- The governor MUST produce exactly one of {Background, Idle, Active,
  Stress} for any valid input.
- The governor MUST NOT require access to protocol-internal state
  beyond what it receives as input.
- The governor MUST satisfy the dwell time invariant: no transition
  SHALL be emitted within dwell_time of the previous transition,
  except when convergence_override is true.

### MetabolicGovernor interface

    trait MetabolicGovernor {
        /// Evaluate current resource pressures and return the
        /// appropriate metabolic state.
        ///
        /// Preconditions:
        ///   - All pressure values MUST be in [0.0, 1.0].
        ///   - elapsed_since_last_transition MUST be non-negative.
        ///
        /// Postconditions:
        ///   - Returns a valid MetabolicState variant.
        ///   - If elapsed_since_last_transition < dwell_time AND
        ///     NOT convergence_override, returns the current state
        ///     unchanged.
        fn evaluate(
            &self,
            pressures: &ResourcePressures,
            current_state: MetabolicState,
            elapsed_since_last_transition: TimeDelta,
            convergence_override: bool,
        ) -> MetabolicState;
    }

    /// Opaque resource pressure readings provided by the governor
    /// implementation. The protocol layer does not inspect these
    /// values; it passes them through to the governor.
    ///
    /// The concrete structure is defined by the governor
    /// implementation (e.g., verkko-mesh Conductor).
    struct ResourcePressures {
        /// Each pressure is a value in [0.0, 1.0] where 0.0 means
        /// no pressure and 1.0 means maximum pressure.
        /// The governor implementation defines the number and
        /// meaning of pressure dimensions.
        values: [f64; N],  // N defined by implementation
    }
```

The MetabolicState state machine (Section 6.3) defines the four states,
their parameter maps (heartbeat interval, LEDBAT target delay, bandwidth
allocation), and the dwell time invariant. The MetabolicGovernor contract
defines how transitions are triggered. The protocol layer calls
`governor.evaluate()` on each heartbeat cycle and transitions if the
returned state differs from the current state.

**State machine transitions:**

```
    Background <-> Idle <-> Active <-> Stress
```

All transitions are bidirectional. Adjacent-only transitions are NOT
required; the governor MAY emit any state from any state (subject to
dwell time). This permits the convergence override to jump directly
from Background to Active or Stress.

**Error handling:**

| Condition | Response | State after |
|-----------|----------|-------------|
| Governor returns invalid state | Log error, retain current state | Unchanged |
| All pressure values are NaN | Log error, retain current state | Unchanged |
| Governor panics or is absent | Default to Active | Active |

The concrete governor implementation (verkko-mesh: Conductor) provides
the `ResourcePressures` values and implements the `min()` meet
projection. This is the sole coupling point between protocol and mesh
for metabolic state.

### P0-3. HeartbeatExtension Defined Concept

**Addresses:** Run 10 P0-3, Consensus Finding 3
**Target document:** verkko-mesh (new defined concept section, before Body)
**Depends on:** nothing

#### Draft normative text

```
### HeartbeatExtension

Mesh-layer state carried in the heartbeat frame's extension region.
This is the sole carrier of mesh-layer state observable by remote
peers via the transport layer.

Invariants:
- HeartbeatExtension fields are written by the mesh layer and
  read by remote mesh layers.
- The protocol layer transports HeartbeatExtension as an opaque
  18-byte payload. The protocol layer MUST NOT interpret, modify,
  or act on HeartbeatExtension field contents.
- HeartbeatExtension MUST be present in every heartbeat frame.
  A heartbeat without a HeartbeatExtension is malformed.
- Field layout is fixed. No version field. Changes to the layout
  require a protocol version increment (see verkko-protocol:
  version negotiation).

Wire format (18 bytes, big-endian, inside encrypted payload):

    Offset  Width  Field                Type     Description
    ------  -----  -----                ----     -----------
     0      8      filter_root          u64      Keyed truncated Merkle root
                                                 for divergence detection.
                                                 Computed per verkko-matter:
                                                 ContentFilter.
     8      4      epoch_current        u32      G-Counter macro-epoch.
                                                 Monotonic per sender.
    12      1      work_capacity        u8       Re-encryption and donation
                                                 capacity. Computed as
                                                 min(bw_kbps/128,
                                                 flash_kb/256, 255).
                                                 Intermediate computation
                                                 MUST use u16 or wider to
                                                 avoid overflow.
    13      1      health_canary        u8       Bits 7-4: worst health
                                                 classification (0=Healthy,
                                                 1=Degraded, 2=Bad).
                                                 Bits 3-0: count of peers
                                                 with Degraded or worse
                                                 classification (0-15,
                                                 clamped).
    14      4      observation_summary  u32      Aggregated cross-cluster
                                                 FlowSummary. See
                                                 ObservationPathForwarding.

The state byte (offset 0 in the fixed header, defined by
verkko-protocol) carries the following mesh-relevant bits:

    Bit 2: gossip_mode        0 = Plumtree, 1 = flood.
    Bit 3: convergence_active 0 = normal, 1 = convergence in progress.
    Bit 4: admin_flag         0 = non-admin, 1 = current admin.

Bits 0-1 carry the metabolic state (protocol-layer).
Bits 5-7 are reserved and MUST be set to 0.

Consumers:
  1. Epoch sync (epoch_current) — verkko-crypto: EpochRatchet
  2. Filter divergence detection (filter_root) — verkko-matter
  3. Donation scheduling (work_capacity) — verkko-relay
  4. Health observation (health_canary) — verkko-mesh: HealthPipeline
  5. Gossip mode coordination (gossip_mode) — verkko-mesh: Gossip
  6. Convergence coordination (convergence_active) — verkko-mesh
  7. Observation aggregation (observation_summary) — verkko-mesh
```

**Error handling for HeartbeatExtension:**

| Condition | Error code | Recovery | State after |
|-----------|-----------|----------|-------------|
| Extension region < 18 bytes | HB_EXT_SHORT | Drop heartbeat. Do not update any mesh state from this heartbeat. Log diagnostic. | Unchanged; failure detection timer continues. |
| epoch_current < previously received epoch from same sender | HB_EPOCH_REGRESS | Drop heartbeat. Log warning with both epoch values. | Unchanged. |
| work_capacity overflow (intermediate > u8) | (local computation error) | Clamp to 255. | work_capacity = 255. |
| health_canary worst_class > 2 | HB_CANARY_INVALID | Treat as worst_class = 2 (Bad). Log warning. | health_canary interpreted as Bad. |

### P0-5. ConvergenceGate Defined Concept

**Addresses:** Run 10 P0-5, Consensus Finding 8
**Target document:** verkko-mesh (new defined concept section)
**Depends on:** P0-4 (named fronts)

#### Draft normative text

```
### ConvergenceGate

Boolean predicate gating the transition between convergence fronts.
Consumers observe the gate's boolean output (open or closed) but
MUST NOT inspect the inputs or mechanisms behind the gate
evaluation (opaque projection principle).

Invariants:
- A gate is either OPEN (true) or CLOSED (false). No partial states.
- A gate, once OPEN, MUST NOT close within the same convergence event.
  Gates are monotonic within one convergence cycle.
- Gates are evaluated locally by each peer. No distributed consensus
  is required for gate evaluation.
- Gates have timeout fallbacks. A gate that does not open within its
  timeout opens with degraded semantics (specified per gate).

Gate definitions:

    Gate 1: ControlGate (gates ControlFront -> ConsensusFront)
        Predicate:
            all_control_entries_processed
            AND all_reachable_keys_obtained
            AND all_ema_resets_applied
        Timeout: 5 minutes.
        On timeout: orphan unretrievable channels. Apply EMA resets
            even if key retrieval is incomplete. Gate opens.
        After timeout: orphaned channels are logged. Admin is
            notified via SuccessionEvent-style notification
            (fire-and-forget POST, 3 attempts, 1s/5s/30s backoff).

    Gate 2: ConsensusGate (gates ConsensusFront -> ReencryptionFront)
        Predicate:
            highest_fencing_token_confirmed
            AND dominance_resolved
            AND ring_coloring_valid
        Timeout: 10 minutes.
        On timeout: highest fencing token seen wins. Dominance
            computed from available GSet state. Gate opens.
        After timeout: peers with divergent fencing tokens log a
            scar (ScarAttribution::CorrelatedSuspicion).

    Gate 3: ReencryptionGate (gates ReencryptionFront -> exit)
        Predicate:
            all_affected_content_reencrypted_or_epoch_forwarded
            AND vital_queue_depth < 25%
        Timeout: none (bounded by tiered deadlines: P1=4h, P2=24h,
            P3=bandwidth-derived, P4=epoch-forward).
        Note: this gate does not open until the ConsensusGate is
            open. The vital_queue_depth < 25% precondition prevents
            re-encryption traffic from starving control messages.

    Gate 4: ExitGate (gates exit sequence completion)
        Predicate:
            gossip_mode_ack_complete
            AND spanning_tree_connected
            AND health_baselines_updated
            AND ring_checksum_verified
        Timeout: 3 * wavefront_deadline.
        On timeout: log the incomplete steps. Force exit. Resume
            steady-state operation. Incomplete steps (e.g., spanning
            tree not fully optimized) are handled by steady-state
            mechanisms.

Interface (opaque projection):

    fn is_open(&self) -> bool;

Consumers MUST use only `is_open()`. The gate implementation
MAY change its internal predicate evaluation without affecting
consumers, provided the boolean outcome is preserved.
```

### P0-6. ScarSeverityTier Defined Concept

**Addresses:** Run 10 P0-6, Consensus Finding 6
**Target document:** verkko-mesh (new defined concept section)
**Depends on:** nothing

#### Draft normative text

```
### ScarSeverityTier

Classification of scar types into severity tiers for weighted
scoring. Each scar type maps to exactly one tier. New scar types
default to Tier 2 unless explicitly classified.

    #[repr(u8)]
    enum ScarSeverityTier {
        Excluded  = 0,  // Tier 0: excluded from dominance scoring
        Transient = 1,  // Tier 1: weight multiplier 0.5
        Concerning = 2, // Tier 2: weight multiplier 1.0 (default)
        Severe    = 3,  // Tier 3: weight multiplier 2.0
    }

Invariants:
- Every ScarAttribution variant MUST have a tier assignment.
- New ScarAttribution variants MUST default to Tier 2 (Concerning)
  until explicitly reclassified.
- Tier assignment is static per scar type: it does not change based
  on context, frequency, or runtime conditions.
- Tier 0 (Excluded) scars contribute to weighted_scar_count for
  health scoring but are excluded from dominance scoring.

Tier assignment table:

    ScarAttribution variant          Tier   Weight  Rationale
    ---------------------------      ----   ------  ---------
    PartitionAbsence                 0      0.0x    Absence is not fault.
                                            (dominance)  Excluded from dominance
                                            0.5x    by construction.
                                            (health)
    SourceConfirmed (transient)      1      0.5x    Hardware flaps, brief
                                                    WiFi dropouts.
    NoAlternatePath                  1      0.5x    Topology limitation,
                                                    not peer fault.
    PathFault                        2      1.0x    Persistent path issue
                                                    with unclear attribution.
    CorrelatedSuspicion              2      1.0x    Multiple observers see
                                                    issues; concerning but
                                                    not confirmed severe.
    SourceConfirmed (persistent)     2      1.0x    Repeated confirmed
                                                    degradation from same
                                                    source.
    IntegrityFailure                 3      2.0x    Content hash mismatch.
                                                    Data corruption or
                                                    malicious behavior.

SourceConfirmed classification rule:
- Transient (Tier 1): first occurrence within a 24-hour window
  from this observer for this target.
- Persistent (Tier 2): second or subsequent occurrence within
  a 24-hour window from this observer for this target.

The 24-hour window is computed from scar observation_epoch
timestamps, not wall clock time.

Weighted scar count with tiers:

    weighted_scar_count(peer, now) =
        sum(
            tier_weight(s.tier) * decay_weight(s, now)
            for s in scars_targeting(peer)
        )

    where:
        tier_weight(Excluded)   = 0.5  (health) or 0.0 (dominance)
        tier_weight(Transient)  = 0.5
        tier_weight(Concerning) = 1.0
        tier_weight(Severe)     = 2.0
        decay_weight(s, now)    = max(0.0, 1.0 - (now - s.observation_epoch) / 180_days)

This replaces the unweighted weighted_scar_count formula in
Section 5.3 (Memoria). The three consumers (health scoring,
dominance cascade, gateway ETT) continue to use
weighted_scar_count as input; their formulas are unchanged.
```

### P0-7. DomainSeparator Enum

**Addresses:** Run 10 P0-7, Disagreement 1
**Target document:** verkko-crypto (new defined concept section)
**Depends on:** nothing

#### Draft normative text

```
### DomainSeparator

repr(C) enum enumerating all registered domain-separated hash and
derivation usages. This enum is the sole API for domain-separated
hashing in the verkko protocol. Raw BLAKE3 MUST NOT be called
directly from protocol crates; all BLAKE3 usage MUST go through
a function that takes a DomainSeparator variant.

    #[repr(C)]
    enum DomainSeparator {
        /// Key identification: BLAKE3(channel_key || index)[0..8]
        /// Input: 32-byte channel key + u16 index
        /// Output: 8 bytes
        KeyId         = 0,

        /// Re-encryption nonce derivation:
        /// BLAKE3(new_key || blob_id || epoch)[0..24]
        /// Input: 32-byte key + 32-byte blob_id + u32 epoch
        /// Output: 24 bytes
        ReencNonce    = 1,

        /// Founding transcript hash:
        /// BLAKE3(founding_transcript)
        /// Input: variable-length founding transcript
        /// Output: 32 bytes
        FoundingHash  = 2,

        /// Metabolic phase jitter:
        /// BLAKE3(DK_pub || epoch)[0..2]
        /// Input: 32-byte DK_pub + u32 epoch
        /// Output: 2 bytes
        PhaseJitter   = 3,

        /// Heartbeat jitter:
        /// BLAKE3(DK_pub || epoch)[0..2]
        /// Input: 32-byte DK_pub + u32 epoch
        /// Output: 2 bytes
        HbJitter      = 4,

        /// Session ID derivation:
        /// BLAKE3(session_key || channel_path || generation)[0..8]
        /// Input: 32-byte session_key + variable channel_path + u32 generation
        /// Output: 8 bytes
        SessionId     = 5,

        /// Destination peer ID (outer header):
        /// BLAKE3(CIK_pubkey)[0..4]
        /// Input: 32-byte CIK pubkey
        /// Output: 4 bytes
        DestinationId = 6,
    }

Invariants:
- No two variants share the same domain separator string.
- The domain separator string for each variant is fixed at
  compile time.
- Adding a new domain-separated hash usage REQUIRES adding a
  new variant to this enum. The enum is the registry.
- Protocol crates MUST NOT import blake3::hash or
  blake3::Hasher directly. They MUST use the verkko-crypto
  domain-separated API.

Domain separator strings:

    Variant        String literal
    -------        --------------
    KeyId          "saalis-keyid-v1"
    ReencNonce     "saalis-reenc-v1"
    FoundingHash   (raw BLAKE3, no domain separation — the
                    founding transcript is structurally unique)
    PhaseJitter    "saalis-jitter-v1"
    HbJitter       "saalis-hbjitter-v1"
    SessionId      "saalis-sessid-v1"
    DestinationId  "saalis-destid-v1"

API surface:

    /// Domain-separated BLAKE3 hash.
    ///
    /// Preconditions:
    ///   - input MUST match the expected input format for the
    ///     given domain (see variant documentation above).
    ///
    /// Postconditions:
    ///   - Returns BLAKE3 keyed hash with the domain separator
    ///     as the key, except for FoundingHash which uses
    ///     unkeyed BLAKE3.
    ///   - Output length is variant-specific (see above).
    fn domain_hash(domain: DomainSeparator, input: &[u8]) -> [u8; 32];

    /// Convenience: truncate domain_hash output to the specified
    /// length for this domain.
    fn domain_hash_truncated(domain: DomainSeparator, input: &[u8]) -> Vec<u8>;

Lint rule: a custom lint MUST flag any direct BLAKE3 invocation
outside the verkko-crypto crate boundary. Severity: error.
```

Note: the relay_key_id derivation uses HMAC-BLAKE3 with channel_salt,
not the DomainSeparator API. This is intentional: HMAC-BLAKE3 is a
different construction from keyed BLAKE3 with domain separation. The
relay_key_id derivation is documented in verkko-protocol Section 3.1
and does not participate in the DomainSeparator registry.

### P0-8. HealthPipeline Two-Mode State Machine

**Addresses:** Run 10 P0-8, Consensus Finding 5, Disagreement 3
**Target document:** verkko-mesh (HealthPipeline section, Section 5.1)
**Depends on:** P0-3 (HeartbeatExtension carries convergence_active), P0-5 (ConvergenceGate)

#### Draft normative text

Insert after the current "Tap 1: Health classification" section heading, before the
classification table:

```
#### HealthPipeline operating modes

The HealthPipeline operates in one of two modes:

    #[repr(u8)]
    enum HealthMode {
        Normal = 0,
        Damped = 1,
    }

**NORMAL mode.** Default. Health observations are processed with
unmodified EMA alpha values (fast=0.1, slow=0.01).

**DAMPED mode.** Active during convergence. Health observations are
processed with attenuated EMA alpha values to prevent the
convergence-health feedback loop from oscillating.

    damped_alpha_fast = alpha_fast * damping_coefficient_fast
    damped_alpha_slow = alpha_slow * damping_coefficient_slow

Damping coefficients:
- damping_coefficient_fast: default 0.3. Calibration item.
  Valid range: (0.0, 1.0) exclusive. Setting 1.0 is a protocol
  violation (eliminates damping).
- damping_coefficient_slow: default 0.1. Calibration item.
  Valid range: (0.0, 1.0) exclusive.

Recovery bounds at default coefficients:
- Fast EMA damped alpha = 0.1 * 0.3 = 0.03.
  Observations to recover from 0 to 0.9: ceil(ln(0.1)/ln(0.97)) = 76.
  At 5-second Active heartbeat: ~380 seconds (~6.3 minutes).
- Slow EMA damped alpha = 0.01 * 0.1 = 0.001.
  Observations to recover from 0 to 0.9: ceil(ln(0.1)/ln(0.999)) = 2302.
  At 5-second Active heartbeat: ~11,510 seconds (~3.2 hours).

**Transition conditions:**

    NORMAL -> DAMPED:
        Trigger: any ConvergenceGate opens (i.e., convergence begins).
        The convergence_active bit (bit 3 of state byte) is set in
        the local peer's heartbeat.
        Transition is immediate (no dwell time).

    DAMPED -> NORMAL:
        Trigger: ALL ConvergenceGates have closed (ExitGate opened
        and exit sequence completed).
        The convergence_active bit is cleared.
        Transition is immediate (no dwell time).

**Distributed coordination.** Remote peers detect convergence via
the convergence_active bit in the HeartbeatExtension state byte.
When a peer receives a heartbeat with convergence_active=1, it
MUST enter DAMPED mode locally. When all received heartbeats show
convergence_active=0, the peer MAY exit DAMPED mode.

**Zero-observation bypass.** If a peer has sent zero heartbeats for
3 or more consecutive heartbeat periods (i.e., the peer is absent,
not merely degraded by convergence), the damping mask MUST NOT
apply to observations about that peer. The absence of heartbeats
is a genuine liveness signal, not convergence disruption. For
absent peers, observations (or the absence of observations)
proceed with NORMAL-mode alpha values regardless of HealthMode.

**Invariants:**
- The existence of DAMPED mode is a protocol constant. It MUST NOT
  be made optional or configurable.
- The coefficients within DAMPED mode are calibration items with
  the constraint: 0.0 < coefficient < 1.0.
- HealthMode transitions do not reset EMA state. The EMA history
  is preserved across mode transitions.
```

#### HeartbeatExtension canonical generator

    function generate_heartbeat_extension(
        filter_root: u64,
        epoch_current: u32,
        work_capacity_bw_kbps: u32,
        work_capacity_flash_kb: u32,
        worst_health_class: u8,     // 0, 1, or 2
        degraded_count: u8,         // 0-15
        observation_summary: u32,
    ) -> [u8; 18]:

        // work_capacity: intermediate must be wider than u8
        let wc_bw: u16 = min(work_capacity_bw_kbps / 128, 255)
        let wc_flash: u16 = min(work_capacity_flash_kb / 256, 255)
        let wc: u8 = min(wc_bw, wc_flash) as u8

        // health_canary packing
        let canary: u8 = (min(worst_health_class, 2) << 4)
                       | min(degraded_count, 15)

        // big-endian serialization
        let out: [u8; 18]
        out[0..8]   = filter_root.to_be_bytes()
        out[8..12]  = epoch_current.to_be_bytes()
        out[12]     = wc
        out[13]     = canary
        out[14..18] = observation_summary.to_be_bytes()
        return out

#### Version negotiation

Version negotiation occurs during the Noise handshake as part

**Protocol version identifier:**

    verkko_version: u32 = (major << 16) | (minor << 8) | patch

    v1.0.0 = 0x00010000

    Offset  Width  Field

- Patch version change: no wire format impact. Bug fixes only.

| Condition | Error code | Recovery |

### Authentication and session establishment

A seat authenticates to a host peer using a user credential, not
a device key (DK/DEK). The host peer validates the credential and
establishes an asymmetric session.

#### Credential types

    #[repr(u8)]
    enum SeatCredentialType {
        Password   = 0,  // Argon2id-hashed password
        Token      = 1,  // Opaque bearer token (for programmatic access)
        Resumption = 2,  // 0-RTT session resumption token

#### Session establishment (full handshake)

1. Seat initiates a Noise_KK or Noise_XX handshake to the host
   peer's local address (LAN only; seats do not connect over WAN).
2. After Noise handshake completes, the seat sends a SeatAuth
   message in the encrypted channel.
3. The host peer validates the credential and returns SeatAuthResult.
4. On success, the seat sends a SubscriptionRequest listing the

5. The host peer confirms subscriptions and sends an initial

     ?      32     schema_hash: [u8; 32] (protobuf schema hash)

     2      8      session_id: u64

     42     var    schema_update: [u8; ?] (present only if
                   schema_hash did not match; protobuf schema)

A seat with a valid resumption_token MAY skip the full handshake:

1. Seat sends Noise_KK message 1 with the resumption_token
   in the encrypted payload.
2. Host peer validates the token (not expired, not revoked).
3. If valid: session resumes. No SeatAuth exchange needed.
4. If invalid: host peer responds with RESUME_FAILED and the
   seat falls back to full handshake.

Resumption token validity: 24 hours or until the host peer
restarts, whichever comes first.

### Consistency model

The seat operates in one of two consistency modes:

    #[repr(u8)]
    enum SeatConsistencyMode {
        Consistent = 0,
        Snapshot   = 1,
    }

**CONSISTENT mode** (default). The host peer serves current mesh
state. Reads reflect the latest committed state. Writes are
applied immediately and propagated to the mesh.

**SNAPSHOT mode** (during convergence). When the host peer enters
convergence (convergence_active=1), it MUST switch the seat to
SNAPSHOT mode:

1. Host peer sends a ConsistencyModeChange event to all
   connected seats.
2. Reads return a frozen snapshot taken at convergence entry.
3. Writes are queued in a deferred-write buffer on the host peer.
4. All responses include a staleness indicator:
   `staleness_ms: u32` (milliseconds since snapshot was taken).
5. When convergence completes, host peer replays the deferred
   writes (in order received) and switches back to CONSISTENT.

**Deferred write semantics:**
- Writes are replayed in FIFO order after convergence.
- If a deferred write conflicts with a state change during
  convergence (e.g., the target entity was modified by the mesh),
  the host peer applies last-writer-wins with the mesh state
  winning. The seat is notified of the conflict via a
  WriteConflict event.
- Maximum deferred write queue: 1024 entries. If the queue
  is full, additional writes return DEFERRED_QUEUE_FULL and
  the seat MUST NOT retry until convergence ends.

**Convergence duration bound.** The host peer MUST NOT remain
in SNAPSHOT mode for longer than 600 seconds (configurable,
default 120 seconds per wavefront deadline). If convergence
exceeds this bound, the host peer proceeds with available
state and transitions back to CONSISTENT.

                    0x03 = INVALID_REQUEST
                    0x04 = INTERNAL_ERROR
                    0x05 = DEFERRED_QUEUE_FULL
                    0x06 = CONVERGENCE_ACTIVE (write blocked)
     5      4      staleness_ms: u32 (0 in CONSISTENT mode)
     9      2      payload_len: u16 (big-endian)

    Role    Permissions

    The responder selects the first suite it supports from the
    initiator's preference list. If no common suite exists, the
    handshake fails with CIPHER_MISMATCH.

In v1, this field is omitted. Both peers assume suite 0x01.
The version negotiation mechanism (P1-4) determines whether
cipher negotiation is present.
```

#### Invite token wire format

The invite token encodes the information needed for a new cluster

    Offset  Width  Field
    ------  -----  -----
     0      1      version: u8 (0x01)

     ?      4      expiry_epoch: u32 (big-endian)

     ?      64     signature: [u8; 64] (Ed25519 by admin DK)

    AddressHint (20 bytes):
        [u8  type]       1 byte   0x04=IPv4, 0x06=IPv6, 0x10=mDNS

        [u16 port]       2 bytes  big-endian
        [u8  addr[16]]  16 bytes  (IPv4: first 4 used; mDNS:

generate a new token for each enrollment.

| Condition | Error code | Recovery |

| Expired token | INVITE_EXPIRED | Request new invite from admin. |
| Signature invalid | INVITE_SIG_BAD | Token corrupted. Request new invite. |
| PSK already used | INVITE_USED | Token consumed. Request new invite. |
| No reachable hints | INVITE_UNREACHABLE | Try relay via relay_dht_key. |

#### Relay operation error codes

All relay operations (PUT, GET, DELETE, TOUCH, SIGNAL) return
a status byte as the first byte of the response.

    #[repr(u8)]
    enum RelayStatus {
        Ok             = 0x00,
        NotFound       = 0x01,  // GET/DELETE/TOUCH: relay_key_id not found
        AlreadyExists  = 0x02,  // PUT: relay_key_id exists (idempotent success)
        StorageFull    = 0x03,  // PUT: relay has no capacity
        TtlExpired     = 0x04,  // GET/TOUCH: blob existed but TTL expired
        AuthFailed     = 0x05,  // SIGNAL: authentication failure
        RateLimited    = 0x06,  // Any: too many operations per window
        BadRequest     = 0x07,  // Any: malformed request
        InternalError  = 0xFF,  // Relay internal failure
    }

**Relay operation wire format:**

    PUT request:
        [u8  op_code]         1 byte    0x01
        [u64 relay_key_id]    8 bytes
        [u32 blob_len]        4 bytes   big-endian
        [u8  blob[blob_len]]  variable  encrypted blob
        [u16 auth_tag_ref]    2 bytes   offset into blob where
                                        AEAD tag starts (for
                                        relay integrity check)
        Total header: 15 bytes.

    PUT response:
        [u8 status]           1 byte    RelayStatus

    GET request:
        [u8  op_code]         1 byte    0x02
        [u8  id_count]        1 byte    (1-16)
        [u64 relay_key_ids[]] 8*count   relay_key_ids to fetch
        [u8  decoy_count]     1 byte    (0-16)
        [u64 decoy_ids[]]     8*count   fake IDs (constant-rate padding)
        Total header: 3 + 8*(id_count + decoy_count) bytes.

    GET response (per ID):
        [u8  status]          1 byte    RelayStatus
        [u64 relay_key_id]    8 bytes
        [u32 blob_len]        4 bytes   (0 if not found)
        [u8  blob[blob_len]]  variable

    DELETE request:
        [u8  op_code]         1 byte    0x03
        [u64 relay_key_id]    8 bytes

    DELETE response:
        [u8 status]           1 byte    RelayStatus

    TOUCH request:
        [u8  op_code]         1 byte    0x04
        [u64 relay_key_id]    8 bytes
        [u32 new_ttl_days]    4 bytes   (0 = use default 90 days)

    TOUCH response:
        [u8 status]           1 byte    RelayStatus

    SIGNAL request:
        [u8  op_code]         1 byte    0x05
        [u8  signal_type]     1 byte    0x01=CONNECT, 0x02=PING
        [u64 target_conn_id]  8 bytes   connection ID of target
        [var payload]         variable  connection setup data

    SIGNAL response:
        [u8 status]           1 byte    RelayStatus

**Recovery actions by error code:**

| Code | Action |
|------|--------|
| NotFound | No retry. Blob may have expired or never existed. |
| AlreadyExists | Success. Idempotent PUT completed. |
| StorageFull | Defer operation. Retry after 60 seconds. |
| TtlExpired | Re-upload if content still needed. |
| AuthFailed | Re-authenticate with relay. |
| RateLimited | Exponential backoff: 1s, 2s, 4s, 8s, max 60s. |
| BadRequest | Do not retry. Fix the request. Log error. |
| InternalError | Retry once after 5s. If persistent, use alternate relay. |
```

### P3-3 (partial). Control Log Entry Format

**Addresses:** Run 10 P3-3, gap M-4
**Target document:** verkko-mesh (ControlLog section, Section 1.7)
**Depends on:** nothing

#### Draft normative text

```
#### Control log entry wire format

    Offset  Width  Field
    ------  -----  -----
     0      1      version: u8 (0x01)
     1      1      entry_type: u8 (see below)
     2      2      scope: u16 (channel_index, 0xFFFF = global)
     4      8      lamport: u64 (big-endian, Lamport clock)
    12      32     prev_hash: [u8; 32] (BLAKE3 of previous entry;
                    0x00 * 32 for first entry after genesis)
    44      32     author_dk_pub: [u8; 32] (Ed25519 pubkey of author)
    76      2      payload_len: u16 (big-endian)
    78      var    payload: [u8; payload_len]
     ?      64     signature: [u8; 64] (Ed25519 over bytes 0..78+payload_len)

    Entry type enum:
        0x01  ENROLLMENT_CERT     payload = enrollment certificate
        0x02  KEY_BUNDLE          payload = KeyBundle
        0x03  CIK_ROTATION        payload = new CIK pubkey (32 bytes)
        0x04  SCAR_ENTRY          payload = ScarEntry
        0x05  EDT_ISSUANCE        payload = EDT
        0x06  PARTITION_ANNOUNCE   payload = PartitionAnnouncement
        0x07  GATEWAY_CLAIM       payload = PeerId (4 bytes)
        0x08  GATEWAY_RELEASE     payload = FlowSummaryTable (44 bytes)
        0x09  SUCCESSION_EVENT    payload = new admin PeerId (4 bytes)
        0x0A  SNAPSHOT            payload = compressed GSet snapshot
        0x0B  DOMINANCE_OVERRIDE  payload = (reserved for future)

    Entry hash:
        entry_hash = BLAKE3(entry[0..78+payload_len+64])

**GSet merge rule (for concurrent entries from different partitions):**
- Union all entries from both partitions.
- Sort by (lamport, author_dk_pub) for deterministic ordering.
- Re-verify hash chain: entries that break the chain are rejected.
- Re-verify signatures: entries with invalid signatures are rejected.

**Compaction:**
- Entries older than 90 days MAY be truncated.
- A SNAPSHOT entry (type 0x0A) MUST be written before truncation.
- The SNAPSHOT entry is signed by the current admin.
- Snapshot payload: BLAKE3 of the serialized GSet state + the
  serialized GSet state (compressed with zstd).
```
