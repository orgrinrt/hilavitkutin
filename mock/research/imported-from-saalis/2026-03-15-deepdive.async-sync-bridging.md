# Deep Dive: Async-to-Sync Bridging Patterns in Rust

**Date:** 2026-03-15
**Scope:** Architecture patterns for "sync core, async shell" systems in Rust
**Relevance:** saalis uses sync Rust for all core logic (scheduler, DB, plugins) with async only at the axum web boundary

---

## 1. The "Sync Core, Async Shell" Pattern

### Origin

The pattern descends from Gary Bernhardt's [Functional Core, Imperative Shell](https://www.destroyallsoftware.com/screencasts/catalog/functional-core-imperative-shell) (2012) and its companion talk [Boundaries](https://www.destroyallsoftware.com/talks/boundaries). The original idea: keep the core logic pure and free of side effects; push all I/O and mutation to an outer shell. The core is easy to test, easy to reason about, and composes well.

The Rust-specific variant — **synchronous core, asynchronous shell** — maps the same separation onto Rust's async/sync split. The sync core contains all business logic, data transformations, scheduling decisions, and database access. The async shell handles network I/O (HTTP, SSE, WebSocket) and is the only code that touches a tokio runtime.

### Why It Works

1. **Function colouring becomes a guardrail.** Rust's async functions are "coloured" — you cannot call an async function from sync code without ceremony. Rather than fighting this, the pattern uses it as an architectural boundary. If your core is sync, you literally cannot accidentally introduce async I/O into it.

2. **Testability.** Sync code is trivially testable with `#[test]`. No `#[tokio::test]`, no runtime setup, no `.await`. The core can be tested in isolation, quickly, deterministically.

3. **Plugin compatibility.** Sync `repr(C)` FFI boundaries are straightforward. Async FFI is a research problem. By keeping the plugin interface sync, you avoid forcing plugin authors to depend on a specific async runtime.

4. **Reduced dependency surface.** The core crates do not depend on tokio at all. Only the web crate pulls in the async runtime. This shrinks compile times and reduces the attack surface for supply-chain issues.

5. **Portability.** Sync code runs anywhere — embedded, WASM, `no_std` contexts. Async runtimes impose platform constraints.

### Who Uses It

- [sulami's blog post](https://blog.sulami.xyz/posts/sync-core-async-shell/) documents this pattern explicitly in Rust, citing it as a direct application of Bernhardt's idea.
- The Rust Users Forum has a [dedicated discussion thread](https://users.rust-lang.org/t/sync-core-async-shell-pattern-in-practice/127683) on applying it in practice.
- Diesel (the most widely-used sync ORM in Rust) is designed around this model — it is entirely sync, and the async wrappers (`deadpool-diesel`, `diesel-async`) are external.
- SeaQL published ["The rainbow bridge between sync and async Rust"](https://www.sea-ql.org/blog/2024-05-20-async-rainbow-bridge/), a detailed exploration of bridging strategies.
- The [Tokio documentation itself](https://tokio.rs/tokio/topics/bridging) has a section on bridging sync and async code.

### Relevance to saalis

saalis's architecture is a textbook instance of this pattern: sync scheduler, sync SQLite persistence, sync plugin ABI, with async confined to the axum web layer. The rest of this document covers the specific bridging mechanisms needed where the async shell meets the sync core.

## 2. `block_on` at the Boundary

### What It Is

`tokio::runtime::Runtime::block_on(future)` blocks the calling OS thread until the given future completes. It is the entry point from sync-land into async-land.

`tokio::runtime::Handle::block_on(future)` does the same but uses a handle to an existing runtime rather than owning one.

### When It Is Safe

- **At the top level of `main()`.** The canonical `#[tokio::main]` macro expands to a `Runtime::block_on` call. This is always safe — there is no existing runtime context to conflict with.
- **In a dedicated thread.** If you spawn a `std::thread` and call `block_on` inside it, this is safe as long as the thread is not a tokio worker thread. This is the pattern Tokio recommends for running an async runtime in a separate thread and sending messages to it from sync code.
- **Inside `spawn_blocking`.** This is supported but has constraints (see below).

### When It Deadlocks

1. **Calling `block_on` from within an async task.** If you call `Runtime::block_on` from code already running on a tokio worker thread, you get a panic: "Cannot start a runtime from within a runtime." `Handle::block_on` will not panic but [can deadlock](https://github.com/tokio-rs/tokio/discussions/4563) — the blocked worker thread cannot process the very futures it needs to complete.

2. **`Handle::block_on` with a `current_thread` runtime.** With a current_thread runtime, the timer driver and I/O driver run on the same (single) thread. Calling `Handle::block_on` on that thread blocks it, so the drivers never run, and any timer-dependent or I/O-dependent future will hang forever.

3. **`block_on` inside `spawn_blocking` with pool exhaustion.** If your `spawn_blocking` task calls `block_on` on a future that requires *another* `spawn_blocking` task to complete, and the blocking thread pool is at capacity, you deadlock. The [Tokio discussion #3717](https://github.com/tokio-rs/tokio/discussions/3717) documents this precisely: "any task running on it (using block_on or not) should eventually exit on its own even if all other threads were paused."

### Recommendation for saalis

saalis should never need `block_on` inside the core — the core is sync. The only `block_on` call should be the one in `main()` that starts the axum server. If sync code ever needs to trigger an async operation (e.g., sending a notification via the web layer), use a channel, not `block_on`.

## 3. `spawn_blocking` for Calling Sync Code from Async Handlers

### Mechanism

`tokio::task::spawn_blocking(|| { ... })` moves the closure to a dedicated blocking thread pool, returning a `JoinHandle<T>` that can be `.await`ed in async code. This is the primary mechanism for axum handlers to call into sync core logic.

```rust
async fn get_entity(
    State(core): State<Arc<Core>>,
    Path(id): Path<EntityId>,
) -> Result<Json<Entity>, AppError> {
    let entity = tokio::task::spawn_blocking(move || {
        core.get_entity(id)
    }).await??;
    Ok(Json(entity))
}
```

### Overhead

`spawn_blocking` is not free. It involves:
- Scheduling work onto the blocking thread pool (lock contention on the pool's work queue).
- A context switch to the blocking thread.
- A context switch back when the work completes and the async task resumes.

For operations completing in **<1ms**, the overhead can exceed the work itself. For typical database queries (1-100ms), the overhead is negligible.

### Thread Pool Sizing

The default blocking thread pool has a very large upper limit — **512 threads**. This default exists because `spawn_blocking` is commonly used for I/O operations that may block for extended periods.

Key configuration knobs via `tokio::runtime::Builder`:
- `max_blocking_threads(n)` — sets the upper bound.
- Idle threads are reaped after a configurable timeout.

For CPU-bound work, the default pool is too large. Tokio's documentation explicitly recommends using **rayon** or a custom thread pool for CPU-bound parallelism, and using a **semaphore** to limit concurrency if you must use `spawn_blocking` for CPU work.

For saalis, SQLite queries are I/O-bound and typically short. The default pool size is fine. If contention becomes an issue, consider a dedicated connection pool pattern (see section 9).

### `block_in_place` Alternative

`tokio::task::block_in_place(|| { ... })` is an alternative that converts the current worker thread into a blocking thread in-place, moving other tasks off it. This avoids the context switch to a separate thread but has drawbacks:
- Cannot be used with `current_thread` runtimes.
- The current task cannot be cancelled while `block_in_place` is running.
- Other tasks on the same worker get shuffled, which can hurt their cache locality.

For saalis, `spawn_blocking` is the safer default. Use `block_in_place` only if profiling reveals the context-switch overhead is a bottleneck.

## 4. Channels Between Async and Sync

Channels are the fundamental primitive for communication across the async/sync boundary. Three main options:

### flume

[Flume](https://github.com/zesterer/flume) is purpose-built for the async/sync bridge. Key properties:
- Both `Sender` and `Receiver` implement `Send + Sync + Clone`.
- Provides both blocking (`recv()`, `send()`) and async (`recv_async()`, `send_async()`) methods on the same channel.
- The async `recv_async()` takes `&self` (not `&mut self`), unlike tokio's `mpsc::Receiver::recv(&mut self)`. This allows multiple async readers without wrapping in a `Mutex`.
- No `unsafe` code in the crate.
- Performance is competitive with or faster than `std::sync::mpsc` and sometimes `crossbeam-channel`.

**Status:** Flume is in [casual maintenance mode](https://lib.rs/crates/flume) — critical fixes only, no new features. This is fine for a stable channel primitive.

**Best fit for saalis:** Flume is the natural choice for the channel between the sync scheduler and the async SSE handler. The sync side calls `sender.send(event)` (blocking, but non-blocking in practice since sends are fast). The async side calls `receiver.recv_async().await`.

### tokio::sync::mpsc

Tokio's own multi-producer, single-consumer channel. Key properties:
- Async-native: `recv()` is async.
- `Sender` is `Send + Sync + Clone`, but `Receiver` is not `Clone` and requires `&mut self` for `recv()`.
- Does **not** support zero-capacity (rendezvous) channels.
- The `Sender` has a `blocking_send()` method for use from sync code.

**Use case:** Good when both ends are within tokio's ecosystem, or when you need backpressure via bounded channels. The `blocking_send()` method makes it usable from the sync side, but it blocks the calling thread.

### crossbeam-channel

The gold standard for sync-only channels. Key properties:
- Extremely fast, well-optimized for sync workloads.
- Supports bounded, unbounded, and zero-capacity (rendezvous) channels.
- `select!` macro for multiplexing across multiple channels.
- **No async support.** Cannot `.await` on a receive.

**Use case:** Ideal for communication between sync threads (e.g., within the scheduler). Not suitable for the async/sync bridge unless you wrap it in a `spawn_blocking` loop.

### Recommendation for saalis

| Path | Channel | Rationale |
|------|---------|-----------|
| Sync scheduler -> async SSE handler | flume | Sync send, async recv on same channel |
| Async handler -> sync core (commands) | flume or `tokio::sync::mpsc` with `blocking_recv` | Either works; flume is simpler |
| Sync-to-sync (within scheduler) | crossbeam-channel | Maximum performance, no async overhead |

## 5. Axum Handlers Calling Into Sync Core

### The Standard Pattern

Every axum handler that touches sync core logic should use `spawn_blocking`:

```rust
async fn handler(State(core): State<Arc<Core>>) -> impl IntoResponse {
    tokio::task::spawn_blocking(move || {
        core.do_sync_work()
    })
    .await
    .expect("blocking task panicked")
}
```

### Dedicated Thread Pool vs `spawn_blocking`

For most applications, `spawn_blocking` is sufficient. However, if the sync core has specific concurrency requirements (e.g., SQLite allows only one writer at a time), a dedicated thread pool gives more control:

```rust
// At startup
let db_pool = rayon::ThreadPoolBuilder::new()
    .num_threads(4)
    .build()
    .unwrap();

// In handler
let (tx, rx) = tokio::sync::oneshot::channel();
db_pool.spawn(move || {
    let result = core.do_sync_work();
    let _ = tx.send(result);
});
let result = rx.await.expect("db task panicked");
```

This gives explicit control over concurrency, avoids contention with other `spawn_blocking` users, and allows tuning the thread count to match the database's capabilities.

### The deadpool-diesel Pattern

[deadpool-diesel](https://crates.io/crates/deadpool-diesel) wraps Diesel's sync connections in an async-compatible pool. The pattern:

1. A pool manages N sync database connections.
2. `pool.get().await` acquires a connection (async wait if all are busy).
3. `conn.interact(|conn| { ... }).await` runs a closure on the pooled connection's thread.

This is essentially `spawn_blocking` with connection pooling baked in. The [axum diesel-postgres example](https://github.com/tokio-rs/axum/blob/main/examples/diesel-postgres/src/main.rs) in the axum repository demonstrates this.

For saalis, since SQLite uses a single file with limited write concurrency, a simpler model suffices: a single `Arc<Mutex<Connection>>` accessed via `spawn_blocking`. The mutex ensures write serialization, and `spawn_blocking` ensures the tokio runtime is not blocked.

## 6. SSE with a Sync Core

### The Problem

Server-Sent Events (SSE) require an async stream that yields events over time. The sync scheduler produces events (task completed, metadata updated, download progress). How do you connect them?

### The Channel-Bridge Pattern

The [axum-sse-from-channel](https://github.com/mouton0815/axum-sse-from-channel) repository demonstrates the canonical approach:

1. **Sync side:** The scheduler holds a `flume::Sender<Event>` (or `tokio::sync::broadcast::Sender<Event>`). When something interesting happens, it calls `sender.send(event)`.

2. **Async side:** The SSE handler subscribes to the channel and converts it into a `Stream`:

```rust
async fn sse_handler(
    State(event_tx): State<broadcast::Sender<Notification>>,
) -> Sse<impl Stream<Item = Result<Event, Infallible>>> {
    let rx = event_tx.subscribe();
    let stream = BroadcastStream::new(rx)
        .filter_map(|result| async move {
            result.ok().map(|notif| {
                Ok(Event::default()
                    .event(notif.kind)
                    .data(serde_json::to_string(&notif.payload).unwrap()))
            })
        });
    Sse::new(stream).keep_alive(KeepAlive::default())
}
```

### Broadcast vs MPSC

- **`tokio::sync::broadcast`** — multiple subscribers, each gets every message. Ideal for SSE where multiple browser tabs may be open. Late subscribers miss messages sent before they subscribed (configurable buffer).
- **`flume` unbounded** — can be cloned for multiple receivers, but each message goes to only one receiver (work-stealing). Not suitable for broadcast SSE.

For saalis's notification system, `tokio::sync::broadcast` is the right choice for the async-side fan-out. The sync scheduler writes to a single `broadcast::Sender` (using `send()`, which is sync-safe on broadcast channels — it does not require `.await`). Each SSE client subscribes independently.

### Connection Lifecycle

When an SSE client disconnects, the stream is dropped, which drops the `broadcast::Receiver`. The [axum discussions](https://github.com/tokio-rs/axum/discussions/1060) show that you can detect disconnection by implementing `Drop` on a guard type or by detecting `SendError` (all receivers dropped) on the sender side.

## 7. Testing the Sync Core Independently

### The Core Advantage

The entire point of the sync core pattern is testability. Core tests are plain `#[test]` functions:

```rust
#[test]
fn test_scheduler_prioritizes_enrichment() {
    let scheduler = Scheduler::new();
    scheduler.enqueue(WorkUnit::Enrich(entity_id));
    scheduler.enqueue(WorkUnit::Download(entity_id));
    let next = scheduler.next();
    assert_eq!(next, WorkUnit::Enrich(entity_id));
}
```

No tokio runtime, no async, no `.await`, no `#[tokio::test]`. Tests run fast, are deterministic, and exercise real logic.

### Testing the Boundary

For integration tests that exercise the axum handlers:

```rust
#[tokio::test]
async fn test_get_entity_handler() {
    let core = Arc::new(Core::new_test());
    let app = Router::new()
        .route("/entity/:id", get(get_entity))
        .with_state(core);

    let response = app
        .oneshot(Request::builder().uri("/entity/42").body(Body::empty()).unwrap())
        .await
        .unwrap();

    assert_eq!(response.status(), StatusCode::OK);
}
```

These tests need `#[tokio::test]` because they exercise the async shell, but the core logic being tested is still sync under the hood.

### Key Principle

If a test does not involve network I/O or async channels, it should not require an async runtime. The sync core pattern makes this the default rather than the exception.

## 8. Performance Implications

### Context Switching Overhead

Every `spawn_blocking` call involves at least two context switches: one to move work to the blocking pool, one to resume the async task when the work completes. On modern hardware, a context switch costs roughly 1-10 microseconds. For database queries taking 1-100ms, this is noise. For sub-microsecond operations, it dominates.

**Mitigation:** Batch multiple sync operations into a single `spawn_blocking` call rather than spawning one per field access.

### Thread Pool Contention

The blocking thread pool uses a shared work queue protected by a mutex. Under high concurrency (hundreds of concurrent `spawn_blocking` calls), this queue becomes a bottleneck.

**Mitigation:** Use a dedicated thread pool (rayon or custom) for hot paths. Reserve `spawn_blocking` for infrequent or varied work.

### Memory Overhead

Each blocking thread consumes a full OS thread stack (typically 2-8 MB). With the default limit of 512 threads, the worst case is ~4 GB of stack memory. In practice, idle threads are reaped, and the pool rarely approaches its limit.

### SQLite-Specific Considerations

SQLite serializes writes internally. Having many blocking threads contending on SQLite writes gains nothing — they serialize anyway. For saalis on a single-user appliance:
- A single writer thread (or `Mutex<Connection>`) is optimal.
- Read concurrency can be achieved with WAL mode and multiple reader connections.
- The thread pool size should match the actual parallelism available, not the default 512.

### Versus Pure-Async Alternatives

A fully async architecture (e.g., using `sqlx` with its async SQLite driver) avoids the context-switch overhead but introduces its own costs: futures state machines, waker allocations, and runtime complexity. For SQLite specifically, the "async" drivers often use `spawn_blocking` internally anyway — the database API is fundamentally synchronous. The sync core approach is honest about this reality.

## 9. Real-World Examples

### Diesel + axum

The [axum diesel-postgres example](https://github.com/tokio-rs/axum/blob/main/examples/diesel-postgres/src/main.rs) and the [axum-diesel-real-world](https://github.com/Quentin-Piot/axum-diesel-real-world) template demonstrate the standard pattern:
- Diesel connections are managed by `deadpool-diesel`.
- Each handler calls `pool.get().await` then `conn.interact(|c| { ... }).await`.
- The `interact` method internally uses `spawn_blocking`.
- Business logic inside `interact` is entirely sync.

This is the sync core pattern applied to database access, even if the projects do not name it as such.

### SQLx's Compile-Time Checking

SQLx offers an "offline mode" where queries are checked at compile time against a cached schema (in a `sqlx-data.json` file). The runtime query execution is async, but the compile-time checking is a fully sync process. This is another instance of keeping the computationally interesting work (type checking, schema validation) sync while the I/O wrapper is async.

### actix-web with `web::block`

Actix-web provides [`web::block()`](https://docs.rs/actix-web/latest/actix_web/web/fn.block.html), which is functionally equivalent to `spawn_blocking`. The pattern is identical:

```rust
async fn handler(data: web::Data<AppState>) -> impl Responder {
    let result = web::block(move || {
        data.sync_service.do_work()
    }).await?;
    HttpResponse::Ok().json(result)
}
```

Actix-web also supports [synchronous actors](https://actix.rs/) — actors that run on dedicated OS threads. This is analogous to a dedicated thread pool for specific sync services.

### GrepTimeDB

[GrepTimeDB's engineering blog](https://greptime.cn/blogs/2023-03-09-bridging-async-and-sync-rust) documents their experience bridging async and sync Rust in a production time-series database, covering the same patterns: `spawn_blocking`, channel bridges, and dedicated thread pools.

## 10. Common Pitfalls

### Pitfall 1: Holding a `MutexGuard` Across `.await`

```rust
// BUG: MutexGuard held across .await
async fn bad_handler(state: Arc<Mutex<State>>) {
    let guard = state.lock().unwrap();
    let data = guard.get_data();
    some_async_call(data).await;  // guard is still alive here!
    drop(guard);
}
```

`std::sync::MutexGuard` is `!Send`. Tokio can move a task between worker threads at any `.await` point. If a `MutexGuard` is alive across an `.await`, the compiler will reject the code because the future is not `Send`. This is a compile-time error, not a runtime bug — Rust protects you.

However, if you use `tokio::sync::Mutex` (whose guard *is* `Send`), the code compiles but the lock is held for the entire duration of the async call, causing unnecessary contention. The fix is the same in both cases — scope the lock:

```rust
async fn good_handler(state: Arc<Mutex<State>>) {
    let data = {
        let guard = state.lock().unwrap();
        guard.get_data().clone()
    };  // guard dropped here
    some_async_call(data).await;
}
```

[Tokio's shared state documentation](https://tokio.rs/tokio/tutorial/shared-state) covers this in detail. [Turso's blog post](https://turso.tech/blog/how-to-deadlock-tokio-application-in-rust-with-just-a-single-mutex) demonstrates how a single mutex can deadlock an entire tokio application.

### Pitfall 2: Blocking the Tokio Runtime

```rust
// BUG: blocks a tokio worker thread
async fn bad_handler(core: Arc<Core>) {
    let result = core.expensive_sync_computation();  // blocks!
    Json(result)
}
```

If `expensive_sync_computation` takes 100ms, a tokio worker thread is blocked for 100ms. With the default 1-per-CPU-core worker threads, this can stall the entire runtime. Always wrap sync work in `spawn_blocking`:

```rust
async fn good_handler(core: Arc<Core>) {
    let result = tokio::task::spawn_blocking(move || {
        core.expensive_sync_computation()
    }).await.unwrap();
    Json(result)
}
```

### Pitfall 3: `Send` Bounds on Sync Types

Types used in `spawn_blocking` closures must be `Send` (they cross thread boundaries). Common issues:
- `Rc<T>` is `!Send` — use `Arc<T>`.
- `Cell<T>` and `RefCell<T>` are `!Send` — use `Mutex<T>` or `RwLock<T>`.
- Raw pointers are `!Send` — wrap in a newtype with an unsafe `Send` impl, or restructure.

For saalis's plugin system with `repr(C)` types, ensure all types that cross the async/sync boundary are `Send + Sync`. Plugin handles passed into `spawn_blocking` closures must satisfy these bounds.

### Pitfall 4: Nested Runtime Creation

```rust
// BUG: panics if called from within a tokio context
fn sync_function_that_needs_async() {
    let rt = tokio::runtime::Runtime::new().unwrap();
    rt.block_on(async { ... });  // panics if already in a runtime
}
```

Creating a new runtime inside an existing one is usually a mistake. If the sync function is called from `spawn_blocking` (which runs within tokio's context), this will panic. Instead, use `Handle::current()` to get the existing runtime's handle, or better yet, restructure so the sync code does not need to call async code at all.

### Pitfall 5: Unbounded Channel Memory Growth

If the sync producer (scheduler) generates events faster than the async consumer (SSE handler) can send them, an unbounded channel grows without limit. Use bounded channels with backpressure, or `broadcast` with a fixed buffer size (where old messages are dropped when the buffer is full).

### Pitfall 6: `spawn_blocking` for Long-Lived Tasks

`spawn_blocking` threads count against the pool limit. If you use `spawn_blocking` for a task that runs for the lifetime of the application (e.g., a background sync loop), you permanently consume a pool slot. Use `std::thread::spawn` for long-lived background threads instead.

## Summary: Recommended Patterns for saalis

| Concern | Pattern | Notes |
|---------|---------|-------|
| Runtime entry point | `#[tokio::main]` / `Runtime::block_on` in `main()` | Single async entry point |
| axum handler -> sync core | `tokio::task::spawn_blocking` | Wrap every sync call |
| Sync scheduler -> SSE | `tokio::sync::broadcast` channel | `send()` is sync-safe |
| Sync-to-sync channels | `crossbeam-channel` or `flume` | No async overhead |
| Async-to-sync commands | `flume` (sync recv in scheduler loop) | Or `tokio::sync::mpsc` with `blocking_recv` |
| SQLite access | `Arc<Mutex<Connection>>` via `spawn_blocking` | WAL mode for read concurrency |
| Long-lived background threads | `std::thread::spawn` | Not `spawn_blocking` |
| CPU-bound parallelism | `rayon` thread pool | Not tokio's blocking pool |
| Testing core logic | Plain `#[test]` | No async runtime needed |
| Testing web layer | `#[tokio::test]` with `oneshot` | Integration tests only |
