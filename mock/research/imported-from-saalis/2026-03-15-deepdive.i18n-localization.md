# Deep Dive: i18n/l10n for Rust Server-Rendered Web Applications

## Context

Saalis serves a web UI via Maud templates rendered server-side, delivered through HTMX
partial exchanges. The application manages a content library (games initially, later
movies/books/music) where metadata---including titles---varies by region and language. This
research covers the full i18n stack: translation frameworks, template integration,
locale detection, RTL support, CLDR-based formatting, game-title localization, and
translation workflow tooling.

---

## 1. fluent-rs (Project Fluent)

### 1.1 Overview

Project Fluent is Mozilla's localization system, designed to give translators full
expressive power over natural language. The Rust implementation lives in the `fluent-rs`
workspace, with `fluent-templates` providing the high-level integration layer. Fluent's
key insight is that localization is *not* string interpolation---it requires giving
translators control over grammar, pluralization, gender, and sentence structure on a
per-locale basis.

### 1.2 FTL File Format

FTL (Fluent Translation List) files use a purpose-built syntax where indentation is
significant:

```ftl
# Simple message
hello-world = Hello, world!

# Variable interpolation via placeables
welcome = Welcome, { $userName }!

# Attributes (sub-messages attached to a message)
login-input = Placeholder text
    .placeholder = email@example.com
    .aria-label = Login email input

# Terms (reusable, translator-facing, prefixed with -)
-brand-name = Saalis

about-page = About { -brand-name }
```

### 1.3 Select Expressions: Pluralization and Gender

Select expressions let translators branch on runtime values. The `*` marks the default
variant (required):

```ftl
# Pluralization (CLDR plural categories: zero, one, two, few, many, other)
emails =
    { $count ->
        [one] You have one unread email.
       *[other] You have { $count } unread emails.
    }

# Gender selection
shared-photos =
    { $userName } { $photoCount ->
        [one] added a new photo
       *[other] added { $photoCount } new photos
    } to { $userGender ->
        [male] his
        [female] her
       *[other] their
    } stream.
```

Critically, each locale defines its *own* variants. Russian needs `one`, `few`, `many`,
`other`; Arabic needs `zero`, `one`, `two`, `few`, `many`, `other`. The translator---not
the developer---decides which branches exist.

### 1.4 Built-in Functions

FTL provides `NUMBER` and `DATETIME` functions for locale-aware formatting within
messages:

```ftl
price = The price is { NUMBER($amount, style: "currency", currency: "EUR") }.
event-date = Event starts on { DATETIME($date, month: "long", year: "numeric") }.
```

### 1.5 fluent-templates and static_loader!

The `fluent-templates` crate provides a high-level loader that maps directory structures
to language identifiers:

```
locales/
  en-US/
    main.ftl
    errors.ftl
  fi/
    main.ftl
    errors.ftl
  ja/
    main.ftl
    errors.ftl
  core.ftl          # shared across all locales
```

The `static_loader!` macro compiles all `.ftl` resources into the binary at compile time:

```rust
use fluent_templates::static_loader;

static_loader! {
    static LOCALES = {
        locales: "./locales",
        fallback_language: "en-US",
        // Optional: shared messages available in all locales
        core_locales: "./locales/core.ftl",
        // Optional: customize each FluentBundle
        customise: |bundle| bundle.set_use_isolating(false),
    };
}
```

The `Loader` trait provides lookup methods:

```rust
use fluent_templates::Loader;
use unic_langid::langid;

let lang = langid!("fi");

// Simple lookup
let msg = LOCALES.lookup(&lang, "hello-world");

// Lookup with arguments
let mut args = HashMap::new();
args.insert("userName", FluentValue::from("Matti"));
let msg = LOCALES.lookup_with_args(&lang, "welcome", &args);

// Attribute access via dot notation
let placeholder = LOCALES.lookup(&lang, "login-input.placeholder");
```

There is also `ArcLoader` for runtime loading (e.g., loading locale files from disk at
startup rather than embedding them), useful if translations are updated independently of
the binary.

### 1.6 Assessment for Saalis

Fluent is the strongest choice for a project that values correctness and expressiveness.
Its translator-centric design means Finnish (with its 15 noun cases), Japanese, and Arabic
can each express their grammar naturally. The `static_loader!` macro fits the
single-binary deployment model (Batocera appliance). The main trade-off is that FTL is a
bespoke format---translators need to learn it, though it is simpler than ICU MessageFormat.

## 2. rust-i18n

### 2.1 Overview

The `rust-i18n` crate (by Longbridge) takes a Rails/Ruby-inspired approach: translations
live in YAML, TOML, or JSON files, and the `t!` macro performs compile-time code
generation to embed them in the binary.

### 2.2 Initialization and the t! Macro

```rust
#[macro_use]
extern crate rust_i18n;

// Load translations from ./locales/, fall back to English
i18n!("locales", fallback = "en");

// Basic usage
t!("hello");                           // uses current locale
t!("hello", locale = "fi");            // explicit locale
t!("messages.hello", name = "world");  // interpolation
t!("serial", sn = 123 : {:08});       // Rust format specs
```

Locale is set globally via `rust_i18n::set_locale("fi")` and retrieved with
`rust_i18n::locale()`.

### 2.3 Translation File Formats

**Version 1 (one file per locale):**

```yaml
# locales/en.yml
_version: 1
hello: "Hello world"
messages.hello: "Hello, %{name}"
```

```yaml
# locales/fi.yml
_version: 1
hello: "Hei maailma"
messages.hello: "Hei, %{name}"
```

**Version 2 (all locales in one file):**

```yaml
# locales/app.yml
_version: 2
hello:
  en: Hello world
  fi: Hei maailma
  ja: こんにちは世界
messages.hello:
  en: "Hello, %{name}"
  fi: "Hei, %{name}"
  ja: "こんにちは、%{name}"
```

### 2.4 Backend System

The `Backend` trait allows custom translation sources (database, remote server):

```rust
pub trait Backend: Send + Sync {
    fn available_locales(&self) -> Vec<String>;
    fn translate(&self, locale: &str, key: &str) -> Option<String>;
}
```

### 2.5 String Extraction

The `cargo i18n` CLI tool (`cargo install rust-i18n-cli`) scans source code for `t!()`
invocations and generates skeleton translation files, reducing the risk of missing keys.

### 2.6 Assessment for Saalis

rust-i18n is simpler to adopt and uses familiar file formats. However, it lacks Fluent's
expressive power for complex grammar (pluralization is manual, no select expressions, no
gender support built in). The `%{name}` interpolation is string-level, not
linguistically-aware. For a project serving Finnish, Japanese, and potentially Arabic
users, the expressiveness gap matters. The `Backend` trait is appealing for loading
translations from the database, though.

## 3. Integrating i18n with Maud Templates

### 3.1 The Challenge

Maud is a compile-time HTML macro, not a string-template engine. There are no template
files to scan for translation keys---everything is Rust code. This means i18n integration
is done by splicing translated strings directly into Maud expressions.

### 3.2 Pattern: Translator Function

Define a helper that wraps the Fluent loader and the current user's locale:

```rust
use maud::{html, Markup, PreEscaped};
use unic_langid::LanguageIdentifier;

struct I18n {
    lang: LanguageIdentifier,
}

impl I18n {
    fn t(&self, key: &str) -> String {
        LOCALES.lookup(&self.lang, key)
    }

    fn t_args(&self, key: &str, args: &HashMap<String, FluentValue>) -> String {
        LOCALES.lookup_with_args(&self.lang, key, args)
    }
}

fn game_card(i18n: &I18n, game: &Game) -> Markup {
    html! {
        div.card {
            h2 { (game.localized_title(&i18n.lang)) }
            p { (i18n.t("game-card.description-label")) }
            span.rating { (i18n.t_args("game-card.rating", &args)) }
        }
    }
}
```

Maud's `(expr)` syntax renders any `Display` value, so translated strings splice in
naturally. For HTML content in translations (bold, links), use `(PreEscaped(i18n.t(...)))`,
but be cautious about XSS---only trust translations from controlled sources.

### 3.3 Pattern: Locale-Aware Layout

The root layout sets the `lang` and `dir` attributes based on the user's locale:

```rust
fn layout(i18n: &I18n, content: Markup) -> Markup {
    let dir = if is_rtl(&i18n.lang) { "rtl" } else { "ltr" };
    html! {
        html lang=(i18n.lang.to_string()) dir=(dir) {
            head {
                title { (i18n.t("site-title")) }
                // ...
            }
            body { (content) }
        }
    }
}
```

### 3.4 Extracting Translation Keys

Since Maud templates are Rust code, not separate template files, there is no
Maud-specific extraction tool. Strategies:

1. **Convention**: All `i18n.t()` calls use string literals as keys. The `cargo i18n`
   tool (rust-i18n) or a custom grep-based script can extract these.
2. **Compile-time validation**: With `fluent-templates`, missing keys at startup can be
   caught by iterating over a known key list and calling `try_lookup`.
3. **CI check**: A build step that loads all FTL files and checks for unused/missing keys
   against the codebase.

## 4. Accept-Language Header Handling

### 4.1 How Browsers Advertise Language Preference

Browsers send an `Accept-Language` header with quality values indicating preference:

```
Accept-Language: fi, en-US;q=0.9, en;q=0.8, ja;q=0.5
```

This means: prefer Finnish, then American English, then generic English, then Japanese.

### 4.2 The accept-language Crate

The `accept-language` crate provides parsing and negotiation:

```rust
use accept_language::{parse, intersection};

// Parse into ordered list of language tags
let preferred = parse("fi, en-US;q=0.9, en;q=0.8");
// => ["fi", "en-US", "en"]

// Intersect with supported languages
let supported = vec!["en-US", "fi", "ja"];
let matched = intersection("fi, en-US;q=0.9", supported);
// => ["fi", "en-US"]
```

### 4.3 Language Resolution Strategy for Saalis

The resolution order should be:

1. **User setting** (stored in user entity's settings/metadata): highest priority, the
   user has explicitly chosen their language.
2. **Accept-Language header**: automatic detection for anonymous users or users who
   haven't set a preference.
3. **Fallback locale**: `en-US` as the ultimate default.

In an axum handler:

```rust
async fn resolve_locale(
    user: Option<&User>,
    headers: &HeaderMap,
) -> LanguageIdentifier {
    // 1. Explicit user preference
    if let Some(user) = user {
        if let Some(lang) = user.setting::<Locale>() {
            return lang;
        }
    }

    // 2. Accept-Language negotiation
    if let Some(accept) = headers.get(ACCEPT_LANGUAGE) {
        if let Ok(header_str) = accept.to_str() {
            let supported = LOCALES.locales().collect::<Vec<_>>();
            // negotiate against available locales
            let negotiated = fluent_langneg::negotiate_languages(
                &parse_accepted_languages(header_str),
                &supported,
                Some(&langid!("en-US")),
                NegotiationStrategy::Matching,
            );
            if let Some(lang) = negotiated.first() {
                return (*lang).clone();
            }
        }
    }

    // 3. Fallback
    langid!("en-US")
}
```

The `fluent-langneg` crate (part of the fluent-rs workspace) implements the full Unicode
CLDR language negotiation algorithm, handling subtag matching (e.g., `en-GB` falling back
to `en`).

## 5. Right-to-Left (RTL) Language Support

### 5.1 Which Languages Are RTL?

Arabic, Hebrew, Persian (Farsi), and Urdu are the most common RTL languages. If saalis
supports Arabic or Hebrew UI translations, RTL layout must be handled.

### 5.2 The `dir` Attribute

Set `dir="rtl"` on the `<html>` element. This triggers automatic flipping of:

- Text alignment
- Table column order
- Flex and grid layout direction (if using `start`/`end` rather than `left`/`right`)

```html
<html lang="ar" dir="rtl">
```

There is also `dir="auto"`, which uses the Unicode Bidirectional Algorithm to detect
direction from content---useful for user-generated text in mixed-direction contexts.

### 5.3 CSS Logical Properties

Physical properties (`margin-left`, `padding-right`, `border-left`, `text-align: left`)
break in RTL. CSS Logical Properties replace them with direction-aware equivalents:

| Physical Property    | Logical Property          |
|---------------------|---------------------------|
| `margin-left`       | `margin-inline-start`     |
| `margin-right`      | `margin-inline-end`       |
| `padding-left`      | `padding-inline-start`    |
| `padding-right`     | `padding-inline-end`      |
| `border-left`       | `border-inline-start`     |
| `text-align: left`  | `text-align: start`       |
| `left`              | `inset-inline-start`      |
| `right`             | `inset-inline-end`        |
| `width`             | `inline-size`             |
| `height`            | `block-size`              |

Browser support for logical properties is excellent in all modern browsers.

### 5.4 Practical Approach

For Saalis, the safest strategy is:

1. **Use logical properties everywhere** in the CSS from the start. This is a
   write-once approach that works for both LTR and RTL without duplication.
2. **Set `dir` attribute dynamically** in the Maud layout based on the resolved locale.
3. **Test with `dir="rtl"` on the root** even before adding Arabic translations, to
   catch layout assumptions early.
4. **Bidirectional content**: Game titles in Arabic or Hebrew may appear within an LTR
   page or vice versa. Use `<bdi>` elements or `unicode-bidi: isolate` for
   user-generated content to prevent reordering issues.

## 6. Date/Time Formatting

### 6.1 The Problem

Date formats vary dramatically by locale:

- `en-US`: March 15, 2026 / 3/15/2026
- `fi`: 15.3.2026 / 15. maaliskuuta 2026
- `ja`: 2026年3月15日
- `de`: 15.03.2026 / 15. M\u00e4rz 2026

Using `chrono`'s `format()` with a hardcoded format string produces only one locale's
output.

### 6.2 ICU4X for Date/Time

ICU4X (`icu` crate) provides CLDR-based date/time formatting. As of ICU4X 2.0 (released
2025), the datetime module uses Unicode Semantic Skeletons:

```rust
use icu::datetime::{DateTimeFormatter, fieldsets};
use icu::locale::locale;
use icu::calendar::Date;
use icu::time::Time;

// Choose field set: Year-Month-Day, medium length
let formatter = DateTimeFormatter::try_new(
    locale!("fi").into(),
    fieldsets::YMD::medium(),
)?;

let date = Date::try_new_iso(2026, 3, 15)?;
let formatted = formatter.format(&date);
// => "15.3.2026"

// With time
let formatter_dt = DateTimeFormatter::try_new(
    locale!("ja").into(),
    fieldsets::YMD::long().with_time_hm(),
)?;
// => "2026年3月15日 16:09"
```

Field sets include `YMD` (year/month/day), `YMDT` (with time), `MD` (month/day),
and time-only sets. Length options are `short`, `medium`, `long`, `full`.

### 6.3 Binary Size Consideration

ICU4X bundles CLDR data. For a resource-constrained appliance (Batocera), use
`icu4x-datagen` to generate a data blob containing only the locales you ship, or use
`FixedCalendarDateTimeFormatter` to avoid linking all calendar systems (Saalis only needs
Gregorian).

### 6.4 chrono with Locale (Alternative)

The `chrono` crate can do basic locale formatting via the `chrono::Locale` feature, but
it covers far fewer locales and formatting options than ICU4X. For a project aiming at
correctness across many locales, ICU4X is the right tool.

## 7. Number Formatting

### 7.1 Locale Differences

| Locale | One million and a half |
|--------|----------------------|
| en-US  | 1,000,000.5          |
| fi     | 1 000 000,5          |
| de     | 1.000.000,5          |
| ja     | 1,000,000.5          |

Grouping separators, decimal marks, and digit grouping rules (Indian numbering uses
lakh/crore groups) all vary.

### 7.2 ICU4X DecimalFormatter

```rust
use icu::decimal::DecimalFormatter;
use icu::locale::locale;

let formatter = DecimalFormatter::try_new(
    locale!("fi").into(),
    Default::default(),
)?;

let value = FixedDecimal::from(1_000_007);
let formatted = formatter.format(&value);
// => "1 000 007" (Finnish uses non-breaking space as grouping separator)

// Bangla numerals
let formatter_bn = DecimalFormatter::try_new(
    locale!("bn").into(),
    Default::default(),
)?;
// => "১০,০০,০০৭" (Bangla digits, Indian grouping)
```

`DecimalFormatterPreferences` allows controlling grouping strategy (always, auto, min2)
and numbering system. ICU4X handles the correct Unicode characters for separators (e.g.,
narrow no-break space U+202F for French).

### 7.3 Recommendation

Use ICU4X for both date/time and number formatting. Since the `icu` crate is already
needed for dates, adding decimal formatting is zero additional dependency cost. This
avoids the fragmentation of using one library for dates and another for numbers.

## 8. Game Title Localization

### 8.1 The Problem

Game titles are not translatable strings---they are marketing-driven proper nouns that
differ by region for historical, legal, and cultural reasons:

| Region     | Title                          |
|-----------|-------------------------------|
| Japan      | バイオハザード (Biohazard)        |
| West       | Resident Evil                  |
| Japan      | ソニック・ザ・ヘッジホッグ            |
| West       | Sonic the Hedgehog             |
| Japan      | Mother 2                       |
| West       | EarthBound                     |
| Japan      | ゼルダの伝説 神々のトライフォース       |
| West       | The Legend of Zelda: A Link to the Past |
| Japan      | 逆転裁判                         |
| West       | Phoenix Wright: Ace Attorney   |
| Japan      | ぷよぷよ (Puyo Puyo)             |
| West (Gen) | Dr. Robotnik's Mean Bean Machine |

The renaming reasons include: religious reference removal (Triforce of the Gods ->
A Link to the Past), trademark conflicts (Biohazard was taken in the US), marketing
decisions, and complete rebranding for cultural fit.

### 8.2 Data Model Implications

This is NOT a UI i18n problem---it is a **metadata** problem. Game titles per region
should be stored as structured data in the entity metadata, not in `.ftl` files.

In saalis's entity/metadata model, this maps to:

```
Entity (game)
  └── Titles metadata
        ├── (region=JP, lang=ja) → "バイオハザード"
        ├── (region=US, lang=en) → "Resident Evil"
        ├── (region=EU, lang=en) → "Resident Evil"
        ├── (region=EU, lang=de) → "Resident Evil"   (same, but explicit)
        └── (region=JP, lang=en) → "Biohazard"       (romanized)
```

Key design points:

- **Region and language are separate axes.** A Japanese user might want to see Western
  game titles in English, or they might want Japanese titles. A Finnish user playing a
  Japanese import sees the Japanese title.
- **Primary/canonical title**: Each entity should have a designated display title resolved
  by: user's preferred title-region setting > user's language > platform region > original
  release title.
- **Search must index all title variants.** When a user searches "Biohazard", they should
  find Resident Evil and vice versa.
- **Sorting uses locale-aware collation.** Japanese titles sort by kana reading
  (五十音順), not by kanji code point. ICU4X provides `Collator` for this.

### 8.3 Integration with UI i18n

The UI i18n system handles *interface* strings ("Added to library", "Download", "Settings").
Game titles come from the metadata store. The rendering layer combines both:

```rust
fn game_card(i18n: &I18n, game: &Game) -> Markup {
    let title = game.display_title(&i18n.lang, &i18n.region);
    html! {
        div.card {
            h2 { (title) }   // from metadata, not from .ftl
            p { (i18n.t("game-card.added-on")) }  // from .ftl
        }
    }
}
```

## 9. UI String Management and Translation Workflow

### 9.1 Key Extraction from Maud Templates

Since Maud templates are Rust code, translation key extraction uses Rust-aware tooling:

1. **rust-i18n's `cargo i18n` CLI**: Scans for `t!()` macro invocations and generates
   skeleton locale files. Works for `rust-i18n` only.
2. **Custom extraction script**: For Fluent, a script that greps for `i18n.t("...")`
   patterns in `.rs` files and cross-references against `.ftl` files to find missing or
   unused keys.
3. **Compile-time validation**: At build time, load all FTL files and verify every key
   used in source exists in every locale. Fail the build on missing translations.
4. **Convention enforcement**: Namespace keys by component (`game-card.title`,
   `settings-page.language-label`, `nav.home`). This makes extraction and review easier.

### 9.2 Translation Workflow

A practical workflow for Saalis:

1. Developer adds new UI in Maud with `i18n.t("new-key")`.
2. Developer adds the English message to `locales/en-US/component.ftl`.
3. CI detects that other locales are missing `new-key` and flags it.
4. Translators add the key to their locale's FTL files.
5. PR merges when all required locales have the key (or the fallback is accepted).

### 9.3 Community Translation Platforms

**Weblate** (recommended for open source):
- Self-hostable, AGPL-licensed, free for libre projects.
- Native support for Fluent FTL files.
- Git-based workflow: Weblate monitors the repo, translators work in the web UI,
  translations are committed back as PRs.
- Supports translation memory, glossaries, and automatic suggestions.
- Quality checks for placeholders, formatting, length, and consistency.
- Over 1150 open-source projects use it (Fedora, phpMyAdmin, LibreOffice).

**Crowdin**:
- Proprietary SaaS, free for open-source projects.
- Supports FTL via a marketplace plugin ("Mozilla Project Fluent FTL File Translator").
- GitHub integration: auto-syncs source strings, delivers translated files as PRs.
- Larger community of professional translators.
- In-context translation (see strings in a preview of the UI).
- More polished UX, but closed-source and dependent on Crowdin's service availability.

**Recommendation for Saalis**: Weblate, because it can be self-hosted alongside the
project (consistent with the self-hosted/appliance philosophy), supports FTL natively,
and has no vendor lock-in. The Git-based workflow fits naturally with the existing
development process.

### 9.4 Translation File Organization

Recommended FTL file structure for Saalis:

```
locales/
  en-US/
    common.ftl          # nav, footer, shared UI
    game-library.ftl    # game list, cards, filters
    game-detail.ftl     # game detail page
    settings.ftl        # settings/preferences page
    errors.ftl          # error messages, 404, etc.
    notifications.ftl   # notification text
  fi/
    common.ftl
    game-library.ftl
    ...
  ja/
    common.ftl
    game-library.ftl
    ...
  core.ftl              # brand name terms, shared across locales
```

Split by UI component/page, not by feature domain. This makes it easy for translators to
work on one screen at a time and keeps file sizes manageable.

## 10. Synthesis: Recommended Stack for Saalis

| Concern                | Solution                        | Crate                    |
|------------------------|---------------------------------|--------------------------|
| UI string translation  | Project Fluent / FTL files      | `fluent-templates`       |
| Compile-time loading   | `static_loader!` macro          | `fluent-templates`       |
| Language negotiation   | CLDR-based matching             | `fluent-langneg`         |
| Accept-Language parse  | Header parsing + intersection   | `accept-language`        |
| Date/time formatting   | CLDR Semantic Skeletons         | `icu` (icu4x)            |
| Number formatting      | Locale-aware decimal            | `icu` (icu4x)            |
| Collation/sorting      | Locale-aware string comparison  | `icu` (icu4x)            |
| Game title resolution  | Metadata store, not i18n files  | Custom (entity metadata) |
| Translation workflow   | Git-integrated community tool   | Weblate (self-hosted)    |
| RTL support            | CSS logical properties + `dir`  | CSS (no crate needed)    |

### Key Design Decisions

1. **Fluent over rust-i18n**: Fluent's expressive grammar handling (select expressions,
   plurals, gender) is essential for Finnish, Japanese, and potential RTL languages.
   rust-i18n's simpler model would require awkward workarounds for non-trivial grammar.

2. **ICU4X over chrono locales**: ICU4X provides correct CLDR-based formatting for all
   target locales with a single dependency. Use `icu4x-datagen` to generate a trimmed
   data blob for only the locales shipped, keeping binary size reasonable for Batocera.

3. **Game titles are metadata, not i18n**: Title variants are stored per-entity in the
   metadata system with region and language dimensions. The UI i18n system is only for
   interface strings.

4. **CSS logical properties from day one**: Adopt logical properties (`inline-start/end`,
   `block-start/end`) in all CSS. This is zero-cost preparation for RTL that also
   produces cleaner, more semantic stylesheets.

5. **Weblate for community translations**: Self-hosted, FTL-native, Git-integrated. No
   vendor dependency, consistent with the project's self-hosted philosophy.
