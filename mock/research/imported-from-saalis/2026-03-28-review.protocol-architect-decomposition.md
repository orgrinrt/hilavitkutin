# Protocol Stack Decomposition Review

**Reviewer:** Protocol Architect (18 years; mesh networking, real-time comms, IETF WG)
**Date:** 2026-03-28
**Documents reviewed:** verkko-crypto, verkko-protocol, verkko-mesh, verkko-relay, verkko-seat, verkko-matter, verkko-taxonomy
**Scope:** Layering, message flow, state machine composition, upgrade independence, missing elements, seat gap

---

## 1. Layering Analysis

### 1.1 The decomposition is broadly correct

The six-document split follows a sound layering principle:

```
verkko-crypto          (primitives, no network concepts)
verkko-protocol        (point-to-point, no mesh concepts)
verkko-mesh            (collective coordination)
verkko-relay           (forwarding + storage membrane)
verkko-seat            (thin client tunnel)
verkko-matter          (distributed content tracking)
```

This mirrors the OSI model's discipline: each layer has a single concern, and dependencies flow strictly downward. The taxonomy's dependency graph is acyclic. Good.

### 1.2 Layer violations found

**Violation 1: verkko-crypto contains network-adjacent bootstrapping.**
Section 1.3 (genesis bootstrap) and 1.4 (device recovery) in verkko-crypto describe enrollment flows, QR codes, address hints, and multi-step join protocols. These are network operations that reference "any enrolled device," "Noise XX with PSK," and "admin signs enrollment cert." The abstract promises "no network concepts, no peers, no messages, pure cryptography," but sections 1.3 and 1.4 breach that promise. The key generation and self-test belong in crypto. The enrollment flow, QR code content, and join protocol belong in verkko-mesh. The founding transcript is a boundary object: its construction is crypto (hash of keys), but its use (mesh identity) is mesh.

**Recommendation:** Split genesis into two parts. verkko-crypto defines the key material and founding transcript hash. verkko-mesh defines the bootstrap state machine (self-enrollment, initial CRDT state, first heartbeat).

**Violation 2: verkko-crypto section 2.6 reaches into mesh scheduling.**
The PCS mechanism section references "polygon schedule (Section 3.6)" and "per-channel rate limiting," which are mesh-layer scheduling concepts. The crypto document should define the HKDF re-seed derivation and stop. The trigger scheduling belongs in verkko-mesh (and indeed verkko-mesh section 2.6 duplicates much of this). The current split has both documents partially specifying the same flow, which creates ambiguity about which is authoritative.

**Recommendation:** verkko-crypto section 2.6 should define only the `on_cpsk_renegotiation` derivation. All scheduling, rate limiting, and distribution references should be removed and live solely in verkko-mesh.

**Violation 3: verkko-protocol carries mesh-layer heartbeat semantics.**
The heartbeat payload (section 3.4) includes `filter_root`, `epoch_current`, `work_capacity`, `health_canary`, `observation_summary`, and `gossip_mode` flag. These are mesh-layer concepts embedded in the protocol-layer frame. The abstract says "just two peers exchanging frames over UDP," but the heartbeat payload encodes mesh topology awareness (filter roots), convergence state (gossip mode), and resource governance (work capacity).

This is not necessarily wrong. It is an optimization: carrying mesh state in protocol heartbeats avoids a separate mesh heartbeat. But it should be explicit. The protocol layer should define the heartbeat frame format with extension fields. The mesh layer should define what goes in those fields and how to interpret them. Currently verkko-protocol defines both format and semantics, which means changing mesh health semantics requires changing the protocol document.

**Recommendation:** verkko-protocol should define the heartbeat as a fixed header (state, rtt, loss, jitter, queue, bw, hlc) plus a variable-length extension region. verkko-mesh should define the extension fields (filter_root, epoch, work_capacity, health_canary, observation_summary). This preserves the layering without adding wire cost.

**Violation 4: verkko-relay's dependency on verkko-protocol is too thin.**
The taxonomy says verkko-relay depends on verkko-protocol for "frame forwarding." But the relay document defines its own storage operations (PUT, GET, DELETE, TOUCH, SIGNAL) without specifying how these map to verkko-protocol frames or streams. The relay has a dual identity: it is both a transit forwarder (protocol-layer concern) and a storage membrane (application-layer concern). The transit forwarding aspect is underspecified.

**Recommendation:** verkko-relay should explicitly define how relay operations map to verkko-protocol streams. Which stream type carries PUT? How is SIGNAL distinguished from a normal connection? Is there a relay-specific handshake?

### 1.3 Clean layering (positive findings)

- verkko-seat correctly isolates itself from verkko-mesh. The host peer translation boundary is the right abstraction.
- verkko-matter correctly depends on mesh, relay, and crypto without reaching past them into protocol internals.
- The Defined Concepts section acts as a genuine contract boundary. Documents reference each other only through defined concepts. This is excellent protocol architecture.

## 2. Message Flow Traces

### 2.1 New device joins the mesh

**Expected path:** crypto (key gen) -> protocol (connection) -> mesh (enrollment)

**Trace:**

1. New device generates DK + DEK (verkko-crypto: identity model). Clear.
2. Existing device generates QR code containing CIK pubkey, PSK, address hints, relay DHT key, founding hash, expiry, nonce (verkko-crypto: 1.4). This step is specified in crypto but belongs in mesh (layer violation noted above).
3. New device connects via Noise XX with PSK (verkko-protocol: Session). The PSK distribution (QR code) is not specified in verkko-protocol. Protocol only defines the Noise handshake mechanics.
4. Admin signs enrollment cert (verkko-mesh: Enrollment). verkko-mesh section 1.2 specifies the cert structure.
5. SAS verification within 24 hours (verkko-crypto: 1.4 mentions it, verkko-mesh: Enrollment Invariants require it).
6. KeyBundle(Grant) delivered after SAS (verkko-mesh: KeyBundle). Key delivery is SAS-gated.
7. Filter sync from connected clusters (verkko-matter: ContentFilter).
8. Post-entry checklist: filter sync, key delivery, control log replay, health init (verkko-mesh: 2.5).

**Gaps:**
- **Discovery gap.** How does the new device find the existing device's IP address? The QR code contains "address hints," but verkko-protocol's PeerDiscovery fallback chain (mDNS -> invite hint -> cached peers -> DHT -> relay) is defined only as invariants. The actual protocol messages for discovery are not specified. For a new device that has never connected, the only viable paths are mDNS (LAN) or the address hint in the QR code. The DHT and cached peers paths are unavailable. This is fine for LAN enrollment but creates a gap for remote enrollment.
- **SchemaSync gap.** verkko-protocol defines SchemaSync (protobuf schema exchange during handshake) but the enrollment flow never references it. When does the new device learn the mesh's protobuf schema? Before or after enrollment? This matters because enrollment messages presumably use protobuf.
- **Control log replay ordering.** Step 8 says "control log replay" but does not specify whether this happens before or after KeyBundle delivery. If the control log contains encrypted entries (wrapped to DEK), the device needs its keys first. If the control log is plaintext (signed but not encrypted), order does not matter. The control log structure (GSet, hash-linked, signed) is specified but its encryption state is not.

### 2.2 Key revocation and content re-encryption

**Expected path:** crypto (new key derivation) -> mesh (key distribution) -> relay (re-encryption) -> matter (filter update)

**Trace:**

1. Admin issues KeyBundle(Revoke) with fresh random keys for affected channels (verkko-mesh: KeyBundle). Delivered via reliable control channel.
2. Fencing token prevents conflicting operations (verkko-mesh: 2.3, fencing tokens).
3. EpochRatchet broken; fresh key injected, micro-epoch resets (verkko-crypto: 2.4).
4. Dual-key window disabled for revocation (verkko-crypto: 2.4). Immediate cutover.
5. Hash ring assigns re-encryption territory (verkko-relay: TerritorialReencryption).
6. Deterministic nonces computed: `BLAKE3("saalis-reenc-v1" || new_key || blob_id || epoch)[0..24]` (verkko-crypto: 4.1, verkko-relay: 4.4).
7. XChaCha20-Poly1305 re-encryption (verkko-crypto: StorageAEAD).
8. Idempotent PUT to relay (verkko-relay: IdempotentPUT).
9. Tiered deadlines: P1 4h, P2 24h, P3 bandwidth-derived, P4 no re-encryption (verkko-relay: 4.4).
10. Pre-batch nonce persist via double-write (verkko-crypto: NonceSafety).
11. Content filter does not change (matter_id = BLAKE3(plaintext) is invariant across re-encryption).

**Gaps:**
- **Revoked device's relay data.** What happens to matter stored on the relay that was encrypted under the old key by the revoked device? The re-encryption covers blobs owned by the revoking mesh, but if the revoked device had matter on a relay node, is that matter re-encrypted or abandoned? The relay sees opaque blobs keyed by `relay_key_id`. On revocation, the `relay_key_id` changes (because `channel_salt` rotates on revocation). The old `relay_key_id` blobs become orphaned. There is no explicit DELETE of old relay blobs after re-encryption. The 90-day TTL with TOUCH will eventually expire them, but that is a long window for potentially compromised content to persist.
- **Relay notification.** How does the relay know to expect new blobs under new `relay_key_id` values? The relay is a dumb membrane; it does not know about revocations. This is by design (trust boundary). But it means old blobs persist until TTL expiry. This should be explicitly acknowledged as a design trade-off.
- **Matter layer involvement.** The trace above does not involve verkko-matter except through content filters. But matter defines Stash (per-peer tracking). When a key is revoked, does the stash update? The stash maps matter_id to local path; matter_id does not change. But the encryption epoch changes, which might affect the stash's "last verified" timestamp. This interaction is unspecified.

### 2.3 Seat connects and requests content

**Expected path:** crypto (session encryption) -> protocol (connection, framing) -> seat (auth, request/response) -> [host peer internally consults mesh + matter]

**Trace:**

1. Seat authenticates with user credential (verkko-seat: SeatSession). Not a DK/DEK. Credential type unspecified.
2. Session encrypted via TransportAEAD (verkko-crypto).
3. Connection established via verkko-protocol (Connection, Session, Frame).
4. SchemaSync during handshake (verkko-protocol: SchemaSync). Seat gets protobuf definitions.
5. Seat sends SeatRequest with correlation ID (verkko-seat: SeatRequest). Access-controlled by role.
6. Host peer translates request to mesh query (internal to host peer; no document specifies this translation).
7. Host peer queries content via verkko-matter source selection (possession AND access AND reachability AND health).
8. Host peer initiates MatterTransfer if content not local (verkko-matter: MatterTransfer).
9. Host peer streams content to seat... how? Via SeatEvent? Via a dedicated stream?
10. Seat subscribes to event categories during session setup (verkko-seat: SeatEvent).

**Gaps (critical, all in verkko-seat's domain):**
- **Authentication protocol.** "User credential" is undefined. Is it a password? A token? A certificate? How is it transmitted? Is it part of the Noise handshake or a post-handshake message?
- **Session key derivation.** The seat does not have a DK/DEK. How does it establish a Noise session? Noise KK requires both parties to know each other's static key. Noise XX could work, but what is the seat's static key? Is the seat ephemeral-only? If so, this is a Noise NK pattern (initiator has no static key), which is not mentioned anywhere.
- **Content streaming.** No mechanism is defined for bulk data transfer from host peer to seat. SeatRequest/SeatResponse is request-response. SeatEvent is push notification. Neither is a streaming transfer protocol. For a 2GB ROM download, the seat needs a streaming mechanism, probably using verkko-protocol's Stream multiplexing, but this connection is not made.
- **Reconnection and state.** The seat document mentions "0-RTT with a stored resumption token" but does not define the token format, where it is stored, or how it interacts with key rotation on the host peer side.
- **Instruction acknowledgement.** SeatInstruction defines delegated operations (download, place, link, remove, verify) but does not define the acknowledgement protocol. Is it a separate SeatRequest? A special message type? What happens if the seat crashes mid-instruction?
- **Event subscription protocol.** "Seat subscribes to event categories during session setup" but no subscription message is defined. What are the event categories? How does the seat filter events?
- **Rate limiting and backpressure.** The seat can presumably send requests at any rate. What prevents a misbehaving seat from overwhelming the host peer? The host peer has LEDBAT and VitalQueue for mesh traffic, but seat traffic enters through a different path.

### 2.4 Partition heals

**Expected path:** protocol (reconnection) -> mesh (convergence) -> relay (re-encryption) -> matter (filter reconciliation)

**Trace:**

1. First CPSK reconnection starts wavefront timer (verkko-mesh: Convergence entry).
2. Gossip mode switches to flood (verkko-mesh: Gossip).
3. Metabolic override forces Active/Stress (verkko-protocol: MetabolicState).
4. Wavefront deadline expires or all expected pairs reconnect (verkko-mesh: 7.1).
5. Front 1: Control log merge (GSet union), PartitionAnnouncement, health EMA reset, observation replay, key retrieval, speculative P1 re-encryption (verkko-mesh: 7.1 Front 1).
6. Front 2: Revocations before grants, fencing token resolution, dominance recomputation, hash ring recomputation with ring coloring (verkko-mesh: 7.1 Front 2).
7. Edge-colored CPSK renegotiation via polygon schedule: 11+ slots, 2s spacing (verkko-mesh: 3.6).
8. Sealed-range catch-up before CPSK re-seed per pair (verkko-matter: SealedRange, verkko-mesh: 2.6).
9. PCS re-seed via HKDF on fresh CPSK (verkko-crypto: PCSMechanism).
10. Front 3: Bulk re-encryption under new keys, flash budget moderated, pre-batch nonce persist (verkko-relay: TerritorialReencryption, verkko-crypto: NonceSafety).
11. Filter reconciliation via Merkle tree walk (verkko-matter: MerkleReconciliation). ~720 KB for 36 pairs.
12. Exit transition: 6-step gated sequence (verkko-mesh: 7.1 Exit).
13. Total: 37-43 seconds.

**Gaps:**
- **Filter reconciliation timing.** Step 11 (filter reconciliation) is listed as "concurrent" in the convergence burst cost table but is not explicitly sequenced relative to the three fronts. Does it run during Front 1? After Front 2? The Merkle root is in every heartbeat; divergence is detected immediately. But when does the reconciliation data exchange happen? During flood mode (high reliability) or after exit to Plumtree (lower overhead)? This matters because filter reconciliation is 720 KB, the largest single component of convergence cost.
- **Seat notification during partition heal.** If a seat is connected to a host peer when a partition heals, the host peer enters convergence. Does the seat see anything? Is the seat's traffic deprioritized? Is the seat notified that the mesh is reconverging? The seat does not know about the mesh, so it would see degraded performance without explanation. This is a UX gap that should be acknowledged even if it is not solvable at the protocol level.
- **Relay during partition.** During a partition, relay peers in the partitioned network may have stale `relay_key_id` salts (90-day rotation, but revocation triggers immediate rotation). If one partition revokes a device and the other does not, the relay state diverges. On heal, the revocation propagates (revoke-wins), but the relay blobs encrypted under the non-revoked key in the other partition need re-encryption. This flow is covered by territorial re-encryption, but the interaction between partition-scoped relay state and convergence is implicit, not explicit.

## 3. State Machine Composition Analysis

### 3.1 Identified state machines

| Document | State Machine | States |
|----------|--------------|--------|
| verkko-protocol | MetabolicState | Background, Idle, Active, Stress |
| verkko-protocol | OperationalMode | (implied: Normal, Converging) |
| verkko-mesh | ConvergenceState | Entry, Front1, Front2, Front3, ExitStep0-6, Done |
| verkko-mesh | SourceBreaker | Closed, HalfOpen, Open |
| verkko-mesh | KeyLifecycle | Active, Orphaned |
| verkko-mesh | GossipMode | Plumtree, Flood |
| verkko-crypto | EmptyLog/ActiveLog | EmptyLog, ActiveLog |
| verkko-matter | SealedRange | FastForward, ActiveWithSealedRange, Suspended |

### 3.2 Composition issues

**Issue 1: MetabolicState and ConvergenceState sequencing.**
The design specifies a sequencing invariant: "OperationalMode::transition_to(ConvergenceEntry) requires metabolic_state in {Active, Stress}." The metabolic override suspends dwell time for this one transition. This is well-specified. However, the exit transition (Step 3) restores dwell time enforcement. Between Steps 2 (spanning tree construction) and Step 5 (health baseline update), the metabolic state machine is live with dwell time enforcement but the mesh is still in a partially converged state. If hardware pressure forces the metabolic SM to Background during this window, heartbeat intervals lengthen to 30s. Step 5 requires "3 heartbeat rounds at Active interval," but if the SM has moved to Background, this takes 90s instead of 15s. This is a potential slow convergence exit path that is not acknowledged.

**Recommendation:** Step 3 should restore dwell time but pin a minimum metabolic state of Idle until Step 6 completes. This ensures convergence exit completes in bounded time regardless of hardware pressure.

**Issue 2: GossipMode and ConvergenceState tight coupling.**
GossipMode transitions are gated by ConvergenceState. Enter convergence -> flood. Exit Step 1 -> Plumtree. But the anomalous-message detection also triggers local flood independently of convergence state. This means a peer can be in GossipMode=Flood while ConvergenceState=Done. The flaky gateway dampening addresses the oscillation risk, but the state space is larger than it first appears: {(ConvergenceState x GossipMode x MetabolicState)} has valid but surprising corners.

Specifically: a peer exits convergence, switches to Plumtree (Step 1), then immediately receives an anomalous eager message and switches back to flood. ConvergenceState is Done, but GossipMode is Flood. The system handles this correctly (the peer is in local flood, not mesh-wide convergence), but the distinction between "convergence flood" and "anomalous flood" is implicit. They have different exit conditions: convergence flood exits via the 6-step sequence; anomalous flood exits via the dampening timer.

**Recommendation:** Name the two flood modes explicitly. ConvergenceFlood (mesh-wide, gated by convergence exit) and AnomalousFlood (local, gated by dampening timer). They share the same wire behavior but have different lifecycle management.

**Issue 3: SealedRange and KeyLifecycle interaction.**
SealedRange has three states: FastForward, ActiveWithSealedRange, Suspended. KeyLifecycle has two states: Active, Orphaned. When a peer reconnects after a revocation, its channels enter KeyLifecycle=Orphaned (old key) and need a CatchUpGrant with the new key. Simultaneously, the peer enters SealedRange state (either ActiveWithSealedRange or Suspended depending on offline duration). The SealedRange defines that CatchUpGrant "implicitly unseals" for Revocation sealed ranges. But this means the KeyLifecycle transition (Orphaned -> Active via CatchUpGrant) and the SealedRange transition (ActiveWithSealedRange -> Active) are triggered by the same message. These coupled transitions are not explicitly documented as a compound state transition.

**Recommendation:** Document the compound transition: receiving CatchUpGrant simultaneously transitions KeyLifecycle from Orphaned to Active AND SealedRange from sealed to unsealed. This is currently implicit and correct, but should be explicit to prevent implementors from ordering these transitions incorrectly.

**Issue 4: CircuitBreaker and ConvergenceState interaction.**
During convergence, the health EMA is reset for partition-absent peers (Front 1, Step 3). This reset could trigger CircuitBreaker state changes. A peer that was Open (excluded) before the partition might have its EMA reset to a healthier value, causing a transition to HalfOpen. But the two-gate probe condition (Recovering AND variance below threshold) might not be met immediately after a reset. The interaction between "EMA reset during convergence" and "circuit breaker state evaluation" is unspecified. Does the breaker re-evaluate immediately on EMA reset? Does it wait for new observations?

**Recommendation:** Specify that CircuitBreaker state is re-evaluated after EMA resets during Front 1. If the reset value satisfies the two-gate condition, the breaker transitions. If not, it waits for post-convergence observations.

## 4. Upgrade Independence Assessment

### 4.1 What can be upgraded independently

| Document | Can change without touching others? | Condition |
|----------|-------------------------------------|-----------|
| verkko-crypto (AEAD construction) | Yes, if TransportAEAD invariants hold | Key size, auth tag size must remain the same |
| verkko-crypto (ratchet internals) | Yes, if EpochRatchet invariants hold | HKDF can be replaced if forward secrecy preserved |
| verkko-protocol (LEDBAT tuning) | Yes | Metabolic state mapping is the only interface |
| verkko-protocol (wire format) | No | Everyone references the 55-byte frame overhead |
| verkko-mesh (gossip algorithm) | Partially | Plumtree can be replaced if mixing time guarantees hold, but heartbeat payload carries gossip_mode flag |
| verkko-mesh (dominance formula) | Yes, if deterministic-from-public-state invariant holds | Formula can change; argmin contract is stable |
| verkko-relay (hash ring) | Yes, if territorial completeness invariant holds | Ring algorithm is internal |
| verkko-relay (storage ops) | No | PUT/GET/DELETE/TOUCH are referenced by verkko-matter |
| verkko-seat | Yes | No other document depends on it |
| verkko-matter (filter type) | Partially | Cuckoo filter internals can change; Merkle root size in heartbeat cannot |

### 4.2 Problematic coupling

**Coupling 1: Heartbeat payload format.**
The heartbeat payload is defined in verkko-protocol (section 3.4) with fixed byte offsets. The fields include mesh-layer semantics (filter_root, epoch_current, observation_summary). Any change to mesh health semantics (adding a field, changing field width) requires changing verkko-protocol's wire format. This couples mesh evolution to protocol evolution.

**Coupling 2: BLAKE3 domain separator registry.**
The registry lives in verkko-crypto but has entries from multiple documents (relay key ID, metabolic jitter, heartbeat jitter, founding hash, etc.). Adding a new domain separator in any document requires updating the crypto document's registry. This is intentional (collision prevention) but creates a coordination cost.

**Coupling 3: Convergence references across four documents.**
The convergence flow (verkko-mesh: 7.1) references: verkko-crypto (PCSMechanism, NonceSafety), verkko-protocol (MetabolicState, VitalQueue), verkko-relay (TerritorialReencryption, IdempotentPUT), verkko-matter (SealedRange, MerkleReconciliation). This is the most coupled flow in the system. Changing any of these mechanisms could affect convergence timing. The 37-43 second convergence budget is a cross-document constraint.

**Coupling 4: Nonce safety crosses crypto and protocol.**
The three-layer nonce defense is defined in verkko-crypto, but the NonceSource range partitioning ([0, 2^95) for event loop, [2^95, 2^96) for re-encryption) is specified in verkko-protocol (section 6.2). The double-write persist is crypto; the thread that calls persist is protocol. This is acceptable coupling (crypto defines the mechanism, protocol defines the threading), but a change to the threading model in protocol could violate nonce safety invariants.

### 4.3 Upgrade independence verdict

The system has reasonable upgrade independence at the extremes (crypto primitives, seat) but tight coupling at the convergence flow. This is inherent to protocol stacks: the interesting work happens at layer boundaries, and convergence crosses all of them.

The Defined Concepts pattern provides the right abstraction for evolution. As long as invariants are preserved, internals can change. The risk is that the invariants are too specific in some cases (e.g., heartbeat payload byte offsets) and too vague in others (e.g., "session encrypted via TransportAEAD" for seats).

## 5. Missing Protocol Elements

### 5.1 Version negotiation

No document specifies how peers negotiate protocol versions. The wire format has no version field in the outer header. The BroadcastEnvelope has a `version: u8` field, but the point-to-point frame does not. When verkko-protocol v2 ships with a different wire format, how do v1 and v2 peers coexist?

**Minimum required:**
- Version field in the outer frame header (1 byte).
- Version negotiation during Noise handshake (carry supported versions as handshake payload).
- Graceful rejection of unsupported versions with an actionable error.

### 5.2 Capability advertisement

No document specifies how peers advertise their capabilities. Examples:
- "I support XChaCha20 but not AES-GCM."
- "I have 50GB of relay storage."
- "I can relay but not store."
- "I support bao-tree streaming verification."

The system assumes all peers are homogeneous. This is reasonable for v1 (all same binary), but becomes a problem at the first protocol upgrade.

**Minimum required:**
- Capability set exchanged during session establishment.
- Capability-aware source selection in verkko-matter.
- Capability-aware relay selection in verkko-relay.

### 5.3 Error handling at the protocol level

No document defines a general error frame or error code registry. Individual mechanisms have error paths (NoncePersistFailed, SAS timeout, enrollment rejection), but there is no protocol-level error message that one peer can send to another to say "I could not process your last frame because X."

**Minimum required:**
- Error frame type in verkko-protocol's msg_type byte.
- Error code registry (u16 or u32) with categories: crypto error, protocol error, mesh error, matter error.
- Defined behavior for unrecognized error codes (log and continue, or disconnect).

### 5.4 Graceful shutdown

No document specifies how a peer announces it is going offline intentionally (as opposed to crashing). An intentional shutdown could:
- Transfer gateway role before disconnecting.
- Hand off territorial re-encryption ownership.
- Notify connected seats.
- Suppress false scar generation on peers that observe the disconnect.

Currently, an intentional shutdown looks identical to a crash from other peers' perspectives. The health pipeline will generate scars, the circuit breaker will open, and convergence will trigger. All of this is unnecessary overhead for a planned departure.

**Minimum required:**
- Graceful shutdown message (GOODBYE frame).
- Gateway pre-transfer on shutdown.
- Scar suppression for announced departures.

### 5.5 Backpressure signaling

The LEDBAT congestion control handles bandwidth-level backpressure, and the VitalQueue handles prioritization. But there is no application-level backpressure signal. If a peer's pending buffer is full, or its flash budget is exhausted, or it is mid-convergence and cannot accept new work, it has no way to tell senders "slow down" at the application layer.

The congestion feedback in SACK ACKs (recv_buffer_pct, recv_rate_kbps) is transport-level. Application-level backpressure (e.g., "do not send me more KeyBundle rotations for 30 seconds") is absent.

### 5.6 Diagnostic and debugging protocol

No document defines how operators diagnose mesh problems remotely. There is structured logging (section 6.4) and stability metrics, but no protocol-level diagnostic exchange. A remote admin cannot query a peer's health pipeline state, circuit breaker state, convergence state, or nonce velocity without SSH access.

For a household appliance (Batocera), this is a significant operational gap. The seat could serve as the diagnostic channel (seat requests for diagnostic data), but this is not specified.

## 6. Seat Protocol Requirements

verkko-seat is a skeleton with invariants but no protocol. From a protocol architect's perspective, here is what it needs.

### 6.1 Session establishment

**Authentication:** The seat needs a Noise pattern that works without a device key. Two options:
1. **Noise NK** (server has static key, client is ephemeral). The seat has no persistent identity. Server authenticates via CIK or a seat-specific key derived from the user credential. Simple but no 0-RTT.
2. **Noise KK with derived seat key.** The host peer derives a seat static key from the user credential (e.g., `HKDF(credential, "seat-static-key")`). The seat stores this key. Enables 0-RTT resumption. More complex but supports resumption.

**Recommendation:** Use Noise NK for initial connection. On first successful authentication, the host peer issues a resumption token (a signed, time-limited credential). On reconnection, the seat uses Noise KK with the resumption token as PSK. This gives both simplicity (first connection) and performance (subsequent connections).

**User credential:** Define a credential type enum: password-derived (Argon2id), TOTP, or platform credential (biometric via WebAuthn). The credential is transported inside the Noise handshake payload (encrypted under the handshake's symmetric key).

### 6.2 Message types

Minimum message set:

| Type | Direction | Purpose |
|------|-----------|---------|
| AUTH | seat -> host | User credential for authentication |
| AUTH_RESULT | host -> seat | Success/failure with session token |
| SUBSCRIBE | seat -> host | Event category subscription |
| REQUEST | seat -> host | Query or operation request |
| RESPONSE | host -> seat | Response to a request |
| EVENT | host -> seat | Pushed event notification |
| INSTRUCTION | host -> seat | Delegated operation |
| INSTRUCTION_ACK | seat -> host | Instruction completion/failure |
| PROGRESS | seat -> host | Progress on long operation |
| STREAM_START | host -> seat | Begin bulk transfer (content download) |
| STREAM_DATA | host -> seat | Chunk of bulk transfer |
| STREAM_END | host -> seat | End bulk transfer |
| PING/PONG | both | Liveness (lighter than mesh heartbeat) |
| GOODBYE | both | Graceful disconnect |

### 6.3 Session state machine

```
Disconnected
    |  [connect]
    v
Handshaking  (Noise NK or KK with resumption)
    |  [auth success]         |  [auth failure]
    v                         v
Subscribing                Disconnected
    |  [subscriptions confirmed]
    v
Active
    |  [idle timeout]     |  [host going offline]     |  [explicit disconnect]
    v                     v                            v
Suspended              Disconnected                 Disconnected
    |  [resume]
    v
Active
```

**Key properties:**
- Suspended state stores the resumption token. The seat can reconnect without re-authenticating for a configurable window (e.g., 24 hours).
- If the host peer goes offline, the seat is immediately disconnected. No failover to another host peer (this is a design choice documented in verkko-seat).
- Multiple seats can be in Active state simultaneously on the same host peer.

### 6.4 Stream multiplexing for seats

The seat connection should use verkko-protocol's Stream multiplexing with at least three streams:
- Stream 0: Control (AUTH, SUBSCRIBE, PING/PONG, GOODBYE)
- Stream 1: Request/Response (REQUEST, RESPONSE, INSTRUCTION, INSTRUCTION_ACK, PROGRESS)
- Stream 2+: Bulk transfer (STREAM_START, STREAM_DATA, STREAM_END), one stream per concurrent transfer

This prevents a large content download from blocking control messages or request/response traffic (no head-of-line blocking, as promised by the Stream invariants in verkko-protocol).

### 6.5 Access control model

The seat has a role (Admin, User, Child). The host peer maps this to mesh-level access control. Define:

| Role | Allowed operations |
|------|-------------------|
| Admin | All queries, all operations, diagnostic data, mesh configuration |
| User | Content queries, download requests, inventory sync, personal settings |
| Child | Content queries (filtered by parental controls), download requests (pre-approved only) |

The access control check happens at the host peer before executing any request. Unauthorized requests receive a RESPONSE with an access-denied error code.

### 6.6 Event categories

Minimum event categories for subscription:

| Category | Events |
|----------|--------|
| content | New content available, content removed, content status change |
| download | Download started, progress, completed, failed |
| mesh | Peer joined, peer left, convergence started, convergence completed |
| system | Flash budget warning, CPU pressure, connectivity change |

The seat subscribes during the Subscribing phase. Event categories can be changed during Active state via a SUBSCRIBE message.

## 7. Recommended Changes

### 7.1 Critical (protocol coherence requires these)

1. **Split genesis bootstrap.** Move enrollment flow, QR code, and join protocol from verkko-crypto sections 1.3-1.4 to verkko-mesh. Keep key generation and founding transcript hash in crypto.

2. **Add version negotiation.** Add a version byte to the outer frame header. Define version exchange during Noise handshake. This is a prerequisite for any future protocol evolution.

3. **Define error frames.** Add an error message type to verkko-protocol with an error code registry. Without this, peers have no way to communicate processing failures.

4. **Specify seat authentication protocol.** Define the Noise pattern (NK for first contact, KK with resumption for subsequent), credential types, and session state machine. Without this, verkko-seat is not implementable.

### 7.2 High (prevents implementation ambiguity)

5. **Separate heartbeat format from semantics.** verkko-protocol should define heartbeat as fixed header + extension region. verkko-mesh should define extension field contents. This decouples mesh evolution from protocol wire format.

6. **Specify relay operation framing.** verkko-relay should define how PUT/GET/DELETE/TOUCH/SIGNAL map to verkko-protocol streams or message types.

7. **Specify filter reconciliation timing.** verkko-mesh convergence sequence should explicitly place filter reconciliation (currently "concurrent") in the timeline. State when it starts, what gates it, and when it must complete.

8. **Name flood modes.** Distinguish ConvergenceFlood from AnomalousFlood in verkko-mesh. Same wire behavior, different lifecycle.

9. **Add graceful shutdown.** Define GOODBYE frame in verkko-protocol. Define pre-departure gateway transfer in verkko-mesh. Define scar suppression for announced departures.

### 7.3 Medium (improves robustness)

10. **Pin minimum metabolic state during convergence exit.** Ensure metabolic SM stays at Idle or above between convergence exit Step 3 and Step 6.

11. **Document compound state transitions.** Explicitly document that CatchUpGrant triggers both KeyLifecycle and SealedRange transitions simultaneously.

12. **Specify CircuitBreaker re-evaluation after EMA reset.** Define whether breaker state is re-evaluated immediately on convergence EMA reset.

13. **Add capability advertisement.** Define capability set exchange during session establishment for forward compatibility.

14. **Define old relay blob cleanup.** Specify whether old `relay_key_id` blobs are explicitly DELETEd after re-encryption or left to TTL expiry. Acknowledge the trade-off.

15. **Define seat diagnostic channel.** Specify that Admin-role seats can query mesh health, convergence state, and resource governance state. This is the operational visibility path for household appliances.
