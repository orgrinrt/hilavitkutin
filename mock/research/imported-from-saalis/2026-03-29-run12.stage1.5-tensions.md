## Tension 1: SansIO Polling Convention -- Synchronous vs Two-Phase vs Arena-Return

**Name:** The Polling Contract Incompatibility

### Conflicting positions

**Expert B** proposes a four-method protocol trait modeled on quinn:

> "Consumer calls `protocol.handle(input: ProtocolInput, arena: &Arena)`
> which processes one input event and mutates internal state. Consumer
> calls `protocol.poll_transmit(arena: &Arena)` in a loop to drain all
> pending transmissions. Consumer calls `protocol.poll_timeout()` to get
> the next timer deadline. Consumer calls `protocol.poll_event()` in a
> loop to drain all pending application-level events."

Expert B ties arena lifetime to the arena reference parameter: the caller
passes the arena into each method, borrows out slices, and must drain all
outputs before calling `arena.reset()`. The handle/poll_transmit/
poll_timeout/poll_event split mirrors quinn's four-phase calling convention.

**The Savant** proposes a two-method feed/poll pattern with a separate
deadline query:

> "trait VerkkoProtocol { fn feed(&mut self, input: ProtocolInput) ->
> usize; fn poll_output(&mut self) -> Option<ProtocolOutput>;
> fn next_deadline(&self) -> Option<Instant>; }"

The Savant's model collapses transmit outputs and application events into
a single `poll_output()` drain, rather than separating them by type. Buffer
ownership differs too: "SendBytes data is a slice into the hot arena. The
slice is valid until the next call to feed()." The consumer copies or sends
before the next feed, not before an explicit arena.reset(). The Savant also
handles arena exhaustion via a new input variant: `ProtocolInput::ArenaProvided
{ base: *mut u8, len: usize }`, whereas Expert B treats arena as an explicit
parameter to every method.

**Expert B on timer management** (related part of the same split):

> "Replace the SetTimer/TimerExpired pair with a single poll_timeout()
> method on the Protocol trait [...] The protocol manages all timers
> internally. The consumer only needs to track one deadline."

Expert B further proposes `fn handle_timeout(&mut self, now_ms: u64,
arena: &ArenaRef) -> Result<(), ProtocolError>` as a dedicated timeout
entry point.

**The Savant** handles this with `fn next_deadline(&self) -> Option<Instant>`
and implicit timer cancellation: "SetTimer implicitly cancels any existing
timer with the same timer_id. CancelTimer is not needed as a separate input."

### Why this tension matters

The polling contract is the API surface that everything above the protocol
layer builds on. Expert B's four-method split provides type-level separation
between transmission outputs (which need to be sent to the OS socket) and
application events (which need to be dispatched to the mesh layer). Collapsing
them into one `poll_output()` (Savant) means the consumer must branch on the
output variant before knowing whether to call `sendto()` or dispatch a mesh
event. At 10,000 frames/second, that branch is in the hot path.

On the other hand, the Savant's two-phase (feed/poll) design explicitly
prevents reentrancy by making it structurally impossible to call feed() from
within output processing, since poll_output() does not call back into the
consumer. Expert B's design must state the reentrancy invariant as a prose
contract rather than enforcing it at the type level.

The arena lifetime rule also differs: Expert B's "valid until arena.reset()"
gives the consumer a single explicit reset point; the Savant's "valid until
the next feed()" makes every input call a potential invalidation point, which
is simpler but less flexible for consumers that want to hold arena-backed
slices across multiple feed iterations.

### Question for Stage 2

Should the polling interface be a typed four-method split (transmit drain,
event drain, timeout query, handle) in the style of quinn and Expert B, or
a unified two-phase feed/poll with a single output stream in the style of
the Savant? The answer determines buffer ownership semantics, timer
management, and whether reentrancy is type-enforced or contract-enforced.
Which design property -- type-level separation of output classes, or
structural reentrancy prevention -- is the higher priority for verkko?

## Tension 2: Relay-First vs Hole-Punch-First NAT Traversal

**Name:** The Connectivity Strategy Inversion

### Conflicting positions

**Expert C** recommends inverting the design's stated approach:

> "I recommend changing the stated approach from 'hole punching attempted
> before relay fallback' to 'relay-first, then attempt hole punching in
> parallel.' Rationale: relay-first guarantees connectivity within ~500ms
> (one RTT to relay). Hole punching can take 2-10 seconds and may fail
> entirely. For a household mesh where users expect instant connectivity,
> relay-first is correct. [...] The transient relay load is negligible."

Expert C cites Tailscale's DERP model as prior art: always connect to relay
first, then attempt direct connections in the background, migrate to direct
if hole punching succeeds.

**Expert C explicitly labels this a known limitation of his own recommendations:**

> "Relay-first vs hole-punch-first. I recommend relay-first. The design
> currently specifies hole-punch-first. This is a design philosophy question,
> not a technical one. Both work. My recommendation optimizes for
> user-perceived latency at the cost of higher relay load during the first
> seconds of a session."

**Expert A (indirect position)** does not address NAT traversal directly,
but the convergence partition-heal sequence she specifies assumes that
inter-cluster connectivity is established before convergence begins. The
convergence wavefront timer starts "on reconnection." If relay-first means
connectivity is always established within ~500ms, the wavefront timer can
fire reliably. If hole-punch-first means connectivity takes 2-10 seconds
and sometimes fails, the convergence trigger is less reliable.

**The Savant** specifies relay-first behavior in the synthesis section:

> "Relay failover: when a relay stops responding (3 missed keepalive probes
> at 70% of mapping lifetime), the gateway queries the DHT for alternative
> relays and initiates new SIGNAL(HELLO) exchanges. The connection_id is
> preserved across relay migration."

The Savant does not explicitly address hole-punch-first vs relay-first,
but the synthesis describes a relay as always-present infrastructure (with
health monitoring, warm standby, failover), not as a last resort.

**Expert B** raises a consequential interaction: the sans-IO state machine
must model the NAT traversal state as a protocol concern. If relay-first is
adopted, the state machine always starts with a relay path and upgrades to
direct. If hole-punch-first is adopted, the state machine starts with direct
candidate checking and falls back. These produce different state machine
topologies and different timer arrangements (connectivity check timeout vs
relay registration timer).

### Why this tension matters

The design's stated invariant (TransitRelay invariant 3: "transit relay is
last resort after hole punching fails") directly contradicts Expert C's
recommendation. This is not a gap; it is a design choice that Expert C is
asking the review to reconsider. The choice has downstream consequences:

1. If relay-first, relay load is higher during session establishment but
   lower after direct upgrade. The relay resource governance (Expert C's
   forwarding table limits, bandwidth caps) must handle bursts from all
   new sessions.

2. If hole-punch-first, relay load is lower but initial connectivity latency
   is higher (2-10 seconds). Expert A's convergence sequences depend on
   inter-cluster sessions being available promptly.

3. Expert C identifies that at 40% symmetric NAT prevalence, a significant
   fraction of peers will never successfully hole-punch. For these peers,
   hole-punch-first means 2-10 seconds of failed checks before relay is
   used on every new session establishment.

### Question for Stage 2

Should the design adopt relay-first with direct-upgrade (Expert C's
recommendation, aligned with Tailscale's model) or preserve hole-punch-first
with relay-fallback (the current design invariant)? Expert A must address
whether the relay-first model provides the connectivity stability the
convergence wavefront timer needs. Expert B must address whether the state
machine topology change is significant.

## Tension 3: Intra-Cluster Key Choice -- CIK vs DEK for Noise KK

**Name:** The Intra-Cluster Session Key Dispute

### Conflicting positions

**Expert A** recommends DEK-based Noise KK for intra-cluster sessions:

> "I recommend DEK-based Noise KK for intra-cluster sessions, with one
> session per peer pair within the cluster. This provides: per-pair forward
> secrecy (compromise of one session does not affect others); independent
> session lifecycle (one peer restarting does not invalidate other sessions);
> compatibility with the existing Noise framework. [...] Static key: DEK
> (not CIK). Reason: CIK is shared, so CIK-based sessions would all be
> compromised by any single peer compromise."

**The Savant** reaches the same conclusion but includes a critical observation
about why CIK cannot work at all, not just why DEK is preferable:

> "Noise KK with identical static keys on both sides degenerates: the
> handshake pattern produces k=DH(CIK, CIK_ephemeral) which is the same
> for both directions. This is a fundamental problem. The correct approach:
> intra-cluster sessions use Noise KK with each peer's DEK as the static
> key."

Both Expert A and the Savant agree on DEK. The tension is elsewhere.

**Expert A's intra-cluster model** proposes DEK-based Noise KK with one
session per peer pair, no polygon scheduling (LAN RTT is negligible), and
gateway routing via dest_peer lookup table. She also specifies what happens
on gateway change:

> "Old gateway transfers CPSK session state to new gateway. This requires
> an intra-cluster reliable transfer. Use vital tier stream within the
> intra-cluster session. [...] New gateway establishes inter-cluster sessions.
> Old gateway's inter-cluster sessions are torn down. In-flight frames may
> be lost during the handoff window."

**The Savant's synthesis** describes a different heartbeat model for
intra-cluster sessions:

> "Intra-cluster heartbeats run at a fixed 1-second interval (LAN latency
> is negligible; the variable interval is for WAN conservation). The
> heartbeat carries the same transport header as inter-cluster heartbeats
> but uses a simplified 8-byte extension region: filter_root (8 bytes)
> only."

The Savant also specifies simplified health scoring for intra-cluster:
"No dual-EMA. A single heartbeat counter. 3 missed intra-cluster heartbeats
(3 seconds) triggers a local-only health degradation."

**Expert B** declines to address intra-cluster communication as outside his
domain: "Outside my domain -- no comment." But his sans-IO polling model
has a direct implication: the protocol state machine treats all connections
uniformly (BytesReceived does not distinguish intra- from inter-cluster).
This means the mesh layer must classify connections by type after receiving
them from the state machine, applying different heartbeat intervals and
resource governance to intra-cluster vs inter-cluster sessions.

**Expert A** does not address the resource accounting gap the Savant
identifies: "intra-cluster sessions use a separate ThresholdGuard instance
with a 10KB arena allocation per peer (60KB maximum for a 6-peer cluster).
This is added to the managed arena baseline (200KB + 60KB = 260KB total)."
Expert A's proposal implies a similar resource impact (5 sessions per peer
in a 6-peer cluster) but does not provide the accounting.

### Why this tension matters

The DEK vs CIK question is resolved by both (DEK wins), but the handoff
semantics create a genuine incompatibility. Expert A says in-flight frames
may be lost during gateway handoff, with heartbeat failure detection as the
recovery trigger. The Savant says connection_id is preserved across relay
migration (for relay failover, §4.2) but does not specify whether
intra-cluster connection_ids survive gateway handoff. If the connection is
torn down (Expert A's model), the VitalQueue is flushed and any in-flight
KeyBundle distributions or CIK rotations during handoff are lost. If the
connection migrates (connection_id preserved, Savant's relay model extended
to intra-cluster), the KeyBundle delivery is not interrupted.

This matters because Expert A identifies CIK rotation -> intra-cluster
distribution -> CPSK renegotiation as a critical ordering: "the ordering
between 'distribute new CIK intra-cluster' and 'renegotiate CPSK
inter-cluster' is unspecified." A gateway handoff that tears down intra-cluster
connections mid-CIK-rotation could leave the cluster in a state where some
peers have the new CIK and some do not, while the old gateway's CPSK sessions
are already torn down.

### Question for Stage 2

Should gateway handoff tear down and re-establish intra-cluster Noise KK
sessions (Expert A's model, simpler but potentially disruptive to in-flight
critical key material), or should connection_id be preserved across gateway
handoff (Savant's model, consistent with relay migration, but requires the
new gateway to take ownership of existing session state)? Expert B must
address how the sans-IO state machine models the gateway handoff event --
is it a connection migration (PATH_CHALLENGE) or a connection close followed
by re-establishment?

## Tension 4: Replication Orchestration -- Independent Probabilistic vs Coordinated Pull

**Name:** The Thundering Herd vs Coordination Tradeoff

### Conflicting positions

**Expert A** proposes a pull-based anti-entropy replication protocol with
probabilistic thundering herd mitigation:

> "Each peer independently evaluates replication state from its local view.
> Multiple peers may initiate redundant transfers; the content-addressed
> nature of matter makes this idempotent."

Expert A's thundering herd mitigation is BLAKE3-based probabilistic selection:

> "fn should_replicate(blob_id, known_holders, target_replicas) -> bool {
> [...] let probability = deficit as f64 / (known_holders as f64 + 1.0);
> let roll = BLAKE3('saalis-repl-v1' || blob_id || my_dk || epoch)[0]
> as f64 / 255.0; roll < probability }"

The key property: "Deterministic per (blob_id, my_dk, epoch) to prevent
re-evaluation jitter." Different peers roll different values and only a
fraction of eligible peers actually initiate replication.

**The Savant** proposes a different orchestration model that does not use
explicit thundering herd mitigation but instead uses a priority queue:

> "for _ in 0..replication_budget: matter = most_under_replicated();
> if matter.replica_count < matter.min_replicas: target =
> select_placement_target(matter); initiate_transfer(matter, target)"

The Savant adds cluster-diversity-first placement: "For target_replicas = 3,
at least 2 replicas must be in different clusters." Expert A's probabilistic
approach is cluster-agnostic; the Savant's explicit placement is
cluster-diversity-aware but requires each peer to have a consistent view of
which clusters hold which replicas.

**Expert A explicitly calls out the thundering herd risk as Problem 5**:

> "At the scale of this system (2GB ROM files, 36 peers), a thundering herd
> of redundant transfers wastes significant bandwidth. If three peers each
> initiate a 2GB transfer to the same target for the same blob, that is 6GB
> of wasted WAN bandwidth. On a 5 Mbps uplink, that is 2.7 hours of
> unnecessary traffic."

**The Savant's approach does not address thundering herd** explicitly. The
most_under_replicated() priority queue means multiple peers could independently
identify the same piece of matter as most_under_replicated and all initiate
transfers simultaneously. The only limiting factor in the Savant's model is
the replication_budget (1-5 chunks per tick depending on metabolic state).

**Expert A and the Savant disagree on when replication runs:**

Expert A: filter reconciliation triggers replication evaluation on
reconnection, plus heartbeat-driven divergence detection during steady state.

Savant: "in Active/Stress mode: replication_budget = 0 // no background
replication." But Expert A says the Conductor should govern replication
aggressiveness with the existing metabolic state. The Savant's rule halts
replication entirely during Active/Stress, which means a partition heal
(which forces Active/Stress via convergence override) produces zero
replication while the system is most in need of filling gaps.

### Why this tension matters

Expert A's probabilistic approach is coordination-free (consistent with the
AP model) but potentially wasteful. The Savant's priority-queue approach is
more deterministic but assumes peers have a consistent view of replica counts,
which requires the content filter to be fully reconciled before replication
decisions are made. During partition heal, filter reconciliation may not be
complete when replication needs to start (this is a direct interaction with
Expert A's Problem 4: "filter reconciliation timing relative to convergence
fronts").

The Savant's rule of no replication in Active/Stress creates a deadlock
scenario: convergence forces Active/Stress, but the post-convergence replication
catch-up (filling gaps from the partition period) requires Active operation.
Expert A's approach does not have this deadlock because it uses the Conductor
for replication aggressiveness rather than halting replication entirely.

Furthermore, Expert A's probabilistic roll uses floating-point division
(`probability = deficit as f64 / (known_holders as f64 + 1.0)` and
`roll = [...] as f64 / 255.0`). The Savant identifies floating-point usage
in the abstract state machine as a critical lint violation (Problem 7, the
fixed-point boundary). The replication decision function is part of the
mesh layer's abstract behavior and should be fixed-point.

### Question for Stage 2

Should replication use Expert A's probabilistic deduplication (coordination-
free, AP-consistent, but potentially redundant transfers) or the Savant's
priority-queue approach (more efficient, but requires consistent filter views
and halts during convergence)? Can the two be combined: priority-queue
ordering with probabilistic per-peer selection? Expert A must address whether
the Savant's Active/Stress halt creates a post-convergence catch-up deadlock.
The Savant must address the thundering herd risk explicitly.

## Tension 5: The hop_count Fix and What It Implies About Relay Trust

**Name:** The AEAD Boundary and Relay Authentication Surface

### Conflicting positions

**Expert C** identifies a critical correctness bug: hop_count is part of
the AEAD associated data, but the relay must decrement it. This breaks
authentication for all relayed frames. Expert C proposes:

> "Exclude hop_count from AEAD associated data. Redefine AEAD AD as:
> AEAD_AD = outer_header[0..8] || outer_header[9..29] (28 bytes,
> destination(4) + connection_id(4) + relay_key_id(8) + nonce(12),
> hop_count (offset 8) excluded). Security analysis: an attacker who
> can modify hop_count can: set hop_count to 0 (drop the frame) or
> set hop_count to 4 (allow one extra hop). Neither attack gives the
> adversary anything beyond what dropping the packet gives."

Expert C's fix is minimal: one field excluded from AD. He argues the security
cost is negligible and cites RFC 9001, IPsec ESP, and WireGuard as precedents.

**Expert C also notes that the relay authenticates via full Noise KK:**

> "A relay MUST NOT forward a frame to an address from which it has not
> received a recent (within 30 seconds) authenticated frame. This prevents
> the relay from being used as a reflector for DDoS amplification."

In Expert C's model, the relay is a full Noise KK participant: it has its
own DK/DEK, it establishes Noise sessions with both endpoints, and it
authenticates forwarded-to addresses via session presence.

**The Savant** accepts the hop_count bug implicitly (by specifying "R
decrements hop_count" without noting the AEAD conflict) but proposes a
more minimal relay identity:

> "The relay is a pseudo-peer with a DK/DEK pair but no CIK (it does not
> belong to a cluster). The relay authenticates to the mesh via a relay-
> specific enrollment process: Admin generates an invite token with
> token_type = 0x04 (RELAY). [...] The relay does not participate in
> gossip, dominance, or convergence. The relay receives a subset of
> heartbeat information sufficient to maintain its forwarding table."

The Savant also frames the relay identity question as an open item:

> "Relay peer identity. Does a relay have a DK/DEK? If yes, it is a
> pseudo-peer with limited capabilities (no channel keys, no gossip, no
> health scoring). If no, it authenticates via a different mechanism
> (PSK from invite token?). This is a design decision, not a specification
> gap."

**Expert A (distributed systems perspective)** does not address the hop_count
bug, but her replication design depends on relay peers participating in
content storage:

> "the Replication defined concept says 'Relay peers participate' [in
> replication]."

If relay peers are full Noise KK participants with DK/DEK (Expert C's model),
they can participate in territorial re-encryption (they have a hash ring
position via their DK pubkey). If relay peers are pseudo-peers with a different
enrollment path (Savant's model), their re-encryption territory assignment
is unclear.

**Expert B** does not address relay authentication but his sans-IO interface
assumes connections are symmetric: the consumer provides BytesReceived and
the state machine emits SendBytes. If the relay is a full Noise KK peer,
it runs the same protocol state machine as any other peer, and Expert B's
interface applies. If the relay is a special-purpose forwarder with only a
forwarding table and no protocol state machine, Expert B's interface does
not apply to it, and a separate relay runtime must be specified.

### Why this tension matters

The hop_count fix is uncontested -- Expert C identifies the bug, all
subsequent design work must incorporate the fix. But the fix's full implications
are not addressed. Excluding hop_count from the AEAD AD means that a malicious
intermediary can increase hop_count to allow a frame to traverse more hops
than intended. Expert C dismisses this as "does not leak plaintext," which
is correct, but does not address traffic amplification from the perspective
of a relay used as a reflector.

More significantly, Expert C's relay model (full Noise KK, DK/DEK, relay
acts as AEAD-level participant in the connection) contrasts with the Savant's
relay model (pseudo-peer, limited capabilities, possibly PSK authentication).
These produce different memory requirements (Expert C's relay runs the full
protocol state machine; Savant's relay runs a simpler forwarding loop),
different enrollment mechanics (full Noise XX enrollment vs relay-specific
token), and different participation in territorial re-encryption (hash ring
member vs non-member).

Expert C's amplification prevention rule ("MUST NOT forward a frame to an
address from which it has not received a recent authenticated frame") requires
the relay to maintain session state for both endpoints of every forwarded
connection. This is consistent with his forwarding table model but creates
a liveness constraint: if one endpoint has not sent a frame in 30 seconds,
the relay drops forwarded frames from the other endpoint, even if the
connection is still live on both sides. A peer in Background mode sends
heartbeats every 30 seconds (the Conductor's background interval is 30s);
at exactly the limit, the relay drops the first heartbeat after a 30-second
gap, triggering a heartbeat miss on the receiver.

### Question for Stage 2

Is the relay a full protocol peer (DK/DEK, Noise KK sessions with endpoints,
full protocol state machine, hash ring member) or a pseudo-peer with a
minimal forwarding runtime? Expert A must address how relay participation
in territorial re-encryption works under the pseudo-peer model (no hash ring
position). Expert B must address whether the relay uses the same Protocol
trait interface or a separate ForwardingRelay interface. The Savant must
address the 30-second amplification prevention rule's interaction with
Background-mode heartbeat intervals.

## Summary: Tensions and the Questions They Generate

| # | Tension | Primary Conflicting Parties | Stage 2 Action |
|---|---------|----------------------------|----------------|
| 1 | SansIO polling convention (4-method vs 2-phase) | Expert B vs Savant | Experts B + C respond; Expert A evaluates convergence implications |
| 2 | Relay-first vs hole-punch-first NAT traversal | Expert C vs design invariant | Expert A evaluates wavefront timer reliability; Expert B evaluates state machine topology |
| 3 | Gateway handoff -- connection teardown vs migration | Expert A vs Savant | Expert B must address state machine; Expert C must address relay path continuity |
| 4 | Replication orchestration -- probabilistic vs priority-queue | Expert A vs Savant | Expert A addresses Active/Stress halt deadlock; Savant addresses thundering herd |
| 5 | Relay trust model -- full peer vs pseudo-peer | Expert C vs Savant | Expert A addresses re-encryption territory; Expert B addresses protocol interface scope |

The most load-bearing tension for the cross-review is Tension 2 (relay-first
vs hole-punch-first) because it changes a stated design invariant, not a gap.
Changing TransitRelay invariant 3 from "last resort" to "first path" would
require the relay resource governance and failover design (Tensions 4 and 5)
to be redesigned for higher steady-state load rather than burst fallback load.

The most dangerous tension for implementation correctness is Tension 5's
embedded sub-issue: the hop_count AEAD fix is uncontested and must be
incorporated immediately, but the 30-second amplification prevention rule
creates a liveness hazard in Background mode that no expert has flagged.
