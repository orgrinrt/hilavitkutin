# Deep Dive: `inventory`, `dlopen`, and Plugin Systems in Rust

Research date: 2026-03-15

---

## 1. How `inventory` Works Internally

The [`inventory`](https://crates.io/crates/inventory) crate (by dtolnay) provides typed distributed
plugin registration. Plugins can self-register from any source file linked into the binary without a
central list.

### Platform Constructor Sections

Under the hood, `inventory` depends on the [`ctor`](https://github.com/mmastrac/rust-ctor) crate.
`ctor` uses platform-specific linker sections to run code before `main()`:

| Platform | Linker Section |
|---|---|
| Linux / Android / BSDs | `.init_array` |
| macOS / iOS (Apple) | `__DATA,__mod_init_func,mod_init_funcs` |
| Windows | `.CRT$XCU` |
| WebAssembly | Synthetic `__wasm_call_ctors()` |

The `#[ctor]` macro expands roughly to:

```rust
#[used]
#[cfg_attr(target_os = "linux", link_section = ".init_array")]
#[cfg_attr(target_vendor = "apple", link_section = "__DATA,__mod_init_func,mod_init_funcs")]
#[cfg_attr(target_os = "windows", link_section = ".CRT$XCU")]
static FOO: extern fn() = {
    extern fn foo() { /* registration logic */ }
    foo
};
```

The `#[used]` attribute prevents the linker from stripping the static during dead-code elimination.
The dynamic loader (ld-linux.so, dyld, ntdll) invokes every function pointer in the appropriate
section when loading the binary or shared library.

### Internal Data Structure

The registry is a lock-free atomic singly-linked list:

- A `Registry<T>` holds an `AtomicPtr<Node<T>>` as its head.
- Each `Node<T>` contains a `value: T` and `next: Option<NonNull<Node<T>>>`.
- `submit` boxes a new `Node`, leaks it, and CAS-swaps it into the head with `SeqCst` ordering.
- `iter::<T>()` loads the head and traverses the list, yielding `&'static T` references.

There is **no ordering guarantee** between registered items of the same type.

## 2. The `submit!` and `collect!` Macros

### `inventory::collect!(MyType)`

Must be called exactly once per type, in the crate that defines the type, outside any function body.
It generates the `Collect` trait implementation for that type, which provides a `registry()` method
returning a `&'static Registry<Self>`. This establishes the per-type linked list head.

### `inventory::submit!(MyType { ... })`

Placed at module scope (not inside functions). Each invocation generates a `#[ctor]`-annotated
initializer that:

1. Evaluates the expression to produce a value of the registered type.
2. Wraps it in a `Node<T>`.
3. Boxes it and submits it to the type's `Registry` via atomic CAS.

All `submit!` invocations across the entire dependency graph take effect simultaneously at startup
(or at `dlopen` time for shared libraries). The generated initializer names include a hash to avoid
collisions, though in practice name collisions have been a reported issue when macros duplicate
submissions (GitHub issue #8).

### Iteration

```rust
for plugin in inventory::iter::<MyPlugin> {
    plugin.do_something();
}
```

Returns an iterator over `&'static T`. The iterator is available after all constructors have run
(i.e., after program start or after `dlopen` completes).

## 3. `inventory` and `dlopen`: Does It Work?

**Yes, with caveats.** When you `dlopen()` a `.so` / `.dylib` containing `inventory::submit!` calls,
the OS loader executes the constructor-section functions during the `dlopen` call. These constructors
register items into the same global `Registry` in the host process. Subsequent calls to
`inventory::iter::<T>()` will include the newly registered items.

### Gotchas

1. **Shared registry requirement.** The host binary and the plugin must share the same `Registry<T>`
   static. This means the plugin must link against the same copy of the `inventory` crate as the
   host. If the plugin statically links its own copy, it gets a separate registry that the host
   cannot see. In practice this means the host should export the registry symbols, or both sides
   should dynamically link a shared `libinventory.so`.

2. **No unregistration.** The linked list is append-only. If you `dlclose()` a library, the nodes it
   registered become dangling pointers. The `ctor` crate documentation notes that on macOS,
   "unloading of shared libraries may not actually happen until process exit, even if explicitly
   unloaded" due to thread-local storage pinning. In general, `dlclose` + `inventory` = UB.

3. **ABI compatibility.** The host and plugin must be compiled with the same Rust compiler version
   (or use `repr(C)` types). The `Node<T>` layout is `repr(Rust)`, meaning it is **not stable**
   across compiler versions.

4. **WASM special handling.** On WASM targets, the linker synthesizes `__wasm_call_ctors()` but may
   not call it automatically. Multi-call WASM modules must explicitly export and invoke it.

5. **Ordering non-determinism.** Constructors run in an unspecified order. If plugin B depends on
   plugin A being registered first, there is no way to enforce that.

### Bottom Line for Saalis

`inventory` works well for **statically linked** plugin discovery within a single binary. For
runtime `dlopen`-based plugins, it is fragile: shared-symbol requirements, no unloading, and
unstable internal ABI make it unsuitable as the primary plugin-loading mechanism. It remains
excellent for the `#[register]` pattern within the monolithic saalis binary itself.

## 4. Alternative Crates: `linkme`, `ctor`

### `linkme` (dtolnay)

[`linkme`](https://github.com/dtolnay/linkme) provides **distributed slices** -- collections of
static elements gathered into a contiguous binary section by the linker.

```rust
#[distributed_slice]
pub static PLUGINS: [fn() -> Box<dyn Plugin>];

#[distributed_slice(PLUGINS)]
static MY_PLUGIN: fn() -> Box<dyn Plugin> = || Box::new(MyPlugin);
```

Key properties:

- **Zero-cost.** No life-before-main, no runtime initialization. The linker arranges elements into a
  contiguous section and `linkme` exposes it as a `&'static [T]` slice.
- **Compile-time type safety.** Element type must match the slice declaration.
- **Platform support:** Linux, macOS, Windows, FreeBSD, OpenBSD, illumos.
- **No dynamic library support.** Elements in a `dlopen`-loaded library are **not** added to the
  distributed slice. The slice is fixed at link time. Issue #36 in the linkme repo confirms this
  limitation.

There is an open issue (inventory#13) discussing rewriting `inventory` on top of `linkme`-style data
segments instead of `ctor`, which would improve portability but would sacrifice `dlopen`
compatibility entirely.

### `ctor` (mmastrac)

[`ctor`](https://github.com/mmastrac/rust-ctor) is the low-level building block. It provides
`#[ctor]` and `#[dtor]` attributes.

- `#[ctor]` places a function pointer in the platform's constructor section.
- `#[dtor]` generates a `#[ctor]` that calls `libc::atexit` to register the destructor.
- No ordering guarantees between constructors.
- Safety caveat: code in constructors must avoid Rust stdlib services that may not be initialized
  yet (stdout, thread-locals, etc.).
- MSRV: 1.56.0 (edition 2021); WASM requires 1.85+.

### Comparison Matrix

| Feature | `inventory` | `linkme` | `ctor` |
|---|---|---|---|
| Mechanism | Constructors (ctor) | Linker data sections | Constructors |
| Life-before-main | Yes | No | Yes |
| Runtime cost | Atomic linked list ops | Zero | Function call |
| dlopen support | Yes (with caveats) | No | Yes |
| Type safety | Via `Collect` trait | Compile-time | Manual |
| Iteration | `&'static T` iterator | `&'static [T]` slice | N/A (raw) |
| Ordering | Unspecified | Unspecified | Unspecified |
| Unloading safety | No | N/A | No |

## 5. The `dlopen` / `libloading` Crate Ecosystem

### `libloading`

The most widely adopted crate (~297M all-time downloads). Provides safe-ish wrappers around
`dlopen`/`dlsym` (Unix) and `LoadLibrary`/`GetProcAddress` (Windows).

- Minimal API: `Library::new(path)`, `library.get::<T>(symbol_name)`.
- Returns `Symbol<T>` with lifetime tied to `Library`.
- Potential thread-safety issues with `dlerror()` on some platforms (FreeBSD).
- No support for loading symbols into structs automatically.

### `dlopen2`

Fork/successor of the original `dlopen` crate by szymonwieloch. Adds features over `libloading`:

- **Symbol-to-struct loading**: derive macros to load multiple symbols into a struct at once.
- **Load from program itself**: can open the current executable as a library.
- **`dladdr` support**: resolve symbol addresses on all platforms.
- **Full thread safety**: proper error handling without `dlerror()` races.
- Object-oriented API with `SymBorApi` and `WrapperApi` traits.

### `sharedlib`

Older crate, less maintained. Has some overhead and lacks features like `dladdr` and
self-loading. Not recommended for new projects.

### Recommendation for Saalis

`libloading` for simple symbol lookup; `dlopen2` if you need structured symbol loading or
`dladdr`. Both are actively maintained. `sharedlib` is effectively dead.

## 6. `repr(C)` ABI Stability Across Compiler Versions

### What `repr(C)` Guarantees

`#[repr(C)]` applies the C layout algorithm: fields are ordered as declared, with C-compatible
padding and alignment rules. This is documented in the
[Rustonomicon](https://doc.rust-lang.org/nomicon/other-reprs.html) and the
[Reference](https://doc.rust-lang.org/reference/type-layout.html).

### What It Does NOT Guarantee

1. **Primitive alignment is platform-specific, not compiler-version-specific.** `u64` alignment
   differs between x86 (4 bytes) and x86_64 (8 bytes) on some platforms. But for a given target
   triple, alignments are stable.

2. **`repr(C)` layout is deterministic for a given set of field types.** If the sizes and alignments
   of all field types are the same, the struct layout will be the same. For primitive types (`u8`,
   `u32`, `f64`, `*const T`, function pointers), this is stable across Rust compiler versions on the
   same target.

3. **No SemVer contract.** There is no official Rust policy stating that `repr(C)` struct layouts
   are semver-stable. Reordering fields or changing types is a breaking change in practice but not
   enforced.

4. **Enums are tricky.** `repr(C)` enums follow C enum rules, but niche optimization is disabled.
   `repr(C, u32)` gives a fixed discriminant size. However, `Option<T>` and other std types are
   `repr(Rust)` and NOT ABI-stable.

### Practical Stability

For a plugin ABI using only `repr(C)` structs with primitive fields and `extern "C"` function
pointers, the layout **is stable** across Rust compiler versions on the same target. This is the
foundation that `abi_stable` and `stabby` build on.

## 7. Designing a Stable Plugin ABI

### Function Pointer Tables (VTables)

The canonical pattern: define a `repr(C)` struct containing `extern "C" fn` pointers.

```rust
#[repr(C)]
pub struct PluginVTable {
    pub api_version: u32,
    pub name: extern "C" fn() -> *const c_char,
    pub init: extern "C" fn(host: *const HostAPI) -> i32,
    pub shutdown: extern "C" fn(),
    pub process: extern "C" fn(input: *const u8, len: usize) -> *mut u8,
    pub free: extern "C" fn(ptr: *mut u8),
}
```

The host loads the library and calls a single well-known symbol:

```rust
extern "C" fn saalis_plugin_entry() -> *const PluginVTable;
```

### Version Negotiation

1. The vtable includes an `api_version` field.
2. The host checks `api_version` before calling any other function.
3. For extensibility, use a prefix-type pattern (as in `abi_stable`): new fields are appended at the
   end, and the host checks the version to know which fields are valid.
4. `stabby` goes further with **canary symbols** embedded in the `.so` encoding the rustc version,
   optimization level, and target triple, checked at load time.

### Error Handling Across the Boundary

- **Never unwind across `extern "C"`.** A Rust panic unwinding through an FFI boundary is UB.
  Use `std::panic::catch_unwind` or the `sabi_extern_fn` macro from `abi_stable`.
- **Return error codes** (C-style `i32`) or a `repr(C)` result type.
- **Allocator discipline.** Memory allocated by the plugin must be freed by the plugin. Provide an
  explicit `free` function in the vtable. Alternatively, both sides must agree on the same allocator
  (e.g., the System allocator, which cdylibs use by default).

### Existing Frameworks

**`abi_stable`** provides:
- `StableAbi` derive macro for layout verification at load time.
- `RootModule` trait for the plugin entry point.
- Prefix types for forward-compatible vtables.
- `sabi_extern_fn` for panic-safe FFI.
- Runtime type-layout checking between host and plugin.

**`stabby`** provides:
- `#[stabby::stabby]` attribute for `repr(C)` with compile-time layout assertions.
- Compact sum-type encoding (`repr(stabby)`) that emulates niche optimization across FFI.
- Canary symbols for build-environment compatibility checks.
- `IStable` trait with associated types representing layout, including niches.
- `StabbyLibrary` trait extending `libloading` with report-aware symbol getters.
- SemVer Prime versioning: `version = 2^api * 3^abi`.

## 8. Real-World Rust Plugin Systems

### Bevy Dynamic Linking

Bevy had `bevy_dynamic_plugin` which loaded `.dylib`/`.so` files exporting a `_bevy_create_plugin`
symbol returning `*mut dyn Plugin` (via `Box::into_raw`). The system was **deprecated in Bevy 0.15**
because:

- Rust has no stable ABI; plugins must be compiled with the exact same Bevy and Rust versions.
- Loading a library runs constructor code unconditionally.
- The `dyn Plugin` pointer crosses the FFI boundary unsafely.
- Not suitable for modding; only viable for same-build-session DLC-style content.

Bevy now uses `bevy_dylib` purely for faster recompilation during development (dynamic linking of
Bevy itself), not for third-party plugins.

### Cargo Subcommands

Cargo uses a process-based plugin model: any binary named `cargo-<subcommand>` in `$PATH` is
invoked as `cargo <subcommand>`. Arguments are forwarded. `cargo --list` discovers available
plugins. This is the simplest possible plugin architecture -- no ABI concerns, no shared memory,
just subprocess execution with IPC via stdin/stdout/exit codes. It is robust but high-overhead for
fine-grained interactions.

### Zellij Plugins (WASM)

Zellij uses WASM for plugins, having migrated from `wasmtime` to the `wasmi` interpreter (v0.51) in
Zellij v0.44.0. Architecture:

- Plugins implement the `ZellijPlugin` trait from the `zellij-tile` crate, compiled to WASI preview
  1 modules.
- Communication uses Protocol Buffers (not WIT).
- Each plugin runs in an isolated memory space with `wasmi_wasi` for controlled file I/O.
- Resource limits via `wasmi::StoreLimits` for memory and stack.
- Plugins can manipulate panes, tabs, and other plugins.

The shift from wasmtime (JIT) to wasmi (interpreter) traded raw speed for smaller binary size and
simpler embedding.

### Lapce Plugins (WASI)

Lapce uses WASI-based plugins. Any language that compiles to WASI can produce a Lapce plugin (Rust,
C, AssemblyScript). This gives language-agnostic extensibility with sandboxing. The plugin system
provides LSP-like capabilities: syntax highlighting, language servers, themes.

## 9. WASM as an Alternative to `dlopen`

### Runtimes

| Runtime | Type | Key Properties |
|---|---|---|
| **Wasmtime** | JIT/AOT | Bytecode Alliance, fastest cold start, Cranelift backend |
| **Wasmer** | JIT/AOT/Interpreter | Multiple backends (LLVM, Cranelift, Singlepass) |
| **wasmi** | Interpreter | Small, embeddable, no JIT, used by Zellij |
| **WasmEdge** | JIT/AOT | Cloud-native focus, WASI-NN for ML |

### Performance vs. Native `dlopen`

Benchmarks show:

- `dlopen` setup: ~140 microseconds
- Wasmer setup: ~968 microseconds
- Wasmtime setup: ~989 microseconds
- Steady-state execution: WASM is within 1.2-2x of native for compute-heavy tasks.
- Memory overhead: native ~1.8 MB resident vs. WASM ~10-15 MB per instance.

### Pros of WASM Plugins

1. **Sandboxing.** Memory isolation by design. A plugin cannot corrupt host memory.
2. **Language-agnostic.** Any language targeting WASM/WASI works (Rust, C, Go, Zig, AssemblyScript).
3. **Distribution.** Shipping a `.wasm` blob is simpler than platform-specific `.so`/`.dylib`/`.dll`.
4. **Deterministic.** No UB from ABI mismatches, allocator conflicts, or unwinding.
5. **Hot-reloadable.** Drop the old instance, load the new one. No `dlclose` hazards.

### Cons of WASM Plugins

1. **Performance overhead.** 5-7x setup latency vs. `dlopen`. Compute overhead 1.2-2x native.
2. **Memory overhead.** 10-15 MB per plugin instance vs. shared address space for native.
3. **Limited system access.** WASI provides a capability-based subset of POSIX. No raw sockets, no
   mmap, limited filesystem. Custom host functions needed for anything beyond basic I/O.
4. **Serialization boundary.** All data crossing the WASM/host boundary must be serialized
   (typically via shared linear memory + length-prefixed buffers, or via the Component Model).
5. **Ecosystem maturity.** WASI preview 2 and the Component Model are still stabilizing. WIT
   tooling is improving but not yet seamless.
6. **Binary size.** Embedding wasmtime adds ~20-40 MB to the host binary. wasmi is much smaller
   (~1-2 MB) but slower.

### WASM Component Model

The emerging standard for structured WASM interfaces. Uses WIT (WebAssembly Interface Types) to
define typed APIs between host and guest. Eliminates manual serialization. Still maturing but
represents the future of WASM plugin systems. Zellij chose Protobuf over WIT; newer projects may
prefer WIT as tooling stabilizes.

## 10. Testing Plugins

### The `test-cdylib` Crate

[`test-cdylib`](https://github.com/Jarcho/test-cdylib) (inspired by dtolnay's trybuild) is the
go-to for testing cdylib plugin projects:

```rust
#[test]
fn test_plugin_loads() {
    let path = test_cdylib::build_current_project();
    let lib = unsafe { libloading::Library::new(&path).unwrap() };
    let entry: libloading::Symbol<extern "C" fn() -> *const PluginVTable> =
        unsafe { lib.get(b"saalis_plugin_entry").unwrap() };
    let vtable = unsafe { &*entry() };
    assert_eq!(vtable.api_version, 1);
}
```

Three entry points:

- `build_current_project()` -- builds the current crate as a cdylib.
- `build_example(name)` -- builds a named example as a cdylib.
- `build_path(path)` -- builds an arbitrary `.rs` file as a cdylib.

Use `once_cell` / `LazyLock` to cache the library path across multiple tests in the same test
binary, avoiding redundant rebuilds.

### Testing Strategies

1. **Trait-based dual testing.** Define the plugin interface as a trait. Test pure-Rust
   implementations directly (unit tests). Then test the FFI boundary separately via cdylib loading
   (integration tests). This isolates logic bugs from ABI bugs.

2. **Round-trip serialization tests.** For every `repr(C)` type crossing the boundary, write tests
   that construct the type on one side, pass it through `extern "C"`, and verify it on the other
   side.

3. **Version negotiation tests.** Build plugin cdylibs with different `api_version` values and
   verify the host rejects incompatible versions gracefully.

4. **Allocator mismatch tests.** Verify that memory allocated by the plugin is freed by the
   plugin's `free` function, not the host's allocator.

5. **Panic safety tests.** Trigger panics inside plugin FFI functions and verify they are caught
   (via `catch_unwind`) rather than unwinding across the boundary.

6. **Multi-plugin interaction tests.** Load multiple plugins and verify they coexist without symbol
   collisions or registry corruption.

7. **WASM plugin testing.** For WASM-based plugins, use `wasmtime`'s or `wasmi`'s embedding API in
   tests. Instantiate the module, call exported functions, and verify results. No `unsafe` needed;
   the runtime handles isolation.

### CI Considerations

- cdylib builds rebuild all dependencies, which is slow. Cache aggressively.
- Test on all target platforms (Linux `.so`, macOS `.dylib`, Windows `.dll`).
- For `inventory`-based registration, test both the statically-linked case and (if applicable) the
  `dlopen` case, since they exercise different code paths.

## Implications for Saalis

The `#[register]` + `inventory` pattern in the saalis architecture works well for the monolithic
single-binary design. All entity types, metadata tables, and column descriptors self-register at
startup via constructor sections. Since saalis is a single Rust binary (not a plugin host), the
`dlopen` gotchas do not apply to the core use case.

If saalis ever needs runtime-loadable plugins (e.g., user-contributed content-type adapters), the
options are:

1. **Native `dlopen` with `abi_stable` or `stabby`** -- best performance, complex ABI management,
   requires same-target compilation.
2. **WASM/WASI plugins via `wasmi`** -- sandboxed, portable, language-agnostic, but higher overhead
   and serialization cost. Good fit for untrusted third-party plugins.
3. **Process-based plugins (cargo-style)** -- simplest, most robust, but highest per-call overhead.
   Good for infrequent operations like metadata enrichment.

For the current scope (single binary on Batocera/loisto), `inventory` + `linkme` for static
registration is the right call. The deferred presentation-layer plugins (themes, UI components)
would be natural candidates for WASM if that path is ever pursued.
