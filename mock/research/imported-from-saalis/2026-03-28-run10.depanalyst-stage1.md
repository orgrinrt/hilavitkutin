### 1.4 Cycle analysis

**Critical finding: two potential back-edges detected.**

**Back-edge 1: verkko-crypto -> verkko-mesh (IMPLICIT)**

In verkko-crypto body:
- S1.1: "Bound at enrollment (see verkko-mesh: Enrollment)"
- S1.3: "For mesh initialization using these keys... see verkko-mesh: Mesh bootstrap"
- S1.4: "For the enrollment and key delivery flow... see verkko-mesh: Enrollment, Device recovery enrollment"
- S2.4: "see verkko-mesh: KeyBundle"
- S2.6: "Distribution of the new epoch key is specified in verkko-mesh"

This creates the edge: `verkko-crypto -> verkko-mesh -> verkko-crypto`, which is a cycle.

**Assessment:** These are informational forward-references ("for how this is used, see..."), not definitional dependencies. verkko-crypto does not need verkko-mesh concepts to define its own concepts. The crypto primitives are self-contained. However, these cross-references are structurally ambiguous. A reader following these references discovers that verkko-crypto's body text relies on mesh concepts to complete the operational picture. The defined concepts (DK, DEK, ChannelKey, etc.) are self-standing, but the body's protocol flows are not.

**Verdict:** Not a true definitional cycle, but a documentation coupling cycle. The body of verkko-crypto is not understandable in isolation; it explains flows that span both documents. This is a coupling smell, not a build-order violation.

**Back-edge 2: verkko-protocol -> verkko-mesh (IMPLICIT)**

In verkko-protocol body:
- S6.3: "MetabolicState transitions are driven by a governor... see verkko-mesh: Conductor"
- S3.4: Heartbeat extension fields reference "see verkko-mesh for semantics" and "see verkko-mesh: HealthPipeline"
- S3.4: SACK defined concept says "see verkko-mesh: HealthPipeline"

This creates the edge: `verkko-protocol -> verkko-mesh -> verkko-protocol`, another cycle.

**Assessment:** The MetabolicState defined concept in verkko-protocol states its transitions "are driven by a governor that provides metabolic state as input. See verkko-mesh: Conductor for the governor specification." This is a dependency on a mesh concept to complete the protocol's own defined concept. The heartbeat extension fields are explicitly split: transport header in protocol, extension semantics in mesh. This is cleaner -- the protocol defines the frame format and mesh defines what goes in the extension slots.

The MetabolicState case is more concerning. The protocol defines the state machine but defers its input signal to mesh. This means the protocol's MetabolicState defined concept is incomplete without verkko-mesh: Conductor. A consumer of MetabolicState cannot understand what drives transitions without reading verkko-mesh.

**Verdict:** Partial definitional dependency. MetabolicState's invariant "transitions are driven by a governor" is meaningful without knowing Conductor's internals, but only barely. This is a tight coupling point.

**Back-edge 3: verkko-crypto -> verkko-ops (IMPLICIT)**

In verkko-crypto body:
- S2.4: "See verkko-ops: Pattern Registry, Structure 3"

**Assessment:** verkko-ops is a leaf document that declares "None" for defined concepts. This reference is informational only. Not a cycle concern because verkko-ops defines no concepts that crypto depends on.

**Verdict:** Informational reference. Not a dependency.

### 1.5 Hidden edge: verkko-matter -> verkko-protocol

verkko-matter's Dependencies section does not list verkko-protocol. However:
- BandwidthPolicy defined concept states: "LEDBAT (from verkko-protocol) enforces at transport level; policies set the ceiling"
- ContentFilter states: "Filter root (keyed Merkle) included in every heartbeat" (heartbeat is a verkko-protocol concept)
- MatterTransfer references "bao-tree verified streaming" with chunk-level integrity, which operates over verkko-protocol's Frame and Stream concepts

**Assessment:** verkko-matter uses LEDBAT and Heartbeat concepts from verkko-protocol without declaring the dependency. It reaches through verkko-mesh's dependency on verkko-protocol. The filter_root field is placed in the heartbeat extension region, which is defined in verkko-protocol's frame format but given semantics by verkko-mesh. So the real path is: matter -> mesh (gossip/heartbeat extension) -> protocol (heartbeat frame). The LEDBAT reference is a direct undeclared dependency on verkko-protocol.

**Verdict:** Missing declared dependency. verkko-matter should list verkko-protocol in its Dependencies section, at least for LEDBAT.

### 1.6 Corrected dependency graph

```
verkko-crypto       -> (none)                          [CLEAN]
verkko-protocol     -> verkko-crypto                   [DECLARED]
                    -> verkko-mesh (MetabolicState)     [IMPLICIT, partial cycle]
verkko-mesh         -> verkko-crypto, verkko-protocol   [DECLARED]
verkko-relay        -> verkko-crypto, verkko-protocol, verkko-mesh [DECLARED]
verkko-seat         -> verkko-crypto, verkko-protocol   [DECLARED]
verkko-matter       -> verkko-crypto, verkko-mesh, verkko-relay [DECLARED]
                    -> verkko-protocol (LEDBAT)         [HIDDEN, undeclared]
verkko-ops          -> all six                          [DECLARED]
```

### 1.7 Topological sort result

With only declared edges: **passes** (no cycles).

With implicit edges included: **fails** on the `verkko-protocol <-> verkko-mesh` mutual reference through MetabolicState/Conductor. This is a 2-node cycle.

With the crypto -> mesh back-references: **fails** on `verkko-crypto <-> verkko-mesh`. Another 2-node cycle.

**Root cause:** These cycles arise from the separation of "what the state machine is" (protocol) from "what drives it" (mesh). The design splits the mechanism from the driver across two documents that already depend on each other in the other direction.

## 4. Interface Sufficiency Assessment

For each declared dependency edge: does the dependent document use only defined concepts from the depended-on document, or does it reach into body internals?

### verkko-protocol -> verkko-crypto

**Declared dependency on:** FrameKey, TransportAEAD, NonceSafety, EpochRatchet, KeyZeroization

**Interface sufficiency: SUFFICIENT**

verkko-protocol uses these concepts at the interface level. It references ChaCha20-Poly1305 specifically (which is the TransportAEAD invariant), uses nonce ranges from NonceSafety, and references per-frame key derivation from FrameKey. The wire format (S3.1) specifies the nonce field size (96-bit) which is documented in TransportAEAD's invariants.

**Stability test:** If verkko-crypto changed HKDF to a different KDF but kept the same EpochRatchet invariants (forward-only, deterministic), verkko-protocol would not need to change.

Minor concern: verkko-protocol S3.1 wire format includes `key_id` which uses `BLAKE3(K_epoch_N || channel_index)[0..8]` -- this is an internal detail from crypto S1.5. The wire format hardcodes the key_id length (8 bytes) which is documented in crypto's body, not its defined concepts. If crypto changed key_id length, protocol's wire format breaks.

### verkko-mesh -> verkko-crypto

**Declared dependency on:** DK, DEK, ChannelKey, EpochRatchet, BroadcastEncryption, PCSMechanism, DomainSeparatorRegistry, FoundingHash

**Interface sufficiency: MOSTLY SUFFICIENT, with one internal leak**

verkko-mesh uses DK for signing, DEK for encryption, ChannelKey for channel management, EpochRatchet for advancement, BroadcastEncryption for multi-recipient, PCSMechanism for re-seeding, FoundingHash for mesh identity. All at the interface level.

**Internal leak:** Mesh S2.3 (dominance cascade) uses `BLAKE3(DK_pub || current_epoch)` as the hash function for dominance scoring. This relies on BLAKE3 being the hash function, which is documented in crypto's DomainSeparatorRegistry body, not in any defined concept's invariants. If crypto changed from BLAKE3 to SHA3, mesh's dominance formula would need to change.

**Internal leak 2:** Mesh S1.2 (enrollment binding) uses `Sign(admin_DK, new_DK_pub || new_DEK_pub || enrollment_epoch || cluster_CIK)` which specifies the signing operation format. This relies on DK being Ed25519 specifically, which is in DK's definition ("Ed25519 signing key").

### verkko-mesh -> verkko-protocol

**Declared dependency on:** Connection, Heartbeat, Session, Frame, VitalQueue, MetabolicState, SACK

**Interface sufficiency: SUFFICIENT**

Mesh uses heartbeat for observation, connection for session management, frame for delivery, VitalQueue for priority queuing, MetabolicState for resource-driven behavior, SACK for health measurement. All at the defined concept level.

The heartbeat extension region is well-designed: protocol defines the frame format and explicitly designates the extension region as mesh-defined. Mesh fills in the semantics. This is clean interface segregation.

### verkko-relay -> verkko-crypto

**Declared dependency on:** StorageAEAD, ChannelKey, EpochRatchet, DomainSeparatorRegistry, NonceSafety

**Interface sufficiency: MOSTLY SUFFICIENT**

Relay uses StorageAEAD (XChaCha20-Poly1305) for at-rest encryption, ChannelKey for key management, NonceSafety for pre-batch persist. Clean interface usage.

**Internal leak:** Relay S4.4 specifies `re_enc_nonce = BLAKE3("saalis-reenc-v1" || channel_key_new || blob_id || re_encryption_epoch)[0..24]`. This uses a specific BLAKE3 domain separator from crypto's DomainSeparatorRegistry body. The domain separator string is an internal detail. However, it is registered in the registry, so this may be considered part of the contract.

### verkko-relay -> verkko-protocol

**Declared dependency on:** Frame, Connection, NATTraversal

**Interface sufficiency: SUFFICIENT**

Relay uses Frame for forwarding, Connection for transit, NATTraversal for fallback. All at the defined concept level.

### verkko-relay -> verkko-mesh

**Declared dependency on:** Cluster, Gateway, Convergence

**Interface sufficiency: SUFFICIENT**

Relay uses Cluster for topology, Gateway for routing decisions, Convergence for re-encryption timing (Front 3). The convergence interaction is well-defined: relay waits for mesh to signal Front 3.

### verkko-seat -> verkko-crypto

**Declared dependency on:** TransportAEAD, KeyZeroization

**Interface sufficiency: SUFFICIENT**

Minimal, clean interface. Seat uses transport encryption and key cleanup. No internal leaks.

### verkko-seat -> verkko-protocol

**Declared dependency on:** Connection, Session, Frame, Stream, SchemaSync, SansIO

**Interface sufficiency: SUFFICIENT**

Seat uses connection establishment, session, frame delivery, stream multiplexing, schema sync, and sans-IO architecture. All at the defined concept level. Clean.

### verkko-matter -> verkko-crypto

**Declared dependency on:** StorageAEAD, DomainSeparatorRegistry, EpochRatchet, ChannelKey

**Interface sufficiency: MOSTLY SUFFICIENT**

Matter uses BLAKE3 for content addressing (`blob_id = BLAKE3(plaintext)`). This relies on BLAKE3 being the hash function, which is a crypto internal. However, BLAKE3 is registered in DomainSeparatorRegistry and is a fundamental identity property, so changing it would be a breaking change by definition.

### verkko-matter -> verkko-mesh

**Declared dependency on:** Cluster, Gateway, Gossip, HealthPipeline, Convergence, Channel

**Interface sufficiency: SUFFICIENT**

Matter uses Cluster for topology, Gateway for routing, Gossip for advertisement, HealthPipeline for source selection (effective_health threshold), Convergence for sequencing, Channel for access control. All at defined concept level.

**Note:** IntegrityFailure scar attribution is defined in verkko-mesh but fires on matter-layer events. This is a reverse coupling point (see H8 above).

### verkko-matter -> verkko-relay

**Declared dependency on:** Membrane, TerritorialReencryption, IdempotentPUT

**Interface sufficiency: SUFFICIENT**

Matter uses relay for at-rest storage, re-encryption, and idempotent writes. Clean.

## 5. Transitive Dependency Leaks

### 5.1 Does verkko-mesh reference verkko-crypto directly?

**Yes, declared.** Mesh depends on both crypto and protocol. Its crypto references are direct (DK, DEK, ChannelKey, EpochRatchet, etc.) and do not go through protocol. This is correct: mesh needs crypto concepts that protocol does not mediate (e.g., BroadcastEncryption, PCSMechanism, FoundingHash).

### 5.2 Does verkko-relay reference verkko-crypto directly?

**Yes, declared.** Relay depends on crypto directly for StorageAEAD, NonceSafety, DomainSeparatorRegistry. These are not mediated through protocol or mesh. This is correct: relay's at-rest encryption is a different AEAD domain than protocol's transport encryption.

### 5.3 Does verkko-matter reference verkko-crypto directly?

**Yes, declared.** Matter depends on crypto for StorageAEAD, DomainSeparatorRegistry, EpochRatchet, ChannelKey. Content addressing uses BLAKE3 directly. This is correct: matter's content integrity is a direct crypto concern.

### 5.4 Does verkko-matter reference verkko-protocol through verkko-mesh?

**Partially.** Matter references LEDBAT (a protocol concept) directly without declaring a protocol dependency. The filter_root in heartbeats flows through mesh's extension region, which is properly mediated. But the LEDBAT reference in BandwidthPolicy is a transitive leak: matter reaches through mesh to protocol's congestion control.

### 5.5 Does verkko-seat reference verkko-mesh?

**No.** This is explicitly noted: "Does NOT depend on verkko-mesh." Clean isolation. The host peer serves as the boundary. This is well-designed.

## 6. Concept Ownership Conflicts

### 6.1 CIK (Cluster Identity Key)

**Contested between:** verkko-crypto and verkko-mesh

- verkko-crypto S1.1 describes CIK as "X25519, Noise/DHT" and discusses CIK replication to cluster peers.
- verkko-mesh defines CIK as a defined concept with invariants.

**Current resolution:** CIK is defined in verkko-mesh. Crypto mentions it in the body but does not list it as a defined concept.

**Assessment:** Correct placement. CIK is a cluster-level concept (mesh), not a pure cryptographic primitive (crypto). The key type (X25519) is a crypto concern, but CIK's semantics (shared among cluster, rotation via control log) are mesh concerns.

### 6.2 CPSK (Cluster-Pair Session Key)

**Contested between:** verkko-crypto and verkko-mesh

- verkko-crypto S2.6 discusses CPSK-seeded PCS mechanism in detail.
- verkko-mesh defines CPSK as a defined concept.

**Current resolution:** CPSK is defined in verkko-mesh. Crypto discusses how CPSK feeds PCS.

**Assessment:** Reasonable. CPSK is the output of a Noise handshake (protocol/mesh concern), not a pure crypto primitive. Its creation involves mesh topology (cluster pairs). Crypto correctly limits itself to the derivation math (PCSMechanism).

### 6.3 MetabolicState / Conductor

**Contested between:** verkko-protocol and verkko-mesh

- verkko-protocol defines MetabolicState as a defined concept.
- verkko-mesh defines Conductor as a defined concept and claims it "drives MetabolicState transitions."

**Current resolution:** Protocol owns the state machine. Mesh owns the governor that drives it.

**Assessment:** This is the weakest ownership boundary in the design. The state machine is meaningless without its driver. A consumer of MetabolicState must understand Conductor to know what triggers transitions. This is a layering inversion: the driven mechanism is in a lower layer than the driver. If protocol is level 1 and mesh is level 2, then level-1's behavior depends on level-2's Conductor. This is architecturally uncomfortable but not circular in the strict DAG sense because protocol does not import Conductor's invariants -- it just says "something drives me."

**Recommendation:** Either (a) move MetabolicState to verkko-mesh (since mesh is where the driver lives), or (b) define MetabolicState's transitions in terms of abstract input signals that any governor could provide, without naming Conductor. Option (b) is currently almost achieved but the explicit "see verkko-mesh: Conductor" in the defined concept invariants breaks it.

### 6.4 Heartbeat extension fields

**Contested between:** verkko-protocol and verkko-mesh

- verkko-protocol defines the Heartbeat frame format including the extension region.
- verkko-mesh defines the extension field semantics.

**Current resolution:** Protocol owns wire format. Mesh owns extension semantics.

**Assessment:** Well-designed split. The interface is clear: protocol provides slots; mesh fills them. This is the correct way to handle cross-layer data. No conflict.

### 6.5 PhysicalBoundary trait

**Contested between:** verkko-mesh and verkko-ops

- verkko-mesh defines PhysicalBoundary as a defined concept with trait definition.
- verkko-ops Pattern Registry lists PhysicalBoundary as Structure 5.

**Current resolution:** PhysicalBoundary is defined in verkko-mesh. Ops references it.

**Assessment:** Correct. Ops defines no concepts. Mesh owns the trait.

### 6.6 ScarAttribution::IntegrityFailure

**Contested between:** verkko-mesh and verkko-matter

- The ScarAttribution enum is in verkko-mesh (S5.1).
- IntegrityFailure fires on verkko-matter events (MatterIntegrity).
- verkko-matter defines blob_id, which appears inside the ScarAttribution variant.

**Current resolution:** ScarAttribution is in mesh. The IntegrityFailure variant references matter concepts.

**Assessment:** This creates an upward dependency from mesh to matter. Mesh must know about blob_id (a matter concept) to define the IntegrityFailure variant. This is an ownership conflict. Options: (a) move IntegrityFailure to matter and have matter produce scar entries, or (b) define a generic "content failure" scar type in mesh that does not reference blob_id, and let matter-layer code populate the details.

## 7. The Ops Document

### 7.1 Does ops create implicit coupling?

verkko-ops depends on all six documents and defines no concepts. This makes it a pure leaf node. No document depends on ops. The question is whether ops creates implicit coupling between documents that do not otherwise depend on each other.

**Analysis:** Ops contains tables that cross-reference all six documents:
- Mechanism Reference Table: maps mechanisms to home documents
- Pattern Registry: maps algebraic structures to instances across documents
- Boundary Object Table: maps shared objects to home documents and consumers
- Verification Frame: references all documents

These tables create a single document that readers use to navigate the entire design. If a reader sees "Structure 2 (EMA)" in the Pattern Registry and follows the instances, they see connections between mesh (HealthPipeline) and protocol (PLL correction) that neither document declares.

**Verdict:** Ops creates navigational coupling but not definitional coupling. No document's definitions change because of ops. However, ops is a **coupling amplifier** for humans: it makes implicit connections visible and may lead developers to treat cross-document patterns as shared implementations when they are explicitly designed as independent. The Pattern Registry explicitly warns against this ("No. Pattern only."), which is the correct mitigation.

### 7.2 Is the all-six dependency a problem?

**No.** Ops is a deployment/verification document. It is consumed by implementors, not by other design documents. Its dependencies are read-only references. If ops were a design document that defined concepts used by others, the all-six dependency would create a choke point. As a leaf node, it is harmless.

The risk is scope creep: if ops starts defining patterns or conventions that other documents reference (as is already starting to happen with back-references from crypto S2.4 and mesh body sections to "verkko-ops: Pattern Registry"), ops effectively becomes a shared dependency, contradicting its "defines no concepts" claim.

**Current state:** Three documents already back-reference ops:
- verkko-crypto S2.4: "See verkko-ops: Pattern Registry, Structure 3"
- verkko-mesh body: "see verkko-ops: Pattern Registry" (multiple locations)
- verkko-relay S4.4: "verkko-ops: Pattern Registry, Structure 7"
- verkko-mesh: "For the Verification Frame, see verkko-ops"
- verkko-mesh: "For Compile-Time Enforcement table, see verkko-ops"

These back-references are informational (pointing to organizational tables), but they create a navigational dependency. A reader of verkko-crypto or verkko-mesh needs verkko-ops to understand references to "Structure 3" or "Structure 7."

## 8. Evolution Impact Analysis

### Scenario A: verkko-crypto adds a new key type

Example: adding a post-quantum key type alongside Ed25519/X25519.

**Impact propagation:**

| Document | Needs change? | Why |
|----------|--------------|-----|
| verkko-crypto | YES | New defined concept for the key type |
| verkko-protocol | MAYBE | If wire format changes to accommodate larger keys. Session might need new Noise pattern. |
| verkko-mesh | YES | Enrollment, CIK, CPSK would need to support new key type. Dominance hash might change. |
| verkko-relay | MAYBE | If StorageAEAD changes construction |
| verkko-seat | MAYBE | If session setup changes |
| verkko-matter | NO | Content addressing is independent of identity keys |
| verkko-ops | YES | Tables, verification plans update |

**Localization:** Poor. A new key type propagates to 3-5 documents. This is inherent to cryptographic key types -- they are foundational. The DAG structure means the blast radius is proportional to the depth of the crypto dependency.

### Scenario B: verkko-protocol changes the wire format

Example: changing frame overhead from 55 bytes to 64 bytes (adding a field).

**Impact propagation:**

| Document | Needs change? | Why |
|----------|--------------|-----|
| verkko-crypto | NO | Wire format is not crypto's concern |

| verkko-ops | YES | Cost tables, build order |

**Localization:** Medium. The Frame defined concept's invariants are abstract enough ("one frame = one logical message") that most changes are localized. But relay's outer header format is tightly coupled to protocol's wire format.

## 9. Structural Findings Summary

### 9.1 DAG violations (by severity)

1. **MEDIUM: verkko-protocol <-> verkko-mesh cycle via MetabolicState/Conductor.** Protocol's MetabolicState defined concept is incomplete without mesh's Conductor. This is a soft definitional cycle. The defined concept's invariant text says transitions "are driven by a governor" without defining what a governor is, then defers to mesh. This makes the defined concept partially opaque to consumers who have not read mesh.

2. **LOW: verkko-crypto -> verkko-mesh body back-references.** Crypto's body contains 5+ forward-references to mesh concepts. These are informational but create documentation coupling. Crypto's defined concepts are self-standing.

3. **MEDIUM: verkko-matter missing verkko-protocol dependency.** BandwidthPolicy directly references LEDBAT without declaring the dependency.

4. **MEDIUM: verkko-mesh -> verkko-matter upward reference via IntegrityFailure.** ScarAttribution::IntegrityFailure contains `blob_id: [u8; 32]`, a matter concept, creating an undeclared upward dependency.

5. **LOW: verkko-mesh -> verkko-relay upward reference via Convergence Front 3.** Mesh's convergence specification delegates to relay's TerritorialReencryption without declaring a relay dependency.

### 9.2 Interface leaks (by severity)

1. **LOW: key_id length (8 bytes) hardcoded in protocol wire format.** This is a crypto body internal, not a defined concept invariant.

2. **LOW: BLAKE3 as hash function used directly in mesh dominance formula.** If crypto changed the hash function, mesh would break.

3. **LOW: BLAKE3 domain separator strings used directly in relay re-encryption nonce derivation.** These are registered in the DomainSeparatorRegistry but the specific strings are body internals.

### 9.3 What works well

1. **verkko-seat isolation.** Clean dependencies on only crypto and protocol. Explicitly does NOT depend on mesh. This is excellent layering.

2. **Heartbeat extension region.** Protocol defines the wire format with extension slots. Mesh defines the extension semantics. Clean cross-layer interface.

3. **verkko-ops as leaf.** Defines no concepts. Pure consumer. Cannot create circular dependencies.

4. **Defined Concepts as contracts.** The pattern of defined concepts with invariants works well. When documents use only the defined concepts of their dependencies, the interface is clean and stable.

5. **verkko-crypto as root.** Zero dependencies. All other documents can depend on it without creating cycles. The crypto primitives are self-contained.

6. **BLAKE3 Domain Separator Registry.** Centralizes all hash domain separators in one authoritative list. Prevents cross-domain collisions. This is a model for managing shared constants in a modular design.

### 9.4 Recommendations

1. **Remove the explicit "see verkko-mesh: Conductor" from MetabolicState's defined concept invariants.** Replace with "transitions are driven by an external governor providing metabolic state as input." This preserves the invariant without naming Conductor, breaking the soft cycle.

2. **Add verkko-protocol to verkko-matter's declared Dependencies.** BandwidthPolicy references LEDBAT directly. The dependency should be explicit.

3. **Resolve the IntegrityFailure ownership conflict.** Either (a) make ScarAttribution::IntegrityFailure generic (no blob_id, just a content hash), or (b) acknowledge the upward dependency from mesh to matter in mesh's Dependencies section.

4. **Consider adding verkko-relay to verkko-mesh's declared Dependencies.** Convergence Front 3 directly delegates to relay's TerritorialReencryption. The mesh-relay relationship is bidirectional in practice, even if the taxonomy claims relay depends on mesh and not vice versa.

5. **Convert verkko-crypto body back-references to mesh from "see verkko-mesh: X" to "consumers use this concept for X (defined elsewhere)."** This makes the informational nature explicit and reduces the impression of a dependency.

6. **Guard against verkko-ops scope creep.** The Pattern Registry is already being back-referenced from three design documents. If this pattern grows, ops becomes a de facto shared dependency while claiming to define no concepts. Consider whether the Pattern Registry entries should be moved to their home documents with ops merely collecting references.
