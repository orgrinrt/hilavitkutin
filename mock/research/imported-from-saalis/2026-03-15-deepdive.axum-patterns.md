# Deep Dive: Axum Patterns for Server-Rendered HTMX with SSE

> Research date: 2026-03-15
> Axum version context: 0.8.x (current stable as of early 2026)
> Focus: patterns relevant to a sync-core Rust application serving HTMX over SSE with session management and role-based access

---

## 1. Router Structure -- Nested Routers, State Sharing, Layer Composition

Axum's `Router` is the composition unit. Routes are defined with `Router::route()`, grouped
with `Router::nest()`, and merged with `Router::merge()`. Each of these interacts with state
and middleware differently.

### Nesting

`Router::nest("/prefix", child_router)` strips the matched prefix before the child sees the
request URI. This is critical for modular route trees:

```rust
let admin_routes = Router::new()
    .route("/users", get(list_users))
    .route("/users/{id}", get(show_user));

let api_routes = Router::new()
    .nest("/admin", admin_routes)
    .route("/health", get(health_check));

let app = Router::new()
    .nest("/api", api_routes);
// GET /api/admin/users -> list_users (sees URI "/users")
// GET /api/health      -> health_check (sees URI "/health")
```

Nested routers inherit the outer router's fallback unless they define their own. Only one
router in a `merge()` operation can have an explicit fallback (use `reset_fallback()` if both
have one).

### State and Nesting

`Router<S>` is generic over its missing state type. `Router<()>` is a "complete" router
requiring no state -- only `Router<()>` can be passed to `axum::serve()`. You call
`.with_state(state)` to provide the state and collapse `Router<S>` into `Router<()>`.

Nested routers can have different state types if each is finalized with `.with_state()` before
nesting:

```rust
let inner = Router::new()
    .route("/bar", get(inner_handler))
    .with_state(InnerState { /* ... */ });

let app = Router::new()
    .route("/", get(outer_handler))
    .nest("/foo", inner)       // inner is Router<()> already
    .with_state(OuterState { /* ... */ });
```

This is how saalis can give the web layer its own `WebState` while nesting API sub-routers
that carry different service handles.

### Layer Composition

Middleware is applied via `Router::layer()` (wraps all previously-added routes, including the
fallback) or `Router::route_layer()` (wraps only matched routes, not the fallback). The
distinction matters for auth:

```rust
let app = Router::new()
    .route("/dashboard", get(dashboard))
    .route_layer(RequireAuth::new())  // 404s bypass auth check
    .route("/login", get(login_page));
// GET /dashboard without auth -> 401
// GET /nonexistent without auth -> 404 (not 401)
```

Middleware ordering follows the onion model: layers added later wrap layers added earlier.
`tower::ServiceBuilder` reverses this -- layers listed first execute first (top-to-bottom),
which is easier to reason about:

```rust
let app = Router::new()
    .route("/", get(handler))
    .layer(
        ServiceBuilder::new()
            .layer(TraceLayer::new_for_http())   // executes first
            .layer(CompressionLayer::new())       // executes second
            .layer(CorsLayer::permissive()),      // executes third
    );
```

## 2. Extractors -- Custom Extractors, FromRequestParts vs FromRequest, Rejection Handling

### The Two Traits

| Trait | Consumes Body? | Position Constraint | Use Case |
|-------|---------------|---------------------|----------|
| `FromRequestParts<S>` | No | Any position | Headers, path, query, state, cookies, session |
| `FromRequest<S>` | Yes | Must be **last** parameter | JSON body, form data, multipart |

Extractors run left-to-right in the handler signature. The type system enforces that at most
one `FromRequest` extractor appears, and only as the final parameter.

### Custom Extractor: ResolvedUser Pattern

For saalis, a `ResolvedUser` extractor that reads the session, loads the user entity, and
checks role permissions is a natural fit for `FromRequestParts`:

```rust
pub struct ResolvedUser {
    pub entity_id: EntityId,
    pub role: Role,
}

#[async_trait]
impl<S> FromRequestParts<S> for ResolvedUser
where
    S: Send + Sync,
{
    type Rejection = AuthError;

    async fn from_request_parts(
        parts: &mut Parts,
        state: &S,
    ) -> Result<Self, Self::Rejection> {
        // Extract session from parts (tower-sessions makes Session an extractor)
        let session = Session::from_request_parts(parts, state)
            .await
            .map_err(|_| AuthError::SessionMissing)?;

        let user_id: EntityId = session
            .get("user_id")
            .await
            .map_err(|_| AuthError::SessionCorrupt)?
            .ok_or(AuthError::NotAuthenticated)?;

        // In a sync-core design, the actual DB lookup would go through
        // spawn_blocking or a pre-resolved cache in request extensions.
        Ok(ResolvedUser {
            entity_id: user_id,
            role: session.get("role").await?.unwrap_or(Role::User),
        })
    }
}
```

### Rejection Handling

Rejections are the error half of an extractor's `Result`. They must implement `IntoResponse`.
Two strategies:

**Strategy A: Wrap in Result in the handler** -- lets the handler decide how to respond:

```rust
async fn handler(user: Result<ResolvedUser, AuthError>) -> impl IntoResponse {
    match user {
        Ok(u) => render_dashboard(u),
        Err(AuthError::NotAuthenticated) => Redirect::to("/login").into_response(),
        Err(e) => (StatusCode::FORBIDDEN, e.to_string()).into_response(),
    }
}
```

**Strategy B: Implement IntoResponse on the rejection** -- the extractor itself decides:

```rust
impl IntoResponse for AuthError {
    fn into_response(self) -> Response {
        match self {
            Self::NotAuthenticated => Redirect::to("/login").into_response(),
            Self::Forbidden => (StatusCode::FORBIDDEN, "Access denied").into_response(),
            Self::SessionCorrupt => {
                (StatusCode::INTERNAL_SERVER_ERROR, "Session error").into_response()
            }
        }
    }
}
```

Strategy B is cleaner for consistent auth behavior across many handlers.

### Composing Extractors Inside Custom Extractors

Use `RequestPartsExt` to call other extractors from within your `from_request_parts`:

```rust
use axum::RequestPartsExt;

let State(app): State<Arc<AppState>> = parts.extract().await?;
let Extension(db): Extension<DbPool> = parts.extract().await?;
```

## 3. Response Types -- IntoResponse, Typed Headers, HTML Responses for Maud

### IntoResponse Trait

Anything returned from a handler must implement `IntoResponse`. Axum provides blanket
implementations for common types:

- `()` -- 200 OK with empty body
- `&'static str`, `String` -- 200 OK with text/plain
- `StatusCode` -- that status with empty body
- `(StatusCode, impl IntoResponse)` -- override status code
- `(HeaderMap, impl IntoResponse)` -- append headers
- `(StatusCode, HeaderMap, impl IntoResponse)` -- both
- Tuples with up to 16 `IntoResponseParts` elements before the final body type
- `Json<T>` -- 200 OK with application/json
- `Response<Body>` -- pass through directly

### Maud Integration

With the `"axum"` feature enabled, Maud's `Markup` type implements `IntoResponse` directly,
returning `text/html; charset=utf-8`:

```rust
use maud::{html, Markup};

async fn dashboard(user: ResolvedUser) -> Markup {
    html! {
        h1 { "Welcome, " (user.entity_id) }
        div #content hx-get="/api/content" hx-trigger="load" {
            "Loading..."
        }
    }
}
```

For HTMX partial responses (fragments), the same type works -- HTMX swaps the returned HTML
into the target element regardless of whether it is a full page or a fragment.

### HTMX Response Headers

The `axum-htmx` crate provides responders that implement `IntoResponseParts`, allowing
composition via tuples:

```rust
use axum_htmx::{HxRetarget, HxReswap};

async fn handler() -> impl IntoResponse {
    (
        HxRetarget("#notifications".to_string()),
        HxReswap("afterbegin".parse().unwrap()),
        html! { div .toast { "Item saved" } },
    )
}
```

The `htmxtools` crate offers an alternative approach built on axum-extra's typed headers for
more structured header support.

### Typed Headers

For custom headers beyond HTMX, use axum-extra's `TypedHeader` extractor and response type,
which wraps headers from the `headers` crate with compile-time validation of header names and
values.

## 4. Middleware -- Tower Layers, Rate Limiting, CORS, Compression

### Tower Integration

Axum does not have its own middleware system. It integrates with Tower directly, meaning the
entire tower and tower-http ecosystem works out of the box. Middleware can be applied at three
levels:

1. **Router-level**: `Router::layer()` / `Router::route_layer()`
2. **MethodRouter-level**: `MethodRouter::layer()` / `MethodRouter::route_layer()`
3. **Handler-level**: `Handler::layer()`

### Writing Middleware

Three approaches in increasing order of control and complexity:

1. **`axum::middleware::from_fn`** -- async closure, easiest for application-specific logic:

```rust
async fn log_request(
    req: Request,
    next: middleware::Next,
) -> impl IntoResponse {
    let method = req.method().clone();
    let uri = req.uri().clone();
    let response = next.run(req).await;
    tracing::info!(%method, %uri, status = %response.status());
    response
}

let app = Router::new()
    .route("/", get(handler))
    .layer(axum::middleware::from_fn(log_request));
```

2. **`axum::middleware::from_extractor`** -- uses an extractor as a guard; the request
   proceeds only if extraction succeeds:

```rust
let app = Router::new()
    .route("/admin", get(admin_panel))
    .route_layer(axum::middleware::from_extractor::<RequireAdmin>());
```

3. **Custom `tower::Layer` + `tower::Service`** -- full control, required for publishing
   reusable middleware crates.

### CORS

Use `tower_http::cors::CorsLayer`:

```rust
let cors = CorsLayer::new()
    .allow_origin("https://saal.is".parse::<HeaderValue>().unwrap())
    .allow_methods([Method::GET, Method::POST])
    .allow_headers([CONTENT_TYPE, AUTHORIZATION]);

let app = Router::new()
    .nest("/api", api_routes)
    .layer(cors);
```

Per-route CORS is achieved by applying different `CorsLayer` instances via `route_layer()` on
different sub-routers before merging.

### Compression

`tower_http::compression::CompressionLayer` compresses responses automatically based on
Accept-Encoding. For an HTMX app serving many small HTML fragments, gzip/brotli compression
yields significant bandwidth savings:

```rust
let app = Router::new()
    .route("/", get(index))
    .layer(CompressionLayer::new());
```

### Rate Limiting

No built-in rate limiter; use `tower-governor` (mature, based on the `governor` crate):

```rust
use tower_governor::{GovernorLayer, GovernorConfigBuilder};

let governor_conf = GovernorConfigBuilder::default()
    .per_second(10)
    .burst_size(30)
    .finish()
    .unwrap();

let app = Router::new()
    .route("/api/search", get(search))
    .layer(GovernorLayer { config: governor_conf });
```

Per-route rate limiting: apply different `GovernorLayer` configs to different sub-routers
before merging. The `axum-governor` crate (unrelated to tower-governor) offers route-specific
and global modes with override capabilities.

## 5. SSE -- Sse Type, Event Builder, Keep-Alive, Backpressure

### Core Types

Axum provides `axum::response::sse::Sse` and `axum::response::sse::Event`. `Sse::new()`
accepts any `Stream<Item = Result<Event, E>>` where `E: Into<BoxError>`.

### Event Builder

```rust
use axum::response::sse::Event;

Event::default()
    .event("notification")       // event type (client-side addEventListener target)
    .data("payload here")        // data field; newlines auto-split into multiple data: lines
    .id("evt-42")                // lastEventId for reconnection
    .retry(Duration::from_secs(5))  // client reconnection interval
    .comment("heartbeat")        // SSE comment line (ignored by most clients)
```

The `.json_data(value)` method (requires the `json` feature) serializes a value as JSON into
the data field.

Panics: `.event()`, `.id()`, and `.retry()` panic if called more than once per Event.
`.id()` panics if the value contains null characters. `.comment()` panics on newlines.

### Keep-Alive

```rust
use axum::response::sse::{Sse, KeepAlive};

async fn event_stream() -> Sse<impl Stream<Item = Result<Event, Infallible>>> {
    let stream = /* ... */;
    Sse::new(stream)
        .keep_alive(
            KeepAlive::new()
                .interval(Duration::from_secs(15))
                .text("ping")
        )
}
```

`KeepAlive::default()` sends a comment-only event (`: keep-alive`) every 15 seconds. For
HTMX, which uses `EventSource` under the hood, this prevents browser-side timeout and
reconnection.

### Backpressure and Slow Clients

SSE is a unidirectional push protocol -- there is no client-side flow control signal. If a
client cannot consume events fast enough, the TCP send buffer fills, which causes the server's
`poll_ready` / write futures to suspend. This ties up a task in the Tokio runtime.

Strategies for saalis:

1. **Bounded broadcast channel with `Lagged` handling**: Use `tokio::sync::broadcast` with a
   bounded capacity. Slow receivers get a `RecvError::Lagged(n)` error, meaning `n` events
   were dropped. The SSE stream can send a "you missed N events, full-refresh" event:

```rust
let (tx, _) = tokio::sync::broadcast::channel::<Notification>(64);

// In the SSE handler:
let mut rx = tx.subscribe();
let stream = async_stream::stream! {
    loop {
        match rx.recv().await {
            Ok(notification) => {
                yield Ok(Event::default()
                    .event("notification")
                    .json_data(&notification)
                    .unwrap());
            }
            Err(broadcast::error::RecvError::Lagged(n)) => {
                yield Ok(Event::default()
                    .event("lagged")
                    .data(format!("{n}")));
            }
            Err(broadcast::error::RecvError::Closed) => break,
        }
    }
};
```

2. **Per-client mpsc with timeout**: Give each client its own `mpsc::Sender`. If the channel
   is full (`.try_send()` returns `Full`), drop the event or disconnect the client. This
   avoids one slow client affecting broadcast throughput.

3. **Client disconnect detection**: When the client drops the connection, the stream's
   `poll_next` will eventually return an error or the `Sse` response future will be dropped.
   Use `tokio_util::sync::CancellationToken` per-client to clean up resources.

## 6. State Management -- Arc<AppState> vs Extension, Avoiding Mutex

### State<T> Extractor (Recommended)

`axum::extract::State<T>` is the primary mechanism. The state must implement `Clone` because
axum clones it for each request. The standard pattern wraps expensive-to-clone data in `Arc`:

```rust
#[derive(Clone)]
struct AppState {
    db: Arc<DatabasePool>,
    config: Arc<Config>,
    notification_tx: broadcast::Sender<Notification>,
}

let state = AppState { /* ... */ };
let app = Router::new()
    .route("/", get(handler))
    .with_state(state);

async fn handler(State(state): State<AppState>) -> impl IntoResponse {
    // state.db, state.config available
}
```

### State vs Extension

`Extension<T>` (from `tower_http`) predates `State` and is less type-safe: if the extension
is missing at runtime, the handler panics or returns 500. `State<T>` is checked at compile
time -- if you forget `.with_state()`, the router won't be `Router<()>` and `axum::serve()`
won't accept it. Prefer `State` for application data.

`Extension` is still useful for per-request data injected by middleware (e.g., a resolved user
identity set by an auth middleware layer).

### Avoiding Mutex in State

For saalis's sync-core design, the web layer state should be as lock-free as possible:

- **Immutable after init**: `Arc<T>` alone -- no lock needed. Configuration, static
  registries, compiled templates all fall here.
- **Read-heavy, rare writes**: `arc_swap::ArcSwap<T>` -- readers get a snapshot without
  locking; writers atomically swap the entire value. Good for hot-reloadable config.
- **Write-through channels**: Instead of `Mutex<Vec<WorkUnit>>`, use
  `mpsc::Sender<WorkUnit>` in state. The handler sends the work unit and returns immediately
  (fire-and-forget). The scheduler owns the receiver and processes work units on its own
  thread.
- **Concurrent maps**: `dashmap::DashMap` for shared lookup tables that need concurrent
  read/write (e.g., active SSE client registry).

Holding `std::sync::Mutex` across `.await` points produces `!Send` futures, which axum
rejects. If you must use a mutex across await, use `tokio::sync::Mutex` -- but the channel or
`arc_swap` patterns above are almost always better.

## 7. Graceful Shutdown -- Signal Handling, Connection Draining, Background Task Cleanup

### Signal Handling

The standard pattern from axum's official examples:

```rust
async fn shutdown_signal() {
    let ctrl_c = async {
        tokio::signal::ctrl_c()
            .await
            .expect("failed to install Ctrl+C handler");
    };

    #[cfg(unix)]
    let terminate = async {
        tokio::signal::unix::signal(
            tokio::signal::unix::SignalKind::terminate(),
        )
        .expect("failed to install SIGTERM handler")
        .recv()
        .await;
    };

    #[cfg(not(unix))]
    let terminate = std::future::pending::<()>();

    tokio::select! {
        _ = ctrl_c => {},
        _ = terminate => {},
    }
}
```

### Connection Draining

`axum::serve()` returns a `Serve` future with `.with_graceful_shutdown(signal)`. Once the
signal resolves, the server stops accepting new connections and waits for in-flight requests
to complete:

```rust
let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await?;
axum::serve(listener, app)
    .with_graceful_shutdown(shutdown_signal())
    .await?;
```

For finer control (e.g., a maximum drain duration for Docker's SIGTERM grace period), use
`axum_server::Handle`:

```rust
let handle = axum_server::Handle::new();
let h = handle.clone();

tokio::spawn(async move {
    shutdown_signal().await;
    h.graceful_shutdown(Some(Duration::from_secs(10))); // 10s drain
});

axum_server::bind(addr)
    .handle(handle)
    .serve(app.into_make_service())
    .await?;
```

### Background Task Cleanup

For saalis's scheduler and subsystems (Curator, Housekeeper, Doctor), use a shared
`CancellationToken`:

```rust
let token = CancellationToken::new();

// Spawn background work
let bg_token = token.clone();
tokio::spawn(async move {
    loop {
        tokio::select! {
            _ = bg_token.cancelled() => break,
            work = rx.recv() => { /* process */ }
        }
    }
    // cleanup: flush pending writes, close DB connections
});

// In shutdown:
shutdown_signal().await;
token.cancel();  // signals all background tasks
// axum's graceful_shutdown drains HTTP; token drains background work
```

## 8. Error Handling -- Mapping Domain Errors to HTTP Responses

### The Pattern

Define a domain error enum, implement `IntoResponse`, and use `Result<T, AppError>` as the
handler return type. The `?` operator then works with any error type that has a `From` impl:

```rust
#[derive(Debug, thiserror::Error)]
pub enum AppError {
    #[error("Not found: {0}")]
    NotFound(String),

    #[error("Forbidden")]
    Forbidden,

    #[error("Validation: {0}")]
    Validation(String),

    #[error(transparent)]
    Internal(#[from] anyhow::Error),
}

impl IntoResponse for AppError {
    fn into_response(self) -> Response {
        let (status, message) = match &self {
            Self::NotFound(msg) => (StatusCode::NOT_FOUND, msg.as_str()),
            Self::Forbidden => (StatusCode::FORBIDDEN, "Access denied"),
            Self::Validation(msg) => (StatusCode::UNPROCESSABLE_ENTITY, msg.as_str()),
            Self::Internal(e) => {
                tracing::error!(error = %e, "Internal error");
                (StatusCode::INTERNAL_SERVER_ERROR, "Internal server error")
            }
        };

        // For HTMX: return an HTML error fragment, not JSON
        (status, html! {
            div .error-toast role="alert" { (message) }
        }).into_response()
    }
}
```

### Extracting Error Context Without Side Effects

Avoid logging inside `IntoResponse` -- instead, use a middleware or tower layer that inspects
response extensions:

```rust
impl IntoResponse for AppError {
    fn into_response(self) -> Response {
        let status = self.status_code();
        let mut response = (status, self.user_message()).into_response();
        response.extensions_mut().insert(self); // attach for middleware
        response
    }
}
```

A tracing middleware layer can then extract the `AppError` from extensions and log it with
full context, keeping the `IntoResponse` impl pure.

### Fallible Extractors

When an extractor fails (e.g., `Json` receives malformed input), axum calls the rejection's
`IntoResponse`. To customize these rejections globally, wrap the built-in extractor:

```rust
pub struct ValidJson<T>(pub T);

#[async_trait]
impl<S, T> FromRequest<S> for ValidJson<T>
where
    Json<T>: FromRequest<S>,
    S: Send + Sync,
{
    type Rejection = AppError;

    async fn from_request(req: Request, state: &S) -> Result<Self, Self::Rejection> {
        Json::<T>::from_request(req, state)
            .await
            .map(|Json(v)| ValidJson(v))
            .map_err(|e| AppError::Validation(e.to_string()))
    }
}
```

## 9. Testing -- tower::ServiceExt::oneshot, TestClient Patterns

### The oneshot Pattern (Recommended)

Because axum's `Router` implements `tower::Service`, you can test it without starting an HTTP
server. Import `tower::ServiceExt` for the `.oneshot()` method:

```rust
#[tokio::test]
async fn test_health_check() {
    let app = app(test_state());  // returns Router<()>

    let response = app
        .oneshot(
            Request::builder()
                .uri("/health")
                .body(Body::empty())
                .unwrap()
        )
        .await
        .unwrap();

    assert_eq!(response.status(), StatusCode::OK);

    let body = response.into_body().collect().await.unwrap().to_bytes();
    assert_eq!(&body[..], b"ok");
}
```

Key constraint: `.oneshot()` consumes the service. For multiple requests in one test, clone
the router or rebuild it, or use `.call()` (requires `poll_ready` first).

### State in Tests

`oneshot` only works on `Router<()>`. Always call `.with_state(test_state)` before testing:

```rust
fn test_app() -> Router<()> {
    Router::new()
        .route("/", get(handler))
        .with_state(AppState::test_default())
}
```

### TestClient (axum-test crate)

For more ergonomic multi-request testing with cookie jars, session persistence, and response
helpers:

```rust
use axum_test::TestServer;

#[tokio::test]
async fn test_login_flow() {
    let server = TestServer::new(app(test_state())).unwrap();

    // Login
    let response = server
        .post("/login")
        .form(&[("username", "admin"), ("password", "secret")])
        .await;
    response.assert_status(StatusCode::SEE_OTHER);

    // Authenticated request (cookies auto-forwarded)
    let response = server.get("/dashboard").await;
    response.assert_status_ok();
    response.assert_text_contains("Welcome");
}
```

This is particularly valuable for testing session-based auth flows in saalis.

### Testing SSE

For SSE endpoints, use `reqwest::Client` with `.get(url).send().await` and read the
`EventSource` stream, or use `oneshot` and collect the response body as a stream, asserting
on individual events.

## 10. Performance -- Connection Limits, Thread Pool Sizing, Benchmarks

### Benchmarks: Axum vs Actix-Web vs Rocket

Based on 2025 benchmarks (TechEmpower and independent tests):

| Metric | Axum | Actix-Web | Rocket |
|--------|------|-----------|--------|
| Requests/sec (plaintext) | Very high (within ~5% of Actix) | Highest overall | Significantly lower |
| Latency p99 | Comparable to Actix | Comparable to Axum | Higher |
| Memory per connection | Lowest | Slightly higher | Higher |
| Max concurrent connections | High | Highest (tuned by default) | Lower |

Actix-Web has historically topped TechEmpower benchmarks across all languages. Axum is within
striking distance on throughput and often wins on memory efficiency. For saalis running on a
Batocera appliance with constrained resources, Axum's lower memory footprint and Tokio
ecosystem integration make it the better fit. The performance difference is negligible for a
single-user/household appliance.

### Connection Limits

Axum (via hyper/tokio) does not impose a default connection limit. Connections are bounded by:
- OS file descriptor limits (`ulimit -n`)
- Available memory for connection state
- Tokio's task scheduler capacity

For an appliance, consider using `tower::limit::ConcurrencyLimitLayer` to cap concurrent
requests:

```rust
use tower::limit::ConcurrencyLimitLayer;

let app = Router::new()
    .route("/", get(handler))
    .layer(ConcurrencyLimitLayer::new(100));  // max 100 concurrent requests
```

### Thread Pool Sizing for spawn_blocking

Tokio's blocking thread pool (used by `spawn_blocking`) defaults to 512 threads. For saalis's
sync-core pattern where SQLite access goes through `spawn_blocking`:

- The pool is lazy -- threads are only spawned when needed
- Each `spawn_blocking` call occupies one thread until the closure returns
- If all threads are busy, new calls queue until a thread frees up
- Configure via `tokio::runtime::Builder::max_blocking_threads()`

For an appliance with 2-4 cores, a blocking pool of 8-16 threads is more appropriate than the
default 512:

```rust
let runtime = tokio::runtime::Builder::new_multi_thread()
    .worker_threads(2)           // async workers = CPU cores
    .max_blocking_threads(8)     // sync workers for SQLite/IO
    .enable_all()
    .build()
    .unwrap();
```

### Appliance-Specific Tuning

- **Worker threads**: Match CPU core count (Batocera devices: typically 2-4 cores)
- **Blocking threads**: 2-4x core count for SQLite-heavy workloads
- **Keep-alive timeout**: Reduce from default 75s to 30s to free connections faster on
  memory-constrained devices
- **Body size limit**: `DefaultBodyLimit::max(2 * 1024 * 1024)` (2MB default is fine; lower
  for API-only routes)

## Session Management: tower-sessions

For completeness, since saalis needs session management for its web UI:

`tower-sessions` is the current standard for cookie-based sessions in axum. It provides
`Session` as both a tower middleware layer and an axum extractor:

```rust
use tower_sessions::{SessionManagerLayer, MemoryStore, Session};

let session_store = MemoryStore::default();
// For persistence across restarts, use a SQLite-backed store

let app = Router::new()
    .route("/", get(handler))
    .layer(SessionManagerLayer::new(session_store));

async fn handler(session: Session) -> impl IntoResponse {
    let count: i32 = session.get("visits").await.unwrap().unwrap_or(0);
    session.insert("visits", count + 1).await.unwrap();
    format!("Visit #{}", count + 1)
}
```

For auth, `axum-login` builds on top of `tower-sessions`, providing `AuthSession` as an
extractor with `login()`, `logout()`, and permission-checking methods. It supports an
`AuthzBackend` trait for RBAC, which maps well to saalis's fixed `Role` enum (Admin, User,
Child).

## Key Takeaways for saalis-web

1. **Router structure**: Nest admin/api/public routes with separate state types, finalize each
   sub-router with `.with_state()` before nesting.
2. **ResolvedUser extractor**: Implement `FromRequestParts` with session lookup and role
   resolution. Return `AuthError` rejection that renders an HTMX-friendly error fragment or
   redirect.
3. **Maud responses**: Return `Markup` directly from handlers. Use `(HxHeader, Markup)` tuples
   for HTMX response headers.
4. **Middleware layering**: Use `route_layer` for auth so 404s are not gated. Apply
   compression and CORS at the outer router level.
5. **SSE for notifications**: Use `broadcast::channel` with `Lagged` handling. Keep-alive at
   15s. Per-client `CancellationToken` for cleanup.
6. **State**: `Arc<AppState>` with channel senders for fire-and-forget writes. No Mutex in
   hot paths.
7. **Graceful shutdown**: `with_graceful_shutdown` for HTTP draining + `CancellationToken` for
   scheduler/subsystem cleanup.
8. **Error handling**: Single `AppError` enum with `IntoResponse` returning HTML fragments for
   HTMX error display.
9. **Testing**: `tower::ServiceExt::oneshot` for unit tests; `axum-test` for session/auth
   integration tests.
10. **Performance**: Axum is within ~5% of Actix-Web throughput with lower memory. Configure
    2 async workers + 8 blocking threads for appliance hardware.
