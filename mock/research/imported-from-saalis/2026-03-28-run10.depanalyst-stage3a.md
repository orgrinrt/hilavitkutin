## 1. How the Savant's Mechanisms Affect the Dependency Graph

### 1.1 Convergence Damping Coefficient (Section 3.1) -- New Conditional Edge

The damping coefficient introduces a new runtime coupling: HealthPipeline must observe ConvergenceGate state. Currently, HealthPipeline is a mesh-internal mechanism that processes heartbeat observations. The damping mask requires HealthPipeline to know whether convergence is active. This creates a new internal edge within verkko-mesh:

```
HealthPipeline -> ConvergenceGate (new, INTERNAL to mesh)
```

This is not a cross-document edge. Both mechanisms live in mesh. The coupling is acceptable because it is contained within a single document. However, it creates a bidirectional internal cycle: ConvergenceGate uses HealthPipeline output (health scores feed into gate predicates), and now HealthPipeline must observe ConvergenceGate state (to apply the damping mask). This internal cycle is the damping mechanism itself; it is intentional and well-defined.

**Graph impact:** No new cross-document edges. No topological changes. Internal mesh complexity increases.

### 1.2 Scar Severity Tiering (Section 3.2) -- Changed Edge Weight

Scar severity tiering changes the edge between verkko-mesh (ScarAttribution, dominance scoring) and verkko-matter (IntegrityFailure). My Stage 1 review identified this as edge H8 (mesh -> matter, IMPLICIT, severity Medium). The tiering proposal places IntegrityFailure in Tier 1 (transient, 0.5x weight, 90-day decay).

The tier assignment itself does not change the dependency graph structure. The edge H8 remains: mesh's ScarAttribution::IntegrityFailure still references matter's blob_id. Tiering changes the runtime weight, not the compile-time coupling. My Stage 2a recommendation (genericize IntegrityFailure to use `content_hash` instead of `blob_id`) remains necessary regardless of tier assignment.

**Graph impact:** No structural change. The H8 edge persists. Tiering is a runtime behavior change, not a dependency change.

### 1.3 Lattice Rigidity Layer / Wire Contracts (Section 3.3) -- Removed Ambiguity, No New Edges

The prioritized wire contract approach (grain boundaries first, internals last) does not change the dependency graph. Wire contracts pin the byte-level semantics of existing defined concepts. They do not create new cross-document dependencies. A wire contract for HeartbeatExtension formalizes the existing mesh -> protocol interface; it does not create a new edge.

However, the Savant's prioritization reveals something I missed. The "Priority 1" grain boundary concepts (HeartbeatExtension, ConvergenceGate, SourceSelectionPredicate, DomainSeparatorRegistry entries) are the exact concepts where two documents share ownership. From a dependency graph perspective, shared ownership is a bidirectional edge masquerading as a defined concept. Wire contracts at these points do not remove the bidirectional coupling; they make it explicit and stable. The edge becomes a formal contract rather than an informal agreement.

**Graph impact:** No new edges. Existing edges become more precisely defined. Edge stability under evolution improves.

### 1.4 Seat Isolation Valve (Section 3.4) -- New Edge, Properly Contained

The seat isolation valve introduces a new internal mechanism in the HostPeer: `snapshot_state` and `live_state`. This creates a new runtime edge within the host peer's scope:

```
seat requests -> snapshot_state (read-only during convergence)
mesh operations -> live_state (mutable during convergence)
snapshot_state <- live_state (snapshot taken at convergence start)
```

From a cross-document dependency perspective, this does not change the graph. Seat still depends on crypto and protocol. Seat still does NOT depend on mesh. The HostPeer remains the boundary. The valve is an implementation mechanism within the HostPeer's mediation contract.

The important constraint: snapshot_state must not expose mesh-internal concepts to the seat. If the snapshot includes convergence metadata (e.g., "this data is stale because convergence is active"), that is a new seat -> mesh information flow. The Savant's specification avoids this: seat queries return data from the snapshot without convergence context. This is correct. The seat sees entity metadata and stash status, not convergence state.

**Graph impact:** No new cross-document edges. The seat -> mesh vacancy (edge 19 in my Stage 2a table, rated F) remains absent, which is correct by design. The valve fills the mediation gap without creating a dependency.

### 1.5 Purpose-Specific Health Projections (Section 4.3) -- New Internal Edges, One External Risk

The Savant proposes four projection functions:
```
health.for_source_selection()   -> matter consumes
health.for_gateway_election()   -> mesh consumes
health.for_metabolic_state()    -> protocol (via Conductor in mesh) consumes
health.for_convergence_gate()   -> mesh/relay consumes
```

Currently, HealthPipeline produces a single scalar that consumers interpret. The projections formalize what is already happening informally. From a dependency graph perspective:

- `for_source_selection()`: matter already depends on mesh (declared). The projection does not create a new edge; it refines the existing matter -> mesh interface.
- `for_gateway_election()`: mesh-internal. No new edge.
- `for_metabolic_state()`: mesh-internal (Conductor consumes it within mesh). No new edge. Protocol's MetabolicState receives state transitions, not raw health projections.
- `for_convergence_gate()`: relay currently reads mesh's convergence state. If relay consumes `health.for_convergence_gate()` directly from HealthPipeline, this is still mediated through mesh (relay depends on mesh, declared). No new edge.

**External risk:** If projections are exposed as defined concepts in mesh, and matter or relay references specific projection semantics (e.g., "source selection uses throughput-weighted health"), those consumers become coupled to the projection's weighting formula. A change to the weighting formula would propagate to consumers. This is the same brittleness the Savant identified for HeartbeatExtension: heavily consumed interfaces are brittle regardless of cleanliness.

**Graph impact:** No new cross-document edges. Internal mesh complexity increases. Evolution brittleness depends on whether projections are treated as opaque (stable) or transparent (brittle).

## 2. What the Proposals Mean for Evolution Independence

### 2.1 ConvergenceGate as a Decoupling Firewall

The Savant's sharpened ConvergenceGate proposal (Section 8.2) is the most important mechanism for evolution independence in the entire review series. The invariant -- "Consumers of ConvergenceGate must not inspect the inputs to the gate predicates" -- is a dependency barrier.

Taking this further: ConvergenceGate should be the sole interface through which relay observes mesh's convergence state. Currently, relay's body text references specific mesh thresholds ("Front 1 complete when GSet merge ACKed by all live peers"). If relay observes a ConvergenceGate predicate (`front_1_complete: bool`) instead, relay is insulated from mesh's internal GSet merge logic. Mesh can change the merge algorithm, change the ACK mechanism, change the liveness definition, and relay sees only the boolean.

**Evolution independence gain:** Relay can evolve independently of mesh's convergence internals. Mesh can evolve its convergence logic without breaking relay. The ConvergenceGate is a one-way mirror: relay sees outcomes, not mechanisms.

This principle should extend to all grain boundary defined concepts. HeartbeatExtension should be the sole carrier of mesh-layer state observable by remote peers (the Savant's restored sharp edge in 8.1). SourceSelectionPredicate should be the sole interface through which matter selects sources. Each grain boundary concept should be an opaque projection, not a transparent window.

If this principle is applied consistently, the graph's evolution behavior changes fundamentally. Currently, a change to mesh's internal health scoring formula propagates to every consumer that reads the formula. With opaque grain boundary projections, the change propagates only to the projection function. The consumers see the same interface. This converts the graph from a tightly coupled lattice (every node coupled to every neighbor's internals) into a loosely coupled DAG (nodes coupled only to defined interfaces).

**Quantified improvement:** My Stage 1 evolution analysis for Scenario D (relay changes hash ring) showed mesh needing changes because "convergence Front 2 step 4 mentions hash ring recomputation." With ConvergenceGate as firewall, this propagation is blocked. Mesh sees relay's re-encryption completion through the gate, not through relay's internal hash ring state. The blast radius of Scenario D drops from 3-4 documents to 1-2.

### 2.2 Named Fronts and Insertion Stability

Named fronts (ControlFront, ConsensusFront, ReencryptionFront) convert position-dependent references into identity-stable references. From an evolution perspective, this means the graph is stable under insertion operations.

Taking this further: the naming principle should apply to all cross-document ordinal references. If any document references another document's concept by position (e.g., "Queue 3" for bulk transfer, or "Structure 7" in the Pattern Registry), that reference is fragile under insertion. The graph edge is correct today but breaks when the target document inserts a new item before the referenced position.

I identified edge H10 (mesh -> ops via Pattern Registry Structure references) as LOW severity in Stage 1. The named fronts principle reveals that ALL Pattern Registry structure numbers create the same fragility. "Structure 3 (AEAD cascade)" in verkko-ops could be renumbered if new structures are added. The back-references from crypto S2.4 and relay S4.4 to "Structure 3" and "Structure 7" would break.

The fix is the same as for convergence fronts: name the structures. "AEAD Cascade Pattern" instead of "Structure 3." This is a LOW priority fix because ops is a leaf node and the back-references are informational, but it is the same structural principle.

### 2.3 The Savant's "Amorphous Lattice" Has a DAG Interpretation

The Savant's claim that "the lattice is amorphous, not crystalline" (Section 6) translates to a precise DAG property: the edges are typed correctly (declared, implicit, hidden) but their contracts are underspecified. An edge in a dependency graph carries two properties: (1) existence (A depends on B) and (2) contract (A depends on B for specific behaviors X, Y, Z). The current graph has property (1) well-established but property (2) partially specified.

Wire contracts add property (2). Without them, edge stability assessments are conditional. My Stage 1 scores assumed that defined concept invariants constitute sufficient contracts. The Devil's Advocate's HKDF example proved this wrong: two implementations can satisfy the prose invariants while producing incompatible byte sequences. My edge scores are upper bounds, not measurements.

**Revised interpretation of my scorecard:** All A and A+ scores in my Stage 1 health scorecard (Section 10) should be read as "A (conditional on wire contracts)." The actual scores are unknown until wire contracts exist. This does not change the graph structure or the ordering of fixes; it changes the confidence interval around each score.

## 3. Mechanisms That Create New Coupling

### 3.1 Convergence Damping Creates a Temporal Coupling Hazard

The damping coefficient (Section 3.1) introduces a temporal coupling between HealthPipeline and ConvergenceGate that is not present in the current design. Currently, HealthPipeline processes observations statelessly with respect to convergence. The damping mask makes HealthPipeline's behavior depend on convergence timing.

This temporal coupling creates a new failure mode: if the ConvergenceGate's open/close transitions are buggy (gate stays open too long, or flaps), the damping mask stays applied incorrectly, producing systematically biased health scores. The bias is invisible to consumers because they see health scores, not the damping state.

From a dependency perspective, this is an internal mesh coupling, not a cross-document edge. But it is worth noting: the damping mechanism trades feedback loop stability for internal coupling complexity. The trade is correct (the feedback loop is more dangerous than the coupling), but the coupling should be documented as a design-intentional bidirectional binding within mesh.

### 3.2 Purpose-Specific Projections Risk Interface Proliferation

The four projection functions (`for_source_selection`, `for_gateway_election`, `for_metabolic_state`, `for_convergence_gate`) create four interfaces where one existed before. Each projection is a potential coupling point. If matter depends on `for_source_selection()` and the projection formula changes, matter's source selection behavior changes. This is correct behavior (the formula should change when health semantics change), but it multiplies the number of interfaces that consumers are coupled to.

The risk is interface proliferation: future designers may add `for_download_priority()`, `for_relay_selection()`, `for_display()`, creating unbounded growth. Each new projection is a new coupling point.

The mitigation is the Savant's own ConvergenceGate principle: projections should be opaque. Consumers should not know the weighting formula. They should receive a scalar and trust it. If projections are opaque, they are stable under formula changes. If they are transparent (documenting that `for_source_selection` weights throughput 0.6 and packet loss 0.4), every formula change propagates to consumer documentation.

**Recommendation:** If purpose-specific projections are adopted, they should be defined concepts with invariants that describe the semantic intent ("a score suitable for selecting the best data source") without specifying the formula. The formula is body content, not invariant content. This keeps the interface stable.

### 3.3 Scar Tiering Introduces a Cross-Concern Classification Problem

The three-tier scar classification (transient, concerning, severe) requires a mapping from scar types to tiers. This mapping is a new piece of shared knowledge. Currently, all scars are equal. With tiering, every new scar type must be assigned a tier. The tier assignment is a design decision, not an implementation decision.

Where does the tier mapping live? If it lives in mesh (alongside ScarAttribution), then mesh must classify scar types that originate in other documents. IntegrityFailure originates from matter-layer events. ConnectivityFlap originates from protocol-layer events. Mesh must classify events from lower and higher layers.

This does not create a new dependency edge (mesh already observes these events), but it creates a classification coupling: mesh's tier assignment for IntegrityFailure depends on understanding whether integrity failures are typically transient (cheap hardware) or persistent (actual data corruption). That understanding lives in matter, not in mesh. The tier assignment is an implicit dependency on matter's domain knowledge.

**Recommendation:** Tier assignment should be a property of the scar type definition, not a separate mapping table. If ScarAttribution::IntegrityFailure is defined in mesh, its tier (Tier 1, transient) should be part of its definition. If scar types are eventually moved to their originating documents (matter defines IntegrityFailure, protocol defines ConnectivityFlap), the tier travels with the type. This keeps classification local to the concept owner.

## 5. Anti-Agreement: The Savant Is Wrong About Seat Isolation Valve Timing

The Savant proposes the seat isolation valve (Section 3.4) at priority position #6 in the repair list, after the grain boundary formalizations (HeartbeatExtension, ConvergenceGate, named fronts), after wire contracts, and after domain separator registration. The Savant places it here because it "fills the largest vacancy with a user-experience-aware design."

I disagree with both the priority and the framing.

**On priority:** The seat isolation valve is an implementation mechanism, not a design specification concern. It specifies how the HostPeer manages state during convergence (snapshot vs. live). This is body content for the seat document, not a defined concept at a grain boundary. The grain boundary formalizations (priorities 1-3) affect the DAG structure. Wire contracts (priority 5) affect interoperability. The seat isolation valve affects user experience during a specific runtime scenario. These are different categories of concern. Mixing user experience mechanisms into a dependency graph repair list confuses structural fixes with behavioral designs.

The Savant correctly identifies that the seat isolation valve does not create new cross-document edges (I confirmed this in Section 1.4). It does not change the topological sort. It does not resolve any soft cycle. It does not formalize any implicit edge. From a dependency graph perspective, the seat isolation valve has zero structural impact. Placing it in the same priority list as ConvergenceGate (which converts an implicit edge to a declared interface) and HeartbeatExtension (which hardens the most consumed grain boundary) conflates two different kinds of work.

**On framing:** The Savant calls this "filling the largest vacancy." My Stage 2a review reclassified the seat -> mesh vacancy as intentional. The seat does not depend on mesh by design. The vacancy is not a hole to be filled; it is a wall to be maintained. The seat isolation valve does not "fill" the vacancy. It specifies how the HostPeer behaves at the wall. This is mediation contract specification, not vacancy repair. The vocabulary matters because calling it a vacancy repair implies the graph should eventually have a seat -> mesh edge. It should not. The valve preserves the absence of that edge while specifying the boundary behavior.

The seat isolation valve belongs in a separate work stream: "seat document body specification." It should be tracked alongside the GapHunter's S-1 through S-10 findings, not alongside the dependency graph structural repairs. Mixing them creates the impression that all nine items in the Savant's list are commensurable. They are not. Items 1-5 are structural. Items 6-9 are behavioral. Structural fixes change what the graph is. Behavioral designs change what the graph does. Both are necessary. They are not the same kind of work.

## 6. Summary

The Savant's Stage 2b proposals are structurally sound. None of the four mechanisms (convergence damping, scar tiering, wire contract prioritization, seat isolation valve) introduce new cross-document edges. Two mechanisms (ConvergenceGate as firewall, wire contracts at grain boundaries) improve existing edge quality. The graph remains acyclic with the same topological ordering.

The most important contribution is the ConvergenceGate firewall principle: "Consumers must not inspect gate inputs." If this invariant is adopted and applied to all grain boundary defined concepts, the graph's evolution behavior improves from "tightly coupled lattice" to "loosely coupled DAG with opaque interfaces." This is the single largest evolution independence gain available from any proposal in the review series.

The largest remaining structural defect is the undeclared matter -> protocol dependency (edge 12/20), which has been confirmed by all reviewers across all rounds and remains unfixed.

All edge scores in my health scorecard are now explicitly conditional on wire contracts existing. Without wire contracts, edge contract strength is unknown. The graph structure is correct; the graph semantics are underspecified.
