## My Disagreement

I disagree with the savant's relay-first/hole-punch-first traffic class
split. The savant writes:

> "Relay-first for control plane (heartbeats, CPSK, convergence).
> Hole-punch-first for data plane (matter transfers, bulk operations).
> Separate strategies for separate traffic classes."

This is conceptually clean but operationally expensive. It requires the
relay forwarding table to distinguish traffic classes. The relay sees
only encrypted outer headers (destination, connection_id, hop_count,
msg_type). The msg_type field identifies HEARTBEAT(0x01), DATA(0x02),
STREAM(0x03), etc. The relay would need to inspect msg_type to route
control frames through the relay path and data frames through the
direct path. But this means the relay must maintain two forwarding
decisions per connection pair: one for control traffic (always relay)
and one for data traffic (attempt direct, fall back to relay). The
consumer must maintain two NAT traversal state machines per connection:
one for the relay session (always up) and one for the direct session
(opportunistic). This doubles the connection state for every relayed
pair.

The correct approach is simpler: relay-first for all traffic during
session establishment and convergence. Once a direct path is
established, migrate all traffic classes to the direct path. Keep
the relay session alive as a warm standby. This is the model I
specified in Stage 1 and I maintain it. The savant's traffic class
split adds complexity without proportional benefit. I develop this
argument with numbers in Section 3.

## Section 1: WiFi-Aware Scar Attenuation and Relay Health Monitoring

### The Question

The savant proposes a three-layer scar classification:
- Layer 1: Thermal defects (WiFi-induced), 1-7 day adaptive decay.
- Layer 2: Structural defects (genuine faults), 180-day decay.
- Layer 3: Grain boundary defects (relay/path faults), 30-day decay.

Does the three-layer classification affect relay health monitoring? If
intra-cluster WiFi noise is filtered out by the thermal classification,
does the relay still see accurate health signals for topology decisions?

### Analysis

The relay does not participate in health scoring. The relay forwards
encrypted frames; it does not decrypt them; it does not read heartbeat
extensions; it does not compute EMA health tracks. The relay's only
health signal is the forwarding table activity: "is this endpoint
sending frames?" The relay's forwarding table has a last_seen timestamp
per endpoint. If last_seen exceeds the amplification prevention window
(30 seconds in Background mode, adjusted per my devil's advocate
recommendation to 2x heartbeat interval), the relay drops frames
from the silent endpoint's partner.

The three-layer scar classification operates at the mesh layer. It
classifies scars based on the diagnostic triangle (direct_health,
relay_health, wifi_quality). The relay health EMA track that feeds into
this triangle is computed by the mesh peers, not by the relay itself.
The mesh peers observe relay path performance by measuring SACK ratios
on traffic routed through the relay. These SACK ratios are unaffected
by the scar classification; they are raw transport-layer measurements.

So the three-layer classification does not degrade relay health signals.
The relay health EMA track continues to measure actual relay path
performance. The classification only determines what happens downstream:
whether a health degradation event produces a thermal scar (short decay)
or a grain boundary scar (30-day decay) or a structural scar (180-day
decay).

### Concrete Numbers

Consider a relay path with true 2% frame loss (relay is slightly
overloaded). The mesh peer observes this via SACK ratios on the relay
path. The dual-EMA health pipeline produces a relay_health score of
approximately 0.98 (Q16.16: 0x0000FA00). This is above the 0.75
diagnostic threshold, so PathDiagnosis = AllHealthy. No scar is
generated. The relay health score correctly reflects the relay's
performance.

Now add 15% WiFi loss on the first hop (source to relay, both on WiFi).
The effective relay path loss is 1 - (0.85 * 0.98) = 16.7%. The mesh
peer observes relay_health of approximately 0.83. Direct path loss is
15% (WiFi only). wifi_quality is 0.85.

The diagnostic triangle:
- direct_health = 0.85 (above 0.75 threshold).
- relay_health = 0.83 (above 0.75 threshold).
- wifi_quality = 0.85 (above 0.75 threshold).

PathDiagnosis = AllHealthy. No scar generated. Both paths are degraded
by WiFi but neither crosses the 0.75 threshold. The relay's 2% true
loss is masked by the WiFi noise floor.

This is a problem. The relay's true health is 0.98 but the observed
relay path health is 0.83 because of WiFi on the first hop. The
diagnostic triangle cannot distinguish "relay is fine, WiFi is noisy"
from "relay has 17% loss." To separate these, the triangle needs:

```
relay_intrinsic_health = relay_path_health / wifi_quality
                       = 0.83 / 0.85
                       = 0.976
```

This normalization removes the WiFi contribution from the relay path
measurement. The relay's intrinsic health (0.976) is close to its
true health (0.98). The difference (0.004) is measurement noise.

I propose adding this normalization to the diagnostic triangle.
The relay_health field should be the WiFi-normalized relay path health,
not the raw relay path SACK ratio. The normalization is a single Q16.16
division per heartbeat tick per relay path. Cost: ~20ns on ARM Cortex-A53
(fixed-point division). At 11 connections with relay paths: 220ns per
tick. Negligible.

### Updated Relay Health Formula

```
struct RelayHealthNormalized {
    raw_relay_sack: HealthScore,      // observed SACK ratio on relay path
    wifi_quality: CapacityRatio,      // local WiFi delivery ratio
    normalized: HealthScore,          // raw / wifi_quality, clamped to [0, 1.0]

    fn compute(raw: u32, wifi: u32) -> u32 {
        if wifi == 0 { return 0; }  // no WiFi = no relay path
        let result = (raw << 16) / wifi;
        if result > 0x00010000 { 0x00010000 } else { result }
        // Clamped: normalized cannot exceed 1.0.
        // If raw > wifi (WiFi loss on return path but not forward),
        // normalization exceeds 1.0; clamp preserves invariant.
    }
}
```

The diagnostic triangle then uses normalized relay health for its
threshold comparison:

```
fn diagnose(&self) -> PathDiagnosis {
    let d = self.direct_health.value_q16();
    let r = self.relay_health_normalized.value_q16();  // normalized
    let w = self.wifi_quality.value_q16();
    let t = 0x0000C000u32;  // 0.75

    match (d >= t, r >= t, w >= t) {
        // ... same 8-case match as savant's proposal
    }
}
```

With normalization, the relay's intrinsic 2% loss is correctly measured
even when WiFi is noisy. Without normalization, the relay appears to
have 17% loss when WiFi has 15% loss, and the diagnostic triangle
cannot tell the relay is fine.

### Impact on Relay Topology Decisions

Relay topology decisions (failover, warm standby selection, relay
preference ranking) should use the normalized relay health, not the
raw relay path health. This ensures:

1. Relay failover triggers when the relay itself degrades, not when
   WiFi fluctuates. False failover rate at 15% WiFi loss:
   - Without normalization: relay_health crosses 0.75 threshold when
     WiFi dips to 12% delivery ratio (relay path = 0.12 * 0.98 = 0.118,
     well below 0.75). False failover when WiFi dips to ~77% delivery
     (relay path = 0.77 * 0.98 = 0.754, just below 0.75). WiFi dips
     to 77% delivery approximately once per hour in a congested apartment.
     False relay failover rate: ~1/hour.
   - With normalization: relay failover triggers only when normalized
     relay health drops below 0.75, meaning the relay itself has ~25%+
     loss. False failover rate from WiFi noise: effectively zero (WiFi
     noise is divided out).

2. Warm standby relay selection uses intrinsic relay quality. Two
   relays R1 (on WiFi) and R2 (on Ethernet) are correctly ranked even
   when the observer is on WiFi. Without normalization, R1 and R2 both
   appear degraded by WiFi. With normalization, R2's Ethernet connection
   is correctly measured as higher quality.

### Verdict on Question 1

The three-layer classification does not harm relay health monitoring.
The classification operates downstream of the health measurement.
However, the diagnostic triangle's relay_health input needs WiFi
normalization to provide accurate relay intrinsic health. Without
normalization, WiFi noise masks relay degradation and causes false
relay failover. I add the normalization formula to my relay design
(Section 6).

## Section 2: The Diagnostic Path Triangle -- Operational Soundness

### The Question

The savant expanded my relay health monitoring into a three-input
diagnostic triangle (direct_health, relay_health, wifi_quality) with
8 PathDiagnosis outputs. Is this operationally sound? What are the
false positive and false negative rates?

### False Positive Analysis

A false positive occurs when the triangle diagnoses a fault that does
not exist. I analyze each diagnosis:

**RelayFault (direct good, relay bad, WiFi good):**
False positive requires direct_health >= 0.75, relay_health_normalized
< 0.75, wifi_quality >= 0.75, but the relay is actually fine.
With normalization, relay_health_normalized < 0.75 means the relay
itself has >25% loss. In practice, a healthy relay on modern hardware
has <0.1% loss (forwarding is a memcpy). A relay with >25% loss is
genuinely overloaded or failing. False positive rate: <0.1% (requires
a measurement error of 25+ percentage points in the SACK ratio EMA,
which requires 25+ consecutive lost SACK reports, probability
0.15^25 at 15% WiFi loss = effectively zero with normalization).

**NatExpired (direct bad, relay good, WiFi good):**
False positive requires the direct path to degrade while the relay
path stays healthy, but the NAT mapping is actually fine. This can
occur during a transient routing change (the direct WAN path flaps
while the relay path uses a different WAN route). In practice, NAT
mappings expire silently (no ICMP notification) and the direct path
degrades gradually (SACK ratios drop as the mapping expires and
return packets are dropped by the NAT). The triangle correctly
identifies this. False positive rate: ~2% (transient routing flaps
cause direct_health to dip below 0.75 for 1-2 heartbeat intervals
before recovering). Duration of false positive: 2-10 seconds (until
the next EMA update corrects the direct_health score).

**DestinationFault (both direct and relay bad, WiFi good):**
False positive requires both paths to degrade simultaneously, but
the destination is actually fine. This occurs when the destination's
upstream link flaps (ISP-level issue). The triangle diagnoses this as
DestinationFault, which is correct: from the observer's perspective,
the destination is unreachable regardless of path. The "false"
positive is actually a true positive from the observer's viewpoint.
False positive rate: <1%.

**WifiTransient (both direct and relay good, WiFi bad):**
False positive requires wifi_quality to drop below 0.75, but WiFi is
actually fine. wifi_quality is measured locally (observed delivery
ratio over 30 seconds). A false positive requires the local
measurement to report <75% delivery ratio when the true ratio is
>75%. This would require a correlated measurement error across the
30-second window. At 1000 frames/second per connection and 11
connections, the 30-second window contains ~330,000 observed frames.
The measurement noise at 330k samples is <0.1 percentage points.
False positive rate: <0.01%.

**LocalLinkDown (all three bad):**
False positive requires all three measurements to simultaneously
report <0.75. This is the most restrictive diagnosis. A false positive
requires three independent measurement errors. Rate: product of
individual false positive rates = effectively zero.

### False Negative Analysis

A false negative occurs when the triangle fails to diagnose a fault
that exists.

**Missed RelayFault:** The relay has >25% loss but the triangle
reports AllHealthy. This requires relay_health_normalized >= 0.75
when the relay's true loss is >25%. With normalization, this requires
the WiFi quality measurement to be wrong (dividing by an incorrect
wifi_quality inflates the normalized value). If wifi_quality is
measured as 0.70 but is actually 0.90, the normalization divides
by 0.70 instead of 0.90, inflating the result by 0.90/0.70 = 1.29x.
A relay with 30% true loss (raw = 0.70 * 0.70 = 0.49, normalized =
0.49 / 0.70 = 0.70) would appear as normalized = 0.70, which is
below 0.75, so the fault is still detected. A relay with 26% true
loss (raw = 0.70 * 0.74 = 0.518, normalized = 0.518 / 0.70 = 0.74)
would just barely miss the threshold. False negative rate for relay
faults between 25-30% loss when WiFi is simultaneously degraded:
~5%. For relay faults >30%: <1%.

**Missed NatExpired:** The NAT mapping has expired but the triangle
reports AllHealthy. This requires direct_health >= 0.75 despite the
expired mapping. An expired NAT mapping causes 100% packet loss on
the direct path (no return packets). The EMA converges to 0.0 within
3-5 heartbeat intervals (15-25 seconds at 5-second intervals). During
the convergence window, direct_health may remain above 0.75. False
negative duration: 15-25 seconds after NAT expiry. Rate during this
window: 100% (the EMA has not converged yet). After convergence: 0%.

### Summary of Error Rates

| Diagnosis | False Positive Rate | False Negative Rate | FN Window |
|-----------|--------------------|--------------------|-----------|
| RelayFault | <0.1% | <5% (25-30% loss range) | steady-state |
| NatExpired | ~2% | 100% (first 15-25s) | EMA convergence |
| DestinationFault | <1% | <1% | steady-state |
| WifiTransient | <0.01% | ~3% (borderline WiFi) | measurement |
| LocalLinkDown | ~0% | <1% | steady-state |

### Operational Verdict

The triangle is operationally sound. The highest false negative rate is
the NAT expiry detection window (15-25 seconds of blindness after the
mapping expires). This is acceptable because NAT keepalives prevent
expiry in the common case, and the 15-25 second detection window is
shorter than most NAT mapping lifetimes (30-120 seconds for aggressive
CGNAT, 120-600 seconds for consumer routers).

The highest false positive rate is NatExpired at ~2%, caused by
transient WAN routing flaps. The response to a false NatExpired
diagnosis is "send a NAT keepalive." A spurious keepalive is harmless
(91 bytes, once per false positive event). At 2% false positive rate
and 5-second heartbeat intervals: ~1 spurious keepalive per 250
seconds = 0.35 bytes/second overhead. Negligible.

I adopt the diagnostic triangle into my relay design.

## Section 3: Relay-First vs Hole-Punch-First Traffic Class Split

### The Question

The savant proposes relay-first for control traffic (heartbeats, CPSK,
convergence) and hole-punch-first for bulk traffic (matter transfers).
Is this complexity justified for household deployment? What is the
concrete cost in relay bandwidth?

### The Complexity Cost

The traffic class split requires:

1. **Two forwarding decisions per connection pair at the relay.** The
   relay must know which connections carry control traffic (always
   forward) and which carry data traffic (forward only if no direct
   path). The relay sees msg_type in the outer header but does not
   know the connection's traffic class assignment. The mesh layer
   makes the assignment; the relay layer does not participate in mesh
   decisions. To implement the split, either:
   (a) The relay inspects msg_type and forwards HEARTBEAT/CPSK frames
       always, DATA frames only when the direct path is down. This
       requires the relay to track per-connection direct-path status,
       which it cannot observe (the relay does not see direct-path
       frames). The mesh would need to SIGNAL the relay with
       direct-path status updates. New SIGNAL type, new forwarding
       table field, new state machine.
   (b) The mesh establishes two connections per cluster pair: one
       control connection (always relayed), one data connection
       (direct-first, relay-fallback). This doubles the connection
       count and CPSK session count per cluster pair. At 11 cluster
       pairs: 22 connections instead of 11. The CPSK polygon schedule
       (11 rounds for 11 connections) becomes 22 rounds for 22
       connections. The polygon extension formula from the digest:
       Lucas 1-factorization on 22 peers requires 21 rounds (22 is
       even, n-1 = 21). That is nearly double the convergence time.

   Option (a) leaks information to the relay (the relay learns when
   direct paths succeed or fail). Option (b) doubles convergence time.
   Neither is acceptable.

2. **Two NAT traversal state machines per connection.** The consumer
   must maintain one Noise KK session via the relay (always up) and
   one Noise KK session via the direct path (opportunistic). Each
   session requires ~2 KB of state (Noise CipherState + replay window
   + SACK tracking). At 11 cluster pairs: 22 sessions * 2 KB = 44 KB
   instead of 11 * 2 KB = 22 KB. The extra 22 KB is manageable on a
   Pi 4 but meaningful on a Pi Zero (22 KB out of 50-100 MB available
   is not the concern; the CPU cost of maintaining 22 active Noise
   sessions is).

3. **Complexity in the connection migration path.** When a direct path
   succeeds, data traffic migrates from relay to direct. Control
   traffic stays on the relay. The SACK stream must span both paths
   (the remote peer sends SACKs for data received on the direct path,
   but these SACKs must be correlated with frames that may have been
   sent on the relay path during migration). This cross-path SACK
   correlation is a new mechanism with no precedent in the design.

### The Bandwidth Cost

Relay-first for all traffic (my Stage 1 proposal):
- Heartbeat: 91 bytes * 1 per 5 seconds * 11 connections = ~200 bytes/s
  = 1.6 kbps.
- SACK: ~146 bytes * 1 per second * 11 connections = ~1.6 KB/s = 12.8 kbps.
- Control log entries during convergence: ~200 bytes * 50 entries *
  6 cluster pairs = 60 KB burst over ~30 seconds = 2 KB/s = 16 kbps.
- Matter transfer (data plane): A 2 GB transfer at LEDBAT pace through
  a Pi Zero relay at 750 kbps per session.

Total relay bandwidth for control traffic: ~30 kbps steady-state, ~46
kbps during convergence. This is 0.2-0.3% of a Pi Zero's 15 Mbps WiFi
budget. The control traffic cost is negligible regardless of strategy.

Savant's split (relay-first for control, hole-punch-first for data):
- Control traffic through relay: same ~30 kbps.
- Data traffic bypasses relay (if direct path succeeds): saves the
  750 kbps per matter transfer session.

The bandwidth saving is 750 kbps per active data session, but only when
the direct path succeeds. For symmetric NAT pairs (40% of deployments),
the direct path never succeeds. The relay carries all traffic regardless.
The split saves bandwidth only for non-symmetric NAT pairs that could
have established a direct path anyway.

### The Simpler Alternative

My Stage 1 proposal: relay-first for all traffic. When a direct path
is established (via background hole-punch), migrate all traffic to the
direct path. Keep the relay session as a warm standby. One connection
per cluster pair. One NAT traversal state machine. One SACK stream.

The cost: during the 2-10 second hole-punch establishment window, data
traffic flows through the relay. At 750 kbps, a 10-second window
consumes ~938 KB of relay bandwidth. For a 2 GB transfer, this is
0.046% of the total transfer. For heartbeat-only traffic (91 bytes
per 5 seconds), the relay cost during the window is ~182 bytes. Both
are negligible.

The benefit: no traffic class splitting, no dual connections, no
cross-path SACK correlation, no information leakage to the relay about
direct-path status.

### Verdict on Question 3

The savant's traffic class split is not justified for household
deployment. The bandwidth savings are negligible for control traffic
(30 kbps, always) and the data traffic savings only apply to the 60%
of connection pairs where hole-punch succeeds. The complexity cost
(dual connections, dual state machines, cross-path SACK) outweighs the
bandwidth savings. I maintain relay-first for all traffic with
opportunistic direct-path migration.

## Section 4: The Pi Zero Relay Stall During Convergence

### The Question

My devil's advocate flagged a 144ms DH stall on a Pi Zero relay during
convergence (12 Noise handshakes * 4 DH operations * 3ms per DH). Does
the savant's pseudo-peer model reduce this?

### What the Savant Proposed

The savant's pseudo-peer model limits the relay's capabilities: the
relay has DK/DEK but does not participate in gossip, dominance scoring,
or health pipeline computation. The relay authenticates peers and
forwards frames but does not process mesh-layer semantics.

This does not reduce the DH stall. The DH stall occurs during Noise KK
session establishment, which is a protocol-layer operation, not a
mesh-layer operation. Whether the relay is a full peer or a pseudo-peer,
it still performs Noise KK handshakes with each endpoint. The pseudo-peer
model saves mesh-layer CPU (no gossip processing, no health pipeline,
no dominance computation) but does not save protocol-layer CPU (Noise KK
handshake, frame forwarding, SACK processing if the relay participates
in reliable delivery).

### Quantifying the Stall

The 144ms number from my devil's advocate analysis:
- 12 Noise handshakes (6 cluster pairs * 2 endpoints per pair).
- 4 DH operations per handshake (2 DH in Noise KK: static-static and
  ephemeral-static, but actually 2 DH per initiator + 2 per responder;
  the relay is the responder in all cases, so 2 DH per handshake).
- Corrected: 12 handshakes * 2 DH per handshake = 24 DH operations.
- At 3ms per DH on Cortex-A53: 24 * 3ms = 72ms.

I overcounted in my devil's advocate. The relay performs 2 DH per
handshake (as responder), not 4. The corrected stall is 72ms, not
144ms.

However, there are two corrections in the opposite direction:

1. Each endpoint pair requires a separate relay session. The relay has
   sessions with both endpoints of a relayed pair. For 6 relayed cluster
   pairs: 12 relay sessions (one per endpoint). Each session has its own
   Noise KK handshake. The relay performs 12 * 2 = 24 DH operations.
   At 3ms each: 72ms. This matches my corrected calculation.

2. During convergence, not all 6 cluster pairs re-establish
   simultaneously. Convergence is sequenced: Front 1 (GSet merge)
   completes before Front 2 (CPSK renegotiation). Session
   re-establishment happens during Front 2. If Front 2 uses the
   polygon schedule (11 rounds), sessions are established in rounds,
   not simultaneously. Each round has at most ceil(N/2) concurrent
   handshakes. At 12 sessions and 11 rounds: ~1-2 handshakes per round.
   Per-round DH cost: 2-4 DH * 3ms = 6-12ms. Total over 11 rounds:
   66-132ms, spread over the polygon schedule's duration (11 rounds *
   2 RTT per round at 50ms = 1100ms). The DH work is ~6-12% of the
   polygon schedule duration, not a blocking stall.

### The Real Stall: Initial Mesh Join

The worst case for the DH stall is not convergence (which is sequenced)
but initial mesh join. When a new relay peer joins, it must establish
Noise KK sessions with all existing peers. At 36 peers (max mesh size):
36 handshakes * 2 DH * 3ms = 216ms of DH on a Pi Zero.

With Expert B's CryptoRequest/CryptoResult pattern, the DH operations
are offloaded to a crypto thread. On a 4-core Pi Zero 2W: the crypto
thread processes 4 DH operations concurrently (one per core). 36
handshakes / 4 cores * 2 DH * 3ms = 54ms. The event loop is not
blocked; it continues forwarding existing traffic while the crypto
thread processes DH.

On a single-core Pi Zero W (discontinued but deployed): the crypto
thread shares the core with the event loop. The 216ms DH blocks the
event loop intermittently (context switching between event loop and
crypto thread). Effective stall: ~108ms (50% of DH time is on the
event loop's core).

### The Pseudo-Peer Savings

The pseudo-peer model saves the following per-tick CPU on a Pi Zero
relay:

| Operation | Full Peer | Pseudo-Peer | Savings |
|-----------|-----------|-------------|---------|
| Noise KK handshakes | 72ms/convergence | 72ms/convergence | 0% |
| Frame forwarding | 3ms/s at 10k fps | 3ms/s at 10k fps | 0% |
| Health pipeline (dual-EMA) | ~0.5ms/s | 0 | 0.5ms/s |
| Gossip processing | ~1ms/s during convergence | 0 | 1ms/s |
| Dominance computation | ~0.1ms per election | 0 | 0.1ms/election |
| Control log verification | ~2ms per 50-entry burst | 0 | 2ms/burst |

Total pseudo-peer savings: ~1.5ms/s steady-state, ~3ms/s during
convergence. On a Pi Zero at 1 GHz quad-core: 1.5ms of 1000ms =
0.15% CPU savings. Not significant.

### Verdict on Question 4

The savant's pseudo-peer model does not reduce the DH stall. The DH
stall is a protocol-layer cost, not a mesh-layer cost. The corrected
stall is 72ms during convergence, spread over the polygon schedule
(~6-12% of each round). This is acceptable on a 4-core Pi Zero 2W
with a crypto thread. It is problematic on a single-core device.

The pseudo-peer model saves ~0.15% CPU steady-state. This is not
a meaningful improvement for the DH stall. The meaningful improvement
is Expert B's CryptoRequest/CryptoResult pattern, which parallelizes
DH across cores. The pseudo-peer model's value is in specification
simplicity (separate RelayForwarder trait, as Expert B proposed), not
in performance.

I update my relay design to require multi-core for relay peers.
Single-core devices (Pi Zero W) should not serve as relay peers. This
is a deployment recommendation, not a protocol constraint.

## Section 5: Cloud Relay Bootstrap and the Nucleation Model

### The Question

My devil's advocate flagged the cloud relay bootstrapping problem:
a 2-peer mesh behind symmetric NATs is non-functional without a cloud
relay or rendezvous server. Does the savant's "nucleation" model help?

### What the Savant Proposed

The savant describes mesh formation as crystal nucleation: the first
few peers form a nucleus, and the crystal grows by attracting new
peers to the nucleus surface. The savant says:

> "Cloud relay bootstrapping: The devil's advocate correctly identifies
> that a 2-peer mesh behind symmetric NATs is non-functional without a
> cloud relay. This is a deployment constraint, not a protocol design
> issue. The minimum fix (hardcode DHT bootstrap nodes) is correct."

The savant classifies this as a deployment constraint, not a protocol
issue. I partially agree but must push back on the specifics.

### The DHT Bootstrap Chain

The chain from "I have an invite token" to "I can communicate with
the mesh" requires:

1. Parse invite token (188 bytes + address_hints + admin signature).
2. Try address_hints: direct connection to the inviting peer. If both
   peers are behind symmetric NATs, all address_hints fail (NAT mapping
   does not match the invitee's source port).
3. Fall back to relay_dht_key: look up the relay in Mainline DHT.
   This requires DHT bootstrap. My devil's advocate identified the
   circular dependency: DHT bootstrap requires knowing at least one
   DHT node, which requires network connectivity, which for
   symmetric NAT peers requires a relay.

The "hardcode DHT bootstrap nodes" fix resolves the circular dependency.
Standard Mainline DHT bootstrap nodes (router.bittorrent.com:6881,
dht.transmissionbt.com:6881, router.utorrent.com:6881) are well-known,
long-lived, and accessible from behind any NAT type (they use UDP and
respond to incoming packets regardless of NAT mapping type).

But this introduces a new dependency: the mesh's availability depends
on third-party DHT bootstrap nodes. If router.bittorrent.com is down,
the 2-peer symmetric NAT mesh cannot bootstrap. This is a fragile
dependency for a self-hosted system.

### The Nucleation Alternative

Instead of relying on DHT for the symmetric NAT case, I propose a
three-tier discovery chain:

**Tier 1: Direct (address_hints from invite token).**
Cost: 1 RTT per hint, ~50-200ms per attempt.
Success rate: ~95% on LAN, ~60% across NATs (non-symmetric), ~5%
across symmetric NATs. This is the fast path.

**Tier 2: STUN + hole-punch (using STUN servers from invite token).**
The invite token currently contains address_hints and relay_dht_key.
I propose adding stun_server_hints (2 STUN server addresses, 8 bytes
each = 16 bytes added to invite token). Both peers send STUN binding
requests to discover their public address/port, then attempt
simultaneous hole-punch using the SIGNAL(PUNCH) mechanism via the
STUN server as a rendezvous point.

Cost: 2 STUN requests + 1 hole-punch attempt, ~200-500ms.
Success rate: ~80% for address-dependent NAT, ~10-15% for symmetric
NAT. Partial improvement over Tier 1.

**Tier 3: Cloud relay (fallback for symmetric NAT pairs).**
For the 40% symmetric NAT case where Tiers 1 and 2 fail, the design
needs a relay. Options:

(a) **DHT-discovered relay:** The invite token's relay_dht_key points
    to a DHT entry containing the relay's public address. Requires
    DHT bootstrap (hardcoded nodes). Cost: ~2-5 seconds for DHT
    lookup. Success rate: ~95% if DHT is available.

(b) **Cloud relay service:** A lightweight relay service (the
    TransitRelay forwarding logic, ~300ns per frame, no decryption)
    running on a $5/month VPS. The service's address is embedded in
    the invite token or hardcoded in the binary. Cost: $5/month
    recurring. Success rate: ~99.9% (VPS uptime).

(c) **Hybrid:** Try DHT first, fall back to cloud relay if DHT is
    unreachable. This requires both a DHT bootstrap list and a cloud
    relay address. The invite token carries both.

### Concrete Numbers for Cloud Relay

A cloud relay (TransitRelay only, no mesh participation) on a
$5/month VPS (1 vCPU, 1 GB RAM, 1 TB bandwidth):

- Memory per relayed pair: 100 bytes (forwarding table entry) + 2 KB
  (Noise session state, if authenticated) = ~2.1 KB. At 1000 relayed
  pairs (far beyond household use): 2.1 MB. Fits in 1 GB.

- CPU per frame: ~300ns (hash lookup + memcpy). At 10,000 fps: 3ms/s.
  0.3% of one core. Fits in 1 vCPU.

- Bandwidth: 10,000 fps * 1232 bytes = 12.32 MB/s = ~100 Mbps. A
  $5/month VPS typically has 1 Gbps port with 1 TB monthly bandwidth.
  At 100 Mbps sustained: exhausts 1 TB in ~2.8 hours. For household
  use (1-5 relayed pairs, heartbeats + occasional matter transfer):
  ~10 kbps steady-state, ~1 Mbps during matter transfer. Monthly
  bandwidth: ~3 GB steady-state + ~10 GB per 2 GB file transfer.
  Well within the 1 TB limit.

- Cost per household: $5/month for the VPS. Split across users of the
  relay: if 100 households share one relay, $0.05/month each.

### The Self-Hosted Ethos Problem

The savant correctly notes that a cloud relay "contradicts the design's
self-hosted ethos." But the alternative is worse: a 2-peer mesh behind
symmetric NATs simply does not work without a relay. The choices are:

1. Require a cloud relay. Contradicts self-hosted ethos.
2. Require at least one peer with non-symmetric NAT. Limits deployment
   (40% of users have only symmetric NAT).
3. Document that 2-peer symmetric NAT meshes are unsupported. Honest
   but limits adoption.
4. Ship a free community relay. The verkko project runs a relay. Users
   can opt out by running their own relay or having a non-symmetric NAT.

Option 4 is the least bad. The community relay is a TransitRelay
(opaque forwarding, no decryption, no content visibility). It sees
connection_id and frame sizes. It cannot read content. The privacy
implications are similar to a TURN server or Tailscale DERP relay.

### Nucleation Model Assessment

The savant's nucleation model (crystal grows from a nucleus) maps well
to deployment: the first peer is the nucleus. But the nucleation model
does not solve the symmetric NAT problem because nucleation requires
at least one reachable seed. If the seed is behind symmetric NAT, no
crystal can form.

The nucleation model helps once the first relay is available: new peers
discover the relay, connect through it, and the mesh grows. The relay
is the nucleation site. Without the relay, nucleation cannot start.

### Verdict on Question 5

The savant's nucleation model does not solve the cloud relay problem.
It describes mesh growth after bootstrapping, not bootstrapping itself.
The minimum solution is a three-tier discovery chain (direct, STUN +
hole-punch, cloud relay) with a community relay as a fallback. I add
this to my relay design. The invite token gains 16 bytes for STUN
server hints and optionally 6 bytes for a cloud relay address hint.

## Section 6: Updated Relay Design

Based on the savant's Stage 2b, my devil's advocate findings, and the
analysis in Sections 1-5, I update my relay design with the following
changes:

### Change 1: WiFi-Normalized Relay Health

The relay health EMA track is replaced with a WiFi-normalized variant.
The diagnostic triangle uses normalized relay health for all threshold
comparisons and topology decisions.

```
relay_health_normalized = raw_relay_sack_ratio / wifi_quality
```

Impact: one Q16.16 division per heartbeat tick per relay path. At 11
connections: 220ns per tick. No memory impact (replaces existing field).

### Change 2: Diagnostic Triangle Adoption

I adopt the savant's diagnostic triangle into the relay design. The
triangle provides automated fault attribution for relay failover
decisions. The relay failover trigger changes from "relay SACK ratio
below threshold for 5 consecutive heartbeats" (my Stage 1) to
"PathDiagnosis == RelayFault for 5 consecutive heartbeats."

Impact: the relay failover mechanism gains immunity to WiFi-induced
false failovers. False failover rate drops from ~1/hour (at 15% WiFi
loss, without normalization) to effectively zero (with normalization).

### Change 3: Relay-First for All Traffic (No Traffic Class Split)

I reject the savant's traffic class split. All traffic uses the relay
path during session establishment. Once a direct path is available, all
traffic migrates to the direct path. The relay session remains as a
warm standby.

The migration procedure (from my Stage 1, unchanged):
1. Background hole-punch attempt starts after relay session is
   established.
2. If hole-punch succeeds, PATH_CHALLENGE on the direct path.
3. On PATH_RESPONSE, migrate all traffic to direct path.
4. Stop sending data frames through the relay.
5. Continue sending keepalive frames through the relay (1 per 25
   seconds, 91 bytes each = 3.6 bytes/second) to maintain the relay
   session as a warm standby.

Relay bandwidth overhead for warm standby: 3.6 bytes/second per relayed
pair. At 6 relayed pairs: 21.6 bytes/second = 173 bps. Negligible.

### Change 4: Multi-Core Requirement for Relay Peers

Relay peers must have at least 2 CPU cores. Single-core devices are
excluded from relay duty. This is a deployment recommendation
documented in the relay specification, not a protocol constraint.

Rationale: the crypto thread is essential for non-blocking DH during
Noise handshakes. On a single-core device, the 72ms convergence DH
stall blocks the event loop, causing heartbeat misses that trigger
false relay failure detection by mesh peers.

### Change 5: Three-Tier Discovery Chain

The peer discovery fallback chain is extended:

```
Tier 1: address_hints from invite token (direct connection)
Tier 2: STUN + hole-punch (new: stun_server_hints in invite token)
Tier 3: cloud relay (relay_address_hint in invite token, or
         relay_dht_key for DHT discovery, or community relay)
```

The invite token wire format gains two optional fields:

```
InviteToken (extended):
    ... existing fields (188 bytes) ...
    stun_hint_count: u8           // 0-2 STUN server hints
    stun_hints: [IpPort; count]   // 6 bytes each (4 IP + 2 port)
    relay_hint_count: u8          // 0-1 direct relay address hints
    relay_hints: [IpPort; count]  // 6 bytes each
    ... admin signature (64 bytes) ...
```

Maximum invite token growth: 1 + 12 + 1 + 6 = 20 bytes. Total invite
token size: 208 bytes + address_hints + admin signature. Still fits in
one 1232-byte UDP frame.

### Change 6: Amplification Prevention Window Dynamic Adjustment

The relay's amplification prevention window is set to 2x the current
heartbeat interval, not a fixed 30 seconds. The relay infers the
heartbeat interval from frame arrival patterns (moving average of
inter-frame intervals from each endpoint). The window adjusts
automatically as the metabolic state changes the heartbeat interval.

```
amp_window = 2 * moving_avg_inter_frame_interval
           = 2 * 5s = 10s  (Active mode, 5s heartbeat)
           = 2 * 30s = 60s (Background mode, 30s heartbeat)
```

Impact on one-directional flow at 15% WiFi loss:
- Fixed 30s window: one-directional flow every ~200s in Background mode.
- Dynamic 60s window: one-directional flow every ~7000s in Background
  mode (requires 2 consecutive missed heartbeats, P = 0.15^2 = 0.0225,
  at 30s interval: once per 30/0.0225 = 1333 intervals = 40,000s).

The dynamic window reduces one-directional flow events by 200x in
Background mode.

### Change 7: Relay Grain Boundary Scar Decay

I adopt the savant's three-layer scar classification for relay path
faults. Relay-attributed scars (ScarLayer::GrainBoundary) use a 30-day
decay instead of 180-day decay. The scar is attributed to the relay
peer, not to the destination peer. This prevents relay infrastructure
issues from polluting the destination peer's dominance score.

The relay's dominance score is affected by grain boundary scars but
the relay does not participate in gateway elections (it is a pseudo-peer
or a full peer with relay-only enrollment). Grain boundary scars affect
only the relay's selection priority in the relay preference ranking:
relays with fewer/older grain boundary scars are preferred.

### Change 8: Community Relay Specification

For meshes where all peers are behind symmetric NATs, the protocol
specification includes a community relay option. The community relay is:

- A TransitRelay running on a public-IP server.
- Opaque forwarding only. No decryption, no content visibility.
- No mesh membership. The community relay does not have a DK/DEK. It
  authenticates peers via a simple challenge-response (the peer signs
  a nonce with its DK; the relay verifies against the DK from the
  SIGNAL(HELLO) registration).
- Rate-limited: 100 kbps per relayed pair, 1 Mbps aggregate per relay.
  Sufficient for heartbeats and metadata. Matter transfers must use
  direct paths or household relays.

This is an optional deployment component. The protocol works without it
if at least one mesh peer has a non-symmetric NAT.

## Section 7: Remaining Open Items in My Domain

### Open 1: Multi-Hop Relay Path Selection

The digest lists D-007 (multi-hop forwarding table, path selection)
as a Run 13 item. With the relay design updates in this document, the
single-hop relay case is fully specified. Multi-hop relay (hop_count
> 1) is needed only when:
- Two peers are behind symmetric NATs in different households.
- No common relay peer is reachable by both.
- A two-hop path exists through two relays (peer A -> relay R1 ->
  relay R2 -> peer B).

This scenario requires R1 and R2 to have a direct connection. If R1
and R2 are both cloud relays or one is a cloud relay, the two-hop
path is viable. The forwarding table at R1 must contain an entry
pointing to R2 as the next hop for peer B's destination cluster.

I defer multi-hop path selection to Run 13 as specified in the digest.
The hop_count field (max 4) and TTL-decrement behavior are already
specified. The forwarding table structure supports multi-hop (next_hop
can be another relay's address). The missing specification is: how does
R1 learn that R2 can reach peer B? This requires relay-to-relay
gossip or a centralized relay registry, neither of which is specified.

### Open 2: Relay/Seat Failover Interaction

The digest lists D-009 (seat failover when host peer dies). The seat
connects to a host peer via WebSocket. If the host peer dies, the
seat loses its connection. The failover path:

1. Seat detects WebSocket disconnection (TCP FIN or timeout).
2. Seat tries alternative host peers from its cached peer list.
3. If all host peers are unreachable (entire cluster is down), the
   seat enters offline mode.

The interaction with relay failover: if the host peer was also a relay
peer, its death triggers both seat failover and relay failover
simultaneously. The relay failover migrates relayed pairs to the warm
standby relay. The seat failover reconnects the seat to an alternative
host peer. These two failovers are independent (different layers, different
state machines) but may interact through resource contention: the
warm standby relay absorbs the relayed traffic from the failed relay,
increasing its load, while simultaneously accepting seat reconnections.

I defer the full seat failover specification to the verkko-seat domain
expert. Outside my domain -- no comment on the seat's internal
failover state machine.

### Open 3: LEDBAT WiFi Correction

The savant proposes:

> (1 / wifi_quality^2)."

Outside my domain -- no comment on LEDBAT parameter tuning. The relay
does not run LEDBAT (it forwards frames without congestion control).
LEDBAT is a protocol-layer concern.

### Open 4: Convergence-Gated Replication Start

The savant proposes a REPLICATION_READY convergence gate. Outside my
domain -- no comment on replication scheduling. The relay participates
in replication only as a storage target (membrane PUTs), not as a
replication decision-maker.

### Open 5: Supersaturation-Weighted Replication Growth

Outside my domain -- no comment. The relay does not make replication
