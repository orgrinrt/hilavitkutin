## Part 1: How the Twelve Gaps Were Received

### 1.1 The ArchSurgeon's Blanket Classification

The ArchSurgeon classified all 12 gaps (B1-B12) as "content gaps, not structural defects." The reasoning: none of the 12 require new DAG edges. The dependency graph is sound. Every gap can be filled by writing body text within the existing document topology. The Surgeon's test is binary: "does this require a new edge in the dependency graph?" If no, it is content. If yes, it is structural.

This classification is technically correct and cartographically incomplete.

The Surgeon's test measures one dimension of structural health: topological correctness. The DAG is acyclic, the edges are declared, the levels are consistent. But a map can have correct borders and still be useless if the interior is blank. A country with correct borders and no roads, no cities, no infrastructure is structurally sound in the border-line sense but structurally deficient in the habitation sense.

Consider B1 (SansIO consumer interface). The Surgeon says: "The interface does not change the DAG." True. But the interface is the entry point to the protocol territory. Without it, the protocol has no port. Ships with cargo (inbound bytes) arrive and find no dock. The territory's borders are correct. The territory's interior is well-mapped (wire formats, state machines). But the port -- the place where the outside world interacts with the territory -- does not exist.

Is this structural? The Surgeon says no, because "structural" means "DAG topology." I say yes, because "structural" also means "can the system be entered and used." A building with a correct floor plan but no doors is structurally deficient in a way that the floor plan alone cannot reveal.

The distinction matters for triage. If all 12 are "content gaps," they all have equal structural priority (zero). The Surgeon cannot distinguish between B1 (the protocol's entire public API) and B11 (DK encryption at rest, an open design question). Both are zero from the DAG perspective. But one blocks every implementer and the other blocks no one until deployment.

**My updated position:** The Surgeon's classification is correct within its frame. But the frame is too narrow for triage. I propose a two-dimensional classification:

| Gap | DAG-structural | Access-structural |
|-----|---------------|-------------------|
| B1 (SansIO) | No | Yes -- protocol has no entry point |
| B2 (Intra-cluster) | No | Yes -- cluster interior has no roads |
| B3 (Channel lifecycle) | No | No -- channels work, creation is missing |
| B4 (Matter GC) | No | No -- policy, not access |
| B5 (Matter chunking) | No | Yes -- matter cannot be transferred |
| B6 (Multi-hop relay) | No | No -- single-hop works |
| B7 (Relay failover) | No | No -- degraded operation, not blocked |
| B8 (Seat failover) | No | No -- degraded, not blocked |
| B9 (Seat write conflicts) | No | No -- addressed by SpecWriter |
| B10 (Bridge) | No | No -- application layer |
| B11 (DK at rest) | No | No -- open question |
| B12 (Connection teardown) | No | Yes -- connections cannot end cleanly |

Four of the twelve are access-structural: B1, B2, B5, B12. These are not DAG violations, but they are structural holes in the territory's usability. The Surgeon's lens cannot see them because the Surgeon looks at edges, not at roads.

### 1.2 The SpecWriter's Triage

The SpecWriter accepted B12 (connection teardown) into Run 11 scope and deferred everything else. The SpecWriter's reasoning for B12: "This is a gap in the existing Run 10 scope that should have been included. Connection teardown is part of the connection lifecycle, which P1-3 covers." For B1: "The SansIO consumer interface is critical for implementation but not for design document completion."

The SpecWriter draws a line between "design document completion" and "implementation readiness." The design documents describe what the protocol does, not the API surface for consuming it. B1 is an implementation API, not a protocol specification.

This distinction is false for a sans-IO protocol. A sans-IO protocol IS its consumer interface. The entire point of sans-IO is that the protocol is a pure state machine whose behavior is defined by its input/output types. The protocol document says "Sans-IO: the protocol is a state machine. The consumer provides the socket." This is not a footnote. It is the first design decision. The state machine's input/output surface IS the protocol specification, in the same way that a CPU's instruction set IS the CPU specification.

The SpecWriter has written wire formats for heartbeats, SACK, connection IDs, streams, version negotiation. These are the internal representation. The consumer interface is the external contract. Specifying internals without the external contract is specifying a CPU's microarchitecture without its instruction set architecture. You can build the microarchitecture, but nobody can write programs for it.

I acknowledge the SpecWriter's point that B1 may be "too large for Run 11." The SansIO interface requires enumerating all input events, all output events, the polling model, the buffer ownership model, and the error semantics. This is substantial work. But "too large" is a scheduling constraint, not a correctness argument. The gap is real regardless of whether Run 11 addresses it.

The SpecWriter's own cross-review inadvertently proves my point. In Part III of specwriter-stage2a, the SpecWriter lists missing inputs and outputs for my proposed ProtocolMachine interface: ConnectionClosed, StreamCreated, StreamClosed, HeartbeatTick, ConfigChanged, HeartbeatExtensionRequest, ConnectionMigrated, StreamReset, MetabolicStateChanged. The SpecWriter knows exactly what the event surface should contain because the SpecWriter wrote the mechanisms that produce and consume those events. The event surface is implicit in the SpecWriter's work. Making it explicit is the gap.

**Is B12 really more important than B1?** No. B12 is smaller (one message type, one handshake procedure, ~40 lines of spec). B1 is larger (~200+ lines). The SpecWriter chose B12 because it fits within Run 11's scope, not because it is more critical. This is pragmatic triage, and I do not object to it as a scheduling decision. But the priority ordering in the map remains: B1 > B5 > B2 > B12.

### 1.3 The WireEng's Partial Acceptance

The WireEng accepted B5 (matter chunking) and B12 (connection teardown) into Run 11 scope and rated B1 as "partial." The WireEng's reasoning for B1 being partial: "The event surface defines what the wire format parser emits. Without it, my wire formats have no consuming API."

The WireEng sees B1 from the wire perspective: the event surface is the consuming API for every wire format. The WireEng produced 11 wire format specifications. Each one defines byte layouts. But byte layouts are inputs to a parser, and the parser emits typed events. Without the event surface, the wire formats are layouts without a consumer contract.

The WireEng then does something the other experts did not: co-designs the event surface with the wire formats. In Finding W-SA-1, the WireEng maps each wire format to a ProtocolOutput variant:

```
SendBytes             -> all wire formats (serialization)
SessionEstablished    -> Noise handshake frame completion
SessionLost           -> heartbeat timeout detection
FrameReady            -> data frame + stream header
HeartbeatReceived     -> heartbeat payload (36 bytes)
SackReceived          -> SACK payload
PathChallengeReceived -> PATH_CHALLENGE (8 bytes)
```

This is the correct approach. The event surface is not a separate specification; it is the integration contract for all the wire formats. The WireEng sees this because the WireEng works at the byte level and understands that bytes need a destination. The SpecWriter works at the semantic level and sees the event surface as a downstream concern.

**"Partial" is the right word.** The wire-format-facing half of B1 (what events exist, what wire formats they correspond to) belongs in Run 11. The implementation-facing half (polling model, buffer ownership, error semantics) can extend into Run 12. The WireEng's triage is the most accurate of the four experts.

**WireEng:** Accepted and co-designed. Mapped each wire format to a ProtocolOutput variant. This is the most useful response: the event surface becomes concrete when grounded in wire formats.

**ArchSurgeon:** Classified as a content gap. Structurally sound. No DAG changes needed. True but unhelpful for prioritization.

The port metaphor holds: the port is accepted as necessary; the disagreement is about whether to build it now or later.

### 2.2 N2: IntraClusterTransport

**SpecWriter:** Accepted the concept. Raised a technical concern: using CIK as the Noise KK static key for all peers in a cluster means the static-static DH (ss) in KK produces the same result for all peer pairs, because all peers share the same CIK. Authentication within the cluster relies entirely on ephemeral DH (ee). The SpecWriter suggested an alternative: Noise KK with per-peer DEK keys for peer-specific authentication.

This is a valid technical point. The SpecWriter is correct that CIK-based Noise KK does not provide peer-specific authentication within a cluster. Any peer in the cluster can impersonate any other peer. The question is whether this matters.

Within a cluster of 2-6 peers, all peers are fully trusted (they share CIK, they share all channel keys, they are part of the same trust domain). Peer-specific authentication within a trusted group adds defense-in-depth but not new security properties. If any cluster peer is compromised, the attacker has all CIK and channel key material anyway. DEK-based authentication would prevent the compromised peer from impersonating other peers within the cluster, but the compromised peer already has all the information those other peers have.

The SpecWriter's conclusion -- "CIK-based Noise KK is simpler, works with the existing key material, and provides adequate security within a trusted cluster" -- aligns with my proposal. The territory remains as mapped.

**WireEng:** Partially accepted. Found that no new wire format is needed. The existing data frame format handles intra-cluster frames. The `dest_peer` field (u16) in the encrypted payload was designed for intra-cluster addressing: `0x0000` = broadcast, other values = specific peer. The WireEng's key insight: "What IS needed is specification of how dest_peer values are assigned and how CIK is used as the Noise KK static key. This is a protocol/mesh specification task, not a wire format task."

This is an excellent observation. The existing wire format already supports intra-cluster communication. The gap is not in the wire format; it is in the specification of how the existing wire format fields are used for intra-cluster traffic. The blank interior of the cluster on my map is not a missing road; it is a road without signs.

**ArchSurgeon:** Content gap. Structurally sound. The proposal uses existing dependencies (mesh -> protocol for Noise handshake, mesh -> crypto for CIK).

**Devil's Advocate:** Argued for inclusion in Run 11. Correctly noted that without intra-cluster specification, different implementations will make different choices and interop will break.

**Assessment:** N2 was adopted in concept but the scope was refined. The wire format already supports it. What is needed is specification of the session model and key usage, not new frame types. This refines my original proposal: IntraClusterTransport is not a new mechanism, it is a specification of how existing mechanisms are used within a cluster. The territory has roads; it needs signposts.

### 2.3 N3: MatterChunkProtocol

**SpecWriter:** Accepted the concept. Corrected my chunk_data size estimate from 1200 bytes to ~1145 bytes based on the corrected frame budget. Noted that bao_proof size is variable (log2(N) * 32 bytes), which for a 2GB blob with 1145-byte chunks means ~672 bytes of proof per chunk, leaving only ~473 bytes for chunk data per frame. The SpecWriter concluded: "the bao_proof size is larger than the chunk_data itself at small chunk sizes."

This is a significant refinement. My original proposal assumed a fixed PROOF_LEN, but the actual proof size grows logarithmically with blob size. For large blobs, the proof dominates the frame. The SpecWriter suggests either sending the proof separately or using larger chunk sizes (16 KB, requiring stream-level reassembly). This changes the territory: the bridge between 2GB blobs and 1280-byte frames is not a single-span bridge; it is a multi-segment bridge with proof delivery as a separate span.

**WireEng:** Accepted and produced a concrete wire format. The WireEng's chunk frame format is 52 bytes of header + variable chunk_data + variable bao_proof. The WireEng also produced wire formats for TransferRequest, TransferBegin, and TransferAbort, adding three new msg_type values (0x0D-0x0F). The WireEng corrected my frame budget: "Maximum chunk payload per frame: 1280 - 29 - 10 - 6 - 16 - 52 = 1167 bytes" (though this should be 1232 - 29 - 10 - 6 - 16 - 52 = 1119 bytes after the UDP payload correction).

The WireEng notes that at 1024-byte bao chunks with ~320 bytes of proof, chunk_data max is approximately 847 bytes, requiring 2 frames per chunk. This is acceptable for transfer throughput but doubles the frame count for large blobs. An alternative: 512-byte bao chunks fit in a single frame but double the chunk count.

**ArchSurgeon:** Content gap. Structurally sound. The chunking protocol uses the matter -> protocol dependency that OP-1 already added.

**Devil's Advocate:** Argued for inclusion. "The system exists to transfer matter. Without B5, the system can catalogue content but cannot actually download."

**Assessment:** N3 was the most enthusiastically received of my three proposals. All four experts accept the need. The WireEng produced a concrete wire format. The refinements (variable proof size, corrected frame budget, chunk size tradeoff) improve the proposal. The territory now has bridge girders; the span calculation needs calibration.

### 3.3 PB3: BlobId/relay_key_id identity systems

**Was this addressed by anyone?** Partially, by the SpecWriter alone.

The SpecWriter responded to my crash-between-derivations scenario: "The stash tracks {blob_id, relay_key_id, status} tuples. If the peer crashes after computing blob_id but before completing the PUT, the stash entry remains in Pending status. This is a correct recovery mechanism. The deterministic nature of both derivations (blob_id = BLAKE3(plaintext), relay_key_id = HMAC-BLAKE3(channel_salt, key_id)) means the mapping can always be reconstructed from the channel key material. The peer does not lose the mapping permanently; it loses it transiently and reconstructs it.

But the SpecWriter did not address my broader point: the two identity systems share no common root. This is by design (relay cannot learn content identity). The phantom border remains as documented -- two nations with different currencies and no exchange rate. The exchange happens inside the peer's head. The SpecWriter confirmed this is intentional and recoverable.

**My updated position:** PB3 is a documented design feature, not a defect. The phantom border exists and is intentional. The recovery mechanism (deterministic re-derivation from channel keys) prevents permanent data loss. The territory is correctly mapped; the border is intentionally opaque. No action needed.

## Part 4: The Five Tensions -- Updated Map

### 4.1 T1: Pre-handshake version negotiation won. Is the territory safe?

The consensus is clear: pre-handshake version negotiation (WireEng's approach) wins. The SpecWriter conceded. The ArchSurgeon endorsed. The WireEng defended. Only the Devil's Advocate dissented.

The Advocate raised five attack vectors against plaintext version probes: protocol fingerprinting, version oracle, amplification, downgrade attack, and version poisoning DoS. These are real attacks on the territory's border checkpoint. Let me assess each:

**Protocol fingerprinting (W1 in the Advocate's weakness list):** Real. The 4-byte magic `0x56524B4F` is visible in every version probe. Any network observer can identify verkko traffic. For a household mesh protocol, this is a moderate concern. The SpecWriter's response: "version information is already available via other means (port numbers, timing patterns, packet sizes)." True, but the magic bytes make identification trivial rather than probabilistic. The territory is not safe from identification; it is safe from exploitation. The fingerprint reveals the protocol, not the keys.

**Downgrade attack:** The most serious. The Advocate correctly identified that an on-path attacker can modify the VersionProbe to strip versions. The SpecWriter's response: the prologue binding mitigates this. The selected version is hashed into the Noise prologue. If the probe was tampered with, the peers compute different prologues, and the handshake fails. The SpecWriter also added: "If a Noise handshake fails immediately after a successful version probe, the initiator SHOULD retry with direct version specification."

This is a sound defense. The downgrade attack is detected (handshake failure) and recovered from (retry with direct version). The territory is safe from silent downgrade. It is not safe from DoS via repeated probe tampering, but DoS is a broader class of attack that no version negotiation scheme fully prevents.

**The SpecWriter's three caveats are correct:** rate limiting (100 responses/second per source), no flags in v1 (reserve all bits), and downgrade detection (retry on handshake failure after successful probe). These should be normative.

**My updated position on T1:** The territory is safe enough. The pre-handshake approach wins on cost/benefit. The security mitigations (prologue binding, rate limiting, downgrade detection retry) close the most dangerous attack vectors. The remaining risk (protocol fingerprinting) is acceptable for a household mesh. If a future deployment requires traffic indistinguishability, the version probe can be encrypted under a pre-shared hint derived from the founding hash, but this is v2 work.

### 4.2 T2: BLAKE2b won. Consistent territory.

The WireEng conceded. The Advocate argued for BLAKE2b. The ArchSurgeon deferred (not a structural concern). The SpecWriter maintained BLAKE2b throughout.

The territory is now consistent: `Noise_KK_25519_ChaChaPoly_BLAKE2b` and `Noise_XX_25519_ChaChaPoly_BLAKE2b`. All implementations use BLAKE2b. The Noise protocol name string is fixed. The handshake state initialization is deterministic.

The Advocate's analysis was the strongest argument: verkko targets 64-bit hardware (Pi 4/5, x86_64 NUCs), BLAKE2b is 20-40% faster on 64-bit, and the WireGuard rationale (32-bit MIPS routers) does not transfer to verkko's deployment. The decision is effectively irreversible after v1 ships (changing the Noise hash function requires a major version bump that changes the handshake state). BLAKE2b is the correct lock.

No further cartographic update needed. The territory is settled.

### 4.3 T3: random_u32 won. Does the territory lose migration-safety?

The Advocate argued that random_u32 is correct. The WireEng conceded. The ArchSurgeon endorsed random_u32 for structural simplicity. The SpecWriter maintained random_u32 throughout.

**Does the territory lose migration-safety?** No. The Advocate's analysis is decisive: "Migration-safe is a property of the usage pattern (session-scoped ID), not the generation algorithm." A random_u32 assigned at session creation survives NAT rebinding and address migration equally well. The connection_id is a session-level identifier regardless of derivation method.

The territory gains simplicity: no DomainSeparator variant for connection IDs, no persistence requirement for a session counter, no layer coupling (protocol does not need to know about CIK). The collision analysis is identical (birthday bound at ~65K concurrent IDs, negligible for a 12-cluster mesh with ~132 connections per peer).

**One observation:** The Advocate noted that the BLAKE3 derivation "introduces unnecessary coupling" by requiring CIK public keys (mesh concepts) in protocol. The ArchSurgeon identified this as a naming dependency. Both are correct. The random_u32 approach eliminates this coupling entirely. The territory's border between protocol and mesh is cleaner.

No further cartographic update needed. The territory is simplified.

### 4.4 T4: Scope -- My Battle

The SpecWriter wants to contain scope to Run 10's 23 actions, with B12 as the sole exception. The WireEng accepts B5 and B12, with B1 partially. The ArchSurgeon says all 12 are content gaps of equal structural priority. The Advocate argues for B1, B2, B5, B12, swapping out P0-4, P0-6, P2-6.

This is the central border dispute of the entire review.

**The case for containment (SpecWriter/ArchSurgeon):** Run 10 produced a prioritized, dependency-ordered plan. Four experts have produced Stage 1 work against that plan. Expanding scope fragments the work and risks never completing the 23 actions. The Savant's gaps are real but they are the next run's work.

**The case for inclusion (Advocate/Savant):** The 23 actions produce precise text for partially-specified areas while leaving the most critical gaps untouched. After Run 11, the design will have byte-level wire formats for heartbeats and SACK (which are well-understood) but no specification for how bytes enter and leave the protocol (B1), no specification for how content is transferred (B5), and no specification for how connections end (B12). This is like surveying a territory with precise elevation measurements on every hilltop while leaving the roads between hilltops unmarked.

**My updated cartographic argument:**

The map after Run 11 (if only the 23 actions land) has this profile:

- Hilltops: precisely surveyed (wire formats, defined concepts, byte layouts)
- Valley floors: precisely surveyed (crypto primitives, key derivation, domain separators)
- Roads between hilltops: **unmarked** (SansIO consumer interface, matter chunking, connection teardown)
- Interior of one major city: **blank** (intra-cluster communication)

A traveler (implementer) can verify every hilltop measurement. They can check every valley floor derivation. But they cannot walk from one hilltop to another because the roads are not on the map. They cannot enter the interior of the city because the streets are blank.

The risk is not that the map is wrong. The map is correct wherever it has content. The risk is that implementers will draw their own roads between hilltops, and different implementers will draw different roads, and the implementations will not interoperate at the road intersections.

**Concrete example: matter transfer.** After Run 11 (without B5), the map has:
- Precise wire format for the data frame outer header (29 bytes)
- Precise wire format for the encrypted payload
- msg_type 0x04 = CHUNK (registered but empty)
- LEDBAT congestion control for bulk streams
- Source selection predicates
- Cuckoo filter advertisement
- Merkle reconciliation for sync

All the pieces are specified except the actual transfer protocol. An implementer knows the frame format, the congestion control, the source selection, and the reconciliation. But they do not know how to chunk a 2GB blob, how to verify chunks incrementally, how to resume a partial transfer, or how to interleave chunks from multiple sources. They will invent all of this independently. Two implementations will produce incompatible transfer sessions.

**Scope recommendation (unchanged from 1.4):** Include B1 (partial), B5, and B12 in Run 11. Defer P0-4 and P2-6 to accommodate. Keep P0-6 because the work is already done. This is a net increase of 1 action (24 total), which is manageable. B2 belongs in a design round, not a specification run.

### 4.5 T5: HostPeer -- Has the phantom been resolved or just moved?

**Has the phantom been resolved?** Moved, not resolved. But moved to the correct location.

The phantom border in Stage 1 was: "The bridge is drawn on the map, but neither territory claims the bridge." The consensus says: "The bridge belongs to a different country (the application layer)." This moves the phantom from the verkko map to the saalis map. From verkko's perspective, the phantom is resolved: verkko specifies both banks of the river (seat-side wire protocol, mesh-side operations) and explicitly states that the bridge is an application concern. From saalis's perspective, the phantom still exists: saalis needs to build the bridge, and the bridge construction manual does not exist.

The SpecWriter's proposed additions (normative note + capability enumeration) are the correct boundary markers. They tell the application implementer: "You must build a bridge here, and it must support these capabilities." The bridge construction itself is the application's responsibility.

This is the correct architecture. The SpecWriter's river metaphor rebuttal is persuasive: "You do not put the bridge on the crypto map or the mesh map. You put it on the application map. And the application map is not a verkko document." The verkko spec provides primitives. The application provides the composition.

## Part 5: The Devil's Advocate's Eight Weaknesses

The Advocate found 8 weaknesses (W1-W8) that no other expert identified. I assess each as a threat to the territory.

### W1: Magic byte incompatibility between SpecWriter and WireEng

**Severity: Critical. Real threat.**

The SpecWriter uses per-message magic ("VKKK", "VKKR", "VXX1", etc.). The WireEng uses unified magic (`0x56524B4F`) with frame_type/hs_type discriminators. These are incompatible frame formats. A receiver checking the first 4 bytes would not know which format to expect.

The emerging consensus (SpecWriter conceded to unified magic) resolves this. But the Advocate is right that nobody noticed the incompatibility until the cross-review. This is a process failure, not a content failure: Stage 1 produced four independent deliverables without a shared frame format contract.

**Cartographic assessment:** Two roads were being built to the same destination by different crews, starting from different ends, using different gauge. The roads meet in the middle and do not connect. The solution is to agree on one gauge (unified magic + type bytes). The SpecWriter's concession resolves this.

### W2: Dual SACK formats

**Severity: Critical. Real threat.**

The SpecWriter specifies a stream-level SACK with 12-byte fixed header. The WireEng specifies a connection-level SACK with 14-byte fixed header. Different field sets: the WireEng has ack_timestamp/ack_delay (essential for LEDBAT RTT measurement), the SpecWriter has recv_window (essential for flow control).

The SpecWriter proposed a hybrid approach in the cross-review: stream-level SACK for reliability + connection-level congestion feedback. This is the correct resolution. The WireEng endorsed it. The merged SACK should have both timestamp fields and recv_window.

**Cartographic assessment:** Two parallel roads serving the same traffic. Merge them into one wider road. The merger is straightforward: include all fields from both formats.

### W3: Prologue binding inconsistency

**Severity: Critical. Real threat.**

The WireEng specifies a 7-byte prologue (`MAGIC || selected_version || hs_type`). The SpecWriter does not specify a prologue at all. The Noise specification requires both peers to agree on the prologue byte-for-byte. A one-byte difference produces completely different handshake keys.

This is the most dangerous of the Advocate's findings. A prologue mismatch means every handshake fails with an opaque decryption error and no useful diagnostic. The SpecWriter's concession to pre-handshake version negotiation implicitly accepts the WireEng's prologue binding, but this must be made explicit.

**Cartographic assessment:** The border checkpoint's password was written down twice, differently, by two different clerks. Every traveler's papers are rejected. Fix: one clerk writes the password once, in one place.

### W4: State byte bit 3 conflict

**Severity: High. Real threat but easily resolved.**

The SpecWriter added `convergence_active` at bit 3. The ArchSurgeon's HeartbeatExtension does not include bit 3. The SpecWriter's DAMPED mode distributed coordination depends on bit 3 existing. If the Surgeon's version is taken as authoritative, DAMPED mode coordination breaks.

The resolution is simple: the SpecWriter's bit layout is more complete (assigns all 8 bits), the Surgeon's is a subset (assigns only bit 2). Merge by adopting the SpecWriter's full bit layout. The Surgeon did not contradict bit 3; the Surgeon simply did not assign it. Addition, not conflict.

**Cartographic assessment:** Two cartographers surveyed the same mountain. One drew 5 peaks, the other drew 3 peaks. The 3 peaks are a subset of the 5. Use the fuller map.

### W5: Seat transport assumption (UDP vs HTTP)

**Severity: High. Real threat to the territory's coherence.**

This is a genuinely novel finding. All four Stage 1 experts assumed the seat uses the same UDP transport as mesh peers. But the seat is a web UI served by axum. Web browsers cannot send raw UDP. The seat's transport must be HTTP/WebSocket, not UDP.

The design documents say: "The seat authenticates to a host peer and communicates via request/response and event subscription." The seat document lists verkko-protocol (Connection, Session, Frame, Stream) as a dependency. But if the seat is a web browser, it cannot use verkko-protocol's UDP frames directly.

**Two possible resolutions:**

1. **Seat uses WebSocket for transport, verkko frames for content.** The seat opens a WebSocket to the host peer's axum server. The WebSocket carries verkko protocol frames as binary WebSocket messages. The Noise handshake happens inside the WebSocket tunnel. This means the seat's "connection" is a WebSocket connection, not a UDP association. The verkko protocol's SansIO interface handles this: the consumer drives I/O, and the consumer can use any transport (UDP, WebSocket, TCP) to deliver bytes to the protocol state machine.

2. **Seat uses a separate REST/SSE protocol, not verkko frames.** The seat communicates via HTTP REST endpoints for requests and SSE for events. No Noise handshake, no verkko frames. Authentication uses HTTP-level mechanisms (JWTs, cookies). This would mean the seat does not depend on verkko-protocol at all, which contradicts the dependency declaration.

The design documents say the seat depends on verkko-protocol. This implies option 1: verkko frames over WebSocket. But this is not specified anywhere. The Advocate is correct that this is a gap nobody has raised.

**Cartographic assessment:** There is a road on the map between the seat territory and the protocol territory. But the road is labeled "UDP" and the seat territory is an island accessible only by sea (HTTP). The road exists on the map but not in the terrain. Either the road needs to be rebuilt as a sea route (WebSocket tunnel) or the map needs to be redrawn (seat uses HTTP natively, not verkko frames).

**My recommendation:** Specify that the seat-to-host-peer connection uses verkko protocol frames encapsulated in WebSocket binary messages. This preserves the dependency on verkko-protocol, uses the existing wire formats, and works with web browsers. The WebSocket is the transport binding; the verkko frames are the content. This should be a new defined concept in verkko-seat: `SeatTransport: WebSocket binary messages carrying verkko protocol frames.`

This is a significant finding. It should be addressed in Run 11 as part of the seat specification work.

### W6: PSK in QR code attack window

**Severity: Medium. Known limitation, partially mitigated.**

The PSK is in the QR code. Anyone who photographs the QR code has the PSK. The attacker can use the token before the legitimate invitee, enrolling a malicious device. The mitigation (SAS verification after handshake, key delivery withheld until SAS) means the attacker enters Active state but cannot access channel keys without SAS approval from the admin.

The Advocate's concern about headless Batocera appliances (admin may not notice for hours) is valid but is a UX concern, not a protocol concern. The protocol correctly withholds key material until SAS verification. A headless appliance should push a notification to the admin's seat (phone, tablet) requiring SAS confirmation. The notification mechanism is part of the seat event system, which exists (SeatEvent).

The Advocate's suggestion of a 5-minute PSK window is reasonable. The token already has an `expiry_epoch` field. Setting expiry to creation_time + 5 minutes would limit the attack window. This is a calibration change, not a protocol change.

**Cartographic assessment:** The border checkpoint has a known vulnerability (stolen passports). The guard (SAS verification) catches most attackers, but a distracted guard misses some. Adding a time limit on passport validity is a simple fix. The territory is safe with the time limit.

### W7: f64 determinism across platforms

**Severity: Medium. Cross-cutting concern, not yet critical.**

The health pipeline, dominance computation, EMA coefficients, decay weights, and scar tier weights all use floating-point. On different platforms (x86_64 with SSE vs ARM with NEON), floating-point operations may produce different results in the least significant bits. For health scoring, small differences could cause different peers to compute different health classifications for the same observations.

This is a real concern but not yet critical. The health scores are used for relative ordering (source selection, dominance cascade), not for consensus. Two peers can disagree on the third decimal place of a health score without causing protocol failure. The dominance cascade uses `total_ordering(score, peer_id)` as a tiebreaker, which makes the ordering deterministic even with floating-point differences.

However, the Advocate's concern about NaN propagation is more serious. The SpecWriter's guard ("All pressure values are NaN -> retain current state") only catches the all-NaN case. A single NaN in an array of 8 pressure values passes the guard and produces undefined governor behavior. This is a real bug.

**Cartographic assessment:** The survey instruments (floating-point) have slight calibration differences on different models. The hill heights differ by millimeters. This does not affect navigation. But a broken instrument (NaN) produces readings of "undefined height," which corrupts the map. The single-NaN case must be guarded.

**My recommendation:** The governor should treat any NaN in the pressure array as invalid input and retain the current state. The guard should be `any(is_nan)`, not `all(is_nan)`. This is a one-word fix in the spec text.

### W8: "Consumer" prose pattern creates orphaned prose

**Severity: Low. Cosmetic, not structural.**

The ArchSurgeon's back-reference cleanup replaces "see verkko-mesh: X" with "the consumer (verkko-mesh)." The Advocate notes this still names verkko-mesh five times in a document that claims no dependencies. The cognitive dependency remains.

This is valid but minor. The parenthetical "(verkko-mesh)" provides context to the reader without creating a code dependency. A future consumer of crypto would need to update the parentheticals, which is a documentation maintenance task. The structural dependency is eliminated; the cognitive breadcrumb is acceptable.

**Cartographic assessment:** The Surgeon removed the roads from the map but left the signposts. The signposts point to the neighboring territory. A traveler might follow the signposts, but the roads do not exist, so the traveler cannot actually reach the neighboring territory from this one. The signposts are informative, not navigational. Acceptable.

## Part 6: Emergent Interactions Nobody Has Noticed

### 6.1 The UDP Payload Size Bug Ripples Through Everything

The WireEng uses 1280 bytes as the UDP payload limit. The SpecWriter identified this as a bug: the IPv6 minimum MTU is 1280 bytes at the IP layer; the UDP payload limit is 1232 bytes (1280 - 40 IPv6 header - 8 UDP header). The WireEng's frame size calculations are 48 bytes too generous.

No expert traced the ripple effects:

- **Data frame max payload:** WireEng says 1225 bytes. Corrected: 1177 bytes. Every frame carries 48 fewer bytes of data than the WireEng assumes.
- **Matter chunk payload:** WireEng says 1167 bytes. Corrected: 1119 bytes. The bao proof + chunk data budget shrinks by 48 bytes.
- **Heartbeat + extension:** 36 bytes. Unaffected (well within either limit).
- **SACK with piggyback:** WireEng says 1195 bytes. Corrected: 1147 bytes.
- **Invite token QR code size:** Potentially affected if token exceeds 1232 bytes (unlikely; tokens are ~200-250 bytes).

The 48-byte correction matters most for matter transfer. At 847 bytes of chunk data per frame (WireEng's estimate), the corrected figure is 799 bytes. At 1024-byte bao chunks, this means 2 frames per chunk instead of the WireEng's marginal 1 frame per chunk. The chunk size tradeoff shifts: 512-byte chunks (fitting in 1 frame) become more attractive after the correction.

**This correction should be applied globally before any frame size calculation is finalized.** The SpecWriter noted it. The WireEng should confirm.

### 6.2 The ConvergenceGate Dependency Inversion Is More Dangerous Than Stated

The ArchSurgeon identified that the ConvergenceGate's ConsensusGate predicate "ring_coloring_valid" implies mesh evaluates relay state, creating an undeclared mesh -> relay edge. The Surgeon classified this as HIGH priority and recommended clarifying the predicates as opaque boolean signals.

But the Advocate's W4 finding (state byte bit 3 conflict) interacts with this. The `convergence_active` bit in the state byte (bit 3) is the wire-observable effect of convergence. If the ConvergenceGate predicates are opaque signals from relay, then the relay layer must provide those signals to mesh. How? Via the heartbeat extension? Via a dedicated control message? Via the protocol layer's event surface?

The data flow would be: relay evaluates ring coloring -> relay produces a boolean signal -> signal reaches mesh -> mesh updates ConvergenceGate -> gate state affects convergence_active bit -> bit appears in heartbeat.

But relay depends on mesh (relay -> mesh in the DAG). If relay produces a signal that mesh consumes, that signal must flow upward in the dependency graph (from Level 3 to Level 2). This is not a dependency inversion (the signal is data, not a code dependency), but it is a runtime data flow that goes against the dependency direction. The ArchSurgeon's own back-reference cleanup in crypto established the principle: "crypto describes what it provides; mesh describes what it consumes." By the same principle: relay should describe what it provides (ring coloring status), and mesh should describe what it consumes (an opaque boolean).

The mechanism for delivering this signal is unspecified. It could be:
1. Relay sets a field in its heartbeat extension. But relay does not own heartbeat extension fields (mesh does).
2. Relay sends a dedicated control message. This requires a new msg_type in the protocol layer.
3. The host peer's runtime reads relay state directly (same-process access). This is an implementation detail, not a protocol specification.

Option 3 is the likely runtime behavior: the mesh layer and relay layer run in the same process, and the mesh layer reads relay's ring coloring status via a function call. This is not a wire format concern. But it means the "opaque boolean signal" is not a wire signal; it is a same-process function call. The ArchSurgeon's recommendation to clarify the signal as opaque is correct, but the mechanism (same-process access, not wire protocol) should be stated.

### 6.3 The SansIO Interface Interacts With the Seat Transport Gap

The Advocate's W5 (seat transport is HTTP, not UDP) and my B1 (SansIO consumer interface) interact. If the seat uses WebSocket as transport, the SansIO consumer interface must handle WebSocket messages as input. The `ProtocolInput::BytesReceived` event would carry bytes from a WebSocket binary message, not from a UDP datagram.

This is not a fundamental problem -- the SansIO interface is transport-agnostic by design. But it means the consumer interface specification must explicitly state: "The source of bytes is opaque. The consumer may deliver bytes from UDP datagrams, WebSocket messages, TCP streams, or any other reliable or unreliable transport. The protocol machine does not know or care about the transport."

This statement is implicit in sans-IO design but should be normative in the spec. Without it, an implementer might assume UDP and build transport-specific parsing into the protocol machine.

### 6.4 The Chunk Protocol and the Connection Teardown Interact at Transfer Abort

My B5 (matter chunking) and B12 (connection teardown) interact when a transfer is in progress and the connection is closing. The WireEng defined a TransferAbort message (msg_type 0x0F) with reason codes (integrity_failure, source_gone, cancelled). The WireEng also defined a Connection Close message (msg_type 0x10) with its own reason codes (NORMAL, IDLE_TIMEOUT, etc.).

What happens when a connection closes during a transfer? Two possibilities:
1. The Connection Close message implicitly aborts all in-progress transfers on that connection. The TransferAbort is unnecessary; the connection close covers it.
2. The TransferAbort and Connection Close are independent. A peer could abort a transfer (TransferAbort) without closing the connection, or close the connection (Connection Close) with transfers still in progress.

Option 2 is correct: TransferAbort is a stream-level operation (canceling one transfer), Connection Close is a connection-level operation (shutting down all streams). But the interaction must be specified: "When a connection is closed, all in-progress transfers on that connection are implicitly aborted. The remote peer SHOULD NOT send TransferAbort for transfers on a connection that has received a Connection Close."

### 6.5 The SpecWriter's Errata and the WireEng's Corrections Are Not Symmetric

The SpecWriter found 6 errata in the WireEng's work:
1. UDP payload limit (1280 -> 1232)
2. T4 in handshake payload (not a wire field)
3. Missing stream flags (FIN, RST)
4. Missing invite token signature
5. BLAKE2s -> BLAKE2b
6. Missing "saalis-dht-v1" in DomainSeparator enum

The WireEng found errata in the SpecWriter's work (per-message magic, missing timestamps in SACK, missing user_id in seat auth, missing request_id in relay ops). But the WireEng did not formally list these as "errata" -- they are embedded in findings (W-SW-9 through W-SW-18).

The asymmetry matters for reconciliation: the SpecWriter produced a clean errata list (6 items). The WireEng produced 18 findings with mixed agreement/disagreement. The reconciliation process should extract the WireEng's findings into a parallel errata list so that both experts' corrections are tracked symmetrically.

### 6.6 The f64 NaN Issue Is Broader Than the Advocate Noticed

The Advocate found W7 (f64 determinism) and identified the single-NaN case in ResourcePressures. But NaN can propagate from multiple sources:

- **EMA coefficients** (0.1, 0.01, 0.3): If a health observation is NaN (e.g., division by zero in a throughput calculation), the EMA update produces NaN. The NaN then propagates through all subsequent EMA updates. The health score for that peer becomes NaN permanently.
- **Weighted scar count:** If a scar weight is NaN (e.g., due to a malformed ScarSeverityTier lookup), the weighted sum is NaN. Peer health becomes NaN.
- **Dominance computation:** If the self-dominance score is NaN, the dominance cascade breaks (NaN comparisons are always false).

The fix is not just in the governor. Every floating-point computation in the health pipeline, scar system, and dominance cascade should be NaN-guarded. The spec should state: "No floating-point computation in the health pipeline MAY produce NaN. If any input is NaN, the computation MUST return the previous valid result." This is a cross-cutting invariant that should be stated once in verkko-mesh's health pipeline section and referenced by all consumers.
