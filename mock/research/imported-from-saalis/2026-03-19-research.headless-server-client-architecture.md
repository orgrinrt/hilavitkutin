# Headless Server + Client Architecture

**Date:** 2026-03-19
**Status:** Design direction, not yet in a design round

---

## Core insight

saalis is a headless server. No UI, no filesystem assumption about
where it runs. Any device runs a client that talks to the server
over HTTP. The server orchestrates (catalogue, enrich, schedule,
manage metadata). The client executes locally (download, write
files, symlink, launch games, render UI).

---

## Server responsibilities

The server owns:
- Metadata catalogue (entities, metadata tables, composites)
- Enrichment pipeline (connectors, API calls)
- Source resolution (where to get content)
- Settings engine (7-layer precedence, access control)
- Notification dispatch (SSE)
- Schema management (migrations, _saalis_schema)
- Search (FTS5, hot store prefix matching)
- Scheduling (work units, DAG, batching)
- Extension loading (dlopen, connector plugins)

The server does NOT:
- Store content files (ROMs, media) — client's filesystem
- Render UI — client's job
- Know the client's filesystem layout — client reports paths
- Launch games — client does
- Download files directly to the client — instructs the client to

---

## Client responsibilities

**Design choice: dumb client.** The client is a thin instruction
executor. No DAG scheduler, no columnar storage, no hot/cold
bridge, no morsel execution, no hilavitkutin runtime. All
intelligence lives server-side. The client executes filesystem
operations, reports results, and maintains a local Stash.

The client owns:
- Local filesystem (ROM storage, saves, cache, themes)
- Content downloads (instructed by server, executed locally)
- Symlinking / file placement (instructed by server)
- Game launching (emulator integration)
- UI rendering (whatever framework fits the platform)
- Local Stash (mapping of server entities → local paths)

---

## Client instruction protocol

The server sends structured instructions to the client. The client
executes them and reports back results. All instructions are
idempotent — the client can retry safely.

### Instruction types

**Download:**
```
server → client:
  download(
    url: Str,
    expected_hash: Str,        // BLAKE3, for verification
    expected_size: u64,
    entity_id: EntityId,
  )

client → server (streaming):
  progress(downloaded: u64, total: u64)

client → server (on complete):
  download_complete(
    entity_id: EntityId,
    temp_path: Str,            // where client stored it
    actual_hash: Str,
    actual_size: u64,
  )
```

**Place (write to target directory):**
```
server → client:
  place(
    entity_id: EntityId,
    source: Str,               // client's temp path
    target_dir: Str,           // destination directory
    filename: Str,             // target filename
    strategy: PlacementStrategy,  // move, copy, extract
  )

client → server:
  placed(
    entity_id: EntityId,
    final_path: Str,
  )
```

**Link (symlink/hardlink for consumer-expected paths):**
```
server → client:
  link(
    entity_id: EntityId,
    source: Str,               // client's canonical path
    target: Str,               // consumer-expected path (e.g. Batocera ROM dir)
    link_type: LinkType,       // symlink, hardlink, reflink, copy
  )

client → server:
  linked(
    entity_id: EntityId,
    link_path: Str,
  )
```

**Remove:**
```
server → client:
  remove(
    entity_id: EntityId,
    path: Str,
    preserve_saves: bool,
  )

client → server:
  removed(entity_id: EntityId)
```

**Verify (check file integrity):**
```
server → client:
  verify(
    entity_id: EntityId,
    path: Str,
    expected_hash: Str,
  )

client → server:
  verified(entity_id: EntityId, matches: bool, actual_hash: Str)
```

**Inventory query:**
```
server → client:
  inventory_sync()

client → server:
  inventory(entries: [InventoryEntry])
  // InventoryEntry: entity_id, path, hash, size, last_modified
```

### Instruction sequencing

The server can send compound instructions:

```
download(url, hash, size, entity_id)
  → on complete → place(entity_id, temp, target_dir, filename, extract)
  → on complete → link(entity_id, source, batocera_rom_dir, symlink)
  → on complete → remove(entity_id, temp)
  → report back all paths
```

The client executes sequentially. Each step reports back. The
server updates its metadata with the client-reported paths.

### Progress streaming

Downloads and large operations stream progress back to the server
via the same connection (SSE or WebSocket). The server aggregates
and re-broadcasts to any connected UI clients.

## Local state inventory

The client maintains a local inventory — a mapping of server
entity IDs to local filesystem state:

```
entity_id → {
  canonical_path: Str,      // where the file actually lives
  links: [Str],             // all symlinks/hardlinks pointing to it
  hash: Str,                // BLAKE3 at last verification
  size: u64,
  last_verified: timestamp,
  status: LocalStatus,      // present, downloading, missing, corrupt
}
```

This inventory is:
- Persisted locally (small SQLite or flat file)
- Synced with the server on reconnect
- Used to answer "what do I have locally?" without network round-trip
- Source of truth for the client's filesystem state

The server knows what SHOULD exist on a client. The client knows
what DOES exist. Sync reconciles the two.

## Deployment scenarios

### Scenario 1: All-in-one (console box)

Server and client run on the same Batocera device.
- Server: localhost API
- Client: connects to localhost
- Files: local filesystem, no network transfer needed
- Downloads: server instructs client, client downloads directly

### Scenario 2: NAS server + console client

Server runs on a NAS or home server. Console runs thin client.
- Server: NAS IP/hostname
- Client: Batocera device connects over LAN
- Files: NFS/SMB mount from NAS, or client downloads via API
- Low-power console: no enrichment/catalogue CPU usage

### Scenario 3: Cloud server + multiple clients

Server runs on a VPS or cloud instance.
- Server: public or VPN-accessible
- Clients: any device — console, phone, desktop
- Files: each client manages its own storage
- Server orchestrates, clients execute locally

### Scenario 4: Multi-console household

One server, multiple Batocera consoles.
- Each console is a client with its own inventory
- Server tracks per-client state
- Content can be on shared NFS or downloaded independently
- User profiles (entity metadata) shared via server

## Crate architecture

```
--- hilavitkutin ecosystem (extracted later) ---

hilavitkutin-str           no_std — Str, str_const!, interning
hilavitkutin-persistence   no_std — rkyv cold store, SIEVE
hilavitkutin-ctx           no_std — provider-gated context framework
verkko                     no_std — transport protocol
verkko-protobuf            no_std — micropb payload
verkko-client              no_std — client transport contracts
verkko-cli                 std — CLI tool + lib

--- server (one binary, no_std) ---

saalis-primitives          no_std, zero deps
saalis-contracts           no_std (shared/ + host/ + client/ modules)
saalis-derive              proc-macro
saalis-api                 no_std facade
saalis-scheduler           no_std (wraps hilavitkutin, deferred)
saalis-persistence         no_std (wraps hilavitkutin-persistence)
saalis-server              no_std (composition + verkko endpoint)
extensions (cdylib)        connector-steam, connector-igdb, etc.

--- clients (dumb executors, no hilavitkutin deps) ---

saalis-web                 std — browser GUI (verkko→HTTP bridge)
(saalis-cli)               std — future CLI client
(community clients)        any platform
```

**Dumb client design:** clients have zero hilavitkutin
dependencies. No DAG scheduler, no columnar storage, no
hot/cold bridge. They receive instructions via verkko, execute
filesystem ops, report results, and maintain a local Stash.
All intelligence is server-side.

Clients depend only on:
- `saalis-api` (shared + client modules)
- `verkko-client`
- Platform syscalls

The client instruction protocol is defined in saalis-contracts
(instruction types, response types). Any client implements the
protocol. The server doesn't care what runs on the other end.

## Impact on existing design

### StoragePath / FileStorage / ManagedStorage

These become CLIENT-side operations. The server doesn't touch
the filesystem directly. It sends instructions. The
`StoragePath` contract and `FileStorage` trait move to the
client instruction protocol.

The server's `Storage` trait family (D9) simplifies:
- `PersistentStorage` — SQLite (server-local, stays)
- `ColumnarStorage` — hilavitkutin hot store (server-local, stays)
- `FileStorage` / `ManagedStorage` — become client instructions
- `StorageLocation` markers — define instruction targets, not
  local directories

### ConnectorContext

`ctx.fs::<L>()` no longer returns a local handle — it queues
a client instruction. Or it's removed from the connector context
entirely (connectors don't write files, they resolve source URLs).
File operations are a separate instruction pipeline.

### Downloads

The download engine is split:
- Server resolves sources (URL, hash, size) via connectors
- Server sends download instruction to client
- Client executes download, streams progress
- Client reports completion, server updates metadata

### Settings

Server settings are server-side. Client settings (UI theme,
display preferences) are client-side. The settings API serves
both — clients query/update their own profile settings via
the API.
