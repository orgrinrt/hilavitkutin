# Deep Dive: Security Considerations for Saalis

Research date: 2026-03-15

---

## 1. PIN-Based Authentication Security

### The Problem With Short PINs

Saalis runs on a household media appliance where users authenticate with PINs (stored as `PinHash`
in the `AuthConfig` metadata table). OWASP [explicitly recommends against 4-digit PINs for
authentication](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html),
because the keyspace (10,000 combinations) is trivially brute-forceable. Even with Argon2id, a
4-digit PIN hashed at 1 second per attempt takes under 3 hours to exhaust on a single core.

**Recommendation:** Support 4--8 digit PINs. The UI should default to 6 digits (1 million
combinations), matching the OWASP guidance that [PINs should be 6--12 digits when used in limited
contexts](https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html). For a
living-room appliance where an attacker has physical access to the same LAN, PIN length is the
weakest link; Argon2id only buys time.

### Argon2id Parameters for Low-Power ARM

Saalis targets Batocera on ARM SBCs (Raspberry Pi 4/5, Odroid, RockPro64). These boards typically
have 1--8 GB RAM and 4 cores at 1.5--2.4 GHz. The
[RFC 9106 recommendations](https://datatracker.ietf.org/doc/rfc9106/) (m=64 MiB, t=3, p=1) are
designed for server hardware and would cause multi-second stalls on a Pi 4.

Recommended parameters for this class of device:

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `m` (memory) | 16 MiB (16384 KiB) | Fits comfortably on 1 GB boards. Still makes GPU attacks expensive -- a GPU with 8 GB VRAM can only run ~500 parallel hashes. |
| `t` (iterations) | 3 | More iterations over less memory is less effective against ASIC/GPU attacks than more memory with fewer passes, but t=3 provides a minimum floor. |
| `p` (parallelism) | 2 | Pi 4 has 4 cores; p=2 keeps one hash from monopolising the CPU while still getting speedup. |
| Target time | 200--500 ms | Acceptable for a PIN entry flow. Benchmark on actual hardware and adjust `m` upward if the target allows. |

The `argon2` crate (from RustCrypto) supports Argon2id with configurable parameters. Always
benchmark on target hardware: `let start = Instant::now(); hash(pin); let elapsed = start.elapsed();`
then adjust `m` until `elapsed` is within the target window.

### Timing Attack Resistance

Argon2id itself runs in data-independent time for the first half of the first pass (the Argon2i
portion), providing [side-channel resistance](https://en.wikipedia.org/wiki/Argon2). However, the
**verification step** must use constant-time comparison. The `argon2` crate's `verify_password()`
uses `subtle::ConstantTimeEq` internally, which is correct. If implementing manually, never use
`==` on hash bytes; always use `subtle::ct_eq()` or the `constant_time_eq` crate.

Even with constant-time comparison, an attacker who can distinguish "valid user, wrong PIN" from
"invalid user" gets a user-enumeration oracle. Mitigate by always running the full Argon2id
computation even for nonexistent users (hash against a dummy stored hash).

### Brute Force Protection

With a 6-digit PIN, the keyspace is 1 million. At 300 ms per hash, exhaustive search on-box takes
~83 hours. But the real threat is network brute force from another device on the LAN.

**Attempt limiting strategy:**

| Threshold | Action |
|-----------|--------|
| 5 failures in 60 seconds | 30-second lockout (delay response) |
| 15 failures in 10 minutes | 5-minute lockout |
| 50 failures in 1 hour | 30-minute lockout + admin notification |

Store attempt counts in-memory (not the database) to avoid write amplification. Use a per-user
counter keyed by `EntityId`, not by IP address -- on a local network, DHCP churn makes IP-based
tracking unreliable. Reset the counter on successful authentication.

The `Governor` crate (tower-compatible) provides rate limiting middleware for axum, but PIN
attempts should use application-level counters since the lockout semantics are user-scoped, not
request-scoped.

## 2. Session Security

### Token Generation

Session tokens must be cryptographically random and long enough to resist brute-force guessing.
The [OWASP Session Management Cheat
Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html)
recommends at least 128 bits of entropy; 256 bits provides a comfortable margin.

```rust
use rand::RngCore;

fn generate_session_token() -> [u8; 32] {
    let mut token = [u8; 32]; // 256 bits
    rand::thread_rng().fill_bytes(&mut token);
    token
}
```

Encode as URL-safe base64 (43 characters) for cookie storage. The `rand` crate's `thread_rng()`
uses the OS CSPRNG (`getrandom` -> `/dev/urandom` on Linux), which is appropriate.

### Cookie Attributes

```
Set-Cookie: saalis_session=<token>;
    HttpOnly;
    SameSite=Lax;
    Path=/;
    Max-Age=86400;
    Secure  // conditional, see §6
```

- **HttpOnly** -- prevents JavaScript access, mitigating XSS-based session theft. Mandatory.
- **SameSite=Lax** -- blocks cross-origin subrequests (POST, PUT, DELETE) from carrying the
  cookie, which neutralises most CSRF vectors. `Strict` would break navigating to saalis from a
  bookmark manager or link, so `Lax` is the pragmatic choice.
- **Secure** -- only set this flag when HTTPS is active (see section 6). On plain HTTP, the
  `Secure` flag causes the browser to never send the cookie.
- **Path=/** -- scope to the entire application. Do not scope to `/api` since the web UI and API
  share the same origin.
- **Max-Age** -- 24 hours for a household appliance is reasonable. Shorter sessions (e.g., 1 hour)
  would annoy users on a media center. Consider "remember me" extending to 30 days for the
  Admin role, with re-authentication required for sensitive operations (credential access, role
  changes).

### Session Fixation Prevention

Session fixation occurs when an attacker sets a known session ID before the victim authenticates,
then hijacks the session after login. Prevention is straightforward: **always generate a new
session token on successful authentication**, never reuse a pre-authentication session ID.

In the Saalis architecture (§11.1.3), `SessionData` is a core-only metadata table. On successful
PIN verification:

1. Delete any existing `SessionData` row for the old token.
2. Generate a new 256-bit token.
3. Insert a new `SessionData` row with the new token, user's `EntityId`, creation timestamp, and
   expiration.
4. Set the cookie with the new token.

The `tower-sessions` crate handles this via `session.cycle_id()`, which regenerates the session
identifier while preserving session data.

## 3. Credential Storage Encryption

### Why ChaCha20-Poly1305

The architecture (section 11.5) specifies application-level encryption for all `Credential`-typed
settings fields. ChaCha20-Poly1305 (RFC 8439) is the right choice for this appliance context:

- **No hardware AES on low-end ARM.** Raspberry Pi 3 lacks AES-NI; ChaCha20 is
  [3x faster than AES-GCM in software](https://docs.rs/chacha20poly1305) on such hardware.
- **Misuse-resistant nonce handling.** XChaCha20-Poly1305 uses a 192-bit nonce, making random
  nonce generation safe (collision probability is negligible at 2^-96 for 2^48 messages).
- **Authenticated encryption.** Poly1305 MAC detects tampering. An attacker who modifies the
  ciphertext in the SQLite database gets a decryption failure, not a corrupt credential.

The Rust crate `chacha20poly1305` (from RustCrypto) is pure Rust, audited, and widely used.

### Key Derivation from Master Password/Key

The architecture defines a master key hierarchy (section 11.5):

1. `SAALIS_MASTER_KEY` environment variable
2. OS keychain entry
3. Auto-generated secret in data directory

When deriving encryption keys from a master secret, use
[HKDF-SHA256](https://crates.io/crates/hkdf) (RFC 5869) with domain-separated info strings:

```rust
use hkdf::Hkdf;
use sha2::Sha256;

fn derive_credential_key(master_key: &[u8]) -> [u8; 32] {
    let hkdf = Hkdf::<Sha256>::new(None, master_key);
    let mut key = [0u8; 32];
    hkdf.expand(b"saalis-credential-encryption-v1", &mut key)
        .expect("32 bytes is a valid length for HKDF-SHA256");
    key
}
```

The info string `"saalis-credential-encryption-v1"` provides domain separation: the same master
key produces different derived keys for different purposes (e.g., a separate key for backup
encryption). The version suffix enables key rotation by deriving a new key with `v2`.

### Key Rotation

Key rotation for at-rest credential encryption follows three phases:

1. **Derive new key.** Bump the version in the HKDF info string (or supply a new master key).
2. **Re-encrypt.** Read each encrypted credential, decrypt with the old key, re-encrypt with the
   new key, write back. This is a batch operation -- wrap in a SQLite transaction.
3. **Retire old key.** After verifying all credentials decrypt successfully with the new key,
   remove or archive the old master key material.

For API keys stored by connectors (IGDB, ScreenScraper, Steam), saalis should log key rotation
events but does not need to rotate the external API keys themselves -- that is the provider's
responsibility. The [90-day rotation
recommendation](https://blog.gitguardian.com/api-key-rotation-best-practices/) applies to keys
that saalis *issues* (e.g., static API keys for machine-to-machine vault access per section 11.6),
not keys it stores on behalf of external services.

### Nonce Management

Each credential value stored in the database needs a unique nonce. With XChaCha20-Poly1305's
192-bit nonce, generating a random nonce per encryption is safe. Store the nonce alongside the
ciphertext (prepend the 24-byte nonce to the ciphertext blob). The storage format is:

```
[24-byte nonce][ciphertext][16-byte Poly1305 tag]
```

This is the standard construction used by the `chacha20poly1305` crate's `encrypt_in_place` method.

## 4. CSRF Protection

### HTMX and CSRF

Saalis serves an HTMX-driven web UI. HTMX makes requests via `hx-get`, `hx-post`, etc., which
are XHR requests from the browser. Unlike traditional form submissions, HTMX requests are subject
to CORS preflight for cross-origin requests, but the primary CSRF concern is same-origin POST/PUT/
DELETE requests triggered by a malicious page.

The [HTMX security guide](https://htmx.org/essays/web-security-basics-with-htmx/) recommends
using CSRF tokens for all mutating requests. HTMX supports injecting headers on every request via
`hx-headers` or the `htmx:configRequest` event.

### Defense Layers

**Layer 1: SameSite=Lax cookies.** This blocks cross-site subrequests from carrying the session
cookie for POST/PUT/DELETE. It does NOT protect against same-site attacks (e.g., if another
application on the same host has an XSS vulnerability). On a single-application appliance, this
risk is minimal.

**Layer 2: Double-submit cookie pattern.** The
[OWASP CSRF Prevention Cheat
Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html)
describes the signed double-submit cookie pattern:

1. On session creation, generate a random CSRF token. Store an HMAC of the token in a
   `__Host-csrf` cookie (HttpOnly=false so JavaScript can read it, SameSite=Strict, Secure if
   HTTPS).
2. HTMX reads the cookie value and includes it as an `X-CSRF-Token` header on every mutating
   request (configured via `htmx:configRequest`).
3. Server-side middleware recomputes the HMAC from the header value and compares it to the cookie
   value. If they match, the request originated from a page that could read the cookie (i.e.,
   same origin).

The HMAC signing prevents an attacker who can set cookies (e.g., via a subdomain) from forging
valid tokens. Use HMAC-SHA256 keyed with a server-side secret derived from the master key.

**Layer 3: `Origin` / `Referer` header validation.** As a belt-and-suspenders check, reject
mutating requests where the `Origin` header does not match the expected host. This catches attacks
that somehow bypass SameSite (e.g., browser bugs).

For a local-network appliance, the combination of SameSite=Lax + signed double-submit cookie
provides strong CSRF protection without server-side state.

## 5. XSS Prevention

### Maud's Automatic HTML Escaping

[Maud](https://maud.lambda.xyz/) is a compile-time HTML template engine for Rust. All dynamic
values interpolated into templates are [automatically HTML-escaped by
default](https://rustworks.github.io/maud-Compile-time-HTML-template/text-escaping.html). The
string `<script>alert('xss')</script>` renders as
`&lt;script&gt;alert(&#x27;xss&#x27;)&lt;/script&gt;`.

To render trusted HTML, the developer must explicitly wrap it in `PreEscaped()`. This opt-in model
is significantly safer than opt-out escaping (e.g., Django's `|safe` filter or Jinja2's
`|safe`), because forgetting to escape is the common vulnerability -- and with Maud, the default
is safe.

**Rules for saalis:**
- Never use `PreEscaped()` with user-supplied data.
- If rich text display is needed (e.g., game descriptions from connectors), sanitize with the
  `ammonia` crate before passing to `PreEscaped()`. Ammonia is a whitelist-based HTML sanitizer
  written in Rust.
- Connector-supplied metadata (titles, descriptions) should always go through Maud's default
  escaping path, even though it is not "user input" in the traditional sense -- a malicious
  connector could inject scripts via metadata.

### Content-Security-Policy Headers

CSP is the second line of defense if an XSS escaping bug slips through. Recommended policy for
saalis:

```
Content-Security-Policy:
    default-src 'self';
    script-src 'self';
    style-src 'self' 'unsafe-inline';
    img-src 'self' data: https:;
    connect-src 'self';
    font-src 'self';
    object-src 'none';
    frame-ancestors 'none';
    base-uri 'self';
    form-action 'self';
    require-trusted-types-for 'script';
```

- **`script-src 'self'`** -- no inline scripts, no `eval()`. HTMX is loaded as a file, not
  inline. This single directive blocks most XSS payloads.
- **`style-src 'unsafe-inline'`** -- HTMX and many CSS frameworks use inline styles. If the
  saalis theme system avoids inline styles, tighten to `'self'` only.
- **`img-src 'self' data: https:`** -- cover art images may be proxied from external HTTPS
  sources or served as data URIs from the cache.
- **`frame-ancestors 'none'`** -- prevents clickjacking. Equivalent to `X-Frame-Options: DENY`.
- **`require-trusted-types-for 'script'`** -- [Trusted
  Types](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Content-Security-Policy/require-trusted-types-for)
  prevents DOM-based XSS by requiring all DOM sink assignments to go through a Trusted Type
  policy. Browser support is Chrome/Edge only as of 2026, but it degrades gracefully.

Set these headers via tower middleware on every response. The `tower-http` crate's
`SetResponseHeaderLayer` is suitable.

## 6. Local Network Security Model

### Does a LAN Appliance Need HTTPS?

This is the central tension. Saalis serves credentials (API keys, passwords) over its web UI and
vault API. On a trusted home network, the threat model is different from the public internet:

**Arguments against HTTPS on LAN:**
- No CA will issue certificates for `.local` mDNS names or RFC 1918 IP addresses.
- Self-signed certificates cause browser warnings that train users to click through security
  prompts -- [arguably worse than no
  HTTPS](https://community.letsencrypt.org/t/certificates-for-devices-only-reachable-via-local/22908).
- Complexity: certificate generation, renewal, and distribution on a headless appliance.

**Arguments for HTTPS on LAN:**
- Any device on the same WiFi network can sniff unencrypted traffic (ARP spoofing, promiscuous
  mode).
- Modern browsers increasingly restrict features (Service Workers, Secure cookies, Clipboard API)
  to secure contexts.
- Mixed-content policies make it impossible to embed saalis in an HTTPS page.

### Recommended Approach

**Default: HTTP with security headers.** For v1, serve over plain HTTP on the local network. Set
`Secure` cookie flag only when HTTPS is detected (check `X-Forwarded-Proto` or the listener
binding). This matches the threat model of similar appliances (Home Assistant, Jellyfin, Plex --
all default to HTTP on LAN).

**Optional: self-signed CA.** For users who want HTTPS, saalis can generate a self-signed CA on
first run and issue a server certificate. The CA certificate is downloadable from the web UI for
installation on client devices. [Home Assistant's approach of a local
CA](https://community.home-assistant.io/t/certificate-authority-and-self-signed-certificate-for-ssl-tls/196970)
is a proven model for home appliances.

**mDNS discovery.** Saalis should advertise via mDNS (`_http._tcp` or a custom `_saalis._tcp`
service type) so clients can discover it at `saalis.local`. mDNS itself is unauthenticated -- any
device on the network can respond to queries. This is acceptable for service discovery (the user
sees the name in a browser) but should never be trusted for security decisions.

### Credential Exposure Mitigation

Even without HTTPS, mitigate credential exposure:

- **Vault API responses** include `Cache-Control: no-store` (section 11.6) to prevent caching.
- **Credential values** are never included in HTML responses. The vault API returns JSON; the web
  UI fetches credentials via XHR and displays them transiently (e.g., a "reveal" button that
  shows for 5 seconds).
- **`Display` impl masking** (section 11.5): `Credential` types render as `"sk-****"` in all
  non-vault contexts, preventing accidental leakage in logs, error messages, or SSE event
  payloads.

## 7. Multi-User Isolation

### Role-Based Credential Visibility

Saalis has three first-party roles: `Admin`, `User`, `Child` (section 11.4). Credential isolation
is enforced by the `SetAccess` compile-time trait system (section 10.8):

| Role | `Credentials` domain | `Infrastructure` domain | Other settings |
|------|----------------------|------------------------|----------------|
| Admin | ReadWrite | ReadWrite | ReadWrite |
| User | Own profile only | ReadOnly | ReadWrite (own scope) |
| Child | Hidden | Hidden | ReadOnly (own scope) |

"Own profile only" means a `User` can read/write credentials scoped to their own profile but
cannot see credentials scoped to other profiles or to the system scope. This is enforced at two
levels:

1. **Query layer.** The `ConnectorContext` resolves settings with the active user's profile scope.
   A `User`'s `ctx.settings()` call returns their own credentials, never another user's.
2. **API layer.** The vault API (section 11.6) checks the session's role before serving
   cross-user credential requests. `GET /api/vault/{provider}/{user_id}` is admin-only.

### Child Role Restrictions

The `Child` role is designed for parental controls on a family media center:

- **No credential access.** The `Credentials` domain is `Hidden` for `Child`. They cannot see or
  modify any API keys or passwords, even their own.
- **Age-restricted content.** The `AgeRestriction` column in `UserProfile` metadata is respected
  by query filters. Connectors' catalogue results are filtered before display.
- **No settings modification.** Infrastructure and system-level settings are hidden. Children can
  only modify their own profile-scoped display preferences.
- **No role escalation.** `UserRole` is a core-only metadata table (section 11.1.2). Plugins
  cannot write to it, and the web UI validates role changes server-side (only admins can assign
  roles).

### Implementation Considerations

The type-system-based security boundary (section 11.1.2 -- `AuthConfig` and `SessionData` in
`saalis-core`, not `saalis-sdk`) is elegant but relies on plugins being compiled separately and
not having access to core internals. For `dlopen`-loaded `.saalix` extensions, this is enforced by
the compilation boundary: plugins link against the SDK, not core. They literally cannot name the
`AuthConfig` type.

## 8. Extension Sandboxing

### Threat Model: Malicious `.saalix` Extensions

A `.saalix` extension is a shared library loaded via `dlopen`. Unlike WebAssembly, native code has
**no sandbox**. A malicious or compromised extension could:

- Read arbitrary memory (including decrypted credentials in the host process).
- Make arbitrary system calls (network, filesystem, process spawning).
- Overwrite function pointers or corrupt host state.
- Exfiltrate credentials to an external server.

This is the most significant security risk in the architecture.

### Mitigation via ConnectorContext Scoping

The `ConnectorContext<C>` (section 1.2.4) limits *intended* access:

- **HTTP requests** go through `ctx.http()`, which the host can rate-limit and log.
- **Storage** goes through `ctx.storage()`, which is scoped to the connector's own storage
  location via `LocationHandle<L, S>` bounds.
- **Settings** are resolved per-connector via `ctx.settings()`, returning only that connector's
  `Settings` struct.
- **Metadata writes** are gated by `SupportsEntity` trait bounds -- a connector cannot write to
  metadata tables it has not declared an association for.

However, these are **soft boundaries**. A native-code extension that deliberately bypasses the SDK
API can do anything the host process can do. `ConnectorContext` scoping protects against bugs and
honest-but-curious connectors, not against actively malicious code.

### Hardening Options

**Short-term (v1):**
- **Extension signing.** Require `.saalix` files to be signed with an Ed25519 key. The host
  verifies the signature before `dlopen`. First-party extensions are signed with the project key.
  Users can add trusted keys for third-party extensions.
- **Permission manifest.** Each `.saalix` declares its required capabilities (network access,
  filesystem paths, metadata tables). The host displays these to the user before enabling the
  extension.
- **Curated registry.** Host a registry of reviewed extensions. Warn users when loading
  unregistered extensions.

**Medium-term:**
- **Process isolation.** Run each extension in a separate process, communicating via IPC
  (Unix domain sockets + `repr(C)` message passing). This prevents memory access across
  extensions but adds latency and complexity.
- **Seccomp/Landlock filtering (Linux).** On Linux, apply seccomp-bpf or Landlock policies to
  the extension process, restricting system calls to a whitelist. This is how Chromium sandboxes
  renderers.

**Long-term:**
- **WebAssembly migration.** Compile extensions to WASM and run in
  [Wasmtime](https://docs.wasmtime.dev/security.html) or Wasmer. WASM provides true
  sandboxing: isolated linear memory, no direct system call access, fine-grained capability
  control. The `repr(C)` ABI would need to be replaced with WIT (WebAssembly Interface Types)
  component model interfaces. This is a significant architectural change but provides provable
  isolation.

For a household appliance where extensions are installed deliberately by the admin user, the
signing + manifest approach is a pragmatic balance. The WASM migration path should remain an
option for when the extension ecosystem grows beyond trusted first-party connectors.

## 9. API Authentication

### Dual Authentication Model

Section 11.6 specifies two authentication paths:

**Path 1: Session cookies (web UI).** The browser sends the `saalis_session` cookie on every
request. The session middleware resolves the cookie to a `ResolvedUser` (section 11.1.3). This
is the primary path for interactive use.

**Path 2: Static API key (machine-to-machine).** For scripts, Batocera profile-switch hooks, and
automation, a static API key is configured in the saalis settings. The key is sent as a
`Bearer` token in the `Authorization` header:

```
GET /api/vault/screenscraper HTTP/1.1
Authorization: Bearer sk-xxxxxxxxxxxxxxxxxxxx
```

The API key resolves to a user context (the admin user, or a service account entity). This path
does NOT require CSRF protection because [browsers do not automatically send `Authorization`
headers](https://rifatcse09.medium.com/csrf-cookies-and-bearer-tokens-what-to-use-when-and-why-most-apps-get-it-wrong-76ddd791946b)
-- only cookies are sent automatically.

### Route Protection

Every API route (except the login endpoint and static assets) must check authentication. In axum,
this is a tower middleware layer that:

1. Checks for `Authorization: Bearer <token>` header first.
2. Falls back to the `saalis_session` cookie.
3. If neither is present or valid, returns `401 Unauthorized`.
4. If valid, inserts the `ResolvedUser` into the request extensions for handler access.

```rust
// In handler:
async fn get_credentials(user: ResolvedUser) -> impl IntoResponse {
    // user.role, user.entity_id, etc. are available
    // axum extracts ResolvedUser from request extensions
}
```

### API Key Security

Static API keys should be:

- At least 256 bits of entropy, generated with `rand::thread_rng()`.
- Stored hashed (SHA-256 is sufficient since the keys have high entropy -- unlike passwords, they
  do not need Argon2id).
- Rotatable: the admin can regenerate the key from the web UI, which invalidates the old one.
- Scoped: in future, API keys could carry permission scopes (read-only, vault-only, etc.), but
  for v1, a single admin-level key is sufficient for the Batocera integration use case.

## 10. Audit Logging

### What to Log

The [Berkeley Security Audit Logging
Guideline](https://security.berkeley.edu/security-audit-logging-guideline) and [NIST SP
800-92](https://nvlpubs.nist.gov/nistpubs/legacy/sp/nistspecialpublication800-92.pdf) recommend
logging security-relevant events. For saalis, the relevant events are:

**Authentication events:**
- Successful login (user, timestamp, source IP)
- Failed login attempt (user attempted, timestamp, source IP)
- Lockout triggered (user, duration, attempt count)
- Lockout expired / manual unlock
- Session created, expired, or invalidated
- API key used (timestamp, source IP, endpoint accessed)

**Authorization events:**
- Role change (who changed, old role, new role, by whom)
- Access denied (user, resource attempted, role)
- Cross-user credential access (admin accessing another user's credentials)

**Credential events:**
- Credential created or updated (provider, user, timestamp -- never the value)
- Credential accessed via vault API (provider, user, requesting user, source IP)
- Credential decryption failure (indicates tampering or key mismatch)
- Master key rotation

**Extension events:**
- Extension loaded (name, signature status, hash)
- Extension signature verification failure
- Extension runtime error

**System events:**
- Application startup and shutdown
- Configuration changes (setting key, old value hash, new value hash, by whom)
- Database migration executed

### Log Format and Storage

Audit logs should be structured (JSON lines or similar) for machine parsing. Each entry should
include:

```json
{
  "ts": "2026-03-15T14:30:00Z",
  "event": "auth.login.success",
  "user_id": 42,
  "source_ip": "192.168.1.100",
  "details": {}
}
```

**Storage:** Write audit logs to a separate SQLite table (`audit_log`) rather than the application
log file. This provides:
- Queryability (e.g., "show all credential accesses in the last 7 days").
- Integrity -- the audit table can be write-only from the application's perspective (no UPDATE or
  DELETE operations exposed).
- Separation from debug/application logs, which may be rotated aggressively.

**Retention:** For a household appliance, 90 days is reasonable. Implement a scheduled cleanup
(via the Housekeeper subsystem) that deletes audit entries older than the retention period. If
the appliance runs out of storage, audit logs should be preserved over application cache data
(the Sieve eviction system should not touch the audit table).

### Tamper Resistance

For a local appliance, full tamper-proofing (hash chains, remote attestation) is overkill.
Pragmatic measures:

- The audit table should not be writable via the web UI or API. Only the application process
  inserts rows.
- Include a running HMAC over the log entries (keyed with a derived key from the master key).
  This detects deletion or modification of entries, even if it does not prevent it.
- Synchronize timestamps with NTP if available on the network. The Batocera host typically runs
  NTP by default.

## Summary of Recommendations

| Area | Key Recommendation |
|------|--------------------|
| PIN auth | Default to 6-digit PINs; Argon2id with m=16MiB, t=3, p=2; per-user lockout after 5 failures |
| Sessions | 256-bit random tokens; HttpOnly + SameSite=Lax cookies; regenerate on login |
| Credential encryption | XChaCha20-Poly1305; HKDF-SHA256 key derivation; random 192-bit nonces |
| CSRF | SameSite=Lax + signed double-submit cookie; HTMX injects `X-CSRF-Token` header |
| XSS | Maud auto-escaping by default; `script-src 'self'` CSP; never `PreEscaped()` on untrusted data |
| Network | HTTP by default; optional self-signed CA; `Cache-Control: no-store` on vault responses |
| Multi-user | Compile-time `SetAccess` per role+domain; core-only metadata for auth tables |
| Extensions | Signing + manifest for v1; process isolation medium-term; WASM long-term |
| API auth | Session cookies for web UI; Bearer tokens for M2M; middleware on all routes |
| Audit logging | Structured logs in SQLite; auth + credential + extension events; 90-day retention |
