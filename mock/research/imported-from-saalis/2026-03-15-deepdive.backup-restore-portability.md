Research date: 2026-03-15

## 1. The `.saalis` Snapshot Format

A `.saalis` file is the canonical unit of backup and portability. It must be self-describing, versioned, and independently verifiable. The format is a ZIP archive with a custom magic header and a mandatory manifest.

### Container Structure

```
snapshot-2026-03-15T14:30:00Z.saalis
  ├── manifest.json            # version, created_at, saalis_version, sections[]
  ├── db/
  │   └── saalis.db            # full SQLite database (VACUUM INTO output)
  ├── config/
  │   ├── settings.toml        # application settings (non-secret)
  │   └── profiles/            # per-user profile configs
  │       ├── admin.toml
  │       └── user_*.toml
  ├── credentials/
  │   └── vault.enc            # encrypted credential blob (XChaCha20-Poly1305)
  ├── media/                   # optional — only if user opts in
  │   ├── covers/
  │   ├── screenshots/
  │   └── thumbnails/
  └── meta/
      ├── hot_manifest.json    # last warm segment list for fast startup
      └── checksums.blake3     # per-file BLAKE3 hashes for integrity
```

### Why ZIP

ZIP is the container, not the compression. Individual entries use zstd compression (via the
`zip` crate's `CompressionMethod::Zstd` support). ZIP was chosen over tar because it provides
random access to individual entries --- critical for partial restore (section 9). The `.saalis`
extension and a leading magic sequence (`SAAL` + version u16 LE, prepended before the ZIP
stream) allow file-type detection without relying on the extension.

### Manifest Schema

```json
{
  "format_version": 1,
  "saalis_version": "0.4.0",
  "created_at": "2026-03-15T14:30:00Z",
  "machine_id": "a1b2c3d4",
  "sections": ["db", "config", "credentials", "media", "meta"],
  "db_schema_version": 12,
  "credential_encryption": {
    "algorithm": "xchacha20-poly1305",
    "kdf": "argon2id",
    "kdf_params": { "m": 16384, "t": 3, "p": 2 },
    "salt": "<base64>"
  }
}
```

The `format_version` field governs how the archive itself is structured. The `db_schema_version`
field records which migration the database has applied, enabling forward-compatible restore (see
section 6). The `sections` array lists which top-level directories are present --- a snapshot
without `media/` is still a valid snapshot. `machine_id` is a stable hash of hardware identifiers
used to detect cross-machine restore (relevant for credential re-encryption, section 4).

## 2. SQLite Backup API: Online Backup Without Downtime

Saalis maintains an in-memory hot store with SQLite as the cold persistent layer. Backup must
not block the scheduler or the web UI. SQLite provides two mechanisms suited to this.

### `VACUUM INTO`

`VACUUM INTO '<path>'` creates a defragmented, minimal-size copy of the database in a single
transaction. It captures a consistent snapshot even under concurrent writes because it holds a
read lock for the duration of the copy
([SQLite VACUUM docs](https://sqlite.org/lang_vacuum.html)). The output file is a standalone
database --- no WAL, no SHM, no journal. This makes it ideal for the `.saalis` archive because
the backup file is self-contained.

Advantages:
- **Atomic snapshot.** The output reflects a single point-in-time state.
- **Compacted output.** Free pages are reclaimed, reducing archive size.
- **No progress callbacks needed** for saalis-scale databases (metadata-only, typically < 50 MB).

Disadvantages:
- Requires temporary disk space equal to the database size.
- No incremental option --- always a full copy.
- On very large databases (GB+), it holds a read transaction for the entire copy duration, which
  can delay WAL checkpointing.

For saalis, where the SQLite database stores metadata (not media files), `VACUUM INTO` is the
primary backup method. A 50 MB metadata database copies in under a second on eMMC/SD.

### `sqlite3_backup` API (via rusqlite)

The [online backup API](https://sqlite.org/backup.html) copies a database page-by-page, allowing
the caller to yield between steps. In Rust, `rusqlite::backup::Backup` wraps this:

```rust
use rusqlite::{Connection, backup};

let src = Connection::open("saalis.db")?;
let mut dst = Connection::open("backup.db")?;
let backup = backup::Backup::new(&src, &mut dst)?;
// Copy 100 pages at a time, sleeping 250ms between steps
backup.run_to_completion(100, std::time::Duration::from_millis(250), None)?;
```

The `run_to_completion` method implements the "online backup of a running database" pattern from
SQLite's documentation. Between each step, the source database is unlocked, allowing other
connections to read and write. If the source changes between steps, the backup restarts from the
modified pages (not from scratch) as long as the backup is performed within the same process that
writes to the database
([rusqlite backup docs](https://docs.rs/rusqlite/latest/rusqlite/backup/index.html)).

**When to use which:**

| Scenario | Method |
|----------|--------|
| User-initiated "Export backup" | `VACUUM INTO` --- clean, compacted, single-shot |
| Scheduled background backup | `sqlite3_backup` with yield --- avoids blocking the scheduler |
| Pre-migration safety copy | `VACUUM INTO` --- the pre-migration copy must be a frozen snapshot |

### Interaction with WAL Mode

Saalis runs SQLite in WAL mode with application-controlled checkpoints (as established in the
persistence strategy synthesis). During `VACUUM INTO`, SQLite internally reads from both the main
database file and the WAL, producing a consistent output that includes all committed WAL
transactions. The WAL is not modified or checkpointed by the backup operation itself. This means
a backup can run concurrently with the epoch-batched dirty flush cycle without interference.

## 3. Incremental Backups: What Changed Since Last Backup?

Full `VACUUM INTO` snapshots are sufficient for weekly or on-demand backups, but for continuous
protection (especially on appliances where SD card failure is the primary threat), we need
incremental strategies.

### WAL-Based Change Detection

SQLite's WAL file is an append-only log of modified pages. Between checkpoints, it contains
every page that has been written since the last checkpoint. This is conceptually an incremental
diff of the database. However, SQLite does not expose a stable "WAL offset" API that would let
an external tool say "give me all changes since offset X."

[Litestream](https://litestream.io/how-it-works/) solves this by taking over WAL management
entirely: it starts a long-running read transaction to prevent automatic checkpointing, copies
new WAL pages to a "shadow WAL" as they appear, and streams them to a replica. It then manually
triggers checkpoints when appropriate. This approach provides near-real-time incremental backup
with sub-second RPO (recovery point objective).

### Epoch-Based Change Tracking (Application-Level)

Saalis already tracks dirty segments by epoch number in the hot-to-cold sync layer. We can
exploit this for incremental backup without relying on WAL internals:

1. **Backup epoch counter.** Maintain a `_backup_watermark` table with a single row:
   `last_backup_epoch INTEGER`. When a full backup completes, record the current epoch.
2. **Dirty-since query.** Each entity row has a `last_modified_epoch` column. An incremental
   backup is: `SELECT * FROM entities WHERE last_modified_epoch > :last_backup_epoch`, plus the
   same for metadata tables.
3. **Incremental archive.** The incremental `.saalis` file contains a SQL dump of only the
   changed rows, stored as `db/incremental.sql` instead of `db/saalis.db`. The manifest records
   `"backup_type": "incremental"` and `"base_epoch": N, "through_epoch": M`.
4. **Restore chain.** To restore from incrementals, apply the last full backup, then replay each
   incremental in epoch order. The restore process runs each `incremental.sql` within a single
   transaction with `INSERT OR REPLACE` semantics.

This gives saalis control over incremental granularity without depending on SQLite internals or
external tools like Litestream. The trade-off is that change tracking is at the application
level (row-granularity), not at the page level. For a metadata database this is fine; for a
multi-GB media cache it would not be.

### Litestream as an Optional Sidecar

For users who want continuous, zero-RPO backup to cloud storage, saalis can document (but not
bundle) Litestream as a sidecar process. Litestream watches the SQLite database file and streams
WAL changes to S3, Backblaze B2, or a local directory. This requires no code changes in saalis
--- Litestream operates at the filesystem level. The only constraint is that saalis must not use
`PRAGMA wal_autocheckpoint=0` if Litestream is managing the WAL; instead, Litestream's own
checkpoint policy takes over. A configuration flag (`backup.litestream_mode = true`) would
disable saalis's manual checkpoint calls.

## 4. Credential Encryption in Backups

### Encryption Scheme

Credentials in the `.saalis` archive are stored in `credentials/vault.enc`, encrypted with
XChaCha20-Poly1305 (192-bit nonce, 256-bit key). The key is derived from a user-supplied backup
password using Argon2id.

**Key derivation pipeline:**

```
backup_password (UTF-8 bytes)
    │
    ▼
Argon2id(password, salt, m=16384, t=3, p=2)
    │
    ▼
256-bit encryption key
    │
    ▼
XChaCha20-Poly1305(key, random_nonce, plaintext=credential_blob)
    │
    ▼
vault.enc = nonce ‖ ciphertext ‖ tag
```

The salt is random (16 bytes), generated fresh for each backup, and stored in the manifest's
`credential_encryption.salt` field. The nonce is random (24 bytes for XChaCha20) and prepended
to the ciphertext. Argon2id parameters are also recorded in the manifest so that future versions
can adjust parameters without breaking decryption of old backups.

**Why XChaCha20 over ChaCha20?** The extended nonce (192 bits vs 96 bits) makes random nonce
generation safe without a counter. Since each backup generates a fresh nonce, there is no risk
of nonce reuse even without state tracking. This is important because the backup process should
be stateless --- it should not need to remember which nonces it has used before.

### What Goes in the Vault

The `vault.enc` blob contains a serialized map of all `Credential`-typed settings: API keys
(IGDB, ScreenScraper), download service tokens, and any connector-registered credentials. The
plaintext format is a simple JSON object keyed by `(profile_id, credential_name)`. Non-secret
settings (sort preferences, UI layout, scan paths) are stored unencrypted in `config/`.

### Cross-Machine Restore

When restoring a `.saalis` backup on a different machine, the `machine_id` in the manifest will
not match. The restore process:

1. Prompts for the backup password.
2. Decrypts `vault.enc` in memory.
3. Re-encrypts credentials using the new machine's at-rest encryption key (derived from hardware
   identifiers or a local master password, as established in the security considerations doc).
4. Writes the re-encrypted credentials to the new machine's credential store.

Credentials that are inherently machine-bound (e.g., a device-specific OAuth token) should be
flagged with a `machine_bound: true` attribute in their `Credential` descriptor. These are
**not** included in the backup vault. On restore, saalis reports which credentials were skipped
and need to be re-entered.

### Backup Without Password

If the user declines to set a backup password, credentials are **excluded entirely** from the
snapshot. The `sections` array in the manifest will not contain `"credentials"`. This is the
safe default: a `.saalis` file found on a USB stick should not leak API keys. The UI must make
this trade-off visible: "This backup does not include credentials. You will need to re-enter
API keys after restore."

## 5. Import/Export Formats: Interoperability with Other Tools

Saalis is a content library manager entering a space with established tools. Import support is
a first-run experience differentiator: users should be able to point saalis at their existing
library metadata and have it ingested without re-scraping.

### EmulationStation `gamelist.xml`

The dominant metadata format in the retro gaming space. Used by EmulationStation, ES-DE,
RetroPie, Batocera, and RecalBox. The format is documented in
[EmulationStation's GAMELISTS.md](https://github.com/Aloshi/EmulationStation/blob/master/GAMELISTS.md).

**Structure:**

```xml
<gameList>
  <game>
    <path>./Chrono Trigger (USA).sfc</path>
    <name>Chrono Trigger</name>
    <desc>A group of adventurers travel through time...</desc>
    <image>./images/Chrono Trigger-image.png</image>
    <releasedate>19950311T000000</releasedate>
    <developer>Square</developer>
    <publisher>Square</publisher>
    <genre>Role-Playing</genre>
    <players>1</players>
    <rating>0.9</rating>
  </game>
</gameList>
```

Key mapping considerations:
- `<path>` is relative to the system ROM folder or absolute. Saalis must resolve this against
  the import source directory.
- `<releasedate>` uses a non-standard datetime format (`%Y%m%dT%H%M%S%F%q`), not ISO 8601.
  The parser must handle this quirk.
- `<image>`, `<thumbnail>`, `<video>`, `<marquee>` point to media files. Saalis should import
  these into its own media store if the files exist, or record the paths for later retrieval.
- `<genre>` is a free-text string with no controlled vocabulary. Saalis should normalize against
  its own genre taxonomy during import, with fuzzy matching and a manual resolution step for
  ambiguous cases.
- Batocera extends the base schema with `<emulator>`, `<core>`, `<ratio>`, and other fields
  relevant to launch configuration. Saalis should preserve these as connector-specific metadata
  rather than discarding them.

**Export:** Saalis must be able to generate `gamelist.xml` files per system, enabling round-trip
compatibility. A user who tries saalis and decides to go back to stock Batocera should lose
nothing.

### LaunchBox XML

LaunchBox stores metadata in per-platform XML files under `Data/Platforms/<Platform>.xml`. The
format includes rich fields: `<Title>`, `<ReleaseDate>`, `<Developer>`, `<Publisher>`,
`<Genre>`, `<MaxPlayers>`, `<Rating>`, `<Overview>`, `<VideoURL>`, and a
[`<DatabaseID>`](https://wiki.retro-exo.com/index.php/LaunchBox_Platform_XML) for matching
against LaunchBox's online database.

Import strategy: match games first by `DatabaseID` (if saalis has a LaunchBox ID cross-reference
table), then by filename, then by title fuzzy match. LaunchBox's genre taxonomy is more
structured than EmulationStation's, making normalization easier.

### RetroArch `.lpl` Playlists

RetroArch playlists are JSON files (since v1.7.5) with a simple schema
([libretro docs](https://docs.libretro.com/guides/roms-playlists-thumbnails/)):

```json
{
  "version": "1.0",
  "items": [
    {
      "path": "/roms/snes/Chrono Trigger.sfc",
      "label": "Chrono Trigger",
      "core_path": "/cores/snes9x_libretro.so",
      "core_name": "Snes9x",
      "crc32": "DETECT",
      "db_name": "Nintendo - Super Nintendo Entertainment System.lpl"
    }
  ]
}
```

The `.lpl` format contains launch configuration (core path, core name) alongside library data
(label, CRC). Saalis should import `label` as the entity name, `path` as the ROM location,
`crc32` for verification matching, and `db_name` to infer the platform/system. The `core_path`
and `core_name` are launch-config metadata that map to saalis's connector settings.

### Export to Open Formats

Beyond round-trip compatibility with specific tools, saalis should support:

- **CSV export.** Flat table of all entities with selected metadata columns. Universal, opens in
  any spreadsheet. Useful for users who want to do their own analysis or share a collection list.
- **JSON export.** Full entity tree with nested metadata. Machine-readable, suitable for feeding
  into other tools or scripts.
- **SQLite dump.** The raw database file (or a `VACUUM INTO` copy). For power users who want to
  run their own SQL queries. This is the highest-fidelity export since it is literally the
  canonical data store.

## 6. Data Migration Between Versions

Schema changes across saalis versions are inevitable. The migration strategy must handle:
upgrading in place, upgrading from a backup, and detecting version mismatches on restore.

### Migration Framework

Saalis uses a linear sequence of numbered migrations, embedded in the binary via
compile-time inclusion. The `refinery` crate
([rust-db/refinery](https://github.com/rust-db/refinery)) provides this pattern for rusqlite:

```
migrations/
  V001__initial_schema.sql
  V002__add_entity_metadata.sql
  V003__credential_profiles.sql
  ...
```

On startup, refinery checks its `refinery_schema_history` table against the embedded migration
list and runs any unapplied migrations in order. Migrations are transactional --- if a migration
fails, the database is left at the previous version and saalis refuses to start (with a clear
error message and instructions to restore from backup).

### Backup-Aware Migration

When restoring a `.saalis` file whose `db_schema_version` is older than the current binary's
schema version:

1. **Extract** `db/saalis.db` to a temporary location.
2. **Run migrations** against the extracted database, advancing it from the backup's schema
   version to the current version.
3. **Validate** post-migration integrity (foreign key checks, expected table existence).
4. **Replace** the live database with the migrated copy (using the backup API for atomicity).

If the backup's schema version is *newer* than the running binary (downgrade scenario), saalis
refuses to restore and instructs the user to upgrade. Schema versions only move forward.

### Forward Compatibility Headers

The manifest's `format_version` field (distinct from `db_schema_version`) governs the archive
structure itself. A future saalis version that changes the archive layout (e.g., adding a new
top-level directory) increments `format_version`. Older versions that do not recognize the
format version can still attempt to read the archive if the ZIP structure is intact, but will
warn the user about potential incompatibility.

### Escape Hatch: SQL Dump

Every `.saalis` backup should optionally include a `db/schema.sql` file containing the full
`CREATE TABLE` / `CREATE INDEX` DDL and a `db/data.sql` file containing `INSERT` statements.
This plain-text representation survives any format evolution --- even if the `.saalis` container
format changes beyond recognition, a user can reconstruct their data from raw SQL. This is the
last line of defense for data portability.

## 7. Cloud Backup

### Optional Sync to Object Storage

Cloud backup is opt-in and never required. Saalis is an appliance-first application; it must
function fully offline. When the user configures cloud backup, saalis uploads `.saalis`
snapshots to an S3-compatible endpoint.

**Supported backends:**

| Backend | Notes |
|---------|-------|
| AWS S3 | Industry standard. Use `rust-s3` or `aws-sdk-s3` crate. |
| Backblaze B2 | S3-compatible API. [10 GB free tier](https://litestream.io/guides/backblaze/). Free uploads, pay only for storage and egress. |
| MinIO | Self-hosted S3. For users who want cloud-like backup without leaving their network. |
| Local path / NAS | Simple file copy to a mounted network share. No cloud dependency. |

### Upload Strategy

- **Scheduled full snapshots.** Weekly by default, configurable. The `.saalis` file is uploaded
  as a single object.
- **Retention policy.** Keep the last N full snapshots (default: 4, configurable). Older
  snapshots are deleted from cloud storage automatically. Users can override with "keep forever."
- **Bandwidth awareness.** Uploads run at lowest scheduler priority and respect
  bandwidth-limiting settings. On metered connections, cloud backup pauses automatically.
- **Encryption in transit.** HTTPS to the storage endpoint. The `.saalis` file's credential
  vault is already encrypted at rest, so even if the cloud provider is compromised, API keys
  are protected.

### Privacy Considerations

- **No telemetry in backups.** The `.saalis` file contains no analytics data, no usage
  statistics, no fingerprinting beyond the `machine_id` hash.
- **Credential vault is end-to-end encrypted.** The cloud provider never sees plaintext
  credentials. The backup password never leaves the user's device.
- **Game library metadata is visible** to the cloud provider (entity names, genres, etc.) unless
  the user enables full-archive encryption. Full-archive encryption wraps the entire `.saalis`
  ZIP in an additional XChaCha20-Poly1305 layer, at the cost of preventing partial restore from
  cloud storage without downloading the entire archive first.
- **No cloud lock-in.** The `.saalis` format is the same regardless of backend. Users can
  switch from S3 to B2 to local NAS by changing one configuration value. The archive format
  has zero dependency on any cloud provider's proprietary features.

## 8. Disaster Recovery

### Scenario: The SD Card Dies

This is the primary disaster scenario for Batocera appliances. SD cards have limited write
endurance, and a media center running 24/7 with periodic catalogue scans will eventually wear
one out.

**Recovery procedure:**

1. Flash a new SD card with Batocera + saalis (the standard installation image).
2. On first boot, saalis detects an empty database and enters **recovery mode**.
3. The user is prompted: "Restore from backup?" with options:
   - **USB drive.** Plug in a drive containing a `.saalis` file. Saalis scans for `.saalis`
     files and lists them by date.
   - **Cloud storage.** Enter S3/B2 credentials (or scan a QR code that encodes them). Saalis
     downloads the latest snapshot.
   - **Network share.** Enter a SMB/NFS path.
   - **Skip.** Start fresh. The old library can be re-catalogued from ROM files on disk.
4. After selecting a backup, saalis:
   - Extracts and migrates the database (section 6).
   - Restores configuration files.
   - Prompts for the backup password to decrypt credentials (or skips if no credential vault).
   - Reports what was restored and what needs manual attention (machine-bound credentials,
     paths that do not exist on the new system).

### Cold Start Behavior

If no backup is available and no previous database exists, saalis performs a clean cold start:

1. Creates a new SQLite database with the current schema.
2. Generates a default admin profile with a randomly generated PIN (displayed once on screen).
3. Scans configured ROM directories for known file extensions.
4. Begins a catalogue work unit to identify and enrich discovered entities.

The cold start path must be fast enough that a user with no backup can get a usable library
within minutes of installation. The first scan is read-only and non-destructive --- it never
modifies ROM files.

### RPO and RTO Targets

| Metric | Target | Mechanism |
|--------|--------|-----------|
| RPO (recovery point objective) | < 1 week | Weekly scheduled full backup |
| RPO with cloud | < 1 day | Daily incremental + weekly full |
| RPO with Litestream | < 1 second | Continuous WAL streaming |
| RTO (recovery time objective) | < 10 minutes | Flash SD + restore from USB |
| RTO from cloud | < 30 minutes | Depends on download speed |

### WAL + Database File Integrity

If the SD card is not dead but the filesystem is corrupt, SQLite's built-in recovery mechanisms
apply. SQLite can recover from a corrupt WAL by ignoring it (the main database file contains all
checkpointed data). The `sqlite3_recover` extension
([SQLite recovery docs](https://sqlite.org/recovery.html)) can salvage data from a corrupt main
database file by scanning for valid B-tree pages. Saalis should include a "Repair database"
option in its admin settings that runs `PRAGMA integrity_check` and, if needed, attempts
recovery.

## 9. Partial Restore

Full restore is the common case, but users sometimes need surgical precision: restore just one
system's games, just settings, or just the data from a specific connector.

### Mechanism: Section-Based Restore

The `.saalis` archive's top-level directories (`db`, `config`, `credentials`, `media`) are
independently extractable thanks to the ZIP container. The restore UI presents checkboxes:

- **Database** --- entities, metadata, history
- **Configuration** --- settings, profiles, scan paths
- **Credentials** --- API keys, service tokens (requires backup password)
- **Media** --- cover art, screenshots, thumbnails

### Fine-Grained Database Restore

Within the database section, partial restore requires application-level logic since SQLite's
backup API operates at the whole-database level. The approach:

1. Extract the backup's `saalis.db` to a temp path.
2. Attach it as a secondary database: `ATTACH ':temp_path:' AS backup`.
3. Run selective queries to merge data:
   - **By entity type:** `INSERT OR REPLACE INTO main.entities SELECT * FROM backup.entities WHERE entity_type = 'snes_game'`
   - **By connector:** `INSERT OR REPLACE INTO main.metadata_* SELECT * FROM backup.metadata_* WHERE source_connector = 'screenscraper'`
   - **By date range:** `INSERT OR REPLACE INTO main.entities SELECT * FROM backup.entities WHERE last_modified_epoch BETWEEN :start AND :end`
4. Detach the backup database.

This ATTACH-based approach is a standard SQLite pattern. It runs within a single transaction on
the live database, so it is crash-safe. Conflicts are resolved by `INSERT OR REPLACE`, meaning
the backup's version wins. For more nuanced conflict resolution (e.g., "keep whichever has more
metadata fields populated"), application logic compares rows before inserting.

### Connector-Scoped Restore

Each metadata value in saalis is tagged with its source connector (the enricher or scraper that
produced it). A connector-scoped restore replaces all metadata from a specific connector with
the backup's version while leaving other connectors' data untouched. This is useful when a
connector's API changes and produces bad data --- the user can roll back just that connector's
contributions from a known-good backup.

## 10. Data Portability Philosophy

### Users Own Their Data

Saalis stores all user data in two places: a single SQLite file and a single configuration
directory. There is no cloud-only state, no "sync required" flag, no data that exists only in
memory without a path to persistence. The user can always:

- Copy the SQLite file and have a complete, queryable backup.
- Open it in any SQLite client (`sqlite3`, DB Browser, DBeaver) and run arbitrary queries.
- Export to CSV, JSON, or `gamelist.xml` from the UI.
- Delete saalis entirely and retain full access to their data via the SQLite file.

### No Lock-In by Design

Lock-in is not just a feature to avoid; it is an architectural constraint that shapes every
design decision:

- **No proprietary binary format.** The `.saalis` container is ZIP. The database is SQLite.
  Config is TOML. Credentials are standard AEAD ciphertext. Every component uses an open,
  well-documented format.
- **No required network dependency.** Cloud backup is optional. Metadata enrichment works
  offline against cached data. The application is fully functional on an air-gapped network.
- **Round-trip import/export.** Importing from `gamelist.xml` and exporting back to
  `gamelist.xml` must be lossless for the fields that both formats support. Users who try
  saalis and leave should be in exactly the same position as before.
- **Schema is inspectable.** The SQLite schema is self-documenting via `sqlite_master`. The
  registered descriptor pattern means every metadata table has a machine-readable definition.
  A user or third-party tool can enumerate all tables, all columns, and their types without
  consulting saalis documentation.

### The Portability Test

A useful litmus test for every feature: "If saalis disappeared tomorrow, could the user recover
their library from the files on disk?" The answer must always be yes. The SQLite database is the
escape hatch. The `.saalis` archive is the convenience wrapper. The CSV/JSON exports are the
universal translators. No single mechanism is required; all are available.

### Comparison with Existing Tools

| Capability | Saalis | EmulationStation | LaunchBox | Playnite |
|------------|--------|-----------------|-----------|----------|
| Data format | SQLite (open) | XML (open) | XML (open) | SQLite (open) |
| Export formats | CSV, JSON, XML, SQL, .saalis | XML only | XML only | JSON, CSV |
| Encrypted backup | Yes (XChaCha20-Poly1305) | No | No | No |
| Incremental backup | Yes (epoch-based) | No | No | No |
| Cloud backup | Optional (S3/B2) | No | No | No |
| Partial restore | Yes (section + entity-level) | No | No | No |
| Cross-tool import | gamelist.xml, .lpl, LB XML | N/A | gamelist.xml | Various |
| Offline-first | Yes | Yes | Yes | Yes |

The goal is not to compete on lock-in avoidance (all the tools listed use open formats) but to
provide the strongest backup and recovery story in the space. For a single-board computer running
off an SD card, backup is not a nice-to-have --- it is a survival mechanism.
