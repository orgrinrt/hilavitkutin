## 1. Response to Each Expert's Findings

### 1.1 Response to SpecAuditor

The SpecAuditor produced 102 findings across all documents. From a dependency analysis perspective, the findings that matter to my domain are:

**S1 (CRITICAL): verkko-seat has no body.** I rated seat's interface sufficiency as SUFFICIENT in my Stage 1 review. The SpecAuditor's finding exposes why that rating is misleading. Seat's declared dependencies (crypto, protocol) are clean at the interface level, but the absence of a body means there is no way to verify that seat's dependencies are actually sufficient for its purpose. My interface sufficiency analysis tests "does the dependent use only defined concepts from the dependency?" It cannot test "does the dependent have enough concepts to define its own behavior?" The SpecAuditor is right that seat is a skeleton. I was right that the skeleton's connections are clean. Both statements are true and neither is complete without the other.

**P1 (CRITICAL): Noise protocol name unspecified.** This is a specification completeness gap, not a dependency graph issue. The Noise handshake lives entirely within verkko-protocol's scope and does not create any cross-document coupling. The SpecAuditor correctly identifies this as an interoperability stopper.

**X5 (MEDIUM): verkko-relay and verkko-mesh circular reference.** The SpecAuditor calls this a "circular reference at the specification level" and proposes clarifying that the dependency is unidirectional. I agree with the diagnosis but disagree with the proposed fix. My Stage 1 review identified this as edge H9 (mesh -> relay: TerritorialReencryption, severity Low). The SpecAuditor rates it MEDIUM. The truth is somewhere between: the cross-reference is informational in the sense that mesh does not import relay's internals, but it is operational in the sense that mesh's convergence gate cannot close without relay completing re-encryption. This is a runtime dependency even if it is not a definitional dependency. The fix should acknowledge the bidirectional relationship explicitly rather than pretending it is unidirectional.

**X3 (MEDIUM): Cross-document constant consistency.** The SpecAuditor verified that constants are consistent (12 clusters, 11 connections, 18-byte extension, 55-byte overhead). This is a valuable cross-check that my analysis did not perform. The proposal to add a constants table to verkko-ops is sound. From a dependency perspective, a shared constants table in ops would not create new coupling because ops is already a leaf node that depends on everything.

**C15 (LOW): DomainSeparatorRegistry is not closed.** I will address this under the savant response, where the severity disagreement is most acute.

### 1.2 Response to GapHunter

The GapHunter's 59 findings are framed as implementation stoppers. The dependency-relevant findings:

**Seat gaps (S-1 through S-10).** The GapHunter rates all of these as BLOCKING. My Stage 1 analysis found seat's interface sufficiency SUFFICIENT. The disagreement is the same as with the SpecAuditor: I was testing the wrong question. Seat's outgoing edges are clean. ## 2. What I Learned from the Savant

### 2.1 Brittleness as a first-class dependency property

My Stage 1 review measured edges in terms of correctness (declared vs implicit), cleanliness (interface-only vs body-internal), and direction (upward vs downward). The Savant introduced a fourth dimension: brittleness under evolution. The cleavage plane analysis (C1-C4) measures how many documents must change when a single concept evolves. This is a propagation analysis, not a static analysis. I should have performed it.

Concrete learning: the heartbeat format change (C1) propagates to five documents. I rated the heartbeat extension split as clean. Both are true. A clean interface can still be brittle if it is heavily consumed. Brittleness is proportional to the number of consumers times the specificity of the contract.

### 2.2 Vacancies as structural load

I cataloged hidden edges (things that are there but undeclared). The Savant cataloged vacancies (things that should be there but are not). Both are structural defects. Hidden edges create false acyclicity; vacancies create false sufficiency. My interface sufficiency rating for seat (SUFFICIENT) was a false sufficiency created by the vacancy V1.

### 2.3 Grain boundaries as a concept

The Savant's grain boundary concept formalizes what I was calling "internal leaks" in my interface sufficiency analysis. My "internal leak" description was binary (leak or no leak). The Savant's grain boundary concept is richer: it identifies the exact location where two documents share ownership of a structure (the heartbeat extension region, the convergence gate conditions, the source selection predicate). This is a better vocabulary for the same phenomenon.

### 2.4 Named fronts prevent naming cascades

This is a simple insight I should have had. I identified relay's reference to "Front 1", "Front 3" as an implicit edge (H9) with low severity. The Savant identified it as a cleavage plane (C2) with medium-high severity. The difference is that I was measuring the current state (the reference exists and is correct) while the Savant was measuring the insertion cost (adding a front between Front 2 and Front 3 cascades to three documents). Named fronts are stable under insertion. This is a graph property: named nodes have identity-stable references; numbered nodes have position-dependent references.

## 3. Where the Savant Is Wrong

### 3.1 DomainSeparatorRegistry severity is overstated

The Savant rates G4/C3 (unregistered BLAKE3 domain separator causing silent mesh partition) as the single most dangerous defect and the highest-priority repair. The SpecAuditor rates the same issue as C15 / LOW. I side with the SpecAuditor on severity, though not for the SpecAuditor's reasons.

The Savant's argument: an unregistered domain separator causes silent mesh partition, the worst failure mode. Prevention is the only defense. Therefore it is catastrophic.

The flaw in this argument: the failure mode requires a developer to add a new BLAKE3 use without registering it. This is a developer error during implementation, not a design defect. The design explicitly requires registration ("Future additions must register a unique domain separator"). The risk is that a future developer fails to follow the documented rule. This is an enforcement gap, not a design gap.

The Savant's proposed fix (S2: compile-time lint) is good practice for implementation but is not a design document concern. The design documents specify the rule. Whether the rule is enforced at compile time, review time, or test time is an implementation decision. The design is correct; the enforcement is an implementation quality concern.

Severity from a dependency graph perspective: LOW. The DomainSeparatorRegistry is centralized in a single document (crypto). All consumers reference it. The edges are clean. The risk is implementation error, not structural coupling.

**This is my specific disagreement with the Savant.** The Savant treats implementation enforcement gaps as design defects. They are not. A specification that says "you must register" is a correct specification even if it lacks a compile-time check. The compile-time check is valuable but belongs in the implementation plan (verkko-ops build order), not in the design document severity ranking.

### 3.2 The mesh-relay relationship is not a vacancy

The Savant does not explicitly call mesh -> relay a vacancy, but the framing of grain boundary G2 (Relay-Mesh convergence interface) implies that mesh is missing a bond to relay. My Stage 1 review found this as edge H9 (mesh references relay for TerritorialReencryption). The Savant's proposal S6 (ConvergenceGate defined concept) is the right fix. But the characterization matters: this is not a missing bond. It is an imprecise bond. Mesh and relay are connected; the connection is just informal (references to "Front 3" by name rather than through a defined concept). The fix is to formalize the interface, not to add a new dependency edge.

### 4.2 Revised topological sort

With all implicit and hidden edges, the graph has three soft cycles:

1. **protocol <-> mesh** via MetabolicState/Conductor. Soft: protocol defines the state machine; mesh drives it. Neither imports the other's internal implementation.

2. **crypto <-> mesh** via body back-references. Soft: crypto's defined concepts are self-standing. The body references are informational.

3. **mesh <-> relay** via Convergence/TerritorialReencryption. Soft: mesh gates on relay's completion; relay waits for mesh's signal. This is a runtime coordination cycle, not a definitional cycle.

None of these cycles prevent topological build ordering. The declared graph remains acyclic. The implementation can use the declared order:

```
Level 0: verkko-crypto
Level 1: verkko-protocol
Level 2: verkko-mesh, verkko-seat
Level 3: verkko-relay
Level 4: verkko-matter
Level 5: verkko-ops
```

If matter declares protocol as a dependency (edge 12/20), the levels do not change because matter is already at level 4 and protocol is at level 1. No topological reordering is needed.

## 5. Responses to Tension Map Questions

### Tension 1: Back-edges -- coupling defects or acceptable forward-references?

The back-edges in crypto and protocol are acceptable forward-references with one exception. The crypto -> mesh body references are informational ("for how this is used, see mesh"). The protocol -> mesh MetabolicState reference is different: it defers a definitional component (transition predicates) to a higher-level document. This is not a build-order violation, but it is a definitional incompleteness.

The heartbeat extension split is both a design strength and a grain boundary. It is not a contradiction. The current interface is correct. The Savant's HeartbeatExtension defined concept would make it resilient. I endorse the defined concept as a hardening measure, not as a defect fix.

### Tension 2: Severity scale

Three scales exist: GapHunter's implementer-halts, SpecAuditor's interoperability, Savant's emergent-risk. From a dependency analysis perspective, the relevant scale is: does this finding change the DAG structure? By this criterion:

- Undeclared dependencies (matter -> protocol): HIGH, because the DAG is incorrect.
- Missing body content (seat): MEDIUM for DAG purposes (the DAG is technically correct but vacuously so).
- Enforcement gaps (DomainSeparatorRegistry lint): LOW for DAG purposes (the DAG edges are correct; enforcement is implementation).
- Brittleness (heartbeat format change blast radius): MEDIUM for DAG purposes (the DAG is correct but fragile under evolution).

For overall project prioritization, all three scales should be applied. The GapHunter's BLOCKING rating for wire formats is correct for implementation sequencing. The SpecAuditor's CRITICAL rating for Noise protocol name is correct for interoperability. The Savant's "catastrophic" rating for domain separator enforcement is correct for failure mode analysis. No single scale is wrong; they measure different things.

### Tension 3: MetabolicState ownership

MetabolicState transition predicates should live in verkko-protocol with explicit delegation to Conductor. The state machine should be self-contained enough that a consumer can understand the states and their semantics without reading mesh. The transition triggers can be defined abstractly: "a governor provides a resource pressure signal in [0.0, 1.0]; when the signal drops below threshold_T for dwell_time_ms, the state transitions." This keeps the state machine in protocol without naming Conductor. Mesh's Conductor then implements the governor interface.

This is option (b) from my Stage 1 recommendation 6.3, and it resolves the soft cycle. Protocol defines the abstract governor interface; mesh implements it. The dependency direction remains mesh -> protocol (mesh imports the governor interface), not protocol -> mesh.

### Tension 4: Matter's undeclared protocol dependency

Both a declaration fix AND a specification fix are required. Adding verkko-protocol to matter's Dependencies is necessary but not sufficient. Matter needs to specify: (a) MatterTransfer uses protocol's bulk Stream (Queue 3), (b) flow control is via LEDBAT at the bulk tier, (c) bao-tree verification operates over protocol's reliable ordered stream.

The topological sort does not change. Matter is at level 4; adding a dependency on protocol (level 1) does not affect levels.

### Tension 5: Empty cells

Seat and matter should be treated as known design debt requiring a separate specification pass, not as out-of-scope. My interface sufficiency rating of SUFFICIENT for seat is a warning sign, not a reassurance. It means the gaps are invisible to dependency analysis because there is nothing in the body to create improper edges. An empty document has perfect interface cleanliness and zero structural value.

The cross-review can make meaningful progress on the documents that interact with seat and matter by identifying the expected interfaces. The Savant's S5 (seat-mesh mediation contract) and S4 (source selection health threshold) show that even without seat/matter body content, the interface contracts can be specified from the consuming side. This is the correct approach: define what the consuming documents need from seat and matter, then use those requirements to guide the specification pass.

## 6. Revised Recommendations (Prioritized)

1. **Add verkko-protocol to verkko-matter's Dependencies.** One-line fix with specification follow-up. Eliminates the only undeclared dependency in the declared graph.

2. **Remove "see verkko-mesh: Conductor" from MetabolicState's invariants.** Replace with an abstract governor interface. Breaks the protocol <-> mesh soft cycle.

3. **Define ConvergenceGate as a defined concept in verkko-mesh.** Formalizes the mesh-relay convergence interface. Converts informal cross-document references into a single consumable concept. Hardens the second-most fragile interface.

4. **Define HeartbeatExtension as a defined concept in verkko-protocol.** Formalizes the protocol-mesh heartbeat interface. Hardens the most trafficked bond. Reduces the heartbeat cleavage plane blast radius from 5 documents to 2 (protocol and mesh).

5. **Genericize ScarAttribution::IntegrityFailure.** Replace `blob_id` with `content_hash`. Eliminates the mesh -> matter upward reference without changing semantics.

6. **Name convergence fronts instead of numbering them.** ControlFront, ConsensusFront, ReencryptionFront. Prevents cascade on front insertion. Stabilizes relay's cross-references.

7. **Guard ops against scope creep.** The Pattern Registry back-references are navigational. They should remain navigational. If any document begins importing ops-defined conventions into its own definitions, the back-references must be migrated to the home documents.

## 7. Anti-Agreement Statement

**I disagree with the Savant's severity ranking of the DomainSeparatorRegistry enforcement gap.** The Savant places it as the #1 priority repair, above the heartbeat interface hardening (S1) and the undeclared matter-protocol dependency. From a dependency analysis perspective, the DomainSeparatorRegistry is the best-structured shared resource in the design: it is centralized in a single root document, all consumers reference it explicitly, and the registration rule is clearly stated. The risk of an unregistered separator is a developer compliance issue, not a structural coupling issue. A compile-time lint is good engineering practice and belongs in the ops build plan. Elevating it to the top priority repair over actual structural defects (undeclared edges, soft cycles, unformalized interfaces) is a severity calibration error. The Savant's failure mode analysis is correct (silent partition is catastrophic), but the probability assessment is missing (how likely is it that a developer adds BLAKE3 without reading the crypto document that defines all BLAKE3 usage?). Structural defects are certain; developer compliance failures are probabilistic. Fix the certain problems first.
