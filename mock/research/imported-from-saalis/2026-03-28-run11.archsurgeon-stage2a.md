**Date:** 2026-03-28
**Role:** Architecture Surgeon (structural specialist)

**Scope:** Structural coherence review of SpecWriter, WireEng, and Savant outputs;

---

## Method

I review each expert's Stage 1 output through one lens: does this change
preserve or violate the DAG? Every proposed addition, every wire format,
every gap finding is evaluated against the post-surgery dependency graph
from my OP-5:

```
Level 0: verkko-crypto
Level 1: verkko-protocol
Level 2: verkko-mesh, verkko-seat
Level 3: verkko-relay
Level 4: verkko-matter
Level 5: verkko-ops
```

A structural defect is an edge that violates the topological ordering, an
implicit dependency that is used but not declared, or a concept that
references something from a document it should not depend on. A content
gap is a body section that is empty but whose placeholder does not create
any undeclared edges. Structural defects must be fixed before content is
written. Content gaps can be filled independently.

### 1.1 MetabolicGovernor Contract (P0-2)

**What the SpecWriter proposes:** An abstract `MetabolicGovernor` trait in
verkko-protocol that receives `ResourcePressures` and returns a
`MetabolicState`. The concrete implementation is in verkko-mesh's Conductor.

**Structural assessment: SOUND.**

This is exactly the right structural fix. The abstract contract lives in
protocol (Level 1). The concrete governor lives in mesh (Level 2). The
dependency direction is correct: mesh depends on protocol, mesh implements
the trait, protocol calls the trait. No new edges are introduced.

However, there is one structural concern in the `ResourcePressures` struct:

```
struct ResourcePressures {
    values: [f64; N],  // N defined by implementation
}
```

The SpecWriter uses `N` as an implementation-defined constant. This is a
const-generic parameterization that protocol defines and mesh fills. But
there is a subtlety: if `N` is defined by the implementation, then protocol
must accept any `N`, which means protocol's `evaluate()` function must be
generic over `N`, which means protocol's compiled binary includes mesh-
specific code. This is not a dependency-graph problem (the trait is in
protocol, the impl is in mesh), but it is a compilation coupling concern.

**Recommendation:** Either fix `N` in the protocol document (e.g., `N=8`
with defined pressure dimensions) or make `ResourcePressures` opaque to
protocol entirely (a `&[u8]` blob that only the governor interprets). The
second option is structurally cleaner: protocol passes the blob through
without knowing its shape. The first option is easier to implement.

**Implicit dependency check:** The MetabolicGovernor contract references
"convergence_override" as a boolean input. Convergence is a mesh concept
(Level 2). Protocol (Level 1) receives this boolean as an opaque flag;
it does not need to understand what convergence is. The flag is passed into
`evaluate()` by the caller. This is acceptable: protocol defines the
interface, mesh provides the semantics of the flag. No back-reference.

**Verdict:** Structurally sound. One recommendation on `ResourcePressures`
sizing. No DAG violations.

### 1.2 HeartbeatExtension (P0-3)

**What the SpecWriter proposes:** A defined concept in verkko-mesh with an
18-byte wire format carrying filter_root, epoch_current, work_capacity,
health_canary, and observation_summary. Plus a state byte layout in the
protocol-owned transport header with mesh-relevant bits (gossip_mode,
convergence_active, admin_flag).

**Structural assessment: SOUND, with one boundary clarification needed.**

The SpecWriter adds two bits to the protocol-owned state byte that carry
mesh semantics: bit 3 (`convergence_active`) and bit 4 (`admin_flag`).
In my OP-4 surgery, I noted that `gossip_mode` (bit 2) is a boundary
object. The SpecWriter extends this pattern to bits 3 and 4.

This is structurally acceptable if and only if:
1. Protocol defines the bit positions (it owns the state byte layout).
2. Mesh defines the semantic meaning (it owns what convergence and admin mean).
3. Both documents cross-reference the same boundary object table in
   verkko-ops.

The SpecWriter's text handles this correctly: "Bits 0-1 carry the metabolic
state (protocol-layer). Bits 5-7 are reserved and MUST be set to 0." The
mesh-owned bits (2, 3, 4) are listed with their mesh semantics.

**One structural question:** The SpecWriter lists 7 consumers for
HeartbeatExtension:

1. Epoch sync (verkko-crypto: EpochRatchet)
2. Filter divergence detection (verkko-matter)
3. Donation scheduling (verkko-relay)
4. Health observation (verkko-mesh: HealthPipeline)
5. Gossip mode coordination (verkko-mesh: Gossip)
6. Convergence coordination (verkko-mesh)
7. Observation aggregation (verkko-mesh)

Consumer 1 creates an implicit edge from mesh (which writes the extension)
to crypto (which reads epoch_current). But crypto does not depend on mesh.
Crypto depends on nothing. The epoch sync works because protocol reads the
heartbeat, extracts epoch_current, and passes it to the epoch ratchet via
the protocol-level EpochRatchet interface. So the data flow is:
mesh -> heartbeat extension -> protocol (transport) -> crypto (EpochRatchet).
The dependency direction is: crypto <- protocol <- mesh. All edges point
downward. No structural defect.

Consumer 2 (verkko-matter reads filter_root) creates a data flow from
matter reading a heartbeat field that mesh wrote. But matter already
depends on mesh. No new edge.

Consumer 3 (verkko-relay reads work_capacity) creates a data flow from
relay reading a heartbeat field. Relay already depends on mesh. No new edge.

**Verdict:** Structurally sound. The consumer list does not create any
undeclared edges. All data flows follow declared dependencies.

### 1.3 ConvergenceGate (P0-5)

**What the SpecWriter proposes:** A boolean predicate gating convergence
front transitions. Four gates (ControlGate, ConsensusGate, ReencryptionGate,
ExitGate) with predicates, timeouts, and timeout behaviors. An opaque
projection interface: `fn is_open(&self) -> bool`.

**Structural assessment: SOUND, but the predicates create conceptual
coupling that needs documentation.**

The ConvergenceGate defined concept lives in verkko-mesh. Its predicates
reference concepts from multiple documents:

- ControlGate predicate: "all_reachable_keys_obtained" references key
  distribution (crypto + mesh). "all_ema_resets_applied" references the
  HealthPipeline (mesh). Sound: all within mesh or mesh's dependencies.

- ConsensusGate predicate: "ring_coloring_valid" references
  TerritorialReencryption (relay). Mesh does not depend on relay; relay
  depends on mesh. This is a structural concern.

Wait. Let me re-check. "ring_coloring_valid" is a hash ring concept from
verkko-relay's TerritorialReencryption. If mesh's ConsensusGate evaluates
ring coloring validity, mesh needs to know about the hash ring. But mesh
does not declare a dependency on relay.

Examining the actual situation: the convergence section in verkko-mesh
already discusses ring coloring in the context of convergence orchestration.
Mesh orchestrates convergence, and convergence includes a re-encryption
phase. The ring coloring constraint is evaluated by the relay layer, not
by mesh directly. So the ConsensusGate's predicate should be evaluated
by checking whether the relay layer has confirmed ring coloring, not by
mesh inspecting the ring directly.

This means the predicate "ring_coloring_valid" is not mesh reading relay
state; it is mesh receiving a boolean signal from the relay layer. The
relay layer evaluates its own ring and reports a boolean. This is
consistent with the opaque projection principle.

However, the SpecWriter's text does not make this delegation explicit.
The predicate is listed as "ring_coloring_valid" inside a mesh concept
as if mesh evaluates it. If mesh evaluates the ring coloring, mesh needs
to depend on relay, which inverts the declared dependency (relay -> mesh,
not mesh -> relay).

**Recommendation:** The ConsensusGate predicate should state that
ring_coloring_valid is provided by the relay layer as an opaque boolean,
not evaluated by mesh. The SpecWriter should add: "ring_coloring_valid:
reported by verkko-relay's TerritorialReencryption as a boolean signal.
Mesh does not inspect the hash ring."

- ReencryptionGate predicate: "all_affected_content_reencrypted_or_
  epoch_forwarded." This is a relay operation (re-encryption) reported to
  mesh as completion status. Same pattern as ring_coloring_valid. The text
  should make the delegation explicit.

- ExitGate predicate: "spanning_tree_connected" and "ring_checksum_verified."
  spanning_tree is a mesh concept (acceptable). ring_checksum is a relay
  concept. Same concern as ring_coloring_valid.

**Verdict:** Structurally sound IF the predicates that reference relay
concepts are documented as opaque boolean signals from the relay layer,
not as mesh-evaluated conditions. Without this clarification, the
ConvergenceGate creates three undeclared mesh -> relay edges (ring_coloring_
valid, all_affected_content_reencrypted, ring_checksum_verified).

This is the single most important structural finding in my review of the
SpecWriter's output. The gates are correct in concept (opaque booleans)
but the predicate descriptions imply mesh-side evaluation of relay state.

### 1.5 DomainSeparator Enum (P0-7)

**What the SpecWriter proposes:** A `repr(C)` enum in verkko-crypto with
7 variants (KeyId, ReencNonce, FoundingHash, PhaseJitter, HbJitter,
SessionId, DestinationId) and a domain-separated BLAKE3 API.

**Structural assessment: SOUND, with one observation.**

The DomainSeparator enum lives in verkko-crypto (Level 0). It enumerates
all domain-separated hash usages across the protocol. The enum itself does
not reference any other document. The documentation for each variant
mentions where it is used (e.g., "DestinationId: BLAKE3(CIK_pubkey)[0..4]")
but these are usage descriptions, not dependencies.

The lint rule ("a custom lint MUST flag any direct BLAKE3 invocation outside
the verkko-crypto crate boundary") is a build-system concern, not a DAG
concern. Sound.

**Observation:** The SpecWriter notes that relay_key_id uses HMAC-BLAKE3
with channel_salt, not the DomainSeparator API. This means relay_key_id
is explicitly excluded from the registry. This is an intentional design
decision (HMAC-BLAKE3 is a different construction from keyed BLAKE3), but
it means the DomainSeparator enum is NOT the complete registry of all
BLAKE3 usage in the system. The SpecWriter acknowledges this. The structural
implication is that relay has a BLAKE3 usage that bypasses the domain
separation API. This is acceptable because relay depends on crypto, and
crypto can provide both the domain-separated API and the raw HMAC
construction. But the DomainSeparator invariant ("all BLAKE3 usage MUST
go through a function that takes a DomainSeparator variant") is technically
violated by relay_key_id derivation.

**Recommendation:** Either add a DomainSeparator variant for relay_key_id
(renaming the API to accommodate HMAC-BLAKE3 alongside keyed BLAKE3), or
soften the invariant to "all keyed BLAKE3 usage" (excluding HMAC-BLAKE3).
The current text contradicts itself: it says "all BLAKE3 usage MUST go
through DomainSeparator" but then says relay_key_id is intentionally
excluded.

**Verdict:** Structurally sound at the DAG level. One invariant
self-contradiction on completeness of the registry.

belong in verkko-seat (seat must not depend on mesh).

### 2.3 Connection ID Generation (BLAKE3-derived)

**What the WireEng proposes:** Connection ID derived via BLAKE3 keyed hash
from CIK public keys, handshake nonce, and session counter. Domain
separator: `"saalis-connid-v1"`.

**Structural assessment: INTRODUCES A COUPLING.**

The WireEng's connection ID derivation uses CIK public keys as input.
CIK is a mesh concept (Cluster Identity Key, defined in verkko-mesh).
verkko-protocol does not declare a dependency on verkko-mesh. If protocol
derives connection IDs from CIK, protocol needs to know about CIK, which
creates an undeclared protocol -> mesh edge.

Wait. Let me re-examine. Protocol depends on crypto. CIK is an X25519 key.
From protocol's perspective, the "local static key" and "remote static key"
used in the Noise handshake are just X25519 keys. Protocol does not need
to know they are CIKs. Protocol uses whatever static keys the caller
provides.

So the question is: does the connection ID derivation formula need to name
`CIK_pub` specifically, or can it use "local_static_key" and "remote_
static_key" (which happen to be CIKs when used in mesh context)?

The WireEng names them `local_CIK_pub` and `remote_CIK_pub`, which creates
a naming dependency on the mesh concept. If the formula instead uses
`local_static_pub` and `remote_static_pub`, the derivation is structurally
clean: protocol uses whatever static keys the Noise handshake uses, and
mesh provides CIKs as those keys.

**Structural finding:** The WireEng's connection ID derivation creates a
naming dependency from protocol to mesh's CIK concept. The fix is trivial:
rename the inputs to `local_static_pub` and `remote_static_pub` in the
protocol specification. The mesh document can note that these are CIKs.

**Second structural concern:** The domain separator `"saalis-connid-v1"` is
added to the DomainSeparatorRegistry in verkko-crypto. The WireEng lists
this in Appendix B. But the SpecWriter's DomainSeparator enum (P0-7) does
not include a `ConnId` variant. Either the WireEng's domain separator needs
to be added to the SpecWriter's enum, or the connection ID derivation should
not use domain-separated BLAKE3.

This is a content inconsistency between the SpecWriter and WireEng, not a
structural defect. But it highlights a coordination gap: both experts
independently defined the DomainSeparator concept (SpecWriter as a repr(C)
enum, WireEng as a registry addition) without cross-referencing each other's
variant list.

**Recommendation:** Add `ConnId = 7` to the SpecWriter's DomainSeparator
enum. Or, if the SpecWriter's random-u32 approach is preferred, remove the
domain separator addition entirely.

**Verdict:** One naming dependency (CIK_pub in protocol). Trivial fix.
One content inconsistency with SpecWriter's DomainSeparator enum.

### 2.9 Data Frame Format (Consolidated)

**Structural assessment: SOUND.**

The WireEng consolidates the existing data frame format from verkko-protocol
section 3.1 into byte-level diagrams. The outer header (29 bytes) and
encrypted payload format are protocol-internal. The msg_type registry
(section 9.3) lists 12 message types. These reference concepts from
multiple documents (heartbeat, gossip, control, chunk, SACK, relay_op,
filter_update, schema_sync, chain_proof) but protocol treats all payloads
as opaque except for heartbeat and SACK (which protocol processes).

The encrypted payload includes channel_index (mesh concept), epoch (crypto
concept), micro_epoch (crypto concept), and dest_peer (mesh concept). These
fields are inside the encrypted region, meaning protocol layer processes
them after decryption. Protocol depends on crypto (for keys and epochs).
The channel_index and dest_peer fields mean protocol must know about
channels and peer IDs, which are mesh concepts.

**Structural finding:** The encrypted payload format in protocol includes
`channel_index` (u16) and `dest_peer` (u16). These are mesh-layer concepts
(channels are defined in mesh, peer IDs are mesh identifiers). If protocol
needs to interpret these fields (not just pass them through), protocol
has an implicit dependency on mesh.

However, examining the design: protocol does not interpret these fields.
Protocol decrypts the frame and passes the decrypted payload (including
channel_index, epoch, dest_peer, and data) to the consumer. The consumer
(mesh layer) interprets channel_index and dest_peer. Protocol only uses
epoch and micro_epoch for key derivation (crypto dependency, declared).

So the channel_index and dest_peer fields are in the wire format owned by
protocol, but their interpretation is deferred to the consumer. This is
the same pattern as the HeartbeatExtension: protocol transports, mesh
interprets. Sound.

**Verdict:** Structurally sound. The channel_index and dest_peer fields
in the encrypted payload are transported by protocol, interpreted by mesh.
No undeclared dependency.

## Section 5: Tension Analysis

### T1: Version Negotiation Timing

**SpecWriter position:** Inside the Noise handshake encrypted payload.
Both peers exchange version ranges post-handshake.

**WireEng position:** Before the Noise handshake as a dedicated UDP
exchange. VersionProbe/VersionResponse with nonce echoing.

**Structural analysis:**

From a dependency-graph perspective, both placements are equivalent: version
negotiation is entirely within verkko-protocol. Neither creates new edges.

From a protocol-design perspective, the structural question is: does the
version negotiation mechanism need to reference any concepts from other
documents?

The WireEng's pre-handshake approach introduces a new frame type (probe_type
0x01/0x02) and a new magic number usage. These are all protocol-internal.
The SpecWriter's post-handshake approach adds 8 bytes to the encrypted
payload. Also protocol-internal.

**My position: WireEng's pre-handshake approach is structurally preferable.**

Reason: the Noise handshake is defined in terms of the Noise protocol name
string (e.g., `Noise_KK_25519_ChaChaPoly_BLAKE2s`). The protocol name
string is hashed into the handshake state. If two peers disagree on the
protocol version and this affects the Noise protocol name (e.g., a future
version uses a different cipher), they will produce non-interoperable
handshake states. The SpecWriter's approach would complete a handshake
(potentially expensive, 2-3 round trips for XX) before discovering the
incompatibility.

The WireEng's approach fails fast: no common version -> no handshake ->
no wasted computation. This is structurally cleaner because it avoids
creating ephemeral Noise state that will be immediately discarded.

**Caveat:** The WireEng's pre-handshake probe is unauthenticated. An
attacker can forge VersionResponse datagrams with `selected_version = 0`
to deny connections. This is a security concern, not a structural concern.
The nonce echoing provides weak replay protection (off-path attacker
cannot forge, on-path attacker can).

**Verdict:** WireEng's pre-handshake approach. Structurally cleaner
dependency graph: no wasted ephemeral state. The unauthenticated nature
is a known trade-off, acceptable for version negotiation (the actual
session is authenticated by Noise).

---

### T2: BLAKE2b vs BLAKE2s in Noise Protocol Name

**SpecWriter position:** BLAKE2b (64-bit variant).
**WireEng position:** BLAKE2s (32-bit variant).

**Structural analysis:**

This is not a structural concern. The Noise protocol name is a protocol-
internal constant that affects only the handshake state initialization.
It does not affect the dependency graph, the defined concepts, or the
document topology.

Both variants produce equally sound DAGs. The choice affects performance
(BLAKE2s is ~20% faster on 32-bit platforms, BLAKE2b is ~40% faster on
64-bit platforms), not structure.

**My position: This is a content decision, not a structural decision.**

However, for completeness: verkko targets Batocera appliances. Batocera
runs on x86_64 and ARM64 (64-bit). BLAKE2b would be the natural choice
for 64-bit targets. BLAKE2s is the Noise specification's recommendation
for the `BLAKE2s` hash function in the Noise protocol framework. The Noise
spec defines `BLAKE2s` and `BLAKE2b` as separate hash functions with
separate protocol names.

I defer to whichever expert has the stronger implementation rationale. The
DAG does not care.

**Verdict:** Content decision. No structural implications. Both experts
must agree on exactly one value before any wire format can be implemented.

### T3: Connection ID -- Random vs. BLAKE3-Derived

**SpecWriter position:** `connection_id = random_u32()`.
**WireEng position:** `connection_id = BLAKE3("saalis-connid-v1" || CIK_pub || ...) [0..4]`.

**Structural analysis:**

The SpecWriter's random approach creates no dependencies. A random u32
requires only a CSPRNG, which protocol already uses.

The WireEng's BLAKE3 approach creates:
1. A dependency on BLAKE3 (via crypto's domain-separated API). Protocol
   already depends on crypto. No new edge.
2. A naming dependency on CIK (mesh concept). As discussed in Section 2.3,
   this is a naming issue, not a DAG edge. Fix by using `local_static_pub`.
3. A dependency on the DomainSeparatorRegistry (to add "saalis-connid-v1").
   Protocol depends on crypto, which owns the registry. No new edge.

**My position: SpecWriter's random approach is structurally simpler.**

The WireEng's BLAKE3 derivation binds the connection ID to session key
material. This provides unpredictability (off-path observers cannot predict
the next ID) and migration safety (ID is bound to the session, not the
address). But a random u32 provides the same unpredictability (CSPRNG output
is by definition unpredictable) and the same migration safety (the ID is
stable across address changes regardless of how it was generated).

The BLAKE3 derivation adds complexity (domain separator registration, key
material inputs) without a clear structural benefit. The collision analysis
is identical (birthday bound at ~65K for both approaches).

The one advantage of the BLAKE3 approach is deterministic regeneration:
if a peer crashes, it can regenerate the same connection ID from persisted
key material. But this is a resilience feature, not a structural concern.

**Verdict:** SpecWriter's random approach is structurally preferable. Fewer
moving parts, no domain separator registration needed, no naming dependency
on CIK. If the WireEng's approach is chosen for resilience reasons, the
naming dependency on CIK must be resolved (use `local_static_pub`).

### T5: HostPeer -- Structural Dependency Violation?

**Savant position:** HostPeer creates a "phantom border" where seat depends
on mesh through the host peer, dissolving the seat's independence.

**My position: HostPeer is NOT a structural dependency violation.**

The DAG says:
- verkko-seat depends on: verkko-crypto, verkko-protocol.
- verkko-seat does NOT depend on: verkko-mesh.

The HostPeer is a runtime composition point. At runtime, a mesh peer
instantiates both mesh behavior and seat-serving behavior. The seat sends
requests via the protocol transport layer. The host peer receives them,
translates them into mesh operations, and sends responses via the protocol
transport layer. The data flow crosses the seat/mesh boundary, but the
document-level dependency does not.

This is the same pattern as a web server that uses both an HTTP library
and a database library. The web server depends on both, but the HTTP
library does not depend on the database library, even though HTTP requests
cause database queries at runtime.

The Savant's concern is valid at the implementation level: the HostPeer
translation layer will be the place where seat and mesh concepts interact,
and implementers may create coupling. But the structural defense is clear:

1. seat defines SeatRequest, SeatResponse, SeatEvent types.
2. mesh defines how to serve those requests using mesh state.
3. The host peer's translation behavior belongs in mesh (or ops), not seat.
4. seat never imports mesh concepts. seat never depends on mesh.

If an implementer puts the translation in the seat crate, that is a lint
violation (seat importing mesh types). The structural boundary is enforced
at compile time.

**Where should the HostPeer translation be specified?** In verkko-mesh.
Mesh defines a section titled "HostPeer behavior" that describes how a mesh
peer translates SeatRequests into mesh operations. This is a mesh-side
behavior because the host peer IS a mesh peer. The seat document describes
the client side of the protocol; the mesh document describes the server
side.

Alternatively, the translation could go in verkko-ops (the integration
document). But ops depends on both seat and mesh, so it can reference
both. This placement is also structurally sound.

**Verdict:** No structural dependency violation. HostPeer translation
belongs in verkko-mesh (or verkko-ops). The DAG does not need a seat ->
mesh edge.
