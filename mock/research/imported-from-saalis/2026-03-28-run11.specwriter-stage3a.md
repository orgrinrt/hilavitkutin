## Part 1: The Access-Structural Gap Classification

### 1.1 Do I Accept the Classification?

The Savant proposes a two-dimensional classification: DAG-structural (does it require a new dependency edge?) and access-structural (can the system be entered and used?). Four gaps are classified as access-structural: B1 (SansIO), B2 (intra-cluster), B5 (matter chunking), B12 (connection teardown).

**I accept the classification with one qualification.**

The Savant's argument is precise: the ArchSurgeon's single-dimension test (does this require a new DAG edge?) correctly identifies topological health but cannot distinguish between gaps that block all implementers and gaps that block no one until deployment. The door metaphor is apt: a building with correct floor plans but no doors is structurally deficient in a way that floor plan analysis cannot reveal.

My Stage 2a position was that the distinction between "design document completion" and "implementation readiness" justified deferring B1. The Savant correctly identifies that this distinction is false for a sans-IO protocol. I quote from my own Stage 2a: "The design documents describe what the protocol does, not the API surface for consuming it." The Savant rebuts this by pointing out that for a sans-IO protocol, the API surface IS what the protocol does. The protocol is defined by its input/output contract, not by its internal mechanisms.

I was wrong on this specific point. A sans-IO protocol's consumer interface is not an implementation API; it is the protocol specification's external contract. Specifying the internal mechanisms (wire formats, state machines, heartbeat schedules) without the external contract is specifying a CPU's microarchitecture without its instruction set architecture. The Savant's rebuttal is technically correct.

**The qualification:** The access-structural classification is useful for triage but should not be treated as a binary. B1 and B5 are access-structural in different ways. B1 blocks all consumers of the protocol (nobody can drive the state machine). B5 blocks one subsystem's purpose (matter cannot be transferred). B12 blocks resource cleanup (connections leak). B2 blocks one communication domain (intra-cluster). The severity gradient matters: B1 > B5 > B12 > B2 in terms of how many implementers are blocked and how severely.

The Savant's priority ordering (B1 > B5 > B2 > B12) differs from mine in that the Savant ranks B2 above B12. I disagree. B12 is a smaller, self-contained specification task that can be completed in Run 11 without design decisions. B2 requires design decisions about session models and key usage that the review process is not equipped to make. Priority should weight feasibility alongside criticality.

**Updated priority ordering:** B1 > B5 > B12 > B2. This matches the Savant's recommendation in section 1.4, where the Savant also concludes B2 "requires a design round."

## Part 3: The Six Emergent Interactions

The Savant identified six cross-cutting issues in Part 6 of Stage 2b. I address each.

### 3.1 The UDP Payload Size Bug (6.1)

**I raised this bug in Stage 2a. The Savant amplified it correctly.**

The ripple effects the Savant identifies are accurate:
- Data frame max payload: 1177 bytes (not 1225).
- Matter chunk payload: 1119 bytes (not 1167).
- SACK with piggyback: 1147 bytes (not 1195).
- Heartbeat + extension: unaffected at 36 bytes.

The Savant correctly notes that the 48-byte correction shifts the chunk size tradeoff. At corrected figures, 512-byte bao chunks are more attractive because they fit in a single frame with room for the proof. I recommend 512-byte bao chunks as the default, with the spec permitting 1024-byte chunks for transfers where the implementer accepts 2-frame-per-chunk overhead.

**Action required:** This correction MUST be applied before any frame size calculation is finalized. Every expert's frame budget is wrong by 48 bytes. I will use 1232 bytes as the UDP payload limit in all normative text going forward.

**This affects my normative text.** My Stage 1 handshake message sizes used 1280 implicitly. They are within budget at 1232 (handshake messages are well under 200 bytes), so no structural change is needed, but the frame budget tables must be corrected.

### 3.2 The ConvergenceGate Dependency Inversion (6.2)

**The Savant correctly traces a data flow issue that the ArchSurgeon and I both missed.**

The Savant's analysis: relay evaluates ring coloring, produces a boolean signal, that signal must reach mesh, mesh updates ConvergenceGate, gate state affects the convergence_active heartbeat bit. This is a runtime data flow from Level 3 (relay) to Level 2 (mesh), which goes against the dependency direction.

The Savant identifies three possible delivery mechanisms:
1. Relay sets a field in its heartbeat extension. (Incorrect: relay does not own heartbeat extension fields.)
2. Relay sends a dedicated control message. (Requires a new msg_type.)
3. The host peer's runtime reads relay state directly via same-process function call. (Implementation detail, not protocol specification.)

**I agree with the Savant that Option 3 is the likely runtime behavior and that the mechanism should be documented as same-process access, not wire protocol.**

The ConvergenceGate predicates (ring_coloring_valid, all_affected_content_reencrypted, ring_checksum_verified) are evaluated within a single process that runs both mesh and relay layers. The predicate evaluation is a function call, not a network message. This is consistent with the ArchSurgeon's principle: "crypto describes what it provides; mesh describes what it consumes." By extension: relay describes what it provides (ring coloring status as a queryable boolean); mesh describes what it consumes (an opaque boolean signal obtained via same-process query).

**Change to my normative text:** The ConvergenceGate specification MUST state: "The ConsensusGate's ring_coloring_valid predicate is provided by the relay layer as an opaque boolean signal. In a single-process deployment, this is a direct function call to the relay layer's ring coloring evaluator. The mesh layer does not evaluate relay state; it queries a boolean result. This query is not a wire protocol operation."

This is a documentation clarification, not a protocol change. But it eliminates the ambiguity that the Savant and the ArchSurgeon both identified.

### 3.3 The SansIO and Seat Transport Interaction (6.3)

**The Savant correctly connects B1 (SansIO) with W5 (seat transport).**

If the seat uses WebSocket as transport, the SansIO consumer interface receives bytes from WebSocket binary messages, not UDP datagrams. The protocol machine must be transport-agnostic.

The Savant's proposed normative statement is exactly what I would write: "The source of bytes is opaque. The consumer may deliver bytes from UDP datagrams, WebSocket messages, TCP streams, or any other reliable or unreliable transport. The protocol machine does not know or care about the transport."

**I accept this and will include it in the B1 (partial) event surface specification.** The ProtocolInput::BytesReceived event SHALL specify that the byte source is opaque. The protocol machine SHALL NOT make assumptions about transport framing (e.g., it SHALL NOT assume that each BytesReceived event corresponds to exactly one UDP datagram; WebSocket messages may carry multiple frames or partial frames).

**This affects my normative text.** The event surface specification must include explicit transport-agnosticism language. This is a small addition (3-4 sentences) to the B1 partial spec.

### 3.4 Chunk Protocol and Connection Teardown Interaction (6.4)

**The Savant correctly identifies that B5 and B12 interact during transfer abort.**

The Savant proposes: "When a connection is closed, all in-progress transfers on that connection are implicitly aborted. The remote peer SHOULD NOT send TransferAbort for transfers on a connection that has received a Connection Close."

**I accept this normative statement with one addition.** The CONNECTION_CLOSE message MUST document its interaction with in-progress transfers:

```
When a peer sends or receives CONNECTION_CLOSE:
  1. All in-progress transfers on that connection are implicitly aborted.
  2. The peer SHOULD NOT send TransferAbort for transfers on a connection
     that has sent or received CONNECTION_CLOSE.
  3. The peer SHOULD send TransferAbort for transfers it wishes to cancel
     on a connection that remains open.
  4. A peer receiving CONNECTION_CLOSE while a transfer is in progress
     MAY immediately free all transfer state for that connection without
     waiting for explicit TransferAbort.
```

This is four sentences of normative text that eliminate the ambiguity. I will include it in the B12 (connection teardown) spec.

### 3.6 f64 NaN Propagation Is Broader Than Initially Identified (6.6)

**I agree with the Savant's analysis. The NaN guard must be cross-cutting.**

The Savant identifies three NaN propagation paths beyond my Stage 2a analysis:
1. EMA coefficients: a single NaN observation corrupts all subsequent EMA updates permanently.
2. Weighted scar count: a malformed ScarSeverityTier lookup producing NaN corrupts peer health.
3. Dominance computation: NaN self-dominance score breaks the cascade (NaN comparisons are always false in IEEE 754).

My Stage 1 MetabolicGovernor spec guarded only the all-NaN case. The Savant's recommendation to change the guard from `all(is_nan)` to `any(is_nan)` is correct but insufficient. The fix must be cross-cutting, as the Savant proposes.

**Change to my normative text:** I will add the following cross-cutting invariant to my health pipeline spec:

```
NaN Safety Invariant:

No floating-point computation in the health pipeline, scar system,
dominance cascade, or metabolic governor MAY produce or propagate NaN.

If any input to a floating-point computation is NaN, the computation
MUST substitute the previous valid result for the NaN input and
continue with the substituted value. If no previous valid result
exists (e.g., at startup), the computation MUST use the type's
default value (0.0 for scores, 1.0 for multiplicative weights).

Specific guards:
- MetabolicGovernor::evaluate(): If any(ResourcePressures.values.is_nan()),
  retain current MetabolicState. Do not transition.
- EMA update: If the new observation is NaN, skip the update.
  Retain the previous EMA value.
- ScarSeverityTier weight lookup: If the weight is NaN, use 1.0
  (neutral weight).
- Dominance score comparison: If either score is NaN, the non-NaN
  score wins. If both are NaN, fall back to peer_id ordering.
```

This is 15 lines of normative text. It is a meaningful addition that prevents an entire class of silent corruption bugs. The Savant's finding here is the most practically important of the six emergent interactions.

## Part 4: W5 -- Seat Transport

### 4.1 The Problem

The Devil's Advocate found this in Stage 2a (W5). The Savant amplified it in Stage 2b and recommended specifying WebSocket encapsulation. Both are right. All four Stage 1 experts assumed UDP transport for everything, but the seat is a web client. Web browsers cannot send raw UDP.

The design documents are clear: verkko-seat depends on verkko-protocol (Connection, Session, Frame, Stream, SansIO). This implies the seat uses verkko protocol frames, not a separate REST/SSE protocol. But the transport binding for those frames between a web browser and the host peer's axum server is unspecified.

### 4.2 My Position

**The seat body spec needs a transport binding section, not a revision of the frame format.**

The wire formats I specified (SeatAuth, SeatRequest, SeatResponse, SeatEvent, SeatInstruction) are transport-agnostic payloads. They describe typed messages with correlation IDs and protobuf-encoded payloads. They do not assume UDP. They are equally valid inside WebSocket binary messages.

What is missing is the specification of how these messages travel between the seat and the host peer. The Savant's recommendation is correct: verkko protocol frames encapsulated in WebSocket binary messages.

**Proposed SeatTransport defined concept for verkko-seat:**

```
### SeatTransport

The transport binding between a seat and its host peer.

Invariants:
- The seat connects to the host peer via a WebSocket connection
  to the host peer's HTTP server.
- The WebSocket subprotocol is "verkko-seat-v1".
- Each WebSocket binary message carries exactly one verkko protocol
  frame (the outer header + encrypted payload).
- The Noise handshake (KK or XX) is performed inside the WebSocket
  tunnel. The first WebSocket messages carry Noise handshake frames.
  Subsequent messages carry post-handshake data frames.
- WebSocket text messages are not used. All communication is binary.
- WebSocket close frames follow RFC 6455. A WebSocket close
  is equivalent to CONNECTION_CLOSE with reason NORMAL.
- The WebSocket connection provides reliable, ordered delivery.
  The verkko protocol's SACK and retransmission mechanisms are
  NOT used for seat connections (WebSocket/TCP provides reliability).
  Stream-level sequence numbers are still used for ordering and
  deduplication but lost-frame retransmission is unnecessary.

The SansIO consumer interface handles WebSocket-delivered bytes
identically to UDP-delivered bytes. The protocol machine does not
distinguish between transport types.
```

**This is a significant finding that requires a new defined concept in verkko-seat.** It is not a revision of the existing frame formats but an addition specifying how they are carried. I recommend including this in Run 11 as part of the seat specification work. The text above is approximately 20 lines; the implementation impact is bounded.

### 4.3 One Complication: Reliability Semantics

The Savant's recommendation (WebSocket encapsulation of verkko frames) creates a semantic mismatch. The verkko protocol is designed for unreliable transport (UDP): it has SACK, retransmission, and congestion control. WebSocket runs over TCP, which provides reliable, ordered delivery. Running the verkko protocol's reliability layer over TCP's reliability layer is redundant.

Two approaches:

**A. Tunnel mode (simple).** The seat connection uses the full verkko protocol stack, including SACK and retransmission, over WebSocket/TCP. The redundancy is harmless: the SACK will always show all frames as delivered, the retransmission timer will never fire, and LEDBAT will operate on TCP's delivery timing. The only cost is 14 bytes of SACK overhead per acknowledgment cycle, which is negligible on a LAN.

**B. Slim mode (optimized).** The seat connection uses a reduced verkko protocol stack that omits SACK, retransmission, and LEDBAT. Frames are sent once and assumed delivered (because TCP guarantees delivery). This saves complexity but requires a different protocol mode that the SansIO interface must support.

**I recommend Approach A (tunnel mode) for v1.** The redundancy is harmless, and it avoids bifurcating the protocol into two modes (reliable and unreliable). The SansIO consumer interface remains identical for both UDP and WebSocket consumers. A future optimization (slim mode) can be specified in a later version if profiling shows the redundancy is measurable.

The normative statement: "When the seat transport is WebSocket (reliable, ordered), the verkko protocol's reliability mechanisms (SACK, retransmission) operate but have no practical effect. The implementation MAY skip retransmission timers for sessions known to be on reliable transports, but this is an optimization, not a protocol change."

## Part 5: Phantom Border PB1 -- HostPeer

### 5.1 The Savant's Critique: Moved, Not Resolved

The Savant writes: "Renaming. The phantom border was: 'The bridge between seat and mesh exists on the map but neither territory claims it.' The consensus says: 'The bridge belongs to a different country (the application layer).' This moves the bridge to a different map (the saalis map, not the verkko map)."

The Savant argues that moving the phantom to the application layer does not resolve it; it renames it. The phantom still exists from the application's perspective.

**I partially agree and partially disagree.**

### 5.2 Where I Agree

The Savant is correct that from the application's perspective, the HostPeer translation layer is still unspecified. Moving it from "not in verkko-seat, not in verkko-mesh" to "in the application layer" does not produce a specification. It produces a labeled gap: "someone else's problem." If nobody writes the application-layer spec, the gap persists.

The Devil's Advocate's conformance testing argument is also valid: without the translation specification, two implementations can have identical seat-side code and completely different HostPeer behavior. The verkko spec cannot be conformance-tested end-to-end without specifying the translation.

### 5.3 Where I Disagree

The phantom is correctly resolved at the verkko level. The verkko spec is a protocol specification, not an application specification. The seven documents define wire formats, state machines, and cryptographic operations. They do not define application logic. The HostPeer translation is application logic: it maps application-specific requests ("list my games") to protocol-specific operations (cuckoo filter query + metadata lookup + channel access check).

The Savant's updated position in Stage 2b (section 3.1) acknowledges this: "I accept that the full translation spec is outside verkko's scope. I insist that the capability enumeration belongs in the verkko spec." This is the correct boundary. The verkko spec provides:

1. A normative note stating that HostPeer translation is an application responsibility.
2. A capability enumeration listing mesh operations the HostPeer MUST support.

The application spec (saalis-core or a verkko-saalis adapter) provides:

3. The actual translation logic.
4. The orchestration of mesh operations in response to specific SeatRequest types.

**I maintain my Stage 2a position but strengthen it.** The normative note and capability enumeration belong in verkko-seat (as I proposed). They are the correct boundary markers. The Savant's concern about conformance testing is valid but is an application-level concern, not a protocol-level concern. Verkko conformance testing tests the wire protocol (does the seat send correctly formed SeatRequest frames? does the host peer respond with correctly formed SeatResponse frames?). Application conformance testing tests the translation (does the HostPeer correctly translate a QUERY request into a cuckoo filter lookup with channel access control?). These are different testing scopes.

### 5.4 Concrete Addition

I will add the following to my verkko-seat spec:

```
### HostPeer Capability Enumeration

The host peer MUST be able to perform the following mesh operations
on behalf of connected seats:

1. Filter query: Query the local cuckoo filter for content matching
   seat-provided criteria.
2. Channel access check: Determine whether the seat's user has access
   to a given channel based on the user's role and the channel's
   access control list.
3. Matter transfer initiation: Initiate a MatterTransfer from a
   source peer on behalf of the seat.
4. Control log query: Read control log entries visible to the seat's
   user role.
5. Health pipeline query: Provide mesh health status appropriate to
   the seat's user role.
6. Configuration update: Apply settings changes submitted by an
   admin-role seat.

The orchestration of these operations (the order in which they are
invoked, the error handling between steps, the caching strategy)
is the responsibility of the application layer. This document does
not specify orchestration because it is application-specific.

A conformance test for the HostPeer capability enumeration verifies
that each operation is available and produces correct results in
isolation. End-to-end orchestration testing is an application-level
concern.
```

This is 22 lines of normative text. It tells the application implementer what they must build without telling them how to build it. The Savant should find this acceptable; the Devil's Advocate may not, because the orchestration remains unspecified. But orchestration is application logic, and I will not specify application logic in a protocol document.

### 9.2 "A sans-IO protocol IS its consumer interface"

**I concede this point.** The Savant is right. I was drawing a distinction between "the protocol" and "the API to the protocol" that does not exist in sans-IO design. In sans-IO, the protocol is the state machine, and the state machine is defined by its input/output types. The input/output types ARE the specification. My wire formats are the internal representation; the event surface is the external contract. Both are part of the protocol specification.

This does not mean the entire consumer interface (polling model, buffer ownership, error semantics) must be specified in Run 11. But the event enumeration (what goes in, what comes out) is as much a part of the protocol specification as the wire formats.

The wire-format-facing event surface (mapping wire format types to ProtocolOutput variants, mapping consumer actions to ProtocolInput events) is the partial B1 for Run 11. The implementation-facing details (polling conventions, buffer allocation, arena lifetime, error propagation) are Run 12.

### 9.4 "IntraClusterTransport is not a new mechanism"

The Savant writes (Stage 2b, section 2.2): "IntraClusterTransport is not a new mechanism, it is a specification of how existing mechanisms are used within a cluster. The territory has roads; it needs signposts."

## Part 12: Remaining Disagreements

### 12.1 B2 Priority

The Savant ranks B2 (intra-cluster) above B12 (connection teardown) in priority. I rank B12 above B2. The Savant's ordering is B1 > B5 > B2 > B12. Mine is B1 > B5 > B12 > B2.

Both orderings agree on B1 and B5 at the top. The disagreement is whether intra-cluster communication (B2) or connection teardown (B12) is more urgent. The Savant argues intra-cluster is more critical because "different implementations will make different choices and interop will break." I argue connection teardown is more urgent because it is a bounded specification task (~40 lines) that can be completed in Run 11, while intra-cluster requires design decisions that the review process cannot make.

This disagreement is about feasibility vs. criticality. Both are valid dimensions. For Run 11, feasibility matters because we are in a specification run, not a design round. B12 can be specified; B2 must be designed first. The Savant agrees (Stage 2b, section 1.4): "B2 (intra-cluster communication) is important but requires design decisions that the review process is not equipped to make." We are in agreement on the action (defer B2); we disagree only on the priority ranking, which is academic since both are deferred.

### 12.2 HostPeer Translation Scope

The Devil's Advocate argues the HostPeer translation belongs in verkko-mesh. The Savant argues it should be partially in verkko (capability enumeration) and partially in the application layer (orchestration). I argue it is entirely an application-layer concern, with only the capability enumeration and normative note in verkko-seat.

This is a three-way disagreement about where to draw the line. The Advocate wants maximum specification (translation logic in verkko-mesh). The Savant wants moderate specification (capability enumeration in verkko, orchestration in application). I want minimum specification at the verkko level (normative note + enumeration only).

I maintain my position. The HostPeer translation is application-specific. Specifying it in verkko-mesh would couple the protocol specification to saalis-specific concepts (entities, hot store, metadata schema). A different application using verkko would have a completely different translation. The verkko spec should not know about games, movies, or content metadata.

### 12.3 Seat Transport Reliability Mode

The Savant recommends specifying WebSocket encapsulation but does not address the reliability semantic mismatch (verkko's SACK/retransmission running over TCP's reliability). I recommend tunnel mode (full protocol stack, redundant reliability) for v1. This is not a disagreement with the Savant; the Savant did not take a position. But a future expert may argue for slim mode (omitting redundant reliability). I preemptively argue against slim mode for v1: it bifurcates the protocol, complicates the SansIO interface, and saves negligible overhead on a LAN.
