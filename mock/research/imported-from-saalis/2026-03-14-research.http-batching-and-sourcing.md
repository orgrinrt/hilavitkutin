# HTTP API Batching and External Sourcing Pipeline Optimisation

**Date:** 2026-03-14
**Context:** Saalis content library manager -- connector layer that fetches metadata from external APIs (Steam, IGDB, ScreenScraper, TheGamesDB, MobyGames, RAWG, GiantBomb, and future sources for movies/music/books).

---

## 1. HTTP Request Batching Patterns

### 1.1 Connection Pooling

Connection pooling reuses established TCP connections across multiple HTTP requests, avoiding the overhead of TCP handshake and TLS negotiation for each request. For a connector layer making hundreds or thousands of requests to the same host, this is the single largest performance win available at the transport level.

**How it works:** A pool of persistent TCP connections is maintained per host. When a request needs to be sent, an idle connection is pulled from the pool rather than opening a new one. After the response is received, the connection is returned to the pool for reuse. Keep-alive headers signal to the server that the connection should remain open.

**Best practices:**
- Create one HTTP client per connector (or per host) and reuse it for all requests
- Configure idle connection timeouts to match server expectations (most servers close idle connections after 60-120 seconds)
- Set a maximum pool size per host to avoid exhausting server-side connection limits
- Monitor pool metrics: active connections, idle count, wait times, acquisition failures

### 1.2 HTTP/2 Multiplexing

HTTP/2 multiplexing allows multiple requests and responses to be interleaved on a single TCP connection simultaneously. Unlike HTTP/1.1 pipelining (which requires responses in order), HTTP/2 streams are independent -- a slow response on one stream does not block others.

**Benefits for API clients:**
- Multiple in-flight requests over a single connection
- Header compression (HPACK) reduces overhead for repetitive API headers (auth tokens, content types)
- Reduces latency by up to 50% compared to HTTP/1.1 for same-host requests ([source](https://webscraping.ai/faq/reqwest/can-i-use-reqwest-with-http-2-and-what-are-the-benefits))
- Server push (rarely used by APIs, but available)

**Limitation:** Most servers set `SETTINGS_MAX_CONCURRENT_STREAMS` to ~100 as recommended by the HTTP/2 spec. A single HTTP/2 connection cannot have more than this many concurrent requests. For very high-throughput scenarios, multiple HTTP/2 connections may be needed ([source](https://github.com/seanmonstar/reqwest/issues/976)).

### 1.3 HTTP/3 and QUIC

HTTP/3 builds on QUIC (UDP-based transport) and eliminates head-of-line blocking at the transport layer. A packet loss on one stream does not stall others. For saalis running on potentially unreliable home networks, this could be beneficial -- but API server support is still limited. Most game metadata APIs do not yet advertise HTTP/3 support.

**Current status:** Experimental. The Rust ecosystem has the `h3` crate, and reqwest has an open issue for experimental HTTP/3 support ([reqwest #1558](https://github.com/seanmonstar/reqwest/issues/1558)). Not recommended as a dependency for saalis at this stage.

### 1.4 HTTP Pipelining

HTTP/1.1 pipelining sends multiple requests on a single connection without waiting for responses, but responses must arrive in order. This creates head-of-line blocking problems and is widely considered deprecated in favour of HTTP/2 multiplexing. Most servers and intermediaries have poor pipelining support. Not recommended.

Sources: [ureq GitHub](https://github.com/algesten/ureq), [reqwest docs](https://docs.rs/reqwest/), [ureq HTTP/2 issue](https://github.com/algesten/ureq/issues/219), [Rust HTTP client comparison](https://blog.logrocket.com/best-rust-http-client/)

## 2. API-Specific Batch Endpoints and Rate Limits

### 2.1 Steam Web API

| Property | Value |
|---|---|
| **Auth** | API key (free, requires Steam account) |
| **Rate limit** | 100,000 requests/day; 200 requests per 5 minutes per key |
| **Batch support** | Limited -- `appdetails` accepts comma-separated appids but only returns `price_overview` filter for multi-ID requests |
| **Best batch endpoint** | `IStoreService/GetAppList` returns all apps in one call; individual detail lookups via `appdetails?appids=<id>` |
| **Recommended strategy** | Fetch full app list once (cache it), then fetch details per-app with rate limiting |

The `store.steampowered.com/api/appdetails` endpoint technically accepts multiple appids as CSV, but returns null/400 for any filter other than `price_overview` when multiple IDs are provided. This is effectively a bug/limitation that makes true batching impractical for full metadata. Each game requires its own request for complete details.

Sources: [Steam Web API docs](https://partner.steamgames.com/doc/webapi/ISteamApps), [Steam API rate limiting](https://steamcommunity.com/discussions/forum/1/601902348018676495/), [Steam API guide](https://www.steamnavigator.com/blog/steam-web-api-guide)

### 2.2 IGDB API (Twitch)

| Property | Value |
|---|---|
| **Auth** | Twitch OAuth2 Client Credentials (free for non-commercial) |
| **Rate limit** | 4 requests/second; max 8 open requests concurrently |
| **Batch support** | Yes -- Multi-Query endpoint |
| **Best batch endpoint** | `POST https://api.igdb.com/v4/multiquery` -- up to 10 queries per request |
| **Query language** | APICalypse (custom query DSL in POST body) |

IGDB's Multi-Query is the most powerful batch mechanism among these APIs. A single POST to `/v4/multiquery` can contain up to 10 separate queries, each targeting different endpoints (games, covers, platforms, etc.). Combined with the APICalypse query language that supports filtering, sorting, and field selection in the request body, this allows fetching rich, related data in very few requests.

**Multi-Query syntax:**
```
query games "My Games" {
  fields name, cover;
  where id = (1, 2, 3);
};
query platforms "My Platforms" {
  fields name, abbreviation;
  where id = (1, 2);
};
```

At 4 req/s with 10 queries per request, effective throughput is 40 queries/second. This is very efficient.

Sources: [IGDB API docs](https://api-docs.igdb.com/), [IGDB Multi-Query](https://medium.com/igdb/its-here-the-new-igdb-api)

### 2.3 ScreenScraper API

| Property | Value |
|---|---|
| **Auth** | Username/password (requires registration at screenscraper.fr) |
| **Rate limit** | 50,000 requests/day max; thread count varies by membership tier |
| **Batch support** | No dedicated batch endpoint |
| **Primary endpoint** | `api2/jeuInfos.php` -- one game per request (by ROM hash, name, or system+ROM) |
| **Concurrency** | Non-registered: 1 thread; registered: 1-3 threads; donors/contributors: more |

ScreenScraper is primarily designed for ROM-based lookups (matching by file hash). It has no batch endpoint -- each game requires a separate request. The API compensates by allowing concurrent threads for registered/contributing users. The media download limit (50,000/day) is shared across all asset types, and composite "mix" images consume 3 downloads each (screenshot + box + marquee).

**Recommendation:** Implement aggressive local caching of ScreenScraper responses. Hash-based lookups are deterministic -- once a ROM is identified, cache the result indefinitely (or until a manual refresh is requested).

Sources: [Skyscraper scraping modules](https://github.com/muldjord/skyscraper/blob/master/docs/SCRAPINGMODULES.md), [Batocera scraping docs](https://wiki.batocera.org/scrape_from)

## 3. Rate Limiting Strategies

### 3.1 Algorithm Comparison

| Algorithm | Behaviour | Burst tolerance | Smoothness | Best for |
|---|---|---|---|---|
| **Token bucket** | Tokens added at fixed rate; request consumes a token; bucket has max capacity | Yes (up to bucket size) | Moderate | APIs that allow bursts (Steam, TMDB) |
| **Leaky bucket** | Requests enqueued; processed at fixed rate; overflow rejected | No | Very smooth | APIs with strict per-second limits (MusicBrainz, MobyGames) |
| **Sliding window** | Counts requests in a moving time window | Moderate | Good | APIs with per-minute/per-hour limits (GiantBomb's 450/15min) |
| **Fixed window** | Counts requests in fixed time slots | Yes (at window boundaries) | Poor | Simple rate caps; not recommended for client-side |

Sources: [Rate limiting algorithms overview](https://api7.ai/blog/rate-limiting-guide-algorithms-best-practices), [Token bucket vs leaky bucket](https://api7.ai/blog/token-bucket-vs-leaky-best-rate-limiting-algorithm), [Rate limiting algorithms](https://www.geeksforgeeks.org/system-design/rate-limiting-algorithms-system-design/)

### 3.2 Rust Implementation: The `governor` Crate

The `governor` crate implements the Generic Cell Rate Algorithm (GCRA), a variant of the leaky bucket that is both efficient and precise. It is the de facto standard for Rust rate limiting.

**Key properties for saalis:**
- Works in both sync and async contexts -- `check()` for non-blocking sync, `until_ready()` for async
- `Send + Sync` by default -- safe to share across threads
- No interior mutability tricks needed -- multiple references are fine
- Configurable quota: rate (elements per unit time) with burst allowance
- `no_std` compatible (via the underlying `ratelimit_meter` algorithm)

**Per-connector configuration:** Each connector should own its own `RateLimiter` instance configured to match the target API's limits. The limiter is checked before each request; if the quota is exhausted, the connector waits (blocks) until a token becomes available.

Sources: [governor crate](https://github.com/boinkor-net/governor), [governor docs](https://docs.rs/governor/latest/governor/_guide/index.html), [Rust rate limiting guide](https://www.shuttle.dev/blog/2024/02/22/api-rate-limiting-rust)

### 3.3 Adaptive Rate Limiting

Some APIs return rate limit state in response headers (e.g., `X-RateLimit-Remaining`, `X-RateLimit-Reset`, `Retry-After`). An adaptive rate limiter adjusts its internal token rate based on these signals rather than relying on a static configuration.

**Implementation approach:**
1. Start with the documented rate limit as the initial token rate
2. After each response, inspect rate limit headers
3. If `X-RateLimit-Remaining` drops below a threshold (e.g., 20% of quota), reduce the token rate
4. If a `429 Too Many Requests` is received, pause and respect the `Retry-After` header
5. Gradually restore the token rate as the server signals capacity recovery

This is particularly valuable for APIs like TheGamesDB that report `remaining_monthly_allowance` in every response.

## 4. Retry and Backoff

### 4.1 Exponential Backoff with Jitter

When a request fails transiently (network timeout, 5xx error, 429 rate limit), retrying with exponential backoff prevents overwhelming the already-stressed server.

**Formula:** `wait = min(cap, base * 2^attempt) + random(0, jitter_range)`

- **base:** Initial wait (e.g., 1 second)
- **cap:** Maximum wait (e.g., 60 seconds)
- **attempt:** Retry count (0-indexed)
- **jitter_range:** Random component to desynchronise retries across clients

**Why jitter matters:** Without jitter, if multiple saalis instances (or other clients) all hit a rate limit simultaneously, they all retry at the same exponential intervals, creating synchronised retry storms. Jitter spreads retries randomly across time, reducing collision probability. AWS recommends "full jitter" where the wait is `random(0, min(cap, base * 2^attempt))` for best results.

Sources: [AWS Exponential Backoff and Jitter](https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/), [AWS Builders Library: Timeouts, retries, backoff with jitter](https://aws.amazon.com/builders-library/timeouts-retries-and-backoff-with-jitter/), [Better Retries with Backoff and Jitter](https://www.baeldung.com/resilience4j-backoff-jitter)

### 4.2 Circuit Breaker

A circuit breaker monitors failure rates and "trips" (opens) when failures exceed a threshold, preventing further requests to a failing service for a cooldown period.

**States:**
1. **Closed** (normal): Requests pass through. Failures are counted.
2. **Open** (tripped): All requests are immediately rejected without being sent. A timer runs.
3. **Half-open** (probing): After the cooldown, a single request is allowed through. If it succeeds, the circuit closes. If it fails, the circuit re-opens.

**Configuration per connector:**
- Failure threshold: e.g., 5 consecutive failures or 50% failure rate in a 1-minute window
- Cooldown period: e.g., 30 seconds (short for transient issues), 5 minutes (for sustained outages)
- Half-open probe count: typically 1

**Why this matters for saalis:** If ScreenScraper goes down (which happens regularly during high-traffic periods), a circuit breaker prevents saalis from wasting its request quota and CPU time on requests that will never succeed. The scheduler can redirect enrichment work to other connectors in the meantime.

### 4.3 Retry Classification

Not all errors should be retried:

| Status Code | Meaning | Retry? | Strategy |
|---|---|---|---|
| 200 | Success | No | -- |
| 304 | Not Modified | No | Use cached response |
| 400 | Bad Request | No | Bug in connector; log and fix |
| 401/403 | Auth failure | No | Re-authenticate, then retry once |
| 404 | Not Found | No | Content does not exist on this source |
| 408 | Request Timeout | Yes | Exponential backoff |
| 429 | Rate Limited | Yes | Respect `Retry-After` header, then backoff |
| 500 | Server Error | Yes | Exponential backoff with jitter |
| 502/503/504 | Gateway/Unavailable | Yes | Exponential backoff; trip circuit breaker if sustained |

## 5. Caching and Conditional Requests

### 5.1 ETags and Conditional Requests

ETags are server-generated identifiers for a specific version of a resource. The client stores the ETag from a response and includes it in subsequent requests via `If-None-Match`. If the resource has not changed, the server returns `304 Not Modified` with no body, saving bandwidth and processing time.

**Flow:**
1. First request: `GET /games/123` -> Response includes `ETag: "abc123"`
2. Subsequent request: `GET /games/123` with `If-None-Match: "abc123"`
3. If unchanged: `304 Not Modified` (no body, very fast)
4. If changed: `200 OK` with new body and new ETag

**If-Modified-Since** works similarly but uses timestamps. ETags are preferred because they avoid clock synchronisation issues between client and server. RFC 9110 recommends servers send both; when both `If-None-Match` and `If-Modified-Since` are present, `If-None-Match` takes precedence.

**API support:** Not all game metadata APIs support conditional requests. IGDB and TMDB have good support. Steam's store API does not consistently return ETags. For APIs without ETag support, implement client-side caching with TTLs.

Sources: [MDN Conditional Requests](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Conditional_requests), [Optimizing REST APIs with ETags](https://zuplo.com/learning-center/optimizing-rest-apis-with-conditional-requests-and-etags), [ETag Wikipedia](https://en.wikipedia.org/wiki/HTTP_ETag)

### 5.2 Local Response Caching

For APIs without conditional request support, saalis should implement its own caching layer:

**Cache key:** `(connector_id, endpoint, parameters_hash)` -- stored in SQLite alongside the entity metadata.

**TTL strategy (tiered):**
- **Static metadata** (game title, release date, developer): TTL 30 days. This data rarely changes.
- **Dynamic metadata** (ratings, player counts, prices): TTL 24 hours.
- **Media/images** (box art, screenshots): TTL 90 days. Cache on disk, not in SQLite.
- **Search results** (discovery queries): TTL 1 hour. These are inherently ephemeral.

**Cache invalidation triggers:**
- User manually requests a refresh for a specific entity
- Scheduled background re-enrichment cycle (e.g., weekly)
- Source API reports data changed (via ETags or changelists)

### 5.3 Bandwidth Estimation

For a library of 1,000 games with 5 sources each:
- Without caching: ~5,000 API calls for initial population, ~5,000 calls per refresh cycle
- With caching + conditional requests: ~5,000 calls initially, ~500-1,000 actual data transfers on refresh (most return 304)
- Bandwidth savings: 80-90% on subsequent refreshes

## 6. Request Prioritisation

### 6.1 Priority Levels

Saalis should implement a priority queue for outbound API requests with at least three tiers:

| Priority | Use Case | Examples |
|---|---|---|
| **Critical** | User is actively waiting for a result | User searches for a game; user opens a game detail page that needs enrichment |
| **Normal** | Automated enrichment triggered by events | New game added to library; scheduled re-enrichment of stale metadata |
| **Background** | Bulk operations, speculative prefetch | Initial library scan; periodic full-library refresh; fetching "nice-to-have" metadata |

### 6.2 Implementation Approach

**Single priority queue per connector:** Each connector maintains a priority queue of pending requests. The connector's rate limiter gates dequeue speed. Higher-priority requests are dequeued first.

**Cross-connector prioritisation:** When multiple connectors compete for the same entity's metadata, the scheduler should:
1. Dispatch requests to the fastest/cheapest connector first (IGDB and Steam have the best rate limits)
2. Use slower connectors (MobyGames, MusicBrainz) only for data not available from primary sources
3. Cancel pending low-priority requests if a higher-priority request for the same entity arrives

**Starvation prevention:** Background tasks should not be starved indefinitely. A simple approach: reserve a percentage of rate limit capacity for background work (e.g., 20%). If no critical/normal work is pending, background work gets 100% of capacity.

Sources: [Priority Queue pattern (Azure)](https://learn.microsoft.com/en-us/azure/architecture/patterns/priority-queue), [API request prioritisation](https://datatas.com/how-to-implement-api-request-prioritization-for-performance-optimization/)

### 6.3 Connector Selection Order

For game metadata, the recommended connector priority (based on rate limits, data quality, and batch capabilities):

1. **IGDB** -- Best batch support (multi-query), good rate limit (4/s), rich data
2. **Steam** -- High rate limit (200/5min), authoritative for Steam games
3. **GiantBomb** -- Good batch support (100/request), decent rate limit
4. **TheGamesDB** -- True batch by ID, but monthly allowance is limited
5. **RAWG** -- No batch, but generous rate limit (5/s)
6. **ScreenScraper** -- No batch, thread-limited, but essential for retro ROM metadata
7. **MobyGames** -- Very slow rate limit; use for enrichment only

## 7. Connection Management

### 7.1 ureq Connection Management

ureq's `Agent` type holds an internal connection pool. All clones of an Agent share the same pool (via `Arc`). Connections are reused for subsequent requests to the same host.

**Configuration recommendations:**
- Create one `Agent` per connector, not per request
- ureq handles keep-alive automatically for HTTP/1.1 connections
- Set reasonable timeouts: connect timeout (5-10s), read timeout (30s for API responses)
- ureq does not support HTTP/2, so each connection handles one request at a time

### 7.2 reqwest Blocking Connection Management

If using reqwest's blocking client for connectors that benefit from HTTP/2:
- The `Client` maintains an internal connection pool with configurable idle timeout and max idle connections per host
- HTTP/2 is negotiated automatically via ALPN during TLS handshake
- Multiple requests to the same host multiplex over a single HTTP/2 connection
- Important: the blocking client must not be used inside an async runtime (it will panic)

### 7.3 DNS Caching

API hostnames rarely change their IP addresses, but DNS lookups add latency to each new connection. Options:
- **OS-level caching:** macOS and Linux cache DNS by default (typically 60s TTL). Usually sufficient.
- **Client-level:** reqwest uses the system resolver. ureq also uses the system resolver. Neither implements client-side DNS caching.
- **For saalis:** Rely on OS DNS caching. The connection pool reuses connections, so DNS lookups are infrequent after warmup.

## 8. Compression

### 8.1 Algorithm Comparison

| Algorithm | Compression ratio | Decompression speed | API support | Rust support |
|---|---|---|---|---|
| **gzip** | ~65% reduction | Fast | Universal (all APIs) | ureq (default), reqwest (default) |
| **Brotli** | ~70% reduction | Moderate | Common (TMDB, IGDB, most CDN-fronted APIs) | ureq (feature), reqwest (feature) |
| **zstd** | ~68% reduction, 42% faster compression than brotli | Very fast | Rare (newer APIs, Cloudflare-fronted) | reqwest (feature) |

Sources: [Compression comparison](https://paulcalvano.com/2024-03-19-choosing-between-gzip-brotli-and-zstandard-compression/), [HTTP compression guide](https://www.debugbear.com/blog/http-compression-gzip-brotli), [zstd vs brotli vs gzip for APIs](https://koder.ai/blog/zstd-vs-brotli-vs-gzip-api-compression)

### 8.2 Practical Impact for Saalis

Game metadata API responses are JSON (text-heavy, highly compressible). Typical response sizes:
- Uncompressed game detail: 5-20 KB
- With gzip: 1.5-6 KB (~70% reduction)
- With brotli: 1.3-5 KB (~75% reduction)

For a library of 1,000 games fetched from 5 sources:
- Uncompressed: ~50 MB total
- With gzip: ~15 MB total
- Savings: ~35 MB per full refresh cycle

On Batocera/loisto appliances with potentially limited bandwidth, this matters. Both ureq and reqwest send `Accept-Encoding: gzip` by default and decompress transparently. Enable brotli via feature flags for additional savings where supported.

### 8.3 Recommendation

Enable gzip by default (already the case for both ureq and reqwest). Enable brotli as a feature flag. Do not depend on zstd support from APIs -- it is too rare among game metadata services. The `Accept-Encoding` header should list all supported encodings; the server picks the best one it supports.

## 9. Summary: Recommended Architecture

### Per-Connector Components

Each connector in saalis should encapsulate:

1. **HTTP client** (ureq Agent or reqwest blocking Client) -- one per connector, reused for all requests
2. **Rate limiter** (governor `RateLimiter`) -- configured to match the target API's documented limits
3. **Retry policy** -- exponential backoff with full jitter, max 3-5 retries, with retry classification
4. **Circuit breaker** -- trips after N consecutive failures, cooldown before probing
5. **Response cache** -- keyed by endpoint + parameters, stored in SQLite with TTLs
6. **Priority queue** -- incoming work items sorted by priority tier
7. **Batch assembler** (where supported) -- collects individual requests and combines them into batch API calls

### Request Flow

```
User action / Scheduler event
        |
        v
  Priority Queue (per connector)
        |
        v
  Batch Assembler (if API supports batching)
        |
        v
  Cache Check (return cached if valid)
        |
        v
  Rate Limiter (block until token available)
        |
        v
  Circuit Breaker Check (fail-fast if open)
        |
        v
  HTTP Request (with conditional headers if ETag cached)
        |
        v
  Response Handler:
    - 200: Parse, cache, return
    - 304: Return cached version
    - 429: Respect Retry-After, re-enqueue
    - 5xx: Retry with backoff
    - 4xx: Log, do not retry (except 401 -> re-auth)
```
