# Deep Dive: SeaORM Migration System Evaluation

**Date:** 2026-03-15
**Scope:** Should saalis use SeaORM exclusively for migrations while using rusqlite directly for data access? Comprehensive evaluation of `sea-orm-migration`, `sea-orm-sync`, and the Entity First Workflow against saalis's own type-level migration system and lighter alternatives.
**Context:** The prior research (`2026-03-15-deepdive.seaorm-vs-rusqlite.md`) established that rusqlite is the correct data-access layer. This document specifically evaluates whether SeaORM's migration subsystem is worth depending on independently, given that migrations are critical, potentially destructive, and easy to get wrong.

---

## 1. sea-orm-sync: Status and API

### 1.1 What It Is

`sea-orm-sync` is a separate crate published on crates.io (not a feature flag on `sea-orm`). It is an automated derivative of the main `sea-orm` crate — the SeaQL team runs a script that converts async functions into synchronous ones and continuously rebases the sync crate on top of the main async crate. This means it inherits all new features from SeaORM automatically rather than being a manually maintained fork.

**Current version:** 2.0.0-rc.37 (published 2026-03-09). Still in release candidate status — not yet a stable 2.0.0 release.

**Backend:** rusqlite only. No PostgreSQL or MySQL support. The connection is wrapped in `Arc<Mutex<rusqlite::Connection>>` for thread-safe access.

**API surface:** The full SeaORM API minus async — Entity, ActiveModel, query builder, transactions (including nested), relationship loading, Entity Loader, Nested ActiveModel. The method signatures are identical to `sea-orm` but return `Result<T>` instead of `impl Future<Output = Result<T>>`. No async runtime is required.

**Dependencies:** sea-query (=1.0.0-rc.31), chrono, uuid, serde_json, sea-orm-macros, and numerous optional feature-gated dependencies. The dependency tree is substantial — this is the full ORM, just without tokio.

**Feature flags of note:**
- `schema-sync` — enables DDL schema comparison and synchronization (the Entity First Workflow)
- `entity-registry` — enables `inventory`-based entity auto-discovery
- `mock` — testing support
- `debug-print` — SQL logging

### 1.2 Stability Assessment

The crate is at RC37 with 233 downloads/month. By comparison, the main `sea-orm` crate has 250k+ weekly downloads. The sync variant is new (announced December 2025), actively developed, and not yet battle-tested at scale. The RC status means the API may still change before 2.0.0 stable.

For a project like saalis that values stability and minimal dependency churn, depending on a release candidate of a mechanically-derived crate is a risk. The mechanical derivation is clever engineering but introduces a novel failure mode: bugs in the async-to-sync translation script that only manifest at the sync layer.

**Source:** [How we made SeaORM synchronous (2025-12-12)](https://www.sea-ql.org/blog/2025-12-12-sea-orm-2.0/), [sea-orm-sync on lib.rs](https://lib.rs/crates/sea-orm-sync)

## 2. SeaORM Migration System: Architecture and Coupling

### 2.1 How sea-orm-migration Works

`sea-orm-migration` (current stable: 1.1.19) provides a migration framework where each migration is a Rust struct implementing `MigrationTrait`:

```rust
#[async_trait]
pub trait MigrationTrait: MigrationName + Send + Sync {
    async fn up(&self, manager: &SchemaManager) -> Result<(), DbErr>;
    async fn down(&self, manager: &SchemaManager) -> Result<(), DbErr>;
}
```

The `SchemaManager` wraps a `sea-query` builder and a database connection, providing methods like `create_table()`, `alter_table()`, `create_index()`, etc. Migrations are registered in a `Migrator` struct that implements `MigratorTrait`, which provides `up()`, `down()`, `status()`, and `fresh()` methods.

Migration state is tracked in a `seaql_migrations` table (not PRAGMA user_version). The table records migration name, applied timestamp, and version.

### 2.2 Async Requirement

The migration trait is `#[async_trait]`. The `up()` and `down()` methods are async. The `SchemaManager` methods are async. Running migrations requires an async runtime (tokio or async-std).

For the stable 1.x line, there is no synchronous migration path. You must have an async runtime available to run migrations.

### 2.3 sea-orm-migration-sync

A synchronous migration variant exists as `sea-orm-migration-sync` (version 2.0.0-rc.27+). It depends on `sea-orm-sync` with the `schema-sync` feature for DDL operations. However, this crate:

- Is even newer than `sea-orm-sync` (also RC status)
- Has negligible adoption (not found in standard crate searches)
- Is tightly coupled to the 2.0 RC ecosystem

### 2.4 Can You Use sea-orm-migration Independently?

No, not meaningfully. The dependency chain is:

```
sea-orm-migration 1.1.19
├── sea-orm ~1.1.19          (the full ORM)
├── sea-schema ^0.16.2       (schema introspection)
├── async-trait ^0.1          (async requirement)
├── tracing ^0.1
└── tracing-subscriber ^0.3
    (optional: clap, dotenvy, sea-orm-cli)
```

`sea-orm-migration` has a hard dependency on `sea-orm` itself (version-locked with `~`). You cannot depend on just the migration crate without pulling in the full ORM — entity system, query builder, connection abstraction, and all of sea-query. The migration crate does not expose a "schema-only" or "DDL-only" mode.

The migration crate re-exports `sea_orm` and `sea_query` from its own module tree. Migration implementations use `sea_orm::DbErr`, `sea_query::*` for table/column builders, and the `SchemaManager` which wraps a `sea_orm::DatabaseConnection`. This is deep structural coupling, not a surface-level dependency.

### 2.5 Hybrid Approach: SeaORM Migrations + rusqlite Data Access

The question of using SeaORM only for migrations while using rusqlite directly for data access runs into practical problems:

**Connection sharing:** SeaORM migrations operate on a `DatabaseConnection` (which in async mode wraps an sqlx pool, and in sync mode wraps a rusqlite connection inside `Arc<Mutex<>>`). Rusqlite operates on a bare `rusqlite::Connection`. These are different types wrapping the same underlying SQLite file. You would need to:
1. Open a SeaORM connection, run migrations, close it.
2. Open a rusqlite connection for data access.

This is workable but means two separate connection lifecycles and two separate dependency trees compiled into your binary. The SeaORM connection is only used for the few seconds of migration execution at startup.

**Version conflicts:** SeaORM pins specific versions of `libsqlite3-sys` through its dependency chain. Rusqlite also pins `libsqlite3-sys`. If both depend on incompatible versions, Cargo will refuse to compile (the `-sys` crate uses links = "sqlite3", which enforces a single version). In practice, the versions tend to align, but it is an ongoing maintenance burden to keep them synchronized.

**Binary size:** You pay the full SeaORM dependency cost (sea-query, sea-schema, sea-orm, async-trait, and in the async case, tokio/sqlx) for a feature used only once per startup. For a single-binary appliance, this is unnecessary bloat.

**Verdict:** The hybrid approach is technically possible but architecturally incoherent. You are paying for the full ORM to use 5% of its surface. The migration system is not designed as a standalone tool.

**Sources:** [sea-orm-migration on docs.rs](https://docs.rs/sea-orm-migration), [sea-orm-migration on crates.io](https://crates.io/crates/sea-orm-migration), [Setting Up Migration docs](https://www.sea-ql.org/SeaORM/docs/migration/setting-up-migration/)

## 3. SeaORM 2.0 Entity First Workflow

### 3.1 How Schema Sync Works

SeaORM 2.0 introduces an Entity First Workflow via the `schema-sync` and `entity-registry` feature flags. Instead of writing migration files, you define entity structs and call:

```rust
db.get_schema_registry("my_crate::entity::*").sync(db)?;
```

The system:
1. Discovers all registered entities via the `inventory` crate.
2. Reads the live database schema via `sea-schema`.
3. Compares in-memory entity definitions against actual tables/columns.
4. Creates missing tables, columns, foreign keys, and unique constraints idempotently.
5. Automatically resolves foreign key dependencies and creates tables in topological order.

### 3.2 What It Handles

- Missing tables: created with all columns and constraints.
- Missing columns: added with default values/expressions.
- Missing foreign keys and unique constraints.
- Column renames: via `renamed_from` annotation.
- Index management: dropping and recreating indexes.

### 3.3 What It Does NOT Handle

The documentation is explicit: **"schema sync would not attempt to do any destructive actions, so meaning no `DROP` on tables, columns and foreign keys."** Exception: dropping indexes is permitted.

This means no:
- Dropping tables
- Dropping columns
- Changing column types
- Deleting rows as part of schema evolution
- Data migration between tables
- Any transformation that could lose data

### 3.4 Relevance to Saalis

The Entity First Workflow is strikingly similar to saalis's own auto-detection of additive changes (diffing registered `Column<In<M>, As<V>>` impls against actual DB columns and auto-creating missing ones). Both use `inventory` for registration. Both are additive-only for the automatic path.

However, the Entity First Workflow replaces traditional migrations for the additive case. It does not address destructive changes at all — those still require the old migration system or manual handling. Saalis's `Migration<V>` trait with `MigrationOpSet` covers both cases: additive changes are auto-detected from Column registrations, destructive changes use the type-level op system.

The Entity First Workflow would be redundant with saalis's own additive detection, and it does not solve the hard problem (destructive migrations). It also requires the full SeaORM entity model, which conflicts with saalis's ZST column descriptors.

**Sources:** [SeaORM 2.0: Entity First Workflow](https://www.sea-ql.org/blog/2025-10-30-sea-orm-2.0/), [Entity First Workflow docs](https://www.sea-ql.org/SeaORM/docs/generate-entity/entity-first/)

## 4. Alternative Migration Crates Compared

### 4.1 rusqlite_migration

**Version:** 2.4.1 (actively maintained)
**Mechanism:** PRAGMA user_version — a 32-bit integer in the SQLite database header. No tracking table.
**Dependencies:** rusqlite (^0.38), log, optionally include_dir.
**Sync/async:** Fully synchronous. Works directly with `&mut rusqlite::Connection`.

**API:**
```rust
use rusqlite_migration::{Migrations, M};

let migrations = Migrations::new(vec![
    M::up("CREATE TABLE ..."),
    M::up("ALTER TABLE ...").down("..."),
    M::up(include_str!("003_fts5.sql")),
]);

migrations.to_latest(&mut conn)?;       // apply pending
migrations.to_version(&mut conn, 2)?;   // migrate to specific version
migrations.validate()?;                  // test integrity
```

**Safety features:**
- Up and down migrations (down is optional per migration, but if you migrate downward, all reversed migrations must have `.down()` defined or the operation fails).
- `validate()` method for testing migration integrity.
- Foreign key check error reporting via `ForeignKeyCheckError`.
- Snapshot testing integration with `insta`.
- No unsafe code (`#![forbid(unsafe_code)]`).

**Limitations:**
- No checksum/divergence detection (if someone edits a previously-applied migration, the system does not detect it).
- No dry-run capability in the public API.
- The user_version is shared — if any other code modifies `PRAGMA user_version`, behavior is undefined.
- Limited to SQLite (by design).
- i32 version limit (~2 billion migrations — not a practical concern).

**Suitability for saalis:** Excellent. Direct rusqlite integration, zero async overhead, PRAGMA user_version aligns with SQLite best practices (version is atomic with database state, survives backup/restore, no extra table). The crate is purpose-built for exactly saalis's use case.

**Sources:** [rusqlite_migration docs](https://docs.rs/rusqlite_migration/latest/rusqlite_migration/), [GitHub](https://github.com/cljoly/rusqlite_migration)

### 4.2 refinery

**Version:** 0.9.0
**Mechanism:** `refinery_schema_history` tracking table with version, name, applied timestamp, and checksum.
**Dependencies:** refinery-core, plus the selected driver feature (rusqlite, postgres, mysql, etc.).
**Sync/async:** Both — `Runner::run()` for sync (rusqlite), `Runner::run_async()` for async backends.

**API:**
```rust
mod embedded {
    use refinery::embed_migrations;
    embed_migrations!("migrations");
}

embedded::migrations::runner().run(&mut conn)?;
```

Migrations follow a naming convention: `V{version}__{description}.sql` or `V{version}__{description}.rs`.

**Safety features:**
- **Divergent migration detection:** Checksums of applied migrations are stored. If a migration file is modified after application, refinery detects the divergence and refuses to run.
- **Missing migration detection:** If a migration that was applied is no longer present in the code, refinery warns.
- **Grouped transactions:** Option to wrap all migrations in a single transaction (`set_grouped(true)`) for atomic all-or-nothing application.
- **Per-migration transactions:** Default behavior wraps each migration individually.

**Limitations:**
- Uses a tracking table instead of PRAGMA user_version. This table must be backed up/restored alongside the data. If the database is restored from backup without the tracking table, refinery's state desyncs. With PRAGMA user_version, the version is embedded in the database header and always consistent.
- Cross-database support (Postgres, MySQL) is wasted dependency weight for SQLite-only usage.
- No built-in down-migration support — the philosophy is "to undo a migration, write a new migration that reverses it."

**Suitability for saalis:** Acceptable but not ideal. The checksum-based divergence detection is genuinely useful for team environments, but saalis is a single-binary appliance — there is no team running migrations in staging. The tracking table approach is less robust than PRAGMA user_version for backup/restore scenarios (which are a first-class feature for saalis — the appliance needs reliable backup to SD card).

**Sources:** [refinery GitHub](https://github.com/rust-db/refinery), [refinery docs](https://docs.rs/refinery/latest/refinery/)

### 4.3 migrant_lib

**Version:** 0.34.0
**Mechanism:** Configuration-file-based tracking.
**Dependencies:** Multiple, including database drivers enabled via features (d-sqlite).
**Sync/async:** Synchronous for rusqlite.

Migrations can be files, string literals, or Rust functions with signature `fn(ConnConfig) -> Result<()>`. Supports embedding migrations in the binary via `include_str!`.

**Suitability for saalis:** Poor. The configuration-file-based approach is designed for traditional server applications with a `.migrant.toml` or similar config. It is over-engineered for an embedded appliance and under-documented compared to alternatives.

**Sources:** [migrant_lib on crates.io](https://crates.io/crates/migrant_lib), [migrant_lib docs](https://docs.rs/migrant_lib/latest/migrant_lib/)

### 4.4 barrel

**Version:** 0.7.0 (released 2021-11-08, minimal maintenance)
**What it is:** A schema builder DSL, not a migration runner. It generates SQL DDL from Rust code:

```rust
m.create_table("users", |t| {
    t.add_column("name", types::varchar(255));
    t.add_column("age", types::integer());
});
```

Barrel integrates with Diesel and refinery as a way to write migration SQL in Rust instead of raw SQL. It does not manage migration state, versioning, or execution.

**Suitability for saalis:** Not applicable. Barrel solves a problem saalis already solves differently — the ZST column descriptor system and `#[register]` macro generate schema DDL from type-level declarations. Barrel would be a redundant, less type-safe alternative to saalis's own system.

**Sources:** [barrel on lib.rs](https://lib.rs/crates/barrel), [barrel GitHub](https://github.com/rust-db/barrel)

### 4.5 Hand-Written PRAGMA user_version

The simplest approach: write a `migrate()` function that reads `PRAGMA user_version`, applies numbered SQL files in order, and updates the version:

```rust
const MIGRATIONS: &[&str] = &[
    include_str!("migrations/001_initial.sql"),
    include_str!("migrations/002_manifest.sql"),
];

fn migrate(conn: &Connection) -> Result<()> {
    let current: i32 = conn.query_row("PRAGMA user_version", [], |r| r.get(0))?;
    let tx = conn.transaction()?;
    for (i, sql) in MIGRATIONS.iter().enumerate() {
        let version = (i + 1) as i32;
        if version > current {
            tx.execute_batch(sql)?;
        }
    }
    tx.execute_batch(&format!("PRAGMA user_version = {}", MIGRATIONS.len()))?;
    tx.commit()
}
```

**Advantages:** Zero dependencies. Complete control. ~20 lines of code. Transparent behavior.

**Disadvantages:** No down-migration. No validation. No error types. No foreign key checking. Every project that writes this re-invents the same patterns.

**Suitability for saalis:** This is what `rusqlite_migration` wraps with proper error handling, down-migration support, and validation. Using `rusqlite_migration` over hand-rolling is a small dependency cost for meaningful ergonomic and safety gains.

**Sources:** [Cargo's internal SQLite migration pattern](https://doc.rust-lang.org/stable/nightly-rustc/src/cargo/util/sqlite.rs.html), [SQLite forum: migration strategies](https://www.sqliteforum.com/p/sqlite-versioning-and-migration-strategies)

## 5. Migration Safety: What Makes Migrations Dangerous

### 5.1 Data Loss Scenarios

1. **Column drop without backup:** `ALTER TABLE DROP COLUMN` (SQLite 3.35.0+) permanently removes data. If the migration is wrong, the data is gone.
2. **Table recreation dance:** Before SQLite 3.35.0, dropping a column requires creating a new table, copying data, dropping the old table, and renaming. Any bug in the copy step loses data.
3. **Type change with data truncation:** Changing a column from TEXT to INTEGER silently truncates non-numeric values.
4. **Interrupted migration:** If a migration crashes mid-execution without a transaction, the database is left in a half-migrated state that neither the old nor the new code can handle.
5. **Version tracking desync:** If migration tracking (table or PRAGMA) gets out of sync with the actual schema, future migrations may apply against the wrong base state — either skipping needed changes or re-applying already-applied ones.

### 5.2 Rollback Strategies

**Transaction wrapping:** The simplest safety net. SQLite supports transactional DDL — `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE` can all be rolled back within a transaction. If any step fails, the entire migration is reverted. Both `rusqlite_migration` and refinery support this.

**Pre-migration backup:** Copy the database file before running migrations. SQLite's `Connection::backup()` API (which saalis already has enabled via the `backup` feature) supports online backup — copying the database while it is open. This is the nuclear fallback.

**Down migrations:** Writing explicit reverse-migration SQL. Supported by `rusqlite_migration` (optional per migration) and SeaORM's migration system (required `down()` method). Not supported by refinery (philosophy: write a new forward migration to undo).

**Data capture before destructive ops:** Saalis's own `MigrationOpSet` system auto-captures affected data before executing destructive ops (DropTable, DropColumn, DeleteRows) and stores it in `migration_data_reverts`. This is more sophisticated than any external migration crate provides.

### 5.3 What SeaORM Provides for Safety That Raw SQL Doesn't

SeaORM's migration system uses the sea-query builder for DDL, which:
- Prevents SQL syntax errors through a typed builder API (you cannot misspell `CREATE TBALE`).
- Handles SQLite-specific DDL quirks (e.g., SQLite's limited ALTER TABLE support).
- Provides the `SchemaManager` abstraction which checks whether tables/columns exist before creating/dropping them.

However, these safety benefits are modest for a SQLite-only project:
- SQL syntax errors in migration files are caught immediately on first run (migrations run on startup, fail fast).
- SQLite DDL quirks are well-known and finite (no ADD CONSTRAINT, limited ALTER TABLE). A small number of SQL migration files are easy to validate.
- Existence checks (`IF NOT EXISTS`, `IF EXISTS`) are standard SQL that does not require an ORM.

The real safety gap between raw SQL migrations and SeaORM's system is **idempotency checking** — SeaORM's SchemaManager can detect whether a table or column already exists before trying to create it. For raw SQL, you use `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE ADD COLUMN` (which errors on duplicate — requiring a try/catch pattern in SQLite). This is a minor inconvenience, not a fundamental safety issue.

## 6. Saalis's Own Migration System vs External Crates

### 6.1 What Saalis's Design Already Provides

The architecture document (`2026-03-10-final-architecture-decisions.md`, section 1.2.6) defines a migration system that is significantly more sophisticated than any external crate:

**Additive changes are automatic:** The host diffs registered `Column<In<M>, As<V>>` impls against actual DB columns at startup. Missing columns are auto-created. No migration code needed for the common case.

**Destructive changes use type-level ops:** `Migration<const V: u32>` trait with `type Ops: MigrationOpSet` — a tuple of ZST operation types. The closed set of ops (CreateTable, DropTable, CreateColumn, DropColumn, RenameColumn, RenameTable, ModifyType, CreateIndex, DropIndex, CopyRows, DeleteRows) ensures every operation has a known reversal.

**Auto-capture of affected data:** All destructive ops capture data before execution, stored in `migration_data_reverts`. Rollback replays captured data.

**Compile-time validation:** Migration ops reference ZSTs with trait bounds — invalid ops are compile errors. `#[register]` bakes descriptor data at compile time.

**Pre-migration snapshot:** Host creates `.saalis` backup before any migration run.

### 6.2 What External Crates Can Add

Given saalis's own system, the question is what an external migration crate provides that is not already covered:

| Capability | Saalis system | rusqlite_migration | refinery | sea-orm-migration |
|------------|---------------|-------------------|----------|-------------------|
| Version tracking | Custom (migration tracking table) | PRAGMA user_version | refinery_schema_history table | seaql_migrations table |
| Additive auto-detect | Yes (Column registration diff) | No | No | Yes (Entity First) |
| Destructive ops | Type-level MigrationOpSet | Raw SQL | Raw SQL | sea-query builder |
| Compile-time validation | Yes (ZST trait bounds) | No | No | No |
| Auto data capture | Yes (migration_data_reverts) | No | No | No |
| Down migrations | Auto-derived from op reversal | Optional per migration | No (write new forward) | Required down() method |
| Checksum/divergence | No | No | Yes | No |
| Dry-run | No | No | No | No |
| Pre-migration backup | Yes (.saalis snapshot) | No | No | No |

The external crates are strictly less capable than saalis's own system for the destructive/transformative case. Their value is limited to:

1. **Bootstrapping version tracking** — reading/writing PRAGMA user_version or a tracking table.
2. **Providing a clean API for the "apply pending migrations" loop** — the `to_latest()` / `runner().run()` pattern.

These are both trivially implementable (20-30 lines of code), but `rusqlite_migration` packages them with proper error handling, validation, and testing support.

### 6.3 Where rusqlite_migration Fits

`rusqlite_migration` is the right tool for the **execution layer** of saalis's migration system. The architecture calls for:

1. Host startup: create .saalis snapshot.
2. Discover MigrationDescriptors via registry.
3. Check per-table version.
4. Run pending migrations.
5. After explicit migrations: diff columns, auto-create missing.

Steps 1-3 and 5 are saalis's own logic. Step 4 needs an execution mechanism that:
- Tracks which version has been applied.
- Applies SQL statements in order.
- Handles transactions correctly.

`rusqlite_migration` handles step 4 cleanly. The migration ops from `MigrationOpSet` compile down to SQL statements (via `OpDescriptor`). Those SQL statements can be fed to `rusqlite_migration` as `M::up(sql).down(reverse_sql)` entries.

The key architectural decision is: **saalis generates the migration SQL from its type-level op system; rusqlite_migration executes and tracks it.** This gives:
- Compile-time safety from saalis's ZST migration ops.
- Auto-derived down migrations from saalis's op reversal.
- PRAGMA user_version tracking from rusqlite_migration.
- Proper error handling and validation from rusqlite_migration.
- Zero async overhead, zero ORM dependency.

## 7. Comparison Summary and Recommendation

### 7.1 SeaORM for Migrations Only: Rejected

Using SeaORM (or sea-orm-migration, or sea-orm-sync + sea-orm-migration-sync) exclusively for migrations is not recommended because:

1. **Cannot be used independently.** sea-orm-migration depends on the full sea-orm crate. You pay for the entire ORM to use the migration subsystem.
2. **Async requirement (1.x).** The stable migration API is async, requiring a tokio/async-std runtime for a startup-only operation in a sync-core application.
3. **RC status (2.x sync).** sea-orm-sync and sea-orm-migration-sync are release candidates with minimal adoption. Depending on them for critical infrastructure (migrations) is premature.
4. **Redundant with saalis's own system.** SeaORM's migration system provides less safety than saalis's type-level MigrationOpSet (no compile-time validation, no auto data capture, no auto-derived reversals). The Entity First Workflow duplicates saalis's additive auto-detection. Neither adds value.
5. **Dependency and binary cost.** The full SeaORM dependency tree (sea-query, sea-schema, sea-orm, async-trait, and potentially tokio) for a feature used once per startup is architecturally disproportionate.
6. **Connection management friction.** Running SeaORM migrations on a SeaORM connection and then switching to rusqlite for data access requires two connection lifecycles, potential `libsqlite3-sys` version conflicts, and conceptual overhead.

### 7.2 Recommended Stack

```
┌─────────────────────────────────────────────────────┐
│                  saalis migration system             │
│  Migration<V> + MigrationOpSet → OpDescriptor → SQL  │
│  (compile-time ZST ops, auto-reversal, data capture) │
└──────────────────────┬──────────────────────────────┘
                       │ generates SQL strings
                       ▼
┌─────────────────────────────────────────────────────┐
│             rusqlite_migration (2.4.1)               │
│  Migrations::new(vec![M::up(sql).down(reverse)])     │
│  .to_latest(&mut conn)                               │
│  PRAGMA user_version tracking                        │
└──────────────────────┬──────────────────────────────┘
                       │ executes on
                       ▼
┌─────────────────────────────────────────────────────┐
│              rusqlite (0.32, bundled)                 │
│  Direct sync SQLite access                           │
│  Single writer + reader connection                   │
└─────────────────────────────────────────────────────┘
```

**Migration flow at startup:**
1. Open rusqlite Connection.
2. Create `.saalis` backup via `Connection::backup()`.
3. Collect `MigrationDescriptor`s from `inventory` registry.
4. Generate SQL from `OpDescriptor` sequences.
5. Build `rusqlite_migration::Migrations` from generated SQL.
6. Call `migrations.to_latest(&mut conn)` — applies pending, tracks via PRAGMA user_version.
7. Diff registered `Column<In<M>, As<V>>` impls against DB schema.
8. Auto-create missing columns (additive changes).

### 7.3 What This Means Practically

- **No SeaORM dependency at all.** Not for data access, not for migrations, not for schema building.
- **rusqlite_migration is the only migration dependency.** It is small (rusqlite + log), synchronous, uses the correct SQLite mechanism (PRAGMA user_version), and integrates directly with the rusqlite connection saalis already owns.
- **Saalis's type-level migration ops remain the source of truth.** They generate SQL. rusqlite_migration executes it. The boundary is clean.
- **All SeaORM-unique safety features are superseded.** Saalis's compile-time validation, auto data capture, and auto-derived reversals are strictly more capable than anything SeaORM's migration system provides.
- **The only thing "lost" vs SeaORM migrations is the sea-query DDL builder** — which prevents SQL syntax typos in migration files. For saalis, the migration SQL is machine-generated from `OpDescriptor` sequences, not hand-written. Typos in generated SQL are bugs in the generator, caught by integration tests, not in migration files.
