# Gap Hunter Review: Verkko Design Documents (Stage 1)

**Date:** 2026-03-28
**Reviewer identity:** Protocol implementer (15 years, 4 protocol stacks from spec, 80+ interop-critical gap finds)
**Method:** Mental implementation walkthrough of each document, noting every point where coding stops because the spec does not tell me what to do.
**Input:** verkko-crypto, verkko-protocol, verkko-mesh, verkko-relay, verkko-seat, verkko-matter, verkko-ops, verkko-taxonomy.

---

## Severity ratings

| Rating | Meaning |
|--------|---------|
| BLOCKING | Cannot write code without this. Implementation halts. |
| AMBIGUOUS | Could guess, might guess wrong. Two implementers will produce incompatible code. |
| MISSING-ERROR | Happy path specified, error/failure path absent. |
| UNTESTABLE | No way to verify correctness from the spec alone. |

## 1. verkko-crypto gaps

### C-1. CSPRNG self-test: what constitutes "non-repeating across 3 samples"? [AMBIGUOUS]

Section 1.3 says "32 bytes, non-zero, non-repeating across 3 samples." Does "non-repeating" mean no two of the three 32-byte samples are identical? Or that no byte within a sample repeats? The first interpretation is trivially satisfied by any working CSPRNG; the second is wrong (birthday paradox makes repeating bytes expected in 32 bytes). An implementer needs the exact predicate.

**Minimum fix:** "Generate three independent 32-byte samples. All three must be distinct (bitwise not-equal). Any two equal samples indicate CSPRNG failure."

### C-2. Enrollment cert format: what is `genesis_timestamp`? [AMBIGUOUS]

Section 1.3 founding transcript includes `genesis_timestamp` but its format is unspecified. Unix seconds? Milliseconds? HLC? This is a peer-agreement-critical derivation (FoundingHash). Two peers using different timestamp formats produce different founding hashes and silently partition.

**Minimum fix:** "genesis_timestamp is a u64 encoding Unix milliseconds since epoch, big-endian."

### C-3. Key identification collision handling [MISSING-ERROR]

Section 1.5 says `key_id = BLAKE3(K_epoch_N || channel_index)[0..8]` and "birthday collision at 2^32 keys (mesh will never reach this)." But what does a peer do if it encounters a collision in practice (bug, malice)? The rotation commit procedure (Section 2.4) uses key_id to look up keys. If two channels produce the same key_id, the receiver tries the wrong key, gets an AEAD auth failure, and then what?

**Minimum fix:** "On AEAD decryption failure with a matched key_id, iterate remaining keys with the same key_id. If none decrypt, treat as corrupt frame (silent drop, no scar)."

### C-4. BroadcastEnvelope: `algorithm` field values [BLOCKING]

Section 2.1 includes `algorithm: u8` but no registry of valid values. If I am implementing, what value do I write for ChaCha20-Poly1305? For XChaCha20? Without a numeric mapping, two implementations will use different constants and fail to interoperate.

**Minimum fix:** "`0x01 = ChaCha20-Poly1305 (transport), 0x02 = XChaCha20-Poly1305 (re-encryption). Values 0x00 and 0x03-0xFF reserved.`"

### C-5. BroadcastEnvelope: `flags` field semantics [AMBIGUOUS]

Section 2.1 includes `flags: u8` with no definition of what any flag bit means. Implementer writes 0x00 and moves on, but another implementer might set bits. Interop risk.

**Minimum fix:** "All flags reserved, must be 0x00. Receivers must ignore unknown flags."

### C-6. BroadcastEnvelope: `wrapped_keys` encryption scheme [BLOCKING]

Section 2.1 says "Key wrapped N-1 times (one per recipient cluster)" but does not specify the key wrapping algorithm. Is it X25519 DH + HKDF + AES-256-KW? X25519 DH + ChaCha20-Poly1305? Age uses X25519 + HKDF + ChaCha20-Poly1305. The spec says "age-style" but does not commit to a construction. The 48-byte wrapped key size constrains the choice but does not fully specify it.

**Minimum fix:** "Each wrapped key is: ephemeral X25519 pubkey (32 bytes) + ChaCha20-Poly1305(key=HKDF(DH(ephemeral, recipient_CIK), 'saalis-wrap-v1'), nonce=0, content_key) (32 + 16 = 48 bytes total)." or equivalent.

### C-7. HKDF-Expand without Extract [AMBIGUOUS]

Section 2.4 uses `HKDF-Expand` directly: `K_epoch_N = HKDF-Expand(prk=K_epoch_{N-1}, info="epoch:advance", len=32)`. RFC 5869 says HKDF-Expand input must be a pseudorandom key from HKDF-Extract. Using a previous HKDF-Expand output as the PRK is technically outside the security proof. Is this intentional (the initial channel key from CSPRNG is sufficient PRK quality)? An implementer needs to know whether to call Extract first or use Expand directly.

**Minimum fix:** "K_epoch_0 is a CSPRNG output (sufficient PRK entropy). Subsequent HKDF-Expand uses the previous output as PRK. This is safe because CSPRNG output has full entropy and HKDF-Expand preserves PRK quality for chain derivation [Krawczyk2010, Section 3.3]."

### C-8. Micro-epoch checkpoint: what is stored? [AMBIGUOUS]

Section 2.4 says "every 720 micro-epoch advances, a checkpoint records the micro-epoch key." Where is this checkpoint stored? In the nonce persist file? In SQLite? In memory only? If a peer crashes and restarts, can it recover the checkpoint? This determines whether catch-up after restart requires re-deriving from the macro-epoch key.

**Minimum fix:** "Micro-epoch checkpoints are stored in the managed arena (volatile). On restart, the peer re-derives from the current macro-epoch key. Checkpoints are an optimization, not a durability requirement."

### C-9. Session_ID: `generation` field undefined [BLOCKING]

Section 2.4 defines `Session_ID = BLAKE3(session_key || channel_path || generation)[0..8]` but `generation` is never defined. What is it? A counter? A timestamp? Without this, I cannot compute Session_IDs.

**Minimum fix:** "generation is a u64 counter, incremented on each session renegotiation for the same channel_path. Starts at 0."

### C-10. BLAKE3 test vectors: empty table [UNTESTABLE]

Section 4.1 includes a test vector table with "(defined at implementation time)" placeholder. The spec itself says divergent BLAKE3 output will silently partition the mesh and "prevention is the only defense." Yet the only defense (test vectors) is deferred. This is a correctness-critical gap.

**Minimum fix:** Provide at least one concrete test vector per registered domain separator using known input values.

## 2. verkko-protocol gaps

### P-1. Connection ID generation [BLOCKING]

Section 3.1 shows `connection_id: u32` in the wire format but never specifies how it is generated. Random? Negotiated during handshake? Monotonic? This is critical for connection migration (QUIC-style PATH_CHALLENGE requires stable connection IDs). Two implementations could use different generation schemes and fail migration.

**Minimum fix:** "connection_id is a 32-bit random value generated by the initiator during Noise handshake. Communicated in the first handshake message. Both peers use the same connection_id for the session lifetime."

### P-2. `destination` field: BLAKE3(CIK_pubkey) or what? [AMBIGUOUS]

Section 3.1 says `destination = BLAKE3(CIK_pubkey)[0..4]`. But CIK is per-cluster, not per-peer. If I am a relay forwarding a frame, I look up the destination cluster by this truncated hash. What happens on collision (two clusters with the same 4-byte hash)? At 12 clusters the probability is negligible, but the spec does not say what to do.

**Minimum fix:** "Collision is negligible at 12 clusters. If observed, the peer uses connection_id to disambiguate."

### P-3. Noise handshake message format [BLOCKING]

The spec says "Noise KK for established pairs, Noise XX for first contact" but does not specify the Noise prologue, the payload contents of each handshake message, or how the handshake maps to UDP frames. A Noise implementation needs: prologue bytes, handshake pattern tokens, payload structure per message, and how to detect KK vs XX on the wire.

**Minimum fix:** Specify: (a) prologue = `"saalis-verkko-v1"`, (b) first byte of first message indicates pattern (0x01 = KK, 0x02 = XX), (c) each handshake message is one UDP frame with the outer header, (d) payload of each handshake message carries the protobuf schema hash (SchemaSync) and OBSERVED_ADDR.

### P-4. SchemaSync: when exactly does it happen? [AMBIGUOUS]

Section defines SchemaSync (protobuf schema exchange during handshake) but the handshake is Noise KK/XX. When in the handshake does schema exchange occur? In the first payload? After handshake completion? Is it part of the encrypted payload of the last handshake message?

**Minimum fix:** "SchemaSync occurs in the first post-handshake application message. Client sends schema hash (32 bytes). Server responds with full schema if mismatch, or empty response if match."

### P-5. Stream multiplexing: no wire format [BLOCKING]

Defined Concepts lists Stream with independent sequence numbers and flow control, but the Body section does not define stream IDs, stream creation/teardown, or how streams map to the wire format. The inner payload has `channel_index` and `msg_type` but no `stream_id`. An implementer cannot multiplex.

**Minimum fix:** Define `stream_id: u8` in the encrypted payload (after msg_type). Document stream open/close signaling.

### P-6. 0-RTT resumption: no specification [BLOCKING]

The taxonomy lists 0-RTT as missing. verkko-seat references "0-RTT with a stored resumption token." No mechanism is specified. What token? What key material? What replay protection?

**Minimum fix:** "Deferred to v2. All sessions begin with full Noise handshake in v1."

### P-7. LEDBAT: which variant exactly? [AMBIGUOUS]

Section 3.3 says "LEDBAT [RFC6817] with adaptive target delay" but the target delay table has values (10ms-100ms) that differ from RFC 6817's fixed 100ms. Is this LEDBAT++ [RFC9330]? A custom variant? An implementer needs to know whether to use the gain function from RFC 6817 or a modified one.

**Minimum fix:** "LEDBAT as specified in RFC 6817 with the following modifications: (1) target_delay is metabolic-state-dependent per the table, (2) dual controllers for asymmetric links."

### P-8. Reliable ordered: sequence number width [BLOCKING]

Section 3.3 mentions "monotonic sequence, SACK ACKs" for reliable ordered streams but does not specify the sequence number width. u16? u32? u64? This determines the wire format and wrap-around behavior.

**Minimum fix:** "u32 sequence number per stream, wrapping at 2^32."

### P-9. SACK format [BLOCKING]

Section 3.3 mentions SACK ACKs but never defines the SACK wire format. How many selective ack blocks? What is the encoding? This is needed for both reliability and the health pipeline (delivery ratio).

**Minimum fix:** Define SACK as: `[u32 cum_ack] [u8 num_blocks] [u32 start, u32 end]*num_blocks`. Limit num_blocks to fit in one frame.

### P-10. PMTU probing mechanism [AMBIGUOUS]

Section 3.3 says "PMTU probing on connection establishment" but does not specify the probing algorithm. RFC 4821? Custom? What probe sizes? What happens if probing fails?

**Minimum fix:** "RFC 4821 binary search. Initial: 1280. Max: 1452. Probe at connection start and every 10 minutes. On failure: fall back to 1280."

### P-11. Cristian's clock sync: how is the RTT sample taken? [AMBIGUOUS]

Section 3.3 says "Cristian's algorithm during Noise handshake (32 bytes)" but does not specify which handshake messages carry the timestamps. The accuracy of Cristian's algorithm depends on which round-trip is measured.

**Minimum fix:** "Timestamps are carried in the last two handshake messages (response and final). 32 bytes = two u128 timestamps (send time and receive time)."

### P-12. NAT traversal: hole punching mechanism [BLOCKING]

Section defines NATTraversal with "hole punching attempted before relay fallback" but does not specify the hole punching protocol. STUN? ICE? Custom? What STUN server? How are candidates exchanged?

**Minimum fix:** "STUN binding request to discover server-reflexive address. Candidates exchanged in Noise handshake payload. Simultaneous open attempted. Timeout: 5 seconds, then relay fallback."

### P-13. Peer discovery: DHT mechanism [AMBIGUOUS]

Section says "Mainline DHT" but does not specify the key used for lookup, the value stored, or how the DHT client bootstraps. An implementer needs to know: what key is published, what value is returned, and what bootstrapping nodes to use.

**Minimum fix:** "Key = BLAKE3(founding_hash || 'dht-v1')[0..20]. Value = encrypted address hint (CIK-encrypted). Bootstrap: standard Mainline DHT bootstrap nodes."

## 3. verkko-mesh gaps

### M-1. Invite token format [BLOCKING]

Section 1.4 says the QR code contains "CIK pubkey, PSK, address hints, relay DHT key, founding hash, expiry, nonce" but does not specify the binary format or encoding. How is this serialized? What delimiter? What encoding for the QR code (binary, base64, protobuf)?

**Minimum fix:** Define a binary format: `[u8 version][u8[32] CIK_pub][u8[32] PSK][u8 num_hints][AddrHint*][u8[32] relay_dht_key][u8[32] founding_hash][u64 expiry_unix_ms][u8[12] nonce]`. QR encodes base64url of this.

### M-2. SAS verification: what algorithm? [BLOCKING]

Section 1.2 says "SAS verification required" and the taxonomy says "6-emoji or word sequence." But what is the SAS derivation? ZRTP-style (RFC 6189)? Hash of shared secret? Without the exact derivation, two implementations will show different SAS to the user.

**Minimum fix:** "SAS = BLAKE3('saalis-sas-v1' || noise_handshake_hash)[0..4]. Displayed as 4-word sequence from the BIP-39 English wordlist (indices = bytes mod 2048)."

### M-3. Enrollment cert: what fields does it contain? [AMBIGUOUS]

Section 1.2 shows `enrollment_cert = Sign(admin_DK, new_DK_pub || new_DEK_pub || enrollment_epoch || cluster_CIK)` but does not specify the exact byte layout. Is `enrollment_epoch` a u32? u64? How is `cluster_CIK` represented? The enrollment cert is logged to the control log and used in dominance cascade, so format disagreement would break the mesh.

**Minimum fix:** `[u8[32] new_DK_pub][u8[32] new_DEK_pub][u32 enrollment_epoch][u8[32] cluster_CIK_pub][u8[64] admin_sig]` = 192 bytes.

### M-4. Control log entry format [BLOCKING]

Section 1.7 says "signed, hash-linked, append-only GSet. Scope (2B channel_index) per entry. Lamport clock (8B)." But the entry format is not fully specified. What are the entry types? What is the hash link (BLAKE3 of previous entry?). What is the exact serialization?

**Minimum fix:** Define: `ControlLogEntry { prev_hash: [u8; 32], lamport: u64, scope: u16, entry_type: u8, payload: [u8], author_dk: [u8; 32], signature: [u8; 64] }`. List entry_type values.

### M-5. Gateway election: max-register CRDT semantics [AMBIGUOUS]

Section 7.2 says "Max-register CRDT. No explicit election." but does not define what value is stored in the register, how peers propose themselves, or how the CRDT state is disseminated. A CRDT needs a merge function and a dissemination path.

**Minimum fix:** "Each cluster peer publishes (availability_score, ett_score, dk_hash) to an LWW-Register. Merge: max by (availability, -ett, dk_hash). Disseminated via intra-cluster heartbeat extension."

### M-6. Gateway handoff: CPSK session transfer [AMBIGUOUS]

Section 7.2 says "gateway-release with CPSK session state (~200 bytes)" but does not specify what CPSK session state includes. Is it the Noise session keys? The CPSK itself? Nonce state? A new gateway needs to know what it receives and how to resume the session.

**Minimum fix:** "CPSK session state = current CPSK (32 bytes) + Noise CipherState pair (2x 32-byte keys + 2x 8-byte nonces) + session epoch (4 bytes) = 116 bytes. New gateway resumes Noise session without re-handshake."

### M-7. Convergence: what triggers "all expected pairs"? [AMBIGUOUS]

Section 7.1 entry says "Front 1 waits until the wavefront deadline expires or all expected pairs reconnect." How does a peer know which pairs are "expected"? Is it all pairs that were connected before the partition? All pairs in the mesh? A peer that was never connected to certain clusters does not know they exist until after GSet merge.

**Minimum fix:** "Expected pairs = all cluster pairs present in the local GSet enrollment records at partition-heal time. Pairs discovered during GSet merge are added to the expected set."

### M-8. Gossip mode ACK: what message? [AMBIGUOUS]

Section 7.1 Exit Step 1 says "All peers ACK gossip_mode=0 within wavefront_timer." But the heartbeat gossip_mode flag is a 1-bit field in a periodic heartbeat, not a request/response. How does a peer "ACK" a mode switch? Does it just set the bit in its next heartbeat? Or is there a dedicated control message?

**Minimum fix:** "ACK = peer's next heartbeat carries gossip_mode=0. Admin considers the switch ACK'd when it has received a heartbeat with gossip_mode=0 from each gateway."

### M-9. PartitionAbsence: excluded from dominance but how? [AMBIGUOUS]

Section 5.1 says `PartitionAbsence` excluded from dominance score, and Section 2.3 shows the dominance formula uses `weighted_scar_count`. Does "excluded" mean PartitionAbsence scars are filtered out before computing weighted_scar_count? Or is there a separate sum?

**Minimum fix:** "`weighted_scar_count` for dominance computation excludes all ScarAttribution::PartitionAbsence entries. The same scar GSet is used but with a filtered iterator."

### M-10. CIK rotation: dual acceptance window details [MISSING-ERROR]

Section 2.3 says "accept both CIKs for 48 hours" but does not specify: how does a peer know which CIK to use for outbound? Does it try the new CIK first? What if the remote peer has not rotated yet? What if the 48-hour window expires and some peers have not migrated?

**Minimum fix:** "Outbound: use new CIK. Inbound: try new CIK first, fall back to old CIK. After 48 hours: reject old CIK, peers still using old CIK must re-enroll."

## 4. verkko-relay gaps

### R-1. SIGNAL operation: undefined [BLOCKING]

Section 4.3 lists five relay operations: PUT, GET, DELETE, TOUCH, SIGNAL ("connection setup"). SIGNAL is never defined. What does it do? What is its wire format? When is it sent? This is one of five operations and it is completely unspecified.

**Minimum fix:** "SIGNAL initiates a relay-mediated connection between two peers. Format: `[u32 source_conn_id][u32 dest_cluster_hash][u8[32] encrypted_hint]`. The relay forwards the signal to the destination. The destination responds with a SIGNAL_ACK containing its address for hole-punching."

### R-2. Relay topology selection algorithm [BLOCKING]

Section defines RelayTopology but the body has no algorithm for selecting which peer relays for which. How is a relay path chosen? Shortest health-weighted path? Is there a routing table? How many hops are allowed (the wire format has hop_count, max 4)?

**Minimum fix:** "Each gateway maintains a relay table: for each unreachable cluster, the best relay is the connected cluster with highest health score toward the destination. Hop limit: 2 (direct or one relay). Relay selection re-evaluated on each heartbeat."

### R-3. Transit relay: forwarding semantics [BLOCKING]

The spec says the relay "forwards frames without decrypting content" but does not specify: does the relay decrement hop_count? Does it rewrite any header fields? Does it buffer or forward immediately? What if the destination is not connected to the relay?

**Minimum fix:** "Relay decrements hop_count by 1. If hop_count reaches 0, frame is dropped. Relay forwards immediately (no buffering). If destination is not connected, relay returns a RELAY_UNREACHABLE control frame to the source."

### R-4. DELETE operation: when is it called? [MISSING-ERROR]

Section 4.3 lists DELETE as a relay operation but never specifies when or why a peer would delete a blob from the relay. After successful re-encryption? After TTL expiry? Is it automatic or explicit?

**Minimum fix:** "DELETE is called: (1) after successful re-encryption under a new key (old blob deleted), (2) on TTL expiry (relay auto-deletes, no peer action), (3) on admin command (channel revocation cleanup)."

### R-5. Relay authentication [BLOCKING]

The spec does not say how a peer authenticates to a relay. Is the relay a peer in the mesh? Is there a separate auth mechanism? How does a peer discover which relay to use? The abstract says "a relay peer is not a full mesh participant" but then what is its identity?

**Minimum fix:** "Relay authenticates via Noise XX with a relay-specific static key published in the mesh control log. The relay is not enrolled as a peer; it holds no channel keys. Authentication provides confidentiality of relay_key_ids from network observers."

### R-6. Relay storage: capacity limits and back-pressure [MISSING-ERROR]

Section 4.3 specifies PUT/GET/DELETE/TOUCH/SIGNAL but says nothing about what happens when the relay runs out of storage. Does it reject PUTs? With what error? Is there back-pressure to the mesh?

**Minimum fix:** "When relay storage exceeds 90% capacity, PUT returns RELAY_FULL. Peer retries on alternate relay. Relay evicts expired blobs (TTL) before rejecting."

## 5. verkko-seat gaps

### S-1. Authentication mechanism [BLOCKING]

Section says "seat authenticates with a user credential (not a DK/DEK)" but does not specify what a "user credential" is. Password? Token? Certificate? How is it verified? What is the session establishment flow?

**Minimum fix:** Define: "Seat authenticates via Noise NK (seat knows host peer's public key, host peer does not know seat). Seat sends a credential payload in the first post-handshake message: `[u8 cred_type][u8[] credential]`. cred_type 0x01 = password (Argon2id-hashed, compared to stored hash). cred_type 0x02 = resumption token."

### S-2. Session establishment flow [BLOCKING]

No handshake flow is specified. How does the seat discover the host peer's address? How does it know the host peer's static key? What if the host peer has rotated its keys? What is the complete message sequence from "seat opens socket" to "seat can send requests"?

**Minimum fix:** Define a 3-step flow: (1) seat resolves host via mDNS or cached address, (2) Noise NK handshake (seat is initiator), (3) post-handshake: seat sends auth credential + schema hash, host responds with auth result + schema (if needed) + subscription catalog.

### S-3. Request types [BLOCKING]

SeatRequest says "typed via protobuf schema" but no request types are defined. What can a seat ask for? The implementer needs at minimum: query entities, trigger download, get status, list content, subscribe to events. Without defined request types, the seat protocol is unusable.

**Minimum fix:** Define a minimum request type enum: `{ Query = 1, Download = 2, Status = 3, Subscribe = 4, Unsubscribe = 5, InstructionResult = 6 }`.

### S-4. Event categories [BLOCKING]

SeatEvent says "seat subscribes to event categories during session setup" but no categories are defined. What events can a seat receive?

**Minimum fix:** Define: `{ ContentChanged = 1, DownloadProgress = 2, MeshStatus = 3, Instruction = 4, SettingsChanged = 5 }`.

### S-5. Error codes [BLOCKING]

No error codes are defined for any seat operation. What does the host return when a request fails? When auth fails? When the seat's role does not permit an operation?

**Minimum fix:** Define: `{ Ok = 0, AuthFailed = 1, Forbidden = 2, NotFound = 3, RateLimited = 4, InternalError = 5, MeshUnavailable = 6 }`.

### S-6. Disconnection semantics [MISSING-ERROR]

No disconnection protocol. How does a seat gracefully disconnect? How does the host detect seat death (no heartbeats from seat)? What happens to in-flight requests? To instruction state?

**Minimum fix:** "Seat sends DISCONNECT frame. Host cleans up session after 30-second idle timeout. In-flight requests are abandoned. Instructions in progress continue; seat receives results on reconnect via instruction log."

### S-7. Seat heartbeat [AMBIGUOUS]

verkko-protocol defines heartbeats between peers but verkko-seat does not reference heartbeats. Does the seat send heartbeats? If not, how does the host detect liveness?

**Minimum fix:** "Seat sends a lightweight heartbeat (8 bytes: connection_id + sequence) every 30 seconds. Host considers seat dead after 3 missed heartbeats (90 seconds)."

### S-8. Multiple seats: resource limits [MISSING-ERROR]

"Multiple seats can connect to the same host peer simultaneously" but no limit is specified. 100 seats? 1000? What happens when the limit is reached?

**Minimum fix:** "Max 16 concurrent seats per host peer. 17th connection attempt receives AUTH_REJECTED with reason CAPACITY. Configurable via settings."

### S-9. Role-to-access mapping [BLOCKING]

Section says "the host peer maps the role to mesh-level access control" but the mapping is not defined. What can Admin do that User cannot? What can User do that Child cannot? The architecture docs say roles are `Admin, User, Child` but the seat doc does not map these to capabilities.

**Minimum fix:** "Admin: all operations. User: query, download, subscribe. Cannot trigger revocations or enrollments. Child: query, subscribe. Cannot download. Cannot view content marked as restricted."

### S-10. Instruction lifecycle [AMBIGUOUS]

SeatInstruction lists instruction types (download, place, link, remove, verify, inventory sync) but does not define: what is the wire format? How does the host assign instruction IDs? What are the completion states? Can an instruction be cancelled?

**Minimum fix:** Define: `Instruction { id: u64, type: u8, payload: bytes, state: Pending|InProgress|Complete|Failed }`. Host assigns monotonic IDs. Seat reports state transitions. Cancel = host sends Cancel instruction with the original ID.

## 6. verkko-matter gaps

### T-1. Matter transfer protocol: no specification [BLOCKING]

Section says "LAN path: direct streaming with BLAKE3 verification at end. WAN path: bao-tree verified streaming (chunk-level integrity)." But there is no wire protocol. How does a peer request a transfer? What message initiates it? What is the chunk size for bao streaming? How does flow control work? How does multi-source sequential fallback work?

**Minimum fix:** Define: (a) REQUEST_MATTER message with blob_id, (b) MATTER_DATA stream with chunk_index and chunk data, (c) chunk size = 64 KiB (bao default), (d) flow control via LEDBAT on the bulk queue, (e) multi-source: on timeout (10s) or integrity failure, try next source from source selection list.

### T-2. Replication: no protocol [BLOCKING]

Section defines Replication with "configurable min_replicas and target_replicas" but does not specify: how does a peer know the replication count for a piece of matter? How does it select targets for replication? How does it detect under-replication? Is there a replication coordinator?

**Minimum fix:** "Replication count is tracked via cuckoo filter: count of clusters advertising the blob_id. Under-replication detected when count < min_replicas. Target selection: peer with lowest storage utilization and highest health. No coordinator; each cluster independently maintains its own replicas. Replication uses the standard matter transfer protocol."

### T-3. Sync reconciliation: merge rules not specified [BLOCKING]

Section says "merge rules: field-level, deterministic, reproducible on all peers" but no merge rules are defined. What fields? What deterministic resolution? LWW? Vector clocks? This is the core of distributed state consistency.

**Minimum fix:** "Sync reconciliation applies to entity metadata (not matter content, which is immutable). Merge: LWW per field with HLC timestamp. Tie-break: higher DK_pub lexicographic. Deletions: tombstone with 90-day TTL."

### T-4. Stash: persistence and recovery [AMBIGUOUS]

Section defines Stash with status enum (Present, Downloading, Missing, Corrupt) but does not specify: where is the stash stored? SQLite? File? How is it recovered after crash? What if the filesystem has files that the stash does not know about?

**Minimum fix:** "Stash is a SQLite table: `(blob_id BLOB PRIMARY KEY, path TEXT, size INTEGER, last_verified INTEGER, status INTEGER)`. On startup: scan filesystem, reconcile with stash (mark Missing for absent files, add unknown files as Present pending verification)."

### T-5. BandwidthPolicy: wire format and enforcement [AMBIGUOUS]

Section says "policies published via gossip (time windows, rate limits)" but does not define the policy format. What does a policy look like? How is it serialized? How is it published? Is it a control log entry?

**Minimum fix:** "BandwidthPolicy = `{ source_cluster: u32, target_cluster: u32, window_start: u32 (seconds since midnight), window_end: u32, max_rate_kbps: u16 }`. Published via gossip as a control log entry."

### T-6. SealedRange: UnsealGrant format [AMBIGUOUS]

Section 2.8 says admin can issue UnsealGrant for Absence sealed ranges but does not define what an UnsealGrant message contains or how it is transmitted.

**Minimum fix:** "UnsealGrant is a KeyBundle(Grant) for the sealed epoch range. Transmitted via the standard KeyBundle distribution mechanism."

### T-7. Content routing: request wire format [BLOCKING]

Section 4.2 defines source selection logic but not the wire format for content routing requests. How does a peer ask "who has blob_id X?" Is it a gossip query? A direct request to all clusters? A lookup in the local cuckoo filter?

**Minimum fix:** "Content routing is local: peer checks its own cuckoo filters (one per cluster) for the blob_id. No network request needed. If multiple clusters have it, source selection (Section 4.2) picks the best. Requesting peer sends REQUEST_MATTER directly to the selected source peer."

### T-8. Download coordination from external sources [BLOCKING]

Section says "Mesh-first: local peer -> mesh peers -> external source. External sources are last resort." But when all mesh sources fail, how does a peer download from an external source? Is this the connector system from the saalis architecture? How does the mesh know the external source URL? How is the download assigned?

**Minimum fix:** "External source download is delegated to the saalis scheduler (out of scope for verkko). Verkko provides a callback: `on_matter_unavailable(blob_id)` that the host application handles."

## 7. verkko-ops gaps

### O-1. No gaps in verkko-ops itself. It is an implementation guide.

verkko-ops is a consumer of all design documents and defines no concepts. Its gaps are downstream of the other documents' gaps. The build order, calibration items, and game day exercises are all well-specified contingent on the other documents being complete.

## 8. Cross-document gaps

### X-1. Mesh-to-relay re-encryption trigger: exact interface [BLOCKING]

verkko-mesh Section 7.1 Front 3 says "re-encryption performed per verkko-relay: TerritorialReencryption." verkko-relay specifies the re-encryption mechanism. But the exact interface between mesh and relay is unspecified. What message does the mesh send to trigger re-encryption? What acknowledgement does the relay return? How does the mesh know re-encryption is complete?

**Minimum fix:** "Mesh sends RE_ENCRYPT_BATCH to relay with: `[u32 epoch_new][u8[32] channel_key_new][u32[] blob_ids]`. Relay processes and responds with RE_ENCRYPT_ACK per blob. Gate opens when all ACKs received or deadline expires."

### X-2. KeyBundle delivery across document boundaries [AMBIGUOUS]

verkko-mesh defines KeyBundle. verkko-crypto defines the key material. verkko-relay needs new keys for re-encryption. verkko-matter needs new keys for sealed-range catch-up. But the delivery path from "admin creates KeyBundle" to "relay has new key" to "matter can decrypt" is never spelled out as an end-to-end flow.

**Minimum fix:** "KeyBundle delivery path: admin creates KeyBundle -> gossip (reliable control channel) -> all peers receive -> peers update local key state -> peers notify relay module (local) -> relay begins re-encryption. For sealed-range: KeyBundle(Grant) delivers keys -> matter module updates sealed_range state -> matter begins catch-up decryption."

### X-3. Convergence and matter: sealed-range ordering [AMBIGUOUS]

verkko-mesh Section 2.6 says "sealed-range catch-up completes before CPSK re-seed." verkko-matter Section 2.8 defines sealed-range catch-up. But the interaction is underspecified: who initiates the catch-up? The mesh layer? The matter layer? How does the polygon schedule know when catch-up is complete?

**Minimum fix:** "Mesh layer initiates sealed-range catch-up by calling matter::catch_up(peer, channel, epoch_range). Matter layer responds with catch_up_complete when all sealed content is either fast-forwarded or suspended. Polygon schedule defers CPSK re-seed for the pair until catch_up_complete."

### X-4. Relay and protocol: frame forwarding path [BLOCKING]

verkko-protocol defines frames. verkko-relay defines transit forwarding. But the mechanism for a peer to tell the relay "forward this frame" is unspecified. Is it a special frame type? A separate protocol? How does the relay know where to forward?

**Minimum fix:** "Transit relay uses the standard frame format. The relay examines the outer header `destination` field (BLAKE3(CIK)[0..4]), looks up the connection to that cluster, and forwards the frame on that connection. No special frame type needed."

### X-5. Seat and matter: how does a seat trigger a download? [AMBIGUOUS]

verkko-seat defines SeatRequest and SeatInstruction (download type). verkko-matter defines matter transfer. But how does a seat's download request become a matter transfer? Does the host peer create a scheduler work unit? Does it directly initiate the transfer?

**Minimum fix:** "Host peer receives SeatRequest(Download, blob_id). Host checks source selection. If available in mesh, initiates matter transfer. Sends SeatEvent(DownloadProgress) as chunks arrive. On completion, sends SeatInstruction(place, local_path) to the seat."

### X-6. Protocol and mesh: how does mesh learn about new connections? [AMBIGUOUS]

verkko-protocol is sans-IO and manages connections. verkko-mesh needs to know when connections form and drop. What is the callback interface? The protocol layer does not define events or callbacks.

**Minimum fix:** "Protocol layer emits events: `ConnectionEstablished(peer_id, connection_id)`, `ConnectionLost(peer_id, reason)`, `HandshakeComplete(peer_id, session_type)`. Mesh layer consumes these events via a trait callback."

### X-7. Version negotiation across all documents [BLOCKING]

No document specifies protocol version negotiation. What happens when a v1 peer connects to a v2 peer? How are incompatible changes detected? The wire format has no version field (except BroadcastEnvelope's `version: u8`). The Noise prologue could include a version, but this is not specified.

**Minimum fix:** "Version negotiation occurs in the Noise prologue: `'saalis-verkko-v' || u8_version`. Incompatible version: handshake fails with PROTOCOL_VERSION_MISMATCH. Compatible: continue with the lower version's capabilities."

## 10. The matter gap (minimum viable specs)

### Transfer protocol (BLOCKING)

Needed: request message, response stream format, chunk size, chunk verification (bao integration), flow control (LEDBAT on bulk queue), multi-source fallback, timeout handling, integrity failure scar attribution, cancellation.

### Replication (BLOCKING for durability)

Needed: under-replication detection (cuckoo filter count vs min_replicas), target selection algorithm, replication scheduling (background work unit), replication as matter transfer, deduplication (content-addressed so idempotent).

### Sync reconciliation (BLOCKING for multi-writer scenarios)

Needed: what state is synced (entity metadata, not matter content), merge algorithm (LWW or CRDT), conflict resolution (deterministic), tombstone handling, reconciliation trigger (Merkle root divergence in heartbeat).

### Stash management (deferrable but needed for correctness)

Needed: persistence format, startup reconciliation, garbage collection of orphaned files, verification scheduling.

## 12. Version negotiation assessment

**Is the absence of version negotiation blocking?** Yes, but only for the second release. For the very first deployment (all peers are v1), there is nothing to negotiate. However, the wire format must include a version field FROM v1, or the first upgrade will require a coordinated flag-day deployment of all peers, which is operationally unacceptable for a household mesh.

**Minimum fix for v1:** Add version to the Noise prologue. Add version to the outer frame header (steal 1 byte from padding or extend header by 1 byte). Document the forward-compatibility rule: unknown fields are ignored, unknown message types are dropped with a log entry.
