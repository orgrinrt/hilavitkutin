**Assessment:** The crypto territory was already the strongest and remains so. The BLAKE2b lock and DomainSeparator cleanup are minor refinements to an already solid map. The remaining blanks are implementation-level details that do not block any downstream territory.

### 1.2 verkko-protocol: Substantially Mapped (Major Improvement)

This territory underwent the largest transformation in Run 11. In Stage 1 I called it "lightly mapped" with major roads but no port. The port is now under construction.

**Newly mapped:**
- **Connection ID generation:** random_u32, with reserved values 0x00000000 and 0xFFFFFFFF. The two-phase ID scheme is eliminated. The WireEng's handshake frame simplifies to 17 bytes fixed header. No layer coupling to mesh (no CIK dependency). Territory simplified.
- **Version negotiation:** Pre-handshake VersionProbe/VersionResponse, 24/20 bytes respectively. Prologue binding at 7 bytes (MAGIC + selected_version + hs_type). Rate limiting at 100 responses/second. Downgrade detection via prologue mismatch causing handshake failure with retry. Territory settled.
- **Connection teardown:** CONN_CLOSE (msg_type 0x10), 8-byte header, 9 reason codes. Two-phase close handshake with timer (min(2*RTT, 5000ms)). CLOSING state behavior specified (process SACKs and CONN_CLOSE, discard other frames). Abrupt close path for fatal violations. Heartbeat timeout close path. This was blank B12; it is now mapped.
- **Session resumption:** 80-byte plaintext token, 120-byte encrypted. Resumption handshake via hs_type 0x03. 0-RTT safety constraints (only idempotent msg_types in early_data). Fresh token issued in accept frame. Ticket counter prevents replay. Territory new and well-specified.
- **Corrected frame budget:** UDP payload limit fixed at 1232 bytes. All frame size calculations recalculated globally. Outer header 29 bytes, inner header 10 bytes, stream header 7 bytes, AEAD tag 16 bytes. Maximum data frame payload: 1170 bytes. All experts use consistent numbers.
- **Stream header:** 7 bytes with flags byte (FIN, RST, SACK_PRESENT). Stream lifecycle flags now exist where before they were missing (WireEng erratum E-WE-3 resolved).
- **Metabolic state machine:** MetabolicGovernor abstract contract. Protocol defines the state machine; mesh provides the concrete governor. ResourcePressures is opaque to protocol. The city that straddled the border now has a checkpoint.
- **SansIO event surface (partial):** The SpecWriter accepted the partial B1 scope. ProtocolOutput variants mapped to wire format types (the WireEng's W-SA-1 approach). ProtocolInput enumeration accepted (BytesReceived, TimerExpired, ConnectionRequested, ConnectionClosed, StreamCreated, StreamClosed, HeartbeatTick, ConfigChanged). Polling model and buffer ownership deferred to Run 12.

**Still blank:**
- **SansIO polling model:** How the consumer drives the state machine. The event enumerations exist but the calling convention does not. An implementer knows what goes in and what comes out but not how to pump the machine.
- **SansIO buffer ownership:** Transient arena vs persistent state. The SpecWriter acknowledged the distinction but deferred it.
- **MemoryProvider contract:** No action in Run 11 covers this.
- **Reliability mode selection per stream:** Reliable ordered vs unreliable datagram. Still unspecified how a stream selects its mode.
- **SACK hybrid format:** The merger of SpecWriter's recv_window with WireEng's timestamps is accepted in principle but the concrete merged wire format is pending. The WireEng awaits the SpecWriter's field ordering confirmation.

**Assessment:** The protocol territory went from "lightly mapped" to "substantially mapped." The port is partially constructed: ships know what cargo types exist (event enumerations) but the docking procedure (polling model) is missing. The most important remaining blank is the SansIO polling model, which is a Run 12 topic by consensus. The SACK merger is a convergence task, not a design gap.

### 1.6 verkko-matter: Partially Mapped (Major Improvement)

- **Bao proof size analysis:** The WireEng produced comprehensive analysis at 512/1024/2048-byte chunk sizes across blob sizes from 1MB to 2GB. **Assessment:** The matter territory went from "lightly mapped" to "partially mapped." The chunking protocol is the bridge I identified as missing in Stage 1 (my N3 mechanism). ## 5. Remaining Dangers

The most dangerous remaining blank regions after Run 11's work is applied, ranked by severity.

### Danger 1: SansIO Polling Model (B1 remainder) -- HIGH

**What is missing:** The consumer does not know how to drive the state machine. The event enumerations (ProtocolInput, ProtocolOutput) tell the consumer what events exist. They do not tell the consumer the calling convention.

**Why it is dangerous:** Without the polling model, two implementations will make different assumptions. One might use a poll-and-dispatch loop. Another might use callbacks. A third might use async/await with a channel. These are not interop-affecting (the wire format is shared), but they create implementation fragmentation that makes testing, debugging, and library sharing difficult.

**Why it is not critical:** The wire formats are specified. Two implementations that use different polling models will still produce and consume identical bytes on the wire. The interoperability is preserved at the wire level. The fragmentation is at the integration level.

**My assessment:** This is the most important thing to specify in Run 12. The partial B1 in Run 11 (event surface) was the right scoping. The polling model requires design decisions about buffer lifetime, arena allocation strategy, and error propagation that deserve a focused specification effort.

### Danger 2: Intra-Cluster Communication (B2) -- HIGH

**What is missing:** How peers within a cluster communicate with each other. Session model, peer addressing, gateway routing.

**Why it is dangerous:** The mesh specification assumes cluster peers can communicate but never says how. The wire format supports it (dest_peer field). The crypto supports it (CIK as Noise KK static key). But the session model and addressing convention are unspecified. Two implementations could assign dest_peer values differently and fail to route intra-cluster traffic.

**Why it is not critical:** For a household mesh with 2-6 peers per cluster, the intra-cluster communication path is simple. The gateway aggregates health observations. Non-gateway peers send directly to the gateway. The session count within a cluster is small (5-15 sessions for a 6-peer cluster). The design decisions are constrained by the small scale.

**My assessment:** This is the second most important thing to address in Run 12. A design round (not a specification task) is needed to decide CIK vs DEK for Noise KK, dest_peer assignment convention, and gateway routing model.

### Danger 3: Chunk Size Calibration -- MEDIUM

**What is missing:** The final decision between 512-byte and 1024-byte bao chunks.

**Why it is dangerous:** The WireEng's efficiency analysis shows ~40% wire efficiency for 1024-byte chunks at the worst case (2GB blob). The SpecWriter recommends 512-byte default. The WireEng chose 1024-byte standard. The choice affects every matter transfer's frame count, throughput, and proof overhead.

**Why it is not critical:** The TransferBegin message includes a chunk_size field, allowing the source to communicate the chunk size. If the default is wrong, implementations can negotiate. The bao standard uses 1024-byte chunks, which argues for consistency with existing tooling.

**My assessment:** The WireEng's choice of 1024-byte chunks (bao standard) with stream-layer reassembly is the correct default. The 40% efficiency is dominated by bao proofs, not by chunk size; switching to 512-byte chunks does not significantly improve efficiency because the proof overhead grows (more chunks means more proof per chunk for small blobs, and the logarithmic proof growth means larger blobs are similar either way). The proof batching optimization (v2) is the real efficiency lever.

### Danger 4: SACK Format Merger -- MEDIUM

**What is missing:** The final merged SACK wire format incorporating both experts' fields.

**Why it is dangerous:** SACK is the reliability backbone. Without a single canonical format, the heartbeat/congestion control/flow control loop cannot be fully implemented. The SpecWriter's recv_window (flow control) and the WireEng's ack_timestamp/ack_delay (LEDBAT RTT measurement) are both necessary.

**Why it is not critical:** Both experts agree the merger should include all fields. The task is mechanical (combine two known field sets into one wire format). No design decisions remain. This is a convergence task.

**My assessment:** This should be completed in the convergence stage that follows Run 11. The WireEng awaits the SpecWriter's field ordering confirmation. Once confirmed, the merged format is straightforward.

### Danger 5: Relay Territory (General) -- MEDIUM

**What is missing:** Transit forwarding, relay authentication, multi-hop forwarding, relay failover.

**Why it is dangerous:** The relay is the mechanism by which peers that cannot directly communicate reach each other. For NAT-traversed household networks, the relay is essential. Without the transit forwarding specification, the relay's core function is unimplemented.

**Why it is not critical:** Run 10 action P2-4 covers transit forwarding. The relay membrane model (5 operations) is well-specified. The SIGNAL operation types (HELLO, PUNCH, REDIRECT) are specified by the WireEng. The relay territory's blanks are body text within a correct topological structure.

**My assessment:** The relay territory needs focused attention in Run 12 or Run 13. It was correctly deprioritized in Run 11 relative to B1, B5, and B12, but it cannot be deferred indefinitely.

### Danger 6: Seat Failover (B8) -- LOW-MEDIUM

**What is missing:** What happens when the host peer goes offline. Reconnection to alternate peers, handoff protocol, state persistence.

**Why it is dangerous:** For user experience, a seat that loses its host peer and has no recovery path means the web UI goes blank until the host peer returns. For a media appliance in a household, this is a noticeable service disruption.

**Why it is not critical:** The seat is a thin client. It can reconnect to any other peer in the cluster that runs the host peer service. The reconnection is a new Noise handshake to a different host peer's WebSocket endpoint. The seat's state is minimal (session token, current view). The main question is whether the seat automatically discovers alternate host peers.

**My assessment:** Seat failover can be addressed as part of the seat body writing (P2-1 execution) without requiring a separate design round. The mechanism is simple: the seat knows the cluster's founding hash and the list of known host peers from the initial configuration. On disconnection, the seat tries the next host peer in the list.

### Danger 7: Matter Garbage Collection (B4) -- LOW

**What is missing:** When and how content is deleted. No deletion policy, no storage pressure response.

**Why it is dangerous in theory:** Unbounded storage growth. A device with limited storage will eventually fill up.

**Why it is low priority in practice:** The deployment target is a media appliance with dedicated storage. The initial content library is curated by the user. Storage pressure is a deployment concern, not a protocol concern. A simple LRU policy at the application layer addresses the immediate need.

## 8. The Cartographer's Final Note

Run 11 achieved what I hoped it would but did not expect it to: the experts engaged with each other's work, changed their positions when presented with better arguments, and converged on a substantially more complete map.

The SpecWriter's most important concession was accepting that a sans-IO protocol IS its consumer interface. This unlocked the partial B1 inclusion that gives the protocol its port. Without this concession, Run 11 would have produced precise interior maps of a territory that nobody could enter.

The WireEng's most important contribution was the matter chunk protocol wire format. This is the bridge I identified in Stage 1 as the third most dangerous blank. The bridge now has girders, a deck, and a toll booth.

The ArchSurgeon's most important contribution was the two-dimensional classification framework (DAG-structural + interface-complete). This replaces my "access-structural" terminology with more precise language. I adopt it. The practical result is identical: B1, B2, B5, B12 are distinguished as interface-incomplete gaps that block implementers more severely than content gaps like B4 or B6.

The territory is not complete. It will not be complete after Run 12 either. But the map is now accurate where it has content, honest about where it is blank, and precise about what the blanks contain. The remaining blank regions are correctly triaged: some need design rounds (B2, B3), some need focused specification (SansIO polling model, relay transit), and some need policy decisions (garbage collection, storage pressure). None of the remaining blanks are phantom borders or phantom dependencies. The topology is sound. The interfaces are identified. The roads between hilltops are being built.

The most dangerous thing I see looking at the full map is not any single blank region. It is the relay territory's general under-specification. Crypto, protocol, mesh, seat, and matter have all received multiple rounds of expert attention. Relay has received one: the base document plus boundary clarifications. The relay membrane model is strong but the transit forwarding core is still a concept diagram, not a specification. For a household mesh that will routinely traverse NAT boundaries, the relay is not optional. It is the trade route that makes the territory habitable. Run 12 should include a relay-focused work stream alongside the intra-cluster design round and SansIO completion.

The map is better. It is honest. The territory awaits its travelers.
