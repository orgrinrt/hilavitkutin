# Deep Dive: Testing Strategies for saalis

Research date: 2026-03-15

---

## 1. Unit Testing the Sync Core

saalis follows a sync-core/async-shell architecture: all domain logic -- work units, settings resolution, query builders, the columnar in-memory store, entity model -- runs synchronously. This is a testing advantage. Synchronous code can be tested with plain `#[test]` functions, no executor required, no `tokio::test` ceremony.

### Work units

Work units are the atomic operations the DAG scheduler dispatches. Each work unit implements a trait with associated types for its triggers and emits. Testing a work unit means:

1. Constructing the unit with test doubles for any context service handles it depends on.
2. Calling its `execute` method with a crafted trigger value.
3. Asserting the returned emits (or errors) match expectations.

Because work units are sync and take their dependencies via handle patterns (not `&dyn Fn` callbacks), test doubles are straightforward structs that implement the same handle trait. No mocking framework is strictly necessary, though [`mockall`](https://crates.io/crates/mockall) can reduce boilerplate for traits with many methods.

```rust
#[test]
fn enrich_work_unit_emits_metadata_write() {
    let ctx = TestContext::new();
    let unit = EnrichWorkUnit::new(ctx.handle());
    let trigger = Trigger::EntityDiscovered { id: EntityId::test(42) };
    let emits = unit.execute(trigger).unwrap();
    assert!(emits.iter().any(|e| matches!(e, Emit::WriteMetadata { .. })));
}
```

### Settings resolution

Settings resolution follows a layered cascade: default -> profile -> user override -> runtime. Each layer is a pure data structure. Testing the resolution logic means building a `SettingsStack` with known layers and asserting that lookups return the correct value at each precedence level. This is ideal for both unit tests and property-based tests (see section 6).

### Query builders

The `.of::<T>(id)` query API and its builder chain produce SQL or in-memory column lookups. Unit-test the builder by asserting on the generated SQL string (snapshot-test it with `insta`) or on the returned `QueryPlan` struct, without ever touching a database. The builder is a pure function from inputs to a query representation.

## 2. Integration Testing with SQLite

### In-memory vs tempfile databases

SQLite supports `:memory:` URIs for fully in-memory databases that vanish when the connection closes. This is the fastest option for tests that need a real schema but no persistence. For tests that exercise connection pooling (r2d2) or file-level locking, use [`tempfile::NamedTempFile`](https://docs.rs/tempfile) to create a throwaway `.sqlite` file that is automatically deleted when the `NamedTempFile` guard drops.

```rust
#[test]
fn migration_applies_cleanly() {
    let tmp = tempfile::NamedTempFile::new().unwrap();
    let conn = rusqlite::Connection::open(tmp.path()).unwrap();
    run_migrations(&conn).unwrap();
    // assert tables exist
    let tables: Vec<String> = conn
        .prepare("SELECT name FROM sqlite_master WHERE type='table'")
        .unwrap()
        .query_map([], |row| row.get(0))
        .unwrap()
        .collect::<Result<_, _>>()
        .unwrap();
    assert!(tables.contains(&"entities".to_string()));
}
```

### Fixture loading

Create a `fixtures/` directory under `tests/` containing SQL files that populate known data sets. A helper function opens a tempfile database, runs migrations, then executes the fixture SQL. Every integration test that needs data calls this helper, ensuring consistent starting state.

### Migration testing

The [`rusqlite_migration`](https://docs.rs/rusqlite_migration) crate supports loading migrations from `*.sql` files in a directory and applying them in order. For saalis, test two properties:

1. **Forward migration**: apply all migrations from scratch and assert the resulting schema matches expectations.
2. **Idempotency**: applying migrations to an already-migrated database is a no-op.

If reversible migrations are supported, also test round-trip: migrate up, migrate down, migrate up again, and assert schema equality.

### Hot-cold store integration

saalis uses an in-memory columnar store (hot) with SQLite as the cold store. Integration tests should verify the flush path: write to the hot store, trigger a flush work unit, and assert the data lands in SQLite. Conversely, test the load path: populate SQLite, trigger a hydration, and assert the hot store reflects the data.

## 3. Proc Macro Testing

saalis has two key proc macros: `#[register]` (attribute macro on impl blocks) and `#[derive(Settings)]` (derive macro on structs). Both live in the `saalis-derive` crate.

### trybuild for compile-fail tests

[`trybuild`](https://docs.rs/trybuild) (by dtolnay) is the standard tool for testing that a proc macro produces the right compiler errors. You write small `.rs` files that should fail to compile, and `.stderr` files containing the expected error output. trybuild invokes `rustc` on each file and compares the output.

```rust
// tests/ui.rs
#[test]
fn ui_tests() {
    let t = trybuild::TestCases::new();
    t.compile_fail("tests/ui/register-on-non-trait-impl.rs");
    t.compile_fail("tests/ui/register-missing-named.rs");
    t.pass("tests/ui/register-valid-connector.rs");
}
```

Key test cases for `#[register]`:

- Applied to an inherent impl (not a trait impl) -- should error with a clear message.
- Applied to an impl of a trait that is not registrable (no `HasDescriptor`) -- should error.
- Applied to a valid registrable trait impl -- should compile and produce the descriptor + `inventory::submit!`.

Set `TRYBUILD=overwrite` to regenerate `.stderr` files when error messages change (e.g., after a rustc upgrade).

### macrotest for expansion snapshots

[`macrotest`](https://docs.rs/macrotest) works like trybuild but compares the expanded output rather than compiler errors. It invokes `cargo expand` and diffs the result against `.expanded.rs` files. This is valuable for asserting the exact shape of generated descriptors, ensuring the `repr(C)` layout, field order, and `inventory::submit!` call are correct.

```rust
#[test]
fn expansion_tests() {
    macrotest::expand("tests/expand/*.rs");
}
```

### Unit-testing macro internals

The proc-macro-patterns deep dive established the canonical pattern: the `#[proc_macro_attribute]` entry point delegates to a fallible function `fn register_impl(attr: TokenStream2, item: TokenStream2) -> syn::Result<TokenStream2>`. This inner function can be unit-tested by constructing `TokenStream2` values from string literals via `syn::parse_str` or `quote!`, calling the function, and asserting on the output tokens. This does not require a compiler context because `proc_macro2` types are usable outside of macro expansion.

## 4. Plugin Testing

saalis plugins are shared libraries (`.so` on Linux, `.dylib` on macOS) that register `repr(C)` descriptors via `inventory::submit!` in platform constructor sections. Testing this system requires exercising the full `dlopen` path.

### Loading test plugins in integration tests

Build a minimal test plugin as a separate `cdylib` crate within the workspace (e.g., `test-plugins/dummy-connector/`). In integration tests, use [`libloading`](https://docs.rs/libloading) to load the compiled `.so`/`.dylib` and verify that `inventory::iter::<ConnectorDescriptor>()` yields the plugin's descriptor after loading.

```rust
#[test]
fn dummy_connector_registers_on_load() {
    let before = inventory::iter::<ConnectorDescriptor>().count();
    unsafe {
        let _lib = libloading::Library::new("target/debug/libdummy_connector.so").unwrap();
    }
    let after = inventory::iter::<ConnectorDescriptor>().count();
    assert_eq!(after, before + 1);
}
```

A CI build step must compile the test plugin crate before running integration tests. Use a `build.rs` or a Makefile/justfile target to sequence this.

### Mocking the plugin ABI

For unit tests that should not load real shared libraries, create a mock descriptor directly:

```rust
fn test_descriptor() -> ConnectorDescriptor {
    ConnectorDescriptor {
        name: c"test-connector",
        fetch: test_fetch as extern "C" fn(...) -> ...,
        // ...
    }
}
```

Submit it via `inventory::submit!` in the test module. Since `inventory` is process-global, use `#[serial_test::serial]` (from the [`serial_test`](https://crates.io/crates/serial_test) crate) for any tests that mutate or depend on the global registry to avoid inter-test interference.

### ABI stability testing

Because `repr(C)` descriptors cross the shared-library boundary, their layout must never silently change. Maintain a set of `static_assertions::assert_eq_size!` and `assert_eq_align!` checks in the SDK crate. Any descriptor field reorder or type change becomes a compile error.

## 5. Web UI Testing

saalis serves an HTMX-driven web UI via axum. Testing spans three layers: handler unit tests, HTMX response validation, and browser-level end-to-end tests.

### Testing axum handlers

Axum handlers and routers are `tower::Service` implementations. Use `tower::ServiceExt::oneshot` to send a synthetic `Request<Body>` and assert on the `Response`. No TCP listener needed.

```rust
#[tokio::test]
async fn library_page_returns_200() {
    let app = build_router(test_state());
    let req = Request::get("/library").body(Body::empty()).unwrap();
    let resp = app.oneshot(req).await.unwrap();
    assert_eq!(resp.status(), StatusCode::OK);
}
```

For handlers that read HTMX-specific headers (e.g., `HX-Request`, `HX-Target`), the [`axum-htmx`](https://docs.rs/axum-htmx) crate provides extractors and responders. In tests, set these headers on the request:

```rust
let req = Request::get("/library/partial")
    .header("HX-Request", "true")
    .header("HX-Target", "content-area")
    .body(Body::empty())
    .unwrap();
```

### HTMX response validation

HTMX responses are HTML fragments, not full pages. Validate them by:

1. **Snapshot testing** (see section 7): `insta::assert_snapshot!` on the response body string.
2. **Structural assertion**: parse the HTML fragment with [`scraper`](https://docs.rs/scraper) and assert on CSS selectors. For example, verify that a partial response contains a `<div id="entity-list">` with the expected number of child elements.
3. **Header assertion**: HTMX responses often include `HX-Trigger`, `HX-Retarget`, or `HX-Push-Url` headers. Assert on these directly from the `Response`.

### Browser-level E2E tests with Playwright

For full end-to-end testing, [`playwright-rust`](https://crates.io/crates/playwright-rs) (v0.8.4, March 2026) provides Rust bindings to Playwright's browser automation. Alternatively, run Playwright tests in JavaScript/TypeScript against the running saalis binary -- this is the more mature path, since the Node.js Playwright library is production-ready while the Rust bindings are still stabilizing.

E2E test structure:

1. Spawn the saalis binary with a test database (tempfile SQLite, known fixtures).
2. Wait for the HTTP port to be ready.
3. Use Playwright to navigate, click, and assert on page content.
4. Tear down the process.

For HTMX-specific assertions, Playwright can wait for HTMX swap events by evaluating JavaScript:

```javascript
await page.evaluate(() => {
    return new Promise(resolve => {
        document.body.addEventListener('htmx:afterSwap', resolve, { once: true });
    });
});
```

Given the overhead, limit E2E tests to critical user journeys: login, browse library, trigger enrichment, view entity detail, change settings.

## 6. Property-Based Testing

[`proptest`](https://github.com/proptest-rs/proptest) is the preferred property-based testing framework for saalis. Unlike `quickcheck`, proptest defines generation strategies per-value rather than per-type, which is more flexible for testing complex domain types. Proptest also provides superior shrinking: it is strategy-aware and avoids generating values that violate constraints.

### Settings resolution

The settings cascade is a prime candidate. Properties to test:

- **Override dominance**: for any key, a higher-precedence layer always wins.
- **Absent key passthrough**: if a key is absent in a higher layer, the lower layer's value is returned.
- **Type preservation**: the resolved type matches the schema-declared type regardless of input combinations.

```rust
proptest! {
    #[test]
    fn higher_layer_wins(
        default_val in any::<i32>(),
        override_val in any::<i32>(),
    ) {
        let stack = SettingsStack::new()
            .with_default("key", default_val)
            .with_override("key", override_val);
        prop_assert_eq!(stack.resolve("key"), Some(override_val));
    }
}
```

### Query builder correctness

Generate random combinations of filters, orderings, and limits. Assert:

- The generated SQL is syntactically valid (parse it with a SQL parser or execute it against an in-memory SQLite).
- Idempotency: building the same query twice produces identical output.
- Commutativity of filters: `.where_eq("a", 1).where_eq("b", 2)` produces the same result set as `.where_eq("b", 2).where_eq("a", 1)`.

### Entity model invariants

Generate arbitrary `EntityId` values and metadata maps. Assert that store -> retrieve round-trips preserve all data, that entity type registrations never collide, and that the `Named::NAME` uniqueness constraint holds across generated types.

## 7. Snapshot Testing

[`insta`](https://insta.rs/) (by Armin Ronacher, v1.64+) provides snapshot testing with excellent ergonomics. Snapshots are stored as `.snap` files alongside tests.

### HTML output

For template rendering, snapshot the rendered HTML:

```rust
#[test]
fn entity_card_renders_correctly() {
    let entity = test_entity();
    let html = render_entity_card(&entity);
    insta::assert_snapshot!(html);
}
```

On first run, insta writes the snapshot. On subsequent runs, it diffs against the stored snapshot. Use `cargo insta review` to interactively accept or reject changes.

### API / handler responses

Snapshot the response body of axum handlers:

```rust
#[tokio::test]
async fn api_entity_response_snapshot() {
    let app = build_router(test_state());
    let req = Request::get("/api/entity/42").body(Body::empty()).unwrap();
    let resp = app.oneshot(req).await.unwrap();
    let body = axum::body::to_bytes(resp.into_body(), usize::MAX).await.unwrap();
    let json: serde_json::Value = serde_json::from_slice(&body).unwrap();
    insta::assert_json_snapshot!(json);
}
```

### Query plans and SQL output

Snapshot the SQL generated by query builders. This catches unintended changes to query shape:

```rust
#[test]
fn query_builder_sql_snapshot() {
    let sql = Query::of::<Game>(EntityId::test(1))
        .with_metadata::<GameMetadata>()
        .order_by("name")
        .build_sql();
    insta::assert_snapshot!(sql);
}
```

### Settings with insta's redaction

Use `insta::Settings` to redact volatile values (timestamps, generated IDs) before snapshotting:

```rust
let mut settings = insta::Settings::clone_current();
settings.add_filter(r"\d{4}-\d{2}-\d{2}T", "[TIMESTAMP]");
settings.bind(|| {
    insta::assert_snapshot!(render_audit_log(&entries));
});
```

## 8. Benchmark Testing

### Criterion

[`criterion`](https://github.com/bheisler/criterion.rs) is the established statistics-driven benchmarking library. It runs benchmarks multiple times, computes confidence intervals, and reports whether changes are statistically significant. For saalis, benchmark:

- **Column operations**: insert, scan, filter on the in-memory columnar store. These are the hottest paths during enrichment and query.
- **Morsel processing**: if the system processes data in morsels/batches, benchmark per-morsel throughput.
- **Settings resolution**: deep cascade with many layers.
- **Template rendering**: maud/askama rendering of complex entity pages.

```rust
fn bench_column_scan(c: &mut Criterion) {
    let store = create_test_store(10_000);
    c.bench_function("scan_10k_entities", |b| {
        b.iter(|| store.scan::<GameMetadata>(black_box(Filter::All)))
    });
}
criterion_group!(benches, bench_column_scan);
criterion_main!(benches);
```

### Divan

[`divan`](https://github.com/nvzqz/divan) is a newer alternative (1000+ GitHub stars, February 2025 CodSpeed integration) with a simpler attribute-based API and built-in allocation counting. It is more ergonomic for parameterized benchmarks:

```rust
#[divan::bench(args = [100, 1_000, 10_000])]
fn column_insert(bencher: divan::Bencher, n: usize) {
    bencher.bench(|| insert_n_entities(n));
}
```

Divan's allocation tracking is particularly useful for saalis, where the sync core should minimize allocations. Use `divan::AllocProfiler` to detect unexpected heap usage in hot paths.

### Recommendation

Use **criterion** for tracked regressions in CI (it generates machine-readable JSON output that [`bencher`](https://bencher.dev/) can ingest for continuous benchmarking). Use **divan** for developer-facing micro-benchmarks during development, where its ergonomics shine.

## 9. Fuzzing

### cargo-fuzz (libFuzzer)

[`cargo-fuzz`](https://github.com/rust-fuzz/cargo-fuzz) uses LLVM's libFuzzer for coverage-guided fuzzing. It is the de facto choice for Rust projects. For saalis, fuzz:

- **DAT file parsing**: No-Intro and Redump DAT files are XML. Feed arbitrary bytes to the parser and assert it never panics.
- **gamelist.xml parsing**: EmulationStation gamelist files from the wild can be malformed. Fuzz the parser.
- **API response parsing**: JSON responses from IGDB, ScreenScraper, and other sources. Fuzz the deserialization path.
- **Settings file parsing**: TOML/YAML config files from user input.

```rust
// fuzz/fuzz_targets/dat_parser.rs
#![no_main]
use libfuzzer_sys::fuzz_target;

fuzz_target!(|data: &[u8]| {
    if let Ok(s) = std::str::from_utf8(data) {
        let _ = saalis_core::parsers::parse_dat(s);
    }
});
```

### AFL (afl.rs)

[`afl.rs`](https://github.com/rust-fuzz/afl.rs) wraps American Fuzzy Lop. It is an alternative to cargo-fuzz with different coverage instrumentation and mutation strategies. AFL excels at finding different classes of bugs than libFuzzer. If resources permit, run both.

### Corpus management

Seed the fuzzer with real-world examples: actual DAT files, gamelist.xml files, API response JSON. Store these in `fuzz/corpus/<target>/`. The fuzzer mutates from this seed corpus, dramatically improving coverage compared to starting from random bytes.

### Structured fuzzing with Arbitrary

The [`arbitrary`](https://docs.rs/arbitrary) crate derives structured input generation. Instead of fuzzing raw bytes, derive `Arbitrary` for domain types and fuzz with typed inputs:

```rust
#[derive(arbitrary::Arbitrary)]
struct FuzzSettings {
    layers: Vec<(String, SettingValue)>,
    query_key: String,
}

fuzz_target!(|input: FuzzSettings| {
    let stack = build_stack(input.layers);
    let _ = stack.resolve(&input.query_key);
});
```

## 10. CI/CD Pipeline

### What to run in CI

Organize CI into tiers by speed:

| Tier | What | Time budget | Trigger |
|------|------|-------------|---------|
| 1 (fast) | `cargo check`, `cargo clippy`, `cargo fmt --check`, unit tests | < 5 min | Every push |
| 2 (standard) | Integration tests, proc macro UI tests, snapshot tests | < 15 min | Every push |
| 3 (extended) | Property-based tests (high iteration count), benchmark regression | < 30 min | PR merge to main |
| 4 (nightly) | Fuzzing (time-boxed to 30-60 min per target), full E2E with Playwright | < 2 hours | Nightly / weekly |

### Test matrix: ARM + x86

saalis targets Batocera on ARM (aarch64) and x86_64. Use the [`cross`](https://github.com/cross-rs/cross) tool for cross-compilation, which provides pre-built Docker images with the correct toolchains and sysroots. The CI matrix:

```yaml
strategy:
  matrix:
    target:
      - x86_64-unknown-linux-gnu    # primary dev
      - aarch64-unknown-linux-gnu   # batocera ARM
      - x86_64-unknown-linux-musl   # static binary
    include:
      - target: x86_64-unknown-linux-gnu
        runner: ubuntu-latest
        use_cross: false
      - target: aarch64-unknown-linux-gnu
        runner: ubuntu-latest
        use_cross: true
      - target: x86_64-unknown-linux-musl
        runner: ubuntu-latest
        use_cross: true
```

For ARM integration tests, use QEMU user-mode emulation (which `cross` sets up automatically). Note that SQLite with `bundled` feature compiles a C library, so cross-compilation needs a C cross-compiler -- `cross` handles this, but custom Docker images may be needed if additional native dependencies appear.

### Coverage: cargo-llvm-cov vs cargo-tarpaulin

[`cargo-llvm-cov`](https://github.com/taiki-e/cargo-llvm-cov) uses LLVM's native instrumentation and works on Linux, macOS, and Windows. It provides line, region, and branch coverage. It is the recommended tool for saalis because:

- It works on macOS (where development happens) and Linux (where CI runs).
- It supports merging coverage from multiple test runs (unit + integration + doc tests) into a single report.
- It outputs LCOV format for integration with Codecov or Coveralls.

[`cargo-tarpaulin`](https://github.com/xd009642/tarpaulin) is Linux-only but has been the longest-standing Rust coverage tool. It can use LLVM instrumentation as a backend (`--engine llvm`). For CI on Linux, either works; for local development on macOS, `cargo-llvm-cov` is the only option.

```yaml
# CI step
- name: Coverage
  run: |
    cargo llvm-cov --all-features --workspace --lcov --output-path lcov.info
    cargo llvm-cov --no-run --lcov --output-path lcov.info  # merge
- name: Upload coverage
  uses: codecov/codecov-action@v4
  with:
    files: lcov.info
```

### Additional CI checks

- **`cargo deny`**: audit dependencies for license violations, known vulnerabilities, and duplicate versions.
- **`cargo udeps`**: detect unused dependencies (important as the crate list grows).
- **`cargo semver-checks`**: when publishing SDK crates, verify semver compliance.
- **Snapshot review**: in CI, set `INSTA_UPDATE=no` so that any unreviewed snapshot change fails the build.

## 11. The Testing Pyramid for saalis

### Distribution

Applying the classic 70/20/10 heuristic, adjusted for saalis's architecture:

```
                    /\
                   /  \       E2E (Playwright)
                  / 5% \      ~10-15 tests: critical user journeys
                 /------\
                / 10%    \    Snapshot tests (insta)
               /  ~50-100 \   HTML output, SQL shape, API responses
              /------------\
             / 15%          \  Integration tests
            /  ~100-200      \  SQLite, plugin loading, hot-cold flush
           /------------------\
          / 60%                \  Unit tests
         /  ~500-1000           \  Work units, settings, query builders,
        /                        \  entity model, column operations
       /--------------------------\
      / 10%                        \  Property-based + fuzz
     /  ~30-50 properties,          \  Settings cascade, query builder,
    /   ~5-10 fuzz targets            \  parsers
   /------------------------------------\
```

### What is worth testing

**High value, low cost (always test):**
- Settings resolution cascade -- pure logic, easy to test, subtle bugs.
- Query builder SQL generation -- snapshot tests catch regressions cheaply.
- `repr(C)` descriptor layout -- `static_assertions` are zero-cost compile-time checks.
- Proc macro error messages -- trybuild tests ensure developer experience does not regress.
- Migration forward-application -- a broken migration is catastrophic.

**High value, moderate cost (test thoroughly):**
- Work unit execute paths -- the core business logic.
- Hot-cold store flush/hydration -- data loss bugs live here.
- HTMX partial responses -- snapshot the HTML fragments; structural regressions break the UI silently.
- Plugin load/unload lifecycle -- `dlopen` bugs are hard to diagnose in production.

**Moderate value, higher cost (test selectively):**
- Full E2E browser tests -- limit to 10-15 critical journeys. These are slow and brittle.
- Cross-platform plugin loading -- test on CI matrix, not locally.
- Benchmark regressions -- run on PR merge, not every push.

**Low value relative to cost (defer or skip):**
- Testing third-party crate behavior (rusqlite, axum, serde) -- trust upstream.
- Visual/CSS regression testing -- overkill for an appliance UI; HTMX partial snapshots suffice.
- Load/stress testing -- saalis runs on a single appliance with one user; not a scaling concern initially.
