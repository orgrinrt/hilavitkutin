## hilavitkutin crate structure (decided)

```
hilavitkutin-api       consumer-facing types, traits, platform contracts
hilavitkutin           engine (scheduling, dispatch, codegen, platform impls)
hilavitkutin-build     build-time optimisations (LLVM, MIR, codegen, cfg)
arvo                   numeric primitives (no_std, cross-project)
```

Runtime deps: consumer → hilavitkutin → hilavitkutin-api
Build deps: all three use hilavitkutin-build in build.rs

No deps on polka-*, std, alloc, or external crates. Designed for
extraction to own repo.

## Consolidated resolutions (R1-R5)

### R1: Column<T> only — IR is consumer-side

hilavitkutin-api defines `Column<T>` as a typed storage slot.
Schema-as-traits (`Column<In<IR>, As<T>>`) is consumer-side
(saalis-contracts does this with `Column<In<M>, As<V>>`).
hilavitkutin doesn't know about domain schemas — schedules
based on Read/Write column sets only.

**Implication for saalis:** Our `Column<In<M>, As<V>>` pattern
wraps hilavitkutin's `Column<T>`. Translation happens in the
registration layer (saalis-derive or saalis-scheduler bridge).

### R2: Incremental execution, evict/inject API

hilavitkutin owns per-morsel generation counters and skip-unchanged
DAG propagation. For persistence, provides:
- **evict/dump**: extract column data out (no copies, no retained refs)
- **inject/import**: move column data in

Column type registration is static (compile-time). No runtime
column type creation. Consumers build hot-cold layering on top.
On-disk state is consumer's responsibility.

**Implication for saalis:** saalis-persistence builds the hot↔cold
bridge using evict/inject. SIEVE eviction decides WHAT to evict,
hilavitkutin's evict API handles HOW.

### R3: Type-native stride, no flags byte, 16-byte max

Stride = `size_of::<T>()` per column. No per-entry flags byte.
Null/tombstone via separate bitmask. Value constraint:
`size_of::<T>() <= 16`. Supersedes earlier 128-bit tagged stride.

### R4: ColumnValue trait with blanket impl

```rust
pub trait ColumnValue: Copy + 'static {
    const BIT_WIDTH: usize;
}

// Blanket via min_specialization
impl<T: Copy + 'static> ColumnValue for T {
    default const BIT_WIDTH: usize = core::mem::size_of::<Self>() * 8;
}
```

No pack/unpack, no SlotSize. Type IS the storage.

**Implication for saalis:** Our `ValueType` trait with
`type Kind: ValueKind` wraps this. The `ColumnValue` in
hilavitkutin-api is the storage-level type; our `ValueType`
adds semantic meaning (sql affinity, gui rendering, access
control via ValueKind).

### R5: Resource model — scalars + const-sized collections

Resources have three field types:
- `Field<T>` — ≤16 bytes, register-cached
- `Seq<T, N>` — const-sized array, arena-allocated
- `Map<K, V, N>` — const-sized map, arena-allocated

No dynamic collections. ColumnStorage returns raw pointers.

## Additional consolidated domains (L2-L5, read 2026-03-19)

### Store types (domain 06)

Three store types, one scheduling mechanism:
```rust
pub struct Resource<T>(PhantomData<T>);  // singleton, one value
pub struct Column<T>(PhantomData<T>);    // collection, morsel-chunked
pub struct Virtual<T>(PhantomData<T>);   // zero-data, DAG edge only
```

- `StoreId(u16)` — dense index assigned at plan time
- `AccessMask(u64)` — bitmask, one bit per StoreId (≤64 stores)
- `AccessSet` trait on tuples: compile-time store descriptors
- ZSTs erased to plain usize indices at execution time

**Implication for saalis:** Our `Column<In<M>, As<V>>` wraps
hilavitkutin's `Column<T>`. `Resource<T>` maps to context services
(HttpClient, Storage, etc. are singletons the host provides).
`Virtual<T>` maps to our event system — fire-only, no data.

### Column storage (domain 07, R5 elaborated)

- Raw pointers, NOT slices (slices cause aliasing UB when WUs fused)
- Consumer provides backing memory (arena, mmap)
- Library never calls allocator directly
- 64-byte alignment on column arrays
- release(column) advisory for column lifetime
- evict/dump + inject/import APIs for persistence (R2)
- Separate arena for resource Seq/Map allocations

### Determinism (domain 08)

- Ordering IS the data flow — DAG edges from Write→Read overlap
- `COMMUTATIVE: bool` on WorkUnit
- No partial writes — WUs buffer internally, write on return
- Version stamps increment only on completion
- No triggers/emits — data access patterns fully capture ordering

**Implication for saalis:** Our event-driven DAG (Triggers/Emits)
is a HIGHER level than hilavitkutin's data DAG. They coexist.
saalis orchestrates (which connector runs when), hilavitkutin
executes (how data flows through the pipeline).

### WorkUnit trait (domain 09, consolidated)

```rust
pub trait WorkUnit<Schedule = Always>: Send + Sync + 'static {
    const NAME: &'static str;
    type Read: AccessSet;
    type Write: AccessSet;
    type Hint: SchedulingHint = (Immediate, Atomic, Normal);
    const COMMUTATIVE: bool;
    fn execute(&self, ctx: Context<Self::Read, Self::Write>);
}
```

- Schedule conditions: `Always` (default), `On<V>` (virtual fires)
- Context: `&self` receiver (load-bearing for aliasing safety)
- Raw pointer access: `unsafe fn read<T>(&self, i) -> T`,
  `unsafe fn write<T>(&self, i, val)`
- Registration: no #[register] in hilavitkutin — consumers handle
  it. Explicit scheduler builder API.
- ≤8 unique columns per WU (guideline, not constraint)

**Implication for saalis:** saalis wraps this with its own WorkUnit
that adds Subsystem, Triggers/Emits, context services. The bridge
translates saalis's event-driven work into hilavitkutin's data-
pipeline WorkUnits.

### Virtual flag system (domain 10)

- Bit-packed fire flags, one bit per registered Virtual
- Hierarchical check: u64→u8 scanning
- Epoch-based reset (one u8 increment, not memset)
- Per-(virtual, consumer) bit tracking — no lost fires
- Virtuals are pure flags, not data carriers

### Phase decomposition (domain 11, merged with 18)

Pipeline execution model:
- **Phase:** wide DAG section between waist points
- **Trunk:** column-disjoint path within phase, zero cross-trunk sync
- **Branch:** side path attached to trunk
- **Bridge:** fan-in node from multiple trunks
- **Fiber:** unit of column co-location and morsel windowing (domain 14)

Execution strategies (bench-validated):
- Multi-trunk: parallel on separate cores
- Single-trunk: head+tail convergence (~2x)
- Pipelined: phases OVERLAP via progress counters
- Core-pinned: trunks assigned to pool threads at plan time
- Pre-allocated pool eliminates small-N catastrophe

### Morsel model (domain 12)

```
morsel_size = (L1_usable / Σ write_sizes).clamp(MIN, MAX) & !3
```

- L1_usable = detected_L1 × 0.75
- Write columns + write resource collections consume L1 budget
- Read-only rides L2 prefetcher
- Per-fiber morsel size
- Change detection: per-morsel generation counter, skip if unchanged
- evict/inject APIs handle persistence

### Fiber formation (domain 14, NEWLY CONSOLIDATED)

Replaces "chain formation." Vocabulary: chain → fiber, chain group
→ trunk, partition → phase.

**Fiber:** unit of column co-location and morsel windowing. WUs
within a fiber share columns in one arena allocation:
```
arena_base + col_A_offset + i * stride_A
arena_base + col_B_offset + i * stride_B
```

**Column classification within fibers:**
- Fiber-internal: register-to-register, DSE skips memory ops
- Fiber-input: read from memory at morsel start
- Fiber-output: written to memory at morsel end

**Algorithm:** Kahn's topo sort → greedy grouping walk with
holistic feasibility check (register file + L1 write budget +
L1+L2 total + no fan-in + no pipeline breaker).

**Holistic feasibility:** register file is the binding constraint
(~14 aarch64, ~10 x86), not L1 capacity.

**Per-phase adaptive configs:** MAX_FUSE, BALANCED, MAX_SPLIT.
Selection by topology (domain 21).

**Implication for saalis:** Fiber formation is entirely internal
to hilavitkutin. saalis never sees fibers. saalis submits
WorkUnits with Read/Write column sets, hilavitkutin forms fibers
from them. The evict/inject API is the only boundary saalis uses
for persistence.

## Cross-cutting constraints from consolidation

- **No-alloc:** no Vec, Box, String, alloc crate in hilavitkutin
- **dyn ban:** entirely banned, no exceptions
- **std::any ban:** no TypeId, no type erasure
- **min_specialization:** always-on nightly feature
- **ExpandedLto:** REQUIRED for devirtualisation of fn pointer arrays
- **Own implementation:** DuckDB as reference, not dependency

## hilavitkutin-api key types (from consolidation)

### Platform traits

```rust
pub trait MemoryProvider: Send + Sync {
    unsafe fn allocate(&self, len: usize, align: usize) -> *mut u8;
    unsafe fn deallocate(&self, ptr: *mut u8, len: usize);
    unsafe fn protect(&self, ptr: *mut u8, len: usize, read: bool, write: bool);
}

pub trait ThreadPool: Send + Sync {
    fn spawn(&self, f: impl FnOnce() + Send + 'static);
    fn worker_count(&self) -> usize;
}

pub trait Clock: Send + Sync {
    fn now_ns(&self) -> u64;
}
```

### Column<T>

Typed storage slot. hilavitkutin schedules based on which columns
a WorkUnit reads and writes. Consumer wraps this with domain-
specific schema types.

### WorkUnit (NOT YET CONSOLIDATED — L3 domain)

From the pre-consolidation state (may change):
```rust
trait WorkUnit {
    type Read: ColumnSlices;
    type Write: ColumnSlicesMut;
    fn process_batch(read: Self::Read, write: Self::Write) -> Result<()>;
}
```

This is the data-pipeline WorkUnit. saalis's event-driven WorkUnit
(Triggers/Emits, context services, I/O-bound) is higher-level.
The two coexist as separate layers.

### ColumnStorage (NOT YET CONSOLIDATED — L2 domain)

Store types mentioned but not yet fully traced:
- Column data stored in co-located arenas within chains
- One base pointer per morsel, columns at `base + col_offset + i * stride`

### What we can design now

With the evict/inject API pattern known (R2) and Column<T> wrapping
confirmed (R1), we can design:

- **saalis-persistence**: ColdStore backed by rusqlite, migration
  runner, FTS5, _saalis_schema. Uses evict/inject pattern for
  hot↔cold bridge (exact hilavitkutin types TBD but the flow is
  known: SIEVE evicts → dump column data → write to SQLite).

- **saalis-core**: plugin loading, settings engine, notification
  dispatch, file management. Independent of hilavitkutin internals.

- **saalis-web**: pure presentation, calls into core. Fully
  independent of hilavitkutin.

### What remains blocked

- saalis-scheduler: exact WorkUnit bridge shape, ColumnStorage

  with event-driven (saalis) context
- Hot store management: SIEVE integration with hilavitkutin's
