# Deep Dive: Image Handling in Rust for a Gallery-First Game Library UI

**Date:** 2026-03-15
**Scope:** Image processing pipeline for saalis -- decoding, resizing, format conversion, thumbnail generation, placeholder hashing, and HTTP serving via HTMX
**Context:** Single Rust binary (axum) serving cover art, screenshots, and box art for a game library on Batocera/loisto ARM SBC appliances (RPi4/5 class, 2-8 GB RAM). Gallery-first design: the primary view is a grid of cover images. Performance on ARM with limited memory is a hard constraint.

---

## 1. The `image` Crate -- Decoding, Resizing, Format Conversion

The [`image`](https://crates.io/crates/image) crate (image-rs) is the de facto standard for image manipulation in Rust. It provides a unified API for decoding, encoding, and pixel-level manipulation across a wide range of formats.

### Supported formats relevant to saalis

| Format | Decode | Encode | Notes |
|--------|--------|--------|-------|
| JPEG   | Yes (default) | Yes (default) | Most common source format from scrapers |
| PNG    | Yes (default) | Yes (default) | Lossless, used by some metadata sources |
| WebP   | Yes (default) | Yes (lossless only) | Pure-Rust backend via `image-webp`; lossy encode requires `libwebp-sys` |
| GIF    | Yes (default) | Yes (default) | Animated GIF support limited |
| AVIF   | Yes (default) | Yes (feature-gated) | Uses `ravif` for encoding, `dav1d` for decoding |
| BMP    | Yes (default) | Yes (default) | Occasionally seen in retro game cover scans |

### Resizing algorithms

The `image` crate exposes resizing through `imageops::resize()` and `DynamicImage::resize()`. Available `FilterType` variants, ordered from fastest to highest quality:

| Algorithm | Kernel size | Use case | Quality | Speed |
|-----------|------------|----------|---------|-------|
| `Nearest` | 1x1 | Pixel art upscaling (preserve sharp edges) | Lowest | Fastest |
| `Triangle` (Bilinear) | 2x2 | Quick previews, low-priority thumbnails | Low | Fast |
| `CatmullRom` (Bicubic) | 4x4 | Good general-purpose default | High | Medium |
| `Gaussian` | -- | Soft, slightly blurred output | Medium | Medium |
| `Lanczos3` | 6x6 | Best quality downscaling (gallery covers) | Highest | Slowest |

For saalis cover art, `Lanczos3` is the right default for pre-generated thumbnails (quality matters, and the resize happens once). `CatmullRom` is a reasonable fallback for on-demand resizing where latency matters more than the last bit of sharpness.

### Performance characteristics

The `image` crate's resizing implementation is pure Rust with no SIMD specialisation. On a 4928x3279 RGBA source image resized to 505x336, the `image` crate's Lanczos3 takes approximately 100-150ms on ARM64 (RPi4 class). This is acceptable for pre-generation but too slow for on-demand resizing under load. The crate also fully decodes images into memory as `DynamicImage`, meaning a 4K RGBA image consumes ~64 MB of RAM. This is manageable for one-at-a-time processing but would be dangerous if multiple images are decoded concurrently.

### Partial decoding

Some decoders implement `ImageDecoderRect`, which allows decoding only a rectangular region of an image. This is useful for extracting a crop from a large source image without loading the entire image into memory. JPEG and PNG decoders support this. For saalis, this is relevant when source images are significantly larger than needed -- a 4K box art scan only needs a central crop for a thumbnail.

## 2. `fast_image_resize` -- SIMD-Accelerated Resizing

The [`fast_image_resize`](https://crates.io/crates/fast_image_resize) crate (fir) is purpose-built for high-performance image resizing using SIMD instructions. It supports SSE4.1, AVX2 on x86_64 and NEON on ARM64 -- both architectures saalis targets.

### ARM64 benchmark results (from the project's benchmark suite)

Source image: nasa-4928x3279.png, resized to 505x336, Lanczos3 filter:

| Implementation | RGBA (ms) | RGB (ms) | Speedup vs `image` |
|----------------|-----------|----------|---------------------|
| `image` crate  | ~150      | ~110     | 1.0x (baseline)     |
| `fir` (Rust, no SIMD) | ~110 | ~80    | ~1.4x               |
| `fir` (NEON)   | ~62       | ~45      | ~2.4x               |
| `libvips`      | ~89       | ~65      | ~1.7x               |

On x86_64 with AVX2, the speedup is more dramatic -- `fir` achieves 5-10x over the `image` crate for Lanczos3, thanks to wider SIMD registers and more aggressive vectorisation.

### API design

`fast_image_resize` does not decode or encode images. It operates on raw pixel buffers. The typical workflow is:

1. Decode with the `image` crate (or any decoder) into a pixel buffer
2. Wrap the buffer in an `fir::Image`
3. Create a `Resizer` with the desired algorithm
4. Resize into a pre-allocated destination buffer
5. Encode the result with the `image` crate

This separation is a feature: it lets you swap the resizing engine without changing your decode/encode pipeline.

### When to use it

For saalis, `fast_image_resize` should be the resizing backend whenever available. The NEON optimisations on ARM64 cut Lanczos3 resize time nearly in half compared to the `image` crate. On-demand thumbnail generation becomes feasible at ~45ms per image instead of ~110ms. For batch pre-generation (e.g., initial library scan of 500 covers), the difference compounds: 22 seconds vs 55 seconds.

### `pic-scale` as an alternative

[`pic-scale`](https://crates.io/crates/pic-scale) is a newer contender offering SIMD (NEON, SSE, AVX-2, AVX-512) and multithreading. It claims parity or faster performance than `fast_image_resize` for equal SIMD and pixel type. It also supports color space transformations (CIE L\*a\*b, Oklab, Sigmoidal) which can improve perceptual quality of resized images. Worth watching, but `fast_image_resize` has a longer track record and more complete benchmarks.

## 3. libvips Bindings -- Industrial-Grade Image Processing

[libvips](https://www.libvips.org/) is a C library for image processing that uses a demand-driven, streaming architecture. It processes images in small tiles, keeping memory usage low regardless of image size. Several Rust binding crates exist:

| Crate | Status | Notes |
|-------|--------|-------|
| [`libvips`](https://crates.io/crates/libvips) | Active | Safe wrapper based on introspection API, generated from libvips 8.17.3 |
| [`vips-rs`](https://github.com/houseme/vips-rs) | Active | Ergonomic API with `VipsInstance` lifecycle management |
| [`rs-vips`](https://lib.rs/crates/rs-vips) | Active | Fork of `libvips-rust-bindings`, uses `VOption` for optional arguments |

### When libvips makes sense

libvips excels when you need to process many large images with minimal memory. Its streaming tile-based architecture means a 20,000x15,000 pixel scan can be thumbnailed using only a few MB of RAM. It also handles colour management (ICC profiles), smart cropping (attention-based), and format conversion in a single pipeline call.

### When it does not make sense for saalis

libvips is a C library dependency. Saalis targets a single static Rust binary deployed on Batocera. Adding libvips means either:

1. Statically linking libvips (complex, pulls in libjpeg, libpng, libwebp, etc.)
2. Dynamically linking against the host's libvips (Batocera may not ship it)
3. Bundling a .so and managing the library path

Given that saalis processes images one at a time (or a small batch during library scan), and the target images are cover art (typically under 2000x3000 pixels), the `image` crate + `fast_image_resize` combination is sufficient. libvips would be justified if saalis ever needed to process massive archival scans or hundreds of images concurrently on a server with ample resources. For the appliance target, pure Rust is the pragmatic choice.

## 4. Thumbnail Generation Strategies

### Pre-generated vs on-demand

| Strategy | Pros | Cons |
|----------|------|------|
| **Pre-generated** | Zero latency at serving time; simple HTTP serving of static files; predictable disk usage | Disk space for all size variants; upfront processing cost; wasted work if some covers are never viewed |
| **On-demand** | No wasted disk; covers only the sizes actually requested; simpler initial import | Latency on first request; CPU spikes under load; requires in-memory or disk caching |
| **Hybrid** | Best of both -- pre-generate the most common size, generate others on demand and cache | Slightly more complex code; two code paths |

For saalis, the hybrid approach is best. Pre-generate the gallery grid thumbnail (the most common view) during the library scan/enrichment phase, and generate detail-view and list-view sizes on demand with caching. The gallery grid is the hot path -- every visit loads dozens of covers simultaneously.

### Size presets

Game cover art has several common aspect ratios depending on the platform era:

| Platform era | Typical aspect ratio | Example dimensions |
|-------------|---------------------|-------------------|
| Modern (PS4/5, Switch) | ~2:3 (portrait) | 264x352 cover |
| Classic console (SNES, Genesis) | ~3:4 (portrait) | 264x352 box art |
| PC (Steam banners) | ~92:43 (landscape) | 460x215 header |
| Square (some mobile, icons) | 1:1 | 264x264 |

Recommended size presets for saalis:

| Preset name | Max dimension | Use case |
|-------------|--------------|----------|
| `grid` | 264px wide | Gallery grid view (primary) |
| `detail` | 528px wide | Detail/info panel |
| `hero` | 1280px wide | Full-width hero banner |
| `list` | 80px wide | Compact list view |
| `micro` | 40px wide | Notification/toast context |

All presets should preserve the source aspect ratio and constrain by the larger dimension, not crop. The `object-fit: cover` CSS property handles filling grid cells without server-side cropping.

### Responsive images with srcset

For the gallery grid, generate two sizes (1x and 2x for retina) and serve with `srcset`:

```html
<img
  src="/img/cover/{id}/grid"
  srcset="/img/cover/{id}/grid 264w, /img/cover/{id}/detail 528w"
  sizes="(min-width: 1024px) 264px, 50vw"
  alt="{title}"
  loading="lazy"
  width="264"
  height="352"
>
```

This lets the browser pick the right resolution. On a TV-distance 1080p display, the 264px version is sufficient. On a close-up retina tablet, the 528px version loads instead. There is no need for more than two or three variants per image -- diminishing returns set in quickly, and each variant costs disk space and cache pressure.

## 5. WebP and AVIF for Web Serving

### Compression characteristics

| Format | Compression vs JPEG | Encoding speed | Decoding speed | Browser support (2025) |
|--------|---------------------|---------------|----------------|----------------------|
| JPEG   | Baseline            | Very fast      | Very fast       | 100% |
| WebP   | 25-34% smaller      | Fast           | Fast            | ~96% |
| AVIF   | ~50% smaller        | Slow (5-20x JPEG) | Moderate    | ~94% |

For photographic content like game cover art and screenshots, AVIF delivers striking compression improvements. A 100 KB JPEG cover becomes ~50 KB in AVIF at equivalent visual quality, or ~70 KB in WebP.

### Encoding speed trade-offs

AVIF encoding is the bottleneck. The `ravif` crate (pure Rust, based on `rav1e`) offers quality (1-100, default 80) and speed (1-10, default 4) parameters. On ARM64:

- Speed 4, quality 80: ~500ms for a 528x700 cover image
- Speed 8, quality 75: ~150ms but visibly softer
- Speed 10, quality 80: ~80ms but unacceptably blurry

For pre-generated thumbnails, speed 4 is fine -- it runs during the enrichment phase, not on the hot path. For on-demand generation, WebP is the safer choice: encoding is 5-10x faster than AVIF with still-good compression.

### Serving strategy

The `image` crate's WebP encoder only supports lossless encoding (resulting in larger files than lossy). For lossy WebP, use `libwebp-sys` (a C dependency, but small and widely available) or accept lossless WebP for the modest file-size penalty. Alternatively, encode to JPEG at quality 85 for on-demand responses and reserve AVIF/WebP for pre-generated assets.

For saalis, the recommended approach:

1. **Pre-generated thumbnails:** Encode to WebP (lossy, via `webp` crate with `libwebp-sys`) and AVIF (via `ravif`). Store both.
2. **Serving:** Use content negotiation via the `Accept` header. If the browser sends `Accept: image/avif`, serve AVIF. If `Accept: image/webp`, serve WebP. Otherwise, serve JPEG.
3. **Fallback:** Always keep a JPEG version. Older browsers, unusual clients, and debugging tools expect it.

Content negotiation in axum is straightforward -- parse the `Accept` header and select the file extension accordingly. No `<picture>` tag gymnastics needed on the HTMX side.

## 6. Lazy Loading in HTMX

### Native lazy loading

The simplest approach is the `loading="lazy"` attribute on `<img>` tags. This is supported by all modern browsers and requires zero JavaScript. The browser defers loading until the image is near the viewport.

```html
<img src="/img/cover/{id}/grid" loading="lazy" width="264" height="352" alt="{title}">
```

This works well for a gallery grid. The browser's built-in heuristics determine "near the viewport" -- typically 1-2 viewport heights ahead.

### HTMX-driven lazy loading

For more control, HTMX provides the `intersect` trigger, which uses the Intersection Observer API under the hood:

```html
<div hx-get="/covers/{id}" hx-trigger="intersect once threshold:0.1" hx-swap="innerHTML">
  <!-- placeholder here -->
</div>
```

When the div scrolls into view (10% visible), HTMX fires a GET request. The server responds with the full `<img>` tag (or a richer card component). This approach allows:

- **Server-rendered placeholders:** The initial HTML contains only the BlurHash/ThumbHash placeholder, reducing initial page weight
- **Batched loading:** The server can return multiple covers in a single response if the endpoint is designed for it
- **Progressive enhancement:** If JavaScript fails, the placeholder is visible but nothing breaks

The `revealed` trigger is a simpler alternative -- it fires once when the element enters the viewport, without threshold configuration.

### Placeholder strategies

| Strategy | Bytes | Visual quality | Implementation complexity |
|----------|-------|---------------|--------------------------|
| Solid colour (dominant) | 7 (hex) | Minimal | Low |
| CSS gradient | ~50 | Low | Low |
| BlurHash | 20-30 | Good (blurred approximation) | Medium |
| ThumbHash | 25-35 | Better (preserves aspect ratio, supports alpha) | Medium |
| Tiny JPEG (inline data URI) | 200-500 | Very good | Low |
| Skeleton/shimmer | 0 (CSS) | N/A (loading indicator) | Low |

For saalis, ThumbHash in a `data-` attribute is the recommended approach. The hash is stored in the database alongside the entity metadata, inlined into the initial HTML, and decoded client-side to a tiny canvas. This gives an immediate visual approximation of every cover in the gallery without any image requests.

## 7. BlurHash and ThumbHash -- Tiny Placeholders

### BlurHash

[BlurHash](https://blurha.sh/) encodes an image as a short ASCII string (20-30 bytes) representing the low-frequency colour components. A 4x3 component BlurHash looks like: `LEHV6nWB2yk8pyo0adR*.7kCMdnj`.

Rust crates:
- [`blurhash`](https://crates.io/crates/blurhash) -- pure Rust, encode and decode
- [`blurhash-ng`](https://crates.io/crates/blurhash-ng) -- optimised implementation
- [`fast-blurhash`](https://crates.io/crates/fast-blurhash) -- speed-focused variant

Parameters: `components_x` (1-9) and `components_y` (1-9) control detail level. 4x3 is a good default for cover art -- enough detail to show the dominant colour regions without excessive string length.

### ThumbHash

[ThumbHash](https://evanw.github.io/thumbhash/) is a newer algorithm that encodes more detail in roughly the same byte budget. Key advantages over BlurHash:

- Encodes the image's **aspect ratio** -- the placeholder has the correct proportions
- Supports **alpha channel** (transparency)
- No configurable parameters -- the algorithm auto-selects the optimal encoding
- Slightly better perceptual quality at comparable sizes

Rust crate: [`thumbhash`](https://crates.io/crates/thumbhash) (official, by the algorithm's author).

The typical workflow:

1. During enrichment, resize the cover to ~100px wide
2. Encode the small image with `thumbhash::rgba_to_thumb_hash()`
3. Store the resulting `Vec<u8>` (typically 25-35 bytes) as a blob in the entity metadata
4. In the HTML template, base64-encode the hash into a `data-thumbhash` attribute
5. A small client-side script decodes the hash to a data URI and sets it as the `<img>` `src`
6. When the real image loads, it replaces the placeholder

### Recommendation for saalis

ThumbHash is the better choice. The automatic aspect ratio encoding is valuable for a gallery with mixed cover art dimensions (portrait console covers vs landscape Steam banners). The lack of configurable parameters also simplifies the pipeline -- there is no "what component count should we use?" question.

The decode step on the client is fast -- sub-millisecond in JavaScript for a single hash. For a gallery of 50 visible covers, decoding all ThumbHashes takes under 10ms total.

### Storage cost

At ~30 bytes per image, 10,000 covers require ~300 KB of ThumbHash data. This is negligible in SQLite and can be included in the initial page payload without concern.

## 8. Aspect Ratio Detection and CSS Layout Stability

### The CLS problem

When images load without dimensions, the browser does not know how much space to reserve. The page reflows as each image loads, causing Cumulative Layout Shift (CLS). In a gallery grid, this manifests as covers jumping around -- a terrible user experience, especially on a TV-distance gamepad-controlled UI where the selected item might move off-screen.

### Prevention

Three mechanisms, all of which saalis should use:

**1. Width and height attributes on `<img>`**

Always include `width` and `height` attributes matching the intrinsic (or rendered) dimensions:

```html
<img src="/img/cover/{id}/grid" width="264" height="352" loading="lazy" alt="{title}">
```

Modern browsers use these to calculate the aspect ratio before the image loads, reserving the correct space. This single technique eliminates most CLS from images.

**2. CSS `aspect-ratio` on containers**

For grid cells that should have a consistent aspect ratio regardless of content:

```css
.cover-cell {
  aspect-ratio: 3 / 4;       /* portrait cover art default */
  overflow: hidden;
}
.cover-cell img {
  width: 100%;
  height: 100%;
  object-fit: cover;          /* crop to fill, no distortion */
  object-position: center top; /* keep the top of the art visible */
}
```

This ensures every grid cell occupies the same space, even if images have slightly different native ratios (a 2:3 cover and a 3:4 cover both fit in the 3:4 cell via `object-fit: cover`).

**3. Handling mixed aspect ratios**

Game cover art comes in various aspect ratios. Options:

| Approach | Visual result | Trade-off |
|----------|--------------|-----------|
| Force uniform ratio (`aspect-ratio: 3/4`, `object-fit: cover`) | Clean grid, some images slightly cropped | Loses edges of non-conforming art |
| Uniform ratio with `object-fit: contain` | No cropping, but letterboxing/pillarboxing | Visual gaps, less polished |
| Dynamic ratio per item (masonry layout) | Pinterest-style, each item at native ratio | Complex layout, hard to navigate with gamepad |
| Group by ratio class (portrait/landscape/square) | Sections with consistent grids | More complex data model |

For saalis, **forced 3:4 with `object-fit: cover`** is the right default. Cover art is almost always portrait and close to 3:4. The occasional landscape Steam banner should be handled differently -- detected at enrichment time and flagged as `aspect_class: landscape`, displayed in a separate row or with a landscape cell template.

### Detecting aspect ratio at enrichment time

When a cover image is first downloaded and processed, compute and store:

```rust
struct CoverMeta {
    width: u32,
    height: u32,
    aspect_class: AspectClass, // Portrait, Landscape, Square
    thumbhash: Vec<u8>,
}

enum AspectClass {
    Portrait,   // height > width * 1.1
    Landscape,  // width > height * 1.1
    Square,     // within 10% of 1:1
}
```

This metadata lets the template engine choose the right CSS class without inspecting the image at render time.

## 9. Memory Usage -- Processing Large Images on ARM

### The problem

A 4928x3279 RGBA image (common high-resolution box art scan) occupies:

```
4928 * 3279 * 4 bytes = ~64.6 MB
```

On a RPi4 with 2 GB RAM (after OS and Batocera overhead, ~1.2 GB available to saalis), loading two such images simultaneously consumes ~130 MB, or about 10% of available memory. During a bulk import of 500 covers, processing them sequentially is fine. Processing them concurrently would be catastrophic.

### Mitigation strategies

**1. Sequential processing with explicit drop**

Process one image at a time. After resizing and encoding, explicitly drop the source buffer before starting the next image. Rust's ownership model makes this natural:

```rust
for cover in covers {
    let img = image::open(&cover.path)?;       // ~65 MB allocated
    let thumb = fir_resize(&img, 264, 352);    // small buffer
    encode_and_save(&thumb, &cover.thumb_path)?;
    // `img` dropped here -- 65 MB freed
}
```

**2. Decode at reduced resolution**

JPEG decoders support DCT scaling -- decoding directly to 1/2, 1/4, or 1/8 size without fully decoding the image. The `image` crate does not expose this directly, but the `jpeg-decoder` crate (which `image` uses internally) supports scale factors. For a 4928px source that needs a 264px thumbnail, decoding at 1/8 scale (616px) and then resizing from 616 to 264 uses ~1.5 MB instead of 65 MB.

If using `image::io::Reader`, you can set memory limits:

```rust
let reader = image::io::Reader::open(path)?
    .with_guessed_format()?;
let mut limits = Limits::default();
limits.max_alloc = Some(32 * 1024 * 1024); // 32 MB max
let img = reader.with_limits(limits).decode()?;
```

This prevents a malformed or unexpectedly large image from exhausting memory.

**3. Streaming with `ImageDecoderRect`**

For decoders that support it (JPEG, PNG), `ImageDecoderRect` allows decoding only a rectangular region. If you only need the centre of a large image for a thumbnail, this saves significant memory.

**4. Memory budget**

Saalis should enforce a global memory budget for image processing. A simple approach: use a semaphore to limit concurrent image operations. On a 2 GB device, allow one concurrent resize; on 4 GB+, allow two or three.

```rust
static IMAGE_SEMAPHORE: Semaphore = Semaphore::new(1); // or from config

async fn generate_thumbnail(id: EntityId) -> Result<Bytes> {
    let _permit = IMAGE_SEMAPHORE.acquire().await?;
    // blocking image work in spawn_blocking
    tokio::task::spawn_blocking(move || {
        // decode, resize, encode
    }).await?
}
```

### Peak memory per operation

| Operation | Memory (4928x3279 RGBA source) |
|-----------|-------------------------------|
| Full decode | ~65 MB |
| Decode at 1/4 scale | ~4 MB |
| Decode at 1/8 scale | ~1.5 MB |
| Resize destination (264x352 RGBA) | ~370 KB |
| WebP encode buffer | ~50-200 KB |
| AVIF encode (rav1e internals) | ~5-15 MB |
| ThumbHash encode (100px input) | ~40 KB |

Total peak for a single thumbnail pipeline (1/8 decode + resize + WebP encode): ~2-3 MB. This is comfortable on any target device.

## 10. Caching Strategy

### URL design

For CDN-friendly, cache-friendly image URLs, embed a content hash in the path:

```
/img/{entity_id}/{preset}/{content_hash}.webp
```

Example: `/img/42/grid/a1b2c3d4.webp`

The `content_hash` is a truncated hash (first 8 hex chars of SHA-256 or xxHash of the image bytes). When a cover image changes (user replaces it, re-enrichment fetches a different version), the hash changes, and the old cached version is automatically invalidated.

### HTTP cache headers

For content-hashed URLs (hash in the path):

```
Cache-Control: public, max-age=31536000, immutable
```

One year, immutable. The hash in the URL guarantees that if the content changes, the URL changes. Browsers, proxies, and CDNs can cache aggressively without ever revalidating.

For non-hashed URLs (e.g., `/img/{entity_id}/grid` as a convenience redirect):

```
Cache-Control: public, max-age=60
ETag: "a1b2c3d4"
```

Short max-age with an ETag derived from the content hash. The browser revalidates after 60 seconds, the server responds with 304 Not Modified if the ETag matches. This provides freshness with minimal bandwidth.

### ETag generation

Compute the ETag from the image content hash, not the file modification time. File mtime is unreliable across filesystems and after restores. A content hash is deterministic and consistent across nodes (if saalis ever runs on multiple devices).

```rust
fn etag_for_image(image_bytes: &[u8]) -> String {
    let hash = xxhash_rust::xxh3::xxh3_64(image_bytes);
    format!("\"{:016x}\"", hash)
}
```

xxHash is preferred over SHA-256 for ETag purposes -- it is non-cryptographic but extremely fast (>10 GB/s on modern hardware), and collision resistance is sufficient for cache invalidation.

### On-disk thumbnail cache

Pre-generated and on-demand thumbnails should be stored on disk in a flat directory structure:

```
/data/cache/img/{entity_id}/{preset}.webp
/data/cache/img/{entity_id}/{preset}.avif
/data/cache/img/{entity_id}/{preset}.jpg
```

The axum handler checks for the cached file first. If it exists, serve it directly (with `tokio::fs::File` and axum's `Body::from_stream`). If not, generate, write to cache, and serve. This pattern keeps the hot path as fast as a static file server.

### Cache eviction

On space-constrained devices, implement LRU eviction for the thumbnail cache. Track access time (or use filesystem atime if available). When the cache exceeds a configured size (e.g., 500 MB), evict the least-recently-accessed thumbnails. The `grid` preset should be exempt from eviction (or given a much higher priority) since it is accessed on every gallery view.

### Summary of headers by URL pattern

| URL pattern | Cache-Control | ETag | Rationale |
|-------------|--------------|------|-----------|
| `/img/{id}/{preset}/{hash}.{ext}` | `public, max-age=31536000, immutable` | No | Hash guarantees freshness |
| `/img/{id}/{preset}` (redirect) | `public, max-age=60` | Yes (content hash) | Short-lived, revalidate via ETag |
| `/img/{id}/original` | `private, max-age=3600` | Yes | Original source, less aggressive |

## Summary of Recommendations

| Decision | Recommendation | Rationale |
|----------|---------------|-----------|
| Decode/encode library | `image` crate | Pure Rust, format breadth, no C deps |
| Resize engine | `fast_image_resize` with NEON | 2-2.5x faster than `image` on ARM64 |
| libvips | Do not use | C dependency incompatible with single-binary goal |
| Thumbnail strategy | Hybrid (pre-generate `grid`, on-demand others) | Balances latency and disk use |
| Web format | WebP (lossy) primary, AVIF for pre-gen, JPEG fallback | WebP is fast to encode; AVIF best compression |
| Placeholder | ThumbHash in DB, inline in HTML | Aspect-aware, no config, ~30 bytes |
| Lazy loading | `loading="lazy"` + HTMX `intersect` for richer patterns | Native browser support + progressive enhancement |
| CLS prevention | `width`/`height` attrs + `aspect-ratio: 3/4` + `object-fit: cover` | Zero layout shift |
| Memory safety | Decode at reduced scale, 1-concurrent semaphore, `Limits` | Safe on 2 GB ARM |
| Caching | Content-hash URLs, `immutable` header, on-disk cache with LRU | Aggressive caching, correct invalidation |
