# verkko-protocol

## Abstract

Point-to-point communication protocol for the verkko ecosystem.
How two endpoints establish a connection, exchange authenticated
frames, manage congestion, and handle network address changes.

Everything here is symmetric: both sides are equal participants.
This document knows nothing about clusters, meshes, trust
hierarchies, or content. Just two peers exchanging frames over
UDP.

Sans-IO: the protocol is a state machine. The consumer provides
the socket. verkko-protocol never touches the network directly.

**Attribution (v10).** This design is the product of forty expert
perspectives across eleven review phases. v10 validated by three
independent reviewers (cryptographic protocol specialist, distributed
systems architect, applied mathematician). See v10 document for full
attribution.

## Dependencies

- **verkko-crypto**: FrameKey, TransportAEAD, NonceSafety,
  EpochRatchet, KeyZeroization

## Defined Concepts

### Frame

The atomic unit of transmission. Fixed overhead, authenticated,
encrypted.

Invariants:
- One frame = one logical message (message-oriented, not
  byte-stream).
- Frame integrity verified before any processing of contents.
- Frames are never partially delivered.

### Session

A Noise KK or Noise XX handshake producing a shared session
state between two endpoints.

Invariants:
- Noise KK for endpoints that already know each other's static
  keys. Noise XX for first contact (with SAS verification).
- Session produces forward-secret transport keys.
- Session keys are independent of all other sessions.
- On KK failure, automatic fallback to XX with notification.

### Connection

A stateful association between two endpoints, identified by
a connection ID that survives IP address changes.

Invariants:
- Connection ID is stable across network migration (WiFi to
  ethernet, IP change, NAT rebinding).
- One connection carries multiple multiplexed streams.
- Connection state is bounded (const-generic capacity limits).

#### Connection ID generation

    connection_id = random_u32()

Generated by each endpoint via CSPRNG at session initiation.
Each direction of a bidirectional session has its own connection_id.

Reserved values:
    0x00000000  Handshake frame (no session established yet).
    0xFFFFFFFF  Reserved for future use.

Lifecycle:
- Generated at session initiation (Noise message 1 sent).
- Valid for the lifetime of the session.
- Invalidated on session termination or renegotiation.
- Preserved across connection migration.

Error handling:
- connection_id = 0x00000000 or 0xFFFFFFFF: regenerate.
- Collision with existing session: regenerate.

### Stream

A multiplexed logical channel within a connection. Independent
sequence numbers and flow control per stream.

Invariants:
- No head-of-line blocking between streams.
- Each stream has independent ordering guarantees.
- Stream count is bounded by const generic.

#### Stream multiplexing

Stream header (7 bytes, inside encrypted payload):

    Offset  Width  Field
    ------  -----  -----
     0      2      stream_id: u16
     2      4      sequence: u32 (per-stream monotonic)
     6      1      flags: u8
                    Bit 0: FIN (stream termination)
                    Bit 1: RST (stream reset)
                    Bit 2: SACK_PRESENT
                    Bits 3-7: reserved (MUST be 0)

Stream ID allocation (three-tier):
    0x0000           Control stream (VitalQueue Tier 1).
    0x0001-0x00FF    Vital tier streams (control log, keys).
    0x0100-0x0FFF    Reactive tier streams (gossip, filters).
    0x1000-0xFFFE    Bulk tier streams (matter, relay ops).
                     LEDBAT-paced.
    0xFFFF           Reserved.

Stream lifecycle:
- Implicit open on first frame with a new stream_id.
- FIN flag closes the stream gracefully.
- RST flag aborts the stream immediately.

HEARTBEAT and SACK-only frames do not carry stream headers
(they are connection-level, not stream-level).

### Heartbeat

Periodic liveness signal between connected endpoints. The
heartbeat frame has two regions: a fixed 18-byte transport
header (owned by this document) and an 18-byte extension
region (owned by verkko-mesh: HeartbeatExtension).

Invariants:
- Heartbeat interval adapts to metabolic state (2s-30s).
- Missed heartbeats trigger failure detection (3 missed for
  routine, 1 for emergency).
- Transport header carries: metabolic state (bits 0-1),
  gossip_mode (bit 2), convergence_active (bit 3),
  admin_flag (bit 4), reserved (bits 5-7), RTT, loss,
  jitter, queue depth, available bandwidth, HLC timestamp.
- Extension region carries mesh-layer state; protocol treats
  these bytes as opaque payload.
- Heartbeat is the sole input to failure detection; no other
  mechanism declares a peer dead.

### LEDBAT

Low Extra Delay Background Transport. Congestion control that
yields to foreground traffic.

Invariants:
- verkko traffic never degrades the user's foreground network
  experience (gaming, streaming).
- Coupled controllers at gateway: one LEDBAT instance per
  physical uplink, fair-share across connections.
- Backs off on any sign of congestion (one-way delay increase).

### VitalQueue

Three-tier priority queue for outbound frames.

Invariants:
- Tier 1 (vital): control + gossip frames. 70/30 weighted fair
  queuing between control and gossip sub-queues.
- Tier 2 (normal): application data.
- Tier 3 (bulk): background transfers. LEDBAT-paced.
- Vital frames are never starved by bulk traffic.

### ReplayWindow

Sliding window for detecting replayed frames. Per-session.

Invariants:
- Duplicate frames are silently dropped.
- Window size W=1024 (128 bytes per session).
- Window resets on session renegotiation (safe because keys
  change).

### NATTraversal

Mechanisms for establishing connections through NAT devices.

Invariants:
- Relay-first fallback: if direct connection fails, traffic
  routes through a relay peer.
- Hole punching attempted before relay fallback.
- OBSERVED_ADDR in handshake response for address discovery.

### PeerDiscovery

How endpoints find each other before connecting.

Invariants:
- Fallback chain: mDNS (LAN) -> invite hint (QR/link) ->
  cached peers -> DHT (Mainline) -> relay.
- Each mechanism is attempted in order; first success wins.
- Discovery result is a (address, identity_fingerprint) pair.

### SchemaSync

Protobuf schema exchange during connection handshake.

Invariants:
- Client sends schema hash. Server compares.
- On mismatch, server sends updated schema.
- Client always has current message definitions after handshake.

### SansIO

The protocol is a state machine that produces outbound bytes
and consumes inbound bytes. No socket ownership.

Invariants:
- verkko-protocol never calls send(), recv(), or any syscall.
- Consumer drives I/O: provides received bytes, collects
  outbound bytes.
- All buffers are arena-allocated from a MemoryProvider.
  No heap allocation.

### MetabolicState

Four-state machine governing resource usage: Background, Idle,
Active, Stress.

Invariants:
- Minimum dwell time before state transitions (prevents
  oscillation).
- Heartbeat interval scales with metabolic state.
- Phase jitter desynchronizes transitions across peers.
- MetabolicState transitions are driven by a governor that
  provides metabolic state as input. See verkko-mesh: Conductor
  for the governor specification.

### MetabolicGovernor (abstract contract)

An abstract contract that drives MetabolicState transitions. The protocol
layer defines the state machine and this contract; mesh or other layers
provide concrete implementations.

Invariants:
- MetabolicGovernor is a pure function from resource pressure readings
  to MetabolicState.
- The governor MUST produce exactly one of {Background, Idle, Active,
  Stress} for any valid input.
- The governor MUST NOT require access to protocol-internal state
  beyond what it receives as input.
- The governor MUST satisfy the dwell time invariant: no transition
  SHALL be emitted within dwell_time of the previous transition,
  except when convergence_override is true.
- ResourcePressures is opaque to the protocol layer. Protocol passes
  the pressure readings through to the governor without inspecting
  their contents.

    #[repr(u8)]
    enum MetabolicState {
        Background = 0b00,
        Idle       = 0b01,
        Active     = 0b10,
        Stress     = 0b11,
    }

State machine transitions:

    Background <-> Idle <-> Active <-> Stress

All transitions are bidirectional. Adjacent-only transitions are NOT
required; the governor MAY emit any state from any state (subject to
dwell time). This permits the convergence override to jump directly
from Background to Active or Stress.

Error handling:

| Condition | Response | State after |
|-----------|----------|-------------|
| Governor returns invalid state | Log error, retain current state | Unchanged |
| Any pressure value is NaN | Log error, retain current state | Unchanged |
| Governor panics or is absent | Default to Active | Active |

### SACK

Selective acknowledgement. Delivery ratio measurement for
congestion control and health observation.

Invariants:
- Per-session tracking of delivered vs sent frame counts.
- Delivery ratio computed from SACK state, EMA-smoothed over
  the metabolic dwell time.
- SACK state is the input to receiver-observed health scoring
  (see verkko-mesh: HealthPipeline).

### ConnectionMigration

QUIC-style PATH_CHALLENGE/PATH_RESPONSE for surviving network
address changes.

Invariants:
- Connection survives IP address changes (WiFi to ethernet,
  NAT rebinding).
- Rate-limited to 1KB/s before path validation completes.
- Uses RFC 9000 Section 9 mechanism.

### SessionResumption

0-RTT session resumption using a stored resumption token.
Avoids full Noise handshake on reconnection.

Invariants:
- Resumption token is encrypted under the session's export
  key (first 32 bytes of GetHandshakeHash()) and stored locally.
- 0-RTT data is replay-vulnerable: only idempotent operations
  permitted in the 0-RTT window (heartbeat, gossip, filter queries).
- The responder validates the resumption token's freshness
  (expiry bound, monotonic ticket counter).
- Fallback to full handshake if resumption fails (token
  expired, key rotated, peer restart).
- Resumption does not weaken forward secrecy for post-0-RTT data:
  the resumed session derives new transport keys via Noise KK.
  0-RTT data has weaker forward secrecy (encrypted under the
  previous session's export key, not fresh ephemeral DH).

Wire format:
- Plaintext token: 80 bytes (session_id [32], session_key_hash [32],
  expiry [u64], ticket_counter [u32], reserved [4]).
- Encrypted token: 120 bytes (nonce [24] + ciphertext [80] + tag [16]).
- Resumption handshake uses hs_type = 0x03 (RESUME).
- Token nonce derived via DomainSeparator::ResumeToken.

## Body

### Version Negotiation

Two peers MUST agree on a protocol version before any authenticated
communication occurs. Version negotiation is the first exchange on
any new UDP association, before the Noise handshake.

VersionProbe (18-28 bytes, unauthenticated):
- Magic: 0x56524B4F ("VRKO"), 4 bytes.
- probe_type: 0x01, 1 byte.
- flags: reserved (all zero in v1), 1 byte.
- version_count: u16, range [1, 6].
- versions: u16 * version_count, descending preference.
- probe_nonce: u64, CSPRNG-generated.

VersionResponse (20 bytes):
- Magic: 0x56524B4F, 4 bytes.
- probe_type: 0x02, 1 byte.
- flags: reserved, 1 byte.
- version_count: 0x0001, 2 bytes.
- selected_version: u16 (highest common, or 0x0000 for none).
- reserved: u16, zero.
- echo_nonce: u64 (echoed probe_nonce).

Negotiation algorithm:
- Initiator sends probe with supported versions.
- Responder selects max(intersection(probe.versions, local.versions)).
- Empty intersection: respond with selected_version = 0x0000.
- Timeout: 5 seconds, 3 retries, then abort with VERSION_TIMEOUT.

Security:
- The responder SHOULD rate-limit VersionResponse to at most 100
  responses per second per source address.
- Selected version is bound into the Noise prologue:
  prologue = 0x56524B4F || u16(version) || u8(hs_type) = 7 bytes.
- If a Noise handshake fails immediately after a successful version
  probe, the initiator SHOULD retry with direct version specification
  (highest supported version, no probe) to detect downgrade attacks.

### 3.1 Wire format

```
Outer header (plaintext, read by transit peers):
  [u32 destination]        4 bytes  -- BLAKE3(CIK_pubkey)[0..4]
  [u32 connection_id]      4 bytes  -- O(1) session lookup, path validation
  [u8  hop_count]          1 byte   -- TTL, max 4 for 12-cluster mesh
  [u64 relay_key_id]       8 bytes  -- HMAC(channel_salt, key_id)
  [u96 nonce]             12 bytes  -- sequential monotonic counter per-sender per-session
                          29 bytes outer preamble

Encrypted payload (ChaCha20-Poly1305 [RFC8439] AEAD):
  [u16 channel_index]      2 bytes  -- numeric, not string
  [u32 epoch]              4 bytes  -- macro-epoch (full u32 G-Counter)
  [varint micro_epoch]     1 byte   -- micro-epoch counter (1 byte typical, 2 bytes high activity)
  [u16 dest_peer]          2 bytes  -- intra-cluster target (0 = broadcast)
  [u8  msg_type]           1 byte   -- heartbeat / gossip / control / chunk
  [data]                  variable
                          10 bytes inner overhead (typical)

  [u128 auth_tag]         16 bytes  -- Poly1305 [Bernstein2005]

Total fixed overhead: 29 + 10 + 16 = 55 bytes per frame (typical).
```

**Replay detection:** separate sliding window W=1024 (IPsec RFC 4303 [RFC4303] style). 128 bytes per session.

**Per-channel relay salt:**
```
channel_salt = HKDF-Expand(prk=channel_key, info="relay:salt:" || floor(unix_days/90), len=32)
relay_key_id = HMAC-BLAKE3(key=channel_salt, data=key_id)[0..8]
```
Immediate rotation on revocation.

### Data Frame Format

Outer Header (29 bytes, plaintext):
    Offset  Width  Field
    ------  -----  -----
     0      4      destination: u32 (BLAKE3(CIK_pub)[0..4])
     4      4      connection_id: u32
     8      1      hop_count: u8 (max 4, decremented per relay)
     9      8      relay_key_id: u64
    17     12      nonce: u96 (sequential monotonic per-session)

Encrypted Payload (inside ChaCha20-Poly1305 AEAD):
    Offset  Width  Field
    ------  -----  -----
     0      2      channel_index: u16 (0xFFFF = control)
     2      4      epoch: u32
     6      1-2    micro_epoch: varint (LEB128)
     7-8    2      dest_peer: u16 (0x0000 = broadcast)
     9-10   1      msg_type: u8 (bit 7 = SACK piggybacked)
    10-11   var    data
    end    16      auth_tag

AEAD associated data: outer_header[0..8] || outer_header[9..29] (28
bytes). The hop_count field (offset 8, 1 byte) is excluded from AEAD
AD because relay peers must decrement hop_count during forwarding.
Excluding hop_count preserves authentication of destination (4 bytes),
connection_id (4 bytes), relay_key_id (8 bytes), and nonce (12 bytes).

Security analysis: an attacker who modifies hop_count can set it to 0
(frame dropped, equivalent to packet drop) or increase it up to max 4
(bounded amplification). Neither threatens confidentiality or integrity.
Precedent: IPsec ESP (RFC 4303 Section 3.3.3.1.1) excludes mutable IP
header fields (TTL) from ICV computation. QUIC (RFC 9001 Section 5.4)
encrypts the packet number separately for similar reasons.

Implementation: ChaCha20-Poly1305 supports multi-part associated data
via incremental authentication. The AD is fed in two parts:

```
fn build_aead_ad(outer_header: &[u8; 29]) -> [&[u8]; 2] {
    [&outer_header[0..8], &outer_header[9..29]]
}
```

msg_type registry:
    0x01 = HEARTBEAT         0x08 = KEY_BUNDLE
    0x02 = GOSSIP            0x09 = RELAY_OP
    0x03 = CONTROL           0x0A = FILTER_UPDATE
    0x04 = CHUNK             0x0B = SCHEMA_SYNC
    0x05 = SACK              0x0C = CHAIN_PROOF
    0x06 = PATH_CHALLENGE    0x0D = TRANSFER_REQUEST
    0x07 = PATH_RESPONSE     0x0E = TRANSFER_BEGIN
    0x10 = CONN_CLOSE        0x0F = TRANSFER_ABORT

### 3.3 Transport (verkko)

**Segmentation.** UDP payload limit: 1,232 bytes (IPv6 minimum MTU 1,280 - IPv6 header 40 - UDP header 8 = 1,232). PMTU probing on connection establishment.

Maximum data payload per frame:
    1232 - 29 (outer) - 10 (inner) - 7 (stream) - 16 (auth tag) = 1170 bytes

**LEDBAT congestion control** [RFC6817] with adaptive target delay and dual controllers:

| State | Target delay | Bandwidth |
|-------|-------------|-----------|
| Background | 10ms | 30-50% |
| Idle | 25ms | 50-70% |
| Active | 50ms | 70-90% |
| Stress | 100ms | 80-90% |

Target delay is a resource-budget parameter derived from the metabolic state (see MetabolicState). Measured delay (`min_rtt`, `current_delay`) enters through convention-enforced boundary gates independently.

Dual controllers for uplink/downlink on asymmetric links. Timestamp source: kernel SO_TIMESTAMPING (~100us accuracy).

**Coupled congestion control at gateway.** Single LEDBAT controller per physical uplink. Fair-share weighted scheduler across 11 inter-cluster connections. Cross-connection vital queue guarantee.

**Explicit congestion feedback** in SACK ACKs (3 bytes):
```
[u8 recv_buffer_pct]    -- 0-100
[u16 recv_rate_kbps]    -- over last RTT
```

**Three-tier send queue with weighted vital fair queuing:**
```
Queue 1 (Vital):     Heartbeats, KeyBundle, Noise handshakes, epoch advance.
                     Cap: min(4KB, available_bandwidth * 0.25).
                     Internal: 70/30 weighted fair queuing (control/gossip).
                     Control sub-queue: heartbeats, KeyBundle, Noise, epoch.
                     Gossip sub-queue: gossip mode transitions, flood messages.
                     Both sub-queues guaranteed minimum share. Neither starves.

Queue 2 (Reactive):  Gossip, filter updates, control log, key requests.
                     Interleaved with bulk at 1:4 ratio.

Queue 3 (Bulk):      Chunks, relay blobs, re-encryption, donation.
                     Governed by LEDBAT.
```

The 70/30 split ensures that during concurrent enrollment plus CPSK bursts, heartbeats continue flowing (2.4 KB/s gossip share at 8 KB/s vital budget provides 87x headroom over 27.5 B/s heartbeat requirement in Stress mode). CPSK messages at 128 bytes see 23ms queuing delay, well within the 2000ms polygon schedule round.

### Reliability Modes

All streams are reliable-ordered. Heartbeats and standalone SACKs are
connection-level (no stream header) and inherently unreliable.

Gossip IHAVE messages MUST use unreliable datagrams at the protocol
level. This is an arena-driven design requirement: during convergence,
a gossip burst of 2,200+ frames would require 2.7 MB of retransmission
state if sent on reliable-ordered streams, exceeding the 256 KB pool
arena by 10x. The mesh layer's Plumtree IHAVE/GRAFT mechanism provides
gossip reliability independently of stream-level retransmission.

Retransmission state per reliable stream:

```rust
struct StreamRetransmitState {
    unacked: RingBuf<UnackedFrame, 32>,
    rto_ms: u32,
    retransmit_count: u8,
    cumulative_ack: u32,
    sack_blocks: [SackBlock; 4],
    sack_block_count: u8,
    recv_window: u32,
}
```

Memory per stream: ~816 bytes. At 11 connections * 10 active streams
each = ~88 KB. Fits within the pool arena (256 KB).

### SACK Format

SACK is carried as msg_type = 0x05 (standalone) or piggybacked
via bit 7 of msg_type on any data frame.

SACK payload:
    Offset  Width  Field
    ------  -----  -----
     0      4      cumulative_ack: u32
     4      4      ack_timestamp: u32 (truncated HLC)
     8      2      ack_delay: u16 (microseconds, max 65535)
    10      1      block_count: u8 (0-16)
    11      1      recv_buffer_pct: u8 (0-100)
    12      2      recv_rate_kbps: u16
    14      4      recv_window: u32 (remaining bytes)
    18      var    sack_blocks: [SackBlock; block_count]

SackBlock (8 bytes):
    [u32 block_start] [u32 block_end] (exclusive end)

Fixed header: 18 bytes. Per block: 8 bytes.
Maximum (16 blocks): 146 bytes.

Note: The SACK carries both congestion feedback fields
(ack_timestamp, ack_delay, recv_rate_kbps, recv_buffer_pct
for LEDBAT) and flow control (recv_window for per-stream
backpressure). This hybrid avoids duplicating congestion
feedback in every stream's SACK.

Field ordering is canonical: ACK fields (offset 0-9), then
metadata (block_count, offset 10), then congestion feedback
(offset 11-13), then flow control (recv_window, offset 14),
then variable-length blocks (offset 18). Naturally aligned.

**Connection migration.** QUIC-style PATH_CHALLENGE/PATH_RESPONSE [RFC9000] (RFC 9000 Section 9). Rate-limit: 1KB/s before validation.

### ConnectionClose

Explicit connection termination. msg_type = 0x10.

Wire format (inside encrypted payload):
    Offset  Width  Field
    ------  -----  -----
     0      1      reason: u8
     1      1      reserved: u8 (zero)
     2      2      detail_len: u16
     4      var    detail: [u8; detail_len] (UTF-8, max 256 bytes)

Reason codes:
    0x00 = NORMAL           -- clean shutdown
    0x01 = IDLE_TIMEOUT     -- no traffic for timeout period
    0x02 = KEY_EXPIRED      -- session keys expired
    0x03 = MIGRATION_FAIL   -- PATH_RESPONSE not received
    0x04 = INTERNAL_ERROR   -- local error
    0x05 = VERSION_MISMATCH -- discovered after SchemaSync
    0x06 = HEARTBEAT_TIMEOUT -- 3 missed heartbeats
    0x07 = PROTOCOL_ERROR   -- protocol violation
    0xFF = UNSPECIFIED       -- generic close

Close handshake:
1. Initiator sends CONN_CLOSE frame with reason.
2. Responder drains pending data, sends its own CONN_CLOSE.
3. Both sides enter CLOSING state: accept no new stream data,
   process remaining SACKs and CONN_CLOSE frames, discard
   other frame types.
4. After min(2*RTT, 5000ms): destroy session state and
   zeroize keys.

Abrupt close: on fatal protocol violation, a peer MAY send
CONN_CLOSE with reason PROTOCOL_ERROR and immediately destroy
session state without waiting for a response.

Heartbeat timeout close: when heartbeat timeout fires (3
missed heartbeats), the peer enters CLOSING state as if it
had received CONN_CLOSE with reason HEARTBEAT_TIMEOUT.

Interaction with in-progress transfers:
- CONN_CLOSE implicitly aborts all in-progress transfers.
- Peers SHOULD NOT send TransferAbort for transfers on a
  connection that has sent or received CONN_CLOSE.
- Implicitly aborted transfers do not generate scars.

**Clock synchronization.** Cristian's algorithm [Cristian1989] during Noise handshake (32 bytes). PLL correction via heartbeat HLC (alpha = 0.01, verkko-ops: Pattern Registry Structure 2 instance).

**CPSK management.** Noise KK [Perrin2018] for established pairs. Symmetric ratchet every 2 minutes, full re-handshake every 30 minutes. First-time: Noise XX (2 round trips). On KK failure: fall back to XX with TOFU + admin notification.

**CPSK epoch binding.** Gateway singularity is enforced per connected component. During partition, each partition may independently elect a gateway. On partition heal, CPSK sessions are renegotiated.

### Noise Handshake Framing

Handshake Frame (17-byte fixed header):
    Offset  Width  Field
    ------  -----  -----
     0      4      magic: 0x56524B4F ("VRKO")
     4      1      frame_type: 0x03 (handshake)
     5      1      hs_type: 0x01=KK, 0x02=XX, 0x03=RESUME
     6      1      msg_index: 0, 1, or 2
     7      4      initiator_cid: u32 (random_u32)
    11      4      responder_cid: u32 (0x00000000 in msg 0)
    15      2      hs_payload_len: u16
    17      var    noise_payload

Noise protocol names:
    Noise_KK_25519_ChaChaPoly_BLAKE2b
    Noise_XX_25519_ChaChaPoly_BLAKE2b

Prologue (7 bytes, not on wire):
    prologue = 0x56524B4F || u16(selected_version) || u8(hs_type)

KK-to-XX fallback: on KK decryption failure, discard KK state,
generate new ephemeral, send XX msg 0 with hs_type=0x02. Fire
admin notification (TOFU alert).

Distinguishing pre- and post-handshake datagrams: check first
4 bytes. If == 0x56524B4F, it is pre-handshake. Otherwise, first
4 bytes are the destination field of a data frame.

**NAT traversal.** Adaptive keepalive: discover mapping lifetime via probes, set interval to 70% of lifetime. Prewarm relay path when lifetime < 60 seconds.

**WiFi-layer rate ceiling for coupled LEDBAT.** On WiFi links, the LEDBAT controller's target rate is capped at the WiFi link's observed capacity. WiFi capacity estimated from `MCS_throughput * (1 - retry_fraction)` via nl80211.

### SansIO Consumer Interface

The protocol is a sans-IO state machine. The consumer drives I/O.

ProtocolInput events (fed to the state machine by the consumer):
- BytesReceived { source: opaque, data: bytes }
- ConnectionRequested { peer_identity, address }
- ConnectionClosed { reason }
- StreamCreated { stream_id, tier }
- StreamClosed { stream_id }
- HeartbeatTick
- ConfigChanged { governor_output }
- CryptoResult { request_id: u32, result: CryptoVerifyResult }
- ResourceMeasurement { resource: ResourceType, value: CapacityRatio }

```rust
#[repr(u8)]
enum CryptoVerifyResult {
    Valid = 0,
    Invalid = 1,
    Error = 2,
}

#[repr(u8)]
enum ResourceType {
    CpuPressure = 0,
    FlashIoPressure = 1,
    NonceIoPressure = 2,
    TransportCapacity = 3,
    WifiQuality = 4,
}
```

The consumer periodically samples OS-level resource statistics, converts
them to Q16.16 CapacityRatio via the PhysicalBoundary gate, and feeds
them to the protocol state machine. The MetabolicGovernor uses all five
dimensions in its min-projection. WifiQuality maps to CapacityRatio
directly: 1.0 means full delivery, 0.85 means 85% delivery ratio. No
new PhysicalBoundary newtype needed; CapacityRatio already captures the
semantics.

ProtocolOutput events (emitted by the state machine):
- SendBytes { destination: opaque, data: bytes }
- SessionEstablished { connection_id, peer }
- SessionLost { connection_id, reason }
- FrameReady { connection_id, stream_id, payload }
- HeartbeatReceived { connection_id, transport_header, extension }
- SackReceived { connection_id, sack_data }
- PathChallengeReceived { connection_id, challenge }
- SchemaRequired { connection_id }
- TransferRequested { connection_id, transfer_request }
- ConnectionClosing { connection_id, reason }
- SessionResumed { connection_id, peer }
- NeedArena { min_bytes }
- CryptoRequest { request_id: u32, operation: CryptoOperation }

```rust
enum CryptoOperation<'a> {
    VerifyEd25519 {
        public_key: &'a [u8; 32],
        message: &'a [u8],
        signature: &'a [u8; 64],
    },
    VerifyBao {
        root_hash: &'a [u8; 32],
        chunk_index: u32,
        chunk_data: &'a [u8],
        proof: &'a [u8],
    },
}
```

When the protocol needs crypto verification (control log entry, matter
chunk), it emits CryptoRequest and enters a "pending verification"
state for that connection/stream. It continues processing other
connections normally. When CryptoResult arrives via handle(), the
protocol resumes processing the verified frame. AEAD (ChaCha20-
Poly1305) remains synchronous inline (~1us, too fast to warrant async).

Timer management is internal to the protocol state machine. The
consumer calls `next_deadline()` to get the earliest deadline across
all connections, then calls `handle_timeout(now_ms, arena)` when the
deadline elapses. The protocol processes all expired timers internally.

SetTimer and CancelTimer are removed from the ProtocolInput/
ProtocolOutput enumerations. The consumer tracks one deadline (the
earliest returned by `next_deadline()`). Timer types (heartbeat, RTO,
path challenge, CPSK renegotiation, connection close drain, metabolic
dwell) are internal protocol state.

Mesh-layer timers (convergence wavefront, convergence duration bound,
dwell time restoration) are NOT protocol timers. The consumer manages
mesh-layer timers separately. The consumer's timer wheel holds both
protocol deadlines (from `next_deadline()`) and mesh-layer deadlines.

The source of bytes is opaque. The consumer may deliver bytes from
UDP datagrams, WebSocket messages, TCP streams, or any other transport.
The protocol machine MUST NOT assume that each BytesReceived event
corresponds to exactly one UDP datagram.

## SansIO Polling Model

The Protocol trait is the sole entry point for the consumer (event loop)
to interact with the protocol state machine. All I/O is external to the
trait. The protocol never touches a socket, timer, or filesystem.

### Protocol Trait (Rust Implementation)

```rust
trait Protocol {
    /// Process one input event. May mutate internal state. May enqueue
    /// outputs. Returns Ok(()) on success, Err on fatal error.
    /// On ArenaExhausted: consumer must provide more arena, then retry.
    fn handle<'a>(
        &mut self,
        input: ProtocolInput<'a>,
        arena: &'a ArenaRef<'a>,
    ) -> Result<(), ProtocolError>;

    /// Drain the next pending transmission. Returns None when no more
    /// are pending. Transmissions returned in priority order (vital
    /// first). The returned Transmit borrows from the arena.
    fn poll_transmit<'a>(
        &mut self,
        arena: &'a ArenaRef<'a>,
    ) -> Option<Transmit<'a>>;

    /// Drain the next pending application event. Returns None when no
    /// more are pending. Events borrow from the arena.
    fn poll_event<'a>(
        &mut self,
        arena: &'a ArenaRef<'a>,
    ) -> Option<Event<'a>>;

    /// Get the next timer deadline across all connections.
    /// Returns None if no timers are active.
    fn next_deadline(&self) -> Option<Instant>;

    /// Process all timers that have expired at or before now_ms.
    fn handle_timeout<'a>(
        &mut self,
        now_ms: u64,
        arena: &'a ArenaRef<'a>,
    ) -> Result<(), ProtocolError>;

    /// Returns true if any CryptoRequest outputs are pending.
    fn has_pending_crypto(&self) -> bool;
}
```

### Error Type

```rust
#[repr(u8)]
enum ProtocolError {
    ArenaExhausted = 0,
    CryptoFailure = 1,
    ProtocolViolation = 2,
    NonceHalted = 3,
}
```

### Calling Convention

```
loop {
    let datagrams = socket.recvmmsg();
    for dgram in datagrams {
        let input = ProtocolInput::BytesReceived {
            source: dgram.addr.into(),
            data: dgram.payload,
        };
        match protocol.handle(input, &arena) {
            Ok(()) => {},
            Err(ProtocolError::ArenaExhausted) => {
                drain_all(&mut protocol, &arena, &socket);
                arena.reset();
                protocol.handle(input, &arena).unwrap();
            },
            Err(e) => { log_error(e); }
        }
    }

    // Process expired timers
    if let Some(deadline) = protocol.next_deadline() {
        if now >= deadline {
            protocol.handle_timeout(now_ms, &arena).ok();
        }
    }

    // Drain transmissions (sent to network)
    while let Some(tx) = protocol.poll_transmit(&arena) {
        socket.sendto(tx.destination, tx.payload);
    }

    // Drain application events (dispatched to mesh layer)
    while let Some(event) = protocol.poll_event(&arena) {
        mesh.handle_event(event);
    }

    // Reset arena for next iteration
    arena.reset();
}
```

### Buffer Ownership Rules

- `ProtocolInput::BytesReceived::data`: borrowed for the duration of
  `handle()`. The protocol copies what it needs into the arena or
  managed state.
- `Transmit::payload`: borrows from the arena. Valid until
  `arena.reset()`.
- `Event` payloads: borrow from the arena. Valid until `arena.reset()`.
- The consumer MUST drain all `poll_transmit()` and `poll_event()`
  before calling `arena.reset()`.

### Backpressure

- If the internal queue is full, subsequent `handle()` calls that would
  enqueue new transmissions return `Err(ArenaExhausted)`.
- Vital-tier frames are never dropped due to arena exhaustion. The
  protocol reserves a fixed portion of the arena for vital frames.
- If the vital reserve is exhausted, the protocol enters degraded mode:
  heartbeats and SACKs only; all other frames dropped until arena is
  drained.

### Reentrancy

`handle()` MUST NOT be called while a previous `handle()` is in
progress. All methods take `&mut self`, enforcing exclusive access at
the type level in Rust. For non-Rust implementations: the polling model
is structurally non-reentrant because output methods do not accept
inputs.

### Abstract Specification (Language-Neutral)

The abstract interface is two-phase: `feed(input)` produces state
changes; `poll_output()` drains all consequences. `next_deadline()`
queries the earliest timer. This provides structural reentrancy
prevention independent of any language's type system. The Rust 4-method
model is one valid optimization of this abstract interface.

### 3.4 Adaptive heartbeat

**Payload (18 bytes fixed header + 18 bytes extension region = 36 bytes inside encrypted frame):**

Fixed header (transport-layer fields):
```
  [u8  state]              1 byte   -- bits 0-1: metabolic state; bit 2: gossip_mode; bit 4: admin flag
  [u16 rtt_us]             2 bytes
  [u16 loss_permille]      2 bytes  -- 0-1000 (diagnostics only, not used for gateway election)
  [u16 jitter_us]          2 bytes  -- 99th percentile, max 65ms
  [u8  queue_depth]        1 byte   -- 0-100%
  [u16 available_bw_kbps]  2 bytes
  [u64 hlc_timestamp]      8 bytes
```

Extension region (mesh-defined fields; see verkko-mesh for semantics):
```
  [u64 filter_root]        8 bytes  -- keyed truncated Merkle root
  [u32 epoch_current]      4 bytes  -- G-Counter sync (macro-epoch)
  [u8  work_capacity]      1 byte   -- re-encryption/donation capacity
  [u8  health_canary]      1 byte   -- top 4: worst classification; bottom 4: degraded count
  [u32 observation_summary] 4 bytes -- aggregated cross-cluster observation
```

The extension fields (filter_root, epoch_current, work_capacity, health_canary, observation_summary, gossip_mode flag) carry mesh-layer semantics. The protocol layer defines the frame format and transport-layer fields; the mesh layer defines extension field contents. See verkko-mesh: Heartbeat extension fields.

Total frame: 29 + 10 + 36 + 16 = 91 bytes.

**work_capacity computation note:** `min(bw_kbps/128, flash_kb/256, 255)`. Intermediate computation must use u16 or u32 before the final min with 255 to avoid silent overflow.

**Gossip mode flag (bit 2 of state byte).** 0 = Plumtree, 1 = flood.

| State | Interval | Wire cost (11 connections) |
|-------|----------|---------------------------|
| Background | 30s | ~37 B/s |
| Idle | 20s | ~55 B/s |
| Active | 5s | ~220 B/s |
| Stress | 2s | ~550 B/s |

Heartbeat interval is a resource-budget parameter derived from the metabolic state (see MetabolicState).

**Heartbeat jitter.** DK-hash-derived per-heartbeat jitter to prevent WiFi PSM synchronization. Jitter range: 20% of interval during steady state, 40% during convergence. Computed as `BLAKE3("saalis-hbjitter-v1" || DK_pub || epoch)[0..2] as i16 % (base_interval_ms * jitter_range_fraction)`. Zero additional bytes.

**Redundancy on lossy links.** When receiver-observed delivery ratio drops below 90%, send duplicate heartbeats. At 5% frame loss: one duplicate. At 20% loss: two duplicates.

### 6.2 Event loop and threading

- **Event loop thread:** single-threaded, edge-triggered epoll with io_uring upgrade path. Owns NonceSource [0, 2^95) (see verkko-crypto: NonceSafety).
- **Crypto thread:** dedicated for Ed25519, bao-tree verification. SPSC ring buffers.
- **Batch AEAD thread:** dedicated for batch AEAD operations. Owns NonceSource [2^95, 2^96). Pre-batch persist on this thread's NonceSource before each batch.
- **Write manager:** coalesced flash writes. Nonce HWMs persisted via double-write protocol.

**Syscall strategy.** Minimum (kernel 4.14+): sendmmsg, recvmmsg, SO_TIMESTAMPING, timerfd. Preferred (5.19+): io_uring.

**Timestamp architecture.** Four domains: measurement (kernel SO_TIMESTAMPING), logical (HLC [Kulkarni2014]), monotonic (VDSO CLOCK_MONOTONIC), wall-clock (VDSO CLOCK_REALTIME + PLL).

### 6.3 Metabolic state machine with phase jitter and Conductor invariant

Four states: Background, Idle, Active, Stress. Two state machines: MetabolicState and OperationalMode. Separate constructs with a sequencing invariant (see below). **Minimum dwell time:** 4 heartbeat intervals (20 seconds at 5-second heartbeats).

MetabolicState transitions are driven by a governor that provides metabolic state as input. The governor is a mesh-layer concept; see verkko-mesh: Conductor for the governor specification. The protocol layer defines the state machine and its parameter maps; the mesh layer drives transitions.

**Convergence metabolic override.** On convergence entry, the metabolic state machine immediately transitions to Active or Stress (whichever is appropriate given current resource readings). Dwell time enforcement is suspended for this transition only.

**Metabolic phase jitter:**
```
phase_delay_ms = BLAKE3("saalis-jitter-v1" || DK_pub || current_epoch)[0..2] as u16 % (dwell_time_ms / 4)
```
At 20-second dwell: up to 5 seconds of stagger per peer. Recomputed per epoch. Reduces Kuramoto order parameter [Kuramoto1975] from ~1.0 to ~0.5-0.7.
