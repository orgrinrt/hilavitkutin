# Systems Integrator Review: Verkko Design Decomposition

**Date:** 2026-03-28
**Reviewer identity:** Systems integrator, 15 years building on protocol specs
**Documents reviewed:** verkko-crypto (535 lines), verkko-protocol (420 lines), verkko-mesh (1,464 lines), verkko-relay (189 lines), verkko-seat (113 lines), verkko-matter (237 lines), verkko-taxonomy (366 lines)
**Review question:** Can a team of engineers who have never seen v10 build a working system from these six documents?

---

## 1. Per-document implementability assessment

### 1.1 verkko-crypto

**Verdict: Implementable with one blocking gap.**

This is the strongest document. The defined concepts are crisp: DK, DEK, ChannelKey, EpochRatchet, MicroEpochRatchet, FrameKey, TransportAEAD, StorageAEAD, NonceSafety, DoubleWritePersist, BroadcastEncryption, PCSMechanism, DomainSeparatorRegistry, KeyZeroization. Each has clear invariants and concrete construction details.

An engineer could implement:
- Key generation (DK, DEK, CIK, signing key) from the genesis sequence.
- The epoch and micro-epoch ratchets from the HKDF-Expand formulas.
- Both AEAD constructions from the explicit algorithm choices and nonce strategies.
- The three-layer nonce safety mechanism from the detailed Layer 1/2/3 descriptions.
- The double-write persist from the explicit byte layout and recovery protocol.
- Broadcast encryption from the BroadcastEnvelope struct.
- The BLAKE3 domain separator registry from the table.

**Where an engineer would get stuck:**

- The BLAKE3 test vector table is empty ("defined at implementation time"). This is a blocking gap. The document itself says "there is no runtime detection mechanism for BLAKE3 divergence" and "prevention is the only defense." You cannot ship prevention without test vectors.
- The enrollment cert format is described but the signing payload construction (what bytes are signed, in what order, with what framing) is split between verkko-crypto (genesis bootstrap) and verkko-mesh (enrollment binding). An implementor must read both and synthesize.
- The `algorithm: u8` field in BroadcastEnvelope has no registry of values. There are two AEAD constructions (transport and re-encryption), but which u8 maps to which?
- How CIK is "replicated to all cluster peers, encrypted to each DEK" is stated as fact but the mechanism is not specified. Is it a KeyBundle? A direct Noise session? This falls between verkko-crypto and verkko-mesh.

### 1.2 verkko-protocol

**Verdict: Partially implementable. Missing pieces are significant but not blocking.**

The wire format is fully specified at the byte level: 29 bytes outer preamble, 10+ bytes encrypted payload, 16 bytes auth tag. The heartbeat payload is byte-level specified (36 bytes). LEDBAT parameters are tabulated per metabolic state. The three-tier send queue is described with capacity caps and weighted fair queuing ratios.

An engineer could implement:
- Frame serialization and deserialization.
- The heartbeat payload.
- The LEDBAT congestion controller (given RFC 6817 as supplementary reading).
- The metabolic state machine with phase jitter.
- The replay window (W=1024, IPsec-style).
- Clock synchronization (Cristian's algorithm during handshake, PLL in heartbeat).

**Where an engineer would get stuck:**

- Stream multiplexing is listed in Defined Concepts with invariants (no head-of-line blocking, independent ordering, bounded count) but the body has no multiplexing specification. How are streams identified in the wire format? How does flow control work per-stream? The wire format shows no stream ID field.
- SchemaSync is defined conceptually (client sends hash, server compares, sends update on mismatch) but there is no wire-level specification. What is the protobuf schema exchange format? When in the handshake does it happen? What is the schema hash algorithm?
- 0-RTT resumption is mentioned in the taxonomy as missing. There is no specification in the document.
- The Connection concept mentions a `connection_id` but the handshake sequence that establishes it is not specified. How is connection_id generated? The wire format shows `[u32 connection_id]` but not its derivation.
- NAT traversal is defined conceptually but the hole-punching procedure is not specified. STUN? ICE-lite? Custom? The build order in verkko-mesh mentions "STUN" but verkko-protocol says "hole punching attempted before relay fallback" without details.
- The SACK ACK format is partially specified (3 bytes congestion feedback) but the SACK bitmap itself is not described. How are selective acknowledgements encoded? What does a SACK ACK frame look like?
- PATH_CHALLENGE/PATH_RESPONSE is referenced to QUIC RFC 9000 Section 9 but not specified inline. Does an implementor need to read the QUIC RFC?
- The `destination` field in the outer header is `BLAKE3(CIK_pubkey)[0..4]`. This is 4 bytes, giving a 32-bit address space. Birthday collision at 2^16 = 65,536 clusters. With a 12-cluster ceiling this is fine, but the document does not state this analysis or confirm that collision handling is unnecessary.
- Explicit congestion feedback format (3 bytes in SACK ACK) is specified, but the SACK ACK itself is not framed. Is it a separate frame type? A field inside heartbeat? A response piggybacked on data?

### 1.3 verkko-mesh

**Verdict: Implementable for core mechanisms. Extremely dense; requires careful cross-referencing.**

This is the largest document (1,464 lines) and contains the most sophisticated mechanisms. The defined concepts are well-structured: Cluster, CIK, CPSK, Enrollment, EDT, KeyBundle, DominanceCascade, HealthPipeline, Scar, CircuitBreaker, Gossip, Convergence, PartitionAnnouncement, Gateway, ResourceGovernance, InviteFlow, LKH.

An engineer could implement:
- Enrollment binding from the explicit enrollment_cert construction.
- The KeyBundle format and conflict resolution rules (Revoke > Grant > Rotate).
- The dominance cascade from the explicit formula and argmin rule.
- The HealthPipeline including dual EMA, PhysicalBoundary gates, NaN guard, scar timing, circuit breaker.
- The convergence three-front sequence with gates and timeouts.
- The polygon CPSK renegotiation schedule from the 1-factorization construction.
- PartitionAnnouncement processing with observer quorum.
- Scar provenance, decay, and the three-consumer cospan.
- Observation-path forwarding with FlowSummary.
- Gateway election via max-register CRDT with ETT tiebreaker.
- Resource governance (GovernedResource and ThresholdGuard).

**Where an engineer would get stuck:**

- The InviteFlow defined concept says "Invite contains: mesh identity, bootstrap peer address, PSK for first handshake" but the body has no invite flow section. How is the invite token constructed? What is the QR code format? How does the bootstrap peer validate the invite? The taxonomy acknowledges this is a critical gap.
- The control log GSet merge is described at a high level but the merge semantics under concurrent admin operations are complex. What happens when two partitions each issue enrollments? The document says "Grants: auto-ratified if valid enrollment cert" which answers this, but the merge of enrollment certs from different partitions' admins needs more precision.
- The max-register CRDT for gateway election is mentioned but not formally specified. What is the register value? How is "availability" measured and compared?
- The "Conductor invariant" says all resource-budget parameters factor through metabolic state, but the mapping functions `f_p(metabolic_state)` are not enumerated. An implementor knows heartbeat interval maps to metabolic state (the table is given), but what about other parameters? LEDBAT target delay is tabulated. What else?
- Intra-cluster communication model is implicit. The document says "reliable broadcast (iterated unicast)" for vital messages within a cluster, but there is no specification of intra-cluster session establishment, key management, or frame routing.

### 1.4 verkko-relay

**Verdict: Partially implementable. The "what" is clear; the "how" for transit forwarding is not.**

The defined concepts cover TransitRelay, RelayTopology, Membrane, TerritorialReencryption, IdempotentPUT, ConstantRatePadding, and RelayCorrelationThreatModel. The membrane model (5 ops, TTL, TOUCH) is concrete. The territorial re-encryption with consistent hashing, ring coloring, and deterministic nonces is well-specified.

An engineer could implement:
- The membrane storage layer (PUT, GET, DELETE, TOUCH, SIGNAL).
- The consistent hash ring with 32 vnodes/peer and ring coloring.
- The ring coloring algorithm (greedy O(N) reassignment).
- Deterministic re-encryption nonces from the BLAKE3 formula.
- Idempotent PUT from the relay_key_id collision check.
- Constant-rate padding from the batched window description.
- Tiered re-encryption deadlines (P1-P4).

**Where an engineer would get stuck:**

- Transit relay forwarding is listed as a defined concept but there is no body section on it. How does a relay peer decide to forward a frame? Does it inspect the outer header's destination field? Does it maintain a routing table? How is the relay path established? The document says "frames without decrypting content" but not the forwarding decision logic.
- RelayTopology is defined conceptually (selection based on connectivity graph and health) but there is no algorithm. How does a peer discover which relay to use? How is relay selection communicated? Is it part of the heartbeat? Part of discovery?
- The SIGNAL operation is listed as one of the 5 membrane ops but never defined. What does SIGNAL do? The document says "connection setup" but no more.
- The relationship between relay peers and mesh peers is ambiguous. Is a relay peer always a mesh peer? The abstract says "not a full mesh participant in the content sense" but the territorial re-encryption section talks about peers on the hash ring. Are relay-only nodes possible?
- BatchGET mentions "10 real + randomized decoys" but does not specify how decoys are generated. Are they random relay_key_ids? How does the relay handle a GET for a nonexistent key? Is the response distinguishable from a decoy response?

### 1.5 verkko-seat

**Verdict: Not implementable. Concept definition only; no protocol specification.**

The defined concepts (Seat, SeatSession, HostPeer, SeatRequest, SeatEvent, SeatInstruction, SeatProgress) are clean and the invariants are clear. But the body is empty. There is no wire format, no authentication protocol, no request/response framing, no event subscription mechanism, no instruction encoding.

The taxonomy acknowledges this: "~0%. The concept is established but no design exists."

**An engineer would need answers to:**

- How does a seat authenticate? The document says "user credential (not a DK/DEK)" but what credential type? Password? Token? Certificate?
- What Noise pattern is used for seat sessions? Or is it TLS? Or something else?
- How is 0-RTT resumption implemented for seats?
- What is the protobuf schema for SeatRequest, SeatEvent, SeatInstruction, SeatProgress?
- How does the host peer map seat roles to mesh access control?
- What happens when the host peer restarts? Does the seat reconnect transparently?
- How does event subscription work? Subscribe-by-type at session setup? Modifiable mid-session?

### 1.6 verkko-matter

**Verdict: Partially implementable. Core mechanisms specified; transfer protocol missing.**

The defined concepts cover Matter, MatterIntegrity, ContentFilter, MerkleReconciliation, MatterTransfer, Stash, Replication, SyncReconciliation, SealedRange, and BandwidthPolicy. The cuckoo filter specification is concrete (10-bit fingerprints, 2-bit saturating counters, 4-entry buckets, 90% load, 162KB/filter). The sealed-range catch-up protocol is well-specified with the three-state model.

An engineer could implement:
- The counting cuckoo filter with the specified parameters.
- The Merkle reconciliation protocol (depth-8 tree, dirty-bit optimization).
- The sealed-range catch-up (three states, SealedReason enum, CatchUpGrant).
- Content integrity (blob_id = BLAKE3(plaintext), IntegrityFailure scar).
- Source selection predicate.

**Where an engineer would get stuck:**

- The matter transfer protocol is not specified. How does a peer request matter from another peer? What is the request/response format? How are chunks sized? The document says "LAN path: direct streaming" and "WAN path: bao-tree verified streaming" but there is no wire-level protocol for either.
- Stash is defined conceptually but there is no storage format. How is the stash persisted? What data structures?
- Replication targets and background replication work are not specified. How does a peer decide what to replicate? How is replication priority determined?
- SyncReconciliation mentions "merge rules: field-level, deterministic, reproducible" but does not define what fields exist or what the merge rules are.
- BandwidthPolicy mentions "policies published via gossip (time windows, rate limits)" but there is no policy format, no gossip integration, no enforcement mechanism.
- The relationship between matter and relay blobs is implicit. Is a matter that lives on a relay stored as a single blob or chunked? The relay membrane deals in blobs; matter can be 2GB. How are large matters handled in the relay?

## 2. Build order recommendation

The taxonomy proposes a dependency graph:

```
verkko-crypto
     |
verkko-protocol
   |    |        \
verkko-mesh  verkko-seat  verkko-relay
     |          /
verkko-matter
```

This is correct and the documents respect it. My recommended implementation build order, accounting for testability at each stage:

**Phase 1: verkko-crypto (standalone, fully testable)**
- All key generation, HKDF ratchets, AEAD constructions, nonce safety, double-write persist.
- Can be unit-tested in isolation. No network needed.
- Blocking prerequisite: fill in the BLAKE3 test vector table.

**Phase 2: verkko-protocol (depends only on verkko-crypto)**
- Wire format, frame serialization, replay window, LEDBAT, heartbeat, metabolic state machine.
- Can be tested with a sans-IO harness (feed bytes in, collect bytes out).
- Does NOT require Noise handshake to be tested (can use fixed session keys).
- Missing pieces (stream mux, schema sync, 0-RTT) can be deferred.

**Phase 3a: verkko-mesh (depends on crypto + protocol)**
- Can be built incrementally: enrollment first, then KeyBundle, then dominance, then health pipeline, then gossip, then convergence.
- Requires a multi-peer test harness (simulated network).
- The polygon schedule, convergence fronts, and gateway election are the most complex and should come last.

**Phase 3b: verkko-relay (depends on crypto + protocol, parallel with mesh)**
- Membrane storage layer can be built and tested independently.
- Transit forwarding depends on having the mesh topology, so that piece must wait for mesh.
- Territorial re-encryption needs the hash ring, which is self-contained.

**Phase 3c: verkko-seat (depends on crypto + protocol, parallel with mesh)**
- Cannot be built until the protocol is designed. Currently blocked.

**Phase 4: verkko-matter (depends on crypto + mesh + relay)**
- Cuckoo filters and Merkle reconciliation can be built first (testable with mock mesh state).
- Transfer protocol needs mesh + relay infrastructure.
- Sealed-range catch-up integrates with mesh convergence.

**Minimum viable subset:** verkko-crypto + verkko-protocol (sans-IO) gives you encrypted authenticated point-to-point communication. Add verkko-mesh enrollment + KeyBundle + dominance and you have a functional multi-peer mesh. Add cuckoo filters from verkko-matter and you have content routing. This is enough for a proof-of-concept.

## 3. Top 10 integration test scenarios

### Test 1: Genesis to first heartbeat
**Documents:** verkko-crypto, verkko-protocol, verkko-mesh
**What is tested:** A single device executes the genesis bootstrap sequence (verkko-crypto 1.3), establishes initial state, and sends its first heartbeat (verkko-protocol 3.4) with correct epoch (0), filter root (empty), and metabolic state (Background).
**Is behavior fully specified?** Yes. The genesis sequence is deterministic. The heartbeat format is byte-level specified.

### Test 2: Two-device enrollment with SAS verification
**Documents:** verkko-crypto, verkko-protocol, verkko-mesh
**What is tested:** Device A (genesis founder) creates an invite. Device B generates DK+DEK, joins via Noise XX with PSK, receives enrollment cert, completes SAS verification, receives KeyBundle(Grant) with channel keys.
**Is behavior fully specified?** Partially. The enrollment cert construction and KeyBundle format are specified. The invite token format and QR code content are not. The SAS verification interaction is referenced to RFC 6189 but the UX flow is not specified.

### Test 3: Epoch ratchet synchronization across peers
**Documents:** verkko-crypto, verkko-protocol
**What is tested:** Two peers connected for > 1 hour. Verify macro-epoch advances independently on both (hourly floor). Verify micro-epoch advances every 100 frames. Verify a peer that missed frames can catch up via micro-epoch HKDF iteration (bounded by 720). Verify key_id matching works.
**Is behavior fully specified?** Yes. The ratchet formulas, key_id computation, and catch-up iteration are all specified.

### Test 4: Nonce safety under crash
**Documents:** verkko-crypto, verkko-protocol
**What is tested:** Simulate crash during AEAD burst. Verify double-write persist recovers valid HWM. Verify gap sizing prevents nonce reuse. Verify velocity monitoring triggers adaptive persist before gap exhaustion.
**Is behavior fully specified?** Yes. The persist format, recovery protocol, gap formula, and velocity thresholds are all specified.

### Test 5: Partition heal and convergence
**Documents:** verkko-mesh, verkko-crypto, verkko-protocol, verkko-relay
**What is tested:** 6 clusters partitioned into two groups of 3 for 2 hours. Partition heals. Verify: wavefront gating waits for reconnections, gossip switches to flood, Front 1 merges control logs and resets EMAs, Front 2 resolves fencing tokens and recomputes dominance, polygon schedule renegotiates CPSKs with extended rounds for rate-limited channels, Front 3 re-encrypts territorial blobs, exit sequence restores Plumtree.
**Is behavior fully specified?** Mostly. The convergence fronts, gates, and exit sequence are detailed. The wavefront deadline formula is explicit. The polygon schedule construction is explicit. Ambiguity: what constitutes "all expected pairs" for wavefront completion? How does a peer know which pairs are expected?

### Test 6: Revocation and re-encryption
**Documents:** verkko-mesh, verkko-crypto, verkko-relay
**What is tested:** Admin revokes a peer. KeyBundle(Revoke) distributed with fresh channel keys. Territorial re-encryption of affected blobs with deterministic nonces. Verify: idempotent relay PUT (two peers re-encrypting the same blob produce identical ciphertext). Verify: P1 tiered deadline met.
**Is behavior fully specified?** Yes. The revocation flow, deterministic nonce construction, and idempotent PUT contract are all specified.

### Test 7: Health pipeline end-to-end
**Documents:** verkko-mesh, verkko-protocol
**What is tested:** Inject 10% frame loss on one connection. Verify: SackRatio computes correctly, dual EMA converges to Degraded classification within ~10 observations (fast EMA), scar timing fires with Intermittent/Degrading = DelayedShort (5 observations), circuit breaker transitions to HalfOpen. Remove loss. Verify: two-gate probe condition (Recovering AND low variance) before rate increase. Full recovery to Closed.
**Is behavior fully specified?** Yes. The EMA formulas, classification thresholds, scar timing rules, and circuit breaker gates are all specified.

### Test 8: Broadcast encryption with epoch advance
**Documents:** verkko-crypto, verkko-mesh
**What is tested:** A message broadcast to 11 clusters. Verify: BroadcastEnvelope constructed correctly (one content encryption, 11 key wraps). Trigger macro-epoch advance. Verify: micro-epoch counter resets. Second broadcast under new epoch key. Verify: recipients derive new key via HKDF and decrypt.
**Is behavior fully specified?** Yes. The envelope format and ratchet mechanics are specified.

### Test 9: Sealed-range catch-up after extended absence
**Documents:** verkko-matter, verkko-mesh, verkko-crypto
**What is tested:** Peer offline for 12 hours (no revocation). Reconnects. Verify: epoch fast-forward via HKDF, sealed range created with SealedReason::Absence, current-epoch traffic decryptable immediately. Admin issues UnsealGrant. Verify: sealed range cleared. Then: peer offline 12 hours WITH revocation. Verify: SealedReason::Revocation, CatchUpGrant with fresh key implicitly unseals.
**Is behavior fully specified?** Yes. The three-state model, SealedReason enum, and CatchUpGrant behavior are specified.

### Test 10: Constant-rate padding and relay correlation
**Documents:** verkko-relay, verkko-crypto
**What is tested:** Relay peer processes a mix of real PUTs and GETs. Verify: batch size is constant within epoch, decoys fill remaining slots, observer cannot distinguish real from decoy operations. Rotate epoch. Verify: batch size changes. Verify: relay_key_id rotation on 90-day boundary.
**Is behavior fully specified?** Partially. The mechanism is described but the decoy generation algorithm, batch timing interval, and batch_size derivation from epoch key are not specified at a level sufficient for two independent implementations to produce identical behavior.

## 4. Ambiguity inventory

### Wire format and framing

1. **SACK ACK framing.** The congestion feedback format (3 bytes) is specified inside SACK ACKs, but the SACK ACK itself is not framed. Is it a separate frame type (msg_type value)? Piggybacked on data frames? How are selective acknowledgements encoded?

2. **msg_type registry.** The encrypted payload has `[u8 msg_type]` with four listed values (heartbeat, gossip, control, chunk). Are these u8 values 0-3? Is there a formal registry? What about KeyBundle, enrollment, CPSK handshake messages?

3. **Stream ID absent from wire format.** The Stream defined concept exists but the wire format has no stream identifier field. How are streams demultiplexed?

4. **connection_id generation.** The outer header has `[u32 connection_id]` used for O(1) session lookup and path validation. How is it generated? Random? Derived from session keys? What happens on collision?

### Handshake and session

5. **Noise handshake framing.** Noise KK and XX handshakes are used but the framing of Noise messages within verkko frames is not specified. Do Noise handshake messages use the same outer header? Are they encrypted with a different key? How does a peer distinguish a handshake frame from a data frame before the session is established?

6. **CPSK ratchet vs re-handshake.** "Symmetric ratchet every 2 minutes, full re-handshake every 30 minutes." The symmetric ratchet is not specified. Is it an HKDF advancement of the CPSK? What is the ratchet's domain separator?

7. **Noise KK to XX fallback trigger.** "On KK failure, automatic fallback to XX with notification." What constitutes KK failure? Handshake timeout? Decryption failure on the first response? How many retries before fallback?

### Key management

8. **CIK replication mechanism.** CIK is "replicated to all cluster peers, encrypted to each DEK." Via what mechanism? A special intra-cluster KeyBundle? A direct DH exchange? This is stated as fact in verkko-crypto but the mechanism is neither there nor in verkko-mesh.

9. **KeyBundle delivery reliability.** KeyBundle is "delivered via reliable control channel." What is the reliable control channel? Is it the VitalQueue tier 1? What if a peer is offline when a KeyBundle is issued? How does it catch up? The "control log replay" path is mentioned but the interaction between KeyBundle delivery and control log storage is ambiguous.

10. **Channel key initial distribution.** When a mesh has 50 channels and a new peer joins, the KeyBundle(Grant) carries all 50 channel keys (34 bytes each = 1,700 bytes). This fits in a frame. But how are the channel keys organized? Is there a channel registry? How does a new peer know what channels exist?

### Health and observation

11. **`expected_observer_count` for partition quorum.** The observer quorum check in PartitionAnnouncement requires `expected_observer_count`. How is this determined? Is it the number of peers in the observer's cluster? The total mesh peer count? The document says `actual_observer_count < expected_observer_count / 2` but does not define where expected_observer_count comes from.

12. **Health EMA initialization.** When a peer first connects to another, what is the initial health EMA value? 1.0 (fully healthy)? The document says "health init" in the post-entry checklist but does not specify the initial value.

13. **Scar auto-clear.** Calibration item 8 mentions "scar auto-clear obs count: 100 over 30 days" but the body text says scars are "append-only (GSet). Never removed." These are contradictory. The scar entry is permanent but the weight decays; does "auto-clear" mean the weight decays below a threshold and the scar stops being counted? Or is there an actual removal mechanism?

### Gossip

14. **Plumtree spanning tree construction.** The exit transition says "spanning tree is connected" and is "deterministic from ring coloring." How does the ring coloring (which partitions relay_key_id space) determine the Plumtree spanning tree? These seem to be different graph structures. The tree is a communication topology; the ring is a data partition.

15. **Gossip message deduplication.** Flood mode uses "deduplication via message ID." What is the message ID? How is it generated? How long are message IDs retained for deduplication? What is the deduplication table size?

16. **recent_ihave_set for anomalous detection.** The anomalous-message detection checks against `recent_ihave_set`. The managed arena table shows "Recent IHAVE set (ring buffer, 100 entries)" but the retention policy and eviction strategy are not specified. Is 100 entries enough for the flood/Plumtree boundary to work correctly?

### Convergence

17. **"All expected pairs" in wavefront.** The wavefront gate waits until "all expected pairs reconnect." How does a peer determine the set of expected pairs? From the pre-partition GSet state? What if a cluster was destroyed during the partition?

18. **Front timeout interactions.** Front 1 has a 5-minute timeout. Front 2 has a 10-minute timeout. What happens if Front 1 times out and channels are orphaned? Does Front 2 proceed with incomplete key state? The document says "Apply EMA resets even if key retrieval incomplete" for Front 1, and Front 2 says "Highest fencing token wins after 10 minutes." But the interaction between partial Front 1 completion and Front 2 correctness is not analyzed.

19. **Convergence re-entry.** What happens if a new partition occurs during convergence? Does the current convergence abort? Does a new convergence start after the current one finishes? The document does not address nested failures.

### Relay

20. **Relay peer identity.** Is a relay peer a mesh peer with a special role? Or is it a distinct entity? The relay document says it is "not a full mesh participant in the content sense" but it participates in territorial re-encryption (which requires being on the hash ring, which requires a DK). Clarify: does a relay peer enroll in the mesh?

21. **SIGNAL operation.** Listed as one of 5 membrane ops. Not defined anywhere. What does it do?

22. **Relay storage backend.** The membrane is a "content-addressed blob store" but the storage backend is not specified. Filesystem? SQLite? Custom? What about atomicity of PUT?

### Territorial re-encryption

23. **"Same-cluster adjacent vnodes" coloring constraint.** The ring coloring prevents adjacent same-cluster vnodes. But with 12 clusters and 32 vnodes/peer, the ring has up to 12 * max_peers_per_cluster * 32 vnodes. The expected collisions analysis assumes 96 vnodes per cluster (3 peers * 32 vnodes). But the cluster size varies (1-6 devices). Does the coloring algorithm handle variable-size clusters correctly? The termination proof is for the general case, but what about degenerate cases (one cluster with 6 peers = 192 vnodes, others with 1)?

24. **Retain-until-handoff mechanism.** "A peer retains ownership of its territorial partition until it explicitly hands off to the new owner." What is the handoff protocol? How does a peer know who the new owner is? What if the new owner is unreachable?

### Seat

25. **Seat authentication credential type.** "User credential (not a DK/DEK)" is the entire specification. What credential? This blocks implementation entirely.

26. **Seat event subscription granularity.** "Seat subscribes to event categories during session setup." What are the event categories? How is the subscription expressed?

### Matter

27. **Large matter and relay chunking.** Matter can be 2GB. The relay membrane stores "opaque encrypted blobs." How is a 2GB matter stored on a relay? As one blob? Chunked? If chunked, what is the chunk size? How is chunk reassembly coordinated?

28. **Replication target configuration.** "Configurable min_replicas and target_replicas per matter priority." Configurable by whom? Per-channel? Per-mesh? What are the defaults?

29. **Merge rules for SyncReconciliation.** "Merge rules: field-level, deterministic, reproducible on all peers." What fields? What rules? This is completely abstract.

30. **Download coordination.** The taxonomy lists "download engine" as a deferred topic. verkko-matter defines MatterTransfer but has no specification for how downloads are initiated, prioritized, scheduled, or coordinated across peers.

### Cross-document

31. **Heartbeat payload evolution.** The heartbeat is 36 bytes with a rigid structure. How are new fields added? Is the format versioned? What happens when peers run different versions?

32. **Error handling and error codes.** No document specifies error codes, error frames, or error handling behavior. What does a peer send when it receives an invalid frame? A KeyBundle with a bad signature? An enrollment cert that fails validation?

33. **Protobuf schema for control messages.** SchemaSync (verkko-protocol) and SeatRequest (verkko-seat) reference protobuf, but the protobuf message definitions are not provided in any document.

34. **Versioning and protocol negotiation.** BroadcastEnvelope has a `version: u8` field. The persist format is "versioned (1 byte header)." But there is no protocol version negotiation specified. How do peers agree on the protocol version? What happens when a v1 peer connects to a v2 peer?

## 5. Defined concepts as API assessment

### verkko-crypto -> verkko-protocol

**Status: Sufficient with minor gaps.**

verkko-protocol depends on FrameKey, TransportAEAD, NonceSafety, EpochRatchet, and KeyZeroization. All of these are well-defined in verkko-crypto with enough information for verkko-protocol to use them:
- TransportAEAD provides the algorithm (ChaCha20-Poly1305), key size (256-bit), nonce size (96-bit), and tag size (128-bit). verkko-protocol uses this to construct frames. Sufficient.
- NonceSafety provides the three-layer guarantee. verkko-protocol's event loop thread owns NonceSource [0, 2^95). Sufficient.
- EpochRatchet provides the HKDF advancement. verkko-protocol carries epoch and micro-epoch in the encrypted payload. Sufficient.

**Gap:** verkko-protocol's wire format includes `[u64 relay_key_id]` which is computed from `HMAC-BLAKE3(key=channel_salt, data=key_id)`. The channel_salt derivation uses `HKDF-Expand(prk=channel_key, info="relay:salt:" || floor(unix_days/90), len=32)`. This computation is specified inline in verkko-protocol, not in verkko-crypto's defined concepts. It relies on internals (HKDF-Expand, BLAKE3 keyed MAC) rather than going through a defined concept. This is a minor encapsulation violation; it works, but it means verkko-protocol needs to know about HKDF-Expand and BLAKE3 keyed MAC details that are not surfaced in the defined concepts.

### verkko-crypto -> verkko-mesh

**Status: Sufficient.**

verkko-mesh depends on DK, DEK, ChannelKey, EpochRatchet, BroadcastEncryption, PCSMechanism, and DomainSeparatorRegistry. These are all well-defined. The PCSMechanism provides the re-seed formula and verkko-mesh provides the trigger and distribution. Clean split.

### verkko-crypto -> verkko-relay

**Status: Sufficient.**

verkko-relay depends on StorageAEAD, ChannelKey, EpochRatchet, DomainSeparatorRegistry, and NonceSafety. StorageAEAD (XChaCha20-Poly1305 with deterministic nonces) is well-defined. The re-encryption nonce formula is explicit. The idempotent PUT follows from deterministic nonces. Clean dependency.

### verkko-crypto -> verkko-seat

**Status: Sufficient for the defined dependency.**

verkko-seat depends on TransportAEAD and KeyZeroization. These are well-defined. However, the seat session is "asymmetric" (not a peer-to-peer session), so it may need a different Noise pattern or credential type that is not in verkko-crypto's defined concepts. This is a gap in verkko-seat, not in verkko-crypto.

### verkko-crypto -> verkko-matter

**Status: Sufficient.**

verkko-matter depends on StorageAEAD and DomainSeparatorRegistry (for BLAKE3 content addressing). The blob_id = BLAKE3(plaintext) construction is straightforward. The domain separator for content integrity could be registered but is not in verkko-crypto's registry table (it uses raw BLAKE3 without a domain separator key, which is actually correct for content hashing).

### verkko-protocol -> verkko-mesh

**Status: Sufficient.**

verkko-mesh depends on Connection, Heartbeat, Session, Frame, VitalQueue, and MetabolicState. These are all defined with invariants. The heartbeat payload format is byte-level specified. The VitalQueue's three tiers and 70/30 weighted fair queuing are explicit. MetabolicState's four states and dwell time are explicit.

### verkko-protocol -> verkko-relay

**Status: Sufficient but relay needs more.**

verkko-relay depends on Frame, Connection, and NATTraversal. Frame and Connection are defined. NATTraversal is defined conceptually but not specified at the wire level. For transit relay forwarding, verkko-relay needs to know how to read the outer header (destination, connection_id, hop_count) without decrypting the payload. This information IS in the wire format specification, so it works as a defined concept dependency. However, the relay forwarding decision logic (not specified) would need to consume frame internals that are not in the defined concepts.

### verkko-mesh -> verkko-matter

**Status: Sufficient.**

verkko-matter depends on Cluster, Gateway, Gossip, HealthPipeline, and Convergence from verkko-mesh. These are well-defined. Source selection uses health scores. Content routing uses gossip. Sealed-range catch-up interacts with convergence. The dependencies are through defined concept invariants.

### verkko-relay -> verkko-matter

**Status: Sufficient.**

verkko-matter depends on Membrane, TerritorialReencryption, and IdempotentPUT from verkko-relay. These are well-defined with explicit invariants.

**Overall assessment:** The defined-concepts-as-API pattern works well. The documents mostly reference each other through defined concepts with stated invariants. The main encapsulation violation is the relay_key_id computation in verkko-protocol which uses verkko-crypto internals directly rather than through a defined concept.

## 6. Gap analysis

### Blocking (cannot build without this)

**B1. BLAKE3 test vector table (verkko-crypto).** Empty. The document explicitly states there is no runtime detection for BLAKE3 divergence. Without test vectors, two independent implementations could silently partition. Severity: blocking. Fix: compute and publish concrete test vectors for all 7 registered domain separators.

**B2. Seat protocol design (verkko-seat).** No wire format, no authentication protocol, no request/response framing. Cannot build any seat functionality. Severity: blocking for seat. Not blocking for mesh-only deployment.

**B3. Transit relay forwarding logic (verkko-relay).** No specification for how a relay peer decides to forward a frame. Cannot build relay-assisted connectivity. Severity: blocking for WAN meshes behind NAT. Not blocking for LAN-only.

**B4. NAT traversal procedure (verkko-protocol).** Hole-punching mentioned but not specified. Cannot build WAN connectivity without NAT traversal or relay. Severity: blocking for WAN meshes.

**B5. Invite flow (verkko-mesh).** No specification for invite token format, QR code content, or bootstrap handshake. Cannot onboard new clusters. Severity: blocking for multi-cluster mesh formation.

### Annoying (could guess but should not have to)

**A1. SACK ACK framing (verkko-protocol).** The congestion feedback format is specified but the SACK itself is not framed. An implementor would need to invent the SACK format. Could follow IPsec/QUIC conventions, but the spec should be explicit.

**A2. msg_type value assignments (verkko-protocol).** Four message types listed but no numeric values assigned. An implementor would assign 0-3 but might do it differently than another implementor.

**A3. Noise handshake framing (verkko-protocol).** How Noise messages are framed within verkko is not specified. Could use the same outer header with a special msg_type, but two engineers might do it differently.

**A4. connection_id generation (verkko-protocol).** Not specified. Random u32? Derived from handshake? Two implementations would likely differ.

**A5. Health EMA initial value (verkko-mesh).** Not specified. Probably 1.0, but could be 0.5 (unknown until observed). Affects initial scar and circuit breaker behavior.

**A6. expected_observer_count source (verkko-mesh).** Used in observer quorum for PartitionAnnouncement. Not defined where this number comes from.

**A7. Relay storage backend (verkko-relay).** Not specified. Filesystem with content-addressed paths? SQLite? An implementor could choose either, and the performance characteristics differ significantly.

**A8. Gossip message ID generation (verkko-mesh).** Used for flood deduplication. Not specified. Could be a hash of the message, a random ID, or a (sender, sequence) pair.

**A9. Channel registry and discovery (verkko-mesh).** A new peer receives KeyBundle(Grant) with channel keys, but how does it learn what channels exist, what they are for, and how to use them?

**A10. CPSK symmetric ratchet domain separator (verkko-protocol).** "Symmetric ratchet every 2 minutes" is stated but the ratchet construction is not specified. Missing HKDF info string or domain separator.

### Nice-to-have (would make implementation easier)

**N1. Reference implementation of the polygon schedule.** The 1-factorization construction is described mathematically. Pseudocode would reduce implementation risk.

**N2. Managed arena allocation strategy.** The 200KB steady state is tabulated by component but the allocation strategy (bump? slab? region?) within the managed arena is not specified.

**N3. Gateway handoff message format.** Gateway-claim and gateway-release are described conceptually (~200 bytes) but no wire format is given.

**N4. Convergence velocity tracking format.** "Every CRDT component tracks its derivative. 7-day rolling day-of-week baselines (1.4KB)." No specification of the derivative tracking or baseline format.

**N5. Gossip config per message type.** The GossipConfig struct is shown (ack_mode, retransmit, priority) but the mapping of which messages use which config is not specified.

**N6. Cuckoo filter insertion failure handling.** 90% load factor is specified, but what happens when the filter is full? How does insertion failure surface?

**N7. Clock synchronization error bounds.** Cristian's algorithm provides the initial sync, PLL correction in heartbeat provides ongoing adjustment. The axiom says "bounded clock skew (1 second)" but the PLL convergence time to achieve this bound is not specified.

## 7. Verdict

**Not ready for full implementation. Ready for Phase 1-2 implementation with targeted fixes.**

The verkko design is remarkably thorough for a research-phase document set. The cryptographic layer (verkko-crypto) is nearly implementation-ready. The mesh coordination layer (verkko-mesh) is the most sophisticated distributed systems design I have seen in a household-scale specification, and it is implementable for its core mechanisms.

**What needs to happen before implementation begins:**

1. **Fill the BLAKE3 test vector table** (verkko-crypto). This is a 1-hour task that blocks everything.

2. **Specify the following wire-level details** (verkko-protocol):
   - msg_type value assignments
   - SACK ACK frame format
   - Noise handshake message framing
   - connection_id generation
   - Stream multiplexing wire format (if streams are in scope for v1)

3. **Specify transit relay forwarding logic** (verkko-relay). The decision logic, routing table maintenance, and relay path establishment need a body section.

4. **Specify the invite flow** (verkko-mesh). Token format, QR code content, bootstrap handshake, and validation.

5. **Design verkko-seat** or explicitly defer it to post-MVP.

**What can proceed now:**

- verkko-crypto implementation (after test vectors are added)
- verkko-protocol sans-IO implementation (frame serialization, LEDBAT, heartbeat, metabolic state machine, replay window)
- verkko-mesh core types (KeyBundle, enrollment cert, dominance score, PhysicalBoundary trait, health pipeline)
- verkko-relay membrane storage layer (PUT/GET/DELETE/TOUCH, hash ring, ring coloring, deterministic re-encryption nonces)
- verkko-matter cuckoo filters and Merkle reconciliation

These components can be built, unit-tested, and integration-tested before the blocking gaps are resolved. The blocking gaps affect system-level integration (connecting peers across networks, onboarding new clusters, thin client access) but not the core algorithmic components.

**Assessment of the taxonomy as a decomposition strategy:** The six-document split is correct. The dependency graph is acyclic. The defined-concepts-as-API pattern works well and should be preserved. The main risk is not the decomposition but the unspecified wire-level details that two independent implementors would resolve differently.
