## §1 Plugin Architecture, Store, and the SDK Boundary

### §1.1 Plugin registration: `inventory` + `repr(C)` descriptors

Plugins register via `inventory::submit!` of `repr(C)` descriptors, never via `dyn Trait`. The `#[register]` macro on an impl block generates a `repr(C)` descriptor with stable `extern "C"` function pointers and submits it to `inventory`. A sealed `HasDescriptor<D>` trait gates registration at compile time, generated by a separate registry-creating macro applied to the registrable trait definition.

| Name | Kind | Purpose |
|------|------|---------|
| `Named` | trait | Machine-readable identifier via `const NAME: &'static str` |
| `NAME` | associated const (on `Named`) | The single source of truth string identifier |
| `Registrable` | trait | Base supertrait: `Named + Display + Send + Sync + 'static` |
| `HasDescriptor<D>` | trait | Sealed compile-time gate: "this type can produce descriptor `D`" |
| `SdkVersion` | struct (data) | Typed ABI version with `major: u16, minor: u16` |
| `CURRENT` | associated const (on `SdkVersion`) | Current SDK version constant |
| `Descriptor` | trait | Common contract for all `repr(C)` descriptors |
| `Registry` | trait | Common contract for typed registries; provides `iter`, `get`, `validate`, `validate_domain` |
| `Descriptor` (assoc type) | associated type (on `Registry`) | The descriptor type this registry holds |
| `ConnectorRegistry` | (domain registry impl) | Filter by capability, capability matrix queries |
| `WorkUnitRegistry` | (domain registry impl) | DAG construction, cycle detection, topological ordering |
| `SubsystemRegistry` | (domain registry impl) | Initialization ordering, cadence timer setup |
| `#[register]` | macro | Universal registration attribute for any trait impl block |

### §1.2 Persistence: Descriptor-based schema, SeaORM host-only

SeaORM is used only on the host side; the entire persistence API is expressed through trait contracts and `repr(C)` descriptors. Plugins never see SeaORM types.

#### §1.2.1 Schema as traits: MetadataTable, Column, References

Tables, columns, and relationships are trait contracts in the SDK; all implementors are ZSTs. The `#[register]` macro generates `repr(C)` descriptors from them.

| Name | Kind | Purpose |
|------|------|---------|
| `MetadataTable` | trait | Pure marker: "I am a metadata table." No methods. |
| `In<M>` | struct (ZST) | Semantic wrapper: "column lives in table `M`" |
| `As<V>` | struct (ZST) | Semantic wrapper: "column is of value type `V`" |
| `sealed::TableRef` | trait | Sealed — only `In<M>` satisfies |
| `sealed::TypeRef` | trait | Sealed — only `As<V>` satisfies |
| `Column<I, A>` | trait | Typed column association (`I: TableRef`, `A: TypeRef`) |
| `References<T>` | trait | FK relationship: table references `T` via associated `FkColumn` |
| `FkColumn` | associated type (on `References<T>`) | The FK column type (`Column<In<Self>, As<i64>>`) |
| `MetadataTableMeta` | (implied interface) | Provides `TABLE_NAME: &'static str` associated const |
| `ColumnMeta` | (implied interface) | Provides `COLUMN_NAME: &'static str` associated const |
| `TABLE_NAME` | associated const | Compile-time table name string |
| `COLUMN_NAME` | associated const | Compile-time column name string |

#### §1.2.2 Implicit plumbing columns

Every metadata table automatically receives host-managed columns (`id`, `entity_id`, `source`, `created_at`, `updated_at`). The `source` column is auto-stamped from `ConnectorContext<C>` — provenance is guaranteed by the type system. No traits or types are defined here; this is a host convention.

#### §1.2.3 Query API: typed, descriptor-backed

Plugins query through `ctx.query::<M>()` using ZST types as compile-time constraints. No new types beyond the query builder are defined here; the key API is `QueryBuilder<M>` and the `SupportsEntity`/`HasMetadata` pattern from §4.5.

#### §1.2.4 Write API: fire-and-forget work units

`ctx.write()` drops a work unit into the scheduler; "Ok" means "accepted," not "written." Write ordering is automatic from the FK graph derived from `References` descriptors. No new types defined beyond `WriteBuilder<M>`.

#### §1.2.5 Table ownership and cross-plugin sharing

Three levels: first-party SDK tables, plugin-private tables, cross-plugin schema crates. This section establishes convention, not new types.

#### §1.2.6 Migrations: type-level ops, auto-additive, safe destructive

Migrations are a registered trait `Migration<const V: u32>`. Additive changes (new `Column` impls) are auto-applied. Destructive/transformative changes require explicit type-level migration ops. All ops are ZST generic structs forming a closed set.

| Name | Kind | Purpose |
|------|------|---------|
| `Migration<const V: u32>` | trait | Per-table migration at version `V`; associated `type Ops: MigrationOpSet` |
| `Ops` | associated type (on `Migration`) | Tuple-based type-level set of migration ops |
| `MigrationOpSet` | trait | Type-level set constraint for migration op tuples |
| `CreateTable<T>` | struct (ZST) | Migration op: create table for `T: MetadataTable` |
| `DropTable<T>` | struct (ZST) | Migration op: drop table (captures all rows before execution) |
| `CreateColumn<C, T>` | struct (ZST) | Migration op: add column `C` to table `T` |
| `DropColumn<C, T>` | struct (ZST) | Migration op: drop column (captures values) |
| `RenameColumn<C, T, NewName>` | struct (ZST) | Migration op: rename column |
| `RenameTable<T, NewName>` | struct (ZST) | Migration op: rename table |
| `ModifyType<C, T, FromType, ToType>` | struct (ZST) | Migration op: change column value type |
| `CreateIndex<Cols, T>` | struct (ZST) | Migration op: create index |
| `DropIndex<Name, T>` | struct (ZST) | Migration op: drop index |
| `CopyRows<Cols, From, To>` | struct (ZST) | Data migration op: copy rows between tables |
| `DeleteRows<T, Filter>` | struct (ZST) | Data migration op: delete rows matching filter (captures) |
| `From<S>` | struct (ZST) | Semantic wrapper for `CopyRows` source table parameter |
| `To<D>` | struct (ZST) | Semantic wrapper for `CopyRows` destination table parameter |
| `MigrationDescriptor` | struct (repr(C)) | Descriptor carrying version, table name, and `&[OpDescriptor]` |
| `OpDescriptor` | enum/struct (repr(C)) | Runtime representation of a migration op |

## §2 Priority Levels

### §2.1 Decision: 5 levels, `repr(C)`, explicit discriminants

The `Priority` enum crosses the plugin ABI boundary. Five levels are the final count because `High` and `Low` fill gaps that cannot be added later without breaking the ABI. Lower discriminant = higher priority.

| Name | Kind | Purpose |
|------|------|---------|
| `Priority` | enum (repr(C)) | Work priority with explicit discriminants: `Immediate=0`, `High=1`, `Normal=2`, `Low=3`, `Background=4` |

## §3 Notification Architecture

### §3.1 Decision: `NotificationPayload` registered trait

The payload IS the notification — each notification kind is a struct implementing `NotificationPayload` with urgency as an associated const. Extensions can define new notification kinds by implementing and registering the trait.

| Name | Kind | Purpose |
|------|------|---------|
| `NotificationPayload` | trait | Registered trait: `Named + Registrable + Send + Sync + 'static`; `const URGENCY: Urgency` |
| `URGENCY` | associated const (on `NotificationPayload`) | The intrinsic urgency of this notification kind |
| `Urgency` | enum (repr(C)) | `Critical=0`, `High=1`, `Normal=2`, `Low=3` |
| `ApiKeyExpired` | struct (ZST) | Example notification: API key has expired (Critical urgency) |
| `DownloadFailed` | struct (data) | Example notification: download failed; carries `entity: EntityId, reason: ErrorClass` |
| `MetadataStale` | struct (data) | Example notification: metadata is stale; carries `entities: Box<[EntityId]>` |

### §3.2 Notification delivery

Delivery flow is: logging macro → `ctx.notify(payload)` → host enriches with execution context → async-broadcast channel → SSE handlers → persistence. No new types defined.

### §3.3 Supplementary: Notification work units

Work units for aggregated/debounced/digest notifications. Named work units (not fully specified as types in this section): `BatchSummaryWork`, `NotificationDeduplicator`, `DigestWork`, `QueueEmptyCheck`. These are work unit impls, not new type definitions.

## §4 Content Model

### §4.1 Decision: Flat entity model with metadata

Everything is an entity. Two anchor DB tables: `entity_types` and `entities`. Users, games, movies, downloads, sessions all use this. No new SDK types defined here — this is the DB schema decision.

### §4.2 Systems are metadata, not structure

Platform/system is a metadata label on an entity, not a structural concept. Convention, not a type.

### §4.3 Hierarchy is metadata relationships

The game hierarchy (franchise → work → edition → variant) is metadata relationships between entities. Convention, not types.

### §4.5 Metadata association — dual-trait pattern with `min_specialization`

Solving the orphan rule for cross-crate metadata association. Two traits, each owned by the declaring side. A blanket bridge via `min_specialization` unifies them for the query/write API.

| Name | Kind | Purpose |
|------|------|---------|
| `SupportsEntity<E>` | trait | Table-side: "this metadata table accepts entity type `E`" |
| `HasMetadata<M>` | trait | Entity-side: "this entity type carries metadata table `M`" |

#### §4.5.1 Blanket bridge via `min_specialization`

The `default impl<E, M> SupportsEntity<E> for M where E: HasMetadata<M>` blanket. No new type names; this is the `min_specialization` pattern enabling either direction of declaration to satisfy the single `SupportsEntity<E>` query bound.

#### §4.5.2 `min_specialization` — nightly requirement and soundness caveat

Nightly requirement acknowledged. Mitigation: never specialize on lifetime bounds. No new types.

## §5 Scheduler & Subsystems

### §5.1 Decision: Event-driven DAG

The scheduler is pure infrastructure — events, ordering, batching, dispatch. Domain knowledge lives in subsystems which are peers. No new types; architecture diagram only.

## §6 Cross-Cutting Conventions

### §6.1 Architectural idiom: Blanket marker traits for compile-time association

Named convention: generic marker traits blanket-implemented for types satisfying a domain constraint. No new types — pattern documentation.

### §6.2 The plugin boundary paradigm: Trait impls only

Plugins define trait implementations, never extending closed enums. This is the single rule for all extensibility questions. Convention, no types.

### §6.3 `repr(C)` enum convention

Every `repr(C)` enum has explicit discriminants, represents a closed set, and uses lower-discriminant = higher priority where ordering applies. Convention, no new types.

### §19.4 What stays (valid concrete types)

Confirms legitimacy of: IR value types (`Color`, `Length`, `FontAsset`, `FontStyle`), credential impls (`ApiKey`, `OAuthToken`), settings derive targets, host-internal types (`DownloadEngine`), newtypes (`EntityId`, `ExternalId<C>`, `SdkVersion`, `Fragment<F>`), and ZSTs universally.

### §19.5 Principle: when is a concrete struct warranted?

Four valid cases: IR value type, trait implementation, newtype, host-internal. Four invalid cases: pipeline data shuttle, appears in trait signatures, aggregates results, "we need to carry X Y Z together." No new types.
