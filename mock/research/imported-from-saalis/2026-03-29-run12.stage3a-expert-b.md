## Disagreement Statement

I disagree with the Savant's maintained position that a two-phase
feed/poll model with revised arena-epoch lifetime is equivalent to
the four-method model for specification quality. The Savant concedes
arena-epoch lifetime (my argument) and claims the two-phase model
now provides zero-copy batching equal to mine. That concession is
real and appreciated. But the Savant's revised trait still collapses
SendBytes and HeartbeatReceived into one `ProtocolOutput` enum
drained through one `poll_output()` method. The cross-language
portability argument does not survive close examination of what
"cross-language" actually means for this protocol. I will show that
the Savant's cross-language argument, when taken seriously, demands
a C ABI specification that neither model provides, and that in the
absence of a C ABI specification, the Rust-native four-method model
is the correct design target.

## Section 1: The Two-Phase vs Four-Method Debate -- Language Lock-In

### 1.1 The Savant's Core Argument

The Savant writes (Stage 2b, Section 2, Domestication 1):

> "Verkko is a protocol. The specification must be implementable in
> C, in Go, in any language. The borrow checker argument is a Rust
> implementation detail."

And:

> "My structural reentrancy prevention works in any language: the
> protocol only produces outputs through poll_output; poll_output
> does not accept inputs; therefore the consumer cannot feed inputs
> during output processing."

The Savant's argument is that the two-phase model's reentrancy
prevention is a property of the API shape, independent of any
language's type system. This is true. It is also irrelevant to the
question we are actually deciding.

### 1.2 The Question We Are Actually Deciding

We are not writing a protocol specification for multi-language
implementation. We are writing a Rust library. The design documents
specify Rust as the implementation language. The crate table lists
Rust crates. The wire format is specified (W1-W18). Any future C or
Go implementation would implement the wire format from the wire
specification, not by translating the Rust trait into a C header.

The two-phase model's cross-language reentrancy guarantee is a
property that would matter if we were designing an abstract protocol
interface specification (like QUIC RFC 9000's abstract API
description). But QUIC RFC 9000 does not specify a polling model.
It specifies wire behavior. Implementations (quinn in Rust, quiche
in C, ngtcp2 in C) each design their own polling model optimized
for their language. Quinn uses a four-method model. Quiche uses a
two-phase model. Both implement the same protocol correctly.

The lesson from QUIC: the polling model is an implementation
concern, not a protocol specification concern. The wire format is
the protocol specification. The Rust trait is the Rust implementation
of that specification.

### 1.3 Why Rust-Specific Optimization Is Not Language Lock-In

Language lock-in occurs when a design decision prevents future
implementations in other languages. The four-method model does not
prevent C or Go implementations because:

1. The wire format (W1-W18) is language-independent. Any language
   can implement the wire format.
2. The protocol state machine (Noise KK/XX, SACK, LEDBAT, stream
   mux) is defined by its behavior, not by the Rust trait shape.
3. A C implementation would use a two-function model (feed/poll)
   because C lacks borrow checking. A Go implementation might use
   channels. A Python implementation might use callbacks. Each is
   valid.

The Savant is arguing that we should design the Rust trait to be
directly translatable to C. This is the wrong goal. The Rust trait
should be the best possible Rust API. The C implementation should
be the best possible C API. Both implement the same wire protocol.
The wire protocol is the contract. The API is the implementation.

### 1.4 The Causal Ordering Argument Examined

The Savant writes:

> "Outputs emerge in the order the protocol generated them, which
> is the causal order."

This is the strongest argument for the two-phase model and I want
to address it carefully. The Savant claims that a single drain loop
preserves causal ordering between transmit outputs and application
events: the SACK frame is enqueued before the HeartbeatReceived
event because the protocol processes the frame, generates the SACK,
then generates the event. The consumer sees them in causal order.

In my four-method model, the consumer drains all transmissions
first, then all events. This reorders the outputs: transmissions
are batched before events. The Savant argues this breaks causal
order.

I concede that the four-method model reorders outputs across the
transmit/event boundary. I do not concede that this matters. The
causal relationship between a SACK frame and a HeartbeatReceived
event is not semantically meaningful to the consumer. The consumer
sends the SACK frame to the network. The consumer dispatches the
HeartbeatReceived event to the mesh layer. These are independent
operations on independent subsystems (network socket vs mesh state
machine). The ordering between them is not observable to the
protocol's correctness.

The ordering that matters is:
- **Within transmissions:** priority ordering (vital > foreground >
  bulk). My poll_transmit() provides this.
- **Within events:** causal ordering (SessionEstablished before
  HeartbeatReceived for that session). My poll_event() provides
  this.
- **Between transmissions:** must be sent before the next feed
  cycle to avoid head-of-line blocking. My drain-then-reset
  convention provides this.

The ordering between a transmission and an event does not affect
correctness. The SACK frame will reach the network whether it is
sent before or after the mesh layer processes HeartbeatReceived.
The mesh layer's HeartbeatReceived processing does not depend on
whether the SACK was sent yet.

### 1.5 Revised Position on Two-Phase vs Four-Method

I maintain the four-method model. My arguments are unchanged from
Stage 2a:

1. Type-level output separation (compile-time guarantee that
   poll_transmit never returns an application event).
2. Arena-epoch batching (the Savant now concedes this).
3. Transmit-first drain ordering (reduced heartbeat tail latency).
4. Reentrancy prevention via borrow checker (the Savant does not
   dispute this for Rust, only for cross-language portability, which
   I have argued is not the design target).

The Savant's causal ordering argument is elegant but does not
identify a correctness issue. Cross-transmit/event ordering is not
observable to the protocol's correctness properties.

However, I make one new concession: if the project ever produces a
formal protocol specification document (like an RFC), that document
should describe the abstract interface as two-phase (feed/poll) for
language neutrality. The Rust implementation should use four methods.
The specification and the implementation serve different audiences.
The specification audience is "anyone implementing the protocol."
The implementation audience is "Rust programmers using the verkko
crate."

## Section 2: Arena Budget Survival Under Convergence Burst

### 2.1 The Savant's Convergence Burst Exhaustion Finding

The Savant identifies (Stage 2b, Structural Hole 2) that 500
WiFi-buffered frames overflow the 64KB hot arena, requiring 16
drain-reset cycles. The Savant also identifies the compounding
interaction between drain-reset cycles and nonce persist stalls.

The devil's advocate calculated the convergence gossip flood:
during convergence Front 1, all peers flood gossip. In a 12-cluster
mesh with 36 peers, each peer sends its control log entries to all
connected peers. If a 4-hour partition produced 200 control log
entries per cluster, the reconnecting gateway receives 200 * 11
= 2,200 gossip messages in the first few seconds. Each gossip
message is a UDP datagram. Combined with WiFi power-save buffering,
these 2,200 messages arrive in a burst.

### 2.2 Does the 256KB Managed Arena Survive?

The managed arena (256KB) is not affected by the gossip burst. The
managed arena holds connection tables, stream state, and SACK
tracking. These are long-lived allocations that do not grow with
incoming traffic volume. The managed arena's occupancy is bounded
by the number of active connections (16 connections * ~16KB each =
~256KB), not by the number of incoming frames.

The hot arena (64KB) is the one under pressure. Each incoming
datagram requires transient allocation for:
- Frame parsing (outer header: 29 bytes)
- Decryption (inner payload: up to 1170 bytes)
- Event construction (variable, ~100 bytes for gossip)
- SACK generation (~146 bytes max)

Total per-frame hot arena cost: ~1,500 bytes. At 64KB hot arena
capacity: 64,000 / 1,500 = ~42 frames per arena epoch before
exhaustion.

The 2,200-frame gossip burst requires 2,200 / 42 = ~52 drain-reset
cycles. Not 16. The Savant's estimate of 500 frames (WiFi power-save
only) underestimates the convergence case. The combined WiFi burst +
convergence gossip produces a larger burst.

### 2.3 Why 52 Drain-Reset Cycles Is Acceptable

Each drain-reset cycle:
1. Drains all pending transmissions (poll_transmit until None).
2. Drains all pending events (poll_event until None).
3. Sends batched transmissions via sendmmsg.
4. Dispatches events to mesh layer.
5. Resets the arena.

The cost per cycle is dominated by sendmmsg latency (~10us for a
batch of 42 frames on Linux). 52 cycles * 10us = ~520us total
sendmmsg time. On a Pi Zero (slower syscall overhead), multiply by
3: ~1.5ms.

The nonce persist interaction the Savant identifies is mitigated by
async nonce persistence. With async persist, the nonce persist cost
per cycle is zero (the persist is fire-and-forget to the crypto
thread). The crypto thread processes the persist asynchronously.
The only cost is the SPSC ring buffer push: ~50ns per cycle. 52
cycles * 50ns = ~2.6us. Negligible.

The 52 drain-reset cycles complete in ~1.5ms on a Pi Zero. This is
within the 2-second heartbeat interval. The system processes the
convergence gossip burst in under 0.1% of the heartbeat budget.

### 2.4 The Real Risk: Pool Arena Exhaustion

The managed and hot arenas survive the gossip burst. The pool arena
(256KB, used for retransmission buffers) is the actual risk.

During convergence, the system generates SACK frames and
retransmission requests for all 2,200 incoming gossip messages. If
gossip uses reliable-ordered streams (which it should, since control
log entries must arrive in order and not be lost), each gossip
message consumes retransmission buffer space: ~1,232 bytes per frame
* 2,200 frames = ~2.7MB of retransmission state.

This exceeds the 256KB pool arena by 10x.

The mitigation: gossip does not use reliable-ordered streams. Gossip
uses the Plumtree protocol, which has its own reliability mechanism
(lazy push for recovery). Gossip messages are sent as unreliable
datagrams at the protocol level. The mesh layer handles gossip
reliability via the IHAVE/GRAFT mechanism, not via stream-level
retransmission.

If gossip is unreliable at the protocol level, no retransmission
state is allocated. The pool arena is unaffected. The hot arena
handles the transient parsing and SACK generation. The 256KB pool
arena survives.

This is a constraint on the mesh layer: gossip MUST use unreliable
datagrams, not reliable-ordered streams. Control log entries
themselves are idempotent (GSet append-only, Ed25519-signed), so
the loss of a single gossip message is recoverable via IHAVE/GRAFT.
The mesh layer's reliability is independent of the protocol layer's
reliability.

### 2.5 Revised Arena Budget

No change to the budget from my Stage 2a proposal:

```
Hot arena:    64KB   (per-frame transients, reset per iteration)
Pool arena:  256KB   (retransmission buffers, assembly, verification)
Managed:     256KB   (connection tables, stream state, SACK tracking)
──────────────────────────────────
Total:       576KB
```

The critical constraint: gossip uses unreliable datagrams. This must
be stated as an arena-driven design requirement, not just a mesh-layer
preference. If gossip ever moves to reliable-ordered streams, the pool
arena budget must increase to at least 3MB, which violates the design
target of keeping total protocol memory under 1MB.

## Section 3: The Diagnostic Path Triangle and the Sans-IO Interface

### 3.1 Does the Triangle Add State to the Protocol Trait?

No. The diagnostic path triangle is a mesh-layer construct. The
Protocol trait does not know about direct paths, relay paths, or
WiFi quality. The Protocol trait handles connections identified by
connection_id. The Protocol trait delivers SACK events, heartbeat
events, and loss statistics. The mesh layer classifies connections
into direct/relay/intra-cluster categories and builds the
PathTriangle from Protocol-layer events.

The Savant's PathTriangle struct lives in verkko-mesh, not in
verkko-protocol. The Protocol trait does not change.

### 3.2 How WiFi Quality Enters the Sans-IO Interface

The Savant writes (Stage 2b, Section 2, Structural Hole 1):

> "When wifi_quality drops below 0.85, the Conductor should NOT push
> toward Active mode."

And (Stage 2b, Section 5, Remaining Amorphous Regions, item 4):

> "But the PhysicalBoundary gate (how the OS-level WiFi statistics
> enter the fixed-point abstract state machine) needs specification.
> This is a new PhysicalBoundary gate type alongside the existing
> six newtypes."

This is a genuine interface question within my domain. The WiFi
quality measurement is an OS-level statistic (link quality, signal
strength, delivery ratio). It must enter the protocol state machine
as a ProtocolInput. Here is my proposal:

```rust
enum ProtocolInput<'a> {
    // Existing variants...
    BytesReceived { source: PeerAddr, data: &'a [u8] },
    ConnectionRequested { destination: PeerAddr },
    ConfigChanged { config: &'a ProtocolConfig },
    CryptoResult { request_id: u32, result: CryptoOutcome<'a> },

    // New variant for external resource measurements
    ResourceMeasurement {
        resource: ResourceType,
        value: CapacityRatio,  // Q16.16 fixed-point, 0x0000..0x10000
    },
}

#[repr(u8)]
enum ResourceType {
    CpuPressure = 0,
    FlashIoPressure = 1,
    NonceIoPressure = 2,
    TransportCapacity = 3,
    WifiQuality = 4,       // new
}
```

The consumer (event loop) periodically samples the OS-level WiFi
statistics, converts them to a Q16.16 CapacityRatio, and feeds them
to the protocol state machine via ResourceMeasurement. The protocol
state machine's MetabolicGovernor uses the wifi_quality dimension
in its min-projection. The MetabolicGovernor does not know that this
measurement came from a WiFi interface; it treats it as one of five
resource dimensions. The min-projection naturally selects the worst
dimension as the bottleneck.

### 3.3 The WiFi-Metabolic State Interaction

The Savant identifies (Stage 2b, Structural Hole 1) that the
current metabolic state machine's responses are tuned for CPU/
bandwidth bottlenecks, not for link-quality bottlenecks. The
response to "WiFi is lossy" should be "send less, tolerate more
jitter, widen scar thresholds," not "send at the same rate but
wait longer between events."

This is a MetabolicGovernor implementation concern. The governor's
threshold table and response table need a second dimension: the
bottleneck resource type. Currently, the governor output is a
single MetabolicState enum. The state drives heartbeat interval,
LEDBAT target, and replication budget. These are uniform responses
regardless of which resource is the bottleneck.

The correct approach: the MetabolicGovernor identifies both the
state AND the bottleneck. The consumer uses both to select the
appropriate response profile:

```rust
struct GovernorOutput {
    state: MetabolicState,        // Background/Idle/Active/Stress
    bottleneck: ResourceType,     // which resource drove the decision
    pressure: CapacityRatio,      // Q16.16, the bottleneck's value
}
```

The consumer's response table becomes two-dimensional:

| State    | CPU bottleneck         | WiFi bottleneck            |
|----------|------------------------|----------------------------|
| Active   | heartbeat 2s, budget 3 | heartbeat 5s, budget 3     |
| Stress   | heartbeat 5s, budget 1 | heartbeat 10s, budget 1    |

When WiFi is the bottleneck, heartbeat intervals increase more
aggressively (to reduce traffic on the lossy link), but replication
budget stays the same (replication is not CPU-bound). When CPU is
the bottleneck, heartbeat intervals increase moderately, and
replication budget drops (replication is CPU-bound).

This does not change the Protocol trait. The GovernorOutput struct
is an internal detail of the MetabolicGovernor implementation. The
consumer reads GovernorOutput and configures the protocol via
ConfigChanged. The Protocol trait receives the configured parameters
(heartbeat_interval_ms, ledbat_target_ms) without knowing why they
were chosen.

### 3.4 PhysicalBoundary Gate for WiFi Quality

The Savant is correct that WiFi quality needs a PhysicalBoundary
gate. The existing six newtypes (SackRatio, CapacityRatio,
TimeDelta, HealthScore, ScarWeight, DominancePenalty) do not cover
the WiFi measurement input path.

However, WifiQuality maps naturally to CapacityRatio. The WiFi
delivery ratio (0.0 to 1.0) is a capacity ratio: 1.0 means full
capacity, 0.85 means 85% capacity. The existing CapacityRatio
newtype (Q16.16) is sufficient. No new PhysicalBoundary newtype
is needed. The ResourceMeasurement input variant uses
CapacityRatio for all resource types, including WiFi quality.

The PhysicalBoundary gate enforcement (lint-enforced, no raw f32
at the boundary) applies: the consumer MUST convert OS-level WiFi
statistics to CapacityRatio before feeding them to the protocol.
The conversion function is outside the protocol state machine
(it is consumer-level code) but it must produce a CapacityRatio,
not an f32. The lint catches any f32 in the protocol crate.

## Section 4: Fixed-Point Violations -- Remaining f32/f64 Remnants

### 4.1 The Savant's Identified Violations

The Savant (Stage 1, Problem 7, and Stage 2b, Section 2) identifies
fixed-point boundary violations in:

1. Expert A's replication roll probability: `deficit as f64 /
   (known_holders as f64 + 1.0)`.
2. Expert A's replication probability threshold: `roll as f64 /
   255.0`.
3. Scar decay formula: implicit floating-point division in some
   expressions.
4. CorrelatedSuspicion confidence field: unspecified type (should
   be Q8.8).
5. EMA replay function: power function approximation.

### 4.2 Additional Violations I Identified in Stage 2a

In my Stage 2a, I identified that Expert A's replication roll MUST
use fixed-point for cross-platform determinism. The BLAKE3 roll
is deterministic per (blob_id, my_dk, epoch), so all peers with
the same inputs must produce the same decision. Floating-point
division can produce different results on different hardware
(x87 vs SSE, ARM vs x86).

### 4.3 New Violations Found in the Savant's Stage 2b

The Savant's own Stage 2b introduces new fixed-point boundary
crossings that should be flagged:

**Violation 1: LEDBAT WiFi correction factor.**

The Savant writes (Stage 2b, Section 4, LEDBAT OWD):

> "When wifi_quality < 0.90, the LEDBAT target widens by
> (1 / wifi_quality^2)."

This is floating-point. The squaring and reciprocal must be
Q16.16 integer operations. The correction factor in fixed-point:

```rust
fn ledbat_wifi_correction_q16(wifi_quality_q16: u32) -> u32 {
    // wifi_quality_q16 is Q16.16, range 0x0000..0x10000
    // We want 1/q^2 in Q16.16
    // q^2 in Q16.16: (q * q) >> 16
    let q_squared = ((wifi_quality_q16 as u64 * wifi_quality_q16 as u64) >> 16) as u32;
    if q_squared == 0 { return 0x00100000; } // cap at 16x
    // 1/q^2 in Q16.16: 0x10000 * 0x10000 / q_squared
    let one_q16 = 0x10000u64;
    let result = (one_q16 * one_q16) / (q_squared as u64);
    result.min(0x00100000) as u32 // cap at 16x
}
```

At wifi_quality 0.85 (Q16.16: 0xD999):
- q_squared = (0xD999 * 0xD999) >> 16 = 0xB8E3 (~0.7225)
- 1/q_squared = 0x10000^2 / 0xB8E3 = ~0x16180 (~1.38 in Q16.16)
- Result: 1.38x target widening. Matches the Savant's f64 calculation.

**Violation 2: WiFi scar decay period calculation.**

The Savant writes (Stage 2b, Mechanism 3):

```
let days_q8_8 = 0x0100 + ((q * 6) >> 0);
days_q8_8 >> 8  // truncate to integer days
```

This is almost correct but the `>> 0` is a no-op, which suggests
the Savant intended a different shift. The formula should be:

```rust
fn wifi_scar_decay_days(wifi_quality_q8_8: u16) -> u16 {
    // wifi_quality_q8_8 is Q8.8, range 0x0000..0x0100
    // days = 1 + wifi_quality * 6
    // In Q8.8: days_q8_8 = 0x0100 + (q * 6)
    // But q * 6 produces Q8.8 * integer = Q8.8
    let days_q8_8 = 0x0100u16.saturating_add(wifi_quality_q8_8.saturating_mul(6));
    // Convert to integer days (truncate)
    days_q8_8 >> 8
}
```

At wifi_quality 0.85 (Q8.8: 0xD9):
- days_q8_8 = 0x0100 + (0xD9 * 6) = 0x0100 + 0x0516 = 0x0616
- days = 0x0616 >> 8 = 6 days.

At wifi_quality 0.60 (Q8.8: 0x99):
- days_q8_8 = 0x0100 + (0x99 * 6) = 0x0100 + 0x039A = 0x049A
- days = 0x049A >> 8 = 4 days.

The Savant's prose says "wifi_quality 0.60 -> 1 day." The
fixed-point formula gives 4 days. There is a discrepancy. The
formula `1 + wifi_quality * 6` at wifi_quality=0.60 gives
1 + 3.6 = 4.6, not 1. The Savant's prose numbers and formula
disagree. The formula is correct; the prose examples are wrong.

If the Savant wants 0.60 -> 1 day, the formula should be
nonlinear, perhaps `days = 1 + (wifi_quality^2 * 6)`. At
wifi_quality 0.60: 1 + 0.36*6 = 3.16 ~ 3 days. Still not 1 day.
For 0.60 -> 1 day, the formula would need to be
`days = ceil(wifi_quality * 7)` which at 0.60 gives ceil(4.2) = 5.
None of these match the Savant's prose examples.

I flag this as a specification inconsistency. The fixed-point
formula needs alignment with the intended behavior curve. The
formula and the prose examples cannot both be correct.

**Violation 3: Supersaturation scaling.**

The Savant's supersaturation formula (Stage 2b, Mechanism 1):

```
let saturation_q16 = (deficit as u32) << 16
    / (priority.target_replicas() as u32);
```

This produces Q16.16 if deficit and target_replicas are small
integers. But the division is integer division of a shifted
numerator, which is the correct fixed-point pattern. No violation
here. The formula is correct as written.

### 4.4 Summary of Remaining Fixed-Point Work

| Location | Violation | Fix |
|----------|-----------|-----|
| LEDBAT WiFi correction | f64 reciprocal-square | Q16.16 integer reciprocal-square with 16x cap |
| WiFi scar decay | Prose/formula mismatch | Align formula to intended behavior curve |
| EMA power approximation | exp(-alpha*dt) | 2KB lookup table per alpha (Savant's proposal, accepted) |
| Replication roll | f64 division | Q16.16 integer division (agreed in Stage 2a) |
| CorrelatedSuspicion.confidence | Untyped | Q8.8 u16 (agreed in Stage 2a) |

No new structural violations beyond these. The protocol state
machine (governor, SACK, LEDBAT, stream mux) is clean. The
violations are all in mesh-layer computations that feed into the
protocol via ConfigChanged or are entirely mesh-internal.

## Section 5: RelayForwarder Refinements from the Pseudo-Peer Model

### 5.1 The Savant's Pseudo-Peer Model

The Savant describes the relay as a pseudo-peer: DK/DEK pair for
Noise session establishment, no CIK (no cluster membership). The
relay runs the Protocol trait for its own sessions and the
RelayForwarder trait for forwarding. The relay does not participate
in gossip, dominance, or convergence.

I accepted this in Stage 2a. The Savant's Stage 2b adds concrete
detail: the relay's consumer (event loop) is ~40% the code of a
full peer's consumer. This is consistent with my RelayForwarder
design.

### 5.2 Refinements to the RelayForwarder Trait

Based on the Savant's Stage 2b additions, I refine the
RelayForwarder trait in three areas:

**Refinement 1: SIGNAL operation type safety.**

My Stage 2a RelayForwarder accepts a generic `&SignalMessage` for
handle_signal(). The Savant's Stage 2b and Expert C's Stage 1
specify three SIGNAL types: HELLO, PUNCH, REDIRECT. Each has a
different payload format. The handle_signal() method should be
type-safe:

```rust
trait RelayForwarder {
    fn forward(
        &mut self,
        frame: &mut [u8],
        source: PeerAddr,
        now_ms: u64,
    ) -> ForwardAction;

    fn handle_hello(
        &mut self,
        hello: &SignalHello,
        source: PeerAddr,
        now_ms: u64,
    ) -> HelloResponse;

    fn handle_punch(
        &mut self,
        punch: &SignalPunch,
        source: PeerAddr,
        now_ms: u64,
    ) -> PunchResponse;

    fn handle_redirect(
        &mut self,
        redirect: &SignalRedirect,
        source: PeerAddr,
        now_ms: u64,
    ) -> RedirectResponse;

    fn poll_expiry(&self) -> Option<u64>;

    fn expire(&mut self, now_ms: u64);
}
```

The three handle methods replace the single handle_signal method.
Each accepts the concrete SIGNAL payload type and returns the
concrete response type. No enum branching inside the forwarder.
The consumer parses the SIGNAL type from the frame and dispatches
to the correct method. This is the same type-separation principle
I apply to poll_transmit/poll_event.

**Refinement 2: now_ms parameter for forward().**

My Stage 2a forward() method does not accept a timestamp. The
Savant's anti-amplification rule (forwarding only to addresses
from which a recent authenticated frame was received within 60
seconds) requires timestamp comparison. The forward() method
needs the current time to check the recency of the last
authenticated frame from the destination address.

Adding `now_ms: u64` to forward() is the simplest solution. The
alternative (having the consumer pre-filter destinations by
recency before calling forward()) moves the anti-amplification
check outside the forwarder, which splits the forwarding logic
between two locations.

**Refinement 3: ForwardAction with metrics.**

The forwarding action should carry basic metrics for the relay's
consumer to use in resource governance:

```rust
enum ForwardAction {
    /// Forward the frame. hop_count already decremented in-place.
    Send {
        destination: PeerAddr,
        frame_len: u16,        // for bandwidth accounting
    },
    /// Process locally (destination is this relay's cluster).
    ProcessLocally,
    /// Drop the frame. Reason provided for diagnostics.
    Drop {
        reason: DropReason,
    },
}

#[repr(u8)]
enum DropReason {
    TtlExhausted = 0,
    NoRoute = 1,
    LoopDetected = 2,
    NotAuthenticated = 3,    // anti-amplification
    RateLimited = 4,
}
```

The frame_len field allows the relay's consumer to track bandwidth
consumption without re-parsing the frame. The DropReason enum
provides diagnostic information for logging and monitoring.

### 5.3 Relay Event Loop Structure

With the refined traits, the relay event loop is:

```rust
loop {
    let datagrams = socket.recvmmsg(batch_size);
    for dgram in datagrams {
        let outer = parse_outer_header(dgram);
        if outer.destination == my_cluster_id {
            // This frame is for us. Process via Protocol trait.
            protocol.handle(
                ProtocolInput::BytesReceived {
                    source: dgram.source,
                    data: dgram.data,
                },
                &arena,
            )?;
        } else {
            // This frame is for someone else. Forward.
            match forwarder.forward(dgram.data, dgram.source, now_ms) {
                ForwardAction::Send { destination, frame_len } => {
                    forward_iovecs.push(iovec {
                        base: dgram.data,
                        dest: destination,
                    });
                    bandwidth_accounting.add(frame_len);
                }
                ForwardAction::ProcessLocally => {
                    // Parse SIGNAL message from the frame.
                    match parse_signal(dgram.data) {
                        Signal::Hello(h) => {
                            let resp = forwarder.handle_hello(&h, dgram.source, now_ms);
                            // ... enqueue response ...
                        }
                        Signal::Punch(p) => {
                            let resp = forwarder.handle_punch(&p, dgram.source, now_ms);
                            // ... enqueue response ...
                        }
                        Signal::Redirect(r) => {
                            let resp = forwarder.handle_redirect(&r, dgram.source, now_ms);
                            // ... enqueue response ...
                        }
                    }
                }
                ForwardAction::Drop { reason } => {
                    diagnostics.record_drop(reason);
                }
            }
        }
    }

    // Drain Protocol trait outputs
    while let Some(tx) = protocol.poll_transmit(&arena) {
        send_iovecs.push(iovec { base: tx.payload, dest: tx.destination });
    }
    while let Some(event) = protocol.poll_event(&arena) {
        relay_mesh.handle_event(event);
    }

    // Batch send all outgoing frames
    socket.sendmmsg(&[&forward_iovecs, &send_iovecs].concat());

    // Expire stale forwarding entries
    if let Some(expiry) = forwarder.poll_expiry() {
        if now_ms >= expiry {
            forwarder.expire(now_ms);
        }
    }

    arena.reset();
}
```

The relay event loop is clean. The outer header parse determines
whether to route through Protocol (own sessions) or RelayForwarder
(transit forwarding). The Protocol trait's four-method drain
(poll_transmit, poll_event) interleaves cleanly with the
RelayForwarder's forwarding actions in the sendmmsg batch.

This is one of the concrete advantages of the four-method model
for the relay: the relay needs to batch both forwarded frames
(from RelayForwarder) and protocol-generated frames (from
poll_transmit) into a single sendmmsg call. The four-method model
allows this because poll_transmit returns Transmit structs that
borrow from the arena. The relay event loop collects forwarded
frames (which borrow from the recvmmsg buffer) and protocol
frames (which borrow from the arena) into a single iovec batch.

With the Savant's two-phase model, the poll_output() returns both
SendBytes and application events. The relay event loop must branch
on every output to separate forwarded frames from protocol frames
from application events. The four-method model avoids this: the
relay knows that poll_transmit() always returns network-bound data
and poll_event() always returns application events.

### 6.1 Protocol Trait (confirmed from Stage 2a)

```rust
trait Protocol {
```

No changes from Stage 2a. The Savant's Stage 2b does not identify

### 6.2 New: ResourceMeasurement Input

```rust
enum ProtocolInput<'a> {
    BytesReceived { source: PeerAddr, data: &'a [u8] },
    ConnectionRequested { destination: PeerAddr },
    ConfigChanged { config: &'a ProtocolConfig },
    CryptoResult { request_id: u32, result: CryptoOutcome<'a> },
    ResourceMeasurement { resource: ResourceType, value: CapacityRatio },
}
```

The ResourceMeasurement variant replaces the previous implicit
assumption that the consumer configures the governor externally.
The governor is part of the protocol state machine. Resource
measurements feed into it via the standard input path.

### 6.3 New: GovernorOutput in Event Stream

```rust
enum Event<'a> {
    // Existing...
    SessionEstablished { connection_id: u32, peer_dk: &'a [u8; 32] },
    HeartbeatReceived { connection_id: u32, extension: &'a [u8; 18] },
    StreamData { connection_id: u32, stream_id: u16, data: &'a [u8] },
    ConnectionClosing { connection_id: u32, reason: CloseReason },
    CryptoRequest { request_id: u32, operation: CryptoOperation<'a> },

    // New: governor state change notification
    MetabolicStateChanged {
        new_state: MetabolicState,
        bottleneck: ResourceType,
        pressure: CapacityRatio,
    },
}
```

When the MetabolicGovernor evaluates a state transition (due to a
ResourceMeasurement input or an internal timer), and the state
changes, the protocol emits a MetabolicStateChanged event. The
consumer (mesh layer) uses this event to adjust mesh-layer
parameters (replication budget, scar thresholds, gossip mode).

The MetabolicStateChanged event carries the bottleneck resource
type, enabling the consumer's two-dimensional response table
(Section 3.3).

### 6.4 Confirmed: Arena Budget

```
Hot arena:    64KB   (per-frame transients, reset per iteration)
Pool arena:  256KB   (retransmission buffers, assembly, verification)
Managed:     256KB   (connection tables, stream state, SACK tracking)
──────────────────────────────────
Total:       576KB
```

Constraint: gossip uses unreliable datagrams at the protocol level.
Control log entry reliability is provided by the mesh layer's
Plumtree IHAVE/GRAFT mechanism, not by stream-level retransmission.

### 6.5 Confirmed: Connection State Machine (relay-first)

Unchanged from Stage 2a. The relay-first connection establishment
with background hole-punch checking produces a simpler state
machine than hole-punch-first. The Savant's split proposal
(relay-first for control plane, hole-punch-first for data plane)
adds complexity without clear benefit: the connection is the same
regardless of traffic class. The NAT traversal state applies to
the connection, not to individual streams. Once a direct path is
established via background checking, all traffic (control and data)
migrates to the direct path.

### 6.6 Confirmed: Timer Management

Internal timer wheel, 4 levels, 256 slots, 1ms resolution. Timer
IDs computed from (connection_index, timer_type, stream_id). New
BackgroundCheck timer type for relay-first model.
