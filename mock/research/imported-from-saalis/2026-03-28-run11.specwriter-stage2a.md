## Part I: Review of the Architecture Surgeon (OP-1 through OP-5)

### OP-1: Declare matter -> protocol dependency

**Verdict: Accept without modification.**

The Surgeon correctly identifies that verkko-matter references Stream, LEDBAT, and Connection without declaring verkko-protocol as a dependency. The fix is mechanical: add one dependency line and update two defined concept bodies. The Surgeon's proposed diff text is clean.

One minor observation: the Surgeon writes "BandwidthPolicy sets the ceiling; LEDBAT enforces the floor" (archsurgeon-stage1, line 100). This phrasing is slightly misleading. LEDBAT does not enforce a floor; it provides an upper bound on sending rate under congestion. BandwidthPolicy provides an administratively configured ceiling. They both constrain from above, with LEDBAT reacting dynamically and BandwidthPolicy set statically. The Surgeon's replacement text at line 132 is better: "intersection of BandwidthPolicy ceilings and LEDBAT congestion control." I recommend using the line 132 phrasing and revising line 100 to match. The word "floor" should not appear in the context of LEDBAT; LEDBAT has no floor by design (it can back off to near-zero).

### OP-2: Promote remaining informal concepts

**Verdict: Accept the SessionResumption promotion. One technical concern.**

The Surgeon's audit of the 13 informal concepts is thorough. 12 of 13 were already promoted; the Surgeon correctly identifies 0-RTT Resumption as the remaining gap and places it as SessionResumption in verkko-protocol.

**Technical concern with the SessionResumption invariant text (archsurgeon-stage1, lines 177-194):**

The Surgeon writes: "Resumption does not weaken forward secrecy: resumed session derives new transport keys via the Noise KK pattern." This is technically incorrect as stated. Noise KK is a full handshake that requires the responder's static key to be known in advance. 0-RTT resumption, by definition, sends data before the handshake completes. The forward secrecy of 0-RTT data is weaker than post-handshake data because 0-RTT data is encrypted under the session export key (derived from the previous session), not under fresh ephemeral keys. The Noise KK pattern used for the resumed session produces fresh transport keys, but those keys protect post-handshake data, not the 0-RTT window.

The invariant should read:

```
- Resumption does not weaken forward secrecy for post-0-RTT data:
  the resumed session derives new transport keys via Noise KK.
  0-RTT data has weaker forward secrecy (encrypted under the
  previous session's export key, not fresh ephemeral DH).
```

This distinction matters because an attacker who compromises the previous session's state can decrypt 0-RTT data but not post-handshake data. The Surgeon's "only idempotent operations permitted in the 0-RTT window" invariant (line 186) correctly addresses the replay risk, but the forward secrecy claim needs qualification.

Additionally, the Surgeon writes "Server validates the resumption token's freshness (expiry bound, monotonic ticket counter)" (line 189). The word "server" is inappropriate in verkko's symmetric protocol model. Both sides are peers. The host peer validates the token. The Surgeon should use "host peer" or "responder" consistently.

## Part II: Review of the Wire Format Engineer (11 wire format diagrams)

### Section 1: Version Negotiation

**This is the subject of Tension 1. Full analysis in Part IV.**

Summary position: I concede on placement. The WireEng's pre-handshake approach is superior for the reasons detailed in the tension analysis below. However, several details of the WireEng's wire format need revision.

**Specific issues with the WireEng's VersionProbe format (wireeng-stage1, lines 36-63):**

1. **Variable-size probe.** The WireEng specifies "16 + 2*version_count bytes. Minimum 18, maximum 28." Variable-size unauthenticated datagrams create parsing complexity. For a field that currently has exactly one valid version (0x0001), sending a variable-width list is over-engineering. I recommend a fixed 24-byte probe with room for 4 versions (8 bytes of version list), zero-padded if fewer than 4. This simplifies the parser and aligns to 8-byte boundaries.

2. **Probe nonce width.** The WireEng uses a u64 probe nonce (lines 50, 61). This is sufficient for replay detection. No objection.

3. **Magic constant.** `0x56524B4F` ("VRKO") is acceptable. I note that my own handshake framing used different per-message magic values ("VKKK", "VKKR", "VXX1", etc. at specwriter-stage1 lines 801-848). If we adopt the WireEng's unified magic for version probes and handshake frames, we need to reconcile. I am willing to adopt the unified magic with a `frame_type` discriminator, as the WireEng proposes. Per-message magic is unnecessary if the frame_type byte is unambiguous.

4. **flags byte.** Bit 0 is `schema_sync_required` (wireeng-stage1, line 58). This flag has questionable value in the version probe phase. SchemaSync is a post-handshake operation. Advertising it pre-handshake adds no benefit because the responder cannot act on it until after the handshake completes. I recommend reserving all 8 flag bits in v1.

### Section 2: Noise Handshake Framing

**The WireEng's handshake frame format (wireeng-stage1, lines 156-187) is well-structured.**

Observations:

1. **Unified 18-byte header for all handshake messages.** Clean. The initiator_cid / responder_cid / msg_index layout is explicit and unambiguous.

2. **connection_id = 0x00000000 for handshakes.** Correct. Handshake frames cannot have a session-bound connection_id because the session does not yet exist.

3. **Prologue binding (lines 195-203).** The WireEng binds `MAGIC || selected_version || hs_type` as the Noise prologue. This is a 7-byte prologue. Good. The binding of the selected version into the prologue means that if version negotiation is tampered with (downgrade attack on the unencrypted probe), the handshake will fail because the peers compute different prologues. This is a meaningful security property that partially mitigates my concern about unencrypted version negotiation.

4. **Handshake message sizes (lines 206-214).** The WireEng lists raw Noise payload sizes. These are correct for the standard Noise XX and KK patterns with X25519 keys.

5. **Handshake Application Payload (lines 216-261).** The WireEng includes Cristian's algorithm timestamps (T1-T4), OBSERVED_ADDR, and schema_hash in the handshake payload. This is denser than my spec, which separated clock sync and OBSERVED_ADDR. I prefer the WireEng's consolidated approach; it reduces round trips.

**One issue: the T4 field (lines 253-254).** The WireEng says "Initiator receive timestamp (filled on receipt, not on wire)." If T4 is not on the wire, it should not appear in the wire format diagram. This is confusing. T4 is computed locally by the initiator when it receives the response. It should be documented as a local computation, not a wire field. The 8 bytes allocated to T4 on the wire are wasted. The WireEng should remove T4 from the wire format and document it as a local-only value. This saves 8 bytes in the handshake payload.

6. **KK-to-XX fallback (lines 263-274).** Correct. On KK decryption failure, discard KK state, generate new ephemeral, send XX msg 0 with hs_type = 0x02. The responder identifies the fallback by the changed hs_type. This matches my spec at specwriter-stage1 lines 785-793.

### Section 3: Connection ID Generation

**This is the subject of Tension 3. Full analysis in Part IV.**

Summary: the WireEng's BLAKE3-derived connection ID is over-engineered for a u32 field. A random_u32 is sufficient and simpler. However, I acknowledge that the WireEng's approach provides one real benefit: migration-safety by binding to session material. See the tension analysis.

### Section 4: SACK Format

**The WireEng's SACK format (wireeng-stage1, lines 346-456) is solid.**

Observations:

1. **Fixed header: 14 bytes.** Compared to my stream-level SACK at 9 + 8*block_count (specwriter-stage1, line 1001), the WireEng's format adds ack_timestamp (4), ack_delay (2), and congestion feedback (3 bytes: recv_buffer_pct + recv_rate_kbps). The congestion feedback fields are valuable for LEDBAT. I did not include them in my SACK-only format. The WireEng is right to add them.

2. **Piggybacked SACK via msg_type bit 7 (lines 422-439).** Using the MSB of msg_type as a SACK piggyback flag is clever and efficient. One bit, zero additional framing. I endorse this.

3. **Sequence number wrap-around (lines 444-446).** Standard modular comparison `(A - B) as i32 > 0`. Correct.

4. **Block count limit of 16 (line 452).** My spec uses 15 (line 993). The difference is minor. I defer to 16 as it maps cleanly to 4 bits. But the WireEng uses a full u8 for block_count, which wastes 4 bits. If block_count max is 16, it fits in 4 bits. The WireEng should consider packing block_count into 4 bits of a shared byte, freeing the other 4 bits for future use. Alternatively, keep the u8 for simplicity and accept the waste.

5. **Missing: recv_window field.** My SACK format includes recv_window (u32, remaining receive window in bytes) at specwriter-stage1 line 997. The WireEng's format does not include a receive window field. The recv_buffer_pct field (percentage, 0-100) is a coarser substitute. For flow control, an absolute byte count (recv_window) is more useful to the sender than a percentage. I recommend adding recv_window to the WireEng's format, or documenting why percentage suffices.

### Section 5: Invite Token Format

**The WireEng's invite token (wireeng-stage1, lines 459-575) is comprehensive.**

Differences from my invite token spec (specwriter-stage1, lines 1541-1594):

1. **Token size.** WireEng: 120 bytes fixed + variable hints. My spec: 182 + 20*hint_count. The difference is my 64-byte Ed25519 signature. The WireEng's token has no signature. This is a significant omission. Without a signature, anyone who obtains the raw token bytes can modify the CIK, PSK, or address hints. The invite token is transmitted via QR code or link, possibly through insecure channels (SMS, email). A signature by the admin DK authenticates the token's origin. I strongly recommend adding the 64-byte signature.

2. **Magic.** WireEng uses `0x53414C49` ("SALI"). Reasonable.

3. **relay_dht_key derivation.** The WireEng specifies `BLAKE3("saalis-dht-v1" || founding_hash)[0..8]` (line 519). This introduces a domain separator ("saalis-dht-v1") not present in the DomainSeparator enum I specified in P0-7 (specwriter-stage1, lines 401-467). If we adopt the DomainSeparator registry principle (all BLAKE3 usage goes through DomainSeparator), then "saalis-dht-v1" must be added as a variant. The WireEng should add it to the enum.

4. **Base45 encoding for QR (line 546).** Good choice. Base45 is the alphanumeric mode of QR codes, maximizing data density. The size estimate (196 chars for 138 bytes, QR version 10) is approximately correct.

5. **Token validation (lines 556-562).** The WireEng's validation includes nonce replay check with 7-day expiry. My spec also includes signature verification. The WireEng omits this step because the token has no signature. Again, the signature should be added.

### Section 6: Control Log Entry Format

**The WireEng's control log entry (wireeng-stage1, lines 578-762) is the most detailed wire format in the entire Stage 1 output.**

Observations:

1. **Fixed overhead: 174 bytes (line 638).** This is substantial. My spec has a smaller header (specwriter-stage1, lines 1702-1716: version + entry_type + scope + lamport + prev_hash + author_dk + payload_len = 1 + 1 + 2 + 8 + 32 + 32 + 2 = 78 bytes before payload). The WireEng adds an entry_hash field (32 bytes) at the front. This self-referential hash is redundant if the signature covers the raw fields directly. However, the WireEng's design (lines 709-726) signs over the entry_hash, not the raw fields. This is a deliberate choice: it allows verification with a single hash + signature check rather than hashing the entire entry every time the signature is verified.

   The tradeoff: 32 bytes of additional overhead per entry vs. faster verification for entries that are verified multiple times (e.g., during gossip propagation). For a control log that is append-only and verified at insertion, the single extra hash is acceptable. For entries that are relayed through gossip to many peers, the pre-computed hash saves repeated computation. I accept this design.

2. **Entry types (lines 640-657).** The WireEng defines 13 entry types (0x01 through 0x0D). My spec defines 11 (0x01 through 0x0B). Differences:
   - WireEng has GATEWAY_CLAIM (0x0C) and GATEWAY_RELEASE (0x0D). I have GATEWAY_CLAIM (0x07) and GATEWAY_RELEASE (0x08) with different numbering. The numbering is arbitrary; the semantics match.
   - WireEng has SNAPSHOT (0x0A). I have SNAPSHOT (0x0A). Match.
   - WireEng has SUCCESSION (0x0B). I have SUCCESSION_EVENT (0x09). Different numbering, same semantics.
   - WireEng has PARTITION_ANN (0x08). I have PARTITION_ANNOUNCE (0x06). Different numbering.
   - WireEng does not have DOMINANCE_OVERRIDE. I have it as 0x0B reserved for future use.

   These must be reconciled to a single numbering. I recommend adopting the WireEng's numbering since the WireEng provides more detailed payload specifications for each type.

3. **Entry type payloads (lines 660-707).** The WireEng specifies byte-level payloads for enrollment certificates, KeyBundle, EDT, PartitionAnnouncement, and ScarEntry. These are dense and useful. My spec covers enrollment certificates separately but with the same fields. The WireEng's payload sizes are well-documented.

4. **GSet merge wire protocol (lines 729-741).** The WireEng specifies batch format: `[u32 batch_count] [entries...]`. Clean. The self-delimiting nature of entries (payload_len determines total size) means no additional framing is needed. I accept this.

5. **Compaction (lines 743-750).** The WireEng's 90-day compaction window matches my spec. The WireEng adds that the snapshot entry is never pruned. Correct invariant.

6. **Forward compatibility (line 759).** "Unknown entry_type: Accept entry. Store but do not process." This is the right behavior for a GSet. Unknown entries are still valid set elements; processing them is the consumer's problem. If a future version adds entry type 0x0E, old peers will store it and forward it via gossip without understanding it. This enables protocol evolution without breaking the hash chain.

### Section 7: Relay Operations Wire Format

**The WireEng's relay format (wireeng-stage1, lines 765-975) is thorough.**

Observations:

1. **13-byte relay operation header (lines 776-797).** Op code, flags, request_id, relay_key_id, reserved. Clean. The request_id (u16) allows 65,536 outstanding requests per session, which is more than sufficient.

2. **Operation codes (lines 799-812).** Response codes use `0x80 | op_code`. This is a standard convention (high bit = response). Clean.

3. **PUT idempotency (lines 834-837).** "If relay_key_id already exists, the PUT is a no-op." This is critical for the idempotent-PUT invariant from the design docs. The WireEng correctly notes that deterministic nonces guarantee identical ciphertext, so no content comparison is needed. This matches my spec at specwriter-stage1, line 1614 (AlreadyExists status).

4. **Batched GET with decoys (lines 855-866).** The WireEng's approach to constant-rate padding through batched GET with decoy key_ids is elegant. The relay processes all key_ids identically because it cannot distinguish real from decoy without the channel key. This matches the constant-rate padding specification from the design.

5. **SIGNAL operation (lines 916-938).** The WireEng defines HELLO, PUNCH, and REDIRECT signal types. My spec defines CONNECT and PING (specwriter-stage1, line 1672). These should be reconciled. The WireEng's HELLO/PUNCH/REDIRECT is more descriptive and maps to distinct NAT traversal phases. I prefer the WireEng's naming.

6. **Constant-rate padding batch_size derivation (lines 959-973).** The WireEng derives batch_size from HKDF-Expand with epoch_key, producing a value in [10, 25]. This is specific, deterministic, and changes each epoch. Good. My spec does not specify the batch_size derivation. The WireEng's approach is better because it prevents the relay from detecting traffic patterns via batch size variation.

### Section 8: Seat Session Establishment

**The WireEng's seat authentication (wireeng-stage1, lines 977-1143) is a wire-level complement to my seat protocol spec (specwriter-stage1, lines 1062-1337).**

Observations:

1. **Authentication types.** The WireEng has three types: Password (0x01), Resumption Token (0x02), PIN (0x03). My spec has three: Password (0), Token (1), Resumption (2). The WireEng adds PIN as a separate type. This is reasonable for Batocera/loisto appliance usage where a numeric PIN is appropriate for the Child role. I accept PIN as a third authentication type.

2. **Seat Auth Request inside Noise XX handshake payload (line 992).** The WireEng places the authentication request inside the Noise handshake payload, not as a separate post-handshake message. This reduces round trips: the seat authenticates during the handshake rather than after it. My spec (specwriter-stage1, lines 1091-1100) describes authentication as a separate post-handshake message sequence. The WireEng's approach is more efficient. However, it couples seat authentication to the Noise handshake, which means the authentication credential is encrypted under the Noise handshake keys, not the transport keys. For XX (first contact), this is fine because the handshake establishes encryption. For KK (known peer), the credential is encrypted under the KK session keys. Both are acceptable.

   I concede on placing authentication inside the handshake payload for efficiency. But the WireEng should clarify what happens if authentication fails: the Noise handshake has already completed, so transport keys exist. The session must be torn down explicitly after an auth failure. My spec handles this with the SeatAuthResult status codes.

3. **Event subscriptions in the auth request (lines 1020-1023).** The WireEng includes event category subscriptions directly in the auth request, combining authentication and subscription in one message. My spec separates them into SeatAuth and SubscriptionRequest (specwriter-stage1, lines 1096-1098). The WireEng's approach is more compact but less flexible: the seat cannot change subscriptions later without re-authenticating. I recommend keeping subscriptions separate (as a post-auth message), or adding a separate SubscriptionUpdate message type for runtime subscription changes.

4. **Seat Auth Response (wireeng-stage1, lines 1050-1087).** The WireEng includes consistency_mode, session_token, token_expiry, and schema_update in the response. This matches my spec's SeatAuthResult (specwriter-stage1, lines 1113-1128). The WireEng's format is more compact (binary framing vs my spec's field-by-field description). Both are functionally equivalent.

5. **Seat Message Frame (lines 1094-1114).** The WireEng defines a 6-byte message frame: direction (1), msg_type (1), correlation_id (2), payload_length (2). My spec uses an 8-byte request envelope: correlation_id (4), request_type (2), payload_len (2). Differences:
   - WireEng uses u16 correlation_id; I use u32. For a seat session that handles a few hundred requests, u16 (65,536 values) is sufficient. But u32 is safer against long-lived sessions. I marginally prefer u32.
   - The WireEng adds a `direction` byte to distinguish request, response, event, instruction, and progress. My spec uses separate message types for each direction. The WireEng's approach is cleaner: one framing format with a direction discriminator rather than separate envelope formats per direction.

### Section 9: Data Frame Format

**The WireEng's consolidated data frame format (wireeng-stage1, lines 1146-1255) is the canonical reference wire format.**

Observations:

1. **29-byte outer header (lines 1158-1185).** Destination (4), connection_id (4), hop_count (1), relay_key_id (8), nonce (12). This matches the design docs. The WireEng adds byte offsets and endianness annotations. Clean.

2. **Encrypted payload inner header (lines 1186-1217).** Channel_index (2), epoch (4), micro_epoch (varint 1-2), dest_peer (2), msg_type (1). The varint for micro_epoch is a pragmatic choice: 1 byte covers values 0-127, which handles most cases. 2 bytes covers up to 16,383.

3. **AEAD associated data (lines 1218-1220).** The outer header (29 bytes) is the associated data for ChaCha20-Poly1305. This binds ciphertext to routing information. Correct.

4. **msg_type registry (lines 1223-1240).** 12 message types defined. PATH_CHALLENGE (0x06) and PATH_RESPONSE (0x07) are included for connection migration. CHAIN_PROOF (0x0C) for re-seed chain proofs. All documented.

5. **Maximum frame size (lines 1242-1254).** 1280 - 29 - 10 - 16 = 1225 bytes max data payload. With piggybacked SACK: 1195 bytes. These numbers are tight but correct for IPv6 minimum MTU. The WireEng correctly uses 1280 bytes (IPv6 min MTU minus IP/UDP headers would be 1280 - 40 - 8 = 1232, but the WireEng appears to use 1280 as the UDP payload limit directly). Actually, the IPv6 minimum MTU is 1280 for the entire IP packet; the UDP payload is 1280 - 40 (IPv6 header) - 8 (UDP header) = 1232 bytes. The WireEng's 1280-byte UDP payload assumption is 48 bytes too generous. This should be corrected to 1232 bytes, making the max data payload 1232 - 29 - 10 - 16 = 1177 bytes.

   **This is a bug.** The WireEng should use 1232 bytes as the UDP payload limit, not 1280. The 1280-byte figure is the IPv6 minimum MTU at the IP layer, not the UDP payload. If the verkko design docs state 1280-byte UDP segments, the docs should be corrected too.

### Section 10: Stream Multiplexing

**The WireEng's stream format (wireeng-stage1, lines 1258-1300) is compatible with my spec.**

Differences:

1. **Stream header size.** WireEng: 6 bytes (stream_id 2 + sequence 4). My spec: 7 bytes (stream_id 2 + sequence 4 + flags 1). The WireEng omits the flags byte. My flags byte carries FIN, RST, and SACK_PRESENT bits (specwriter-stage1, lines 948-953). FIN and RST are necessary for stream lifecycle management. The WireEng should add a 1-byte flags field to the stream header, making it 7 bytes.

   Without FIN and RST, streams cannot be closed or reset. The WireEng's stream header as-is creates streams that can be opened (implicitly, on first use) but never closed. This is a gap.

2. **Stream ID allocation ranges.** The WireEng (lines 1292-1298) uses:
   - 0x0000: default stream (control)
   - 0x0001-0x00FF: vital tier
   - 0x0100-0x0FFF: reactive tier
   - 0x1000-0xFFFE: bulk tier

   My spec (specwriter-stage1, lines 958-971) uses:
   - 0x0000: control stream (vital tier 1)
   - 0x0001: gossip stream (vital tier 2)
   - 0x0002-0x00FF: application streams (reactive/bulk)
   - 0x0100-0xFFFE: bulk transfer streams (LEDBAT-paced)
   - 0xFFFF: reserved

   The WireEng's allocation is more structured (range-based tier mapping). I prefer it. The explicit tier ranges eliminate per-stream configuration: any stream in 0x1000-0xFFFE is automatically LEDBAT-paced. Clean.

3. **HEARTBEAT and SACK exclusion (lines 1287-1288).** The WireEng correctly states that heartbeat and SACK are connection-level, not stream-level. They do not carry stream headers. This matches the design.

### Section 11: SchemaSync Wire Format

**The WireEng's SchemaSync (wireeng-stage1, lines 1302-1323) is minimal but correct.**

The schema exchange piggybacks on the handshake payload via schema_hash (CRC-32C). On mismatch, a SCHEMA_SYNC frame carries the full protobuf FileDescriptorSet. The version field enables unambiguous schema ordering.

One concern: CRC-32C as the schema hash. CRC-32C is a 32-bit check value, not a cryptographic hash. Two different schemas could collide on CRC-32C with probability 1/2^32. For a protocol that may evolve over years with many schema versions, a cryptographic hash (even truncated BLAKE3[0..4]) would be safer. However, schema collision is not a security issue (it would cause a false "schemas match" result, leading to protobuf parsing errors). The failure mode is detectable at the application layer. CRC-32C is acceptable for v1.

### Appendix A: Magic Numbers and Frame Type Summary

**The WireEng's summary (wireeng-stage1, lines 1326-1349) provides a clear frame disambiguation rule.**

"The receiver checks the first 4 bytes. If they match the magic (0x56524B4F), it is a pre-handshake frame. Otherwise, it is a post-handshake data frame." The probability of collision with BLAKE3(CIK_pubkey)[0..4] is 1/2^32. This is a clever optimization that avoids a separate port or protocol discriminator.

**Potential issue:** If an attacker can craft a CIK whose BLAKE3 hash starts with 0x56524B4F, they could cause frame misclassification. This is computationally feasible (~2^32 attempts) but would only affect routing, not security (the attacker cannot decrypt the frame). The practical impact is negligible.

### Appendix B: Domain Separator Registry Additions

**The WireEng adds "saalis-connid-v1" to the DomainSeparatorRegistry (wireeng-stage1, lines 1350-1358).** If we adopt the WireEng's connection ID generation, this is necessary. If we adopt my random_u32 approach, it is unnecessary. See Tension 3.

## Part III: Review of the Cartographic Savant

### Territory Survey

### Novel Mechanisms Analysis

The Savant proposes three novel mechanisms (N1, N2, N3). I evaluate each as a protocol specification writer.

**N1: SansIO Event Surface (lines 237-270).**

The proposed ProtocolInput/ProtocolOutput/ProtocolMachine trait is a reasonable starting point but has specification gaps:

1. **Missing inputs.** The Savant lists BytesReceived, TimerExpired, ConnectionRequested, ArenaRefill. Missing from this list: ConnectionClosed (local close request), StreamCreated (new stream request), StreamClosed (local stream close), HeartbeatTick (the protocol needs a periodic tick for heartbeat generation, not just timer expiry), ConfigChanged (MetabolicGovernor output delivered to the protocol machine).

2. **Missing outputs.** The Savant lists SendBytes, SetTimer, SessionEstablished, SessionLost, FrameReady, NeedArena. Missing: HeartbeatExtensionRequest (the protocol needs to request mesh-layer extension data for outbound heartbeats), ConnectionMigrated (IP address changed, session preserved), StreamReset (remote RST received), MetabolicStateChanged (governor transition notification to the consumer).

3. **Polling interface.** The Savant specifies `fn handle(&mut self, input: ProtocolInput) -> SmallVec<[ProtocolOutput; 4]>` and `fn poll_timeout(&self) -> Option<u64>`. This is an event-driven interface. An alternative is a readiness-based interface: `fn poll(&mut self) -> ProtocolOutput` returning the next ready output, called in a loop until it returns None. The event-driven approach batches outputs per input; the readiness approach decouples input processing from output delivery. For a sans-IO protocol, the readiness approach is more common (cf. quinn-proto, which uses a combined poll/handle design). The Savant's choice is acceptable but should be evaluated against the readiness pattern during detailed design.

4. **Arena ownership.** The Savant writes "The consumer owns the allocation. The protocol borrows. No heap allocation inside the protocol." This is a strong constraint. It means the protocol machine cannot hold state that outlives a single handle() call unless the arena is persistent. For session state (connection tables, stream state, ratchet keys), the protocol needs persistent storage. The ArenaRefill model works for transient buffers (frame assembly, decryption scratch space) but not for persistent state. The interface should distinguish between transient buffers (arena-allocated) and persistent state (owned by the protocol machine).

**N2: IntraClusterTransport (lines 275-293).**

The Savant proposes Noise KK with CIK as static key for intra-cluster communication. Technical assessment:

1. **CIK as static key for Noise KK.** This means all peers in a cluster use the same static key. In Noise KK, both peers must know each other's static key. If all peers share the same CIK, then Noise KK degenerates: both sides use the same static key. The DH computations still produce unique session keys (because ephemeral keys are unique), but the static-static DH (ss) in KK is a computation with a known result (CIK * CIK is the same for all peer pairs). This weakens the authentication: any peer in the cluster can impersonate any other peer in the cluster. This is acceptable if intra-cluster peers are fully trusted (they are; they share CIK). But it means that the Noise KK pattern does not provide peer-specific authentication within the cluster. The session key uniqueness comes entirely from the ephemeral DH (ee).

2. **Alternative: Noise KK with per-peer DEK.** Instead of using CIK (shared) as the static key, each intra-cluster session could use the peers' individual DEK keys. This provides peer-specific authentication: peer A and peer B produce a session key that cannot be computed by peer C, even though C holds CIK. However, this requires each peer to know every other peer's DEK, which means a full key distribution within the cluster. For clusters of 2-6 peers, this is feasible.

3. **The Savant's proposal is the simpler choice.** CIK-based Noise KK is simpler, works with the existing key material, and provides adequate security within a trusted cluster. I would note the authentication limitation explicitly but accept the design.

**N3: MatterChunkProtocol (lines 296-325).**

Reasonable overall. My concerns:

1. **transfer_id (u64).** Sufficient for uniqueness. But the Savant does not specify how transfer_id is generated. If random, it should be from CSPRNG. If sequential, it leaks transfer order.

2. **bao_proof variable size.** The Savant writes `bao_proof: [u8; PROOF_LEN]` without specifying PROOF_LEN. For a bao tree with N chunks, each proof is `ceil(log2(N)) * 32` bytes. For a 2GB blob at 1145-byte chunks, N = ~1.8M, so proof length = 21 * 32 = 672 bytes. This is larger than the chunk_data itself at small chunk sizes. The Savant's format would need the proof to be sent separately or the chunk size to be much larger (e.g., 16 KB chunks, requiring stream-level reassembly across multiple frames).

3. **Multi-source chunk interleaving.** The Savant correctly identifies that bao tree proofs enable multi-source transfer: chunks from different sources can be verified independently. This is a key advantage of bao over Merkle trees with ordered verification.

## Part IV: Tension Positions

### T1: Version Negotiation Timing

**Tension:** I placed version negotiation inside the Noise handshake encrypted payload. The WireEng placed it before the Noise handshake as an unencrypted VersionProbe/VersionResponse exchange.

**I concede. The WireEng's pre-handshake approach is superior.**

My reasoning for placing it inside the handshake was confidentiality: version information could leak to passive observers. But this reasoning is flawed for three reasons:

1. **Wasted Noise computation.** If two peers have no common version, my approach requires completing a full Noise handshake (2 messages for KK, 3 for XX) before discovering the incompatibility. The handshake involves X25519 DH operations, HKDF, and ChaCha20-Poly1305 encryption. For a version mismatch (which is the most common failure mode during protocol upgrades across a mesh), this is pure waste.

2. **Version information is not secret.** The protocol version tells an observer what software the peer is running, but this information is already available via other means (port numbers, timing patterns, packet sizes). Encrypting the version adds no meaningful security.

3. **Prologue binding mitigates downgrade attacks.** The WireEng binds the selected version into the Noise prologue (wireeng-stage1, lines 195-203). This means that if an active attacker modifies the unencrypted VersionProbe to force a downgrade, the Noise handshake will fail because the two peers compute different prologues (one with the real version, one with the forged version). This is the standard defense against downgrade attacks in protocol negotiation (cf. TLS 1.3's downgrade sentinel).

4. **The pre-handshake probe is cheap.** 24 bytes, one round trip, no cryptographic operations. The cost of the version probe is negligible compared to the cost of a Noise handshake.

**Security analysis of pre-handshake version negotiation:**

The pre-handshake approach has three attack surfaces that must be addressed:

1. **Downgrade attack.** An active attacker modifies the VersionProbe to advertise only an old, vulnerable version. The WireEng's prologue binding mitigates this: the selected version is bound into the Noise prologue, so a tampered version selection produces a handshake failure. The initiator can detect this: if the handshake fails after a version probe succeeded, the initiator should suspect tampering and retry with direct version specification (skipping the probe entirely and attempting KK with the latest version).

2. **Amplification attack.** An attacker spoofs the source IP of a VersionProbe and directs VersionResponses to a victim. The amplification factor is approximately 1:1 (24 bytes in, 24 bytes out), so this is not a useful amplifier. No mitigation needed beyond standard IP-level anti-spoofing.

3. **Fingerprinting.** An observer who sees VersionProbes in the clear can determine which verkko version a peer runs. For a home mesh appliance, this leaks the software version to the ISP or any on-path observer. The information value is low (the ISP already knows the device type via MAC address OUI and traffic patterns), but it is nonzero. If this concerns users, a future version could add an optional "version probe encryption" mode using a pre-shared hint (e.g., derived from the founding hash). For v1, this is not worth the complexity.

**Concession with caveats:** The WireEng's pre-handshake probe is the right placement. However, I recommend three modifications:

- **Rate limiting.** The VersionProbe is unauthenticated. An attacker can flood a peer with VersionProbes. The responder should rate-limit VersionResponse generation (e.g., 100 per second per source IP). The WireEng does not specify rate limiting for version probes. Add: "The responder SHOULD rate-limit VersionResponse to at most 100 responses per second per source address. Probes exceeding the limit are silently dropped."

- **No flags in v1.** Remove the `schema_sync_required` flag from the VersionProbe. Reserve all flag bits. Schema sync is a post-handshake concern that cannot be acted on pre-handshake.

- **Downgrade detection.** Add: "If a Noise handshake fails immediately after a successful version probe, the initiator SHOULD retry with direct version specification (highest supported version, no probe) before declaring the peer unreachable. This detects active downgrade attacks on the version probe."

### T2: BLAKE2b vs. BLAKE2s in the Noise Protocol Name

**Tension:** I specified `Noise_KK_25519_ChaChaPoly_BLAKE2b`. The WireEng specified `Noise_KK_25519_ChaChaPoly_BLAKE2s`.

**I maintain BLAKE2b. The WireEng should concede.**

Technical reasoning:

1. **BLAKE2b is the standard choice for 64-bit platforms.** verkko targets x86_64 and aarch64 (Batocera runs on both). BLAKE2b operates on 64-bit words and is faster on 64-bit hardware. BLAKE2s operates on 32-bit words and is faster on 32-bit hardware (embedded systems, microcontrollers). Since verkko does not target 32-bit platforms, BLAKE2b is the correct choice.

2. **BLAKE2b produces 64-byte output; BLAKE2s produces 32-byte output.** The Noise framework uses the hash for handshake state hashing. BLAKE2b's larger internal state provides more security margin. For a protocol that handles key material, the additional margin is worth the negligible performance cost.

3. **Consistency with the broader verkko crypto choices.** verkko uses BLAKE3 (which is based on BLAKE2s internally but produces 256-bit output) for content hashing and domain separation. However, the Noise protocol name string is part of the Noise specification, not the verkko crypto layer. The Noise framework defines specific protocol name strings for specific hash functions. Using BLAKE2b in the Noise handshake and BLAKE3 in the application layer is standard practice; they serve different roles.

4. **Interoperability with Noise libraries.** Both BLAKE2b and BLAKE2s are supported by all major Noise libraries (snow, noiseprotocol, noise-java). The choice does not constrain implementation.

5. **The Noise specification recommends BLAKE2b for general use.** The Noise spec (rev 34, section 12.1) lists `BLAKE2b` as the default recommendation and `BLAKE2s` as the embedded systems alternative.

**The WireEng's choice of BLAKE2s appears to be an error, not a deliberate design decision.** The WireEng does not provide a rationale for BLAKE2s. The Noise protocol name string is not a casual choice; it is hashed into the handshake state and determines the hash function used for all handshake computations. Getting this wrong means all implementations must agree, or sessions will silently fail.

**Resolution:** `Noise_KK_25519_ChaChaPoly_BLAKE2b` and `Noise_XX_25519_ChaChaPoly_BLAKE2b`. All implementations MUST use BLAKE2b.

### T3: Connection ID Generation

**Tension:** I specified `connection_id = random_u32()`. The WireEng specified a BLAKE3-derived connection ID from CIK pair, handshake nonce, and session counter.

**I maintain random_u32, but I acknowledge one valid concern from the WireEng.**

Arguments for random_u32:

1. **Simplicity.** One CSPRNG call vs. BLAKE3 hash with 4 inputs and a domain separator. The additional complexity buys almost nothing for a u32 field.

2. **Same collision probability.** Both approaches produce uniformly distributed u32 values. The birthday bound is identical: ~65,536 sessions for 50% collision probability. The WireEng's session_counter input (wireeng-stage1, line 340-341) mitigates collisions by regenerating on collision, but random_u32 can do the same: generate, check for collision with existing sessions, regenerate if collision.

3. **No new domain separator needed.** The WireEng's approach requires adding "saalis-connid-v1" to the DomainSeparatorRegistry. This adds complexity to the registry for no security benefit.

4. **Pre-handshake ID assignment.** With random_u32, the initiator can assign a connection_id before the Noise handshake begins (as stated in my spec, line 905: "Generated: at session initiation (Noise message 1 sent)"). With the WireEng's approach, the connection_id cannot be generated until the handshake nonce (the Noise ephemeral key) is available, which is during message 1 construction. In practice, both happen at the same time, so this difference is minimal.

**The WireEng's valid concern: unpredictability for off-path observers (wireeng-stage1, line 326).** A random_u32 from a CSPRNG is unpredictable to off-path observers. A BLAKE3-derived ID is also unpredictable because the handshake nonce is unknown to off-path observers. Both provide the same security property. The WireEng's claim that derivation provides additional unpredictability is incorrect: CSPRNG output is, by definition, computationally indistinguishable from random. There is no "more unpredictable."

**The WireEng's valid concern: migration-safety (wireeng-stage1, line 328).** The WireEng writes "The connection_id is bound to the session, not the IP address." This is true for both approaches. A random_u32 assigned at session initiation is also bound to the session for its lifetime. The binding is logical (the session state associates the connection_id), not cryptographic. The WireEng's derivation makes the binding cryptographic but this adds no practical benefit because the session state already maintains the association.

**Resolution:** `connection_id = random_u32()`. Reserved values: 0x00000000 (handshake) and 0xFFFFFFFF. Regenerate on collision with existing sessions.

### T5: HostPeer Translation Layer

**Tension:** The Savant argues my seat spec is incomplete without specifying HostPeer's internal translation from SeatRequest to mesh operations. My seat spec treats HostPeer as opaque.

**I partially concede. The seat spec is implementable from the seat's perspective, but the system spec is incomplete without the translation layer.**

The Savant's analysis (savant-stage1, lines 197-205) is technically correct. The HostPeer translation layer is the bridge between two territories (seat and mesh) that neither claims. Every implementer will make ad-hoc decisions about how to translate SeatRequest types into mesh operations (filter lookups, channel access checks, matter transfer initiation).

However, the Savant is wrong about where the translation layer belongs. The Savant writes (line 203): "It is not in seat (seat does not know mesh). It is not in mesh (mesh does not know seat). It lives in the application layer, outside all seven documents."

This is exactly right, and it is the correct outcome. The HostPeer translation layer is the integration point between the verkko protocol stack and the saalis application. It belongs in neither verkko-seat nor verkko-mesh. It belongs in the saalis application layer (saalis-core or a dedicated saalis-verkko adapter crate). The verkko documents specify the two sides of the bridge:

- verkko-seat specifies SeatRequest types, SeatResponse types, SeatEvent types, access control rules, and consistency model.
- verkko-mesh specifies the mesh operations that the HostPeer performs: filter queries, channel access checks, matter transfer, control log operations.

The translation is a mapping from one to the other. This mapping is application-specific: saalis maps "list my games" to a cuckoo filter query plus metadata lookup. A different application using verkko would map different request types to different mesh operations.

**Where I concede:** The seat spec should explicitly state that the HostPeer translation layer is an application-layer responsibility, not a verkko-layer responsibility. My spec currently treats HostPeer as opaque without saying why. The spec should add a normative note:

```
HostPeer translation: The mapping from SeatRequest types to mesh
operations is the responsibility of the application layer (e.g.,
saalis-core). This document specifies the wire protocol between
seat and host peer. It does not specify how the host peer
fulfills requests internally, because that mapping is
application-specific. Two applications using the verkko stack
may implement different HostPeer translation layers.
```

This makes the boundary explicit without expanding the verkko spec into application territory.

**Where I disagree with the Savant:** The Savant implies (lines 204-205) that without the translation specification, "the HostPeer will be the place where every implementer makes ad-hoc decisions." This is true but acceptable. The HostPeer translation layer IS the application layer. It is supposed to be where the application makes decisions. The verkko spec provides the primitives (SeatRequest/SeatResponse wire protocol + mesh operations). The application provides the mapping. This is standard layered protocol design.

The Savant's river metaphor (line 201) is evocative but misleading. The bridge across the river is not drawn on either territory's map because the bridge IS the application. You do not put the bridge on the crypto map or the mesh map. You put it on the application map. And the application map is not a verkko document.

**Concrete example of why the translation layer belongs in the application:**

Consider the SeatRequest "list content matching query Q." In saalis, this means:
1. Check the seat's role permits listing (access control -- specified in verkko-seat).
2. Query the hot store for entities matching Q (saalis-core concern -- not a verkko operation).
3. For each entity, check channel access (verkko-mesh ChannelKey lifecycle check).
4. Return the filtered entity list as a SeatResponse payload (verkko-seat wire format).

Steps 1 and 4 are verkko-seat. Step 3 is verkko-mesh. Step 2 is pure saalis (entity model, hot store query, metadata schema). The translation layer must know about saalis entities, which are outside the verkko specification entirely. A different application using verkko (e.g., a file sharing tool instead of a game library) would have a completely different step 2 and might not have step 3 at all.

This confirms that the translation layer is application-specific and must live outside the verkko spec. The Savant's concern is valid (the gap exists) but the gap is intentional and correct.

**What the verkko spec should provide to make the translation layer implementable:**

Rather than specifying the translation, the verkko spec should provide two things that it currently lacks:

1. **A normative note in verkko-seat** stating that the HostPeer translation layer is an application-layer responsibility, with a list of verkko-mesh operations that the HostPeer may need to invoke (filter queries, channel access checks, matter transfer initiation, control log queries, health pipeline queries). This gives the application implementer a checklist without dictating the translation logic.

2. **A HostPeer capability enumeration in verkko-seat** listing the mesh operations that the HostPeer is expected to support. This is not a wire format; it is a contract: "the HostPeer MUST be able to perform the following mesh operations on behalf of the seat." The application implements these operations by calling mesh APIs.

These two additions make the seat spec implementable without specifying the translation logic itself.

- SpecWriter (lines 154-181): filter_root(8) + epoch_current(4) + work_capacity(1) + health_canary(1) + observation_summary(4) = 18. Big-endian.
- Surgeon (lines 349-360): Same field order, same sizes. Match.

- **Status: consistent.**

**Outer header (29 bytes):**

- WireEng (lines 1158-1185): Same layout, same field widths. Explicit byte diagram.
- **Status: consistent.**

**Noise handshake message sizes:**
- SpecWriter (lines 800-849): KK msg1 = 87 bytes total (magic 4 + version 1 + payload_len 2 + noise 48 + encrypted 32). Non-standard framing with per-message magic.
- WireEng (lines 206-214): KK msg0 = 48 bytes noise payload (e, es, ss). KK msg1 = 48 bytes (e, ee, se). Raw Noise payload only; add 18-byte handshake frame header.
- **Discrepancy:** SpecWriter uses a 7-byte custom header (magic 4 + version 1 + payload_len 2). WireEng uses an 18-byte standard handshake frame header. These are different framing approaches. The WireEng's framing is more extensible (includes initiator_cid, responder_cid, msg_index, reserved). The SpecWriter's framing is more compact but less informative.
- **Resolution needed:** Adopt the WireEng's 18-byte handshake frame header. It carries connection IDs that enable session multiplexing at the transport layer.

**SACK format:**
- SpecWriter (lines 986-1001): stream-level SACK. cumulative_ack(4) + block_count(1) + sack_blocks(8*N) + recv_window(4) + recv_buffer_pct(1) + recv_rate_kbps(2). Total overhead: 12 + 8*N.
- WireEng (lines 356-416): connection-level SACK. cumulative_ack(4) + ack_timestamp(4) + ack_delay(2) + block_count(1) + recv_buffer_pct(1) + recv_rate_kbps(2) + sack_blocks(8*N). Fixed header: 14 bytes.
- **Discrepancy:** Different header sizes (12 vs 14). The WireEng adds ack_timestamp and ack_delay fields that the SpecWriter omits. The WireEng omits recv_window that the SpecWriter includes.
- **Resolution needed:** Merge. Include ack_timestamp, ack_delay (from WireEng), and recv_window (from SpecWriter). Final fixed header: 18 + 8*N. The additional 4 bytes (ack_timestamp + ack_delay total: 6; recv_window: 4; net addition from WireEng base: +4 over WireEng, +6 over SpecWriter) are justified by the congestion control information they carry. At typical block_count=0, a standalone SACK is 18 bytes of payload inside the encrypted frame. Well within budget.

**Invite token:**
- SpecWriter (lines 1556-1577): version(1) + cik_pub(32) + psk(32) + hint_count(1) + hints(20*N) + relay_dht_key(8) + founding_hash(32) + expiry_epoch(4) + nonce(8) + signature(64) = 182 + 20*N.
- WireEng (lines 468-520): magic(4) + version(1) + token_type(1) + flags(1) + hint_count(1) + cik_pub(32) + psk(32) + founding_hash(32) + expiry_epoch(4) + nonce(8) + relay_dht_key(8) + hints(variable) = 124 + hints. No signature.
- **Discrepancies:**
  1. WireEng has magic (4 bytes), token_type (1 byte), flags (1 byte). SpecWriter does not.
  2. WireEng hint format is 7/19 bytes (addr_type + address + port). SpecWriter hint format is 20 bytes (fixed, padded).
  3. WireEng has no signature. SpecWriter has 64-byte Ed25519 signature.
- **Resolution needed:** Adopt the WireEng's magic and type/flags fields (useful for extensibility). Add the SpecWriter's 64-byte Ed25519 signature (required for authentication). Use the SpecWriter's fixed-size hint format (20 bytes, padded) for parser simplicity.

**Control log entry:**
- SpecWriter (lines 1702-1732): version(1) + entry_type(1) + scope(2) + lamport(8) + prev_hash(32) + author_dk_pub(32) + payload_len(2) + payload(var) + signature(64) = 142 + payload.
- WireEng (lines 586-638): entry_hash(32) + prev_hash(32) + lamport(8) + entry_type(1) + reserved(1) + channel_index(2) + author_dk(32) + payload_len(2) + payload(var) + signature(64) = 174 + payload.
- **Discrepancy:** WireEng adds a 32-byte entry_hash at the front. WireEng also omits the version byte. The entry_hash is self-referential (BLAKE3 of the remaining fields). As noted in Part II, this pre-computed hash enables faster verification during gossip propagation.
- **Resolution needed:** Adopt the WireEng's entry_hash approach. Add the SpecWriter's version byte for forward compatibility. Final fixed overhead: 175 + payload.

- SpecWriter: EpochRatchet covered in DomainSeparator test vectors.
- WireEng: epoch (u32) and micro_epoch (varint) in the encrypted payload (lines 1210-1212). The micro_epoch varint encoding is LEB128.
- **Potential issue:** The design docs do not specify whether micro_epoch is LEB128 or fixed-width. The WireEng's choice of LEB128 is reasonable for saving 1 byte in the common case (micro_epoch < 128). But LEB128 introduces variable-width parsing inside the encrypted payload, which adds complexity. A fixed u16 would be simpler and waste at most 1 byte. Given that the encrypted payload is already variable-length (due to msg_type-specific data), one more byte of overhead is negligible.
- **Recommendation:** Use fixed u16 for micro_epoch instead of LEB128 varint. This simplifies the parser without meaningful cost. The maximum micro_epoch value of 65,535 is more than sufficient (at 1 micro-epoch per second, this covers 18.2 hours within a 1-hour macro-epoch, which is impossible by construction since macro-epochs are hourly).

## Part VII: Technical Errata Found During Cross-Review

### Errata in my own Stage 1

1. **Handshake magic values.** I used per-message magic ("VKKK", "VKKR", "VXX1", etc.) in P1-3 (specwriter-stage1, lines 801-848). The WireEng uses a unified magic with frame_type discriminator. I should adopt the unified approach.

2. **OBSERVED_ADDR format.** My spec (specwriter-stage1, lines 851-857) uses 18 bytes with a 16-byte fixed address field. The WireEng (wireeng-stage1, lines 256-257) uses variable-size (4 or 16 bytes based on addr_type). My fixed-size approach wastes 12 bytes for IPv4 but simplifies parsing. The WireEng's variable-size approach is more compact. For a field that appears once per handshake, the 12-byte waste is negligible. I prefer fixed-size for parser simplicity.

3. **SeatAuth credential_len as u16.** My spec (specwriter-stage1, line 1108) uses u16 for credential_len. The WireEng (wireeng-stage1, line 1018) also uses u16 with max 256 bytes. Agreement.

4. **SACK msg_type.** I used msg_type 0xFE for SACK-only frames (specwriter-stage1, line 1470, 1486). The WireEng uses msg_type 0x05 (wireeng-stage1, line 434). The WireEng's 0x05 is better because it fits in the msg_type registry alongside other message types. 0xFE suggests a "reserved" value that is not really reserved. I adopt the WireEng's 0x05.

5. **SessionResumption forward secrecy claim.** As noted in the Surgeon review, my P0-2 reference to 0-RTT forward secrecy needs qualification. The 0-RTT window has weaker forward secrecy than post-handshake data.

### Errata in the WireEng's Stage 1

1. **UDP payload limit.** 1280 bytes is the IPv6 minimum MTU at the IP layer. The UDP payload limit is 1232 bytes (1280 - 40 IPv6 header - 8 UDP header). All frame size calculations based on 1280 must be corrected.

2. **T4 in handshake payload.** Not a wire field. Should not appear in the wire format diagram.

3. **Missing stream flags (FIN, RST).** Streams cannot be closed without lifecycle flags.

4. **Missing invite token signature.** No authentication of token origin without Ed25519 signature.

5. **BLAKE2s in Noise protocol name.** Should be BLAKE2b. See T2.

6. **Missing "saalis-dht-v1" in DomainSeparator enum.** If the relay_dht_key derivation uses this domain separator, it must be registered.

### Errata in the Surgeon's Stage 1

1. **"LEDBAT enforces the floor" (line 100).** LEDBAT has no floor. It enforces a dynamic ceiling based on congestion feedback. The revised text at line 132 is correct; line 100 should match.

2. **"Server" in SessionResumption (line 189).** Should be "host peer" or "responder."

3. **0-RTT forward secrecy claim (line 193).** Needs qualification. See Part I analysis.

### Errata in the Savant's Stage 1

1. **chunk_data size estimate (line 311).** 1200 bytes does not match the corrected frame budget. Should be approximately 1145 bytes.

2. **SmallVec in ProtocolMachine interface (line 264).** `SmallVec<[ProtocolOutput; 4]>` is a Rust implementation detail, not a protocol specification. The abstract contract should specify "zero or more output events" without naming a Rust collection type.

## Part VIII: Open Questions for Stage 2b

These are questions that I cannot resolve unilaterally. They require input from the project lead or from other experts in Stage 2b.

### Q1: Should the control log support CHANNEL_CREATE entries?

The Savant identifies channel lifecycle (B3) as a gap. Neither my Stage 1 nor the WireEng's control log entry types include a CHANNEL_CREATE entry. If channels are created via the control log (which seems correct: channel creation is a trust operation), then a new entry type is needed. But channel creation also requires generating a ChannelKey, distributing it via KeyBundle, and creating filter entries. The relationship between CHANNEL_CREATE (mesh concept) and KEY_GRANT (crypto distribution mechanism) needs clarification.

**Options:**
- A: CHANNEL_CREATE is a distinct entry type that precedes the first KEY_GRANT for that channel.
- B: The first KEY_GRANT for a channel implicitly creates the channel (no separate CHANNEL_CREATE).
- C: CHANNEL_CREATE is an application-layer concept that does not appear in the control log.

I lean toward option A for auditability: the control log should record when a channel was created, by whom, and with what access policy. But this requires design input.

### Q2: Should micro_epoch be LEB128 varint or fixed u16?

As noted in the consistency audit, the WireEng uses LEB128 for micro_epoch. I recommend fixed u16 for parser simplicity. But this changes the encrypted payload format from the WireEng's spec. Need agreement.

### Q3: What is the maximum number of concurrent seats per host peer?

My seat spec (specwriter-stage1, line 1121) includes TOO_MANY_SEATS as an error code, but I never specify the maximum. The WireEng also does not specify a limit. This is an operational concern, not a protocol concern, but a recommended default should be stated. Proposal: 16 concurrent seats. A home mesh serving a family needs at most 8-10 devices; 16 provides 2x headroom.

### Q4: How does the protocol distinguish between "peer went offline" and "peer is unreachable due to network partition"?

The heartbeat is the sole input to failure detection. Three missed heartbeats = peer declared dead. But this does not distinguish between a crashed peer and a partitioned network. The Savant's territory survey notes that PARTITION_ANN exists as a control log entry type, but the trigger for generating a PARTITION_ANN is unspecified. Does the peer generate a PARTITION_ANN about itself when it detects connectivity loss? That seems backwards (the peer cannot reach anyone to deliver the announcement). Does another peer generate a PARTITION_ANN about the missing peer? Then the partition announcement is really a "peer disappeared" announcement, not a partition detection mechanism.

This is an existing design question, not a Run 11 introduction. But it surfaced during cross-review.

### Q5: What happens to in-flight relay operations during CIK rotation?

The 48-hour dual acceptance window for CIK rotation (design docs) means that relay operations started under the old CIK must complete before the old CIK is discarded. But the relay operations wire format (WireEng section 7) does not include a CIK identifier. The relay uses relay_key_id (derived from channel_salt and key_id, not CIK). So CIK rotation should not affect relay operations. But it does affect the Noise sessions to the relay peer. If the relay peer's Noise session was established with old CIK, does it need renegotiation? Or does the session survive CIK rotation because the transport keys are independent of CIK?

The answer should be: transport keys are independent of CIK (they are derived from the Noise handshake ephemeral DH, not from CIK). But the Noise KK pattern uses CIK as the static key. If CIK rotates, the next KK handshake uses the new CIK, but existing sessions (established with old CIK) continue with their existing transport keys. This is correct behavior, but it should be stated explicitly in the spec. The 48-hour dual acceptance window means that during rotation, a peer accepts KK handshakes with either old or new CIK. After the window, only new CIK is accepted, and existing sessions established with old CIK remain valid until their natural expiry.

### Q6: Should the invite token include a minimum protocol version field?

The WireEng's invite token (section 5) includes a version byte for the token format but not a minimum protocol version. If a new device scans an invite token and supports only protocol version 1, but the mesh has upgraded to version 2 with breaking changes, the new device will successfully parse the token, attempt to connect, and fail at the Noise handshake because of version mismatch. Adding a `min_protocol_version: u16` field to the invite token would allow the new device to fail fast (before attempting a network connection) with a useful error message: "This mesh requires protocol version 2. Your software supports version 1. Please update."

This is a quality-of-life improvement, not a correctness issue. The failure mode without it (version mismatch at handshake time) is detectable but less user-friendly. Recommend: add `min_protocol_version: u16` to the invite token in the reserved bytes.
