# Integration: hilavitkutin + arvo into saalis

**Date:** 2026-03-14
**Status:** in progress
**Depends on:** `2026-03-10-final-architecture-decisions.md` (saalis), polka-dots design rounds (hilavitkutin/arvo)

---

## Context

saalis currently specifies its own scheduler, WorkUnit trait, DAG
dispatch, and numeric newtypes. The polka-dots project is extracting
two shared crates:

- **`hilavitkutin`** — `no_std + alloc` pipeline execution engine:
  DAG scheduling, WorkUnit dispatch, morsel-driven batching,
  work-stealing, progress/cancellation
- **`arvo`** — `no_std` numeric primitives: fixed-point types,
  float wrappers, semantic aliases. Bans raw primitives in
  data-facing positions.

Both are designed to be shared across polka-dots, saalis, and loimu.
This document maps integration points and design decisions needed.

## §1 WorkUnit trait alignment

### Current saalis (§5.2)

```rust
trait WorkUnit<
    F: ForTarget,
    W: WithInput = With<()>,
    R: ReturnsOutput = Returns<()>,
>: Send + Sync + 'static {
    type Subsystem: Subsystem;
    type Triggers: EventKindSet;
    type Emits: EventKindSet;

    const BATCH_TARGET: usize = 1;
    const BATCH_TIMEOUT: Duration = Duration::from_secs(60);
    const PRIORITY: Priority = Priority::Normal;
    const MAX_TRANSACTION_CHUNK: Option<usize> = None;
    const MAX_RETRIES: u32 = 5;

    fn execute(
        &self,
        ctx: &SchedulerContext,
        target: &F::Target,
        input: W::Input,
    ) -> Result<R::Output>;
}
```

### hilavitkutin's WorkUnit

```rust
trait WorkUnit {
    type Read: ColumnSlices;
    type Write: ColumnSlicesMut;
    fn process_batch(read: Self::Read, write: Self::Write) -> Result<()>;
}
```

### Key differences

| Aspect | saalis | hilavitkutin |
|--------|--------|--------------|
| Generic params | `For<T>`, `With<I>`, `Returns<O>` | None — `Read`/`Write` associated types |
| Scheduling | `type Triggers`/`Emits` (event DAG) | Read/Write column deps (data DAG) |
| Batching | `BATCH_TARGET`/`BATCH_TIMEOUT` consts | Morsel sizing from L1 cache + column count |
| Execution | `fn execute(&self, ctx, target, input)` | `fn process_batch(read, write)` (stateless) |
| Context | `SchedulerContext` passed in | No context — consumer provides `ColumnStorage` |
| Subsystem | `type Subsystem` grouping | No subsystem concept |

### Questions to resolve

**Q1: Do saalis work units map to hilavitkutin work units 1:1?**

saalis work units are higher-level: "catalogue Steam games", "enrich
from IGDB", "check connector health." These involve HTTP calls, DB
queries, external I/O. hilavitkutin work units are lower-level:
"transform column A → column B" in a tight inner loop with no I/O.

Options:
- (a) saalis wraps hilavitkutin at the scheduler level but keeps its
  own WorkUnit trait for I/O-bound connector work. hilavitkutin
  handles the data pipeline (query execution, batch transforms).
- (b) saalis's WorkUnit IS hilavitkutin's WorkUnit with domain
  extensions (subsystem, triggers/emits, retry). Connector work
  units are a special case with `type Read = ()`.
- (c) Two layers: hilavitkutin for the inner data pipeline,
  saalis's event-driven DAG for orchestrating connector work.
  Connector work units emit events that feed the data pipeline.

**Q2: How do Triggers/Emits (event DAG) relate to Read/Write (data DAG)?**

hilavitkutin builds its DAG from column read/write declarations.
saalis builds its DAG from event kind triggers/emits. These are
different dependency models:

- Data DAG: "I read column X, so I must run after whoever writes X"
- Event DAG: "I trigger on event Y, so I run when Y is emitted"

Are these complementary (both exist, different layers) or should
saalis's event DAG be expressed as column dependencies?

**Q3: What happens to `SchedulerContext`?**

saalis's `ctx.query()`, `ctx.write()`, `ctx.http()`, `ctx.notify()`,
`ctx.progress()` are all on the context. hilavitkutin has no context —
it takes column slices in and writes column slices out.

For I/O-bound work (HTTP calls, credential lookups), the context is
essential. For pure data transforms (aggregation, dedup, integrity
checks), column slices suffice.

**Q4: Morsel-driven execution for saalis?**

saalis's current design has fire-and-forget writes and scheduler
cadence floors. Morsel-driven execution (cache-fitting batches of
column data) makes sense for query execution (DataView projections)
and batch metadata operations. It doesn't make sense for HTTP-bound
connector work.

Does saalis use morsels for its data pipeline and a simpler dispatch
for connector orchestration?

## §2 Numeric types: arvo integration

### Current saalis newtypes

| saalis type | Wraps | arvo replacement |
|-------------|-------|------------------|
| `EntityId(i64)` | `i64` | `arvo::Count` or custom `arvo` newtype? |
| `Port(u16)` | `u16` | `arvo` semantic alias or stays as-is? |
| `RateLimit(NonZeroU32)` | `NonZeroU32` | `arvo` type or stays? |
| `As<Str>` column | `Cow<'static, str>` | Not numeric — stays |
| `As<i64>` column | `i64` | `arvo::IFixed<64,0>` or similar |
| `As<f64>` column | `f64` | `arvo::FastFloat` or `arvo::StrictFloat` |
| `As<bool>` column | `bool` | Not numeric — stays |

### Questions to resolve

**Q5: Which saalis types become arvo types?**

The principle from arvo: raw `f64`, `u64`, etc. are banned in
data-facing positions. But saalis's newtypes already wrap them.
The question is whether to wrap arvo types instead of raw primitives:

- `EntityId` wraps `i64` → wrap `arvo::IFixed<64,0>` instead?
  Or is EntityId an opaque ID, not a "numeric value"?
- Column value types (`As<i64>`, `As<f64>`) → `As<arvo::Count>`,
  `As<arvo::FastFloat>`?
- `Priority` discriminants are `u8` → stays as `repr(C)` enum?

**Q6: Does `Str` move to arvo or stay in saalis-sdk?**

arvo owns numeric primitives. `Str` is a text primitive. Options:
- (a) `Str` stays in `saalis-sdk` — arvo is for numbers only
- (b) `Str` moves to a shared `primitives` crate that arvo also
  depends on (or arvo expands scope to "all value primitives")
- (c) `Str` moves to arvo as a non-numeric value type

**Q7: 15-byte column stride compatibility?**

arvo types are designed for polka-dots' 128-bit (16-byte) stride
with 1 byte for discriminant/null flag = 15 bytes payload. saalis
uses SeaORM/SQLite, not columnar storage. Does the 15-byte
constraint apply to saalis? Or is it a polka-dots concern that
arvo types just happen to satisfy?

## §3 Crate dependency changes

### Current saalis dependency graph

```
saalis-primitives          (no_std, zero deps)
       ↑
  saalis-derive            (proc-macro)
       ↑
   saalis-sdk              (no_std + alloc)
     ↑     ↑
saalis-core  saalis-web
     ↑         |
     └─────────┘
```

### With hilavitkutin + arvo

```
arvo                       (no_std, zero deps beyond core)
  ↑
hilavitkutin               (no_std + alloc, depends on arvo)
  ↑
saalis-primitives          (no_std, depends on arvo for numeric types)
       ↑
  saalis-derive            (proc-macro)
       ↑
   saalis-sdk              (no_std + alloc, depends on hilavitkutin)
     ↑     ↑
saalis-core  saalis-web
     ↑         |
     └─────────┘
```

### Questions to resolve

**Q8: Does saalis-primitives depend on arvo?**

If `Priority` discriminants or other `repr(C)` enums use arvo
types, primitives needs arvo. But currently primitives is "zero
deps" — adding arvo changes that. Alternatively, primitives stays
zero-deps and arvo types enter at the SDK level.

**Q9: Does saalis-sdk depend on hilavitkutin directly?**

If saalis's WorkUnit trait IS hilavitkutin's (or wraps it), the
SDK needs the dependency. If hilavitkutin is host-internal (only
saalis-core uses it for scheduling), the SDK stays independent.

**Q10: Does hilavitkutin replace or wrap the scheduler in saalis-core?**

Options:
- (a) hilavitkutin IS the scheduler — saalis-core configures and
  runs it, provides `ColumnStorage` impl backed by SeaORM
- (b) saalis-core has its own orchestration layer (event DAG,
  subsystems, cadence timers) that uses hilavitkutin internally
  for data pipeline execution
- (c) saalis-core wraps hilavitkutin with a thin domain adapter

## §4 What stays, what changes

### Definitely stays (saalis-specific, no hilavitkutin equivalent)

- Event kinds as registered types (EventKindSet, triggers/emits)
- Subsystem grouping (Curator, Housekeeper, Doctor)
- Cadence timers (MIN_CADENCE floors)
- Context services (ctx.query, ctx.write, ctx.http, ctx.notify)
- Notification system (NotificationPayload)
- Settings system (SettingsValue, SetAccess, derive)
- Schema system (MetadataTable, Column, References)
- Theme system
- Session/auth system
- Error classification (ErrorKind, AppError)

### Definitely changes

- Raw numeric newtypes → arvo types where appropriate
- Column value types → arvo numeric types for numeric columns

### Needs discussion

- WorkUnit trait shape
- Scheduler implementation (event DAG vs data DAG vs both)
- Morsel-driven execution for data operations
- ColumnStorage integration with SeaORM-backed persistence
