# Crate Taxonomy Revision

**Date:** 2026-03-16
**Context:** Post-research, post-D1–D9 decisions, post-mockspace bootstrap

## Current state

The mockspace has 5 crates, decided before the research round:

```
saalis-primitives    (no_std, zero deps — repr(C) enums, ABI const)
       ↑
  saalis-derive      (proc-macro — #[register], #[derive(Settings)])
       ↑
   saalis-sdk        (no_std + alloc — trait contracts, ZSTs, Str, ColumnValue)
     ↑     ↑
saalis-core  saalis-web
     ↑         |
     └─────────┘
```

`saalis-core` is responsible for: scheduler (wrapping hilavitkutin),
`inventory` collection, rusqlite (bundled, FTS5), ColdStore hot↔cold
bridge, SIEVE hot store management, migration runner, plugin loader
(dlopen), notification dispatch, settings engine. Plus `main.rs`.

`saalis-web` is responsible for: axum HTTP shell, Maud templates,
HTMX, SSE, ResolvedUser extractor, DataView rendering, ActionFeedback,
ApiResult/ApiError.

## Questions to resolve

### Q1: Should saalis-core be split?

`saalis-core` is accumulating significant responsibility:

| Concern | Subsystem |
|---------|-----------|
| Scheduler | Wraps hilavitkutin, subsystem dispatch, event DAG |
| Persistence | ColdStore (rusqlite), SIEVE, epoch flush, migrations |
| Plugin loading | dlopen, descriptor collection, ABI validation |
| Settings | Resolution engine, 7-layer precedence, encryption |
| Notifications | Dispatch, logging macro integration, SSE bridge |
| File management | ManagedStorage, CAS blob store, reflink/hardlink |
| Search | FTS5 sync, fuzzy fallback |

Options:
- (a) Keep monolithic `saalis-core` — simpler dependency graph,
  single compilation unit, easier internal refactoring
- (b) Split into `saalis-scheduler`, `saalis-persistence`,
  `saalis-plugins` — cleaner responsibility boundaries, but more
  crates = more compile time, more dependency edges
- (c) Split only the largest concern — `saalis-persistence` for
  ColdStore + SIEVE + migrations, everything else stays in core

Considerations:
- polka-dots has 8 core crates (data, catalogue, project, pipeline,
  scheduler, audit, cli, plus 6 extensions). But polka-dots is a
  different domain with different access patterns.
- loimu has 34+ crates but it's a full UI framework with rendering,
  compositing, animation, etc.
- saalis at v1 has ~5-7 real concerns. Is splitting premature?
- The mockspace workflow means we can split later without disruption —
  the trait boundaries are in SDK, not in core's internal structure.

### Q2: Where does hilavitkutin sit in the dependency graph?

hilavitkutin is the shared pipeline execution engine. Options:

- (a) Core-internal dependency — only `saalis-core` depends on it.
  Plugins never see hilavitkutin types. Core wraps it behind SDK
  trait contracts (ColumnStorage → Storage trait family).

- (b) SDK dependency — `saalis-sdk` depends on hilavitkutin. Plugins
  can use hilavitkutin types directly (e.g., WorkUnit trait). But this
  couples the SDK to hilavitkutin's API evolution.

- (c) Between SDK and core — a new `saalis-engine` crate that wraps
  hilavitkutin with saalis-specific adapters (SIEVE, ColdStore bridge).
  SDK and engine are peers; core composes them.

Current decision (from integration topic): hilavitkutin is core-internal.
Saalis's `WorkUnit` trait in SDK is separate from hilavitkutin's
`WorkUnit`. Core bridges them. Does this still hold?

### Q3: Where does the Storage trait family live?

D9 defined a unified Storage trait family:

```
Storage (base) → FileStorage, ColumnarStorage, PersistentStorage, ManagedStorage
```

- `Storage`, `FileStorage`, `ManagedStorage` are in SDK (plugins call
  `ctx.fs()` which returns handles bounded by these traits)
- `ColumnarStorage` — this is hilavitkutin's trait. If hilavitkutin
  is core-internal, does this trait leak into SDK or stay in core?
- `PersistentStorage` — SQLite/rusqlite concern, definitely core-only

If some Storage traits are SDK and others are core, the "unified
family" is split across crate boundaries. Is that a problem?

### Q4: Where does arvo sit?

arvo provides numeric primitives. Options:

- (a) saalis-primitives depends on arvo — arvo types available at
  the lowest level. But primitives is "zero deps."
- (b) saalis-sdk depends on arvo — numeric types available to plugins.
  Primitives stays zero-deps.
- (c) arvo types only enter at core level — SDK uses plain i64/f64
  in ColumnValue, core converts to arvo for hilavitkutin.

The research concluded: saalis-primitives stays zero-deps, arvo
enters at SDK level. Still correct?

### Q5: First-party extension crates?

polka-dots has first-party extensions as separate crates:
- `polka-source-brew`, `polka-source-mas`, `polka-platform-macos`, etc.

Should saalis have:
- `saalis-connector-steam` — Steam Web API connector
- `saalis-connector-igdb` — IGDB/Twitch API connector
- `saalis-connector-screenscraper` — ScreenScraper connector
- `saalis-format-nointro` — No-Intro DAT verification

These would be cdylib crates depending only on `saalis-sdk`, loaded
via dlopen. This is the natural end-state of the plugin architecture.

But for v1, should they live in core (compiled in) or as separate
crates from the start?

Arguments for separate from day one:
- Proves the plugin architecture works
- Forces clean SDK boundaries
- Each connector is independently testable

Arguments for in-core initially:
- Less ceremony during initial development
- Can always extract later
- Plugin loading (dlopen) is complex to debug

### Q6: Should the theme system be its own crate?

The theme system (§16) has significant surface area:
- `Theme`, `ThemeProperty`, `Provides<P, G>`, `AsVariantOf<T>`
- `ThemeTranslator<F>`, `ThemeBuilder<F>`, `Fragment<F>`
- `Color`, `Length`, `FontFamily`, `FontAsset`, `FontStyle`
- IR value types, format markers, property resolver

This is all SDK-level (extensions define themes). But it's
conceptually separate from the entity/connector/scheduler domain.

Options:
- (a) Keep in `saalis-sdk` — it's all trait contracts
- (b) Extract `saalis-theme` — focused SDK crate for theme system
- (c) Keep traits in SDK, extract IR types to their own crate

### Q7: Does saalis-web need splitting?

`saalis-web` handles:
- Axum routing and middleware
- Maud template components
- HTMX fragment rendering
- SSE event stream
- DataView builder and rendering
- Action/ActionFeedback
- Auth middleware (ResolvedUser)
- Static asset serving

For a presentation-only crate, this is manageable. But if the
DataView system grows complex (gallery, table, board, list views
with progressive disclosure), it might warrant extraction.

For now: keep monolithic. Revisit if it grows past ~5k LOC.
