## §1 Plugin Architecture, Store, and the SDK Boundary

### 1.1 Plugin registration: `inventory` + `repr(C)` descriptors

Plugins register via `inventory::submit!` of `repr(C)` descriptors.
NOT `dyn Trait` objects (vtable layout is unstable across compilation
units). NOT explicit `extern "C" fn register()` symbol lookup —
`inventory` works across `dlopen` boundaries via platform constructor
sections (`__DATA,__mod_init_func` on macOS, `.init_array` on Linux).

The `#[register]` attribute macro, applied to a trait impl block,
generates:
1. A `repr(C)` descriptor struct with stable `extern "C"` function
   pointers that bridge to the developer's trait impl methods.
2. An `inventory::submit!` call for that descriptor.

```rust
#[register]
impl Connector for Steam { ... }

#[register]
impl Cataloguer for Steam { ... }
```

The macro inspects the trait name in the impl block, maps it to the
appropriate descriptor type (`ConnectorDescriptor`, `CataloguerDescriptor`,
etc.) via a fixed lookup table in the proc macro, and checks that the
implementing type satisfies `HasDescriptor<D>`. One macro, context-aware
from the impl it's applied to.

**`Named` trait and `Registrable` base trait.** Every type that
participates in the registration system implements `Named` (machine-
readable identifier) and `fmt::Display` (human-readable form). These
are collected into a single base supertrait:

```rust
/// Machine-readable identifier. Config keys, DB, descriptor lookup.
trait Named {
    const NAME: &'static str;
}

/// Base supertrait for all registrable types. One place to change
/// if bounds evolve — propagates to every registrable trait.
trait Registrable: Named + fmt::Display + Send + Sync + 'static {}

// Blanket impl — satisfy the bounds, you're Registrable.
impl<T: Named + fmt::Display + Send + Sync + 'static> Registrable for T {}
```

A type implements `Named` and `Display` once, and gains `Registrable`
automatically. All registrable traits (`Connector`, `EntityType`,
`SourceType`, `MetadataTable`, `Subsystem`, `SettingsDomain`, etc.)
use `Registrable` as their supertrait — never repeating the bounds.

**Descriptor registration gate: `HasDescriptor<D>`.** Sealed marker
trait that gates registration. When `#[register]` is applied to an
impl block, the macro checks `HasDescriptor<D>` at compile time —
if the type doesn't satisfy the right trait bounds for that
descriptor kind, the compiler (and LSP) immediately reports an error.

```rust
/// Sealed — never hand-implemented.
trait HasDescriptor<D: Descriptor>: Registrable {}
```

`HasDescriptor<D>` impls are generated by the **registry-creating
macro** — the macro applied to a trait definition that makes it
registrable. When a trait is declared as registrable, the macro
generates the descriptor type, the registry, and a blanket impl:

```rust
// Generated by the registry-creating macro on `trait Connector`:
impl<T: Connector> HasDescriptor<ConnectorDescriptor> for T {}
```

`#[register]` never writes `HasDescriptor` impls — it only checks
them. This is how it detects whether it's in the right place: if
applied to an impl of a non-registrable trait, no `HasDescriptor`
impl exists, and the compiler errors.

Extensions work the same way: an extension that defines a new
registrable trait uses the same registry-creating macro, which
generates its own `HasDescriptor<ExtensionDescriptor>` blanket
impl. `#[register]` then works on those extension trait impls
automatically.

This is distinct from `Registrable` (the identity supertrait).
`Registrable` means "has the common bounds for the registration
system." `HasDescriptor<D>` means "can produce a specific
descriptor of type `D`."

**`SdkVersion` type.** Proper type for ABI version, not a raw `u32`:

```rust
struct SdkVersion {
    major: u16,
    minor: u16,
}

impl SdkVersion {
    const CURRENT: Self = Self { major: 1, minor: 0 };

    fn is_compatible_with(&self, host: &SdkVersion) -> bool {
        self.major == host.major && self.minor <= host.minor
    }
}

impl From<u32> for SdkVersion { ... }     // packed repr(C) format
impl From<(u16, u16)> for SdkVersion { ... }
```

Different sources produce `SdkVersion` via `Into<SdkVersion>` —
the packed `u32` from `repr(C)` descriptors, tuple literals in Rust
code, etc. The type owns the compatibility logic.

**`Descriptor` trait.** Common contract for all `repr(C)` descriptors:

```rust
trait Descriptor: 'static + Send + Sync {
    fn name(&self) -> &'static str;
    fn sdk_version(&self) -> SdkVersion;
}
```

Every descriptor, regardless of domain, has a name and an SDK version.
The `Descriptor` trait provides Rust-side typed access to `repr(C)`
struct fields — the data lives in the concrete struct, the trait is
the interface. The descriptor's `name` field is a const pointer to
`Named::NAME` — not a copy, the same `&'static str` pointing to the
same `.rodata` bytes. `Named::NAME` is the single source of truth;
the descriptor just holds a reference to it. Similarly,
`sdk_version` points to `SdkVersion::CURRENT`.

**`Registry` trait.** Common contract for all typed registries, with
default implementations that use `inventory` via the `Descriptor`
trait:

```rust
trait Registry {
    type Descriptor: Descriptor;

    fn iter(&self) -> impl Iterator<Item = &'static Self::Descriptor> {
        inventory::iter::<Self::Descriptor>()
    }

    fn get(&self, name: &str) -> Option<&'static Self::Descriptor> {
        self.iter().find(|d| d.name() == name)
    }

    fn len(&self) -> usize {
        self.iter().count()
    }

    /// Non-overridable. Runs base checks (version compat, duplicate
    /// names) then calls `validate_domain()` for domain-specific
    /// checks. Domain registries never need to remember to check
    /// versions — it's always done.
    fn validate(&self) -> Result<()> {
        let current = SdkVersion::CURRENT;
        let mut seen = HashSet::new();
        for desc in self.iter() {
            if !desc.sdk_version().is_compatible_with(&current) {
                bail!("{}: incompatible SDK version", desc.name());
            }
            if !seen.insert(desc.name()) {
                bail!("{}: duplicate registration", desc.name());
            }
        }
        self.validate_domain()
    }

    /// Domain-specific validation hook. Override this, not
    /// `validate()`. Default: no additional checks.
    fn validate_domain(&self) -> Result<()> { Ok(()) }
}
```

Version checking and duplicate detection are non-overridable — baked
into `validate()`. Domain registries override `validate_domain()` to
add their own checks: `WorkUnitRegistry` adds DAG cycle detection,
`ConnectorRegistry` checks capability coherence, etc. They can't
accidentally skip the base checks.

Domain registries implement `Registry` plus domain-specific queries:

- **`ConnectorRegistry`** — filter by capability (all cataloguers,
  all enrichers), capability matrix queries
- **`WorkUnitRegistry`** — DAG construction from `Triggers`/`Emits`
  associated types, cycle detection, topological ordering
- **`SubsystemRegistry`** — initialization ordering, cadence timer
  setup

**This is the universal registration pattern.** Everything registered
via `inventory` — connectors, enrichers, work units, subsystems — uses
`#[register]` on the impl block, `HasDescriptor<D>` for compile-time
validation, `Descriptor` for common metadata, and `Registry` for
typed discovery with built-in validation. First-party and plugin code
use the same path. No special cases.

**Constraints:**
- The SDK crate containing `inventory::collect!` must be dynamically
  linked (shared) between host and plugins, not statically duplicated.
- No `dlclose` — plugins are load-only. `inventory` stores `&'static`
  references that would dangle on unload.
- Plugin dylibs are compiled as `cdylib`.

**What was rejected:**
- Dual registration path (`inventory` + `extern "C" fn register()`) —
  solving a problem that doesn't exist. `inventory` already handles
  dylib registration.
- `dyn Trait` objects in `inventory` — vtable layout instability
  across compilation units.
- Subprocess/JSON-RPC for plugins — unnecessary complexity for the
  current use case. Can be revisited if non-Rust plugin support is
  needed later.

### 1.2 Persistence: Descriptor-based schema, SeaORM host-only

SeaORM is the persistence layer — but **only on the host side**.
Plugins never see SeaORM types. The entire persistence API is
expressed through the same trait contract + `repr(C)` descriptor +
`#[register]` pattern used for connectors, work units, and subsystems.

**Store trait, two-type pattern: Rejected.** `Store`, `Migrate`,
`FromRow`, `Backend` as independent traits are not used. The
convergence's "plugins need domain types in a `no_std` SDK crate"
argument was invalid — plugins don't hold model structs. But instead
of exposing SeaORM entities directly to plugins, the schema is
expressed entirely through trait contracts.

#### 1.2.1 Schema as traits: MetadataTable, Column, References

Tables, columns, and relationships are all trait contracts in the SDK.
Concrete types are ZSTs (zero-sized types) — pure compile-time markers
that carry no data. The `#[register]` macro generates `repr(C)`
descriptors. The host discovers descriptors via registries and creates
actual SeaORM entities / DB tables.

**MetadataTable** — pure marker trait:

```rust
trait MetadataTable: Send + Sync + 'static {}
```

No methods. A ZST that implements `MetadataTable` declares "I am a
metadata table." The host reads the `TableDescriptor` generated by
`#[register]` for any runtime information (table name, etc.).

**Column** — typed column association with sealed bounds:

```rust
// Semantic wrapper ZSTs:
struct In<M: MetadataTable>(PhantomData<M>);
struct As<V: ValueType>(PhantomData<V>);

// Sealed marker traits — only In<M> and As<V> satisfy them:
mod sealed {
    pub trait TableRef {}
    pub trait TypeRef {}
}
impl<M: MetadataTable> sealed::TableRef for In<M> {}
impl<V: ValueType> sealed::TypeRef for As<V> {}

// Column trait — registrable, strictly constrained:
trait Column<I: TableRef, A: TypeRef>: Send + Sync + 'static {}
```

A column declaration reads as natural language:

```rust
struct Score;
struct Source;

#[register]
impl Column<In<Rating>, As<i32>> for Score {}

#[register]
impl Column<In<Rating>, As<Str>> for Source {}
```

"Score is a Column In Rating As i32." One `#[register]` per impl.
The macro generates `ColumnDescriptor` + `inventory::submit!`.

**ZST column reuse across tables.** Because ZSTs carry no data,
the same column type can appear in multiple tables:

```rust
#[register]
impl Column<In<Rating>, As<Str>> for Source {}

#[register]
impl Column<In<Description>, As<Str>> for Source {}

#[register]
impl Column<In<Media>, As<Str>> for Source {}
```

The `Source` concept is one struct. Each `Column` impl is a separate
fact registered independently. The host sees all of them.

**References** — FK relationships as trait contracts:

```rust
trait References<T>: MetadataTable {
    type FkColumn: Column<In<Self>, As<i64>>;
}

#[register]
impl References<ContentInstance> for Rating {
    type FkColumn = EntityId;
}
```

The host reads `ReferenceDescriptor`s to build the FK graph. The
scheduler uses this graph for write ordering — no manual dependency
declarations needed.

#### 1.2.2 Implicit plumbing columns

Every metadata table automatically gets host-managed columns that
plugins never declare and never set:

| Column | Type | Purpose |
|--------|------|---------|
| `id` | integer | Primary key |
| `entity_id` | integer | FK to `entities` |
| `source` | text | Connector that produced this row (auto-stamped) |
| `created_at` | datetime | Row creation timestamp |
| `updated_at` | datetime | Last modification timestamp |

The `source` column is auto-stamped from the `ConnectorContext<C>`
type parameter — the plugin literally cannot specify a different
source. Provenance is guaranteed by the type system.

Plugin-declared columns (via `Column<In<M>, As<V>>`) are only the
domain-specific fields. The host adds plumbing when creating the
actual DB table.

#### 1.2.3 Query API: typed, descriptor-backed

Plugins query through `ctx.query()` using ZST types as compile-time
constraints. The host resolves descriptors and translates to SeaORM's
query builder internally.

```rust
let results = ctx.query::<Rating>()
    .filter::<Score>(Op::Gt, 80.into())
    .execute()?;
```

`Score: Column<In<Rating>, _>` is checked at compile time. The host
maps `Score` → `ColumnDescriptor` → SeaORM column → actual SQL.
No SeaORM types cross the plugin boundary.

The `SupportsEntity` / `HasMetadata` pattern (§4.5) gates which
metadata an entity type can query:

```rust
ctx.query::<Rating>()              // Rating: MetadataTable ✓
    .of::<Game>(id)       // Rating: SupportsEntity<Game> ✓
    .execute()?;

ctx.query::<CompatInfo>()
    .of::<Movie>(id)      // CompatInfo: SupportsEntity<Movie> ✗
    // compile error
```

#### 1.2.4 Write API: fire-and-forget work units

`ctx.write()` does not execute immediately. It drops a work unit into
the scheduler. The scheduler batches writes by table, orders them
by FK dependency graph (derived from `References` trait impls), and
flushes when optimal.

```rust
fn enrich(&self, ctx: &ConnectorContext<Self>, entity_id: i64) -> Result<()> {
    let data = ctx.http::<Get>(igdb_url)?;

    ctx.write::<Rating>(entity_id, |row| {
        row.set::<Score>(85)
    })?;  // Ok = "accepted into scheduler", not "written to DB"

    ctx.write::<Description>(entity_id, |row| {
        row.set::<Text>(data.summary.clone())
    })?;

    Ok(())
}
```

The source column is stamped automatically from `ConnectorContext<Self>`
— the plugin's connector type. No source parameter, no impersonation.

Write ordering is automatic:
1. Scheduler collects all pending writes
2. Groups by table
3. Reads FK graph from `References` descriptors
4. Flushes in dependency order (entities before ratings)
5. Handles retries, batching, health-gating per §5

#### 1.2.5 Table ownership and cross-plugin sharing

Three levels of table ownership:

1. **First-party tables** (Rating, Media, Description, etc.) — ZST
   markers defined in `saalis-sdk`. All plugins can write to these.
   These cover the common metadata types from §4.4.

2. **Plugin-private tables** — a plugin defines its own `MetadataTable`
   ZST + `Column` impls. Only that plugin writes to it. Other plugins
   cannot reference the ZST (it's in the plugin's crate).

3. **Cross-plugin shared tables** — a plugin author publishes a
   lightweight schema crate containing just ZSTs + Column impls (no
   logic, no dependencies beyond `saalis-sdk`). Other plugin authors
   depend on the schema crate to write to those tables. This is the
   standard Rust "types crate" pattern (`http`, `tower-service`, etc.).

Level 3 exists automatically by necessity — the only way to write to
a table is to know its ZST statically. No additional framework support
is needed. The first-party tables (level 1) cover the vast majority
of use cases; level 3 is an emergent capability, not a designed-for
feature.

#### 1.2.6 Migrations: type-level ops, auto-additive, safe destructive

Migrations follow the same registrable pattern. `Migration<const V: u32>`
is a trait with a const generic version, registered via `#[register]`.

**Additive changes are automatic.** When the host starts, it diffs
registered `Column<In<M>, As<V>>` impls against actual DB columns.
Missing columns are auto-created. No migration code needed for the
common case of adding new fields.

**Destructive/transformative changes require `Migration<V>`** with a
closed set of type-level ops — no raw SQL, no escape hatches. Every op
is a known primitive with guaranteed bidirectional behavior:

```rust
trait Migration<const V: u32>: MetadataTable + Send + Sync + 'static {
    type Ops: MigrationOpSet;  // tuple-based type-level set, same pattern as EventKindSet
}
```

##### Type-level migration ops

Ops use the same ZST markers as the schema. No strings — the
`#[register]` macro resolves ZSTs to const data at compile time
via `MetadataTableMeta::TABLE_NAME` and `ColumnMeta::COLUMN_NAME`
associated consts.

**Structural ops (schema shape):**

| Op type | Reversal | Data capture |
|---------|----------|:---:|
| `CreateTable<T>` | `DropTable<T>` | no |
| `DropTable<T>` | `CreateTable<T>` + re-insert | yes — captures all rows |
| `CreateColumn<C, T>` | `DropColumn<C, T>` | no |
| `DropColumn<C, T>` | `CreateColumn<C, T>` + restore | yes — captures column values |
| `RenameColumn<C, T, NewName>` | `RenameColumn<NewName, T, C>` | no |
| `RenameTable<T, NewName>` | `RenameTable<NewName, T>` | no |
| `ModifyType<C, T, FromType, ToType>` | swap from/to | no |
| `CreateIndex<Cols, T>` | `DropIndex` | no |
| `DropIndex<Name, T>` | `CreateIndex` | no |

**Data ops (row-level):**

| Op type | Reversal | Data capture |
|---------|----------|:---:|
| `CopyRows<Cols, From<S>, To<D>>` | no-op (non-destructive) | no |
| `DeleteRows<T, Filter>` | re-insert captured rows | yes — captures matching rows |

All destructive ops (Drop*, DeleteRows) auto-capture affected data
before executing. The host stores captured data in a
`migration_data_reverts` table keyed by migration version. Rollback
for destructive ops replays the captured data. Rollback for
non-destructive ops is structural reversal only.

**Move = Copy + Drop.** Not a separate primitive. The developer
composes:

```rust
CopyRows<(Score, MaxScore), From<Scoring>, To<Rating>>,
DropTable<Scoring>,  // auto-captures remaining rows
```

##### Compile-time snapshotting

The `#[register]` macro on a Migration impl resolves all ZST types
to concrete data at compile time via associated consts:

```rust
// #[register] generates:
inventory::submit!(MigrationDescriptor {
    version: 1,
    table: <Scoring as MetadataTableMeta>::TABLE_NAME,
    ops: &[
        OpDescriptor::CreateTable {
            table: <Rating as MetadataTableMeta>::TABLE_NAME,
        },
        OpDescriptor::CopyRows {
            columns: &[<Score as ColumnMeta>::COLUMN_NAME],
            from: <Scoring as MetadataTableMeta>::TABLE_NAME,
            to: <Rating as MetadataTableMeta>::TABLE_NAME,
        },
        OpDescriptor::DropTable {
            table: <Scoring as MetadataTableMeta>::TABLE_NAME,
        },
    ],
});
```

The descriptor is self-contained. After compilation, the runtime only
uses descriptors — trait impls are compile-time scaffolding.

##### Legacy ZSTs

When a table or column is dropped via migration, its ZST struct
persists in the codebase (needed by the Migration's `type Ops` tuple).
These are organized in a `migrations/legacy.rs` module. They:

- Have no runtime cost (zero-sized, compiler erases unused code)
- Are only referenced by migration type signatures
- Provide compile-time safety: accidentally deleting a ZST that a
  migration references causes a compile error (vs. string-based
  migrations that fail at runtime)

```
saalis-sdk/src/
  schema/
    tables.rs          ← active: Rating, Media, Description...
    columns.rs         ← active: Score, Text, Url...
  migrations/
    legacy.rs          ← dropped ZSTs: Scoring, OldScore...
    v1.rs              ← Migration<1> impls
    v2.rs              ← Migration<2> impls
```

##### Example: split table

```rust
// v1: Scoring table exists with Score, MaxScore, StatValue columns
// v2: Split into Rating (Score, MaxScore) + Stats (StatValue)

#[register]
impl Migration<1> for Scoring {
    type Ops = (
        CreateTable<Rating>,
        CreateTable<Stats>,
        CopyRows<(Score, MaxScore), From<Scoring>, To<Rating>>,
        CopyRows<(StatValue,), From<Scoring>, To<Stats>>,
        DropTable<Scoring>,
    );
}
// Rollback auto-derived:
// 1. CreateTable<Scoring> + re-insert captured rows
// 2. DropTable<Stats>
// 3. DropTable<Rating>
// (reverse order of ops, each reversed individually)
```

##### Safety guarantees

1. **Pre-migration snapshot:** host always creates `.saalis` snapshot
   before any migration run (nuclear fallback)
2. **Closed op set:** no raw SQL. Every op is a known primitive with
   guaranteed bidirectional behavior. The host is in full control.
3. **Auto-capture:** all destructive ops capture affected data before
   executing, stored in `migration_data_reverts` table
4. **Compile-time validation:** migration ops reference ZSTs with trait
   bounds — invalid ops (e.g., copying a column that doesn't exist in
   the source table) are compile errors
5. **Type-level snapshotting:** `#[register]` bakes descriptor data at
   compile time. Legacy ZSTs can be removed from active code without
   affecting migration execution.

**Host startup migration flow:**

1. Create `.saalis` snapshot
2. Discover all `MigrationDescriptor`s via registry
3. Group by table (from `MetadataTable` supertrait on `Migration`)
4. Check per-table version in migration tracking table
5. Run pending migrations in `V` order:
   - For each op: if destructive → capture data first
   - Execute op (translate to `sea-query` SQL)
   - Store capture reference in `migration_data_reverts`
6. After explicit migrations: diff registered columns vs DB, auto-create
   any missing (additive changes from new `Column` registrations)

Plugin migrations use the same path. A plugin that adds `Migration<1>`
for its own table gets discovered and executed alongside first-party
migrations.

#### 1.2.7 What was rejected

- **Two-type pattern** (domain structs in SDK + entity models in core
  + `From` conversion) — serving a crate boundary that doesn't need
  to exist.
- **Store, Query, Backend, Executor, Migrate, FromRow** wrapper traits
  — SeaORM subsumes all of these on the host side.
- **SeaORM types in the SDK** — plugins don't need SeaORM. Trait
  contracts + descriptors express the full schema.
- **String-based table/column references** — replaced by ZST markers
  with compile-time checking.
- **ContentRepository trait** (fixed method API like `get_metadata`,
  `list_by_type`) — replaced by the typed query builder that reuses
  the descriptor registry.
- **Buffered-flush write model** — replaced by fire-and-forget work
  units into the scheduler. The scheduler already handles batching,
  ordering, and retry.
- **Manual `DependsOn<T>` trait for write ordering** — unnecessary.
  FK graph from `References` trait impls provides the same information
  automatically.
- **Per-plugin tables for shared metadata** — fragments queries.
  Shared first-party tables with source-tagging are more efficient.
- **Raw SQL in migrations** — replaced by closed set of type-level
  ops with guaranteed bidirectional behavior. No escape hatches.
- **String-based migration ops** — replaced by ZST type parameters
  with compile-time snapshotting into descriptors.
- **Separate schema and data migration stages** — unified into one
  `Migration<V>` with one ordered tuple of ops. Schema and data ops
  are interleaved when needed (e.g., create table → copy data →
  drop old table).

### 1.3 What plugins see vs what the host manages

**Plugins define:** Trait implementations (`Connector`,
`Cataloguer<E>`, `Enricher<E>`, `Sourcer<E, S>`).
Capability traits are generic over entity type (and source type for
`Sourcer`) — each impl is registered separately per entity type
(§1.5). The `#[register]` attribute macro on each impl generates
`repr(C)` descriptors and registers them via `inventory`.

**Plugins also define:** Schema declarations — `MetadataTable`,
`Column<In<M>, As<V>>`, `References<T>`, `HasMetadata<M>` /
`SupportsEntity<E>` impls (§4.5).
These are ZST markers with `#[register]`, following the same pattern
as behavioral trait impls. First-party tables are pre-defined in the
SDK; plugins can add their own.

**Plugins receive (via method parameters):** Context objects that the
host constructs and passes into trait methods. The context is generic
over the connector type: `ConnectorContext<C: Connector>`.

Within a trait impl method body, the plugin can call:

- `ctx.query::<M>()` — typed query builder (§1.2.3)
- `ctx.write::<M>(id, |row| ...)` — fire-and-forget write (§1.2.4)
- `ctx.http::<M>(&str)` — typed HTTP request; `M` is an HTTP method
  ZST (`Get`, `Post`, etc.). `HttpClient` trait defines the contract,
  host provides the impl (rate-limited, retried)
- `ctx.fs::<L>()` — `LocationHandle<L>` (bound-gated filesystem
  ops over `Storage` trait §14)
- `ctx.progress(completed, total)` — fire-and-forget progress update;
  drops a work unit for the scheduler, never blocks
- `ctx.notify(payload)` — emit typed notification (§3); payload is a
  `NotificationPayload` struct, urgency baked into the type
- `ctx.settings()` — `&C::Settings` (strongly-typed, pre-resolved
  from 7-layer precedence §10.9; credentials included as `Credential`
  fields, not a separate system)

All context services follow the same pattern: the SDK defines trait
contracts (`HttpClient`, `Storage`, `NotificationPayload`, etc.),
and the host provides implementations. Context methods return
trait-bounded types. How the ABI bridge delivers host impls to
plugins is an implementation detail of the `#[register]` macro
system, not a design concern. `LocationHandle<L>` (§14) is an
example of ergonomic bound-gating on top of the `Storage` trait.

**`HttpClient` trait — typed HTTP entry points:**

```rust
/// HTTP method ZSTs — turbofish selects the method.
struct Get;
struct Post;
struct Put;
struct Delete;
struct Patch;
struct Head;

trait HttpMethod {}
impl HttpMethod for Get {}
impl HttpMethod for Post {}
impl HttpMethod for Put {}
impl HttpMethod for Delete {}
impl HttpMethod for Patch {}
impl HttpMethod for Head {}

/// SDK trait contract — host provides the impl (rate-limited,
/// retried, with connector-scoped base URL if applicable).
trait HttpClient {
    fn request<M: HttpMethod>(&self, url: &str) -> Result<Response>;
}
```

Usage from a plugin:

```rust
let data = ctx.http::<Get>("https://api.igdb.com/v4/games")?;
let result = ctx.http::<Post>("https://api.example.com/submit")?;
```

The turbofish on `http` selects the HTTP method at the call site,
consistent with the turbofish patterns elsewhere (`ctx.query::<M>()`,
`ctx.fs::<L>()`). The host's `HttpClient` impl handles rate
limiting, retries, and timeout enforcement internally.

Query and write operations use ZST metadata types as compile-time
constraints. The host resolves them through descriptor registries
and translates to SeaORM internally. Writes are auto-stamped with
the connector's identity from the `ConnectorContext<C>` type parameter.

**Plugins do NOT:**
- See SeaORM types (host-only, behind the descriptor abstraction)
- Execute writes directly (writes are work units for the scheduler)
- Specify their own source identity (auto-stamped from context)
- Access the scheduler directly (host dispatches work)

**The host manages:** SeaORM entities (generated from descriptors),
scheduler, database connections, write ordering (from FK graph),
configuration, service lifecycle. The host constructs
`ConnectorContext<C>`, calls the plugin's trait methods through
the descriptor's function pointers, and processes the results.

### 1.5 Connector and capability trait contracts

#### 1.5.1 Connector — identity + connection validation

```rust
trait Connector: Registrable {
    type Settings: Settings;

    /// Check prerequisites: local directories exist, API reachable,
    /// credentials valid. Ensures the connector's environment is ready.
    /// NOT ongoing health monitoring (that's §5.7 health work units).
    fn check_prerequisites(&self, ctx: &ConnectorContext<Self>) -> Result<()>;
}
```

Identity comes from `Registrable` (`Named::NAME` for machine ID,
`Display` for human-readable). `type Settings` associates the
connector with its strongly-typed settings struct (§10). The host
resolves all settings layers into the concrete struct at context
construction; `ctx.settings()` returns `&C::Settings`.

`check_prerequisites` is connection lifecycle (startup validation —
directories exist, API reachable, credentials valid), distinct from
ongoing health monitoring (§5.7 — regular work units that write
health metadata). Capabilities are separate traits registered
independently.

#### 1.5.2 Cataloguer — "this content exists"

```rust
trait Cataloguer<E: EntityType>: Connector {
    fn catalogue(
        &self,
        ctx: &ConnectorContext<Self>,
    ) -> Result<()>;
}
```

Generic over entity type. Each impl is registered separately:

```rust
#[register]
impl Cataloguer<Game> for Steam {}

#[register]
impl Cataloguer<Game> for Igdb {}
#[register]
impl Cataloguer<Movie> for Igdb {}
```

Discovery is a write — the cataloguer writes identity-level data
through the context API, not by returning a collection:

```rust
fn catalogue(&self, ctx: &ConnectorContext<Self>) -> Result<()> {
    for item in api_response {
        ctx.write::<Entities>(|row| {
            row.set::<ExternalId<Self>>(item.id);
            row.set::<Title>(item.title);
        });
    }
    Ok(())
}
```

The host deduplicates, creates entities, emits `EntityDiscovered`
events. Cataloguers do NOT write metadata — if a connector is both
`Cataloguer` and `Enricher` (like Steam), the host dispatches
enrichment separately after cataloguing.

#### 1.5.3 Enricher — "here's context about it"

```rust
trait Enricher<E: EntityType>: Connector {
    fn enrich(
        &self,
        ctx: &ConnectorContext<Self>,
    ) -> Result<()>;
}
```

No `enriches()` declaration method. The enricher queries what needs
work and writes results through the context API:

```rust
fn enrich(&self, ctx: &ConnectorContext<Self>) -> Result<()> {
    let entities = ctx.query::<Titles>().filter::<NeedsEnrichment>()?;
    for entity in entities {
        let data = ctx.http::<Get>(api_url)?;
        ctx.write::<Rating>(entity.id, |row| {
            row.set::<Score>(data.score);
        });
    }
    Ok(())
}
```

`SupportsEntity` bounds (§4.5) enforce at compile time what it can
write. The Curator's `EnrichWork` handles routing — which enrichers get
dispatched for which content is work unit logic, not trait API.
`NeedsEnrichment` is a registered filter ZST — composable, extensible,
type-safe. The work unit returns `Result<()>` — the host knows what
was written (it processes the fire-and-forget work units) and derives
counts and error tracking from the scheduler's batch execution.

#### 1.5.4 Sourcer — "get it here"

```rust
trait Sourcer<E: EntityType, S: SourceType>: Connector {
    fn resolve(
        &self,
        ctx: &ConnectorContext<Self>,
    ) -> Result<()>;
}
```

Generic over both entity type AND source type. Sourcing is writing
source metadata through the context API:

```rust
fn resolve(&self, ctx: &ConnectorContext<Self>) -> Result<()> {
    let entities = ctx.query::<Sources>().filter::<NoUrl>()?;
    for entity in entities {
        let url = /* look up download URL */;
        ctx.write::<Source>(entity.id, |row| {
            row.set::<Url>(url);
            row.set::<SizeBytes>(size);
        });
    }
    Ok(())
}
```

Registration is per entity type + source type:

```rust
#[register]
impl Sourcer<Game, DirectDownload> for Myrient {}

#[register]
impl Sourcer<Game, SteamApp> for Steam {}

#[register]
impl Sourcer<Game, LocalFile> for FilesystemScanner {}
```

The host collects `SourcerDescriptor`s and routes by entity type +
source type. The Download Engine (§12) dispatches by source type
descriptor.

#### 1.5.5 SourceType — extensible via `#[register]`

```rust
trait SourceType: Registrable {
    /// The URL shape this source type produces.
    type Url: SourceUrl;
}

trait SourceUrl: AsRef<str> + fmt::Display + Send + Sync + 'static {}
```

First-party source types in the SDK:

```rust
struct DirectDownload;

impl Named for DirectDownload { const NAME: &'static str = "direct_download"; }
impl fmt::Display for DirectDownload { ... } // "Direct Download"

#[register]
impl SourceType for DirectDownload { type Url = HttpUrl; }

struct LocalFile;
// Named + Display impls...
#[register]
impl SourceType for LocalFile { type Url = LocalPath; }
```

Plugins extend with their own:

```rust
struct SteamApp;
impl Named for SteamApp { const NAME: &'static str = "steam_app"; }
impl fmt::Display for SteamApp { ... } // "Steam"

#[register]
impl SourceType for SteamApp {
    type Url = SteamUri; // steam://install/{app_id}
}
```

NOT a `repr(C)` enum — plugins must be able to define new source
types. Same trait + ZST + `#[register]` + descriptor pattern as
everything else.

#### 1.5.6 Data transfer types

Pipeline data types (`Discovered<E, C>`, `EnrichCandidate<E>`,
`EnrichSummary`, `SourceCandidate<E>`, `ResolvedSource<E, S>`) are
eliminated — these were concrete structs shuttling data between
pipeline stages. The context API (`ctx.query()`, `ctx.write()`)
replaces them (see §19.1). What remains are newtypes and value types:

**`EntityId` — opaque entity identifier:**

```rust
#[repr(C)]
#[derive(Copy, Clone, Eq, PartialEq, Hash)]
struct EntityId(i64);
```

Newtype, not trait. The ID strategy is a host decision (SQLite rowid).
Plugins pass it around opaquely. If the backing type changes, the
newtype internals change — API stays the same.

**URL types:**

| Type | Wraps | Used by |
|------|-------|---------|
| `HttpUrl` | `Cow<'static, str>` (validated URL) | `DirectDownload` |
| `LocalPath` | `PathBuf` | `LocalFile`, `LocalDirectory` |
| `SteamUri` | `u64` (app ID) | `SteamApp` (plugin-defined) |
| `MagnetLink` | `Cow<'static, str>` | `Torrent` (plugin-defined) |

#### 1.5.7 ExternalId — typed cross-source identity

```rust
struct ExternalId<C: Connector> {
    id: Cow<'static, str>,
    _connector: PhantomData<C>,
}
```

Compile-time guarantee that an external ID is tied to a specific
connector. No stringly-typed source name.

External IDs are metadata columns queried through the standard query
API — no standalone `ExternalIdSet` container needed:

```rust
let steam_id = ctx.query::<ExternalIds>()
    .of::<Game>(entity_id)
    .get::<Steam>()?;
```

`ExternalId<C>` the newtype stays — it's the typed value for a
single ID. Multi-connector access is handled by the query API,
not a runtime container.

#### 1.5.8 Pipeline flow through the scheduler

The connector pipeline is expressed entirely as scheduler events
(§5.1) and work units (§5.2), not as a method chain:

```
Sweep event
  → CatalogueWork dispatched per Cataloguer<E>
    → catalogue() writes entities via ctx.write()
    → returns Result<()>
    → host deduplicates from written data
    → emits EntityDiscovered events

EntityDiscovered event
  → EnrichWork dispatched per Enricher<E> matching the entity type
    → enrich() queries via ctx.query(), writes via ctx.write()
    → returns Result<()>
    → emits EntityEnriched events

EntityEnriched event
  → SourceWork dispatched per Sourcer<E, S> matching the entity type
    → resolve() queries via ctx.query(), writes via ctx.write()
    → returns Result<()>
    → emits SourceResolved events

SourceResolved event
  → Download Engine queues downloads (§12)
```

Each stage is a work unit in the DAG. The Curator subsystem (§5.6)
owns `CatalogueWork`, `EnrichWork`, `SourceWork`. Routing is
type-driven at the descriptor level — when a `Game` is discovered,
the scheduler queries all `EnricherDescriptor`s registered for
entity type `Game`. All data flows through context writes — no
pipeline structs, no return collections.

#### 1.5.9 What was rejected

- **`async` trait methods** — superseded by sync-core (§5.2).
  HTTP is accessed via `ctx.http` which the host provides as a
  sync-presenting interface.
- **`dyn Connector` at registry boundary** — superseded by
  `#[register]` + `repr(C)` descriptors (§1.1).
- **`as_cataloguer()` / `as_enricher()` downcasts on Connector** —
  unnecessary. Capabilities are separate traits with separate
  descriptors. The host queries the appropriate registry.
- **`enriches()` declaration method** — unnecessary. `SupportsEntity`
  bounds (§4.5) enforce write permissions at compile time. Runtime
  routing is the scheduler's job.
- **`SourceType` as `repr(C)` enum** — closed set, plugins can't
  extend. Same trait + ZST + `#[register]` pattern as everything else.
- **`url: String` on resolved sources** — stringly-typed URL is a
  parsing swamp. `SourceType::Url` associated type makes URL shape
  a compile-time contract per source type.
- **`entity_id: i64` raw** — replaced by `EntityId` newtype.
  Opaque to plugins, host controls the ID strategy.
- **`ExternalId.source: &'static str`** — stringly-typed connector
  reference. Replaced by `ExternalId<C: Connector>` with phantom
  generic. Multi-connector access via the query API (§1.5.7).
- **`Discovered<E, C>`, `EnrichCandidate<E>`, `EnrichSummary`,
  `SourceCandidate<E>`, `ResolvedSource<E, S>`** — pipeline data
  structs replaced by context API writes (§19.1).
- **`ExternalIdSet`** — runtime container replaced by the query API
  (§19.2).

## §2 Priority Levels

### 2.1 Decision: 5 levels, `repr(C)`, explicit discriminants

```rust
#[repr(C)]
pub enum Priority {
    Immediate  = 0,
    High       = 1,
    Normal     = 2,
    Low        = 3,
    Background = 4,
}
```

Lower discriminant = higher priority, enabling direct comparison
(`if a < b { a is higher priority }`).

**Rationale:** The `repr(C)` ABI argument is decisive. This enum
crosses the plugin boundary via descriptors. Adding variants later
is a breaking change for every compiled plugin. The cost of unused
variants is zero (discriminant values, no code paths, no memory).
The cost of needing them later is a forced SDK version bump and
plugin recompilation.

**Divergence from original:** The original scheduler design
(scheduler-and-subsystems-design.md) used 3 levels. The upgrade to 5
is accepted because:

1. `High` serves user-initiated batch operations ("re-enrich all PS2
   games") that should preempt normal enrichment but not bypass
   interactive responses.
2. `Low` serves plugin-contributed deferrable work that should run
   above `Background` but below core `Normal` operations.
3. The `repr(C)` constraint means we cannot add these later without
   breaking the ABI. With `inventory`-collected descriptors, all
   plugins share the same enum definition — it must be right from
   the start.

**Explicit discriminants** prevent accidental ABI breakage from
reordering variants. Every `repr(C)` enum in the SDK must have
explicitly assigned discriminant values.

## §3 Notification Architecture

### 3.1 Decision: `NotificationPayload` registered trait

**The payload IS the notification.** Each notification kind is a
struct implementing the `NotificationPayload` trait — a registered
trait like `Connector`, `WorkUnit`, etc. The payload type carries its
own urgency as an associated const, and any context-specific data
(entity IDs, error details) as fields.

```rust
/// Registered trait — extensions can define their own notification kinds.
trait NotificationPayload: Named + Registrable + Send + Sync + 'static {
    const URGENCY: Urgency;
}

#[repr(C)]
enum Urgency {
    Critical = 0,  // immediate attention required
    High     = 1,  // important, time-sensitive
    Normal   = 2,  // informational, worth surfacing
    Low      = 3,  // background, batchable
}
```

**Notification kinds are types, not enum variants.** Each defines its
own urgency and carries whatever context is relevant as struct fields:

```rust
struct ApiKeyExpired;
impl Named for ApiKeyExpired { const NAME: &'static str = "api_key_expired"; }
#[register]
impl NotificationPayload for ApiKeyExpired {
    const URGENCY: Urgency = Urgency::Critical;
}

struct DownloadFailed { entity: EntityId, reason: ErrorClass }
impl Named for DownloadFailed { const NAME: &'static str = "download_failed"; }
#[register]
impl NotificationPayload for DownloadFailed {
    const URGENCY: Urgency = Urgency::High;
}

struct MetadataStale { entities: Box<[EntityId]> }
impl Named for MetadataStale { const NAME: &'static str = "metadata_stale"; }
#[register]
impl NotificationPayload for MetadataStale {
    const URGENCY: Urgency = Urgency::Low;
}
```

**Call site — payload type inferred from argument:**

```rust
ctx.notify(ApiKeyExpired);
ctx.notify(DownloadFailed { entity: id, reason });
ctx.notify(MetadataStale { entities: batch });
```

No turbofish needed. The host extracts urgency from the type's
associated const, name from `Named::NAME`, and descriptor from the
registry. **Category is not a parameter** — the host derives it from
the execution context (which subsystem, which connector, which work
unit is currently executing).

**Ergonomics via logging macro abstraction.** We define our own
`warn!`, `error!`, `info!`, `debug!`, `trace!` macros that wrap
`tracing` and accept an optional `notify(...)` parameter:

```rust
// Just a log — no notification
warn!("download retry attempt {attempt} for {game}");

// Log + typed notification — payload carries all context
warn!(
    notify(DownloadFailed { entity: game.id, reason }),
    "Download failed — will retry automatically."
);

// Info-level log + notification
info!(
    notify(EnrichmentComplete { entity: game.id }),
    "IGDB enrichment complete for {game}"
);
```

When `notify(...)` is absent, the macro expands to a plain
`tracing::warn!()` (or `info!`, `error!`, etc.). When present, it
also emits through `ctx.notify(payload)`.

This gives:
- **One macro family** — same `warn!`, `error!`, `info!` everywhere.
  Notification is an optional annotation, not a separate API.
- **Full type safety** — notification kinds are types, not strings or
  enum variants. The compiler enforces that only registered payloads
  can be emitted.
- **Zero ceremony for non-notification logs** — the common case (just
  logging) has no extra noise.
- **Dual output when needed** — log gets the structured event for
  observability, notification bus gets the typed payload for user
  delivery.
- **Extensible** — extensions define new notification kinds by
  implementing `NotificationPayload` and registering. No host changes
  needed.
- **Single notification path** — all notifications flow through one
  bus, one set of tests, one place to debug.

### 3.2 Notification delivery

```
our warn!/error!/info! macro
  → always: tracing::warn/info/error (for log/observability)
  → if notify(...) present:
      ctx.notify(payload)
        → host enriches with execution context (subsystem, connector, work unit)
        → async-broadcast channel
        → SSE handlers (filter by Audience)
        → persistence (notification history table)
```

### 3.3 Supplementary: Notification work units

For aggregated, debounced, scheduled, and absence-based notifications,
the notification subsystem registers work units with the scheduler:

- `BatchSummaryWork` — triggers on `BatchComplete`, aggregates, emits
  one summary notification per batch
- `NotificationDeduplicator` — absorbs repeated failures within a
  window, emits one summary
- `DigestWork` — runs on cadence (Background priority), queries DB,
  emits digest notification
- `QueueEmptyCheck` — triggers on `DownloadComplete`, checks if queue
  is empty, notifies

These work units also emit through `ctx.notify(payload)` — same
typed path, same bus.

### 3.4 What was rejected

- **`NotificationEvent` struct with category/urgency/message fields** —
  Untyped. Category is derivable from execution context. Urgency is an
  intrinsic property of the notification kind, not a per-call-site
  choice. `message: String` violates the no-String principle. Replaced
  by typed `NotificationPayload` structs where the type IS the event.
- **`NotificationCategory` enum as a parameter** — category is not
  something the plugin should specify. The host knows which subsystem,
  connector, and work unit is executing. Category is derived, not
  declared.
- **Const generic urgency (`ctx.notify::<{ Urgency::Critical }>(...)`)** —
  urgency is an intrinsic property of the notification kind, not a
  per-call-site decision. `ApiKeyExpired` is always critical. Baking
  urgency into the payload type via associated const is cleaner.
- **NotificationLayer as the primary notification source** — strings
  for category/urgency, log messages ≠ notification messages, dual
  paths. The tracing Layer concept is captured in our logging macros
  instead — same ergonomics, typed output.
- **Tracing Layer for observability as a separate system** — deferred.
  Our logging macros already emit tracing events. If Prometheus
  metrics are needed later, a Layer can scrape from the existing
  tracing output without affecting the notification architecture.

## §4 Content Model

### 4.1 Decision: Flat entity model with metadata

Two anchor tables:

```sql
CREATE TABLE entity_types (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL UNIQUE      -- "game", "movie", "user", etc.
);

CREATE TABLE entities (
    id INTEGER PRIMARY KEY,
    type_id INTEGER NOT NULL REFERENCES entity_types(id),
    created_at DATETIME NOT NULL DEFAULT (datetime('now'))
);
```

**Everything else is metadata.** Ratings, descriptions, media, compat
info, platform tags, franchise membership, variant relationships,
credits, classifications, stats, links — all in separate metadata
tables keyed by `entity_id` FK to `entities`.

**Users are entities** (§11.1). A user is an entity with `type_id`
pointing to `User`. User profiles, roles, linked identities, play
sessions, credentials — all metadata. One data model for everything.

### 4.2 Systems are metadata, not structure

An entity tagged `platform: "nes"` is the same structural entity as
one tagged `platform: "steam"`. Platform/system is a filterable
property — a metadata label — not a structural concept.

### 4.3 Hierarchy is metadata relationships

The game hierarchy (franchise → work → edition → variant) exists as
metadata relationships between entities, not as hardcoded structural
tables. A franchise relationship links two entities. An "edition-of"
relationship links two entities. A "variant-of" relationship links
two entities.

This means adding a new hierarchy level or relationship type is a
metadata change, not a schema change. Cross-type relationships (e.g.,
User → Game for ownership or play history) use the same system (§11.7).

### 4.4 Metadata tables

Each metadata table is a ZST marker implementing `MetadataTable`
(§1.2.1). First-party tables are defined in the SDK. All tables
automatically get implicit plumbing columns (§1.2.2): `id`,
`entity_id`, `source`, `created_at`, `updated_at`.

Domain-specific columns are declared via `Column<In<M>, As<V>>`
impls. The host discovers all column descriptors at startup and
creates the actual DB tables via SeaORM.

| ZST | Table | Purpose | Domain columns |
|-----|-------|---------|----------------|
| `CompatInfo` | `compat_info` | Emulator compatibility | `Emulator`, `Tier`, `Notes` |
| `Rating` | `ratings` | Scores (normalized 0-100) | `Score`, `MaxScore` |
| `Description` | `descriptions` | Summaries, full text | `Text`, `Language`, `Kind` |
| `Media` | `media` | Cover art, screenshots, video | `Url`, `MediaType`, `Width`, `Height` |
| `Credits` | `credits` | Developers, publishers | `Name`, `Role` |
| `Classifications` | `classifications` | Genres, tags, age ratings | `Label`, `Scheme` |
| `Stats` | `stats` | Downloads, reviews, popularity | `Metric`, `Value` |
| `Links` | `links` | Store pages, wikis, guides | `Url`, `LinkType` |
| `Titles` | `titles` | Regional/localized names | `Name`, `Language`, `IsPrimary` |
| `Composites` | `composites` | Cached aggregates | (host-managed, not plugin-writable) |

Multiple connectors write to the same shared table. Source provenance
is automatic — the `source` plumbing column is stamped from
`ConnectorContext<C>` (§1.2.4). Multi-source metadata with
preferred-entry selection is a query over one table filtered by source,
not a union across fragmented tables.

### 4.5 Metadata association — dual-trait pattern with `min_specialization`

The old `Associate<M>` trait hit the **orphan rule**: a plugin
defining a new `EntityType` cannot `impl Associate<Rating> for
PluginEntity` because neither the trait (`Associate`) nor the foreign
type (`Rating`) is local to the plugin crate. A plugin defining a new
`MetadataTable` has the symmetric problem.

**Solution:** Two traits, each anchored to the side that "owns" the
declaration:

```rust
/// Table-side: "this metadata table accepts this entity type."
/// Implemented where the MetadataTable ZST lives.
trait SupportsEntity<E: EntityType>: MetadataTable {}

/// Entity-side: "this entity type carries this metadata."
/// Implemented where the EntityType ZST lives.
trait HasMetadata<M: MetadataTable>: EntityType {}
```

**Four association cases and where the impl lives:**

| Case | Impl | Crate |
|------|------|-------|
| Core entity + core metadata | `impl HasMetadata<Rating> for Game` | core/SDK |
| Core entity + plugin metadata | `impl SupportsEntity<Game> for PluginTable` | plugin |
| Plugin entity + core metadata | `impl HasMetadata<Rating> for PluginEntity` | plugin |
| Plugin entity + plugin metadata | Either trait, both are local | plugin |

Every case has at least one local type. The orphan rule is satisfied
in all four.

#### 4.5.1 Blanket bridge via `min_specialization`

The query and write APIs use a single bound: `M: SupportsEntity<E>`.
To make `HasMetadata` impls (entity-side) automatically satisfy this
bound, a blanket impl bridges the two traits:

```rust
#![feature(min_specialization)]

default impl<E, M> SupportsEntity<E> for M
where
    E: HasMetadata<M> + EntityType,
    M: MetadataTable,
{}
```

The `default` keyword requires `min_specialization`. Without it, this
blanket impl would conflict with any explicit `impl SupportsEntity<E>
for M` — specialization allows the explicit impl to take priority
over the blanket.

**Result:** Either direction of declaration works, and the query API
only needs one bound:

```rust
// ✅ Game has ratings (via HasMetadata<Rating> for Game → blanket bridge)
ctx.query::<Rating>().of::<Game>(id).execute()?;

// ❌ Movie doesn't have compat info — compile error
ctx.query::<CompatInfo>().of::<Movie>(id).execute()?;

// ✅ Writing also gated by SupportsEntity bound
ctx.write::<Rating>(entity_id, |row| row.set::<Score>(85))?;

// ✅ Plugin entity with core metadata (plugin wrote HasMetadata<Rating> for PluginEntity)
ctx.query::<Rating>().of::<PluginEntity>(id).execute()?;

// ✅ Core entity with plugin metadata (plugin wrote SupportsEntity<Game> for PluginTable)
ctx.query::<PluginTable>().of::<Game>(id).execute()?;
```

`SupportsEntity` is the query/write permission system. A plugin
literally cannot read or write metadata it hasn't declared an
association for — from either direction.

#### 4.5.2 `min_specialization` — nightly requirement and soundness caveat

This design requires **nightly Rust** with `#![feature(min_specialization)]`.

`min_specialization` is a deliberately scoped subset of the full
specialization RFC. It restricts which impls can be marked `default`
to avoid most soundness holes. It is used extensively in `std`
(e.g., `ToString` blanket impl for `Display` types) and is proven
reliable in practice.

**Known soundness issue:** `min_specialization` has a known gap
involving lifetime-dependent specialization. The compiler cannot
fully verify that a specializing impl is strictly more specific
than the default when lifetimes are involved. This is the primary
reason the feature remains unstable.

**Mitigation:** We do not specialize on lifetime bounds. Our
blanket bridge uses only trait bounds (`EntityType`, `MetadataTable`,
`HasMetadata<M>`), never lifetime parameters. The soundness issue
does not apply to our usage pattern. This must be **actively
maintained** — any future specialization involving lifetimes must
be manually reviewed for soundness. The compiler will not catch
lifetime-dependent specialization bugs.

**Why accept nightly:** The trait architecture (ZST markers,
cross-crate blanket impls, descriptor bridges) already pushes
against stable Rust's limits. `min_specialization` is one feature
flag that unlocks the dual-trait bridge cleanly, versus working
around the orphan rule with less ergonomic patterns (wrapper types,
re-exports, or runtime checks). The project commits to nightly
for this reason.

### 4.6 Entity type registration

Entity types (`Game`, `Movie`, `User`, etc.) are ZST markers
implementing an `EntityType` trait, registered via `#[register]` —
same pattern as everything else. The `entity_types` DB table is
populated from `EntityTypeDescriptor`s at startup.

```rust
trait EntityType: Send + Sync + 'static {}

struct Game;
#[register]
impl EntityType for Game {}

struct User;
#[register]
impl EntityType for User {}
```

`HasMetadata` impls (entity-side) and `SupportsEntity` impls
(table-side) bind entity types to metadata tables at compile time
(§4.5). Runtime DB rows (`entity_types` table) mirror the registered
entity types for query purposes. The host syncs descriptors → DB
rows at startup.

Plugin-contributed entity types follow the same pattern: a plugin
registers a new `EntityType` ZST and declares `HasMetadata` impls
for whichever metadata tables it supports. If a plugin defines a
new metadata table, it writes `SupportsEntity` impls for existing
entity types. The host discovers the new entity type through the
registry and adds it to the DB.

### 4.7 What was rejected

- **6-level structural hierarchy** (franchises → canonical_works →
  canonical_editions → canonical_games → game_variants as separate
  tables) — over-specified. The hierarchy is metadata relationships,
  not structural schema.
- **2-level convergence** (canonical_works → canonical_editions only)
  — lost the metadata tables and the flat entity model.
- **`canonical_works` / `canonical_editions` naming** — the FRBR
  naming was adopted by the convergence but the underlying model is
  `entities` with metadata, not a work/edition hierarchy. The naming
  discussion is moot when the hierarchy is metadata.
- **`contents` / `library_items` naming** — too narrow once users
  are entities. Kept `entities` as the universal table name (§11.1).
- **Per-plugin metadata tables** — fragments queries. One `ratings`
  table with source-tagging is more efficient than N plugin-specific
  rating tables that need union queries for aggregation.
- **`Metadata` trait with `EntityTrait` supertrait** — tied the SDK
  to SeaORM. Replaced by pure `MetadataTable` marker + `Column`
  traits with no ORM dependency.
- **Separate user tables** — users are entities (§11.1). No parallel
  data model for users.
- **Single `Associate<M>` trait** — orphan rule prevents cross-crate
  associations (plugin entity + core metadata, or core entity + plugin
  metadata). Replaced by dual-trait `SupportsEntity<E>` /
  `HasMetadata<M>` with `min_specialization` blanket bridge (§4.5).
- **`MetadataTable<E>` (generic over entity type)** — moves the orphan
  problem rather than solving it: a plugin adding a new entity type
  to an existing metadata table still can't impl the foreign generic.
  Rejected in favour of the dual-trait pattern.

### 4.8 Naming convention

Tables use `entities` (universal, type-agnostic), not
`canonical_works` or `canonical_editions`. The name reflects the
design: everything — games, movies, users — is an entity.
Structure comes from metadata, not schema.

## §5 Scheduler & Subsystems

### 5.1 Decision: Event-driven DAG (preserved from original)

The scheduler replaces loop-and-sleep polling. Events drive work, the
scheduler batches and orders intelligently, and minimum cadence is a
safety net, not the primary mechanism.

```
┌──────────────────────────────────────────────────┐
│                   Scheduler                       │
│                                                   │
│  Event Queue ──► DAG Resolution ──► Dispatch      │
│       ▲              │                  │         │
│       │         batch + order           │         │
│       │              │                  ▼         │
│       └──────── Event Output ◄── WorkUnit::execute│
│                                                   │
│  Min Cadence Timer ──► Synthetic Sweep Events     │
├──────────────────────────────────────────────────┤
│              Subsystems (peers)                   │
│                                                   │
│  ┌─────────┐  ┌────────────┐  ┌────────┐        │
│  │ Curator  │  │ Housekeeper│  │ Doctor │  ...   │
│  └─────────┘  └────────────┘  └────────┘        │
└──────────────────────────────────────────────────┘
```

The scheduler is pure infrastructure — events, ordering, batching,
dispatch. Domain knowledge lives in the subsystems. Subsystems are
peers: they implement the same registration trait, produce typed work
units, and the scheduler weaves them into a single execution graph.

At startup, the scheduler collects all work units from all subsystems,
reads their `type Triggers` and `type Emits` associated types, and
builds a directed acyclic graph. The DAG determines what can run in
parallel, what must be sequential, and detects cycles as startup errors.

### 5.2 WorkUnit: Semantic generic parameters

Generic parameters use semantic wrapper ZSTs so usage reads like
English instead of anonymous positional generics:

```rust
/// Semantic wrappers — self-documenting generic parameters.
struct For<T: ?Sized>(PhantomData<T>);
struct With<I = ()>(PhantomData<I>);
struct Returns<O = ()>(PhantomData<O>);

/// Sealed extraction traits — only For/With/Returns implement these.
/// The host uses these to resolve inner types for scheduling and
/// type-safe dispatch.
trait ForTarget { type Target: ?Sized; }
trait WithInput { type Input; }
trait ReturnsOutput { type Output; }

impl<T: ?Sized> ForTarget for For<T> { type Target = T; }
impl<I> WithInput for With<I> { type Input = I; }
impl<O> ReturnsOutput for Returns<O> { type Output = O; }
```

```rust
trait WorkUnit<
    F: ForTarget,
    W: WithInput = With<()>,
    R: ReturnsOutput = Returns<()>,
>: Send + Sync + 'static {
    /// Identity comes from `Named::NAME` via `Registrable` (§1.1).

    /// Which subsystem owns this work unit (§5.6).
    type Subsystem: Subsystem;

    /// Static event declarations — compile-time DAG wiring.
    type Triggers: EventKindSet;
    type Emits: EventKindSet;

    const BATCH_TARGET: usize = 1;
    const BATCH_TIMEOUT: Duration = Duration::from_secs(60);
    const PRIORITY: Priority = Priority::Normal;
    const MAX_TRANSACTION_CHUNK: Option<usize> = None;
    const MAX_RETRIES: u32 = 5;

    fn execute(
        &self,
        ctx: &SchedulerContext,
        target: &F::Target,
        input: W::Input,
    ) -> Result<R::Output>;
}
```

**`EventKindSet` — tuple-based type-level set:**

```rust
trait EventKindSet {}
impl EventKindSet for () {}
impl<A: EventKind> EventKindSet for (A,) {}
impl<A: EventKind, B: EventKind> EventKindSet for (A, B) {}
impl<A: EventKind, B: EventKind, C: EventKind> EventKindSet for (A, B, C) {}
// ... macro-generated up to reasonable arity
```

**Note:** `With` and `Returns` use stable generic type parameter
defaults — not `associated_type_defaults` (unstable, known bugs, no
stabilization timeline). Callers write `impl WorkUnit<For<Steam>>`
and get `With<()>`, `Returns<()>` automatically.

**Key design decisions:**

- **`For<T>`** — type-safe per target. Compile-time guarantee that
  `CatalogueWork` only targets `Cataloguer` connectors. The wrapper
  makes intent explicit at every usage site.
- **`With<I>` / `Returns<O>` with defaults** — each work unit declares
  what it consumes and produces. Defaults to `()` when not needed.
  Tuple types for multiple inputs: `With<(&Config, &mut Stats)>`.
  Each work unit chooses its own input/output types directly.
- **Sealed extraction** — `ForTarget`, `WithInput`, `ReturnsOutput`
  are sealed traits. Only the wrapper ZSTs implement them. No one can
  pass a raw type where a wrapper is expected.
- **Sync `fn execute()`** — sync core principle. The scheduler runs on
  a plain OS thread. Only the axum web shell is async.
- **`&[Event]` was considered** for the batch parameter but replaced by
  the `With<I>` type, which is more expressive. Specialized traits fix
  `With` to the appropriate batch type for their domain.

### 5.3 Subsystem grouping via `type Subsystem`

The specialized trait hierarchies (`ConnectorWorkUnit`, `DbWorkUnit`,
`SystemWorkUnit`) are eliminated (see §19.3). They were convenience
aliases that locked input/output types. Now that `WorkUnit` has
`type Subsystem` (§5.2), subsystem grouping is handled at the type
level. Each work unit chooses its own input/output types.
`Cataloguer`, `Enricher`, `Sourcer` become lightweight marker ZSTs
that assert the correct subsystem, not separate trait hierarchies
with constrained generics.

The DAG wiring layer uses `type Emits: EventKindSet` (§5.2) for
cascade routing. The scheduler knows at registration time which
events a work unit can emit — the DAG is wired statically from the
type-level sets. No runtime `into_events()` needed.

### 5.4 Registration: Same `#[register]` pattern

Work units and subsystems use the same `#[register]` + `HasDescriptor<D>`
+ `Descriptor` + `Registry` pattern as connectors (§1.1). Everything
that goes through `inventory` uses `repr(C)` descriptors with
`extern "C"` function pointers. Domain registries (`WorkUnitRegistry`,
`SubsystemRegistry`) implement the `Registry` trait with domain-
specific `validate()` overrides — `WorkUnitRegistry` adds DAG cycle
detection, `SubsystemRegistry` checks initialization ordering.

```rust
struct MyApiHealthCheck;
impl Named for MyApiHealthCheck { const NAME: &'static str = "my_api_health"; }

#[register]
impl WorkUnit for MyApiHealthCheck {
    type Subsystem = Doctor;
    type Triggers = (Sweep,);
    type Emits = (HealthCheckFailed,);

    const PRIORITY: Priority = Priority::Background;

    fn execute(&self, ctx: &SchedulerContext, target: &Steam,
               input: ()) -> Result<()> {
        let status = if api_ok { HealthStatus::Healthy }
                     else { HealthStatus::Degraded };
        ctx.write::<Health>(connector_id, |row| {
            row.set::<Status>(status);
        });
        Ok(())
    }
}
```

Each concrete `(WorkUnit, T)` pair generates a separate
`WorkUnitDescriptor` — same pattern as `Cataloguer<E>` generating
per-entity-type descriptors. The generic `T` is resolved at
registration time, monomorphized into the descriptor's function
pointers. No `dyn` at the boundary.

The `#[register]` macro checks `HasDescriptor<WorkUnitDescriptor>`,
generates the `repr(C)` descriptor, and submits to
`Registry::<WorkUnitDescriptor>`. Plugins can register their own work
units — an enricher plugin that adds a health check for its API gets
woven into the host's DAG automatically.

**Descriptor table (expanded from §1.4):**

| Descriptor | `#[register]` on | Bridges to |
|-----------|-----------------|-----------|
| `ConnectorDescriptor` | `impl Connector for X` | `Connector` methods |
| `CataloguerDescriptor` | `impl Cataloguer<E> for X` | `Cataloguer` methods + entity type |
| `EnricherDescriptor` | `impl Enricher<E> for X` | `Enricher` methods + entity type |
| `SourcerDescriptor` | `impl Sourcer<E, S> for X` | `Sourcer` methods + entity type + source type |
| `SourceTypeDescriptor` | `impl SourceType for X` | `SourceType` consts + `Url` associated type |
| `WorkUnitDescriptor` | `impl WorkUnit<For<T>, ...> for X` | `WorkUnit` methods |
| `SubsystemDescriptor` | `impl Subsystem for X` | `Subsystem` methods |
| `ErrorDescriptor` | `impl AppError<K> for X` | `ErrorKind` methods |

Collection is per-specialization:

```rust
// Host startup
for desc in Registry::<ConnectorDescriptor>::iter() { ... }
for desc in Registry::<WorkUnitDescriptor>::iter() { ... }
for desc in Registry::<SubsystemDescriptor>::iter() { ... }
```

### 5.5 Events

Events are the currency of the scheduler. They carry typed payloads
and flow through the DAG:

Event kinds are ZST markers registered via `#[register]`, same
pattern as everything else. NOT a `repr(C)` enum — plugins must
be able to define their own event kinds (§6.2: extensible domain
concepts are trait impls, not enum variants).

```rust
trait EventKind: Registrable {}

// --- Curator events ---
struct Sweep;
#[register] impl EventKind for Sweep {}

struct EntityDiscovered;
#[register] impl EventKind for EntityDiscovered {}

struct EntityEnriched;
#[register] impl EventKind for EntityEnriched {}

struct SourceResolved;
#[register] impl EventKind for SourceResolved {}

struct DownloadComplete;
#[register] impl EventKind for DownloadComplete {}

struct ExtractionComplete;
#[register] impl EventKind for ExtractionComplete {}

struct DownloadFolded;
#[register] impl EventKind for DownloadFolded {}

// --- Housekeeper events ---
struct DbWalThresholdExceeded;
#[register] impl EventKind for DbWalThresholdExceeded {}

// --- Doctor events ---
struct HealthCheckFailed;
#[register] impl EventKind for HealthCheckFailed {}

struct SystemThresholdExceeded;
#[register] impl EventKind for SystemThresholdExceeded {}

// --- User-initiated ---
struct ManualTrigger;
#[register] impl EventKind for ManualTrigger {}
```

Plugins register their own event kinds the same way. The
scheduler discovers all registered event kinds at startup
and wires the DAG accordingly.

### 5.6 Three subsystems

```rust
trait Subsystem: Registrable {
    const MIN_CADENCE: Duration;
    fn initialize(&self, ctx: &SchedulerContext) -> Result<()> { Ok(()) }
    fn shutdown(&self, ctx: &SchedulerContext) -> Result<()> { Ok(()) }
}
```

Work units declare their subsystem, not the other way around — the
host collects work units per subsystem by scanning the registry at
startup. Extensions adding new work units just set
`type Subsystem = Curator;` — no modification to the subsystem needed.

```rust
// On WorkUnit (§5.2):
type Subsystem: Subsystem;
```

| Subsystem | Cadence | Work Units |
|-----------|---------|------------|
| **Curator** | 1h floor | CatalogueWork, EnrichWork, SourceWork |
| **Housekeeper** | 2h floor | WalCheckpoint, VacuumDb, PruneStaleData, DeduplicateContent, InvalidateCaches |
| **Doctor** | 5min floor | ConnectorHealthCheck, DataIntegrityCheck, AutoRecover, DiskSpaceCheck |

Cadence is a **floor** — if nothing triggers work, the subsystem gets
a `Sweep` event at most `MIN_CADENCE` apart. Not a scheduling
interval. If work was triggered by real events within the window, the
timer resets.

Adding a fourth subsystem (e.g., Notifier for alerts) is a struct
implementing `Subsystem` with `#[register]`. No scheduler changes.

### 5.7 Health status and recovery strategies

The `Diagnosable` trait and `DiagnosticReport` struct are eliminated
(see §19.3). Health checking is a regular work unit that writes health
metadata through the context API, same pattern as everything else:

```rust
// Health check work unit — same pattern as everything else
fn execute(&self, ctx: &SchedulerContext, target: &Steam,
           input: ()) -> Result<()> {
    let status = if api_ok { HealthStatus::Healthy }
                 else { HealthStatus::Degraded };
    ctx.write::<Health>(connector_id, |row| {
        row.set::<Status>(status);
    });
    Ok(())
}

// Scheduler reads health for dispatch decisions:
let health = ctx.query::<Health>().of(connector_id)?;
```

`HealthStatus` remains as a `repr(C)` enum — it's a value type, not
a pipeline struct:

```rust
#[repr(C)]
enum HealthStatus {
    Healthy   = 0,
    Degraded  = 1,
    Unhealthy = 2,
}
```

Recovery strategies remain as registered types — the Doctor subsystem
dispatches recovery work units triggered by health status changes
through the event DAG:

```rust
/// Registered trait — recovery strategies are types, not enum variants.
/// Each strategy declares WHAT kind of recovery it represents.
/// Strategies can carry runtime data as struct fields when needed
/// (e.g., WaitForRateLimit carries reset_at).
trait RecoveryStrategy: Registrable {}

struct ReAuthenticate;
#[register] impl RecoveryStrategy for ReAuthenticate {}

struct WaitForRateLimit { reset_at: Instant }
#[register] impl RecoveryStrategy for WaitForRateLimit {}

struct Reconnect;
#[register] impl RecoveryStrategy for Reconnect {}

/// Type-level set, same pattern as EventKindSet (§5.2).
trait RecoveryStrategySet {}
impl RecoveryStrategySet for () {}
impl<A: RecoveryStrategy> RecoveryStrategySet for (A,) {}
impl<A: RecoveryStrategy, B: RecoveryStrategy> RecoveryStrategySet for (A, B) {}
impl<A: RecoveryStrategy, B: RecoveryStrategy, C: RecoveryStrategy>
    RecoveryStrategySet for (A, B, C) {}
```

Health status feeds dispatch decisions:

| Health | Dispatch |
|--------|----------|
| Healthy | Normal — dispatch as configured |
| Degraded | Throttled — larger batches, inter-batch delay |
| Unhealthy | Defer — accumulate silently until recovery |

No `Diagnosable` trait on connectors. Health monitoring is handled
by the Doctor subsystem's work units, not by a trait on the connector.

### 5.8 Batching heuristics

Beyond static `BATCH_TARGET` and `BATCH_TIMEOUT`, the scheduler uses
seven composable heuristic dimensions:

1. **Adaptive yield** — tick duration (10–200ms) adjusts based on
   event velocity. Burst mode tightens the loop; idle mode loosens.
2. **Velocity-scaled timeout** — at low velocity, effective timeout
   shortens so small batches dispatch promptly.
3. **Health-gated dispatch** — Doctor's health metadata (queried via
   `ctx.query::<Health>()`) feeds dispatch decisions: Normal,
   Throttled, or Defer.
4. **Error-class retry** — transient failures enter a deferred queue
   with exponential backoff. Fixable/permanent errors escalate to
   notifications.
5. **Priority-reserved concurrency** — thread pool slots reserved for
   `Immediate` priority so user actions never wait behind sweeps.
6. **Per-connector cap** — no single connector can monopolize the
   thread pool.
7. **Rate limit token bucket** — per-connector token buckets enforce
   declared rate limits. Batches dispatch only when both a concurrency
   slot AND rate-limit tokens are available.

`Immediate` priority events bypass batching entirely — they dispatch
as a batch of 1, right now.

### 5.9 Event cascade example

```
User links Steam credentials
  → ManualTrigger { connector: "steam" } (Immediate priority)
  → Curator dispatches CatalogueWork for Steam
  → Steam cataloguer discovers 500 games
  → 500× EntityDiscovered events emitted
  → Scheduler batches into groups of 50 (BATCH_TARGET)
  → Curator dispatches EnrichWork (10 batches, Normal priority)
  → Curator dispatches SourceWork (10 batches, Normal priority)
  → Enrichment completes for first batch
  → 50× EntityEnriched events emitted
  → Doctor's DataIntegrityCheck triggers (validates FK consistency)
  → Housekeeper's CacheInvalidation triggers (stale views)
```

No polling. No sleep. Each step triggers the next. If nothing is
discovered, the cascade stays quiet.

### 5.10 What the convergence changed (and what we kept)

**Preserved faithfully:** DAG architecture, three subsystems with
same work units and cadence values, `EventKind` types, `Subsystem`
trait, health-aware dispatch via health metadata and recovery
strategies, batching heuristics, `inventory`-based registration.

**Adopted from convergence:**
- Sync `fn execute()` — correct per sync-core principle (original
  had `async fn execute` which predated the sync-core decision).
- `WorkResult` struct with `succeeded`/`failed`/`emitted` fields —
  better than raw event collections for retry and notification decisions.
  Now superseded by context writes and `Result<()>` returns.

**Corrected:**
- `WorkUnit` with `type Subsystem` for grouping — convergence v1 draft
  dropped type safety for object safety. Type safety always trumps.
  Subsystem association at the type level prevents dispatch errors.
- Specialized trait hierarchies (`ConnectorWorkUnit`, `DbWorkUnit`,
  `SystemWorkUnit`) removed — plain `WorkUnit` with `type Subsystem`
  handles grouping (§19.3). Each work unit chooses its own context
  interaction pattern.
- `impl IntoIterator<Item = Event>` for batch parameter replaced by
  associated `Input` type — more expressive, specialized per domain.
- Registration uses `#[register]` + `HasDescriptor<D>` + `Registry<D>`
  — same universal pattern as connectors (§1.1), not bare `inventory`
  with `dyn Trait` objects.

## §6 Cross-Cutting Conventions

### 6.1 Architectural idiom: Blanket marker traits for compile-time association

The project uses blanket-implemented marker traits as a standard
pattern for expressing valid type combinations at compile time:

- `SupportsEntity<E: EntityType>` / `HasMetadata<M: MetadataTable>` —
  "this metadata table accepts this entity type" / "this entity type
  carries this metadata" (§4.5, dual-trait with blanket bridge)
- `HasDescriptor<D: Descriptor>` — "this type can produce a descriptor
  of type `D` for registration" (§1.1, sealed)

The pattern:

```rust
// 1. Define the marker, generic over what it associates with
trait MarkerTrait<Assoc> {}

// 2. Blanket-impl for types satisfying the domain constraint
impl<T: DomainTrait> MarkerTrait<DomainDescriptor> for T {}

// 3. Use as bound in generic code
fn do_thing<T: MarkerTrait<X>>() { ... }
```

This is a **named project convention**. When a new domain needs
compile-time association checking, reach for this pattern first.

### 6.2 The plugin boundary paradigm: Trait impls only

Plugins define trait implementations. The host runs them. This is
the single rule that resolves all boundary extensibility questions:

- A plugin wanting a custom notification kind **impls
  `NotificationPayload`** and registers it. The host discovers it via
  the registry and routes it through the notification bus.
- A plugin wanting a custom event kind **impls a trait** and
  registers it. The host weaves it into the DAG.
- A plugin wanting a custom entity type **impls a trait** and
  registers it. The host adds it to the entity type registry.

- A plugin wanting custom error types **impls `AppError`** with
  its own error enum. The host reads `class()` and `guidance()`
  generically — same scheduler retry logic, same notification
  routing.

**Consequence for `repr(C)` enums:** Fixed `repr(C)` enums are for
descriptor fields that describe static properties (priority level,
error class discriminant, HTTP method). They are NOT for domain
concepts that plugins might extend. Extensible domain concepts are
trait impls + `#[register]`, not enum variants.

If you're asking "should this enum have room for plugin extensions?"
— the answer is: it shouldn't be an enum. It should be a trait.

### 6.3 `repr(C)` enum convention

Every `repr(C)` enum in the SDK:

1. Has explicitly assigned discriminant values (no implicit ordering)
2. Represents a **closed set** of values — things that are
   definitionally fixed and should not be extended by plugins
3. Uses `lower discriminant = higher priority/precedence` where an
   ordering relationship exists

Current `repr(C)` enums: `Priority`, `ErrorClass` discriminant,
`HealthStatus`, `Urgency`, `FieldKind`. These are all closed sets —
a plugin has no business defining a new priority level or urgency.
`HttpMethod` is expressed as ZSTs + trait (§1.3) rather than
`repr(C)` enum — they're used as generic type parameters.

If a concept seems like it might need plugin extension, it's a trait
+ registration, not a `repr(C)` enum.

## §7 Error Classification

### 7.1 Decision: Three error classes (closed taxonomy)

```rust
#[repr(C)]
enum ErrorClass {
    Transient  = 0,  // retry with backoff
    Fixable    = 1,  // notify user with guidance
    Permanent  = 2,  // log + notify admin
}
```

This is a **closed set** — every error in the system falls into one
of these three classes. No plugin has business inventing a fourth
class. `ErrorClass` is `repr(C)` with explicit discriminants per §6.3.

**What about `Informational`?** The research doc listed it as a fourth
class for "expected conditions that look like errors but aren't" (no
match, already done, skipped). These are not errors — they're
batch execution outcomes. Tracked in the scheduler's result
aggregation, not error handling.

### 7.2 `ErrorKind` trait: Class behavior is type-level

The three classes aren't just discriminants — they encode recovery
behavior. Each class is a ZST implementing `ErrorKind`:

```rust
trait ErrorKind: 'static {
    /// Max retry attempts. 0 = no retry.
    const MAX_RETRIES: u32 = 0;
}

struct Transient;
impl ErrorKind for Transient {
    const MAX_RETRIES: u32 = 5;
}

struct Fixable;
impl ErrorKind for Fixable {}

struct Permanent;
impl ErrorKind for Permanent {}
```

All three are ZSTs — no runtime data on the error kind itself.
Retry logic is compile-time: the host checks
`attempt < K::MAX_RETRIES`. Notification is handled via
`NotificationPayload` (§3) — errors that should notify the user
emit a typed notification through `ctx.notify()`, not via an
`ErrorKind` method. Guidance (per-error user-facing message) is
an associated const on `AppError`, not on the error kind:

### 7.3 `AppError<K>` trait: Generic over class

```rust
trait AppError<K: ErrorKind>: std::error::Error + Send + Sync {
    /// Per-error guidance string for user-facing messages.
    /// Fixable errors should always set this; others default to None.
    const GUIDANCE: Option<&'static str> = None;
}
```

`fn kind(&self) -> &K` is removed — the error kind is the type
parameter `K`, a ZST. There's nothing to return. The host
knows the kind from the type: `AppError<Fixable>` IS fixable.

Error types are per-class, not per-domain-enum. Each error is a
focused type with one class baked in at compile time:

```rust
struct NetworkTimeout(reqwest::Error);

#[register]
impl AppError<Transient> for NetworkTimeout {}

struct DiskFull { needed: u64, available: u64 }

#[register]
impl AppError<Fixable> for DiskFull {
    const GUIDANCE: Option<&'static str> =
        Some("Free up disk space or change storage path");
}

struct MissingCredentials;

#[register]
impl AppError<Fixable> for MissingCredentials {
    const GUIDANCE: Option<&'static str> =
        Some("Configure credentials in Settings");
}
// Host derives which connector's credentials are missing from
// the execution context — no `provider` field needed.
```

No runtime classification. No `match err.class()`. The type IS the
class. `AppError<Transient>` can only be a retryable error — the
scheduler knows at compile time.

### 7.4 Type erasure via descriptors

At the dispatch boundary (scheduler, notification routing), errors
use the same `#[register]` + descriptor pattern as everything else.
`#[register]` on `impl AppError<K> for T` generates an
`ErrorDescriptor` that captures:

- The error class (as `repr(C)` discriminant from `K: ErrorKind`)
- `K::MAX_RETRIES` (const, baked into descriptor at compile time)
- `GUIDANCE` (const `Option<&'static str>`, baked into descriptor)
- Error name, display

The scheduler works with `ErrorDescriptor`, not `dyn AppError<K>`.
The descriptor IS the type-erased interface — same pattern as
`ConnectorDescriptor`, `WorkUnitDescriptor`, etc. No `Box<dyn>`
at the dispatch boundary.

Plugins define their own error types, impl `AppError<K>`, and
`#[register]` generates the descriptor. The host discovers error
types through `Registry::<ErrorDescriptor>` and handles them via
descriptor function pointers.

### 7.5 ABI boundary

`ErrorClass` remains a `repr(C)` enum for descriptor fields. The
`ErrorDescriptor` carries the class discriminant and const values
(`MAX_RETRIES`, `GUIDANCE`), same pattern as all other descriptors.
Error behavior is baked into the descriptor at compile time — no
runtime function pointers needed for these, no trait object vtables.

### 7.6 Error handling patterns

**Errors that should propagate:** Use `?`. Standard Rust.

**Errors that should warn but continue:** Explicit `if let Err` with
our `warn!` macro (§3):

```rust
if let Err(e) = db.update_progress(id, bytes) {
    warn!("progress update failed: {e}");
}
```

**Errors that truly don't matter:** `let _ =` — should be rare and
intentional. Each instance in the codebase needs individual review.

**Batch operations:** The scheduler processes batch outcomes as an
iterator of results. Counts successes, handles failures for retry via
`AppError<K>` (which already carries `MAX_RETRIES` and `GUIDANCE`),
counts skips. No intermediate `BatchResult` collection or separate
`BatchError` struct needed (see §19.2).

### 7.7 What was rejected

- **`let_warn!` macro** — hides control flow. `if let Err` + `warn!`
  is explicit, idiomatic Rust, and already supported by our logging
  macros. No special macro needed.
- **`Informational` error class** — not an error. Tracked in the
  scheduler's batch result aggregation as skipped items.
- **`ResultExt` trait** (Option C from research) — opt-in
  classification is too easy to forget. Domain error types with
  `AppError` impl make classification mandatory at the type level.
- **Wrapper/conversion approach** — no `AppError` enum wrapping
  domain errors, no `Into<AppError>` chain. The trait IS the
  contract. Domain types implement it directly.

### 7.8 Notification integration

| Error class | Notification? | Routing |
|-------------|:---:|---------|
| Transient (auto-retry) | No | — |
| Transient (retries exhausted) | Yes | User who triggered |
| Fixable | Yes, with `guidance` | User or admin |
| Permanent | Yes | Admin |
| Batch partial failure | Yes, summary | User who triggered |

### 9.3 Crate structure

```
saalis-primitives          (leaf — zero deps)
       ↑

       ↑

     ↑     ↑
saalis-core  saalis-web    (core: sync host + SeaORM; web: async axum)
     ↑         |
     └─────────┘           (web depends on core's lib.rs)
```

`saalis-web` and the thin `main.rs` bridge are async.

collection happens in `saalis-sdk` (re-exported) or `saalis-core`
(host-side) depends on how `dlopen` interacts with the platform constructor
sections in practice. Not a design decision — an implementation detail.

## §10 Configuration & Settings

### 10.1 Type taxonomy for SDK values

Every type in the SDK falls into one of four categories. The choice
is systematic, not ad-hoc:

| Category | When | Examples |
|----------|------|---------|
| **Trait** | Varying behavior, shared contract, extensible | `Connector`, `Locale`, `Credential`, `SourceType`, `SettingsDomain` |
| **Newtype** | Fixed representation, validation at construction | `EntityId(i64)`, `Port(u16)`, `RateLimit(NonZeroU32)` |
| **`repr(C)` enum** | Closed set, crosses ABI boundary | `Priority`, `SettingsScope`, `FieldKind`, `HealthStatus`, `Urgency` |
| **Primitive** | No semantic meaning beyond the raw value | `bool`, `usize`, `Str` (free-form text — `Cow<'static, str>`, avoids heap when static) |

**The test:** "Can I imagine two implementations with different
behavior?" If yes → trait. `Locale`: yes (BCP 47, POSIX, simplified).
`Credential`: yes (static key, OAuth token, certificate). `Port`:
no, always a validated u16. `EntityId`: no, always an opaque i64.

### 10.2 `SettingsValue` — the shared contract for storable values

Every type that appears in a `#[derive(Settings)]` struct implements
`SettingsValue`. This gives the derive macro a uniform interface:

```rust
trait SettingsValue: Serialize + DeserializeOwned + fmt::Display + Send + Sync + 'static {
    const FIELD_KIND: FieldKind;
}
// Serialization via serde (host chooses format — JSON into SQLite TEXT).
// Display is purely for user-facing rendering (masked for Credentials).
// No custom store()/parse() — serde handles it.

/// Compile-time marker — the derive macro uses `T: Secret` bounds
/// to generate masking behavior and `FieldKind::Password`.
trait Secret {}

// Blanket: all Credential types are Secret.
impl<T: Credential> Secret for T {}

#[repr(C)]
enum FieldKind {
    Text         = 0,
    Password     = 1,
    Number       = 2,
    Toggle       = 3,
    Select       = 4,
    Range        = 5,
    LocalePicker = 6,
}
```

Type → UI field mapping is inherent in the type, not annotated:

| Rust type | `FIELD_KIND` | `Secret`? | Notes |
|-----------|----------------|-----------|-------|
| `T: Credential` | `Password` | yes (blanket) | Secret by type, not annotation |
| `T: Locale` | `LocalePicker` | no | Options from known locales |
| `Port` | `Number` | no | min=1 max=65535 from newtype |
| `RateLimit` | `Number` | no | min=1 from `NonZeroU32` |
| `bool` | `Toggle` | no | |
| `Str` | `Text` | no | Free-form text — `Cow<'static, str>` |

### 10.3 `Locale` trait — language/region contract

```rust
trait Locale: Registrable + SettingsValue {
    fn language(&self) -> &'static str;
    fn region(&self) -> Option<&'static str>;
    fn script(&self) -> Option<&'static str>;
}
```

Multiple valid representations — the trait ensures consistent
access patterns regardless of implementation:

```rust
/// BCP 47 language tags: en-US, fi-FI, zh-Hans-CN
struct Bcp47Locale(/* validated */);
impl Locale for Bcp47Locale { ... }

// Other impls possible: PosixLocale, SimplifiedLocale, etc.
```

Settings fields use a concrete impl:

```rust
#[setting(scope = Profile, domain = Preferences)]
preferred_language: Bcp47Locale,
```

### 10.4 `Credential` trait — secret value contract

```rust
trait Credential: Registrable + SettingsValue {
    fn is_expired(&self) -> bool { false }
    fn needs_refresh(&self) -> bool { false }
}
```

`Credential: SettingsValue` implies `Secret` via blanket impl.
The derive macro checks `T: Secret` at compile time to generate
masking behavior and `FieldKind::Password`. No runtime
`is_secret()` method — secrecy is a type-level property.

**`Display` vs serde on Credentials:** `Display` is for user-facing
output — `Credential` types MUST mask the value (`"sk-****"`).
DB persistence uses serde `Serialize`/`Deserialize` — the actual
unmasked value for encrypted storage. These are independent:
`Display::fmt()` masks, serde serializes the real value.

First-party credential types:

```rust
/// Static API key — never expires.
struct ApiKey(Cow<'static, str>);
impl Credential for ApiKey {}
// Display: "sk-****"

/// OAuth bearer token with optional expiry.
struct OAuthToken {
    token: Cow<'static, str>,
    expires_at: Option<Instant>,
}
impl Credential for OAuthToken {
    fn is_expired(&self) -> bool {
        self.expires_at.map_or(false, |e| e < Instant::now())
    }
    fn needs_refresh(&self) -> bool { self.is_expired() }
}
```

### 10.5 `#[derive(Settings)]` — single source of truth

```rust
#[derive(Settings)]
#[settings(section = "steam")]
struct SteamSettings {
    #[setting(domain = Credentials)]
    api_key: ApiKey,

    #[setting(scope = Profile, domain = Curation)]
    preferred_language: Bcp47Locale,

    #[setting(domain = SteamAdvanced)]
    fetch_workshop: bool,

    #[setting(domain = Infrastructure)]
    rate_limit: RateLimit,
}
```

**What the derive macro generates:**

- Schema (field definitions, types, validation from `SettingsValue`)
- Resolution logic (walk the 7-layer precedence per field)
- Typed accessors (`settings.api_key` returns `ApiKey`, not `String`)
- Field labels (derived from field names: `api_key` → "API Key";
  override with `#[setting(label = "Custom")]` when needed)
- UI field kinds (from `SettingsValue::field_kind()`)
- Secret detection (from `T: Secret` marker trait, §10.2)
- Change diffing (derive generates per-field ZSTs; diff returns
  an iterator of changed field descriptors, not string names)

**Defaults:**

- `scope` defaults to `System` (most settings are system-wide,
  runtime-mutable). Override for `Global` (deployment-fixed) or
  `Profile` (per-profile overridable).
- `domain` is **required** — no sensible default. Every setting
  must explicitly declare its access domain.
- `label` derived from field name. Override only when the generated
  label isn't right.

### 10.6 Settings scope — resolution depth

Two orthogonal axes on each setting: **scope** (where the value
comes from) and **domain** (who can see/change it).

Scope determines which layers participate in resolution:

```rust
#[repr(C)]
enum SettingsScope {
    /// CLI, env, TOML, compiled defaults only. Not DB-mutable.
    /// Deployment-level configuration.
    Global  = 0,

    /// + DB settings table. Writable at runtime by authorized roles.
    /// Shared across all profiles.
    System  = 1,

    /// + per-profile overrides. Each profile can carry its own value.
    Profile = 2,
}
```

| Scope | CLI | Env | Profile override | Global override | DB settings | TOML | Default |
|-------|-----|-----|-----------------|----------------|-------------|------|---------|
| `Global` | yes | yes | — | — | — | yes | yes |
| `System` | yes | yes | — | — | yes | yes | yes |
| `Profile` | yes | yes | yes | yes | yes | yes | yes |

### 10.7 Settings domain — access control

`SettingsDomain` is a trait, extensible by plugins via `#[register]`.
Domains are semantic categories for access gating — orthogonal to
scope and UI sections.

```rust
trait SettingsDomain: Registrable {}
```

First-party domains in the SDK:

```rust
struct Credentials;      // API keys, passwords, tokens
struct Infrastructure;   // ports, paths, scheduler tuning
struct Curation;         // enrichment priorities, quality thresholds
struct Preferences;      // theme, language, display options
```

Plugins register their own:

```rust
struct SteamAdvanced;
#[register]
impl SettingsDomain for SteamAdvanced {}
```

The same setting section in the UI ("Steam") can span multiple
domains. A curator sees `preferred_language` (Curation domain)
but never sees `api_key` (Credentials domain) — it doesn't exist
as far as they know.

### 10.8 `SetAccess` — compile-time role-domain access matrix

Access control between roles and settings domains is a compile-time
relationship expressed as trait impls. No runtime mapper, no
`&self`, no function calls — the access level IS the impl.

```rust
/// Access levels are a closed set expressed as ZSTs — not because
/// they're extensible, but because Rust enum variants aren't types.
/// `SetAccess<R, ReadWrite>` requires `ReadWrite` to be a type, not
/// a value. No runtime cost — purely compile-time, zero bytes.
trait AccessLevel {}

struct Hidden;    // setting doesn't exist for this role
impl AccessLevel for Hidden {}

struct ReadOnly;  // can see value, can't change it
impl AccessLevel for ReadOnly {}

struct ReadWrite; // full access
impl AccessLevel for ReadWrite {}

/// User-facing trait — the nice declaration syntax.
/// Access level is a type parameter, not a const.
trait SetAccess<R: Role, A: AccessLevel>: SettingsDomain {}

/// Sealed bridge — resolves the unique access level for a
/// (Domain, Role) pair. Only one impl per pair is possible;
/// duplicate SetAccess impls for the same (Domain, Role) with
/// different access levels cause a compile error via this bridge.
trait ResolveAccess<R: Role>: SettingsDomain {
    type Level: AccessLevel;
}

/// Blanket bridge — SetAccess automatically provides ResolveAccess.
impl<D, R, A> ResolveAccess<R> for D
where D: SetAccess<R, A>, R: Role, A: AccessLevel {
    type Level = A;
}
```

**Default access via specialization:**

```rust
// Non-admin roles: Hidden by default (safest)
default impl<R: Role, D: SettingsDomain> SetAccess<R, Hidden> for D {}

// Admin: full access to everything by default
impl<D: SettingsDomain> SetAccess<Admin, ReadWrite> for D {}
```

**Specific overrides — reads like English:**

```rust
impl SetAccess<User, ReadWrite> for Preferences {}
impl SetAccess<Child, ReadOnly> for Preferences {}
```

"User has access ReadWrite to Preferences." "Child has access
ReadOnly to Preferences." Everything else is Hidden for non-admin
roles by default.

**The settings engine queries access at compile time** via
`<Domain as ResolveAccess<R>>::Level` when:

- **Rendering settings UI** — `Hidden` fields omitted entirely
  (not disabled — absent)
- **Serving API reads** — `Hidden` fields excluded from response
- **Accepting API writes** — `Hidden` and `ReadOnly` rejected

**Orphan-rule safe for extensions:**

- Extension adds new domain `MyDomain`:
  `impl SetAccess<User, ReadWrite> for MyDomain {}` — Self is local ✓
- Extension adds new role `Moderator`:
  `impl SetAccess<Moderator, ReadWrite> for Preferences {}` —
  `Moderator` in type params, Self is concrete foreign type ✓

Example access matrix (expressed as impls, not a runtime table):

| Role | Credentials | Infrastructure | Curation | Preferences |
|------|-------------|----------------|----------|-------------|
| Admin | ReadWrite (default) | ReadWrite (default) | ReadWrite (default) | ReadWrite (default) |
| User | Hidden (default) | Hidden (default) | Hidden (default) | ReadWrite (impl) |
| Child | Hidden (default) | Hidden (default) | Hidden (default) | ReadOnly (impl) |

### 10.9 7-layer precedence

Settings resolution walks layers top-down, returning the first
explicitly-set value per field. Scope determines which layers
are checked (§10.6):

```
1. CLI arguments          (--setting steam.api_key=xxx)
2. Environment variables  (SAALIS_STEAM_API_KEY=xxx)
3. Per-profile overrides  (settings_overrides WHERE profile_id = N)
4. Global DB overrides    (settings_overrides WHERE scope = 'global')
5. DB settings            (settings table, web UI mutable)
6. saalis.toml            (static file on disk)
7. Compiled defaults      (from #[derive(Settings)] field defaults)
```

The derive macro generates resolution logic per field. Each field
knows its scope and thus which layers to check. `Global`-scoped
fields skip layers 3–5. `System`-scoped fields skip layers 3–4.

**Hot-reload:** Layers 3–7 (DB, TOML, defaults) can be hot-
reloaded. The host watches the DB for changes and reconstructs
affected settings structs. Layers 1–2 (CLI, env) are immutable
after startup. Changed settings emit events through the scheduler
bus (§5.5) so affected subsystems can react.

### 10.10 Connector settings integration

`Connector` has `type Settings: Settings`. The host resolves
all layers at `ConnectorContext<C>` construction time:

```rust
impl Connector for Steam {
    type Settings = SteamSettings;

    fn check_prerequisites(&self, ctx: &ConnectorContext<Self>) -> Result<()> {
        // Strongly typed — no string lookup, no runtime type error.
        let key = &ctx.settings().api_key;
        // key is &ApiKey, not &str
        ctx.http::<Get>(&format!("...?key={key}"))?;
        Ok(())
    }
}
```

The plugin never sees the layering. It gets a resolved struct.

### 10.11 What was rejected

- **`PublicSettings` / `CredentialProvider` as separate host-provided
  service traits** — replaced by `type Settings` on `Connector`.
  Credentials are settings with `Credential` type, not a parallel
  system.
- **`per_user = true/false` boolean** — too coarse. Replaced by
  `SettingsDomain` trait + `SetAccess` compile-time impls for
  proper role-based access gating.
- **`const NAME` / `const DISPLAY_NAME` on every trait** — replaced
  by shared `Named` + `Display` via `Registrable` supertrait.
- **String-typed settings values** (`ApiKey` as `String`, `Locale`
  as `String`) — replaced by semantic types implementing
  `SettingsValue`. Types carry their own validation, UI mapping,
  and secret behavior.
- **`secret = true` attribute on settings fields** — redundant when
  the field type implements `Credential`. The type IS the annotation.
- **Config merge at startup** (research doc Option B) — doesn't
  support per-profile overrides or hot-reload.
- **Unified SettingsRegistry with string keys** (research doc
  Option A) — replaced by strongly-typed `#[derive(Settings)]`
  structs.

## §11 User Identity & Credentials

### 11.1 Decision: Users are entities

Users are `EntityType`s. A user is an entity with metadata, the same
as a game or a movie. No special user tables, no parallel data model,
no user-anchored vs content-anchored distinction.

```rust
struct User;

#[register]
impl EntityType for User {}
```

A user is an entity instance with `type_id` pointing to `User`.
Everything about them — name, role, linked identities, play sessions,
currency balance — is metadata.

This unifies the entire data model. The same query API, the same
metadata tables, the same enrichment pipeline, the same relationship
system. One model for everything.

#### 11.1.1 User metadata tables

| Metadata Table | Purpose | Domain columns |
|----------------|---------|----------------|
| `UserProfile` | Display name, avatar | `DisplayName`, `AvatarPath`, `AgeRestriction` |
| `UserRole` | Role assignment | `RoleName` (stores `Named::NAME` of the `Role` type) |
| `LinkedIdentity` | External service identities | `Provider`, `ExternalId`, `DisplayName`, `LinkedAt` |
| `PlaySession` | Game playtime tracking | `GameEntityId`, `StartedAt`, `EndedAt`, `Duration` |
| `CurrencyBalance` | Balance and transaction log | `Balance`, `TransactionType`, `Amount`, `Reason` |
| `Watchlist` | Per-user media tracking | `TargetEntityId`, `AddedAt`, `Notes` |
| `AuthConfig` | Per-user auth configuration | `PinHash`, `AuthMethodId` |

`HasMetadata` impls (§4.5) paint what each entity type carries:

```rust
// Users
impl HasMetadata<UserProfile> for User {}
impl HasMetadata<UserRole> for User {}
impl HasMetadata<LinkedIdentity> for User {}
impl HasMetadata<PlaySession> for User {}
impl HasMetadata<CurrencyBalance> for User {}
impl HasMetadata<Watchlist> for User {}
impl HasMetadata<AuthConfig> for User {}

// Games
impl HasMetadata<Rating> for Game {}
impl HasMetadata<CompatInfo> for Game {}
impl HasMetadata<Media> for Game {}
impl HasMetadata<LinkedIdentity> for Game {}  // external IDs (Steam app ID, IGDB ID)
impl HasMetadata<Description> for Game {}
```

Shared metadata tables associate with multiple entity types when it
makes sense:

```rust
// Both games and users can have media (cover art / avatars)
impl HasMetadata<Media> for Game {}
impl HasMetadata<Media> for User {}

// Both can have linked identities (game's external IDs / user's accounts)
impl HasMetadata<LinkedIdentity> for Game {}
impl HasMetadata<LinkedIdentity> for User {}
```

#### 11.1.2 Security boundary for user metadata

`AuthConfig` (PIN hash) and `SessionData` (session tokens, §18)
are security-sensitive. Plugins must not be able to query them.

**Solution:** Define `AuthConfig` and `SessionData` as
`MetadataTable` ZSTs in `saalis-core`, NOT in `saalis-sdk`. Plugins
can't reference a type outside their dependency graph.
`ctx.query::<AuthConfig>()` is a compile error in plugin code — the
ZST doesn't exist for them.

The query API is strictly type-based (§1.2.3) — no string-based table
access. Core-only metadata tables are invisible to plugins. The type
system IS the security boundary.

Similarly, `UserRole` is defined in core. Plugins can query user
profiles and linked identities (if those ZSTs are in the SDK), but
cannot read or write role assignments.

#### 11.1.3 Auth performance

User lookup happens on every HTTP request. Session token lookup is
an indexed query on the `SessionData` metadata table (§18). The
resolved user (entity ID + profile + role + age restriction) is
cached in-memory. The session token maps to a cached `ResolvedUser`
struct. Re-query metadata only on session creation and explicit
refresh, not on every request.

Cache invalidation ties to the event bus (§5.5) — when a user's
role or profile metadata changes, the event triggers invalidation
of all cached `ResolvedUser` entries for that user's sessions.

### 11.2 Decision: Credentials are settings

No parallel credential system. Credentials are `Credential`-typed
fields in `#[derive(Settings)]` structs. The settings system's
7-layer precedence (§10.9) handles scoping. No `CredentialProvider`
trait, no `linked_credentials` table.

```rust
#[derive(Settings)]
#[settings(section = "screenscraper")]
struct ScreenScraperSettings {
    #[setting(scope = Profile, domain = Credentials)]
    username: Username,  // SettingsValue newtype — not Credential, not Secret

    #[setting(scope = Profile, domain = Credentials)]
    password: ApiKey,  // Credential type → is_secret() = true

    #[setting(domain = Infrastructure)]
    rate_limit: RateLimit,
}
```

Per-user credentials = `Profile`-scoped `Credential` fields. Each
user (entity) can carry their own ScreenScraper password as a profile
override. Global credentials = `System`-scoped. `SetAccess` impls
(§10.8) gate visibility per role.

### 11.3 Decision: Linked identities via metadata

External service identities are metadata on the `User` entity type.
Not a setting, not a secret — a fact about the user.

```rust
// SDK — available to plugins for display/matching
struct LinkedIdentity;

#[register]
impl MetadataTable for LinkedIdentity {}

struct Provider;
struct ExternalId;
struct ProviderDisplayName;
struct LinkedAt;

#[register] impl Column<In<LinkedIdentity>, As<Str>> for Provider {}
#[register] impl Column<In<LinkedIdentity>, As<Str>> for ExternalId {}
#[register] impl Column<In<LinkedIdentity>, As<Str>> for ProviderDisplayName {}
#[register] impl Column<In<LinkedIdentity>, As<i64>>    for LinkedAt {}
```

Discovery of available identity providers uses the same registration
pattern:

```rust
trait LinkedIdentityProvider: Registrable {
    /// Label for the external ID field (e.g., "Steam64 ID")
    const ID_LABEL: &'static str;

    /// Optional: validate the external ID format before storing
    fn validate_id(&self, id: &str) -> Result<()> { Ok(()) }
}

struct SteamIdentity;
#[register]
impl LinkedIdentityProvider for SteamIdentity {}
impl Named for SteamIdentity { const NAME: &'static str = "steam"; }
```

Querying is the standard metadata API:

```rust
// All linked identities for a user
ctx.query::<LinkedIdentity>().of::<User>(user_id).execute()?;

// Filter by provider
ctx.query::<LinkedIdentity>()
    .of::<User>(user_id)
    .filter::<Provider>(Op::Eq, "steam".into())
    .execute()?;
```

`LinkedIdentity` is in the SDK (not core-only) because plugins may
need to read external IDs for enrichment matching — e.g., a Steam
enricher needs the user's Steam64 ID to fetch their library.

Games also use linked identities for cross-source matching:

```rust
impl HasMetadata<LinkedIdentity> for Game {}

// A game's Steam app ID, IGDB ID, etc.
ctx.query::<LinkedIdentity>()
    .of::<Game>(game_id)
    .filter::<Provider>(Op::Eq, "igdb".into())
    .execute()?;
```

### 11.4 Decision: Roles are a registered trait

Roles are an extensible domain concept — extensions can define new
roles (e.g., `Moderator`, `Curator`). Per §6.2: extensible concepts
= registered traits, not closed enums.

```rust
trait Role: Registrable {}

struct Admin;
#[register] impl Role for Admin {}

struct User;
#[register] impl Role for User {}

struct Child;
#[register] impl Role for Child {}
```

First-party roles cover the household appliance use case.
Extensions register additional roles for their domain — a parental
controls plugin might add `Curator`, a multi-tenant plugin might
add `Moderator`.

Role *assignment* is metadata on the `User` entity. The `UserRole`
metadata table (core-only, not SDK) stores the role's `Named::NAME`.

Access control is expressed via `SetAccess` impls (§10.8) — each
`(Role, SettingsDomain)` pair's access level is a compile-time
trait relationship, not a runtime lookup.

### 11.5 Decision: Encryption at application level, both ends

Any settings field whose type implements `Secret` (all `Credential`
types do via blanket impl, §10.2) gets encrypted before DB write,
decrypted after DB read. The settings engine handles this
transparently — connectors never see ciphertext.

```
Write path:  value → SettingsValue::store() → encrypt(master_key) → DB
Read path:   DB → decrypt(master_key) → SettingsValue::parse() → typed value
```

Master key source (checked in order):
1. Environment variable: `SAALIS_MASTER_KEY`
2. OS keychain entry: `saalis-master-key`
3. Auto-generated secret in data directory (first-run, backed up in
   `.saalis` snapshots with `includes_secrets = true`)

`Display` impls for `Credential` types MUST mask values (`"sk-****"`).
This prevents accidental exposure in tracing output, error messages,
debug prints, or notification text.

### 11.6 Decision: Dedicated vault API

Credential serving is a dedicated API namespace, separate from
general settings. Special security considerations warrant isolation.

```
GET  /api/vault/providers
     → all registered LinkedIdentityProviders + Credential-bearing
       settings sections
     → field definitions, no values
     → public (for UI rendering)

GET  /api/vault/{provider}
     → resolved credentials for the active session's user
     → plaintext over local network (same trust model as web UI)
     → requires valid session OR static API key from config
     → audit-logged

GET  /api/vault/{provider}/{user_id}
     → admin-only: specific user's credentials
     → audit-logged

GET  /api/vault/identities/{user_id}
     → linked identities for a user (public info, no secrets)
     → any authenticated user can see their own
     → admin can see all
```

Security measures on all vault endpoints:
- **Rate limiting** — prevent brute-force enumeration of credentials
- **Audit logging** — every credential access logged: who, which
  provider, when, from which IP
- **`Cache-Control: no-store`** — prevent credential caching in
  proxies or browsers
- **Admin-only cross-user access** — non-admin users can only access
  their own credentials
- **Dual auth** — session token via cookie (web UI) OR static API
  key from config (machine-to-machine). Both paths resolve to a
  user context. See §18 for session system details.

### 11.7 Cross-entity relationships

With users as entities, the relationship system (§4.3) handles all
user-content associations:

```
User ──owns──► Game          (library)
User ──played──► Game        (with PlaySession metadata on the edge)
User ──wishlists──► Game     (watchlist)
User ──favorited──► Game     (favorites)
Game ──variant-of──► Game    (editions)
Game ──in-franchise──► Game  (series)
```

One relationship system. One query API. One enrichment pipeline.

The enrichment pipeline works for users: an `Enricher<User>`
connector can fetch a user's Steam profile, RetroAchievements
progress, avatar, and playtime stats — same trait, same pipeline,
same scheduler dispatch as game enrichment.

### 11.8 Profile-scoped settings integration

"Profile" in the settings system (§10.6, §10.9) maps to a `User`
entity. The `settings_overrides` table keys per-profile overrides by
entity ID:

```sql
-- Layer 3: per-profile overrides
SELECT value FROM settings_overrides
WHERE key = 'screenscraper.password'
  AND profile_entity_id = ?;
```

The settings engine resolves `Profile`-scoped fields by looking up
overrides keyed to the active user's entity ID. The active user is
determined by the session (§18). External clients (Batocera scripts,
other machines) authenticate via sessions like any other client —
there is no global "active box user" concept. Work units that need
user context declare this through their trait contracts; the
scheduler provides it from the triggering session or omits it for
system-level operations.

### 11.9 What was rejected

- **`CredentialProvider` trait as a standalone system** — credentials
  are settings fields with `Credential` types. No parallel system.
- **`linked_credentials` table** — superseded by `Profile`-scoped
  `Credential` settings fields in the 7-layer precedence.
- **`linked_identities` as a special host-managed table** — superseded
  by `LinkedIdentity` metadata table on the `User` entity type. Same
  pattern as all other metadata.
- **Separate user tables** — users are entities. No `users` table.
  User data is metadata on entity instances of type `User`.
- **String-typed roles** (`"admin"`, `"user"`, `"child"`) — replaced
  by `Role` registered trait with ZST markers (`Admin`, `User`, `Child`).
- **Roles as closed `repr(C)` enum** — extensions need to define
  their own roles. Extensible domain concept = registered trait (§6.2).
- **`RoleAuthority` runtime mapper** — replaced by `SetAccess<R, A>`
  compile-time trait impls with specialization defaults (§10.8).
- **SQLCipher (encrypted SQLite)** — too heavy. Application-level
  encryption for secret fields only.
- **Per-user PIN key derivation** — PINs are optional and short.
  Master key is more reliable and recoverable.
- **OAuth in v1** — deferred. Manual credential entry (API keys,
  username/password) covers all current providers. OAuth flow
  (callback URLs, token refresh, consent screens) can be added later
  as an `AuthMethod` trait impl (§18) without architectural changes.
- **Global "active box user" / box status** — saalis is
  platform-agnostic, not tied to a single appliance. External clients
  use sessions like any other client. Work unit context requirements
  are expressed through trait contracts; settings scoping (§10.6)
  determines whether user context is needed.
- **`AuthData` combining PIN hash and session tokens** — conflates
  per-user auth configuration with per-session state. Split into
  `AuthConfig` (on User) and `SessionData` (on Session entity, §18).

## §12 Download Engine

### 12.1 Decision: Downloads are transient entities that fold on completion

A download is an entity while active. It has metadata (progress,
speed, source URL, lane assignment), relationships
(`Download ──targets──► Game`), and participates in the query API
and notification system like any other entity.

```rust
struct Download;

#[register]
impl EntityType for Download {}

impl HasMetadata<DownloadProgress> for Download {}
impl HasMetadata<DownloadSource> for Download {}
```

When a download completes — fully downloaded, extracted, verified —
it performs a **completion fold**: the host creates permanent
`DownloadHistory` metadata on the target entity and deletes the
Download entity.

```
Active:   Download entity (id=500, type=Download)
            ──targets──► Game entity (id=42)
            metadata: DownloadProgress, DownloadSource

Folded:   Download entity deleted
          Game entity (id=42) gains DownloadHistory metadata:
            source_url, downloaded_at, file_size, checksum, source_connector
```

**Completion fold** is the lifecycle pattern where a transient entity
completes its purpose and folds into permanent metadata on its
target. The entity ceases to exist; its essential record persists as
facts about the target. This pattern may apply to other transient
operations that produce lasting records.

The fold is a host-managed operation. The scheduler's completion
handler for download work units performs the fold: create metadata
row, delete entity, emit `DownloadFolded` event.

#### 12.1.1 Download metadata tables

**While active (on Download entity):**

| Metadata Table | Purpose | Columns |
|----------------|---------|---------|
| `DownloadProgress` | Live progress tracking | `BytesDownloaded`, `TotalBytes`, `Lane`, `StartedAt` |
| `DownloadSource` | Active source info | `SourceUrl`, `SourceConnector`, `Checksum` |
| `DownloadStatus` | State machine state | `Status`, `RetryCount`, `Error`, `PausedAt` |

**After settling (on target entity, e.g., Game):**

| Metadata Table | Purpose | Columns |
|----------------|---------|---------|
| `DownloadHistory` | Permanent provenance record | `SourceUrl`, `SourceConnector`, `DownloadedAt`, `FileSize`, `Checksum`, `ExtractionManifest` |

`DownloadHistory` is always retained. A household appliance
accumulates at most a few thousand downloads over its lifetime —
negligible for SQLite. History has permanent value: redownload
decisions, source quality tracking, provenance auditing.

### 12.2 Decision: Source-type dispatch via `Downloader<S>` trait

Different source types need different download strategies. The
download engine dispatches via a registered `Downloader<S>` trait:

```rust
trait Downloader<S: SourceType>: Registrable {
    /// Execute the download. Called on a worker thread.
    ///
    /// `cancel`: checked between chunks for pause/cancel.
    /// Progress reported via `ctx.progress(downloaded, total)` — fire-and-forget.
    /// Download metadata (path, checksum, etc.) written via `ctx.write()`.
    fn download(
        &self,
        ctx: &DownloadContext,
        dest: &Path,
        cancel: &AtomicBool,
    ) -> Result<()>;
}
```

The downloader writes download metadata through `ctx.write()` — no
`DownloadOutcome` return struct needed (see §19.1). Source data is
queried through the context, not passed as a `ResolvedSource` parameter.

First-party downloaders:

```rust
struct HttpDownloader;

#[register]
impl Downloader<DirectDownload> for HttpDownloader {}
// Uses ctx.http (ureq) with Range resume, streaming, chunk-level cancel

struct FileCopyDownloader;

#[register]
impl Downloader<LocalFile> for FileCopyDownloader {}
// std::fs::copy, cancel check via periodic poll

struct DirectorySyncDownloader;

#[register]
impl Downloader<LocalDirectory> for DirectorySyncDownloader {}
// Recursive copy/sync
```

Plugin-contributed source types come with their own `Downloader`:

```rust
struct SteamCmdDownloader;

#[register]
impl Downloader<SteamApp> for SteamCmdDownloader {}
// Shells out to steamcmd or launches steam:// protocol
```

The download engine queries the `DownloaderRegistry` for the
appropriate `Downloader<S>` given the `SourceTypeDescriptor`. If no
downloader is registered for a source type, the download fails with
a `Fixable` error and guidance ("No downloader available for source
type 'magnet'").

### 12.3 Decision: ArchiveFormat + Extractor\<F\> traits

The format IS the thing — it knows its file extensions, and the
extractor is parameterized by format. Same pattern as `SourceType` →
`Downloader<S>`.

```rust
/// Registered trait — each archive format is a type.
trait ArchiveFormat: Registrable {
    /// File extensions at the filesystem boundary (strings are
    /// unavoidable here — filenames are inherently strings).
    const EXTENSIONS: &'static [&'static str];
}

struct Zip;
impl Named for Zip { const NAME: &'static str = "zip"; }
#[register]
impl ArchiveFormat for Zip {
    const EXTENSIONS: &'static [&'static str] = &["zip"];
}

struct SevenZip;
impl Named for SevenZip { const NAME: &'static str = "7z"; }
#[register]
impl ArchiveFormat for SevenZip {
    const EXTENSIONS: &'static [&'static str] = &["7z"];
}

struct Tar;
impl Named for Tar { const NAME: &'static str = "tar"; }
#[register]
impl ArchiveFormat for Tar {
    const EXTENSIONS: &'static [&'static str] = &["tar", "tar.gz", "tar.zst", "tgz"];
}
```

Extractor is parameterized by format — the host matches file
extension → `ArchiveFormat` via registry scan, then dispatches to the
registered `Extractor<F>`:

```rust
trait Extractor<F: ArchiveFormat>: Registrable {
    /// Progress reported via `ctx.progress(extracted, total)` — fire-and-forget.
    /// Extracted file metadata written via `ctx.write()`.
    fn extract(
        &self,
        ctx: &ExtractionContext,
        archive: &Path,
        dest: &Path,
    ) -> Result<()>;
}
```

The extractor writes extraction metadata through `ctx.write()` — no
`ExtractedFile` struct or `Manifest` associated type needed (see §19.1).
Same fire-and-forget write pattern as all other pipeline stages:

```rust
fn extract(&self, ctx: &ExtractionContext, archive: &Path, dest: &Path) -> Result<()> {
    for entry in archive_entries {
        // extract file to dest
        ctx.write::<ExtractedEntry>(entity_id, |row| {
            row.set::<FilePath>(path);
            row.set::<FileSize>(size);
            row.set::<FileChecksum>(checksum);
        });
    }
    Ok(())
}
```

First-party extractors:

```rust
struct ZipExtractor;

#[register]
impl Extractor<Zip> for ZipExtractor {}

struct SevenZipExtractor;

#[register]
impl Extractor<SevenZip> for SevenZipExtractor {}

struct TarExtractor;

#[register]
impl Extractor<Tar> for TarExtractor {}
```

Plugins can contribute formats AND extractors (e.g., RAR support
without bundling a restrictive-license native library in core):

```rust
struct Rar;
impl Named for Rar { const NAME: &'static str = "rar"; }
#[register]
impl ArchiveFormat for Rar {
    const EXTENSIONS: &'static [&'static str] = &["rar"];
}

struct RarExtractor;

#[register]
impl Extractor<Rar> for RarExtractor {}
// extensions: ["rar"] — shells out to system `unrar` binary
```

**Final-format detection:** Files that don't match any registered
extractor's extensions are assumed to be final format (`.chd`, `.cso`,
`.zso`, `.iso`, `.bin`, `.cue`, etc.) and skip extraction entirely.
This is determined at runtime from the `ExtractorRegistry` — no
hardcoded extension list.

**Extraction as a separate work unit:** Download completion emits
`DownloadComplete`. `ExtractWork` triggers on this event, runs
extraction via the matched `Extractor`, and emits
`ExtractionComplete`. The download lane is freed immediately when the
HTTP transfer finishes — extraction runs on the scheduler's thread
pool, not in a download slot.

### 12.4 Decision: Status state machine

```
queued → downloading → complete → [ExtractWork via scheduler]
              ↓                          ↓
           paused                    extracted → [completion fold]
              ↓
           shelved ←───────────────── failed (after max retries)
              ↓                          ↓
           queued (unshelve)          queued (retry, up to max)
                                         ↓
                                      cancelled (user-initiated, terminal)
```

| Status | Meaning | Transitions to |
|--------|---------|----------------|
| `Queued` | Waiting for a free lane | `Downloading` |
| `Downloading` | Active HTTP transfer in a lane | `Complete`, `Paused`, `Failed` |
| `Paused` | Temporarily stopped, lane released, partial file kept | `Queued` (resume) |
| `Shelved` | Indefinitely set aside by user. Sits at end of queue, inactive. Not terminal — can be unshelved. | `Queued` (unshelve) |
| `Complete` | HTTP transfer done, awaiting extraction | `Extracted` (via ExtractWork) |
| `Extracted` | Fully processed, ready to fold | Folded (entity deleted, history written) |
| `Failed` | Error after exhausting retries | `Queued` (manual retry), `Shelved`, `Cancelled` |
| `Cancelled` | User-initiated termination. Terminal. | — |

**`Shelved`** is distinct from `Paused`:
- `Paused` = "stop temporarily, I'll be right back" — stays in queue
  position, auto-resumes if user hits "resume all"
- `Shelved` = "I don't know when I'll get to this" — moved to end of
  queue, skipped by the dispatcher, only reactivated by explicit user
  action ("unshelve")

**`Failed` is not terminal.** Failed downloads can be manually
retried (back to `Queued`) or shelved. Only `Cancelled` is terminal
(and even then, the entity can be deleted by the user or by
housekeeping).

Per-download pause: save `bytes_downloaded` in `DownloadProgress`
metadata, release lane, set status to `Paused`. Resume: set status
back to `Queued`. The existing partial file + HTTP Range header
handles continuation automatically.

Cancellation: set the `AtomicBool` cancel flag, which the downloader
checks between chunks. The in-flight HTTP read returns early with a
cancellation error. Partial file is deleted for cancelled downloads
(unlike pause, which preserves it).

### 12.5 Decision: Lane management

Thread pool with configurable capacity (default 4 lanes).
`std::thread::spawn` per download, not tokio tasks.

```rust
struct DownloadEngine {
    semaphore: Semaphore,  // std parking_lot semaphore, capacity = lanes
    cancel_flags: DashMap<EntityId, Arc<AtomicBool>>,
}
```

The dispatcher watches for `Queued` downloads via scheduler events
(not polling). When a lane is free and a queued download exists:

1. Acquire semaphore permit
2. Create `AtomicBool` cancel flag for this download
3. Update download entity status to `Downloading`, assign lane number
4. Spawn worker thread:
   a. Look up `Downloader<S>` from registry by source type
   b. Call `downloader.download()` with `DownloadContext` and cancel flag
   c. Plugin calls `ctx.progress()` inside download loop — fire-and-forget
      work units flow through scheduler → in-memory progress map → SSE
   d. On completion: emit `DownloadComplete` event
   e. On failure: update retry count, emit failure event
5. Permit released on thread exit (Drop)

Lane numbers are assigned from the semaphore permit — each permit
corresponds to a lane (1–N). The UI displays active downloads by lane.

**Recovery on startup:** Any download entities with status
`Downloading` are reset to `Queued`. The partial file (if present)
enables Range-header resume on the next attempt.

### 12.6 Decision: Progress reporting

Progress uses the same fire-and-forget pattern as other context
services. The plugin calls `ctx.progress(downloaded, total)` inside
its download loop. This drops a lightweight progress work unit into
the scheduler — the plugin never blocks.

```
Downloader chunk read
  → ctx.progress(bytes_downloaded, total_bytes)
  → fire-and-forget work unit → scheduler
  → scheduler coalesces updates, refreshes in-memory progress map
  → SSE push to connected clients (throttled to ~2/sec)
```

The scheduler naturally coalesces progress work units — if multiple
queue up before the next cycle, only the latest values matter. This
gives throttling as a property of scheduler cadence, not a separate
mechanism.

The `DownloadProgress` metadata table in the DB is updated
periodically (every ~5 seconds or on status change), not on every
chunk. This keeps the DB write load low while the UI gets real-time
updates via SSE.

Speed and ETA are computed client-side from the progress snapshots
(current bytes + timestamp). No server-side speed tracking needed —
the client has the same information and can compute rolling averages
locally.

### 12.7 Decision: Pipeline integration

The download engine is part of the scheduler DAG (§5):

```
SourceResolved event
  → host creates Download entity (type=Download)
  → sets relationship: Download ──targets──► target entity
  → sets status: Queued
  → Download entity metadata: DownloadSource (url, connector, checksum)
  → scheduler dispatches to download engine

DownloadComplete event
  → ExtractWork triggers (if needs_extraction)
  → Extractor matched by file extension from registry
  → extraction runs on scheduler thread pool
  → ExtractionComplete event emitted

ExtractionComplete event (or DownloadComplete if no extraction needed)
  → completion fold: DownloadHistory metadata created on target entity
  → Download entity deleted
  → DownloadFolded event emitted
```

Priority inheritance: the download inherits the priority of the work
unit that produced the `SourceResolved` event. User-initiated
downloads ("download this game now") get `Immediate` priority.
Background catalogue sweeps produce `Normal` or `Low` priority
downloads.

### 12.8 What was rejected

- **Downloads as pure infrastructure** (not entities) — loses the
  query API, relationship system, and notification integration for
  no benefit. Downloads have metadata, relationships, and lifecycle
  states — they are entities.
- **Hardcoded source-type dispatch** — closed set, plugins can't
  contribute new download strategies. `Downloader<S>` trait follows
  the universal registration pattern.
- **Hardcoded extraction** — same reasoning. `Extractor` trait,
  registered, extensible. Plugin-contributed format support without
  core modifications.
- **Multi-source parallel download** — would trigger rate-limit bans
  on most ROM sources. Serial source rotation (failure → blacklist →
  try next) is the correct approach.
- **Download history pruning** — history has permanent value and
  negligible storage cost. Always retained as metadata on the target
  entity.
- **tokio-based download engine** — superseded by sync-core
  architecture. `std::thread::spawn` + `ureq` + `AtomicBool`
  cancellation.
- **DB-per-chunk progress writes** — excessive I/O. Progress flows
  through `flume` channels to SSE. DB updated periodically.
- **Bandwidth limiting as architectural concern** — implementation
  detail. The `HttpClient` host implementation can enforce rate
  limits if configured. Not a download engine design decision.

## §13 Transactional Consistency

### 13.1 Decision: Three-layer consistency model

Different operations need different consistency guarantees. Three
layers, composable:

| Layer | Mechanism | When |
|-------|-----------|------|
| **SeaORM transactions** | `db.transaction()` | Pure-DB atomicity: scheduler batch flushes, user-initiated batch operations |
| **Transient entities** | Entity with state machine + completion fold | Multi-step operations spanning DB + filesystem + time |
| **Accept-and-recover** | Scheduler retry + deferred queue (§5) | Background tasks where partial results are acceptable |

### 13.2 Decision: SeaORM transactions for pure-DB atomicity

The scheduler's batch flush wraps each batch in a SeaORM transaction.
If flushing 50 rating writes fails on row 30, all 50 roll back. This
is internal to the host's flush logic.

User-initiated atomic operations (e.g., "queue these 10 games, deduct
currency for all or none") are also single SeaORM transactions. The
entire batch succeeds or fails atomically. User batches are small
enough to fit in a single transaction without write-lock concerns.

Background batch processing (enriching 500 games) uses
`max_transaction_chunk` (§5.2) to chunk into smaller transactions,
releasing the SQLite write-lock between chunks to prevent starvation.

### 13.3 Decision: Transient entities subsume intent logging

The entity model + completion fold (§12.1) naturally provides intent
logging for multi-step operations. No separate `operation_log` table
is needed.

A transient entity IS the intent record:
- **Creation** = recording the intent
- **Metadata** = tracking progress and operation-specific state
- **Status state machine** = tracking the operation phase
- **Completion fold** = recording the permanent result
- **Startup recovery** = scanning for in-progress transient entities

Examples:

| Operation | Transient entity | Folds into |
|-----------|-----------------|------------|
| Download | `Download` entity with progress metadata | `DownloadHistory` on target game |
| Import | `Import` entity tracking processed rows | Import summary on user/collection |
| Extraction | Part of Download lifecycle (ExtractWork) | `ExtractionManifest` on target |
| Dedup merge | `Merge` entity recording source/target entities | Merge audit on surviving entity |

If the process crashes mid-operation, the transient entity persists.
On startup, the scheduler discovers entities with in-progress status
and resumes or retries them. The entity IS the recovery mechanism.

### 13.4 Decision: DB-first ordering for mixed operations

When an operation spans both DB and filesystem:

1. **Create transient entity (DB)** — records the intent with source
   path, destination path, operation type
2. **Perform filesystem operation** — entity metadata tracks what
   was done
3. **Update entity status (DB)** — records completion
4. **Completion fold** — permanent metadata on target, entity deleted

If filesystem fails: entity stays in current state, scheduler retries.
If DB update fails after filesystem succeeds: entity persists,
startup recovery verifies filesystem state and reconciles.

There is no window where the operation is lost — the entity exists
from intent to fold. The only risk is a crash between entity creation
(step 1) and filesystem operation (step 2), which is harmless — the
entity records the intent, startup recovery replays it.

### 13.5 Decision: Write ordering from FK graph

Write ordering is automatic (§1.2.4). The scheduler reads
`ReferenceDescriptor`s (derived from `References<T>` trait impls)
to build the FK dependency graph. Batch flushes proceed in
topological order: parent tables before child tables (e.g., `entities`
before `ratings`).

This eliminates manual dependency declarations. Adding a new metadata
table with a `References` impl automatically places it in the correct
flush order.

### 13.6 What was rejected

- **`operation_log` table** — superseded by transient entities. The
  entity model already provides intent recording, state tracking,
  and crash recovery. A separate log table would be redundant.
- **Saga pattern with compensating actions** — compensations can
  fail (turtles all the way down). Transient entity state machines
  handle multi-step operations more cleanly: if a step fails, the
  entity stays in its current state and the scheduler retries.
- **Manual `DependsOn<T>` for write ordering** — FK graph from
  `References` trait impls provides the same information
  automatically (§1.2.4).
- **Explicit `Executor` trait in SDK** — core-internal concern.
  Plugins use `ctx.query()` and `ctx.write()`, never raw SQL or
  transaction handles.
- **All-or-nothing for background batches** — background enrichment
  of 500 games should not fail entirely because game #237 had a
  transient error. Partial results + retry is the correct model.
  All-or-nothing is reserved for user-initiated operations where
  the user expects atomic feedback.

## §14 Filesystem Abstraction

### 14.1 Decision: Typed storage locations (ZST markers + trait composition)

Filesystem locations are ZST markers registered via `#[register]`,
following the same pattern as `EntityType` (§4) and all other
registered traits (§1). Plugins address filesystem locations by
type, never by raw path. The type system constrains what a plugin
can access.

**Identity via `Named`.** Location markers use the existing `Named`
trait (§1.1) for their machine-readable identifier — same as every
other registered type. No `const ID` field; identity comes from
the trait system.

**Capabilities via marker traits.** Whether a location is managed
or redirectable is expressed as trait impls, not const bools.
Supertrait constraints ensure only valid combinations compile:

```rust
/// Base marker trait for a named filesystem location.
/// Each implementor is a ZST, registered via `#[register]`.
/// Identity comes from `Named` (§1.1).
trait StorageLocation: Named + Registrable {}

/// Saalis owns the lifecycle of files at this location.
/// Only `StorageLocation` types can be managed.
/// Enables: install, remove, write operations.
trait Managed: StorageLocation {}

/// This location can be redirected to another volume via symlink
/// (storage target pattern). Only managed locations can be
/// redirected — you can't redirect a location you don't own.
/// Enables: symlink/redirect operations.
trait Redirectable: Managed {}
```

Compile-time enforcement: `ctx.fs::<Saves>().install(...)` is a
compile error — `Saves` doesn't impl `Managed`, so `.install()`
does not exist on that handle. No runtime check, no error variant,
no possibility of misuse.

**First-party locations:**

| Marker | Traits | Purpose |
|--------|--------|---------|
| `Roms` | `StorageLocation + Managed + Redirectable` | Installed content files |
| `Temp` | `StorageLocation + Managed + Redirectable` | In-flight downloads, extraction scratch |
| `Saves` | `StorageLocation` | Game save data (external, read-only to saalis) |
| `Cache` | `StorageLocation + Managed` | Enrichment cache, thumbnails |
| `Themes` | `StorageLocation + Managed` | Installed theme assets |
| `PluginData` | `StorageLocation + Managed` | Per-plugin data directory (auto-scoped) |

```rust
struct Roms;

#[register]
impl StorageLocation for Roms {}
impl Managed for Roms {}
impl Redirectable for Roms {}

impl Named for Roms {
    const NAME: &'static str = "roms";
}
impl fmt::Display for Roms {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "ROM Storage")
    }
}

struct Saves;

#[register]
impl StorageLocation for Saves {}
// No Managed impl — saalis cannot write here.
// No Redirectable impl — can't redirect what you don't own.

impl Named for Saves {
    const NAME: &'static str = "saves";
}
```

Plugins may register custom `StorageLocation` markers for their
own data directories. The host resolves the physical path based
on the `Named::NAME` identifier and the current configuration.

### 14.2 Decision: `Storage` trait contract (dyn-free, bound-gated)

The `Storage` trait defines filesystem operations parameterized
by location type. Generic methods make the trait deliberately
non-object-safe — no `dyn Storage`, no vtable dispatch. The host
provides a concrete `RealStorage` implementation; all calls are
monomorphized.

Methods are gated by trait bounds on the location parameter.
The compiler enforces that write operations are only callable
on `Managed` locations and redirect operations only on
`Redirectable` locations.

```rust
/// Filesystem operations, parameterized by location.
/// Non-object-safe by design — monomorphized at each call site.
///
/// Methods are grouped by required capability bound:
/// - StorageLocation: read-only discovery and inspection
/// - Managed: write operations (saalis owns lifecycle)
/// - Redirectable: volume redirect via symlink
trait Storage {
    // ── Any StorageLocation (read-only) ──────────────────

    /// Resolved root path for a location.
    fn root<L: StorageLocation>(&self) -> PathBuf;

    /// Resolve a name within this location to its canonical path,
    /// following symlinks.
    fn resolve<L: StorageLocation>(&self, name: &str) -> PathBuf;

    /// Available space on the volume backing this location.
    /// Uses `nix::sys::statvfs`.
    fn available_space<L: StorageLocation>(&self) -> Result<u64>;

    /// Read file contents.
    fn read<L: StorageLocation>(
        &self, name: &str,
    ) -> Result<impl AsRef<[u8]>>;

    /// Walk the location directory up to `max_depth`, returning
    /// file paths relative to the location root.
    fn walk<L: StorageLocation>(
        &self, max_depth: usize,
    ) -> Result<impl IntoIterator<Item = PathBuf>>;

    /// Check if a named file exists at this location.
    fn exists<L: StorageLocation>(&self, name: &str) -> bool;

    /// File metadata (size, mtime, is_symlink).
    fn file_metadata<L: StorageLocation>(
        &self, name: &str,
    ) -> Result<FileMetadata>;

    /// Validate that a path does not escape the location root
    /// (no `..` traversal, no absolute escape).
    fn is_safe(&self, path: &Path) -> bool;

    // ── Managed locations (write operations) ─────────────

    /// Move a file into this location (rename or cross-device copy).
    fn install<L: Managed>(
        &self, from: &Path, name: &str,
    ) -> Result<PathBuf>;

    /// Remove a file. `preserve_saves` triggers save-data detection
    /// heuristics before deletion.
    fn remove<L: Managed>(
        &self, name: &str, preserve_saves: bool,
    ) -> Result<()>;

    /// Write file contents (creates parent dirs as needed).
    fn write<L: Managed>(
        &self, name: &str, contents: &[u8],
    ) -> Result<()>;

    // ── Redirectable locations (volume redirect) ─────────

    /// Create a symlink redirecting this location to a path on
    /// another volume (the storage target pattern).
    fn redirect<L: Redirectable>(
        &self, target: &Path,
    ) -> Result<()>;
}
```

The host implementation in `saalis-core`:

```rust
pub struct RealStorage {
    /// Location name → resolved root path
    roots: HashMap<&'static str, PathBuf>,
    /// Volume info cache for space queries
    volumes: HashMap<PathBuf, VolumeInfo>,
}

impl Storage for RealStorage {
    fn root<L: StorageLocation>(&self) -> PathBuf {
        self.roots[L::NAME].clone()
    }

    fn install<L: Managed>(
        &self, from: &Path, name: &str,
    ) -> Result<PathBuf> {
        let dest = self.roots[L::NAME].join(name);
        utils::fs::safe_move(from, &dest)
    }

    // ...
}
```

**Plugin ABI boundary.** Cross-`dlopen` calls use a `repr(C)`
`StorageDescriptor` with `extern "C"` function pointers, same
pattern as `ConnectorDescriptor` (§1.1). The plugin SDK wraps the
descriptor in a typed `Storage`-conforming wrapper. Within the
host binary, first-party code calls `RealStorage` directly —
fully monomorphized, zero indirection.

### 14.3 Decision: `LocationHandle` ergonomic wrapper (bound-gated methods)

`LocationHandle<L, S>` is a zero-cost wrapper that captures the
location type parameter. Conditional `impl` blocks expose methods
only when `L` satisfies the required capability bound — the
compiler removes methods that don't apply.

```rust
pub struct LocationHandle<'a, L: StorageLocation, S: Storage> {
    storage: &'a S,
    _marker: PhantomData<L>,
}

// ── Read-only operations (any StorageLocation) ──────────

impl<L: StorageLocation, S: Storage> LocationHandle<'_, L, S> {
    pub fn root(&self) -> PathBuf {
        self.storage.root::<L>()
    }

    pub fn read(&self, name: &str) -> Result<impl AsRef<[u8]>> {
        self.storage.read::<L>(name)
    }

    pub fn walk(&self, max_depth: usize) -> Result<impl IntoIterator<Item = PathBuf>> {
        self.storage.walk::<L>(max_depth)
    }

    pub fn exists(&self, name: &str) -> bool {
        self.storage.exists::<L>(name)
    }

    pub fn resolve(&self, name: &str) -> PathBuf {
        self.storage.resolve::<L>(name)
    }

    pub fn file_metadata(&self, name: &str) -> Result<FileMetadata> {
        self.storage.file_metadata::<L>(name)
    }

    pub fn available_space(&self) -> Result<u64> {
        self.storage.available_space::<L>()
    }
}

// ── Write operations (Managed locations only) ───────────

impl<L: Managed, S: Storage> LocationHandle<'_, L, S> {
    pub fn install(&self, from: &Path, name: &str) -> Result<PathBuf> {
        self.storage.install::<L>(from, name)
    }

    pub fn remove(&self, name: &str, preserve_saves: bool) -> Result<()> {
        self.storage.remove::<L>(name, preserve_saves)
    }

    pub fn write(&self, name: &str, contents: &[u8]) -> Result<()> {
        self.storage.write::<L>(name, contents)
    }
}

// ── Redirect operations (Redirectable locations only) ───

impl<L: Redirectable, S: Storage> LocationHandle<'_, L, S> {
    pub fn redirect(&self, target: &Path) -> Result<()> {
        self.storage.redirect::<L>(target)
    }
}
```

Context integration:

```rust
impl<C: Connector> ConnectorContext<C> {
    pub fn fs<L: StorageLocation>(&self) -> LocationHandle<'_, L, RealStorage> {
        LocationHandle {
            storage: &self.storage,
            _marker: PhantomData,
        }
    }
}

// Compiles — Roms is Managed + Redirectable:
ctx.fs::<Roms>().install(downloaded_file, "Game (USA).iso")?;
ctx.fs::<Roms>().redirect(Path::new("/mnt/hdd/roms"))?;

// Compiles — Temp is Managed:
ctx.fs::<Temp>().write("scratch.bin", &data)?;

// Compiles — read-only, available on all locations:
let saves = ctx.fs::<Saves>().walk(2)?;
let free = ctx.fs::<Saves>().available_space()?;

// COMPILE ERROR — Saves is not Managed:
// ctx.fs::<Saves>().install(file, "save.srm");
//                   ^^^^^^^ method not found

// COMPILE ERROR — Cache is not Redirectable:
// ctx.fs::<Cache>().redirect(path);
//                   ^^^^^^^^ method not found
```

### 14.4 Decision: Utility functions (core-internal)

Standalone helper functions in `saalis-core/utils/fs.rs`. Not in
the SDK, not trait-ified. Used internally by `RealStorage` and
other core components.

```rust
/// Resolve a path through symlinks to the real location.
pub fn resolve_real_path(path: &Path) -> PathBuf;

/// Move a file: atomic rename, fallback to copy+delete for
/// cross-device moves.
pub fn safe_move(from: &Path, to: &Path) -> Result<PathBuf>;

/// Delete a file, logging on failure. For cleanup paths where
/// failure is acceptable.
pub fn remove_best_effort(path: &Path);

/// Check for path traversal (no `..`, no absolute escape).
pub fn is_safe_path(path: &Path) -> bool;

/// Walk a directory up to max_depth, yielding file paths.
pub fn walk_files(root: &Path, max_depth: usize) -> impl Iterator<Item = PathBuf>;

/// Volume info via nix::sys::statvfs (no unsafe).
pub fn statvfs(path: &Path) -> Result<VolumeInfo>;

/// Pre-allocate a reserve file via fallocate(2).
pub fn create_reserve(path: &Path, size: u64) -> Result<()>;

/// Delete a reserve file (emergency space recovery).
pub fn delete_reserve(path: &Path) -> Result<()>;
```

**`nix` crate for POSIX operations.** Replaces raw `unsafe`
`libc::statvfs` with safe `nix::sys::statvfs::statvfs()`.
Also provides safe `fallocate` for reserve file creation.
No `unsafe` code in the filesystem layer.

### 14.6 Decision: No MemFs

In-memory filesystem mocks are complex to implement correctly
(path resolution, directory semantics, permissions, symlinks)
and provide less confidence than real filesystem tests.

**Testing strategy:**
- Integration tests use `tempdir` with real filesystem operations
- `TestStorage` implementation wraps real temp directories but
  records all operations for assertion (install log, remove log)
- Tests verify both outcomes (files exist) and operations
  (install was called with correct location + name)

### 14.7 What was rejected

- **Full `Filesystem` trait wrapping `std::fs`** — too large an
  API surface, leaky abstraction. Filesystem operations don't map
  cleanly to a trait (permissions, race conditions, platform
  differences). The domain-level `Storage` trait is the right
  boundary.
- **`MemFs` / `vfs` crate** — complex to implement correctly,
  provides less confidence than real filesystem tests. `tempdir`
  is simpler and more realistic.
- **Raw `&Path` in plugin API** — plugins should not construct
  arbitrary paths. Typed `StorageLocation` markers constrain
  what a plugin can address. The host resolves physical paths
  internally.
- **`&dyn Storage`** — violates the architecture's dyn-free
  principle. `Storage` has generic methods (parameterized by
  `L: StorageLocation`), making it deliberately non-object-safe.
  The host monomorphizes all calls. Plugin ABI uses `repr(C)`
  descriptors, same as every other cross-boundary call (§1.1).
- **Raw `unsafe libc::statvfs`** — replaced by `nix` crate's
  safe wrapper. No `unsafe` code in the filesystem layer.
- **Separate `ScanOps` trait** — splitting read/write into
  separate traits fragments the API. The `Storage` trait is
  unified; compile-time access control comes from the `Managed`
  and `Redirectable` capability traits on the location marker,
  not from splitting the operation trait.
- **`const MANAGED: bool` / `const REDIRECTABLE: bool`** —
  runtime const bools pretending to be type information. Whether
  a location is managed or redirectable is a type-level fact
  expressed via trait impls (`Managed`, `Redirectable`), not
  runtime data. The compiler enforces valid access at compile
  time; invalid operations are not expressible, not just checked.

## §15 Presentation Layer

### 15.1 Decision: DataViews as typed query projections

A DataView is not a separate presentation abstraction — it IS a
typed query over the entity model, with presentation metadata
derived from the column ZSTs. No `DataViewRead` / `DataViewWrite`
traits, no presentation structs, no manual `fields()` / `values()`
impls. The column types that define the schema carry enough
information to auto-generate the presentation.

```rust
// The DataView builder composes from column ZSTs:
let view = DataView::for_entity::<Game>("browse")
    .col::<Title>()                     // text, sortable
    .col::<Rating>()                    // number, sortable
    .col::<Genre>().filterable()        // badge, filterable
    .col::<CoverArt>().render_as(Image) // image
    .from_params(&url_params)           // user sort/filter/page
    .execute(&ctx)?;                    // query + build
```

**How `.col::<M>()` works.** Each `MetadataTable` type has columns
defined via `Column<In<M>, As<V>>` ZSTs (§4). The column's `As<V>`
type maps to a field type automatically:

| Column type (`As<V>`) | Auto-detected field type |
|------------------------|-------------------------|
| `As<Str>` | Text (sortable, not filterable) |
| `As<f64>` / `As<i64>` | Number (sortable) |
| `As<bool>` | Boolean (sortable, filterable) |

The metadata type's `Named::NAME` provides the machine-readable
ID; `Display` provides the human-readable label. Presentation
modifiers on the builder override these defaults.

**How `.execute()` works.** The builder:
1. Reads column metadata from the configured ZSTs
2. Constructs a multi-metadata join query against the entity table
3. Applies entity type filter (`WHERE entity_type IN (...)`)
4. Applies user-facing sort, filter, and pagination from URL params
5. Fetches the data
6. Builds the template-ready `DataView` payload

Under the hood, the query is a join across metadata tables:

```sql
SELECT e.id, t.value AS title, r.score AS rating, g.value AS genre
FROM entities e
LEFT JOIN title t ON t.entity_id = e.id
LEFT JOIN rating r ON r.entity_id = e.id
LEFT JOIN genre g ON g.entity_id = e.id
WHERE e.entity_type IN ('game')
  AND r.score > 8              -- user filter
ORDER BY r.score DESC          -- user sort
LIMIT 50 OFFSET 100           -- pagination
```

The DataView builder generates this from the type parameters. No
hand-written SQL in the web layer.

### 15.2 Decision: Multi-entity views with `Any<(T1, T2)>`

DataViews can show entities of multiple types using a type-level
union filter:

```rust
// Universal browse — Games AND Movies
DataView::new("browse")
    .col::<Title>()
    .col::<Rating>()
    .entities::<Any<(Game, Movie)>>()
    .from_params(&params)
    .execute(&ctx)?;
```

`Any<(Game, Movie)>` generates `WHERE entity_type IN ('game',
'movie')`. The compiler enforces that only metadata types
registered for ALL entity types in the union are valid columns.
Adding `.col::<RomFormat>()` is a compile error if `Movie` has
no `RomFormat` metadata.

Single-entity views use the simpler `for_entity::<Game>()` form,
which is sugar for `.entities::<Game>()`.

### 15.3 Decision: Presentation modifiers on the builder

The builder supports chainable modifiers that override the
auto-detected defaults:

```rust
DataView::for_entity::<Game>("library")
    .col::<Title>()
    .col::<FileSize>().format("bytes")     // "1.2 GB" not "1234567890"
    .col::<Genre>().filterable()            // enable filter dropdown
    .col::<CoverArt>().render_as(Image)     // render URL as <img>
    .col::<SystemId>().label("Platform")    // override display name
    .col::<InternalScore>().hidden()        // fetch but don't display
    .card_mode()                            // default to card layout
    .page_size(48)                          // items per page
    .group_by::<Genre>()                    // visual grouping
    .execute(&ctx)?;
```

| Modifier | Effect |
|----------|--------|
| `.filterable()` | Add filter dropdown for this column |
| `.render_as(Image)` | Override field type to Image |
| `.format("bytes")` | Display format hint |
| `.label("...")` | Override display name |
| `.hidden()` | Include in data, don't render |
| `.no_sort()` | Disable sort on this column |
| `.card_mode()` | Default renderer: cards not rows |
| `.page_size(n)` | Items per page (default 50) |
| `.group_by::<M>()` | Visual grouping by metadata field |
| `.action_budget(n)` | Max inline action buttons (default 3) |

### 15.4 Decision: Actions via callback

Actions are context-dependent (a queued download offers "cancel",
a completed one doesn't). They remain a callback on the builder,
receiving the entity row and its metadata:

```rust
DataView::for_entity::<Download>("queue")
    .col::<DownloadProgress>()
    .col::<TargetEntity>()
    .col::<SourceInfo>()
    .actions(|row| {
        match row.col::<DownloadStatus>() {
            Status::Queued => vec![
                Action::post("cancel", "Cancel",
                    format!("/api/queue/{}/cancel", row.id()))
                    .danger()
                    .replace_row(),
            ],
            Status::Failed => vec![
                Action::post("retry", "Retry",
                    format!("/api/queue/{}/retry", row.id()))
                    .ghost()
                    .replace_row(),
            ],
            Status::Shelved => vec![
                Action::post("unshelve", "Unshelve",
                    format!("/api/queue/{}/unshelve", row.id()))
                    .accent()
                    .replace_row(),
            ],
            _ => vec![],
        }
    })
    .execute(&ctx)?;
```

The `Action` builder is unchanged from the research docs. Builder
pattern with chainable modifiers (`.danger()`, `.confirm()`,
`.replace_row()`, `.async_stream()`, etc.).

### 15.5 Decision: `Actionable` trait in SDK

The `Actionable` trait is defined in the SDK — it's a trait
contract that entity types can implement. Plugin connectors may
define actions on their entity types.

```rust
/// Any entity that can have actions performed on it.
/// Defined in saalis-sdk, implementations in web layer or plugins.
trait Actionable {
    type Actions: IntoIterator<Item = Action>;
    fn actions(&self) -> Self::Actions;
}
```

The associated type lets impls return `&'static [Action]` for
static action sets or a small `ArrayVec`/array for dynamic ones —
no heap allocation required.

`Action`, `ActionStyle`, `ActionFeedback`, `SwapTarget` are SDK
types. The web layer's DataView builder uses `Actionable` impls
when they exist, or falls back to the explicit `.actions()`
callback on the builder.

### 15.6 Decision: Scheduler dispatch for GUI actions

Every user action that touches an external service or might take
>100ms goes through the scheduler (§5). The API handler dispatches
intent, not work.

```
Browser                    Handler                   Scheduler
   │                         │                          │
   ├─ POST /catalogue ──────►│                          │
   │                         ├─ emit ManualTrigger ────►│
   │◄─ 202 "working..." ────┤                          │
   │                         │                          │
   │◄─── SSE: progress ─────┤◄── CatalogueProgress ───┤
   │◄─── SSE: complete ─────┤◄── CatalogueComplete ───┤
```

**What goes through the scheduler:**

| Action | Event | Priority |
|--------|-------|----------|
| Test connection | `ManualTrigger(connector, HealthCheck)` | Immediate |
| Force catalogue | `ManualTrigger(connector, Catalogue)` | Immediate |
| Enrich all | `ManualTrigger(*, Enrich)` | Immediate |
| Deploy credentials | `CredentialDeploy(connector)` | Immediate |
| Import data | `ImportWork(payload)` | Immediate |

**What stays synchronous:** Save setting, toggle flag, load page
data, dismiss stray, create user profile — instant DB reads/writes
with no external service.

**Feedback paths.** The `ActionFeedback` enum determines how
results reach the browser. This is a **core→web presentation-layer
internal** — it exists only at the boundary where Rust meets HTML
attributes (`hx-get`, `sse-swap`). `&'static str` is used because
these values are compile-time known route/event constants, and
strings are unavoidable at the HTML boundary. This type MUST NOT
leak into SDK, plugin, or scheduler APIs.

```rust
// saalis-web internal only — NOT part of the SDK surface.
enum ActionFeedback {
    /// Swap response directly (instant operations).
    Inline,
    /// Show working state, poll scheduler status.
    Poll { status_url: &'static str, interval_ms: u32 },
    /// Show working state, listen for SSE event.
    Stream { event_name: &'static str },
}
```

### 15.7 Decision: Template engine — Maud

HTML is Rust code via the `maud` crate. Templates compile to
zero-allocation `Markup` values. Full IDE support: autocomplete,
refactoring, go-to-definition. Closing tags are auto-generated,
preventing invalid HTML.

With query-projection DataViews, most HTML is generated
programmatically by generic components. The template engine
handles page skeletons and small HTMX fragments — both of which
Maud handles ergonomically inline:

```rust
/// Generic DataView component — renders any DataView
fn render_dataview(view: &DataView) -> Markup {

    }
}

/// Action button component — renders any Action

Every handler becomes:

}
```

### 15.9 Decision: Guard extractors

Pre-handler validation via axum extractors. Check runs before
handler body; rejection flows through `ApiError`.

| Guard | Purpose | Replaces |
|-------|---------|----------|
| `AdminGuard` | Reject non-admin requests | 9 copy-pasted checks |
| `MutationGuard` | Reject mutations in preview mode | Ad-hoc preview checks |
| `CurrencyGuard` | Reject insufficient balance | Manual balance checks |
| `Validated<T>` | Server-side input validation | Zero validation |

```rust
pub struct AdminGuard;

impl<S: Send + Sync> FromRequestParts<S> for AdminGuard {
    type Rejection = ApiError;

    async fn from_request_parts(
        parts: &mut Parts, state: &S,
    ) -> Result<Self, Self::Rejection> {
        let user = CurrentUser::from_request_parts(parts, state).await
            .map_err(|_| ApiError::Forbidden("not authenticated".into()))?;
        if !user.has_role::<Admin>() {
            return Err(ApiError::Forbidden("admin only".into()));
        }
        Ok(Self)
    }
}
```

### 15.10 Decision: SSE event stream

Server-Sent Events deliver async operation results to the browser:

```rust
async fn event_stream(
    State(state): State<AppState>,
) -> Sse<impl Stream<Item = Result<Event, Infallible>>> {
    let rx = state.scheduler.subscribe_events();
    let stream = ReceiverStream::new(rx).map(|event| {
        Ok(Event::default()
            .event(event.kind.as_str())
            .data(event.to_fragment()))
    });
    Sse::new(stream)
}
```

Each event carries a rendered Maud fragment as its data. The
event name matches the `sse-swap` attribute on the listening
element. When the event arrives, HTMX swaps the fragment in.
No polling needed for the SSE path. Polling is the fallback for
clients that don't support SSE.

### 15.11 What was rejected

- **`DataViewRead` / `DataViewWrite` traits** — superseded by
  query-projection DataViews. The column ZSTs carry enough
  metadata to auto-generate presentation. No separate trait
  impls needed.
- **Presentation structs** (`SystemRow`, `StrayRow`, etc.) — no
  intermediate mapping layer. DataView rows ARE entity query
  results, projected through column ZSTs.
- **`Store` / `Migrate` / `Query` traits** — superseded by the
  entity model (§4), descriptor-based schema, and
  `ctx.query::<M>().of::<E>(id)` API.
- **`FromRow` trait** — superseded by SeaORM entities with derive.
- **Askama templates** — replaced by Maud. With query-projection
  DataViews generating most HTML programmatically, the template
  engine handles less page-specific HTML. Maud's inline approach
  provides full Rust IDE support (autocomplete, refactoring,
  go-to-def), auto-closing tags, and aligns with the "everything
  is Rust types" philosophy.
- **Derive macro for DataViewRead** — moot. There is no
  `DataViewRead` trait to derive. The schema IS the presentation
  source.
- **Separate `ScanOps` trait for read/write split** — the DataView
  builder handles both reads (query) and writes (actions via
  scheduler dispatch). No need for trait-level read/write
  separation in the presentation layer.

## §16 Theme System

### 16.1 Decision: Themes are extensions

Themes are not a special data-only format with their own loading
pipeline. A theme is a standard extension — `.saalix` archive,
`#[register]`, same SDK as connectors and subsystems. No separate
manifest parsing, no data-only mode, no extra infrastructure.

The `Theme` trait is a pure identity marker — it carries no
methods. Its purpose is to constrain `AsVariantOf<T>`: the
`T: Theme` bound ensures that only types explicitly marked as
themes can be used as grouping targets. Without this constraint,
nothing prevents `AsVariantOf<AdminUser>` or
`AsVariantOf<GameTrailer>` from compiling. The trait adds no
behavior; it exists solely to make invalid groupings a compile
error.

```rust
trait Theme: Named + Registrable {}

struct Catppuccin;

#[register]
impl Theme for Catppuccin {}

impl Named for Catppuccin {
    const NAME: &'static str = "catppuccin";
}
```

### 16.2 Decision: ThemeProperty as IR

Properties are the internal representation of theming. The ZST
marker plus its associated `Value` type IS the IR. Properties
have no awareness of any output format — they are pure data
definitions.

```rust
trait ThemeProperty: Named + Registrable {
    type Value: 'static;
}
```

First-party properties:

```rust
struct Background;

struct Foreground;
#[register]
impl ThemeProperty for Foreground { type Value = Color; }

struct Accent;
#[register]
impl ThemeProperty for Accent { type Value = Color; }

struct Surface;
#[register]
impl ThemeProperty for Surface { type Value = Color; }

#[register]

struct AccentDim;
#[register]

#[register]

struct CornerRadius;
#[register]

struct CornerRadiusSm;
#[register]

struct BodyFont;

```

`FontAsset` is a value type describing a font file bundled in
the `.saalix` archive:

```rust
struct FontAsset {
    family: &'static str,
    weight: u16,
    style: FontStyle, // Normal, Italic
    /// Path within the .saalix archive.
    path: &'static str,
}

enum FontStyle { Normal, Italic }
```

Fonts follow the same `Provides` pattern as everything else.
A theme or variant that bundles fonts declares them via
`Provides<BundledFonts>`:

```rust
impl Provides<BundledFonts, AsVariantOf<Catppuccin>> for Mocha {
    const VALUE: &'static [FontAsset] = &[
        FontAsset {
            family: "JetBrains Mono",
            weight: 400,
            style: FontStyle::Normal,
            path: "fonts/JetBrainsMono-Regular.woff2",
        },
        FontAsset {
            family: "JetBrains Mono",
            weight: 700,

            path: "fonts/JetBrainsMono-Bold.woff2",
        },
    ];
}
```

A variant that bundles no fonts simply doesn't implement
`Provides<BundledFonts>` — there is no empty default to
override. The `ThemeBuilder<Css>` checks for `BundledFonts`
and generates `@font-face` rules if present; it does nothing
if absent.

Value types (`Color`, `Length`, `FontFamily`, `FontAsset`,
`bool`) are SDK-provided IR types. The SDK also provides
manipulation utilities (`color::darken()`, `color::lighten()`,
etc.) that operate on these IR types directly — derivation is
Rust code using typed values, not string manipulation.

Extensions can register new `ThemeProperty` types. There is no
distinction between first-party and extension-defined properties
— the registration system treats them identically.

### 16.3 Decision: Provides\<P, G\> and AsVariantOf\<T\>

Variants provide IR values through `Provides<P, G>`:

```rust
trait Provides<P: ThemeProperty, G = ()> {
    const VALUE: P::Value;
}
```

Theme values are compile-time known — colors, booleans, font
lists. `const` ensures they can cross the ABI boundary from
extension dylibs without runtime dispatch.

The second generic `G` is the grouping parameter. It defaults
to `()` (no grouping — a standalone theme that is its own
identity). For variants grouped under a theme, the grouping
is `AsVariantOf<T>`:

```rust
struct AsVariantOf<T: Theme>(PhantomData<T>);
```

The `T: Theme` bound is the constraint that prevents nonsense
groupings. `AsVariantOf<Catppuccin>` compiles because
`Catppuccin: Theme`. `AsVariantOf<AdminUser>` does not compile
because `AdminUser` does not implement `Theme`. This is the
sole reason the `Theme` trait exists — it gates the generic
parameter.

This design eliminates `VariantOf<T>` as a separate trait. The
variant-to-theme relationship is expressed directly on each
`Provides` impl. There is no separate trait that says "Mocha
belongs to Catppuccin" — the `Provides` impls themselves carry
that information through the grouping parameter.

There is no `DefaultVariant` trait. A theme IS its own default
variant. `Catppuccin` implements `Theme` and can also implement
`Provides<P>` directly (with the default `G = ()` grouping).
When no variant is explicitly selected, the theme type itself
is the active variant. Named variants like Mocha group under
the theme via `AsVariantOf<Catppuccin>` and are alternatives
the user can switch to.

No compile-time completeness bound. Compile-time exhaustiveness
checking across dylib boundaries is not possible — proc macros
run at compile time, `inventory` collects at load time. These
are different phases; the macro cannot see properties registered
by other crates that will be linked later.

Completeness is validated at load time via registry walk: the
host checks that the active variant has `Provides` for all
registered properties. Missing properties fall back through a
resolution chain (see §16.7). A variant that covers every
registered property is fully self-contained; one that covers a
subset is still valid — the system is always total.

**Theme as its own default variant (Catppuccin):**

```rust
struct Catppuccin;
impl Named for Catppuccin { const NAME: &'static str = "catppuccin"; }

#[register]
impl Theme for Catppuccin {}

// Catppuccin IS its own default variant.
// These use the default G = () grouping.
impl Provides<Background> for Catppuccin {
    const VALUE: Color = Color::hex("#1e1e2e");
}
impl Provides<Foreground> for Catppuccin {
    const VALUE: Color = Color::hex("#cdd6f4");
}
impl Provides<Accent> for Catppuccin {
    const VALUE: Color = Color::hex("#89b4fa");
}
impl Provides<IsDark> for Catppuccin {
    const VALUE: bool = true;
}
// ... all properties — this is what you get when you
// select "Catppuccin" without choosing a variant.
```

**Named variant grouped under a theme (Mocha):**

```rust
struct Mocha;
impl Named for Mocha { const NAME: &'static str = "mocha"; }

impl Provides<Background, AsVariantOf<Catppuccin>> for Mocha {
    const VALUE: Color = Color::hex("#1e1e2e");
}
impl Provides<Foreground, AsVariantOf<Catppuccin>> for Mocha {
    const VALUE: Color = Color::hex("#cdd6f4");
}
impl Provides<Accent, AsVariantOf<Catppuccin>> for Mocha {
    const VALUE: Color = Color::hex("#89b4fa");
}
impl Provides<IsDark, AsVariantOf<Catppuccin>> for Mocha {
    const VALUE: bool = true;
}
// ... all properties
```

**Standalone theme (no variants, no grouping):**

```rust

impl Named for Corporate { const NAME: &'static str = "corporate"; }

    const VALUE: Color = Color::hex("#ffffff");

```

**Cross-extension variant packs** work naturally: a separate
extension can register
`impl Provides<P, AsVariantOf<Catppuccin>> for CustomVariant`
for all properties without modifying the original theme
extension.

### 16.4 Decision: Fragment\<F\> and ThemeTranslator\<F\>

`Fragment<F>` is a generic newtype tagged by a format marker.
No per-format concrete types, no associated types on translators,
no `ThemeFormat` trait. The format marker is a plain ZST; the
generic parameter on `Fragment` provides type-level
differentiation.

```rust
struct Fragment<F>(Cow<'static, str>, PhantomData<F>);
```

`ThemeTranslator<F>` is the contract for translating an IR value
type to a target format. Implemented by value types for each
format they support:

```rust
trait ThemeTranslator<F> {
    fn translate(&self) -> Fragment<F>;
}
```

Implementations:

```rust
struct Css; // ZST format marker

impl ThemeTranslator<Css> for Color {
    fn translate(&self) -> Fragment<Css> {
        Fragment::new(format!(
            "#{:02x}{:02x}{:02x}", self.r, self.g, self.b
        ))
    }
}

impl ThemeTranslator<Css> for Length {
    fn translate(&self) -> Fragment<Css> {
        Fragment::new(format!("{}{}", self.value, self.unit))
    }
}

impl ThemeTranslator<Css> for FontFamily {
    fn translate(&self) -> Fragment<Css> {
        Fragment::new(format!("'{}'", self.0))
    }
}
```

The static constraint: a builder targeting format `F` can only
process properties where `P::Value: ThemeTranslator<F>`. If a
property's value type has no translator for the target format,
it is a compile error — not a runtime gap.

The structure is implicitly bidirectional: if
`Color: ThemeTranslator<Css>` exists, a corresponding
`ThemeParser<Css> for Color` could parse CSS color strings back
to `Color` values. Not critical, but useful for e2e testing or
theme import tooling without additional design work.

### 16.5 Decision: ThemeBuilder\<F\>

The builder is a trait contract. Format markers implement it:

```rust
trait ThemeBuilder<F>: Registrable {
    type Output;
    fn build(
        &self,
        resolver: &PropertyResolver,
    ) -> Self::Output;
}

#[register]
impl ThemeBuilder<Css> for Css {
    type Output = Stylesheet;
    fn build(
        &self,
        resolver: &PropertyResolver,
    ) -> Stylesheet {
        // Walk registered ThemeProperty descriptors
        // Resolve each value from active variant via Provides<P>
        // Translate via <P::Value as ThemeTranslator<Css>>::translate()
        // Map property name → CSS variable name (builder's concern)
    }
}
```

CSS variable naming (`--bg`, `--accent`, etc.) is the builder's
responsibility, not the property's. The property only knows its
`Named::NAME`. The builder decides how names map to format-
specific identifiers. A `ThemeBuilder<Terminal>` would map the
same property names to ANSI escape sequences instead.

### 16.6 Decision: Scopes are not a theme concern

Scopes — regions of the UI tree where a different theme or
variant is active — belong entirely to the presentation layer
(§15). The theme system has no awareness of scopes.

A scope is a subtree root: a single node in the UI tree that
optionally switches which theme or variant is active from that
point downward. The rendering context tracks the active variant
as it walks the tree. When a scope switch is encountered, the
context updates. Property resolution always uses whatever
variant is currently active — the theme itself is always a
single, complete set of `Provides<P, G>` values.

This means there is no `ScopedValue`, no `ScopeRegistry`, no
scope-aware property resolution in the theme system. If the
sidebar should be darker, the scope points the sidebar subtree
at a darker variant. The variant doesn't know it's being used
for a sidebar — it's just a complete palette.

### 16.7 Decision: Per-user theme selection

Users are entities (§11). Theme preference is user metadata.

```rust
struct ThemePreference;

#[register]
impl MetadataTable for ThemePreference {
    // Columns: ThemeId, VariantId
}

// Entity-metadata association via dual-trait pattern (§4.5)
impl HasMetadata<ThemePreference> for User {}
```

Resolution order:
1. User's `ThemePreference` metadata (if set, includes variant)
2. System-wide theme setting (global `Settings`)
3. First registered theme (the theme itself is the default variant)

### 16.8 Decision: Serving and caching

Theme output is served via a dedicated endpoint:

```
GET /theme/current.css
```

The endpoint:
1. Resolves the active theme and variant for the current user
2. Checks the output cache (keyed by `(theme_id, variant_id)`)
3. If cached, returns with ETag
4. If not, runs `ThemeBuilder<Css>::build()`, caches, returns
   with ETag

Cache invalidation: when theme settings change, the cache entry
is evicted. The ETag changes, and the browser fetches new output
on next page load.

Page templates include the theme output via:

```rust
html! {
    link rel="stylesheet" href="/theme/current.css";
}
```

### 16.9 Decision: Font and asset serving

Fonts are declared via `Provides<BundledFonts>` (see §16.2).
The `ThemeBuilder<Css>` handles font serving:

1. Resolves `BundledFonts` from the active variant
2. Extracts font files from the `.saalix` archive to a
   serve-able location
3. Generates `@font-face` CSS rules referencing the served paths
4. Includes the `@font-face` rules in the builder output

Font files are served via `/theme/fonts/{theme_id}/{filename}`.
The host validates paths (no traversal) and sets appropriate
cache headers.

### 16.10 What was rejected

- **Data-only theme format** — themes are extensions, same as
  everything else. No special TOML-based theme pipeline. No
  separate manifest parsing. Theme logic is Rust code using the
  SDK. This eliminates an entire subsystem (theme TOML parser,
  variant TOML parser, variable derivation engine, scope
  resolution engine) and replaces it with trait impls.
- **`fn css() -> String` on themes/variants** — themes never
  produce format output. They provide typed IR values via
  `Provides<P>`. The host's `ThemeBuilder<F>` owns format
  translation. This inversion means themes are format-agnostic
  and the same property values can target CSS, terminal, native
  UI, or any future format.
- **CorePalette supertrait bundle** — compile-time exhaustiveness
  checking across dylib boundaries is not possible. Proc macros
  run at compile time; `inventory` collects at load time. These
  are different phases. Completeness is validated at load time
  with fallback resolution for gaps.
- **`ThemeFormat` trait with associated types** — unnecessary.
  Format markers are plain ZSTs. `Fragment<F>` carries the
  format tag via generic parameter alone.
- **Per-format concrete fragment types** (`CssFragment`,
  `AnsiSequence`, etc.) — `Fragment<F>` handles differentiation
  through the generic parameter. No separate named types needed.
- **Format-specific conversion traits** (`IntoCss`, etc.) —
  replaced by generic `ThemeTranslator<F>`, one contract for
  all formats.
- **CSS variable names on properties** — properties are format-
  agnostic IR. Variable naming is the builder's concern.
- **Engine-level variable derivation** — derivation is Rust code
  in the variant's `Provides<P>` impls. The SDK provides typed
  utility functions (`color::darken()`, etc.) that operate on IR
  value types directly. Different themes can use different
  derivation strategies.
- **`VariantOf<T>` as a separate trait** — the variant-to-theme
  relationship is expressed directly on `Provides<P, G>` via the
  `AsVariantOf<T>` grouping parameter. A separate trait declaring
  "this variant belongs to that theme" is redundant when every
  `Provides` impl already carries the grouping. This also
  eliminates `is_dark()` as a trait method — it becomes a
  `ThemeProperty` like any other (`IsDark`).
- **`DefaultVariant<T>` capability trait** — a theme IS its own
  default variant. The theme type itself implements `Provides<P>`
  with the default `G = ()` grouping. When no variant is selected,
  the theme's own values are used. No separate marker needed.
- **TOML variant files** — variants are registered Rust types
  implementing `Provides<P, G>`. Palette data can be constants,
  computed, or loaded from bundled files — the variant decides.
- **`type = "theme"` routing to a separate pipeline** — themes
  use the same extension loading pipeline as code plugins. The
  `type` field in the manifest can distinguish them for UI
  purposes (theme browser vs plugin manager) but the loading
  mechanism is identical.
- **`ScopedValue<S, P>` per-property scope overrides** — scopes
  are not a theme concern. A scope switches which variant is
  active for a subtree; the variant itself is always a complete
  set of values. Per-property overrides within a scope are
  redundant — just use a different variant.
- **`ScopeRegistry` in theme builder signatures** — the builder
  resolves properties from a single active variant. Scope
  switching is the presentation layer's job (§15). The builder
  never sees scopes.
- **Theme-defined scope hierarchy** — scope parent-child
  relationships are a UI tree concern, not a theme concern.
  Static parent tracking in the theme layer is not sensible
  when scope resolution depends on runtime tree walking.

## §17 State Management

### 17.1 Decision: Universal data flow pattern

All external data operations — import, export, connector sync,
backup — are the same pattern:

```
Source → Query (select/filter) → Pipeline (transform) → Sink
```

| Operation | Source | Sink |
|-----------|--------|------|
| IGDB sync | IGDB connector | Local store |
| Steam import | Steam connector | Local store |
| Archive import | Archive connector | Local store |
| Archive export | Local store | Archive connector |
| Backup | Local store | Remote/file sink |
| Dev preview | Archive connector | (none — inspect only) |

There is no separate "import/export subsystem." Import and
export are use cases of the universal data flow, not special
operations.

### 17.2 Decision: Source and Sink as connector capabilities

The `Connector` trait is the universal abstraction for data
flow endpoints. Every connector can be a source, a sink, or
both, via capability traits:

```rust
trait Connector: Named + Registrable { ... }
trait Source: Connector { ... }
trait Sink: Connector { ... }
```

Capability by connector type:

| Connector | Source | Sink |
|-----------|--------|------|
| IGDB | yes | no |
| Steam | yes | no |
| Archive | yes | yes |
| Local store | yes | yes |

The direction is not baked into the type — it's determined by
which capability is used in a given operation. The archive
connector reading from a `.saalis` file acts as `Source`. The
same archive connector writing a `.saalis` file acts as `Sink`.

The local entity store is itself a connector implementing both
`Source` and `Sink`. This is what makes "export" natural: the
local store is the source, the archive connector is the sink.
No special export infrastructure needed — it's a data flow
from one connector to another.

### 17.3 Decision: Query API as the selector

The query API selects what data flows through the pipeline.
It is the filter, not the mechanism. The same query
abstraction used for DataViews and entity access parameterizes
what moves between source and sink.

**Export:** a query over the local store as source. "Export all
RPG games" is a query filtered by genre, with the archive
connector as sink.

**Import:** a query over the archive connector as source.
"Import only games I don't already have" is a query over the
archive, filtered against what exists locally.

**Presets** are saved query templates per entity type for common
mass operations:

- "All games" — query all Game entities and their registered
  metadata tables
- "All games + settings" — query Game entities plus all
  Setting entities
- "Everything" — query all registered entity types

The query API already knows which metadata tables belong to
which entity type via registration, so selecting an entity
type implicitly includes all its associated metadata.

### 17.4 Decision: Conflict resolution on the Sink

When data arrives at a sink that already contains matching
entities, the sink's conflict strategy determines what happens.
This is not import-specific — it's the same question when IGDB
sends updated metadata for a game the local store already has.

Conflict strategies:

- **Replace** — incoming data overwrites existing. The source
  wins unconditionally.
- **Skip** — existing data is kept. Incoming duplicates are
  discarded.
- **NewerWins** — the record with the more recent timestamp
  is kept, regardless of source.

Strategies are registered types implementing a trait:

```rust
trait ConflictStrategy: Named + Registrable { ... }

struct Replace;
struct Skip;
struct NewerWins;
```

Each entity type declares its default strategy via
registration. Game entities might default to `NewerWins`
(prefer fresher metadata). User entities might default to
`Skip` (don't overwrite local users with imported ones). The
default is overridable per operation.

The conflict strategy links naturally with the query API's
join/union semantics — a union of two query sources with a
conflict resolution parameter is how the system resolves
overlapping entity sets.

### 17.5 Decision: Cataloguer pipeline as processor

The cataloguer pipeline sits between source and sink. It
transforms, validates, and deduplicates data in transit.

For archive data: minimal processing. The data is already in
entity format — it came from the same system. The pipeline
validates schema compatibility and applies the conflict
strategy.

For connector data (IGDB, Steam): heavier transformation.
API responses are mapped to entity types, metadata is
normalized, duplicates are resolved against the local store.

The pipeline is the same infrastructure regardless of
source and sink. The amount of processing varies by source
type, but the pipeline contract is uniform.

### 17.6 Decision: Credential encryption in archives

When the archive connector acts as a sink (export), it
separates credential-marked settings from the rest of the
data. The `Credential` trait marker (§11) identifies which
settings are credentials programmatically — no manual list.

Export flow:
1. Query settings from source
2. Partition by `Credential` marker
3. Write non-credential settings to main archive data
4. Serialize credential settings separately
5. Encrypt credential blob with ChaCha20-Poly1305
   (key derived via Argon2id from user passphrase)
6. Include encrypted blob in `.saalis` archive

Import flow:
1. Read main archive data normally
2. Detect encrypted credential blob
3. Prompt for passphrase, derive key, decrypt
4. Merge credentials into local settings via conflict
   strategy

Credential inclusion is controlled by the export operation:
- Explicit flag: include or exclude
- Fallback: read `export_include_secrets` setting (default
  `false`)

### 17.7 Decision: Preview mode

Preview mode is a data flow that stops before the sink.
The Source → Query → Pipeline path runs normally, producing
a changeset. The changeset is not committed to the sink.

```
Source → Query → Pipeline → [changeset] → (inspect, discard)
```

A changeset is the natural output of the pipeline before
the final write. Preview mode simply withholds the commit.
The changeset can be inspected: "42 new games, 15 updated
metadata records, 0 conflicts."

In the web layer, preview mode is a runtime flag:
- Scheduler subsystems do not dispatch write work units
- Web layer guard extractors (§15) enforce read-only on
  mutating endpoints
- UI shows a banner indicating preview state
- The flag is not persisted — it's a CLI argument, gone on
  next normal launch

This enables the dev workflow: import a production snapshot,
browse locally to inspect library state, discard on exit.

### 17.8 Decision: Archive manifest

The archive manifest shares the same format and versioning
scheme as the extension manifest (§9.2). Both use
`manifest.toml` in a `.saalis` / `.saalix` archive. The
format is unified — same parser, same version fields, same
structure.

Fields shared with extension manifests:
- `schema_version` — for forward-compatible migration on
  import
- `saalis_version` — the saalis version that produced the
  archive
- `name` — archive identifier
- `description` — optional human-readable description
- `created_at` — timestamp

Fields specific to state archives (not valid in extension
manifests):
- `hostname` — the machine that produced the archive
- `entity_types` — which entity types are included
- `has_credentials` — whether an encrypted credential blob
  is present
- `source_db_version` — schema version of the source
  database

Fields valid only in extension manifests (not in state
archives):
- `type` — extension type (connector, cataloguer, theme,
  etc.)
- `entry_point` — dylib entry
- `sdk_version` — SDK version the extension was built
  against
- `permissions` — requested capabilities

The manifest parser validates that archive-only fields do
not appear in extension manifests and vice versa. Invalid
fields are a hard error, not a warning — this prevents
accidental confusion between `.saalis` state archives and
`.saalix` extension archives.

### 17.9 What was rejected

- **StateDomain trait with table name lists** — the old
  design grouped raw table name strings into domain structs.
  In the entity model, tables are registered `MetadataTable`
  ZSTs. Domains as table lists are meaningless when the
  registration system already knows which metadata tables
  belong to which entity types.
- **StateBundle struct for compile-time enforcement** — the
  old design used a struct with typed fields to force new
  domains to be added. The registration system replaces this
  — unregistered metadata tables cannot exist by
  construction. `inventory` collection at load time provides
  the enforcement.
- **Separate ConfigDomain and SecretsDomain** — config is
  settings (§11). Credentials are settings with the
  `Credential` trait marker. No separate domain types needed
  — the marker-based identification is more precise and
  works with the existing settings infrastructure.
- **Separate import/export subsystem** — import and export
  are use cases of the universal data flow pattern
  (Source → Query → Pipeline → Sink), not a separate
  subsystem with its own traits, CLI commands, and
  infrastructure. The connector pattern with Source/Sink
  capabilities handles all external data operations
  uniformly.
- **Direct `Connection` usage in domain traits** — the old
  design passed `rusqlite::Connection` directly to domain
  export/import methods. The persistence layer abstracts
  this. Data flows through the entity model and query API,
  not raw SQL.
- **`ATTACH DATABASE` as the import mechanism** — an
  implementation detail of the old SQLite-specific design.
  The connector/source/sink pattern is storage-agnostic.

## §18 Session System

### 18.1 Decision: Session is an EntityType (delete-on-expiry lifecycle)

Sessions are transient entities, following the precedent set by
Downloads (§12). A session has metadata, a relationship to a user,
and a finite lifetime. Unlike Downloads which fold on completion
(creating permanent metadata on the target entity), sessions use
a **delete-on-expiry** lifecycle — expired sessions are simply
deleted by the Housekeeper subsystem with no permanent record.

This establishes a second transient entity lifecycle pattern:

| Pattern | Example | On completion/expiry |
|---------|---------|---------------------|
| **Fold-on-completion** | Download | Permanent metadata created on target entity, transient entity deleted |
| **Delete-on-expiry** | Session | Entity deleted, no trace |

```rust
struct Session;

#[register]
impl EntityType for Session {}

impl HasMetadata<SessionData> for Session {}
```

`SessionData` columns:

| Column | Type | Purpose |
|--------|------|---------|
| `Token` | `Str` | Opaque random token (256-bit, base64url-encoded) |
| `CreatedAt` | `i64` | Timestamp of session creation |
| `ExpiresAt` | `i64` | Timestamp of session expiry |
| `UserAgent` | `Str` | Client identification |
| `IpAddress` | `Str` | Originating IP |

Session → User relationship:

```
Session ──belongs-to──► User
```

The relationship system (§4.3) links each session to its owning
user. Session lookup by token returns the Session entity; the
`belongs-to` relationship resolves to the User entity.

`SessionData` is defined in `saalis-core` (not `saalis-sdk`).
Plugins cannot query session state — same security boundary
pattern as `AuthConfig` (§11.1.2).

### 18.2 Decision: AuthConfig replaces AuthData

The old `AuthData` metadata table conflated per-user auth
configuration (PIN hash) with per-session state (tokens, expiry).
With sessions as their own entity type, auth configuration is
a separate metadata table on User:

```rust
struct AuthConfig;

#[register]
impl MetadataTable for AuthConfig {}

impl HasMetadata<AuthConfig> for User {}
```

`AuthConfig` columns:

| Column | Type | Purpose |
|--------|------|---------|
| `PinHash` | `Option<Str>` | Argon2id hash of PIN, or `None` (no PIN). Runtime-computed, never static. |
| `AuthMethodId` | `AuthMethodId` (newtype) | Registered `AuthMethod` `Named::NAME`, stored as text |

One row per user. `AuthConfig` is core-only (not SDK).

### 18.3 Decision: AuthMethod is a registered trait

Auth methods follow the universal pattern: trait + `#[register]`.
The host discovers available auth methods through the registry
and dispatches verification through the trait contract.

```rust
trait AuthMethod: Registrable {
    /// The credential data this method expects from the client.
    type Credential;

    /// Verify the submitted credential against the user's auth config.
    /// Returns Ok(()) if verification succeeds.
    fn verify(
        &self,
        credential: &Self::Credential,
        user_id: EntityId,
        ctx: &AuthContext,
    ) -> Result<()>;

    /// Whether this method requires a multi-step flow
    /// (OAuth redirect, WebAuthn challenge-response).
    const IS_INTERACTIVE: bool;
}
```

PIN is the first implementation:

```rust
struct PinAuth;

#[register]
impl AuthMethod for PinAuth {
    type Credential = Pin;

    fn verify(
        &self,
        pin: &Pin,
        user_id: EntityId,
        ctx: &AuthContext,
    ) -> Result<()> {
        let config = ctx.query::<AuthConfig>()
            .of::<User>(user_id)
            .execute()?;
        // Argon2id verify against stored hash
        verify_pin(pin, &config.pin_hash)
    }

    const IS_INTERACTIVE: bool = false;
}
```

Future auth methods (OAuth, WebAuthn, LDAP) are additional
trait impls — core or plugin. The host owns all session
creation, cookie handling, and security enforcement. The
`AuthMethod` impl only provides credential verification logic
with the inputs the host decides to pass.

`AuthContext` provides access to the query API for reading
`AuthConfig` metadata. It does NOT provide write access — the
host creates sessions, not the auth method.

### 18.4 Decision: Login flow

Login is a two-step process:

1. **User selection** — the client identifies which user to
   authenticate as. In the web UI, this is a profile picker.
   For API clients, the user ID is provided in the request.

2. **Credential verification** — the host looks up the user's
   `AuthConfig` to determine their `AuthMethod`, then
   dispatches to the registered `AuthMethod` impl for
   verification. If the user has no PIN set (`PinHash` is
   `None`), verification is skipped — immediate access.

On successful verification:

3. **Session creation** — the host generates a 256-bit random
   token, creates a `Session` entity with `SessionData`
   metadata, creates the `Session ──belongs-to──► User`
   relationship, and returns the token to the client (as a
   cookie for web UI, as a response body for API clients).

4. **ResolvedUser caching** — the host resolves the user's
   profile, role, and age restriction, caches this as a
   `ResolvedUser` struct keyed by session token in an
   in-memory map.

Session creation is a **synchronous host operation** — direct
DB write, same category as "create user profile" and "save
setting" (§15.6). No scheduler involvement. The operation is
purely local with no external service calls.

### 18.5 Decision: Auth middleware (axum extractor)

The web layer provides a `ResolvedUser` axum extractor that
resolves the active user from the request:

```
Request
  → read session token from cookie
  → look up in-memory ResolvedUser cache
  → if miss: query SessionData by token, check expiry,
    follow belongs-to → User, resolve profile/role,
    populate cache
  → return ResolvedUser
```

Fallback path for machine-to-machine:

```
Request
  → no cookie, check X-Api-Key header
  → validate against static API key from config (§11.6)
  → resolve to the API key's associated user
  → return ResolvedUser
```

Unauthenticated requests: endpoints that don't require auth
(public pages, login endpoint itself) don't use the extractor.
All mutating endpoints and data endpoints require it.

All session logic (token validation, PIN verification, user
resolution) is sync code in `saalis-core`. The axum extractor
is a thin async wrapper calling sync core functions.

### 18.6 Decision: Session lifecycle

**Duration:** Configurable via `SessionSettings`
(`#[derive(Settings)]`, `Infrastructure` domain). Default 30
days. Admin can set different defaults. Sessions refresh their
`ExpiresAt` on activity (sliding window).

**Max concurrent sessions:** Configurable. When exceeded, the
oldest session is invalidated (deleted). No hard limit by
default.

**Logout:** Deletes the Session entity. The in-memory cache
entry is evicted.

**Expiry cleanup:** The Housekeeper subsystem deletes Session
entities where `ExpiresAt < now`. Standard housekeeping work
unit, same scheduling as other cleanup tasks.

**Token format:** 256-bit cryptographically random bytes,
base64url-encoded. Opaque — no JWT, no signed tokens. The DB
is the single source of truth. No distributed verification
need.

```rust
#[derive(Settings)]
#[settings(section = "sessions")]
struct SessionSettings {
    #[setting(scope = Global, domain = Infrastructure)]
    duration_secs: u64,         // default: 30 days

    #[setting(scope = Global, domain = Infrastructure)]
    max_concurrent_per_user: u32,  // default: 0 (unlimited)

    #[setting(scope = Global, domain = Infrastructure)]
    refresh_on_activity: bool,  // default: true
}
```

`SetAccess` (§10.8) gates access to `SessionSettings` — Admin
has `ReadWrite` by default, others `Hidden` unless explicit impls
say otherwise. Pure compile-time trait relationship.

### 18.7 Decision: Events

Session lifecycle events flow through the scheduler event bus
(§5.5):

| Event | Trigger | Consumers |
|-------|---------|-----------|
| `SessionCreated` | Successful login | Notification (if configured), audit log |
| `SessionExpired` | Housekeeper cleanup | Cache invalidation |
| `SessionRevoked` | Explicit logout or admin revocation | Cache invalidation, SSE push |
| `AuthFailed` | Bad PIN / credential | Doctor (rate limiting), audit log |

`AuthFailed` events are monitored by the Doctor subsystem for
brute-force detection. After configurable N failures within a
window, the Doctor can lock the account temporarily.

Error classification (§7):
- Wrong credential → `Fixable` (user can correct and retry)
- Expired session → `Transient` (re-login resolves it)
- Rate limited → `Transient` (retry after backoff)
- No user entities exist → `Fixable` (first-run setup needed)

### 18.8 Decision: No "box status" concept

There is no global "active box user." Saalis is
platform-agnostic — it runs on Batocera appliances, Linux
servers, macOS dev machines, or distributed across multiple
hosts. Every client authenticates via sessions.

External automation (Batocera profile-switch scripts, cron
jobs, other machines) authenticates using sessions like any
other client — via the API with static API key auth or PIN.

Background operations (scheduled cataloguing, periodic
enrichment) that need user context declare this through their
work unit trait contracts. The scheduler provides context from
the triggering session. System-level operations that don't
need a user run without one — settings resolve using only
non-profile layers (§10.9: CLI, env, global DB, TOML,
defaults).

### 18.9 Decision: First-run state

When no `User` entities exist (fresh installation):

1. The web UI detects this via query (no entities of type
   `User`) and shows a setup flow.
2. The first user created is assigned the `Admin` role.
3. The setup flow creates the user entity, sets `AuthConfig`
   (optional PIN), and creates an initial session.

This is a Housekeeper startup validation — "ensure at least
one admin exists." If all admin users are somehow deleted, the
Housekeeper emits a warning and the web UI shows the setup
flow again.

No special "unauthenticated admin mode." The system always
has users — the first-run flow creates one before anything
else happens.

### 18.10 What was rejected

- **`AuthData` as a combined metadata table** — conflated
  per-user auth config (PIN hash) with per-session state
  (tokens). Split into `AuthConfig` (User metadata) and
  `SessionData` (Session entity metadata).
- **Session as metadata on User** — breaks the architectural
  pattern: transient state with its own metadata and
  relationships should be an entity (Download precedent).
  Multiple sessions per user means multiple redundant rows
  carrying the same auth config.
- **JWT / signed tokens** — no distributed verification need.
  Single SQLite DB is the source of truth. Opaque random
  tokens with DB lookup are simpler and avoid key management.
- **Global "active box user" / box status** — assumes
  single-appliance deployment. Saalis is platform-agnostic.
  External clients use sessions. Work unit context is declared
  via trait contracts.
- **Session policy in access control** — `SetAccess` (§10.8) is
  a compile-time `(Role, Domain) → AccessLevel` trait relationship.
  Session duration and limits are runtime settings, not access
  control decisions. Session policy lives in `SessionSettings`
  governed by `SetAccess` impls.
- **Auth methods as `repr(C)` enum** — auth methods are an
  extensible domain concept. Trait + `#[register]`, not a
  closed set. Plugins can contribute auth methods; the host
  controls all execution and security enforcement.
- **Auth methods as core-only trait** — the SDK boundary model
  inherently constrains what plugins can do. An `AuthMethod`
  impl only provides verification logic; the host owns session
  creation, cookie handling, and all security enforcement. No
  reason to restrict auth method registration to core.

## §19 Clarifications: Concrete Types → Trait Contracts

This section documents design decisions that were initially expressed
as concrete structs but were later recognized as violations of the
architecture's core principle: **define contracts (traits + ZSTs),
not concrete shapes (structs with fields).** The context API
(`ctx.query()`, `ctx.write()`, `ctx.notify()`, `ctx.progress()`)
is the universal interface for all data flow. Pipeline types that
carry data between stages are unnecessary intermediaries — the
context handles it.

### 19.1 Pipeline data types eliminated

These concrete structs existed to shuttle data between pipeline
stages. They're replaced by context API calls — query what you
need, write what you produce.

**BAD — `Discovered<E, C>` as return collection:**
```rust
// Plugin returned a collection of discovered entities.
struct Discovered<E: EntityType, C: Connector> {
    external_id: ExternalId<C>,
    title: String,
    _type: PhantomData<E>,
}

trait Cataloguer<E: EntityType>: Connector {
    type Discoveries: IntoIterator<Item = Discovered<E, Self>>;
    fn catalogue(&self, ctx: &ConnectorContext<Self>) -> Result<Self::Discoveries>;
}
```

**GOOD — write through context:**
```rust
fn catalogue(&self, ctx: &ConnectorContext<Self>) -> Result<()> {
    for item in api_response {
        ctx.write::<Entities>(|row| {
            row.set::<ExternalId<Self>>(item.id);
            row.set::<Title>(item.title);
        });
    }
    Ok(())
}
```
Discovery IS a write. No struct, no collection, no return type.

---

**BAD — `EnrichCandidate<E>` as input struct:**
```rust
struct EnrichCandidate<E: EntityType> {
    entity_id: EntityId,
    external_ids: ExternalIdSet,
    _type: PhantomData<E>,
}
```

**GOOD — query through context with ZST filters:**
```rust
fn enrich(&self, ctx: &ConnectorContext<Self>) -> Result<()> {
    let entities = ctx.query::<Titles>().filter::<NeedsEnrichment>()?;
    for entity in entities {
        let data = ctx.http::<Get>(api_url)?;
        ctx.write::<Rating>(entity.id, |row| {
            row.set::<Score>(data.score);
        });
    }
    Ok(())
}
```
Query what needs work, write results. `NeedsEnrichment` is a
registered filter ZST — composable, extensible, type-safe.

---

**BAD — `SourceCandidate<E>` and `ResolvedSource<E, S>`:**
```rust
struct SourceCandidate<E: EntityType> {
    entity_id: EntityId,
    external_ids: ExternalIdSet,
    _type: PhantomData<E>,
}

struct ResolvedSource<E: EntityType, S: SourceType> {
    entity_id: EntityId,
    url: S::Url,
    size_bytes: Option<u64>,
    checksum: Option<Checksum>,
    _type: PhantomData<(E, S)>,
}
```

**GOOD — query + write through context:**
```rust
fn resolve(&self, ctx: &ConnectorContext<Self>) -> Result<()> {
    let entities = ctx.query::<Sources>().filter::<NoUrl>()?;
    for entity in entities {
        let url = /* look up download URL */;
        ctx.write::<Source>(entity.id, |row| {
            row.set::<Url>(url);
            row.set::<SizeBytes>(size);
        });
    }
    Ok(())
}
```
Sourcing IS writing source metadata. No intermediate types.

---

**BAD — `EnrichSummary` as return struct:**
```rust
struct EnrichSummary {
    processed: usize,
    written: usize,
    skipped: usize,
    errors: Box<[ErrorDescriptor]>,
}
```

**GOOD — `Result<()>`, host counts inline:**
The work unit returns `Result<()>`. The host knows what was written
(it processes the fire-and-forget work units). Counts and error
tracking are derived from the scheduler's batch execution, not
reported by the plugin.

---

**BAD — `DownloadOutcome` as return struct:**
```rust
struct DownloadOutcome {
    bytes_downloaded: u64,
    resumed: bool,
    checksum: Option<Checksum>,
}
```

**GOOD — write through context:**
The downloader writes download metadata through `ctx.write()`.
No return struct needed.

---

**BAD — `ExtractedFile` manifest as return collection:**
```rust
struct ExtractedFile {
    path: PathBuf,
    size: u64,
    checksum: Checksum,
}

trait Extractor<F: ArchiveFormat>: Registrable {
    type Manifest: IntoIterator<Item = ExtractedFile>;
    fn extract(&self, ...) -> Result<Self::Manifest>;
}
```

**GOOD — write through context:**
```rust
fn extract(&self, ctx: &ExtractionContext, ...) -> Result<()> {
    for entry in archive_entries {
        // extract file to dest
        ctx.write::<ExtractedEntry>(entity_id, |row| {
            row.set::<FilePath>(path);
            row.set::<FileSize>(size);
            row.set::<FileChecksum>(checksum);
        });
    }
    Ok(())
}
```
Extraction is scheduler-driven. Same fire-and-forget write pattern.

### 19.2 Container types eliminated

**BAD — `ExternalIdSet` as runtime container:**
```rust
struct ExternalIdSet { /* internal: keyed by descriptor-derived ID */ }

impl ExternalIdSet {
    fn get<C: Connector>(&self) -> Option<&str>;
    fn insert<C: Connector>(&mut self, id: String);
}
```

**GOOD — query API is the typed accessor:**
```rust
let steam_id = ctx.query::<ExternalIds>()
    .of::<Game>(entity_id)
    .get::<Steam>()?;
```
External IDs are metadata columns. The query API provides typed
access. No standalone container needed. `ExternalId<C>` the
newtype stays — it's the typed value for a single ID.

---

**BAD — `BatchResult<T>` + `BatchError` as aggregation structs:**
```rust
struct BatchResult<T> {
    succeeded: Box<[T]>,
    failed: Box<[BatchError]>,
    skipped: usize,
}

struct BatchError {
    entity_id: EntityId,
    class: ErrorClass,
    guidance: Option<&'static str>,
}
```

**GOOD — host iterates `Result<T, E>` inline:**
The scheduler processes batch outcomes as an iterator of results.
Counts successes, handles failures for retry via `AppError<K>`
(which already carries `MAX_RETRIES` and `GUIDANCE`), counts skips.
No intermediate collection, no separate error struct.

### 19.3 Trait hierarchies eliminated

**BAD — `Diagnosable` trait + `DiagnosticReport`:**
```rust
trait Diagnosable: Connector {
    type Strategies: RecoveryStrategySet;
    fn diagnose(&self, ctx: &DiagnosticContext) -> DiagnosticReport;
}

struct DiagnosticReport {
    status: HealthStatus,
}
```

**GOOD — health is metadata, health check is a regular work unit:**
```rust
// Health check work unit — same pattern as everything else
fn execute(&self, ctx: &SchedulerContext, target: &Steam, ...) -> Result<()> {
    let status = if api_ok { HealthStatus::Healthy }
                 else { HealthStatus::Degraded };
    ctx.write::<Health>(connector_id, |row| {
        row.set::<Status>(status);
    });
    Ok(())
}

// Scheduler reads health for dispatch decisions:
let health = ctx.query::<Health>().of(connector_id)?;
```
No `Diagnosable` trait. No `DiagnosticReport`. Health status is
metadata queried and written like everything else. Recovery
strategies remain as registered types — the Doctor subsystem
dispatches recovery work units triggered by health status changes
through the event DAG.

---

**BAD — `ConnectorWorkUnit`, `DbWorkUnit`, `SystemWorkUnit`
as separate trait hierarchies:**
```rust
trait ConnectorWorkUnit<T: Connector>:
    WorkUnit<For<T>, With<EventBatch>, Returns<WorkResult>> {}

trait DbWorkUnit<T: DbBackend>:
    WorkUnit<For<T>, With<EntityIds>, Returns<IntegrityReport>> {}

trait SystemWorkUnit<T: SystemResource>:
    WorkUnit<For<T>, With<()>, Returns<Box<[ThresholdAlert]>>> {}
```

**GOOD — just `WorkUnit` with `type Subsystem`:**
These were convenience aliases that locked input/output types.
Now that `WorkUnit` has `type Subsystem`, the subsystem grouping
is handled at the type level. Each work unit chooses its own
input/output types. `Cataloguer`, `Enricher`, `Sourcer` become
lightweight marker ZSTs that assert the correct subsystem, not
separate trait hierarchies with constrained generics.

---

**BAD — `IntoEvents` for DAG wiring:**
```rust
trait IntoEvents {
    fn into_events(self) -> Vec<Event>;
}
```

**GOOD — `type Emits: EventKindSet` (already fixed in §5.2):**
The DAG is wired statically from type-level sets. The scheduler
knows at registration time which events a work unit emits. No
runtime `into_events()` needed.

### 19.4 What stays (valid concrete types)

Not all concrete structs are wrong. These are legitimate:

- **IR value types** (`Color`, `Length`, `FontAsset`, `FontStyle`) —
  pure data, const-constructible, nothing to abstract over.
- **Credential impls** (`ApiKey(Cow<'static, str>)`, `OAuthToken { ... }`) —
  concrete implementations of the `Credential` trait contract.
- **Settings derive targets** (`SteamSettings`, etc.) — concrete
  structs that implement the `Settings` trait via derive macro.
- **Host-internal types** (`DownloadEngine`) — `saalis-core`
  implementation details, not SDK surface.
- **Newtypes** (`EntityId(i64)`, `ExternalId<C>`, `SdkVersion`,
  `Fragment<F>`) — thin wrappers providing type safety.
- **ZSTs** — the vast majority of types in the architecture are
  zero-sized types used as type-level markers. These are always
  correct.

### 19.5 Principle: when is a concrete struct warranted?

A concrete struct is the right choice when:

1. **It's an IR value type** — pure data with no behavioral contract.
   `Color { r, g, b }`, `FontAsset { family, weight, ... }`. No
   reason to abstract; the data IS the thing.
2. **It's a trait implementation** — the struct implements a trait
   contract. `ApiKey(Cow<'static, str>): Credential`, `SteamSettings: Settings`.
   The trait is the abstraction; the struct is one concrete shape.
3. **It's a newtype** — wrapping one value for type safety.
   `EntityId(i64)`, `ExternalId<C>`.
4. **It's host-internal** — implementation details of `saalis-core`
   that never cross the SDK boundary.

A concrete struct is WRONG when:

1. **It shuttles data between pipeline stages** — use context API
   methods instead (query + write).
2. **It appears in trait signatures** — use trait bounds
   (`IntoIterator`, `AsRef`, associated types).
3. **It aggregates results** — use iterators, not collections.
4. **It exists because "we need to carry X, Y, Z together"** — that's
   a context method, not a struct.
