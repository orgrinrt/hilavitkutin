# Crate Taxonomy Decisions

**Date:** 2026-03-16
**Resolves:** 202603161200_topic.crate-taxonomy-revision.md

## Decisions

### Split by weight, not by every concern

Heavy domains get their own crate. Light concerns stay in core
as the composition layer.

### Final taxonomy (11 crates)

```
saalis-primitives              (no_std, zero deps)
       ↑
  saalis-derive                (proc-macro)
       ↑
   saalis-sdk                  (no_std + alloc, themes included)
     ↑  ↑  ↑
     |  |  saalis-scheduler    (hilavitkutin, event DAG, subsystems)
     |  |       ↑
     |  saalis-persistence     (rusqlite, ColdStore, SIEVE, FTS5)
     |       ↑
  saalis-core                  (plugins, settings, notifications, files, main.rs)
       ↑
  saalis-web                   (axum, maud, HTMX, SSE)

First-party extensions (cdylib, SDK-only):
  saalis-connector-steam
  saalis-connector-igdb
  saalis-connector-screenscraper
  saalis-format-nointro
```

### Per-crate responsibilities

| Crate | std? | Responsibility |
|-------|------|---------------|
| `saalis-primitives` | `no_std` | `repr(C)` enums (`Priority`, `ErrorClass`, `HealthStatus`, `Urgency`, `FieldKind`, `Op`, `SettingsScope`), ABI version const |
| `saalis-derive` | proc-macro | `#[register]`, `#[derive(Settings)]` |
| `saalis-sdk` | `no_std + alloc` | Trait contracts, ZSTs, `Str`, `ColumnValue`, theme system, `Storage`/`FileStorage`/`ManagedStorage` traits, context types, query/write builders |
| `saalis-scheduler` | `std` | Wraps hilavitkutin, subsystem dispatch, event DAG, `WorkUnit` bridge, `Curator`/`Housekeeper`/`Doctor` |
| `saalis-persistence` | `std` | rusqlite (bundled, FTS5), `ColdStore`, SIEVE hot store management, epoch flush, `_saalis_schema` migrations, FTS5 sync |
| `saalis-core` | `std` | Plugin loading (dlopen), settings engine, notification dispatch, file management (`ManagedStorage` impl), `main.rs` — composes scheduler + persistence |
| `saalis-web` | `std`, async | axum, Maud, HTMX, SSE, `ResolvedUser`, DataView rendering, API types |
| `saalis-connector-steam` | cdylib | Steam Web API connector |
| `saalis-connector-igdb` | cdylib | IGDB/Twitch API connector |
| `saalis-connector-screenscraper` | cdylib | ScreenScraper connector |
| `saalis-format-nointro` | cdylib | No-Intro/TOSEC DAT verification |

### Q&A resolution

| Q | Decision | Rationale |
|---|----------|-----------|
| Q1: Split core? | Yes — `saalis-scheduler` + `saalis-persistence` extracted | Heavy compile isolation (SQLite C, hilavitkutin). Core becomes lightweight composition layer. |
| Q2: hilavitkutin placement? | `saalis-scheduler` internal | Plugins never see hilavitkutin types. Scheduler wraps them behind SDK traits. |
| Q3: Storage trait family? | `Storage`/`FileStorage`/`ManagedStorage` in SDK. `ColumnarStorage` in scheduler. `PersistentStorage` in persistence. | Plugins only need file-level traits. Internal storage traits stay where their impls live. |
| Q4: arvo placement? | SDK level | Primitives stays zero-deps. arvo enters via SDK. |
| Q5: First-party extensions? | Separate crates from day one | Proves plugin architecture. Forces clean SDK boundaries. |
| Q6: Theme system? | Stays in SDK | All trait contracts and ZSTs. Not heavy enough for own crate. |
| Q7: Split web? | Keep monolithic | Revisit if >5k LOC. |
