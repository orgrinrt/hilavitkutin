## Â§2 Changes needed (research contradicts or improves current design)

### 2.1 SeaORM â†’ rusqlite

**Current:** Â§9.3 lists SeaORM in `saalis-core`.
**Research says:** SeaORM wraps SQLite in async `spawn_blocking`,
adding 12x overhead per epoch flush. SQLite has no native async API.
Our sync core makes async wrapping pure waste.

**Change:** Replace SeaORM with direct rusqlite (bundled, FTS5
enabled). Use `rusqlite_migration` for schema versioning via
`PRAGMA user_version`. Typed ZSTâ†’`ToSql`/`FromSql` bridge via
`ColumnDescriptor` trait. Static writer + reader connection pair
(no r2d2 pool).

**Source:** `seaorm-vs-rusqlite.md`, `persistence-strategy.md`

### 2.2 Hot/cold architecture (new â€” not in current design)

**Current:** Â§1.2 describes persistence as "SeaORM host-only" with
direct DB access. No hot store concept.
**Research says:** In-memory columnar store (hilavitkutin) as the
runtime data surface, SQLite as cold durability journal. SIEVE
eviction, epoch-batched WAL flush, prioritized startup warming.

**Change:** Add a new section (or revise Â§1.2) describing:
- Hot store: hilavitkutin's `ColumnStorage` impl, SIEVE-managed
- Cold store: SQLite via rusqlite, write-only during normal operation
- Sync layer: Housekeeper work unit, epoch-batched flush
- Startup: prioritized partial preload from manifest, background fill
- Memory budget: partitioned between hot store and SQLite page cache

**Source:** `hot-cold-storage-strategies.md`, `persistence-strategy.md`, `sieve-eviction.md`

### 2.3 SettingsValue serialization â†’ serde

**Current:** Already changed in this session â€” `store()/parse()`
removed, `Serialize + DeserializeOwned` added as supertraits.
**Status:** Done in the doc, but the research adds detail:
use RON for config/manifests, JSON for web API, postcard for

**Source:** `serde-patterns.md`

### 2.4 Content-addressable blob store (new)

**Current:** Â§14 Filesystem Abstraction has `Storage` trait with

**Research says:** BLAKE3-keyed flat blob store with reflink to

**Source:** `content-addressable-storage.md`, `content-acquisition-pipeline.md`

## Â§5 Decision log

### D1: SeaORM â†’ rusqlite (approved 2026-03-15)

**Decision:** Drop SeaORM entirely. Use rusqlite (bundled, FTS5) for
all cold store access. `rusqlite_migration` for schema version
tracking via `PRAGMA user_version` (host tables) and a
`_saalis_schema` tracking table (plugin tables).

**Rationale:** `sea-orm-migration` cannot be used independently â€” it
pulls the full ORM. `sea-orm-sync` is RC status with negligible
adoption. Our `Migration<V>` + `MigrationOpSet` type system generates
migration SQL; we only need an execution/tracking layer.

**Schema versioning:** `_saalis_schema(table_name, version,
last_migrated_at)` for ALL tables â€” host and plugin alike. One
mechanism, no parallel systems. `PRAGMA user_version` is not used.
The `_saalis_schema` table itself is bootstrapped on first run
(if absent, create it â€” this is the only self-referential case).
Migration ordering follows FK dependency graph (same topological
sort as write ordering).

### D2: Column value types â€” `ColumnValue` trait (approved 2026-03-15)

**Decision:** One trait, one mechanism for all column value types:

```rust
trait ColumnValue: Serialize + DeserializeOwned + Send + Sync + 'static {
    type Affinity: SqlAffinity;
}
```

- `Affinity` is for DDL generation only (4 ZSTs: `Integer`, `Real`,
  `Text`, `Blob`)
- `Serialize`/`DeserializeOwned` is the sole serialization path for
  ALL types â€” primitives and custom alike
- No special fast path for primitives, no `to_storage`/`from_storage`
- The host uses one serde-to-rusqlite bridge for everything
- Extensions declare affinity (one line) and implement serde (which
  they'd do anyway)
- Validation is the type's own constructor / serde `Deserialize` impl

**Replaces:** The `SqlPrimitive` / `SqlType` / `SqlValue` concepts
discussed earlier. Also replaces the `As<V>` bound â€” `As<V>` now
requires `V: ColumnValue` instead of an ad-hoc bound.

### D3: Hotâ†”cold bridge â€” ColdStore flush/load (approved 2026-03-15)

**Decision:** `ColdStore` is a host-internal struct in `saalis-core`
that owns rusqlite connections and bridges the hot store â†” SQLite.
Not a trait â€” plugins never see it.

**Directions:**
- Cold â†’ hot (load): rusqlite row â†’ serde `Deserialize` via
  `ColumnValue` â†’ typed value â†’ hilavitkutin `ColumnStorage` segment
- Hot â†’ cold (flush): `ColumnStorage` segment â†’ typed value â†’
  serde `Serialize` via `ColumnValue` â†’ rusqlite `INSERT OR REPLACE`

**Flush triggers:**
- Epoch timer (aligned to scheduler cadence)
- Batch complete (after work unit batch finishes)
- Memory pressure (SIEVE eviction persists before discarding)
- Graceful shutdown

**Dirty tracking:** Per `(entity_id, MetadataTable)` pair. When
`ctx.write::<Rating>(entity_id, ...)` is called, the host marks
that pair dirty. Epoch flush collects all dirty pairs.

**Transaction shape:** One transaction per epoch flush, all dirty
rows across all tables. SQLite WAL allows concurrent reads
(hot store serves all reads anyway).

**ColdStore API (host-internal):**
- `load_table<M: MetadataTable>()` â€” bulk SELECT, deserialize, push
  to hot store
- `flush_dirty(dirty: &DirtySet)` â€” collect, serialize, INSERT OR
  REPLACE
- `checkpoint()` â€” WAL checkpoint when threshold exceeded

### D5: Hot store memory management (approved 2026-03-15)

**Decision:** SIEVE eviction is saalis-side, not hilavitkutin. Saalis
wraps hilavitkutin's `ColumnStorage` with a `WeightedSieveCache`
keyed by `(TableId, SegmentRange)`, weighted by byte size.

**Eviction granularity:** Per-table-segment. All columns of one
metadata table for a row range are evicted together â€” matches the
natural access pattern (DataView touches specific tables).

**Phase-based budget:**

| Phase | Hot store | Trigger |
|-------|-----------|---------|
| Active browsing | 60% available RAM | Default |
| Background idle | 40% | User idle timeout |
| Game running | ~0% | Game launch detected |
| Post-game resume | Ramp up | Game exit detected |

Game launch: flush all dirty â†’ `cache.clear()` â†’ `PRAGMA
shrink_memory`. Post-game: load Composites first, then manifest
working set, then background fill.

**Startup warming:** 4-tier priority:
1. `_saalis_schema` + `entity_types` (immediate, tiny)
2. Composites table (gallery view â€” first thing user sees)
3. Last-session working set (from `_saalis_warmup_manifest`)
4. Background fill by access frequency

**Key invariant:** Hot store is fully reconstructable from cold store
at any time. No data exists only in hot store â€” everything dirty
is flushed before eviction.

**Configuration:** `Infrastructure` domain setting with three
presets: Aggressive (default â‰¤4GB), Balanced, Keep-alive.

### D7: Composites as registered CompositeField trait (approved 2026-03-15)

**Decision:** Composites table is NOT host-hard-coded. It follows the
same registration pattern as everything else.

```rust
trait CompositeField: Registrable {
    type Source: MetadataTable;
    type Value: ColumnValue;
    fn composite(&self, ctx: &CompositeContext) -> Self::Value;
}
```

- Extensions register `CompositeField` impls â†’ host discovers via
  `inventory` â†’ creates composites table with one column per field
- `composite()` uses the standard query context API â€” can be as
  complex as needed (weighted averages, cross-table lookups, etc.)
- First-party fields: `PrimaryTitle`, `CoverArtUrl`,
  `CoverDimensions`, `PrimaryRating`, `PrimaryGenre`, `Platform`
- Extensions add their own (e.g., `SteamReviewScore`)
- Gallery view reads whatever fields are available in Composites

**Refresh triggers (unchanged):**
- Immediate: on write to a CompositeField's `Source` table
- Deferred: after batch enrichment completes
- Full rebuild: Doctor subsystem periodic sweep

**ThumbHash:** `CoverThumbHash` is a `CompositeField` whose
`composite()` reads the cover art URL, generates a 30-byte
ThumbHash, and stores it. Gallery renders instantly from Composites.

### D6: FTS5 search â€” dual path (approved 2026-03-15)

**Decision:** Two search paths, each serving a different need:

**Fast path (hot store):** Prefix autocomplete as the user types.
Scans the Titles column in the hot store directly â€” brute-force
prefix matching on in-memory strings, sub-millisecond at 100k
entities. No SQLite round-trip. Serves the instant-feedback UX
("sup..." â†’ shows matches live).

**Full path (cold store FTS5):** Ranked full-text search for
complete queries. FTS5 external content table synced from Titles
metadata. Returns ranked entity IDs â†’ hot store serves full
metadata for display.

**FTS5 configuration:**
- External content table (content synced from Titles via epoch flush)
- `unicode61` tokenizer with `remove_diacritics 2`
- Prefix indexes: `prefix='2 3 4'`
- BM25 column boosting: title 10x, developer 5x, tags 3x,
  description 1x
- `strsim` Jaro-Winkler fuzzy fallback when FTS5 returns nothing

**FTS5 sync:** Updated during epoch flush â€” when dirty Titles
rows are flushed to cold store, the FTS5 index is updated in the
same transaction. No separate sync mechanism.

### D9: Unified Storage trait family (approved 2026-03-15)

**Decision:** All storage concerns share one trait family with
capability supertraits:

```rust
trait Storage: Registrable {}

trait FileStorage: Storage { /* read, write, move, link, walk */ }
trait ColumnarStorage: Storage { /* allocate, slice, release â€” hilavitkutin */ }
trait PersistentStorage: Storage { /* flush, checkpoint, backup â€” SQLite */ }
trait ManagedStorage: Storage { /* store, link, gc, dedup â€” file lifecycle */ }
```

- Hot store: `ColumnarStorage`
- Cold store: `PersistentStorage`
- File system: `FileStorage`
- Managed file store: `FileStorage + ManagedStorage`
- Cold store bridge connects `ColumnarStorage` â†” `PersistentStorage`

**Managed file store:** Saalis-managed files live in a central store
(`/userdata/saalis/store/<entity_id>/`), linked to consumer-expected
paths. Observed (external) files are tracked by reference only.

**Link strategy:** reflink (btrfs/APFS) â†’ hardlink (same fs) â†’
symlink (cross fs) â†’ copy (fallback). Detected per filesystem.

**Dedup:** Hash before storing. Same BLAKE3 digest = reflink/hardlink
existing copy instead of duplicating.

**File metadata per entity:**
- `store_path` â€” canonical path (managed files)
- `external_path` â€” external location (observed files)
- `hash` â€” BLAKE3 digest
- `size` â€” file size
- `managed` â€” bool, saalis owns lifecycle or not

Specifics of each capability trait's API deferred to implementation.

### D8: QueryFilter as registered trait (approved 2026-03-15)

**Decision:** Query filters are registered ZSTs with a `matches()`
method. Extensions define their own filters.

```rust
trait QueryFilter: Registrable {
    fn matches(&self, ctx: &FilterContext, entity: EntityId) -> bool;
}
```

- `FilterContext` provides read access to the hot store
- Compound predicates are just Rust logic in `matches()`
- First-party filters: `NeedsEnrichment`, `NoUrl`, `Unverified`,
  `Stale`, `HasSource`, etc.
- Extensions register their own (e.g., `NeedsSteamReview`)
- Applied via `ctx.query::<M>().filter::<F>()`
- The query engine calls `matches()` per entity, returns matching set
- Composable: `.filter::<A>().filter::<B>()` = AND

### D4: Nullable columns via `Option<V>` (approved 2026-03-15)

**Decision:** `As<Option<V>>` for nullable columns. `As<V>` is NOT
NULL. Uses the type system directly â€” no attributes, no wrappers.

```rust
// Required â€” generates NOT NULL
impl Column<In<Rating>, As<i64>> for UserScore {}

// Nullable â€” generates without NOT NULL constraint
impl Column<In<Rating>, As<Option<i64>>> for CriticScore {}
```

`ColumnValue` blanket for `Option<V: ColumnValue>`:
- `type Affinity = V::Affinity` (same column type)
- Serde handles null naturally (serializes as SQL NULL / JSON null)
- Hot store: hilavitkutin's stride already has a null flag byte
  (from polka-dots 128-bit stride design: 1 byte discriminant +
  15 bytes payload)
