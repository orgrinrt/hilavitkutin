# Implementation Roadmap — From Stubs to Working Core

**Date:** 2026-03-16
**Context:** Design round 202603161400 closed. 11-crate mockspace
with compiled type signatures. No implementations. Goal: working
core where extensions can register, catalogue, enrich, and serve
content via the web UI.

## Current state

Every crate has `src/lib.rs` with real type signatures that compile.
No method bodies, no logic, no runtime behavior. The SDK defines
contracts, primitives defines enums, derive has a passthrough
`#[register]` macro that does nothing.

The old romhoard code in `src/` at repo root is the working system.
It will be replaced once the new crates reach feature parity.

## Critical path (in dependency order)

### Phase 1: Registration infrastructure

**saalis-derive — `#[register]` proc macro**

The foundation. Without real registration, nothing else works.

- Parse the impl block to extract: trait name, type name, generic params
- Generate a `repr(C)` descriptor struct with:
  - `name: &'static str` (from `Named::NAME`)
  - `sdk_version: SdkVersion` (from `SdkVersion::CURRENT`)
  - `extern "C"` function pointers for each trait method
- Generate `inventory::submit!` call
- Generate `HasDescriptor<D>` compile-time check
- Generate `Named` + `Display` impl if not already present

**saalis-sdk — registration traits with real impls**

- `Named`: already has `const NAME` — done
- `Registrable`: supertrait — needs `Display` default
- `Descriptor`: `fn name()` and `fn sdk_version()` read from fields
- `Registry`: `fn iter()` uses `inventory::iter`, `fn get()` finds by
  name, `fn validate()` checks versions and duplicates

### Phase 2: Schema and persistence

**saalis-sdk — ColumnValue + serde**

- `ColumnValue` trait: `Serialize + DeserializeOwned` supertraits,
  `type Affinity: SqlAffinity` — wire up the serde bounds
- Impls for `i64`, `f64`, `bool`, `Str`, `Option<V>`
- `SqlAffinity` ZSTs stay as-is

**saalis-persistence — ColdStore with rusqlite**

- `rusqlite` connection management (writer + reader)
- DDL generation from descriptors (CREATE TABLE from MetadataTable
  + Column descriptors, column affinity from `ColumnValue::Affinity`)
- `_saalis_schema` version tracking table
- `rusqlite_migration` integration for `Migration<V>` execution
- Serde-based row serialization (ColumnValue → rusqlite params)
- Serde-based row deserialization (rusqlite rows → ColumnValue)
- Basic `load_table()` and `flush_dirty()` (no SIEVE yet — direct
  SQLite reads as v1, hot store comes later)
- FTS5 external content table for Titles

**Note:** Phase 2 does NOT need hilavitkutin or SIEVE. The v1
persistence layer uses SQLite directly for both reads and writes.
The hot store is layered on top in a later phase.

### Phase 3: Connector context and work units

**saalis-sdk — context types with real methods**

- `ConnectorContext<C>`: `fn query()`, `fn write()`, `fn http()`,
  `fn notify()`, `fn progress()`, `fn settings()`
- `QueryBuilder<M>`: `fn of()`, `fn filter()`, `fn execute()` —
  returns iterator over rows from ColdStore
- `WriteBuilder<M>`: `fn set()` — queues a dirty write
- `HttpClient` / `HttpMethod`: delegate to ureq
- These are SDK-side trait definitions; the actual implementations
  are in core (constructed by the host, passed to plugins)

**saalis-scheduler — basic work unit dispatch**

- Subsystem registration via inventory
- Event DAG construction from `Triggers`/`Emits` associated types
- Basic sequential dispatch (no morsel-driven, no hilavitkutin —
  just run work units in topological order)
- Cadence timers (MIN_CADENCE floor per subsystem)
- Epoch flush trigger after work unit batch completes

### Phase 4: Core composition

**saalis-core — wiring everything together**

- `main.rs`: boot sequence (open DB, run migrations, collect
  registrations, start scheduler, start web server)
- Plugin loading: for v1, compiled-in only (inventory from linked
  crates). dlopen comes later.
- Settings engine: parse `config.toml`, 7-layer precedence resolution,
  `#[derive(Settings)]` integration
- Notification dispatch: basic logging, no SSE yet
- File management: basic `Roms`, `Temp`, `Cache` locations with
  standard filesystem ops. No CAS blob store yet.

### Phase 5: Web UI

**saalis-web — minimal serving**

- Axum router with basic routes (library, settings, system)
- Maud templates for: gallery view (cover art grid), game detail page,
  settings page
- `ResolvedUser` extractor (single-user v1 — no auth required initially)
- DataView builder: basic gallery view from Composites metadata
- Static asset serving (CSS, HTMX from CDN, gamepad nav JS)
- SSE endpoint for progress updates

### Phase 6: First-party extensions

**saalis-connector-steam — first real connector**

- `Connector` impl with `SteamSettings` (API key)
- `Cataloguer<Game>` — fetch user's Steam library via Steam Web API
- `Enricher<Game>` — basic metadata from Steam store API
- This validates the entire pipeline: register → catalogue → persist →
  display in gallery

**saalis-connector-igdb — primary enricher**

- `Connector` impl with IGDB credentials (Twitch OAuth)
- `Enricher<Game>` — multi-query batch enrichment via Apicalypse
- Cover art, ratings, genres, descriptions

**saalis-connector-screenscraper — retro media**

- `Connector` impl with ScreenScraper credentials
- `Enricher<Game>` — ROM hash-based identification, box art

**saalis-format-nointro — ROM verification**

- DAT file parsing via `listinfo` crate
- Verification as enrichment work unit (CRC32/MD5/SHA1 matching)

## What is deferred (not needed for v1 parity)

- hilavitkutin hot store integration (use SQLite directly for v1)
- SIEVE eviction (no hot store = no eviction)
- Phase-based memory budget (no hot store = no memory pressure)
- dlopen plugin loading (compiled-in only for v1)
- CAS blob store (plain file copies for v1)
- Quality profiles (manual source selection for v1)
- Declarative TOML connectors (Rust-only for v1)
- Theme system (default theme, no customization for v1)
- Multi-user auth (single user for v1)
- Download engine (use existing romhoard download code as reference)
- Gamepad navigation (mouse/keyboard first)
- i18n (English only for v1)
- Import from gamelist.xml / LaunchBox (manual catalogue for v1)
- EmulationStation integration (standalone for v1)

## Implementation order within each phase

Each phase is a design round:
1. Write topic file with current state + changes
2. Write doc changelist → update DESIGN.md.tmpl files
3. Lock docs → write src changelist
4. Implement → cargo check → cargo mock
5. Lock src → close round

Phases can overlap: e.g., Phase 3 (context) can start while Phase 2
(persistence) is in progress, since the SDK side of context is
independent of the ColdStore implementation.
