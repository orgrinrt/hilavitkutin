## Context

With every peer running the full runtime, data management becomes
distributed. Downloads, enrichment, content routing, and sync all
need mesh-aware protocols. This topic covers the operational design
for how data flows through the mesh.

## Scope

### Download coordinator per-peer
- Each peer runs its own download coordinator locally
- Lane pool with bandwidth management (bulk, curator, system)
- Bandwidth loaning when lanes are idle
- Game-time awareness (pause downloads during gameplay)
- ManagedDownload + MonitoredDownload traits per-peer
- Mesh-first sourcing: pull from mesh peers before external CDNs

### Inter-cluster bandwidth
- Time-based bandwidth policies (published via gossip)
- Per-cluster peer rules (max request rates, blocked windows)
- Bandwidth negotiation protocol
- QoS-driven source selection for transfers
- Schedule format: UTC hours, Option<u32> semantics (None = uncapped)

### Content routing
- Intra-cluster: full content map (every entity, every file)
- Inter-cluster: cuckoo filter summaries (support deletion)
- Content availability announcements via PlumTree gossip
- Mesh-first resolution: local → mesh → external fallback

### Sync and reconciliation
- HLC (Hybrid Logical Clock) per entity for versioning
- Per-peer-pair last_sync_hlc for independence detection
- XOR fingerprint over sorted hash ranges for set reconciliation
- Divergent history merge:
  - Enrichment: field-level, HLC-ordered tiebreak
  - User data: per-user authority
  - Source URLs: union
  - File hashes: trust verified over unverified
- Delta sync for incremental changes
- Bloom/cuckoo filter pre-check before full hash exchange

### Stash (local content tracking)
- Per-peer Stash: entity → local path mapping
- StashEntry: path, hash, size, last_verified, status
- LocalStatus enum: Present, Downloading, Missing, Corrupt
- Advertised to mesh via content announcements
- Synced with mesh on reconnect

### Replication and redundancy
- Configurable min_replicas / target_replicas
- Background replication work unit maintains targets
- Relay participates in redundancy (encrypted at rest)
- Salted hash keys for relay storage (90-day rotation)
- Natural backup: replication IS the backup

### File transfer
- LAN: stream whole file, BLAKE3 verify at end (skip bao-tree)
- WAN: bao-tree verified streaming via verkko-blobs
- Multi-source: sequential provider fallback (start simple)
- Mesh-first: check local → check mesh peers → external CDN

### Cold store sync
- rkyv archives synced across peers via verkko-blobs
- Str handles: content-hash on wire, re-intern on receive
- String table: per-peer runtime + shared const table
- Locale overlays: separate rkyv files, keyed by content hash

### Mesh-first data lifecycle
1. External API/CDN seeds data into mesh (one peer enriches/downloads)
2. Peer advertises new content to mesh via gossip
3. Other peers pull from mesh (LAN speed between cluster peers)
4. External sources never called again for same content
5. Redundancy maintains availability across peer outages
