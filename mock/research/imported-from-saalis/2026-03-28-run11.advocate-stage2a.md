## T1: Version Negotiation Timing -- Steelmanning Encrypted (SpecWriter)

The emerging consensus favors WireEng's plaintext pre-handshake probe. The
argument: why waste a Noise handshake on a peer you cannot talk to? Simple,
efficient, obvious. The SpecWriter's encrypted approach looks like unnecessary
work. I will now demonstrate why that "obvious" choice is a trap.

### The attack surface WireEng opens

WireEng's VersionProbe is a 24-byte unauthenticated, unencrypted UDP
datagram with a fixed 4-byte magic (`0x56524B4F`) at offset 0. This
creates at minimum five attack vectors:

**1. Protocol fingerprinting.** Any observer on the network path can
identify verkko traffic by scanning for the magic bytes. The VersionProbe
is sent before any encryption. A passive adversary (ISP, state actor,
curious neighbor on shared WiFi) can trivially identify which devices
are running verkko. For a household mesh protocol designed for privacy,
leaking "this device runs verkko" on every connection setup is a real
cost. SpecWriter's encrypted approach leaks nothing: the first byte on
the wire is a Noise handshake message, indistinguishable from any other
Noise-based protocol.

**2. Version oracle.** The VersionProbe advertises the exact protocol
versions the peer supports. An active attacker can send VersionProbes to
every IP on a subnet and build a map of which devices support which
verkko versions. This is valuable for targeted attacks: old versions may
have known vulnerabilities. The SpecWriter's approach reveals versions
only to authenticated peers.

**3. Amplification.** The VersionProbe is 18-28 bytes. The VersionResponse
is 20 bytes. The amplification factor is close to 1:1, so this is not a
strong amplification vector. But the VersionResponse echoes the nonce
without any rate limiting or authentication. An attacker who spoofs the
source address can cause the responder to send unsolicited traffic to a
victim. WireEng specifies no rate limiting on VersionResponse generation.
A responder that answers every VersionProbe it receives is a reflector.

**4. Downgrade attack.** This is the most serious. WireEng's negotiation
selects `max(intersection(probe.versions, local.versions))`. An active
man-in-the-middle can intercept the VersionProbe, strip all versions
except the lowest, and forward it. The responder selects the lowest
version. The initiator receives a VersionResponse selecting a version
it supports but did not prefer. The initiator cannot detect this
because the probe-nonce echo only proves the response corresponds to
*some* probe, not that the probe was received unmodified.

WireEng's nonce-echo mechanism prevents replay of old responses but
does not prevent modification of the probe itself. The probe is
unauthenticated. There is no MAC, no signature, no binding to the
peer's identity. Any on-path attacker can modify the version list.

SpecWriter's encrypted approach makes downgrade attacks impossible:
version ranges are exchanged inside the Noise-authenticated channel.
An attacker cannot read or modify them without breaking Noise.

**5. Denial of service by version poisoning.** An attacker floods
a peer with VersionProbes containing `version_count = 6` with all
invalid versions. The responder computes `max(intersection(...))`,
finds an empty intersection, and sends a response with
`selected_version = 0x0000`. The responder has done work (intersection
computation, nonce echo, packet construction) for zero benefit. At
high rates, this is a CPU and bandwidth denial of service. The fixed
magic makes filtering harder because legitimate probes also start
with the same magic.

### The "wasted handshake" argument is weak

WireEng's primary argument against encrypted negotiation is: "Two
peers with no common version will complete a Noise handshake before
discovering the incompatibility and aborting."

How often does this actually happen? In a household mesh of 2-12
clusters with ~36 peers, all running saalis, how many version
mismatches occur?

- During normal operation: zero. All devices run the same version.
- During a rolling upgrade: some devices are on v1, some on v2.
  They negotiate. Even with SpecWriter's approach, the handshake
  completes in 1-2 round trips (KK: 1 RTT, XX: 2 RTT). On a LAN,
  that is <1ms. The "wasted" handshake costs microseconds.
- On a WAN connection (inter-cluster over the internet): the handshake
  costs maybe 50-100ms. Still negligible compared to the session lifetime
  of hours or days.

The cost of "wasting" a handshake on version mismatch is measured in
microseconds on LAN and milliseconds on WAN. The cost of revealing
versions in plaintext is permanent metadata leakage and a downgrade
attack surface. The tradeoff does not favor plaintext.

### SpecWriter's approach needs one fix

The one legitimate weakness of encrypted negotiation: if the Noise
protocol name itself changes between versions (e.g., BLAKE2b in v1,
BLAKE2s in v2), the peers cannot even complete the handshake. But this
is a non-problem if the Noise protocol name is fixed forever and only
the application-layer version changes. The Noise pattern, DH, cipher,
and hash are baked into the protocol name string. Version negotiation
should control application-layer semantics (heartbeat format,
control log entry types, etc.), not the cryptographic primitives.

**Verdict on T1:** The winning position (plaintext negotiation) is
wrong. Encrypted version negotiation is safer and the performance
cost is negligible for verkko's deployment scenario. The downgrade
attack alone should disqualify plaintext negotiation.

## T2: BLAKE2b vs BLAKE2s -- Steelmanning BLAKE2b (SpecWriter)

WireEng chose BLAKE2s. The argument: 32-bit platforms (Raspberry Pi,
household devices). BLAKE2s is optimized for 32-bit. This sounds
reasonable on the surface. Let me attack it.

### What are verkko's actual target devices?

The saalis architecture document says: "Single Rust binary on
Batocera/loisto appliance." Batocera is a Linux distribution for
retro gaming. The target hardware:

- Raspberry Pi 4/5: **64-bit** ARM (AArch64). The Pi 4 has been
  64-bit since 2019. The Pi 5 is exclusively 64-bit.
- Intel NUCs: 64-bit x86_64.
- Mini PCs (AceMagician, Beelink, etc.): 64-bit x86_64.
- Old hardware (Pi 3, Pi Zero 2W): also 64-bit ARM (ARMv8-A).

The Raspberry Pi 3 and later all have 64-bit CPUs. The Pi Zero 2W
has a 64-bit CPU. Running a 32-bit OS on these is a choice, not a
hardware limitation. Batocera's official builds for Pi 4/5 are 64-bit.

There are no 32-bit-only devices in the verkko target deployment.
The BLAKE2s optimization advantage does not apply.

### BLAKE2b is faster on 64-bit

On AArch64 (Pi 4/5), BLAKE2b is approximately 20-40% faster than
BLAKE2s due to its 64-bit word size matching the native register
width. On x86_64 (NUCs, mini PCs), the advantage is similar or
larger, especially with AVX2 instructions.

BLAKE2s's 32-bit word size means it processes 32-byte blocks in
10 rounds. BLAKE2b's 64-bit word size means it processes 128-byte
blocks in 12 rounds. The throughput-per-cycle ratio favors BLAKE2b
on 64-bit hardware.

### The Noise specification's own position

The Noise Protocol Framework specification (revision 34) lists both
BLAKE2b and BLAKE2s as valid hash functions. It does not recommend
one over the other. But the WireGuard protocol -- the most successful
real-world deployment of Noise -- uses BLAKE2s with Noise_IKpsk2. The
WireGuard choice was driven by its target: Linux kernel space on
embedded devices, including 32-bit MIPS routers.

verkko is not WireGuard. verkko targets 64-bit household devices
running a full Linux userspace. The WireGuard rationale does not
transfer.

### Interoperability risk

The Noise protocol name string is hashed into the handshake state.
As the tension map notes, two peers using different protocol name
strings produce non-interoperable sessions that fail with an opaque
decryption error. This is the most dangerous kind of bug: silent,
hard to diagnose, no error message says "you are using the wrong
hash function."

Choosing the wrong variant is permanently damaging because changing
it later requires a major version bump (the protocol name string
changes, which changes the handshake state initialization). This
decision is effectively irreversible after v1 ships.

### Counterargument: what if a future device IS 32-bit?

If a future verkko device has a 32-bit-only CPU, BLAKE2b still works.
It is slower on 32-bit than BLAKE2s, but verkko's hash throughput
requirements are modest. The Noise handshake hashes at most a few
hundred bytes. Heartbeat hashing (36 bytes every 2-30 seconds) is
negligible. The only high-throughput hash use is BLAKE3 for content
addressing, which is a separate choice from the Noise hash.

The performance difference between BLAKE2b and BLAKE2s in the Noise
handshake on a 32-bit device would be measured in single-digit
microseconds. Undetectable by any human or system.

**Verdict on T2:** BLAKE2b is correct for verkko's target hardware.
The BLAKE2s choice is based on a false premise (32-bit devices) and
loses 20-40% hash performance for no benefit. Choose BLAKE2b and
lock it forever.

## T3: Connection ID Generation -- Steelmanning random_u32() (SpecWriter)

WireEng's BLAKE3-derived connection ID binds the ID to CIK public keys,
handshake ephemeral, and a session counter. This sounds rigorous and
professional. I will now argue it is overengineered for a 32-bit field.

### The BLAKE3 derivation provides no security benefit

The connection_id appears in the **plaintext** outer header. It is
readable by every transit peer and every on-path observer. Binding it
to key material does not make it secret -- it is still transmitted in
the clear. The "unpredictability for off-path observers" argument
from WireEng is valid for both approaches: a CSPRNG-generated random
u32 is equally unpredictable to an off-path observer.

For on-path observers, neither approach provides protection. The
connection_id is visible in every frame regardless of how it was
generated.

### BLAKE3 derivation introduces unnecessary coupling

WireEng's derivation requires:
- `local_CIK_pub` (32 bytes from the mesh layer)
- `remote_CIK_pub` (32 bytes from the mesh layer)
- `handshake_nonce` (32 bytes from the Noise handshake)
- `session_counter` (4 bytes, persisted across restarts)

This couples connection ID generation to:
1. The mesh layer (CIK keys). Protocol is not supposed to know about
   CIK. CIK is a mesh concept (verkko-mesh: CIK). Protocol depends
   on crypto, not mesh.
2. The Noise handshake state (ephemeral key). The connection ID
   cannot be generated until the handshake is at least partially
   complete.
3. A persistent counter that must survive restarts.

SpecWriter's `random_u32()` requires: a CSPRNG. That is it. No
layer violations, no persistence, no timing constraints.

### The "migration-safe" argument is illusory

WireEng argues: "The connection_id is bound to the session, not the
IP address. It survives NAT rebinding and WiFi-to-ethernet migration."

SpecWriter's `random_u32()` is also bound to the session, not the
IP address. The connection_id is assigned at session creation and
used for the session lifetime. It survives migration equally well
regardless of how it was generated. "Migration-safe" is a property
of the usage pattern (session-scoped ID), not the generation
algorithm.

### The collision analysis is identical

Both approaches produce a uniformly distributed 32-bit value. The
birthday collision probability is identical: ~50% at 2^16 = 65,536
concurrent IDs. WireEng's own analysis confirms this. The BLAKE3
derivation provides no collision resistance advantage over a CSPRNG.

### The session_counter introduces a persistence burden

WireEng's derivation requires persisting a monotonic counter across
restarts. If the counter file is lost or corrupted, what happens?
If two instances of the same peer share a counter, what happens? If
the counter wraps (u32 max is 4 billion), what happens?

A CSPRNG requires no persistence. It produces a fresh random value
every time. The only failure mode is a broken CSPRNG, which is a
catastrophic failure for the entire protocol (Noise handshakes also
need the CSPRNG).

### The DomainSeparator pollution is real

WireEng adds "saalis-connid-v1" to the DomainSeparatorRegistry for
this derivation. The DomainSeparator enum is a compile-time registry
that controls all BLAKE3 usage. Adding an entry for a 32-bit
convenience value dilutes the registry. The registry should contain
security-critical derivations (key material, nonces, authentication).
Connection IDs are not security-critical -- they are routing
convenience.

### When WOULD BLAKE3 derivation be justified?

If the connection_id were used for authentication or key derivation,
binding it to key material would be appropriate. If the connection_id
were 128+ bits and could serve as a session fingerprint, the
derivation might add value. For a 32-bit routing shortcut in a
plaintext header, the complexity is unjustified.

**Verdict on T3:** `random_u32()` is correct. The BLAKE3 derivation
adds complexity, layer violations, persistence requirements, and
DomainSeparator pollution for zero measurable security or collision
benefit. Simple is better than clever.

## T4: Scope -- Steelmanning the Savant (Include B1-B12)

The emerging consensus is to accept Run 10's 23 actions and defer
B1-B12. Three of four experts produced deliverables within the 23-action
scope. Only the Savant dissents. I will argue that the Savant is right
and the consensus is dangerously wrong.

### B1 (SansIO consumer interface) is not deferrable

The protocol is specified as "sans-IO: the protocol is a state machine.
The consumer provides the socket." This is stated in the Abstract. It is
an architectural invariant, not an optimization. Every other mechanism
in verkko-protocol -- handshakes, heartbeats, LEDBAT, SACK, streams --
assumes the sans-IO contract exists.

But the sans-IO contract does not exist. There is no `ProtocolInput`
enum. There is no `ProtocolOutput` enum. There is no `ProtocolMachine`
trait. There is no specification of what happens when the consumer calls
`handle(BytesReceived {...})`. The protocol's entire public surface is
a blank page.

Without B1, the Run 10 deliverables are a machine with no ignition
switch. The SpecWriter has specified wire formats for heartbeats,
SACK, connection IDs, streams, version negotiation. The WireEng has
specified byte-level formats for control log entries, invite tokens,
relay operations. All of this assumes a consumer can feed bytes in
and receive bytes out. But the mechanism for feeding and receiving
is unspecified.

An implementer reading the spec after Run 11 will know the exact
byte layout of every frame type. They will not know how to drive the
state machine that produces and consumes those frames. The wire
format is the internal representation. The consumer interface is the
external API. Specifying internals without the API is specifying a
CPU without an instruction set.

### B2 (intra-cluster communication) creates implementation divergence

The mesh spec has exhaustive inter-cluster detail: CPSK, broadcast
encryption, polygon schedule, coupled LEDBAT. Intra-cluster gets
one sentence: "reliable broadcast (iterated unicast) for Vital/Reactive
messages."

This means every implementer invents intra-cluster communication
from scratch. Does it use CIK for Noise KK? Does it use DEK pairwise
sessions? Does it use a different wire format? Is the heartbeat format
the same within a cluster as between clusters? None of this is
specified.

The Savant proposed a specific mechanism: Noise KK with CIK as static
key, producing pairwise session keys. This is 15 lines of pseudocode.
It resolves the ambiguity completely. Deferring it means accepting that
different implementations will make different choices and interop will
break at the intra-cluster boundary.

### B5 (matter chunking) blocks the system's purpose

The system exists to transfer matter. Matter can be 2GB. The transport
uses 1280-byte UDP segments. The bridge between 2GB and 1280 bytes is
unspecified.

Without B5, the system can catalogue content (cuckoo filters), detect
divergence (Merkle reconciliation), and negotiate who to download from
(source selection). But it cannot actually download. The entire matter
transfer pipeline -- the reason verkko exists -- terminates at a gap.

The bao tree is mentioned. BLAKE3 verified streaming is mentioned. But
the chunk size, the transfer session lifecycle, the chunk request/
response format, the multi-source interleaving strategy, the resumption
protocol -- all of this is blank.

### B12 (connection teardown) is a protocol completeness issue

Connections are established via Noise handshakes. Connections are
migrated via PATH_CHALLENGE/PATH_RESPONSE. But connections never end.
There is no CLOSE message. There is no GOAWAY. There is no half-close.
There is no specification for what happens to in-flight frames when a
connection terminates.

This is not an edge case. Every connection eventually ends. If the
teardown is unspecified, implementations will disagree on resource
cleanup, session state invalidation, and reconnection timing. One
implementation might wait 60 seconds for a timeout; another might
immediately reconnect. The resulting behavior will be unpredictable
and potentially resource-exhausting.

### Can Run 11 address 23 + 12 = 35 actions?

No. That is the counterargument. But the counterargument proves the
Savant's deeper point: if Run 11 cannot address the most critical
gaps (B1, B2, B5, B12), and Run 11 is the current review cycle, then
when DO they get addressed? The review process is designed to converge.
Each run should reduce gaps, not defer them.

The correct approach is triage: drop the lowest-priority items from
the 23 actions to make room for B1, B2, B5, and B12. Not all 12 gaps
are equally urgent. But the top 4 (B1, B2, B5, B12) are more critical
than several of the 23 actions (P0-4 rename convergence fronts, P0-6
ScarSeverityTier, P2-6 health pipeline dimensions are important but
not blocking).

**Verdict on T4:** The Savant is right that B1, B2, B5, and B12 are
more dangerous than some of the accepted 23 actions. The consensus
should triage: include B1, B2, B5, B12; defer P0-4, P0-6, and P2-6
to a future run. This keeps the total action count manageable while
addressing the most critical gaps.

## T5: HostPeer Translation Layer -- Steelmanning the Savant

SpecWriter's seat spec is self-contained at the seat-protocol boundary.
It specifies SeatRequest, SeatResponse, SeatEvent, SeatInstruction with
wire formats. From the seat's perspective, the spec is complete. The
emerging consensus may accept this as sufficient. I will argue it is not.

### The HostPeer is where every ambiguity concentrates

The seat spec says: "The host peer translates between the seat's
request/response world and the mesh's collective state." But the
translation is unspecified. Consider the concrete operations:

**"List my games" (SeatRequest type: QUERY)**
The host peer must:
1. Read the seat's role to determine access control.
2. Query the local hot store for entities matching the query.
3. Filter by channel access (the seat's user has access to certain
   channels; games in other channels are invisible).
4. Return protobuf-encoded entity metadata.

Step 3 requires the HostPeer to know about channels (a mesh concept).
The seat spec cannot describe this because "the seat does not know
about the mesh." But if the HostPeer's channel-aware filtering is
unspecified, implementers will:
- Some will filter on the HostPeer side (correct but unspecified).
- Some will return all entities and filter on the seat side (wrong:
  leaks channel membership information to the seat).
- Some will require the seat to specify a channel in the request
  (wrong: the seat does not know about channels).

**"Download this ROM" (SeatRequest type: DOWNLOAD)**
The host peer must:
1. Check that the seat's role permits download.
2. Locate the matter (cuckoo filter lookup across mesh peers).
3. Initiate a MatterTransfer from the best source.
4. Stream the matter to the seat via the seat's session.
5. Send SeatProgress events during the transfer.
6. Issue a SeatInstruction::PLACE to put the file in the right
   directory on the seat's device.

Steps 2-3 require mesh operations (source selection, matter transfer).
Step 4 requires bridging from the mesh's bulk-tier stream to the
seat's session stream. Step 6 requires the HostPeer to know the
seat's filesystem layout.

None of this is specified. The seat spec says "the host peer handles
all mesh operations on behalf of the seat." But which mesh operations,
in what order, with what error handling?

### Without the translation spec, seat is untestable

The seat spec defines wire formats and error codes. But a conformance
test for the seat protocol requires testing the HostPeer's behavior.
If the HostPeer translation is unspecified, there is no conformance
test. Two implementations can have identical seat-side code and
completely different HostPeer behavior.

The Savant's cartographic metaphor is precise: "The bridge is drawn
on the map, but neither territory claims the bridge as its own.
Travelers arriving at the bridge find no customs office on either
side." The HostPeer is where seat and mesh meet. If nobody specifies
what happens at the meeting point, the meeting point dissolves into
ad-hoc implementation.

### Where the translation spec belongs

The SpecWriter argues the seat spec is complete "at the seat-protocol
boundary." This is a valid modular design position. But it leaves the
question: where DOES the translation spec live?

- Not in verkko-seat (seat does not know mesh).
- Not in verkko-mesh (mesh does not know seat).
- Not in verkko-protocol (protocol is point-to-point, not application-layer).

The answer is: it lives in a new boundary document, or it lives in
verkko-mesh as a "seat-serving" subsection. The HostPeer IS a mesh
peer. Its seat-serving behavior is part of its mesh behavior. The
translation is a mesh operation triggered by seat requests. It
belongs in mesh.

The Savant identified this correctly: "It is not in seat (seat does
not know mesh). It is not in mesh (mesh does not know seat). It lives
in the application layer, outside all seven documents." The solution
is to put it in mesh, because the HostPeer is already a mesh entity,
and mesh already defines all the operations the HostPeer needs to
perform.

**Verdict on T5:** The Savant is right. The seat spec is incomplete
without the HostPeer translation layer. The translation belongs in
verkko-mesh as a new section defining how a mesh peer serves seat
requests using mesh operations. Without it, the seat spec is a
facade with nothing behind it.

## Weaknesses in the Emerging Consensus

Beyond the five tensions, I find at least seven weaknesses that no
expert has identified.

### W1: The magic byte collision with other protocols

WireEng uses `0x56524B4F` ("VRKO") as the magic for all verkko
datagrams -- version probes, handshake frames, and (presumably)
data frames. The SpecWriter uses different magic values per message
type: `0x564B4B4B` ("VKKK"), `0x564B4B52` ("VKKR"), `0x56585831`
("VXX1"), etc.

Neither expert has addressed the interaction. If both systems coexist
in the final spec, what magic does the responder check first? The
responder receives a UDP datagram. The first 4 bytes are:
- `0x56524B4F`: WireEng version probe or handshake frame.
- `0x564B4B4B`: SpecWriter KK message 1.
- `0x56585831`: SpecWriter XX message 1.

But wait -- the SpecWriter's handshake format uses a different magic
per message type, while the WireEng's handshake format uses a unified
magic with a `frame_type` byte. These are incompatible frame formats.
The SpecWriter has a 7-byte header (magic + version + payload_len).
The WireEng has an 18-byte header (magic + frame_type + hs_type +
hs_len + initiator_cid + responder_cid + msg_index + reserved).

No expert has reconciled these formats. They are proposing different
wire protocols. The final spec must choose one, but the tension map
does not identify this as a tension. It is treated as an implementation
detail when it is actually a fundamental incompatibility in the frame
format.

### W2: The SACK format is specified twice, incompatibly

SpecWriter specifies SACK at the stream level (P1-3) with a 9-byte
fixed header and SACK blocks. WireEng specifies SACK at the frame
level (Section 4) with a 14-byte fixed header, different field ordering,
and a piggybacked SACK mechanism using bit 7 of msg_type.

The SpecWriter's SACK has: cumulative_ack (4), block_count (1),
sack_blocks (var), recv_window (4), recv_buffer_pct (1), recv_rate_kbps
(2). Total fixed: 12 bytes (at block_count=0).

The WireEng's SACK has: cumulative_ack (4), ack_timestamp (4),
ack_delay (2), block_count (1), recv_buffer_pct (1), recv_rate_kbps
(2). Total fixed: 14 bytes.

Key differences:
- WireEng includes ack_timestamp and ack_delay (8 bytes). SpecWriter
  does not.
- SpecWriter includes recv_window (4 bytes). WireEng does not.
- WireEng's piggybacked SACK uses bit 7 of msg_type. SpecWriter's
  SACK uses a flags byte in the stream header with SACK_PRESENT at
  bit 2.

Two different SACK formats from two experts, both claiming to specify
the definitive wire format. Both internally consistent. Both incompatible
with each other. Nobody noticed because each expert wrote in isolation.

### W3: The Prologue binding is inconsistent

WireEng specifies the Noise prologue as:
```
prologue = MAGIC || selected_version || hs_type
         = 0x56524B4F || u16(version) || u8(hs_type)
         = 7 bytes
```

SpecWriter does not specify a prologue at all. The SpecWriter's
handshake format uses per-message-type magic values ("VKKK", "VKKR",
"VXX1", etc.), which means the prologue would need to be either:
- The specific magic for that message type (but which one? there are
  5 different magics across KK and XX messages).
- Something else entirely.

The Noise specification requires both peers to agree on the prologue
byte-for-byte, or the handshake will fail with an opaque decryption
error. If the SpecWriter and WireEng approaches are mixed in any way,
the prologue will not match and every handshake will fail.

This is not a theoretical concern. The prologue is hashed into the
Noise handshake state during initialization. A one-byte difference
produces completely different handshake keys. The result is a
handshake that completes the DH operations but fails to decrypt any
authenticated payload, with no useful error message.

### W4: The heartbeat state byte bit allocation has a silent conflict

The SpecWriter's HeartbeatExtension draft (P0-3) specifies:
```
Bit 2: gossip_mode        0 = Plumtree, 1 = flood.
Bit 3: convergence_active 0 = normal, 1 = convergence in progress.
Bit 4: admin_flag         0 = non-admin, 1 = current admin.
Bits 0-1: metabolic state (protocol-layer).
Bits 5-7: reserved.
```

The existing verkko-protocol spec (Section 3.4) specifies:
```
bits 0-1: metabolic state
bit 2: gossip_mode
bit 4: admin flag
```

Bit 3 is not mentioned in the existing spec. The SpecWriter adds
`convergence_active` at bit 3. This is new information that the
HealthPipeline two-mode state machine (P0-8) depends on: "Remote
peers detect convergence via the convergence_active bit in the
HeartbeatExtension state byte."

But the ArchSurgeon's HeartbeatExtension defined concept (OP-4)
specifies the gossip_mode flag as "bit 2 of state byte" and says
nothing about a convergence_active bit. The ArchSurgeon's ownership
table lists the `state` byte as "(metabolic + gossip_mode + admin)"
with no convergence_active.

So: the SpecWriter's P0-3 adds convergence_active at bit 3. The
ArchSurgeon's OP-4 does not include convergence_active. The
SpecWriter's P0-8 depends on convergence_active existing. If the
ArchSurgeon's version is taken as authoritative (no bit 3), the
entire DAMPED mode distributed coordination mechanism breaks.

This is a silent conflict because both experts claim to be specifying
the same HeartbeatExtension, but they assign different semantics to
the state byte. Nobody noticed because the ArchSurgeon focused on
structure and the SpecWriter focused on semantics.

### W5: All four experts assume UDP, but the seat is LAN-only

The SpecWriter specifies that seats connect via Noise KK or XX
handshake. The existing seat spec says seats do not connect over WAN
("LAN only; seats do not connect over WAN"). The WireEng's entire
wire format is built around UDP datagrams.

But the seat is a web UI served by axum. The web UI communicates via
HTTP/WebSocket/SSE, not raw UDP. The seat-to-host-peer connection is
not a peer-to-peer UDP connection. It is a web client connecting to
an HTTP server.

The SpecWriter's seat authentication wire format (P2-1) specifies
byte-level formats "inside Noise XX handshake payload." But the
saalis web layer uses axum (HTTP). The seat's transport is
HTTP-over-TCP, not the verkko UDP protocol.

How does the seat's Noise-encrypted request/response map to
HTTP requests? Does the seat open a WebSocket and send verkko frames
through it? Does it use raw UDP despite being a web browser? Web
browsers cannot send raw UDP.

This is not addressed by any expert. The implicit assumption is
that the seat uses the same UDP transport as mesh peers, but the
architecture says the seat is a web client and the host peer runs
an axum HTTP server. These are contradictory transport assumptions.

The seat spec may need to specify a different transport binding:
verkko frames encapsulated in WebSocket messages, or a REST-like
mapping of SeatRequest/SeatResponse to HTTP verbs. None of the
four experts have raised this.

### W6: The invite token carries the PSK in plaintext (in the QR code)

WireEng specifies the invite token with a 32-byte PSK at offset 40.
SpecWriter also specifies a 32-byte PSK. Both agree: the PSK is
included in the invite token, which is encoded in a QR code.

The QR code is displayed on a screen. Anyone who photographs the
screen, reads the QR code, or intercepts the deep link has the PSK.
The PSK is the sole authentication for the Noise XX handshake. An
attacker who obtains the QR code can impersonate the invitee and
enroll a malicious device.

Both experts specify "single-use" tokens with nonce replay prevention.
But "single-use" means the token works once. If the attacker uses
it before the legitimate invitee, the attacker is enrolled and the
invitee gets TOKEN_REPLAYED. The attacker is now a trusted peer in
the mesh.

The mitigation is SAS verification (Short Authentication String).
But both experts specify that SAS verification happens *after* the
handshake. WireEng: "key delivery is SAS-gated per enrollment rules."
SpecWriter: "SAS verification required before channel keys are
delivered." The attacker completes the handshake, enters "Active
state but channel keys are withheld," and now the admin must
actively notice that the wrong device enrolled. On a headless
Batocera appliance, the admin may not notice for hours or days.

This is a known limitation of QR-based enrollment, but no expert
has proposed a mitigation beyond "SAS verification." A time-limited
PSK with a 5-minute window would reduce the attack surface
significantly. Neither expert specifies a time limit on PSK
validity (only an expiry_epoch on the token, which is macro-epoch
granularity -- hours, not minutes).

### W7: The ResourcePressures struct uses f64 in a no-heap protocol

The SpecWriter's MetabolicGovernor contract (P0-2) specifies:

```rust
struct ResourcePressures {
    values: [f64; N],  // N defined by implementation
}
```

This uses `f64` (floating-point) values in a protocol that targets
embedded-ish Linux devices. Floating-point is not forbidden by the
verkko spec, but it introduces platform-dependent behavior:

- Different FPU implementations may produce different results for
  the same inputs (denormals, rounding modes).
- The governor's `evaluate()` function takes `elapsed_since_last_transition`
  as a `TimeDelta`. If this is also floating-point, NaN propagation
  can cause the governor to silently produce invalid states. The
  SpecWriter addresses this: "All pressure values are NaN -> retain
  current state." But what about a single NaN in an array of 8
  pressure values? The SpecWriter says "All pressure values are NaN"
  (all, not any). A single NaN mixed with valid values would pass
  the NaN check and produce undefined governor behavior.

The broader issue: four experts are writing spec text that mixes
IEEE 754 floating-point with deterministic protocol behavior. The
EMA coefficients (0.1, 0.01, 0.3), the decay weights, the scar
tier weights -- all use floating-point. On different platforms, these
computations may produce different results in the least significant
bits. For health scoring and dominance computation, small differences
in floating-point rounding can cause different peers to compute
different health classifications for the same observations.

This is a cross-cutting concern that no expert has addressed: should
the protocol use fixed-point arithmetic for all health/scoring
computations to ensure deterministic cross-platform behavior?

### W8: The ArchSurgeon's "consumer" pattern creates orphaned prose

The ArchSurgeon converts crypto's back-references from "see
verkko-mesh: X" to "the consumer (verkko-mesh)." This eliminates
the dependency direction violation. But it creates a new problem:
the crypto document now says "the consumer" five times without
defining what "consumer" means in this context.

A reader of verkko-crypto who does not already know about verkko-mesh
will encounter:
- "DK and DEK are bound to a device at enrollment time; the enrollment
  ceremony is defined by the consumer (verkko-mesh)."
- "Mesh initialization using these keys is the responsibility of
  the consumer (verkko-mesh)."
- "The enrollment and key delivery flow is the responsibility of
  the consumer (verkko-mesh)."

Each of these says "(verkko-mesh)" parenthetically. If a new consumer
of crypto appears (say, a different mesh implementation), these
parentheticals become misleading. The ArchSurgeon's pattern works for
a single consumer but does not generalize.

More importantly, the prose still names "verkko-mesh" five times. The
declared dependency graph says crypto has no dependencies. The prose
still names a dependent document five times. The structural dependency
is eliminated; the cognitive dependency remains. A reader following
the "(verkko-mesh)" breadcrumbs will read mesh as if it were a
dependency of crypto, which is the inverse of the DAG.

This is not a blocking problem, but it is a design smell that the
ArchSurgeon's surgery did not fully resolve.

Three of the undiscovered weaknesses (W1, W2, W3) are critical
wire-format incompatibilities that would cause silent handshake
failures between implementations following different expert
proposals. These must be resolved before any spec text is merged.

The cross-review should focus on:
1. Reconciling the magic/frame-format conflict (W1).
2. Choosing one SACK format (W2).
3. Specifying the prologue exactly once, in one place (W3).
4. Resolving the state byte bit 3 assignment (W4).
5. Addressing the seat transport binding (W5).

Without resolving these five, the four expert deliverables cannot
be composed into a coherent specification.

## A Note on Process

The four experts produced excellent work in isolation. The SpecWriter
wrote comprehensive normative text. The WireEng wrote precise wire
formats with test-vector-ready detail. The ArchSurgeon performed clean
structural surgery. The Savant mapped unmapped territory.

The problem is composition. Each expert optimized for internal
consistency within their own deliverable. Nobody optimized for
cross-expert consistency. The result is four internally consistent
documents that contradict each other on critical wire format decisions.

This is not a failure of any expert. It is a failure of the stage
structure: Stage 1 produces four independent deliverables without a
shared frame format contract. The tension map (Stage 1.5) identified
content disagreements (which hash, which timing) but missed format
disagreements (which header layout, which SACK encoding, which
magic bytes).

Stage 2 must resolve the format conflicts before resolving the
content disagreements. A consistent frame format is the prerequisite
for everything else.
