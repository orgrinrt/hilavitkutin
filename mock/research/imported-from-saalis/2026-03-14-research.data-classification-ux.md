# Data Classification, Flexibility, and User Experience: Lessons from Notion, Coda, Fibery, Obsidian, Logseq, and Emacs Org-mode

**Date:** 2026-03-14
**Context:** Research for saalis, a content library manager with a flexible entity/metadata data model, DataViews, actions, filters, and sorts.

---

## 1. Notion

### The Block-Based Data Model

Notion's core architectural insight is that everything --- text, images, lists, database rows, pages --- is a **block**. Despite differing front-end representations, every element is stored as a block entity in PostgreSQL with a consistent structure, schema, and associated metadata. Blocks use "downward pointers" to refer to their children, forming a **render tree** that different block types render in different ways. This atomic, graph-like data model gives users the ability to customise how information is moved, organised, and shared.

At scale, Notion manages over 200 billion block rows. Their Postgres database is sharded across 96 physical instances with 480 logical shards. Critically, 90% of upserts are *updates* to existing blocks, not inserts --- a data pattern that challenges traditional data warehouses and is relevant to any content library where metadata enrichment dominates over creation.

**Source:** [The Data Model Behind Notion](https://www.notion.com/blog/data-model-behind-notion)

### Database Views and Properties

Notion offers six view types for any database: **Table, Board (Kanban), Timeline (Gantt), Calendar, Gallery, and List**. Each view is a lens on the same underlying data, and users can create multiple views with independent filters, sorts, and groupings. This "one source, many views" pattern is the single most important UX concept for saalis to adopt.

The **property system** supports types including text, number, select, multi-select, date, person, file, checkbox, URL, email, phone, formula, relation, and rollup. Relations connect rows across databases; rollups aggregate data from related rows. Formulas compute derived values from other properties.

**Linked databases** allow the same database to appear on multiple pages with different filters and layouts. Changes in any view propagate to the source, maintaining a single source of truth. This is directly analogous to saalis's DataView concept --- typed query projections over a shared entity store.

**Source:** [Views, Filters, Sorts & Groups](https://www.notion.com/help/views-filters-and-sorts), [Using Linked Databases](https://www.notion.com/help/guides/using-linked-databases)

### What Makes It Intuitive

Notion's intuitiveness stems from three design choices:

1. **Progressive onboarding:** Instead of a blank workspace, Notion asks "What will you use Notion for?" and preloads relevant templates. Users who start with templates have significantly higher retention than those starting from scratch.
2. **Slash commands:** Typing `/` reveals available block types, teaching the system through usage.
3. **Direct manipulation:** Blocks can be dragged, transformed (e.g., text to toggle, list to database), and nested freely. The interface teaches its own capabilities.

**Source:** [How Notion Solved the Blank Page Problem](https://onboardme.substack.com/p/how-notion-solved-the-blank-page-product-strategy-deepdive), [Notion's Onboarding](https://www.candu.ai/blog/how-notion-crafts-a-personalized-onboarding-experience-6-lessons-to-guide-new-users)

### What Frustrates Power Users

1. **Performance cliffs:** Databases degrade noticeably beyond 5,000 rows; severe slowdowns appear at 10,000--20,000 rows. Complex formulas depending on rollups that depend on other formulas create cascading computation. This is a fundamental architectural constraint of the block-per-row model.
2. **Rollup limitations:** You cannot roll up a rollup. Nested aggregations require workarounds.
3. **Formula language:** Notion formulas are powerful but opaque to non-technical users. There is no debugger, no autocomplete for complex expressions, and error messages are unhelpful.
4. **Relation limit:** Approximately 1,000 related items per relation before performance suffers.

**Source:** [Optimize Database Load Times](https://www.notion.com/help/optimize-database-load-times-and-performance), [Pushing Notion to the Limits](https://notionmastery.com/pushing-notion-to-the-limits/)

## 2. Coda

### Tables as First-Class Documents

Where Notion is wiki-first with databases bolted on, Coda is **document-first with tables as the core primitive**. A Coda doc is a living application: tables, buttons, automations, and formulas coexist in a single surface. Tables are not separate databases --- they are sections of the document with full formula access.

This means Coda's formulas can appear nearly everywhere, not just in column definitions. Formulas can reference other tables, canvas controls, and external services. Buttons can execute multi-step actions: update rows, send emails, trigger webhooks, press other buttons.

**Source:** [Coda vs Notion: Formulas](https://coda.io/blog/tool-consolidation/coda-vs-notion-formulas), [Coda vs Notion: Automations and Buttons](https://coda.io/blog/tool-consolidation/coda-vs-notion-buttons-automations)

### Cross-Doc Packs

Coda's **Packs** are integrations that bring external data into tables (e.g., pulling GitHub issues, Slack messages, or API endpoints into queryable rows). Cross-doc packs allow tables in one document to reference tables in another, creating a federation layer.

This is relevant to saalis's plugin architecture: Packs demonstrate that external data sources (scrapers, metadata APIs, download services) should surface as first-class table-like entities, not second-class imports.

### Flexibility vs. Structure: Lessons Learned

Coda's learning curve is significantly steeper than Notion's once you go beyond basic notes. The power of "everything is a formula" creates confusion:

1. **The spreadsheet mental model breaks down.** Users expect columns to behave like Excel, but Coda formulas can have side effects (via buttons and automations), making the system harder to reason about.
2. **Discoverability suffers.** With formulas available everywhere, users struggle to find where logic lives. Debugging requires understanding the entire document's formula dependency graph.
3. **Teams bifurcate.** The person who builds the Coda doc becomes a single point of failure. Non-technical team members use it but cannot modify it. Coda's own product research acknowledges the risk of over-indexing on advanced users' needs.

The key takeaway: **formula power should be layered, not pervasive.** The base experience must work without formulas; advanced logic should be opt-in.

**Source:** [How Coda Builds Product](https://www.lennysnewsletter.com/p/how-coda-builds-product), [Coda: PM's Guide to User Research](https://coda.io/blog/product-teams/pm-guide-to-research-design-principles)

## 3. Fibery

### Entity-Relationship Model

Fibery is the closest analogue to saalis's architecture among these tools. Its core abstraction is **custom entity types connected by typed relations**. You define Databases (entity types) with custom fields, then connect them via relations that are visible from both ends. A CRM space might have Accounts and Contacts; a product space might have Features, Milestones, and Bugs. Everything is connectable to everything else.

This is directly parallel to saalis's `EntityType` trait, where games, movies, users, and other entities all live in a unified `entities` table with typed metadata. Fibery validates that this approach works at the product level.

**Key architectural decisions in Fibery that saalis should note:**

- **Relations are the core of the workspace.** Not an afterthought or an advanced feature --- they are the primary organisational mechanism.
- **Many-to-many is native.** Unlike Notion, where many-to-many requires workaround junction databases, Fibery supports it directly.
- **Hierarchies emerge from relations.** You define that Features contain User Stories, that Projects contain Tasks. The hierarchy is not hardcoded; it is a consequence of relation configuration.

**Source:** [Custom Domain & Connected Processes](https://fibery.com/features/custom-domain), [How to Design a Database in Fibery](https://fibery.com/blog/guides/how-to-design-a-database-in-fibery/)

### Views and Reports

Fibery offers **Board, Table, List, Timeline, Calendar, Feed, Chart, Map, Form, and Entity views** --- the widest view selection of any tool reviewed. Each view can be filtered, sorted, and grouped independently. Smart Folders allow embedding filtered views inside entity pages, enabling contextual dashboards.

The breadth of views demonstrates a principle: **different data contexts demand different visual representations**, and a content library must offer at least gallery (for cover art), table (for metadata comparison), and board (for status tracking) as first-class views.

**Source:** [Customizable Views](https://fibery.com/features/customizable-views)

### Knowledge Organisation Philosophy

Fibery's essay "The Knowledge Organisation" argues that a **single generic hierarchical network** can hold all of an organisation's knowledge. Each node has a type; types form hierarchies and cross-references. This rejects both pure folder hierarchies (too rigid) and pure tag clouds (too loose) in favour of typed graphs.

This aligns with saalis's design: entities with typed metadata and relations form a knowledge graph where games relate to platforms, genres, series, and download sources. The graph is the organising principle, not folders or tags.

**Source:** [The Knowledge Organisation](https://fibery.com/blog/gems/the-knowledge-organization/), [Fibery Philosophy](https://fibery.io/blog/essays/fibery-philosophy/)

## 4. Obsidian

### Dataview: SQL-Like Queries Over Markdown

Obsidian's Dataview plugin is the most instructive example of **querying structured metadata within an unstructured medium**. It indexes all markdown files, extracting:

- **YAML frontmatter** properties (typed key-value pairs at the top of files)
- **Inline fields** using `Key:: Value` syntax anywhere in the document
- **Implicit metadata** like tags, links, creation date, and file path

The Dataview Query Language (DQL) is SQL-inspired but operates on files rather than rows:

```
TABLE rating, genre, platform
FROM "games"
WHERE rating > 8
SORT rating DESC
```

For advanced users, JavaScript queries provide full programmatic access. This three-tier approach (simple queries, DQL, JavaScript) is a model of progressive disclosure applied to a query system.

**Source:** [Dataview Documentation](https://blacksmithgu.github.io/obsidian-dataview/), [Dataview GitHub](https://github.com/blacksmithgu/obsidian-dataview)

### Properties and Frontmatter

As of Obsidian 1.9+, the **Properties** system provides vault-wide property management with typed values. Renaming a property through the Properties view updates all occurrences across the vault. Properties power search, filters, Dataview queries, and the newer Bases feature (Obsidian's native database views).

The lesson: **structured metadata should be easy to add, easy to query, and easy to refactor.** Obsidian's approach of storing metadata inline with content (frontmatter) and providing vault-wide refactoring tools is user-friendly for power users. But the requirement to manually type YAML is a barrier for casual users --- saalis should provide GUI-based property editing with the structured storage happening behind the scenes.

**Source:** [Properties - Obsidian Help](https://help.obsidian.md/properties)

### Emergent Structure from Tags, Links, and Folders

Obsidian's organisational philosophy is **emergent rather than prescribed**. Users choose their own mix of:

- **Folders** for coarse categorisation
- **Tags** for cross-cutting concerns (e.g., `#to-play`, `#completed`)
- **Links** for semantic relationships between notes
- **Properties** for structured, queryable metadata

This flexibility is powerful but creates the "methodology paralysis" problem --- new users spend more time deciding how to organise than actually organising. The community has developed dozens of competing organisational systems (PARA, Zettelkasten, Johnny Decimal, etc.).

For saalis, the lesson is: **provide a strong default organisational scheme** (entity types with standard metadata fields) while allowing user-defined properties and tags for personalisation. Do not force users to invent their own schema from scratch.

### Community Plugins for Data Organisation

The plugin ecosystem includes Kanban (markdown-backed boards), Database Folder, DB Folder, and dozens of others that attempt to turn Obsidian into a database tool. The Kanban Status Updater plugin automatically syncs note properties when cards move between columns --- a pattern directly applicable to saalis's download status boards.

**Source:** [Obsidian Plugins](https://obsidian.md/plugins), [Obsidian Kanban](https://github.com/mgmeyers/obsidian-kanban)

## 5. Logseq

### Outline-Based Structure with Properties

Logseq takes an **outliner-first** approach where every piece of content is a block in a hierarchical outline. Properties attach to blocks or pages using `key:: value` syntax. Child blocks inherit their parents' tags, creating implicit structure through nesting.

The outliner model has a distinctive strength: **context is preserved through hierarchy**. A game's metadata, notes, and tasks naturally nest under the game's block. This is more intuitive for note-taking than Notion's flat database rows, but less suitable for structured querying.

### Queries: Simple and Advanced

Logseq provides two query tiers:

1. **Simple queries** use a visual builder to filter by tags, properties, and status. These are accessible to non-technical users.
2. **Advanced queries** use Datalog (a logic programming language), which is extremely powerful but has a notoriously steep learning curve. Building real-world catalogue queries with custom block properties requires understanding Datalog patterns, rules, and bindings.

The DB version of Logseq (beta as of late 2025) introduces a proper database backend with typed properties (Text, Number, Date, Checkbox, URL, Node). This moves Logseq closer to a structured data tool, but advanced query capabilities were in flux during the transition.

**Source:** [Logseq Properties Limitations](https://discuss.logseq.com/t/what-are-the-limitations-to-using-logseq-properties-to-make-a-lightweight-queryable-graph-database-of-pages/7651), [Logseq DB Version](https://github.com/logseq/docs/blob/master/db-version.md)

### What Works vs. What Is Clunky

**Works well:**
- Daily journal workflow with properties for tracking habits, tasks, and references
- Tag-based organisation with inherited structure
- Simple queries for TODO management and page lists

**Clunky:**
- Custom block property queries require Datalog expertise
- Performance degrades with too many query results
- The outliner model makes tabular views feel bolted-on rather than native
- Mixing structured data (properties) with unstructured content (outline blocks) creates ambiguity about where data lives

For saalis, Logseq demonstrates the danger of **two-tier query systems where the gap between tiers is too large**. Simple queries handle 60% of needs; advanced queries handle 95% --- but the 35% gap requires a qualitative skill jump (learning Datalog). Saalis's filter/sort system should have a smooth gradient from basic to advanced.

**Source:** [Advanced Logseq Queries](https://www.eriksuniverse.com/advanced-logseq-queries-real-world.html), [Logseq Views and Query System](https://deepwiki.com/logseq/logseq/4.5-views-and-query-system)

## 6. Emacs Org-mode

### The Properties Drawer System

Org-mode stores structured metadata in **property drawers** immediately below headlines:

```org
* Game Title
:PROPERTIES:
:platform: SNES
:genre:    RPG
:rating:   9
:status:   completed
:END:
```

Properties are inherited down the outline hierarchy unless overridden. This inheritance model is powerful for defaults: set `:platform: SNES` on a parent heading and all child entries inherit it.

**Source:** [Org Properties and Columns](https://alphaalgorithms.github.io/2019/05/22/emacs-properties-and-columns/)

### Column View

Column view overlays a **tabular display** on the outline, turning each headline into a row with columns derived from properties. Users define column formats specifying which properties to display and their widths. This transforms a hierarchical document into an ad-hoc spreadsheet --- the original "view switching" concept, predating Notion by over a decade.

Column view works because it does not replace the outline; it is a **temporary overlay** that can be toggled on and off. The underlying data model (outline with properties) remains authoritative. This principle --- views as overlays, not transformations --- is directly applicable to saalis's DataView system.

**Source:** [Org Column View Tutorial](https://orgmode.org/worg/org-tutorials/org-column-view-tutorial.html)

### Agenda Views

Org-mode's agenda system collects items from multiple files based on **tags, properties, TODO states, and dates**, then presents them in filtered, sorted views. Custom agenda commands allow users to define named views with specific filter criteria. The agenda is essentially a saved DataView query.

The power comes from composability: agenda commands can combine tag filters (`+work-meeting`), property matchers (`PRIORITY="A"`), and date ranges into a single view. This is the most mature implementation of "filtered views over heterogeneous data" in any tool reviewed.

**Source:** [Agenda Commands](https://orgmode.org/manual/Agenda-Commands.html), [Emacs Agenda Views](https://alphaalgorithms.github.io/2019/05/17/emacs-agenda-views/)

### Why It Works Despite Complexity

Org-mode succeeds for power users because:

1. **No complexity ceiling.** The system can be as simple as a text file with asterisks for headlines, or as complex as a full project management system with custom Elisp functions. There is no point where users "outgrow" the tool.
2. **Plain text as foundation.** Every feature is a layer on top of readable plain text. If all extensions disappeared, the data would still be useful. This is the ultimate portability guarantee.
3. **Composable primitives.** Properties, tags, TODO states, dates, and links are orthogonal features that combine freely. Users build their own systems from these primitives rather than choosing between pre-built templates.
4. **Keyboard-driven UX.** Every action has a keybinding. The lack of a GUI is a feature for users who value speed over discoverability.

The lesson for saalis: **power users value composability and no-ceiling depth more than polish.** The system should expose primitives (entity types, properties, filters, sort criteria, view types) that combine freely, even if it means the most advanced configurations require some learning.

**Source:** [The Right Way to Use Org Mode](https://karl-voit.at/2021/08/30/the-org-mode-way/), [Org Mode for Beginners](https://orgmode.org/worg/org-tutorials/org4beginners.html)

### org-roam

org-roam adds a knowledge graph layer: each note gets a unique ID, and bidirectional links are tracked in a SQLite database. The org-roam-ui package provides an interactive graph visualisation in the browser. This demonstrates that **graph visualisation is a complementary view**, not a replacement for text or table views --- useful for discovering connections but not for daily workflow.

**Source:** [org-roam](https://www.orgroam.com/), [org-roam-ui](https://github.com/org-roam/org-roam-ui)

## 7. Common Patterns Across All Tools

### Flexible Schemas Without Overwhelming Users

Every tool reviewed grapples with the tension between schema flexibility and user overwhelm. The successful patterns:

| Strategy | Used by | Mechanism |
|---|---|---|
| **Typed properties with GUI pickers** | Notion, Fibery | Drop-down to select property type; no raw schema editing |
| **Templates as starting points** | Notion, Fibery, Coda | Pre-configured schemas for common use cases |
| **Inline metadata** | Obsidian, Logseq, Org-mode | Properties co-located with content, reducing context-switching |
| **Schema-on-read** | Obsidian Dataview | No schema definition required; metadata is extracted from existing content |
| **Progressive query complexity** | Obsidian (DQL tiers), Logseq (simple/advanced) | Basic filters for everyone, full query language for power users |

### View Switching

All six tools support multiple views over the same data. The common view types and their purposes:

| View | Purpose | Best for |
|---|---|---|
| **Table** | Dense metadata comparison | Power users; bulk editing |
| **Board/Kanban** | Status-based workflow | Tracking progress (download status, play status) |
| **Gallery/Grid** | Visual browsing with thumbnails | Cover art-heavy content (games, movies) |
| **List** | Compact, scannable | Quick reference, search results |
| **Calendar** | Time-based organisation | Release dates, play schedules |
| **Timeline** | Duration and dependency tracking | Project management; less relevant for content libraries |

### Filtering and Sorting UX

Best practices distilled from all tools:

1. **Inline filter chips** (Notion, Fibery): Applied filters are visible as removable chips above the view, providing immediate feedback about what is filtered.
2. **Column-header controls** (Notion, Coda): Clicking a column header to sort or filter leverages the spreadsheet mental model users already have.
3. **Natural language labels** (all): "Low to high" instead of "Ascending". "Has any of" instead of "IN".
4. **Saved filter sets** (Org-mode agenda, Notion views): Named filter configurations that persist as reusable views.

**Source:** [Filter UX Design Patterns](https://www.pencilandpaper.io/articles/ux-pattern-analysis-enterprise-filtering), [Designing Filter & Sort for Better UX](https://medium.com/design-bootcamp/designing-filter-sort-for-better-ux-9b88f40081db)

### Relationship Visualisation

- **Notion:** Relations appear as linked chips in table cells; clicking opens the related entity. No graph view.
- **Fibery:** Relations visible from both ends; entity views show related items in embedded tables.
- **Obsidian/org-roam:** Graph visualisation as an exploratory tool, not a primary interface.
- **Logseq:** Bidirectional links with a linked references section below each block.

The consensus: **graph visualisation is a discovery tool, not a daily driver.** Relations should be surfaced primarily through inline references and filtered views, with graph exploration available as an optional "big picture" mode.

### Bulk Operations

Enterprise table UX research identifies bulk operations as one of the most valued features. Patterns:

- **Multi-select with checkboxes** (Notion, Fibery): Select rows, then apply batch actions (delete, move, change property).
- **Keyboard-driven multi-select** (Org-mode): Bulk tag, bulk refile, bulk state change via agenda.
- **"Select all matching filter"** (Coda): Apply an action to all rows matching the current filter without manual selection.

For a content library, bulk operations are critical: "Mark all selected games as downloaded", "Add tag 'retro' to all SNES games", "Move all completed downloads to archive".

### Templates and Defaults

- **Notion:** Templates per database --- click "New" and choose a template that pre-fills properties.
- **Fibery:** Workspace templates for entire domain configurations (CRM, Product Management).
- **Coda:** Doc templates that include tables, formulas, and automations.
- **Org-mode:** Capture templates that define where new entries go and what properties they have.

The pattern: **templates should operate at multiple granularities** --- entity-level (a new game entry with default properties), view-level (a pre-configured dashboard), and workspace-level (an entire content library configuration).

## 8. What Saalis Should Adopt

### View Types for Content Libraries

Based on the patterns above, saalis should offer these views as first-class citizens:

1. **Gallery view** --- the primary browsing experience. Large cover art thumbnails with title, platform, and status overlay. This is how game libraries (Playnite, LaunchBox, Steam) present content and what users expect. LaunchBox's success demonstrates that visual presentation of game collections with custom artwork is a primary user desire.
2. **Table view** --- dense metadata for power users. Sortable columns, inline editing, column visibility toggles. Essential for bulk operations and metadata auditing.
3. **Board view** --- Kanban for status tracking. Columns for download status (Queued, Downloading, Downloaded, Verified, Failed) or play status (Backlog, Playing, Completed, Dropped). The Obsidian Kanban Status Updater pattern (moving a card updates the property) should be native.
4. **List view** --- compact display for search results and large collections. Title, platform, and one or two key properties per row.

Calendar and Timeline views are lower priority but could serve release-date browsing and download scheduling respectively.

### Exposing the DataView System

Saalis's DataView system (typed query projections) should be exposed to users through:

1. **Saved views** --- named configurations of view type + filters + sorts + visible properties. These are the user-facing manifestation of DataViews.
2. **View switching** --- a toolbar that lets users switch between Gallery, Table, Board, and List for the same underlying query. Filters and sorts persist across view switches.
3. **Default views** --- every entity type (Game, Movie, etc.) ships with sensible default views. Games get a Gallery sorted by name, a Board grouped by download status, and a Table with all properties. Users can customise or create new views but never start from zero.
4. **Linked/embedded views** --- views that can be embedded in dashboards or other entity pages, each with independent filter configurations. This mirrors Notion's linked databases and Fibery's Smart Folders.

### Filter/Sort UX for Non-Technical Users

1. **Visual filter builder** --- drop-downs for property, operator, and value. No typing of query syntax. Property types constrain available operators (text gets "contains/starts with/equals", numbers get ">/</between", selects get "is any of/is none of").
2. **Active filter chips** --- applied filters displayed as dismissible chips above the view. Users always know what is filtered.
3. **Smart defaults** --- "Show all" by default, with suggested filters based on entity type ("By platform", "By genre", "By status").
4. **Sort by column header** --- clicking a column header in Table view cycles through ascending, descending, and unsorted. The spreadsheet convention is universally understood.
5. **No query language in the primary UI** --- the filter builder handles 95% of needs. An advanced query mode can exist for power users, but it must never be required.

**Source:** [Helpful Filter Categories and Values](https://www.nngroup.com/articles/filter-categories-values/), [Search Filters: 5 Best Practices](https://www.algolia.com/blog/ux/search-filter-ux-best-practices)

### Relation Handling

Following Fibery's lead:

- Relations should be first-class and visible from both ends. If Game A relates to Platform B, both the game and the platform show the relationship.
- Many-to-many should be native (a game has multiple genres; a genre has multiple games).
- Related entities should be browsable inline --- clicking a platform chip on a game should show that platform's details and all its games.

## 9. What to Avoid

### Over-Flexibility (The Blank Page Problem)

Notion's early growth was hampered by users opening a blank page and not knowing what to do. Coda's formula-everywhere approach creates documents only their creators can modify. Obsidian's "choose your own methodology" leads to methodology paralysis.

**Saalis must avoid the blank canvas.** A content library has a well-defined domain: entities have cover art, titles, platforms, genres, ratings, and download status. The default experience should present this structure immediately. Users should be customising a working system, not building one from scratch.

### Formula/Query Languages for Non-Technical Users

Notion formulas, Coda formulas, Dataview DQL, and Logseq Datalog all create a bifurcation: the person who writes the formulas becomes a single point of failure. Non-technical users cannot debug or modify them.

**Saalis should not expose a query language in the primary UI.** Filters, sorts, and views should be configurable through GUI controls. If an advanced query mechanism exists (for plugins or power users), it should be a separate, clearly-marked layer that does not gate any core functionality.

### Performance Cliffs from Relations and Rollups

Notion's architecture demonstrates that per-block storage with complex cross-references creates O(n*m) computation costs. Fibery mitigates this by using a relational database backend rather than a block store.

**Saalis should:**
- Store relations in a proper junction table, not as serialised lists in entity rows.
- Compute aggregations (counts, averages) lazily or via background workers, not on every page load.
- Set sensible limits on displayed related entities (show first 50 with "load more") rather than rendering thousands inline.
- Index all filterable and sortable properties in SQLite for sub-second query times, even with tens of thousands of entities.

### Two-Tier Query Gaps

Logseq's simple-vs-Datalog divide and Obsidian's basic-search-vs-DQL divide create a frustrating middle ground where users have outgrown simple queries but cannot yet write advanced ones.

**Saalis should offer a smooth gradient:** basic filters (GUI) -> compound filters (AND/OR groups in GUI) -> saved filter sets (named views) -> plugin-defined custom queries (for developers). Each step adds capability without requiring a paradigm shift.

## 10. Progressive Disclosure

Progressive disclosure is the principle of showing users only the most important options initially, revealing more on request. Nielsen Norman Group defines it as reducing cognitive load by deferring secondary features until the user seeks them.

### Applied to Saalis

**Layer 0: Zero-configuration browsing.**
Open saalis, see your game library in Gallery view with cover art. Search by title. Click to see details. No configuration required. This is the Playnite/Steam experience.

**Layer 1: View customisation.**
Switch between Gallery, Table, Board, and List views. Apply basic filters (by platform, by genre, by status). Sort by name, rating, or date added. All through toolbar buttons and drop-downs.

**Layer 2: Advanced filtering and saved views.**
Create compound filters with AND/OR logic. Save custom views with specific filter/sort/property configurations. Create board views with custom column definitions. Reorder and resize table columns.

**Layer 3: Custom properties and entity types.**
Add user-defined properties to existing entity types. Create new entity types (e.g., a user might want to track gaming peripherals or custom ROM patches). Define relations between entity types.

**Layer 4: Automation and plugins.**
Trigger actions on status changes (e.g., auto-verify checksums when download completes). Write or install plugins that add new metadata sources, download providers, or view types.

### Implementation Principles

1. **Default views that just work.** Every entity type ships with pre-configured views. Users never see "No views configured".
2. **Contextual actions, not menus.** Right-click a game for contextual actions (download, edit metadata, change status). Do not bury common actions in deep menu hierarchies.
3. **Fewer than three disclosure levels per interaction.** A filter builder might show property and operator on the first level, with advanced options (case sensitivity, regex) on a second level. Never require a third click to complete a filter.
4. **Teach through use.** When a user switches to Board view for the first time, briefly explain what the columns represent. When they add their first custom property, show how to filter by it. In-context guidance, not upfront tutorials.
5. **Visible escape hatches.** Advanced features should be discoverable (a "More options" link, an "Advanced" toggle) but never required. A user who never clicks "Advanced" should have a complete, satisfying experience.

**Source:** [Progressive Disclosure - NN/G](https://www.nngroup.com/articles/progressive-disclosure/), [Progressive Disclosure: Simplifying the Complexity - Shopify](https://www.shopify.com/partners/blog/progressive-disclosure)

## Summary of Recommendations

| Principle | Adopt from | Avoid from |
|---|---|---|
| One source, many views | Notion linked databases, Fibery views | --- |
| Gallery as primary view | Playnite, LaunchBox, Steam | --- |
| Visual filter builder | Notion, Fibery | Coda formula-everywhere |
| Entity types with typed relations | Fibery entity model | Notion's limited relations |
| Templates and default views | Notion onboarding, Fibery workspace templates | Obsidian's blank-slate methodology |
| Smooth filter gradient | Obsidian Dataview's 3 tiers (adapted) | Logseq's simple/Datalog cliff |
