**Date:** 2026-03-15
**Scope:** CAS system survey with lessons for ROM/game file deduplication in saalis

## 1. Git Object Store

### Architecture

Git's object store is the canonical example of content-addressable storage. Every piece of data -- file contents (blobs), directory listings (trees), commits, and tags -- is stored as an object identified by the cryptographic hash of its content. Git constructs a header consisting of the object type, a space, the size in bytes, and a null byte, then concatenates this with the raw content and hashes the result.

Objects exist in two forms:

- **Loose objects:** Individual zlib-compressed files stored at `.git/objects/ab/cdef...`, sharded by the first two hex characters of the hash. Simple, one file per object.
- **Packfiles:** Multiple objects combined into a single `.pack` file, paired with a `.idx` index file. Packfiles use **delta compression**: Git finds objects of similar size and name, stores one fully and the rest as binary deltas against it. The `--window` parameter (default 10) controls how many objects are compared, and `--depth` (default 50) limits delta chain length.

Pack index files store object IDs in lexicographic order, enabling binary search for O(log n) lookups. Git 2.28+ introduced the **multi-pack index (MIDX)**, which indexes objects across multiple packfiles in a single file, critical for repositories with millions of objects.

### SHA-1 to SHA-256 Migration

Git historically used SHA-1 (160-bit). The transition to SHA-256 (256-bit) has been a multi-year effort:

- **Git 2.29 (Oct 2020):** First experimental SHA-256 repository support.
- **Git 2.42 (Aug 2023):** SHA-256 repositories declared no longer "experimental curiosity."
- **Git 2.45 (2024):** SHA-1/SHA-256 interoperability code landed.
- **Git 2.51 (Aug 2025):** SHA-256 set as default for new repositories, targeting Git 3.0 by late 2026.

The approach uses a bidirectional mapping table between SHA-1 and SHA-256 names, allowing a SHA-256 repository to interoperate with SHA-1 remotes. GitHub still does not support SHA-256 repositories; Forgejo/Codeberg added support in April 2024.

### Storage Overhead and Scale

The pack index per object is roughly 24 bytes (20/32-byte hash + 4-byte offset + 4-byte CRC). For 1M objects, the index overhead is approximately 24 MB. Delta compression routinely achieves 50%+ compression on versioned text. The `git maintenance` incremental repack coalesces small packfiles under a 2 GB threshold.

### Lessons for Saalis

Git's whole-file addressing (blob = entire file content) maps directly to ROM dedup. However, Git's delta compression is optimized for text diffs between versions of the same file, not for deduplicating across unrelated binary files. The pack index overhead is modest and the MIDX approach is proven at GitHub's scale (hundreds of millions of objects).

## 2. IPFS Content Addressing

### CID Structure

IPFS uses **Content Identifiers (CIDs)** -- self-describing, future-proof content addresses. A CID contains:

- **Multicodec:** Identifies the data format (e.g., `dag-pb` for Protocol Buffers, `raw` for raw binary, `dag-cbor` for CBOR).
- **Multihash:** Self-describing hash: hash function code + digest length + digest bytes. Default is SHA-256.
- **Multibase:** Encoding for string representation (base58btc for CIDv0, base32 for CIDv1).

CIDv0 is a bare multihash (always SHA-256, always dag-pb). CIDv1 adds the version byte, multicodec, and multibase prefix, supporting arbitrary hash functions and codecs.

### DAG-PB and Chunking

When adding a file, IPFS:

1. **Chunks** the file (default: fixed 256 KiB chunks).
2. **Wraps** each chunk in a UnixFS protobuf (DAG-PB codec), adding type and size metadata.
3. **Builds a DAG:** If the file exceeds one chunk, creates intermediate nodes whose Links point to the leaf chunks, forming a balanced Merkle DAG.

The CID depends on both content AND structure -- changing the chunk size or DAG layout produces a different CID for the same file. Maximum block size is 2 MiB (bitswap protocol limit). IPFS also supports buzhash-based content-defined chunking via `--chunker=buzhash`, though it is not the default.

### Pinning and Garbage Collection

IPFS nodes cache data opportunistically. Without explicit action, cached blocks are eligible for garbage collection. The `ipfs repo gc` command removes all unpinned blocks. Pinning marks a root CID and recursively protects all blocks in its DAG. Remote pinning services (Pinata, Filebase, etc.) exist because local pinning alone does not guarantee network availability.

### Lessons for Saalis

IPFS demonstrates the importance of separating the hash from the encoding/structure metadata (the CID's self-describing nature). For ROM files, the CID approach is over-engineered -- we do not need DAG-PB or chunking for whole-file addressing. However, the **multihash** concept (hash algorithm identifier + digest) is worth adopting to future-proof the hash format. IPFS's GC model (pin = keep, unpin = eligible for removal) is a clean design for reference-counted stores.

## 3. Perkeep (formerly Camlistore)

### Blob Store Design

Perkeep, created by Brad Fitzpatrick (author of memcached, LiveJournal), is a personal storage system built entirely on content-addressed blobs. A **blobref** is `hashname-hexdigest` (e.g., `sha224-a1b2c3...`). The currently recommended hash is SHA-224.

The blob server interface is minimal: `Get(blobref)`, `Put(blob)`, `Enumerate(sorted by blobref)`. The only metadata tracked per blob is its size. This simplicity is the system's strength -- storage backends are composable: local disk, S3, Google Cloud Storage, and layered proxies for sharding, replication, encryption, and conditional routing.

### Schema Blobs and Permanodes

Raw data blobs are dumb byte sequences. **Schema blobs** are JSON-encoded blobs that describe other blobs -- they create structure over the flat blob namespace. Types include:

- **File schema blobs:** Reference ordered lists of data blobs composing a file.
- **Directory schema blobs:** Reference other file/directory schema blobs.
- **Claim blobs:** Signed assertions (e.g., "set the title of permanode X to Y").
- **Permanodes:** Immutable schema blobs with no mutable content -- they serve as stable identifiers for mutable objects. Mutations are recorded as signed claims pointing at the permanode.

### Indexing

Perkeep maintains a separate index (backed by MySQL, PostgreSQL, SQLite, or LevelDB) derived from schema blobs. The index is a **materialized view** -- it can be rebuilt from scratch by replaying blobs. This separation means the blob store is the source of truth, and the index is a performance optimization.

### Storage Optimization: Blobpacked

The **blobpacked** storage type initially stores one physical blob per logical blob, then asynchronously repacks small blobs into large contiguous files organized by access patterns. An index maps logical blobrefs to physical offsets within packed files. This is conceptually similar to Git's packfiles.

### Lessons for Saalis

Perkeep's architecture is remarkably relevant. The separation of "dumb blob store" from "schema blobs that describe structure" from "derived index" maps cleanly to a ROM CAS store. The permanode concept (stable ID for a mutable entity) parallels our entity model. The index-as-materialized-view pattern means the store can survive index corruption. The blobpacked optimization shows that whole-file blob stores can achieve good I/O performance with deferred compaction.

## 4. Nix Store

### Store Paths and Content Addressing

Nix traditionally uses **input-addressed** store paths: the output path is derived from a hash of the derivation's inputs (source code, dependencies, build script), not the output itself. This means two builds of the same source with different compilers produce different store paths even if the output is bit-identical.

**Content-addressed (CA) derivations** (RFC 0062, experimental since Nix 2.4) change this: the store path is derived from the hash of the build output. This enables **early cutoff** -- if a dependency is rebuilt but produces identical output, downstream rebuilds are skipped.

### Garbage Collection

Nix GC uses a reachability algorithm from **GC roots**:

1. Roots are symlinks under `/nix/var/nix/gcroots` and all profile generations under `/nix/var/nix/profiles`.
2. The collector traverses store path references recursively, marking reachable paths as live.
3. All unreachable paths are deleted.

The `keep-derivations` (default: true) and `keep-outputs` (default: false) options control whether derivations and their outputs are protected beyond direct reachability.

### Deduplication via Hard Links

`nix-store --optimise` scans the store for files with identical content and replaces duplicates with hard links, typically saving 25-35% of store space. This is a post-hoc deduplication pass, not an integrated CAS mechanism -- files are stored individually and deduplicated as a maintenance operation.

### Lessons for Saalis

Nix's GC root model is clean: explicit roots keep things alive, everything else is collectible. The distinction between input-addressed and content-addressed paths is instructive -- for ROM files, we want pure content addressing (the hash of the file IS the identity, regardless of where/how it was obtained). Nix's hard-link deduplication is a pragmatic optimization that avoids the complexity of a full CAS store while still saving significant space.

## 5. Restic and Borg Backup

### Content-Defined Chunking (CDC)

Both Restic and Borg split files into variable-length chunks using rolling hash algorithms, enabling deduplication even when data is inserted or deleted within files.

**Restic** uses a Rabin fingerprint rolling hash:
- Window size: 64 bytes
- Average chunk size: 1 MiB (determined by testing lowest 21 bits of fingerprint for zero)
- Minimum chunk: 512 KiB
- Maximum chunk: 8 MiB
- A random irreducible polynomial of degree 53 is generated per repository and stored in the config, defeating watermark attacks.

**Borg** uses the Buzhash algorithm (cyclic polynomial hash):
- Default CHUNK_MIN_EXP = 19 (min 512 KiB)
- Default CHUNK_MAX_EXP = 23 (max 8 MiB)
- HASH_MASK_BITS = 21 (target 2 MiB average)
- Buzhash is used only for finding chunk boundaries; a separate cryptographic hash (HMAC-SHA-256 or BLAKE2b) is used for chunk identity/deduplication.

### Deduplication Ratios

Both tools typically achieve **60-85% deduplication** on incremental backups, depending on data change patterns. Borg tends to be slightly more bandwidth-efficient (15-18% savings over Restic in comparative tests), partly due to default lz4 compression. For initial backups with no prior data, deduplication across unrelated files is minimal -- CDC primarily benefits incremental scenarios.

### Architecture

Restic stores chunks as individual files in a content-addressed repository, organized into directories by hash prefix. Each snapshot is a JSON document referencing a tree of nodes, each node listing the chunk IDs composing a file. Index files map chunk IDs to pack files (which aggregate chunks, similar to Git packfiles).

Borg uses a similar model: chunks stored in segment files (append-only logs), with an in-memory hash table mapping chunk IDs to segment offsets. Compaction reclaims space from deleted chunks.

### Lessons for Saalis

CDC is powerful for backup (incremental changes to large files), but for ROM dedup the question is whether ROM files from different sources share internal content blocks. ROMs are typically not modified incrementally -- they are distinct releases. **Whole-file hashing is likely sufficient** for ROM dedup. CDC would only help if the same ROM is obtained from different dumpers who prepend different headers (a rare edge case). The overhead of CDC (rolling hash computation, chunk index) is not justified for a ~100K file collection where most dedup is exact-match.

## 6. Cross-Cutting Analysis

### Hash Algorithm Choice: BLAKE3 vs SHA-256

| Property | SHA-256 | BLAKE3 |
|---|---|---|
| Output size | 256 bits | 256 bits (default) |
| Collision resistance | 2^128 (birthday bound) | 2^128 (birthday bound) |
| Single-core throughput | ~500 MB/s (with HW accel) | ~1 GB/s single-core |
| Multi-core throughput | N/A (inherently serial) | **8+ GB/s** (tree hashing, parallelizable) |
| Small input (<4 KiB) | Comparable or faster (HW SHA) | Slightly slower on x86_64 due to setup |
| Ecosystem adoption | Universal (Git, IPFS, TLS, etc.) | Growing (used by Iroh, b3sum, Bao) |
| Hardware acceleration | SHA-NI on x86, ARMv8 CE | AVX-512, NEON (software-level parallelism) |

For saalis on an appliance (likely ARM SBC), BLAKE3's NEON-accelerated parallelism is attractive. At the ROM scale (~100K files, 1-10 TB), hashing throughput matters primarily during initial import. BLAKE3 at 1+ GB/s on a single core means a 1 GB ROM file hashes in ~1 second. SHA-256 without hardware acceleration could take 3-5x longer.

**Recommendation:** BLAKE3 for the primary content hash, stored with a multihash-style prefix (`blake3-<hex>`) to allow future algorithm changes. SHA-256 can be computed on demand for interop (e.g., verifying against ROM databases that use SHA-256).

**Sources:**
- [BLAKE3 vs SHA-256 Performance Showdown (Slaptijack)](https://slaptijack.com/programming/blake3-vs-sha256-performance.html)
- [SHA-256 Alternatives: BLAKE3 vs SHA-3 vs xxHash3 Benchmarks](https://devtoolspro.org/articles/sha256-alternatives-faster-hash-functions-2025/)
- [Hash Collision Probabilities (Preshing)](https://preshing.com/20110504/hash-collision-probabilities/)

### Chunk Size: Whole-File vs Content-Defined

| Strategy | Best For | Overhead | Dedup Quality |
|---|---|---|---|
| Whole-file hash | Immutable binary files (ROMs) | Minimal (one hash per file) | Exact-match only |
| Fixed-size chunking (256K-1M) | Large files with partial overlap | Medium (index per chunk) | Moderate |
| Content-defined chunking (CDC) | Files with insertions/deletions | High (rolling hash + index) | High for incremental |

For ROM files, which are immutable binary artifacts, **whole-file hashing is the correct choice**. The dedup opportunity is exact-duplicate detection (same ROM from multiple sources, same ROM in multiple collections), not partial-content overlap. CDC adds complexity and index overhead without meaningful benefit.

### Garbage Collection Strategies

Three models emerge from the survey:

1. **Reference counting (Perkeep/Nix):** Explicit roots protect reachable objects. GC deletes everything unreachable. Clean and deterministic.
2. **Pinning (IPFS):** Objects are cached opportunistically and GC'd unless explicitly pinned. Good for distributed systems.
3. **Snapshot-based (Restic/Borg):** Objects referenced by any snapshot are kept. Pruning removes snapshots, then GC removes orphaned chunks.

For saalis, the **reference counting / GC roots model** (Nix-style) fits best. An entity referencing a blob in the CAS store acts as a GC root. When no entity references a blob, it becomes eligible for collection. A separate `gc` maintenance command walks the reference graph and removes orphans.

### Storage Overhead of the Index

| System | Index Overhead per Object | At 100K Objects |
|---|---|---|
| Git loose | ~0 (filesystem is the index) | 100K inodes |
| Git packed | ~24 bytes (hash + offset + CRC) | ~2.4 MB |
| IPFS | ~62 bytes (CIDv1 + block metadata) | ~6.2 MB |
| Restic | ~72 bytes (hash + pack ref + offset + length) | ~7.2 MB |
| Perkeep | ~40 bytes (blobref + size in SQLite) | ~4 MB |
| Simple CAS (hash + size + refcount) | ~40 bytes | ~4 MB |

At the saalis scale (10K-100K files), index overhead is negligible -- well under 10 MB for a SQLite table mapping `blake3_hash -> (size, refcount, storage_path)`.

### Handling Hash Collisions

None of the surveyed systems actively handle hash collisions for 256-bit hashes. The birthday bound of 2^128 operations means:

- At 100K files: collision probability is approximately 2^-94, or about 1 in 10^28. Effectively zero.
- At 1 billion files: collision probability is approximately 2^-68. Still astronomically unlikely.
- The entire Bitcoin network has performed roughly 2^93 SHA-256 operations total as of 2025. Finding a collision would require 2^35 times more work.

Git famously had a SHA-1 collision demonstrated (SHAttered, 2017) with 2^63 operations, but SHA-256 and BLAKE3 at 256 bits provide a comfortable 2^128 security margin. **No collision handling is needed.**

### Performance at Scale (10K-100K Files, 1-10 TB)

| Operation | Expected Performance |
|---|---|
| Initial import (hash all files) | BLAKE3: ~1 GB/s per core, so 1 TB takes ~17 min (single-threaded) |
| Lookup by hash | SQLite B-tree on 32-byte key: <1 ms |
| Insert new blob | Write file + SQLite INSERT: bounded by disk I/O |
| GC pass | Walk entities table for live hashes, diff against blobs table: seconds |
| Enumerate all blobs | SQLite full scan of blobs table: <1 second at 100K rows |

The bottleneck at this scale is always disk I/O, not hash computation or index lookup. A simple flat-file CAS store (files named by their hash, sharded into 256 directories by first byte) handles 100K files with no performance issues.

## 7. Reflinks and CAS Integration

### The `reflink-copy` Rust Crate

The [`reflink-copy`](https://crates.io/crates/reflink-copy) crate provides cross-platform copy-on-write file cloning. API:

```rust
// Attempt reflink; fail if unsupported
reflink_copy::reflink("source.rom", "dest.rom")?;

// Attempt reflink; fall back to regular copy
reflink_copy::reflink_or_copy("source.rom", "dest.rom")?;
```

Platform support:
- **Linux:** btrfs (full support), XFS (reflink enabled by default since xfsprogs 5.1). ext4 does NOT support reflinks.
- **macOS/iOS:** APFS (full support, the default filesystem since macOS 10.13).
- **Windows:** ReFS on Windows Server 2012+ (block cloning). NTFS does NOT support reflinks.

### CAS + Reflink Architecture

The key question: **can you reflink from a CAS blob store to a user-visible path?**

Yes, with constraints:

1. **Same filesystem requirement:** The CAS store and the target path must be on the same filesystem (same mount point). Cross-filesystem reflinks fail with EXDEV. On Linux kernels before 5.17, even the same filesystem on different mount points could fail.
2. **Read-only source:** The CAS blob should be marked read-only (mode 0444). Reflinks are copy-on-write, so if the user modifies the linked file, only their copy changes -- the CAS blob is unaffected. But belt-and-suspenders: make the CAS blob immutable.
3. **Space accounting:** Reflinked files initially consume zero additional disk space. Space is only allocated when blocks are modified. `du` reports the logical size, not the physical size. Use `btrfs fi du` or `stat` to see shared extents.

### Practical Architecture for Saalis

```
/data/saalis/
  cas/                          # Content-addressable store
    ab/cdef0123...              # Blob files named by BLAKE3 hash, sharded
    ...
  library/                      # User-visible paths
    games/
      Super Mario World.sfc     # reflink -> cas/a1/b2c3...
      Zelda ALTTP.sfc           # reflink -> cas/d4/e5f6...
```

When a user "adds" a ROM, saalis:
1. Hashes the file with BLAKE3.
2. If the hash already exists in `cas/`, skips the copy (dedup hit).
3. If new, moves/copies the file into `cas/` with a hash-based name.
4. Creates a reflink from `cas/<hash>` to `library/<user-visible-name>`.

When a user "deletes" a ROM from their library:
1. Removes the reflinked file from `library/`.
2. Decrements the refcount in the blobs index.
3. If refcount reaches zero, the GC can remove the blob from `cas/`.

This gives:
- **Zero-cost dedup:** Multiple library entries pointing to the same ROM share physical blocks.
- **Zero-cost organization:** Users can reorganize their library without copying data.
- **Graceful fallback:** On ext4 or other non-reflink filesystems, `reflink_or_copy` falls back to a regular copy. The CAS store still provides dedup via hardlinks or symlinks as alternatives.

### Reflink vs Hardlink vs Symlink for CAS

| Mechanism | Copy-on-write? | Cross-filesystem? | Separate permissions? | Delete safety |
|---|---|---|---|---|
| Reflink | Yes | No | Yes (independent inodes) | Safe (independent files) |
| Hardlink | No (shared inode) | No | No (shared inode) | Risky (modification affects all) |
| Symlink | N/A (pointer) | Yes | N/A | Dangling if target deleted |

Reflinks are the ideal mechanism: they behave like independent files from the user's perspective but share storage. Hardlinks are a usable fallback but require care (read-only CAS blobs prevent accidental mutation). Symlinks work cross-filesystem but break if the CAS blob is moved or deleted.

## 8. Summary of Recommendations for Saalis

| Decision | Recommendation | Rationale |
|---|---|---|
| Hash algorithm | BLAKE3 (primary), SHA-256 (interop) | 4-10x faster, same security, NEON acceleration on ARM |
| Hash format | `blake3-<64 hex chars>` (multihash-style prefix) | Future-proof, algorithm-agile |
| Chunking | Whole-file | ROMs are immutable artifacts; CDC overhead unjustified |
| GC model | Reference-counted roots (Nix-style) | Entities are roots; orphan blobs are collectible |
| Index | SQLite table: `(hash PRIMARY KEY, size, refcount)` | <10 MB at 100K files, sub-ms lookups |
| Storage layout | Flat files sharded by hash prefix (2-char dirs) | Simple, proven, no packing needed at this scale |
| User-visible files | Reflink from CAS to library path | Zero-cost dedup and organization; fallback to copy |
| Collision handling | None | 2^128 birthday bound; not a practical concern |
| Filesystem | APFS (macOS), btrfs/XFS (Linux) for reflink support | ext4 fallback uses hardlinks or copies |
