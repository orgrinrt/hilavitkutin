# Deep Dive: ureq — Synchronous HTTP Client for Rust

## Overview

[ureq](https://github.com/algesten/ureq) is a blocking HTTP client for Rust that deliberately
avoids async/await in favour of simple, thread-per-request blocking I/O. It is an ideal fit for a
"sync core, async shell" architecture where the core application logic stays synchronous and only
the outermost web-serving layer (e.g. axum/tokio) uses async. The current major version is **3.x**,
which introduced a Sans-IO architecture with protocol logic separated into the `ureq-proto` crate.

ureq forbids all `unsafe` code (`#![forbid(unsafe_code)]`), making it one of the safest HTTP clients
available in the Rust ecosystem.

---

## 1. Architecture — Blocking I/O Without Async

### Design Philosophy

ureq's position is explicit: the costs of async are worth paying for HTTP *servers* that must handle
thousands of concurrent clients, but for HTTP *clients* — which typically issue a handful of
concurrent requests — blocking I/O is simpler, produces smaller binaries, and has fewer
dependencies. Each concurrent request requires its own OS thread, which is perfectly acceptable for
client-side workloads.

### Sans-IO Architecture (3.x)

ureq 3.x separates protocol logic from I/O:

- **`ureq-proto`** — A pure, sans-IO crate that handles HTTP/1.1 protocol parsing and encoding with
  no I/O dependencies. It manages request serialisation, response parsing, chunked transfer
  decoding, and header processing as pure state machines.
- **`ureq`** — The main crate that provides I/O, connection pooling, TLS, configuration, cookies,
  and the user-facing API. It drives `ureq-proto`'s state machines with actual TCP reads/writes.

### Transport Trait

The I/O layer is abstracted behind a `Transport` trait, enabling a chain-of-responsibility connector
pattern:

- **TcpTransport** — Direct TCP connections
- **RustlsTransport** — TLS via rustls
- **NativeTlsTransport** — TLS via the OS-native stack
- **SOCKS proxy transport** — SOCKS4/4a/5 tunnelling
- **HTTP CONNECT transport** — HTTP proxy tunnelling
- **TestTransport** — Mock implementation for testing

Each connector in the chain either handles the connection or passes it to the next, allowing
transparent layering of proxies, TLS, and direct connections.

## 2. Agent Configuration

The `Agent` is ureq's central stateful object. It holds configuration, a connection pool, an
optional cookie jar, and middleware. All `Agent` clones share state through internal `Arc`, making
cloning cheap and enabling a single process-wide agent across threads.

### Creation

```rust
use ureq::Agent;
use std::time::Duration;

let agent: Agent = Agent::config_builder()
    .timeout_global(Some(Duration::from_secs(30)))
    .https_only(true)
    .build()
    .into();
```

### Configuration Options (Config)

| Option | Type | Default | Description |
|---|---|---|---|
| `timeout_global` | `Option<Duration>` | `None` | Overall request timeout |
| `timeouts` | `Timeouts` | (none set) | Granular timeout struct |
| `https_only` | `bool` | `false` | Reject non-HTTPS URIs |
| `http_status_as_error` | `bool` | `true` | Treat 4xx/5xx as `Error::StatusCode` |
| `max_redirects` | `u32` | `10` | Maximum redirect chain length |
| `max_redirects_will_error` | `bool` | `true` | Error on exceeding max redirects |
| `redirect_auth_headers` | `RedirectAuthHeaders` | `Never` | Whether to forward auth headers on redirect |
| `save_redirect_history` | `bool` | `false` | Record redirect locations in response extensions |
| `no_delay` | `bool` | `true` | TCP_NODELAY (disables Nagle's algorithm) |
| `ip_family` | `IpFamily` | `Any` | IPv4, IPv6, or both |
| `max_idle_connections` | `usize` | `10` | Total idle connections in pool |
| `max_idle_connections_per_host` | `usize` | `3` | Idle connections per host |
| `max_idle_age` | `Duration` | `15s` | How long idle connections are kept |
| `input_buffer_size` | `usize` | `128 KB` | Read buffer size |
| `output_buffer_size` | `usize` | `128 KB` | Write buffer size |
| `max_response_header_size` | `usize` | `64 KB` | Max response header size |
| `user_agent` | `AutoHeaderValue` | `"ureq/<ver>"` | User-Agent header |
| `accept` | `AutoHeaderValue` | `"*/*"` | Accept header |
| `accept_encoding` | `AutoHeaderValue` | (feature-dependent) | gzip/brotli based on features |
| `proxy` | `Option<Proxy>` | env-based | HTTP/SOCKS proxy |
| `tls_config` | `TlsConfig` | (see TLS section) | TLS settings |
| `allow_non_standard_methods` | `bool` | `false` | Permit non-standard HTTP methods |

### Proxy Configuration

```rust
use ureq::{Agent, Proxy};

let proxy = Proxy::new("http://user:password@proxy.example:8080")?;
let agent: Agent = Agent::config_builder()
    .proxy(Some(proxy))
    .build()
    .into();
```

SOCKS proxies (4, 4a, 5) are supported via the `socks-proxy` feature flag.

## 3. TLS Backends

### rustls (Default)

The default TLS backend is **rustls** with the **ring** cryptographic provider. This is a pure-Rust
implementation with no dependency on OpenSSL or platform-native TLS libraries. It produces fully
statically-linked binaries — important for appliance targets like Batocera.

Feature flags:
- `rustls` — rustls + ring + webpki-roots (default, included by default)
- `rustls-no-provider` — rustls without ring; bring your own `CryptoProvider`
- `platform-verifier` — use OS-native certificate verification with rustls via
  `rustls-platform-verifier`

### native-tls

Uses the platform's TLS stack (SChannel on Windows, Secure Transport on macOS, OpenSSL on Linux).
Must be explicitly opted in and is "never picked up as a default."

Feature flags:
- `native-tls` — enable with webpki-roots
- `native-tls-no-default` — enable without default root certs

### Root Certificate Configuration

```rust
use ureq::tls::{TlsConfig, RootCerts};

let tls = TlsConfig::builder()
    .root_certs(RootCerts::WebPki)          // Mozilla's bundled roots (default)
    // or: RootCerts::PlatformVerifier      // OS-native cert store
    // or: RootCerts::Specific(certs)       // Custom CA certificates
    .build();
```

The `RootCerts` enum has three variants:

| Variant | Description |
|---|---|
| `WebPki` | Mozilla's root certificate bundle compiled into the binary (default) |
| `PlatformVerifier` | Delegates to the OS certificate store; requires `platform-verifier` feature |
| `Specific(Arc<Vec<Certificate>>)` | User-provided root certificates for custom/internal CAs |

Additional TLS options:
- `client_cert()` — client certificate chain with private key for mutual TLS
- `use_sni()` — Server Name Indication (default: true)
- `disable_verification()` — disables certificate verification (dangerous, testing only)
- `unversioned_rustls_crypto_provider()` — plug in a custom `CryptoProvider` (e.g. `aws-lc-rs`)

## 4. HTTP/2 Support Status

**ureq does not support HTTP/2 and has no plans to add it.**

GitHub issue [#219](https://github.com/algesten/ureq/issues/219) was closed in November 2020 with
the maintainer (Martin Algesten) explaining:

- HTTP/2 multiplexing is fundamentally at odds with ureq's synchronous, blocking model. Multiplexing
  requires managing multiple concurrent streams over a single connection, which effectively demands
  async-style concurrency management.
- For HTTP clients (as opposed to servers), the maintainers believe HTTP/1.1 is sufficient. Most
  API endpoints and server-to-server communication work fine over HTTP/1.1.
- Adding HTTP/2 would undermine ureq's core value proposition: simplicity. The codebase is
  deliberately "simple, at times even naive," making it accessible to contributors.

**Practical impact:** Nearly all REST APIs, metadata services (IGDB, ScreenScraper), and download
servers work over HTTP/1.1. HTTP/2 is primarily beneficial for browser-like workloads with many
parallel small requests to the same origin — not typical for a content library manager's backend
HTTP calls.

## 5. Cookie Handling

Enabled via the `cookies` Cargo feature. When active, the Agent maintains a `CookieJar` shared
across all clones.

```rust
// Cookies are automatically stored from Set-Cookie response headers
// and sent back on subsequent requests to matching domains.
let agent: Agent = Agent::new_with_defaults();

// Manual cookie jar access:
let mut jar = agent.cookie_jar_lock();
// Manipulate cookies directly...
// IMPORTANT: drop the lock before making requests to avoid deadlocks
drop(jar);
```

Cookie handling is automatic: `Set-Cookie` headers from responses are parsed and stored in the jar,
then included in subsequent requests to matching domains. The implementation uses the `cookie_store`
crate internally.

## 6. Streaming Response Bodies

The `Body` struct implements `std::io::Read`, enabling streaming without buffering the entire
response into memory. This is critical for downloading large files (ROMs, ISOs, media).

### Reading Methods

```rust
// Full body as string (10 MB default limit)
let text: String = response.body_mut().read_to_string()?;

// Full body as bytes (10 MB default limit)
let bytes: Vec<u8> = response.body_mut().read_to_vec()?;

// Streaming via borrowed reader (no limit by default)
let reader = response.body_mut().as_reader();
std::io::copy(&mut reader, &mut output_file)?;

// Owned reader — can be moved to another thread
let (_, body) = response.into_parts();
let reader = body.into_reader();
// reader is 'static and Send, usable across threads
```

### Default Size Limits

The convenience methods (`read_to_string`, `read_to_vec`, `read_json`) apply a **10 MB default
limit** to protect against memory exhaustion from malicious or misbehaving servers. For larger
responses, either:
- Increase the limit via `with_config().limit()`
- Use `as_reader()` or `into_reader()` directly, which have no limit

### Important Detail for Connection Pooling

Reading the response body to completion is required for the connection to be returned to the pool.
If you abandon a body mid-read, the connection is dropped rather than pooled. When using
`as_reader()` with a limit that truncates the body, the connection will not be reused.

### Body Metadata

```rust
response.body_mut().mime_type();        // e.g. "application/json"
response.body_mut().charset();          // e.g. "utf-8"
response.body_mut().content_length();   // Option<u64>, None for chunked
```

## 7. Request Building

### HTTP Method Shortcuts

```rust
let agent = ureq::agent();

agent.get("https://api.example.com/items").call()?;
agent.post("https://api.example.com/items").send("body")?;
agent.put("https://api.example.com/items/1").send("body")?;
agent.delete("https://api.example.com/items/1").call()?;
agent.patch("https://api.example.com/items/1").send("body")?;
agent.head("https://api.example.com/items").call()?;
agent.options("https://api.example.com/items").call()?;
```

### Headers

```rust
let response = agent.get("https://api.example.com/data")
    .header("Authorization", "Bearer token123")
    .header("Accept", "application/json")
    .call()?;
```

### Query Parameters

ureq does not have a dedicated query parameter builder. Parameters must be included in the URL:

```rust
let url = format!("https://api.example.com/search?q={}&page={}", query, page);
agent.get(&url).call()?;
```

### JSON (requires `json` feature)

```rust
// Sending JSON
#[derive(serde::Serialize)]
struct Item { name: String, count: u32 }

agent.post("https://api.example.com/items")
    .send_json(&Item { name: "test".into(), count: 42 })?;

// Receiving JSON
#[derive(serde::Deserialize)]
struct Response { id: u64, status: String }

let item: Response = agent.get("https://api.example.com/items/1")
    .call()?
    .body_mut()
    .read_json()?;
```

### Form Data

```rust
agent.post("https://example.com/login")
    .send_form(&[("username", "user"), ("password", "pass")])?;
```

### Multipart (requires `multipart` feature)

Multipart form uploads are supported via the `multipart` feature flag.

### Raw Request Execution

For full control, build an `http::Request` and run it directly:

```rust
let req = http::Request::builder()
    .method("POST")
    .uri("https://api.example.com/data")
    .header("X-Custom", "value")
    .body("raw body")?;

let response = agent.run(req)?;
```

## 8. Middleware / Interceptor Patterns

ureq 3.x provides a first-class `Middleware` trait for chained request/response interception.

### Trait Definition

```rust
pub trait Middleware: Send + Sync + 'static {
    fn handle(
        &self,
        request: Request<SendBody<'_>>,
        next: MiddlewareNext<'_>,
    ) -> Result<Response<Body>, Error>;
}
```

### Function-Based Middleware (Simple)

Any function matching the signature is automatically a `Middleware`:

```rust
fn my_middleware(
    req: http::Request<SendBody>,
    next: MiddlewareNext,
) -> Result<http::Response<Body>, ureq::Error> {
    // modify request before sending
    let res = next.handle(req)?;
    // modify response after receiving
    Ok(res)
}
```

### Stateful Middleware (Struct-Based)

For maintaining state between invocations (e.g. rate limiting, retry counters):

```rust
use std::sync::{Arc, Mutex};

struct RateLimiter {
    last_request: Arc<Mutex<std::time::Instant>>,
    min_interval: Duration,
}

impl Middleware for RateLimiter {
    fn handle(
        &self,
        request: Request<SendBody>,
        next: MiddlewareNext,
    ) -> Result<Response<Body>, Error> {
        {
            let mut last = self.last_request.lock().unwrap();
            let elapsed = last.elapsed();
            if elapsed < self.min_interval {
                std::thread::sleep(self.min_interval - elapsed);
            }
            *last = std::time::Instant::now();
        }
        // Lock released before continuing the chain
        next.handle(request)
    }
}
```

### Use Cases

The middleware system enables:
- **Logging** — log request method/URI before, status code after
- **Rate limiting** — enforce minimum intervals between requests
- **Retry** — catch transient errors and retry with backoff
- **Authentication** — inject/refresh auth tokens transparently
- **Metrics** — measure request duration, count errors

The `Send + Sync + 'static` bounds ensure middleware is safe to share across threads when the Agent
is cloned.

## 9. Comparison with reqwest::blocking

| Dimension | ureq 3.x | reqwest::blocking |
|---|---|---|
| **Async runtime** | None | Spawns a hidden tokio runtime |
| **Binary size** | Small (~hundreds of KB with rustls) | Larger (pulls in tokio, hyper, h2) |
| **Compile time** | Fast (few dependencies) | Slower (large dependency tree) |
| **HTTP/2** | No | Yes (via h2 crate) |
| **Unsafe code** | `#![forbid(unsafe_code)]` | Allows unsafe (via dependencies) |
| **Connection pooling** | Built-in Agent | Built-in Client |
| **Middleware** | First-class `Middleware` trait | No built-in middleware; custom via wrapper |
| **Streaming** | `impl Read` on Body | `impl Read` on Response |
| **Cookie support** | Feature flag | Feature flag (cookie_store) |
| **Proxy** | HTTP CONNECT, SOCKS4/5 | HTTP CONNECT, SOCKS5 |
| **TLS** | rustls (default) or native-tls | native-tls (default) or rustls |
| **JSON** | Feature flag (serde) | Feature flag (serde) |
| **Multipart** | Feature flag | Built-in |
| **Websocket** | No | No (separate crate) |
| **Query params API** | None (manual URL building) | `.query(&[("key", "val")])` builder |
| **Architecture** | Sans-IO + blocking I/O | Async core with blocking facade |

### Key Trade-offs

**reqwest::blocking** always spins up a tokio runtime in a background thread, even for a single
request. This is invisible to the caller but adds overhead in binary size, startup time, and
dependency count. For a "sync core" architecture, this hidden async runtime is architecturally
dishonest — it creates exactly the async machinery you're trying to avoid.

**ureq** is genuinely synchronous. There is no hidden runtime. A blocking `call()` maps directly to
blocking socket reads/writes. This makes the program's execution model fully predictable, debuggable
with standard tools, and honest about its threading model.

**Feature gap:** The main thing reqwest offers that ureq does not is HTTP/2. If your target APIs
require HTTP/2 (rare for REST APIs), reqwest is the only option. For everything else, ureq's simpler
model is advantageous.

## 10. Connection Pooling Details

### Pool Configuration

| Setting | Default | Description |
|---|---|---|
| `max_idle_connections` | 10 | Total idle connections across all hosts |
| `max_idle_connections_per_host` | 3 | Maximum idle connections to a single host |
| `max_idle_age` | 15 seconds | Idle connections older than this are evicted |

### Behaviour

- Connections are organised per host (scheme + hostname + port).
- After a response body is fully read, the connection is returned to the pool for reuse.
- If the body is not fully consumed, the connection is dropped (not pooled).
- The pool holds an internal `Mutex`, briefly locked when borrowing or returning connections.
- All `Agent` clones share the same pool via `Arc`, enabling a single process-wide pool.
- Expired connections are cleaned up via `cleanup_expired()`.

### DNS Resolution

ureq provides a `Resolver` trait (in the `unversioned::resolver` module, not yet under semver) for
pluggable DNS resolution. The default resolver uses the system's standard DNS mechanisms.

**DNS caching is not built in.** The system resolver may cache at the OS level, but ureq itself does
not maintain a DNS cache. For high-frequency requests to the same hosts, connection pooling
effectively bypasses DNS since pooled connections already have established TCP sockets.

## 11. Error Handling

### Error Enum

`ureq::Error` is `#[non_exhaustive]` with approximately 29 variants:

```rust
pub enum Error {
    // HTTP-level
    StatusCode(u16),              // 4xx/5xx when http_status_as_error is true
    TooManyRedirects,
    RedirectFailed,               // 307/308 redirect with body

    // Network/Transport
    Io(io::Error),
    Timeout(Timeout),
    HostNotFound,
    ConnectionFailed,

    // Protocol
    Http(http::Error),            // Invalid headers, etc.
    Protocol(ureq_proto::Error),  // HTTP/1.1 parse failures
    BadUri(String),

    // TLS
    Tls(&'static str),
    Rustls(rustls::Error),        // (feature-gated)
    NativeTls(native_tls::Error), // (feature-gated)
    Pem(rustls_pemfile::Error),   // (feature-gated)
    TlsRequired,

    // Content
    BodyExceedsLimit(u64),
    LargeResponseHeader(usize, usize),
    Decompress(&'static str, Error),  // gzip/brotli
    Json(serde_json::Error),          // (feature-gated)
    UnknownCharset(String),
    InvalidMimeType(FromStrError),
    BodyStalled,                      // Body read made no progress

    // Proxy
    InvalidProxyUrl,
    ConnectProxyFailed(String),

    // Cookies (feature-gated)
    Cookie(CookieError),
    CookieValue(&'static str),
    CookieJar(Error),

    // Configuration
    RequireHttpsOnly(String),

    // Catch-all
    Other(Box<dyn Error + Send + Sync>),
}
```

### Idiomatic Error Handling

```rust
use ureq::Error;

match agent.get("https://api.example.com/data").call() {
    Ok(response) => {
        // 2xx response (or any status if http_status_as_error is false)
    }
    Err(Error::StatusCode(code)) => {
        // Server returned 4xx/5xx
        eprintln!("HTTP error: {}", code);
    }
    Err(Error::Timeout(which)) => {
        // Request timed out (connect, read, or global)
    }
    Err(Error::HostNotFound) => {
        // DNS resolution failed
    }
    Err(e) => {
        // All other errors (I/O, TLS, protocol, etc.)
        eprintln!("Request failed: {}", e);
    }
}
```

### Disabling Status-as-Error

When `http_status_as_error` is set to `false`, 4xx/5xx responses are returned as `Ok(Response)`
rather than `Err(Error::StatusCode)`. This is useful when you need to inspect error response bodies:

```rust
let agent: Agent = Agent::config_builder()
    .http_status_as_error(false)
    .build()
    .into();

let response = agent.get("https://api.example.com/data").call()?;
// Now you must check response.status() yourself
if response.status() >= 400 {
    let error_body = response.body_mut().read_to_string()?;
}
```

### Conversion

`Error` provides `into_io(self) -> std::io::Error` for converting any ureq error into a standard
I/O error. `From` conversions exist for `io::Error`, `http::Error`, `ureq_proto::Error`,
`rustls::Error`, `native_tls::Error`, `serde_json::Error`, and cookie errors.

## Saalis Relevance

For saalis's "sync core, async shell" architecture, ureq is the natural HTTP client choice:

1. **Genuinely synchronous** — no hidden tokio runtime, honest blocking I/O model.
2. **Pure Rust with no unsafe** — aligns with safety-first design philosophy.
3. **rustls default** — produces fully static binaries for Batocera/loisto appliance deployment
   without requiring OpenSSL on the target.
4. **Middleware system** — rate limiting (for API quotas on IGDB/ScreenScraper), retry logic, and
   authentication token injection can all be implemented as composable middleware.
5. **Streaming bodies** — essential for downloading large game files without buffering entire ROMs
   in memory. The `into_reader()` method returns a `'static + Send` reader that can be handed off
   to a download work unit.
6. **Connection pooling** — the 3-per-host, 15-second idle defaults are reasonable for metadata
   API polling. A single `Agent` shared via `Arc` across scheduler subsystems keeps connection
   reuse automatic.
7. **No HTTP/2 needed** — REST APIs (IGDB, ScreenScraper) and ROM download servers all operate over
   HTTP/1.1. HTTP/2's multiplexing benefits are irrelevant for a sync client model.

### Recommended Feature Set for Saalis

```toml
[dependencies]
ureq = { version = "3", features = ["json", "cookies", "gzip", "brotli", "socks-proxy"] }
```

This gives JSON serialisation for API responses, cookie persistence for sites requiring sessions,
compressed transfer support to minimise bandwidth, and SOCKS proxy support for privacy-conscious
users.
