# Deep Dive: Proc Macro Patterns for `#[register]` and `#[derive(Settings)]`

## 1. The syn / quote / proc-macro2 Ecosystem

Rust procedural macros are built on a trinity of crates maintained primarily by David Tolnay:

- **`proc-macro2`** (1.0) -- a wrapper around the compiler-internal `proc_macro` crate. All library code (parsing helpers, code generation utilities) should be written against `proc_macro2` types rather than `proc_macro` types. The reason is simple: `proc_macro` types can only exist inside a procedural macro invocation, so they cannot appear in unit tests, helper functions in non-proc-macro crates, or any context outside the compiler's macro expansion phase. `proc_macro2` lifts that restriction while remaining ABI-compatible via `.into()` conversions at the proc-macro entry point boundary.

- **`syn`** (2.0) -- a complete parser for Rust syntax. It can parse any valid Rust source into a structured AST. Key entry points for proc macros: `syn::DeriveInput` (for derive macros), `syn::ItemImpl` (for attribute macros on impl blocks), `syn::ItemFn`, `syn::ItemStruct`, etc. Syn 2.0 introduced a more modular feature-flag system; enable `features = ["full"]` for the complete AST, or `features = ["derive"]` for just derive-related types.

- **`quote`** (1.0) -- quasi-quoting for generating `TokenStream` output. The `quote!` macro interpolates Rust variables using `#var` syntax and produces a `proc_macro2::TokenStream`. For repetition over iterables, `#( #items )*` is used. `quote_spanned!` attaches a specific `Span` to generated tokens, which is critical for error reporting (see section 8).

### Canonical structure of a proc-macro crate

```
saalis-derive/
  Cargo.toml          # [lib] proc-macro = true
  src/
    lib.rs            # #[proc_macro_derive], #[proc_macro_attribute] entry points
    register.rs       # impl logic for #[register]
    settings.rs       # impl logic for #[derive(Settings)]
```

Entry points in `lib.rs` accept `proc_macro::TokenStream` and return `proc_macro::TokenStream`. All internal logic works with `proc_macro2::TokenStream`. The conversion happens at the boundary:

```rust
#[proc_macro_attribute]
pub fn register(attr: proc_macro::TokenStream, item: proc_macro::TokenStream) -> proc_macro::TokenStream {
    register_impl(attr.into(), item.into())
        .unwrap_or_else(|e| e.into_compile_error())
        .into()
}
```

This pattern -- delegating to a fallible function returning `syn::Result<proc_macro2::TokenStream>` -- is the gold standard. It enables unit-testing the inner function without needing a compiler context.

## 2. Attribute Macros vs Derive Macros

### Derive macros

- Triggered via `#[derive(MyTrait)]` on structs, enums, or unions.
- Receive the full token stream of the annotated item as input.
- Can only **append** new items (impls, structs, consts) after the original item; they cannot modify or remove the original.
- May define helper attributes (e.g., `#[setting(...)]`) that are consumed during expansion and stripped from the final output.
- Ideal for: auto-implementing traits, generating companion types or impls based on field structure.

### Attribute macros

- Triggered via `#[my_attr(...)]` on any item (fn, struct, impl block, module, etc.).
- Receive **two** token streams: the attribute arguments and the annotated item.
- Can **replace** the annotated item entirely -- they output whatever tokens they want, including a modified version of the original item, additional items, or nothing.
- Ideal for: transforming items, injecting code around functions, processing impl blocks, generating registration shims.

### For saalis

- **`#[register]`** should be an **attribute macro** applied to an `impl` block (e.g., `impl EntityType for Game { ... }`). It needs to parse the impl, extract associated types and constants, generate a `#[repr(C)]` descriptor struct, emit the original impl unchanged, and append an `inventory::submit!` call. Only an attribute macro can do all of this because it needs to both preserve and augment the impl block.

- **`#[derive(Settings)]`** should be a **derive macro** on a struct. It reads the struct's fields and their `#[setting(...)]` helper attributes to generate a `SettingsValue` impl, scope/domain extraction logic, and (optionally) registration. Derive macros are the natural fit because the input is a struct definition and the output is trait impls.

## 3. Generating `repr(C)` Descriptor Structs from Trait Impl Blocks

The `#[register]` macro needs to parse an `impl Trait for Type` block, extract information, and emit a `#[repr(C)]` descriptor struct. Here is the parsing strategy:

### Parsing the impl block

```rust
fn register_impl(attr: TokenStream, item: TokenStream) -> syn::Result<TokenStream> {
    let impl_block: syn::ItemImpl = syn::parse2(item.clone())?;

    // Extract the trait being implemented
    let (_, trait_path, _) = impl_block.trait_.as_ref()
        .ok_or_else(|| syn::Error::new_spanned(&impl_block, "expected trait impl"))?;

    // Extract the self type
    let self_ty = &impl_block.self_ty;

    // Walk associated items
    let mut assoc_types = Vec::new();
    let mut assoc_consts = Vec::new();

    for item in &impl_block.items {
        match item {
            syn::ImplItem::Type(ty) => {
                assoc_types.push((&ty.ident, &ty.ty));
            }
            syn::ImplItem::Const(c) => {
                assoc_consts.push((&c.ident, &c.expr));
            }
            _ => {}
        }
    }
    // ... generate descriptor
}
```

### Generating the descriptor

The descriptor struct must be `#[repr(C)]` so it can cross an ABI boundary (plugin FFI). The macro generates it alongside the original impl:

```rust
let descriptor_ident = format_ident!("__{}_DESCRIPTOR", self_ty.to_token_stream().to_string().to_uppercase());

let output = quote! {
    #impl_block  // emit original impl unchanged

    #[repr(C)]
    pub struct #descriptor_ident {
        pub type_name: &'static str,
        // fields derived from associated types/consts
    }

    inventory::submit! {
        #descriptor_ident {
            type_name: stringify!(#self_ty),
            // ...
        }
    }
};
```

Key considerations:
- Use `format_ident!` from `quote` to construct identifiers programmatically.
- The descriptor struct's fields are determined by the trait contract. The macro knows the trait's shape at design time and generates fields accordingly.
- For ZST marker types (like `MetadataTable`, `Column<In<M>, As<V>>`), the descriptor may store type IDs or function pointers rather than the types themselves, since `repr(C)` cannot represent Rust generics.

## 4. Inspecting Trait Bounds in Macros

A proc macro operates on syntax, not semantics. It runs before type checking. Therefore, **a proc macro cannot directly query whether a type implements a trait**. However, it can generate code that the compiler will check:

### Pattern: generated trait-bound assertions

```rust
// Generated by the macro -- fails compilation if T doesn't implement the trait
const _: () = {
    fn _assert_implements_entity_type<T: EntityType>() {}
    fn _check() {
        _assert_implements_entity_type::<Game>();
    }
};
```

This is a zero-cost compile-time check: the function is never called at runtime, but the compiler must verify the trait bound.

### Pattern: static_assertions crate

The `static_assertions` crate provides ready-made macros:

```rust
static_assertions::assert_impl_all!(Game: EntityType, Send, Sync);
```

A proc macro can emit these assertions in its generated code.

### Pattern: const-block assertions (Rust 1.79+)

```rust
const { assert!(std::mem::size_of::<MyDescriptor>() <= 256) };
```

Useful for asserting size constraints on `repr(C)` descriptors.

### Pattern: the `impls` crate

The `impls!` macro from the `impls` crate evaluates trait implementation checks as compile-time `bool` constants, supporting logical operators:

```rust
const GAME_IS_ENTITY: bool = impls!(Game: EntityType & Send & Sync);
```

This can be combined with `const { assert!(...) }` for hard failures, or used to drive conditional code generation in declarative macros (though not in proc macros directly, since proc macros run before const evaluation).

## 5. The `inventory` Integration

The `inventory` crate (by dtolnay) provides typed distributed plugin registration. It works via constructor functions (`__attribute__((constructor))` on Unix, equivalent on other platforms) that run before `main()`.

### Two-step setup

1. **Declare the registry** (once, in the crate that defines the type):

```rust
// In saalis-primitives or saalis-core:
pub struct EntityDescriptor { /* repr(C) fields */ }
inventory::collect!(EntityDescriptor);
```

2. **Submit entries** (anywhere in the dependency graph):

```rust
// Generated by #[register] macro:
inventory::submit! {
    EntityDescriptor {
        type_name: "Game",
        // ...
    }
}
```

### How it works internally

Each `inventory::submit!` call expands to a platform-specific constructor function that, at program load time, pushes the value into a global `Vec` behind a `Mutex`. The `inventory::iter::<EntityDescriptor>()` function returns an iterator over all submitted values.

### Integration with `#[register]`

The proc macro generates the `inventory::submit!` call as part of its output. The macro must reference `inventory` by path, which means `inventory` must be a dependency of the crate where `#[register]` is used. A common pattern is to re-export it:

```rust
// In saalis-sdk:
pub use inventory;
```

Then the macro generates `::saalis_sdk::inventory::submit!` to avoid requiring users to directly depend on `inventory`.

### Crate path resolution

Proc macros cannot use `$crate` (that is a `macro_rules!` feature). The standard workaround is to use the `proc-macro-crate` crate, which reads `Cargo.toml` at compile time to find the correct path for a dependency. Alternatively, accept a `#[register(crate = "...")]` attribute for overrides.

Platform support: Linux, macOS, iOS, FreeBSD, Android, Windows, WebAssembly (via wasm-bindgen), and others.

## 6. Compile-Time Validation Patterns

### Size assertions for `repr(C)` descriptors

```rust
const _: () = {
    assert!(
        std::mem::size_of::<EntityDescriptor>() == EXPECTED_SIZE,
        "descriptor size mismatch -- ABI break"
    );
};
```

### Layout assertions

```rust
const _: () = {
    assert!(std::mem::align_of::<EntityDescriptor>() >= 8);
};
```

### Field offset assertions (for FFI stability)

Use `memoffset::offset_of!` or the unstable `std::mem::offset_of!` (stabilised in Rust 1.77):

```rust
const _: () = {
    assert!(std::mem::offset_of!(EntityDescriptor, type_name) == 0);
    assert!(std::mem::offset_of!(EntityDescriptor, version) == 8);
};
```

### Type-level checks via sealed traits

The macro can generate a "sealed" impl that only compiles if the type satisfies certain constraints:

```rust
const _: () = {
    // Ensure the type is Sized + 'static (required for inventory)
    fn _assert<T: Sized + 'static>() {}
    fn _check() { _assert::<Game>(); }
};
```

### The `proc_macro_assertions` crate

This crate generates const-context code that type-checks conditions without runtime cost. Useful for asserting that macro inputs satisfy constraints that cannot be checked syntactically.

## 7. Derive Macro for Settings

A `#[derive(Settings)]` macro on a struct needs to:

1. Parse the struct's fields and their types.
2. Extract `#[setting(...)]` helper attributes from each field.
3. Generate a `SettingsValue` impl (or equivalent trait).
4. Extract scope/domain metadata from struct-level or field-level attributes.

### Parsing with syn

```rust
#[proc_macro_derive(Settings, attributes(setting))]
pub fn derive_settings(input: proc_macro::TokenStream) -> proc_macro::TokenStream {
    let input = parse_macro_input!(input as DeriveInput);
    // ...
}
```

The `attributes(setting)` declaration tells the compiler that `#[setting(...)]` is a helper attribute for this derive, preventing "unknown attribute" errors.

### Extracting field attributes

```rust
if let Data::Struct(data) = &input.data {
    if let Fields::Named(fields) = &data.fields {
        for field in &fields.named {
            let field_name = field.ident.as_ref().unwrap();
            let field_ty = &field.ty;

            for attr in &field.attrs {
                if attr.path().is_ident("setting") {
                    // Parse the attribute contents
                    attr.parse_nested_meta(|meta| {
                        if meta.path.is_ident("scope") {
                            let value = meta.value()?;
                            let lit: syn::LitStr = value.parse()?;
                            // store scope = lit.value()
                        }
                        if meta.path.is_ident("domain") {
                            // ...
                        }
                        if meta.path.is_ident("default") {
                            let value = meta.value()?;
                            let lit: syn::Expr = value.parse()?;
                            // store default expression
                        }
                        Ok(())
                    })?;
                }
            }
        }
    }
}
```

### Using darling for ergonomic attribute parsing

The `darling` crate by Ted Driggs eliminates boilerplate when parsing attributes. Instead of manually walking `NestedMeta`, you define a struct that derives `FromDeriveInput` and `FromField`:

```rust
#[derive(FromDeriveInput)]
#[darling(attributes(setting))]
struct SettingsInput {
    ident: syn::Ident,
    data: darling::ast::Data<(), SettingsField>,
}

#[derive(FromField)]
#[darling(attributes(setting))]
struct SettingsField {
    ident: Option<syn::Ident>,
    ty: syn::Type,
    #[darling(default)]
    scope: Option<String>,
    #[darling(default)]
    domain: Option<String>,
    #[darling(default)]
    default: Option<syn::Expr>,
}
```

Darling handles error collection, default values, renaming, and validation. It also provides `#[darling(from_none)]` for optional fields and `darling::util::Flag` for boolean-style attributes.

This approach is recommended for `#[derive(Settings)]` because the attribute grammar is likely to grow in complexity (scope, domain, visibility, credential vs. regular setting, validation rules, etc.), and darling scales well for that.

### Generating the impl

```rust
let field_impls = fields.iter().map(|f| {
    let name = &f.ident;
    let ty = &f.ty;
    let scope = f.scope.as_deref().unwrap_or("global");
    quote! {
        SettingsEntry {
            name: stringify!(#name),
            scope: #scope,
            type_id: std::any::TypeId::of::<#ty>(),
        }
    }
});

quote! {
    impl SettingsValue for #struct_name {
        fn entries() -> &'static [SettingsEntry] {
            &[ #( #field_impls ),* ]
        }
    }
}
```

## 8. Error Reporting in Proc Macros

Good error messages are critical for macro usability. There are several strategies, ranked from best to worst:

### `syn::Error` and `.to_compile_error()` (preferred)

```rust
fn my_macro_impl(input: TokenStream) -> syn::Result<TokenStream> {
    let input: DeriveInput = syn::parse2(input)?;
    if !meets_requirements(&input) {
        return Err(syn::Error::new_spanned(
            &input.ident,
            "type must have at least one field"
        ));
    }
    // ...
}
```

`syn::Error::new_spanned` attaches the span of the given syntax node to the error. The compiler will underline exactly the right tokens in the source. This is far superior to `Span::call_site()`, which highlights the entire macro invocation.

### Span choices

- **`Span::call_site()`** -- points to the macro invocation site. Use as a last resort when no better span is available.
- **`Span::mixed_site()`** -- behaves like `call_site` for name resolution but like `def_site` for hygiene. Rarely needed in practice.
- **`node.span()`** -- the span of a specific syntax node. Prefer passing the node directly to `new_spanned` rather than calling `.span()`, because `.span()` on a multi-token node may lose information about the end position, resulting in only the first token being highlighted.

### Multiple errors

`syn::Error` supports accumulating multiple errors via `.combine()`:

```rust
let mut errors = Vec::new();
// ... collect errors ...
if !errors.is_empty() {
    let mut combined = errors.remove(0);
    for e in errors {
        combined.combine(e);
    }
    return Err(combined);
}
```

This produces multiple compiler diagnostics from a single macro expansion, which is much more helpful than failing on the first error.

### `compile_error!` emission

Under the hood, `.to_compile_error()` generates a `compile_error!("message")` invocation with the appropriate span. You can also emit `compile_error!` directly via `quote_spanned!`:

```rust
quote_spanned! { field.span() =>
    compile_error!("field type is not supported");
}
```

### What NOT to do

- **`panic!`** in a proc macro produces a generic "proc macro panicked" error with no span information and a stack trace. It is acceptable only for internal bugs (ICE-style), never for user-facing errors.
- **`proc-macro-error`** crate (unmaintained) -- used to provide `abort!` and `emit_error!` macros. Its successor `proc-macro-error2` exists but the standard `syn::Error` pattern has become dominant and is preferred.

## 9. Testing Proc Macros

### Unit testing the inner function

Because the macro logic is written against `proc_macro2::TokenStream`, it can be tested directly:

```rust
#[test]
fn test_register_generates_descriptor() {
    let input: proc_macro2::TokenStream = quote! {
        impl EntityType for Game {
            type Metadata = GameMeta;
            const NAME: &'static str = "game";
        }
    };
    let output = register_impl(TokenStream::new(), input).unwrap();
    let output_str = output.to_string();
    assert!(output_str.contains("DESCRIPTOR"));
    assert!(output_str.contains("repr(C)"));
}
```

This gives fast, isolated tests that don't need a compiler. However, it only tests that the output tokens look right, not that they compile.

### `trybuild` -- compile-fail and compile-pass tests

The `trybuild` crate (also by dtolnay) runs `rustc` on test files and checks the output:

```rust
#[test]
fn ui_tests() {
    let t = trybuild::TestCases::new();
    t.pass("tests/ui/pass/*.rs");         // must compile
    t.compile_fail("tests/ui/fail/*.rs"); // must fail with expected errors
}
```

Each compile-fail test has a corresponding `.stderr` file with the expected error output. Running with `TRYBUILD=overwrite` regenerates these files, letting you review changes via `git diff`.

This is the primary tool for testing error messages -- ensuring that your `syn::Error` spans produce clear, well-located diagnostics.

### `macrotest` -- expansion snapshot tests

`macrotest` invokes `cargo expand` on test files and compares the expanded output with `.expanded.rs` snapshot files:

```rust
#[test]
fn expansion_tests() {
    macrotest::expand("tests/expand/*.rs");
}
```

If the `.expanded.rs` file does not exist, it is created automatically. This is valuable for catching unintended changes in generated code.

### Integration tests

Standard `#[test]` functions in a crate that depends on your proc-macro crate exercise the macros end-to-end:

```rust
#[register]
impl EntityType for TestGame {
    type Metadata = TestMeta;
    const NAME: &'static str = "test_game";
}

#[test]
fn registered_types_include_test_game() {
    let descriptors: Vec<_> = inventory::iter::<EntityDescriptor>().collect();
    assert!(descriptors.iter().any(|d| d.type_name == "test_game"));
}
```

### Debugging tips

- **`cargo expand`** shows the fully expanded output of a specific file. Install via `cargo install cargo-expand`. Pass `--lib` or `--bin` flags and a specific item.
- **`eprintln!`** in proc macro code prints to stderr during compilation, visible in `cargo build` output. Useful for quick debugging but remove before committing.
- **Structuring code for testability** (Ferrous Systems recommendation): keep the proc macro entry point as thin as possible; put all logic in a regular library function that takes and returns `proc_macro2::TokenStream`.

## 10. Performance: Incremental Compilation Impact

Proc macros have a significant impact on compilation performance. Understanding these costs informs architectural decisions about how many macros to define and how to structure them.

### The core problem

Proc macros are opaque to the compiler. Unlike declarative macros (`macro_rules!`), the compiler cannot inspect or cache their expansion. When any token in a file changes, **all proc macro invocations in that file are re-expanded from scratch**, even if the change is unrelated to the macro input. This is because the compiler cannot prove that the proc macro's output didn't depend on the changed tokens.

### Measured impact

Real-world measurements show that caching proc macro expansions can yield **11-40% faster incremental build times**. The polkadot-sdk project achieved a 35% reduction in incremental compile times by implementing macro expansion caching. This gives a sense of how much time proc macros cost during iterative development.

### Why proc macros block the pipeline

Rustc uses a pipelining model where downstream crates can begin compiling as soon as the upstream crate's metadata is available, even before codegen completes. However, **proc macros must be fully compiled and linked before any crate using them can begin parsing**. This means `saalis-derive` becomes a critical-path dependency.

### Mitigation strategies

1. **Minimise the number of distinct proc-macro crates.** Each proc-macro crate is a separate compilation unit that blocks downstream crates. Putting `#[register]`, `#[derive(Settings)]`, and any other macros into a single `saalis-derive` crate avoids compounding this cost.

2. **Keep proc-macro crates lean.** Heavy dependencies in the proc-macro crate (e.g., pulling in `serde` or `regex` for macro logic) increase its compile time, which blocks everything downstream. Use only `syn`, `quote`, `proc-macro2`, and minimal helpers like `darling`.

3. **Prefer derive macros over attribute macros where possible.** The compiler has slightly more information about derive macro inputs (it knows the item is a struct/enum/union), which can help in some edge cases. More importantly, derive macros are more familiar to Rust developers and produce less surprising behaviour.

4. **Avoid re-exporting heavy types in macro output.** If the generated code references types from large crates, those crates must be compiled before the macro's output can be type-checked. Structure dependencies so that the generated code references only types from lightweight crates (like `saalis-primitives`).

5. **Use `cfg(test)` guards** on test-only macro invocations to avoid expanding them during production builds.

### Caching status (as of early 2026)

There is an ongoing effort in the Rust compiler to cache proc macro expansions (tracking the mapping from input `TokenStream` to output `TokenStream`). However, this is not yet stable. The challenge is that proc macros are not guaranteed to be pure functions -- they can read files (`include_bytes!`), access environment variables, or use randomness. The compiler cannot safely cache results without some guarantee of determinism. For now, assume no caching and design accordingly.

## Summary: Recommended Patterns for Saalis

| Macro | Kind | Input | Output |
|-------|------|-------|--------|
| `#[register]` | Attribute macro | `impl Trait for Type { ... }` | Original impl + `#[repr(C)]` descriptor struct + `inventory::submit!` |
| `#[derive(Settings)]` | Derive macro | `struct MySettings { ... }` | `SettingsValue` impl + entry metadata + optional registration |

Both macros should live in a single `saalis-derive` crate. Use `darling` for attribute parsing in `#[derive(Settings)]`. Use `syn::ItemImpl` parsing for `#[register]`. Generate compile-time trait-bound assertions rather than trying to inspect types at macro expansion time. Test with `trybuild` for error diagnostics and `macrotest` for expansion snapshots.
