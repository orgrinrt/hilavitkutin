# SQLite Alternatives for Saalis Cold Store

**Date:** 2026-03-14
**Status:** Research / Decision Input
**Context:** Saalis is a single-binary Rust content library manager targeting Batocera/loisto appliances (ARM and x86). The current persistence layer uses SQLite via SeaORM as cold storage, with an in-memory columnar hot store (hilavitkutin) handling active data. This document evaluates whether SQLite remains the right choice for the cold store, or whether an alternative better fits our access patterns and constraints.

---

## 1. SQLite: Strengths and Limitations for Our Use Case

### Strengths

SQLite is the most deployed database engine in the world and its strengths are substantial for embedded single-binary applications:

- **Single-file database.** The entire database is one file on disk. Backup is `cp`. This maps directly to our `.saalis` snapshot format requirement.
- **Zero configuration.** No daemon, no socket, no config file. Link the library and call `sqlite3_open()`.
- **Library footprint.** Approximately 600KB compiled. Negligible contribution to our binary size.
- **WAL mode.** Write-Ahead Logging enables concurrent readers alongside a single writer without blocking. On Raspberry Pi hardware, SQLite in WAL mode is [2x faster for small databases and up to 60x faster for large databases](https://spin.atomicobject.com/sqlite-raspberry-pi/) compared to the default rollback journal.
- **ACID transactions.** Full atomicity, consistency, isolation, and durability guarantees with configurable fsync behavior.
- **FTS5 full-text search.** Built-in extension for content name search with BM25 ranking. Battle-tested on billions of devices daily.
- **ARM support.** Runs natively on every ARM board in existence. No cross-compilation surprises.
- **Ecosystem maturity.** Decades of testing, fuzzing, and production hardening. The test suite is orders of magnitude larger than the source code.
- **SeaORM integration.** We already have a working ORM layer, though async SQLite performance is notably worse than sync due to SQLite's lack of a native async API.

### Limitations

- **Single writer.** Only one write transaction can execute at a time. Concurrent write attempts queue behind a mutex. For our burst-write pattern (scheduler work units flushing hot store to cold), this means writes are serialized.
- **Write amplification.** SQLite's page-based storage (default 4KB pages) means even small updates rewrite entire pages. On flash storage (SD cards, eMMC common on Batocera appliances), this accelerates wear. WAL mode mitigates this somewhat but does not eliminate it.
- **Row-oriented storage.** Analytical queries that scan a single column across many rows (e.g., "find all entities where metadata field X matches Y") must read entire rows. Our hot store is columnar precisely because this pattern is common, but the cold store still suffers when queried directly.
- **Large blob handling.** SQLite stores blobs inline in B-tree pages. Cover art, cached screenshots, and similar binary data bloat the database file and fragment page access. The conventional workaround is to store blobs as external files and reference paths, which we already plan to do.
- **No native compression.** Data is stored uncompressed by default. Extensions like `sqlite-zstd` exist but add complexity.
- **Memory usage under load.** SQLite's page cache can consume significant memory when handling complex queries. On our 2-8GB RAM budget, this competes with the hot store.

### Verdict

SQLite's limitations are real but largely mitigated by our architecture: the hot store handles analytical queries, burst writes can be serialized through the scheduler, and blobs will be external. The single-writer constraint is a non-issue when writes are funneled through a single scheduler thread by design.

## 2. DuckDB: Columnar Analytical Engine

[DuckDB](https://duckdb.org/) is an embeddable columnar OLAP database that has seen explosive adoption since 2023.

### Fit for Our Use Case

- **Columnar storage with vectorized execution.** For scanning metadata across thousands of entities (e.g., "all games with genre RPG released after 2000"), DuckDB can be [10-50x faster than SQLite](https://www.datacamp.com/blog/duckdb-vs-sqlite-complete-database-comparison).
- **Embedded, zero-ops.** No server process. Links as a library into a single binary.
- **Rust bindings.** The `duckdb-rs` crate provides ergonomic bindings. The `frozen-duckdb` crate offers precompiled binaries for faster build times.
- **ARM support.** DuckDB [runs TPC-H queries at scale factor 1000 on a Raspberry Pi 5](https://duckdb.org/2025/01/17/raspberryi-pi-tpch), demonstrating genuine ARM viability. Peak memory stays below 2.5GB even on 2TB datasets thanks to strict buffer management.
- **Compression built-in.** Columnar storage with dictionary encoding, bitpacking, and other compression schemes reduces disk footprint.

### Concerns

- **OLTP weakness.** DuckDB is designed for analytical queries, not transactional point lookups. Inserting or updating individual entity records is slower than SQLite. Our burst-write pattern (batch inserts/updates from the hot store) partially mitigates this, since DuckDB handles bulk loads well.
- **Binary size.** Approximately 5MB compiled library vs SQLite's ~600KB. Not a deal-breaker for a content library manager, but notable.
- **Relational model maturity.** DuckDB supports foreign keys, constraints, and standard SQL, but its optimizer is tuned for scan-heavy workloads. Single-row lookups by primary key (e.g., "load entity by ID") will not be faster than SQLite.
- **Not officially distributed for Raspberry Pi OS.** Must be built from source for 32-bit ARM. 64-bit ARM (aarch64) builds are available.
- **Concurrency model.** Supports multiple concurrent readers and uses multi-threaded query execution, but write concurrency is limited similarly to SQLite.

### Verdict

DuckDB would shine if the cold store were frequently queried with analytical patterns. However, our architecture pushes analytical work to the in-memory hot store, making the cold store primarily a durable write target and occasional point-lookup source. DuckDB's strengths are underutilized here, and its OLTP weaknesses become liabilities. **Not recommended as a replacement**, though it could serve as a complementary analytical engine if complex cold-store queries become a requirement.

## 3. libSQL / Turso / Limbo

[libSQL](https://github.com/tursodatabase/libsql) is an open-source, open-contribution fork of SQLite maintained by Turso.

### What It Adds

- **Embedded replicas.** Replicate the database inside the application. Irrelevant for our single-node use case.
- **Vector search.** Native vector indexing. Potentially useful for future content similarity features, but not a current requirement.
- **WASM UDFs.** User-defined functions written in WebAssembly. Interesting but unnecessary for our use case.
- **BEGIN CONCURRENT (experimental).** MVCC-based concurrent writes. This directly addresses SQLite's single-writer limitation, though our scheduler design already serializes writes intentionally.
- **Full backward compatibility with SQLite.** libSQL reads and writes standard SQLite files.

### Limbo: SQLite Rewritten in Rust

Turso is [rewriting SQLite entirely in Rust](https://turso.tech/blog/introducing-limbo-a-complete-rewrite-of-sqlite-in-rust) under the project name "Turso Database" (formerly Limbo). Key features:

- **Async from the ground up.** Uses `io_uring` on Linux. This would eliminate the async performance penalty we currently pay with SeaORM + SQLite.
- **Native WASM support.** Irrelevant for our use case.
- **Currently in beta (alpha released July 2025).** Not production-ready. The on-disk format and API are still stabilizing.
- **Testing via Antithesis.** Ambitious automated testing to match SQLite's legendary reliability, but unproven at scale.

### Verdict

libSQL is the most interesting "keep SQLite but better" option. The async-native Rust rewrite could eventually eliminate our SeaORM async overhead. However, the project is not yet stable enough for production. **Worth watching closely. Revisit when Turso Database reaches 1.0.** In the meantime, libSQL (the C fork) could be a low-risk swap for standard SQLite if we need vector search or concurrent writes.

## 4. ReDB: Pure Rust Key-Value Store

[ReDB](https://github.com/cberner/redb) is an embedded key-value database written entirely in Rust, using copy-on-write B-trees.

### Strengths

- **Pure Rust, no unsafe.** No C dependencies. Compiles natively for any Rust target including ARM.
- **ACID with configurable durability.** Transactions can be non-durable for performance without sacrificing atomicity, consistency, or isolation.
- **MVCC.** Single writer, multiple concurrent readers without blocking. Serializable isolation.
- **Stable file format.** Since redb 3.0.0, the file format is backward-compatible.
- **Zero-copy reads.** Data is returned as direct references to memory-mapped storage, avoiding deserialization overhead.

### Limitations

- **Key-value only.** No SQL, no relational model, no query optimizer, no full-text search. We would need to implement our own indexing, query planning, and relationship management on top of raw key-value operations.
- **Performance vs LMDB.** Benchmarks show redb is competitive but [consistently slower than LMDB](https://github.com/cberner/redb): ~2600ms bulk load vs LMDB's ~1100ms, ~975ms random reads vs LMDB's ~567ms.
- **No compression.** Data stored uncompressed.
- **Small ecosystem.** Far fewer users and battle-testing than SQLite.

### Verdict

ReDB is technically sound but a poor fit. Dropping SQL means rebuilding query, indexing, and relationship layers ourselves. The relational model (entities, metadata tables, foreign keys) is fundamental to our architecture. **Not recommended.** The engineering cost of building a relational layer on top of key-value storage far outweighs any benefits.

## 5. Sled: Lessons from a Perpetual Beta

[Sled](https://github.com/spacejam/sled) is a Rust embedded database that has been under development since 2016.

### Current Status

Sled remains in an alpha/unstable state with no 1.0 release. The APIs and on-disk format are still [changing in non-forward-compatible ways](http://sled.rs/). A full rewrite of the storage subsystem is underway as part of the "komora" project, which aims to dramatically lower disk space usage and garbage collection overhead.

### What Happened

Sled's development trajectory is instructive:

- **Ambitious scope.** Lock-free data structures, concurrent writers, modern storage engine design. All technically interesting but creating a compounding stability debt.
- **Perpetual rewrite cycle.** The komora rewrite is the latest in a series of architectural rethinks. Each promises to solve the previous version's problems.
- **Data loss reports.** Community reports of data corruption in production have damaged trust, even if many were user error or fixed in subsequent versions.
- **Single maintainer.** The project depends heavily on one developer (Tyler Neely), creating bus-factor risk.

### Lessons

1. Embedded database reliability requires conservative engineering, not cutting-edge concurrency research.
2. Stable on-disk formats matter more than raw performance for production use.
3. SQLite's "boring technology" approach is a feature, not a limitation.

### Verdict

**Not recommended.** Sled is a research project, not production infrastructure. Its instability and uncertain future make it unsuitable for a content library that must reliably persist user data on appliance hardware.

## 6. RocksDB / LevelDB: LSM-Tree Databases

[RocksDB](https://rocksdb.org/) (Facebook's fork of Google's LevelDB) is an LSM-tree key-value store optimized for fast storage.

### Fit for Our Use Case

- **Write-optimized.** LSM-trees buffer writes in memory and flush sorted runs to disk. Excellent for our burst-write pattern.
- **Compression built-in.** Snappy, LZ4, Zstd, Zlib, Bzip2 all supported.
- **Rust bindings.** The `rust-rocksdb` crate is [actively maintained](https://github.com/rust-rocksdb/rust-rocksdb) with support for transactions, column families, backups, and advanced features.
- **Battle-tested.** Used by Facebook, CockroachDB, TiKV, and dozens of other production systems.

### Concerns

- **C++ dependency.** RocksDB is a large C++ library. Cross-compilation for ARM requires a C++ toolchain. The `bindgen-static` feature supports musl/static linking, but the build process is heavier than SQLite's single-file amalgamation.
- **Key-value only.** Same limitation as ReDB: no SQL, no relational model.
- **Space amplification.** LSM-trees can use 2-10x the logical data size due to compaction levels. On constrained storage (SD cards), this is problematic.
- **Memory overhead.** RocksDB's block cache, memtables, and bloom filters consume significant memory. Default configurations assume server-class hardware with abundant RAM.
- **Tuning complexity.** Hundreds of configuration knobs. Getting good performance on constrained hardware requires expertise and extensive benchmarking.
- **Write amplification during compaction.** While individual writes are fast, background compaction rewrites data multiple times. On flash storage, this is worse than SQLite's write amplification.

### Verdict

**Not recommended.** RocksDB's key-value model requires us to reimplement relational abstractions, its C++ dependency complicates cross-compilation, and its resource consumption and tuning complexity are inappropriate for appliance hardware. LevelDB shares these limitations with fewer features.

## 7. LanceDB: Vector + Columnar

[LanceDB](https://lancedb.com/) is an embedded database built on the Lance columnar format, designed for vector search and multimodal data.

### Strengths

- **Native Rust.** Core engine written in Rust.
- **Columnar + vector search.** Combines structured metadata queries with vector similarity search in a single engine.
- **Embedded, no server.** Library-mode deployment.
- **Automatic indexing.** Creates IVF-PQ vector indices automatically for vector columns.
- **Metadata filtering.** Secondary bitmap indices enable fast filtered queries even at 500M+ rows.

### Concerns

- **Not a general-purpose relational database.** Designed for AI/ML workloads. The data model is table-oriented but not relational in the traditional sense (no foreign keys, no JOIN optimization, no referential integrity).
- **Young project.** Less battle-tested than SQLite by orders of magnitude.
- **Overkill for our needs.** We do not currently need vector search. Full-text search for content names is handled well by SQLite's FTS5.
- **Storage format.** Uses Lance format files, not a single database file. Backup/restore would require archiving a directory rather than copying a single file.

### Verdict

**Not recommended for the cold store.** LanceDB solves problems we do not have. If we later need content similarity or recommendation features based on embeddings, LanceDB could serve as a complementary engine alongside SQLite, not a replacement.

## 8. Polars / DataFusion as Cold Store

[Polars](https://github.com/pola-rs/polars) and [Apache DataFusion](https://datafusion.apache.org/) are Rust-native query engines built on Apache Arrow.

### Architecture Mismatch

Both are compute/query engines, not storage engines. They process data in Arrow columnar format and can read/write Parquet files, but they [do not provide durable transactional storage](https://dasroot.net/posts/2026/01/rust-data-processing-polars-vs-datafusion/). They are designed to sit on top of a storage layer, not replace it.

- **No ACID transactions.** Neither provides transaction isolation, rollback, or crash recovery.
- **No point writes.** Updating a single entity requires rewriting an entire Parquet file or partition.
- **No referential integrity.** No foreign keys, no constraints.
- **Session-oriented.** Designed for batch analytical processing, not persistent application state.

DataFusion is used as a foundation by other database projects (notably InfluxDB's storage engine), but always combined with a proper storage layer that provides durability and transaction semantics.

### Verdict

**Fundamentally unsuitable.** These are query engines, not databases. Our hot store (hilavitkutin) already occupies this niche in our architecture. Using Polars or DataFusion as the cold store would mean building our own transactional storage layer on top, which is exactly what SQLite already provides.

## 9. LMDB: Lightning Memory-Mapped Database

[LMDB](http://www.lmdb.tech/) is a memory-mapped B-tree key-value store created by Howard Chu for OpenLDAP.

### Strengths

- **Extreme read performance.** LMDB is [several orders of magnitude faster than alternatives for reads](https://github.com/marvelousmlops/database_comparison) due to zero-copy memory-mapped access. Data is returned as direct pointers into the memory map.
- **Tiny footprint.** 32KB of object code. The smallest option evaluated.
- **ACID with MVCC.** Readers never block writers. Read transactions see a consistent snapshot and can live indefinitely without affecting write performance.
- **Copy-on-write.** No write-ahead log needed. The database is always in a consistent state. Crash recovery is instant.
- **No compaction.** Unlike LSM-trees, LMDB's B-tree structure does not require background compaction, eliminating write amplification from that source.
- **Rust bindings.** The [heed](https://github.com/meilisearch/heed) crate (maintained by Meilisearch) provides fully typed, safe LMDB wrappers with minimal overhead. Actively maintained as of 2025.

### Concerns

- **Key-value only.** Same fundamental limitation as ReDB and RocksDB. No SQL, no relational model.
- **Memory-mapped limitations.** LMDB maps the entire database into virtual address space. On 32-bit systems, this limits database size to ~2GB. On 64-bit ARM (our target), this is not an issue, but 32-bit Batocera builds would be constrained.
- **Single writer.** Write transactions are globally serialized via a mutex, same as SQLite.
- **No built-in compression.** Data stored as-is.
- **Fixed maximum database size.** Must be configured at open time. Underestimating requires re-creation.
- **Page-level copy-on-write.** Modified pages are copied entirely, leading to write amplification for small updates within large pages.

### Verdict

LMDB is technically excellent for read-heavy key-value workloads and is used successfully by Meilisearch (search engine) and Monero (cryptocurrency). However, like all key-value stores in this evaluation, it requires building relational abstractions from scratch. **Not recommended as a direct SQLite replacement**, but heed/LMDB could serve as the underlying storage for a custom persistence layer if we ever outgrow SQLite's query performance on the cold store.

## Recommendation

**Keep SQLite as the cold store.** Here is the reasoning:

### Why SQLite Wins for Saalis

1. **Our architecture already compensates for SQLite's weaknesses.** The in-memory columnar hot store handles analytical queries. The scheduler serializes writes. Blobs are stored as external files. The limitations that would push us toward alternatives are already addressed at the architecture level.

2. **Relational model is non-negotiable.** Our entity system (entities table, metadata tables, foreign keys, the `.of::<T>(id)` query API) requires a proper relational database. Every key-value store evaluated (ReDB, Sled, RocksDB, LMDB) would require us to rebuild SQL-level abstractions, indexing, and query planning from scratch. This is months of engineering work to achieve what SQLite provides out of the box.

3. **FTS5 covers our search needs.** Content name search with BM25 ranking is built into SQLite. No alternative except libSQL (which inherits FTS5) matches this without bolting on a separate search engine.

4. **Single-file backup maps to our snapshot format.** The `.saalis` backup format can simply include a copy of the SQLite file. Key-value stores that use directory-based storage (RocksDB, LanceDB) would complicate this.

5. **ARM compatibility is proven.** SQLite runs on every ARM device ever made. DuckDB and RocksDB require more complex cross-compilation setups.

6. **Resource efficiency on constrained hardware.** SQLite's ~600KB footprint and configurable page cache make it the most predictable option for 2-8GB RAM budgets. RocksDB and DuckDB have higher baseline memory requirements.

7. **Risk-adjusted maturity.** SQLite has decades of production hardening, a test suite larger than its source code, and backward-compatible file formats. Choosing an alternative trades this reliability for marginal gains in areas our architecture already handles.

### What Could Change This Recommendation

- **If Turso Database (Limbo) reaches 1.0 with stable async I/O**, it would be worth evaluating as a drop-in SQLite replacement that eliminates our async performance penalty. This is the most likely future change.
- **If we need vector similarity search** for content recommendations, LanceDB or libSQL's vector extensions could supplement (not replace) SQLite.
- **If cold-store analytical queries become a bottleneck** (unlikely given the hot store), DuckDB could be added as a read-only analytical view materialized from SQLite.
- **If write amplification on flash storage becomes measurable**, we could investigate SQLite's `PRAGMA page_size` tuning, the `sqlite-zstd` extension for compression, or LMDB's copy-on-write model as an alternative write path.

### Immediate Action Items

1. **No migration needed.** Continue with SQLite via SeaORM.
2. **Consider sync SQLite access.** Given that [async SQLite crates perform considerably worse than sync counterparts](https://aarambhdevhub.medium.com/rust-orms-in-2026-diesel-vs-sqlx-vs-seaorm-vs-rusqlite-which-one-should-you-actually-use-706d0fe912f3), evaluate whether the cold store should use rusqlite directly (sync) rather than SeaORM's async layer, since our architecture is sync-core with an async shell only at the web boundary.
3. **Enable WAL mode and tune pragmas.** Ensure `PRAGMA journal_mode=WAL`, `PRAGMA synchronous=NORMAL`, and appropriate `PRAGMA cache_size` are set for our target hardware.
4. **Monitor Turso Database (Limbo).** Track the Rust SQLite rewrite for a potential future migration when it stabilizes.
