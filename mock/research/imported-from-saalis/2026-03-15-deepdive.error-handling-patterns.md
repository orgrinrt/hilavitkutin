# Deep Dive: Error Handling Patterns for a Layered Rust Architecture

Research date: 2026-03-15

---

## 1. The `thiserror` vs `anyhow` Split

The Rust ecosystem has settled on a clear division of labour between two complementary
error-handling crates, both maintained by David Tolnay.

**`thiserror` is for libraries; `anyhow` is for applications.**

`thiserror` is a derive macro that removes boilerplate from hand-written error enums. It
generates `Display`, `Error`, and `From` implementations. The resulting types are concrete,
matchable, and inspectable --- callers can distinguish a `ParseError` from a `NetworkError`
at compile time. This matters for library code (and SDK code) where consumers need to react
differently to different failure modes.

`anyhow` provides a single `anyhow::Error` type that can hold any `std::error::Error`.
It excels at rapid propagation with the `?` operator, attaching human-readable context via
`.context("doing X")`, and producing rich error reports with full causal chains. The
trade-off is that the concrete error type is erased --- the caller sees one opaque error and
must use `downcast_ref` to inspect it.

### How this maps to a five-crate architecture

| Crate | Error style | Rationale |
|---|---|---|
| `saalis-primitives` | `thiserror` enums | SDK consumers and plugins match on these |
| `saalis-derive` | Compile errors via `syn` | Proc macros report via `compile_error!` |
| `saalis-sdk` | `thiserror` enums, re-exports primitives errors | Public API surface, plugins depend on this |
| `saalis-core` | `thiserror` for domain errors, `anyhow` for internal orchestration | Domain types are matchable; scheduler/persistence internals use `anyhow` |
| `saalis-web` | `anyhow` with `IntoResponse` conversion | Errors become HTTP responses; no downstream consumer needs to match |

The guiding rule: if the error crosses a crate boundary where a consumer might want to match
on it, use `thiserror`. If the error propagates upward toward a human (log line, HTTP
response, notification), use `anyhow`.

A common pattern in the ecosystem is to define `thiserror` errors in the library layers and
then wrap them in `anyhow` at the application boundary:

```rust
// saalis-sdk (library layer)
#[derive(Debug, thiserror::Error)]
pub enum ConnectorError {
    #[error("network timeout after {elapsed:?}")]
    Timeout { elapsed: Duration, source: ureq::Error },
    #[error("authentication failed for {provider}")]
    AuthFailed { provider: &'static str },
    #[error("rate limited, retry after {retry_after:?}")]
    RateLimited { retry_after: Option<Duration> },
}

// saalis-core (application layer)
fn run_connector(name: &str) -> anyhow::Result<()> {
    let data = connector.fetch()
        .context(format!("running connector '{name}'"))?;  // ConnectorError -> anyhow::Error
    Ok(())
}
```

### `snafu` as an alternative

The `snafu` crate offers a middle ground: it generates "context selectors" that act as
typed error constructors, encouraging one error enum per module. Each selector captures
source errors and additional fields. This produces very detailed error chains with
backtraces. For large codebases like GreptimeDB and Databend, `snafu` has proven effective
at module-level error ownership. The trade-off is more ceremony per error site compared to
`anyhow`'s `.context()`.

### `error-stack` for rich reports

The `error-stack` crate (by HASH) provides `Report<C>`, a context-aware error wrapper that
tracks the full chain from root cause through every abstraction boundary. Unlike `anyhow`,
it preserves the ability to query for specific context types attached at any layer, and it
integrates with `tracing` to capture `SpanTrace` automatically. This makes it suitable for
systems where post-mortem analysis of error chains is a priority.

## 2. Error Propagation Across ABI Boundaries

When plugins are loaded via `dlopen` and communicate through `extern "C"` function pointers
in `repr(C)` descriptors, Rust's standard error machinery (`Result`, `?`, trait objects)
cannot cross the boundary. Several problems arise:

1. **Vtable layout instability.** `dyn Error` uses Rust-internal vtable layout, which is
   not guaranteed stable across compilation units or compiler versions.
2. **Stack unwinding.** A panic that unwinds across an FFI boundary is undefined behaviour
   (unless using `extern "C-unwind"`, stabilised in Rust 1.71, but even then both sides
   must agree on unwinding ABI).
3. **Type erasure.** The host cannot understand plugin-side error types without a shared
   schema.

### The `repr(C)` error descriptor pattern

The standard approach is to define a C-compatible error representation in the SDK crate
that both host and plugins link against:

```rust
/// FFI-safe error representation. Lives in saalis-sdk.
#[repr(C)]
pub struct FfiResult {
    /// 0 = success, non-zero = error code from ErrorCode enum
    pub code: u32,
    /// Null-terminated UTF-8 error message, or null on success.
    /// Allocated by the plugin, freed by the host via a provided free function.
    pub message: *const c_char,
    /// Optional structured context (e.g., retry_after_ms for rate limits).
    pub context: u64,
}

#[repr(C)]
pub enum ErrorCode {
    Ok = 0,
    NetworkTransient = 1,
    AuthFailed = 2,
    RateLimited = 3,
    ParseError = 4,
    InternalError = 5,
    Unknown = 255,
}
```

Every `extern "C"` function pointer in a descriptor returns `FfiResult` instead of
`Result<T, E>`. The SDK provides conversion functions:

```rust
impl FfiResult {
    pub fn from_result<E: std::error::Error + Classified>(result: Result<(), E>) -> Self {
        match result {
            Ok(()) => FfiResult { code: 0, message: std::ptr::null(), context: 0 },
            Err(e) => {
                let code = match e.class() {
                    ErrorClass::Transient => ErrorCode::NetworkTransient as u32,
                    ErrorClass::Fixable => ErrorCode::AuthFailed as u32,
                    ErrorClass::Permanent => ErrorCode::InternalError as u32,
                };
                let msg = CString::new(e.to_string()).unwrap_or_default();
                FfiResult {
                    code,
                    message: msg.into_raw(),
                    context: e.retry_after()
                        .map(|d| d.as_millis() as u64)
                        .unwrap_or(0),
                }
            }
        }
    }
}
```

### The thread-local `LAST_ERROR` pattern

An alternative (used by libgit2, the Pact FFI, and the `ffi_helpers` crate) avoids
returning error strings entirely. Functions return a sentinel value (0, -1, or NULL), and
the caller retrieves the error from a thread-local:

```rust
thread_local! {
    static LAST_ERROR: RefCell<Option<Box<dyn Error>>> = RefCell::new(None);
}

pub fn set_last_error(e: impl Error + 'static) {
    LAST_ERROR.with(|cell| *cell.borrow_mut() = Some(Box::new(e)));
}

#[no_mangle]
pub extern "C" fn last_error_message() -> *const c_char {
    LAST_ERROR.with(|cell| {
        match &*cell.borrow() {
            Some(e) => CString::new(e.to_string()).unwrap().into_raw(),
            None => std::ptr::null(),
        }
    })
}
```

This is simpler but loses structured error data. For saalis, the `repr(C)` result pattern
is preferable because the error code carries classification information (transient vs
permanent) that the scheduler needs to make retry decisions without parsing strings.

### Memory management at the boundary

Error messages allocated by the plugin must be freed by the host (or vice versa). The
safest approach is to include a `free_message` function pointer in the descriptor, or to
use a shared allocator. The SDK should provide `ffi_free_string(ptr: *const c_char)` and
document the contract.

## 3. The `ErrorKind` Pattern --- Generic `AppError<K>` with Compile-Time Classification

Rust's standard library established the `ErrorKind` pattern with `std::io::Error`, which
wraps an `io::ErrorKind` enum plus an optional boxed source error. This allows callers to
match on the kind (`NotFound`, `PermissionDenied`, `WouldBlock`) without knowing the
underlying OS error code.

### Generalising to `AppError<K: ErrorKind>`

The pattern can be generalised so that each layer defines its own `ErrorKind` while sharing
a common `AppError<K>` wrapper:

```rust
pub trait ErrorKind: Copy + fmt::Debug + fmt::Display + Send + Sync + 'static {
    /// Whether this error class is worth retrying.
    fn is_transient(&self) -> bool;
    /// User-facing guidance, if any.
    fn guidance(&self) -> Option<&'static str> { None }
}

pub struct AppError<K: ErrorKind> {
    kind: K,
    source: Box<dyn Error + Send + Sync>,
    context: ErrorContext,
}

pub struct ErrorContext {
    pub operation: &'static str,
    pub subject: Option<String>,
    pub attempt: Option<u32>,
    pub max_attempts: Option<u32>,
}
```

Each subsystem defines its own kind enum:

```rust
#[derive(Debug, Clone, Copy)]
pub enum CuratorErrorKind {
    ConnectorTimeout,
    ConnectorAuthFailed,
    EnrichmentParseFailed,
    EntityNotFound,
    StorageFull,
}

impl ErrorKind for CuratorErrorKind {
    fn is_transient(&self) -> bool {
        matches!(self, Self::ConnectorTimeout)
    }
    fn guidance(&self) -> Option<&'static str> {
        match self {
            Self::ConnectorAuthFailed => Some("Check your API credentials in Settings"),
            Self::StorageFull => Some("Free up disk space or change the storage path"),
            _ => None,
        }
    }
}
```

The benefit of the generic approach is that `AppError<K>` provides a uniform interface
(`kind()`, `is_transient()`, `guidance()`, `context()`) while the specific `K` type
encodes domain knowledge. Functions that are generic over `K: ErrorKind` can implement
retry logic, notification routing, and logging without knowing which subsystem produced the
error.

### Relationship to `thiserror`

The `ErrorKind` variants are not full error types with sources --- they are lightweight
classification markers. The actual source chain lives in `AppError.source`. This differs
from the typical `thiserror` pattern where each variant holds its own source. The two
approaches are complementary: `thiserror` defines the source chain, `ErrorKind` classifies
how to handle it.

## 4. Retry-Aware Errors

A scheduler that processes work units needs to decide, per failure, whether to retry or
fail permanently. This decision depends on two inputs: the error classification and the
attempt count.

### Error classification for retry decisions

The `backoff` crate formalises this with a three-variant result:

```rust
enum OperationResult<T, E> {
    Ok(T),
    Retry(E),    // Transient error: try again
    Err(E),      // Permanent error: stop immediately
}
```

For richer classification, the pattern extends naturally:

```rust
enum RetryVerdict {
    Retry { after: Duration },
    RetryWithBackoff,
    FailPermanent,
    FailNeedsIntervention { guidance: &'static str },
}

fn classify_for_retry<K: ErrorKind>(err: &AppError<K>, attempt: u32) -> RetryVerdict {
    if !err.kind().is_transient() {
        return if let Some(g) = err.kind().guidance() {
            RetryVerdict::FailNeedsIntervention { guidance: g }
        } else {
            RetryVerdict::FailPermanent
        };
    }

    let max = err.context().max_attempts.unwrap_or(MAX_RETRIES);
    if attempt >= max {
        RetryVerdict::FailPermanent  // Exhausted
    } else {
        RetryVerdict::RetryWithBackoff
    }
}
```

### Backoff strategies

The `backoff` crate and AWS SDK for Rust both use exponential backoff with jitter:

```
delay = min(base * multiplier^attempt + jitter, max_delay)
```

Typical defaults: base = 1s, multiplier = 2, max_delay = 30s, jitter = random 0..base.

For rate-limited APIs (HTTP 429), the server often provides a `Retry-After` header. The
error type should carry this as structured data (`retry_after: Option<Duration>`) rather
than embedding it in the error message. The `backoff` crate supports this via
`Error::retry_after(duration)`.

### Scheduler integration

In a DAG scheduler with work units, each work unit execution returns a result. The
scheduler examines the result:

1. **Success** --- mark complete, enqueue downstream dependents.
2. **Transient error, attempts remaining** --- re-enqueue with incremented attempt counter
   and computed backoff delay.
3. **Transient error, attempts exhausted** --- mark failed, emit notification
   ("download failed after 10 attempts"), do not retry.
4. **Permanent/fixable error** --- mark failed, emit notification with guidance, do not
   retry. Optionally mark the entity for manual review.

The attempt counter and max-attempts live on the work unit, not the error. The error merely
reports its class; the scheduler owns the retry policy.

## 5. Error Context Enrichment

As an error propagates from a connector through the scheduler to the web layer, it crosses
several abstraction boundaries. At each boundary, new context becomes available: which
connector, which entity, which work unit, which HTTP request.

### `anyhow::Context` for string-based enrichment

The simplest mechanism is `anyhow`'s `.context()` method:

```rust
connector.fetch(entity_id)
    .context("fetching metadata")?;
// becomes: "fetching metadata: network timeout after 30s"
```

This produces a chain of string messages. It is ergonomic but unstructured --- you cannot
programmatically extract "which entity" from the chain without parsing strings.

### Structured context via `error-stack`

The `error-stack` crate's `Report<C>` supports attaching arbitrary typed data:

```rust
use error_stack::{Report, ResultExt};

connector.fetch(entity_id)
    .change_context(CuratorError::ConnectorFailed)
    .attach_printable(format!("connector: {}", connector.name()))
    .attach(EntityId(entity_id))
    .attach(WorkUnitId(wu_id))?;
```

Later, any handler can query the report for specific attachments:

```rust
if let Some(entity_id) = report.request_ref::<EntityId>().next() {
    // Use the entity ID for notification routing, DB tracking, etc.
}
```

### `tracing` span context

The `tracing-error` crate captures the current span context at error creation time. If the
code is instrumented with spans that carry structured fields (connector name, entity ID,
work unit ID), these fields are automatically available in the error's `SpanTrace`:

```rust
#[tracing::instrument(fields(connector = %name, entity_id = %id))]
fn enrich_entity(name: &str, id: EntityId) -> Result<()> {
    // Any error created here will capture the span's fields
    let data = fetch_metadata(id)?;
    Ok(())
}
```

This avoids duplicating context between tracing and error handling --- the span is the
context. The `SpanTrace` is captured automatically when `error-stack` is used with its
`tracing` feature enabled.

### The enrichment pattern for a layered system

Each layer adds its own context and wraps the error kind:

```
Plugin layer:    ConnectorError::Timeout { elapsed: 30s }
                 + "IGDB API"
SDK boundary:    FfiResult { code: 1, message: "...", context: 0 }
Host reconst.:   AppError<CuratorErrorKind> { kind: ConnectorTimeout, source: ... }
                 + ErrorContext { operation: "enrich", subject: "Super Mario World" }
Scheduler:       + attempt: 3, max_attempts: 10
Web layer:       HTTP 503 + "Temporary issue: IGDB API timeout. Will retry."
Notification:    "Enrichment of 'Super Mario World' failed: IGDB timeout (attempt 3/10)"
```

## 6. User-Facing Error Messages vs Developer Error Details

Rust's `Display` and `Debug` traits serve fundamentally different audiences.

### `Display` is for users

The `Display` implementation of an error type produces the message that end-users (or API
consumers) see. It should be:

- Written in plain language, free of implementation jargon.
- Actionable where possible ("check your API credentials", not "IGDB auth handshake
  failed with 401 during OAuth2 token exchange").
- Brief --- a single sentence or short phrase.
- Free of internal identifiers (memory addresses, internal enum variant names).

```rust
#[derive(Debug, thiserror::Error)]
pub enum ConnectorError {
    #[error("connection timed out after {elapsed:?}")]
    Timeout { elapsed: Duration, #[source] source: ureq::Error },

    #[error("authentication failed --- check your credentials in Settings")]
    AuthFailed { provider: &'static str },
}
```

### `Debug` is for developers

The `Debug` output includes the full type name, all fields, and the source chain. It
appears in log files, panic messages, and `{:?}` format strings. `#[derive(Debug)]` is
almost always sufficient.

### The dual-output pattern

At the web boundary, errors are split:

```rust
impl IntoResponse for AppError {
    fn into_response(self) -> Response {
        // User sees Display
        let user_message = self.to_string();

        // Developer sees Debug in logs
        tracing::error!(error = ?self, "request failed");

        // HTTP response carries only the user message
        (StatusCode::INTERNAL_SERVER_ERROR, user_message).into_response()
    }
}
```

For notification payloads, the same split applies: the notification message uses `Display`,
while the notification metadata (for admin dashboards) can include the `Debug`
representation or structured fields from `ErrorContext`.

### Formatting the error chain

`anyhow` prints the full causal chain by default when using `{:?}`. For `Display`, only
the top-level message is shown. If you want the full chain in a user-facing context (e.g.,
an admin error details view), iterate the chain explicitly:

```rust
fn format_error_chain(err: &dyn std::error::Error) -> String {
    let mut chain = vec![err.to_string()];
    let mut current = err.source();
    while let Some(cause) = current {
        chain.push(cause.to_string());
        current = cause.source();
    }
    chain.join(": ")
}
```

## 7. Error Aggregation in Batch Operations

Batch operations --- enriching 500 entities, resolving strays across a library, importing a
folder of ROMs --- must not stop on the first failure. The standard Rust
`collect::<Result<Vec<_>, _>>()` idiom short-circuits on the first `Err`, which is wrong
for batch semantics.

### The partition pattern

```rust
let results: Vec<Result<Entity, AppError>> = entities
    .iter()
    .map(|e| enrich(e))
    .collect();

let (successes, failures): (Vec<_>, Vec<_>) = results
    .into_iter()
    .partition(Result::is_ok);

let ok: Vec<Entity> = successes.into_iter().map(Result::unwrap).collect();
let errs: Vec<AppError> = failures.into_iter().map(Result::unwrap_err).collect();
```

### A dedicated `BatchResult` type

For systems where batch outcomes are common, a purpose-built result type is cleaner:

```rust
pub struct BatchResult<T, E> {
    pub succeeded: Vec<T>,
    pub failed: Vec<(String, E)>,  // (item identifier, error)
    pub skipped: Vec<(String, &'static str)>,  // (item identifier, reason)
}

impl<T, E: fmt::Display> BatchResult<T, E> {
    pub fn summary(&self) -> String {
        format!(
            "{} succeeded, {} failed, {} skipped",
            self.succeeded.len(),
            self.failed.len(),
            self.skipped.len(),
        )
    }

    pub fn has_failures(&self) -> bool {
        !self.failed.is_empty()
    }

    pub fn into_notification(&self) -> Option<NotificationPayload> {
        if self.failed.is_empty() { return None; }
        // Build notification with failure details
        Some(/* ... */)
    }
}
```

### Iterator adaptors for batch error collection

The `itertools` crate provides `partition_map` for splitting an iterator of `Result` into
two collections in one pass:

```rust
use itertools::Itertools;

let (successes, failures): (Vec<Entity>, Vec<AppError>) = entities
    .iter()
    .map(|e| enrich(e))
    .partition_result();
```

For concurrent batch operations (e.g., multiple HTTP requests), `futures::join_all`
collects all results regardless of individual failures, then the partition happens on the
collected `Vec<Result<_, _>>`.

### Error budget pattern

Some batch operations want to fail the entire batch if too many items fail (e.g., if >50%
of enrichments fail, something systemic is wrong). This is an "error budget":

```rust
const FAILURE_THRESHOLD: f64 = 0.5;

let total = results.len();
let failure_count = results.iter().filter(|r| r.is_err()).count();

if failure_count as f64 / total as f64 > FAILURE_THRESHOLD {
    return Err(AppError::systemic("too many failures in batch, aborting"));
}
```

## 8. Panic Safety

Panics in Rust are intended for unrecoverable bugs, not expected errors. However, in a
plugin system with `dlopen`-loaded code, panics must be contained.

### `catch_unwind` at plugin boundaries

Every `extern "C"` function in a plugin descriptor should wrap its body in
`std::panic::catch_unwind`:

```rust
extern "C" fn plugin_fetch(ctx: *const FfiContext) -> FfiResult {
    let result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
        let ctx = unsafe { &*ctx };
        // ... actual plugin logic ...
    }));

    match result {
        Ok(Ok(())) => FfiResult::ok(),
        Ok(Err(e)) => FfiResult::from_error(e),
        Err(panic_payload) => {
            // Panic was caught --- convert to error
            let msg = panic_payload
                .downcast_ref::<&str>()
                .map(|s| s.to_string())
                .or_else(|| panic_payload.downcast_ref::<String>().cloned())
                .unwrap_or_else(|| "unknown panic".to_string());
            FfiResult::panic(msg)
        }
    }
}
```

The `#[register]` macro should generate this wrapping automatically for every function
pointer in the descriptor, so plugin authors never need to think about it.

### `AssertUnwindSafe` and its risks

`catch_unwind` requires the closure to be `UnwindSafe`. Most types are, but `&mut T` is
not (the referent might be in an inconsistent state after a panic). Wrapping in
`AssertUnwindSafe` is a promise that the caller will not observe broken invariants. At a
plugin boundary, this is acceptable because:

- The plugin's internal state is discarded on panic (the descriptor can be marked as
  poisoned).
- The host never shares mutable references across the boundary (all communication is via
  `repr(C)` values).

### Poisoned locks

`std::sync::Mutex` (and `RwLock`) implement "poisoning": if a thread panics while holding
the lock, the lock is marked poisoned, and subsequent `lock()` calls return
`Err(PoisonError)`. This is a safety mechanism that prevents observing inconsistent state.

In a plugin system, poisoned locks can occur if a plugin panics while a shared resource is
locked. The host should:

1. Catch the panic at the boundary (as above).
2. If a shared lock is poisoned, either recover by calling `.into_inner()` on the
   `PoisonError` (if the data is known to be safe) or propagate the poisoned state as a
   permanent error.

For saalis, where the core is sync and shared state is minimal, poisoned locks are unlikely
but should be handled. The `parking_lot` crate's mutexes do not implement poisoning at
all (panics just release the lock), which is a simpler model if poisoning detection is not
needed.

### Panic-in-Drop

A known footgun (Rust issue #86027): if a type's `Drop` implementation panics, and that
type is used as a panic payload caught by `catch_unwind`, dropping the caught payload
triggers a second panic, which aborts the process. Mitigations:

- Never panic in `Drop` implementations. Use `Drop` only for cleanup that cannot fail.
- If a `Drop` can fail (e.g., flushing a buffer), log the error and swallow it.
- The `scopeguard` crate provides `defer!` blocks that catch panics internally.

### `extern "C-unwind"`

Stabilised in Rust 1.71 (RFC 2945), `extern "C-unwind"` allows Rust panics to propagate
through C frames that are prepared for it. This is relevant for plugins that call back into
the host: if the host callback panics, the unwind can propagate through the plugin's C
frame back to the host's `catch_unwind`. Without `-unwind`, such unwinding is UB. However,
using `extern "C-unwind"` requires both sides to agree on the ABI, and it does not help
with foreign (non-Rust) plugins. For a Rust-only plugin system, it is a viable option; for
a heterogeneous plugin system, stick with `catch_unwind` at every boundary.

## 9. The `?` Operator Chain --- Ergonomic Error Conversion Across Layers

The `?` operator desugars to:

```rust
match expr {
    Ok(val) => val,
    Err(e) => return Err(From::from(e)),
}
```

The key mechanism is `From::from(e)` --- the error type of the expression is converted into
the error type of the enclosing function's return type. This means error conversion is
driven by `From` implementations.

### Designing the `From` graph

For a layered architecture, `From` implementations form a directed graph from specific
errors to more general ones:

```
ureq::Error --From--> ConnectorError --From--> CuratorError --From--> anyhow::Error
io::Error ---From--> StorageError ---From--> HousekeeperError --From--> anyhow::Error
```

Each layer defines `From<LowerError> for HigherError`, enabling `?` to work across
boundaries. `thiserror`'s `#[from]` attribute generates these implementations:

```rust
#[derive(Debug, thiserror::Error)]
pub enum CuratorError {
    #[error("connector failed: {0}")]
    Connector(#[from] ConnectorError),

    #[error("storage failed: {0}")]
    Storage(#[from] StorageError),

    #[error("entity not found: {id}")]
    NotFound { id: EntityId },
}
```

Now `?` works on any function returning `ConnectorError` inside a function returning
`CuratorError`.

### Pitfall: diamond conversions

If two error types both implement `From<io::Error>`, and a function returns a type that has
`From` for both, the compiler cannot decide which path to take. This manifests as
"conflicting implementations of trait `From`". Solutions:

- Use `.map_err()` to explicitly choose the conversion path.
- Avoid `#[from]` on `io::Error` in multiple error types within the same call chain.
- Create wrapper newtypes (`StorageIoError(io::Error)`) to disambiguate.

### Context-adding with `?`

The `anyhow` approach of `.context("message")?` adds a string message but erases the
concrete type. A middle ground is to implement `From` with context:

```rust
impl From<ConnectorError> for CuratorError {
    fn from(e: ConnectorError) -> Self {
        CuratorError::Connector(e)
        // The variant name IS the context --- no string needed
    }
}
```

Each layer wraps the lower error in a variant whose name documents the conversion context.
The full chain is recoverable via `.source()`.

### The `?` chain across sync/async boundaries

In a sync-core / async-shell architecture, `?` works identically in both contexts (it is
not an `.await`-aware operator). The error types must satisfy `Send + Sync` if they are used
in async code (which they should be anyway for thread safety). The async shell (`axum`)
requires `IntoResponse`, which is separate from `From` --- do not conflate error conversion
with response conversion.

## 10. Testing Error Paths

Error handling code is notoriously under-tested. Rust's type system helps (you must handle
`Result`), but the quality of error messages, the correctness of classification, and the
propagation of context all need explicit testing.

### Ensuring `Send + Sync`

Error types that cross thread boundaries (which is nearly all of them in a system with a
scheduler) must be `Send + Sync`. Assert this at compile time:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    fn assert_send_sync<T: Send + Sync>() {}

    #[test]
    fn error_types_are_send_sync() {
        assert_send_sync::<ConnectorError>();
        assert_send_sync::<CuratorError>();
        assert_send_sync::<AppError<CuratorErrorKind>>();
    }
}
```

This catches accidental inclusion of `Rc`, `Cell`, or other non-thread-safe types in error
types. It is a zero-cost test (the function body is empty; the check is purely in the type
system).

### Testing `Display` output

Error messages are part of the user-facing contract. Test them:

```rust
#[test]
fn timeout_message_is_human_readable() {
    let err = ConnectorError::Timeout {
        elapsed: Duration::from_secs(30),
        source: make_fake_ureq_error(),
    };
    assert_eq!(err.to_string(), "connection timed out after 30s");
}

#[test]
fn auth_error_includes_guidance() {
    let err = ConnectorError::AuthFailed { provider: "IGDB" };
    let msg = err.to_string();
    assert!(msg.contains("credentials"), "message should mention credentials: {msg}");
}
```

### Testing error classification

If errors carry classification (via `ErrorKind` or `Classified` trait), test that the
classification is correct:

```rust
#[test]
fn timeout_is_transient() {
    let err = make_timeout_error();
    assert!(err.kind().is_transient());
}

#[test]
fn auth_failure_is_not_transient() {
    let err = make_auth_error();
    assert!(!err.kind().is_transient());
    assert!(err.kind().guidance().is_some());
}
```

### Testing error conversion chains

Verify that `From` implementations work correctly across layers:

```rust
#[test]
fn connector_error_converts_to_curator_error() {
    let connector_err = ConnectorError::Timeout { /* ... */ };
    let curator_err: CuratorError = connector_err.into();
    assert!(matches!(curator_err, CuratorError::Connector(_)));
}

#[test]
fn curator_error_converts_to_anyhow() {
    let err = CuratorError::NotFound { id: EntityId::new(42) };
    let anyhow_err: anyhow::Error = err.into();
    // Can downcast back
    assert!(anyhow_err.downcast_ref::<CuratorError>().is_some());
}
```

### Testing `#[should_panic]` with expected messages

For code paths that should panic (invariant violations, impossible states), use
`#[should_panic(expected = "...")]`. Note that the `expected` string matches against the
`Debug` representation, not `Display`:

```rust
#[test]
#[should_panic(expected = "invalid state transition")]
fn rejects_invalid_transition() {
    work_unit.transition(State::Complete, State::Pending);
}
```

### Property testing error paths

The `proptest` and `quickcheck` crates can generate random inputs to exercise error paths
that are hard to hit manually:

```rust
proptest! {
    #[test]
    fn parse_never_panics(input in ".*") {
        let _ = parse_metadata(&input);  // Should return Err, never panic
    }
}
```

### Integration testing with error injection

For testing retry logic and batch error aggregation, inject errors via trait
implementations:

```rust
struct FailingConnector {
    fail_count: AtomicU32,
    fail_until: u32,
}

impl Connector for FailingConnector {
    fn fetch(&self, _id: EntityId) -> Result<Metadata, ConnectorError> {
        let n = self.fail_count.fetch_add(1, Ordering::Relaxed);
        if n < self.fail_until {
            Err(ConnectorError::Timeout { elapsed: Duration::from_secs(1), /* ... */ })
        } else {
            Ok(Metadata::default())
        }
    }
}
```

This lets you verify that the scheduler retries the correct number of times, that the
batch result accurately reports partial failures, and that notifications are emitted at
the right thresholds.
