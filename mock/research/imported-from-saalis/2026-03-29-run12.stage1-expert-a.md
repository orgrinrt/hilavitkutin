## Part 1: Independent Assessment

### 1.1 Summary of what exists

The verkko design specifies a household P2P mesh (2-12 clusters, ~36
peers) with an AP consistency model (Gilbert & Lynch, 2002). The
replicated state primitives are well-chosen:

- **Control log:** GSet (Shapiro et al., 2011) with hash-linking and
  Lamport clocks. Append-only, merge = union. This is textbook CRDT
  usage and converges by construction.

- **Epoch counters:** G-Counter (merge = max per component). Again,
  standard and convergent.

- **Gateway election:** Max-register CRDT. Convergent by construction.

- **Scar entries:** GSet membership with read-time decay weighting.
  Append-only merge. Convergent.

- **Content filters:** Counting cuckoo filters with Merkle
  reconciliation. Anti-entropy protocol (Demers et al., 1987) for
  divergence detection.

- **Dominance cascade:** Pure function of GSet state. Deterministic
  admin succession without election. This is the correct approach
  for avoiding FLP impossibility (Fischer, Lynch & Paterson, 1985).

The convergence architecture (three fronts with gated sequencing) is
structurally sound. The use of typestate for enforcing front ordering
(Pattern Registry Structure 4) prevents the most dangerous class of
convergence bugs at compile time.

The sealed-range catch-up mechanism is a good design for handling
offline peers: fast-forward for short absences, sealed range for
medium absences, suspension for long absences. This mirrors the
approach used in Bayou (Terry et al., 1995) and later in CRDTs for
mobile devices.

### 1.2 What is specified well

**Epoch synchronization.** The G-Counter approach with heartbeat-carried
macro-epoch and HKDF fast-forward is correct. Each peer can independently
derive any epoch key from any earlier key. The micro-epoch sub-ratchet
adds per-frame forward secrecy without coordination. The rotation commit
protocol (sender-asynchronous, no quorum) correctly avoids the need for
distributed agreement.

**Health pipeline.** The dual-EMA approach with per-observer computation
avoids the need for consensus on health state. Each peer computes health
from its own observations. The asymmetric recovery/degradation timing
(~10 obs to degrade, ~100 to recover) is a well-known pattern from
circuit breaker design. The DAMPED mode during convergence prevents
the convergence-health feedback loop identified in Alvaro et al.'s
lineage-driven fault injection work.

**Fencing tokens.** The grace window mechanism (10 minutes, Revoke only
for stale tokens) correctly handles split-brain KeyBundle conflicts.
The priority ordering (Revoke > Grant > Rotate) ensures that the most
security-critical operation always wins, which follows the "safety over
liveness" principle.

**Territorial re-encryption.** The deterministic nonce construction
enables idempotent re-encryption without coordination. Multiple peers
re-encrypting the same blob independently produce identical ciphertext.
This is a correct application of deterministic encryption for achieving
convergence in replicated storage, similar to the approach in
convergent encryption (Douceur et al., 2002, though for a different
purpose).

### 1.3 What is missing or underspecified

The design has several critical gaps in my domain. I will organize
these from most to least severe.

**P2-3: Matter sync and replication.** The SyncReconciliation defined
concept states "Merge rules: field-level, deterministic, reproducible
on all peers" and "Conflict resolution is deterministic: given the same
inputs, all peers produce the same result." These are requirements, not
a specification. The body section says nothing about how mesh-wide
replicated state is synchronized beyond filter reconciliation. The
Replication defined concept says "Background replication work maintains
targets" but specifies no protocol for:
- How a peer discovers it is under-replicated
- How replication targets are distributed as configuration
- How replication work is coordinated (or not) across peers
- What happens when two peers simultaneously try to replicate to
  the same third peer
- How replication interacts with partition healing

This is the most significant gap in the design from my perspective.

**D-004: Intra-cluster communication.** The dest_peer field (u16) exists
in the wire format but the session model is blank. Key questions:
- Does intra-cluster traffic use CIK or DEK for Noise KK?
- Is there one Noise session per peer pair within a cluster, or does
  the gateway multiplex?
- How does a non-gateway peer receive frames addressed to it from
  outside the cluster?
- What happens to intra-cluster sessions when the gateway changes?

This gap is particularly concerning because intra-cluster communication
is the foundation for CIK replication, gateway handoff, and observation
path forwarding. All of these mechanisms are specified at a level that
assumes working intra-cluster transport.

**D-005: Channel lifecycle.** The CHANNEL_CREATE entry type (0x0E) is
reserved in the control log but creation semantics are unspecified.
From a replication perspective, channel creation is a write to shared
state (the control log GSet). The design needs to specify:
- Who can create channels (admin only? any enrolled peer?)
- What the CHANNEL_CREATE payload contains
- How the initial channel key is distributed after creation
- How channel creation interacts with ongoing convergence
- Whether channel deletion exists (it should not, given GSet semantics)

**Relay transit forwarding (P2-4).** The relay operations wire format
defines five operations (PUT, GET, DELETE, TOUCH, SIGNAL) but the
forwarding path for transit relay is blank. The hop_count field in the
outer header exists and decrements per relay, but:
- How does a relay peer decide where to forward?
- How does the forwarding table get populated?
- How is the forwarding authenticated (relay sees connection IDs
  but cannot decrypt)?
- What prevents forwarding loops?

### 1.4 Convergence analysis of existing mechanisms

I will now systematically check whether each stateful mechanism
converges after partition heal, which is the core question for any
AP system.

**Control log (GSet).** Converges. GSet merge = union. Idempotent,
commutative, associative (Shapiro et al., 2011). Hash-linking provides
tamper evidence but does not affect convergence. The Lamport clock
provides total ordering within the control plane. After partition heal,
both sides exchange their logs, and union produces the merged state
on all peers deterministically. No issues here.

**Epoch counters (G-Counter).** Converges. Each peer maintains a
component that only increases. Merge = componentwise max. After
partition heal, peers fast-forward via HKDF chain. The micro-epoch
counter resets on macro-epoch advance, which could cause ambiguity
if two peers in different partitions advance to the same macro-epoch
with different micro-epoch states. However, since micro-epoch keys
are derived deterministically from the macro-epoch key, and
macro-epoch keys are G-Counter-synchronized, this converges correctly.
The key_id matching protocol (iterate HKDF forward up to 720 steps)
handles temporary disagreement.

**Gateway election (max-register CRDT).** Converges. Max-register
merge = max. After partition heal, each partition's gateway may
differ. The max-register resolves this deterministically. However,
I note a subtlety: the gateway election uses availability as primary
criterion and ETT as tiebreaker. "Availability" during a partition
is ambiguous -- a peer may be available within its partition but not
across the partition boundary. The design says "per-connected-component"
gateway singularity (Invariant 2), which correctly handles this. After
partition heal, the CPSK epoch binding renegotiates sessions, and the
merged state produces a single gateway per cluster. This converges,
but there is a transient period where two peers may believe they are
gateway. The design acknowledges this (Section 2.3, transient
dominance disagreement).

**Dominance cascade.** Converges. The dominance score is a pure
function of GSet state (enrollment certs) and current epoch. After
GSet merge completes, all peers with the same GSet state compute the
same dominance scores deterministically. The BLAKE3 hash mixing with
epoch provides temporal rotation. No consensus required. This is one
of the strongest parts of the design.

**Health scores (dual-EMA).** Does NOT need to converge across peers.
Health is per-observer. Each peer computes health from its own SACK
observations. This is the correct approach: trying to achieve
consensus on health state would introduce coordination requirements
that conflict with the AP model. The observation replay during
convergence (per-observer only, stale cutoff, sorted by timestamp)
is also correct.

**Content filters (cuckoo filters).** Converge via anti-entropy
(Merkle reconciliation). The Merkle root in every heartbeat provides
continuous divergence detection. Delta sync for incremental changes.
Full-filter-sync as fallback. This follows the classic Demers et al.
(1987) anti-entropy pattern. Convergence time is proportional to
gossip diameter, which is bounded by the 12-cluster mesh size. No
issues.

**Scar entries (GSet).** Converge. Same as control log. The read-time
decay weighting does not affect convergence because it is a pure
function applied at query time, not a mutation of the GSet.

**Fencing tokens.** Converge with caveats. The grace window (10 min,
Revoke only for stale tokens) handles the split-brain scenario where
two admins issue conflicting KeyBundles during a partition. After
partition heal, the higher fencing token wins (monotonic). Stale
Grant and Rotate are discarded. Revoke is always honored. This is
correct: the Revoke-wins policy ensures safety (no unauthorized
access survives partition heal) at the cost of potential loss of
Grants issued during partition. This is the right tradeoff for a
security-sensitive system.

**Replication state.** UNSPECIFIED. The Replication defined concept
says "configurable min_replicas and target_replicas per matter
priority" but does not specify how replication targets are
replicated (ironic) or how peers reach agreement on target levels.
This must be addressed.

### 1.5 Partition behavior analysis

I will now walk through the complete partition-heal sequence from a
replication architect's perspective, identifying where the design is
solid and where gaps exist.

**Pre-partition state.** All clusters have consistent GSet state.
Epoch counters agree. Content filters agree (within anti-entropy
convergence window). Health scores are per-observer.

**During partition.** Each partition operates independently (AP model).
Epoch counters advance independently in each partition (they may
diverge if hourly boundaries are crossed). Scar entries accumulate
independently. Admin succession may differ between partitions (both
partitions may have different admins if GSet state differs). Content
changes happen independently in each partition.

**Partition heal (convergence).** The three-front convergence
protocol handles this:

Front 1 (Control): GSet merge, PartitionAnnouncement, health EMA
reset. This is correct and sufficient for control state.

Front 2 (Consensus): Fencing token resolution, dominance
recomputation. Correct: the Revoke-wins policy and monotonic fencing
tokens ensure safety.

Front 3 (Re-encryption): Territorial re-encryption under new keys.
Correct: deterministic nonces ensure idempotent convergence.

**What is missing from the partition-heal sequence:**

1. **Content filter reconciliation timing.** The filter reconciliation
   happens "on reconnection" but is not explicitly sequenced with the
   three convergence fronts. Should it happen before Front 1? During
   Front 1? After convergence? If filter reconciliation happens
   during convergence while content state is still being merged, the
   reconciled filter may be immediately outdated.

2. **Replication target reconciliation.** If replication targets
   changed during the partition (e.g., admin in partition A raised
   min_replicas for critical content), how is this reconciled?
   Replication targets are presumably part of the control log or
   channel configuration, but the mechanism is unspecified.

3. **Stash reconciliation.** Each peer's stash is local (not
   replicated). After partition heal, peers may have different
   content. The content filter reconciliation handles advertising
   what each cluster has, but the mechanism for bringing
   under-replicated content up to target is unspecified.

4. **Transfer session recovery.** If a matter transfer was in
   progress when the partition occurred, the TransferSession has
   state (have_bitmap, transfer_id) that is per-session. The
   design says CONN_CLOSE implicitly aborts all in-progress
   transfers. But what triggers the restart? Is there a mechanism
   to resume interrupted transfers after partition heal, or does
   the requester start from scratch? The have_bitmap in
   TransferRequest suggests resumption is intended, but the trigger
   for restarting transfers is unspecified.

### 1.6 Assessment of CRDT usage

The design uses CRDTs correctly for the mechanisms that are specified.
Specifically:

- GSet for append-only state (control log, scars): correct.
- G-Counter for epoch synchronization: correct.
- Max-register for gateway election: correct.

These are all monotonic lattice types (Shapiro et al., 2011, Structure
1 in the Pattern Registry). The merge function is well-defined, and
convergence follows from the lattice properties (idempotent,
commutative, associative merge).

I note that the design does NOT use any multi-writer CRDTs
(LWW-Register, MV-Register, OR-Set). This is appropriate for this
domain: the control log is a GSet (add-only), epoch counters are
G-Counters (increment-only), and gateway claims are max-registers
(max-only). The single-writer pattern for KeyBundles (fencing token)
avoids the need for multi-writer conflict resolution.

**One concern:** the design mentions "field-level merge rules" for
SyncReconciliation but does not specify which CRDT type is used for
each field. If any field requires last-writer-wins semantics, the
design needs to specify the conflict resolution function explicitly.
LWW requires a total order on writes, which typically requires
timestamps. The HLC provides this, but the design does not connect
HLC to SyncReconciliation.

### 1.7 Analysis of causal ordering

The design uses two clock types:

- **Lamport clocks** for control log total ordering.
- **HLC** (Kulkarni et al., 2014) for data plane causal ordering.

This separation is correct. Control operations (enrollment, revocation,
key operations) need total ordering. Data operations (content changes,
metadata updates) need causal ordering.

The HLC provides causal consistency within the data plane. The design
says "HLC + peer_id tiebreaker total order" for the data plane, which
provides total ordering when needed. This is stronger than pure causal
ordering and is appropriate for LWW conflict resolution if needed.

**Concern:** The causal relationship between control plane and data
plane is not specified. If a key revocation (control plane, Lamport
clock) should causally precede all data encrypted under the new key
(data plane, HLC), the design needs to specify how the two clock
domains interact. Currently, the convergence front sequencing
(Front 1 before Front 2 before Front 3) provides this ordering
implicitly during convergence, but during normal operation (no
partition), the ordering between a KeyBundle(Rotate) and the first
frame encrypted under the new key is not formally specified.

### 1.8 Evaluation of the consistency model

The design explicitly claims AP (CAP theorem, Gilbert & Lynch, 2002)
with eventual consistency (Bailis, 2013). This is the correct choice
for a household mesh that must operate during network partitions
(WiFi outages, power failures, devices going offline).

The consistency model table in Section 9 of verkko-mesh correctly
identifies five domains with different consistency levels:

1. Control log: causal within scope, total via Lamport + dominance.
2. Key state: single-writer with fencing token.
3. Content routing: eventual with periodic anti-entropy.
4. Health scores: local-only, per-observer.
5. Resources: proportional control, ISS-stable.

This is a well-structured consistency taxonomy. Each domain has the
weakest consistency model that satisfies its requirements, which is
the correct application of Bailis' coordination avoidance principle
(Bailis et al., 2014).

The Hellerstein CALM theorem (Hellerstein, 2010) states that
monotonic programs can be computed in an eventually consistent
fashion without coordination. The verkko design achieves this by
using monotonic data structures (GSet, G-Counter, max-register)
for all replicated state. This is not an accident; it is a
deliberate design choice that the Pattern Registry documents
(Structure 1: JoinSemilattice, 14 instances).

## Part 2: The 5-7 Most Significant Problems

### Problem 1: SyncReconciliation has no merge specification (P2-3)

**(a) Problem statement.**

The SyncReconciliation defined concept claims "Merge rules:
field-level, deterministic, reproducible on all peers" and
"Conflict resolution is deterministic." But the body contains no
merge rules. The Replication defined concept says "Background
replication work maintains targets" but specifies no protocol.
This is not an underspecification; it is an absence. The entire
replication domain beyond filter reconciliation is a blank page.

**(b) Why it matters.**

Replication is the reason the mesh exists. Without a replication
protocol, matter that exists on one peer cannot reliably reach
another. The design has excellent content transfer mechanics
(MatterChunk, TransferSession, bao verification) but no protocol
for deciding when and where to replicate. This is like having a
postal system with trucks and routes but no mechanism for deciding
what mail to deliver where.

Without merge rules, the system cannot recover from the following
scenario: during a 4-hour partition, cluster A creates 50 new
entities and cluster B creates 30 new entities. After partition
heal, how does each cluster learn about the other's entities?
The content filter reconciliation handles advertising (Merkle
delta sync), but the actual replication decisions (what to fetch,
from whom, in what priority, to what replica count) are unspecified.

**(c) References/prior art.**

The classic reference is Demers et al. (1987), which defines three
anti-entropy modes: push, pull, and push-pull. The verkko filter
reconciliation is a pull-based anti-entropy mechanism (compare Merkle
roots, request deltas). The replication protocol should use a similar
approach but for the actual content, not just the advertisement.

Dynamo (DeCandia et al., 2007) uses a preference list and sloppy
quorum for replication placement. Cassandra uses a similar approach
with configurable replication factor. For a household mesh, the
simpler approach from Riak's "hinted handoff" (Klophaus, 2010) is
more appropriate: when a peer discovers under-replicated content,
it requests the content from any peer that has it.

**(d) Recommended approach.**

I propose a pull-based anti-entropy replication protocol. The design
already has the building blocks (content filters, source selection,
matter transfer). What is missing is the orchestration:

```
ReplicationPolicy {
    min_replicas: u8,       // minimum copies for durability
    target_replicas: u8,    // desired copies for availability
    priority: MatterPriority,
    scope: ReplicationScope, // PerCluster | MeshWide
}

// Distributed via control log entry (CHANNEL_CONFIG or similar)
// Each peer independently evaluates replication state:

on_filter_reconciliation_complete():
    for blob_id in local_stash.all():
        known_holders = count_holders_from_filters(blob_id)
        target = replication_policy(blob_id).target_replicas
        if known_holders < target AND i_am_holder(blob_id):
            candidates = peers_without_blob(blob_id)
                .filter(health_ok)
                .filter(has_capacity)
                .sort_by(relationship_age)
            for candidate in candidates.take(target - known_holders):
                enqueue_push_replication(blob_id, candidate)

on_heartbeat_filter_divergence(remote_cluster):
    for blob_id in filter_diff(remote_cluster):
        if not local_stash.has(blob_id):
            if replication_policy(blob_id).scope == MeshWide:
                if count_local_replicas(blob_id) < min_replicas:
                    enqueue_pull_replication(blob_id, remote_cluster)
```

Key properties:
- **No coordination.** Each peer independently evaluates replication
  state from its local view. Multiple peers may initiate redundant
  transfers; the content-addressed nature of matter makes this
  idempotent.
- **Convergent.** The filter reconciliation eventually reveals all
  content. The replication evaluation eventually brings counts to
  target. Convergence follows from the anti-entropy property.
- **Priority-aware.** The replication policy's priority field maps
  to the existing MatterPriority enum (P1-P4).
- **Partition-safe.** During partition, each partition independently
  maintains replication within its connected component. After
  partition heal, filter reconciliation reveals new content, and
  replication pulls fill the gaps.

The ReplicationPolicy should be distributed via the control log
(a new entry type or as payload of a CHANNEL_CONFIG entry). This
makes it a GSet member, convergent by construction.

Replication work should use the existing bulk tier (LEDBAT-paced)
and should be governed by the Conductor (metabolic state determines
replication aggressiveness). This follows the existing resource
governance architecture.


### Problem 2: Intra-cluster session model is absent (D-004)

**(a) Problem statement.**

The dest_peer field (u16) exists in every encrypted payload, but the
intra-cluster communication model is entirely unspecified. The
design specifies inter-cluster communication in great detail (Noise
KK/XX between gateways, CPSK management, polygon scheduling) but
says nothing about how peers within a cluster communicate.

**(b) Why it matters.**

Intra-cluster communication is the foundation for:
- CIK replication (encrypted to each peer's DEK)
- Gateway handoff (CPSK session state, observation summaries)
- Observation path forwarding (intra-cluster heartbeats)
- KeyBundle distribution (admin to cluster peers)
- Content filter updates (stash changes within a cluster)
- Intra-cluster replication (local copies for durability)

Every one of these mechanisms assumes working intra-cluster
transport. Without a session model, implementors must guess at
the security properties, key usage, and session lifecycle.

The question "CIK vs DEK for intra-cluster Noise KK" has
significant security implications. If CIK is used, compromise
of any cluster peer compromises all intra-cluster sessions (CIK
is shared). If DEK is used, each peer pair has an independent
session, but key management becomes more complex.

**(c) References/prior art.**

Local area network communication in P2P systems is typically
simpler than WAN communication because latency is low and
bandwidth is high. Membership protocols for LANs (SWIM, Serf)
use UDP gossip with periodic full state exchange.

For security within a trusted group, the Signal protocol's
"sender keys" pattern (one key per sender, shared with all
recipients) provides efficient broadcast within a group without
per-pair key management. This is similar to the broadcast
encryption approach already used for inter-cluster communication.

**(d) Recommended approach.**

I recommend DEK-based Noise KK for intra-cluster sessions, with
one session per peer pair within the cluster. This provides:
- Per-pair forward secrecy (compromise of one session does not
  affect others)
- Independent session lifecycle (one peer restarting does not
  invalidate other sessions)
- Compatibility with the existing Noise framework

The session model:

```
IntraClusterSession {
    // One Noise KK session per peer pair within a cluster
    // Static key: DEK (not CIK)
    // Reason: CIK is shared, so CIK-based sessions would all
    // be compromised by any single peer compromise. DEK-based
    // sessions provide per-pair isolation.

    peer_a: PeerId,      // u16
    peer_b: PeerId,      // u16
    session_state: NoiseKKState,

    // Session lifecycle
    // - Initiated on peer join (enrollment complete)
    // - Renegotiated on DEK rotation (rare)
    // - Torn down on peer departure
    // - No polygon scheduling (LAN RTT is negligible)
}

// Gateway routing within cluster
//
// All inter-cluster frames arrive at the gateway.
// The gateway demultiplexes by dest_peer:
//   dest_peer = 0x0000: broadcast to all cluster peers
//   dest_peer = specific: forward to that peer's intra-cluster session
//
// Non-gateway peers route inter-cluster frames through gateway.
// Gateway peer is the sole inter-cluster session holder.

// Gateway handoff for intra-cluster state
on_gateway_change(old_gateway, new_gateway):
    // Old gateway transfers CPSK session state to new gateway
    // This requires an intra-cluster reliable transfer
    // Use vital tier stream within the intra-cluster session
    //
    // New gateway establishes inter-cluster sessions
    // Old gateway's inter-cluster sessions are torn down
    // In-flight frames may be lost during the handoff window
    // This is acceptable: the heartbeat failure detection (3 missed)
    // provides the recovery trigger
```

**Addressing.** The dest_peer field is u16. Within a cluster of
1-6 peers, a simple assignment suffices: peers receive their
dest_peer ID at enrollment time. The gateway's dest_peer is
always the CIK-derived cluster ID. A broadcast to all cluster
peers uses dest_peer = 0x0000.

**Partition within a cluster.** If a cluster peer becomes
unreachable on the LAN (WiFi dropout, device sleep), the gateway
detects this via intra-cluster heartbeat timeout. The gateway
does not forward frames to the unreachable peer. When the peer
reconnects, a simplified convergence (GSet merge, filter sync)
restores state. This is simpler than inter-cluster convergence
because all cluster peers share the CIK and all channel keys.


### Problem 3: Channel lifecycle unspecified (D-005)

**(a) Problem statement.**

The CHANNEL_CREATE entry type (0x0E) is reserved in the control
log, but creation semantics, authorization model, and key
distribution for new channels are entirely absent. Channels are
the organizational unit for access control and key management,
but there is no specification for how they come into existence.

**(b) Why it matters.**

Without channel creation semantics, the system cannot add new
data partitions after initial setup. The channel abstraction is
central to the access control model (off/opaque/read/write
per-peer per-channel), but the lifecycle from creation through
key distribution to steady-state operation is a gap.

From a replication perspective, channel creation is a write to
shared state that must be disseminated to all mesh participants.
The GSet control log provides the dissemination mechanism, but
the channel creation entry must contain enough information for
all peers to initialize their local channel state (key, access
control, replication policy).

More critically, channel creation during a partition could lead
to two partitions creating channels with the same channel_index.
The current design uses channel_index as a u16, and the control
log entry includes channel_index. If two admins in different
partitions both create a channel and happen to pick the same
index, the GSet merge after partition heal would contain two
CHANNEL_CREATE entries for the same index with different keys.
This is a conflict the GSet cannot resolve because GSet merge
is union, not conflict resolution.

**(c) References/prior art.**

Amazon's DynamoDB uses a coordinated namespace for table creation
(requires the control plane). CRDTs typically handle namespace
conflicts via unique identifiers (UUIDs) rather than sequential
indices. The OR-Set (Shapiro et al., 2011) handles concurrent
add/remove but verkko uses GSet, which does not support removal.

For channel index assignment without coordination, a UUID-based
approach or a partition-aware index allocation scheme is needed.

**(d) Recommended approach.**

Channel creation should be admin-only (consistent with the
single-admin-authority model). The CHANNEL_CREATE control log
entry should contain:

```
ChannelCreatePayload {
    channel_index: u16,       // assigned by admin
    channel_key: [u8; 32],    // initial 256-bit key (CSPRNG)
    access_defaults: AccessDefaults, // per-role default access
    replication_policy: ReplicationPolicy,
    display_name_hash: [u8; 8], // truncated BLAKE3 of name
}
// Wrapped in KeyBundle(Grant) for key distribution
```

**Index conflict prevention.** Channel indices are assigned by
the admin. The admin increments a local counter for new channels.
Since admin authority is single-writer (fencing token), concurrent
channel creation with conflicting indices is prevented by the same
mechanism that prevents conflicting KeyBundles. During a partition,
if both partitions have different admins (transient dominance
disagreement), they could in theory create conflicting channels.
This is mitigated by:

1. The fencing token: a stale-token CHANNEL_CREATE (Grant) is
   rejected during the grace window (only Revoke accepted for
   stale tokens).
2. If both admins have valid fencing tokens (each incremented
   independently during the partition), the higher fencing token
   wins during Front 2 consensus. The losing partition's channel
   creation is discarded.
3. The control log stores both entries (GSet union), but only the
   winning admin's channel is active. The losing admin's channel
   is orphaned and requires re-creation after convergence.

**Channel deletion.** Channels should not be deleted (GSet
semantics). A channel can be revoked (all keys revoked, access
set to off for all peers) but the control log entry persists.
This is consistent with the append-only security model.

**Channel creation during convergence.** Channel creation should
be blocked during convergence (convergence_active = 1). This
prevents the race condition where a channel is created and
immediately invalidated by convergence state changes.


### Problem 4: Content filter reconciliation lacks explicit convergence timing

**(a) Problem statement.**

Content filter reconciliation uses Merkle tree delta sync, which
is a correct anti-entropy mechanism. However, the timing of filter
reconciliation relative to the three convergence fronts is not
specified. The design says filter reconciliation happens "on
reconnection" (Section 3.7) but the convergence specification
(Section 7.1) does not mention filter reconciliation in any front.
The convergence burst cost table shows "Filter reconciliation
(Merkle)" as "concurrent" but does not specify which front it is
concurrent with.

**(b) Why it matters.**

Filter reconciliation is the mechanism by which peers learn about
content changes that occurred during the partition. If filter
reconciliation happens before Front 1 (control log merge), the
filter state may be based on stale access control. If it happens
during Front 3 (re-encryption), the filter is advertising content
under old keys. If it happens after convergence, there is a window
where peers have converged control and key state but do not know
about each other's content.

The correct ordering depends on the invariant: "filter-stash
consistency" (Invariant 5), which is explicitly acknowledged as
"partially enforced" with "convergence time proportional to
gossip diameter." But the design should specify when during the
convergence sequence filter reconciliation begins and what gate,
if any, ensures it completes before steady-state operations resume.

**(c) References/prior art.**

Bayou (Terry et al., 1995) performs anti-entropy after session
establishment but before accepting new writes. This prevents
writes against stale state. The LBFS (Muthitacharoen et al., 2001)
approach of Merkle-based reconciliation is content-first: reconcile
what exists before worrying about metadata.

In verkko's case, the control state (keys, access control) must be
reconciled before content state (filters) because content access
depends on key state. This argues for filter reconciliation after
Front 2 but before or concurrent with Front 3.

**(d) Recommended approach.**

Filter reconciliation should be explicitly sequenced:

1. **Begin:** After Front 1 ControlGate opens (access control
   is resolved).
2. **Concurrent with:** Front 2 (consensus) and Front 3
   (re-encryption). Filter reconciliation does not depend on
   key state finalization because filters advertise blob_ids
   (content hashes), not encrypted content.
3. **Gate:** No blocking gate. Filter reconciliation is
   asynchronous anti-entropy. If a peer has not completed
   filter reconciliation by the time convergence exits, it
   continues in steady state. This is consistent with the
   "eventual consistency" property of Invariant 5.
4. **Exit sequence interaction:** Step 5 of the exit sequence
   (update health baselines) should wait for filter
   reconciliation to complete, or at minimum for the initial
   Merkle delta exchange to complete. This ensures that
   health baselines are computed with accurate filter state.

Add to the convergence specification:

```
Front 1.5 (concurrent with Fronts 2-3):
    Filter reconciliation begins after ControlGate opens.
    Merkle delta exchange with each reconnected cluster pair.
    Completes asynchronously. No blocking gate.
    Initial delta exchange expected within 2 heartbeat intervals
    (~10 seconds at Active).
    Remaining divergence resolves via steady-state anti-entropy.
```


### Problem 5: Replication coordination absence creates thundering herd risk

**(a) Problem statement.**

The design says "Background replication work maintains targets"
(Replication defined concept) but provides no mechanism for
preventing multiple peers from simultaneously replicating the
same content to the same destination. In a mesh where all peers
independently evaluate replication state from content filters,
multiple peers may simultaneously discover that a particular
blob is under-replicated and all initiate transfers to the same
target peer.

**(b) Why it matters.**

At the scale of this system (2GB ROM files, 36 peers), a
thundering herd of redundant transfers wastes significant
bandwidth. If three peers each initiate a 2GB transfer to the
same target for the same blob, that is 6GB of wasted WAN
bandwidth. On a 5 Mbps uplink, that is 2.7 hours of
unnecessary traffic.

The LEDBAT pacing provides congestion control (transfers will
not degrade foreground traffic), and the content-addressed
nature of matter means redundant transfers are not harmful to
correctness. But the efficiency loss is significant for
household networks with constrained uplinks.

**(c) References/prior art.**

Dynamo (DeCandia et al., 2007) uses a coordinator node for each
key to prevent thundering herd. Cassandra uses a similar
approach. For an AP system without coordination, the standard
approach is probabilistic deduplication: each peer independently
decides whether to replicate with a probability inversely
proportional to the number of known holders. This is the
approach used in epidemic algorithms (Demers et al., 1987) where
the "push probability" decreases as the number of infected nodes
increases.

**(d) Recommended approach.**

Use probabilistic replication with content-addressed deduplication:

```
fn should_replicate(blob_id: BlobId, known_holders: u8,
                     target_replicas: u8) -> bool {
    if known_holders >= target_replicas {
        return false;
    }
    // Probability of initiating replication decreases with
    // number of known holders that could also initiate
    let deficit = target_replicas - known_holders;
    let probability = deficit as f64 / (known_holders as f64 + 1.0);
    // Deterministic per (blob_id, my_dk, epoch) to prevent
    // re-evaluation jitter
    let roll = BLAKE3("saalis-repl-v1" || blob_id || my_dk || epoch)[0]
               as f64 / 255.0;
    roll < probability
}
```

This provides:
- **No coordination.** Each peer decides independently.
- **Probabilistic deduplication.** Expected redundant transfers
  = O(1) regardless of holder count.
- **Convergent.** The deficit decreases as replicas are created,
  eventually reaching zero.
- **Deterministic per epoch.** The BLAKE3 roll prevents a peer
  from re-evaluating and flipping its decision within the same
  epoch.

Additionally, the TransferRequest message should include a
"replication_priority" field so the source can prioritize
transfer sessions. If a source receives multiple
TransferRequests for the same blob_id from different requesters,
it can serve them sequentially (reusing the bao tree state) or
decline excess requests.


### Problem 6: Seat consistency model during convergence lacks causal guarantees

**(a) Problem statement.**

The seat consistency model specifies SNAPSHOT mode during
convergence: "Reads return a frozen snapshot taken at convergence
entry." Deferred writes are replayed after convergence with
"mesh-wins" conflict resolution. However, the design does not
specify how the snapshot is taken, what consistency guarantees
the snapshot provides, or how deferred writes are causally ordered
relative to convergence state changes.

**(b) Why it matters.**

From a distributed systems perspective, a "frozen snapshot" without
a defined consistency level is ambiguous. Does the snapshot reflect
the last committed state before convergence? The state at the
moment the convergence_active flag was set? The state at the
moment the seat received the ConsistencyModeChange event?

The deferred write replay with "mesh-wins" conflict resolution is
last-writer-wins (LWW) with the mesh as the winner. This means
that any entity modified by both the mesh (during convergence) and
the seat (deferred write) will have the mesh's version. The seat
is not notified until the WriteConflict event, which may arrive
after the seat has already built UI state on the assumption that
its write succeeded. This violates the principle of least surprise
for the user.

**(c) References/prior art.**

Bayou (Terry et al., 1995) handles tentative and committed writes
explicitly: the client knows that a write is tentative until the
primary commits it. This provides clear expectations. The COPS
system (Lloyd et al., 2011) provides causal+ consistency, which
ensures that reads reflect a causally consistent snapshot.

For verkko's seat model, the simplest correct approach is to use
the HLC timestamp at convergence entry as the snapshot timestamp
and ensure that the snapshot reflects all state with HLC <=
that timestamp.

**(d) Recommended approach.**

1. **Define the snapshot precisely.** The snapshot is taken at
   the HLC timestamp when the host peer enters convergence
   (sets convergence_active = 1). The snapshot reflects all
   committed state with HLC <= snapshot_hlc.

2. **Deferred writes carry HLC.** Each deferred write is
   timestamped with the seat's local HLC (derived from the
   session's HLC sync). After convergence, deferred writes are
   replayed in HLC order.

3. **Conflict detection, not resolution.** Instead of
   silently applying "mesh-wins," the host peer should detect
   conflicts and present them to the seat as WriteConflict
   events with both versions (seat's write and mesh's state).
   The seat (or user) chooses. This is more appropriate for a
   content library manager where user intent matters.

4. **Bounded staleness indicator.** The staleness_ms field in
   SeatResponse should be a precise measurement: `staleness_ms =
   host_peer_hlc.now() - snapshot_hlc`. This gives the seat
   accurate information about snapshot freshness.

```
// Snapshot semantics
SeatSnapshot {
    snapshot_hlc: HlcTimestamp,
    // All reads return state as of snapshot_hlc
    // Deferred writes are buffered with their own HLC
}

// Conflict resolution
enum WriteConflictResolution {
    MeshWins,     // default if seat does not respond
    SeatWins,     // seat explicitly chooses its version
    Manual,       // user resolves
}
```

**Counterargument to my position:** One could argue that for a
content library manager, automatic "mesh-wins" is safer because
the mesh represents the consensus state and user writes during
convergence are likely to be configuration changes that should
not override security-critical state changes. This is a valid
point for KeyBundle-related state, but for content metadata
(ratings, tags, descriptions), user intent should take precedence.
A hybrid approach (mesh-wins for security state, seat-wins for
user content) may be optimal.


### Problem 7: No specification for how replication targets survive partition

**(a) Problem statement.**

Replication targets (min_replicas, target_replicas per matter
priority) are mentioned in the Replication defined concept but
their distribution mechanism is unspecified. If replication
targets are configuration state, they must be replicated
themselves. If they are part of the control log, they are
GSet-convergent. If they are local configuration, they may
diverge across the mesh after a partition.

**(b) Why it matters.**

Consider a partition where cluster A's admin raises min_replicas
from 2 to 3 for P1 content. During the partition, cluster B's
admin (different admin due to dominance disagreement) keeps
min_replicas at 2. After partition heal, which value wins?

If replication targets are in the control log (GSet), both
values are preserved (union). But then the system has two
conflicting policies and no resolution mechanism.

If replication targets are a max-register (merge = max), the
higher value always wins. This is safe (more replication is
always at least as durable) but may be undesirable if the higher
value was set by a partition admin who lost dominance.

If replication targets are per-channel configuration managed by
KeyBundle, they follow the fencing token ordering and the
dominant admin's configuration wins. This is the most consistent
approach.

**(c) References/prior art.**

Amazon S3 uses fixed replication factors per storage class. HDFS
uses configurable replication per file. In both systems,
replication configuration is managed by a central coordinator.
For an AP system, replication configuration should be treated as
part of the control state and use the same convergence mechanism
as other control state.

**(d) Recommended approach.**

Replication targets should be per-channel configuration,
distributed via the control log as a new entry type
(CHANNEL_CONFIG, or as payload within KEY_GRANT). This makes
them subject to the fencing token ordering: the dominant admin's
configuration wins after convergence.

```
// New control log entry type: CHANNEL_CONFIG (0x0F)
ChannelConfigPayload {
    channel_index: u16,
    min_replicas: u8,
    target_replicas: u8,
    replication_scope: ReplicationScope,
    bandwidth_policy: BandwidthPolicyRef,
}

// Merge rule: latest by fencing token (same as KeyBundle)
// During grace window: only changes that increase min_replicas
// are accepted from stale tokens (analogous to Revoke-wins)
```

The "increase-wins" policy for stale-token CHANNEL_CONFIG follows
the same safety principle as Revoke-wins for KeyBundle: the more
conservative (safer) operation takes precedence during partition
heal.

## Part 3: Open Items and Known Limitations

### Open items within my domain

1. **SyncReconciliation body content.** The defined concept exists
   but the body section is entirely absent. This review proposes
   a pull-based anti-entropy approach (Problem 1), but the design
   team should validate this against the content model.

2. **Transfer session resumption trigger.** After partition heal
   or peer restart, interrupted transfers should be resumed. The
   have_bitmap in TransferRequest supports this, but the trigger
   for re-evaluation (when does a peer decide to resume an
   interrupted transfer?) is unspecified. I recommend: on filter
   reconciliation completion, re-evaluate all blobs with status =
   Downloading in the local stash.

3. **Replication policy distribution timing.** If replication
   targets change during convergence (admin changes min_replicas),
   when does the new policy take effect? I recommend: after Front 2
   consensus (fencing token resolved, dominant admin determined).
   Replication evaluation based on new targets begins after the
   exit sequence completes.

4. **Intra-cluster partition detection.** The design specifies
   inter-cluster partition detection (3 missed heartbeats) but
   does not specify intra-cluster heartbeat behavior. If a cluster
   peer goes offline on the LAN, how long before the gateway
   detects this? I recommend: same heartbeat mechanism, but with
   shorter intervals (LAN latency permits 1-2 second heartbeats
   at negligible cost).

5. **Content filter false positive handling during replication.**
   The cuckoo filter has a ~0.78% false positive rate. During
   replication evaluation, a peer may believe that a remote cluster
   holds a blob (filter says yes) when it does not (false positive).
   The TransferRequest will fail (source does not have the blob),
   but this wastes a round trip. For replication evaluation, the
   design should specify how false positives are handled: retry
   with a different source, or mark the blob as "needs verification"
   and use the full Merkle reconciliation to confirm.

6. **Channel creation and initial key distribution atomicity.** When
   a channel is created, the initial key must be distributed to all
   authorized peers. This is a two-step operation (control log entry
   + KeyBundle Grant). If the admin crashes between the control log
   entry and the KeyBundle, the channel exists but no peer has the
   key. I recommend: the CHANNEL_CREATE control log entry should
   contain the encrypted key material (wrapped to each authorized
   DEK), making creation and initial key distribution atomic.

### Known limitations of my analysis

1. I have not analyzed the cryptographic properties of the epoch
   ratchet or the PCS mechanism. Those are outside my domain. I
   trust the crypto layer's convergence (HKDF chain is deterministic
   from shared state) but cannot verify the security properties.

2. I have not analyzed the metabolic governor or resource governance
   mechanisms in detail. These are primarily systems engineering
   concerns, not replication concerns.

3. My replication protocol proposal (Problem 1) is deliberately
   simple. A production system may need more sophisticated
   replication placement (rack-aware, storage-tier-aware). For a
   household mesh of 2-12 clusters, the simple approach is
   appropriate.

4. I have not analyzed the relay's role in replication. The Membrane
   model (relay holds ciphertext) means relay peers can participate
   in replication without accessing plaintext. The design mentions
   this but does not specify how relay-held replicas count toward
   replication targets or how they are accessed during content
   retrieval.

5. My analysis of intra-cluster communication (Problem 2) assumes
   DEK-based sessions. The design team may choose CIK-based
   sessions for simplicity. The security tradeoffs should be
   evaluated by the crypto expert.

## Part 4: Ideal Design Synthesis

The verkko design has a strong foundation in replicated state theory.
The choice of monotonic CRDTs (GSet, G-Counter, max-register) for
all replicated state is correct and avoids the coordination
requirements that would be imposed by stronger consistency models.
The dominance cascade (consensus-free admin succession) is an
elegant application of deterministic functions over eventually
consistent state, directly inspired by the CALM theorem's insight
that monotonic programs do not require coordination.

The convergence protocol (three fronts, gated sequencing, typestate
enforcement) is structurally sound. The Revoke-wins policy for
fencing tokens ensures safety during partition heal. The
deterministic nonce construction for re-encryption enables
coordination-free convergence of encrypted storage state.

The primary deficiency is the absence of replication orchestration.
The design has all the building blocks:
- Content addressing (blob_id = BLAKE3(plaintext))
- Content advertisement (cuckoo filters with Merkle reconciliation)
- Content transfer (MatterChunk, TransferSession, bao verification)
- Source selection (health-aware, possession-aware, access-aware)
- Bandwidth governance (LEDBAT, BandwidthPolicy, metabolic state)

What is missing is the glue: the protocol that connects "I know
content X exists at cluster Y" to "I should fetch content X from
cluster Y because my local replica count is below target." This
is a pull-based anti-entropy protocol that should be specified in
verkko-matter.

The intra-cluster communication gap is less severe but still
important. The design assumes working intra-cluster transport for
several mechanisms (CIK replication, gateway handoff, observation
forwarding) without specifying the session model. This should be
addressed in Run 12 to unblock implementors.

The channel lifecycle gap is a design decision, not a specification
task. The key question is whether channel creation requires
coordination (admin-only) or can be decentralized. Given the
single-admin-authority model and the fencing token mechanism, I
strongly recommend admin-only channel creation to avoid the
channel_index conflict problem described in Problem 3.

If I were to design the ideal version of this system from scratch,
I would:

1. **Start with the replication protocol.** Define ReplicationPolicy
   as a per-channel setting distributed via control log. Define a
   pull-based anti-entropy loop driven by filter reconciliation.
   Use probabilistic deduplication to prevent thundering herd.

2. **Specify intra-cluster communication early.** Use DEK-based
   Noise KK for intra-cluster sessions. Define the gateway routing
   model (gateway demultiplexes by dest_peer). Define the handoff
   protocol for gateway changes.

3. **Make channel lifecycle explicit.** Admin-only creation, no
   deletion (GSet), CHANNEL_CREATE control log entry with embedded
   key material. Block channel creation during convergence.

4. **Formalize filter reconciliation timing.** After ControlGate
   opens, concurrent with Fronts 2-3, asynchronous completion.
   No blocking gate. Steady-state anti-entropy continues after
   convergence exit.

5. **Add a DomainSeparator for replication decisions.**
   `DomainSeparator::ReplicationRoll = 9` with string
   `"saalis-repl-v1"` for the probabilistic replication roll.
   This ensures the replication decision is deterministic per
   (blob_id, peer, epoch) and domain-separated from other BLAKE3
   usages.

The design is approximately 62% specified according to the Run 11
synthesis. With the replication protocol, intra-cluster session
model, and channel lifecycle specified, I estimate the coverage
would reach approximately 78-82%. The remaining gaps (relay transit
forwarding, multi-hop relay, seat failover) are lower priority and
can be addressed in subsequent runs without blocking the core
implementation.
