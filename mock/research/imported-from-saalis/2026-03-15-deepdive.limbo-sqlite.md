# Deep Dive: Limbo / Turso Database — SQLite Rewritten in Rust

**Date:** 2026-03-15
**Scope:** Development status, architecture, API surface, feature completeness, performance, io_uring, and feasibility as a future migration target for saalis.

---

## 1. Project Identity and History

**Limbo** was announced in December 2024 by Turso (the company behind libSQL) as a ground-up rewrite of SQLite in Rust. The project was originally codenamed "Project Limbo" and lived at `github.com/tursodatabase/limbo`. In January 2025, Turso announced they were going "all-in" on the rewrite, renaming the project from Limbo to simply **Turso** (now at `github.com/tursodatabase/turso`) and making it the company's flagship effort. The stated goal: build the "next evolution of SQLite" with full memory safety, native async I/O, and modern concurrency — while remaining compatible with SQLite at the file format and SQL dialect level.

The project launched with extraordinary community momentum: 32 contributors on announcement day, 70 within the first week, and as of March 2026, the repository has ~17.8k stars, ~791 forks, ~460 open issues, and over 15,900 commits to main from 50+ contributors.

**License:** MIT (same as libSQL).

**Relationship to libSQL:** Turso (the company) maintains both libSQL (a fork of SQLite's C codebase) and Turso Database (the Rust rewrite). The long-term intent is that if the rewrite succeeds, the Turso Database codebase *becomes* libSQL — a full succession. libSQL remains production-ready today; Turso Database is not yet.

## 3. Architecture: How It Differs from SQLite

Turso is not a binding or wrapper around SQLite — it is a clean-room reimplementation. The core library is approximately 25,000 lines of Rust, organized into distinct subsystems:

### 3.1 Async-First Execution Model

This is the fundamental architectural difference. SQLite's `sqlite3_step()` is synchronous and blocks on I/O. Turso extends this entry point to be **asynchronous**: when the VDBE encounters an I/O operation, it yields control via an `IOResult::IO` variant rather than blocking. The caller can then poll or await completion.

All storage operations return futures/completion objects. The VDBE uses explicit state machines to preserve execution context across yields:

```
enum CursorSeekState {
    Start,
    MovingBetweenPages { eq_seen: bool },
    FoundLeaf { eq_seen: bool },
}
```

This design eliminates the need for one OS thread per database connection in multi-tenant scenarios — a key motivation.

### 3.2 I/O Abstraction Layer

The `IO` trait provides platform-specific implementations:

| Backend | Platform | Mechanism |
|---------|----------|-----------|
| `UringIO` | Linux | io_uring with zero-copy vectored I/O |
| `UnixIO` | Unix (fallback) | epoll/kqueue-based event polling |
| `IOCP` (experimental) | Windows | I/O Completion Ports (new in v0.5) |
| `MemoryIO` | All | In-memory buffers for testing/simulation |
| `GenericIO` | All | Cross-platform synchronous fallback |

Buffer pools use arena allocation for O(1) reuse across async operations.

### 3.3 VDBE (Virtual Database Engine)

Turso reimplements SQLite's register-based bytecode VM with 100+ instruction types. The design mirrors SQLite's VDBE conceptually but adds Rust's memory safety guarantees. Key types:

- **Program**: Immutable instruction sequence with metadata
- **ProgramState**: Mutable execution context (PC, registers, cursors)
- **StepResult**: Non-blocking return values (`Done`, `Row`, `IO`, `Interrupt`)

### 3.4 Storage Layer

- **B-Tree**: Interior and leaf page nodes with pointer arrays, state-machine-based cursor for async traversal, binary search within page cell arrays. Handles SQLite's variable-length record encoding.
- **Pager**: Page allocation, caching, eviction. Atomic flags for thread-safe operations. Page sizes 512–65,536 bytes (matching SQLite).
- **Page Cache**: SIEVE eviction algorithm for hot page retention, O(1) lookup and replacement.
- **WAL**: SQLite-compatible frame format with checksums. Custom 64-bit `TursoRwLock` for reader-writer coordination. Supports checkpoint modes: Passive, Full, Restart, Truncate.

### 3.5 SQL Processing Pipeline

Three-stage transformation:
1. **Parsing** (`turso_parser`): Generates AST from SQL string
2. **Translation** (`translate/emitter.rs`): Emits VDBE bytecode
3. **Optimization** (`translate/optimizer.rs`): Plan rewriting, index selection, join ordering

v0.5 significantly improved the optimizer with hash join bloom filters, correlated subquery decorrelation (EXISTS/NOT EXISTS into semi/anti joins), and disjunctive predicate optimization across multiple indexes.

### 3.6 MVCC Concurrency Control

The most ambitious departure from SQLite. SQLite uses a single-writer lock on the entire database file. Turso implements **Multi-Version Concurrency Control** inspired by Hekaton:

- `MvStore` maintains multiple row versions in-memory via a `SkipMap` keyed by (RowID, TxID)
- `LogicalClock` generates transaction timestamps
- `MvccLazyCursor` merges in-memory versions with B-Tree data per snapshot
- Visibility: `begin_ts <= version.ts <= current_ts`
- `LogicalLog` for persistent crash recovery
- Checkpoint state machine migrates committed versions to B-Tree asynchronously

Enabled via `PRAGMA journal_mode = 'mvcc'`. Moved from tech preview to beta in v0.5.

## 4. io_uring Integration

On Linux, Turso uses `io_uring` — the kernel's ring-buffer-based interface for asynchronous system calls. This is not bolted on; the entire storage layer was designed around it. Key characteristics:

- Ring buffers are shared between user space and kernel space
- I/O requests are submitted without system call overhead per request
- Completion notifications arrive asynchronously
- Zero-copy vectored I/O for page reads/writes

The io_uring implementation was battle-tested through Deterministic Simulation Testing, which uncovered subtle bugs (e.g., partial write handling — an extremely rare condition that is hard to test in production).

On macOS, the fallback is kqueue-based polling. On Windows, an experimental IOCP backend was added in v0.5. The `MemoryIO` backend replaces the actual I/O loop with a simulated one for deterministic testing.

## 5. Performance Benchmarks

### 5.1 Tail Latency (Multi-Tenant)

The headline benchmark simulated a multi-tenant serverless runtime (1–100 tenants, each with their own database, running `SELECT * FROM users LIMIT 100` x 1000). SQLite used one thread per tenant; Turso used Rust coroutines.

| Percentile | SQLite | Turso | Improvement |
|------------|--------|-------|-------------|
| p90 | ~same | ~same | negligible |
| p99 | ~same | ~same | negligible |
| p999 | high | low | **~100x reduction** |

The benefit is specifically in **tail latency**: SQLite's query latency degrades non-gracefully as thread count increases, while Turso's cooperative async model avoids the thundering-herd / context-switch pathology.

### 5.2 MVCC Write Throughput

With `BEGIN CONCURRENT` and MVCC enabled:

- SQLite: ~150,000 rows/second (flat regardless of thread count, due to single-writer lock)
- Turso (4 threads): ~200,000 rows/second
- Turso (8 threads, with 1ms compute per tx): **4x faster** than SQLite

Throughput scales with thread count because MVCC allows optimistic concurrent writes with row-level conflict detection at commit time.

### 5.3 General Performance

The team's stated position: *"Performance of Turso is already good — faster than SQLite in some cases, slower in others. The goal is not to claim that it is faster, but to highlight that a good level of performance was reached early in the project."*

## 6. Feature Completeness (SQL and C API)

Based on the authoritative [COMPAT.md](https://github.com/tursodatabase/turso/blob/main/COMPAT.md) (1,089 lines), here is the current state:

### 6.1 Overall Status

- SQLite file format: **fully supported**
- SQLite query language: **partially supported**
- SQLite C API: **partially supported**

### 6.2 SQL Statements — What Works

Fully supported: `ALTER TABLE`, `ANALYZE`, `ATTACH/DETACH DATABASE`, `BEGIN/COMMIT/ROLLBACK TRANSACTION`, `CREATE/DROP INDEX/TABLE/TRIGGER/VIEW`, `CREATE VIRTUAL TABLE`, `DELETE`, `INSERT` (including `ON CONFLICT`/UPSERT), `REPLACE`, `RETURNING`, `SELECT` (with `WHERE`, `LIKE`, `LIMIT`, `ORDER BY`, `GROUP BY`, `HAVING`, `JOIN`, `NATURAL JOIN`, `USING`), `UPDATE`, `EXPLAIN`, `CHECK`.

### 6.3 SQL Statements — Notable Gaps

| Feature | Status | Detail |
|---------|--------|--------|
| `WITH` (CTEs) | Partial | No `RECURSIVE`, no `MATERIALIZED`, CTEs only in SELECT |
| `WINDOW` functions | Partial | Default frame only; no `rank()`, `row_number()`, etc. |
| `CROSS JOIN` | No | SQLite CROSS JOIN means "do not reorder joins" |
| `RIGHT JOIN` | No | LEFT and FULL OUTER supported, not RIGHT |
| `VACUUM` | No | Not supported at all |
| `INDEXED BY` | No | |
| `REINDEX` | No | |
| `SAVEPOINT` | No | Listed but not implemented |
| `GENERATED` columns | No | |
| `CREATE TABLE ... STRICT` | Partial | Was experimental, promoted to full in v0.5 |
| `agg() FILTER (WHERE ...)` | No | Incorrectly ignored |
| `... OVER (...)` | No | Incorrectly ignored |
| Recursive CTEs | No | |

### 6.4 SQL Functions — Coverage

- **Scalar functions**: Nearly complete (abs, char, coalesce, format, hex, instr, length, lower/upper, substr, trim, typeof, etc.)
- **Mathematical functions**: All 30 complete (acos through trunc)
- **Aggregate functions**: All complete (avg, count, group_concat, sum, total, median, percentile)
- **Date/time functions**: All complete (date, time, datetime, julianday, unixepoch, strftime, timediff) with all modifiers
- **JSON functions**: Nearly all complete (json_extract, json_set, json_array, json_each, etc.); `json_tree` is partial

### 6.5 C API — Key Gaps

The C API is the critical interop surface. Core happy-path functions work:

**Supported:** `sqlite3_open`, `sqlite3_close`, `sqlite3_prepare_v2`, `sqlite3_step`, `sqlite3_exec`, `sqlite3_finalize`, `sqlite3_reset`, all `sqlite3_bind_*` (int, int64, double, text, blob, null), all core `sqlite3_column_*`, error handling (`errcode`, `errmsg`, `errstr`), `sqlite3_changes`, `sqlite3_last_insert_rowid`, WAL checkpoint.

**Not supported (significant):**
- **User-defined functions**: Entire `sqlite3_create_function*` / `sqlite3_result_*` family is stubbed or missing
- **Collation functions**: All `sqlite3_create_collation*` missing
- **Backup API**: Entirely stubbed
- **BLOB I/O**: Entirely stubbed
- **Virtual table C API**: All `sqlite3_create_module*`, `sqlite3_declare_vtab`, etc. missing
- **Loadable extensions**: `sqlite3_load_extension`, `sqlite3_auto_extension` missing
- **Serialization**: `sqlite3_serialize` / `sqlite3_deserialize` stubbed
- **Hooks**: `sqlite3_commit_hook`, `sqlite3_update_hook`, `sqlite3_preupdate_hook` all missing

### 6.6 Journaling Modes

Only WAL mode is supported. Rollback journal modes (delete, truncate, persist, memory) are marked "Not Needed" — Turso explicitly has no plans to support them.

### 6.7 Extensions (Beyond SQLite)

Turso ships built-in extensions not found in vanilla SQLite:

- **UUID**: v4, v7, conversion functions
- **Regexp**: sqlean-compatible (regexp_like, regexp_substr, regexp_replace, etc.)
- **Vector**: Distance metrics (cosine, L2), vector manipulation — exact search only, ANN indexing on roadmap
- **Time**: sqlean-compatible rich time manipulation
- **FTS**: Tantivy-powered full-text search (not SQLite FTS3/FTS4/FTS5)
- **CSV**: Virtual table for CSV file reading
- **Percentile**: median, percentile_cont, percentile_disc
- **Encryption at rest**: AES-256-GCM, AEGIS-256 (page-level AEAD)
- **CDC**: Change Data Capture (stable as of v0.5)
- **Built-in types**: uuid, boolean, json, jsonb, date, time, timestamp, inet, varchar (beyond SQLite's affinity system)

## 7. Rust API Surface and rusqlite Compatibility

### 7.1 Native Rust API

The `turso` crate on crates.io (currently v0.4.3) provides an **async-first** API:

```rust
// Open database
let db = Builder::new_local("my.db").build().await?;
let conn = db.connect()?;

// Execute
conn.execute("CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT)", ()).await?;
conn.execute("INSERT INTO users (name) VALUES (?1)", ["Alice"]).await?;

// Query
let mut rows = conn.query("SELECT * FROM users", ()).await?;
```

Key types: `Database` (via `Builder`), `Connection`, `Statement` (prepared), `Transaction`. All query/execute methods are `async`.

### 7.2 How This Differs from rusqlite

rusqlite is fundamentally **synchronous**. The API shape is different:

| Aspect | rusqlite | turso crate |
|--------|----------|-------------|
| I/O model | Sync (blocks on SQLite C calls) | Async (`.await`) |
| Runtime | None required | tokio |
| `Connection::open()` | Returns `Connection` | `Builder::new_local().build().await` returns `Database` |
| `conn.execute()` | Sync, returns `usize` | Async, returns `usize` via `.await` |
| `conn.prepare()` | Returns `Statement<'_>` | Returns `Statement` via `.await` |
| `stmt.query_map()` | Sync iterator | Async stream |
| User-defined functions | Supported via `create_scalar_function` | Not yet supported via C API |

**There is no drop-in rusqlite compatibility.** The Turso Rust crate is its own API. Any migration from rusqlite would require rewriting database access code to use async patterns.

### 7.3 SeaORM Integration

SeaORM's synchronous variant (`sea-orm-sync`) supports rusqlite as a backend. There is **no Turso Database backend for SeaORM** today. The async/sync impedance mismatch is the core barrier: SeaORM 2.0 moved toward synchronous iteration (replacing Stream with Iterator), while Turso's Rust API is async-first.

A future integration path would likely require either:
1. A `sea-query-turso` adapter (similar to `sea-query-rusqlite`)
2. Turso exposing a synchronous Rust API wrapper
3. SeaORM adding an async SQLite backend via the Turso crate

None of these exist today.

## 8. Testing and Reliability

Turso employs a multi-layered testing strategy to approach SQLite's legendary reliability:

### Deterministic Simulation Testing (DST)

Built into the core from day one. The `MemoryIO` backend replaces real I/O with a deterministic simulator, enabling:
- Perfect reproducibility of failures
- Simulation of years of execution with different event orderings
- Testing of rare conditions (partial writes, io_uring edge cases)
- Fuzzing with `quickcheck` and `proptest`

### Antithesis Partnership

Turso partners with Antithesis, which provides a **deterministic hypervisor** that runs many fuzzing threads in parallel, injecting faults while continuously verifying system properties. This catches bugs that even the internal simulator might miss — including bugs in the simulator itself.

### Bug Bounty

The team offers **$1,000** for data corruption bugs that their testing misses — a signal of confidence in the methodology.

## 10. Feasibility as a Migration Target for Saalis

### 10.1 Saalis Context

Saalis uses SQLite as a cold store via SeaORM (backed by rusqlite). The architecture is sync-core with an async shell (only the axum web layer uses tokio). Database access is synchronous by design, using fire-and-forget writes through a scheduler.

### 10.2 Compatibility Assessment

| Dimension | Assessment | Notes |
|-----------|------------|-------|
| **SQL dialect** | Likely sufficient | Saalis uses standard CRUD + metadata queries. Missing features (recursive CTEs, window functions, VACUUM) are unlikely blockers for a content library manager. |
| **File format** | Compatible | Turso reads/writes SQLite 3.x format. Migration is file-level compatible. |
| **SeaORM integration** | Blocker today | No SeaORM backend exists for Turso. Would need `sea-query-turso` or equivalent. |
| **Sync-core architecture** | Impedance mismatch | Turso's Rust API is async-first. Saalis's sync core would need either: (a) a sync wrapper around Turso's async API, (b) restructuring DB access to async, or (c) using Turso's C API through rusqlite-like bindings. |
| **Single-process, single-writer** | Fine | Saalis runs as a single binary. Turso's limitation of no multi-process concurrent access is not a concern. MVCC concurrent writes could benefit the scheduler's fire-and-forget pattern. |
| **Platform (Batocera/ARM)** | Risk | Turso targets Linux/macOS/Windows/WASM. Batocera on ARM should work (io_uring support in modern kernels), but is not a tested deployment target. No io_uring on older kernels means falling back to the polling backend. |
| **Production readiness** | Blocker today | Beta status. Not suitable for production workloads yet. |
| **User-defined functions** | Risk | The C API for UDFs is entirely unimplemented. If saalis needs custom SQLite functions, this is a gap. |

### 10.3 Recommendation

**Not feasible today. Worth tracking for 2027.**

The primary blockers are:
1. **No SeaORM backend** — saalis would need to either write one or abandon SeaORM for raw Turso API calls
2. **Async/sync impedance** — saalis's sync-core design conflicts with Turso's async-first Rust API
3. **Beta maturity** — no production use cases should depend on it yet

The primary *attractions* for a future migration:
- **MVCC concurrent writes** would pair well with the scheduler's fire-and-forget write pattern (no single-writer bottleneck)
- **100x tail latency improvement** in multi-tenant/multi-connection scenarios
- **Memory safety** guarantees throughout the database engine
- **Built-in FTS** (Tantivy-powered) eliminates needing a separate search index for content metadata
- **Built-in vector search** could enable future ML-based content similarity/recommendation features
- **CDC** could drive the notification system reactively
- **Pure Rust** — no C dependency chain, simpler cross-compilation for Batocera/ARM

**Revisit when:** (a) Turso reaches 1.0 / production-ready, (b) a SeaORM or sea-query backend emerges, and (c) the C API UDF surface is fleshed out. Alternatively, if saalis drops SeaORM in favor of a thinner query-builder layer, the Turso Rust crate could be adopted directly — but that is a larger architectural decision.
