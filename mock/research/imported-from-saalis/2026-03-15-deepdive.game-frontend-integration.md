## 1. EmulationStation Family

### 1.1 Original EmulationStation (Aloshi)

The original EmulationStation by Aloshi is a C++ graphical frontend for game browsing and launching. It is no longer actively maintained but established the foundational formats that all forks still use.

**Game discovery** works through `es_systems.cfg`, an XML document wrapped in `<systemList>`. Each `<system>` entry contains:

| Tag | Required | Purpose |
|-----|----------|---------|
| `<name>` | yes | Short identifier (e.g. `snes`), used in folder paths and error messages |
| `<fullname>` | no | Display name (e.g. "Super Nintendo Entertainment System") |
| `<path>` | yes | Root folder to search for ROMs; `~` expands to home; subdirectories are traversed recursively |
| `<extension>` | yes | Whitespace-delimited list of file extensions including the leading dot (`.smc .sfc .zip`) |
| `<command>` | yes | Shell command to launch a game; `%ROM%` is replaced with the shell-escaped absolute path to the selected ROM |
| `<platform>` | no | Comma/space-delimited platform identifiers for scraper accuracy |
| `<theme>` | no | Theme set to load; defaults to `<name>` if omitted |

EmulationStation scans each system's `<path>` for files matching `<extension>`. If at least one match exists, the system appears in the UI. Metadata is loaded from `gamelist.xml` files found in the ROM directory or in `~/.emulationstation/gamelists/<system>/`.

**Theming** uses an XML-based theme engine with standard views: `basic` (text list only), `detailed` (list + metadata panel with image), `video` (detailed + video playback), and `grid` (thumbnail grid). Custom views must inherit from one of these. Theme elements reference metadata fields prefixed with `md_` (e.g. `md_image`, `md_video`, `md_marquee`, `md_description`). The system is functional but limited -- you cannot add arbitrary UI elements, animations, or custom layouts beyond what the view type supports.

**Scraper integration** is built in. The internal scraper queries TheGamesDB or ScreenScraper, downloads media, and writes `gamelist.xml` entries with absolute or relative paths to the downloaded images and videos.

### 1.2 ES-DE (EmulationStation Desktop Edition)

ES-DE is a ground-up rewrite/heavy fork that targets desktop platforms (Windows, macOS, Linux) and is also used by RetroDECK. Key differences from the original:

- **`es_systems.xml`** (note: `.xml` not `.cfg`) ships bundled with the application and must not be edited directly. Custom systems go in `~/.emulationstation/custom_systems/es_systems.xml`. The format adds support for multiple `<command>` entries with `label` attributes to define alternative emulators for the same system.
- **Media matching by filename, not XML path.** ES-DE does not store media paths in `gamelist.xml`. Instead, it matches media files to ROMs based on filename. A ROM named `Chrono Trigger (USA).sfc` automatically finds `Chrono Trigger (USA).png` in the appropriate media subdirectory. This makes the setup fully portable -- no absolute paths to break when moving files.
- **Gamelist location changed in v2.0.0.** Only gamelists in `~/.emulationstation/gamelists/` are loaded by default. The legacy behavior of reading gamelist.xml from the ROM directory tree can be re-enabled via `LegacyGamelistFileLocation` in `es_settings.xml`.
- **System sorting** uses a separate `es_systems_sorting.xml` file, avoiding the need to modify the main systems config.
- **Scraper improvements.** ES-DE bundles a scraper with ScreenScraper and TheGamesDB support, writing metadata to gamelist.xml but relying on the filename-based media matching rather than embedding paths.

### 1.3 RetroBat

RetroBat is not a frontend itself but a Windows-centric software distribution that bundles and auto-configures EmulationStation with RetroArch and many standalone emulators. It uses a modified EmulationStation fork (batocera-emulationstation adapted for Windows).

Key characteristics:
- **Over 150 systems preconfigured** with appropriate emulator bindings
- **Configuration managed from the UI** -- users rarely need to touch emulator configs directly
- **Supports native Windows games** alongside emulated systems (can auto-scan Steam, GOG, Epic, EA libraries)
- **Uses the same `es_systems.cfg` / `gamelist.xml` formats** as Batocera's ES fork
- Generates emulator configurations dynamically, similar to Batocera's configgen approach but implemented for Windows

## 2. Pegasus Frontend

Pegasus is a cross-platform graphical frontend written in C++ with a QML-based theme engine. It was designed as a modern alternative to EmulationStation.

### 2.1 Metadata Format (`metadata.pegasus.txt`)

Pegasus uses a human-readable plain-text format based on the Debian control file syntax (key-value pairs, blank-line-separated blocks). The preferred filename is `metadata.pegasus.txt`, placed alongside games.

**Collection fields:**

```
collection: Super Nintendo Entertainment System
shortname: snes
extensions: 7z, bin, smc, sfc, fig, swc, mgd, zip
launch: snes9x "{file.path}"
directories: /path/to/more/roms
ignore-file: buggygame.bin
```

| Field | Purpose |
|-------|---------|
| `collection` | Display name; starts a new collection block |
| `shortname` | Identifier used by themes for per-collection assets |
| `extensions` | Comma-separated file extensions (no leading dot) |
| `launch` | Command template; `{file.path}` is replaced with the selected game's path |
| `directories` | Additional directories to scan |
| `files` | Explicit file list |
| `ignore-file` | Files to exclude from scanning |
| `ignore-regex` | Regex pattern to exclude files |
| `command` | Alias for `launch` |
| `workdir` | Working directory for launch command |

**Game fields:**

```
game: Chrono Trigger
file: Chrono Trigger (USA).sfc
developer: Square
publisher: Square
genre: RPG
players: 1
rating: 92%
release: 1995-03-11
summary: A group of adventurers travel through time...
description: (longer multi-line text)
launch: custom-emulator "{file.path}"
assets.boxFront: media/box/chrono.png
assets.screenshot: media/screens/chrono.png
assets.video: media/video/chrono.mp4
x-custom-field: anything
```

| Field | Purpose |
|-------|---------|
| `game` | Display name; starts a new game block |
| `file` | ROM/game file path (relative or absolute) |
| `developer` | Developer name |
| `publisher` | Publisher name |
| `genre` | Genre string |
| `players` | Player count |
| `rating` | Rating as percentage |
| `release` | Release date |
| `summary` | Brief description |
| `description` | Full description (multi-line) |
| `launch` | Per-game launch override |
| `workdir` | Per-game working directory |
| `assets.<type>` | Asset file path (see media section) |
| `x-*` | Custom extension fields |

### 2.2 Differences from EmulationStation

| Aspect | EmulationStation | Pegasus |
|--------|-----------------|---------|
| Config format | XML (`es_systems.cfg`, `gamelist.xml`) | Plain text (`metadata.pegasus.txt`) |
| Game discovery | Extension matching in directory tree | Extension matching + explicit file lists |
| Media paths | Embedded in gamelist.xml (original) or filename-matched (ES-DE) | `assets.*` fields in metadata or automatic directory scanning |
| Theme engine | XML-based, fixed view types (basic/detailed/video/grid) | QML-based, fully programmable UI with animations, shaders, custom layouts |
| Multi-source | Single metadata source per system | Collections can span multiple directories and multiple metadata files |
| Launch config | Single `<command>` per system (ES original) | Per-collection and per-game launch overrides |
| ES compatibility | N/A | Can read `gamelist.xml` and `es_systems.cfg` natively as alternate metadata sources |

### 2.3 QML Theming Advantages

Pegasus themes are written in QML (Qt Meta-object Language), giving theme creators full programmatic control:
- **Complete UI replacement** -- themes can add or remove any UI element, create custom menu screens, transitions, and animations
- **Shader support** -- custom GLSL shaders can be run for visual effects
- **Live reload** -- theme changes are reflected in real-time without restarting the application
- **JavaScript logic** -- QML allows inline JavaScript for complex UI behaviors
- **No fixed views** -- unlike ES's basic/detailed/video/grid constraint, Pegasus themes define the entire UI from scratch

The tradeoff is higher complexity for theme authors compared to ES's declarative XML approach.

## 3. LaunchBox / BigBox

LaunchBox is a Windows-centric game library manager and frontend. BigBox is its fullscreen "10-foot" companion interface.

### 3.1 Database Format

LaunchBox stores all data in XML files under `.\Data\`:
- **`Platforms\<PlatformName>.xml`** -- per-platform game entries
- **`LaunchBox.xml`** -- user edits and overrides
- **`Metadata.xml`** -- the entire GamesDB master database cached locally (can be very large)

A game entry in a platform XML includes fields like:

| Field | Notes |
|-------|-------|
| `<Title>` | Game name (US name preferred for multi-region titles) |
| `<ID>` / `<DatabaseID>` / `<LBDatabaseID>` | Internal and GamesDB cross-reference IDs |
| `<ReleaseDate>` | Release date |
| `<Developer>` | Developer |
| `<Publisher>` | Publisher |
| `<Genre>` | Genre string |
| `<MaxPlayers>` | Computed from max of local/online/co-op player fields |
| `<Rating>` | Content rating |
| `<CommunityStarRating>` | Community score |
| `<Notes>` | Description text |
| `<VideoUrl>` | Link to video |
| `<WikipediaURL>` | Wikipedia link |
| `<ApplicationPath>` | Path to the ROM/executable |
| `<Emulator>` | Assigned emulator |
| `<Platform>` | Platform name |
| `<Series>` | Game series |
| `<SortTitle>` | Custom sort name |
| Custom fields | User-defined key-value pairs stored in separate XML sections |

`<AdditionalApplication>` entries can define per-game extras (e.g., different disc versions, setup utilities, Steam launch shortcuts).

### 3.2 Metadata Richness

LaunchBox's GamesDB is arguably the richest metadata source in the retro gaming space. It stores data contributed by the community with editorial oversight and includes fields not found in other systems (series relationships, Wikipedia links, community ratings, multiple media types). The `Metadata.xml` cache is loaded into RAM for instant access.

### 3.3 Limitations

- **Windows-only** -- both LaunchBox and BigBox require Windows (or Wine, poorly)
- **Closed-source** -- the core application is proprietary; BigBox requires a premium license
- **XML bloat** -- the full metadata cache can grow to hundreds of megabytes
- **No standard export format** -- getting data out requires parsing their XML or using community plugins
- **Plugin API is .NET-based** -- limits integration to Windows/.NET ecosystems

## 4. RetroArch's Built-in Browser

RetroArch includes a built-in game browser (XMB, Ozone, or RGUI menu drivers) that uses playlist files for game organisation.

### 4.1 Playlist Format (`.lpl`)

Since RetroArch 1.7.5, playlists use JSON format (the old 6-line plain-text format is still supported for backwards compatibility):

```json
{
  "version": "1.5",
  "default_core_path": "/path/to/cores/snes9x_libretro.so",
  "default_core_name": "Snes9x",
  "items": [
    {
      "path": "/home/user/roms/snes/Chrono Trigger (USA).sfc",
      "label": "Chrono Trigger (USA)",
      "core_path": "DETECT",
      "core_name": "DETECT",
      "crc32": "2D5B718A|crc",
      "db_name": "Nintendo - Super Nintendo Entertainment System.lpl"
    }
  ]
}
```

| Field | Purpose |
|-------|---------|
| `path` | Absolute path to the ROM file |
| `label` | Display name |
| `core_path` | Path to the libretro core, or `"DETECT"` for auto-detection |
| `core_name` | Human-readable core name, or `"DETECT"` |
| `crc32` | CRC32 hash or serial number for database matching (format: `HASH\|crc`) |
| `db_name` | System database name for metadata lookup; matches the playlist filename |

`default_core_path` and `default_core_name` at the top level set the fallback core for entries that use `"DETECT"`.

### 4.2 Thumbnail Paths

Thumbnails follow a rigid directory convention:

```
thumbnails/
  <Playlist Name>/         # e.g. "Nintendo - Super Nintendo Entertainment System"
    Named_Boxarts/         # Box art / cover art
      <Game Label>.png
    Named_Snaps/           # In-game screenshots
      <Game Label>.png
    Named_Titles/          # Title screens
      <Game Label>.png
```

The `<Playlist Name>` directory must exactly match the playlist filename minus the `.lpl` extension. The `<Game Label>` must exactly match the `label` field in the playlist entry, with special characters (`&*/:\`<>?\|`) replaced by `_`.

### 4.3 Core Association

RetroArch matches ROMs to cores through:
1. **Explicit playlist entry** -- `core_path` in the `.lpl` item
2. **Playlist default** -- `default_core_path` at the top level
3. **Database matching** -- `db_name` links to RetroArch's internal database for automatic core recommendation
4. **Manual selection** -- user picks a core from the menu at launch time

## 5. How Batocera Integrates EmulationStation

Batocera uses a heavily modified EmulationStation fork (`batocera-emulationstation`) as its primary UI, with a Python-based configuration generation system (`configgen`) as the bridge between the UI and emulators.

### 5.1 The configgen Architecture

The launch flow is:

1. **EmulationStation** displays systems based on `/usr/share/emulationstation/es_systems.cfg`
2. The `<command>` for every system is the same generic launcher:
   ```
   python /usr/lib/python3.9/site-packages/configgen/emulatorlauncher.py \
     %CONTROLLERSCONFIG% -system %SYSTEM% -rom %ROM%
   ```
3. **`emulatorlauncher.py`** parses the arguments (system name, ROM path, controller config, plus optional flags for emulator, core, netplay settings)
4. It looks up the appropriate **generator** class for the selected system/emulator combination
5. The generator's `generate()` method is called with: system info, ROM path, player controllers, game metadata, gun/wheel devices, and target resolution
6. The generator **writes emulator config files on the fly** based on the user's settings in EmulationStation, then returns the command line to execute
7. `emulatorlauncher.py` executes the generated command

### 5.2 Per-System Generators

Generators live in `/usr/lib/python3.9/site-packages/configgen/generators/`. Each is a Python class with a `generate()` method. Examples:

- `libretro/libretroGenerator.py` -- generates `retroarch.cfg` overrides and launches RetroArch with the appropriate core
- `dolphin/dolphinGenerator.py` -- writes Dolphin config and launches it
- `ppsspp/ppssppGenerator.py` -- configures PPSSPP
- `ryujinx/ryujinxMainlineGenerator.py` -- configures Ryujinx for Switch emulation

New generators are registered in `setup.py`. The architecture is modular -- adding a new emulator means writing a new generator class and registering it.

### 5.3 Dynamic Configuration

This is Batocera's key differentiator: **emulator configuration files are regenerated on every game launch.** Users change settings in EmulationStation's menus (resolution, shaders, controls), and those settings are translated into emulator-specific config files at launch time. This means:
- Users never need to open an emulator's own settings UI
- Settings are consistent across the system
- Per-game overrides are supported through EmulationStation's game-level settings

### 5.4 Custom System Overrides

Users can override the bundled `es_systems.cfg` by placing a custom version at `/userdata/system/configs/emulationstation/es_systems.cfg`, or add new systems with `es_systems_<custom_name>.cfg` files. ROM paths must start with `/userdata/roms/`.

### 5.5 ES Web Server

Batocera embeds a web server in EmulationStation (enabled via SYSTEM SETTINGS > FRONTEND DEVELOPER OPTIONS > ENABLE PUBLIC WEB ACCESS). Once enabled at `http://<batocera-ip>:1234`, it provides:
- Game browsing and metadata viewing
- **Remote game launching** -- games play on the Batocera device as if launched from the local UI
- Game termination via a Kill button
- Game list reloading
- System information display

This web server is a C++ component embedded directly in batocera-emulationstation, not a separate service.

## 6. The gamelist.xml Format in Detail

### 6.1 Structure

```xml
<?xml version="1.0"?>
<gameList>
  <game>
    <!-- game metadata -->
  </game>
  <folder>
    <!-- folder metadata -->
  </folder>
</gameList>
```

### 6.2 All Standard Fields

| Tag | Type | Example | Notes |
|-----|------|---------|-------|
| `<path>` | string | `./Chrono Trigger (USA).sfc` | Relative or absolute path to ROM. Relative paths are relative to the system's ROM directory |
| `<name>` | string | `Chrono Trigger` | Display name |
| `<desc>` | string | `A group of adventurers...` | Description text; long descriptions auto-scroll in the UI |
| `<image>` | string | `./media/images/Chrono Trigger.png` | Path to primary image (typically screenshot or box art) |
| `<thumbnail>` | string | `./media/thumbnails/Chrono Trigger.png` | Path to thumbnail image |
| `<video>` | string | `./media/videos/Chrono Trigger.mp4` | Path to video snap |
| `<marquee>` | string | `./media/marquees/Chrono Trigger.png` | Path to marquee/wheel/logo art |
| `<rating>` | float | `0.85` | Rating from 0.0 to 1.0 |
| `<releasedate>` | datetime | `19950311T000000` | ISO-ish format: `%Y%m%dT%H%M%S` |
| `<developer>` | string | `Square` | Developer name |
| `<publisher>` | string | `Square` | Publisher name |
| `<genre>` | string | `Role-Playing` | Genre string |
| `<players>` | string | `1` | Player count (sometimes a range like `1-2`) |
| `<playcount>` | int | `5` | Number of times played |
| `<lastplayed>` | datetime | `20260101T120000` | Last played timestamp |

### 6.3 Extended Fields (Batocera / Recalbox Forks)

| Tag | Type | Notes |
|-----|------|-------|
| `<favorite>` | bool | `true`/`false` -- marks game as favorite |
| `<hidden>` | bool | `true`/`false` -- hides game from normal view |
| `<kidgame>` | bool | `true`/`false` -- marks game as kid-safe |
| `<lang>` | string | Language code(s) |
| `<region>` | string | Region code(s) |
| `<md5>` | string | MD5 hash of the ROM file |
| `<cheevosHash>` | string | RetroAchievements hash |
| `<cheevosId>` | int | RetroAchievements game ID |
| `<genreid>` | string | Numeric genre identifier |
| `<arcadesystemname>` | string | Arcade system board name |
| `<bezel>` | string | Bezel/overlay path |

### 6.4 Folder Entries

`<folder>` tags use the same fields as `<game>` but represent directories. They let you assign metadata (name, image, description) to folder nodes in the ROM directory tree.

### 6.5 How Scrapers Populate It

Scrapers (Skraper, ARRM, ES's built-in scraper, Skyscraper) query metadata services (ScreenScraper, TheGamesDB, IGDB) and:
1. Download media files (images, videos) into media subdirectories
2. Write `gamelist.xml` entries with paths pointing to the downloaded media
3. Map service-specific fields to the gamelist.xml tags
4. Preserve unknown tags (e.g., `cheevosHash`) when rewriting the file

### 6.6 Limitations

- **No schema validation** -- malformed XML is common and causes silent failures
- **Path brittleness** -- absolute media paths break when ROMs are moved (ES-DE solved this with filename matching)
- **No relational data** -- no concept of series, sequels, compilations, or game-to-game relationships
- **Single-system scope** -- each gamelist.xml is per-system; cross-system collections are handled separately (favorites, custom collections) and stored in different files
- **No standardisation across forks** -- each fork (Batocera, RetroPie, Recalbox, ES-DE) adds its own extended tags, creating a fragmented ecosystem
- **No scraper source attribution** -- no field tracks where metadata came from or its freshness
- **Media paths are redundant** in ES-DE's model, which ignores them entirely

## 7. Integration Strategies for Saalis

### 7.1 Strategy A: Replace ES Entirely (Serve Web UI)

Saalis acts as the sole frontend, serving a web UI that handles browsing, metadata, and game launching.

**Approach:**
- Saalis runs on the Batocera device as a service
- The web UI (served by the `saalis-web` crate via axum) replaces EmulationStation
- Game launching happens via subprocess calls from the saalis backend (calling emulators directly or through Batocera's configgen)
- Batocera boots directly into a browser (kiosk mode) pointed at saalis's local web UI, or saalis provides its own framebuffer/Wayland UI

**Advantages:**
- Complete control over UX, metadata model, and library management
- No impedance mismatch between saalis's entity model and gamelist.xml
- Single source of truth for all data
- Can provide features ES cannot: smart collections, cross-system search, rich metadata, download management

**Disadvantages:**
- Loses the ES theme ecosystem (hundreds of community themes)
- Must handle controller input in a browser context (possible but non-trivial)
- Must implement all UI/UX from scratch
- Breaking change for existing Batocera users who know ES

**When to choose:** If saalis is the primary product and the Batocera integration is a platform detail, not a feature.

### 7.2 Strategy B: Run Alongside (Sync Metadata to gamelist.xml)

Saalis runs as a background service; EmulationStation remains the primary UI. Saalis syncs its richer metadata model down to `gamelist.xml` files.

**Approach:**
- Saalis manages the library, downloads, scraping, and enrichment
- A sync daemon periodically (or on-demand) generates/updates `gamelist.xml` for each system
- Media files are placed in ES-compatible paths
- ES displays the curated library; users interact with ES as usual

**Advantages:**
- Preserves the full ES theme ecosystem
- Non-disruptive to existing Batocera users
- Saalis handles the hard parts (scraping, deduplication, organisation) while ES handles presentation
- Graceful degradation: if saalis is down, ES still works with the last-synced data

**Disadvantages:**
- Lossy sync: gamelist.xml cannot represent saalis's full metadata model (no series, no entity relationships, no custom fields beyond what ES forks support)
- Two sources of truth: users might edit metadata in ES (e.g., marking favorites) and those changes need to be synced back
- Must handle gamelist.xml format differences across ES forks

**When to choose:** If broad compatibility with the existing Batocera/ES ecosystem matters more than full UX control.

### 7.3 Strategy C: Export Mode (Generate ES-Compatible Files)

Similar to Strategy B but one-directional -- saalis generates ES-compatible files on command, with no continuous sync.

**Approach:**
- User manages library in saalis (web UI, CLI, or API)
- An `export` command generates `es_systems.cfg`, `gamelist.xml` per system, and copies/symlinks media into ES-expected paths
- Also supports exporting to Pegasus `metadata.pegasus.txt` and RetroArch `.lpl` playlists
- ES/Pegasus/RetroArch is then pointed at the exported directory

**Advantages:**
- Clean separation of concerns
- Supports multiple frontends from a single library
- No runtime dependency between saalis and the frontend
- Multi-format export is a natural fit for saalis's content-type-agnostic model

**Disadvantages:**
- No live updates -- user must re-export after library changes
- No bidirectional sync (favorites, play counts, last-played timestamps from ES are not captured)
- User must manually trigger exports

**When to choose:** If saalis is primarily a library management tool and the presentation layer is explicitly deferred to third-party frontends.

### 7.4 Strategy D: Hybrid (Saalis Manages Library, ES Launches Games)

Saalis owns the library and metadata; ES is used purely as a launch/display layer, with tight integration via APIs and file watches.

**Approach:**
- Saalis generates and maintains `gamelist.xml` files in real-time (file-watch or inotify triggers ES reload)
- Saalis provides a web UI for library management, enrichment, and settings
- ES provides the couch/TV experience with controller input and theme support
- Batocera's ES web server API or custom IPC (Unix socket, D-Bus, or simple HTTP) enables saalis to trigger game launches and receive play-session events from ES
- Play statistics (last played, play count, favorites) flow back from ES's gamelist.xml changes to saalis

**Advantages:**
- Best of both worlds: rich library management + polished TV UI
- Leverages ES theme ecosystem for the "lean back" experience
- Web UI available for "lean forward" management
- Can progressively replace ES components over time

**Disadvantages:**
- Most complex integration
- Must handle bidirectional sync carefully to avoid conflicts
- Dependent on ES fork's specific API/behavior (Batocera's ES web server, gamelist.xml location, reload mechanisms)

**When to choose:** If the goal is a premium appliance experience where users interact with ES on the TV but manage their library from a phone/laptop browser.

### 7.5 Recommended Approach for Saalis

Given saalis's design as a Batocera appliance with a web UI:

**Start with Strategy C (export), evolve toward Strategy D (hybrid).**

1. **Phase 1:** Implement multi-format export (gamelist.xml, metadata.pegasus.txt, RetroArch .lpl). This is low-risk, immediately useful, and validates the metadata model.
2. **Phase 2:** Add live gamelist.xml generation (file watch / inotify integration). ES picks up changes automatically or via its reload mechanism.
3. **Phase 3:** Integrate with Batocera's ES web server for remote launch from saalis's web UI, and capture play-session data back from ES.
4. **Phase 4:** Optionally offer a full web-UI-only mode for users who want to bypass ES entirely.

## 8. Launching Emulators from a Web UI

### 8.1 RetroArch CLI

The fundamental command is:

```bash
retroarch -L /path/to/cores/<core>_libretro.so /path/to/rom
```

Key flags:
- `-L PATH` / `--libretro PATH` -- path to the libretro core shared object
- `--config PATH` -- override config file
- `--appendconfig PATH` -- append additional config (e.g., per-game overrides)
- `--subsystem NAME` -- use a core subsystem (for multi-disc, BIOS-dependent systems)
- `--menu` -- start directly in the RetroArch menu

### 8.2 Per-Emulator Command Lines

Standalone emulators each have their own CLI conventions:

| Emulator | Example Command |
|----------|----------------|
| RetroArch | `retroarch -L /cores/snes9x_libretro.so game.sfc` |
| Dolphin | `dolphin-emu -e /path/to/game.iso` |
| PPSSPP | `ppsspp /path/to/game.iso` |
| PCSX2 | `pcsx2 /path/to/game.iso` |
| Cemu | `cemu -g /path/to/game.rpx` |
| DuckStation | `duckstation-qt -fullscreen /path/to/game.cue` |
| Ryujinx | `ryujinx /path/to/game.nsp` |

### 8.3 The Batocera configgen Approach

On Batocera, saalis should not call emulators directly. Instead, invoke the configgen launcher:

```bash
python /usr/lib/python3.9/site-packages/configgen/emulatorlauncher.py \
  -system snes -rom "/userdata/roms/snes/Chrono Trigger.sfc" \
  -emulator libretro -core snes9x
```

This ensures:
- Controller mappings are applied correctly
- Per-game and per-system settings from ES are respected
- Shaders, overlays, and bezels are configured
- The emulator config is regenerated with current settings

### 8.4 Web UI Launch Flow

For saalis's web UI to launch games on the Batocera device:

**Option A: Batocera ES Web Server**
- Use the existing ES web server API at `http://localhost:1234`
- The API supports launching games as if selected from the ES UI
- Games play on the device; the web UI shows a "now playing" state
- Kill button available to stop the game

**Option B: Direct Subprocess from saalis Backend**
- The saalis Rust backend spawns the configgen command as a child process
- stdout/stderr are captured for error reporting
- Process lifecycle is managed (timeout, graceful shutdown via signals)
- No dependency on ES running or its web server being enabled

**Option C: D-Bus / IPC**
- If ES is running, communicate via D-Bus or a Unix socket to request game launches
- More robust than HTTP for local IPC
- Requires ES to expose such an interface (Batocera's fork has limited support)

For Batocera integration, **Option A is simplest** if ES is running, and **Option B is necessary** if saalis replaces ES or runs independently.

### 9.2 Directory Conventions

**EmulationStation (Batocera/RetroPie):**
```
/userdata/roms/<system>/
  gamelist.xml
  media/
    images/        # primary image (box art or screenshot)
    thumbnails/    # secondary image
    videos/        # video snaps (.mp4)
    marquees/      # wheel/logo art
```
Paths in gamelist.xml are typically relative: `./media/images/GameName.png`

**ES-DE:**
```
~/.emulationstation/
  gamelists/<system>/gamelist.xml
  downloaded_media/<system>/
    screenshots/
    covers/
    marquees/
    videos/
    titlescreens/
    fanart/
    backcovers/
    3dboxes/
    physicalmedia/
    miximages/     # composite images generated by the scraper
```
No paths in gamelist.xml -- files are matched by name to the ROM filename.

**Pegasus:**
```
<game-directory>/
  metadata.pegasus.txt
  media/
    box2dfront/    # or boxFront via metadata
    screenshot/
    wheel/
    videos/
```
Pegasus also supports the Skraper asset provider, which looks in `./media/` subdirectories automatically.

**RetroArch:**
```
thumbnails/
  <Playlist Name>/
    Named_Boxarts/<Label>.png
    Named_Snaps/<Label>.png
    Named_Titles/<Label>.png
```
Only PNG is supported. Special characters in labels are replaced with `_`.

### 9.3 Implications for Saalis

Saalis should store media in its own canonical structure (content-addressable or entity-ID-based) and generate symlinks or copies into frontend-specific directory layouts during export. Key considerations:

- **Filename normalisation** is critical: each frontend has different rules for matching filenames to games (ES-DE strips extensions, RetroArch replaces special characters, Pegasus uses explicit paths)
- **Format support varies:** RetroArch requires PNG thumbnails only; ES and Pegasus support PNG and JPG for images, MP4 for video
- **Composite images** (miximages in ES-DE, mixing box art + screenshot + logo into one image) are generated by scrapers; saalis could generate these as part of its export pipeline
- **Video snaps** are the largest media files; consider symlinking rather than copying during export
- **Multiple media per game** is common (box front, box back, screenshot, title screen, video, manual); saalis's entity model should support an arbitrary number of named media assets per entity
- **Media deduplication** matters: the same game across multiple systems (e.g., a multi-platform release) might share the same box art; content-addressable storage handles this naturally
