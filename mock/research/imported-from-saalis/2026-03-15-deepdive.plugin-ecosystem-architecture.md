# Deep Dive: Plugin Ecosystem Architecture — Obsidian and VSCode

**Date:** 2026-03-15
**Purpose:** Technical research into the two most successful consumer-software plugin ecosystems — Obsidian (personal knowledge management) and VSCode (code editing) — to extract concrete architectural lessons for saalis's `.saalix` extension system.

---

## 1. Obsidian Plugin Ecosystem

### 1.1 Scale and Growth

Obsidian's community plugin directory has surpassed **2,000 plugins** with aggregate downloads exceeding **100 million** ([Plugin Milestones | Obsidian Stats](https://www.moritzjung.dev/obsidian-stats/pluginstats/milestones/), [Obsidian Stats](https://www.obsidianstats.com)). The community forums and Discord server host over 60,000 active members ([Obsidian in 2026: Usage, Revenue, Valuation & Growth Statistics](https://fueler.io/blog/obsidian-usage-revenue-valuation-growth-statistics)). Churn rates sit below 10% annually — exceptional for subscription SaaS — driven largely by the plugin ecosystem and the local-file ownership model. The plugin diversity is remarkable: task management, calendar integration, AI-powered writing, diagram editing, music notation, flashcard generation, Kanban boards, and full database query engines ([Obsidian Plugins](https://obsidian.md/plugins)).

What makes this especially notable is that Obsidian's user base is **not primarily developers**. It is a knowledge-centric community — writers, researchers, students, academics. The plugin ecosystem grew despite this, because the contribution barrier was kept low and the reward loop (immediate utility, visible download counts, community recognition) was kept tight. An academic paper studying the Obsidian plugin ecosystem specifically notes this anomaly: a non-developer-centric platform sustaining a massive plugin ecosystem through community engagement and accessible tooling ([Not Only for Developers: Exploring Plugin Maintenance for Knowledge-Centric Communities](https://arxiv.org/html/2602.17018)).

### 1.2 Plugin Architecture and API Surface

Every Obsidian plugin is a TypeScript/JavaScript class extending `Plugin`, which itself extends `Component`. The `Component` base class provides lifecycle management — any resource registered during `onload()` is automatically cleaned up during `onunload()`. This declarative registration pattern eliminates an entire class of resource-leak bugs ([Plugin Development | obsidianmd/obsidian-api | DeepWiki](https://deepwiki.com/obsidianmd/obsidian-api/3-plugin-development), [Plugin - Developer Documentation](https://docs.obsidian.md/Reference/TypeScript+API/Plugin)).

The plugin instance receives `this.app`, the root access point to all Obsidian subsystems:

- **Vault** — read, create, modify, delete, and rename files and folders. Listen for create/modify/delete/rename events. Access file metadata (size, modification time, extension) ([Vault - Developer Documentation](https://docs.obsidian.md/Reference/TypeScript+API/Vault)).
- **Workspace** — manipulate panes (leaves), tabs, sidebars, and split views. Register custom view types via `this.registerView()` with an `ItemView` subclass. Activate views by setting view state on workspace leaves ([Views - Developer Documentation](https://docs.obsidian.md/Plugins/User+interface/Views)).
- **MetadataCache** — access parsed frontmatter (YAML), headings, links, embeds, tags, and block references for every markdown file. This is the indexed, structured representation of vault content.
- **Commands** — register global commands via `this.addCommand()` with an ID, display name, optional hotkey, and callback. Commands appear in the command palette and are available for user-defined hotkey binding.
- **Settings** — register a settings tab via `this.addSettingTab()` using a `PluginSettingTab` subclass. Persist plugin data with `this.loadData()` / `this.saveData()`, which serializes to `.obsidian/plugins/{id}/data.json`.
- **Editor Extensions** — Obsidian's editor is CodeMirror 6 (CM6). Plugins register CM6 extensions via `registerEditorExtension()`, gaining access to view plugins (access to the editor viewport) and state fields (custom editor state). This allows syntax highlighting, custom decorations, inline widgets, and editor behaviour modifications ([Editor extensions | Obsidian Plugin Developer Docs](https://marcusolsson.github.io/obsidian-plugin-docs/editor/extensions)).
- **Ribbon and Status Bar** — add icons to the left sidebar ribbon and items to the bottom status bar.
- **Events** — a publish-subscribe system where classes extending `Events` emit typed events. Handlers are registered via `registerEvent()` which returns an `EventRef` for automatic cleanup. `registerInterval()` wraps `setInterval` with automatic cleanup on unload.

### 1.3 How Plugins Are Loaded

Obsidian plugins are **not compiled** — they are bundled JavaScript. The development workflow is TypeScript source compiled (via esbuild, Rollup, or Webpack) to a single `main.js` file. At runtime, Obsidian loads this file using JavaScript's `eval()` function within the Electron renderer process. The `obsidian` package (the API) is excluded from the bundle — Obsidian provides it at runtime via `require('obsidian')` ([obsidian - npm](https://www.npmjs.com/package/obsidian), [How to debug Obsidian plugins](https://mnaoumov.wordpress.com/2022/05/10/how-to-debug-obsidian-plugins/)).

Each plugin directory under `.obsidian/plugins/{plugin-id}/` contains three files: `manifest.json` (metadata: id, name, version, minAppVersion, description, author), `main.js` (the bundled code), and optionally `styles.css` (custom CSS injected into the app).

This loading model has critical implications:

1. **Full Node.js access.** Because plugins run in Electron's renderer with Node integration enabled, they can `require('fs')`, `require('child_process')`, `require('electron')`, or any Node.js module. A plugin has the same filesystem and process access as the Obsidian application itself.
2. **No compilation barrier.** Plugin authors do not need to target a specific binary ABI. Any JavaScript that exports a class extending `Plugin` will load. This eliminates the most significant friction point in compiled-code plugin systems.
3. **Hot reload during development.** Obsidian can reload plugins without restarting the application — `eval()` the new `main.js`, call the old instance's `onunload()`, call the new instance's `onload()`.

### 1.4 Sandboxing Model (or Lack Thereof)

Obsidian has **no sandbox**. The official documentation states plainly: "Due to technical limitations, Obsidian cannot reliably restrict what plugins can do" ([Plugin security - Obsidian Help](https://help.obsidian.md/plugin-security)). Community plugins have unrestricted access not just to vault files but to any file on the user's machine, to the network, and to child processes.

The security model is instead built on **layers of social trust**:

1. **Restricted Mode.** On first launch, Obsidian runs in Restricted Mode with no community plugins enabled. The user must explicitly opt out of Restricted Mode to install any community plugin. This is the consent boundary — the user acknowledges the risk.
2. **Open-source requirement.** All community plugins must be open-source and hosted on GitHub. Source code is inspectable by anyone before installation.
3. **Initial code review.** When a plugin is first submitted to the community directory, it undergoes manual review by the Obsidian team against developer policies. This review checks for obvious malice, adherence to API guidelines, and basic quality standards ([Submission requirements for plugins - Developer Documentation](https://docs.obsidian.md/Plugins/Releasing/Submission+requirements+for+plugins), [Developer policies - Developer Documentation](https://docs.obsidian.md/Developer+policies)).
4. **No per-update review.** Subsequent updates are published directly via GitHub releases without re-review. The Obsidian team is too small to review every update to every plugin. They rely on community reporting to catch problems after the fact.
5. **Community vigilance.** Popular plugins with many users receive de facto security validation through usage. Issues are reported on GitHub and the Obsidian forum.

This model has drawn criticism. A 2025 analysis noted that the supply chain risk is real — a compromised plugin author account could push a malicious update to thousands of users without any review gate ([Obsidian's Plugin Security Model Faces Community Criticism](https://biggo.com/news/202509200713_Obsidian_Plugin_Security_Concerns)). Community members have proposed sandboxing approaches that would grant vault access while blocking network and disk access, but these have not been implemented.

### 1.5 Community Review and Submission Process

To list a plugin in the community directory, a developer submits a pull request to the [`obsidianmd/obsidian-releases`](https://github.com/obsidianmd/obsidian-releases) repository, adding an entry to `community-plugins.json`. The entry contains the plugin ID (matching `manifest.json`) and the GitHub repository path (`user-name/repo-name`). An automated `validate-plugin-entry.yml` workflow runs checks against the submission. The PR template includes a checklist: attestation of intent to maintain, mobile compatibility declaration, correct release artifact structure ([Plugin Submission Guide | DeepWiki](https://deepwiki.com/obsidianmd/obsidian-releases/6.1-plugin-submission-guide), [Submit your plugin - Developer Documentation](https://docs.obsidian.md/Plugins/Releasing/Submit+your+plugin)).

After approval, the plugin appears in the in-app community plugin browser. Users search by keyword, read descriptions, see download counts and last-update dates, and install with one click.

### 1.6 Update Mechanism

When a plugin author creates a GitHub release with a tag matching the version in `manifest.json`, Obsidian's update mechanism picks it up. The release must contain the individual files (`manifest.json`, `main.js`, optionally `styles.css`) — not just source archives. Draft releases and pre-releases are ignored. Users receive updates automatically without manual review of each version ([Releasing Your Plugin | DeepWiki](https://deepwiki.com/obsidianmd/obsidian-developer-docs/2.8-releasing-your-plugin), [Automatic plugin updates - Obsidian Forum](https://forum.obsidian.md/t/automatic-plugin-updates/13566)).

For beta testing, the community-built BRAT (Beta Reviewer's Auto-update Tool) plugin allows installing plugins from GitHub repositories that are not yet in the community directory ([obsidian42-brat](https://github.com/TfTHacker/obsidian42-brat)).

### 1.7 Inter-Plugin Communication

Obsidian provides no formal inter-plugin API. Two community-established patterns exist: (1) exporting an API object to the global namespace, and (2) accessing another plugin's instance via `plugin.app.plugins.plugins["other-plugin-id"]?.api`. Both are informal — there is no type safety, no dependency declaration, and no guaranteed load order ([Inter-plugin Communication - Obsidian Forum](https://forum.obsidian.md/t/inter-plugin-communication-expose-api-to-other-plugins/23618), [How to create plugin APIs for other plugins - Obsidian Forum](https://forum.obsidian.md/t/how-to-create-plugin-apis-for-other-plugins-to-use-plugin-to-plugin-communication/92296)).

### 1.8 Dataview: A Case Study in Emergent Capability

Dataview ([GitHub](https://github.com/blacksmithgu/obsidian-dataview)) is the definitive example of a plugin that transformed its host application's identity. Released in 2021, it has accumulated nearly **3 million downloads** and is consistently the most-installed community plugin ([Dataview - Obsidian Stats](https://www.obsidianstats.com/plugins/dataview)).

Dataview treats the vault as a queryable database. It indexes frontmatter fields, inline `[key:: value]` annotations, tags, links, and task checkboxes. Users write queries in either the Dataview Query Language (DQL) — a SQL-like syntax — or in full JavaScript via DataviewJS. Queries are embedded as code blocks in markdown and render as live tables, lists, task views, or calendars ([Dataview Documentation](https://blacksmithgu.github.io/obsidian-dataview/)).

The security split is instructive: DQL queries are sandboxed — they can read but not modify vault data. DataviewJS queries run with full plugin-level access and can create, modify, or delete files and make network calls. This two-tier model (safe declarative queries vs. powerful imperative scripts) maps directly to saalis's planned split between declarative connector definitions and full Rust plugin implementations.

Dataview fundamentally changed how people use Obsidian. Users started designing their vault structure around Dataview queries — adding frontmatter fields specifically to enable filtering and aggregation. The plugin became load-bearing infrastructure for workflows it was never designed to support: project management, CRM, habit tracking, book cataloguing, Dungeons & Dragons campaign management ([Real Use Case for Obsidian with Dataview](https://nicolevanderhoeven.com/blog/20221007-real-use-case-for-obsidian-with-dataview/), [How I use Obsidian Dataview](https://cassidoo.co/post/obsidian-dataview/)).

The lesson: **a plugin that lets users query and reshape the host's data model in ways the host team never anticipated will generate more ecosystem value than any number of feature-specific plugins**. Dataview's success prompted Obsidian to build "Bases," a native database feature, effectively absorbing the plugin's core capability into the platform ([Obsidian Dataview Is Dead. Long Live Bases.](https://medium.com/@lennart.dde/obsidian-dataview-is-dead-long-live-bases-9750e8a92877)).

### 1.9 What Drove the Explosion

Synthesising across all the evidence, the factors that drove Obsidian's plugin ecosystem from zero to 2,000+ plugins:

1. **JavaScript entry point.** The largest possible developer pool. No compilation, no ABI concerns, no toolchain installation beyond Node.js.
2. **Fast iteration loop.** Edit TypeScript, build, hot-reload in Obsidian. Seconds, not minutes.
3. **Rich API surface.** Vault, workspace, commands, settings, editor extensions, events — plugins can touch almost everything.
4. **Native discovery.** The in-app plugin browser eliminates distribution friction.
5. **GitHub-based workflow.** Plugin authors already know Git. The submission process (PR to a JSON file) is familiar.
6. **Visible metrics.** Download counts and community feedback create a reward loop for authors.
7. **No gatekeeping after initial review.** Updates ship immediately via GitHub releases.
8. **Community culture.** The Obsidian community actively celebrates plugin authors, shares workflows built on plugins, and reports bugs constructively.

## 2. VSCode Extension Ecosystem

### 2.1 Scale

The Visual Studio Code Marketplace hosts over **50,000 extensions** serving tens of millions of users. Individual extensions like GitLens exceed 40 million downloads. The marketplace processes over 3 million extension downloads monthly ([Visual Studio Marketplace](https://marketplace.visualstudio.com/vscode)). VSCode's dominance as a code editor is inseparable from its extension ecosystem — it is the primary reason developers choose it over alternatives.

### 2.2 Extension Host Process Model

VSCode's most consequential architectural decision is **process isolation**. Extensions do not run in the renderer (UI) process. Instead, VSCode spawns a dedicated **Extension Host** — a separate Node.js process — that executes all activated extensions. Communication between the renderer and the Extension Host uses a bidirectional RPC protocol with typed interfaces: `MainThread*Shape` (callable from Extension Host to renderer) and `ExtHost*Shape` (callable from renderer to Extension Host). All interfaces are defined in `src/vs/workbench/api/common/extHost.protocol.ts` ([Extension Host | Visual Studio Code Extension API](https://code.visualstudio.com/api/advanced-topics/extension-host), [VS Code Architecture Overview](https://skywork.ai/skypage/en/VS-Code-Architecture-Overview/1977611814760935424), [Extension System | DeepWiki](https://deepwiki.com/microsoft/vscode/3-product-configuration-and-policy)).

This architecture provides several guarantees:

- **UI responsiveness.** A misbehaving extension cannot freeze the editor. The user can always type, save, and navigate regardless of what extensions are doing.
- **Crash isolation.** If the Extension Host process crashes, VSCode can restart it without losing the user's work. Extensions restart with their saved state.
- **Controlled API surface.** The Extension Host does not have direct DOM access. It interacts with the UI exclusively through the RPC protocol, which exposes only the `vscode` namespace API. Extensions cannot inject arbitrary HTML, modify CSS, or access the editor's internal DOM structure ([Extension API guidelines - GitHub Wiki](https://github.com/microsoft/vscode/wiki/Extension-API-guidelines), [Extension Capabilities Overview](https://code.visualstudio.com/api/extension-capabilities/overview)).

A type conversion layer mediates between the Extension Host and renderer. Complex objects are serialized for IPC — the API defines "public" types (`vscode.Uri`, `vscode.Position`, `vscode.Range`) distinct from internal "private" types (`URI`, `IPosition`). This boundary prevents extensions from depending on internal implementation details ([Extension Development | DeepWiki](https://deepwiki.com/microsoft/vscode-wiki/5-extension-development)).

Each extension receives its own scoped `vscode` API object via a closure that is called with the extension's `IExtensionDescription` during activation. This means different extensions get different API instances — enabling per-extension capability tracking and eventual per-extension permission scoping.

### 2.3 Activation Events and Performance Budgets

VSCode extensions declare **activation events** in their `package.json` manifest — conditions under which the extension should be loaded. This enables lazy loading: extensions that are not relevant to the current session are never loaded, consuming zero memory and zero startup time ([Activation Events | Visual Studio Code Extension API](https://code.visualstudio.com/api/references/activation-events)).

Common activation events include:

- `onLanguage:python` — activate when a Python file is opened.
- `onCommand:myExtension.doSomething` — activate when a specific command is invoked.
- `onView:myExtension.treeView` — activate when a custom view becomes visible.
- `onFileSystem:sftp` — activate when a file with a specific URI scheme is accessed.
- `workspaceContains:**/package.json` — activate when the workspace contains a matching file.
- `*` — activate on startup. **Strongly discouraged** as it competes with other extensions and VSCode itself during startup.

The performance impact is measurable: bundling the Azure Account extension reduced activation time by 50% and extension size from 6.2MB to 840KB. Lazy loading techniques have shown 20-30x speed improvements on warm and cold boots ([How to Speed Up Your VS Code Extension](https://dev.to/sneezry/how-to-speed-up-your-vs-code-extension-not-only-webpack-48b5), [Speeding up VSCode extensions in 2022](https://jason-williams.co.uk/posts/speeding-up-vscode-extensions-in-2022/)). VSCode's built-in "Developer: Startup Performance" command lets users identify slow-activating extensions.

The key insight: **activation events are a declarative performance contract**. The extension author states when their code is needed, and the host guarantees it will not be loaded before that condition is met. This is fundamentally different from Obsidian's model where all enabled plugins load on startup.

### 2.4 Contribution Points and Manifest-Driven Integration

VSCode extensions declare their UI contributions statically in `package.json` via **contribution points** — JSON declarations that the host can parse without executing any extension code. This allows VSCode to render command palette entries, menu items, keybindings, configuration schemas, tree views, custom editors, and language grammars before the extension has activated ([Contribution Points | Visual Studio Code Extension API](https://code.visualstudio.com/api/references/contribution-points), [Extension Anatomy | Visual Studio Code Extension API](https://code.visualstudio.com/api/get-started/extension-anatomy)).

Available contribution points include: `commands`, `menus`, `keybindings`, `configuration`, `languages`, `debuggers`, `grammars`, `themes`, `snippets`, `views`, `viewsContainers`, `problemMatchers`, `taskDefinitions`, `colors`, `customEditors`, and more.

This manifest-driven approach is architecturally significant: **the host knows what an extension provides without loading it**. A theme extension's colours are available from the manifest. A language extension's file associations are available from the manifest. Command entries appear in the palette from the manifest. Only when the user actually invokes a command does the extension activate.

This maps directly to saalis's `repr(C)` descriptor pattern, where the host reads static descriptor data to enumerate an extension's capabilities before calling any plugin code.

### 2.5 The Webview API

When extensions need custom UI beyond VSCode's built-in components (tree views, quick picks, input boxes), they use the **Webview API**. Webviews are sandboxed iframe-like containers that render HTML/CSS/JavaScript within the editor. Three patterns exist ([Webview API | Visual Studio Code Extension API](https://code.visualstudio.com/api/extension-guides/webview)):

1. **Webview Panels** — shown as editor tabs. Used for custom visualizations, dashboards, documentation viewers.
2. **Custom Editors** — replace the default editor for specific file types. Hook into editor events (undo, redo, save).
3. **Webview Views** — rendered in the sidebar or bottom panel.

Webviews communicate with the extension via a message-passing API (`postMessage` / `onDidReceiveMessage`). They run in a separate, sandboxed context with no direct access to the extension's Node.js runtime or the `vscode` API. The Webview UI Toolkit provides a component library (web components, framework-agnostic) for consistent styling ([Webview UI Toolkit](https://github.com/microsoft/vscode-webview-ui-toolkit)).

### 2.6 Language Server Protocol (LSP)

The Language Server Protocol is VSCode's most influential extension pattern, and it transcends VSCode entirely. LSP defines a standardised JSON-RPC protocol between a **language client** (the editor extension) and a **language server** (the analysis tool). The server runs in its own process and can be written in any language. The client is a thin VSCode extension that manages the server lifecycle and translates between LSP messages and the `vscode` API ([Language Server Extension Guide | Visual Studio Code Extension API](https://code.visualstudio.com/api/language-extensions/language-server-extension-guide), [Language Server Protocol Overview - Microsoft Learn](https://learn.microsoft.com/en-us/visualstudio/extensibility/language-server-protocol?view=visualstudio)).

Benefits:
- **Performance isolation.** Expensive analysis (type checking, linting, indexing) runs in a separate process that cannot block the UI.
- **Language agnosticism.** Servers written in Rust, Go, C++, or Python work with any LSP-compatible editor.
- **Reusability.** A single language server works across VSCode, Neovim, Emacs, Sublime Text, and any other LSP client.
- **Standardised capabilities.** Completions, hover information, diagnostics, go-to-definition, find-all-references, rename — all standardised at the protocol level.

LSP's design insight — **describe interactions at the level of the editor (text positions, document URIs) rather than the language model (AST nodes, type systems)** — is what made standardisation tractable. The protocol is simpler than it would be if it tried to model programming language semantics.

### 2.7 Marketplace Infrastructure and Trust Model

The Visual Studio Marketplace provides discovery, distribution, and trust signals ([Extension Marketplace](https://code.visualstudio.com/docs/configure/extensions/extension-marketplace), [Security and Trust in Visual Studio Marketplace](https://developer.microsoft.com/blog/security-and-trust-in-visual-studio-marketplace), [Publishing Extensions | Visual Studio Code Extension API](https://code.visualstudio.com/api/working-with-extensions/publishing-extension)):

**VSIX packaging.** Extensions are distributed as `.vsix` files — ZIP archives following the Open Packaging Conventions (OPC) standard. They contain `package.json` (the extension manifest), compiled JavaScript, assets, and an `extension.vsixmanifest` XML file with marketplace-specific metadata. The `vsce` CLI tool handles packaging and publishing. Platform-specific VSIX packages are supported for extensions with native dependencies ([File Formats and Specifications | DeepWiki](https://deepwiki.com/microsoft/vscode-vsce/7.2-file-formats-and-specifications)).

**Signature verification.** The Marketplace signs all extensions at publish time. VSCode verifies the signature at install time, ensuring package integrity and provenance.

**Secret scanning.** Every newly published extension is automatically scanned for secrets (API keys, credentials). Publishing is blocked if secrets are detected.

**Verified publishers.** A blue checkmark badge indicates the publisher has: (a) proven domain-name ownership, and (b) maintained at least one extension in good standing for six months or more. This is a trust signal, not a security guarantee — it means "this publisher is who they claim to be" ([Publishing Extensions | DeepWiki](https://deepwiki.com/microsoft/vscode-docs/4.3-publishing-extensions)).

**Workspace Trust.** Extensions declare their compatibility with untrusted workspaces via `untrustedWorkspaces` in `package.json`: `true` (fully functional in Restricted Mode), `false` (disabled in Restricted Mode), or `limited` (some features disabled). This lets users open untrusted projects with reduced extension capabilities ([Workspace Trust Extension Guide | Visual Studio Code Extension API](https://code.visualstudio.com/api/extension-guides/workspace-trust)).

**Runtime sandboxing status.** As of 2026, VSCode extensions do **not** run in a sandbox despite the process isolation. They have full Node.js access within the Extension Host process, similar to Obsidian. A 2025 research paper proposed automated runtime sandboxing with least-privilege policies, but native sandbox enforcement remains an open development area in VSCode itself ([Extension runtime security](https://code.visualstudio.com/docs/configure/extensions/extension-runtime-security), [Automated Security Framework for VS Code Extensions](https://nhsjs.com/2025/automated-security-framework-for-vs-code-extensions-risk-profiling-policy-generation-and-runtime-sandboxing/), [Run extensions inside a sandbox - Issue #59756](https://github.com/microsoft/vscode/issues/59756)).

**Auto-update.** Extensions update automatically to the highest available stable version. Pre-release channels are supported — publishing a stable release with a higher version number automatically migrates pre-release users to stable.

## 3. Comparative Analysis

| Dimension | Obsidian | VSCode |
|-----------|----------|--------|
| **Scale** | 2,000+ plugins, 100M+ total downloads | 50,000+ extensions, millions of users |
| **Language** | JavaScript/TypeScript (bundled to `main.js`) | JavaScript/TypeScript (bundled to VSIX) |
| **Loading** | `eval()` in Electron renderer process | Separate Extension Host Node.js process |
| **Process isolation** | None — plugins share the renderer | Yes — Extension Host is a separate process |
| **Runtime sandbox** | None | None (despite process isolation) |
| **UI access** | Full DOM access | No DOM access; Webview API for custom UI |
| **Lazy loading** | No — all enabled plugins load at startup | Yes — activation events control when extensions load |
| **Manifest-driven discovery** | Minimal (`manifest.json` has metadata only) | Extensive (contribution points declare commands, menus, views, etc.) |
| **API surface** | Wide and somewhat informal (Vault, Workspace, MetadataCache, etc.) | Wide but strictly controlled (`vscode` namespace, versioned, no internals) |
| **Distribution** | GitHub releases (3 files per plugin) | VSIX packages via Marketplace (signed) |
| **Trust model** | Open-source + initial code review + community vigilance | Verified publishers + signature verification + secret scanning |
| **Update review** | No per-update review | No per-update review (but signed) |
| **Inter-plugin communication** | Informal (global namespace or `app.plugins.plugins[]`) | Formal extension dependencies in `package.json` |
| **Performance controls** | None (plugins can block the main thread) | Activation events, separate process, startup profiling tools |

## 4. Lessons for Saalis's `.saalix` Extension System

### 4.1 What Makes Developers WANT to Build Extensions

The evidence across both ecosystems converges on a consistent set of factors:

**Low barrier to entry is non-negotiable.** Obsidian's JavaScript requirement means any web developer can write a plugin. VSCode's JavaScript/TypeScript requirement is equally accessible. Both provide sample plugins, documentation, and CLI tooling. For saalis, the compiled Rust requirement is inherently higher friction than JavaScript — but the declarative connector definition format (TOML-based, no code) fills the low-barrier role that JavaScript fills in Obsidian. The two-tier model (declarative definitions for simple cases, full Rust for complex cases) is the right answer. The Rust toolchain barrier should be offset by excellent scaffolding (`saalis new connector`), pre-configured CI templates, and comprehensive SDK documentation.

**Fast iteration loop.** Both ecosystems enable rapid development cycles — edit, build, reload, test. For saalis, this means the SDK must support a development mode where extensions can be loaded from a local directory without packaging into `.saalix` archives, with hot-reload on file change where architecturally feasible.

**Visible impact and metrics.** Download counts, star counts, and community recognition are real motivators. The extension directory must surface these metrics prominently. Extension authors should see their impact — "your connector was used in 340 library sweeps this week."

**A rich API surface that allows unexpected uses.** Dataview did not exist in Obsidian's roadmap. LSP was not anticipated by every editor it now serves. Extensions that reshape the host's data model or enable workflows the core team never imagined generate the most ecosystem value. For saalis, this means the SDK should expose the entity query system, metadata tables, scheduler work units, and event bus broadly enough that a creative extension author can build something the core team did not anticipate — a Dataview equivalent that lets users query their game library in novel ways.

**Community and culture.** Both ecosystems have active forums, Discord servers, and documentation hubs. Obsidian's community actively celebrates plugin authors. The saalis project should foster this culture from the start — featured extensions, author spotlights, contribution guides.

### 4.2 What Makes Users Trust and Install Extensions

**Native discovery.** Both ecosystems embed extension browsing in the application itself. Users do not need to visit a website, find a download link, figure out where to put files. For saalis, the web UI must include an extension browser as a first-class feature — search, filter by category, sort by installs or rating, one-click install.

**Social proof metrics.** Download counts, ratings, last-update date, and active issue counts are the primary signals users use to evaluate trust. These must be prominent in the saalis extension directory.

**Verified identities.** VSCode's verified publisher badge is a meaningful trust signal. For saalis, extensions that handle credentials (connectors requiring API keys or login) should require verified-publisher status.

**Source availability.** Obsidian requires open-source plugins. This is a strong trust signal. Saalis should strongly encourage (though perhaps not mandate) open-source extensions listed in the community directory.

**Rollback capability.** Users need to know they can undo a bad extension install. Both ecosystems support disabling and uninstalling extensions cleanly. Saalis should support version pinning and one-click rollback to the previous version.

### 4.3 What Technical Decisions Enable or Hinder Ecosystem Growth

**Process/thread isolation enables fearless extension installation.** VSCode's Extension Host means a bad extension cannot crash the editor. Obsidian's lack of isolation means a bad plugin can crash everything. Saalis's architecture — where extensions execute within scheduler work units on the sync core — should enforce timeout-based cancellation and panic recovery at the `repr(C)` boundary. Extensions that segfault or hang should be automatically disabled with a user-visible notification, not crash the host.

**Manifest-driven capability declaration enables lazy loading and safe introspection.** VSCode's contribution points allow the host to know what an extension provides without loading it. Saalis's `repr(C)` descriptors with `inventory` collection serve the same purpose — and go further, because descriptors are statically compiled into the shared library and can be read without calling any `extern "C"` function. This is a genuine advantage over both Obsidian and VSCode.

**Activation events prevent startup degradation.** VSCode's lazy loading prevents the "death by a thousand extensions" problem where each additional extension adds a few hundred milliseconds to startup. Saalis should adopt an equivalent: extensions declare what entity types, events, or conditions trigger their activation. A connector for Steam metadata should not load until a Steam-related entity is present or a user explicitly triggers a Steam sweep.

**Formal inter-extension dependency declaration prevents fragile coupling.** Obsidian's informal inter-plugin communication via global namespace is brittle. VSCode's manifest-declared dependencies are better. Saalis's `manifest.toml` dependency declarations, resolved at load time through the registry, are the right model.

**API versioning and compatibility enforcement prevent ecosystem fragmentation.** VSCode's engine compatibility field in `package.json` ensures extensions declare their minimum supported editor version. Saalis's `SdkVersion { major, minor }` with major-version breaking changes and minor-version additive changes is the correct approach. The host should refuse to load extensions with incompatible major versions and emit warnings for extensions compiled against a newer minor version than the host provides.

**The absence of true sandboxing is a shared weakness — but saalis is better positioned to address it.** Neither Obsidian nor VSCode sandboxes extensions at runtime. Both rely on social trust (code review, verified publishers, community vigilance). Saalis's `ConnectorContext<C>` pattern — where the host controls what services an extension can access — provides a natural sandboxing boundary that JavaScript ecosystems lack. A theme extension receives no context (it is pure static data). A connector receives context scoped to its declared domains and metadata tables. A notification provider receives read-only event access. This capability-based security model, enforced at the `repr(C)` ABI boundary, would be a genuine advance over both reference ecosystems.
