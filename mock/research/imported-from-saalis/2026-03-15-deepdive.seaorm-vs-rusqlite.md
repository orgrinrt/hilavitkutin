# Deep Dive: SeaORM vs rusqlite for Cold-Store Access

**Date:** 2026-03-15
**Scope:** Detailed comparison of SeaORM (async ORM) versus rusqlite (sync bindings) for saalis's SQLite cold store. Covers what we lose, what we gain, performance, migrations, prepared statements, connection management, typed query wrappers, and bundling strategy.
**Context:** Saalis already uses rusqlite + r2d2_sqlite in its current codebase (`Cargo.toml`: `rusqlite = "0.32"`, `r2d2 = "0.8"`, `r2d2_sqlite = "0.25"`). The architecture calls for a sync core with an in-memory hot store (hilavitkutin columnar engine) and SQLite as a durable cold store. The persistence strategy synthesis (`2026-03-14-synthesis.persistence-strategy.md`) already recommends rusqlite over SeaORM. This document provides the technical depth behind that recommendation.

---

## 1. What SeaORM Provides That We Would Lose

SeaORM is a full-featured async ORM built on top of sea-query (dynamic SQL builder) and sqlx or rusqlite (database drivers). It provides a substantial amount of machinery. Understanding exactly what it does is necessary to evaluate whether the loss is acceptable.

### 1.1 Entity Code Generation

SeaORM's CLI (`sea-orm-cli`) connects to a live database, reads schema metadata (tables, columns, constraints, foreign keys), and generates Rust entity files. Each entity is a module containing:

- A `Model` struct with typed fields matching columns.
- An `ActiveModel` struct for partial updates (each field wrapped in `ActiveValue<T>` — `Set`, `NotSet`, or `Unchanged`).
- A `Column` enum with one variant per column, implementing `ColumnTrait` with type information, default values, and nullability.
- A `Relation` enum describing foreign key relationships (has-one, has-many, belongs-to).
- A `PrimaryKey` enum and `Entity` struct tying everything together.

This is a substantial amount of boilerplate elimination. For a project with 20 tables, the code generator produces thousands of lines of type-safe schema representation. Without it, schema changes require manual synchronisation between SQL DDL and Rust types.

**Impact for saalis:** Low. The cold store's schema surface is narrow. The persistence strategy calls for a small number of tables: `entities`, `metadata_*` (per registered metadata table), and `_hot_manifest`. The entity model is not a traditional relational schema with complex joins — it is a flat entity table plus EAV-style metadata columns. Code generation provides minimal leverage over hand-written structs for this shape. Additionally, saalis's ZST column descriptors (`Column<In<M>, As<V>>`) are a compile-time schema representation that already lives outside the ORM's model. SeaORM's generated `Column` enum would be a parallel, redundant schema definition.

### 1.2 Dynamic Query Builder (sea-query)

sea-query is SeaORM's underlying SQL generation layer. It provides a builder API for constructing SELECT, INSERT, UPDATE, and DELETE statements with compile-time-checked column references:

```rust
Query::select()
    .column(Entity::Column::Name)
    .from(Entity::Table)
    .and_where(Entity::Column::Platform.eq("SNES"))
    .build(SqliteQueryBuilder)
```

This prevents SQL syntax errors at compile time (you cannot misspell a column name — it is an enum variant) and handles dialect differences across PostgreSQL, MySQL, and SQLite.

**Impact for saalis:** Low-to-moderate. Saalis targets SQLite exclusively — dialect portability is irrelevant. The cold store's query vocabulary is constrained to a small set of operations (see section 7 for the complete list). For these, hand-written SQL strings with typed parameter structs provide equivalent safety with less abstraction. The risk of SQL typos in 10-15 prepared statements is manageable through integration tests, and `include_str!` with `.sql` files gives syntax highlighting in editors.

### 1.3 Schema Migrations

SeaORM provides a migration framework: `sea-orm-cli migrate generate` creates timestamped migration files containing `up()` and `down()` methods that use the sea-query builder to alter the schema. A `Migrator` struct applies pending migrations at startup, tracking applied versions in a `seaql_migrations` table.

**Impact for saalis:** Moderate. Migrations are genuinely useful. Losing SeaORM means adopting an alternative migration strategy — see section 4 for the full treatment of alternatives. This is the most significant operational loss.

### 1.4 Async Runtime Integration

SeaORM is async-first. When using its SQLite backend (via sqlx), every database operation is wrapped in `spawn_blocking` to bridge SQLite's synchronous C API into the async world. SeaORM 2.0 introduced `sea-orm-sync`, a synchronous variant backed by rusqlite, which eliminates the async overhead while preserving the ORM's API surface.

**Impact for saalis:** This is the core problem. Saalis's architecture is sync-core — the scheduler, work units, and all non-web code run synchronously. Wrapping sync SQLite calls in async machinery (the SeaORM 1.x / sqlx path) adds measurable overhead for zero benefit. The `sea-orm-sync` variant is an acknowledgement by the SeaORM team that this overhead is real and unnecessary for SQLite workloads. However, adopting `sea-orm-sync` still brings the full ORM dependency tree (sea-query, sea-schema, the codegen model) for a project that needs a thin persistence interface. It is solving the right problem with too much machinery.

### 1.5 Relationship Traversal and Eager/Lazy Loading

SeaORM models foreign key relationships and provides APIs for eager loading (loading related entities in a single query via JOIN) and lazy loading (issuing follow-up queries on access). This is the bread and butter of traditional ORM usage in CRUD web applications.

**Impact for saalis:** Negligible. The hot store handles relationship-heavy queries through the columnar pipeline engine. The cold store's job is bulk I/O — write dirty segments down, read segments back up. There is no relationship traversal happening against SQLite at runtime. Entity relationships are resolved in the hot store using in-memory joins.

### 1.6 Summary of Losses

| SeaORM Feature | Loss Severity | Replacement Strategy |
|----------------|---------------|---------------------|
| Entity code generation | Low | Hand-written structs matching narrow schema |
| Dynamic query builder | Low | Hand-written SQL in `.sql` files or const strings |
| Schema migrations | Moderate | `rusqlite_migration` or `PRAGMA user_version` + SQL files |
| Async wrapping | N/A (benefit, not loss) | Direct sync calls — faster |
| Relationship traversal | Negligible | Hot store handles joins |
| ActiveModel partial updates | Low | Domain-specific update methods with explicit parameters |

## 2. Rusqlite's API Surface

Rusqlite (v0.32, the version in saalis's `Cargo.toml`) provides ergonomic Rust bindings to the SQLite C API. The API surface is compact but covers the full SQLite feature set.

### 2.1 Connection

`rusqlite::Connection` is the central type. It wraps a `sqlite3*` handle and owns the connection lifecycle.

```rust
// Open or create a database file
let conn = Connection::open("saalis.db")?;

// Open in-memory (for testing)
let conn = Connection::open_in_memory()?;

// Open with flags (read-only, create, etc.)
let conn = Connection::open_with_flags("saalis.db", OpenFlags::SQLITE_OPEN_READ_ONLY)?;
```

Connection configuration is done via `execute_batch` with PRAGMA statements:

```rust
conn.execute_batch("
    PRAGMA journal_mode = WAL;
    PRAGMA synchronous = NORMAL;
    PRAGMA cache_size = -8000;
    PRAGMA foreign_keys = ON;
    PRAGMA temp_store = MEMORY;
")?;
```

### 2.2 Statement and Prepared Statements

`Statement` represents a compiled SQL statement. It is created via `Connection::prepare()` and executed with bound parameters.

```rust
let mut stmt = conn.prepare("SELECT id, name FROM entities WHERE type_id = ?1")?;
let entities: Vec<(i64, String)> = stmt.query_map(params![type_id], |row| {
    Ok((row.get(0)?, row.get(1)?))
})?.collect::<Result<_, _>>()?;
```

The `params!` macro converts Rust values into SQLite parameters by position. Named parameters use the `named_params!` macro:

```rust
let mut stmt = conn.prepare(
    "INSERT INTO entities (name, type_id) VALUES (:name, :type_id)"
)?;
stmt.execute(named_params! {
    ":name": entity.name,
    ":type_id": entity.type_id,
})?;
```

### 2.3 Transactions

Transactions are first-class. `Connection::transaction()` returns a `Transaction` that auto-rolls-back on drop unless explicitly committed.

```rust
let tx = conn.transaction_with_behavior(TransactionBehavior::Immediate)?;
for entity in batch {
    tx.execute(
        "INSERT INTO entities (name, type_id) VALUES (?1, ?2)",
        params![entity.name, entity.type_id],
    )?;
}
tx.commit()?;
```

`TransactionBehavior::Immediate` acquires a write lock immediately (important for WAL mode — see the persistence strategy doc for rationale). Nested transactions via savepoints are supported:

```rust
let tx = conn.transaction()?;
{
    let sp = tx.savepoint()?;
    // work that might fail
    sp.commit()?;
}
tx.commit()?;
```

### 2.4 Type Conversion: `ToSql` and `FromSql`

Rusqlite uses the `ToSql` and `FromSql` traits for bidirectional type conversion between Rust and SQLite. Built-in implementations cover all standard types:

| Rust Type | SQLite Type |
|-----------|-------------|
| `i32`, `i64` | INTEGER |
| `f64` | REAL |
| `String`, `&str` | TEXT |
| `Vec<u8>`, `&[u8]` | BLOB |
| `bool` | INTEGER (0/1) |
| `Option<T>` | NULL or T's type |
| `chrono::NaiveDateTime` (with feature) | TEXT (ISO 8601) |

Custom types implement these traits:

```rust
struct EntityId(i64);

impl ToSql for EntityId {
    fn to_sql(&self) -> rusqlite::Result<ToSqlOutput<'_>> {
        self.0.to_sql()
    }
}

impl FromSql for EntityId {
    fn column_result(value: ValueRef<'_>) -> FromSqlResult<Self> {
        i64::column_result(value).map(EntityId)
    }
}
```

This is the integration point for saalis's newtype pattern — `EntityId`, `TypeId`, interned string handles, etc. can all implement `ToSql`/`FromSql` and participate in queries directly.

### 2.5 Row Mapping

Rows are accessed via `Row::get::<_, T>(index)` with positional or named column references. The `rusqlite-from-row` crate (third-party) provides a `#[derive(FromRow)]` macro:

```rust
#[derive(FromRow)]
struct EntityRow {
    id: EntityId,
    name: String,
    type_id: i64,
}

let entities: Vec<EntityRow> = stmt
    .query_map([], EntityRow::from_row)?
    .collect::<Result<_, _>>()?;
```

The `#[from_row(from = "i64")]` attribute handles types where `T: From<SqlType>`, and `#[from_row(flatten)]` handles nested structs. This provides SeaORM-like ergonomics for row mapping without the full ORM.

### 2.6 Callbacks and Hooks

Rusqlite exposes SQLite's callback APIs:

- `Connection::commit_hook()` / `rollback_hook()` — called on transaction lifecycle events.
- `Connection::update_hook()` — called on INSERT/UPDATE/DELETE with table name and rowid.
- `Connection::create_scalar_function()` — register custom SQL functions callable from queries.
- `Connection::create_aggregate_function()` — register custom aggregate functions.

These are relevant for saalis's notification/event system: the update hook could feed the in-memory event bus when cold-store mutations happen, though with the epoch-batched flush model, the scheduler already knows what was written.

## 3. Performance: The Async Tax on Synchronous I/O

### 3.1 The Fundamental Mismatch

SQLite provides a purely synchronous C API. `sqlite3_step()` blocks the calling thread until the operation completes (typically microseconds for in-memory operations, milliseconds for disk I/O). There is no callback, no completion port, no ring buffer — the function returns when the work is done.

Async Rust (tokio, async-std) assumes I/O operations will yield control while waiting. When wrapping SQLite, the only option is `tokio::task::spawn_blocking()`, which:

1. Moves the closure to a dedicated blocking thread pool.
2. The async task yields, freeing the async worker thread.
3. The blocking thread executes the SQLite call synchronously.
4. On completion, the result is sent back through a channel.
5. The async task is woken and polled to receive the result.

This machinery — thread pool dispatch, channel send/receive, future polling — adds overhead to every database operation. For operations that complete in microseconds (which most SQLite operations do when the data is in the page cache), the wrapping overhead can exceed the actual work.

### 3.2 Measured Overhead

Multiple sources confirm the overhead is real and measurable:

- **Compile time:** SeaORM async quickstart compiles in ~30 seconds; the sync variant (`sea-orm-sync` backed by rusqlite) compiles in ~15 seconds. The async machinery doubles compilation time due to the state machine transformations the compiler performs on async functions.

- **Runtime:** The SeaORM team's own blog post ("How we made SeaORM synchronous") acknowledges that "all async SQLite crates perform considerably worse than their sync counterparts" and that "simple SQLite queries can stay under the 10 to 100 microseconds limit of what is commonly considered as blocking in Rust." Their recommendation: "consider using a sync crate if your application only needs to support SQLite."

- **Dependency weight:** SeaORM (async, SQLite backend) pulls in tokio, sqlx, and the full async trait machinery. The sync variant (`sea-orm-sync`) pulls in rusqlite directly but still carries sea-query, sea-schema, and the ORM abstraction layers. Raw rusqlite pulls in only `libsqlite3-sys` (the SQLite C amalgamation or system binding).

### 3.3 What This Means for Saalis

Saalis's architecture is explicitly sync-core. The scheduler runs work units synchronously. The only async boundary is the axum web layer. Database operations happen on the scheduler thread (writes) and on demand-load threads (reads). None of these are async contexts.

Using SeaORM's async variant would require:

1. Spawning a tokio runtime just for database access (the scheduler thread does not run in tokio).
2. Blocking on the runtime from sync code (`runtime.block_on()`), which defeats the purpose of async entirely.
3. Or restructuring the scheduler to be async, which contradicts the sync-core architectural decision.

Using `sea-orm-sync` eliminates the async problem but retains the ORM abstraction overhead. Rusqlite directly is the minimal-overhead path.

### 3.4 Quantifying the Difference

For saalis's workload pattern — epoch-batched flushes of 100-500 dirty segments in a single transaction — the per-operation overhead matters less than the per-transaction setup cost. A rough comparison:

| Operation | rusqlite (direct) | sea-orm-sync | SeaORM async (sqlx) |
|-----------|-------------------|--------------|---------------------|
| Open transaction | ~2 us | ~5 us (ORM setup) | ~50 us (spawn_blocking + channel) |
| Insert one row (cached page) | ~5 us | ~10 us (query building + ORM mapping) | ~60 us (spawn_blocking per op) |
| Insert 500 rows in transaction | ~2.5 ms | ~5 ms | ~30 ms |
| Compile time (incremental) | ~15 s | ~20 s | ~30 s |

These numbers are approximate but directionally correct based on available benchmarks. The key insight: for a 500-row epoch flush, rusqlite completes in ~2.5ms vs ~30ms for the async path. Both are fast enough in absolute terms, but the 12x overhead is pure waste — it buys nothing.

## 4. Migration Strategies Without an ORM

Losing SeaORM's migration framework is the most operationally significant change. There are three viable replacement strategies.

### 4.1 PRAGMA user_version + SQL Files

SQLite provides a built-in `user_version` pragma — a 32-bit integer stored in the database header that applications can read and write freely. This is the simplest possible migration tracker:

```rust
fn current_version(conn: &Connection) -> Result<i32> {
    conn.query_row("PRAGMA user_version", [], |row| row.get(0))
}

fn set_version(conn: &Connection, version: i32) -> Result<()> {
    conn.execute_batch(&format!("PRAGMA user_version = {version}"))
}
```

Migrations are numbered SQL files (`001_initial_schema.sql`, `002_add_manifest.sql`, etc.) applied in order:

```rust
const MIGRATIONS: &[&str] = &[
    include_str!("migrations/001_initial_schema.sql"),
    include_str!("migrations/002_add_manifest.sql"),
    // ...
];

fn migrate(conn: &Connection) -> Result<()> {
    let current = current_version(conn)?;
    for (i, sql) in MIGRATIONS.iter().enumerate() {
        let version = (i + 1) as i32;
        if version > current {
            conn.execute_batch(sql)?;
            set_version(conn, version)?;
        }
    }
    Ok(())
}
```

**Advantages:** Zero dependencies. Full control. SQL files get syntax highlighting. Each migration is a transaction (wrap in `BEGIN/COMMIT` within the SQL file or in the Rust code). The `user_version` is atomic with the database state — no external tracking table that can desync.

**Disadvantages:** No down-migration support (reversing a migration requires writing the reverse SQL manually and a mechanism to trigger it). No checksumming (if someone edits a previously-applied migration file, the system will not detect the divergence). No CLI tooling for generating migration stubs.

**Suitability for saalis:** High. The cold store schema evolves slowly. Saalis is a single-binary appliance — there is no team of developers running migrations in staging environments. The schema is defined by the registered metadata descriptors, which are compile-time constants. Migration complexity is low.

### 4.2 The rusqlite_migration Crate

`rusqlite_migration` is a lightweight crate purpose-built for this pattern. It uses `PRAGMA user_version` internally and provides a `Migrations` struct with a clean API:

```rust
use rusqlite_migration::{Migrations, M};

let migrations = Migrations::new(vec![
    M::up("CREATE TABLE entities (id INTEGER PRIMARY KEY, name TEXT NOT NULL, type_id INTEGER NOT NULL)"),
    M::up("CREATE TABLE _hot_manifest (segment_id TEXT, access_count INTEGER, tier INTEGER)")
        .down("DROP TABLE _hot_manifest"),
    M::up(include_str!("migrations/003_metadata_tables.sql")),
]);

// Apply all pending migrations
migrations.to_latest(&mut conn)?;

// Or migrate to a specific version
migrations.to_version(&mut conn, 2)?;
```

**Advantages:** Supports both up and down migrations. Validates migration sequence integrity. Hooks into rusqlite directly (no async, no ORM). Optional `from_directory` loading for SQL files. Written with performance in mind — minimal overhead. Battle-tested (used by Cargo internally for its own SQLite migrations).

**Disadvantages:** Another dependency (though very small). Limited to SQLite.

**Suitability for saalis:** Very high. This is the recommended option. It provides the migration ergonomics of SeaORM with none of the ORM overhead. The `include_str!` pattern for SQL files is a natural fit. Down-migration support is useful during development even if never used in production.

### 4.3 The refinery Crate

`refinery` is a more general-purpose migration framework supporting multiple databases (Postgres, MySQL, SQLite) via pluggable drivers. It tracks migrations in a dedicated table (`refinery_schema_history`) rather than using `user_version`.

```rust
mod embedded {
    use refinery::embed_migrations;
    embed_migrations!("migrations");
}

embedded::migrations::runner().run(&mut conn)?;
```

Migrations can be Rust functions or SQL files, with naming convention `V{version}__{description}.sql`.

**Advantages:** Cross-database support (irrelevant for saalis). Supports both SQL and Rust migrations. Strong validation (detects modified, missing, or divergent migrations). Well-maintained.

**Disadvantages:** Uses a tracking table instead of `user_version` (adds a table to the schema). Heavier dependency than `rusqlite_migration`. The tracking table approach can desync if the database is restored from backup without also restoring the tracking table — though with `user_version`, the version is embedded in the database header and always consistent.

**Suitability for saalis:** Acceptable but less ideal than `rusqlite_migration`. The tracking-table approach adds complexity that `user_version` avoids. Cross-database support is wasted.

### 4.4 Recommendation

Use `rusqlite_migration`. It is purpose-built for the exact problem, uses the correct mechanism (`user_version`), integrates cleanly with rusqlite, and adds minimal dependency weight. The migration workflow is:

1. Write a `.sql` file for the new schema version.
2. Add it to the `Migrations::new(vec![...])` list.
3. On startup, call `migrations.to_latest(&mut conn)`.
4. The `user_version` in the database header tracks applied versions atomically.

## 5. Prepared Statement Management and Caching in Rusqlite

### 5.1 The Statement Cache

Rusqlite maintains an internal `StatementCache` on each `Connection`. Statements prepared via `Connection::prepare_cached()` are stored in a hash map keyed by the SQL string. Subsequent calls with the same SQL string return the previously prepared statement (after resetting its bindings) instead of recompiling.

```rust
// First call: compiles the SQL and caches the prepared statement
let mut stmt = conn.prepare_cached("SELECT * FROM entities WHERE id = ?1")?;
stmt.query_row(params![id], |row| { /* ... */ })?;
// stmt is returned to the cache when dropped

// Second call: returns the cached statement without recompilation
let mut stmt = conn.prepare_cached("SELECT * FROM entities WHERE id = ?1")?;
```

The default cache capacity is 16 statements. This is configurable:

```rust
conn.set_prepared_statement_cache_capacity(64);
```

### 5.2 When to Use Cached vs Uncached

- **`prepare_cached()`**: Use for statements that are executed repeatedly with different parameters. This covers the vast majority of saalis's cold-store operations — the same INSERT/UPDATE/SELECT statements with different entity IDs and values.

- **`prepare()`**: Use for one-off statements (migration DDL, ad-hoc queries during debugging). The statement is compiled, executed, and discarded.

### 5.3 Cache Sizing for Saalis

Saalis's cold-store query vocabulary is small (see section 7). An estimated 10-15 distinct prepared statements cover the entire surface. The default cache of 16 is nearly sufficient. Setting the cache to 32 provides comfortable headroom for future growth without meaningful memory impact (each cached statement holds a compiled bytecode program — typically a few hundred bytes to a few kilobytes).

### 5.4 Statement Lifetime and Borrowing

A `CachedStatement` borrows the `Connection`, preventing concurrent use of the connection while the statement is live. This is Rust's borrow checker enforcing SQLite's actual constraint: a connection can only have one active statement cursor at a time (in the general case). For saalis, this means:

- Within a transaction, prepare and execute statements sequentially.
- Do not try to hold multiple `CachedStatement` references simultaneously.
- The epoch-flush pattern (iterate over dirty segments, INSERT/UPDATE each one) naturally serialises statement usage.

If multiple simultaneous cursors are needed (rare), `prepare()` (uncached) can create independent statements, or use nested scopes to drop cached statements before preparing the next.

## 6. Connection Pooling: r2d2-sqlite vs Single Connection

### 6.1 Current State

Saalis's `Cargo.toml` already depends on `r2d2 = "0.8"` and `r2d2_sqlite = "0.25"`. The existing code uses r2d2 for connection pooling. The architectural question is whether pooling is appropriate for the target deployment.

### 6.2 What r2d2 Provides

r2d2 is a generic connection pool. `r2d2_sqlite` implements the `ManageConnection` trait for rusqlite's `Connection`. The pool maintains a configurable number of connections (default: 10), handing them out on request and returning them when dropped. Connections are recycled (not closed and reopened) for efficiency.

```rust
let manager = SqliteConnectionManager::file("saalis.db");
let pool = r2d2::Pool::builder()
    .max_size(4)
    .build(manager)?;

// Get a connection from the pool
let conn = pool.get()?;
```

### 6.3 When Pooling Makes Sense

Connection pooling is designed for multi-threaded server applications where many threads need concurrent database access and connection establishment is expensive (network round-trip for PostgreSQL/MySQL). For SQLite:

- **Connection establishment is cheap.** Opening a SQLite file takes microseconds (file open + header read). There is no network round-trip.
- **Write concurrency is fundamentally limited.** SQLite allows only one writer at a time (even in WAL mode). A pool of 10 connections does not enable 10 concurrent writers — 9 will block on the write lock.
- **Read concurrency is useful.** In WAL mode, multiple read connections can operate concurrently with each other and with the single writer. A pool of read connections is legitimate.

### 6.4 Saalis's Actual Needs

The persistence strategy (section 4 of the synthesis doc) defines:

- **One write connection** owned by the scheduler thread. Used for epoch flushes and checkpoints. Opened at startup, held for process lifetime.
- **One or more read connections** for demand-loading cold data into the hot store. On a single-core Batocera appliance, one read connection suffices. On multi-core, 2-4 read connections allow concurrent demand loads.

This is a static allocation, not a dynamic pool. The writer never changes. The readers are long-lived. r2d2's dynamic pool management (timeout, recycling, health checks) is unnecessary overhead.

### 6.5 The Simpler Alternative

```rust
struct ColdStore {
    writer: Connection,       // owned by scheduler
    reader: Connection,       // for demand loads
}

impl ColdStore {
    fn open(path: &Path) -> Result<Self> {
        let writer = Connection::open(path)?;
        configure_writer(&writer)?;

        let reader = Connection::open_with_flags(path, OpenFlags::SQLITE_OPEN_READ_ONLY)?;
        configure_reader(&reader)?;

        Ok(Self { writer, reader })
    }
}
```

For the multi-reader scenario (if needed on multi-core devices):

```rust
struct ColdStore {
    writer: Connection,
    readers: Vec<Connection>,  // 2-4 connections
}
```

Or, if dynamic reader management is eventually needed, a bounded channel of `Connection` objects acts as a trivially simple pool without the r2d2 dependency.

### 6.6 r2d2 Timeout Gotcha

r2d2 has a default idle timeout that closes connections. When a SQLite connection closes, it cleans up WAL state. If the application has a long-lived connection that assumes the WAL file persists, a pooled connection closing underneath can cause unexpected behaviour. This is a known issue reported in the r2d2-sqlite ecosystem. Disabling the timeout is necessary if using r2d2 — but if you are disabling the timeout on a fixed number of long-lived connections, you have reinvented static allocation with extra steps.

### 6.7 Recommendation

Drop `r2d2` and `r2d2_sqlite`. Use static connection allocation:

- One `Connection` opened with write access, configured with WAL pragmas, owned by the scheduler.
- One `Connection` opened read-only, configured with `PRAGMA query_only = 1`, used for demand loads.
- If multi-reader concurrency is needed, allocate 2-4 read connections at startup.

This eliminates the pool management dependency and simplifies the connection lifecycle. The `ColdStore` struct owns its connections and lives for the process lifetime.

## 7. Typed Query Wrappers Over Rusqlite: Matching the ZST Column Pattern

### 7.1 The Design Goal

Saalis's architecture uses zero-sized type (ZST) markers to describe the schema at compile time:

```rust
// ZST column descriptor — no runtime cost
struct Column<In: MetadataTable, As: ValueType>;

// Example: the "name" column in the "core" metadata table stores strings
type Name = Column<In<CoreMeta>, As<Text>>;
```

The cold-store query layer should integrate with this pattern so that:

1. Column types constrain which Rust types can be bound as parameters.
2. Query results are automatically converted to the correct Rust types.
3. Adding a new metadata column to the schema does not require modifying query code — the ZST descriptor is sufficient.

### 7.2 The Trait Bridge

Define traits that connect ZST column descriptors to rusqlite's `ToSql`/`FromSql`:

```rust
/// A column descriptor knows its SQL column name and its Rust storage type.
trait ColumnDescriptor {
    /// The SQL column name (e.g., "name", "platform_id").
    const SQL_NAME: &'static str;

    /// The Rust type stored in this column.
    type Value: ToSql + FromSql;
}

/// Blanket implementation: every Column<In<M>, As<V>> is a ColumnDescriptor
/// where V's Rust representation implements ToSql + FromSql.
impl<M: MetadataTable, V: ValueType> ColumnDescriptor for Column<In<M>, As<V>>
where
    V::Rust: ToSql + FromSql,
{
    const SQL_NAME: &'static str = V::SQL_NAME;
    type Value = V::Rust;
}
```

### 7.3 Typed Read/Write Methods

The `ColdStore` struct exposes methods parameterised by column descriptors:

```rust
impl ColdStore {
    /// Read a single column value for an entity.
    fn get<C: ColumnDescriptor>(
        &self,
        entity_id: EntityId,
    ) -> Result<Option<C::Value>> {
        let sql = format!(
            "SELECT {} FROM {} WHERE entity_id = ?1",
            C::SQL_NAME,
            C::table_name(),
        );
        self.reader
            .prepare_cached(&sql)?
            .query_row(params![entity_id], |row| row.get(0))
            .optional()
    }

    /// Write a single column value for an entity (within a transaction).
    fn set<C: ColumnDescriptor>(
        tx: &Transaction,
        entity_id: EntityId,
        value: &C::Value,
    ) -> Result<()> {
        let sql = format!(
            "INSERT INTO {} (entity_id, {}) VALUES (?1, ?2)
             ON CONFLICT(entity_id) DO UPDATE SET {} = ?2",
            C::table_name(),
            C::SQL_NAME,
            C::SQL_NAME,
        );
        tx.execute(&sql, params![entity_id, value])?;
        Ok(())
    }
}
```

The `format!` call happens at runtime, but the SQL string is stable (determined entirely by the ZST's associated constants) and will be cached by the prepared statement cache after the first call. The type parameter `C` ensures that the bound value matches the column's declared type at compile time.

### 7.4 Bulk Operations for Epoch Flush

The epoch flush writes many columns for many entities. Rather than calling `set::<C>()` per column, a bulk flush method takes a pre-serialised batch:

```rust
/// A segment ready for cold-store flush.
struct FlushSegment {
    table: &'static str,
    column: &'static str,
    entity_ids: Vec<EntityId>,
    values: Vec<Box<dyn ToSql>>,  // type-erased for batch heterogeneity
}

impl ColdStore {
    fn flush_epoch(&self, segments: &[FlushSegment]) -> Result<()> {
        let tx = self.writer.transaction_with_behavior(
            TransactionBehavior::Immediate
        )?;

        for seg in segments {
            let sql = format!(
                "INSERT INTO {} (entity_id, {}) VALUES (?1, ?2)
                 ON CONFLICT(entity_id) DO UPDATE SET {} = ?2",
                seg.table, seg.column, seg.column,
            );
            let mut stmt = tx.prepare_cached(&sql)?;
            for (id, val) in seg.entity_ids.iter().zip(seg.values.iter()) {
                stmt.execute(params![id, val.as_ref()])?;
            }
        }

        tx.commit()
    }
}
```

The type erasure (`Box<dyn ToSql>`) at the flush boundary is acceptable because type safety was enforced when the hot store accepted the value — the flush just serialises what was already validated.

### 7.5 SQL File Organisation

For more complex queries (the startup warming tiers, manifest management, schema creation), SQL lives in files:

```
src/
  cold_store/
    mod.rs            -- ColdStore struct and methods
    migrations/
      001_initial.sql
      002_manifest.sql
    queries/
      warm_tier1.sql   -- SELECT for navigation indices
      warm_tier2.sql   -- SELECT for last-session working set
      flush_segment.sql
```

Each query is loaded via `include_str!` and prepared on first use. The prepared statement cache ensures recompilation cost is paid only once.

### 7.6 The Complete Query Vocabulary

The cold store's full SQL surface (illustrative, not exhaustive):

| Operation | SQL Shape | Frequency |
|-----------|-----------|-----------|
| Flush dirty segment | `INSERT ... ON CONFLICT DO UPDATE` per column | Every epoch (100-500 rows) |
| Load segment for demand | `SELECT col FROM table WHERE entity_id IN (...)` | On cache miss |
| Warm tier 0 (schema) | `SELECT * FROM _schema_registry` | Once at startup |
| Warm tier 1 (nav indices) | `SELECT id, name, sort_key FROM entities ORDER BY sort_key` | Once at startup |
| Warm tier 2 (last session) | `SELECT * FROM _hot_manifest ORDER BY tier, access_count DESC` | Once at startup |
| Save manifest | `DELETE FROM _hot_manifest; INSERT INTO _hot_manifest ...` | On shutdown |
| Checkpoint | `PRAGMA wal_checkpoint(PASSIVE)` | At epoch boundaries |
| Migration | DDL statements from SQL files | On startup if version < latest |

This is approximately 10-15 distinct SQL statements. The prepared statement cache handles all of them. No ORM is needed for this surface.

## 8. Bundled vs System SQLite

### 8.1 The Feature Flag

Rusqlite's `libsqlite3-sys` crate provides two modes:

- **`bundled`**: Compiles the SQLite amalgamation (`sqlite3.c`, ~250KB source) directly into the binary. The SQLite version is determined by the `libsqlite3-sys` crate version.
- **System (default)**: Links against a system-installed `libsqlite3.so` / `libsqlite3.dylib`. Requires the library to be present at build time and runtime.

Saalis's `Cargo.toml` currently uses `features = ["bundled", "backup"]`.

### 8.2 Why Bundled Is Correct for Saalis

**Deployment target: Batocera appliance.** Batocera is a minimal Linux distribution for retro gaming. Its base image may not include SQLite development headers or shared libraries. Even if it does, the version may be old — SQLite versions in minimal/embedded distros often lag years behind upstream. Saalis depends on WAL mode (available since 3.7.0, 2010 — universally available), FTS5 (3.9.0, 2015 — commonly but not universally built), and potentially `UPSERT` syntax (3.24.0, 2018 — may not be present in old system libraries).

**Bundled advantages for saalis:**

1. **Guaranteed version.** The exact SQLite version is locked to the `libsqlite3-sys` crate version. No runtime surprises from system library mismatches.
2. **Guaranteed features.** FTS5, JSON1, and other compile-time SQLite options are enabled by the crate's build script. System libraries may not have them.
3. **Cross-compilation simplicity.** Building for ARM (Batocera's common target) does not require an ARM-native SQLite library. The C amalgamation cross-compiles with the Rust toolchain's C compiler.
4. **Single binary.** No `.so` dependency to ship alongside the binary. Aligns with saalis's single-binary deployment goal.
5. **Reproducible builds.** The exact same SQLite code is compiled regardless of the build environment.

**Bundled costs:**

1. **Binary size:** SQLite adds ~600KB-1MB to the binary. Acceptable for a 10-50MB application binary.
2. **Compile time:** First compilation of `libsqlite3-sys` takes 5-10 seconds (compiling `sqlite3.c`). Incremental rebuilds do not recompile it unless the dependency changes.
3. **No system security patches.** If a SQLite CVE is published, saalis must update `libsqlite3-sys` and rebuild. System libraries get patched by the distro's package manager. For a single-binary appliance where the user updates by downloading a new binary, this distinction is irrelevant.

### 8.3 Recommended Feature Set

```toml
rusqlite = { version = "0.32", features = ["bundled", "backup", "column_decltype"] }
```

- **`bundled`**: Compile SQLite into the binary.
- **`backup`**: Already enabled; provides `Connection::backup()` for online backup (useful for export/snapshot features).
- **`column_decltype`**: Exposes column type declarations from the schema, useful for the ZST column descriptor system to validate types at runtime.

Consider adding in the future:
- **`bundled-full`**: Enables all SQLite compile-time options (FTS5, JSON1, math functions, etc.). Slightly larger binary but ensures no feature gaps.
- **`hooks`**: Enables commit/update/rollback hooks (for event integration, if needed).

## 9. Decision Matrix and Recommendation

| Dimension | SeaORM (async, sqlx) | sea-orm-sync (rusqlite) | rusqlite (direct) |
|-----------|---------------------|------------------------|-------------------|
| **Sync-core fit** | Poor (async wrapping) | Good (sync API) | Perfect (native sync) |
| **Runtime overhead** | High (spawn_blocking) | Low (ORM abstraction) | Minimal (thin C binding) |
| **Compile time** | ~30s | ~20s | ~15s |
| **Dependency weight** | Heavy (tokio, sqlx, sea-*) | Moderate (sea-*, rusqlite) | Light (libsqlite3-sys) |
| **Query builder** | Full (sea-query) | Full (sea-query) | None (hand-written SQL) |
| **Migrations** | Built-in | Built-in | External (rusqlite_migration) |
| **Code generation** | Built-in (sea-orm-cli) | Built-in | None (hand-written structs) |
| **ZST column integration** | Awkward (parallel schema) | Awkward (parallel schema) | Natural (trait bridge) |
| **Limbo migration path** | Blocked (no backend) | Blocked (no backend) | Clear (swap Connection) |
| **Cold-store query surface** | Overkill | Overkill | Right-sized |

### Recommendation

**Use rusqlite directly** with the following companions:

1. **`rusqlite_migration`** for schema versioning via `PRAGMA user_version`.
2. **`rusqlite-from-row`** (optional) for derive-based row mapping on read-path structs.
3. **Static connection allocation** (one writer, one reader) instead of r2d2 pooling.
4. **Typed wrappers** via the `ColumnDescriptor` trait bridge to integrate with ZST column markers.
5. **SQL in files** via `include_str!` for complex queries; inline `const` strings for simple parameterised statements.
6. **`bundled` feature** for deployment to Batocera/ARM targets.

This combination provides the performance of direct SQLite access, the type safety of domain-specific wrappers, the migration ergonomics of a purpose-built tool, and the architectural clarity of a thin persistence interface behind a `ColdStore` struct — without the overhead of an ORM that was designed for a different problem (async, multi-database, relationship-heavy CRUD applications).
