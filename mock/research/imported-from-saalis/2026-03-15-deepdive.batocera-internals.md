> Research date: 2026-03-15
> Purpose: Understand Batocera's architecture deeply enough to plan saalis integration on the loisto appliance.

## 1. Filesystem Architecture: Squashfs + Overlay

Batocera's root filesystem is a **read-only squashfs image** stored on the boot partition. At boot, this image is mounted at `/` and provides the entire operating system — binaries, libraries, init scripts, the EmulationStation frontend, all emulator cores and standalone emulators. Because squashfs is compressed and read-only, the system is effectively immutable firmware.

### Writable Layer

On top of the squashfs, Batocera layers a **tmpfs (RAM-backed) overlay** using OverlayFS. This means any writes to the root filesystem during runtime go to RAM and are **lost on reboot** by default. If a user or developer wants to persist changes to the root filesystem (e.g. installing a package, modifying a system config), they must explicitly run:

```
batocera-save-overlay
```

This script creates a patch file at `/boot/boot/overlay` (on the boot partition) that stores the delta between the current filesystem state and the pristine squashfs. On subsequent boots, this overlay file is applied on top of the squashfs. The overlay is limited to **100 MB by default**, though the size can be overridden with a flag. Removing `/boot/boot/overlay` reverts all modifications.

To modify anything under `/boot/` at runtime, the boot partition must be remounted writable:

```
mount -o remount,rw /boot
```

It reverts to read-only on next boot automatically.

### Partition Layout

Batocera uses exactly **two partitions**:

| Partition | Label | Filesystem | Mount Point | Contents |
|-----------|-------|------------|-------------|----------|
| Boot | BATOCERA | FAT32 (x86_64) or platform-dependent | `/boot` | Kernel, initramfs, squashfs image (`boot/batocera`), bootloader files (GRUB/syslinux/U-Boot), `batocera-boot.conf`, overlay file |
| Userdata | SHARE | ext4 (default) | `/userdata` | ROMs, BIOS files, saves, screenshots, `system/batocera.conf`, themes, decorations, custom services |

The boot partition requires at least **6 GB** for the update system to function. The userdata partition expands to fill remaining disk space (auto-resize on first boot, controlled by `autoresize` in `batocera-boot.conf`).

## 2. Userdata Partition Structure

The `/userdata` partition (also referenced as `/share` or `SHARE`) is where all user-facing data lives. It is the only writable persistent partition in normal operation. Key directories:

```
/userdata/
  bios/               # BIOS/firmware files for emulators
  cheats/             # Cheat files
  decorations/        # Bezel/border overlays
  kodi/               # Kodi media center data
  music/              # Background music for ES
  roms/               # Game ROMs, organized by system (roms/nes/, roms/snes/, etc.)
  saves/              # Save states and SRAM saves
  screenshots/        # In-game screenshots
  splash/             # Custom splash screens
  system/             # System configuration
    batocera.conf     # Main configuration file
    configs/          # Per-emulator config overrides
    services/         # User-defined services (v38+)
    logs/             # System and emulator logs
  themes/             # EmulationStation themes
```

### Supported Userdata Filesystems

The userdata partition can be reformatted to several filesystems:

- **ext4** — Default. Full Linux compatibility, no size limits, symlinks supported.
- **BTRFS** — Supported. Copy-on-write, better corruption protection. Slightly slower than ext4. Batocera ships a `btrfs_for_windows` driver on the boot partition for cross-platform access.
- **exFAT** — Supported but limited. No symlinks, no Unix permissions. Considered "outdated" by the Batocera docs.
- **NTFS** — Usable but problematic. Wine, Steam, Cemu, Flatpak, and other advanced systems may malfunction due to missing file attributes and symlink incompatibility.
- **FAT32** — Technically usable, but severe 4 GB file size limit makes it impractical.

## 3. Boot Sequence

### Stage 1: Bootloader

- **x86_64**: GRUB (EFI) or syslinux (legacy BIOS). The GRUB config is on the boot partition.
- **ARM**: U-Boot. Batocera maintains device-specific U-Boot forks (e.g. [RG351P_u-boot](https://github.com/batocera-linux/RG351P_u-boot)).

### Stage 2: Initramfs (BusyBox init)

The bootloader loads the Linux kernel and an initramfs. The initramfs is a minimal BusyBox environment containing an `init` shell script (ash). This script:

1. Mounts the boot partition at `/boot_root`.
2. Checks for `/boot_root/boot/batocera.update` — if present, performs an in-place squashfs update (see Section 9).
3. Mounts the squashfs image at `/new_root` (read-only).
4. If `/boot/boot/overlay` exists, applies it as an OverlayFS upper layer.
5. Mounts a tmpfs as the writable OverlayFS layer.
6. Calls `switch_root /new_root /sbin/init` to hand off to the real init system.

### Stage 3: Buildroot Init (SysV-style)

Batocera is built on **Buildroot** and uses a traditional SysV-style init with scripts in `/etc/init.d/`. Two master scripts orchestrate the sequence:

- `/etc/init.d/rcS` — runs all `S##*` scripts in numerical order at boot.
- `/etc/init.d/rcK` — runs all `K##*` scripts in numerical order at shutdown.

Key init scripts in approximate order:

| Script | Purpose |
|--------|---------|
| `S00bootcustom` | Executes `/boot/boot-custom.sh` (earliest user hook, before network) |
| `S02splash` | Displays splash screen / boot video |
| `S03udevd` | Starts udev for device hotplug |
| `S08connman` | Starts ConnMan network manager |
| `S11share` | Mounts the userdata/SHARE partition |
| `S12populateshare` | Populates default directory structure on SHARE (runs synchronously — blocks boot) |
| `S15bluetooth` | Starts Bluetooth daemon |
| `S31emulationstation` | Launches EmulationStation (the main UI) |
| `S99userservices` | Runs user-defined services from `/userdata/system/services/` |

There is also a **postshare.sh** hook (v35+): `/boot/postshare.sh` executes after `/userdata` is mounted but before the splash video plays and before EmulationStation loads.

### Stage 4: EmulationStation

`S31emulationstation` launches the Batocera fork of EmulationStation, which becomes the primary user interface — a fullscreen, controller-navigable game browser.

## 4. The batocera-services System

Introduced in **Batocera v38**, the services system replaces the deprecated `custom.sh` approach. Services are user-defined scripts that integrate with the boot/shutdown lifecycle.

### Creating a Service

Place a script (without `.sh` extension) in `/userdata/system/services/`. The script must handle `start`, `stop`, and optionally `status` arguments:

```bash
#!/bin/bash
case "$1" in
    start)
        # Launch your daemon
        /path/to/my-daemon &
        ;;
    stop)
        killall my-daemon
        ;;
    status)
        pidof my-daemon >/dev/null && echo "running" || echo "stopped"
        ;;
esac
```

### Naming Rules

- Only `A-Z`, `a-z`, `0-9`, and `_` are allowed in the filename.
- Must not start with a digit.
- Must use Unix line endings (LF, not CR/LF).
- No `.sh` extension.

### Management

```
batocera-services list user       # List all user services
batocera-services enable <name>   # Enable service (starts on boot)
batocera-services disable <name>  # Disable service
batocera-services start <name>    # Start immediately
batocera-services stop <name>     # Stop immediately
```

Services can also be toggled from the EmulationStation **Services** menu, which both starts/stops immediately and persists the enabled/disabled state.

All user services are dispatched by `S99userservices`, which means they run **after** EmulationStation has started. The `S99` script waits for all service `stop` handlers to complete during shutdown, so services must avoid infinite loops or long sleep timers in their stop path.

**Sources:**
- [Batocera Services Wiki](https://wiki.batocera.org/scripting_services_rules_examples)

## 5. EmulationStation: System Discovery and Emulator Launching

### System Discovery

EmulationStation reads system definitions from `/usr/share/emulationstation/es_systems.cfg` (an XML file). Each `<system>` entry defines:

- `<name>` — internal identifier (e.g. `nes`, `snes`, `psx`)
- `<fullname>` — display name
- `<path>` — ROM directory (e.g. `/userdata/roms/nes`)
- `<extension>` — supported file extensions
- `<command>` — launch command template
- `<platform>` — scraping platform identifier
- `<theme>` — theme folder name

A system only appears in the UI if its ROM directory contains at least one file with a matching extension. All emulator cores are pre-installed — there is no need to "add" cores.

Users can create custom system definitions at `/userdata/system/configs/emulationstation/es_systems_<custom_name>.cfg`, which overlay the built-in definitions.

### Launch Pipeline

When a user selects a game, EmulationStation invokes the `<command>` template, which in Batocera is typically:

```
python /usr/lib/python3.X/site-packages/configgen/emulatorlauncher.py \
    %CONTROLLERSCONFIG% -system %SYSTEM% -rom %ROM%
```

The **configgen** system (`emulatorlauncher.py`) is a Python-based configuration generator that:

1. Reads the system name and ROM path from arguments.
2. Determines the correct emulator/core from `batocera.conf` settings (per-system or global defaults).
3. Invokes a system-specific **generator** (e.g. `libretroGenerator`, `rpcs3Generator`, `dolphinGenerator`) that writes the appropriate emulator configuration files.
4. Launches the emulator process, managing video mode switching, controller mapping, and shader configuration.
5. On exit, restores the video mode and returns control to EmulationStation.

### Embedded Web Server

EmulationStation includes a built-in C++ web server (since v32) used internally by the Content Downloader to update game lists. It can be enabled for external access for remote control and game list management.

## 6. batocera.conf Configuration System

The primary configuration file is `/userdata/system/batocera.conf`. It is a flat key-value file with dotted hierarchical keys.

### Syntax

```ini
## This is a comment
#key.that.is.disabled=value    # Commented-out setting

system.hostname=BATOCERA
system.timezone=Etc/UTC

wifi.enabled=1
wifi.ssid=MyNetwork
wifi.key=MyPassword

# Per-system emulator overrides
snes.emulator=libretro
snes.core=snes9x

# Global settings
global.videomode=CEA 4 HDMI
global.shaderset=none

# Update settings
updates.enabled=1
updates.type=stable
```

### Key Sections

- **System**: `system.hostname`, `system.timezone`, `system.language`
- **Network**: `wifi.enabled`, `wifi.ssid`, `wifi.key` (up to 3 SSIDs: `wifi2.*`, `wifi3.*`)
- **Controllers**: `controllers.bluetooth.enabled`, controller mapping settings
- **Global emulator**: `global.videomode`, `global.shaderset`, `global.ratio`
- **Per-system overrides**: `<system>.emulator`, `<system>.core`, `<system>.ratio`
- **Updates**: `updates.enabled`, `updates.type` (`stable` or `butterfly` for dev builds)

### CLI Tools

```
batocera-settings-get <key>          # Read a value
batocera-settings-set <key> <value>  # Write a value
```

These tools work with the standard config file format and are designed for scripting.

### Boot Configuration

A separate file, `/boot/batocera-boot.conf`, controls boot-level settings on the boot partition:

- `sharedevice` — external storage device for userdata
- `sharewait` — seconds to wait for external storage to come online
- `autoresize` — whether to auto-expand userdata partition on first boot

## 7. Network Configuration

Batocera uses **ConnMan** (Connection Manager) as its network daemon.

### Ethernet

Wired connections auto-configure via DHCP. For static IP:

```
connmanctl config ethernet_<MAC>_cable \
    --ipv4 manual <IP> <netmask> <gateway> \
    --nameservers <DNS1> <DNS2>
```

Changes take effect immediately without restart.

### Wi-Fi

Managed via `batocera-wifi`:

```
batocera-wifi start                    # Enable Wi-Fi hardware
batocera-wifi scanlist                 # Scan for networks
batocera-wifi enable <SSID> <key>      # Connect and save
batocera-wifi disable                  # Disable Wi-Fi
batocera-wifi list                     # List saved networks
```

Up to 3 Wi-Fi networks can be stored in `batocera.conf` (`wifi.*`, `wifi2.*`, `wifi3.*`). Special characters in SSID/password must be backslash-escaped.

### SSH Access

SSH is available by default. Default credentials: `root` / no password (or `linux` on some versions). The hostname defaults to `BATOCERA` and is configurable via `system.hostname` in `batocera.conf`.

**Sources:**
- [Wi-Fi Wiki](https://wiki.batocera.org/wifi_ssid)

## 8. Bluetooth and Gamepad Setup

Bluetooth is managed by `batocera-bluetooth` and BlueZ under the hood.

### Configuration

Enable in `batocera.conf`:
```
controllers.bluetooth.enabled=1
```

### Pairing Methods

1. **GUI**: Controller Settings > PAIR A BLUETOOTH CONTROLLER (puts pad in scan mode).
2. **CLI**: `batocera-bluetooth trust-pad` — scans for and trusts only controllers.
3. **Manual**: `bluetoothctl` — `scan on`, `pair <MAC>`, `connect <MAC>`, `trust <MAC>`.

Controller mappings are handled by EmulationStation's built-in input configuration system, which maps buttons to a canonical layout and passes controller configs to emulators via `%CONTROLLERSCONFIG%` in the launch command.

## 9. The Update System

Batocera's update mechanism is elegant in its simplicity, enabled by the squashfs architecture.

### Automatic (OTA) Update Process

1. The updater downloads `boot.tar.xz` to `/userdata/system/upgrade/boot.tar.xz`.
2. Verifies the MD5 checksum.
3. Remounts `/boot` read-write.
4. Backs up boot config files (`batocera-boot.conf`, etc.).
5. Extracts `boot.tar.xz` to `/boot/`, replacing kernel, initramfs, and squashfs image.
6. Restores backed-up config files.
7. Removes the overlay file (since it was built against the old squashfs).
8. On next boot, the initramfs detects the new squashfs and mounts it.

### Manual Update

Users can also manually place a `batocera.update` file (a new squashfs image) on the boot partition. The initramfs `init` script checks for this file during boot and, if found, swaps it in as the new root squashfs.

### Update Channels

Controlled via `batocera.conf`:
- `updates.type=stable` — stable releases
- `updates.type=butterfly` — development/beta builds

The update system requires at least **2 GB free** on the userdata partition for the download staging area.

## 10. Kernel Configuration and Available Modules

Batocera is built on Buildroot and maintains per-architecture kernel defconfig files. The x86_64 defconfig is at [`board/batocera/x86/linux_x86_64-defconfig.config`](https://github.com/batocera-linux/batocera.linux/blob/master/board/batocera/x86/linux_x86_64-defconfig.config) in the source tree.

### Modules Relevant to saalis

| Module | Status | Notes |
|--------|--------|-------|
| **BTRFS** | Available | Supported as a userdata filesystem. Driver included. |
| **FUSE** | Available | Used by MergerFS (included in Batocera) for merging multiple directories. Kernel FUSE module is loaded. |
| **inotify** | Available | Standard Linux kernel feature. `CONFIG_INOTIFY_USER` is enabled in virtually all Linux kernels, including Batocera's. |
| **OverlayFS** | Built-in | Core to Batocera's architecture (squashfs + tmpfs overlay). |
| **ext4** | Built-in | Default userdata filesystem. |
| **FAT/VFAT** | Built-in | Required for the boot partition. |
| **squashfs** | Built-in | Required for the root filesystem. |
| **USB storage** | Available | For external drives and USB sticks. |

### FUSE + inotify Caveat

While both FUSE and inotify are available as kernel features, **inotify does not propagate through FUSE filesystems**. This means if saalis monitors a MergerFS mount via inotify, it will not receive events for changes made on underlying branches. This is a known upstream limitation of the FUSE/fsnotify interaction, not specific to Batocera. Direct monitoring of the actual filesystem (not the FUSE mount) works as expected.

## 11. Integration Strategies for saalis

Given the above architecture, there are three viable approaches for running saalis on Batocera.

### Option A: User Service (Recommended)

Deploy saalis as a **batocera-services user service**. This is the least invasive approach:

- Place the saalis binary somewhere on `/userdata/` (e.g. `/userdata/system/saalis/saalis`).
- Create `/userdata/system/services/saalis` (no extension):

```bash
#!/bin/bash
SAALIS_DIR=/userdata/system/saalis
case "$1" in
    start)
        cd "$SAALIS_DIR"
        ./saalis serve >> "$SAALIS_DIR/saalis.log" 2>&1 &
        echo $! > "$SAALIS_DIR/saalis.pid"
        ;;
    stop)
        [ -f "$SAALIS_DIR/saalis.pid" ] && kill $(cat "$SAALIS_DIR/saalis.pid") 2>/dev/null
        rm -f "$SAALIS_DIR/saalis.pid"
        ;;
    status)
        [ -f "$SAALIS_DIR/saalis.pid" ] && kill -0 $(cat "$SAALIS_DIR/saalis.pid") 2>/dev/null \
            && echo "running" || echo "stopped"
        ;;
esac
```

- Enable: `batocera-services enable saalis`
- saalis starts after EmulationStation (S99), runs alongside it, serves its web UI.
- Survives reboots. Survives Batocera OTA updates (userdata is untouched).
- The SQLite database, configuration, and all saalis state live on `/userdata/`.
- saalis can directly read/manage ROM files in `/userdata/roms/`.

**Advantages**: Zero modification to system partition. Survives updates. Toggleable from ES menu. User-friendly.

**Disadvantages**: Starts after ES. Cannot replace ES as the primary UI without additional work.

### Option B: Overlay-Based System Integration

Use `batocera-save-overlay` to inject saalis deeper into the system:

- Place the binary in `/usr/bin/saalis`.
- Create a proper init.d script (e.g. `S30saalis`) to start before EmulationStation.
- Save with `batocera-save-overlay`.

**Advantages**: Starts earlier in the boot sequence. Can be positioned before or after any init stage.

**Disadvantages**: The overlay is wiped on every OTA update. Limited to 100 MB. Fragile across version upgrades. Requires re-application after updates.

### Option C: Replace EmulationStation

For the loisto appliance where saalis IS the primary interface:

- Modify `S31emulationstation` (via overlay or custom build) to launch saalis instead of EmulationStation.
- saalis would need to handle controller input, video mode management, and emulator launching (potentially delegating to configgen/emulatorlauncher.py for the actual emulator invocation).
- Alternatively, saalis could launch EmulationStation as a subprocess or embed it.

**Advantages**: Full control over the user experience. saalis becomes the shell.

**Disadvantages**: Requires maintaining a custom overlay or fork. Complex integration with Batocera's emulator configuration pipeline.

### Recommended Approach for loisto

For the initial release, **Option A** (user service) is the pragmatic choice. saalis runs as a background web service, accessible via browser from any device on the network. It manages the content library (catalogue, enrich, download, organise) while EmulationStation handles the actual gaming UI. The two coexist without conflict.

For a future "saalis-native" loisto build, **Option C** could be pursued as a custom Batocera image where saalis replaces EmulationStation entirely, built from source using Batocera's Buildroot infrastructure.

### Key Integration Points

- **ROM directory**: `/userdata/roms/` — saalis writes here, ES discovers games here.
- **BIOS directory**: `/userdata/bios/` — saalis can manage BIOS files.
- **Configuration**: saalis stores its own config on `/userdata/system/saalis/` to avoid conflicts with `batocera.conf`.
- **SQLite database**: Lives on `/userdata/` (ext4 or BTRFS — both support SQLite fine; avoid NTFS/exFAT).
- **Network**: ConnMan manages the network; saalis binds to `0.0.0.0:<port>` for its web UI.
- **Filesystem monitoring**: inotify works on ext4/BTRFS userdata partitions for watching ROM directories. Avoid relying on inotify over FUSE/MergerFS mounts.
- **Static binary**: Since Batocera is a minimal embedded system, saalis should be compiled as a **statically-linked binary** (musl target) or bundled with its dependencies to avoid glibc version mismatches.
