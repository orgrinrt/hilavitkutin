## Preamble: My Disagreement

I disagree with the Savant's claim that replication_budget should be 0
during Active/Stress metabolic states. The Savant writes:

> "in Active/Stress mode: replication_budget = 0 // no background
> replication."

This creates a post-convergence deadlock. Convergence forces
Active/Stress via metabolic override. The exit sequence restores
normal metabolic governance. But during the convergence window itself,
filter reconciliation reveals under-replicated content, and the
replication budget is zero. The peer has just discovered the problem
and is prohibited from acting on it. After convergence exits and the
metabolic state decays to Idle or Background, the peer can finally
replicate, but by then the urgency has passed and the system has been
operating with known under-replication for the duration of the
convergence window plus the dwell time for the transition back to
Idle.

The correct approach is that replication during Active/Stress should
be throttled but not halted. The Conductor already provides
bandwidth governance via LEDBAT and the BandwidthPolicy mechanism.
Replication work uses bulk tier streams. The LEDBAT controller will
naturally yield to foreground traffic. Setting replication_budget = 0
is a second throttle applied on top of a throttle that already works,
and the second throttle creates a deadlock where the first does not.

I will develop this position in detail under Tension 4.

## Part 1: Response to Expert B (Sans-IO Protocol Framework Designer)

### Problem 1: No Polling Interface

Expert B identifies the polling interface as the single most critical
gap. I agree with the severity assessment. From a convergence
perspective, the polling interface determines how convergence signals
(HeartbeatReceived with divergent Merkle roots, ConvergenceGates) flow
from the protocol layer to the mesh layer. Without a defined polling
convention, the convergence triggering mechanism is unspecified at the
integration boundary.

Expert B's four-method Protocol trait (handle, poll_transmit,
poll_event, poll_timeout) is well-motivated by quinn and s2n-quic
precedent. The Savant proposes a two-method alternative (feed,
poll_output). I address this in Tension 1 below.

**What matters for my domain:** the poll_event method (or
poll_output in the Savant's model) is where HeartbeatReceived events
emerge. The mesh layer's convergence detection logic consumes
HeartbeatReceived events, compares Merkle roots, and triggers
convergence. The polling convention must guarantee that
HeartbeatReceived events are delivered within the same event loop
iteration as the BytesReceived input that produced them. If
HeartbeatReceived is buffered and delivered in a later iteration,
the convergence detection has one-iteration latency, which at
5-second heartbeat intervals is negligible. But at Active mode
(2-second heartbeats), a one-iteration delay could cause the mesh
to miss the divergence detection window and require an additional
heartbeat round. Both Expert B's and the Savant's models deliver
events within the same iteration (drain loop), so this is not a
distinguishing factor.

### Problem 2: Reliability Mode Selection

Outside my domain -- no comment on the choice between all-reliable
vs mixed-reliability streams. I note only that gossip IHAVE
messages being reliable-ordered (Expert B's Option A) does not affect
convergence correctness. IHAVE is lazy repair (Plumtree). If IHAVE is
lost, eager push still delivers the message. Reliable IHAVE adds
unnecessary retransmission overhead but does not break convergence.

### Problem 3: MetabolicGovernor Concrete Parameter Mapping

Expert B's threshold table with hysteresis is well-specified. The
Q16.16 fixed-point representation is correct and aligns with the
Savant's fixed-point boundary enforcement. The three thresholds
(0.20, 0.50, 0.80) and hysteresis margin (0.05) are calibration items
that can be tuned empirically.

**From a convergence perspective**, the metabolic override during
convergence bypasses the governor thresholds and forces Active or
Stress. Expert B's implementation correctly handles this:

```
if convergence_override {
    return max(candidate, MetabolicState::Active);
}
```

This is correct. The convergence override should always produce at
least Active, and the governor's pressure readings should still push
to Stress if resource pressure is genuinely high. Expert B's
formulation achieves this via `max(candidate, Active)`.

One subtlety: Expert B does not address what happens when the
convergence override is released (exit sequence Step 3: "restore
metabolic dwell time enforcement"). If the governor's current
candidate state is Background (low pressure) but the system was just
in convergence-forced Active, the dwell time should apply to the
Active-to-Background transition. Expert B's dwell time check
(`now_ms - last_transition_ms < dwell_time_ms`) correctly handles
this only if last_transition_ms is updated when the convergence
override is released, not when the override was initially applied. The
exit sequence should set last_transition_ms = now_ms when it releases
the override.

### Problem 4: Crypto Thread Interaction

Expert B's CryptoRequest/CryptoResult model is clean and compatible
with the sans-IO pattern. Outside my domain regarding the
implementation, but the interaction with convergence is relevant:
control log entries received during convergence (Front 1, GSet merge)
require Ed25519 verification. If verification is async (Expert B's
model), the convergence front cannot complete until all control log
entries are verified. This introduces a dependency between convergence
progress and crypto thread throughput.

At 50us per Ed25519 verification on ARM and potentially hundreds of
control log entries during a long partition, the crypto queue could
impose a several-millisecond delay on convergence. This is negligible
compared to the convergence duration bound (120-600 seconds). Expert
B's model is acceptable.

### Problem 5: Timer Management

Expert B's replacement of SetTimer/TimerExpired with poll_timeout
and handle_timeout is the quinn pattern. This simplifies the consumer
but removes the Savant's implicit cancellation (SetTimer with same
timer_id overwrites). Both approaches work for convergence timers.

The convergence wavefront timer is the most critical timer from my
domain. It must fire exactly once after reconnection to gate the
start of convergence. Under Expert B's model, the wavefront timer is
an internal connection timer managed by the protocol. Under the
Savant's model, the wavefront timer is a SetTimer output that the
consumer schedules. Either works, but the wavefront timer is a mesh-
layer concern (convergence is a mesh state machine), so it should not
be internal to the protocol state machine. The protocol state machine
handles transport; the mesh layer handles convergence.

This means the wavefront timer is NOT one of Expert B's connection
timers. It is a mesh-layer timer. Expert B's poll_timeout aggregates
only protocol-level timers (heartbeat, RTO, path challenge). The
mesh layer must manage its own timers separately (wavefront,
convergence duration bound, dwell time). The consumer needs two timer
sources: protocol poll_timeout() and mesh-layer timers. This is
fine; the consumer's timer wheel can hold both.

## Part 2: Response to Expert C (NAT Traversal and Relay Engineer)

### Problem 1: hop_count AEAD Bug

Expert C's identification of the hop_count AEAD bug is correct and
critical. The fix (exclude hop_count from AEAD AD) is the right
approach. I have no disagreement.

From a convergence perspective, this bug would prevent any relayed
connection from functioning, which means convergence between
clusters that require relay connectivity would never complete. The
wavefront timer would fire, convergence would begin, but the CPSK
renegotiation for the relayed pair would fail because all frames
through the relay would have invalid auth tags. The pair would
remain unconverged, creating a persistent partial-partition state.
This confirms Expert C's severity assessment: CRITICAL.

### Problem 2: Transit Relay Forwarding

Expert C's forwarding table specification is complete and practical.
The per-frame cost (~300ns) is negligible. The forwarding table
structure (HashMap<(destination, connection_id), ForwardingEntry>)
is correct.

**Convergence interaction:** During convergence, the polygon schedule
coordinates CPSK renegotiation for each gateway pair. If a pair
communicates through a relay, the Noise KK handshake messages must
transit the relay's forwarding table. Expert C does not explicitly
address whether Noise handshake frames (which use the magic header
0x56524B4F) are forwarded by the same mechanism. The relay
distinguishes pre-handshake (magic header) from post-handshake
(destination header) frames. Handshake frames do not have the
standard outer header with destination and connection_id. The relay
needs a separate routing mechanism for handshake frames, or the
handshake must be bootstrapped through a pre-existing forwarding
table entry.

I believe the correct resolution is that SIGNAL HELLO establishes
the forwarding entry before the Noise handshake begins. The
initiating gateway sends SIGNAL HELLO to the relay with the target
destination, the relay creates the forwarding entry, and subsequent
Noise handshake frames are forwarded via that entry. But this requires
handshake frames to carry the destination field, which the current
wire format does not include in the handshake magic header. This is
a gap that Expert C's forwarding specification does not fully resolve.

### Problem 3: NAT Traversal

Expert C provides a complete NAT traversal state machine based on
RFC 8445 (ICE). The specification is thorough. I address the
relay-first vs hole-punch-first question in Tension 2 below.

### Problem 4: Relay Failover

Expert C's warm standby relay proposal is sound. The failover
latency (2-30 seconds depending on metabolic state) is acceptable.
The relay-specific keepalive probes (2-second interval, 182 bytes/s)
are negligible.

**Convergence interaction:** If a relay fails during convergence, the
gateway pair using that relay loses connectivity. The warm standby
promotes, but the CPSK renegotiation in progress must restart. The
polygon schedule should detect the connection failure and reschedule
the pair's renegotiation for the next polygon slot. This means the
polygon schedule needs a retry mechanism for individual pairs. The
current design treats the polygon schedule as a fixed sequence. It
needs to handle relay failover as a transient failure that
requeues the pair.

### Problem 5: Seat Failover

Outside my domain -- no comment on seat failover mechanisms. The
distributed consistency aspects of seat failover (snapshot semantics,
deferred write replay) are addressed in my Stage 1 Problem 6.

### Problem 6: Multi-hop Relay

Expert C's recommendation to document multi-hop as "v1 supports
1-hop relay only" is pragmatic. I agree. For the household mesh
deployment envelope (2-12 clusters), multi-hop is an edge case.
The hop_count field reserves the capability for future extension.

### Problem 1: The Hollow Unit Cell (Intra-Cluster Communication)

The Savant correctly identifies the CIK key degeneracy problem:
Noise KK with identical static keys on both sides produces identical
DH outputs in both directions. This is a fundamental observation
that my Stage 1 review also reached (DEK-based Noise KK for intra-
cluster). I am glad the Savant independently confirms this.

The Savant's simplified intra-cluster health scoring (single
heartbeat counter, no dual-EMA) is the right approach. LAN peers
do not need the full health pipeline. The 3-second timeout (3
missed 1-second heartbeats) for local health degradation is
appropriate.

**The resource accounting is valuable.** The Savant calculates 10KB
per intra-cluster peer (60KB maximum for 6-peer cluster), bringing
the managed arena to 260KB. My Stage 1 review noted the resource
gap but did not provide concrete numbers. The Savant's numbers are
reasonable.

However, I note that the Savant's 1-second intra-cluster heartbeat
interval is fixed, not variable with metabolic state. This is
correct for LAN (bandwidth is not a constraint), but it means
intra-cluster heartbeats consume a fixed budget regardless of the
system's overall resource pressure. At 91 bytes per heartbeat and
5 intra-cluster sessions, this is 455 bytes/second, or ~3.6 kbps.
Negligible on LAN, but it means the Conductor cannot reduce
intra-cluster heartbeat overhead during Stress mode. For a household
device under heavy load, this is acceptable. The intra-cluster
heartbeat is a fixed cost of cluster membership.

### Problem 2: The Amorphous Relay

The Savant's relay failover specification adds an important

### Problem 3: The SansIO Amorphous Surface

The Savant's polling model (feed/poll_output/next_deadline) differs

The Savant's NeedArena handling (degraded mode: heartbeats and SACKs

### Problem 4: The Missing Replication Lattice

The Savant proposes a priority-queue approach
(most_under_replicated()) with cluster-diversity-first placement.
My Stage 1 proposes a probabilistic approach (should_replicate with
BLAKE3 roll). I address this conflict in Tension 4.

The Savant's observation about metadata sync via alternating Merkle
roots in heartbeats (even heartbeats carry filter root, odd carry
stash root) is elegant. This halves the per-heartbeat overhead
while maintaining the anti-entropy property. Both roots are
eventually exchanged; the detection latency doubles (from N
heartbeats to 2N heartbeats for full divergence detection). At
Active mode (2-second heartbeats), this means ~4 seconds for filter
divergence detection and ~4 seconds for stash divergence detection.
Acceptable.

However, I have a concern about this alternating scheme during
convergence. The convergence wavefront timer gates on detecting
divergence, which requires receiving a heartbeat with a divergent
Merkle root. If the first heartbeat after reconnection happens to be
an "odd" heartbeat (stash root), the convergence trigger may fire on
stash divergence rather than filter divergence. The convergence
mechanism should trigger on either divergence, which the design
already supports (the wavefront timer fires "on reconnection," not
"on filter divergence detection"). So the alternating scheme is
compatible with convergence triggering.

### Problem 5: The Relay Defect Migration Gap

The Savant correctly identifies that relay failure has no defect
migration path. The relay is not a mesh peer, so the health pipeline
does not apply. The Savant proposes relay path health as a separate
EMA track. Expert C proposes warm standby relay with keepalive probes.
These are complementary: the EMA track detects gradual degradation;
the warm standby provides fast failover on hard failure.

From a convergence perspective, relay failure during convergence is a
serious problem. If the relay fails and the warm standby takes over,
the CPSK renegotiation for the affected pair must restart. But the
polygon schedule may have already advanced past that pair's slot. The
polygon schedule should be idempotent: if a pair's renegotiation
fails, the pair is added back to the schedule for the next pass.
This is not currently specified.

### Problem 6: The Uncrystallized Channel Lifecycle

The Savant's channel lifecycle specification (admin-only creation,
flat u16 indices, no deletion, revocation via KeyBundle(Revoke))
aligns with my Stage 1 Problem 3 recommendation. No disagreements.

The Savant adds the observation that EDT scope_flags (u64 bitmask)
references channels by bit position, and dynamic channel creation
requires updating the bitmask, but EDT tokens are signed and
immutable. This is a genuine problem that I did not identify.
However, the resolution is straightforward: the scope_flags bitmask
limits the EDT to channels that existed at EDT creation time. New
channels created after the EDT was issued are not accessible via
that EDT. The admin must reissue the EDT to include new channels.
This is consistent with the security model: EDTs are delegation
instruments with limited scope, and scope expansion requires admin
action.

### Problem 7: The Fixed-Point Conversion Boundary

The Savant's identification of floating-point violations in the
abstract state machine code samples is thorough. The
CorrelatedSuspicion confidence: f32 violation, the scar decay
formula with f64 literals, the CapacityRatio guard with f64
comparison, and the EMA replay_observation with powf() calls are
all legitimate problems.

From a convergence perspective, the fixed-point boundary is critical
because health scores determine when convergence exits (exit
sequence Step 5: "update health baselines"). If two peers compute
different health baselines due to floating-point divergence, one
peer may exit convergence while the other remains in convergence,
creating a split-brain convergence state. The fixed-point enforcement
prevents this.

The Savant's lookup table approach for the alpha_effective
computation (2KB per alpha_base value, 256 entries, compile-time
generated) is the correct solution. The powf() function cannot be
computed in fixed-point without a lookup table or polynomial
approximation. The lookup table provides exact reproducibility
across platforms. 2KB is negligible.

I note that my Stage 1 replication probability formula also uses
floating-point:

```
let probability = deficit as f64 / (known_holders as f64 + 1.0);
let roll = BLAKE3(...)[0] as f64 / 255.0;
roll < probability
```

The Savant's fixed-point boundary rule applies here. The replication
decision function is part of the mesh layer's abstract behavior.
Rewriting in fixed-point:

```
let probability_q16_16 = ((deficit as u32) << 16)
    / (known_holders as u32 + 1);
let roll_q16_16 = (BLAKE3(...)[0] as u32) << 8;
    // 0-255 mapped to 0x0000-0xFF00 (Q16.16 range [0, ~0.996])
roll_q16_16 < probability_q16_16
```

This is deterministic across platforms and avoids the f64 operations.
I update my ideal design to use this fixed-point formulation.

**What the Savant taught me:** The fixed-point boundary is more
pervasive than I initially recognized. Every computation that crosses
peers must be fixed-point. The replication decision function is one
such computation. The Savant's systematic identification of all
floating-point violations provides a template for auditing any new
mechanism added to the design.

## Part 4: Responses to the Five Tensions

### Tension 1: SansIO Polling Convention (Four-Method vs Two-Phase)

**My position:** The four-method split (Expert B) is preferable for
convergence observability, but the difference is not load-bearing
for my domain.

Expert B's separation of poll_transmit and poll_event means the mesh
layer receives application events (HeartbeatReceived, SessionLost)
from a different drain loop than transmission outputs (SendBytes).
This provides type-level separation: the convergence logic processes
HeartbeatReceived events from poll_event without needing to branch
on whether the output is a SendBytes or a HeartbeatReceived.

The Savant's unified poll_output collapses both into one stream. The
convergence logic must match on ProtocolOutput variants:

```
match output {
    ProtocolOutput::SendBytes { .. } => socket.sendto(..),
    ProtocolOutput::HeartbeatReceived { .. } => convergence.check(..),
    ProtocolOutput::SessionLost { .. } => convergence.trigger(..),
    ...
}
```

This is a single branch in the hot path. At the scale of this system
(~36 peers, <1000 events per second), the branch cost is negligible.

From a convergence perspective, the more important property is
whether the drain loop is atomic within an event loop iteration.
Both models guarantee this: Expert B's consumer drains all
poll_transmit then all poll_event before the next handle(); the
Savant's consumer drains all poll_output before the next feed().
Both ensure that all outputs from one input are processed before
the next input is consumed. Convergence detection happens within a
single iteration in both models.

The Savant's structural reentrancy prevention (feed cannot be called
from within output processing because poll_output does not call back)
is a genuine advantage. Expert B achieves the same through the
`&mut self` borrow (you cannot call handle() while poll_event holds
&mut self). Both prevent reentrancy, but the Savant's model makes it
structurally obvious.

**My recommendation:** I slightly prefer Expert B's four-method split
for its type-level output separation, but I would not block on this.
Either model works for convergence. The arena lifetime rule is the
more important question, and there I prefer Expert B's explicit
arena.reset() over the Savant's implicit invalidation on next feed().
The explicit reset gives the convergence logic a clear window to
process HeartbeatReceived events and extract Merkle roots before the
arena-backed data is invalidated.

### Tension 2: Relay-First vs Hole-Punch-First

**My position:** Relay-first is correct. I support Expert C's
recommendation.

Expert C cites Tailscale's DERP model: always connect to relay first,
attempt direct connections in background, migrate to direct if hole
punching succeeds. This is the right approach for convergence
reliability.

The convergence wavefront timer starts "on reconnection." If
reconnection requires hole punching (2-10 seconds, may fail), the
wavefront timer fires on a connection that may not yet be usable. If
hole punching fails, the peer falls back to relay (additional 100ms
for relay path setup). Total time to usable connection: 2-10 seconds
plus relay setup. During this time, the convergence protocol is
stalled.

With relay-first, the connection is usable within ~500ms (one RTT to
relay). The wavefront timer fires on a usable connection. Convergence
can begin immediately. If hole punching later succeeds in the
background, traffic migrates to the direct path, but convergence
does not care about the path quality; it cares about the path existing.

The stronger argument: convergence correctness requires all gateway
pairs to be able to exchange CPSK renegotiation messages. The polygon
schedule sequences these renegotiations. If one pair cannot exchange
messages because hole punching is still in progress, the polygon
schedule stalls for that pair, delaying convergence for the entire
mesh. Relay-first eliminates this stall.

**Counterargument addressed:** The design's TransitRelay invariant 3
("transit relay is last resort after hole punching fails") embodies a
preference for direct connections. This is a good preference in
steady state. But during convergence, the priority is not optimal path
quality; it is path existence. Relay-first optimizes for path
existence. The invariant should be restated as: "transit relay
provides immediate connectivity; direct paths are preferred when
available and are established in background."

**Interaction with Expert C's amplification prevention rule:** Expert
C specifies that a relay MUST NOT forward to an address from which it
has not received a recent (within 30 seconds) authenticated frame.
If relay-first is adopted, both endpoints establish Noise sessions
with the relay before forwarding begins. The amplification prevention
is satisfied by the SIGNAL HELLO exchange, which is an authenticated
frame. The 30-second window aligns with Background mode heartbeats
(30 seconds), but the Tension Map identifies a liveness hazard: a
peer in Background mode sends heartbeats every 30 seconds, which is
exactly at the relay's amplification prevention threshold. The first
heartbeat after a 30-second gap may arrive at 30.001 seconds,
exceeding the threshold.

**My recommendation:** Set the relay's amplification prevention
window to 35 seconds (not 30), providing a 5-second margin for
Background mode heartbeats. Alternatively, use the relay keepalive
mechanism (2-second probes per Expert C's Problem 4) to ensure the
relay always has a recent authenticated frame, regardless of
heartbeat interval. The keepalive probes are separate from heartbeats
and would keep the relay's forwarding entry alive even during
Background mode.

### Tension 3: Gateway Handoff -- Connection Teardown vs Migration

**My position:** Teardown and re-establish. The Savant's connection_id
preservation is elegant but creates unresolvable state transfer
problems.

The Savant proposes connection_id preservation across relay migration:
"The connection_id is preserved across relay migration (same mechanism
as connection migration: PATH_CHALLENGE/PATH_RESPONSE through the new
relay). The connection survives the relay change."

For relay migration (same connection, different relay path), this is
correct. The connection endpoints have not changed; only the relay
has. PATH_CHALLENGE through the new relay verifies reachability, and
the connection continues with the same session keys.

But gateway handoff is different. The gateway is a connection
endpoint. When the gateway changes, the new gateway must establish
new Noise KK sessions with the mesh. The old gateway's session keys
are specific to the old gateway's DEK. The new gateway cannot
continue the old session because it does not have the old gateway's
Noise session state (hash chain, nonce counter, replay window).

The Savant's relay failover model (connection_id preserved) works
because the relay is not a Noise endpoint. The Savant does not
explicitly extend this to gateway handoff; the Tension Map infers
the extension.

I recommend:

1. **Relay migration:** connection_id preserved (Savant's model).
   The relay is not a Noise endpoint. Only the forwarding path
   changes. PATH_CHALLENGE through the new relay verifies reachability.

2. **Gateway handoff:** connection teardown and re-establishment.
   The new gateway is a new Noise endpoint. All inter-cluster sessions
   are torn down and re-established with the new gateway's DEK.
   In-flight frames may be lost during the handoff window.

The concern about CIK rotation during gateway handoff (Tension Map:
"some peers have the new CIK and some do not, while the old gateway's
CPSK sessions are already torn down") is valid but manageable. The
CIK rotation is a control log entry. The GSet merge ensures all peers
eventually receive the new CIK. The ordering problem is that
inter-cluster CPSK sessions use CIK for Noise KK, so the new gateway
cannot establish CPSK sessions with the new CIK until all cluster
peers have the new CIK. This is solved by:

1. CIK distribution is an intra-cluster operation (vital tier,
   reliable delivery).
2. Gateway handoff waits for CIK distribution to complete before
   establishing new inter-cluster sessions.
3. The old gateway's inter-cluster sessions remain active until the
   new gateway's sessions are established (handoff overlap window).

The handoff overlap window ensures continuity. The old gateway
continues forwarding during the overlap. When the new gateway's
sessions are established, traffic migrates. The old gateway tears
down its sessions.

**What this requires:** An explicit gateway handoff protocol, not
just "old gateway's sessions are torn down." The protocol should be:

```
gateway_handoff(old, new):
    // Phase 1: Prepare
    old -> new: transfer CPSK session state (intra-cluster, vital)
    old -> new: transfer observation summaries (intra-cluster, vital)

    // Phase 2: Establish
    new: establish inter-cluster Noise KK sessions
    old: continue forwarding during establishment

    // Phase 3: Migrate
    new: announce gateway change via heartbeat
    remote gateways: migrate to new gateway's sessions
    old: drain remaining in-flight frames

    // Phase 4: Teardown
    old: tear down old inter-cluster sessions
```

This is more complex than "teardown and re-establish" but preserves
continuity. The overlap window prevents frame loss. The VitalQueue
ensures critical frames (KeyBundle, convergence gates) survive the
handoff.

### Tension 4: Replication Orchestration (Probabilistic vs Priority-Queue)

**My position:** Combine both. Priority-queue ordering with
probabilistic per-peer selection.

Expert A (my Stage 1): Probabilistic replication with BLAKE3 roll.
Coordination-free. Each peer independently decides whether to
replicate with probability inversely proportional to known holders.
Deterministic per (blob_id, peer, epoch).

Savant: Priority-queue with most_under_replicated(). Each peer
replicates the most under-replicated content first. No thundering
herd mitigation. Budget is 0 during Active/Stress.

**Why the Savant's approach alone is insufficient:**

1. **Thundering herd.** The Savant's most_under_replicated() is a
   global ordering. If all peers compute the same ordering (they
   will, because they see the same filter state after reconciliation),
   all peers will try to replicate the same blob. With target_replicas
   = 3 and known_holders = 1, all 35 other peers could simultaneously
   initiate a transfer of the same 2GB blob. The LEDBAT pacing limits
   the bandwidth impact, but 35 concurrent LEDBAT flows to the same
   source peer would still saturate the source's uplink.

2. **Active/Stress halt deadlock.** As stated in my preamble.
   Convergence forces Active/Stress. Replication budget = 0. Filter
   reconciliation reveals under-replicated content. No action taken.
   Convergence exits. Metabolic state decays to Idle (dwell time).
   Replication finally starts. The system has operated with known
   under-replication for potentially minutes.

**Why my probabilistic approach alone is insufficient:**

1. **No priority ordering.** My Stage 1 approach replicates whatever
   the peer happens to evaluate first. There is no guarantee that the
   most under-replicated content is prioritized. The peer might
   replicate a P4 blob (low priority) while a P1 blob (critical)
   remains at 1 replica.

2. **The Savant's cluster-diversity constraint is not captured.** My
   probabilistic approach is cluster-agnostic. A peer decides to
   replicate without considering whether the target cluster already
   has a replica. The cluster-diversity-first placement (Savant:
   "at least 2 replicas in different clusters") requires knowledge
   of cross-cluster replica distribution, which only the filter
   state provides.

**Combined approach:**

```
on_heartbeat_tick():
    budget = replication_budget_for_metabolic_state()
    // Budget is NEVER zero. Minimum 1 chunk per tick.
    // In Active/Stress: budget = 1 (throttled, not halted)
    // In Idle: budget = 5
    // In Background: budget = 1

    for _ in 0..budget:
        matter = most_under_replicated()
        // Priority ordering (Savant's contribution)
        if matter.replica_count >= matter.min_replicas:
            break  // nothing under-replicated

        // Probabilistic per-peer selection (my contribution)
        if should_replicate_q16(matter.blob_id,
                                 matter.known_holders,
                                 matter.target_replicas):
            target = select_placement_target_cluster_diverse(matter)
            // Cluster-diversity constraint (Savant's contribution)
            initiate_transfer(matter, target)
```

Where should_replicate_q16 is the fixed-point version of my
probabilistic function:

```
fn should_replicate_q16(blob_id, known_holders, target_replicas) -> bool {
    let deficit = target_replicas - known_holders;
    let probability_q16 = ((deficit as u32) << 16)
        / (known_holders as u32 + 1);
    let roll_q16 = (BLAKE3("saalis-repl-v1" || blob_id || my_dk || epoch)[0]
                    as u32) << 8;

}
```

This combines:
- Priority ordering (most under-replicated first) from the Savant
- Probabilistic deduplication (coordination-free) from my Stage 1
- Cluster-diversity placement from the Savant
- Fixed-point arithmetic from the Savant's boundary rule
- Non-zero budget in all metabolic states (my correction)

**Convergence properties of the combined approach:**

1. **Convergent.** The deficit decreases as replicas are created.
   When all blobs reach target_replicas, the loop breaks. The
   probabilistic deduplication ensures expected O(1) redundant
   transfers per blob per epoch.

2. **Partition-safe.** During partition, each partition independently
   maintains replicas. After partition heal, filter reconciliation
   reveals new content. The priority queue surfaces the most
   under-replicated content first. The probabilistic roll prevents
   thundering herd on the newly-discovered content.

3. **Active/Stress safe.** Budget = 1 during Active/Stress.
   Replication proceeds slowly but does not halt. The LEDBAT pacing
   ensures bulk transfers yield to foreground traffic. One chunk per
   tick is ~1170 bytes per heartbeat interval (2 seconds at Active =
   585 bytes/second). This is negligible bandwidth but ensures
   progress.

4. **Post-convergence catch-up.** After convergence exits and the
   metabolic state decays to Idle, the budget increases to 5 chunks
   per tick. The priority queue ensures the most critical gaps are
   filled first. The probabilistic roll prevents all peers from
   targeting the same gaps simultaneously.

### Tension 5: Relay Trust Model (Full Peer vs Pseudo-Peer)

**My position:** The relay is a pseudo-peer (Savant's model), but it
participates in territorial re-encryption with a special hash ring
position.

Expert C models the relay as a full Noise KK participant with DK/DEK.
The Savant models the relay as a pseudo-peer with DK/DEK but no CIK,
no gossip, no health scoring, no convergence participation.

From a replication perspective, the Replication defined concept says
"Relay peers participate." This means relay peers hold replicas
(encrypted, in the membrane). For replicas held by a relay, the
territorial re-encryption mechanism assigns re-encryption territories
based on DK pubkeys on the consistent hash ring.

If the relay is a pseudo-peer with a DK pubkey, it has a position on
the hash ring. It can be assigned a territory. But the relay does not
perform re-encryption (it cannot; it does not hold channel keys). The
gateway must perform re-encryption for relay-held replicas in the
relay's territory.

**My recommendation:**

1. The relay has a DK/DEK (Savant's model). The DK pubkey gives the
   relay a hash ring position.

2. The relay's hash ring territory is assigned to the nearest
   non-relay peer on the ring. This peer is responsible for
   re-encrypting relay-held replicas.

3. The relay provides GET/PUT operations for the responsible peer
   to read, re-encrypt, and write back the relay-held replicas.

4. The relay counts toward replication targets. A replica held by a
   relay is a valid replica for durability purposes (the relay stores
   ciphertext on disk). But for the cluster-diversity constraint, the
   relay is its own "cluster" (it has no CIK). This means
   relay-held replicas contribute to diversity.

5. When the relay goes offline, its replicas become inaccessible.
   The content filter still advertises them (the filter is per-cluster,
   not per-peer). The responsible peer detects relay failure (via
   keepalive probes), marks the relay's replicas as unavailable, and
   initiates re-replication to bring the blob back to
   target_replicas without counting the relay's copy.

This model is consistent with both Expert C's relay authentication
(the relay has DK/DEK and participates in Noise KK) and the Savant's
pseudo-peer model (the relay does not participate in gossip,
convergence, or dominance). The relay's role in replication is
passive: it stores ciphertext, serves GET requests, and accepts PUT
requests. It does not make replication decisions.

**The 30-second amplification prevention interaction with Background
mode:** As noted in Tension 2, the relay's 30-second amplification
prevention window is exactly at the Background mode heartbeat
interval boundary. The relay keepalive probes (2-second interval per
Expert C's Problem 4) resolve this: the keepalive probe is an
authenticated frame that refreshes the relay's forwarding entry.
Background mode heartbeats at 30 seconds are supplemented by
keepalive probes at 2 seconds. The relay always has a recent
authenticated frame. The amplification prevention window is never
reached.

However, this means the relay keepalive probes are a fixed cost even
in Background mode. At 182 bytes/second per session and 10 relayed
sessions, the total keepalive overhead is 1,820 bytes/second (14.6
kbps). On a 5 Mbps uplink, this is 0.29%. Negligible. But the
keepalive cost does not scale down with metabolic state. In
Background mode (minimal resource usage), the relay keepalive
is the dominant bandwidth consumer. This is acceptable because relay
connectivity is a liveness requirement: if the relay path goes
away, the peer loses connectivity to all relay-dependent clusters.

## Part 5: What the Savant Taught Me

### 1. The fixed-point boundary is universal.

I initially treated the fixed-point boundary as a concern for health
scores and dominance computation. The Savant's systematic audit
revealed that the boundary applies to every computation that crosses
peers: scar decay, EMA replay, capacity ratio, and -- critically for
my domain -- the replication probability function. Every cross-peer
computation must be deterministic across platforms. Fixed-point is
the mechanism for achieving this. I have updated my replication
probability function to use Q16.16 fixed-point.

### 2. The alternating Merkle root in heartbeats.

The Savant's scheme of alternating filter_root (even heartbeats) and
stash_root (odd heartbeats) reduces per-heartbeat overhead while
maintaining anti-entropy properties. I had not considered this
optimization. It doubles the divergence detection latency but halves
the heartbeat extension size, which is a good tradeoff for the
household mesh where bandwidth conservation matters.

### 3. Channel-as-species monotonicity.

The Savant's observation that the entire crystal is monotonic (state
grows, weight decays, nothing is deleted) is a powerful
characterization. This monotonicity is what makes the system
convergent without coordination, per the CALM theorem (Hellerstein,
2010). Every CRDT in the system (GSet, G-Counter, max-register) is
monotonic. Channel creation is monotonic (channels are never
deleted). Scar entries are monotonic (GSet). Even the scar decay
is monotonic in a sense: the weight function is a pure function of
time, applied at read-time, not a mutation of the underlying GSet.
The Savant's crystallographic lens captures this property elegantly.

### 4. The EDT scope_flags interaction with dynamic channels.

I had not identified the interaction between EDT scope_flags (u64
bitmask) and dynamic channel creation. The Savant's observation
that EDT tokens are signed and immutable means the bitmask cannot be
updated to include new channels. This is a real constraint on the
channel lifecycle. The resolution (EDT scope covers only channels
that existed at EDT creation time; admin must reissue for new
channels) is straightforward but should be explicitly documented.

### Where the Savant is wrong

I addressed the primary disagreement in my preamble (replication
budget = 0 during Active/Stress). The Savant's rule creates a
post-convergence deadlock that my combined approach avoids.

I also believe the Savant's two-phase polling model (feed/poll_output)
is slightly less clean than Expert B's four-method model, though
both are workable. The Savant's unified poll_output requires the
consumer to branch on output type, which is trivial at this scale
but less elegant than Expert B's type-separated drain loops.

The Savant's relay health model (separate EMA track, keyed by relay
address) is good but does not address the interaction with the
polygon schedule. When relay health degrades during convergence, the
polygon schedule must handle the failure. The Savant says the polygon
schedule treats late-arriving pairs as still-partitioned, but this
means the pair remains unconverged after convergence, which is not a
failure the Savant proposes to handle. I recommend adding a polygon
retry mechanism for relay-dependent pairs.

## Part 6: Updated Ideal Design

### 6.1 Replication protocol

The combined priority-queue + probabilistic approach described in
Tension 4. Key properties:

- Priority ordering: most under-replicated first
- Probabilistic deduplication: BLAKE3 roll, coordination-free
- Cluster-diversity placement: at least 2 clusters for target >= 3
- Fixed-point arithmetic: Q16.16 for all computations
- Non-zero budget in all metabolic states: minimum 1 chunk per tick
- Relay-held replicas count toward targets
- Post-convergence catch-up: budget increases as metabolic state
  decays to Idle

### 6.2 Intra-cluster communication

DEK-based Noise KK, one session per peer pair, as proposed in
Stage 1. Incorporating the Savant's additions:

- 1-second fixed heartbeat interval (LAN, no metabolic variability)
- Simplified health: single heartbeat counter, no dual-EMA
- Resource accounting: 10KB per peer, 60KB max
- dest_peer: sequential u16 from enrollment, never reused

### 6.3 Gateway handoff protocol

Four-phase handoff with overlap window:

1. Prepare: transfer CPSK session state and observation summaries
2. Establish: new gateway creates inter-cluster Noise KK sessions
3. Migrate: traffic moves to new gateway's sessions
4. Teardown: old gateway tears down old sessions

This is more complex than simple teardown but preserves continuity
for in-flight critical frames (KeyBundle, convergence gates).

### 6.4 Relay model

Pseudo-peer with DK/DEK, no CIK. Relay-specific enrollment
(token_type 0x04). Hash ring position for territorial re-encryption
(territory assigned to nearest non-relay peer). Relay-held replicas
count toward replication targets with cluster-diversity benefit.
Relay keepalive probes (2-second interval) satisfy amplification
prevention window.

### 6.5 Connectivity strategy

Relay-first, direct-upgrade. TransitRelay invariant 3 restated:
"transit relay provides immediate connectivity; direct paths are
preferred when available and are established in background."
Amplification prevention window: 35 seconds (not 30).

### 6.6 Filter reconciliation timing

After ControlGate opens (Front 1 complete), concurrent with
Fronts 2-3, asynchronous completion. No blocking gate. Alternating
Merkle roots in heartbeats (Savant's optimization). Steady-state
anti-entropy continues after convergence exit.

### 6.7 Channel lifecycle

Admin-only creation. Flat u16 indices, monotonically assigned.
No deletion (GSet monotonicity). CHANNEL_CREATE control log entry
with embedded encrypted key material (atomic creation + initial
key distribution). EDT scope covers only channels that existed at
EDT creation time.

### 6.8 Convergence polygon retry

New addition: the polygon schedule includes a retry mechanism for
pairs whose CPSK renegotiation fails due to relay failure or
transient connectivity loss. After the initial polygon pass, failed
pairs are re-queued for a retry pass. Maximum 2 retry passes before
the pair is treated as persistently partitioned.

### 6.9 Fixed-point replication decision

All replication decision computations use Q16.16 fixed-point:

```
fn should_replicate_q16(
    blob_id: BlobId,
    known_holders: u8,
    target_replicas: u8,
    my_dk: &[u8; 32],
    epoch: u64,
) -> bool {
    if known_holders >= target_replicas {
        return false;
    }
    let deficit = target_replicas - known_holders;
    let probability_q16 = ((deficit as u32) << 16)
        / (known_holders as u32 + 1);
    let hash = BLAKE3(
        DomainSeparator::ReplicationRoll,
        blob_id || my_dk || epoch.to_le_bytes()
    );
    let roll_q16 = (hash[0] as u32) << 8;
    roll_q16 < probability_q16
}
```

DomainSeparator::ReplicationRoll = 9 with string "saalis-repl-v1",
added to the existing domain separator table.
