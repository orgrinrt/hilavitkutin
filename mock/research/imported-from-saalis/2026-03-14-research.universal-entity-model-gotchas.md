# Universal Entity Model: Risks, Gotchas, and Mitigations

**Date:** 2026-03-14
**Context:** Saalis content library manager -- everything is an entity (games, movies, users, downloads, sessions). Two anchor tables (`entity_types`, `entities`). All actual data lives in typed metadata tables referencing entities via FK. Entity identity emerges from *which* metadata tables have rows for a given entity, not from the entity row itself.

---

## 1. Classic EAV Problems

The Entity-Attribute-Value model has been called "the most common database anti-pattern" for good reason. In its pure form -- a single table with three columns (entity_id, attribute_name, attribute_value) -- it creates a cascade of interrelated problems that database experts have catalogued for decades.

### 1.1 Query Complexity and Join Explosion

The fundamental issue is *pivoting*. In a traditional relational schema, selecting a product with its name, price, and description is a single-table query. In EAV, each attribute becomes a separate row, requiring self-joins to reconstruct what should be a single row. Selecting entities based on one attribute value requires two joins; two attributes require four joins. For a product page that needs 15 attributes, you are looking at 30 joins or a complex pivot query ([source](https://www.cybertec-postgresql.com/en/entity-attribute-value-eav-design-in-postgresql-dont-do-it/)).

### 1.2 Loss of Database Constraints

Relational databases enforce data integrity through column types, NOT NULL constraints, CHECK constraints, UNIQUE constraints, and foreign keys. In an EAV table, the `value` column must accommodate every data type, so it is typically `TEXT` or `VARCHAR`. The database engine itself cannot perform robust input validation -- a "price" attribute could contain "hello" and the database would accept it. All validation logic migrates into the application layer, which is fragile and easy to bypass ([source](https://en.wikipedia.org/wiki/Entity%E2%80%93attribute%E2%80%93value_model)).

### 1.3 Indexing Challenges

Standard B-tree indexes work well when columns have known types and predictable cardinality. An EAV value column is polymorphic: it contains dates, numbers, strings, and booleans all mixed together. Range queries (`WHERE price > 10.00`) become impossible to index efficiently because the column stores text representations. Even if you split values by type (as Magento does), you still need composite indexes on `(entity_id, attribute_id)` on *every* type table ([source](https://www.sqlservercentral.com/forums/topic/entity-attribute-value-is-evil-how-to-avoid)).

### 1.4 Reporting Difficulty

Reporting and analytics tools assume tabular data: columns with names and types. EAV data must be pivoted into tabular form before any reporting tool can consume it. This pivot operation is expensive, hard to make generic, and often requires the report author to know all possible attribute names in advance.

### 1.5 The Inner Platform Effect

The most damning critique of EAV is the "inner platform effect" -- the tendency to build a poor replica of the development platform on top of itself. An EAV schema is, at its core, a database inside a database. You have re-implemented tables (entities), columns (attributes), and rows (values) using the database's own tables, columns, and rows, but without any of the database engine's optimizations for those constructs. As [The Daily WTF](https://thedailywtf.com/articles/The_Inner-Platform_Effect) describes it: "you are trying to build a database engine inside your database engine." The result is universally slower, harder to query, and more complex than the thing it replaced.

## 2. Our Variant vs Pure EAV

Saalis does *not* use pure EAV. The critical distinction: instead of a single `(entity_id, attribute, value)` table, we use **typed metadata tables**, each with proper columns, proper types, and proper constraints. A `game_metadata` table has typed columns for `title TEXT NOT NULL`, `release_year INTEGER`, `developer TEXT`. A `user_metadata` table has `username TEXT UNIQUE NOT NULL`, `email TEXT`, `role INTEGER NOT NULL`.

### 2.1 What This Preserves

- **Column-level type safety.** Each metadata table enforces types, NOT NULL, UNIQUE, CHECK, and FOREIGN KEY constraints natively. The database engine does what it was designed to do.
- **Meaningful indexes.** You can create an index on `game_metadata.release_year` and it indexes actual integers, not text blobs.
- **Standard query patterns within a metadata table.** Querying "all games released after 2020" is `SELECT entity_id FROM game_metadata WHERE release_year > 2020` -- no pivoting, no self-joins.
- **Schema documentation.** Each table's DDL is self-documenting. A developer can read `game_metadata` and understand the shape of a game without consulting application code.

### 2.2 What This Does NOT Solve

- **Cross-table reconstruction.** To get "all data about entity X", you still must join across every metadata table that might have rows for that entity. This is the N-join problem, just with N metadata tables instead of N attributes.
- **Polymorphic queries.** "Show me all entities sorted by name" requires knowing that games store names in `game_metadata.title`, users store names in `user_metadata.username`, and movies store names in `movie_metadata.title`. There is no generic `name` column to sort by.
- **Schema awareness.** Any code that handles "an entity" generically must know the full set of metadata tables. Adding a new metadata table means updating every place that reconstructs full entities.
- **The anchor table is still a god table.** The `entities` table contains one row per entity of any type. It is a single point of contention and, if it has few columns of its own, it is essentially just an ID allocator.

### 2.3 Where We Sit on the Spectrum

The design sits between pure EAV and pure relational. Compared to pure EAV, we regain most of the database engine's power (types, constraints, indexes). Compared to pure relational (separate, unrelated tables for each domain concept), we gain a universal identity system and the ability to write code that is entity-type-agnostic. This is a legitimate trade-off, but it is still a trade-off, and the remainder of this document explores where it can bite.

## 3. Query Performance Implications

### 3.1 The N-Join Problem at Scale

Consider loading a game entity's full profile: entity base row + game_metadata + ratings + media_assets + credits + download_status. That is 5-6 JOINs. With 100k entities, each join must seek through an indexed foreign key on `entity_id`. If the metadata tables are reasonably sized (100k rows each with indexed FKs), each join is an index lookup -- O(log N) per join, totalling roughly 5-6 index lookups. On SQLite with data in the page cache, this is sub-millisecond.

The problem arises with *list queries*: "show me 50 games sorted by rating, filtered by year, with thumbnails." This requires joining `game_metadata` (for year filter), `ratings` (for sort), and `media_assets` (for thumbnails) across 50 entities. The query planner must navigate three index lookups per entity, times 50. On SQLite, this is still fast at 100k entities -- but it becomes fragile. If any of these tables grow to millions of rows without proper indexes, a single missing index turns a 5ms query into a 500ms table scan.

### 3.2 Indexing Strategies

**Covering indexes** are the single most impactful optimization for this pattern. A covering index on `game_metadata(entity_id, release_year, title)` lets SQLite answer "games released after 2020 with their titles" without ever touching the main table. The query planner "simply walks the index from one end to the other and delivers the output in time proportional to N" ([SQLite docs](https://sqlite.org/queryplanner.html)).

**Composite indexes** on foreign keys must follow the "left to right, no skipping" rule. An index on `(entity_id, attribute_a, attribute_b)` supports lookups by entity_id alone, or by (entity_id + attribute_a), but *not* by attribute_b alone.

**Foreign key indexes** are not automatically created in SQLite. Every metadata table's `entity_id` column needs an explicit index. Without it, joins degrade to full table scans.

### 3.3 SQLite-Specific Concerns

SQLite does not use AND-by-INTERSECT to leverage multiple indexes in AND-connected WHERE clauses. If you filter by both `release_year > 2020 AND developer = 'Nintendo'`, SQLite will use *one* index and scan the rest. A composite index on `(release_year, developer)` or `(developer, release_year)` is required, and the order matters based on selectivity ([source](https://sqlite.org/optoverview.html)).

`EXPLAIN QUERY PLAN` is essential. Every new query pattern against metadata tables should be verified to ensure the expected indexes are being used.

## 4. Schema Evolution Trade-offs

### 4.1 The Easy Part

Adding a new entity type is trivial: create a new metadata table, register its entity type. No existing tables are modified. No migrations touch existing data. This is the primary selling point of the pattern, and it is genuinely powerful. In a traditional relational schema, adding a new domain concept often requires ALTER TABLE on existing tables or complex migration scripts.

### 4.2 The Hard Part

Every piece of code that needs "all data about an entity" must be updated when a new metadata table is added. This manifests in several ways:

- **Generic entity serialization** (e.g., API responses) must know about all metadata tables.
- **Search and indexing** must be updated to include new metadata tables in full-text search.
- **Backup and export** routines must enumerate all metadata tables.
- **Schema introspection** -- the application must maintain a registry of "which metadata tables exist and what entity types they serve."

### 4.3 How Others Handle This

**Notion** solves this by having *one* block type with a flexible property bag. All blocks share the same schema; the "type" field determines which properties are meaningful. This avoids the N-table problem but sacrifices type safety ([source](https://www.notion.com/blog/data-model-behind-notion)).

**Salesforce** uses a Universal Data Dictionary (UDD) where custom objects and fields are metadata entries, not physical tables. When you create a custom object, "the platform does not create an actual table in a database, but instead stores some metadata that it can then use at runtime to dynamically materialize virtual application components." This is the inner platform effect executed at massive scale with enormous engineering investment ([source](https://architect.salesforce.com/fundamentals/platform-multitenant-architecture)).

**Our approach** (compile-time registration via `#[register]` + `inventory`) means the set of metadata tables is known at compile time. This eliminates runtime discovery costs but means schema changes require recompilation. For an appliance binary, this is acceptable. For a plugin system, it means plugins must be compiled against the current schema.

## 5. Successful (and Cautionary) Implementations

### 5.1 Magento -- The Canonical Cautionary Tale

Magento is the most cited example of EAV causing real-world pain. Products are stored across separate tables per data type per entity type: `catalog_product_entity_int`, `catalog_product_entity_varchar`, `catalog_product_entity_decimal`, etc. Loading a single product requires 5-7 JOINs. For a catalog listing page showing 50 products with 20 attributes each, the query complexity is staggering ([source](https://yegorshytikov.medium.com/magento-eav-database-architecture-20b25c09af98)).

Magento attempted to mitigate this with "flat index tables" -- precomputed denormalized tables for catalog and category data. This worked but created a new problem: maintaining flat indexes required cron jobs that became long-running at scale, and the flat tables fell out of sync with EAV tables during high-write periods. Adobe eventually advised against flat indexes for Magento 2.1+ ([source](https://meetmagentoacademy.github.io/magento2-training-resources/backend/performance/eav.html)).

**Lesson:** Type-split EAV tables reduce type-safety problems but multiply the join problem. Denormalization caches are effective but introduce consistency challenges.

### 5.2 WordPress wp_postmeta -- Death by a Thousand Rows

WordPress stores post metadata in a single `(post_id, meta_key, meta_value)` table. For WooCommerce, each product has approximately 29 metadata rows. A store with 10,000 orders generates 300,000+ postmeta rows just for order data. Store owners reported 30+ second page loads for the orders admin page ([source](https://core.trac.wordpress.org/ticket/30814)).

The fix was radical: WooCommerce 6.9 introduced High-Performance Order Storage (HPOS), migrating order data from postmeta to dedicated custom tables with proper columns and types. The results: **5x faster order creation** and **40x faster backend filtering**. HPOS is enabled by default for new WooCommerce installations since version 8.2 ([source](https://developer.woocommerce.com/docs/features/high-performance-order-storage/)).

**Lesson:** Pure EAV (key-value postmeta) is a performance trap that eventually requires migration to typed tables. This validates our typed-metadata-table approach over pure EAV, but also shows that even "typed tables" can fail if the underlying pattern requires too many joins.

### 5.3 Notion -- Everything is a Block (200 Billion of Them)

Notion stores every piece of content as a "block" row in PostgreSQL. Pages, text paragraphs, images, database rows, embeds -- everything is a block with a type field and a properties column. By early 2021, Notion had 20 billion blocks. By mid-2024, over 200 billion. They sharded PostgreSQL to 480 logical shards across 96 physical instances ([source](https://www.notion.com/blog/sharding-postgres-at-notion)).

Notion works despite the universal-entity pattern because of massive infrastructure investment: PostgreSQL sharding by workspace ID, PgBouncer connection pooling, a dedicated data lake (Apache Hudi on S3) for analytics, and extensive application-layer caching. Users *do* experience slowdowns with large databases -- a known limitation that Notion continues to optimize ([source](https://latwy.co/knowledge-base/why-is-notion-slow-with-large-databases)).

**Lesson:** The universal-entity pattern can work at extraordinary scale, but it requires proportional infrastructure investment. Notion's "block" is closer to our pattern than pure EAV: blocks have typed properties within their type, not arbitrary key-value pairs.

### 5.4 Salesforce -- The Inner Platform, Industrialized

Salesforce stores all tenant data in shared "heap" tables (MT_Data, MT_Data_0 through MT_Data_N) with columns named Value0 through Value500. Custom objects do not get physical tables; they are metadata entries in the Universal Data Dictionary. This is EAV taken to its logical extreme, but with billions of dollars of engineering behind it: custom query optimizer, metadata cache layer, automatic indexing, tenant-scoped partitioning ([source](https://cirra.ai/articles/salesforce-database-architecture-explained)).

**Lesson:** You *can* build the inner platform, but only if your entire business model is the inner platform. For an appliance binary running on a Raspberry Pi, this is not a viable reference architecture.

### 5.5 Datomic -- EAV Done Right?

Datomic stores all data as immutable (entity, attribute, value, transaction) tuples -- "datoms." This is pure EAV, but with three key differences: immutability (no updates, only additions and retractions), transaction-time tracking (every fact knows when it was asserted), and mandatory attribute schemas (value types and cardinalities must be declared). Datomic maintains four index trees sorted by EAVT, AEVT, AVET, and VAET, enabling efficient access patterns across all query shapes ([source](https://docs.datomic.com/datomic-overview.html)).

**Lesson:** EAV can be a principled data model if you accept its constraints and build storage specifically for it. Datomic does not try to make a relational database behave like an EAV store; it built a purpose-specific storage engine. We are using SQLite, a relational engine, which means we must work *with* relational patterns, not against them.

### 5.6 Airtable -- Flexible Schema on MongoDB

Airtable uses MongoDB rather than a relational database, sidestepping the EAV join problem entirely. Each "base" is a document collection where records have flexible schemas. There are no joins; related data is embedded or linked at the application layer ([source](https://community.airtable.com/development-apis-11/what-database-used-in-airtable-3458)).

**Lesson:** If your data model is inherently flexible and document-shaped, a document store avoids the relational impedance mismatch entirely. Our choice of SQLite commits us to relational patterns and their associated costs.

## 6. Performance Mitigations

### 6.1 Composite / Cached Aggregation Tables

Our planned `Composites` metadata table is a denormalized read cache: frequently-accessed attributes from multiple metadata tables precomputed into a single row per entity. This eliminates the N-join problem for the most common query patterns (list views, search results, thumbnails).

**Trade-off:** Every write to a source metadata table must update the composite. This can be synchronous (slow writes, consistent reads) or asynchronous (fast writes, eventually consistent reads). Our fire-and-forget scheduler with work units naturally supports the async approach.

**Risk:** Composite staleness. If the composite update fails or is delayed, the UI shows stale data. For a personal content library, brief staleness is acceptable. For user-facing data like download progress, it may not be.

### 6.2 Materialized Views / Precomputed Joins

SQLite does not natively support materialized views. The equivalent is a physical table populated by a trigger or a periodic rebuild. This is functionally identical to the composites approach above, with the same trade-offs. PostgreSQL and MySQL users have better options here, but we are committed to SQLite.

### 6.3 Column-Oriented Storage for Hot Paths

The planned hilavitkutin integration provides columnar storage for analytics and bulk queries. This sidesteps the row-oriented join problem entirely: when you need "all ratings sorted by score", a columnar scan of the ratings column is O(N) with excellent cache locality, compared to O(N log N) random-access joins in row storage.

**Risk:** Synchronization between SQLite (source of truth) and the columnar store. Writes go to SQLite; the columnar store must be populated from SQLite data. This is a form of CQRS (Command Query Responsibility Segregation), which introduces eventual consistency ([source](https://learn.microsoft.com/en-us/azure/architecture/patterns/cqrs)).

### 6.4 CQRS / Denormalized Read Models

The pattern of "normalize for writes, denormalize for reads" is the canonical mitigation for the EAV performance problem. Our architecture naturally supports this: writes go through the scheduler as work units, reads come from composites or the columnar store. The key discipline is ensuring that every write path has a corresponding read-model update path.

### 6.5 Smart Indexing

Practical checklist for our metadata tables:

1. **Every `entity_id` foreign key column** must have an explicit index. SQLite does not auto-index FKs.
2. **Covering indexes** for list-view queries: `CREATE INDEX idx_game_list ON game_metadata(release_year, entity_id, title)` lets the "browse by year" view never touch the main table.
3. **Partial indexes** for common filters: `CREATE INDEX idx_active_downloads ON download_metadata(entity_id) WHERE status = 'active'` speeds up the active-downloads dashboard.
4. **ANALYZE after bulk imports** so the query planner has up-to-date statistics in the `sqlite_stat1` table.

## 7. Type Safety Mitigations

### 7.1 The Problem

When "everything is an entity," the type system must answer: "What metadata tables are valid for this entity type?" Without guardrails, code that operates on a `User` entity might accidentally query `game_metadata`, returning zero rows silently instead of failing loudly.

### 7.2 Our Approach: Compile-Time ZST Column Types

The planned architecture uses zero-sized types (ZSTs) as compile-time markers:

- `SupportsEntity<Game>` bounds on metadata table types ensure that `game_metadata` operations can only be invoked for game entities.
- `HasMetadata<GameMetadata>` bounds on entity types ensure that code expecting game metadata only compiles if the entity type declares it.
- `Column<In<GameMetadata>, As<Text>>` types make column references type-safe: you cannot accidentally use a column from one metadata table in a query against another.

This pushes schema validation from runtime to compile time. The cost is verbosity and compile-time complexity. The benefit is that invalid metadata access is a compile error, not a silent empty result.

### 7.3 How Others Handle This

**Magento** handles type safety in PHP with abstract model classes and getter methods. It is entirely runtime-checked, and attribute-name typos are a common source of bugs.

**Datomic** requires attribute schemas to be installed before use, with explicit value types and cardinalities. Schema violations are transaction-time errors, not compile-time errors, but they are at least loud.

**Salesforce** enforces types through its metadata layer: custom fields have declared types, and the platform rejects type-mismatched data at the API level.

**Our advantage** is Rust's type system. No other production EAV-like system has compile-time trait bounds preventing invalid metadata access. This is a genuine differentiator that mitigates one of EAV's biggest risks.

## 8. The "God Table" Anti-Pattern

### 8.1 When Everything Becomes a Row

The "god table" anti-pattern occurs when a single table tries to represent all concepts in the system. In its worst form, it is a table with columns like `id, type, name, value1, value2, value3, ...` where different types use different columns. It leads to sparse data, meaningless column names, and impossible constraints ([source](https://hackernoon.com/an-overview-of-sql-antipatterns)).

### 8.2 Our `entities` Table: God Table or Identity Allocator?

Our `entities` table contains: `entity_id` (PK), `entity_type_id` (FK to `entity_types`), and possibly creation timestamp. It has no domain-specific columns. It is not a god table in the traditional sense -- it does not store attributes of different types in shared columns. It is an **identity allocator**: its job is to assign unique IDs and track type membership.

However, it shares one property with the god table: *every entity in the system has a row here*. This means:

- **Contention on inserts.** Every new entity, regardless of type, inserts into this table. On SQLite (single-writer), this is sequential by design. At our scale (10k-100k entities, not millions per second), this is not a problem.
- **Meaningless aggregate queries.** `SELECT COUNT(*) FROM entities` tells you nothing useful. You almost always need to filter by type or join to a metadata table.
- **Orphan risk.** If a metadata row is deleted but the entity row remains, the entity becomes a ghost -- an ID with no data. Cascading deletes or application-level cleanup is required.

### 8.3 Staying on the Right Side

The line between "universal identity system" and "god table" is:

1. **The anchor table stores identity, not data.** If you start adding `name`, `description`, or `status` columns to `entities` because "every entity has one," you are sliding toward the god table. Each entity type should define its own name/description in its own metadata table.
2. **Joins are always to specific metadata tables, not to the anchor.** If your queries start with `SELECT * FROM entities JOIN ... JOIN ... JOIN ...` you are using the anchor correctly. If they start with `SELECT * FROM entities WHERE type = 'game' AND ...`, you are treating the anchor as a data table.
3. **The anchor table stays small.** Two or three columns, no nullable "optional" columns, no type-specific data.

## 9. Alternatives Considered

### 9.1 Table-Per-Type (Classical Relational)

Each domain concept gets its own table: `games`, `users`, `movies`, `downloads`. No shared identity system.

**What we lose by not using this:**
- Simpler queries (no entity_id indirection).
- Standard ORM support.
- Foreign keys between domain tables are straightforward.
- Schema is immediately comprehensible from DDL alone.

**What we gain by not using this:**
- Cross-type operations (tagging, permissions, audit logs) require polymorphic associations, which have their own well-documented problems -- notably, foreign keys cannot reference multiple parent tables, so referential integrity must be enforced in application code ([source](https://www.oreilly.com/library/view/sql-antipatterns/9781680500073/f_0043.html)).
- Adding a new entity type requires creating tables *and* modifying every cross-cutting system.

### 9.2 Single Table Inheritance (STI)

One table for all entity types, with a discriminator column and nullable columns for type-specific attributes. Used by many Rails applications.

**What we lose by not using this:**
- Simplest possible query pattern: one table, one query.
- No joins for basic entity retrieval.

**What we gain by not using this:**
- STI tables become massively wide and sparse as types diverge. A game has `release_year` but not `email`; a user has `email` but not `release_year`. With 10 entity types and 15 attributes each, the table has 150+ columns, most null for any given row.
- Adding attributes to one type adds columns that are meaningless for all other types.
- Constraints cannot distinguish "required for games but not for users" without CHECK constraints referencing the discriminator, which are complex and fragile.

### 9.3 JSON Columns

Store type-specific data in a JSON/JSONB column: `entities(id, type, data JSON)`.

**What we lose by not using this:**
- Flexible per-row schema without multiple tables.
- Single-table queries for full entity reconstruction.
- Good support in PostgreSQL (JSONB with GIN indexes).

**What we gain by not using this:**
- SQLite's JSON support is functional but limited compared to PostgreSQL. JSON path queries cannot use standard B-tree indexes; they require generated columns or expression indexes.
- No column-level constraints, types, or NOT NULL enforcement inside JSON.
- Query syntax is non-standard and database-specific.
- JSON columns are effectively EAV in disguise, with the same loss of database-enforced integrity.

### 9.4 Document Stores (MongoDB, CouchDB)

Abandon relational storage entirely. Each entity is a document with flexible schema.

**What we lose by not using this:**
- Natural fit for entities with varying attributes.
- No join problem: all data for an entity is in one document.
- Horizontal scaling is built-in.

**What we gain by not using this:**
- SQLite is embedded, zero-config, single-file. MongoDB requires a separate server process, which conflicts with our "single binary on Batocera" deployment target.
- Relational integrity between entities (e.g., "this download belongs to this game") must be enforced in application code.
- ACID transactions across multiple documents are complex in document stores.

## 10. Specific Risks for Our Use Case

### 10.1 The Content List View

The most common operation in a content library is "show me a paginated list of games, sorted by name, filtered by genre, with thumbnails and ratings." This query touches:

- `game_metadata` (name, genre filter)
- `ratings` (sort by rating)
- `media_assets` (thumbnail URL)

Three JOINs for every row in the result set. At 10k games, this is manageable with proper indexes. At 100k games with complex filters, it becomes the kind of query that looks fast in development (50ms) and slow in production (500ms) because the query planner makes different choices with different data distributions.

**Mitigation:** The composites table pre-joins these fields. The list view queries composites, not the source tables.

### 10.2 Search

Full-text search across entity names, descriptions, and tags requires data from multiple metadata tables. SQLite's FTS5 extension operates on a single table. You must either:

- Maintain a separate FTS5 table populated with data from all metadata tables (another denormalized read model), or
- Run multiple FTS5 queries (one per metadata table) and merge results in application code.

**Mitigation:** A unified search index table, populated by the scheduler when metadata changes.

### 10.3 "What Is This Entity?"

Given an entity_id with no type hint, determining what the entity *is* requires either:

- Querying the `entities` table for its type (fast, single lookup), or
- Probing each metadata table to see which ones have rows for it (slow, N queries).

If the `entity_type_id` column in `entities` is always accurate, this is trivial. But if type is determined by "which metadata tables have rows," then type is an emergent property that can become inconsistent -- a game entity might have a `user_metadata` row due to a bug, and there is no constraint preventing this.

**Mitigation:** The `entity_type_id` column in `entities` is authoritative. `SupportsEntity<E>` trait bounds prevent writing metadata to incompatible entity types at compile time.

### 10.4 Cascading Deletes

Deleting an entity requires deleting rows from every metadata table that might reference it. Miss one, and you have orphaned data. With 15 metadata tables, that is 15 DELETE statements (or relying on `ON DELETE CASCADE` foreign keys, which SQLite supports but does not enforce unless `PRAGMA foreign_keys = ON`).

**Risk:** Forgetting `PRAGMA foreign_keys = ON` at connection startup. This is a notorious SQLite gotcha -- foreign keys exist in the schema but are silently unenforced by default.

**Mitigation:** Set `PRAGMA foreign_keys = ON` at connection initialization, verified by a startup check. Wrap entity deletion in a transaction that deletes from all registered metadata tables.

### 10.5 Migration Complexity

When a metadata table schema changes (e.g., adding a column to `game_metadata`), the migration is straightforward -- standard ALTER TABLE. But when the *entity model* changes (e.g., splitting "media" entities into "movies" and "TV shows"), you must:

1. Create new metadata tables (`movie_metadata`, `tvshow_metadata`).
2. Migrate data from the old table to the new tables.
3. Update `entity_type_id` in the anchor table for affected entities.
4. Update all composite/cache tables.
5. Update all application code that referenced the old type.

This is not unique to our pattern -- any schema refactoring is complex. But the universal entity model adds step 3 (anchor table updates) and step 4 (cache invalidation across multiple denormalization layers).

### 10.6 Write Amplification

Every entity creation involves:

1. INSERT into `entities` (anchor row).
2. INSERT into one or more metadata tables.
3. UPDATE composites table.
4. UPDATE search index.
5. UPDATE columnar store (hilavitkutin).

A single logical write becomes 4-5 physical writes. On SQLite's single-writer model, this serializes writes across all entity types. At low throughput (a personal content library, not a multi-tenant SaaS), this is fine. But bulk imports (e.g., scanning a ROM collection of 10,000 games) will be I/O bound.

**Mitigation:** Batch writes in transactions. Defer composite and search index updates to the scheduler. Use WAL mode for concurrent reads during writes.

## Summary of Risk-Mitigation Matrix

| Risk | Severity | Mitigation |
|---|---|---|
| N-join query complexity | Medium | Composites table, covering indexes |
| Missing FK indexes on metadata tables | High | Explicit index creation checklist, startup validation |
| Polymorphic queries across types | Medium | Type-specific query APIs, no generic "all entities" endpoint |
| Schema awareness (knowing all metadata tables) | Medium | Compile-time `#[register]` + `inventory` |
| Inner platform effect | Low | Typed metadata tables, not key-value; no runtime schema creation |
| God table anchor | Low | Keep anchor to ID + type only, never add domain columns |
| Orphaned entities / ghost data | Medium | `PRAGMA foreign_keys = ON`, cascading deletes, startup integrity check |
| Write amplification on bulk import | Medium | Batch transactions, deferred index updates, WAL mode |
| Search across entity types | Medium | Unified FTS5 search index table |
| Composite staleness | Low | Fire-and-forget scheduler with retry, acceptable staleness window |
| SQLite PRAGMA foreign_keys off by default | High | Connection initialization check, fail-fast on missing pragma |
