# Deep Dive: Cross-Compilation and CI/CD for ARM Deployment

## Context

Saalis ships as a single Rust binary targeting Batocera appliances (primarily aarch64 / ARM64), built from x86_64 development machines and CI runners. This document covers the full pipeline from cross-compilation tooling through release automation and profile-guided optimization.

---

## 1. cross-rs: Docker-Based Cross-Compilation

[cross-rs](https://github.com/cross-rs/cross) provides a drop-in replacement for `cargo` that runs compilation inside Docker (or Podman) containers pre-configured with the correct cross-toolchain, sysroot, and system libraries for each target. The CLI is identical to Cargo -- `cross build --target aarch64-unknown-linux-gnu` just works, with no manual toolchain setup required.

### Custom Dockerfiles via Cross.toml

When a project needs system-level dependencies (e.g., OpenSSL headers, libfuse, udev), cross supports custom Dockerfiles and pre-build hooks through `Cross.toml`:

```toml
[target.aarch64-unknown-linux-gnu]
pre-build = [
    "dpkg --add-architecture $CROSS_DEB_ARCH",
    "apt-get update && apt-get --assume-yes install libssl-dev:$CROSS_DEB_ARCH"
]
```

Alternatively, a fully custom Dockerfile can be specified:

```toml
[target.aarch64-unknown-linux-gnu.dockerfile]
file = "docker/Cross.aarch64.Dockerfile"
context = "."
build-args = { EXTRA_PACKAGES = "libsqlite3-dev" }
```

Cross provides the `CROSS_BASE_IMAGE` build argument automatically, so custom Dockerfiles can extend the official image with `FROM $CROSS_BASE_IMAGE`.

### Cross-Testing via QEMU

cross supports running `cargo test` on non-native targets using QEMU user-mode emulation. For aarch64, the default runner is `qemu-user`. Tests run inside the Docker container with binfmt_misc transparently routing ARM binaries to QEMU. The runner can be overridden per-target:

```toml
[target.aarch64-unknown-linux-gnu]
runner = "qemu-user"   # default; also supports "qemu-system" or "native"
```

Setting `QEMU_STRACE=1` provides syscall tracing for debugging test failures that stem from QEMU limitations rather than actual bugs.

### Container-in-Container for CI

When running cross inside a CI container (e.g., `FROM rust:1`), set `CROSS_CONTAINER_IN_CONTAINER=true` so that cross mounts volumes correctly via the host Docker daemon rather than trying to nest containers.

### Saalis Implications

For saalis, the SQLite bundled feature (`libsqlite3-sys` with `features = ["bundled"]`) compiles SQLite from C source during the build. This works seamlessly inside cross's Docker images since the C cross-compiler is already present. No custom Dockerfile should be needed unless additional system dependencies emerge.

## 2. cargo-zigbuild: Zig as a Cross-Linker

[cargo-zigbuild](https://github.com/rust-cross/cargo-zigbuild) uses Zig's cross-compilation infrastructure (specifically its bundled libc and linker) as a backend for `cargo build`. Rather than running builds inside Docker, it runs directly on the host, using Zig purely as the linker/C-compiler.

### Advantages Over cross-rs

- **Build speed:** Real-world reports show builds dropping from 15+ minutes (cross with Docker overhead) to under 2 minutes, because cargo-zigbuild leverages the host's native cargo cache and incremental compilation. No Docker image pull, no container startup, no duplicated compilation state.
- **No Docker dependency:** Simpler CI setup. Works on macOS (including Apple Silicon), Windows, and Linux without Docker. Particularly valuable for local development iteration.
- **glibc version targeting:** cargo-zigbuild allows pinning a minimum glibc version by appending it to the target triple:
  ```
  cargo zigbuild --target aarch64-unknown-linux-gnu.2.28
  ```
  This is critical for Batocera, which ships a specific glibc version. Zig's linker resolves symbols against that exact version, avoiding "GLIBC_2.XX not found" errors at runtime.
- **Simpler Docker images:** When deploying via Docker, the cross-compiled binary can be built on the host and `COPY`-ed into a minimal (`scratch` or `distroless`) image, rather than compiling inside a fat builder image.

### Limitations

- Does not support all targets cross-rs supports (notably MSVC and some exotic embedded targets).
- The musl target (`aarch64-unknown-linux-musl`) works but the glibc version suffix is irrelevant there since musl has its own versioning.
- Some crates with complex build scripts or inline assembly may hit edge cases with Zig's linker.

### Recommendation for Saalis

cargo-zigbuild is the better default for local development cross-compilation and CI builds where Docker is undesirable. Use it with `aarch64-unknown-linux-gnu.2.28` (or whatever glibc version Batocera's target image ships). Fall back to cross-rs if a system dependency requires a full sysroot.

## 3. GitHub Actions Matrix Builds

### Matrix Strategy

A typical CI matrix builds for both x86_64 (for development/testing on standard hardware) and aarch64 (for Batocera deployment):

```yaml
strategy:
  matrix:
    include:
      - target: x86_64-unknown-linux-gnu
        os: ubuntu-latest
      - target: aarch64-unknown-linux-gnu
        os: ubuntu-latest
        cross: true
```

The [`houseabsolute/actions-rust-cross`](https://github.com/houseabsolute/actions-rust-cross) action wraps this pattern: it uses native `cargo` for x86_64 targets and automatically invokes `cross` for ARM targets. By default it integrates `Swatinem/rust-cache@v2` for dependency caching.

### Caching Strategies

**Swatinem/rust-cache:** Caches `~/.cargo/registry`, `~/.cargo/git`, and the `target/` directory. Cache key includes the platform, `Cargo.lock` hash, and rustc version. Simple, effective for most projects. Each matrix entry (target triple) gets its own isolated cache.

**sccache (Shared Compilation Cache):** Wraps `rustc` as a shim, intercepting compilation requests and caching object files. Unlike target-dir caching, sccache caches individual compilation units, providing finer-grained reuse. Configuration with GitHub Actions:

```yaml
- uses: mozilla-actions/sccache-action@v0.0.9
- name: Build
  env:
    SCCACHE_GHA_ENABLED: "true"
    RUSTC_WRAPPER: "sccache"
    SCCACHE_GHA_CACHE_TO: "sccache-${{ matrix.target }}"
    SCCACHE_GHA_CACHE_FROM: "sccache-"
```

sccache stores each cache entry separately in the GHA cache (unlike `@actions/cache` which saves a single large archive). It can also use S3 as a backend for cross-repository sharing, with `SCCACHE_BUCKET` and standard AWS credentials. For pull requests from forks, set `SCCACHE_S3_NO_CREDENTIALS=1` to use a public read-only cache bucket.

**Recommendation:** Use sccache for saalis. It handles cross-compilation caching better than target-dir caching because the host tools (proc-macros, build scripts) share cache entries even when the target changes.

### Important CI Note

As of early 2025, cross-rs does not publish pre-built Linux ARM binary releases. If running on ARM-based CI runners (e.g., GitHub's new `ubuntu-24.04-arm` runners), pin `cross-version` to a specific commit from the cross repo.

## 4. Static Linking: musl vs glibc

### The Core Trade-Off

| Aspect | glibc (gnu) | musl |
|---|---|---|
| Dynamic linking | Default, required for NSS/DNS | Optional |
| Static linking | Not reliably supported | First-class, produces truly static binaries |
| Binary portability | Depends on host glibc version | Runs on any Linux kernel 2.6.39+ |
| Performance | Faster for most workloads | Slightly slower malloc, faster startup |
| DNS resolution | Full NSS support | No NSS; uses own resolver |
| Compatibility | Broad ecosystem support | Rare edge cases with C dependencies |

glibc cannot be statically linked reliably because of its NSS (Name Service Switch) subsystem, which dynamically loads shared libraries at runtime. Attempting static glibc linking produces binaries that still try to `dlopen` at runtime.

### musl for Single-Binary Deployment

Target `aarch64-unknown-linux-musl` to produce a fully static binary with zero runtime dependencies:

```
cross build --target aarch64-unknown-linux-musl --release
# or
cargo zigbuild --target aarch64-unknown-linux-musl --release
```

The resulting binary can run on any Linux system with a compatible kernel -- including Batocera, even if its userspace glibc version differs.

### SQLite Bundled Implications

With `libsqlite3-sys` configured with `features = ["bundled"]`, SQLite is compiled from source and statically linked into the binary. This is the correct approach for musl targets: there is no system SQLite to link against, and bundling guarantees a known SQLite version with the exact compile-time options saalis needs (WAL mode, FTS5, etc.). The bundled SQLite C source compiles cleanly with musl's headers.

### Recommendation for Saalis

Use `aarch64-unknown-linux-musl` for the Batocera release binary. This eliminates all runtime library dependencies, making the binary maximally portable across Batocera versions and Linux distributions. The slight musl performance overhead is negligible for saalis's workload (web server + SQLite, not compute-bound). Use `aarch64-unknown-linux-gnu` only if a future C dependency requires glibc-specific APIs.

## 5. Binary Size Optimization

A single-binary appliance benefits from aggressive size optimization. The full Cargo.toml profile:

```toml
[profile.release]
opt-level = "z"        # optimize for size over speed
lto = "fat"            # whole-program link-time optimization
codegen-units = 1      # single codegen unit for maximum optimization
panic = "abort"        # no unwinding tables (~10% size reduction)
strip = "symbols"      # strip debug symbols from binary
```

### Technique Breakdown

**opt-level = "z" vs "s":** Both optimize for size. `"z"` is more aggressive, sometimes at the cost of runtime performance. `"s"` is a middle ground. For saalis, `"z"` is appropriate since the binary is I/O-bound (HTTP, SQLite), not CPU-bound. If profiling reveals hot loops where `"z"` hurts, consider a per-crate override.

**LTO (Link-Time Optimization):** `"fat"` LTO performs whole-program optimization across all crates, enabling cross-crate inlining and dead code elimination. This both reduces binary size (by eliminating unused code) and improves runtime speed (by 10-20%). The trade-off is significantly longer compile times (2-5x), so use only in release profiles.

**codegen-units = 1:** Forces the compiler to process each crate as a single unit, enabling more optimization opportunities. Default is 16 for parallelism. Setting to 1 improves both size and speed at the cost of compilation time.

**panic = "abort":** Eliminates the unwinding machinery (landing pads, personality functions, `.eh_frame` section). Since saalis runs as a single long-lived process, unwinding is not useful -- a panic should crash and restart via the system supervisor.

**strip = "symbols":** Removes symbol tables and debug info. Can also be done post-build with `strip -s target/release/saalis`. Note: stripped binaries are harder to debug with `perf` or backtraces. Ship a separate debug-info artifact for crash analysis.

### Analysis with cargo-bloat

[cargo-bloat](https://github.com/RazrFalcon/cargo-bloat) measures how much each crate and function contributes to the final binary:

```
cargo bloat --release --crates
cargo bloat --release -n 30   # top 30 largest functions
```

This identifies unexpectedly large dependencies. Common findings: `serde_derive` monomorphization, `regex` automata tables, unused features of large crates. Use `cargo-bloat` before and after optimization changes to quantify impact.

### Real-World Expectations

With all optimizations applied, a typical Rust HTTP server binary compresses to under 2MB -- over 40% smaller than a default release build. For saalis (with SQLite, an HTTP server, and template rendering), expect 3-6MB for the final stripped binary.

## 6. Release Automation

### cargo-dist

[cargo-dist](https://github.com/axodotdev/cargo-dist) is a purpose-built tool for shipping Rust binaries. Running `dist init` in the workspace generates a `release.yml` GitHub Actions workflow that implements the full pipeline: plan, build, host, publish, announce.

Key capabilities:
- Builds release tarballs and platform-specific installers.
- Uploads artifacts to GitHub Releases automatically on tag push.
- Generates CycloneDX SBOM (Software Bill of Materials) files for supply chain transparency.
- Supports cross-compilation targets in the build matrix.
- Handles changelog generation and release notes.

### Versioning Strategy

For an appliance binary, semantic versioning works well: `MAJOR.MINOR.PATCH`. Tag-triggered releases (`v0.1.0`, `v0.2.0`) are the standard pattern. cargo-dist and cargo-release both support this workflow.

[cargo-release](https://github.com/crate-ci/cargo-release) handles the pre-release checklist: bumps version in `Cargo.toml`, creates the git tag, and pushes. Combined with cargo-dist's CI pipeline, the workflow becomes:

```
cargo release patch  # bumps 0.1.0 -> 0.1.1, commits, tags, pushes
# GitHub Actions picks up the tag and runs cargo-dist
```

### Recommendation for Saalis

Use cargo-dist for release automation. It handles the matrix build (x86_64 + aarch64), artifact naming, and GitHub Release creation. Pair with cargo-release for the version bump workflow. Keep a `CHANGELOG.md` and use conventional commits so cargo-dist can auto-generate release notes.

## 7. Testing on ARM

### QEMU User-Mode Emulation

cross-rs integrates QEMU user-mode emulation for running `cargo test` on non-native targets. The workflow is transparent: `cross test --target aarch64-unknown-linux-gnu` builds the test binary for ARM and executes it under `qemu-aarch64`:

- Tests run 5-20x slower than native execution.
- Some tests may fail due to QEMU limitations (incomplete syscall emulation, timing-sensitive tests, mmap behavior differences), not actual bugs.
- `QEMU_STRACE=1` enables syscall tracing for debugging emulation issues.
- Linux `binfmt_misc` kernel support is required (available on all modern CI runners).

### Real Hardware CI Runners

For higher-fidelity testing, GitHub now offers ARM-based runners (`ubuntu-24.04-arm`). These run tests at native speed and eliminate QEMU-related false failures. The trade-off is availability and cost -- ARM runners may have longer queue times.

A practical hybrid: run the full test suite on x86_64 (fast, reliable), run a smoke test subset on aarch64 via QEMU in CI, and run full integration tests on real ARM hardware for release candidates.

### Strategy for Saalis

Since saalis targets Batocera on real ARM hardware, the smoke-test-on-QEMU + full-test-on-native approach is ideal. Unit tests and most integration tests run on x86_64. ARM-specific tests (filesystem paths, endianness, pointer-width assumptions) run via `cross test`. Pre-release validation happens on a physical Batocera device.

## 8. Batocera Package Format

Batocera (v27+) uses **pacman** as its package manager, adopting the same format as Arch Linux's pacman. Packages are `.tar.zst` (preferred, faster decompression) or `.tar.xz` archives containing:

1. The directory structure to overlay onto `/`, starting from root.
2. A `.PKGINFO` metadata file with package name, version, description, architecture, and dependencies.

### Creating a Saalis Package

The package structure would look like:

```
saalis-0.1.0-1-aarch64/
  .PKGINFO
  usr/
    bin/
      saalis
  userdata/
    system/
      configs/
        saalis.toml        # default configuration
  etc/
    init.d/
      S99saalis            # init script for auto-start
```

Build the package with `batocera-makepkg` on a Batocera system, or construct the archive manually:

```bash
# From the package directory:
tar -czf saalis-0.1.0-1-aarch64.pkg.tar.zst \
    --zstd \
    .PKGINFO usr/ userdata/ etc/
```

### Distribution Options

- **Custom pacman repository:** Host package files on a web server or GitHub Pages. Users add the repo to `/userdata/system/pacman/pacman.conf` and install with `pacman -S saalis`.
- **Direct download:** Provide the `.pkg.tar.zst` file on GitHub Releases. Users install locally with `pacman -U /path/to/saalis-0.1.0-1-aarch64.pkg.tar.zst`.
- **Buildroot recipe:** For potential inclusion in Batocera's official distribution, create a Buildroot package recipe (`.mk` file + `Config.in`) under `package/batocera/utils/saalis/`. This compiles from source during the Batocera image build.

### Saalis Recommendation

Start with GitHub Releases distribution (direct `.pkg.tar.zst` download). As adoption grows, set up a custom pacman repository hosted on GitHub Pages or the saal.is domain. The Buildroot recipe is a long-term goal for official Batocera inclusion.

## 9. Reproducible Builds and Supply Chain Security

### cargo-auditable

[cargo-auditable](https://github.com/rust-secure-code/cargo-auditable) embeds the full dependency tree (crate names, versions, source) as JSON in a dedicated ELF section (`.dep-v0`) of the compiled binary. This enables post-hoc vulnerability scanning of deployed binaries without access to the source repository.

Key properties:
- The embedded data contains no timestamps and is sorted deterministically, so it does not break reproducible builds.
- Multiple Linux distributions (Alpine, NixOS, openSUSE, Void, Chimera, Wolfi) build all Rust packages with cargo-auditable by default.
- Scanning is done with `cargo audit bin /path/to/binary`, checking embedded deps against the RustSec advisory database.

### cargo-vet

[cargo-vet](https://github.com/mozilla/cargo-vet) (Mozilla) maintains a ledger of audited crate versions. Each dependency must be marked as reviewed by a trusted party before it can be used. Teams can import audit sets from other organizations (Mozilla, Google, etc.) to reduce individual review burden.

This is a pre-build check (part of CI), not a runtime property. It prevents new unaudited dependencies from entering the build.

### Reproducibility

For deterministic builds:
- Pin all dependencies in `Cargo.lock` (committed to the repo).
- Pin the Rust toolchain version in `rust-toolchain.toml`.
- Use `cargo-auditable` to embed provenance data.
- Consider `SOURCE_DATE_EPOCH` for any timestamps in build scripts.
- cargo-dist can generate CycloneDX SBOMs alongside release artifacts.

### Recommendation for Saalis

Enable cargo-auditable in the release build (it adds negligible overhead -- a few KB to the binary). Adopt cargo-vet once the dependency tree stabilizes, importing Mozilla's and Google's audit sets as a starting point. Commit `Cargo.lock` and `rust-toolchain.toml` for reproducibility.

## 10. Profile-Guided Optimization (PGO)

PGO uses runtime profiling data to guide compiler optimizations. Instead of relying on heuristics, the compiler knows which branches are taken, which functions are hot, and which call patterns dominate.

### The PGO Workflow

1. **Instrumented build:** Compile with profiling instrumentation inserted into the binary.
   ```
   cargo pgo build
   ```
2. **Profile collection:** Run the instrumented binary under representative workloads. Each execution generates `.profraw` files.
   ```
   cargo pgo run -- --serve --port 8080
   # Exercise typical workflows: browse library, trigger metadata enrichment, etc.
   ```
3. **Profile merging:** Merge raw profiles into an indexed format.
   ```
   # cargo-pgo handles this automatically
   ```
4. **Optimized build:** Recompile using the merged profile data.
   ```
   cargo pgo optimize
   ```

### cargo-pgo

[cargo-pgo](https://github.com/Kobzol/cargo-pgo) wraps this workflow with an ergonomic CLI. It also supports LLVM BOLT as a post-link optimization pass.

Commands: `cargo pgo build`, `cargo pgo run`, `cargo pgo test`, `cargo pgo bench`, `cargo pgo optimize`. Profiles are stored under `target/release/pgo/` and merged into `merged.profdata` automatically before the optimized build.

### BOLT: Post-Link Optimization

BOLT (Binary Optimization and Layout Tool) operates on the final linked binary, reordering functions and basic blocks based on profiling data to improve instruction cache locality. It is complementary to PGO -- PGO optimizes at the compiler level, BOLT optimizes at the binary layout level.

Combined PGO + BOLT has shown 34-68% performance improvements in compiled benchmarks. For I/O-bound applications like saalis, the improvement will be smaller but still meaningful for reducing latency in the web UI.

### Collecting Profiles on Target Hardware

For maximum benefit, profiles should be collected on the actual target platform (ARM Batocera device) running representative workloads:

1. Build an instrumented binary for aarch64.
2. Deploy to the Batocera device.
3. Run typical usage scenarios (library browsing, metadata enrichment, downloads).
4. Copy `.profraw` files back to the build machine.
5. Merge and rebuild with `cargo pgo optimize`.

This is a release-engineering step, not a per-commit CI step. Run PGO optimization for major releases, not every build.

### Cross-Architecture PGO Caveat

PGO profiles are partially architecture-dependent. Profiles collected on x86_64 can still improve an aarch64 build (branch frequencies and call counts transfer), but architecture-specific optimizations (register allocation, instruction scheduling) will be less effective. For best results, collect profiles on the target architecture.

### Recommendation for Saalis

Defer PGO until after the first stable release. The optimization workflow adds significant complexity and build time. When ready, use cargo-pgo with profiles collected on a real Batocera device. BOLT is optional and requires building LLVM from source (or using LLVM 16+ releases); evaluate only if latency profiling identifies instruction cache pressure as a bottleneck.

## Summary: Recommended Saalis CI/CD Pipeline

| Stage | Tool | Notes |
|---|---|---|
| Local cross-compile | cargo-zigbuild | Fast iteration, no Docker needed |
| CI cross-compile | cross-rs or cargo-zigbuild | Matrix: x86_64 + aarch64-unknown-linux-musl |
| CI caching | sccache (GHA backend) | Per-compilation-unit caching |
| Testing (CI) | cargo test (native) + cross test (QEMU) | Smoke tests on ARM |
| Testing (pre-release) | Real Batocera hardware | Full integration tests |
| Binary optimization | strip + LTO + panic=abort + opt-level=z | 3-6MB expected binary |
| Release automation | cargo-dist + cargo-release | Tag-triggered GitHub Releases |
| Package format | .pkg.tar.zst (Batocera pacman) | Direct download or custom repo |
| Supply chain | cargo-auditable + cargo-vet | Embedded deps + audit ledger |
| PGO | cargo-pgo (deferred) | For major releases, profile on real hardware |
