### The one remaining promotion: 0-RTT Resumption

The SeatSession defined concept in verkko-seat (line 56) states: "Session can be resumed (0-RTT) with a stored resumption token." This references a protocol behavior that has no defined concept in verkko-protocol. The revised taxonomy identified this as belonging in verkko-protocol.

However, 0-RTT resumption is a feature of the Noise XX/KK handshake (if the Noise protocol variant supports it) or a separate session ticket mechanism. Since verkko-seat depends on verkko-protocol, and SeatSession already references 0-RTT, the concept needs a home.

**Assessment:** 0-RTT resumption is a seat-specific optimization. verkko-protocol's Session defined concept covers full handshakes. The 0-RTT mechanism is a session variant. Two possible placements:

1. Add to verkko-protocol as a defined concept (makes protocol aware of resumption semantics).
2. Keep in verkko-seat as a SeatSession-specific behavior (seat defines how it resumes, protocol provides the transport).

**Surgical decision:** The Run 10 synthesis places 0-RTT in verkko-protocol. I follow that decision. The concept belongs in protocol because it concerns the handshake state machine, not seat-specific logic. The seat consumes it.

```

### SessionResumption

0-RTT session resumption using a stored resumption token.
Avoids full Noise handshake on reconnection.

Invariants:
- Resumption token is encrypted under the session's export
  key and stored locally.
- 0-RTT data is replay-vulnerable: only idempotent operations
  permitted in the 0-RTT window.
- Server validates the resumption token's freshness (expiry
  bound, monotonic ticket counter).
- Fallback to full handshake if resumption fails (token
  expired, key rotated, server restart).
- Resumption does not weaken forward secrecy: resumed session
  derives new transport keys via the Noise KK pattern.
```

## OP-4: Create HeartbeatExtension defined concept and specify ownership split

### Structural defect

The heartbeat frame has an 18-byte transport header owned by verkko-protocol and an 18-byte extension region whose fields are defined by verkko-mesh. Six consumers share this extension layout with no formal concept owning it. The Run 10 synthesis (P0-3) requires a HeartbeatExtension defined concept.

Currently:
- verkko-protocol Section 3.4 defines the full 36-byte heartbeat payload but notes "The extension fields carry mesh-layer semantics. See verkko-mesh: Heartbeat extension fields."
- verkko-mesh has a "Heartbeat extension fields" section (line 910) that lists the fields and their semantics.
- There is no defined concept for the extension region.

### What this operation does

1. Creates a HeartbeatExtension defined concept in verkko-mesh (it owns the semantics).
2. Specifies exactly which fields protocol owns vs which fields mesh defines.
3. Updates the cross-references to use the new defined concept name.

### Diff: verkko-mesh.md, Defined Concepts section (insert after ObservationPathForwarding, before Body)

**Insert after the ObservationPathForwarding defined concept (after line 323, before `## Body`):**

```

### HeartbeatExtension

The 18-byte extension region of the heartbeat frame. Carries
mesh-layer state observable by remote peers. The protocol layer
(verkko-protocol: Heartbeat) defines the frame format and the
18-byte transport header. The mesh layer defines the extension
region contents.

Invariants:
- The extension region is exactly 18 bytes.
- Field layout is fixed:
  - `filter_root` (8 bytes): keyed truncated Merkle root for
    divergence detection (see ContentFilter).
  - `epoch_current` (4 bytes): G-Counter sync, macro-epoch.
  - `work_capacity` (1 byte): re-encryption/donation capacity.
    Computed as `min(bw_kbps/128, flash_kb/256, 255)`.
  - `health_canary` (1 byte): top 4 bits = worst health
    classification; bottom 4 bits = degraded source count.
  - `observation_summary` (4 bytes): aggregated cross-cluster
    observation (see ObservationPathForwarding).
- `gossip_mode` is carried in the transport header's state
  byte (bit 2), not in the extension region.
- The extension region is the sole carrier of mesh-layer
  state observable by remote peers in steady-state operation.
- Protocol layer treats extension bytes as opaque payload.
  Mesh layer interprets them.
- No version field in the extension region. Versioning is
  handled at the session level (SchemaSync).
```

### Ownership table

| Field | Owner | Location | Bytes |
|-------|-------|----------|-------|
| `state` (metabolic + gossip_mode + admin) | verkko-protocol | Transport header | 1 |
| `rtt_us` | verkko-protocol | Transport header | 2 |
| `loss_permille` | verkko-protocol | Transport header | 2 |
| `jitter_us` | verkko-protocol | Transport header | 2 |
| `queue_depth` | verkko-protocol | Transport header | 1 |
| `available_bw_kbps` | verkko-protocol | Transport header | 2 |
| `hlc_timestamp` | verkko-protocol | Transport header | 8 |
| `filter_root` | verkko-mesh | Extension region | 8 |
| `epoch_current` | verkko-mesh | Extension region | 4 |
| `work_capacity` | verkko-mesh | Extension region | 1 |
| `health_canary` | verkko-mesh | Extension region | 1 |
| `observation_summary` | verkko-mesh | Extension region | 4 |

The `gossip_mode` flag (bit 2 of `state` byte) is a boundary object: the bit position is owned by protocol (it defines the state byte layout), the semantic meaning is owned by mesh (it drives gossip mode transitions). This is already documented in the Boundary Object Table in verkko-ops.
