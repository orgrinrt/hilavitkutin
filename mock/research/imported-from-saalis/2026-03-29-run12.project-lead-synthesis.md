#### R12-001: Exclude hop_count from AEAD associated data

**Source:** Expert C, Stage 1, Problem 1
**Confirmed by:** Expert A (Stage 2a), Expert B (Stage 2a), Savant (Stage 2b)
**Consensus:** Unanimous
**Target document:** verkko-protocol
**Location:** Section 3.1 (Frame Format), AEAD associated data definition
**Action:** REPLACE

**Current text:**

> AEAD associated data: outer_header (29 bytes)

**Replacement text:**

> AEAD associated data: outer_header[0..8] || outer_header[9..29] (28
> bytes). The hop_count field (offset 8, 1 byte) is excluded from AEAD
> AD because relay peers must decrement hop_count during forwarding.
> Excluding hop_count preserves authentication of destination (4 bytes),
> connection_id (4 bytes), relay_key_id (8 bytes), and nonce (12 bytes).
>
> Security analysis: an attacker who modifies hop_count can set it to 0
> (frame dropped, equivalent to packet drop) or increase it up to max 4
> (bounded amplification). Neither threatens confidentiality or integrity.
> Precedent: IPsec ESP (RFC 4303 Section 3.3.3.1.1) excludes mutable IP
> header fields (TTL) from ICV computation. QUIC (RFC 9001 Section 5.4)
> encrypts the packet number separately for similar reasons.
>
> Implementation: ChaCha20-Poly1305 supports multi-part associated data
> via incremental authentication. The AD is fed in two parts:
>
> ```
> fn build_aead_ad(outer_header: &[u8; 29]) -> [&[u8]; 2] {
>     [&outer_header[0..8], &outer_header[9..29]]
> }
> ```

**Rationale:** This is a correctness bug. Without this fix, every frame
forwarded through a relay has an invalid auth tag and is silently dropped
by the recipient. Transit relay forwarding is non-functional. At 40%
symmetric NAT prevalence, this affects a majority of multi-household
deployments.

#### R12-002: Define the SansIO Protocol trait (4-method polling model)

**Source:** Expert B, Stage 1, Problem 1
**Refined by:** Expert B Stage 2a (revised trait), Savant Stage 2b
(arena-epoch concession), Expert B Stage 3a (final form)
**Consensus:** Expert B maintains 4-method; Savant concedes for Rust
implementation but maintains 2-phase for protocol specification.
**Resolution:** 4-method for Rust crate; 2-phase described in ops as the
abstract specification model.
**Target document:** verkko-protocol
**Location:** New section "SansIO Polling Model" (after Section 3, before
Section 4)
**Action:** INSERT

**Exact text:**

> ## SansIO Polling Model
>
> The Protocol trait is the sole entry point for the consumer (event loop)
> to interact with the protocol state machine. All I/O is external to the
> trait. The protocol never touches a socket, timer, or filesystem.
>
> ### Protocol Trait (Rust Implementation)
>
> ```rust
> trait Protocol {
>     /// Process one input event. May mutate internal state. May enqueue
>     /// outputs. Returns Ok(()) on success, Err on fatal error.
>     /// On ArenaExhausted: consumer must provide more arena, then retry.
>     fn handle<'a>(
>         &mut self,
>         input: ProtocolInput<'a>,
>         arena: &'a ArenaRef<'a>,
>     ) -> Result<(), ProtocolError>;
>
>     /// Drain the next pending transmission. Returns None when no more
>     /// are pending. Transmissions returned in priority order (vital
>     /// first). The returned Transmit borrows from the arena.
>     fn poll_transmit<'a>(
>         &mut self,
>         arena: &'a ArenaRef<'a>,
>     ) -> Option<Transmit<'a>>;
>
>     /// Drain the next pending application event. Returns None when no
>     /// more are pending. Events borrow from the arena.
>     fn poll_event<'a>(
>         &mut self,
>         arena: &'a ArenaRef<'a>,
>     ) -> Option<Event<'a>>;
>
>     /// Get the next timer deadline across all connections.
>     /// Returns None if no timers are active.
>     fn next_deadline(&self) -> Option<Instant>;
>
>     /// Process all timers that have expired at or before now_ms.
>     fn handle_timeout<'a>(
>         &mut self,
>         now_ms: u64,
>         arena: &'a ArenaRef<'a>,
>     ) -> Result<(), ProtocolError>;
>
>     /// Returns true if any CryptoRequest outputs are pending.
>     fn has_pending_crypto(&self) -> bool;
> }
> ```
>
> ### Error Type
>
> ```rust
> #[repr(u8)]
> enum ProtocolError {
>     ArenaExhausted = 0,
>     CryptoFailure = 1,
>     ProtocolViolation = 2,
>     NonceHalted = 3,
> }
> ```
>
> ### Calling Convention
>
> ```
> loop {
>     let datagrams = socket.recvmmsg();
>     for dgram in datagrams {
>         let input = ProtocolInput::BytesReceived {
>             source: dgram.addr.into(),
>             data: dgram.payload,
>         };
>         match protocol.handle(input, &arena) {
>             Ok(()) => {},
>             Err(ProtocolError::ArenaExhausted) => {
>                 drain_all(&mut protocol, &arena, &socket);
>                 arena.reset();
>                 protocol.handle(input, &arena).unwrap();
>             },
>             Err(e) => { log_error(e); }
>         }
>     }
>
>     // Process expired timers
>     if let Some(deadline) = protocol.next_deadline() {
>         if now >= deadline {
>             protocol.handle_timeout(now_ms, &arena).ok();
>         }
>     }
>
>     // Drain transmissions (sent to network)
>     while let Some(tx) = protocol.poll_transmit(&arena) {
>         socket.sendto(tx.destination, tx.payload);
>     }
>
>     // Drain application events (dispatched to mesh layer)
>     while let Some(event) = protocol.poll_event(&arena) {
>         mesh.handle_event(event);
>     }
>
>     // Reset arena for next iteration
>     arena.reset();
> }
> ```
>
> ### Buffer Ownership Rules
>
> - `ProtocolInput::BytesReceived::data`: borrowed for the duration of
>   `handle()`. The protocol copies what it needs into the arena or
>   managed state.
> - `Transmit::payload`: borrows from the arena. Valid until
>   `arena.reset()`.
> - `Event` payloads: borrow from the arena. Valid until `arena.reset()`.
> - The consumer MUST drain all `poll_transmit()` and `poll_event()`
>   before calling `arena.reset()`.
>
> ### Backpressure
>
> - If the internal queue is full, subsequent `handle()` calls that would
>   enqueue new transmissions return `Err(ArenaExhausted)`.
> - Vital-tier frames are never dropped due to arena exhaustion. The
>   protocol reserves a fixed portion of the arena for vital frames.
> - If the vital reserve is exhausted, the protocol enters degraded mode:
>   heartbeats and SACKs only; all other frames dropped until arena is
>   drained.
>
> ### Reentrancy
>
> `handle()` MUST NOT be called while a previous `handle()` is in
> progress. All methods take `&mut self`, enforcing exclusive access at
> the type level in Rust. For non-Rust implementations: the polling model
> is structurally non-reentrant because output methods do not accept
> inputs.
>
> ### Abstract Specification (Language-Neutral)
>
> The abstract interface is two-phase: `feed(input)` produces state
> changes; `poll_output()` drains all consequences. `next_deadline()`
> queries the earliest timer. This provides structural reentrancy
> prevention independent of any language's type system. The Rust 4-method
> model is one valid optimization of this abstract interface.

**Rationale:** The polling model is the highest-priority gap in the
entire verkko specification (D-003). Without it, two independent
implementations produce wire-compatible but integration-incompatible
state machines. The 4-method model follows quinn precedent and provides
type-level output separation and arena-epoch batching for zero-copy
sendmmsg.

#### R12-005: Define reliability mode binding

**Source:** Expert B, Stage 1, Problem 2
**Confirmed by:** Expert B Stage 3a (gossip unreliable constraint)
**Consensus:** All streams reliable-ordered except gossip IHAVE (unreliable
datagrams). Arena-driven constraint.
**Target document:** verkko-protocol
**Location:** Section 3.3 (Streams)
**Action:** REPLACE

**Current text:**

> Reliable ordered: monotonic sequence, SACK ACKs, exponential backoff.
> Unreliable datagram: per-source monotonic sequence for gap detection.

**Replacement text:**

> ### Reliability Modes
>
> All streams are reliable-ordered. Heartbeats and standalone SACKs are
> connection-level (no stream header) and inherently unreliable.
>
> Gossip IHAVE messages MUST use unreliable datagrams at the protocol
> level. This is an arena-driven design requirement: during convergence,
> a gossip burst of 2,200+ frames would require 2.7 MB of retransmission
> state if sent on reliable-ordered streams, exceeding the 256 KB pool
> arena by 10x. The mesh layer's Plumtree IHAVE/GRAFT mechanism provides
> gossip reliability independently of stream-level retransmission.
>
> Retransmission state per reliable stream:
>
> ```rust
> struct StreamRetransmitState {
>     unacked: RingBuf<UnackedFrame, 32>,
>     rto_ms: u32,
>     retransmit_count: u8,
>     cumulative_ack: u32,
>     sack_blocks: [SackBlock; 4],
>     sack_block_count: u8,
>     recv_window: u32,
> }
> ```
>
> Memory per stream: ~816 bytes. At 11 connections * 10 active streams
> each = ~88 KB. Fits within the pool arena (256 KB).

**Rationale:** The simplicity of uniform reliable-ordered streams outweighs
the ~22 KB memory savings from mixed modes. The gossip exception is driven
by the hard arena budget constraint, not by preference.

#### R12-006: Replace SetTimer/TimerExpired with internal timer management

**Source:** Expert B, Stage 1, Problem 5
**Confirmed by:** Savant (next_deadline signature adopted by Expert B)
**Target document:** verkko-protocol
**Location:** ProtocolInput/ProtocolOutput enumerations, timer specification
**Action:** REPLACE

**Current text:**

> SetTimer { timer_id: u32, deadline_ms: u64 }
> TimerExpired { timer_id: u32 }

**Replacement text:**

> Timer management is internal to the protocol state machine. The
> consumer calls `next_deadline()` to get the earliest deadline across
> all connections, then calls `handle_timeout(now_ms, arena)` when the
> deadline elapses. The protocol processes all expired timers internally.
>
> SetTimer and CancelTimer are removed from the ProtocolInput/
> ProtocolOutput enumerations. The consumer tracks one deadline (the
> earliest returned by `next_deadline()`). Timer types (heartbeat, RTO,
> path challenge, CPSK renegotiation, connection close drain, metabolic
> dwell) are internal protocol state.
>
> Mesh-layer timers (convergence wavefront, convergence duration bound,
> dwell time restoration) are NOT protocol timers. The consumer manages
> mesh-layer timers separately. The consumer's timer wheel holds both
> protocol deadlines (from `next_deadline()`) and mesh-layer deadlines.

**Rationale:** Quinn and smoltcp both use this pattern. Internal timer
management eliminates timer ID allocation, collision handling, and
cancellation as consumer concerns.

**Source:** Expert C, Stage 1, Problem 3

**Rejected alternatives:** Traffic class split (Savant Stage 2b, rejected

**Current text:**

**Replacement text:**

> Invariant 2: Relay-first connectivity. Transit relay provides immediate
> connectivity (~500ms, one RTT). Direct paths are established in
> background via hole punching. When a direct path is verified via
> PATH_CHALLENGE/PATH_RESPONSE, all traffic migrates from relay to

**Rationale:** Expert C demonstrates that at 40% symmetric NAT prevalence,

eliminates this latency. Expert C's devil's advocate analysis shows the
relay bandwidth cost during the establishment window is negligible (938 KB
for a 10-second window on a 2 GB transfer = 0.046%).

#### R12-008: Define intra-cluster session model

**Source:** Expert A, Stage 1, Problem 2; Savant, Stage 1, Problem 1
**Confirmed by:** All experts agree on DEK-based Noise KK
**Target document:** verkko-mesh
**Location:** New section "Intra-Cluster Communication" (after Section 2,
Cluster)
**Action:** INSERT

**Exact text:**

> ## Intra-Cluster Communication
>
> ### Session Model
>
> Intra-cluster peers use Noise KK with each peer's DEK as the static
> key. One session per peer pair within the cluster. CIK MUST NOT be
> used as the static key for intra-cluster Noise KK: CIK is shared
> across all cluster peers, and Noise KK with identical static keys on
> both sides degenerates (the handshake produces symmetric DH outputs
> in both directions).
>
> Session lifecycle:
> - Initiated on peer enrollment (enrollment cert contains new_DEK_pub).
> - Renegotiated on DEK rotation (rare).
> - Torn down on peer departure (enrollment revocation, clean shutdown).
> - No polygon scheduling (LAN RTT is negligible).
>
> ### dest_peer Assignment
>
> Sequential u16 at enrollment, starting from 0x0001, incrementing for
> each new peer. The assignment is logged in the enrollment cert (one
> additional u16 field). dest_peer 0x0000 is broadcast (all cluster
> peers). dest_peer values are never reused within a cluster's lifetime
> (even after peer revocation, the value is retired).
>
> ### Gateway Routing Within Cluster
>
> The gateway maintains a routing table mapping dest_peer values to
> intra-cluster session handles. When a frame arrives from an inter-
> cluster session with dest_peer != 0x0000, the gateway looks up
> dest_peer in the routing table and forwards the frame on the
> corresponding intra-cluster session. The inner header is not modified;
> only the outer header changes (new connection_id, new nonce,
> decremented hop_count).
>
> ### Intra-Cluster Heartbeats
>
> Fixed 1-second interval (LAN bandwidth is not a constraint; the
> variable interval is for WAN conservation). Same transport header as
> inter-cluster heartbeats. Simplified 8-byte extension region:
> filter_root (8 bytes) only. Other extension fields (epoch_current,
> work_capacity, health_canary, observation_summary) are inter-cluster
> concerns.
>
> ### Intra-Cluster Health Scoring
>
> Simplified. No dual-EMA. A sliding window of the last 32 heartbeat
> results (u32 bitmask). Health degradation triggers when fewer than 22
> of the last 32 heartbeats were received (approximately 30% loss
> sustained over 32 seconds). This threshold absorbs WiFi burst losses
> of up to 10 consecutive misses without triggering degradation.
>
> Scar threshold: 30 consecutive missed heartbeats (30 seconds). At 15%
> WiFi packet loss, the probability of 30 consecutive misses is
> 0.15^30, which is effectively zero. Only genuine 30+ second outages
> produce scars. WiFi-induced scars use the WifiInduced attribution
> (see Scar Classification) with 1-7 day adaptive decay.
>
> Gateway election dampening: gateway re-election requires sustained
> intra-cluster health degradation (10+ seconds below threshold) or
> explicit peer departure. Transient WiFi-induced degradation (< 10
> seconds) does not trigger gateway re-election.
>
> ### Resource Accounting
>
> Intra-cluster sessions use a separate ThresholdGuard instance: 10 KB
> arena allocation per peer, 60 KB maximum for a 6-peer cluster. Total
> managed arena: 256 KB (200 KB inter-cluster + 56 KB intra-cluster).

**Rationale:** Intra-cluster communication is the foundation for CIK
replication, gateway handoff, observation-path forwarding, and KeyBundle
distribution. Expert A and Savant independently identified the CIK key
degeneracy as a fundamental cryptographic problem. The WiFi-aware
thresholds (sliding window, 30-miss scar threshold, gateway dampening)
are driven by Expert C's devil's advocate analysis showing that
Ethernet-tuned thresholds (3-miss scar threshold) produce false scars
every 4.9 minutes at 15% WiFi packet loss.

#### R12-009: Define gateway handoff procedure

**Source:** Expert A, Stage 2a (4-phase handoff)
**Confirmed by:** Expert A Stage 2a (teardown, not migration)
**Target document:** verkko-mesh
**Location:** Within intra-cluster communication section
**Action:** INSERT

**Exact text:**

> ### Gateway Handoff
>
> Gateway handoff tears down and re-establishes inter-cluster sessions.
> Connection_id is NOT preserved across gateway handoff (the new gateway
> is a new Noise endpoint with a different DEK).
>
> Handoff procedure (4 phases):
>
> 1. **Prepare:** New gateway receives CPSK session parameters from old
>    gateway via intra-cluster vital-tier reliable stream.
> 2. **Establish:** New gateway initiates Noise KK sessions with all
>    inter-cluster gateways using the transferred CPSK parameters.
> 3. **Migrate:** Old gateway stops accepting new inter-cluster frames.
>    New gateway begins accepting. Overlap window: both gateways forward
>    intra-cluster frames during the transition.
> 4. **Teardown:** Old gateway tears down its inter-cluster sessions.
>    New gateway is sole inter-cluster endpoint.
>
> CIK rotation ordering: CIK distribution (intra-cluster, vital tier,
> reliable delivery) MUST complete before the new gateway establishes
> inter-cluster CPSK sessions. The new gateway waits for intra-cluster
> CIK distribution confirmation before initiating inter-cluster
> handshakes.
>
> Relay migration (distinct from gateway handoff): connection_id IS
> preserved. The relay is not a Noise endpoint; only the forwarding path
> changes. PATH_CHALLENGE through the new relay verifies reachability.

**Rationale:** Expert A's analysis demonstrates that connection migration
is not feasible for gateway handoff because the new gateway cannot
continue the old gateway's Noise session state. Relay migration is
different because the relay is not a Noise endpoint.

#### R12-010: Define channel lifecycle

**Source:** Expert A, Stage 1, Problem 3; Savant, Stage 1, Problem 6
**Confirmed by:** Expert A Stage 2a (EDT scope_flags resolution)
**Target document:** verkko-mesh
**Location:** Channel defined concept body text
**Action:** INSERT

**Exact text:**

> ### Channel Lifecycle
>
> Channel creation is admin-only, consistent with the single-admin-
> authority model. Channel creation is blocked during convergence
> (convergence_active = 1).
>
> CHANNEL_CREATE control log entry (type 0x0E) payload:
>
> ```
> ChannelCreatePayload {
>     channel_index: u16,
>     channel_key: [u8; 32],       // initial 256-bit key (CSPRNG)
>     access_defaults: u8,         // per-role default access level
>     replication_policy: u8,      // encoded ReplicationPolicy
>     display_name_hash: [u8; 8],  // truncated BLAKE3 of name
> }
> ```
>
> The initial channel key is wrapped in a KeyBundle(Grant) and
> distributed via the existing BroadcastEnvelope mechanism.
>
> Channel index conflict prevention: indices are assigned by the admin
> via a local counter. Single-admin-authority (fencing token) prevents
> concurrent conflicting assignments. During partition with split admin,
> the higher fencing token wins during Front 2 consensus; the losing
> partition's channel creation is orphaned and requires re-creation.
>
> Channel deletion does not exist. GSet semantics: append-only. A
> channel is deactivated by revoking all keys (KeyBundle(Revoke)).
>
> EDT scope_flags interaction: the u64 bitmask limits the EDT to
> channels that existed at EDT creation time. New channels created
> after EDT issuance are not accessible via that EDT. The admin must
> reissue the EDT to include new channels.

**Rationale:** The CHANNEL_CREATE entry type was reserved (0x0E) but
unspecified. Without creation semantics, the system cannot add new data
partitions after initial setup. The admin-only constraint and fencing
token mechanism reuse existing security primitives.

#### R12-011: Define scar classification (three-layer defect model)

**Source:** Savant, Stage 2b, Mechanism 3
**Confirmed by:** Expert A Stage 3a (CRDT-compatible, convergent),
Expert C Stage 3a (does not harm relay health monitoring)
**Target document:** verkko-mesh
**Location:** Scar defined concept body text
**Action:** INSERT

**Exact text:**

> ### Scar Classification
>
> Scars carry a ScarAttribution variant that determines the decay period.
> The classification is a CRDT-compatible payload extension: the GSet
> grows monotonically; merge (set union) is unaffected by payload
> complexity; the decay function is a pure function of (scar,
> current_time).
>
> Three scar layers:
>
> **Layer 1: Thermal (WiFi-induced).** Decay: 1 + (1 - wifi_quality) * 6
> days. Determined by the diagnostic triangle's PathDiagnosis at scar
> creation time. The wifi_quality_at_observation is stored in the scar
> entry (observation-time state, not query-time local state). All peers
> compute the same decay from the same stored value.
>
> ```
> ScarAttribution::WifiInduced {
>     observer: DK,
>     wifi_quality_at_observation: u16,  // Q8.8
> }
> ```
>
> Corrected decay formula (Q8.8 fixed-point):
>
> ```
> fn wifi_scar_decay_days(wifi_quality_q8_8: u16) -> u16 {
>     let one_minus_q = 0x0100u16.saturating_sub(wifi_quality_q8_8);
>     let days_q8_8 = 0x0100u16.saturating_add(
>         one_minus_q.saturating_mul(6));
>     days_q8_8 >> 8
> }
> ```
>
> At wifi_quality 0.60: decay = 3 days. At 0.85: decay = 1 day. At 0.95:
> decay = 1 day. Worse WiFi produces shorter decay (the scar is more
> likely environmental noise).
>
> **Layer 2: Structural (genuine faults).** Decay: 180 days.
> DirectObservation or CorrelatedSuspicion attribution. Full dominance
> impact.
>
> **Layer 3: Grain boundary (relay/path faults).** Decay: 30 days.
> PathFault attribution. Moderate dominance impact on relay selection
> priority. Does not affect the destination peer's dominance score
> (relay is a pseudo-peer).
>
> ### Observation Window
>
> The scar attribution is determined by the minimum WiFi quality observed
> during the observation window, not the instantaneous value at scar
> creation time. The observation window starts when the first missed
> heartbeat occurs and ends when either (a) the scar threshold is
> reached, or (b) heartbeats resume.
>
> ```
> struct ObservationWindow {
>     consecutive_misses: u16,
>     min_wifi_quality_q8_8: u16,
>     window_start_epoch: u64,
> }
> ```
>
> The running minimum ensures that a scar is classified as WifiInduced
> if WiFi was bad at any point during the observation window, even if it
> recovered before the threshold was reached. The min_wifi_quality is
> local state (not replicated); it affects only the attribution field
> written to the GSet.

**Rationale:** Expert C's devil's advocate analysis demonstrates that
a 30-second WiFi burst produces a 180-day scar under the current design.
The three-layer classification reduces this to a 1-7 day scar. Expert A
proves the classification is CRDT-compatible: the attribution is stored
in the GSet entry; all peers compute the same decay from the same stored
value. The running minimum (Expert A correction) prevents misclassification
when WiFi quality oscillates during the observation window.

#### R12-017: Define replication orchestration (combined triad)

**Source:** Expert A, Stage 1, Problem 1; Savant, Stage 2b, Mechanism 1;
Expert A, Stage 3a (session-based correction)
**Target document:** verkko-mesh (replication section) + verkko-matter
**Location:** Replication defined concept body text
**Action:** INSERT

**Exact text (verkko-mesh):**

> ### Replication Orchestration
>
> Three interlocking components:
>
> **Component 1: Priority Queue (WHAT).** most_under_replicated() selects
> the blob with the highest deficit. Deficit is weighted by priority:
> P1 at 3.0x, P2 at 2.0x, P3 at 1.0x, P4 at 0.5x. The queue determines
> the replication face: the system replicates the most under-replicated,
> highest-priority content first.
>
> **Component 2: Probabilistic Roll (WHO).** Each peer independently
> decides whether to initiate replication. The roll is deterministic per
> (blob_id, my_dk, epoch):
>
> ```
> let probability_q16_16 = ((deficit as u32) << 16)
>     / (known_holders as u32 + 1);
> let roll_q16_16 = (BLAKE3("saalis-repl-v1" || blob_id || my_dk
>     || epoch)[0] as u32) << 8;
> should_replicate = roll_q16_16 < probability_q16_16
> ```
>
> Fixed-point Q16.16 arithmetic. No floating-point. Coordination-free.
> Expected redundant transfers = O(1) regardless of holder count.
>
> **Component 3: Supersaturation Rate (HOW MANY).** Controls concurrent
> TransferSessions (not individual chunks). Each TransferSession runs
> its own bao streaming at LEDBAT pace.
>
> ```
> fn replication_sessions_to_open(
>     deficit: u8,
>     priority: MatterPriority,
>     metabolic: MetabolicState,
> ) -> u8 {
>     let target = priority.target_replicas();
>     let saturation_q16 = ((deficit as u32) << 16)
>         / (target as u32);
>     let priority_weight_q16 = match priority {
>         P1 => 0x00030000,  // 3.0x
>         P2 => 0x00020000,  // 2.0x
>         P3 => 0x00010000,  // 1.0x
>         P4 => 0x00008000,  // 0.5x
>     };
>     let weighted = ((saturation_q16 as u64
>         * priority_weight_q16 as u64) >> 16)
>         .min(0x10000) as u32;
>     let base_rate = match metabolic {
>         Background => 1u32,
>         Idle => 3,
>         Active => 2,
>         Stress => 1,
>     };
>     let scaled = (base_rate * weighted) >> 16;
>     scaled.max(1) as u8
> }
> ```
>
> Minimum is always 1 session. Budget is NEVER zero in any metabolic
> state.
>
> ### Replication Gate
>
> REPLICATION_READY = filter_reconciliation_complete AND
> convergence_front >= Front 2. The gate prevents replication decisions
> on stale filter data. After the gate opens, sessions come online
> gradually under Conductor governance.
>
> ### Replication Targets
>
> Replication targets (min_replicas, target_replicas) are per-channel
> configuration, distributed via the control log as CHANNEL_CONFIG
> entries (type 0x0F). Subject to fencing token ordering: the dominant
> admin's configuration wins after convergence.

**Rationale:** Expert A's probabilistic roll and the Savant's
supersaturation-weighted growth rate are complementary, not competing.
Expert A provides coordination-free deduplication; the Savant provides
deficit-proportional throughput. Expert A's Stage 3a correction changes
the unit from chunks-per-tick to concurrent TransferSessions, eliminating
the 39-day replication time for 2 GB blobs. The replication gate (Savant
Stage 2b) prevents thundering herd on stale filter data.

> ### Transit Forwarding
>

>

#### R12-026: Define relay failover (warm standby)

**Source:** Expert C, Stage 1, Problem 4
**Target document:** verkko-relay
**Location:** After relay resource governance
**Action:** INSERT

**Exact text:**

> ### Relay Failover
>
> When a peer registers with a relay via SIGNAL HELLO, it also registers
> with a secondary relay (the next-best candidate from relay selection).
> The secondary relay holds a dormant forwarding entry.
>
> Failover procedure:
>
> 1. Relay heartbeat timeout (3 missed keepalive probes at 70% of
>    mapping lifetime).
> 2. Promote secondary relay: activate forwarding entries.
> 3. Notify remote endpoint via secondary relay: SIGNAL REDIRECT.
> 4. Select new secondary relay, register via SIGNAL HELLO.
>
> Failover timing: 1 heartbeat timeout (2-30 seconds depending on
> metabolic state) + 1 RTT for redirect (~200ms). Total: 2.2-30.2
> seconds.
>
> Relay keepalive probes: 2-second interval regardless of metabolic
> state. Cost: 2 * 91 bytes/s = 182 bytes/s per relayed session = 3.6
> bytes/s as warm standby.

**Rationale:** Without relay failover, a relay peer reboot causes 17-100
seconds of outage. Expert C's warm standby reduces worst-case failover
to 30.2 seconds (Background mode). The 2-second keepalive probes are
independent of the heartbeat interval.

#### R12-034: Add abstract polling model to ops

**Source:** Savant, Stage 3b (concession); Expert B, Stage 3a
**Target document:** verkko-ops
**Location:** VoiceSpec or new "Abstract Interface" section
**Action:** INSERT

> ### Abstract Protocol Interface (Language-Neutral)
>
> The abstract polling model is two-phase:
>
> 1. feed(input): processes one input event, mutates internal state.
> 2. poll_output(): drains one output event. Returns None when empty.
> 3. next_deadline(): queries the earliest timer deadline.

two-phase for language neutrality. The Savant concedes the Rust
implementation uses four methods. Both agree the wire protocol is the

## Part 6: Remaining Dangers

Ranked by severity and likelihood of causing implementation problems.

### Danger 1: SyncReconciliation merge rules (HIGH)

The entire matter metadata sync mechanism beyond content filters is a
blank page. Expert A identifies this as the most significant gap from a
replication perspective. The Replication defined concept says "field-level,
deterministic, reproducible" but specifies no merge rules. Without merge
rules, two implementations will produce different post-partition state for
matter metadata. Content filter reconciliation handles advertisement (what
exists), but metadata reconciliation (attributes of what exists) is
unspecified.

**Impact:** Matter metadata (ratings, tags, descriptions, priority
assignments) can remain inconsistent indefinitely after partition heal.
The convergence exit sequence does not gate on matter metadata sync.

**Recommendation:** Dedicated design round in Run 13.

### Danger 2: Convergence body gaps (MEDIUM-HIGH)

The polygon schedule (11 rounds, Lucas 1-factorization) and the 6-step
exit sequence are referenced in verkko-ops but lack body text in
verkko-mesh. Expert A's Stage 2a identifies that the polygon schedule
needs a retry mechanism for relay failover (if a pair's CPSK
renegotiation fails, the pair must be re-queued). The exit sequence's
interaction between "restore dwell time" and "convergence override
released" needs explicit sequencing (Expert A: set last_transition_ms
= now_ms on override release).

**Impact:** The convergence mechanism is the most complex state machine
in the design. Without body text, implementors must reverse-engineer
the sequence from invariants and the ops mechanism reference table.

**Recommendation:** Run 13 priority item.

### Danger 3: Handshake frame forwarding through relay (MEDIUM)

Noise handshake frames use a different header format (magic 0x56524B4F)
than data frames (destination field). The relay's forwarding table is
keyed by (destination, connection_id). Handshake frames do not carry
these fields. Expert A identifies this gap: SIGNAL HELLO must establish
the forwarding entry before the handshake begins, but the inverted
ordering (register relay entry, then perform handshake through the
relay) is not fully specified.

**Impact:** First-time inter-cluster session establishment through a
relay may fail if the handshake frame cannot be forwarded.

**Recommendation:** Expert C should address in Run 13 by specifying
whether handshake frames carry a destination field or use a separate
forwarding path.

### Danger 4: Matter garbage collection (MEDIUM)

Storage grows without bound. The Savant identifies this as a "pressure
defect that will cause lattice fracture at runtime." No garbage
collection mechanism exists. The flash budget moderation (diminuendo
curve) handles write pressure but does not reclaim storage.

**Impact:** Long-running deployments will exhaust storage on
resource-constrained devices (Pi Zero, 32 GB SD card).

**Recommendation:** Run 13 design round.

### Danger 5: WiFi-metabolic feedback loop (LOW-MEDIUM)

The Savant identifies in Stage 2b that WiFi power-save buffering creates
a feedback loop with the metabolic state machine: WiFi jitter causes
LEDBAT to back off, reducing transport_capacity, pushing toward Stress,
which changes heartbeat intervals, which changes WiFi burst patterns.
Expert B's GovernorOutput with bottleneck type provides the mechanism
to break this loop (WiFi-bottleneck gets different responses than CPU-
bottleneck), but the feedback dynamics are not formally modeled.

**Impact:** On Pi Zero with WiFi power management, the metabolic state
machine may oscillate between Active and Stress driven by WiFi jitter
rather than actual resource pressure. The hysteresis margin (0.05) may
be insufficient for WiFi-induced pressure fluctuations.

**Recommendation:** Empirical calibration on target hardware. The
hysteresis margin and WiFi-specific response thresholds are calibration
items.

### Danger 6: DK encryption at rest (LOW-MEDIUM)

The DK private key is the identity secret. Its storage encryption is
unspecified. On Batocera (SD card, no TPM), the encryption mechanism
must balance security against the threat model (physical theft of SD
card) and usability (no password entry on headless appliance).

**Impact:** Physical access to the device recovers the DK private key,
enabling identity theft and mesh impersonation.

**Recommendation:** Run 13 design round.
