# Deep Dive: `min_specialization` in Rust

Research date: 2026-03-15

Saalis depends on nightly Rust's `min_specialization` for three architectural patterns: the `SupportsEntity`/`HasMetadata` blanket bridge, the `SetAccess` default-access matrix, and the `Secret` blanket from `Credential`. This document covers everything a contributor needs to understand about what the feature is, what it is not, why it exists as a subset, and what happens if it never stabilizes.

---

## 1. What `min_specialization` allows vs full specialization

RFC 1210 introduced specialization: the ability to have overlapping trait implementations where a more specific impl takes priority over a more general one. The `specialization` feature gate enables the full RFC. The `min_specialization` feature gate enables a deliberately restricted subset that the compiler team considers sound (with caveats).

**Full specialization** allows marking any item in an impl as `default` â€” functions, associated types, associated consts â€” and then providing a more specific impl that overrides those items. The specializing impl need only be "more specific" in the sense that it applies to a strict subset of types.

**`min_specialization`** restricts this to impls that are "always applicable." An impl is always applicable if, whenever the base impl's bounds are satisfied and the specializing impl's bounds are satisfied for *some* choice of lifetimes, the specializing impl applies for *any* choice of lifetimes. In practice this means:

- No specializing on lifetime bounds (no `T: 'static` as a specialization axis).
- No repeating generic parameters in unconstrained positions.
- The only new bounds a specializing impl may introduce must either (a) already exist on the base impl after substitution, or (b) be well-formed predicates implied by the trait's type arguments, or (c) be bounds on traits specifically marked as specialization-safe in the compiler (`#[rustc_specialization_trait]` or `#[rustc_unsafe_specialization_marker]`).

The practical effect: you can specialize a blanket `impl<T> Trait for T` with a more specific `impl<T: SomeMarker> Trait for T`, but you cannot specialize based on whether a lifetime is `'static`, and you cannot introduce arbitrary new trait bounds that the compiler cannot verify as "always applicable."

## 2. The soundness problem with full specialization

The canonical demonstration of the unsoundness comes from Aaron Turon's 2017 blog post "Shipping specialization: a story of soundness" and the corresponding tracking issue (#40582). The attack constructs a `make_static` function that converts any `&'a str` into `&'static str` â€” obviously unsound.

The exploit relies on the fact that Rust erases lifetimes before code generation. The type checker sees that `'a` is not `'static` and selects the default impl; but after lifetime erasure, the code generator cannot distinguish `&'a str` from `&'static str` and may select the wrong specialization. The disagreement between type checking (which knows lifetimes) and codegen (which does not) is the root cause.

**Concrete sketch of the attack:**

```rust
#![feature(specialization)]

trait IsStatic {
    fn is_static() -> bool { false }  // default
}

default impl<T> IsStatic for T {}

impl<T: 'static> IsStatic for T {
    fn is_static() -> bool { true }  // specialized for 'static
}
```

After lifetime erasure, the codegen cannot determine which impl applies. This class of bug allows constructing use-after-free, type confusion, and other memory-safety violations without any `unsafe` block.

**How `min_specialization` avoids this:** The "always applicable" restriction ensures that the specializing impl's applicability does not depend on lifetime information. If `T: SomeMarker` holds for some choice of lifetimes, it must hold for all choices of lifetimes. The compiler verifies this by checking that:

1. No direct use of `'static` appears in the specializing impl's type arguments.
2. Generic parameters appear at most once in unconstrained argument positions.
3. All new predicates on the specializing impl are either present on the base impl (after substitution) or are well-formed predicates for the trait's type arguments.
4. Any trait used as a new specialization bound is annotated with `#[rustc_specialization_trait]` (which requires all impls of that trait to themselves be always-applicable) or `#[rustc_unsafe_specialization_marker]` (which is unsound w.r.t. lifetimes but permitted for marker traits like `Copy`, `Eq`, `FusedIterator`).

## 3. What `default impl` means â€” syntax and semantics

The `default` keyword on an impl or an impl item signals that the item may be overridden by a more specific impl. Without `default`, specialization is not possible for that item.

### Syntax

There are two granularities:

**Per-item default** (used in std):

```rust
impl<T: Clone> RcFromSlice<T> for Rc<[T]> {
    default fn from_slice(v: &[T]) -> Self {
        // general clone-based implementation
    }
}

impl<T: Copy> RcFromSlice<T> for Rc<[T]> {
    fn from_slice(v: &[T]) -> Self {
        // optimized memcpy-based implementation
    }
}
```

**Whole-impl default** (what saalis uses):

```rust
default impl<E, M> SupportsEntity<E> for M
where
    E: HasMetadata<M> + EntityType,
    M: MetadataTable,
{}
```

This marks the entire impl as overridable. Any explicit `impl SupportsEntity<Game> for Rating {}` takes priority over the blanket.

### Semantics

When multiple impls apply to the same type, the compiler selects the most specific one. The "most specific" relationship must form a total chain â€” there cannot be two impls that are each more specific in different ways (that would be an ambiguity error, not a specialization). The `default` keyword is the opt-in: if an impl is not `default`, the compiler rejects any overlapping impl as a coherence violation just like it does on stable Rust.

For associated types and consts marked `default`, downstream code cannot assume the default value. The type checker treats a `default type Output = ()` as an opaque associated type unless a non-default impl is known to apply. This is essential for soundness: if code could assume the default associated type, a specializing impl that changes it would violate the assumption.

## 4. Real-world usage

### Standard library

The std library is the primary consumer of `min_specialization`. The std-dev-guide explicitly states: "Only specialization using the `min_specialization` feature should be used. The full `specialization` feature is known to be unsound."

Notable uses:

- **`ToString` for `Display` types.** The blanket `impl<T: Display> ToString for T` is specialized for `str` to avoid going through the `fmt` machinery â€” `"hello".to_string()` directly allocates and copies rather than formatting.

- **Iterator forwarding.** The `impl Iterator for &mut I where I: Iterator` forwards methods to the inner iterator. Recent work (PR #138862, 2025) added specialization for all applicable methods: if the inner iterator is `Sized`, more methods can be forwarded. A private helper struct hides the specialization from the public API.

- **`Rc`/`Arc` from slice.** `RcFromSlice` uses `default fn from_slice` for `Clone` types, specialized to a `memcpy` path for `Copy` types.

- **`SpecExtend` for iterators.** Various `collect()` and `extend()` operations are specialized for iterators with known sizes (e.g., `TrustedLen`) to pre-allocate exact capacity.

The standard library follows a strict pattern: specialization is used only for performance optimization, never for behavioral differences visible to callers. A private helper trait does the dispatch, and the public method delegates to it. This avoids leaking specialization into the stable API surface.

### Other notable crates

- **Solana.** The Solana validator historically used `#![feature(specialization)]` (the full, unsound version). This was a known source of concern in their codebase.

- **`serde`.** Does not use specialization, but its `Serialize`/`Deserialize` ecosystem would benefit enormously from it â€” a frequently cited motivation for stabilization.

- **Bevy.** The Bevy game engine builds on stable Rust and does not use `min_specialization`. It uses procedural macros and the type system (sealed traits, `SystemParam` derive) to achieve similar compile-time dispatch without specialization.

## 5. The `default impl<T: Trait>` pattern â€” specializing on trait bounds

The most common `min_specialization` pattern is a blanket impl marked `default`, with a more specific impl that adds a trait bound:

```rust
trait Fast {
    fn fast_path(&self) -> bool { false }
}

default impl<T> Fast for T {}

impl<T: Copy> Fast for T {
    fn fast_path(&self) -> bool { true }
}
```

This works under `min_specialization` because `Copy` is annotated with `#[rustc_unsafe_specialization_marker]` in the standard library. The compiler has a built-in allow-list of traits that are considered safe (or at least acceptable) for specialization:

- **`#[rustc_specialization_trait]`**: All impls of this trait must be always-applicable. Implementing such a trait is unstable, so it cannot appear on stable public traits. `Sized` is the notable exception (it already cannot be manually implemented).

- **`#[rustc_unsafe_specialization_marker]`**: The trait has no associated items. Specializing on it is "unsafe" in the sense that the compiler does not verify lifetime soundness for its impls â€” but the attribute is acceptable for existing marker traits like `Copy`, `FusedIterator`, `Eq`, and `TrustedLen`. These traits are all lifetime-agnostic in practice.

For user-defined traits on nightly, you can specialize on a trait bound only if the trait is marked with one of these attributes. Without that annotation, the compiler rejects specialization with "cannot specialize on trait `YourTrait`."

## 6. Limitations â€” what you CANNOT do with `min_specialization`

### Cannot specialize on concrete types (without a marker trait)

You cannot write:

```rust
default impl<T> MyTrait for T { ... }
impl MyTrait for String { ... }  // ERROR under min_specialization
```

The concrete-type impl is not "always applicable" in the required sense â€” the compiler needs a trait-based proof of specificity, not a type-identity proof. Full `specialization` would allow this; `min_specialization` does not.

**Workaround:** Define a marker trait, implement it for the concrete type, and specialize on the marker bound. This is more boilerplate but satisfies the always-applicable check.

### Cannot specialize on lifetime bounds

```rust
default impl<T> MyTrait for T { ... }
impl<T: 'static> MyTrait for T { ... }  // REJECTED
```

This is the exact pattern that causes the soundness hole. `min_specialization` explicitly forbids it.

### Cannot specialize on `const` trait bounds

`min_specialization` does not equate `T: ~const Trait` with `T: Trait` for specialization purposes (tracking issue #95187). This means you cannot specialize a `~const` bound implementation with a non-const one, or vice versa.

### Cannot specialize associated types freely

A `default type` in an impl means the associated type is opaque to downstream code. You cannot write code that assumes the default associated type value and then have a specialization change it â€” the compiler prevents projection through default associated types. This is intentional but makes certain patterns awkward.

### Cannot specialize on user-defined traits without compiler attributes

The `#[rustc_specialization_trait]` and `#[rustc_unsafe_specialization_marker]` attributes are internal to rustc. They are not available to crate authors outside of the standard library / compiler. This means external crates using `min_specialization` can only specialize on the compiler-blessed set of traits (`Copy`, `Sized`, `Eq`, `FusedIterator`, `TrustedLen`, etc.) plus well-formed predicate bounds.

**Exception:** If the specializing impl's additional bounds are well-formed predicates implied by the trait's type arguments (e.g., a where clause that already follows from the trait definition), they are accepted. This is the loophole that makes saalis's `SupportsEntity` blanket bridge work â€” the bounds `E: HasMetadata<M>`, `E: EntityType`, `M: MetadataTable` are trait bounds that narrow the generic space, not marker-trait specializations.

## 7. Interaction with associated types and consts

### Associated types

Under specialization, an associated type in a `default impl` is treated as an opaque projection. Code consuming the trait cannot assume `<T as Trait>::Assoc == ConcreteType` if the impl providing that associated type is `default`. This prevents the compiler from making unsound inferences that a later specialization might invalidate.

```rust
trait Engine {
    type Output;
}

default impl<T> Engine for T {
    default type Output = ();  // opaque â€” cannot project through this
}

impl Engine for SpecialCase {
    type Output = Vec<u8>;     // concrete â€” projection now works
}

// fn use_engine<T: Engine>() {
//     let x: <T as Engine>::Output = ();  // ERROR: cannot assume Output = ()
// }
```

This restriction is sound but limits expressiveness. You cannot provide a useful default associated type and then specialize it â€” the default type is usable only within the impl that declares it, not by generic code.

### Associated consts

The same rules apply. A `default const` cannot be assumed by generic code. In practice, associated consts are less commonly specialized because the use cases tend to be better served by associated types or methods.

## 8. Current stabilization status

### RFC and tracking issues

- **RFC 1210** ("impl specialization"): Accepted in 2016. Full text at [rust-lang/rfcs](https://rust-lang.github.io/rfcs/1210-impl-specialization.html).
- **Tracking issue #31844**: The umbrella issue for all specialization work. As of early 2026, still open. No stabilization report has been filed.
- **Tracking issue #40582**: Specifically tracks the lifetime-dispatch unsoundness.
- **PR #71420** (Ralf Jung, 2020): Officially marked `specialization` as unsound with a compile-time warning.
- **PR #68970** (Matthew Jasper, 2020): Implemented `min_specialization` as the sound subset.

### Relationship to the next-generation trait solver

The next-gen trait solver (formerly "chalk-inspired") has been a Rust project goal since 2024H2. As of November 2025, it was adopted by Rust Analyzer and was undergoing crater runs to identify regressions. The goal for 2025H2 was to prepare it for stabilization.

The next-gen solver is a prerequisite for *properly* supporting specialization â€” the old solver's handling of overlap and specificity has known gaps. However, stabilizing the solver does not automatically stabilize specialization. They are separate features with separate soundness requirements.

### Compiler team sentiment

A May 2025 forum thread ("Specialization: are we getting there?") asked about the current state. A November 2025 thread ("New trait solver and simple specialization") explored whether a minimal form could ride the solver stabilization. Neither thread produced a concrete timeline or commitment from the compiler team.

The general sentiment: `min_specialization` is stable enough for std's internal use and is unlikely to be *removed*, but stabilizing it for general use requires resolving the `#[rustc_specialization_trait]`/`#[rustc_unsafe_specialization_marker]` story for external crates. Without that, external crates can only specialize on the compiler's built-in marker traits, which is too restrictive for a stable feature.

### Timeline estimate

**Realistic assessment:** There is no timeline for stabilization. The feature has been unstable since 2016 (a decade). The compiler team's bandwidth is focused on the trait solver, async closures, and edition work. Specialization is not listed as a 2025H2 or 2026H1 project goal. A conservative estimate is 2-4 more years for any stabilization, if it happens at all.

**Risk for saalis:** Low in practice. Nightly Rust is a committed choice for the project (ZST markers, `inventory`, `repr(C)` ABI patterns). `min_specialization` is used extensively in std and is therefore maintained even if unstable. The risk is not breakage but perpetual nightly requirement.

## 9. Alternatives if it never stabilizes

### Sealed trait hierarchies

The most common stable-Rust workaround. Instead of specialization, define a trait hierarchy where a private supertrait controls dispatch:

```rust
mod private {
    pub trait Sealed {}
}

pub trait Fast: private::Sealed {
    fn do_work(&self);
}

impl private::Sealed for Vec<u8> {}
impl Fast for Vec<u8> {
    fn do_work(&self) { /* optimized */ }
}

// General impl via a blanket on a different trait
```

This pattern works when you control all implementations, but it does not solve the orphan rule problems that `min_specialization` solves for saalis.

### Macro-generated impls

Procedural macros can generate specific impls for each type combination, avoiding the need for overlapping impls entirely. The `#[derive(Settings)]` macro could, for instance, enumerate all `(Role, Domain)` combinations and emit non-overlapping impls. This eliminates `default impl` but requires the macro to have knowledge of all roles and domains at derive time â€” feasible for a closed set, problematic for an extensible one.

### Autoref specialization (dtolnay trick)

Documented in dtolnay's [case studies](https://github.com/dtolnay/case-studies/tree/master/autoref-specialization). The trick exploits Rust's method resolution order: if a method can be called without autoref, it takes priority over one requiring `&self`. By defining methods on wrapper types at different reference levels, you get dispatch that resembles specialization.

**Limitations:** Only works in non-generic contexts (typically inside macros). Cannot be used in generic functions where the type is a parameter. The `spez` and `castaway` crates package this pattern for convenient use, but they are fundamentally limited to macro-expansion-time dispatch, not compile-time trait resolution.

### The `try-specialize` crate

Provides `TypeFn` and `TrySpecialize` traits that attempt specialization at the value level using `TypeId` comparison. Works on stable Rust but has a runtime cost (an `if` branch on `TypeId`) and cannot specialize associated types or consts â€” only function behavior.

### For saalis specifically

If `min_specialization` were removed from nightly (extremely unlikely given std's dependence on it), saalis would need to:

1. **`SupportsEntity` blanket bridge:** Replace with explicit `impl SupportsEntity<E> for M` for every association. The `#[register]` derive macro already knows both sides of the association and could emit both impls. More boilerplate, but mechanically automatable.

2. **`SetAccess` defaults:** Replace `default impl<R: Role, D: SettingsDomain> SetAccess<R, Hidden> for D` with macro-generated impls for each `(Role, Domain)` pair that lacks an explicit override. The derive macro has the role list and domain list at hand.

3. **`Secret` blanket from `Credential`:** This is `impl<T: Credential> Secret for T {}` â€” a simple blanket impl, not a specialization. It does *not* actually require `min_specialization` unless there is a conflicting blanket. If it stands alone, stable Rust accepts it.

## 10. How saalis uses `min_specialization`

### 10.1 `SupportsEntity`/`HasMetadata` blanket bridge (section 4.5)

The orphan rule prevents a plugin from implementing `Associate<Rating> for PluginEntity` because neither `Associate` nor `Rating` is local to the plugin crate. The dual-trait pattern splits this into two traits:

- `SupportsEntity<E>` â€” table-side, implemented where the `MetadataTable` ZST lives.
- `HasMetadata<M>` â€” entity-side, implemented where the `EntityType` ZST lives.

A blanket bridge unifies them:

```rust
#![feature(min_specialization)]

default impl<E, M> SupportsEntity<E> for M
where
    E: HasMetadata<M> + EntityType,
    M: MetadataTable,
{}
```

The `default` keyword is essential: without it, any explicit `impl SupportsEntity<Game> for PluginTable` would conflict with the blanket. With `default`, the explicit impl takes priority and the blanket serves as the fallback for `HasMetadata`-declared associations.

The query API uses a single bound â€” `M: SupportsEntity<E>` â€” and both directions of declaration satisfy it. This is the architectural keystone of the plugin metadata system.

**Soundness note:** This blanket uses only trait bounds (`EntityType`, `MetadataTable`, `HasMetadata<M>`). No lifetime parameters are involved. The always-applicable check passes because the bounds narrow the generic space without depending on lifetime information.

### 10.2 `SetAccess` default-access matrix (section 10.8)

The settings system uses `SetAccess<R, A>` to declare at compile time what access level role `R` has to a settings domain. A specialization provides safe defaults:

```rust
// Non-admin roles: Hidden by default (safest)
default impl<R: Role, D: SettingsDomain> SetAccess<R, Hidden> for D {}

// Admin: full access to everything by default
impl<D: SettingsDomain> SetAccess<Admin, ReadWrite> for D {}
```

Specific overrides read naturally:

```rust
impl SetAccess<User, ReadWrite> for Preferences {}
impl SetAccess<Child, ReadOnly> for Preferences {}
```

Without `min_specialization`, the `default impl` for non-admin roles would conflict with both the admin blanket and the specific overrides. Specialization resolves the overlap: specific impl > admin blanket > default blanket.

### 10.3 `Secret` blanket from `Credential` (section 10.2)

```rust
trait Secret {}

// Blanket: all Credential types are Secret.
impl<T: Credential> Secret for T {}
```

This particular blanket does not itself require specialization â€” it is a standard blanket impl. However, it coexists in a system where other blankets may overlap, and the `default impl` machinery on `SetAccess` and `SupportsEntity` creates an environment where `min_specialization` is already required. The `Secret` blanket benefits from this environment: the derive macro uses `T: Secret` bounds to generate `FieldKind::Password` and masking behavior, and the settings engine uses `T: Secret` to trigger encryption on DB write.

The critical property is that secrecy is a type-level fact, not a runtime flag. `ApiKey` implements `Credential`, therefore `ApiKey: Secret`, therefore any settings field of type `ApiKey` gets encrypted storage and masked display. No annotation needed â€” the type carries the policy.
