# Hot/Cold Storage: Eviction and Resurfacing Strategies for In-Memory Database Systems

**Date:** 2026-03-14
**Context:** Saalis content library manager — columnar in-memory hot store (hilavitkutin pipeline engine) with SQLite as persistent cold storage. Single-binary appliance targeting Batocera/loisto, limited RAM, read-heavy workload with bursty writes during catalogue/enrich cycles.

---

## 1. Eviction Strategies

Eviction is the process of removing data from the hot (in-memory) tier when memory pressure demands it. The choice of algorithm determines what stays resident and what gets pushed to cold storage. Every algorithm makes a bet about future access patterns based on past behavior.

### 1.1 LRU (Least Recently Used)

The baseline. Evicts whichever item was accessed least recently. Implemented as a doubly-linked list where every access moves the item to the head; eviction takes from the tail. Simple and well-understood, but has well-known pathologies: a single sequential scan of cold data can flush the entire hot working set (the "scan pollution" problem). Metadata cost is one timestamp or list position per entry.

LRU works well for workloads with strong temporal locality but breaks down under sequential scans and frequency-skewed access patterns. DuckDB's buffer manager initially used LRU but found it inadequate for analytical table scans of larger-than-memory tables ([DuckDB Memory Management, 2024](https://duckdb.org/2024/07/09/memory-management)).

### 1.2 LFU (Least Frequently Used)

Evicts the item with the lowest access count. Better than LRU for frequency-skewed workloads (a few items are very popular) but suffers from "frequency stagnation" — items that were popular long ago accumulate high counts and are never evicted even when they become cold. Requires either decay mechanisms or windowed counting. Metadata cost is one counter per entry, plus optional aging metadata.

Redis implements an approximated LFU using a Morris counter (logarithmic counter) with a configurable decay time, tracking frequency with just 8 bits per key ([Redis LFU vs LRU](https://redis.io/blog/lfu-vs-lru-how-to-choose-the-right-cache-eviction-policy/)).

### 1.3 ARC (Adaptive Replacement Cache)

Developed at IBM Almaden ([Megiddo & Modha, FAST '03](https://www.usenix.org/conference/fast-03/arc-self-tuning-low-overhead-replacement-cache)), ARC maintains four lists: T1 (recently accessed once), T2 (accessed at least twice), B1 (ghost entries evicted from T1), and B2 (ghost entries evicted from T2). The key innovation is that ARC dynamically adjusts the partition between recency and frequency based on workload — if ghost hits on B1 are frequent, it grows T1; if ghost hits on B2 dominate, it grows T2.

ARC is **parameter-free** and self-tuning, unlike 2Q, LRU-2, LRFU, and LIRS which require user-defined parameters. Studies show ARC achieves 26% miss rate reduction over LRU on representative workloads ([IEEE Computer, 2004](https://theory.stanford.edu/~megiddo/pdf/IEEE_COMPUTER_0404.pdf)). The major downside is that ARC is **patented by IBM**, which has historically limited its adoption in open-source projects (ZFS famously had to work around this).

Metadata cost: four lists plus a tuning parameter `p`. Approximately 2x the memory overhead of plain LRU due to ghost lists.

### 1.4 CLOCK (Second Chance)

An approximation of LRU using a circular buffer and a single "hand" pointer. Each entry has a 1-bit reference flag. On access, set the bit to 1. On eviction, the hand sweeps: if bit=1, reset to 0 and advance (give a "second chance"); if bit=0, evict. CLOCK achieves near-LRU miss rates with dramatically lower overhead — no list reordering on every access, just a bit flip ([Wikipedia: Cache Replacement Policies](https://en.wikipedia.org/wiki/Cache_replacement_policies)).

CLOCK is the standard in operating system page replacement (Linux, BSD) precisely because the per-access cost is O(1) with no locking. The trade-off is less precision than true LRU.

### 1.5 2Q (Two Queue)

Proposed by Johnson and Shasha, 2Q uses a FIFO admission queue (A1) and an LRU main queue (Am). New items enter A1; if accessed again while in A1, they promote to Am. Eviction comes from the tail of Am (or from A1 if it exceeds its size limit). This filters out scan pollution — single-access items never pollute Am.

The weakness is the **A1 size parameter**: too large wastes memory on cold data; too small fails to capture warming items. There is no universally correct value. 2Q typically requires tuning per workload.

### 1.6 LIRS (Low Inter-reference Recency Set)

LIRS uses "inter-reference recency" (IRR) — the number of distinct items accessed between two consecutive accesses to the same item — instead of simple recency. Items with small IRR are "hot" (LIR set, always resident); items with large IRR are "cold" (HIR set, candidates for eviction). LIRS can detect patterns that fool LRU, particularly loop-like access patterns.

The downside is implementation complexity and the need to maintain a variable-length history stack. MySQL's InnoDB buffer pool uses a variant of LIRS with "young" and "old" sublists.

### 1.7 SIEVE (2024 — State of the Art)

SIEVE, presented at NSDI 2024 ([Yang et al., NSDI '24](https://www.usenix.org/conference/nsdi24/presentation/zhang-yazhuo)), is a breakthrough: simpler than LRU yet more effective than ARC on most workloads. Like CLOCK, it uses a single pointer ("hand") and a 1-bit visited flag. The critical difference from CLOCK: SIEVE uses a **FIFO insertion order** (not circular reuse) and **does not move objects on access** — it only sets the visited bit.

Key results:
- Up to **63.2% lower miss ratio than ARC** on tested traces
- Lower miss ratio than 9 state-of-the-art algorithms on **45% of 1559 traces** (next best: 15%)
- **2x throughput** of an optimized 16-thread LRU
- Implemented in 5 production cache libraries with **fewer than 20 lines of code** on average

SIEVE is particularly interesting for saalis because it is patent-free, trivially simple to implement, and performs well on both recency-dominant and frequency-dominant workloads. Metadata cost: 1 bit per entry plus a single pointer — the same as CLOCK.

Source: [SIEVE website](https://cachemon.github.io/SIEVE-website/), [NSDI 2024 paper](https://junchengyang.com/publication/nsdi24-SIEVE.pdf)

## 2. Resurfacing and Cache Warming Strategies

Resurfacing is the reverse of eviction: bringing cold data back into the hot store. The key question is whether to do it reactively (on demand) or proactively (prefetch/predict).

### 2.1 Demand-Driven (Reactive)

The simplest approach: on a cache miss, fetch the required data from SQLite into the hot store. This is the "cache-aside" or "lazy loading" pattern. The hot store acts as a read-through cache. Pros: no wasted prefetch work, simple implementation. Cons: first access to any cold item incurs full SQLite read latency; burst patterns (e.g., browsing a collection) cause cascading misses.

### 2.2 Sequential Prefetching

When access patterns are sequential (e.g., scanning a game library sorted by name), prefetch the next N items from cold storage. DuckDB does this for analytical scans — when it detects sequential page access, it reads ahead. SQLite's own OS layer uses `mmap` advisory hints for readahead on sequential access patterns.

For saalis, sequential prefetching is relevant when a user browses a sorted list. If the UI requests items 50-60, prefetch 61-80.

### 2.3 Predictive / Pattern-Based Loading

More sophisticated systems analyze access patterns to predict future needs:

- **Temporal patterns:** If enrichment cycles run nightly, pre-warm the entities that will be processed.
- **Correlation-based:** If accessing game X often leads to accessing games Y and Z (same series, same genre), prefetch related entities.
- **Query-plan-based:** If a query will need a join across entities and metadata, prefetch both sides.

Redis implements cache prefetching by analyzing access logs and preloading data that is likely to be requested soon ([Redis Cache Prefetching](https://redis.io/learn/howtos/solutions/caching-architecture/cache-prefetching)). Production systems often combine this with ML-based prediction for content recommendation engines.

### 2.4 Working-Set Tracking

Microsoft's Hekaton engine (SQL Server in-memory OLTP) tracks hot/cold classification using **exponential smoothing** of sampled record accesses. Rather than tracking every access (expensive), they sample a fraction and use a decay function to estimate access frequency. This was shown to be more accurate than LRU-2 and ARC for identifying cold records that can be safely migrated to secondary storage ([Levandoski et al., ICDE 2013](https://www.microsoft.com/en-us/research/wp-content/uploads/2013/04/ColdDataClassification-icde2013-cr.pdf)).

The companion "Siberia" project extended this to a full migration framework, moving cold records to SSD while maintaining transparent access ([Eldawy et al., VLDB 2014](https://www.microsoft.com/en-us/research/wp-content/uploads/2016/02/p1016-eldawy.pdf)). The key finding: OLTP workloads exhibit highly skewed access — a small fraction of records account for most accesses, making eviction highly effective.

### 2.5 VoltDB's Anti-Caching

VoltDB inverts the traditional model: instead of caching hot data from a disk database, it starts with everything in memory and "anti-caches" cold data to disk. When memory pressure rises, cold tuples (identified via access frequency tracked with a CountMin Sketch) are serialized to disk in blocks. On access to an evicted tuple, the entire block is fetched back. The system merges cold blocks back selectively if data becomes hot again within a time window ([DeBrabant et al., VLDB 2013](https://www.vldb.org/pvldb/vol6/p1942-debrabant.pdf)).

This is conceptually close to what saalis needs — the columnar hot store is the primary working surface, and SQLite is the "anti-cache" destination.

## 3. Dirty Tracking and Write-Back Strategies

When the hot store allows mutations, we need to track what has changed and flush it to cold storage for durability.

### 3.1 Write-Ahead Logging (WAL)

The industry standard. Before modifying data in memory, write a log record describing the change to a sequential append-only file. On crash, replay the log to recover. SQLite's WAL mode is the most relevant implementation for saalis:

- Changes are written to a separate `.wal` file, not directly to the main database.
- A "wal-index" in shared memory allows readers to locate pages quickly.
- **Checkpointing** periodically transfers changes from WAL to the main database file.
- Default checkpoint threshold: 1000 pages. Configurable via `wal_autocheckpoint`.
- WAL allows concurrent readers during writes — readers see a consistent snapshot.

SQLite WAL is well-suited because it converts random writes into sequential appends and allows concurrent read access ([SQLite WAL documentation](https://sqlite.org/wal.html)). For saalis, the hot store mutations could be journaled to a lightweight WAL structure and periodically checkpointed to SQLite.

Source: [Fly.io: SQLite Internals WAL](https://fly.io/blog/sqlite-internals-wal/), [SQLite WAL](https://sqlite.org/wal.html)

### 3.2 Dirty Page/Record Tracking

The most common approach for buffer managers. Each cached item (page, record, or column segment) carries a dirty bit. When modified, the bit is set. A background flusher periodically scans for dirty items and writes them to persistent storage. This is how virtually every database buffer pool works:

- **PostgreSQL** uses a background writer daemon that continuously writes dirty pages when clean buffer count is low.
- **SQL Server** uses both checkpoints (periodic bulk flush) and a lazy writer (continuous background flush under memory pressure).
- **Apache Ignite** marks pages dirty when updated in memory, appends the change to WAL, and later writes dirty pages to partition files during checkpoints ([Ignite Native Persistence](https://ignite.apache.org/docs/latest/persistence/native-persistence)).

For saalis's columnar store, dirty tracking at the **column segment** granularity (rather than individual rows) aligns well with the columnar layout. A segment that has any mutation gets marked dirty and is flushed as a unit.

Source: [CMU 15-445 Lecture Notes](https://15445.courses.cs.cmu.edu/fall2025/notes/04-bufferpool.pdf), [SQL Server Checkpoint/Lazy Writer](https://www.sqlshack.com/sql-server-checkpoint-lazy-writer-eager-writer-and-dirty-pages-in-sql-server/)

### 3.3 Shadow Paging / Copy-on-Write

Instead of modifying pages in place, allocate a new copy, write changes there, then atomically swap the pointer. LMDB and CouchDB use this approach. Advantages: no WAL needed for crash recovery (the old page is always intact), instant recovery (just use the last committed root). Disadvantages: write amplification (full page copy for any change), fragmentation, and historically poor random write performance on spinning disks ([Wikipedia: Shadow Paging](https://en.wikipedia.org/wiki/Shadow_paging), [Ayende: Storage Wars](https://ayende.com/blog/163393/the-storage-wars-shadow-paging-log-structured-merge-and-write-ahead-logging)).

For saalis, shadow paging is less attractive because SQLite already provides WAL-based durability. Layering another CoW mechanism would add complexity without clear benefit.

### 3.4 Epoch-Based Batching

Rather than tracking individual dirty items, group all mutations within a time window (epoch) and flush them as a batch. This amortizes the cost of persistence operations. Combined with the scheduler's work-unit model in saalis, this maps naturally: each catalogue/enrich cycle produces a batch of mutations that can be flushed as one epoch to SQLite within a transaction.

## 4. Industry Implementations

### 4.1 Redis (RDB + AOF)

Redis is purely in-memory with optional persistence. Two mechanisms:

- **RDB snapshots:** Point-in-time dump via `fork()` + copy-on-write. The child process writes the full dataset to disk while the parent continues serving. Compact file, fast restarts, but data loss window equals the snapshot interval.
- **AOF (Append-Only File):** Logs every write command. Three fsync policies: always (safest, slowest), every second (default, up to 1s loss), never (OS-managed, fastest). AOF files are larger and replay is slower than RDB load.
- **Hybrid (since Redis 4.0):** `aof-use-rdb-preamble` combines an RDB snapshot as the AOF prefix with subsequent commands appended. Faster restart than pure AOF, better durability than pure RDB.

Redis has **no eviction-to-disk** — when `maxmemory` is reached, it evicts data entirely (LRU, LFU, random, volatile-ttl policies). Data is gone, not moved to cold storage. This is fundamentally different from what saalis needs.

Source: [Redis Persistence Docs](https://redis.io/docs/latest/operate/oss_and_stack/management/persistence/)

### 4.2 Apache Ignite

Ignite implements true hot/cold tiering with its multi-tier storage:

- **Memory tier:** Off-heap memory with page-based allocation. When native persistence is enabled, Ignite stores all data on disk and caches what fits in RAM.
- **Eviction (persistence off):** FIFO, LRU, or Random-LRU policies when memory regions hit `maxSize`.
- **Page replacement (persistence on):** Random-LRU, where each page stores a last-access timestamp. When a new page is needed and memory is full, the algorithm samples pages and evicts the one with the oldest timestamp.
- **Dirty page flush:** Dirty pages are tracked, appended to WAL, then written to partition files during checkpoints.

Ignite's approach of "everything on disk, cached in memory" is very close to what saalis needs. The key lesson: when persistent storage is the source of truth, the eviction policy choice matters less (data is never lost) and the optimization target shifts to minimizing I/O rather than preventing data loss.

Source: [Ignite Multi-Tier Storage](https://ignite.apache.org/arch/multi-tier-storage.html), [Ignite Eviction Policies](https://ignite.apache.org/docs/ignite2/latest/memory-configuration/eviction-policies)

### 4.3 SingleStore (MemSQL)

SingleStore provides automatic three-tier storage: memory, local SSD cache, and object storage.

- **Rowstore tables:** Fully in-memory, optimized for OLTP point queries.
- **Columnstore tables:** Stored as sorted, compressed segments on disk with memory caching. Data is pushed to object storage as it cools.
- The system automatically manages promotion/demotion between tiers.

The columnstore approach is directly relevant to saalis: store column segments compressed on disk (SQLite), cache hot segments in memory. SingleStore's segment-level caching (rather than row-level) matches a columnar hot store design.

Source: [SingleStore Architecture](https://www.singlestore.com/blog/how-memsql-works/), [SingleStore Database Storage](https://docs.singlestore.com/cloud/manage-data/database-storage/)

### 4.4 DuckDB Buffer Manager

DuckDB's buffer manager is particularly instructive because DuckDB is also an embedded analytical database:

- **Unified memory pool:** Unlike traditional databases that split memory between page cache and operator scratch space, DuckDB's buffer manager controls all process memory. Pages can be evicted to make room for intermediate query results, and vice versa.
- **Large pages (256KB):** Much larger than traditional 4-16KB pages, reducing management overhead for analytical workloads.
- **Lock-free LRU eviction queue:** A concurrent priority queue with LRU policy. Buffers enter the eviction queue when unpinned; eviction happens when new allocations would exceed the memory limit.
- **Temporary directory spill:** When memory is exhausted, operator state (hash tables, sort buffers) spills to a temporary directory on disk.
- **Configurable eviction:** DuckDB records timestamps of `unpin()` events, enabling custom eviction policies.

Key lesson for saalis: the unified memory pool concept is powerful. Rather than having separate budgets for "cached entities" and "query working memory," a single pool with a unified eviction policy avoids fragmentation and simplifies tuning.

Source: [DuckDB Memory Management (2024)](https://duckdb.org/2024/07/09/memory-management), [DuckDB Buffer Manager Issue #748](https://github.com/duckdb/duckdb/issues/748)

### 4.5 SQLite Page Cache

SQLite's own page cache (`pcache1.c`) is what we are building on top of:

- **Hash-table indexed slots:** Pages are stored in slots accessible via a hash table for O(1) lookup.
- **Pinned vs. unpinned:** Pinned pages (actively in use) cannot be evicted. Unpinned pages are candidates.
- **Pluggable interface:** `sqlite3_pcache_methods2` allows replacing the default cache with a custom implementation.
- **Minimum cache size:** 10 pages guaranteed to prevent deadlock during complex operations.
- **Shrink callback:** `xShrink()` requests the cache to free heap memory under pressure.

For saalis, we are effectively building a **layer above** SQLite's page cache. Our columnar hot store caches deserialized, query-ready columnar data. SQLite's page cache handles the raw disk I/O buffering beneath. These are two distinct cache layers with different eviction concerns.

Source: [SQLite Page Cache API](https://sqlite.org/c3ref/pcache_methods.html), [SQLite Architecture](https://sqlite.org/arch.html)

### 4.6 RocksDB Tiered Storage

RocksDB's LSM-tree architecture naturally separates hot and cold data by level:

- **Tiered compaction:** Higher levels (L0, L1) contain recent/hot data; the last level contains the oldest/coldest data.
- **Time-aware placement:** `preclude_last_level_data_seconds` places data older than a threshold on the last level (cold tier) and newer data on the penultimate level (hot tier).
- **Temperature hints:** RocksDB assigns temperature metadata to SST files, hinting to the filesystem which storage medium to use (SSD for hot, HDD/object storage for cold).

The lesson: even within a single store, data naturally stratifies by age/temperature. RocksDB makes this explicit with per-file temperature hints.

Source: [RocksDB Time-Aware Tiered Storage (2022)](https://rocksdb.org/blog/2022/11/09/time-aware-tiered-storage.html), [RocksDB Tiered Storage Wiki](https://github.com/facebook/rocksdb/wiki/Tiered-Storage-(Experimental))

### 4.7 Apache Arrow / DataFusion

DataFusion is an embeddable query engine built on Arrow's columnar format:

- **MemoryPool trait:** All operators must reserve memory before allocating. The pool tracks total usage.
- **Spill-to-disk:** When the pool is exhausted, pipeline-breaking operators (sort, aggregation, hash join) spill intermediate results to local disk.
- **Greedy vs. Fair pools:** Greedy pool works for single-operator spill; FairSpillPool divides memory evenly among partitions when multiple operators need to spill.
- **MemoryReservation:** Operators hold a reservation handle; when they need more, they request from the pool, which may trigger spill in other operators.

DataFusion does not implement a persistent cache — it reads source data on every query. But its memory management patterns (reservation-based allocation, cooperative spill) are directly applicable to saalis's hot store memory budget management.

Source: [DataFusion SIGMOD 2024 Paper](https://andrew.nerdnetworks.org/pdf/SIGMOD-2024-lamb.pdf), [DataFusion Memory Pool PR](https://github.com/apache/datafusion/pull/1526)

## 5. Patterns for the Saalis Use Case

### 5.1 Workload Characteristics

Saalis has a distinctive workload profile:

- **Read-dominant:** Users browse, search, filter, and view entity details far more than they modify data. Read:write ratio likely 50:1 or higher during normal use.
- **Bursty writes:** Catalogue scans and enrichment cycles produce concentrated bursts of writes — hundreds or thousands of entity updates in a short window, then quiet.
- **Small working set:** A user's active browsing session touches maybe 100-500 entities out of a total library of 10,000-100,000+. The "hot" set at any moment is a tiny fraction of total data.
- **Predictable access patterns:** Library browsing is sequential (sorted lists), hierarchical (drill into a collection/genre), or search-driven. These are relatively predictable.
- **Limited RAM:** Batocera appliance may have 1-4 GB total system RAM, of which saalis might get 256-512 MB.

### 5.2 Recommended Eviction Strategy: SIEVE

Given the constraints, **SIEVE** is the strongest candidate for saalis's hot store eviction:

1. **Simplicity:** ~20 lines of code to implement. Fits the single-binary, embedded philosophy.
2. **No parameters to tune:** Unlike 2Q or LIRS, SIEVE adapts without configuration — critical for an appliance that must work out of the box.
3. **Scan resistance:** SIEVE's FIFO ordering with visited-bit filtering naturally resists scan pollution, which matters when browsing large sorted lists.
4. **Low metadata overhead:** 1 bit per entry. On a hot store with 10,000 cached column segments, that is ~1.2 KB of eviction metadata.
5. **High concurrency:** Cache hits only set a bit (no list reordering), so concurrent reads do not contend. This matters even in a sync-core design because the scheduler's work units may access the hot store from multiple subsystems.
6. **Proven performance:** Outperforms ARC on 45% of real-world traces with lower implementation complexity.

Fallback consideration: if SIEVE proves insufficient for saalis's specific access patterns (unlikely but possible), ARC provides the next tier of sophistication — but note the IBM patent concern.

### 5.3 Recommended Write-Back Strategy: Epoch-Based Dirty Tracking + SQLite WAL

For the write path:

1. **Track dirty segments:** Each column segment in the hot store carries a dirty bit. Any mutation sets it.
2. **Epoch batching:** The scheduler batches dirty segments within a work unit's lifecycle. When a catalogue/enrich cycle completes, all dirty segments are flushed to SQLite in a single transaction.
3. **SQLite WAL mode:** Use SQLite in WAL mode so that:
   - Flush writes do not block concurrent reads (UI browsing continues during write-back).
   - Sequential WAL writes are fast even on flash storage.
   - Checkpoint can be triggered at natural boundaries (end of enrichment cycle, idle timeout).
4. **No separate WAL for the hot store:** Since SQLite already provides WAL-based durability, the hot store does not need its own write-ahead log. The hot store is a cache; if it is lost (crash), it is rebuilt from SQLite. The only data at risk is mutations not yet flushed — bounded by the epoch interval.

### 5.4 Recommended Resurfacing Strategy: Demand + Predictive Prefetch

A two-tier approach:

1. **Demand loading (baseline):** Any cache miss triggers a synchronous fetch from SQLite. The fetched data is inserted into the hot store and subject to normal SIEVE eviction. This guarantees correctness with zero wasted work.
2. **Predictive prefetch (optimization):** When the UI requests a page of results (e.g., items 20-40 of a sorted list), prefetch the next page (41-60) asynchronously. This is a simple sequential prefetch that covers the dominant access pattern (scrolling through lists). The scheduler can issue this as a low-priority work unit.
3. **Relation-based prefetch (future):** When viewing entity details, prefetch related entities (same series, same platform, same genre). This is a second-order optimization that can be added later once access patterns are observed in production.

### 5.5 Granularity: Column Segments, Not Rows

The hot store should cache at **column segment** granularity rather than individual rows:

- Columnar stores naturally organize data into segments (groups of values for one column).
- Evicting/surfacing a segment is a single I/O operation.
- Dirty tracking per segment is cheap (one bit each).
- Segment size can be tuned (e.g., 1000-10000 rows per segment) to balance granularity vs. overhead.
- This aligns with how SingleStore and DuckDB manage columnar data.

### 5.6 Memory Budget Management

Adopt DuckDB's unified pool concept:

- A single `MemoryBudget` controls all hot store memory.
- Both cached column segments and transient query working memory draw from the same pool.
- When the budget is exhausted, the SIEVE eviction policy frees segments to make room.
- A configurable `max_hot_memory` setting (default: 50% of available RAM after OS/Batocera overhead) bounds the pool.
- Reserve a minimum cache size (similar to SQLite's 10-page minimum) to prevent thrashing.

## 6. Startup Strategies

How to populate the hot store when saalis starts up.

### 6.1 Lazy (Cold Start)

Start with an empty hot store. Every access triggers a cache miss and a SQLite fetch. This is the simplest approach and guarantees minimal startup time. The downside: the first few minutes of use are slower as the cache warms organically.

Production systems that use lazy warming: DuckDB (no persistent cache, reads source data per query), DataFusion (same pattern).

### 6.2 Full Preload

Load the entire SQLite database into the hot store at startup. Guarantees zero cache misses but is only feasible when the dataset fits in RAM. For a large library on a 1-2 GB RAM appliance, this is not viable.

### 6.3 Prioritized Partial Preload (Recommended)

The middle ground, and what most production systems recommend:

1. **Persist eviction metadata:** On clean shutdown, write a manifest of what was in the hot store (segment IDs + access metadata). On next startup, reload those segments first. This is what SAP ASE (`-cr` startup flag) and Oracle True Cache do — they save the "cache signature" and restore it on restart ([Oracle True Cache Warmup](https://blogs.oracle.com/database/oracle-true-cache-learn-about-cache-warmup), [SAP ASE Cache Warming](http://dcx.sybase.com/1201/en/dbusage/perform-s-4988196.html)).

2. **Priority order:** Load segments in descending order of their last access frequency or recency. Stop when the memory budget is exhausted.

3. **Background loading:** Startup is not blocked by warming. The UI becomes responsive immediately (serving from SQLite on miss), while a background task streams hot segments from SQLite into the cache.

4. **Startup budget:** Reserve a fraction of the memory budget (e.g., 80%) for preloaded segments, leaving 20% for transient query use during the warming phase.

For saalis specifically, the implementation would be:
- On shutdown: write `hot_manifest.json` (or a SQLite table) listing segment IDs and access counts.
- On startup: spawn a background work unit that reads the manifest and loads segments in priority order.
- The manifest is advisory — if it is missing or corrupt, fall back to cold start.

This approach provides near-instant startup (UI works immediately) with rapid convergence to full performance (most-used data loads within seconds).

### 6.4 Domain-Specific Warming

For a content library, we can exploit domain knowledge:

- **Recently added:** New entities are likely to be browsed first (user just added them).
- **Favorites/most played:** If the UI tracks usage stats, warm those entities first.
- **Current collection view:** If the user's last session was viewing "SNES games," warm that collection on startup.

This is analogous to e-commerce platforms warming cache for products featured on the homepage before a sale event ([Cache Warming Explained, Aerospike](https://aerospike.com/blog/cache-warming-explained/)).

## 7. Summary of Recommendations for Saalis

| Concern | Recommendation | Rationale |
|---|---|---|
| **Eviction algorithm** | SIEVE | Simpler than LRU, outperforms ARC, no parameters, patent-free, 1-bit metadata |
| **Eviction granularity** | Column segment | Matches columnar layout, single I/O per evict/surface, cheap dirty tracking |
| **Write-back** | Epoch-batched dirty flush to SQLite WAL | Leverages SQLite's existing WAL, non-blocking reads during flush, natural batch boundaries from scheduler work units |
| **Resurfacing** | Demand + sequential prefetch | Demand for correctness, prefetch for scroll-ahead performance |
| **Memory management** | Unified pool (DuckDB-style) | Avoids fragmentation between cache and query working memory |
| **Startup** | Prioritized partial preload from persisted manifest | Near-instant responsiveness with rapid cache warming |
| **Dirty tracking** | Per-segment dirty bit | 1 bit per segment, flushed in epoch batches |
| **Crash recovery** | SQLite is source of truth; hot store rebuilt from cold | No separate WAL needed for hot store; at most one epoch of mutations lost |
