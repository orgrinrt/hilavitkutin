## Part 1: Cross-Review of SpecWriter (specwriter-stage1.md)

The SpecWriter produced 20 specifications across ~1,480 lines covering P0-2 through P3-4. I focus on wire-level consistency between the SpecWriter's normative text and my wire format specifications.

### 1.1 MetabolicGovernor (P0-2): Wire Format Impact Assessment

The SpecWriter defines a `MetabolicGovernor` abstract contract with a `trait MetabolicGovernor` interface and a `ResourcePressures` struct. This is a host-internal interface; it does not appear on the wire. However, there is a wire-level coupling that the SpecWriter does not address.

**The `state` byte in the heartbeat transport header.** My wire format (Section 9.2, encrypted payload) uses `msg_type = 0x01` for heartbeats. The heartbeat transport header has a 1-byte `state` field at offset 0. The SpecWriter's MetabolicGovernor drives transitions between the four metabolic states encoded in bits 0-1 of this byte.

**Wire-level consistency check:**
- My spec: bits 0-1 of the `state` byte carry the metabolic state. Two bits, four values. Consistent with {Background=0, Idle=1, Active=2, Stress=3}.
- SpecWriter: four states `{Background, Idle, Active, Stress}`. The MetabolicGovernor returns exactly one of these. The SpecWriter does not specify the numeric encoding.

**Finding W-SW-1 (minor).** The SpecWriter's MetabolicGovernor text does not assign numeric values to the four MetabolicState variants for the wire representation. The `state` byte bit layout needs to be explicitly bound to the MetabolicGovernor's output. I propose:

```
#[repr(u8)]
enum MetabolicState {
    Background = 0b00,
    Idle       = 0b01,
    Active     = 0b10,
    Stress     = 0b11,
}
```

This assignment maps directly to bits 0-1 of the `state` byte. The SpecWriter should add this repr(u8) enum to the MetabolicGovernor section. Without it, implementers will choose different encodings and the heartbeat will be silently non-interoperable.

**Finding W-SW-2 (no impact).** The `convergence_override` parameter in the governor interface is not observable on the wire. It affects the dwell time invariant internally. The `convergence_active` bit (bit 3 of `state` byte, per SpecWriter's P0-3) is the wire-observable effect. No conflict.

### 1.2 HeartbeatExtension (P0-3): Wire Format Consistency

The SpecWriter specifies an 18-byte HeartbeatExtension. My Stage 1 data frame format (Section 9) specifies a 36-byte heartbeat payload (18-byte transport header + 18-byte extension). Direct comparison:

| Field | SpecWriter offset | WireEng offset | Width | Match? |
|-------|-------------------|----------------|-------|--------|
| filter_root | 0 | 0 (within extension) | 8 | Yes |
| epoch_current | 8 | 8 | 4 | Yes |
| work_capacity | 12 | 12 | 1 | Yes |
| health_canary | 13 | 13 | 1 | Yes |
| observation_summary | 14 | 14 | 4 | Yes |

**Wire-level agreement:** complete. Both specifications produce identical byte layouts for the 18-byte extension region.

**Finding W-SW-3 (minor).** The SpecWriter assigns `convergence_active` to bit 3 of the state byte and `admin_flag` to bit 4. My wire format in the data frame encrypted payload does not enumerate the individual state byte bits beyond bits 0-1 (metabolic) and bit 2 (gossip_mode). The SpecWriter's bit assignments are additive and consistent. I accept them without reservation.

**Finding W-SW-4 (consistency).** The SpecWriter says "health_canary: Bits 7-4: worst health classification" and "Bits 3-0: count." My wire format in the heartbeat section uses the same packing convention. Both produce `canary = (worst_class << 4) | count`. Consistent.

However, the SpecWriter uses "Bits 7-4" (MSB-first notation) while I use "top 4 bits / bottom 4 bits" (colloquial). Both mean the same thing, but the normative text should pick one bit numbering convention. My conventions section (Stage 1 line 19) says "bit 0 is the least significant bit." Under this convention, the SpecWriter's "Bits 7-4" corresponds to the upper nibble. No inconsistency; just a style alignment note.

### 1.3 ConvergenceGate (P0-5): Wire Format Needs

The SpecWriter defines ConvergenceGate with a `fn is_open(&self) -> bool` interface and four named gates (ControlGate, ConsensusGate, ReencryptionGate, ExitGate). The interface is `is_open()`: a boolean. The opaque projection principle says consumers MUST NOT inspect internals.

**Wire format assessment:** The ConvergenceGate itself does not need a wire representation. It is a local predicate evaluated by each peer independently. The wire-observable effect is the `convergence_active` bit in the heartbeat `state` byte (bit 3). When any gate opens, convergence is in progress. When ExitGate opens and the exit sequence completes, convergence ends and the bit clears.

**Finding W-SW-5 (no wire format needed).** The ConvergenceGate is correctly specified as a host-local boolean predicate. The wire-visible signal is already carried by the `convergence_active` bit. No additional wire format is required. The Savant's tension map question about whether ConvergenceGate needs a wire representation: it does not. The bit in the state byte is sufficient.

However, the timeout behavior of each gate has an indirect wire effect. When a gate times out and "opens with degraded semantics," the peer may emit scars (ScarAttribution::CorrelatedSuspicion for ConsensusGate timeout). These scars are serialized as control log entries (my Stage 1 Section 6, entry type `0x09`). The scar entry payload format I defined (89 bytes) does not distinguish timeout-triggered scars from observation-triggered scars. The `scar_type` field (1 byte) could encode this. I recommend reserving scar_type values for timeout-sourced scars:

```
scar_type values:
  0x01 = SourceConfirmed (transient)
  0x02 = SourceConfirmed (persistent)
  0x03 = PathFault
  0x04 = CorrelatedSuspicion
  0x05 = IntegrityFailure
  0x06 = NoAlternatePath
  0x07 = PartitionAbsence
  0x08 = GateTimeout       // NEW: scar triggered by ConvergenceGate timeout
```

This would allow the control log to distinguish organic scars from timeout-induced scars, which matters for scar severity tiering (the SpecWriter's P0-6). A GateTimeout scar should be Tier 1 (Transient, weight 0.5x) because it reflects a convergence timing issue, not a persistent fault.

### 1.4 ScarSeverityTier (P0-6): Wire Encoding

The SpecWriter defines `ScarSeverityTier` as a `#[repr(u8)]` enum with 4 variants (Excluded=0, Transient=1, Concerning=2, Severe=3). This enum is used in the weighted_scar_count formula but does not appear directly on the wire. The scar_type in my control log entry payload determines the tier mapping.

**Finding W-SW-6 (no wire format needed).** The ScarSeverityTier does not need a wire representation because it is derived from the scar_type, which is already on the wire. The tier assignment table is a lookup function, not a wire field. This is correct.

### 1.5 DomainSeparator Enum (P0-7): Wire Implications

The SpecWriter defines 7 DomainSeparator variants (KeyId, ReencNonce, FoundingHash, PhaseJitter, HbJitter, SessionId, DestinationId). I introduced one additional domain separator in my Stage 1: `"saalis-connid-v1"` for connection ID generation.

**Finding W-SW-7 (missing variant).** The SpecWriter's DomainSeparator enum does not include `ConnId` for my `"saalis-connid-v1"` domain separator. If my connection ID generation approach is accepted (see Tension 3 below), this variant must be added:

```
ConnId = 7,  // "saalis-connid-v1"
             // Input: 32-byte local_CIK + 32-byte remote_CIK
             //        + 32-byte handshake_nonce + u32 session_counter
             // Output: 4 bytes (truncated)
```

If the SpecWriter's `random_u32()` approach is accepted instead, this variant is unnecessary. The resolution depends on Tension 3.

**Finding W-SW-8 (padding rule critical).** The SpecWriter specifies that domain separator strings shorter than 32 bytes are zero-padded to 32 bytes for BLAKE3 keyed mode. This is critical for interoperability and must be explicitly stated in any wire contract that references domain-separated hashing. My relay_key_id derivation (HMAC-BLAKE3) is a different construction, so the padding rule does not apply to relay_key_id. The SpecWriter correctly notes this distinction. Consistent.

### 1.6 Noise Handshake Wire Format (P1-3): Fundamental Disagreement

The SpecWriter specifies different magic numbers for each Noise message type: `0x564B4B4B` ("VKKK"), `0x564B4B52` ("VKKR"), `0x56585831` ("VXX1"), `0x56585832` ("VXX2"), `0x56585833` ("VXX3").

My wire format uses a single magic `0x56524B4F` ("VRKO") for all pre-handshake frames, with a `frame_type` byte (`0x03` = handshake) and an `hs_type` byte (`0x01` = KK, `0x02` = XX) and a `msg_index` byte (0, 1, 2) to distinguish messages.

**Finding W-SW-9 (fundamental).** These approaches are incompatible. The SpecWriter uses 5 different magic values; I use 1 magic + 2 discriminator bytes.

**Wire format argument for my approach:**

1. A single magic value simplifies the receiver's first-four-bytes demux. The receiver checks `== 0x56524B4F`, then reads the type byte. The SpecWriter's approach requires checking against 5 different magic values (or checking the first 3 bytes `0x56` + one of `KK`/`XX` and then the suffix).

2. The `msg_index` byte in my format provides explicit sequencing information that the SpecWriter's magic-only approach encodes implicitly (the message number is the last byte of the magic). This means the SpecWriter's format cannot distinguish "KK message 1 from initiator" from "KK message 2 from responder" by magic alone -- both use the same magic pair but different message types are distinguished by which side sent them. My format uses `msg_index` explicitly.

3. Extensibility: my format can add new handshake patterns (e.g., Noise NK for relay-assisted connections) by adding a new `hs_type` value. The SpecWriter's approach would require allocating a new family of magic numbers (3 for NK: VNK1, VNK2, VNK3).

4. The SpecWriter's format uses a `version: 0x01` byte at offset 4. My format does not have a version in the handshake frame because versioning is handled by the pre-handshake version negotiation (Section 1). The SpecWriter's version byte is redundant if version negotiation happens first (Tension 1 decides this).

**Recommendation:** Use my single-magic + type-byte approach. The SpecWriter's multi-magic approach is aesthetically clean but mechanically inferior for parsing and extensibility.

### 1.7 Connection ID (P1-3): Fundamental Disagreement (Tension 3)

The SpecWriter specifies `connection_id = random_u32()`. I specify a BLAKE3-derived connection ID. This is Tension 3 and is addressed in Part 4 below.

### 1.8 Stream Multiplexing (P1-3): Minor Differences

**SpecWriter's stream header:** 7 bytes (stream_id u16 + sequence u32 + flags u8).
**My stream header:** 6 bytes (stream_id u16 + sequence u32).

**Finding W-SW-10 (flags byte).** The SpecWriter adds a 1-byte `flags` field with FIN, RST, and SACK_PRESENT bits. My format omits this, relying on msg_type to signal stream lifecycle events (FIN/RST would be separate msg_type values). The SpecWriter's approach is more efficient: it avoids needing a separate msg_type for FIN and RST, which would consume two of the limited 7-bit msg_type space. I accept the SpecWriter's 7-byte stream header.

**However:** the SpecWriter's stream_id ranges differ from mine.

| Range | SpecWriter | WireEng |
|-------|-----------|---------|
| 0x0000 | Control stream | Default stream (control) |
| 0x0001 | Gossip stream | Vital tier (0x0001-0x00FF) |
| 0x0002-0x00FF | Application (reactive/bulk) | Vital tier |
| 0x0100-0xFFFE | Bulk transfer | Reactive (0x0100-0x0FFF), Bulk (0x1000-0xFFFE) |
| 0xFFFF | Reserved | Reserved |

**Finding W-SW-11 (stream_id ranges).** My allocation provides a three-tier mapping (vital 0x0001-0x00FF, reactive 0x0100-0x0FFF, bulk 0x1000-0xFFFE) that directly mirrors the VitalQueue's three tiers. The SpecWriter's allocation overloads the 0x0002-0x00FF range for "application streams" that could be either reactive or bulk depending on msg_type. This requires the stream multiplexer to inspect msg_type to determine the queue tier, which adds coupling. My approach determines the tier from stream_id alone (O(1) comparison), which is cleaner for the priority scheduler.

**Recommendation:** Use my three-tier stream_id allocation. The stream_id should encode the queue tier implicitly.

### 1.9 SACK Format: Structural Disagreement

**SpecWriter's SACK:** Inside the stream header (SACK_PRESENT flag in the flags byte), the SACK follows the data payload within the same stream context. Stream-level SACK. Fixed header: 9 bytes (cumulative_ack u32 + block_count u8 + recv_window u32).

**My SACK:** A standalone msg_type `0x05` with a 14-byte fixed header (cumulative_ack u32 + ack_timestamp u32 + ack_delay u16 + block_count u8 + recv_buffer_pct u8 + recv_rate_kbps u16). Also supports piggybacking via bit 7 of msg_type.

**Finding W-SW-12 (SACK scope).** Two disagreements:

1. **Timestamp fields.** My SACK carries `ack_timestamp` (4 bytes) and `ack_delay` (2 bytes). The SpecWriter's SACK does not carry timestamps. Timestamps in SACK are essential for accurate RTT measurement in the LEDBAT congestion controller. Without them, the sender cannot distinguish between a delayed ACK and a long path RTT. The SpecWriter's SACK assumes RTT is measured from heartbeats alone, but heartbeats are sent at 2-30 second intervals. SACK-based RTT measurement provides per-frame granularity. This is important for LEDBAT's one-way delay estimation.

2. **Connection-level vs stream-level.** My SACK is connection-level (it uses a dedicated msg_type with no stream_id). The SpecWriter's SACK is stream-level (each stream has its own SACK state). The design documents specify "per-session tracking of delivered vs sent frame counts" (verkko-protocol SACK defined concept). A "session" corresponds to a connection, not a stream. Stream-level SACK is finer-grained and correct for per-stream reliability, but the congestion feedback (recv_buffer_pct, recv_rate_kbps) is connection-level. Both are needed.

**Recommendation:** Hybrid approach. Use the SpecWriter's stream-level SACK (with the flags byte) for per-stream reliability. Add my timestamp fields to the SACK structure. Use a connection-level congestion feedback field separate from stream-level SACK. This avoids duplicating congestion feedback in every stream's SACK.

### 1.11 Seat Protocol (P2-1): Wire Format Consistency

The SpecWriter produces a comprehensive seat protocol spec. My Stage 1 also specifies seat authentication (Section 8). Comparison:

**Seat Auth Request:**
- SpecWriter: `credential_type u8 + credential_len u16 + credential var + role_requested u8 + client_version u16 + schema_hash [u8;32]`. Variable offset fields (after credential), 32-byte schema hash.
- WireEng: `auth_type u8 + role u8 + credential_len u16 + user_id [16] + schema_hash u32 + sub_count u8 + reserved [3] + event_subs var + credential var`. Fixed-offset fields up to byte 28, CRC-32C schema hash (4 bytes).

**Finding W-SW-13 (seat auth format).** Three specific disagreements:

1. **Schema hash width.** I use CRC-32C (4 bytes) for the schema hash. The SpecWriter uses a 32-byte BLAKE3 hash. CRC-32C is sufficient for schema identity (it detects accidental mismatches, not adversarial substitution, because the schema is exchanged over an authenticated channel). 32 bytes is wasteful in a bandwidth-constrained seat auth message. Recommendation: keep CRC-32C (4 bytes). The adversarial case is covered by the Noise session's authentication.

2. **User ID.** My format includes a 16-byte UUID for user identification. The SpecWriter's format does not include a user ID in the auth message; it relies on the credential alone. A user ID is needed because multiple users may share the same host peer, and the credential (password hash) does not identify the user (different users could have the same password). The user ID is essential. Recommendation: include the 16-byte user_id.

3. **Event subscriptions.** My format includes event subscriptions in the auth request. The SpecWriter separates this into a post-auth SubscriptionRequest message. The SpecWriter's approach is cleaner (separation of concerns) but costs one additional round trip. For LAN-only seat connections, the extra round trip is negligible. Recommendation: accept the SpecWriter's approach. Move subscriptions out of the auth request.

**Finding W-SW-14 (seat message framing).** My seat message frame uses: `direction u8 + msg_type u8 + correlation_id u16 + payload_length u16 + payload`. The SpecWriter uses separate envelopes for each direction (SeatRequest, SeatResponse, SeatEvent, SeatInstruction, SeatProgress). These are compatible -- the SpecWriter's envelopes can be unified into my framing by setting the `direction` byte appropriately. But the correlation_id width differs: mine is u16, the SpecWriter's is u32. At u16, the maximum concurrent outstanding requests is 65,535, which is more than sufficient for a seat. However, the SpecWriter's u32 correlation_id aligns with the SeatRequest envelope. Since seats are LAN-only and low-throughput, u16 is adequate, but u32 costs only 2 extra bytes and provides future-proofing. Recommendation: use u32 correlation_id.

### 1.12 Relay Operations (P3-4): Wire Format Consistency

The SpecWriter specifies relay operations with a different wire structure from my relay operation header (Section 7).

**SpecWriter's relay format:** `op_code u8 + relay_key_id u64 + blob_len u32 + blob var` (no request_id, no flags byte, no is_decoy bit).

**My format:** 13-byte header with `op_code u8 + flags u8 + request_id u16 + relay_key_id u64 + reserved u8`.

**Finding W-SW-15 (relay format).** The SpecWriter's format omits:

1. **request_id.** Without a correlation ID, the relay cannot match responses to requests when multiple operations are in flight. The SpecWriter assumes one operation at a time per session, but the design documents specify constant-rate padding with batch operations (batch_size 10-25). Concurrent operations need correlation. My request_id (u16) is essential.

2. **is_decoy flag.** The constant-rate padding mechanism requires the relay to process real and decoy operations identically. But the sender needs to mark decoys so it can ignore their responses. Without the is_decoy bit, the sender has no way to distinguish real GET responses from decoy GET responses. The SpecWriter's format includes `decoy_count` and `decoy_ids` in the GET request, which is an alternative approach. However, this leaks the decoy count to the relay (the relay sees `id_count` vs `decoy_count`). My approach (per-operation is_decoy flag) keeps the relay unable to distinguish real from decoy even at the operation level. Recommendation: use my per-operation is_decoy flag rather than a separate decoy list.

### 1.13 Control Log Entry: Overlapping Specifications

Both the SpecWriter (P3-3) and I (Section 6) specify control log entry formats. These overlap significantly.

**SpecWriter's format:** `version u8 + entry_type u8 + scope u16 + lamport u64 + prev_hash [32] + author_dk_pub [32] + payload_len u16 + payload var + signature [64]`. Signature covers bytes 0..(78+payload_len).

**My format:** `entry_hash [32] + prev_hash [32] + lamport_clock u64 + entry_type u8 + reserved u8 + channel_index u16 + author_dk [32] + payload_len u16 + payload var + signature [64]`. Signature covers entry_hash (which is BLAKE3 of the content fields).

**Finding W-SW-16 (control log entry design choice).** Two differences:

1. **entry_hash prefix.** My format includes a 32-byte entry_hash at the top. The SpecWriter's format does not include the hash in the serialized entry; it is computed by the receiver. My approach is self-verifying: the entry carries its hash, so the receiver can verify without re-hashing during initial deserialization (just compare). The SpecWriter's approach saves 32 bytes per entry but requires re-hashing for verification. For GSet merge (gossip exchange), my approach enables O(1) duplicate detection by hash lookup before signature verification. The SpecWriter's approach requires hashing before duplicate detection. Recommendation: include the entry_hash prefix. The 32-byte cost is amortized over the entry lifetime (entries are stored, not just transmitted).

2. **Version byte.** The SpecWriter includes a `version u8` at offset 0. My format does not. The SpecWriter's version byte provides forward compatibility for control log format changes. But the control log format is tightly coupled to the mesh state machine; changing it requires a protocol version increment anyway. The version byte is low-cost (1 byte) and low-risk. I accept it as a reasonable precaution.

3. **Entry types.** The SpecWriter defines 11 entry types (0x01-0x0B). I define 13 (0x01-0x0D, adding SUCCESSION and GATEWAY_RELEASE as separate types). The SpecWriter combines GATEWAY_CLAIM and GATEWAY_RELEASE under different op types. My separation is cleaner because claim and release have different payload sizes and semantics. Recommendation: use my 13-type enumeration with the SpecWriter's version byte.

### 1.14 Invite Token: Overlapping Specifications

Both the SpecWriter (P3-3) and I (Section 5) specify invite token formats. The most significant difference:

**SpecWriter:** 182 + (20 * hint_count) bytes. Includes a 64-byte Ed25519 signature. Uses 20-byte AddressHint with mDNS support (type 0x10).

**My format:** 124 + variable address hints. Uses 7/19-byte AddressHint (without padding). No signature.

**Finding W-SW-17 (invite token signature).** The SpecWriter's invite token includes an admin signature (64 bytes). My format does not. The signature prevents token forgery: without it, anyone who knows the CIK and PSK can craft a valid invite token. But the PSK is CSPRNG-generated and single-use, so knowledge of the PSK implies the token was legitimately issued. The signature adds defense-in-depth (even if the PSK leaks, the forger would need the admin DK to sign a new token). The 64-byte cost is significant for QR codes (the SpecWriter's total is ~242 bytes vs my ~138 bytes, pushing QR version from 10 to 13-14). Recommendation: include the signature. The QR code size increase is acceptable, and the defense-in-depth is valuable for a security-critical bootstrap token.

**Finding W-SW-18 (address hint padding).** The SpecWriter's AddressHint is always 20 bytes (with padding for IPv4). My AddressHint is 7 bytes for IPv4 and 19 bytes for IPv6 (no padding). Fixed-size hints simplify parsing (constant offset calculation). Variable-size hints save 13 bytes per IPv4 hint. For a QR code payload where every byte matters, variable-size is preferable. But for wire parsing in the gossip layer (if tokens are forwarded), fixed-size is simpler. Since invite tokens are primarily encoded in QR codes and parsed once, I accept the SpecWriter's fixed 20-byte format for parsing simplicity.

## Part 2: Cross-Review of Architecture Surgeon (archsurgeon-stage1.md)

The Architecture Surgeon performed 5 operations: OP-1 (matter->protocol dependency), OP-2 (SessionResumption promotion), OP-3 (crypto back-references), OP-4 (HeartbeatExtension defined concept), OP-5 (DAG verification). I evaluate wire-level impacts.

### 2.1 OP-1: matter -> protocol dependency declaration

**Wire impact: none.** This operation adds `verkko-protocol` to verkko-matter's dependency list. No wire formats change. MatterTransfer already implicitly uses Stream and LEDBAT; the declaration just makes the dependency explicit.

However, the Surgeon's updated MatterTransfer text adds: "All matter transfers use Stream (bulk tier). LEDBAT pacing ensures matter traffic never degrades foreground use." This has a wire-format implication: matter transfer frames must carry a stream_id in the bulk tier range. Per my stream_id allocation (Section 10.2), bulk streams use `0x1000-0xFFFE`. The MatterTransfer specification should reference this range explicitly.

**Finding W-AS-1 (minor).** The Surgeon's MatterTransfer update should specify that matter transfer streams use stream_id values in `0x1000-0xFFFE` (bulk tier). This binds the "bulk tier" text to the concrete wire format.

### 2.2 OP-2: SessionResumption defined concept

**Wire impact: yes, requires a resumption token wire format.**

The Surgeon promotes SessionResumption as a defined concept in verkko-protocol with these invariants:
- "Resumption token is encrypted under the session's export key and stored locally."
- "0-RTT data is replay-vulnerable: only idempotent operations permitted in the 0-RTT window."
- "Server validates the resumption token's freshness (expiry bound, monotonic ticket counter)."

**Finding W-AS-2 (wire format needed).** The SessionResumption concept needs a wire format for the resumption token. Neither the Surgeon nor the SpecWriter provides byte-level detail for the token itself. The SpecWriter's seat protocol (P2-1) specifies a resumption token for seat sessions (32-byte `session_token` + 8-byte `token_expiry`), but this is specific to seat-to-host-peer sessions. The Surgeon's SessionResumption is for peer-to-peer protocol-level sessions.

I propose the following resumption token wire format for peer-to-peer sessions:

```
SessionResumption Token (80 bytes, encrypted under session export key):
  Offset  Width  Field
  ------  -----  -----
   0      32     session_id: [u8; 32]   -- identifies the session to resume
  32      32     session_key_hash: [u8; 32] -- BLAKE3(session_transport_keys),
                                           for freshness verification
  64      8      expiry: u64            -- HLC timestamp of token expiry
  72      4      ticket_counter: u32    -- monotonic, prevents replay
  76      4      reserved: [u8; 4]      -- zero, future use
```

The token is encrypted with XChaCha20-Poly1305 under the session's export key (from the Noise handshake's `GetHandshakeHash()` output). The 24-byte nonce is derived deterministically: `BLAKE3("saalis-resume-v1" || session_id)[0..24]`. This requires a new DomainSeparator variant:

```
ResumeToken = 8,  // "saalis-resume-v1"
```

**Finding W-AS-3 (0-RTT frame format).** The Surgeon states "0-RTT data is replay-vulnerable: only idempotent operations permitted in the 0-RTT window." This implies a frame format distinction between 0-RTT frames and 1-RTT frames. The receiver needs to know whether a frame arrived in the 0-RTT window (and thus must be treated as potentially replayed) or in the 1-RTT window (replay-safe). I propose adding a bit to the handshake frame:

```
Handshake frame flags (offset 5, currently used for hs_type):
  Bit 7: 0-RTT data follows (1 = this message carries 0-RTT application data)
```

The receiver processes 0-RTT data with the constraint: only idempotent operations (gossip, heartbeat, filter queries). Non-idempotent operations (key distribution, control log entries) must wait for 1-RTT confirmation.

### 2.3 OP-3: crypto back-reference cleanup

**Wire impact: none.** This is a documentation change. The crypto primitives' wire formats are unchanged. The back-references were navigational, not structural. No wire format depends on the direction of the cross-reference.

### 3.5 B5: Matter Chunking

**Wire format needed: yes, critical.**

The Savant correctly identifies this as the third most dangerous blank. My data frame format (Section 9) uses msg_type `0x04` (CHUNK) but does not specify the chunk payload format. The Savant proposes a MatterChunkProtocol with `transfer_id`, `blob_id`, `chunk_index`, `total_chunks`, `chunk_data`, and `bao_proof`.

**Finding W-SA-5 (chunk frame format needed).** I concur with the Savant. A chunk frame format is needed. Building on the Savant's proposal and my data frame format:

```
Matter Chunk Frame (inside encrypted payload, msg_type = 0x04):
  Offset  Width  Field
  ------  -----  -----
   0      8      transfer_id: u64      -- identifies the transfer session
   8      32     blob_id: [u8; 32]     -- BLAKE3 of complete plaintext
  40      4      chunk_index: u32      -- position in the bao tree
  44      4      total_chunks: u32     -- total chunks in the blob
  48      2      chunk_len: u16        -- length of chunk_data (max 1024)
  50      2      proof_len: u16        -- length of bao proof
  52      var    chunk_data: [u8; chunk_len]
   ?      var    bao_proof: [u8; proof_len]
```

Maximum chunk payload per frame:
```
1280 (UDP) - 29 (outer) - 10 (inner) - 6 (stream hdr) - 16 (auth tag) - 52 (chunk hdr) = 1167 bytes
```

Split between chunk_data and bao_proof. A bao proof for a 2GB blob with 1024-byte chunks is ~320 bytes (10 levels * 32 bytes). So chunk_data max is approximately 1167 - 320 = 847 bytes. At 1024-byte chunks (the bao standard chunk size), we need 2 frames per chunk. This is acceptable; the alternative is reducing chunk size to 847 bytes, which works with the bao tree structure.

**Finding W-SA-6 (transfer control frames needed).** The Savant's lifecycle (TransferRequest, TransferBegin) also needs wire formats:

```
TransferRequest (msg_type = 0x0D, new):
  [8]   transfer_id: u64
  [32]  blob_id: [u8; 32]
  [4]   chunk_bitmap_len: u32 (number of u64 words in bitmap)
  [var] chunk_bitmap: [u64; chunk_bitmap_len]  -- bitfield of already-held chunks

TransferBegin (msg_type = 0x0E, new):
  [8]   transfer_id: u64
  [32]  blob_id: [u8; 32]
  [4]   total_chunks: u32
  [32]  bao_root: [u8; 32]  -- root hash for bao tree verification

TransferAbort (msg_type = 0x0F, new):
  [8]   transfer_id: u64
  [1]   reason: u8  (0x01=integrity_failure, 0x02=source_gone, 0x03=cancelled)
```

These three new msg_type values (0x0D-0x0F) must be added to the msg_type registry (Section 9.3).

**Scope recommendation:** B5 should be in Run 11 scope. Without the chunk frame format, the system cannot transfer matter. This is the primary payload of the protocol.

### 3.12 B12: Connection Teardown

**Wire format needed: yes.**

The Savant correctly identifies this as a gap: "connections start but never end." The SpecWriter's stream multiplexing (P1-3) has FIN and RST flags for streams, but there is no connection-level close.

**Finding W-SA-8 (connection close frame needed).** I propose a connection close frame:

```
Connection Close (msg_type = 0x10, new):
  Offset  Width  Field
  ------  -----  -----
   0      1      reason: u8
   2      2      detail_len: u16
   4      var    detail: [u8; detail_len] (UTF-8 diagnostic, max 256)

Reason codes:
  0x00 = NORMAL           -- clean shutdown
  0x01 = IDLE_TIMEOUT     -- no traffic for timeout period
  0x02 = KEY_EXPIRED      -- session keys expired, need rehandshake
  0x03 = MIGRATION_FAIL   -- PATH_RESPONSE not received
  0x04 = INTERNAL_ERROR   -- local error
  0x05 = VERSION_MISMATCH -- discovered after SchemaSync
  0xFF = PROTOCOL_ERROR   -- unspecified protocol violation
```

**Connection close handshake:**
1. Initiator sends CLOSE frame with reason.
2. Responder drains pending data, sends its own CLOSE frame.
3. Both sides enter CLOSING state: accept no new stream data, process remaining SACKs.
4. After 2*RTT or 5 seconds (whichever is shorter): destroy session state and zeroize keys.

This is analogous to TCP's FIN-ACK handshake and QUIC's CONNECTION_CLOSE frame. Without it, connections can only end by heartbeat timeout (3 missed heartbeats = 6-90 seconds depending on metabolic state), which wastes resources and delays cleanup.

**Scope recommendation:** B12 should be in Run 11 scope. Connection teardown is a basic protocol requirement.

## Part 4: Tension Responses

### Tension 1: Version Negotiation Timing

**My position: pre-handshake version negotiation (I defend my Stage 1 approach).**

Arguments:

1. **Wasted work.** If version negotiation happens inside the Noise handshake (SpecWriter's approach), two peers with no common version will: (a) generate ephemeral DH keys, (b) perform 1-2 DH operations, (c) encrypt/decrypt handshake messages, (d) only then discover they are incompatible. This wastes ~50us of CPU per attempt and 1-2 round trips. My pre-handshake probe discovers incompatibility in 1 RTT with zero cryptographic work.

2. **Denial-of-service resistance.** An attacker that sends version-incompatible Noise handshakes forces the victim to perform DH operations before discovering the mismatch. My version probe is unauthenticated but cheap (24 bytes, no crypto). An attacker can spam version probes, but the cost to the victim is a comparison and a 24-byte response, not a DH computation.

3. **The probe is tiny.** 24 bytes. It fits in any UDP datagram. The 5-second timeout with 3 retries is robust against packet loss.

4. **Noise prologue binding.** My format includes the selected version in the Noise prologue (`prologue = MAGIC || selected_version || hs_type = 7 bytes`). This cryptographically binds the handshake to the negotiated version, preventing downgrade attacks. The SpecWriter's approach also includes the version in the handshake payload, but after the handshake is already bound to a specific Noise protocol name (which is version-independent). The prologue binding in my approach is strictly stronger.

5. **The unencrypted-version-probe security concern is minimal.** An active attacker can forge a VersionResponse with `selected_version = 0x0000` (no common version), causing a denial of service. But an active attacker can also forge the first Noise handshake message, causing the same result. The version probe does not leak any secrets (it contains only supported version numbers, which are not secret).

**Concession:** If the SpecWriter's concern is that the version probe reveals which protocol versions the initiator supports (allowing version-targeted attacks), this is valid. However, the same information leaks from the Noise protocol name string, which is transmitted in the clear during XX handshake message 1 (the ephemeral public key is unencrypted). The version set is not more sensitive than the protocol name.

**Verdict: I maintain pre-handshake version negotiation.** The cost/benefit analysis strongly favors avoiding wasted DH operations. The security properties are equivalent.

### Tension 2: BLAKE2b vs BLAKE2s

**My position: BLAKE2s (I defend my Stage 1 approach, with caveats).**

The Noise protocol name strings are:
- SpecWriter: `Noise_KK_25519_ChaChaPoly_BLAKE2b` (64-bit variant)
- WireEng: `Noise_KK_25519_ChaChaPoly_BLAKE2s` (32-bit variant)

Arguments for BLAKE2s:

1. **Target platform.** saalis targets Batocera appliances, which are typically ARM-based single-board computers (Raspberry Pi, Odroid). ARM processors (especially ARMv7, Cortex-A53/A72) execute 32-bit operations natively. BLAKE2s (32-bit operations) is ~20% faster than BLAKE2b (64-bit operations) on 32-bit ARM. On 64-bit ARM (Cortex-A76+), BLAKE2b is ~15% faster.

2. **Noise framework usage.** The Noise handshake uses the hash function for: prologue hashing, handshake hashing (chaining key derivation), and HKDF-like key derivation. The security level of BLAKE2s (128-bit collision resistance) is more than sufficient for these uses. The transport encryption uses ChaCha20-Poly1305, which provides 128-bit security. Using BLAKE2b (256-bit collision resistance) in the handshake provides no practical security improvement when the transport cipher is 128-bit.

3. **Compatibility.** The Noise specification defines both `BLAKE2b` and `BLAKE2s` as valid hash functions. Both are widely implemented in Noise libraries (snow, ring, etc.).

**Caveat and concession:** BLAKE2b is the more commonly used variant in existing Noise deployments (WireGuard uses `BLAKE2s`, but most non-WireGuard Noise implementations default to `BLAKE2b`). Using BLAKE2s might surprise implementers who expect BLAKE2b. However, since verkko is a new protocol and not interoperating with existing Noise deployments, this is not a compatibility concern.

**Performance numbers (approximate, Cortex-A53):**
- BLAKE2s: ~500 MB/s (32-bit native)
- BLAKE2b: ~420 MB/s (64-bit emulated on 32-bit)
- On 64-bit Cortex-A76: BLAKE2b ~650 MB/s, BLAKE2s ~550 MB/s

The Noise handshake performs ~5-8 hash operations. The performance difference is negligible in absolute terms (<1us). The choice is principled, not practically impactful.

**Revised position:** I yield to BLAKE2b. The rationale:

1. The performance difference is negligible for handshake operations.
2. BLAKE2b is the more conventional choice in the Noise ecosystem.
3. verkko's primary BLAKE3 usage (data hashing, domain separation) already uses 32-bit operations. The handshake hash is not a performance bottleneck.
4. If we ever target 64-bit ARM exclusively (likely for future Batocera appliances), BLAKE2b will be faster.

**Verdict: Switch to BLAKE2b.** The Noise protocol name strings become `Noise_KK_25519_ChaChaPoly_BLAKE2b` and `Noise_XX_25519_ChaChaPoly_BLAKE2b`. This is a one-line change in my Stage 1 Section 2.1.

### Tension 3: Connection ID -- Random vs. Derived

**My position: I revise to random_u32() with reservations.**

My Stage 1 specified BLAKE3-derived connection IDs. The SpecWriter specified `random_u32()`. Reviewing the arguments:

**Arguments for derived (my original):**
- Unpredictability for off-path observers (off-path cannot predict next connection_id).
- Migration-safety (bound to session, not IP).
- Domain separation via `"saalis-connid-v1"`.

**Arguments against derived (SpecWriter's critique, implied):**
- `random_u32()` has the same unpredictability property (CSPRNG output is unpredictable by definition).
- `random_u32()` is also migration-safe (the connection_id is a session-level identifier regardless of how it was generated).
- Derived connection_id cannot be generated until the Noise handshake has produced the ephemeral public key. This means connection_id is unavailable during the handshake itself. My handshake frame (Section 2.2) works around this with `initiator_cid` and `responder_cid` fields, but these are pre-handshake IDs that differ from the post-handshake derived ID. This creates a two-phase ID scheme (handshake IDs -> derived IDs) that adds complexity.
- Adding a DomainSeparator variant for connection ID generation adds a registry entry for a non-security-critical use. The connection_id is not a secret; it is visible in the outer header (plaintext).

**Revised analysis:** The connection_id serves one purpose: demultiplexing incoming datagrams to sessions. For this purpose, `random_u32()` is sufficient. The derived approach solves a problem that does not exist: off-path observers cannot inject frames even if they know the connection_id, because the frame payload is authenticated (ChaCha20-Poly1305 AEAD with the outer header as associated data). Knowing the connection_id without the session key is useless.

**Verdict: Switch to random_u32().** This simplifies the implementation, removes a DomainSeparator variant, and eliminates the two-phase ID scheme. The connection_id is reserved values `0x00000000` (handshake) and `0xFFFFFFFF` (future use). All other values are valid.

**Remaining concern:** The collision probability at ~65,536 concurrent connections is ~50% for birthday collisions. A 12-cluster mesh with 12 connections per peer has at most 132 connections per peer. At this scale, the birthday collision probability for u32 is negligible. If collision is detected (received frame with unknown connection_id after a new session is established), regenerate. This is the SpecWriter's approach and it is adequate.

### Tension 5: HostPeer -- Does the Seat-to-Mesh Translation Need Wire-Level Specification?

**Verdict: no additional wire format work for HostPeer.** The wire formats are complete. The missing piece is the protobuf message catalog, which is application-level specification work outside the wire engineer's scope.

## Part 6: Open Questions for Stage 2b

1. **Noise hash function change.** Switching to BLAKE2b requires updating the Noise protocol name strings in Section 2.1, the prologue computation in Section 2.3, and any test vectors that depend on the Noise handshake output. This is a mechanical change but must be verified for consistency.

2. **Hybrid SACK.** The hybrid approach (stream-level SACK for reliability + connection-level congestion feedback) needs a concrete wire format for the connection-level congestion feedback. Proposal: piggyback 3 bytes (recv_buffer_pct u8 + recv_rate_kbps u16) on every SACK, whether stream-level or standalone. This matches the existing verkko-protocol spec (Section 3.3, "Explicit congestion feedback in SACK ACKs").

3. **0-RTT frame marking.** The 0-RTT bit (Finding W-AS-3) needs agreement from the SpecWriter, since it modifies the handshake frame format.

4. **SansIO event surface co-design.** The ProtocolOutput variants (Finding W-SA-1) must be co-designed with the SpecWriter's seat protocol and the Surgeon's SessionResumption to ensure the event surface covers all wire format types, including the new ones proposed in this cross-review.

5. **Chunk payload sizing.** The matter chunk frame header (52 bytes) and bao proof (~320 bytes for 2GB blobs) leave ~847 bytes per frame for chunk data. At 1024-byte bao chunks, this requires 2 frames per chunk. The alternative (using 512-byte bao chunks) would fit in a single frame but doubles the proof overhead for small blobs. This is a calibration decision that needs input from the matter transfer specification author.
