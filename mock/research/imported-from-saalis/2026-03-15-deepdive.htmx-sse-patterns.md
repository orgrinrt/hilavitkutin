# Deep Dive: HTMX + SSE Patterns for Real-Time UI on Low-Power Hardware

**Date:** 2026-03-15
**Scope:** Presentation layer technology selection for saalis web UI
**Context:** Single Rust binary (axum) serving an HTMX-driven web UI on Batocera/loisto ARM SBC appliances (RPi4/5 class). Server-push notifications, live progress updates, library state changes. Gamepad-first, TV-distance interaction.

---

## 1. SSE vs WebSocket for Server-Push-Only Workloads

Server-Sent Events (SSE) and WebSocket solve different problems. For saalis, where the server pushes state updates to the browser and the client never streams data back (user actions are normal HTTP form submissions / `hx-get` / `hx-post`), SSE is the correct choice. Here is why.

### Protocol differences

| Dimension | SSE | WebSocket |
|---|---|---|
| Direction | Server-to-client only | Full duplex |
| Transport | Plain HTTP (text/event-stream) | Upgraded TCP (ws://) |
| Reconnection | Built into the spec (automatic) | Manual implementation required |
| HTTP/2 multiplexing | Yes -- shares the connection | No -- separate TCP socket |
| Proxy/firewall friendliness | Excellent (just HTTP) | Often blocked or requires config |
| Browser API | `EventSource` -- ~20 lines | `WebSocket` -- comparable, but reconnection adds complexity |
| Memory model | Streaming -- processed messages discarded | Buffered -- application manages message lifecycle |
| Connection limit (HTTP/1.1) | 6 per domain (browser limit) | Unlimited (separate socket) |
| Connection limit (HTTP/2) | 100 streams default (negotiable) | Same as HTTP/1.1 for WS |

### Why SSE wins for saalis

1. **Unidirectional fit.** Saalis pushes progress bars, notification toasts, library-changed signals, and scheduler status. The client never streams data upstream -- it sends discrete HTTP requests. SSE matches this exactly.

2. **Automatic reconnection.** The `EventSource` API reconnects automatically with a configurable retry interval. The HTMX SSE extension layers exponential-backoff on top. WebSocket reconnection must be hand-coded.

3. **HTTP/2 multiplexing.** On HTTP/2 (which axum supports via hyper), SSE connections multiplex over a single TCP connection. This eliminates the old HTTP/1.1 six-connection-per-domain limit entirely. A single RPi4 serving its own UI to one browser has no connection pressure.

4. **Memory efficiency on ARM.** SSE connections discard processed messages without buffering. On a 1-4 GB ARM device, this matters. WebSocket libraries tend to accumulate message buffers until application code drains them.

5. **No additional runtime dependency.** axum already speaks HTTP. SSE is just a long-lived response with `Content-Type: text/event-stream`. No need for a WebSocket upgrade path, no `tokio-tungstenite` dependency.

6. **Proxy transparency.** If a user accesses saalis through an SSH tunnel or nginx reverse proxy, SSE works without configuration changes. WebSocket requires explicit proxy support.

### When WebSocket would be needed

If saalis ever needed collaborative editing, real-time chat, or high-frequency bidirectional streaming (game streaming, voice), WebSocket would be appropriate. None of these are in scope.

### Performance parity

Benchmarks from Timeplus and SoftwareMill show SSE and WebSocket deliver comparable throughput and latency for server-push workloads. The performance difference is negligible; the protocol choice should be driven by the communication pattern, not raw speed.

## 2. HTMX SSE Extension: Connection, Reconnection, Event Filtering

### Installation and connection

The SSE extension is loaded as a separate script (not bundled in core htmx). A connection is established declaratively:

```html
<div hx-ext="sse" sse-connect="/events">
  <div sse-swap="library-updated">
    <!-- replaced when "library-updated" event arrives -->
  </div>
  <div sse-swap="download-progress">
    <!-- replaced when "download-progress" event arrives -->
  </div>
</div>
```

The `sse-connect` attribute points to the SSE endpoint URL. The connection is opened when the element enters the DOM and closed when it is removed (or when a parent is swapped out).

### Event filtering

Each child element declares which named SSE event it responds to via `sse-swap="EventName"`. Only matching events trigger a swap on that element. Multiple events can target the same element with comma separation: `sse-swap="event1,event2"`.

For events that should trigger an HTTP request rather than a direct swap, use `hx-trigger`:

```html
<div hx-ext="sse" sse-connect="/events">
  <div hx-get="/library/stats" hx-trigger="sse:library-changed">
    <!-- makes a GET request when "library-changed" fires -->
  </div>
</div>
```

This is powerful: the SSE event acts as a signal, and the element fetches fresh HTML from a normal endpoint. This decouples event payloads from rendered HTML -- the server SSE stream sends lightweight signals, and the browser fetches the expensive rendered fragment only when needed.

### Reconnection

The extension provides two layers of reconnection:

1. **Browser-native:** The `EventSource` API automatically reconnects after connection loss, defaulting to a 3-second retry. The server can set the retry interval via `retry: <ms>` in the SSE stream.

2. **HTMX extension layer:** On top of native reconnection, the extension implements exponential backoff. If the server is down, reconnection attempts space out progressively rather than hammering the endpoint.

### Swap behavior

By default, `sse-swap` replaces the innerHTML of the element bearing the attribute. The swap strategy can be changed with `hx-swap`:

```html
<div sse-swap="notification" hx-swap="beforeend">
  <!-- new notifications are appended, not replaced -->
</div>
```

### Connection lifecycle events

The extension fires HTMX events that can be listened to:
- `htmx:sseOpen` -- connection established
- `htmx:sseError` -- connection error
- `htmx:sseBeforeMessage` -- before processing (cancelable)
- `htmx:sseMessage` -- after processing

## 3. Out-of-Band Swaps for Multi-Region Updates

### The core pattern

`hx-swap-oob` lets a single server response update multiple independent DOM regions. The server returns the primary swap content plus additional elements marked with `hx-swap-oob="true"`:

```html
<!-- Primary content (swapped into the target) -->
<div id="main-content">
  ...updated content...
</div>

<!-- Out-of-band: update the notification badge -->
<span id="notification-count" hx-swap-oob="true">3</span>

<!-- Out-of-band: update the sidebar status -->
<div id="sidebar-status" hx-swap-oob="true">
  <span class="status-dot green"></span> All systems operational
</div>
```

HTMX parses the response, swaps the primary content into the target, then finds all elements with `hx-swap-oob` and swaps each into its matching DOM element (matched by `id`).

### Combining OOB with SSE

This is where the pattern becomes particularly powerful for saalis. A single SSE event can carry multiple HTML fragments that update different page regions simultaneously:

```
event: system-update
data: <div id="download-progress" hx-swap-oob="true">
data:   <progress value="67" max="100"></progress>
data: </div>
data: <div id="queue-count" hx-swap-oob="true">
data:   <span>12 remaining</span>
data: </div>
data: <div id="disk-usage" hx-swap-oob="true">
data:   <meter value="0.73">73%</meter>
data: </div>
```

One event, three DOM updates, zero additional HTTP requests.

### OOB swap strategies

The `hx-swap-oob` attribute supports the same swap strategies as `hx-swap`:

- `hx-swap-oob="true"` -- replaces innerHTML (default)
- `hx-swap-oob="outerHTML"` -- replaces the entire element
- `hx-swap-oob="beforeend"` -- appends to the element
- `hx-swap-oob="afterbegin"` -- prepends to the element
- `hx-swap-oob="delete"` -- removes the element

### Saalis application

For saalis, a single `/events` SSE endpoint can push:
- Download progress updates (progress bars in the main view)
- Notification toasts (appended to a notification container)
- Library stats (game count, storage used -- sidebar or header)
- Scheduler status (curator/housekeeper/doctor subsystem states)
- Scraper results (newly discovered metadata appearing in real time)

All from one SSE connection, all updating different page regions, all without the browser making additional HTTP requests.

### Scale validation

One developer demonstrated running a 50x50 tile grid (2,500 elements) with 1,000 OOB updates per second through SSE without visible performance issues. Saalis will never approach this volume -- typical update rates will be single-digit events per second at most.

## 4. The Idiomorph Extension for Smooth DOM Updates

### Problem: innerHTML replacement destroys state

Default HTMX swaps replace innerHTML. This destroys:
- Focus state (the focused input loses focus)
- Scroll position within replaced containers
- CSS transition mid-states
- Video/audio playback state
- Form input values the user was typing

### Solution: DOM morphing

Idiomorph is a morphing algorithm created by the HTMX team (bigskysoftware). Instead of replacing DOM nodes, it diffs the old and new trees and mutates existing nodes in place. Only changed attributes and text content are updated; unchanged nodes are left alone.

### How it works

Idiomorph builds "id sets" -- for each element in both old and new content, it creates a set of all `id` values found within that subtree. It then uses these id sets to match elements between old and new trees, achieving better matching accuracy than older algorithms like morphdom.

### Usage

```html
<body hx-ext="morph">
  <div hx-get="/content" hx-swap="morph:innerHTML">
    <!-- content morphed, not replaced -->
  </div>
</body>
```

Three swap strategies:
- `hx-swap="morph"` -- morphs outerHTML of the target
- `hx-swap="morph:outerHTML"` -- same, explicit
- `hx-swap="morph:innerHTML"` -- morphs children of the target

### HTMX 4.0 integration

As of htmx 4.0, morph swap styles based on the idiomorph algorithm ship built-in:
- `innerMorph` -- morphs children of the target element
- `outerMorph` -- morphs the target element itself

No separate extension script needed in htmx 4.0+.

### Performance

Benchmarks show idiomorph is approximately 10% slower than morphdom for large DOM morphs, and equal to or faster for smaller morphs. For saalis, where we morph content cards, list items, and status regions (not massive data tables), the overhead is negligible.

### Combining morph with SSE + OOB

This is the recommended pattern for saalis: SSE delivers HTML fragments, OOB distributes them to multiple page regions, and morph ensures updates are smooth without focus loss or flicker. The combination is:

```html
<body hx-ext="sse,morph">
  <div sse-connect="/events">
    <div id="game-grid" sse-swap="library-updated" hx-swap="morph:innerHTML">
      <!-- game cards morphed in place -->
    </div>
  </div>
</body>
```

### hx-preserve

For elements that should never be touched by any swap (e.g., a playing audio element, a video player), `hx-preserve` pins them in the DOM across swaps entirely.

## 5. hx-trigger Patterns: Debounce, Lazy Loading, Infinite Scroll

### Debounced search

```html
<input type="search"
       name="q"
       hx-get="/search"
       hx-trigger="input changed delay:300ms"
       hx-target="#results"
       hx-swap="morph:innerHTML"
       hx-indicator="#search-spinner">
```

- `input` -- fires on every keystroke
- `changed` -- only if the value actually changed (prevents re-firing on focus/blur)
- `delay:300ms` -- debounces: resets the timer on each keystroke, fires 300ms after the user stops typing

For gamepad text entry (on-screen keyboard), `delay:500ms` or higher is appropriate since input is slower.

### Throttled updates

```html
<div hx-get="/status"
     hx-trigger="every 5s"
     hx-swap="morph:innerHTML">
```

The `every` modifier triggers periodic polling. Useful as a fallback or complement to SSE for non-critical status displays.

### Lazy loading

```html
<div hx-get="/game/123/details"
     hx-trigger="revealed"
     hx-swap="outerHTML">
  <div class="skeleton-loader">Loading...</div>
</div>
```

`revealed` fires when the element scrolls into the viewport (uses IntersectionObserver internally). The skeleton placeholder is replaced with real content on first visibility.

For more control:

```html
<div hx-get="/game/123/boxart"
     hx-trigger="intersect threshold:0.5"
     hx-swap="outerHTML">
```

`intersect` with `threshold:0.5` fires when 50% of the element is visible.

### Infinite scroll

```html
<div id="game-list">
  <div class="game-card">...</div>
  <div class="game-card">...</div>
  <!-- sentinel at the end -->
  <div hx-get="/games?page=2"
       hx-trigger="revealed"
       hx-swap="outerHTML"
       hx-target="this">
    <div class="loading-spinner"></div>
  </div>
</div>
```

The sentinel element at the bottom of the list, when scrolled into view, fetches the next page. The server response includes the next batch of game cards plus a new sentinel pointing to page 3. The sentinel replaces itself (`hx-target="this"` + `hx-swap="outerHTML"`), creating a chain.

### Combining triggers

```html
<div hx-get="/library"
     hx-trigger="sse:library-changed, click from:#refresh-btn, every 60s">
```

Multiple triggers can be combined with commas. This element refreshes on SSE signal, manual button click, or every 60 seconds as a fallback.

## 6. Gamepad-Friendly Navigation and TV-Distance UI

This is the area requiring the most custom work, as HTMX has no built-in gamepad or spatial navigation support. The solution is a layered approach.

### The 10-foot UI problem

At TV viewing distance (3+ meters), interfaces need:
- Minimum 24px body text, 32px+ headings (roughly 2x desktop sizes)
- Large touch/focus targets (minimum 48x48 CSS pixels, ideally 64x64+)
- High-contrast focus indicators visible from across the room
- Grid-based layouts with clear spatial relationships
- No hover-dependent interactions (gamepads have no cursor)

### Spatial navigation: current state

The W3C CSS Spatial Navigation Level 1 specification (WICG, migrated to CSS WG) proposes native arrow-key navigation between focusable elements based on visual position. However, as of early 2026, no browser ships an implementation. We cannot rely on it.

### Practical approach for saalis

**Layer 1: CSS focus management (no JS)**

```css
/* All interactive elements must be focusable and visibly focused */
.game-card, button, a, input, select {
  outline: 3px solid transparent;
  outline-offset: 4px;
  transition: outline-color 0.15s, transform 0.15s;
}

.game-card:focus-visible, button:focus-visible,
a:focus-visible, input:focus-visible {
  outline-color: var(--focus-ring-color, #4da6ff);
  transform: scale(1.03);
  z-index: 10;
}

/* Scroll focused element into view */
:focus-visible {
  scroll-margin: 2rem;
}
```

**Layer 2: Gamepad-to-keyboard mapping (~50 lines JS)**

The Gamepad API (`navigator.getGamepads()`) is available in all modern browsers. A small script maps d-pad and analog stick to arrow key events, and face buttons to Enter/Escape/Tab. This is a proven pattern used by Spotify for TV navigation and by gaming console browsers.

```js
// Conceptual -- the actual implementation dispatches KeyboardEvent
function pollGamepad() {
  const gp = navigator.getGamepads()[0];
  if (!gp) return;
  if (gp.buttons[12].pressed) dispatchKey('ArrowUp');
  if (gp.buttons[13].pressed) dispatchKey('ArrowDown');
  if (gp.buttons[14].pressed) dispatchKey('ArrowLeft');
  if (gp.buttons[15].pressed) dispatchKey('ArrowRight');
  if (gp.buttons[0].pressed)  dispatchKey('Enter');
  if (gp.buttons[1].pressed)  dispatchKey('Escape');
  requestAnimationFrame(pollGamepad);
}
```

**Layer 3: Spatial navigation engine (~200 lines JS)**

With arrow key events dispatched, a spatial navigation handler intercepts them and moves focus to the nearest focusable element in the pressed direction. The algorithm:

1. Get the bounding rect of the currently focused element.
2. Filter all focusable elements to those in the pressed direction.
3. Score by distance (Euclidean from center-to-center, weighted toward the primary axis).
4. Focus the closest match.

Spotify's engineering team documented this approach extensively. Their key lessons: keep the focusable-element set small per screen (under 200), use CSS `tabindex` and `data-` attributes to define navigation groups, and handle edge wrapping (left from the leftmost column wraps to the previous row's last element, or not, depending on layout).

**Layer 4: Navigation group hints**

```html
<div class="nav-group" data-nav="grid" data-nav-cols="4">
  <div class="game-card" tabindex="0">...</div>
  <div class="game-card" tabindex="0">...</div>
  <!-- spatial nav within this group respects 4-column grid -->
</div>
```

Grid-aware navigation (knowing the column count) prevents jarring jumps when items have different heights.

### Focus preservation across HTMX swaps

The morph extension preserves focus during swaps. Additionally, HTMX fires `htmx:afterSwap` events where a small handler can restore focus to the previously focused element or the nearest equivalent:

```js
document.body.addEventListener('htmx:afterSwap', (e) => {
  const focusId = e.detail.target.querySelector('[data-restore-focus]');
  if (focusId) focusId.focus();
});
```

## 7. Progressive Enhancement: JS Budget

### What works without JS (pure HTML/CSS)

- Full page navigation (links, form submissions) -- server renders complete pages
- `hx-boost` degrades gracefully: boosted links/forms fall back to normal navigation
- Layout, typography, color themes (CSS custom properties)
- Focus-visible styling
- `<details>`/`<summary>` for accordions
- CSS scroll-snap for carousel-like browsing

### What needs HTMX (14 KB gzipped)

- Partial page updates without full reload
- SSE connections for live updates
- Infinite scroll, lazy loading
- Debounced search
- OOB multi-region updates
- Morph swaps for flicker-free updates
- Toast notifications from SSE events

### What needs custom JS

- Gamepad API polling and spatial navigation (~250 lines)
- On-screen keyboard for gamepad text input (~200 lines)
- Focus management across HTMX swaps (~50 lines)
- Optional: service worker for offline shell (~100 lines)

### Total JS budget

| Component | Size (gzipped) |
|---|---|
| htmx core | ~14 KB |
| SSE extension | ~3 KB |
| Idiomorph extension (htmx <4.0) | ~3 KB |
| Custom gamepad/spatial nav | ~2 KB |
| **Total** | **~22 KB** |

Compare: React + ReactDOM = ~42 KB gzipped, plus application bundle typically 100-500 KB. The saalis UI would ship under 25 KB of JS total, a budget that is trivial for even the most constrained ARM browser.

### Degradation strategy

If JS fails entirely (broken download, CSP issue, ancient browser), saalis should remain functional for browsing and basic operations via standard HTML forms and links. Real-time updates degrade to refresh-the-page. This is acceptable for an appliance UI where the browser environment is controlled.

## 8. Performance on RPi4/5: Browser Rendering Budget

### Hardware context

| Spec | RPi4 | RPi5 |
|---|---|---|
| CPU | Cortex-A72, 4-core, 1.8 GHz | Cortex-A76, 4-core, 2.4 GHz |
| GPU | VideoCore VI | VideoCore VII |
| RAM | 1-8 GB | 2-8 GB |
| Typical browser | Chromium (Bookworm) | Chromium (Bookworm) |

RPi5 is roughly 2-3x faster than RPi4 for browser workloads. Both run Chromium adequately for simple pages; both struggle with heavy JS frameworks.

### Rendering budget constraints

At 60fps, each frame budget is 16.67ms. On RPi4 Chromium:
- CSS animations using `transform` and `opacity` (GPU-composited) run smoothly
- CSS animations involving layout (width, height, margin) cause jank
- DOM operations over ~1000 nodes per frame cause visible stutter
- WebGL performance is poor (7-8fps for moderate scenes on RPi5)
- Scrolling large DOM trees (5000+ nodes) becomes choppy without optimization

### DOM size guidelines for ARM

- **Target:** Keep visible DOM under 1,500 nodes
- **Acceptable:** Up to 3,000 nodes with careful CSS (no expensive selectors, no box-shadow on scrolling elements)
- **Danger zone:** 5,000+ nodes -- scrolling and swaps become sluggish
- **Virtual scrolling threshold:** If the game library has 1,000+ titles, use infinite scroll / pagination rather than rendering all at once

### SSE connection overhead

An idle SSE connection consumes:
- One HTTP/2 stream (negligible -- it's just a held-open response)
- ~50 KB of browser memory for the EventSource object
- Zero CPU when no events are arriving
- A periodic TCP keepalive (every 15-45 seconds, operating system managed)

This is negligible. Even on a 1 GB RPi4, a single SSE connection is unmeasurable.

### Optimization checklist for ARM browsers

1. Use `transform` and `opacity` for all animations (GPU-composited layer)
2. Add `will-change: transform` sparingly to elements that animate frequently
3. Avoid `box-shadow`, `filter: blur()`, and `backdrop-filter` on scrolling containers
4. Use `content-visibility: auto` on off-screen sections for rendering skip
5. Keep CSS selectors simple -- avoid deep descendant selectors
6. Use morph swaps to minimize DOM churn
7. Paginate or infinite-scroll large lists -- never render 1,000 game cards at once
8. Use `loading="lazy"` on all `<img>` elements (native browser lazy loading, no JS needed)
9. Serve images in WebP/AVIF at display resolution -- no oversized assets
10. Set Chromium kiosk-mode flags: `--disable-smooth-scrolling --enable-gpu-rasterization --num-raster-threads=4`

## 9. Comparison with Alternatives

### HTMX vs Stimulus

Stimulus (Hotwire/Basecamp) provides structured JS controllers that attach to DOM elements. It complements server-rendered HTML but does not provide AJAX, SSE, or swap mechanics. You would still need Turbo (or hand-rolled fetch) for partial updates. HTMX subsumes both Stimulus and Turbo in a single declarative system. For saalis, HTMX alone is simpler than the Stimulus + Turbo combination.

### HTMX vs Alpine.js

Alpine.js (~17 KB) provides reactive client-side state and templating, inspired by Vue. It excels at toggling dropdowns, modals, and local UI state without a build step. It does not handle server communication. The common pattern is **HTMX + Alpine.js together**: HTMX manages server interaction, Alpine manages ephemeral client state (open/close, active tab, temporary selections). For saalis, Alpine is worth considering for complex interactive widgets (filter panels, multi-select), but most UI can be handled with pure HTMX + CSS.

### HTMX vs Phoenix LiveView

LiveView maintains a persistent WebSocket connection and a server-side process per session. It diffs rendered HTML on the server and sends minimal patches (5-10x smaller than full fragments). This is architecturally elegant but requires:
- Elixir/Erlang runtime (not Rust)
- Persistent server-side state per connection
- WebSocket infrastructure

For saalis, LiveView's model is inappropriate: we are a Rust binary, we want stateless HTTP for most interactions, and we want the server to do minimal work (ARM SBC with limited CPU). HTMX's "send HTML fragments over HTTP" model is simpler, lighter, and works naturally with axum.

### HTMX vs Turbo (Hotwire)

Turbo provides Turbo Drive (page acceleration), Turbo Frames (partial page updates), and Turbo Streams (WebSocket/SSE multi-region updates). Turbo Streams are conceptually similar to HTMX SSE + OOB. However, Turbo is tightly coupled to the Rails ecosystem, carries more JS weight, and has a steeper learning curve. HTMX is framework-agnostic and lighter.

### Summary matrix

| Criterion | HTMX | Alpine.js | Stimulus+Turbo | LiveView |
|---|---|---|---|---|
| Server communication | Built-in | No | Built-in (Turbo) | Built-in |
| SSE support | Extension | No | Turbo Streams | WebSocket |
| Client state | Minimal | Excellent | Minimal | Server-side |
| JS size (gzipped) | 14 KB | 17 KB | ~25 KB | ~30 KB |
| Server runtime | Any | Any | Any (Rails-oriented) | Elixir only |
| Stateless HTTP | Yes | N/A | Partial | No |
| Progressive enhancement | hx-boost | N/A | Turbo Drive | No |
| Morphing | Idiomorph | No | Morphing (limited) | Diffs |
| Complexity | Low | Low | Medium | Medium-High |

**Recommendation:** HTMX as the primary interaction layer, with the option to add Alpine.js later for complex client-side widgets if pure CSS/HTML proves insufficient.

## 10. Real-World HTMX + SSE Projects at Scale

### Contexte: React-to-HTMX port

The most cited production case study. Contexte, a French media SaaS, replaced their 2-year-old React frontend with Django templates + HTMX:
- **67% code reduction** (21,500 LOC to 7,200 LOC)
- **96% reduction in JS dependencies** (255 packages to 9)
- **88% faster build time** (40s to 5s)
- **50-60% improvement in time-to-interactive**
- **140% increase in Python code** (500 to 1,200 LOC -- server does more work)
- Team became fully full-stack; the hard frontend/backend split disappeared

Important caveat: Contexte is a media reading application (text + images, sophisticated filtering), which is highly amenable to the hypermedia model. Not every application will see these results.

### Three Dots Labs: Go + SSE + HTMX

Three Dots Labs built a microblogging platform demonstrating live SSE updates using Go, HTMX, and Watermill (event-driven architecture). Their architecture: events published through Watermill, SSE endpoint subscribes and pushes HTML fragments, HTMX swaps them into the page. They document the pattern as production-ready for any Go backend.

### OOB + SSE stress test

Adam Giacomelli demonstrated pushing 1,000 OOB updates per second through SSE to a 50x50 tile grid (2,500 DOM elements), with no visible degradation. This validates that the SSE + OOB pattern can handle far more throughput than saalis will ever need.

### HTMX in production: general findings

From a survey of HTMX production users (hamy.xyz, 2024):
- Teams report HTMX works well for small-to-medium codebases
- The biggest adjustment is thinking in HTML fragments rather than JSON APIs
- Server-side template rendering becomes the bottleneck to optimize, not client JS
- Debugging is straightforward: browser network tab shows HTML responses, no source maps needed
- The pattern scales horizontally (stateless HTTP requests) as easily as any REST API

### Applicability to saalis

Saalis is a single-user (or small household) appliance. It will never face "scale" in the concurrent-users sense. The relevant scale dimension is:
- **Library size:** Thousands of games with metadata, artwork, descriptions
- **Update frequency:** Low (scraping runs, downloads completing)
- **DOM complexity:** Grid of game cards with images, each with metadata overlay

This is firmly within HTMX's demonstrated sweet spot. The Contexte case study (media library with search and filtering) is the closest analogue, and their results were excellent.

## Summary: Recommended Pattern for Saalis

```
axum (Rust)
  |
  |-- GET /events          -> SSE endpoint (text/event-stream)
  |     |-- event: library-updated    -> HTML fragment with hx-swap-oob
  |     |-- event: download-progress  -> progress bar fragment
  |     |-- event: notification       -> toast HTML appended to container
  |     |-- event: scheduler-status   -> status badges for 3 subsystems
  |
  |-- GET /games           -> full page (server-rendered HTML)
  |-- GET /games?page=2    -> HTML fragment (infinite scroll page)
  |-- GET /search?q=mario  -> HTML fragment (search results)
  |-- POST /download/123   -> triggers download, returns updated card
  ...

Browser (Chromium on RPi4/5):
  htmx (14 KB) + SSE extension (3 KB) + gamepad nav (2 KB) = ~19 KB JS
  Single SSE connection, morph swaps, OOB multi-region updates
  Gamepad spatial navigation via Gamepad API + custom 200-line handler
  Progressive enhancement: works without JS for basic browsing
```

The total JS payload is under 25 KB. The architecture is stateless HTTP with a single SSE push channel. DOM stays under 1,500 nodes through pagination. Focus is preserved across swaps via morphing. Gamepad navigation requires a small custom JS layer but no framework. This is the lightest viable stack for a real-time, gamepad-navigable library UI on ARM hardware.
