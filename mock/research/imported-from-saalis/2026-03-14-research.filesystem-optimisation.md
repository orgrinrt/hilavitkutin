# Filesystem Access Optimisation Across Platforms

**Date:** 2026-03-14
**Context:** saalis content library manager — installs, moves, links, and manages game ROM/ISO files on Linux (Batocera), macOS, and potentially other platforms.

---

## 1. Copy-on-Write and Reflinks

Copy-on-write (CoW) reflinks allow creating a file copy that shares the same underlying disk blocks as the original. No data is physically duplicated until one of the copies is modified. For a content library manager handling multi-gigabyte game ISOs, reflinks can turn a minutes-long copy into a sub-second operation.

### Filesystem Support

- **btrfs (Linux):** Full reflink support via `FICLONE` and `FICLONERANGE` ioctls. Reflinks are essentially a no-op on btrfs — `copy_file_range` on btrfs shares extents without copying data. Batocera supports btrfs as a userdata partition format and ships a `btrfs_for_windows` driver on its boot partition.
- **XFS (Linux):** Reflink support when the filesystem is created with `reflink=1` (default since `mkfs.xfs` 5.10+). Uses the same `FICLONE` ioctl as btrfs.
- **APFS (macOS):** Supports file cloning via the `clonefile(2)` system call (macOS 10.12+). APFS is the default filesystem on all modern Macs. Clones are instant and space-free until divergence.
- **ext4 (Linux):** No reflink support. ext4 is Batocera's default userdata filesystem.
- **ReFS (Windows):** Supports block cloning via `FSCTL_DUPLICATE_EXTENTS_TO_FILE`, but only on Windows Server editions. Not relevant for our use case.

### Rust API

Two crates provide cross-platform reflink support:

- **[`reflink-copy`](https://lib.rs/crates/reflink-copy):** Exposes `reflink(src, dst)` and `reflink_or_copy(src, dst)`. On Linux, uses `FICLONE` ioctl. On macOS, uses `clonefile(2)`. The `reflink_or_copy` variant returns `Ok(None)` if reflinked, or `Ok(Some(bytes_written))` if it fell back to a regular copy.
- **[`clone-file`](https://github.com/iliazeus/clone-file):** Similar API with `clone_file()` and `clone_or_copy_file()`. Also supports `FICLONERANGE` for partial file cloning.

### Impact for Game Installation

When installing a 4 GB ISO on a btrfs or APFS filesystem, a reflink completes in microseconds instead of the 10-30 seconds a byte-copy would take. The space cost is zero until the user modifies the file (which, for ROMs, is never). This is the single highest-impact optimisation available.

**Recommendation:** Use `reflink-copy::reflink_or_copy` as the default file installation path. It attempts reflink first and falls back transparently. On ext4 (Batocera's default), the fallback adds negligible overhead — it fails fast on `FICLONE` and then performs a regular copy.

## 2. splice / sendfile / copy_file_range — Zero-Copy Kernel Transfers

When reflinks are not available (e.g., on ext4), the next best option is zero-copy kernel-level file transfer, which avoids bouncing data through userspace buffers.

### System Calls

| Syscall | Linux | macOS | Windows | Use Case |
|---------|-------|-------|---------|----------|
| `copy_file_range` | 4.5+ | No | No | File-to-file copy. Creates reflinks on CoW fs. |
| `sendfile` | 2.2+ | Yes (file-to-socket only) | TransmitFile (socket only) | File-to-socket; Linux also supports file-to-file. |
| `splice` | 2.6.17+ | No | No | Pipe-mediated zero-copy between arbitrary fds. |
| `copyfile` | No | Yes | No | macOS high-level copy with metadata. Not zero-copy. |

### Rust Standard Library Integration

Rust's `std::io::copy` on Linux already specialises for these syscalls with the preference ranking: **copy_file_range > sendfile > splice > read+write loop**. When copying from a `std::fs::File` to another `std::fs::File` on Linux, `io::copy` will automatically attempt `copy_file_range` first, which on btrfs/XFS will produce reflinks. If that fails, it falls through to `sendfile`, then `splice`, then the userspace fallback.

This means that on Linux, `std::io::copy` between two `File` handles is already near-optimal. There is no need to manually invoke these syscalls for basic file-to-file copies.

### Crate Access

- **`nix`:** Provides `nix::fcntl::copy_file_range`, `nix::sys::sendfile::sendfile`, and `nix::fcntl::splice` for manual control when needed (e.g., progress reporting with chunked copies).
- **`libc`:** Raw FFI bindings to all these syscalls.
- **`tokio-splice2`:** Provides `splice(2)` integration for tokio async contexts.

### macOS Limitations

macOS's `sendfile` only works for file-to-socket transfers and cannot be used for file-to-file copies. There is no `copy_file_range` or `splice` equivalent. For file-to-file copies on macOS, the best path is: try `clonefile` (APFS reflink) first, then fall back to the userspace read-write loop. The `fcopyfile(3)` function handles metadata but is not zero-copy.

**Recommendation:** Rely on `std::io::copy` for non-reflink file copies — it already does the right thing on Linux. Use `reflink-copy` as a pre-step before falling back to `io::copy`.

## 3. io_uring on Linux

io_uring is Linux's modern async I/O interface (kernel 5.1+), replacing the older `aio` subsystem. It uses a pair of ring buffers shared between userspace and kernel, allowing batched submission and completion of I/O operations with minimal syscall overhead.

### Performance Characteristics

- Throughput can increase by **3.4x** with latency reductions of **68%** compared to traditional syscall-per-operation patterns.
- The primary benefit is **batching**: hundreds of operations submitted via a single `io_uring_enter()` call, amortising syscall overhead.
- Early benchmarks show **~60% improvement** over epoll for network I/O, and **~25% more throughput** for mixed workloads.
- For a single large file copy, io_uring provides minimal advantage over `copy_file_range` — the win comes from batching many small operations.

### Rust Crates

| Crate | Model | Status | Notes |
|-------|-------|--------|-------|
| **`io-uring`** | Low-level ring API | Stable | Direct ring buffer manipulation. No runtime required. |
| **`tokio-uring`** | Tokio-integrated | Active development | Runs on a tokio runtime. Buffer ownership model differs from standard tokio. |
| **`glommio`** | Thread-per-core | Mature (Datadog) | Leverages io_uring from inception. Battle-tested at Datadog. |
| **`monoio`** | Thread-per-core | Active (ByteDance) | Pure io_uring runtime. ~2-3x tokio throughput at 4-16 cores. |

### Batocera Kernel Support

Batocera uses **Linux kernel 6.15** (as of 2025), which has full io_uring support including all modern features (multi-shot accept, fixed files, registered buffers, direct I/O, linked SQEs). io_uring has been available since kernel 5.1 and every Batocera release since the 5.x kernel era supports it.

### Is It Worth It for Bulk File Operations?

For our use case (installing/moving game files), io_uring is **most valuable when performing many operations concurrently** — e.g., extracting an archive with hundreds of small files, or scanning a ROM directory and stat'ing thousands of entries. For a single large file copy, `copy_file_range` is already optimal.

The complication is that io_uring requires an **ownership-based buffer model** — buffers are handed to the kernel and cannot be accessed until the operation completes. This is fundamentally incompatible with Rust's standard `AsyncRead`/`AsyncWrite` traits, which assume borrowed buffers. This is a significant architectural decision point.

### Safety Concerns

Recent research highlights that **async Rust is not inherently safe with io_uring** — cancellation of in-flight operations can lead to use-after-free if buffers are dropped before the kernel finishes with them. Production use requires careful lifetime management.

**Recommendation:** Do not adopt io_uring as a primary I/O backend for v1. The complexity and safety concerns outweigh the benefits for our workload (mostly large sequential file copies, not high-concurrency network I/O). Revisit for directory scanning and archive extraction in a future version. Use `copy_file_range` (via `std::io::copy`) for file transfers — it is already near-optimal for our dominant use case.

## 4. Symlink and Hardlink Strategies

Saalis needs redirectable storage locations — a game's logical path (`/roms/snes/game.sfc`) may point to physical storage on an external drive, a network share, or a different partition. Symlinks are the natural tool for this.

### Platform Behaviour

**Linux:** Symlinks are straightforward. `std::os::unix::fs::symlink(target, link)` works for both files and directories. No privilege requirements. The dominant strategy for ROM organisation.

**macOS:** Standard Unix symlinks work identically to Linux. macOS also has **firmlinks** — bidirectional directory links used internally by APFS to merge the read-only System volume and the writable Data volume. Firmlinks are not user-creatable at runtime; they are configured via `/etc/synthetic.conf` and require a reboot. They are irrelevant for application-level file management. Standard symlinks are the correct tool.

**Windows:** Three distinct link types exist:
- **Symbolic links:** Require `SeCreateSymbolicLinkPrivilege` (admin-only by default, though Developer Mode relaxes this since Windows 10 1703). Separate API calls for files (`symlink_file`) and directories (`symlink_dir`).
- **Junctions:** Directory-only. No privilege requirements. Always use absolute paths. Created via `FSCTL_SET_REPARSE_POINT`. Rustup uses junctions exclusively on Windows.
- **Hard links:** File-only. Same filesystem required. `std::fs::hard_link` works cross-platform.

### Rust API

- **`std::os::unix::fs::symlink`:** Unix symlinks (Linux + macOS).
- **`std::os::windows::fs::symlink_file` / `symlink_dir`:** Windows symlinks.
- **`std::fs::hard_link`:** Cross-platform hard links (same filesystem only).
- **[`symlink` crate](https://crates.io/crates/symlink):** Thin cross-platform wrapper that selects the right call per platform.

### What Breaks

- Symlinks across filesystem boundaries work fine (unlike hard links).
- Symlinks to removable media break when the drive is disconnected — saalis must handle dangling symlinks gracefully.
- Some applications resolve symlinks and operate on the canonical path, which can break relative path assumptions.
- On Windows, `std::fs::remove_dir` must be used to remove directory symlinks, not `remove_file`.
- NFS and CIFS/SMB may not support symlinks or may translate them in unexpected ways.

### Hardlink Considerations

Hard links share the same inode and cannot span filesystems. They are useful for deduplication (two ROM entries pointing to the same physical file) but cannot redirect to external storage. They are invisible to `ls -l` (no visual indicator), which makes debugging harder.

**Recommendation:** Use symlinks as the primary redirection mechanism. On Linux and macOS, use `std::os::unix::fs::symlink`. If Windows support is added, use junctions for directories (no privilege required) and symlinks for files (require Developer Mode). Always validate symlink targets before use and handle `ENOENT` on dangling links.

## 5. Atomic File Operations

When saalis writes metadata databases, configuration files, or download state, a crash mid-write must not corrupt existing data. The standard pattern is **write-to-temp, fsync, rename**.

### The Pattern

1. Create a temporary file in the **same directory** as the target (same filesystem guarantees atomic rename).
2. Write all data to the temporary file.
3. Call `fsync()` on the file descriptor to flush data to disk.
4. Call `fsync()` on the **directory** file descriptor to persist the directory entry.
5. `rename()` the temporary file to the target path. On POSIX, rename is atomic — the target is either the old file or the new file, never a partial write.

### Crates

- **[`tempfile`](https://docs.rs/tempfile/):** The standard Rust crate for temporary files. `NamedTempFile::persist()` performs a rename but does **not** fsync the file or directory before renaming. The documentation explicitly warns: "Neither the file contents nor the containing directory are synchronized, so the update may not yet have reached the disk when persist returns." Insufficient for crash safety on its own.

- **[`atomic-write-file`](https://docs.rs/atomic-write-file/):** Purpose-built for crash-safe atomic writes. Creates a temp file in the same directory, uses `openat`/`linkat`/`renameat` to guard against directory renames during the operation, and calls `fsync` before the final rename. This is the correct tool for writing database files and configuration.

- **[`renamore`](https://docs.rs/renamore/):** Provides additional rename semantics for atomic operations.

### Platform Notes

- On Linux, `rename(2)` is atomic within a single filesystem. Cross-filesystem rename fails with `EXDEV`.
- On macOS, `renamex_np` provides `RENAME_SWAP` and `RENAME_EXCL` flags for additional atomicity guarantees. Standard `rename(2)` is also atomic on APFS/HFS+.
- The `ext4` filesystem with `data=ordered` (default) ensures that data blocks are written before metadata, which provides reasonable crash safety even without explicit fsync — but explicit fsync is still recommended for critical data.

**Recommendation:** Use `atomic-write-file` for all critical writes (SQLite WAL notwithstanding — SQLite handles its own atomicity). Use `tempfile` only for truly temporary scratch files that do not need crash safety.

### Sources

- [atomic-write-file crate](https://docs.rs/atomic-write-file/)
- [tempfile::NamedTempFile documentation](https://docs.rs/tempfile/latest/tempfile/struct.NamedTempFile.html)
- [Rust forum discussion on atomic writes](https://users.rust-lang.org/t/how-to-write-replace-files-atomically/42821)

## 6. Directory Traversal Optimisation

Scanning ROM directories with 10,000+ files is a recurring operation. The choice of traversal crate has a measurable impact.

### Crate Comparison

| Crate | Parallelism | Sorting | gitignore | Speed vs walkdir |
|-------|-------------|---------|-----------|------------------|
| **[`walkdir`](https://crates.io/crates/walkdir)** | Single-threaded | Yes | No | Baseline |
| **[`jwalk`](https://github.com/jessegrosjean/jwalk)** | Multi-threaded (rayon) | Yes (parallel) | No | **~4x walkdir** for sorted results with metadata |
| **[`ignore`](https://crates.io/crates/ignore)** | Multi-threaded (crossbeam) | No (unordered) | Yes (.gitignore, .ignore) | Similar to jwalk; lock-free parallel iteration |

### Analysis

- **`walkdir`:** Single-threaded, well-tested, small dependency. Adequate for directories under ~1,000 files. For 10k+ files, the single-threaded stat() calls become the bottleneck, especially on network-mounted or spinning-disk storage.

- **`jwalk`:** Parallelises at the directory level using rayon. Each directory's entries are read and stat'd in parallel. The ~4x speedup comes primarily from overlapping stat() calls across directories. The streaming iterator API is compatible with walkdir patterns. Built on top of rayon, so it integrates with existing thread pools.

- **`ignore`:** Extracted from ripgrep. Provides `WalkParallel` with a lock-free crossbeam-based parallel iterator. Supports `.gitignore` / `.ignore` filtering with `RegexSet` for matching multiple glob patterns simultaneously. The filtering adds overhead but is useful if ROM directories contain build artifacts or temporary files that should be skipped.

### Directory Entry Caching

On Linux, `getdents64` returns `d_type` which can eliminate stat() calls for type-only checks (is this a file or directory?). Both `jwalk` and `ignore` take advantage of this. On filesystems that do not provide `d_type` (notably XFS with `ftype=0`), a separate stat() is required per entry.

**Recommendation:** Use `jwalk` for ROM directory scanning. It provides the best balance of speed, sorted output, and API simplicity. The `.gitignore` awareness of `ignore` is unnecessary for our use case and adds complexity. For Batocera on ext4 (which supports `d_type`), `jwalk` can skip most stat() calls during pure directory listing.

- [jwalk crate](https://github.com/jessegrosjean/jwalk)
- [ignore crate documentation](https://docs.rs/ignore)
- [WalkDir performance experiment (Rust forum)](https://users.rust-lang.org/t/walkdir-performance-a-small-experiment/24112)

## 7. Archive Extraction

Game ROMs are commonly distributed as ZIP, 7z, RAR, and tar.gz archives. Performance and memory usage during extraction directly affect the user experience.

### Crate Ecosystem

| Format | Crate | Pure Rust | Streaming | Notes |
|--------|-------|-----------|-----------|-------|
| **ZIP** | [`zip`](https://crates.io/crates/zip) | Yes | Yes | Supports store, deflate, bzip2, zstd. The standard Rust ZIP library. |
| **tar** | [`tar`](https://crates.io/crates/tar) | Yes | Yes | Streaming by nature — tar is a sequential format. |
| **gzip/zstd/lz4** | [`flate2`](https://crates.io/crates/flate2), [`zstd`](https://crates.io/crates/zstd), [`lz4_flex`](https://crates.io/crates/lz4_flex) | flate2: optional; zstd: C binding; lz4_flex: yes | Yes | Composable with `tar` for `.tar.gz`, `.tar.zst`, etc. |
| **7z** | [`sevenz-rust2`](https://docs.rs/sevenz-rust2) | Yes | Partial | Fork of unmaintained `sevenz-rust`. LZMA/LZMA2 decompression. |
| **7z (multi-threaded)** | [`sevenzip-mt`](https://lib.rs/crates/sevenzip-mt) | Yes | Yes | Splits files into blocks, compresses/decompresses in parallel on rayon thread pool. Memory-efficient: blocks freed immediately after writing. |
| **RAR** | [`unrar`](https://crates.io/crates/unrar) | No (C binding) | Yes | Wraps the official UnRAR library. RAR's format is proprietary. |
| **Async streaming** | [`async-compression`](https://docs.rs/async-compression) | Yes | Yes | Tokio-compatible `AsyncRead`/`AsyncWrite` adaptors for gzip, zstd, lz4, brotli, etc. |

### Performance Considerations

- **Streaming vs. full-load:** Streaming extraction reads from the archive and writes to disk without holding the entire decompressed content in memory. For a 4 GB ISO inside a 2 GB 7z archive, full-load decompression would require 4+ GB of RAM. Streaming extraction uses only the compression dictionary size (typically 64 MB-1 GB for LZMA2). Always prefer streaming.

- **Parallel decompression:** The `sevenzip-mt` crate splits data into blocks and decompresses in parallel. This is significant for 7z archives with solid compression (where all files are compressed as a single stream), as single-threaded decompression is CPU-bound.

- **ZIP random access:** ZIP's central directory allows extracting individual files without reading the entire archive. This is useful for selective installation (e.g., extracting only a specific ROM from a multi-game archive).

- **Memory management:** The `sevenzip-mt` approach of freeing compressed blocks immediately after decompression is worth emulating. For large archives, memory usage should be bounded to the compression dictionary size plus one output buffer.

**Recommendation:** Use `zip` for ZIP files (most ROM distributions), `tar` + `flate2`/`zstd` for tarballs, and `sevenz-rust2` or `sevenzip-mt` for 7z files. Wrap extraction in a progress-reporting layer that tracks bytes read from the archive stream. For RAR, depend on `unrar` (C binding) — the format is proprietary and no pure-Rust implementation exists. Consider `async-compression` if extraction is integrated into the async web layer for download-and-extract pipelines.

## 8. Checksum Computation

Verifying file integrity after download or installation requires hashing multi-gigabyte files. Hash algorithm choice directly affects verification time.

### Algorithm Comparison

| Algorithm | Speed (single-core) | Speed (multi-core) | SIMD | Cryptographic | Crate |
|-----------|--------------------|--------------------|------|---------------|-------|
| **BLAKE3** | ~3-4 GB/s | ~92 GB/s (16 cores) | SSE2, SSE4.1, AVX2, AVX-512, NEON, WASM | Yes | [`blake3`](https://crates.io/crates/blake3) |
| **xxHash3 (XXH3)** | ~31 GB/s | Single-threaded only | Yes | No | [`xxhash-rust`](https://crates.io/crates/xxhash-rust) |
| **SHA-256** | ~0.5 GB/s | Not parallelisable | SHA-NI extension | Yes | [`sha2`](https://crates.io/crates/sha2) |
| **CRC32** | ~20+ GB/s | Not parallelisable | SSE4.2 | No | [`crc32fast`](https://crates.io/crates/crc32fast) |

### BLAKE3 Architecture

BLAKE3 is internally a Merkle tree, which enables two forms of parallelism:

1. **SIMD parallelism:** Multiple hash blocks processed in a single instruction. The `blake3` crate auto-detects CPU features at runtime and selects the optimal implementation (AVX-512 on modern x86, NEON on ARM/Batocera).
2. **Multi-threading:** The Merkle tree structure allows different chunks of the file to be hashed on different threads, then combined. The `blake3` crate provides `Hasher::update_rayon()` for automatic multi-threaded hashing.

For a 4 GB ISO on a modern machine with AVX2, single-threaded BLAKE3 hashing takes approximately 1 second. With `update_rayon()` on 4 cores, this drops to ~250 ms. This is fast enough to hash files during or immediately after download without noticeable delay.

### xxHash3 vs BLAKE3

xxHash3 is faster on a single core (~31 GB/s vs ~4 GB/s for BLAKE3) but is not cryptographic — it does not resist deliberate collisions. For **integrity verification against accidental corruption** (bit rot, incomplete downloads), xxHash3 is sufficient and faster. For **integrity verification against tampering** (verifying ROM authenticity), BLAKE3 is required.

xxHash3 is not parallelisable across multiple threads for a single file, so for very large files on multi-core systems, BLAKE3 with rayon can actually be faster in wall-clock time despite lower single-core throughput.

### Practical Considerations

- Most ROM distribution sites provide CRC32 or MD5 checksums. Saalis should support verifying these legacy checksums even if it uses BLAKE3 internally.
- Hashing is I/O-bound on spinning disks and most SSDs. On a SATA SSD (~500 MB/s), even SHA-256 is not the bottleneck. The algorithm matters primarily on NVMe storage (3-7 GB/s) or when hashing from the page cache.
- Advising the kernel with `posix_fadvise(POSIX_FADV_SEQUENTIAL)` before hashing can improve read-ahead behaviour and throughput by 10-20% on some filesystems.

**Recommendation:** Use BLAKE3 as the primary internal hash algorithm. Its combination of cryptographic strength, SIMD acceleration, and multi-core parallelism makes it the best all-around choice. Store BLAKE3 hashes alongside legacy CRC32/MD5 for compatibility with ROM databases. Use `posix_fadvise` (via the `nix` crate) to hint sequential access before hashing large files.

## 9. File Metadata Caching and Change Detection

Avoiding redundant `stat()` calls and detecting when ROM files have been added, removed, or modified externally is important for keeping the library up to date without full re-scans.

### The `notify` Crate

The [`notify`](https://crates.io/crates/notify) crate (v8.x) is the standard cross-platform file watching library in Rust. It is used by rust-analyzer, alacritty, deno, cargo-watch, and watchexec.

**Backend selection:**
| Platform | Backend | Notes |
|----------|---------|-------|
| Linux/Android | inotify | Per-file/directory watch. 8192 default watch limit (`/proc/sys/fs/inotify/max_user_watches`). |
| macOS | FSEvents | Coarse-grained directory-level events. Lower overhead than kqueue for large trees. |
| macOS (alt) | kqueue | File-level precision. Higher fd usage. |
| Windows | ReadDirectoryChangesW | Directory-level with recursive flag. |
| Fallback | Polling | Works everywhere. Configurable interval. |

**Caching:** `notify` v8 includes a `RecommendedCache` feature that enables file ID caching on Windows and macOS (where rename detection requires it) and disables it on Linux (where inotify provides rename cookies natively).

### inotify Limitations on Batocera

inotify watches are per-directory. For a ROM library with 500 directories and 10,000 files, this requires ~500 watches — well within the default 8,192 limit. However, inotify does **not** work across filesystem boundaries (e.g., if a ROM directory is a mount point for an external drive, the parent directory's watch will not see changes within the mount). Each mount point requires its own set of watches.

inotify also does not detect changes made while the watcher was not running. On startup, saalis must perform a full scan and compare against the cached state.

### Metadata Caching Strategy

Rather than watching every file individually, a practical approach is:

1. **On startup:** Full directory scan with `jwalk`, recording `(path, mtime, size)` tuples. Compare against the stored catalogue to detect additions, removals, and modifications.
2. **While running:** Use `notify` to watch ROM root directories. On events, update the catalogue incrementally.
3. **Optimisation:** Cache `stat()` results during a scan and reuse them for subsequent operations (checksum verification, display) within the same session. Invalidate on `notify` events.

**Recommendation:** Use `notify` with the FSEvents backend on macOS and inotify on Linux. Perform a full scan on startup (unavoidable for correctness). Use `notify` for incremental updates during runtime. Cache metadata aggressively within a session, invalidating on watch events.

- [notify crate (GitHub)](https://github.com/notify-rs/notify)
- [notify 8.2.0 documentation](https://docs.rs/crate/notify/latest)
- [Notify Rust Guide 2025](https://generalistprogrammer.com/tutorials/notify-rust-crate-guide)

## 10. Platform-Specific Filesystem Features

### Filesystem Comparison for Batocera/Embedded

| Feature | ext4 | btrfs | f2fs | XFS |
|---------|------|-------|------|-----|
| **Default on Batocera** | Yes | Optional | No | No |
| **Reflinks** | No | Yes | No | Yes (mkfs flag) |
| **Compression** | No | Yes (zstd, lzo) | No | No |
| **Checksumming** | Metadata only | Data + metadata | Metadata only | Metadata only |
| **fallocate** | Yes | Yes | Yes | Yes |
| **d_type in getdents** | Yes | Yes | Yes | Yes (ftype=1) |
| **Max file size** | 16 TB | 16 EB | 3.94 TB | 8 EB |

### Performance Characteristics (2025 Benchmarks)

Based on Phoronix Linux 6.11 and 6.15 benchmarks on NVMe storage:

- **ext4 and XFS** are consistently the fastest for sequential writes and general file operations. ext4 is the safest default for Batocera.
- **btrfs** pays a performance penalty for its CoW semantics on write-heavy workloads (ranked last or second-to-last in SQLite and sequential write benchmarks). However, its reflink support makes file "copies" effectively free, which can offset this for ROM management workflows.
- **f2fs** shows mixed results — slightly better than ext4 for file writes, slightly worse for reads with very large file counts. Designed for flash storage (SD cards, eMMC), which is relevant for some Batocera hardware.

### fallocate for Preallocation

`fallocate(2)` preallocates disk blocks without writing data, preventing fragmentation for large files. On ext4, this is a metadata-only operation (fast). On btrfs, it reserves space in the allocator.

The `nix` crate exposes `nix::fcntl::fallocate` with flags:
- `FALLOC_FL_KEEP_SIZE`: Allocate beyond EOF without changing file size (useful for download preallocation).
- `FALLOC_FL_ZERO_RANGE`: Zero a range without deallocation.
- `FALLOC_FL_PUNCH_HOLE | FALLOC_FL_KEEP_SIZE`: Create sparse holes (useful for partial downloads).

**Use case for saalis:** When beginning a large file download (e.g., 4 GB ISO), call `fallocate` to preallocate the full size. This:
1. Reports out-of-space errors **before** the download starts, not 3 GB in.
2. Reduces file fragmentation on ext4.
3. Allows the filesystem to allocate contiguous extents.

### posix_fadvise for I/O Hints

`posix_fadvise(2)` tells the kernel how a file will be accessed:
- `POSIX_FADV_SEQUENTIAL`: Aggressive read-ahead (useful before hashing or copying a large file).
- `POSIX_FADV_DONTNEED`: Drop pages from cache after use (useful after hashing a file to avoid evicting other cached data).
- `POSIX_FADV_WILLNEED`: Begin read-ahead immediately (useful to prefetch a file before the user clicks "install").

The `nix` crate exposes these via `nix::fcntl::posix_fadvise`.

### macOS-Specific Features

- **APFS clonefile:** Covered in section 1. The primary macOS-specific optimisation.
- **F_NOCACHE:** Via `fcntl`, disables the unified buffer cache for a file descriptor. Useful for large sequential reads (hashing, copying) to avoid polluting the cache.
- **F_RDAHEAD:** Controls read-ahead behaviour. `fcntl(fd, F_RDAHEAD, 1)` enables aggressive read-ahead.

**Recommendation:** Default to ext4 on Batocera for reliability and performance. Recommend btrfs to users who want instant "copies" via reflinks (document the trade-off: slower writes, free copies). Use `fallocate` for download preallocation on all Linux filesystems. Use `posix_fadvise(SEQUENTIAL)` before large sequential reads. On macOS, use `F_NOCACHE` for large file operations to avoid cache pollution.
