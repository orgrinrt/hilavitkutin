# API Designer Review: Verkko Decomposition Quality and Interface Contract Cleanliness

**Reviewer:** API Designer archetype (20+ years, interface contracts, dependency hygiene)
**Date:** 2026-03-28
**Scope:** All six verkko design documents + taxonomy
**Method:** Full read of every document, cross-referencing every defined concept, every dependency declaration, every cross-document reference

---

## 1. Per-Document Findings

### 1.1 verkko-crypto

**Boundary cleanliness.**

- **Boundary leak (outward): CPSK.** Section 2.6 describes `on_cpsk_renegotiation` which calls `distribute_via_keybundle(channel, K_epoch_new, Rotate)`. KeyBundle is a verkko-mesh concept. The function signature references `shared_channels(cluster_a, cluster_b)`, which requires knowledge of cluster topology (a mesh concept). The section header even says "CPSK-seeded post-compromise security (PCS) -- mechanism" but the code walks cluster-pair channels and distributes via KeyBundle. The *mechanism* (HKDF derivation math) belongs here. The *trigger scheduling and distribution* belongs in verkko-mesh. The document acknowledges this split ("For PCS trigger scheduling and distribution, see verkko-mesh") but the `on_cpsk_renegotiation` pseudocode still contains the distribution call. The pseudocode should stop at producing `K_epoch_new` and let verkko-mesh handle distribution.

- **Boundary leak (outward): Device recovery flow (Section 1.4).** This section references enrollment, SAS verification, KeyBundle delivery, filter sync, admin revocation, and re-encryption. All of these are verkko-mesh concepts. The device recovery *cryptographic* operations (generating new DK + DEK, the Noise handshake with PSK) belong here. The *enrollment flow*, *SAS verification window*, *KeyBundle delivery*, and *filter sync* belong in verkko-mesh. The section mixes both without clear separation.

- **Boundary leak (outward): Genesis bootstrap (Section 1.3).** References "CRDTs at zero", "control log entry", "Metabolic SM in Background" -- all mesh concepts. The cryptographic genesis (key generation, self-test, founding transcript) belongs here. The CRDT initialization and metabolic state belong in verkko-mesh.

- **Boundary leak (outward): Channel key identification (Section 1.5).** References "channel_index" and "channel" as organizational concepts. These are mesh-layer concepts. The crypto document should define key_id derivation without assuming a channel abstraction.

- **Boundary leak (outward): Epoch triggers (Section 2.4).** References membership change, KeyBundle, verkko-mesh concepts. The ratchet math belongs here; the trigger policy belongs in verkko-mesh.

**Dependency hygiene.** Correctly declares no dependencies. However, body content references verkko-mesh concepts (Enrollment, KeyBundle, Convergence, filter sync) extensively. These are documented as forward references ("see verkko-mesh: X") but the amount of mesh-layer knowledge embedded in this supposedly dependency-free document is concerning. A reader of verkko-crypto alone would encounter numerous unexplained concepts.

**Defined concept completeness.**

- **Missing defined concept: CIK.** The body (Section 1.1) introduces CIK (Cluster Identity Key) and its signing key. CIK is not a defined concept in verkko-crypto, yet it is generated here (genesis bootstrap) and used in multiple crypto operations. CIK IS defined in verkko-mesh, but the genesis bootstrap in verkko-crypto generates it. This is a boundary problem: either CIK generation belongs in verkko-mesh (and verkko-crypto provides the primitives), or CIK should be a defined concept here.

- **Missing defined concept: FoundingHash.** Section 1.3 defines a founding_hash that serves as mesh identity. This is referenced nowhere in the defined concepts of any document but is clearly a significant identifier.

- **Missing defined concept: CPSK.** Section 2.6 operates on CPSKs extensively. CPSK is defined in verkko-mesh, not here, yet the PCS mechanism section treats it as a core input.

- **All declared defined concepts are referenced.** Every concept in the Defined Concepts section appears in the body.

**Cross-reference integrity.**

- "(see verkko-mesh: Enrollment)" -- verkko-mesh defines Enrollment. Correct.
- "(see verkko-mesh: KeyBundle)" -- verkko-mesh defines KeyBundle. Correct.
- "(see verkko-protocol: Frame)" -- verkko-protocol defines Frame. Correct.
- "(see verkko-relay: TerritorialReencryption)" -- verkko-relay defines TerritorialReencryption. Correct.
- "(see verkko-protocol: MetabolicState)" in the domain separator registry (metabolic phase jitter) -- verkko-protocol defines MetabolicState. Correct.

**Standalone readability.** Moderate. A reader encounters CIK, CPSK, KeyBundle, clusters, control logs, enrollment, filter sync, and metabolic state without definitions. The forward references are present but numerous. The document reads more like a chapter of a unified design than an independent specification.

### 1.2 verkko-protocol

**Boundary cleanliness.**

- **Boundary leak (inward from mesh): Section 3.3.** References "Conductor (see verkko-mesh: ResourceGovernance)" for LEDBAT target delay. ResourceGovernance is a verkko-mesh concept, but verkko-protocol does not declare a dependency on verkko-mesh. This is either an undeclared dependency or a boundary violation. The protocol document uses the Conductor concept to parameterize LEDBAT but does not list verkko-mesh in its dependencies.

- **Boundary leak (inward from mesh): Section 3.4 (Heartbeat).** The heartbeat payload includes `observation_summary` with the reference "see verkko-mesh: ObservationPathForwarding". verkko-protocol should not reference mesh concepts if it does not depend on mesh. The heartbeat format is defined here, but the semantic meaning of several fields (observation_summary, gossip_mode, filter_root) requires mesh knowledge.

- **Boundary leak (inward from mesh): Section 6.3 (MetabolicState).** References "Conductor (see verkko-mesh: ResourceGovernance)" extensively. The MetabolicState is defined as a verkko-protocol concept but its governor inputs are described in terms of mesh-layer resource governance. This creates a semantic dependency on verkko-mesh without a declared dependency.

- **Boundary leak (inward from relay): Section 6.2.** References "re-encryption thread" and "pre-batch persist on this thread's NonceSource before each batch". Re-encryption is a verkko-relay concept. The threading model includes relay-specific threads.

**Dependency hygiene.**

- Declared: verkko-crypto (FrameKey, TransportAEAD, NonceSafety, EpochRatchet, KeyZeroization). All used in body. Correct.
- **Undeclared dependency on verkko-mesh.** The body references ResourceGovernance, ObservationPathForwarding, and Conductor from verkko-mesh. These are used to parameterize heartbeat intervals, LEDBAT targets, and metabolic state transitions. This is a real dependency that is not listed.
- **Alternative interpretation:** The protocol document defines MetabolicState as its own concept and merely notes that the governor *could* be implemented using mesh-layer concepts. But the text says "derived from the Conductor (see verkko-mesh: ResourceGovernance)" which is a concrete dependency, not a note.

**Defined concept completeness.**

- **Missing defined concept: SACK.** The body references SACK acknowledgements and SACK delivery ratio extensively (Sections 3.3, 3.4) but SACK is not a defined concept. It is used as an input to health scoring (verkko-mesh) and to congestion control.

- **Missing defined concept: 0-RTT Resumption.** Listed in the taxonomy's scope for verkko-protocol but not present as a defined concept or in the body. The taxonomy lists it as a medium-priority gap.

- **Missing defined concept: ConnectionMigration.** Section 3.3 describes QUIC-style PATH_CHALLENGE/PATH_RESPONSE connection migration but it is not a defined concept despite being a significant protocol behavior.

- **MetabolicState defined here but heavily depends on mesh concepts.** The defined concept declares invariants about dwell time and phase jitter, but the body's description of governor inputs requires ResourceGovernance from verkko-mesh.

**Cross-reference integrity.**

- "(see verkko-mesh: ResourceGovernance)" -- verkko-mesh defines ResourceGovernance. Correct name, but undeclared dependency.
- "(see verkko-mesh: ObservationPathForwarding)" -- verkko-mesh body contains Section 5.4 "Observation-path forwarding" but there is no defined concept called "ObservationPathForwarding" in verkko-mesh's Defined Concepts section. **Broken reference.**
- "(see verkko-crypto: NonceSafety)" -- verkko-crypto defines NonceSafety. Correct.

**Standalone readability.** Good for the wire format and transport sections. Degrades significantly when the metabolic state machine and heartbeat semantics are discussed, due to undeclared mesh dependencies.

### 1.3 verkko-mesh

**Boundary cleanliness.**

- **Boundary leak (inward from matter): Mechanism Reference Table entry 16.** Cuckoo filters and Merkle reconciliation are listed with home "4.2 (verkko-matter)" but the mesh document contains no body section 4.2. The table correctly attributes the home to verkko-matter. However, several body sections reference cuckoo filters and filter state without deferring to verkko-matter (e.g., filter sync in Section 2.5, filter root in the heartbeat description, filter reconciliation in convergence cost).

- **Boundary leak (inward from relay): Convergence Front 3.** Describes bulk re-encryption with flash budget moderation and deterministic nonces. The re-encryption mechanism belongs in verkko-relay. The mesh document should reference verkko-relay concepts without describing the re-encryption internals (deterministic nonces, flash budget moderation).

- **Boundary leak (inward from relay): Section 7.1 convergence sequencing step 3.** References "speculative P1 re-encryption" with airtime caps. This is relay-layer behavior orchestrated by mesh-layer convergence. The orchestration belongs here; the re-encryption details (airtime caps, deterministic nonces) belong in verkko-relay.

- **Boundary leak (inward from protocol): Terminology section.** Redefines DK, DEK, CIK, CPSK, HLC with full descriptions. These are either crypto or protocol concepts. The terminology section duplicates their definitions rather than citing the defined concepts from their home documents.

- **Boundary leak (inward from crypto): VoiceSpec.** The VoiceSpec section describes crate structure including voice_wire (AEAD, nonce sequencing, NonceSourceFactory, double-write persist, PhysicalBoundary) -- these are all crypto-layer concerns placed in a mesh document's implementation plan.

- **Massive document size.** At 1,464 lines, this document is 3-10x larger than the others. It contains implementation-level details (VoiceSpec, build order, calibration items, game day exercises, rejected proposals, compile-time enforcement points) that arguably belong in a separate implementation planning document rather than a design contract document. The taxonomy says documents should have "Deep-nested headers for searchability" but the mesh document has become an implementation specification, not just a design document.

**Dependency hygiene.**

- Declared: verkko-crypto (DK, DEK, ChannelKey, EpochRatchet, BroadcastEncryption, PCSMechanism, DomainSeparatorRegistry). All used. Correct.
- Declared: verkko-protocol (Connection, Heartbeat, Session, Frame, VitalQueue, MetabolicState). All used. Correct.
- **Undeclared dependency on verkko-relay.** Body references TerritorialReencryption, IdempotentPUT, relay membrane, and re-encryption extensively. Convergence Front 3 depends entirely on relay concepts. The Mechanism Reference Table lists relay mechanisms. The Invariants table references verkko-relay concepts.
- **Undeclared dependency on verkko-matter.** Body references cuckoo filters, content integrity, filter-stash consistency, MerkleReconciliation, and SealedRange. The four-reservoir model includes "Content space" with cuckoo filters. The Invariants table includes "Filter-stash consistency".

**Defined concept completeness.**

- **Missing defined concept: PhysicalBoundary.** Section 2.0a defines the PhysicalBoundary trait in detail. It is referenced throughout the document and by other documents, but it is not in the Defined Concepts section. This is a significant omission: multiple documents depend on this concept.

- **Missing defined concept: Memoria.** Section 5.3 names and describes the Memoria (scar institutional memory) as a named composition. It is referenced in emergent mechanisms and terminology. Not a defined concept.

- **Missing defined concept: Conductor.** The terminology section defines Conductor as "the governor's meet projection over the resource product lattice. Sole coherence point for resource-budget parameters." It is referenced in verkko-protocol. Not a defined concept.

- **Missing defined concept: ControlLog.** Section 1.7 describes the control log (signed, hash-linked, append-only GSet). It is referenced in every section that modifies shared state. Not a defined concept.

- **Missing defined concept: ObservationPathForwarding.** Section 5.4 describes this mechanism. verkko-protocol references "ObservationPathForwarding" from verkko-mesh. But it is not a defined concept.

- **Missing defined concept: FencingToken.** Referenced extensively in KeyBundle, dominance, and convergence. The fencing token is a critical ordering mechanism but is not a defined concept.

- **Unused defined concept: none detected.** All defined concepts appear in the body.

**Cross-reference integrity.**

- All "(see verkko-crypto: X)" references check out against verkko-crypto's defined concepts.
- "(see verkko-relay: TerritorialReencryption)" -- correct.
- "(see verkko-relay: IdempotentPUT)" -- correct.
- "(see verkko-matter: SealedRange)" -- correct.
- "(see verkko-matter: MerkleReconciliation)" -- correct.
- "(see verkko-matter: MatterIntegrity)" -- correct.
- "(see verkko-protocol: MetabolicState)" -- correct.
- "(see verkko-protocol: VitalQueue)" -- correct.

**Standalone readability.** Poor for a first-time reader due to sheer volume. Excellent as a reference document once the reader knows the structure. The document assumes deep familiarity with all five other documents despite only declaring two dependencies.

### 1.4 verkko-relay

**Boundary cleanliness.**

- **Clean.** The document stays within its scope. It references verkko-crypto and verkko-protocol concepts through defined concept names. The re-encryption mechanism, membrane model, and padding are all relay-specific.

- **Minor leak (inward from mesh): Section 4.4.** References "see verkko-mesh: Convergence" for speculative P1 re-encryption. The convergence interaction is described from the relay's perspective (what the relay does during convergence, not how convergence works). This is acceptable cross-referencing, not a boundary violation.

**Dependency hygiene.**

- Declared: verkko-crypto (StorageAEAD, ChannelKey, EpochRatchet, DomainSeparatorRegistry, NonceSafety). All used. Correct.
- Declared: verkko-protocol (Frame, Connection, NATTraversal). Frame and Connection used. NATTraversal is referenced only implicitly (transit relay as fallback after hole punching). Marginal but acceptable.
- **Undeclared dependency on verkko-mesh.** Section 4.4 references "see verkko-mesh: Convergence" and the body uses concepts like "cluster", "cluster peers", "CIK pubkeys", "membership change", "revocation" without declaring verkko-mesh as a dependency. The territorial partitioning uses DK pubkeys (a crypto concept) but the ring membership changes with mesh membership changes. The capacity computation references "cluster" as an organizational concept.

**Defined concept completeness.**

- **All defined concepts referenced in body.** Correct.
- **Missing defined concept: RelayStorageTTL.** The 90-day TTL with 30-day TOUCH interval is described in the body and is a significant contract for relay consumers. It is mentioned in the Membrane defined concept invariants but does not have its own concept entry. The invariant in Membrane covers it, so this is a minor issue.

**Cross-reference integrity.**

- "(see verkko-crypto: StorageAEAD)" -- correct.
- "(see verkko-mesh: Convergence)" -- correct.
- All references check out.

**Standalone readability.** Good. The document is concise and self-contained. A reader with only crypto and protocol defined concepts could understand it, except for the cluster and membership concepts from verkko-mesh.

### 1.5 verkko-seat

**Boundary cleanliness.**

- **Clean.** The document is well-bounded. It explicitly states what it does NOT depend on (verkko-mesh) and why. It stays within its scope of seat-to-host-peer communication.

- **Potential gap: authentication mechanism.** The document says "Seat authenticates with a user credential (not a DK/DEK -- seats do not have device keys)" but does not specify what a "user credential" is or how it integrates with the crypto layer. This is not a boundary violation but an underspecification.

**Dependency hygiene.**

- Declared: verkko-crypto (TransportAEAD, KeyZeroization). TransportAEAD is implied for session encryption. KeyZeroization is implied for key cleanup. Neither is explicitly referenced in the body. The body has no Body section content beyond the defined concepts. **Declared but unreferenced due to empty body.**
- Declared: verkko-protocol (Connection, Session, Frame, Stream, SchemaSync, SansIO). None explicitly referenced in the body. Same issue: empty body.
- **No undeclared dependencies.** The explicit "Does NOT depend on verkko-mesh" is useful documentation.

**Defined concept completeness.**

- All defined concepts are self-contained.
- **Missing defined concept: SeatReconnection.** The Seat invariant mentions "If that peer goes offline, the seat is disconnected" but there is no concept for reconnection behavior. SeatSession mentions 0-RTT resumption but the reconnection policy (does the seat try another host peer? does it wait?) is undefined.
- **SeatInstruction types are listed but not formalized.** "download, place, link, remove, verify, inventory sync" is a list of instruction types that could grow. Whether this is a closed set (repr(C) enum) or extensible (trait) is not specified.

**Cross-reference integrity.** No cross-references to verify (the document has no body content with references).

**Standalone readability.** Good. The document is concise and clear about what a seat is and is not. The empty body means it is essentially a skeleton awaiting design.

### 1.6 verkko-matter

**Boundary cleanliness.**

- **Clean.** The document stays within its scope. Content routing, transfer, stash, replication, and sync are all matter-specific concerns.

- **Minor leak (outward to mesh): ScarAttribution.** Section 4.2 introduces `ScarAttribution::IntegrityFailure` as a new variant of the ScarAttribution enum defined in verkko-mesh. This is a legitimate extension point (matter defines an integrity failure that creates a mesh-layer scar), but it means verkko-matter is extending a verkko-mesh type. The dependency direction (matter depends on mesh) is correct, so this is acceptable.

**Dependency hygiene.**

- Declared: verkko-crypto (StorageAEAD, DomainSeparatorRegistry). StorageAEAD not explicitly referenced in body text. DomainSeparatorRegistry referenced via "BLAKE3" for content addressing. BLAKE3 is used directly but DomainSeparatorRegistry is the formal concept. Marginal.
- Declared: verkko-mesh (Cluster, Gateway, Gossip, HealthPipeline, Convergence). All referenced or implied. Gossip used for content filter advertisement. HealthPipeline for source selection. Convergence for sequencing. Correct.
- Declared: verkko-relay (Membrane, TerritorialReencryption, IdempotentPUT). All referenced. Correct.
- **Potentially missing dependency on verkko-protocol.** Section 4.2 references LEDBAT pacing for filter reconciliation ("Paced at 128 KB/s") and the MatterTransfer concept implies use of protocol-layer transport. But the dependency could be considered transitive through verkko-mesh.
- **Potentially missing dependency on verkko-crypto: EpochRatchet, ChannelKey.** The SealedRange mechanism (Section 2.8) references macro-epoch keys, epoch advancement, and HKDF chain. These are verkko-crypto concepts (EpochRatchet, ChannelKey) not listed in the dependency declaration.

**Defined concept completeness.**

- All defined concepts are referenced in the body.
- **Missing defined concept: BlobId.** The body uses `blob_id = BLAKE3(plaintext_content)` as a fundamental identifier. It is the identity of matter. But `blob_id` is not a defined concept; it is described within MatterIntegrity and Matter. Given that blob_id is used across documents (verkko-relay references it for re-encryption nonce derivation), it should arguably be a defined concept in verkko-matter.
- **ContentFilter vs "cuckoo filter".** The defined concept is ContentFilter but the body frequently uses "cuckoo filter" without referencing the defined concept name. Inconsistent terminology.

**Cross-reference integrity.**

- "(see verkko-mesh: KeyBundle)" -- correct.
- "(see verkko-mesh: Scar)" -- verkko-mesh defines Scar. Correct.
- "(see verkko-relay: TerritorialReencryption)" -- correct.
- All references check out.

**Standalone readability.** Good. The document is readable with knowledge of its declared dependencies' defined concepts, with the caveat that EpochRatchet and ChannelKey concepts from verkko-crypto are used without being in the dependency list.

The undeclared dependencies in verkko-mesh are particularly concerning because they create apparent circular dependencies:

- verkko-mesh depends on verkko-relay (uses TerritorialReencryption, IdempotentPUT)
- verkko-relay depends on verkko-mesh (uses cluster, membership -- undeclared)
- verkko-mesh depends on verkko-matter (uses cuckoo filters, SealedRange)
- verkko-matter depends on verkko-mesh (declared)

These are not true circular dependencies in the defined-concept sense (the mesh references relay/matter internals but does not need their defined concepts to define its own concepts). However, the body content creates a bidirectional knowledge coupling that undermines the acyclic dependency graph claimed in the taxonomy.

### 2.3 Dependency graph discrepancy

The taxonomy declares:
```
verkko-crypto
     |
verkko-protocol
   |    |        \
verkko-mesh  verkko-seat  verkko-relay
     |          /
verkko-matter
```

The actual dependency graph based on body content usage:
```
verkko-crypto
     |
verkko-protocol  -----> verkko-mesh (undeclared)
   |    |        \          ^  |
verkko-mesh  verkko-seat  verkko-relay (bidirectional with mesh)
     |    \       /
     |   verkko-matter
     +------^  (mesh references matter concepts)
```

The bidirectional coupling between verkko-mesh and verkko-relay, and between verkko-mesh and verkko-matter, means the "strictly acyclic" claim in the taxonomy is violated in practice, even if not in the Defined Concepts dependency declarations.

### 2.6 Questionable concept ownership

**MetabolicState.** Defined as a verkko-protocol concept but its governor (the Conductor) is described in verkko-mesh's ResourceGovernance. The terminology section in verkko-mesh describes it as "Hierarchical data stream. Access controlled per-channel." This is a foundational concept that lacks formal status.

## 4. Recommended Changes

### 4.1 Critical (contract integrity)

**C1. Add PhysicalBoundary to verkko-mesh Defined Concepts.** This is a shared trait referenced across documents. It has clear invariants (finite, bounded, sign-constrained). Without it as a defined concept, dependents cannot formally rely on its guarantees.

**C2. Add Channel to verkko-mesh Defined Concepts.** Channel is used in verkko-crypto (key identification, epoch ratchet), verkko-mesh (access control, KeyBundle, gossip), and verkko-matter (content routing). It has no formal definition. Define it with invariants: hierarchical path, per-channel access level, independent key lifecycle.

**C3. Add ControlLog to verkko-mesh Defined Concepts.** The control log is the backbone of trust state. It has clear invariants: signed entries, hash-linked, append-only GSet, scoped per channel, Lamport-clocked. Multiple documents depend on it.

**C4. Add FencingToken to verkko-mesh Defined Concepts.** The fencing token's ordering guarantee and grace-window behavior are critical to admin authority. Dependents need to know these invariants without reading the KeyBundle internals.

**C5. Add ObservationPathForwarding to verkko-mesh Defined Concepts.** verkko-protocol references it by name. It must exist as a defined concept for the cross-reference to be valid.

**C6. Declare verkko-mesh dependency on verkko-relay.** The mesh body extensively uses relay concepts (TerritorialReencryption, IdempotentPUT, re-encryption, relay membrane). Either declare the dependency or remove the relay details from the mesh body, replacing them with abstract references.

**C7. Declare verkko-mesh dependency on verkko-matter, or restructure.** The mesh body references cuckoo filters, filter-stash consistency, MerkleReconciliation, and SealedRange. Either declare the dependency or move the filter/content-routing references out.

**C8. Resolve verkko-protocol's undeclared dependency on verkko-mesh.** The protocol's MetabolicState governor is parameterized by mesh-layer ResourceGovernance. ## Summary

The decomposition is structurally sound. The six documents cover distinct domains with mostly clean boundaries. The defined concept pattern (name + invariant as public API) is well-executed and would survive a major version transition.

The primary issues are:

1. **verkko-mesh is doing too much.** It contains implementation planning (VoiceSpec, build order, calibration, game days, rejected proposals) that should be in a separate document. At 1,464 lines it is 3-10x larger than its peers and accumulates undeclared dependencies on both verkko-relay and verkko-matter.

2. **verkko-crypto leaks mesh-layer concerns.** Device recovery, genesis bootstrap, and PCS distribution cross the boundary into enrollment, KeyBundle, and cluster topology.

3. **verkko-protocol has an undeclared upward dependency on verkko-mesh.** The MetabolicState governor is parameterized by mesh-layer ResourceGovernance, creating a layering inversion.

4. **Thirteen concepts are used informally but never formalized.** The most critical are Channel, PhysicalBoundary, ControlLog, and FencingToken -- all are load-bearing across multiple documents.

5. **One broken cross-reference.** verkko-protocol references "ObservationPathForwarding" from verkko-mesh, which is not a defined concept.

The dependency graph is acyclic at the declared level but bidirectionally coupled at the body level between mesh-relay and mesh-matter. This needs resolution either by declaring the dependencies (and accepting the graph complexity) or by extracting the boundary-crossing content.
