## 1. Access-Structural Classification: Accept, Reject, or Modify?

### 1.1 The Savant's Argument

The Savant argues that my blanket classification of B1-B12 as "content gaps,
not structural defects" is "technically correct and cartographically
incomplete." The Savant proposes a new category: **access-structural**. A gap
is access-structural if it does not violate the DAG but blocks all access paths
to a critical capability. Four gaps are classified as access-structural: B1
(SansIO), B2 (intra-cluster), B5 (matter chunking), B12 (connection teardown).

The Savant's analogy: "A country with correct borders and no roads, no cities,
no infrastructure is structurally sound in the border-line sense but
structurally deficient in the habitation sense."

### 1.2 My Assessment

I concede the triage problem. I do not concede the terminology.

**What I concede:** My Stage 2a classification treats all 12 gaps as having
equal structural priority (zero). The Savant is correct that this is unhelpful
for triage. B1 (protocol has no entry point) and B11 (DK encryption at rest,
an open question) do not have the same implementation urgency, and a
classification system that assigns them both "zero structural priority" fails
to distinguish between "the building has no front door" and "the building has
no doorbell."

**What I do not concede:** The term "access-structural" is imprecise. It
smuggles a different concept (implementability) into the structural domain by
attaching the word "structural" to it. The word "structural" in my framework
has a specific, testable meaning: the DAG has a defect (missing edge, inverted
edge, undeclared dependency). Access-structural does not mean the DAG has a
defect. It means the body text within a correctly-placed node is insufficient
for a consumer to use that node.

This is not structural in the DAG sense. It is what I would call
**interface completeness** -- whether a node in the DAG exposes a sufficient
contract for its dependents to consume it. Interface completeness is a real
property. It matters for implementability. But it is not the same thing as
structural soundness, and conflating the two erodes the precision of both
terms.

### 1.3 My Counter-Proposal

I accept a two-dimensional classification, but with different labels:

| Dimension | Definition | Test |
|-----------|-----------|------|
| **DAG-structural** | Edge correctness in the dependency graph | Does filling this gap require adding, removing, or modifying a dependency edge? |
| **Interface-complete** | Contract sufficiency at node boundaries | Can a downstream consumer of this node's defined concepts build against it? |

Under this classification:

| Gap | DAG-structural | Interface-complete |
|-----|---------------|--------------------|
| B1 (SansIO) | Sound | Incomplete -- protocol's consumer contract is undefined |
| B2 (Intra-cluster) | Sound | Incomplete -- mesh's intra-cluster session model is undefined |
| B3 (Channel lifecycle) | Sound | Partial -- channel access control is implicit |
| B4 (Matter GC) | Sound | Complete enough -- deletion is a policy, not a contract |
| B5 (Matter chunking) | Sound | Incomplete -- matter's transfer contract is undefined |
| B6 (Multi-hop relay) | Sound | Partial -- hop_count exists, forwarding rules do not |
| B7 (Relay failover) | Sound | Complete enough -- degraded operation, not contract failure |
| B8 (Seat failover) | Sound | Partial -- fallback mechanism not specified |
| B9 (Seat write conflicts) | Sound | Complete enough -- addressed by SpecWriter |
| B10 (Bridge) | Sound | Out of scope -- application layer |
| B11 (DK at rest) | Sound | Complete enough -- open design question |
| B12 (Connection teardown) | Sound | Incomplete -- connections have no close contract |

The result is the same four gaps the Savant identified (B1, B2, B5, B12), but
the label is "interface-incomplete" rather than "access-structural." The
difference matters: "access-structural" suggests a DAG problem that I should
fix. "Interface-incomplete" correctly identifies a body-text problem that
the SpecWriter should fix. The responsibility and the remediation path are
different.

### 1.4 Verdict on the Classification

**Modified acceptance.** The Savant's two-dimensional approach is correct.
The Savant's labeling is imprecise. I adopt the two dimensions (DAG-structural,
interface-complete) with my own terminology. The practical result is identical:
B1, B2, B5, B12 are distinguished from B3-B4, B6-B11 as higher priority.

## 2. ConvergenceGate Dependency Inversion: Resolved?

### 2.1 My Original Finding

In Stage 2a Section 1.3, I identified that the ConsensusGate predicate
"ring_coloring_valid" references a relay concept (TerritorialReencryption's
hash ring). If mesh evaluates ring coloring validity, mesh requires a
dependency on relay. The declared dependency is relay -> mesh, not
mesh -> relay. This would be a dependency inversion.

I recommended: "ring_coloring_valid is provided by the relay layer as an opaque
boolean signal. Mesh does not inspect the hash ring."

### 2.2 The Savant's Response

The Savant confirms the finding and extends it. In Stage 2b Section 6.2, the
Savant traces the full data flow:

```
relay evaluates ring coloring ->
relay produces boolean signal ->
signal reaches mesh ->
mesh updates ConvergenceGate ->
gate state affects convergence_active bit ->
bit appears in heartbeat
```

The Savant then asks the correct question: **how does the signal travel from
relay (Level 3) to mesh (Level 2)?** Three options:

1. Relay sets a field in its heartbeat extension. Problem: relay does not own
   heartbeat extension fields (mesh does).
2. Relay sends a dedicated control message. Problem: requires a new msg_type.
3. Same-process function call. The runtime reads relay state directly.

The Savant favors option 3 and notes that the "opaque boolean signal" is not
a wire signal; it is a same-process function call. The Savant says the
mechanism should be documented.

### 2.3 My Assessment

The Savant is correct that option 3 is the runtime reality. Let me verify
whether this creates a structural problem.

**Test:** Does same-process function call from mesh reading relay state create
a DAG violation?

The DAG says relay depends on mesh. At the code level: the relay crate imports
mesh types. The mesh crate does NOT import relay types. If mesh reads relay
state via a function call, mesh must have access to relay's interface, which
means mesh would need to import relay types. This would create mesh -> relay,
which is the dependency inversion I flagged.

**But.** If the signal is delivered via an abstract interface (a trait defined
in mesh that relay implements), the dependency direction is preserved. Mesh
defines a trait: `trait ConvergenceSignalProvider { fn ring_coloring_valid(&self) -> bool; }`. Relay implements this trait. The runtime passes relay's implementation to mesh via dependency injection. Mesh depends on its own trait. Relay depends on mesh (to implement the trait). No inversion.

This is exactly the MetabolicGovernor pattern. Protocol defines the abstract
contract. Mesh provides the concrete implementation. The same pattern applies
here: mesh defines the signal contract. Relay provides the signal.

### 2.4 Resolution Status

**Partially resolved.** The concept is correct (opaque boolean via abstract
trait). The mechanism is identified (same-process function call via trait
implementation). What is NOT resolved:

1. The trait is not defined. The SpecWriter's ConvergenceGate text does not
   include a `ConvergenceSignalProvider` trait or equivalent. The predicates
   still read as if mesh evaluates relay state.

2. The wording of the three predicates (ring_coloring_valid,
   all_affected_content_reencrypted, ring_checksum_verified) still implies
   mesh-side evaluation. The SpecWriter has not updated the text.

3. The Savant correctly notes that the signal delivery mechanism should be
   documented. Without this, an implementer may create a direct mesh -> relay
   import, introducing the very inversion I flagged.

**Action required:** The SpecWriter must:
- Add a signal provider trait (or equivalent abstract interface) to verkko-mesh
  that relay implements.
- Reword the three predicates as: "provided by the ConvergenceSignalProvider
  implementation (supplied by verkko-relay at runtime); mesh does not evaluate
  relay state."
- Document that the signal is delivered via same-process trait object, not via
  wire protocol.

**Verdict:** Not resolved. The concept is agreed upon. The text is not updated.
The finding remains open until the predicate wording is fixed and the signal
provider interface is defined.

## 3. Scope Triage: Structural Implications of Including B1, B5, B12

topology.

**B5 (matter chunking):**

My OP-1 in Stage 2a already added the matter -> protocol dependency (for
Stream transport). Matter chunking uses this existing edge. The chunk wire
format lives in verkko-matter. The transfer lifecycle messages
(TransferRequest, TransferBegin, TransferAbort) are matter-internal defined

Including B1 (partial), B5, and B12 in Run 11 requires **zero DAG changes**

From my perspective as the structural reviewer, there is no structural

## 5. The HostPeer Phantom Border: Final Position

### 5.1 The Dispute

My Stage 2a position: The HostPeer is runtime composition, not a document-level
dependency violation. The DAG does not need a seat -> mesh edge. The
translation behavior belongs in verkko-mesh or verkko-ops.

The Savant's Stage 2b response: Moving the HostPeer to the application layer
"moves the phantom, not resolves it." The phantom is resolved at the verkko
level by the SpecWriter's normative note and capability enumeration, but it
moves to the saalis level.

### 5.2 My Assessment

The Savant and I agree on the architecture. We disagree on the framing.

**Agreement:**
1. The HostPeer translation does NOT belong in verkko-seat.
2. Seat does NOT depend on mesh in the DAG.
3. The host peer is a runtime composition point.
4. The SpecWriter's normative note and capability enumeration are the correct
   boundary markers within verkko.
5. The full translation specification is an application-layer concern.

**Disagreement:** The Savant says "moved, not resolved." I say "resolved at
the correct layer."

### 5.3 Why "Resolved at the Correct Layer" Is the Right Framing

A phantom border is a border that exists on the map but is not claimed by
either adjacent territory. The Savant's original finding in Stage 1 was
precisely this: the HostPeer sits between seat and mesh, and neither document
specifies it.

After the cross-reviews:
- verkko-seat specifies the seat-side protocol completely.
- The SpecWriter's normative note states that HostPeer translation is an
  application-layer responsibility.
- The capability enumeration lists what the HostPeer must support.

The phantom is resolved within verkko because verkko now explicitly declares
the boundary and enumerates the capabilities. The fact that the application
layer (saalis) must build the bridge is not a phantom; it is a declared
interface. Declared interfaces are the opposite of phantom borders.

The Savant says "the phantom still exists from saalis's perspective." I
disagree. From saalis's perspective, there is a clearly defined set of
capabilities to implement, documented in the verkko spec. That is a design
task, not a phantom border. A phantom border is unacknowledged. A design task
is acknowledged and enumerated.

### 5.4 Final Position

**The HostPeer phantom border is resolved at the verkko level.** The resolution
is:
1. Normative note in verkko-seat: "HostPeer translation is an application-layer
   responsibility."
2. Capability enumeration in verkko-seat (or verkko-ops): the HostPeer MUST
   support filter queries, channel access checks, matter transfer initiation,
   control log queries, health pipeline queries.
3. The remaining work (building the translation) is a saalis design task with
   a clear contract.

**What I concede to the Savant:** The capability enumeration should be in the
verkko spec, not deferred to the application. The Savant is correct that
without the enumeration, the boundary is declared but not useful. The
enumeration turns a vague "application layer handles it" into a specific
"application layer must implement these five capabilities." This is the
difference between a road sign that says "bridge ahead" and a road sign that
says "bridge closed."

**What I do not concede:** That the phantom "moves" to saalis. There is no
phantom in saalis. There is a design task in saalis. Phantom borders are a
property of incomplete specifications, not of unimplemented code.
