## 1. Response to Dependency Graph Analyst (DepAnalyst)

### 1.1 Back-edges and cycles

DepAnalyst identifies two implicit back-edges: crypto -> mesh (body references) and protocol -> mesh (MetabolicState/Conductor). DepAnalyst correctly calls the crypto -> mesh case "informational forward-references, not definitional dependencies" and the protocol -> mesh case "a tight coupling point."

I agree with both verdicts. From a spec completeness perspective, informational cross-references do not create interoperability risk. Two implementations that never read the forward-reference text in verkko-crypto would still produce compatible crypto primitives. The crypto defined concepts (DK, DEK, EpochRatchet, etc.) are self-standing. DepAnalyst is right that this is "a documentation coupling cycle" rather than a build-order violation.

On the MetabolicState/Conductor coupling, DepAnalyst's recommendation to either move MetabolicState to mesh or define abstract input signals is sound from a spec precision standpoint. My Stage 1 finding P13 offered the same two options. I now lean toward option (b): define abstract input signals in protocol, because MetabolicState's heartbeat interval table and LEDBAT target delay table belong at the transport layer. The state machine is protocol's; the driver is mesh's. The fix is surgical: remove the explicit "see verkko-mesh: Conductor" from MetabolicState's invariants and replace it with "transitions are driven by a governor that emits one of {Active, Idle, Background, Stress} based on resource pressure signals. See the consuming document for the governor specification." This preserves the DAG.

### 1.2 Hidden edge H7: matter -> protocol (LEDBAT)

DepAnalyst flags this as a missing declared dependency (medium severity). I did not raise this in Stage 1, and in retrospect that was an oversight. DepAnalyst is correct: BandwidthPolicy in matter explicitly names LEDBAT and references protocol-layer concepts. The dependency should be declared.

However, I disagree that this is purely a declaration paperwork fix. The dependency is substantive. Matter's transfer protocol needs to specify which VitalQueue tier it uses and how it interacts with LEDBAT pacing. My Stage 1 finding MT1 covers the transfer protocol gap, and DepAnalyst's H7 finding complements it: the declaration must be added AND the transfer-to-transport binding must be specified. The Savant's vacancy V3 makes the same point more forcefully. All three of us converge on the same conclusion from different angles. This is high-confidence.

### 1.3 Hidden edge H8: mesh -> matter (ScarAttribution::IntegrityFailure)

DepAnalyst identifies an upward reference from mesh (level 2) to matter (level 4) through the IntegrityFailure scar variant. This is a genuine ownership conflict. The mesh document should not contain a variant that references blob_id (a matter concept).

DepAnalyst offers two options: move IntegrityFailure to matter, or make the scar type generic. From a spec precision standpoint, option (b) is better. The ScarAttribution enum in mesh should define a generic `ContentIntegrityFailure { source: PeerId, content_hash: [u8; 32] }` where content_hash is an opaque byte array at the mesh layer. The matter layer interprets it as blob_id. This removes the upward dependency without moving the scar mechanism out of mesh (where it belongs, since health scoring consumes scars).

### 1.4 Interface sufficiency assessment

DepAnalyst rates most declared dependency edges as SUFFICIENT. I confirm this assessment for the edges I independently examined. The protocol -> crypto interface is clean. The mesh -> crypto interface has the two internal leaks DepAnalyst identifies (BLAKE3 in dominance, Ed25519 in enrollment). These are real leaks but they reference properties that are definitionally stable (changing from Ed25519 or BLAKE3 would be a breaking change to crypto itself). I classify these leaks as LOW risk, not medium.

I disagree with DepAnalyst's assessment of the heartbeat extension split (H6, rated "Low" severity). DepAnalyst calls it "a well-designed split" and "clean interface segregation." The Savant calls it grain boundary G1 and the "most heavily trafficked bond in the lattice." I side with the Savant here. The split is well-designed in concept but incomplete in specification. The 18-byte extension region has no version field, no reserved space, no extension mechanism. My Stage 1 finding M20 covers this. DepAnalyst's "clean" rating underestimates the maintenance risk because the extension region is the only data path for six different consumers (failure detection, health scoring, epoch sync, filter divergence, gossip mode, metabolic observation). A "clean" interface that is also brittle under change is a time bomb. The Savant's proposal S1 (HeartbeatExtension defined concept) is the right fix.

## 2. Response to Gap Hunter (GapHunter)

### 2.1 Severity scale alignment

GapHunter uses BLOCKING/AMBIGUOUS/MISSING-ERROR/UNTESTABLE. My Stage 1 uses MISSING/INCOMPLETE/AMBIGUOUS/UNJUSTIFIED with severity levels CRITICAL/HIGH/MEDIUM/LOW. The tension map (Tension 2) asks which scale is correct. My answer:

Both scales are correct for their purpose, but they answer different questions. GapHunter's BLOCKING means "I cannot write a function without this." My CRITICAL means "two implementations will produce incompatible wire traffic." These are different failure modes. A BLOCKING gap is a development velocity problem. A CRITICAL gap is an interoperability problem.

For the purposes of Stage 2, I recommend a merged triage:
- **Interop-critical** (cannot ship): findings where two implementations will produce incompatible behavior on the wire. My P1, P2, S1, X1. Also GapHunter's X-7 (version negotiation).
- **Implementation-blocking** (cannot code): findings where a developer cannot write correct code. GapHunter's P-3, P-5, P-8, P-9, R-1 through R-5, S-1 through S-10.
- **Correctness-threatening** (code will exist but may be wrong): Savant's C3 (unregistered domain separator), my MT3 (SyncReconciliation), GapHunter's T-3 (merge rules).
- **Maintainability concerns** (will cause pain later): Savant's G1, G2, G4; DepAnalyst's H8.

The Savant's severity scale (catastrophic/high/medium for emergent risk) captures something neither GapHunter nor I captured in Stage 1: delayed-onset failures that are invisible until production stress. The domain separator registry gap (Savant's C3/G4) is the clearest example. I rated it LOW in Stage 1 (C15). The Savant rates it "catastrophic, silent." The Savant is right and I was wrong. I am revising my assessment. See section 4.

### 2.2 Convergence on specific findings

GapHunter and I identified 47 of the same gaps (using different finding IDs and wording). I spot-checked 15 of the overlapping findings and found no substantive disagreements in the diagnosis. The fixes we propose are compatible but mine tend to be more prescriptive (specifying exact byte layouts and field orderings) while GapHunter's are more minimal ("enough to unblock coding").

For a protocol specification, my approach is correct. The purpose of a spec is to produce interoperable implementations. A "minimum fix" that unblocks one developer still leaves two developers free to diverge. Every wire format fix must specify exact byte layouts, endianness, and field ordering. GapHunter's minimum fixes are necessary but not sufficient.

### 2.3 Stream multiplexing as optional

GapHunter's Section 11 assessment says stream multiplexing is "No" for v1 blocking ("Single-stream works for v1"). I disagree with this classification. If the wire format is defined without a stream_id field in v1, then adding stream multiplexing in v2 requires either: (a) breaking the wire format, (b) using a frame type to signal stream support, or (c) reserving a stream_id field in v1 that is always zero.

Option (c) costs 1 byte per frame and prevents a future breaking change. The spec should include stream_id in v1 even if single-stream is the only mode. This is a standard protocol design practice (reserve extensibility fields early). My Stage 1 finding P3 rates this HIGH, not because multiplexing is needed now, but because omitting it from the wire format creates a version negotiation problem later. GapHunter's pragmatism here trades future cost for present simplicity. In a household mesh that must handle rolling upgrades without coordination, this trade is wrong.

### 2.4 Where GapHunter found things I missed

GapHunter's C-9 (Session_ID `generation` field undefined) is a finding I missed entirely. The `generation` parameter appears in a BLAKE3 derivation and is never defined. This is BLOCKING for any implementation that computes Session_IDs. I am adding this to my updated assessment as a HIGH finding.

GapHunter's X-7 (version negotiation across all documents) is a cross-cutting concern I should have flagged. I noted the absence of version fields in individual findings (C5, M20) but did not elevate it to a cross-document finding. GapHunter correctly identifies that version negotiation must be in the wire format from v1 or the first upgrade requires a flag-day deployment.

## 3. Response to Structural Integrity Savant

### 3.1 What I learned from the Savant

The Savant's crystallographic lens produces three insights I want to formalize as spec methodology:

**Insight 1: Grain boundaries as specification interfaces.** The Savant's grain boundary concept (G1-G4) is a useful formalization of what I call "split ownership." When two documents share responsibility for a concept (heartbeat extension: protocol owns format, mesh owns semantics), the interface between them is a grain boundary. The Savant's key observation is that grain boundaries require their own specification (a defined concept that both documents reference). This is exactly how IETF handles cross-layer interfaces: define an abstract interface that both layers reference. The HeartbeatExtension defined concept (Savant S1) and ConvergenceGate defined concept (Savant S6) follow this pattern.

I am adopting the term "grain boundary" in my methodology. Every grain boundary in the spec should be formalized as a defined concept in the document that owns the lower-layer side of the interface.

**Insight 2: Cleavage plane analysis as a spec stress test.** The Savant's cleavage planes (C1-C4) formalize the question "what happens when we change X?" as a structural analysis. This is similar to protocol extensibility analysis in IETF work but the Savant's formulation is more systematic. Each cleavage plane identifies: the stress, the fracture path (which documents break), and the severity. This is a useful tool for identifying which parts of the spec need extensibility hooks.

I am adopting cleavage plane analysis as a checklist for future spec reviews: for each major mechanism, trace the change propagation path and count the documents affected.

**Insight 3: Vacancies as structural weakness.** The Savant distinguishes between "missing body" (a finding I make) and "missing bond" (a structural gap where two documents should interact but do not). Vacancy V1 (seat has no bond to mesh) is different from my S1 (seat has no body). My finding says "you cannot implement this." The Savant's finding says "even if you implement this, the seat is structurally isolated from the mesh and will behave unpredictably under mesh stress." The Savant is identifying a design gap, not a specification gap. Both need to be fixed, but they are different categories.

I am now distinguishing between specification gaps (missing text that would enable implementation) and structural gaps (missing relationships between documents that would enable correct behavior). My Stage 1 focused on the former. The Savant identifies the latter.

### 3.2 Where the Savant is wrong

**The Savant overstates vacancy V2 (matter has no bond to seat).** The Savant says "The seat cannot directly address matter" and treats this as a structural weakness. But this is intentional design. Seat is a thin client; it does not need to address matter directly. The HostPeer mediates. The flow is seat -> host peer -> mesh -> matter. This is not a vacancy; it is an indirection by design. The seat's isolation from mesh and matter is a feature (security boundary), not a defect.

The specification gap is in the HostPeer: the HostPeer's mediation logic is unspecified (my S3, the Savant's V1). But the absence of a direct seat-to-matter bond is correct architecture. The Savant treats all missing bonds as weaknesses; I treat some missing bonds as intentional isolation.

**The Savant underestimates the domain separator registry risk timeline.** The Savant calls the DomainSeparatorRegistry enforcement gap (G4/C3) the "most dangerous defect" and rates it "catastrophic, silent." The diagnosis is correct but the risk timeline needs qualification. This risk materializes only when a new BLAKE3 use is added without registration. In v1 (the current specification), all BLAKE3 uses are registered. The catastrophe requires a specification change that violates the registry invariant. This makes it a process failure, not a specification failure. The Savant's proposed fix (S2, a domain separator registration lint) is excellent and should be implemented, but the current specification is not broken; the defense against future breakage is absent. I originally rated this LOW (C15) because the current spec is complete on this point. The Savant's "catastrophic" rating is correct for the failure mode but overstated for the current risk level. I am revising my rating to MEDIUM: the failure mode is catastrophic but the probability in v1 is zero. The fix (S2) should be implemented before v2 development begins.

**The Savant's "empty cell" framing obscures actionable findings.** The Savant says verkko-seat is "~15% filled" and verkko-matter is "~40% filled." These percentages are imprecise and not actionable. My Stage 1 lists 14 specific findings for seat and 13 for matter, each with a concrete fix. GapHunter lists 10 seat gaps and 8 matter gaps, each with a minimum fix. The Savant's "fill percentage" framing is useful for executive summary but harmful for engineering triage because it suggests the problem is uniformly distributed across the document. It is not. The seat document has adequate defined concepts (the "what" is ~60% specified) but zero body (the "how" is 0% specified). "15% filled" conflates these two dimensions.

### 4.2 New findings from cross-review

**CR1. MISSING / HIGH -- Session_ID `generation` field undefined.**
(From GapHunter C-9.) verkko-crypto Section 2.4 uses `generation` in the Session_ID derivation without defining it. This is BLOCKING for Session_ID computation.

*Fix:* "generation is a u64 counter, incremented on each session renegotiation for the same channel_path. Initial value: 0. Carried in the session establishment handshake."

**CR2. MISSING / HIGH -- Version negotiation absent from all documents.**
(From GapHunter X-7.) No document specifies protocol version negotiation. The wire format has no version field. Rolling upgrades will require a flag-day deployment.

*Fix:* "Add version to the Noise prologue: `'saalis-verkko-v' || u8_version`. Add version_hint byte to the outer frame header. Incompatible version: handshake fails. Compatible: negotiate to the lower version."

**CR3. INCOMPLETE / MEDIUM -- ScarAttribution::IntegrityFailure creates upward dependency.**
(From DepAnalyst H8.) The mesh-layer ScarAttribution enum references blob_id, a matter-layer concept. This is an ownership violation.

*Fix:* "Replace IntegrityFailure { source: PeerId, blob_id: [u8; 32] } with ContentIntegrityFailure { source: PeerId, content_hash: [u8; 32] }. content_hash is opaque at the mesh layer. The matter layer interprets it as blob_id."

### 4.3 Tension resolutions

**Tension 1 (coupling cycles vs. deliberate interface split).** The back-edges are acceptable forward-references that document consumers should tolerate, with one surgical fix: remove the explicit "see verkko-mesh: Conductor" from MetabolicState's defined concept invariants. Replace with abstract input signal language. The heartbeat extension split needs a HeartbeatExtension defined concept (Savant S1). It is a design strength that currently lacks spec precision.

**Tension 2 (severity triage).** No single scale is correct. The merged triage in section 2.1 above is the right approach. Interop-critical findings are the hardest gate. Implementation-blocking findings are the broadest gate. Correctness-threatening findings are the most dangerous gate (they pass the first two but fail in production).

**Tension 3 (MetabolicState ownership).** MetabolicState stays in verkko-protocol. Transition predicates are explicitly delegated to the consuming document's governor via abstract input signals. The protocol layer defines states, dwell times, and parameter tables. The mesh layer defines the governor that selects states. This is a layering choice, not a defect, but the current spec text must be clarified to remove the direct naming of Conductor.

**Tension 4 (matter's undeclared protocol dependency).** Both declaration fix AND structural specification needed. verkko-matter must: (a) add verkko-protocol to its Dependencies section, (b) specify that MatterTransfer uses bulk-tier streams (VitalQueue Queue 3), (c) specify LEDBAT pacing for transfer bandwidth. This does not change the topological sort level; matter is already at level 4 (after relay at level 3, which already depends on protocol at level 1).

**Tension 5 (empty cells).** Seat and matter are NOT out-of-scope. Their gaps affect cross-document interfaces that the other reviewers need to evaluate. However, the gap resolution should be staged: first, define the cross-document interfaces (HostPeer mediation, MatterTransfer-to-Stream binding), then fill the internal body content. DepAnalyst's "interface sufficiency: SUFFICIENT" for seat is technically correct (seat uses only defined concepts from crypto and protocol) but misleading, because seat's own defined concepts are insufficient for its consumers. The interfaces seat consumes are fine; the interfaces seat exports are empty.

### 4.4 Disagreement statement

I disagree with DepAnalyst's claim that the heartbeat extension split (H6) has "Low" severity and is "clean interface segregation." The split is conceptually clean but specificationally incomplete. The 18-byte extension region has no reserved space, no version field, and no extension mechanism. Six consumers depend on it. Any change to the extension field layout requires coordinated edits in two documents. This is the definition of a brittle interface. The severity is MEDIUM at minimum, and the Savant's HeartbeatExtension defined concept is the correct fix.

## 5. Scope boundary

The following topics were raised by other reviewers but are outside my domain (specification completeness). I offer no comment:

- Whether the crystal metaphor accurately represents document relationships (Savant's methodology)
- Whether the DAG structure is optimal vs. alternative decompositions (DepAnalyst's recommendation to consider merging)
- Whether implementation velocity concerns justify deferring spec completeness (GapHunter's v1 vs v2 pragmatism for stream multiplexing)
- The ops document's role as a "coupling amplifier" (DepAnalyst section 7.1)

These are architectural and process questions, not specification completeness questions.
