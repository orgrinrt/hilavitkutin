#### R11-006: Add DomainSeparator defined concept

- **Source:** SpecWriter P0-7, Stage 1
- **Location:** New defined concept section, before Body
- **Action:** INSERT

```
### DomainSeparator

repr(C) enum enumerating all registered domain-separated hash and
derivation usages. This enum is the sole API for domain-separated
hashing in the verkko protocol. Raw BLAKE3 MUST NOT be called
directly from protocol crates; all keyed BLAKE3 usage MUST go
through a function that takes a DomainSeparator variant.

    #[repr(C)]
    enum DomainSeparator {
        KeyId         = 0,  // "saalis-keyid-v1"
        ReencNonce    = 1,  // "saalis-reenc-v1"
        FoundingHash  = 2,  // (raw BLAKE3, structurally unique)
        PhaseJitter   = 3,  // "saalis-jitter-v1"
        HbJitter      = 4,  // "saalis-hbjitter-v1"
        SessionId     = 5,  // "saalis-sessid-v1"
        DestinationId = 6,  // "saalis-destid-v1"
        ResumeToken   = 7,  // "saalis-resume-v1"
        DhtKey        = 8,  // "saalis-dht-v1"
    }

Invariants:
- No two variants share the same domain separator string.
- The domain separator string for each variant is fixed at
  compile time.
- Adding a new domain-separated hash usage REQUIRES adding a
  new variant to this enum. The enum is the registry.
- Protocol crates MUST NOT import blake3::hash or
  blake3::Hasher directly. They MUST use the verkko-crypto
  domain-separated API.
- HMAC-BLAKE3 (used for relay_key_id) is a different construction
  from keyed BLAKE3 and is explicitly excluded from this registry.

Padding rule:
    key = domain_separator_bytes || 0x00 * (32 - len(domain_separator_bytes))
    All domain separator strings are <= 32 bytes.

API surface:
    fn domain_hash(domain: DomainSeparator, input: &[u8]) -> [u8; 32];
    fn domain_hash_truncated(domain: DomainSeparator, input: &[u8]) -> &[u8];
```

- **Rationale:** Centralizes all BLAKE3 domain separation into a compile-time registry, preventing domain collision.

#### R11-008: Add MetabolicGovernor abstract contract

- **Source:** SpecWriter P0-2, Stage 1; ArchSurgeon structural endorsement
- **Location:** MetabolicState defined concept section
- **Action:** INSERT after MetabolicState
- **Text:**

```
### MetabolicGovernor (abstract contract)

An abstract contract that drives MetabolicState transitions. The protocol
layer defines the state machine and this contract; mesh or other layers
provide concrete implementations.

Invariants:
- MetabolicGovernor is a pure function from resource pressure readings
  to MetabolicState.
- The governor MUST produce exactly one of {Background, Idle, Active,
  Stress} for any valid input.
- The governor MUST NOT require access to protocol-internal state
  beyond what it receives as input.
- The governor MUST satisfy the dwell time invariant: no transition
  SHALL be emitted within dwell_time of the previous transition,
  except when convergence_override is true.
- ResourcePressures is opaque to the protocol layer. Protocol passes
  the pressure readings through to the governor without inspecting
  their contents.

    #[repr(u8)]
    enum MetabolicState {
        Background = 0b00,
        Idle       = 0b01,
        Active     = 0b10,
        Stress     = 0b11,
    }

State machine transitions:

    Background <-> Idle <-> Active <-> Stress

All transitions are bidirectional. Adjacent-only transitions are NOT
required; the governor MAY emit any state from any state (subject to
dwell time). This permits the convergence override to jump directly
from Background to Active or Stress.

Error handling:

| Condition | Response | State after |
|-----------|----------|-------------|
| Governor returns invalid state | Log error, retain current state | Unchanged |
| Any pressure value is NaN | Log error, retain current state | Unchanged |
| Governor panics or is absent | Default to Active | Active |
```

- **Rationale:** Breaks the protocol -> mesh dependency cycle by making the governor an abstract contract that protocol defines and mesh implements.

#### R11-009: Add SessionResumption defined concept

- **Source:** ArchSurgeon OP-2, Stage 1; WireEng Stage 3a wire format
- **Location:** After ConnectionMigration defined concept, before Body
- **Action:** INSERT
- **Text:**

```
### SessionResumption

0-RTT session resumption using a stored resumption token.
Avoids full Noise handshake on reconnection.

Invariants:
- Resumption token is encrypted under the session's export
  key (first 32 bytes of GetHandshakeHash()) and stored locally.
- 0-RTT data is replay-vulnerable: only idempotent operations
  permitted in the 0-RTT window (heartbeat, gossip, filter queries).
- The responder validates the resumption token's freshness
  (expiry bound, monotonic ticket counter).
- Fallback to full handshake if resumption fails (token
  expired, key rotated, peer restart).
- Resumption does not weaken forward secrecy for post-0-RTT data:
  the resumed session derives new transport keys via Noise KK.
  0-RTT data has weaker forward secrecy (encrypted under the
  previous session's export key, not fresh ephemeral DH).

Wire format:
- Plaintext token: 80 bytes (session_id [32], session_key_hash [32],
  expiry [u64], ticket_counter [u32], reserved [4]).
- Encrypted token: 120 bytes (nonce [24] + ciphertext [80] + tag [16]).
- Resumption handshake uses hs_type = 0x03 (RESUME).
- Token nonce derived via DomainSeparator::ResumeToken.
```

- **Rationale:** Promotes the remaining informal concept (0-RTT resumption) to a defined concept with wire format.

#### R11-011: Add version negotiation (pre-handshake)

- **Source:** WireEng Section 1, Stage 1; Tension T1 resolution
- **Location:** New section before Section 3.1 Wire format
- **Action:** INSERT
- **Text:**

```
### Version Negotiation

Two peers MUST agree on a protocol version before any authenticated
communication occurs. Version negotiation is the first exchange on
any new UDP association, before the Noise handshake.

VersionProbe (18-28 bytes, unauthenticated):
- Magic: 0x56524B4F ("VRKO"), 4 bytes.
- probe_type: 0x01, 1 byte.
- flags: reserved (all zero in v1), 1 byte.
- version_count: u16, range [1, 6].
- versions: u16 * version_count, descending preference.
- probe_nonce: u64, CSPRNG-generated.

VersionResponse (20 bytes):
- Magic: 0x56524B4F, 4 bytes.
- probe_type: 0x02, 1 byte.
- flags: reserved, 1 byte.
- version_count: 0x0001, 2 bytes.
- selected_version: u16 (highest common, or 0x0000 for none).
- reserved: u16, zero.
- echo_nonce: u64 (echoed probe_nonce).

Negotiation algorithm:
- Initiator sends probe with supported versions.
- Responder selects max(intersection(probe.versions, local.versions)).
- Empty intersection: respond with selected_version = 0x0000.
- Timeout: 5 seconds, 3 retries, then abort with VERSION_TIMEOUT.

Security:
- The responder SHOULD rate-limit VersionResponse to at most 100
  responses per second per source address.
- Selected version is bound into the Noise prologue:
  prologue = 0x56524B4F || u16(version) || u8(hs_type) = 7 bytes.
- If a Noise handshake fails immediately after a successful version
  probe, the initiator SHOULD retry with direct version specification
  (highest supported version, no probe) to detect downgrade attacks.
```

- **Rationale:** Pre-handshake probe avoids wasted DH computation on version mismatch. Prologue binding provides cryptographic downgrade protection.

#### R11-012: Add connection teardown (B12)

- **Source:** WireEng Stage 3a; SpecWriter Stage 3a acceptance
- **Location:** New subsection after Connection Migration
- **Action:** INSERT
- **Text:**

```
### ConnectionClose

Explicit connection termination. msg_type = 0x10.

Wire format (inside encrypted payload):
    Offset  Width  Field
    ------  -----  -----
     0      1      reason: u8
     1      1      reserved: u8 (zero)
     2      2      detail_len: u16
     4      var    detail: [u8; detail_len] (UTF-8, max 256 bytes)

Reason codes:
    0x00 = NORMAL           -- clean shutdown
    0x01 = IDLE_TIMEOUT     -- no traffic for timeout period
    0x02 = KEY_EXPIRED      -- session keys expired
    0x03 = MIGRATION_FAIL   -- PATH_RESPONSE not received
    0x04 = INTERNAL_ERROR   -- local error
    0x05 = VERSION_MISMATCH -- discovered after SchemaSync
    0x06 = HEARTBEAT_TIMEOUT -- 3 missed heartbeats
    0x07 = PROTOCOL_ERROR   -- protocol violation
    0xFF = UNSPECIFIED       -- generic close

Close handshake:
1. Initiator sends CONN_CLOSE frame with reason.
2. Responder drains pending data, sends its own CONN_CLOSE.
3. Both sides enter CLOSING state: accept no new stream data,
   process remaining SACKs and CONN_CLOSE frames, discard
   other frame types.
4. After min(2*RTT, 5000ms): destroy session state and
   zeroize keys.

Abrupt close: on fatal protocol violation, a peer MAY send
CONN_CLOSE with reason PROTOCOL_ERROR and immediately destroy
session state without waiting for a response.

Heartbeat timeout close: when heartbeat timeout fires (3
missed heartbeats), the peer enters CLOSING state as if it
had received CONN_CLOSE with reason HEARTBEAT_TIMEOUT.

Interaction with in-progress transfers:
- CONN_CLOSE implicitly aborts all in-progress transfers.
- Peers SHOULD NOT send TransferAbort for transfers on a
  connection that has sent or received CONN_CLOSE.
- Implicitly aborted transfers do not generate scars.
```

- **Rationale:** Connections had no close mechanism; could only terminate by heartbeat timeout. This wastes resources and delays cleanup.

#### R11-013: Connection ID generation (random_u32)

- **Source:** SpecWriter P1-3, Tension T3 resolution
- **Location:** Connection defined concept section
- **Action:** INSERT
- **Text:**

```
#### Connection ID generation

    connection_id = random_u32()

Generated by each endpoint via CSPRNG at session initiation.
Each direction of a bidirectional session has its own connection_id.

Reserved values:
    0x00000000  Handshake frame (no session established yet).
    0xFFFFFFFF  Reserved for future use.

Lifecycle:
- Generated at session initiation (Noise message 1 sent).
- Valid for the lifetime of the session.
- Invalidated on session termination or renegotiation.
- Preserved across connection migration.

Error handling:
- connection_id = 0x00000000 or 0xFFFFFFFF: regenerate.
- Collision with existing session: regenerate.
```

- **Rationale:** Simpler than BLAKE3 derivation, no layer coupling to mesh, no persistent counter, identical collision probability.

```
#### Stream multiplexing

Stream header (7 bytes, inside encrypted payload):

    Offset  Width  Field
    ------  -----  -----
     0      2      stream_id: u16
     2      4      sequence: u32 (per-stream monotonic)
     6      1      flags: u8
                    Bit 0: FIN (stream termination)
                    Bit 1: RST (stream reset)
                    Bit 2: SACK_PRESENT
                    Bits 3-7: reserved (MUST be 0)

Stream ID allocation (three-tier):
    0x0000           Control stream (VitalQueue Tier 1).
    0x0001-0x00FF    Vital tier streams (control log, keys).
    0x0100-0x0FFF    Reactive tier streams (gossip, filters).
    0x1000-0xFFFE    Bulk tier streams (matter, relay ops).
                     LEDBAT-paced.
    0xFFFF           Reserved.

Stream lifecycle:
- Implicit open on first frame with a new stream_id.
- FIN flag closes the stream gracefully.
- RST flag aborts the stream immediately.

HEARTBEAT and SACK-only frames do not carry stream headers
(they are connection-level, not stream-level).
```

- **Rationale:** Three-tier stream ID allocation encodes the queue tier implicitly, enabling O(1) priority scheduling without inspecting msg_type.

#### R11-017: Add HeartbeatExtension defined concept

- **Source:** ArchSurgeon OP-4, SpecWriter P0-3; merged specification
- **Location:** Defined Concepts section, after ObservationPathForwarding
- **Action:** INSERT
- **Text:**

```
### HeartbeatExtension

The 18-byte extension region of the heartbeat frame. Carries
mesh-layer state observable by remote peers. The protocol layer
(verkko-protocol: Heartbeat) defines the frame format and the
18-byte transport header. The mesh layer defines the extension
region contents.

Invariants:
- The extension region is exactly 18 bytes.
- Field layout is fixed (big-endian):
  - filter_root (8 bytes): keyed truncated Merkle root.
  - epoch_current (4 bytes): G-Counter macro-epoch.
  - work_capacity (1 byte): min(bw_kbps/128, flash_kb/256, 255).
  - health_canary (1 byte): bits 7-4 = worst health class
    (0=Healthy, 1=Degraded, 2=Bad); bits 3-0 = degraded count
    (0-15, clamped).
  - observation_summary (4 bytes): aggregated FlowSummary.
- Protocol layer treats extension bytes as opaque payload.
  Mesh layer interprets them.
- No version field in the extension region. Changes require
  a protocol version increment.

State byte (offset 0 in transport header, protocol-owned):
    Bits 0-1: MetabolicState (protocol-layer)
    Bit 2:    gossip_mode (0=Plumtree, 1=flood)
    Bit 3:    convergence_active (0=normal, 1=converging)
    Bit 4:    admin_flag (0=non-admin, 1=admin)
    Bits 5-7: reserved (MUST be zero)

gossip_mode, convergence_active, and admin_flag are boundary
objects: protocol owns the bit position; mesh owns the
semantic meaning.

Ownership table:

| Field | Owner | Region | Bytes |
|-------|-------|--------|-------|
| state | verkko-protocol | Transport header | 1 |
| rtt_us | verkko-protocol | Transport header | 2 |
| loss_permille | verkko-protocol | Transport header | 2 |
| jitter_us | verkko-protocol | Transport header | 2 |
| queue_depth | verkko-protocol | Transport header | 1 |
| available_bw_kbps | verkko-protocol | Transport header | 2 |
| hlc_timestamp | verkko-protocol | Transport header | 8 |
| filter_root | verkko-mesh | Extension region | 8 |
| epoch_current | verkko-mesh | Extension region | 4 |
| work_capacity | verkko-mesh | Extension region | 1 |
| health_canary | verkko-mesh | Extension region | 1 |
| observation_summary | verkko-mesh | Extension region | 4 |

Error handling:
| Condition | Response |
|-----------|----------|
| Extension region < 18 bytes | Drop heartbeat. Log diagnostic. |
| epoch_current < previous from same sender | Drop heartbeat. Log warning. |
| health_canary worst_class > 2 | Treat as 2 (Bad). Log warning. |
```

- **Rationale:** Formalizes the ownership split between protocol (transport) and mesh (extension) for the most critical frame in the system.

Boolean predicate gating the transition between convergence fronts.
Consumers observe the gate's boolean output (open or closed) but
MUST NOT inspect the inputs or mechanisms behind the gate evaluation
(opaque projection principle).

Invariants:
- A gate is either OPEN (true) or CLOSED (false). No partial states.
- A gate, once OPEN, MUST NOT close within the same convergence event.
- Gates are evaluated locally by each peer.
- Gates have timeout fallbacks.

Gate definitions:

    Gate 1: ControlGate (ControlFront -> ConsensusFront)
        Predicate: all_control_entries_processed
            AND all_reachable_keys_obtained
            AND all_ema_resets_applied
        Timeout: 5 minutes.
        On timeout: orphan unretrievable channels. Gate opens.

    Gate 2: ConsensusGate (ConsensusFront -> ReencryptionFront)
        Predicate: highest_fencing_token_confirmed
            AND dominance_resolved
            AND ring_coloring_valid (provided by ConvergenceSignalProvider)
        Timeout: 10 minutes.
        On timeout: highest fencing token seen wins. Gate opens.

    Gate 3: ReencryptionGate (ReencryptionFront -> exit)
        Predicate: all_affected_content_reencrypted_or_epoch_forwarded
            (provided by ConvergenceSignalProvider)
            AND vital_queue_depth < 25%
        Timeout: none (bounded by tiered deadlines).

    Gate 4: ExitGate (exit sequence completion)
        Predicate: gossip_mode_ack_complete
            AND spanning_tree_connected
            AND health_baselines_updated
            AND ring_checksum_verified (provided by ConvergenceSignalProvider)
        Timeout: 3 * wavefront_deadline.

Interface (opaque projection):
    fn is_open(&self) -> bool;

Relay-dependent predicates (ring_coloring_valid,
all_affected_content_reencrypted, ring_checksum_verified)
are provided by the ConvergenceSignalProvider implementation
(supplied by verkko-relay at runtime). Mesh does not evaluate
relay state; it queries opaque boolean signals via same-process
trait implementation.

### ConvergenceSignalProvider

Abstract trait defined in verkko-mesh. Relay implements it.
The dependency direction is preserved: relay depends on mesh
(to know the trait); mesh depends on its own trait (to call it).

#### R11-029: Add matter chunk protocol (B5)

- **Source:** WireEng Stage 3a Part 4; Savant N3
- **Location:** New section in Body
- **Action:** INSERT
- **Text:**

```
### MatterChunkProtocol

Bridge between multi-gigabyte blobs and 1232-byte UDP payloads.
Uses bao verified streaming with 1024-byte chunks (bao standard).

Chunk frame (msg_type = 0x04, 52-byte header):

    Offset  Width  Field
    ------  -----  -----
     0      8      transfer_id: u64 (identifies transfer session)
     8     32      blob_id: [u8; 32] (BLAKE3 of complete plaintext)
    40      4      chunk_index: u32 (0-based bao tree position)
    44      4      total_chunks: u32
    48      2      chunk_data_len: u16
    50      2      proof_len: u16
    52      var    chunk_data
     ?      var    bao_proof

Chunk header appears once per chunk, not once per frame. Chunks
that span multiple frames use stream-layer reassembly.

Frame budget:
    Available per first frame: 1118 bytes (chunk_data + bao_proof)
    Available per continuation frame: 1170 bytes
    Worst case (2GB blob): 2 frames per chunk

TransferRequest (msg_type = 0x0D, 44-byte header):
    [8]  transfer_id (CSPRNG-generated by requester)
    [32] blob_id
    [4]  have_bitmap_len (u32, number of u64 words)
    [var] have_bitmap (bitfield of already-held chunks)

TransferBegin (msg_type = 0x0E, 88-byte header):
    [8]  transfer_id (echoed)
    [32] blob_id (echoed)
    [4]  total_chunks
    [8]  blob_size (u64, total bytes)
    [32] bao_root (root hash for verification)
    [2]  chunk_size (u16, typically 1024)
    [2]  reserved

TransferAbort (msg_type = 0x0F, 12-byte header):
    [8]  transfer_id
    [1]  reason (0x00=COMPLETED, 0x01=INTEGRITY_FAILURE,
         0x02=SOURCE_GONE, 0x03=CANCELLED, 0x04=STORAGE_FULL,
         0x05=RATE_LIMITED, 0x06=CONN_CLOSING, 0xFF=UNSPECIFIED)
    [1]  reserved
    [2]  detail_len

Transfer lifecycle:
1. Requester sends TransferRequest with blob_id and have_bitmap.
2. Source replies with TransferBegin (total_chunks, bao_root).
3. Source streams chunks on bulk-tier stream (LEDBAT-paced).
4. Requester verifies each chunk against bao_proof.
5. On integrity failure: abort, scar the source.
6. On completion: verify blob_id = BLAKE3(reassembled plaintext).

Control messages use reactive tier (0x0100-0x0FFF). Chunk data
uses bulk tier (0x1000-0xFFFE).

Multi-source: requester opens separate transfer sessions with
different sources. Each has a unique transfer_id. The have_bitmap
communicates partial state for resumption.
```

- **Rationale:** Without chunking, the system cannot transfer the content it exists to serve. This is the primary payload of the protocol.

## Part 4: Rejected Items

#### REJ-001: Per-message handshake magic values

- **What:** Different magic bytes for each Noise message (VKKK, VKKR, VXX1, VXX2, VXX3).
- **Proposed by:** SpecWriter, Stage 1
- **Why rejected:** Inferior parsing (5 magic comparisons vs 1 magic + type byte), worse extensibility (new patterns need new magic families), version byte redundant with pre-handshake negotiation. SpecWriter conceded in Stage 2a.

#### REJ-002: BLAKE3-derived connection ID

- **What:** `connection_id = BLAKE3("saalis-connid-v1" || CIK_pub || CIK_pub || nonce || counter)[0..4]`
- **Proposed by:** WireEng, Stage 1
- **Why rejected:** Creates protocol -> mesh naming dependency (CIK_pub), requires persistent counter, adds DomainSeparator entry for non-security-critical use, no collision benefit over random_u32(). WireEng conceded in Stage 2a.

#### REJ-003: BLAKE2s for Noise protocol name

- **What:** `Noise_KK_25519_ChaChaPoly_BLAKE2s` instead of BLAKE2b.
- **Proposed by:** WireEng, Stage 1
- **Why rejected:** verkko targets 64-bit hardware where BLAKE2b is 20-40% faster. The WireGuard precedent (BLAKE2s for 32-bit MIPS routers) does not transfer. BLAKE2b is the Noise spec's default recommendation. WireEng conceded in Stage 2a.

#### REJ-004: Version negotiation inside Noise handshake

- **What:** Exchange version ranges in the encrypted payload after Noise handshake completion.
- **Proposed by:** SpecWriter, Stage 1
- **Why rejected:** Wastes DH computation on version mismatch. Pre-handshake probe with prologue binding provides equivalent security (downgrade detected via handshake failure). The "wasted handshake" costs microseconds on LAN but the pre-handshake approach is structurally simpler. SpecWriter conceded in Stage 2a.

#### REJ-005: schema_sync_required flag in VersionProbe

- **What:** Bit 0 of flags byte in VersionProbe signals schema sync needed.
- **Proposed by:** WireEng, Stage 1
- **Why rejected:** SchemaSync is a post-handshake concern. The responder cannot act on this flag pre-handshake. All flag bits reserved in v1.

## Part 6: Open Questions

These items were raised during the review but not resolved. They require
human decision.

### OQ-1: Chunk size default (512 vs 1024 bytes)

The SpecWriter recommends 512-byte bao chunks as default. The WireEng
chose 1024-byte chunks (bao standard). The TransferBegin message includes
a chunk_size field, so the choice is negotiable. The WireEng's efficiency
analysis shows ~40% wire efficiency for 1024-byte chunks at worst case
(2GB blob). The Savant recommends 1024 for bao standard consistency.

**Recommendation:** Accept 1024-byte chunks as the default. The bao
standard alignment is more valuable than marginal frame-packing
improvement. The chunk_size field enables future negotiation.

### OQ-2: Fixed-point arithmetic for health scores

Should health scores, EMA computations, and dominance cascades use
fixed-point instead of f64? The NaN Safety Invariant (R11-021) mitigates
the immediate danger, but floating-point determinism across platforms
remains a concern for convergence correctness.

**Recommendation:** Defer to Run 12. The NaN guard handles the acute
case. A thorough evaluation of fixed-point tradeoffs (precision loss vs
determinism gain) belongs in a focused design round.

### OQ-3: Peer offline vs partition detection

The heartbeat is the sole input to failure detection. The
PARTITION_ANN control log entry type exists but the trigger mechanism
is unspecified. When should a peer declare "I am partitioned" vs
"the other peer went offline"?

**Recommendation:** Defer to Run 12. This is a distributed systems
design question that requires careful specification.

### OQ-4: SACK field ordering finalization

R11-037 specifies the hybrid SACK format with all fields from both
experts. The exact field ordering needs final confirmation. The
WireEng awaits the SpecWriter's sign-off.

**Recommendation:** Accept the ordering in R11-037 (recv_window at
offset 14). This is the most logical position (after congestion
feedback fields, before SACK blocks).

## Part 7: Remaining Dangers

Ranked by severity per the Savant's final assessment (Stage 3b).

### Danger 1: SansIO Polling Model (B1 remainder) -- HIGH

The event enumerations (R11-016) tell the consumer what events exist.
They do not tell the consumer the calling convention. Without the
polling model, two implementations will use different driving patterns.
Wire interoperability is preserved (identical bytes on wire) but
integration fragmentation makes testing and library sharing difficult.

**Impact:** Implementation fragmentation, not interoperability failure.
**Target:** Run 12 (highest priority item).

### Danger 2: Intra-Cluster Communication (B2) -- HIGH

One sentence for an entire communication domain. The wire format
supports it (dest_peer field). The crypto supports it (CIK as Noise KK
static key). But the session model and addressing convention are
unspecified. Two implementations could assign dest_peer values
differently and fail to route intra-cluster traffic.

**Impact:** Potential interoperability failure within clusters.
**Target:** Run 12 design round.

### Danger 3: Chunk Size Calibration -- MEDIUM

1024-byte vs 512-byte default. The TransferBegin chunk_size field
enables negotiation, so the impact of a wrong default is bounded.
See OQ-1.

**Impact:** Suboptimal transfer efficiency, not failure.
**Target:** Final decision before merge.

### Danger 4: SACK Format Finalization -- MEDIUM

The hybrid format (R11-037) is drafted but the field ordering needs
explicit confirmation. See OQ-4.

**Impact:** Blocks full LEDBAT implementation.
**Target:** Convergence stage.

### Danger 5: Relay Territory (General) -- MEDIUM

Transit forwarding, relay authentication, multi-hop, failover are all
unspecified in the body. The membrane model and 5 operations have wire
formats (R11-040) but the transit relay's core function has no body text.

**Impact:** Relay is essential for NAT-traversed networks. Unspecified
relay behavior means the system cannot serve users behind strict NATs.
**Target:** Run 12 or Run 13 (focused relay run).

### Danger 6: Seat Failover (B8) -- LOW-MEDIUM

Host peer dies; seat has no recovery path. Simple retry mechanism
suffices for v1 (try next known host peer). The SeatTransport defined
concept (R11-024) makes reconnection mechanically straightforward.

**Impact:** User-visible service disruption until host peer returns.
**Target:** Run 12 (during seat body writing).

### Danger 7: Matter Garbage Collection (B4) -- LOW

Unbounded storage growth on devices with limited storage. A simple
LRU policy at the application layer addresses the immediate need.

**Impact:** Long-term storage exhaustion.
**Target:** Run 13 or later.
