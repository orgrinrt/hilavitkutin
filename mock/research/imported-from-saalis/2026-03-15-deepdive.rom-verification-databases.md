# Deep Dive: ROM Verification Databases (No-Intro, TOSEC, Redump)

## 1. Overview

ROM verification databases are community-maintained catalogues that fingerprint every known dump of a game or software title using cryptographic checksums. They do not distribute copyrighted content; they distribute *metadata* — file names, sizes, CRC32, MD5, and SHA-1 hashes — packaged as DAT files. Tools such as CLRMAMEPro, RomVault, and RomCenter consume these DAT files to audit, rename, and rebuild local ROM collections so that every file can be confirmed as a bit-perfect match against the canonical reference.

Three databases dominate the space, each with a distinct philosophy:

| Database | Focus | Media | Philosophy |
|----------|-------|-------|------------|
| **No-Intro** | Cartridge / card / digital | ROM chips, flash | One verified-good dump per game per region |
| **TOSEC** | Everything | All media types | Every known variant, historical completionism |
| **Redump** | Optical disc | CD, DVD, GD, BD, UMD | Bit-perfect disc preservation with sector-level accuracy |

## 2. DAT File Formats

DAT files come in two primary formats, both of which encode the same logical structure: a *header* block describing the database, followed by a list of *game* entries each containing one or more *rom* (or *disk*) entries with checksums.

### 2.1 CLRMAMEPro (ListInfo) Plain-Text Format

The original format, sometimes called "ListInfo" after the MAME command that generates it. It uses a brace-delimited, whitespace-sensitive syntax:

```
clrmamepro (
    name "Nintendo - Game Boy Advance"
    description "Nintendo - Game Boy Advance"
    version "20240301"
    author "No-Intro"
)

game (
    name "Legend of Zelda, The - The Minish Cap (USA)"
    description "Legend of Zelda, The - The Minish Cap (USA)"
    rom ( name "Legend of Zelda, The - The Minish Cap (USA).gba" size 16777216 crc 1b4221ec md5 a15f6c8e66e9688cee7c3c0ad8e920b6 sha1 c1e12ff46f6dbe51ab07d0dbe1e81c04a77c2260 )
)
```

Key syntax rules:

- The space between a tag name and its attribute (or opening parenthesis) is significant.
- Description values must be quoted; if a value contains quotes, escape with `\"`.
- Tags parsed: `name`, `description`, `cloneof`, `romof`, `year`, `manufacturer`, `rom` (with `name`, `size`, `crc`/`crc32`, `md5`, `sha1`, `merge`, `baddump`, `nodump`, `status`), `disk`, `sample`, `sampleof`.
- Tag order and case do not matter.
- Header can include `forcemerging` (`none`/`split`/`full`), `forcezipping` (`zip`/`unzip`), and `rebuildto`.

The official documentation lives at: https://mamedev.emulab.it/clrmamepro/docs/htm/datfile.htm

### 2.2 Logiqx XML Format

The XML-based successor, now the dominant distribution format for No-Intro and Redump. It follows a DTD originally published at `http://www.logiqx.com/Dats/datafile.dtd`:

```xml
<?xml version="1.0"?>
<!DOCTYPE datafile PUBLIC "-//Logiqx//DTD ROM Management Datafile//EN"
  "http://www.logiqx.com/Dats/datafile.dtd">
<datafile>
  <header>
    <name>Nintendo - Game Boy Advance</name>
    <description>Nintendo - Game Boy Advance</description>
    <version>20240301-091200</version>
    <author>No-Intro</author>
    <homepage>No-Intro</homepage>
    <url>https://www.no-intro.org</url>
    <clrmamepro header="No-Intro_GBA.xml"/>
  </header>
  <game name="Legend of Zelda, The - The Minish Cap (USA)">
    <description>Legend of Zelda, The - The Minish Cap (USA)</description>
    <rom name="Legend of Zelda, The - The Minish Cap (USA).gba"
         size="16777216"
         crc="1b4221ec"
         md5="a15f6c8e66e9688cee7c3c0ad8e920b6"
         sha1="c1e12ff46f6dbe51ab07d0dbe1e81c04a77c2260"/>
  </game>
</datafile>
```

Key elements:

- **`<header>`**: `name`, `description`, `category`, `version`, `author`, `homepage`, `url`, `clrmamepro` (with optional `header`, `forcemerging`, `forcezipping` attributes), `romcenter` (with `plugin`, `rommode`, `biosmode`, `samplemode` attributes).
- **`<game>`**: attribute `name`; children `description`, `year`, `manufacturer`, `rom`, `disk`.
- **`<rom>`**: attributes `name`, `size`, `crc`, `md5`, `sha1`, `status` (`baddump`/`nodump`/`good`/`verified`).
- **`<disk>`**: for disc/CHD media; attributes `name`, `md5`, `sha1`.

The Logiqx XML DTD has been extended by many projects beyond its original scope. The most complete documentation of all known extensions is maintained by the SabreTools project: https://github.com/SabreTools/SabreTools/wiki/DatFile-Formats

### 2.3 Parent-Clone DAT (P/C XML)

No-Intro also publishes "Parent-Clone" XML DATs. These add `cloneof` attributes to `<game>` elements, grouping regional variants under a parent entry. This enables 1G1R (one game, one ROM) filtering where a tool picks the preferred region's ROM and discards duplicates.

## 3. How Checksum Matching Works

ROM verification is conceptually simple: compute a fingerprint of your file, look it up in the database.

### 3.1 Hash Algorithms

- **CRC32** (32-bit): Fast to compute, stored inside ZIP directory headers (so archives can be audited without extraction). Used as the primary quick-match key. Collision risk is non-trivial for adversarial inputs but acceptable for ROM identification where the file size also acts as a discriminator.
- **MD5** (128-bit): Stronger fingerprint. Most DATs include this alongside CRC32.
- **SHA-1** (160-bit): The strongest hash commonly used in DAT files. Considered the authoritative match — if SHA-1 matches, the file is identical to the reference dump.

### 3.2 Matching Process

1. **Scan**: The tool walks a directory tree. For each file, it computes CRC32 (and optionally MD5/SHA-1). If the file is inside a ZIP archive, CRC32 and size can be read directly from the ZIP central directory without decompression.
2. **Lookup**: The computed checksums are compared against every `<rom>` entry in the loaded DAT(s). A match requires **all provided checksums to agree** plus a matching file size.
3. **Classify**: Each ROM in the DAT is marked as *have* (matched), *missing* (no match found), *bad dump* (CRC mismatch or flagged `baddump` in DAT), or *unneeded* (file exists but matches no DAT entry).
4. **Fix/Rebuild**: The tool can rename misnamed-but-correct files, move matched files into the correct directory structure, and source missing ROMs from a "to-sort" pool.

### 3.3 Special Cases

- **Headerless matching**: Some platforms (NES, Atari Lynx) have copier headers prepended to ROM dumps. The DAT's `<clrmamepro header="..."/>` element points to an XML header definition that tells the tool how many bytes to skip before hashing. No-Intro hashes are computed on the headerless content.
- **CHD / compressed disc images**: Redump DATs describe uncompressed bin/cue images, but users often store disc images as CHD (MAME's compressed hunks of data) or RVZ (Dolphin's format). Tools like `verifydump` decompress to the canonical format before hash comparison.

## 4. The Three Major Databases

### 4.1 No-Intro

**Mission**: Catalogue the best available dump of each commercially released cartridge/card game. "No-Intro" means no intros, no scene-group modifications — just the raw chip data.

**Scope**: Cartridge-based consoles (NES, SNES, Game Boy, GBA, N64, DS, etc.), handheld card media, and increasingly digital-only platforms.

**Characteristics**:
- Only verified-good dumps are listed. Overdumps, underdumps, hacked, trained, or otherwise modified ROMs are excluded.
- One entry per unique dump. A game with three regional releases (USA, Europe, Japan) appears as three separate entries.
- Parent-Clone DATs group these regional entries under a single parent for 1G1R filtering.
- Versioned: the `version` field in the header is a datestamp. New DATs are released as new dumps are verified or naming corrections are made.

**Source**: https://datomatic.no-intro.org/ (DAT-o-MATIC)

**Download**: The DAT-o-MATIC website requires navigating a multi-step form with POST requests and cookie handling. There is no official public API. Programmatic access requires scraping the download flow (system selection via POST, cookie persistence, file retrieval). Community scripts exist (Python, shell) but are fragile against site changes. The main download page is at `https://datomatic.no-intro.org/index.php?page=download&op=dat`.

### 4.2 TOSEC (The Old School Emulation Center)

**Mission**: Catalogue *everything* — every known variant, every region, every revision, every crack, every bad dump. Historical completionism over curation.

**Scope**: Extremely broad. Covers consoles, home computers (Amiga, Atari ST, C64, ZX Spectrum, etc.), firmware, operating systems, applications, magazines, cover disks, comic books, and more. A single game may have hundreds of entries across regional variants, cracks, trainer versions, and alternate dumps.

**Characteristics**:
- The TOSEC Naming Convention (TNC) is its signature contribution. Every catalogued file follows a rigid naming format: `Title version (demo) (date)(publisher)(system)(video)(country)(language)(copyright)(devstatus)(media type)(media label)[dump info flags][more info]`.
- Country codes follow ISO 3166-1 alpha-2.
- Dump info flags include `[cr]` (cracked), `[f]` (fixed), `[h]` (hacked), `[o]` (overdump), `[a]` (alternate), `[b]` (bad dump), `[t]` (trained), `[!]` (verified good).
- DAT files use the ClrMAMEPro plain-text format.
- Less strict verification than No-Intro — TOSEC may include dumps of uncertain provenance. For systems without No-Intro or Redump coverage, TOSEC is often the only available catalogue.

**Source**: https://www.tosecdev.org/downloads

**Download**: TOSEC publishes complete DAT packs as ZIP archives. These are also mirrored on the Internet Archive (https://archive.org/details/tosec-main). Programmatic download is straightforward since the packs are direct-download ZIPs.

**Naming Convention Specification**: https://www.tosecdev.org/tosec-naming-convention

### 4.3 Redump

**Mission**: Preserve optical disc media at the bit-perfect level. Every track, every sector, raw and unmodified.

**Scope**: CD, DVD, GD (Dreamcast), BD (Blu-ray), UMD (PSP) — PlayStation, PlayStation 2, Saturn, Dreamcast, GameCube, Wii, Xbox, PC Engine CD, and dozens more.

**Characteristics**:
- DATs describe uncompressed disc images, typically in bin/cue format (one `.bin` per track, one `.cue` cuesheet).
- Each track has its own set of checksums (CRC32, MD5, SHA-1). Multi-track discs have multiple `<rom>` entries per `<game>`.
- Redump uses raw data tracks and write-offset correction on audio tracks, without RIFF headers. This methodology diverges from TOSEC's disc-dumping approach.
- The database stores metadata but not the disc images themselves.
- DATs can optionally include serial numbers and version information by appending parameters to the download URL.

**Source**: http://redump.org/downloads/

**Download**: Redump DAT files follow a predictable URL pattern: `http://redump.org/datfile/{system}/`, e.g., `http://redump.org/datfile/ps2/`. Optional parameters can be appended: `http://redump.org/datfile/ps2/serial,version`. A complete list of system codes is maintained at: http://wiki.redump.org/index.php?title=List_of_DB_Download_Links. Direct HTTP GET downloads make programmatic retrieval trivial compared to No-Intro.

**Cuesheet downloads**: Redump also provides cuesheet-only downloads at http://redump.org/downloads/ for use with compressed formats.

## 5. ROM Management Tools

### 5.1 CLRMAMEPro

The oldest and most established ROM manager (Windows-native, runs under Wine). It loads DAT files as "profiles", scans ROM directories, and produces detailed audit reports. It can rebuild sets by moving/renaming/zipping files to match the DAT. Supports both CLRMAMEPro plain-text and Logiqx XML DAT formats. Advanced features include merge handling (split, non-merged, full-merged sets for arcade ROMs), header skip support, and batch operations.

Homepage: https://mamedev.emulab.it/clrmamepro/

### 5.2 RomVault

A more modern ROM manager. Uses three directory concepts: **DatRoot** (where DATs live), **RomRoot** (the organised collection), and **ToSort** (unsorted incoming files). Workflow: import DATs, scan ROMs, find fixes (calculates what can be rebuilt from available files), then execute fixes. Supports multiple DAT formats, nested directory trees, and can generate reports of missing/unfixable ROMs. Written in C#, cross-platform via .NET.

Homepage/Wiki: https://wiki.romvault.com/

### 5.3 RomCenter

Windows-based ROM manager with a visual, database-driven approach. Colour-coded status indicators show which databases are complete/incomplete/missing and which ROMs are playable/misnamed/incomplete/unknown. Supports CRC verification, renaming, dummy ROM creation, and reporting. Uses its own internal database format but imports standard DAT files.

Homepage: https://www.romcenter.com/

### 5.4 Igir

A modern, zero-setup, cross-platform CLI ROM collection manager (Node.js). Features not found in other managers: ROM patching, multi-disc playlist creation, ROM extension correction. By default uses CRC32 + file size for matching (fast path via ZIP directory inspection); supports `--input-checksum-min` to escalate to MD5, SHA-1, or SHA-256 for stronger verification.

Homepage: https://igir.io/

### 5.5 oxyROMon

A cross-platform CLI ROM organizer **written in Rust**. Designed with archiving in mind — only supports original and lossless ROM formats. Can import Logiqx DAT files and has built-in commands to download No-Intro and Redump DATs. Supports console, computer, and arcade (WIP) systems.

Repository: https://github.com/alucryd/oxyromon
Crate: https://crates.io/crates/oxyromon

## 6. Existing Rust Crates for DAT Parsing

The Rust ecosystem has several relevant crates:

### 6.1 `listinfo`

A zero-copy parser for the MAME ListInfo (CLRMAMEPro plain-text) DAT format. Provides two APIs: a low-level expression tree (`parse::parse_document()`) and a higher-level Serde deserialization API (feature-gated behind `"deserialize"`). Does *not* handle Logiqx XML — only the plain-text format.

- Crate: https://crates.io/crates/listinfo
- Docs: https://docs.rs/listinfo
- Repository: https://github.com/SnowflakePowered/listinfo-rs

### 6.2 `shiratsu-naming`

A zero-copy file name parser for No-Intro, TOSEC, and GoodTools naming conventions. Parses structured ROM entry names into typed components (title, region, version, flags, etc.) without regular expressions. Includes TOSEC name repair utilities. From the same author/org as `listinfo` (SnowflakePowered).

- Crate: https://crates.io/crates/shiratsu-naming
- Docs: https://docs.rs/shiratsu-naming

### 6.3 `oxyromon` (application, not library)

While published as a crate, oxyROMon is an application binary, not a reusable library. Its internal DAT parsing and ROM matching logic is not exposed as a public API. However, its source code (https://github.com/alucryd/oxyromon) is a valuable reference implementation for Logiqx XML parsing, No-Intro/Redump DAT downloading, and hash-based ROM auditing in Rust.

### 6.4 Notable gap: Logiqx XML parser crate

There is no standalone Rust crate that parses Logiqx XML DATs into typed structures. The XML format is straightforward enough to parse with `quick-xml` or `roxmltree` + Serde, but a purpose-built crate with proper `<header>`, `<game>`, `<rom>`, `<disk>` types and checksum validation does not exist on crates.io as of this writing. This is a potential area for saalis to contribute upstream.

## 7. Integration Strategy for Saalis

ROM verification fits naturally into saalis's architecture as a **metadata enrichment step** within the Curator subsystem. Here is how the pieces connect:

### 7.1 DAT Files as Metadata Sources

DAT files are, in saalis terms, just another metadata source — analogous to scraper APIs like ScreenScraper or IGDB, but offline and checksum-based rather than title-based. A DAT connector plugin would:

1. **Ingest**: Parse DAT files (Logiqx XML for No-Intro/Redump, CLRMAMEPro plain-text for TOSEC) into saalis's internal metadata model.
2. **Index**: Build a hash map keyed by `(crc32, size)` for fast lookup, with MD5 and SHA-1 available for confirmation.
3. **Persist**: Store the parsed DAT data as metadata entries against a `DatSource` entity type, or as columns on game entities.

### 7.2 Verification as Enrichment

When saalis catalogues a ROM file:

1. Compute CRC32 (fast) and file size during the initial file scan. This is cheap enough to do eagerly.
2. Look up `(crc32, size)` in the in-memory DAT index.
3. On match, record the DAT-provided canonical name, region, and verification status as metadata on the entity.
4. Optionally compute SHA-1 for confirmation (heavier, could be deferred to a background work unit).
5. If no match, flag the entity as "unverified" — it may be a bad dump, a modified ROM, or simply not yet catalogued.

### 7.3 DAT Acquisition

- **Redump**: Trivial. HTTP GET to predictable URLs. Can be automated with a periodic scheduler work unit.
- **TOSEC**: Direct ZIP download from tosecdev.org or Internet Archive mirrors. Also automatable.
- **No-Intro**: Harder. DAT-o-MATIC requires form-based interaction. Options: (a) ship a bundled snapshot and update manually, (b) scrape the download flow (fragile), (c) use community mirrors on the Internet Archive, (d) let the user manually place DAT files into a watched directory. Option (d) is the most robust for initial implementation.

### 7.4 Data Model Sketch

In saalis's entity/metadata model:

- A `VerificationSource` (No-Intro, TOSEC, Redump) is a registered entity type.
- Each DAT file import creates or updates entries under that source.
- A ROM entity gains metadata columns like `verification_status` (verified | unverified | bad_dump | no_match), `dat_canonical_name`, `dat_region`, `dat_version` (the DAT version that provided the match).
- The `(crc32, md5, sha1, size)` tuple is stored as intrinsic metadata on every file-backed entity, computed at scan time.

### 7.5 Implementation Priorities

1. **Phase 1**: Parse Logiqx XML DATs (covers No-Intro + Redump). Use `quick-xml` with Serde. Build CRC32+size index. Verify on catalogue.
2. **Phase 2**: Parse CLRMAMEPro plain-text DATs (covers TOSEC, older MAME DATs). Use the `listinfo` crate.
3. **Phase 3**: Use `shiratsu-naming` to extract structured metadata (region, version, flags) from DAT entry names.
4. **Phase 4**: Automated DAT acquisition for Redump and TOSEC. Manual/watched-directory for No-Intro.
5. **Phase 5**: 1G1R filtering using Parent-Clone DATs — let users define region priority and surface a single preferred ROM per game.

### 7.6 Crate Dependencies

| Purpose | Crate | Notes |
|---------|-------|-------|
| Logiqx XML parsing | `quick-xml` + serde | No existing typed crate; roll our own structs |
| CLRMAMEPro plain-text parsing | `listinfo` | Zero-copy, serde support |
| ROM name parsing | `shiratsu-naming` | No-Intro + TOSEC naming conventions |
| CRC32 computation | `crc32fast` | SIMD-accelerated, standard choice |
| MD5 computation | `md5` | Only needed for confirmation pass |
| SHA-1 computation | `sha1` | Only needed for confirmation pass |
| HTTP downloads | `ureq` (sync) | For DAT file retrieval; sync fits saalis's core |
