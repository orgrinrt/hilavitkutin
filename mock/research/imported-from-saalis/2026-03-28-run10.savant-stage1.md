**Reviewer:** Structural Integrity Savant (crystallographic lens)

## Lattice Overview

The crystal has seven unit cells arranged in a strict DAG:

```
crypto (root)
  |
protocol
  |    \
mesh    seat
  |  \
relay  matter
         |
   (also -> relay)

ops (terminal leaf, no outgoing bonds)
```

The lattice is not regular. Three cells (crypto, mesh, protocol) are dense with internal structure. Two cells (seat, matter) are partially hollow. One cell (relay) is mid-density. One cell (ops) is a passive accumulator with no bonds flowing out.

## 1. Lattice Regularity

### Section ordering

The taxonomy mandates: Abstract, Dependencies, Defined Concepts, Body, References. Six of seven documents follow this order. All six design documents place Defined Concepts before Body, matching the taxonomy. verkko-ops has no Defined Concepts section (correct; it defines none).

**Regularity score: high.** The pattern holds across all cells. No structural disorder in document shape.

### Defined concept format

Each defined concept has: Name (PascalCase), 1-2 sentence description, Invariants list. This format is uniform across all six documents that define concepts. No document breaks the pattern.

**One irregularity:** verkko-crypto's `FoundingHash` defined concept mentions mesh-layer artifacts (invite tokens, snapshots, join confirmations) in its invariants. These are mesh concerns leaking into a crypto invariant declaration. The invariant should state the hash's properties (determinism, uniqueness), not where it appears in the mesh lifecycle. This is a minor inclusion defect (see section 4).

### Attribution blocks

Four of seven documents carry attribution blocks (crypto, protocol, mesh, relay). Matter has one. Seat and ops have none. The inconsistency is cosmetic but suggests seat and ops were written in a different pass. Not a structural defect.

## 3. Vacancies (Missing Bonds)

### Vacancy V1: Seat has no bond to Mesh

**Crystallographic:** The seat unit cell shares faces with crypto and protocol but has zero bonds to mesh. It is a dangling cell in the lattice, connected only at the bottom.

**Protocol:** Seat defines a HostPeer concept that "translates between the seat's request/response world and the mesh's collective state." But HostPeer's invariants never reference any mesh concept. The translation is asserted, not specified. What mesh operations does the host peer perform? What access control model maps seat roles to channel access levels? The taxonomy says "host peer handles all mesh operations on behalf of the seat" but there is no specification of which operations or how.

**Emergent:** Under load (many seats, partition event), the host peer must handle convergence while serving seat requests. There is no specification of what happens to seat sessions during convergence. Does the seat see a degraded service? Does it get disconnected? The absence of this bond means seat behavior during mesh stress is undefined.

### Vacancy V2: Matter has no bond to Seat

**Crystallographic:** Matter depends on mesh, relay, and crypto. Seat depends on crypto and protocol. There is no path from seat to matter except through a host peer (which is mesh-side). The seat cannot directly address matter.

**Protocol:** A seat user wants to download a game ROM. The seat sends a SeatRequest to the host peer. The host peer looks up matter via ContentFilter. The host peer initiates MatterTransfer. The host peer delegates a SeatInstruction (download, place, verify) back to the seat. This flow is implied but never specified. The SeatInstruction defined concept lists "download, place, link, remove, verify, inventory sync" but these instruction types are not bound to matter operations.

**Emergent:** Without this bond, there is no specification for how a seat learns what matter exists, requests matter, or reports matter status. The entire user-facing content workflow is a vacancy.

### Vacancy V3: Matter has no direct bond to Protocol

**Crystallographic:** Matter depends on crypto, mesh, and relay. It does not list protocol as a dependency. But MatterTransfer's invariants mention "LAN path: direct streaming" and "WAN path: bao-tree verified streaming." These are transport concerns. How does matter get from peer A to peer B? Through protocol's Connection and Stream abstractions? Through relay's TransitRelay? The transport binding is unspecified.

**Protocol:** Matter's transfer protocol section says "multi-source: sequential provider fallback" and "mesh-first: local peer -> mesh peers -> external source." But it never references how a transfer uses protocol's Stream multiplexing, LEDBAT pacing, or VitalQueue priority assignment. Is a matter transfer a bulk-tier stream? A reactive-tier stream?

**Emergent:** Without this bond, matter transfer performance is undetermined. A naive implementation might starve foreground traffic (violating LEDBAT's purpose) or be starved itself (preventing replication targets from being met).

### Vacancy V4: No KeyBootstrap in crypto

**Crystallographic:** The taxonomy lists "Key bootstrap -- first mesh formation" as a critical gap. The crypto document defines genesis bootstrap sequence (DK, DEK, CIK generation and founding transcript) but not how a second device discovers and trusts the first device's keys during initial mesh formation. The InviteFlow in mesh provides the user-facing ceremony, but the cryptographic binding between invite token and key exchange is split between crypto (Noise XX with PSK) and mesh (invite contains mesh identity, bootstrap peer address, PSK). Neither document fully owns the bootstrap handshake.

**Protocol:** The invite flow says "SAS verification required for trust establishment" and crypto says "Noise XX with PSK." But the PSK distribution (QR code contents, expiry, nonce) is specified in crypto's device recovery section (1.4), not in mesh's InviteFlow. InviteFlow is a defined concept with minimal invariants: "invite contains: mesh identity, bootstrap peer address, PSK for first handshake" and "one-time-use credential."

**Emergent:** First mesh formation and new-cluster joining share the same cryptographic ceremony but are split across documents with different levels of specificity. The device recovery path in crypto (section 1.4) is more specified than the invite flow in mesh. A new cluster joining for the first time may hit unspecified edge cases (what if the PSK expires during the Noise handshake? what if mDNS fails and the bootstrap address is stale?).

## 4. Dislocations (Concepts in the Wrong Cell)

### Dislocation D1: FoundingHash invariants reference mesh artifacts

**Crystallographic:** FoundingHash is defined in verkko-crypto. Its invariant says "FoundingHash appears in invite tokens, snapshots, join confirmations." These are mesh concepts. A crypto invariant should not enumerate its consumers. The invariant should be: "FoundingHash is deterministic and unique per founding transcript." Where it appears is the consuming document's concern.

**Protocol:** Harmless today. But if a new document (say verkko-admin) adds a new use of FoundingHash, the crypto invariant would need updating. This is a maintenance burden from a misplaced invariant.

**Emergent:** Low risk. The invariant is overly specific but not wrong. The directional error (crypto knowing about mesh's use) is minor because the list is short.

### Dislocation D2: Heartbeat extension fields duplicated across protocol and mesh

**Crystallographic:** The heartbeat frame format is defined in verkko-protocol section 3.4. The extension field semantics are defined in verkko-mesh section "Heartbeat extension fields." Both sections contain the field list (filter_root, epoch_current, work_capacity, health_canary, observation_summary, gossip_mode). The protocol section defines the byte layout and the mesh section defines the semantics. But the field names and sizes appear in both places.

**Protocol:** Any change to extension field sizes requires updating both documents. The `work_capacity` computation note appears in protocol ("min(bw_kbps/128, flash_kb/256, 255)") and the same formula appears in mesh ("Computed as min(bw_kbps/128, flash_kb/256, 255)"). This is a duplication defect.

**Emergent:** A change to extension field size (e.g., expanding observation_summary from 4 to 8 bytes) requires coordinated edits in two documents. If one is updated and the other is not, the documents disagree on heartbeat frame size. This is a grain boundary (see section 5).

### Dislocation D3: Section 2.6 numbering collision

**Crystallographic:** Both verkko-crypto and verkko-mesh have a section numbered 2.6 about PCS. Crypto's 2.6 is "CPSK-seeded post-compromise security (PCS) -- mechanism." Mesh's 2.6 is "CPSK-seeded PCS -- trigger and distribution." The v10 provenance markers explain why (both came from the same v10 section 2.6), but in the decomposed lattice, this is a naming collision that creates ambiguity in cross-references.

**Protocol:** When relay says "see verkko-crypto: PCSMechanism," the reference is unambiguous because it uses the defined concept name. But internal references like "Section 2.6" are ambiguous. The mesh document itself says "For the PCS key derivation mechanism, see verkko-crypto: PCSMechanism" but also has its own 2.6.

**Emergent:** Low risk if cross-references always use defined concept names. Medium risk if any consumer references "section 2.6" without specifying the document.

## 5. Grain Boundaries (Imperfect Document Interfaces)

### Grain Boundary G1: Protocol-Mesh heartbeat interface

**Crystallographic:** The heartbeat is the most heavily trafficked bond in the lattice. It carries transport-layer fields (defined in protocol) and extension fields (defined in mesh). The total frame size (91 bytes) is computed in protocol but depends on extension field sizes defined by mesh. This is a grain boundary: neither document fully owns the heartbeat frame layout.

**Protocol mechanism:** Protocol says "18 bytes fixed header + 18 bytes extension region = 36 bytes inside encrypted frame." Mesh defines what those 18 extension bytes contain. If mesh needs 19 bytes, protocol's frame format changes. If protocol changes the extension region allocation, mesh's field layout changes.

**Emergent behavior:** This grain boundary is load-bearing. The heartbeat is the sole input to failure detection, health scoring, epoch sync, filter divergence detection, gossip mode switching, and metabolic state observation. Any crack at this interface propagates to all six consumers. The interface should be formalized as a defined concept (HeartbeatExtension) with explicit size bounds.

### Grain Boundary G2: Relay-Mesh convergence interface

**Crystallographic:** Relay's body references mesh's convergence fronts by number ("Front 1", "Front 3") and mesh's internal thresholds ("vital queue depth < 25%"). Mesh's convergence section references relay's re-encryption ("see verkko-relay: TerritorialReencryption"). The interface is bidirectional and informal.

**Protocol mechanism:** Mesh defines three convergence fronts. Relay defines what happens during Front 3 (bulk re-encryption) and what happens speculatively during Front 1 (P1 re-encryption). Relay also defines a gate condition: "Blocked until: Consensus gate open AND vital queue depth < 25%." The "vital queue depth" is a protocol concept (VitalQueue), not a mesh concept. Relay is reaching across two document boundaries.

**Emergent behavior:** Adding a fourth convergence front (e.g., for seat synchronization) would require updating both mesh and relay. Changing the vital queue threshold from 25% to 30% requires finding the reference in relay's body. This interface is brittle.

### Grain Boundary G3: Matter-Mesh source selection interface

**Crystallographic:** Matter's source selection predicate is "possession AND access AND reachability AND health(effective_score) AND NOT assembling(raptor)." The health(effective_score) term comes from mesh's HealthPipeline. But matter does not list HealthPipeline as a dependency; it references "verkko-mesh: Cluster, Gateway, Gossip, HealthPipeline, Convergence, Channel." HealthPipeline is listed as a dependency. The interface is through effective_score, which is min(fast_ema, slow_ema). But matter never defines what health threshold it requires for source selection.

**Protocol mechanism:** The Boundary Object Table in ops lists "Source selection predicate" with contract "Requires effective_health >= threshold." But the threshold value is not specified in matter or ops. This is a missing parameter.

**Emergent behavior:** Without a defined threshold, implementations will choose their own, leading to inconsistent source selection across peers. This could cause matter transfers to fail on one peer (strict threshold) while succeeding on another (lenient threshold).

### Grain Boundary G4: Crypto-Mesh domain separator ownership

**Crystallographic:** The DomainSeparatorRegistry is defined in crypto and contains 7 registrations. Two of them ("saalis-jitter-v1", "saalis-hbjitter-v1") serve mesh/protocol purposes. The registry's invariant says "No two derivations with different purposes share the same domain separator" and "All domain separators are versioned or structurally distinct." But the enforcement is documentation-only.

**Protocol mechanism:** Crypto says "Future additions must register a unique domain separator." But there is no compile-time or lint-time enforcement that a new BLAKE3 use in matter or relay has been registered. The BLAKE3 implementation equivalence requirement (crypto section 4.1) says "A peer whose BLAKE3 output diverges from the reference on any registered domain separator will silently partition from the mesh." Silent partitioning from an unregistered separator would be undetectable.

**Emergent behavior:** As the system grows, the probability of an unregistered domain separator increases. The failure mode (silent mesh partition) is the worst kind: no error, no notification, just divergent state. This is the crystal's most dangerous latent defect.

## 6. Cleavage Planes

Where does this crystal fracture under stress?

### Cleavage Plane C1: Heartbeat format change

**Stress:** Adding a new extension field to the heartbeat.

**Fracture path:** protocol (frame format) -> mesh (extension semantics) -> relay (work_capacity interpretation) -> matter (filter_root divergence detection) -> ops (boundary object table, mechanism reference). Five documents affected.

**Severity:** High. The heartbeat is the lattice's load-bearing bond. Any change to its format propagates to every document except crypto and seat.

### Cleavage Plane C2: New convergence front

**Stress:** Adding a fourth convergence front (e.g., seat resynchronization or matter replication verification).

**Fracture path:** mesh (convergence section) -> relay (front numbering, gate conditions) -> matter (sealed-range catch-up ordering) -> ops (mechanism reference table, game day exercises). Four documents affected.

**Severity:** Medium-high. The convergence specification is tightly coupled across mesh, relay, and matter. The front numbering is used as a coordination protocol. Adding a front between Front 2 and Front 3 would require renumbering in three documents.

### Cleavage Plane C3: New BLAKE3 domain separator

**Stress:** A new feature requiring a BLAKE3 derivation (e.g., matter deduplication hint).

**Fracture path:** If the new domain separator is not registered in crypto's DomainSeparatorRegistry, peers running different code versions will silently diverge. If it is registered, only crypto needs updating (low fracture). The asymmetry between registered (clean) and unregistered (catastrophic) makes this a hidden cleavage plane.

**Severity:** Catastrophic if unregistered. Trivial if registered. The gap between these outcomes is the defect.

### Cleavage Plane C4: MetabolicState expansion

**Stress:** Adding a fifth metabolic state (e.g., "Emergency" below Background).

**Fracture path:** protocol (state machine definition, heartbeat interval table, LEDBAT target delay table) -> mesh (Conductor parameter maps, convergence metabolic override) -> ops (calibration items). Three documents affected, but the parameter maps in mesh would need the most work (every "f_p(metabolic_state)" function gains a new case).

**Severity:** Medium. The Conductor invariant was designed for extensibility ("New resource dimensions enter the governor's min() without changing the metabolic state machine"), but adding a new state to the state machine itself is the opposite of adding a new resource dimension. The invariant protects against resource growth, not state growth.

## 7. Empty Cells

### verkko-seat: ~15% filled

**Current content:** 6 defined concepts with minimal invariants. No body section. No algorithms. No wire formats. No cost analysis.

**Minimum lattice for structural soundness:**

1. **SeatSession wire format.** How does the seat authenticate? The defined concept says "user credential" but does not specify the credential type. Password? Token? Client certificate? This determines the bond to crypto (which AEAD? which key exchange?).

2. **HostPeer translation spec.** Which mesh operations can a seat trigger? The defined concept says "handles all mesh operations on behalf" but "all" is unbounded. At minimum: query matter (ContentFilter lookup), request transfer (MatterTransfer initiation), observe status (health, connectivity). Each operation needs access control mapping from seat role to channel access level.

3. **Seat-during-convergence behavior.** What happens to active seat sessions when the host peer enters convergence? Options: (a) seat sessions continue with stale data, (b) seat sessions are paused and resume after convergence, (c) seat sessions see a "convergence in progress" event. This must be specified because seat users are the humans who notice problems.

4. **SeatInstruction protocol.** The defined concept lists instruction types but not the protocol for executing them. How does the seat report partial progress on a 2GB download? How does the host peer retry a failed instruction? What is the idempotency contract?

### verkko-matter: ~40% filled

**Current content:** 11 defined concepts, partial body (sealed-range catch-up, filter reconciliation, cuckoo filters, content integrity). Missing: transfer protocol, stash management, replication policy, sync reconciliation, bandwidth policies.

**Minimum lattice for structural soundness:**

1. **MatterTransfer protocol.** Must bond to protocol's Stream and VitalQueue. A matter transfer is a bulk-tier stream (Queue 3). Bao-tree verified streaming must specify chunk size, verification frequency, and what happens on verification failure mid-transfer.

2. **Stash-to-ContentFilter binding.** Stash is local state. ContentFilter is mesh-wide. The bridge between them (how local stash changes propagate to the filter) is unspecified. This determines the convergence time for matter discovery.

3. **Replication policy.** min_replicas and target_replicas are mentioned but their enforcement mechanism is not. Who decides which peer replicates what? Is it the gateway? The admin? A hash-ring assignment (like territorial re-encryption)? The replication scheduling mechanism is a vacancy.

## Strengthening Proposals

### Proposal S1: Formalize HeartbeatExtension as a defined concept in protocol

**Crystallographic:** Create a bond point at the protocol-mesh grain boundary. The HeartbeatExtension defined concept would own the extension region layout, with invariants specifying the maximum size (18 bytes currently) and the field allocation.

**Protocol mechanism:** Protocol defines:
```
HeartbeatExtension {
    max_size: 18 bytes,
    fields: [
        filter_root: [u8; 8],
        epoch_current: u32,
        work_capacity: u8,
        health_canary: u8,
        observation_summary: [u8; 4],
        // gossip_mode is bit 2 of state byte (transport header, not extension)
    ],
    reserved: 0 bytes,
}
```
Invariant: total extension bytes <= max_size. Mesh consumes HeartbeatExtension by name, not by byte offset.

**Emergent:** Adding a new extension field requires updating HeartbeatExtension (one place in protocol) and the consuming logic (mesh). The grain boundary becomes a clean bond. Protocol's frame size computation references HeartbeatExtension::max_size instead of a magic 18.

### Proposal S2: Domain separator registration lint

**Crystallographic:** Seal the lattice defect at G4 by making unregistered BLAKE3 usage a compile-time error.

**Protocol mechanism:** A lint rule (like the existing forbidden-imports lints) scans all crate source for BLAKE3 calls. Each call must reference a domain separator string that appears in a canonical registry (a const array or enum). Unregistered strings are a lint error.

**Emergent:** The silent-partition failure mode (C3) becomes impossible. New BLAKE3 uses force registration, which forces a review of the crypto document. The DomainSeparatorRegistry invariant ("no two derivations with different purposes share the same domain separator") becomes enforceable rather than aspirational.

### Proposal S3: Name convergence fronts, do not number them

**Crystallographic:** Replace the "Front 1, Front 2, Front 3" numbering with named fronts: ControlFront, ConsensusFront, ReencryptionFront. Names are stable under insertion. Numbers are not.

**Protocol mechanism:** Mesh defines:
```
ControlFront: control log merge, partition processing, EMA reset
ConsensusFront: revocations, fencing, dominance, hash ring
ReencryptionFront: bulk re-encryption under new keys
```
Relay references "ReencryptionFront" instead of "Front 3." Matter references "ControlFront" for sealed-range ordering.

**Emergent:** Adding a fourth front (e.g., SeatSyncFront) does not require renumbering. Cross-document references are stable. The cleavage plane C2 is sealed: insertion no longer cascades through naming.

### Proposal S4: Defined concept for source selection health threshold

**Crystallographic:** Fill the vacancy at grain boundary G3 by specifying the health threshold for source selection.

**Protocol mechanism:** Matter defines a const or calibration parameter:
```
SOURCE_HEALTH_THRESHOLD: HealthScore = HealthScore::try_new(0.50).unwrap()
```
With invariant: "A peer with effective_health below SOURCE_HEALTH_THRESHOLD is excluded from source selection for matter transfer." The value 0.50 aligns with the "Bad" health state boundary in mesh (slow EMA < 0.50).

**Emergent:** All peers use the same threshold for source selection, preventing inconsistent transfer behavior. The threshold is a calibration item (add to ops calibration table). The Boundary Object Table entry for "Source selection predicate" gains a concrete contract.

### Proposal S5: Explicit seat-mesh mediation contract

**Crystallographic:** Fill vacancy V1 by defining the HostPeer's mesh-facing obligations as a set of mesh operations with access control mappings.

**Protocol mechanism:** Seat defines a table:

| Seat Role | Permitted Mesh Operations |
|-----------|--------------------------|
| Admin | All: query, transfer, configure, revoke, enroll |
| User | Query matter, request transfer, observe status, report progress |
| Child | Query matter (filtered), request transfer (filtered), observe status |

Each operation maps to specific mesh-layer interactions:
- "query matter" -> ContentFilter lookup via gateway
- "request transfer" -> MatterTransfer initiation (host peer acts as requesting peer)
- "observe status" -> subscribe to HealthPipeline classification changes
- "report progress" -> SeatProgress (already defined)

**Emergent:** The seat's relationship to mesh operations is specified. During convergence, the host peer can defer "query matter" requests (filter is reconciling) and pause "request transfer" requests (territorial re-encryption in progress) while continuing "observe status" events. The seat user sees "mesh converging, transfers paused" instead of silent failure.

### Proposal S6: Convergence gate as a defined concept in mesh

**Crystallographic:** Extract the gate conditions scattered across mesh and relay into a defined concept that both documents reference.

**Protocol mechanism:** Mesh defines:
```
ConvergenceGate {
    control_gate: bool,     // all control entries processed, all reachable keys obtained
    consensus_gate: bool,   // highest fencing token confirmed, dominance resolved
    reencryption_gate: bool, // all affected content re-encrypted or epoch-forwarded
    vital_queue_clear: bool, // vital queue depth < 25%
}
```
Invariant: ReencryptionFront begins only when consensus_gate AND vital_queue_clear. Relay references ConvergenceGate::consensus_gate instead of "Consensus gate open."

**Emergent:** The relay-mesh grain boundary (G2) becomes a clean bond through ConvergenceGate. Relay no longer reaches across to protocol (VitalQueue depth) directly; it consumes the gate condition from mesh.

## Summary: Crystal Health Report

**Overall lattice quality:** The crypto-protocol-mesh core is a well-formed crystal with high internal symmetry and strong bonds. The relay cell is adequately bonded but coupled to mesh internals through informal references. The seat and matter cells are partially hollow, creating structural weakness at the periphery.

**Strongest region:** crypto -> protocol. Clean dependency. Protocol uses only crypto's invariants. No leakage in either direction. This is the lattice's backbone.

**Weakest region:** seat. Zero bonds to mesh, zero bonds to matter. The cell exists as a structural placeholder. Under any load (user interaction, convergence, matter transfer), the seat cell contributes nothing to the lattice's strength because its interfaces are unspecified.

**Most dangerous defect:** The DomainSeparatorRegistry enforcement gap (G4/C3). Silent mesh partitioning from an unregistered BLAKE3 domain separator is a catastrophic failure mode with no detection mechanism. This is the crystal's hidden fracture waiting to happen.

**Cleavage plane hierarchy (most fragile first):**
1. C3: Unregistered domain separator (catastrophic, silent)
2. C1: Heartbeat format change (high blast radius, 5 documents)
3. C2: New convergence front (medium-high, 4 documents)
4. C4: MetabolicState expansion (medium, 3 documents)

**Priority repairs:**
1. S2 (domain separator lint) -- seals the most dangerous defect
2. S1 (HeartbeatExtension defined concept) -- hardens the most trafficked bond
3. S3 (named convergence fronts) -- prevents cascade from insertion
4. S4 (source selection threshold) -- fills a functional gap
5. S6 (ConvergenceGate defined concept) -- cleans relay-mesh boundary
6. S5 (seat-mesh mediation) -- fills the largest vacancy but is design work, not repair
