# Deep Dive: Serde Patterns and Performance Optimization

## Context

Saalis uses serde extensively: settings serialization to/from SQLite, API response parsing
(IGDB, ScreenScraper), configuration files (RON/TOML), and descriptor data in the plugin
ABI layer. The project enforces `&'static str` everywhere with `String` disallowed in the
trait/descriptor surface, and uses `u32` interning handles for runtime string comparison.
This research covers serde usage patterns, performance characteristics, and integration
strategies relevant to that architecture.

---

## 1. Derive vs Manual `impl Serialize / Deserialize`

### When derive is sufficient

Serde's derive macro generates code that the Rust compiler can optimize down to the same
speed as a handwritten serializer for most struct/enum shapes. The generated code handles
generics, trait bounds, and attribute annotations transparently. For the vast majority of
types — settings structs, API response DTOs, configuration models — derive is the correct
choice.

```rust
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub struct LibraryConfig {
    pub scan_interval_secs: u64,
    #[serde(default)]
    pub auto_enrich: bool,
}
```

### When manual impl pays off

Manual implementations become worthwhile in three situations:

1. **Avoiding intermediate allocations.** If the derived code would allocate a `String` or
   `Vec` only to immediately convert it into a different representation (e.g., an interned
   `u32` handle), a manual `Deserialize` that calls `visitor.visit_str` and interns in-place
   eliminates that allocation entirely.

2. **Non-standard wire representations.** When a type's serialized form diverges
   significantly from its in-memory layout — for instance serializing a bitflag set as
   a comma-separated string, or serializing an `EntityId` as `"game:42"` — derive cannot
   express the transformation and a manual impl is required.

3. **Cross-crate `repr(C)` descriptors.** For ABI-stable types that must roundtrip through
   FFI boundaries, manual impls give full control over the exact byte layout and field
   ordering, avoiding any derive-generated code that might change between serde versions.

### Performance note

A manual `Serialize` impl that writes fields directly to the serializer (avoiding any
intermediate `Content` or `Value` buffering) can be measurably faster for hot-path types.
Benchmark before committing to manual impls — the derive output is already quite good and
maintenance cost of hand-rolled impls is real.

## 2. Common Attribute Patterns

### `#[serde(rename_all = "...")]`

Applied at the container level. Renames all fields or enum variants according to a case
convention. Processing happens at compile time with zero runtime cost.

```rust
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]  // user_name -> userName
pub struct ApiResponse { ... }

#[derive(Serialize, Deserialize)]
#[serde(rename_all = "SCREAMING_SNAKE_CASE")]  // enum variants
pub enum Priority { VeryHigh, Medium, Low }
```

Available conventions: `lowercase`, `UPPERCASE`, `PascalCase`, `camelCase`, `snake_case`,
`SCREAMING_SNAKE_CASE`, `kebab-case`, `SCREAMING-KEBAB-CASE`.

For external APIs (IGDB returns camelCase JSON, ScreenScraper returns XML with its own
conventions), `rename_all` on the response struct avoids per-field `#[serde(rename)]`
annotations.

### `#[serde(default)]`

Fills missing fields with `Default::default()` during deserialization. Essential for
backwards-compatible configuration: when a new setting is added, old config files that
lack the field still deserialize successfully.

```rust
#[derive(Serialize, Deserialize)]
pub struct ScanSettings {
    pub roots: Vec<PathBuf>,
    #[serde(default = "default_depth")]
    pub max_depth: u32,
    #[serde(default)]            // uses bool::default() = false
    pub follow_symlinks: bool,
}
fn default_depth() -> u32 { 5 }
```

Combine with `#[serde(skip_serializing_if = "is_default")]` to keep serialized output
clean — fields at their default value are omitted.

### `#[serde(skip)]`

Excludes a field from both serialization and deserialization. The field must implement
`Default` (or have a `default = "..."` function) because serde needs to construct it
during deserialization.

Use cases: computed caches, runtime handles, fields that must not leak into config files
or API responses.

```rust
#[derive(Serialize, Deserialize)]
pub struct PluginState {
    pub name: &'static str,
    #[serde(skip)]
    pub handle: Option<dlopen::Handle>,  // runtime-only
}
```

### `#[serde(flatten)]`

Flattens a nested struct's fields into the parent's serialized representation. Powerful
for composition but carries a **hidden deserialization cost**: serde buffers all
key-value pairs into a `Vec<Option<(Content, Content)>>`, then replays them to each
flattened field via `FlatMapDeserializer`. This roughly **doubles deserialization time**
compared to the unflattened equivalent. Serialization is cheaper — `FlatMapSerializer`
writes directly to the parent map.

```rust
#[derive(Serialize, Deserialize)]
pub struct Entity {
    pub id: EntityId,
    #[serde(flatten)]
    pub metadata: CommonMetadata,  // fields merged into Entity's map
}
```

**Recommendation for saalis:** Avoid `#[serde(flatten)]` on hot-path types (e.g., bulk
entity lists from API responses). Use it freely for configuration types where
deserialization happens once at startup. For hot paths, consider the `serde_fast_flatten`
crate which approaches unflattened performance.

## 3. Zero-Copy Deserialization

### The mechanism

Serde's lifetime system allows deserializing into types that borrow directly from the
input buffer rather than allocating new memory. This is controlled by the `'de` lifetime
on `Deserialize<'de>`.

- `&'de str` and `&'de [u8]` are implicitly borrowed from input.
- Other borrowing types (like `Cow<'de, str>`) require `#[serde(borrow)]`.

```rust
#[derive(Deserialize)]
pub struct ApiHit<'a> {
    pub name: &'a str,                    // implicitly borrowed
    #[serde(borrow)]
    pub description: Cow<'a, str>,        // borrowed when possible, owned when escaped
}
```

### `Cow<'de, str>` — the practical sweet spot

`Cow` is the cornerstone of zero-copy serde. When the input contains a plain string (no
escape sequences in JSON, for example), serde borrows it directly (`Cow::Borrowed`). When
escapes force transformation, it falls back to an owned `String` (`Cow::Owned`). This
gives best-of-both-worlds: zero allocations in the common case, correctness always.

### Format compatibility

Not all formats support zero-copy:

| Format      | Zero-copy `&str` | Zero-copy `&[u8]` | Notes                         |
|-------------|------------------|--------------------|-------------------------------|
| serde_json  | Yes (from `&[u8]`) | Yes              | Only from `from_slice`, not `from_str` for `&[u8]` |
| postcard    | Yes              | Yes                | Binary, direct buffer refs    |
| bincode     | Yes              | Yes                | Binary, direct buffer refs    |
| toml        | Limited          | No                 | Strings often require escaping |
| ron         | Limited          | No                 | Similar to TOML               |

### Relevance to saalis's `Str` primitive

Since saalis disallows `String` in the trait/descriptor surface and uses `&'static str`
for compile-time-known names and `u32` interned handles for runtime strings, zero-copy
deserialization applies primarily at the **boundary layer** — parsing API responses and
config files before interning. A custom `Deserialize` impl for the interned handle type
can accept a `&str` from the visitor and immediately intern it, never allocating a
`String`:

```rust
impl<'de> Deserialize<'de> for InternedStr {
    fn deserialize<D: Deserializer<'de>>(deserializer: D) -> Result<Self, D::Error> {
        struct InternVisitor;
        impl<'de> Visitor<'de> for InternVisitor {
            type Value = InternedStr;
            fn expecting(&self, f: &mut fmt::Formatter) -> fmt::Result {
                f.write_str("a string")
            }
            fn visit_str<E: de::Error>(self, v: &str) -> Result<InternedStr, E> {
                Ok(InternedStr::intern(v))  // intern in place, no String
            }
            fn visit_string<E: de::Error>(self, v: String) -> Result<InternedStr, E> {
                Ok(InternedStr::intern(&v))  // accept owned too
            }
        }
        deserializer.deserialize_str(InternVisitor)
    }
}
```

## 4. Custom Serializers and Deserializers

### Serializing `Cow<'static, str>`

For types that hold `Cow<'static, str>` (borrowed from a static string table or owned
from runtime), the default `Serialize` derive works correctly — both `Borrowed` and
`Owned` variants serialize as a plain string. No custom impl needed.

For *deserialization*, a custom impl is needed if you want to map known values back to
`&'static` references (avoiding allocation):

```rust
fn visit_str<E: de::Error>(self, v: &str) -> Result<Cow<'static, str>, E> {
    // Check static lookup table first
    if let Some(static_ref) = KNOWN_STRINGS.get(v) {
        Ok(Cow::Borrowed(static_ref))
    } else {
        Ok(Cow::Owned(v.to_owned()))
    }
}
```

### Serializing the `u32` interned handle

On serialization, resolve the handle back to its string for human-readable formats
(JSON, TOML, RON) and emit the raw `u32` for binary formats (postcard, bincode):

```rust
impl Serialize for InternedStr {
    fn serialize<S: Serializer>(&self, serializer: S) -> Result<S::Ok, S::Error> {
        if serializer.is_human_readable() {
            serializer.serialize_str(self.resolve())
        } else {
            serializer.serialize_u32(self.0)
        }
    }
}
```

The `is_human_readable()` method is available on all serde `Serializer` and
`Deserializer` implementations. JSON, TOML, RON return `true`; postcard, bincode return
`false`.

### `#[serde(with = "...")]` for field-level customization

When a type's default serde impl is wrong for a specific field but right elsewhere, use
`with` to apply a custom module:

```rust
mod unix_timestamp {
    pub fn serialize<S: Serializer>(ts: &SystemTime, s: S) -> Result<S::Ok, S::Error> { ... }
    pub fn deserialize<'de, D: Deserializer<'de>>(d: D) -> Result<SystemTime, D::Error> { ... }
}

#[derive(Serialize, Deserialize)]
pub struct Entry {
    #[serde(with = "unix_timestamp")]
    pub created_at: SystemTime,
}
```

## 5. Format Comparison

### Human-readable formats

| Format | Crate         | Use case                   | Strengths                              | Weaknesses                         |
|--------|---------------|----------------------------|----------------------------------------|------------------------------------|
| JSON   | `serde_json`  | API responses, web UI      | Universal, tooling everywhere          | Verbose, no comments, no trailing commas |
| TOML   | `toml`        | Cargo-style config files   | Familiar to Rust devs, Cargo uses it   | Weak for deeply nested structures  |
| RON    | `ron`         | Rust-native config/data    | Supports enums/tuples natively, comments, trailing commas | Less known outside Rust ecosystem |

**Recommendation for saalis:** RON for internal configuration and extension manifests
(it handles Rust enums naturally, which fits the `#[register]` descriptor model). JSON
for the web API layer and anything touching the browser. TOML only if user-facing config
must be Cargo-familiar.

### Binary formats

| Format   | Crate      | Size    | Speed                | no_std | Zero-copy | Notes                              |
|----------|------------|---------|----------------------|--------|-----------|------------------------------------|
| postcard | `postcard` | Small   | Fast                 | Yes    | Yes       | Documented wire format, varint encoding |
| bincode  | `bincode`  | Medium  | Very fast            | Partial| Yes       | v2 split from v1; fixed-size encoding  |
| bitcode  | `bitcode`  | Smallest| Fast                 | Yes    | Yes       | Bit-level packing, best compression |
| rkyv     | `rkyv`     | Medium  | Fastest (zero-deser) | Yes    | Yes       | Zero-copy *access* (no deserialization step) |

**Benchmark summary** (from rust_serialization_benchmark):
- **rkyv** wins nearly every speed benchmark due to true zero-deserialization (archived data
  is accessed in-place without any conversion step).
- **bitcode** wins on size — bit-level packing produces the smallest output.
- **postcard** and **bincode** are very close in speed; postcard consistently beats bincode
  on size thanks to varint encoding.

**Recommendation for saalis:** postcard for internal persistence and IPC (documented
stable wire format, excellent no_std support, good size/speed tradeoff). JSON for the
web boundary. If bulk entity storage in SQLite blobs becomes a bottleneck, evaluate
bitcode for maximum compression or rkyv for maximum read speed.

## 6. Error Handling

### The problem with default errors

Standard serde errors report *what* went wrong and (in JSON) a line/column number, but
for large or minified JSON (single-line API responses), finding column 4832 is painful.

### `serde_path_to_error`

Wraps any serde `Deserializer` and tracks the field path to the first error:

```rust
let jd = &mut serde_json::Deserializer::from_str(input);
let result: Result<Config, _> = serde_path_to_error::deserialize(jd);
match result {
    Err(err) => {
        let path = err.path().to_string();  // e.g. "settings.scan.roots[2]"
        let inner = err.into_inner();       // original serde error
        tracing::error!(%path, %inner, "config parse failed");
    }
    Ok(config) => { ... }
}
```

### `serde_json_path_to_error`

A drop-in replacement for `serde_json` that enriches all errors with path information
automatically — no wrapper boilerplate. Preferable as a default for development builds.

### `format_serde_error`

Produces human-readable error messages with source context (underlines the offending
line). Useful for user-facing config validation in CLI output.

### Custom error messages via `de::Error::custom`

For domain validation during deserialization:

```rust
fn visit_str<E: de::Error>(self, v: &str) -> Result<Priority, E> {
    match v {
        "low" => Ok(Priority::Low),
        "medium" => Ok(Priority::Medium),
        "high" => Ok(Priority::High),
        other => Err(E::custom(format_args!(
            "unknown priority '{}', expected one of: low, medium, high", other
        ))),
    }
}
```

**Recommendation for saalis:** Use `serde_path_to_error` at every deserialization
boundary (API parsing, config loading, extension manifest validation). The path context
is invaluable for debugging plugin descriptors with nested structures.

## 7. Schema Validation with `schemars`

### What it does

`schemars` derives JSON Schema (2020-12) from Rust types via `#[derive(JsonSchema)]`. It
reads `#[serde(...)]` attributes and mirrors them in the schema — `rename_all`,
`default`, `skip`, `tag`, `flatten` all produce the correct schema output.

### Integration with serde

```rust
use schemars::JsonSchema;

#[derive(Serialize, Deserialize, JsonSchema)]
#[serde(rename_all = "camelCase")]
pub struct ExtensionManifest {
    pub name: String,
    pub version: String,
    #[serde(default)]
    pub permissions: Vec<Permission>,
}
```

Generates a JSON Schema where `name` appears as `name` (not renamed because snake_case
to camelCase is identity here), `permissions` has a `default: []`, etc.

### Override without affecting serde

`#[schemars(...)]` attributes override `#[serde(...)]` for schema generation only:

```rust
#[serde(default)]
#[schemars(range(min = 1, max = 3600))]
pub scan_interval_secs: u32,
```

### Validator integration

Schemars reads `#[validate(...)]` (from the `validator` crate) and `#[garde(...)]`
attributes to produce more precise schemas with `minLength`, `pattern`, `minimum`, etc.

### Use case for saalis

Extension manifest validation is the primary use case. Generate a JSON Schema from the
Rust `ExtensionManifest` type and validate extension manifests at load time before
attempting deserialization. This catches structural errors early with clear messages.
The schema can also be published for extension authors to validate against in their
editors.

## 8. Performance

### JSON parsers compared

Benchmarks from serde-rs/json-benchmark on representative datasets:

| Parser       | Twitter (us) | CITM (ms)  | Canada (ms) | Notes                          |
|--------------|--------------|------------|-------------|--------------------------------|
| serde_json   | ~2300        | ~3.0       | ~9.4        | Baseline, mature, reliable     |
| simd-json    | ~1100        | ~2.1       | ~8.1        | Tape-based, then struct parse  |
| sonic-rs     | ~830         | ~1.4       | ~4.0        | Direct-to-struct, no tape      |

**sonic-rs** is 2-3x faster than serde_json because it parses directly into Rust structs
using SIMD without an intermediate tape representation. simd-json is 1.5-2x faster
but still uses a two-stage tape-then-parse approach.

Both SIMD libraries require x86_64 (SSE2/AVX2) or aarch64 (NEON) — they include scalar
fallbacks for other architectures but lose most of their advantage.

### Compile-time cost of derives

Serde derives are a significant contributor to compile times in large projects. Each
`#[derive(Serialize, Deserialize)]` invocation parses the full `TokenStream` into
`syn::DeriveInput`, generates monomorphized serialization code, and produces generic
impls that LLVM must later optimize.

**Mitigation strategies:**

1. **Feature-gate serde.** Put `serde` behind a cargo feature in library crates. Only
   enable it in leaf crates that actually perform I/O. This is particularly relevant for
   `saalis-primitives` and `saalis-sdk` — most internal types do not need serde, only
   types that cross the persistence or web boundary do.

2. **Minimize derive scope.** Only derive `Serialize`/`Deserialize` on types that are
   actually serialized. Use separate DTO types at boundaries rather than deriving on
   core domain types.

3. **Consider `serde_repr` for C-like enums.** For `repr(C)` enums with no data (like
   `Role`), `serde_repr` generates far less code than full `Serialize`/`Deserialize`
   derives — it serializes as the integer discriminant.

4. **Proc-macro optimization.** Set `opt-level = 2` for build dependencies in
   `Cargo.toml` to speed up proc-macro compilation itself:

   ```toml
   [profile.dev.build-override]
   opt-level = 2
   ```

### Runtime performance patterns

- Avoid `#[serde(flatten)]` on hot paths (2x deserialization cost).
- Prefer `from_slice` over `from_str` for JSON (enables zero-copy `&[u8]` borrowing).
- Use `serde_json::to_writer` instead of `to_string` + `write_all` to avoid the
  intermediate `String` allocation.
- For bulk operations, reuse a `serde_json::Deserializer` with `StreamDeserializer`
  to parse newline-delimited JSON without re-parsing the entire buffer.

## 9. Versioned Serialization and Schema Evolution

### Built-in serde tools

**`#[serde(default)]`** — the simplest evolution strategy. New fields added with defaults
are backwards-compatible: old data missing the field gets the default value.

**`#[serde(alias = "...")]`** — accept old field names alongside new ones. Zero-cost at
runtime (resolved at compile time):

```rust
#[derive(Deserialize)]
pub struct ScanConfig {
    #[serde(alias = "scan_dirs")]   // old name
    pub roots: Vec<PathBuf>,        // new name
}
```

**`#[serde(deny_unknown_fields)]`** — strict mode that rejects unexpected fields.
Useful for validating extension manifests where typos should be caught, but harmful for
forwards compatibility (new fields from newer versions would be rejected).

### Tagged version enums

For major schema changes, use an internally tagged enum:

```rust
#[derive(Deserialize)]
#[serde(tag = "version")]
pub enum Settings {
    #[serde(rename = "1")]
    V1(SettingsV1),
    #[serde(rename = "2")]
    V2(SettingsV2),
}
```

This pattern lets you deserialize any version and migrate to the latest in application
code.

### External versioning libraries

- **serde-evolve:** Compile-time verified migrations. Separates wire format from domain
  types. Good fit for saalis descriptors where the wire format is `repr(C)` stable but
  the application type may evolve.
- **revision** (SurrealDB): Revision-tolerant serialization. Fields can be added/removed
  across revisions with explicit annotations. Designed for database storage.
- **serde-flow:** Defines multiple struct variants for loading historical data. Simpler
  than serde-evolve but less compile-time safety.

### Recommendation for saalis

Use `#[serde(default)]` + `#[serde(alias)]` for settings evolution — they cover 90% of
cases with zero ceremony. For the plugin ABI descriptor format where binary compatibility
matters across versions, evaluate `serde-evolve` for its compile-time migration
verification. Reserve tagged version enums for rare major schema breaks.

## 10. `no_std` + `alloc` Serde

### Feature flags

Serde provides three tiers:

| Feature | Includes                                    | Requires      |
|---------|---------------------------------------------|---------------|
| (none)  | Primitive types, fixed-size arrays, `&str`  | `core` only   |
| `alloc` | `String`, `Vec<T>`, `Box<T>`, `Cow<T>`, `BTreeMap` | `alloc` crate |
| `std`   | `HashMap`, `HashSet`, `OsString`, `PathBuf`, etc.  | `std`         |

```toml
[dependencies]
serde = { version = "1", default-features = false, features = ["derive", "alloc"] }
```

### What works without `std`

- All derive macros (with one exception below).
- `Serialize` and `Deserialize` for all core and alloc types.
- Custom `Serializer`/`Deserializer` implementations.
- `#[serde(borrow)]`, zero-copy deserialization.
- `serde_repr` for C-like enums.

### What breaks without `std`

- **Untagged enums** in `derive(Deserialize)` — these require `std` because the
  implementation uses `Content` buffering that depends on `std::error::Error`. This is
  a known limitation.
- `HashMap`/`HashSet` impls — only available with `std`. Use `BTreeMap`/`BTreeSet` from
  `alloc` instead, or bring in `hashbrown` with its serde feature.
- `std::io::Read`/`Write`-based APIs (e.g., `serde_json::from_reader`) — these require
  `std`. Use `from_slice`/`to_vec` instead.

### Format crate `no_std` support

| Crate       | `no_std` | `alloc` | Notes                                    |
|-------------|----------|---------|------------------------------------------|
| postcard    | Yes      | Optional| Primary design target is `no_std`        |
| bincode     | Partial  | Yes     | v2 has better `no_std` support           |
| bitcode     | Yes      | Yes     | Designed for constrained environments    |
| serde_json  | No       | No      | Requires `std` (open issue #1040)        |
| toml        | No       | No      | Requires `std`                           |
| ron         | No       | No      | Requires `std`                           |

### Relevance to saalis

Saalis targets a single-binary Linux appliance (Batocera) and uses `std` — `no_std` is
not a constraint for the application itself. However, `saalis-primitives` and
`saalis-sdk` may benefit from `no_std` + `alloc` compatibility to keep them lightweight
and to allow potential future use in WASM plugins. The key takeaway: if those crates need
serde, use `default-features = false` with `features = ["derive", "alloc"]` and avoid
`HashMap` (use `BTreeMap` or `hashbrown`). Avoid untagged enums in types that must be
`no_std`-compatible.

## Summary of Recommendations for Saalis

| Area                  | Recommendation                                                    |
|-----------------------|-------------------------------------------------------------------|
| Derive vs manual      | Derive by default. Manual impl for `InternedStr` and `repr(C)` ABI types. |
| Attributes            | `rename_all` for API DTOs, `default` for all settings, avoid `flatten` on hot paths. |
| Zero-copy             | `Cow<'de, str>` at parsing boundaries, intern immediately after.  |
| Formats               | RON for config/manifests, JSON for web API, postcard for binary persistence. |
| Error handling        | `serde_path_to_error` at every deserialization boundary.          |
| Schema validation     | `schemars` for extension manifest validation + publishing schema. |
| JSON performance      | `serde_json` is sufficient; evaluate `sonic-rs` only if API parsing benchmarks demand it. |
| Compile time          | Feature-gate serde in primitives/SDK crates, `opt-level = 2` for build deps. |
| Versioning            | `#[serde(default)]` + `#[serde(alias)]` for settings; tagged enums for major breaks. |
| `no_std`              | Use `default-features = false` + `alloc` in primitives/SDK if WASM plugins are pursued. |
