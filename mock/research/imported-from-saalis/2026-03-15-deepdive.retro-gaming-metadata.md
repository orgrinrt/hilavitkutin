# Deep Dive: The Retro Gaming Metadata Landscape

**Date:** 2026-03-15
**Scope:** Comprehensive survey of platforms, ROM formats, naming conventions, verification,
BIOS requirements, regional variants, patching, modern platforms, essential metadata fields,
and how existing tools handle the complexity.

---

## 1. Platform/System Count: The Scale of the Problem

The retro gaming ecosystem is vastly larger than most people assume.

### RetroArch / Libretro

RetroArch's libretro API exposes **over 200 cores** spanning emulators, game engines, media
players, and standalone games. These cores cover **40+ distinct gaming platforms**, though the
exact count depends on how you slice variants (e.g., Game Boy vs Game Boy Color vs Super Game
Boy). Core availability varies by host platform -- a desktop build has the full catalogue,
while iOS or 3DS builds offer a subset.

Key systems with dedicated libretro cores include: NES, SNES, Game Boy / GBC / GBA, N64,
Nintendo DS, Virtual Boy, Sega Master System, Genesis/Mega Drive, Sega CD, Sega 32X, Saturn,
Dreamcast, PlayStation, PS2 (via PCSX2), PSP (via PPSSPP), Atari 2600/5200/7800/Jaguar/Lynx,
PC Engine / TurboGrafx-16, Neo Geo (AES/MVS/CD/Pocket), WonderSwan, 3DO, Vectrex, ColecoVision,
Intellivision, MSX/MSX2, Amstrad CPC, ZX Spectrum, Commodore 64/128/Amiga, DOS (via DOSBox),
ScummVM, and dozens of arcade board variants through FinalBurn Neo and MAME cores.

### MAME

MAME (which merged with MESS) documents **over 32,000 individual systems** -- arcade boards,
home computers, consoles, calculators, chess computers, and electronic amusement devices from
the last five decades. Not all are fully playable; roughly 10,000+ titles are classified as
working. The sheer count comes from the fact that MAME treats each PCB revision, regional
variant, and hardware configuration as a distinct system entry.

### Active ROM Preservation

Two primary preservation organisations curate verified databases:

- **No-Intro** focuses on cartridge-based media. Their DAT-o-Matic system provides DAT files
  (XML databases of verified hashes) for well over 100 distinct platforms. Each DAT contains
  every known verified dump for that platform, with CRC32, MD5, and SHA-1 hashes. No-Intro's
  philosophy is to catalogue only the *best* available dump of each title -- clean, unmodified,
  as close to the original silicon as possible.

- **Redump** focuses on optical media (CD, DVD, GD-ROM, UMD, Blu-ray). They have surpassed
  **50,000 verified PC disc entries** alone, and cover dozens of disc-based systems including
  PlayStation, PS2, PS3, PSP, Saturn, Dreamcast, GameCube, Wii, Xbox, 3DO, CD-i, PC Engine CD,
  Sega CD, Neo Geo CD, and more. Every entry is verified through a rigorous community dumping
  process using tools like Redumper and DiscImageCreator.

- **TOSEC** (The Old School Emulation Center) takes a broader cataloguing approach, listing
  *everything* found in the wild -- not just verified best dumps but also bad dumps, alternate
  dumps, trainer versions, and non-game content like magazines and manuals. TOSEC is especially
  valuable for less well-known systems (home computers, obscure consoles) where No-Intro/Redump
  coverage is thin.

Combined, these projects cover **several hundred distinct platforms** when you include all
variants of home computers, handhelds, consoles, and arcade systems.

## 2. ROM Format Differences by Platform

ROM file formats are not just different file extensions -- they encode fundamentally different
information about the original hardware, and getting the format wrong can mean broken emulation.

### NES: .nes (iNES / NES 2.0)

The `.nes` format wraps raw PRG and CHR ROM data in a **16-byte iNES header**. The header
begins with the ASCII magic `NES` followed by `0x1A` (DOS EOF marker). Bytes 4-5 specify
PRG ROM size (in 16 KiB units) and CHR ROM size (in 8 KiB units). Bytes 6-7 encode hardware
flags: mapper number (which memory-mapping chip the cartridge used), mirroring mode, battery
backup presence, and trainer presence.

The **NES 2.0** extension expands the header to describe submappers, extended ROM sizes, and
additional hardware details that the original iNES format couldn't represent. An optional
512-byte "trainer" block (legacy from copier devices) may follow the header before the actual
ROM data.

Correct mapper identification is critical -- the same game data with the wrong mapper number
will produce garbage. No-Intro DATs include mapper information and some ROM managers can
auto-detect it.

### SNES: .sfc / .smc

The fundamental data is the same; the difference is a **512-byte copier header**:

- `.sfc` = **headerless** (raw ROM data). This is the modern preferred format.
- `.smc` = **headered** (512-byte Super Magicom copier header prepended to the ROM data).

You can detect the copier header by checking `file_size % 1024`: if the remainder is 512,
a copier header is present; if 0, the ROM is headerless. The copier header itself is a relic
of backup devices like the Super Magicom and serves no purpose for emulation. Most modern
emulators handle both formats, but ROM patches (IPS/BPS) are typically authored for one
format or the other, meaning a headered patch on a headerless ROM will corrupt the output.

The SNES also has an **internal ROM header** (part of the actual cartridge data) at a fixed
offset that contains the game title, mapping mode (LoROM/HiROM/ExLoROM/ExHiROM), ROM size,
RAM size, region code, and a checksum. Emulators read this to configure memory mapping.

### Game Boy Family: .gb / .gbc / .gba

- `.gb` = original Game Boy ROMs
- `.gbc` = Game Boy Color ROMs (may be Color-only or Color-enhanced originals)
- `.gba` = Game Boy Advance ROMs

These are straightforward binary dumps of the cartridge ROM. The GB/GBC internal header (at
offset 0x100-0x14F) contains the Nintendo logo (used as a boot check), title, CGB flag
(Color support), cartridge type (MBC chip), ROM size, and RAM size. GBA ROMs have their own
header format with the game title, game code, and maker code at the start.

### Sega Genesis/Mega Drive: .md / .gen / .bin

- `.md` = Mega Drive (Japanese/European naming)
- `.gen` = Genesis (North American naming)
- `.bin` = generic binary dump

These are all the same format -- raw 16-bit ROM data in big-endian byte order. The internal
header at offset 0x100-0x1FF contains the console name, copyright, game name (domestic and
overseas), serial number, checksum, and region codes. The `.smd` extension (Super Magic Drive)
indicates an interleaved format from a copier device and needs deinterleaving.

### N64: .z64 / .n64 / .v64 (Byte Order Matters)

N64 ROMs exist in three byte orderings, identifiable by the first 4 bytes of the file:

| Format | Extension | Byte Order | Magic Bytes |
|--------|-----------|------------|-------------|
| Native (Big-Endian) | `.z64` | ABCD | `80 37 12 40` |
| Byte-Swapped | `.v64` | BADC | `37 80 40 12` |
| Little-Endian | `.n64` | DCBA | `40 12 37 80` |

The `.z64` (big-endian, native) format is the canonical one. The others are artefacts of
different copier devices. Modern emulators auto-detect the byte order, but ROM managers
should normalise to `.z64`. The internal header occupies the first 0x40 bytes and contains
the game title, code, and CRC checksums.

### Disc-Based Systems: .iso / .bin+.cue / .chd / .pbp

Disc-based games introduce significant format complexity:

- **BIN/CUE**: The `.bin` file is a raw sector-by-sector dump of the disc; the `.cue` file is
  a text "cue sheet" describing track layout (data tracks, audio tracks, indexes, gaps). A
  single disc may produce one `.bin` per track or one monolithic `.bin`. This is Redump's
  preferred raw format.

- **ISO**: A raw dump of the data track only, using 2048-byte sectors (Mode 1). Suitable for
  data-only discs but loses audio tracks and subchannel data. Common for PS2, GameCube, PSP.

- **CHD (Compressed Hunks of Data)**: MAME's lossless compression format. A single `.chd` file
  replaces a BIN/CUE pair, achieving 15-95% compression depending on content. Managed by the
  `chdman` tool. CHD supports configurable hunk sizes (16 bytes to 1 MiB) and multiple
  compression algorithms. Widely supported by RetroArch cores and standalone emulators. CHD
  files are single-disc only -- multi-disc games need multiple CHD files.

- **PBP (PlayStation Portable)**: Originally Sony's PSP game packaging format. The community
  repurposed it to compress PS1 disc images using the PSX2PSP tool. Uniquely, PBP can
  consolidate **multiple discs into a single file**, making it the only common format that
  solves multi-disc in one file. However, PBP conversion is lossy (the original BIN/CUE cannot
  be perfectly reconstructed), and it only works for PS1 content.

- **ECM (Error Code Modeler)**: A compression scheme that strips predictable error correction
  data from raw disc images, achieving good compression. Needs decompression before use.

- **CSO/ZSO**: Compressed ISO formats used for PSP, reducing file size while allowing random
  access reads.

## 3. BIOS Requirements per Platform

Many emulated systems require original BIOS/firmware dumps to function. BIOS files are
copyrighted, cannot be distributed with emulators, and must be sourced from the user's own
hardware. This is a major usability challenge for library managers.

### Systems That Require BIOS

| System | BIOS Files | Notes |
|--------|-----------|-------|
| PlayStation 1 | `scph5500.bin` (JP), `scph5501.bin` (US), `scph5502.bin` (EU) | Region-specific. Some emulators (DuckStation) include HLE BIOS as fallback |
| PlayStation 2 | Named by version, e.g. `ps2-0200a-20040614.bin` or `SCPH-70012.bin` | Region-specific, multiple revisions |
| Sega CD | `bios_CD_U.bin`, `bios_CD_E.bin`, `bios_CD_J.bin` | Region-specific, required |
| Sega Saturn | `sega_101.bin` (JP), `mpr-17933.bin` (US/EU) | Required for all Saturn emulators |
| Sega Dreamcast | `dc_boot.bin`, `dc_flash.bin` | Optional for some emulators (Flycast), required for NAOMI/Atomiswave |
| Nintendo DS | `bios7.bin`, `bios9.bin`, `firmware.bin` | Required for DS; DSi additionally needs its own BIOS |
| Game Boy Advance | `gba_bios.bin` | Optional -- most cores include HLE BIOS |
| Neo Geo | `neogeo.zip` (contains multiple BIOS ROMs) | Required for arcade emulation, contains region variants |
| Atari 5200 | `5200.rom` | Required |
| Atari 7800 | `7800 BIOS (U).rom` | Required for some titles |
| PC Engine CD | `syscard3.pce` | Required for CD games |
| 3DO | `panafz10.bin` (Panasonic) | Required |
| MSX/MSX2 | Various BIOS ROMs per machine variant | Complex -- many machine variants |
| Amiga | Kickstart ROMs by model/version | Required, version-specific |

### Systems That Do NOT Require BIOS

NES, SNES, Game Boy/GBC/GBA (with HLE), Genesis/Mega Drive, N64, most Atari systems,
WonderSwan, Virtual Boy. These systems either have no BIOS in the original hardware or their
emulators implement HLE (High-Level Emulation) replacements.

### BIOS Verification

BIOS files should be verified against known MD5/SHA-1 hashes. Wrong BIOS versions cause
subtle compatibility issues -- a US BIOS won't boot JP-region PS1 games, and older BIOS
revisions may lack compatibility fixes present in later versions.

## 4. Naming Conventions

Three major naming systems have dominated ROM distribution:

### No-Intro Standard

Format: `Title (Region) (Languages) (Version) (Devstatus) (Additional) (Special) (License)`

Examples:
- `Super Mario Bros. (World)`
- `Legend of Zelda, The - A Link to the Past (USA) (En,Fr,De,Es)`
- `Castlevania - Symphony of the Night (USA) (Rev 1)`
- `Sonic the Hedgehog 2 (USA, Europe) (Beta)`

Rules:
- Title uses official publisher name; articles like "The" are moved to after a comma
- Region uses full country/region names: USA, Europe, Japan, World, etc.
- Languages use ISO 639-1 codes in parentheses: (En,Fr,De,Es,It)
- Version/revision: (Rev 1), (v1.1), (Beta), (Proto), (Demo)
- Only the *best* verified dump is named; bad dumps, hacks, and trainers are excluded

No-Intro is the de facto standard for cartridge-based ROM sets and is what most frontends and
scrapers expect.

### TOSEC Standard

Format: `Title vVersion (Year)(Publisher)(System)(Country)(Language)(CopyrightStatus)(DevelopmentStatus)(MediaType)(MediaLabel)(DumpFlags)`

Examples:
- `Pac-Man (1982)(Atari)(US)`
- `Lemmings (1991)(Psygnosis)(en)(Disk 1 of 2)[cr FLT]`

TOSEC uses a more exhaustive approach:
- Year is included in the filename
- Publisher is explicit
- Media type and label for multi-disc
- Dump quality flags: `[!]` (verified good), `[b]` (bad dump), `[cr]` (cracked), `[f]` (fixed),
  `[h]` (hacked), `[o]` (overdump), `[t]` (trained), `[tr]` (translated)
- TOSEC catalogs *everything* -- good, bad, modified, and everything in between

### GoodTools Convention

Format: `Title (Region) [flags]`

GoodTools (GoodNES, GoodSNES, GoodGEN, etc.) popularised "GoodSets" -- complete archives of
every known ROM for a platform. Their flag system uses brackets:

- `[!]` = verified good dump
- `[b]` = bad dump
- `[a]` = alternate version
- `[h]` = hack
- `[o]` = overdump
- `[p]` = pirate
- `[t]` = trained (cheats added)
- `[f]` = fixed to run on emulator
- `[T+Eng]` = English translation patch applied
- `(U)` = USA, `(E)` = Europe, `(J)` = Japan, `(W)` = World

GoodTools is largely considered legacy now. No-Intro superseded it for verification accuracy,
and GoodSets often contained bad dumps, hacks, and overdumps mixed with clean copies.

### Redump Naming

Redump uses a convention similar to No-Intro but for disc-based media:
`Title (Region) (Languages) (Disc N)` with disc numbering for multi-disc releases.

## 5. Multi-Disc Handling

Multi-disc games are common on PS1, PS2, Saturn, Dreamcast, Sega CD, and PC Engine CD.

### M3U Playlists

The standard solution is a `.m3u` playlist file -- a plain text file listing one disc image
per line, in order:

```
Final Fantasy VII (USA) (Disc 1).chd
Final Fantasy VII (USA) (Disc 2).chd
Final Fantasy VII (USA) (Disc 3).chd
```

The frontend displays the `.m3u` file as a single game entry. Disc swapping is handled through
the emulator's disc control interface (eject, change index, insert). Most RetroArch cores
and standalone emulators support M3U.

### CHD Multi-Disc

CHD files are inherently single-disc. Multi-disc CHD sets use one `.chd` per disc, tied
together with an `.m3u` playlist. This is the recommended approach for most disc-based
systems on Batocera and similar appliance distributions.

### PBP Multi-Disc

PBP is unique in supporting up to 5 discs in a single file. Tools like PSX2PSP merge
multiple disc images into one `.pbp` with built-in disc switching. However:
- PBP is PS1-only
- The conversion is lossy (original BIN/CUE cannot be reconstructed)
- Some emulators/cores have limited PBP multi-disc support
- RetroAchievements does not support PBP for achievement tracking

### Folder-Based Organisation

Some frontends support a "game folder" model where each multi-disc game lives in its own
directory, with the frontend presenting the folder as a single entry. ES-DE supports this
natively.

## 6. Regional Variants and Languages

### PAL vs NTSC: More Than a Label

The PAL/NTSC distinction affects far more than video output:

- **Frame rate**: NTSC = 60 Hz (59.94), PAL = 50 Hz. This is a ~17% speed difference.
- **Resolution**: NTSC = 720x480 (480i/p), PAL = 720x576 (576i).
- **CPU clock**: Many systems tie CPU speed to the video standard. PAL systems run slower.
- **Game behaviour**: Games not optimised for PAL run ~17% slower with lower-pitched music.
  Some games were "PAL-optimised" (speed-corrected), some partially (music fixed, gameplay
  still slow), and some not at all.

### Region Codes

Regions go beyond simple PAL/NTSC:
- **Japan (NTSC-J)**: Japanese language, 60 Hz, may have different content (censorship, bonus)
- **USA (NTSC-U)**: English, 60 Hz
- **Europe (PAL)**: Subdivided into numerous sub-regions (UK, France, Germany, etc.) often
  with localised text. Some PAL regions had unique censorship requirements.
- **Brazil (PAL-M)**: PAL encoding but at 60 Hz -- a unique hybrid
- **Korea (NTSC-K)**: Often based on JP release with Korean text
- **Asia (NTSC)**: English or Chinese, 60 Hz, sometimes different content from USA release

### Language Implications

Many games shipped as multi-language within a single region, but languages are not always
tied to region predictably. A European release might contain En, Fr, De, Es, It, while
a separate Scandinavian release contains En, Sv, No, Da. The No-Intro convention captures
this with language tags: `(Europe) (En,Fr,De,Es,It)`.

For a library manager, region and language are independent axes that both matter.

## 7. Fan Translations and ROM Hacks

### Patch Formats

ROM hacks and fan translations are distributed as patch files, not modified ROMs, for
legal reasons. The patch encodes only the byte differences from the original.

| Format | Full Name | Max ROM Size | Checksum | Notes |
|--------|-----------|-------------|----------|-------|
| IPS | International Patching System | 16 MiB | None | Oldest, simplest, most widespread. No integrity check. |
| UPS | Universal Patching System | No limit | CRC32 | Successor to IPS, removes size limit, adds checksum |
| BPS | Beat Patching System | No limit | CRC32 | Modern standard. Validates source ROM before patching. |
| xdelta | xdelta/VCDIFF | No limit | Yes | Best for large files (PSP, DS). Handles data relocation. |
| PPF | PlayStation Patch Format | No limit | MD5 | Designed specifically for disc images |

### Tracking Patched vs Unpatched

This is a significant challenge for library managers:

- Patches are authored against a specific ROM version (No-Intro verified, headered/headerless,
  specific revision). Applying a patch to the wrong base ROM produces corruption.
- A patched ROM's hashes will not match any No-Intro/Redump entry, making verification
  impossible against standard databases.
- The relationship is: (base ROM hash) + (patch file) = (patched ROM hash).
- RHDN (ROMHacking.net) maintains a database of patches with the expected base ROM hash
  and the resulting patched ROM hash.
- Some emulators (RetroArch, mGBA) support **soft-patching**: automatically applying an IPS/BPS
  file to a ROM at load time without modifying the original. The patch file is placed alongside
  the ROM with the same name and the appropriate extension.

For a library manager, the ideal model stores the unpatched (verified) ROM and the patch file
separately, applying patches at runtime or generating the patched ROM on demand.

## 8. Modern Platforms

### GOG DOSBox Packages

GOG ships classic DOS games as Windows/Mac/Linux installers that bundle:
- The original game data files
- A preconfigured DOSBox installation
- A `dosbox.conf` with game-specific settings

The game data can be extracted using **innoextract** (for Windows installers) without running
the installer. The extracted data files can then be used with any DOSBox build (DOSBox-X,
DOSBox Staging, RetroArch DOSBox cores). There is no DRM on the game data.

### GOG ScummVM Packages

GOG similarly bundles ScummVM games with a packaged ScummVM binary and the game's data files.
The data files can be extracted and used with any ScummVM installation. ScummVM identifies
games by their data file signatures, so extracted files can simply be pointed at.

### Steam Retro Games

Many classic/retro games on Steam ship with:
- The original ROM/data files embedded in the installation directory
- A bundled emulator (often DOSBox, ScummVM, or a custom wrapper)
- Usually no DRM on the actual game data

The ROM files can often be found in the Steam game's installation directory and used with
external emulators. Some publishers (Sega, SNK, Capcom) sell classic game compilations where
the ROMs are extractable.

### Extraction Challenges

Modern platform integration requires:
- Parsing installer formats (Inno Setup for GOG)
- Navigating Steam directory structures
- Identifying game data files among engine/wrapper files
- Handling wrapper-specific configuration (DOSBox .conf files, ScummVM game IDs)

## 9. Essential Metadata for a Library Manager

Based on the landscape analysis, a library manager for retro content needs these metadata fields:

### Core Identity
- **Platform/System**: The target hardware (NES, SNES, Genesis, etc.)
- **Title**: Canonical game name (preferably No-Intro normalised)
- **Region(s)**: Country/region codes (USA, Europe, Japan, etc.)
- **Languages**: ISO 639-1 codes for available languages
- **Revision/Version**: (Rev 1), (v1.1), (Proto), (Beta), (Demo)

### Verification & Integrity
- **Hash values**: CRC32, MD5, SHA-1 of the ROM file
- **Verification status**: No-Intro verified, Redump verified, unverified, bad dump
- **DAT source**: Which database validated this hash (No-Intro, Redump, TOSEC)
- **Header status**: Headered vs headerless (for NES, SNES, etc.)
- **Byte order**: For N64 ROMs (z64/v64/n64)

### Hardware & Emulation
- **BIOS requirements**: Which BIOS files are needed and for which region
- **Mapper/board type**: NES mapper number, SNES mapping mode, etc.
- **Save type**: SRAM, EEPROM, Flash, battery-backed (especially for GBA)
- **Special hardware**: Light gun, multitap, link cable, rumble, etc.
- **Recommended core/emulator**: Best emulator for this specific title

### File & Media
- **ROM format**: File extension and internal format details
- **File size**: Expected size for verification
- **Disc number**: For multi-disc games
- **Media type**: Cartridge, CD, DVD, UMD, floppy, cassette
- **Compression**: CHD, PBP, 7z, zip -- and whether the original can be reconstructed

### Provenance & Relationships
- **Parent/clone**: Relationship to other versions of the same title
- **Patch lineage**: If patched, link to base ROM hash and patch file
- **Source**: No-Intro set, Redump set, TOSEC set, GOG, Steam, manual dump
- **Series/franchise**: For organisational grouping

### Presentation
- **Box art / cover**: Front, back, spine, cartridge/disc art
- **Screenshots**: Title screen, in-game
- **Video**: Preview/gameplay clips
- **Description/synopsis**: Game summary
- **Genre, developer, publisher, release date, player count, rating**

## 10. How Existing Tools Handle This Complexity

### Skyscraper

Skyscraper is a powerful CLI scraper written in C++ that uses a **two-phase approach**:

1. **Scrape phase**: Queries online databases (ScreenScraper, TheGamesDB, IGDB, MobyGames,
   OpenRetro, ArcadeDB) and caches results locally in a resource cache (SQLite or file-based).
   Identification uses either filename matching or hash-based lookup (ScreenScraper supports
   CRC/MD5/SHA-1 lookup).

2. **Generate phase**: Combines cached resources into a `gamelist.xml` for the target frontend,
   compositing artwork from multiple sources (screenshot + box art + wheel logo = single image).

Skyscraper handles platforms via a platform configuration mapping ROM directories to system
identifiers. It supports ES-DE, AttractMode, and Pegasus frontends. It does *not* manage
ROMs, BIOS, or verification -- it is purely a metadata/media scraper.

### ES-DE (EmulationStation Desktop Edition)

ES-DE is a frontend/launcher that:
- Organises ROMs by system in a directory-per-system structure
- Reads `gamelist.xml` for metadata and media paths
- Supports folder-based multi-disc grouping
- Has built-in scraper support (ScreenScraper and TheGamesDB)
- Handles file extensions per-system via `es_systems.xml` configuration
- Launches games by invoking emulators with configured command-line templates

ES-DE's metadata model in `gamelist.xml` includes: name, description, image, marquee, video,
thumbnail, rating, release date, developer, publisher, genre, players, play count, last
played, favourite flag, and custom collections.

ES-DE does not handle ROM verification, BIOS management, patching, or download.

### LaunchBox / BigBox

LaunchBox is the most comprehensive existing solution, featuring:
- Its own **community-maintained metadata database** (gamesdb.launchbox-app.com) covering
  hundreds of platforms with extensive metadata and media
- **Import wizards** that scan ROM directories, match by filename, and pull metadata
- **Multi-system support** with per-platform emulator configuration
- **Playlist/collection** management for multi-disc and curated lists
- **PC game integration** with Steam, GOG, Epic via plugin ecosystem
- Metadata scraping from its own database, IGDB, and other sources
- Stores metadata in XML files per-platform

LaunchBox handles complexity through a thick desktop application (Windows-only for the full
version), which is the opposite of saalis's appliance-oriented approach.

### ScreenScraper (Database)

ScreenScraper.fr is the most comprehensive retro gaming metadata database, fed by community
contributions. It supports:
- **Hash-based identification**: CRC32, MD5, SHA-1 lookups (most accurate method)
- **Filename-based fallback**: Matches against known naming conventions
- **Multi-system coverage**: Hundreds of platforms
- **Rich media**: Box art, screenshots, title screens, in-game shots, videos, manuals, wheels,
  marquees, 3D box renders, cartridge/disc art, fan art, bezels
- **Multilingual metadata**: Game descriptions in multiple languages
- **API access**: RESTful API with rate limiting and user-level quotas

ScreenScraper's hash-based identification is the gold standard for ROM matching -- it works
regardless of filename, compression, or naming convention.

### ROM Managers (ClrMamePro, RomVault)

These tools focus specifically on ROM set management and verification:

- Load DAT files (No-Intro, Redump, TOSEC, MAME)
- Scan ROM directories and compute CRC32/MD5/SHA-1 hashes
- Verify ROMs match DAT entries; identify missing, extra, and corrupt files
- Handle header detection and removal (NES, SNES, N64, etc.)
- Rebuild/reorganise ROM sets to match DAT structure
- Manage merged, split, and non-merged MAME ROM set configurations

ClrMamePro offers more control and flexibility. RomVault offers better performance and TOSEC
handling. Neither provides game metadata, media, or any frontend functionality.

## Implications for Saalis

This landscape reveals several key challenges for a content-type-agnostic library manager
targeting retro gaming as its first domain:

1. **Platform identification cannot be guessed from file extension alone.** The same `.bin`
   extension is used by Genesis, Sega CD BIOS, Atari 2600, and raw disc images. Directory
   context (platform folder) or hash-based identification is required.

2. **Verification is hash-based.** CRC32+MD5+SHA-1 against No-Intro/Redump DATs is the only
   reliable way to identify and verify ROMs. Filename matching is a useful heuristic but not
   authoritative.

3. **BIOS management is a first-class concern.** Many platforms are unplayable without correct
   BIOS files, and BIOS requirements are per-platform, per-region, and per-emulator.

4. **Multi-disc is a solved problem** (M3U + CHD), but the library manager must understand
   the relationship between playlist files, individual disc files, and the logical "game" entity.

5. **Patched ROMs break verification.** A library manager that tracks both the original ROM
   and applied patches (with soft-patching support) provides the best of both worlds.

6. **Region and language are independent axes**, both essential for matching user preferences
   and avoiding duplicate entries in the library.

7. **Metadata sourcing should be hash-first.** ScreenScraper's CRC/MD5/SHA-1 lookup is the
   most reliable identification method and should be the primary strategy, with filename
   parsing as a fallback.

8. **The format landscape is messy but finite.** A finite set of format-specific parsers
   (iNES header, SNES header detection, N64 byte order, CHD, M3U, etc.) covers the vast
   majority of cases.
