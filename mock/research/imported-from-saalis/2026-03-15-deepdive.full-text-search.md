# Deep Dive: Full-Text Search for a Game Library Manager

**Date:** 2026-03-15
**Scope:** Search infrastructure for saalis -- indexing, querying, and ranking game metadata
**Context:** SQLite-backed single binary on Batocera/loisto ARM appliances. 10k-100k entity records (game titles, descriptions, tags, developer/publisher names). Content-type agnostic model means the same search must eventually cover movies, books, and music. Web UI served via axum + HTMX with instant search / autocomplete UX.

## 1. FTS5 Fundamentals

FTS5 (Full-Text Search 5) is SQLite's built-in full-text search extension, shipping with every modern SQLite build since 3.9.0. It creates virtual tables backed by an inverted index -- a mapping from tokens (words) to the documents and positions where they appear.

### Virtual table creation

```sql
CREATE VIRTUAL TABLE games_fts USING fts5(
    title,
    description,
    developer,
    tags,
    tokenize = "unicode61 remove_diacritics 2"
);
```

Each named column becomes a searchable field. The virtual table stores the inverted index, positional data, and (by default) a copy of the original content.

### Content vs contentless vs external content tables

FTS5 offers three storage modes:

| Mode | Syntax | Stores original text | Use case |
|---|---|---|---|
| **Regular** | (default) | Yes -- full copy | Standalone FTS table |
| **Contentless** | `content=''` | No | Index-only; cannot return column values or use `highlight()`/`snippet()` |
| **External content** | `content='source_table'` | No -- reads from source on demand | Index alongside existing tables; avoids data duplication |

For saalis, external content tables are the clear choice: our metadata already lives in the entities/metadata tables, and we do not want to double-store every title and description.

### Querying

Three equivalent syntaxes:

```sql
SELECT * FROM games_fts WHERE games_fts MATCH 'mario kart';
SELECT * FROM games_fts WHERE games_fts = 'mario kart';
SELECT * FROM games_fts('mario kart');
```

FTS5 query syntax supports AND, OR, NOT, phrase queries (`"super mario"`), column filters (`title:mario`), and prefix queries (`mari*`).

## 2. Tokenizer Configuration for Game Names

Game titles are a hostile environment for naive tokenizers. Consider: "The Legend of Zelda: Twilight Princess HD", "FINAL FANTASY VII REMAKE", "Sid Meier's Civilization VI", "Pokémon Legends: Arceus", "NieR:Automata".

### The unicode61 tokenizer

This is FTS5's default and most capable built-in tokenizer. It applies Unicode 6.1 character classification rules to split text into tokens, with these configurable options:

| Option | Values | Default | Effect |
|---|---|---|---|
| `remove_diacritics` | 0, 1, 2 | 1 | 0 = keep accents; 1 = remove from single-codepoint chars; **2 = remove all** |
| `tokenchars` | string | (none) | Characters to treat as part of tokens rather than separators |
| `separators` | string | (none) | Characters to force-treat as separators |
| `categories` | space-separated | `"L* N* Co"` | Unicode general categories considered token characters |

For game names, the recommended configuration:

```sql
tokenize = "unicode61 remove_diacritics 2 tokenchars '-'"
```

- **`remove_diacritics 2`**: "Pokemon" matches "Pokemon" -- critical since users rarely type accented characters. Mode 2 removes diacritics from all characters including multi-codepoint sequences, unlike mode 1 which skips some edge cases.
- **`tokenchars '-'`**: Keeps hyphens as token characters so "Mega-Man" tokenises as `mega-man` rather than two separate tokens `mega` and `man`. This preserves phrase integrity while still allowing a search for "mega man" to match via separate token hits.
- **Case folding**: Applied automatically by unicode61. "FINAL FANTASY" and "final fantasy" produce identical tokens.

### Roman numeral normalisation

FTS5 has no built-in Roman numeral handling. "Civilization VI" will not match a search for "Civilization 6". This requires application-level preprocessing:

**Strategy: Index-time expansion.** When inserting into the FTS index, detect Roman numerals in titles and store both forms:
- Original: "Final Fantasy VII Remake"
- Normalised: "Final Fantasy 7 Remake"

This can be done by inserting both the original and normalised title into separate FTS columns, or by concatenating them (`"Final Fantasy VII 7 Remake"`). The normalisation function is straightforward -- match `\b(I{1,3}|IV|VI{0,3}|IX|X{0,3}L?C?)\b` and convert. The common range for game sequels is I-XX (1-20), covering virtually all cases.

### Subtitle and colon handling

Titles like "The Legend of Zelda: Breath of the Wild" contain colons that unicode61 treats as separators by default. This is correct behaviour -- it allows searching for just "breath of the wild". The colon is naturally a separator and should remain one.

## 3. FTS5 Ranking: BM25 and Custom Functions

### Built-in BM25

FTS5's `bm25()` function implements the Okapi BM25 algorithm. It returns negative scores (more negative = more relevant) so that `ORDER BY rank` naturally sorts best-first with ascending order.

```sql
SELECT *, rank FROM games_fts WHERE games_fts MATCH 'mario'
ORDER BY rank;
```

### Column weight boosting

The critical feature for game search: `bm25()` accepts per-column weight arguments. Weights are positional, matching the column order in the virtual table definition.

```sql
-- title matches worth 10x, description 1x, developer 5x, tags 3x
SELECT * FROM games_fts WHERE games_fts MATCH ?
ORDER BY bm25(games_fts, 10.0, 1.0, 5.0, 3.0);
```

This ensures that a search for "mario" ranks "Super Mario Bros." (title match) above a game whose description mentions Mario in passing. For a game library, title matches should overwhelmingly dominate.

### Persistent rank configuration

Rather than specifying weights on every query, set a default:

```sql
INSERT INTO games_fts(games_fts, rank) VALUES('rank', 'bm25(10.0, 1.0, 5.0, 3.0)');
```

Now `ORDER BY rank` uses these weights automatically, and the query planner can optimise more aggressively than with an explicit function call.

### Custom auxiliary functions

FTS5 supports registering custom ranking functions via C API (`fts5_api.xCreateFunction`). In Rust, this would require unsafe FFI through `rusqlite::ffi`. Potential use: boost exact title matches, penalise very short descriptions, or factor in a game's popularity score. For our scale, BM25 with column weights is likely sufficient without custom functions.

## 4. Prefix Queries for Autocomplete

Prefix queries are the backbone of instant search. When the user types "super mar", the query `super mar*` should match "Super Mario Bros.", "Super Mario 64", "Super Mario World", etc.

### Syntax

```sql
SELECT highlight(games_fts, 0, '<b>', '</b>') AS title
FROM games_fts
WHERE games_fts MATCH 'super mar*'
ORDER BY rank
LIMIT 10;
```

The `*` wildcard matches any token starting with the prefix. It works at the token level: `mar*` matches "mario", "marble", "mars". Multiple terms are implicitly ANDed: `super mar*` requires both "super" (exact) AND a token starting with "mar".

### Prefix indexes

By default, prefix queries require a range scan of the term dictionary. For fast autocomplete, configure prefix indexes at table creation:

```sql
CREATE VIRTUAL TABLE games_fts USING fts5(
    title, description, developer, tags,
    tokenize = "unicode61 remove_diacritics 2 tokenchars '-'",
    prefix = '2 3 4'
);
```

This pre-builds index entries for all 2-, 3-, and 4-character prefixes. The cost is additional index size (roughly 2-3x for the prefix portion), but for 10k-100k game titles with short-to-medium text, total FTS index size remains under 50 MB even with prefix indexes. The performance gain is dramatic: prefix queries drop from milliseconds-with-scan to sub-millisecond lookups.

**Prefix length selection:** `prefix='2 3 4'` covers the common autocomplete case where the user has typed 2-4 characters. Longer prefixes fall back to range scans, but by that point the scan range is narrow and fast. Single-character prefix indexes (`prefix='1'`) are wasteful -- they match too broadly and the index overhead is large.

### Autocomplete query pattern

For a responsive search box:

```sql
-- User has typed "zel"
SELECT highlight(games_fts, 0, '<mark>', '</mark>') AS title,
       snippet(games_fts, 1, '<mark>', '</mark>', '...', 30) AS description_snippet
FROM games_fts
WHERE games_fts MATCH 'zel*'
ORDER BY rank
LIMIT 8;
```

The `highlight()` function wraps matched terms in the title. The `snippet()` function extracts the most relevant fragment from the description, limited to 30 tokens, with `...` indicating truncation.

## 5. External Content FTS5 Tables

Since saalis already stores game metadata in structured tables, external content tables let FTS5 index that data without duplication.

### Setup

```sql
-- Assume our metadata is stored with entity_id as the key
CREATE TABLE game_metadata (
    entity_id INTEGER PRIMARY KEY,
    title TEXT NOT NULL,
    description TEXT,
    developer TEXT,
    tags TEXT
);

CREATE VIRTUAL TABLE games_fts USING fts5(
    title, description, developer, tags,
    content = 'game_metadata',
    content_rowid = 'entity_id',
    tokenize = "unicode61 remove_diacritics 2 tokenchars '-'",
    prefix = '2 3 4'
);
```

The `content` option names the source table. The `content_rowid` option maps the FTS rowid to the source table's primary key. When FTS5 needs to return column values (for `highlight()`, `snippet()`, or direct column reads), it queries: `SELECT entity_id, title, description, developer, tags FROM game_metadata WHERE entity_id = ?`.

### Synchronisation via triggers

FTS5 does not automatically track changes to the content table. You must install triggers:

```sql
CREATE TRIGGER game_meta_ai AFTER INSERT ON game_metadata BEGIN
    INSERT INTO games_fts(rowid, title, description, developer, tags)
    VALUES (new.entity_id, new.title, new.description, new.developer, new.tags);
END;

CREATE TRIGGER game_meta_ad AFTER DELETE ON game_metadata BEGIN
    INSERT INTO games_fts(games_fts, rowid, title, description, developer, tags)
    VALUES ('delete', old.entity_id, old.title, old.description, old.developer, old.tags);
END;

CREATE TRIGGER game_meta_au AFTER UPDATE ON game_metadata BEGIN
    INSERT INTO games_fts(games_fts, rowid, title, description, developer, tags)
    VALUES ('delete', old.entity_id, old.title, old.description, old.developer, old.tags);
    INSERT INTO games_fts(rowid, title, description, developer, tags)
    VALUES (new.entity_id, new.title, new.description, new.developer, new.tags);
END;
```

The delete syntax is non-obvious: you insert into the special shadow column (same name as the table) with the value `'delete'`, along with the old row values. FTS5 uses these old values to locate and remove the correct index entries.

### Rebuild command

If triggers are missed or the index becomes inconsistent:

```sql
INSERT INTO games_fts(games_fts) VALUES('rebuild');
```

This re-reads the entire content table and reconstructs the index. For 100k records with moderate text, this takes 1-5 seconds on ARM hardware.

### Integration with saalis entity model

In saalis's descriptor-based schema where everything is an entity with metadata columns, the content table for FTS would likely be a view joining entity base fields with their metadata. The `content_rowid` would map to `EntityId`. This fits cleanly: the FTS index is just another derived view of the entity data, kept in sync by the same write path that updates metadata.

## 6. FTS5 with rusqlite

### Feature flags

rusqlite provides FTS5 support when using the `bundled` feature flag, which compiles SQLite from source with FTS5 enabled. The current version (0.38.0) bundles SQLite 3.51.1.

```toml
[dependencies.rusqlite]
version = "0.38.0"
features = ["bundled"]
```

FTS5 is enabled by default in bundled builds. If linking against a system SQLite, you must ensure it was compiled with `-DSQLITE_ENABLE_FTS5`.

### API usage

FTS5 tables are transparent to rusqlite -- they behave as normal tables for queries:

```rust
// Creating the FTS table
conn.execute_batch("
    CREATE VIRTUAL TABLE IF NOT EXISTS games_fts USING fts5(
        title, description, developer, tags,
        content = 'game_metadata',
        content_rowid = 'entity_id',
        tokenize = \"unicode61 remove_diacritics 2\",
        prefix = '2 3 4'
    );
")?;

// Querying with parameterised input
let query = format!("{}*", user_input.replace('"', "\"\""));
let mut stmt = conn.prepare(
    "SELECT highlight(games_fts, 0, '<b>', '</b>') AS title,
            snippet(games_fts, 1, '<b>', '</b>', '...', 30) AS desc_snippet,
            rank
     FROM games_fts
     WHERE games_fts MATCH ?1
     ORDER BY rank
     LIMIT 10"
)?;

let results: Vec<SearchResult> = stmt.query_map(params![query], |row| {
    Ok(SearchResult {
        title: row.get(0)?,
        description_snippet: row.get(1)?,
        score: row.get(2)?,
    })
})?.collect::<Result<_, _>>()?;
```

### Query sanitisation

User input must be sanitised before use in MATCH expressions. FTS5 query syntax has special characters (`*`, `"`, `AND`, `OR`, `NOT`, `NEAR`, `(`, `)`, `:`). A safe approach: treat the entire user input as a phrase by wrapping in quotes, or escape special characters. For autocomplete, append `*` to the last token only.

```rust
fn sanitise_fts_query(input: &str) -> String {
    let tokens: Vec<&str> = input.split_whitespace().collect();
    if tokens.is_empty() {
        return String::new();
    }
    let mut parts: Vec<String> = tokens.iter()
        .map(|t| format!("\"{}\"", t.replace('"', "\"\"")))
        .collect();
    // Make last token a prefix query for autocomplete
    if let Some(last) = parts.last_mut() {
        last.push('*');
    }
    parts.join(" ")
}
```

### Performance characteristics

On an RPi4-class ARM SBC with 100k game records:
- Index build (full rebuild): 2-5 seconds
- Simple MATCH query: < 1 ms
- Prefix query with prefix indexes: < 1 ms
- Prefix query without prefix indexes: 1-5 ms
- BM25 ranking with column weights: < 1 ms additional overhead
- `highlight()` / `snippet()` processing: negligible

FTS5 adds approximately 1-2x the size of the indexed text to the database file. For 100k games with average 500 bytes of indexed text per game, expect 50-100 MB of FTS overhead.

## 7. Tantivy: Rust-Native Full-Text Search

Tantivy is a full-text search engine library written in Rust, inspired by Apache Lucene. It provides capabilities well beyond FTS5, at the cost of additional complexity and a separate data store.

### Architecture

Tantivy follows Lucene's segment-based architecture:

- **Index**: Top-level container, a collection of segments stored in a `Directory` abstraction (filesystem, RAM, or custom).
- **Segments**: Immutable, self-contained units containing an inverted index, positional data, stored fields, and fast fields (columnar numeric data). New documents go into new segments; background merge consolidates them.
- **Term dictionary**: Stored as a Finite State Transducer (FST), providing O(1)-like lookups with extreme compression. A 10M-term dictionary might occupy 10 MB.
- **Postings**: Compressed using SIMD-friendly integer compression (bitpacking, PFor-Delta). Memory-mapped for zero-copy access.

Design principles: search is O(1) in memory (only the FST and skip data are resident; postings are streamed), and indexing is O(1) in memory (bounded by a configurable buffer, default 100 MB, flushed to segments).

### Capabilities relevant to saalis

| Feature | Tantivy | FTS5 |
|---|---|---|
| BM25 ranking | Yes | Yes |
| Fuzzy queries (Levenshtein) | Yes, built-in `FuzzyTermQuery` | No (requires trigram workaround) |
| Phrase queries | Yes | Yes |
| Prefix queries | Yes | Yes |
| Boolean queries | Yes, full algebra | Yes (AND, OR, NOT) |
| Faceted search | Yes, hierarchical | No |
| Range queries | Yes (numeric, date) | No |
| Regex queries | Yes | No |
| Custom tokenizers | Yes, trait-based | Yes, C API |
| Highlighting | Via `tantivy-highlight` | Built-in `highlight()` / `snippet()` |
| Multi-threaded indexing | Yes | No (SQLite single-writer) |
| Incremental indexing | Yes (segment merges) | Yes (insert/delete) |
| In-memory index | Yes (`RamDirectory`) | No (always on disk) |

### Performance at saalis scale

For 10k-100k documents with short text fields (game titles, descriptions):

- **Index build**: Sub-second for 100k documents. Tantivy can index millions of documents per second for short text.
- **Query latency**: Sub-millisecond for simple queries. Fuzzy queries with edit distance 1-2 add ~15% overhead.
- **Memory**: The FST for 100k unique terms fits in ~1 MB. Postings are memory-mapped; resident set depends on OS page cache. Minimal overhead for small datasets.
- **Disk**: Tantivy's compressed index is typically smaller than FTS5's for equivalent data, due to more aggressive compression.
- **Startup**: Under 10 ms to open an index.

### CJK support

Tantivy has robust CJK tokenization through third-party integrations:
- **Japanese**: lindera-tantivy (morphological analysis with IPADIC/UniDic dictionaries), Vaporetto, tantivy-tokenizer-tiny-segmenter
- **Chinese**: tantivy-jieba, cang-jie
- **Korean**: lindera with ko-dic

### Trade-offs vs FTS5 for saalis

**In favour of Tantivy:**
- Native fuzzy queries (Levenshtein automaton, not brute-force)
- Faceted search for filtering by genre, platform, year
- Multi-threaded indexing for large imports
- Richer query algebra (regex, ranges)
- Rust-native API with no FFI boundary

**In favour of FTS5:**
- Zero additional dependency -- ships with SQLite
- Transactional consistency with the rest of the database (single WAL)
- External content tables avoid data duplication
- Simpler operational model -- no separate index directory, no segment merges
- `highlight()` and `snippet()` are built-in
- Triggers keep the index in sync automatically
- Smaller binary size impact

**Verdict for saalis:** FTS5 is the pragmatic default. The dataset is small (100k titles is trivial for FTS5), the index lives in the same SQLite database as everything else, and the operational simplicity on an embedded appliance is compelling. Tantivy becomes worth considering if we need fuzzy queries, faceted search, or the dataset grows to millions of records. It could be introduced later as an optional enhancement without replacing FTS5 -- they serve different layers.

## 8. MeiliSearch: Embedded Mode Viability

MeiliSearch is a search engine written in Rust, designed for instant, typo-tolerant search with < 50 ms response times. It runs as an HTTP server, but its core engine is the `milli` crate.

### Typo tolerance

MeiliSearch's headline feature. It uses a prefix-based distance algorithm that handles:
- Character transpositions: "mraio" matches "mario"
- Missing characters: "maio" matches "mario"
- Extra characters: "marioo" matches "mario"
- Wrong characters: "marlo" matches "mario"

Tolerance scales with word length: 1 typo allowed for 5-8 character words, 2 typos for 9+ characters. This is configurable per-index and can be disabled for specific attributes or words.

### Faceted search

Built-in faceting with instant filter counts. Define filterable attributes, then query with filter expressions: `genre = "RPG" AND platform = "Switch"`. Facet distributions update per-query.

### The milli crate: embedded use

The `milli` crate is MeiliSearch's core indexing and search engine. It is not published on crates.io but can be used as a git dependency. It uses LMDB (via heed) as its storage backend.

**Challenges for saalis:**
- **Not a stable public API.** milli's API is internal to MeiliSearch and changes without semver guarantees.
- **LMDB dependency.** Introduces a second storage engine alongside SQLite. On ARM, LMDB works well but adds memory-mapping complexity.
- **Binary size.** milli pulls in a substantial dependency tree.
- **No crates.io release.** Depends on a git URL, making builds less reproducible.

### Assessment

MeiliSearch's typo tolerance and relevance tuning are best-in-class, but embedding milli introduces significant architectural complexity. For saalis's scale, FTS5 with application-level fuzzy matching provides 80% of the benefit at 10% of the complexity. If typo tolerance becomes critical, a lighter approach (Levenshtein on the top-N FTS results, or Tantivy's `FuzzyTermQuery`) is preferable to embedding milli.

## 9. Fuzzy Matching for Game Names

Game names present unique fuzzy matching challenges: abbreviations ("GTA" for "Grand Theft Auto"), alternate regional titles ("Biohazard" vs "Resident Evil"), transliterations, Roman numerals, and user typos.

### Algorithms and their fitness

| Algorithm | Strength | Weakness | Use case |
|---|---|---|---|
| **Levenshtein distance** | Edit-distance based, intuitive | Slow for large vocabularies (O(n*m)) | Post-filter on short candidate lists |
| **Damerau-Levenshtein** | Handles transpositions ("teh" -> "the") | Same cost as Levenshtein | Better for typo correction |
| **Jaro-Winkler** | Prefix-weighted similarity, fast | Less effective for long strings with mid-word errors | Ranking candidates by similarity |
| **Trigram similarity** | Substring overlap, index-friendly | Minimum 3-char overlap required | Pre-filter in SQL/FTS |
| **Soundex / Metaphone** | Phonetic matching | English-centric, many false positives | Limited value for game names |

### Rust crates

- **`strsim`** (rapidfuzz/strsim-rs): Implements Hamming, Levenshtein, OSA, Damerau-Levenshtein, Jaro, Jaro-Winkler, and Sorensen-Dice. Normalised variants return 0.0-1.0 similarity scores. Well-maintained, zero dependencies.
- **`fuzzt`**: Built on strsim, adds top-N matching from a collection and Gestalt pattern matching. Useful for "did you mean?" suggestions.
- **`rapidfuzz`**: Rust port of the Python rapidfuzz library. Optimised for batch comparisons.

### Practical strategy for saalis

A layered approach:

1. **FTS5 prefix/token matching** (primary): Handles the 90% case -- user types partial title, gets instant results.
2. **FTS5 trigram tokenizer** (secondary index): A second FTS table with `tokenize="trigram"` enables substring matching. "ario" matches "Mario", "raft" matches "Minecraft". Useful for mid-word searches but not true fuzzy matching.
3. **Levenshtein post-filter** (fallback): When FTS returns few or no results, compute edit distance between the query and the top-100 game titles (sorted by popularity). The `strsim` crate makes this trivial. For 100k titles, comparing against a pre-sorted popularity list of the top 1000 takes < 1 ms.
4. **Alias/alternate title table**: Store known aliases ("GTA" -> "Grand Theft Auto", "Biohazard" -> "Resident Evil", "FF7" -> "Final Fantasy VII") in a lookup table. Index these in FTS alongside primary titles.

## 10. Search UX Patterns

### Instant search with debouncing

For the HTMX-based UI, the search flow is:

1. User types in search box.
2. Client-side debounce (200-300 ms) prevents firing on every keystroke.
3. After debounce, `hx-get="/search?q=..."` fires.
4. Server runs FTS5 prefix query, returns HTML fragment with results.
5. Fragment is swapped into the results container via `hx-target`.

HTMX provides `hx-trigger="keyup changed delay:250ms"` which handles debouncing natively. No JavaScript required.

**Debounce timing:** 200-300 ms is optimal. Below 200 ms, too many requests fire for fast typists. Above 400 ms, the UI feels sluggish. For our ARM appliance, 250 ms is a safe default -- it gives the server enough time to complete the query (< 1 ms for FTS5) and render the response (< 5 ms for a small HTML template) before the next keystroke arrives.

### Highlighting matched terms

FTS5's `highlight()` function returns the column value with matched tokens wrapped in user-specified markers:

```sql
SELECT highlight(games_fts, 0, '<mark>', '</mark>') AS title
FROM games_fts WHERE games_fts MATCH 'mario';
-- Returns: "Super <mark>Mario</mark> Bros."
```

This HTML can be injected directly into the HTMX response. The `<mark>` element has native browser styling (yellow highlight) and is semantically correct.

For description previews, `snippet()` extracts the most relevant fragment:

```sql
SELECT snippet(games_fts, 1, '<mark>', '</mark>', '...', 25) AS preview
FROM games_fts WHERE games_fts MATCH 'plumber';
-- Returns: "...iconic <mark>plumber</mark> on a quest to rescue..."
```

### "Did you mean?" suggestions

When a search returns zero results:

1. Run the query against the trigram FTS index (catches substring matches the primary index missed).
2. If still zero results, compute Jaro-Winkler similarity between the query and the top-N most popular titles.
3. Present the closest matches as "Did you mean: Super Mario Bros.?"

```rust
use strsim::jaro_winkler;

fn suggest_alternatives(query: &str, titles: &[&str], threshold: f64) -> Vec<&str> {
    let mut scored: Vec<_> = titles.iter()
        .map(|t| (*t, jaro_winkler(query, &t.to_lowercase())))
        .filter(|(_, score)| *score > threshold)
        .collect();
    scored.sort_by(|a, b| b.1.partial_cmp(&a.1).unwrap());
    scored.into_iter().take(3).map(|(t, _)| t).collect()
}
```

A Jaro-Winkler threshold of 0.75-0.80 produces reasonable suggestions without too many false positives.

### Empty state and progressive disclosure

- **Empty query**: Show recently played, popular, or newly added games.
- **Single character**: Do not search yet (too broad). Show a hint: "Keep typing..."
- **Two characters**: Begin prefix search if prefix='2' index exists.
- **Three+ characters**: Full prefix search with ranking.
- **No results**: Show "did you mean?" suggestions and a link to browse by genre/platform.

## 11. Multi-Language Search: CJK Tokenization

### The problem

Chinese, Japanese, and Korean text does not use whitespace to separate words. The sentence "ゼルダの伝説" (The Legend of Zelda) contains no spaces. FTS5's unicode61 tokenizer treats it as a single enormous token, making it unsearchable except by exact match.

Japanese adds further complexity with three scripts (hiragana, katakana, kanji) and mixed-script titles: "ファイナルファンタジーVII" (Final Fantasy VII) mixes katakana with Roman numerals.

### FTS5 solutions

**Trigram tokenizer**: The simplest CJK-compatible approach. Every 3-character sequence becomes a token. "ゼルダの伝説" produces tokens: `ゼルダ`, `ルダの`, `ダの伝`, `の伝説`. A search for "ゼルダ" matches because that trigram exists in the index.

```sql
CREATE VIRTUAL TABLE games_fts_cjk USING fts5(
    title, description,
    tokenize = "trigram"
);
```

Trade-offs: trigram indexes are larger (3x or more) and produce more false positives than morphological tokenizers. But they require zero external dictionaries and work across all CJK languages.

**ICU tokenizer**: SQLite can be compiled with ICU (International Components for Unicode) support, providing dictionary-based word segmentation. This produces better tokens but requires bundling ICU libraries (10+ MB), which may be prohibitive on an embedded appliance.

**lindera-sqlite**: A Rust-based solution providing a custom FTS5 tokenizer with morphological analysis for Japanese (IPADIC dictionary), Korean, and Chinese. It compiles as a loadable SQLite extension. For saalis's Rust/SQLite stack, this is the most natural fit if trigram matching proves insufficient.

### Tantivy solutions

Tantivy has first-class CJK support through the lindera ecosystem:

- **lindera-tantivy**: Provides a `LinderaTokenizer` that plugs into Tantivy's tokenizer pipeline. Supports IPADIC (Japanese), UniDic (Japanese), ko-dic (Korean), and CC-CEDICT (Chinese) dictionaries. Dictionaries are embedded via feature flags, adding 20-50 MB to the binary depending on the language.
- **tantivy-jieba**: Chinese segmentation via the jieba algorithm (statistical, no dictionary required; ~5 MB model).

### Practical recommendation for saalis

**Phase 1 (MVP):** Use FTS5 with unicode61 for Latin-script titles and a parallel FTS5 table with trigram tokenizer for CJK titles. Detect script at index time and route to the appropriate table. Query both tables and merge results. This requires zero external dependencies.

**Phase 2 (if CJK quality matters):** Integrate lindera-sqlite as a loadable extension for proper Japanese/Chinese morphological tokenisation. This improves precision (fewer false positives) and recall (finds morphological variants) at the cost of dictionary size.

**Phase 3 (if Tantivy is adopted):** Use lindera-tantivy for unified multilingual search. One index, one query path, proper tokenisation for all languages.

### Cross-language search

Game libraries often need to search across languages: a user might search for "zelda" and expect to find "ゼルダの伝説". This requires an alias/translation table mapping titles across languages. The IGDB and ScreenScraper APIs provide localised titles which can populate this table. Index all known translations for each game, linked by entity ID.

## Summary: Decision Matrix

| Criterion | FTS5 (SQLite) | Tantivy | milli (MeiliSearch) |
|---|---|---|---|
| **Dependency weight** | Zero (bundled with SQLite) | ~5 MB binary increase | ~15 MB + LMDB |
| **Data consistency** | Same WAL as all other data | Separate index directory | Separate LMDB store |
| **Fuzzy search** | Trigram only (no Levenshtein) | Built-in FuzzyTermQuery | Built-in typo tolerance |
| **Prefix autocomplete** | Excellent with prefix indexes | Excellent | Excellent |
| **BM25 ranking** | Built-in with column weights | Built-in | Custom relevance rules |
| **Faceted search** | No | Yes | Yes |
| **CJK support** | Trigram or loadable extension | lindera-tantivy | Built-in (charabia) |
| **Highlighting** | Built-in highlight()/snippet() | Via tantivy-highlight crate | Built-in |
| **API stability** | Extremely stable (SQLite) | Stable (semver) | Unstable (internal API) |
| **ARM embedded fit** | Ideal | Good | Marginal |
| **Operational complexity** | Minimal | Moderate (segment merges) | High (LMDB tuning) |

**Recommended approach for saalis:** Start with FTS5 using external content tables, unicode61 tokenizer with `remove_diacritics 2`, prefix indexes, and BM25 with column weights. Add application-level fuzzy matching via `strsim` for "did you mean?" suggestions. Revisit Tantivy only if faceted search or native fuzzy queries become requirements that justify the additional operational complexity.
