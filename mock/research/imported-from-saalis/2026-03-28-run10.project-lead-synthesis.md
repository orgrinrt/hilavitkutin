## Section 1: Resolved Decisions

### Tension 1: Document coupling cycles — are the back-edges a problem?

**Decision: the back-edges are real coupling but not structural defects. They become defects only if left implicit.**

The DepAnalyst is correct that the protocol-mesh MetabolicState coupling and the crypto-mesh enrollment deferral are implicit back-edges requiring resolution. The Savant is correct that the forward-reference pattern is a strength when the interfaces are opaque. Both are right about different aspects.

The resolution is the one all reviewers converged on: replace the back-references with abstract contracts. MetabolicState transition predicates get an abstract governor interface in protocol; Conductor in mesh implements it. Crypto's enrollment deferral gets an explicit cross-reference stating mesh owns enrollment ceremony; crypto provides the primitives. The back-edges become documented forward-references through opaque interfaces, not implicit couplings through body text.

The HeartbeatExtension split is a grain boundary requiring its own defined concept (Savant's position, endorsed by all reviewers by Stage 3). It is both a design strength (clean interface segregation, per DepAnalyst) and a load-bearing surface needing formalization (per Savant). These are not contradictory; they describe different properties of the same boundary.

Rejected: the position that back-edges are inherently acceptable as-is. They must be made explicit or resolved.

### Tension 2: Severity triage — which scale?

**Decision: use a merged two-axis scale. Axis 1: interoperability impact (SpecAuditor). Axis 2: implementation blocking (GapHunter). Report both.**

The three scales (SpecAuditor's interop framing, GapHunter's implementer-halts framing, Savant's emergent-risk framing) are not competing answers to the same question. They answer different questions. A gap can be non-blocking for implementation (an implementer can write code around it) but critical for interoperability (two implementations diverge). The DomainSeparatorRegistry is the proof: LOW for implementation blocking, MEDIUM for interop, potentially HIGH for emergent risk. Collapsing these into one number loses information.

Going forward, every finding carries two ratings:
- **IMPL** severity: does this halt an implementer? (BLOCKING / NEEDED / NICE)
- **INTEROP** severity: does this cause two correct implementations to diverge? (CRITICAL / HIGH / MEDIUM / LOW)

The Savant's emergent-risk framing is valuable for feedback loops and interaction effects but is not a general triage axis. It applies to a specific class of findings (closed loops, cliff effects) and should be noted as a flag, not a third axis.

Rejected: the Savant's claim that emergent risk should override interop severity for DomainSeparatorRegistry. The emergent risk is real but the probability in v1 is zero; it is a v2 process concern.

### Tension 3: MetabolicState predicates — where do they live?

**Decision: transition predicates live in verkko-protocol as an abstract governor contract. Conductor in mesh implements the contract.**

The SpecAuditor's "two acceptable fixes" framing is correct, but only one of them resolves the soft cycle. Putting predicates in protocol makes the state machine self-contained but couples protocol to mesh-specific concepts (health scores, convergence state). Putting an abstract governor contract in protocol and having Conductor implement it preserves the layering: protocol defines what transitions exist and what the governor interface looks like; mesh's Conductor provides the concrete trigger logic.

This is the position all four reviewers converged on by Stage 3. The abstract governor contract is the mechanism. The "see verkko-mesh: Conductor" cross-reference in protocol's invariants is replaced with the governor interface definition. The soft cycle is resolved.

Rejected: putting the predicates directly in mesh without an interface in protocol. This would leave protocol's state machine incomplete; the transitions would have no specified triggers.

### Tension 4: matter -> protocol dependency — declare it or restructure?

**Decision: both. Declare the dependency AND specify the bond.**

This is not an either/or. The DepAnalyst is correct that the declaration fix is a one-line change to matter's Dependencies section. The Savant is correct that the declaration alone does not specify how MatterTransfer maps to Stream (bulk tier) and LEDBAT pacing. Both are needed.

The dependency declaration places matter at Level 4 in the topological sort (after protocol at Level 1), which is where it already sits. The topological ordering does not change. The specification work is: MatterTransfer uses protocol's Stream abstraction with bulk tier priority, LEDBAT pacing is the congestion control mechanism for matter transfers, and the BandwidthPolicy body must specify the LEDBAT variant.

Rejected: treating this as only a "paperwork fix." The missing specification of how matter uses protocol services is the actual gap; the missing declaration is a symptom.

The GapHunter is right that seat's 10 BLOCKING gaps and matter's body voids halt implementation. The Savant is right that these are characterized vacancies, not emergencies for the design review. ### Disagreement 1: DomainSeparatorRegistry — MEDIUM, LOW, or design defect?

**Decision: MEDIUM severity. The fix is both a type-level constraint (DomainSeparator enum) AND a process guard (lint). Not a pure design defect, not a pure process concern.**

The Savant's proposed fix is correct and should be adopted: a `DomainSeparator` repr(C) enum as the sole API surface for domain-separated hashing, with raw BLAKE3 not directly accessible from protocol crates. This is a type-level constraint that makes unregistered usage unrepresentable, consistent with the project's architectural rules (repr(C) enums for closed sets, no dyn, type safety over runtime checks).

The SpecAuditor is right that this is not analogous to TLS 1.0's padding oracle (developer error vs attacker input). The GapHunter is wrong that the wrapper requires runtime registry lookups; the Savant's enum approach is compile-time, not runtime. The Savant is right about the mechanism but overstates the severity; in v1 with a single implementation team, the risk is effectively zero. The process concern escalates to real risk only when extension crates or multiple teams exist.

The fix: implement `DomainSeparator` enum (Savant's design). Add a lint catching raw BLAKE3 usage (Savant's S2). Rate as MEDIUM: zero risk in v1, critical if violated in v2+, and the type-level fix is low-cost.

Rejected: the GapHunter's position that a type-level approach is impractical. The Savant demonstrated a concrete, practical mechanism.
Rejected: the Savant's framing as a design defect on par with TLS 1.0 padding oracle. The failure mode requires a developer to bypass the API, not an attacker to exploit normal operation.

### Disagreement 2: Scar penalty function — smooth or cliff?

**Decision: keep the cliff at 10.0 weighted units. The tiered severity system is sufficient to prevent unfair penalties.**

The Savant's argument is more convincing: the threshold at 10.0 weighted units is an intentional phase boundary between "trusted with some issues" and "demoted due to accumulated concern." The tiered severity system (transient scars at 0.5x weight, concerning at 1.0x, severe at 2.0x) ensures that only genuinely problematic patterns reach the threshold. A household device with transient hardware flaps accumulates Tier-1 scars at 0.5x weight; it needs 20 such scars to reach the threshold, which is a reasonable signal of persistent problems.

The SpecAuditor's smooth function `1 + max(0, weighted - 10)^1.5` eliminates the cliff but creates an ambiguous transition zone where a peer's status is unclear. The cliff provides a clear, predictable boundary that operators and the admin UI can reason about.

Rejected: the smooth penalty function. A clear phase boundary is preferable to a gradient for a household mesh appliance where users and administrators need simple mental models.

### Disagreement 3: Damping coefficients — constants or calibration items?

**Decision: the DAMPED mode is a protocol constant. The coefficients within it (0.3x, 0.1x) are calibration items with specified defaults and recovery bounds.**

The Savant's sharp edge is correct: setting coefficients to 1.0/1.0 must be forbidden because it eliminates damping entirely, re-enabling the convergence-health feedback loop oscillation. The existence of the DAMPED state in the HealthPipeline state machine is a structural invariant.

The SpecAuditor and GapHunter are correct that the specific values (0.3x, 0.1x) are unjustified and the recovery time estimates were wrong by 3x. The correct recovery bounds (29 observations for fast EMA, 299 for slow EMA) should be stated. The coefficients should be calibration items with the constraint that they must be strictly less than 1.0 (i.e., damping must attenuate, never amplify or pass through).

The specification must state:
- DAMPED mode is mandatory (protocol constant).
- Coefficients are calibration items with defaults 0.3 (fast) and 0.1 (slow).
- Valid range: (0.0, 1.0) exclusive. Setting 1.0 is a protocol violation.
- Recovery bounds must be computed from actual coefficients and documented.

Rejected: treating the entire damping mechanism as optional/calibration. The mechanism itself is structural; only the parameters are tunable.

## Section 2: Consensus Findings (No Decision Needed)

These findings had unanimous or near-unanimous agreement across all reviewers. They are facts established by the review, not opinions requiring resolution.

1. **Matter -> protocol dependency is undeclared.** All four reviewers confirmed independently across all rounds. Fix: add verkko-protocol to matter's Dependencies; specify MatterTransfer's mapping to Stream (bulk tier) and LEDBAT pacing.

2. **Wire contracts are required before implementation.** Prose invariants do not constrain byte layout. The Devil's Advocate's two-team interop failure scenario is non-hypothetical. Every reviewer accepted this with no substantive objection. Wire contracts must include: canonical byte layout, test vectors, and canonical generators (pseudocode algorithms).

3. **HeartbeatExtension needs its own defined concept.** Six consumers share an 18-byte layout with no version field. All reviewers agreed this is a grain boundary requiring formalization.

4. **Named convergence fronts replace numbered fronts.** ControlFront, ConsensusFront, ReencryptionFront. Position-dependent references break on insertion. All reviewers agreed.

5. **The convergence-health feedback loop exists and needs damping.** The Savant identified it; the SpecAuditor formalized it as a two-mode state machine (NORMAL/DAMPED); the GapHunter identified the distributed coordination gap (convergence_active heartbeat bit); all accepted the mechanism.

6. **The scar system needs severity tiering.** Three tiers (transient, concerning, severe) with criteria-driven classification. ScarSeverityTier should be a defined concept. All reviewers agreed on the direction; the parameters were debated.

7. **The seat needs a consistency model.** CONSISTENT (normal) vs SNAPSHOT (convergence active). The SpecAuditor named it; the Savant adopted it; the GapHunter endorsed staleness indicators and conflict resolution for deferred writes.

8. **ConvergenceGate should follow the opaque projection principle.** Consumers observe boolean outcomes, not the inputs or mechanisms behind them. The DepAnalyst generalized this to all grain boundary defined concepts. All reviewers endorsed.

9. **All bond strength scores are conditional on wire contracts.** The Savant conceded this explicitly in Stage 3b. The DepAnalyst added the conditionality annotation to all edge scores. The actual interop strength is unknown until wire contracts exist.

10. **The seven-document-to-seven-crate mapping is unlikely to survive.** 67% of pairwise boundaries have implicit coupling. The Devil's Advocate estimates three crates (crypto-protocol, mesh-relay, matter-seat). No reviewer defended the seven-crate outcome. This is noted as an implementation-time decision, not a specification-time action.

11. **The health pipeline's internal dimensions are unspecified.** Whether projections (Savant) or dimensions (Devil's Advocate) are exposed, the internal model must first be specified. The current spec has only `effective_health`, a single scalar. Both approaches require the same specification work for the internal model.

12. **Two priority orderings are needed.** Design priority (grain boundaries first, per Savant) and implementation priority (first bytes on the wire, per GapHunter). Neither subsumes the other. The ops document's build order should reference both.

13. **The seat -> mesh vacancy is intentional.** The DepAnalyst confirmed this; the Savant's seat isolation valve preserves the absence of that edge. The HostPeer mediates without creating a dependency.

14. **The zero-observation bypass for damping is mandatory.** If a peer sends zero heartbeats for 3+ consecutive periods, the damping mask must not apply. Zero heartbeats signals an absent peer, not convergence disruption.

## Section 4: What NOT to Do

1. **Do not restructure the seven documents into three.** The document structure is a specification organization, not a crate boundary commitment. Restructuring creates churn without improving specification quality. The implementation team decides crate boundaries.

2. **Do not add a smooth penalty function to scar scoring.** The cliff at 10.0 weighted units is an intentional phase boundary. The tiered severity system prevents unfair penalties. Smoothing creates ambiguity.

3. **Do not expose health dimensions directly to consumers.** Use opaque projections. The coupling direction matters more than the specification effort.

4. **Do not implement four health projections in v1.** Two projections (availability, throughput) are sufficient. Four creates a 12-parameter tuning surface with no empirical data to calibrate against.

5. **Do not make the DAMPED mode optional or set coefficients to 1.0.** The existence of damping is a protocol constant. Only the coefficient values are calibration items.

6. **Do not add a runtime DomainSeparator registry.** The fix is a compile-time repr(C) enum, not a runtime lookup table. The GapHunter's objection applies to runtime registries, not type-level constraints.

7. **Do not write seat or matter body content before Phase 0 structural fixes.** The defined concepts (HeartbeatExtension, ConvergenceGate, ScarSeverityTier) must exist before the body content that references them can be written coherently.

8. **Do not treat the protobuf message catalog as a Phase 3 gap.** It is a Priority 1 wire contract (Phase 1). The seat body depends on it.

9. **Do not attempt to resolve the seven-crate question in specification.** It is an implementation-time decision that does not affect spec completeness or correctness.

10. **Do not write wire contracts without canonical generators.** Test vectors alone are insufficient. The canonical generator algorithm is what makes the specification complete. Test vectors without generators are examples, not specifications.
