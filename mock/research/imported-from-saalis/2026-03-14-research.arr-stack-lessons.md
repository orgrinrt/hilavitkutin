# Research: The *arr Stack -- Architecture Lessons for Saalis

**Date:** 2026-03-14
**Subject:** Sonarr, Radarr, Lidarr, Readarr, Prowlarr -- architecture evolution, design patterns, and lessons learned
**Purpose:** Inform saalis design by studying a decade of evolution in the most successful open-source content library management ecosystem

---

## 1. Architecture Evolution

### The NzbDrone Era (v1/v2)

Sonarr began life as **NzbDrone**, a C# application targeting .NET Framework 4.0 on Windows and Mono on Linux/macOS. The original database file is still called `nzbdrone.db` -- a vestige of this lineage. The early architecture was monolithic: a single process serving a web UI, running scheduled tasks, and managing downloads. The application used Mono for cross-platform support, which introduced significant performance overhead and platform-specific bugs that plagued the project for years.

The internal project structure still reflects the NzbDrone heritage. The core business logic lives in `NzbDrone.Core`, shared utilities in `NzbDrone.Common`, and the application host in `NzbDrone.Host`. This naming persists even in 2026, demonstrating how deeply embedded early naming decisions become.

### Sonarr v3: Stabilisation

Sonarr v3 upgraded to .NET Framework 4.6.1 (still requiring Mono on non-Windows platforms) and introduced:

- A new React-based UI replacing the original Backbone.js frontend
- Quality profile grouping (allowing multiple qualities to be treated as equivalent)
- Language Profiles as a separate concept from quality
- Remux and WebRip quality tiers

The v3 era was characterised by incremental improvement rather than architectural revolution. The Mono dependency remained, causing ongoing cross-platform pain.

### Sonarr v4: The .NET Core Migration

After roughly a year of development and nearly 1000 commits from over 100 contributors, Sonarr v4 shipped with the most significant architectural change in the project's history: **migration from .NET Framework + Mono to .NET 6**. This:

- Eliminated the Mono dependency entirely, resolving years of platform-specific bugs
- Brought substantial performance improvements from the modern .NET runtime
- Replaced the v2 API entirely (breaking backward compatibility)
- Introduced **Custom Formats** to replace the simpler "Preferred Words" system
- Replaced MediaInfo with ffprobe for media scanning (eliminating crash-inducing native library issues)

The database schema changed enough that rollback from v4 to v3 is not possible -- a one-way migration.

**Currently in development:** Sonarr v5 introduces a new `Sonarr.Api.V5` endpoint layer alongside the existing V3 API, signaling continued evolution.

### The Fork Ecosystem

**Radarr** forked from Sonarr to handle movies (single items rather than series/episodes). **Lidarr** forked for music (artists/albums/tracks via MusicBrainz). **Readarr** forked for books. Each fork is essentially a "gigantic diff against Sonarr's repo" -- they share the NzbDrone.Core infrastructure but diverge in content-type-specific logic.

This fork-based development model has significant consequences:

- Every upstream Sonarr commit requires manual conflict resolution and cherry-picking into every fork
- The workflow is "much closer to a centralized VCS like Perforce than the normal Git workflow"
- The complexity "scares off potential new contributors"
- Bug fixes in shared infrastructure must be independently applied to each project

A [proposal to extract NzbDrone as a standalone shared package](https://github.com/Sonarr/Sonarr/issues/7528) was discussed but not committed to. The proposal envisioned content-specific namespaces (`NzbDrone.TV`, `NzbDrone.Movie`, `NzbDrone.Music`, `NzbDrone.Book`) with the shared framework as a Git submodule. The maintainers acknowledged the value but declined, citing the effort required.

### Lesson for Saalis

Saalis's "everything is an entity" model with `EntityType` as a trait avoids the *arr stack's fundamental mistake: building content-type assumptions into the core framework. The *arr approach -- forking the entire application per content type -- creates an O(N) maintenance burden. Saalis's single-binary, trait-based approach where games, movies, books are all entities with metadata is architecturally superior, but we must ensure the `EntityType` abstraction is genuinely flexible enough that we never face pressure to fork.

## 2. Data Model

### Project Structure

Sonarr's codebase is organized into clearly separated layers:

| Project | Role |
|---------|------|
| `Sonarr.Common` | Foundational utilities, disk ops, HTTP abstractions, platform detection |
| `Sonarr.Core` | Business logic: TV management, downloading, indexers, quality, scheduling, DB |
| `Sonarr.Http` | Base HTTP request/response handling, auth, REST resource handling |
| `Sonarr.Api.V3` / `Sonarr.Api.V5` | Versioned REST API endpoints |
| `Sonarr.SignalR` | Real-time UI communication |
| `Sonarr.Mono` / `Sonarr.Windows` | Platform-specific abstractions (disk, process, permissions) |
| `Sonarr.Host` | Application entry point, DI container (DryIoc), middleware pipeline |
| `Sonarr.Update` | Self-update mechanism |

### Database: SQLite with WAL

All *arr applications use **SQLite** as their primary database, with WAL (Write-Ahead Logging) journal mode enabled. The database file is `nzbdrone.db` for Sonarr and similarly named for other *arr apps.

Key database tables in the *arr ecosystem include:

**Sonarr:** `Series`, `Episodes`, `EpisodeFiles`, `Config`, `Commands`, `AutoTagging`, `Blocklist`, `CustomFilters`, `QualityProfiles`, `Tags`, `Indexers`, `DownloadClients`, `Notifications`, `History`, `ScheduledTasks`

**Radarr:** `Movies`, `MovieFiles`, `MovieMetadata`, `AlternativeTitles`, `Credits`, `Collections`, `Config`, `Commands`, `Blocklist`, `QualityProfiles`, `CustomFormats`, `Tags`, `Indexers`, `DownloadClients`, `Notifications`, `History`

Radarr uses a **split data model** that separates movie definition from movie metadata -- multiple movie entries can reference the same metadata record. Alternative titles are pre-loaded and grouped by `MovieMetadataId` as a performance optimization.

### Database Migrations

The *arr apps implement automated schema migrations that run at startup. Radarr notably [squashed its pre-fork Sonarr migrations](https://github.com/Radarr/Radarr/pull/4029) to clean up inherited migration history. The migration system logs operations and validates schema integrity during startup, with corruption detection and error handling.

### PostgreSQL Support

Sonarr v5 and Radarr added optional PostgreSQL support for users who want a more robust database backend. However, migrating an existing SQLite database to PostgreSQL is officially unsupported -- community members have documented workarounds using tools like `pgloader`, but it is not a first-class migration path.

### Lesson for Saalis

The *arr stack validates our choice of SQLite with WAL mode. Their split metadata model (separating the "thing you want" from "metadata about the thing") maps well to our entity + metadata table design. However, the *arr apps' per-content-type table structure (`Series` vs `Movies` vs `Artists`) is exactly what our unified `entities` table with `EntityType` discriminator avoids. Their migration squashing (cleaning up inherited migrations from forks) is a cautionary tale about fork-based development creating migration debt.

## 3. Indexer Abstraction (Prowlarr)

### The Problem Prowlarr Solves

Before Prowlarr, each *arr application independently managed its own indexer (source) configurations. Users had to configure the same indexers in Sonarr, Radarr, and Lidarr separately, and tools like Jackett emerged as third-party indexer proxies. Prowlarr unified this into a single indexer manager that syncs configurations to all connected *arr applications.

### Three-Component Architecture

Every indexer implementation in Prowlarr follows a consistent three-component pattern:

1. **Indexer class** -- defines metadata: name, base URLs, privacy level, supported capabilities, authentication requirements
2. **RequestGenerator** -- constructs HTTP requests tailored to each source's API format (JSON POST, query strings, custom headers)
3. **Parser** -- interprets responses (JSON APIs, HTML tables, XML feeds) and extracts release information into standardised `ReleaseInfo`/`TorrentInfo` objects

This pattern enables consistent interfaces despite wildly diverse underlying APIs.

### Protocol Standards: Newznab and Torznab

Two standardized protocols reduce the per-indexer implementation burden:

- **Newznab** (Usenet): XML-based API with standardised categories, capabilities advertisement via `/api?t=caps`, and NZB file downloads
- **Torznab** (BitTorrent): Extension of Newznab for torrent indexers, adding torrent-specific attributes, magnet link support, and `.torrent` file downloads

Prowlarr supports 24 indexers natively for Usenet, plus any Newznab-compatible indexer via "Generic Newznab". For torrents, it supports 500+ trackers natively plus any Torznab-compatible tracker.

### Cardigann: YAML-Based Indexer Definitions

The most architecturally interesting aspect of Prowlarr is the **Cardigann framework** -- a YAML-based DSL for defining indexer behaviour without writing code. Each YAML definition describes:

- Search URL patterns and query construction
- Authentication flow (login forms, cookies, API keys, 2FA)
- Response parsing rules (CSS selectors, regex, JSON paths)
- Category mappings
- Rate limiting parameters

An `IndexerDefinitionUpdateService` downloads definitions from `indexers.prowlarr.com` and caches them locally. The Cardigann interpreter executes these definitions at runtime. This enables supporting 500+ trackers without compiled code for each one -- only trackers requiring truly custom authentication (cookie sessions, multi-step login, 2FA) get native C# implementations.

### Lesson for Saalis

Prowlarr's architecture maps directly to our Connector/Cataloguer pattern. The three-component split (Indexer/RequestGenerator/Parser) is essentially what we call a Connector (source definition + request building + response parsing). The Cardigann YAML approach is particularly relevant -- saalis plugins that define source behaviour declaratively (via descriptors or configuration) rather than imperatively (via compiled code) would dramatically lower the barrier for community contributions. Our `#[register]` + descriptor pattern could support a similar declarative approach for simple sources while allowing full Rust plugins for complex ones.

## 4. Download Client Integration

### Abstraction Layer

The *arr apps abstract download clients behind a common interface that supports two client families:

- **Usenet clients:** SABnzbd, NZBGet
- **Torrent clients:** qBittorrent, Deluge, Transmission, rTorrent, Flood, Aria2

Each download client implementation handles protocol-specific communication (SABnzbd's REST API, qBittorrent's WebUI API, Deluge's JSON-RPC) while presenting a uniform interface to the core application.

### The Download Pipeline

The pipeline follows a clear state machine:

1. **Grab** -- Sonarr/Radarr sends a download request to the client, associating it with a configured category label. The category prevents conflicts with non-*arr downloads.

2. **Downloading** -- The download client handles the actual transfer. The *arr app polls the client periodically for status updates.

3. **Completed Download Handling** -- When the client reports a download as ready, the *arr app scans the completed file location for usable files. It parses filenames to match against the requested content.

4. **Import** -- Matched files are renamed according to configurable naming templates and moved/hardlinked to the library folder. The system attempts hardlinks first (preserving data for seeding), falling back to copy if hardlinks are not supported (e.g., cross-filesystem moves).

5. **Cleanup** -- If configured, leftover download files are removed via the client's API.

### BitTorrent-Specific Handling

Torrents that are still seeding are left in place. Hardlinks allow the library copy to exist without duplicating disk space. When seeding completes:

- If hardlinks are supported: atomic move of the original file
- If not: copy + delete

File locks from active seeding can prevent renaming, requiring a temporary pause-rename-resume workaround.

### Error Handling

The pipeline includes:

- Stalled download detection with configurable timeouts
- Failed download blocklisting (preventing re-grab of known-bad releases)
- Automatic re-search on failure
- Queue management API for manual intervention (remove, blocklist, force-import)

### Lesson for Saalis

The *arr download pipeline is mature and battle-tested. Our sourcing pipeline should follow a similar state machine pattern (discover -> grab -> download -> import -> organise). The hardlink strategy for seeding is essential -- saalis must handle this from day one if we support torrent sources. The category/label system for isolating saalis downloads from other client usage is a simple but critical UX detail. The blocklist pattern (permanently avoiding known-bad sources) is directly applicable to our sourcing quality system.

## 5. Quality Profiles and Upgrades

### Quality Profile Architecture

The *arr quality system is a multi-layered decision engine:

**Quality Tiers:** A ranked list of quality levels (e.g., SDTV < WEBDL-480p < Bluray-720p < WEBDL-1080p < Bluray-1080p < Remux-1080p < WEBDL-2160p < Bluray-2160p < Remux-2160p). The ordering within the profile determines preference.

**Cutoff:** A quality level at which the system stops actively seeking upgrades. Once the cutoff quality is reached, the *arr app no longer searches for better releases -- but if a higher-quality release appears (e.g., via RSS), it will still be downloaded.

**Quality Groups:** Multiple qualities can be grouped together and treated as equivalent. This allows saying "720p WEB and 720p Blu-ray are equally acceptable" without preferring one over the other.

### Custom Formats (v4+)

Custom Formats replaced the simpler "Preferred Words" system and are the most sophisticated part of the quality engine:

- **Pattern matching:** Custom Formats define regex-based rules that match against release attributes (name, source, codec, resolution, language, release group, etc.)
- **Scoring:** Each Custom Format has a numeric score assigned per quality profile. Scores are additive -- a release matching multiple Custom Formats gets the sum of their scores.
- **Minimum score:** Profiles can set a minimum Custom Format score below which releases are rejected.
- **Upgrade threshold:** Profiles specify an "Upgrade Until" score -- the system upgrades existing files until this cumulative score is reached.

### Decision Logic

When evaluating whether to grab a release:

1. **Quality check first** -- Is this release's quality acceptable in the profile? Is it at or above the existing file's quality?
2. **Custom Format scoring** -- Calculate the cumulative score from all matching Custom Formats.
3. **Upgrade decision** -- If the release is the same quality as the existing file, prefer the one with the higher Custom Format score. If qualities differ, prefer the higher-ranked quality (up to cutoff).
4. **Negative scoring** -- Formats with very negative scores (e.g., -10000) effectively act as hard rejects, even if quality would otherwise be acceptable.

### Known Issues

There are edge cases where the upgrade logic misbehaves: Custom Format score upgrades can sometimes occur even when "Upgrades Allowed" is unchecked, if the existing file's score is below the "Upgrade Until Custom Format Score" threshold. This has been [reported as a bug](https://github.com/Sonarr/Sonarr/issues/5330).

### Lesson for Saalis

The Custom Format scoring system is powerful and should inform our sourcing priority system. The two-tier approach (quality tier + format score) allows both coarse-grained (resolution/source type) and fine-grained (codec, release group, language) preferences. However, the system's complexity generates significant user confusion -- TRaSH Guides exists primarily to help users configure quality profiles correctly. Saalis should aim for similar expressiveness with better defaults and clearer mental models. The "Upgrade Until" concept maps directly to our quality thresholds.

## 6. Metadata Handling

### The SkyHook Proxy Pattern

Radarr's metadata system centres on `SkyHookProxy`, which implements `IProvideMovieInfo` and `ISearchForNewMovie`. Rather than calling TMDb/IMDb APIs directly, the *arr apps route through their own metadata proxy servers (skyhook.sonarr.tv, etc.). This provides:

- API key management (users don't need their own TMDb keys)
- Response normalisation across metadata sources
- Server-side caching reducing load on upstream APIs
- Ability to supplement/correct upstream data

### Multi-Source Search

The metadata system supports diverse query patterns:

| Query Type | Example | Implementation |
|-----------|---------|----------------|
| IMDb ID | `imdb:tt1234567` | Direct lookup |
| TMDb ID | `tmdb:12345` | Direct lookup |
| URL pattern | TMDb/IMDb URLs | ID extraction via regex |
| Title search | `Movie Title 2023` | Text-based with optional year filtering |

### Data Transformation Pipeline

External API responses are transformed into internal domain objects through a `MapMovie` method that handles:

- **Title processing:** Clean titles, sort titles, original titles
- **Date handling:** Cinema, physical, and digital release dates with automatic status determination
- **Rating aggregation:** Combines ratings from TMDb, IMDb, Metacritic, Rotten Tomatoes, and Trakt
- **Media assets:** Poster and fanart image URLs
- **Localisation:** Alternative titles and translations
- **Relationships:** Collection membership and recommendation linking

### Discovery Features

Beyond search, the metadata system provides discovery:

- `GetTrendingMovies()` for currently popular content
- `GetPopularMovies()` for sustained interest
- `GetChangedMovies()` for monitoring metadata updates (uses 15-minute caching buffer)

### Metadata Source Tensions

A notable ongoing tension: **Sonarr is locked to TVDB** for TV series metadata, with [significant community demand](https://forums.sonarr.tv/t/tmdb-as-metadata-source/32403) for TMDb as an alternative or primary source. TVDB's transition to a paid model created anxiety about dependency on a single metadata provider. This is a lesson in avoiding tight coupling to external services.

### Lidarr's MusicBrainz Challenge

Lidarr maintains its own metadata API server (`LidarrAPI.Metadata`) that proxies MusicBrainz data. This requires running a MusicBrainz PostgreSQL mirror and Solr search server -- significant infrastructure just for metadata. MusicBrainz schema changes have broken Lidarr searches, and cache-busting requires manual intervention by team members.

### Lesson for Saalis

The proxy/intermediary pattern for metadata is smart -- it decouples the application from upstream API changes and rate limits. Saalis should consider a similar pattern for game metadata (IGDB, MobyGames, etc.) rather than direct API calls. However, the TVDB lock-in lesson is critical: we must support multiple metadata sources per entity type, with user-configurable priority. The SkyHook pattern also demonstrates the value of server-side response normalisation -- our Connector/Cataloguer pattern should normalise diverse source responses into our unified metadata model at the connector level, not in core business logic.

## 7. File Management

### Naming Conventions

The *arr apps use a token-based naming system with configurable templates:

**Sonarr example:**
```
{Series.CleanTitleYear}.S{season:00}E{episode:00}.{QUALITY.REAL}.{QUALITY.PROPER}.{Quality.Title}.{MediaInfo.VideoCodec}-{RELEASE.GROUP}
```

**Radarr example:**
```
{Movie.CleanTitle}.{Release.Year}.{Quality.Full}.{MediaInfo.AudioCodec}.{MediaInfo.AudioChannels}.{MediaInfo.VideoCodec}-{RELEASE.GROUP}
```

Tokens support various formatting modifiers (`.` separator, case control, clean title variants). MediaInfo tokens require video file analysis via ffprobe.

### Hardlink Strategy

The hardlink system is central to *arr file management:

- When importing, the app creates a **hardlink** from the download location to the library location
- Both paths point to the same disk blocks -- no additional space consumed
- The torrent client continues seeding from the original path
- The library serves from the renamed/organised path

**Requirements:** Both paths must be on the same filesystem and same mount. This is why TRaSH Guides recommend a shared parent folder structure:

```
/data/
  /downloads/
    /torrents/
    /usenet/
  /media/
    /movies/
    /tv/
```

### Root Folder Management

Each *arr app manages one or more "root folders" -- base directories where media is stored. Content is organised into subfolders within root folders (one folder per series/movie). Radarr requires individual folders per movie; flat file structures are explicitly unsupported.

### Multi-Disk Considerations

The hardlink requirement (same filesystem) creates complications for multi-disk setups. Users must use filesystem-level solutions (mergerfs, JBOD pools, ZFS) to present multiple disks as a single mount point, or accept copy-based imports for cross-filesystem moves.

### Lesson for Saalis

For games, file management differs significantly from media files -- ROMs may need specific directory structures per emulator, and game installations have their own folder conventions. However, the hardlink pattern for maintaining seed-ability is directly applicable if saalis supports torrent-based sourcing. The token-based naming system is a good UX pattern for configurable file organisation. We should note the multi-disk pain point and ensure our filesystem abstraction handles it gracefully from the start.

## 8. API Design

### REST API Architecture

The *arr apps implement comprehensive REST APIs following OpenAPI 3.0 specifications:

- **Radarr** exposes 145+ endpoints under `/api/v3/`
- **Sonarr** maintains both V3 (`/api/v3/`) and V5 (`/api/v5/`) endpoint versions
- API documentation is auto-generated from controller attributes and method signatures

### Resource Model Pattern

API responses follow a consistent convention:

- `{Entity}Resource` for single entities
- `{Entity}ResourcePagingResource` for paginated collections
- `{Entity}BulkResource` for bulk operations

Pagination is standardised with `page`, `pageSize`, `sortKey`, `sortDirection` parameters.

### Authentication

Three authentication mechanisms coexist:

1. **API Key** -- `X-Api-Key` header or `apikey` query parameter for programmatic access
2. **Form-based** -- Cookie/session authentication for the web UI
3. **SignalR** -- Specialised auth for WebSocket connections

### DI and Startup

Sonarr uses **DryIoc** as its DI container. The startup sequence is:

1. `ConfigureServices()` -- register all DI services
2. `Configure()` -- build HTTP middleware pipeline
3. Database initialisation -- run migrations, validate schema
4. Event publishing -- system-ready notifications
5. Single-instance check -- prevent concurrent instances on the same database

The middleware pipeline is carefully ordered: forwarded headers, logging, URL base rewriting, exception handling, routing, CORS, auth, compression, caching, WebSocket, SignalR hubs, controller mapping.

### Command/Task Queue

The *arr apps implement an asynchronous command pattern:

- Commands are published and executed asynchronously
- Each command has: `TimeAdded`, `Command`, `TimeExecuted`, `Duration`, `CommandBody`, `Priority`, `Error`
- Commands can be queried by ID for status
- Long-running operations do not block API responses
- The UI can manipulate the queue: cancel running tasks, remove pending items

This is effectively a persistent work queue -- tasks are tracked to completion with full observability.

### SignalR for Real-Time Updates

SignalR provides push-based updates from server to UI:

- Download progress updates
- Import notifications
- Health check alerts
- Queue state changes

External tools (Bazarr, Home Assistant) also connect to SignalR feeds for real-time synchronisation, demonstrating the value of making the real-time channel externally accessible.

### Lesson for Saalis

The command queue pattern maps directly to our scheduler work unit system. Their approach of tracking every task to completion with status, priority, and error information validates our event-driven DAG scheduler design. The API Key authentication for programmatic access is essential -- saalis will be accessed by scripts and other tools. The auto-generated OpenAPI documentation is valuable for ecosystem development. However, our sync-core/async-shell boundary means we should not adopt SignalR wholesale -- our web layer (axum) can provide WebSocket-based real-time updates while keeping the core synchronous.

## 9. Notification System

### Connect Providers

The *arr notification system (called "Connect" in the UI) supports a wide range of providers:

- **Chat platforms:** Discord, Slack, Telegram
- **Push services:** Pushover, Pushbullet, Boxcar, Simplepush, ntfy
- **Email:** SMTP-based with STARTTLS
- **Media servers:** Plex, Emby, Jellyfin (library update triggers)
- **Webhooks:** Generic HTTP POST with JSON payloads
- **Custom scripts:** Execute local scripts with event data as environment variables
- **Specialised:** Synology Indexer, Kodi (XBMC), Gotify, Notifiarr

### Trigger Events

Each provider can be selectively enabled for specific events:

| Trigger | Description |
|---------|-------------|
| On Grab | Content identified and sent to download client |
| On Import | Content successfully imported to library |
| On Upgrade | Existing content replaced with better quality |
| On Rename | Files renamed (without quality change) |
| On Health Issue | System health check failure |
| On Health Restored | Previously failing health check resolved |
| On Application Update | Application version updated |
| On Manual Interaction Required | Human intervention needed |

### Provider Abstraction

Each notification provider implements a common interface that receives event-specific payload data. Providers are independently configurable -- users can have Discord notifications for grabs but Telegram for health issues.

### Webhook Data Parity Issue

A [known issue](https://github.com/Sonarr/Sonarr/issues/6822): Discord notifications contain more data than generic webhooks. The webhook payload is supposed to be the canonical complete format, but Discord-specific formatting led to data being added to Discord notifications without updating the webhook payload. This is a cautionary tale about maintaining parity across notification channels.

### Community Extensions

The built-in notification system's limitations have spawned significant community tooling:

- **Bettarr-Notifications** -- Enhanced notification formatting for Discord, Telegram, Slack, Pushover, and ntfy
- **Notifiarr** -- Centralised notification aggregator with configurable reactions and rich formatting
- **Custom script ecosystem** -- Shell/Python scripts triggered via the Custom Script provider

### Lesson for Saalis

Our notification system design (NotificationPayload as a registered trait) aligns well with the *arr approach, but we should learn from the webhook data parity bug: the canonical notification payload must be complete and identical regardless of delivery channel. Channel-specific formatting should be a presentation concern, not a data concern. The trigger event list is a good starting point for our own event taxonomy. The Custom Script provider is valuable for extensibility -- saalis should support a similar "run arbitrary executable on event" mechanism.

## 10. What They Got Wrong

### The Fork Problem (Critical)

The single biggest architectural mistake in the *arr ecosystem is the fork-based content-type extension model. Each content type (TV, movies, music, books) requires an entirely separate application -- separate process, separate database, separate configuration, separate maintenance. This creates:

- **User complexity:** Running a full media stack requires 4-6 separate applications (Sonarr + Radarr + Lidarr + Readarr + Prowlarr + Bazarr), each with its own UI, settings, and port
- **Developer burden:** Shared bug fixes must be cherry-picked across all forks
- **Configuration sprawl:** Indexers, download clients, and notification providers must be configured in each app (Prowlarr partially addresses this for indexers)
- **Resource waste:** Multiple identical web servers, schedulers, and database engines running for what is conceptually one application

The community has noticed: projects like [Bobarr](https://github.com/iam4x/bobarr) market themselves as "all-in-one alternatives," and [Buildarr](https://buildarr.github.io/) exists primarily to manage the configuration complexity. Users describe the apps as "all definitely feel like small parts of a single package."

### Metadata Source Lock-in

Sonarr's hard dependency on TVDB (with no option to switch to TMDB) has been a persistent source of frustration, especially as TVDB moved to a paid model. The system should have been designed with metadata source as a swappable strategy from the start.

### Quality Profile Complexity

The quality profile system, while powerful, is notoriously difficult to configure correctly. The existence of TRaSH Guides (a comprehensive community guide just for *arr configuration) and tools like Recyclarr and Profilarr (which automate quality profile configuration from YAML) demonstrates that the UX failed. Users should not need a third-party guide to configure basic preferences.

### Database Limitations

- SQLite's single-writer limitation causes "database is locked" errors under concurrent load
- The `nzbdrone.db` filename persists from the pre-Sonarr era, confusing new users
- No clean SQLite-to-PostgreSQL migration path despite PostgreSQL being offered as an option
- SQLite 3.41.0's change to disable double-quoted strings [broke Sonarr](https://github.com/Sonarr/Sonarr/issues/5464), revealing fragile SQL construction

### VPN Incompatibility

Putting *arr apps behind a VPN "can and will make the applications unusable" because image/metadata providers block VPN IPs and Cloudflare sits in front of most *arr servers. The apps were not designed with proxy/VPN-awareness in mind.

### Disk I/O Spikes

The "Refresh" task (re-scanning library folders) causes heavy I/O that can overwhelm NAS devices. The only mitigation is adjusting a "Rescan after Refresh" setting. Better incremental scanning (inotify/fsevents) would solve this.

### Flat File Structure

Radarr's inability to support flat file structures (requiring one folder per movie) limits flexibility. This is described as requiring "substantial backend modifications" to change, indicating the folder-per-item assumption is deeply embedded.

### Lesson for Saalis

Every one of these is avoidable:

1. **Single binary, multiple content types** -- already our design
2. **Pluggable metadata sources** -- our Connector pattern handles this
3. **Sensible defaults with progressive disclosure** -- our settings system should have opinionated defaults that work out of the box
4. **SQLite discipline** -- use parameterized queries, WAL mode, and design for the single-writer constraint from day one; our fire-and-forget write scheduler already accounts for this
5. **Proxy-aware networking** -- build HTTP client configuration (proxy, custom CA, timeout) into the core from the start
6. **Filesystem events** -- use notify/kqueue for incremental scanning rather than periodic full rescans
7. **Flexible file organisation** -- don't hardcode folder structure assumptions into the data model

## 11. Community Extensions and Extensibility

### The Ecosystem

The *arr ecosystem has spawned an enormous set of companion tools, documented in the [Awesome *Arr](https://ravencentric.cc/awesome-arr/) collection:

**Configuration Management:**
- **Buildarr** -- Declarative YAML-based configuration for entire *arr stacks
- **Recyclarr** -- Syncs TRaSH Guides quality profiles to *arr instances
- **Profilarr** -- Configuration management platform for Radarr/Sonarr

**Companion Applications:**
- **Bazarr** -- Subtitle management that connects to Sonarr/Radarr SignalR feeds
- **Overseerr/Jellyseerr** -- Request management and media discovery
- **Tdarr** -- Transcode automation that notifies Radarr/Sonarr of changes
- **Maintainerr** -- Automatic library cleanup based on rules

**Alternative Interfaces:**
- **Reiverr** -- Combined interface for Jellyfin + TMDB + Radarr + Sonarr
- **Ruddarr** -- Native iOS companion app
- **LunaSea** -- Mobile management app

**Discovery and Automation:**
- **SuggestArr** -- Recommendation engine based on viewing history
- **Listrr** -- Automatic list generation from filters
- **Upgradinatorr** -- Keeps media synced with Custom Format scoring

**Monitoring:**
- **Notifiarr** -- Notification aggregation and formatting
- **Home Assistant integrations** -- Dashboard widgets and automation triggers

### Extensibility Patterns That Work

1. **REST API as the primary extension point** -- Every community tool interacts through the documented REST API. This is the single most important extensibility decision.

2. **SignalR for real-time integration** -- Tools like Bazarr subscribe to real-time events rather than polling, enabling responsive companion apps.

3. **Webhook + Custom Script** -- The Connect system's webhook and custom script providers serve as escape hatches for any integration not directly supported.

4. **Category/tag system** -- Tags and categories allow external tools to organise and filter content without modifying core data structures.

5. **OpenAPI auto-generation** -- Machine-readable API specs enable automated client library generation (pyarr, sonarr-py, arrapi).

### Patterns That Do Not Work

1. **No plugin system** -- There is no way to extend *arr functionality without forking the entire project or building a separate companion app. Every extension is an external process.

2. **Per-app configuration** -- The lack of shared configuration across *arr instances led to Buildarr and Prowlarr as workarounds.

3. **Limited internal hooks** -- Custom Scripts receive event data but cannot modify *arr behaviour (e.g., cannot influence quality decisions or modify metadata before import).

### Lesson for Saalis

The *arr ecosystem validates that a good REST API is the most important extensibility mechanism. But it also demonstrates the ceiling of API-only extensibility -- the community constantly wants to modify internal behaviour (quality decisions, naming logic, metadata enrichment), which requires either forking or building entire companion applications.

Saalis's plugin system (registered traits, `repr(C)` ABI, `inventory` pattern) addresses this directly. Plugins that can register new EntityTypes, Connectors, quality scorers, and notification providers without forking are a genuine differentiator. However, we should also provide the API-level extension points (REST API, webhooks, custom scripts) because they have a much lower barrier to entry than compiled plugins.

## Summary: Key Takeaways for Saalis

| *arr Pattern | Saalis Equivalent | Lesson |
|-------------|-------------------|--------|
| Fork per content type | Unified `EntityType` trait | Avoid O(N) maintenance; one binary handles all content |
| Prowlarr indexer abstraction | Connector/Cataloguer pattern | Three-component split (source/request/parse) is proven |
| Cardigann YAML definitions | Declarative descriptor registration | Declarative source definitions lower contribution barrier |
| Quality profiles + Custom Formats | Sourcing priority system | Two-tier scoring (coarse quality + fine format) is powerful but needs better UX |
| SkyHook metadata proxy | (Consider for game metadata) | Proxy pattern decouples from upstream API changes |
| SignalR real-time updates | axum WebSocket layer | Real-time updates are essential for UX, but keep sync core |
| Command queue | DAG scheduler work units | Track every task to completion with full observability |
| REST API + webhooks | Web API + notification system | API is the primary extensibility mechanism; provide escape hatches |
| Connect notification providers | Registered NotificationPayload trait | Maintain payload parity across all channels |
| No plugin system | `#[register]` + `repr(C)` + `inventory` | Genuine differentiator; fill the gap *arr cannot |
| SQLite + WAL | SQLite + WAL (same) | Proven for single-user appliance; respect single-writer constraint |
| Hardlinks for seeding | Filesystem abstraction | Essential for torrent source support |
| Mono -> .NET Core migration | Rust from day one | Avoid the platform abstraction tax entirely |

The *arr stack is the most successful open-source content library management ecosystem. Its decade of evolution validates many of our architectural choices while clearly demonstrating the pitfalls of fork-based extension and metadata source lock-in. Saalis's entity-centric, plugin-extensible, single-binary design avoids their biggest mistakes while learning from their proven patterns.
