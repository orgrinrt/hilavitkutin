# Hilavitkutin per-crate documentation audit, 2026-05-05

**Date:** 2026-05-05.
**Scope:** `mock/crates/*/{README,DESIGN,BACKLOG}.md.tmpl` across 11 crates. No `DEEPDIVE_*.md.tmpl` files exist in the hilavitkutin repo.
**Method:** Manual read pass against `documentation-writing.md` + `writing-style.md` + `vocabulary.md` + `ai-agent-framing.md` workspace rules, with mechanical greps for em-dash, banned vocabulary, leakage patterns, and stale-name cross-reference checks against `crates/*/src/`.

## Summary

Templates are largely free of em-dashes (clean) and free of marketing/hype-word violations. The dominant issues are Tier 1 leakage (28 hits across 11 templates referencing `mockspace`, `mock/research/sketches/`, `mock/design_rounds/`, design-round IDs, and the bare phrase "design round"); engine-vocabulary drift (the word `entry`/`entries` doubling for both engine records and arena/string-table slots, plus residual `chain` and `entities` in engine-domain prose); two ASCII tree diagrams in `hilavitkutin-extensions/DESIGN.md.tmpl` and `hilavitkutin-linking/DESIGN.md.tmpl`; surface-level inconsistencies in README heading levels (h1 vs h3) and DESIGN-title naming ("Design" / ": Design" / ", Design" / "Design Overview" / no suffix); a handful of stale type names (`MemoryProvider`/`ThreadPool`/`Clock` in api docs where source has `*Api` suffix; `WeightedSieveCache` in persistence docs where source has plain `SieveCache`; `tuple_generic!` in ctx docs where no such macro is exported); and one Round 4 stale Kit-trait holdover requiring an explicit "deleted" note rather than the prose currently used.

Severity skew: most findings are mechanical-cleanup grade. Two findings warrant design discussion (the engine-vocabulary `entry` doubling in providers + str storage prose, and the `MemoryProviderApi` vs `MemoryProvider` rename drift between api/DESIGN.md.tmpl prose and source).

## Cross-cutting patterns

1. **Tier 1 leakage of `mock/` paths and round-ID phrasing** is the single highest-volume failure across the audit (28 hits). The recurring shapes are: (a) `mock/research/sketches/<datetime>_<topic>/FINDINGS.md` references threaded through DESIGN.md.tmpl files; (b) `mock/design_rounds/<datetime>_topic_<name>.md` references; (c) the bare phrase "design round" or "during design round YYYYMMDDHHMM"; (d) the phrase "src CL" or "src CL of round X"; (e) the phrase "mockspace (workspace convention)". The fix is mechanical and cross-cutting: a single sweep removing each of these references. The audit-trail value lives in `mock/research/` itself; the rendered docs do not need the cross-reference.

2. **Engine-vocabulary `entry`/`entries` doubling** between (a) the engine's banned record-synonym usage and (b) legitimate domain meanings (arena slots in `MemoryArena`, string-table entries in `StringInterner`, list-items in BACKLOG section headers). The vocab.md rule already carves out hash-table entry / interner-arena entry / changelist entry / .init_array entry as narrow exceptions. The issue is that in practice the boundaries are blurred: providers/DESIGN treats arena slots as `Entry`, str/DESIGN says "268M unique entries" (string-table entries), and the same word "entry" appears in engine-domain prose (api/BACKLOG L154 "trait-solver chain" + "duplicate count" + nearby usage). Recommend either (a) a sweep that renames source `Entry` to `Slot` or `Bucket` to mechanically eliminate the overlap, or (b) an explicit per-crate vocab carveout block at the top of providers/DESIGN.md.tmpl + str/DESIGN.md.tmpl that documents "in this crate, `entry` refers to the string-table slot type, not the engine record". Option (a) is cleaner.

3. **Engine-vocabulary `chain` doubling** between (a) the banned engine dead term, (b) algorithm-name vocabulary ("matrix chain DP"), (c) compile-time-trait-solver context ("trait-solver chain length"), (d) Rust-syntax method-chain context ("builder chain"). Two distinct shapes here: (b) is acceptable as algorithm vocab; (c) is the actual `chain` ban in cons-list/accumulator context (rename to "cons-list"); (d) is borderline — "builder chain" is widely-understood Rust idiom, and replacing with "builder pipeline" risks confusion with the engine's `pipeline` term. Recommend keeping (b) and (d), renaming all (c) instances.

4. **`hilavitkutin-persistence` tabular vocabulary clash** with the engine's record vocabulary (`RowCount`, `ColumnCount` as types; `row_count` as field name; `entities` as informal prose). Persistence is fundamentally a tabular-serialisation concern, but its tabular vocab leaks into the engine's record-only world via the doc-template surface. This is a substantive design question: either rename persistence types to use record vocabulary (`RecordCount` / `ColumnCount` stays, no row), or harmonise the workspace vocab rule to allow tabular vocab inside persistence-scoped contexts. Either direction is a design-round-level decision, not a mechanical cleanup.

5. **README heading-level inconsistency** — four crates use h3 (`### hilavitkutin-ctx`, `### hilavitkutin-kit`, `### hilavitkutin-persistence`, `### hilavitkutin-str`); five use h1; one uses h1 with backticks (`# `hilavitkutin-providers``). Pick one convention and apply across all 11 README.md.tmpl files. The hilavitkutin/.claude/rules/readmes.md says READMEs are "SHORT (3-10 lines)" but doesn't specify a heading level. Suggest h1 (matching most siblings) without backticks (cleaner); standardise across all 11.

6. **DESIGN-title naming inconsistency** — variants observed: "X" / "X, Design" / "X: Design" / "X Design Overview" / "X, Design Overview". One workspace convention applied across all 11 DESIGN.md.tmpl files. Suggest `# <crate-name>, design` matching the writing-style heading rule (sentence case after the comma, lowercase imperative).

7. **Rendered Tier 1 vs. Tier 2 boundary** — these `mock/crates/*/*.md.tmpl` files are templates that compose into the rendered `docs/DESIGN.md` (per `documentation-writing.md` Tier 3 rules). The Tier 1 rules apply to whatever portion of these templates ends up in the rendered output. Without knowing the exact strip-or-not behaviour of the doc-generation pipeline, the safe assumption is that all template prose ships into a public-facing surface.
