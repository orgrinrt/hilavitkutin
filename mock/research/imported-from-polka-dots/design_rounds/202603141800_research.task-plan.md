# Research Task Plan: hilavitkutin Core Design

**Created:** 2026-03-14
**Purpose:** Overnight research sprint to prepare for hilavitkutin core
design decisions. The outputs feed into the topic file at
`202603141800_topic.hilavitkutin-core-design.md`.

**Context:** hilavitkutin is a shared pipeline execution engine for three
Rust projects: polka-dots (dotfiles build engine), saalis (game library
manager), and loimu (UI framework). The design must find common
denominators between what all three need for DAG-based scheduling, typed
read/write access, morsel-driven column processing, and cooperative
yielding for I/O-bound work. loimu has the most mature design and is the
primary reference. Academic literature and industry practice provide the
external validation. loisto (a Zig-based desktop environment) provides
a lower-level perspective on scheduling and efficiency.

**Target output:** 40,000+ lines of research documentation across all
phases, written to `mock/design_rounds/` as research files.

---

## Phase 1: Loimu Design Research (8 documents)

loimu is a Rust UI framework at `~/Dev/loimu/`. Its docs directory
contains ~80 design documents organized as `LOIMU_<SYSTEM>_OVERVIEW.md`
for high-level and `LOIMU_<SYSTEM>_<TOPIC>.md` for detail. It also has
research documents at `~/Dev/loimu/mock/research/` (SOURCES.md,
SUMMARIES.md). All eight tasks in this phase produce research documents
that exhaustively cover loimu's relevant design.

The purpose of Phase 1 is to deeply understand loimu's architecture so
we can identify what to adopt, what to simplify, and what to skip when
designing hilavitkutin. loimu is not a template to copy — it's a
reference implementation that solved many of the same problems in a
UI-framework context. Our job is to extract the general-purpose core.

### Task 1.1: Loimu Storage Architecture Research

**Output file:** `202603141800_research.loimu-storage.md`
**Minimum length:** 3000 words

**Source material — read ALL of these from `~/Dev/loimu/docs/`:**
- `LOIMU_STORAGE_OVERVIEW.md` — high-level storage architecture
- `LOIMU_STORAGE_STORAGE.md` — detailed storage internals
- `LOIMU_STORAGE_RECORDS.md` — record (row) storage model
- `LOIMU_STORAGE_VIEWS.md` — SoA view abstraction
- `LOIMU_STORAGE_FUSION.md` — how the storage engine fuses iterations
- `LOIMU_STORAGE_TEMPORAL.md` — temporal storage (pre-computed futures)
- `LOIMU_STORAGE_MARKER_STORAGE.md` — marker storage (tags/labels)
- `LOIMU_BUFFERS_OVERVIEW.md` — buffer primitives (SlabColumn, etc.)
- `LOIMU_PRIMITIVES_OVERVIEW.md` — fixed-point types and their storage

Also check `~/Dev/loimu/mock/crates/` for storage-related crate stubs
(likely `loimu-storage`, `loimu-buffers`) and read their `src/lib.rs`.

**Content requirements (cover ALL of these):**

1. **SlabColumn internals** — the core storage primitive. Memory layout,
   allocation strategy, growth model, backing implementation. How does
   a SlabColumn store typed data? Relationship to the stride model.

2. **The 16-byte stride model** — 1 byte flags + 15 bytes payload. Why
   16 bytes? Cache line interaction (64B on most CPUs, 128B coherency on
   Apple Silicon). Flag byte contents. How types smaller than 15 bytes
   are stored. How types larger than 15 bytes are handled.

3. **Field<T> vs Collection<T>** — the two field kinds in resources.
   Field<T> is a single value. Collection<T> is a variable-length sequence.
   CollectionHandle (index + length). CollectionStorage.

4. **Three storage backends** — Record (persistent, SwapBuffer +
   SnapshotBuffer), Snap (ephemeral signals, double-buffered arenas),
   Reel (pre-computed future values, FIFO). Purpose, allocation,
   access, lifetime for each.

5. **SwapBuffer and SnapshotBuffer** — double-buffering for concurrent
   read/write. Buffer swapping between passes. Snapshotting for fault
   rollback. Sole-writer vs shared-writer differences.

6. **Occupancy BitSet** — sparse storage. One bit per slot. Vectorized
   iteration. Interaction with morsel-style chunking.

7. **Packing strategies** — bit width → packing rules. SIMD unpack
   kernels. Compile-time tables.

8. **View abstraction and SoA layout** — Views, column independence,
   addressability, multi-column composition.

9. **Storage fusion** — how the storage engine fuses compatible iterations
   from different scheduling units. Fusibility criteria. Performance
   impact.

10. **Temporal storage** — the Reel backend. Pre-computed future values.
    How does temporal storage enable prediction-based skipping in the
    scheduler? Feed rate, read cursor, FIFO lifecycle.

11. **Marker storage** — tags, labels, entity state markers. How do
    these differ from regular columns?

12. **Design rationale and critique** — why these choices? What works
    well? Where is there unnecessary complexity for hilavitkutin's needs?

**Content requirements (cover ALL of these):**

   signals; scheduler wakes, runs DAG pass, sleeps. Comparison to game
   loops and Tokio runtimes.

2. **Three policy types** — Continuous, Adaptive, OnDemand. Interval
   computation, overrun handling.

3. **List scheduling with upward rank** — step-by-step algorithm.
   Kahn's for topological order. Upward rank formula. Initial heuristic
   cost. EMA replacement after 10 passes. Why upward rank vs HEFT.

4. **Lane assignment and worker dispatch** — mapping units to threads.
   Highest-rank-first assignment. Vertical-first ordering. Why
   vertical-first unlocks more downstream work.

5. **Work-stealing** — overflow lists, lane_compat bitmask, upstream_rank
   priority. Comparison to Rayon/Cilk deque stealing.

6. **Per-unit metadata** — lane, lane_compat, depth, upstream_rank,
   fusible. Computation timing (startup + recalculation signal).

7. **Rotation groups and prediction-based skipping** — N rotation groups,
   1/N active per pass. Adaptive tolerance formula. Prediction quality
   heuristic.

8. **Version stamps and phantom writes** — version:u64, early-release
   of downstream units, zero atomics.

9. **Fault handling and rollback** — sole-writer snapshot restore,
   shared-writer undo journal, BehaviorFaulted signal.

10. **SchedulerMetrics** — pass count, avg duration, activity level.
    Adaptive policy tuning.

11. **Deferred models** — Naughty Dog fibers, work-stealing wavefront.
    Why deferred. What would change.

12. **Relevance and critique** — what to adopt, what to simplify, what
    hilavitkutin adds (yielding).

   stamps.

8. **Frame generation** — what the GRAPH_FRAME_GENERATION doc covers.
   How does the graph produce frames (or passes)? Generation scheduling.

9. **Signal flow edges** — emit/observe edges. Same-pass delivery.
   Mid-pass signal deferral.

10. **Fusible vs non-fusible units** — criteria for storage engine

11. **Exclusive access guarantee** — DAG-enforced exclusive writes.
    Comparison to Rust's borrow checker.
