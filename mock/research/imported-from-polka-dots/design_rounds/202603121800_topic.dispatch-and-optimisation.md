## Open questions — to be resolved interactively

### From reference system comparison (DuckDB, BLIS, Polars)

These were stated as "conscious differences" in the comparison but
were NOT explicitly discussed or decided. Each needs review:

**Q1. NUMA awareness** — RESOLVED

DuckDB assigns NUMA-local morsels and uses per-socket storage areas.
We assumed "single-socket consumer hardware, NUMA doesn't apply."
Is this the right scope, or should the design at least be
NUMA-compatible for future server use?

**Background.** NUMA (Non-Uniform Memory Access) is a memory
architecture on multi-socket servers. Each CPU socket has local RAM;
accessing another socket's RAM costs 2-3x more latency (~150-200 ns
vs ~80 ns) due to inter-socket interconnect traversal. DuckDB handles
this with morsel locality (assign morsels to threads on the same socket
as the data), thread-local storage (per-socket hash table build
buffers), and interleaved allocation (shared read-only structures
spread across sockets).

NUMA awareness in our library would require: socket topology detection,
NUMA-local allocation hints in ColumnStorage, locality-aware chain-to-
core assignment, and possibly per-socket progress tracking. This is
significant complexity.

**Decision: NUMA is out of scope, but the design must not preclude it.**

Our target is consumer hardware — single socket, unified memory (even
Apple Silicon Ultra presents unified address space). Our data volumes
(KB to low MB) fit in L2, so memory controller locality is irrelevant.

However, the Executor trait and ColumnStorage trait together provide
the extension points for a future NUMA-aware implementation:

1. **Chain-to-core assignment is the Executor's choice.** The library
   provides the chain dependency graph; the Executor decides which
   core runs which chain. A NUMA-aware Executor assigns chains to
   cores on the socket where input data resides.
2. **ColumnStorage allocation is consumer-controlled.** The consumer
   implements both ColumnStorage and Executor, so they can coordinate:
   Executor knows which core runs chain X, ColumnStorage allocates
   chain X's columns on that core's local NUMA node.
3. **AtomicUsize progress counters work across NUMA boundaries.**
   Higher latency for cross-socket atomics, but at morsel granularity
   (one atomic per morsel per chain), this is negligible.

No library-level NUMA abstractions needed. The chain model is
accidentally NUMA-friendly: chains don't share data across cores
during execution.

**Q2. Out-of-core / spill to disk** — RESOLVED

Polars can spill morsels to disk when memory is tight. We assumed
"data is KB to low MB, always fits in memory." Is a memory ceiling
needed? Should the ColumnStorage trait at least allow for a spilling
implementation, even if we don't build one?

**Background.** Out-of-core execution means processing datasets that
don't fit in RAM by spilling intermediate data to disk and paging it
back when needed. Polars serializes morsels to Arrow IPC format and
restores them under memory pressure. DuckDB's buffer manager evicts
pages to a temp directory. Both add significant complexity:
serialization formats, temp file management, memory pressure monitors,
page-in/page-out logic in every data access path.

**Analysis across three target scales:**

polka-dots (500 rows × 15 cols × 16 B = 120 KB): fits in L2 cache,
let alone RAM. Spill is absurd here.

saalis (200K–1M rows × 30–60 cols × 16 B = 96 MB–960 MB): fits in
RAM on any machine running a media cataloger. Even worst case (1M ×
60 cols = 960 MB) is fine on 16–32 GB machines. Pipeline-level
batching (process 100K rows at a time, call Pipeline::run() multiple
times) handles extreme cases without library-level spill.

ECS game engine (10M entities × 50 component types × 16 B = 8 GB):
spill is actively harmful — NVMe random read latency (~100 µs) would
consume 0.6% of a 16.6 ms frame budget per page-in. Games are latency-
bound; disk I/O in the hot loop is unacceptable.

**Decision: No spill support. No library-enforced memory ceiling.**

Spill-to-disk is the wrong solution for every use case: polka-dots
doesn't need it, saalis fits in RAM, game engines can't tolerate disk
latency. The consumer controls allocation via ColumnStorage — if they
want a ceiling, they return Err from allocate().

**Design-for: column lifetime notification.** The library should signal
when a column is dead (no remaining chain reads from it). The Schedule
already knows the full dependency graph, so it can compute column
liveness statically. Addition to ColumnStorage:

```rust
fn release(&mut self, column: usize);
```

Advisory — the implementation can ignore it or use it to free/reuse
memory. Impact at scale: saalis saves ~50% peak memory by freeing
upstream-only columns; ECS at 10M entities needs this to fit in RAM.

### Design-for notes from scale analysis (saalis + ECS)

The Q2 discussion explored what happens when this pipeline engine is
reused at larger scales: saalis (200K–1M rows, media cataloging) and
a hypothetical ECS game engine (10M entities, 60fps). Four design
implications surfaced that affect later questions:

**DF1. Dispatch overhead scales with morsel count.** At 10M entities
with 600-row morsels, you get ~16,667 morsels per system. Even 1 µs
per dispatch = 16.7 ms — the entire frame budget. At this scale,
lock-free dispatch (Q8) stops being optional, and two-level granularity
(Q5) becomes necessary — larger scheduling morsels with L1-fitting
inner chunks, exactly like DuckDB's morsel/vector split. For polka-dots
(2–4 morsels), dispatch overhead is irrelevant. The design should not
prevent either approach.

**DF2. Intra-chain parallelism needed at scale.** polka-dots uses
single-core-per-chain because 2–4 morsels don't benefit from multiple
cores on the same chain. At 16,667 morsels, you absolutely want 8
cores processing the same system's morsels in parallel — morsel-driven
dispatch within a chain (Q4), with ordering guarantees if determinism
is required (Q6). The design should treat single-core-per-chain as a
policy choice of the Executor, not a hard constraint of the model.

**DF3. Column lifetime notification.** Documented in Q2 above. The
library signals ColumnStorage when a column is no longer needed by
any remaining chain. Essential for ECS-scale memory (8 GB → ~2–3 GB
peak with lifetime management), useful for saalis (~50% peak reduction),
irrelevant for polka-dots.

**DF4. Consumer-side batching and archetypes.** At ECS scale, the
consumer manages data granularity: archetypes mean separate Schedules
per component set, not one giant Schedule for all entities. Pipeline-
level batching (multiple Pipeline::run() calls on subsets) handles
saalis-scale data. The library doesn't need to know about archetypes
or batching — it just runs the Schedule it's given. But the design
must not assume a single all-encompassing run.

### Design-for notes from OS/compiler pipeline comparison

**DF5. LLVM pass manager validates fused chains and column lifetime.**
LLVM's pass pipeline is structurally isomorphic to our model: passes
declare read/write analysis dependencies (our WorkUnit Read/Write),
passes are fused into function-level pipelines that process all passes
on one function before moving to the next (our fused chains processing
all ops on one morsel before the next), and module-level passes are
pipeline breakers (our fan-in chain breaks). LLVM's analysis
preservation/invalidation model is the mature solution to our column
lifetime problem: each pass returns a PreservedAnalyses set, the pass
manager tracks consumer counts per analysis, and analyses are freed
when the last consumer completes. Our ColumnStorage::release(column)
follows the same pattern. The Schedule should pre-compute a consumer
count per column (number of chains that read it) and decrement on
chain completion — release fires at zero.

**DF6. Per-core-type detection, not just global minimum.** macOS GCD
maps QoS classes to core types (P-core vs E-core). Linux EAS (Energy-
Aware Scheduling, kernel 5.18+) uses per-core capacity values from
firmware to model performance asymmetry. Both systems distinguish
core types at the OS level.

Our hardware detection (decision 2) currently takes min L1 across core
types. This is conservative — it sizes morsels for the weakest core.
But a P-core on Apple M2 has 192 KB L1 while the E-core has 64 KB.
A chain running on a P-core could use 3x larger morsels.

The library should detect per-core-type hardware parameters:
- L1 data cache per core type (not just global min)
- Core count per type (e.g., 4 P-cores + 4 E-cores)
- Core type identity (which physical cores are P vs E)

This enables the default Executor to make informed assignments:
- Wide chains (many columns, benefit from larger morsels) → P-cores
- Narrow chains (few columns, smaller morsels sufficient) → E-cores
- Critical-path chains (most downstream dependents) → P-cores
- Leaf chains (no dependents) → E-cores

Morsel sizing becomes per-core-type: a chain assigned to a P-core
uses morsel_size(p_core_l1, chain_cols), a chain on an E-core uses
morsel_size(e_core_l1, chain_cols). The formula is the same, only the
L1 input differs.

This is the same category of concern as physical core count detection
(decision 2, 20) — hardware facts the library owns. Not an Executor
implementation detail.

**DF7. L2 cache utilisation — two-tier partitioning and co-location.**

Currently the design targets L1 only. L2 is unused by design. But L2
presents two concrete opportunities:

**L2 landscape across platforms:**

| Platform | L2 per core | Sharing model |
|----------|-------------|---------------|
| Intel Haswell-Skylake | 256 KB | Private per core |
| AMD Zen 3-4 | 512 KB | Private per core |
| Intel Alder Lake P-core | 1.25 MB | Private per core |
| Apple M1 P-core cluster | 12 MB / 4 cores | Shared among cluster |
| Apple M1 E-core cluster | 4 MB / 4 cores | Shared among cluster |
| Apple M2 P-core cluster | 16 MB / 4 cores | Shared among cluster |

Key difference: L2 is private on x86, shared per cluster on Apple
Silicon.

**Two-tier chain partitioning.** The current break condition is:
union_cols > max_cols_l1 → break. But breaking means two data passes.
For sequential access patterns, L2 throughput is ~80-90% of L1
(hardware prefetcher hides most latency). So one pass at L2 speed may
beat two passes at L1 speed, especially at small row counts where
dispatch overhead dominates.

Proposed policy:
- union_cols ≤ max_cols_l1 → fuse, morsel sized to L1 (optimal)
- union_cols > max_cols_l1 but ≤ max_cols_l2 → fuse, morsel sized
  to L2 (one pass, slightly higher per-access latency, no extra
  dispatch overhead)
- union_cols > max_cols_l2 → break into chains

max_cols_l2 = usable_l2 / (MIN_MORSEL × 16), same formula, different
cache input.

**Dependent chain co-location.** When chain A writes a column and
chain B reads it (pipeline parallelism, B lags one morsel), the data
A wrote is still in A's core's L2 (or shared L2 cluster). If B runs
on a core in the same L2 cluster, B reads from L2 (~5 ns) instead of
L3/DRAM (~15-80 ns). This is a 3-16x latency reduction on the inter-
chain data path.

The default Executor should place dependent chains on cores sharing
L2. Independent chains (no shared columns) can go on any core/cluster.

**At polka-dots scale, everything fits.** 500 rows × 60 columns × 16B
= 480 KB. Fits in one core's L2 on Zen 3+ (512 KB) and trivially on
Apple Silicon (12 MB). The entire pipeline's data stays in L2 across
all chain transitions.

NOTE: latency and throughput numbers here are approximate order-of-
magnitude values, not measured. They should be validated with
benchmarks on target hardware before committing to specific threshold
formulas.

**DF8. Chain priority as a library concept.** GCD's QoS classes and
CFS's vruntime both express priority. Our dispatch priority (decision
22, "unblocks most") is already a form of priority — chains that
unblock the most downstream work run first. This priority should also
inform core-type assignment: high-priority chains (critical path) get
P-cores, low-priority chains (leaf nodes) get E-cores. The Schedule
can pre-compute both dispatch order and core-type preference from the
same dependency graph.

### Multi-dimensional cache tiling model

The design-for notes above (DF1–DF8) address individual concerns:
morsel sizing, chain partitioning, co-location, priority. But they
treat these as separate decisions. In reality, they form a single
multi-dimensional tiling problem — analogous to level-of-detail (LOD)
selection in rendering engines, but with more dimensions than a simple
cache-level hierarchy.

This section formalises the tiling model. It supersedes the simple
"count columns, fit in L1, break if over" model from the memory layout
topic (decisions 9, 12, 13) and replaces it with a richer framework
that accounts for read/write asymmetry, cache hierarchy, column access
patterns, commutativity, and static independence.

#### The LOD analogy

In a rendering engine (Unreal's Nanite, terrain LOD systems, HLOD
hierarchies), the renderer must decide how much geometric detail to
use for each object. The scene is organised as a hierarchy:

- **Root:** the entire scene as a single bounding volume
- **Clusters:** groups of nearby objects
- **Objects:** individual meshes
- **LOD levels:** each mesh at multiple detail levels (high-poly
  for close-up, low-poly for distant)
- **Triangles:** the finest geometric unit

The renderer traverses this hierarchy top-down with a **budget** —
usually screen-space error in pixels. At each node, it asks: "does
this node's approximation meet the budget?" If yes, use this level
(stop descending). If no, descend to children (more detail, more
cost).

The result is a **cut** across the hierarchy — a frontier that
separates "rendered at this level" from "not descended into." The cut
position depends on the budget (camera distance, screen resolution,
GPU capability). The same scene produces different cuts on different
hardware.

Cache tiling is the same problem with different vocabulary. Replace
"geometric detail" with "data locality" and "screen-space error" with
"cache fit":

- **Root:** the entire DAG as one unit
- **Partitions:** independent subgraphs sharing no columns
- **Chain groups:** dependent chains sharing columns via L2
- **Chains:** fused op sequences, one cache-tier working set
- **Morsels:** row batches sized to fit in the chain's cache tier
- **Entries:** individual row × column values

The "budget" is cache capacity. The "cut" is where to partition — at
each level, the tiling algorithm asks: "does this grouping fit in the
available cache?" If yes, keep grouped (fewer passes over data, more
cache utilisation). If no, split (more passes, each cache-fitting).

#### Why "count all columns equally" is wrong

The memory layout topic (decision 9) defined:

```
max_cols = usable_l1 / (MIN_MORSEL × 16)
```

This treats every column identically — read columns and write columns
consume the same L1 budget. But CPU caches handle reads and writes
fundamentally differently, and understanding this difference changes
the partitioning equation.

**How reads work in cache (MESI protocol):**

When a CPU core reads a cache line that it doesn't have locally, the
cache controller fetches it. If no other core has it, the line arrives
in Exclusive state (this core owns it, but hasn't modified it). If
another core already has it in Shared or Exclusive state, the cache
coherence protocol negotiates: the line ends up in Shared state on
both cores. Multiple cores can simultaneously hold the same cache
line in Shared state with zero coherence traffic after the initial
fetch.

For sequential reads (scanning through a column array), the hardware
prefetcher detects the stride pattern and begins fetching cache lines
ahead of the read pointer. On modern hardware, a well-predicted
sequential read stream achieves near-peak memory bandwidth — the
prefetcher keeps the pipeline full so the core rarely stalls waiting
for data. This works at every cache level: L1, L2, and L3.

**How writes work in cache (MESI protocol):**

When a CPU core writes to a cache line, the line must transition to
Modified state. If any other core holds a copy (in Shared or Exclusive
state), that copy is invalidated — the other core receives an
invalidation message and drops its copy. This is cache coherence
traffic: messages on the interconnect between cores.

After writing, the line is Modified in the writing core's L1. It
stays there until eviction (LRU replacement), at which point it's
written back to L2 (and eventually L3/DRAM). Subsequent reads of
this line by the same core are L1 hits. Subsequent reads by OTHER
cores require a coherence event: the Modified line is either written
back to shared cache and downgraded to Shared, or transferred
directly to the requesting core (cache-to-cache transfer, depending
on the architecture).

**What this means for our column model:**

Read columns and write columns have different cache residency needs:

1. **Read-only columns** (loaded by an upstream chain, consumed by
   this chain) can live in any cache level. L2 reads with prefetching
   achieve ~80-90% of L1 throughput for sequential access. L3 reads
   achieve ~50-70%. Multiple chains reading the same column on
   different cores share the cache lines in Shared state — no
   coherence cost, no invalidation.

2. **Write columns** (this chain's output) MUST be in L1 for
   performance. Each write transitions the cache line to Modified
   state. If the write set doesn't fit in L1, write cache lines are
   evicted to L2 prematurely, and subsequent writes to the same
   morsel region miss in L1 and must re-fetch from L2. This is
   write thrashing — vastly more expensive than read overflow to L2,
   because every eviction-then-refetch involves two data movements
   instead of one.

3. **Read-write columns** (read and modified in place by this chain)
   behave like write columns — the first read fetches the line, the
   write transitions to Modified. The line stays in L1 as Modified.
   This is actually the cheapest access pattern per line: one fetch,
   then all subsequent accesses (reads and writes) are L1 hits.

**The revised partitioning constraint:**

Instead of the flat `max_cols` budget, the constraint should be:

```
write_cols × STRIDE ≤ L1_write_budget
read_cols  × STRIDE ≤ L2_read_budget  (overflow from L1 to L2 is OK)
total_cols × STRIDE ≤ L1_write_budget + L2_read_budget
```

Where `L1_write_budget` is the portion of L1 reserved for write
columns (and in-place read-write columns), and `L2_read_budget` is
the L2 capacity available for read-only columns.

Concretely: a chain that reads 20 columns but writes 3 has a write
set of 3 × 16 = 48 bytes per row. On a 128 KB L1 with 0.75
utilisation, the L1 write budget supports 96,000 / 48 = 2,000 rows
per morsel for the write set. The 20 read columns (320 bytes per row)
stream from L2 via prefetcher. Total data per morsel row: 368 bytes,
but only 48 bytes need L1 residency.

Under the old flat model: 23 columns at 16 bytes = 368 bytes per row.
`max_cols_l1 = 96,000 / (64 × 16) = 93` — fits. But on a 32 KB L1:
`max_cols_l1 = 24,000 / (64 × 16) = 23` — barely fits, and could be
wrong because it assumes all 23 columns need L1 residency. In reality,
only the 3 write columns need L1; the chain would be fine on a 32 KB
L1 as long as L2 can serve the read stream.

Under the revised model: the chain fits because 3 write columns × 64
rows × 16 bytes = 3,072 bytes ≪ 24,000 byte L1 budget. The read
columns stream from L2. No chain break needed.

This means fewer chain breaks, fewer data passes, and better overall
throughput — especially on platforms with small L1 but large L2 (x86
with 32 KB L1 and 512 KB L2).

#### Front-back processing and commutativity

A separate dimension of the tiling problem is whether a chain's
processing order matters. This determines what intra-chain parallelism
strategies are available.

**Commutative operations:** An operation is commutative (with respect
to row ordering) when the result of processing row R does not depend
on the result of processing any other row. Each row is independent.
Examples in polka-dots: loading catalogue entries from TOML files
(each entry is independent), auditing individual apps against a
registry (each app's audit is independent), resolving a keybind
configuration (each keybind is independent).

**Non-commutative operations:** An operation is non-commutative when
processing order matters. Examples: running totals (row N depends on
row N-1's accumulated value), sequential override resolution (later
entries override earlier ones), any operation with inter-row state.

For commutative chains, we have a parallelism option that doesn't
require morsel ordering or synchronisation:

**Front-back processing.** Two threads process the same chain's column
arrays from opposite ends:
- Thread A processes morsels from the front: rows 0..M, M..2M, ...
- Thread B processes morsels from the back: rows N-M..N, N-2M..N-M, ...

They work on opposite ends of the same column arrays. Because columns
are contiguous arrays, the two threads access completely different
cache lines — no false sharing, no coherence traffic. They "meet in
the middle" and the chain is done in half the wall-clock time.

This is simpler than full morsel-driven dispatch because:
- Exactly two threads — no dynamic dispatch, no task queue
- Deterministic assignment — each thread knows its morsel range at
  start. Thread A gets the first half, thread B gets the second half.
  No atomic morsel counter needed.
- No ordering concerns — for commutative ops, the output is the same
  regardless of which thread processed which row.

The WorkUnit should declare commutativity as a property:

```rust
impl WorkUnit for LoadCatalogue {
    type Read = ();
    type Write = (In<CatalogueIR, Name>, In<CatalogueIR, Installed>,
                  In<CatalogueIR, Version>);
    const COMMUTATIVE: bool = true;

    fn process_batch(...) -> Result<()>;
}
```

The chain partitioner propagates commutativity: a fused chain is
commutative if ALL of its constituent WorkUnits are commutative. If
any op in the chain is non-commutative, the entire chain is non-
commutative (because ops run sequentially within a morsel, and if
one op's output depends on row order, the morsel order matters).

The Executor uses this property to decide:
- Commutative chain → eligible for front-back processing (2 threads)
  or full morsel-driven dispatch (N threads, at ECS scale)
- Non-commutative chain → single-core sequential processing only

Front-back processing extends naturally to more than two threads:
the row range is divided into N equal segments, one per thread. Each
thread processes its segment independently. This is conceptually
identical to Polars' partition-based parallelism — but only applicable
to commutative chains, and with the advantage that no ordering or
lineariser is needed.

#### The full LOD hierarchy

With read/write asymmetry and commutativity established, the complete
tiling hierarchy has six levels. Each level is a grouping decision
with its own budget constraint and parallelism model.

**Level 0: Entry (single row × single op)**

The finest grain. One value in one column, processed by one operation.
This is the register level — the value is loaded into a CPU register,
the operation executes, the result is stored back.

Budget: CPU register file (typically 16-32 general-purpose registers
on x86/ARM).

Parallelism: SIMD vectorisation. If the operation is the same across
rows (which it is — process_batch applies the same logic to all rows),
the compiler can vectorise: load 2-4 entries per SIMD register
(depending on SIMD width and stride size), process them in parallel.
This is the BLIS micro-kernel level — the innermost computation.

We rely on LLVM autovectorisation here (Q3), not hand-tuned assembly.

**Level 1: Morsel (batch of rows × fused chain)**

A contiguous range of rows, processed by all ops in a fused chain,
back-to-back. The morsel is the unit of cache-fitting: its total
data (morsel_rows × chain columns × 16 bytes) fits in the target
cache tier.

Budget: L1 for write columns + L2 for read columns (the revised
model above). The morsel_size formula uses the write set for L1
sizing and allows the read set to overflow to L2.

Parallelism: within a morsel, all ops run sequentially on one thread.
Between morsels of the same chain, parallelism depends on
commutativity (Level 1 → Level 2 interaction):
- Commutative: morsels are independent, can be processed in parallel
  (front-back or multi-thread)
- Non-commutative: morsels are sequential, one thread only

The morsel is what stays "hot" in L1 during the fused chain's inner
loop. When the loop advances to the next morsel, the current morsel's
write data is evicted to L2 (where the next chain can read it), and
the next morsel's data is fetched into L1 (from L2, via prefetcher).

**Level 2: Chain (fused op sequence)**

A sequence of WorkUnits that run back-to-back on each morsel. The
chain is the unit of fusion — ops within a chain share data through
L1 (the morsel stays in L1 as each op processes it).

Budget: the chain's write set must fit in L1 (one morsel's worth),
and the chain's read set must fit in L2 (or be prefetchable from L2).
The chain break conditions are:
- Write set exceeds L1 write budget → must break
- Total set exceeds L1 + L2 combined → must break
- Fan-in (reads from multiple chains) → must break
- Pipeline breaker (op requires all input before producing output)

This level determines how many ops fuse together and how many passes
over the data are needed. Fewer chains = fewer passes = less total
data movement, but each chain must fit its cache budget.

The chain is also the unit of progress tracking (AtomicUsize counter)
and the unit of error propagation (one chain succeeds or fails as a
whole).

**Level 3: Chain group (dependent chains sharing L2)**

Chains that share columns (one writes, another reads) and benefit
from co-location on cores that share L2 cache. The chain group is the
unit of cache locality across chains — within a group, inter-chain
data handoff goes through shared L2 (~5 ns on Apple Silicon) rather
than L3/DRAM (~15-80 ns).

Budget: the shared L2 capacity of a core cluster. On Apple Silicon,
4 P-cores share 12-16 MB L2 — a chain group's total live column data
should fit in this shared L2 for optimal inter-chain handoff. On x86
where L2 is private, the relevant shared level is L3 (8-24 MB per
socket), but the principle is the same: co-locate chains that share
data on cores that share cache.

Parallelism: pipeline parallelism between chains within the group.
Chain B lags chain A by one morsel. Both run simultaneously on
different cores in the same L2 cluster. A's written data is in
shared L2 when B needs to read it.

The group is the unit of Executor placement — the Executor assigns
an entire chain group to a core cluster, then assigns individual
chains within the group to specific cores in that cluster.

**Level 4: Partition (independent subgraph)**

A connected component of the column-sharing graph. Two chains are in
the same partition if they share any column (directly or transitively).
Chains in different partitions share NO columns — they are statically
independent.

Budget: no cache constraint between partitions. Each partition is
self-contained. Partitions can run on different L2 clusters, different
NUMA nodes, or even different machines — there is zero data
dependency between them.

Parallelism: embarrassingly parallel. No progress counters, no
coordination, no fence checks between partitions. Each partition is
a fully independent sub-pipeline.

How to find partitions: build a graph where nodes are columns and
edges connect columns that appear together in any WorkUnit's Read ∪
Write set. The connected components of this graph are the column
partitions. Each WorkUnit belongs to exactly one partition (determined
by its column set). If all WorkUnits share at least one column
transitively, there is one partition (the common case for polka-dots,
where everything flows through the catalogue).

At polka-dots scale: likely one partition. At saalis scale: movies
and games might be independent partitions. At ECS scale: physics,
rendering, AI, audio are often independent partitions.

**Level 5: Pipeline (full DAG)**

The entire set of WorkUnits submitted by the consumer. This is the
coarsest level — the whole computation.

Budget: total RAM. If the full DAG's column data exceeds available
memory, the consumer must batch: split the work into multiple
Pipeline::run() calls, each with a subset of rows or a subset of
the DAG.

Parallelism: consumer-controlled. The library processes one
Pipeline::run() call at a time. The consumer can run multiple calls
concurrently if they provide separate ColumnStorage instances.

#### Column access compositions

Beyond the read/write distinction, columns within a chain have
different access patterns that affect optimal placement in the cache
hierarchy. These compositions determine how aggressively the tiling
algorithm can assign columns to different cache tiers.

**Read-only shared columns.** A column written by an upstream chain
and read by multiple downstream chains. Example: the catalogue's
Name column is loaded once and read by audit, resolution, and
deployment chains. This column is Shared in the cache coherence
protocol — all reading cores hold copies with zero coherence traffic
after the initial fetch. These columns are ideal candidates for L2/L3
residency: they're read-only, prefetchable, and shared. They should
NOT consume L1 write budget.

**Write-only columns (streaming writes).** A column written by this
chain but not read again until much later (or never — it's a final
output). The write data passes through L1 (each write is an L1 store)
but can be evicted to L2/L3 immediately after the morsel advances.
The column doesn't need to STAY in L1 beyond the current morsel —
the cache eviction policy naturally handles this. These columns
consume L1 write budget but only for one morsel at a time.

**Read-then-write columns (in-place transform).** A column that this
chain reads and modifies in place. The first access per cache line is
a read (fetch to L1 in Exclusive state), subsequent writes modify it
in place (stays in L1 as Modified). This is the most cache-efficient
pattern: one fetch, then all subsequent accesses are L1 hits until
the morsel advances. These columns consume L1 budget but get maximum
benefit from it.

**Dense vs sparse access.** Not all read columns are accessed on every
row. A chain might read column A on every row (dense access — e.g.,
reading every app's name) but column B only for lookups (sparse
access — e.g., checking a condition that's true for 5% of rows, then
reading column B only for those rows). Dense read columns benefit
from L1/L2 residency and prefetching. Sparse read columns gain less
from prefetching (the prefetcher fetches lines that won't be used)
and can tolerate L3 latency because the access is infrequent.

The tiling algorithm could use access pattern hints to refine column
placement. For now, treating all columns as dense sequential access
is correct for our domain (dotfiles operations process every entry).
But at ECS/saalis scale, sparse access patterns (e.g., "only process
entities where Health < 0") are common, and the distinction becomes
meaningful.

**Column temperature summary:**

| Access pattern | Cache requirement | Budget category |
|---------------|-------------------|-----------------|
| Write-only | L1 during current morsel | L1 write budget |
| Read-write (in-place) | L1 during current morsel | L1 write budget |
| Read-only dense | L2 (prefetched), L1 opportunistic | L2 read budget |
| Read-only sparse | L3 tolerable | Minimal budget |
| Read-only shared (multi-chain) | Shared L2/L3, zero coherence | L2 read budget (amortised) |

#### Tiling algorithm sketch (multi-dimensional)

Pulling together all dimensions, the tiling algorithm operates
top-down through the LOD hierarchy:

**Step 1: Identify partitions (Level 4).**
Build the column-sharing graph. Find connected components. Each
component is an independent partition. Partitions are scheduled
with zero coordination between them.

**Step 2: Within each partition, form chains (Level 2).**
Topological sort of WorkUnits. Greedily fuse adjacent ops into
chains, respecting the revised budget constraints:
- Compute the chain's write set (Write columns ∪ ReadWrite columns)
  and read set (Read-only columns).
- Check: write_set × STRIDE ≤ L1_write_budget? If no → break.
- Check: write_set × STRIDE + read_set × STRIDE ≤ L1 + L2 combined?
  If no → break.
- Check: fan-in (reads from multiple chains)? → break.
- Check: pipeline breaker? → break.
- Tag each chain with its cache tier: "L1-only" (everything fits in
  L1) or "L1+L2" (writes in L1, reads overflow to L2).

**Step 3: Within each partition, form chain groups (Level 3).**
Group chains that share columns (one's write is another's read) and
would benefit from shared-L2 co-location. The group is the unit of
Executor placement — all chains in a group should run on cores in
the same L2 cluster.

Grouping algorithm: build a graph where chains are nodes and edges
represent column sharing. Find clusters that fit within an L2
cluster's core count (e.g., 4 cores on Apple Silicon P-core cluster).
If a group has more chains than cores in a cluster, split the group.

**Step 4: Compute morsel sizes (Level 1).**
For each chain, compute the morsel size based on its cache tier and
the core type it will run on:
- L1-only chain on P-core: morsel_size(p_core_l1, total_cols)
- L1-only chain on E-core: morsel_size(e_core_l1, total_cols)
- L1+L2 chain: morsel_size(core_l1, write_cols) — sized so the
  write set fits in L1. The read set streams from L2.

**Step 5: Tag commutativity (cross-cutting).**
For each chain, check if all constituent WorkUnits are commutative.
Commutative chains are eligible for front-back processing or multi-
core morsel dispatch. Non-commutative chains are single-core only.

**Step 6: Executor placement.**
The Executor receives the tiled plan:
- Partition-level: independent partitions assigned to different L2
  clusters (or run sequentially if fewer clusters than partitions).
- Group-level: chain groups assigned to specific L2 clusters, with
  dependent chains on cores that share L2.
- Chain-level: individual chains assigned to specific cores within
  the cluster. Priority chains (critical path) get P-cores.
  Commutative chains may get multiple cores (front-back).
- Morsel-level: morsel sizes pre-computed per chain. The Executor
  drives the dispatch loop using progress counters and the pre-
  computed dispatch table.

#### What this replaces

This multi-dimensional model replaces the following decisions from
the memory layout topic, which were simpler single-dimension models:

- Decision 9: `max_cols = usable_l1 / (MIN_MORSEL × 16)` →
  replaced by split L1 write budget + L2 read budget.
- Decision 12: "chain partitioning minimizes chain count respecting
  max_cols" → refined by the revised budget constraints (write vs
  read) and the two-tier L1/L2 fuse policy.
- Decision 13: "chain breaks at fan-in, max_cols, pipeline breakers"
  → refined: max_cols is now split into write-set-exceeds-L1 and
  total-set-exceeds-L1+L2.
- Decision 14: "morsel size is per-chain" → refined: morsel size
  is per-chain AND per-core-type.

Decisions 9, 12, 13, 14 remain correct in spirit (cache-aware
partitioning, minimize chains, break on structural boundaries) but
the mechanics are richer.

NOTE: the latency numbers, throughput ratios, and cache behavior
descriptions in this section are based on published architectural
documentation and general microarchitecture knowledge. They should
be validated with targeted benchmarks on polka-dots' actual inner
loops before committing to specific threshold formulas or budget
ratios. The structural model (hierarchy, read/write asymmetry,
commutativity) is sound regardless of exact numbers.

#### The morsel as a 3D cube

The tiling model above treats morsels as 2D windows: a row range
(X) times a column set (Y). This is correct for a single exhaustive
pipeline run, where each morsel is processed once and discarded. But
it misses structure in two places:

1. In a **real-time loop** (ECS game at 60fps, a daemon watching for
   config changes, any repeated execution of the same Schedule on
   evolving data), the same morsel is processed on successive
   frames/ticks. The data from the previous frame may still be in
   cache.

2. **Within** the morsel's execution, the fused chain runs a sequence
   of ops, each accessing different columns. The morsel's inner
   structure has an op-sequence dimension that determines which
   columns are active at each moment.

Both of these add a meaningful third dimension to the morsel,
transforming it from a 2D window into a 3D cube. The two dimensions
are distinct (one is "outer time," one is "inner op sequence") but
connected through the cache hierarchy.

##### Outer dimension: temporal extent (T)

In a real-time loop, the pipeline executes repeatedly. Frame 0
processes all morsels. Frame 1 processes all morsels again. The data
evolves between frames (some entities change, most don't), but the
column arrays are at the same memory addresses and the morsel
boundaries cover the same row ranges.

This means the cache state from frame N-1 is physically present in
the cache hierarchy when frame N starts:

**L1** is per-core and small (32-192 KB). Between frames, other work
runs on the same core — other systems, OS scheduler, interrupts. L1
is almost certainly clobbered between frames. A morsel's data from
the previous frame will NOT be in L1.

**L2** is larger (256 KB to 16 MB depending on platform) and is
either per-core (x86) or per-cluster (Apple Silicon). Between frames,
L2 may survive if the total pipeline data is small relative to L2
capacity. For polka-dots (480 KB total pipeline data, vs 512 KB-16 MB
L2), the entire pipeline's data from the previous frame is likely
still in L2 when the next frame starts. For larger workloads, only
the most recently processed morsels survive in L2.

**L3** is larger still (8-24 MB) and shared across all cores on a
socket. L3 is the most likely to retain data across frames, even for
larger workloads.

The morsel's temporal extent T is: **how many frames does this
morsel's data survive in cache?** This is not a fixed number — it
depends on:

- **Cache capacity.** If the pipeline's total data fits in L2, T is
  effectively unlimited (data survives across all frames until
  explicitly evicted). If total data exceeds L2, T is determined by
  LRU eviction: morsels processed most recently survive longest.

- **Core affinity.** If morsel M always runs on core C, core C's
  L1/L2 retains M's data between frames. If M migrates to a
  different core each frame, the new core's cache is cold and T = 0
  for L1/L2 (only L3 might help, if shared). Core affinity is an
  Executor policy: pin morsel-to-core mapping across frames.

- **Morsel boundary stability.** If morsel boundaries are stable
  (same row ranges every frame), the cache lines from the previous
  frame are at the same addresses. If entity count changes between
  frames (entities added or removed, shifting row indices), morsel
  boundaries shift, cache lines don't align, and temporal reuse
  breaks. Stable boundaries require stable entity allocation — a
  concern for the consumer's ColumnStorage implementation, not the
  library.

- **Inter-frame interference.** What other work runs on the same core
  between frames? If the core is idle or doing unrelated work (on
  different memory regions), the pipeline's L2 data survives. If
  another heavy pipeline runs on the same core between frames, L2
  is evicted.

**What the library should provide for temporal reuse:**

1. **Schedule reuse.** The Schedule should be constructable once and
   reusable across frames. Schedule construction (DAG analysis, chain
   partitioning, morsel sizing, tiling) is non-trivial work. In a
   real-time loop, the Schedule is built once (first frame or at
   startup) and reused every frame. Only the data content changes,
   not the data structure. The Schedule's internal state (chain
   assignments, morsel sizes, column indices) is immutable after
   construction.

2. **Morsel-to-core affinity hint.** The Schedule or the Executor
   should support a "sticky" morsel-to-core mapping. On the first
   frame, the Executor assigns morsels to cores (possibly
   arbitrarily). On subsequent frames, the Executor reuses the same
   assignment. The mapping is a simple array: morsel_index →
   core_index. The Executor stores it and reuses it.

3. **Stable morsel boundaries.** The library should not recompute
   morsel boundaries per frame. Given a fixed row count (which is
   stable if entity count doesn't change), morsel boundaries are
   deterministic: morsel 0 = rows [0, morsel_size), morsel 1 = rows
   [morsel_size, 2 × morsel_size), etc. These don't change between
   frames unless the row count changes. If the row count DOES change
   (entities added or removed), morsel boundaries recompute, and
   temporal reuse for affected morsels is lost. This is acceptable —
   structural changes are expected to be infrequent relative to
   per-frame processing.

##### Change detection and incremental execution

The temporal dimension becomes much more powerful with change
detection. If the library tracks which morsels contain data that
changed since the last frame, it can skip unchanged morsels entirely.

**Mechanism: per-morsel generation counter.**

Each morsel has a generation number — a monotonic counter that
increments whenever any row in the morsel's range is modified. The
ColumnStorage (or a separate change-tracking layer) bumps the
generation when a write occurs.

Each chain records the generation it last processed for each of its
input morsels. On the next frame:
- If input morsel generation == last processed generation → skip.
  The output from the previous frame is still valid (in cache AND in
  the output columns).
- If input morsel generation > last processed generation → must
  reprocess. The data has changed.

This is **incremental execution** — only re-process what changed.
The cost of a frame drops from O(total_rows × ops) to O(changed_rows
× ops). For a dotfiles daemon watching for config changes, 99% of
morsels are unchanged between checks. For an ECS game, the fraction
depends on the system — physics changes every entity every frame,
but rendering data (mesh, material) is mostly static.

**Generation propagation through chains.** If chain A produces
output that chain B reads, and chain A's output morsel generation
hasn't changed (chain A was skipped), then chain B's input hasn't
changed either. Change detection propagates through the DAG — an
unchanged root means all downstream chains can be skipped. A changed
root only forces reprocessing of the chains that transitively depend
on the changed columns.

This is the same structure as build systems (Make, Bazel): if a
source file hasn't changed, all downstream build targets that depend
on it can be skipped. Our pipeline's DAG IS a dependency graph, and
change detection makes it an incremental build system.

**The temporal cube, concretely:**

The morsel cube has three dimensions:
- **X (rows):** spatial extent — which entities this morsel covers.
  Sized to fit write columns in L1.
- **Y (columns):** schema extent — which attributes this morsel
  accesses. Determined by chain partitioning.
- **T (temporal):** how many frames this morsel's cached result
  remains valid. Determined by change detection — unchanged morsels
  have T = ∞ (skip processing), changed morsels have T = 0 (must
  reprocess).

For a single exhaustive run, T = 1 for all morsels (every morsel
processed exactly once). The cube degenerates to a 2D window.

For a real-time loop, T varies per morsel:
- Stable morsels (unchanged data, affinitized core): T = many frames.
  Cache-warm, skip-eligible.
- Volatile morsels (frequently changing data): T = 1. Must reprocess
  every frame.

The temporal dimension transforms the pipeline from a batch processor
into an incremental processor. The same Schedule, the same chain
partitioning, the same morsel sizes — but the Executor only dispatches
morsels whose input generations have advanced.

##### Inner dimension: op sequence (Z)

Within a single morsel's execution, the fused chain runs ops
sequentially. Each op accesses a different subset of the morsel's
total column set. At any moment during morsel processing, only one
op's columns are actively being read/written.

**Visualizing the inner structure:**

Consider a chain with three ops and columns {A, B, C, D, E}:

```
Op₁ reads {A, B},    writes {C}
Op₂ reads {B, C},    writes {D}
Op₃ reads {C, D},    writes {E}
```

The morsel's union column set is {A, B, C, D, E} = 5 columns. But
during execution:

```
Z=0 (Op₁ executing): columns {A, B, C} active    (3 cols in L1)
Z=1 (Op₂ executing): columns {B, C, D} active    (3 cols in L1)
Z=2 (Op₃ executing): columns {C, D, E} active    (3 cols in L1)
```

At no point are all 5 columns active simultaneously. The maximum
simultaneously active set is 3, not 5. If L1 can hold 3 columns'
worth of morsel data but not 5, the morsel still fits — as long as
columns cycle through L1 correctly.

This is the Z dimension of the inner cube:
- **x (rows):** same as outer X, or a sub-range (micro-morsel)
- **y (active columns):** the columns this specific op accesses
  (subset of the chain's total column set)
- **z (op index):** which op is executing within the fused chain

**Why Z matters for cache sizing:**

The current model sizes the morsel based on the UNION column set:
`morsel_size(l1, union_cols)`. This is conservative — it ensures the
entire union fits in L1 simultaneously. But as the example shows, the
union is an overestimate of the peak live set.

The peak live set at any Z-slice is:

```
peak_live = max over all ops:
    (op.read_cols ∪ op.write_cols ∪ previous_op.write_cols)
```

The `previous_op.write_cols` term accounts for the fact that the
previous op's output (which the current op might read) should still
be in L1. This is the "handoff" between ops within the chain.

If `peak_live < union_cols`, the morsel can be larger than the union-
based formula suggests, because the cache only needs to hold
`peak_live` columns at any moment, not the full union.

For the example above:
- Z=0: {A, B, C} = 3 (op₁'s reads + writes)
- Z=1: {B, C, D} = 3 (op₂'s reads + writes, C carried from op₁)
- Z=2: {C, D, E} = 3 (op₃'s reads + writes, D carried from op₂)
- peak_live = 3

Under the union model: morsel_size(l1, 5) — sized for 5 columns.
Under the peak-live model: morsel_size(l1, 3) — sized for 3 columns.
That's a 67% larger morsel (fewer morsels, less dispatch overhead).

For chains with many ops that each touch a small subset of columns
(common in ECS — position update, velocity update, collision check,
rendering prep — each touching 2-4 components out of a total of 20+),
the difference between union and peak-live is dramatic.

**Inner tiling (micro-morsels):**

The Z dimension also enables a tiling strategy WITHIN the morsel. If
the chain's union column set doesn't fit in L1 but the peak-live set
DOES, we process the morsel as-is — the cache naturally cycles through
column subsets as ops execute.

But if even the peak-live set doesn't fit in L1 (an extremely wide
chain or very small L1), we can introduce an inner tiling level:

```
// Outer loop: advance through row range in micro-morsels
for row_chunk in morsel.chunks(micro_morsel_size) {
    // Inner loop: run all ops on this micro-morsel
    op₁.process(row_chunk);
    op₂.process(row_chunk);
    op₃.process(row_chunk);
}
```

The micro-morsel is sized so that peak_live columns × micro_morsel
rows × STRIDE fits in L1. The outer morsel can be larger (sized for
L2 or for scheduling overhead). This is the DuckDB two-level model
(morsel for scheduling, vector for L1 fitting) but derived from our
chain structure rather than hardcoded.

This inner tiling is "only relevant within the morsel" — the chain
partitioner, the Executor, and the progress counters all work at
morsel granularity. Only the fused chain's inner execution loop sees
the micro-morsel level.

##### Connecting outer and inner cubes

The morsel's full 3D structure has two nested cubes:

**Outer cube (scheduling level):**
- X: row range — sized by cache tier and column budget
- Y: column set — determined by chain partitioning and read/write
  classification
- T: temporal extent — determined by change detection and core
  affinity

The outer cube is the unit of:
- Dispatch (the Executor assigns outer morsels to cores)
- Progress tracking (AtomicUsize advances per outer morsel)
- Change detection (generation counter per outer morsel)
- Skip decisions (unchanged T > 0 → skip)

**Inner cube (execution level):**
- x: micro-morsel row sub-range — sized so peak-live columns fit L1
- y: active column subset — which columns the current op touches
- z: op index — which op in the fused chain is executing

The inner cube is the unit of:
- Cache fitting (peak-live set in L1 during each z-slice)
- Vectorisation (SIMD across x within each z-slice)
- Column cycling (columns enter/leave L1 as z advances)

For simple pipelines (polka-dots, small ECS), the inner cube
degenerates: the micro-morsel equals the full morsel (everything fits
in L1), z has 2-5 steps, and the inner structure is trivial. No inner
tiling needed.

For complex pipelines (wide chains, large ECS, saalis at scale), the
inner cube becomes important: micro-morsels enable L1 fitting when
the outer morsel is L2-sized, and the peak-live analysis enables
larger outer morsels than the union-based model would allow.

For real-time loops, the outer T dimension enables incremental
execution (skip unchanged morsels) and temporal cache reuse (warm L2
from previous frame). These are the highest-leverage optimisations
for repeated execution — they reduce the per-frame cost from
O(total_data) to O(changed_data), which for typical workloads (few
entities change per frame) is a 10-100x reduction.

#### Volumetric analysis

The 3D cube model isn't just a structural description — it enables
a class of analysis that was not possible with the flat 2D window
model. The Schedule can compute a **volume map** of the pipeline
before execution starts, and the Executor can refine it at runtime
across frames. This analysis informs scheduling, placement,
prediction, and adaptive reorganisation.

The analysis has two phases: static (computed once at Schedule
construction time, zero runtime cost) and runtime (accumulated
across frames in a real-time loop, near-zero per-frame cost).

##### Static analysis: what the Schedule computes at construction

**1. Cache pressure profile per chain.**

For each chain, the Schedule has the full op sequence and each op's
Read/Write column declarations. It can compute the cache pressure at
each Z-slice (each op in the chain):

```
For chain C with ops [op₁, op₂, ..., opₙ]:
  For z in 0..n:
    live_set[z] = op[z].read ∪ op[z].write
    if z > 0:
        live_set[z] = live_set[z] ∪ op[z-1].write   // handoff
    pressure[z] = |live_set[z]| × morsel_size × STRIDE
```

This produces a pressure curve along Z for each chain. The curve
reveals:

- **Peak pressure point:** the Z-slice where the most columns are
  simultaneously live. This determines the true cache budget: the
  morsel must be sized so that the PEAK fits, not the average or the
  union.

- **Pressure transitions:** Z-slices where new columns enter the
  live set (cache-cold fetch required) or old columns leave (cache
  lines can be evicted). These transitions are where L1 misses are
  most likely.

- **Flat regions:** sequences of Z-slices with the same live set
  (consecutive ops accessing the same columns). These are cache-
  stable — no eviction/fetch churn, maximum reuse.

The Schedule uses the pressure profile to:
- Size the morsel based on peak_live, not union (decision 59)
- Identify whether inner tiling (micro-morsels) is needed (decision 60)
- Determine the chain's cache tier (L1-only vs L1+L2)

**2. Data flow volume map.**

The Schedule can compute the total data volume flowing through each
edge in the dependency DAG:

```
edge_volume(chain_A → chain_B) =
    |shared_columns(A.write, B.read)| × total_rows × STRIDE
```

This produces a weighted DAG where edges have byte volumes. The
weights are exact (not estimates) because column counts are known
statically and row counts are known after loaders complete.

The volume map informs chain group formation (Level 3 in the LOD
hierarchy):

- **Heavy edges** (large data volume flowing between chains): these
  chains MUST be co-located on shared-L2 cores. The data handoff is
  large, so each L2 hit (vs DRAM fetch) saves significant absolute
  time. Prioritize these edges in the grouping algorithm.

- **Light edges** (small data volume, few shared columns): these
  chains can tolerate cross-cluster placement. The data handoff is
  small enough that even DRAM fetches are fast.

- **Zero edges** (no shared columns): these chains are in different
  partitions (Level 4). No co-location benefit — schedule
  independently.

The grouping algorithm becomes a **weighted graph partitioning
problem**: given a graph of chains with weighted edges (data volume)
and a constraint (cluster core count), find groups that minimize
cross-group edge weight. This is the same class of problem as
circuit placement in chip design (minimize cross-boundary wire length)
and mesh partitioning in parallel FEM simulation (minimize cross-
partition communication volume). Known algorithms: Kernighan-Lin,
METIS, spectral bisection.

For our scale (2-10 chains), even brute-force enumeration of
groupings is feasible. At ECS scale (dozens of systems), a proper
graph partitioning algorithm would be needed.

**3. Column lifetime map.**

The Schedule can compute the exact lifetime of each column — from the
chain that creates it (allocate) to the last chain that reads it
(release). This is already described in decision 36 (consumer-count
model). The volume map adds byte-level detail:

```
For each column C:
    born_at     = chain that writes C
    dies_at     = last chain that reads C
    live_bytes  = total_rows × STRIDE
    live_frames = dies_at.order - born_at.order  (topological distance)
```

The total live bytes at any point in execution is the sum of
live_bytes for all columns whose lifetime spans that point. The peak
of this curve is the **memory watermark** — the minimum RAM required
to execute the pipeline.

The Schedule can compute the memory watermark statically. If the
watermark exceeds a threshold (consumer-specified, or heuristically
derived from detected RAM), the Schedule can warn the consumer or
suggest pipeline-level batching (process subsets of rows).

**4. Partition independence analysis.**

Finding connected components of the column-sharing graph (decision
48) is itself a form of volumetric analysis: it identifies
disconnected sub-volumes in the (rows × columns × ops) space. Each
partition is a self-contained volume with zero data flow to/from
other partitions.

The Schedule can report partition structure to the consumer:
- Number of independent partitions
- Column set and op set per partition
- Estimated data volume per partition
- Whether partitions can run concurrently (always yes — they're
  independent by definition)

This is valuable for the consumer's own scheduling: if polka-dots
discovers that "process apps" and "process keybinds" are independent
partitions, it can report this to the CLI layer for progress display
("apps: 80%, keybinds: 100%").

##### Runtime analysis: what the Executor learns across frames

In a real-time loop, the Executor accumulates statistics across
frames. This transforms the static volume map into a refined,
data-driven model.

**5. Per-morsel processing time.**

The Executor times each morsel's processing. Over multiple frames,
it builds a per-morsel timing distribution:

```
morsel_times[chain][morsel_index] = [
    frame_0_time, frame_1_time, frame_2_time, ...
]
```

The timing data reveals:
- **Consistent morsels:** similar processing time each frame. These
  are predictable and can be scheduled tightly.
- **Variable morsels:** processing time fluctuates. These may contain
  data-dependent branches (e.g., some rows trigger expensive lookups,
  others don't). Variable morsels should get scheduling slack.
- **Outlier morsels:** occasionally very slow. These may indicate
  cache misses (data evicted between frames), contention, or data
  skew.

**6. Change frequency tracking.**

The generation counter (decision 57) bumps on each write. Over many
frames, the Executor can track per-morsel change frequency:

```
change_rate[chain][morsel_index] =
    changes_in_last_N_frames / N
```

This classifies morsels by temperature:
- **Hot morsels** (change_rate ≈ 1.0): change every frame. Must be
  processed every frame. Benefit from P-core assignment (fast
  processing) and core affinity (warm L2 from processing just last
  frame).
- **Warm morsels** (0.0 < change_rate < 1.0): change occasionally.
  Can be skipped most frames. Benefit from core affinity (when they
  DO need processing, warm L2 helps).
- **Cold morsels** (change_rate ≈ 0.0): rarely or never change. Can
  be skipped almost always. Don't need P-core assignment. Can be
  placed on E-cores or deferred entirely.

The temperature classification drives core-type assignment:
- Hot morsels → P-cores (processed every frame, need maximum speed)
- Cold morsels → E-cores (rarely processed, don't waste P-core
  capacity)
- Warm morsels → either, based on availability

This is the GCD QoS mapping (DF8) refined by runtime data instead of
static priority alone.

**7. Cache residency prediction.**

The Executor knows:
- Which morsels ran on which core this frame
- Each morsel's data size (rows × columns × STRIDE, precomputed)
- Each core's L2 capacity (detected at startup)

From this, it can predict which morsels' data will survive in L2
until the next frame:

```
For core C with L2 capacity L:
    morsels_on_C = [m1, m2, m3, ...]  // in execution order
    cumulative = 0
    for m in morsels_on_C.reverse():   // most recent first
        cumulative += m.data_size
        if cumulative <= L:
            m.predicted_in_l2 = true   // will survive until next frame
        else:
            m.predicted_in_l2 = false  // will be evicted by LRU
```

Morsels predicted to be in L2 benefit from core affinity (keep them
on the same core next frame). Morsels predicted to be evicted gain
nothing from affinity — the Executor is free to move them to a
different core if that improves other placement decisions (e.g.,
load balancing).

This is a simple LRU simulation. It's not perfectly accurate (the
actual cache replacement policy may differ, OS interference evicts
lines unpredictably), but it provides a useful heuristic for
affinity decisions.

**8. Frame time prediction.**

Before frame N+1 starts, the Executor can predict its duration:

```
predicted_frame_time = sum over all changed morsels:
    avg_processing_time[chain][morsel_index]
```

Where "changed morsels" are those whose generation counter advanced
since last processing. The prediction uses the historical average
processing time per morsel (from runtime stat 5 above).

This prediction is directly useful for:

- **Frame budgeting in games.** A game engine's frame budget is
  16.6 ms (60fps) or 33.3 ms (30fps). If predicted_frame_time
  exceeds the budget, the game engine can reduce pipeline scope
  before execution starts: skip non-critical systems, reduce entity
  count, defer cold morsels.

- **Daemon scheduling in polka-dots.** A dotfiles daemon that checks
  for changes periodically can predict "will this check be fast
  (few changes) or slow (many changes)?" and adjust its check
  frequency accordingly. Frequent checks when changes are rare (low
  cost, detect changes quickly). Less frequent checks when many
  changes are happening (avoid overload during bulk operations).

- **Progress estimation.** For a single exhaustive pipeline run (not
  real-time), the Executor can estimate total processing time after
  the first few morsels complete:
  `estimated_total = (elapsed / morsels_done) × total_morsels`.
  The CLI can display a progress bar with ETA.

**9. Throughput trend detection.**

Over many frames, per-morsel timing data reveals throughput trends:

- **Degradation:** processing time increasing over frames. Possible
  causes: memory fragmentation, cache pollution from growing
  background processes, data growth (more entities). The Executor
  can flag this for the consumer.

- **Improvement:** processing time decreasing. Possible cause: data
  becoming more static (change_rate dropping), cache warming up.
  Normal for startup — performance improves over the first few
  frames as caches populate.

- **Periodic patterns:** processing time spikes every N frames.
  Possible cause: periodic background work (OS updates, GC sweeps),
  periodic data changes (scheduled tasks). The Executor can avoid
  scheduling heavy chains during predicted spike frames.

##### Executor reorganisation triggers

The runtime analysis enables the Executor to adaptively reorganise
its execution plan between frames. The Schedule remains immutable
(the DAG, chains, column indices don't change). What changes is the
Executor's mapping of the Schedule to hardware.

**Trigger 1: Affinity rebalance.**

If the Executor detects load imbalance (one core consistently takes
longer than others), it can migrate morsels between cores:

```
If core_C.avg_frame_time > 1.3 × mean(all_cores.avg_frame_time):
    // Core C is overloaded
    Find morsel M on C with lowest affinity benefit
        (predicted_in_l2 = false, or change_rate ≈ 0)
    Migrate M to the lightest core
```

The migration cost (cold cache on the new core) is weighed against
the imbalance cost (the overloaded core is the bottleneck). If the
morsel is cold (rarely processed) or not in L2 anyway, migration is
nearly free.

This is the CFS load-balancing analogy: check imbalance at frame
boundaries, migrate the cheapest-to-move morsel to the lightest core.

**Trigger 2: Chain group re-clustering.**

If the Executor discovers (via edge volume profiling) that the actual
data flow between chains differs from the static prediction —
perhaps because runtime selectivity filters mean fewer rows flow
through certain edges than expected — it can recompute chain groups:

```
If actual_edge_volume(A→B) < 0.1 × predicted_edge_volume(A→B):
    // Very little data actually flows A→B
    // Co-location is less valuable than predicted
    Consider moving A or B to a different cluster for better
    load balance or better co-location with other heavy edges
```

Chain group re-clustering requires re-computing the weighted graph
partitioning. This is more expensive than affinity rebalance but less
frequent — only triggered when actual data flow diverges
significantly from prediction.

**Trigger 3: Morsel size adaptation.**

If the Executor detects that a chain's morsels are consistently
suboptimal:

```
If morsel_processing_time < 2 × dispatch_overhead:
    // Morsels too small — dispatch dominates
    Increase morsel size (double it, re-check L1 fit)

If cache_miss_rate > threshold (via hardware counters or timing):
    // Morsels too large — L1 thrashing
    Decrease morsel size (halve it)
```

Morsel size adaptation is bounded by the cache tier constraints
(MIN_MORSEL, MAX_MORSEL) and the L1/L2 budget. The Executor adjusts
within these bounds.

Note: morsel size changes affect morsel boundaries, which breaks
temporal cache alignment (morsels cover different row ranges than
before). The Executor should only change morsel sizes when the
improvement from better cache fitting outweighs the cost of losing
temporal reuse. This is typically worth it during the first few
frames (calibration), then stabilizing once good sizes are found.

**Trigger 4: Priority re-ranking.**

The initial "unblocks most" priority (decision 22) is static —
derived from the DAG structure at Schedule construction. But runtime
profiling reveals actual chain processing times, which may differ
from the structural prediction:

```
If chain_C.actual_time >> chain_C.predicted_time:
    // Chain C is slower than expected
    // Its dependents are starved — waiting longer than predicted
    Increase C's dispatch priority (start it earlier next frame)

If chain_D.actual_time << chain_D.predicted_time:
    // Chain D is faster than expected
    // Its priority is higher than it needs to be
    Decrease D's dispatch priority (let slower chains go first)
```

Priority re-ranking doesn't change the dependency order (dependents
still wait for their dependencies), but it changes the ORDER in which
ready chains are dispatched when multiple chains are dispatchable
simultaneously.

##### The octree analogy for analysis operations

Each of the analysis operations above maps to a standard operation on
a spatial data structure over the (X, Y, T) volume:

| Analysis operation | Octree / voxel analogy |
|--------------------|----------------------|
| **Static pressure profile** | Building the octree — initial volume structure from the DAG |
| **Runtime profiling** | Refining the octree — adjusting cell densities as real data flows through |
| **Change detection (skip unchanged)** | Empty-space skipping — octree traversal skips empty (unchanged) cells |
| **Frame budgeting** | Level-of-detail selection — reduce "resolution" (skip non-critical chains) when budget is tight |
| **Morsel temperature classification** | Density classification — dense (hot) vs sparse (cold) regions of the volume |
| **Cache residency prediction** | Visibility query — which cells are "visible" (in L2) from the current frame? |
| **Scope reduction (query predicate)** | Frustum culling — only process the sub-volume matching the query |
| **Partition independence** | Disconnected component detection — identify sub-volumes with no mutual interaction |
| **Load rebalancing** | Re-meshing — redistribute volume cells across processing units to balance load |
| **Throughput trending** | Temporal mipmap — coarser analysis at longer time scales reveals trends |

The value of the octree analogy is not that we literally build an
octree data structure (we don't — the volume is described by the
Schedule's DAG and the Executor's runtime stats). The value is that
it provides a vocabulary and a set of known algorithms for the
operations we need:

- Spatial queries ("which morsels are on core C?") → bounding box
  queries on the octree
- Temporal queries ("which morsels changed in the last 5 frames?")
  → temporal range queries
- Budget-constrained selection ("which morsels fit in 10 ms?") →
  LOD selection with cost budget
- Reorganisation ("move morsel M from core A to core B") → cell
  migration between octree nodes

These are well-studied operations with known complexities and
implementations. We can draw on decades of spatial data structure
research for our Executor's adaptive scheduling.

The runtime analysis doesn't apply (T = 1, no frames). But the

#### Matrix methods for pipeline analysis

The pipeline's structure has a natural matrix representation. Every
analysis operation described above — chain formation, partition
identification, chain group formation, op ordering — can be
expressed as a standard operation on a sparse matrix. This connects
our scheduling problem to decades of research in sparse matrix
algorithms, graph partitioning, and numerical linear algebra.

##### The access matrix

Build a matrix A where rows = ops (WorkUnits) and columns = data
columns. Each entry indicates the access type:

```
         col_A  col_B  col_C  col_D  col_E
op_1  [   R      R      W      ·      ·   ]
op_2  [   ·      R      R      W      ·   ]
op_3  [   ·      ·      R      R      W   ]
```

R = read, W = write, · = no access. This is a sparse matrix with
at most (num_ops × num_columns) entries, but typically much sparser
(each op accesses a small subset of all columns).

The access matrix encodes everything the Schedule needs:
- The non-zero pattern determines dependencies (if op_i writes
  column C and op_j reads column C, j depends on i)
- The row structure determines chain formation (consecutive rows
  with overlapping column spans can fuse)
- The column structure determines data flow (columns shared between
  rows carry data between ops)
- The block structure determines partitions (independent blocks on
  the diagonal share no columns)

Every analysis we've described is a matrix operation:

| Pipeline operation | Matrix operation |
|-------------------|-----------------|
| Chain formation | Bandwidth minimization (row/col reordering) |
| Partition identification | Block diagonal permutation |
| Chain group formation | Graph partitioning on the column intersection graph |
| Op ordering for cache reuse | Row reordering to minimize bandwidth |
| Column memory layout | Column reordering for spatial locality |
| Dead column elimination | Structural rank detection |
| Common subexpression detection | Repeated column patterns across rows |
| Dependency DAG | Structural upper triangular form |

##### Immediately applicable technique 1: Cuthill-McKee algorithm

**Problem it solves:** finding an op execution order and column
memory layout that maximizes cache reuse.

**Background.** The Cuthill-McKee (CM) algorithm reorders the rows
and columns of a sparse matrix to reduce its **bandwidth** — the
maximum distance between a row index and its farthest non-zero
column entry. A low-bandwidth matrix has its non-zero entries
clustered near the diagonal. In physical terms: elements that are
"connected" (share non-zero entries) end up near each other in the
ordering.

The reverse Cuthill-McKee (RCM) variant reverses the CM ordering
and tends to produce even smaller bandwidth in practice. Both run
in O(n + nnz) time where n = number of rows and nnz = number of
non-zero entries.

**Applied to our access matrix:**

**Row reordering (ops).** RCM produces an op ordering where ops
accessing similar columns are adjacent. When the chain formation
algorithm then walks this ordered sequence and greedily fuses, it
naturally produces chains with high column reuse between consecutive
ops. The reordering ensures that the greedy algorithm sees ops with
shared columns in sequence, rather than interspersed with unrelated
ops.

Our current chain formation uses topological sort as the ordering.
Topological sort respects dependencies but ignores column sharing —
if two independent ops both access columns {A, B, C}, topological
sort might place them far apart (with other ops in between), while
RCM would place them adjacent (same column span → low bandwidth).

The practical difference: when the DAG has multiple valid topological
orders (common when many ops are independent at the same depth), the
topological sort picks one arbitrarily. RCM picks the one that
maximizes column reuse. For a linear pipeline (A → B → C), they give
the same answer. For a wide DAG with fan-out (A → {B, C, D, E}),
RCM can produce significantly better chain formation.

**Column reordering (memory layout).** RCM applied to the column
dimension produces a column ordering where columns accessed by the
same ops are adjacent in memory. If the ColumnStorage allocates
columns in this order, adjacent columns occupy adjacent cache lines.
When the chain processes its column set, the hardware prefetcher
benefits from spatial locality — it prefetches the next cache line,
which happens to be the next column the chain will access.

Without reordering, column 0 might be "Name" and column 1 might be
"DeployTarget" — accessed by completely different chains. With RCM
reordering, column 0 and column 1 are guaranteed to be co-accessed
by the same chain, so prefetching one brings the other into cache.

**Implementation:** RCM is available in many sparse matrix libraries.
In Rust, the `sprs` crate provides sparse matrix operations. The
algorithm itself is simple enough to implement directly — it's a
BFS traversal of the column intersection graph, starting from a
peripheral node (the node with the highest eccentricity).

Complexity: O(ops + nnz) for both row and column reordering. At our
scale (tens of ops, tens of columns), this is sub-microsecond. At
ECS scale (hundreds of systems, hundreds of component types), it's
still microseconds — negligible compared to Schedule construction.

##### Immediately applicable technique 2: block diagonal detection

**Problem it solves:** identifying independent partitions of the
pipeline (decision 48).

**Background.** A matrix has **block diagonal structure** if it can
be permuted into a form where non-zero entries exist only in
square blocks along the diagonal, with zeros everywhere else:

```
[ B₁  ·   ·  ]
[ ·   B₂  ·  ]
[ ·   ·   B₃ ]
```

Each block Bᵢ represents an independent sub-problem — ops and
columns within the block interact with each other but not with
anything outside the block.

**Applied to our access matrix:**

Finding the block diagonal structure IS partition identification.
Each block is an independent partition. The standard algorithm:

1. Build the bipartite graph of the access matrix: ops on one side,
   columns on the other, edges where entries are non-zero
2. Find connected components of this bipartite graph
3. Each connected component is a block — the ops and columns in
   that component form an independent partition
4. Permute the matrix to group each component's rows and columns
   together → block diagonal form

This is the same algorithm we described for partition identification
(decision 48), but phrased in matrix vocabulary.

**The Dulmage-Mendelsohn decomposition** goes further. It finds the
**fine decomposition** within each block — a further structural
analysis that identifies:

- **Horizontal components:** ops that don't contribute to any output
  column (structurally redundant ops — their write columns are never
  read by any other op). These can be eliminated entirely.
- **Vertical components:** columns that are never written by any op
  (input-only columns from loaders — these are the DAG's roots).
- **Square components:** the core of each block where ops and columns
  are structurally matched.

The fine decomposition gives us dead column elimination for free: if
the Dulmage-Mendelsohn decomposition identifies ops in the horizontal
component, those ops (and their output columns) can be removed from
the Schedule. Their output is never consumed — computing it is
wasted work.

**Implementation:** Available in SuiteSparse's `cs_dmperm` (C),
MATLAB's `dmperm`, and implementable in Rust using the `sprs`
crate's sparse matrix utilities. The algorithm runs in O(n + nnz)
time.

##### Immediately applicable technique 3: spectral graph partitioning

**Problem it solves:** forming chain groups for L2 co-location
(decision 49).

**Background.** Spectral partitioning uses the eigenstructure of a
graph's Laplacian matrix to find partitions that minimize the total
weight of edges crossing between partitions. It's provably near-
optimal for the "normalized cut" objective — a balanced partition
where both halves have similar total weight and the cut weight is
minimized.

**The algorithm:**

1. Build the weighted chain graph: nodes = chains, edge weight =
   data volume flowing between them (|shared_columns| × rows ×
   STRIDE).

2. Compute the graph Laplacian L = D - W, where:
   - W is the weighted adjacency matrix (W[i,j] = edge weight
     between chain i and chain j)
   - D is the degree matrix (D[i,i] = sum of edge weights incident
     to chain i)

3. Find the **Fiedler vector**: the eigenvector corresponding to the
   second-smallest eigenvalue of L. The smallest eigenvalue of L is
   always 0 (with eigenvector = all-ones). The second-smallest
   eigenvalue (the algebraic connectivity) measures how well-
   connected the graph is. Its eigenvector — the Fiedler vector —
   encodes the optimal 2-way partition.

4. Partition by sign: nodes with positive Fiedler vector components
   go in group A, nodes with negative components go in group B.
   This partition approximately minimizes the total data volume
   flowing between groups A and B.

5. For k-way partitioning: apply recursive bisection (split the
   graph in two, then split each half), or compute k eigenvectors
   and apply k-means clustering in the spectral embedding space.

**Why spectral beats greedy:**

Greedy graph partitioning (start with a node, greedily add the
neighbor with the heaviest edge, break when the group is full) finds
locally optimal solutions. Spectral partitioning finds globally
near-optimal solutions because it considers the entire graph
structure simultaneously — the Fiedler vector encodes global
connectivity, not just local neighborhoods.

For simple graphs (linear chains: A → B → C), greedy and spectral
give the same answer. For complex DAGs with fan-out and fan-in
(common at ECS scale where many systems share component types in
non-obvious ways), spectral can find significantly better groupings.

**Example:** Consider 6 chains with data flow:

```
    A ──(100MB)── B ──(50MB)── C
    |                           |
  (10MB)                     (10MB)
    |                           |
    D ──(80MB)── E ──(60MB)── F
```

Greedy starting from A: {A, B, D} (follows heaviest edges).
Spectral: {A, B, C} and {D, E, F} (minimizes the total cut:
10 + 10 = 20 MB instead of greedy's cut which varies but is worse).

The spectral partition places the heavily-connected chains (A-B-C
and D-E-F) together, with only light edges (10 MB each) crossing
the cut. The greedy algorithm's local decisions lead to a suboptimal
global partition.

**At our scale:** For 2-10 chains, the Laplacian is a tiny matrix
(at most 10×10). Computing its eigenvectors is trivial — a few
microseconds. Even a brute-force enumeration of all possible 2-way
partitions (2¹⁰ = 1,024 possibilities) is instant. Spectral methods
become valuable at ECS scale (50+ systems) where brute force is
impractical.

**Implementation:** Eigenvalue computation is available in Rust
via `nalgebra` (dense matrices) or `sprs` + `arpack`-style iterative
solvers (sparse matrices). For our tiny matrices, dense eigenvalue
decomposition via `nalgebra` is simplest.

##### Immediately applicable technique 4: matrix chain ordering

**Problem it solves:** finding the optimal grouping of ops into
chains that minimizes total data passes.

**Background.** The matrix chain multiplication problem: given a
sequence of matrices M₁ × M₂ × ... × Mₙ, find the parenthesization
(grouping of multiplications) that minimizes total scalar
multiplications. The classic dynamic programming solution runs in
O(n³) time and O(n²) space.

**Applied to our chain formation:**

Each op in the topologically-sorted sequence is a "matrix." The
"cost" of fusing ops i through j into one chain is:

```
cost(i, j) = rows × |union_cols(op_i, ..., op_j)| × STRIDE
```

This is the total data volume for one pass through the chain's
column set. The constraint is that the chain must fit in the cache
budget:

```
feasible(i, j) =
    |write_cols(op_i, ..., op_j)| × STRIDE ≤ L1_write_budget
    AND
    |total_cols(op_i, ..., op_j)| × STRIDE ≤ L1 + L2 budget
```

If the grouping (i..j) is not feasible (exceeds cache budget), it
cannot be a chain — it must be split.

The DP recurrence:

```
dp[i][j] = min over all valid split points k in i..j:
    dp[i][k] + dp[k+1][j]
    where both (i..k) and (k+1..j) are feasible
```

Base case: dp[i][i] = cost of a single-op chain = rows × |op_i.cols|
× STRIDE.

The optimal solution minimizes total data volume across all chains.

**Why DP beats greedy:**

Greedy fusion joins ops left-to-right, breaking only when forced.
Consider three ops where:

```
op_1: cols {A, B, C}          — 3 cols
op_2: cols {D, E, F}          — 3 cols
op_3: cols {A, B, C, D, E, F} — 6 cols
```

Cache budget: max 5 columns per chain.

Greedy: fuse op_1 + op_2 → union {A,B,C,D,E,F} = 6 cols → exceeds
budget → break. Result: chain {op_1}, chain {op_2, op_3} (op_2 + op_3
union = {A,B,C,D,E,F} = 6 → also exceeds → break again). Final:
three chains, three data passes.

DP: considers all groupings. Chain {op_1, op_3} has union
{A,B,C,D,E,F} = 6 → exceeds. Chain {op_2, op_3} has union
{A,B,C,D,E,F} = 6 → exceeds. All pairs exceed — three chains is
indeed optimal here. But in cases where the column overlap pattern
is more nuanced, DP finds groupings that greedy misses.

More realistic example where DP wins:

```
op_1: cols {A, B}       — 2 cols
op_2: cols {B, C}       — 2 cols
op_3: cols {C, D}       — 2 cols
op_4: cols {D, E}       — 2 cols
Budget: max 3 cols per chain.
```

Greedy (left to right): {op_1, op_2} = {A,B,C} = 3 ✓, then
{op_3, op_4} = {C,D,E} = 3 ✓. Two chains. Cost: 3 + 3 = 6 col-
passes.

DP: also finds {op_1,op_2} + {op_3,op_4}. But consider:
{op_1} + {op_2, op_3} + {op_4}: {A,B} + {B,C,D} + {D,E} = 2+3+2 =
7 col-passes — worse.
{op_1, op_2, op_3} = {A,B,C,D} = 4 → exceeds budget. Break required.

In this case greedy happens to match DP. But when the DAG has
multiple valid topological orders and the column overlap pattern is
irregular, DP explores the full solution space. The O(n³) cost is
negligible for n < 100 ops.

**Implementation:** The DP is straightforward to implement. The
feasibility check for each (i, j) range requires computing the
union column set, which can be done incrementally (maintain a
bitset, OR in each new op's column mask). For n ops with c total
columns, the DP runs in O(n³ × c/64) time using 64-bit bitmask
operations for the union computation.

##### Useful but not immediately applicable

The following techniques from matrix algebra and sparse computation
are relevant to our model but not needed for the initial
implementation. They become valuable at scale or when specific
performance problems arise.

**Sparse matrix-vector multiply (SpMV) patterns.**

The execution of one chain on one morsel is structurally similar to
SpMV (sparse matrix-vector multiplication): the access matrix
determines which columns are accessed (the sparse pattern), and the
morsel data is the "vector" being transformed.

SpMV optimisation techniques include:
- **Register blocking:** group nearby non-zero entries into small
  dense blocks (e.g., 2×2 or 4×4) and process each block with
  register-resident computation. In our model, this would mean
  grouping columns that are always accessed together into a
  "column block" that's loaded into registers as a unit.
- **Cache blocking:** partition the SpMV into chunks that fit in
  cache. This IS our morsel model — already applied.
- **SIMD-friendly storage:** reorder non-zero entries so that SIMD
  loads fetch contiguous values. The column reordering from RCM
  partially achieves this.

The SpMV connection is more structural than immediately practical,
because our access pattern is sequential (scan through contiguous
column arrays), not sparse random access. But if we ever add
selection vectors (filtered rows where only a subset of entries are
active), the access pattern becomes genuinely sparse, and SpMV
techniques for sparse iteration would directly apply.

**LU decomposition and triangular solve analogy.**

Our topologically-sorted DAG is already in "upper triangular" form:
each chain depends only on earlier chains (chains with lower
topological index). Executing the pipeline in topological order is
a **forward substitution** — the standard algorithm for solving a
triangular linear system:

```
For i = 1 to n:
    x[i] = (b[i] - Σ_{j<i} L[i,j] × x[j]) / L[i,i]
```

In our terms:
```
For chain_i in topological_order:
    output[chain_i] = process(input[chain_i], outputs_of_dependencies)
```

The analogy goes deeper:

- **Fill-in during LU decomposition** corresponds to fan-in creating
  chain breaks. In LU, eliminating a variable can create new non-zero
  entries (fill-in) in rows that previously had zero entries at that
  column. In our DAG, when two chains merge at a fan-in point, the
  downstream chain may need to read columns from BOTH upstream chains,
  creating a wider column set than either upstream chain had alone.
  This "fill-in" forces a chain break (the union exceeds the budget).

- **Pivoting strategies** in LU (choosing which row to eliminate next
  for numerical stability) correspond to dispatch priority strategies
  in our Executor (choosing which chain to process next for cache
  efficiency). In LU, good pivoting reduces fill-in. In our model,
  good dispatch ordering reduces cache misses.

- **Fill-reducing orderings** (Minimum Degree, Approximate Minimum
  Degree, Nested Dissection) in LU reorder the matrix to minimize
  fill-in during elimination. Applied to our DAG, these orderings
  would reorder ops to minimize the column "blow-up" at fan-in points
  — choosing merge orders that keep the union column set small.

The fill-reducing ordering is the most actionable part of this
analogy. When a DAG has multiple fan-in points with choices about
which branches to merge first, the merge order affects the column
set sizes. A fill-reducing ordering (Minimum Degree) would merge
branches with the smallest column overlap first, keeping intermediate
column sets small. This is a refinement of the chain partitioning
algorithm that our current greedy approach doesn't consider.

**Strassen-like decomposition / common subexpression elimination.**

Strassen's algorithm reduces the number of multiplications in matrix
multiply by computing clever linear combinations of sub-results. The
analogy in our pipeline: if two ops both compute the same function
of the same input column, the computation can be shared.

Concretely, if op_3 and op_5 both need `f(column_A)` — the same
transformation of the same input — the pipeline could compute
`f(column_A)` once into an intermediate column and have both op_3
and op_5 read from it. This eliminates redundant computation.

Detection: look for repeated column patterns across rows of the
access matrix. If two rows have identical non-zero patterns in their
read columns AND the ops are the same function, the computation can
be shared.

In practice, this is **common subexpression elimination (CSE)** — a
standard compiler optimisation. The matrix representation makes it
easy to detect: identical row sub-patterns in the access matrix
indicate potential sharing. The Schedule could run a CSE pass after
DAG construction, inserting shared intermediate columns where
duplicated computation is detected.

This is unlikely to matter at polka-dots scale (few ops, little
redundancy). At ECS scale, where many systems independently compute
the same derived quantity (e.g., "distance to player" computed
separately by AI, audio, and rendering systems), CSE could
eliminate significant redundant work.

**Morton / Hilbert curve ordering.**

Space-filling curves (Morton Z-order, Hilbert curve) provide
orderings of 2D grid elements that maximize spatial locality — nearby
elements in the 2D grid are nearby in the 1D ordering.

Applied to our (morsel_index × chain_index) execution space: instead
of processing all morsels of chain 0, then all morsels of chain 1,
process (morsel, chain) pairs in Z-order or Hilbert-curve order.
This interleaves morsel processing across chains.

The benefit: if chain 0 and chain 1 share columns, processing
morsel 0 of chain 0 then morsel 0 of chain 1 keeps the shared data
in cache. Processing all morsels of chain 0 first evicts the shared
data before chain 1 can use it.

The constraint: dependency ordering. Chain 1 can't process morsel M
until chain 0 has progressed past morsel M (pipeline parallelism).
So the space-filling curve can only reorder INDEPENDENT chains at
the same dependency depth, or dependent chains within the pipeline-
parallel lag window.

For independent chains (different partitions, or siblings in the
DAG), Morton ordering could improve L2 utilisation by interleaving
their morsel processing. The Executor would process:

```
(morsel 0, chain A), (morsel 0, chain B),
(morsel 1, chain A), (morsel 1, chain B),
(morsel 2, chain A), (morsel 2, chain B), ...
```

Instead of:

```
(morsel 0, chain A), (morsel 1, chain A), ..., (morsel N, chain A),
(morsel 0, chain B), (morsel 1, chain B), ..., (morsel N, chain B)
```

The interleaved order keeps both chains' column data warm in L2 if
the combined morsel data of both chains fits. The sequential order
evicts chain A's data from L2 by the time chain B starts (if chain
A's total data exceeds L2).

This is a scheduling optimisation for the Executor, not a structural
change to the Schedule. The dispatch loop already processes "the
next dispatchable morsel" — the Morton ordering simply provides a
different definition of "next" that accounts for cross-chain cache
sharing.

##### Summary: matrix methods as a toolbox

The access matrix is the unifying representation. All four immediate
techniques and all four deferred techniques operate on it:

| Technique | Input | Output | Complexity | When to use |
|-----------|-------|--------|-----------|-------------|
| **RCM reordering** | Access matrix | Op ordering + column layout | O(n + nnz) | Schedule construction — always |
| **Block diagonal** | Access matrix | Partition identification | O(n + nnz) | Schedule construction — always |
| **Dulmage-Mendelsohn** | Access matrix | Fine decomposition + dead column elimination | O(n + nnz) | Schedule construction — always |
| **Spectral partitioning** | Chain weight graph | Chain groups for L2 co-location | O(chains² to chains³) | Schedule construction — when > 5 chains |
| **Matrix chain DP** | Op sequence + budgets | Optimal chain grouping | O(n³ × c/64) | Schedule construction — when > 10 ops |
| **SpMV patterns** | Access matrix | Register/cache blocking hints | Analysis only | When selection vectors are added |
| **Fill-reducing ordering** | DAG + column sets | Merge order minimizing column blow-up | O(n + nnz) | When DAG has complex fan-in |
| **CSE detection** | Access matrix rows | Shared intermediate columns | O(n² × c) | When many ops share identical reads |
| **Morton/Hilbert ordering** | (morsel, chain) grid | Interleaved dispatch order | O(morsels × chains) | When independent chains share columns |

The first three (RCM, block diagonal, Dulmage-Mendelsohn) are cheap
enough to run unconditionally at Schedule construction. They improve
chain formation, identify partitions, and eliminate dead columns with
zero downside.

Spectral partitioning and matrix chain DP are slightly more expensive
but still fast enough for our scale. They should run when the DAG is
complex enough to benefit (more than a few chains/ops).

The deferred techniques (SpMV, fill-reducing, CSE, Morton) are
specialised optimisations for specific scenarios. They should be
implemented when those scenarios are encountered, not speculatively.

#### Intake node placement and unified scheduling

The current two-step scheduler separates concerns:

1. **Schedule** (static): DAG → chains → chain groups → partitions.
   Determines *what* executes in *what order*.
2. **Executor** (runtime): Takes the Schedule, determines *where* —
   which core, which morsel, in what dispatch sequence.

This is a sequential decomposition: first partition, then place. But
there's a question about whether this decomposition loses global
optimality, particularly for **intake nodes** — the DAG roots (zero
in-degree ops) that read raw column data with no upstream dependency.

##### What intake nodes are

Intake nodes are the sources of the pipeline. Every chain starts with
either an intake node (reads from raw storage) or a dependent node
(reads another chain's output). In our DAG:

```
intake_A → op_1 → op_2 → ...   (chain 0)
intake_B → op_3 → ...           (chain 1)
intake_C → op_4 → op_5 → ...   (chain 2)
           ↑
        intake_D (fan-in: D also feeds chain 2)
```

Intake nodes determine the initial data flow. Their placement —
which chain they belong to, which morsel range they process, which
core they run on — sets the initial conditions for the entire
pipeline's cache behaviour. If intake_A and intake_B read overlapping
columns, placing them on the same L2 cluster means their shared data
stays warm. Placing them on different clusters means the shared data
is loaded twice from DRAM.

Currently, chain membership is determined by the chain partitioning
algorithm (greedy or DP, decisions 41, 70), and core placement is
determined by the Executor. Intake nodes are just the first op in
their chain — they get no special treatment. But their placement
has outsized impact because they're the *first* thing that runs, so
they determine what's in cache when everything else starts.

##### The placement problem in 3D space

With the 3D morsel cube model (decisions 52-53), every execution
unit has coordinates:

- **X** (row range): which morsel slice of the data
- **Y** (column set): which columns are active
- **Z** (position): op index within chain (inner z) or chain index
  in dependency order (outer, structural)
- **T** (temporal): which frame / run iteration

Intake nodes occupy the z = 0 position within their chain. Their X
is determined by morsel boundaries. Their Y is determined by which
columns they read. Their T is "every frame" (real-time) or "once"
(batch). So the placement question reduces to:

1. **Which chain should this intake belong to?** (Y + Z placement)
2. **Which core should process this chain's morsels?** (physical
   placement, affects cache residency)
3. **In what order relative to other intakes?** (dispatch sequencing,
   affects cross-intake cache sharing)

These three decisions are currently made independently: (1) by the
chain partitioner, (2) by the Executor, (3) implicitly by
topological sort. A joint optimisation could make better decisions.

##### Why ray marching doesn't map directly

Ray marching works by stepping along a ray through a **continuous
signed distance field (SDF)**, sampling the field at each step to
determine how far to the nearest surface. The algorithm converges
on surface intersections in O(log(1/ε)) steps for smooth fields.

The intuition is attractive: cast a ray from an intake node into the
3D execution space, find the optimal "face" to attach to. But our
space isn't a continuous SDF:

- The morsel cube is a **discrete lattice**, not a continuous field.
  Morsels have integer indices, chains are discrete, ops are discrete.
- The "distance" between an intake and a target isn't geometric — it's
  a function of column overlap, cache state, dependency depth, and
  core affinity. These are graph-theoretic and cache-model distances.
- There's no smooth gradient to follow. Moving an intake from chain A
  to chain B is a discrete jump, not a continuous displacement.

Ray marching solves "find the nearest surface in a continuous field."
We need "find the optimal attachment point in a weighted discrete
graph." Different problem structure.

However, the *spirit* of ray marching — "probe outward from a point,
find what you hit first using a distance heuristic" — does have a
discrete analogue. See the nearest-neighbour placement approach below.

##### What does map: force-directed and graph placement models

Several well-studied approaches handle this kind of discrete placement
optimisation:

**Force-directed placement (spring-electrical model).**

Treat each node (including intakes) as a particle in abstract space.
Define forces:

- **Attraction** between nodes sharing columns: proportional to column
  overlap count. If intake_A reads columns {1, 2, 3} and chain C
  uses columns {2, 3, 4}, the attraction is proportional to
  |{2, 3}| = 2.
- **Repulsion** between all nodes: prevents everything from collapsing
  to one core. Proportional to 1/distance².
- **Gravity** toward assigned core: once a chain has a core assignment,
  gravity keeps its nodes near that physical location.

The system iterates until equilibrium. Intake nodes settle near the
chain groups where their column affinity is strongest. This is
essentially what the spectral partitioning (decision 69) computes
analytically — the Fiedler vector gives the optimal bipartition under
exactly this force model.

For more than two groups, iterative force-directed layout (e.g.,
Fruchterman-Reingold) handles k-way placement. Complexity:
O(n² × iterations) for n nodes, typically converging in 50-100
iterations. At our scale (< 100 ops), this is sub-millisecond.

**Multi-level graph partitioning (METIS-style).**

METIS is the standard for balanced graph partitioning. It works in
three phases:

1. **Coarsening:** Repeatedly merge the two most similar nodes
   (heaviest-edge matching) until the graph has k nodes (where k =
   number of target partitions / cores).
2. **Initial partitioning:** Partition the tiny coarsened graph
   (trivial at this size).
3. **Uncoarsening with refinement:** Un-merge nodes one level at a
   time, running a local refinement algorithm (Kernighan-Lin or
   Fiduccia-Mattheyses) at each level to improve the partition.

The key insight: coarsening captures global structure (which nodes
are strongly connected), while refinement captures local structure
(boundary nodes that could be swapped to reduce edge cut).

Applied to our model:
- Nodes = ops (including intakes)
- Edge weight = column overlap between ops
- Partition target = k cores (or k L2 clusters)
- Balance constraint = roughly equal work per partition
- Refinement objective = minimize cross-partition column traffic

METIS handles intake nodes naturally — they're just nodes in the
graph. Their placement falls out of the global partitioning. If
intake_A shares many columns with ops on core 0, METIS places
intake_A on core 0. If intake_A shares columns with ops on BOTH
core 0 and core 1, METIS decides based on which placement minimizes
total cross-partition traffic.

Complexity: O(n + edges) per coarsening/refinement level, typically
O(n log n) total. Sub-millisecond for < 1000 nodes.

**Simulated annealing on the access matrix.**

Take the access matrix (decision 66), the core topology (P-cores,
E-cores, L2 clusters), and the chain dependency graph. Define a
cost function:

```
cost = Σ (cross_cluster_column_loads × DRAM_latency)
     + Σ (L1_misses × L2_latency)
     + Σ (load_imbalance² × penalty)
```

Start with the greedy chain partitioning as the initial solution.
Randomly perturb: move an op from one chain to another, swap two
ops' chain assignments, move a chain from one core to another.
Accept improvements always, accept degradations with probability
exp(-Δcost / temperature). Cool the temperature over iterations.

This finds near-global-optima but is slower than METIS. Useful
when the cost function has complex interactions that partitioning
doesn't capture — e.g., when temporal reuse (T dimension) affects
placement value.

Complexity: O(iterations × cost_eval), where cost_eval is O(n × c).
For 1000 iterations at our scale, still sub-millisecond.

##### Nearest-neighbour placement: the discrete "ray march"

The closest discrete analogue to ray marching is **greedy nearest-
neighbour placement**:

1. Build the full DAG (all ops, including intakes).
2. Compute the column affinity between every pair of ops:
   `affinity(a, b) = |cols(a) ∩ cols(b)| / |cols(a) ∪ cols(b)|`
   (Jaccard similarity on column sets).
3. Place ops one at a time, in priority order (critical path first):
   - For each unplaced op, find the already-placed op with highest
     affinity (the "nearest surface" in the discrete graph).
   - Place the new op in the same chain / core as its nearest
     neighbour, subject to budget constraints.
   - If placement would violate a constraint (column budget exceeded,
     chain too long), find the next-nearest neighbour.

This is the discrete version of "cast a ray outward, attach to the
first surface you hit." The "ray" is the affinity ranking, and the
"surface" is an existing chain or core assignment. It's greedy, so
it can miss global optima (same weakness as greedy chain
partitioning), but it's simple and handles intakes naturally —
intakes are placed first (high priority, z = 0) and become the
"surfaces" that later ops attach to.

The advantage over the current sequential approach: the current
approach partitions into chains first (ignoring core topology), then
assigns chains to cores (ignoring chain formation choices). The
nearest-neighbour approach makes both decisions together — an op's
chain membership is determined by which core cluster has the highest
affinity, not just which adjacent op has column overlap.

##### Intake compatibility with T and z dimensions

**T dimension (temporal / frame axis):**

Intake nodes are fully compatible. In the real-time loop:

- **Hot intakes** (column changes every frame): should be placed on
  cores where their morsel data has L2 temporal affinity (decision
  55). Even though the data changed, the L2 cache line is still
  allocated — the write updates it in-place rather than loading from
  DRAM. Hot intakes benefit most from morsel-to-core pinning.

- **Cold intakes** (column rarely changes): benefit from change
  detection (decision 57). If the column's generation counter hasn't
  bumped, the intake doesn't execute. Its morsel data may have been
  evicted from L2 — but that's fine, because it won't be read.

- **Warm intakes** (changes some frames): this is where placement
  matters most. A warm intake should be co-located with the ops that
  consume its output, so that when it DOES fire, the output flows
  to consumers through L2 rather than DRAM. The Executor can use
  change frequency (runtime analysis, decision 62) to predict which
  intakes will fire next frame and pre-place their morsels.

The temporal dimension adds a *weight* to intake placement: hot
intakes get stronger affinity to their current core (keep them
where L2 is warm), cold intakes get weaker affinity (placement
doesn't matter much), warm intakes get affinity toward their
consumers (co-locate for when they fire).

**z dimension (op sequence within chain):**

Intake nodes are z = 0 by definition — they're the first op in any
chain. The z dimension doesn't constrain intake *placement*; rather,
intake placement constrains the z dimension for all subsequent ops
in the same chain. If intake_A is placed on core 0, all ops in
intake_A's chain execute on core 0 (or at least in core 0's L2
cluster, per decision 42). The intake's placement propagates forward
through the z axis.

This is actually the key insight: **intake placement determines the
z-axis topology.** The chain extends from the intake (z = 0) through
each subsequent op (z = 1, 2, ...). If we get intake placement
right, the rest of the chain inherits good placement. If we get it
wrong, every subsequent op in the chain suffers.

For fan-in nodes (z > 0 ops with multiple upstream chains), the
placement is overconstrained: they need data from multiple chains,
possibly on different cores. Fan-in placement should favor the
upstream chain with the largest data transfer — place the fan-in on
the same core as its heaviest input, pay the cross-cluster cost for
the lighter input.

##### Should we unify the two-step scheduler?

The question is whether to merge Schedule construction and Executor
placement into a single joint optimisation.

**Arguments for unification:**

- Intake placement has outsized impact on cache behaviour, and the
  current sequential approach doesn't account for core topology when
  forming chains.
- The sequential approach can produce locally optimal chain partitions
  that are globally suboptimal when core topology is considered.
- At our scale (< 100 ops, < 20 chains, 4-16 cores), the joint
  optimisation is cheap enough to run at Schedule construction time.
- The access matrix (decision 66) already captures all the
  information needed for joint optimisation.

**Arguments against unification:**

- Separation of concerns: Schedule is the *logical* execution plan,
  Executor is the *physical* execution. Mixing them makes the
  Schedule hardware-dependent.
- Schedule reuse (decision 54): if the Schedule encodes core
  assignments, it can't be reused on a different machine (different
  core count, different L2 topology).
- The Executor trait is the consumer's extension point (decision 31).
  If placement is baked into the Schedule, consumers lose control.
- The sequential approach produces solutions that are typically
  within 5-15% of global optima for our scale. The improvement from
  joint optimisation may not justify the complexity.

**Middle ground: placement hints.**

The Schedule can compute **placement hints** without making binding
decisions:

```
struct PlacementHint {
    chain: ChainId,
    preferred_cluster: Option<L2ClusterId>,
    affinity_weights: SmallVec<[(ChainId, f32); 4]>,
    intake_temperature: Temperature,  // Hot, Warm, Cold
}
```

The Schedule runs the affinity analysis (column overlap, Jaccard
similarity, spectral partitioning) at construction time and attaches
hints to each chain. The Executor reads these hints when assigning
chains to cores. The hints are hardware-independent (expressed as
relative affinities, not absolute core numbers), so the Schedule
remains reusable.

The default Executor follows hints. A custom Executor can override
them (e.g., a NUMA-aware Executor might use socket topology instead).

This preserves the two-step separation while capturing the benefits
of joint optimisation. The Schedule has the global view (access
matrix, DAG structure, column overlap), the Executor has the local
view (hardware topology, current cache state, runtime data).

##### Practical algorithm: placement-aware Schedule construction

Combining the techniques from the matrix methods section with the
placement concerns:

```
1. Build DAG from WorkUnit declarations.
2. Construct access matrix A (ops × columns).
3. Run Dulmage-Mendelsohn decomposition → partitions, dead columns.
4. Within each partition:
   a. Run RCM reordering → op ordering, column layout.
   b. Run chain partitioning (DP if > 10 ops, greedy otherwise).
   c. For each chain, classify intake temperature:
      - Has generation counter? Check change frequency if available.
      - Multiple consumers? Higher affinity weight.
      - Large column set? Higher placement impact.
   d. Build chain-weight graph (edge = shared column count).
   e. Run spectral partitioning → chain groups.
   f. Compute placement hints per chain:
      - Preferred cluster = chain group assignment.
      - Affinity weights = column overlap with each neighbor chain.
      - Intake temperature from step (c).
5. Attach placement hints to Schedule.
6. Executor reads hints at dispatch time.
```

Steps 1-5 run once at Schedule construction. Step 6 runs every
frame. The hints are stable across frames unless the DAG changes
(new ops added, ops removed), which triggers a full Schedule
reconstruction.

The placement-aware construction adds steps 4c, 4d, 4e, 4f to the
existing Schedule construction pipeline. These are all O(chains²)
or better — negligible overhead for < 20 chains.

**Q3. Per-architecture codegen** — RESOLVED

BLIS hand-optimizes micro-kernels per CPU in assembly. We assumed
"Rust compiler handles it, LLVM autovectorization is sufficient."
Is this true for our inner loop? Should we at least benchmark
whether the compiler autovectorizes our stride processing?

**Background.** BLIS (BLAS-like Library Instantiation Software)
achieves 80-95% of theoretical peak FLOPS on matrix multiplication by
hand-writing the innermost computation — the **micro-kernel** — in
assembly for every CPU microarchitecture. The micro-kernel is
typically 50-200 lines of assembly that performs an `mr × nr`
register-resident multiply-accumulate. Different architectures get
different micro-kernels because register counts, SIMD widths, and
pipeline depths vary:

- Haswell (AVX2): mr=6, nr=8. Uses 12 YMM registers (256-bit) for
  the 6×8 tile, 2 for broadcast. FMA throughput: 2 per cycle, 4-cycle
  latency.
- Zen3 (AVX2): Same mr/nr but different instruction scheduling for
  5-cycle FMA latency.
- Apple M1 (NEON): mr=12, nr=8. ARM NEON has 32 128-bit registers,
  allowing larger tiles.
- Skylake-X (AVX-512): mr=14, nr=16. 512-bit registers double the
  SIMD width.

BLIS's sub-configuration system bundles all variants into one binary
and selects the right micro-kernel at runtime via CPU detection +
function pointer dispatch. The overhead is one pointer indirection per
micro-kernel invocation — negligible because each invocation processes
hundreds to thousands of FLOPs.

The entire BLIS framework (five-loop nest, three packing stages,
cache-oblivious tiling) exists for one purpose: keeping the micro-
kernel's registers and L1 cache fed with data so it never stalls.
This works because matrix multiplication is **compute-bound**: the
bottleneck is FPU throughput, not memory bandwidth. Every wasted
cycle, every register spill, every suboptimal instruction schedule
directly reduces peak performance.

**Why our inner loop is fundamentally different.**

Our inner loop processes one morsel through a fused chain of ops:

```rust
for row in morsel_start..morsel_end {
    for op in chain.ops() {
        op.process_row(columns, row);
    }
}
```

Each op reads some input columns and writes some output columns for a
batch of rows. The operations are consumer-defined (through WorkUnit),
not a fixed mathematical operation like matrix multiply.

Three properties make our case different from BLIS:

1. **Memory-bound, not compute-bound.** Our operations are typically
   simple transforms: read a value, apply a function, write a result.
   The bottleneck is getting data into registers from L1 cache (even
   at L1 speeds, a 64-byte cache line load takes ~4 cycles, while
   processing 4-8 values from that cache line takes 1-2 cycles each).
   This is why our entire morsel/chain/tiling framework focuses on
   cache utilisation rather than compute throughput. Hand-tuned
   assembly would squeeze out at most a few percent here — the
   pipeline is stalled on memory, not ALU.

2. **The inner function is unknown at library compile time.** BLIS
   knows it's computing `C[i,j] += A[i,k] × B[k,j]` — a fixed
   operation that maps to a fixed assembly sequence. Our inner
   function is whatever the consumer's `WorkUnit::execute()` does. We
   can't hand-write assembly for an operation we haven't seen. The
   consumer could implement anything from simple arithmetic to hash
   lookups to string parsing.

3. **The SIMD pattern varies per op.** Some ops are pure column-to-
   column maps (SIMD-friendly: apply identical operation to every
   element). Others are reductions (partially SIMD-friendly: vertical
   accumulation then horizontal reduce). Others have branching (poorly
   SIMD-friendly: conditional logic per element). Others involve
   pointer chasing (not SIMD-friendly at all: hash table probes).
   There is no single SIMD strategy that applies uniformly.

**What LLVM autovectorization does for us.**

LLVM's autovectorizer (Loop Vectorizer + SLP Vectorizer passes)
converts scalar loops into SIMD operations. It handles:

- **Contiguous array access:** `for i in 0..n { out[i] = f(in[i]); }`
  → SIMD load, apply f to vector register, SIMD store. This IS our
  morsel iteration pattern when ops are simple transforms.
- **Reductions:** `sum += a[i]` → vector accumulation with horizontal
  reduce at the end.
- **Conditional writes:** `if a[i] > 0 { out[i] = a[i]; }` → masked
  SIMD (with AVX-512 mask registers or NEON predication).

LLVM autovectorizes effectively when:
- Loop body is simple (no opaque function calls)
- Memory accesses are contiguous and aligned
- Trip count is known or bounded
- No loop-carried dependencies

LLVM autovectorizes poorly when:
- Loop body calls through function pointers or `dyn Trait` (can't see
  the function body → can't analyze for vectorization safety)
- Memory accesses are strided or indirect (gather/scatter)
- Complex control flow in the body (deeply nested branches)
- Cross-iteration dependencies exist

**Our no-dyn rule makes autovectorization work.**

This is a non-obvious but significant advantage of our architecture
decision to ban `dyn` (no dynamic dispatch anywhere). All WorkUnit
dispatch is static and monomorphised — the compiler generates a
separate copy of the chain execution loop for each concrete WorkUnit
type. LLVM can inline the entire call chain from the outer morsel
loop through each op's execute() body, see the full data flow, and
autovectorize.

Contrast with DuckDB: operators are virtual functions (`virtual void
Execute(DataChunk&)` — the C++ equivalent of `dyn WorkUnit`). The
compiler can't see through the virtual call to the operator body, so
it can't autovectorize. DuckDB compensates by manually vectorizing
every operator implementation with explicit SIMD intrinsics. This is
the work that BLIS-style per-architecture kernels address.

We don't pay this cost. Our monomorphised generics give LLVM full
visibility, so the compiler does the work that DuckDB's authors do
manually.

**Our columnar storage is already SIMD-optimal.**

The stride concern would arise if we used row-major storage (Array of
Structs): `struct Row { a: f64, b: f64, c: f64 }; data: Vec<Row>`.
Accessing column `a` would be a strided access (every 3rd element),
which doesn't autovectorize well.

But our columnar storage (Struct of Arrays) means each column is a
contiguous array. Processing column `a` for a morsel is:
`a[morsel_start..morsel_end]` — a contiguous slice. This is the ideal
access pattern for SIMD: LLVM generates aligned vector loads and
stores with no gather/scatter overhead.

**Where autovectorization will realistically fail.**

These are not framework problems — they're inherent to the
operations:

- **String/variable-length columns.** Heap-allocated, accessed via
  pointers. No SIMD for variable-length data. Even DuckDB processes
  most string operations with scalar code.
- **Hash-based lookups.** Pointer chasing, branching on collisions,
  data-dependent memory access. Inherently sequential.
- **Transcendental math.** `sin()`, `exp()`, `log()` — iterative
  algorithms. LLVM can use SVML (x86) or vectorized libm (ARM) for
  partial vectorization, but speedup is limited.
- **Gather patterns.** `B[A[i]]` (column B indexed by values from
  column A). AVX2 has `vgatherdd` but it's 5-10× slower than
  contiguous load. This pattern arises in join-like ops.

In all cases, hand-written assembly wouldn't help. The operations
are inherently not SIMD-friendly. BLIS doesn't face this because
`C += A × B` is always SIMD-friendly by construction.

**Should we provide SIMD helpers in the SDK?**

Some frameworks provide explicit SIMD utilities (DuckDB: `__m256i`
intrinsics, Arrow: SIMD kernels, Polars: delegates to Arrow). We
could provide `polka_sdk::simd` utilities:

```rust
pub fn map_column_simd<T: SimdElement>(
    input: &[T], output: &mut [T],
    f: impl Fn(Simd<T, N>) -> Simd<T, N>
) { ... }
```

This has significant downsides:
- Couples SDK to SIMD width (N varies: 4 for SSE/NEON-128, 8 for
  AVX2, 16 for AVX-512).
- Pushes complexity onto op authors — most want to write
  `output[i] = input[i] + 1`, not explicit SIMD.
- `std::simd` is still nightly-only (as of early 2026). Using it
  forces nightly or adds a crate dependency.
- LLVM already generates the same code from the scalar loop.

Explicit SIMD helpers add complexity without benefit when the
compiler already autovectorizes. If a consumer needs hand-tuned
SIMD for a specific hot operation, they can use Rust's `#[target_
feature]` and `std::arch` intrinsics in their WorkUnit — that's
the consumer's choice, not the library's.

**What the library SHOULD do: constrain the existing abstraction
so vectorization is guaranteed, not hoped for.**

The 15-byte column value const-assert is the precedent. It doesn't
say "please keep values small" — it makes oversized values a compile
error. The same philosophy applies to vectorization: rather than
documenting "write SIMD-friendly code," we make non-vectorizable
patterns structurally impossible through the constraints already
inherent in our design — and make the few remaining gaps explicit.

The critical insight: **our existing design already eliminates most
vectorization killers.** The 128-bit stride, columnar layout, no-dyn
rule, external progress tracking, and process_batch with separate
Read/Write slices were designed for cache efficiency — but they
also happen to be exactly what LLVM's autovectorizer needs.

##### What defeats autovectorization — and what we already prevent

LLVM's loop vectorizer fails on a specific, exhaustive list of
patterns. Here is each pattern and its status in our existing
design:

1. **Opaque function calls.** The compiler can't inline, can't prove
   purity, can't analyze side effects. → **Already eliminated.** The
   no-dyn rule means all WorkUnit dispatch is monomorphised. LLVM
   sees the full call chain from the Executor's morsel loop through
   process_batch() into the op author's code.

2. **Heap allocation in the loop body.** `Box::new()`, `Vec::push()`,
   `String::from()` — the allocator is a global side effect with
   synchronisation. → **Already eliminated.** The 15-byte column
   value constraint (paradigm §11.6) makes heap types (`String`,
   `Vec`, `PathBuf`) compile errors as column values. Column data
   can only contain fixed-size values. Output columns are pre-
   allocated by the framework via ColumnStorage before chain
   execution begins. process_batch() receives slices into already-
   allocated arrays — there is no allocation handle to call.

3. **Non-contiguous memory access.** Indexed reads
   (`data[index_array[i]]`), pointer chasing (`node.next`), strided
   access (AoS layout). → **Already eliminated.** Columnar storage
   means each column is a contiguous array. process_batch() receives
   `&[Stride]` slices — contiguous by definition. The 128-bit fixed
   stride means every element is 16 bytes, sequentially laid out.
   No pointer chasing (15-byte payload can't hold a pointer to heap),
   no strided access (SoA, not AoS), no indirect indexing built into
   the API.

4. **Loop-carried dependencies.** When iteration N depends on N-1's
   result. → **Partially addressed.** Read and Write are separate
   associated types. process_batch() receives `Self::Read` (immutable
   slices) and `Self::Write` (mutable slices). The type system
   prevents accidental read/write aliasing — LLVM can prove the read
   and write slices don't overlap, enabling vectorization. Loop-
   carried state (running totals, sequential overrides) is the op
   author's choice within process_batch — the framework can't prevent
   it, but COMMUTATIVE: bool (decision 45) tells the scheduler when
   it's present.

5. **Complex control flow.** Branches, early exits. → **Not
   directly addressable.** process_batch() gives the op author full
   control over the inner loop. This is deliberate (paradigm §11.4.7:
   "the Executor does not decompose the function further"). Complex
   branching within the body is the op author's domain. However: the
   15-byte Copy values constrain WHAT the branches operate on (no
   heap data, no pointer chasing), limiting the practical complexity.

6. **Interior mutability / shared mutation.** `Cell`, `RefCell`,
   `AtomicU64`. → **Already eliminated (needs explicit constraint).**
   Column values that fit in 15 bytes and are Copy cannot contain
   RefCell (not Copy). Cell<T: Copy> is layout-transparent and
   doesn't defeat vectorization. But the Copy bound should be
   explicit on As<T> — see below.

7. **Non-`Copy` types.** Types with `Drop` require destructor calls
   — opaque function calls that prevent reordering. → **Already
   eliminated (needs explicit constraint).** The 15-byte payload
   constraint effectively requires Copy-like types (no String, no
   Vec, no heap), but the bound isn't explicitly stated. Making
   `As<T>: Copy` an explicit constraint in the Column trait
   formalises what the size constraint already implies.

8. **Misaligned data.** Unaligned SIMD loads are slower (1-2 cycle
   penalty per load) and force scalar prologue/epilogue loops. →
   **Partially addressed.** The 128-bit stride is naturally 16-byte
   aligned (one SIMD lane). Morsel boundaries are multiples of 4
   entries = 64 bytes = one cache line. But the column ARRAY base
   address also needs 64-byte alignment — this should be a
   ColumnStorage requirement.

Score: of eight vectorization killers, our existing design already
eliminates items 1, 2, 3, and the structural parts of 4. Items 6
and 7 need one explicit trait bound (Copy on As<T>). Item 8 needs
one ColumnStorage constraint (64-byte alignment). Item 5 is
inherently the op author's domain.

##### What we need to add: explicit constraints on existing types

The gap is small. We need to make implicit guarantees explicit:

**1. `As<T>: Copy` — the column value must be Copy.**

The existing constraint:
```rust
const _: () = assert!(
    size_of::<T>() <= 15,
    "Column value type exceeds 15-byte stride payload"
);
```

The additional constraint:
```rust
impl<I: IR, T, C> Column<In<I>, As<T>> for C
where
    T: Copy + 'static,
{
    const _SIZE_CHECK: () = assert!(size_of::<T>() <= 15);
}
```

Copy eliminates:
- Drop types (no destructor calls in the inner loop)
- Heap types (String, Vec, Box cannot be Copy)
- Interior mutability (RefCell is not Copy)
- Any type that requires cleanup or indirection

This is not a new constraint — it's making explicit what the 15-byte
size limit already implies in practice. No type larger than 15 bytes
that isn't Copy would be useful as a column value. But making it
explicit lets LLVM see it and lets the compiler reject types that
happen to be small but have Drop impls.

**2. Column array alignment — 64 bytes.**

The ColumnStorage trait should require that allocated column arrays
start at 64-byte aligned addresses:

```rust
trait ColumnStorage {
    /// Allocate storage for a column.
    /// The returned memory MUST be aligned to at least 64 bytes.
    fn allocate(&mut self, column: usize, rows: usize) -> Result<()>;

    /// Slice of column data. Returned pointer inherits the
    /// allocation alignment — if the allocation is 64-byte
    /// aligned, slices starting at morsel boundaries (which
    /// are multiples of 4 entries = 64 bytes) are also 64-byte
    /// aligned.
    fn slice(&self, column: usize, start: usize, end: usize)
        -> &[Stride];
    fn slice_mut(&mut self, column: usize, start: usize, end: usize)
        -> &mut [Stride];
}
```

64 bytes = cache line size = AVX-512 vector width. This ensures:
- SIMD loads are always aligned (no penalty, no scalar prologue)
- Morsel boundaries never split cache lines (morsel = multiple of
  4 entries × 16 bytes = 64 bytes)
- Column arrays start on cache line boundaries (no false sharing
  with adjacent allocations)

The constraint doesn't need a const-assert in the trait — it's a
contract on the implementation. The default ColumnStorage
implementation uses `std::alloc::Layout::from_size_align(size, 64)`
or an arena with 64-byte alignment. A consumer who violates this
gets misaligned loads, not a compile error — but the library's
default implementation guarantees it.

**3. Morsel size is a multiple of 4 entries (already in place).**

The morsel formula already rounds down to multiple of 4:
```rust
clamped & !3 // round down to multiple of 4
```

4 entries × 16 bytes = 64 bytes = one cache line = one AVX-512
vector. This means:
- Every morsel starts and ends on a cache line boundary
- No partial SIMD vector at morsel boundaries (no remainder loop)
- Combined with 64-byte array alignment: every morsel slice is
  itself 64-byte aligned

This is already decided (decision 6) and needs no change.

**4. The stride IS the SIMD lane.**

The 128-bit (16-byte) stride is exactly one SIMD lane in SSE/NEON.
In AVX2 (256-bit), one register holds exactly 2 strides. In AVX-512
(512-bit), one register holds exactly 4 strides. The stride was
chosen for data layout reasons (§1 of the memory layout topic), but
it has a happy consequence: the compiler never needs to handle
partial lanes, cross-lane shuffles, or misaligned elements within a
SIMD register. Each stride maps to one SIMD lane, cleanly.

This is already decided (decision 1) and needs no change. But it's
worth recognising explicitly that the 128-bit stride is a
vectorization enabler, not just a data layout choice.

##### What we get for free from existing decisions

Reviewing the existing design through a vectorization lens reveals
that the design is already a near-complete set of vectorization
guarantees — not by accident, but because the same properties that
make code cache-friendly also make it SIMD-friendly:

| Existing design property | Decision | Vectorization effect |
|--------------------------|----------|---------------------|
| 128-bit fixed stride | 1 | Uniform SIMD lane size |
| 15-byte payload + const-assert | paradigm §11.6 | No heap types in column data |
| Columnar contiguous arrays | paradigm §5.4 | Sequential access, no gather/scatter |
| No-dyn, no-std::any | Paradigm rule | Full monomorphisation, LLVM sees through |
| Separate Read/Write types | §5.7 | Non-aliasing proof for LLVM |
| External progress tracking | 15-18 | No branches in inner loop |
| Morsel = multiple of 4 | 6 | Cache-line aligned boundaries |
| Framework controls morsel sizing | 5, 14 | Trip count known, data fits L1 |
| COMMUTATIVE: bool | 45 | Scheduler knows split/reorder safety |
| process_batch receives slices | §8.12 | Contiguous, bounded, framework-allocated |

The additions needed fall into two categories: things traits can
express, and things they can't.

##### What traits CAN express

**`As<T>: Copy`** — the column value type must be Copy. This is a
trait bound on the Column/As type parameter. It prevents Drop types,
heap types, and most interior mutability. The 15-byte size constraint
already makes this true in practice, but the explicit bound lets the
compiler see it and reject types that are small but have Drop impls.

**`As<T>: 'static`** — no borrowed data in column values. Already
implied by the column storage model (columns outlive any single
process_batch call), but making it explicit prevents lifetime
pollution in the type system.

##### What traits CANNOT express — const-asserts required

There are properties the compiler needs to guarantee vectorization
that Rust's trait system simply cannot express. These must be
enforced via const-assert or derive-macro checks:

**1. Value size fits payload (already exists).**
```rust
const _: () = assert!(
    size_of::<T>() <= 15,
    "Column value type exceeds 15-byte stride payload"
);
```
Already in place (paradigm §11.6). This is the precedent for
everything below.

**2. Value alignment divides stride.**
```rust
const _: () = assert!(
    16 % align_of::<T>() == 0,
    "Column value alignment must divide the 16-byte stride"
);
```
There is no `AlignDivides16` trait. `align_of::<T>()` is a const fn
that returns a compile-time value, so this works as a const-assert.
If T has alignment 8, 4, 2, or 1: passes (16 is divisible). If T
has alignment 16: passes (16 / 16 = 1). If T has alignment 32:
fails — but no 15-byte-or-smaller type has alignment > 16, so this
is a safety net, not a practical restriction.

Why it matters: if a value's alignment doesn't divide the stride,
consecutive values in the column array would have padding gaps
between them, breaking the contiguous layout that SIMD loads expect.
The 16-byte stride must pack values without gaps.

**3. No padding within composite column values.**
```rust
// For a struct used as a column value:
const _: () = assert!(
    size_of::<MyValue>() == /* sum of field sizes */,
    "Column value must have no internal padding"
);
```
Rust's default (`repr(Rust)`) may reorder and pad struct fields.
`repr(C)` gives predictable layout but can still insert alignment
padding between fields. A const-assert on total size vs field sum
catches this.

Why it matters: if a column value has internal padding, SIMD
register loads include garbage bytes in the padding positions. When
the value is extracted from a SIMD lane via shift/mask, the padding
bytes are unpredictable. For most operations this doesn't matter
(the padding is ignored), but for bitwise operations or comparisons
on the full lane, padding can cause incorrect results.

There is no `NoPadding` trait. This must be a const-assert. For
primitive types (u8, u32, f64, etc.) it's trivially true. For
structs, the `#[derive(ColumnValue)]` macro can compute and assert
the sum.

**4. repr(C) layout.**

There is no `ReprC` trait in Rust. The `repr(C)` attribute controls
layout but is not visible to the type system — you can't write
`T: ReprC` as a bound. Two enforcement mechanisms:

a. **Derive macro enforcement.** A `#[derive(ColumnValue)]` macro
   that requires `#[repr(C)]` on the type and emits a compile error
   if it's missing. The derive macro can inspect the item's
   attributes at expansion time. This is the same pattern serde uses
   for `#[derive(Serialize)]` — the macro checks structural
   properties that traits can't express.

b. **Layout stability const-assert.** If we can't mandate a derive
   macro (e.g., for primitive types or third-party types), we can
   assert that the layout matches C rules:
   ```rust
   // Fields at predictable offsets (repr(C) guarantee):
   const _: () = assert!(
       offset_of!(MyValue, field_a) == 0,
       "First field must be at offset 0 (repr(C))"
   );
   ```
   `offset_of!` is stable as of Rust 1.77. Combined with the
   no-padding assert, this verifies repr(C)-equivalent layout
   without requiring the attribute.

Why it matters: Rust's default representation may reorder fields for
packing efficiency. If two copies of the same type have different
field orderings (which can't happen in practice, but the compiler
is free to do in theory), SIMD operations that load multiple values
into one register would see inconsistent layouts. `repr(C)` removes
this uncertainty. More practically, it ensures that `memcpy` of a
column slice is equivalent to copying each value — there are no
hidden invariants maintained by field ordering.

**5. Morsel size is multiple of 4 entries (already in place).**

The morsel formula already rounds down:
```rust
clamped & !3 // round down to multiple of 4
```

4 entries × 16 bytes = 64 bytes = cache line = AVX-512 width. This
is already decided (decision 6) but should be a const-assert on
the computed morsel size:
```rust
const _: () = assert!(
    morsel_size % 4 == 0,
    "Morsel size must be a multiple of 4 entries (64 bytes)"
);
```

**6. Column array base alignment (runtime check).**

This is the one constraint that CANNOT be a const-assert because
it depends on the allocator, which runs at runtime. The
ColumnStorage trait specifies 64-byte alignment as a contract, but
enforcement is a runtime debug_assert in the default implementation:

```rust
fn allocate(&mut self, column: usize, rows: usize) -> Result<()> {
    let layout = Layout::from_size_align(rows * STRIDE, 64)
        .expect("column layout");
    let ptr = unsafe { alloc::alloc(layout) };
    debug_assert!(
        ptr as usize % 64 == 0,
        "Column array must be 64-byte aligned"
    );
    // ...
}
```

For a custom ColumnStorage (arena, mmap), the consumer is
responsible for alignment. The library documents the requirement;
misalignment causes performance degradation (unaligned SIMD loads),
not incorrect results.

##### Summary of constraint enforcement

| Property | Trait bound? | Const-assert? | Why needed |
|----------|-------------|---------------|------------|
| Copy | Yes (`T: Copy`) | — | No Drop/heap in inner loop |
| 'static | Yes (`T: 'static`) | — | No borrowed data in columns |
| Size ≤ 15 bytes | No | Yes (`size_of`) | Fits stride payload |
| Alignment divides 16 | No | Yes (`align_of`) | Clean stride packing |
| No internal padding | No | Yes (`size_of` vs fields) | SIMD lane consistency |
| repr(C) layout | No | Derive macro or `offset_of!` | Predictable field order |
| Morsel multiple of 4 | No | Yes (on computed value) | Cache line / SIMD alignment |
| Array 64-byte aligned | No | `debug_assert!` (runtime) | Aligned SIMD loads |

The first two rows are trait bounds on `As<T>` in the Column
definition. The middle five rows are const-asserts generated by
`#[derive(ColumnValue)]` or by the `#[register]` macro on Column
impls. The last row is a runtime check in ColumnStorage.

##### Column values are newtypes — the type IS the contract

Our architecture has two principles that constrain this design
space tightly:

1. **We only use our own primitives.** Column values are domain
   newtypes like `VersionStr`, `OurBool`, `PackageName` — not raw
   `u32` or `f64`. Every type that carries semantic meaning is a
   newtype (paradigm §4.3). Raw primitives have "no semantic meaning
   beyond raw value" (type taxonomy, paradigm §3). Column values
   always have semantic meaning — they represent domain concepts.

2. **Everything is a hard contract via traits.** Not "a derive macro
   will validate it" — the trait system itself must express and
   enforce the constraints. If the trait bound compiles, the type is
   valid. No runtime checks, no macro-hidden asserts as the primary
   mechanism.

These two principles together mean: column values are newtypes, and
the newtype's trait impls ARE the layout contract. The question
isn't "how do we const-assert properties of arbitrary types" but
"how do we define our newtypes so the trait system guarantees their
properties."

**The problem with const-assert-only enforcement.**

Const-asserts (`assert!(size_of::<T>() <= 15)`) fire at compile
time, but they're not visible in the type system. A function with
`T: Copy` communicates its constraint to callers — the compiler
checks it at every call site and gives clear error messages. A
function with a const-assert inside it just explodes at
monomorphisation with a cryptic message and no indication in the
signature. Const-asserts are a last resort, not a design tool.

The goal: make the column value contract expressible as a trait
bound `T: Payload` where the trait system itself guarantees all
properties — size, alignment, layout, copy semantics — without
any hidden asserts.

**The approach: sealed slot sizes.**

The fundamental constraint is: column payloads must fit in 15
bytes. On stable Rust, you can't write `where size_of::<T>() <= 15`
as a trait bound. But you CAN enumerate the valid sizes and make
them a sealed trait:

```rust
/// A fixed-size byte representation for column payloads.
/// Sealed — only implemented for the sizes that fit in a
/// 15-byte stride payload.
///
/// All slot types are repr(C), Copy, align(1), and have
/// no padding by construction (they're byte arrays).
pub trait SlotSize: Copy + 'static {
    const SIZE: usize;
}

#[repr(C)] #[derive(Copy, Clone)]
pub struct Slot1(pub [u8; 1]);
impl SlotSize for Slot1 { const SIZE: usize = 1; }

#[repr(C)] #[derive(Copy, Clone)]
pub struct Slot2(pub [u8; 2]);
impl SlotSize for Slot2 { const SIZE: usize = 2; }

#[repr(C)] #[derive(Copy, Clone)]
pub struct Slot4(pub [u8; 4]);
impl SlotSize for Slot4 { const SIZE: usize = 4; }

#[repr(C)] #[derive(Copy, Clone)]
pub struct Slot8(pub [u8; 8]);
impl SlotSize for Slot8 { const SIZE: usize = 8; }

#[repr(C)] #[derive(Copy, Clone)]
pub struct Slot12(pub [u8; 12]);
impl SlotSize for Slot12 { const SIZE: usize = 12; }

#[repr(C)] #[derive(Copy, Clone)]
pub struct Slot15(pub [u8; 15]);
impl SlotSize for Slot15 { const SIZE: usize = 15; }
```

SlotSize is sealed — only our crate implements it, only for these
specific sizes. There is no `Slot16`, no `Slot32`. The size
constraint is enforced by the type system: if your payload doesn't
fit any slot, you can't implement Payload. No const-assert needed.

Each slot type is `repr(C)`, `Copy`, has alignment 1 (byte arrays),
and has no internal padding (byte arrays can't have padding). These
properties are guaranteed by construction — `[u8; N]` wrapped in a
`repr(C)` struct IS the simplest possible fixed-layout type. The
compiler can't introduce padding or reorder fields because there's
only one field and it's a byte array.

Alignment divides stride: `align_of::<[u8; N]>()` is 1. `16 % 1
== 0`. Always true. No assert needed.

**The Payload trait.**

```rust
/// A domain value that can be stored in a column's stride payload.
///
/// Column values are always domain newtypes (VersionStr, OurBool,
/// etc.), never raw primitives. The Payload trait connects the
/// newtype to its slot representation.
pub trait Payload: Copy + 'static {
    /// Which slot size this payload uses. Determines the byte
    /// representation in column storage.
    type Slot: SlotSize;

    /// Pack the domain value into its slot representation.
    fn pack(self) -> Self::Slot;

    /// Unpack a slot representation back into the domain value.
    fn unpack(slot: Self::Slot) -> Self;
}
```

The trait is the complete contract:
- `Copy + 'static` — no Drop, no heap, no borrowed data (trait
  supertraits, checked by compiler)
- Size ≤ 15 bytes — guaranteed by `Slot: SlotSize` (sealed trait,
  no slot > 15 bytes exists)
- repr(C), no padding — guaranteed by slot types (byte arrays)
- Alignment divides stride — guaranteed by slot types (align 1)

No const-assert, no derive macro, no `unsafe`. Pure traits.

**Domain newtypes implement Payload:**

```rust
/// Version string stored as 8 bytes (e.g., semver encoded).
#[repr(C)]
#[derive(Copy, Clone)]
pub struct VersionStr([u8; 8]);

impl Payload for VersionStr {
    type Slot = Slot8;

    fn pack(self) -> Slot8 {
        Slot8(self.0)
    }

    fn unpack(slot: Slot8) -> Self {
        Self(slot.0)
    }
}

/// Boolean with domain meaning (not raw bool).
#[repr(C)]
#[derive(Copy, Clone)]
pub struct OurBool(u8);

impl Payload for OurBool {
    type Slot = Slot1;

    fn pack(self) -> Slot1 {
        Slot1([self.0])
    }

    fn unpack(slot: Slot1) -> Self {
        Self(slot.0[0])
    }
}
```

Each newtype declares its slot — the compiler checks that the slot
is a valid SlotSize. The pack/unpack functions are the canonical
conversion between the domain type and its byte representation.
These are trivial for simple newtypes (byte copy), more involved
for encoded types (e.g., a VersionStr that encodes major.minor.patch
into 8 bytes).

**No raw primitives as column values.**

This follows directly from the newtype mandate (paradigm §4.3).
You don't store `u32` in a column — you store `PackageCount(u32)`.
The newtype carries the semantic meaning AND implements Payload.

There are no blanket `impl Payload for u32` or `impl Payload for
f64`. Raw primitives don't implement Payload. If an extension needs
a u32 column, it defines a newtype:

```rust
#[repr(C)]
#[derive(Copy, Clone)]
struct DependencyCount(u32);

impl Payload for DependencyCount {
    type Slot = Slot4;
    fn pack(self) -> Slot4 { Slot4(self.0.to_ne_bytes()) }
    fn unpack(slot: Slot4) -> Self { Self(u32::from_ne_bytes(slot.0)) }
}
```

This is the newtype mandate applied consistently. The boilerplate
is small (5 lines for the Payload impl), and a derive macro can
reduce it further — but the derive is a convenience, not the
enforcement mechanism. The trait IS the contract.

**How Payload connects to Column and WorkUnit.**

The `As<T>` type parameter in `Column<In<IR>, As<T>>` requires
`T: Payload`:

```rust
impl<I: IR, T: Payload, C> Column<In<I>, As<T>> for C { ... }
```

Any generic operating on column data uses `T: Payload` as the
bound. The bound propagates through the type system:

```rust
fn morsel_size<T: Payload>(l1: usize, cols: usize) -> usize {
    // T::Slot::SIZE is known at compile time via the trait
    let stride = 1 + T::Slot::SIZE; // 1 flag byte + payload
    let usable = (l1 as f64 * 0.75) as usize;
    (usable / (cols * stride)).clamp(64, 8192) & !3
}
```

Wait — this reveals something. The current design uses a UNIFORM
16-byte stride for ALL columns. But if Payload has a type-level
slot size, each column could have a DIFFERENT stride (1 + Slot::SIZE
bytes per entry). A `OurBool` column would use 2 bytes per entry
(1 flag + 1 payload), not 16 bytes.

This is a design tension. There are two options:

**Option A: Uniform 16-byte stride (current design).**

Every column entry is 16 bytes regardless of payload size. An
OurBool (1 byte payload) wastes 14 bytes per entry. A VersionStr
(8 bytes) wastes 7 bytes. The advantage: stride is constant, SIMD
loads are uniform, morsel sizing is simple (cols × 16 per row).

The slot size still exists in the trait — it determines how much of
the 16-byte stride is meaningful. But storage is always 16 bytes.
pack/unpack handle the padding:

```rust
/// The stride representation. Always 16 bytes.
#[repr(C)]
#[derive(Copy, Clone)]
pub struct Stride {
    flags: u8,
    payload: [u8; 15],
}

impl Stride {
    fn write<T: Payload>(&mut self, value: T) {
        let slot = value.pack();
        // Copy Slot::SIZE bytes into payload[..Slot::SIZE]
        // Remaining bytes are undefined (not read)
        self.payload[..T::Slot::SIZE]
            .copy_from_slice(&slot.as_bytes());
    }

    fn read<T: Payload>(&self) -> T {
        let mut slot_bytes = [0u8; T::Slot::SIZE];
        slot_bytes.copy_from_slice(&self.payload[..T::Slot::SIZE]);
        T::unpack(T::Slot::from_bytes(slot_bytes))
    }
}
```

**Option B: Variable stride per column type.**

Each column has stride = 1 + T::Slot::SIZE. Denser packing, less
wasted cache space. A 5-column morsel with (Slot1, Slot4, Slot8,
Slot1, Slot2) uses 2+5+9+2+3 = 21 bytes per row instead of 80.
This is 4× denser.

The downside: stride varies per column, complicating SIMD patterns
(can't load N entries with a single aligned vector load unless all
entries have the same size), morsel sizing formula must account for
variable strides, and the Executor needs per-column stride metadata.

**This is a significant design question that deserves its own
analysis.** The current design (decision 1 from the memory layout
topic) established the 128-bit uniform stride. The Payload trait
with typed slots gives us the information to revisit this — we now
know at compile time how large each column's values actually are.
Whether to use that information for variable strides or keep the
uniform approach is a separate decision that connects to the stride
encoding in paradigm §11.5.

For now, the Payload trait works with EITHER option:
- Uniform stride: Slot determines which bytes are meaningful within
  the fixed 16-byte entry
- Variable stride: Slot determines the column's entry size

The trait structure is the same either way. The slot IS the
contract regardless of how storage handles padding.

**Why this approach satisfies both principles.**

1. **Our own primitives:** Every column value is a domain newtype.
   Raw u32/f64/bool never appear as column types. The newtype
   carries the meaning, the Payload impl carries the layout
   contract.

2. **Hard contracts via traits:** The entire constraint chain is
   traits: `Payload: Copy + 'static`, `Slot: SlotSize`. No
   const-asserts needed for size (sealed slot set), alignment
   (byte arrays), padding (byte arrays), or Copy/lifetime. The
   only place a const-assert COULD still live is in the Stride
   itself:

   ```rust
   const _: () = assert!(size_of::<Stride>() == 16);
   ```

   This asserts a property of OUR type (Stride), not of an
   arbitrary T. It's a sanity check on our own code, not a
   constraint on users.

**A derive macro is a convenience, not the enforcement.**

A `#[derive(Payload)]` could generate the pack/unpack boilerplate:

```rust
#[derive(Payload)]  // generates impl Payload with slot inference
#[repr(C)]
struct VersionStr([u8; 8]);
// → Slot inferred as Slot8 from size_of::<[u8; 8]>()
// → pack/unpack generated as trivial byte copy
```

But this is OPTIONAL. The trait impl can always be written by hand.
The derive reduces boilerplate; it doesn't enforce invariants. The
invariants are enforced by the trait system itself (sealed SlotSize,
Copy supertrait, 'static supertrait).

##### The process_batch contract — what the op author controls

The existing design (paradigm §11.4.7) deliberately gives the op
author control of the inner loop:

> "Within process_batch(), the extension author has direct access to
> input and output slices. They can iterate row by row, use SIMD, or
> apply any transformation. The Executor does not decompose the
> function further — it controls WHEN the function runs and WHICH
> rows it processes."

This is correct. The framework controls the outer loop (morsel
boundaries, dispatch order, core assignment), and the op author
controls the inner loop (what computation to perform on the slices).
The framework doesn't need to control the inner loop because the
constraints on what enters the inner loop are tight enough:

- The slices contain only Copy, 15-byte, fixed-stride values
- The slices are contiguous and 64-byte aligned
- The slices are bounded (morsel_size elements)
- Read and write slices are type-distinct (non-aliasing)
- No fence entries, no sentinels, no metadata inline

A simple process_batch implementation:
```rust
fn process_batch(read: Self::Read, write: Self::Write) -> Result<()> {
    let (installed, version) = read;
    let (latest, outdated) = write;

    for i in 0..installed.len() {
        latest[i] = resolve_latest(version[i]);
        outdated[i] = installed[i] != latest[i];
    }
    Ok(())
}
```

LLVM sees: two contiguous reads, two contiguous writes, no aliasing,
no allocation, no function pointers, no loop-carried dependency,
Copy values. It vectorizes this automatically.

An op author who introduces loop-carried state:
```rust
fn process_batch(read: Self::Read, write: Self::Write) -> Result<()> {
    let (values,) = read;
    let (running_sum,) = write;
    let mut acc = 0u64;
    for i in 0..values.len() {
        acc += values[i].as_u64();
        running_sum[i] = acc.into();
    }
    Ok(())
}
```

This won't fully vectorize (loop-carried dependency via `acc`).
But the framework already knows: this WorkUnit declared
`COMMUTATIVE: bool = false`. The scheduler handles it correctly —
single core, no splitting, no front-back. The vectorization loss
is accepted because the operation inherently requires sequential
processing. No trait hierarchy needed to express this — COMMUTATIVE
already communicates it.

##### Compiler hints beyond constraints

Beyond the hard constraints (Copy, alignment, stride), there are
softer things the library can do to help the compiler:

**1. `#[inline(always)]` on the morsel dispatch loop.**

The Executor's inner morsel dispatch — the code that calls
process_batch — should be `#[inline(always)]`. This ensures LLVM
sees the process_batch body in the same compilation unit as the
morsel loop, enabling cross-function vectorization. Without inlining,
LLVM treats process_batch as an opaque call and can't vectorize
across the call boundary.

Since process_batch is monomorphised (no dyn), the compiler CAN
inline it — but it might choose not to for code size reasons.
`#[inline(always)]` forces the issue.

**2. `#[repr(C)]` on column value types.**

Column values should be `repr(C)` to ensure predictable layout. Rust
doesn't guarantee struct field ordering (it may reorder for packing).
`repr(C)` means the layout matches what LLVM expects — no surprises
when loading/storing values through SIMD registers.

This can be enforced by requiring `repr(C)` on the As<T> type:
```rust
// The #[register] macro on Column<In<IR>, As<T>> checks:
const _: () = assert!(
    std::mem::size_of::<T>() <= 15,
    "Column value type exceeds 15-byte payload"
);
// repr(C) can't be const-asserted, but can be verified by the
// derive macro at compile time by checking layout stability.
```

Alternatively, column values could be required to be primitive
types or `#[derive(ColumnValue)]` types where the derive macro
enforces `repr(C)`.

**3. Slice pointer annotations.**

In Rust, `&[T]` and `&mut [T]` are already non-aliasing (the borrow
checker guarantees this). But in the monomorphised code, LLVM may
not always prove non-aliasing across multiple slice parameters. The
`#[restrict]` equivalent in Rust is the existing borrow rules — but
for extra safety, the Read and Write types being separate associated
types (not two slices of the same type) gives LLVM additional type-
based aliasing information.

**4. Loop structure within process_batch.**

The framework can provide utility functions that encode the optimal
loop structure:

```rust
/// Process each row in a morsel. This function is designed to
/// autovectorize — it iterates over contiguous slices with a
/// known trip count and no loop-carried state.
#[inline(always)]
pub fn for_each_row<R: ColumnSlices, W: ColumnSlicesMut>(
    read: &R, write: &mut W,
    f: impl Fn(R::Row, W::RowMut),
) {
    for i in 0..read.len() {
        f(read.row(i), write.row_mut(i));
    }
}
```

This is a convenience, not a requirement. Op authors who use
`for_each_row` get vectorization-friendly loop structure
automatically. Op authors who write their own loop get the same
outcome as long as they iterate sequentially — which is the natural
thing to do with slices.

The key difference from the MapUnit/FuseUnit approach: this doesn't
change the WorkUnit trait or introduce new traits. It's a utility
function in the SDK that op authors can optionally use. process_batch
remains the single entry point. The framework doesn't control the
inner loop — it provides a helper for the common case.

##### Summary: the existing design is the vectorization contract

No per-architecture codegen. No new trait hierarchy. The existing
design — 128-bit stride, Copy values, columnar contiguous storage,
no-dyn, separate Read/Write, external progress tracking, COMMUTATIVE
marker — already constitutes a near-complete set of vectorization
guarantees.

Trait-expressible additions:
- `As<T>: Copy + 'static` on the Column trait

Const-assert additions (enforced via `#[derive(ColumnValue)]` or
`#[register]` expansion):
- Value size ≤ 15 bytes (already exists)
- Value alignment divides 16
- No internal padding
- repr(C) layout (derive macro attribute inspection)
- Morsel size multiple of 4 entries

Runtime enforcement:
- Column array 64-byte alignment (`debug_assert!` in ColumnStorage)

Compiler hints:
- `#[inline(always)]` on morsel dispatch loop
- Optional `for_each_row` utility for op authors

`#[derive(ColumnValue)]` is the enforcement point for everything
traits can't express. Primitives get blanket impls.

**Revised decision: No per-architecture codegen. The existing
abstraction IS the vectorization contract. Tighten it with explicit
bounds (Copy), const-asserts (alignment, padding, repr(C)), and
runtime checks (array alignment). No new traits needed.**

**Q4. Partition-based vs morsel-driven parallelism** — RESOLVED

Polars pre-partitions data across N pipeline instances (one per
thread). We assumed morsel-driven dispatch (closer to DuckDB).
Should we explicitly confirm this choice? What are the trade-offs
at our scale?

**Answer: hybrid — partition-based at the macro level, deterministic
segment-based within chains, with commutativity gating parallelism.**

The tiling model (decisions 45–51) already resolved this. The design
is neither pure Polars nor pure DuckDB — it takes the best of both:

**1. Partition-level (Polars-like): embarrassingly parallel.**

Decision 48 defines partitions as connected components of the
column-sharing graph. Independent partitions run on different L2
clusters with zero coordination. This is exactly Polars'
pre-partitioning — but derived structurally from the access matrix,
not manually specified.

At polka-dots scale: usually one partition (everything shares the
catalogue). At saalis/ECS scale: multiple independent partitions
emerge naturally.

**2. Chain-level (DuckDB-like): dispatch-table driven.**

Within a partition, chains execute according to the pre-computed
dispatch table (memory layout §7.5-7.6). The Executor walks the
table, checks progress atomics, and dispatches ready morsels. This
is morsel-driven dispatch — but with a static dispatch table rather
than DuckDB's dynamic task queue.

**3. Morsel-level: deterministic segment assignment, not work-stealing.**

For commutative chains (decision 45), the Executor can assign
morsel segments to multiple cores — front-back processing or N-way
split. Each core gets a deterministic, pre-computed row range. No
atomic morsel counter, no contention, no work-stealing.

For non-commutative chains: single-core sequential. One core
processes all morsels in order.

**Why not pure work-stealing (DuckDB model)?**

DuckDB's dynamic morsel dispatch makes sense at 10M+ rows where
load imbalance across cores matters. At our scale (hundreds to low
thousands of rows, 2-4 morsels per chain), the overhead of a shared
atomic counter or task queue exceeds any load-balancing benefit.
Deterministic segment assignment means zero dispatch contention.

Design-for note: **DF9.** If a future consumer has highly variable
per-row processing time (e.g., one row takes 100× longer than
another), deterministic segmentation will cause load imbalance. The
Executor trait allows a work-stealing implementation — but the
default library Executor should use deterministic assignment.

**Decision: hybrid parallelism. Partition-based across independent
column groups. Dispatch-table driven within partitions. Deterministic
segment assignment for multi-core chains. Work-stealing is Executor-
implementor's option, not the default.**

### From gaps identified during the memory layout discussion

**Q5. Two-level granularity (scheduling unit vs cache-fitting unit)**
— RESOLVED

DuckDB has morsel (122,880 rows) for thread scheduling AND vector
(2,048 rows) for L1-fitting. We use one level — the morsel is both.
At our data volumes (hundreds to low thousands of rows), we may
have only 2-4 morsels per chain. Is the single-level model correct
for us, or should we separate scheduling granularity from cache
granularity?

**Answer: two levels exist — morsel (L1 scheduling unit) and
micro-morsel (inner tiling when peak-live exceeds L1). Decision 60
already established this.**

The morsel is the scheduling unit — the Executor dispatches work
in morsel-sized chunks. The morsel is sized to fit the chain's write
set in L1 (decision 43: write set must fit L1, read set can overflow
to L2).

The micro-morsel (decision 60) is the inner tiling unit. When a
chain's peak-live column set exceeds L1, the morsel is subdivided
into micro-morsels that DO fit L1. This is analogous to DuckDB's
vector-within-morsel — but activated conditionally, only when cache
pressure demands it.

At polka-dots scale (< 20 columns per chain, 128-byte morsel
column footprint): micro-morsels are almost never needed. The morsel
IS the L1-fitting unit. At ECS scale (hundreds of columns):
micro-morsels activate automatically.

**Why not always use two levels?**

DuckDB always processes vectors of 2,048 within morsels of 122,880
because at their data volumes, the morsel is always too large for L1.
Our morsels are sized to L1 by construction — the morsel formula
already accounts for column count. Adding a mandatory inner level
would just be the morsel again.

The two-level structure exists in the design (morsel + micro-morsel)
but the inner level only materialises when needed. This is strictly
better than DuckDB's fixed two-level split: no overhead in the
common case, full tiling capability in the pathological case.

**Decision: single-level by default (morsel = L1-fitting scheduling
unit). Micro-morsel inner tiling activates when peak_live > L1
(decision 60). No change needed.**

**Q6. Morsel ordering / determinism** — RESOLVED

Polars maintains morsel ordering via MorselSeq and a linearizer.
Our model processes morsels sequentially within a chain on one core,
so ordering is inherently preserved. But if we ever split one chain
across multiple cores, we'd need ordering. Should the design account
for this, or is single-core-per-chain the final answer?

**Answer: ordering is guaranteed by construction. No lineariser
needed.**

Two cases:

**Non-commutative chains:** single-core, sequential morsel
processing. Ordering is trivially preserved — morsels execute in
row order on one thread.

**Commutative chains:** by definition, output is order-independent.
Multiple cores process disjoint row segments simultaneously.
Each core writes to its own row range — no interleaving, no
reordering. The output columns contain the same values regardless
of which core processed which segment.

Polars needs a lineariser because it uses dynamic work-stealing:
morsels complete out of order and must be reassembled. Our
deterministic segment assignment means each row is processed by
exactly one core at a deterministic position — the output is
always in row order, even under multi-core dispatch.

**What about future N-thread non-commutative chains?**

The design explicitly forbids this (decision 46): non-commutative
chains are single-core only. If a future consumer needs parallel
non-commutative processing, they must implement their own
synchronisation in the WorkUnit — the library won't provide it.
This is the correct boundary: the library guarantees ordering
for free because it never dispatches non-commutative work to
multiple cores.

**Decision: no ordering mechanism needed. Non-commutative =
single-core = trivially ordered. Commutative = deterministic
segments = no reordering. No lineariser.**

**Q7. Backpressure** — RESOLVED

Polars has explicit backpressure with consume tokens. A fast
producer chain writes freely, the consumer reads when fences allow.
At our scale, unbounded production fits in memory. Should backpressure
be part of the design, or explicitly scoped out?

**Answer: no backpressure mechanism. Column data is fully
materialised. The progress counter IS the implicit flow control.**

Backpressure exists to prevent a fast producer from overwhelming
a slow consumer when data flows through a pipe. In a streaming
system, unbounded buffering means unbounded memory.

Our system is not streaming. Column data is fully materialised
before the pipeline runs — all rows exist in ColumnStorage at
pipeline start. The pipeline transforms data in-place (or into
output columns). There is no producer-consumer pipe where one side
generates rows faster than the other consumes them.

The progress counter (memory layout §7.3) provides implicit flow
control: chain B cannot process morsel [0..512) until chain A's
progress reaches 512. This prevents B from reading uninitialised
data — which is a correctness concern, not a memory concern.

**Why Polars needs backpressure and we don't:**

Polars is a streaming query engine. `filter → groupby → join`
can produce intermediate results of wildly different sizes. A
filter that passes 99% of rows into a slow groupby will accumulate
unbounded intermediate state without backpressure.

Our model has fixed column sizes — the number of rows is known
at pipeline start (it's the entity count). No op creates more rows
than existed at the start. Output columns are pre-allocated at
their final size. There is nothing to back-pressure against.

Design-for note: **DF10.** If a future consumer needs streaming
intake (rows arriving during pipeline execution), the progress
counter model extends naturally: intake ops update their progress
counter as new rows arrive, and downstream chains only process
up to the current progress. No explicit backpressure token needed
— the progress counter IS the backpressure mechanism.

**Decision: no explicit backpressure. Data is pre-materialised,
column sizes are known, progress counters provide implicit flow
control. Streaming intake (DF10) extends the existing model.**

**Q8. Lock-free dispatch mechanism** — RESOLVED

DuckDB's dispatcher is lock-free and inline on the worker thread.
We haven't specified the dispatch mechanism. At 4-8 cores and 2-5
chains, even a mutex has negligible contention. Should we specify
the mechanism, or leave it as an Executor implementation detail?

**Answer: the dispatch mechanism IS lock-free by construction.
Specify it, because it falls out of existing decisions for free.**

The dispatch table (memory layout §7.6) is a static, read-only
list of (op, start_row, end_row, dependency) entries. Worker
threads read the table + read progress atomics + dispatch ready
morsels. The only writes are progress counter stores (plain
`store` with `Release` ordering — §7.4).

No locks anywhere:

- **Dispatch table:** read-only after Schedule construction.
  Multiple cores read it concurrently. No lock needed.
- **Progress counters:** one `AtomicUsize` per op. Exactly one
  core writes each counter (the core running that op). All other
  cores only read. No contention.
- **Morsel assignment:** deterministic. Each core knows its row
  range at dispatch-table construction time. No shared counter
  to increment.

The only potential lock point is Executor-to-core communication:
"core X, start processing chain Y's segment [a..b)." But this
is a one-time setup per pipeline run, not a per-morsel operation.
The Executor can use a simple per-core channel or even direct
function calls if using thread::scope.

Compare to DuckDB: DuckDB needs lock-free dispatch because it
uses dynamic work-stealing with millions of morsels. The
dispatcher is in the hot path — every morsel completion triggers
a "what should I do next?" lookup. Our dispatch is static: each
core's work is determined at Schedule construction time.

**Decision: dispatch is lock-free by construction. Static dispatch
table (read-only), per-op progress counters (one writer, N
readers), deterministic morsel assignment (no shared counter).
Specify this as a library guarantee, not an Executor implementation
detail.**

**Q9. Memory budget per thread / per chain** — RESOLVED

DuckDB budgets (max_memory / num_threads) / 4 per operator. We
haven't addressed memory limits. Should the library enforce a memory
ceiling? Should ColumnStorage have a capacity limit?

**Answer: already resolved by decision 33 — no library-enforced
memory ceiling. Consumer controls memory via ColumnStorage.**

Decision 33 established this: no library-enforced memory ceiling.
The consumer implements ColumnStorage and controls all allocation.
The library has no global allocator, no memory pool, no ceiling.

DuckDB needs memory budgeting because it handles arbitrary-size
datasets: a query over 100GB of data must spill to disk when RAM
runs out. The per-operator budget ensures no single operator
monopolises memory.

Our column data is consumer-owned. The consumer knows:
- How many entities exist (row count)
- Which columns are needed (the WorkUnit set)
- How much RAM is available

The consumer can compute total memory: `row_count × column_count
× 16 bytes`. If this exceeds available memory, the consumer batches
(Pipeline::run() with subsets). This is a consumer decision, not a
library decision.

**What the library DOES provide:**

- Decision 34: `ColumnStorage::release(column)` — advisory
  notification when a column is dead (all readers done).
- Decision 36: consumer-count model — Schedule pre-computes
  readers per column, decrements on chain completion.
- Decision 59: peak-live column analysis — the Schedule reports
  the maximum simultaneous live column count. The consumer can
  use this to compute peak memory.
- Decision 61: static memory watermark — Schedule construction
  reports the total and peak memory needed.

These give the consumer full information to make memory decisions
without the library imposing a ceiling.

**Decision: no library-enforced memory budget. Consumer controls
allocation via ColumnStorage. Library provides memory information
(peak-live count, watermark, release notification). No change
from decision 33.**

### Additional execution model questions

**Q10. Error propagation** — RESOLVED

When process_batch returns Err within a fused chain, what happens?
Options: fail the entire chain, fail all downstream chains, fail
the entire pipeline. The Executor needs a strategy.

**Answer: fail the chain, poison all downstream chains, report
all errors at pipeline completion.**

Error propagation follows the dependency graph:

**1. Morsel-level:** process_batch returns `Result<()>`. On `Err`,
the Executor stops processing further morsels in that chain. The
chain's progress counter freezes at the last successful morsel
boundary.

**2. Chain-level:** the chain is marked as failed. Its progress
counter will never reach the total row count. This is already the
chain's completion model — "the chain succeeds or fails as a
whole" (tiling model Level 2, line 576).

**3. Downstream chains:** any chain that depends on the failed
chain's progress will block forever (its dependency constraint
`progress >= N` will never be satisfied for N > frozen progress).
The Executor detects this: if a chain is failed and has unmet
dependents, those dependents are "poisoned" — marked as failed
without executing.

**4. Independent chains:** chains in different partitions, or
chains that don't depend on the failed chain, continue executing
normally. A failure in the catalogue loading chain doesn't prevent
the keybind resolution chain from completing (if they're
independent).

**5. Pipeline result:** `Pipeline::run()` returns a result type
that carries:
- Per-chain completion status (success or error)
- The original error from the failed chain
- Which downstream chains were poisoned (and why)

The consumer decides how to handle partial failure. For polka-dots,
any chain failure is likely fatal (abort the build). For a game
ECS, partial failure might be acceptable (skip the failed system
this frame).

**Implementation:**

```rust
enum ChainStatus {
    Completed,
    Failed(anyhow::Error),
    Poisoned { upstream: ChainId },
}

struct PipelineResult {
    chain_results: Vec<(ChainId, ChainStatus)>,
}

impl PipelineResult {
    fn is_ok(&self) -> bool { ... }
    fn failed_chains(&self) -> impl Iterator<Item = &ChainId> { ... }
    fn first_error(&self) -> Option<&anyhow::Error> { ... }
}
```

**Why not fail the entire pipeline immediately?**

Early termination wastes completed work. If chain A fails but
chain C (independent) has already completed, throwing away C's
results is wasteful. The consumer can inspect `PipelineResult`
and use whatever partial results are available.

Additionally, the Executor may not be able to cancel chains that
are already running on other cores. Poisoning is a cooperative
mechanism — the Executor checks chain status before dispatching
the next morsel.

**Decision: chain failure → freeze progress, poison dependents,
continue independent chains. PipelineResult carries per-chain
status. Consumer decides whether partial failure is acceptable.**

**Q11. Completion / result extraction** — RESOLVED

After all chains complete, output columns contain results. How does
the consumer read them? Direct column access by index? A structured
result type? Should the Pipeline::run() return type carry type-safe
access to output columns?

**Answer: Pipeline::run() returns PipelineResult (status) + the
consumer reads columns directly from their own ColumnStorage.
No result copying.**

The consumer owns ColumnStorage. Output columns are written
directly into the consumer's storage by the pipeline. After
`Pipeline::run()` completes, the data is already where the
consumer put it — there is nothing to "extract."

**The flow:**

```rust
// Consumer creates storage and registers columns
let mut storage = MyColumnStorage::new();
let name_col = storage.create_column::<Name>();
let installed_col = storage.create_column::<Installed>();
let resolved_col = storage.create_column::<ResolvedPath>();

// Consumer builds pipeline with WorkUnits that reference these columns
let pipeline = Pipeline::new()
    .add(LoadCatalogue { write: (name_col, installed_col) })
    .add(ResolvePaths { read: name_col, write: resolved_col });

// Pipeline runs — writes directly into consumer's storage
let result = pipeline.run(&mut storage, &executor)?;

// Consumer reads results from their own storage — no extraction step
for row in 0..storage.row_count() {
    let name = storage.read::<Name>(name_col, row);
    let path = storage.read::<ResolvedPath>(resolved_col, row);
    // use name, path...
}
```

**Why no structured result type?**

A structured result type would mean copying data out of
ColumnStorage into a result struct — an unnecessary copy. The
consumer already has typed access to their columns through
ColumnStorage. The column handles (`name_col`, `resolved_col`)
are typed — `storage.read::<Name>(name_col, row)` returns a
`Name`, not an untyped blob.

**What about type safety?**

The column handle carries the type:

```rust
struct ColumnHandle<T: Payload> {
    index: usize,
    _marker: PhantomData<T>,
}
```

`storage.read::<Name>(handle, row)` is type-safe at compile time.
The consumer can't accidentally read a `Name` column as an
`Installed` — the types don't match.

**PipelineResult is for status, not data:**

`Pipeline::run()` returns `PipelineResult` (defined in Q10) which
tells the consumer what succeeded and what failed. The actual
data is in ColumnStorage. This separation is clean:

- `PipelineResult`: did it work? which chains failed? why?
- `ColumnStorage`: where is the data? (where it always was)

**Decision: no result extraction mechanism. Consumer reads output
columns from their own ColumnStorage using typed column handles.
Pipeline::run() returns PipelineResult (per-chain status) only.
No data copying.**
