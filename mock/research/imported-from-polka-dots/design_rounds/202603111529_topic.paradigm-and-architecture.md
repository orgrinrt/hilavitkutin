# Topic: The Paradigm — How polka-dots Works

**Date:** 2026-03-11
**Status:** IN PROGRESS — being built interactively, point by point

## Purpose

Establish the complete architectural paradigm for polka-dots. This document
becomes the canonical reference that gets embedded into every crate's lib.rs
as a doc comment wall-of-text, so that any agent reading source code — instead
of the documentation files like they should — still gets the full picture.

---

## §1 What polka-dots Is

polka-dots is a plugin-extensible data compiler and pipeline engine.

**The core loop:**

1. **Intake** — loads files of any format (TOML, JSON, YAML, INI, .polka,
   bash scripts, whatever) into a unified internal state. Each format has
   a registered Format plugin that tells the host how to parse it. Executable
   files have registered Runner plugins that tell the host how to invoke them.
   All data, regardless of source format, ends up in the same internal shape.

2. **Processing** — pipeline stages operate on the unified data. Pre-process,
   codegen, compile-time evaluation, post-process, arbitrary transformations.
   Everything works on the same shape regardless of where the data came from.

3. **Output** — compiles the processed data into build artifacts. Templates
   get resolved, variables get inlined, everything gets materialized into
   concrete files. The output is "compiled" — no unresolved templates or
   variables remain (unless explicitly kept, like env vars in the artifacts).

4. **Deploy** (optional) — places the compiled artifacts where they belong.
   Symlinks, file copies, machine orchestration — whatever the deployment
   target requires.

**What makes it more than a compiler:** The extensibility gives it specific
purposes. With dotfile-management extensions, it becomes a dotfile manager
(deploying with symlinks, tracking changes, syncing back modifications).
With orchestration extensions, it becomes a homelab manager (running things
on other machines — data in, processing, data out to different targets).
The core is format-agnostic data compilation. Extensions give it domain
meaning.

**The compiler analogy is intentional:** Data in (any format, mixed together)
→ unified IR → processing passes → build artifacts out. This is what compilers
do. polka-dots is a compiler whose input language is "all your config files"
and whose output is "all your generated/deployed config."

## §2 The Plugin Model

polka-dots is a HOST that loads PLUGINS (cdylib shared libraries). The SDK
defines trait contracts. Plugins implement traits and register themselves
via `inventory` + `repr(C)` descriptors. The host discovers registered
implementations at startup and calls them through `repr(C)` descriptor
structs with `extern "C"` function pointers.

First-party extensions (brew, mas, .polka DSL, macOS platform) use the
exact same plugin path as hypothetical third-party ones. Zero special
treatment.

### 2.1 What plugins do

Plugins define trait implementations and register them. The things plugins
provide all run on the HOST side of the boundary. Only definitions exist
on the plugin side. No data passes from host to plugins. Everything is
trait contracts and the minimal `repr(C)` ABI surface needed to bridge.

### 2.2 What extensions teach the system

Extensions don't just add capabilities — they **teach the system new shapes**.

Without the keybinds extension, the system doesn't know what a keybind is.
The extension gives it the shape (how keybinds are stored), the parsing
(how to read keybind definitions), the transformation rules (how to produce
skhd/aerospace/emacs output). Without it, nobody can define transformations
or mutations on keybind data, because the concept doesn't exist in the
system.

This is the extension model: **an extension introduces a domain concept
by giving the system the shape, the parsing, and the processing rules for
that concept.** The host provides the infrastructure (loading, unified state,
pipeline execution, deployment). Extensions provide the domain knowledge.

### 2.3 Registration pattern

The `#[register]` attribute macro, applied to a trait impl block, generates:
1. A `repr(C)` descriptor struct with `extern "C"` function pointers
   that bridge to the developer's trait impl methods.
2. An `inventory::submit!` call for that descriptor.

The macro inspects the trait name in the impl block, maps it to the
appropriate descriptor type via a fixed lookup table in the proc macro.

### 2.4 Discovery

`Registry` trait provides typed discovery with built-in validation.
Base validation (version compat, duplicate names) is non-overridable.
Domain registries add domain-specific checks via `validate_domain()`.

Generic `InventoryRegistry<D>` in the SDK handles the 80% (iter, get,
version check). Core crates provide domain wrappers with specialized
validation.

### 2.5 Cross-cutting base traits (polka-primitives)

Three foundational traits that everything composes from:

- **`Named`** — identity. Returns a Cow-based newtype (not raw `&str`).
  Every registered type implements Named. This is why domain traits like
  WorkUnit do NOT have `fn name()` — Named is separate and composable.

- **`Registrable`** — marker trait meaning "this type can be registered
  via `#[register]`." Nothing else. Note: "Registry" in polka-dots means
  ONLY the inventory/link-time static collection concept. The app/package
  catalog is `Catalogue` (separate concept, separate crate).

- **`Descriptor`** — the `repr(C)` bridge struct that `#[register]`
  generates. Contains `extern "C"` function pointers and static metadata.
  The host reads descriptors to discover and call plugin implementations.

These are the base layer. Specifics will evolve as we learn from saalis
and loimu implementations.

### 2.6 Constraints

- The SDK crate must be dynamically linked (shared) between host and
  plugins, not statically duplicated.
- No `dlclose` — plugins are load-only. `inventory` stores `&'static`
  references that would dangle on unload.
- Plugin dylibs are compiled as `cdylib`.

## §3 Type Taxonomy

Every type in the SDK falls into exactly one of four categories:

| Category | When | polka examples |
|----------|------|---------------|
| **Trait** | Varying behavior, shared contract, extensible | Runner, Format, Source, Platform, PipelineStage |
| **Newtype** | Fixed representation, validation at construction | Hash, CataloguePath, EntryId, StageName |
| **`repr(C)` enum** | Closed set, crosses ABI boundary | ErrorClass, SettingsScope, FieldKind |
| **Primitive** | No semantic meaning beyond raw value | bool, usize |

**The test:** "Can I imagine two implementations with different behavior?"
If yes → trait.

**The plugin boundary rule:** If you're asking "should this enum have room
for plugin extensions?" — the answer is: it shouldn't be an enum. It should
be a trait.

**`repr(C)` enum convention:** Every `repr(C)` enum has explicitly assigned
discriminant values (no implicit ordering). Represents a closed set of values.
Uses `lower discriminant = higher priority/precedence` where an ordering
relationship exists. If a concept might need plugin extension → trait, not enum.

## §4 Banned Constructs

### 4.1 Entirely banned

- **`dyn`** — no trait objects anywhere. Not in SDK, not in core, not in
  extensions. No exceptions. Not even for host-provided services.
- **`std::any`** — no type erasure. Everything is statically typed.
- **Raw `String`** — must be behind a documented newtype explaining why
  the allocation is necessary.
- **Raw `Vec`** — must be behind a documented newtype.
- **Raw `Box`** — must be behind a documented newtype.

### 4.2 Allowed with constraints

- **`HashMap` / `BTreeMap`** — allowed directly.
- **`Option`** — allowed directly.
- **`Result`** — allowed, but strive for domain-specific error types
  (not `Result<T, String>`).
- **`PathBuf` / `Path`** — allowed in non-`no_std` crates.
- **`&'static str` / `&str`** — allowed, but prefer a `Cow`-based
  newtype for the canonical string representation.

### 4.3 The newtype mandate

Every use of `String`, `Vec<T>`, or `Box<T>` in public API must be:
1. Wrapped in a newtype
2. Documented with WHY the allocation is necessary
3. The newtype carries the semantic meaning (not just a wrapper)

## §5 Data Model — The In-Memory Database

### 5.1 The core realization

polka-dots' data layer IS a typed in-memory database. This is not
accidental — it's the natural consequence of being a data compiler:

- **saalis** is a database system with schema-as-traits + Column<In<Table>, As<Type>>
- **polka-dots** is a data compiler with the same patterns
- **LLVM, rustc** — every serious compiler has database-like IR internals

The data model is the heart of the system. Everything else (pipeline,
CLI, extensions) builds on top of the database.

### 5.2 Schema-as-traits IS the IR definition mechanism

Data schema is declared through trait impls on ZSTs with compile-time
type constraints. NOT through a runtime value enum. NOT through `dyn`.
NOT through `toml::Value`.

**Schema-as-traits and domain IRs are the same concept.** The ZST trait
declarations ARE the IR definitions. They're not separate layers.

The pattern (same as saalis `Column<In<Table>, As<ValueType>>`):

```rust
// Define an IR (table)
struct CatalogueIR;
impl IR for CatalogueIR {
    type Key = PackageName;  // what identifies a row (entry)
}

// Define columns in that IR
struct Version;
struct Installed;

#[register]
impl Column<In<CatalogueIR>, As<VersionStr>> for Version {}
#[register]
impl Column<In<CatalogueIR>, As<OurBool>> for Installed {}
```

Reading: "Version is a Column In CatalogueIR As VersionStr."

Note: we use `Column` (same name as saalis) not `Kvp`, because the
underlying model IS columnar storage and we share the same bones.

### 5.3 The IR trait

Each domain IR declares its key type — what identifies an entry (row):

```rust
trait IR {
    type Key: Eq + Hash;  // entry identity
}
```

Identity is domain-specific:
- Catalogue entries: identified by PackageName
- Keybinds: identified by KeyCombo (+ context)
- Config files: identified by output path / template name

Override matching: when loading from multiple files, entries with the
same Key are the same entry. How conflicts resolve (last-wins? merge?
error?) is domain-specific — the IR or its passes handle merge logic.

### 5.4 Columnar storage

**Everything is a column.** The in-memory database stores data in
columnar layout:

- **Table** = a domain IR (CatalogueIR, KeybindIR, etc.)
- **Row** = an entry (one managed thing, identified by Key)
- **Column** = a property declared via schema-as-traits

Columnar contiguous storage means:
- Each column is a contiguous array of its type: `[VersionStr; N]`
- Type erasure is by column sizing (packed size from the trait, statically
  available, encoded into the IR/descriptor)
- Stride is uniform within a column — efficient to process
- All column data can live in an arena allocator

**No dyn/Any needed:** Each column is homogeneous (all values same type).
The "heterogeneity" is across columns, but columns are accessed
individually through their ZST key which carries type information
statically. Access: `get_column::<Version>()` returns `&[VersionStr]`.

### 5.5 Shared columns across IRs

The same ZST can be a column in multiple IRs:

```rust
impl Column<In<CatalogueIR>, As<VersionStr>> for Version {}
impl Column<In<BrewPackageIR>, As<VersionStr>> for Version {}
```

`Version` is a reusable "column concept." Storage is per-table (separate
column in each IR). But the semantic meaning is shared.

### 5.6 Nesting — foreign keys

Nested/hierarchical data is expressed as foreign keys to other IRs:

```rust
struct Taps;
impl Column<In<BrewConfigIR>, As<ForeignKey<TapListIR>>> for Taps {}
```

A cell stores a key (or set of keys) into the referenced IR. The host
resolves references when needed. Exact FK representation is deferred
to implementation.

### 5.7 WorkUnit access — scoped to table + column

WorkUnits declare data access scoped to specific IRs and columns:

```rust
impl WorkUnit for AuditPass {
    type Read = (
        In<CatalogueIR, Installed>,    // read Installed column
        In<CatalogueIR, Version>,       // read Version column
    );
    type Write = (
        In<AuditIR, AuditResult>,       // write AuditResult column
    );
    fn process_batch(read: Self::Read, write: Self::Write) -> Result<()>;
}
```

Extensions can add columns to any IR (register new Column impls
`In<SomeIR>`) — making any IR extensible without modifying the
defining extension.

### 5.8 Domain IRs (Intermediate Representations)

Each domain defines its own IR — an intermediate representation for that
domain's data. The system holds multiple domain-specific IRs.

**The three basic operations on an IR:**

| Operation | Signature | Who provides it |
|-----------|-----------|----------------|
| **Intake** | N → IR | Format bridge extensions (e.g., keybinds-toml) |
| **Pass** | IR → IR | Domain extensions, or any extension |
| **Output** | IR → M | Output bridge extensions (e.g., skhd-keybinds) |

Extensions are composable, not monolithic. Any extension can bring any
combination of these three operations:

- `polka-keybinds` — defines `KeybindIR` shape + IR→IR validation passes
- `keybinds-toml` (hypothetical) — TOML → KeybindIR + KeybindIR → TOML
- `skhd-keybinds` (hypothetical) — KeybindIR → skhd config format

### 5.9 The Catalogue is just another IR

The Catalogue (app/package registry) is NOT special infrastructure. It's
a domain IR whose entries are declared via schema-as-traits, same as
any other domain concept. Source extensions (brew, mas) provide intake
bridges. The audit system provides IR→IR passes.

### 5.10 Format agnosticism

The data layer is format-agnostic. Format bridges handle serialization
specifics. The IR shape and domain logic never reference any specific
format. Source format is an implementation detail of the intake bridge.

### 5.11 Shared bones with saalis

polka-dots and saalis have identical architectural bones:

| Shared concept | saalis | polka-dots |
|---------------|--------|-----------|
| Schema-as-traits | Column<In<Table>, As<Type>> | Same |
| Registration | inventory + #[register] | Same |
| Base traits | Named, Registrable, Descriptor | Same |
| Plugin model | cdylib + repr(C) descriptors | Same |
| Error model | ErrorClass + ErrorKind + AppError | Same |
| WorkUnit | DAG scheduler | Same (different scheduling needs) |
| **Backend** | **SeaORM / SQLite** | **In-memory columnar** |

**Strategic decision:** Same names, implement independently, merge later.

Both projects use identical names for the shared concepts (Column, Named,
Registrable, Descriptor, ErrorClass, IR). Both design docs reference
each other's decisions. But the implementations are independent — no
git dependency, no coordination overhead during design phase.

Rationale:
- Both are in design phase (no working code to extract from)
- "Same patterns" at doc level may diverge at implementation level
  (SeaORM needs migration support; polka needs arena allocation)
- Premature extraction creates coupling that slows design iteration
- When BOTH have working implementations, extract the genuinely shared
  parts into a common crate (likely in loimu or standalone)

**loimu relationship:** loimu's full framework (54 crates, signal-driven
runtime, temporal storage, HEFT scheduler) is too specialized for
polka-dots or saalis to depend on right now. The shared patterns emerged
independently. loimu convergence is a long-term goal, not a current
dependency.

## §6 Plugin Trait Taxonomy

### 6.1 Runner — "how to execute this"

Definition of how to invoke an interpreter/runner on content. Bash → pipe
to `bash`. Python → `python3 eval`. TypeScript → `ts-node`. Just a command
pattern. The host does the actual execution.

### 6.2 Format — "how to parse/serialize this"

How to convert between structured data and text format. TOML, JSON, INI,
YAML, KDL. Parse text into structured data (intake). Serialize structured
data into text (output). Replaces the previous "Language" and "Transform"
traits with a single bidirectional concept.

### 6.3 Source — "how to manage packages from this provider"

Definition of how to install/uninstall/query packages from an external
source (brew, mas, apt, etc.). Host does the actual shell execution.

### 6.4 Platform — "what environment are we running on"

Platform detection, paths, capabilities. macOS, Linux, WSL, NixOS. The
host queries registered platforms to determine the current environment.

### 6.5 WorkUnit — "a unit of work in the compilation DAG"

The pipeline is a flat DAG, not a phased sequence. "Intake → process →
output → deploy" are descriptive labels, not enforced phases. Ordering
emerges from the dependency graph.

**The WorkUnit trait** is the core pipeline abstraction. It is maximally
minimal — WorkUnit only has `execute()`. Everything else is expressed
through the type system or cross-cutting traits:

- **Identity** — via `Named` trait (cross-cutting, not per-trait)
- **DAG wiring** — ECS-style data access patterns, NOT explicit events.
  The WorkUnit's context is a generic parameter that declares what
  resources the unit reads and writes (like Bevy system parameters).
  The `#[register]` macro extracts `Read<T>` and `Write<T>` types from
  the context into the `repr(C)` descriptor. The host builds the DAG
  from these data access patterns:
  - `Read/Read` on same resource → can run in parallel
  - `Read/Write` or `Write/Write` on same resource → must be sequential
  - No shared resources → fully independent, can parallelize
- **No triggers/emits** — data access patterns fully capture ordering
  for a single-run exhaustive pipeline. If two units need ordering,
  there must be a data dependency. Make it explicit through a typed
  resource rather than an abstract event.
- **No priority** — the pipeline runs to completion, not a frame-rate
  loop. No scheduling tiebreakers needed.

```rust
// Conceptual shape (not final syntax):
trait WorkUnit {
    type Read: ColumnSlices;       // input column slices
    type Write: ColumnSlicesMut;   // output column slices
    fn process_batch(read: Self::Read, write: Self::Write) -> Result<()>;
}
```

The Read/Write types declare data dependencies at the type level.
The Schedule resolves these declarations into a dependency graph.
The Executor processes morsels (batches of rows), calling
process_batch() with column slices for each morsel.

Extensions register WorkUnit impls. A keybinds extension registers a
WorkUnit with `(Read<ParsedFormats>, Write<KeybindState>)` — the DAG
automatically places it after format-parsing and before anything that
reads KeybindState. No hardcoded phases, no explicit event wiring.

### 6.6 The .polka DSL (polka-lang)

The .polka DSL is a separate, optional extension. It has its own parser,
AST, macro system, expander — but that internal complexity is invisible
to the SDK. It's just another extension that registers Format + Runner
capabilities for .polka files. The system works without it.

- `AppError<K>` trait — generic over error kind

- Errors are type-erased via descriptors at the boundary

## §9 The Macro System — Consistent Building Blocks

Every fundamental concept gets a macro that generates ALL the boilerplate:
the type, the descriptor, Named impl, Registrable impl, inventory::submit.
This makes it impossible for the generated code to diverge between crates,
because no one writes it by hand. The macro IS the single source of truth.

### 9.1 `define_*!` macros — declaring new concepts

These macros create something new that the system needs to know about:

| Macro | What it creates |
|-------|----------------|
| `define_registry!` | A registry for a domain — category ZST, descriptor type, inventory discovery |
| `define_ir!` | A domain IR (table) — IR trait impl, Key type, column ZSTs + Column<In<_>, As<_>> impls |
| `define_id!` | A typed domain ID — integer equality, interned lookup |
| `define_error!` | An error type — ErrorKind impl, diagnostic descriptor |

Each macro generates:
1. The type itself (struct/ZST)
2. A `*Descriptor` wrapper (repr(C))
3. `Registrable` impl pointing to the descriptor
4. `Named` impl on the descriptor
5. `inventory::submit!` call for auto-discovery

### 9.2 `#[register]` — registering trait implementations

The `#[register]` attribute macro applies to trait impl blocks. It
registers an implementation of an existing trait:

| What's registered | Example |
|-------------------|---------|
| Format (N→IR / IR→N) | `#[register] impl Format for TomlFormat {}` |
| Source | `#[register] impl Source for BrewSource {}` |
| Runner | `#[register] impl Runner for BashRunner {}` |
| Platform | `#[register] impl Platform for MacosPlatform {}` |
| WorkUnit | `#[register] impl WorkUnit for AuditPass {}` |

`#[register]` generates the same boilerplate as `define_*!` (descriptor,
inventory submit) but for an IMPLEMENTATION rather than a DECLARATION.

### 9.3 The distinction

`define_*!` = "this concept exists" (declaring new things).
`#[register]` = "this type implements that concept" (registering impls).

Both ensure consistency through code generation. Neither allows hand-
written registration boilerplate. The lints reject any manual descriptor
creation, any manual inventory::submit!, any manual Named impl on a
registered type.

### 9.4 No `#[derive(Settings)]` — settings are just data

There is no "settings" concept. Settings are key-value data in IRs like
everything else. If a domain needs scoped configuration (global < platform
< app < local override chain), it defines scope columns in its IR and a
resolution WorkUnit that collapses them.

`#[derive(Settings)]` does not exist. The define_ir! macro + standard
column definitions + a resolution pass already cover this. Adding a
separate settings abstraction would create a parallel concept for
something the data model already handles. Every column in every IR is
already a "setting" — some just happen to have scope metadata.

## §11 Runtime Architecture — Schedule + Executor

The runtime separates planning from execution. The Schedule builds a
dependency graph from registered WorkUnits. The Executor (a trait)
takes the Schedule and the data, optimizes access patterns, and runs
the operations. The CLI implements or uses an Executor.

### 11.1 Schedule — the planner

The Schedule is built from a `LoadedProject`. At construction time it:

1. Discovers all registered IR types via `inventory` → assigns numeric IDs
2. Discovers all registered WorkUnits via `inventory`
3. Extracts Read/Write declarations from each WorkUnit's descriptor
4. Builds a dependency graph from data access patterns

The Schedule does NOT execute anything. It produces an ordered
dependency graph: "these are the WorkUnits, these are their data
accesses, these must come before those." Pure topology.

The set of IR types and WorkUnit types is fixed at startup. All are
known at link time via `inventory`. No dynamic creation.

### 11.2 Dependency graph

Each WorkUnit descriptor (generated by `#[register]`) carries metadata
about its data access:

- Which (IR, Column) pairs it reads
- Which (IR, Column) pairs it writes

The Schedule builds edges: if WorkUnit A writes to (IR_x, Column_y)
and WorkUnit B reads from (IR_x, Column_y), A must execute before B.

Rules (decision 30):
- Read/Read → no conflict (parallel-safe)
- Read/Write or Write/Write → dependency edge

The graph is static. It does not change during execution.

### 11.3 Executor — the trait

The Executor is a trait. It takes a Schedule (the dependency graph)
and the IR data, then runs the operations.

The Executor's responsibilities:
1. **Analyze ordering** — take the dependency graph's constraints
2. **Build archetypes** — group columns that are accessed together by
   the same set of WorkUnits into static archetypes
3. **Batch by column** — reformat operations into column-oriented
   execution order, not WorkUnit-oriented order
4. **Execute tile-based** — process column by column, running the ops
   that touch each column in dependency-respecting order

The Executor decides threading, error handling, incremental tracking,
and all other execution concerns. The Schedule doesn't care.

### 11.4 Morsel-driven batch execution

#### 11.4.1 Why not row-at-a-time or column-at-a-time

**Row-at-a-time** (process one row across all columns, then next row):
Breaks cache locality. If a row spans 10 columns stored at different
addresses, each row access touches 10 different memory regions. The
cache fills with data from all 10 columns, but only uses one entry
from each per row. Wasteful.

**Pure column-at-a-time** (process all rows of col1, then all of col2):
Maximizes cache locality for a single column, but multi-column
operations (read col1 + col2, write col3) require buffering. Read
all of col1 into a buffer, then read all of col2 while applying
the buffered col1 values. The buffer is extra memory, and the data
is touched twice (once to buffer, once to transform). Wasteful.

**Morsel-driven batch execution** (process N rows at a time, all
relevant columns together): Best of both worlds. A "morsel" is a
batch of N rows. For each morsel, load slices of all relevant
columns into cache. Process the morsel — direct reads and writes
to column slices, no buffering. Then move to the next morsel.

This is the model used by DuckDB and other modern analytical
databases. polka-dots implements its own lightweight version
tailored to 128-bit strides and the plugin architecture.

#### 11.4.2 Why columns in different memory regions are still efficient

Each column is stored as an independent contiguous array. Columns
are NOT co-located in memory. The Version column might be at address
0x1000_0000 and the LatestVersion column at 0x5000_0000. This is
fine because of how CPU hardware works:

**Hardware prefetching.** When the CPU reads memory, it detects
sequential access patterns. If it sees reads at addresses 0, 16,
32, 48 (our 128-bit stride), it prefetches the next cache lines
before they're requested. Modern CPUs (including Apple Silicon)
track 8-16 independent sequential streams simultaneously. Each
column is one stream. The prefetcher runs ahead on ALL of them.

Concrete example — processing Version + LatestVersion together:

```
Version at 0x1000_0000:       [V0][V1][V2][V3] [V4][V5][V6][V7] ...
LatestVersion at 0x5000_0000: [L0][L1][L2][L3] [L4][L5][L6][L7] ...
```

1. Read V0 — cache miss. CPU fetches cache line: V0-V3 arrive.
2. Read L0 — cache miss. CPU fetches cache line: L0-L3 arrive.
3. Read V1 — cache hit. Already loaded in step 1.
4. Read L1 — cache hit. Already loaded in step 2.
5-8. V2, L2, V3, L3 — all cache hits.
9. Read V4 — prefetcher already sent for this. Cache hit.
10. Read L4 — prefetcher already sent for this. Cache hit.

**Two cache misses total for the first entries. Everything else
is hits.** The prefetcher tracks both streams independently. As
long as each stream is sequential (which columnar guarantees),
the CPU stays fed. Where the streams are in memory doesn't matter.

**Why row-based layout wastes cache:** A row-based layout stores
`[V0, L0, Name0, Desc0, ...][V1, L1, Name1, Desc1, ...]`. If
you only need Version + LatestVersion (2 of 10 columns), 80% of
every cache line is data you never read. Columnar layout means
cache lines contain ONLY the column you're reading. 100% useful.

#### 11.4.3 Batch sizing — fitting in L1

The morsel size (N rows per batch) is chosen so that ALL column
slices for that batch fit in L1 data cache simultaneously.

Apple Silicon (M-series) numbers:
- L1 data cache: 128 KB per performance core
- 128-bit stride: 16 bytes per entry
- 1024 rows × 16 bytes = 16 KB per column slice
- At 1024 rows: 7 column slices fit in L1 (7 × 16 KB = 112 KB)
- At 2048 rows: 3 column slices fit in L1 (3 × 32 KB = 96 KB)

For most operations in polka-dots (2-5 columns), a morsel of 1024
rows is ideal. All column slices fit in L1 with room to spare.
The Executor tunes morsel size based on the column count of each
operation.

#### 11.4.4 The full hardware picture

The prefetcher is not the only consideration. Every layer of the
memory hierarchy is involved:

**TLB (Translation Lookaside Buffer).** Every memory access goes
through virtual-to-physical address translation. The TLB caches
these translations. A TLB miss triggers a page table walk (4-5
chained memory accesses — expensive).

Why it's manageable: Apple Silicon uses 16 KB pages. A morsel of
1024 rows at 16 bytes = 16 KB = exactly 1 page per column per
batch. So 5 columns = 5 TLB entries. Apple Silicon L1 TLB has
~160 entries. 5/160 = 3% of TLB capacity. The TLB entry is loaded
ONCE per column per batch, then reused for all 1024 sequential
accesses within that page. Cost is amortized across the entire
morsel.

TLB would hurt with hundreds of columns or huge batches spanning
many pages per column. Not our case.

**Cache associativity.** L1 cache is 8-way set-associative on
Apple Silicon. Multiple column addresses COULD map to the same
cache set, causing evictions even when total data fits ("conflict
misses"). Statistically unlikely with typical allocator behavior
(addresses are spread across sets), but possible. The memory
allocator naturally spreads column arrays across different address
ranges, minimizing conflicts.

**Memory bandwidth.** There's a ceiling on how fast data moves
from RAM to CPU. Apple Silicon unified memory: ~100-200 GB/s.
polka-dots operates on kilobytes to low megabytes. We will never
approach this limit. Bandwidth matters at DuckDB scale (gigabytes),
not dotfiles scale.

**False sharing.** If two threads write to the same cache line
(64 bytes), the line "bounces" between cores — each write
invalidates the other core's copy. Since columns are independent
arrays and the Executor assigns column ranges to threads at morsel
boundaries, different threads write to different cache lines. The
128-bit stride (16 bytes) means 4 entries per cache line — as long
as morsel boundaries align to cache line boundaries (every 4 rows),
no conflict.

**Store buffer / write combining.** When writing output columns,
the CPU batches sequential writes in a store buffer. Writes to the
same cache line are combined before flushing to cache. Our model
(sequential writes to contiguous array) is the optimal pattern.
The hardware combines 4 sequential 16-byte writes into one cache
line write.

**Branch prediction.** The fence check (is this entry data or a
fence?) is a branch per entry. Since fences are rare — one per
dependency boundary, maybe 1 per thousand entries — the branch
predictor predicts "data" with ~99.9% accuracy. Mispredictions
(~15 cycles each on Apple Silicon) are negligible at that rate.

#### 11.4.5 Hardware concern summary

| Concern | Impact at our scale | Why |
|---------|-------------------|-----|
| Prefetching | Handled | Sequential access, CPU tracks multiple streams |
| TLB | ~5 entries per batch | One page per column per morsel, 160-entry TLB |
| Cache capacity | ~48 KB for 3 cols | L1 is 128 KB, plenty of room |
| Cache associativity | Unlikely conflict | Allocator spreads addresses |
| Memory bandwidth | Nowhere near limit | KB-MB scale, not GB |
| False sharing | Not applicable | Per-column, per-thread morsel boundaries |
| Store buffer | Optimal pattern | Sequential writes, hardware combines |
| Branch prediction | ~99.9% hit rate | Fences are rare |

#### 11.4.6 Archetype as scheduling hint, not storage layout

An "archetype" is the set of columns co-accessed by the same
operations. It is a SCHEDULING optimization, not a STORAGE layout.

The problem with archetype-as-storage: if column Version is needed
by Op A (reads Version + LatestVersion) AND Op B (reads Version +
Name), Version would need to live in two memory regions. Duplicating
column data is wasteful and incoherent.

Instead: columns are stored independently. The Executor uses
archetype information to SCHEDULE operations. Ops that share input
columns run consecutively on the same morsel, so shared column
slices are still in L1 cache from the previous op. Cache reuse
without storage duplication.

#### 11.4.7 WorkUnit decomposition — batch slice processing

WorkUnits are decomposed for batch execution. A WorkUnit does NOT
run as a single opaque function call over the entire dataset. It
receives **column slices** for a morsel and processes them directly.

Conceptual shape:

```rust
trait WorkUnit {
    type Read: ColumnSlices;       // input column slices
    type Write: ColumnSlicesMut;   // output column slices
    fn process_batch(read: Self::Read, write: Self::Write) -> Result<()>;
}
```

The Executor:
1. Partitions rows into morsels (N rows, sized to L1)
2. For each morsel, extracts column slices from stored arrays
3. Calls process_batch() for each WorkUnit whose inputs are ready
4. Orders WorkUnit calls within a morsel by the dependency graph

Within process_batch(), the extension author has direct access to
input and output slices. They can iterate row by row, use SIMD,
or apply any transformation. The Executor does not decompose the
function further — it controls WHEN the function runs and WHICH
rows it processes.

No buffering. Column slices point directly into the stored column
arrays. Reads are direct pointers. Writes go directly to the output
column array. One memory touch per value.

### 11.5 Column entry encoding — 128-bit tagged stride

Every column entry is a fixed 128-bit (16-byte) stride. The first
byte is a packed flags byte. The remaining 15 bytes are payload.

```
Byte 0 (flags):
  bit 0: is_fence (0 = data, 1 = fence)
  bit 1: reached  (fence handshake status)
  bits 2-7: reserved for future flags
Bytes 1-15: payload (15 bytes)
```

For data entries (bit 0 = 0): payload holds the column value.
For fence entries (bit 0 = 1): payload holds a pointer to the
dependency fence entry (in another column's data).

The 128-bit stride aligns with NEON (ARM/Apple Silicon) and SSE
SIMD register widths. The Executor loads each entry as a full
128-bit stride (one NEON `ldr q` instruction). Flags extraction
and payload extraction are register operations (shifts, masks) —
zero additional memory cost. One memory load per entry.

Payload starts at byte 1, not byte 8. No alignment padding
between flags and payload. This gives 15 bytes of usable payload
space instead of 8. Since the Executor always loads the full
stride, internal alignment of the payload is irrelevant — the
payload is extracted via a register shift (`loaded >> 8`), which
is a single-cycle operation.

### 11.6 The 15-byte column value constraint

The `As<T>` type parameter in `Column<In<IR>, As<T>>` specifies
what gets stored in the column. Since the payload is 15 bytes,
`T` must fit in 15 bytes. This is enforced at compile time:

```rust
const _: () = assert!(
    size_of::<T>() <= 15,
    "Column value type exceeds 15-byte stride payload"
);
```

Generated by `define_ir!` for every column definition. Zero
runtime cost. If `size_of::<T>() > 15`, the crate does not
compile.

This constraint forces the right design patterns:

- Typed IDs (u32, u64) → 4-8 bytes, fits
- `repr(C)` enums → 1-4 bytes, fits
- Newtypes wrapping integers → fits
- Booleans → 1 byte, fits
- Small structs (up to 15 bytes) → fits
- `String` → 24 bytes → **compile error** → must use an interned
  ID (u32 index into intern table)
- `PathBuf` → 24 bytes → **compile error** → must use an interned
  ID
- `Vec<T>` → 24 bytes → **compile error** → must use an arena
  index

The constraint directly enforces decision 62 (typed IDs for
lookups). Variable-length data physically cannot live in a column.
It must be interned and referenced by ID. The 15-byte limit makes
the right thing the only thing that compiles.

### 11.7 Per-column fences — cross-column synchronization

Fences are inline within column data, encountered during
iteration. They synchronize cross-column dependencies (e.g.,
"col2 can't be read until col1's writes are done").

Column-local ordering is automatic — the Schedule's dependency
graph determines the order of ops within each column's processing.
Fences handle ONLY cross-column synchronization.

The fence handshake:

1. Op iterates column, encounters fence entry (flags bit 0 = 1)
2. Sets own "reached" bit (flags bit 1 = 1) — "I arrived here"
3. Follows payload pointer to the dependency fence entry
4. Checks dependency's "reached" bit:
   - If 1 → dependency was reached, continue processing
   - If 0 → hold until the Executor signals continuation
5. On hold, the Executor resumes the op when the dependency
   completes and sets its "reached" bit

Fences are rare (one per dependency boundary, not per row). The
pointer follow is an occasional cache miss — acceptable because
the common case (data entry) is a single flags-byte check per
stride.

The Schedule computes fence placement at plan time from the
dependency graph. The Executor inserts fence entries into column
data before execution begins.

### 11.8 Incremental execution

Change tracking at the column level. State persisted between runs
via on-disk manifests and content hashes. An operation re-runs
only if its input column data changed since the last execution.

This is an Executor concern, not a Schedule concern. The Schedule
produces the same graph regardless of what changed. The Executor
decides which operations to skip based on incremental state.

### 11.9 Error handling

When an operation fails:

1. The error is recorded
2. All downstream dependents (operations that need this operation's
   output column data) are **poisoned** — they do not execute
3. Independent branches (no data dependency) continue
4. The result contains actual errors + poisoned (skipped) operations

This is an Executor policy. Different Executors could implement
different strategies (fail-fast, collect-all, etc.).

### 11.10 Deployment is just data

Deployment WorkUnits read from compilation output IRs and write to
deployment-related columns (DeployOp, TargetPath, etc.). These are
data like any other — processed column-by-column by the Executor.

After execution, deployment columns contain the plan: which files
go where, what operations to perform. The CLI reads these columns
and executes the I/O (symlinks, copies, remote sync).

Dry-run = CLI reads the deployment columns and renders the plan
without executing it. The Executor ran identically — only the CLI's
response to the data changes.

### 11.11 The full flow

1. CLI parses args
2. `polka_project::load(path)` → `LoadedProject`
3. `Schedule::new(loaded_project)` → discovers IRs/WorkUnits,
   builds dependency graph
4. `executor.run(schedule, data)` → tile-based column execution
   → returns execution result
5. CLI reads result + final IR data (including deployment columns)
6. CLI renders output, executes deployment ops if requested

All commands go through this flow. No command bypasses `load()`.
No command creates its own Schedule. The Executor never does
user-facing I/O — it processes data. The CLI does I/O.

## §10 Design Principles — Why It Works This Way

### 10.1 Contracts before implementations — why everything is a trait

Every capability in the system is a trait contract FIRST. You define
what something CAN DO before you define what something IS. A `Format`
trait says "anything implementing this can parse and serialize a specific
data format." A `Source` trait says "anything implementing this can
manage packages from a provider." The contract exists independently of
any implementation.

This is not just good practice — it's the foundation of the plugin
model. The SDK contains ONLY traits. Implementations live in extensions.
If you find yourself writing a concrete struct before writing the trait
it implements, you're working backwards.

### 10.2 Why ZSTs are the natural primitive

A Zero-Sized Type carries meaning without data. `struct AutoUpdate;`
says "this concept exists" without allocating a single byte. ZSTs are
the natural primitive for a system where logic is defined through type
relationships, not through stored data.

In our schema-as-traits model, `AutoUpdate` is just a marker. Its
meaning comes from the trait impl: `impl Column<In<BrewConfig>, As<OurBool>>
for AutoUpdate {}`. The ZST has no fields, no methods, no runtime cost.
It exists purely as a compile-time handle that the type system uses to
wire things together.

This is why "everything defines the logic without the concrete types."
The ZSTs are the logic layer. The data lives in columnar storage, keyed
by these ZSTs. Logic and data are cleanly separated — logic is types,
data is tables.

### 10.3 Why `dyn` and `std::any` are entirely banned

Dynamic dispatch (`dyn Trait`) and type erasure (`std::any::TypeId`,
`Any`) are escape hatches from the type system. They say "I don't know
what this is at compile time." In our architecture, you ALWAYS know
what things are at compile time.

The plugin boundary uses `repr(C)` function pointers, not trait objects.
The storage engine uses typed columns, not `Box<dyn Any>`. The registry
uses `inventory` with concrete descriptor types, not runtime `TypeId`
lookups.

**The principle:** If you need dynamic dispatch, your abstraction is
wrong. Either:
- Make it a trait with concrete impls registered via `#[register]`
- Make it a `repr(C)` enum if the set is closed
- Make it a generic parameter if the caller knows the type

There is no fourth option. If none of these work, redesign.

### 10.4 Why raw `Vec`, `String`, `Box` require documented newtypes

Every heap allocation is a design decision. A `Vec<T>` says "I need a
growable, heap-allocated, contiguous sequence." WHY do you need that?
The answer must be documented in a newtype:

```rust
/// Ordered list of package dependencies that grows during
/// intake as new sources are discovered.
struct DependencyList(Vec<PackageName>);
```

If you can't explain why you need the allocation, you probably don't.
Use a slice, a reference, an array, a fixed-size buffer, or an arena
allocation instead.

This forces intentionality about memory. In a system with arena-backed
columnar storage, most data lives in the storage engine, not in ad-hoc
heap allocations. Every raw `Vec` outside the storage engine is
suspicious.

### 10.5 Why columnar storage — data is tables, not objects

Traditional OOP stores data in objects: a `Package` struct has name,
version, source, installed status all together. Accessing all installed
statuses means touching every Package object (scattered memory).

Columnar storage separates concerns: names are one contiguous array,
versions are another, installed statuses are another. Scanning all
installed statuses is a single contiguous memory sweep. Cache-friendly,
SIMD-friendly, parallelizable.

But the deeper reason is: **columns map to the schema-as-traits model.**
Each column IS a ZST's worth of typed data. The type system declares
columns, the storage engine allocates them. There's no impedance
mismatch between the type-level declarations and the runtime storage.

### 10.6 Why data access patterns replace explicit ordering

Traditional pipelines have explicit phase ordering: "parse, then
validate, then transform, then output." This is fragile — adding a new
step requires knowing where it goes, and the ordering is implicit
knowledge in someone's head.

Our approach: WorkUnits declare what they read and write. The DAG
builder figures out the ordering. Two units that read the same data
can run in parallel. A unit that writes data must run before anything
that reads it.

The ordering IS the data flow. If you can't determine ordering from
data access patterns alone, you have a hidden data dependency that
should be made explicit as a typed resource.

### 10.7 Why extensions add shapes, not methods

An extension can't add a method to an existing trait. It can't modify
someone else's IR. It CAN:
- Define a new IR type (a new table in the database)
- Add columns to an existing IR (new properties for existing entries)
- Register WorkUnits that read/write existing IRs

This is the "open for extension, closed for modification" principle
taken to its logical extreme. You extend the system by adding new
shapes (new IRs, new columns), not by modifying existing behavior.

### 10.8 Why `repr(C)` + function pointers for the plugin boundary

Alternatives considered:
- **Cap'n Proto / FlatBuffers**: serialization overhead, runtime schema
  validation, adds large dependency
- **WASM**: sandboxing overhead, limited type sharing, complex toolchain
- **gRPC / IPC**: process boundary overhead, serialization cost
- **Trait objects (`dyn`)**: BANNED (§10.3)

`repr(C)` + function pointers is the minimal ABI surface. The descriptor
struct IS the interface. Function pointers call directly into plugin
code with no serialization, no marshaling, no sandbox overhead. The
cost is exactly one indirect call per function pointer invocation.

The constraint: plugins must be compiled with the same SDK version
(ABI compatibility). This is acceptable — plugins are cdylibs compiled
as part of the same workspace, not downloaded binaries from the internet.

### 10.9 Why `inventory` (link-time) over runtime registration

`inventory` makes registration a link-time operation. When a plugin
cdylib is loaded, its `inventory::submit!` calls have already placed
descriptors into a static collection. Discovery is just iterating a
static slice — no builder pattern, no `register()` calls, no ordering
concerns.

This means: the set of all registered types is known as soon as plugins
finish loading. The host can build the complete DAG, allocate all
storage, and validate all schemas BEFORE any work begins. There's no
"oh, a new type was registered mid-pipeline" surprise.

Runtime registration adds complexity (what if registration order
matters? what if a registration fails after others succeeded?). Link-
time registration eliminates these problems by making registration
atomic and complete.

### 10.10 Why the compiler metaphor matters

Thinking of polka-dots as a "dotfile manager" leads to a file-centric
architecture: copy files, symlink files, diff files. This is what
every other dotfile tool does. It's also why they're all limited.

Thinking of polka-dots as a "data compiler" leads to a completely
different architecture:
- **Input** is not "files" — it's data in various formats
- **Processing** is not "copying" — it's transformation passes on IR
- **Output** is not "files" — it's materialized artifacts from IR
- **Extensions** don't "manage files" — they teach the system new shapes

The compiler metaphor forces you to think about data transformations,
not file operations. It makes the architecture format-agnostic (any
input format → unified IR → any output format). It enables optimizations
that file-centric tools can't do (dead data elimination, cross-domain
analysis, dependency-ordered deployment).

### 10.11 No ad-hoc logic — reusable abstractions everywhere

Before writing ANY function, search for existing abstractions with
similar signatures or names. If something similar exists, use it or
extend it. If nothing exists, you're creating a new abstraction —
make it reusable, not bespoke.

The mockspace lints enforce this: duplicate signatures, overlapping
functionality, and unused abstractions are detected. The compiler will
refuse to build and commits won't go through if you violate this rule.

Ad-hoc logic is the enemy of a plugin system. Every bespoke function
is an abstraction that could have been a trait impl, a registered type,
or a column in the IR. If you're writing one-off logic, you're probably
working at the wrong level of abstraction.

### 10.12 NEVER take the "pragmatic" shortcut

If you are about to use words like:
- "pragmatic solution"
- "YAGNI"
- "for v1 we do X, then later..."
- "future work"
- "stub — actual impl in v2"
- "good enough for now"

**STOP. BREATHE. Then do the opposite.**

These phrases are signals that you're about to compromise the design
for short-term convenience. In this project, the design IS the product.
A "pragmatic" shortcut becomes a permanent architectural scar that every
future extension has to work around.

The right thing to do, even when it seems idiotic, is to implement the
design as specified. If the design is wrong, change the DESIGN (through
a design round), not the implementation. The design is the source of
truth. The implementation follows.

### 10.13 Source code lies — trust only documentation

**NEVER read source code to understand how something should work.**
Source code is what someone ACTUALLY wrote, which may be entirely wrong
and against the design. Some lazy agent or hurried developer has
probably cut corners, skipped validation, used a banned pattern, or
misunderstood the architecture.

**ALWAYS read documentation (.md, .org files) to understand the design.**
The design documents are the canonical truth. If the code doesn't match
the docs, the CODE is wrong, not the docs.

Furthermore: documentation is not siloed. Important details about one
domain may be explained in a side note in an entirely different domain's
design docs. Nothing exists in isolation. When you need to understand
something, search ALL documentation, not just the immediately related
files.

### 10.14 The Enforcement Rule

Every enforceable rule is linted. A principle, workflow step, or
convention that is not enforced by a lint, hook, or compile-time check
**does not exist.** When adding a new rule, you must also write the
lint that enforces it and the test that validates the lint. No exceptions.

Enforcement tiers:
- **Build** (`cargo check`): type errors, macro expansion failures
- **Pre-commit** (git hooks): AST lints, commit message format, byline
  policy, file size limits
- **Pre-push** (git hooks): `cargo check` (full workspace), `cargo test`

A principle without enforcement is a wish, not a rule. If we can't lint
it, we either make it lintable or accept it's advisory guidance.

### 10.15 Rules must live in templates

When a new rule, principle, or convention is discovered, it goes into
the appropriate mockspace template (PRINCIPLES.md.tmpl, WORKFLOW.md.tmpl,
agent rules). NOT into:
- Memory files (`~/.claude/`)
- Plan files or session notes
- Conversation summaries

Those are ephemeral. Other agents and future sessions cannot read them
reliably. If a rule is not in a mockspace template, it does not exist
for the project.

### 10.16 Typed IDs everywhere

Every domain object has a typed ID. Strings are for display and
serialization. Internally, always resolve to a typed ID.

"If you are comparing strings in a loop, the design is broken."

Use interned IDs (pointer-based or integer-based equality) for all
runtime lookups. The `Named` trait provides the human-readable name,
but keyed lookups use typed IDs, not string comparisons.

### 10.17 Production quality in the mockspace

The mock workspace contains production Rust code. Not stubs. Not
skeletons. Not "contracts with stubbed mechanics."

- No `todo!()`. If a function exists, it has a real implementation.
- No "simplified" implementations. If the design says arena-backed,
  the mock uses arena-backed storage.
- No `Vec<T>` where a domain container is specified.
- Trait bounds reflect reality. If the real implementation won't need
  `T: Clone`, the mock doesn't require it either.

The previous "stubs are valid" approach was too permissive. Agents could
not reliably determine which simplifications were safe. The line between
"contract is real" and "mechanics are stubbed" was too fuzzy. Everything
must be real.

### 10.18 Responsible AI tooling

- Prefer agents as assistants over autonomous agents. A human in the
  loop catches mistakes, provides context, and takes responsibility.
- When an agent is the primary contributor (making design decisions,
  authoring logic), the commit must include `Co-Authored-By`. Reviewers
  see at a glance whether a human or agent drove the work.
- When the human drives and the agent assists (implementing human-
  provided design, generating boilerplate), no byline needed.
- The line is about who does the THINKING. If the agent chooses the
  approach and makes the decisions, it's the primary contributor.

### 10.19 Topic files are frozen once committed

A committed topic file is history. It cannot be edited, regardless of
the current design round phase. If corrections are needed, create a
new topic file (only possible during TOPIC phase, before any changelist
exists).

This ensures full traceability. Git history preserves every previous
version of the design. No accidental drift from partially-applied edits.

### 10.20 Deprecation, not addendum

If a changelist needs revision after it was written, **deprecate** it
and write a new one. Never add addendums. The new changelist must
include a "Comparison to deprecated changelist" section explaining
what changed and why.

Deprecated changelists are preserved in the archive as history. They
are never deleted. This ensures every contributor can reconstruct the
full design evolution.
