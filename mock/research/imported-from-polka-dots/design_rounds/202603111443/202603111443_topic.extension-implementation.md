# Topic: First-Party Extension Implementation

**Date:** 2026-03-11
**Status:** PRELIMINARY ‚Äî pending interactive review before promoting to DOC phase

## Context

The mock workspace has 6 first-party extension crates (cdylib, depend on SDK only):
- `polka-lang-polka` ‚Äî the `.polka` DSL language plugin
- `polka-source-brew` ‚Äî Homebrew/Cask/Tap source
- `polka-source-mas` ‚Äî Mac App Store source
- `polka-platform-macos` ‚Äî macOS platform
- `polka-platform-linux` ‚Äî Linux platform
- `polka-keybinds` ‚Äî keybind resolution

All are stubs or minimal implementations. This round covers implementing them
with real logic.

## Current state per extension

### polka-lang-polka ‚Äî skeleton only
- `process()` returns source unchanged
- Needs: pest grammar, parser, AST nodes, macro expander, builtin dispatch,
  resolution pass
- Complexity: HIGH ‚Äî essentially rebuilding the nuked DSL correctly

### polka-source-brew ‚Äî all methods are no-ops
- Five Source methods all return hardcoded empty results
- Needs: shell-out to `brew list`, `brew install`, `brew outdated --json`
- Missing from SDK: tap concept, update/upgrade, source availability check
- Complexity: MEDIUM

### polka-source-mas ‚Äî identical empty shape
- Needs: shell-out to `mas list`, `mas install`, `mas outdated`
- Package IDs are numeric (app store IDs), not names
- `mas` may not be installed ‚Äî needs graceful detection
- Complexity: LOW-MEDIUM

### polka-platform-macos ‚Äî partially real
- `detect()`, `config_dir()`, `home_dir()` work correctly
- Missing from Platform trait: `arch()`, version detection, brew prefix
  path (`/opt/homebrew` vs `/usr/local`), launchctl integration,
  `data_dir()`, `cache_dir()`
- Complexity: LOW (current trait), MEDIUM (with expanded trait)

### polka-platform-linux ‚Äî partially real
- `detect()`, `config_dir()` (with XDG), `home_dir()` work correctly
- Missing: distro detection (`/etc/os-release`), WSL detection
  (`/proc/version`), systemd detection, distro-specific paths
- Complexity: LOW-MEDIUM

### polka-keybinds ‚Äî data types real, resolution stubbed
- `ResolvedKeybind` and `KeybindFormat` well-defined
- Working implementation exists in `src/keybinds.rs` with 4 formats
  (skhd, aerospace, emacs, display) ‚Äî needs porting
- Complexity: MEDIUM (porting existing work)

## SDK contract gaps blocking extensions

### Source trait gaps
- No `update(package)` / `upgrade(package)` method
- No `available(package)` check ("does this source know about this?")
- No tap/repository concept (brew tap != brew install)
- No `generate_lockfile()` / `export()` for Brewfile generation
- No execution context (timeout, dry-run, env override, progress callback)
- No `is_available()` to check if the backing tool exists

### Language trait gaps
- No source file path ‚Äî errors can't reference originating file
- Single output only ‚Äî can't fan out to multiple files
- No partial results ‚Äî one error loses all work, no warnings
- `LanguageError` has `line`/`column` but no `file` field

### Platform trait gaps
- No `arch()` or `is_apple_silicon()`
- No `distro()` / `distro_info()`
- No `is_wsl()`
- No `service_manager()` for launchctl/systemd
- No `data_dir()`, `cache_dir()`

### KeybindFormat not in SDK
- Defined in `polka-keybinds`, not `polka-sdk`
- Other extensions can't register keybind formats without depending
  on `polka-keybinds`, violating "extensions depend on SDK only"

### DataValue ‚Äî no array iteration
- No `as_array()` or `iter()` ‚Äî can't iterate TOML arrays through
  the `DataValue` trait

## Shared patterns needing SDK support

### Shell command execution
Both source and platform plugins need `Command::new().args().output()` with
exit-code-to-error mapping. Should be a `polka_sdk::process` utility.

### Source availability precheck
Every source needs "is my backing tool installed?" ‚Äî identical shape
across brew, mas, apt, etc. Should be `Source::is_available()` or an
SDK helper.

### HOME / XDG resolution
Both platforms duplicate `env::var_os("HOME")` logic. SDK should provide
`polka_sdk::paths::home_dir()`.

### `#[register]` macro is a stub
Generates no descriptor, no `inventory::submit!`. ALL registration is
silently non-functional.

## Missing extensions (not in scope for this round, but noted)

- `polka-source-apt` / `polka-source-dnf` / `polka-source-pacman` ‚Äî Linux
- `polka-source-nix` ‚Äî NixOS
- `polka-platform-windows` / `polka-platform-wsl`
- `polka-format-*` ‚Äî Transform implementations (toml, json, ini, yaml, kdl)
  ‚Äî zero exist anywhere in the mock
- `polka-format-skhd` / `polka-format-aerospace` / etc. ‚Äî keybind formats
  as separate registered plugins

## Implementation order (preliminary)

1. Fix SDK contract gaps first (core-design-review round must land first)
2. Platform plugins (simplest, fewest SDK deps)
3. Source plugins (need shell execution helper from SDK)
4. Keybind resolver (port existing working code)
5. Language plugin (highest complexity, depends on all SDK contracts being stable)
