# Topic: Core Design Review — Holes, Smells, Improvements

**Date:** 2026-03-11
**Status:** PRELIMINARY — pending interactive review before promoting to DOC phase

## Context

The architecture-signatures round (`202603110821`) established the plugin-first
architecture with 17 crates across SDK / Core:Data / Core:Execution / CLI layers,
plus first-party extensions. All crates compile with real trait/struct/enum signatures.

This round reviews the core design for:
- Missing abstractions
- Design smells (wrong boundaries, leaky abstractions, premature thinning)
- Code smells (duplication, inconsistency, type mismatches)
- Dependency graph issues

## Preliminary findings

### Critical (blocks real implementation)

1. **No concrete Registry implementations** — `Registry` trait exists in primitives
   but no `LanguageRegistry`, `SourceRegistry`, etc. The bridge between
   `inventory::submit!` and runtime dispatch is entirely absent.

2. **No concrete Descriptor types** — `Descriptor` trait defined but no
   `LanguageDescriptor`, `SourceDescriptor`, etc. The cross-dylib dispatch
   mechanism described in design docs has zero concrete representation.

3. **`DataValue` has no array variant** — TOML has first-class arrays. `DataEntry`
   enumerates String/Int/Bool/Null/Map but no Array. Array data falls into a
   black hole when `polka-project::load()` reads polka.toml.

4. **`StageContext` is too thin** — carries `project_root` and `data` but not
   `catalogue` or `files`. Pipeline stages can't see resolved files without
   re-resolving (which requires I/O, violating the no-I/O rule).

5. **Language -> Transform handoff is untyped** — `Language::process()` returns
   `String`, `Transform::render()` takes `&dyn DataValue`. No intermediate IR
   connects the two. The pipeline data flow has a gap.

6. **`pipeline::run()` stage discovery** — takes `&[&dyn PipelineStage]` from
   the caller but the design says stages are inventory-registered. No
   `discover_stages()` or `from_registry()` exists.

7. **`DataStore::data()` returns empty data** — `add_layer()` appends to layers
   but never updates `merged`. The `data()` accessor always returns empty.

### Medium (design inconsistencies, will cause friction)

8. **Error types don't implement `std::error::Error`** — `LanguageError`,
   `SourceError`, `TransformError`, `StageError` implement `Display` but not
   `Error`. Can't use `?` into `anyhow::Result` without manual conversion.

9. **`StageName`/`RoutineName` newtypes exist but aren't used** —
   `PipelineResult.stages` keys on `String`, `SchedulerState.last_run` keys on
   `String`, despite dedicated newtypes existing in primitives.

10. **`BuildManifest` and `DeployManifest` are identical** — same fields, same
    methods, same logic. Textbook code duplication.

11. **`Vec<String>` platform filtering** — `Routine.platforms` and
    `CatalogueEntry.platforms` use `Vec<String>` instead of the `Platform` trait.
    Violates the "no string comparison for platforms" rule.

12. **`audit()` can't discover Sources** — depends on `polka-sdk` but not
    `inventory`. Has no way to call `inventory::iter()` for registered sources.
    Stub comment says "queries Source impls via inventory" — impossible.

13. **`inventory` on SDK is unused** — SDK declares `inventory` dep but never
    calls it. Extensions need `inventory` directly or re-exported, neither exists.

14. **`polka-keybinds` layer placement** — declared cdylib extension but
    participates in the core pipeline. `ResolvedKeybind` type can't flow back
    to `polka-pipeline` because no core crate depends on an extension.

15. **Duplicate `#[public_api]` macros** — `polka-visibility` and `polka-derive`
    both define independent `#[public_api]` proc macros. Will drift when enhanced.

### Low (inconsistencies, minor gaps)

16. **`SettingsValue::parse()` returns `Result<Self, String>`** — bare String
    error, no `SettingsError` type. Not machine-readable.

17. **`Schedule::Cron` can't express wildcards** — `Option<u8>` conflates
    absent with wildcard. Missing `day_of_month`, `month`. Can't express
    "every hour on the 15th minute on weekdays."

18. **Missing `Default` derives** — `DataMap`, `DataStore`, `Catalogue`,
    `BuildManifest`, `DeployManifest`, `SchedulerState` all have `fn new()`
    but no `#[derive(Default)]`.

19. **`BuildManifest::is_stale()` takes `&PathBuf` not `&Path`** — forces
    owned reference where borrowed would do.

20. **`audit()` takes `&Catalogue` not `&LoadedProject`** — bypasses the
    "all commands go through `load()`" rule.

21. **`polka-cli` missing direct deps on `polka-data`/`polka-catalogue`** —
    relies on transitive resolution through `polka-project`.

22. **`DataValue::keys()` returns `Option<Vec<&str>>`** — allocates on every
    call. Should return an iterator.

23. **`Named::name()` returns `&'static str`** — prevents runtime-constructed
    registrables (e.g., routines loaded from config).

24. **`polka-scheduler` missing `anyhow` dep** — will need it for persistence.

25. **`CatalogueEntry.metadata` is flat `BTreeMap<String, String>`** — no
    `ServiceConfig` struct for service entries despite `is_service: bool` flag.

26. **`Source::installed()` returns `Vec<String>`** — no typed package
    identifier. Cross-source reconciliation is stringly-typed.

27. **`KeybindResolver` not registered** — implements `Named`/`Display` but
    no `#[register] impl` for any trait. No integration point with the pipeline.

28. **`FilePattern` has no `platforms` field** — platform-filtered file
    resolution mentioned in design but no field to filter on.

29. **`polka-audit` depends on `polka-catalogue` directly** — creates a path
    that bypasses `polka-project::load()`.
