## 1. The Store<T> Abstraction

The central problem Q1 poses: singletons and collections must create DAG edges
through the same mechanism, but consumers access them differently. A singleton is
one value of type T, accessed as `&T` or `&mut T`. A collection is N values of
type T, accessed as an iterator over `&T` or `&mut T`, potentially
morsel-chunked. The scheduler cares about who reads T and who writes T. It does
not care about N.

The Store trait is the common interface. It identifies a named, typed slot in the
world. It does not describe how data is stored; it describes what the scheduler
tracks.

```rust
/// Marker trait for anything the scheduler can track as a dependency.
///
/// Implementors identify a named, typed storage slot. The scheduler
/// builds DAG edges from Store identifiers; it never inspects the
/// backing memory.
pub trait Store: 'static {
    /// The element type held in this store.
    type Item: Send + Sync + 'static;

    /// Whether this store holds one element, many elements, or no
    /// elements (pure ordering edge).
    const CARDINALITY: Cardinality;

    /// Unique identifier for this store, used for DAG edge matching.
    /// Derived from the type itself via define_id! or equivalent.
    fn store_id() -> StoreId;
}
```

Cardinality is a closed enum with three variants:

```rust
#[repr(u8)]
pub enum Cardinality {
    /// Exactly one value. Accessed via direct reference.
    Single = 0,
    /// Zero or more values. Accessed via iteration.
    Many = 1,
    /// No backing storage. Exists only for DAG edge creation.
    Virtual = 2,
}
```

The scheduler does not branch on cardinality. It sees `StoreId` values in
`AccessSet` declarations and builds edges from write-to-read matches. Cardinality
influences only the access context: which methods are available on the context
object inside `execute()`. This separation is the core insight that makes
unification work.

### Why one trait, not three

An earlier design sketch used three separate traits: `ResourceStore`,
`ColumnStore`, `VirtualStore`. Each carried its own `store_id()` and its own
entry in the access set. The scheduler would need to compare across three
namespaces, tripling the edge construction logic. Worse, a consumer that wanted
to refactor a singleton into a collection (common during development: a global
config becomes per-entity) would need to change every WorkUnit declaration from
`Read<Resource<Config>>` to `Read<Column<Config>>`, propagating through the
entire codebase.

With a single `Store` trait, refactoring from singleton to collection changes one
type alias and the backing storage. WorkUnit declarations stay the same because
the scheduler edge is `StoreId`-based, not cardinality-based. The access context
methods change (`.resource::<S>()` becomes `.column::<S>()`) but those changes
are local to the WorkUnit bodies, not to their declarations.

### StoreId generation

`StoreId` must be unique per Store implementation, deterministic across
compilations, and fast to compare. `std::any::TypeId` is banned (per project
rules). The replacement is `define_id!`-generated interned identifiers. Each
`Store` impl gets a `StoreId` derived from the type's module path and name,
interned at link time via `inventory` or `linkme`. Comparison is a single `u64`
equality check.

```rust
/// Interned store identifier. Equality is pointer-width integer comparison.
#[derive(Copy, Clone, Eq, PartialEq, Hash)]
pub struct StoreId(u64);
```

The `u64` value is a hash of the fully qualified type path, computed at compile
time. Collisions are detected at startup by the registration sweep (iterate all
registered stores, check for duplicate `StoreId` values with different type
names, panic if found). This is O(n) at startup, where n is the number of
registered stores (typically 20-100), so the cost is negligible.

## 2. Type-Level Singleton vs Collection

Should `Resource<T>` and `Column<T>` be different types implementing `Store`, or
should there be one `Store<T>` with a runtime cardinality field? This decision
affects compile-time safety, ergonomics, and the complexity of the access context.

### Option A: type-level (compile-time)

```rust
pub struct Resource<T>(PhantomData<T>);
impl<T: Send + Sync + 'static> Store for Resource<T> {
    type Item = T;
    const CARDINALITY: Cardinality = Cardinality::Single;
    fn store_id() -> StoreId { /* hash of "Resource<T>" */ }
}

pub struct Column<T>(PhantomData<T>);
impl<T: Send + Sync + 'static> Store for Column<T> {
    type Item = T;
    const CARDINALITY: Cardinality = Cardinality::Many;
    fn store_id() -> StoreId { /* hash of "Column<T>" */ }
}

pub struct Virtual<T>(PhantomData<T>);
impl<T: Send + Sync + 'static> Store for Virtual<T> {
    type Item = T;
    const CARDINALITY: Cardinality = Cardinality::Virtual;
    fn store_id() -> StoreId { /* hash of "Virtual<T>" */ }
}
```

With type-level cardinality, the compiler enforces correct access. A WorkUnit
that declares `type Read = (Resource<AppConfig>,)` gets a context where
`.resource::<Resource<AppConfig>>()` returns `&AppConfig`. Calling
`.column::<Resource<AppConfig>>()` is a compile error because `Resource<T>`
implements a `SingleAccess` marker trait but not `ManyAccess`.

```rust
// Compile-time enforcement via trait bounds
impl<'a, R, W> Context<'a, R, W> {
    pub fn resource<S>(&self) -> &S::Item
    where
        S: Store<CARDINALITY = {Cardinality::Single}>,
        R: Contains<S>,
    { /* ... */ }

    pub fn column<S>(&self) -> ColumnRef<'_, S::Item>
    where
        S: Store<CARDINALITY = {Cardinality::Many}>,
        R: Contains<S>,
    { /* ... */ }
}
```

The `CARDINALITY = {Cardinality::Single}` constraint uses const generics in
where clauses. Rust does not support this syntax directly today, but it can be
approximated with marker traits:

```rust
pub trait SingleStore: Store {}
pub trait ManyStore: Store {}
pub trait VirtualStore: Store {}

impl<T: Send + Sync + 'static> SingleStore for Resource<T> {}
impl<T: Send + Sync + 'static> ManyStore for Column<T> {}
impl<T: Send + Sync + 'static> VirtualStore for Virtual<T> {}
```

### Option B: value-level (runtime)

```rust
pub struct AnyStore<T> {
    cardinality: Cardinality,
    _marker: PhantomData<T>,
}
```

The scheduler treats all stores identically. Access methods check cardinality at
runtime:

```rust
impl<'a, R, W> Context<'a, R, W> {
    pub fn resource<S: Store>(&self) -> &S::Item {
        assert!(S::CARDINALITY == Cardinality::Single, "not a singleton");
        // ...
    }
}
```

This compiles but panics at runtime if a WorkUnit tries to use `.resource()` on
a collection. The safety is weaker: the mistake is caught at test time, not
compile time.

### What the references do

loimu uses type-level. `define_resource!` and `define_record!` produce different
types with different access patterns. The scheduler does not distinguish them
(both produce typed DAG edges), but the access API does (`resource_ref::<T>()`
vs `each (field: &T)`).

Bevy uses type-level. `Res<T>` and `Query<&T>` are different system parameter
types. Both create scheduling edges through the same mechanism
(`SystemParamState::access`), but the access API is completely different
(`.into_inner()` vs `.iter()`).

EnTT (C++) uses type-level. `registry.ctx<T>()` for singletons, `registry.view<T>()` for components. Same registry, different access patterns.

DuckDB uses type-level internally. Catalog entries (singleton metadata) and table
columns (multi-row data) are stored differently and accessed through different
APIs, but the transaction manager tracks dependencies on both uniformly.

All four references confirm: type-level for access ergonomics, unified at the
scheduler level. No reference uses runtime cardinality checks.

### Recommendation

Type-level. `Resource<T>`, `Column<T>`, and `Virtual<T>` are different types
implementing `Store`. The scheduler sees `StoreId` values and does not
distinguish cardinality. The access context provides different methods based on
marker traits (`SingleStore`, `ManyStore`). Compile-time errors for cardinality
misuse. No runtime panics for wrong access patterns.

The consequence: `Resource<AppConfig>` and `Column<AppConfig>` are different
stores with different `StoreId` values. A WorkUnit that writes
`Resource<AppConfig>` does not create an edge to a WorkUnit that reads
`Column<AppConfig>`. This is correct: a singleton config and a column of configs
are semantically different things. If a consumer wants a single shared config, it
uses `Resource<AppConfig>`. If it wants per-entity configs, it uses
`Column<AppConfig>`. The two do not interfere.

## 3. Memory Layout for Singletons vs Collections

Both singletons and collections are backed by `SlabColumn<T>`. The question is
whether this unification is wasteful for singletons.

### SlabColumn overhead for a singleton

SlabColumn reserves virtual address space via `mmap(PROT_NONE)`. The minimum
physical commitment is one page (4 KB on most platforms). For a 16-byte resource
(a config struct with a few fields), the memory layout is:

```
Virtual reservation:  [============================================]
                       ^ 64 KB minimum reservation (configurable)
Physical commitment:  [====]
                       ^ 4 KB (one page, demand-committed)
Actual data:          [T]
                       ^ 16 bytes of actual data
Wasted:               4096 - 16 = 4080 bytes per singleton
```

With 50 singletons, that is 50 pages of physical memory (200 KB) for perhaps
800 bytes of actual data. The overhead is real but small: 200 KB is irrelevant
on any machine that runs polka, saalis, or loimu. Virtual address space usage is
50 * 64 KB = 3.2 MB, which is nothing on a 48-bit (256 TB) address space.

### Alternative: dedicated singleton storage

Bevy stores resources in a `HashMap<ComponentId, ResourceData>` separate from
entity component storage. This avoids per-resource page overhead. The tradeoff:

| Aspect | SlabColumn singleton | Dedicated HashMap |
|--------|---------------------|-------------------|
| Memory overhead per singleton | 4 KB (one page) | ~64 bytes (HashMap entry + allocation) |
| Access latency | Direct pointer (base + 0) | Hash lookup + pointer chase |
| Code complexity | One storage path | Two storage paths |
| Dirty tracking | Same DirtyTracker | Separate mechanism needed |
| Version stamps | Same VersionedResource wrapper | Same (works either way) |
| Aliasing analysis | Participates naturally | Excluded or special-cased |

The HashMap approach saves memory (64 bytes vs 4096 bytes per singleton) but
adds a code path. Every place that touches storage must branch on "is this a
singleton or a collection?" The dirty tracker needs to handle both. The aliasing
analysis needs to either include or exclude singletons. The version stamp
mechanism is the same either way, but the storage location differs.

### loimu's approach

loimu stores resources in SlabColumn with cardinality 1. A resource is a view
with one slot. The overhead per resource is one page of physical memory. loimu
accepts this because the alternative (dual storage paths) creates maintenance
burden that outweighs the memory savings. With ~30 framework resources and ~20
app resources, the total overhead is 200 KB. Not worth optimizing.

### Recommendation for hilavitkutin

Use SlabColumn for both. One storage path, one dirty tracker, one aliasing
analysis, one version stamp mechanism. The memory overhead (4 KB per singleton
instead of 64 bytes) is negligible. The code simplicity gain is significant: no
branching on storage kind anywhere in the scheduler, executor, or dirty tracker.

If a consumer has hundreds of singletons (unlikely but possible), a future
optimization can introduce a small-object pool that packs multiple singletons
into a single page. This optimization is internal to the storage engine and
invisible to the scheduler and access context.

### Concrete layout: singleton

```
SlabColumn<AppConfig>:
  virtual_base:  0x7f0000100000
  committed:     1 page (4096 bytes)
  stride:        size_of::<AppConfig>() aligned
  count:         1

  [slot 0: AppConfig { theme: "dark", indent: 4, ... }]
  [unused: 4096 - sizeof(AppConfig) bytes]
```

### Concrete layout: collection

```
SlabColumn<Entity>:
  virtual_base:  0x7f0000200000
  committed:     N pages (N * 4096 bytes, demand-paged)
  stride:        size_of::<Entity>() aligned
  count:         variable (grows as entities are added)

  [slot 0: Entity { id: 0, name: "firefox", ... }]
  [slot 1: Entity { id: 1, name: "vim", ... }]
  [slot 2: Entity { id: 2, name: "tmux", ... }]
  ...
  [slot N-1: last entity]
  [uncommitted pages beyond slot N-1]
```

### Concrete layout: virtual

```
Virtual<EntityDiscovered>:
  No SlabColumn allocated.
  No virtual reservation.
  No physical pages.
  StoreId exists in the AccessSet.
  DAG edges created from StoreId matching.
  Context provides no access methods for this store.
```

## 4. Version Stamps on Singletons vs Collections

Every store needs change tracking for two purposes: (1) the scheduler's
early-cutoff optimization (skip downstream units if nothing actually changed),
and (2) incremental rebuild (polka skips expansion if the parser produced the
same AST).

### Granularity options

| Granularity | Description | Cost per check | Phantom write detection |
|---|---|---|---|
| Per-Store | One stamp per `Resource<T>` or `Column<T>` | O(1) | Conservative: any write to any element bumps the stamp |
| Per-Element | One stamp per row in a collection | O(N) to scan all | Precise: only changed rows bump their stamp |
| Per-Column-Field | One stamp per field within a struct (SoA decomposition) | O(fields) | Field-precise: unchanged fields keep old stamp |
| Per-Morsel | One stamp per chunk of rows | O(N/M) where M is morsel size | Chunk-level granularity |

### What loimu does

loimu uses per-resource stamps. A resource (singleton or record type) has one
`version: u64`. Any write to any field increments the stamp. The scheduler
captures versions before a unit runs and compares after. If the version did not
change, downstream units depending on that resource can skip. This is the
early-cutoff optimization.

Phantom write detection uses `memcmp` for resources under 4 KB. If the unit
writes a value that is bitwise identical to the existing value, the version stamp
is not incremented. This catches the common case of "read config, compute same
result, write back." For resources larger than 4 KB, the developer calls
`.mark_changed()` explicitly.

### Per-element stamps: when do they pay off?

Consider a collection with 10,000 entities. A WorkUnit writes to 5 of them. With
per-store stamps, the stamp increments once, and all downstream units that read
the collection must re-execute, even if they only care about the 9,995 unchanged
entities.

With per-element stamps, each of the 5 written elements gets a bumped stamp.
Downstream units can check per-element and skip the unchanged ones. But the
check cost is O(N) to scan all stamps, which may be more expensive than just
re-processing the unchanged elements.

The break-even depends on per-element processing cost. If processing one element
takes 1 microsecond and stamp checking takes 1 nanosecond, per-element stamps
save time when fewer than 99.9% of elements change. If processing takes 10
nanoseconds and stamp checking takes 1 nanosecond, the overhead of checking
dominates when more than ~10% of elements change.

For polka (processing dotfiles with 100-1000 entries), per-store stamps are
sufficient. For saalis (processing query results with 10,000-100,000 rows),
per-morsel stamps might help. For loimu (processing UI state with 100-10,000
elements), per-store stamps are what they already use and find adequate.

### Recommendation

Per-Store stamps for the initial design. One `Version` (u64) per `Resource<T>`
and one `Version` per `Column<T>`. The `VersionedStore<S: Store>` wrapper
handles increment-on-write and memcmp-based phantom detection.

```rust
pub struct VersionedStore<S: Store> {
    version: Version,
    storage: SlabColumn<S::Item>,
}

impl<S: Store> VersionedStore<S> {
    /// Returns the current version stamp.
    pub fn version(&self) -> Version {
        self.version
    }

    /// Write a singleton value. Increments version only if the value
    /// actually changed (memcmp for types under 4 KB, unconditional
    /// increment for larger types).
    pub fn write_single(&mut self, value: S::Item) -> bool
    where
        S: SingleStore,
    {
        let changed = !byte_equal(&self.storage[0], &value);
        if changed {
            self.storage[0] = value;
            self.version.increment();
        }
        changed
    }

    /// Acquire mutable access to the collection. The version stamp
    /// is unconditionally incremented because tracking per-element
    /// changes across arbitrary mutation is not feasible at this
    /// granularity level.
    pub fn write_many(&mut self) -> ColumnMut<'_, S::Item>
    where
        S: ManyStore,
    {
        self.version.increment();
        ColumnMut { slab: &mut self.storage }
    }
}
```

Per-morsel stamps can be layered on later. The `DirtyTracker` already provides
hierarchical bitset tracking that maps naturally to morsel-level granularity. A
future `MorselVersionedStore` would use `DirtyTracker` to track which morsels
were written and expose `dirty_morsels() -> impl Iterator<Range<usize>>` for
downstream units to check. This optimization is invisible to the scheduler; it
only affects the access context internals.

## 5. Access Context Design

The context object that `execute()` receives. This is where the type-level
singleton/collection distinction becomes ergonomically relevant.

### The Context type

```rust
pub struct Context<'a, R: AccessSet, W: AccessSet> {
    store_map: &'a StoreMap,
    _read: PhantomData<R>,
    _write: PhantomData<W>,
}
```

`StoreMap` is the runtime container holding all `VersionedStore` instances. It is
keyed by `StoreId` and stores type-erased pointers. The context does not own any
data; it borrows from the `StoreMap` for the duration of `execute()`.

### Read access: singletons

```rust
impl<'a, R: AccessSet, W: AccessSet> Context<'a, R, W> {
    /// Read a singleton resource. Returns a shared reference that is
    /// valid for the duration of execute().
    ///
    /// Compile error if S is not in the Read set.
    /// Compile error if S is not a SingleStore.
    pub fn resource<S>(&self) -> &S::Item
    where
        S: SingleStore,
        R: Contains<S>,
    {
        let store = self.store_map.get::<S>();
        // Safety: S has cardinality Single, so slot 0 is always valid.
        // The scheduler guarantees no concurrent writer exists.
        unsafe { store.storage.get_unchecked(0) }
    }
}
```

The `Contains<S>` bound is a compile-time proof that S appears in the Read
access set. If a WorkUnit declares `type Read = (Resource<AppConfig>,)` and its
body calls `ctx.resource::<Resource<DbPool>>()`, the `Contains` bound fails
because `DbPool` is not in the Read set. This is a compile error, not a runtime
panic.

### Read access: collections

```rust
impl<'a, R: AccessSet, W: AccessSet> Context<'a, R, W> {
    /// Read a collection. Returns a column reference supporting
    /// iteration and random access.
    pub fn column<S>(&self) -> ColumnRef<'_, S::Item>
    where
        S: ManyStore,
        R: Contains<S>,
    {
        let store = self.store_map.get::<S>();
        ColumnRef {
            slab: &store.storage,
            len: store.len(),
        }
    }
}
```

`ColumnRef` supports three iteration modes:

```rust
pub struct ColumnRef<'a, T> {
    slab: &'a SlabColumn<T>,
    len: usize,
}

impl<'a, T> ColumnRef<'a, T> {
    /// Full linear iteration.
    pub fn iter(&self) -> impl Iterator<Item = &T> {
        (0..self.len).map(move |i| unsafe { self.slab.get_unchecked(i) })
    }

    /// Morsel-chunked iteration. The callback receives slices of
    /// morsel_size elements (last chunk may be smaller).
    pub fn morsels(&self, morsel_size: usize) -> impl Iterator<Item = &[T]> {
        self.slab.as_slice()[..self.len].chunks(morsel_size)
    }

    /// Filtered iteration via selection vector.
    pub fn select<'s>(&self, sel: &'s SelectionVector) -> impl Iterator<Item = &T> + 's
    where
        'a: 's,
    {
        sel.indices().map(move |&idx| unsafe { self.slab.get_unchecked(idx as usize) })
    }

    /// Random access by index.
    pub fn get(&self, index: usize) -> Option<&T> {
        if index < self.len {
            Some(unsafe { self.slab.get_unchecked(index) })
        } else {
            None
        }
    }

    /// Element count.
    pub fn len(&self) -> usize {
        self.len
    }
}
```

### Write access: singletons

```rust
impl<'a, R: AccessSet, W: AccessSet> Context<'a, R, W> {
    /// Write a singleton resource. The closure receives a mutable
    /// reference. Version stamp is incremented after the closure
    /// returns if the value changed.
    pub fn resource_mut<S, F, Ret>(&mut self, f: F) -> Ret
    where
        S: SingleStore,
        W: Contains<S>,
        F: FnOnce(&mut S::Item) -> Ret,
    {
        let store = self.store_map.get_mut::<S>();
        let slot = unsafe { store.storage.get_unchecked_mut(0) };

        // Snapshot for phantom write detection (types under 4 KB).
        let snapshot = if core::mem::size_of::<S::Item>() <= 4096 {
            Some(unsafe { core::ptr::read(slot as *const S::Item) })
        } else {
            None
        };

        let ret = f(slot);

        // Increment version only if value actually changed.
        match snapshot {
            Some(ref old) if byte_equal(old, slot) => { /* phantom write */ }
            _ => store.version.increment(),
        }

        ret
    }
}
```

The closure-based API ensures the version stamp check happens after the mutation
is complete. A direct `&mut T` return would require the caller to signal when
they are done writing, which is error-prone.

### Write access: collections

```rust
impl<'a, R: AccessSet, W: AccessSet> Context<'a, R, W> {
    /// Write to a collection. Returns a mutable column reference.
    /// Version stamp is unconditionally incremented.
    pub fn column_mut<S>(&mut self) -> ColumnMut<'_, S::Item>
    where
        S: ManyStore,
        W: Contains<S>,
    {
        let store = self.store_map.get_mut::<S>();
        store.version.increment();
        ColumnMut {
            slab: &mut store.storage,
            len: &mut store.len,
        }
    }
}
```

`ColumnMut` supports push, swap-remove, and mutable iteration:

```rust
pub struct ColumnMut<'a, T> {
    slab: &'a mut SlabColumn<T>,
    len: &'a mut usize,
}

impl<'a, T> ColumnMut<'a, T> {
    pub fn push(&mut self, value: T) {
        let idx = *self.len;
        self.slab.set(idx, value);
        *self.len += 1;
    }

    pub fn swap_remove(&mut self, index: usize) -> T {
        assert!(index < *self.len, "index out of bounds");
        *self.len -= 1;
        if index == *self.len {
            unsafe { self.slab.take(index) }
        } else {
            let last = unsafe { self.slab.take(*self.len) };
            let removed = unsafe { core::mem::replace(self.slab.get_unchecked_mut(index), last) };
            removed
        }
    }

    pub fn iter_mut(&mut self) -> impl Iterator<Item = &mut T> {
        (0..*self.len).map(move |i| unsafe { self.slab.get_unchecked_mut(i) })
    }
}
```

### Virtual stores: no access methods

Virtual stores have no backing storage. The context provides no methods for them.
A WorkUnit that declares `type Write = (Virtual<EntityDiscovered>,)` does not
call any context method to "write" the virtual store. The DAG edge is created
from the declaration alone. The execute body does whatever work it needs to do;
the fact that it declared `Write<Virtual<EntityDiscovered>>` is a pure ordering
constraint that tells the scheduler "run this before anything that reads
EntityDiscovered."

```rust
// This WorkUnit writes Virtual<EntityDiscovered> but never calls
// any context method for it. The declaration alone creates the edge.
impl WorkUnit for CatalogScanner {
    type Read = (Resource<HttpClient>,);
    type Write = (Column<Entities>, Virtual<EntityDiscovered>);

    fn execute(&mut self, ctx: &mut Context<Self::Read, Self::Write>) -> Progress {
        let client = ctx.resource::<Resource<HttpClient>>();
        let mut entities = ctx.column_mut::<Column<Entities>>();
        // ... scan catalog, push entities ...
        // No mention of EntityDiscovered in the body.
        Progress::Complete
    }
}
```

## 6. DAG Edge Unification

The scheduler builds a directed acyclic graph where edges represent data
dependencies. Edge construction is the same regardless of store cardinality.

### Edge construction algorithm

```
for each pair (unit_a, unit_b) where a != b:
    for each store_id in unit_a.writes():
        if store_id in unit_b.reads():
            dag.add_edge(unit_a, unit_b)
```

This is O(U^2 * W * R) in the worst case, where U is the number of units, W is
the max write set size, and R is the max read set size. In practice, U is
10-100, W is 1-3, R is 1-5, so the total comparisons are 100-15000. Each
comparison is a single u64 equality check. The entire DAG construction takes
microseconds.

### Concrete example

```rust
impl WorkUnit for ConfigLoader {
    type Read = ();
    type Write = (Resource<AppConfig>,);
    // Reads nothing, writes the config singleton.
}

impl WorkUnit for CatalogScanner {
    type Read = (Resource<AppConfig>, Resource<HttpClient>);
    type Write = (Column<Entities>, Virtual<EntityDiscovered>);
    // Reads config and HTTP client, writes entity collection and
    // emits the EntityDiscovered ordering signal.
}

impl WorkUnit for EntityEnricher {
    type Read = (Virtual<EntityDiscovered>, Column<Entities>);
    type Write = (Column<EnrichedEntities>,);
    // Waits for EntityDiscovered, reads raw entities, writes enriched.
}

impl WorkUnit for ReportGenerator {
    type Read = (Column<EnrichedEntities>, Resource<AppConfig>);
    type Write = (Resource<Report>,);
    // Reads enriched entities and config, writes report singleton.
}
```

DAG edges derived from these declarations:

```
ConfigLoader --[AppConfig]--> CatalogScanner
ConfigLoader --[AppConfig]--> ReportGenerator
CatalogScanner --[EntityDiscovered]--> EntityEnricher
CatalogScanner --[Entities]--> EntityEnricher
EntityEnricher --[EnrichedEntities]--> ReportGenerator
```

The scheduler sees five edges. It does not know that AppConfig is a singleton,
Entities is a collection, and EntityDiscovered is virtual. All edges are
StoreId-based. The topological sort produces:

```
Step 1: ConfigLoader (no dependencies)
Step 2: CatalogScanner (depends on ConfigLoader via AppConfig)
Step 3: EntityEnricher (depends on CatalogScanner via EntityDiscovered + Entities)
Step 4: ReportGenerator (depends on EntityEnricher via EnrichedEntities,
                         depends on ConfigLoader via AppConfig — already satisfied)
```

### Write-write conflicts

Two units that both write the same store create a conflict. The scheduler must
serialize them. The edge direction is determined by topological depth: the unit
with lower depth runs first. If both have the same depth, the scheduler picks
arbitrarily (deterministic tie-breaking by StoreId hash).

```rust
impl WorkUnit for WriterA {
    type Write = (Column<Entities>,);
}
impl WorkUnit for WriterB {
    type Write = (Column<Entities>,);
}
// Conflict: both write Column<Entities>.
// Scheduler adds edge WriterA -> WriterB (or B -> A, based on depth).
// They never run in parallel.
```

### Read-read parallelism

Two units that both read the same store have no conflict. They can run in
parallel. No edge is created between them.

```rust
impl WorkUnit for ReaderA {
    type Read = (Column<Entities>,);
}
impl WorkUnit for ReaderB {
    type Read = (Column<Entities>,);
}
// No conflict. No edge. Can run in parallel.
```

### Read-write ordering

A reader and a writer of the same store get a directed edge: writer before
reader (if the reader depends on the writer's output) or reader before writer
(if the writer must see the reader's version). The default is writer-before-reader,
which matches the natural data flow direction. Reverse ordering requires explicit
annotation.

## 7. Virtual Columns in the Unified Model

Virtual columns are zero-sized types that create DAG edges without allocating
storage. They are the mechanism for expressing control-flow dependencies that
carry no data.

### Use cases

**Event edges (saalis).** A connector discovers entities and signals downstream
enrichment. No data flows through the signal itself; the data flows through the
entity collection. The signal is purely ordering.

**Phase boundaries (polka).** The parser must complete before expansion begins.
Parser writes `Virtual<ParseComplete>`, expander reads it. No data in the signal;
the AST flows through `Column<AstNode>`.

**Ordering constraints (loimu).** A behavior that must run after another but does
not read its output. The ordering is structural, not data-dependent.

### Memory layout

Virtual stores have no `VersionedStore` entry in the `StoreMap`. The `StoreMap`
tracks their `StoreId` for edge construction but allocates nothing.

```rust
impl StoreMap {
    pub fn register<S: Store>(&mut self) {
        match S::CARDINALITY {
            Cardinality::Single | Cardinality::Many => {
                // Allocate SlabColumn and VersionedStore.
                let store = VersionedStore::<S>::new();
                self.stores.insert(S::store_id(), Box::new(store));
            }
            Cardinality::Virtual => {
                // Register StoreId for edge matching. No allocation.
                self.virtual_ids.insert(S::store_id());
            }
        }
    }
}
```

### Version stamps on virtual stores

Virtual stores have no version stamp. The DAG edge they create is unconditional:
if the writer ran, the reader runs. There is no early-cutoff optimization for
virtual stores because there is no data to compare. The writer's completion is
the signal; the scheduler does not need to check whether "nothing changed."

This is correct because virtual stores express ordering, not data flow. If the
ordering is unnecessary (the writer did not produce meaningful output), the
writer should not declare the virtual write. The scheduler cannot optimize away
an ordering constraint the developer explicitly declared.

### Interaction with yielding

A WorkUnit that declares `Write<Virtual<EntityDiscovered>>` and returns
`Progress::Yield(Pending)` has not satisfied its write. The EntityDiscovered
edge is not satisfied. Downstream readers of EntityDiscovered remain blocked.
This is identical to how data-carrying writes interact with yielding: the edge
is satisfied on `Complete`, not on `Yield`.

## 8. Comparison: hilavitkutin's Model vs Bevy vs loimu

| Aspect | hilavitkutin (proposed) | Bevy 0.15 | loimu |
|--------|------------------------|-----------|-------|
| **Store types** | `Resource<T>`, `Column<T>`, `Virtual<T>` | `Res<T>`, `Query<&T>`, Events (typed) | `define_resource!`, `define_record!`, signals |
| **Type-level cardinality** | Yes (marker traits) | Yes (different parameter types) | Yes (different macro forms) |
| **Scheduler unification** | One edge type (StoreId match) | One edge type (ComponentId conflict) | One edge type (typed access match) |
| **Singleton storage** | SlabColumn with one slot | HashMap<ComponentId, ResourceData> | SlabColumn with one slot |
| **Collection storage** | SlabColumn (dense) | Archetype tables (SoA per archetype) | SlabColumn + views (SoA per view) |
| **Virtual deps** | Zero-sized Store impl, no allocation | Events (allocated, ring-buffered) | Signal bus (arena-allocated, double-buffered) |
| **Version stamps** | Per-Store, memcmp phantom detection | Per-archetype tick + per-component tick | Per-resource, memcmp phantom detection |
| **Access modes** | Read, Write | Ref, Mut, Option, With, Without, Added, Changed | 7 modes (Pinned, Immediate, MutImmediate, Sample, Post, Approx, Defer) |
| **Context type** | `Context<R, W>` with compile-time bounds | System parameters (each borrows from World) | Handler closures with access-analyzed bodies |
| **Dirty tracking** | Hierarchical DirtyTracker bitset | Changed<T> filter on queries | Hierarchical DirtyTracker per-resource |
| **Morsel iteration** | ColumnRef::morsels(size) | Not built-in (manual chunking) | `each` blocks with per-element fusion |
| **Selection vectors** | ColumnRef::select(sel) | Query filters (With, Without, Added, Changed) | Marker-based tree access |
| **Yielding** | Progress::Yield(token) | No (systems run to completion) | No (behaviors run to completion, deferred to post-1.0) |
| **Macros** | Optional (WorkUnit is manually implementable) | Optional (SystemParam derive, but manual impl supported) | Mandatory (define_* macros required) |
| **no_std** | Core is no_std, executor is std-gated | No (requires std) | No (requires std) |

### Key differences from Bevy

Bevy's archetype model groups entities by component set. An entity with
components (A, B) lives in a different archetype table than an entity with
(A, B, C). This optimizes iteration for systems that query specific component
combinations but creates overhead when components are added or removed
(archetype migration). hilavitkutin does not use archetypes. Each column is
independent. Iteration over multiple columns requires the consumer to zip or
join them manually. This is simpler and sufficient for polka/saalis workloads
where entity-component-system patterns are not the primary data model.

Bevy's events are allocated and ring-buffered: events persist for two frames.
hilavitkutin's virtual stores have zero allocation and zero persistence. The
ordering edge is the event. If a consumer needs persistent event data, it uses a
regular `Column<EventData>` instead of a `Virtual<Signal>`.

### Key differences from loimu

loimu's seven access modes are reduced to two (Read, Write). loimu adds its
richer modes as a layer on top. The macro mandate is relaxed: `WorkUnit` is a
trait, not a macro output. loimu's signal bus (arena-allocated, double-buffered)
is replaced by virtual stores (zero-allocation DAG edges). loimu can build its
full signal bus on top of hilavitkutin's virtual stores plus its own arena
storage.

loimu's view system (dynamic affinity analysis, population hysteresis) is not
present in hilavitkutin. Views in hilavitkutin are static: the consumer defines
which columns are co-iterated, and the storage engine lays them out accordingly.
No runtime restructuring.

## 9. Concrete Implementation Sketch

The minimal set of types needed to implement the unified store model. This is
not a complete API; it is the skeleton that the topic document's design decisions
build on.

### Core traits

```rust
// -- store.rs --

use core::marker::PhantomData;

/// Interned store identifier. Comparison is u64 equality.
#[derive(Copy, Clone, Eq, PartialEq, Hash, Debug)]
pub struct StoreId(u64);

/// Store cardinality.
#[derive(Copy, Clone, Eq, PartialEq, Debug)]
#[repr(u8)]
pub enum Cardinality {
    Single = 0,
    Many = 1,
    Virtual = 2,
}

/// A named, typed storage slot tracked by the scheduler.
pub trait Store: 'static {
    type Item: Send + Sync + 'static;
    const CARDINALITY: Cardinality;
    fn store_id() -> StoreId;
}

/// Marker: this store holds exactly one element.
pub trait SingleStore: Store {}
/// Marker: this store holds zero or more elements.
pub trait ManyStore: Store {}
/// Marker: this store holds no data (ordering edge only).
pub trait VirtualStore: Store {}
```

### Concrete store types

```rust
// -- resource.rs --

/// Singleton store. One value of type T.
pub struct Resource<T>(PhantomData<T>);

impl<T: Send + Sync + 'static> Store for Resource<T> {
```

### Access sets

```rust
// -- access.rs --

/// A set of store accesses (reads or writes) declared by a WorkUnit.
pub trait AccessSet {
    /// Returns the StoreId values this set contains.
    fn store_ids() -> &'static [StoreId];
}

/// Compile-time proof that access set A contains store S.
pub trait Contains<S: Store>: AccessSet {}

// Implemented for tuples via macro:
// impl AccessSet for () { ... }
// impl<A: Store> AccessSet for (A,) { ... }
// impl<A: Store, B: Store> AccessSet for (A, B) { ... }
// ...up to 16 elements.
```

The tuple implementations are generated by a macro. Each tuple variant collects
the `StoreId` values from its elements into a static array. The `Contains`
implementations are generated for each element position:

```rust
// Generated by macro for 2-tuples:
impl<A: Store, B: Store> AccessSet for (A, B) {
    fn store_ids() -> &'static [StoreId] {
        // Computed at compile time via const fn.
        &[A::store_id(), B::store_id()]
    }
}

impl<A: Store, B: Store> Contains<A> for (A, B) {}
impl<A: Store, B: Store> Contains<B> for (A, B) {}
```

The `store_ids()` method returns a `&'static [StoreId]` so the scheduler can
iterate over it without allocation. The static lifetime is achieved by leaking
a const-constructed array (or using a `LazyLock` on platforms where const
construction of the hash is not possible).

### WorkUnit trait

```rust
// -- work_unit.rs --

pub trait WorkUnit: Send + 'static {
    type Read: AccessSet;
    type Write: AccessSet;

    fn execute(&mut self, ctx: &mut Context<Self::Read, Self::Write>) -> Progress;
}
```

### StoreMap

```rust
// -- store_map.rs --

/// Runtime container for all versioned stores.
pub struct StoreMap {
    stores: HashMap<StoreId, ErasedStore>,
    virtual_ids: HashSet<StoreId>,
}

/// Type-erased store entry. The actual VersionedStore<S> is behind
/// a raw pointer. Access is type-checked at the Context level, not
/// at the StoreMap level.
struct ErasedStore {
    ptr: *mut u8,
    drop_fn: unsafe fn(*mut u8),
    version_fn: fn(*const u8) -> Version,
}
```

The `ErasedStore` avoids `dyn` (which is banned) and `std::any::TypeId` (also
banned). Type safety is ensured by the `Context` layer: the `Contains<S>` bound
proves that S was registered, and the `StoreId` lookup returns the correct
erased pointer. The `ptr` is cast to `*mut VersionedStore<S>` inside the
`Context` methods, where S is known.

### DAG builder

```rust
// -- dag_builder.rs --

pub fn build_edges<U: WorkUnit>(
    units: &[(UnitId, &dyn WorkUnitMeta)],
) -> Dag<UnitId, ()> {
    let mut dag = Dag::new();

    for &(id, _) in units {
        dag.add_node(id);
    }

    for &(id_a, meta_a) in units {
        for &(id_b, meta_b) in units {
            if id_a == id_b { continue; }

            // Write A -> Read B: data flow edge.
            for &write_id in meta_a.writes() {
                for &read_id in meta_b.reads() {
                    if write_id == read_id {
                        let _ = dag.add_edge(id_a, id_b, ());
                    }
                }
            }

            // Write A -> Write B: conflict edge (serialize writers).
            for &write_a in meta_a.writes() {
                for &write_b in meta_b.writes() {
                    if write_a == write_b && id_a < id_b {
                        let _ = dag.add_edge(id_a, id_b, ());
                    }
                }
            }
        }
    }

    dag
}
```

The `WorkUnitMeta` trait provides `reads()` and `writes()` as `&[StoreId]`
slices, derived from the associated `AccessSet` types. This avoids generic
instantiation in the DAG builder; the builder works with erased StoreId slices,
not with concrete types.

### Putting it all together

A complete execution cycle:

```
1. Registration
   - Consumer registers WorkUnit implementations.
   - For each WorkUnit, extract AccessSet -> StoreId slices.
   - Register all referenced stores in StoreMap (allocate SlabColumn for
     Single/Many, record StoreId for Virtual).

2. DAG construction
   - build_edges() compares write sets against read sets.
   - Produces Dag<UnitId, ()> with data flow and conflict edges.

3. Plan computation
   - Topological sort (Kahn's algorithm).
   - Group independent units into parallel steps.
   - Compute upward rank for work-stealing priority.
   - Assign lanes.
   - Output: ExecutionPlan<UnitId>.

4. Execution
   - For each step in the plan:
     - For each unit in the step (parallel):
       - Construct Context<Read, Write> from StoreMap references.
       - Call unit.execute(ctx).
       - If Complete: mark edges as satisfied.
       - If Yield: re-queue for later execution.
     - Barrier between steps.

5. Version check (early-cutoff)
   - After each unit completes, compare version stamps.
   - If a store's version did not change, downstream units that read
     only that store can be skipped.
   - Propagate skip decisions forward through the DAG.
```

## 10. Open Questions for the Topic Document

This research establishes the design space. Several decisions remain for the
topic document to resolve:

**Q1a. type_path_hash stability.** The `StoreId` is derived from a hash of the
type path. If a consumer renames a type or moves it to a different module, the
StoreId changes. This breaks incremental execution plans cached from a previous
run. Is StoreId stability across compilations a requirement? If so, should
StoreId be a user-provided string constant instead of an auto-derived hash?

**Q1b. AccessSet compile-time construction.** The `store_ids()` method on
`AccessSet` returns `&'static [StoreId]`. If StoreId construction requires
runtime hashing (because const fn does not support the hash algorithm), the
static slice needs `LazyLock` or a build script. Is this acceptable, or should
StoreId use a simpler hash that is const-evaluable?

**Q1c. Singleton write API.** The closure-based `resource_mut` API prevents
leaking `&mut T` references but is less ergonomic than Bevy's `ResMut<T>`. Bevy
tracks mutation via `DerefMut` detection: when the `ResMut` is dropped, it
checks if `DerefMut` was called. Should hilavitkutin use a similar RAII guard
instead of a closure?

**Q1d. Collection version stamp granularity.** Per-store stamps are conservative.
If profiling shows significant phantom-write waste on collections, per-morsel
stamps via `DirtyTracker` are the upgrade path. Should the initial implementation
include the `DirtyTracker` integration as an opt-in feature, or defer it
entirely?

**Q1e. ErasedStore and the dyn ban.** The `ErasedStore` struct uses raw pointers
and function pointers instead of `dyn Store`. This is correct per project rules
but requires unsafe code in the StoreMap. Should the erasure be encapsulated in a
single unsafe module with a safe public API, or should Context methods each
contain their own unsafe blocks?

These questions are implementation-level. The architectural answer to Q1 is
confirmed: unified at the scheduler level (one edge type, one conflict algorithm),
differentiated at the access level (type-level cardinality with compile-time
enforcement). The Store trait, the three concrete types (Resource, Column,
Virtual), and the Context-based access pattern are the design.
