## 1. Signal bus architecture

Signals are the sole communication mechanism in the loimu runtime. There
are no callbacks, no observer patterns, no event buses. Every inter-behavior
communication travels as a signal. A keystroke is a signal. A policy timer
firing is a signal. A window resize is a signal. There is no other mechanism.

Signals are not records. The record model (`define_record!`, `StorageRecord`,
views, temporal columns) applies to persistent data only. Signals are managed
by the snap backend of the storage engine. They are ephemeral: emitted during
a pass, consumed, then discarded.

### SignalBus

`SignalBus` is internal framework storage, not a resource. This distinction
is load-bearing for three reasons:

- **No version stamps.** Resources carry `version: u64` for the scheduler's
  early-release optimization. Signals do not need this; their presence or
  absence IS the information.
- **No DAG edges from bus access.** Signal edges come exclusively from
  handler declarations (`on Signal => ...`), not from resource access
  patterns. Behaviors never declare resource access for signals.
- **Not visible to plugins as a resource.** Plugins interact with signals
  through `SignalEmit` and `SignalRead` providers, not by projecting the
  bus directly.

### Storage layout

The bus owns two `ErasedArena<SignalId>` instances: one read arena and one
write arena. Each arena is backed by `SparseArray` (not HashMap). Keys are
`SignalId` (interned u64). Column lookup is O(1).

Within each arena, there is one `ErasedColumn` per signal type, keyed by
`SignalId`. Columns are pre-allocated at registration time, not on first
emit.

```
SignalBus
  read_arena:  ErasedArena<SignalId>
    +-- per-type ErasedColumn  (frozen, read-only during pass)
  write_arena: ErasedArena<SignalId>
    +-- per-type ErasedColumn  (mutable, accumulates during pass)
```

### Pre-allocation and the settling guarantee

After initialization and all plugin loads complete, the signal bus arenas
perform no new allocations during schedule passes. This is the settling
guarantee.

The guarantee holds because:

1. All signal types are registered at init time (native) or plugin load
   time (WASM). Each registration pre-allocates an `ErasedColumn` in both
   arenas for that signal type.
2. During passes, `emit()` writes into pre-allocated column slots via
   bump-pointer. Arena reset at pass boundary rewinds to zero without
   freeing.
3. Column capacity may grow (page commit) on volume spikes, but this is a
   page-level operation within the existing virtual reservation, not a heap
   allocation.

Plugin pre-allocation: when a plugin is loaded, the loader reads the plugin
manifest's signal declarations and pre-allocates columns in both arenas for
each declared signal type. This happens during the plugin load step, before
the plugin's behaviors enter the DAG. By the time the first pass runs with
the new plugin, all columns exist.

Scope: the guarantee applies after the system finishes initialization and
all pending plugin loads. Hot-reload with new signal types (schema-reset
reload) pre-allocates during the reload step. Init and load/reload steps
themselves are not covered.

### Arena allocation characteristics

| Operation | Cost | Notes |
|-----------|------|-------|
| Per-emit | O(1) bump-pointer | No per-signal heap alloc, no fragmentation |
| Arena clear | O(1) reset | Rewind to zero, no per-signal dealloc, no destructor walks |
| Queue mode | Coexist in arena slot | Multiple signals of same type in same column |
| Replace / ReplaceBy | Overwrite in-place | Within the arena slot |

Contrast with `Vec<Box<dyn Signal>>`: per-signal allocation, per-signal
drop, cache-hostile pointer chasing, type erasure overhead. The arena
avoids all of these.

## 2. Signal types

### The Signal trait

The `Signal` trait carries two associated constants: `NAME` (the interned
string identity, yielding a `SignalId`) and `BUFFER_MODE` (the buffering
strategy). All signals must be `Clone + Send + Sync + 'static`.

```rust
trait Signal: Clone + Send + Sync + 'static {
    const NAME: /* interned str type */;
    const BUFFER_MODE: BufferMode;
}
```

Identity is `SignalId` (the interned u64 IS the signal's identity), not
a string constant. The `define_id!` macro produces the `SignalId` type.

### define_signal! macro

The macro is the only sanctioned way to define a signal. Hand-writing
`impl Signal` is explicitly forbidden because it misses `Registrable`
(the signal won't appear in the registry) and `HasShape` (plugin projection
fails).

For each invocation, the macro generates:

1. **Struct** -- the signal type itself.
2. **`Signal` impl** -- sets `NAME` and `BUFFER_MODE`.
3. **`HasShape` impl** -- for shape-based plugin projection.
4. **`Registrable` impl** -- returns `SignalDescriptor` for `inventory`
   registration.

### Zero-size vs data-carrying signals

Unit signals (zero-size) are declared without fields:

```rust
define_signal!(QuitRequested);  // unit signal, default Queue
```

Data-carrying signals carry named fields:

```rust
define_signal!(KeyPressed { key: Str } buffering: Queue);
define_signal!(MouseMoved { x: Coord, y: Coord } buffering: Replace);
define_signal!(NodeChanged { node: NodeId } buffering: ReplaceBy(node));
```

If no `buffering:` clause is given, the default is `Queue`.

### Wrong vs right

```rust
// WRONG: manual impl loses Registrable, HasShape, and breaks registry consistency.
impl Signal for MySignal {
    const BUFFER_MODE: BufferMode = BufferMode::Queue;
}

// RIGHT: one line, everything generated
define_signal!(MySignal);
```

### Supporting types

| Type | Purpose |
|------|---------|
| `Signal` trait | Typed message with buffer mode |
| `BufferMode` | Replace, Queue, ReplaceBy(field) |
| `SignalBus` | Central signal storage (arena-allocated, keyed by SignalId) |
| `SignalSet` | Named group of related signal types (backed by `DenseColumn<SignalId>`) |
| `SignalSetStore` | Signal set data |
| `SignalDescriptor` | Registry entry (generated by `define_signal!`) |
| `ContainsSignal<S>` | Compile-time tuple membership check |

### Infra macros

| Macro | Where | Why |
|-------|-------|-----|
| `define_signal!` | Signal type declarations | Generates Signal + HasShape + Registrable |
| `define_registry!` | `SignalRegistry` | ZST registry with `discover()` and HasName for descriptor |
| `define_provider!` | `SignalEmit`, `SignalRead`, `SignalCount`, `SignalQuery` | Providers for behavior contexts |
| `define_id!` | `SignalId` | Interned string ID type |

### Conventions

- Each crate defines only its own domain signals.
- The `loimu-signal` crate itself defines only `SignalOverflow` (meta-signal
  about the signal system).

## 3. BufferMode semantics

`BufferMode` controls what happens when the same signal type is emitted
multiple times within a single arena (within a single pass).

```rust
enum BufferMode {
    Replace,
    Queue,
    ReplaceBy(/* field ident */),
}
```

### Replace

Latest value wins. Last emission per DAG execution order overwrites all
earlier emissions of the same type in that arena. Each arena holds at most
one value for a Replace signal.

Use case: mouse position, scroll offset, cursor position; any continuous
value where only the latest matters.

```rust
define_signal!(MouseMoved { x: Coord, y: Coord } buffering: Replace);
define_signal!(CursorMoved { x: Coord, y: Coord } buffering: Replace);
```

Observer behavior: sees the write-arena value (current pass) if available,
otherwise falls back to the read-arena value (previous pass).

### Queue

All signals preserved in emission order (FIFO). Readers iterate all. Multiple
emitters contributing the same signal type all have their values preserved.
Default mode when no `buffering:` clause is specified.

Use case: key presses, discrete events; any event where every occurrence
matters.

```rust
define_signal!(KeyPressed { key: Str } buffering: Queue);
define_signal!(QuitRequested);  // unit signal, default Queue
```

Observer behavior: iterates all signals from both arenas, but the arenas
remain logically separate. No merged super-queue, no double-counting.

### ReplaceBy(field)

Latest emission per unique key field value wins. Keyed deduplication: one
slot per distinct value of the named field, last write per key wins.

Use case: per-node deduplication where only the latest change to each
distinct node matters.

```rust
define_signal!(NodeChanged { node: NodeId } buffering: ReplaceBy(node));
```

Observer behavior: sees one value per unique key from the read arena (prev
pass) and one value per unique key from the write arena (current pass).

### BufferMode per-arena truth table

BufferMode applies independently to each arena. This prevents double-counting
across pass boundaries.

| Mode | Read arena (prev pass) | Write arena (current pass) | Observer sees |
|------|----------------------|--------------------------|---------------|
| `Queue` | All previous-pass signals | All current-pass signals | Both sets, logically separate |
| `Replace` | One value (last prev-pass emit) | One value (last current-pass emit) | Write-arena value if present, else read-arena |
| `ReplaceBy(f)` | One value per key (prev pass) | One value per key (current pass) | Both key sets, per-arena |

### Conflict resolution

If two behaviors both emit a `Replace` signal in the same pass, the one
running later in the DAG determines the final value within the write arena.
For `Queue`, both are preserved and the reader sees them in DAG-execution
order.

## 4. Double buffering mechanics

### The problem

Without double-buffering, the scheduler faces a dilemma when signals arrive
mid-pass (platform events, async callbacks, or behaviors emitting signals
whose observers already ran):

- **Stall**: wait for all signals before running any behavior. Kills
  parallelism and adds latency.
- **Drop**: discard signals that arrive after their observers ran. Loses
  data silently.

Signal double-buffering avoids both: the pass runs at full speed, and late
signals are deferred to the next pass with zero cost.

### Read/write arenas

```
read_arena:   previous pass's emissions. Frozen, read-only during the pass.
write_arena:  current pass's emissions. Writable, accumulates during the pass.
```

### Pointer swap at pass boundary

The scheduler performs the swap at the very start of each pass, before
emitting `ScheduleStart`. This ensures `ScheduleStart` goes into a clean
write arena.

```
old write_arena -> new read_arena  (pointer swap, zero cost)
old read_arena  -> reset           (O(1) arena rewind, zero per-signal cost)
```

No memcpy. No allocation. The arena being written becomes the arena being
read. The arena being read is cleared and reused as the new write target.

### O(1) reset

`clear()` drops values but keeps the `SparseArray` structure and column
allocations for reuse. The structure is preserved so that subsequent passes
do not need to rebuild column indices. The column allocations remain
available. This is how the settling guarantee is maintained across passes.

### Full pass lifecycle

```
pass N-1:  behaviors emit into write_arena
           |
pass boundary: write_arena -> read_arena (pointer swap)
               old read_arena resets
           |
pass N:    scheduler emits ScheduleStart into write_arena
           behaviors read from read_arena + write_arena (DAG-ordered)
           behaviors emit into write_arena
           scheduler emits ScheduleEnd
           |
pass boundary: write_arena -> read_arena (pointer swap)
               old read_arena resets
```

### Emission rules

All signal emissions go into the write arena:

- Behaviors calling `ctx.emit(signal)`
- The scheduler emitting `ScheduleStart` / `ScheduleEnd`
- External sources (platform input drains, async callbacks)

`BufferMode` applies within the write arena. Multiple `Replace` emissions
of the same type within one pass: last writer wins. Multiple `Queue`
emissions: all preserved in emission order.

### Reading rules

Behaviors read from both arenas:

1. **Read arena** (previous pass): always visible. Contains signals emitted
   during the previous pass, including any that arrived too late for their
   observers back then.
2. **Write arena** (current pass): visible only for signals from behaviors
   that already ran this pass. The scheduler knows this from the DAG: if A
   has `emit S;` in a handler and B has `on S => ...`, the DAG edge A->B
   guarantees A ran before B, so B sees A's write-arena signals.

### Relationship to other buffering mechanisms

Signal double-buffering is separate from record double-buffering (frozen/live
column swap) and temporal triple-buffering (old/current/projected). Same
`SwapBuffer` primitive from loimu-storage, different modes:

| | Signals | Markers |
|---|---------|---------|
| **SwapBuffer mode** | `Exchange` (pointer swap + reset) | `Snapshot` (memcpy live -> frozen) |
| **Persistence** | Ephemeral (cleared each pass) | Persistent (live across passes) |
| **Swap cost** | O(1) pointer swap | O(n) memcpy |
| **Read source** | Both arenas (DAG-ordered) | Snapshot only (opportunistic live upgrade) |
| **DAG edges** | From handler declarations (`on S`, `emit S;`) | From `read nodes`/`write nodes` access |
| **Default latency** | Zero (same-pass for declared paths) | One pass (snapshot is always previous) |

## 5. Signal flow in the DAG

### Emission = write edge, observation = read edge

Emission (`emit S;` detected by macro) registers the behavior as an emitter
of signal S. Observation (`on S => ...`) registers the behavior as an
observer of S. The scheduler constructs a DAG edge from these declarations:

```
A.emits_ids contains S  AND  B.observes_ids contains S  =>  edge A -> B
```

This check is `SignalId` u64 equality; one integer comparison per pair.

### Same-pass delivery guarantee

For declared signal paths (handler `on S => ...` combined with `emit S;`
creating a DAG edge), same-pass delivery is guaranteed. The emitter runs
before the observer in DAG topological order. The observer sees the signal
in the write arena immediately. No latency.

This is the common case for behavior-to-behavior signal flow.

### DAG and parallelism

Scheduling units with disjoint resource access (and disjoint signal
read/write sets) can run in parallel on separate lanes. The DAG edges
constrain ordering only where signal dependencies exist.

### Steady-state DAG

The scheduler rebuilds the DAG when scheduling units are added or removed
(hot reload, plugin load). During steady-state, the DAG is fixed.
`RecalculateSchedulePlan` is emitted on plugin load/unload, triggering
a full plan recomputation between passes.

## 6. Mid-pass vs cross-pass delivery

### Same-pass delivery (declared paths)

When behavior A emits S and behavior B has `on S => ...`, the macro
detects this and creates a DAG edge A->B. The scheduler guarantees A runs
before B within the same pass. B reads S from the write arena. Latency:
zero.

All declared `emit S;` + `on S => ...` pairs create DAG edges. This covers
the overwhelming majority of behavior-to-behavior signal flow.

### Cross-pass delivery (late arrivals)

Signals can arrive too late for a given pass in two situations:

1. **External signals mid-pass**: platform input drains, async callbacks
   arriving after the observer already ran this pass.
2. **Undeclared signal paths**: a behavior emits a signal whose observer
   has no DAG edge to it. This is rare; it means the `emit` is not
   visible to the macro, or the signal is unobserved in any handler
   declaration.

These signals land in the write arena and become visible next pass via
the read arena. Latency: one pass.

No signals are lost. All late arrivals are available next pass.

### Cascading latency

Worst case: behavior A emits S, but observer B already ran this pass
(no DAG edge for S). B picks up S next pass from the read arena. If B
then emits T observed by C, and C already ran, that is a two-pass chain.
Each undeclared hop adds one pass of latency.

For declared signal paths, this never happens. Same-pass delivery is
guaranteed by the DAG. Cascading latency is exclusively a consequence of
undeclared signal paths or external event timing.

### External signal handling

External signals arriving mid-pass (platform events, async callbacks)
go into the write arena. If their observers already ran this pass, those
signals are picked up next pass. The scheduler's signal filtering checks
both arenas before dispatching each behavior, so the next pass will
correctly detect the presence of the signal.

## 7. Signal filtering

### Mechanism

Before executing a scheduling unit, the scheduler checks whether any of
its `observes_ids` are present in either signal arena. If none are present,
the unit is skipped entirely for this pass.

This check uses `SignalId` (interned u64) equality, not string matching.
Both arenas are checked. The cost is one integer comparison per observed
signal type per behavior.

### Effective rate control

Because different policies wake the scheduler at different rates, and each
policy emits its own signal, behaviors observing different policy signals
effectively run at different rates. They are simply skipped on passes where
their policy signal is absent.

This is how loimu achieves variable-rate behavior execution without explicit
tick counters or timers in the scheduler itself. The signal filtering IS the
rate control mechanism.

## 8. SignalSets

`SignalSet` groups related signals for compound observation. A behavior
observing a `SignalSet` effectively observes all member signals.

```rust
define_signal!(set InputSignals = [KeyPressed, KeyReleased, MouseMoved]);
```

SignalSets do not store data. They are compile-time metadata that expands
into multiple handler registrations. At compile time, observing
`InputSignals` expands into observing `KeyPressed`, `KeyReleased`, and
`MouseMoved` individually.

`SignalSet` is backed by `DenseColumn<SignalId>` for the set membership
data. `ContainsSignal<S>` provides a compile-time tuple membership check,
allowing the type system to verify that a signal is part of a set.

## 9. Comparison to virtual columns

The hilavitkutin topic (Q3) asks about virtual columns: zero-width marker
types that create DAG edges without carrying data. This is directly
comparable to loimu's signal system, but with significant differences in
scope.

### What loimu signals do

- Signals create DAG edges between behaviors: `emit S;` -> `on S => ...`
  produces edge A->B.
- Signals can be zero-size (unit signals like `QuitRequested`) or carry data
  (`KeyPressed { key: Str }`).
- The signal bus is a full runtime subsystem with arena allocation,
  double-buffering, three buffer modes, signal filtering, and compound
  observation via SignalSets.
- Signals are the ONLY communication mechanism. All inter-behavior data
  flow goes through them.

### What saalis needs (from the topic)

- Events like `EntityDiscovered` that create control-flow dependencies
  without carrying persistent column data.
- DAG edges so that when a connector discovers an entity, downstream
  processing units are triggered.
- Saalis's events are closer to zero-width DAG markers than to loimu's
  full signal bus. They need the edge-creation behavior, not the arena
  storage, double-buffering, or buffer mode semantics.

### What polka needs

- Nothing. Polka's pipeline is a static DAG of N->IR, IR->IR, IR->M
  transforms. Dependencies are structural (column read/write), not event-driven.
  There are no signals and no runtime events to respond to.

### The zero-width column approach

saalis's request from the topic: `Write<EntityDiscovered>` as a zero-sized
type creating a DAG edge with no data stored. This is conceptually a
degenerate case of loimu's signal system:

- It has the edge-creation behavior (emit = write edge, observe = read edge).
- It does not need buffer modes (there is no data to replace or queue).
- It does not need double-buffering (the event's existence is a one-shot
  trigger, not a value that persists across passes).
- It does not need arena allocation (zero-size types have no storage).

The zero-width column IS the common denominator between loimu's signals and
saalis's events: it is the DAG edge without the bus.

## 10. Relevance to hilavitkutin

### Three projects, three needs

| Project | Signal/event need | Scope |
|---------|-------------------|-------|
| polka-dots | None | Static DAG, no runtime events |
| saalis | Event edges | `Write<EntityDiscovered>` triggers downstream units |
| loimu | Full signal bus | Arena-allocated, double-buffered, three buffer modes, filtering, SignalSets |

### What is the common denominator?

The common denominator is the DAG edge mechanism, not the signal bus.

All three projects share the concept that declaring `Write<T>` and `Read<T>`
on work units creates a dependency edge in the DAG. For polka and saalis
with data columns, T is a real data type with storage. For saalis's events
and loimu's zero-size signals, T is a marker type with no storage but the
same edge semantics.

hilavitkutin should provide:

1. **Typed DAG edge construction** from `Read<T>` / `Write<T>` declarations.
   This is the universal primitive.
2. **Zero-width marker type support** so that `Write<EntityDiscovered>`
   creates a DAG edge without allocating storage. This covers saalis's
   event use case and the trivial-signal subset of loimu's needs.
3. **No signal bus.** The full signal bus (arena allocation, double-buffering,
   three buffer modes, signal filtering, SignalSets, providers) is
   loimu-specific. It should remain in loimu-signal as a layer on top of
   hilavitkutin's DAG primitives.

### What loimu builds on top

loimu-signal would continue to own:

- `SignalBus` with `ErasedArena<SignalId>` pairs
- Double-buffering with `SwapBuffer<_, Exchange>` mode
- `BufferMode` (Replace, Queue, ReplaceBy)
- `define_signal!` macro generating Signal + HasShape + Registrable
- `SignalSet` compound observation
- Signal filtering (skip units with no observed signals in either arena)
- `SignalEmit`, `SignalRead`, `SignalCount`, `SignalQuery` providers
- The settling guarantee (pre-allocation at init/load time)

All of these sit on top of hilavitkutin's DAG scheduling. The DAG edge
from `emit S;` + `on S => ...` maps directly onto hilavitkutin's
`Write<S>` / `Read<S>` edge construction. loimu-signal translates handler
declarations into the access declarations that hilavitkutin understands.

### The boundary

hilavitkutin owns the DAG and the edge semantics. loimu-signal owns the
bus, the storage, and the delivery semantics. The translation layer is
thin: loimu's handler declarations (`on S => ...`, `emit S;`) produce
`Read<S>` / `Write<S>` access sets that hilavitkutin uses for topological
ordering.

This keeps hilavitkutin minimal (DAG + dispatch + storage abstraction)
while letting loimu build its full signal architecture on top without
compromising any of the settling guarantee, double-buffering, or buffer
mode semantics.

Key dependencies: `loimu-storage` (arena and `SwapBuffer` primitives),
`loimu-id` (`SignalId` interning), `loimu-registry` (`inventory`-based

hilavitkutin world, the storage and id primitives might come from the
shared crate, but the signal-specific layers (bus, providers, registry)
stay in loimu-signal.
