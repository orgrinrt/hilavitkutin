## 1. The Fundamental Tension

CPU-bound pipeline processing and I/O-bound task execution are structurally
adversarial. They compete for the same scarce resource (worker threads) and
their performance characteristics are inverted: CPU work wants uninterrupted
execution with cache-hot data; I/O work spends most of its wall-clock time
waiting for external responses and benefits from high concurrency.

A morsel-driven pipeline achieves throughput by keeping each worker thread
saturated with arithmetic. The worker pulls a morsel, runs the fused chain
of operators on it, writes output, pulls the next morsel. At steady state,
the critical data structures (input columns, hash tables, aggregation
buffers) sit in L1/L2 cache and the CPU pipeline rarely stalls. Interrupting
this flow is expensive. A context switch evicts cache lines. A system call
flushes the store buffer. A thread park/unpark round-trip costs 2-5 us on
Linux even with futex fast-path, and the resumed thread returns to cold
caches.

I/O-bound work has the opposite profile. An HTTP request to an external
API takes 50-500 ms of wall-clock time, during which the issuing thread
does zero useful computation. A disk read on NVMe takes 10-100 us. A
database query against a remote Postgres instance takes 1-50 ms. If a
worker thread blocks on any of these operations, it cannot process morsels.
With N worker threads and M blocked on I/O, effective parallelism drops to
N-M, potentially to zero if all workers are waiting on external responses.

The naive solution -- more threads -- causes its own problems. Oversubscription
leads to contention on shared data structures (morsel queues, hash table
build buffers, atomic progress counters). Context switching overhead grows
linearly with thread count. Cache pollution from thread interleaving destroys
the locality that makes morsel processing fast.

Systems that need both CPU-bound pipeline processing and I/O-bound external
communication must solve this tension explicitly. The literature and industry
practice show three broad strategies: separation (dedicated I/O threads that
never touch the compute path), integration (I/O expressed as pipeline stages
that yield cooperatively), and delegation (I/O submitted asynchronously to
kernel or runtime, completion notification wakes the relevant pipeline stage).
Each has trade-offs in complexity, latency, and throughput. The rest of this
document surveys how specific systems choose among them.

## 2. Database Approaches

Database systems are the most direct precedent. Every database must handle
network I/O (client connections, replication, distributed queries) alongside
compute-heavy query processing (scans, joins, aggregations). The strategies
they use span the full separation-integration-delegation spectrum.

### 2.1 PostgreSQL: process-per-connection, blocking I/O

PostgreSQL assigns one OS process per client connection. Each process runs
the full query lifecycle: parse, plan, execute, send results. I/O (reading
pages from disk, sending results over the network) happens inline with
computation. When a process waits for a disk read, it blocks. The OS
scheduler preempts it and runs another process.

This model is simple and correct. Each process has its own address space,
so there is no shared-state contention. The OS handles all scheduling
decisions. The cost is scaling: each process consumes 5-10 MB of memory
for stack and connection state, and context switches between hundreds of
processes are expensive. PostgreSQL typically runs well up to a few hundred
connections; beyond that, connection poolers (PgBouncer, Odyssey) multiplex
many client connections onto fewer backend processes.

For pipeline-style execution, this model is instructive mainly as a
counter-example. It provides no separation between I/O and compute. A
process blocked on network send cannot do useful query work. There is no
morsel-level scheduling -- each process owns its entire query and runs it
sequentially. The model works because PostgreSQL's target workload (OLTP:
many short queries) does not require sustained CPU throughput on any single
query.

### 2.2 CockroachDB: goroutine-per-request with cooperative scheduling

CockroachDB is written in Go and uses goroutines for concurrency. Each SQL
statement runs in its own goroutine. Distributed queries spawn additional
goroutines for remote fragment execution. I/O (network RPCs to other nodes,
disk reads via Pebble/RocksDB) is handled by Go's runtime: when a goroutine
issues an I/O call, the runtime parks it and schedules another goroutine on
the same OS thread. The I/O call is dispatched to a background poller (epoll
on Linux, kqueue on macOS) and the goroutine is woken when the response
arrives.

The Go runtime acts as a user-space scheduler with built-in I/O integration.
Goroutines are cooperative in practice (the compiler inserts yield points at
function calls and channel operations), and the runtime preempts long-running
goroutines via signal injection (since Go 1.14). This means compute-heavy
goroutines do not block the entire scheduler, and I/O-bound goroutines do
not consume OS threads while waiting.

CockroachDB's vectorized execution engine (introduced in v19.2, based on the
MonetDB/X100 model) adds a wrinkle. Vectorized operators process batches of
1024 tuples at a time, similar to morsels. The engine chains operators into
pipelines and calls Next() on the root operator, which pulls batches through
the chain. When a pipeline stage needs remote data (e.g., a distributed
hash join reading from a remote node), the Next() call issues an RPC and
the goroutine parks until the response arrives. Other goroutines (other
queries, other pipeline stages) run while this one waits.

This is the integration model: I/O is expressed as a pipeline stage that
blocks (from the caller's perspective) but yields (from the runtime's
perspective). It works because Go's runtime provides the yield/resume
mechanism transparently. The programmer writes blocking code; the runtime
makes it cooperative.

The limitation is that the Go runtime's scheduler is general-purpose. It
does not understand morsel locality, cache affinity, or DAG dependencies.
A goroutine woken from I/O might be scheduled on a different OS thread
(and therefore a different core) than the one that started it, causing
cache misses on the batch data. CockroachDB mitigates this with thread-local
batch pools but cannot fully control scheduling the way a purpose-built
morsel dispatcher can.

### 2.3 DuckDB: single-process, explicit I/O stages as pipeline breakers

DuckDB takes the separation approach. The execution model is morsel-driven
(directly inspired by HyPer/Leis et al.) with explicit pipeline boundaries.
I/O operations are pipeline breakers: they sit between pipeline segments,
not inside them.

A scan from a Parquet file, for example, works like this: the scan operator
reads a row group from disk (I/O), decompresses it into an in-memory
column chunk (compute), and exposes it as a source for the downstream
pipeline. The I/O and decompression happen before the morsel pipeline starts.
Once the data is in memory, the pipeline runs without any I/O.

For remote sources (e.g., reading from S3 via the httpfs extension), DuckDB
uses a prefetch model. A background thread issues HTTP range requests for
upcoming row groups while the main pipeline processes current row groups.
The pipeline never blocks on network I/O; instead, it picks up the next
row group when its prefetch is complete. If the prefetch is not ready, the
pipeline stalls at the pipeline breaker boundary, not inside the hot loop.

This design keeps the morsel processing path free of I/O entirely. Worker
threads inside a pipeline segment never issue system calls, never block,
never yield. The pipeline scheduler (DuckDB's parallel hash join coordinator,
for example) handles pipeline breakers as synchronization points where all
worker threads complete the current segment before the next segment starts.

The trade-off is latency at the pipeline boundary. If a prefetch is slow,
all worker threads idle at the breaker waiting for data. There is no
interleaving of I/O-wait and useful compute within a single pipeline. For
DuckDB's target workload (analytical queries on local or cached data), this
is acceptable because I/O latency is dominated by disk read time (10-100 us
for NVMe) and pipeline segments are long enough to hide prefetch latency.
For a workload where I/O is the bottleneck (e.g., querying a slow remote
API), this model would leave CPUs idle for extended periods.

### 2.4 DataFusion: Tokio integration with Stream trait

Apache DataFusion, the Rust-based query engine underlying Apache Arrow,
takes the delegation approach. DataFusion operators implement the `Stream`
trait (from the futures crate), which returns `Poll::Ready(Some(batch))` or
`Poll::Pending`. The Tokio async runtime drives these streams: when an
operator returns Pending, Tokio parks the task and runs other tasks. When
the I/O source signals readiness (via epoll/kqueue/io_uring), Tokio wakes
the task and resumes the operator.

This is the most direct precedent for hilavitkutin's `Progress::Yield(Pending)`
model. The structural parallel is close:

| DataFusion | hilavitkutin |
|------------|--------------|
| `Stream::poll_next()` | `WorkUnit::execute()` |
| `Poll::Ready(batch)` | `Progress::Complete` |
| `Poll::Pending` | `Progress::Yield(Pending)` |
| Tokio waker | consumer-provided waker |
| `RecordBatch` output | column writes via `Write<T>` |

DataFusion's I/O integration works because Tokio provides the full
event-loop machinery: I/O resource registration, readiness notification,
waker management, task scheduling. A DataFusion operator that reads from
S3 uses the `object_store` crate, which uses `reqwest`, which uses `hyper`,
which uses Tokio's TCP driver. When the HTTP response arrives, the chain
of wakers fires and the operator's `poll_next()` is called again.

The important lesson from DataFusion is that the query engine itself does
not implement an I/O runtime. It exposes a cooperative interface (Stream /
Poll) and delegates I/O mechanics to whatever async runtime the consumer
is using. DataFusion works with Tokio, but the Stream trait is runtime-
agnostic in principle.

The limitation is that DataFusion's execution is fully async. Every operator
is an async task, even purely CPU-bound operators like filter, project, and
aggregate. This means CPU-bound operators pay the overhead of the async
state machine (generated by the compiler for each .await point) and the
Tokio scheduler's task switching logic. For small batches, this overhead
is measurable. DataFusion mitigates it with large batch sizes (typically
8192 rows) and by using `spawn_blocking` for truly CPU-heavy work (e.g.,
Parquet decompression), but the fundamental tax of running CPU work through
an async scheduler remains.

## 3. Game Engine Approaches

Game engines face the I/O integration problem under harder constraints than
databases. A database can tolerate milliseconds of latency on a query plan
decision; a game engine has a 16.6 ms frame budget (60 fps) or 8.3 ms
(120 fps), and any I/O stall in the main thread is visible as a dropped
frame. Game engines have developed mature strategies for keeping I/O off
the critical path.

### 3.1 Asset loading during gameplay

Streaming open-world games (Unreal Engine, Unity DOTS, custom engines like
Naughty Dog's) load assets (textures, meshes, audio, level geometry) from
disk while gameplay runs. The standard pattern is a dedicated I/O thread
pool (typically 1-2 threads, since disk I/O is bandwidth-limited not
CPU-limited) that services a priority queue of load requests.

The game thread submits a load request (asset path, priority, callback).
The I/O thread reads the file from disk, decompresses it, and signals
completion via an atomic flag, a channel, or a fence. The game thread
polls the flag during its per-frame update. When the asset is ready, the
game thread integrates it (uploads to GPU, adds to scene graph). The game
thread never blocks on I/O; it renders a placeholder or LOD (level of
detail) proxy until the full asset is ready.

The completion notification mechanism varies:

- **Atomic flag**: simplest. The I/O thread sets an `AtomicBool` to true.
  The game thread checks it once per frame. Zero synchronization overhead
  (a single relaxed load), but the game thread must poll.
- **Channel**: the I/O thread sends a completion message through an mpsc
  channel. The game thread drains the channel once per frame. Slightly
  higher overhead (channel lock/unlock) but supports multiple completions
  per frame without polling each request individually.
- **Callback / continuation**: the I/O thread invokes a callback on
  completion, which enqueues work into the game thread's task queue.
  Most flexible but requires careful thread safety.

### 3.2 Network I/O alongside game logic

Multiplayer games process network packets (player inputs, state updates,
chat messages) alongside game simulation. The standard architecture is a
dedicated network thread that owns the socket(s) and communicates with the
game thread via lock-free queues.

The network thread runs an event loop (epoll, kqueue, IOCP, or a library
like ENet). It receives packets, deserializes them, and pushes them into
a lock-free SPSC (single-producer single-consumer) or MPSC queue. The game
thread drains the queue once per tick and processes the messages. Outgoing
messages follow the reverse path: game thread pushes into a queue, network
thread drains and sends.

This is strict separation. The game thread never touches a socket. The
network thread never touches game state. The queue is the only
synchronization point, and it is designed to be wait-free in the common
case (ring buffer with atomic head/tail pointers).

Valve's Source engine, id Tech engines, and most custom AAA engines use
this pattern. It extends naturally to multiple network protocols: one
thread for game state (UDP), another for voice (UDP with jitter buffer),
another for HTTP (matchmaking API, telemetry uploads). Each I/O concern
gets its own thread; game logic runs independently of all of them.

### 3.3 Audio streaming

Real-time audio is the most latency-sensitive I/O in games. The audio
callback (called by the OS audio subsystem at the hardware sample rate,
typically every 5-10 ms) must fill a buffer with audio samples. If it
blocks or runs overtime, the result is an audible glitch.

The solution is a ring buffer. A background thread (or the game thread
during its audio phase) reads compressed audio from disk, decodes it,
and writes PCM samples into a ring buffer. The audio callback reads from
the ring buffer. If the ring buffer has data, the callback never blocks.
If the ring buffer is empty (the producer fell behind), the callback
writes silence -- a graceful degradation rather than a stall.

This is delegation with a fixed-size intermediary. The producer (I/O +
decode) and consumer (audio callback) never directly synchronize. The ring
buffer absorbs timing jitter between them. The same pattern appears in
video decoding (decode frames ahead into a ring buffer, the render thread
reads the next frame) and log systems (application writes to a ring buffer,
a background thread flushes to disk).

### 3.4 Frame budget accommodation

Some engines give I/O work a time slice within the frame budget. Unreal
Engine's `FQueuedThreadPool` and Unity's job system both support this: the
main thread has a "service" phase where it processes completed I/O requests,
integrates loaded assets, and dispatches new I/O requests. This phase has a
fixed time budget (e.g., 2 ms out of a 16.6 ms frame). If the budget is
exhausted, remaining work is deferred to the next frame.

This is the game engine equivalent of cooperative yielding. The I/O
integration code runs a loop: process one completion, check the timer,
process another completion, check the timer, and so on until the budget
is spent or the queue is drained. The timer check corresponds directly
to `Progress::Yield(BudgetExhausted)` in hilavitkutin's model.

Naughty Dog's fiber-based engine (described in GDC talks by Christian
Gyrling, 2015) takes this further. All work (gameplay, rendering, I/O
integration) runs as fibers on a shared worker pool. When a fiber needs
I/O, it submits the request and yields the fiber. The scheduler runs
another fiber (potentially from a completely different subsystem) on the
same worker thread. When the I/O completes, the original fiber is
rescheduled. This is structurally identical to async/await but implemented
at the engine level rather than the language level.

The fiber model achieves maximum worker utilization: a worker thread never
blocks on I/O because fibers yield before blocking. But it requires
explicit fiber management (fiber pools, stack allocation, save/restore of
fiber state), which adds complexity that async runtimes like Tokio provide
for free.

## 4. io_uring and Modern Kernel I/O

Traditional Unix I/O requires a system call per operation: read(), write(),
sendmsg(), recvmsg(). Each system call crosses the user/kernel boundary,
which costs 100-500 ns on modern hardware (Spectre mitigations increased
syscall overhead by 2-5x on Intel). For a workload that issues thousands
of I/O operations per second, syscall overhead is measurable.

io_uring (Linux 5.1+, Jens Axboe, 2019) eliminates most of this overhead
by batching I/O submissions and completions through shared memory ring
buffers.

### 4.1 Submission queue / completion queue model

The application and kernel share two ring buffers: the submission queue (SQ)
and the completion queue (CQ). The application writes I/O requests (sqe:
submission queue entry) into the SQ. The kernel reads them, performs the I/O,
and writes results (cqe: completion queue entry) into the CQ. The application
reads completions from the CQ.

In the ideal case, no system call is needed for either submission or
completion. The application writes sqes, the kernel polls the SQ (using
`IORING_SETUP_SQPOLL` flag, which dedicates a kernel thread to poll the
SQ), and the application polls the CQ for completions. This is "system-call-
free I/O" -- the only system calls are the initial io_uring_setup() and
memory mapping.

Without SQPOLL, the application calls `io_uring_enter()` to notify the
kernel that new sqes are available. This is one system call per batch of
submissions, amortized across potentially hundreds of I/O operations.

### 4.2 Integration with user-space schedulers

io_uring integrates with event loops via the completion queue's file
descriptor. The application can register the CQ fd with epoll, or poll
it directly in a select/poll loop. This means an existing event loop
(libuv, mio, Tokio) can monitor io_uring completions alongside socket
readiness, timer expirations, and other event sources.

For a pipeline scheduler like hilavitkutin, the integration point would be:

1. Work unit submits I/O via io_uring (writes sqe to the SQ).
2. Work unit returns `Progress::Yield(Pending)` with a waker tied to the CQ fd.
3. The consumer's event loop (Tokio, mio, or a custom loop) monitors the CQ fd.
4. When the cqe appears, the event loop invokes the waker.
5. The scheduler resumes the work unit.

This avoids the overhead of spawning an OS thread for I/O, the overhead
of Tokio's task scheduling, and the overhead of system calls for each
individual I/O operation. The work unit submits I/O directly to the kernel
and yields; the scheduler never knows or cares that I/O happened.

### 4.3 Tokio's io_uring support

The tokio-uring crate (maintained by the Tokio project) provides an async
interface to io_uring. It runs on a single-threaded Tokio runtime (io_uring
SQEs are tied to the thread that created the ring) and exposes async methods
for read, write, accept, connect, and other operations.

tokio-uring is relevant because saalis connectors will likely use reqwest
(which uses hyper, which uses Tokio). On Linux, a future version of Tokio
could use io_uring internally for all socket I/O, making reqwest transparently
benefit from reduced syscall overhead. The saalis connector code would not
change; only the underlying I/O driver would switch.

### 4.4 Platform considerations

io_uring is Linux-only (kernel 5.1+). macOS has no equivalent; kqueue is
the best available (event-driven but still one-syscall-per-operation for
non-socket I/O). Windows has I/O Completion Ports (IOCP), which provide
similar batched-completion semantics but with a different API surface.

For hilavitkutin, which targets macOS (polka-dots primary platform) and
Linux (saalis servers, CI), the design cannot depend on io_uring. The
yield/resume contract must be I/O-mechanism-agnostic. The consumer chooses
the I/O backend (epoll, kqueue, io_uring, IOCP) and provides the waker
integration. hilavitkutin never submits I/O and never polls for I/O
readiness.

## 5. The Parking/Waking Model

The core abstraction for integrating I/O with cooperative scheduling is
parking and waking. A task that cannot make progress parks itself (removes
itself from the runnable set) and provides a waker (a callback or handle
that the scheduler can use to re-add the task to the runnable set when
the blocking condition is resolved).

### 5.1 Tokio park/unpark

Tokio's task model is the most mature implementation of this pattern in
Rust. When a future returns `Poll::Pending`, the Tokio runtime parks the
task. The future, before returning Pending, must have registered interest
with an I/O driver (via `cx.waker().clone()`). When the I/O driver detects
readiness (the socket has data, the timer fired), it calls `waker.wake()`,
which enqueues the task in the runtime's run queue. The runtime eventually
picks up the task and calls `poll()` again.

Key properties:

- **Zero busy-waiting**: a parked task consumes no CPU. The waker is a
  pointer-sized handle; thousands of parked tasks cost kilobytes, not
  threads.
- **Waker is runtime-agnostic**: the `Waker` type (from `std::task`) is
  a trait object with `wake()` and `clone()` methods. Any runtime can
  provide a waker. Futures do not know which runtime drives them.
- **Waker is edge-triggered**: wake() means "this task might be able to
  make progress." It does not mean "the I/O is definitely complete." The
  task must re-check the condition and may return Pending again.

### 5.2 Progress::Yield(Pending) as a parking mechanism

hilavitkutin's `Progress::Yield(Pending)` is structurally equivalent to
`Poll::Pending`. When a work unit returns `Yield(Pending)`, the scheduler
removes it from the runnable set. But unlike Tokio, hilavitkutin does not
include an I/O driver. There is no built-in mechanism to detect I/O
readiness and wake the parked work unit.

The question is: who wakes the work unit?

In Tokio, the waker is provided by the runtime via the `Context` parameter
to `poll()`. The future clones the waker and passes it to the I/O driver.
When I/O is ready, the driver calls `waker.wake()`.

In hilavitkutin, the work unit must receive a waker from somewhere and
pass it to whatever I/O mechanism it uses. Options:

1. **Scheduler provides a waker**: the scheduler creates a waker per work
   unit that, when invoked, moves the unit back to the runnable queue. The
   work unit passes this waker to its I/O code (e.g., a Tokio runtime
   running in the consumer's process).

2. **Work unit provides a readiness check**: instead of a waker callback,
   the work unit returns a function `fn() -> bool` that the scheduler
   polls periodically. The function checks if the I/O is complete (e.g.,
   tests an atomic flag set by a background thread). This is simpler but
   wastes CPU on polling.

3. **Consumer mediates**: the consumer runs both hilavitkutin's scheduler
   and an I/O runtime (Tokio). The consumer wraps work units in an adapter
   that bridges the two: when the work unit yields Pending, the adapter
   registers a Tokio waker that calls the scheduler's resume function.

Option 1 is the cleanest for hilavitkutin's API. The scheduler defines a
`Waker` type (not `std::task::Waker`, since hilavitkutin is `no_std`-
compatible) with a single method:

```
trait Wake {
    fn wake(&self);
}
```

The scheduler provides a `Waker` to each work unit's `execute()` context.
The work unit clones the waker, passes it to its I/O code, and returns
`Yield(Pending)`. When I/O completes, the I/O code calls `waker.wake()`.
The scheduler moves the work unit back to the runnable queue.

### 5.3 The waker problem in no_std

hilavitkutin is designed to be `no_std`-compatible (critical for loimu's
embedded targets and polka-dots' minimal runtime footprint). In `no_std`,
there is no `std::task::Waker` (it requires alloc for the vtable clone),
no epoll/kqueue (they are OS-specific), and no Tokio (it requires std).

This constrains the waker design. hilavitkutin's waker must be:

- Statically sized (no heap allocation for vtable or closure).
- Platform-independent (no OS-specific I/O primitives).
- Simple enough that a `no_std` consumer can implement it (e.g., an
  embedded system with interrupt-driven I/O).

A practical shape: the waker is an index into a scheduler-owned array of
atomic flags. The work unit receives the index. The I/O completion code
(running on any thread, in any context) sets the flag via
`AtomicBool::store(true, Relaxed)`. The scheduler polls the flags of
all pending work units before each scheduling round and resumes any that
are flagged.

This is polling, not event-driven. But the polling happens at scheduling
granularity (once per morsel dispatch round, not per instruction), and
checking an array of atomic bools is a single cache line scan. For
hilavitkutin's scale (tens to hundreds of work units, not millions), this
is fast enough.

Consumers that want event-driven waking (Tokio integration, io_uring
completion notification) can wrap the atomic flag with their own mechanism:
the Tokio waker sets the atomic flag and then notifies the Tokio runtime,
which calls into the scheduler. The scheduler's flag-scan finds the flagged
unit and resumes it. The flag is the lowest-common-denominator mechanism;
event-driven consumers add a notification layer on top.

### 5.4 Hybrid model: scheduler owns yield/resume, consumer owns I/O

The emerging design: hilavitkutin provides the yield/resume contract
(Progress enum, waker flags, runnable queue management). The consumer
provides the I/O integration (Tokio runtime, io_uring ring, background
threads, interrupt handlers). The boundary between them is the waker flag:
consumer-side I/O completion sets the flag, scheduler-side dispatch checks
the flag.

This separation means hilavitkutin never imports any I/O crate. No tokio
dependency, no mio dependency, no libc dependency (for epoll/kqueue). The
scheduler is pure scheduling logic: DAG analysis, morsel dispatch, work
stealing, progress tracking, and flag scanning. Everything I/O-related
lives in the consumer.

## 6. Separation of Concerns: Scheduler vs I/O Runtime

The survey above reveals a consistent pattern across databases, game engines,
and kernel I/O: the systems that age well are the ones that separate
scheduling concerns from I/O concerns.

### 6.1 What the scheduler owns

The scheduler's responsibilities are:

- **Task ordering**: determining which work units are eligible to run based
  on DAG dependencies (upstream units must complete before downstream units
  start), resource availability (Read/Write conflict analysis), and priority
  (critical-path units first).
- **Dependency tracking**: maintaining the DAG of work units, tracking which
  units have completed, which are pending, and which are runnable.
  Decrementing dependency counters when upstream units complete. Adding
  downstream units to the runnable set when all dependencies are satisfied.
- **Work-stealing**: balancing load across worker threads. When a worker's
  local queue is empty, it steals from another worker's queue. The scheduler
  owns the deque data structures and the stealing protocol.
- **Morsel dispatch**: assigning morsel ranges to workers. Tracking how many
  morsels of each input column have been processed. Knowing when a work
  unit has consumed all its input (all morsels processed = unit complete).
- **Progress tracking**: monitoring the state of each work unit (Runnable,
  Running, Yielded, Complete, Failed). Handling state transitions (Running
  to Yielded on Yield, Yielded to Runnable on wake, Running to Complete on
  Complete).

### 6.2 What the I/O runtime owns

The I/O runtime's responsibilities are:

- **I/O submission**: issuing system calls (read, write, connect, sendmsg)
  or writing submission queue entries (io_uring). This is platform-specific
  and often library-specific (reqwest for HTTP, rusqlite for SQLite, tokio
  for TCP).
- **Readiness notification**: detecting when an I/O operation is complete or
  when a file descriptor is ready for reading/writing. This is the event
  loop: epoll_wait, kevent, GetQueuedCompletionStatus, or io_uring CQ
  polling.
- **Waker invocation**: when an I/O operation completes, calling the waker
  associated with the waiting task. This bridges the I/O world back to the
  scheduling world.
- **Connection management**: pooling TCP connections, managing TLS sessions,
  handling DNS resolution. These are I/O-specific concerns that a scheduler
  should never know about.

### 6.3 The integration point

The scheduler and I/O runtime communicate through a narrow interface:

1. **Yield(Pending)**: work unit tells the scheduler "I cannot make progress
   now; I have arranged to be woken when I can." The scheduler parks the
   work unit.
2. **Wake**: I/O completion triggers the waker. The waker sets the flag (or
   calls a callback, or sends a message). The scheduler detects the wake
   and moves the work unit back to runnable.
3. **Resume**: the scheduler dispatches the now-runnable work unit to a
   worker thread. The work unit continues from where it left off (either
   by re-executing from the top with saved state, or by resuming a
   suspended coroutine/fiber).

This interface is small enough to be stable across consumers. A Tokio-based
consumer (saalis) and a bare-metal consumer (loimu on embedded) both use
the same Progress enum and the same waker flag mechanism. Only the I/O code
inside the work unit differs.

### 6.4 Why hilavitkutin should NOT include an I/O runtime

Including an I/O runtime would:

- **Force a dependency on Tokio (or mio, or libc)**: this breaks `no_std`
  compatibility and adds transitive dependencies that not all consumers want.
  loimu does not use Tokio. polka-dots does not need async I/O (its pipeline
  is purely CPU-bound). Only saalis needs Tokio, and saalis should bring its
  own.
- **Duplicate functionality**: saalis already runs a Tokio runtime for its
  HTTP connectors. Adding another I/O runtime inside hilavitkutin would mean
  two runtimes competing for the same OS resources (epoll fds, thread pools).
- **Constrain platform support**: an I/O runtime must handle platform
  differences (epoll vs kqueue vs IOCP vs io_uring). hilavitkutin's scheduler
  is platform-independent. Mixing I/O in would couple scheduling code to
  platform-specific I/O code.
- **Violate separation of concerns**: the scheduler's job is to decide which
  work runs when and where. The I/O runtime's job is to get bytes in and out
  of the process. These are orthogonal concerns. Combining them would make
  the scheduler harder to test (tests would need mock I/O), harder to reason
  about (scheduling bugs vs I/O bugs in the same codebase), and harder to
  optimize (scheduling hot paths would contain I/O cold paths).

The correct design: hilavitkutin provides `Progress::Yield(Pending)`, a
waker mechanism, and runnable-queue management. Consumers bring their own
I/O. The scheduler does not know what the work unit is waiting for.

## 7. Relevance to hilavitkutin

The research above maps directly onto hilavitkutin's three consumers and
their distinct I/O profiles.

### 7.1 saalis connectors: HTTP calls via reqwest

saalis's connector subsystem makes HTTP requests to external APIs (game
store APIs, metadata services, health check endpoints). These are I/O-
bound: each request takes 50-500 ms of wall-clock time, during which the
CPU does nothing useful.

The integration path:

1. The saalis process runs a Tokio runtime (it already does, for reqwest).
2. A connector work unit's `execute()` starts an HTTP request via reqwest.
   reqwest returns a future. The work unit stores the future in its state.
3. The work unit clones the hilavitkutin waker and creates a Tokio task
   that awaits the future and calls `waker.wake()` on completion.
4. The work unit returns `Progress::Yield(Pending)`.
5. The hilavitkutin scheduler parks the work unit and dispatches other
   work (other connectors, CPU-bound data pipeline stages).
6. When the HTTP response arrives, Tokio completes the future, the Tokio
   task calls `waker.wake()`, hilavitkutin's scheduler detects the flag
   and moves the connector work unit back to runnable.
7. The work unit is dispatched again. It reads the HTTP response from its
   stored state, processes it (parses JSON, maps to internal types), writes
   results to output columns, and returns `Progress::Complete`.

This pattern handles multiple concurrent HTTP requests naturally. Each
connector work unit is an independent work unit in the DAG. If 10 connectors
are all waiting on HTTP, 10 work units are parked and the scheduler runs
CPU-bound work on all worker threads. As responses trickle in, the scheduler
resumes connectors one by one, interleaving their processing with ongoing
morsel work.

Retry logic fits the model: a connector that gets a 429 (rate limited) or
503 (service unavailable) returns `Progress::Yield(Retry { attempt: 1,
max: 3, backoff: Duration::from_secs(5) })`. The scheduler parks the work
unit with a timer. When the backoff expires, the scheduler moves the unit
back to runnable. The retry attempt, max, and backoff are all visible to
the scheduler for monitoring and cancellation.

### 7.2 polka pipeline: no I/O in hot path

polka-dots' pipeline is CPU-bound. All input data (polka.toml, .polka files,
catalogue entries) is loaded into memory before the pipeline starts. The
pipeline transforms this data through N-to-IR, IR-to-IR, and IR-to-M stages,
each processing morsels of column data. No stage makes network calls or disk
reads during execution.

For polka-dots, `Progress::Yield(Pending)` is never returned. Every work
unit returns `Progress::Complete` on every call. The yield/resume machinery
exists in hilavitkutin but polka-dots never exercises it. This is fine; the
yield code path is cold (never taken), so it adds zero overhead to the hot
morsel-processing path.

The one exception is polka-dots' file output stage: writing generated
config files to disk. This happens after the pipeline completes, not during
it. The pipeline produces in-memory output; a separate output phase writes
files. If file writes were integrated as pipeline stages, they would be
I/O-bound, but they are not latency-sensitive (writing a few KB to local
disk takes microseconds) and do not benefit from concurrent execution
(sequential writes to the same directory are faster than concurrent writes
due to filesystem journaling).

### 7.3 loimu: GPU fence waiting

loimu submits work to the GPU (draw calls, compute dispatches, buffer
uploads). After submission, the CPU must wait for the GPU to complete
before reading results (e.g., readback buffers, query results). GPU fences
are the synchronization primitive: the CPU submits a fence with the GPU
work and polls or waits for the fence to signal.

This is structurally identical to I/O waiting:

1. A loimu behavior submits GPU work and a fence.
2. The behavior returns `Progress::Yield(Pending)` with a waker tied to
   the fence.
3. The scheduler parks the behavior and runs other work (other behaviors,
   CPU-side computation).
4. The GPU completes the work and signals the fence.
5. The fence monitoring code (a background thread polling fence status, or
   a GPU driver callback on some platforms) calls `waker.wake()`.
6. The scheduler resumes the behavior, which reads the GPU results.

The pattern is the same as saalis's HTTP waiting: submit work to an external
processor, yield, resume when the external processor signals completion.
The scheduler does not know whether the work unit is waiting for a network
response, a disk read, or a GPU fence. It only knows the unit has yielded
Pending and will be woken by its waker.

### 7.4 The common pattern

All three consumers follow the same sequence:

1. Submit work to an external system (HTTP API, GPU, disk).
2. Store a handle to the pending result (future, fence, file descriptor).
3. Arrange for the waker to fire when the external system completes.
4. Return `Progress::Yield(Pending)`.
5. On resume, read the result and continue.

hilavitkutin provides steps 4 and 5 (yield and resume). The consumer
provides steps 1, 2, and 3 (submission, handle storage, waker hookup).
The scheduler sees an opaque yield/resume cycle. The consumer sees an
opaque scheduling decision about when to resume.

This is the correct factoring. The scheduler does not know about HTTP,
GPU fences, or disk I/O. The I/O code does not know about DAG dependencies,
morsel dispatch, or work stealing. The waker flag is the only shared data
structure. Each side can be tested, optimized, and evolved independently.

### 7.5 Summary of surveyed approaches and their mapping

| System | Strategy | I/O mechanism | Scheduler integration | hilavitkutin parallel |
|--------|----------|---------------|----------------------|----------------------|
| PostgreSQL | None (blocking) | Inline syscalls | OS scheduler | N/A |
| CockroachDB | Delegation | Go runtime | Goroutine yield | Progress::Yield via runtime |
| DuckDB | Separation | Pipeline breakers | No I/O in hot path | polka-dots model |
| DataFusion | Delegation | Tokio Stream | Poll::Pending | Progress::Yield(Pending) |
| Game asset loading | Separation | Background threads | Atomic flag poll | Waker flag scan |
| Game networking | Separation | Dedicated I/O thread | Lock-free queue | Consumer I/O thread |
| Game audio | Separation | Ring buffer | Producer/consumer | N/A |
| Naughty Dog fibers | Integration | Fiber yield | Fiber scheduler | Progress::Yield model |
| io_uring | Delegation | Kernel ring buffer | CQ fd in event loop | Consumer io_uring integration |

The consistent lesson: production systems either keep I/O entirely off the
compute path (DuckDB, game engines) or provide a cooperative yield mechanism
and delegate I/O to a separate runtime (DataFusion, CockroachDB, Naughty
Dog fibers). No successful system mixes blocking I/O into the morsel
processing path. hilavitkutin's design -- yield/resume contract in the
scheduler, I/O ownership in the consumer -- aligns with the delegation
strategy used by DataFusion and the Naughty Dog fiber model, while preserving
the option for consumers like polka-dots to use the separation strategy
(no I/O at all in the pipeline).
