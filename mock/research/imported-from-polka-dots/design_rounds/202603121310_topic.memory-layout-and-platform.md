## Purpose

Nail down the memory layout for column storage and op execution across
platforms. Determine whether the 128-bit stride is universal or needs
adaptation, explore no_std feasibility, and consider custom allocator
strategies for the columnar pipeline.

---

## §1 The 128-bit Stride Is Universal

The 128-bit (16-byte) stride per column entry is a data layout decision,
not a SIMD-width decision. It works on every platform:

- On 128-bit SIMD (NEON, SSE): one stride = one register. Optimal.
- On 256-bit SIMD (AVX/AVX2): one register holds 2 strides. Process
  2 entries per SIMD op. Not wasteful, just different throughput.
- On 64-bit only (no SIMD): stride is 2 u64 loads. Still fine for
  sequential access.

The stride encoding (1-byte flags + 15-byte payload) is identical on
all platforms. No adaptation needed for the data layout itself.

## §2 Cross-Platform Hardware Landscape

| Platform | Era | SIMD width | L1 data cache | Cache line | Page size |
|----------|-----|-----------|---------------|------------|-----------|
| Intel Ivy Bridge | 2012 | SSE 128 / AVX 256 | **32 KB** | 64 bytes | 4 KB |
| Intel Haswell-Skylake | 2013-2017 | AVX2 256 | **32 KB** | 64 bytes | 4 KB |
| AMD Zen 1-4 | 2017-2023 | AVX2 256 | **32 KB** | 64 bytes | 4 KB |
| Intel Alder Lake+ | 2021+ | AVX2 256 / AVX-512 | **48 KB** (P-core) | 64 bytes | 4 KB |
| Apple Silicon M1+ | 2020+ | NEON 128 | **128-192 KB** | 64 bytes | 16 KB |
| ARM server (Graviton) | 2018+ | NEON 128 / SVE var | **64 KB** | 64 bytes | 4-64 KB |

**Key observation:** L1 varies 4x across the range (32 KB to 128 KB).
Page size varies 4x (4 KB to 16 KB). Cache line size is universally
64 bytes. SIMD width varies but doesn't affect the stride layout.

**The real variable is L1 cache size.** Everything else is either
universal (cache line, stride layout) or handled by the compiler
(SIMD width).

## §3 Morsel Sizing — How Others Do It

### 3.1 DuckDB — fixed compile-time constant

STANDARD_VECTOR_SIZE = 2048 tuples. Compile-time constant. No runtime
detection, no platform adaptation. Conservative default that fits in
any L1 at 8 bytes/entry (16 KB per column slice).

Morsel size for parallelism = 122,880 rows (60 × 2048). This is the
work unit for thread scheduling, not the cache-fitting unit.

DuckDB does NOT probe hardware. The 2048 constant was chosen once and
works everywhere. For their 8-byte integer columns, 2048 × 8 = 16 KB
fits in even 32 KB L1 with room for 2 columns.

For OUR 16-byte stride, 2048 × 16 = 32 KB per column — the entire
32 KB L1 for just one column. Too aggressive for our wider entries.

### 3.2 Polars — formula-based, not hardware-driven

Polars computes chunk size from thread count and column count:

```
thread_factor = max(12 / n_threads, 1)
chunk_size = max(50_000 / n_cols * thread_factor, 1000)
```

No cache detection. The formula accounts for data width indirectly
(more columns = smaller chunks) but doesn't know the actual L1 size.
Manual override available via `set_streaming_chunk_size()`.

### 3.3 BLIS — runtime detection, the gold standard

BLIS detects CPU microarchitecture via CPUID at startup. Selects
pre-compiled kernel configuration + blocksizes for the specific cache
hierarchy. "Hot swap" — can switch kernels mid-execution.

- Loop tiling sized per cache level (L1, L2, L3)
- Detection happens ONCE at init
- Optimal sizes used throughout the session
- Fallback to generic kernels if detection fails

BLIS is the only project in this survey that actually adapts to the
hardware it's running on. It's also the highest-performance.

### 3.4 fjall — platform-agnostic, fixed blocks

Fixed configurable block sizes (4-64 KiB). No runtime cache or page
detection. Manual configuration via `Config::cache_size()`. Relies on
the OS kernel for page mapping.

Interesting pattern: ByteView — 24-byte struct with inline storage
for values ≤20 bytes, larger values heap-allocated via pointer. Same
concept as our 128-bit stride with 15-byte inline payload.

### 3.5 Apache Arrow DataFusion — configuration-based

Default batch size: 8,192 rows. Configurable, not auto-detected.
Manual tuning for memory-constrained scenarios.

## §4 polka-dots Morsel Strategy

### 4.1 Detection (BLIS-style, primary heuristic)

Detect L1 data cache size once at startup:

- **x86** — CPUID instruction via `raw-cpuid` or `cache-size` crate
- **macOS ARM** — `sysctl hw.l1dcachesize`
- **Linux ARM** — `/sys/devices/system/cpu/cpu0/cache/index0/size`
- **Fallback** — 32 KB (the minimum supported L1, Ivy Bridge era)

Detection runs once. The result is used for the entire session.

### 4.2 Formula (Polars-influenced, secondary factor)

Morsel size is computed from detected L1, column count, and stride:

```
detected_l1 = detect() or 32_768      // fallback 32 KB
usable = detected_l1 * 0.75           // leave room for other cache residents
morsel = usable / (col_count * 16)    // fit all column slices in L1
morsel = clamp(morsel, 512, 8192)     // floor 512, ceiling 8192
morsel = round_down_multiple_of_4(morsel)
```

The 0.75 utilization factor leaves room for instruction cache,
stack, and other transient data that shares L1 capacity.

Morsel size is rounded to a multiple of 4, NOT a power of 2.
Power-of-2 matters for SIMD lane widths and hash tables, not for
batch row counts. The only alignment that matters here is cache
line alignment: a cache line is 64 bytes = 4 entries at 16-byte
stride. Multiples of 4 ensure morsel boundaries don't split cache
lines. Any multiple of 4 works — 500, 768, 1500 are all fine.

Thread count may further reduce morsel size: if multiple threads
process different morsels of the same column, each thread's working
set must fit independently.

### 4.3 Fallback — 32 KB L1, 512-row morsels

When detection fails, assume 32 KB L1. At 512 rows × 16 bytes:

- 8 KB per column slice
- **4 columns fit** in 32 KB L1
- A 3-input + 1-output op works without thrashing
- Justified tradeoff: smaller morsels = slightly more per-batch
  overhead, but correct behavior on constrained hardware

### 4.4 Per-chain morsel sizing

Morsel size is computed per-chain (§8), not per-op or globally. A
chain is a fused sequence of ops that run back-to-back on the same
row range. The morsel size is determined by the **union** of all
unique columns across all ops in the chain — reads and writes both
count, because all must be cache-resident simultaneously.

The principle: having ALL columns in the chain cache-hot simultaneously
is more valuable than processing more rows per batch. A cache miss
costs ~100 cycles. A per-batch setup cost is ~10-20 instructions.
Smaller morsels that keep everything in L1 always win over larger
morsels that thrash.

Different chains have different morsel sizes. A chain touching 3
columns gets large morsels. A chain touching 7 columns gets smaller
ones. Independent chains run on separate cores (each with its own L1),
so their morsel sizes don't interact.

### 4.5 Concrete examples

| Platform | L1 | Usable | 2-col morsel | 5-col morsel | 9-col morsel |
|----------|-----|--------|-------------|-------------|-------------|
| Ivy Bridge | 32 KB | 24 KB | 768 | 512 | 168 |
| AMD Zen 4 | 32 KB | 24 KB | 768 | 512 | 168 |
| Alder Lake | 48 KB | 36 KB | 1128 | 512 | 252 |
| Apple M1 | 128 KB | 96 KB | 3072 | 1200 | 668 |
| Apple M1 Pro | 192 KB | 144 KB | 4608 | 1800 | 1004 |

Per-op sizing means Apple Silicon gets 4-6x larger morsels for
simple ops compared to x86 — AND complex ops with many columns
still fit entirely in L1 everywhere. Neither performance nor
correctness is sacrificed.

## §5 Why Columnar Storage Works Across Memory Regions

Columns are stored as independent contiguous arrays. They are NOT
co-located in memory. This is efficient because of how CPU hardware
works at multiple levels:

### 5.1 Hardware prefetching

When the CPU reads memory sequentially, it detects the pattern and
prefetches future cache lines before they're requested. Modern CPUs
(including Apple Silicon) track 8-16 independent sequential streams
simultaneously. Each column is one stream.

Concrete example — processing Version + LatestVersion:

```
Version at 0x1000_0000:       [V0][V1][V2][V3] [V4][V5][V6][V7] ...
LatestVersion at 0x5000_0000: [L0][L1][L2][L3] [L4][L5][L6][L7] ...
```

1. Read V0 — cache miss. CPU fetches cache line: V0-V3 arrive (64B).
2. Read L0 — cache miss. CPU fetches cache line: L0-L3 arrive (64B).
3-8. V1, L1, V2, L2, V3, L3 — all cache hits (already loaded).
9. Read V4 — prefetcher saw the pattern, already fetched. Hit.
10. Read L4 — prefetcher saw the pattern, already fetched. Hit.

Two cache misses for the first entries. Everything else is hits.
Where the columns are in memory doesn't matter — only that each
column is contiguous and access is sequential.

Row-based layout (`[V0, L0, Name0, Desc0][V1, L1, Name1, Desc1]`)
wastes cache: if you only need 2 of 10 columns, 80% of every cache
line is data you never read. Columnar means 100% useful cache fill.

### 5.2 TLB (Translation Lookaside Buffer)

Every memory access goes through virtual-to-physical address
translation. The TLB caches these translations (~160 L1 entries on
Apple Silicon, ~64 on x86). A TLB miss triggers a page table walk
(4-5 chained memory accesses).

At 512-row morsels (our minimum):
- 512 × 16 bytes = 8 KB per column per morsel
- 4 KB pages (x86): 2 pages per column → 5 columns = 10 TLB entries
- 16 KB pages (Apple Silicon): 1 page per column → 5 columns = 5 entries
- Both fit comfortably in L1 TLB

At 2048-row morsels (Apple Silicon):
- 2048 × 16 bytes = 32 KB per column
- 16 KB pages: 2 pages per column → 5 columns = 10 TLB entries
- Still fine (160-entry TLB)

TLB would hurt with hundreds of columns or huge morsels spanning
many pages. Not our case at any supported morsel size.

### 5.3 Cache associativity

L1 cache is set-associative (8-way on Apple Silicon, varies on x86).
Multiple column addresses COULD map to the same cache set, causing
evictions even when total data fits ("conflict misses"). Statistically
unlikely with typical allocator behavior — addresses are spread across
sets naturally. If column data lives in a single arena (§6), adjacent
columns ARE in nearby addresses, which distributes across cache sets.

### 5.4 Memory bandwidth

Ceiling on RAM-to-CPU throughput. Apple Silicon: ~100-200 GB/s.
polka-dots operates on kilobytes to low megabytes. We will never
approach this limit. Bandwidth matters at GB scale, not ours.

### 5.5 False sharing

If two threads write to the same cache line (64 bytes), the line
bounces between cores. Our 128-bit stride = 16 bytes = 4 entries
per cache line. Morsel boundaries aligned to cache line boundaries
(every 4 rows) prevent different threads from writing to the same
cache line in the same column.

### 5.6 Store buffer / write combining

Sequential writes to the same cache line are combined in the store
buffer before flushing. Our model (sequential writes to contiguous
column array) is the optimal pattern. 4 sequential 16-byte writes
combine into one cache line write.

### 5.7 Branch prediction

With external progress tracking (§7), there are no fence sentinel
entries in column data and therefore no per-entry "data or fence?"
branch in the inner loop. The inner loop is a straight sequential
scan — branch prediction is effectively perfect for the data path.

The only conditional in the dispatch loop (between morsels) is the
progress check, which is a single comparison outside the hot path.

### 5.8 Hardware concern summary

| Concern | Impact at our scale | Why |
|---------|-------------------|-----|
| Prefetching | Handled | Sequential access, CPU tracks multiple streams |
| TLB | ~5-10 entries per morsel | 1-2 pages per column per morsel |
| Cache capacity | Morsel-sized to fit L1 | Detection + formula ensures fit |
| Cache associativity | Unlikely conflict | Allocator spreads addresses |
| Memory bandwidth | Nowhere near limit | KB-MB scale, not GB |
| False sharing | Prevented | Morsel boundaries at cache line boundaries |
| Store buffer | Optimal pattern | Sequential writes, hardware combines |
| Branch prediction | Perfect in inner loop | No fence sentinels — pure sequential data (§7) |

## §6 Static Morsel Sizing

Column count per op is statically known from the `type Read` and `type Write`
associated types. These are ZST marker tuples — the actual column data lives in
storage, not in the types. Rust has no built-in `tuple_arity::<T>()`, so we use
the standard macro-generated trait pattern (same as Bevy ECS, serde, etc.):

```rust
trait ColumnSlices {
    const WIDTH: usize;
    // ... fetch/bind methods
}

macro_rules! impl_column_slices {
    ($($T:ident),*; $n:expr) => {
        impl<$($T: ColumnMarker),*> ColumnSlices for ($($T,)*) {
            const WIDTH: usize = $n;
        }
    };
}

impl_column_slices!(; 0);
impl_column_slices!(A; 1);
impl_column_slices!(A, B; 2);
impl_column_slices!(A, B, C; 3);
// ... up to max supported arity
```

With WIDTH available as a const, individual op column counts are
compile-time known:

```rust
const fn col_count<W: WorkUnit>() -> usize {
    W::Read::WIDTH + W::Write::WIDTH
}
```

`col_count::<W>()` resolves to a literal at compile time via
monomorphization. No manual `COL_COUNT` annotation per WorkUnit.

The morsel size is computed per **chain** (§8), not per op. A chain's
unique column count is the union across all ops in the chain. The
formula uses the chain's union count as input:

```rust
const STRIDE: usize = 16;
const MIN_MORSEL: usize = 64;
const MAX_MORSEL: usize = 8192;
const L1_UTILIZATION: f64 = 0.75;

fn morsel_size(detected_l1: usize, chain_col_count: usize) -> usize {
    let usable = (detected_l1 as f64 * L1_UTILIZATION) as usize;
    let raw = usable / (chain_col_count * STRIDE);
    let clamped = raw.clamp(MIN_MORSEL, MAX_MORSEL);
    clamped & !3 // round down to multiple of 4
}
```

The individual `col_count::<W>()` values feed into chain partitioning
(§8.3) — the Schedule sums the unique columns across all ops in each
chain and uses that total for morsel sizing.

`max_cols` — the maximum columns per chain — is derived from the
platform:

```rust
fn max_cols(detected_l1: usize) -> usize {
    let usable = (detected_l1 as f64 * L1_UTILIZATION) as usize;
    usable / (MIN_MORSEL * STRIDE)
}
```

On 32 KB L1 (24 KB usable): `max_cols = 24576 / (64 × 16) = 24`.
On 128 KB L1 (96 KB usable): `max_cols = 96000 / (64 × 16) = 93`.
Chains exceeding `max_cols` would thrash at any morsel size — the
Schedule breaks them at this boundary (§8.3).

## §7 Dependency Tracking — External Progress, Not Inline Fences

### 7.1 The problem

Morsels are cache-sized batches of rows. Dependencies between ops mean
"don't read past row N until the upstream op has written past row N."
These two concepts must compose: how does the Executor know when a
downstream op's morsel is safe to dispatch?

### 7.2 Why NOT inline fence entries

The paradigm document (§11.7) originally proposed encoding fences as
sentinel entries inline in the column data — a special 16-byte entry
with an `is_fence` flag bit, a `reached` status bit, and a pointer to
the dependency fence. This was motivated by wanting self-describing
column data where dependency information travels with the data.

We are abandoning inline fences for several reasons:

**They pollute the data stream.** Every fence entry is 16 bytes of
non-data in an otherwise contiguous data array. The inner processing
loop must check every entry's flags byte to distinguish data from
fence — that's a branch per entry in the hottest path. Even with
~99.9% branch prediction accuracy, the check itself costs an
instruction per entry, and the mere existence of mixed-type entries
prevents the compiler from autovectorizing the inner loop.

**They require pointer chasing.** The inline fence model stores a
pointer to the upstream fence entry. Resolving a fence means following
that pointer to another column's memory region, reading its `reached`
bit, potentially waiting. This is a random memory access in the middle
of what should be a sequential scan — a cache miss by design.

**They are redundant.** The Schedule builds the full dependency DAG.
The Executor knows every op, its morsel size, and which ops depend on
which. Fence positions are computable from this information — they
don't need to be stored in the data. Encoding them inline duplicates
information the Executor already has.

**They waste column space.** Each fence entry displaces a real data
entry. With per-column fences at every dependency boundary, a column
participating in multiple dependency chains accumulates multiple
16-byte fence entries. This shifts all subsequent row indices, meaning
"row 500" in the logical data is not at byte offset `500 * 16` in the
physical column — indexing becomes indirect.

**Self-describing data is not a real use case.** Columns are not
persisted with dependency metadata. They are not passed between
systems. The dependency graph is a runtime execution concern that
lives and dies with the Executor. There is no consumer of "column
data with embedded fence markers" other than the Executor itself,
which already has better information.

### 7.3 External progress tracking

Each op gets one `AtomicUsize` — a monotonically increasing row index
representing how far the op has progressed. Starts at 0. Updated after
each morsel completes.

**Producer side** (op A finishes a morsel covering rows 0..768):

```rust
progress_a.store(768, Ordering::Release);
```

**Consumer side** (op B needs rows 0..512 from A's output):

```rust
if progress_a.load(Ordering::Acquire) >= 512 {
    // safe to dispatch B's morsel [0..512)
}
```

### 7.4 Why a plain store, not fetch_add or fetch_or

We considered three atomic signaling strategies:

**Bitfield with fetch_or** — flip bit N when morsel N completes.
`progress.fetch_or(1 << morsel_idx, Release)`. Consumer checks
individual bits. Appealing because bit flipping feels cheap.

**Counter with fetch_add** — increment by morsel size per completion.
`progress.fetch_add(morsel_size, Release)`. Consumer checks
`progress >= needed`.

**Monotonic store** — write the completed-up-to row index directly.
`progress.store(completed_row, Release)`. Consumer checks
`progress >= needed`.

Both `fetch_or` and `fetch_add` are **read-modify-write (RMW)**
operations. On x86 they compile to `lock or` / `lock xadd` — the CPU
must take exclusive ownership of the cache line, read the old value,
modify it, and write back, all atomically. On ARM they compile to a
`ldaxr`/`stlxr` (load-exclusive / store-exclusive) retry loop that
can fail under contention.

A plain `store` with `Release` ordering is NOT an RMW. On x86 it
compiles to a single `mov` — x86's Total Store Order means Release
semantics are free; every store is already Release. On ARM it compiles
to a single `stlr` (store-release) instruction. No cache line lock,
no retry loop, no contention.

The producer already knows exactly what value to write — it just
finished processing up to row N, so it stores N. There is no need to
read the previous value. This makes the plain store strictly cheaper
than any RMW alternative.

The bitfield approach also imposes a hard ceiling: 64 morsels per op
on a 64-bit system (one bit per morsel). A column with 100,000 rows
and 512-row morsels needs ~195 morsels — exceeds 64 bits. A usize
row index has no such limit.

### 7.5 The dispatch loop

All dependency checking lives in the Executor's dispatch loop,
**between** morsels. The inner processing loop — `process_batch()` —
never sees a fence, never checks a flag, never branches on dependency
state. It receives a contiguous slice of pure data and processes it
sequentially.

```
Executor dispatch loop:
  for each ready morsel:
      if all dependencies satisfied:          ← progress.load() here
          op.process_batch(read_slice, write_slice)
          progress.store(new_row_idx, Release) ← progress.store() here
```

The `process_batch` inner loop:

```
process_batch(read: &[Stride], write: &mut [Stride]):
    for i in 0..morsel_len:
        write[i] = transform(read[i])        ← pure data, no branches
```

This is the critical property: **zero fence-related overhead in the
hot path.** The dispatch loop runs once per morsel (every 512-8192
rows). The inner loop runs once per row. All dependency logic is
amortized over the morsel — one load + one compare + one store per
morsel, versus zero cost per row.

### 7.6 Pre-computed dispatch table

The Schedule knows the DAG topology. After L1 detection and morsel
sizing, the Executor builds a dispatch table: an ordered list of
(op, start_row, end_row) work items with their dependency constraints.

```
Dispatch table for a 2000-row pipeline, op A → op B:

A morsel [0, 768)     depends on: nothing
A morsel [768, 1536)  depends on: nothing
A morsel [1536, 2000) depends on: nothing
B morsel [0, 512)     depends on: A.progress >= 512
B morsel [512, 1024)  depends on: A.progress >= 1024
B morsel [1024, 1536) depends on: A.progress >= 1536
B morsel [1536, 2000) depends on: A.progress >= 2000
```

Note: B's morsel boundaries differ from A's (different column count
→ different morsel size, per §4.4). The dependency constraint is on
row index, not morsel index — B's morsel [0, 512) needs A to have
completed at least up to row 512, regardless of how A's morsels are
sized.

The dispatch table is built once per Schedule. No fence scanning, no
pointer chasing, no per-entry flag checks. The Executor walks the
table, checks progress atomics, and dispatches ready morsels.

### 7.7 Column data is pure data

With external progress tracking, the 128-bit stride encoding
simplifies. There are no fence sentinel entries. Every entry in a
column is real data:

```
Byte 0: flags (reserved for data-only concerns — null, tombstone, etc.)
Bytes 1-15: payload
```

The `is_fence` and `reached` bits from the original design are gone.
The flags byte is entirely available for future data-level concerns.
Column indexing is direct: row N is at byte offset `N * 16`. No
displaced indices from interleaved fence entries.

### 7.8 Implications for the paradigm document

This section supersedes §11.7 (Per-Column Fences) in the paradigm
document. The fence concept is replaced by external atomic progress
counters. The column encoding in §11.5 loses its fence-related flag
bits. These changes should be reflected in the paradigm document when
it is next updated.

## §8 Execution Model — Two-Step Scheduling and Fused Chains

### 8.1 Overview

The execution model has three components, each with a distinct role:

1. **Schedule** (step 1) — builds the dependency DAG from WorkUnit
   declarations. Pure topology: "op A must complete before op B."
   Assumes infinite resources.
2. **Chain partitioner** (step 2) — partitions the DAG into fused
   chains, respecting `max_cols`. Computes morsel sizes per chain,
   places fences between chains, assigns dispatch priority.
3. **Executor** — runs the chains. The library ships a default
   thread-pool Executor. The Executor trait is available for
   consumers who want to override (test harness, profiling, etc.).

The library handles all three internally. The consumer calls
`Pipeline::run()` and gets results.

### 8.2 Hardware detection

The library detects at startup (once):

- **Minimum L1 data cache** across all core types — handles
  asymmetric architectures (Apple Silicon P-cores vs E-cores).
  Uses the minimum to guarantee morsels fit regardless of which
  core the OS assigns a thread to.
- **Physical core count** — excludes hyperthreads / SMT logical
  cores. This is the maximum concurrent chains.

Detection methods (same as §4.1 for L1, extended for core count):

- **x86** — CPUID for L1, `/proc/cpuinfo` or `sysctl` for physical
  core count
- **macOS ARM** — `sysctl hw.l1dcachesize` (minimum across core
  types), `sysctl hw.physicalcpu`
- **Linux ARM** — sysfs for L1 per core (take minimum),
  `/sys/devices/system/cpu/` for physical count
- **Fallback** — 32 KB L1, 1 core (safe minimum)

From these, the library derives:

```
usable_l1  = min_l1 * 0.75
max_cols   = usable_l1 / (MIN_MORSEL * 16)
max_threads = physical_core_count
```

### 8.3 Step 1 — DAG construction

The Schedule builds the dependency DAG from all registered WorkUnit
types. Each WorkUnit declares `type Read` and `type Write` as tuples
of column markers. A dependency edge exists when one op writes a
column that another op reads.

This DAG is the canonical dependency order. It is correct (respects
all data dependencies) but NOT optimized — it assumes every column
is readable at any time, with infinite cache and infinite parallelism.

### 8.4 Step 2 — Chain partitioning

The chain partitioner transforms the DAG into an execution plan.
The goal: **minimize the number of chains** (each chain = one
iteration through the data). Fewer chains = fewer data walks =
less redundant cache loading.

A **chain** is a fused sequence of ops that run back-to-back within
one morsel. Within a chain, for each morsel (row range), every op
runs in sequence: writes happen before the reads that depend on them.
No fences, no atomics, no synchronization within a chain. Just a
tight loop.

**What makes an op eligible to join a chain:**

An op joins a chain if ALL of its read columns are written by ops
already in the chain (or are pre-existing source columns loaded by
a chain member). The write→read dependency is what links ops: an
op joins because it reads what the chain already writes.

**What forces a chain break:**

1. **Fan-in** — an op reads columns from two different chains. It
   cannot fuse with either; it starts a new chain with fence
   dependencies on both upstream chains.
2. **Column count limit** — adding the op would push the chain's
   unique column count past `max_cols`. The data would not fit in
   L1 at any reasonable morsel size. The chain breaks, and the op
   starts a new chain.
3. **Pipeline breakers** — ops that require ALL input rows before
   producing any output (sort, full-table aggregate). They cannot
   process data morsel-by-morsel, so they force a materialized
   boundary.

**The algorithm:**

```
1. Topologically sort all ops in the DAG
2. For each op in topological order:
   a. Identify which chain(s) contain the ops that write
      this op's read columns
   b. If exactly one upstream chain, AND adding this op's
      new columns keeps union_cols ≤ max_cols:
      → append op to that chain (writes before dependent reads)
   c. Otherwise (multiple upstream chains, or would exceed
      max_cols, or pipeline breaker):
      → start a new chain
      → record fence dependencies on upstream chain(s)
3. For each chain, compute morsel_size from union_cols and usable_l1
4. Compute dispatch priority per chain (§8.6)
```

### 8.5 Fused chain execution

Within a chain, all ops run in sequence per morsel. No fences, no
progress tracking, no atomics. The morsel size is derived from the
chain's union column count — all columns across all ops in the chain
must fit in one core's L1 simultaneously.

```
Chain [LoadCatalogue → ResolveVersions → AuditPass → FormatAudit]
  Union columns: Name, Installed, Version, LatestVer, Outdated,
                 AuditResult, AuditOutput = 7
  Morsel size: usable_l1 / (7 × 16), rounded to multiple of 4

Morsel [0..N):
  LoadCatalogue.process_batch(0..N)     // writes Name, Installed, Version
  ResolveVersions.process_batch(0..N)   // reads Installed, Version → writes LatestVer, Outdated
  AuditPass.process_batch(0..N)         // reads 4 cols → writes AuditResult
  FormatAudit.process_batch(0..N)       // reads AuditResult, Name → writes AuditOutput
  ← all 7 columns cache-hot throughout, single walk through the data →

Morsel [N..2N):
  LoadCatalogue.process_batch(N..2N)
  ResolveVersions.process_batch(N..2N)
  AuditPass.process_batch(N..2N)
  FormatAudit.process_batch(N..2N)

... until all rows processed.
```

One walk. One core. One L1 cache. All ops fused. This is the default
execution mode for any linear dependency chain that fits in `max_cols`.

### 8.6 Inter-chain fences and pipeline parallelism

Fences exist only between chains, never within. A fence is an
`AtomicUsize` per chain — a monotonic row index tracking how far the
chain has progressed. Updated after each morsel completes.

A downstream chain's morsel [s, e) can dispatch as soon as all
upstream chains have progressed past row e. The downstream chain
lags by one morsel — not waiting for full completion. This is
pipeline parallelism: chains overlap in time.

```
Chain A (independent): [LoadCatalogue → ResolveVersions → AuditPass → FormatAudit]
Chain B (independent): [LoadKeybinds → ResolveKeybinds]
Chain C (depends on A and B): [FinalReport]  ← reads from both chains

Execution on 3 cores:

Core 0 (Chain A): morsel [0..N) → morsel [N..2N) → morsel [2N..3N) → ...
Core 1 (Chain B): morsel [0..M) → morsel [M..2M) → ...
Core 2 (Chain C): (waits) → morsel [0..K) → morsel [K..2K) → ...
                            ↑ starts when A.progress ≥ K AND B.progress ≥ K
```

Chain C starts reading as soon as ONE morsel's worth of rows is
available from both upstream chains. It does not wait for A or B
to finish entirely. If A completes morsel [0..N) quickly and B is
still loading, C waits for B — the fence on the slower chain is
the bottleneck.

The morsel sizes differ between chains (different union column
counts). Chain A might have morsel size 340 (7 cols), Chain B
might have 3072 (2 cols), Chain C might have 1500 (3 cols). The
fence dependency is on row index, not morsel index — Chain C's
morsel [0..1500) needs A.progress ≥ 1500 and B.progress ≥ 1500,
regardless of how A and B's own morsels are sized.

### 8.7 Dispatch priority — "unblocks most"

When multiple chains are ready (all fence deps satisfied), the
library prefers the chain that transitively unblocks the most
downstream work. This is pre-computed from the chain dependency
graph:

```
for each chain in reverse dependency order:
    downstream_weight = 0
    for each chain that depends on this one:
        downstream_weight += 1 + that_chain.priority
    chain.priority = downstream_weight
```

A chain at the root of a deep dependency tree gets high priority.
A leaf chain (nothing depends on it) gets priority 0. The default
Executor uses priority to decide which chain gets a core first.

### 8.8 Thread model

The library's default Executor creates one thread per physical core
(detected at startup). Each thread processes one chain at a time.
Independent chains run on separate cores simultaneously — each core
has its own L1, so no cache contention.

No more chains run concurrently than there are physical cores. If
there are 12 chains and 8 cores, at most 8 run at once. The
remaining chains wait for a core to free up.

L1 cache is per-core, not shared. Two chains on different cores
each get full L1. The morsel size is computed for one core's L1 —
parallel execution does not reduce the effective cache per chain.

### 8.9 Column storage

Column storage is a trait. The library defines the contract:

```rust
trait ColumnStorage {
    fn allocate(&mut self, column: usize, rows: usize) -> Result<()>;
    fn slice(&self, column: usize, start: usize, end: usize) -> &[Stride];
    fn slice_mut(&mut self, column: usize, start: usize, end: usize) -> &mut [Stride];
}
```

Columns are identified by plain `usize` indices assigned by the
Schedule during DAG construction. Column marker types (ZSTs) do
their job at compile time — constraining the DAG, enforcing type
safety, determining morsel sizes. They are erased by the time
execution begins. The Schedule lowers types to indices.

The consumer's storage impl decides the backing memory: arena,
mmap, vec-per-column, or anything else. The library never calls
an allocator directly.

### 8.10 Data loading — loaders as root ops

Data loading is part of the pipeline, not a separate phase.
Loader ops are WorkUnits with no Read columns (only Write):

```rust
impl WorkUnit for LoadCatalogue {
    type Read = ();
    type Write = (In<CatalogueIR, Name>, In<CatalogueIR, Installed>,
                  In<CatalogueIR, Version>);
    fn process_batch(read: Self::Read, write: Self::Write) -> Result<()>;
}
```

Loaders are root nodes in the DAG — no dependencies, highest
priority. They read from their data source (filesystem, HTTP,
whatever) and write to columns. The data source is the loader's
concern; the Schedule doesn't know or care where data comes from.

Row counts are not known until loaders complete. Downstream chains
compute their morsel ranges just-in-time — when their fence
dependencies are first satisfied, the upstream column lengths are
known. This is not a special case: the progress counter naturally
provides the current row count, and morsel ranges are computed from
(0..current_progress).step_by(morsel_size).

### 8.11 Row count and just-in-time morsel ranges

The dispatch table's morsel ranges are not ALL computed up front.
The chain structure, morsel sizes, and fence dependencies are
computed at plan time. But the concrete row ranges depend on data
size, which is only known after loaders finish.

The Executor computes morsel ranges per chain when the chain first
becomes dispatchable (all fence deps satisfied). At that point,
all upstream columns are finalized and their lengths are known.
This is not a runtime cost — it's a single division
(`total_rows / morsel_size`) done once per chain.

For chains whose inputs come from other transform chains (not
loaders), the row count cascades: if the upstream chain processed
N rows, the downstream chain also processes N rows. The count
propagates through the chain graph.

### 8.12 The consumer API

The library provides one ergonomic entry point:

```rust
let results = Pipeline::<MyStorage>::run(my_storage)?;
```

Internally, this:
1. Detects hardware (min L1, physical core count)
2. Builds the DAG from registered WorkUnits
3. Partitions into chains (respecting max_cols)
4. Computes morsel sizes and fence dependencies
5. Spawns thread pool (physical core count)
6. Dispatches chains with priority-based scheduling
7. Returns storage with all output columns populated

The consumer provides a `ColumnStorage` implementation. Everything
else — scheduling, morsel sizing, fencing, threading — is handled
by the library. The `Executor` trait exists as an override point
for consumers who need custom dispatch (testing, profiling, exotic
platforms), but the default Executor is built-in and handles all
hardware concerns.

### 8.13 Concrete example — full pipeline

```
Ops:
  LoadCatalogue:    writes [Name, Installed, Version]
  LoadKeybinds:     writes [KeybindRaw]
  ResolveVersions:  reads [Installed, Version]          writes [LatestVer, Outdated]
  AuditPass:        reads [Installed, Version, LatestVer, Outdated]  writes [AuditResult]
  FormatAudit:      reads [AuditResult, Name]           writes [AuditOutput]
  ResolveKeybinds:  reads [KeybindRaw]                  writes [Resolved]
  FinalReport:      reads [AuditOutput, Resolved]       writes [Report]

DAG:
  LoadCatalogue → ResolveVersions → AuditPass → FormatAudit ─┐
                                                              ├→ FinalReport
  LoadKeybinds → ResolveKeybinds ─────────────────────────────┘
```

Chain partitioning (on 128 KB L1, max_cols = 93):

```
Chain A: [LoadCatalogue → ResolveVersions → AuditPass → FormatAudit]
  Union cols: Name, Installed, Version, LatestVer, Outdated, AuditResult, AuditOutput = 7
  Morsel: 96000 / (7 × 16) = 857 → 856 rows (multiple of 4)

Chain B: [LoadKeybinds → ResolveKeybinds]
  Union cols: KeybindRaw, Resolved = 2
  Morsel: 96000 / (2 × 16) = 3000 → 3000 rows

Chain C: [FinalReport]
  Union cols: AuditOutput, Resolved, Report = 3
  Morsel: 96000 / (3 × 16) = 2000 → 2000 rows
  Fences: A.progress ≥ end AND B.progress ≥ end
```

Chain C is a separate chain because it reads from both A and B
(fan-in). It cannot fuse with either without running the other
chain's data twice.

Execution on 3 cores, 2000 catalogue entries, 500 keybinds:

```
Core 0 (Chain A): [0..856) → [856..1712) → [1712..2000)
  Each morsel: Load → Resolve → Audit → Format (4 ops, fused)
  Progress: 856 → 1712 → 2000

Core 1 (Chain B): [0..500)
  One morsel covers all 500 keybinds (morsel 3000 > 500)
  Progress: 500 (done)

Core 2 (Chain C): waits → [0..2000)
  Needs A.progress ≥ 2000 AND B.progress ≥ 2000
  B finishes instantly. A takes 3 morsels. C dispatches after A completes.
  One morsel covers all 2000 rows (morsel 2000 = row count).
```

Total data walks: Chain A walks its 7 columns once. Chain B walks
its 2 columns once. Chain C walks its 3 columns once. Three walks
total — the minimum possible for this DAG structure.
