## 1. Database Columnar Storage

### 1.1 Apache Arrow memory format

Apache Arrow defines a language-independent columnar memory format designed for
analytical computation. The format specification (as of Arrow 17.0) covers
fixed-width, variable-width, and nested types with a consistent buffer layout
that enables zero-copy reads across process boundaries and programming
languages.

**Buffer layout for a fixed-width column (e.g., Int32):**

```
[validity bitmap buffer][data buffer]
```

The validity bitmap uses one bit per value to track nulls. Bit index `i`
corresponds to value index `i`. A set bit means the value is valid (non-null);
a clear bit means null. The bitmap is padded to 64-byte boundaries. Arrow
allocates a validity bitmap even for non-nullable columns unless the producer
explicitly omits it, because consumers cannot statically distinguish "no nulls"
from "bitmap not present" without schema metadata.

The data buffer holds values contiguously at the type's natural width. For
Int32, this is 4 bytes per value. For Float64, 8 bytes. The buffer is also
padded to 64-byte boundaries to enable SIMD-aligned reads.

**Variable-width types (e.g., Utf8, Binary):**

```
[validity bitmap][offsets buffer (int32 or int64)][data buffer]
```

The offsets buffer has `n + 1` entries for `n` values. The string at index `i`
spans bytes `offsets[i]..offsets[i+1]` in the data buffer. This allows O(1)
random access to any string without scanning. The Large variants (LargeUtf8,
LargeBinary) use int64 offsets for data buffers exceeding 2 GB.

**Dictionary encoding:**

Arrow supports dictionary-encoded columns where the data buffer stores integer
indices into a separate dictionary array. The dictionary array holds the unique
values. This is Arrow's equivalent of categorical/factor columns in R or pandas.
Dictionary encoding reduces memory consumption when cardinality is low relative
to row count: a column of 10 million country codes (200 unique values) stores
10M int16 indices plus 200 strings, rather than 10M inline strings.

**Nested types (Struct, List, Map, Union):**

Struct columns decompose into child arrays, one per field. A `Struct<name:
Utf8, age: Int32>` column is stored as two separate child arrays (a Utf8 array
and an Int32 array) plus a validity bitmap at the struct level. This is
columnar decomposition applied recursively: even nested fields are stored in
contiguous, type-homogeneous buffers.

List columns use an offsets buffer (like Utf8) pointing into a single child
array. A `List<Int32>` column stores all integers from all lists contiguously
in one child Int32 array, with the offsets buffer delineating list boundaries.

**What Arrow does NOT do:**

Arrow does not bit-pack below byte boundaries. The smallest fixed-width type
is 1 byte (Boolean uses 1 bit per value in the data buffer, but this is a
special case with dedicated accessor functions). There is no general mechanism
for storing a 4-bit integer in 4 bits. Arrow prioritizes zero-copy reads:
every value is accessible via a pointer dereference at a byte-aligned offset,
without shift-and-mask extraction.

This is a deliberate trade-off. Arrow's design assumes that analytical
workloads are memory-bandwidth-bound, not memory-capacity-bound. Bit-packing
saves space but requires extraction logic on every access, which adds
instructions to the critical path. Arrow bets that the simpler access pattern
(load from byte-aligned address) is faster in practice for vectorized
execution, where the CPU is processing thousands of values per operator
invocation.

Compare this with loimu's bit-packing approach. loimu stores `uint4` values
in 4-bit slots (16 values per u64 word). This prioritizes storage density
over access simplicity. The extraction cost is paid in the unpack kernel,
which converts from packed storage to computation-width registers before
any operator runs. loimu's bet is that the unpack cost is amortized over
the computation performed on each morsel, and that the storage density gain
improves cache utilization enough to compensate. The two approaches reflect
genuinely different workload assumptions: Arrow targets analytical SQL (large
scans over many columns), loimu targets UI frameworks (smaller data volumes
but tighter memory budgets and more diverse value widths).

### 1.2 DuckDB storage architecture

DuckDB (Raasveldt and Muhlbauer, SIGMOD 2019; continued development through
2026) implements a single-file analytical database with a storage layer
designed for mixed OLAP/OLTP workloads on local machines.

**Row groups.** DuckDB organizes data into row groups of approximately 122,880
tuples (120 * 1024). Each row group contains one segment per column. Segments
are independently compressed and independently readable. The row group is the
unit of parallel scan: different threads process different row groups.

**Per-segment compression.** Each segment selects its compression algorithm
independently based on the data distribution:

- **Constant:** All values in the segment are identical. Stores one value plus
  a count. Common for nullable columns where an entire segment is NULL, or for
  partitioned data where a partition key has one value per segment.
- **Run-Length Encoding (RLE):** Consecutive runs of identical values. Stores
  (value, run_length) pairs. Effective for sorted or nearly-sorted columns.
- **Bit-packing:** Values are stored at their minimum required bit width. A
  column of integers in range 0..15 uses 4 bits per value. DuckDB computes the
  minimum bit width per segment, not per table, so a column with values 0..15
  in one segment and 0..1000000 in another uses 4-bit packing for the first
  and 20-bit packing for the second. This is the closest analogue to loimu's
  sub-word packing, but applied at the segment level rather than the type level.
- **Frame of Reference (FOR):** Subtracts a base value from all values in the
  segment, then bit-packs the residuals. A column of timestamps within a
  10-second window stores the base timestamp once and the microsecond offsets
  in a few bits each.
- **Dictionary:** Like Arrow's dictionary encoding, but per-segment. Low-
  cardinality segments use dictionary encoding; high-cardinality segments fall
  through to other methods.
- **FSST (Fast Static Symbol Table):** String-specific compression developed
  by Boncz, Neumann et al. (VLDB 2020). Builds a 256-entry symbol table of
  common substrings and replaces occurrences with 1-byte codes. Achieves 3-5x
  compression on typical string columns while maintaining random access
  (each compressed string can be decompressed independently).

**Vectorized execution.** DuckDB processes data in vectors of 2048 values. This
vector size was chosen to fit in L1 cache on most architectures (2048 * 8 bytes
= 16 KB for 64-bit values; typical L1 data cache is 32-64 KB). Operators
receive a DataChunk (a collection of vectors, one per column referenced by the
operator) and process all 2048 values in a tight loop.

The vector is the unit of work within a thread. The row group is the unit of
parallelism across threads. A scan operator processes one row group at a time
on its assigned thread, producing 2048-value vectors from each segment. Filter,
project, hash, and aggregate operators consume and produce vectors of the same
size (or smaller, after filtering).

**Buffer management.** DuckDB uses a buffer manager that pages row group
segments in and out of memory. Cold segments (not recently accessed) are evicted
to disk; hot segments stay in memory. The buffer manager maintains a reference
count per buffer block and uses a clock-based eviction policy. This is
conceptually similar to OS page cache management but at the segment granularity
rather than the page granularity.

### 1.3 Vectorized vs compiled execution

The analytical database community has debated two execution paradigms since the
mid-2000s.

**Vectorized execution (MonetDB/X100, Zukowski et al. 2005):** Operators are
interpreted (function pointers or virtual dispatch), but they process vectors
of values rather than individual tuples. The interpretation overhead
(function call per operator) is amortized over the vector size. A filter
operator processes 2048 values per function call, so the per-value cost of
the function call is negligible. The inner loop over the vector is a tight,
branch-free loop that the CPU can pipeline and vectorize (SIMD). MonetDB/X100
demonstrated 10-100x speedup over the Volcano iterator model by eliminating
per-tuple function call overhead.

**Compiled execution (HyPer, Neumann 2011):** The query is compiled to machine
code at query time. No function pointers, no virtual dispatch, no vector
boundaries. The generated code processes one tuple at a time through the entire
pipeline (from scan to materialization) in a single tight loop. Pipeline
breakers (hash build, sort) cause materialization, but within a pipeline, the
tuple flows through all operators without ever being written to memory. This
eliminates materialization overhead between operators and enables the CPU to
keep the tuple in registers throughout the pipeline.

**DuckDB chose vectorized.** The pragmatic reasons: vectorized execution is
simpler to implement (each operator is a standalone function operating on
arrays), simpler to debug (you can inspect vectors between operators), and
simpler to extend (adding a new operator means writing one function, not
modifying a code generator). The performance difference between vectorized and
compiled is small on modern hardware with wide SIMD registers, because the
vectorized inner loop already achieves near-optimal throughput.

**Umbra (Kersten et al., VLDB 2021) chose compiled with adaptive fallback.**
Umbra is the successor to HyPer, also from TU Munich. It compiles queries to
machine code but falls back to vectorized interpretation for queries where
compilation latency would exceed execution time (simple point lookups,
short-running queries). The adaptive approach gives the best of both worlds:
compiled execution for long-running analytical queries, interpreted execution
for OLTP-style point queries.

**Implications for hilavitkutin's data layout:** Both paradigms assume columnar
storage. Vectorized execution reads columns into vectors and processes them;
compiled execution generates code that reads column values directly. The storage
format (contiguous, fixed-stride, type-homogeneous buffers) is the same. The
difference is in how operators are dispatched, not how data is stored. This
means hilavitkutin's storage abstraction can serve either execution model. The
storage contract is: provide contiguous, typed slices of values. Whether the
consumer iterates those slices in an interpreted loop (vectorized) or in
generated code (compiled) is the consumer's concern.

## 2. ECS Storage in Game Engines

### 2.1 Archetype storage (Bevy ECS)

Bevy ECS (as of Bevy 0.15) organizes entity data by archetype. An archetype is
the set of component types that an entity has. All entities with components
`(Position, Velocity, Sprite)` share one archetype; entities with
`(Position, Velocity, Health)` share a different archetype.

Each archetype owns a `Table`. The Table contains one `Column` per component
type in the archetype. Each Column is a contiguous array of values of that
component type. Entity data within an archetype is stored in struct-of-arrays
(SoA) layout: all Position values contiguous, all Velocity values contiguous.

**Why archetype storage works well for iteration:** A query like
`Query<(&Position, &Velocity)>` matches all archetypes that contain both
Position and Velocity. For each matching archetype, the query iterates the
Position column and Velocity column in lockstep. Both columns have the same
length (the number of entities in that archetype) and the same row order (row
`i` in Position and row `i` in Velocity belong to the same entity). The
iteration is two pointer walks over contiguous arrays, cache-friendly by
construction.

**The migration problem:** Adding or removing a component changes an entity's
archetype. If entity E has archetype `(Position, Velocity)` and you add a
Health component, entity E must migrate to archetype
`(Position, Velocity, Health)`. Migration involves:

1. Allocating a row in the target archetype's Table.
2. Copying the entity's Position and Velocity values from the source Table to
   the target Table.
3. Moving the last entity in the source Table into the vacated row (swap-remove
   to avoid holes).
4. Updating the entity-to-archetype lookup.

Migration is O(number of components on the entity). For entities with many
components, migration is expensive. For entities that frequently add/remove
components, archetype storage creates churn. In practice, most game entities
have stable component sets after initialization, so migration is rare during
steady-state gameplay. But ECS patterns that use marker components (add a
`Poisoned` component to mark an entity as poisoned, remove it when cured)
generate frequent migrations.

**Bevy's Table and Column internals:** Bevy's `Table` is a struct containing a
`HashMap<ComponentId, Column>` (mapping component type IDs to their columns),
plus entity metadata. Each `Column` wraps a `BlobVec`: a type-erased `Vec<u8>`
that stores values at a runtime-known `Layout` (size and alignment). `BlobVec`
manages the raw allocation, grow/shrink, and slot access. Typed access is
recovered through `Column::get_data_unchecked<T>()` methods that cast the raw
byte pointer to `&T`, relying on the caller to provide the correct type. This
is safe at the API boundary because Bevy's query system enforces type
correctness at compile time through `Query<&T>` parameters, but the storage
itself is type-erased.

### 2.2 Sparse set storage (EnTT, C++)

EnTT (Caini, C++17) uses a different storage strategy: sparse sets. Each
component type has its own `sparse_set<T>`, independent of other component
types.

A sparse set consists of two arrays:

- **Dense array:** Stores (entity_id, component_value) pairs contiguously.
  Iteration over all entities with this component is a linear scan of the
  dense array. The dense array is packed: no holes, no invalid entries.
- **Sparse array:** Maps entity ID to dense array index. Conceptually a
  `Vec<Option<usize>>` indexed by entity ID, but implemented as a paged
  sparse array to avoid allocating memory proportional to the maximum entity
  ID.

**Operations:**

- **Has component:** Check `sparse[entity_id]`; if it maps to a valid dense
  index and `dense[index].entity == entity_id`, the entity has the component.
  O(1).
- **Get component:** Same check, then return `&dense[index].value`. O(1).
- **Add component:** Append to the end of the dense array, set
  `sparse[entity_id] = dense.len() - 1`. O(1) amortized (Vec growth).
- **Remove component:** Swap-remove from the dense array: move the last entry
  into the removed slot, update the sparse array for the moved entity, clear
  the sparse array entry for the removed entity. O(1).
- **Iterate all:** Linear scan of the dense array. O(n) where n is the number
  of entities with this component.

**The tradeoff vs archetype storage:**

Adding and removing components is O(1) with sparse sets (no migration between
tables). This makes sparse sets better for ECS patterns with frequent component
mutation (marker components, temporary state). Single-component iteration is
equally cache-friendly (dense array is contiguous).

Multi-component iteration is where sparse sets lose. To iterate all entities
with both Position and Velocity, EnTT iterates the smaller sparse set (say
Velocity) and, for each entity in Velocity's dense array, probes Position's
sparse set to check if that entity also has Position. This probe is a random
access into Position's sparse array, which is not cache-friendly. If Position
and Velocity are on mostly the same entities, this amounts to a random-access
lookup per entity, versus archetype storage's lockstep linear scan.

EnTT's `sort()` method can mitigate this by sorting one sparse set to match
another's order, improving probe locality. But this sorting is explicit and
must be maintained as entities are added/removed.

### 2.3 Table storage (flecs, C)

flecs (Vos, C99/C11) uses archetype-style table storage similar to Bevy but
with a different vocabulary and some distinct design choices.

Each archetype (called a "type" in flecs, being the set of component IDs) maps
to a table. Tables store component data column-wise, one array per component
type. Entities are rows in the table.

**flecs' query system** matches tables by component sets. A query for
`(Position, Velocity)` returns all tables whose component set is a superset of
`{Position, Velocity}`. The query caches the matching table list, so subsequent
iterations do not re-evaluate which tables match.

**flecs' migration** strategy handles large tables more efficiently than
naive copy by batching: when multiple entities migrate simultaneously (e.g.,
a system adds Health to all entities in a query), flecs can batch the
migrations into a single table-to-table block copy. This amortizes the
per-entity migration cost.

**flecs and relationships:** flecs extends the archetype concept with
component pairs (relationships). An entity can have `(ChildOf, parent_entity)`
as a component. The archetype includes the pair, so entities with the same
parent are grouped in the same table. This enables cache-friendly iteration
over children of a specific parent, which is valuable for scene graph
traversal.

### 2.4 Iteration strategies compared

**Archetype iteration:** For a query matching components {A, B, C}, the ECS
finds all archetypes containing {A, B, C}. For each matching archetype, it
provides column slices for A, B, and C. The consumer iterates these slices in
lockstep. If there are K matching archetypes with N_k entities each, the total
work is sum(N_k) iterations plus K archetype lookups. When most entities share
one archetype (common in games), K is small and the iteration is nearly
optimal.

**Sparse set iteration:** For a query matching {A, B, C}, the ECS picks the
smallest sparse set (say C has the fewest entities). It iterates C's dense
array and probes A and B for each entity. The iteration cost is |C| iterations
plus |C| probes into A's sparse array and |C| probes into B's sparse array.
Each probe is a random access. If entities with C almost always also have A
and B, the probing overhead is pure waste (you always find a match). If
entities with C rarely have A and B, the early rejection saves work.

**Hybrid approaches:** Some ECS implementations use archetype storage for
common component combinations and sparse sets for rare or frequently-mutated
components. Bevy ECS supports this explicitly: component storage can be
declared as `Table` (archetype) or `SparseSet` per component type. Table
storage is the default; SparseSet storage is opt-in for components that are
frequently added/removed. The scheduler and query system handle both
transparently.

## 3. Rust-Specific SoA Approaches

### 3.1 The soa_derive crate

`soa_derive` (Lumol, available on crates.io) is a proc macro that transforms
an AoS (array-of-structs) definition into an SoA (struct-of-arrays) layout.

```rust
#[derive(StructOfArray)]
struct Particle {
    x: f32,
    y: f32,
    mass: f32,
}
```

This generates `ParticleVec`, a struct containing three `Vec`s:

```rust
struct ParticleVec {
    x: Vec<f32>,
    y: Vec<f32>,
    mass: Vec<f32>,
}
```

The generated type provides `push`, `pop`, `len`, `get`, and iteration methods
that mirror `Vec<Particle>` semantics but operate on the SoA layout. Iteration
yields `ParticleRef { x: &f32, y: &f32, mass: &f32 }` structs (generated
reference types).

`soa_derive` is simple and effective for its purpose: converting a struct into
SoA layout with zero boilerplate. It does not provide ECS-style entity
management, component queries, or dependency tracking. It is a data layout
tool, not a framework. The generated `ParticleVec` is a replacement for
`Vec<Particle>`, not a replacement for an ECS World.

The relevance to hilavitkutin: `soa_derive` demonstrates that Rust's proc
macros can generate SoA types from struct definitions with full type safety.
hilavitkutin's schema-as-traits approach (where each domain defines its IR
as traits with associated column types) is a more general version of the same
idea, but driven by trait definitions rather than struct definitions.

### 3.2 Bevy ECS internals in detail

Bevy's storage layer (bevy_ecs crate, `storage` module) provides three
abstractions layered on top of each other.

**BlobVec:** A type-erased dynamically-sized array. Stores `u8` bytes with a
known `Layout` (size and alignment). Manages allocation, reallocation, and
drop (via a stored drop function pointer). BlobVec is the raw building block:
it knows nothing about component types, entities, or ECS semantics.

**Column:** Wraps a BlobVec for the component data plus a separate Vec for
change detection ticks (added_tick and changed_tick per row). Column provides
the component-level storage API: `initialize`, `replace`, `swap_remove_and_drop`,
`get_data_ptr`. It still does not know about entities or archetypes.

**Table:** A collection of Columns (one per component type in an archetype)
plus entity metadata. Table manages rows: allocating a new row, swap-removing
a row, accessing a Column by ComponentId. Table is the archetype's data store.

**ComponentSparseSet:** An alternative to Table storage for components declared
with `StorageType::SparseSet`. Uses a two-level sparse array (pages of 256
entries) mapping EntityId to dense index, plus a BlobVec for the dense data.
This is EnTT's sparse set pattern, implemented in Rust.

**Archetype:** Metadata struct (not a storage container). Contains the set of
ComponentIds, a reference to the Table storing the data, and edge information
(which archetype do you transition to if component X is added/removed). The
archetype does not own any component data directly; it indexes into a shared
pool of Tables.

### 3.3 Rust type system advantages for columnar/ECS storage

Rust's type system provides several guarantees that are valuable for typed
storage over type-erased backing stores.

**Compile-time query validation.** Bevy's `Query<&Position, &mut Velocity>`
is validated at compile time. The type parameters encode what the query reads
(Position) and what it writes (Velocity). The system parameter extraction
machinery ensures that two systems cannot simultaneously hold `&Position` and
`&mut Position` (the borrow checker catches this at the system level through
exclusive/shared access analysis). This is the Rust borrow checker applied to
ECS data access, providing data-race freedom without runtime locking.

**Generic typed accessors over type-erased storage.** BlobVec stores raw bytes.
Column provides `get_data_ptr()` returning a raw pointer. The typed access
boundary is at the Query level: `Query<&T>::get()` calls
`Column::get_data_ptr()`, casts the raw pointer to `*const T`, and returns
`&T`. This cast is sound because the Query system guarantees (via ComponentId
matching) that the Column actually contains values of type T. The type erasure
in storage is an implementation detail invisible to consumers.

**repr(C) for predictable layout.** When types are declared `repr(C)`, their
field layout is deterministic and matches the C ABI. This is critical for
three scenarios: (1) SIMD operations that require specific field offsets within
a struct, (2) FFI across the plugin boundary (hilavitkutin's `repr(C)`
descriptors and column types must have identical layout on both sides of a
cdylib boundary), and (3) memory-mapped storage where the on-disk format must
match the in-memory layout for zero-copy reads.

loimu uses `repr(transparent)` on its fixed-point types, which guarantees that
`UFixed<I, F>` has the same layout as its inner `u64`. This means a
`SlabColumn<UFixed<8, 8>>` has the same memory layout as a `SlabColumn<u64>`,
enabling unsafe but sound transmutes for bulk operations (copying, zeroing,
SIMD loading) without going through the fixed-point API.

## 4. The Singleton-vs-Collection Duality

This section addresses Q1 from the hilavitkutin topic: can columns and
resources be unified under a single `Read<T>` / `Write<T>` mechanism?

### 4.1 How databases handle global state

Database systems distinguish sharply between relation data (in tables) and
system state (connection pools, buffer managers, lock managers, transaction
logs, catalog metadata). No database stores its connection pool in a SQL table.
No database expresses "the buffer manager needs the page cache" as a
foreign key constraint.

In DuckDB, the `DatabaseInstance` holds the catalog, buffer manager, and
transaction manager as process-level singletons. Query operators receive a
`ClientContext` reference that provides access to these singletons. The
dependency is implicit: every operator can access the buffer manager, because
the ClientContext is threaded through every operator invocation. There is no
dependency graph tracking "operator X reads the buffer manager." The buffer
manager is ambient infrastructure, not a tracked dependency.

PostgreSQL is similar: `BackendGlobals` holds session state, the shared memory
segment holds the buffer pool and lock table, and every backend process has
implicit access to both. The shared memory structures use their own
concurrency control (LWLocks, spinlocks, atomic operations) independent of
the transaction/MVCC system that protects table data.

The database pattern is: **table data is tracked, global state is ambient.**
This works because the global state has a fundamentally different access
pattern: it is always available, never partitioned, never iterated in bulk.
Nobody runs a "scan the connection pool" query.

### 4.2 How ECS handles singletons

Bevy ECS distinguishes between components (stored in archetype Tables, one
value per entity) and resources (stored separately, one value per type).

Resources are accessed via `Res<T>` (shared reference) and `ResMut<T>`
(exclusive reference). The system scheduler tracks resource access with the
same mechanism it uses for component access: if system A has `ResMut<Config>`
and system B has `Res<Config>`, the scheduler orders them (A before B or B
before A, depending on other constraints). The dependency tracking is unified
even though the storage is not.

Internally, resources are stored in a `Resources` struct (a sparse array
indexed by `ResourceId`, each entry being a `ResourceData` that wraps a
`Column` with a single row). This is already close to loimu's model: a
resource is a Column with one element.

The critical insight from Bevy: **the scheduler does not need to know whether
a dependency target is a single value or a collection.** The scheduler tracks
`Read(ComponentId)` and `Write(ComponentId)` edges. Whether ComponentId refers
to a component type (many values, one per entity) or a resource type (one
value total) does not affect the scheduling algorithm. A write-read edge is a
write-read edge regardless of the data's cardinality.

### 4.3 loimu's unified approach

loimu goes further than Bevy. In loimu, resources and columns are both backed
by `SlabColumn`. A resource is a `SlabColumn` with capacity for one element.
A collection column is a `SlabColumn` with capacity for many elements. The
scheduler tracks `Read<T>` and `Write<T>` dependencies uniformly. It does not
distinguish between "reading a resource" and "reading a column." The dependency
graph has edges, and edges have resource IDs; the row count behind each
resource ID is irrelevant to the scheduler.

This unification simplifies the scheduler. There is one dependency tracking
mechanism, one conflict detection algorithm, one topological sort. The
scheduler does not have separate code paths for "resource conflicts" and
"column conflicts."

The access API does differ. Accessing a resource is a single-value read:
`resource::<T>()` returns `&T`. Accessing a collection is morsel-based:
`column::<T>()` returns a slice `&[T]` sized to the current morsel. But this
difference is in the work unit's `execute()` method, not in the scheduler. The
scheduler sees `Read<T>` either way.

### 4.4 The case for unification

Saalis's `type Ctx` request (from the hilavitkutin topic) becomes unnecessary
under the unified model. Saalis wants work units to access HTTP clients, DB
connection pools, and configuration objects. In a split model, these would be
"context" parameters separate from the column dependency system. In the unified
model, they are resources: `Read<HttpClient>`, `Read<DbPool>`, `Read<Config>`.
The scheduler tracks them like any other dependency. If two work units both
write to the DB pool's internal state, the scheduler detects the conflict and
serializes them. No special-case "context" mechanism needed.

The unified model also simplifies the mental model for work unit authors.
There is one way to declare dependencies: `Read<T>` for shared access,
`Write<T>` for exclusive access. Whether T is a singleton or a collection
is T's concern, not the declaration's.

### 4.5 The case against unification

Singleton access patterns differ from collection access patterns in ways that
matter for performance:

- **No morsel chunking for singletons.** A resource is one value. There is no
  point in morsel-splitting it. If the scheduler treats resources and columns
  identically, it must either special-case "morsel size = 1" for resources or
  ignore the distinction and waste scheduling overhead.
- **Different cache behavior.** A resource (say, a Config struct of 256 bytes)
  fits in a single cache line. A column of 100,000 Position values spans
  thousands of cache lines. Prefetching strategies differ. Memory access
  patterns differ. The scheduler could, in theory, use this information to
  co-schedule work units that access the same resource (cache-friendly) or
  separate work units that access large columns (reduce contention).
- **Different contention patterns.** Many work units might read the same
  resource (Config), creating a many-reader pattern. Few work units typically
  write the same column simultaneously (columns are usually written by one
  producer). The read-write ratio skew means different concurrency strategies
  might be optimal.

However, these arguments are about performance optimization, not
correctness. The unified model is correct regardless of cardinality. The
optimizations (skip morsel dispatch for singletons, co-schedule
resource readers) can be layered on top without breaking the unified
dependency model. The scheduler can check `if cardinality == 1 { skip morsel
dispatch }` without needing a fundamentally different dependency tracking
mechanism.

### 4.6 Evidence from other systems

**Legion ECS (Rust, amethyst project):** Separated resources from components.
Resources were stored in a `Resources` struct outside the World. Systems
declared resource access separately from component queries. This worked but
created two parallel dependency mechanisms in the scheduler.

**Specs ECS (Rust, amethyst project):** Also separated resources from
components. `ReadStorage<T>` for components, `Read<T>` for resources. The
scheduler (shred) tracked both, but through different type-system paths.

**Unity DOTS (C#):** Uses the term "singleton component" for a component type
that exists on exactly one entity. There is no separate resource concept; a
singleton is just a component with one instance. The query system handles it
normally. This is the fully unified approach: even singletons are components in
the archetype table. The overhead is minimal because the archetype has one
entity.

**Shipyard (Rust ECS):** Uses sparse set storage. All data (including what
other ECSs call "resources") is stored as components. A "resource" is a
component on a unique entity. The scheduler sees no distinction. This is the
same philosophy as Unity DOTS.

The trend across ECS implementations is toward unification. Earlier systems
(Specs, Legion) separated resources from components. Later systems (Bevy,
Unity DOTS, Shipyard) unify them or nearly unify them. The separation created
complexity without a commensurate benefit.

## 5. Memory-Mapped and Arena-Based Storage

### 5.1 mmap for persistent columnar data

loimu's SlabColumn uses `mmap` with `PROT_NONE` for the initial reservation,
then `mprotect` to commit pages on demand. This gives three properties:

1. **No realloc.** The virtual address range is stable. Pointers to slot 0
   remain valid as the column grows. This eliminates the reallocation hazard
   of `Vec`: growing a Vec may move its backing allocation, invalidating all
   raw pointers. SlabColumn's pointer stability means that raw pointers to
   column data can be safely held across growth events (within the same thread,
   respecting Rust's aliasing rules).

2. **Zero-copy growth.** Committing a new page is a kernel call that maps
   physical memory to the next page in the reserved range. No data is copied.
   Compare with `Vec::reserve()`, which allocates a new buffer, copies all
   existing data, and frees the old buffer. For a column of 1 GB, Vec growth
   copies 1 GB; SlabColumn growth commits one 4 KB page.

3. **Lazy physical allocation.** Virtual pages mapped with `PROT_NONE` consume
   no physical memory and no swap. Only committed pages use RAM. A SlabColumn
   can reserve 1 TB of virtual address space and use 4 KB of physical memory if
   only one page is committed. On 64-bit systems, virtual address space is
   effectively infinite (48-bit or 57-bit virtual addresses give 256 TB or
   128 PB).

**DuckDB's buffer manager** provides similar semantics at a higher level of
abstraction. DuckDB allocates fixed-size buffer blocks (256 KB default) and
manages them in a pool. Segments reference buffer blocks. When a block is
needed, the buffer manager either returns an in-memory block or reads it from
disk. When memory pressure is high, the buffer manager evicts blocks to disk
(writing dirty blocks, discarding clean blocks). This is demand paging
implemented in user space, giving DuckDB control over eviction policy that the
OS page cache does not provide.

**Arrow IPC (Inter-Process Communication)** supports memory-mapped reads. An
Arrow IPC file can be mmap'd, and the RecordBatch data is accessible without
copying. The IPC format stores the schema, buffer lengths, and buffer data
contiguously. The reader computes buffer pointers from the mmap'd base address
and buffer offsets. This is zero-copy in the truest sense: no deserialization,
no allocation, no copying. The drawback is that the mmap'd data must be in
Arrow format (byte-aligned, padded, with validity bitmaps), which means
bit-packed formats (like loimu's sub-word columns) would need to be unpacked
before storing in Arrow IPC.

### 5.2 Arena allocation for ephemeral data

loimu defines three storage backends: Record, Snap, and Reel. Each uses
SlabColumn as the underlying storage atom but with different lifecycle
semantics.

**Record (persistent, mmap-backed):** The default backend. Values persist
across scheduler passes. Record columns use the full mmap lifecycle: reserve,
commit, read, write. Suitable for entity data, configuration, any state that
must survive across passes.

**Snap (ephemeral, signal arena):** Values exist for one scheduler pass. Snap
columns use arena allocation: bump-allocate during the pass, bulk-clear
(pointer reset, no per-element drop) at pass boundaries. The clear is O(1)
regardless of how many values were written. Snap columns are used for per-pass
intermediate results, event buffers, and temporary computation state.

The arena model maps to database transaction-local memory pools. PostgreSQL's
MemoryContext system uses per-transaction and per-query memory contexts that
are bulk-freed at transaction/query end. The concept is the same: allocate
fast, free in bulk, never free individual allocations. The difference is that
loimu's Snap backend is typed (each Snap column stores values of one type with
known stride) while PostgreSQL's MemoryContext is a general-purpose allocator.

**Reel (pre-computed future values, FIFO):** Reel columns are ring buffers
that store predicted future values. A prediction behavior writes N future
values; a consumer behavior reads them in order. When the scheduler advances,
the oldest value is consumed and the newest is appended. Reel columns are
loimu-specific (rotation groups, temporal prediction) and unlikely to be needed
by polka-dots or saalis.

### 5.3 Mapping to hilavitkutin

The three-backend model maps to hilavitkutin's consumers as follows:

- **polka-dots:** All data is in-memory. No persistence across invocations
  (polka-dots runs once and exits). Record backend behavior with in-memory
  allocation (no mmap needed; a simple Vec or arena suffices). Snap backend
  for pipeline intermediate results.
- **saalis:** Data lives in SQLite for persistence. In-memory columns are
  materialized views of query results. Record backend for cached/hot data with
  a DB-backed eviction path (when memory is tight, evict to SQLite). Snap
  backend for per-query intermediate results.
- **loimu:** Full three-backend model. Record for persistent entity state,
  Snap for per-pass intermediates, Reel for temporal prediction.

hilavitkutin needs a storage trait that abstracts over these backends. The
trait contract: provide typed, contiguous, fixed-stride access. How the
backing memory is managed (mmap, Vec, arena, DB materialization) is behind
the trait.

## 6. Open-Source Discussions and Community Knowledge

### 6.1 Linux kernel data structures

The Linux kernel faces a version of the singleton-vs-collection problem in its
per-CPU data model. Each CPU core has its own copy of certain data structures
(per-CPU counters, per-CPU caches, per-CPU run queues). These are singletons
from the perspective of each core, but collections from the perspective of the
system. Access is via `get_cpu_var()` / `put_cpu_var()` macros that disable
preemption for the duration of access, ensuring that the current core's copy
is accessed without races.

The kernel's slab allocator (`kmem_cache`) provides typed object pools. A slab
cache is created for a specific struct type and allocates/frees objects of that
type. Objects within a slab are contiguous in memory (cache-friendly for
iteration). The slab allocator is conceptually similar to a typed column: it
provides fast allocation of same-type objects in contiguous memory. The
difference is that slab objects are individually allocated and freed (like a
pool), while column rows are bulk-managed (insert at end, swap-remove).

The kernel's radix tree (now XArray, as of kernel 5.x) maps integer keys to
pointers. The page cache uses XArray to map (inode, offset) pairs to page
pointers. XArray is a sparse data structure: it allocates tree nodes on demand
and can represent large key spaces efficiently. This is analogous to a sparse
array in ECS (mapping entity IDs to component data), though XArray is designed
for pointer storage, not typed value storage.

The kernel's approach to lockless access is relevant to hilavitkutin's
scheduling model. RCU (Read-Copy-Update) allows readers to access data without
locks. Writers create a copy, modify the copy, and atomically swap the pointer.
Readers that started before the swap continue reading the old data until they
reach a quiescent state. This is conceptually similar to double-buffered signal
arenas (loimu's Snap backend pointer swap) and to MVCC in databases (readers
see a snapshot, writers create new versions).

### 6.2 ECS performance in the Rust ecosystem

The Rust gamedev community has extensively benchmarked ECS implementations.
The key comparisons from 2023-2025 benchmarks (ecs_bench_suite, bevy's
internal benchmarks):

**Iteration performance:** Archetype-based systems (Bevy, hecs) dominate
multi-component iteration benchmarks. The lockstep column scan is hard to
beat. Sparse-set systems (Shipyard in sparse mode) are slower for
multi-component iteration due to probe overhead.

**Component add/remove performance:** Sparse-set systems dominate component
mutation benchmarks. No migration cost. Archetype systems show linear
degradation with component count per entity (more components = more data to
copy during migration).

**Memory usage:** Archetype systems have fragmentation overhead when there are
many archetypes with few entities each. Each archetype allocates its own Table
with its own column Vecs. If there are 1000 archetypes with 10 entities each,
there are 1000 Vecs per component type, each with wasted capacity. Sparse-set
systems have one Vec per component type regardless of how entities are
distributed.

**The SoA vs AoS debate in Rust gamedev:** The community consensus (from
Rust gamedev discord, Reddit, and conference talks) is that SoA layout is
correct for ECS workloads where systems iterate subsets of components. AoS
layout is correct for workloads where all fields of a struct are always
accessed together (particle systems, vertex buffers). The ECS pattern
inherently favors SoA because queries select subsets of components.

### 6.3 The data-oriented design movement

Mike Acton's CppCon 2014 talk ("Data-Oriented Design and C++") established the
principle that drives modern ECS design: "where there is one, there are many."
The argument: if your program processes one Entity, it almost certainly
processes many Entities. Design your data layout for the many case (arrays of
each field) not the one case (struct with all fields). The one case is a
degenerate many case (array of length 1), not a separate concept.

This principle directly supports the singleton-as-degenerate-collection model.
A resource is a column with one row. A configuration singleton is a column with
one row. The data layout and access machinery are the same; the row count
differs. Acton's argument is that designing separate mechanisms for "one" and
"many" is a premature optimization that adds complexity without improving
performance, because the "one" case is rare enough that its performance is
irrelevant.

Andrew Kelley (Zig language creator) has discussed similar principles in the
context of Zig's standard library data structures. Zig's `MultiArrayList`
(formerly `StructOfArrays`) provides SoA layout for any struct type,
similar to Rust's `soa_derive`. Kelley argues that SoA should be the default
for all bulk data, with AoS reserved for individual instances. Zig's hash
maps use SoA layout internally: keys, values, and metadata are stored in
separate arrays. This reduces cache pollution when only keys are being probed
(the values array stays cold).

The Data-Oriented Design wiki (dataorienteddesign.com) catalogs patterns for
transforming OOP hierarchies into flat, cache-friendly data layouts. The
recurring theme: replace inheritance with composition (ECS components), replace
polymorphism with data-driven dispatch (type IDs and function pointer tables),
replace pointer-linked structures with flat arrays indexed by integer IDs.
These patterns converge on the same storage model: typed, contiguous, fixed-
stride arrays indexed by integer keys, which is exactly what a columnar store
provides.

## 7. Relevance to hilavitkutin

### 7.1 The unified store is the right answer for the scheduler

The evidence from ECS evolution (Specs/Legion separating resources, Bevy/Unity
DOTS/Shipyard unifying them) and from loimu's design (resources and columns
both backed by SlabColumn, scheduler tracks dependencies uniformly) converges
on one conclusion: unify.

The scheduler does not need to distinguish between singletons and collections.
An edge in the dependency graph says "work unit A writes T, work unit B reads
T, therefore A must complete before B starts." Whether T has one row or one
million rows does not change the scheduling constraint. The scheduler
implements topological ordering over read/write edges. The data behind each
edge is opaque to the scheduler.

This is the same conclusion that the database world reached for a different
reason: the query optimizer does not distinguish between "table with one row"
and "table with one million rows" for correctness. It distinguishes them for
cost estimation (cardinality affects join order selection), but the algebraic
rules (commutativity, associativity of joins) apply regardless of cardinality.
Correctness is cardinality-independent; optimization is cardinality-aware.

hilavitkutin should follow the same principle: the dependency tracking is
cardinality-independent (one mechanism for both singletons and collections),
but the dispatch strategy can be cardinality-aware (skip morsel chunking for
singletons).

### 7.2 The access API can differ

While the scheduler unifies singletons and collections, the work unit author
needs different APIs for different access patterns.

`.resource::<T>()` returns `&T` (or `&mut T` for `Write<T>`). This is a
single-value access. No slicing, no morsel offset, no iteration. The work unit
reads or writes one value.

`.column::<T>()` returns `&[T]` (or `&mut [T]` for `Write<T>`). This is a
morsel-sized slice. The work unit processes the slice in a loop. The scheduler
dispatches multiple morsels to multiple threads.

Both methods generate the same dependency declaration: `Read<T>` or `Write<T>`.
The scheduler sees the same edge regardless of which method the work unit calls.
The difference is ergonomic, not semantic. A resource is a column that happens
to have one row, and `.resource::<T>()` is sugar for `.column::<T>()[0]`.

### 7.3 The storage backend varies by consumer

The three hilavitkutin consumers have different persistence requirements:

- **polka-dots:** In-memory only. No persistence across invocations. Simple
  `Vec`-backed columns. Arena-backed intermediates for pipeline stages.
- **saalis:** SQLite-backed persistence. In-memory columns are materialized
  views of SQL query results. The storage trait must support "materialize from
  DB, operate in-memory, flush back to DB."
- **loimu:** mmap-backed persistent columns (Record), arena-backed ephemeral
  columns (Snap), ring-buffered prediction columns (Reel).

The storage trait abstracts over these backends. The trait provides:
- `fn as_slice(&self) -> &[T]` for typed, contiguous read access.
- `fn as_mut_slice(&mut self) -> &mut [T]` for typed, contiguous write access.
- `fn len(&self) -> usize` for cardinality.

How the backing memory is allocated, persisted, and evicted is behind the
trait. The scheduler and work units see typed slices. The backend manages
memory.

### 7.4 The common abstraction

Synthesizing across database columnar formats, ECS storage patterns, and the
three hilavitkutin consumers, the common abstraction is:

**Typed, contiguous, fixed-stride storage with read/write access tracking.**

- **Typed:** Each column stores values of one type. The type is known at
  compile time (generic parameter) or at registration time (ComponentId
  mapped to a Layout). Type safety is enforced at the access API boundary.
- **Contiguous:** Values are stored in a contiguous memory region. No linked
  lists, no pointer indirection, no hash table overhead. Iteration is a
  linear memory scan. SIMD operations can load consecutive values without
  gather instructions.
- **Fixed-stride:** Each value occupies `size_of::<T>()` bytes. Value at index
  `i` is at offset `i * stride`. O(1) random access. No variable-length
  encoding (variable-length data uses offsets into a separate data buffer,
  like Arrow's Utf8 format).
- **Read/write access tracking:** Every access is declared as `Read<T>` or
  `Write<T>`. The scheduler builds a dependency graph from these declarations
  and enforces ordering constraints. No ambient access; every dependency is
  explicit and trackable.

Everything else is consumer-specific:

- Bit-packing (loimu): Consumer-specific storage optimization. The common
  abstraction provides byte-aligned typed slices; loimu's SlabColumn unpacks
  sub-word values to byte-aligned form before presenting them through the
  common API.
- Compression (DuckDB): Consumer-specific. Compressed segments are
  decompressed into vectors before processing. The operator sees uncompressed
  vectors.
- Persistence (saalis/loimu): Consumer-specific. Materialization from DB,
  mmap-backed files, or arena allocation are behind the storage trait.
- Change detection (Bevy): Consumer-specific. Bevy's Column stores ticks
  alongside data for change detection. hilavitkutin could add version stamps
  (loimu's approach) as an optional layer.

The hilavitkutin storage trait is the intersection of what Arrow, DuckDB, Bevy,
EnTT, flecs, and loimu all provide: a typed slice of contiguous, fixed-stride
values with a known length. The rest is policy.
