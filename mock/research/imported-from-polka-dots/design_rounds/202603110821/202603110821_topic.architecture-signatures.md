**Date:** 2026-03-11

## Context

polka-dots is an opinionated dotfiles build tool. The February 2026 round proposed
decomposing the monolithic codebase into 13 crates across 4 layers. That round was
archived without implementation â€” the decomposition was sound for an internal
refactor, but the architecture has since evolved to follow the same plugin-first
pattern used in saalis and loimu.

## Architectural vision

polka-dots has two fundamental responsibilities:

1. **File-backed data store** â€” hierarchical structured data (settings, catalogue
   entries, loadouts) resolved through a precedence chain. Files are the database.
   Plugins define their own data types through SDK traits. The host resolves values
   through file inheritance, loadout overrides, platform conditionals.

2. **Plugin-agnostic pipeline engine** â€” builds, templates, generates, syncs,
   deploys, schedules. The pipeline doesn't know what data flows through it or what
   the concrete steps do. It knows trait contracts. The `.polka` DSL is just one
   language plugin. Brew is just one source plugin. TOML is just one format plugin.

## Design principles

### Type taxonomy (from saalis Â§10.1)

Every type in the SDK falls into one of four categories:

| Category | When | polka examples |
|----------|------|---------------|
| **Trait** | Varying behavior, shared contract, extensible | `Language`, `Source`, `Transform`, `Platform`, `DataValue`, `PipelineStage` |
| **Newtype** | Fixed representation, validation at construction | `CataloguePath(PathBuf)`, `Hash(String)` |
| **`repr(C)` enum** | Closed set, crosses ABI boundary | `SettingsScope`, `FieldKind`, `ErrorClass` |
| **Primitive** | No semantic meaning beyond the raw value | `bool`, `usize`, `String` (free-form only) |

**The test:** "Can I imagine two implementations with different behavior?"
If yes â†’ trait.

**The plugin boundary rule:** Fixed `repr(C)` enums are for descriptor fields
that describe static properties. They are NOT for domain concepts that plugins
might extend. Extensible domain concepts are trait impls + `#[register]`, not
enum variants.

If you're asking "should this enum have room for plugin extensions?" â€” the
answer is: it shouldn't be an enum. It should be a trait.

### Plugin registration

Same pattern as saalis/loimu:

- `inventory::submit!` of `repr(C)` descriptors â€” NOT `dyn Trait` vtables
- `#[register]` macro on impl blocks, generates descriptors + `inventory::submit!`
- `Named` + `Registrable` base traits with blanket impl
- `Descriptor` trait for common metadata (name, SDK version)
- `Registry` trait for typed discovery with built-in validation

First-party and plugin code use the same registration path. No special cases.

### Trait roles

Traits serve two distinct roles (both valid, used as appropriate):

1. **Marker traits** â€” compile-time categorization. ZSTs, no runtime cost.
   Used for type-level filtering and validation.
2. **Interface traits** â€” runtime behavior. Real methods, real dispatch.
   Used where operations need to be polymorphic.

Not a strict rule â€” use the right kind for the right job.

### The plugin boundary paradigm

Plugins define trait implementations. The host runs them. This resolves all
boundary extensibility questions:

- Plugin wants a new output format â†’ impls `Transform` trait, registers it
- Plugin wants a new source type â†’ impls `Source` trait, registers it
- Plugin wants a new platform â†’ impls `Platform` trait, registers it
- Plugin wants a new language â†’ impls `Language` trait, registers it
- Plugin wants custom settings â†’ `#[derive(Settings)]` with `SettingsValue` types

## Crate decomposition

### SDK crates (traits only â€” what extensions depend on)

**`polka-primitives`**
`repr(C)` atoms, newtypes, `Named`, `Registrable`, `Descriptor`, `Registry`
base trait. `no_std + alloc`. Zero concrete implementations.

**`polka-derive`**
Proc macros: `#[register]`, `#[derive(Settings)]`.

**`polka-sdk`**
Re-exports primitives + derive. Adds domain trait contracts:
- `Language` â€” parse config source, produce output (what `.polka`, bash, etc. implement)
- `Source` â€” provide apps/packages from a source (what brew, mas, etc. implement)
- `Transform` â€” convert data to an output format (what toml, json, ini implement)
- `Platform` â€” platform detection + platform-specific behavior (extensible, not closed)
- `PipelineStage` â€” a step in the build/deploy pipeline
- `DataValue` â€” typed data access (not an enum â€” extensible)
- `SettingsValue` â€” storable/resolvable setting types
- `SettingsDomain` â€” access control categories for settings

### Core crates: Data

**`polka-data`**
Data store implementation. File-backed settings resolution through precedence
chain. Typed access, merge semantics, file glob resolution. The host side of
`DataValue` and `SettingsValue` contracts.

**`polka-catalogue`**
App/package/service catalogue. Loads the TOML registry files (what was
previously called "registry" â€” renamed to avoid collision with the inventory-
based `Registry` trait from SDK primitives). Loading, validation, querying.

**`polka-project`**
Project loading. Reads polka.toml, populates data store, coordinates data
crates. The single entry point for project initialization.

### Core crates: Execution

**`polka-pipeline`**
Pipeline orchestration. Build, stage, deploy through trait contracts.
Manifest tracking, staleness detection, hash comparison. Calls registered
plugins through SDK traits, never knows concrete implementations.

**`polka-scheduler`**
Task scheduling. Routine management, cadence timers. Pure scheduling
logic â€” daemon I/O lives in the CLI layer.

**`polka-audit`**
System audit. Compares installed state against catalogue: what's present,
missing, outdated, orphaned. Returns structured audit results.

### CLI

**`polka-cli`**
clap shell, I/O execution, output rendering. The only crate that touches
stdout/stderr/filesystem for user-facing operations. Thin wrappers that call
core crates and render results.

### First-party extensions (each a dylib)

**`polka-lang-polka`** â€” the `.polka` DSL: pest parser, AST, expander, builtins.
Just one language plugin â€” not special, follows the same `Language` trait contract.

**`polka-source-brew`** â€” Homebrew/Cask/Tap source provider.

**`polka-source-mas`** â€” Mac App Store source provider.

**`polka-format-*`** â€” Output transforms: toml, json, ini, kdl, etc.

**`polka-platform-macos`** â€” macOS platform detection + platform-specific behavior.

**`polka-platform-linux`** â€” Linux platform detection + distro-specific behavior.

**`polka-keybinds`** â€” Keybind resolution with inventory-registered formats.

## What the original 9 problems become

| # | Original problem | Resolution |
|---|-----------------|-----------|
| 1 | Zero traits | Everything is a trait. SDK exports only traits. |
| 2 | `toml::Value` leaked | `DataValue` trait in SDK. No concrete format types cross the boundary. |
| 3 | Schema types are data bags | Schema concepts are traits with behavioral methods. |
| 4 | I/O in business logic | Host (core) owns I/O. Plugins return data through SDK traits. |
| 5 | `&Ui` bleeding | Doesn't exist in SDK or core. CLI-only concern. |
| 6 | Bypassing `load()` | Data store is single source. SDK trait access only. |
| 7 | Cross-command coupling | Pipeline engine orchestrates. Commands are thin CLI entries. |
| 8 | Stringly-typed dispatch | Type taxonomy: trait / newtype / repr(C) enum / primitive. |
| 9 | Scattered platform code | Platform is an extensible trait, registered by plugins. |

## What this does NOT change

- polka.toml as project entry point
- The user-facing CLI commands
- File-based configuration (TOML files are the database)
- The concept of loadouts and catalogue entries

## What this DOES change from the February design

- 13 internal crates â†’ 10 core crates + 6 first-party extensions (16 total in workspace)
- Everything-concrete â†’ everything-trait (SDK boundary)
- `.polka` DSL is special â†’ `.polka` DSL is just a language plugin
- Brew/Cask/Mas are built-in â†’ they're source plugins
- Platform is an enum â†’ Platform is an extensible trait
- `registry/` (ambiguous) â†’ `catalogue/` (distinct from SDK `Registry`)
- Internal refactor â†’ plugin-extensible architecture
