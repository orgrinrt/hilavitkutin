# Src Changelist: Architecture Signatures

**Round:** 202603110821
**Phase:** SRC

Replace all nuked stubs with real Rust trait/struct/enum signatures that compile.
Implementation order is bottom-up through the dependency graph.

## 1. polka-primitives (Layer 0, no deps)

- [ ] `Named` trait — `const NAME: &'static str`
- [ ] `Registrable` trait — supertrait: `Named + Display + Send + Sync + 'static`, blanket impl
- [ ] `Descriptor` trait — common metadata for `repr(C)` descriptors (name, sdk_version)
- [ ] `Registry` trait — typed discovery with validation
- [ ] `SdkVersion` newtype — ABI version type
- [ ] `repr(C)` enums: `ErrorClass`, `SettingsScope`, `FieldKind`
- [ ] Newtypes: `Hash(String)`, `CataloguePath(PathBuf)`

## 2. polka-derive (proc-macro, depends on primitives)

- [ ] `#[register]` attribute macro — stub that compiles (expand later)
- [ ] `#[derive(Settings)]` derive macro — stub that compiles (expand later)

## 3. polka-sdk (depends on primitives + derive)

- [ ] Re-export polka-primitives and polka-derive
- [ ] `Language` trait — parse config source, produce output
- [ ] `Source` trait — provide apps/packages from a source
- [ ] `Transform` trait — convert data to output format
- [ ] `Platform` trait — platform detection + behavior
- [ ] `PipelineStage` trait — a step in the build/deploy pipeline
- [ ] `DataValue` trait — typed data access
- [ ] `SettingsValue` trait — storable/resolvable setting types
- [ ] `SettingsDomain` trait — access control categories

## 4. polka-data (Core:Data, depends on SDK)

- [ ] `DataStore` struct — file-backed data store
- [ ] Settings resolution — precedence chain walker
- [ ] Merge semantics — layer overrides onto base
- [ ] File glob resolution types

## 5. polka-catalogue (Core:Data, depends on SDK)

- [ ] `Catalogue` struct — loaded catalogue with indexed entries
- [ ] `CatalogueEntry` struct — single app/package/service definition
- [ ] `load()` — load catalogue from TOML registry files
- [ ] Query methods: `get()`, `active_apps()`, `by_source()`, `services()`

## 6. polka-project (Core:Data, depends on SDK + data + catalogue)

- [ ] `LoadedProject` struct — fully resolved project state
- [ ] `load()` — THE single entry point for project initialization

## 7. polka-pipeline (Core:Execution, depends on SDK + project)

- [ ] `PipelineResult` — aggregate result type
- [ ] `BuildManifest` / `DeployManifest` — tracking types
- [ ] `run()` — orchestrate registered pipeline stages

## 8. polka-scheduler (Core:Execution, depends on SDK)

- [ ] `Schedule` — parsed schedule definition
- [ ] `Routine` — named task with schedule
- [ ] `SchedulerState` — persistent state
- [ ] `due_routines()` — compute which routines are due

## 9. polka-audit (Core:Execution, depends on SDK + catalogue)

- [ ] `AuditResult` — aggregate audit findings
- [ ] `AppAudit` — per-app status
- [ ] `ServiceAudit` — per-service status

## 10. polka-cli (Layer 3, depends on all core)

- [ ] `Cli` struct — clap derive stub
- [ ] Placeholder command enum

## 11. First-party extensions (cdylib, depend on SDK only)
