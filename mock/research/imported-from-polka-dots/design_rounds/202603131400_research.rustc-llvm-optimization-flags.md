## 1. RUSTFLAGS delivery mechanism

Four mutually exclusive sources, checked in this order (first match wins):

| Priority | Source | Syntax |
|----------|--------|--------|
| 1 (highest) | `CARGO_ENCODED_RUSTFLAGS` env | `0x1f`-separated flags. Used when standard escaping is unreliable. |
| 2 | `RUSTFLAGS` env | Space-separated string or array. |
| 3 | `target.<triple>.rustflags` + `target.<cfg>.rustflags` in `.cargo/config.toml` | All matching entries are joined together (not first-wins). |
| 4 (lowest) | `build.rustflags` in `.cargo/config.toml` | Global default. |

Key behaviors:

- Sources 1-4 are **mutually exclusive**: if `RUSTFLAGS` is set, config.toml
  `build.rustflags` is ignored entirely.
- When `--target` is passed to cargo, rustflags only apply to the target
  compiler invocation. Build scripts and proc macros do NOT receive them.
- Without `--target`, all compiler invocations receive the flags.
- Target-specific env override: `CARGO_TARGET_<TRIPLE>_RUSTFLAGS` (triple in
  uppercase, hyphens become underscores).

Config file examples:

```toml
# .cargo/config.toml

[build]
rustflags = ["-C", "target-cpu=native", "-C", "opt-level=3"]

[target.x86_64-unknown-linux-gnu]
rustflags = ["-C", "target-feature=+avx2,+fma"]

[target.'cfg(target_arch = "aarch64")']
rustflags = ["-C", "target-feature=+neon,+dotprod"]
```

## 2. How -C llvm-args works

`-C llvm-args=<string>` passes arguments directly to the LLVM backend.

### Syntax rules

- Space-separated within a single flag: `-C llvm-args="--flag1 --flag2=val"`
- Multiple `-C llvm-args` flags: YES, they accumulate.
  `-C llvm-args="--inline-threshold=300" -C llvm-args="--force-vector-width=4"` works.
- Comma separation: NO. Use spaces or multiple flags.
- In RUSTFLAGS env: `RUSTFLAGS='-C llvm-args="--inline-threshold=300 --force-vector-width=4"'`

### Discovering available flags

```bash
rustc -C llvm-args="--help"         # public flags
rustc -C llvm-args="--help-hidden"  # all flags (3000+)
```

### Stability

LLVM args are NOT subject to Rust's stability guarantees. They can change
between LLVM versions (and therefore between rustc versions). The LLVM
version bundled with rustc changes roughly every 6-8 weeks.

## 3. Cargo profile settings

These are set in `Cargo.toml` under `[profile.release]` (or custom profiles).

| Setting | Default (release) | Recommended | Notes |
|---------|-------------------|-------------|-------|
| `opt-level` | `3` | `3` | `s`/`z` reduce binary size but disable loop vectorization |
| `lto` | `false` (thin-local) | `"thin"` or `true` | See LTO section |
| `codegen-units` | `16` | `1` | Fewer = better optimization, slower compile |
| `panic` | `"unwind"` | `"abort"` | Saves binary size and unwind overhead |
| `strip` | `"none"` | `"symbols"` for prod | Smaller binary |
| `debug` | `false` | `1` or `"line-tables-only"` | Needed for profiling |
| `debug-assertions` | `false` | `false` | Keep off in release |
| `overflow-checks` | `false` | `false` | Keep off in release |
| `incremental` | `false` | `false` | Inhibits optimization |

Custom profile example:

```toml
[profile.release-perf]
inherits = "release"
opt-level = 3
lto = "thin"
codegen-units = 1
panic = "abort"
strip = "symbols"
debug = "line-tables-only"  # enables profiling
```

Per-crate overrides:

```toml
# Optimize all dependencies even in dev builds
[profile.dev.package."*"]
opt-level = 2

# Critical hot-path crate at max optimization
[profile.release.package.polka-pipeline]
opt-level = 3
```

|------|-------------|--------------|----------------|-----------------|-----------|
| `inline-threshold` (deprecated) | `-C inline-threshold=N` | Does nothing since ~2020. Use LLVM arg instead. | N/A | N/A | Stable (no-op) |

### 4.2 Rustc -Z flags (nightly)

| Flag | How to pass | What it does | Compile impact | Runtime benefit |
|------|-------------|--------------|----------------|-----------------|
| `inline-llvm` | `-Z inline-llvm=yes` | Enable/disable LLVM inlining pass entirely. Default: yes. | N/A | Disabling would be catastrophic |

| `inline-mir-hint-threshold` | `-Z inline-mir-hint-threshold=100` | Threshold for `#[inline]`-hinted functions in MIR. Default: 100. | Moderate | Higher means more `#[inline]` functions get inlined at MIR level |

| `cross-crate-inline-threshold` | `-Z cross-crate-inline-threshold=N` | Controls which functions are eligible for cross-crate inlining. | Moderate | Important for library code with many small functions |

## 5. Vectorization

### 5.1 Rustc stable flags

| Flag | How to pass | What it does | Compile impact | Runtime benefit | Notes |
|------|-------------|--------------|----------------|-----------------|-------|
| `no-vectorize-loops` | `-C no-vectorize-loops` | Disable the LLVM loop vectorizer entirely. | Faster compile | Catastrophic for data processing | Never use for data workloads |
| `no-vectorize-slp` | `-C no-vectorize-slp` | Disable SLP (superword-level parallelism) vectorizer. | Slightly faster compile | Moderate loss | Rarely useful to disable |
| `target-cpu=native` | `-C target-cpu=native` | Generate code for the host CPU. Enables all available SIMD. | Negligible | Significant. Unlocks AVX2/AVX-512/NEON/SVE. | Not portable across machines |
| `target-feature` | `-C target-feature=+avx2,+fma` | Enable specific SIMD instruction sets. | Negligible | Significant for specific codepaths | Comma-separated, `+` to enable |

### 5.2 Target features for data workloads

#### x86_64

| Feature | What it enables | Impact |
|---------|-----------------|--------|
| `+sse4.2` | 128-bit SIMD, CRC32, string ops | Baseline for modern x86 |
| `+avx` | 256-bit float SIMD | 2x float throughput |
| `+avx2` | 256-bit integer SIMD | Critical for columnar integer operations |
| `+fma` | Fused multiply-add | Better float precision and throughput |
| `+avx512f` | 512-bit SIMD foundation | 2x over AVX2 but may cause frequency throttling |
| `+avx512bw` | 512-bit byte/word ops | Text/string processing |
| `+avx512vl` | AVX-512 on 128/256-bit | Most useful AVX-512 extension |
| `+avx512vbmi` | Byte-level permute | Fast shuffles for string ops |
| `+bmi1,+bmi2` | Bit manipulation | Fast popcount, pdep, pext |
| `+popcnt` | Population count | Bitmap operations |
| `+lzcnt` | Leading zero count | Division, log2 |
| `+aes` | AES acceleration | Hashing |

#### aarch64

| Feature | What it enables | Impact |
|---------|-----------------|--------|
| `+neon` | 128-bit SIMD (always on for aarch64) | Baseline |
| `+dotprod` | Dot product instructions | Aggregation operations |
| `+i8mm` | Int8 matrix multiply | ML-style reductions |
| `+sve` | Scalable Vector Extension (128-2048 bit) | Hardware-length-agnostic vectorization |
| `+sve2` | SVE2 (extended ops) | More operation coverage |
| `+fp16` | Half-precision float | Memory bandwidth savings |
| `+bf16` | BFloat16 | ML workloads |
| `+crc` | CRC32 | Checksums/hashing |
| `+aes` | AES acceleration | Hashing |
| `+lse` | Large system extensions (atomics) | Better atomic performance |
| `+mops` | memcpy/memset/memmove acceleration | Faster bulk memory ops |

## 14. PGO-related

### 14.1 Standard PGO workflow

**Step 1: Instrument**
```bash
RUSTFLAGS="-Cprofile-generate=/tmp/pgo-data" \
  cargo build --release --target=x86_64-unknown-linux-gnu
```

**Step 2: Train**
```bash
./target/x86_64-unknown-linux-gnu/release/myprogram < typical_workload.dat
./target/x86_64-unknown-linux-gnu/release/myprogram < another_workload.dat
```

**Step 3: Merge**
```bash
llvm-profdata merge -o /tmp/pgo-data/merged.profdata /tmp/pgo-data
```

**Step 4: Optimize**
```bash
RUSTFLAGS="-Cprofile-use=/tmp/pgo-data/merged.profdata" \
  cargo build --release --target=x86_64-unknown-linux-gnu
```

### 14.2 PGO flags

| `profile-use=<path>` | `-C profile-use=/path/merged.profdata` | Use merged profile data for optimization. |

| `--enable-npm-pgo-inline-deferral` | `-C llvm-args=--enable-npm-pgo-inline-deferral` | Defer inlining decisions during PGO. |
| `--enable-post-pgo-loop-rotation` | `-C llvm-args=--enable-post-pgo-loop-rotation` | Run loop rotation after PGO instrumentation. |
| `--preinline-threshold=N` | `-C llvm-args=--preinline-threshold=N` | Inlining threshold during pre-instrumentation pass. Default: 75. |

### 14.4 BOLT (post-link optimization)

BOLT is a post-link optimizer from LLVM that reorders machine code using profile data.
Not directly integrated into rustc, but can be applied to compiled Rust binaries.

Workflow:
```bash
# 1. Build with labels for BOLT
RUSTFLAGS="-C link-arg=-Wl,--emit-relocs" cargo build --release

# 2. Instrument
llvm-bolt target/release/mybinary -instrument -o mybinary.instr

# 3. Collect profile
./mybinary.instr < workload.dat
# produces /tmp/prof.fdata

# 4. Optimize
llvm-bolt target/release/mybinary -o mybinary.bolt \
  -data=/tmp/prof.fdata \
  -reorder-blocks=ext-tsp \
  -reorder-functions=cdsort \
  -split-functions \
  -split-all-cold \
  -split-eh \
  -dyno-stats
```

BOLT can provide 5-15% additional improvement on top of PGO + LTO.

### 14.5 Profile format

| File type | Description |
|-----------|-------------|
| `.profraw` | Raw profile data from instrumented run. One per run. |
| `.profdata` | Merged profile. Created by `llvm-profdata merge`. |
| `.prof` | Sample profile for AutoFDO. Created by `perf` + `create_llvm_prof`. |
| `.fdata` | BOLT profile. Created by instrumented binary. |

| `--do-counter-promotion` | `-C llvm-args=--do-counter-promotion` | Promote counters out of loops. |
| `--iterative-counter-promotion` | `-C llvm-args=--iterative-counter-promotion` | Allow counter promotion across entire loop nest. |
| `--max-counter-promotions-per-loop=N` | `-C llvm-args=--max-counter-promotions-per-loop=N` | Cap promotions per loop to limit register pressure. |

## 15. Floating-point optimization

| Flag | How to pass | What it does | Gotchas |
|------|-------------|--------------|---------|
| `--fp-contract=fast` | `-C llvm-args=--fp-contract=fast` | Aggressively form fused FP ops (FMA). | Changes results vs strict IEEE |
| `--fp-contract=on` | `-C llvm-args=--fp-contract=on` | Only fuse "blessed" FP ops. | - |
| `--enable-no-nans-fp-math` | `-C llvm-args=--enable-no-nans-fp-math` | Assume no NaN values. Enables optimizations. | UB if NaN actually occurs |
| `--enable-no-infs-fp-math` | `-C llvm-args=--enable-no-infs-fp-math` | Assume no infinity values. | UB if Inf actually occurs |
| `--enable-no-signed-zeros-fp-math` | `-C llvm-args=--enable-no-signed-zeros-fp-math` | Ignore sign of zero. | Changes `-0.0 + 0.0` behavior |
| `--enable-no-trapping-fp-math` | `-C llvm-args=--enable-no-trapping-fp-math` | FP operations don't trap. | Suppresses FP exceptions |
| `--denormal-fp-math=preserve-sign` | `-C llvm-args=--denormal-fp-math=preserve-sign` | Flush denormals to zero (preserving sign). | Significant speedup on some hardware |
| `--denormal-fp-math-f32=preserve-sign` | `-C llvm-args=--denormal-fp-math-f32=preserve-sign` | Same, f32 only. | - |
| `--acc-reassoc` | `-C llvm-args=--acc-reassoc` | Reassociate accumulation chains (FP reductions). | Changes FP result ordering |

These are all **unsafe** in the sense that they violate IEEE 754 semantics. Safe for
workloads where exact FP reproducibility is not required (aggregations, analytics,
ML inference).

### 17.6 Development build (fast compile, decent performance)

```toml
[profile.dev]
opt-level = 1

[profile.dev.package."*"]
opt-level = 2

# Don't use LTO, keep codegen-units high for fast dev builds
```

### 17.7 Flags that are safe at dev optimization level

Most LLVM args are only meaningful at `-C opt-level=2` or higher. At `opt-level=0`:
- Target CPU/features still apply (SIMD intrinsics work)
- Frame pointer and debug info settings work
- LTO is ignored
- Inlining thresholds are largely ignored (no inlining at O0)
- Vectorization is disabled

At `opt-level=1`:
- Basic inlining occurs
- Loop vectorization may fire on simple loops
- LLVM args for thresholds will take effect but with limited scope

### 17.8 Flags to avoid

| Flag | Why |
|------|-----|
| `--force-vector-width=N` (nonzero) | Overrides cost model. Benchmark first. |
| `--unroll-count=N` (global) | Overrides per-loop analysis. Too aggressive can bloat icache. |
| `--enable-no-nans-fp-math` | Undefined behavior if NaN occurs. Only use if you control all float inputs. |
| `opt-level = "z"` | Disables loop vectorization. Catastrophic for data workloads. |
| `-Z unsound-mir-opts` | Known to cause miscompilation. Never use in production. |
| `-C no-vectorize-loops` | Disables vectorization entirely. |
| `--inline-threshold=2000+` | Compile time explosion for diminishing returns. |
