## 1. Foundational Algorithms

### 1.1 Cilk (MIT, 1990s-2000s)

Cilk is the canonical work-stealing scheduler. The theoretical foundation was
laid by Blumofe and Leiserson in "Scheduling Multithreaded Computations by Work
Stealing" (JACM, 1999). The implementation was developed at MIT over a decade,
starting with Cilk-1 in 1994, through Cilk-5 in 1998, and continuing as
Intel Cilk Plus (2010-2018) and OpenCilk (2020-present).

The central data structure is the deque (double-ended queue). Each worker
thread owns one deque. When a worker spawns a new task, it pushes the task
onto the bottom of its own deque (LIFO order). When a worker finishes its
current task, it pops from the bottom of its deque to get the next task.
The LIFO discipline preserves temporal locality: the most recently spawned
task is likely to operate on data that is still in the worker's L1/L2 cache.

Stealing operates from the opposite end. When a worker's deque is empty, it
becomes a thief. It selects a random victim (another worker) and attempts to
steal a task from the top of the victim's deque (FIFO order). The stolen
task is typically a large-granularity task (one that was spawned early in a
recursive decomposition), which means the thief gets a substantial chunk of
work rather than a leaf-level micro-task. This asymmetry is deliberate:
stealing incurs overhead (cache misses, synchronization), so each steal
should yield enough work to amortize the cost.

The provable time bound is:

```
E[T_P] <= T_1/P + O(T_inf)
```

where T_1 is the total work (serial execution time), P is the number of
processors, and T_inf is the critical path length (longest sequential chain
of dependent tasks). The expected number of steal operations is O(P * T_inf),
which means steals are rare when the parallelism (T_1 / T_inf) is large
relative to P. In practice, on well-parallelized workloads, workers spend
over 99% of their time executing local tasks and less than 1% stealing.

The space bound is equally important:

```
S_P <= S_1 * P
```

where S_1 is the serial stack depth. Each worker can have at most S_1 frames
on its deque (the full depth of the recursive decomposition), and there are
P workers. This linear space guarantee prevents memory explosion under
parallelism; a program that uses 1 MB of stack serially will use at most
P MB under P-way parallel execution.

Cilk enforces fully strict computations: a spawned task can only be stolen
before it begins executing. Once execution starts, the task runs to completion
on the worker that started it. There is no task migration, no preemption, no
suspension. This simplification avoids the complexity of checkpointing and
resuming task state, and it enables the deque to be implemented without
locks on the owner's push/pop path (only the steal operation needs
synchronization).

Cilk Plus (Intel, 2010) extended the model with vector parallelism (SIMD
array notation), reducers (commutative reduction variables that avoid
serialization), and holders (thread-local storage with Cilk semantics).
Intel discontinued Cilk Plus in 2018. OpenCilk (MIT, 2020-present)
continues the research lineage with a modern LLVM-based implementation,
adding features like structured dags for provably efficient cache behavior
and reducer hyperobjects.

The key insight from Cilk for hilavitkutin: work-stealing achieves
near-optimal load balancing with minimal overhead when the work-to-depth
ratio is high. The randomized victim selection is theoretically optimal and
practically simple. The deque structure is the right primitive.

### 1.2 HEFT (Heterogeneous Earliest Finish Time)

HEFT was introduced by Topcuoglu, Hariri, and Wu in "Performance-Effective
and Low-Complexity Task Scheduling for Heterogeneous Computing" (IEEE TPDS,
2002). It remains the most widely cited algorithm for scheduling DAGs onto
heterogeneous processors, with over 7,000 citations. Its popularity comes
from a rare combination: simplicity, speed, and good empirical performance.

HEFT operates in two phases.

**Phase 1: Upward rank computation.** Each task node n in the DAG receives
a rank value computed bottom-up:

```
rank_u(n) = w_avg(n) + max_{s in succ(n)} (c_avg(n,s) + rank_u(s))
```

where w_avg(n) is the average computation cost of task n across all
processor types, c_avg(n,s) is the average communication cost between task
n and successor s, and the max is taken over all successors. Exit tasks
(no successors) have rank_u(n) = w_avg(n). The rank represents the length
of the critical path from task n to the DAG exit, weighted by average
costs. Tasks on the critical path have the highest ranks.

**Phase 2: Task assignment.** Tasks are sorted in decreasing order of
upward rank. For each task (in priority order), HEFT evaluates all
available processors and assigns the task to the processor that yields the
earliest finish time. The earliest finish time accounts for: the task's
computation cost on that processor, the availability time of the processor
(when it finishes its currently assigned work), and the data arrival time
(when all predecessors' outputs are available, including communication
delay).

HEFT supports insertion-based scheduling: when evaluating a processor, it
checks for gaps in the processor's schedule where the task could be
inserted (between two already-assigned tasks) if the gap is large enough.
This improves utilization on processors with uneven workloads.

The time complexity is O(e * p) where e is the number of edges in the DAG
and p is the number of processors. For a DAG with 1000 tasks and 10
processors, this is on the order of tens of thousands of operations, which
completes in microseconds. HEFT is a static algorithm: it computes the
entire schedule before execution begins.

Empirical evaluations consistently show HEFT producing makespans within 2x
of optimal for practical DAGs. Topcuoglu et al. reported average schedule
length ratios (SLR) of 1.1-1.3 across Gaussian elimination, FFT, and
random DAG benchmarks with 10-50 tasks on 4-8 processors. HEFT
outperforms most other static heuristics (MCP, DLS, ETF) while being
simpler to implement.

**Limitations.** HEFT is static: it does not adapt to runtime variations
(task execution times deviating from estimates, processor failures,
dynamic load). It assumes accurate cost estimates, which are often
unavailable before the first execution. It does not model memory
constraints, cache effects, or I/O contention. And it requires
communication cost modeling, which is irrelevant for shared-memory
systems where all processors access the same address space.

loimu uses a variant of HEFT: upward rank computation with lane assignment.
The variant drops communication cost modeling (all data is in shared
memory) and replaces processor heterogeneity with lane grouping (logical
groups of work units that can share resources). The upward rank computation
is identical; the assignment phase is simplified because there is no
communication delay term.

### 1.3 List Scheduling Variants

List scheduling is a family of algorithms, not a single algorithm. The
common structure: maintain a priority queue of ready tasks (tasks whose
predecessors have all completed), and whenever a processor becomes
available, assign the highest-priority ready task to it.

The family differs in two dimensions: the priority function and the
assignment policy.

**Priority functions:**
- **Upward rank** (HEFT): critical path length from task to exit.
  Prioritizes tasks that would delay the overall makespan if delayed.
- **Downward rank**: critical path length from entry to task. Prioritizes
  tasks that are "closest to done" in terms of predecessors completed.
- **Level-based**: the longest path from the DAG entry to the task.
  Prioritizes deep tasks. Equivalent to BFS ordering.
- **Critical path**: the task's position on the longest path through the
  DAG. A special case of upward rank.
- **Shortest job first**: smallest estimated execution time. Reduces
  average turnaround time but can starve long tasks.
- **HLFET (Highest Level First with Estimated Times)**: Adam, Chandy, and
  Dickson (1974). One of the earliest DAG scheduling heuristics. Uses
  level as priority, breaks ties by estimated time.

**Assignment policies:**
- **First available**: assign to whichever processor finishes its current
  task first.
- **Earliest finish time** (HEFT): assign to whichever processor would
  finish this specific task earliest, accounting for data arrival times.
- **Random**: assign to a random available processor. Surprisingly
  effective when task costs are uniform.

**Static vs dynamic list scheduling.** Static list scheduling computes the
full priority ordering and assignment before execution begins. Dynamic list
scheduling re-evaluates priorities and assignments as tasks complete,
potentially reordering the remaining tasks based on actual execution times.
The static approach is cheaper (one computation) but less adaptive. The
dynamic approach handles cost estimation errors but adds overhead to the
critical path (re-prioritization at each task completion).

loimu uses static list scheduling with a hybrid twist: the initial schedule
uses heuristic cost estimates (1 + number of writes + number of emits per
work unit), but after 10 execution passes, the scheduler switches to
EMA-based (exponential moving average) cost estimates derived from measured
execution times. This is not truly dynamic scheduling (the schedule is
recomputed periodically, not per-task), but it converges toward accurate
priorities over time.

## 2. Modern Implementations (Post-2018)

### 2.1 Taskflow (C++, Tsung-Wei Huang et al.)

Taskflow is a header-only C++ library for parallel and heterogeneous task
graph computing. The primary publication is "Taskflow: A Lightweight
Parallel and Heterogeneous Task Graph Computing System" (IEEE TPDS, 2021).
The library has seen significant adoption in EDA (electronic design
automation), machine learning inference, and scientific computing.

Taskflow defines five task types that cover the space of practical parallel
patterns:

1. **Static tasks.** A callable with no runtime control flow. The task
   runs once when its dependencies are satisfied. This is the common case
   for pure computation.

2. **Dynamic tasks (subflows).** A task that, when executed, spawns a
   subgraph of additional tasks. The parent task is not considered complete
   until all spawned subtasks finish. This enables recursive decomposition:
   a divide-and-conquer algorithm spawns subtasks dynamically based on
   input size, without pre-declaring the graph structure.

3. **Composed tasks (modules).** A reusable subgraph that can be embedded
   in multiple parent graphs. The module is defined once and instantiated
   multiple times. Dependencies between the parent graph and the module's
   entry/exit tasks are declared explicitly. This enables library-level
   composability: a library publishes a Taskflow module, and consumers
   embed it in their own graphs.

4. **Conditional tasks.** A task that returns an integer selecting which
   outgoing edge to follow. This enables in-graph branching (if/else) and
   looping (return to a previous task) without rebuilding the graph.
   Conditional tasks are the mechanism for iterative algorithms: a
   convergence-check task returns 0 (continue) or 1 (done), routing
   execution back to the loop body or forward to the exit.

5. **cudaFlow tasks.** A task that encapsulates a CUDA kernel graph.
   Dependencies between CPU tasks and GPU tasks are expressed uniformly
   in the same graph. Data transfers (host-to-device, device-to-host) are
   explicit tasks in the graph.

The executor uses a work-stealing scheduler. Benchmarks in the TPDS paper
report approximately 2 microseconds of per-task overhead (the cost of
dequeueing a task, checking its dependencies, and invoking its callable).
Scaling benchmarks demonstrate near-linear speedup to 40 cores on a
dual-socket Xeon system for graph analytics workloads (BFS, SSSP,
PageRank).

Taskflow includes a profiling tool (tfprof) that generates timeline
visualizations of task execution. Each task's start time, end time, and
worker thread are recorded and rendered as a Gantt chart. This is useful
for identifying load imbalance, critical path bottlenecks, and scheduling
overhead.

The key insight from Taskflow for hilavitkutin: the five task types are
not arbitrary; they are the minimal set that covers all practical parallel
patterns without requiring escape hatches. Static for known work, dynamic
for recursive decomposition, conditional for control flow, composed for
modularity, and GPU for heterogeneous compute. hilavitkutin does not need
GPU tasks, but the other four patterns are directly relevant.

### 2.2 Rayon (Rust)

Rayon is the standard work-stealing library for data parallelism in Rust.
It provides a global thread pool with work-stealing load balancing, exposed
through two primary interfaces.

The `join(op_a, op_b)` function is the core primitive. It executes two
closures potentially in parallel: one runs on the current thread, the other
is pushed onto the current thread's deque and may be stolen by an idle
worker. `join` is recursive: each closure can call `join` again, creating a
tree of parallel tasks. The recursion bottoms out when tasks are small
enough to execute sequentially.

The `ParallelIterator` trait transforms iterator chains into parallel work.
`.par_iter()` replaces `.iter()`, and the iterator adaptors (map, filter,
fold, collect) are automatically parallelized using a divide-and-conquer
splitting strategy. The split point is chosen by the `IndexedParallelIterator`
trait: the iterator reports its length, and Rayon splits it in half
recursively until chunks are below a threshold.

Rayon uses `crossbeam-deque` internally for the work-stealing deques. The
thread pool is sized to the number of logical cores by default, but can be
configured. Thread pinning is not built in; the OS scheduler places threads
on cores.

The `scope` API provides structured concurrency: spawned tasks are bounded
by the scope's lifetime, ensuring that all spawned work completes before
the scope exits. This prevents dangling references and enables safe
borrowing of stack data by parallel tasks.

What Rayon does NOT provide: task graphs, dependency declarations,
priority scheduling, heterogeneous task types, or conditional execution.
Rayon is a data-parallel runtime, not a DAG scheduler. It is the right
tool for parallelizing the inner loop of morsel processing (applying a
transform to each row in a morsel), but it is not the right tool for
orchestrating the pipeline DAG (deciding which stages run in what order
on which cores).

For hilavitkutin, Rayon is relevant as the inner parallelism engine:
within a single pipeline stage, Rayon can parallelize morsel processing
across cores. The outer DAG scheduling (which stage runs when, dependency
tracking, progress monitoring) is hilavitkutin's responsibility.

### 2.3 Tokio (Rust)

Tokio is an asynchronous runtime for Rust, designed for I/O-heavy
applications (network servers, database clients, HTTP services). Its
scheduler is a work-stealing design optimized for a different workload
profile than Rayon: tasks that frequently suspend (awaiting I/O) rather
than tasks that run to completion.

Tokio's multi-threaded scheduler has N worker threads (default: one per
logical core). Each worker has a local run queue (bounded, 256 slots) and
shares a global injection queue. New tasks are pushed to the local queue
(LIFO for locality). When the local queue is full, half the tasks are
moved to the global queue. When the local queue is empty, the worker
first checks the global queue, then attempts to steal from a random other
worker's local queue.

The I/O driver is integrated: epoll (Linux), kqueue (macOS), or io_uring
(Linux 5.1+) readiness notifications wake tasks directly into the
scheduler. When a socket becomes readable, the I/O driver finds the
waiting task and pushes it onto the notifying worker's local queue. This
keeps I/O-related tasks on the worker that is already handling that
connection, preserving cache locality for connection state.

The central challenge with Tokio is mixing CPU-bound and I/O-bound work.
An async task that performs heavy computation (parsing a large JSON
document, compressing data, running a query plan) blocks the worker thread
and prevents it from polling I/O readiness. Other tasks on the same worker
starve.

Tokio's mitigations:
- **spawn_blocking()**: offloads CPU-bound work to a separate,
  expandable thread pool. The async task awaits the result without blocking
  a worker.
- **yield_now()**: manually yields the current task, returning it to the
  back of the queue. Requires programmer discipline.
- **Cooperative budgeting**: since Tokio 0.2.14, each task has a budget of
  128 poll iterations. After 128 polls, the task is forced to yield. This
  prevents a single task from monopolizing a worker indefinitely.

For hilavitkutin, Tokio is relevant to saalis's use case: saalis has
I/O-bound connector tasks (HTTP requests to game store APIs, database
queries) interleaved with CPU-bound data pipeline tasks (deduplication,
aggregation, scoring). Tokio alone cannot handle this mix efficiently.
The `Progress::Yield(Pending)` model proposed in the hilavitkutin topic
is a different approach: I/O tasks yield immediately when they would
block, returning control to the scheduler, which can run other ready
tasks on the same worker. This is cooperative scheduling with explicit
yield points, rather than Tokio's preemptive budgeting.

### 2.4 Intel TBB Flow Graph

Intel Threading Building Blocks (now oneTBB) includes a flow graph
component for streaming dataflow computation. Unlike Taskflow's task
graph (which represents a DAG of one-shot computations), TBB's flow
graph represents a persistent topology through which messages flow
continuously.

Node types include:
- **function_node**: applies a functor to each incoming message, produces
  an output message. Concurrency policy (serial, unlimited, or
  N-parallel) is declared per node.
- **continue_node**: fires when all predecessors have sent a continue
  signal. Used for synchronization barriers.
- **join_node**: waits for messages from all input ports before producing
  a tuple.
- **split_node**: takes a tuple and sends each element to a separate
  output port.
- **buffer_node**, **queue_node**, **sequencer_node**: buffer messages with
  various ordering guarantees.
- **limiter_node**: rate-limits the flow to prevent downstream overload.

The flow graph topology is fixed at construction. Messages (data items)
flow through the graph, and nodes process them as they arrive. This is a
push-based streaming model, not a pull-based DAG model. The distinction
matters: in a DAG scheduler, a task runs once (or once per morsel batch);
in a flow graph, a node runs once per incoming message, potentially
millions of times.

TBB's flow graph is relevant to hilavitkutin as a conceptual reference
for morsel pipelines. A morsel flowing through a chain of pipeline stages
is analogous to a message flowing through a chain of function_nodes.
The concurrency control per node (serial, unlimited, N-parallel) maps to
hilavitkutin's lane concept: a pipeline stage that must be serial (e.g.,
writing to an output file) has concurrency = 1; a stage that is
embarrassingly parallel (e.g., parsing rows) has concurrency = unlimited.

### 2.5 crossbeam-deque (Rust)

crossbeam-deque is the low-level building block that Rayon and Tokio use
for their work-stealing implementations. It provides a lock-free
Chase-Lev deque (Chase and Lev, "Dynamic Circular Work-Stealing Deque",
SPAA 2005).

The Chase-Lev deque has one owner and multiple stealers. The owner pushes
and pops from the bottom (LIFO). Stealers steal from the top (FIFO). The
implementation uses atomic operations on the bottom and top indices, with
a growable circular buffer for storage.

The steal operation returns one of three results:
- `Steal::Success(T)`: a task was successfully stolen.
- `Steal::Empty`: the deque is empty.
- `Steal::Retry`: a concurrent operation (another steal or a pop)
  conflicted. The stealer should retry.

The `Retry` result is essential for correctness without locks: rather than
blocking on a mutex, the stealer simply tries again. In practice, retries
are rare because steal operations are infrequent relative to push/pop
operations (the Cilk bound again: O(P * T_inf) steals total).

crossbeam-deque also provides an `Injector` queue: a multi-producer,
multi-consumer queue for injecting new work from outside the thread pool.
External code pushes tasks into the injector; workers steal from it when
their local deques are empty (after checking other workers' deques).

For hilavitkutin, crossbeam-deque is the implementation primitive. Each
worker thread gets a `Worker<Task>` deque. Idle workers steal from other
workers' `Stealer<Task>` handles. External task injection (new pipeline
runs, wakeups from I/O completion) uses the `Injector<Task>`.

## 3. Cache-Aware and NUMA-Aware Scheduling

### 3.1 Work-Stealing with Locality Hints

The fundamental tension in work-stealing is between load balancing and
data locality. Stealing moves work to a different core, which means the
stolen task operates on data that is likely in the victim's cache, not the
thief's cache. For tasks with large working sets, this cache miss penalty
can dominate the computation time.

Random victim selection (Cilk's approach) ignores locality entirely. It
produces optimal load balance (provably) but worst-case locality. On
modern multi-core CPUs with hierarchical caches (L1 private, L2 per-core
or per-cluster, L3 shared, DRAM), the performance impact depends on where
the data currently resides.

Locality-guided stealing restricts the victim selection to nearby cores:
1. First, attempt to steal from a core sharing the same L2 (e.g., the
   same P-core cluster on Apple Silicon, where two P-cores share an L2).
2. If no work is found, widen to cores sharing the same L3.
3. If still no work, steal from any core (fall back to random).

This hierarchical stealing was analyzed by Acar, Blelloch, and Blumofe in
"The Data Locality of Work Stealing" (Theory of Computing Systems, 2002).
They proved that restricting steals to nearby processors preserves the
asymptotic work-stealing bound while reducing cache misses by a factor
proportional to the cache hierarchy depth.

LLAMA (Bauer et al., 2020, SC '20) applied locality-aware scheduling to
graph analytics workloads (BFS, PageRank, connected components on billion-
edge graphs). Their approach partitions the graph by NUMA node, assigns
initial work to the owning node, and restricts stealing to within-node
first. They reported 1.3-2.1x speedup over locality-unaware work-stealing
on a 4-socket AMD EPYC system with 256 cores.

In practice, most production implementations (Rayon, Tokio, Go's runtime)
use random stealing because the implementation is simpler and the locality
benefit is workload-dependent. Random stealing is a good default;
locality-aware stealing is an optimization for NUMA systems or workloads
with large per-task working sets.

For hilavitkutin's target workloads, data locality is naturally good
because morsels are sized to fit in L2 cache. A stolen morsel task will
miss on the morsel data (loading it from L3 or DRAM), but the data is
small enough that the cold-start penalty is bounded. Random stealing is
the right default.

### 3.2 Cache-Partitioned Scheduling

The goal of cache-partitioned scheduling is to prevent interference
between tasks sharing a cache level. If two tasks on the same L2 cluster
both have working sets that fill 80% of the L2, they will thrash each
other's cache lines, and both will run at DRAM speed.

Intel Cache Allocation Technology (CAT), available on Xeon processors
since Broadwell, provides hardware cache partitioning. The OS or
application can assign cache ways to specific cores or processes, ensuring
that one workload's cache footprint does not evict another's.

Software-based cache partitioning does not require hardware support: the
scheduler assigns tasks to cores such that the total working set of tasks
on each core (or core cluster) fits within the cache. For morsel
processing, this means:

```
morsel_count_per_core * morsel_size * element_size < L2_size
```

Typical L2 sizes: 256 KB (Intel E-core), 1.25 MB (Intel P-core), 4 MB
(Apple Silicon P-core cluster of 2 cores), 16 MB (Apple Silicon shared L2
per 4 E-cores). A morsel of 1024 rows at 16 bytes per element is 16 KB,
which fits comfortably even in the smallest L2. The constraint becomes
relevant when multiple columns are accessed simultaneously: 10 columns of
1024 rows at 16 bytes each is 160 KB, which still fits in every L2 except
the smallest E-core configurations.

The morsel sizing formula from the dispatch topic (202603121800) already
accounts for cache capacity. hilavitkutin inherits this: morsel sizes are
chosen to fit the target core's cache at construction time, and the
scheduler does not need to perform runtime cache management.

### 3.3 NUMA-Aware Morsel Assignment

Leis et al. ("Morsel-Driven Parallelism: A NUMA-Aware Query Evaluation
Framework for the Many-Core Age", SIGMOD 2014) introduced NUMA-aware
morsel assignment. The core idea: partition the input data by NUMA node,
and assign morsels to workers on the same NUMA node as the data.

The implementation requires:
1. **NUMA topology detection** at startup (Linux: libnuma, /sys/devices/
   system/node/; macOS: not applicable, unified memory).
2. **NUMA-aware allocation**: allocate column data on a specific NUMA
   node using `numa_alloc_onnode()` or relying on Linux's first-touch
   policy (the page is allocated on the node of the thread that first
   writes to it).
3. **Worker-to-node binding**: pin worker threads to specific NUMA nodes
   using `pthread_setaffinity_np()` or `sched_setaffinity()`.
4. **Morsel assignment**: when dispatching morsels, prefer assigning a
   morsel to a worker on the same NUMA node as the morsel's data.

On a 4-socket system with 64 cores per socket, NUMA-aware morsel
assignment can yield 2-4x speedup over NUMA-oblivious assignment for
memory-bandwidth-bound workloads (table scans, hash joins). The speedup
comes from avoiding cross-socket memory accesses, which cost 2-3x more
latency (100-200 ns remote vs 50-80 ns local on modern AMD EPYC).

Apple Silicon uses unified memory with a single memory controller. There
is no NUMA topology. However, Apple Silicon does have heterogeneous
cache sizes: P-core clusters have larger L2 (4 MB shared between 2 cores
on M1/M2, or 16 MB shared between 4 cores on M3 Pro/Max) compared to
E-core clusters (4 MB shared between 4 E-cores). This is not NUMA (all
memory accesses have uniform latency), but it does affect optimal morsel
sizing per core type, which is addressed in section 4.

The dispatch topic (202603121800, Q1) decided that NUMA is out of scope
for hilavitkutin's initial design, but the architecture must not preclude
a future NUMA-aware implementation. The Executor trait and ColumnStorage
trait together provide the extension points: a NUMA-aware Executor assigns
chains to cores on the socket where input data resides, and a NUMA-aware
ColumnStorage allocates data on the appropriate node.

## 4. Heterogeneous Core Scheduling

### 4.1 big.LITTLE and P-core / E-core Architectures

ARM's big.LITTLE architecture (2011) introduced the concept of
heterogeneous cores on the same die: high-performance "big" cores paired
with energy-efficient "LITTLE" cores. Apple Silicon adopted this pattern
with P-cores (Performance) and E-cores (Efficiency). Intel followed with
Alder Lake (12th gen, 2021), pairing P-cores (Golden Cove) with E-cores
(Gracemont).

The cores differ in:
- **Clock speed**: P-cores run at higher frequencies (3.2-4.0 GHz on
  Apple Silicon M-series vs 2.0-2.4 GHz for E-cores).
- **Issue width**: P-cores can decode and execute more instructions per
  cycle (8-wide decode on Apple M-series P-core vs 4-wide on E-core).
- **Cache size**: P-cores typically have larger L1/L2 caches.
- **Power consumption**: E-cores use 2-5x less power per operation.
- **Instruction set**: on Intel, E-cores lack AVX-512 (12th-13th gen) or
  may have different SIMD capabilities. On ARM/Apple, all cores share the
  same ISA.

For a work-stealing scheduler, heterogeneous cores create an asymmetry:
a task that takes 1 ms on a P-core might take 2-3 ms on an E-core. If
the scheduler treats all cores as equal (uniform work-stealing), tasks
assigned to E-cores take longer, creating load imbalance. The P-cores
finish their work and steal from E-cores, but the stolen tasks have
already started on the E-core and cannot be migrated (fully strict
semantics).

### 4.2 Thread Placement Strategies

**All on P-cores.** Pin all worker threads to P-cores, ignoring E-cores.
Maximizes per-task throughput but wastes available hardware. On an Apple
M2 (4P + 4E), this uses only half the cores.

**Round-robin.** Distribute threads across all cores without
distinguishing types. Simple but ignores the performance asymmetry.

**Classification-based.** Classify tasks by their characteristics and
assign accordingly:
- **Latency-sensitive tasks** (user-facing responses, interactive UI
  updates) go to P-cores. The higher single-thread performance minimizes
  response time.
- **Throughput/batch tasks** (background indexing, bulk transforms) go to
  E-cores. Power efficiency matters more than per-task latency.
- **Morsel processing** can go to any core. Morsels are designed to fit
  in cache, so the computation is ALU-bound, not memory-bound. P-cores
  process morsels faster but E-cores process them at higher power
  efficiency.
- **I/O-waiting tasks** go to E-cores. A task that is blocked on network
  I/O uses negligible CPU; a P-core is wasted on it.

**Adaptive.** Measure task execution time on both core types and
dynamically adjust placement. If a task consistently takes > 2x longer
on E-cores, prefer P-cores for it. If a task shows < 1.5x difference,
prefer E-cores for power efficiency.

### 4.3 Linux Scheduling for Heterogeneous Cores

Linux's Completely Fair Scheduler (CFS) gained heterogeneous core support
through Energy-Aware Scheduling (EAS), introduced in kernel 5.0 (2019).
EAS uses an energy model (provided by the hardware or device tree) to
estimate the energy cost of running a task on each core type, and places
tasks to minimize total energy consumption while meeting performance
targets.

Per-Entity Load Tracking (PELT) estimates each task's CPU utilization as
an exponential moving average of recent activity. High-utilization tasks
(consistently CPU-bound) are placed on P-cores. Low-utilization tasks
(frequently sleeping, I/O-bound) are placed on E-cores.

`sched_ext` (Linux 6.12+, December 2024) enables user-defined scheduling
policies via BPF programs. The kernel provides hooks (enqueue, dequeue,
dispatch, running, stopping), and the BPF program implements the
scheduling logic. This allows applications to implement domain-specific
scheduling without kernel modifications. loisto (the build system) uses
sched_ext for custom BPF scheduling where the userspace scheduler decides
task placement and the kernel executes the decisions.

### 4.4 Impact on Morsel Sizing

On heterogeneous hardware, the optimal morsel size differs by core type.
P-cores have larger caches and higher throughput, so they can process
larger morsels efficiently. E-cores have smaller caches and lower
throughput, so smaller morsels avoid cache pressure and keep the core
busy without wasting energy on cache misses.

A concrete example on Apple M2:
- P-core cluster: 4 MB L2 shared between 2 cores = 2 MB effective per
  core. A morsel of 4096 rows at 16 bytes across 10 columns = 640 KB,
  fitting comfortably.
- E-core cluster: 4 MB L2 shared between 4 cores = 1 MB effective per
  core. The same 4096-row morsel (640 KB) fits, but leaves little room
  for other working set data. A 2048-row morsel (320 KB) would be safer.

Dynamic morsel sizing: the scheduler adjusts morsel size based on which
core type will process it. P-cores get 4096-row morsels; E-cores get
2048-row morsels. The morsel is the unit of work-stealing, so stolen
morsels from a P-core deque (4096 rows) could end up on an E-core. The
scheduler can either: (a) split stolen morsels before dispatching to an
E-core, or (b) accept the suboptimal size and let the E-core process
the larger morsel with higher cache pressure.

loimu's approach is static morsel sizing with runtime detection: morsel
size is chosen once at startup based on the available hardware.
hilavitkutin could be more adaptive, adjusting morsel sizes per-core-type
or even per-task based on measured cache miss rates. Whether this
complexity is justified depends on the workload: for polka-dots (120 KB
total data), morsel sizing is irrelevant. For saalis (100 MB-1 GB), it
could matter.

## 5. Mixed-Granularity Scheduling

### 5.1 The Scale Problem

The three projects that hilavitkutin serves have work units spanning five
orders of magnitude in execution time:

| Work type | Typical duration | Example |
|-----------|-----------------|---------|
| Morsel processing | 1-100 us | Parsing 1024 TOML rows |
| Column transform | 100-1000 us | Deduplicating 10K entries |
| File I/O | 1-50 ms | Reading a polka.toml file |
| HTTP request | 50-5000 ms | Querying a game store API |
| Database query | 10-500 ms | SQLite aggregation query |

A single work-stealing deque cannot handle this range efficiently. If a
5-second HTTP request sits in a deque alongside 50-microsecond morsel
tasks, the worker processing the HTTP request is occupied for 100,000x
longer than a morsel task would take. Other workers finish their morsels
and find no work to steal (the HTTP worker has only one task, and it is
in progress, so it cannot be stolen under fully strict semantics).

### 5.2 Priority Inversion

Priority inversion occurs when a high-priority task is blocked behind a
low-priority task. In a work-stealing context: a high-priority morsel
task is in a deque behind a low-priority I/O task. The worker pops the
I/O task (it is at the bottom, LIFO), and the morsel task is stuck until
the I/O task completes.

Solutions from the literature:
- **Separate queues per priority level.** High-priority tasks in one
  deque, low-priority in another. Workers drain the high-priority deque
  first. Effective but doubles the data structure overhead.
- **Preemption.** The scheduler interrupts a running low-priority task
  when a high-priority task arrives. Complex to implement (requires
  saving and restoring task state) and can cause correctness issues if
  the interrupted task holds locks or partially-written data.
- **Cooperative yielding.** Tasks voluntarily yield at defined points,
  allowing the scheduler to interleave high-priority work. Requires
  programmer discipline but avoids the complexity of preemption.

hilavitkutin's `Progress::Yield` model is a form of cooperative yielding.
I/O tasks never block: they initiate the I/O operation, return
`Yield(Pending)`, and are parked until the I/O completion callback wakes
them. The worker is immediately free to process the next task in its
deque. Priority inversion is impossible because I/O tasks never hold a
worker for more than the time to initiate the I/O call (microseconds).

### 5.3 Starvation Prevention

Starvation occurs when a task in a deque is never executed because new
tasks keep being pushed on top of it (LIFO order) and stolen from above
it (FIFO order). In theory, a task in the middle of a deque could wait
indefinitely.

In practice, starvation is rare in work-stealing because the deque is
small (bounded by the recursive depth of the computation) and tasks
complete quickly. But with mixed-granularity work, starvation is a real
risk: a steady stream of morsel tasks keeps the worker busy, and a
long-pending I/O task at the bottom of the deque never gets processed.

Solutions:
- **Aging.** Increase a task's priority the longer it waits in the deque.
  When it reaches the highest priority, it jumps to the front. Simple but
  requires per-task metadata and periodic scanning.
- **Dedicated workers.** Reserve one or more workers for low-priority or
  I/O tasks. These workers do not process morsel tasks, so they are
  always available for I/O. This wastes a core if I/O is infrequent.
- **The yield model.** hilavitkutin's approach avoids starvation by
  construction. I/O tasks yield immediately (never blocking a worker),
  and they are woken externally (by I/O completion) rather than waiting
  in a deque. The wakeup pushes them back into the scheduler's injection
  queue, where they are picked up by the next available worker.

### 5.4 The Long-Tail Problem

In morsel processing, most morsels complete in similar time (the data is
uniform, the computation is deterministic). But some morsels take much
longer: skewed data distributions (one morsel contains all the large
strings), cache misses (a morsel's data was evicted by another task),
or branch mispredictions (a morsel triggers a rare code path).

These long-tail morsels create load imbalance at the end of a pipeline
stage: all workers finish their morsels except one, which is still
processing a slow morsel. The other workers are idle.

Cilk's solution is recursive splitting: if a task is splittable, the
worker splits it into subtasks and pushes them onto its deque, where
other workers can steal them. This works for divide-and-conquer
algorithms but not for morsels: a morsel is a contiguous range of rows,
and splitting it mid-processing requires checkpointing the partial
result and restarting on a different worker.

Practical solutions:
- **Smaller morsels.** Reduce morsel size to reduce the maximum possible
  tail latency. A 256-row morsel has at most 4x the variance of a
  1024-row morsel (assuming uniform row cost). But smaller morsels
  increase scheduling overhead (more tasks to manage).
- **Speculative execution.** When a worker finishes early, it starts
  processing the next pipeline stage's morsels speculatively (if
  dependencies allow). This hides the tail latency of the current stage
  behind useful work from the next stage.
- **Dynamic rebalancing.** The scheduler monitors morsel completion times.
  If one worker is significantly slower than others (still processing
  when others are idle), it splits the remaining unprocessed morsels of
  the slow pipeline stage into smaller chunks and distributes them.

hilavitkutin's morsels are fixed-size once dispatched, but the scheduler
can rebalance by reassigning remaining undispatched morsels. If worker A
has processed 80% of its assigned morsels and worker B has processed 20%,
the scheduler can transfer worker B's remaining morsels to worker A's
deque. This works because undispatched morsels are just ranges in a
column; reassignment is a metadata operation, not a data copy.

## 6. Adaptive Scheduling

### 6.1 Self-Tuning Schedulers

A self-tuning scheduler measures task execution times and adjusts its
scheduling parameters to improve future decisions. The feedback loop:

1. **Predict.** Assign a cost estimate to each task before execution.
   Initial estimates come from heuristics (e.g., loimu's formula:
   cost = 1 + number_of_writes + number_of_emits).
2. **Schedule.** Use the cost estimates to compute a schedule (list
   scheduling with upward rank, HEFT assignment, etc.).
3. **Execute.** Run the schedule and measure actual execution times.
4. **Update.** Replace cost estimates with measured values. Use an
   exponential moving average (EMA) to smooth out variance:

```
estimate_new = alpha * measured + (1 - alpha) * estimate_old
```

where alpha is the smoothing factor (typically 0.1-0.3). A higher alpha
responds faster to changes but is more sensitive to outliers. A lower
alpha is more stable but slower to adapt.

5. **Repeat.** On the next execution, use the updated estimates to
   compute a better schedule.

loimu's implementation: for the first 10 passes, use heuristic estimates
(the DAG is not recomputed). After 10 passes, switch to EMA-based
estimates and recompute the schedule. The threshold of 10 is chosen to
accumulate enough measurements for the EMA to converge. The
`RecalculateSchedulePlan` signal triggers a full schedule recomputation
when estimates change significantly.

More sophisticated approaches exist in the literature:
- **Bayesian optimization** for schedule parameters (Alistarh et al.,
  "Distributionally-Aware Scheduling", ASPLOS 2022). Models the
  relationship between scheduling parameters and makespan as a Gaussian
  process, then optimizes parameters using expected improvement. Provably
  converges to the global optimum but requires hundreds of iterations,
  which is impractical for a runtime scheduler.
- **Reinforcement learning** for task placement (Mao et al., "Learning to
  Schedule with Graph Neural Networks", ICLR 2019). Trains a neural
  network to map DAG structure to scheduling decisions. Achieves
  near-optimal schedules on synthetic benchmarks but requires training
  data and inference overhead.

For hilavitkutin, the EMA approach is the right level of complexity.
Bayesian optimization and RL are research-grade techniques that add
latency, complexity, and dependencies without proportional benefit for
the workload sizes (hundreds to thousands of tasks, not millions).

### 6.2 Runtime Cost Estimation and Rebalancing

The quality of a list schedule depends directly on the quality of cost
estimates. With perfect estimates, list scheduling with upward rank
produces near-optimal schedules. With inaccurate estimates, the schedule
can be arbitrarily bad: a task estimated at 1 ms that actually takes
1 second will delay all its dependents, creating a critical path
bottleneck that the scheduler did not anticipate.

**Initial heuristic estimates.** Before any execution, costs must be
guessed. Useful heuristics:
- Constant cost (all tasks cost 1). Simple but ignores heterogeneity.
- Work-proportional (cost = number of input rows * number of columns
  accessed). Better for morsel processing where work scales with data
  size.
- Declaration-based (cost = 1 + writes + emits, loimu's approach).
  Captures the intuition that tasks with more outputs do more work.
  Crude but better than constant.
- Type-based (different cost models per task type). I/O tasks get high
  estimates, morsel tasks get low estimates, aggregation tasks get
  medium estimates based on expected row counts.

**Divergence detection.** After several executions, the scheduler
compares actual costs to estimates. If the ratio actual/estimated exceeds
a threshold (e.g., 2x) for any task, the schedule may be significantly
suboptimal. The scheduler triggers a recomputation.

loimu's `RecalculateSchedulePlan` is triggered by:
- A task's measured cost exceeding 2x its estimate for 3 consecutive
  passes.
- A change in the DAG structure (tasks added or removed).
- An explicit signal from the application.

**Recomputation cost.** For a DAG with N tasks and E edges, recomputing
the upward rank and list schedule takes O(N + E) time. For hilavitkutin's
expected DAG sizes (10-1000 tasks), this is sub-millisecond. Recomputation
is cheap enough to do periodically (every 10-100 passes) without
measurable overhead.

## 7. Relevance to hilavitkutin

This section synthesizes the findings into concrete recommendations for
hilavitkutin's scheduler design.

### 7.1 DAG Construction from Type Declarations

All three projects derive DAG edges from typed declarations: polka-dots'
`Read<T>/Write<T>` on WorkUnits, saalis's resource access declarations,
loimu's field-granular read/write analysis. The pattern is also used by
Unity DOTS (ECS-style systems declare component access, the job scheduler
infers dependencies) and Bevy (Rust ECS with automatic system ordering
based on resource access).

hilavitkutin should derive edges automatically from `Read<T>/Write<T>`
declarations on work units. Two work units conflict (must be ordered) if
one writes a type that the other reads or writes. Two work units that
only read the same type can run in parallel. This is standard
read-write conflict analysis, identical to database transaction
isolation.

The DAG is constructed once at startup (or when the set of registered
work units changes) and cached. Edge derivation is O(N^2 * C) where N is
the number of work units and C is the average number of accessed types
per unit. For N=1000 and C=5, this is 5 million comparisons, completing
in single-digit milliseconds.

### 7.2 List Scheduling with Upward Rank

The literature and loimu's experience both validate list scheduling with
upward rank as a strong default. HEFT's empirical performance (within 2x
of optimal) is well-documented. Taskflow's benchmarks confirm near-linear
scaling with work-stealing underneath.

hilavitkutin should implement:
1. Upward rank computation (O(N + E), bottom-up traversal of the DAG).
2. Lane assignment: group work units into lanes (logical execution
   groups). Units in the same lane share a worker pool and are scheduled
   together.
3. Topological dispatch: ready tasks (all predecessors complete) are
   dispatched in decreasing rank order to available workers.

The initial cost model is declaration-based (1 + writes). After 10
execution passes, switch to EMA-based costs. Recompute the schedule when
costs diverge by more than 2x.

### 7.3 Work-Stealing via crossbeam-deque

Cilk's work-stealing bounds are proven optimal. Rayon's production use
validates the approach in Rust. crossbeam-deque provides the lock-free
data structure.

hilavitkutin should implement:
1. One `Worker<Task>` deque per worker thread.
2. Workers push ready tasks to their local deque (LIFO).
3. Idle workers steal from a random victim's `Stealer<Task>` (FIFO).
4. External injection via `Injector<Task>` for new pipeline runs and
   I/O wakeups.
5. Random victim selection (Cilk-validated, simple, provably optimal).

No locality-aware stealing in the initial design. hilavitkutin's morsels
are cache-sized, so the cold-start penalty on a stolen morsel is bounded
(a few microseconds for L3 access). Locality-aware stealing can be added
later if profiling shows it matters.

### 7.4 Explicit Yield for I/O Integration

This is hilavitkutin's novel contribution. Neither Cilk (CPU-only), Rayon
(CPU-only), nor Tokio (all-async, everything yields) handles the specific
pattern of mixed CPU-bound morsel processing and I/O-bound connector
tasks.

The `Progress` enum:

```
Complete           -- done, outputs written
Yield(BudgetExhausted) -- cooperative yield, resume next quantum
Yield(Pending)     -- waiting on I/O, park until woken
Yield(Retry { attempt, max, backoff }) -- transient failure, retry
Failed(Error)      -- permanent failure
```

Key properties:
- I/O tasks never block a worker. They initiate I/O, return
  `Yield(Pending)`, and are parked. The worker is immediately free.
- Wakeup is external: the I/O completion callback (from Tokio, mio, or a
  platform-specific mechanism) pushes the task back into the injector
  queue.
- `BudgetExhausted` enables cooperative preemption for CPU-bound tasks
  that exceed a time budget. The scheduler tracks time per task and sets
  a cooperative budget (e.g., 1 ms). Tasks that check the budget and
  yield are resumed in the next scheduling round.
- `Retry` handles transient failures (network timeout, lock contention)
  with configurable backoff. The scheduler parks the task for the
  backoff duration and then re-enqueues it.

This model avoids the mixed-granularity problem entirely. I/O tasks do
not occupy workers. Morsel tasks run to completion (no yielding needed
for cache-sized work). The scheduler never has a 5-second HTTP request
blocking a worker while morsel tasks starve.

### 7.5 Morsel-Based Data Dispatch

Morsel-driven execution (Leis et al., SIGMOD 2014) divides column data
into fixed-size chunks (morsels) and dispatches them to workers
independently. Each morsel is a contiguous slice of rows in a column.

hilavitkutin should implement:
1. Morsel size chosen at startup based on hardware detection (cache sizes,
   core count, core types).
2. Default morsel size: 1024 rows (matches common L2 sizes for typical
   element widths).
3. Per-core-type morsel sizing on heterogeneous hardware (larger for
   P-cores, smaller for E-cores) as an optimization.
4. Morsels are dispatched to workers via the work-stealing deque. Each
   morsel is a task containing a (stage_id, row_offset, row_count)
   triple.
5. Workers process morsels by calling the pipeline stage's execute()
   with a slice of the input column.

### 7.6 Adaptive Cost Estimation

hilavitkutin should implement the EMA-based cost estimation that loimu
has validated:

1. Initial cost: declaration-based heuristic.
2. After each execution pass, record measured cost per task.
3. After 10 passes, switch to EMA estimates (alpha = 0.2).
4. Monitor divergence: if any task's actual cost exceeds 2x its estimate
   for 3 consecutive passes, trigger schedule recomputation.
5. Recomputation: re-run upward rank computation and lane assignment
   with updated costs.

The EMA smoothing factor of 0.2 gives a half-life of approximately 3
passes (the estimate reaches 50% of the new measured value after 3
updates). This is responsive enough to adapt to workload changes (a
table growing from 1K to 100K rows) while stable enough to ignore
transient outliers (a single slow morsel due to a GC pause or page
fault).

### 7.7 Summary of Design Choices

| Concern | Decision | Validation |
|---------|----------|------------|
| DAG construction | Automatic from Read/Write declarations | Unity DOTS, Bevy, loimu |
| Scheduling algorithm | List scheduling with upward rank | HEFT (Topcuoglu 2002), loimu |
| Load balancing | Work-stealing with random victim | Cilk (Blumofe-Leiserson 1999) |
| Implementation | crossbeam-deque (Chase-Lev) | Rayon, Tokio |
| I/O integration | Progress::Yield(Pending), external wakeup | Novel; avoids Tokio's async coloring |
| Data dispatch | Morsel-driven, cache-sized chunks | Leis et al. (SIGMOD 2014) |
| Cost estimation | Heuristic initial, EMA after 10 passes | loimu's production experience |
| Heterogeneous cores | Static morsel sizing with detection | Apple Silicon P/E core model |
| NUMA | Out of scope, architecture permits future addition | Dispatch topic Q1 decision |
| Locality-aware stealing | Not initially, random is sufficient | Cilk bounds, cache-sized morsels |

What hilavitkutin should implement, in priority order:

1. **DAG from Read/Write declarations.** Core, non-negotiable. This is
   the foundation that all three projects share.
2. **List scheduling with upward rank + lane assignment.** Validated by
   HEFT literature and loimu's implementation. Good default, well-
   understood performance characteristics.
3. **Work-stealing via crossbeam-deque.** Proven in Rust by Rayon and
   Tokio. Lock-free, bounded space, asymptotically optimal.
4. **Explicit yield for I/O integration.** Novel design that solves the
   mixed-granularity problem without async coloring or separate I/O
   runtimes. Neither Cilk, Rayon, nor Tokio does exactly this.
5. **Morsel-based data dispatch for column processing.** The morsel model
   is well-validated for analytical query processing and applies directly
   to polka-dots' column transforms and saalis's data pipelines.
6. **Adaptive cost estimation.** EMA after heuristic warmup. Cheap to
   implement, measurable improvement in scheduling quality after the
   first few executions.
