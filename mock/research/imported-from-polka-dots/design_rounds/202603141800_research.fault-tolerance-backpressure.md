## 1. Retry Patterns

Retry is the most common fault tolerance mechanism in distributed and pipeline
systems. The naive approach (retry immediately, same parameters) is almost
always wrong. It amplifies load on an already-struggling service, causes
thundering herd effects, and wastes resources when failure is permanent.
Production systems use structured retry patterns that address these failure
modes.

### 1.1 Exponential backoff with jitter

The standard retry delay formula is:

```
delay = min(base_delay * 2^attempt + random(0, max_jitter), max_delay)
```

AWS's architecture blog (Brooker, "Exponential Backoff And Jitter", 2015)
established this as the industry recommendation. The exponential growth
ensures that retry frequency drops rapidly as failures accumulate: attempt 0
waits 100ms, attempt 1 waits 200ms, attempt 2 waits 400ms, attempt 3 waits
800ms. Without the exponential component, a fixed-delay retry at 100ms would
hammer a failing service 10 times per second indefinitely.

The jitter component is critical and frequently omitted by naive
implementations. Without jitter, all clients that failed at the same time
will retry at the same time (base_delay * 2^attempt is deterministic). This
creates a synchronized retry storm: the service recovers briefly, gets hit
by all retries simultaneously, fails again, and all clients retry at the
same later time. Jitter desynchronizes clients by adding a random offset.
AWS recommends "full jitter" (random(0, base_delay * 2^attempt)) over
"equal jitter" (base_delay * 2^(attempt-1) + random(0, base_delay *
2^(attempt-1))) because full jitter produces better spread in practice.

The max_delay cap prevents absurd waits. Without it, attempt 20 at 100ms
base would be 100ms * 2^20 = ~29 hours. A typical cap is 30-60 seconds.

### 1.2 Circuit breakers (Hystrix model)

Netflix's Hystrix library (2012, now in maintenance mode but the pattern is
universal) introduced the circuit breaker to software systems. The circuit
breaker is a state machine with three states:

```
CLOSED -> (N failures in window) -> OPEN
OPEN   -> (timeout expires)      -> HALF-OPEN
HALF-OPEN -> (probe succeeds)    -> CLOSED
HALF-OPEN -> (probe fails)       -> OPEN
```

In the CLOSED state, requests flow normally. The breaker counts failures
within a rolling time window (e.g., 10 failures in 60 seconds). When the
threshold is exceeded, the circuit opens.

In the OPEN state, all requests are rejected immediately without attempting
the call. This prevents wasting resources on a known-bad service and gives
the service time to recover. A timer starts; when it expires (e.g., 30
seconds), the circuit transitions to HALF-OPEN.

In the HALF-OPEN state, a single probe request is allowed through. If it
succeeds, the circuit closes (service recovered). If it fails, the circuit
re-opens and the timer resets.

The circuit breaker prevents cascade failures. Without it, service A calls
failing service B, exhausts its thread pool waiting for timeouts, and
becomes unable to serve its own requests. With a circuit breaker, service A
detects that B is down after N failures and stops trying, preserving its
own resources.

Hystrix added sophistication: configurable failure thresholds, rolling
windows with statistical sampling, fallback methods when the circuit is
open, and bulkhead isolation (see below). The core three-state machine is
what matters for hilavitkutin.

### 1.3 Bulkhead pattern

The bulkhead pattern isolates failure domains by giving each subsystem its
own resource pool. The name comes from ship construction: a hull is divided
into watertight compartments so that a breach in one does not sink the ship.

In software, this means each dependency gets its own thread pool, connection
pool, or semaphore. If service B is slow, the threads waiting on B are
confined to B's pool. Service A's pool and service C's pool are unaffected.
Without bulkheads, a single slow dependency can exhaust the shared thread
pool and block all other work.

Hystrix enforced bulkheads by assigning each command group its own thread
pool with configurable size. The Resilience4j library (Hystrix's spiritual
successor) offers both thread pool and semaphore-based bulkheads. Semaphore
bulkheads are lighter (no thread creation) but do not isolate blocking calls.

For a pipeline system like hilavitkutin, bulkheads map naturally to resource
categories. I/O-bound work (HTTP calls, file reads) gets one pool. CPU-bound
transforms get another. A stalled HTTP call does not consume a CPU worker
slot.

### 1.4 Dead letter queues

When a work item fails after exhausting all retries, it must go somewhere.
Dropping it silently loses data. Retrying forever wastes resources. A dead
letter queue (DLQ) captures permanently failed items for later inspection
and potential replay.

Messaging systems (RabbitMQ, SQS, Kafka) formalized this pattern. In SQS,
messages that exceed `maxReceiveCount` are automatically moved to a
configured DLQ. Operators can inspect the DLQ, fix the underlying issue
(deploy a bug fix, correct a schema), and replay messages back into the
main queue.

For a pipeline system, the DLQ concept maps to a failed-work-items store.
When a pipeline stage permanently fails on a morsel, the morsel is moved to
a quarantine rather than retried. The pipeline continues with the remaining
morsels. After the run, the operator inspects quarantined morsels to
diagnose the failure.

### 1.5 Idempotency

Retries are only safe if the operation is idempotent: applying it once
produces the same result as applying it N times. This is trivially true for
read operations and pure transforms. It is not trivially true for writes.

Techniques for achieving idempotency:

- **Idempotency keys.** Each request carries a unique key. The server
  deduplicates by key, returning the cached result for duplicate requests.
  Stripe's payment API uses this pattern: each charge attempt has an
  idempotency key, so retrying a failed network call does not double-charge.
- **Version-conditioned writes.** Write operations include a version
  precondition: "write X if current version is V." If the first attempt
  succeeded but the response was lost, the retry sees version V+1 and
  rejects (or returns the existing value). DynamoDB conditional puts and
  Firestore transactions use this pattern.
- **Append-only logs.** Instead of mutating state, append an event to a log.
  Duplicate appends produce duplicate entries, but the consumer deduplicates
  during log replay. Kafka consumers with exactly-once semantics use this:
  each record has an offset, and the consumer tracks which offsets it has
  processed.

For hilavitkutin's morsel pipeline, idempotency comes naturally from the
execution model. Pipeline stages are pure transforms over column data. A
morsel contains input rows; the stage produces output rows. Re-running the
stage on the same input morsel produces the same output. Stages with side
effects (file writes, HTTP calls) need explicit idempotency handling, which
is the consumer's responsibility (polka wraps file writes with atomic
rename; saalis uses idempotency keys for API calls).

## 2. Backpressure in Streaming Systems

Backpressure is the mechanism by which a slow consumer controls the rate of
a fast producer. Without backpressure, the producer overwhelms the consumer,
causing either unbounded memory growth (buffering), data loss (dropping), or
consumer crash (OOM).

### 2.1 Reactive Streams specification (JVM)

The Reactive Streams specification (2014, now part of java.util.concurrent.Flow
since JDK 9) defines demand-based flow control. The consumer calls
`subscription.request(n)` to signal that it can accept N more items. The
producer sends at most N items. When the consumer processes those items, it
requests more. The producer never sends unrequested items.

Implementations: Akka Streams (Lightbend), RxJava 2+ (Netflix/Google),
Project Reactor (Pivotal/VMware), Vert.x. All follow the same protocol:

```
Subscriber -> request(10) -> Publisher
Publisher  -> onNext(item) x 10 -> Subscriber
Subscriber -> request(5)  -> Publisher
Publisher  -> onNext(item) x 5  -> Subscriber
```

The key property is that backpressure is cooperative and explicit. The
producer knows exactly how much demand exists. There is no guessing, no
buffer management, no drop policy. The subscription protocol IS the
backpressure mechanism.

Akka Streams adds "fused" processing for operators within the same
materialization: adjacent operators share a thread and do not go through
the request/onNext protocol. This avoids the overhead of demand signaling
for in-process chains, applying the protocol only at async boundaries.

### 2.2 Kafka consumer groups

Kafka takes a different approach. The broker does not implement producer-side
backpressure. Producers write to partitions at whatever rate they choose.
Consumers pull from partitions at their own pace by committing offsets.

Backpressure manifests as consumer lag: the difference between the latest
offset in the partition and the consumer's committed offset. If the consumer
is slow, lag grows. The broker retains messages according to its retention
policy (time-based or size-based). If the consumer falls far enough behind,
messages are deleted before the consumer reads them.

This is a fundamentally different model from Reactive Streams. Reactive
Streams prevents the producer from outrunning the consumer. Kafka lets the
producer run freely and relies on the broker's retention capacity to absorb
bursts. For high-throughput data pipelines (millions of events per second),
Kafka's model is better: producers are never blocked, and consumers can
catch up during quiet periods. For memory-constrained systems, Reactive
Streams is better: no unbounded buffering.

### 2.3 TCP flow control analogy

TCP's sliding window protocol is the original backpressure mechanism. The
receiver advertises a window size (how many bytes it can accept). The
sender transmits at most that many bytes before waiting for
acknowledgments. When the receiver processes data and frees buffer space,
it advertises a larger window.

Application-level backpressure follows the same principle. The difference
is granularity: TCP operates on bytes; application-level backpressure
operates on logical units (messages, records, morsels).

### 2.4 Backpressure in a morsel pipeline

In a morsel-driven pipeline (Leis et al. 2014), work is decomposed into
fixed-size morsels dispatched to worker threads. The pipeline has multiple
stages connected by pipeline breakers (hash build, sort, aggregation). Each
stage processes morsels from an input queue and produces output into the
next stage's input.

When a downstream stage is slower than its upstream stage, morsels
accumulate at the boundary. Three options:

**(a) Unbounded buffering.** Let morsels accumulate in memory. Simple to
implement. Risk: unbounded memory growth if the slow stage is persistently
slower than the fast stage. DuckDB uses this with a memory budget; when the
budget is exceeded, spill to disk.

**(b) Backpressure (slow down upstream).** When the buffer between stages
reaches a threshold, stop dispatching morsels to the upstream stage. Workers
assigned to the upstream stage are redirected to the downstream stage (or
idle). This is what HyPer does implicitly: the dispatcher assigns workers
to pipeline stages based on progress, naturally directing workers to the
bottleneck.

**(c) Drop morsels (lossy).** Discard excess morsels. Acceptable for
real-time streaming (audio/video) where stale data has no value. Never
acceptable for data processing where every record matters.

For hilavitkutin, option (b) is correct. The scheduler controls morsel
dispatch. If it detects that a stage's output buffer is growing (or that
workers assigned to a stage are consistently completing faster than the
downstream stage can consume), it redirects workers. This is implicit
backpressure: no protocol, no demand signaling, just scheduling policy.

The morsel model's elasticity is the mechanism. Because work assignment is
per-morsel (not per-stage), the scheduler can dynamically rebalance workers
across stages within a single pipeline execution. A stage that is falling
behind gets more workers; a stage that is blocked on downstream consumption
gets fewer. This is the morsel dispatcher's natural behavior; backpressure
is a scheduling consequence, not a separate system.

## 3. Database Fault Handling

Database engines have decades of experience with fault handling in
execution engines. The patterns are directly applicable to pipeline systems.

### 3.1 DuckDB out-of-memory handling

DuckDB (Raasveldt and Muhleisen, SIGMOD 2019) handles memory pressure
through operator-level spill-to-disk. When an operator (hash join, sort,
aggregation) exceeds its memory budget, it partitions its data into
disk-backed segments and processes them incrementally.

The hash join operator illustrates the pattern: during the build phase, hash
table partitions that exceed the memory budget are written to temporary
files. During the probe phase, these partitions are loaded back one at a
time, probed, and discarded. The operator continues with degraded
performance (I/O overhead) rather than failing.

This approach has two key properties. First, the operator handles the fault
locally. The scheduler and pipeline framework do not need to know that
spilling occurred. The operator's API is unchanged; it simply runs slower.
Second, degradation is graceful. Performance scales with available memory,
not with data size. A query that runs in 1 second with 8GB of RAM runs in
10 seconds with 1GB of RAM, not in infinite seconds (OOM crash).

### 3.2 Query cancellation

Database engines use cooperative cancellation. Each operator checks a
cancellation flag at regular intervals (typically once per morsel or once
per N tuples). If the flag is set, the operator cleans up its resources
(releases memory, closes file handles) and returns a cancellation error.
The pipeline framework propagates the error upward.

There is no forced termination. Operators are trusted to check the flag
in a timely manner. If an operator enters an infinite loop or blocks on
a system call, the cancellation flag is not checked and the query hangs.
This is a trade-off: cooperative cancellation is simple and composable
(no signal handlers, no thread interruption, no cleanup races), but
depends on well-behaved operators.

PostgreSQL uses this model with `CHECK_FOR_INTERRUPTS()` macro calls
scattered throughout operator code. DuckDB uses a similar per-morsel check.
Both accept the risk of unresponsive operators in exchange for simpler
architecture.

For hilavitkutin, cooperative cancellation through the `Progress` return
value is the same pattern. A work unit that detects cancellation returns
`Progress::Failed(Error::Cancelled)`. The scheduler propagates the
cancellation to dependent work units. No forced termination, no signal
handling complexity.

### 3.3 Transaction rollback

When a database transaction fails, all changes must be reverted atomically.
Two primary mechanisms exist:

**Undo log (ARIES, Mohan et al. 1992).** Before each write, the old value
is recorded in an undo log. On rollback, the undo log is replayed in
reverse order, restoring each modified value. This is the mechanism behind
most OLTP databases (PostgreSQL, MySQL/InnoDB, Oracle). The undo log is
append-only during the transaction and replayed only on failure.

**MVCC (multi-version concurrency control).** Each write creates a new
version of the record. Readers see the version that was current at their
transaction's start time. On rollback, the new versions are simply
discarded (garbage collected). No explicit undo needed. Used by PostgreSQL
(tuple versioning), CockroachDB (intent resolution), and FoundationDB
(optimistic concurrency).

loimu's SnapshotBuffer is a variant of the undo log approach. For
sole-writer resources (one scheduling unit writes to the resource), a
snapshot is taken before execution. On fault, the snapshot is restored.
For shared-writer resources (multiple scheduling units write to the same
resource), an undo journal records each write. On fault, the journal is
replayed in reverse.

The distinction between sole-writer and shared-writer is a compile-time
optimization. The scheduler analyzes access declarations at plan time and
determines whether each resource has a single writer or multiple writers.
Sole-writer snapshot is cheaper: one memcpy at the start, one memcpy on
fault. Shared-writer undo journal is more general but has per-write
overhead (recording each old value).

## 4. Pipeline-Specific Fault Patterns

Pipeline systems face fault patterns that differ from request/response
systems. A pipeline is a DAG of stages, and failure in one stage affects
all downstream stages. The fault handling strategy must account for the
DAG structure.

### 4.1 DAG stage failure

When a node in the DAG fails, the system has four options:

**(a) Fail the entire DAG.** Simple and correct. Any failure aborts the
whole pipeline. Used when partial results are meaningless (e.g., a
compiler: a parse error means no binary output).

**(b) Skip the failed node and continue with degradation.** The failed
node's output is marked as unavailable. Downstream nodes that depend on
it are also skipped. Nodes that do not depend on the failed node continue
normally. Used when partial results have value (e.g., a data pipeline:
one bad source does not invalidate data from other sources).

**(c) Retry the node.** The scheduler re-dispatches the failed node with
backoff. Useful for transient failures (network timeout, lock contention).
Must be bounded (max retries) to avoid infinite loops.

**(d) Substitute a default/fallback result.** The failed node's output is
replaced with a default value. Downstream nodes proceed with the default.
Used when the failed node's contribution is optional (e.g., an enrichment
stage: missing enrichment data means the record passes through
un-enriched rather than being dropped).

The choice depends on the pipeline's semantics. For polka (data compiler),
option (a) is the default: a template parse error should fail the build.
For saalis (data pipeline), option (b) is appropriate: one failing
connector should not block all other connectors. Option (c) applies to
transient failures in both systems. Option (d) is rarely appropriate
because it silently masks errors.

### 4.2 Partial DAG re-execution

When a mid-pipeline node fails and then succeeds on retry, only the
failed node and its downstream dependents need re-execution. Upstream
nodes that already completed successfully do not need to re-run.

This requires that upstream outputs are still available. If the pipeline
has already freed upstream buffers (to save memory), the outputs are gone
and upstream nodes must re-run. The trade-off: holding upstream outputs
for potential re-execution costs memory; freeing them eagerly saves memory
but forces full re-execution on failure.

The version stamp model from loimu helps here. Each stage's output carries
a version stamp. If the output has not been freed, the version stamp
confirms it is still valid. If the output has been freed, the version
stamp is invalid and the stage must re-run. The scheduler can make the
retention decision based on the stage's retry policy: stages with high
retry probability keep their outputs longer.

Apache Spark's RDD model (Zaharia et al., NSDI 2012) takes the extreme
position: RDDs are lazily evaluated and recomputable. If a partition is
lost (node failure), it is recomputed from its lineage (the chain of
transformations that produced it). This is partial DAG re-execution by
default, at the cost of maintaining the full lineage graph.

### 4.3 Poison pills

A poison pill is a malformed input that crashes every worker that attempts
to process it. Without detection, the poison pill is retried indefinitely:
it fails, is re-queued, assigned to another worker, fails again, re-queued,
and so on. Each retry wastes a worker's time and pollutes error logs.

Detection heuristic: if the same morsel (identified by its position in the
input or a content hash) fails on N different workers, it is a poison pill.
The morsel is quarantined (sent to the dead letter queue) and processing
continues with the remaining morsels.

Kafka implements poison pill handling at the consumer level: a
`DeserializationException` on a record can trigger a configurable error
handler that logs the record and advances the offset. Spring Kafka's
`SeekToCurrentErrorHandler` with a `DeadLetterPublishingRecoverer` is the
canonical pattern.

For hilavitkutin, poison pill detection is a scheduler concern. The
scheduler tracks which morsels have failed and on which workers. If a
morsel exceeds a failure threshold across distinct workers, it is
quarantined. The pipeline stage is not aware of poison pill handling;
it sees a normal morsel, processes it (fails), and returns
`Progress::Failed`. The scheduler makes the quarantine decision.

### 4.4 loimu's approach

loimu handles per-scheduling-unit faults with a three-step process:

1. **Detection.** The unit panics (native) or traps (WASM). The
   scheduler catches the panic at the dispatch boundary.
2. **Rollback.** For sole-writer resources, restore from the snapshot
   buffer. For shared-writer resources, replay the undo journal in
   reverse. The resource is returned to its pre-execution state.
3. **Notification.** A `BehaviorFaulted { id, unit, reason }` signal is
   emitted into the signal bus. Remaining units in the same handler are
   skipped. The DAG continues; downstream nodes that depend on the
   faulted unit's output are skipped because the output was not written.

This approach is option (b) from the DAG failure taxonomy: skip the failed
node and continue with degradation. The signal notification enables
monitoring and diagnostic behaviors to observe faults without coupling
to the fault handling logic.

WASM units get an additional mechanism: per-unit fuel budgets. Each
scheduling unit is allocated a fuel budget (default 10M instructions).
If the unit exhausts its fuel, it is terminated (fuel trap), and the
same rollback sequence applies. This prevents infinite loops in plugin
code from hanging the scheduler.

### 4.5 Error propagation through the DAG

Should errors propagate through the DAG like data? Two approaches:

**Sentinel value propagation.** The failed node writes a sentinel value
(e.g., a "Failed" variant in the column's sum type) to its output column.
Downstream nodes see the sentinel and handle it (skip the row, propagate
the sentinel further, substitute a default). This makes error handling
explicit in every downstream stage.

**DAG-level skip propagation.** The scheduler marks the failed node's
output as unavailable. Downstream nodes are skipped entirely (not
dispatched). No sentinel values; downstream nodes never see the failure.
This keeps error handling out of the stage logic but loses the ability
for downstream stages to recover or degrade gracefully.

loimu uses DAG-level skip propagation. The faulted unit's output is not
written, so downstream units that read it find no new version (version
stamp unchanged) and skip via the early cutoff mechanism. This is elegant:
fault handling and incremental computation use the same version stamp
infrastructure.

For polka, DAG-level skip propagation is appropriate. A failed parse stage
means no AST was produced; downstream expand, resolve, and codegen stages
have nothing to work with. For saalis, sentinel value propagation may be
more appropriate: a failed connector should produce an "unavailable"
marker that downstream aggregation stages can count and report, rather
than silently skipping.

## 5. Our Progress::Retry Model

The topic document proposes a `Progress` enum returned by `execute()`:

```
Complete
Yield(BudgetExhausted)
Yield(Pending)
Yield(Retry { attempt, max, backoff })
Failed(Error)
```

This section analyzes how the `Retry` variant maps to established patterns
and what gaps remain.

### 5.1 Mapping to exponential backoff

`Retry { attempt, max, backoff }` is a direct encoding of exponential
backoff. The work unit returns `Retry { attempt: 2, max: 5, backoff:
Duration::from_millis(400) }`, and the scheduler delays re-dispatch by
400ms. The next attempt increments: `Retry { attempt: 3, max: 5, backoff:
Duration::from_millis(800) }`. When `attempt >= max`, the scheduler
converts to `Failed`.

This is clean because the work unit controls its own retry parameters. A
network-dependent unit can set max: 5 with exponential backoff. A
file-dependent unit can set max: 2 with short backoff (file lock
contention is typically brief). The scheduler does not need to know the
retry policy for each type of failure; the work unit encodes the policy
in its `Retry` return value.

The jitter component is a scheduler responsibility, not a work unit
responsibility. The work unit specifies the base backoff; the scheduler
adds jitter before scheduling the re-dispatch. This is the correct
separation: the work unit knows its failure domain (network vs file vs
compute); the scheduler knows the global scheduling state (how many other
units are also retrying, whether jitter is needed to avoid thundering
herd).

### 5.2 What is missing: circuit breakers

The current model handles per-work-unit retry. It does not handle
cross-work-unit failure correlation. Consider: 10 work units all make HTTP
calls to the same external service. The service goes down. Each unit
independently retries with backoff, consuming 10 * max_retries attempts
before all 10 reach permanent failure.

A circuit breaker at the resource level would detect the pattern: after
the 3rd unit fails talking to `Read<HttpClient>`, the circuit opens. The
remaining 7 units are immediately failed without attempting the call. When
the circuit enters HALF-OPEN (after timeout), one unit is allowed to probe.
If it succeeds, the circuit closes and the remaining units are retried.

The work unit cannot implement this because it does not know about other
work units' failures. The scheduler can, because it sees all `Retry` and
`Failed` returns and can correlate them by resource. The implementation
sketch:

```
For each resource R:
  track: failure_count_in_window, last_failure_time, state (closed/open/half-open)

When a work unit with Read<R> returns Failed:
  failure_count_in_window += 1
  if failure_count_in_window >= threshold:
    circuit[R].state = OPEN
    circuit[R].open_until = now + timeout

Before dispatching a work unit with Read<R>:
  if circuit[R].state == OPEN:
    if now > circuit[R].open_until:
      circuit[R].state = HALF_OPEN
      allow dispatch (probe)
    else:
      return Failed immediately (circuit open)
  if circuit[R].state == HALF_OPEN:
    if work unit succeeds:
      circuit[R].state = CLOSED
    else:
      circuit[R].state = OPEN
      circuit[R].open_until = now + timeout
```

This is a scheduler-level concern, not a hilavitkutin primitive. The core
library provides the `Progress::Retry` and `Progress::Failed` signals. The
consumer-layer scheduler (saalis's scheduler, polka's scheduler) builds
circuit breakers on top using resource-level failure tracking.

### 5.3 Per-work-unit vs per-service granularity

The circuit breaker must track failures at the right granularity. If
tracking at the work unit level, each unit has its own circuit breaker;
this is just max_retries (already handled by `Retry { max }`). If tracking
at the resource level, all units sharing a resource share a circuit breaker;
this catches correlated failures.

The resource type (`Read<HttpClient>`, `Read<DatabasePool>`,
`Read<FileSystem>`) is the natural grouping key. Two work units reading
the same `HttpClient` resource are likely talking to the same service. A
failure in one is evidence of a failure in the other.

But resource types can be too coarse. An `HttpClient` resource might be
used to call ten different services. A failure calling service A should
not trip the circuit for service B. The granularity depends on how
resources are modeled. If each service has its own resource type
(`Read<SteamApiClient>`, `Read<IgdbApiClient>`), per-resource circuit
breakers work. If there is a single shared `HttpClient`, the circuit
breaker needs sub-resource keying (by URL prefix, by host, by API
endpoint).

For hilavitkutin, the recommendation is: track circuit state per resource
type. Consumers that need finer granularity (per-service, per-endpoint)
should model each service as a separate resource. This keeps the circuit
breaker logic simple (one circuit per resource type) and pushes the
granularity decision to the domain (where it belongs).

### 5.4 Backpressure integration

The `Yield(BudgetExhausted)` variant is the backpressure signal. When a
work unit returns `Yield(BudgetExhausted)`, it means: "I have more work to
do but ran out of my time quantum." The scheduler re-queues the unit for
the next quantum.

If a pipeline stage consistently yields (overwhelmed by input volume), the
scheduler can slow down the upstream stage by delaying morsel dispatch.
This is implicit backpressure through yield frequency: a stage that yields
every quantum is clearly slower than its upstream, and the scheduler
reduces upstream dispatch rate to match.

The mechanism:

```
For each pipeline stage S:
  track: yield_count_in_window, complete_count_in_window

If yield_count >> complete_count for stage S:
  stage S is overwhelmed
  reduce morsel dispatch to S's upstream stage(s)
  assign more workers to S (morsel-driven elasticity)
```

This is not explicit backpressure (no demand signaling protocol). It is
scheduling policy that achieves the same effect: the producer slows down
because workers are redirected to the consumer. Morsel-driven scheduling
makes this natural: workers are assigned to the stage that needs them most.

## 6. Fault Categories and Handling Strategy

Not all failures deserve the same response. Classifying failures by
recoverability determines the appropriate handling strategy.

### 6.1 Transient failures

Transient failures are temporary and will resolve without intervention.
Examples: network timeout, rate limit (429 Too Many Requests), temporary
service unavailability (503), lock contention, DNS resolution hiccup.

Handling: retry with exponential backoff and jitter. The `Retry` variant
in `Progress` is designed for this category. The work unit sets appropriate
retry parameters based on the failure signal (rate limits often include a
`Retry-After` header that should be respected as the minimum backoff).

### 6.2 Fixable failures

Fixable failures require a change to the input or configuration, not a
retry. Examples: invalid input data (malformed JSON, schema mismatch),
type errors, validation failures, expired credentials that need rotation.

Handling: do not retry. Return `Failed(Error)` with a diagnostic message
explaining what is wrong and how to fix it. Retrying with the same input
will produce the same failure. The error message should include the failing
input (or a reference to it) and the expected format.

### 6.3 Permanent failures

Permanent failures cannot be resolved by the pipeline system. Examples:
service decommissioned (404 Not Found on a previously valid endpoint),
permission denied (403) with no credential available, hardware failure.

Handling: do not retry. Return `Failed(Error)`. The circuit breaker (if
present) should open after detecting a pattern of permanent failures from
the same resource, preventing wasted attempts by other work units.

### 6.4 saalis-specific categories

- **API rate limits (transient).** Steam, IGDB, and other game data APIs
  impose rate limits. Retry with the delay specified in the `Retry-After`
  header. Circuit breaker: if all calls to an API are rate-limited, stop
  trying and wait for the window to reset.
- **Invalid Steam ID (fixable).** A malformed Steam ID cannot be retried;
  the input must be corrected. The connector should return `Failed` with
  a diagnostic identifying the bad ID.
- **IGDB API key expired (permanent until rotated).** The key is invalid
  and no automatic rotation exists. Return `Failed`. The circuit breaker
  for the IGDB resource opens, preventing all other work units from
  wasting attempts.

### 6.5 polka-specific categories

- **File I/O errors (transient).** A locked file (editor has it open)
  resolves when the lock is released. Retry with short backoff (100ms base,
  max 3 attempts). If still locked, return `Failed` with the path and
  advice to close the editor.
- **Template parse errors (fixable).** A syntax error in a `.polka` file
  cannot be retried. Return `Failed` with span information (line, column,
  context) for the user to fix.
- **Missing source file (permanent).** A file referenced in `polka.toml`
  does not exist. No retry. Return `Failed` with the missing path and the
  reference location in `polka.toml`.

## 7. Relevance to hilavitkutin

This section maps the surveyed patterns to hilavitkutin's concrete design
decisions.

### 7.1 Progress::Retry covers the common case

The `Retry { attempt, max, backoff }` variant in `Progress` maps directly
to exponential backoff. The work unit owns its retry parameters. The
scheduler owns jitter and dispatch timing. This separation matches AWS's
recommendation and Hystrix's design: the caller (work unit) knows its
failure domain; the infrastructure (scheduler) knows the global state.

No changes are needed to the `Progress` enum to support the standard retry
pattern. The existing shape is correct.

### 7.2 Circuit breakers are consumer-layer

Circuit breakers require cross-work-unit failure correlation, which the
scheduler can implement using resource-level failure tracking. This is not
a hilavitkutin primitive; it is a scheduling policy that saalis and polka
build on top.

saalis needs circuit breakers because it talks to external services with
correlated failure modes. polka probably does not need them: file I/O
failures are rarely correlated (each file is independent). The circuit
breaker belongs in the consumer's scheduler configuration, not in the
shared library.

hilavitkutin provides the building blocks: `Progress::Retry` and
`Progress::Failed` carry the failure information, and the resource typing
(`Read<T>` / `Write<T>`) provides the grouping key. The consumer wires
these into a circuit breaker if needed.

### 7.3 Backpressure is implicit through yield frequency

In a morsel-driven pipeline, backpressure emerges from scheduling policy.
The scheduler monitors yield frequency per stage and rebalances workers
accordingly. No explicit demand signaling protocol is needed.

This is a deliberate simplification over Reactive Streams. Reactive
Streams' demand protocol is designed for unbounded, asynchronous producer-
consumer chains where the producer and consumer run independently. In a
morsel pipeline, the scheduler controls both producer and consumer
dispatch. It does not need a protocol to coordinate them; it makes the
dispatch decisions directly.

### 7.4 Error propagation through the DAG via Progress::Failed

When a work unit returns `Progress::Failed(Error)`, the scheduler marks the
unit's output as unavailable. Downstream units that depend on the output
are skipped via the version stamp early cutoff mechanism (the output's
version stamp was not incremented, so downstream units detect no change
and skip).

This is the same mechanism loimu uses for `BehaviorFaulted`. Error
propagation and incremental computation share the same infrastructure.
No separate error propagation system is needed.

### 7.5 Rollback via version stamps

loimu's sole-writer snapshot and shared-writer undo journal patterns
apply directly. hilavitkutin's scheduler determines at plan time whether
each resource has a single writer or multiple writers (from access
declarations). Sole-writer resources are snapshotted before execution.
Shared-writer resources use an undo journal.

On `Progress::Failed`, the scheduler replays the snapshot or undo journal
to restore the resource to its pre-execution state. The resource's version
stamp is also restored, so downstream units see no change and skip.

The version stamp restoration is critical. Without it, a failed write that
partially modified a resource and incremented the version stamp would cause
downstream units to run on corrupted input. Restoring the version stamp
ensures that partial writes are invisible to the rest of the DAG.

### 7.6 Poison pill quarantine

hilavitkutin should provide a morsel quarantine mechanism at the scheduler
level. The scheduler tracks morsel failure counts. When a morsel exceeds a
threshold across distinct workers, it is quarantined (removed from the
dispatch queue and recorded for later inspection). The pipeline stage is
not involved in quarantine decisions.

The quarantine threshold should be configurable per pipeline stage. A stage
processing untrusted external input (user-provided templates, API
responses) should have a lower threshold than a stage processing validated
internal data.

### 7.7 Summary of recommendations

| Pattern | Ownership | Implementation |
|---------|-----------|----------------|
| Exponential backoff | hilavitkutin (Progress::Retry) | Work unit returns retry params; scheduler adds jitter and delays |
| Circuit breakers | Consumer layer (saalis, polka) | Scheduler tracks per-resource failure rates; trips circuit after threshold |
| Bulkhead isolation | Consumer layer | Separate worker pools for I/O-bound vs CPU-bound stages |
| Dead letter queue | Consumer layer | Quarantined morsels stored for inspection |
| Backpressure | hilavitkutin (implicit) | Scheduler monitors yield frequency; rebalances workers |
| Rollback | hilavitkutin (core) | Sole-writer snapshot or shared-writer undo journal via version stamps |
| Poison pill detection | hilavitkutin (scheduler) | Track per-morsel failure count; quarantine after threshold |
| Cooperative cancellation | hilavitkutin (Progress::Failed) | Work unit checks cancellation flag; returns Failed(Cancelled) |

The pattern is consistent: hilavitkutin owns the primitives (Progress enum,
version stamps, scheduling dispatch) and the mechanisms that require
DAG-level visibility (poison pill detection, rollback, implicit
backpressure). Consumer-layer systems own the policies that require
domain-specific knowledge (circuit breaker thresholds, bulkhead pool sizes,
DLQ storage backends, fault category classification).
