## 1. Fixed-point type system

### Core types

loimu's numeric foundation is two const-generic fixed-point types:

```rust
#[repr(transparent)]
pub struct UFixed<const I: u8, const F: u8>(u64);

#[repr(transparent)]
pub struct IFixed<const I: u8, const F: u8>(i64);
```

`UFixed` is unsigned. `I` is the number of integer bits, `F` is the number of
fractional bits. The total logical width is `I + F`. The physical container is
always the next power-of-two that fits: a `UFixed<6,2>` occupies 8 logical bits
but lives in a `u64`. The `repr(transparent)` guarantee means the in-memory
layout is identical to the inner integer, which matters for FFI, transmute
safety, and columnar storage.

`IFixed` is signed. Layout is 1 sign bit + `I` integer bits + `F` fractional
bits. The sign bit is implicit in the two's complement representation of `i64`,
so the const generics only specify `I` and `F`. `IFixed<1,14>` gives 1 sign +
1 integer + 14 fractional = 16 logical bits.

Both types define `ZERO` and `ONE` constants. `ONE` is `1u64 << F` (or
`1i64 << F` for signed), which places the value 1.0 at the correct bit
position. This is the standard fixed-point encoding: the integer value stored
in the container is the real value multiplied by `2^F`.

### FixedPointMeta

```rust
pub struct FixedPointMeta {
    pub integer_bits: u8,
    pub fractional_bits: u8,
    pub is_signed: bool,
    pub total_bits: u8,
}
```

Produced by a `const fn meta()` on both types. This allows runtime code
(serialization, storage, debug formatting) to inspect the fixed-point layout
without const-generic monomorphization. The storage layer uses `total_bits` to
determine packing strategy; the debug formatter uses `fractional_bits` to
display the decimal value.

### Arithmetic semantics

All arithmetic is saturating. This is a deliberate design choice: fixed-point
overflow in a UI framework or data pipeline should clamp to min/max rather than
wrap. Wrapping arithmetic in fixed-point produces values that are not merely
wrong but catastrophically wrong (a color channel wrapping from 0.99 to 0.0,
for instance).

**Addition and subtraction** are straightforward saturating operations on the
underlying integer.

**Multiplication** widens to the next-larger integer type (u128 for UFixed,
i128 for IFixed), performs the multiply, shifts right by `F` to re-normalize,
then clamps to the representable range. The widening prevents intermediate
overflow: `UFixed<8,8>` * `UFixed<8,8>` produces a 32-bit intermediate in a
128-bit container, with headroom to spare.

**Division** saturates on divide-by-zero (returns MAX or MIN depending on sign)
rather than panicking. The numerator is shifted left by `F` bits before
dividing to maintain fractional precision.

The standard operators (`Add`, `Sub`, `Mul`, `Div`) delegate to the saturating
variants. There are no wrapping or checked alternatives exposed. If you need
wrapping semantics, you work with the raw bits via `from_bits` / `to_bits`.

### Const-generic design rationale

The const-generic approach (`UFixed<const I: u8, const F: u8>`) rather than a
trait-based approach (`impl FixedPoint for MyType`) gives several properties:

1. **Zero-cost abstraction.** The compiler monomorphizes each combination. A
   `UFixed<0,8>` is a distinct type from `UFixed<2,8>` at compile time, with
   all operations specialized to those specific bit counts.
2. **Type-level documentation.** Reading `Ratio` = `UFixed<0,8>` tells you
   immediately: unsigned, zero integer bits (range 0..1), 8-bit fractional
   precision.
3. **Compile-time safety.** You cannot accidentally add a `Ratio` to a `Coord`
   without an explicit conversion, because they are different types.
4. **Alias composability.** Semantic aliases are just type aliases, not
   newtypes. No boilerplate forwarding impls. The full arithmetic API is
   inherited.

The downside is monomorphization cost: each `(I, F)` combination generates its
own code. loimu accepts this because the set of actually-used combinations is
small (the alias taxonomy below) and the operations are simple enough that
LLVM inlines most of them anyway.

## 2. Alias taxonomy

loimu defines three categories of type aliases over UFixed/IFixed. Together
they form a complete numeric vocabulary.

### Semantic aliases

These are domain-specific names for commonly used fixed-point configurations.
Each has a specific interpretation:

| Alias | Type | Range | Purpose |
|-------|------|-------|---------|
| `Ratio` | `UFixed<0,8>` | 0.0 .. 1.0 | Normalized unsigned value. Color channels, opacity, blend factors, progress bars. |
| `Factor` | `UFixed<2,8>` | 0.0 .. 4.0 | Unsigned multiplier. Scale factors, speed multipliers, gain. 2 integer bits give range up to 4x. |
| `Norm` | `IFixed<1,14>` | -1.0 .. 1.0 | Signed normalized value. Audio samples, direction components, blend weights. 14-bit fractional gives ~0.00006 precision. |
| `Coord` | `IFixed<16,16>` | -32768.0 .. 32768.0 | Spatial coordinate with sub-pixel precision. 16-bit fractional gives 1/65536 pixel resolution. |
| `Angle` | `UFixed<9,7>` | 0.0 .. 512.0 | Degrees. 9 integer bits cover 0-360 with headroom. 7-bit fractional gives ~0.78 degree precision. |
| `Bool` | `UFixed<1,0>` | 0 or 1 | 1-bit boolean. Aliases to `uint1`. |
| `Nibble` | `UFixed<4,0>` | 0 .. 15 | 4-bit unsigned integer. |

The rationale for `Ratio` being 8-bit fractional: color channels are the
primary consumer, and 8-bit precision (256 levels) matches sRGB. No point
paying for 16-bit precision when the output is 8-bit anyway.

`Coord` at `IFixed<16,16>` mirrors the common fixed-point convention in 2D
graphics (used by FreeType, Cairo, Skia). 16 integer bits cover typical screen
coordinates; 16 fractional bits give sub-pixel precision for anti-aliasing and
smooth animation.

`Angle` using `UFixed<9,7>` is a design choice: degrees rather than radians,
unsigned because angles are conventionally 0-360. The 9 integer bits can
represent up to 512, which accommodates angles beyond 360 without wrapping
logic. The geometry crate (loimu-geom) redefines `Angle` differently; see
section 5.

### Integer aliases (F=0)

When fractional bits are zero, the fixed-point type degenerates to a
sub-word-width integer:

- **Unsigned:** `uint1` through `uint28`, defined as `UFixed<1,0>` through
  `UFixed<28,0>`.
- **Signed:** `int2` through `int28`, defined as `IFixed<1,0>` through
  `IFixed<27,0>`. (Signed needs at least 2 total bits: 1 sign + 1 integer.)

These fill the gap between Rust's built-in integer types (u8, u16, u32, u64)
and the actual bit widths needed by domain logic. A `uint4` (nibble) stores 16
values; a `uint12` stores 4096 values. In columnar storage, a column of
`uint4` values packs 16 values per u64, which is 16x denser than using `u8`.

The storage layer uses the declared bit width to choose packing strategy. A
column of `uint1` uses bitset packing (64 values per u64). A column of
`uint12` uses 12-bit packing with shift/mask extraction. This is the primary
motivation for sub-word integers: storage density in the columnar engine.

### Shaped aliases (mixed integer/fractional)

These provide a vocabulary of balanced, precision-biased, and range-biased
fixed-point types at each total width:

**Wide (75% integer, 25% fractional) — range-biased:**

| Alias | Type | Total bits |
|-------|------|-----------|
| `wide_f8` | `IFixed<6,2>` | 8 |
| `wide_f16` | `IFixed<12,4>` | 16 |
| `wide_f32` | `IFixed<24,8>` | 32 |
| `wide_f64` | `IFixed<48,16>` | 64 |

**Precise (25% integer, 75% fractional) — precision-biased:**

| Alias | Type | Total bits |
|-------|------|-----------|
| `prec_f8` | `IFixed<2,6>` | 8 |
| `prec_f16` | `IFixed<4,12>` | 16 |
| `prec_f32` | `IFixed<8,24>` | 32 |
| `prec_f64` | `IFixed<16,48>` | 64 |

**Standard (50% integer, 50% fractional) — balanced:**

| Alias | Type | Total bits |
|-------|------|-----------|
| `std_f8` | `IFixed<4,4>` | 8 |
| `std_f16` | `IFixed<8,8>` | 16 |
| `std_f32` | `IFixed<16,16>` | 32 |
| `std_f64` | `IFixed<32,32>` | 64 |

All shaped aliases are signed by default. Unsigned variants use a `u_` prefix:
`u_wide_f8` = `UFixed<6,2>`, etc.

The naming convention encodes two properties: the shape (wide/prec/std) tells
you the integer-to-fractional ratio, and the `_f<N>` suffix tells you the
total width. `wide_f32` is a 32-bit wide-shape: lots of range, limited
fractional precision. `prec_f32` is a 32-bit precise-shape: limited range,
high fractional precision.

The rationale for three shapes at four widths: most fixed-point code needs to
choose between range and precision at a given storage budget. Rather than
forcing users to compute `(I, F)` pairs, the shaped aliases express the common
trade-offs directly. A physics simulation might use `prec_f32` for
accumulated sub-pixel error; a map coordinate system might use `wide_f32` for
large world coordinates with coarse fractional precision.

## 3. Conversion trait hierarchy

loimu defines six conversion traits that form a hierarchy from lossless to
lossy, from fixed-to-primitive to fixed-to-fixed.

### FromPrimitive / ToPrimitive

```rust
trait FromPrimitive<P> { fn from_primitive(p: P) -> Self; }
trait ToPrimitive<P> { fn to_primitive(&self) -> P; }
```

These convert between fixed-point types and Rust's standard numeric types
(f32, f64, u8, u16, u32, u64, i8, i16, i32, i64). `FromPrimitive` is
saturating: values outside the representable range clamp to min/max.
`ToPrimitive` truncates fractional bits that cannot be represented in the
target type.

These are the primary entry/exit points between the fixed-point world and
external data (user input, file I/O, interop with floating-point libraries).

### TryFromPrimitive

```rust
trait TryFromPrimitive<P> { fn try_from_primitive(p: P) -> Option<Self>; }
```

Fallible conversion from standard types. Returns `None` if the value is
outside the representable range, rather than clamping. Use this when
out-of-range input is an error condition rather than a clamp condition.
Validation at construction boundaries (parsing user input, deserializing
config) should use `TryFromPrimitive`; internal arithmetic should use
`FromPrimitive` (saturating) to avoid option-chaining in hot paths.

### FromFixed

```rust
trait FromFixed<T> { fn from_fixed(t: T) -> Self; }
```

Lossless conversion between fixed-point types. Only implemented when the
target type has equal or greater width in both integer and fractional
components. `FromFixed<UFixed<0,8>>` for `UFixed<2,8>` is valid (target has
more integer bits). `FromFixed<UFixed<2,8>>` for `UFixed<0,8>` is not
implemented (would lose integer bits).

The compiler enforces losslessness: if the conversion would lose information,
it does not compile. No runtime check needed.

### TruncateFrom

```rust
trait TruncateFrom<T> { fn truncate_from(t: T) -> Self; }
```

Lossy conversion between fixed-point types. Implemented for any source/target
pair. Discards bits that do not fit: high integer bits are clamped (saturated),
low fractional bits are truncated. This is the fixed-to-fixed equivalent of
`as` casting in Rust: always succeeds, may lose information.

The separation between `FromFixed` (lossless, compile-time enforced) and
`TruncateFrom` (lossy, always available) forces the programmer to explicitly
acknowledge precision loss. A function that takes `impl FromFixed<Ratio>` can
only be called with types that are at least as precise as `Ratio`. A function
that takes `impl TruncateFrom<Coord>` accepts any fixed-point type and handles
the truncation.

### MulInto

```rust
trait MulInto<Rhs, Target> { fn mul_into(self, rhs: Rhs) -> Target; }
```

Cross-type multiplication with explicit target type. The result of multiplying
two fixed-point types depends on the desired output precision: `Ratio * Factor`
could produce a `Ratio`, a `Factor`, or a `UFixed<2,16>`. `MulInto` makes the
target explicit:

```rust
let r: Ratio = ratio.mul_into::<Ratio>(factor);
```

This avoids the ambiguity of `Mul` output types. The standard `Mul` impl
returns `Self` (same type as the left operand). `MulInto` allows the caller to
choose a different output type, with the trait implementation handling the
intermediate widening and re-normalization.

### Trait relationships

The six traits form a lattice of increasing lossyness:

```
FromFixed (lossless, compile-time)
    |
TruncateFrom (lossy, always available)
    |
FromPrimitive / ToPrimitive (cross-domain, saturating)
    |
TryFromPrimitive (cross-domain, fallible)
```

`MulInto` is orthogonal; it handles a specific operation (multiplication) where
the output type differs from the input types.

The design principle: make the easy thing safe. `FromFixed` is the easiest
conversion to use and the safest. `TruncateFrom` requires an explicit method
call that signals "I know I'm losing precision." `FromPrimitive` is saturating
because out-of-range primitives should clamp, not panic. `TryFromPrimitive`
exists for the minority case where clamping is wrong.

## 4. Arithmetic strategy system

### The problem

Different code paths have different precision/performance trade-offs. A
color blending loop processes millions of pixels per frame and can tolerate
truncation to SIMD lane width. A financial calculation processes one value and
needs every fractional bit. loimu's strategy system makes this trade-off
explicit and configurable.

### Strategy trait and types

```rust
trait ArithmeticStrategy: 'static {}

struct Hot;    // lane-capped, may truncate for SIMD
struct Cold;   // full-width, non-critical path
struct Warm;   // runtime configurable (default)
struct Precise; // full-width, critical path
```

Each strategy is a zero-sized type that implements `ArithmeticStrategy`. They
carry no data; they exist only as type-level markers.

**Hot:** Truncates values to fit SIMD lane widths. A `UFixed<16,16>` in a Hot
context might be processed as `UFixed<8,8>` to fit four values in a 128-bit
NEON lane or eight values in a 256-bit AVX2 lane. The truncation is lossy but
the throughput gain is worth it for bulk operations (color blending, particle
updates, geometry transforms).

**Cold:** Full-width computation on code paths that run infrequently. No
truncation, no SIMD. Used for initialization, configuration parsing, one-time
calculations. The "cold" annotation also serves as a hint to the compiler's
code layout (unlikely paths).

**Warm:** Runtime-configurable. The default strategy. Whether Warm behaves
like Hot or Cold depends on a runtime configuration resource:

```rust
enum FixedPointMode { PreciseMode, FastMode }

struct FixedPointConfig {
    mode: FixedPointMode,
}
```

Default is `PreciseMode` (Warm behaves like Cold). Switching to `FastMode`
makes Warm behave like Hot. This lets users toggle precision/performance
globally without recompiling.

**Precise:** Full-width, critical path. Like Cold, but semantically distinct:
Precise means "this calculation must be exact," not "this calculation is
infrequent." Physics integration, financial rounding, coordinate transforms
that feed into hit-testing. The strategy system never truncates Precise
operations regardless of runtime configuration.

### DSL integration

loimu's behavior DSL (its equivalent of polka-dots' `.polka` files) uses
annotations to assign strategies:

```
#[hot]     → Hot strategy
#[cold]    → Cold strategy
#[warm]    → Warm strategy (default, can be omitted)
#[precise] → Precise strategy
```

These annotations are resolved at compile time. The code generator selects
the appropriate arithmetic implementation based on the strategy. For Hot
paths, it emits SIMD intrinsics (when available) with truncated operands. For
Cold/Precise paths, it emits scalar operations at full width.

### Interaction with storage

The strategy system connects to the storage layer's bit-packing. Storage
always stores values at their declared width (a `UFixed<16,16>` is stored as
32 bits in a column). Computation unpacks to the strategy-determined width:

- Hot: unpack to SIMD lane width (may truncate)
- Cold/Warm(Precise)/Precise: unpack to container width (u64/i64, no truncation)
- Warm(Fast): unpack to SIMD lane width (like Hot)

This separation between storage width and computation width is fundamental.
You do not lose stored precision by choosing Hot for a compute pass; the
stored values remain at full width. Only the in-flight computation truncates.

## 5. Geometry and composite types

### Vector and matrix types

```rust
struct Vec2<T> { x: T, y: T }
struct Vec3<T> { x: T, y: T, z: T }
struct Vec4<T> { x: T, y: T, z: T, w: T }
```

Generic over the element type. Component-wise `Add` and `Sub`, scalar `Mul<T>`.
No dot product, cross product, or normalization; those are higher-level
operations that depend on the element type's arithmetic properties.

```rust
struct Mat2<T> { cols: [Vec2<T>; 2] }
struct Mat3<T> { cols: [Vec3<T>; 3] }
struct Mat4<T> { cols: [Vec4<T>; 4] }
```

Column-major layout. No matrix multiplication or inversion implemented at
the primitives level. loimu-geom provides `Affine` (see below) with
composition; full matrix ops are deferred to the rendering layer.

The rationale for column-major: GPU APIs (Vulkan, Metal, WebGPU) expect
column-major matrices. Column-major also means each column is a contiguous
`Vec<T>`, which simplifies SIMD loading.

### Transform and color (primitives crate)

```rust
struct Transform2D {
    translation: Vec2<Coord>,
    rotation: Angle,
    scale: Vec2<Factor>,
}
```

A 2D affine transform decomposed into translation, rotation, and scale. Uses
the semantic aliases from the primitives crate: `Coord` for position, `Angle`
for rotation, `Factor` for scale. This is a high-level representation; the
geometry crate provides `Affine` for the matrix form.

```rust
struct Color {
    r: Ratio,
    g: Ratio,
    b: Ratio,
    a: Ratio,
}
```

Linear color space. Each channel is a `Ratio` (`UFixed<0,8>`), giving 256
levels per channel. Constants: `TRANSPARENT` (0,0,0,0), `WHITE` (1,1,1,1),
`BLACK` (0,0,0,1). The `SrgbConvert` trait handles gamma correction for
display output.

Linear color space is the correct choice for computation: blending, lighting,
and interpolation produce correct results only in linear space. sRGB is a
display encoding, not a computation space.

### Curve types (primitives crate)

```rust
struct CurvePoint<T> { t: f32, value: T }
```

A single point on a curve. The time parameter `t` is deliberately `f32`, not a
fixed-point type. Rationale: curve evaluation involves division and
interpolation where floating-point's dynamic range is genuinely useful, and
the precision requirements are low (animation timing does not need sub-pixel
accuracy in the time domain).

```rust
struct Curve<T, const CAP: usize> { /* ... */ }
struct StaticCurve<T, const N: usize> { /* ... */ }
```

`Curve` is dynamic with a compile-time capacity bound. Fixed-size array
internally; unused slots duplicate the last defined point. This enables
branchless evaluation: the binary search or linear scan never encounters
uninitialized memory, and evaluating past the last point returns the last
value naturally.

`StaticCurve` is compile-time known. All points are const. Used for easing
functions and other curves that never change.

The `CurveEval` trait unifies both:

```rust
trait CurveEval {
    fn evaluate(&self, t: f32) -> T;
    fn len(&self) -> usize;
    fn is_empty(&self) -> bool;
}
```

Interpolation is linear. The `t` parameter is clamped to 0..1. Evaluation
converts the fixed-point values to f32, interpolates, and converts back. This
round-trip through f32 is the one place where loimu's fixed-point purity
breaks, and it is intentional: linear interpolation in fixed-point requires
intermediate widening that the const-generic system cannot express generically.

`EasingCurve` is aliased to `Curve<Ratio, 8>`: an easing function is a curve
from `Ratio` (0..1 normalized value) with up to 8 control points.

### Geometry types (loimu-geom, separate crate)

The geometry crate redefines several semantic aliases with different parameters:

| Primitives alias | Geom alias | Type | Rationale |
|-----------------|------------|------|-----------|
| `Coord` = `IFixed<16,16>` | `Coord` = `IFixed<16,16>` | Same | Shared convention |
| `Factor` = `UFixed<2,8>` | `Factor` = `IFixed<2,14>` | Different | Geom factors can be negative (flip), need more precision |
| `Angle` = `UFixed<9,7>` | `Angle` = `IFixed<16,16>` | Different | Geom uses radians (signed, high precision) |
| (none) | `Dimension` = `UFixed<16,16>` | New | Unsigned size/extent |
| (none) | `ColorChannel` = `UFixed<0,8>` | New (= Ratio) | Domain-specific name |
| (none) | `GradientPosition` = `UFixed<0,16>` | New | Higher precision for gradient stops |

The divergence between primitives and geom aliases is notable. `Factor` in
primitives is unsigned (scale factors are positive); in geom it is signed
(affine transforms can mirror). `Angle` in primitives uses degrees with
coarse precision (UI rotation widgets); in geom it uses radians with high
precision (trigonometric computation).

This divergence is a design tension. arvo will need to decide whether to
unify these or maintain separate alias sets per domain.

**Core geometry types:**

- `Point { x: Coord, y: Coord }` — a position in 2D space
- `Size { width: Dimension, height: Dimension }` — an extent (unsigned)
- `Rect { origin: Point, size: Size }` — axis-aligned rectangle with
  `contains`, `intersects`, `union`, `intersection`, `inset` operations
- `Affine([IFixed<16,16>; 6])` — 3x2 affine matrix in row-major form.
  Factory methods: `translate`, `rotate`, `scale`, `IDENTITY`. Composition
  via `Mul` (matrix multiply). This is the concrete matrix type that
  `Transform2D` decomposes into.
- `GradientStop { position: GradientPosition, color: Color }` — color stop
  for gradient rendering

## 6. ID system

### Design

loimu's ID system provides type-safe, Copy, cheap-to-compare identifiers.
The core trait:

```rust
trait Identifier: Copy + Eq + Hash + Debug + Display {}
```

All IDs are `Copy` (they are u64-sized or smaller). All IDs support equality,
hashing (for use as map keys), and debug/display formatting.

### The define_id! macro

`define_id!` is the only way to create ID types. It generates a newtype around
`u64` with the `Identifier` trait impl and the requested generation mode.

Four modes:

**Default (interned string):**
```rust
define_id!(WidgetId);
```
Created from string names. The string is interned (stored once in a global
table) and mapped to a u64 index. Hot-path comparison is always u64 equality;
the string is only needed for debug formatting and serialization.

**Namespace (prefixed interned):**
```rust
define_id!(ComponentId, namespace = "component");
```
Like default, but the interned string is prefixed with the namespace. This
prevents collisions between IDs from different domains: `"component:button"`
and `"widget:button"` intern to different u64 values.

**Sequential (atomic counter):**
```rust
define_id!(EntityId, sequential);
```
IDs are assigned by an `AtomicU64` counter. No string interning. Used for
entities and other objects that are created at runtime and do not have stable
names. The counter is global and monotonically increasing; IDs are unique but
not reusable.

**UUID (interned UUID):**
```rust
define_id!(SessionId, uuid);
```
UUIDs are generated and then interned to u64 indices. Gives globally unique
IDs (for distributed systems or persistent storage) with the same hot-path
u64 comparison cost.

### IdGen and IdMap

`IdGen` is a standalone atomic counter for sequential ID generation.
`IdMap<K, V>` (in loimu-storage, not loimu-primitives) is a typed `HashMap`
wrapper that enforces the key is an `Identifier`. It provides no additional
functionality over HashMap but serves as documentation and prevents accidental
use of raw u64 keys.

### Hot-path optimization

The critical design constraint: ID comparison on hot paths must be a single
u64 equality check. String interning achieves this for named IDs. Sequential
and UUID modes achieve it trivially (already u64). The interning table is
write-once, read-many: strings are interned at creation time and never removed.
The table grows monotonically, which means no locking on the read path (the
index is stable once assigned).

## 7. Storage integration

### Bit-packing in columns

loimu's columnar storage (SlabColumn) stores fixed-point values at their
declared bit width, not their container width. A column of `uint4` values packs
values into 4-bit slots, not 8-bit or 64-bit slots. This density is the
primary motivation for sub-word integer types.

The packing strategy is determined at compile time from a table that maps bit
width to:

- **Packing strategy:** how values are packed into u64 words (bitshift + mask
  for sub-word, direct for word-aligned)
- **Unpack kernel:** the SIMD or scalar routine that extracts values from
  packed storage into computation-width registers
- **Values per cache line:** how many values fit in one 64-byte cache line
  (or 128-byte on Apple Silicon). This feeds into morsel sizing; a morsel
  should be a multiple of this value to avoid partial cache line loads.
- **Chunk size:** the granularity of column allocation. Columns grow in chunks,
  and the chunk size is chosen to be a multiple of the cache line value count.

### SIMD unpack

Computation operates on unpacked values at the next power-of-two SIMD lane
width. The unpack kernel converts from packed storage to computation registers:

- `uint1` (1-bit): bitset extraction, 64 values per u64 word
- `uint4` (4-bit): shift-and-mask, 16 values per u64 word
- `uint8` (8-bit): byte extraction, 8 values per u64 word
- `uint12` (12-bit): shift-and-mask with carry across word boundaries
- `uint16` (16-bit): halfword extraction, 4 values per u64 word
- `uint32` (32-bit): word extraction, 2 values per u64 word
- `uint64` (64-bit): direct read, 1 value per u64 word

For SIMD computation, the unpack kernel widens values to the lane width. A
column of `uint8` values unpacked for AVX2 processing becomes a vector of 32
u8 lanes (256 bits / 8 bits per lane). A column of `uint16` values becomes a
vector of 16 u16 lanes.

The Hot arithmetic strategy may further truncate: a `UFixed<16,16>` (32-bit
logical width) might be truncated to 16 bits to double the SIMD throughput.
This truncation happens at the unpack stage, not in storage.

### Compile-time table

The bit-width-to-strategy mapping is a compile-time table, not a runtime
dispatch. Each `UFixed<I, F>` type has a const-evaluable total bit width
(`I + F`), and the storage layer selects the packing strategy via const
generics or a const-fn lookup. This means adding a new bit width does not
require modifying the storage layer; the table is exhaustive over the range
1..64.

## 8. Relevance to arvo

arvo is the numeric primitives crate for hilavitkutin. It serves polka-dots,
saalis, and loimu. The following analysis identifies what to port from loimu,
what to redesign, and what to add.

### What to port directly

**UFixed/IFixed core types.** The const-generic fixed-point design is sound and
well-tested in loimu. The `repr(transparent)` over u64/i64 container, the
saturating arithmetic, the `from_bits`/`to_bits` escape hatch, and the `meta()`
introspection all carry over unchanged.

**Integer aliases (uint1..uint28, int2..int28).** These are mechanical type
aliases. Port them verbatim.

**Shaped aliases (wide/prec/std at f8/f16/f32/f64).** The three-shape, four-width
grid with signed default and `u_` prefix for unsigned is a clean design. Port
it.

**Conversion traits (all six).** The hierarchy from `FromFixed` (lossless) through
`TruncateFrom` (lossy) to `FromPrimitive`/`ToPrimitive` (cross-domain) is
well-designed. Port the trait definitions and the blanket impls.

**FixedPointMeta.** The runtime introspection struct is needed by the storage
layer and debug formatting. Port it.

**define_id! macro with four modes.** The ID system is project-agnostic. Port
it to arvo. The `IdMap` wrapper stays in the storage layer (hilavitkutin, not
arvo) since it depends on `HashMap`.

### What to redesign

**Semantic aliases.** loimu defines `Ratio`, `Factor`, `Coord`, `Angle` etc.
in the primitives crate, then loimu-geom redefines some with different
parameters. This creates confusion: which `Factor` do you import? arvo should
define the semantic aliases once, with parameters that work for all three
projects. Decisions needed:

- `Factor`: signed or unsigned? loimu-geom needs signed (negative scale =
  mirror). polka-dots probably does not need `Factor` at all. saalis does not
  need it. Decision: signed, following the more general case.
- `Angle`: degrees or radians? loimu primitives uses degrees (UI), loimu-geom
  uses radians (math). arvo should pick one canonical representation. Radians
  are the mathematically correct choice; degree conversion is a display
  concern. Decision: radians, `IFixed<16,16>`, matching loimu-geom.
- Whether `Coord`, `Ratio`, `Norm` should live in arvo or in domain crates.
  These are semantic names for specific `(I, F)` combinations. If arvo defines
  them, all three projects share the same types. If domain crates define them,
  each project can tune the parameters. Recommendation: arvo defines the types,
  domain crates define aliases only if they need different parameters.

**Arithmetic strategy system.** The four-strategy model (Hot/Cold/Warm/Precise)
and the runtime `FixedPointConfig` are loimu-specific. polka-dots does not have
a behavior DSL with `#[hot]` annotations. saalis does not have SIMD-optimized
column processing (its data lives in SQLite). arvo should define the strategy
trait and the four marker types, but the DSL integration and the runtime config
resource stay in loimu's layer.

The compile-time strategy selection (which operations to emit for Hot vs
Precise) belongs in hilavitkutin-build, not arvo. arvo provides the types;
hilavitkutin-build provides the codegen.

**Curve types.** The f32 round-trip in curve evaluation is pragmatic but
impure. arvo should provide curves, but the interpolation implementation
should be generic over an `Interpolate` trait rather than hard-coded to
f32 round-trip. This lets loimu use f32 interpolation (fast, good enough for
animation) while polka-dots could use fixed-point interpolation (if it ever
needs curves, which is unlikely).

### What to add

**FastFloat.** loimu's primitives are exclusively fixed-point. arvo needs to
support IEEE 754 floats as first-class citizens for projects (or subsystems)
that need dynamic range. A `FastFloat` newtype over f32/f64 with the same
trait vocabulary (`FromPrimitive`, `ToPrimitive`, `Interpolate`) would let code
be generic over "any arvo numeric type" without caring whether the underlying
representation is fixed or floating.

The key constraint: `FastFloat` must NOT be `repr(transparent)` over f32
indiscriminately. It should enforce specific semantics (no NaN, no infinity,
or saturating-to-finite) so that arvo code can assume total ordering. A
`FastFloat` that can be NaN breaks `Eq` and `Ord`, which breaks use as map
keys, sort keys, and comparison operands.

**Composable trait bounds.** loimu's conversion traits are implemented
pairwise. arvo should add blanket trait bounds that express composability:
"any type that is `FromFixed<Ratio>` is also `FromFixed<Bool>`" (because
`Bool` fits in `Ratio`). This requires careful trait hierarchy design to avoid
conflicting blanket impls, but it eliminates boilerplate where-clauses in
generic code.

**no_std support.** loimu targets desktop/mobile with full std. arvo targets
three projects, one of which (polka-dots) might run in constrained
environments (WASM, embedded). The fixed-point types themselves are inherently
no_std (no allocation, no I/O). The ID system's string interning requires
`alloc` (global interning table). The strategy system's runtime config
requires a global resource. arvo should be structured as:

- `arvo-core`: `no_std`, no `alloc`. Fixed-point types, aliases, conversion
  traits, FixedPointMeta, vector/matrix types.
- `arvo`: re-exports `arvo-core` plus `alloc`-dependent features (ID system,
  curves with dynamic capacity, strategy runtime config).

This split ensures that the numeric primitives are usable in any context,
while the convenience features are available when the allocator is.

**Compile-time validation.** loimu does not validate that `I + F <= 64` at
compile time (it relies on runtime assertions in from_bits). arvo should use
const-generic bounds or a `const { assert!(I + F <= 64) }` block to catch
invalid combinations at compile time. Rust's const evaluation is stable enough
for this.

**Display and serialization.** loimu's primitives have `Debug` but limited
`Display` and no `serde` support. arvo should provide `Display` (formatted as
decimal with appropriate precision) and optional `serde` support behind a
feature flag. Serialization format: the raw bits as an integer, with the
`FixedPointMeta` available for human-readable rendering.

### What to leave out

**loimu-geom types (Rect, Affine, GradientStop).** These are domain-specific
to UI/rendering. polka-dots and saalis do not need axis-aligned rectangles or
affine transforms. loimu-geom remains a loimu-only crate.

**Color type.** Same reasoning. Color is a rendering primitive. If saalis or
polka-dots ever need colors (unlikely), they can define their own using arvo's
`Ratio` type.

**SrgbConvert.** Display color space conversion is rendering-specific.

**The full storage integration.** The bit-packing table, unpack kernels, and
SIMD dispatch live in hilavitkutin's storage layer, not arvo. arvo provides
`FixedPointMeta` (which gives the storage layer the bit width) and the types
themselves (which the storage layer packs/unpacks). The storage layer owns the
packing strategy selection.

### Summary of arvo's scope

arvo provides:
- `UFixed<I, F>`, `IFixed<I, F>` with saturating arithmetic
- `FixedPointMeta` for runtime introspection
- Integer aliases (uint1..uint28, int2..int28)
- Shaped aliases (wide/prec/std at f8/f16/f32/f64, signed and unsigned)
- Semantic aliases (Ratio, Factor, Norm, Coord, Angle) with unified parameters
- Conversion traits (FromFixed, TruncateFrom, FromPrimitive, ToPrimitive,
  TryFromPrimitive, MulInto)
- Vec2/Vec3/Vec4 and Mat2/Mat3/Mat4 (generic, minimal ops)
- ArithmeticStrategy trait + Hot/Cold/Warm/Precise markers
- define_id! macro with four modes (behind `alloc` feature for interned modes)
- FastFloat newtype (total-ordering f32/f64 wrapper)
- Curve types (behind `alloc` feature for dynamic curves)
- `no_std` core with `alloc`-gated features
- Compile-time validation of `I + F <= 64`
- Optional serde support behind feature flag

arvo does not provide:
- Geometry types (Rect, Affine, Point, Size) — domain crate
- Color / sRGB conversion — domain crate
- Bit-packing / unpack kernels — hilavitkutin storage layer
- DSL annotations / codegen — hilavitkutin-build
- Runtime strategy configuration — consumer crate
- IdMap — hilavitkutin storage layer
