## 1. Architecture Evolution Across Design Rounds

loisto went through seven formal design rounds (Round 0 through Round 6) between
2026-02-24 and 2026-02-25, followed by a consolidated review and new
architecture document on 2026-02-26, a crate taxonomy on 2026-02-28, and a
taxonomy refinement on 2026-03-01. The evolution shows a project that started
with pragmatic component selections and progressively replaced each one with
purpose-built alternatives as the design clarified what the system actually
needed.

### Rounds 0-1: Foundation

The initial rounds established the base platform: AMD AM4 as the hardware target
(treating a "dead" consumer socket as fixed hardware, like a console),
gamescope as the Wayland compositor, a native Rust UI framework (vello/wgpu/
parley with Xilem), and a nested wheel navigation model for the shell. Key
rejections in these early rounds: Chromium kiosk UI (too heavy), batocera-run
and RetroArch as runner frameworks (too opinionated), and Batocera as a base OS
(wrong model entirely). The project committed to building its own OS, not
modifying an existing distribution.

### Round 2: Runner Architecture and Labels

Round 2 introduced three runner types (libretro, process, media) with a layered
architecture: sandbox engine + runner engine + manifest ABI. The configgen
pipeline was established (shell -> data broker -> runner -> emulator config). The
labels system replaced flat emulator tiers: instead of a single "compatibility"
score, each content item gets multi-dimensional scores across processing,
graphics, memory, storage, input, and composite compatibility. This round also
established the "everything is media" philosophy: the system is a media OS, not
a gaming OS. Music, video, ebooks, and browsing are first-class content types.

### Round 3: Zig PID 1

Round 3 made the controversial decision to write PID 1 in Zig. The rationale
was: PID 1 must never crash, must be tiny (~100 LOC), and must interact with
the kernel's process model (mounting, signal forwarding, zombie reaping) in ways
that are ergonomic in Zig but awkward in Rust (no libstd dependency, no
allocator, direct syscalls). The four-process idle target (PID 1, init, IPC bus,
compositor) was established here. Input handling was pinned to evdev +
gamecontrollerdb (SDL2's controller database format without the SDL2 runtime
dependency).

### Round 4: Cargo Workspace Unification

All crates (both Rust and Zig) became members of a single cargo workspace. This
was a build system decision: cargo manages the dependency graph and build
orchestration even for Zig crates, using `zig build` as the actual compiler for
Zig crates but cargo as the workspace manager. Each crate declares its primary
language in `[package.metadata.loisto]`.

### Rounds 5-6: Refinement Under Review

Rounds 5 and 6 covered game lifecycle (runners, configgen, save states, cgroups,
rendering pipeline, data model) and infrastructure (input routing, audio
pipeline, network, power management, storage layout, display/peripherals). These
rounds were written quickly and contained many decisions that the review round
(2026-02-26) subsequently reworked. The rounds are important not for their
specific decisions (many were overridden) but for establishing the SCOPE of
concerns the system must address.

### 2026-02-26 Review: Consolidation

The consolidated review walked through all 31 research documents and the Round
5-6 decisions, producing a single "new architecture" document with definitive
decisions. This is where the system's current form crystallized. Major changes
from the review: power profiles replaced by continuous power governance (from
PS5, Xbox, ADPF research), RetroAchievements made into an extension implementing
a generic trophy ABI, the storage layout restructured from retro-gaming-specific
paths to extension-centric zones, the 12-node graph architecture formalized, and
the three-dimensional build system (targets x profiles x domains) specified in
detail.

### 2026-02-28 Taxonomy: 52 Crates

The crate taxonomy round took the 9 existing mock crates and expanded to 52,
organizing them into four workspace groups (sys/, shell/, ext/, image/). Major
structural decisions: loisto-model (centralized type dumping ground) removed in
favor of types living where they are owned, fjall replaced by LMDB for the KV
store (C-native, Zig-friendly), PID 1 split from init for crash isolation, IPC
implemented as Zig with Rust bindings in the same crate, domain-specific runners
moved to ext/ as extensions rather than core crates.

### 2026-03-01 Refinement: 84 Crates

The taxonomy refinement expanded to 84 crates across six workspace groups (abi/,
boot/, root/, sys/, shell/, ext/) with naming conventions that indicate each
crate's kind: -d for daemons, -mon for monitors, -ctl for controllers, lib for
libraries, driver- for hardware drivers, frag- for configuration fragments. The
frame pipeline was split from four monolithic stages into ten composable purpose
libraries (fplib-*). Boot chain crates were extracted from sys/ into a new boot/
group with a distinct lifecycle (starts, finishes, hands off). The image/ group
was renamed to root/ and expanded from 2 to 20 crates covering build tooling,
hardware drivers (14 driver-* crates for the known hardware bounds), and system
configuration fragments.

## 2. Key Architecture Decisions

### 2.1 Extension ABI via Rust Proc Macros Generating C

The ABI schema is authored in Rust using `#[repr(C)]` structs with custom proc
macros (`#[derive(AbiStruct)]`, `#[derive(AbiEnum)]` from loisto-abi-macros).
Field annotations use `#[loisto(required)]`, `#[loisto(default = val)]`, etc.

The output is plain C that crosses language boundaries with zero runtime
overhead. Zig consumes the generated C headers via `@cImport`. Rust uses the
`repr(C)` types directly. A codegen tool produces JSON Schema (for IDE
autocomplete) and TOML templates (for extension developer defaults) from the
same Rust definitions.

This is the single source of truth for all cross-domain types. There are 14 ABI
crates (one per architectural node plus abi-media and abi-macros), each defining
the types for its domain: init lifecycle, IPC message envelopes, scheduler
workload descriptors, broker queries, pacer timing, compositor framebuffer
management, runner manifests, shell UI context, input events, audio routing,
network state, update slots, and media transport.

The early design (Round 5 / new-architecture document) initially proposed C
headers as the schema source of truth with annotation macros
(`LOISTO_STRUCT`/`LOISTO_FIELD`/`LOISTO_ENUM`). This was later reversed: Rust
proc macros generate the C headers, which is more ergonomic for the primary
development language and produces identical output at the ABI boundary.

### 2.2 Three-Dimensional Build System

The build system operates on three orthogonal dimensions:

**Targets** (hardware variants): znver2, znver3, rdna2, sdeck. Each specifies
`-march`, CPU features, and Zig/Rust target triples. The target dimension means
the same codebase produces differently optimized binaries for different hardware
variants of the AM4 platform.

**Profiles** (optimization tiers): dev (Cranelift, -O0, no LTO, unwind),
test (Cranelift, -O1, no LTO), preview (-O2, LLVM, ThinLTO, strip debuginfo),
release (-O3, LLVM, Full+Cross LTO, PGO+BOLT, strip all). Cranelift is used for
dev/test because it compiles 2-4x faster than LLVM, but cannot do cross-language
LTO or vectorization.

**Domains** (workload character): frame (sustained rendering, huge pages, PGO),
query (throughput data processing, PGO), core (system foundation, size-prefer),
stream (real-time audio, mandatory no-lazy-binding), cold (rarely active, -Os),
transient (one-shot setup, no special treatment), kernel (sched_ext BPF, separate
toolchain), hardware (driver sysexts).

The build pipeline reads `build/targets.toml`, each crate's domain from
`[package.metadata.loisto]`, `build/profiles.toml`, and `build/domains.toml`,
then computes RUSTFLAGS and Zig flags per domain group. Post-link passes apply
BOLT (profile-guided binary optimization), stripping, and huge page alignment.

All compilation flags are computed; there are no manual `[profile.*.package.*]`
blocks in the workspace Cargo.toml. One source of truth per concern, zero
divergence risk.

### 2.3 Zig-as-Linker

`zig cc` is the linker for all Rust and Zig crates across all profiles. This
enables: cross-language LTO (Rust + Zig share LLVM IR at the link boundary),
Identical Code Folding (`--icf=all`, deduplicating monomorphized generics across
languages), static musl linking (no glibc dependency, smaller footprint,
deterministic builds), and hermetic cross-compilation via `zig cc --target`.

### 2.4 DuckDB + LMDB Data Backend

The data architecture uses two stores, both owned exclusively by the Data Broker
(loisto-brokerd):

**DuckDB** handles structured persistent data: content catalog, metadata, labels,
trophies, download state, NPU telemetry. Column-oriented, vectorized SQL. This
is the cold path, used for analytics queries and complex joins.

**LMDB** handles the asset index: resolved file paths, texture cache keys,
frequently accessed key-value lookups. Memory-mapped B+ tree with zero-copy
reads. This is the warm path during UI navigation.

The choice of LMDB over fjall (the initial Rust-only KV store) was driven by
the language decision: both DuckDB and LMDB have C APIs that Zig consumes
natively via `@cImport`, while fjall is Rust-only and would require FFI
wrappers. All data access goes through IPC bus messages; no other crate touches
the databases directly.

Additional data patterns exist outside the broker: runtime state flows through
IPC bus shared memory ring buffers, user config lives on the filesystem (users
can hand-edit, filesystem-watched into the database), generated configs are
ephemeral (reconstructed at launch from database state), and save states are
filesystem blobs.

## 4. Scheduling Decisions

### 4.1 sched_ext Adoption

The decision to use sched_ext was made in the initial research phase and
reinforced in the Round 6 review. The key argument: loisto knows exactly which
processes it manages and what they need. A custom BPF scheduler can make
decisions that CFS cannot because CFS operates without knowledge of application
semantics. The BPF scheduler sees workload descriptors (is this a game renderer,
an audio thread, a background monitor?) and makes placement decisions
accordingly.

sched_ext also provides hot-reloadability: the BPF scheduler can be updated
without rebooting. This matters for an appliance OS that aims for console-grade
uptime.

### 4.2 Synthetic E-Cores

The synthetic E-core concept emerged from the Round 6 power management redesign.
Homogeneous Zen cores are differentiated via CPPC hints: some cores get low max
frequency (economy), others get high (performance). The BPF scheduler pins
latency-sensitive threads to performance cores and background work to economy
cores.

This is a soft hybrid. The BPF scheduler owns all placement and can flex between
configurations. On 8C/16T (5700G), a typical split is 6 performance threads on
cores 0-2 and 2 economy threads on core 3, but this adapts at runtime based on
workload demands and thermal state.

### 4.3 Workload Classification

The workload classification system connects the labels system (multi-dimensional
per-content-item scores) with the scheduler. Each content item has scores across
dimensions: CPU intensity, GPU intensity, memory pressure, storage I/O, input
requirements, depth buffer source, motion vector source, upscaling benefit,
audio complexity, NPU utilization class. Labels and the Pacer project different
subsets of the same dimension registry. The scheduler consumes the subset
relevant to CPU placement.

The classification feeds a two-layer scheduling model: static WorkloadDescriptors
(computed once per content item, cached in DuckDB, reused on every launch) and
dynamic Pacer adaptation (per-frame decisions based on actual metrics). The static
layer pre-configures the system; the dynamic layer adapts to reality.

## 5. Efficiency Findings

### 5.1 Build Targets Per Hardware

The multi-target build system produces differently optimized binaries for each
hardware variant. `-march=znver2` vs `-march=znver3` selects different
instruction schedules and vectorization strategies. RDNA 2 targets include
dGPU-specific shader compilation. Steam Deck targets include Aerith-specific
power management.

This is possible because loisto controls the entire stack and knows its hardware
bounds. A general distribution cannot ship znver3-optimized binaries because it
must run on any x86-64 machine. loisto ships per-target images, like a console
ships per-console firmware.

### 5.2 CPU Mitigations

The kernel configuration selectively enables CPU vulnerability mitigations:
AMD-relevant mitigations are on, Intel-only mitigations are off. On an AMD-only
platform, this eliminates the performance overhead of mitigations for
vulnerabilities that do not affect the target hardware (Meltdown, various
Intel-specific speculative execution issues). The research review estimated this
at 2-5% performance recovery depending on workload.

### 5.3 Memory Binning with Cgroups

Services are assigned memory budgets enforced by cgroups. The categorization is
tied to the domain system: frame-domain crates get larger memory allocations
(huge pages for code and heap), cold-domain crates get minimal allocations (they
are rarely active and should not consume resources). The cgroup configuration is
computed from domain metadata, not manually specified per crate.

Memory binning interacts with the huge page strategy: with ~4-8 active processes
and known memory budgets, the system has minimal memory fragmentation. THP
(Transparent Huge Pages) is configured as `madvise` (explicit opt-in per
allocation), and frame-domain crates opt in aggressively. This is more aggressive
than general distributions but justified by the known, bounded process count.

### 5.4 Huge Pages

Frame-domain crates use 2MB `.text` alignment (huge code pages) and
`MADV_HUGEPAGE` on heap allocations. Query-domain crates use heap-only huge
pages (DuckDB mmap). Stream-domain crates use heap-only huge pages (audio
buffers). Core, cold, and transient domains skip huge pages because their
binaries are too small to benefit and the alignment padding would waste memory.

The kernel reserves 128MB of 2MB huge pages (`vm.nr_hugepages=64`) for
explicit use, with THP available for the rest via madvise. This is calibrated
for the 32GB RAM target.

## 6. Data Architecture

### 6.1 DuckDB + LMDB

The final data architecture uses two stores, both owned by brokerd:

DuckDB serves as the structured data store for content catalog, metadata,
labels, trophies, download state, and NPU telemetry. It is column-oriented and
vectorized, making it efficient for analytics queries (e.g., "show me all PS2
games with compatibility score > 0.8 sorted by last played"). DuckDB is updated
periodically with eventual consistency acceptable for analytics workloads.

LMDB serves as the fast-access asset index for resolved paths, texture cache
keys, and any data that needs sub-millisecond read latency during UI navigation.
Memory-mapped B+ tree with zero-copy reads means LMDB lookups are essentially
pointer dereferences into the mapped file.

The evolution from the initial design is notable: the earliest rounds used fjall
(a Rust KV store). The taxonomy round (2026-02-28) replaced fjall with LMDB
because LMDB has a C API that Zig consumes natively, while fjall is Rust-only.
The language choice for the broker (Zig) drove the storage choice.

### 6.2 Data Broker as Sole Database Owner

No crate other than brokerd touches DuckDB or LMDB directly. All data access
goes through IPC bus messages. This is an explicit architectural constraint:
the broker is the "database owner" pattern. Extensions query content through IPC,
the shell queries metadata through IPC, the scheduler queries workload
descriptors through IPC.

The benefit is singular: database schema changes, storage backend changes, and
query optimization changes are isolated to one crate. The rest of the system
sees only typed IPC messages defined in abi-broker.

### 6.3 TOML Ingestion and Config Classes

User-facing configuration uses TOML files on the filesystem. The Round 5 review
established three config classes:

1. **User configs**: human-readable files, editable by hand, filesystem-watched
   into the database (not the other way around). The UI writes these too.
2. **App userdata**: extension saves, shader caches, etc. Cannot move to the
   database. Indexed in the database for recovery/management, but files stay on
   the filesystem.
3. **Generated artifacts**: ephemeral, reconstructed from the database at launch
   time. Never persisted. The harness crate (loisto-harness) produces config
   files, CLI arguments, save file formats, environment variables, and any
   ephemeral artifacts extensions need before launch.

The flow for user config is: user edits TOML file (or UI writes it) ->
filesystem watcher detects change -> broker ingests into DuckDB/LMDB -> IPC
subscribers receive change notification. This means the filesystem is the source
of truth for user config, and the database is a derived cache.

### 6.4 Storage Zones

The /data partition is organized into six zones with Landlock enforcement:

- `ext/<id>/`: per-extension private data, opaque to loisto, read-write only by
  that extension
- `res/content/`: playable/viewable media, broker-managed, read-only for
  extensions (filtered by manifest declarations)
- `res/firmware/`: shared platform firmware/BIOS files, read-only for extensions
- `res/media/`: display assets (thumbnails, box art), read-only for shell/overlay
- `idx/`: DuckDB + LMDB, read-write only by broker
- `cfg/`: system-level user config, read-write by shell and system services
- `log/`: rotated logs, append-only by system services
- `sys/`: machine state, root-only (init, RAUC)

Extensions never use filesystem paths directly. They declare resource needs using
the `verb:target(specifier)` grammar, and the system generates constants for
resolved paths at launch time.

## 7. Relevance to hilavitkutin

### 7.1 Design Round Process

The most immediately transferable aspect of loisto's design rounds is the
PROCESS, not the content. Seven rounds over two days, 31 research documents,
then a consolidated review that overwrites earlier decisions without attempting
backward compatibility. The project does not accumulate decisions; it replaces
them. Round 6's power management section was entirely rewritten during the
review. The storage layout was restructured from a retro-gaming-specific tree to
extension-centric zones. The runner architecture from Round 2 was flagged for
complete rework against the extension model.

For hilavitkutin, this means: design rounds should not be incremental patches on
earlier rounds. Each round should re-evaluate its domain from first principles
against the current state of all other domains. Decisions are cheap to override
during design; they become expensive only after implementation. The design docs
are the contract; source code is corrected to match them, never the reverse.

### 7.2 Two-Layer Scheduling

loisto's static + dynamic scheduling model (WorkloadDescriptor + Pacer)
translates directly to hilavitkutin. The static layer is the pipeline IR: domain
transforms produce typed descriptions of what work needs to happen, what
resources it needs, and what its dependencies are. This is computed once per
pipeline configuration change and cached. The dynamic layer is the runtime
scheduler: it reads actual throughput, backpressure, and resource availability
and makes per-batch decisions about work assignment.

The lesson from loisto is: keep the static layer rich (many dimensions of
description) and the dynamic layer cheap (simple decisions based on current
metrics). Do not try to make the dynamic layer smart; make the static layer
informative enough that the dynamic layer's decisions are obvious.

### 7.3 Domain-Based Optimization

loisto's domain system (frame, query, core, stream, cold, transient) assigns
different compilation and runtime strategies to crates based on their workload
character. This is a build-time concern that hilavitkutin cannot replicate
directly (it is a library, not an OS), but the concept of classifying work units
by their character and applying different scheduling strategies per class is
directly applicable.

A hilavitkutin pipeline stage that processes data in bulk (analogous to loisto's
query domain) should be scheduled for throughput. A stage that produces output
consumed by a latency-sensitive consumer (analogous to loisto's frame domain)
should be scheduled for deadline. A stage that runs rarely (analogous to loisto's
cold domain) should be scheduled with minimal resource reservation.

### 7.4 Sole Owner Pattern

The broker's "sole database owner" pattern is relevant to hilavitkutin's data
store design. If the pipeline has a central data structure (the typed in-memory
database from the paradigm document), access should be mediated through a single
owner with defined query and mutation interfaces, not through direct access from
arbitrary pipeline stages. This enables schema evolution, storage backend
changes, and consistency guarantees without distributed coordination.

### 7.5 ABI-First Design

loisto defines cross-domain contracts as ABI types before implementing anything.
The 14 abi/ crates define every type that crosses a domain boundary. This is the
type taxonomy approach: the contract is the design, implementations follow.

For hilavitkutin, this means: define the IR types (what columns exist, what
transforms operate on them, what the scheduler sees) as explicit contract types
before implementing transforms or schedulers. The IR IS the ABI between pipeline
stages, and it should be defined with the same rigor as a C ABI.

### 7.6 What Does NOT Transfer

Several loisto patterns are system-level concerns that do not apply to a
library-level scheduler:

- Immutable rootfs and A/B update slots
- Hardware driver sysexts and boot chain management
- PID 1 and init process supervision
- Landlock/seccomp/namespace sandboxing
- Physical device management (evdev, PipeWire, DRM/KMS)
- CachyOS kernel fork and kernel configuration
- PCIe topology and hardware-specific compute unit management
- RAUC, systemd-sysext, dm-verity, SquashFS

These are the MECHANISM layer. The POLICY patterns (workload classification,
two-layer scheduling, domain-based optimization, sole owner data access, ABI-
first type contracts) are the transferable insights.

### 7.7 Crate Taxonomy as Architectural Discipline

The progression from 9 to 52 to 84 crates, with naming conventions that tell you
what each crate IS (-d, -mon, -ctl, lib, driver-, frag-, abi-), demonstrates an
architectural discipline that transfers to any project: every distinct concern
gets its own module, and the name tells you its kind. "Nothing is too thin" is
a principle that prevents the accumulation of multi-purpose modules that become
difficult to reason about and split later.

For hilavitkutin, this means: if a concern is distinct enough to name, it is
distinct enough to be its own module. A pipeline stage, a transform, a column
type, and a scheduler policy are all separate concerns even if their initial
implementations are small. The taxonomy is the architecture; the code fills it
in.
