## 1. LLVM's default O3 pipeline

Source: `llvm/lib/Passes/PassBuilderPipelines.cpp` (LLVM 22.1).

The O3 pipeline runs passes in this order (grouped by phase):

### Phase 1: Module simplification

ForceFunctionAttrsPass, InferFunctionAttrsPass, Annotation2MetadataPass,
LowerExpectIntrinsicPass, CoroEarlyPass, IPSCCPPass,
CalledValuePropagationPass, GlobalOptPass, PromotePass, InstCombinePass,
AlwaysInlinerPass, DeadArgumentEliminationPass.

### Phase 2: Inlining and CGSCC optimization

ModuleInlinerWrapperPass (wraps InlinerPass, PostOrderFunctionAttrsPass,
ArgumentPromotionPass inside a CGSCC walk).

### Phase 3: Function simplification (the main loop)

Per function, iteratively:
EarlyCSEPass, GVNHoistPass*, GVNSinkPass*, SpeculativeExecutionPass,
JumpThreadingPass, CorrelatedValuePropagationPass, InstCombinePass,
AggressiveInstCombinePass, LibCallsShrinkWrapPass, TailCallElimPass,
ReassociatePass, ConstraintEliminationPass.

Then the loop nest pipeline:
LoopInstSimplifyPass, LoopSimplifyCFGPass, LICMPass, LoopRotatePass,
SimpleLoopUnswitchPass, LoopIdiomRecognizePass, IndVarSimplifyPass,
LoopDeletionPass, LoopFullUnrollPass.

Then post-loop:
SROAPass, VectorCombinePass, MergedLoadStoreMotionPass, GVNPass,
SCCPPass, BDCEPass, InstCombinePass, ADCEPass, MemCpyOptPass, DSEPass,
MoveAutoInitPass, CoroElidePass.

### Phase 4: Vectorization

LoopDistributePass (only if `EnableLoopDistribute` is true -- default false),
InjectTLIMappings, LoopVectorizePass, LoopLoadEliminationPass,
InstCombinePass, SimplifyCFGPass, SLPVectorizerPass.

### Phase 5: Post-vectorization cleanup

VectorCombinePass, InstCombinePass, LoopUnrollPass, LoopSinkPass,
LoopDeletionPass, LICMPass, SimplifyCFGPass.

### Phase 6: Module optimization

GlobalDCEPass, ConstantMergePass, CGProfilePass,
RelLookupTableConverterPass.

**Passes marked with * are behind flags that default to off.**

### What this means for data processing

The default O3 pipeline gives you:
- Loop vectorization (LoopVectorizePass)
- SLP vectorization (SLPVectorizerPass)
- Loop unrolling (LoopFullUnrollPass + LoopUnrollPass)
- LICM (LICMPass, runs twice)
- GVN (load elimination, redundancy elimination)
- Dead store elimination
- Loop idiom recognition (memset/memcpy patterns)
- Basic loop simplification

But it does NOT give you:
- Loop distribution (off by default)
- Loop fusion (off by default)
- Loop interchange (not in default pipeline)
- Loop flatten (off by default)
- Loop unroll-and-jam (off by default)
- Loop versioning for LICM (off by default)
- Data prefetching (not in default pipeline)
- Load-store vectorization (not in default pipeline)
- Sandbox vectorizer (experimental, off)
- Histogram vectorization (off by default)
- Loop bound split (not in default pipeline)
- Loop idiom vectorize (not in default pipeline)

## 2. Passes not enabled by default

These passes exist in LLVM, compile successfully, but are gated behind
flags or absent from the default pipeline. Each is available via
`-C passes=<name>` or `-C llvm-args=<flag>`.

### 2.1. LoopDistributePass

**Pass name:** `loop-distribute`
**Flag:** `-enable-loop-distribute` (default: false)
**What:** Splits loops with dependence cycles into multiple loops.
Isolates problematic dependences so the remaining partitions can be
vectorized.
**Data processing benefit:** High. A loop that reads columns A and B,
computes a result, and writes column C might have a carried dependence
on C that prevents vectorization. Distribution splits the computation
into vectorizable parts.
**Activation:**
```
-C llvm-args=-enable-loop-distribute
# or
-C passes=loop-distribute
```
**Runs before the vectorizer** in the pipeline position. The default
pipeline already has the slot but gates it with the flag.

### 2.2. LoopFusePass

**Pass name:** `loop-fusion`
**Flag:** Controlled by `PTO.LoopFusion` in the pipeline builder (default: false)
**What:** Combines adjacent loops over the same iteration space into a
single loop. Reduces loop overhead and improves data locality by
touching data once instead of twice.
**Maturity:** Moderate. The profitability model currently returns true
unconditionally ("primarily to test the pass"). Dependence analysis is
conservative. Loop peeling support exists to make trip counts match.
**Data processing benefit:** High for chained operators (filter then
project over the same morsel). Eliminates intermediate writes.
**Known issues:**
- Profitability model is a stub (always fuses)
- Does not handle non-empty exit blocks
- SCEV tracking through fusion is incomplete
**Activation:**
```
-C passes=loop-fusion
```

### 2.3. LoopInterchangePass

**Pass name:** `loop-interchange`
**Flag:** Not behind a flag; simply not in the default pipeline
**What:** Swaps nested loop order to improve cache access patterns.
Converting column-major access to row-major (or vice versa) can
eliminate cache misses.
**Data processing benefit:** Very high for multi-column processing
where the loop order determines whether access is sequential or strided.
**Limitations:**
- Only supports 2-10 deep nests
- Tightly nested loops only (no intervening code)
- No triangular loop support
- Maximum 64 load/store instructions analyzed
**Activation:**
```
-C passes=loop-interchange
```

### 2.4. LoopFlattenPass

**Pass name:** `loop-flatten`
**Flag:** `EnableLoopFlatten` (default: false)
**What:** Converts nested `for i in 0..N { for j in 0..M }` into
`for k in 0..(N*M)` when the inner body uses `A[i*M+j]`. Eliminates
the overhead of the outer loop and enables better vectorization of the
single resulting loop.
**Data processing benefit:** Moderate. Useful when iterating over 2D
column data (e.g., nested groups), but most columnar access is 1D.
**Risks:** Must prove `N*M` does not overflow. Uses IV widening or
runtime overflow checks. If the inner loop is small, the flattened loop
may execute speculative instructions more often.
**Activation:**
```
-C llvm-args=-enable-loop-flatten
```

### 2.5. LoopUnrollAndJamPass

**Pass name:** `loop-unroll-and-jam`
**Flag:** `EnableUnrollAndJam` (default: false)
**What:** Unrolls the outer loop and fuses the resulting inner loop
bodies. Improves register reuse when the inner loop accesses data that
is invariant in the outer loop.
**Data processing benefit:** Moderate to high for nested column
operations where the outer loop iterates over columns and the inner
loop iterates over rows.
**Activation:**
```
-C llvm-args=-enable-unroll-and-jam
```

### 2.6. LoopVersioningLICMPass

**Pass name:** `loop-versioning-licm`
**Flag:** `UseLoopVersioningLICM` (default: false)
**What:** Creates two versions of a loop: one optimized assuming no
aliasing (all memory marked `noalias`), one conservative fallback.
Runtime checks determine which runs. Enables LICM on loops where
alias analysis returns `MayAlias`.
**Data processing benefit:** High for multi-column operations through
pointer arguments where LLVM cannot prove non-aliasing. The optimized
path hoists invariant loads outside the loop.
**Requirements:** Innermost loop, simplified form, single backedge,
contains invariant loads/stores with ambiguous aliases.
**Activation:**
```
-C llvm-args=-enable-loop-versioning-licm
```

### 2.7. LoopDataPrefetchPass

**Pass name:** `loop-data-prefetch`
**Flag:** Not in default pipeline. Must be added manually.
**What:** Inserts `llvm.prefetch` intrinsics for strided memory accesses
in inner loops. Calculates how many iterations ahead to prefetch based
on target-specific prefetch distance divided by loop iteration size.
**Data processing benefit:** High for streaming through column data that
exceeds L1 cache. Prefetching the next cache line while processing the
current one hides memory latency.
**Configuration:**
- `loop-prefetch-writes` (default: false) -- prefetch store addresses too
- `prefetch-distance` -- instructions ahead (target-dependent default)
- `min-prefetch-stride` -- minimum stride to trigger prefetching
- `max-prefetch-iters-ahead` -- cap on lookahead
**Limitations:** Inner loops only. No profile-guided insertion.
**Activation:**
```
-C passes=loop-data-prefetch
-C llvm-args=-loop-prefetch-writes
```

### 2.8. LoadStoreVectorizerPass

**Pass name:** `load-store-vectorizer`
**Flag:** Not in default pipeline.
**What:** Merges sequential scalar loads/stores into vector loads/stores.
Unlike SLP vectorizer, this operates on memory operations directly,
combining `load i32, ptr` + `load i32, ptr+4` into `load <2 x i32>, ptr`.
**Originally designed for:** GPU targets (NVPTX, AMDGPU) where vectorized
loads go to consecutive scalar registers with free extraction.
**Data processing benefit:** Low for CPU targets. The SLP vectorizer
already handles most cases. Potentially useful for specific patterns the
SLP misses.
**Activation:**
```
-C passes=load-store-vectorizer
```

### 2.9. SandboxVectorizerPass

**Pass name:** `sandbox-vectorizer`
**Status:** Experimental. Under active development.
**What:** A new modular vectorizer framework built on "Sandbox IR" -- a
shadow IR layer. Uses a configurable pipeline of sub-passes: seed
collection, bottom-up vectorization, accept-or-revert.
**Data processing benefit:** Unknown. Too early to evaluate.
**Default sub-pipeline:** `seed-collection<tr-save,bottom-up-vec,tr-accept-or-revert>`
**Activation:**
```
-C passes=sandbox-vectorizer
```

### 2.10. LoopIdiomVectorizePass

**Pass name:** `loop-idiom-vectorize`
**Flag:** Not in default pipeline.
**What:** Recognizes loop patterns that the standard loop vectorizer
cannot handle due to early exits. Currently recognizes:
1. Byte-comparison loops (find first mismatch between two arrays)
2. `find_first_of` patterns (find first matching element from a set)
**Hardware target:** SVE2 targets benefit most ("256 comparisons in one
go"). Also works on fixed-width SIMD.
**Data processing benefit:** High for string/byte comparison in filter
predicates. A filter scanning a column for the first match of a set of
values is exactly this pattern.
**Activation:**
```
-C passes=loop-idiom-vectorize
```

### 2.11. LoopBoundSplitPass

**Pass name:** `loop-bound-split`
**Flag:** Not in default pipeline.
**What:** Splits a loop with a conditional branch into two loops: one
with a tighter bound (condition always true), one handling the rest
(condition always false). Eliminates the branch from the main loop body.
**Data processing benefit:** Moderate. Useful for loops with a
"process while condition holds, then switch to different logic" pattern.
**Limitations:** Only innermost loops. No `ICMP_NE/EQ` support. No
negative step support.
**Activation:**
```
-C passes=loop-bound-split
```

### 2.12. LoopPredicationPass

**Pass name:** `loop-predication`
**What:** Converts per-iteration guard/range checks into a single
loop-invariant check. Instead of checking `i < len` every iteration,
proves the check holds for all iterations and hoists it.
**Data processing benefit:** High for Rust code. Rust's bounds checks
compile to per-iteration guards. Loop predication can prove them
redundant and hoist them, yielding the same performance as
`get_unchecked()` without the `unsafe`.
**Activation:**
```
-C passes=loop-predication
```

### 2.13. IRCEPass (Inductive Range Check Elimination)

**Pass name:** `irce`
**What:** Splits a loop into pre/main/post loops where the main loop
provably needs no range checks. The pre-loop handles iterations before
the safe range, the main loop runs without checks, the post-loop
handles remaining iterations.
**Data processing benefit:** High. Same motivation as loop predication
but uses loop splitting instead of hoisting. Eliminates bounds checks
from the vectorizable main loop.
**Activation:**
```
-C passes=irce
```

### 2.14. ConstraintEliminationPass

**Pass name:** `constraint-elimination`
**What:** Collects constraints from dominating branch conditions and
eliminates redundant comparisons. If `i < n` was checked in a branch,
later checks of `i < n` in dominated blocks are removed.
**Data processing benefit:** Moderate. Eliminates redundant bounds
checks that survive after other passes.
**Note:** This IS in the default O3 pipeline already.

### 2.15. GVNHoistPass and GVNSinkPass

**Pass names:** `gvn-hoist`, `gvn-sink`
**Flags:** `EnableGVNHoist` (default: off), `EnableGVNSink` (default: off)
**What:** GVNHoist moves computations earlier to merge redundant
executions. GVNSink moves computations later for the same purpose.
Complementary to GVN proper.
**Data processing benefit:** Low. GVN already handles most cases.
**Activation:**
```
-C llvm-args=-enable-gvn-hoist
-C llvm-args=-enable-gvn-sink
```

### 2.16. NewGVNPass

**Pass name:** `newgvn`
**Flag:** `RunNewGVN` (default: false, uses GVNPass instead)
**What:** A rewrite of GVN using a more sophisticated congruence-based
algorithm. Theoretically more powerful but less battle-tested.
**Data processing benefit:** Unknown. May find more redundancies in
complex expression trees.

### 2.17. DFAJumpThreadingPass

**Pass name:** `dfa-jump-threading`
**Flag:** `EnableDFAJumpThreading` (default: false)
**What:** Threads switch statements inside loops. When the next case
is statically determinable from the current case, converts the switch
to direct jumps. Eliminates indirect branch overhead.
**Data processing benefit:** Moderate. Useful if your operator dispatch
uses a switch-in-loop pattern (state machine over operator types).
**Activation:**
```
-C llvm-args=-enable-dfa-jump-threading
```

### 2.18. ControlHeightReductionPass (CHR)

**Pass name:** `chr`
**Flag:** `EnableCHR` (default: true at O3)
**What:** Merges multiple biased conditional branches into a single
branch using profile data. Requires branches with >99% bias.
**Data processing benefit:** Moderate. Reduces branch overhead in
hot paths with predictable conditions (e.g., null checks on
non-nullable columns).
**Note:** Already enabled at O3 when profile data is available.

### 2.19. LoopLoadEliminationPass

**Pass name:** `loop-load-elim`
**What:** Detects store-to-load forwarding patterns in loops where
iteration N stores to `A[i+1]` and iteration N+1 loads from `A[i]`.
Replaces the load with the previously stored value using a PHI node.
**Data processing benefit:** Moderate. Reduces memory bandwidth in
loops with inter-iteration data flow (e.g., running aggregates).
**Note:** IS in the default O3 pipeline already.

### 2.20. InterleavedAccessPass and InterleavedLoadCombinePass

**Pass names:** `interleaved-access`, `interleaved-load-combine`
**What:** InterleavedAccess lowers interleaved memory operations to
target-specific instructions (e.g., ARM NEON `vld2`/`vst2`).
InterleavedLoadCombine detects shuffle patterns that represent
interleaved loads and combines them into wider loads that the
InterleavedAccess pass can lower.
**Data processing benefit:** High for AoS (array of structs) data
accessed in a loop. Less relevant for pure columnar (SoA) layouts.
**Activation:**
```
-C passes=interleaved-load-combine,interleaved-access
```

### 2.21. Histogram Vectorization (inside LoopVectorizePass)

**Flag:** `-enable-histogram-loop-vectorization` (default: false)
**What:** Detects `buckets[indices[i]] += step` patterns and vectorizes
them using target-specific histogram instructions (e.g., SVE2 `histcnt`).
**Data processing benefit:** Very high for GROUP BY aggregation and
hash table probing. The histogram pattern is the core of count-based
aggregation.
**Hardware:** SVE2 on AArch64 has native `histcnt`. x86 has no direct
equivalent (would use conflict detection + scatter).
**Activation:**
```
-C llvm-args=-enable-histogram-loop-vectorization
```

### 2.22. Explicit Vector Length (EVL) Tail Folding

**Flag:** Controlled by target TTI `foldTailWithEVL()`.
**What:** Uses EVL (explicit vector length) instructions instead of
traditional mask-based tail folding. The processor executes only the
valid lanes without masking overhead.
**Data processing benefit:** High for variable-length morsel tails on
targets supporting EVL (RISC-V V, potentially future ARM/x86).
**Note:** Target-dependent; not directly user-controlled via flags.

### 2.23. MergeFunctionsPass

**Pass name:** `mergefunc`
**Flag:** `EnableMergeFunctions` (default: false)
**What:** Identifies and merges structurally identical functions. In a
heavily generic/monomorphized Rust codebase, many instantiations produce
identical machine code.
**Data processing benefit:** Low on performance, high on binary size.
Reduces instruction cache pressure from duplicated monomorphizations.
**Activation:**
```
-C llvm-args=-enable-merge-functions
```

### 2.24. HotColdSplittingPass

**Pass name:** `hotcoldsplit`
**Flag:** `EnableHotColdSplit` (default: false at O3)
**What:** Extracts cold code regions into separate functions. Keeps hot
paths compact for better instruction cache utilization.
**Data processing benefit:** Moderate. Error handling, logging, and
rare-path code in operator implementations gets moved out of the hot
loop's instruction footprint.
**Activation:**
```
-C llvm-args=-enable-hot-cold-split
```

### 2.25. PartialInlinerPass

**Pass name:** `partial-inliner`
**Flag:** `RunPartialInlining` (default: false)
**What:** Outlines cold regions of a function, then inlines the
remaining hot part. Reduces effective inline cost.
**Data processing benefit:** Low to moderate. Helps when operator
functions are too large to inline but have small hot paths.
**Activation:**
```
-C llvm-args=-enable-partial-inlining
```

## 3. Vectorization passes

### Full taxonomy

| Pass | In default O3? | Purpose | Data processing relevance |
|------|---------------|---------|--------------------------|
| `loop-vectorize` | Yes | SIMD-ify loops with known trip counts | Core: all column loops |
| `slp-vectorizer` | Yes | Combine independent scalar ops | High: multi-column arithmetic |
| `vector-combine` | Yes | Optimize scalar/vector interactions | Medium: cleanup |
| `load-store-vectorizer` | No | Merge adjacent loads/stores | Low for CPU |
| `sandbox-vectorizer` | No | Experimental modular framework | Unknown |
| `loop-idiom-vectorize` | No | Vectorize early-exit search loops | High for string filters |
| `interleaved-access` | Post-codegen | Lower interleaved ops to HW | Medium |
| `interleaved-load-combine` | Post-codegen | Detect interleaved patterns | Medium |
| `complex-deinterleaving` | Post-codegen | Complex number ops | None |

### Key loop vectorizer flags

| Flag | Default | Effect |
|------|---------|--------|
| `-force-vector-width=N` | auto | Force SIMD width (4, 8, 16, ...) |
| `-force-vector-interleave=N` | auto | Force unroll factor inside vectorized loop |
| `-enable-loop-distribute` | false | Enable loop distribution before vectorizer |
| `-vectorizer-maximize-bandwidth` | false | Use maximum register bandwidth |
| `-enable-interleaved-mem-accesses` | true | Vectorize strided access patterns |
| `-enable-masked-interleaved-mem-accesses` | false | Masked variant |
| `-prefer-predicate-over-epilogue` | false | Use tail folding instead of scalar epilogue |
| `-enable-early-exit-vectorization` | false | Vectorize loops with multiple exits |
| `-enable-histogram-loop-vectorization` | false | Vectorize histogram patterns |
| `-vectorize-with-block-frequency` | false | PGO-enhanced: aggressive in hot regions, conservative in cold |

### Gather/scatter support

LLVM supports masked gather and scatter via:
- `llvm.masked.gather` / `llvm.masked.scatter` intrinsics
- The loop vectorizer generates these for non-contiguous (strided) access patterns
- Profitability depends on `TTI::isLegalMaskedGather/Scatter()` returning true
- x86 AVX2 supports gather (not scatter); AVX-512 supports both
- AArch64 SVE supports both with overhead modeled at ~10 cycles each

Gather/scatter is used automatically by the loop vectorizer when it
encounters strided access patterns. There is no separate pass to enable.
The cost model determines whether to use scalar loops or gather/scatter
based on the target.

### Reduction optimization

The loop vectorizer handles reductions (sum, product, min, max, AND,
OR, XOR) automatically. Key decisions:
- `TTI::preferInLoopReduction()` -- should the reduction happen inside the
  vector loop body or after?
- `TTI::enableOrderedReductions()` -- allow non-deterministic FP reduction
  order for speed?
- Tree reductions vs. in-loop reductions are chosen per-target

No separate reduction pass exists; it is integrated into the loop
vectorizer.

### Pass interactions

- `loop-distribute` MUST run before `loop-vectorize` (it creates the
  partitions that the vectorizer can handle)
- `loop-interchange` should run before `loop-vectorize` (correct loop
  order is needed for vectorization legality)
- `loop-fusion` should run after inlining (adjacent loops from different
  functions become visible only after inlining)
- `loop-data-prefetch` should run after `loop-vectorize` (prefetch the
  vectorized access pattern, not the scalar one)
- `loop-flatten` competes with `loop-interchange`; only one should apply
  to a given nest

## 5. Memory and cache passes

### Passes with cache/memory impact

| Pass | In default O3? | What it does for memory |
|------|---------------|----------------------|
| `licm` | Yes | Hoists invariant loads out of loops |
| `gvn` | Yes | Eliminates redundant loads |
| `dse` | Yes | Removes dead stores |
| `memcpyopt` | Yes | Optimizes memcpy/memmove/memset sequences |
| `sroa` | Yes | Breaks aggregates into scalars (registers) |
| `mldst-motion` | Yes | Sinks/hoists loads/stores across diamonds |
| `loop-data-prefetch` | No | Inserts software prefetch instructions |
| `loop-versioning-licm` | No | Alias-versioned hoisting |
| `separate-const-offset-from-gep` | No | Splits GEPs for better CSE |
| `loop-load-elim` | Yes | Eliminates redundant loads via forwarding |
| `pgo-memop-opt` | PGO only | Specializes memcpy/memset for hot sizes |

### SeparateConstOffsetFromGEP

**Pass name:** `separate-const-offset-from-gep`
**What:** Splits `getelementptr` instructions with multiple indices into
a variable base + constant offset. Enables the base to be hoisted by
LICM and the offset to fold into addressing modes.
**Data processing benefit:** High for columnar access where the column
base pointer is loop-invariant but the row offset varies. Reducing GEP
complexity frees registers and enables CSE of shared base calculations.
**Originally designed for:** NVPTX (GPU register pressure), but benefits
CPU targets too.
**Activation:**
```
-C passes=separate-const-offset-from-gep
```

### Cache hierarchy information available to LLVM

The `TargetTransformInfo` exposes:
- `getCacheLineSize()` -- typically 64 bytes (x86, AArch64)
- `getCacheSize(L1)` -- 32 KiB typical
- `getCacheSize(L2)` -- 256 KiB typical
- `getCacheAssociativity(L1/L2)` -- 8-way typical
- `getPrefetchDistance()` -- target-dependent
- `getMinPrefetchStride()` -- minimum stride to trigger prefetching
- `getMaxPrefetchIterationsAhead()` -- cap on lookahead

These are queried by loop-data-prefetch and the loop vectorizer's
interleave factor calculation. They are NOT directly tunable via
command-line flags; they come from the target definition. Using
`-C target-cpu=native` ensures the correct values for the host CPU.

## 6. Branch and control flow passes

### Branch elimination and branchless optimization

LLVM does NOT have a dedicated "convert branches to selects" pass.
The opposite exists: **SelectOptimize** (in CodeGen) converts selects
BACK to branches when profile data shows the condition is highly
predictable and one operand is expensive.

Branch-to-select conversion happens in multiple places:
- **SimplifyCFG** (`simplifycfg`) converts simple diamond CFG patterns
  to select instructions. This IS in the default pipeline.
- **Loop vectorizer** performs if-conversion (flattening control flow
  to selects/masks) as part of vectorization.
- **SpeculativeExecution** (`speculative-execution`) moves instructions
  to execute speculatively before branches, enabling folding. In default
  O3 pipeline.

For data processing: the loop vectorizer's if-conversion is the primary
mechanism. It converts `if (predicate) { result = expr; }` into masked
operations. No separate pass needed.

### Control flow passes relevant to data processing

| Pass | In default O3? | Purpose |
|------|---------------|---------|
| `simplifycfg` | Yes (multiple) | Merge blocks, convert to selects |
| `jump-threading` | Yes | Thread branches through known conditions |
| `speculative-execution` | Yes | Speculatively execute before branches |
| `chr` | Yes (O3 + PGO) | Merge biased branches |
| `dfa-jump-threading` | No (off by default) | Thread switch-in-loop |
| `select-optimize` | CodeGen | Convert selects back to branches |
| `guard-widening` | No | Merge runtime guards |
| `irce` | No | Eliminate range checks by loop splitting |
| `loop-predication` | No | Hoist loop guards |

## 7. Profile-guided passes

### What profile data enables

When PGO data is available (`-C profile-use=...`), these passes change
behavior:

| Pass | How profile data is used |
|------|-------------------------|
| InlinerPass | Inline hot callsites more aggressively |
| PGOIndirectCallPromotion | Promote hot indirect calls to direct |
| PGOMemOPSizeOpt | Specialize memcpy/memset for observed sizes |
| LoopVectorizePass | More aggressive in hot regions when `vectorize-with-block-frequency` |
| LoopUnrollPass | Adjust unroll thresholds based on trip counts |
| CHR | Merge biased branches (>99% one-way) |
| SelectOptimize | Convert selects to branches when predictable |
| LoopSinkPass | Sink to lower-frequency blocks |
| HotColdSplittingPass | Extract cold code to separate functions |
| MachineBlockPlacement | Reorder basic blocks for fallthrough optimization |
| MachineFunctionSplitter | Split cold basic blocks to separate sections |
| BasicBlockSections | Group hot blocks together, cold blocks separately |

### Specific PGO-related flags

**`--profile-guided-section-prefix`** (default: true)
What: Assigns `.text.hot` prefix to hot functions and `.text.unlikely`
to cold functions. The linker groups these sections, improving
instruction cache locality.
How: CodeGenPrepare checks `ProfileSummaryInfo` to classify functions.
Activation: Enabled by default when profile data is present.

**`--split-machine-functions`** (Machine Function Splitter)
What: Splits individual functions into hot and cold sections. Cold basic
blocks (those with no profile counts, or below the 99.995th percentile)
go to `.text.unlikely.*`. Hot blocks stay in `.text.hot.*`.
Requires: Profile data (instrumentation or sample). Without profile data,
it only splits exception handling code if `-mfs-split-ehcode` is set.
Effect: Reduces instruction cache pollution. The hot path of a function
fits in fewer cache lines.
Activation:
```
-C llvm-args=-split-machine-functions
```

**Codegen Data ThinLTO Two Rounds**
What: A two-pass ThinLTO compilation where the first round collects
codegen data (outlined function patterns, stable function maps), and
the second round uses that data for better outlining and function
merging decisions.
Status: Implemented in `llvm/lib/CGData/CodeGenData.cpp`.
Activation: Via linker/build system flags (not directly via rustc flags).

**`-vectorize-with-block-frequency`** (LoopVectorizeWithBlockFrequency)
What: When PGO data is available, the loop vectorizer becomes "more
aggressive in hot regions" and minimizes "code growth in cold regions."
Without this flag, the vectorizer treats all loops equally.
Activation:
```
-C llvm-args=-vectorize-with-block-frequency
```

**Profile-guided prefetch insertion:** Does not exist. The
LoopDataPrefetch pass uses static analysis only. No LLVM pass uses
profile data to guide prefetch placement.

**Profile-guided loop tiling:** Does not exist in LLVM. Polly can use
profiling information for its optimizer, but LLVM's built-in passes
do not perform profile-guided tiling.

## 8. Industry compiler additions

### Intel DPC++/oneAPI (intel/llvm, sycl branch)

Intel's LLVM fork adds passes in `llvm/lib/SYCLLowerIR/`:
- **SYCLOptimizeBarriers** -- optimize synchronization barriers
- **SYCLJointMatrixTransform** -- joint matrix (tensor core) operations
- **LowerInvokeSimd** -- lower SIMD invocation patterns
- **LowerESIMD** (ESIMD subdirectory) -- lower Explicit SIMD constructs
- **ESIMDOptimizeVecArgCallConv** -- optimize vector argument passing
- **LowerESIMDVLoadVStore** -- optimize vector load/store operations

These are GPU/SYCL-specific. None are usable as general CPU optimization
passes. Intel does NOT add extra general-purpose CPU loop optimization
passes to their LLVM fork.

Intel's CPU-specific optimizations ship in ICC/ICX (Intel C++ Compiler),
which is proprietary and does not expose its passes as loadable plugins.

### AMD AOCC

AMD's AOCC (AMD Optimizing C/C++ Compiler) is based on LLVM with
proprietary additions. Known additions from documentation and release notes:
- Enhanced loop tiling for Zen microarchitecture cache hierarchy
- Improved prefetch distance tuning for Zen 4/5 memory subsystem
- Zen-specific cost model adjustments in the vectorizer
- Aggressive function merging and inlining for Zen
- AMD-specific `znver4`/`znver5` scheduling models

These are proprietary. No source code or loadable plugins are available.
The optimizations are baked into AOCC's modified LLVM.

### ARM Compiler

ARM's commercial compiler (Arm Compiler for Embedded, armclang) uses
LLVM with:
- Neoverse-specific scheduling models (upstreamed to LLVM)
- SVE/SVE2 cost model tuning (largely upstreamed)
- AArch64-specific histogram support via `histcnt` instruction (upstreamed)

Most ARM-specific optimizations are upstreamed to LLVM. The AArch64 TTI
(TargetTransformInfo) in upstream LLVM includes:
- SVE gather/scatter overhead: ~10 cycles each
- Histogram (`histcnt`) base cost: 8 cycles
- Streaming SVE mode control
- Tail folding threshold: 15 instructions
- Preference for fixed vs. scalable vectors when costs are equal

Using `-C target-cpu=apple-m4` or `-C target-cpu=neoverse-v2` picks up
all of these tunings automatically.

### Apple LLVM fork (swiftlang/llvm-project)

Apple's LLVM fork (`apple/llvm-project`, now `swiftlang/llvm-project`)
adds Swift-specific passes but NO additional general-purpose optimization
passes for Apple Silicon. Apple's CPU-specific optimizations are contributed
upstream to LLVM's AArch64 backend.

There are no secret Apple Silicon optimization passes. The upstream
`-C target-cpu=apple-m4` is the best available.

### Google's LLVM work

Google has contributed:
- **Propeller** -- profile-guided code layout at the basic block level.
  Implemented as BasicBlockSections in LLVM + a linker-side reordering
  tool. Alternative to BOLT that works at compile time instead of
  post-link. Uses `-fbasic-block-sections=list` + a profile file.
- **BOLT** (originally Facebook/Meta, now in LLVM): Post-link binary
  optimizer using profile data. Reorders functions and basic blocks.
  Not an LLVM IR pass; operates on the binary.
- **ML-guided optimizations**: Infrastructure for using ML models to
  guide inlining, register allocation, and other heuristic decisions.
  Available via `InlineAdvisorAnalysis` and related passes.
- **Profile-guided code layout**: Extended TSP algorithm for basic block
  reordering, contributed to MachineBlockPlacement.

### Meta's BOLT

BOLT is a post-link binary optimizer, NOT an LLVM IR pass. It operates
on compiled binaries using profile data (from `perf record`).

**Optimizations:**
- Basic block reordering (ext-tsp algorithm)
- Function reordering (hfsort+)
- Function splitting (hot/cold)
- Identical code folding (ICF)

**Complementary pre-link passes:**
- `profile-guided-section-prefix` (groups hot/cold functions for BOLT)
- `split-machine-functions` (provides initial hot/cold split that BOLT refines)
- `basic-block-sections` (Propeller-style, alternative to BOLT)

BOLT and Propeller are alternatives, not complements. BOLT works
post-link on any binary; Propeller requires recompilation with the
profile. For a Rust project, BOLT is easier to adopt.

## 9. Academic and research passes

### Published LLVM passes for data processing

**AoS-to-SoA conversion at IR level:**
No LLVM pass exists for this. The LLVM data layout model represents
struct types but does not transform aggregate layouts. MLIR's
`MemRefType` with affine maps can represent layout transformations,
but no LLVM IR pass performs AoS-to-SoA conversion.
Research exists in the polyhedral community (Bondhugula et al.) but
implementations target MLIR or custom IR, not LLVM.

**Cache-oblivious loop transformations:**
Polly implements cache-aware (not cache-oblivious) tiling using the
isl scheduler. No standalone LLVM pass implements cache-oblivious
algorithms. Research papers (Frigo et al., "Cache-Oblivious Algorithms")
describe algorithms, not compiler passes.

**SIMD optimization beyond built-in vectorizer:**
- **Minotaur** (Zhengyang Liu, LLVM Dev Meeting 2022): A SIMD-oriented
  superoptimizer that finds better SIMD instruction sequences for
  LLVM's portable vector operations and Intel AVX intrinsics. Research
  prototype, not a loadable plugin.
- **SEED Vectorizer** (research): Region-based SLP vectorizer. Inspired
  the Sandbox Vectorizer work in LLVM.
- **RV (Region Vectorizer)** (Saarland University): Whole-function
  vectorizer. Outer-loop vectorization support. Research prototype.
  Not maintained for recent LLVM versions.

**Prefetch optimization:**
No academic LLVM pass for profile-guided prefetch insertion was found.
The existing LoopDataPrefetch pass is the state of the art for LLVM.
Google's AutoFDO has discussed profile-guided prefetch but no public
LLVM pass implements it.

**Branch-free computation:**
No standalone LLVM pass converts branches to selects beyond what
SimplifyCFG and the loop vectorizer already do. Research on predicated
execution (Intel MPX, ARM predication) targets the backends, not
IR-level passes.

## 11. Pass ordering and interaction

### Does pass ordering matter?

Yes, significantly. LLVM's default pipeline ordering is the result of
years of tuning and represents a good general-purpose order. However,
for data-heavy workloads, specific reorderings can help.

### Key ordering rules for data processing

1. **Loop distribution BEFORE vectorization**: The default pipeline has
   the slot but disables it. Enabling `loop-distribute` places it
   correctly before `loop-vectorize`.

2. **Loop interchange BEFORE vectorization**: If the wrong loop is
   innermost, the vectorizer cannot vectorize it. Interchange must
   happen first.

3. **Loop fusion AFTER inlining**: Adjacent loops from inlined functions
   become visible only after the inliner runs. Fusion should run in
   Phase 5 (post-vectorization) or in a second pipeline run.

4. **Prefetch AFTER vectorization**: Prefetch the vectorized access
   pattern, not the scalar one. The prefetch distance calculation depends
   on the final loop body size.

5. **IRCE BEFORE vectorization**: Eliminating bounds checks before
   vectorization gives the vectorizer a cleaner loop body.

6. **Loop predication BEFORE vectorization**: Same reasoning as IRCE.

### Can `-C no-prepopulate-passes` outperform default?

Theoretically yes, practically dangerous. The default pipeline has
implicit dependencies (e.g., `loop-rotate` must run before `loop-vectorize`;
LCSSA form must be maintained). Removing prepopulated passes and
specifying your own requires deep knowledge of pass prerequisites.

A safer approach: keep the default pipeline and ADD extra passes:
```
-C passes=loop-interchange,loop-distribute,loop-data-prefetch
```
These run AFTER the default pipeline, which is suboptimal for some
(interchange should run before vectorization), but avoids breaking
implicit dependencies.

The ideal approach (if it were supported): use PassBuilder extension
points to inject passes at specific pipeline positions. This is how
LLVM plugins work in C++. Rust's `-C passes=` does not support
specifying insertion points.

### Known beneficial orderings for data-heavy code

From the LLVM pass ordering literature and empirical tuning:

1. **Standard O3 + distribution + interchange**:
   ```
   -C opt-level=3
   -C llvm-args=-enable-loop-distribute
   -C passes=loop-interchange
   ```
   Loop interchange runs after the default pipeline. Not ideal, but
   catches outer-loop vectorization opportunities.

2. **Aggressive vectorization + prefetch**:
   ```
   -C opt-level=3
   -C llvm-args=-force-vector-width=8
   -C llvm-args=-force-vector-interleave=4
   -C passes=loop-data-prefetch
   ```
   Forces wider vectors and deeper interleaving, then adds prefetch
   for the resulting access pattern.

3. **PGO-enhanced pipeline**:
   ```
   # First: build with instrumentation
   -C profile-generate=/tmp/pgo
   # Run representative workload
   # Merge profiles
   llvm-profdata merge -o merged.profdata /tmp/pgo
   # Rebuild with profile
   -C profile-use=merged.profdata
   -C llvm-args=-vectorize-with-block-frequency
   -C llvm-args=-split-machine-functions
   ```
