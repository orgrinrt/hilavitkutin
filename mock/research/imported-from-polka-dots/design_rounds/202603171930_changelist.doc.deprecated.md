independent of polka-dots. Two new crates in the mock workspace,
zero dependency on any polka-* crate. Designed for later extraction
to its own repository.

## Crate structure

```
mock/crates/hilavitkutin-api/   consumer-facing types, traits, platform contracts
mock/crates/hilavitkutin/       engine (scheduling, dispatch, codegen, platform impls)
```

Dependency graph:
```
consumer (polka-scheduler, saalis, loimu)
  └─ hilavitkutin
       └─ hilavitkutin-api
```

Consumers write against hilavitkutin-api (WorkUnit, Store, Context,
MemoryProvider, ThreadPool, Clock). The engine crate implements the
scheduling, dispatch, and codegen. It re-exports the api types.

hilavitkutin-api has zero dependencies. The engine depends on the api
crate only. No dependency on polka-*, std, alloc, or external crates.

Platform trait definitions (MemoryProvider, ThreadPool, Clock) live
in the api crate because consumers implement them. Platform tier
implementations (mmap wrapper, std::thread wrapper) live in the
engine crate behind feature gates.

### DESIGN.md.tmpl sections:

**§ Store types**
- `Resource<T>`: singleton, one value
- `Column<T>`: collection, N values, morsel-chunked
- `Virtual<T>`: zero-data, DAG edges only
- `StoreId(u16)`: dense index assigned at plan time
- `AccessMask(u64)`: bitmask for set operations (AND/OR)
- `AccessSet` trait on tuples: compile-time store descriptors
- Plan phase converts to runtime bitmasks via monomorphised fns

**§ WorkUnit trait**
- `WorkUnit<Schedule = Always>` with Read/Write associated types
- Schedule conditions: `Always`, `On<V>`
- Required declarations: read column set, write column set, order
- Write implies read (no redundant declaration)
- Authoring guideline: ≤8 unique columns per WU
- `HasCtx<S>` blanket supertrait for per-impl Context resolution
- Multi-impl types work: same struct, different Schedule params

**§ Context type**
- `Context<'a, R, W>` with `Contains<S>` bounds
- `&self` receiver (load-bearing: prevents noalias UB on fused WUs)
- `unsafe fn read<T>(&self, i: usize) -> T` — raw pointer, no bounds check
- `unsafe fn write<T>(&self, i: usize, val: T)` — raw pointer through &self
- No slice methods (from_raw_parts causes aliasing UB when WUs fused)
- Resource access: pointer to external storage, not inline in Context
- `ctx.each()` / `ctx.batch()` / `ctx.reduce()` op registration

**§ SlabColumn**
- Contiguous, fixed-stride, zero-copy-growth column
- Type-native stride: `size_of::<T>()` per element
- Backed by MemoryProvider
- Morsel-scoped views: `.morsel(start, end)` returns `&[T]`
- Push/len/as_slice/as_mut_slice

**§ Scheduling hints**
- Three axes: urgency, divisibility, significance
- ZST carriers with `const VALUE: u8`
- Per-axis value traits (UrgencyValue, DivisibilityValue, SignificanceValue)
- Tuple `(U, D, S)` implements SchedulingHint
- Built-in values: Immediate/Steady/Relaxed/Deferred, Atomic/Adaptive/
  Interruptible, Critical/Important/Normal/Opportunistic/Optional

**§ Virtual flags**
- Per-(virtual, consumer) epoch bytes
- fire / is_fired / clear_on_dispatch / advance_epoch
- fire_external for cross-thread signalling (atomic store)
- Epoch wraparound every 256 passes: full clear

**§ Version stamps**
- Parallel arrays `[u64; MAX_STORES]` for before/current comparison
- capture_before(write_mask) / changed(write_mask) → AccessMask
- increment(store_id)

**§ Platform traits**
- `MemoryProvider`: allocate(len, align), deallocate(ptr, len), protect(ptr, len, r, w)
- `ThreadPool`: spawn(FnOnce + Send), worker_count()
- `Clock`: now_ns() -> u64
- Trait-only; no implementations in this crate
- Consumers bring their own impls or use engine-provided defaults

## 2. CREATE mock/crates/hilavitkutin/

**New crate.** The execution engine.

### Files to create:

- `Cargo.toml` — depends on hilavitkutin-api only
- `src/lib.rs` — stub
- `README.md.tmpl` — 3-line summary
- `DESIGN.md.tmpl` — full template (below)

### DESIGN.md.tmpl sections:

**§ Overview**
- Morsel+chain+phase execution model
- Shared across polka-dots, saalis, loimu
- All scheduling computed at plan time from WU declarations
- No alloc, no heap in pipeline code; SlabColumn or stack arrays only

**§ Chain formation**
- Algorithm: Kahn's topo sort → greedy fusion walk
- Column-count limit: `min(L1_limit, register_budget)`
  - aarch64: ~14 unique cols, x86-64: ~10
  - L1_limit: `L1_usable / (MIN_MORSEL * sizeof(T))`
  - register_budget: GPRs - reserved (31-17=14 on aarch64, 16-6=10 on x86-64)
- Unique column count (not sum of per-WU counts)
- Fan-in breaks chain (predecessors in multiple chains)
- Inter-chain deps from WU-level predecessor analysis

**§ Morsel sizing**
- Formula: `(L1_usable / (chain_col_count * sizeof(T))).clamp(64, 8192) & !3`
- L1_usable: L1 * 0.75 (macOS 128K → 96K, Linux 32K → 24K)
- Write-only variant: use write_col_count instead of total (~5% gain)
- Per-chain morsel size (each chain has its own)
- Morsel = spatial window: rows × columns that fit in L1

**§ Phase decomposition**
- Waist detection: positions where concurrent path count is minimal
- Trunk identification: column-disjoint maximal paths within a phase
- Branch/bridge assignment: side paths, fan-in resolution
- Trunks run in parallel (zero sync, zero shared write columns)
- Bridges run after parent trunks complete (progress counter gated)
- Read-only shared columns between trunks: fine (reads don't conflict)

**§ Dispatch codegen**
- Monomorphised tuple dispatch (Approach A)
- Each chain = tuple of WU ZSTs implementing ChainDispatch
- LOCAL `&[WuFn]` slice constructed inside dispatch fn → LLVM devirtualises
- `#[inline(never)]` on chain dispatch, `#[inline(always)]` on WU functions
- ASM verification checklist: 0 blr, indexed addressing, 0 per-row [sp],
  morsel size constant-folded, 0 bl to WU functions
- What does NOT devirtualise: struct-field arrays, global static SLOTS,
  const-generic &[fn; N] params, run_chain(&[WuFn]) functions

**§ Compiled dispatch (per-core programs)**
- Each core gets a monomorphised function for its pipeline
- Baked-in: phases, row ranges, chain sequences, morsel sizes, sync points
- Stack AtomicUsize for phase sync (not Arc — zero heap alloc)
- Spin loop: isb + ldapr + cmp + branch (aarch64)
- Head+tail convergence: two threads process same chain from opposite ends
- Stack-local resource accumulators, merged after convergence

**§ Strategy selection**
- Row count thresholds: <10K sequential, >100K threading
- Pipeline shape: wide (adaptive), deep (sequential), mixed (phased)
- Depth-to-width ratio: `depth > chains/2 && roots ≤ 2` → sequential
- Per-project defaults: polka=seq, saalis=mixed, loimu=phased
- Frame budget: controls incrementality, not throughput

**§ Resource handling**
- Stack-local caching: snapshot resources before morsel loop
- Redirect res_ptr to stack array → LLVM promotes to registers
- Write back mutable resources after loop via stp
- Pointer separation from column storage (aliasing correctness)
- &self receiver on Context is load-bearing (prevents noalias)

**§ Deep-chain codegen**
- DSE: chain-internal columns → register locals, no memory ops
- Chain-input: read from memory at loop start
- Chain-output: written to memory at loop end
- Stores-at-end: all output stores grouped after computation
- Flattened dispatch: 0.95-1.00x hand-written
- Pipeline dispatch: 1.09-1.18x (store-reload between fused WUs)
- Multi-chain scenarios: 0.95-1.02x (chain boundaries are natural materialisation)

**§ What does NOT help**
- Row sorting (Morton, data-flow, state): 5-10% negative on synthetic data
- Explicit prefetch: ~2x slower on Apple Silicon
- likely/unlikely in dispatch loop: prevents LLVM vectorisation
- Deep-chase (both-ends CAS on serial deps): CAS contention exceeds savings

**§ Scheduler builder**
- `Scheduler::builder()` with const generics: MAX_UNITS, MAX_STORES, MAX_LANES
- `.add::<W>()` / `.add_on::<V, W>()` for work unit registration
- `.resource::<T>(value)` / `.column::<T>()` for store registration
- `.memory()` / `.threads()` / `.clock()` for platform injection
- `.build()` → computes DAG, plan, lane assignments

**§ Scheduling data structures**
- PlanState: read_masks, write_masks, dag adjacency (all bitmasks)
- PartitionMap: connected components via stack-allocated DFS
- ChainPlan: contiguous ranges of topo-ordered units, packed ops array
- MorselPlan: pre-partitioned row ranges per worker per chain
- All const-generic arrays, zero heap, ~30 KB total for 256 units

**§ Thread management**
- Pre-allocated pool, pool_size = physical_core_count
- Hybrid wake: spin 128 iterations → thread::park/unpark
- Core-pinned trunks for L1 warmth
- Leftover threads do convergence on heaviest trunk, then bridges

**§ Platform tier implementations**
- Feature-gated: `no_os` (default), `os`, `std`
- no_os: consumer must provide all platform impls via builder
- os: raw syscall mmap (Linux aarch64 svc, macOS extern C), future pthread
- std: std::alloc, std::thread, Instant::now wrappers
- Each tier gates impls behind `#[cfg(feature = "...")]`

**§ Definitive numbers (game world, 1M agents, 17 WUs, 36 cols, 2 threads)**

| layer | from → to | contribution |
|-------|-----------|--------------|
| devirtualisation (mono tuples) | 37→24ms | -35% |
| phase pipelining + convergence | 24→18ms | -25% |
| stack atomics (no Arc) | 18.2→17.7ms | -3% |
| write-only morsel sizing | 17.7→16.8ms | -5% |
