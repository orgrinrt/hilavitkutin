## 1. `define_resource!` -- Syntax, Generated Code, and Persistence

### Syntax

`define_resource!` is the mandatory way to declare a resource in loimu. Manual `impl Resource` is explicitly prohibited because it misses `HasShape`, `Registrable`, persistence configuration, and `inventory` registration. The macro lives in `loimu-resource-macros` and is re-exported by `loimu-resource`.

Basic syntax with persistence:

```rust
define_resource!(
    #[singleton]
    #[userconfig = "audio.toml"]
    AudioSettings {
        volume: Factor = Factor::from_f32(0.8),
        muted: Bool = Bool::FALSE,
    }
);
```

Owner-controlled write access:

```rust
define_resource!(
    #[singleton]
    #[owner(AudioBehavior)]
    AudioState {
        current_track: Option<TrackId> = None,
        playback_pos: Coord = Coord::ZERO,
    }
);
```

Temporal fields:

```rust
define_resource!(
    #[singleton]
    CameraState {
        #[temporal(cadence = RenderCycle)]
        position: Vec3<Coord> = Vec3::ZERO,
        zoom: Factor = Factor::ONE,
    }
);
```

### Generated code

For a `define_resource!` invocation, the proc macro generates all of the following:

1. **Struct** with all declared fields and their types.
2. **`Resource` trait impl** -- carries name and singleton flag.
3. **`StorageRecord` impl** -- field descriptors, temporal metadata. This delegates to `define_record!` internally.
4. **`HasShape` impl** -- field names, sizes, and offsets computed via `offset_of!`. This is used by the storage engine for columnar projection and by plugins for shape-based field access.
5. **`Registrable` impl** -- returns a `ResourceDescriptor` for the `inventory`-based registry system.
6. **`Default` impl** -- generated when all fields have defaults (explicit value, `Option` defaulting to `None`, or `Collection::new()`).
7. **`PersistableResource` impl** -- generated when any persistence attribute (`#[userconfig]`, `#[userdata]`, `#[cache]`, `#[systemconfig]`) is present. Returns a `PersistenceConfig` at zero runtime cost (all `const`).
8. **`inventory::submit!`** -- automatic registry discovery at link time.
9. **Field accessor methods** -- typed getters for each field.
10. **Field type validation** -- rejects forbidden collection types at compile time.

### Persistence attributes

Resource-level persistence attributes control where data is written to disk:

| Attribute | Directory | Default format |
|-----------|-----------|----------------|
| `#[userconfig]` | `config_dir/<app>/` | TOML |
| `#[userdata]` | `data_dir/<app>/` | bincode |
| `#[cache]` | `cache_dir/<app>/` | bincode |
| `#[systemconfig]` | platform system config | TOML |
| `#[appconfig]` | `app_config_dir/<app>/` | TOML |

Path overrides use `= "filename"` syntax: `#[userconfig = "audio.toml"]`.

### Persistence field control modes

Two mutually exclusive modes per resource:

**Opt-out mode** (default): all fields inherit the resource-level persistence location. Mark individual fields with `#[ephemeral]` to exclude them.

```rust
define_resource!(#[singleton] #[userconfig = "app.toml"] AppConfig {
    theme: Str = Str::from("dark"),
    #[ephemeral] session_token: Option<Str> = None,
});
```

**Opt-in mode**: triggered by the presence of any `#[persist]` field. Only marked fields are persisted; all others are implicitly ephemeral.

```rust
define_resource!(#[singleton] #[userdata = "state.bin"] GameState {
    #[persist] high_score: u32 = 0,
    #[persist] player_name: Str = Str::from("Player"),
    pass_counter: u32 = 0,  // not persisted
});
```

Mixing `#[persist]` and `#[ephemeral]` in the same resource is a **compile error**. The proc macro detects this and emits a clear diagnostic.

Per-field location overrides are supported -- a field can target a different persistence directory than the resource default:

```rust
define_resource!(#[singleton] #[userconfig = "app.toml"] MixedResource {
    settings: u32 = 0,              // inherits #[userconfig]
    #[cache] render_cache: u32 = 0,  // overridden to cache dir
    #[ephemeral] transient: bool = false,
});
```

### Hot reload

`#[hot_reload]` watches the persistence file for external changes. When the file changes on disk, the framework reloads the resource and emits `ResourceReloaded { resource_type, resource_name }`. Behaviors observing this signal can react.

```rust
define_resource!(#[singleton] #[cache = "render.bin"] #[hot_reload] RenderCache {
    width: u32 = 1920,
    height: u32 = 1080,
});
```

### PersistenceConfig

The generated `PersistableResource` implementation returns a `PersistenceConfig`:

```rust
pub struct PersistenceConfig {
    pub default_location: Option<PersistenceLocation>,
    pub default_file_path: Option<Str>,
    pub fields: &'static [FieldPersistence],
    pub hot_reload: bool,
    pub opt_in: bool,
}
```

`opt_in: false` means opt-out mode; the `fields` array contains only overrides and exclusions. `opt_in: true` means opt-in mode; `fields` contains only explicitly persisted fields. The entire config is `const` -- zero runtime cost.

### Load/save timing

- **Load**: persisted resources are loaded from disk during `ResourcesInitialized`, before `BehaviorsReady`.
- **Save**: written back on `RuntimeStopping`.
- **Hot reload**: file watchers run on a background thread; changes are applied at the start of the next schedule pass (before behaviors run).

## 2. `define_record!` -- Base Macro for Storage Records

`define_record!` is the lower-level primitive that `define_resource!` delegates to internally. It generates the `StorageRecord` trait implementation, which is the foundation for the storage engine's columnar layout.

Records are the persistent data model in loimu. Everything that lives in the storage engine -- resources, bindings, collection elements -- is a record. Signals are explicitly NOT records; they use arena-allocated ephemeral storage.

### StorageRecord trait

The `StorageRecord` trait provides the storage engine with the structural metadata it needs for columnar decomposition: field names, types, sizes, alignments, and offsets. `define_record!` generates this implementation.

### FieldDescriptor

Each field in a record produces a `FieldDescriptor` -- a compile-time structure containing the field name, type information, byte size, alignment, and offset within the struct. The storage engine uses these descriptors to build columnar layouts, perform projection (reading a subset of fields), and validate schema compatibility when plugins are loaded.

### Forbidden field types

The macro validates field types at compile time and rejects standard library collections that are not storage-engine-compatible:

| Forbidden | Replacement |
|-----------|-------------|
| `Vec<T>` | `Collection<T>` |
| `HashMap<K, V>` | `Dictionary<K, V>` |
| `HashSet<T>` | `Collection<T>` or `IdSet<T>` |
| `BTreeMap<K, V>` | `Dictionary<K, V>` |
| `VecDeque<T>` | `Collection<T>` |

The storage-engine replacements (`Collection`, `Dictionary`, `IdSet`) are columnar-friendly types that the engine knows how to decompose, version, and project.

## 3. `define_signal!` -- Signal Types, Buffer Modes, and Generated Code

### Signal model

Signals are typed ephemeral messages that drive the entire loimu runtime. Everything happens through signals -- no callbacks, no observers, no event buses. Signals are NOT records; they are arena-allocated, double-buffered, managed by the snap backend of the storage engine.

### Syntax

```rust
define_signal!(KeyPressed { key: Str } buffering: Queue);
define_signal!(MouseMoved { x: Coord, y: Coord } buffering: Replace);
define_signal!(NodeChanged { node: NodeId } buffering: ReplaceBy(node));
define_signal!(QuitRequested);  // unit signal, default Queue buffering
```

Signals can be zero-size (unit signals like `QuitRequested`) or data-carrying (like `KeyPressed`). Unit signals use `Queue` buffering by default.

### BufferMode

Three buffer modes control what happens when the same signal type is emitted multiple times within a single arena:

| Mode | Behavior on multiple emits |
|------|---------------------------|
| `Queue` | All preserved in emission order. Readers iterate all. |
| `Replace` | Last emission (per DAG order) wins. Earlier values overwritten in-place. |
| `ReplaceBy(field)` | Last emission per unique key field value wins. Keyed deduplication. |

Buffer modes apply **independently** to each arena (read and write). This prevents double-counting across pass boundaries.

### Generated code

For each `define_signal!` invocation, the macro produces:

1. **Struct** -- the signal type itself (zero-size for unit signals, data-carrying otherwise).
2. **`Signal` trait impl** -- sets `NAME` (const) and `BUFFER_MODE` (const).
3. **`HasShape` impl** -- for shape-based plugin projection.
4. **`Registrable` impl** -- returns `SignalDescriptor` for `inventory` registration.

### Why not hand-write `impl Signal`

```rust
// WRONG: manual impl loses Registrable, HasShape, and breaks registry consistency.
// Identity is SignalId (the interned u64 IS the signal's identity), not a string constant.
impl Signal for MySignal {
    const BUFFER_MODE: BufferMode = BufferMode::Queue;
}

// RIGHT: one line, everything generated
define_signal!(MySignal);
```

Hand-written impls miss `Registrable` (signal does not appear in registry), miss `HasShape` (plugin projection fails), and create maintenance burden when the `Signal` trait evolves.

### Double-buffered signal bus

The `SignalBus` maintains two arenas: a read arena (frozen, signals from previous pass) and a write arena (mutable, current pass emissions). At pass boundary, the write arena becomes the read arena (pointer swap, zero copy), and the old read arena resets (O(1) arena rewind).

Signals are `Clone + Send + Sync + 'static`. Each crate defines only its own domain signals. Identity is `SignalId` (interned u64), matched via equality, never string comparison.

### Settling guarantee

After initialization and all plugin loads complete, the signal bus arenas perform no new allocations during schedule passes. Pre-allocated `ErasedColumn` slots absorb all emissions via bump-pointer writes.

### Infrastructure macros in the signal crate

| Macro | Where | Purpose |
|-------|-------|---------|
| `define_signal!` | Signal type declarations | Generates Signal + HasShape + Registrable |
| `define_registry!` | `SignalRegistry` | ZST registry with `discover()` and HasName |
| `define_provider!` | `SignalEmit`, `SignalRead`, `SignalCount`, `SignalQuery` | Provider types for behavior contexts |
| `define_id!` | `SignalId` | Interned string ID type |

## 4. `define_behavior!` -- Handler Dispatch, Scheduling Units, Block Types, and Access Derivation

### Syntax and model

Behaviors are the single unit of application logic in loimu. There is no Responder, no FrameBehavior, no System -- just `Behavior`. Each behavior contains one or more `on Signal => name { body }` handler blocks. The macro lives in `loimu-behavior-macros` and is re-exported by `loimu-behavior`.

```rust
define_behavior!(FocusBehavior {
    on FocusRequest => focus_handler {
        let state = ctx.read::<FocusState>();

        each (spatial: Spatial2d, focus: &mut FocusTarget) {
            if spatial.bounds.contains(signal.position) {
                focus.focused = Bool::TRUE;
            }
        }

        ctx.write::<FocusNavigation>(|nav| {
            let next = nav.compute_next(signal.key);
            ctx.emit(FocusChanged { node: next });
        });
    }
});
```

`#[derive(Behavior)]` on an impl block is also supported. Both forms coexist; the DSL form is conventional for docs and examples.

### Complete keyword reference

| Keyword | Purpose | Scheduling unit? |
|---------|---------|-----------------|
| `on Signal => name { body }` | Signal handler block | Container |
| `ctx.read::<T>()` | Singleton resource read | No (captured) |
| `ctx.write::<T>(\|t\| { ... })` | Singleton resource write | No (captured) |
| `each (fields) { body }` | Collection iteration | Yes (fusible) |
| `tree<Marker> (node) { body }` | Tree traversal | Yes (non-fusible) |
| `bind Type { fields };` | GPU binding write | Yes |
| `compute { groups, local_size }` | GPU compute dispatch | Yes |
| `draw { target, instances }` | GPU draw dispatch | Yes |
| `copy { src, dst, size }` | GPU copy dispatch | Yes |
| `emit Type { fields };` | Signal emission | Yes |
| `let name = expr;` | Intermediate computation | No (scalar) |
| `anim name: ClipType;` | Referenced clip declaration | No |
| `anim_graph { ... }` | Persistent animation topology | Yes |
| `start(clip)` | Begin clip playback | No |
| `crossfade(a, b).over(dur)` | Cross-fade between clips | No |
| `spring_to::<R>(\|r\| r.field, val)` | Spring to target value | No |

### Generated code

For a `define_behavior!` invocation, the macro generates:

1. A **unit struct** (behaviors are stateless -- no fields).
2. **`Behavior` trait impl** with handler dispatch logic.
3. **Scheduling units** per `each`/`tree`/`emit`/`bind`/`compute`/`draw`/`copy` block.
4. **Internal subgraph edges** from access overlap analysis -- these are DAG edges between scheduling units within the same behavior, NOT unconditional sequential chains.
5. **Computed portals** -- claim and release points per resource, enabling early release by the scheduler.
6. **`Registrable` impl** with `BehaviorDescriptor`.
7. **Per-handler context types** restricting API access to the resources and signals declared in that handler.
8. **`HasName` impl** for registry integration.

### Scheduling unit decomposition

Behaviors are subgraphs in the main DAG, not monolithic nodes. Each `each`/`tree`/scalar/`emit` block becomes one scheduling unit (one DAG node). Internal nodes are connected by DAG edges derived from access analysis, not by unconditional sequential chains.

Example decomposition:

```
on PhysicsStep => physics_handler:
  Unit 0: each (pos, vel) { ... }     <- fusible
  Unit 1: emit Collision { ... }      <- standalone

on RenderFrame => render_handler:
  Unit 0: each (pos, color) { ... }   <- fusible
  Unit 1: bind ParticleBinding        <- standalone
  Unit 2: draw { ... }               <- standalone
```

Two scheduling units within the same behavior that touch different fields can be parallelized by the scheduler.

### Computed portals

Portals are exclusively a subgraph concept. Only subgraphs have portals; modules are linkable extensions that do not participate in the portal model.

- **Claim point**: first scheduling unit in the subgraph accessing resource R.
- **Release point**: last scheduling unit in the subgraph accessing resource R.
- External nodes can use R freely before the claim point and after the release point.

### Fusible vs opaque

These are separate concerns:

- **Fusible**: a storage engine question -- can the iteration passes of multiple scheduling units be merged into a single columnar scan?
- **Opaque**: a scheduler question -- can the scheduler see inside the scheduling unit and interleave its work with other units?

### Conditional guards

```rust
on Tick => update_handler {
    read config: AppConfig;

    if config.particles_enabled {
        each (pos: &mut Spatial2d, vel: Velocity) {
            pos.position += vel.value;
        }
    }
}
```

Guards must read only singletons or `let` bindings. The guarded `each` block is pessimistically allocated as a scheduling unit with full access declarations (even if the guard evaluates to false at runtime). Guard is evaluated once at dispatch time: true means the unit runs normally and is fusible; false means it returns immediately with zero iteration. The fusion engine excludes false-guarded units from fused groups.

## 5. The 16-Stage Parsing Pipeline

The `loimu-behavior-macros` proc macro processes `define_behavior!` input through a 16-stage pipeline. Each stage transforms an intermediate representation into a more refined form, culminating in the generated Rust code.

| Stage | Function | Input | Output |
|-------|----------|-------|--------|
| 1 | `parse_handler_list()` | Raw token stream | List of `on Signal => name { body }` blocks |
| 2 | `parse_handler_body()` | Handler body tokens | `each`, `tree`, `read`, `write`, `emit` patterns |
| 3 | `extract_each_access()` | `each (a: A, b: &mut B)` | Read A, Write B (per-parameter direction) |
| 4 | `extract_field_level_access()` | Body field access expressions | Per-field read/write sets (`FieldRef`) |
| 5 | `detect_access_modes()` | Wrapper types in parameters | Pinned, Sample, Post, Approx, Defer mode tags |
| 6 | `detect_guard_expressions()` | `if expr { each ... }` | Guard expression + singleton-only validation |
| 7 | `generate_scheduling_units()` | Analyzed handler body | Unit list at each/tree/scalar/emit boundaries |
| 8 | `generate_subgraph_edges()` | Unit access sets | Internal DAG edges (data-dependency, not sequential) |
| 9 | `compute_portals()` | Per-resource unit access | Claim and release points per resource |
| 10 | `generate_access_metadata()` | Body scan results | `observes_names`, `emits_names`, `access_set` |
| 11 | `parse_anim_declarations()` | `anim name: Type;` and inline blocks | Animation clip references |
| 12 | `parse_anim_graph()` | `anim_graph { ... }` | Blend/physics/IK node list |
| 13 | `generate_blend_graph()` | anim_graph DSL | `BlendGraph` construction code |
| 14 | `detect_animation_commands()` | `start()`, `.then()`, `crossfade()`, `spring_to()` | Animation command scheduling units |
| 15 | `generate_spring_setup()` | `spring()` in anim_graph | Temporal field annotation code |
| 16 | `parse_dispatch_blocks()` | `compute`/`draw`/`copy` blocks | `DispatchConfig` + `QueueAffinity` |

Stages 1-10 are the core pipeline for access analysis and scheduling unit generation. Stages 11-15 handle animation DSL features. Stage 16 handles GPU dispatch configuration.

The pipeline is strictly sequential: each stage consumes the output of the previous stage. No parallelism within the macro expansion itself -- parallelism is in the generated scheduling units at runtime.

## 6. Access Analysis and Subgraph Construction

### Access derivation rules

The proc macro scans each handler body for access patterns and derives scheduling metadata without any explicit access declarations from the user:

| Pattern | Derived access |
|---------|---------------|
| `each (a: A, b: B, c: &mut C) { ... }` | read A (Pinned), read B (Pinned), write C |
| `each (a: Immediate<A>) { ... }` | Immediate read A |
| `tree<M> (node) { ... }` | read tree M |
| `ctx.read::<R>()` | read singleton R |
| `ctx.write::<R>(\|t\| { ... })` | write singleton R (implies read) |
| `read name: R;` | read singleton R |
| `write name: R;` | write singleton R (implies read) |
| `emit Type { ... };` | emits signal Type |
| `bind Type { ... };` | write binding descriptor |
| `compute { ... }` | GPU compute dispatch, Compute queue |
| `draw { ... }` | GPU draw dispatch, Graphics queue |
| `copy { ... }` | GPU copy dispatch, Copy queue |

`&mut` in an `each` tuple indicates write access. Wrapper types (like `Pinned<T>`, `Immediate<T>`) encode direction; no `&`/`&mut` prefix is needed on wrappers. Bare `&T` in an `each` tuple maps to `Pinned<T>` (frozen snapshot, no ordering edge).

### Field-level inference

The macro does not stop at whole-record access -- it infers which specific fields within a record are actually read or written by analyzing the body expressions:

```rust
each (spatial: Spatial2d) {
    let pos = spatial.position;  // reads Spatial2d.position only
    // Spatial2d.velocity not accessed -> no DAG edge to velocity writers
}
```

This field-level granularity means that two behaviors accessing different fields of the same record produce no DAG edge and can run in parallel. This is a critical optimization for the scheduler.

### Pessimistic emit detection

All code paths within a handler body are scanned for `emit` statements. If any code path can emit a signal (even inside an `if` branch that might not execute), the behavior is marked as emitting that signal. This pessimistic approach ensures that the DAG always has the correct edges -- it may be overly conservative, but it is never wrong.

### Subgraph edge construction

Stage 8 (`generate_subgraph_edges()`) takes the access sets of all scheduling units within a behavior and constructs internal DAG edges. The key insight is that these are **data-dependency edges, not sequential chains**. Two scheduling units within the same behavior that have no overlapping access sets get no edge between them and can be scheduled in parallel.

Example: if unit 0 reads `Spatial2d.position` and unit 1 writes `Spatial2d.velocity`, there is no edge because the fields do not overlap. If unit 0 reads `Spatial2d.position` and unit 2 writes `Spatial2d.position`, there is a read-after-write edge from unit 2 to unit 0 (or write-after-read, depending on order).

### Three fusion tiers

| Tier | Access modes | Fusibility |
|------|-------------|------------|
| 1 (Unconditional) | Pinned, Approx, Defer | Always fusible -- no ordering constraints |
| 2 (Ordered) | Immediate, MutImmediate | Per-element order preserved |
| 3 (Chaos) | Sample, Post | Fusible, but relaxed ordering |

Two `MutImmediate` writers on the same column are NOT fusible -- they must be serialized.

### Sole-writer detection

The scheduler detects at plan computation time whether a resource has a single writer across all scheduling units in the entire DAG. Sole-writer resources use simpler rollback (snapshot restore instead of undo journal), which saves memory and computation.

## 7. Seven Access Modes

The access mode system is central to loimu's scheduling model. Each mode encodes direction (read vs write), freshness guarantee, ordering constraints, and determinism.

| Type | Direction | Freshness | Ordered | Deterministic | Default? |
|------|-----------|-----------|---------|---------------|----------|
| `Pinned<T>` | Read | Frozen snapshot | No | Yes | Default for reads |
| `Immediate<T>` | Read | Current frame | Yes | Yes | Opt-in |
| `&mut T` / `MutImmediate` | Write | Current frame | Yes | Yes | Default for writes |
| `Sample<T>` | Read | Current | No | No | Opt-in |
| `Post<T>` | Write | Current | No | No | Opt-in |
| `Approx<T>` | Read | Future | No | Yes | Opt-in |
| `Defer<T>` | Write | Deferred | No | Yes | Opt-in |

**`Pinned<T>`** is the default read mode. Bare `&T` in an `each` tuple maps to `Pinned<T>`. Reads a frozen snapshot of the data -- no ordering edge to writers, which maximizes parallelism.

**`Immediate<T>`** opts into current-frame ordered reads, which adds a DAG edge to the writer. Use when you need to see the most recent write from the current frame, in DAG order.

**`MutImmediate`** (or bare `&mut T`) is the default write mode. Current-frame ordered writes.

**`Sample<T>`** reads the current value without ordering guarantees. Non-deterministic -- the value depends on scheduling order.

**`Post<T>`** writes without ordering guarantees. Non-deterministic.

**`Approx<T>`** reads a future/predicted value. Deterministic but not necessarily accurate -- used for speculative computation.

**`Defer<T>`** writes a deferred value that will be applied later. Deterministic.

### Compile-time error table for invalid access mode combinations

| Pattern | Error message |
|---------|--------------|
| `&Pinned<T>` | "Pinned already implies read; remove the & prefix" |
| `&mut Approx<T>` | "Approx is read-only; use &mut T or Defer<T> for writes" |
| Reading `Post<T>` | Compile error (Post is write-only) |
| Reading `Defer<T>` | Compile error (Defer is write-only) |

## 8. Codepath Macros

Crate: `loimu-codepath-macros`. This is a proc macro crate that cannot be re-exported, so each consumer crate depends on it directly.

### `#[optimize_for(hot)]`

Applied to functions on hot paths (scheduler loop, behavior dispatch, signal emit/read, resource access). Behavior varies by build profile:

- **Debug**: wraps the function body with diagnostic timing and error context. Return type is `Outcome<T>` (full error handling).
- **Release (framework mode)**: strips all diagnostics, applies `#[inline(always)]`. Return type becomes `Just<T>` (errors panic).
- **Release (standalone mode)**: keeps `Result` return types (`Outcome<T>`) even in release -- panicking on error in a published library is unacceptable.

### `#[optimize_for(cold)]`

Applied to functions on cold paths (startup, teardown, registry validation, plugin loading, error formatting). Applies `#[cold]` and `#[inline(never)]`. Retains full diagnostics in all build profiles.

### Convention

Every function in the framework should be annotated with one of these two attributes. The annotation serves as documentation of the function's expected call frequency and as an optimization hint for the compiler.

## 9. Visibility Macros

Crate: `loimu-visibility-macros`. Like the codepath macros, this is a proc macro crate used directly by consumer crates.

### `#[public_api]` and `#[internal_api]`

| Attribute | Purpose |
|-----------|---------|
| `#[public_api]` | App developer / external consumer API surface |
| `#[internal_api]` | Framework plumbing shared between sibling crates |

Items with `pub(crate)` or `pub(super)` need no annotation -- they never cross crate boundaries.

### The `standalone` feature flag

Every loimu crate declares `standalone = []` in `[features]`. The default (absent) means the crate is used as a framework dependency.

| Attribute | `standalone` absent (framework dep) | `standalone` enabled (published) |
|-----------|--------------------------------------|----------------------------------|
| `#[public_api]` | `#[doc(hidden)] pub` | `pub` |
| `#[internal_api]` | `#[doc(hidden)] pub` | `pub(crate)` |
| `pub(crate)` | `pub(crate)` | `pub(crate)` |

**Framework mode** (default): all annotated items are accessible to sibling crates but hidden from documentation. The umbrella/facade crate re-exports selected items with proper documentation.

**Standalone mode**: public API items are fully visible and documented. Internal API items become `pub(crate)` and cannot cross crate boundaries.

### Expansion

```rust
#[public_api]
struct AudioSettings { ... }

// Expands to:
#[cfg(feature = "standalone")]
pub struct AudioSettings { ... }

#[cfg(not(feature = "standalone"))]
#[doc(hidden)]
pub struct AudioSettings { ... }
```

The decision happens at the consumer crate's compile time, not the proc macro's. Proc macros cannot read the consumer's feature flags directly, so both variants are emitted behind `cfg`.

### Interaction with codepath optimization

| Build | `standalone` absent | `standalone` enabled |
|-------|---------------------|----------------------|
| Debug | `Outcome<T>` (full error handling) | `Outcome<T>` |
| Release | `Just<T>` (errors panic) | `Outcome<T>` (keeps Result) |

The cfg guard for the stripped version: `#[cfg(all(not(feature = "standalone"), not(debug_assertions)))]`.

### `no-bare-pub` lint

Rejects any `pub` item without `#[public_api]` or `#[internal_api]`. Scoped visibility (`pub(crate)`, `pub(super)`) and `pub use` re-exports are exempt.

Exempt crates (proc macro crates that cannot use these attributes on their own exports): `loimu-visibility-macros`, `loimu-codepath-macros`, `loimu-resource-macros`.

### Umbrella crate documentation pattern

In framework mode, individual crate docs are hidden (`#[doc(hidden)]`). Documentation lives on the umbrella crate's re-exports:

```rust
// In loimu/src/lib.rs (umbrella)
/// State container for long-lived data accessed by behaviors.
///
/// Resources persist across schedules. Use `define_resource!` to declare.
pub use loimu_resource::Resource;
```

## 10. GPU and Render Macros

### `loimu-render-macros` -- ELIMINATED

This crate has been eliminated entirely:
- `define_render_pass!` is removed. Render passes are now GPU handlers (`bind`/dispatch keywords) inside `define_behavior!`.
- `define_binding!` has moved to `loimu-gpu-macros`.

### `define_binding!` (in `loimu-gpu-macros`)

Crate: `loimu-gpu-macros`, re-exported by `loimu-gpu`.

```rust
define_binding!(TransformBuffer {
    model_matrix: Mat4<Coord>,
    normal_matrix: Mat3<Coord>,
} buffering: Double);
```

Generated items:

1. **Struct** -- the binding type with declared fields.
2. **`Binding` impl** -- buffer strategy (e.g., `Double`), layout metadata.
3. **`StorageRecord` impl** -- descriptor record in the storage engine. Binding descriptors are records managed by the storage engine.
4. **`GpuSyncable` impl** -- descriptor-to-GPU-buffer transfer, fused into storage engine batch passes.
5. **`HasShape` impl** -- field layout for projection.
6. **`Registrable` impl** -- returns `BindingDescriptor`.

The separation between storage-engine-managed descriptor records and GPU-backend-managed device memory is explicit and intentional.

### GPU dispatch sub-keywords in `define_behavior!`

No GPU/CPU split from the scheduler's perspective. All handlers are scheduling units with access sets. GPU operations are DSL keywords within `define_behavior!`, not a separate system.

```rust
on RenderFrame => draw_handler {
    let particles = ctx.read::<ParticleState>();

    each (pos: Spatial2d, color: ParticleColor) {
        // CPU record iteration
    }

    bind ParticleBinding {
        positions: particles.positions,
        colors: particles.colors,
    };

    draw {
        target: "main_viewport",
        instances: particles.count,
    };
}
```

Three dispatch sub-keywords determine queue affinity:

| Sub-keyword | Queue | Config struct |
|-------------|-------|---------------|
| `compute { groups, local_size }` | Compute | `ComputeConfig` |
| `draw { target, instances }` | Graphics | `DrawConfig` |
| `copy { src, dst, size }` | Copy | `CopyConfig` |

The compositor reads `queue_affinity` from `SchedulingUnitDescriptor` and routes GPU work to the correct hardware queue. Behaviors can freely mix CPU record iteration, GPU binding writes, signal emission, and resource mutation in a single handler.

Target-conditional GPU pass merging: desktop systems run parallel queues; tiled/constrained GPUs merge compatible handlers.

## 11. Arithmetic Strategy Annotations

The `loimu-behavior-macros` proc macro translates DSL arithmetic operators (`+`, `*`, `-`) on fixed-point types to explicit method calls based on per-type or per-field annotations.

### Annotation types

| Annotation | Strategy |
|------------|----------|
| `#[hot]` | Hot-path arithmetic -- optimized for throughput, may sacrifice precision |
| `#[cold]` | Cold-path arithmetic -- full precision, no optimization shortcuts |
| `#[warm]` | Warm-path arithmetic -- balanced precision/performance |
| `#[precise]` | Precise arithmetic -- highest accuracy, may use higher-cost computation path |

### DSL translation

The macro reads the annotation during stage 5 (`detect_access_modes()`) of the parsing pipeline and uses `FixedPointConfig` to select the concrete method. For example, `a + b` on a `#[hot]`-annotated fixed-point field might translate to `a.add_into(&b)` using a fast approximation, while the same expression on a `#[precise]` field might translate to a full-precision addition path.

This translation is invisible to the behavior author -- they write natural arithmetic syntax, and the macro selects the appropriate implementation based on the annotation and the type system.

## 12. Compile-Time Constraint Enforcement

The macro system enforces a comprehensive set of compile-time constraints to prevent structural errors that would be difficult to diagnose at runtime.

### Complete forbidden pattern table

| Forbidden pattern | Error message / action |
|-------------------|----------------------|
| Nested `each` blocks | "nested each blocks are not supported; flatten into separate handlers" |
| `each` inside `tree` | "each inside tree is not supported; use separate handlers" |
| Guard reading per-element data | "guard must read only singletons or let bindings; move per-element logic inside each body" |
| `&Pinned<T>` | "Pinned already implies read; remove the & prefix" |
| `&mut Approx<T>` | "Approx is read-only; use &mut T or Defer<T> for writes" |
| Reading `Post<T>` | Compile error (Post is write-only) |
| Reading `Defer<T>` | Compile error (Defer is write-only) |
| `Vec<T>` in resource field | Compile error: use `Collection<T>` |
| `HashMap<K, V>` in resource field | Compile error: use `Dictionary<K, V>` |
| `HashSet<T>` in resource field | Compile error: use `Collection<T>` or `IdSet<T>` |
| `BTreeMap<K, V>` in resource field | Compile error: use `Dictionary<K, V>` |
| `VecDeque<T>` in resource field | Compile error: use `Collection<T>` |
| Mixing `#[persist]` and `#[ephemeral]` in same resource | Compile error with diagnostic |
| `pub` item without `#[public_api]` or `#[internal_api]` | `no-bare-pub` lint rejection |
| Manual `impl Behavior` | Convention prohibition (misses scheduling units, subgraph edges, portals) |
| Manual `impl Resource` | Convention prohibition (misses HasShape, Registrable, persistence, inventory) |
| Manual `impl Signal` | Convention prohibition (misses Registrable, HasShape, registry consistency) |

### Design rationale

Every constraint exists because the macro generates interconnected metadata that manual implementations would miss. A manually implemented `Resource` would lack `HasShape` (breaking columnar projection), `Registrable` (breaking `inventory` discovery), `PersistableResource` (breaking persistence), and `inventory::submit!` (breaking automatic registration). The constraints enforce that the entire system stays consistent.

## 13. Relevance to hilavitkutin

The loimu macro paradigm presents a clear design philosophy: **macros are not syntax sugar -- they are the only correct way to declare domain types**. This has several implications for hilavitkutin's Q8 (registration) design.

### What loimu's macros actually do

The `define_*` macros solve a specific problem: a single type declaration needs to produce multiple interconnected trait implementations, and missing any one of them breaks the system. A `define_resource!` call generates 7-10 items (struct, Resource, HasShape, StorageRecord, Registrable, Default, PersistableResource, inventory::submit!, field accessors, type validation). These are not independent -- they must be internally consistent. `HasShape` offsets must match the struct layout. `PersistableResource` must reference the correct fields. `Registrable` must produce a descriptor that matches the `HasShape` output.

### The tradeoff spectrum

At one extreme: mandatory macros for everything (loimu's approach). The user never writes a trait impl by hand. The macro guarantees consistency. The cost is opacity -- users cannot see what code is generated without expanding the macro, and debugging macro-generated code is harder than debugging hand-written code.

At the other extreme: manual trait implementations with convention enforcement via lints. The user writes `impl Registrable for MyType { ... }` by hand. A lint checks that all required traits are implemented. The cost is boilerplate and the risk of inconsistency between trait impls.

### Key questions for hilavitkutin

1. **How many traits does a single type declaration need to implement?** In loimu, a resource needs 7-10 impls. If hilavitkutin's registration requires fewer interconnected impls, manual trait implementation becomes more viable.

2. **How tightly coupled are the generated impls?** loimu's `HasShape` must be byte-offset-consistent with the struct layout, which is impossible to validate without proc macro access to the type's memory layout. If hilavitkutin's impls can be independently correct, macros are less critical.

3. **Is `inventory` the registration mechanism?** loimu uses `inventory::submit!` for link-time discovery. If hilavitkutin uses the same pattern, the macro provides value by automating the `submit!` call. If hilavitkutin uses a different discovery mechanism (e.g., explicit registration at startup), the automation value is lower.

4. **Does hilavitkutin need field-level scheduling metadata?** loimu's `define_behavior!` performs field-level access analysis (stage 4 of the 16-stage pipeline) to enable fine-grained parallelism. polka-dots as a build engine does not have a real-time scheduler with the same parallelism requirements. If field-level access metadata is not needed, the most complex part of loimu's macro system is irrelevant.

5. **Does hilavitkutin need persistence configuration?** loimu's `define_resource!` generates `PersistableResource` with `PersistenceConfig`. polka-dots has `polka-data` for persistence. The question is whether persistence metadata should be declared at the type level (loimu's approach) or configured externally (polka-data's approach).

### Recommendation considerations

The aspects of loimu's macro paradigm most likely relevant to hilavitkutin:

- **`inventory::submit!` automation** -- if hilavitkutin uses `inventory` for plugin discovery, a `#[register]` proc macro that generates the `submit!` call and a `Registrable` impl is high value.
- **`HasShape` generation** -- if hilavitkutin needs runtime reflection on type layout (for serialization, plugin ABI, or columnar storage), auto-generating shape metadata from the struct definition avoids manual offset computation.
- **Forbidden type validation** -- compile-time rejection of types that do not conform to the system's constraints (loimu rejects `Vec<T>` in favor of `Collection<T>`; hilavitkutin might reject types that are not `repr(C)` or not `Send + Sync`).

The aspects least likely relevant:

- **The 16-stage parsing pipeline** -- this is specific to loimu's real-time behavior scheduling model with `each`/`tree`/`emit` block decomposition. polka-dots does not have real-time scheduling.
- **Seven access modes** -- Pinned/Immediate/Sample/Post/Approx/Defer are specific to loimu's multi-writer concurrent scheduling. polka-dots build stages are sequential.
- **GPU dispatch sub-keywords** -- polka-dots is a dotfiles build engine, not a rendering engine.
- **Animation DSL stages** (11-15) -- domain-specific to loimu.

The middle ground -- potentially relevant depending on hilavitkutin's design decisions:

- **Signal/event declaration macros** -- if hilavitkutin has an event system, `define_signal!`-style macros that auto-register and auto-generate `HasShape` could be useful.
- **Visibility macros** -- the `standalone` feature flag pattern and `no-bare-pub` lint are good API hygiene practices that could apply to polka-dots' SDK/extension architecture.
- **Codepath annotations** -- `#[optimize_for(hot)]` and `#[optimize_for(cold)]` are build-profile-dependent optimizations that could apply to polka-dots' pipeline execution paths.
