# Research: Polly (LLVM Polyhedral Optimizer) and Rust-Generated LLVM IR

**Date:** 2026-03-13
**Type:** Research (not a design round topic)
**Context:** Evaluating whether Polly can optimize tight numeric loops in Rust codegen

---

## 1. Polly's SCoP Detection vs Rust IR

### What Polly needs

Polly detects Static Control Parts (SCoPs): loop nests where all control flow is
affine (loop bounds, branch conditions, array subscripts are linear functions of
induction variables and parameters). It models these as integer polyhedra using
the ISL library, then applies tiling, fusion, parallelisation, and vectorisation.

A region qualifies as a SCoP if and only if:

- Loops have computable trip counts (ISL must be able to express them)
- Memory accesses are affine (base + linear combination of induction vars)
- No function calls (except `readnone`/`readonly` intrinsics, memset, memcpy, memmove)
- No `alloca`, `landingpad`, `resume` instructions
- No indirect branches or irreducible control flow
- No volatile memory operations
- No unsupported data types (e.g. scalable vectors)

### What Rust generates that breaks SCoPs

Rust's safety guarantees produce LLVM IR patterns that are hostile to polyhedral
analysis. The specific constructs are:

**Bounds checks (the primary killer).** Every `slice[i]` generates:
```llvm
%cmp = icmp uge i64 %idx, %len
br i1 %cmp, label %panic, label %continue
```
This creates a conditional branch inside the loop body. The panic branch
contains a call to `core::panicking::panic_bounds_check` which is a
non-returning function call. Polly rejects function calls in SCoPs
(`ReportFuncCall`), and non-affine branches (`ReportNonAffBranch`).

**Panic/abort calls.** Overflow checks (`checked_add`, `checked_mul`),
`unwrap()`, `expect()`, `Option` matching, and `Result` error paths all
generate calls to panic functions (`@_ZN4core9panicking5panic17h...`).
These are `noreturn` calls followed by `unreachable` terminators.

**Landing pads and unwinding.** When panic=unwind (the default), Rust
generates `invoke` instructions instead of `call`, with `landingpad`
blocks for unwinding. Polly explicitly rejects `LandingPadInst` and
`ResumeInst`.

**Iterator machinery.** Rust iterators like `.iter().enumerate()` or
`.chunks()` compile through multiple layers of generic monomorphisation.
After inlining and optimisation, simple cases reduce to clean loops, but
complex iterator chains often leave residual pointer arithmetic, phi
nodes, and select instructions that prevent SCEV (Scalar Evolution)
from computing affine trip counts. Polly reports
`Non affine loop bound '***COULDNOTCOMPUTE***'` when SCEV fails.

**Inclusive ranges.** `0..=n` generates a conditional increment to handle
`n == u32::MAX` without overflow. This produces non-canonical loop forms
that LLVM's loop analysis has trouble with. `0..n` is much better
optimised (confirmed by Popov, LLVM Dev Meeting 2024 slide 35).

**Enum discriminants.** Match expressions on enums produce `switch`
instructions on discriminant values. Polly validates switches via
`isValidSwitch` but requires affine conditions.

### What Rust patterns CAN produce Polly-friendly IR

**Simple numeric `for` loops with `get_unchecked`.** This is the most
reliable path:
```rust
unsafe {
    for i in 0..n {
        *dst.get_unchecked_mut(i) = *src.get_unchecked(i) * 2.0;
    }
}
```
This eliminates bounds checks, producing a clean induction variable with
affine access `base + i * sizeof(T)`. The resulting IR has a phi node,
icmp, conditional branch to exit, and GEP with the induction variable.

**`chunks_exact()` with simple inner bodies.** `chunks_exact(N)` where N
is a compile-time constant produces loops where LLVM can prove the
chunk size, eliminating internal bounds checks. The inner loop over
the fixed-size slice often has no bounds checks after optimisation.

**`#[repr(C)]` struct arrays with pointer arithmetic.** When structs have
known, fixed layout, LLVM can compute GEP offsets as affine functions
of the induction variable.

**`iter().zip()` on slices of equal known length.** After inlining, zip
on two slices with matching lengths can sometimes reduce to a clean
indexed loop, but this is fragile and depends on LLVM's ability to
prove the length relationship.

**`#[inline(always)]` on hot inner functions.** Inlining is critical
because Polly needs to see the entire loop body in one function.
Without inlining, the call instruction kills the SCoP.

### Prior art: Julia and Chapel

The Julia GSoC 2016 project (Reisinger) found identical problems. Julia's
bounds checks prevented Polly from detecting SCoPs. The solution was to
disable bounds checks (`@inbounds`) for Polly-targeted code. Even then,
Julia's loop iteration via StepRanges produced non-affine trip counts
that SCEV could not compute. Additional LLVM canonicalisation passes
(loop-rotate, instcombine) were needed to get Polly to recognise loops.

The Chapel GSoC 2018 project found that `-polly-invariant-load-hoisting`
was needed to handle Chapel's array descriptor loads, but this only
worked for simple 0-based arrays, not Chapel's general domain-indexed
arrays.

**No one has systematically studied Rust idioms for Polly compatibility.**
The Rust community has not invested in this because Polly is not shipped
with rustc and requires custom builds.

### Polly's error block heuristic

Polly has a built-in heuristic for exactly the bounds-check pattern.
The `isErrorBlock` function identifies basic blocks that:

1. Terminate with `unreachable`, OR
2. Contain a call to a non-pure, non-returning function that is
   conditionally executed (not dominated by the loop header)

When `-polly-allow-error-blocks` is enabled (it is ON by default), Polly
treats these blocks as "error blocks" that can be speculatively assumed
to never execute. The error block's domain is set to empty, effectively
removing it from the polyhedral model.

This means Rust's bounds-check pattern (`icmp` + branch to
`panic_bounds_check` + `unreachable`) should in theory be handled: the
panic block qualifies as an error block (noreturn call + unreachable).
However, this only works if:

- The bounds check branch is the ONLY non-affine element in the region
- The panic call is in a separate basic block (not fused with loop body)
- LLVM's earlier passes have not already mangled the control flow

In practice, after LLVM's standard optimisation passes run before Polly,
the IR is often restructured in ways that break this heuristic. The
Optimistic Loop Optimization paper (Grosser et al., CGO 2017) showed
that optimistic assumptions can increase analysable SCoPs by 3.9x on
SPEC/NPB benchmarks, but this was tested on C code, not Rust.

### Predicting Polly compatibility from Rust source

There is no tool that analyses Rust source or MIR and predicts Polly
compatibility. The practical approach is:

1. Emit LLVM IR: `rustc --emit=llvm-ir -C opt-level=3 -C codegen-units=1`
2. Run Polly's analysis passes on that IR (see section 3)
3. Examine the rejection diagnostics

MIR patterns that suggest Polly-friendly LLVM IR:
- No `assert` or `panic` in the loop body
- No `match` or `if let` on `Option`/`Result`
- No function calls (everything inlined)
- Simple induction variable (single `i += 1`)
- Array accesses via raw pointer or `get_unchecked`

MIR patterns that predict SCoP rejection:
- `BoundsCheck` MIR statements
- `Call` terminators (especially panic-related)
- `SwitchInt` on non-induction-variable values
- `Unwind` or `Resume` cleanup paths

## 2. Polly as Built-in Passes vs Plugin

### Building rustc with Polly

Rust's bootstrap system supports Polly via `bootstrap.toml`:

```toml
[llvm]
polly = true
```

This passes `-DLLVM_POLLY_LINK_INTO_TOOLS=ON` to LLVM's CMake, which
statically links `libPolly.a` and `libPollyISL.a` into the LLVM libraries
that rustc links against. The Polly passes are then registered in LLVM's
pass registry at startup.

**Known issue:** rust-lang/rust#110787 reports that `build.rs` in
`rustc_llvm` incorrectly links Polly libraries as `dylib` when other
LLVM libraries are linked statically, causing undefined symbol errors.
The fix is to ensure Polly libraries use the same link kind as other LLVM
libraries. As of 2026, this issue may or may not be resolved in trunk.

### Using Polly passes after static linking

Once Polly is statically linked into rustc's LLVM, the passes are
available through `-C llvm-args`:

```bash
# Enable Polly optimisation
rustc -C llvm-args=--polly file.rs

# Polly with strip-mining vectorisation
RUSTFLAGS='-C llvm-args=--polly -C llvm-args=--polly-vectorizer=stripmine' cargo build --release

# Polly with parallelisation (requires OpenMP runtime at link time)
RUSTFLAGS='-C llvm-args=--polly -C llvm-args=--polly-parallel' cargo build --release
```

This was enabled by PR #78566 (merged November 2020). Before that PR,
`-C llvm-args=--polly` was silently ignored.

### Using `-C passes=` for Polly

The `-C passes=` flag feeds pass names into LLVM's new pass manager
pipeline parser. Polly registers its passes through the plugin API:

```cpp
llvm::PassPluginLibraryInfo getPollyPluginInfo() {
  return {LLVM_PLUGIN_API_VERSION, "Polly", LLVM_VERSION_STRING,
    polly::registerPollyPasses};
}
```

When Polly is statically linked, these passes should be available. The
pass names registered for the new PM (from `PollyPasses.def`) include:

- `polly-detect` (SCoP detection)
- `polly-scops` (polyhedral description)
- `polly-opt-isl` (ISL-based optimisation)
- `polly-codegen` (LLVM IR generation)
- `polly-simplify`, `polly-optree`, `polly-delicm`, `polly-prune-unprofitable`
- Print passes: `polly-print-detect`, `polly-print-scops`, etc.

So in theory:

```bash
rustc -C passes=polly-detect,polly-scops,polly-opt-isl,polly-codegen file.rs
```

However, the interaction between rustc's pass pipeline and Polly's pass
registration is not well-tested. The `-C llvm-args=--polly` approach is
more reliable because it triggers Polly's own pipeline insertion logic
in `RegisterPasses.cpp`.

### Loading Polly as a dynamic plugin

If Polly is NOT statically linked (i.e. `llvm.polly = false` or default),
you can load it dynamically on nightly:

```bash
RUSTFLAGS="-Z llvm-plugins=/path/to/LLVMPolly.so -C llvm-args=--polly" cargo +nightly build
```

This requires:
- Nightly rustc (the `-Z llvm-plugins` flag is unstable)
- A `LLVMPolly.so` built against the exact same LLVM version as rustc
- LLVM must have been built with `LLVM_ENABLE_PLUGINS=ON`

This is fragile. LLVM plugin ABI compatibility is not guaranteed across
minor versions. Building Polly against a different LLVM version than
rustc's bundled LLVM will crash.

### Current state of Polly in official rustc

Polly is NOT enabled in any official Rust release (stable, beta, or
nightly). The `llvm.polly` config defaults to `false`. To use Polly, you
must build rustc from source with `llvm.polly = true`.

## 3. Polly Diagnostic/Analysis Mode

### Analysis-only invocation

Polly can run detection and analysis without transforming code. There are
two approaches:

**Approach A: `-polly-no-codegen` flag.**
Runs Polly's full pipeline (detect, model, optimise) but skips IR
generation. Useful to see what Polly would do:

```bash
RUSTFLAGS='-C llvm-args=--polly -C llvm-args=--polly-no-codegen' cargo build --release
```

**Approach B: Run individual analysis passes via `opt`.**
Extract the LLVM IR and run Polly's passes manually:

```bash
# Step 1: Emit LLVM IR from rustc
rustc --emit=llvm-ir -C opt-level=3 -C codegen-units=1 -C debuginfo=0 file.rs

# Step 2: Run Polly SCoP detection on the IR (legacy PM)
opt -polly-detect -polly-scops -analyze file.ll

# Step 2 alt: New pass manager
opt -passes='polly-print-detect' file.ll -disable-output

# Step 3: View detected SCoPs
opt -polly-scops -analyze -polly-process-unprofitable file.ll
```

The `-polly-process-unprofitable` flag tells Polly to report even SCoPs
it considers too small to be worth optimising. Without it, Polly's
profitability heuristic (`isProfitableRegion`, `hasSufficientCompute`)
silently drops many detected SCoPs.

### Diagnostic flags

All of these are passed via `-C llvm-args=` when using rustc:

| Flag | Effect |
|------|--------|
| `-polly-show` | Open graphviz window showing detected SCoPs |
| `-polly-show-only` | Show SCoPs without detailed statements |
| `-polly-dot` | Write `.dot` files for each function's SCoPs |
| `-polly-dot-only` | `.dot` files showing basic blocks only |
| `-polly-export-jscop` | Write JSON SCoP descriptions (`.jscop` files) |
| `-polly-only-func=REGEX` | Limit analysis to matching function names |
| `-polly-dump-before-file=FILE` | Dump IR entering Polly to a file |
| `-polly-no-codegen` | Run analysis and optimisation but skip codegen |
| `-polly-optimizer=none` | Skip optimisation, only detect and model |
| `-polly-process-unprofitable` | Analyse even small/unprofitable SCoPs |
| `-polly-allow-nonaffine` | Allow non-affine subscripts (conservative) |
| `-polly-invariant-load-hoisting` | Hoist loop-invariant loads (helps with struct field access) |

### Using LLVM optimisation remarks

LLVM's remark system captures Polly's diagnostic output:

```bash
# Write remarks to YAML files
RUSTFLAGS='-Cdebuginfo=1 -Cremark=all -Zremark-dir=/tmp/remarks -C llvm-args=--polly' \
  cargo +nightly build --release

# Or just stderr output for a specific pass
RUSTFLAGS='-Cremark=polly -C llvm-args=--polly' cargo +nightly build --release
```

The `-Cremark=polly` filter should capture Polly's optimisation remarks
(applied, missed, analysis). The `-Zremark-dir` flag writes them as YAML
files compatible with LLVM's `opt-viewer.py` tool.

The `cargo-remark` tool can visualise these:

```bash
cargo install cargo-remark
RUSTFLAGS='-C llvm-args=--polly' cargo remark build
```

This generates a static HTML site showing which functions Polly analysed,
which SCoPs it found, and why it rejected others.

### Rejection diagnostics

Polly's rejection tracking produces specific reason types:

| Rejection class | Meaning in Rust context |
|----------------|------------------------|
| `ReportFuncCall` | Bounds check panic call, unwrap, any non-inlined fn |
| `ReportAlloca` | Stack allocation in loop (rare after optimisation) |
| `ReportNonAffBranch` | Bounds check conditional branch |
| `ReportNonAffineAccess` | Complex index expression SCEV cannot analyse |
| `ReportLoopHasNoExit` | Infinite loop (rare in Rust) |
| `ReportLoopHasMultipleExits` | Early return, break, or `?` operator in loop |
| `ReportAlias` | Pointer aliasing that runtime checks cannot resolve |
| `ReportIncompatibleType` | Unsupported LLVM type in loop body |

These appear in the remark output with source locations when
`-Cdebuginfo=1` is set.

## 4. MIR Analysis for Optimisation Prediction

### Dumping MIR

```bash
# Dump all MIR passes
rustc -Z dump-mir=all -Z dump-mir-dir=/tmp/mir file.rs

# Dump specific function
rustc -Z dump-mir=my_function file.rs

# Dump unoptimised MIR
rustc -Z mir-opt-level=0 --emit=mir file.rs

# Dump optimised MIR
rustc -C opt-level=3 --emit=mir file.rs
```

### MIR patterns that predict Polly compatibility

There are no existing tools that analyse MIR for polyhedral optimisation
potential. However, specific MIR patterns are strong predictors:

**Good (likely Polly-friendly after LLVM lowering):**

```
// Simple induction variable loop
bb1: {
    _3 = Lt(_1, _2);
    switchInt(move _3) -> [0: bb4, otherwise: bb2];
}
bb2: {
    _5 = (*_4)[_1];      // unchecked access
    _1 = Add(_1, const 1);
    goto -> bb1;
}
```

**Bad (will break SCoPs):**

```
// Bounds-checked access
bb2: {
    _6 = Len(*_4);
    _7 = Lt(_1, _6);
    assert(move _7, "index out of bounds") -> bb3;
}
```

The `assert` in MIR lowers to an `icmp` + `br` + call to
`panic_bounds_check` in LLVM IR. This is the single most common SCoP
killer in Rust code.

### Relevant rustc flags for LLVM analysis

| Flag | Purpose |
|------|---------|
| `--emit=llvm-ir` | Dump LLVM IR (`.ll` files) |
| `--emit=llvm-bc` | Dump LLVM bitcode (`.bc` files) |
| `-C no-prepopulate-passes` | See raw IR before LLVM optimisation |
| `-C save-temps` | Save all intermediate files |
| `-Z print-llvm-passes` | Print which LLVM passes run |
| `-Z time-llvm-passes` | Time each LLVM pass |
| `-C llvm-args=-print-after-all` | Print IR after every pass |
| `-C llvm-args=-filter-print-funcs=NAME` | Filter IR printing to one function |
| `-C llvm-args=-opt-bisect-limit=N` | Binary search for which pass causes a problem |
| `-Cremark=all` | Print all optimisation remarks to stderr |
| `-Zremark-dir=DIR` | Write remarks as YAML files |

### Practical workflow for auditing Polly potential

1. Build rustc with `llvm.polly = true`
2. Compile the target crate:
   ```bash
   RUSTFLAGS='-C opt-level=3 -C codegen-units=1 -C llvm-args=--polly \
     -C llvm-args=--polly-no-codegen -C llvm-args=--polly-process-unprofitable \
     -C llvm-args=--polly-export-jscop -Cdebuginfo=1 -Cremark=polly' \
     cargo build --release
   ```
3. Examine rejection remarks in stderr output
4. For functions of interest, extract LLVM IR and run analysis manually:
   ```bash
   opt -polly-scops -analyze -polly-process-unprofitable \
       -polly-only-func=target_function target.ll
   ```
5. Visualise detected SCoPs:
   ```bash
   opt -polly-dot -polly-only-func=target_function target.ll
   dot -Tpng scops.target_function.dot -o scops.png
   ```

## 5. Summary of Findings

### What works

- Polly CAN be used with Rust if you build rustc from source with `llvm.polly = true`
- `-C llvm-args=--polly` enables Polly in the LLVM pipeline (since PR #78566, 2020)
- Polly's error block heuristic can, in theory, handle Rust's panic branches
- Analysis-only mode works: `-polly-no-codegen`, `-polly-export-jscop`, remark system
- `get_unchecked` loops with simple induction variables produce clean affine IR

### What does not work

- No Polly in any official rustc distribution (requires custom build)
- Rust's bounds checks are the primary SCoP killer; LLVM's earlier passes
  often restructure the IR before Polly sees it, defeating the error block heuristic
- Iterator adaptor chains produce non-affine trip counts that SCEV cannot compute
- Inclusive ranges (`0..=n`) produce non-canonical loop forms
- `panic=unwind` (default) generates landing pads that Polly rejects outright
- No tooling exists to predict Polly compatibility from Rust source or MIR

### What nobody has studied

- Systematic measurement of which Rust patterns produce Polly-analysable IR
- Whether `panic=abort` (eliminates landing pads) significantly improves SCoP detection
- The interaction between Rust's MIR optimisations and Polly's canonicalisation needs
- Whether `-polly-invariant-load-hoisting` helps with Rust's slice fat pointer loads
- Performance impact of Polly on real Rust numeric workloads

### Key insight for polka-dots

If we need Polly-optimisable loop nests in our execution engine, the
generated Rust code must use `unsafe` + `get_unchecked` + `panic=abort`
+ `#[inline(always)]` for hot inner loops. This is the only reliable
path to producing affine LLVM IR that Polly can analyse. Iterator
abstractions and safe indexing are fundamentally incompatible with
polyhedral analysis as Polly is implemented today.

The alternative is to bypass Polly entirely and do our own loop tiling
and vectorisation at the IR level (our morsel/pipeline scheduler
already has the information needed for data-locality optimisation).
