## 1. Rust Registration Crates

Three crates dominate the "collect things across crate boundaries at link time"
space in Rust. Each takes a different approach to the same underlying problem:
how do you declare a value in one crate and discover it in another without
requiring a central registry file that both crates import?

### inventory

`inventory` is the most widely used approach for type-safe global registration.
The API surface is small. A type implements `inventory::Collect`, then any crate
in the dependency graph can call `inventory::submit!(MyType { ... })` at module
scope. At runtime, `inventory::iter::<MyType>()` returns an iterator over all
submitted instances.

The mechanism relies on linker sections. On Linux, submitted items are placed in
`.init_array` sections. On macOS, they go into `__DATA,__mod_init_func`. On
Windows, the `.CRT$XCU` section is used. At process startup, the C runtime
iterates these sections and calls the initializer functions, which register the
items into a global linked list. `inventory::iter` walks this list.

The key properties:

- **Type safety.** `inventory::submit!` only accepts types that implement
  `Collect`. The iterator is typed. No casting, no string keys.
- **Cross-crate.** Works across any number of crates in the dependency graph.
  A plugin crate can submit a value and the host crate discovers it without
  any explicit wiring.
- **`'static` requirement.** All submitted items must be `'static`. The items
  live in global memory for the lifetime of the process. There is no
  `inventory::remove`.
- **No ordering guarantee.** The order in which `inventory::iter` yields items
  is determined by linker section ordering, which varies across platforms and
  link orders. Consumers must not rely on iteration order.
- **No `no_std`.** The implementation depends on `std` for the linked list and
  `Once` synchronization. Bare-metal or `no_std` targets cannot use it.
- **Low overhead.** Each submitted item adds one function pointer to a linker
  section and one node to a linked list. Iteration is O(n) with no allocation.

polka-dots uses `inventory` internally. The `#[register]` proc macro in
`polka-derive` generates an `inventory::submit!` call for each registered trait
implementation. loimu uses `inventory` for all `define_*` macros: every
`define_resource!`, `define_signal!`, `define_behavior!`, and `define_binding!`
invocation generates an `inventory::submit!` with the corresponding descriptor
type (`ResourceDescriptor`, `SignalDescriptor`, `BehaviorDescriptor`,
`BindingDescriptor`).

### linkme

`linkme` takes a different API approach to the same linker-section mechanism. It
exposes a `#[distributed_slice]` attribute that declares a static slice, and
items are appended to the slice from any crate:

```rust
#[distributed_slice]
pub static PLUGINS: [fn() -> &'static str];

#[distributed_slice(PLUGINS)]
static MY_PLUGIN: fn() -> &'static str = || "my_plugin";
```

At runtime, `PLUGINS` is a regular `&[fn() -> &'static str]` containing all
items from all crates. The implementation uses the same linker sections as
`inventory` but presents them as a contiguous slice rather than a linked list.

Key differences from `inventory`:

- **Slice access.** The result is a `&[T]` rather than an iterator over a
  linked list. This enables O(1) indexing and length queries. For small
  collections (typical in registration scenarios), the difference is
  negligible.
- **Slightly lower overhead.** No linked list nodes. Items are contiguous in
  the linker section. One fewer indirection per item during iteration.
- **Partial `no_std` support.** The core mechanism does not require `std` on
  some platforms (Linux, macOS). The portability story is more nuanced than
  `inventory`'s flat "requires `std`" stance.
- **Same ordering caveat.** Linker section order is platform-dependent.
  Consumers must not rely on the order of items in the slice.
- **Platform dependencies.** Requires specific linker support. The
  `#[used(linker)]` attribute and section placement differ across platforms.
  `linkme` handles this internally but the platform coverage is narrower than
  `inventory`'s.

Neither polka-dots nor loimu uses `linkme`. The `inventory` API is marginally
more ergonomic for the trait-registration pattern (submit a value, iterate
values) compared to `linkme`'s slice pattern (declare a slice, append to the
slice). The performance difference is immaterial for registration counts in the
tens to hundreds.

### ctor

`ctor` provides raw static constructors. A function annotated with `#[ctor]`
runs before `main()`:

```rust
#[ctor]
fn init() {
    GLOBAL_REGISTRY.lock().unwrap().push(MyPlugin);
}
```

This is the lowest-level approach. It provides no type safety, no discovery API,
and no cross-crate coordination beyond "your function runs at startup."

Key properties:

- **No type safety.** The constructor is a raw `fn()`. What it does is up to
  the author. No compile-time check that it registers the right type.
- **Order is undefined.** Constructor execution order across crates (and even
  within a single crate on some platforms) is unspecified. If constructor A
  depends on state initialized by constructor B, the program has a race
  condition.
- **`no_std` support.** Works on bare-metal targets. The mechanism is the same
  C-runtime `.init_array` approach, but without the `std` wrapper.
- **Dangerous for DAG construction.** If registration order matters (because
  you need to detect duplicates, validate references between registered items,
  or build a dependency graph), `ctor`'s undefined ordering is a liability.
  You would need a separate validation pass after all constructors have run.

Not suitable for work unit registration. The lack of type safety and undefined
ordering make it a poor fit for building a validated dependency graph at startup.

### Comparison table

| Feature | inventory | linkme | ctor |
|---|---|---|---|
| Type safety | Yes | Yes (typed slice) | No (raw fn) |
| `no_std` | No | Partial | Yes |
| Cross-crate | Yes | Yes | Yes |
| Ordering | Undefined | Undefined | Undefined |
| Overhead | Low | Lower | Lowest |
| API shape | Submit + iterate | Distributed slice | Raw constructor |
| Used by polka | Yes | No | No |
| Used by loimu | Yes | No | No |

All three share the same fundamental limitation: ordering is undefined. For DAG
construction, this is acceptable because the DAG itself defines the ordering.
Registration order does not need to match execution order. The registrations are
just declarations; the scheduler sorts them after discovery.

## 2. ECS Registration Patterns

Entity Component System architectures solve a registration problem at two
levels: component types (what data exists) and systems (what logic runs).
Both are relevant to hilavitkutin's work unit registration design.

### Bevy

Bevy is the most prominent Rust ECS framework and its registration model is
the closest analog to what hilavitkutin faces.

**Plugin system.** `App::add_plugins(MyPlugin)` is the top-level registration
point. A plugin is a struct that implements the `Plugin` trait with a single
method: `fn build(&self, app: &mut App)`. Inside `build()`, the plugin registers
systems, resources, events, and other plugins. This is explicit registration:
the developer writes `app.add_systems(Update, my_system)` for each system.

**System registration.** Systems are functions with typed parameters. Bevy
extracts the parameter types at registration time (via the `SystemParam` trait)
and uses them to build a dependency graph. The key insight: the function
signature IS the access declaration. `fn my_system(query: Query<&Position,
&mut Velocity>)` declares that it reads `Position` and writes `Velocity`. No
separate access declaration is needed.

**Automatic scheduling.** Bevy's scheduler resolves system dependencies from
their `Query` parameters at runtime. Two systems that both write the same
component type cannot run in parallel; two systems that read different
components can. This is the same model as loimu's access-based DAG construction
and hilavitkutin's planned Read/Write design.

**Ordering constraints.** The automatic scheduling handles parallelism, but
sometimes explicit ordering is needed. Bevy provides:

- `.before(other_system)` / `.after(other_system)` for pairwise ordering.
- System sets: named groups with ordering constraints applied to the entire
  group.
- Run conditions: `.run_if(in_state(GameState::Playing))` to conditionally
  skip systems.

**Ambiguity detection.** Bevy warns at startup when two systems could conflict
(both access the same component with at least one writer) but no ordering
constraint exists between them. This is a diagnostic, not an error. The
developer can add ordering constraints or suppress the warning.

**Component registration.** Automatic on first use. When a component type
appears in a `Query` for the first time, Bevy creates storage for it. No
explicit `register_component::<T>()` call. This is possible because Bevy uses
`TypeId` internally for component identification. (Note: polka-dots bans
`std::any` / `TypeId`. This approach is not directly portable.)

**Resource registration.** `App::insert_resource(MyResource { ... })` at
plugin build time. Resources are singleton values accessible to any system
via `Res<T>` (read) or `ResMut<T>` (write). This maps directly to
hilavitkutin's `Read<T>` / `Write<T>` for singleton resources.

### Specs, Legion, hecs

Older Rust ECS implementations take varying approaches:

**Specs.** Explicit component registration: `World::register::<T>()` for each
component type. System scheduling via `DispatcherBuilder` with explicit
dependency declarations: `.with(SystemA, "a", &[])` and
`.with(SystemB, "b", &["a"])` (SystemB runs after SystemA). Manual, verbose,
but clear.

**Legion.** No explicit registration. Components are stored by `TypeId` on first
insertion. Systems are closures passed to `Schedule::add_system()`. Dependencies
are inferred from query parameters, similar to Bevy but with a simpler
scheduler.

**hecs.** Minimal. No scheduling at all. The developer iterates queries manually
in application code. No registration, no automatic parallelism, no dependency
analysis. Useful as a data structure, not as a scheduling framework.

**Common pattern across all four:** `TypeId`-based component identification with
runtime type registry. The scheduler (where it exists) infers dependencies from
typed parameters. This is the ECS consensus: registration is implicit (types
register themselves on use), scheduling is automatic (dependencies flow from
type access declarations).

### How Bevy handles ordering constraints

Bevy's system scheduling is DAG construction from component access declarations.
The same model as loimu's behavior scheduling and hilavitkutin's planned design.
The process:

1. Each system declares its component access via function parameters.
2. The scheduler builds a conflict graph: systems that access the same
   component with at least one writer are in conflict.
3. Explicit ordering constraints (`.before()`, `.after()`, system sets) add
   edges to the DAG.
4. The scheduler topologically sorts the DAG and groups non-conflicting
   systems into parallel batches.

This is structurally identical to what hilavitkutin needs. The difference is
granularity: Bevy operates at component-type granularity (all of `Position`),
while loimu operates at field granularity (`Position.x` vs `Position.y`).
hilavitkutin should accept `AccessSet` as input without mandating the
granularity level; consumers choose their own granularity.

## 3. Service Discovery and DI Patterns

Traditional dependency injection and service discovery patterns exist in Rust
but have been largely supplanted by the ECS model for the kinds of applications
polka-dots, saalis, and loimu represent. The survey is included for
completeness and to explain why ECS-style access typing is the recommended
approach.

### Service Locator

A global registry of services accessed by type:
`ServiceLocator::get::<HttpClient>()`. Common in game engines (Unity uses it
extensively) and web frameworks.

- **Pros:** Simple. Works from any call site. No wiring boilerplate.
- **Cons:** Hidden dependencies (callers depend on services not visible in
  their signature). Hard to test (must set up the global locator before
  calling any code that uses it). No compile-time safety (runtime panic if
  the service is not registered). Typically relies on `TypeId` / `Any`, which
  polka-dots bans.
- **Verdict:** Anti-pattern for the hilavitkutin use case. The ECS model
  solves the same problem with visible, type-checked dependencies.

### Dependency Injection (Rust crates)

**shaku.** Compile-time DI container. Interfaces and implementations are
registered via derive macros. The container resolves dependencies at
construction time. Supports scoped lifetimes and overrides for testing.
Moderately complex API. Low adoption in the Rust ecosystem.

**inject.** Runtime DI. Register factories, resolve by type. Simpler than
shaku but less type-safe. Even lower adoption.

Neither crate has significant community traction. The Rust ecosystem has not
converged on a DI framework the way Java has (Spring) or C# has (ASP.NET DI).
The reason is that Rust's type system and ownership model make the ECS/typed-
parameter pattern more natural than constructor injection.

### ECS as DI

The key insight: `Read<T>` / `Write<T>` in system parameters IS dependency
injection. The scheduler resolves dependencies automatically from type
declarations. There is no container, no factory, no locator. The type
signature of the work unit's execute method declares its dependencies, and the
scheduler provides them.

Consider saalis's need for host services. Under a traditional DI model, the
work unit would receive an `HttpClient` via constructor injection or a service
locator lookup. Under the ECS model, the work unit declares
`type Read = (HttpClient, DatabasePool)` and the scheduler provides read
access to both as resources. The access declaration is the dependency
declaration. The scheduler is the DI container.

This is why hilavitkutin's `type Read` / `type Write` model works as DI for
host services. No separate DI framework needed. No service locator. The
unified store (columns and resources under one Read/Write, per Q1 of the
topic) makes this natural: `Read<HttpClient>` (singleton resource) and
`Read<Entities>` (collection column) participate in the same dependency
graph.

## 4. polka-dots' Existing #[register]

polka-dots has a working registration system implemented via `polka-derive`.
Understanding what it does and how is essential for the hilavitkutin decision.

The `#[register]` attribute macro targets trait impl blocks:

```rust
#[register]
impl LanguagePlugin for PolkaLang {
    const NAME: &'static str = "polka";
    // ...
}
```

The macro inspects the trait name in the impl block (`LanguagePlugin`) and maps
it to the appropriate descriptor type via a fixed lookup table inside the proc
macro. For `LanguagePlugin`, it generates a `LanguagePluginDescriptor` -- a
`repr(C)` struct containing `extern "C"` function pointers that bridge to the
developer's trait impl methods.

The generated code includes:

1. A `repr(C)` descriptor struct instance with function pointers wrapping each
   trait method.
2. An `inventory::submit!` call that registers the descriptor.
3. The original trait impl block, unmodified.

Discovery at startup: `inventory::iter::<LanguagePluginDescriptor>()` yields
all registered language plugins. The host validates them (version compatibility,
no duplicate names) and stores them for later dispatch.

Key properties of polka's `#[register]`:

- **Named.** Every registration has a `NAME` const for human-readable
  identification. This comes from the `Named` trait in `polka-primitives`,
  not from `#[register]` itself.
- **polka-specific.** The macro targets trait impl blocks and generates
  `repr(C)` descriptors with `extern "C"` function pointers. This is
  designed for the cdylib plugin boundary. It is not a general-purpose
  registration macro.
- **Fixed trait lookup.** The proc macro has a hardcoded table mapping trait
  names to descriptor types. Adding a new registrable trait requires modifying
  the proc macro.
- **Layered.** Registration (`#[register]` + `inventory`) is separate from
  identity (`Named`) and from capability (`Registrable` marker trait).

## 5. saalis's Named + inventory Pattern

saalis uses the same foundational building blocks as polka-dots but with its
own conventions. The `Named` trait provides identity:

```rust
trait Named {
    const NAME: &'static str;
}
```

All registrable types implement `Named` for human-readable identification.
`inventory` handles discovery. The combination is: `Named` tells you WHAT
something is called; `inventory` tells you WHERE to find it.

saalis's registration needs differ from polka's:

- **No cdylib boundary.** saalis's subsystem operations are statically linked.
  No `repr(C)` descriptors, no `extern "C"` function pointers. The
  registration is for internal discovery within a single binary.
- **Named subsystem operations.** Each operation belongs to a subsystem
  (Curator, Doctor, Librarian) and has a name within that subsystem. The
  registry is hierarchical: subsystem -> operations.
- **Cadence metadata.** Operations carry scheduling metadata (minimum interval,
  priority) alongside their identity.

The open question for hilavitkutin: does hilavitkutin define its own `Named`
equivalent, or does it use saalis's? The answer from the paradigm topic is
clear: `Named` lives in `polka-primitives` as a cross-cutting base trait.
hilavitkutin should depend on the same `Named` trait, not define its own.
saalis should also use polka-primitives' `Named`. The trait is simple enough
that sharing it across projects is trivial, and having multiple `Named` traits
with identical semantics is pure waste.

## 6. loimu's Registration

loimu's registration is the most elaborate of the three projects. Every
`define_*` macro generates:

1. A `Registrable` impl returning a domain-specific descriptor type.
2. An `inventory::submit!` call with the descriptor.
3. Additional trait impls (`HasShape`, `HasName`, domain traits) that the
   registry system depends on.

The registry layer is per-domain:

- `SignalRegistry` discovers and validates all `SignalDescriptor` instances.
- `ResourceRegistry` discovers and validates all `ResourceDescriptor` instances.
- `BehaviorRegistry` discovers and validates all `BehaviorDescriptor` instances.
- `BindingRegistry` discovers and validates all `BindingDescriptor` instances.

Discovery happens via `Registry::discover()`, which internally calls
`inventory::iter::<XDescriptor>()`. After discovery, a validation pass checks:

- **No duplicate names.** Two resources with the same name are a hard error.
- **All referenced signals exist.** A behavior that observes `FocusRequest`
  fails validation if no `SignalDescriptor` for `FocusRequest` was registered.
- **Access sets are valid.** A behavior that reads `AudioSettings` fails
  validation if no `ResourceDescriptor` for `AudioSettings` was registered.

This startup validation is critical. It catches configuration errors (missing
plugins, misspelled signal names, unregistered resources) at startup rather
than at runtime. The cost is O(registrations^2) for duplicate detection and
O(registrations * references) for reference validation. For typical
registration counts (tens to hundreds), this is sub-millisecond.

The mandatory macro approach (no manual `impl Resource`, no manual `impl
Signal`) exists because each declaration produces 7-10 interconnected items.
A manually written `impl Resource` would miss `HasShape` (breaking columnar
projection), `Registrable` (breaking discovery), `PersistableResource`
(breaking persistence), and `inventory::submit!` (breaking registration).
The macro guarantees internal consistency.

## 7. Game Engine Registration Patterns Beyond ECS

### Unity

Unity uses C# reflection and attributes. `[System]` classes are discovered via
assembly scanning at startup. Components register themselves implicitly on first
use. Prefabs and ScriptableObjects provide declarative registration for data.
The pattern is reflection-heavy; not portable to Rust without `std::any`.

### Unreal Engine

Unreal's `UCLASS()`, `UPROPERTY()`, `UFUNCTION()` macros generate registration
metadata via Unreal Header Tool (UHT), a preprocessor that runs before
compilation. UHT generates `.generated.h` files containing registration code,
reflection metadata, and serialization boilerplate. Conceptually similar to
Rust proc macros but operating at the preprocessor level.

Unreal's subsystem registration uses `USubsystem` classes discovered via
`GetSubsystem<T>()`. Subsystems are created and destroyed automatically by the
engine based on their lifecycle scope (Engine, Editor, Game Instance, World,
Local Player). This is closer to saalis's subsystem model than to polka's
plugin model.

### Godot

Godot uses `ClassDB::register_class<T>()` for explicit registration. Each
class provides metadata (properties, methods, signals) via `_bind_methods()`.
The registration is mandatory and explicit. No implicit discovery.

### Common Thread

All major game engines converge on the same pattern: metadata-generating macros
or preprocessors that turn type declarations into registration code. The
metadata includes identity (name), capability (what the type can do), and
scheduling hints (when and how it should run). Manual registration exists as a
fallback but is discouraged because it invariably misses some piece of the
interconnected metadata.

## 8. Relevance to hilavitkutin

The core question from Q8: should hilavitkutin define `#[register]` for work
units, or expect consumers to handle registration?

### Shape of the problem

A hilavitkutin work unit needs, at minimum:

1. **Identity** -- a name for diagnostics and the scheduler's internal maps.
2. **Access declarations** -- `type Read` and `type Write` for DAG edge
   construction.
3. **Execute entry point** -- `fn execute()`.
4. **Discovery** -- the scheduler needs to find all work units at startup.

Optionally:

5. **Scheduling hints** -- morsel vs scalar dispatch, priority, cadence.
6. **Fault handling mode** -- retry policy, rollback strategy.
7. **Fusibility** -- whether the storage engine can merge this unit's
   iteration with others.

Items 1-3 are the `WorkUnit` trait. Item 4 is the registration question.
Items 5-7 are metadata that may or may not be part of the trait.

### Option A: hilavitkutin defines WorkUnit trait + #[register] macro + inventory-based discovery

hilavitkutin provides a proc macro crate (`hilavitkutin-macros` or
`hilavitkutin-derive`) with a `#[register]` attribute. Applying it to a
`WorkUnit` impl generates:

1. A `WorkUnitDescriptor` with access metadata and identity.
2. An `inventory::submit!(WorkUnitDescriptor { ... })`.
3. Validation that `Named` is implemented.

The scheduler discovers all work units via
`inventory::iter::<WorkUnitDescriptor>()` and builds the DAG from their
access declarations.

**Pros:**

- Zero-friction registration. Consumers write `#[register] impl WorkUnit for
  MyTransform { ... }` and the DAG discovers it automatically.
- Automatic DAG construction. The scheduler sees all work units and their
  access declarations without any manual wiring.
- Consistent with polka-dots' existing `#[register]` pattern.
- Consistent with loimu's `define_*` pattern (same inventory mechanism).

**Cons:**

- Adds a proc macro dependency to hilavitkutin. Proc macros are slow to
  compile and opaque to debug.
- May conflict with polka's `#[register]`. If both polka-derive and
  hilavitkutin-derive provide `#[register]`, the user must disambiguate.
  This is solvable (use different attribute names, or make polka-derive
  delegate to hilavitkutin-derive for WorkUnit registrations), but it is
  additional complexity.
- Prescribes `inventory` as the discovery mechanism. Consumers that want a
  different mechanism (e.g., explicit `scheduler.register(my_unit)` calls)
  must work against the framework.

### Option B: hilavitkutin defines only the WorkUnit trait, consumers handle registration

hilavitkutin provides the `WorkUnit` trait and nothing else. Consumers use
their own registration mechanism:

- polka uses `#[register]` from polka-derive (which already exists).
- saalis uses `Named` + `inventory` (which it already uses).
- loimu uses `define_behavior!` (which already generates `inventory::submit!`).

The scheduler provides an API for explicit registration:
`Scheduler::register::<T>()` or `Scheduler::register_descriptor(desc)`.
Consumers call this API at startup after their own discovery mechanism has
collected the work units.

**Pros:**

- No proc macro dependency in hilavitkutin. Simpler crate, faster compilation.
- No conflict with existing registration mechanisms. Each consumer keeps its
  own `#[register]` / `define_*` pattern.
- Maximum flexibility. Consumers can register work units from any source:
  inventory discovery, explicit lists, runtime plugin loading, configuration
  files.
- hilavitkutin stays minimal. It defines what a work unit IS (the trait) and
  how to schedule it (the DAG), not how to find it (discovery).

**Cons:**

- Manual boilerplate per work unit. Each consumer must write its own
  registration glue code. For polka-dots, this means adding WorkUnit
  registration to polka-derive's `#[register]` (a one-time cost). For
  saalis, this means adding a call to `scheduler.register()` per operation.
- DAG construction requires explicit API calls. The scheduler cannot build
  the DAG automatically from inventory discovery; the consumer must feed
  it the discovered units.
- Each consumer reinvents the "discover + validate + register" flow. Some
  duplication across polka, saalis, and loimu.

### Option C: hilavitkutin provides both. Trait is always available. Optional `hilavitkutin-macros` crate provides #[register]

The WorkUnit trait lives in hilavitkutin (core crate, no proc macro dep). An
optional `hilavitkutin-macros` crate provides `#[work_unit]` (not `#[register]`
to avoid name collision with polka-derive) that generates the descriptor and
inventory submission.

Consumers choose:

- Use `hilavitkutin-macros` for zero-friction registration.
- Use their own mechanism and call `Scheduler::register()` explicitly.
- Mix both (some work units auto-registered, some explicit).

**Pros:**

- Flexibility + convenience. Consumers pick the registration style that fits.
- No forced dependency on proc macros. The core hilavitkutin crate is lean.
- The optional macro crate can evolve independently of the core.

**Cons:**

- Two crates to maintain. The macro crate must stay in sync with the trait
  definition.
- Optional dependency complexity. Consumers must know that the macro crate
  exists and decide whether to use it.
- The macro crate needs its own attribute name to avoid conflicting with
  polka-derive's `#[register]`.

### Recommendation: Option B

The recommended approach is Option B: hilavitkutin defines only the WorkUnit
trait, and consumers handle registration using their own mechanisms.

The reasoning:

**Registration is consumer-specific.** polka registers plugin trait impls via
`#[register]` on impl blocks, generating `repr(C)` descriptors for the cdylib
boundary. saalis registers named subsystem operations with cadence metadata.
loimu registers behavior handlers with field-granular access analysis, seven
access modes, scheduling unit decomposition, and computed portals. These are
fundamentally different shapes. A single `#[register]` macro in hilavitkutin
would either be too generic (producing a descriptor that none of the consumers
actually want) or too specific (hard-coding one consumer's shape and forcing
others to adapt).

**The WorkUnit trait is the contract.** What hilavitkutin needs from consumers
is not "register yourself" but "tell me your access declarations and give me
an execute function." The `WorkUnit` trait with `type Read`, `type Write`, and
`fn execute()` is the complete contract. How consumers discover and collect
their work units before handing them to the scheduler is their concern.

**Each consumer already has registration.** polka-derive already has
`#[register]`. loimu already has `define_behavior!`. saalis already has its
Named + inventory pattern. Adding a hilavitkutin-specific registration
mechanism means each consumer must now support two registration systems (their
own for non-WorkUnit things, hilavitkutin's for WorkUnit things) or migrate
everything to hilavitkutin's system (which cannot accommodate their
domain-specific metadata).

**The scheduler accepts descriptors, not inventory iterators.** hilavitkutin's
scheduler should expose `fn register(&mut self, descriptor: WorkUnitDescriptor)`
or similar. The consumer collects work units however it wants (inventory, explicit
list, config file, runtime plugin load) and feeds them to the scheduler. The
scheduler does not call `inventory::iter` itself. This keeps hilavitkutin
decoupled from the discovery mechanism.

**Convergence can be extracted later.** If polka, saalis, and loimu all converge
on the same registration pattern (all use inventory, all generate the same
descriptor shape, all validate the same constraints), that pattern can be
extracted into hilavitkutin at that point. Extracting a proven pattern is
cheaper and safer than designing one speculatively.

### The key insight

Registration is not one problem. It is three:

1. **Declaration** -- the consumer defines a work unit (trait impl, macro
   invocation, whatever).
2. **Discovery** -- the consumer collects all declared work units (inventory,
   explicit list, plugin loader).
3. **Registration** -- the consumer hands the collected work units to the
   scheduler.

hilavitkutin should own step 3 (the scheduler API that accepts work units).
Steps 1 and 2 are consumer concerns. polka's step 1 is `#[register] impl
WorkUnit for ...`. loimu's step 1 is `define_behavior!(...)`. saalis's step 1
is a struct with `Named` + manual trait impl. All three can use inventory for
step 2, but the shape of what they submit to inventory differs. Step 3 is
the only part that must be uniform: the scheduler needs a consistent
`WorkUnitDescriptor` regardless of how it was discovered.

### Descriptor shape

hilavitkutin's `WorkUnitDescriptor` should contain:

- **Name** -- from `Named`, for diagnostics and dedup detection.
- **Read set** -- list of type identifiers that the work unit reads.
- **Write set** -- list of type identifiers that the work unit writes.
- **Execute function** -- function pointer or equivalent callable.
- **Scheduling hints** -- morsel/scalar dispatch, priority, fusibility.

The descriptor does NOT contain:

- **`repr(C)` function pointers** -- that is polka's cdylib concern.
- **Field-granular access** -- that is loimu's concern. hilavitkutin accepts
  resource-granularity access; loimu's layer refines it to field granularity
  as a compile pass.
- **Persistence config** -- that is loimu's resource concern.
- **Cadence metadata** -- that is saalis's concern, passed as a scheduling
  hint or handled in saalis's wrapper.
- **Signal metadata** -- that is loimu's concern.

The descriptor is the common denominator. Each consumer wraps it with their
domain-specific metadata and registration mechanism.

### Integration with polka-derive

For polka-dots specifically, the integration path is:

1. polka-derive's `#[register]` already generates descriptors and inventory
   submissions for plugin trait impls.
2. When a `#[register]` targets a `WorkUnit` impl block, polka-derive
   generates both a polka-specific descriptor (for the cdylib boundary) and
   a `WorkUnitDescriptor` (for hilavitkutin's scheduler).
3. At startup, polka iterates `inventory::iter::<WorkUnitDescriptor>()` and
   feeds each to `scheduler.register()`.
4. No changes to hilavitkutin. All integration logic lives in polka-derive
   and polka-pipeline.

This keeps hilavitkutin clean while giving polka zero-friction integration.
The same pattern works for saalis (feed descriptors from its own registry)
and loimu (extract descriptors from BehaviorDescriptor during
`BehaviorRegistry::discover()`).

## 9. Summary of Findings

| Pattern | Where used | Mechanism | Ordering | Type safety |
|---|---|---|---|---|
| `inventory` | polka, loimu | Linker sections | Undefined | Yes |
| `linkme` | -- | Linker sections (slice) | Undefined | Yes |
| `ctor` | -- | Raw constructors | Undefined | No |
| Bevy plugins | Bevy | Explicit `add_plugins()` | Explicit | Yes |
| Bevy systems | Bevy | `add_systems()` + param inference | Auto from types | Yes |
| Specs dispatch | Specs | `DispatcherBuilder` + manual deps | Explicit strings | Partial |
| Service locator | Unity, web | Global registry by type | None | Runtime only |
| shaku DI | Niche Rust | Compile-time container | Constructor | Yes |
| ECS as DI | Bevy, loimu, hilavitkutin | Typed params = deps | Auto from types | Yes |
| loimu `define_*` | loimu | Mandatory macros + inventory | Auto from access | Yes |
| polka `#[register]` | polka-dots | Proc macro + inventory | N/A (plugin) | Yes |

The ecosystem consensus: typed access declarations that double as dependency
declarations, with automatic scheduling from those declarations. The discovery
mechanism (inventory, explicit registration, reflection) is secondary. What
matters is that the scheduler has typed access information for each work unit.
hilavitkutin should own the scheduling contract (WorkUnit trait + scheduler
API) and leave discovery to consumers.
