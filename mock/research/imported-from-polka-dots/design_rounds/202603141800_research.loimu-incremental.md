## 1. Hierarchical DirtyTracker

The `loimu-dirty` crate provides a generic hierarchical bitset with configurable depth.
Each level halves (or otherwise reduces) the number of tracked elements:

```
level 0: per-element bits (finest)
level 1: groups of N elements
level 2: groups of N*N elements
...
level K: single root bit (coarsest)
```

Marking a leaf dirty sets the bit at level 0 and propagates upward through all
parent levels. Querying at a coarse level lets callers skip entire subtrees
without inspecting fine-grained bits. Clearing is O(levels), not O(elements):
reset the coarsest level first, then lazily clear finer levels on the next mark.

No heap allocation occurs after construction. Capacity is fixed at creation;
growing requires re-creating with a larger capacity. Thread-safe reads, exclusive
writes (this matches the scheduler DAG's guarantee that only one writer touches a
resource at a time).

### API sketch

```
DirtyTracker::new(capacity, levels)
DirtyTracker::mark(index)
DirtyTracker::is_dirty(index) -> bool
DirtyTracker::is_any_dirty() -> bool
DirtyTracker::dirty_ranges(level) -> impl Iterator<Range>
DirtyTracker::clear()
```

### Three instantiations

| Consumer | Leaf granularity | Use |
|----------|-----------------|-----|
| loimu-render / loimu-gpu | screen-space tile (4px base) | skip clean raster regions |
| loimu-tree | node ID | skip unchanged subtrees during layout |
| loimu-storage | slot index | track which slots need GPU upload |

The same data structure serves three domains. The renderer uses it for spatial
tile-based skip logic, the tree uses it for structural subtree skip, and storage
uses it to batch GPU uploads to only the slots that changed.

## 2. Three-level dirty tracking

loimu solves the "what changed" problem at three distinct levels, each feeding
into the next.

**Level 1 -- resource version stamps.** Every resource carries a hidden
`version: u64` field, incremented by the write accessor when mutation occurs.
This is not atomic; the DAG guarantees exclusive write access. The scheduler
captures versions before each behavior runs. After a behavior completes, it
compares the captured versions against current values. If a resource's version
did not change, all downstream behaviors that depend only on that resource can
skip.

**Level 2 -- graph early cutoff.** When a node in the DAG produces the same
output as last tick, all downstream nodes that depend solely on that output
skip execution. This extends the version stamp concept across the full graph.
The scheduler precomputes which behaviors to skip before running any of them,
which is possible because dependency declarations are static.

**Level 3 -- tile dirty regions.** Dirty tracking at the resource level tells us
which behaviors re-run. Tile-based dirty tracking tells us which screen regions
need re-rendering. The hierarchical tile structure maps resource changes through
node bounding boxes to screen-space tiles, so the render pass only touches pixels
that actually changed.

These three levels compose: version stamps feed early cutoff, early cutoff reduces
the set of behaviors that run, fewer behavior writes mean fewer tiles get dirtied,
and tile dirty regions limit rasterization to the changed area.

## 3. Version stamp mechanics

### The phantom-write problem

The naive approach increments the version on every write access, even if the
written value is identical to the previous value:

```
Tick N:
  BehaviorA writes ResourceX -> version 41 -> 42
  BehaviorB reads ResourceX, sees version 42, does work

Tick N+1 (naive):
  BehaviorA writes ResourceX with same value -> version 42 -> 43
  BehaviorB reads ResourceX, sees version 43, does work (wasteful)

Tick N+1 (with phantom-write detection):
  BehaviorA writes ResourceX, value unchanged -> version stays 42
  BehaviorB reads ResourceX, sees version 42 (same as last tick) -> skips
```

### Detection strategies

| Strategy | Cost | Quality |
|----------|------|---------|
| Always increment | Zero | No skipping |
| Bitwise comparison | `memcmp(old, new, size)` | Perfect for small resources |
| Hash comparison | `hash(new) != hash(old)` | Good for large resources |
| Developer hint | `.mark_unchanged()` | Zero cost, requires discipline |

loimu uses bitwise comparison for resources under 4KB, which covers more than
95% of framework resources. For larger resources, the developer calls
`.mark_changed()` explicitly; the default is "assume changed."

### Why version stamps instead of dirty flags

Dirty flags are boolean: clean or dirty. Version stamps are monotonic counters.
The advantage: you can tell how stale something is. If a downstream behavior
skipped N ticks because its inputs did not change, the version stamp difference
tells the frame generation system exactly how many frames were static. A resource
unchanged for 100 ticks is "cold" and its region can be deprioritized in the
frame generation ratio.

### Why per-resource, not per-field

Per-field version stamps would enable finer-grained skipping: "behavior X only
reads field A of resource Y; field A did not change, so skip X even though field
B changed." Cost: one u64 per field instead of one u64 per resource. For a
resource with 10 fields, that is 10x more version stamps to compare. The savings
from finer-grained skipping rarely justify the overhead. Design guideline: if
profiling shows a hot behavior frequently skipped due to phantom writes on a
large resource, split the resource into smaller resources with separate version
stamps. This is a manual design choice, not a framework feature.

### Why not hash-based change detection

Hashing (like Salsa uses internally) requires reading the entire resource value
to compute the hash. Version stamps require reading a single u64. For resources
that are written frequently but rarely change value (phantom writes), hashing is
correct but expensive. Version stamps with bitwise comparison are nearly as good
for small resources and cheaper for large ones.

## 4. Graph early cutoff

### The Salsa model

Salsa is Rust-Analyzer's incremental computation engine. It tracks dependencies
at function granularity through runtime discovery:

```
Input: source text
  -> parse(text) -> AST
    -> resolve(AST) -> resolved names
      -> type_check(resolved) -> diagnostics
```

When source text changes, Salsa marks `parse` as dirty, re-runs it, compares the
new AST to the old AST, and if unchanged (e.g., only a comment changed), performs
early cutoff -- skipping `resolve`, `type_check`, everything downstream. If
changed, it marks `resolve` dirty and repeats. Dependencies are runtime-discovered:
each query records which other queries it called. This is flexible but has overhead
(hash lookups for every dependency check).

### The Adapton model

Adapton adds named computation nodes. Instead of tracking dependencies by call
stack, each computation has an explicit name (ID). This enables structural editing
(inserting a list element recomputes only the new element and its neighbors),
cross-execution consistency (the same name always refers to the same computation),
and efficient re-use (named nodes are cached and reused by identity).

### How loimu differs

| Aspect | Salsa | Adapton | loimu |
|--------|-------|---------|-------|
| Dependencies | Runtime-discovered | Runtime-discovered | Static (declared) |
| Granularity | Per-function | Per-named-node | Per-behavior |
| Change detection | Value comparison | Memoization | Version stamps |
| Cost of tracking | Hash lookups | Allocation per node | Zero (compile-time) |

Static declarations mean the DAG is known before execution starts. The scheduler
can precompute which behaviors to skip before running any of them. Neither Salsa
nor Adapton can do this because they discover dependencies during execution.

The trade-off: static declarations cannot express data-dependent dependencies. If
BehaviorA conditionally reads ResourceX, the DAG must conservatively assume it
always reads ResourceX. Salsa would discover on each run whether the dependency
was actually used. In practice, this conservative approach wastes little work. GUI
behaviors have stable access patterns: a layout behavior always reads all its
inputs, a render behavior always reads all visible nodes.

### Early cutoff in the DAG

```
[InputBehavior] -> writes InputState (v41 -> v42)
     |
[LayoutBehavior] -> reads InputState (v42), writes LayoutResult (v15 -> v15*)
     |
[RenderBehavior] -> reads LayoutResult (v15, unchanged) -> SKIP

* LayoutBehavior ran but produced the same output
```

The scheduler captures the version of every resource before and after each
behavior runs. If all written resources have unchanged versions, all downstream
behaviors that read only those resources can skip.

This is O(behaviors x resources_per_behavior) version comparisons. For a typical
app with ~50 behaviors and ~3 resources each, that is ~150 u64 comparisons per
tick. Negligible.

### Comparison to other systems

**React fiber.** React's reconciliation compares virtual DOM trees, walking the
entire tree looking for changes: O(n) in tree size. React's approach is "render
everything, diff the output." loimu inverts this: "track what changed, render only
that." For a 10,000-node UI with 3 nodes changing, React walks 10,000 nodes;
loimu checks 3 version stamps and processes ~10 tiles.

**Druid/Xilem.** Druid uses explicit change tracking with dirty flags. Xilem uses
tree-diffing closer to React. loimu's version stamps are more granular than
Druid's boolean dirty flags: version stamps enable phantom-write detection, which
Druid cannot distinguish from a real write.

**Bevy ECS.** Bevy uses `Ticks`: a global tick counter compared against
per-component last-modified ticks. Conceptually identical to loimu's version
stamps. Key difference: Bevy checks at component granularity (one value per ECS
component); loimu checks at resource granularity (one value per resource, which
may contain multiple fields). Coarser but cheaper.

## 5. Tile-based dirty regions

### Hierarchical tile structure

Following the coarse-to-fine pattern observed across GPU rasterization (Giesen),
Intel occlusion culling, and mesh shaders:

```
Level 0: Screen (1 region)
  +-- Level 1: Macro tiles (64x64 pixels)
       +-- Level 2: Tiles (16x16 pixels)
            +-- Level 3: Pixels
```

At 4K resolution (3840x2160):

- Level 1: 60x34 = 2,040 macro tiles (256 bytes bitmask)
- Level 2: 240x135 = 32,400 tiles (4KB bitmask)

A clean macro tile means all 16 child tiles are clean. The entire 64x64 region is
skipped without checking individual tiles.

### Dirty propagation sequence

1. Behavior writes resource; version stamp increments
2. Resource has associated screen region(s) via node markers
3. Scene tree maps nodes to screen-space bounding boxes
4. Bounding boxes mark tiles as dirty in the bitmask
5. Render pass iterates only dirty tiles
6. After rendering: clear dirty bitmask

Steps 2-4 happen at behavior exit, deferred to pass boundary. The cost is
proportional to the number of nodes that changed position or content, not the
total number of nodes.

### Conservative vs optimistic dirtying

**Conservative (current design).** If a node's resource changed, mark all tiles
the node touches as dirty. This may over-dirty: a color change on a small element
marks its entire bounding box.

**Optimistic (future).** Compare old and new render output per-tile. If the pixels
are identical, clear the dirty bit. This catches phantom visual changes (e.g., a
behavior rewrites the same color value) but requires rendering to a scratch buffer
first.

Conservative dirtying is sufficient for now. The over-dirtying cost is small
because most changes are spatially localized.

### Graph data structure for dirty tracking

```rust
struct DirtyTracker {
    /// One bit per node in the DAG: is this node dirty?
    node_dirty: BitVec,
    /// Version stamps captured before execution
    versions_before: Vec<u64>,
    /// Nodes that were dirty last tick (for "was dirty" queries)
    prev_dirty: BitVec,
}
```

After the execution pass:

1. Compare `versions_before` with current versions
2. For each resource whose version changed, mark all observing nodes dirty
3. For each node marked dirty, propagate to downstream nodes
4. Swap `node_dirty` and `prev_dirty` for next tick

Propagation is a forward sweep over topological order. Each node visited at most
once. Cost: O(V + E) where V = nodes, E = edges in the DAG.

## 6. Resource aliasing

### Core idea

Resources that are never live at the same time can share the same memory. If
render pass A writes to a temporary texture, and render pass C (which runs later,
after A's texture is no longer needed) writes to a different temporary texture of
the same size, both textures can alias the same GPU memory allocation. This is
directly analogous to register allocation in compilers: variables with
non-overlapping lifetimes can share the same register.

### Frostbite FrameGraph compile phase

The Frostbite FrameGraph (O'Donnell, GDC 2017) introduced this pattern:

1. **Cull:** remove unreferenced passes (passes whose outputs are never read)
2. **Compute lifetimes:** for each transient resource, determine the first and
   last pass that accesses it
3. **Alias memory:** resources with non-overlapping lifetimes share backing
   memory via a free list

The key constraint: aliasing only applies to transient resources (created and
destroyed within a single frame). Persistent resources (render targets that
survive across frames) cannot be aliased because their lifetimes span the
entire frame.

### loimu-graph implementation

```rust
pub struct AliasingAnalysis<N: Id> {
    step_of: IdMap<N, usize>,
    lifetimes: DenseColumn<(usize, usize)>,
}
```

For each resource mentioned in the graph's access declarations:

1. Find the earliest step that accesses the resource -> `first_step`
2. Find the latest step that accesses the resource -> `last_step`
3. The lifetime interval is `[first_step, last_step]`

Two resources can alias if their intervals do not overlap:
`a.last_step < b.first_step` or `b.last_step < a.first_step`.

### Greedy interval coloring algorithm

`alias_groups()` returns sets of resources that can share memory:

1. Sort resources by `first_step`
2. For each resource, try to assign it to an existing group where no conflict
   exists (no overlapping lifetime)
3. If no group fits, create a new group

This is O(n * k) where n = number of resources and k = number of groups. For
typical frame graphs (dozens to low hundreds of resources), this is instant.

The aliasing problem is equivalent to graph coloring on an interval graph (where
vertices are resources and edges connect resources with overlapping lifetimes).
Interval graphs are perfect graphs, so the chromatic number equals the maximum
clique size (the maximum number of simultaneously live resources). Optimal interval
graph coloring runs in O(n log n): sort by start time, greedily assign colors.
This is essentially what the linear scan does. The linear scan register allocation
algorithm (Poletto & Sarkar, 1999) is directly applicable.

### Consumers

- **Scheduler:** behavior scratch buffers. Two behaviors that never run in the
  same step can alias their scratch buffers, reducing peak memory usage.
- **Animation:** intermediate blend values. In a sequential chain A -> B -> C,
  A's intermediate is consumed by B and then dead. A's and C's intermediates
  can alias.
- **Render graph (future):** the primary motivation. Alias render targets, depth
  buffers, and temporary compute buffers.

### Dirty tracking constraint

Two resources that alias cannot both be dirty simultaneously because their
version stamps would conflict. The plan computation phase checks for this and
falls back to separate allocations when needed.

### Invariants

1. Aliased resources never overlap in time. No step accesses both.
2. Aliasing is conservative. May miss opportunities, never produces incorrect
   aliasing.
3. Persistent resources are never aliased. Only resources created and destroyed
   within a single plan execution are candidates.
4. Aliasing groups are disjoint. Each resource belongs to exactly one group
   (or no group if it cannot be aliased).

### Bandwidth and cache impact

From Drepper ("What Every Programmer Should Know About Memory"): aliasing has a
positive cache effect when the lifetime gap is short. The second resource's access
finds cache lines warm from the first resource's access (assuming lines have not
been evicted). When the gap is long, cache lines are cold regardless; aliasing
saves heap memory but not bandwidth.

From Forrest Smith's bandwidth analysis: at 30 GB/s CPU bandwidth, a 4K
framebuffer read+write costs ~2.2ms. Aliasing two framebuffers saves 33 MB of
allocation but the bandwidth cost is the same. For GPU-side aliasing (400 GB/s),
memory capacity (not bandwidth) is the bottleneck. This reinforces the Frostbite
FrameGraph's motivation: aliasing was driven by Xbox One's 32MB ESRAM limit, not
by bandwidth constraints.

## 7. Subgraph composition

### The problem

A single flat graph does not scale when systems compose. The scheduler graph
contains behavior nodes; one of those behaviors (e.g. AnimationBehavior) internally
evaluates an animation graph. A naive approach evaluates the animation graph inside
a single behavior `run()` call: sequential, unparallelized.

With subgraphs, the animation graph becomes a child of the scheduler node. The
executor sees it, expands it, and parallelizes its internal steps using the same
infrastructure as the outer graph.

### Node model

```rust
pub enum NodeKind<N: Id> {
    Leaf,
    Subgraph(ExecutionPlan<N>),
}
```

A `Subgraph` node contains a pre-computed `ExecutionPlan`. When the executor
reaches a subgraph node, it executes the inner plan's steps (potentially in
parallel) before proceeding.

### Properties (following Taskflow's `composed_of`)

1. The subgraph is a single node in the outer graph for ordering purposes. All
   predecessors complete before any inner task starts; all inner tasks complete
   before any successor starts.
2. Internal parallelism is preserved. Independent inner tasks run in parallel.
3. The subgraph is a reference, not a copy. Multiple outer graphs can compose
   the same subgraph.
4. No special executor support needed. The executor treats the module node like
   any other task; when it runs, it recursively enters the subgraph's execution.

### Plan caching

The inner plan is computed once and cached. It only recomputes when the inner
graph's topology changes (nodes or edges added/removed). The `DirtyTracker` can
mark the subgraph node itself as dirty when its inner graph changes, triggering
replanning of the subgraph. For animation: the blend tree topology changes rarely
(when switching animation states). Static subgraphs with replanning on change
are sufficient.

### Nesting

Subgraphs can nest: a subgraph node can itself contain subgraph nodes. Since the
graph is acyclic, nesting terminates. In practice two levels suffice (scheduler ->
animation, scheduler -> module -> module's behaviors). Deep nesting indicates a
design problem.

Inner plan node IDs live in the same ID space as the outer graph. Typed IDs from
`define_id!` ensure no collisions.

### Computed portals (boundary analysis)

The parent scheduler inspects a subgraph's internal DAG to compute claim points
and release points for each resource the subgraph accesses. This enables
inter-subgraph parallelism without breaking encapsulation.

- **Claim point (head-side):** the first internal node that accesses a given
  resource. External nodes can freely use that resource until the subgraph
  reaches its claim point.
- **Release point (tail-side):** the last internal node that accesses a given
  resource. External nodes waiting on that resource can start after the release
  point, not after the entire subgraph finishes.

```
Subgraph S: [A, B, C, D, E] in topological order

  A: reads R       <- claim point for R (first access)
  B: writes R
  C: reads R       <- release point for R (last access)
  D: writes Q      <- claim AND release point for Q (only access)
  E: unrelated work

Parent scheduler computes:
  - External writers of R can proceed until S.A starts
  - External readers of R can start after S.C completes (not after S.E)
  - External nodes using Q: can write Q until S.D starts, read Q after S.D
```

Both early release and delayed claiming fall out from static access analysis. The
subgraph author writes normal behaviors; the framework computes boundaries
automatically.

### Opaque subgraphs

A subgraph can be explicitly marked opaque, disabling boundary analysis. The
parent treats it as a single atomic node: full stall on all declared resources, no
internal inspection. Use cases: GPU-driven pipelines where internal synchronization
is managed by GPU hardware (atomics, work graphs), and externally synchronized
subgraphs where correctness depends on running as a monolith.

Opaque marking is opt-in. Non-opaque subgraphs get boundary analysis for free.
Monolithic subgraphs that are not marked opaque are a code smell, not an
optimization opportunity.

### Execution strategies

- **Sequential expansion (current):** executor runs inner plan step by step with
  parallel dispatch within each step. Outer step boundary waits for entire
  subgraph. Correct and simple.
- **Parallel expansion (future):** if the subgraph's first step has no external
  dependencies beyond its predecessors (already complete), the subgraph could
  begin while other outer-step nodes are still running. Higher utilization,
  more complex.
- **GPU offload (future):** compute-heavy subgraphs lowered to GPU compute
  dispatches. CPU submits, fence signals completion. Overlaps GPU compute with
  CPU work on other branches.

### Use cases

Scheduler -> Animation:
```
Scheduler graph:
  InputBehavior -> AnimationBehavior -> RenderBehavior
                       |
              Animation subgraph:
                Driver1, Driver2 -> Blend -> Constraint -> Output
```

Scheduler -> Module:
```
Scheduler graph:
  CoreBehaviors -> ModuleA -> ModuleB -> RenderBehaviors
                    |           |
            ModuleA subgraph  ModuleB subgraph
```

## 8. Frame generation

### The opportunity

GUI applications are mostly static. A typical desktop app redraws the entire
window 60-120 times per second, but between user interactions, less than 5% of
the pixel area actually changes. Frame generation exploits this: render "real"
frames at a reduced rate and synthesize intermediate frames from known scene state.

The key advantage over game frame generation (FSR 3, DLSS 4): loimu knows exactly
what moved. Games estimate motion from pixels (optical flow, motion vectors from
rasterization). loimu has the scene tree, transform stack, animation curves, and
dirty regions. Perfect information.

### Extrapolation (preferred for input response)

At display time, take the last rendered frame and warp dirty regions forward
using known motion:

```
Last rendered state (time T):
  Node A at position (100, 200), velocity (0, -300px/s)  [scrolling up]
  Node B at position (50, 50), static
  Node C at opacity 0.8, fading to 1.0 over 200ms

Display time (T + dt):
  Node A: position (100, 200 - 300*dt)  [exact: scroll is linear]
  Node B: no change, reuse cached
  Node C: opacity 0.8 + (0.2 * easing(dt/0.2))  [exact: easing curve known]
```

For scrolling: trivially correct (linear motion). For animations with easing
curves: follows the curve exactly. For physics simulations: uses known velocity
and acceleration.

### Interpolation (optional, visual polish)

When the scheduler knows both start and end state of an animation (e.g. a 200ms
fade), it can render the start and end frames and interpolate between them.
Smoother results than extrapolation for complex transitions. Adds one frame of
latency (must wait for the "future" frame). At 120Hz that is 8.3ms: acceptable
for non-interactive animations, not for input-driven motion.

### Text bypass

Text is always rendered at native resolution. Frame generation never warps,
interpolates, or applies temporal techniques to text content. The dirty region
system identifies text regions and excludes them. This avoids the conflict between
temporal techniques (subpixel jitter, TAA accumulation) and crisp text rendering.

### Dynamic multiplier

Following DLSS 4.5's dynamic multi-frame generation concept, loimu adjusts the
frame generation ratio based on scene activity:

- Idle (no animations): render at 5-10fps, display cached frames
- Scrolling: render at half display rate, extrapolate intermediates
- Fine input (typing): render at full rate, no generation needed

The scheduler knows which nodes are animating and how fast. It adjusts the ratio
automatically.

### Bandwidth feasibility

From Forrest Smith's bandwidth analysis:

| Resolution | Frame size | CPU read+write | GPU read+write |
|------------|-----------|----------------|----------------|
| 1080p | 8 MB | 0.5ms @ 30 GB/s | 0.04ms @ 400 GB/s |
| 4K | 33 MB | 2.2ms @ 30 GB/s | 0.17ms @ 400 GB/s |
| 4K, 10% dirty | 3.3 MB | 0.2ms @ 30 GB/s | 0.02ms @ 400 GB/s |

Even at 4K with CPU-only processing, frame generation fits within a 120Hz budget
(8.3ms). With dirty region optimization (the typical case for GUIs), it is
essentially free.

### Why GUI frame generation is fundamentally easier

- **Perfect motion information.** Scene tree, transform stack, animation curves,
  dirty regions. No estimation needed.
- **Mostly static content.** A typical GUI frame has >90% static content. Frame
  generation for static regions is zero-cost: reuse the cached tile.
- **No vestibular conflict.** Small interpolation errors are invisible on a
  monitor. Quality thresholds can be relaxed compared to VR.
- **2D transforms only.** GUI transforms are affine (translate, scale, rotate) or
  at most perspective (3D card flip effects). Geometric warping is exact for
  affine transforms. No parallax, lighting changes, or volumetric effects.

### Architecture position

Frame generation is NOT a graph feature. It is a render-side optimization that
consumes graph outputs:

```
Scheduler (loimu-graph consumer) -> behaviors update resources
   | version stamps, dirty regions
Render behavior -> render dirty regions to tiles
   | tile cache + motion state
Frame generation -> warp/extrapolate dirty tiles for display
   |
Display
```

loimu-graph provides: dirty tracking (DirtyTracker), version stamps (via resource
access), and DAG ordering that ensures consistency. The frame generation system
reads this information but does not modify the graph.

## 9. Incremental re-execution flow

The full incremental re-execution cycle across a single tick:

**Step 1: capture versions.** Before execution begins, the scheduler snapshots
`versions_before` for every resource in the DAG. Each resource's `version: u64`
is read into the `DirtyTracker`'s `versions_before` vector.

**Step 2: execute behaviors.** Behaviors run in topological order (with parallel
dispatch within each step). Each behavior reads its declared resources and writes
its declared outputs. Write accessors perform phantom-write detection: for
resources under 4KB, bitwise comparison against the previous value; if unchanged,
the version stamp is not incremented.

**Step 3: compare versions.** After the execution pass completes, the scheduler
compares `versions_before` with current versions for every resource. For each
resource whose version changed, all nodes that observe (read) that resource are
marked dirty in `node_dirty`.

**Step 4: propagate.** For each node marked dirty, propagate to all downstream
nodes via a forward sweep over topological order. Each node is visited at most
once. Cost: O(V + E).

**Step 5: skip.** On the next tick, the scheduler consults `node_dirty` before
running each behavior. If a node is not dirty (none of its input resources changed
last tick), the behavior skips execution entirely. The skip check is a single
bit-test per behavior.

After propagation, `node_dirty` and `prev_dirty` are swapped. The `prev_dirty`
bitset serves "was dirty last tick" queries, which the frame generation system
uses to determine staleness.

The tile dirty region layer runs in parallel: resource changes are mapped through
node bounding boxes to screen-space tiles, and the tile bitmask is updated. The
render pass then iterates only dirty tiles. After rendering, the tile bitmask
clears (O(levels) via the hierarchical DirtyTracker).

## 10. Relevance to hilavitkutin

The hilavitkutin design needs incremental computation for two primary consumers:
polka's incremental rebuild pipeline and saalis's re-query system. loimu's dirty
tracking, aliasing, and subgraph composition offer concrete reference patterns.

### Dirty tracking for polka incremental rebuild

polka's pipeline is a DAG of build stages. When a `.polka` source file changes,
only the stages downstream of that file need to re-execute. loimu's three-level
model maps directly:

- **Version stamps** on polka-data resources (settings, catalogue entries, resolved
  paths). When `polka-data` writes a resolved value and the value is identical to
  what was already stored, the version stamp stays. Downstream pipeline stages
  that depend on that value skip.
- **Graph early cutoff** in the pipeline DAG. If the `parse` stage produces the
  same AST (version unchanged), the `expand` and `resolve` stages skip. This is
  the Salsa pattern but with loimu's static dependency declarations instead of
  runtime discovery. polka's pipeline stages have stable access patterns (each
  stage always reads the same set of inputs), so the static-declaration trade-off
  is favorable.
- **No tile layer needed.** polka does not render to screen-space tiles. The tile
  dirty region concept does not apply. The two levels (version stamps + graph
  early cutoff) are sufficient.

### saalis re-query

saalis is a query system. When underlying data changes, cached query results need
invalidation. loimu's version stamp model applies directly: each data source
carries a version stamp, and queries record which versions they observed. On
re-query, if all observed versions are unchanged, the cached result is valid.

The comparison to Salsa is relevant here. Salsa's runtime-discovered dependencies
are natural for a query system where the set of accessed data varies per query.
loimu chose static declarations because GUI behaviors have stable access patterns.
saalis queries may have variable access patterns (a query result depends on which
records match a predicate, and that set changes). This suggests saalis should lean
toward runtime dependency discovery (Salsa-style) rather than static declaration
(loimu-style), using version stamps for the change detection mechanism itself.

### What to adopt from loimu

- **Version stamps as the change detection primitive.** The `version: u64` per
  resource is cheap, composable, and enables phantom-write detection. Adopt this
  directly.
- **Phantom-write detection with bitwise comparison for small resources.** The
  4KB threshold and memcmp strategy are well-motivated. Adopt this directly.
- **Graph early cutoff pattern.** The scheduler's "capture before, compare after,
  propagate forward" pattern is clean and generic. Adopt the pattern for polka's
  pipeline DAG.
- **DirtyTracker as a reusable primitive.** The hierarchical bitset with
  configurable depth is useful beyond rendering. It can serve as the skip-map
  for build stages, catalogue entries, or data store slots.
- **Subgraph composition for modular pipelines.** polka's pipeline has natural
  subgraph boundaries: each language plugin's processing is a subgraph of the
  main pipeline. Computed portals enable the pipeline to overlap language plugin
  processing with other stages.
- **Aliasing for pipeline scratch data.** Pipeline stages that do not overlap in
  execution can share scratch allocations. The greedy interval coloring algorithm
  is directly applicable.

### What to simplify

- **No tile dirty regions.** polka and saalis do not render to screen. The tile
  layer is rendering-specific.
- **No frame generation.** polka builds files, not frames. The dynamic multiplier
  and extrapolation concepts do not apply.
- **Static dependencies may not suffice for saalis.** Query systems have
  data-dependent access patterns. Plan for runtime dependency tracking on the
  saalis side, even if polka's pipeline uses static declarations.
- **No GPU offload.** polka's pipeline is CPU-only. The GPU compute and work graph
  extensions are irrelevant.
- **Simpler subgraph model.** polka does not need opaque subgraphs or GPU-driven
  planning. The basic `NodeKind::Subgraph(ExecutionPlan)` with plan caching and
  sequential expansion is sufficient.
