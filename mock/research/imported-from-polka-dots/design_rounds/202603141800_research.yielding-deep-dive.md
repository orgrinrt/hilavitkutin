## 1. The State Machine Problem

When a work unit yields, it saves its state and resumes later. The work
unit IS a state machine. The `execute()` method is called repeatedly until
the unit returns `Progress::Complete`. Between calls, the unit holds all
intermediate data in its own struct fields. This section examines what
that looks like in practice, where it breaks down, and what Rust's type
system buys and costs.

### 1.1 Simple case: HTTP connector

An HTTP connector fetches a single resource from an external API. The
states are linear and the intermediate data is small.

```rust
enum ConnectorState {
    Idle,
    RequestSent { future_handle: FutureHandle },
    ResponseReceived { body: Vec<u8> },
    Processing { parsed: ApiResponse },
}

struct HttpConnector {
    url: Url,
    state: ConnectorState,
    wake_slot: Option<WakeSlot>,
}

impl WorkUnit for HttpConnector {
    type Read = (Read<HttpClient>,);
    type Write = (Write<EntityColumn>,);

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        match &mut self.state {
            ConnectorState::Idle => {
                let client = ctx.read::<HttpClient>();
                let handle = client.start_request(&self.url);
                let slot = ctx.wake_slot();
                handle.on_complete(slot.clone());
                self.wake_slot = Some(slot);
                self.state = ConnectorState::RequestSent { future_handle: handle };
                Progress::Yield(YieldReason::Pending(self.wake_slot.unwrap()))
            }
            ConnectorState::RequestSent { future_handle } => {
                match future_handle.try_recv() {
                    Some(body) => {
                        self.state = ConnectorState::ResponseReceived { body };
                        // immediately continue processing, no yield needed
                        self.execute(ctx)
                    }
                    None => {
                        // spurious wake, still waiting
                        Progress::Yield(YieldReason::Pending(self.wake_slot.unwrap()))
                    }
                }
            }
            ConnectorState::ResponseReceived { body } => {
                let parsed = parse_api_response(body);
                self.state = ConnectorState::Processing { parsed };
                self.execute(ctx)
            }
            ConnectorState::Processing { parsed } => {
                let entities = ctx.write::<EntityColumn>();
                for entity in &parsed.entities {
                    entities.push(entity.into());
                }
                self.state = ConnectorState::Idle;
                Progress::Complete
            }
        }
    }
}
```

This is manageable. Four states, each with distinct intermediate data.
The enum variants are small (a URL, a body buffer, a parsed response).
The state transitions are linear. Exhaustive matching ensures every state
is handled. Rust gives us compile-time verification that no state is
forgotten.

### 1.2 Complex case: multi-phase pipeline operator

A hash join operator has two phases: build and probe. Each phase processes
many morsels and may yield between morsels. The state is large and varied.

```rust
enum HashJoinState {
    BuildingHashTable {
        partial_table: HashMap<JoinKey, Vec<Row>>,
        rows_processed: usize,
        total_rows: usize,
    },
    Probing {
        hash_table: HashMap<JoinKey, Vec<Row>>,
        probe_position: usize,
        output_buffer: Vec<JoinedRow>,
    },
    Emitting {
        results: Vec<JoinedRow>,
        emit_position: usize,
    },
}
```

The `BuildingHashTable` variant holds a growing hash table. The `Probing`
variant holds the completed hash table (potentially gigabytes for a large
join) plus a probe cursor. The `Emitting` variant holds the joined results.
The enum's size is the size of the largest variant. If the hash table is
10 MB, the enum is at least 10 MB regardless of which variant is active.

This is the first real cost of explicit state machines: variant bloat.
Every state variant's data occupies the same memory footprint as the
largest variant. A state machine with one small variant (an index) and
one large variant (a hash table) wastes memory when in the small state.

Mitigation: box the large variants.

```rust
enum HashJoinState {
    BuildingHashTable(Box<BuildState>),
    Probing(Box<ProbeState>),
    Emitting(Box<EmitState>),
}
```

Each variant is now pointer-sized. The large data lives on the heap. The
enum itself is small. The cost is one heap allocation per state transition
(when moving from Build to Probe, allocate a new ProbeState and drop the
BuildState). For a hash join that transitions twice across millions of
morsels, the allocation cost is negligible.

### 1.3 How Rust's type system helps

**Exhaustive matching.** Every `match` on the state enum must handle every
variant. If a new state is added (say, `Spilling` for out-of-memory
handling), the compiler flags every match that does not handle it. This is
static verification that the state machine is complete.

**Move semantics for intermediate data.** When transitioning from Build to
Probe, the hash table moves from `BuildState` to `ProbeState`. There is
no copy, no reference counting, no possibility of the build state and
probe state simultaneously holding the same hash table. Ownership transfer
is the state transition.

```rust
// Transition from build to probe
let build = match std::mem::replace(&mut self.state, HashJoinState::Idle) {
    HashJoinState::BuildingHashTable(build) => build,
    _ => unreachable!(),
};
let probe = ProbeState {
    hash_table: build.partial_table,  // moved, not copied
    probe_position: 0,
    output_buffer: Vec::new(),
};
self.state = HashJoinState::Probing(Box::new(probe));
```

After this transition, `build` is consumed. The hash table cannot be
accessed through the build state. The type system enforces the invariant
that only one phase owns the hash table at a time.

**No aliasing bugs.** In a language with shared mutable state (C++, Java),
a state machine that passes a pointer to intermediate data into a callback
can create aliasing: the callback holds a reference, the state transitions,
and the reference is dangling. Rust's borrow checker prevents this. The
intermediate data cannot be borrowed across a state transition because the
`&mut self` on `execute()` invalidates all borrows into `self`.

### 1.4 How Rust's type system hurts

**Cannot yield from a nested function call.** Consider a work unit that
calls a helper function which calls another helper function, and the
innermost function discovers it needs to wait for I/O. In async Rust,
this is trivial: the innermost function is async, it `.await`s, and the
compiler generates a state machine that suspends at the await point,
preserving the entire call stack.

In the explicit model, the innermost function cannot yield. It returns a
value to the middle function, which returns a value to execute(), which
returns `Progress::Yield`. The entire computation up to the yield point
must be refactored to avoid deep call stacks. Either the work unit
inlines all logic into execute() (flat, hard to read), or it manually
saves the call-stack equivalent as fields in its struct (tedious, error-
prone).

This is the central ergonomic cost. A three-level nested computation
becomes three enum variants with saved state at each level. Async/await
generates this automatically. Our explicit model requires the programmer
to write it.

```rust
// What the programmer wants to write:
fn execute(&mut self, ctx: ...) -> Progress {
    let a = compute_phase_1(&ctx);           // 100ms
    let b = fetch_from_api(a).await;         // 200ms wait
    let c = compute_phase_2(b, &ctx);        // 50ms
    let d = fetch_enrichment(c).await;       // 100ms wait
    write_results(d, &ctx);                  // 10ms
    Progress::Complete
}

// What the programmer must write:
fn execute(&mut self, ctx: ...) -> Progress {
    match &mut self.state {
        State::Phase1 => {
            let a = compute_phase_1(&ctx);
            self.state = State::WaitingApi { a };
            start_api_call(a, ctx.wake_slot());
            Progress::Yield(YieldReason::Pending(slot))
        }
        State::WaitingApi { a } => {
            let b = self.api_result.take().unwrap();
            let c = compute_phase_2(b, &ctx);
            self.state = State::WaitingEnrichment { c };
            start_enrichment(c, ctx.wake_slot());
            Progress::Yield(YieldReason::Pending(slot))
        }
        State::WaitingEnrichment { c } => {
            let d = self.enrichment_result.take().unwrap();
            write_results(d, &ctx);
            Progress::Complete
        }
    }
}
```

The async version is 5 lines. The explicit version is 20 lines with three
enum variants, two Option fields for intermediate results, and manual
state tracking. For simple work units this is acceptable. For complex
multi-phase operators with conditional yields, it becomes a maintenance
burden.

### 1.5 Is the ergonomic cost acceptable?

Yes, for three reasons.

First, the common case does not yield. polka-dots' pipeline stages process
morsels and return Complete. The state machine problem only applies to
I/O-bound work units (saalis connectors, loimu GPU fence waits). These
are the minority of work units in any deployment.

Second, I/O-bound work units tend to have simple state machines. An HTTP
connector has 3-4 states. A database query has 2-3. These are manageable
without macro assistance. Complex multi-phase operators (hash join, sort-
merge) typically do not yield for I/O; they yield for budget exhaustion,
and budget yields happen at natural morsel boundaries where the state is
already small (a cursor position, a partial accumulator).

Third, the explicit model is debuggable. The state is a plain enum. You
can print it, serialize it, inspect it in a debugger. Async state machines
are compiler-generated anonymous types that cannot be named, printed, or
inspected. When a work unit misbehaves, the explicit state machine is
visible; the async state machine is opaque.

The trade-off is clear: worse ergonomics for rare I/O cases, better
debuggability and portability for all cases. For a no_std-compatible
library that must work on embedded targets (loimu) and in CPU-bound
pipelines (polka-dots), this is the correct trade.

### 1.6 Comparison: Rust async/await generates the state machine automatically

The async/await desugaring is instructive. Given:

```rust
async fn work() {
    let a = phase_1().await;
    let b = phase_2(a).await;
    phase_3(b).await;
}
```

The compiler generates roughly:

```rust
enum WorkStateMachine {
    Phase1 { future: Phase1Future },
    Phase2 { a: A, future: Phase2Future },
    Phase3 { future: Phase3Future },
    Complete,
}
```

This is exactly the state machine we write manually. The compiler handles
variant sizing, field placement, and state transitions. The programmer
writes sequential code with `.await` and gets the state machine for free.

The question is whether hilavitkutin should use async internally. The
answer is no: async requires a runtime (Tokio, smol, embassy), introduces
`Pin<&mut Self>` complexity, breaks no_std without alloc, and forces every
work unit to be an async fn even when it never yields. The explicit model
avoids all of these costs. Work units that never yield pay nothing. Work
units that yield pay the manual state machine cost, which scales with the
number of yield points (typically 1-3), not the complexity of the compute
between yield points.

## 2. Yield Granularity

How often should a work unit yield? The answer depends on what the work
unit is doing, and getting it wrong has measurable performance consequences
in both directions. Yielding too often wastes scheduler cycles. Yielding
too rarely starves other work and harms responsiveness.

### 2.1 Never (morsel processing)

The hot path. `execute()` processes one morsel (2048-10000 rows), returns
Complete. No yield overhead. No state machine. The work unit is
essentially a function call.

```rust
fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
    let input = ctx.read::<InputColumn>();
    let output = ctx.write::<OutputColumn>();
    for row in input.morsel() {
        output.push(transform(row));
    }
    Progress::Complete
}
```

This is the 99% case for polka-dots. Every N-to-IR, IR-to-IR, and IR-to-M
stage processes a morsel and returns Complete. The morsel size is chosen so
that a single morsel fits in L2 cache and processes in 100-500 us. No
yield is needed because the operation is fast and bounded.

Overhead of never yielding: zero. The scheduler dispatches the work unit,
it runs, it returns Complete, the scheduler dispatches the next unit. The
Progress enum is a single discriminant check on the return value. No state
restoration, no re-queuing, no cache pollution.

### 2.2 Per-batch (HTTP connector doing 50 API calls)

Yield after each batch of 5 calls. Resume 10 times total. Yield cost is
negligible relative to HTTP latency (each batch takes 250-2500 ms of
wall-clock time; the yield/resume overhead is microseconds).

```rust
fn execute(&mut self, ctx: ...) -> Progress {
    match &self.state {
        State::FetchBatch { batch_idx, results } => {
            let start = batch_idx * BATCH_SIZE;
            let end = (start + BATCH_SIZE).min(self.total);
            for i in start..end {
                let response = self.pending_responses[i].take();
                results.push(parse(response));
            }
            if end >= self.total {
                write_all_results(results, &ctx);
                Progress::Complete
            } else {
                self.state.batch_idx += 1;
                start_next_batch(&ctx, end);
                Progress::Yield(YieldReason::Pending(slot))
            }
        }
    }
}
```

Each yield allows the scheduler to run other ready work units while the
next HTTP batch is in flight. The yield cost (state save, re-queue,
context switch) is dominated by the HTTP latency. Even if yielding takes
10 us and resuming takes 10 us, that is 0.004% of a 500 ms HTTP wait.

### 2.3 Per-item (processing a stream of events)

Yield after each event. High yield frequency. Is the Progress enum
overhead acceptable at this granularity?

The overhead of a yield/resume cycle:

1. Return `Progress::Yield` from execute(): one enum construction,
   negligible.
2. Scheduler dequeues the work unit from the running set: one pointer
   write to the work unit's status field. ~1 ns.
3. Scheduler enqueues to the pending/ready set: one atomic push to a
   lock-free queue. ~20-50 ns.
4. Scheduler scans wake flags (for Pending yields) or immediately
   re-queues (for BudgetExhausted): one atomic load per pending unit.
   ~1 ns per unit, O(N) for N pending units.
5. Worker picks up the re-queued unit: one atomic pop from the work
   queue. ~20-50 ns.
6. Worker calls execute() again: one indirect call. ~5 ns.
7. Work unit restores state: one match on the state enum, one branch.
   ~1-3 ns.

Total overhead per yield/resume: roughly 50-110 ns, call it 100 ns.

If events arrive at 10,000 per second (100 us between events), the yield
overhead is 100 ns / 100 us = 0.1%. Negligible.

If events arrive at 1,000,000 per second (1 us between events), the yield
overhead is 100 ns / 1 us = 10%. Significant. At this rate, yielding per
item is too expensive.

The threshold is roughly: if per-item processing time is under 1 us,
batch multiple items per execute() call. If per-item processing time is
over 10 us, per-item yielding is fine. Between 1-10 us, measure.

For hilavitkutin's workloads, per-item yielding is acceptable. saalis
events are HTTP responses (milliseconds apart). loimu events are signal
emissions (microseconds apart, but processed in batches per DAG pass).
polka-dots does not have streaming events. No consumer operates at the
sub-microsecond granularity where yield overhead matters.

### 2.4 Budget-based yielding

The work unit checks a time budget and yields when exhausted. This is the
general-purpose mechanism for CPU-bound work that cannot predict how long
it will take.

Questions:
- Who provides the budget?
- How does the work unit check it?
- What happens if the work unit overshoots?

**Who provides the budget.** The scheduler sets the budget based on
scheduling policy. A pipeline with many ready work units should use a
short budget (spread CPU time across units). A pipeline with few ready
work units should use a long budget (no reason to yield when nothing else
is waiting). The budget is passed to execute() via the Context.

```rust
impl Context<R, W> {
    /// Returns the remaining budget. The work unit should yield
    /// when this returns zero or negative.
    fn budget_remaining(&self) -> Duration;

    /// Returns true if the budget is exhausted.
    fn budget_exhausted(&self) -> bool;
}
```

The work unit checks `budget_exhausted()` at natural boundaries (after
each morsel, after each iteration of a loop, after each phase). It does
not check every instruction; that would be more expensive than the work.

**What happens if the work unit overshoots.** Nothing catastrophic. The
scheduler's time accounting notes that the unit ran over budget. If a unit
consistently overshoots by a large margin, the scheduler can reduce its
budget on the next dispatch (adaptive budgeting) or flag it for
monitoring. But there is no preemption. The scheduler trusts work units to
yield cooperatively. A work unit that ignores the budget and runs for 10
seconds blocks its worker thread. This is the same trust model as DuckDB's
cooperative cancellation and PostgreSQL's `CHECK_FOR_INTERRUPTS()`.

**Budget implementation.** The simplest implementation: the scheduler
records `Instant::now()` at dispatch time and passes the deadline
(dispatch_time + budget) to the Context. `budget_exhausted()` checks
`Instant::now() > deadline`. `Instant::now()` costs ~20 ns on modern
hardware (rdtsc + scaling on x86, cntvct_el0 on ARM). Checking once per
morsel (every 100-500 us) adds 0.004-0.02% overhead. Acceptable.

For no_std environments without a clock, the budget can be expressed as
an instruction count or iteration count rather than wall-clock time. The
work unit decrements a counter on each iteration and yields when it
reaches zero. This is what loimu uses for WASM fuel budgets (10M
instructions default). The counter is checked with a simple integer
comparison, costing effectively zero.

## 3. Waker Design

When a work unit yields `Pending` (waiting on I/O), the scheduler parks
it. Something must wake it when the I/O completes. The waker design
determines how that notification reaches the scheduler.

### 3.1 Option A: Callback-based waker

The work unit receives a closure that, when called, signals the scheduler.

```rust
struct CallbackWaker {
    wake_fn: Box<dyn Fn() + Send + Sync>,
}

impl CallbackWaker {
    fn wake(&self) {
        (self.wake_fn)();
    }
}
```

This is the `std::task::Waker` model. Tokio provides a waker backed by
its internal task queue. When `wake()` is called, the task is pushed onto
the runtime's run queue. The work unit (or the I/O code on its behalf)
clones the waker and passes it to the I/O driver.

Pros:
- Familiar pattern. Every async Rust developer knows wakers.
- Composable. Wakers can be wrapped, chained, combined.
- Event-driven. No polling. The scheduler sleeps until a waker fires.

Cons:
- Requires allocation. The closure (or vtable) lives on the heap. Each
  waker is at least one pointer + vtable (16 bytes on 64-bit). Creating
  and destroying wakers has allocation overhead.
- Requires `alloc`. Not available in no_std without an allocator.
- Lifetime management. The waker must outlive the I/O operation. If the
  work unit is cancelled while I/O is in flight, the waker might fire
  after the work unit is dropped. This is safe (the scheduler ignores
  wake signals for unknown work units) but requires the waker's closure
  to capture only stable handles (not references into the work unit).

### 3.2 Option B: Poll-based

The scheduler periodically polls all Pending work units.

```rust
// Scheduler main loop
loop {
    for unit in &self.pending_units {
        match unit.is_ready() {
            true => self.move_to_runnable(unit),
            false => continue,
        }
    }
    self.dispatch_runnable();
}
```

Pros:
- No waker allocation. No callback. No lifetime issues.
- Simple to implement.
- no_std compatible.

Cons:
- Latency. The work unit is not woken until the next poll cycle. If the
  scheduler polls every 1 ms, average latency is 0.5 ms. For I/O-bound
  work (HTTP responses at 50-500 ms), this is negligible. For tight
  loops where a Pending unit becomes ready in microseconds, this adds
  measurable latency.
- CPU waste. Polling all Pending units every cycle wastes CPU when no
  units are ready. With 100 Pending units polled every 1 ms, that is
  100K atomic loads per second. On a modern CPU this is irrelevant, but
  on an embedded target (loimu) it might matter for power consumption.

### 3.3 Option C: Channel-based

The work unit sends a oneshot channel sender to the I/O task. The I/O
task sends a wake signal on the channel. The scheduler selects on the
channel.

```rust
let (tx, rx) = oneshot::channel();
// give tx to the I/O task
io_task.on_complete(move |_| { let _ = tx.send(()); });
// scheduler selects on rx
self.pending_channels.push((unit_id, rx));
```

Pros:
- No polling. The scheduler blocks on a select/recv over all channels.
- Composable. Channels can carry payloads (not just wake signals).
- Familiar concurrency primitive.

Cons:
- Channel allocation per yield. Each Pending yield creates a channel.
  For work units that yield Pending infrequently (HTTP connectors: 1-2
  times per execute), this is fine. For work units that yield Pending
  rapidly (streaming data: hundreds of times), channel allocation
  pressure is real.
- Requires std or an allocator. Channels are not available in no_std
  without significant effort.
- Select complexity. Selecting over a dynamic set of channels requires
  something like `tokio::select!` or `crossbeam::select!`. This adds a
  runtime dependency.

### 3.4 Option D: Atomic flag array

The scheduler maintains a fixed-size array of atomic bools, one per work
unit slot. The I/O completion sets the flag. The scheduler scans the
array.

```rust
struct WakeSlot(u16);

struct WakeArray {
    flags: [AtomicBool; MAX_WORK_UNITS],
}

impl WakeArray {
    fn wake(&self, slot: WakeSlot) {
        self.flags[slot.0 as usize].store(true, Ordering::Release);
    }

    fn scan_and_clear(&self) -> impl Iterator<Item = WakeSlot> + '_ {
        self.flags.iter().enumerate().filter_map(|(i, flag)| {
            if flag.swap(false, Ordering::Acquire) {
                Some(WakeSlot(i as u16))
            } else {
                None
            }
        })
    }
}
```

Pros:
- Zero allocation. The array is stack- or static-allocated.
- no_std compatible. `AtomicBool` is available everywhere including
  embedded targets.
- Fast scan. Scanning 256 atomic bools is a single cache line read
  (256 bools = 256 bytes = 4 cache lines). On modern hardware this
  completes in ~20 ns. Even scanning 1024 slots takes ~80 ns.
- Simple. No closures, no channels, no vtables. The waker is a u16
  index. The I/O completion code stores `true` at that index. The
  scheduler scans and resumes.
- Thread-safe. `AtomicBool::store` is safe from any thread, any
  context, including interrupt handlers.

Cons:
- Polling, not event-driven. The scheduler must scan the array
  periodically. Same latency concern as Option B, but the scan is
  faster (no per-unit `is_ready()` call, just atomic loads).
- Fixed capacity. MAX_WORK_UNITS must be chosen at compile time. If
  exceeded, either panic or grow (which requires allocation).
- No payload. The wake signal is a single bit. The I/O completion
  cannot send a result through the wake slot. The work unit must obtain
  the result through a separate channel (a shared cell, a pre-allocated
  slot, or by re-invoking the I/O API to retrieve the result).

### 3.5 Recommendation

hilavitkutin should provide Option D (atomic flag array) as the built-in
mechanism. It is the lowest common denominator: works on no_std, works
on embedded, works in the kernel, zero allocation, trivial to implement,
trivial to test.

Consumers that want event-driven waking wrap Option D with their runtime's
native mechanism:

```rust
// Tokio integration (consumer code, not hilavitkutin code)
struct TokioWaker {
    wake_slot: WakeSlot,
    wake_array: Arc<WakeArray>,
    notify: Arc<Notify>,
}

impl TokioWaker {
    fn wake(&self) {
        self.wake_array.wake(self.wake_slot);
        self.notify.notify_one(); // wake the scheduler's Tokio task
    }
}
```

The atomic flag is the truth. The Tokio Notify is a performance
optimization that avoids polling (the scheduler's Tokio task awaits
notify_one instead of polling the flag array). If the Notify is missed
(race condition), the scheduler's periodic scan catches it. The flag
array is the safety net; the event-driven layer is the fast path.

This layering means hilavitkutin never depends on Tokio, mio, or any
async runtime. The scheduler's core loop is:

```rust
loop {
    // Scan wake flags
    for slot in self.wake_array.scan_and_clear() {
        self.move_to_runnable(slot);
    }
    // Dispatch runnable work
    self.dispatch_all_runnable();
    // If nothing is runnable and nothing is pending, exit
    if self.runnable.is_empty() && self.pending.is_empty() {
        break;
    }
}
```

Consumers add their event-driven layer around this loop. saalis wraps it
in a Tokio task that awaits a Notify between iterations. polka-dots runs
it as a plain loop (no Pending work units, so the scan is always empty).
loimu runs it inside its signal-driven scheduler, scanning between DAG
passes.

## 4. Partial Writes and DAG Consistency

When a work unit yields, has it written partial results to its output
columns? This question determines whether yielding can create data
corruption visible to downstream work units.

### 4.1 Scenario: Hash build with budget yield

A hash build work unit processes 10 morsels. After 3 morsels, it yields
BudgetExhausted. The partial hash table exists in the work unit's internal
state but has not been written to the output column.

The DAG handles this correctly without special logic. The dependency edge
from the build unit to the probe unit is not satisfied until the build
unit returns Complete. The probe unit is not in the runnable set. No
downstream work unit can observe the partial state.

This works because the DAG's unit of atomicity is the work unit's
`Complete` return. Between yields, the work unit is a black box. The
scheduler knows it has not completed; that is sufficient to keep
downstream units gated.

### 4.2 Scenario: Streaming connector with partial writes

An HTTP connector writes partial results as they arrive. It receives 5
of 50 entity responses and writes them to the output column. A downstream
aggregation work unit runs and sees only 5 entities.

Is this correct? Only under a streaming execution model where the
aggregation unit is designed to process incremental input. The aggregation
unit would need to run again after the connector writes more entities. This
is a fundamentally different execution model from the atomic-completion
model.

The streaming model requires:
- Partial write support in the DAG (edges that fire on partial writes)
- Version stamps that increment on each partial write
- Downstream units that can run incrementally (process new rows only)
- Early-release semantics (loimu's version stamp model)

This is complex. It mixes two concerns: the yield/resume mechanism (when
does the work unit run again?) and the data flow model (when does a
downstream unit see new data?). Mixing them creates ambiguity: does
`Yield(BudgetExhausted)` mean "I wrote partial results" or "I have not
written anything yet"?

### 4.3 Scenario: Singleton resource with partial update

A work unit modifies a config singleton. It updates field A, yields, and
another work unit reads the config. The reader sees field A updated but
field B still at its old value. The singleton is in an inconsistent state.

The DAG prevents this if the Read/Write declarations are correct: the
reader has `Read<Config>` and the writer has `Write<Config>`. The DAG
edge prevents the reader from running until the writer returns Complete.
Between yields, the writer has exclusive access (Write implies exclusive).

But what if the work unit writes to the column directly during execute()
and then yields? The column now holds partial data. If the scheduler
allows a different work unit with `Read` access to the same column to
run (which it should not, because the Write is still held), corruption
occurs.

The invariant: a work unit that yields retains its Write locks. No other
unit can Read or Write the same column until the yielded unit returns
Complete (or Failed). The scheduler must enforce this. This is
straightforward: the running set tracks which columns are locked by
which work units, and a yielded unit's locks are not released.

### 4.4 Design choice: no partial writes

Work units buffer results internally and write them all on Complete.

```rust
fn execute(&mut self, ctx: ...) -> Progress {
    match &mut self.state {
        State::Fetching { buffer, idx } => {
            // Process one batch
            let batch = self.fetch_batch(*idx);
            buffer.extend(batch);
            *idx += 1;
            if *idx >= self.total_batches {
                // All done, write everything at once
                let output = ctx.write::<EntityColumn>();
                for item in buffer.drain(..) {
                    output.push(item);
                }
                Progress::Complete
            } else {
                Progress::Yield(YieldReason::Pending(slot))
            }
        }
    }
}
```

Pros:
- Simplest correct semantics. No partial visibility. Downstream units
  see either nothing (unit still running) or everything (unit complete).
- Matches loimu's atomic behavior model. Behaviors run to completion;
  partial writes do not exist.
- No version stamp complexity during yield. The version stamp is
  incremented once, on Complete.
- Write locks are held across yields, but no data is exposed. If the
  work unit fails mid-execution, rollback is trivial: nothing was
  written.

Cons:
- Memory pressure. The work unit buffers all intermediate results. For
  a connector that fetches 50,000 entities, the buffer holds all 50,000
  in the work unit's state before writing to the column.
- Latency. Downstream units cannot start until all results are ready.
  A streaming model would allow downstream units to start processing
  early results while later results are still arriving.

For hilavitkutin's initial design, no partial writes is the correct
choice. The memory cost is bounded (work units process finite data), and
the latency cost is acceptable (downstream units wait for Complete, which
is the natural morsel-driven behavior). Streaming/incremental execution
is a future extension that can be built on top of the version stamp
infrastructure without changing the core yield contract.

### 4.5 Write lock retention across yields

The scheduler must track which work units currently hold Write access,
including yielded units. The data structures for this:

```rust
struct SchedulerState {
    // Which columns are Write-locked and by whom
    write_locks: HashMap<TypeId, WorkUnitId>,
    // Which work units are currently yielded
    yielded: Vec<WorkUnitId>,
    // When checking if a new unit can run:
    // - all Read<T> columns must not be Write-locked
    // - all Write<T> columns must not be Read-locked or Write-locked
}
```

When a work unit yields, its Write locks remain in `write_locks`. When it
returns Complete or Failed, its locks are released. This is identical to
how a database transaction holds row locks across multiple operations:
the lock is held for the duration of the transaction, not per-statement.

The implication: a work unit that yields Pending for a long time (seconds,
waiting for HTTP) holds its Write locks the entire time. Any work unit
that needs to Read or Write the same column is blocked. This is correct
behavior: the yielded unit will eventually resume and write its results.
Allowing another unit to write to the same column during the wait would
create a data race.

If lock contention on yielded units becomes a performance problem, the
mitigation is to reduce the scope of Write declarations. A connector that
writes to `EntityColumn` does not need to declare Write on every column;
it declares Write only on the column it actually writes. Other work units
that operate on different columns are not blocked.

## 5. Yield vs Split

An alternative to yielding: split a long-running work unit into multiple
shorter work units connected by DAG edges.

### 5.1 The split model

Instead of one HTTP connector work unit that yields 10 times, create 10
work units: HTTPCall_1 through HTTPCall_10. Each makes one call and
returns Complete. They can run in parallel (no inter-dependencies) or
sequentially (each depends on the previous, for paginated APIs).

```
// Parallel split: all calls are independent
HTTPCall_1 ──┐
HTTPCall_2 ──┤
HTTPCall_3 ──┼── Aggregator ── Output
HTTPCall_4 ──┤
HTTPCall_5 ──┘

// Sequential split: paginated API
HTTPCall_1 ── HTTPCall_2 ── HTTPCall_3 ── ... ── Aggregator ── Output
```

Each HTTPCall_N is atomic: it makes one request, processes the response,
writes its slice of the output, and returns Complete. No yield, no state
machine, no resume. The scheduler handles concurrency and ordering through
the DAG.

### 5.2 When split works

**Known iteration count.** If the work unit knows upfront how many
operations it needs (10 entities to fetch, 5 files to process), it can
be split into N atomic units. Each unit is small, fast, and simple.

**Independent operations.** If the operations are independent (fetch
entity A does not depend on fetching entity B), the split units can run
in parallel across workers. The scheduler maximizes concurrency
automatically.

**Natural phases.** Hash build and hash probe are naturally separate work
units. The build phase produces a hash table; the probe phase consumes
it. No yield needed; the DAG encodes the dependency.

### 5.3 When split fails

**Unknown iteration count.** A paginated API returns pages until the
response says "no more pages." The work unit does not know how many pages
exist. Splitting requires knowing N upfront. Options:

- Create one unit, yield between pages. Simple, works.
- Create an initial unit that fetches page 1, then dynamically creates
  a unit for page 2 upon completion, which creates a unit for page 3,
  and so on. This is dynamic DAG modification during execution.

Dynamic DAG modification is complex. The DAG must be mutable during
execution. Adding a new node requires recalculating topological order,
updating dependency edges, and potentially re-scheduling. This is possible
(Taskflow supports dynamic task creation) but adds significant
implementation complexity to the scheduler.

**Single long operation.** One HTTP request that takes 5 seconds cannot
be split. It is one operation. The work unit starts the request, yields
Pending, and resumes when the response arrives. Split does not apply.

**Shared mutable state across iterations.** A work unit that processes
events and accumulates state (e.g., building a summary from a stream of
updates) needs access to the accumulator across all iterations. Splitting
into separate units requires the accumulator to be an explicit resource
in the DAG, with Read/Write edges from each unit. This encodes the
sequential dependency correctly but adds DAG complexity.

### 5.4 When to split vs yield

| Situation | Recommendation |
|-----------|---------------|
| Known iteration count, independent operations | Split (parallel) |
| Known iteration count, sequential operations | Split or yield (equivalent) |
| Unknown iteration count | Yield |
| Single long operation (I/O wait) | Yield |
| Multi-phase operator (hash build → probe) | Split (natural phases) |
| Accumulating state across iterations | Yield (simpler than shared resource) |

The general heuristic: if the work can be decomposed into independent
atomic pieces at DAG construction time, split. If the decomposition
depends on runtime information (how many pages, when the I/O completes,
whether a retry is needed), yield.

Both mechanisms coexist in hilavitkutin. A pipeline graph contains both
atomic work units (split) and yielding work units. The scheduler handles
both uniformly: a yielded unit is re-queued; an atomic unit is dispatched
once. No separate mechanism is needed.

## 6. Interaction with Work-Stealing

When a work unit yields, the worker thread is free to process other work.
In a work-stealing scheduler, the worker either pops from its local queue
or steals from another worker's queue. This has implications for cache
behavior, affinity, and priority.

### 6.1 Cache impact

The yielded work unit's data is hot in the worker's L1/L2 cache. After
stealing and running other work, the cache is cold for the resumed unit.

For I/O yields (Pending for seconds): cache is cold regardless. The HTTP
response takes 50-500 ms. L2 cache lines are evicted in microseconds
under active computation. By the time the work unit resumes, its data was
evicted long ago. Yielding adds no additional cache cost.

For budget yields (BudgetExhausted after microseconds): cache impact is
real. The work unit ran for, say, 500 us processing morsels. Its working
set (input column, output column, intermediate accumulator) is hot in L2.
The worker steals other work, runs for 500 us, evicts the original data.
When the yielded unit resumes, it incurs L2 misses on its working set.

The cost of L2 misses depends on working set size. For a morsel of 4096
int64 values: 32 KB. L2 miss latency is ~10 ns on modern hardware.
Reloading 32 KB: 32768 / 64 (cache line size) = 512 cache lines * 10 ns
= ~5 us. If the work unit runs for 500 us per budget quantum, the cache
reload cost is 5/500 = 1%. Noticeable but not devastating.

For larger working sets (hash tables in the MB range), cache reload cost
is higher. But MB-scale hash tables live in L3 or DRAM, not L2. The
yield/resume does not change their access pattern; they were already
experiencing L3/DRAM latency during execution.

### 6.2 Worker affinity

Should a yielded work unit resume on the same worker (cache affinity) or
any worker (load balance)?

**Same-worker affinity.** The yielded unit is pushed onto the yielding
worker's local queue. When the worker finishes its current work, it pops
the yielded unit and resumes it. Data is more likely to be in L1/L2
(if the worker ran a short piece of other work between yields).

**Any-worker dispatch.** The yielded unit is pushed onto a global ready
queue. Any idle worker can pick it up. Better load balancing but no cache
affinity guarantee.

loimu's lane_compat bitmask provides a middle ground. Each work unit has
a bitmask indicating which workers (lanes) can execute it. A work unit
that wants same-worker affinity sets only its current worker's bit. A work
unit that accepts any worker sets all bits.

For hilavitkutin, the recommendation: push yielded units onto the
yielding worker's local queue by default (affinity). If the yielding
worker's queue is full, push onto the global overflow queue (any-worker).
This favors affinity when load is balanced and falls back to global
dispatch under load imbalance.

The implementation:

```rust
fn handle_yield(&mut self, worker_id: WorkerId, unit_id: WorkUnitId) {
    let worker = &mut self.workers[worker_id];
    if worker.local_queue.len() < LOCAL_QUEUE_CAPACITY {
        worker.local_queue.push_back(unit_id);
    } else {
        self.global_queue.push(unit_id);
    }
}
```

### 6.3 Priority of resumed work units

Should resumed work units have higher priority than new work? A partially
completed work unit holds Write locks on its output columns. Until it
completes, all downstream units are blocked. Resuming it unblocks more
work than starting a new independent unit.

The upward rank model from loimu handles this naturally. Upward rank is
the length of the longest path from the node to a DAG sink. Nodes with
high upward rank are on the critical path and should be prioritized.

A yielded unit's upward rank does not change between yields. If it was
high-priority before yielding, it remains high-priority after. The
scheduler's priority queue automatically prefers it over lower-rank new
units.

No special "resume priority boost" is needed. The DAG structure encodes
priority through rank. The scheduling algorithm does not need to
distinguish between "new unit" and "resumed unit"; both are entries in the
ready queue sorted by rank.

### 6.4 Stealing from yielded units

What if a worker is idle and the only available work is a yielded unit on
another worker's local queue? The idle worker steals it. This breaks
cache affinity but is correct: an idle worker doing something is better
than an idle worker doing nothing.

The work-stealing protocol handles this without modification. Yielded
units in a worker's local queue are stealable like any other queued work.
The FIFO steal order (steal from the bottom of the deque) means yielded
units that have been waiting longest are stolen first, which is the right
policy (they have been waiting longer, their cache data is already cold).

## 7. Testing Yielding Behavior

Yielding introduces a class of bugs that do not exist in atomic-execution
models: state corruption between yields, missed wakes, incorrect state
restoration, partial-write visibility. Testing must cover these failure
modes explicitly.

### 7.1 Deterministic yield injection

A test harness that forces the work unit to yield after every N operations,
regardless of time budget.

```rust
struct YieldEveryN {
    inner: Box<dyn WorkUnit>,
    n: usize,
    counter: usize,
}

impl WorkUnit for YieldEveryN {
    fn execute(&mut self, ctx: ...) -> Progress {
        self.counter += 1;
        if self.counter % self.n == 0 {
            return Progress::Yield(YieldReason::BudgetExhausted);
        }
        self.inner.execute(ctx)
    }
}
```

This wraps any work unit and forces it to yield at regular intervals. The
test runs the same work unit with N=1 (yield every call), N=2, N=10,
N=infinity (never yield), and verifies identical output across all runs.

If output differs between N values, the work unit's state machine is
broken: it either loses state across yields, writes partial results, or
has a non-deterministic code path triggered by the yield pattern.

### 7.2 Property-based testing

Use a property-based testing framework (proptest, quickcheck) to generate
random yield patterns and verify deterministic output.

```rust
proptest! {
    #[test]
    fn yield_pattern_determinism(
        yield_points in prop::collection::vec(1..100usize, 0..50)
    ) {
        let baseline = run_without_yields(&work_unit, &input);
        let result = run_with_yields_at(&work_unit, &input, &yield_points);
        assert_eq!(baseline, result);
    }
}
```

The property: for any set of yield points, the final output is identical
to the output of a non-yielding execution. This is the fundamental
correctness property of the yielding model.

### 7.3 Chaos testing

Randomly yield at different points across multiple runs. Verify
deterministic output. This is a broader version of property-based testing
that also exercises the scheduler's resume logic, work-stealing behavior,
and DAG dependency tracking.

The chaos test framework:

1. Build a DAG with 10-50 work units, some of which yield.
2. Run the DAG with a single worker (sequential, deterministic).
3. Record the final column state as the baseline.
4. Run the DAG with 4 workers, random yield injection, 100 times.
5. Verify each run matches the baseline.

Failures in chaos testing indicate concurrency bugs: races in the
scheduler's ready queue, incorrect Write lock handling, or cache-related
non-determinism in the work units.

### 7.4 Yield injection via test scheduler

A test scheduler that injects `Progress::Yield(BudgetExhausted)` after
each `execute()` call, forcing the work unit to prove it can resume
correctly from every possible state.

```rust
struct YieldInjector<S: Scheduler> {
    inner: S,
    force_yield: bool,
}

impl<S: Scheduler> Scheduler for YieldInjector<S> {
    fn dispatch(&mut self, unit: &mut dyn WorkUnit, ctx: ...) -> Progress {
        let result = unit.execute(ctx);
        match result {
            Progress::Complete if self.force_yield => {
                // The unit claims it's done, but let's verify it handles
                // the case where we override. This tests that the unit
                // can handle being called again after returning Complete.
                // If it can't, it's a bug (missing terminal state check).
                Progress::Yield(YieldReason::BudgetExhausted)
            }
            other => other,
        }
    }
}
```

This is aggressive: it converts Complete to Yield, testing whether the
work unit handles being called again after it thinks it is done. A
well-written work unit should either re-return Complete (idempotent
completion) or enter a terminal state on first Complete and detect the
extra call as an error.

### 7.5 Waker testing

The waker mechanism (Option D, atomic flag array) needs its own tests:

- Wake before scan: set flag, scan, verify the work unit is moved to
  runnable.
- Wake after scan: scan (nothing found), then set flag, scan again,
  verify the work unit is moved to runnable.
- Concurrent wake: two threads set the same flag simultaneously. Verify
  exactly one wake is delivered.
- Spurious wake: set flag, scan, clear flag, work unit re-executes,
  returns Pending again (I/O still not ready). Verify the unit is re-
  parked without data corruption.
- Double wake: set flag, scan and resume, while the I/O completion sets
  the flag again (redundant). Verify the second wake is harmless (the
  unit is already running or complete).

These tests verify the atomic flag array's correctness under contention.
They are simple because the mechanism is simple: atomic loads and stores
with Release/Acquire ordering.

## 8. Concrete API Sketch for hilavitkutin

Based on all findings, this section presents the complete yielding API.

### 8.1 The Progress enum

```rust
/// The result of a single `execute()` call on a work unit.
pub enum Progress {
    /// The work unit has completed all its work. Results have been
    /// written to output columns. The scheduler releases all locks
    /// and satisfies downstream dependency edges.
    Complete,

    /// The work unit cannot complete right now. It will be re-
    /// dispatched according to the yield reason.
    Yield(YieldReason),
}

/// Why a work unit yielded.
pub enum YieldReason {
    /// The work unit exceeded its time/iteration budget. It has
    /// more work to do and should be re-dispatched as soon as
    /// a worker is available.
    BudgetExhausted,

    /// The work unit is waiting on an external event (I/O, GPU
    /// fence, timer). It should be parked until the wake slot
    /// is signalled.
    Pending(WakeSlot),

    /// The work unit encountered a transient failure and wants
    /// to retry. The scheduler adds jitter to the backoff and
    /// delays re-dispatch accordingly.
    Retry {
        attempt: u32,
        max: u32,
        backoff: Duration,
    },

    /// The work unit encountered a permanent failure. The
    /// scheduler marks the unit as failed, releases locks,
    /// and skips downstream dependents.
    Failed(WorkUnitError),
}
```

Note: `Failed` is a YieldReason variant rather than a top-level Progress
variant. This is a deliberate choice. Failure is a form of "not Complete":
the work unit has not produced its output. Placing it under Yield means
the scheduler handles all non-Complete outcomes in a single match arm.
The scheduler does not re-dispatch a Failed unit; it is a terminal state.
But structurally, it shares the "did not Complete" codepath with
BudgetExhausted, Pending, and Retry.

An alternative placing Failed at the top level:

```rust
pub enum Progress {
    Complete,
    Yield(YieldReason),
    Failed(WorkUnitError),
}
```

Both shapes are functionally equivalent. The nested form (Failed inside
YieldReason) groups all non-Complete outcomes; the flat form (Failed at
top level) makes the terminal/non-terminal distinction explicit. The flat
form is slightly better for readability: `Progress::Complete` and
`Progress::Failed` are the two terminal states; `Progress::Yield` is the
only non-terminal state. The nested form muddles this by putting a
terminal state (Failed) inside a non-terminal variant (Yield).

Recommendation: use the flat form.

```rust
pub enum Progress {
    Complete,
    Yield(YieldReason),
    Failed(WorkUnitError),
}

pub enum YieldReason {
    BudgetExhausted,
    Pending(WakeSlot),
    Retry { attempt: u32, max: u32, backoff: Duration },
}
```

### 8.2 The WakeSlot type

```rust
/// An index into the scheduler's atomic wake array. Work units
/// receive a WakeSlot from the context and pass it to their I/O
/// code. The I/O code calls `wake_array.wake(slot)` when the
/// operation completes.
#[derive(Clone, Copy, Debug)]
pub struct WakeSlot(u16);
```

u16 gives 65,536 possible wake slots. For hilavitkutin's workloads
(tens to hundreds of work units), this is more than sufficient. The
atomic flag array is 65,536 bytes = 64 KB, which fits comfortably in
a single allocation. In practice, the array is sized to the actual
number of work units, not the full u16 range.

### 8.3 The WorkUnit trait

```rust
pub trait WorkUnit: Send + Sync + 'static {
    /// The set of columns this work unit reads.
    type Read: AccessSet;

    /// The set of columns this work unit writes.
    type Write: AccessSet;

    /// Execute one quantum of work. Called repeatedly until
    /// the work unit returns Complete or Failed.
    ///
    /// `&mut self` because the work unit IS the state machine.
    /// It holds mutable state across yields.
    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress;
}
```

**`&mut self` vs `&self`.**

loimu uses `&self` on behavior handlers because behaviors are stateless:
they read from resources, compute, and write to resources. All state
lives in resources, not in the behavior.

hilavitkutin uses `&mut self` because work units hold state across yields.
The HTTP connector stores its ConnectorState enum. The hash join stores
its partial hash table. This state is per-work-unit, not shared, so it
lives in the work unit itself.

`&mut self` guarantees exclusive access: only one thread executes a given
work unit at a time. This is essential for the state machine model. Two
threads executing the same work unit concurrently would race on the state
enum, corrupting it. `&mut self` makes this impossible at compile time.

`&self` with internal mutability (Mutex, Cell) is an alternative but adds
runtime cost and cognitive overhead. The work unit is already guaranteed
single-threaded execution by the scheduler (one work unit is dispatched
to one worker at a time). `&mut self` encodes this guarantee in the type
system rather than relying on runtime synchronization.

### 8.4 The Context type

```rust
pub struct Context<'a, R: AccessSet, W: AccessSet> {
    read_columns: R::Ref<'a>,
    write_columns: W::RefMut<'a>,
    wake_slot: WakeSlot,
    budget_deadline: Option<Instant>,
}

impl<'a, R: AccessSet, W: AccessSet> Context<'a, R, W> {
    /// Get a reference to a read column.
    pub fn read<T: Column>(&self) -> &T
    where
        R: Contains<T>,
    {
        self.read_columns.get::<T>()
    }

    /// Get a mutable reference to a write column.
    pub fn write<T: Column>(&mut self) -> &mut T
    where
        W: Contains<T>,
    {
        self.write_columns.get_mut::<T>()
    }

    /// Get the wake slot assigned to this work unit. Pass this
    /// to I/O code so it can signal the scheduler on completion.
    pub fn wake_slot(&self) -> WakeSlot {
        self.wake_slot
    }

    /// Check if the time budget is exhausted. Work units should
    /// check this at natural boundaries (after each morsel, after
    /// each loop iteration) and yield BudgetExhausted if true.
    pub fn budget_exhausted(&self) -> bool {
        match self.budget_deadline {
            Some(deadline) => Instant::now() >= deadline,
            None => false, // no budget = unlimited
        }
    }
}
```

The Context is lifetime-bound to the scheduler's dispatch scope. It
cannot be stored in the work unit across yields (the borrow checker
prevents it). This is correct: the context is valid only during one
execute() call. The scheduler may rearrange column storage between calls,
invalidating old references. The work unit must obtain fresh references
on each execute() call.

### 8.5 The WakeArray type

```rust
pub struct WakeArray {
    flags: Box<[AtomicBool]>,
}

impl WakeArray {
    pub fn new(capacity: usize) -> Self {
        let flags = (0..capacity)
            .map(|_| AtomicBool::new(false))
            .collect::<Vec<_>>()
            .into_boxed_slice();
        WakeArray { flags }
    }

    /// Signal that the work unit at this slot should be woken.
    /// Safe to call from any thread, any context.
    pub fn wake(&self, slot: WakeSlot) {
        self.flags[slot.0 as usize].store(true, Ordering::Release);
    }

    /// Scan all flags and return slots that have been woken.
    /// Clears each flag atomically (swap, not store) to avoid
    /// missing a wake that arrives during the scan.
    pub fn scan_and_clear(&self) -> Vec<WakeSlot> {
        let mut woken = Vec::new();
        for (i, flag) in self.flags.iter().enumerate() {
            if flag.swap(false, Ordering::Acquire) {
                woken.push(WakeSlot(i as u16));
            }
        }
        woken
    }
}
```

The `swap` in `scan_and_clear` is important. A naive implementation using
`load` then `store(false)` has a race: a wake signal arrives between the
load and the store, and the store clears it. `swap(false)` atomically
reads and clears, ensuring no wake is lost.

### 8.6 Scheduler dispatch logic

The scheduler's main loop, showing how the Progress variants are handled:

```rust
fn dispatch_one(&mut self, worker: WorkerId, unit_id: WorkUnitId) {
    let unit = &mut self.work_units[unit_id];
    let ctx = self.build_context(unit_id, worker);

    let progress = unit.execute(ctx);

    match progress {
        Progress::Complete => {
            self.release_locks(unit_id);
            self.satisfy_downstream(unit_id);
            self.mark_complete(unit_id);
        }
        Progress::Yield(YieldReason::BudgetExhausted) => {
            // Re-queue on the same worker for cache affinity
            self.requeue_local(worker, unit_id);
        }
        Progress::Yield(YieldReason::Pending(slot)) => {
            // Park the unit. Locks are retained.
            self.park(unit_id, slot);
        }
        Progress::Yield(YieldReason::Retry { attempt, max, backoff }) => {
            if attempt >= max {
                // Max retries exceeded, treat as failure
                self.handle_failure(unit_id, WorkUnitError::MaxRetriesExceeded);
            } else {
                let jittered = self.add_jitter(backoff);
                self.schedule_delayed(unit_id, jittered);
            }
        }
        Progress::Failed(error) => {
            self.handle_failure(unit_id, error);
        }
    }
}

fn handle_failure(&mut self, unit_id: WorkUnitId, error: WorkUnitError) {
    self.rollback(unit_id);       // restore pre-execution state
    self.release_locks(unit_id);
    self.skip_downstream(unit_id); // mark all downstream as skipped
    self.record_failure(unit_id, error);
}
```

The key dispatch properties:

- BudgetExhausted: re-queue locally (same worker, cache affinity). No
  lock release. No downstream notification.
- Pending: park (remove from dispatch). No lock release. The wake array
  will trigger resume.
- Retry: delay re-queue with jittered backoff. No lock release.
  Automatic conversion to Failed when max retries exceeded.
- Complete: release locks, satisfy downstream edges, mark done.
- Failed: rollback, release locks, skip downstream.

Locks are released only on terminal states (Complete, Failed). Non-
terminal states (BudgetExhausted, Pending, Retry) retain locks. This
ensures no other work unit can observe partial state.

### 8.7 End-to-end example: saalis connector pipeline

A saalis data pipeline with three stages: fetch from API, transform,
write to store. The fetch stage yields Pending while waiting for HTTP.

```rust
// Stage 1: HTTP fetch (yields Pending)
struct FetchEntities {
    state: FetchState,
    api_url: Url,
    buffer: Vec<Entity>,
}

enum FetchState {
    Ready,
    WaitingResponse(WakeSlot),
    Done,
}

impl WorkUnit for FetchEntities {
    type Read = (Read<HttpClient>,);
    type Write = (Write<RawEntities>,);

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        match self.state {
            FetchState::Ready => {
                let client = ctx.read::<HttpClient>();
                let slot = ctx.wake_slot();
                client.start_get(&self.api_url, slot);
                self.state = FetchState::WaitingResponse(slot);
                Progress::Yield(YieldReason::Pending(slot))
            }
            FetchState::WaitingResponse(slot) => {
                let client = ctx.read::<HttpClient>();
                match client.try_recv_response(slot) {
                    Some(Ok(response)) => {
                        self.buffer = parse_entities(response);
                        let output = ctx.write::<RawEntities>();
                        for entity in self.buffer.drain(..) {
                            output.push(entity);
                        }
                        self.state = FetchState::Done;
                        Progress::Complete
                    }
                    Some(Err(e)) if e.is_transient() => {
                        Progress::Yield(YieldReason::Retry {
                            attempt: 1,
                            max: 3,
                            backoff: Duration::from_millis(500),
                        })
                    }
                    Some(Err(e)) => {
                        Progress::Failed(WorkUnitError::from(e))
                    }
                    None => {
                        // spurious wake
                        Progress::Yield(YieldReason::Pending(slot))
                    }
                }
            }
            FetchState::Done => Progress::Complete,
        }
    }
}

// Stage 2: Transform (no yield, pure compute)
struct NormalizeEntities;

impl WorkUnit for NormalizeEntities {
    type Read = (Read<RawEntities>,);
    type Write = (Write<NormalizedEntities>,);

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        let input = ctx.read::<RawEntities>();
        let output = ctx.write::<NormalizedEntities>();
        for entity in input.iter() {
            output.push(normalize(entity));
        }
        Progress::Complete
    }
}

// Stage 3: Write to store (no yield)
struct PersistEntities;

impl WorkUnit for PersistEntities {
    type Read = (Read<NormalizedEntities>, Read<DatabasePool>,);
    type Write = ();

    fn execute(&mut self, ctx: Context<Self::Read, Self::Write>) -> Progress {
        let entities = ctx.read::<NormalizedEntities>();
        let pool = ctx.read::<DatabasePool>();
        pool.batch_insert(entities.iter());
        Progress::Complete
    }
}
```

DAG edges: FetchEntities --Write<RawEntities>--> NormalizeEntities
--Write<NormalizedEntities>--> PersistEntities.

Execution sequence:

1. Scheduler dispatches FetchEntities to worker 0.
2. FetchEntities starts HTTP request, returns Yield(Pending(slot 0)).
3. Worker 0 is now free. Scheduler dispatches other ready work.
4. HTTP response arrives. I/O completion sets wake_array[0] = true.
5. Scheduler scans wake array, finds slot 0, moves FetchEntities to
   runnable.
6. Scheduler dispatches FetchEntities to worker 0 (affinity) or any
   idle worker. FetchEntities reads response, writes to RawEntities,
   returns Complete.
7. Scheduler releases FetchEntities' Write<RawEntities> lock, satisfies
   the edge to NormalizeEntities.
8. Scheduler dispatches NormalizeEntities. It processes all rows, writes
   to NormalizedEntities, returns Complete.
9. Scheduler dispatches PersistEntities. It writes to the database,
   returns Complete.

During step 3, the worker is free for up to 500 ms (HTTP latency). In a
pipeline with 20 connectors and 4 workers, all 20 connectors can be in
flight simultaneously (all parked on Pending), and the 4 workers process
CPU-bound work while waiting. Maximum utilization. Zero blocking.

## 9. Open Questions for Future Work

Several design questions emerged during this deep dive that are not
resolved here but should be addressed in subsequent design rounds.

**Streaming/incremental execution.** The no-partial-writes constraint
(Section 4.4) is correct for the initial design but limits streaming
use cases. A connector that receives 50,000 entities should ideally
stream them to downstream units as they arrive, not buffer all 50,000
and write once. This requires partial-write semantics, version-stamped
incremental updates, and downstream units that handle incremental input.
The infrastructure (version stamps, early-release) exists in loimu's
design; the question is how to integrate it with the yield model.

**Yield from nested calls.** Section 1.4 identified the ergonomic cost
of manual state machines for deeply nested computations. A procedural
macro that transforms sequential code with `yield_pending!()` calls into
an explicit state machine enum would recover much of async/await's
ergonomics without requiring an async runtime. This is a compiler
engineering problem, not a scheduling problem.

**Budget calibration.** Section 2.4 described budget-based yielding but
did not specify how the scheduler chooses the budget. Adaptive budgeting
(adjust based on observed execution time) is the right approach, but the
specific algorithm (exponential moving average of past execution times,
percentile-based, fixed with feedback) needs empirical tuning.

**Cancellation propagation.** When a work unit is cancelled (e.g., the
user hits Ctrl-C), the scheduler must propagate cancellation to all in-
flight work units. For atomic work units this is straightforward (set a
flag, unit checks on next morsel boundary). For yielded-Pending units,
cancellation requires waking the unit, which then detects the
cancellation flag and returns Failed. The waker must fire even though
the I/O has not completed. This requires a "cancel wake" that is distinct
from an "I/O complete wake." The flag array does not distinguish them; the
work unit must check a separate cancellation flag after being woken.

**Memory pressure under many yields.** Section 4.4 identified that
buffering all results internally (no partial writes) creates memory
pressure for large work units. If 20 connectors each buffer 10,000
entities, that is 200,000 entities in work-unit state, separate from
the column storage. This doubles memory usage compared to a streaming
model. For polka-dots (small data) this is irrelevant. For saalis
(potentially large datasets), it warrants monitoring and possibly a
configurable spill-to-disk mechanism for work-unit internal buffers.
